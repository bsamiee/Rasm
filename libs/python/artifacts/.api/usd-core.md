# [PY_ARTIFACTS_API_USD_CORE]

`usd-core` (dist `usd-core`, import `pxr`) supplies Pixar's Universal Scene Description stage-authoring + USDZ-packaging surface for the artifacts scene rail beside the `vtk` USD exporter: a `pxr.Usd.Stage` authoring root (`CreateNew`/`CreateInMemory`/`Open`/`DefinePrim`/`Save`/`Export`), the `pxr.UsdGeom` typed-prim schema family (`Xform`/`Mesh`/`Points`/`Camera`/`Scope` + the `SetStageUpAxis`/`SetStageMetersPerUnit` stage axes), the `pxr.UsdShade` material/shader graph, the `pxr.Sdf` layer/asset-path/value-type substrate, the `pxr.Gf`/`pxr.Vt` math-value and typed-array vocabulary, and the `pxr.UsdUtils` asset-packaging family whose `CreateNewUsdzPackage(assetPath, usdzFilePath)` zips an authored `.usdc`/`.usda` layer (with all its dependency assets) into the `.usdz` AR/Apple deliverable. The package owner composes `pxr.UsdUtils.CreateNewUsdzPackage` into the `figures/scene#SCENE` `Scene3d` owner's `Export`-`USDZ` arm — the `vtk` `vtkUSDExporter` writes the rendered scene to a `.usdc` layer on the gated worker, then `CreateNewUsdzPackage(Sdf.AssetPath(usdc), usdz)` packages it — and never re-implements the USD crate file format (`.usdc` binary / `.usda` ASCII), the composition-arc layer stack, the Hydra render-graph, or the USDZ zip+alignment record layout the C++ USD runtime already owns. It is authoring/packaging only on the gated band — the cp315-core render path stays at `pyvista`/`vtk`, and mesh-file interchange stays at `geometry`.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: stage, prim, and layer roots
- rail: scene — `pxr.Usd`, `pxr.Sdf`

`Usd.Stage` is the single authoring root; the source kind (a new on-disk layer, an in-memory anonymous layer, or an opened existing layer) is a static-method choice on one type, never a parallel stage class — `CreateNew` takes a file path, `CreateInMemory` allocates an anonymous root layer, and `Open` resolves an existing layer. `Usd.Prim` is the one scene-graph node every typed schema (`UsdGeom.Mesh`, `UsdGeom.Xform`, ...) wraps via `Define`/`Get`; `Sdf.Layer` is the persisted substrate (`.usdc`/`.usda`) a stage composes and saves through. There is no per-format stage subtype — the layer format is the file extension `Stage.Export`/`Sdf.Layer.Export` writes (`.usdc` crate-binary, `.usda` ASCII), discriminated by suffix, never a parallel writer class.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: stage authoring and layer export
- rail: scene — `pxr.Usd`, `pxr.UsdGeom`, `pxr.Sdf`

The whole stage authors lazily — prims compose into the layer stack and write on `Save`/`Export`.

[ENTRYPOINT_SCOPE]: USDZ packaging and asset dependencies
- rail: scene — `pxr.UsdUtils`

`UsdUtils.CreateNewUsdzPackage(assetPath, usdzFilePath)` is the USDZ builder the scene `Export`-`USDZ` arm calls: it takes the root layer as an `Sdf.AssetPath` (NOT a bare `str`), gathers every dependency asset the layer references, and writes the aligned-zip `.usdz` deliverable, returning `bool` success. `CreateNewARKitUsdzPackage` is the same with the ARKit/QuickLook compliance constraints (single root layer, restricted file types) applied. `ExtractUsdzPackage` is the inverse, unpacking a `.usdz` to a directory. `ComputeAllDependencies` discovers the layer/reference/asset closure; `ModifyAssetPaths` rewrites asset paths (relocation). The packaging functions are Boost.Python overload stubs (no Python `__doc__`); the verified C++ signature is reflected below.

## [04]-[IMPLEMENTATION_LAW]

[SCENE_USDZ_PACKAGING]:
- import: `from pxr import Usd, UsdGeom, UsdShade, Sdf, Gf, Vt, UsdUtils` at boundary scope inside the gated scene worker only — `pxr` does NOT import on the cp315 core, so the import lives behind the `runtime` `anyio.to_process.run_sync` subprocess seam on the same sub-3.13 worker the `pyvista`/`vtk` render already crosses onto, never at module scope on the core. The dist name is `usd-core`; the import name is `pxr`; each submodule is a native `pxr/<Module>/_<module>.so` Boost.Python extension.
- band axis: the CMake/C++ OpenUSD wheel ships no cp315 build and the dist metadata declares a HARD `Requires-Python: >=3.9, <3.15` upper cap (NOT merely a wheel gap), the same native-ceiling band as the `pyvista`/`vtk` rows — but on the WIDER `python_version<'3.15'` marker than the `vtk` `python_version<'3.13'` floor, both bands resolving on the one sub-3.13 vtk worker, so the `USD`/`USDZ` `Export` arm crosses `to_process.run_sync(_render_export, ...)` onto that worker where `pxr` imports at module scope. The gate is the declared `<3.15` ceiling, which is why the marker is `python_version<'3.15'` (a companion band) and not a version pin; the cp315 core process imports neither `pyvista` nor `vtk` nor `pxr`.
- packaging axis: `CreateNewUsdzPackage(Sdf.AssetPath(usdc_path), usdz_path)` is the single USDZ builder the scene `Export`-`USDZ` arm calls — it gathers the root layer plus every dependency asset and writes the aligned-zip `.usdz`, returning `bool`; the `vtk` `vtkUSDExporter` writes the `.usdc` layer first (over `Plotter.render_window`), then this function packages it. The FIRST argument is an `Sdf.AssetPath`, NOT a bare `str` — a raw-string call raises `Boost.Python.ArgumentError`. `CreateNewARKitUsdzPackage` is the ARKit-constrained variant for the Apple QuickLook deliverable; `ExtractUsdzPackage` is the inverse — one packaging family, never a per-target zip function.
- authoring axis: a USD scene authored from scratch (rather than exported from VTK) is `Usd.Stage.CreateNew(path)` (or `CreateInMemory()`), then `UsdGeom.<Schema>.Define(stage, primPath)` per typed prim, then `Create<Attr>Attr(Vt.<Type>Array(...))` per attribute over the `Gf`/`Vt` value vocabulary, then `SetStageUpAxis`/`SetStageMetersPerUnit`/`SetDefaultPrim` stage metadata, then `GetRootLayer().Save()` or `Export(path)` — one stage root, typed prims defined by schema, never a parallel per-geometry-kind writer. The layer format (`.usdc` crate-binary / `.usda` ASCII) is the file suffix, discriminated by extension at export.
- value axis: a geometry attribute carries a `Vt` typed array directly (`Vt.Vec3fArray` for points/normals, `Vt.IntArray` for face counts/indices, `Vt.TokenArray` for interpolation tokens), and a transform-op a `Gf` value (`Gf.Vec3f`/`Gf.Matrix4d`/`Gf.Quatf`) — there is no per-vertex or per-element wrapper object, so a mesh's whole point buffer is one `CreatePointsAttr(Vt.Vec3fArray(points))` `Set`, never a vertex-by-vertex append.
- dependency axis: `ComputeAllDependencies(Sdf.AssetPath(layer))` returns the `(layers, references, assets)` closure a USDZ package will embed, and `ModifyAssetPaths(layer, fn)` rewrites asset paths for relocation before packaging — so a scene with external texture/reference assets packages a complete self-contained `.usdz` rather than a layer with dangling references.
- fault axis: the USD runtime raises `Tf.ErrorException` (missing asset, malformed layer, write failure) and the Boost.Python binding raises `Boost.Python.ArgumentError` for a wrong-typed argument (the `Sdf.AssetPath`-vs-`str` mismatch); both are mapped at the `_render_export` worker boundary into the `runtime` `RuntimeRail` fault, never surfaced bare across the interpreter seam, and a `bool` `False` return from `CreateNewUsdzPackage` is a packaging-failure rail, not a silent miss.
- evidence: each USD/USDZ export captures the scene file target (`usdc`/`usdz`), the authored prim count, the up-axis and meters-per-unit stage metadata, the dependency-asset count embedded, and the recovered-package content key as the `figures/scene#SCENE` `ArtifactReceipt.Scene(key, target)` fact (`target` the `SceneTarget` value), so the scene owner records the packaging without minting a separate receipt.
- boundary: usd-core owns USD stage authoring (`Usd.Stage`/`UsdGeom`/`UsdShade`/`Sdf`), layer serialization to `.usdc`/`.usda`, and USDZ packaging/extraction (`UsdUtils`) on the gated sub-3.13 worker — the `Export`-`USDZ` arm packages the `vtk`-written `.usdc` rendered scene (camera/lights/PBR surfaces), distinct from the `geometry` mesh-file codec (the `figures/scene#SCENE` `<-> geometry/mesh/repair#MESH_CODEC_BOUNDARY` seam records that scene = visualization-scene export and geometry mesh = mesh-file codec, no shared owner). Offscreen RENDERING is NOT this package's concern — the rendered raster stays at `pyvista`/`vtk`; the cp315-core render path imports no `pxr`; live USD viewers/Hydra render delegates stay outside the wheel (`usd-core` ships no Storm/embree render delegate).

[STACKING]:
- `usd-core` is the USDZ-packaging back half of the scene `Export`-`USD`/`USDZ` arm whose front half is `vtk` `vtkUSDExporter`: the VTK exporter writes the rendered scene to a `.usdc` layer over `Plotter.render_window`, then `UsdUtils.CreateNewUsdzPackage(Sdf.AssetPath(usdc), usdz)` packages that layer into the `.usdz` AR/Apple deliverable — so the two-package arm produces a USDZ with zero hand-rolled zip alignment, the `vtk` row owning the render-to-layer write and the `usd-core` row owning the layer-to-package zip.
- `Usd.Stage.CreateInMemory()` + `UsdGeom.<Schema>.Define` + `Export` lets the scene owner author a USD layer directly from a `numpy` point/index buffer (bypassing VTK) over `Vt.Vec3fArray(points)`/`Vt.IntArray(indices)`, so a pure-geometry USD export does not require a full render pass — one stage-authoring path the same `Export` arm reaches when the target is `USD` and no render is needed.
- `CreateNewARKitUsdzPackage` hardens the packaging policy to the Apple QuickLook constraint set (single root layer, restricted asset types) as a data choice on the same arm — an AR-deliverable scene selects the ARKit packager, a general USDZ selects `CreateNewUsdzPackage`, never a branching per-platform export owner.
- `ComputeAllDependencies` + `ModifyAssetPaths` close the self-contained-package invariant: before `CreateNewUsdzPackage`, the dependency closure is computed and any external asset path rewritten relative, so a scene authored with referenced textures packages a complete `.usdz` rather than one with broken asset references.
