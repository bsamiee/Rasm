# State-Receipts: Frontier

[TRIE_STRUCTURAL_SHARING]:
- The CHAMP trie underlying `HashMap<K,V>`, `AtomHashMap<K,V>`, and `HashSet<K>` uses an `(UpdateType Type, bool Mutate)` env tuple threaded through every node `Update` call. `Mutate: true` is passed only inside the `TrieMap(IEnumerable<(K,V)>)` and `TrieMap(ReadOnlySpan<(K,V)>)` constructors — a single owner mutates its own freshly-allocated nodes during batch construction. Every public API call after construction passes `Mutate: false`, which forces `SetItem<A>(items, index, value, mutate: false)` to clone the items array before modifying the slot. The mutate gate is structural: the immutable sharing guarantee is enforced by the `Mutate` flag, not by field finality. Code that constructs a trie from an enumerable and never exposes it between construction and the first `Add/SetItem` call avoids one clone per inserted key during initialization.
- The `Entries.Nodes` sub-node array is cloned on every path-rewrite even when `Mutate: false` — `SetItem(Nodes, num4, item2, env.Mutate)` is a full `Array.Copy` when `mutate = false`. A single-key update on a map with depth `d` allocates `d` new `Entries` nodes (path copy), each with a freshly cloned `Nodes[]`. Map depth is bounded by hash collision depth: distinct keys with disjoint hash bits share at most 7 levels (32-bit hash / 5-bit sections). The practical allocation cost of a single update scales with the trie depth, not the map cardinality.
- The `Collision` node type handles keys with identical hashes by holding a flat `(K,V)[]`. Every `Collision.Update` clones this array on every write, regardless of `Mutate` — `Collision` does not thread the `Mutate` flag into its `SetItem`. High-cardinality maps with a flawed `Eq<K>.GetHashCode` that collapses keys into collision nodes lose the path-copy sharing benefit entirely and degrade to linear scan + full clone on every update.

[SEQ_INTERNAL_STRUCTURAL_COSTS]:
- `SeqConcat<A>` (the internal type used when `AtomSeq.Concat(Seq<A>)` merges two `Lazy` or mixed-type segments) computes `Count` as `ms.Sum(s => s.Count)` — an O(segments) traversal each time `Count` is accessed. `AtomSeq<A>.Count` delegates directly to the internal's `Count`, so reading `AtomSeq.Count` on a concat-heavy sequence is not O(1). Observers that call `Count` per `Change` event on a frequently-concatenated `AtomSeq` should maintain a separate `Atom<int>` counter rather than reading structural `Count`.
- `SeqConcat<A>.Add(value)` delegates to the last segment's `Add` and rebuilds the `ms` list: `new SeqConcat<A>(ms.Take(ms.Count - 1).Add(value2))`. `ms` is itself a `Seq<ISeqInternal<A>>`, so `Take` and `Add` on it also exercise segment traversal. Repeated `AtomSeq.Add(x)` calls on a concat-internal sequence accumulate O(segments) overhead per call. Workloads that mix bulk `Concat` with per-element `Add` should use `AtomSeq.Swap(s => s.Add(...))` batched inside one CAS rather than interleaving individual `Add` and `Concat` calls.
- `SeqStrict.Append(right)` has a single-use in-place write gate: `Interlocked.Exchange(ref addDisallowed, 1)`. The first call wins and can copy `right.data` directly into the pre-allocated suffix space of the `data` array (if capacity allows), returning a new `SeqStrict` that shares the same backing array with a larger `count`. The second call finds `addDisallowed == 1` and always allocates a new array. This means two `AtomSeq.Concat(rhs)` calls on a freshly-strict sequence: the first is a potential in-place widen, the second forces a full copy. The gate is not reset by `Clone` or `Reset` — sharing the backing array propagates `addDisallowed` to the new `SeqStrict` wrapper that maps `count + right.count`.

[ATOM_SWAP_VALIDATOR_RETURN_CONTRACT]:
- `Atom<A>.Swap(Func<A,A> f)` returns the pre-swap value (`val`) when `validator(val2)` returns false, and returns the new post-swap value (`val2`) on CAS success. The return value therefore does not distinguish "validator rejected" from "the swap function produced a value identical to the current state" — if `f` returns the exact current value, the validator succeeds, the CAS is a no-op (same object reference for reference types, same boxed value for value types via `Box<A>`), and the method returns `val2 == val`. Callers that need to distinguish rejection from idempotent success must examine the atom's `Value` after the call, or use a validator that always returns `true` and encodes rejection in the function's domain via `SwapMaybe`.
- When `validator` rejects, `Swap` returns the pre-swap snapshot without retrying — the spin loop exits immediately on `!validator(val2)`. This means a rejecting validator is always called exactly once per `Swap` invocation (with the latest observed value), never more. The retry loop is only driven by `Interlocked.CompareExchange` failure, not by validator outcome. A validator that is expensive to compute pays that cost once per `Swap` call regardless of contention.

[REF_FINALIZER_GC_HAZARD]:
- `Ref<A>` has a finalizer that calls `STM.Finalise(id)`, which calls `state.Remove(id)` on the process-global `AtomHashMap<EqLong, long, RefState>`. If a `Ref<A>` becomes unreachable from GC roots outside a transaction while that transaction is active — for example, a `Ref` constructed locally inside an `atomic` lambda with no external capture — the GC can finalize it between the transaction's snapshot read and its `ValidateAndCommit` call. `CommitWrites` then attempts `s[write]` on the now-pruned global state, producing a default/null `RefState` lookup. The version comparison `s[write].Version == refState.Version` will compare against a zero-initialized `RefState.Version`, triggering a spurious `ConflictException` and an infinite retry loop if the same `Ref` is recreated with the same id (not possible — ids are monotonic) or the transaction replays and the id is absent (lookup returns default, version 0 != captured version N, loops forever). The safe pattern is to ensure all `Ref<A>` instances used inside `atomic` are strongly referenced from a live root outside the lambda boundary for the duration of the transaction.
- The GC hazard is exacerbated by `async`/`await` inside `atomic`: since `STM.transaction` is `AsyncLocal`, the transaction flows through continuations. A `Ref<A>` captured only as a local in the async method can become eligible for finalization if the compiler splits the state machine and the local is not live in a suspension frame. Capturing `Ref<A>` instances in fields or in the lambda closure that also captures `this` is the guaranteed safe form.

[REF_VALUEIO_SETTER_DEAD_WRITE]:
- `Ref<A>.ValueIO` has a C# `set` accessor that calls `value.Map(delegate(A v) { STM.Write(Id, v); ... })`. `IO.Map` is a lazy combinator — it returns a new `IO<Unit>` describing the write but never evaluates it. The `set` accessor returns immediately after constructing and discarding the mapped `IO`. Assigning `ref.ValueIO = someIO` is a no-op; the assignment silently does nothing. Any code relying on the `ValueIO` setter to write through an `IO`-producing expression will observe no state change. The correct write pattern inside `atomic()` is `ref.Value = val` or `ref.Swap(f)`.

[ATOMIO_VALIDATOR_ASYMMETRY]:
- `Atom<A>(value, Func<A,bool> validator)` (the synchronous Prelude factory) returns `Option<Atom<A>>` where `None` means the seed value failed validation — callers receive a typed absent value to handle.
- `atomIO<A>(value, Func<A,bool> validator)` (the `IO`-lifting path) throws `Exceptions.ValidationFailed` (an `Error`-subtype exception) when the validator rejects the seed, wrapped inside the `IO`'s effect boundary. The `IO<Atom<A>>` resolves to a failed `IO` on evaluation, not to `IO<Option<Atom<A>>>`. Callers cannot distinguish "IO failed due to validation" from "IO failed due to any other reason" without catching `ValidationFailed` specifically via `IO.Catch`. The two construction paths are not symmetric: the synchronous path returns presence/absence via the type; the `IO` path returns failure via the error rail.

[ATOMHASHMAP_FILTER_MUTATION_ASYMMETRY]:
- `AtomHashMap<K,V>.Filter(Func<K,V,bool> pred)` and `AtomHashMap<K,V>.Filter(Func<V,bool> pred)` return a **new** `AtomHashMap<K,V>` wrapping a filtered snapshot. They do not mutate the receiver and do not fire the `Change` event. The returned map is a detached cell with no shared observer wiring.
- `FilterInPlace(Func<V,bool> pred)` and `FilterInPlace(Func<K,V,bool> pred)` mutate the receiver via a CAS spin loop and fire `Change` with a `HashMapPatch` carrying all removed-key changes in the `Changes` trie. The name divergence (`Filter` vs `FilterInPlace`) is load-bearing: passing the result of `Filter` to a consumer expecting the updated live cell is a silent correctness bug — the live cell is unmodified.

[ATOMHASHMAP_ADDRANGE_OBSERVER_SUBSCRIPTION_RACE]:
- `AtomHashMap.AddRange(range)` evaluates `this.Change != null` before the CAS loop to choose between the `WithLog` (change-tracking) and the plain (no-log) trie path. The chosen path is fixed for the duration of that spin; if an observer subscribes to `Change` between the path selection and the CAS success, the committed operation used the plain path and `changes == null`. `AnnounceChanges(items, trieMap, null)` is a no-op — the new observer does not see the change event for that `AddRange` call. The subscription race window is one CAS spin cycle wide, but it is structurally inherent because the path selection is outside the CAS body. Observers that require guaranteed delivery from their subscription point must initialize their internal state by reading `Items` after subscribing.
- Contrast with scalar mutation paths (`SwapKey`, `SetItem`, `Add`, etc.): these check `this.Change != null` inside each CAS body iteration, so a newly subscribed observer is visible on the retry that follows a CAS miss. The subscription race window for scalar ops is bounded to the single successful CAS; for range ops it spans the entire pre-CAS path selection.

[ADDRANGEWITHLOG_INTERNAL_CHANGE_TRIE_CONSTRUCTION]:
- `TrieMap.AddRangeWithLog(IEnumerable<(K,V)>)` builds its change-log trie using `EmptyForMutating` — a trie instance permitted to mutate its own nodes in place — and accumulates per-key `Change<V>` values via `AddOrUpdateInPlace`. The result is a single immutable change-trie snapshot after all insertions complete. This is more allocation-efficient than the `Swap(TrackingHashMap)` path, which allocates a `TrackingHashMap` wrapper and uses `Change<V>.Combine` on every overlapping key during construction. `AddRangeWithLog` produces one `Change.None` → `Change.EntryAdded` per successful insertion; `Swap`+`TrackingHashMap` may produce `Combine`-collapsed `EntryMapped` records when a key is both present and mutated within one bulk function body.

[REF_IMPLICIT_OPERATOR_TRANSACTION_READ_TRACKING]:
- `Ref<A>` has `implicit operator A(Ref<A> value)` which calls `value.Value` which calls `STM.Read(Id)`. Inside an active `atomic` block, `STM.Read` calls `transaction.Value.ReadValue(id)` which unconditionally adds `id` to the transaction's `reads` HashSet. This means any code path that implicitly converts a `Ref<A>` to `A` — including passing it to an overloaded method that accepts `A`, using it in a LINQ query that expects `A`, or composing it with LanguageExt operators that accept `A` via `implicit operator` — silently tracks that read in the current transaction under `Serialisable` isolation. Unexpected elements appear in the read-set, broadening the conflict surface beyond what the author intended. Explicit `ref.Value` reads carry the same behavior but are visible at the call site.

[MEMO_RESET_MEMORY_ORDERING]:
- `Memo<A>.Reset()` transitions `state` from `2` to `1` via `Interlocked.CompareExchange`, then performs `value = default(A)` as a plain field write, then `state = 0` as a plain field write. The two non-interlocked writes following the CAS are not fenced. On weakly-ordered hardware (ARM64, which includes Apple Silicon), the CPU may reorder `state = 0` before `value = default`, making a thread spinning on `state != 0` observe `state == 0` before seeing `value == default`. In practice, `GetValue()` checks `state != 2` to decide whether to call `GetValue()`, and `GetValue()` wins the `CAS(state, 0, 1)` to transition to `loading` — so a thread that observes `state == 0` early and wins the loading CAS will call `acquire()` and overwrite `value` with the fresh result, making the stale non-null `value` invisible. The hazard is benign in the common case but produces spurious work. A `Memo<A>` that wraps a non-idempotent `acquire` (which is already forbidden by the design contract) would have observable consequences.
