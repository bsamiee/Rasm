## [02]-[EXACT_INSERTION_POINTS]

Where implicit coordinates must survive for the V4 ruled default to hold, in pipeline order:

1. **Birth** â€” `intersect.md` tri-tri / edgeĂ—plane crossing: the endpoint is a 3D line-plane construction over five INPUT points (edge `u,v` Ă— triangle `a,b,c`). The corpus has NO representation for it: `Lpi` is 2D lineĂ—line. Required: an LPI-3D construction row (3 numerators + Î» over 5 defining points).
2. **Carriage** â€” `Crossing`/`ImplicitCrossing`/`Constraint`: the defining-entity tuple (vertex ids of the edge and triangle) must ride to the substrate. Today a `bool IsImplicit` and a rounded `Point3d` ride instead. V4's `Constraint` implicit case is exactly this carrier.
3. **Identity** â€” the same geometric crossing born from adjacent faces (shared edge Ă— same triangle) must intern to ONE vertex. Implicit identity = same defining entities (exact, trivial). Today identity = bit-equality of independently-computed rounded doubles (`arrangement.md:284`; `intersect.md` `Chain` has NO cross-face merge key at all â€” `Section` never records the edge id).
4. **Substrate admission** â€” `Tessellation.Build`'s point set becomes mixed explicit/implicit; the vertex table must widen. `SimplexStore` (int indices) is untouched; only the `Point3d[] Vertices` payload becomes a two-kind row store.
5. **The walk** â€” `Locate`/`ExitFace` on an implicit QUERY against explicit corners = the landed `OrientLPI` shape (supported). But the first implicit vertex STORED makes every later walk touching its star need `Orient2D(implicit-stored, explicit, query)` â€” implicit in a defining position, unsupported. Two crossing constraints in one face put two implicit vertices in one triangle â€” multi-implicit combos, unsupported.
6. **The cavity** â€” `InCircum` with an implicit query against all-explicit corners = `InCircleLPI` (supported). Against corners that are themselves implicit = `incircle_{IIEE,IIIE,IIII}`-class, unsupported. This burden exists ONLY if the Delaunay restoration runs over implicit-bearing triangulations (see re-scope 4).
7. **Recovery-time crossings** â€” two implicit-endpoint constraint segments crossing IN-PLANE: both segments lie in the face's supporting plane and each in its own B-triangle's plane, so the crossing is a THREE-PLANE point over nine INPUT vertices (TPI-3D) â€” depth-1 over input entities is preserved with no nested construction, PROVIDED the constraint carrier keeps the original supporting entities. Sub-segment splits must re-anchor to the original entities (the depth-1 law).
8. **Emission** â€” `ToMesh`/`ToPolylines`/the V10c dual accessor materialize rounded coordinates ONCE at the seam. This is the sanctioned rounding point; topology and orientation are already decided exactly. The honesty contract per `[SEAM_AND_RAIL_LAW]` states it per owner.

## [03]-[WHAT_BREAKS_FIRST]

- **As designed, before any widening**: the exact path is already dead at insertion point 1 â€” `PlaneCrossPoint` rounds, `ImplicitCrossing` discards the payload, and `arrangement.md:467`'s implicit framing is falsified by its own `:258-262` (the brief's E7 row, confirmed on disk).
- **Granting the V4 `Constraint` widening alone**: the first hard break is INSIDE the insertion event â€” `Retriangulate` must append the new vertex to `Point3d[] Vertices` (`delaunay.md:90-91,122`); there is no implicit slot. - **Granting a hybrid vertex table**: the first unsupported predicate call is `ExitFace` on the next insertion whose walk crosses the implicit vertex's star (`Orient2D` with implicit in a defining position, `delaunay.md:154-162`), then `InCircum` with implicit corners (`:184-189`).
- **Floor defects that break the path even where the family applies** (all in `predicates.md`):
  (a) `OrientLPI`/`OrientTPI` mix the exact `Lambda` with the ROUNDED `double Denominator` (`d1.Estimate()`) inside the claimed-exact determinant (`predicates.md:228-242,522-531`) â€” a rounded coordinate INSIDE the exact branch;
  (b) the orient pair has no negative-Î» verdict flip (the in-circum pair flips via `RationalOracle.InCircum`; `Lpi`'s Î» is orientation-dependent and signs freely) â€” wrong verdicts for half the segment orientations;
  (c) `Tpi` drops the Z numerator entirely and computes its plane data through rounded `Vector3d.CrossProduct`/dot intermediates before the expansion lattice â€” `InSphereTPI`'s implicit point is forced to a Z=0 numerator and the construction is exact-over-rounded-intermediates, not exact over input;
  (d) `Lpi.Of`/the exact fallbacks feed rounded leaf differences (`s.X - r.X` as plain double) into `TwoProduct` â€” exact products of already-rounded factors (the reference computes leaf differences as error-free transforms);
  (e) the "a filter would require the very rounded coordinate the construction forbids" rationale (`predicates.md:15,572`) is false â€” the reference frame filters implicit predicates with semi-static + interval stages over the homogeneous coordinates, no rounded point required. Exact-only implicit members mean BigInteger-rational adjudication per cavity test at arrangement scale â€” a feasibility-grade throughput gap, not a correctness one.

## [04]-[REFERENCE_FRAME]

Verified against the exact-computation literature (implementation code where available, per proximity):

- Indirect predicates (Attene, Computer-Aided Design 2020; arXiv:2105.09772): an implicit parameter is a polynomial fraction Î»/d; ANY standard predicate is rewritten by substituting explicit parameters with fractions â€” implicit points in any position, any multiplicity ("an indirect predicate determines which of its input parameters are implicit, and internally calculates Î» and d for each of them"). Three cascaded computation models: FP with semi-static filter â†’ interval arithmetic (dynamic filter, directed rounding) â†’ exact expansions. The CDT of a 2D segment arrangement is the paper's named second-class application, with orient2d AND incircle over generic points.
- `MarcoAttene/Indirect_Predicates` header (the shipped family): `orient2d_indirect_{IEE,IIE,III}` plus `orient2dxy/yz/zx` axis-projected variants (the dominant-axis drop for 3D implicit points); `incircle_indirect_{IEEE,IIEE,IIIE,IIII}` plus `incirclexy` projected; `orient3d_indirect_{IEEE..IIII}`; `inSphere_{IEEEE..IIIII}`; `lessThanOnX/Y/Z_{IE,II}` exact coordinate comparisons (identity/ordering); every member with a `_t` interval-filtered variant. Constructions: `implicitPoint2D_SSI` (segmentĂ—segment, four points â€” the corpus `Lpi`'s true analog), `implicitPoint3D_LPI` (lineĂ—plane, FIVE points), `implicitPoint3D_TPI` (three planes as NINE input points, never derived plane values), `implicitPoint3D_LNC` (line + parameter; the CDT Steiner-on-segment carrier).
- Mesh arrangements (Cherchi-Livesu-Scateni-Attene, SIGGRAPH Asia 2020; `gcherchi/FastAndRobustMeshArrangements`): intersection vertices stay represented by their defining input entities end-to-end; the per-face re-triangulation is a CONSTRAINED (non-Delaunay) triangulation â€” `triangulation.cpp` contains zero incircle/Delaunay references, orientation family only (`orient2Dxy/yz/zx`, `orient3D`, `lessThanP`, `pointInTriangle`, `segmentsIntersectInside`) â€” deliberately dodging multi-implicit in-circum cost; coordinates round once at output.
- CDT-3D (Diazzi-Panozzo-Vaxman-Attene, TOG/SIGGRAPH Asia 2023; `MarcoAttene/CDT`): Steiner points added to make a PLC admit a CDT are carried implicitly through the extended taxonomy (LPI/TPI/LNC) â€” the 3D `Tetrahedralization`/`Facet` analog of this hinge, with the `orient3d`/`inSphere` multi-implicit family as its cost.
