# State-Receipts: Deepening

[STM_ISOLATION_MECHANISM]:
- `Isolation.Snapshot` (default) does not mean reads go unchecked: `CommitWrites` compares `s[write].Version == refState.Version` (the global committed version vs. the transaction-snapshot version) and throws `ConflictException` on mismatch. Write-write conflicts are always detected and cause a retry. Only the read-set validation (`ValidateReads`) is skipped — so non-repeatable reads and phantom reads are possible, but blind writes are not. The correct characterization is optimistic write-lock, not snapshot isolation in the database sense.
- `Isolation.Serialisable` adds `ValidateReads` before commit: it iterates `transaction.reads` (a `HashSet<long>` populated by every `STM.Read` call regardless of isolation mode) and checks version parity against the latest global state. Any version change in the read-set triggers `ConflictException` and a full retry. Reads accumulate unconditionally — a transaction that reads many `Ref`s under `Snapshot` pays the allocation cost with no commit-time benefit. Prefer `Serialisable` only when read-set validation is required.
- `Ref.Value` reads outside `atomic()` bypass read-set tracking — `STM.Read` checks `transaction.Value != null` first and reads directly from `state.Items[id].UntypedValue` when no transaction is active. The `state` field is an `AtomHashMap<EqLong, long, RefState>` and `Items` is a `volatile` trie reference, so direct reads are visibility-safe.

[STM_COMMIT_ARCHITECTURE]:
- The entire commit — validation, write application, and commute application — executes inside a single `state.SwapInternal(delegate ...)` CAS spin loop. Two concurrent `ValidateAndCommit` calls compete on this CAS; one wins, the other re-spins with the updated global state snapshot. This is the only global serialization point for STM commits.
- `commute` applies `f` to the transaction-local state snapshot immediately for in-transaction reads and appends `(id, f)` to the `commutes` list. At commit time, `CommitCommutes` runs after `CommitWrites` against the latest committed global state `s` — so commuted functions apply against values already advanced by `CommitWrites`. A `Ref` may appear in both `writes` and `commutes`; the write sets the version-verified value, then the commute applies `f` to that result. `RefValidationFailedException` is not retried inside `CommitCommutes` — it escapes `atomic()` directly.
- `STM.transaction` is `AsyncLocal<Transaction?>`, not `[ThreadStatic]`. A transaction started before `await` flows through continuations; `Task.Run` inside `atomic()` does not inherit the `AsyncLocal` value and throws `InvalidOperationException` on any `Ref` write.

[ATOMHASHMAP_OBSERVER_OPTIMIZATION]:
- Every scalar mutation on `AtomHashMap` (`SetItem`, `Add`, `Remove`, `SwapKey`, etc.) checks `this.Change != null` inside the CAS spin loop before choosing between the `WithLog` (change-tracking) and plain (no-log) trie path. When no observer is subscribed, the plain path runs and no `Change<V>` is allocated. A single subscribed handler forces the audit-log path on every operation.
- The bulk `Swap(Func<TrackingHashMap, TrackingHashMap>)` path always constructs a `TrackingHashMap<K,V>` wrapper regardless of whether `Change` is subscribed. The wrapper accumulates per-key `Change<V>` values using the `Change<V>` monoid `Combine`, which collapses multiple mutations to the same key into a single compressed change record. The patch delivered to observers reflects the net delta of the entire bulk function body, not intermediate states. For workloads that never subscribe observers, the constant `TrackingHashMap` allocation overhead on every bulk `Swap` call is the trade-off cost of the unified API.
- `HashMapPatch<K,V>` uses two internal representations: a scalar op stores a `(key, change)` pair; a bulk op stores a `TrieMap<K, Change<V>>`. The `Changes` property builds `HashMap<K, Change<V>>` on demand — O(1) single-entry construction for scalar ops, trie wrap for bulk ops. Observers that call `patch.Changes` on every event pay this allocation per event; high-frequency scalar-op observers should check `patch.Changes.Count == 1` first and use a fast path that avoids the full `HashMap` wrap.

[CHANGE_MONOID_COMPOSITION]:
- `Change<A>.Combine(y)` is the semigroup and monoid `Combine` used to collapse sequential changes to the same key. Composition laws verified from source: `NoChange + y = y`; `x + NoChange = x`; `x + Removed = Removed`; `Removed + Added(v) = Mapped(oldValue, v)`; `Added(x) + Mapped(→y) = Added(y)` (the add wins, destination updated); `Mapped(from, x) + Mapped(→y) = Mapped(from, y)` (origin preserved, destination updated). All other combinations reduce to `y`. Code that distinguishes insertion from in-place update based on `EntryAdded` vs. `EntryMapped` must use scalar `SwapKey` calls to receive per-operation atomics; bulk `Swap` collapses them.
- `EntryMapped<A,B>` implements `EntryMappedFrom<A>` and `EntryMappedTo<B>` as separate interfaces. Pattern-matching as `EntryMappedTo<V>` extracts only the new value; `EntryMappedFrom<V>` extracts only the old value; `EntryMapped<V,V>` gives both. Change observers that only need the new state should pattern-match `EntryMappedTo<V>` to avoid binding to the `From` type parameter when a type-changing mutation is possible.

[ATOMQUE_EVENT_ASYMMETRY]:
- `AtomQue<A>` fires distinct `Dequeued` and `Enqueued` events (`AtomDequeuedEvent<A>` and `AtomEnqueuedEvent<A>` delegates) after each successful CAS. `AtomHashMap` fires one `Change` event with a typed patch; `AtomSeq` fires no events; `Atom<A>` fires `Change` with the new value. The per-operation events on `AtomQue` map to producer-consumer notification without requiring a patch diff.
- `AtomQue.Clear()` fires `Dequeued` for every element present at the moment of CAS success by iterating the captured pre-clear queue. A concurrent `Enqueue` after the CAS may add an element that `Clear` does not notify. Observers that must account for every element must use `Dequeue` in a drain loop rather than `Clear`.

[MEMO_FAILURE_AND_RESET]:
- `Memo<A>.Reset()` transitions `state` from `2` to `1` via `Interlocked.CompareExchange`, sets `value = default(A)`, then sets `state = 0`. It is a no-op when `acquire == null` (literal-value memos created via `Memo(A value)` are permanently fixed and cannot be reset). A `Memo<A>` constructed with an acquire function can be reset and will re-acquire on the next `Value` access.
- `Clone()` constructs a new unloaded `Memo<A>` with the same acquire delegate, leaving the original's loaded state untouched. Use `Clone` when multiple independent acquisition lifecycles share the same computation — each clone acquires independently without coordinating through the original.
- `memoK<F,A>(Func<K<F,A>>)` wraps a higher-kinded lazy computation. The `~` operator defined on `Memo<F,A>` forces acquisition by calling `.Value`. Force only at the point where the effect is to be executed; using `~` at composition time defeats the lazy semantics.

[MEMO_WEAK_EVICTION_HAZARD]:
- `memo<A,B>(Func<A,B>)` wraps a `WeakDict<A,B>` whose values are `WeakReference<OnFinalise<B>>`. `OnFinalise` removes its own cache entry in its finalizer. If the caller does not retain the computed value, the entry becomes collectible immediately — `memo<int, SomeRef>(f)(42)` may call `f(42)` on the very next invocation. The per-key `object` lock in `syncMap` prevents double-computation within one GC cycle but provides no cross-GC-cycle guarantee.
- The per-key lock uses `syncMap.GetOrAdd(key, new object())`, which may allocate a lock object even on a cache-hit path and discard it if the key already exists. The lock object is removed via `syncMap.TryRemove` immediately after the protected block, so subsequent calls to the same key are unguarded. This design is correct for idempotent computations and incorrect for non-idempotent ones.
