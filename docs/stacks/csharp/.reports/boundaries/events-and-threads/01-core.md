# Events and Threads — Core Findings

[SUBSCRIPTION_AS_DISPOSABLE_VALUE]:
- A subscription is the detacher token returned by the attach site. The attach operation is `event += handler`, yielding no token; detach is `event -= handler`. Because the delegate identity must match exactly, the wrapper delegate captured for `+=` must be stored and reused for `-=`. The canonical form is a `sealed` struct implementing `IDisposable` that holds a `Seq<Action>` of detach lambdas, each closing over the exact wrapper instance used at subscribe time — not a secondary reference to the handler.
- `CancellationToken.Register` returns `CancellationTokenRegistration`, which is both `IDisposable` and `IAsyncDisposable` (verified against .NET 10 `System.Private.CoreLib`). The registration is live until `Dispose`/`DisposeAsync` completes; failing to dispose leaks the callback reference and prevents the registered action from being collected. When the token is already cancelled at the time of `Register`, the callback fires synchronously on the calling thread before `Register` returns, so no subscription object is enqueued.
- Subscription sets opened outside a reactivation scope accumulate across reactivation cycles. Any collection that appends on open without a matching clear on deactivate leaks. The invariant is: the subscription set is destroyed at the same point the owner transitions away from the live state; reactivation constructs a fresh set rather than appending to a retained one.
- When draining a subscription set (on deactivation or disposal), the fence is: all event handlers are detached first, before any acquired handle is disposed. Disposal that fires before detach allows a racing callback to touch a handle mid-teardown. The order is strictly `detach all → dispose handles`.
- Host-owned buses (message buses, event aggregators, application-level notification centers) deregister independently of the local subscription record. Re-querying availability after any host lifecycle event is required; the consumer must not assume the bus is still alive simply because the subscription object has not been disposed. The defensive pattern is: `bus is { IsAlive: true }` (or equivalent host-specific liveness probe) at every callback entry before acting.

[ATOM_CHANGE_EVENTS]:
- `Atom<A>.Change` is an `event AtomChangedEvent<A>` (a `delegate void` with a single `A` parameter). The event fires synchronously on the swapping thread immediately after the `Interlocked.CompareExchange` succeeds and the spin loop `break`s — before `Swap` returns. The swap function itself may be called multiple times in the spin loop; the event fires exactly once, with the committed value. The swap function must therefore be side-effect free; the change event is the single notification point.
- `Atom<M, A>` (the metadata-carrying form) exposes the same `Change` event but the swap function receives `(M metadata, A current)`. The change event carries only the new value, not the metadata or prior value. Subscribers needing prior-to-current diffs must hold their own shadow read.
- `AtomHashMap<K, V>.Change` fires as `AtomHashMapChangeEvent<K, V>`, carrying a `HashMapPatch<K, V>` that includes the previous map, the current map, and either a single key + `Change<V>` or a `TrieMap` of all changed keys. The event fires synchronously inside `AnnounceChange`/`AnnounceChanges`, which runs after the CAS in `Swap`. Like `Atom`, this fires on the swapping thread.
- `Ref<A>.Change` (STM-backed) fires inside the STM `finally` block for every transaction iteration, including conflict-aborted iterations. `t.changes` is the in-transaction proposed diff accumulated by `WriteValue`; it is not empty on conflict. This means subscribers to `Ref<A>.Change` will receive notifications for rolled-back writes. Consumers must treat these as candidate notifications and re-read the committed value if they require consistency.

[THREAD_MARSHAL_EXPLICIT_BOUNDARY]:
- `IO<A>.Post()` marshals the computation onto the `SynchronizationContext` captured in `EnvIO.SyncContext` at construction time. The implementation allocates an `AutoResetEvent`, posts a delegate that calls `Run(env)` (the synchronous runner, not `RunAsync`), awaits the event handle, then re-throws any captured exception. If `env.SyncContext` is `null`, `Post` falls through to an ordinary `RunAsync` on the current scheduler. The context is captured once at `EnvIO.New()`; all `Post()` calls inside that environment target the same captured context.
- `IO.syncContext` is an `IO<Option<SynchronizationContext>>` effect — it reads the live `EnvIO.SyncContext` from the environment at evaluation time. The pattern for one-time capture is: evaluate `IO.syncContext` at the composition root on the correct thread, inject the result into `EnvIO.New(..., syncContext: ...)`, then pass that environment down. Never re-read `SynchronizationContext.Current` inside an effect; use the captured value.
- `SynchronizationContext.Post` is fire-and-queue (non-blocking); `SynchronizationContext.Send` is synchronous and can deadlock if the target thread is blocked waiting. `IO.Post()` uses `Post` and `AutoResetEvent.WaitOneAsync` for the caller-side rendezvous — the calling fiber blocks on the wait handle while the UI thread processes synchronously. If the host context queues work behind an already-blocked UI thread, this becomes a deadlock. The guard is: only cross to the host thread from a fiber that is not itself holding the host thread.
- `MonadUnliftIO<M>.PostIO` delegates to `MapIO(ma, io => io.Post())`, meaning `Eff<RT, A>.PostIO()` lifts through the runtime transformer and applies `Post()` to the inner `IO`. The sync context must be in the environment at the point the `Eff` runs; it cannot be injected after the fact.

[FORK_AND_CONCURRENCY]:
- `IO<A>.Fork(Option<TimeSpan>)` spawns a `Task.Factory.StartNew` with `TaskCreationOptions.LongRunning`, bypassing the default thread-pool and allocating a dedicated OS thread. The forked computation receives a child `Resources` (constructed from the parent's `Resources`) and a child `CancellationTokenSource`; parent cancellation propagates via `env.Token.Register(tsrc.Cancel)`. Both the `CleanUp` (owns `CancellationTokenSource` + `CancellationTokenRegistration`) and the child `Resources` are disposed in the task's `finally` block on the forked thread.
- `ForkIO<A>` is a `readonly record struct` with two `IO` effects: `Cancel` and `Await`. Neither executes unless bound. The forked task runs unconditionally after `Fork` returns; failing to bind `Cancel` or `Await` does not stop execution. The only cancellation path is binding `Cancel` or cancelling the parent `EnvIO` token.
- Parent-to-child resource sharing is by reference: the child `Resources` holds a reference to the parent via the `parent` field in `Resources`. `Release` walks up to the parent if the key is not found locally. Disposing the parent environment while the fork is live invalidates the parent resource scope; the fork's own `finally` calls `resources.Dispose()` on the child scope only.
- `ThreadPool.SetMaxThreads` and `SetMinThreads` are the construction-time caps for the managed pool. Any native thread-pool or host-specific cap that differs from the managed limit must be recorded as construction evidence on the seam owner; `Fork` bypasses the pool entirely via `LongRunning`, so pool caps do not constrain forked workers but OS thread limits do.

[CHANNEL_AS_SEAM_MARKER]:
- `Channel.CreateUnbounded<T>()` returns an `UnboundedChannel<T>` with `runContinuationsAsynchronously: true` by default. The single-reader optimization (`UnboundedChannelOptions.SingleReader = true`) routes to a lock-free `SingleConsumerUnboundedChannel<T>`, which is safe only when a single consumer issues `ReadAsync`/`TryRead` at a time; concurrent readers on this path corrupt state.
- `Channel.CreateBounded<T>(0)` (capacity zero) returns a `RendezvousChannel<T>` (internal type, .NET 10). This is a synchronization point: every `WriteAsync` blocks until a matching `ReadAsync` is pending, and vice versa. Using it as a queue is an error; using it as a rendezvous signal between a producer and consumer that must meet before either proceeds is the intended shape.
- `Channel.CreateUnboundedPrioritized<T>(UnboundedPrioritizedChannelOptions<T>)` accepts an `IComparer<T>`; items are dequeued in comparator order regardless of insertion order. This is a .NET 10 addition. The option type carries `Comparer`; when `null`, items are ordered by `IComparable<T>` default.
- `BoundedChannelFullMode` has four cases: `Wait` (back-pressure, blocks writer), `DropNewest` (evicts the most recently added item), `DropOldest` (evicts the head), `DropWrite` (silently drops the incoming item, invoking `itemDropped` if provided). `DropWrite` with the `itemDropped` callback is the correct form for observer-owned rate limiting: the channel defines the policy; the callback surfaces the dropped value for telemetry.
- `ChannelReader<T>` and `ChannelWriter<T>` are independent abstract surfaces. Passing `ChannelReader<T>` to a consumer and `ChannelWriter<T>` to a producer enforces the seam: no consumer can complete or write, no producer can read or await completion. The `Channel<T>` owner holds both and is responsible for completing the writer side on shutdown; `Completion` on the reader will settle only after `TryComplete` is called on the writer side.
- `ChannelReader<T>.ReadAllAsync(CancellationToken)` returns `IAsyncEnumerable<T>` that consumes until the channel is completed; it does not complete on cancellation alone — the channel must be closed by the producer. Rate-limited consumers must use `WaitToReadAsync` + `TryRead` in a loop to interpose their own pacing between dequeues.

[STATE_CELLS_AND_SYNCHRONIZATION_GATES]:
- `System.Threading.Lock` (sealed class, .NET 9+, present in .NET 10) replaces `object` as the lock target. The `lock` keyword on a `Lock` instance generates `EnterScope()` / `Scope.Dispose()` instead of `Monitor.Enter/Exit`, eliminating the hidden boxing and the `SyncBlock` slot overhead. `IsHeldByCurrentThread` allows reentrancy detection without a try/finally guard. Use `Lock` over `object` for all synchronization gates.
- `Lock.EnterScope()` returns a `ref struct Scope`, making it the correct pattern inside `unsafe` kernels and stack-allocated contexts where `IDisposable`-in-heap is illegal. The `Scope` is not valid across `await` points; a gate that must span an async region requires a semaphore, not a `Lock`.
- An `Atom<Option<T>>` cell with a token-guarded `Swap` (`SwapMaybe`) is the correct shape for a singleton ownership seam: acquire sets `Some(token)`, release clears back to `None`. The `SwapMaybe` overload short-circuits when `f` returns `None`, leaving the cell unchanged; this is the safe conditional-take pattern. A `Swap(prev => ...)` that returns the prior value captures the race winner atomically.
- The distinction between a synchronous gate and an atomic state cell: a `Lock`/`Monitor` gate is earned at a boundary when the invariant requires multiple in-scope reads and writes to be mutually exclusive — i.e., the guarded region is a critical section. An `Atom` is earned when a single value must transition atomically from one state to another with no requirement for mutual exclusion across multiple values. STM `Ref` is earned when multiple values must change atomically as a unit. Conflating these produces either over-serialization (wrapping independent atoms in a lock) or under-protection (using atoms for multi-value invariants).

[SUBSCRIPTION_FENCING_AND_LATE_SUBSCRIPTION]:
- Late-subscription semantics must be declared at the seam: a subscriber that arrives after the first emission must be told whether it receives replay-last (the most recent value), snapshot-plus-delta (current state and subsequent changes), or rejection (subscription refused after a terminal event). Absence of this declaration causes ordering hazards where a consumer misses the initial state and receives only incremental diffs without a baseline.
- `Channel`-borne delivery is commit-ordered and non-missable for active consumers. A subscriber that starts consuming after items have been written but before the channel is completed will see all buffered items in order. This is snapshot-plus-delta for free when the channel is used as the delivery vehicle for committed changes: the buffer is the snapshot, subsequent writes are the delta.
- Fault receipts must flow on a separate `ChannelWriter<Error>` from projection results. Mixing errors into a `Channel<Either<Error, T>>` forces every consumer to unpack the sum type on every read; a dedicated fault channel lets nominal consumers remain total and fault monitors remain disjoint.
- A hot-path that submits typed intent (command/request value) and consumes cached receipts must never block on compute or storage in the submission loop. The pattern is: `TryWrite` the intent onto an unbounded channel, return a pre-allocated or pooled receipt handle to the caller, and resolve the receipt from a fold worker. The fold worker is the single named serial processor that owns the channel read loop and mutates no external state outside its own result cell.

[PRODUCERS_AND_CONSUMERS]:
- Producer never throttles for consumer. Back-pressure is the consumer's observable, surfaced via the channel's `FullMode`. When a bounded channel is full, `WaitToWriteAsync` blocks the producer until the consumer drains; the consumer sets its own pace by controlling the read rate. A producer that calls `Thread.Sleep` or `Task.Delay` to pace for an assumed consumer speed is incorrect; the channel's back-pressure signal is the correct gate.
- Observer-owned rate limiting: the consumer reads at a rate governed by its own `Schedule`-driven loop, `WaitToReadAsync` return timing, or explicit delay in the read loop. The producer writes unconditionally up to the channel's capacity policy. Combining `DropOldest` or `DropNewest` with a `itemDropped` callback gives the consumer telemetry about shed load without blocking the producer.
- Pre-frame propagation of scale/DPI changes follows the same law: the producer (host notification) writes to a channel; the consumer (frame preparation worker) reads and applies the value before any frame work begins. Reading the change inside the frame work (post-frame-start) causes a one-frame-behind rendering artifact. The channel buffer holds the most recent value; the consumer reads with `TryRead` at frame start, applying the latest or ignoring if nothing is pending.
- `IO<A>.Fold(Schedule, S, Func<S, A, S>)` implements a serial fold worker inside the effect system: the `IO` is repeated on the schedule and each result accumulates into state. This is the single named serial fold worker pattern in the `IO`/`Eff` surface. The fold runs in sequence; concurrent writes to the state cell from outside the fold break the serial invariant. Drain rendezvous is via `RepeatUntil` with a terminal predicate, or by cancelling the environment token.
