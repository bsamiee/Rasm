# Fields, Sampling, and Integration — Deepening

[INTEGRATOR_CONVERGENCE_MECHANICS]:
- `GaussKronrodRule.recursive_adaptive_integrate` applies a cubic change-of-variable before calling the non-adaptive kernel: the finite-interval case maps `t` through `(b-a)/4 * t*(3-t²) + (a+b)/2`, which is a degree-3 polynomial on `[-1,1]` — a different substitution than the GaussLegendre finite-interval path. The error estimate per node is `max(|GK_N - G_{(N-1)/2}|, |GK_N * 2ε|)`, clamping to machine-epsilon relative noise; this prevents the tree from recursing beyond numerical resolution. Depth exhaustion returns silently — `error` carries the accumulated local error at that leaf, not a convergence flag.
- `Integrate.GaussKronrod` at the facade defaults `targetRelativeError = 1e-8`, not `1e-10` as the `GaussKronrodRule.Integrate` signature documents; the two values are independent constructor defaults. Calling the facade with the value-returning overload (two `out` params omitted) discards both `error` and `L1Norm`; the cancellation diagnostic is invisible unless the caller uses the four-parameter overload `Integrate.GaussKronrod(f, a, b, out error, out L1Norm, ...)`.
- `DoubleExponentialTransformation.Integrate` dispatches to `NewtonCotesTrapeziumRule.IntegrateAdaptiveTransformedOdd` with `Enumerable.Range(0, 10)` as the level sequence — exactly 10 levels, each generating a new abscissa/weight array on-the-fly for levels ≥7 (precomputed levels: 0–6). The termination test uses a Richardson-like convergence accelerator: when the ratio `log(err_k)/log(err_{k-1})` falls in `(1.9, 2.1)`, the current error is replaced by its square root before the relative comparison — this detects quadratic convergence in the level refinement and tightens the stop criterion. `targetRelativeError` is divided by `5.0 * (b-a)/2` before the loop, so the effective interval-normalized tolerance is 5× tighter than the caller-specified value.
- Contrary to what the precomputed-array count of 7 suggests, levels 7–9 are generated on-the-fly via the `tanh(π/2 * sinh(t))` kernel with decreasing step sizes; the on-the-fly level count scales as `3 × 2^(level-1)` abscissas. Integrand evaluations at the maximum level (9) use 96×2 = 192 new abscissas per call, bringing total evaluations to ~760 for a converging 10-level integration.
- `GaussLegendrePointFactory._gaussLegendrePoint` is `[ThreadStatic]` with a single slot per thread — it caches only the most recently requested order. A field integration loop that alternates between two non-precomputed orders (e.g., 30 for one boundary, 50 for another) will recompute nodes on every call. The remedy is to construct `GaussLegendreRule` instances once and call their instance `Integrate` method, which stores the point in the instance rather than the thread-local slot.
- The 3D cubature `GaussLegendreRule.Integrate(Func<double,double,double,double>, ...)` reduces to `Integrate(z => Integrate(f(·,·,z), ...), ...)` — a lambda-captured outer quadrature over the 2D inner rule. Each outer evaluation re-enters the 2D kernel, which re-fetches the thread-local `GaussPoint` for the same order; because order is constant, the thread-local is valid and no recomputation occurs. The cost is `n² + n` function evaluations for even order `n` (no symmetry savings in the outer dimension) rather than `((n+1)/2)²` for odd order.

[INTERPOLATION_ALGEBRA_AND_CAPABILITY_MAP]:
- `Barycentric.InterpolateRationalFloaterHormannSorted(x, y, order)` computes barycentric weights as alternating-sign sums of products of inverse pairwise distances over blending windows of width `order`. The `order` parameter is the blending width `d` in the Floater-Hormann 2007 family — `d=0` recovers nearest-neighbor, `d=x.Length-1` is the full-degree polynomial. The factory without an `order` argument defaults to `x.Length - 1`. This interpolant supports neither differentiation nor integration through the `IInterpolation` contract; it is the correct route when data has near-singularities but the caller does not need derivatives or quadrature.
- `CubicSpline.InterpolateMakimaSorted` uses the modified Akima (MATLAB `makima`) slope formula: weights are `|Δm_{i+1} - Δm_i| + 0.5*|Δm_{i+1} + Δm_i|` versus the classic Akima `|Δm_{i+1} - Δm_i|`. The `0.5*|sum|` term prevents zero weights when consecutive slopes cancel — classic Akima assigns zero weight to both neighbors in that case, producing C¹ but visually flat transitions. MAKIMA requires ≥5 points; PCHIP requires ≥3. PCHIP enforces strict monotonicity by zeroing slopes at sign-change intervals and by clamping endpoint slopes with the `PchipEndPoints` helper, which applies the `|endpoint_slope| ≤ 3*|first_interval_slope|` rule.
- `TransformedInterpolation.InterpolateSorted(transform, transformInverse, x, y)` applies `transformInverse` to the `y` values in parallel via `CommonParallel.For` with chunk size 4096, then constructs a `LinearSpline` over the transformed values, and wraps it with a forward `transform` applied at evaluation time. Both capabilities (`SupportsDifferentiation`, `SupportsIntegration`) are `false`; the transform composition would require chain-rule differentiation and change-of-variables integration that are not implemented. Using a `CubicSpline` in the base position is not supported by this factory — it hardcodes `LinearSpline.InterpolateSorted`.
- `FiniteDifferenceCoefficients.CalculateCoefficients` builds the full coefficient table at construction via matrix inverse: for `p` points it constructs a `p×p` Vandermonde-like matrix, inverts it via `DenseMatrix.Inverse()`, multiplies each entry by `p!`, and rounds to the nearest integer divided by `p!`. The rounding is `MidpointRounding.AwayFromZero` and produces exact rational stencil weights. The matrix inverse is the dominant cost: `O(p³)`. For `p=5` this is negligible; at `p=15`, the 3375-operation solve runs once at construction and the resulting table is reused for all subsequent calls. The constructor allocates one `DenseMatrix` per center position (up to `p` matrices) sequentially — no parallelism inside `CalculateCoefficients`.
- `BulirschStoerRationalInterpolation.Interpolate(t)` initializes two working arrays `C` and `D` of length `N` with `y[i]` and `y[i] + 1e-25` respectively. The `1e-25` shift on `D` is the pole-avoidance perturbation; if the rational update denominator `(D[k+1] - C[k])` rounds to `AlmostEqual(0.0)` (the library's default ULP tolerance), the method returns `NaN` rather than throwing. Callers composing this over a field that might have poles must test for `NaN` on each evaluation rather than relying on an exception.
- `StepInterpolation` reports `SupportsDifferentiation: true` despite being a piecewise constant: the derivative is `0.0` everywhere within a segment and `double.NaN` at breakpoints. The implementation returns `0.0` for all `t` without NaN at breakpoints — it silently elides the distributional derivative at the jumps. Callers that use `StepInterpolation` in a quadrature loop expecting derivative accuracy will receive zero gradient everywhere inside each bin.

[SAMPLING_DISTRIBUTION_AND_DENSITY_MECHANICS]:
- `SystemRandomSource.Default` is a `ThreadLocal<SystemRandomSource>` instance, not a process-wide singleton. Each thread holds its own `SystemRandomSource` initialized with `RandomSeed.Robust()`, which calls `RandomNumberGenerator.GetBytes(4)` under a static lock. The `FastDoubles(values)` static method bypasses `Default` for large arrays (length ≥2048): it parallelizes via `CommonParallel.For` with block sizes of 1024, 2048, or 8192, constructing a fresh `System.Random` from `RandomSeed.Robust()` per block. The result is non-deterministic regardless of any per-thread seeding, because each block gets an independent OS-entropy seed.
- `HybridMC`'s default gradient delegate `Grad(DensityLn pdfLnP, double[] x)` is a centered three-point finite difference: `step = max(0.001, 1e-6 * |x[i]|)` for each component, with the absolute floor of `0.001` dominating when `|x[i]| < 1000`. This is a hardcoded absolute threshold, not a relative step derived from density curvature. Supplying a user-provided `DiffMethod` (the eight-parameter constructor overload) is mandatory for fields with sharp gradients near zero or for large-scale parameter spaces where `0.001` is either too coarse or too fine.
- `UnivariateHybridMC.Grad` uses the same step formula: `step = max(0.001, 1e-6 * x)` (scalar). Both constructors that omit the `diff` parameter use this three-point centered difference. The `burnInterval` parameter controls the number of initial Markov chain steps discarded; `Burn(BurnInterval)` is called in the constructor body, so the constructor is side-effectful and constructs the chain's burn-in state synchronously.
- `RandomSource` instances accept a `threadSafe` flag that wraps all `DoSample` calls in `lock(_lock)`. Thread-safe instances are documented as "two and half times slower than non-thread-safe classes." The `MersenneTwister`, `Xoshiro256StarStar`, and other generators all follow this pattern: construct with `threadSafe: false` for single-threaded field sampling, construct with `threadSafe: true` or use per-thread instances for parallel pipelines. Supplying the instance to a `Distribution.Samples` call does not confer additional thread safety — the distribution delegates directly to `DoSample` without additional locking.
- `KernelDensity.Estimate` parallelizes via `CommonParallel.Aggregate(0, count, kernel_call, add, 0.0)` — the per-element computation `kernel((x - samples[i]) / bandwidth)` runs in parallel, and partial sums are combined with floating-point addition. Parallel floating-point reduction does not preserve associativity order, so estimates are non-reproducible across thread schedules even with a fixed seed. If reproducibility is required, the kernel must be evaluated serially, which forfeits the automatic parallelism.

[SPLINE_INTEGRATION_EXACTNESS_AND_LAZY_SEMANTICS]:
- `CubicSpline.ComputeIndefiniteIntegral` computes a prefix-sum of segment integrals: `integral[i+1] = integral[i] + h*(c0 + h*(c1/2 + h*(c2/3 + h*c3/4)))` where `h = x[i+1] - x[i]`. This is Horner-evaluated, exact in floating-point for polynomials, and runs in `O(N)`. The indefinite integral array has length `N` (one fewer than the `x` array), indexed by left-segment index. Calling `Integrate(a, b)` evaluates `Integrate(b) - Integrate(a)`, each requiring one binary search via `Array.BinarySearch`. For a field integration that queries thousands of definite integrals over a fixed set of splines, warming the `Lazy<>` via a single `Integrate` call at construction (not at first query) eliminates lazy-initialization overhead from the hot path.
- The `SplineBoundaryCondition` enum governs `InterpolateBoundariesSorted`: values are `FirstDerivative` (pin `y'` at the boundary), `SecondDerivative` (pin `y''` — Natural at zero, Clamped at non-zero), and `Periodic`. The `Natural` and `Clamped` factories are sugar over `InterpolateBoundariesSorted` with `SplineBoundaryCondition.SecondDerivative`. For field domains with symmetry axes, `SplineBoundaryCondition.SecondDerivative` with value `0.0` (natural) minimizes overshoot; `FirstDerivative` with value `0.0` (clamped flat) is appropriate when the field is known to be extremal at the boundary.
- `LinearSpline.ComputeIndefiniteIntegral` uses the same prefix-sum pattern as cubic but with two coefficients; the lazy array is also `O(N)` to compute. `StepInterpolation.ComputeIndefiniteIntegral` accumulates `integral[i+1] = integral[i] + y[i] * (x[i+1] - x[i])`, which is a rectangle-rule sum over the piecewise-constant segments. All three types share the same `Lazy<double[]>` + `LeftSegmentIndex` via `Array.BinarySearch` pattern; the only difference is the polynomial degree evaluated per segment.

[DE_TRANSFORM_AND_CONTOUR_INTEGRATION]:
- `DoubleExponentialTransformation` does not handle infinite bounds internally — the infinite-domain substitutions live exclusively in `Integrate.DoubleExponential` (the facade method). The `DoubleExponentialTransformation.Integrate` static method on the class itself accepts only finite `[a, b]` and passes the integrand unchanged to `NewtonCotesTrapeziumRule.IntegrateAdaptiveTransformedOdd`. Calling the class method directly with `double.PositiveInfinity` passes infinity into the abscissa evaluation, producing `NaN` weights; only the facade-level `Integrate.DoubleExponential` applies the substitution. This is a documented difference between the facade and the underlying class.
- Both `GaussLegendreRule.ContourIntegrate` and `GaussKronrodRule.ContourIntegrate` accept `Func<double, Complex>` — a real-parameterized complex-valued integrand — and return `Complex`. These are line-integral contour variants: the integration variable is still a real scalar (parameterizing a contour path), not a complex variable. For computing path integrals of complex-valued scalar fields along a line segment in the plane, these are the correct routes. The `L1Norm` output in the Kronrod contour variant is `double`, not `Complex` — it accumulates `Complex.Abs` of each evaluation, making the cancellation diagnostic (`|result|/L1Norm`) still valid for complex-valued cancellation.
- The `Integrate.GaussLegendre` facade applies a distinct cubic substitution for the finite-interval case: `t → (b-a)/4 * t*(3-t²) + (a+b)/2`, which is Gauss-Lobatto-like smoothing of the integration domain. This substitution is applied on top of the already-mapped Gauss-Legendre nodes, effectively composing two changes of variable. The result is not a standard Gauss-Legendre rule on `[a,b]`; the effective quadrature points are the Gauss-Legendre nodes of the mapped domain, not equidistributed. Callers expecting the raw Gauss-Legendre nodes on `[a,b]` must use `new GaussLegendreRule(a, b, order)` with instance method `Integrate`, which applies only the linear affine map without the cubic substitution.

[ADAPTIVE_QUADRATURE_FAILURE_SURFACE_AND_DEPTH_SEMANTICS]:
- `GaussKronrodRule.recursive_adaptive_integrate` initializes `absTol = 0.0` and sets it to `|value * relTol|` at the first call (the root interval). Recursive calls halve `absTol` at each level; the dual condition `num3 < error2 AND absTol < error2` must both hold to recurse. The `absTol` halving prevents infinite recursion even on integrands where the relative criterion is never met, but also means that refinement in deep sub-intervals is not bounded to the original relative tolerance — an integrand with a spike covered by many levels can deliver accumulated error exceeding `relTol * |result|` because `absTol` splits geometrically, not proportionally to local magnitude.
- The `maximumDepth` cap in `recursive_adaptive_integrate` is decremented at each level without counting the actual number of function evaluations. At `maximumDepth = 15` and `order = 15` (15-point Kronrod), each non-adaptive kernel call evaluates `15` points (with the reflection symmetry using `~8` unique values). Full depth recursion generates `2^15 = 32768` leaves, each contributing ~8 evaluations: up to ~260,000 function evaluations. The default `maximumDepth = 15` is appropriate for smooth functions; for oscillatory or near-singular integrands in field integration, reducing to 8–10 trades accuracy for bounded evaluation count.
- Neither `DoubleExponentialTransformation` nor any adaptive Gauss-Kronrod overload catches `NaN` or `Infinity` propagated by the integrand. If the field evaluator returns `NaN` at a quadrature node (e.g., a signed-distance function evaluated at its zero set), the returned integral is `NaN` with no diagnostic. The only defense is to wrap the integrand delegate with a guard that returns a finite fallback at singular points, recorded as a typed failure outside the integral call.

[THREAD_LOCAL_CACHING_AND_PARALLEL_COMPOSITION]:
- `GaussLegendrePointFactory` and `GaussKronrodPointFactory` both hold a `[ThreadStatic] GaussPoint` field that caches only the most recently requested order for that thread. Parallel field integration with `Task.WhenAll` or `Parallel.For`, where each task uses a different order, produces cache misses on every call — the thread-local slot stores only one value. The correct pattern for parallel multi-order integration is to construct named `GaussLegendreRule` instances once at setup and capture them in closures; the instance holds `_gaussLegendrePoint` as a private field rather than in the thread-local slot.
- `CommonParallel.Aggregate` in `KernelDensity.Estimate` uses an internal partitioning scheme that does not expose a degree-of-parallelism cap to the caller. When a field integration pipeline spawns multiple concurrent KDE evaluations (e.g., importance-sampled density estimates for different field regions), the sum of all parallel KDE tasks can saturate the thread pool. Wrapping each KDE call in a `Task.Run` and using `ParallelOptions.MaxDegreeOfParallelism` at the outer parallel level does not limit the inner `CommonParallel.Aggregate` thread usage — the two parallelism layers are independent.
- `Xoshiro256StarStar` serializes its 64-bit state as `[DataMember(Order=1..4)]` via `[DataContract]`. This means a seeded instance can be serialized mid-sequence and deserialized to reproduce the exact continuation of the sample stream. For blue-noise sampling or deterministic field sampling that must checkpoint and resume, serializing the generator state is the correct approach. `MersenneTwister` carries the same `[DataContract]` attribute; `SystemRandomSource` wraps `System.Random` which does not support serialization.
