# Parameter Algebra — Deepening

[CONVERSION_ASYMMETRY_AND_ACCESS_GATES]:
- Ad-hoc unions (`[Union<T1,T2>]`) inherit from `UnionAttributeBase`, which exposes both `ConversionFromValue` (default `Implicit`) and `ConversionToValue` (default `Explicit`). Regular `[Union]` exposes only `ConversionFromValue`. The asymmetry is intentional: ad-hoc unions are structurally typed composites where extracting a member case back to its naked type is an explicit operation, whereas regular unions rely on `Switch`/`Map` for case consumption. Setting `ConversionToValue = ConversionOperatorsGeneration.None` on an ad-hoc union forces all member extraction through generated accessors, eliminating ambiguous implicit downcast paths.
- `UnionAttributeBase.ConstructorAccessModifier` defaults to `UnionConstructorAccessModifier.Public` for all union forms. `ValueObjectAttributeBase.ConstructorAccessModifier` defaults to `AccessModifier.Private`. This asymmetry is the architecture signal: union case construction is open at the call site by default; value-object construction is always factory-gated. Setting `ConstructorAccessModifier = Private` on a union simultaneously privatizes both the constructor and the implicit conversion operators, mandating factory methods for case creation — the constraint is a single attribute property, not a split between two settings.
- `UnionAttributeBase.UseSingleBackingField = true` forces an `object` backing field for all member cases, including struct cases. The struct case is then boxed on storage and unboxed on access. This is a memory-reduction trade-off when the union carries many cases whose sizes differ significantly; by default, each case gets its own typed backing field sized to that case. The parameter algebra consequence: struct stateless cases with `T1IsStateless = true` already store only the discriminator index with zero payload, making `UseSingleBackingField` irrelevant for those slots but still affecting the other slots.
- `FactoryMethodGeneration` on `UnionAttributeBase` controls factory method generation for ad-hoc union members only. `Default` detects triggers: type-parameter members, `interface` or `object` members, or duplicate member types cause factory method generation. `None` suppresses generation globally. `Always` generates for every member regardless of triggers. This is not the same control as `SkipFactoryMethods` on value objects — union factory methods are the only construction path when `ConstructorAccessModifier = Private`, so `FactoryMethodGeneration = None` combined with `ConstructorAccessModifier = Private` produces a type with no public construction path at all; the custom factory must be authored manually.

[STATELESS_CASE_DISCIPLINE]:
- `T1IsStateless = true` on an ad-hoc union member instructs the generator to store only the discriminator index — no payload. The generated accessor `AsT1` returns `default(T1)`. For struct `T1` this is the zero-value struct; for class `T1`, `T1IsNullableReferenceType` is automatically coerced to `true` (confirmed from source: `typeof(T1).IsClass && T1IsStateless`), so the accessor returns `null`. The correct policy for stateless cases: use struct marker types so the null-return is structurally impossible and the accessor always returns a well-formed sentinel.
- A stateless union case carries no information beyond the fact that the case was selected. This is the correct shape for policy entries that have no configuration data: a `None`, `Default`, or `Absent` case in a request union. The case discriminant is the entire semantic content. Passing stateless cases as `Option<T>` parameters adds a second layer of optionality on top of the discriminant; the correct form keeps the stateless case as a union member and leaves `Option<T>` for parameters where the entire union is absent.

[ARGUMENT_REF_STRUCT_CONSTRAINTS]:
- `Argument<T>` is a `readonly ref struct where T : allows ref struct`. It cannot be boxed, stored in a field, used as a generic type argument in `Option<Argument<T>>`, or held across an `await` boundary. Its use is strictly as a method parameter or local variable within a synchronous call frame. The generated `ValidateFactoryArguments` partial method does NOT use `Argument<T>` — it takes `ref T` for each declared field. `Argument<T>` is a design-time primitive for custom factory methods where the factory author needs to distinguish "caller explicitly supplied a value that happens to equal the domain default" from "caller omitted the argument."
- When a custom factory implements `IObjectFactory<T, TValue, TValidationError>`, its `static abstract Validate(TValue? value, IFormatProvider? provider, out T? item)` method receives the raw `TValue?` — not `Argument<TValue>`. The `Argument<T>` pattern is for bespoke factory overloads authored alongside `[ObjectFactoryAttribute<T>]`, not for the generated `Validate` contract. Mixing the two — expecting `Argument<T>` in a generated interface — is a category error that produces a compile-time mismatch against the `IObjectFactory<T, TValue, TValidationError>` static abstract signature.
- `IObjectFactory<TValue>` (one type parameter) is a marker interface with no members. `IObjectFactory<T, TValue, out TValidationError>` (three type parameters) inherits from it and adds `static abstract TValidationError? Validate(TValue?, IFormatProvider?, out T?)`. When `SkipFactoryMethods = true`, neither interface is implemented — the cascade also drops `TypeConverter`, the implicit conversion operator from key type to value object, `SkipIParsable` is forced true, and all arithmetic operator generation is suppressed. The `[ObjectFactoryAttribute<T>]` route is the replacement: it registers a custom factory keyed on `T` and can route `UseForSerialization`, `UseWithEntityFramework`, and `UseForModelBinding` independently per factory, allowing different conversion paths for JSON, EF Core, and model binding from one domain type.

[SCHEDULE_OPERATOR_SEMANTICS]:
- `Schedule + Schedule` (Semigroup `Combine`) calls `SchCombine`, which concatenates the two streams sequentially: the first schedule's durations run to exhaustion, then the second's durations continue. `Schedule | Schedule` calls `SchUnion`, which runs both in parallel and emits the minimum duration at each step until the longer stream exhausts. `Schedule & Schedule` calls `SchIntersect`, emitting the maximum duration at each step until the shorter stream exhausts. These are three semantically distinct policy composition operators sharing no behavior; conflating `+` (sequential concatenation) with `|` (parallel minimum) produces schedules that run more iterations than intended.
- `ScheduleTransformer + ScheduleTransformer` composes two transformers left-to-right: `f + g` produces `x => g.Apply(f.Apply(x))`. The composition is not commutative: `NoDelayOnFirst + jitter(0.1)` removes the first delay and then adds jitter to all remaining, while `jitter(0.1) + NoDelayOnFirst` adds jitter first and then replaces the first jittered duration with zero. Transformer composition order is the stacking order for policy mutation, not for execution of the base schedule.
- `ScheduleTransformer` carries an implicit conversion to `Schedule`: `implicit operator Schedule(ScheduleTransformer t) => Schedule.Forever | t`. A bare `ScheduleTransformer` coerced to `Schedule` without an explicit base produces an infinite schedule bounded only by the transformer's internal logic. `Schedule.recurs(3)` as a transformer becomes `Schedule.Forever.Take(3)` when coerced — three iterations with zero delay between each. This coercion is load-bearing when storing a transformer in a field typed `Schedule`; the `Forever` base is invisible at the declaration site.
- `Schedule.NoDelayOnFirst` and `Schedule.RepeatForever` are static `ScheduleTransformer` fields, not `Schedule` values. They can only enter a policy through transformer application (`| NoDelayOnFirst`) or transformer composition (`myTransformer + NoDelayOnFirst`), never by direct assignment to a `Schedule`-typed field without implicit coercion. Storing them in a `static readonly Schedule` field triggers the implicit coercion and locks in `Forever` as the base.

[FROZEN_TABLE_ALTERNATE_LOOKUP_CONSTRAINTS]:
- `FrozenDictionary<TKey, TValue>.TryGetAlternateLookup<TAlternateKey>` succeeds only when the dictionary's `Comparer` implements `IAlternateEqualityComparer<TAlternateKey, TKey>`. For string-keyed frozen dictionaries, an additional constraint holds: `typeof(TAlternateKey)` must equal `typeof(ReadOnlySpan<char>)` — no other alternate key type is accepted for `string` keys regardless of comparer capabilities. `GetAlternateLookup<TAlternateKey>()` throws `InvalidOperationException` when the constraint is unsatisfied; `TryGetAlternateLookup` returns `false`. The practical shape: `FrozenDictionary<string, TPolicy>` with the default `StringComparer.Ordinal` comparer and a `ReadOnlySpan<char>` alternate lookup is the only span-keyed policy-table path in the BCL.
- `IAlternateEqualityComparer<TAlternate, T>` requires three members: `bool Equals(TAlternate, T)`, `int GetHashCode(TAlternate)`, and `T Create(TAlternate)`. The `Create` method is used by the frozen table's own `ContainsKey(TAlternateKey)` implementation to generate a canonical key from the alternate for probing. Callers accessing policy rows by `ReadOnlySpan<char>` key do not pay a string allocation for the lookup; the `Create` call is only invoked internally when the implementation needs to construct a canonical key for some operations. The `AlternateLookup<TAlternateKey>` struct itself is a lightweight wrapper holding the frozen dictionary reference and a cached delegate for the alternate-compare path.
- `FrozenDictionary<TKey, TValue>.GetValueRefOrNullRef(TKey)` returns `ref readonly TValue`. Callers test for absence via `Unsafe.IsNullRef(in result)` — no boxing of a `TValue` default, no heap allocation for the miss path. Combined with `AlternateLookup`, a policy-table lookup on a `ReadOnlySpan<char>` key is allocation-free end-to-end. The corresponding `FrozenSet<T>.AlternateLookup<TAlternate>` exposes `TryGetValue(TAlternate, out T)` rather than a `ref readonly` path, so membership vocabulary lookups with span keys do not share the ref-return optimization.

[CATCHM_GENERIC_EXTENSION_BLOCK]:
- The `|` operator combining `K<F, A>` with `CatchM<E, F, A>` is defined as a C# 14 extension block in `FallibleExtensions` constrained to `where F : Fallible<E, F>`. It is not per-type (not defined separately on `Fin<A>`, `Eff<A>`, etc.) — any monad implementing `Fallible<E, M>` inherits the operator through the extension block. This means a `static readonly CatchM<Error, Eff<RT>, A>` named catch policy field is directly applicable to `K<Eff<RT>, A>` values via `|` without monad-specific wiring.
- `CatchM<E, M, A>.Run(error, otherwise)` checks `Match(error)`: if the predicate returns `false`, it returns `otherwise` unchanged; if `true`, it invokes `Action(error)`. The `|` operator chains this as `lhs.Catch(rhs)`, which calls `F.Catch(fa, rhs.Match, rhs.Action)`. A chain of `| namedCatch1 | namedCatch2` is evaluated left-to-right; the first matching catch consumes the error. Ordering of named catch values in a chain is semantic: a broad predicate (`@catch(e => true, fallback)`) absorbs all errors and must appear last or it shadows narrower catches.
- `bracketIO`'s `Catch: Func<Error, IO<C>>` parameter is a raw function, not a `CatchM`. A named `static readonly CatchM` cannot be passed directly; the bridge is `e => myCatch.Match(e) ? myCatch.Action(e).As() : IO.fail<C>(e)`. When the bracket catch needs the same recovery policy as a named catch used elsewhere, the recommended shape is a shared `static readonly Func<Error, IO<C>>` that the named catch's `Action` delegates to, keeping the predicate-selection logic in the `CatchM` and the recovery body in the shared function.

[POLICY_VALUE_THREADING_FAILURE_SURFACES]:
- `SmartEnum.ComparisonOperators` coerces `EqualityComparisonOperators` upward: if `ComparisonOperators` is set to `DefaultWithKeyTypeOverloads`, `EqualityComparisonOperators` is automatically promoted to the same level even if not explicitly set. The coercion is one-directional and happens at attribute read time (the property getter enforces it). Setting `EqualityComparisonOperators = None` while `ComparisonOperators = Default` silently has no effect — the getter returns `Default` because `ComparisonOperators >= EqualityComparisonOperators`. The safe pattern: set `ComparisonOperators` as the single authority; never split-set them independently.
- `SkipFactoryMethods = true` on a `ValueObjectAttribute<TKey>` cascades to: no `TypeConverter`, no `IObjectFactory<T, TValue, TValidationError>`, no `IObjectFactory<TValue>` marker, no key-to-value implicit conversion, `SkipIParsable = true` (which forces `SkipISpanParsable = true`), and all arithmetic operator generation forced to `None`. An `[ObjectFactoryAttribute<T>]` on the same type restores serialization and EF Core routes but does NOT restore the arithmetic operators or `IComparable` path — those are permanently suppressed by `SkipFactoryMethods`. Arithmetic operator generation requires the standard factory to be present.
- `SmartEnum.DisableSpanBasedJsonConversion = false` (default) causes the generated JSON converter to use a `ReadOnlySpan<char>`-keyed path via `IAlternateEqualityComparer` during deserialization, avoiding a string allocation per JSON token. Setting it `true` forces a string-based path. The `DisableSpanBasedJsonConversion` flag has no effect when `KeyMemberType` is not `string` — the span-based path is string-specific. When multiple smart enums share a JSON payload and throughput matters, keeping this flag at its default across all string-keyed enums is the zero-configuration allocation reduction.

[OPTION_SEMIGROUP_SEMANTICS]:
- `Option<A>` implements `Semigroup<Option<A>>` via `Combine`: returns `this` if `IsSome`, `rhs` otherwise. The `|` operator delegates to `Choose`, which calls the `Choice<Option>` trait implementation: returns `ma` if `IsSome`, `mb` otherwise. Both are semantically identical: left-biased fallback. Neither computes a union of the contained values — the inner `A` type plays no role. A chain `optA | optB | optC` returns the first `Some` without evaluating later operands. This is not the same as `option1.Combine(option2)` in the monoid sense; the monoid identity is `None`, and `Combine` is not an additive fold of the inner values.
- `Option<A>.operator true` returns `IsSome`; `operator false` returns `IsNone`. These enable `if (opt)` and short-circuit `&&`/`||` in expression trees. At the parameter-algebra layer, `operator true` and `operator false` are the domain-extension hook for guard expressions that must remain expression-form (e.g., inside generated `Switch` arms returning bool): `option1 && option2` returns the second `Option<A>` if the first is `Some`, otherwise `None` — using the standard C# short-circuit operator table applied to `operator true`/`false`.
