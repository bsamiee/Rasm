# Generic-Math Participation and Operator-Interface Synthesis

[DUAL_PATH_OPERATOR_DETECTION]:
- Operator capability on the key is computed once into a per-key flag record and drives every arithmetic and relational satellite; the record is filled by two independent passes that never re-probe. The first pass walks the key's implemented interfaces and, on finding a homogeneous generic-math operator interface, grants the entire operator family in one assignment — addition, subtraction, multiplication, and division each become the full `{ unchecked, checked }` flag pair, and comparison becomes all four relational flags, with no per-operator method inspection. The second pass walks the key's declared operator methods and, for every key that did not already satisfy the interface pass, sets each operator flag individually from the presence of the corresponding `op_*` method. The interface pass is all-or-nothing per family; the method pass is per-operator — the two together mean a key either lights up a whole family through a generic-math interface or accretes operators one method at a time.
- The interface pass requires exact homogeneity: the operator interface must be the framework numerics interface by name and namespace, generic with three type arguments, and the first two type arguments must both equal the key type. The result type argument is unconstrained, so a key whose binary operator yields a third type still triggers full-family generation, but a key implementing the mixed-operand form — left operand the key, right operand a foreign type — fails the equality on the second argument and is invisible to the interface pass entirely. Self-self operand symmetry is the admission gate, not result-type identity.
- The method pass classifies an operator method as arithmetic only when it is static, takes exactly two by-value parameters, and has return type and both parameter types all equal to the key type. A hand-written key operator that returns a delta type, or that mixes operand types, satisfies neither pass and produces no owner operator — the owner's arithmetic surface is strictly the homogeneous closed-operation subset of the key's operators, never the full operator algebra a numerically rich key might expose.
- Checked detection is a separate flag bit fed by a separate method name: the unchecked operator method sets the default bit, the checked operator method sets the checked bit, and a key carrying only the unchecked operator yields an owner with the unchecked operator and no checked variant. The flag pair is the precise reason a key can be partially checked-capable and the owner mirror that partiality rather than fabricating a checked operator the key never defined.

[INTERFACE_EMISSION_GATING]:
- Each arithmetic family declares its generic-math interface on the owner only when the family's default (unchecked) bit is set; the checked bit alone never emits an interface, because the checked operator lives inside the same interface as the unchecked one and the interface is meaningless without the unchecked member. The owner therefore advertises `IAdditionOperators<TSelf,TSelf,TSelf>` exactly when the key advertised the addition family, and a key with only a checked operator and no unchecked operator yields no addition interface at all.
- The comparison interface emits under a three-way disjunction that the arithmetic families lack: all four relational operators present on the key, or a configured comparer accessor, or a string key. The string and accessor arms manufacture the interface over a key that has no relational operators of its own, so the owner satisfies a relational-operator constraint that the key cannot — the owner is strictly more generic-math-capable than its underlying type along the ordering axis, and that lift is policy-driven, not key-derived.
- A partial relational-operator set on the key produces the matching partial operator subset on the owner and suppresses the comparison interface, because the interface declaration is gated on the full four-operator presence. The owner then supports the present operators at call sites yet fails a relational-operator interface constraint — a silent capability split where source-level operator use compiles but generic-constraint dispatch rejects the owner.
- The equality-operator interface is unconditional for the owner — it is always emitted with the self-self type arguments — because equality is generated from the owner's own equality, not from any key operator. Equality participation in generic math is a floor every owner reaches; arithmetic and ordering participation are earned from the key.

[KEY_OPERAND_INTERFACE_DOUBLING]:
- The key-type-overload mode emits a second generic-math interface per family alongside the self-self one: addition gains `IAdditionOperators<TSelf,TKey,TSelf>`, comparison gains `IComparisonOperators<TSelf,TKey,bool>`, equality gains `IEqualityOperators<TSelf,TKey,bool>`. The owner thereby participates in generic math against two distinct right-operand types — itself and its raw key — so a generic kernel constrained on the owner-against-key form admits the owner with an unadmitted key as the second operand, and the kernel composes admitted and raw operands without naming either concrete type.
- The mixed-operand arithmetic bodies route the result back through the factory exactly as the self-self bodies do, so an owner-plus-key addition re-enters admission and can reject; the mixed form is ergonomic raw-key arithmetic adjacent to the owner, but every mixed operation is still an admission site. The owner-against-key interface is a validating arrow disguised as an operator — its presence in a generic-math constraint silently routes the algorithm's intermediate results through the owner's invariant.
- The mixed-operand bodies null-guard the two operands asymmetrically by origin: the owner operand is guarded when the owner is a reference type, the key operand is guarded only when the key may be null. A class owner with a non-nullable struct key produces a mixed operator that throws on a null owner but never on the key, and the asymmetry is structural, computed independently per operand from the operand's own nullability.

[RESULT_TYPE_NARROWING_CAST]:
- Every arithmetic body wraps the raw key arithmetic in a cast back to the key type precisely when the key is one of the integral types that C# promotes to a wider type under binary arithmetic — the byte, signed-byte, short, unsigned-short, and char family — and omits the cast for every wider key. The promotion-then-narrow is the only reason `byte + byte` can be fed to a factory expecting a byte, and it is computed from the key's special type, not from any operator inspection.
- The narrowing cast sits inside the checked wrapper for the checked variant and outside it for the unchecked variant, so the two variants of one narrow-key operator carry different failure surfaces: the unchecked operator promotes to the wide type, computes, then narrows by silent truncation, then re-enters admission on the truncated value — an out-of-range sum wraps modularly and validates the wrapped result with no signal; the checked operator computes under overflow detection on the wide type, so the same sum throws an overflow before any narrowing. A narrow-key owner with a permissive invariant therefore admits silently wrapped arithmetic through its unchecked operator and rejects loudly through its checked one for identical operands.
- The cast is the key type, not the result type the key's operator declares, so a narrow key whose generic-math interface declares a wider result is still narrowed back to the key before the factory call — the owner's arithmetic is closed over the key type by force, and a key designed to widen under arithmetic has that widening erased at the owner boundary.

[ABSENT_IDENTITY_AND_NUMBER_INTERFACES]:
- No identity-element or aggregate numeric interface is ever synthesized — no additive identity, no multiplicative identity, no min-max interface, no full number interface — because an identity element is a constant that would materialize outside admission, an unvalidated owner value the generator refuses to mint. The owner reaches the binary-operator interfaces and stops; the algebraic structure it advertises is a magma under each operator, never a monoid, because the unit is unrepresentable without bypassing the factory.
- A generic kernel written against a full number constraint excludes every owner by construction, because that constraint demands the identity interfaces the owner cannot carry; relaxing the kernel to the exact binary-operator-family constraints is the owner-compatible spelling, and any seed, zero, or bound the kernel needs must enter as an admitted parameter rather than be summoned from a static identity property. The absence is the type-level enforcement that admitted values have no unadmitted siblings, even algebraic units.

```csharp
public static Fin<TOwner> Sum<TOwner>(TOwner seed, ReadOnlySpan<TOwner> values)
   where TOwner : IAdditionOperators<TOwner, TOwner, TOwner> =>
   Fin<TOwner>.Try(() =>
   {
      var acc = seed;
      foreach (var v in values) acc = acc + v;   // each += re-enters admission; seed is admitted, never a synthesized zero
      return acc;
   });
```

[OWNER_AGAINST_KEY_EQUALITY_ROUTING]:
- The key-overload equality operator does not compare the owner's egressed key to the raw key with default equality — it routes through the owner's configured equality comparer, so a string-keyed owner tests `owner == rawKey` under the same case policy that governs owner-to-owner equality, and a comparer-accessor owner applies the accessor's equality comparer. Boundary code can test an admitted owner against unadmitted raw material under the owner's identity semantics without constructing a second owner and without re-validating, and the comparison honors the owner's intrinsic policy rather than a naive key comparison.
- The raw-key operand is never validated by the equality overload, so an owner tested against a raw value the factory would reject returns inequality, indistinguishable from an admissible-but-different value. The key-overload equality is a policy-aware membership test against the owner's value space, not a proof the raw operand is itself admissible.
- The equality-comparison-operators setting clamps up to the comparison-operators setting in the generated output — requesting key-type overloads for ordering transparently grants them for equality — so the owner-against-key equality interface appears whenever the owner-against-key comparison interface is requested, keeping the two mixed-operand interfaces co-present by construction even where the source attribute spells only one.

[CONSTRAINT_DESIGN_PRESSURE]:
- An algorithm meant to admit value objects is constrained against the precise operator-family interfaces the owner can carry — the binary-operator families and the relational interface — never against the aggregate numeric interface and never against the identity interfaces, because those are exactly the surfaces admission withholds. The owner-compatible constraint set is the homogeneous binary-operation closure plus equality, and every algorithm seed or bound is an admitted parameter threaded into the kernel.
- A key chosen to carry a domain operator beyond the homogeneous closed set — a difference operator yielding a delta type, a mixed-operand scaling operator — gains nothing at the owner: the detection passes reject it, so that operator must live on the owner as a hand-written domain operation outside the generated surface, and it cannot satisfy any generic-math constraint. The generated operator surface is deliberately the closed-operation subset; richer key algebra is a domain method, not a synthesized operator.
- Routing arithmetic back through the factory makes every generated operator a validation site, so an owner with a range invariant must treat its own arithmetic as a policy decision: either suppress arithmetic generation and expose only domain-meaningful combinations, or accept that addition and subtraction can throw and place those operations on a result rail. Generated arithmetic on a constrained owner is never free; the closed-operation re-admission is the cost of carrying the invariant through every operator.
