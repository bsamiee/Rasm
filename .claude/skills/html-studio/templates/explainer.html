How the run gate admits work

Every run enters through one admission gate; this page walks the path, the policy that binds it, and the model that decides under load.

TL;DR

Every submitted run passes admit(), which resolves the caller to a policy row and charges one token from that row's bucket. Buckets refill continuously at the row's rate up to its burst ceiling, so short spikes ride the ceiling while sustained pressure converges on the rate. A run refused at the gate returns a typed refusal with the retry horizon — the ledger records admissions only.

Request path

Four steps carry every submission; each step names its exact source location.

1 Resolve the caller to a policy row gate/admission.py:21

The gate reduces the submission to a policy key — the declared lane when one is present, the caller identity otherwise. A submission with no resolvable key takes the default row, which carries the lowest rate in the table.

2 Look up the bucket gate/policy_table.py:9

Rows live in one policy table keyed by lane; each row owns a rate, a burst ceiling, and a refusal horizon. The table reloads on file change without a restart.

3 Refill and charge gate/policy_table.py:31

The bucket refills lazily: elapsed time times rate, clamped to burst, computed at read. Admission charges one token; the charge and the refill are one atomic step.

4 Refuse when empty gate/admission.py:48

An empty bucket returns the typed refusal carrying the row's retry horizon. The refusal is a value, never an exception — callers route on it.

Gotchas

Burst is not rate. A lane idle for one minute can fire its full burst instantly even when its rate is low; capacity planning reads both numbers.

Buckets are per process. A multi-worker host multiplies effective rate by worker count; the policy table states per-process numbers.

The default row is a trap for new lanes. A lane misspelled at the call site silently rides the lowest rate; the refusal's lane field is the diagnostic.

FAQ

How does a lane get a higher rate?

Add or edit its row in config/gate.policy.json; the table reloads live and the next admission reads the new numbers.

Where do refusals surface?

Every refusal increments the gate's pressure metric tagged by lane; the ledger stays admission-only by design.

Can a run bypass the gate?

The gate is the only door — the ledger refuses a run id it did not mint, so a bypass cannot record work.
