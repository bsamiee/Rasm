# Events and Threads — Frontier

[IO_FORK_THREAD_POOL_CORRECTION]:
- `IO<A>.Fork` in beta-77 documents "Queues the specified work to run on the thread-pool" — not `Task.Factory.StartNew(LongRunning)`. The `ForkIO<A>` record struct carries `Cancel: IO<Unit>` and `Await: IO<A>`; neither executes until bound. Parent-to-child resource sharing is by reference through the child `Resources.parent` pointer, which means the OS thread budget is the binding constraint, not managed pool caps. The `IO<A>.Timeout(TimeSpan)` overload wraps a fork-with-cancel race, not a `CancellationTokenSource.CancelAfter`; the timeout is compositional. `awaitAny` from `Prelude` cancels all other forks on the first success (verified in XML docs); `Task.WhenAny` does not — these are not interchangeable when fan-in cleanup matters.
- `awaitAll<A>(Seq<ForkIO<A>>)` collects all results or accumulates all errors; it does not short-circuit on first failure. The error accumulation means a seam that forks N boundary probes and uses `awaitAll` must treat the error sequence as a partial-availability report, not a single fault.

[TASK_COMPLETION_ORDER_AND_SPAN_OVERLOADS]:
- `Task.WhenEach<TResult>(ReadOnlySpan<Task<TResult>>)` yields tasks in completion order as `IAsyncEnumerable<Task<TResult>>` — verified: tasks submitted with delays 250–50 ms return in 5,4,3,2,1 order. This replaces manual `Task.WhenAny`-in-a-loop without tracking a remaining set. `WithCancellation(ct)` on the returned `IAsyncEnumerable` stops iteration; it does not cancel the underlying tasks. Faulted tasks are yielded normally — `completed.IsFaulted` must be checked per task; no exception propagates through the enumeration itself.
- `Task.WhenAll(ReadOnlySpan<Task>)` and `Task.WhenAll<TResult>(ReadOnlySpan<Task<TResult>>)` avoid the `Task[]` allocation for fixed-arity fan-out with `stackalloc`-composable spans. Same for `Task.WhenAny(ReadOnlySpan<Task>)`. `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` eliminates the array allocation for token-merging at seams that link two or more parent tokens.
- `TaskCompletionSource<T>.SetFromTask(Task<T>)` and `TrySetFromTask(Task<T>)` propagate the terminal state (success, fault, or cancellation) of the source task atomically to the TCS — including canceled and faulted states, verified. The prior pattern of manual `SetResult`/`SetException`/`SetCanceled` branching is replaced. `TrySetFromTask` returns `false` if the TCS is already transitioned, making it safe for concurrent completion races.

[CONFIGURE_AWAIT_OPTIONS_CONSTRAINT]:
- `ConfigureAwaitOptions.SuppressThrowing` is valid only on the non-generic `Task`, not `Task<T>`. Applying it to `Task<T>.ConfigureAwait(options)` throws `ArgumentOutOfRangeException` at runtime (not compile time); CA2261 is the diagnostic. The correct pattern: cast `Task<T>` to `Task`, await with `SuppressThrowing`, then inspect the original `Task<T>` handle for `IsFaulted`, `Exception.InnerException`, or `IsCanceled`. This asymmetry means `SuppressThrowing` is a non-result-carrying boundary guard, not a value-path operator.
- `ConfigureAwaitOptions` is a `[Flags]` enum with four values: `None = 0`, `ContinueOnCapturedContext = 1`, `SuppressThrowing = 2`, `ForceYielding = 4`. Composition is bitwise-or. `ForceYielding` forces a continuation post even on an already-completed task — verified: awaiting a pre-completed `Task.CompletedTask` with `ForceYielding` resumes on a different managed thread. This is the correct form for host-event dispatch callbacks that must not run on the event source's stack. The combination `ForceYielding | SuppressThrowing` (value 6) yields before running and does not propagate faults through the await site; the non-generic `Task` cast is still required for the `SuppressThrowing` component.

[CHANNEL_COMPLETION_AND_ERROR_PROPAGATION_PRECISION]:
- `ChannelWriter<T>.Complete(error)` does not surface the error immediately: `ChannelReader<T>.Completion` (the `Task`) only transitions to faulted after all buffered items are drained — verified: `Completion.IsCompleted` is `false` after `Complete()` while items remain in the buffer; it becomes `true` only after the last item has been consumed. The error surfaces from `ReadAsync`, `WaitToReadAsync`, and `ReadAllAsync` after the buffer is empty.
- The error from `Complete(error)` propagates as the original exception type directly — not wrapped in `ChannelClosedException`. `WaitToReadAsync` and `ReadAllAsync` throw `InvalidOperationException` (or whatever the caller passed) directly; verified across both paths. `ChannelClosedException` is thrown only on a normal `Complete()` (no error) when `ReadAsync` is called after the channel is empty and closed. Consumers must catch the writer's original exception type, not `ChannelClosedException`, when the writer signals structured failures via `Complete(error)`.
- The consequence for fault-projection split: a fold worker that calls `primaryWriter.Complete(error)` to terminate projection also delivers the error to all `ReadAllAsync` consumers as a thrown exception of the original type. Separate `ChannelWriter<Error>` is architecturally cleaner when fault consumers and projection consumers must react differently, but the native error is already reachable via the single-channel pattern when a single exception type is sufficient to terminate all consumers uniformly.
- `RendezvousChannel<T>` (the internal type from `Channel.CreateBounded<int>(0)`) does carry `CanCount = true` and `CanPeek = true` — confirmed: these flags do not indicate meaningful buffering, only that the API surface is available. The rendezvous invariant (each write blocks until a matching read is pending) means `Count` is always 0 or 1 in the handoff window, not a useful capacity indicator.

[ASYNC_LOCAL_PROPAGATION_SEMANTICS]:
- `AsyncLocal<T>` propagates down into child scopes (Tasks, threads started with `Thread.Start`) but modifications inside a child scope do not flow back to the parent — verified: `Task.Run(() => al.Value = "child")` leaves the parent's value unchanged after await. Each async scope has its own logical copy. `Thread.UnsafeStart` completely suppresses flow — the new thread starts with `AsyncLocal<T>.Value` returning `default(T)`, not the caller's value, confirmed experimentally.
- `AsyncLocal<T>(AsyncLocalValueChangedHandler<T>)` fires the callback on every value change, including the implicit fork/join transitions: entering a child scope fires with `previousValue = current, currentValue = default, threadContextChanged = true`; the child setting a value fires with `previousValue = default, currentValue = childValue, threadContextChanged = false`; child scope exit fires with the reverse. This gives a seam owner a hook for ambient-context-sensitive subscription management — the callback fires whether the change is explicit or implicit from a fork. The callback runs synchronously on the thread that caused the change.
- `ExecutionContext.SuppressFlow()` returns an `AsyncFlowControl` that must be disposed to restore flow on the current thread. The suppression is thread-local. The common mistake is calling `SuppressFlow()` without capturing the return value: without `Dispose()` the thread's flow remains suppressed for all subsequent work items queued from that thread. `ThreadPool.UnsafeQueueUserWorkItem<TState>` inherently skips EC capture regardless of the ambient flow state; it is allocation-free with a struct `TState` and eliminates both the delegate allocation and EC cost for context-free callbacks.
- `ExecutionContext.Run(ctx, callback, state)` is reusable on .NET 10 — a captured context can be passed to `Run` multiple times without `InvalidOperationException`. Earlier documentation described it as single-use; that constraint does not apply on .NET 10, verified experimentally.

[THREAD_UNSAFE_START_AND_BOUNDARY_SEAMS]:
- `Thread.UnsafeStart(object? parameter)` and `Thread.UnsafeStart()` ship alongside `Thread.Start` on .NET 10. `UnsafeStart` skips `ExecutionContext` capture entirely: the spawned thread starts with no `AsyncLocal<T>` values from the calling context, no ambient security or activity state, and incurs no allocation for EC copy. The correct domain for `UnsafeStart` is callback-only workers that write to channels, swap atoms, or set completion sources — operations that carry no ambient context semantics. Any worker that needs to propagate activity correlation, user identity context, or caller-injected `AsyncLocal` values must use `Start`.
- `ThreadPool.UnsafeRegisterWaitForSingleObject` skips EC capture for `WaitHandle`-backed callbacks. This is the allocation-free registration path for boundary seams that map native wait handles to managed notifications without inheriting ambient context. The symmetric registered `WaitHandle.Unregister(WaitHandle)` must be called on teardown; failing to unregister leaks the OS wait registration even after the calling object is collected.

[PERIODIC_TIMER_AND_RATE_CONTROL_PROTOCOL]:
- `PeriodicTimer(TimeSpan, TimeProvider)` accepts `TimeProvider` injection — the correct testing seam for schedule-driven fold workers. `WaitForNextTickAsync(CancellationToken)` returns `false` when the timer is disposed (not when the token is cancelled — cancellation throws `OperationCanceledException`). Disposing the timer from a parent scope while the worker is between ticks delivers `false` cleanly; if the parent disposes during the loop body, the current iteration completes and `false` surfaces on the next tick. `Period` is a mutable property; updates take effect on the next tick, not the current pending wait.
- The rate-limited consumer pattern: a `PeriodicTimer` drives the outer tick loop; inside each tick, `TryRead` drains all items that arrived in the interval. `WaitToReadAsync` gates on the next interval boundary when no items are ready. The fold worker owns the timer and disposes it on deactivation. The producer writes unconditionally; the channel's `BoundedChannelFullMode` governs what happens under back-pressure. This is the correct composition: producer policy is expressed in channel capacity + `FullMode`, consumer pacing is expressed in the timer period, and the two are independent.

[SUBSCRIPTION_SCOPING_WITH_IO_WITHENVFAIL]:
- `IO<A>.WithEnvFail(env => env.LocalResources)` releases acquired resources only on failure, not on success — the complement of `Bracket`. The seam pattern: an activation computation acquires subscriptions inside `WithEnvFail`; on success, the subscriptions escape the computation bound to an explicit `Atom<Option<Resources>>` ownership token; on failure, `WithEnvFail` cleans up the partial subscription set automatically. Explicit deactivation calls `atom.Swap(opt => opt.Map(r => { r.DisposeU(); return r; }))` — or more precisely, swaps `None` in and calls `Dispose` on the evicted `Resources` value.
- `IO<A>.Retry(Schedule)` releases resources on each failed attempt and retains them on success. A retry loop that opens subscriptions on each attempt never accumulates them across failed tries; each failed attempt's `Resources` scope is cleaned up before the next retry begins. This is the correct shape for a connection-and-subscribe pattern that must retry until the upstream is available: the subscription is re-established fresh on each attempt, and no partial state accumulates.
- `IO<A>.Uninterruptible` composed with a `Bracket` finalizer lambda is the canonical teardown form for subscriptions: cancellation of the parent token does not interrupt the finalizer chain. The finalizer runs the detach-then-dispose sequence to completion even if the cancellation token fires mid-teardown. `IO<A>.Finally<B>(IO<B>)` runs the `finally` effect after `this` regardless of success or failure; unlike `Bracket`, the `Finally` effect's result is discarded and the original result (or error) propagates. For teardown effects that have their own return value (e.g., a drain count), `Bracket` is the correct form; for side-effecting cleanup, `Finally` suffices.

[CONCURRENT_FANOUT_AND_AWAIT_ANY_CANCELLATION]:
- `Prelude.awaitAny<A>(ForkIO<A>[])` cancels all remaining forks when the first one succeeds — different from `Task.WhenAny`, which leaves all tasks running. The cancel effect on the losers is delivered via the fork's `CancellationTokenSource`; the losers' `Resources` scopes are cleaned up in their own `finally` blocks when cancelled. For fan-out patterns where the first successful result wins and all other probes must be stopped (e.g., competitive resolution among multiple boundary endpoints), `awaitAny` is the correct surface. Using `Task.WhenAny` at this seam leaks the loser tasks.
- `Prelude.awaitAll<A>(ForkIO<A>[])` collects every result; if any fork fails, errors are accumulated. This is the correct form when all forks model independent required boundary checks and the caller needs a complete result set or a complete error set — not a partial report.
- `Parallel.ForEachAsync<TSource>(IAsyncEnumerable<TSource>, ParallelOptions, Func<TSource, CancellationToken, ValueTask>)` consumes an async source with bounded parallelism. `ParallelOptions.MaxDegreeOfParallelism` caps concurrent invocations. The `CancellationToken` passed to each body is the conjunction of the `ParallelOptions.CancellationToken` and the method's internal completion gate; cancelling the options token terminates all pending bodies. This is the correct form for bounded-parallel event processing over a channel-backed `IAsyncEnumerable` source.

[CANCELLATION_REGISTRATION_DISPOSAL_HAZARD]:
- `CancellationTokenRegistration.Dispose()` blocks until any in-flight callback on another thread finishes. `DisposeAsync()` returns immediately after unregistering — it does not wait for the in-flight callback. The window created by `await using (var reg = ...)` combined with a callback that accesses a resource disposed in the `finally` is real: `DisposeAsync` returns before the callback finishes, the `finally` runs, and the callback then accesses a disposed resource. Synchronous `Dispose()` is safer when the next step after unregistration disposes resources the callback touches. `UnsafeRegister` with context-free callbacks (`ChannelWriter.TryWrite`, `Atom.Swap`) does not alter this disposal race — the synchronous/async distinction still applies regardless of which registration variant was used.
- `CancellationToken.Register(callback)` captures `ExecutionContext` at registration time and restores it when invoking the callback. `UnsafeRegister` skips this. The allocation difference is meaningful for high-frequency boundary seams (e.g., per-request timeout guards): `UnsafeRegister` is zero-allocation per registration for context-free callbacks. `IsCancellationRequested` is a cheap volatile read and the correct form inside a hot-path callback to exit early rather than inspecting state via closure.
