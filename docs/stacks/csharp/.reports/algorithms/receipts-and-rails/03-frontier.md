# Numeric Evidence Law: Receipts and Rails ‚Äî Frontier

[SVD_ORDERING_CONTRACT_INVERSION]:
- `Svd<T>.S` singular values are stored in descending order: `L2Norm = Math.Abs(S[0])` and `ConditionNumber = Math.Abs(S[0]) / Math.Abs(S[last])` are both correct only if `S[0]` is the maximum. A receipt that stores `S[0]` as the "minimum" or asserts `S[0] <= S[1]` is structurally wrong. The implementation is the authority; any conservation check on a receipt that treats the sorted order as ascending corrupts every downstream rank-threshold, condition-number, and pseudoinverse computation derived from the stored singular values.
- The rank threshold for the `double` concrete type is `S.Maximum().EpsilonOf() * Math.Max(U.RowCount, VT.RowCount)`. Because `S` is descending, `S.Maximum()` scans the full vector in O(n) rather than using `S[0]` directly. A receipt that pre-computes and stores the threshold as `S[0].EpsilonOf() * Math.Max(m, n)` is both correct and avoids the scan ‚Äî but it must name the constant as the algorithm-specific threshold, not as a generic `Tolerance` field, because the formula differs for the `Complex32` type: `tolerance = S.AbsoluteMaximum().Magnitude.EpsilonOf() * (float)Math.Max(m, n)` ‚Äî the `(float)` cast reduces the effective dimension precision and produces a slightly wider threshold than the `double` variant.

[ILUTP_PRECONDITIONER_EVIDENCE_SURFACE]:
- `ILUTPPreconditioner` exposes three independent numeric policy constants: `FillLevel` (default `200.0`, dimensionless multiplier of the input NNZ), `DropTolerance` (default `0.0001`), and `PivotTolerance` (default `0.0`, meaning no pivoting). These three scalars are independent: `FillLevel` controls the storage budget (`NNZ(L+U) <= FillLevel * NNZ(A)`), `DropTolerance` controls which fill-in entries below that threshold are zeroed, and `PivotTolerance` triggers column pivoting only when the diagonal entry is below `PivotTolerance * ‚Äñrow‚Äñ`. A receipt wrapping an ILUTP preconditioner must store all three as named fields; storing only the final `NNZ(L)` and `NNZ(U)` from `UpperTriangle().NonZerosCount` and `LowerTriangle().NonZerosCount` preserves the fill outcome but loses the policy, making re-admission impossible because the same fill outcome can arise from different (`FillLevel`, `DropTolerance`) combinations.
- `ILUTPPreconditioner.UpperTriangle()` and `.LowerTriangle()` expose the factored matrices post-`Initialize`. The actual fill-in ratio `(NNZ(L) + NNZ(U)) / NNZ(A)` is a post-factorization degeneracy counter: when it approaches `FillLevel`, the factorization saturated its budget and preconditioner quality is controlled by the drop tolerance, not the fill target. A receipt that records `ActualFillRatio` alongside `FillLevel` encodes the conservation check `ActualFillRatio <= FillLevel`.
- `ILUTPPreconditioner.Pivots()` returns the pivot permutation array after `Initialize`. When `PivotTolerance == 0`, no pivoting decisions are made. A receipt factory must distinguish between a zero-pivot-tolerance run (unconditional diagonal) and a positive-pivot-tolerance run (conditional pivoting applied) ‚Äî these are qualitatively different evidence states, not both "no pivoting needed."

[ITERATOR_FIRST_WIN_SEMANTICS]:
- `Iterator<T>.DetermineStatus` iterates the stop-criteria list in registration order and returns on the first non-`Continue` status found, not on the most severe. If `ResidualStopCriterion` is registered before `DivergenceStopCriterion`, a diverged solver that simultaneously satisfies the residual threshold is declared `Converged`, not `Diverged`. A receipt factory must capture the full stop-criterion list in the same insertion order ‚Äî not just the terminal `Iterator.Status` ‚Äî so re-admission can replay the priority logic and verify the status was not an order-dependent artifact.
- The `Iterator<T>()` zero-argument constructor populates itself from `Matrix<T>.Build.IterativeSolverStopCriteria(maxIterations: 1000)`. The default set is `ResidualStopCriterion` first, `DivergenceStopCriterion` second, `IterationCountStopCriterion` third. A solver that both converges and exceeds the iteration count on the same step returns `Converged` ‚Äî the iteration count criterion never fires if the residual criterion fires first. A receipt must record whether an explicit `Iterator<T>` was constructed with explicit stop criteria or whether the default constructor was used, because the two paths can produce materially different status outcomes under borderline conditions.
- `ResidualStopCriterion<T>` constructor signature: `(double maximum, int minimumIterationsBelowMaximum = 0)`. The default `minimumIterationsBelowMaximum = 0` means convergence is declared on the first iteration where the residual drops below the threshold, with no hysteresis. A receipt that records both the threshold and the hysteresis count is the only way to re-audit whether a premature convergence declaration was caused by a zero hysteresis policy.
- `DivergenceStopCriterion<T>` constructor: `(double maximumRelativeIncrease = 0.08, int minimumIterations = 10)`. The `minimumIterations` parameter has a hard lower bound of 3 (enforced by the constructor). A receipt must record `MinimumNumberOfIterations` because divergence at a 3-iteration window is a qualitatively weaker claim than at a 10-iteration window: a transient residual spike over 3 steps triggers divergence where a 10-step window would not.

[EXCEPTION_BOUNDARY_HIERARCHY]:
- The exception hierarchy has two distinct roots: `NonConvergenceException` (iterative algorithm failures) and `NativeInterfaceException` (BLAS/LAPACK provider failures). `NumericalBreakdownException : NonConvergenceException` ‚Äî catch it before `NonConvergenceException`. The native root branches into `InvalidParameterException : NativeInterfaceException` (parameter validation, carries an integer code), `MemoryAllocationException : NativeInterfaceException` (no additional fields), and `SingularUMatrixException : NativeInterfaceException` (carries the 1-based index of the zero diagonal entry in U).
- `SingularUMatrixException` carries the 1-based index of the zero diagonal entry in U ‚Äî the boundary-level evidence that identifies which equation is degenerate. At the boundary, `Error.New(SolverErrorCodes.SingularU, ex.Message, Error.New(ex))` preserves both the element index (in the message) and the exception identity. Reconstructing the element index from `ex.Message` string parsing is brittle; catch `SingularUMatrixException` specifically and extract `ex.Element` before constructing the error.
- The optimization exceptions form a separate hierarchy: `OptimizationException` (base, carries only message), `MaximumIterationsException : OptimizationException` (carries the iteration count in the message string), `EvaluationException : OptimizationException` (carries `IObjectiveFunctionEvaluation ObjectiveFunction` ‚Äî the last evaluation snapshot before failure), and `IncompatibleObjectiveException : OptimizationException` (structural incompatibility). `EvaluationException` is the only exception in the library that carries a typed evidence payload; its `ObjectiveFunction` provides the point, gradient, and Hessian at the failing evaluation. A receipt factory that catches `EvaluationException` without extracting `ObjectiveFunction` discards the only post-failure diagnostic evidence the optimizer provides.
- `LevenbergMarquardtMinimizer` defaults all three tolerances to `1e-15`. When a receipt stores these three tolerances as `double`s and they all equal `1e-15`, the receipt cannot distinguish a caller who set them intentionally from one who accepted the defaults. An honesty marker `bool UsesDefaultTolerances` ‚Äî set at construction when all three equal `1e-15` ‚Äî is the minimum distinction needed for re-admission to validate the tolerance regime.

[ATOM_VALIDATOR_RETURN_SEMANTICS]:
- `Atom<A>.New(A value, Func<A, bool> validator)` returns `Option<Atom<A>>`, not `Fin<Atom<A>>`. When the initial value fails the validator, the result is `None` with no error code, message, or failure reason. A domain layer that surfaces atom construction failures on the numeric rail must convert `None` to `Fin.Fail(Error.New(domainCode, "initial state failed invariant"))` at the construction site; the library provides no error information to preserve. The `Prelude.atom(value, validator)` helper returns the same `Option<Atom<A>>`.
- When `Atom<A>.Swap(f)` produces a value that fails the validator, the swap is silently abandoned and the atom retains the prior value. The return value is the prior value ‚Äî identical to a successful swap that happened to hold the prior value ‚Äî so there is no way to distinguish "validator rejected the swap" from "the function computed the current value" from the return value alone. `SwapMaybe(Func<A, Option<A>> f)` is cleaner: when `f` returns `None`, the swap is skipped without ambiguity. However, the validator can still silently reject the `Some` case. The only pattern that fully eliminates ambiguity is to use `SwapMaybe` where `f` checks the invariant itself and returns `None` on failure, making the library validator redundant.

[RUNNING_WEIGHTED_STATISTICS_WEIGHT_SUM_GAP]:
- `RunningWeightedStatistics` tracks `_w1` (sum of weights) and `_w2` (sum of squared weights) internally but exposes only `Count` (number of non-zero-weight pushes) and the computed moment statistics. The effective sample size under reliability weighting is `_w1^2 / _w2` and cannot be computed from the public API. A receipt that needs to report effective sample size for confidence interval derivation must accumulate `_w1^2 / _w2` in a domain-layer parallel accumulator alongside `RunningWeightedStatistics`, or replace it with a domain type that exposes these fields ‚Äî there is no re-entry point on the library type.
- The `Variance` property on `RunningWeightedStatistics` computes `_m2 / _w1` ‚Äî reliability-weighted population variance with no Bessel correction. A receipt that records this as "sample variance" and uses it in downstream interval construction without applying the Bessel correction over-constrains the confidence bounds for small effective sample sizes.

[TENSOR_PRIMITIVES_STDDEV_HONESTY]:
- `TensorPrimitives.StdDev<T>` computes population standard deviation ‚Äî `sqrt(sum((x_i - mean)^2) / N)` with the N normalizer, not the sample standard deviation with N-1. A receipt that stores a `double StdDev` computed via `TensorPrimitives.StdDev<double>` and annotates it as sample standard deviation has a structural honesty violation. The distinction matters for any downstream F-test, chi-square test, or sample-to-population inference that uses the stored value.
- `TensorPrimitives.Dot<T>` does not require `IRootFunctions<T>`, so it works on any algebraic ring including integer types ‚Äî unlike `Norm<T>` and `CosineSimilarity<T>`. A receipt validity predicate for "residual within tolerance" over a custom ring type uses `TensorPrimitives.Dot` to compute the squared norm without requiring the full `IRootFunctions` surface, then applies the tolerance comparison manually ‚Äî preserving exact arithmetic where `sqrt` would introduce rounding.
- `TensorPrimitives.HammingDistance<T>` is unconstrained ‚Äî it operates by equality comparison (not numeric comparison) and returns an `int` count of differing elements. Calling it on two `double[]` pivot or permutation arrays to verify bitwise identity produces the correct conservation check for permutation preservation without floating-point ambiguity.

[FOURIER_CONVENTION_AS_HONESTY_MARKER]:
- `FourierOptions.Default` is `0` ‚Äî symmetric `1/sqrt(N)` scaling on both forward and backward. `FourierOptions.Matlab` (`AsymmetricScaling`) applies no scaling on forward and `1/N` on backward. `FourierOptions.NumericalRecipes` (`InverseExponent | NoScaling`) applies a positive exponent on forward and no scaling on either direction. These three conventions produce outputs that are scale-different by exactly `sqrt(N)` or `N` relative to each other. A receipt that stores `FourierOptions` as an `int` bitfield preserves both orthogonal facts (exponent sign and scale); a receipt that stores a prose string destroys the bitwise re-admission test.
- `Fourier.ForwardReal` throws `NotSupportedException` when `FourierOptions.InverseExponent` is set; the packed real FFT surface supports only non-inverted exponent conventions. A receipt factory for real-signal forward transforms must gate on `(options & FourierOptions.InverseExponent) == 0` before invoking `ForwardReal` and return `Fin.Fail` with a typed domain error on violation. Failing to gate means the exception fires inside the library and must be caught at the external boundary, promoting a domain precondition failure to an `Exceptional` error.
- `FrequencyScale` places DC at index 0, positive frequencies at indices 1 through `length/2`, and wrapped negative frequencies at indices `length/2 + 1` through `length - 1`. For even-length N, the Nyquist frequency at index `N/2` is simultaneously the most positive and most negative frequency representable ‚Äî a shared edge bin. A receipt that stores the frequency axis without flagging the Nyquist-bin ambiguity for even-length transforms leaves downstream spectral analysis to silently misinterpret the edge bin's contribution.

[NONLINEAR_COVARIANCE_CONSTRUCTION_GAP]:
- `NonlinearMinimizationResult` computes `Covariance = Hessian.PseudoInverse() * Value / DegreeOfFreedom` in its constructor via `EvaluateCovariance`, using the SVD pseudoinverse. For near-rank-deficient Jacobians, the pseudoinverse is well-defined but produces a covariance matrix whose diagonal entries for degenerate directions are numerically large ‚Äî not zero. A receipt that gates on `Covariance != null` without recording whether any singular value fell below the pseudoinverse threshold emits statistically credible-looking but inflated standard errors for rank-deficient models. `NonlinearMinimizationResult` does not expose the Hessian SVD rank; a domain receipt wrapper that recomputes `Hessian.Svd(false).Rank` post-construction is the only path to the effective parameter count.
- The `EvaluateCovariance` path calls `objective.EvaluateAt(objective.Point)` unconditionally ‚Äî a second objective evaluation at construction time. If the objective function has side effects, this second evaluation fires them. A receipt wrapping `NonlinearMinimizationResult` must be constructed immediately after the minimizer returns and before any external state the objective reads can change; delayed construction does not re-trigger this evaluation but also cannot detect whether the second evaluation at construction time succeeded or mutated state.

[DESCRIPTIVE_STATISTICS_ACCURACY_FLAG_INVISIBILITY]:
- `DescriptiveStatistics` constructor takes `bool increasedAccuracy = false` which switches internal computation from `double` to `decimal`. After construction, the flag is not stored as a property ‚Äî there is no `bool IncreasedAccuracy` equivalent on the instance. The computed `Mean`, `Variance`, `Skewness`, and `Kurtosis` are stored as `double` regardless of which path was used. Two instances from the same data with `true` versus `false` produce `double` properties that differ by up to O(`DoublePrecision`) on well-conditioned data but can diverge visibly on large-magnitude data near `decimal.MaxValue`. A receipt wrapping `DescriptiveStatistics` must record the flag separately at construction as a named `bool UsesDecimalArithmetic` field ‚Äî the only moment the information exists.
- `DescriptiveStatistics.Variance` and `RunningStatistics.Variance` both use the N-1 Bessel correction (unbiased); `TensorPrimitives.StdDev<T>` uses N (population). These three produce `double` but are not interchangeable in a receipt. A receipt that accumulates statistics from multiple sources must carry a provenance field indicating the normalizer used ‚Äî at minimum a two-case enum `{Population, Sample}` ‚Äî or every downstream confidence computation is silently wrong.

[ERROR_COMBINE_OPERATOR_SEMANTICS]:
- `Error.Combine(Error other)` ‚Äî accessible as `error1 + error2` via `operator +` ‚Äî wraps both into a `ManyErrors` whose internal collection carries them in left-to-right order, preserving diagnostic order. Both `Error.Many(Error[])` and `Error.Many(Seq<Error>)` are available as public static factories for constructing a multi-error from a sequence; the fold pattern `errors.Fold(Error.Empty, (acc, e) => acc + e)` is an alternative but not the only path.
- `Error.Filter<E>()` on a `ManyErrors` returns only errors assignable to `E`, and `Errors.None` if none match. On a single `Error`, it returns `this` if assignable, otherwise `Errors.None`. A re-admission check that needs to inspect only algorithm-specific error cases uses `receivedError.Filter<ExpectedError>()` to isolate expected domain errors and test `.HasCode(SolverErrorCodes.SingularU)`, leaving exceptional errors in the remainder ‚Äî rather than pattern-matching the full `ManyErrors` graph.

[MANAGED_SVD_BLUESTEIN_OVERFLOW_BOUNDARY]:
- `ManagedFourierTransformProvider` uses a Bluestein FFT for non-power-of-two lengths. `BluesteinSequenceLengthThreshold` (documented as "greater than `Math.Sqrt(Int32.MaxValue) + 1`") is the threshold above which `k*k` overflows `int32`; its value is 46341. For `N > 46341`, the Bluestein sequence uses `long`-indexed accumulation internally. A receipt wrapping a non-power-of-two FFT must record `N > 46341` as a boolean `UsesLongBluesteinSequence` honesty marker: when `true`, the frequency output may have accumulated slightly different rounding than the short-path implementation, and any residual-norm check on a round-trip transform is sensitive to this path selection.
