# Request Variant Family Shape

[TWO_SYMBOLS_ONE_ARM]:
- A request arm carries two orthogonal traits on one object, keyed by two distinct symbols never one merged record: `symbolSerializable` holds `Schema<A, I, R>` for the *payload* and `symbolWithResult` holds `{ success: Schema<SA, SI, RR>; failure: Schema<FA, FI, RR> }` for the *outcome* — `SerializableWithResult` is the structural `extends Serializable, WithResult` intersection, so the payload codec and the result-pair codec are two independently-keyed slots, never one schema that has to re-derive which half a consumer wants.
- The up-cast pair projects each trait in isolation: `asSerializable(arm)` narrows to the bare payload `Serializable<Type, Encoded, Context>` and `asWithResult(arm)` narrows to the bare `WithResult<Success, SuccessEncoded, Failure, FailureEncoded, Context>` — a processor needing only the dispatch payload erases the result slot, a processor needing only the outcome codec erases the payload slot, and a full remote procedure keeps both through `asSerializableWithResult` — three trait views off one declaration, never three parallel surfaces.
- The two slots split the arm's `R` requirement into two distinct positions, never one merged channel: `TaggedRequestClass` threads `Struct.Context<Payload>` as the payload-decode requirement and `Schema.Context<Success> | Schema.Context<Failure>` as the *result*-decode requirement in separate proto type-arg slots — so an arm whose payload decodes context-free but whose `success` schema pulls a service requirement carries that requirement only on the outcome channel, and `serialize(arm)` is `Effect<I, ParseError, R_payload>` while `serializeExit(arm, ...)` is `Effect<..., ParseError, R_result>`, the two serializations resolving against different layers.

```typescript
import { Effect, ParseResult, Schema } from 'effect'

class Resolve extends Schema.TaggedRequest<Resolve>()('Resolve', {
    payload: { key: Schema.NonEmptyString },
    success: Schema.Struct({ at: Schema.DateTimeUtc }),
    failure: Schema.TaggedStruct('Absent', { key: Schema.NonEmptyString }),
}) {}
const onWire = (arm: Resolve): Effect.Effect<Schema.Serializable.Encoded<typeof arm>, ParseResult.ParseError> => Schema.serialize(arm)
const outcomeCodec = Schema.successSchema(Schema.asWithResult(Resolve.make({ key: 'x' })))
```

- `Schema.serialize(arm)` reads `symbolSerializable` and encodes the `_tag`+`key` payload alone, `Schema.successSchema(asWithResult(...))` reads `symbolWithResult` and recovers the `{ at }` success codec alone — the same arm yields the dispatch shape and the outcome shape through two symbol reads, the rejected spelling a parallel `PayloadSchema` beside a `ResultSchema` the next field edits twice.

[FOUR_CORNERS_DERIVE_FROM_ONE_DECLARATION]:
- One `class X extends Schema.TaggedRequest<X>()(tag, { payload, success, failure })` declaration fuses six surfaces the loose form spells separately: the tagged value-constructor, the payload codec, structural `Equal`/`Hash`, a runnable `Request.Request<SuccessType, FailureType>`, the `SerializableWithResult` serialization owner, and a `Schema.Union`-joinable nominal class — `class X extends ... {}` is the whole `object → type → const → typeof → instantiation` chain collapsed, no `type XPayload`, no `const xSuccessSchema`, no hand-written `Request.of` constructor.
- The four contract corners recover by trait extractor, never by a mirror type: payload-decoded is `Serializable.Type<typeof X>`, payload-wire `Serializable.Encoded<typeof X>`, outcome-success `WithResult.Success<typeof X>`, outcome-failure `WithResult.Failure<typeof X>` — four `infer`-projections over two traits, plus the encoded duals `WithResult.SuccessEncoded`/`FailureEncoded`, so a function typed "what running this arm yields" is `WithResult.Success<typeof X>` and never `Schema.Type<typeof X.success>` reaching through the live member nor a re-listed `{ at: DateTime.Utc }`.
- The constructor elides two phantom slots by two mechanisms that compose: `Struct.Constructor<Omit<Payload, "_tag">>` strips `_tag` (the `tag<Tag>` default supplies it) and `Request.Request.Constructor` omits `keyof Request.Request.Variance` (the covariant phantom type-id) — so `X.make({ key })` passes neither the discriminant nor the variance witness, two elisions on one call the hand-rolled `Request.of<X>()({ _tag: 'X', key })` would have to spell.
- `Self` is mandatory: omitting it resolves to `MissingSelfGeneric<"TaggedRequest", '"Tag", SuccessSchema, FailureSchema, '>` rather than a usable class — the same self-type gate `TaggedClass` and `TaggedError` carry, so a request arm is a first-class nominal class joinable to a `Schema.Union` exactly like a plain tagged arm, the result-pair carriage invisible to the decode search tree.

[ANNOTATION_AXIS_IS_PAYLOAD_SCOPED]:
- The `annotations` argument of `TaggedRequest` is typed `ClassAnnotations<Self, Simplify<Struct.Type<{ _tag: tag<Tag> } & Payload>>>` — its message-scope sees the *payload* type only, never `success`/`failure` — so an `identifier`, `description`, or custom `message` declared on the arm names the request value as it decodes, and the outcome schemas carry their own annotations independently on their own `Schema.Struct`/`TaggedStruct` declarations.
- The split is load-bearing for diagnostics and `$defs`: a `JSONSchema.make` over a request payload emits the payload contract under the arm's `identifier`, while the outcome contract is a separate emission off `exitSchema(arm)` or `successSchema(arm)` — the request value and its result are two annotatable, two emittable surfaces, so a wire contract for "what to send" and a wire contract for "what comes back" are derived from one arm yet annotated and projected at two scopes, never one merged annotation that conflates request and response.
- The annotation cannot reach the result pair to constrain it, so a per-outcome refinement (a `success` whose value carries `Schema.between`) rides the *outcome schema's* own annotation, recovered through `successSchema(arm)` into `Arbitrary.make`/`JSONSchema.make` — the same owner-resident-annotation discipline a plain field carries, applied to the result channel the payload annotation cannot see.

[FAILURE_SCHEMA_IS_NOT_THE_CAUSE_TREE]:
- The `failure` schema names only the *expected* `Fail.error` arm of a wider cause variant the exit codec actually serializes: `ExitEncoded`'s `Failure` branch carries `CauseEncoded<E, D>`, a six-arm recursive tagged union — `Empty`, `Fail` (the `failure`-typed `error`), `Die` (the defect), `Interrupt` (a `FiberIdEncoded`), and `Sequential`/`Parallel` (the recursive composition) — so an arm declaring `failure: SomeError` projects onto the `Fail` arm alone, while the wire outcome can decode any cause node, including composites a concurrent run accumulates.
- The carriage is what makes the result channel survive concurrency without a hand-flattened error: an arm run as several racing sub-effects whose faults combine serializes the *whole* `Sequential`/`Parallel` cause tree through `serializeExit`, and `deserializeExit` reconstructs the live `Exit` with that tree intact — the failure schema sized for one error never has to model the join, the cause variant the codec already owns carries it, so a parallel-fault outcome round-trips structurally where a `failure`-only serializer would collapse it to whichever error won.
- The half-channel codec sees a different shape than the full exit by exactly this gap: `serializeFailure`/`deserializeFailure` read the `failure` schema in isolation (the bare `Fail.error` shape, no cause wrapper), whereas `serializeExit` wraps that error in the `Fail` cause arm and admits the `Die`/`Interrupt`/composite arms beside it — so a fault-only notification crosses as the flat error and a full outcome crosses as the discriminated cause tree, two projections off one arm selected by whether the defect and interruption axes must travel, not a boolean knob.

[DEFECT_ERASES_TO_UNKNOWN_ON_THE_WIRE]:
- The third `ExitEncoded` slot `D` is pinned to `unknown` by every result accessor: `exitSchema` returns `ExitEncoded<SI, FI, unknown>`, `serializeExit` produces `ExitEncoded<SI, FI, unknown>` — the arm cannot type its own defect schema, because `exitSchema` is built internally as `Exit({ success, failure, defect: Defect })` with the *built-in* `Schema.Defect` codec hardcoded into the `defect` position, never the arm's `failure`.
- `Schema.Defect` is an `unknown → unknown` transform that decodes structured-error-shaped objects (`message`, optional `name`/`stack`) into `Error` instances and stringifies everything else, encoding `Error` back to a plain `{ name, message }` — so a domain defect carried in the `Die` cause arm round-trips as a degraded plain object, never as its original class, and a `Die`-thrown value the interior knew precisely re-enters the consumer as an `unknown` the `Defect` codec reconstructed.
- The boundary this draws: the *expected* failure (the `failure` schema, the `Fail.error` arm) round-trips with full type fidelity, but a *defect* (a `Die`, an unrecovered throw) crosses the wire only at `Defect`'s lossy fidelity — so a fault that must survive serialization typed belongs in the `failure` schema as a `TaggedError` arm, never left to surface as a defect the exit codec erases to `unknown`; modeling the fault on the failure axis is the spelling that keeps it on the wire, leaving it as a thrown defect is the spelling that degrades it.

[EXIT_CODEC_IS_A_PER_PROTO_SINGLETON]:
- `exitSchema(arm)` is memoized in a process-global `WeakMap` keyed on the arm class's `proto`: the first call builds `Exit({ success, failure, defect: Defect })` and stores it against the prototype, every subsequent call returns the same schema instance — so the outcome codec is constructed once per arm *class*, not once per request value nor once per serialization, and `serializeExit`/`deserializeExit` (each `dual(2, ...)` over `encode`/`decodeUnknown` of the cached schema) pay the codec-build cost a single time across the whole process for a given arm.
- The cache key is the prototype, so two distinct request values of the same arm share one exit codec and a structurally-identical-but-nominally-distinct arm gets its own slot — the memoization is per declared class, the unit the declaration-chain collapse already made the single source, so the exit codec inherits the arm's identity discipline rather than allocating per-call as a hand-rolled `Schema.Exit({...})` beside each handler would.
- The consequence for a hot request family: a resolver draining many values of one arm builds the exit codec on the first drain and reuses it for every element, so the per-element cost is the `encode`/`decode` walk alone, never a re-`Exit({...})` per outcome — the rejected spelling is constructing `Schema.Exit({ success: arm.success, failure: arm.failure, defect: Schema.Defect })` inline at the handler, which re-allocates the transform every call and duplicates the exact codec the WeakMap already holds.

[HETEROGENEOUS_BATCH_UPCASTS_TO_THE_TRAIT]:
- A processor over a mixed-arm batch dispatches on the `WithResult` trait, never a per-arm branch: `asWithResult` accepts any `WithResult.All` and the bound `All = Any | WithResult<any, any, never, never, unknown>` admits *both* a fallible arm (`Any`, failure `any`) and an infallible arm (the `never, never` failure branch) — so a queue draining `TaggedRequest.All` values feeds each element to `successSchema`/`failureSchema`/`exitSchema` and each element's own outcome schemas thread through, the batch loop holding one trait view across arms whose result pairs differ.
- `TaggedRequest.All` is the closed admission type the batch keys on: `Any | TaggedRequest<string, any, any, any, any, any, never, never, unknown>` — the trailing `never, never` again the infallible arm — so a function over the whole family types its parameter `TaggedRequest.All` and recovers per element by trait extractor, the `_tag` available for dispatch and the result pair available for outcome serialization, no `switch (req._tag)` enumerating arms to find each one's schemas.
- The infallible arm is encoded structurally, not by a nullable error: `failure: Schema.Never` collapses the `WithResult` failure slot to `never`, so `failureSchema(arm)` is `Schema.Never`, the runnable `Request`'s `E` channel is `never`, and the arm's `ExitEncoded` failure branch is uninhabited — the absence of a fault is a type-level fact the family carries, and the heterogeneous batch's trait up-cast tolerates it through the same `never, never` bound that lets a fallible sibling carry a real failure, one loop over both with no per-arm fallibility branch.

[COMPLETEEFFECT_BRIDGES_SCHEMA_PAIR_TO_RUNNABLE]:
- The handler's `Effect` is checked against the arm's *runnable* `Request.Request<SuccessType, FailureType>` projection, not against the schemas directly: `Request.completeEffect(arm, effect)` demands `effect: Effect<Request.Request.Success<arm>, Request.Request.Error<arm>, R>` and yields `Effect<void, never, R>`, where `Request.Request.Success`/`Request.Request.Error` are the `Type`-side projections of the arm's `success`/`failure` schemas — so the schema-declared result pair becomes the handler's typed obligation through the `Request.Request` interface the `TaggedRequestClass` already extends, the two contracts (schema codec, runnable channel) fused on one class.
- `Request.Request.Result<arm>` is `Exit<Success, Error>` and `Request.Request.OptionalResult<arm>` is `Exit<Option<Success>, Error>` — the same `Exit` shape `exitSchema` serializes — so the *runnable* outcome type and the *serializable* outcome type are one `Exit<SA, FA>` the trait pair derives from the one declaration: the handler completes the request with an `Exit`, the codec serializes that `Exit`, and a remote arm's local execution and its wire round-trip key on the identical outcome shape.
- The two projections must agree at the seam: a handler returning an `Effect<A, E>` whose `A`/`E` diverge from the arm's `success`/`failure` `Type` fails `completeEffect` at that arm, the same single-arm locality a `Match.tag` over the request family gives — so a dispatch surface pairing `Match.tagsExhaustive` (the family decode/dispatch) with `Request.completeEffect` (the per-arm runnable completion) checks both the discriminant coverage and the per-arm outcome type, a fourth request landing as one `TaggedRequest` class plus one matched-arm handler the checker demands.

```typescript
import { Effect, ParseResult, Request, Schema } from 'effect'

class Lookup extends Schema.TaggedRequest<Lookup>()('Lookup', {
    payload: { id: Schema.NonEmptyString },
    success: Schema.Int,
    failure: Schema.TaggedStruct('NotFound', { id: Schema.NonEmptyString }),
}) {}
const run = (arm: Lookup): Effect.Effect<void> => Request.completeEffect(arm, Effect.succeed(arm.id.length))
const decodeOutcome: (wire: unknown) => Effect.Effect<Request.Request.Result<Lookup>, ParseResult.ParseError> =
    Schema.decodeUnknown(Schema.exitSchema(Schema.asWithResult(Lookup.make({ id: 'k' }))))
```

- `completeEffect(arm, Effect.succeed(arm.id.length))` checks `Effect<number, never>` against the arm's `Request<number, NotFound>` projection — `number` matches `success: Schema.Int`, the absent error is admitted by covariance — and `exitSchema(asWithResult(...))` recovers the same arm's `Exit<number, NotFound>` codec, so the runnable channel and the wire channel are one `Exit` shape; the rejected spelling is a hand-written `Schema.Exit({ success: Schema.Int, failure: NotFound, defect: Schema.Defect })` beside a separate `Effect<number, NotFound>` handler signature, two contracts the next outcome edit desynchronizes.

[UNION_FAILURE_THREADS_BOTH_CHANNELS_AT_ONCE]:
- A `Schema.Union` of `TaggedError` arms in the `failure` slot widens *two* derived channels from one declaration with no second edit: the runnable `Request.Request<Success, FailureType>`'s `E` becomes the error union (so `Match.tags` over a caught fault narrows to the exact arm and `completeEffect` checks the handler's `Effect` error against `Request.Request.Error<arm> = the union`), and the wire `failureSchema(arm)` becomes that union's codec decoding by the same search tree — appending a fault arm grows the typed `E` channel and the wire fault contract together, the discriminant spanning the request family and its fault family as one vocabulary.
- The fault union's requirement rides the *result* channel alone, never the payload channel: the arm's `R` splits as `Struct.Context<Payload>` for payload-decode and `Schema.Context<Success> | Schema.Context<Failure>` for result-decode, so a `failure` union whose arms pull a service requirement (a context-dependent `TaggedError` codec) places that requirement only on `failureSchema`/`exitSchema`/`deserializeExit` — `serialize`/`deserialize` of the payload stay context-free, and the layer providing the fault-decode requirement is needed at the outcome seam, never at the dispatch seam.
- The split is what lets the dispatch boundary and the outcome boundary resolve against different layers: a remote caller decoding the request payload provides only `R_payload`, and the same caller decoding the returned `Exit` (whose `Fail` arm holds the requirement-bearing fault union) provides `R_result` — so a request family whose faults need a service to decode does not drag that service into the request-admission layer, the two `R` positions on the one `TaggedRequestClass` proto keeping the requirement at the channel that actually consumes it.
