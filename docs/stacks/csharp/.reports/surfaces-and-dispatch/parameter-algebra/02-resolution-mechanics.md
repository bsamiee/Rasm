# Resolution Mechanics of Parameter Surfaces

[PRIORITY_PRUNING_TOPOLOGY]:
- `[OverloadResolutionPriority(n)]` is not a betterness rule; it is a pruning pass that runs after applicability and before betterness, and it groups candidates by declaring type first. Within each declaring-type group, every candidate whose priority is below the group maximum is deleted; the surviving groups recombine, then ordinary betterness runs on the union. The integer is a signed 32-bit value, so a negative priority demotes a candidate below all unannotated peers in its own type without touching any other type's group.
- Because grouping is per declaring type, a base type can never raise a member above a derived type's member: derived-type member lookup already prunes the base group when an applicable derived candidate exists, so the base priority is read inside a group that has already been discarded. The consequence for a deep entrypoint surface is that priority steering only resolves ambiguity among siblings declared together; splitting a would-be-ambiguous pair across a base/derived boundary makes the attribute inert.
- Priority is read from the least-derived declaration of a virtual member; applying the attribute on an `override` is a compile error, not a silent no-op, and the attribute is never inherited across an interface implementation — an implicit implementation carries priority 0 regardless of the interface member's priority, with no warning on the mismatch.
- Extension members group by their containing static class, so two extension overloads in one static class are priority-ordered against each other, but an extension in a higher-priority static class does not outrank one in another class — the grouping is the static class, never the receiver. A single extension block that owns the full receiver surface keeps every priority decision inside one group; scattering the overloads across blocks fragments the priority lattice.
- Priority can make a default-priority overload uncallable from source entirely: with `M(int)` at 0 and `M(long)` at 1, `M(someInt)` binds the `long` overload despite the identity conversion. The vestigial overload is reachable only through a delegate conversion, `UnsafeAccessor`, or reflection — and the delegate escape fails for a contravariant reference pair such as a prioritized `M(object)` shadowing `M(string)`. A prioritized overload is therefore a hard redirection, not a hint, and the demoted body should be deleted rather than left as dead surface.

[PARAMS_BETTERNESS_LATTICE]:
- The element type dominates every other betterness axis for `params` collections. Two `params` overloads differing only in collection wrapper resolve by the chain `ReadOnlySpan<E>` beats `Span<E>` beats array-or-array-interface `E[]`/`IEnumerable<E>`, but only when an identity conversion exists between the element types; when the element types differ, betterness is decided on the element conversion first and the collection wrapper never enters the comparison. A `params ReadOnlySpan<E>` overload therefore makes a coexisting `params E[]` overload pure dead surface for both expanded calls and collection-expression call sites — no `[OverloadResolutionPriority]` is required to win, so annotating the array form to suppress an ambiguity is the wrong fix; removing it is.
- Normal form beats expanded form unconditionally: a single argument already convertible to the whole collection type binds the value parameter, never the per-element expansion, and the expanded form is only constructed when the normal form is inapplicable. This is the self-ambiguity at the heart of a one-element call against `params ReadOnlySpan<T>` where `T` is itself that span — the resolution prefers passing the lone argument as the collection, and forcing element-wise admission requires a collection-expression bracket at the call site.
- Two `params` overloads whose element types are non-identical but inter-convertible are ambiguous on an empty argument list and on an empty collection expression, because no argument exists to drive the element-conversion comparison and the collection-wrapper tie-break is gated behind "the same set of arguments corresponds to the collection elements for both methods." Exposing `params ReadOnlySpan<int>` beside `params ReadOnlySpan<int?>` on one entrypoint is a latent zero-arity ambiguity; a single arity-collapse parameter with one element type is the only form that resolves cleanly at the empty boundary.
- The collection-wrapper tie-break is positioned last in the betterness order specifically so it cannot fire across a swapped parameter layout: `Test(int x, params ReadOnlySpan<int> y)` versus `Test(int y, params Span<int> x)` stays ambiguous because the "same set of arguments corresponds to the collection elements" guard fails when the spans are populated from different positions. Arity collapse must keep the fixed parameters positionally identical across any overload pair it leaves standing, or the span-over-array preference never engages.
- A non-array `params` collection has no expanded form under the dynamic binder and none in an expression tree; the expanded form is also disqualified when the single surviving candidate is a local function. An entrypoint that must remain callable through `dynamic` or lambda-to-expression-tree compilation cannot rely on non-array `params` arity collapse and must accept the collection value directly.

[OPTIONAL_DEFAULT_TIE_BREAK]:
- Among equally-good candidates, the one whose every parameter received a corresponding argument beats the one that had to substitute a default for an optional parameter; "fewer parameters with omitted optionals" is a real tie-breaker, not folklore. A growing optional tail therefore actively loses resolution against a tighter sibling — adding an optional context parameter to one overload can silently redirect existing call sites onto a different overload whenever both remain applicable.
- This tie-break is the resolution-level argument against the nullable flag tail: the flag overload is structurally penalized against the bundled-record overload exactly when both are applicable, producing call-site drift that no diagnostic surfaces. Collapsing the tail into one `Option<ContextRecord>` parameter removes the parallel candidate, so there is no second member for the omitted-default penalty to choose between.
- A `params` collection may legally follow optional parameters in the signature but can never carry a default value of its own — omission yields a fresh empty collection. The interaction is that a trailing `params` does not count as an omitted optional for the tie-break above, so an arity-collapse tail and an optional-context parameter ahead of it compose without the empty-`params` site being penalized as a missing default.

[NAMED_ARGUMENT_POSITIONAL_LAW]:
- A named argument used out of declaration order is legal only when no positional argument follows it; a positional argument after an out-of-order name is CS1738. A named argument may, however, sit in its own correct positional slot while later positionals continue — the rule is positional integrity, not name-must-be-trailing. This bounds the readability use of names on a wide entrypoint: names can clarify a leading discriminant only if every following argument is also named or already in position.
- Names cannot reconstruct an omitted-then-supplied optional across a gap by position; the gap is closeable only by naming the later optional. An optional-context parameter that callers will frequently skip-then-target must be reachable by name, which makes its parameter identifier load-bearing API surface — renaming it through a generated `SwitchMapStateParameterName`-style override is a breaking change to every named call site, not a cosmetic relabel.
- Caller-information optionals — including the source-text capture attribute — are optional parameters the caller must never pass explicitly; the compiler fills them. A prioritized overload pair built to inject a captured-expression default works precisely because the captured-default overload is selected by priority while its filled parameter stays invisible to the call site. Supplying that parameter by name defeats the capture and is the misuse the attribute design assumes will not occur.

[GENERATED_DISPATCH_SURFACE_RESOLUTION]:
- The exhaustive total `Switch`/`Map` and the partial `SwitchPartially`/`MapPartially` overloads coexist on one generated type without ambiguity because they differ in arity and in the optionality shape of their per-case slots: the total form requires every case handler positionally, the partial form types each case slot as an optional defaulted parameter plus a required-or-defaulted fallback. A call supplying every case binds the total form; a call omitting cases binds the partial form by the omitted-default penalty running in reverse — the total form is inapplicable once a case argument is missing, so no tie-break is even reached.
- The action form and the func form of a generated partial dispatch are distinguished only by return type and by whether the fallback slot is required, which return type alone does not disambiguate; resolution separates them by the fallback parameter's optionality. The action `SwitchPartially` makes the fallback optional, the func `MapPartially`/`SwitchPartially` makes it required, so a call with no fallback can only bind the action form and a call inside an expression context binds the func form by target typing — the parameter algebra, not the body, carries the disambiguation.
- An additive stop-at `Switch` overload generated for a named sub-hierarchy is applicable simultaneously with the exhaustive overload, and the two are not ambiguous only because their arities differ: the stop-at overload collapses an entire leaf set into one handler parameter, lowering the parameter count below the exhaustive form. A dispatch site that lists every leaf binds exhaustive; one that lists the sub-hierarchy once binds stop-at — the choice is recovered from argument arity, never from a mode flag.

[SHORT_CIRCUIT_OPERATOR_RESOLUTION]:
- The absence-carrying option type compiles `||` because it supplies `operator true` (returning the present predicate) and `operator |` (left-biased first-present choice); it deliberately omits `operator &`, so `&&` does not compile and an all-present conjunction cannot be smuggled in as a short-circuit operator — the resolver rejects it at the operator-lookup stage. Guard composition that needs lazy absence-testing is forced onto `||`; everything else routes through bind or property-pattern destructuring, and this is an operator-resolution constraint, not a style choice.
- The eager `|` choice evaluates both operands before the left-biased pick, while the genuinely lazy alternative is a distinct second `Choose` overload taking a deferred, compute-at-most-once right branch resolved by the right operand's static type. Selecting short-circuit semantics is therefore an overload selection driven by the shape of the right-hand argument — a deferred carrier binds the lazy overload, a materialized value binds the eager one — so the call site's expression type, not a flag, decides whether the expensive alternative is skipped.

[REF_STRUCT_GENERIC_ADMISSION]:
- A `readonly ref struct` argument carrier constrained `where T : allows ref struct` is admissible as a generic type argument only to a type parameter that itself declares `allows ref struct`; it cannot be boxed, stored in a field, or held across an `await`. The set-vs-omitted optional-argument carrier is exactly this shape, so its use is confined to a single synchronous call frame and it cannot participate in any deferred or asynchronous dispatch surface — a resolution-time constraint that bounds where the "supplied this exact value versus omitted" distinction can be threaded.
- A factory seam typed on a ref-struct source — a character span admission path — binds a zero-allocation overload directly from a span literal without an intermediate heap allocation, and the same span seam is selected by the serialization reader path. The reverse projection back to a span is an explicit method call, never the implicit conversion operator, because the implicit operator is generated against the value type and not the ref-struct interface — so the allocation-free egress is reachable only by naming the conversion, and resolution will silently pick the allocating implicit path if the explicit call is omitted.
