# Parameter Algebra — Edge

[RETRY_PREDICATE_POLICY_FORMS]:
- `retryWhile(Schedule, K<M, A>, Func<Error, bool>)` and `retryUntil(Schedule, K<M, A>, Func<Error, bool>)` add a predicate gate to the schedule-driven retry loop. The schedule governs spacing and count; the predicate governs the error domain that admits retries. Both are named fields on a policy record, not flags passed at call sites. Storing `static readonly Func<Error, bool> IsTransient = e => e.IsExpected && !e.IsType<TerminalError>()` alongside the schedule makes the full retry policy one cohesive value.
- `repeatWhile` and `repeatUntil` accept the shape `(Schedule, K<M, A>, Func<A, bool>)`; the predicate receives the bound value, not an error. Using `repeatWhile` when the intent is to abort on certain errors keeps repeating until the schedule expires, because a successful value always satisfies `Func<A, bool>`. The correct split: schedule-driven retry for error recovery; schedule-driven repeat with value predicate for polling — never conflated on one call site.

[EXPECTED_EXCEPTIONAL_CATCH_VOCABULARY]:
- `exceptional<M, A>(Func<Exceptional, K<M, A>>)` and `expectedOf<E, M, A>(Func<E, K<M, A>>)` are the two orthogonal catch axes on the `Error` type hierarchy. `Expected` errors (created via `Error.New(string)` or `Error.New(int, string)`) carry `IsExpected = true`; `Exceptional` errors (created via `Error.New(Exception)`) carry `IsExceptional = true`. A policy that handles domain errors and a policy that handles infrastructure faults are two separate `static readonly CatchM` values chained with `|`; merging them into one predicate-catch erases the type distinction.
- `catchOfFold<E, M, A>` and `exceptionalOfFold<E, M, A>` are the monoid-fold variants of `catchOf` and `exceptionalOf`. Where `catchOf` uses `ForAllM` (fail-fast on first handler failure), the `Fold` variants use `FoldM`, which continues processing sub-errors even after individual handler failures. Choosing between `catchOf` and `catchOfFold` is a policy decision about aggregate error semantics: fail-fast vs. collect-all.
- `Error.Filter<E>()` returns an `Error` containing only sub-errors of type `E`; when no sub-errors match, it returns `Errors.None`. `Errors.None` is a `ManyErrors` with an empty `Seq<Error>`. Using `error.Filter<MyError>().IsEmpty` to gate a handler is incorrect because `ManyErrors` has no `IsEmpty`; the correct guard is `error.IsType<MyError>()` before calling `Filter<MyError>()` to collect the matching sub-errors.

[SCHEDULE_TIME_INJECTION_POLICY]:
- `Schedule.upto`, `fixedInterval`, `windowed`, `secondOfMinute`, `minuteOfHour`, `hourOfDay`, and `dayOfWeek` all accept a `Func<DateTime> currentTimeFn` parameter. The default form uses `DateTime.UtcNow`; the injected form accepts any clock function. The policy value carries its own time source rather than reading a global — `static readonly Schedule CronLike = Schedule.minuteOfHour(0, () => _clock.UtcNow)` binds the clock at construction, not at each retry step.
- `Schedule.fixedInterval` suppresses delay (emits zero) when the action overruns the interval, but the next window starts from the completion time of the previous action. `Schedule.windowed` always emits the residual time to the next absolute boundary regardless of action duration. `fixedInterval` is the load-leveling form; `windowed` is the cron-alignment form. Using `windowed` when the intent is load-leveling causes delayed actions to catch up immediately but then run at fixed boundary times, producing burst behavior at boundary crossings.
- `Schedule.repeat(int n)` repeats the entire schedule `n` times — it is a schedule multiplier, not an iteration count. `Schedule.exponential(100*ms) + Schedule.repeat(2)` runs the full exponential stream twice end-to-end, doubling the total iteration count. `recurs` is the iteration-count form; a stored policy should use `recurs` exclusively when the intent is a retry count.

[VALUE_OBJECT_UNSAFE_CONVERSION_AND_NULL_ADMISSION]:
- `ValueObjectAttribute<TKey>.UnsafeConversionToKeyMemberType` (default `Explicit`) generates an `explicit operator TKey(TObj obj)` that can throw `InvalidOperationException` when the value object wraps an unset state (`AllowDefaultStructs = true`). Setting it to `None` removes this operator; callers use `ToValue()` or `.Key` explicitly. The safe implicit operator (`ConversionToKeyMemberType = Implicit`) never throws and is the only conversion generated when `AllowDefaultStructs = false`.
- `NullInFactoryMethodsYieldsNull = true` changes the factory contract so `Create(null)` returns `null` instead of a validation error. Setting `EmptyStringInFactoryMethodsYieldsNull = true` automatically sets `NullInFactoryMethodsYieldsNull = true`; the reverse is not guaranteed. Treating them as independent knobs produces a null-passes-but-empty-fails policy; the only way to make both null and empty yield `null` is to set `EmptyStringInFactoryMethodsYieldsNull = true` alone.

[SKIP_KEY_MEMBER_MANUAL_BACKING]:
- `ValueObjectAttribute<TKey>.SkipKeyMember = true` suppresses generation of the key field/property. The generator reads `KeyMemberName` to locate the manually-declared member, which must be a property with an `init` setter when `KeyMemberKind = MemberKind.Property` — the generated constructor assigns `this.KeyMemberName = value`, requiring a writable accessor. For fields (`MemberKind.Field`), a `readonly` field is accepted because the generated constructor may initialize `readonly` members.

[OBJECT_FACTORY_HAS_CORRESPONDING_CONSTRUCTOR]:
- `ObjectFactoryAttribute.HasCorrespondingConstructor` (bool, default `false`) controls whether the generator emits a direct-constructor expression as a fast path for conversion from a trusted source such as EF Core materialization. When `true`, values arriving from a validated store bypass re-validation; values from wire sources still pass through `Validate`. This is the domain-internal trusted-source seam.
- Multiple `[ObjectFactoryAttribute<T>]` declarations on one type with different `UseForSerialization` values register independent converter registrations per serialization framework. A type can declare `[ObjectFactoryAttribute<Guid>](UseForSerialization = SerializationFrameworks.SystemTextJson)` for JSON and `[ObjectFactoryAttribute<string>](UseForSerialization = SerializationFrameworks.MessagePack)` for binary with different admission validators for each. `UseWithEntityFramework` and `UseForModelBinding` are always per-attribute, so EF Core can use a different factory than the JSON deserializer on the same type.

[ATOM_SWAP_IO_VS_SWAP]:
- `Atom<A>.SwapIO(Func<A, A>)` returns `IO<A>`, deferring the swap to effect evaluation time and allowing it to participate in IO composition and error handling. `Atom<A>.Swap(Func<A, A>)` executes immediately and returns the committed value. The distinction is load-bearing in a pipeline: `atom.SwapIO(update).Bind(newVal => downstream(newVal))` threads the committed state through the subsequent IO computation without a separate read; executing `Swap` inside an IO pipeline produces a side effect that is not sequenced with the monad's execution order.
- `Atom<A>.SwapMaybeIO(Func<A, Option<A>>)` returns `IO<A>`: returning `None` from the function aborts without committing and the IO produces the unchanged value; `Some(next)` commits. Both paths are `IO<A>`, never `IO<Option<A>>`. Callers that need to distinguish "aborted" from "committed" must capture state outside the `SwapMaybeIO` call.
- `Atom<M, A>.SwapIO(Func<M, A, A>)` and `Atom<M, A>.Swap(Func<M, A, Option<A>>)` expose the metadata `M` as the first parameter to the swap function. The correct shape for a policy cell needing access to a fixed configuration record without closing over it: `Atom<PolicyConfig, Schedule>(config, initialSchedule)` with swap `(cfg, s) => s | Schedule.maxDelay(cfg.MaxBackoff)`.

[ATOMHASHMAP_AS_POLICY_REGISTRY]:
- `AtomHashMap<K, V>.SwapKey(K key, Func<Option<V>, Option<V>> f)` receives `None` when the key is absent and `Some(v)` when present; returning `None` removes the key, returning `Some(next)` sets it. Adding, updating, and removing entries are all expressed as one `Func<Option<V>, Option<V>>` with no separate `Add`/`Update`/`Remove` surface. The function may be called multiple times on CAS retry and must be pure and allocation-minimal.
- `AtomHashMapChangeEvent<K, V>` is the event payload type fired after each committed mutation on `AtomHashMap`. The event fires only on committed paths — failed CAS retries and validator rejections do not fire it — making it a reliable policy-change notification: every invocation reflects a state that was actually installed.

[COMPLEX_VALUE_OBJECT_MEMBER_EQUALITY_COMPARER]:
- `[MemberEqualityComparerAttribute<TMember, TComparerAccessor>]` applies to a property or field of a `[ComplexValueObject]` and overrides the equality comparer for that specific member in `GetHashCode` and `Equals`. Multiple `[MemberEqualityComparerAttribute]` declarations on different members of the same complex value object are independent — each member carries its own comparer. This is the per-field equality policy: a complex value object with a `string` field requiring ordinal comparison and a numeric field requiring exact equality declares both without a shared global policy.
- `StringKeyedObjectComparer<T>.Ordinal`, `OrdinalIgnoreCase`, and the culture variants provide `IEqualityComparer<T>` for smart-enum or value-object types keyed on `string`. Using `StringKeyedObjectComparer<T>.Ordinal` as the comparer for a `FrozenDictionary<T, PolicyRow>` is the correct form: it delegates to `StringComparer.Ordinal`, preserving consistency between lookup semantics and enum identity.

[UNION_SKIP_EQUALITY_AND_BACKING_FIELD_INTERACTION]:
- `UnionAttributeBase.SkipEqualityComparison = true` suppresses `IEquatable<T>`, `Equals`, `GetHashCode`, and all comparison operators. This is the correct form for unions used as discriminated command or request types where structural equality is meaningless — equality generation is wasted code at that use site. The suppression makes the `UseSingleBackingField` choice irrelevant to correctness, since the backing-field layout would otherwise affect `GetHashCode` bucket distribution.
- `UseSingleBackingField = true` with a mix of struct and reference-type cases boxes every struct case on storage and unboxes on retrieval. When the union carries only reference-type cases, `UseSingleBackingField` saves memory by eliminating per-case backing fields at the cost of one interface dispatch on unboxing. The struct-case boxing cost is only paid when `T1IsStateless = false`; a stateless struct case has no payload to box.

[PROJECTION_EQUALITY_COMPARER_AS_POLICY_TABLE_KEY]:
- `Thinktecture.Collections.ProjectionEqualityComparer<T, TItem>` constructs an `IEqualityComparer<T>` from a `Func<T, TItem> selector` and an optional inner `IEqualityComparer<TItem>`. A `FrozenDictionary<RequestType, PolicyRow>` keyed on a union-case type uses `ProjectionEqualityComparer<RequestType, Kind>` where `Kind` is the discriminant, so two requests with the same kind map to the same policy row regardless of payload differences. The comparer is the policy definition, not infrastructure: it declares which dimension of the request type governs policy selection.
