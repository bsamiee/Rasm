# [CSHARP_BRANCH_IDEAS]

The cross-package C# concert: ideas that couple two or more C# packages into one capability, distilled from the eight folder pools. A concept that lives inside one folder stays in that folder's `IDEAS.md`; a concept that spans C#, Python, and TypeScript at once lives in `libs/.planning/IDEAS.md` and is referenced here as a wire seam, never restated. Each idea is a card — a bracketed slug leader plus the capability, what it unlocks, and the gap or technique it draws on. `[1]-[OPEN]` holds live ideas; `[2]-[CLOSED]` records a finished or dropped idea with a one-line disposition.

## [1]-[OPEN]

[CAPABILITY_CONTROL_PLANE]: one self-describing op catalog spanning the runtime spine, the execution lane, and the UI.
- Every operation is a typed effect-classed cost-modeled permission-gated `CapabilityDescriptor` minted in `AppHost/capability/registry` from the `Compute/remote/channels` proto op surfaces and the `AppUi/commands/commands-availability` intent table; a shape-discriminated discovery fold drives intent planning, the grant/cost broker previews cost, the command algebra wraps any invocation in a commit-or-rollback transaction routed onto the `Compute/intent/admission` rail, and the MCP projection and SDK codegen are projections of the one descriptor source.
- Unlocks one operation surface every consumer reads — the MCP agent tools, the in-process function-calling, the command palette, and the polyglot SDK all derive from a single catalog, so a new operation is a descriptor row, never a parallel client or a second op-metadata owner.
- Draws on the dependency-policy never-hand-roll rule and the official MCP C# SDK now owning the protocol: the registry is the one place op metadata lives, and the cross-language SDK-codegen consumption is the `libs/.planning` capability-catalog seam.

[ONE_CONTENT_IDENTITY_FABRIC]: the single `XxHash128` content-address seed threading cache, blob residence, diff, model-result reuse, and the codecs.
- The sealed snapshot hash, the Compute interchange identity, the incremental topology naming hash, and the per-node reactive-graph key are all one `XxHash128` law: `Rasm.Geometry/topology/reconciliation` emits the canonical geometry bytes, `Compute/interchange/codecs` folds artifact bytes plus deflection and tolerance into the key, `Persistence/cache/indexes` content-addresses blob residence and the model-result horizon, and `Compute/models/inference` plus the reactive scheduler reuse a sub-result by key.
- Unlocks fetch-by-content-key across every C# package — a warm-start blob, a result cache hit, a deduped snapshot chunk, and a clean reactive-graph node all collapse onto one identity, so no package mints a parallel key and a re-tessellation at a new deflection re-keys geometry and metadata together.
- Draws on the content-addressed-build-graph and self-adjusting-computation technique over the one seed; the cross-runtime bit-identical reproduction of this seed is the `libs/.planning` content-address seam.

[INCREMENTAL_COMPUTE_PIPELINE]: a content-keyed reactive recompute engine spanning the execution lane, the durable store, and the UI.
- `Compute/scheduling/runtime` keys every job-graph node on the content digest of its inputs, schedules only the transitive dirty closure on re-run, and replays every clean node from the `Compute/models/inference` result cache; the `Compute/solver/sweep` frame-budget governor bounds the dirty closure per frame, the `Persistence/sync/collaboration` op-log changefeed records the moved nodes, and `AppUi` receives only the genuine delta for a parametric edit.
- Unlocks interactive parametric editing across the AEC pipeline — a single-parameter tweak re-runs only the affected tessellate/solve/encode chain while every unchanged branch returns from cache, and a coupled geometry-and-field iteration transmits its true delta with no second versioning scheme.
- Draws on demand-driven incremental computation over the one content-address seed and the deterministic result cache, composing the identity and receipt owners the lanes already sit beside rather than minting a second memoization or dependency tracker.

[DETERMINISTIC_REPLAY_OBSERVATORY]: a reproducibility kernel binding the runtime spine, the durable op-log, and the notebook surface.

- Draws on the chained-event-log and verifiable-credential technique over the one content-address seed; cross-machine replay-verify consumes the Compute provider-determinism fingerprint, and a signed grant attestation chained into the event log closes the in-process-only authorization gap.
