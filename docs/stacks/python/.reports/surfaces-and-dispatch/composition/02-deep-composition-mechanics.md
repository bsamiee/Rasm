# [DEEP_COMPOSITION_MECHANICS]

[CURRY_FLIP_N_APPLICATION_ORDER]:
- `curry_flip(N)` is `@curry_flip(1)` for single-source-last flip and `@curry_flip(2)` for two-argument flip; N counts the number of leading positional args to defer, not the total arity. `_wrap_args(*args, **kwargs)` captures all trailing arguments first, then `_curry((), num_args, _wrap_curried)` builds N sequential single-arg closures for the leading params. The application order is therefore: trailing `*args`/`**kwargs` first, then leading params left-to-right one at a time. [expression/core/curry.py 5.6.0, 2026-06-09]
- `result.map2` is `@curry_flip(2)` over `(x: Result[A, E], y: Result[B, E], mapper: Callable[[A, B], C]) -> Result[C, E]`. The decorated call shape is `result.map2(mapper)(x)(y)` â€” mapper first, then `x`, then `y`. Supplying the two `Result` values positionally in the wrong order type-checks but silently swaps the `A` and `B` roles. `block.map2` is an alias of `block.starmap` operating on `Block[tuple[A, B]]`; it is NOT `@curry_flip`-decorated and takes `mapper` directly. These two `map2` surfaces are entirely different shapes and must not be conflated. [expression/core/result.py lines 385-391; expression/collections/block.py lines 784-785, 5.6.0, 2026-06-09]
- A two-surface `curry_flip(2)` partial like `result.map2(mapper)` returns `Callable[[Result[A, E]], Callable[[Result[B, E]], Result[C, E]]]`. Using it inside `pipe` reads left-to-right: `pipe(x_result, result.map2(mapper)(y_result))`. The inner application `result.map2(mapper)(y_result)` binds `y` early; the value piped in becomes `x`. Reversing the two `Result` arguments in the call silently swaps the mapper's argument order without a type error when `A` and `B` unify. [expression/core/curry.py lines 174-205; expression/core/result.py lines 385-391, 5.6.0, 2026-06-09]

[TRAVERSE_SEQUENCE_FOLD_BACK_SEMANTICS]:
- `traverse(fn, lst)` in `expression.extra.result.traversable` is implemented via `seq.fold_back(folder, lst)` seeded with `Ok(block.empty)`. `seq.fold_back` reverses the source list and applies `functools.reduce(lambda x, y: folder(y, x), reversed(list(source)), state)`. This means the iteration over `lst` proceeds right-to-left at the `reduce` level, but the `folder` (which is an `@effect.result`-decorated generator) receives `(head, tail_result)` in left-to-right element order because `fold_back` restores element order through the reversal. The net observable guarantee is: the final `Block[B]` is in the same left-to-right order as the input `Block[A]`. [expression/extra/result/traversable.py lines 16-41; expression/collections/seq.py lines 552-580, 5.6.0, 2026-06-09]
- Short-circuit behaviour: `fold_back` materializes `list(source)` before reversing. `traverse`'s `folder` yields from `fn(head)` first; if that is `Error`, the `@effect.result` builder raises `EffectError` which the builder catches and returns the `Error` immediately. Because `fold_back` has already materialized the full list, ALL elements are consumed before traversal starts â€” only the applicative effect (bind-chaining) short-circuits. The first error encountered in left-to-right element order terminates the chain; elements after the first error are never passed to `fn`. [expression/extra/result/traversable.py lines 23-34; expression/core/builder.py lines 53-68, 5.6.0, 2026-06-09]
- `sequence(lst)` is `traverse(identity, lst)` â€” no independent implementation. The error type `E` must be homogeneous across the entire `Block[Result[A, E]]`; heterogeneous error types require `map_error` normalization before passing to `sequence`. There is no `traverse` overload accepting `Seq` or `Map`; both must be converted to `Block` first via `block.of_seq` or `map.to_list`. [expression/extra/result/traversable.py lines 44-50, 5.6.0, 2026-06-09]
- `fold_back` argument order differs between `seq` and `map`: `seq.fold_back(folder, source)` takes `folder: (A, State) -> State` then source; `map.fold_back(state, folder, table)` takes state first and its `@curry_flip(1)` shape is `map.fold_back(folder)(state)(table)` â€” the state argument is the first flipped-to-last param, not the folder. Mixing these shapes in a polymorphic pipeline silently misroutes arguments. [expression/collections/seq.py lines 552-580; expression/collections/map.py lines 415-421, 5.6.0, 2026-06-09]

[BLOCK_SEQ_MAP_AS_COMPOSITION_SURFACES]:
- `Block.fold(folder, state)` is `functools.reduce(folder, self, state)` â€” left fold, `folder: (State, A) -> State`. The module-level `block.fold` is `@curry_flip(1)` over `(source, folder, state)`, so the pipe form is `pipe(blk, block.fold(folder, state))`. This is a three-argument flip: trailing `(folder, state)` are captured first, then `source` is the single curried leading arg. [expression/collections/block.py lines 157-178, 624-651, 5.6.0, 2026-06-09]
- `block.partition(predicate)` returns `tuple[Block[A], Block[A]]` (truthy first, falsy second); it is `@curry_flip(1)` so pipe form is `pipe(blk, block.partition(pred))`. This is the correct surface for splitting a `Block` into two rails before diverging ROP paths â€” replace any hand-rolled `filter`/`filter` double-pass with one `partition`. [expression/collections/block.py lines 307-331, 830-850, 5.6.0, 2026-06-09]
- `block.try_head` returns `Option[A]` and is not curried (single-arg, no flip). The module-level function takes `source: Block[A]` directly. `block.choose(chooser)` is `@curry_flip(1)` and maps each element to `Option[B]`, collecting `Some` values â€” it is the correct replacement for a `filter` + `map` pair when the mapping is partial. [expression/collections/block.py lines 457-467, 560-567, 5.6.0, 2026-06-09]
- `Map.try_find(key)` returns `Option[V]` and the module-level `map.try_find` is `@curry_flip(1)` â€” pipe form is `pipe(m, map.try_find(key))`. `Map.change(key, fn)` where `fn: Option[V] -> Option[V]` is the atomic upsert/delete primitive; it is `@curry_flip(1)` over `(table, key, fn)`. `Map.fold` folds `(State, (K, V)) -> State` left; `Map.fold_back` folds `((K, V), State) -> State` right. Both `Map` folds operate over key-sorted tree order (AVL), not insertion order. [expression/collections/map.py lines 58-59, 95-99, 160-161, 406-421, 5.6.0, 2026-06-09]
- `block.unfold(generator, state)` is `@curry_flip(1)` over `(state, generator)`: `pipe(seed, block.unfold(gen))`. `generator: State -> Option[tuple[A, State]]` returns `Nothing` to terminate. This is the correct surface for building a `Block` from a recursive fixpoint rather than constructing via mutable list and converting. [expression/collections/block.py lines 469-486, 996-1014, 5.6.0, 2026-06-09]

[RESULT_BUILDER_CALL_OVERLOADS_AND_GENERATOR_CONTRACT]:
- `ResultBuilder.__call__` in `effect/result.py` has one declared `@overload`-free signature: `fn: Callable[_P, Generator[T | None, T, T | None] | Generator[T | None, None, T | None]] -> Callable[_P, Result[T, E]]`. The generator's `send` type is either `T` (when the generator re-receives the unwrapped value) or `None` (when it does not). There is no static overload distinguishing these two generator shapes; the checker accepts both. [expression/effect/result.py lines 35-48, 5.6.0, 2026-06-09]
- The `Builder._send` implementation uses `gen.send(value)`. On the first call `value` is `None` (generator starts). On each subsequent bind step the unwrapped `ok` value is sent back. A `yield from sub_result` inside the effect body means the generator delegates to `Result.__iter__` which either sends the unwrapped value back (Ok path) or raises `EffectError(self)` (Error path). The builder catches `EffectError` in `_send`, sets `done`, and returns `return_from(error_result)`, causing early termination. Bare `yield value` (not `yield from`) does NOT unwrap â€” it sends the `Result` object itself as the next `send` value, producing a type error at the assignment site. [expression/core/builder.py lines 48-72, 95-131; expression/core/result.py lines 251-256, 5.6.0, 2026-06-09]
- The `ResultBuilder.combine` implementation is `xs.bind(lambda _: ys)` â€” sequential combination discards `xs`'s value and returns `ys`. This means multiple independent `yield from` statements inside one `@effect.result` block are sequenced, not zipped. Two independent fallible calls that should both succeed must each `yield from` independently; their values are bound in declaration order. If the first `yield from` produces `Error`, the effect short-circuits before the second is evaluated. [expression/effect/result.py lines 28-30, 5.6.0, 2026-06-09]
- Return type annotation of an `@effect.result[T, E]()`-decorated function must be written as `Result[T, E]` on the outer function. The builder's `__call__` wraps in `wrapper(*args, **kw) -> _TOuter` which erases the outer annotation at runtime. Static checkers infer `Result[T, E]` from the overload; if the outer function omits the annotation, checkers fall back to inferring `Generator[...]` from the undecorated body â€” always annotate the outer return explicitly. [expression/effect/result.py lines 35-48; expression/core/builder.py lines 74-131, 5.6.0, 2026-06-09]

[MAP_ERROR_NORMALIZATION_IN_MULTI_SURFACE_PIPELINES]:
- `result.map_error` is `@curry_flip(1)` over `(result, mapper)` â€” pipe form is `pipe(r, result.map_error(normalize))`. The method form is `r.map_error(normalize)`. Both are identity on the `Ok` track; only the `Error` case is transformed. `map_error` is the only combinator in `expression.core.result` that alters the error type without altering the ok type; `bind`/`map` leave the error type invariant. [expression/core/result.py lines 124-134, 394-397, 5.6.0, 2026-06-09]
- When composing two dispatch surfaces with distinct error types `E1` and `E2` into a single downstream `Result[T, E]`, the normalization seam must be placed at each surface boundary before the first downstream `.bind` or `pipeline` step. The pattern is: `pipe(surface_a(args).map_error(into_e), result.bind(surface_b)).map_error(into_e)` for the second surface, or equivalently via `pipeline`: `pipeline(lambda v: surface_a(v).map_error(into_e), lambda v: surface_b(v).map_error(into_e))(input)`. Applying `map_error` after a `bind` that returns `E2` causes a type mismatch because the bind already widened the error channel. [expression/core/result.py lines 124-134, 136-146; expression/extra/result/pipeline.py lines 71-96, 5.6.0, 2026-06-09]
- `pipeline` in `expression.extra.result` uses `functools.reduce` seeded with `Ok` (the constructor function, not a value). The zero-arg form `pipeline()` returns `lambda x: Ok(x)` â€” a lift into the `Ok` track. Each step is composed via `acc(x).bind(fn)`. The error type `E` must be uniform across all `pipeline` steps at the call site; heterogeneous errors require `map_error` inside each step lambda before the pipeline is built, not after. [expression/extra/result/pipeline.py lines 71-96, 5.6.0, 2026-06-09]
- `Result.or_else_with(other)` accepts `other: Callable[[E], Result[T, E]]` â€” the error value is passed to the recovery function, enabling context-preserving fallback without `map_error`. This is the correct combinator for multi-surface recovery where the downstream surface needs the original error as input (e.g., retry with error context, fallback lookup keyed on the failure). `or_else` takes an already-evaluated `Result` and is eagerly evaluated regardless of the Ok track; `or_else_with` is lazy. [expression/core/result.py lines 212-220, 5.6.0, 2026-06-09]
