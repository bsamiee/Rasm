# Applicative and Monad Sequencing Algebra

[ONE_SKELETON_TWO_COMBINATORS]:
- Applicative `Traverse` and monadic `TraverseM` over a sequence are byte-for-byte the same fold skeleton — `Foldable.fold(add, F.Pure(Empty), ta).Map(Kind)` — and diverge in exactly one closure, the per-element `add`. The applicative arm folds with `Applicative.lift((bs, b) => bs.Add(b), state, f(value))`; the monadic arm folds with `state.Bind(bs => f(value).Bind(b => F.Pure(bs.Add(b))))`. The accumulate-versus-abort distinction is not two algorithms; it is the applicative-versus-monad sequencing law applied to one accumulator, and the only structural difference a reader can point to is `lift` versus `Bind` in the step.
- `lift` evaluates *both* operands unconditionally — `F.Pure(f).Apply(state).Apply(effect)` — so a failing element never stops the remaining effects from running and every failure reaches the result. `Bind` on a failed or empty left never invokes the right, so the first failure aborts the chain and later effects never run. The choice of which to call is the choice of whether downstream effects after a failure execute at all.
- The right-result-type recovery is mandatory: the fold yields the abstract `K<F, K<T, B>>` kind and a terminal `.Map(bs => bs.Kind())` plus a call-site `.As()` recovers the concrete carrier. Omitting `.As()` leaves the value unusable in concrete-typed position; the trait surface always hands back the kind, never the data type.
- `TraverseM`'s own trait-level default body is literally a call to the applicative `traverse` — `static virtual TraverseM(...) => Traversable.traverse(f, ta)`. A structure that does not override `TraverseM` therefore inherits *applicative* sequencing under a monadic name; the short-circuit it advertises exists only where the concrete type supplies its own `TraverseM`, and assuming abort-on-first-failure from an un-overridden structure is a latent correctness bug.

[TRAVERSE_AS_SEQUENCE_AFTER_MAP]:
- The canonical traversal law is carried as a concrete default method: `TraverseDefault(f, ta) => Traversable.sequence(Functor.Map(f, ta))`. Traversal is `sequence ∘ map` — first map every element into an effect, then swap the structure and effect layers. `Sequence` itself is the no-mapping degenerate: `Sequence(ta) => traverse(x => x, ta)`, an identity-mapped traverse. A type implementing only the abstract `Traverse` gets `Sequence`, `TraverseM`, and `SequenceM` for free, and the two `sequence` forms differ only in whether the inner functor is constrained `Applicative` (accumulate) or `Monad` (short-circuit).
- `SequenceM` is `traverseM(x => x, ta)` and inherits monadic short-circuit; `Sequence` is `traverse(x => x, ta)` and inherits applicative accumulation. Picking between them is picking the failure-collection policy at the swap point, expressed purely by the constraint on the inner functor, with no branch in the caller.

[VALIDATION_IS_THE_ACCUMULATING_APPLICATIVE]:
- The accumulating applicative is a monoidal collect: the failure channel is combined through an ad-hoc-resolved `Semigroup` instance, so two failures merge rather than the first winning. The combine instance is resolved at runtime, not constrained at compile time — using a failure type that is not a `Semigroup`/`Monoid` raises a load-time exception instead of a compiler error. That looseness is deliberate: omitting the compile-time `Monoid` constraint is what lets the same type implement coproduct, bifunctor, and bimonad surfaces, which in turn hand back a free `Partition`. The constraint was traded away to buy capability, and the price is a deferred, runtime-surfaced failure-type contract.
- The `&` operator is the applicative-conjunction shape: success values are collected into a `Seq` and failures accumulate through the semigroup; the `|` operator is `Choose` — first-success fallback that never evaluates the right when the left succeeds. Conjunction-accumulate and disjunction-fallback are distinct operators over the same carrier and are not interchangeable at a call site that needs one specific policy.
- The same data type carries *both* an accumulating applicative `Apply` and a short-circuiting monadic `Bind`: `Bind` on the failure case returns the failure untouched and never runs the continuation, while `Apply` merges. A traversal that wants every error needs the applicative path; the moment the code drops into a `Bind`/query block over the same type, it silently reverts to first-failure semantics. The type does not warn at the boundary; the sequencing operator chosen is the policy.

[IO_APPLICATIVE_IS_THE_PARALLEL_ONE]:
- Over an effect carrier, applicative `Apply` is concurrent and monadic `Bind` is sequential — and this is mechanical, not advisory. The applicative apply node launches both operands before awaiting either: it calls `RunAsync` on the function effect and on the argument effect, captures two in-flight value-tasks, and only if neither completed synchronously hands off to an async join node that awaits both. `Bind` cannot do this — the right operand is a function of the left's produced value, so it is structurally forced to await the left first. Choosing `lift`/`Apply`/the tuple-apply form runs independent effects in parallel; choosing `Bind` serializes them.

```csharp
// applicative apply node: both effects in flight before either is awaited
ValueTask<Func<A, B>> ff = Ff.RunAsync(env);
ValueTask<A>          fa = Fa.RunAsync(env);
return ff.IsCompleted && fa.IsCompleted
    ? Next(ff.Result(fa.Result))
    : new ApplyAsyncJoin<A, B, C>(ff, fa, Next);  // awaits both, not one-then-the-other
```

- The reverse-apply node exists so the argument effect can drive: an effect exposes `ApplyBack(f)` that builds the same parallel apply with the function and argument positions swapped, letting either side be the one a pipeline threads through while preserving the two-effects-in-flight evaluation. Empty and failure short-circuits are checked at node construction — an empty or failed function effect collapses to empty/failure without launching the argument at all — so the parallel launch is skipped precisely when one side is already known dead.

[ACTION_IS_SEQUENCING_WITH_A_DISCARD]:
- Discard-sequencing is `lift` with a projection that throws one operand away: forward action is `lift((_, y) => y, ma, mb)` — run the first effect, keep the second's value; back-action is `lift((x, _) => x, ma, mb)` — run both, keep the first. Both still evaluate both effects (it is applicative), so the discarded side's effect is *not* skipped, only its value. This is the parser-combinator `*>`/`<*` shape and the correct primitive for "do this effect for its effect, return that one's value" without a `Bind` that would wrongly imply data dependence.
- A non-empty effect list folds to one combined effect via repeated action: `Actions(fas) => fas.Tail.Fold(fas.Head, (h, t) => h.Action(t))` — left-to-right action sequencing that runs every effect and yields only the final value. The input is a non-empty carrier by type, so the empty-applicative-sequence case is eliminated at the type level rather than guarded at runtime; there is no "sequence of zero actions has no value" branch because the type forbids the empty input.
- Every action overload ships a memoised twin taking the second operand as a deferred, cache-on-first-use value, so the right effect is constructed lazily and evaluated at most once even when the combinator is reused. Reaching for the memoised overload is the way to avoid reconstructing or re-running an inner effect across repeated applicative application.

[CONSEQUENCES_FOR_TRAVERSAL_CHOICE]:
- For a collection of independent effects where every failure must be reported, applicative traversal is correct and monadic traversal is wrong — not slower, wrong: monadic traversal abandons the remaining effects on the first failure and the lost failures never appear. For a collection where the first failure should abort and later effects must *not* run (they are expensive, or they have ordering-dependent side effects), monadic traversal is correct and applicative traversal is wrong — it runs effects that should have been skipped.
- Over an effect carrier the same dichotomy decides concurrency: applicative traversal can evaluate elements in parallel because no element depends on another's value, while monadic traversal is forced sequential by the bind dependency. A traversal that is logically independent but written with `TraverseM` over an effect both loses parallelism and changes failure semantics; the operator is the policy on two axes at once.
- The effect always still enumerates and forces the source structure up to the abort point — short-circuit stops the bind chain, not the source materialization already performed. A lazy backing is evaluated element-by-element as the traversal proceeds regardless of which sequencing is chosen; the short-circuit reclaims downstream *effect* execution, never the upstream structure walk already paid for.
