# Iterative Refinement as a Bounded Effect Loop and the Hidden-SVD Conditioning Cost

[CONDITION_EVIDENCE_HIDES_A_FULL_SVD]:
- The three induced-norm and rank queries on the base matrix surface are each a full singular-value decomposition in disguise: the rank query, the spectral two-norm, and the condition number all resolve to `Svd(computeVectors: false).Rank`, `.L2Norm`, and `.ConditionNumber` respectively — three independent calls construct three separate decompositions of the same operand, with no shared handle and no memoization. A conditioning gate that reads rank, then spectral norm, then condition factors the matrix three times at cubic cost each, when one retained decomposition answers all three from its singular-value vector.
- The cheap induced norms route to the provider as flat-array reductions and never touch a decomposition: the one-norm, the infinity-norm, and the Frobenius norm each call the provider's matrix-norm reduction over the contiguous value buffer at quadratic cost. The expensive surface is exactly the family that needs the spectrum — two-norm, rank, condition — and the cheap surface is exactly the family that needs only the entries. A gate that mistakes the spectral two-norm for a free aggregate pays a decomposition it never reads the vectors of.
- The correct condition estimate for a square operator avoids the spectral decomposition entirely by pairing a cheap induced norm of the operator with the same induced norm of its action on a probe through a retained factorization. The one-norm condition is `‖A‖₁ · ‖A⁻¹‖₁`, and the inverse-norm factor is estimated by a power-iteration probe — solve against a sign-vector right-hand side on the held factorization, renormalize, iterate a fixed small number of times — so the estimate costs one cheap norm plus a handful of triangular solves on a factorization already in hand, never a fresh cubic decomposition. The probe reuses the very handle the solve will use, so the conditioning evidence and the solution share one factorization.

```csharp
// κ₁ estimate: cheap ‖A‖₁ × Hager-style probe of ‖A⁻¹‖₁ on a held factorization — no SVD
static double ConditionOneEstimate(Matrix<double> a, ISolver<double> fac, int probes) =>
    a.L1Norm() * (Schedule.recurs(probes) | Schedule.Forever).Run()
        .Fold((norm: 0.0, x: Vector<double>.Build.Dense(a.RowCount, 1.0 / a.RowCount)),
            (s, _) => fac.Solve(s.x).PointwiseSign() is var sgn
                ? (sgn.L1Norm(), fac.Solve(sgn).Normalize(1.0)) : s).norm;
```

[REFINEMENT_IS_A_RESIDUAL_FOLD_OVER_A_HELD_HANDLE]:
- Single-precision-quality solutions earn full precision by residual correction without ever refactoring: factor once, solve once for the initial estimate, then loop computing the residual against the original operator, solving the correction on the same held factorization, and accumulating it into the estimate. The factorization is read on every iteration and written on none, so each refinement step is one matrix-vector product plus one triangular back-substitution — sub-cubic — against a cubic decomposition paid exactly once. The loop body never reconstructs evidence; it consumes the handle's solve arrow repeatedly.
- The residual must be formed against the original operator in the working precision, never against the reconstructed factors, because the factors already carry the rounding error the refinement exists to cancel. The residual is `b − A·x` with the unfactored operator; the in-place matrix-vector form writes the action into a reused buffer, the subtraction overwrites a second reused buffer, and the correction solve writes a third — three buffers, allocated once, threaded through every iteration. Reconstructing `A` from the factors to form the residual defeats the entire correction.
- The fold carries a typed iteration state, not a mutable accumulator: the running estimate, the previous relative-residual norm, and the held factorization travel as one immutable record through a state-threaded reduction. Each step projects a new state from the prior; convergence is the predicate `relative residual below the working-precision floor`, stagnation is `relative residual fails to shrink by a fixed contraction ratio`, and divergence is `relative residual grows`. The three outcomes are cases of one status discriminant the fold returns, so the loop's termination reason is recoverable from the final state, never from an exception or a sentinel return.

```csharp
// x ← x + A⁻¹(b − A·x) on one held factorization; recurrence cap as schedule, exit as fold predicate
static (Vector<double> X, IterationStatus Status) Refine(
    Matrix<double> a, ISolver<double> fac, Vector<double> b, double floor, int cap)
{
    var ax = Vector<double>.Build.Dense(b.Count);
    var bn = b.L2Norm();
    return (Schedule.recurs(cap) | Schedule.Forever).Run().FoldUntil(
        (x: fac.Solve(b), prev: double.PositiveInfinity, st: IterationStatus.StoppedWithoutConvergence),
        predicate: t => t.State.st != IterationStatus.Continue && t.State.prev < double.PositiveInfinity,
        f: (s, _) =>
        {
            a.Multiply(s.x, ax);                                       // in-place A·x, reused buffer
            var rel = b.Subtract(ax).L2Norm() / bn;
            return rel <= floor  ? (s.x, rel, IterationStatus.Converged)
                 : rel >= s.prev ? (s.x, rel, IterationStatus.Diverged)
                 : (s.x.Add(fac.Solve(b.Subtract(ax))), rel, IterationStatus.Continue);
        }) is var r ? (r.x, r.st) : (b, IterationStatus.Failure);
}
```

[REFACTORIZATION_PRIVILEGE_IS_DEFINITENESS_EXCLUSIVE]:
- Exactly one factorization handle in the dense suite can re-factor a successor operator into its existing factor buffer: the definite handle alone declares an in-place re-factorization method that block-copies a new operand over the held factor storage before re-invoking the provider, reusing the entire factor allocation across a stream of operands. The pivoting handle, both orthogonal handles, the spectral handle, and the rank-revealing handle expose no such method — each demands a freshly constructed handle per operand. The zero-allocation refactorization stream is a privilege of definiteness, not of squareness, so a sequential-update route whose operator mutates per step must hold the definite case to capture it; any other case pays a full handle allocation every step.
- A stream of near-identical definite operators therefore amortizes to one factor-buffer lifetime: construct the definite handle once, then drive successive operators through the in-place re-factorization, solving each against the refreshed buffer. The allocation budget of the whole stream is the single initial factor array plus the reused right-hand-side and result buffers — the per-step cost is the provider's factorization work with no managed allocation at all. This is the only dense path where a moving operator does not allocate per step, and it exists only while the operator stays definite; a step that loses definiteness throws at re-factorization and the rejection is the regime boundary.
- The log-determinant accessor on the definite handle is the underflow-stable evidence that the determinant accessor cannot give: the determinant of a large definite operator is a product of squared diagonal factors that underflows silently to zero, while the log-determinant sums the logarithms of the diagonal and stays finite. A streaming definite refactorization that tracks a log-likelihood — covariance updates, sequential Bayesian filtering — must read the log-determinant accessor per step, because the determinant accessor returns a flush-to-zero value with no underflow indication and the logarithm of that value is negative infinity. The held handle's log-determinant is recomputed from the refreshed factor on each in-place re-factorization at the cost of one diagonal pass.

[SCHEDULE_SHAPES_THE_LOOP_BUT_DELAY_IS_INERT]:
- The recurrence combinator caps the refinement iteration count as a recurrence count with zero inter-step delay, so the iteration policy is a value composed from the schedule algebra rather than a literal loop bound: a fixed recurrence cap composes by intersection with a cumulative-work ceiling and by union with a convergence-driven early stop, and the composed schedule is the single policy the fold consumes. The temporal combinators — spaced, exponential, fibonacci backoff — are inert for a compute-bound refinement because the loop has no I/O to pace; the only load-bearing combinator is the recurrence cap and the convergence predicate that short-circuits it. A backoff delay on a pure numerical refinement is policy that decides nothing and is the rejected form.
- The convergence short-circuit is a rail predicate over the residual, not a schedule property, so the termination reason and the iteration cap occupy two separate layers: the schedule owns the upper bound on steps and the residual predicate owns the early exit, and the two compose without either knowing the other. The fold returns the first state whose status leaves the continue case, so reaching the cap is itself a terminal status — stopped-without-convergence — distinct from converged, diverged, and the failure status that marks a degenerate factorization. Five statuses partition every exit, and the caller's recovery dispatches on the status case, never on inspecting the residual after the fact.
- The cap and the working-precision floor are policy values seated together, not magic constants in the loop body: the floor is the relative-residual target derived from the working precision, the cap is the recurrence count, and the contraction ratio that detects stagnation is a third row of the same policy record. A refinement that hard-codes any of the three in the loop body has smuggled policy into control flow; lifting all three into one record makes the loop body a pure transition function over operand, factorization, and policy, and the same body serves single-precision-promoted and full-precision-tightened refinements by swapping the policy row.

[OVERFLOW_SAFE_NORMS_COST_A_SCALAR_PASS]:
- The vector two-norm that drives every convergence test is overflow-safe at the cost of being scalar: it aggregates the elements through a hypotenuse reduction that rescales to avoid intermediate overflow and underflow, so a residual whose squared sum would overflow a naive accumulation still norms correctly, but the reduction is one managed call per element with no vectorization. For a refinement loop whose residual norm is recomputed every iteration over a large residual, this scalar reduction is the per-iteration floor cost, and the vectorized sum-of-squares primitive over the residual's contiguous buffer replaces it with one wide pass — at the price of forfeiting the overflow rescaling, which is safe only when the residual magnitude is known bounded.
- The relative-residual denominator is computed once and reused, never per iteration: the right-hand-side norm is invariant across the refinement, so a loop that recomputes it each step pays a redundant scalar reduction. Hoisting the denominator into the initial fold state and threading it unchanged makes each iteration cost exactly one residual norm, and the relative residual is a division by the held constant. The same hoist applies to the operator's cheap induced norm when the loop also tracks a running condition estimate — both invariants live in the initial state, computed once.

[RESIDUAL_ARITHMETIC_IS_COLUMN_MAJOR_FUSED]:
- The residual matrix-vector product against the column-major operator is a fused multiply-add per column, not a strided row-dot: each column of the operator is a contiguous slice of the flat value buffer, so the action accumulates one column's contribution per fused multiply-add of the column span scaled by the corresponding solution component into the accumulator span. The row-dot phrasing strides across columns at the row-count stride and defeats the contiguous access; the column-major fused accumulation sweeps each column once in storage order. The residual subtraction then runs as one span-level subtract over the accumulator, and the entire residual formation is two wide passes with no virtual element dispatch.
- The correction accumulation into the running estimate is an in-place vector add over the result buffer, so the refinement reuses the estimate buffer across every iteration rather than allocating a fresh estimate per step: the held factorization writes the correction into a scratch buffer, and the estimate absorbs it in place. Across a capped refinement the only persistent allocations are the estimate, the residual scratch, the action scratch, and the correction scratch — four buffers for the whole loop regardless of iteration count, against a naive phrasing that allocates a fresh vector at every solve and subtraction.
