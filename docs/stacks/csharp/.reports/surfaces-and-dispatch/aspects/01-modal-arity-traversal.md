# Modal Arity As Carrier-Polymorphic Traversal

[ARITY_QUADRANT]:
- One entrypoint owns four modalities and the discriminant is recoverable from the value's shape, not a knob: singular is `Map`/`Bind` over `K<F, A>`; plural-preserving is `Traverse`/`Sequence` (`K<T, A>` to `K<F, K<T, B>>`, same container shape, effect lifted out); plural-reducing is `Fold`/`FoldMap` (`K<T, A>` to one summary value); first-success is `Choice`/`oneOf` (`K<T, K<F, A>>` collapsed to the first inhabited `K<F, A>`). The four are the same carrier `K<F, A>` consumed at different shapes — adding a `mode`, `batch`, or `many` flag beside the value re-describes a discriminant the container already carries and is the rejected form.
- The container being traversed and the effect being threaded are independent type parameters: `Traverse<F, A, B>(Func<A, K<F, B>> f, K<T, A> ta) where F : Applicative<F>` quantifies over the inner effect `F` while the outer `T` is fixed by the receiver. One traversal arrow therefore serves every effect carrier — a validation rail, an async effect, an option — without a per-effect overload, because the effect is resolved at the call site by the function's return type.
- The reducing modality is monoid-keyed, not loop-keyed: the nullary `Fold<A>(K<T, A> tm) where A : Monoid<A>` and `FoldMap<A, B>(Func<A, B> f, K<T, A> ta) where B : Monoid<B>` collapse a container through the element (or projected) monoid with no accumulator parameter and no folder lambda. A "sum these", "concat these", "merge these" batch endpoint is a `Monoid<B>` constraint plus a projection, never a hand-threaded accumulator; the monoid is the policy value that selects the reduction.
- The reducing modality has an effectful sibling that short-circuits inside the effect: `FoldM<A, M, S>(Func<A, Func<S, K<M, S>>> f, S initialState, K<T, A> ta) where M : Monad<M>` and the `FoldWhileM`/`FoldUntilM` predicate forms thread state through a `Monad<M>`, so a batch reduction that must stop on the first failure or first predicate hit is the same fold with an effectful folder, not a separate guarded loop.

[ACCUMULATE_VERSUS_SHORTCIRCUIT_FORK]:
- The plural-preserving modality forks on one constraint and that single choice flips error semantics at the same call site. `Traverse`/`Sequence` require `F : Applicative<F>`; `TraverseM`/`SequenceM` require `M : Monad<M>`. Over an accumulating carrier, the applicative path collects every failure and the monadic path stops at the first — the discriminant between accumulate-all and fail-fast is the free function chosen, with identical arguments.
- The fork is real because the accumulating carrier implements both traits with divergent operations, not because one trait is missing. The carrier's `Apply` merges two failures through the fail-type semigroup; its `Bind` returns the first failure and never inspects the second. Traversal that dispatches through `Apply` accumulates; traversal that dispatches through `Bind` short-circuits — same carrier, two arrows.
- A short-circuiting carrier (a single-error rail) implements `Apply` to return the first failure without merging, so its applicative and monadic traversals coincide. The accumulate-versus-fail-fast choice only has observable consequence on a carrier whose fail type is a semigroup; on a single-slot error rail both paths are fail-fast and the constraint difference is invisible.
- The accumulating fork carries a definition-time hazard: the fail type's semigroup is resolved at runtime through an instance lookup, not by a static type constraint on the carrier. A carrier parameterized over a fail type that is not a semigroup compiles cleanly and throws only at the first applicative apply, accumulate operator, or applicative traverse. The safe fail types are an error monoid or a sequence; the constraint surfaces explicitly only on the convenience folds keyed `where F1 : Monoid<F1>`.

[FIRST_SUCCESS_DISPATCH]:
- The first-success modality is one fold through a binary choice operator and reaches every alternative carrier uniformly. `oneOf<F, A>(params K<F, A>[])` and the sequence form require `F : Alternative<F>`; the body folds the collection left-to-right through `Choose`, returning the first inhabited result and, when none succeed, the last:

```
static virtual K<F, A> Choice<A>(in Seq<K<F, A>> ms)
{
    if (ms.IsEmpty) return Empty<A>();
    var k = ms[0];
    foreach (var item in ms.Tail) k |= item; // |= is Choose
    return k;
}
```

- The choice operator carries a lazy second-argument overload — `Choose<A>(K<F, A> fa, Memo<F, A> fb)` — so the fold does not force later alternatives once an early one succeeds. Any custom alternative carrier that omits the memoised overload makes the parser-combinator surface (`Many`, `Some`, `SepBy`) and the choice fold recurse without short-circuit and overflow; the lazy overload is mandatory infrastructure, not an optimization.
- First-success and accumulate are the two semigroup-shaped collapses and they bias oppositely on the same carrier: the choice combine keeps the first success and merges failures only when all fail, while the applicative accumulate merges every failure regardless. The carrier exposes both as operators — a non-short-circuiting accumulate operator and a first-success choose operator — alongside truthiness operators so the C# short-circuiting boolean operators select first-success while the non-short-circuiting forms accumulate both operands.

[ADMISSION_FEEDS_DISPATCH]:
- The plural entrypoint's element type is an admitted owner, so the traversal is the seam where many raw inputs cross the boundary at once. The generated admission factory is `static abstract TError? Validate(TValue? value, IFormatProvider? provider, out T? item)` returning the error or null — a `params`/span/collection-expression arity at the boundary maps each raw element through this factory and the per-element factory outcome is bridged into the carrier the traversal threads.
- Pairing the accumulating carrier with the per-element factory makes one traversal admit a whole batch and report every rejection in one pass; pairing the short-circuiting carrier with the same factory admits the batch and stops at the first rejection. The batch-admission policy — collect-all versus fail-fast — is therefore the applicative-versus-monadic traversal choice, not a flag on the admission factory, and the factory is written once.
- The owner's factory error is a typed payload constrained `where TError : class, IValidationError<TError>` with `static abstract TError Create(string message)`. Naming a custom error type in the owner declaration replaces the default across the entire generated factory surface, and that same error type becomes the carrier's fail slot at the traversal — so the accumulating traverse merges domain-typed errors through their semigroup, not stringly-typed defaults.
- The smart-enum admission path exposes the batch source itself as a generated surface: `static abstract IReadOnlyList<T> Items { get; }` with `static abstract bool TryGet(TKey? key, out T item)` whose key resolution is fronted by a `TryGetFromKey(object? key, out object? obj, out object error)` delegate seam over a generated lookup table. Traversing `Items` is the canonical bounded-vocabulary sweep — every case admitted and projected in one carrier-polymorphic pass — and key-to-case resolution is the table dispatch that delegate fronts.

[DISPATCH_SHAPE_GOVERNS_PLURAL_GRANULARITY]:
- Generated case dispatch executes synchronously and atomically at the call site, so wrapping a plural dispatch in a composition-time policy treats the dispatch and all its case behavior as one unit. Per-element retry, per-element timeout, or per-element catch is not expressible by wrapping the traversal; it requires each element to first be lifted into the carrier with its own policy, after which the traversal threads the already-wrapped elements. The granularity of the composition-time aspect is fixed by where the carrier is introduced — inside the per-element function, or around the whole traversal.
- The plural-preserving traversal returns the container shape intact inside the effect (`K<F, K<T, B>>`), so downstream dispatch sees a fully-admitted container or a single combined failure — never a mixed container of admitted and raw elements. This is why the discriminant stays recoverable: after the traversal the value is uniformly in the carrier, and the next entrypoint discriminates on its shape without a flag recording whether admission was singular or plural.
- The reducing and preserving modalities compose without an intermediate materialization: a traversal that admits a batch into the carrier can be followed by a monoid fold over the admitted container, or the effectful fold can both admit and reduce in one pass. Choosing the preserving traverse then a pure monoid fold versus a single effectful fold is a question of whether the admitted container is needed downstream, not a performance knob — the effectful fold discards the container shape, the traverse-then-fold retains it.
