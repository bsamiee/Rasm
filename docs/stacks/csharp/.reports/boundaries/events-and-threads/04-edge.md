# Events and Threads — Edge Findings

[IOUNINTERRUPTIBLE_MECHANISM_AND_SCOPE]:
- `IO<A>.Uninterruptible()` constructs `IOUninterruptible<A,A>`, whose `Invoke` creates `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — passing `CancellationToken.None` and `null` for the source. Because source is null, `New` allocates a fresh `CancellationTokenSource`, sets Own bit 1, and because `CancellationToken.None.CanBeCanceled` is false, installs no parent-propagation registration. The computation runs with an uncancellable token completely decoupled from the parent scope. Resources are shared (Own bit 2 = 0). On completion, the child env's `DisposeNonResources` disposes only the locally owned CTS — resources are never touched.
- The `IOUninterruptibleAsync` form carries the `LocalEnv` forward and awaits the task. The uncancellable token is live across the full async span. A computation that awaits on a host service internally polling its own token receives the `None` token — if that service uses `env.Token` to gate its loop, it will never observe cancellation. Finalization paths must themselves complete within finite time.
- `Uninterruptible` shares the parent resource scope and only isolates the cancellation token. `Bracket()` creates a child resource scope via `WithEnv(env => env.LocalResources)` and disposes it on success and failure. `BracketFail()` creates a child resource scope via `WithEnvFail(env => env.LocalResources)` and disposes it only on failure. For teardown sequences that must not be interrupted AND must clean up their own subscriptions, the correct composition is `Uninterruptible` wrapping a `Bracket` whose finalizer runs the detach-and-dispose sequence.

[ENVIO_OWNERSHIP_BITS_AND_SCOPE_INVARIANTS]:
- `EnvIO.New` assigns ownership via two bits in the `Own` field: bit 1 means the caller allocated the CTS and will dispose it; bit 2 means the caller allocated the `Resources` and will dispose them. `DisposeResources` disposes only if bit 2 is set; `DisposeNonResources` disposes the CTS only if bit 1 is set and always disposes the stored `CancellationTokenRegistration`.
- `env.LocalResources` calls `New(null, Token, Source, SyncContext)` — source is the parent's `Source`, not null — so bit 1 is not set and the parent CTS is never disposed. Resources is null so bit 2 is set; a fresh `Resources` with the parent as its `parent` field is allocated and disposed on child env disposal. Child subscriptions accumulate there and are disposed when the `using EnvIO local = ...` exits in `IOLocal.Invoke`.
- `env.LocalCancel` calls `New(Resources, Token, null, SyncContext)` — resources is the parent's `Resources`, not null — so bit 2 is not set. Source is null so bit 1 is set; a new CTS is allocated with the parent token registration installed to propagate cancellation. Disposal of the child env disposes the new CTS and the registration but leaves parent resources intact. When a subscription depends on a resource allocated in the parent scope, detaching it inside the child scope's computation is safe because parent resources survive child env disposal.
- `IOLocalOnFailOnly.Invoke` does not use `using` on the child env. On success it calls only `local.DisposeNonResources()` in the `finally` block — the CTS is disposed but resources are retained. On failure, `local.DisposeResources()` in the `catch` block releases the child resource scope, then `finally` disposes non-resources. This is the implementation of `BracketFail` and `WithEnvFail`.

[IASYNC_DISPOSABLE_IN_RESOURCES_SYNCHRONOUS_HAZARD]:
- `Resources.AcquireAsync<A>(A value) where A : IAsyncDisposable` stores a `TrackedResourceAsyncDisposable<A>`, whose `Release()` returns `IO.liftAsync(async delegate { await Value.DisposeAsync().ConfigureAwait(false); return unit; })`. `Resources.DisposeU` calls `resource.Value.Release().Run(envIO2)` synchronously — `IO.Run` drives the async computation to completion via `GetAwaiter().GetResult()`, blocking the disposing thread until `DisposeAsync()` completes.
- The hazard: `DisposeU` is called from `IOLocal.Invoke`'s `using` block and from `IOLocalOnFailOnly.Invoke`'s `catch`. Both may execute on an async state machine continuation running on the captured `SynchronizationContext`. If `DisposeAsync()` needs to marshal back to the same context for completion, the `.GetResult()` block produces a deadlock. Resources whose `DisposeAsync` marshals to a specific context must not be acquired inside a `WithEnv` or `BracketFail` block that runs on that context. Use `Prelude.use(() => new T(...))` with explicit `release` outside any scope that runs on the target context.
- When `TrackedResourceAsyncDisposable` coexists with `TrackedResourceDisposable` in the same `Resources`, disposal order between them is undefined (hash-trie traversal). When an async-disposable must be released before a synchronous disposable, encode the order explicitly by calling `release` in sequence inside the computation rather than relying on scope disposal.

[REPEAT_AND_RETRY_RESOURCE_SEMANTICS]:
- `IO<A>.RepeatUntil(Func<A,bool> predicate)` is `Bracket().Bind(v => predicate(v) ? IO.pure(v) : RepeatUntil(predicate))`. Each iteration wraps the body in `Bracket()` = `WithEnv(env => env.LocalResources)`, which creates a child resource scope disposed after each cycle. Resources acquired inside the body are released at the end of each cycle. A persistent subscription must be acquired outside the repeat body — opened at the level of the `IO` chain that contains the `Repeat`, not inside the repeated computation.
- `IO<A>.RetryUntil(Func<Error,bool> predicate)` is `Catch(BracketFail(), ...)`. `BracketFail()` = `WithEnvFail(env => env.LocalResources)`: resources are released only on failure. On success, the iteration's resources are retained in the parent scope. A retry loop that opens a connection on each attempt accumulates one live connection (the last successful one); each failed attempt cleans up. The first success's resource escapes the BracketFail scope into the parent.
- Both `Repeat` and `Retry` are CPS-recursive `Bind` chains, not iterative loops. Each iteration adds one frame depth to the `IO` DSL evaluation chain. For schedules with large iteration counts, this builds a deeply nested structure evaluated at `Run` time. `IO<A>.Repeat(schedule).WithState` — or equivalent schedule-driven patterns — encapsulates the iteration as a state machine rather than a recursive `Bind` chain and is the correct form for long-running fold workers.

[ATOM_CHANGE_DELEGATE_LIFECYCLE_AND_GC]:
- `Atom<A>.Change` is a raw C# multicast delegate field. A lambda registered as `atom.Change += v => SomeAction(v)` creates an anonymous delegate whose reference is lost immediately — detach is impossible without capturing the delegate to a variable before registration.
- `Atom<A>` holds a strong reference to every registered delegate, which holds a strong reference to its target. If the atom is a long-lived static cell and subscribers are activation-scoped objects, the atom prevents GC of those objects even after deactivation. Detach all `Atom.Change` subscriptions at deactivation using the exact stored delegate reference.
- `Atom.Swap` fires `this.Change?.Invoke(val2)` synchronously after the CAS succeeds. If the invocation list is long, every handler runs on the swapping thread before `Swap` returns. A concurrent swap from another thread can succeed between two sequential handler invocations in the same `?.Invoke` call, so late handlers in the list may observe a different committed value than early ones if a concurrent swap wins after the first handler ran.

[CTS_TRYRESET_AND_POOLING_CONSTRAINTS]:
- `CancellationTokenSource.TryReset()` returns `true` and unregisters all callbacks via `UnregisterAll()` only when `_state == States.NotCanceledState` AND either the timer is null or the timer has not yet fired. The `_everQueued` flag is set when the timer is queued to fire; once set it is never cleared. A CTS used with `CancelAfter` where the timeout elapsed sets `_everQueued = true` — `TryReset` returns `false` for it permanently.
- The contract for safe CTS pooling: (1) `TryReset` returns `true`; (2) the token was never cancelled; (3) `UnregisterAll` has run. The pooling value is for CTS instances created with `CancelAfter` that are cancelled before the timer fires — `CancelAfter`→`TryReset` before the timer fires is the only poolable pattern. `EnvIO.New` registers a parent-propagation callback at construction; for a pooled CTS to be reused in `EnvIO`, `TryReset` must be called before re-passing it to `New`.

[MANUAL_RESET_VALUE_TASK_SOURCE_CORE_AS_ZERO_ALLOC_EVENT]:
- `ManualResetValueTaskSourceCore<TResult>` (a `struct` in `System.Threading.Tasks.Sources`) is the zero-allocation foundation for reusable async events. `Reset()` clears all fields and increments `Version`; an awaiter holding a stale version number gets `ArgumentOutOfRangeException` from `GetResult` or `GetStatus` — the version token is the stale-detection mechanism.
- `SignalCompletion` dispatch: when `RunContinuationsAsynchronously = false`, the continuation is invoked inline on the completing thread before `SetResult` returns — the same reentrancy hazard as `ChannelOptions.AllowSynchronousContinuations = true`. When `RunContinuationsAsynchronously = true`, continuation is submitted via `ThreadPool.UnsafeQueueUserWorkItem(action, state, preferLocal: true)` — no `ExecutionContext` capture; the `preferLocal` flag promotes same-core resumption for tightly coupled producer-consumer pairs.
- The intended seam use: a host event that fires on a specific thread implements `IValueTaskSource<bool>` over a `ManualResetValueTaskSourceCore<bool>`, calls `Reset()` to prepare a new wait cycle, and calls `SetResult(true)` inside the event callback. `Reset()` must be called before the next consumer `await` — calling it while an awaiter is active corrupts the version. The protocol requires one-producer, one-consumer, reset-before-next-wait discipline.

[EXECUTIONCONTEXT_RESTORE_VS_RUN_AT_SEAM]:
- `ExecutionContext.Restore(ctx)` is a one-way install: it sets the current thread's execution context without saving the prior context or restoring it afterward. It is the building block for async state machines that must restore a captured context at `await` resumption points without the overhead of `Run`'s save-restore pair. Calling `Restore` in application-level code outside an async state machine produces a context leak: the thread continues with the installed context after the call site.
- `ExecutionContext.Run(ctx, callback, state)` saves the prior context, installs `ctx`, invokes the callback, and restores the prior context in a `finally` block. This is correct for event-dispatch wrappers that must not leak the event's context onto the calling thread after dispatch.
- `UnsafeRegister` callbacks that do only context-free work need neither `Run` nor `Restore` because they carry no ambient context semantics. Only callbacks that consume or propagate `AsyncLocal<T>` values need `Run`. Using `Run` in a high-frequency callback introduces allocation (context copy) and full restore overhead; profiling under load is warranted before adding `Run` to a tight event path.
- `Activity.Current` is backed by `AsyncLocal<Activity> s_current`. A host event that fires on a dedicated thread without a managed context (such as a native callback via P/Invoke) carries `null` `Activity.Current`. `UnsafeRegister` callbacks skip EC restore entirely, so `Activity.Current` inside such a callback is the activity current on the cancelling thread — neither the registration thread's activity nor the consuming thread's activity. Set `Activity.Current` explicitly inside the callback or use `Run` with the captured context if the callback must operate under a specific activity.

[RENDEZVOUSCHANNEL_INTERNAL_MECHANICS]:
- `RendezvousChannel<T>` has `CanCount = true`, `Count` always returns `0`, and `CanPeek = true`. `TryPeek` inspects the head of the blocked-writers queue under a lock and returns the item waiting for handoff without consuming it. `TryRead` pops the head writer from the queue, returns its item, and signals the writer's completion by calling `blockedWriteAsyncOperation.DangerousSetResult(default(VoidResult))`. The writer's `WriteAsync` then unblocks.
- The rendezvous is lock-mediated; `_readerSingleton` and `_waiterSingleton` are pooled, pre-allocated async operation objects reused across calls when one reader is active at a time, reducing allocation for the common single-reader-single-writer rendezvous pair.
- `TryRead` without a prior `WaitToReadAsync` returns `false` if no writer is blocked — the reader must be waiting before the writer arrives, or vice versa. The rendezvous is a synchronization point, not a queue; its correct use is explicit phase handoff between two independent async workers.

[IOTHREADPOOL_WORK_ITEM_BOXING_INVARIANT]:
- `ThreadPool.UnsafeQueueUserWorkItem(IThreadPoolWorkItem callBack, bool preferLocal)` stores the work item as `object` internally. A `struct` implementing `IThreadPoolWorkItem` is boxed at the call site — one allocation, better than a delegate wrapper plus closure, but still allocates. The allocation-free path for struct state is `ThreadPool.UnsafeQueueUserWorkItem<TState>(Action<TState>, TState, preferLocal)` with a struct `TState`; the generic overload stores the struct state inline without boxing.
- The `IThreadPoolWorkItem` interface (`void Execute()`) is the correct form when the work item is the context and needs to be cancellable: implement `Execute` to check a volatile flag before doing work. Structures implementing this interface without closures achieve the minimum allocation path for per-event work submission.

[SIGNAL_AND_COUNTDOWN_LIFECYCLE_AT_SEAM]:
- `Signal.autoReset<M>()` and `Signal.manualReset<M>()` both wrap `new EventWaitHandle(...)` with a `use` operator, tracking it in the current `Resources` scope. The underlying `EventWaitHandle` is disposed when the resource scope is released. A `Signal` consumed inside a `Repeat` body is disposed after each iteration; a `Signal` used as a cross-iteration synchronization point must be acquired outside the `Repeat` computation.
- `CountdownSignal<M>.Wait()` calls `CountdownEvent.Wait(CancellationToken)`, blocking the thread pool thread until the count reaches zero or the token fires. For high-count drains, prefer `PeriodicTimer`-driven polling of `handle.IsSet` or submit the wait to a dedicated thread via `Fork`.
- `Signal<M>.Wait()` calls `IO.liftAsync(e => handle.WaitOneAsync(e.Token))`. `WaitOneAsync` uses `ThreadPool.RegisterWaitForSingleObject` with EC capture — not the `Unsafe` variant. The resumption carries the `ExecutionContext` from the `await` site inside the `liftAsync` lambda, not from the thread that called `SetResult` on the TCS. This is correct for most seam uses but incurs one EC capture per `Wait` call.

[FORK_SYNC_CONTEXT_AND_POST_ALLOCATION_PRESSURE]:
- `IO<A>.Fork` passes `env.SyncContext` to the child `EnvIO.New`. The forked thread carries the parent's captured sync context. Any `IO.Post()` call inside the fork targets the same context as the parent — if the parent captured a host thread context, forked work that calls `Post()` marshals back to the same host thread. A fork spawned to protect the host thread from blocking work that still uses `Post()` does not reduce context-switch count; it only moves the blocking off the host thread.
- `IO<A>.Post()` allocates an `AutoResetEvent` per invocation and calls `SynchronizationContext.Post` followed by `await wait.WaitOneAsync(env.Token)`. `WaitOneAsync` uses `ThreadPool.RegisterWaitForSingleObject` internally — three kernel transitions per `Post()` call. For per-frame or per-event dispatch at high frequency, `Post()` is not viable; use `Channel<T>` or `AtomHashMap` state cells to batch work and marshal the batch once.
- When `env.SyncContext == null`, `Post()` falls through to `RunAsync` inline with no context switch. Code that relies on `Post()` for thread safety while running in a null-context environment silently gets inline execution and no actual marshaling. Seam contracts that require specific thread affinity must verify `env.SyncContext != null` before relying on `Post()` for enforcement.

[RESOURCES_DISPOSAL_ORDER_AND_ORDERED_TEARDOWN]:
- `Resources.DisposeU` iterates the `AtomHashMap<object, TrackedResource>` and calls each resource's `Release().Run(envIO2)` synchronously. `AtomHashMap` iteration follows HAMT traversal order, which is determined by hash values — effectively arbitrary from the caller's perspective. There is no LIFO, FIFO, or insertion-order guarantee.
- For teardown sequences where a subscriber must be detached before its connection is disposed, acquire resources with explicit `use` and `release` pairs in sequence. Call `release` on the subscription token explicitly inside the computation before the connection `use` exits. Do not rely on scope disposal order for ordered teardown — encode the order as a sequential `IO` chain:

```csharp
from conn  in use(IO.lift(OpenConnection))
from token in use(IO.lift(() => Subscribe(conn)))
from _     in IO.yieldFor(Duration.Zero) // ... actual work ...
from __    in release(token)             // explicit detach first
select unit                              // conn release follows scope exit
```

- `TrackedResourceDisposable<A>` calls `Value.Dispose()` synchronously. `TrackedResourceAsyncDisposable<A>` calls `await Value.DisposeAsync().ConfigureAwait(false)` wrapped in `IO.liftAsync`, then `Run(envIO2)` blocks via `.GetAwaiter().GetResult()`. This synchronous block-on-async is only safe when the disposing thread is not the `SynchronizationContext` target. Disposing an async-disposable resource from a host UI thread via scope disposal is a deadlock risk when the resource's `DisposeAsync` schedules continuation back to the same thread.
