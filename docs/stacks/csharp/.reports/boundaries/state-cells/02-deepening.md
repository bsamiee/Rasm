# State-Cell Boundary Law — Deepening

[RESOURCES_RELEASE_ORDERING]:
- `Resources.DisposeU` and `ReleaseAll` iterate `foreach (var resource in resources)` over an `AtomHashMap<object, TrackedResource>` backed by a HAMT. HAMT iteration follows hash-tree traversal order, not insertion order. There is no LIFO or FIFO guarantee on resource release. Designs that require ordered teardown must encode the sequence explicitly — a `Seq<IO<Unit>>` tracked alongside the map, or nested `Resources` parent chains where child-scope disposal precedes parent-scope disposal by construction.
- `Resources.Release<A>(A value)` walks the parent chain when the key is absent in the current scope: `resources.Find(obj).Match(f, () => parent != null ? parent.Release(value) : unitIO)`. A forked computation runs inside a child `Resources` constructed as `new Resources(parentResources)`. Any `Release` call that misses the child map traverses into the parent scope and releases a parent-scoped resource. This is the correct design for shared-resource release from a fork, but it is also the mechanism by which a forked computation can accidentally release a parent resource by key collision on `object` identity.
- `Fork` creates `new Resources(parentResources)` as the forked child scope. The fork's `finally` block calls `resources.Dispose()` on the child only; parent resources remain alive. However, any resource the fork explicitly releases via `IO<A>.Bracket()` or `use(...)` that was registered in the parent scope will be released through the parent chain traversal — the parent-scope resource is then gone. Never register a shared parent resource in a child bracket unless the intent is to transfer ownership to the fork.

[ENVIO_ENV_TOPOLOGY_PRECISION]:
- `EnvIO.Local` (property) computes `New(null, Token, null, SynchronizationContext.Current)` — passing `null` for both `resources` and `source`. The factory allocates both a fresh `Resources` and a fresh `CancellationTokenSource`, sets `Own = 3`, and registers `Token.Register(() => source.Cancel())` to forward parent cancellation. This is full isolation: new resource scope, new cancellation scope, parent cancellation propagates inward, parent resources are inaccessible.
- `EnvIO.LocalResources` computes `New(null, Token, Source, SyncContext)` — null resources but the parent `Source` is passed. The factory allocates fresh `Resources` (Own bit 2 set) but reuses the parent `CancellationTokenSource` (Own bit 1 NOT set). No `Registration` is created because the source is not new. Disposing this env does NOT dispose the parent source; cancelling the parent token still cancels operations run inside this env because they share the same `Source`.
- `EnvIO.LocalCancel` computes `New(Resources, Token, null, SyncContext)` — passes parent resources but null source. A new linked `CancellationTokenSource` is created (Own bit 1 set), with `Token.Register(() => source.Cancel())` wiring parent→child. Disposing this env disposes the child source but not the parent resources. Child cancellation does not propagate to parent; parent cancellation propagates to child via the registration.
- `IO<A>.Local()` calls `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — a fresh cancellation scope with shared resources. The `IOLocal` DSL node wraps the mapped env in `using EnvIO local = MapEnvIO(envIO)`, so both the new source and the registration are disposed when the inner operation completes. The confusingly named `.Local()` is a cancellation isolation wrapper, not a resource isolation wrapper; `.Bracket()` is the resource isolation wrapper.
- `IO<A>.Bracket()` calls `WithEnv(env => env.LocalResources)` which produces `IOLocal` (disposes on both success and failure). `IO<A>.BracketFail()` calls `WithEnvFail(env => env.LocalResources)` which produces `IOLocalOnFailOnly`: on success the local env's resources are NOT disposed (only `DisposeNonResources()` runs in `finally`), leaving them live for the caller. On failure, `DisposeResources()` runs in the `catch` before `DisposeNonResources()` in `finally`. The structural difference is `IOLocal` uses `using EnvIO local` while `IOLocalOnFailOnly` manually calls `DisposeResources()` only in catch.

[UNINTERRUPTIBLE_MECHANISM_PRECISE]:
- `IOUninterruptible.Invoke` constructs `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)`. Passing `CancellationToken.None` with a null source means a new `CancellationTokenSource` is created (Own bit 1 set), and because `CancellationToken.None.CanBeCanceled == false`, NO registration is created connecting the parent token to the new source. Parent cancellation does not propagate into the uninterruptible body during execution — isolation via a non-linked fresh source.
- The asynchronous uninterruptible path (`IOUninterruptibleAsync.Invoke`) is `async ValueTask<IO<B>>` that executes `A arg = await Fa; LocalEnv.Dispose(); return +Next(arg)` with no `try/finally` wrapping the `await`. If `Fa` faults, the exception propagates before `LocalEnv.Dispose()` is called, abandoning the isolated `CancellationTokenSource` until GC finalization. For high-frequency fault paths (network faults, timeout races inside uninterruptible regions), this accumulates live `CancellationTokenSource` objects. The correct form wraps `await Fa` in `try { ... } catch { LocalEnv.Dispose(); throw; }`.
- Composing `Uninterruptible` with an outer `Fork` creates a subtlety: the fork creates a child `EnvIO` linked to the parent via `env.Token.Register(() => tsrc.Cancel())`, so the fork's source is cancelled when the parent is cancelled. The `Uninterruptible` inside the fork creates its own local env from `CancellationToken.None`, so the fork's cancellation also does not reach the uninterruptible body. The fork's cleanup disposes its own `tsrc` and the parent registration, so no leak occurs even if the parent fires during the uninterruptible region.

[MEMO_STATE_MACHINE_RACE_SURFACE]:
- `Memo<A>` uses three states: `0` (unloaded), `1` (loading), `2` (loaded). `GetValue()` wins on `CAS(state: 0→1)`, runs `acquire()`, writes `value = result`, then writes `state = 2`. A failed `acquire()` writes `state = 0` in the catch block; the exception is rethrown. The two writes after the CAS are non-atomic; concurrent callers in state `1` spin via `SpinWait.SpinOnce()`.
- `Memo<A>.Reset()` CASes `state: 2→1` (sets to loading), then writes `value = default(A)`, then writes `state = 0`. Both reset writes are sequential after the CAS with no further atomic protection. A concurrent `Reset()` that also tries `CAS(state, 1, 2)` while state is `1` spins until the first Reset writes `state = 0`, then observes the `case 0: return default(Unit)` branch and exits — two concurrent `Reset()` calls are safe, but only one clears the value.
- `Memo<F, A>` (the higher-kinded form) uses the same three-state protocol on `K<F, A>?`. `Clone()` creates a new `Memo` from the same `acquire` delegate without cached state — allowing controlled re-acquisition without the concurrency hazard of `Reset()`. For boundary cells that need periodic re-acquisition under high contention, `Clone()` combined with replacement via an outer `Atom<Memo<A>>` is safer than `Reset()` under load.
- `MemoExtensions.Memo<T>(this Func<T> func)` wraps any function into a lazy-computed memoization cell. The multi-key form `memo<T,R>(this Func<T,R> func)` uses `ConcurrentDictionary<T, R>.GetOrAdd(key, f)`. `GetOrAdd` does NOT guarantee the factory runs exactly once when multiple threads race for the same key; for factories with side effects, wrap with a `Lazy<R>` value in the dictionary using `LazyThreadSafetyMode.ExecutionAndPublication`.

[ATOM_CHANGE_EVENT_TOPOLOGY]:
- `Atom<A>.Change` fires synchronously on the thread that won the CAS, immediately after `Interlocked.CompareExchange` succeeds. It does NOT fire when the validator rejects the candidate value. It does NOT fire when `SwapMaybe` returns the current unchanged value. It fires exactly once per successful committed transition. Between the CAS winning and the `Change` invocation, any concurrent reader of `atom.Value` already sees the new value — the change event lags slightly behind actual visibility.
- `Atom<M, A>` (the metadata form) has a `Change` event with signature `AtomChangedEvent<A>` — it receives only the new `A` value, not the `M` metadata. Handlers that need the metadata must capture it from the construction context.
- `AtomHashMap<K,V>.Change` fires via `AnnounceChange` after the CAS succeeds, carrying a `HashMapPatch<K,V>` with `prev` and `current` trie snapshots. However, `AnnounceChange` gates on `change != null && change.HasChanged` before calling `this.Change?.Invoke(...)`. The path from `Change` subscription to WithLog trie variants is operation-dependent: `SetItem` explicitly checks `this.Change != null` and switches between `SetItemWithLog` and `SetItem`, while `FindOrAdd` always calls `FindOrAddWithLog` regardless of subscription. For high-frequency registry cells where `FindOrAdd` dominates, the `WithLog` allocation cost is always present; the subscription gate matters only for `SetItem`-dominant workloads.

[CANCELLATION_REGISTRATION_LIFETIME_PRECISION]:
- `EnvIO.New` registers `token.Register(() => source.Cancel())` only when it creates a new `CancellationTokenSource` AND the incoming `token.CanBeCanceled`. The resulting `CancellationTokenRegistration` is stored in `Registration` and disposed in `DisposeNonResources()`. Failing to call `Dispose()` or `DisposeNonResources()` on a non-root `EnvIO` leaks the registration, which holds a delegate reference keeping the source alive until the parent token fires or the parent `CancellationTokenSource` is disposed.
- `EnvIO.DisposeNonResources()` uses `Interlocked.CompareExchange(ref nonResourcesDisposed, 1, 0)` to ensure it runs at most once. `DisposeResources()` uses the same pattern on `resourcesDisposed`. `Dispose()` calls both sequentially. Double-dispose is safe — idempotent by CAS. Calling `DisposeNonResources()` then `DisposeResources()` separately (as `IOLocalOnFailOnly` does in the failure path) is equivalent to a single `Dispose()` call; the CAS guards prevent double-release.
- A `CancellationTokenSource` created with a timeout (`new CancellationTokenSource(TimeSpan)`) starts the timer immediately at construction. If the `EnvIO.New` call that creates a timeout-scoped source is delayed inside a lazy `IO` tree, the timeout window begins at `IO` tree evaluation time. Stacking multiple `LocalCancelWithTimeout` calls does not compound the timeout — each call creates a new source from the current token at execution time.

[ATOM_VALIDATOR_GATE_INVARIANT]:
- `Atom<A>.New(A value, Func<A, bool> validator)` returns `Option<Atom<A>>.None` if the initial value fails the validator. The validator is stored as `Func<A, bool>` (substituted with `_ => true` when omitted). If the validator is expensive, it runs once per CAS attempt plus one extra per contended retry — pure O(1) validators are required.
- A concurrent `SwapMaybe(curr => curr.IsNone ? Some(acquire()) : None)` pattern for once-only acquisition has a double-acquisition hazard: two threads both see `None`, both call `acquire()`, one wins the CAS, the other's handle is leaked. The only safe form for once-only resource acquisition is a two-step protocol — gate CAS first (no acquisition), acquire, then a second CAS to commit — or structured disposal of the losing handle via the swap function's return value.

[RESOURCES_MERGE_AND_SCOPE_MIGRATION]:
- `Resources.Merge(Resources rhs)` is `internal` and adds all entries from `rhs.resources` into `this.resources` via `Swap`. It is used by the framework to migrate resources from a completed sub-scope into a parent and is not exposed publicly. Designs that need to manually promote resources from a child scope to a parent must go through the framework's `use`/`bracket` DSL.
- `Resources.Acquire<A>(A value, Func<A, IO<Unit>> release)` stores `(Value, Dispose: Func<A, IO<Unit>>)` in `TrackedResourceWithFree<A>` and calls `Dispose(Value)` when released. The `IO<Unit>` release runs inside the `EnvIO` provided to `DisposeU`, which carries the sync context — correct for resources whose cleanup must run on a specific thread or must be sequenced relative to other effects.
- `Resources.DisposeU(EnvIO envIO)` creates a fresh internal `EnvIO` with `CancellationToken.None` to run releases. Release effects are therefore never cancellable through the caller's token — they run to completion regardless of the outer cancellation state. A teardown effect that itself acquires a temporary resource must use a manually scoped inner `IO.Bracket` to isolate that acquisition, or the temporary resource outlives the teardown call.

[LIFECYCLE_FAULT_REENTRY_CELL_DESIGN]:
- A lifecycle state cell encoded as `Atom<LifecycleState>` with a `validator = _ => true` and transition constraints in the swap function — `curr => curr.IsValidPredecessor(next) ? next : curr` — correctly separates domain invariants from relational transition constraints. The validator is the wrong surface for transition constraints because it receives only the candidate next state; the current state must be read inside the CAS loop. A swap function that returns the current state unchanged when the transition is invalid, combined with `SwapMaybe` returning `None` to signal rejection, is the correct shape.
- Correlation cell replace-versus-reuse under fault re-entry: when a boundary layer transitions `Faulted → Opening → Open`, a `ConcurrentDictionary<CorrelationId, TaskCompletionSource<T>>` representing in-flight requests from the faulted instance must be fully swept before the new open instance begins accepting work. The sweep must fault all outstanding sources via `TrySetException(new BoundaryFaultException(...))` rather than `TrySetCanceled()`, because callers may distinguish cancelled (retriable) from faulted (non-retriable). Remove each entry before faulting it to prevent double-fault from a concurrent drain-plus-replace.
- A three-state initialization cell (`Initializing`, `Initialized`, `InitializationFailed`) prevents the window where a second caller observes `Initialized = true` while the initializer's side effects are still running. The winning initializer holds `Initializing` through the full side-effect sequence and commits `Initialized` or `InitializationFailed` as the final state transition. Callers that see `Initializing` must wait — use a `SemaphoreSlim(0, 1)` or a `TaskCompletionSource<Unit>` that the winner signals on completion.

[GENERATED_REGISTRY_KEY_STABILITY]:
- A generated registry backed by `AtomHashMap<K, V>` where keys derive from `nameof` on a source-generated type survives recompilation because `nameof` resolves at compile time to a stable string literal. Keys derived from `GetType().Name` or `GetType().GUID` are stable across recompilation but not across assembly signing changes or obfuscation. Keys derived from `RuntimeHelpers.GetHashCode` or object identity are per-process and produce ghost entries on every restart.
- When a generated registry accumulates ghost entries, `AtomHashMap<K,V>.Filter(pred)` atomically removes entries matching the predicate. The sweep must execute during the boundary layer's drain phase — before `Closed`, after all in-flight operations referencing the old keys complete. A filter sweep during `Open` state races with concurrent `FindOrAdd` calls; the filter predicate must check entry validity rather than key existence to be safe under concurrent use.
- `AtomHashMap.FindOrAdd(K key, Func<V> None)` is the correct once-only insertion primitive for singleton registries. The factory `None` runs inside the CAS loop and may execute more than once on CAS failure; it must be pure and idempotent. For factories that acquire resources, use `TryAdd(key, new Lazy<V>(factory))` and read `.Value` from the stored lazy — `Lazy<T>` with `LazyThreadSafetyMode.ExecutionAndPublication` ensures the factory runs exactly once.

[NATIVE_LIBRARY_IDENTITY_AND_FINGERPRINTING]:
- Native library identity for conflict detection requires three fields at minimum: absolute path, file size (from `FileInfo.Length`), and last-write timestamp (from `FileInfo.LastWriteTimeUtc`). Path alone is insufficient because file replacement at the same path changes the file without changing the path. `Assembly.LoadFrom(path)` does not prevent loading two physically different DLLs at the same path if one was loaded before the file was replaced.
- `NativeLibrary.Load(string libraryPath)` returns an `IntPtr` handle; `NativeLibrary.TryLoad(string, Assembly, DllImportSearchPath?, out IntPtr)` populates the handle and returns `false` on failure without throwing. The handle must be stored in the gate cell alongside the file fingerprint. `NativeLibrary.Free(handle)` is the symmetric release; releasing a handle still referenced by exported function pointers produces an access violation on next call. The gate cell must hold the handle until the process exits or the library is explicitly unloaded as a deliberate teardown step.
- Two process-static native library cells wrapping the same underlying native library must share a fingerprint check to detect version conflicts at the seam. An `AtomHashMap<string, (IntPtr handle, long size, DateTime modified)>` keyed by canonical name with a `SwapKey` that validates fingerprint consistency on each `TryAdd` surfaces conflicts as a typed failure at the second load attempt rather than silent undefined behavior from loading two incompatible versions.
