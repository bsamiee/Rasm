[BEARTYPE_STRATEGY_AXIS]:
- `BeartypeStrategy` is a four-member enum: `O0` (no-op bypass — disables all checks, useful for production toggles via env var); `O1` (default — samples one random container item per call, constant time); `Ologn` (samples a logarithmic number of items); `On` (samples all items, linear — use only at ingress boundaries with bounded input size). - `is_random: bool = True` controls whether `O1`/`Ologn` samples are random or deterministic (always index 0). Setting `is_random=False` produces reproducible checks for testing, at the cost of losing adversarial coverage.

[BEARTYPE_CONF_NUMERIC_AND_HINT_OVERRIDES]:
- `is_pep484_tower=True` silently widens `float` annotations to `float | int` and `complex` annotations to `complex | float | int`, matching the PEP 484 numeric tower. This is a lossy coercion guarantee, not a type check. The same effect is expressible precisely via `hint_overrides=FrozenDict({float: int | float})` without the tower's implicit complex widening. Prefer explicit `hint_overrides` when only float widening is needed. [beartype `_conf/confmain.py` lines 515–560, 616–631, 2026-06-09]
- `hint_overrides: FrozenDict` accepts `{source_hint: target_hint}` and rewrites annotations at decoration time. This is a blanket annotation substitution, not a structural subtype rule — use it only to normalize wire boundary types before domain logic sees them, not to silence real type errors.

[BEARTYPE_TO_RESULT_BOUNDARY_ADAPTER]:
- The canonical adapter: catch `BeartypeCallHintViolation` (or whatever `violation_type` resolves to) at the module boundary and lift to `Error`. The adapter is a one-shot `@beartype`-decorated function whose body is a single expression returning the validated value; the calling module catches the exception in a `try`-equivalent via `expression`'s `Result.of(lambda: validated_fn(...))` or a boundary-scope `catch` combinator. Do not place `try/except` inside domain flow — the catch belongs at the surface that owns the ingress contract.
- `violation_param_type`, `violation_return_type`, and `violation_door_type` allow routing different violation axes to different error types, enabling distinct `except` arms at the boundary: parameter violations map to `InputError`, return violations to `ContractError`, and `door` predicate failures to `PredicateError`. This allows a single decorated function to serve multiple downstream rail-lifting patterns without overloading one exception type.

```python
from beartype import beartype, BeartypeConf
from beartype.roar import BeartypeCallHintParamViolation, BeartypeCallHintReturnViolation
from expression import Result, Error, Ok

class InputContractError(Exception): ...
class OutputContractError(Exception): ...

_conf = BeartypeConf(
    violation_param_type=InputContractError,
    violation_return_type=OutputContractError,
    strategy=BeartypeStrategy.On,  # full check at ingress
)

@beartype(conf=_conf)
def _validated(payload: PayloadType) -> DomainResult: ...

def ingest(raw: object) -> Result[DomainResult, InputContractError | OutputContractError]:
    return Result.of(lambda: _validated(raw)).map_error(
        lambda e: e if isinstance(e, (InputContractError, OutputContractError))
        else InputContractError(str(e))
    )
```

[PEP_649_749_DEFERRED_ANNOTATIONS_AND_RUNTIME_CONTRACTS]:
- PEP 649 (Python 3.14+): annotations are stored as lazy evaluators (`__annotate__` dunder) rather than eagerly evaluated strings or values. PEP 749 (accepted, Python 3.14) adds `annotationlib` with `Format.FORWARDREF | FORMAT.VALUE | Format.STRING` and `get_annotations(obj, format=)`. [verified via beartype installed source and pydantic `_typing_extra.py`, 2026-06-09]
- Both beartype and pydantic already branch on `sys.version_info >= (3, 14)` to use `annotationlib.get_annotations(obj, format=Format.FORWARDREF)` instead of direct `__annotations__` access. The `FORWARDREF` format returns `annotationlib.ForwardRef` proxies for unresolvable names rather than raising `NameError`. beartype has a dedicated `pep749/` subpackage (`pep749evaluate.py`, `pep649749annotate.py`, `pep484749forwardref.py`) handling PEP 749 evaluators. [beartype `_util/hint/pep/proposal/pep749/`, pydantic `_internal/_typing_extra.py` lines 297–311, 2026-06-09]
- Effect on `@beartype`: on Python 3.14+, beartype calls `get_hintable_pep649749_annotations(hintable, format=FORWARDREF)` instead of `hintable.__annotations__`. Unresolvable annotations become `annotationlib.ForwardRef` objects, which beartype defers to call time for resolution (not decoration time). Circular imports that previously required `from __future__ import annotations` workarounds are resolved transparently.
- Effect on `beartype_this_package()` claw: the claw AST transform operates on the source AST before the module is compiled, so PEP 649 lazy evaluators do not affect it — it injects `@beartype` before any annotation is evaluated. This makes `claw` MORE robust to circular forward references than per-function `@beartype` applied to already-compiled objects.
- Effect on `@validate_call`: pydantic uses `signature_no_eval` (which calls `annotationlib.Format` when available) to read annotations without evaluating them at decoration time, then resolves them during schema generation via `NsResolver`. With `defer_build=True`, schema generation is pushed to first call — at which point all forward references are expected to be resolved. For modules using `from __future__ import annotations` (PEP 563 string mode), pydantic invokes `get_type_hints` with the module globals; on Python 3.14+ in PEP 749 FORWARDREF mode, it uses `get_annotations(format=FORWARDREF)` and resolves lazily. [pydantic `_internal/_typing_extra.py` lines 575–613, 2026-06-09]
- `from __future__ import annotations` (PEP 563 string mode) interacts differently with `@beartype` than PEP 749 FORWARDREF mode: PEP 563 stringifies all annotations to string literals; beartype's `resolve_pep563` function detects this (via `__future__.annotations` in the module's `__dict__`) and re-evaluates each annotation as an expression against the module globals. On Python 3.14+ with native PEP 749, `from __future__ import annotations` remains available but is expected to be deprecated eventually; beartype notes this explicitly. [beartype `peps/_pep563.py` lines 1–246, 2026-06-09]

[FREE_THREADING_SINGLEDISPATCH_DEPTH]:
- `singledispatch` in Python 3.15 (verified via `functools` source) uses `annotationlib.get_type_hints(func, format=Format.FORWARDREF)` inside `register()` when resolving annotation-based dispatch. `register()` mutates `registry[cls] = func` then calls `dispatch_cache.clear()` unconditionally. Under no-GIL (PEP 703/779 free-threaded build), the sequence `registry[cls] = func; dispatch_cache.clear()` is not atomic — a concurrent `dispatch()` call may read a stale `dispatch_cache` entry between the registry write and cache clear, or miss the new registration. [CPython `functools.py` `register` lines, verified 2026-06-09]
- The `dispatch()` function has a `cache_token` TOCTOU pattern: it reads `cache_token`, compares against `get_cache_token()`, and conditionally clears `dispatch_cache`. Under free-threading, two threads can both observe a stale `cache_token`, both clear and rebuild the cache redundantly. This is a performance race, not a data corruption race (writes to `WeakKeyDictionary` are idempotent for the same key), but it invalidates the cache more often than necessary.
- Mitigation: complete all `singledispatch` `.register()` calls at module import time (before any thread could call `dispatch()`). After import, the `registry` dict is read-only and `dispatch_cache` is only rebuilt on ABC virtual subclass invalidation (when `cache_token` changes), which is also a module-level operation in practice. Never call `.register()` from request handlers, task workers, or any code that runs after startup.
- `ContextVar` for per-call aspect state (e.g., correlation ID, tenant scope, observability context) is safe under all threading models including free-threaded: `contextvars` are implemented at the C level with per-task copies that do not share state between concurrent coroutines or threads. A `ContextVar.set()` returns a `Token` that can `reset()` to prior state — the correct pattern for stack-discipline aspect injection. [verified: `sysconfig.get_config_var('Py_GIL_DISABLED') == 0` on Python 3.15.0b1 GIL build; ContextVar C-level safety is build-independent, 2026-06-09]
- A `frozendict` value table (indexed by type, tag, or discriminant) is unconditionally read-safe under free-threading because it is immutable after construction. Prefer a `frozendict` dispatch table over `singledispatch` for closed bounded vocabularies; use `singledispatch` only for open extension points where external code registers new types. For the open case in a free-threaded app, protect `.register()` calls with a `threading.Lock` acquired once during startup (not per-dispatch, as that defeats the purpose of `singledispatch` caching).

[DECORATOR_STACK_ORDER_FOR_BEARTYPE_AND_VALIDATE_CALL]:
- `@beartype` must be the outermost non-caching decorator to guard the full call. Stack order from outermost to innermost: `cache` (`functools.cache` or `stamina.retry`) → `@beartype` → `@validate_call` → `@staticmethod/@classmethod` (if applicable). `functools.cache` must be outside `@beartype`, not inside — caching the wrapper would skip the type check on cache hits. `@beartype` inside `@cache` means the check fires on first call only and cached results are never re-checked.
- Do not stack `@beartype` and `@validate_call` on the same function. The outer decorator governs. `@validate_call` (with `ConfigDict(strict=True)`) and `@beartype` have similar no-coercion semantics, but `@validate_call` builds a Rust-backed `pydantic-core` validator while `@beartype` generates a Python wrapper. At an HTTP ingress boundary that already uses pydantic models, `@validate_call` is the natural choice; at an internal module boundary with no pydantic schema involvement, `@beartype` is correct.
- `warning_cls_on_decorator_exception: Optional[TypeWarning]` converts decoration-time exceptions (when `@beartype` cannot parse an annotation) into warnings instead of raising, defaulting to `_BeartypeConfReduceDecoratorExceptionToWarningDefault`. In strict library code, set `warning_cls_on_decorator_exception=None` to surface annotation errors at decoration time rather than silently degrading to no-op contracts.
