# [08] Prior Art — In-Host Test Bridges Elsewhere

How other ecosystems run tests inside a GUI host that cannot go headless, and what transfers to `tools/rhino-bridge`. Web research conducted 2026-06-10; per-system currency is marked. Baseline for comparison: the current bridge (client → protocol → plugin over named pipe; `.verify.csx` scenarios compiled in-host by RhinoCode; JSON envelope phases; Yak deploy + full Rhino quit/relaunch on plugin update).

## [2]-[SYSTEM_NOTES]

### [2.1] Unity — the reload story that matters

Unity is the only corpus member that solved the exact pain the operator names: code changes without host restart. The editor process is permanent; script/package changes trigger a domain reload that tears down and rebuilds the managed world only. UTF serializes its run state and resumes the run mid-suite after the reload; an unmanaged reload fails the test loudly ("unexpected domain reload") instead of hanging. The .NET-Core analog of reloadable AppDomains is the collectible `AssemblyLoadContext` — which RhinoCode already uses for scenario scripts (`csharp.resolver.isolate = true`). The unsolved Rasm gap is that the *bridge plugin itself* lives in Rhino's default load context, so updating it forces quit/relaunch. Unity's lesson: the permanently-loaded layer must be a dumb, version-stable shell; everything that iterates lives in the reloadable layer.

The player-side runner is the cautionary half: results return over PlayerConnection, and if the link cannot be established the run completes invisibly with no XML. Evidence must be persisted at the execution site, not only streamed.

### [2.2] Unreal — external orchestration + framework sprawl

Two-tier design: in-engine Automation Framework (driven via Session Frontend over the Message Bus / UdpMessaging) for interactive runs, and Gauntlet — an external C# orchestrator — for CI. Gauntlet deliberately knows nothing about test content; it launches sessions, monitors processes, and validates results from logs/artifacts after exit, which converts host crashes from hangs into parsed outcomes. The 2025 landscape survey (Fray) documents the cost of organic growth: six overlapping test families (Automation, Spec, CQTest, Blueprint functional, Map Check, Asset Validation) with disjoint capabilities — "if you could have the parametric tests from Spec, and the Before/AfterAll from CQTest, you'd have the perfect library" — and diagnostics routed to side windows that are "easy to miss." Local iteration via Session Frontend is called "an annoying window for the local automated tester."

### [2.3] Revit — both generations of the same bridge

RTF (2013-era, maintained) is the cold-process design: generate a journal + addin per run, boot a fresh Revit, run NUnit in-process, write NUnit XML to disk, kill Revit. Deterministic and crash-contained, but every run pays full Revit startup. ricaun.RevitTest (active, 2025) is the warm-host redesign and the closest structural cousin to `tools/rhino-bridge`: identical 3-component split (TestAdapter ≈ client, Console ≈ operator, Application ≈ plugin) over "client-server pipe communication" — but fronted by a standard VSTest TestAdapter, so tests are ordinary `dotnet test` / Test Explorer citizens, Revit reuse is default with `NUnit.Open`/`NUnit.Close` opt-outs, the target Revit version is inferred from the test assembly's RevitAPI reference, and the same tests can route to Design Automation (cloud Revit) unchanged. [UNVERIFIED] its in-Revit assembly reload mechanics (AppDomain vs ALC vs shadow copy) — README/marketing pages do not state it; source dive required before copying.

pyRevit demonstrates the persistent in-host server variant: an HTTP micro-framework living inside Revit, decorator-registered routes, host context injected into handlers — plus the warning labels (draft status, no auth). Its Go-based telemetry sidecar is the only corpus example of structured, queryable cross-run diagnostics storage.

### [2.4] McNeel's own prior art

Three first-party precedents bracket the design space. Rhino.Testing: in-process headless `RhinoCore` under NUnit — the right answer on Windows, unavailable on macOS (Rhino.Compute states the constraint plainly: "built on top of Rhino 8 for Windows"); its forum thread also documents the discovery trap — static C# properties touching RhinoCommon "load before this resolver is registered" (Sykes), and `TestCaseSource` with geometry fails at discovery time because discovery runs outside the host. Rhino.Compute: supervisor process (`compute.frontend`) health-checks and restarts the geometry host — the only corpus member with automated host resurrection. rhinocode CLI (Rhino ≥8.11): Rhino ships its own in-host script server reachable over `rhinocode_remotepipe_<pid>` named pipes, running any supported language in a chosen live instance — i.e., McNeel maintains a first-party version of the bridge's transport+execute core. The community MCP wave (2025-26) independently re-derived the same shape (in-host TCP/SSE listener + external server), confirming convergent architecture rather than a Rasm idiosyncrasy.

### [2.5] Test-platform hosts — where the front-end should live

VSTest's runner/testhost split over a custom TCP+JSON protocol is the legacy pattern; MTP collapsed it: the test project is the host, and anything that must survive host death (crash dumps, hang dumps) runs as a spawned out-of-process observer. That inversion — diagnostics watchdog outside the process under test — is directly transferable: the bridge client, not the plugin, should own hang detection and post-mortem collection. ricaun proves a GUI host can sit behind a standard test-platform adapter; MTP (already this repo's test runner) is the modern adapter surface.

## [3]-[TRANSFERABLE_LESSONS]

1. **Two-layer plugin: permanent shell + reloadable worker.** Unity's editor/domain split, RhinoCode's own collectible ALC, and ricaun's installed-once Application all point the same way: the `.rhp` should shrink to a version-stable bootstrap (pipe server + UI-thread marshal + ALC loader) that never needs redeploy; all bridge logic, protocol evolution, and scenario kit loading move into an ALC-loaded worker assembly swapped per run. This deletes the quit/relaunch cycle for everything except shell changes — the single highest-leverage fix for the operator's stated pain.
2. **The supervisor lives outside the host.** Gauntlet, RTF, ricaun's Console, compute.frontend, and MTP's out-of-proc extensions unanimously place lifecycle, timeout, crash-dump, and restart ownership in an external process that treats host death as a parsed result, not a hang. The bridge client already half-does this; completing it means heartbeat/liveness during execute (not just connect), `.ips`/crash-marker harvesting as evidence, and optional auto-relaunch — compute.frontend-style — as a policy.
3. **Persist evidence at the execution site; streaming is an enhancement.** Unity's player runner loses all results when the connection drops; RTF writes NUnit XML to disk in-host; Gauntlet parses artifacts after exit. The plugin should spool phase/fact/capture evidence to disk as it runs so a mid-run crash still yields a partial report (the current `Scenario.Run` try/finally fact flush is the right instinct — extend it below the scenario layer).
4. **Keep test discovery host-free or in-host — never half-and-half.** Rhino.Testing's worst failures are discovery-time: host-native types in `TestCaseSource`/static fields explode outside Rhino. The `.verify.csx` compile-in-host model already sidesteps this; preserve that property in any redesign (an MTP adapter must enumerate scenarios from file paths, not from loading scenario types).
5. **Warm reuse by default, cold-fresh as a knob.** ricaun (reuse + `NUnit.Open`/`Close`) vs RTF (always cold) is the iteration-vs-determinism axis; mature practice exposes both. The bridge has launch/quit primitives but no per-run recycle policy; make "fresh host for proof runs, warm host for iteration" a declared option rather than operator discipline.
6. **Fail loud on host-state transitions.** UTF turns an unexpected domain reload into an explicit test failure with a named cause. Bridge equivalents — plugin version skew, RhinoCode engine restart, document state leakage between scenarios — should be first-class fault categories, not symptoms discovered through downstream weirdness.
7. **Front the bridge with the standard test platform, not a bespoke CLI.** ricaun shows a GUI host behind a VSTest adapter gives `dotnet test`/IDE integration for free; MTP is this repo's existing runner. A thin MTP framework adapter that shells to the bridge client would make scenarios enumerable, filterable, and CI-reportable like any spec — and directly addresses the "linkage direction may be backwards" suspicion: test platform → client → host, with python orchestration optional rather than load-bearing.
8. **One test family, not six.** Unreal's sprawl (and its 2025 "perfect library doesn't exist" lament) is the canonical warning against accreting parallel scenario formats, runners, and result schemas. Whatever the rebuild produces, `.verify.csx` + one envelope schema should remain the only authoring surface.

## [4]-[ANTI_PATTERNS]

| [X] | [ANTI-PATTERN] | [EVIDENCE] |
| --- | --- | --- |
| [1] | Results that exist only over a live connection | Unity player runner: no connection, no XML — run evidence vanishes |
| [2] | Diagnostics routed where nobody looks | Unreal Map Check: "output is just sent to the Message Log, making it easy to miss" |
| [3] | Synthetic GUI input as a test mechanism | McNeel (Sykes): "physically moving the actual mouse cursor around with code will likely fail at some point" |
| [4] | Assuming host state between tests | Unreal guidance: "do not assume the state of the game or the Editor"; leave disk state as found |
| [5] | Host-native types on the discovery path | Rhino.Testing static-property/`TestCaseSource` failures before resolver registration |
| [6] | Bespoke wire protocols when a standard front-end exists | VSTest's custom TCP+JSON cited as legacy burden; MTP replaced it; ricaun rides the adapter instead |
| [7] | Chasing headless the vendor does not support | Fusion: "not a tested or supported configuration"; Rhino headless is Windows-only (Rhino.Compute) — mitigation, not denial, is the correct posture |
| [8] | Parallel test frameworks accreting per niche | Unreal's six families with disjoint fixtures/parameterization and no complete member |
| [9] | In-host hang with no external deadline | Unity batchmode freeze history ("Unity freezes and you have to Force Quit"); UI-thread execution is not server-cancelable — exactly the bridge's `--timeout-ms` caveat, so the client deadline must be authoritative |
