# Entrypoint Law — Deepening

[FROZEN_DISPATCH_TABLE_MECHANICS]:
- `FrozenDictionary.Create<TKey,TValue>(ReadOnlySpan<KeyValuePair<TKey,TValue>>)` (.NET 10 BCL, `System.Collections.Frozen`) accepts a collection-expression literal at the call site — `FrozenDictionary.Create([(key1, handler1), (key2, handler2)])` — with no heap-allocated intermediate because the compiler lowers the collection expression to a stack-allocated span. The comparer overload `Create<TKey,TValue>(IEqualityComparer<TKey>, ReadOnlySpan<...>)` controls key equality without a separate configuration parameter.
- `FrozenDictionary<TKey,TValue>.GetValueRefOrNullRef(TKey)` returns a `ref TValue` (null ref when absent); the caller checks `Unsafe.IsNullRef(ref value)` and reads through the reference without boxing. For dispatch tables where `TValue` is `Func<Cmd,Fin<Result>>` this eliminates both a boxing allocation and a second dictionary access.
- `FrozenDictionary<TKey,TValue>.GetAlternateLookup<TAlternate>()` lets an `FrozenDictionary<string, TValue>` keyed with `StringComparer.OrdinalIgnoreCase` accept a `ReadOnlySpan<char>` at lookup time when the comparer implements `IAlternateEqualityComparer<ReadOnlySpan<char>, string>`. The BCL's `StringComparer.Ordinal` and `OrdinalIgnoreCase` implement this interface. The dispatch table key stays `string`; the call site passes a span sliced from an incoming buffer without allocating. This is the only BCL mechanism for zero-alloc string-key dispatch from a span.
- `FrozenDictionary` is constructed once at module load and assigned to a `static readonly` field. The generator does not need to be a separate method — a collection-expression initializer compiles to a static field initializer, which the runtime executes once with no locking overhead.

[ARITY_COLLAPSE_FAILURE_SURFACES]:
- `params ReadOnlySpan<T>` and `params IEnumerable<T>` are both valid `params` collection forms under C# 13+. The compiler resolves to the `ReadOnlySpan<T>` overload first when both exist, regardless of priority, because span is considered a better conversion target for collection-expression arguments. `[OverloadResolutionPriority]` is only needed to break ties when the candidates are equally good conversions — the priority is not needed to select span over `IEnumerable` because span is already preferred.
- When a `params ReadOnlySpan<T>` parameter appears in a generic method where `T` is unconstrained, the compiler cannot lower a collection-expression argument to a stack-allocated span unless the call site provides a statically-known element count. If the count comes from a variable, the compiler emits a heap-allocated array. The entrypoint must either accept a typed span directly (`ReadOnlySpan<KnownCmd>`) or restrict `T` to `allows ref struct` to prevent the heap fallback.
- `[OverloadResolutionPriority]` negative values are valid and legal. Applying `-1` to a deprecated overload that takes `T[]` while leaving the span overload at `0` ensures the array overload loses at all call sites that pass a collection expression, without needing to delete the array overload for binary compatibility.
- The input-redescribing-parameter anti-pattern has a structural variant: a `CancellationToken` beside a union value is not redescription (it is a capability orthogonal to the discriminant), but a `bool processInParallel` beside a `Seq<Cmd>` is: the batch intent must be encoded in the union case's type, not as a parallel flag. The test: can the discriminant be recovered from the value alone? If yes, the parameter is redundant.

[BRACKETIO_FOUR_OVERLOAD_MECHANICS]:
- `bracketIO<M,A>(K<M,A>)` (single-arg, `Prelude`) creates a local resource environment scoped to the computation. Resources acquired via `IO.use<A>(resource)` (where `resource : IDisposable`) inside the computation are registered in the local scope and released when the scope exits regardless of success or failure. This is the correct shape for an entrypoint whose dispatch arm acquires a scoped connection or handle: wrap the entire dispatch call in `bracketIO(dispatch(request))` to guarantee cleanup.
- `bracketIO<M,A,B,C>(K<M,B> Acq, Func<B,IO<C>> Use, Func<B,IO<A>> Fin)` (three-arg) exists alongside a four-arg form `bracketIO<M,A,B,C>(Acq, Use, Func<Error,IO<C>> Catch, Fin)` that adds error-path handling before the `Fin` cleanup. The four-arg form does not replace the three-arg form: `Fin` always runs in both; `Catch` only runs on failure before `Fin` runs. This is the mechanism for logging or transforming errors from the acquisition or use phase without swallowing the cleanup.
- The single-arg `bracketIO` form is documented as preferred: it avoids the need to provide manual `Fin` handlers because the IO runtime tracks all `use`-registered resources automatically within the scope. The three/four-arg forms are for resources that cannot implement `IDisposable` directly — e.g., a remote connection requiring an async close call (`Func<B,IO<A>>`).

[CATCH_OPERATOR_DISCRIMINATION]:
- The `|` catch operator on `K<M,A>` has two right-hand operand types: `CatchM<TError,M,A>` (produced by `@catch`, `expected`, `exceptional`, and typed variants) and `Finally<M,A>` (produced by calling `@finally(K<M,B>)`). The two overloads exist on `FallibleExtensions` and are distinguished by right-operand type at compile time. Chaining: `computation | catchHandler | @finally(cleanup)` is legal because the result of `computation | catchHandler` is still `K<M,A>`, which then accepts the `Finally<M,A>` right operand. The `finally` clause runs regardless of whether the catch matched.
- `catchOf<E,M,A>(Func<E,K<M,A>>)` restricts the catch to errors whose runtime type is exactly `E` (a subtype of `Error`). `exceptionalOf<E,M,A>` matches `Exceptional`-subtype errors specifically; `expectedOf<E,M,A>` matches `Expected`-subtype errors. The distinction is load-bearing for entrypoints that need to separate infrastructure failures (`Exceptional` = unexpected exception) from domain rejections (`Expected` = anticipated failure with a code). Mixing `exceptional` and `expected` handlers on the same computation with `|` processes them in declaration order; the first predicate match wins.
- Error code matching `@catch(int code, Func<Error,K<M,A>> fail)` compares against `Error.Code`; when `Code == 0` the comparison falls back to message equality. This is documented in the XML docs on `Expected` and `Exceptional` types. An entrypoint that discriminates retryable errors by code must use non-zero codes on its domain errors to avoid false message-equality matches in the catch predicate.

[SCHEDULE_TRANSFORMER_STACKING_ORDER]:
- `Schedule.recurs(n)` is a transformer, not a schedule; it limits an infinite schedule to `n` recurrences. `Schedule.spaced(d)` and `Schedule.exponential(d)` are schedules. Composing `|` (union) and `&` (intersect) mix transformers and schedules freely: `Schedule.recurs(5) | Schedule.exponential(10*ms)` produces a finite exponential schedule of exactly 5 steps. Adding `| Schedule.maxDelay(2000*ms)` caps each delay without changing the step count because `maxDelay` is a transformer applied to the delay values, not to the count.
- `Schedule.NoDelayOnFirst` is a static `Schedule` field (a transformer) that patches the first yielded duration to zero. Composing it: `Schedule.NoDelayOnFirst | Schedule.recurs(3) | Schedule.exponential(100*ms)` gives three retries, the first immediate and the next two exponentially delayed. The `|` union is left-associative; `NoDelayOnFirst` patches the first duration of whatever schedule it combines with, so placement matters: it should be the leftmost operand.
- `Schedule.jitter(minRandom, maxRandom)` adds a random duration in the specified range to each step's delay. `Schedule.decorrelate(factor)` de-correlates durations symmetrically by applying +/- jitter that doubles the schedule length. For parallel retry from multiple entrypoints, `decorrelate` is preferable to `jitter` because it prevents thundering-herd re-collision. The practical composition for distributed retry aspects: `Schedule.recurs(5) | Schedule.exponential(50*ms) | Schedule.decorrelate() | Schedule.maxCumulativeDelay(5000*ms)`.
- `Schedule.repeat(n)` repeats the entire schedule `n` times, distinct from `Schedule.recurs(n)` which limits step count. Using `repeat(2)` on a `Schedule.recurs(3) | Schedule.exponential(100*ms)` produces 6 total steps, not 3. This distinction is non-obvious and a common source of incorrect retry count behavior.

[EXTENSION_BLOCK_COMPOSITION_CONSTRAINTS]:
- C# 14 extension blocks allow `extension(ReceiverType recv) { public ReturnType Method(...) {} }` in any file in the compilation. When the receiver is a generated `[Union]` partial type, the extension block compiles against the generated partial's public surface but cannot access `private` or `protected` members of the generated partial. The dispatch method written in an extension block therefore must invoke the generated `Switch` method through the public `K<M,A>` form — it does not get access to the internal discriminator field.
- Extension blocks are non-virtual and cannot be overridden. An extension block on a `[Union]` base type does not dispatch through any override on a case subtype. This means the extension block is the right place for the module-level dispatch entrypoint (which routes through `Switch`) but not for per-case behavior (which belongs on the `[SmartEnum]` delegate rows or the generated `Switch` arm itself).
- Multiple extension blocks for the same receiver type in different files are additive. The compiler merges them. This means the module can define one extension block with the primary dispatch entrypoint in the module's implementation file, and a separate extension block in a test file adds overloads for test-specific probing — both coexist without modifying the generated partial. The access visibility is controlled by the extension block's declared access modifier, not the receiver's.
- Generic extension blocks `extension<T>(Union<T> req) where T : ...` are valid in C# 14 and allow attaching dispatch logic to generic union types without instantiating per-type copies. The constraint on `T` can reference the module's type vocabulary to restrict which instantiations carry the dispatch method — this is the mechanism for expressing that a dispatch entrypoint only applies to unions whose type parameter satisfies a domain invariant.

[UNION_ENTRYPOINT_PRIVACY_FAILURE_MODES]:
- `UnionConstructorAccessModifier.Public` (the default) allows any call site to construct any case of the union using the generated implicit conversion from the case type. Setting `ConstructorAccessModifier = Internal` on an `internal` union type allows trusted assemblies to construct cases while preventing external consumers from injecting cases directly. This is the correct setting for a request union that crosses internal assembly boundaries but must remain sealed to external consumers.
- `FactoryMethodGeneration.Always` generates static factory methods even for unambiguous member types. The generated factory name follows `CaseName` for regular unions and `From<T1>` for ad-hoc unions. When `ConstructorAccessModifier = Private`, the implicit conversion is also private, so the only way to construct a case is through the generated factory or a hand-written wrapper. If the factory is also suppressed (`FactoryMethodGeneration.None`), the only construction path is the hand-written static factory — this is the maximum-control shape for a sealed entrypoint.
- When a regular `[Union]` case declares a property with a type that is itself a `[Union]`, the outer union's `Switch` receives a case instance that contains an inner union. Destructuring requires nesting `Switch` calls. If both unions use `NestedUnionParameterNameGeneration.Default`, the outer dispatcher parameter names include the full qualified path (`innerUnionCaseName`) and the inner dispatcher is invoked via a chained `Switch`. If `Simple` naming is used on the outer union but two inner union leaf names collide, the collision is a compile error on the outer union's generated code — not the inner union's.

[FROZEN_SET_ENTRYPOINT_GUARDS]:
- `FrozenSet<T>` (`System.Collections.Frozen`) supports `GetAlternateLookup<TAlternate>()` exactly like `FrozenDictionary`. A `FrozenSet<string>` with `StringComparer.OrdinalIgnoreCase` accepts `ReadOnlySpan<char>` lookups via its alternate lookup instance. The guard pattern for entrypoints that pre-validate a command name before dispatching: `if (!_validNames.GetAlternateLookup<ReadOnlySpan<char>>().Contains(nameSpan)) return Fin<Result>.Fail(Error.New("unknown command"))` — this is a zero-allocation pre-check against a frozen vocabulary, expressed as a guard inside the entrypoint without a separate boolean parameter.
- `FrozenSet<T>` and `FrozenDictionary<TKey,TValue>` are both constructed only when the input sequence has no duplicate keys; duplicates throw `ArgumentException` at construction time. When a dispatch table is initialized from a `[SmartEnum]` items collection, the smart enum guarantees key uniqueness, so the `FrozenDictionary.Create` call cannot throw — the smart enum's key invariant propagates through the construction path and eliminates the runtime duplicate check.

[RETRYWHILEUNTIL_PREDICATE_DISPATCH]:
- `retryWhile(K<IO,A>, Func<Error,bool>)` and `retryUntil(K<IO,A>, Func<Error,bool>)` are distinct from `retry(Schedule, K<IO,A>)`. The `While`/`Until` variants do not accept a `Schedule` — the retry fires immediately on each failure without delay. When a delay is needed alongside a predicate, the correct composition is `retry(Schedule.recurs(5) | Schedule.exponential(50*ms), computation)` followed by `retryWhile(result, pred)` — but this is not composable in one expression. The correct shape is `retryWhile(Schedule, computation, pred)` from the four-argument overload: `Prelude.retryWhile(schedule, computation, predicate)` which combines schedule and predicate into one aspect.
- `retryWhile` continues while the predicate returns `true`; `retryUntil` continues until the predicate returns `true`. Both evaluate the predicate after each failure before applying the next schedule step. The predicate receives the `Error` value, not the result — it cannot inspect a successful value to decide whether to retry. For value-dependent retry (retry while the result is below a threshold), use `repeatWhile(K<IO,A>, Func<A,bool>)` which evaluates a predicate against the success value.
