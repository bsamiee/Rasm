# [CAMPAIGN_METHOD]

The doctrinal approach behind the app-package planning corpora and every library campaign that follows — including the first-class sibling library branches `libs/typescript` and `libs/python`, each held to this same bar. Sessions load this file first: it states why these libraries are built this way, the method that produces them, the bar they are held to, and the staged work that remains before implementation. Folder-specific entry points live in each package charter (`Rasm.<Pkg>/.planning/README.md`); open work lives in `TASKLOG.md`.

[TWO_MODES]: PLANNING and IMPLEMENTATION are distinct modes and never interleave. PLANNING — the mode this file governs — does ALL the discovery (research, decompile and spike resolution, gap-hunt, app-ideation, inherited-pool binding) and produces a decision-complete, implementation-ready plan in which every change is already specified to the file, owner, row or case, and signature it lands on. Every working artifact a session builds — the gitignored research briefs and registers under `.artifacts/planning-briefs/`, the integration work-list manifest, the spike proofs under `.artifacts/spikes/` — is a PLANNING output that exists to make the plan decision-complete; none is an implementation-time activity. IMPLEMENTATION is a later, separate run that does no discovery: it dispatches workflows that apply the finalized plan — authoring or transcribing exactly what planning mapped — and finds nothing new. The test is simple: finding, researching, or building a manifest is PLANNING; dispatching workflows to apply a finished plan is IMPLEMENTATION.

[FENCE_LAW]: The unit of work is the transcription-complete CODE FENCE, never prose. For every owner, planning authors the real C# declaration — named members, union and enum cases, record fields, method signatures AND bodies wherever logic exists, and exact decompile-verified API spellings — so implementation copy-pastes it. Implementing, integrating, or ENRICHING an item means authoring or deepening that code fence: the measure of a done item is new or expanded code inside the fence, not new prose around it. An intent card, a rationale paragraph, a lone DENSITY_BAR / FEATURES / ROADMAP row, a TASKLOG closure, or a bulleted description frames a fence — it never substitutes for one. An ENRICH or NEW item that adds only prose, a card, a bare row, or a ledger registration without deepening the owner's code fence is UNDONE: a defect to repair, not a completed item. A pass that produced page insertions but no new or deepened code fence has enriched nothing. The done-measure is the diff to the code fence, not the diff to the page.

[IDEATION_LAW]: Ideation is the campaign's engine and it is AGGRESSIVE and UNANCHORED. Each turn imagines the most ambitious applications the suite must power FROM FIRST PRINCIPLES — the apps a flagship company would build on it — and never lets prior completion cap ambition: a finished, world-class owner is the FLOOR for the next idea, never the ceiling, and "it is already done" is a forbidden stopping condition. Ideation deliberately resists context-anchoring — it does not derive ideas only from what the owners already do (that merely re-describes the present); it derives the most ambitious capability frontier the domain admits and then measures the gap to it (imagine first, compare second). The central `FEATURES.md` and the suite + folder task lists are the idea reservoir that drives this: every turn pushes new high-value concepts into FEATURES, maps exactly where each folder AND the cross-cutting surface fall short of building the imagined apps, and converts every actionable shortfall into authored fences this turn (APP-IDEATION → SHORTFALL → IMPLEMENT, with [FENCE_LAW] applying to the implementation). A turn whose ideation only re-confirms existing capability — surfacing nothing the corpus does not already do — has failed its most important job; the measure of ideation is net-new ambitious capability mapped and implemented, not coverage re-stated. When the folders are near-complete, ideation and the cross-cutting concert surface become the campaign's center of gravity, never another round of per-folder enrichment on already-world-class owners.

## [1]-[THE_BAR]

These are lib-grade foundations, not app scaffolding. The bar for every package, current and future:

- World-class library code a company would adopt if it were public: the library internalizes the heavy logic, boilerplate, host quirks, native lifetimes, and failure handling so downstream apps compose capability instead of re-deriving it.
- No wrappers, no wrapper-faces, no rename adapters, no grab-bag abstractions. Every surface adds real functionality: high-value, higher-order capability built FROM the admitted packages, never a thinner face OVER them.
- Effortless complex apps without sacrifice: single entry points and unified rails reduce downstream agent overhead, and they do it by absorbing complexity internally — never by limiting tooling, hiding capability, or simplifying away power.
- Dense, optimized, polymorphic code: one owner per axis, one entrypoint family per rail, one fault family per package, total Switch dispatch, fold algebras, table-driven logic — density is anti-redundancy and polymorphic collapse, not a line budget. Flat code, parallel rails, and near-duplicate shapes are defects.
- Files are as large as the owned concern requires; size follows the concern, never a line ceiling.
- Full doctrine adherence: `docs/stacks/csharp/` is law, composed as given, never restated.

## [2]-[UNIVERSAL_PRINCIPLES]

- One unified body of functionality, not loose feature arms. A new capability is a row, case, or column on an existing axis — never a sibling surface. The pressure to add a second surface is the signal to deepen the first.
- Modality completeness from the root: every axis that varies by where code runs carries the full row set — host modalities on the AppHost host-profiles axis, process topologies in `FEATURES.md`. Apps load and offload compute, data, and UI across those rows without new surfaces.
- Parameterization as a universal need: anything that can vary is a row value, never a hardcode. A new data source, asset source, theme probe, transport, store engine, or schedule is one row on the owning axis. Host-agnostic sourcing is the default (a UI host is rhino, empty, custom, or any admitted package; the same law applies to every axis).
- External-lib maximization, truthfully: every admitted package is mined to its full capability and integrated through the owning axis; first-party companions and extensions — database extension surfaces included — are first-class rows from the root. An admitted capability no page exploits is a named gap.
- Plug-in-play composition, glove-in-hand integration: packages are complete in isolation and integrate only through the AppHost ports and the ledger seam laws (`region-map/seam-splits.md`; one-owner summary in `FEATURES.md`). Coupling beyond the ports is a defect; so is a seam re-taught instead of consumed.
- Unified telemetry/logging/observability from the source: every package contributes through the contributor and receipt-sink ports; signals are plug-in-play at composition, never bolted on per app.

## [3]-[THE_METHOD]

The repeatable campaign shape that produced this corpus, binding for every future library campaign:

1. INFRA TRUTH FIRST — root config, manifests, lockfiles, tool pins, and classification are made honest before anything else (the P0a pattern); admissions execute with restore proof; every admitted package gets a decompile-grounded catalogue page; every existing catalogue is re-verified against decompiled surfaces (phantom APIs are the named defect class). The build stays clean and centralized — versions, targets, RIDs, and settings live in `Directory.Packages.props` and `Directory.Build.props`, project files stay versionless, and `.csproj` edits are admitted to remove dead RIDs and native assets or to centralize a setting (a single-platform target drops every off-platform native asset). Admission is bleeding-edge: every concern is served by the most advanced, modern, strongest external library available at newest stable, verify-before-pin — retaining a legacy package where a stronger modern equivalent exists is a named defect (the Xbim-to-GeometryGym class of replacement). A dedicated agent subset works only the package manifest and each target folder's `.api/` catalogues, so every admitted package is fully integrated and carries a full, deep `.api/` catalogue with zero member gaps.
2. RESEARCH RESERVOIR — design lanes, adversarial verifiers, gap-hunt lenses, deep-read lanes, decompile probes, and web currency checks run BEFORE authoring; conflicts resolve by the suite standard's binding precedence.
3. IDEATION BEFORE DESIGN — exhaustive concept enumeration (isolation + in-concert, across every topology and domain) drives the pages; imagine everything, then prune bloat ruthlessly; every survivor rides an axis row, and the concert concepts land in `FEATURES.md`. The per-package refinement blueprint converts ideation into per-page cluster plans, binding corrections, density guidance, and anti-bloat vetoes.
4. AUTHOR-THEN-REPAIR — parallel page authors transcribe the blueprint under the suite standard (`README.md` here); consolidated review-and-repair waves find AND fix in one pass; the orchestrator is the only ledger writer.
5. CROSS-FOLDER COHESION — ledger consolidation (owner symbols, seam splits, adjudications), then a final deep pass per folder: under-utilization hunt, heavy code critique on every fence, doc integrity, research items answerable by decompile get answered, zero-guessing validation.
6. EVERYTHING DURABLE — working material persists under `.artifacts/planning-briefs/` (gitignored); law lands in charters, pages, the ledger, atlases, and ROADMAPs; versions live only in `Directory.Packages.props`; the charter ADMISSIONS_RECORD owns the package→page→catalogue→status map, never the pin. ARCHITECTURE.md draws the planned non-flat implementation source tree (namespaced sub-folders), one leaf per BUILD_ORDER file annotated with its transcription owners; the flat BUILD_ORDER file list is the tree's source, never a duplicate rendering. The `.planning/` corpus stays flat.

## [4]-[STAGED_WORK_BEFORE_IMPLEMENTATION]

The sessions that precede implementation, in order; each is its own campaign run with the method above:

1. PER-FOLDER EXTREME-DETAIL DEEPENING — each package in isolation: re-read every catalogue against every page, push capabilities/features/considerations further (the AppUi mined-no-gaps treatment as the bar for all), close TASKLOG rows owned by the folder, deepen thin catalogue pages (named in TASKLOG), and re-run the code critique until any world-class app is buildable from the pages alone.
2. COMPREHENSIVE APP IDEATION — a dedicated ideation campaign across all dimensionalities (local, web-linked, data-sourcing, domain-specific, multi-process) producing new concept rows integrated root-up into pages and atlases; advanced complexity is the target, always with the dual mandate: universal standalone usage AND first-class rhino/gh2 operation.
3. SIBLING LIBRARY BRANCHES — `libs/typescript` (entry: `libs/typescript/.planning/README.md`) and `libs/python` (entry: `libs/python/.planning/README.md`) are first-class sibling branches, not afterthoughts: each is a multi-package branch made lib-grade in isolation with its own central manifest (`pnpm-workspace.yaml`; the single root `pyproject.toml`), catalogue extraction, ideation, pages, charter, and ledger, and each runs this same campaign method on its own cadence with its own workflow passes. The branches are agnostic, universal, professional libraries in their own right — never thin couplings to the C# packages — but aligned hand-in-glove where a shared concern crosses the boundary. TypeScript integrates only through the wire contracts; Python integrates only through the companion and offline seams it owns — the two-hop geometry/IFC tessellation rail (IFC -> IfcOpenShell -> GLB), data and artifact bundle shapes, and C# graduation evidence — never C# interiors, host lifecycles, durable stores, or TypeScript UI state. Each branch keeps the architecture it would have if designed in isolation: re-derive its package/folder/domain topology from first principles every campaign, never anchored to a stale layout. Focus is ADDITIVE, never subtractive: C# stays primary and each sibling branch receives its own dedicated concurrent allocation on top — never effort diverted from the C# packages. Each branch is a complete professional library adoptable in ANY future monorepo, every concern owned ground-up, not a support layer for the C# packages: TypeScript owns its full web/edge platform — infrastructure + provisioning (deploy/Pulumi), the web app surface, the backend/node tier, its OWN store/Postgres/SQL access, and a web UI/UX domain entirely its own (a browser UI/UX concern distinct in kind from the C# Avalonia AppUi); Python owns a full science/compute/data stack — compute, data, geometry, and IFC pipelines driven root-up into a real pipeline that is an alternative to OR complement of the rhino/gh2 + C# stack, pushed far past surface level. The C# planning corpus GUIDES which logical packages each branch needs and lands every cross-boundary BIG idea at its right touchpoint the same turn, but the branches own those concerns natively, never as tacked-on mirrors of the C# owners. Each branch runs its OWN aggressive-ideation and keeps its OWN additive `FEATURES.md`/`TASKLOG.md`/next-loop pool (informed by the C# corpus, never coupled to it); the closeout distillation is tri-language and unified (central C# driving + python + typescript additive). Driving the weaker-language doctrines to match-or-exceed `docs/stacks/csharp` is an OPTIONAL operator-requested side-track, run per session only when asked.
4. FINAL ARCHITECTURE PASS — the closing pre-implementation session: per-package file plan finalized against the charter BUILD_ORDER, logic-flow walkthrough of every entrypoint rail end-to-end, cross-package boot/drain/wire choreography rehearsed on paper, implementation-start gates executed (bridge spikes, native probes), and the proof-gate matrix confirmed green-able; the per-package source tree in ARCHITECTURE.md is refactored each session to match the finalized BUILD_ORDER, so the tree is the live file plan, not a one-time sketch. Exit: implementation begins with transcription only.

## [5]-[QUALITY_MACHINERY]

- Every page graded cold per the suite standard's review law; research sections carry unresolved fact statements only — the executable proof rails live once in each charter PROOF_GATES.
- The ledger (`region-map/`) is the single registry of owner symbols and seam laws; consult before naming any cross-package type; the orchestrating session is its only writer.
- Owner state lives in exactly one surface: the charter DENSITY_BAR `[STATE]` column. FEATURES, GAP_LEDGER, pages, and the gitignored register carry no state marker; FEATURES and ROADMAP route to the owner for state. A state value duplicated across surfaces is the named drift defect — one owner, one state cell, one edit site.
- Bleeding-edge posture is verified, not assumed: newest stable of every package at admission time with restore proof; performance laws (FrozenDictionary/SearchValues/span formatting/zero-alloc capsules/GC posture per profile) are page rows that specs encode.
- Compute/threading/tensors are first-class architecture: one CPU budget across lane readers, ORT pools, and parallel partitions; allocation classes with receipts; solve-path isolation; benchmark claims gated by host fingerprint.
- Proof gates per charter: locked restore, api doctor/resolve, static plan/build, test --target, bridge scenarios for host seams, local mermaid render.
- Spike resolution ([8]): every `SPIKE` is driven to `FINALIZED` during planning by tier-1 decompile and tier-2 throwaway-harness proofs against the real packages, native libraries, and servers; only live-integrated-host probes remain open for the implementation session.

## [6]-[SESSION_ENTRY_PROTOCOL]

1. Read this file, then `TASKLOG.md` and the target folder's `TASKLOG.md`, then the target folder's charter (`Rasm.<Pkg>/.planning/README.md`) end-to-end.
2. Read the suite standard and ledger; doctrine (`docs/stacks/csharp/`) arrives settled.
3. Orchestrate with the method in [3]: ambitious synthesis is opus producer→red-team→merge (never a single agent), with parallel agents for reading, authoring, and repair — per-folder lanes parallel across folders and sequential within each. The orchestrator is the ledger's sole writer and the sole committer: after each phase gate it commits a scoped path set with explicit `git add -- <path>` of that gate's enumerated planning files only, never `git add -A` and never a foreign path; concurrent working-tree edits outside the staged set stay untouched because agents and the operator edit concurrently. Workflow agents never run git.
4. Never re-derive: catalogues are decompile truth; pages are decision-complete; the binding precedence resolves conflicts; anything genuinely new lands as rows through the same machinery.

## [7]-[DEPTH_AND_GAP_HUNT]

The standing enrichment lens binding every per-folder deepening and enrichment pass, beyond the session's starting register. Each session reads every folder in full depth and grades each owner against a world-class library that must power many diverse, ambitious applications at once; a surface-level owner, an under-exploited admitted package, or a missing capability category is a defect, not an acceptable baseline.

- World-class-or-defect: grade every owner at the bar of "nothing a flagship app dev would hand-roll," not "present." For each page, name what an ambitious app would still build by hand from this folder's domain — a query shape, a store engine, a codec, a data type, an import/export path, a chart kind, a transport, a streaming or chunking path, a viewport binding, a data-source adapter — and own it now. An owner that stops at table-stakes is a deepening target.
- Categorical gap-hunt, not row top-up: the hunt finds whole concerns the folder overlooks, not only rows on existing axes. Persistence selects ANY store — Postgres, SQLite, DuckDB, embedded, remote — each fully realized with import/export, the full data-type surface, and streaming/chunking, no engine half-built; Compute streams and chunks at performance; every folder connects to remote, web, and backend so data flows from a source and back to a source; typography, charts, and visuals reach world-class breadth, power, and live-data feeding; rhino/gh2 viewport and panel integration pulls live data from any source. A missing category is a new owned concern, surfaced and integrated this session.
- External-lib maximization: every admitted package is mined to its deepest useful capability from multiple angles — operator, combinator, extension point, generated surface, zero-alloc and performance path — and integrated at the deepest form the package reaches; an admitted capability no owner exploits is a named defect.
- Topology completeness: every modality an app can take — backend, server, remote, on-host, companion, sidecar, standalone, web-fed, hub, paired — is equally and fully realized through the host-modality row set, never one face built and the rest stubbed; a capability present in-process but absent companion, sidecar, or remote is a gap.
- Root-up integration, never flat spam: a found gap lands by deepening the canonical owner with a row, case, column, fold, or policy value, or by a NEW design doc when a genuinely new owned concern warrants one — authored ground-up, woven into the densest owner, integrated where it belongs, never a loose arm, a parallel surface, or a tacked-on helper. Adding a spec page is welcome; adding flat or loose code is the defect. Every addition resolves through `docs/stacks/csharp/` in full and the suite standard.
- Additive and recurring: the gap-hunt runs every session as design pressure on top of the starting register, and each session closes by distilling the next frontier — new capability, feature, and category lists per file and folder — so the corpus deepens every cycle and never plateaus.

## [8]-[SPIKE_RESOLUTION]

A `SPIKE` owner is a transcription-complete fence whose runtime correctness is gated on a probe. Spikes are driven to `FINALIZED` during planning, never deferred to implementation, by proving each probe against the real external packages, native libraries, servers, and runtimes in throwaway scaffolding, then folding the verified spelling and behavior back into the fence and flipping its `[STATE]`. The design corpus stays the deliverable; the proof harness is disposable. When implementation begins, every closable spike is already finalized and transcribes by copy-paste.

The closure tiers, cheapest first; a spike rides the lowest tier that proves it.
1. Decompile and reflection — `uv run python -m tools.assay api query` and the `.api` catalogues confirm member existence and exact spelling with no execution, closing every uncatalogued-member RESEARCH row: the placeholder spelling in the fence is replaced with the verified one and the dependent owner row leaves `SPIKE`.
2. Throwaway harness against the real package — a minimal standalone proof under the gitignored temp root `.artifacts/spikes/<spike-id>/`, referencing the admitted package versions from the manifests, compiled and run against the real runtime to confirm behavior: a native library `dlopen` on the target RID, a live Postgres or DuckDB response against a throwaway database, an ONNX session run, an Avalonia headless render, a serializer round-trip, a byte-identity or codec check. The harness proves exactly the named probe; its verified output — member shapes, return contracts, error taxonomy, performance class — folds into the fence and card, and the temp dir is deleted. This tier closes every native, server, library, and headless spike in-session.
3. Live integrated host — only probes that strictly require the running integrated host (Generic Host boot and unload inside the live Rhino or GH2 plugin ALC, NSView embedding, in-process host-seam behavior) stay `SPIKE` until the host exists, and even these are de-risked standalone first, so only the final in-host confirmation remains for implementation. Any probe requiring interactive OS authentication is tier-3 by the same rule: a macOS keychain item read (`SecItem*` / Security.framework), DPAPI, a biometric or credential-store read, or a signal-to-self (SIGHUP) delivery all prompt the operator and never run unattended — their owners are proven by tier-1 decompile member-shape ONLY (the P/Invoke and Security.framework spellings) and stay `SPIKE` until the integrated host.

- Every closed spike updates the page (RESEARCH row removed or member confirmed), the `[STATE]` column (`SPIKE` to `FINALIZED`), and the charter `GAP_LEDGER` and ROADMAP; a probe that reveals a runtime divergence reshapes the fence root-up — the spike has done its job by surfacing the unknown before it cost the system.
- The throwaway harness never enters package source, the planning pages, or the region ledger; it lives only under `.artifacts/spikes/` (gitignored), is referenced by no durable artifact, and is deleted once its result is folded in. Versions stay in `Directory.Packages.props`; the harness reads them there and never re-declares them, and the ADMISSIONS_RECORD carries no pin.
- [WARNING] A throwaway harness NEVER triggers an OS credential, keychain, biometric, or password prompt: an unsigned harness reading a keychain item it wrote raises the macOS login-keychain dialog and spams the operator with a password request that has no answer. Keychain, secret-store, DPAPI, biometric, and signal-delivery routes are proven by tier-1 decompile member-shape only and stay tier-3 (above). Tier-2 provisioning is clean and additive: public container images only (never `docker login`), operator containers and volumes are sacrosanct (never stopped or destroyed), and the provisioning architecture is full, powerful, and current — the complete server, extension, and native surface (PostgreSQL at newest major with its full extension set, the cache and object-store backends, and the native runtimes) is provisioned so every server, native, and headless spike is reachable, then torn down to its pre-run state except where a later phase reuses the stack.
- A session closes with every tier-1 and tier-2 spike finalized; the residual `SPIKE` set is exactly the tier-3 live-host probes, each named in the page RESEARCH cluster and the ROADMAP close-out list. The `SPIKE` count at session exit measures only host-bridge surface, never unproven library, native, or server capability.

## [9]-[THE_LOOP]

The campaign is a repeatable cycle, not a one-shot; each session is one turn and the corpus deepens every turn. Every step of THE_LOOP is PLANNING work: the research reservoir, inherited-pool integration, gap-hunt, and spike resolution find, map, and FINALIZE the complete work-set to the extreme bar during planning — never deferred to implementation. Implementation is a separate session that transcribes the decision-complete plan; its own first step executes what planning fully mapped and discovers nothing new. An unknown that surfaces during planning is resolved in planning, never parked for implementation.
1. Consume the prior session's distillation as the binding, decision-complete starting work-list: the refreshed concept/feature/capability pool, the per-file and per-folder next-loop capability and task lists, every folder `TASKLOG.md` and the suite `TASKLOG.md`, and the `FEATURES.md` concept rows the prior session emitted. Integrate this entire inherited pool in full this turn before ideating anything new; the loop never restarts from a blank slate or from random ideas, and a pre-existing finding left unintegrated is a defect, not a backlog item.
2. Deepen and gap-hunt per folder ([7]) in one full-depth read that integrates the inherited work-list AND runs the categorical world-class gap-hunt, surfacing and owning every missing category and every surface-level owner root-up. Every pass deep-reads every admitted package the folder declares (its `.api` catalogues) and integrates the full useful capability at depth — operator, combinator, extension point, generated surface, zero-alloc and performance path; a surface-level use of an admitted package is a named defect, and every capability a folder exposes is rich and powerful, never table-stakes. Every addition resolves through all of `docs/stacks/csharp/` (README + the core docs + the relevant `domain/*`).
3. Resolve every open unknown in-session through parallel research-and-artifact tracks opened at session start — one per open question, spike, uncatalogued surface, version or compatibility unknown, and investigation item: decompile and reflection ([8].1), throwaway-harness probes against the real packages, native libraries, servers, and runtimes ([8].2), package-capability deep-reads, and currency/compatibility dossiers. Each track writes a gitignored brief under `.artifacts/planning-briefs/` whose verified result folds back to FINALIZE the dependent fence and flip its `[STATE]` within the same session; an admission-gated probe runs inside the gate that admits its pin. Research generated in a session is implemented in that session — never punted to a later turn — and the residual `SPIKE` set at exit is exactly the tier-3 live-host and hardware probes ([8]).
4. Cohere ground-up within each package AND cross-folder without coupling, run as three ordered HARDENED workflows — each producer -> red-team -> merge in its own right, never a single pass: (a) the parallel per-folder upgrade (step 2 above, one folder per lane); then (b) an ALIGNMENT/REFINEMENT pass that reconciles every folder's owners, seams, charter DENSITY_BAR state, ARCHITECTURE source tree (re-derived from BUILD_ORDER), lean FEATURES, and folder TASKLOG against the ledger and propagates any cross-package signature change (a port/receipt arity change ripples to every sibling caller in this pass); then (c) a CROSS-CUTTING pass that authors the suite-level surface — the CROSS_PACKAGE_LAWS, the suite FEATURES concert atlas, and the cross-folder suite TASKLOG — and verifies anchor integrity and single-state-surface across the whole suite. The cross-folder suite TASKLOG is authored IN the cross-cutting pass (after per-folder alignment), never bolted on; one owner per axis, consumers reference and never re-mint, the ledger records every seam-split; clean drift, collapse duplication, never flatten a sanctioned altitude-split. Reconcile document architecture each turn: ARCHITECTURE source tree re-derived from BUILD_ORDER; FEATURES held lean (feature + owner + page#cluster, no state, no case counts, no owner-symbol chains); owner state edited only on the DENSITY_BAR; ADMISSIONS_RECORD carries page→catalogue→status, never a version; every page#cluster anchor resolves to a live heading; per-folder open work lives in the folder TASKLOG, cross-folder work in the suite TASKLOG, closed rows in neither. A version pin, a size cap, a phantom anchor, or a second state surface found during the turn is repaired before distillation.
5. Distill forward in two ordered movements. First — the integration gate — verify that 100% of this turn's contained work is realized as a transcription-complete fence (not a mention) and driven to `FINALIZED`: every inherited-pool item from step 1, every gap-hunt finding from step 2, and every in-session research result from step 3. Integration completeness is the gate; a single unintegrated finding fails the turn and is repaired before the second movement. Second — and only then — run the closeout deep-read plus comprehensive app-driven ideation: name the most ambitious apps the suite must power, identify in the same pass exactly where each folder and the cross-cutting surface fall short of building them, and emit a refreshed, decision-complete pool of new concepts, features, capabilities, and per-file and per-folder task lists — to the suite `libs/csharp/.planning/` or any of the five folders — as the next session's starting work-list. That forward-emitted ideation pool is the ONLY thing a session defers; every shortfall it can act on this turn lands on an existing owner or a justified new page now, never on a ledger or a future to-do, and only a genuinely new categorical concern that would warrant a new package is surfaced as such. The closeout deep-read is NOT new-concepts-only — it is a two-sided enrichment audit. (i) Grade EVERY EXISTING owner against the world-class bar ([7]) by deep-reading every `.api/` catalogue and every admitted external library end to end, and name each owner that is surface-level or under-exploits an admitted capability — a UI owner not maximizing Skia for an existing visual, a Postgres extension not fully integrated, an admitted store not richly realized, a compute pipeline not optimized or not collapsed to the densest surface, a fence that adds rows by hand where a fold/table/generated owner is available. Pushing an existing body deeper (more capability woven into the same owner) is as much a closeout target as a new concept. (ii) Audit EVERY design-doc fence against ALL of `docs/stacks/csharp/` (README + the core docs + every `domain/*`) for doctrine conformance and surface-collapse opportunities — a fence that violates a code law, or that could collapse parallel types/enums/methods into a denser `[Union]`/`SmartEnum`/fold/data-table, is a named target. Every such existing-owner enrichment and doctrine-conformance target joins the refreshed pool as a detailed, owner-anchored next-loop task. Enrichment is universal and root-up: every existing file is pushed, deep-package-leveraged, and aligned every pass — never left at table-stakes — and its telemetry, diagnostics, and metrics deepen with every capability it gains. The distillation is TRI-LANGUAGE and UNIFIED: the closeout emits the central C# next-loop pool (the driving `FEATURES.md` + suite `TASKLOG.md`) AND a separate ADDITIVE next-loop pool for each sibling branch (`libs/python/.planning` and `libs/typescript/.planning` — their own `FEATURES.md` + `TASKLOG.md`), so the next session starts rich across all three languages at once. Each branch's aggressive-ideation reads the C# corpus for cross-pollination but generates language-owned additive concepts (new apps, new packages, wider branch improvement), never a mirror of the C# owners. Driving the weaker-language doctrines (`docs/stacks/python`, `docs/stacks/typescript`) to match-or-exceed `docs/stacks/csharp` is an OPTIONAL operator-requested side-track (documented in the session-entry prompt), run per session only when asked — not a standing step.

The next session begins at step 1 against that fresh distillation, and `campaign-method.md` records the loop so every turn launches from the same protocol.
