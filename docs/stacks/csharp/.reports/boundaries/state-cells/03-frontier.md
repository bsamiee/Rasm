# State-Cell Boundary Law — Frontier

[ATOM_VOLATILE_BOX_MEMORY_MODEL]:
- `Atom<A>.value` is declared `private volatile object`. All reads carry acquire-fence semantics; all CAS operations via `Interlocked.CompareExchange` carry full memory barriers. For value types, `Box<A>` wraps the struct; `Box<A>.GetValue` is a static IL-emitted delegate (`DynamicMethod`) that field-loads directly from the box object. When `ILCapability.Available` is false (NativeAOT), the fallback is `(object x) => ((Box<A>)x).Value` — correct but adds a cast with no correctness hazard.
- `Memo<A>.state` is `internal int` with no `volatile` annotation. The `Interlocked.CompareExchange` calls in `GetValue` and `Reset` carry implicit full memory barriers, so state machine transitions are coherent. However, the hot-path `if (state != 2)` guard in the `Value` property getter is a plain `int` read. On ARM64, a thread could observe `state == 2` before observing the write to `value` that occurred on the winning CAS thread, reading a stale default. The safe form is `Volatile.Read(ref state) != 2` in the getter, or replacement with `Atom<Option<A>>` which carries the full volatile guarantee on both state and payload simultaneously.

[ENVIO_DISPOSAL_LAYERING_GAPS]:
- `EnvIO.DisposeResources()` checks `(Own & 2) == 2` before touching resources — an `EnvIO` that does not own its resource scope silently skips resource release even when `Dispose()` is called. This is correct for scopes that borrow a parent's resources; `Own bit 2` tracks allocation, not logical ownership, and cannot be selectively suppressed after `New` returns.
- `EnvIO.DisposeResources()` calls `EnvIO.New()` — a fully owned fresh `EnvIO` with `Own=3` — then passes it to `Resources.DisposeU(envIO)`. `DisposeU` constructs a second inline `EnvIO` with `Own=0` using `CancellationToken.None` to actually run releases. The fresh `EnvIO` from `New()` has `Dispose()` never called on it within `DisposeResources()`; its `CancellationTokenSource` and `Resources` scope are abandoned to GC finalization. This is a minor resource leak per disposal — not a correctness hazard but a pressure point for high-frequency disposal paths.
- `Resources.DisposeU(EnvIO envIO)` passes `envIO.Resources` (the outer caller's resources) into the release env. Custom `IO<Unit>` release effects that call `use(...)` or `Acquire(...)` inside them register new resources into the CALLER's resource scope, not a local one. A teardown effect that acquires a temporary resource must use a manually scoped inner `IO.Bracket` to isolate that acquisition.

[IOUNINTERRUPTIBLE_ASYNC_LEAK]:
- `IOUninterruptibleAsync<A,B>.Invoke` is an `async ValueTask<IO<B>>` method that executes `A arg = await Fa; LocalEnv.Dispose(); return +Next(arg)` with no `try/finally`. If `Fa` faults, the awaiter propagates the exception before `LocalEnv.Dispose()` is called. The `LocalEnv` holds an isolated `EnvIO` constructed with `CancellationToken.None` and a fresh `CancellationTokenSource` (Own bit 1 set). On exception, the `CancellationTokenSource` inside `LocalEnv` is abandoned until GC finalization. For high-frequency fault paths, this accumulates live `CancellationTokenSource` objects. The fix wraps `await Fa` in `try { A arg = await Fa; LocalEnv.Dispose(); return +Next(arg); } catch { LocalEnv.Dispose(); throw; }`.

[IOLOCAL_EXCEPTION_PROPAGATION_CONTRACT]:
- `IOLocal<X,A>.Invoke` uses `using EnvIO local = MapEnvIO(envIO)` with no explicit `try/catch`. When `Operation.RunAsync(local)` throws, the compiler-generated `finally` calls `local.Dispose()` — both resources and non-resources are released — before the exception propagates. `Bracket()` therefore guarantees cleanup on any failure path.
- `IOLocalOnFailOnly<X,A>.Invoke` manually separates the cleanup: `catch { local.DisposeResources(); throw; }` followed by `finally { local.DisposeNonResources(); }`. On success, neither `DisposeResources` nor `DisposeNonResources` run in the local scope's guarded body — the `await Operation.RunAsync(local)` returns normally and `Next(result)` is returned without disposal. The caller takes ownership of `local`'s resources by the resources being in the parent scope via `LocalResources`, which shares the parent `Source`.

[ATOMHASHMAP_FACTORY_MULTI_INVOKE_SURFACE]:
- `AtomHashMap<K,V>.FindOrAdd(K, Func<V> None)` calls `Items.FindOrAddWithLog(key, None)` inside a CAS spin loop. If two threads both observe `key` absent, both call `None()` inside the loop body, only one wins the CAS, the loser retries and now finds the key present. The `None` factory therefore executes at least once per concurrent first-insert per key. Any factory that acquires resources will leak the handle produced by the losing thread. The safe surface: `TryAdd(key, new Lazy<V>(factory))` then read `.Value` — `Lazy<T>` with `LazyThreadSafetyMode.ExecutionAndPublication` ensures the factory runs exactly once even if multiple threads race before it completes.
- `AtomHashMap<K,V>.SwapKey(K, Func<V,V>)` returns `unit` immediately when the key is absent — the swap function is never called and the map is unchanged. For conditional insertion semantics, use `SwapKey(key, Func<Option<V>, Option<V>>)` which receives `None` for absent keys.
- `AtomHashMap.Swap(Func<TrackingHashMap<K,V>, TrackingHashMap<K,V>>)` checks reference identity of the swap function's result before attempting the CAS — if the swap function returns the SAME underlying trie reference, the CAS is skipped entirely and no `Change` event fires. A swap function that conditionally modifies should return the original `TrackingHashMap` when no change is desired, not a structurally equivalent new one.

[FORK_CLEANUP_AND_RESOURCE_CHAIN_HAZARDS]:
- `IO<A>.Fork` creates a `CleanUp` record holding `(CancellationTokenSource tsrc, CancellationTokenRegistration reg)`. `CleanUp.Dispose()` uses `Interlocked.Exchange(ref disposed, 1) == 0` as an idempotency guard, then disposes `tsrc` and `reg` independently. The fork task's `finally` block calls `resources.Dispose()` (the child resource scope) and `cleanup.Dispose()`. The `Cancel` IO only cancels the source; it does not call `cleanup.Dispose()` — `Dispose` is called exclusively from the task's `finally`. The `CancellationTokenRegistration` (the parent→child link) is therefore disposed in the task's `finally` regardless of whether cancellation was requested externally or via timeout.
- The fork passes `EnvIO.New(resources, token, tsrc, env.SyncContext)` to `RunAsync` — all four parameters are non-null, so `Own = 0`. Neither the `CancellationTokenSource` nor the `Resources` are owned by this env; `Dispose()` on this env is a no-op. Cleanup responsibility falls entirely on the `finally` block's explicit `resources.Dispose()` and `cleanup.Dispose()` calls.

[REPEAT_STACK_GROWTH_PRESSURE]:
- `IO<A>.RepeatUntil(Func<A,bool> predicate)` — the no-schedule overload — is defined as `Bracket().Bind(v => !predicate(v) ? RepeatUntil(predicate) : IO.pure(v))`. Each iteration adds one `Bind` closure to the pending continuation chain in the trampoline-style interpreter's managed `Seq<StackItem>` stack. `Bracket()` calls `WithEnv(env => env.LocalResources)` which produces an async-path `IOLocal` node, so each iteration suspends and resumes via the managed continuation stack. Use this form only for predicates known to fire within a bounded number of iterations.
- `IO<A>.RepeatUntil(Schedule schedule, Func<A,bool> predicate)` — the scheduled overload — avoids the recursive bind. It uses a local `go(Iterator<Duration> iter, A? value)` function that calls `IO.yieldFor(head).Bind(_ => Bracket().Bind(v => !predicate(v) ? go(tail, v) : IO.pure(v)))` at each schedule step. `IO.yieldFor` injects the schedule delay before each iteration. For long-running polling loops, use this overload to stay in the schedule-driven path.

[IO_POST_ALLOCATION_COST]:
- `IO<A>.Post()` allocates an `AutoResetEvent` per call — a kernel-backed synchronization primitive — to bridge the `SynchronizationContext.Post` callback back to the awaiting thread. An `AutoResetEvent` is `IDisposable`; the `Post()` implementation disposes it correctly in a `finally` block, but there is no pooling. For marshaling high-frequency operations to a UI or platform thread, batch work via a channel whose consumer runs on the target context and returns aggregated results, calling `Post()` at the boundary only once per batch.

[ENVIO_SYNC_CONTEXT_CAPTURE]:
- `EnvIO.New()` with `syncContext = null` captures `SynchronizationContext.Current` at the moment `New()` is called — not at construction of the `IO` tree and not at `Run()` time. If `IO.RunAsync()` (no-arg form) is called from a thread with a non-null `SynchronizationContext`, that context propagates to all `Post()` calls inside the operation. A state cell that must always marshal updates to a specific context must inject the context explicitly via `EnvIO.New(syncContext: theContext)`, not rely on ambient capture.
- `EnvIO.LocalSyncContext` creates a new `EnvIO` with `SynchronizationContext.Current` at the point of the property access, sharing the existing `Resources` and `Source`. The returned `EnvIO` must be used immediately — deferring it into a lazy `IO` node means the context is captured at node creation, not at evaluation, which may be on a different thread.

[RESOURCES_RELEASEALL_TRIE_SWEEP]:
- `Resources.ReleaseAll()` runs via an `IO.lift` lambda that calls `resources.Swap(r => { foreach (item in r) item.Value.Release().Run(envIO); return empty; })`. The swap function runs inside `AtomHashMap.Swap`'s CAS loop. If the CAS fails (concurrent mutation), the entire sweep — including already-executed `Release().Run(envIO)` calls for items in the previous snapshot — re-executes on the retried snapshot. Release effects must be idempotent, or `ReleaseAll` can double-release on CAS contention. The safe form for non-idempotent teardown is `Release<A>(value)` per key, which removes the key before calling the release effect, preventing re-entry on CAS retry.

[LIFECYCLE_FAULT_CELL_WAITING_MECHANISM]:
- A three-state initialization cell (`Initializing`, `Initialized`, `InitializationFailed`) encoded as `Atom<InitState>` with `validator = _ => true` requires waiters that see `Initializing` to block until the winner transitions. A `TaskCompletionSource<InitState>` created before the first `Swap` and signaled in the winning initializer's completion path provides an awaitable seam. The cell's swap function returns the new state; the initializer calls `tcs.TrySetResult(newState)` immediately after the committed `Initialized` swap. Waiters read `atom.Value` first; on `Initializing`, they `await tcs.Task`. If the winner commits before the waiter checks the atom, the waiter reads `Initialized` directly; if after, the waiter blocks on `tcs.Task`. The `TaskCompletionSource` must be process-static alongside the atom — both initialized before any concurrent caller can reach the gate.
- A lifecycle state machine encoded as `Atom<LifecycleState>` with a validator `v => v.IsValidSuccessor(atom.Value)` has a critical flaw: the validator receives the CANDIDATE next state, not the current state. The current state must be read inside the CAS loop. The correct validator for a state machine is `_ => true`; transition constraints belong in the swap function: `curr => curr.IsValidPredecessor(next) ? next : curr`.

[GENERATED_REGISTRY_CHANGE_SUBSCRIPTION_COST]:
- `AtomHashMap<K,V>.Change` subscription affects the path taken by operations that gate on `this.Change != null`. `SetItem` checks this gate and switches between `SetItemWithLog` (allocates a `Change<V>` diff record) and `SetItem` (no allocation). `FindOrAdd` always calls `FindOrAddWithLog` regardless of subscription — the diff allocation is always present for this operation. For a high-frequency registry cell where `FindOrAdd` dominates, `Change` subscription has no allocation impact on that path; the performance argument against subscription applies to `SetItem`-dominant workloads.
- `AnnounceChange(prev, current, key, change)` checks `change != null && change.HasChanged` before calling `this.Change?.Invoke(...)`. For no-op swaps where `SetItemWithLog` still produces a `Change<V>` record before checking `HasChanged`, the allocation occurs before the gate — a pre-materialized-allocation hazard for idempotent swap functions on subscribed maps where `SetItem` is the dominant operation.

[CANCELLATION_LINKED_SOURCE_DISPOSAL_CONTRACT]:
- `CancellationTokenSource.CreateLinkedTokenSource(CancellationToken[])` (the params overload) and `CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` (added in .NET 9) both register callbacks on all parent tokens at construction time. Each parent registration holds a delegate reference back to the linked source. If the linked source is not disposed, those registrations remain live until each parent source fires or is disposed. For bounded-lifetime boundary cells that create linked sources per-operation, `Dispose()` must be called in every path — success, failure, and timeout — or parent cancellation callbacks accumulate as garbage.
- `EnvIO.New` uses `token.Register(() => source.Cancel())` rather than `CreateLinkedTokenSource`, which registers a single callback without allocating a linked source; this is cheaper but does not expose a `Token` derived from both inputs — the `EnvIO.Source.Token` is a new token that is cancelled transitively by the registration. For hand-rolled linked sources outside `EnvIO`, the pattern is a `using CancellationTokenSource linked = CancellationTokenSource.CreateLinkedTokenSource(parent, localToken)` block scoped to the operation.
