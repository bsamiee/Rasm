# State-Cell Boundary Law â€” Edge

[STM_REF_MULTI_CELL_COORDINATION]:
- `Ref<T>` is the correct primitive when two or more state cells must transition atomically. `Atom<A>` CAS-loops on a single cell; two independent atoms cannot be CAS-swapped together without a window where one has committed and the other has not. `Ref<T>` inside `STM.DoTransaction` / `sync(...)` provides snapshot isolation across any number of refs with automatic retry on conflict â€” the unit of atomicity is the transaction body, not a single cell.
- `STM.DoTransaction<T>(Func<T>, Isolation)` nests: when a transaction is already running on the current thread, the inner `DoTransaction` call participates in the outer transaction rather than starting a new one. `STM.TransactionId` returns the currently running transaction id, non-null only when called from within a live transaction body. A state cell that conditionally starts a transaction only when not already inside one can read `STM.TransactionId` as a guard.
- `Isolation.Snapshot` triggers a retry only when a ref written inside the transaction was also written externally between the transaction's read-point and its commit. A ref that is read but not written is not tracked for conflict. `Isolation.Serialisable` tracks both reads and writes; any external write to any ref touched causes retry. For boundary lifecycle cells where the correctness invariant is "the set of open connections did not change while I was deciding to add one", `Serialisable` is required; `Snapshot` permits a torn read.
- `commute(Ref<T>, Func<T,T>)` re-runs the function with the most recently committed value at commit time rather than with the in-transaction snapshot. The in-transaction read returns the snapshot value for consistency within the body; the commit write uses the live value. Use `commute` only for operations where last-one-in-wins is acceptable â€” append-only counters and idempotent flags. For boundary cells whose value determines whether an operation is permitted, `swap(ref, f)` (not `commute`) enforces the snapshot-at-read invariant.
- `Ref<T>.Value` setter called outside a transaction wraps itself in an implicit `Snapshot` transaction. Two consecutive outside-transaction assignments are two separate implicit transactions and can interleave with any concurrent explicit transaction between them.

[STM_ISOLATION_PRESSURE_FOR_LIFECYCLE_CELLS]:
- A lifecycle state machine encoded as multiple `Ref<T>` cells â€” one for state, one for in-flight count, one for drain deadline â€” inside a single `Serialisable` transaction is the correct shape for drain-with-count correctness. The `Draining` transition sets the state ref and reads the in-flight count atomically; any concurrent operation that increments the in-flight count will conflict with the state-read, triggering a retry. Either the drain commits before any in-flight increment, or the in-flight increment commits and the drain retries with the updated count.
- `STM.DoTransaction<T>(Eff<RT, T>, Isolation)` accepts an `Eff<RT, T>` effect as the transaction body. If the transaction retries, the effect's `IO` subtree is re-evaluated. Any effect with external side effects (logging, metrics, I/O) re-executes on each retry. Transaction bodies must be confined to pure `Ref` reads and writes; side-effectful `IO` must run in a continuation after the transaction commits using the committed values.
- `CommuteRef<T>` is returned by `commute(Ref<T>, f)`. It cannot be read back within the same transaction â€” its current value is undefined until commit. A transaction body that reads a `CommuteRef<T>` after commuting it observes the pre-commute snapshot value, not the commuted result. Boundary cells that must read-then-write within one transaction should use `swap`, not `commute`.

[CANCELLATIONTOKEN_ASYNC_CANCEL_PATTERN]:
- `CancellationTokenSource.CancelAsync()` returns a `Task` that completes after all synchronous callbacks registered via `CancellationToken.Register` have executed and any asynchronous continuations scheduled through them have been queued, but not necessarily completed. `Cancel()` completes synchronously after running all callbacks. For a boundary cell that must ensure all token callbacks complete before proceeding to teardown, `await source.CancelAsync()` provides the right seam; `Cancel()` followed by `await Task.Yield()` does not carry the same guarantee.
- `new CancellationTokenSource(TimeSpan delay, TimeProvider provider)` creates a source whose timer uses the injected `TimeProvider`. A `FakeTimeProvider` can advance the clock to trigger the timeout in deterministic tests without real-time elapsed. `PeriodicTimer(TimeSpan, TimeProvider)` accepts the same overload. Both constructors start the timer immediately on construction.
- `PeriodicTimer.Period` is a read-write property. Setting it mid-operation changes the interval immediately: the next `WaitForNextTickAsync` uses the new period. A boundary cell that must rate-adapt its polling loop can mutate `Period` directly rather than disposing and recreating the timer.

[LAZY_EXCEPTION_CACHING_FOR_INIT_CELLS]:
- `Lazy<T>` with `LazyThreadSafetyMode.ExecutionAndPublication` caches the first exception thrown by the factory permanently â€” every subsequent `.Value` access rethrows the cached exception without retrying the factory. This makes it unsuitable as an initialization cell for anything with transient failure modes (network, file system, native library load). A boundary cell whose initialization can fail transiently must use `LazyThreadSafetyMode.PublicationOnly`, which does not cache exceptions and retries on each subsequent `.Value` access until one succeeds, or must use `Atom<Option<T>>` with explicit retry policy via `IO` schedule.
- `LazyThreadSafetyMode.PublicationOnly` allows multiple threads to run the factory concurrently; the first to complete sets the value, and all others discard their results. For factories that acquire a native resource, concurrent initialization means competing factories all acquire the resource but only one is retained â€” the others are silently discarded without disposal. `PublicationOnly` is only safe for factories that do not acquire resources or for resources that are self-cleaning on GC finalization. The safe concurrent-initialization cell for resource-acquiring factories is `Lazy<T>(factory, LazyThreadSafetyMode.ExecutionAndPublication)` with explicit acknowledgment that transient failures are permanent.
- Recursive access to `Lazy<T>.Value` inside the factory throws `InvalidOperationException` under `ExecutionAndPublication` (the lock is not re-entrant). Under `PublicationOnly`, recursive access does not throw â€” the recursion sees the uninitialized state and re-enters the factory. A boundary cell that chains lazy initialization cells must prevent cross-cell circular dependencies or choose `PublicationOnly` explicitly and accept the double-initialization risk.

[PRIORITIZED_DRAIN_CHANNEL_AS_STATE_BOUNDARY_SEAM]:
- `Channel.CreateUnboundedPrioritized<T>(UnboundedPrioritizedChannelOptions<T>)` creates a channel backed by a heap whose dequeue order is determined by `options.Comparer`. Items are dequeued in ascending order â€” smallest-first per the comparer. For a drain-phase boundary cell, an `IComparer<WorkItem>` that sorts by priority class then enqueue timestamp provides structured drain: high-priority in-flight items drain before low-priority ones, avoiding starvation that a FIFO unbounded channel cannot prevent.
- `ChannelReader<T>.TryPeek(out T item)` is available on bounded and prioritized channels but returns `false` on unbounded non-prioritized channels. `CanPeek` is the capability flag. A drain predicate that decides whether to accept new items based on the current queue depth can use `Count` (when `CanCount` is true) and `TryPeek` together to inspect the head of the prioritized queue without dequeuing.
- `BoundedChannelFullMode.DropWrite` causes `TryWrite` to return `false` immediately when the channel is full without blocking or dropping an existing item. This is the correct mode for a boundary cell that surfaces back-pressure as a typed rejection: the caller receives `false` from `TryWrite`, wraps it in a `BoundaryCapacityFault`, and returns it up the `Fin<A>` rail. `DropNewest` and `DropOldest` silently discard items, obscuring loss from the fault rail. `Wait` blocks, preventing the boundary cell from returning a typed fault synchronously.

[INTERLOCK_SMALL_INT_GATE_CELLS]:
- `Interlocked.CompareExchange(ref byte location, byte value, byte comparand)` and the `sbyte`, `short`, `ushort` overloads are available in .NET 9+. Lifecycle state machines with â‰¤255 states can encode state in a `byte` field and CAS-transition without widening to `int`. Four `byte` state fields fit in one cache word versus four `int` fields spanning multiple words. The state values must fit in the unsigned range of the chosen type; a state machine that uses negative sentinel values must use `int` or `sbyte` with care.
- `Interlocked.Or(ref int location, int value)` and the `uint`, `long`, `ulong` overloads atomically set bits in a flag word and return the original value. For a lifecycle state cell that tracks multiple concurrent properties as a bitfield, a single `Interlocked.Or` or `And` can atomically set or clear a flag without a CAS loop. Reading the current flags requires `Volatile.Read` or `Interlocked.CompareExchange(ref state, state, state)` to get a fresh value with acquire semantics; a plain field read is not coherent on weakly ordered hardware.

[TESTCLOCK_INTEGRATION_FOR_BOUNDARY_TIMEOUT_CELLS]:
- `CancellationTokenSource(TimeSpan, TimeProvider)` and `PeriodicTimer(TimeSpan, TimeProvider)` both accept a `TimeProvider` that drives their respective timers. When the same `FakeTimeProvider` instance is injected into the runtime spine record and into every timeout-scoped `CancellationTokenSource` created from the spine, advancing the fake clock in a test synchronously fires all outstanding timeouts. This gives deterministic coverage of timeout-triggered state transitions â€” `Draining` on grace period expiry, `InitializationFailed` on setup timeout, `Closed` on idle timeout â€” without requiring real-time waits in tests. The `TimeProvider` must be part of the spine record, not hidden inside closure captures, for this propagation to be complete.
- `TimeProvider.CreateTimer(callback, state, dueTime, period)` returns `ITimer` whose `Change(TimeSpan, TimeSpan)` reschedules without allocation. When a boundary cell's polling interval must be adaptive, the correct cell is `(ITimer timer, Atom<TimeSpan> currentPeriod)` with the atom driving `timer.Change` calls from the `Atom.Change` handler. The handler fires synchronously on the thread that committed the swap; `timer.Change` is O(1) and safe to call from the handler.
- `ITimer.Change(TimeSpan.Zero, Timeout.InfiniteTimeSpan)` fires the callback immediately and disables periodic rescheduling. This is the correct form for a boundary cell that must immediately trigger a drain-start callback on an explicit `Shutdown()` call when the cell uses a timer as its drain-trigger rather than a `CancellationToken`.

[LOCK_SCOPE_SEMANTICS_VS_MONITOR]:
- `System.Threading.Lock.Enter()` is re-entrant: the same thread can enter multiple times without deadlock and must exit the same number of times. For boundary cells that guard process-static initialization whose callers may themselves be initializers of dependent cells, `Lock` silently permits the recursion while `SemaphoreSlim(1,1)` hard-deadlocks. Choose `SemaphoreSlim(1,1)` explicitly when recursive entry from the same thread must be detected and rejected rather than silently permitted.
- `Lock.EnterScope()` returns `Lock.Scope`, a `ref struct` with a `Dispose()` method. Because `Scope` is a `ref struct`, it cannot be boxed, stored in a field, or captured in a lambda â€” it is mechanically scoped to the method frame. For a boundary cell that must pass a "locked scope" token to a nested method, the token cannot be a `Lock.Scope`; use `Lock.IsHeldByCurrentThread` as an assertion in the nested method instead.
- `Lock.Scope.Dispose()` throws `SynchronizationLockException` if called from a thread other than the one that entered the lock. A `using Lock.Scope` block that spans an `await` will throw when the continuation resumes on a different thread. `Lock` must not span `await` expressions; use `SemaphoreSlim.WaitAsync` and `Release` for asynchronous mutual exclusion.

[ATOMHASHMAP_FINDORMAYBE_ADD_SURFACE]:
- `AtomHashMap<K,V>.FindOrMaybeAdd(K key, Func<Option<V>> None)` calls the `None` delegate only when the key is absent and inserts only when it returns `Some(v)`. If `None` returns `None`, the map is left unchanged and the returned `Option<V>` is `None`. The factory still executes inside the CAS loop and may run multiple times on contention; it must produce the same insertion decision on repeated calls (though possibly different `Some` values, in which case only the CAS winner's value is stored).
- `AtomHashMap<K,V>.FindOrMaybeAdd(K, Option<V>)` â€” the value-direct overload â€” accepts a pre-computed `Option<V>`. Inserting a pre-computed `None` is a read-only probe: if the key is absent, nothing is inserted; if present, the stored value is returned. This is the correct surface for a lookup-or-no-op pattern where key absence is a valid terminal state rather than a trigger for insertion.

[CHANNEL_DRAIN_MID_FLIGHT_EXACT_SEMANTICS]:
- `ChannelWriter<T>.TryComplete(Exception? error)` returns `false` when the channel is already completed (idempotent on the `false` path). `Complete(Exception?)` throws `ChannelClosedException` when already completed. For drain-phase boundary cells that may receive concurrent `Shutdown()` calls, `TryComplete` is the safe form. When `error` is non-null, `ChannelReader<T>.Completion` task faults with that exception after drain; callers awaiting `Completion` receive the fault, not a clean `RanToCompletion`.
- `ChannelReader<T>.ReadAllAsync(CancellationToken)` produces an `IAsyncEnumerable<T>` that completes when the channel is drained and completed. Cancellation of the token cancels the enumeration but does not complete the channel â€” the channel writer must be separately completed. A drain loop using `ReadAllAsync` that is cancelled mid-drain without the channel being completed will restart enumeration on the next consumer call, rediscovering unread items. The correct drain-then-stop pattern: cancel the write-side via `TryComplete()`, then `await foreach` with `CancellationToken.None` until the enumerable completes.

[OBJECTDISPOSEDEXCEPTION_THROWIF_TYPE_OVERLOAD]:
- `ObjectDisposedException.ThrowIf(bool, Type)` accepts a `Type` directly rather than an `object` instance. Use this overload inside static methods, class factories, or extension methods where `this` is not available but the type is known at compile time: `ObjectDisposedException.ThrowIf(_closed, typeof(BoundaryCell))`. The `typeof(T)` form is a compile-time constant and avoids the virtual dispatch on `GetType()` that the `object`-accepting overload implies. Both overloads produce equivalent exception messages containing the full type name.
