# Sparse Systems: Core Law

[TRIPLET_ASSEMBLY]:
- `CoordinateStorage<T>` is the canonical triplet format. The constructor overload accepting pre-filled arrays requires the explicit `nonZerosCount` form; the two-parameter array overload treats the storage as empty and begins appending from index 0, making it useless for pre-built buffers.
- `CoordinateStorage<T>.At(i, j, value)` accumulates — a subsequent call with the same `(i, j)` adds to the existing entry, not replaces it. This is the only correct mechanism for FEM assembly where multiple elements contribute to the same global node pair; calling `At` once per element contribution is the complete assembly loop.
- `Converter.ToCompressedColumnStorage<T>(storage, cleanup: true, inplace: false)` performs duplicate summation and exact-zero dropping during CSC conversion; `cleanup: false` produces unsorted, possibly duplicate rows that break all factorization routines. The `inplace: true` flag destroys the coordinate arrays in place and must not be used when the triplet buffer needs to survive for structural-edit increments.
- `CompressedColumnStorage<T>.OfIndexed(CoordinateStorage<T>, inplace)` is the static factory shorthand for the same conversion. Prefer it over the `Converter` static class when the source is already a `CoordinateStorage<T>`.
- `DropZeros(tolerance)` on `CompressedColumnStorage<T>` removes entries whose absolute value is at or below the tolerance after conversion. The default tolerance is 0.0, so only exact binary zeros are dropped. For assembly results that accumulate near-cancellation residuals, pass a structural tolerance — typically `machineEps * frobenius_norm` — not 0.0.

[COMPRESSED_FORMAT]:
- CSparse stores CSC: `ColumnPointers` (length `n+1`), `RowIndices`, and `Values`, all as public fields on `CompressedColumnStorage<T>`. MathNet stores CSR: `RowPointers` (length `m+1`), `ColumnIndices`, and `Values` as public fields on `SparseCompressedRowMatrixStorage<T>`. These two storage orientations are not interchangeable; a CSC matrix handed to MathNet's `TrySolveIterative` is treated as CSR by the preconditioner's `Initialize` without any conversion, which silently produces wrong preconditioner factors.
- `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` performs an in-memory transposition at construction time, allocating a new independent block. This is the single correct conversion path when bridging from a CSparse solve result back to a MathNet-resident operator for residual arithmetic. `PopulateExplicitZerosOnDiagonal` must be called before passing the resulting storage to Intel MKL DSS — it fills zero entries on the diagonal explicitly as required by the MKL sparse solver interface.
- `SparseCompressedRowMatrixStorage<T>.Normalize()` removes duplicate entries and ensures row-sorted order. `NormalizeDuplicates()` is the targeted form that only sums duplicates without sorting. `NormalizeOrdering()` sorts column indices within each row. `NormalizeZeros()` removes explicit zeros. These four operations are strictly on CSR storage and have no analog on CSparse CSC storage — the parallel on CSC is `Helper.SortIndices` plus `DropZeros`.
- CSparse's `Helper.ValidateStorage(storage, strict: true)` checks that row indices within each column are sorted and contain no duplicates. It returns a bool and does not throw. The `strict: false` form accepts unsorted, possibly duplicate rows — this matches the post-`inplace` state before a `Converter.ConvertInPlace` completes. Calling any factorization on storage that fails `strict: true` validation produces silently incorrect factors; the factorization constructors do not validate internals.
- `CompressedColumnStorage<T>.AutoTrimStorage` (default `true`) causes sparse matrix addition and multiplication to resize the internal arrays to exactly `NonZerosCount` after each structural operation. Disable it when repeatedly building a matrix incrementally from additions; the reallocations dominate runtime.

[STRUCTURAL_INVARIANTS]:
- Symmetry verification via `CompressedColumnStorage<T>.IsSymmetric()` performs a full O(nnz log nnz) fold-and-compare traversal every call. For `SparseMatrix` (Complex), it tests Hermitian symmetry (conjugate equality), not storage symmetry; these are not the same property and are tested by different methods. Neither method caches its result. Calling it inside a solve loop is a per-access revalidation; the correct design is to carry a typed witness — a constructor-verified value object whose admitted values are limited to matrices with confirmed structural symmetry — so the predicate runs once at the admission boundary, not at each solver call site.
- `Keep(Func<int, int, T, bool>)` on both `CoordinateStorage<T>` and `CompressedColumnStorage<T>` drops entries in-place where the predicate returns false. This is the correct path for structural edit operations — pin a node by keeping only diagonal entries in that row/column — without a full decompress-reassemble cycle. The predicate receives the current value, so it can implement both structural and numeric drop conditions (zero-below-threshold, off-diagonal-in-pinned-row) in one pass.

[SINGLE_FORMAT_LAW]:
- A sparse workflow crosses three storage contexts with distinct format requirements: assembly (triplet, `CoordinateStorage<T>`); iterative-solver matrix-vector products (CSR or CSC, depending on operator orientation); and direct-factorization (CSC with sorted, duplicate-free rows for CSparse, CSR with explicit diagonal zeros for MKL). Converting repeatedly between all three is the multi-representation tax. The single-owner law: one CSC `CompressedColumnStorage<T>` instance is the canonical owner. Triplet inputs convert to it once; iterative solvers that need CSR call `Transpose` once at construction to obtain a CSR-oriented copy; factorization reads the CSC form directly. The owner does not carry format metadata as a flag — format is a named type distinction.
- `CompressedColumnStorage<T>.OfCompressedSparseRowFormat` on the MathNet side and `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` are the only zero-copy conversion bridges between formats. Both allocate a fresh independent block. Feeding them the raw `ColumnPointers`/`RowIndices`/`Values` arrays of a CSparse matrix is the correct interop path; no wrapper or intermediate type is justified.

[FILL_REDUCING_ORDERINGS]:
- `ColumnOrdering.MinimumDegreeAtPlusA` is the only valid ordering for symmetric positive-definite (SPD) systems solved by `SparseCholesky` or `SparseLDL`. The `MinimumDegreeAtA` and `MinimumDegreeStS` orderings operate on the normal equations and are documented as not suited for Cholesky; applying them anyway produces permutation vectors that are semantically valid but numerically counterproductive — no exception is thrown.
- `AMD.Generate<T>(A, ColumnOrdering)` computes the approximate minimum degree permutation vector and returns `int[]`; the factorization `Create` overload accepting `int[]` instead of `ColumnOrdering` lets callers cache or share the permutation vector across multiple factorizations of matrices with identical sparsity pattern but different numeric values. This is the mechanism for the weakly-keyed factor cache: key on sparsity pattern identity (verified via row-count, column-count, and `ColumnPointers`/`RowIndices` hash), cache the `int[]` permutation, and pass it to numeric-only re-factorizations.
- `SymbolicFactorization` exposes `lnz` (nonzeros in L) and `unz` (nonzeros in U or R) as direct integer fields. These are the fill-in evidence values. A sparse receipt carries these before the numeric factorization to enable pre-allocation decisions and to surface fill ratio (`lnz / A.NonZerosCount`) as a diagnostic in the algorithm receipt.
- `GraphHelper.EliminationTree` and `GraphHelper.ColumnCounts` are exposed as public static methods. They are normally called inside the symbolic analysis phase; calling them independently is the mechanism for inspecting symbolic fill structure prior to committing to a numeric factorization under a particular ordering.

[FACTORIZATION_LAW]:
- `SparseCholesky.Create(A, ColumnOrdering.MinimumDegreeAtPlusA)` performs symbolic analysis and numeric factorization in one call; it does not expose a two-phase API at the `Create` level. The symmetric matrix requirement is a precondition; only the upper triangular part is read. Passing a non-SPD matrix produces silently incorrect factors rather than a typed failure — the factorization succeeds but `Solve` returns a numerically wrong answer. The receipt must include a post-solve residual recomputation `‖Ax − b‖₂ / ‖b‖₂` as the only reliable correctness witness.
- `SparseLDL` handles indefinite symmetric matrices (negative diagonal entries in D). Its documentation states explicitly that no numeric pivoting is performed; for strongly indefinite systems or those with near-zero pivots, the LDL' path produces inaccurate factors without warning. The receipt law for LDL': carry the diagonal-D sign flags (count of negative pivots) as a structural audit field so callers can detect unexpected indefiniteness.
- `SparseLU.Create` accepts a `tol` parameter in [0, 1] controlling partial pivoting: `tol = 1.0` is full partial pivoting, `tol = 0.001` is near-diagonal (fast, dangerous for ill-conditioned systems). The parameter is not a threshold on pivot size — it is a relative threshold: a pivot is accepted if it is at least `tol` times the magnitude of the largest element in its column. Setting `tol = 0` disables pivoting entirely.
- `SparseCholesky.Update(w)` and `Downdate(w)` implement rank-1 Cholesky updates in-place without re-factorization; `UpDown(sigma, w)` is the combined `+sigma * w*w'` form with `sigma ∈ {+1, -1}`. These return `false` when the updated matrix loses positive definiteness, not a typed error. For structural-edit paths — pinning a node modifies one row/column — these are significantly cheaper than full re-factorization when the modification is rank-1 in nature.

[KRYLOV_ITERATION]:
- `Matrix<T>.TrySolveIterative(input, result, solver, iterator, preconditioner)` returns `IterationStatus` without throwing. The six status values are `Continue`, `Converged`, `Diverged`, `StoppedWithoutConvergence`, `Cancelled`, and `Failure`. Only `Converged` represents an acceptable result. The distinction between `Diverged` (monotone residual increase detected over `MinimumNumberOfIterations` iterations) and `StoppedWithoutConvergence` (count cap hit without convergence) is load-bearing for fallback routing: divergence routes to a regularized retry or direct solve; count-exhaustion routes to a direct solve with the partial iterate as warm start.
- `Iterator<T>` is constructed by passing an `IIterationStopCriterion<T>[]` array. The stacked stop-criteria policy executes all criteria in order on every iteration and halts on the first non-`Continue` status. The correct minimum stack is `[FailureStopCriterion, DivergenceStopCriterion, ResidualStopCriterion, IterationCountStopCriterion]` in that order: NaN detection before divergence detection before convergence test before hard cap.
- `ResidualStopCriterion<T>` carries both a `Maximum` threshold and a `MinimumIterationsBelowMaximum` count — the residual must remain below `Maximum` for at least `MinimumIterationsBelowMaximum` consecutive iterations before `Converged` is signaled. This eliminates false early-termination from oscillating residuals. Setting `MinimumIterationsBelowMaximum = 1` degrades to single-point convergence and misses oscillators.
- `DivergenceStopCriterion<T>` tracks residual history over `MinimumNumberOfIterations` iterations and fires `Diverged` when the residual increases monotonically over that window at a relative rate above `MaximumRelativeIncrease`. The default `MinimumNumberOfIterations` is not zero; the criterion requires a warm-up window before it can detect divergence. Setting it too short triggers spurious divergence on normally oscillating Krylov residuals in the early iterations.
- The `IterationCountStopCriterion<T>` default `DefaultMaximumNumberOfIterations` field is a constant; it is not scaled to system size. For large systems, an iteration cap that is not proportional to `n` (or to a user-supplied relative tolerance on restart dimension) will terminate before the Krylov space is large enough to capture the solution. The correct design carries the size-scaled cap as a policy value computed at factorization time, not a raw constant.
- Double-checked convergence: `TrySolveIterative` returning `Converged` means the `ResidualStopCriterion` saw the internal residual vector fall below its threshold, but that vector is the preconditioned residual computed inside the Krylov algorithm, not the true residual `‖Ax − b‖`. The independent recomputation `‖Ax − b‖₂ / ‖b‖₂` must be performed on the solution in the calling code to verify the actual solve quality, because preconditioning can distort the internal residual's relationship to the true error.

[PRECONDITIONERS]:
- `ILU0Preconditioner` uses the sparsity structure of the input matrix to compute the ILU(0) factorization; fill is limited to positions that are already nonzero in the input. Its `Initialize` method internally casts the matrix to `SparseMatrix` (CSR). Passing a dense `Matrix<T>` causes a dense-to-sparse conversion inside `Initialize` — valid but unnecessary; always pass a sparse matrix directly.
- `ILUTPPreconditioner` (ILU with threshold and pivoting) carries `FillLevel`, `DropTolerance`, and `PivotTolerance` as mutable properties on the instance. The interaction between `FillLevel` (controls how many extra fill entries are kept per row) and `DropTolerance` (numeric drop threshold applied column-by-column) determines effective preconditioner quality; there is no single correct pair — they are problem-dependent. The defaults `DefaultFillLevel = 200.0` and `DefaultDropTolerance = 0.0001` are aggressive fill retention with mild dropping; for memory-constrained settings reduce `FillLevel` first.
- `MILU0Preconditioner` is the modified ILU(0) variant that corrects the row sums of the L and U factors so that the preconditioner matrix exactly preserves the rowsums of the original matrix. This improves convergence for M-matrices and diagonally dominant problems. Its `UseModified` property (default inferred from construction) selects between standard and modified forms; `IsInitialized` allows deferred initialization checks before the first `Approximate` call.
- `DiagonalPreconditioner` stores only the diagonal of the matrix. It is O(n) to apply and effectively zero in setup cost. For well-conditioned systems, it degrades convergence relative to ILU0, but for Krylov methods applied to near-diagonal systems (e.g., Jacobi-style discrete operators), it eliminates preconditioner overhead while still providing scale normalization.

[GAUGE_HANDLING]:
- Sparse systems arising from pure Neumann boundary conditions or floating substructures have a singular stiffness matrix and a non-trivial null space. There is one policy concern with four named dialects: (1) large diagonal penalty — add `α * I` to the singular rows (α ≫ max diagonal entry); (2) row/column elimination — replace the row and column of the gauge node with a zero row, zero column, and 1 on the diagonal, then adjust the right-hand side; (3) gauge-node identity row — replace one row with the identity condition directly in the assembled `CoordinateStorage<T>` before conversion; (4) regularization shift — add `α * I` uniformly to the entire matrix. These are not four code paths — they are four expressions of one operator modification, all implementable through `CompressedColumnStorage<T>.Keep` combined with targeted `At` additions on the triplet before conversion. A single policy function parameterized by `GaugeTreatment` (a `[SmartEnum]` or `[Union]`) produces the appropriate modification pattern; a separate function per dialect is the anti-pattern.
- The large diagonal penalty value `α` must exceed the spectral radius of the stiffness matrix to be numerically effective. A practical lower bound is `α ≥ 1e8 * max_diagonal_entry`. The resulting system is no longer singular but is ill-conditioned; for direct solvers this is acceptable; for iterative solvers it stresses the preconditioner. The receipt for gauge-treated systems carries the gauge treatment type and the effective `α` value.

[REALIFIED_BLOCK_EMBEDDINGS]:
- `CSparse.Complex.*` contains full complex CSC storage and complex `SparseCholesky`/`SparseLU`/`SparseLDL`/`SparseQR`. The `System.Numerics.Complex` element type maps directly into these. However, when the factorization dependency is constrained to real-only solvers (e.g., when MKL DSS is the provider and its complex mode is unavailable), a two-by-two real block embedding must be applied: a complex `n×n` system becomes a `2n×2n` real system with the embedding `[Re(A), -Im(A); Im(A), Re(A)]`. Assembling this from triplets is straightforward — each complex entry `(i, j, a+bi)` contributes four real entries — but the `CoordinateStorage<double>` for the block-embedded system has `4 * nnz` entries before deduplication. The `At` duplicate-summation mechanism handles overlapping block contributions correctly. The admission boundary for complex-to-real block embedding is a single conversion function keyed on the `DssMatrixStructure` enum: `Nonsymmetric` maps to no symmetry; `Symmetric` maps to the real embedding with the upper half only.
- `SparseMatrix.Transpose(storage: true)` on a Complex `SparseMatrix` performs a storage-level transpose (converting CSC to CSR orientation) without complex conjugation; the default `Transpose()` includes conjugation. The distinction is non-trivial when building Hermitian block systems manually.

[FACTOR_CACHE]:
- A weakly-keyed success-only factor cache holds solved factorizations keyed by the complete physical parameter set: matrix dimensions, sparsity fingerprint (hash of `ColumnPointers` and `RowIndices` arrays), and the ordering enum or permutation vector identity. The cache stores only successful factorizations — those where post-solve residual verification passed — so a failed or diverged solve never populates the cache and the next call re-factorizes without stale data.
- The permutation vector `int[]` obtained from `AMD.Generate` is the stable key for numeric-only re-factorizations when the sparsity pattern is invariant but values change. Caching the permutation separately from the numeric factors lets the symbolic phase run once and numeric-only phases run cheaply under a new value assignment.
- `ISparseFactorization<T>.NonZerosCount` exposes the number of nonzeros in the factors (L for Cholesky/LDL; L+U combined for LU). This is the fill-in evidence and belongs in the receipt alongside the pre-ordering nnz of the input matrix, so the fill ratio is auditable without re-running the factorization.
