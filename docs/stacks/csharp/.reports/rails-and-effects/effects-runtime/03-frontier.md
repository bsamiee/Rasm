# Effects-Runtime: Frontier

[IMPLICIT_CONVERSION_AS_LIFT_SURFACE]:
- `Eff<A>` carries nine implicit conversions: from `Pure<A>`, `Fail<Error>`, `Lift<A>`, `Lift<Fin<A>>`, `Lift<MinRT,A>`, `Lift<MinRT,Fin<A>>`, `Either<Error,A>`, `Fin<A>`, and `IO<A>`. Every already-computed carrier value becomes an `Eff<A>` without an explicit lift call; the `Fin<A>` implicit is load-bearing at boundary seams where a validation rail rejoins the effect rail. `Eff<RT,A>` adds one further implicit: from `Eff<A>`, widening the no-RT carrier to the RT form — the single direction of implicit widening matches the type-parameter direction.
- `IO<A>` is not implicitly convertible to `Eff<A>` at the type level; the `Eff<A>.op_Implicit(IO<A>)` is the one-way bridge. When an `IO<A>` value must participate in a `K<Eff, A>` context without explicit wrapping, pass it through `Eff.lift(io)` — the static form is the canonical spelling and matches the core lift shape.
- The `Lift<A>` implicit on `Eff<A>` accepts a `Lift<A>` ref-struct by-ref (`@`). This means an inline `Lift(() => expr)` value cannot be stored in a variable before passing — it must be constructed at the conversion site. The lifetime constraint is not enforced by the compiler for method-parameter contexts but signals that these ref-struct lifts are not general values.

[RUNTIME_SCOPING_WITH_LOCAL]:
- `Eff.local<RT1, RT2, A>(Func<RT1, RT2> f, Eff<RT2, A> ma)` returns `Eff<RT1, A>`. It maps the outer environment through `f` to produce a narrowed inner environment, runs `ma` in that inner environment, and re-surfaces the result in the outer carrier. This is the canonical mechanism for embedding a dependency-isolated sub-computation whose runtime record is a strict subset or projection of the outer one — no capability leakage between the two scopes.
- `Eff.runtime<RT>()` lifts the runtime record itself into the bound value as `Eff<RT, RT>`. Combined with `.Bind(rt => ...)` it enables a computation to inspect its own runtime at composition time without breaking the reader monad abstraction. Used to derive derived capabilities or perform runtime-conditional branching without breaking the monadic spine.
- `Eff.getState<RT>()` exposes the full internal `Eff<RT, RT>` state, including the `EnvIO`. This is the inspection escape hatch when both the runtime record and the execution environment must be read in one bind; it is not equivalent to `runtime` — `runtime` gives only the `RT`, `getState` gives the full `Eff` internals.
- `Eff.localCancel<RT, A>(Eff<RT, A> ma)` wraps `ma` in a fresh cancellation scope linked to but isolated from the outer token. A cancellation of the inner scope does not cancel the outer scope's other computations; but an exception produced by `IO.cancel` inside the local still propagates outward because it is an exception, not a signal. The key invariant: scope isolation prevents regional cancellation from tearing sibling effects; it does not suppress the exception itself.

[STM_AND_REF_INTEGRATION]:
- `STM.DoTransaction<A>(Eff<A>, Isolation)` and `STM.DoTransaction<RT, A>(Eff<RT, A>, Isolation)` are the effect-aware transaction forms; they are distinct from `DoTransaction(Func<A>, ...)` and `DoTransaction(Func<ValueTask<A>>, ...)`. The `Eff` overloads nest cleanly into an effect pipeline because they return `Fin<A>` via the standard run collapse; the `Func` overloads do not participate in the `Fin` rail. `RunTransaction` is the raw form that does not silently re-nest into a parent transaction; `DoTransaction` nests when a parent is active.
- `Isolation.Snapshot` rolls back only when a `Ref` *written within* the transaction is externally written between start and commit. `Isolation.Serialisable` rolls back when any `Ref` *read or written within* the transaction is externally written. Choosing `Snapshot` admits more concurrency but leaves read-write skew possible; `Serialisable` prevents all anomalies at the cost of higher conflict rate.
- `Ref<A>.Commute(Func<A, A>)` re-runs the function at commit time against the most-recently-committed value, not the in-transaction snapshot. This means `Commute` is only correct for commutative or last-writer-wins operations — non-commutative uses will silently produce stale-basis results. The `CommuteRef<A>` wrapper returned by `DoTransaction(Func<CommuteRef<A>>)` provides the same semantics with explicit commute-function registration via `STM`.
- `Ref<A>.SwapIO(Func<A, A>)` returns `IO<A>` — it wraps the atomic swap in the `IO` monad for participation in effect pipelines. The swap is still transactional when called inside a `DoTransaction` boundary, but `SwapIO` can also be run outside a transaction as a standalone CAS loop. The `IO` wrapper gives cancellation-awareness: if the outer `EnvIO` is cancelled while the CAS loop is spinning, the operation aborts. `Atom<A>.SwapIO` has identical semantics and the same return type.
- `Atom<A>.SwapMaybeIO(Func<A, Option<A>>)` enables conditional atomic update as an IO value: when the function returns `None`, the atom is unchanged and the IO yields the current value; when it returns `Some(v)`, the atom is updated to `v`. This is the idiomatic form for test-and-set patterns where the decision to update depends on the current value — the function may be called multiple times on contention, so it must remain side-effect-free.

[TIMEOUT_BEHAVIOR_AND_CARRIER_POLYMORPHISM]:
- `IO<A>.Timeout(TimeSpan)` throws when the limit is exceeded — its return type is `IO<A>`, not `IO<Fin<A>>`. The timeout exception is not wrapped into the `Fin` rail automatically; it propagates as an exception through the trampoline's catch stack. To model a timeout as a typed failure, wrap with `.BracketFail()` followed by a `@catch` on the exception type, or use the carrier-polymorphic `TimeoutIO<M, A>(K<M, A>, TimeSpan)` extension from `MonadUnliftIOExtensions`, which uses `MapIO` and converts the exception to an `Error` inside the carrier's own `Fin` wrapper.
- `MonadUnliftIOExtensions.TimeoutIO<M, A>(K<M, A> ma, TimeSpan timeout)` is carrier-polymorphic via `MonadUnliftIO<M>`: any `Eff<A>`, `Eff<RT, A>`, transformer stack, or custom `MonadUnliftIO` type picks it up without specialization. The `MapIO(ma, io => io.Timeout(ts))` implementation means the timeout clock starts when the underlying `IO` begins executing, not when the `Eff` or transformer is constructed — composition is lazy.
- `IO<A>.WithEnv(Func<EnvIO, EnvIO>)` and `IO<A>.WithEnvFail(Func<EnvIO, EnvIO>)` are the low-level env-mapping primitives that `Bracket`, `BracketFail`, `Timeout`, and `Fork` all build on. `WithEnv` releases the mapped resources on both success and failure; `WithEnvFail` releases only on failure and leaks resources back to the parent scope on success. A custom DSL extension that requires a specialized `EnvIO` shape constructs it via `WithEnv`/`WithEnvFail` — never by accessing `EnvIO` fields directly.

[APPLICATIVE_TUPLE_AND_OPERATOR_FORMS]:
- `EffExtensions` provides `SelectMany` overloads accepting `ValueTuple<K<Eff,A>, K<Eff,B>>` and `ValueTuple<K<Eff,A>, K<Eff,B>, K<Eff,C>>` as the collection source. This enables LINQ query expressions where the `from` clause destructures a tuple of effects: `from (a, b) in (effA, effB)` runs both effects sequentially and binds both results into scope in one step. The semantics are monadic (left-to-right sequential), not parallel — applicative concurrency requires explicit `fork`/`awaitAll`.
- The `*` operator (`op_Multiply`) on `K<Eff, Func<A,B>>` and `K<Eff, A>` is curried applicative application; `IOExtensions` defines the same operator extending to 11-arity curried functions. The extended arity covers `Func<A0,...,A10, B>` applied to `K<IO, A0>` through `K<IO, A10>` — all effects run left-to-right, the function is applied to all results. This is the dense form for combining up to 11 independent but sequentially-executed effects without nested `SelectMany`.
- `>>>` (`op_UnsignedRightShift`) on `K<Eff, A>` with `K<Eff, B>` runs the left effect for its side-effects, discards the value, then runs the right effect and returns its value. It is distinct from `>>` (`Kleisli`): `>>>` ignores the left value entirely, while `>>` feeds it as input. The operator is available on both `Eff` and `IO` via their respective extension classes.
- `MonadIOExtensions.SelectMany<M, A, B, C>(K<M, A>, Func<A, IO<B>>, Func<A, B, C>)` and the symmetric `SelectMany<M, A, B, C>(IO<A>, Func<A, K<M, B>>, Func<A, B, C>)` enable mixed-carrier LINQ queries where one step is `IO` and another is `Eff<RT>` without explicit `.LiftIO`. The first form binds an `IO` step inside an `M`-shaped query; the second binds an `M` step inside an `IO`-shaped query. Both delegate to the carrier's `LiftIO` automatically.

[FALLIBLE_COLLECTION_PARTITIONING]:
- `FallibleExtensionsE.succs<M, A>(Seq<K<M, A>>)` runs each effect in the sequence and returns a `Seq<A>` containing only the successful values; failed effects are silently dropped. `FallibleExtensions.fails<M, A>(Seq<K<M, A>>)` runs each effect and returns a `Seq<Error>` containing only the failure values. Neither form accumulates; both execute eagerly. When both halves are needed, pair two traversals or use `TraverseM` followed by partition — there is no single `partitionM` form in the API.
- `FallibleExtensions.CatchIO<M, A>(K<M, A>, Func<Error, bool>, Func<Error, K<IO, A>>)` matches the `@catch` family but lifts the recovery path to `IO` rather than the outer carrier `M`. This is used at the boundary where recovery produces a raw IO effect (typically a BCL call wrapped in a single `IO.lift`) that does not belong to the outer runtime context. The recovery result is then lifted back into `M` via `MonadIO.liftIO`. The three-overload family covers predicate, Error-equality, and numeric-code matching.
- `guardIO(bool)` returns `IO<Unit>` and emits `Applicative.Empty` (which propagates as `Errors.None`) when the condition is false. `guard<E>(bool, Func<E>)` returns `Guard<E, Unit>` for use in generic query expressions over `Fallible<E>` carriers. The two forms are not interchangeable: `guardIO` is for `IO`-typed query expressions; `guard<Error>(...)` is for `Eff`-typed query expressions that need to emit a typed error on the failure branch.

[SCHEDULE_ADVANCED_PATTERNS]:
- `Schedule.repeat(int times)` is a `ScheduleTransformer` that repeats the *entire base schedule* `n` times — it is not a retry/repeat count. `Schedule.recurs(int times)` limits to `n` *iterations of the retry/repeat loop*. The naming inversion is non-obvious: `recurs(5)` gives 5 total attempts (the natural mental model), while `repeat(5)` gives 5 full schedule cycles each containing as many steps as the base. Use `recurs` for count-bounded loops, `repeat` for count-bounded phase cycling.
- `Schedule.fixedInterval(Duration, currentTimeFn?)` fires at absolute wall-clock boundaries regardless of how long the action took. If the action runs longer than the interval, the *next* fire is immediate — no pile-up of queued executions. `Schedule.windowed(Duration, currentTimeFn?)` subdivides the timeline into windows and sleeps until the next window boundary; it can fire multiple times per window if the execution is fast. Both accept an optional `currentTimeFn` for deterministic test injection.
- `ScheduleTransformer` is `(Func<Schedule, Schedule>)` wrapped in a record with `Apply(Schedule)` and `op_Addition(ScheduleTransformer, ScheduleTransformer)`. Composing transformers with `+` produces a new transformer that applies left-then-right: `(a + b).Apply(s)` equals `b(a(s))`. A custom transformer is just a lambda wrapped in `new ScheduleTransformer(s => ...)` — any `Schedule -> Schedule` function qualifies.
- `Schedule.intersperse(Duration[] durations)` accepts a `params Duration[]` and builds an `Iterable<Duration>` from the array before constructing `SchItems`. The `Schedule.intersperse(Schedule inner)` overload binds each step of `inner` between the base steps. These two overloads do not conflict at call sites because `Duration` and `Schedule` are not implicitly convertible; the array overload admits inline duration literals as a variadic probe series.

[FORK_RESOURCE_AND_AWAIT_SEMANTICS]:
- `IO<A>.Fork(Option<TimeSpan>)` uses `TaskCreationOptions.LongRunning`, which requests a dedicated OS thread rather than a thread-pool worker. For short-lived tasks this creates excess OS-thread pressure; `fork` is designed for computations that would otherwise block the pool. For lightweight parallelism, prefer the applicative `*` operator or `awaitAll` over a `Seq<K<M, A>>`, which uses `TraverseM` without `LongRunning`.
- `awaitAny(Seq<K<M, A>>)` forks each effect internally, uses `Task.WhenAny` over the awaits, picks the first-completed task, and returns its value. Losers are neither cancelled nor awaited — their tasks continue running in the background until their own timeout or completion. When guaranteed cleanup of losers is required, take `ForkIO<A>` handles explicitly, call `awaitAny(Seq<ForkIO<A>>)`, then cancel losers via `forkIO.Cancel.Run(env)` in a `Finally` block.
- `ForkIO<A>.Await` is `IO<A>`, not `IO<Fin<A>>`. A cancellation of the *awaiting* environment (not the fork's own token) causes `Await` to propagate the caller's cancellation exception — the fork continues but the await unblocks. This means `Await` inside a retry loop re-awaits the same fork on each retry, potentially producing duplicate results if the fork completes between retry attempts. Only the first `Await` observes the `ForkIO`'s result; subsequent awaits on the same handle are racing on a completed task.
- `IOExtensions.WaitOneIO(AutoResetEvent)` and `WaitOneIO<A>(AutoResetEvent)` lift BCL synchronization primitives into the `IO` monad. The typed overload returns `IO<A>` where `A` is a phantom — useful when a downstream bind requires a typed IO. These are the only BCL sync-primitive adapters in the extension class; `Semaphore`, `ManualResetEvent`, and `Monitor` have no built-in `IO` wrappers and must be lifted via `IO.lift(() => evt.WaitOne())`.

[DSL_EXTENSION_POINTS_AND_CORRECTNESS]:
- The four legal `IO<A>` DSL extension base types are `InvokeSync<A>`, `InvokeAsync<A>`, `InvokeSyncIO<A>`, and `InvokeAsyncIO<A>`. `InvokeSyncIO<A>` is the only synchronous form whose `Invoke` returns `IO<A>` rather than `A` — it chains more DSL nodes from a synchronous gate, enabling two-phase dispatch where phase selection is cheap-synchronous and the body is effect-producing. Subclassing any other `IO<A>` type throws `Errors.IODSLExtension` at run time, not compile time.
- `IOCatch<X, A>` is the `IOCatch<A>` subtype that carries a typed exception parameter `X`. When implementing a custom recovery node, subclass `IOCatch<X, A>`, implement `MakeOperation()` to return the guarded `IO<A>`, and implement `MakeHandler()` to return `Func<Exception, IO<A>>`. The `Final` field is optional; when set, it is run unconditionally as a `Finally` before the next continuation. Do not allocate these nodes per-call in a tight loop — they allocate a `StackItem` per invocation.
