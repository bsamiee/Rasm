# Entrypoint Law — Deepening

[FROZEN_DISPATCH_TABLE_MECHANICS]:
- `FrozenDictionary.Create<TKey,TValue>(params ReadOnlySpan<KeyValuePair<TKey,TValue>>)` accepts a collection-expression literal at the call site — `FrozenDictionary.Create([(key1, handler1), (key2, handler2)])` — with no heap-allocated intermediate because the compiler lowers the collection expression to a stack-allocated span. The comparer overload `Create<TKey,TValue>(IEqualityComparer<TKey>?, params ReadOnlySpan<...>)` controls key equality; passing the comparer at construction is the only way to configure alternate-key lookup, as there is no post-construction comparer replacement.
- `FrozenDictionary<TKey,TValue>.GetValueRefOrNullRef(TKey)` returns `ref readonly TValue`; absence is detected with `Unsafe.IsNullRef(in slot)`. The dispatch idiom: `ref readonly var slot = ref table.GetValueRefOrNullRef(key); if (Unsafe.IsNullRef(in slot)) return Fin<Result>.Fail(...); return slot(request);` achieves zero-allocation lookup and invocation for `Func<Cmd, Fin<Result>>` values.
- `FrozenDictionary<TKey,TValue>.GetAlternateLookup<TAlternate>()` lets an `FrozenDictionary<string, TValue>` keyed with `StringComparer.OrdinalIgnoreCase` accept a `ReadOnlySpan<char>` at lookup time when the comparer implements `IAlternateEqualityComparer<ReadOnlySpan<char>, string>`. The BCL's `StringComparer.OrdinalIgnoreCase` satisfies this constraint. The dispatch table key stays `string`; the call site passes a span sliced from an incoming buffer without allocating. Storing the `AlternateLookup<TAlternateKey>` struct — a `readonly struct` with `ContainsKey`, `TryGetValue`, and an indexer — in a `static readonly` field avoids reconstructing the lookup handle per call.
- `FrozenDictionary` is constructed once at module load and assigned to a `static readonly` field. A collection-expression initializer compiles to a static field initializer, which the runtime executes once with no locking overhead. When the initialization key source is a `[SmartEnum]` items collection, the smart enum's key uniqueness invariant propagates through `FrozenDictionary.Create` and eliminates the runtime duplicate check — construction cannot throw on valid enum inputs.

[ARITY_COLLAPSE_FAILURE_SURFACES]:
- `params ReadOnlySpan<T>` and `params IEnumerable<T>` are both valid `params` collection forms under C# 13+. The compiler resolves to the `ReadOnlySpan<T>` overload first when both exist — span is already a better conversion target for collection-expression arguments, so `[OverloadResolutionPriority]` is only needed when candidates are equally good conversions, not to select span over `IEnumerable`.
- When a `params ReadOnlySpan<T>` parameter appears in a generic method where `T` is unconstrained, the compiler cannot lower a collection-expression argument to a stack-allocated span unless the call site provides a statically known element count. If the count comes from a variable, the compiler emits a heap-allocated array. The entrypoint must either accept a typed span directly (`ReadOnlySpan<KnownCmd>`) or add `where T : allows ref struct` to prevent the heap fallback for ref-struct-safe types.
- `[OverloadResolutionPriority(-1)]` applied to a deprecated `T[]`-accepting overload downgrades it relative to the span overload without deletion. The priority is evaluated before normal tiebreaking, so the span overload wins at all call sites that could call either, even when the array overload would otherwise win by specificity. Negative priority applies regardless of whether the overload is in the same type or a base type.
- The input-redescribing-parameter anti-pattern has a structural variant: a `CancellationToken` beside a union value is not redescription (it is a capability orthogonal to the discriminant), but a `bool processInParallel` beside a `Seq<Cmd>` is: the batch intent must be encoded in the union case's type, not as a parallel flag. The test is whether the discriminant is recoverable from the value alone — if yes, the parameter is redundant.

[BRACKETIO_OVERLOAD_MECHANICS]:
- `bracketIO<M,A>(K<M,A>)` (single-arg, `Prelude`) creates a local resource environment scoped to the computation. Resources acquired via `IO.use<A>(resource)` (where `resource : IDisposable`) inside the computation are registered in the local scope and released when the scope exits regardless of success or failure. This is the correct shape for an entrypoint whose dispatch arm acquires a scoped connection or handle.
- `bracketIO<M,A,B,C>(K<M,B> Acq, Func<B,IO<C>> Use, Func<B,IO<A>> Fin)` (three-arg) handles resources that cannot implement `IDisposable` directly — e.g., a remote connection requiring an async close call. The `Fin` returns `IO<A>` where `A` is the final result type; the `Fin` arm can therefore contribute a derived value rather than being forced to return `Unit`.
- `bracketIO<M,A,B,C>(Acq, Use, Catch, Fin)` (four-arg) adds `Catch: Func<Error, IO<C>>`, which the XML docs state runs for errors raised within the `Acq` computation — not the `Use` computation. `Fin` always runs in all paths; `Catch` is for reclassifying or logging the acquisition error before cleanup, and cannot suppress the cleanup.
- The single-arg form is documented as preferred: "Consider using `bracketIO(computation)` rather than `bracket(acq, use, fin)`, the semantics are more attractive as there's no need to provide function handlers, the cleanup is automatic."

[CATCH_OPERATOR_DISCRIMINATION]:
- The `|` catch operator on `K<M,A>` has two right-hand operand types: `CatchM<TError,M,A>` (produced by `@catch`, `expected`, `exceptional`, and typed variants) and `Finally<M,A>` (produced by `final<F,X>(K<F,X>)`). The two overloads on `FallibleExtensions` are distinguished by right-operand type at compile time. Chaining `computation | catchHandler | final(cleanup)` is legal because the result of `computation | catchHandler` is still `K<M,A>`, which then accepts the `Finally<M,A>` right operand.
- `catchOf<E,M,A>(Func<E,K<M,A>>)` restricts the catch to errors whose runtime type is exactly `E` (a subtype of `Error`). `exceptionalOf<E,M,A>` matches `Exceptional`-subtype errors; `expectedOf<E,M,A>` matches `Expected`-subtype errors. This distinction is load-bearing for entrypoints that need to separate infrastructure failures (`Exceptional` = unexpected exception) from domain rejections (`Expected` = anticipated failure with a code). Multiple `|` handlers are processed in declaration order; the first predicate match wins.
- Error code matching `@catch(int errorCode, Func<Error,K<M,A>> fail)` compares against `Error.Code`; when `Code == 0`, comparison falls back to message equality. An entrypoint that discriminates retryable errors by code must use non-zero codes on its domain errors to avoid false message-equality matches.

[SCHEDULE_TRANSFORMER_STACKING_ORDER]:
- `Schedule.recurs(n)`, `maxDelay`, `maxCumulativeDelay`, `jitter`, `decorrelate`, `resetAfter`, `intersperse`, and the static fields `NoDelayOnFirst`, `RepeatForever`, `Identity` are `ScheduleTransformer` values; `spaced`, `linear`, `exponential`, `fibonacci`, and cron variants are `Schedule` values. `ScheduleTransformer` carries an implicit conversion to `Schedule` via `Schedule.Forever | t`, so a bare transformer is valid wherever a `Schedule` is expected — `retry(Schedule.recurs(5), computation)` is equivalent to `retry(Schedule.Forever | Schedule.recurs(5), computation)`, producing exactly 5 retries.
- `Schedule.NoDelayOnFirst` patches the first yielded duration to zero. The chain `NoDelayOnFirst | Schedule.recurs(5) | Schedule.exponential(seed)` evaluates left-associatively: `NoDelayOnFirst | Schedule.recurs(5)` resolves via `ScheduleTransformer`'s implicit conversion to `Schedule` followed by the `(Schedule, ScheduleTransformer)` overload, producing a finite zero-first-delay schedule; the outer `|` then unions it with the exponential schedule. Placing `NoDelayOnFirst` as the leftmost operand is the correct form for a zero-first-delay finite exponential retry.
- `Schedule.jitter(minRandom, maxRandom)` adds a random duration in the specified range to each step's delay. `Schedule.decorrelate(factor)` de-correlates durations symmetrically by applying +/- jitter that doubles the schedule length. For parallel retry from multiple entrypoints, `decorrelate` is preferable to `jitter` because it prevents thundering-herd re-collision. The practical composition for distributed retry aspects: `Schedule.recurs(5) | Schedule.exponential(50*ms) | Schedule.decorrelate() | Schedule.maxCumulativeDelay(5000*ms)`.
- `Schedule.repeat(n)` repeats the entire schedule `n` times; `Schedule.recurs(n)` limits the step count to `n`. Using `repeat(2)` on a `Schedule.recurs(3) | Schedule.exponential(100*ms)` produces 6 total steps, not 3 — a common source of incorrect retry count behavior.

[EXTENSION_BLOCK_COMPOSITION_CONSTRAINTS]:
- C# 14 extension blocks must be declared inside a top-level, non-generic, `static class`; placing an `extension(T)` block at top-level or inside a generic class produces CS9283. Multiple `extension(SameReceiver)` blocks in the same static class are additive; multiple static classes each containing an `extension(SameReceiver)` block are also additive across the compilation. This is the mechanism for splitting test-only probing methods into a test-assembly extension block without modifying the production static class.
- Extension blocks cannot access `private` members of the receiver type (CS0122). The dispatch entrypoint written in an extension block must route entirely through the receiver's public API — the generated `Switch`/`Map` methods, public case properties, and explicit interface members. This is not a limitation for union dispatch because the generated `Switch` surface is public by definition.
- The `extension(T)` block itself does not accept an access modifier (CS0106). Visibility is controlled per-member inside the block: `public`, `internal`, and `private` all compile and carry their normal access semantics. A `private` member in an extension block is inaccessible from outside the static class containing the block.
- Extension blocks support property syntax: `public bool IsA => receiver is TypeA;` compiles without restriction. Properties defined this way are non-virtual and compile to static methods with a receiver parameter, suitable for attaching computed predicate properties to a union type without touching the generated partial.

[UNION_ENTRYPOINT_PRIVACY_FAILURE_MODES]:
- `ConstructorAccessModifier.Public` (the default) allows any call site to construct any case via the generated implicit conversion. `ConstructorAccessModifier.Internal` on an `internal` union type allows trusted assemblies to construct cases while preventing external consumers from injecting cases directly — the correct setting for a request union that crosses internal assembly boundaries but must remain sealed externally.
- When `ConstructorAccessModifier = Private` and `FactoryMethodGeneration = None`, the only construction surface is a hand-written `public static` factory method inside the partial class, which remains accessible because it is inside the partial class. No caller can implicitly widen a case, no caller can call a generated factory, and the `Switch` and `Map` methods remain fully operative.
- When a regular `[Union]` case declares a property whose type is itself a `[Union]`, destructuring requires nesting `Switch` calls. If both unions use `NestedUnionParameterNameGeneration.Default`, the outer dispatcher parameter names include the full qualified path and the inner dispatcher is invoked via a chained `Switch`. If `Simple` naming is used on the outer union but two inner union leaf names collide, the collision is a compile error on the outer union's generated code.

[FROZEN_SET_ENTRYPOINT_GUARDS]:
- `FrozenSet<T>.GetAlternateLookup<TAlternate>()` works identically to `FrozenDictionary`'s alternate lookup. A `FrozenSet<string>` with `StringComparer.OrdinalIgnoreCase` accepts `ReadOnlySpan<char>` lookups via its alternate lookup instance. The zero-allocation guard pattern: `if (!_validNames.GetAlternateLookup<ReadOnlySpan<char>>().Contains(nameSpan)) return Fin<Result>.Fail(Error.New("unknown command"))` pre-validates a command name before dispatching without a separate boolean parameter.
- `FrozenSet<T>` and `FrozenDictionary<TKey,TValue>` both throw `ArgumentException` at construction when the input contains duplicate keys. When a dispatch table is initialized from a `[SmartEnum]` items collection, the smart enum guarantees key uniqueness and eliminates the runtime duplicate check.

[RETRYWHILEUNTIL_PREDICATE_DISPATCH]:
- `retryWhile(K<IO,A>, Func<Error,bool>)` and `retryUntil(K<IO,A>, Func<Error,bool>)` fire immediately on each failure without delay. The three-argument schedule form `retryWhile(Schedule, K<IO,A>, Func<Error,bool>)` combines schedule-based delay with a predicate guard: both the predicate and the schedule must permit a retry for it to proceed. `repeatWhile(Schedule, K<IO,A>, Func<A,bool>)` is the success-value counterpart, inspecting the successful result and re-running while the predicate holds.
- `retryWhile` continues while the predicate returns `true`; `retryUntil` continues until the predicate returns `true`. Both evaluate the predicate after each failure, receiving the `Error` value. For value-dependent retry, use `repeatWhile` which evaluates a predicate against the success value, not the error.
