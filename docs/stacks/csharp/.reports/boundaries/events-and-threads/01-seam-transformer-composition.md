# Seam Transformer Composition

[ONE_TRANSFORMER_OWNS_THE_SEAM]:
- Attach, detach, marshal, retry, and fault-fence are not five wiring sites; they are one effect transformer composed by arrow juxtaposition. The bracket arrow owns acquire-project-release as a unit, the catch arrow owns the typed fault rail, the finally arrow owns idempotent teardown, and the schedule-driven retry arrow owns reactivation cadence — composing them yields a single `K<M, A> -> K<M, A>` function that any concern threads through, replacing the enumerated detach-then-dispose sequencing the unstructured form repeats at every seam.
- The trait set the seam binds is structural, not nominal: the unlift trait supplies `MapIO`, `ToIO`, and the bracket arrow; `Fallible<Error, M>` supplies the predicate-keyed catch arrow; `Final<M>` supplies the finally arrow. A seam that needs attach + marshal + retry + fault-fence + ordered teardown is the intersection of these traits — declare the constraint set once and the whole composition is available generically, so the capsule never names the concrete effect type in its body. The constraint list is also the seam's capability manifest: the signature states exactly which concerns the transformer composes.
- Ordered teardown is encoded in the bracket's finalizer arm, not inferred from scope-collection iteration order, which is hash-trie order and therefore undefined. The finalizer arm runs a sequential effect chain — detach the subscription token first, then release the handle — so the order is a property of the composed effect, not of the disposal walk. A racing callback can never touch a handle mid-teardown because the detach effect sequences before the dispose effect inside one bracket finalizer.
- The fault-fence is the catch arrow keyed on a predicate over the error rail, placed once around the marshaled body. Host invoke primitives swallow exceptions thrown inside a posted delegate; the marshal transform's internal capture-and-rethrow restores the throw at the awaiting fiber, and the surrounding catch then routes it to the seam's typed failure. Marshaled work that is not threaded through this composed transformer has no ambient propagation path and silently drops faults — the rule is that no marshaled effect leaves the capsule except through the composed transformer.
- Reactivation builds a fresh transformer instance rather than re-running attach against a retained one. The bracket's child resource scope is constructed per activation; the subscription set is the scope, so deactivation that releases the scope destroys the set atomically, and the next activation's bracket constructs a new scope. There is no append-to-retained path, which is the structural cure for cross-cycle subscription accumulation.
- `toIO` projects the inner `IO` out of the carrier as a value, which is the move that lets a seam fork its host-event delivery onto a dedicated thread and re-bind the awaited result back into the same `M` — `Fork`/`Await` operate on the extracted `IO`, and the surrounding `M`-shaped pipeline never learns that a thread boundary was crossed.

[SCHEDULE_AS_REACTIVATION_POLICY_ALGEBRA]:
- Retry and repeat cadence is a `Schedule` value, not enumerated attempt logic. `Schedule` is a lazy `Iterable<Duration>` stream with a closed algebra over two participant kinds: between two schedules, `|` is union — each step takes the minimum of the two durations and the composed stream runs to the length of the *longer* schedule — and `&` is intersect — the composed stream runs only while *both* continue, the shorter envelope; `+` is sequential append. Between a schedule and a `ScheduleTransformer`, both `|` and `&` collapse to the same thing — applying the transformer to the schedule — so the operator choice is semantic only between two schedules and purely notational when a transformer is an operand.
- Base rows are `exponential(seed, factor)`, `fibonacci(seed)`, `spaced(space)`, `fixedInterval(interval)`; `ScheduleTransformer` rows — `recurs(n)`, `maxDelay(cap)`, `jitter(factor)` — are `Schedule -> Schedule` arrows, so `exponential(seed) | recurs(6)` reads as "the exponential delay sequence, capped to six iterations," and `| maxDelay(cap)` as "with each delay clamped." The policy is a derived value built from one primary cadence and secondary clamps, never a flag set whose combinations the retry loop re-derives.

```csharp
var reactivation = Schedule.exponential(20 * ms) | Schedule.recurs(6) | Schedule.maxDelay(2 * sec) | Schedule.jitter(0.25);
K<M, A> Activate<M>(K<M, A> attach) where M : MonadUnliftIO<M> =>
    MonadUnliftIO.mapIO(io => io.Bracket().Retry(reactivation), attach);
```

- A union of a host-availability probe schedule with a bounded retry schedule expresses "keep probing on the host's recovery cadence, but give up only after the bounded envelope" as a single `Schedule` — the longer-of-two semantics of schedule-on-schedule `|` is the precise tool when one stream is open-ended polling and the other is a hard ceiling. Intersect is the tool when the seam must honor the stricter of two cadences, such as a frame-rate ceiling intersected with a back-off floor.
- `Run()` is the only place the duration stream materializes; it is consumed by the carrier's `Repeat`/`Retry` arrow, which threads accumulator state across iterations without re-entering attach. Because the schedule is a pure value, the same reactivation policy is reused across every seam instance and is unit-testable in isolation by enumerating `Run()` — no live host is required to prove the cadence.

[LIFTING_THE_HOST_EVENT_INTO_THE_CARRIER]:
- A host event that fires on a dedicated thread enters the carrier through an async lift over a reusable zero-allocation awaiter, not through a per-event task allocation. The awaiter's reset-before-wait discipline and version token are owned inside the lifted effect; the carrier sees only `K<M, bool>` and never the underlying value-task-source mechanics. The marshal arrow then posts that lifted event back onto the host thread when the consumer must run there, and `toIO` extracts it for forked off-thread consumption — the same lifted event, two carrier-level destinations, zero rewrites.
- The marshal arrow is viable only off the high-frequency path. Each post allocates a wait handle and performs multiple kernel transitions, so per-frame or per-event marshaling collapses throughput; the carrier instead writes the lifted event into a channel state cell and marshals one batch drain per tick. The transformer that performs this is the repeat arrow over a body that drains the channel, driven by the tick schedule — one composed value owns both the cadence (the schedule) and the marshal point (the drain), replacing a hand-written tick loop with inline post-per-item.
- The context-bearing reads — sync context, token, source — are effects evaluated at the composition root on the correct host thread and injected into the environment once; the ambient-context accessors are property reads that re-capture at access time, so invoking them on a background thread captures the wrong context. The seam transformer must seal the environment on the host thread before any fork, and the carrier then propagates the sealed context to every layer through the marshal arrow without re-reading.
- The structural guard against host-thread deadlock is that the seam crosses to the host only from a fiber that does not itself hold the host thread. The marshal arrow uses fire-and-queue posting with a wait-handle rendezvous; a synchronous send from a fiber already occupying the host context is a self-deadlock the type system cannot see. The capsule encodes the guard by sealing the host context at the root and forking blocking work off it before any marshal, so the marshaling fiber is always distinct from the host fiber by construction.
