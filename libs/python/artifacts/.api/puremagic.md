# [PY_ARTIFACTS_API_PUREMAGIC]

`puremagic` is the pure-Python content-sniffing engine that is the DEFAULT identity path of the artifacts file-control rail — a self-contained `magic_data.json` signature table (1485 header rows, 8 footer rows, 79 extension-only rows, 83 multi-part header/footer rows) plus a `scanners/` deep-scan layer, with NO native libmagic, NO `.mgc` loader, NO C extension and NO data-file provisioning. It carries the head/footer of bytes/path/stream against the magic table, returns a confidence-ranked `PureMagicWithConfidence` roster (the `byte_match`/`offset`/`extension`/`mime_type`/`name`/`confidence` namedtuple), folds the convenience `from_file`/`from_string`/`from_stream` rows down to one cooked extension-or-MIME string, and — this is its categorical-best axis over libmagic — disambiguates the structurally-ambiguous `PK\x03\x04` ZIP and `\xd0\xcf\x11\xe0` CFBF containers into the EXACT OOXML/ODF/EPUB/USDZ subtype (`single_deep_scan` reads `[Content_Types].xml` / the CFB stream+CLSID directory) where libmagic floors a `.docx` to `application/zip` and a legacy `.doc` to `application/CDFV2`. The `exchange/detect#DETECT` owner composes `puremagic` as the on-loader-path default sniffer (no `to_process` worker crossing, because there is no native dependency to reify in a subprocess), holding `python-magic`/libmagic only where its compiled magic database is strictly broader on a leaf signature; the owner reads the `PureMagicWithConfidence` roster into its `DetectIdentity`/`MediaClass`/`Container`/`Trust` fold and never re-implements the signature table, the multi-part header+footer correlation, the per-container deep-scan, or the confidence arithmetic the module already owns.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: the match record and fault rail
- rail: file control

The match record is a `namedtuple`, so the owner reads it positionally or by field and folds it into the page's `DetectIdentity` with `._asdict()`; the deep-scan layer carries its own `scanners.helpers.Match` dataclass internally and the public functions re-wrap it as `PureMagicWithConfidence` before it crosses the module boundary, so the owner only ever sees the namedtuple shape. The fault rail is two narrow exceptions — `PureError` (a `LookupError`: no signature matched) and `PureValueError` (a `ValueError`: empty input) — which the owner lifts to the file-control fault rail at the boundary, never letting them escape as bare exceptions into domain logic. The package `__all__` re-exports `PureError`/`PureMagic`/`PureMagicWithConfidence` from the top-level `puremagic`; `PureValueError` is exported only from the `puremagic.main` implementation module, so the owner imports it as `from puremagic.main import PureValueError` and traps both at the seam regardless.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: cooked single-answer detection
- rail: file control

The `from_*` family is the one polymorphic detection surface over the three source shapes (bytes/str, path, open stream), each returning ONE cooked `str`. There is no flag-pinned cookie as in libmagic. Every row returns the file extension by default, the MIME type under `mime: bool = False` (`from_extension` defaults `mime=True`); a path is `os.PathLike | str`; and `filename: os.PathLike | str | None = None` on `from_string`/`from_stream` (the `…` tail below) enriches the magic match with the extension hint AND (when it names a real file) triggers the deep-scan layer. The owner calls `from_string` on the canonical in-memory `Buffer` (admission already holds the bytes) and `from_file` only on an on-disk `File`; `filename=` is how a buffer-sourced payload still reaches the deep-scan when its bytes were spilled to a temp path.

| [INDEX] | [SURFACE]           | [CALL_SHAPE]                                | [CAPABILITY]                                                        |
| :-----: | :------------------ | :------------------------------------------ | :------------------------------------------------------------------ |
|  [01]   | `from_string`       | `from_string(string: str \| bytes, …)`      | in-memory bytes/str; canonical `Buffer`; `filename=` adds deep-scan |
|  [02]   | `from_file`         | `from_file(filename, mime=False)`           | a path (head+foot read, ext hint, deep-scan); the `File` row        |
|  [03]   | `from_stream`       | `from_stream(stream, …)`                    | seekable stream (head, foot-seek, rewind); fsspec-tolerant          |
|  [04]   | `from_extension`    | `from_extension(extension: str, mime=True)` | reverse: ext → MIME/name; longest sig wins                          |
|  [05]   | `ext_from_filename` | `ext_from_filename(filename)`               | parse on-disk ext, recovering a double ext (`.tar.gz`)              |

[ENTRYPOINT_SCOPE]: ranked match-roster detection
- rail: file control

The `magic_*` family returns the FULL confidence-ranked `list[PureMagicWithConfidence]` (highest first) rather than the single cooked string — this is the roster the `Detect` owner reads to fold MIME + extension tuple + human name + the `keep_going`-equivalent multi-match set in one call, the structural parallel of holding a multi-flag libmagic cookie. Every row returns `list[PureMagicWithConfidence]`; `filename: os.PathLike | str | None = None` on the source rows runs the deep-scan and folds its exact resolution (`confidence == 1.0`) to the front, and a pure-buffer call with no filename returns the table matches only. `magic_file`/`magic_string`/`magic_stream`/`magic_extension` are in the top-level `puremagic` `__all__`; `identify_all` (the no-I/O core matcher row `[05]`) is NOT re-exported and is reached as `puremagic.main.identify_all` only when the owner pre-slices its own head/foot.

[ENTRYPOINT_SCOPE]: head/foot slicing and the deep-scan layer
- rail: file control

These are the lower primitives the owner reaches only to control I/O and the container-disambiguation pass directly. They are NOT in the top-level `puremagic` `__all__` — each is imported from the `puremagic.main` implementation module (`from puremagic.main import single_deep_scan, file_details, get_max_lengths, magic_data`), so the catalog names the module-qualified path the owner binds. The three `*_details` rows return `(head, foot)` bytes; `file_details`/`string_details`/`stream_details` slice exactly `get_max_lengths()` bytes off each end (head = longest header signature + offset, foot = longest footer span) so a multi-gigabyte payload is identified from a bounded head+foot read, never a full load. `single_deep_scan(bytes_match: bytes | bytearray | None, filename, head=None, foot=None, confidence=0) -> scanners.helpers.Match | None` is the categorical-best disambiguator (full signature here, compact head below): it `match`es the leading signature bytes to a per-container scanner (`zip`/`pdf`/`cfbf`/`ebml`/`ogg`/`asf`/`mpeg_audio`/`sndhdr`) and otherwise tries the `pdf`/`python`/`json`/`hdf5` first-match chain (the `text` catch-all running via the separate `catch_all_deep_scan`). `magic_data(filename=<bundled magic_data.json>)` returns `(headers, footers, extension_only, multi_part)`. The env knob `PUREMAGIC_DEEPSCAN=0` disables the entire layer (every `from_*`/`magic_*` falls back to table-only), so an untrusted or latency-bounded ingest can switch it off without changing the call.

| [INDEX] | [SURFACE]          | [CALL_SHAPE]                                 | [CAPABILITY]                                                        |
| :-----: | :----------------- | :------------------------------------------- | :------------------------------------------------------------------ |
|  [01]   | `get_max_lengths`  | `get_max_lengths() -> (int, int)`            | bounded byte budget each end is sliced to; folds multi-part offsets |
|  [02]   | `file_details`     | `file_details(filename)`                     | `(head, foot)` from a path; `PureError` on a non-regular file       |
|  [03]   | `stream_details`   | `stream_details(stream)`                     | `(head, foot)` from a seekable stream, rewind; fsspec-tolerant      |
|  [04]   | `string_details`   | `string_details(string)`                     | `(head, foot)` off an in-memory bytes payload — no I/O              |
|  [05]   | `single_deep_scan` | `single_deep_scan(bytes_match, filename, …)` | route leading bytes to the container scanner for the exact subtype  |
|  [06]   | `magic_data`       | `magic_data(filename=…)`                     | load a custom or the bundled signature DB                           |

[ENTRYPOINT_SCOPE]: the deep-scan result shape and container scanners
- rail: file control

The deep-scan layer is the strict-stronger surface over libmagic — a `scanners/` package of per-container resolvers each exposing a `match_bytes` signature constant and a `main(file_path, head, foot) -> Match | None`; every row below is a `scanners.<name>` module. `scanners.helpers.Match(extension, name, mime_type, confidence=1.0)` is the deep-scan result the public roster re-wraps as `PureMagicWithConfidence` before the boundary. The `cfbf_scanner` magic is `b"\xd0\xcf\x11\xe0\xa1\xb1\x1a\xe1"` (short `b"\xd0\xcf\x11\xe0"`). The owner never calls a scanner directly (the `from_*`/`magic_*` rows route them), but the `zip_scanner` OOXML map and the `cfbf_scanner` legacy-Office resolution are exactly the `Container.ZIP`/`Container.OLE` → exact-subtype disambiguation the `detect.md` owner's `MediaClass`/`Container` fold consumes. The `zip_scanner` resolves the OOXML `.docx`/`.xlsx`/`.pptx`/`.docm`/`.xlsm`/`.xlsb`/`.dotx` set (plus ODF/EPUB/USDZ/CBZ); the `cfbf_scanner` resolves legacy `.doc`/`.xls`/`.ppt`/`.msi`/`.msg`.

| [INDEX] | [SCANNER]                    | [SIGNATURE]               | [RESOLVES]                                                              |
| :-----: | :--------------------------- | :------------------------ | :---------------------------------------------------------------------- |
|  [01]   | `zip_scanner`                | `b"PK\x03\x04"`           | `[Content_Types].xml` → OOXML set, ODF/EPUB/USDZ/CBZ                    |
|  [02]   | `cfbf_scanner`               | `b"\xd0\xcf\x11\xe0"…`    | CFB stream + CLSID → legacy Office (libmagic: `application/CDFV2`)      |
|  [03]   | `ebml` / `ogg` / `asf`       | per-format `match_bytes`  | Matroska/WebM, Ogg stream, ASF/WMV container subtype                    |
|  [04]   | `mpeg_audio` / `sndhdr`      | per-format `match_bytes`  | MPEG-audio frame, AU/HCOM/FSSD/SNDR audio subtype                       |
|  [05]   | `pdf`/`json`/`hdf5`/`python` | first-match chain         | confirm a true PDF/JSON/HDF5/Python payload past a generic signature    |
|  [06]   | `text_scanner`               | `decode_any`; `eml_check` | charset (utf-8/cp1252), line-terminator + CSV + RFC-822 `.eml` recovery |

## [04]-[IMPLEMENTATION_LAW]

[CONTENT_IDENTITY]:
- import: `import puremagic` at boundary scope only; module-level import is banned by the manifest import policy. Unlike `python-magic` there is NO libmagic-absent guard to write — the package is pure-Python with a bundled `magic_data.json`, so the import cannot fail for a missing native library; an `ImportError` here is a genuine packaging fault, not a host-provisioning gap.
- loader-path law: `puremagic` runs IN-PROCESS on the event loop. The `exchange/detect#DETECT` owner composes it WITHOUT the `anyio.to_process.run_sync` `WORKER_BAND` crossing the `python-magic` path requires — there is no native dependency to reify in a subprocess, no `BrokenWorkerProcess` death to recover, no pickle of the payload across a process seam. A bounded head+foot read (`get_max_lengths()`) keeps even a multi-gigabyte path off the heap, so the in-process call is cheap; only a deliberately latency-bounded ingest moves it to a thread via `anyio.to_thread.run_sync`, never a process.
- source axis: bytes, path, and stream are three rows on the one `from_*`/`magic_*` surface, never parallel detector types; the in-memory `from_string`/`magic_string` row is canonical because admission already holds the payload, and the `filename=` keyword is how that buffer row still reaches the deep-scan (it spills to a temp path only when the disambiguation demands a full re-read the head+foot slice cannot serve).
- output axis: the single cooked string (`from_*`) and the ranked roster (`magic_*`) are the two read shapes — `mime=True` selects MIME over extension on the cooked row, and the roster carries `byte_match`/`offset`/`extension`/`mime_type`/`name`/`confidence` per match. There is no per-output flag cookie: a full identity is ONE `magic_string`/`magic_file` call returning the roster the owner folds, with the highest-confidence head as the primary and the tail as the multi-match (`Trust.AMBIGUOUS`) evidence.
- deep-scan axis: `single_deep_scan` IS the categorical-best capability — it disambiguates the `PK\x03\x04` ZIP family (OOXML via `[Content_Types].xml`, ODF, EPUB, USDZ, CBZ) and the `\xd0\xcf\x11\xe0` CFBF compound (legacy Office via CFB stream+CLSID inspection) to the EXACT subtype, plus EBML/Ogg/ASF/MPEG-audio/AU/JSON/HDF5/Python/EML/text confirmation, where the `python-magic` libmagic database floors a `.docx` to `application/zip` and a `.doc` to `application/CDFV2`. A deep-scan hit folds in at `confidence == 1.0`. The env knob `PUREMAGIC_DEEPSCAN=0` is the one runtime switch (read by every `from_*`/`magic_*` entry) the owner sets through `os.environ` for an untrusted-ingest table-only pass, never a per-call argument.
- fault axis: a no-match raises `PureError` (a `LookupError`) on `from_*`/`from_extension`; empty input raises `PureValueError` (a `ValueError`) on `perform_magic`/`magic_*`. The ranked `magic_file`/`magic_string`/`magic_stream` rows run `raise_on_none=False`, so an unidentifiable payload returns the table matches (or `[]`) rather than raising — the owner reads an empty/low-confidence roster as `MediaClass.UNKNOWN`. Lift `PureError`/`PureValueError` to the file-control fault rail once at the boundary.
- evidence: each detection captures source kind, the highest-confidence MIME, the extension, the human name, the confidence score, and the multi-match tail as a content-identity receipt folded into the runtime `DetectIdentity`, never a re-minted identity; the float confidence is the page-native signal `python-magic` lacks (libmagic returns one unranked string), so a low-confidence head is the `Trust` gate's ambiguity input rather than a silently-trusted answer.
- boundary: `puremagic` owns content sniffing only. It mints no `ContentKey` (the `evidence/identity#IDENTITY` content hash is a producing owner's concern) and contributes no `ArtifactReceipt`. Its resolved identity is the admission gate the per-format readers dispatch on — `python-docx`/`python-pptx`/`openpyxl` read the exact OOXML subtype the `zip_scanner` resolved, `odfpy` the ODF subtype, `pymupdf`/`pypdf` the `application/pdf` branch, `pillow`/`pyvips` the image branches, `msoffcrypto-tool` the legacy/encrypted-Office branch the `cfbf_scanner` resolved — and the descriptive metadata INSIDE the identified container is the `exchange/metadata#METADATA` owner's concern, never re-read here. The `magic_data(filename=...)` custom-DB hook stays unused unless a private signature must be added; the bundled table is the default.

[INTEGRATION_STACK]:

- categorical-best vs `python-magic` (the supersession evidence the brief demands per-page): on the DEFAULT path `puremagic` is strictly stronger on four axes — (1) NO native libmagic / `.mgc` / data-file provisioning (pure-Python, bundled `magic_data.json`), so it rides the loader path with no `to_process` seam; (2) the `zip_scanner`/`cfbf_scanner` deep-scan resolves OOXML/ODF/EPUB/USDZ + legacy-Office to the EXACT subtype where libmagic floors to the generic container; (3) the float `confidence` ranking gives the `Trust` gate a native ambiguity signal; (4) multi-part header+footer correlation and double-extension recovery are in the data table. `python-magic`/libmagic is retained ONLY where its broader compiled database recognizes a leaf signature `magic_data.json` lacks (the long tail of niche binary formats) — a strict-stronger-on-a-distinct-axis retention under the categorical-best mandate, not an overlap; the two never both own the default sniff. The `python-magic` `mime_encoding` charset facet maps to `puremagic`'s `text_scanner.decode_any` `(text, encoding)` return on the default path; the `MagicParam` recursion/ELF-bomb caps have no `puremagic` analogue (it has no recursive magic-indirection engine to bomb — a smaller attack surface, not a missing capability), and an untrusted ingest sets `PUREMAGIC_DEEPSCAN=0` to skip the container scanners entirely.
