# [CLAIM_DISCIPLINE]

Durable documentation states only claims the artifact owns. Research, command output, package pages, transcripts, review notes, and model runs are working material; the final artifact carries the resulting rule, contract, status, or gap.

## [1]-[USE_WHEN]

Apply this standard when documentation states:
- commands, flags, outputs, quality gates, or expected success signals
- package, platform, runtime, language, host, API, or support behavior
- generated contracts, schemas, public API surfaces, diagrams, codemaps, or machine-facing indexes
- support status, retirement state, security posture, operational procedure, rollback, escalation, or recovery behavior
- artifact replacement, relocation, or deletion.

## [2]-[CLAIM_LAW]

State a confirmed fact plainly. Cut unsupported claims or turn them into explicit owner-local gaps. Do not attach external reference, origin trail, refresh, research-log, or confirmation-tail sections to ordinary docs.

[CLAIM_FORMS]:
- Stable rule: direct prose.
- Concrete artifact behavior: local owner record beside the artifact.
- Generated artifact behavior: generated artifact record with owner and refresh trigger.
- Missing confirmation: gap record with close trigger.
- Replacement work: preservation record.

[OMISSION_RULE]:
- Omit research path, package page, search query, prompt, transcript, and review history.
- Omit dates unless the artifact’s own domain is dated.
- Omit version narration unless the artifact’s job is a concrete support or compatibility matrix.
- Omit external references from reusable docs, prompts, standards, skills, examples, and templates.

## [3]-[CONFIDENCE_ORDER]

When claims conflict, use the material closest to the executing system:
1. generated contracts, manifests, lockfiles, schemas, project files, checked code, and checked generated API output
2. executed local command, render, scenario, build, test, or tool output
3. maintained platform or package documentation
4. maintained release notes, migration notes, or issue records.

The final artifact states the winning behavior. It does not narrate the conflict unless the document’s purpose is the conflict record itself.

## [4]-[FIELD_SHAPES]

Use fields only when an agent or machine consumer acts on them.

| [INDEX] | [FIELD]            | [ADD_WHEN]                                                       |
| :-----: | :----------------- | :--------------------------------------------------------------- |
|   [1]   | `Owner:`           | one manifest, generator, contract, route, or tool owns the claim |
|   [2]   | `Generated by:`    | an artifact is regenerated from an owning route                  |
|   [3]   | `Observed result:` | a local result changes whether the claim is accepted             |
|   [4]   | `Gap:`             | confirmation is missing and the claim must remain bounded        |
|   [5]   | `Refresh trigger:` | a concrete event requires re-checking or regeneration            |
|   [6]   | `Preserved:`       | replacement moved load-bearing content                           |

Do not publish empty fields. Do not put field blocks at page tail unless the page is a machine artifact whose consumer reads them there.

[GENERATED_ARTIFACT_RECORD]:
- Artifact: `<artifact>`.
- Owner: `<owner>`.
- Generated by: `<generator-or-route>`.
- Refresh trigger: `<owner-change>`.

[GAP_RECORD]:
- Claim: `<bounded-claim>`.
- Gap: `<missing-confirmation>`.
- Close when: `<observable-condition>`.

## [5]-[PLACEMENT]

Place confidence fields beside the claim, row, diagram, command, generated artifact, or procedure they affect.

[PLACEMENT_LAW]:
- Row-specific confidence belongs in a row-owned record.
- Diagram confidence belongs in the caption or adjacent record.
- Generated artifact confidence belongs beside the generated artifact record.
- A page-level record is valid only when one owner and one refresh trigger cover every material claim on the page.

## [6]-[UNCERTAINTY]

Do not soften claims with hedges. A fact is stated as law, narrowed to its exact scope, or cut.

[GAP_LAW]:
- Use `Gap:` only when the missing confirmation changes action.
- Give every gap a close condition.
- Do not use `n/a`, `unknown`, `maybe`, `probably`, `latest`, `current`, `future`, or similar filler as a substitute for a bounded claim.
- Preserve qualifiers such as `optional`, `if present`, and `where supported` only when the qualifier is part of the contract.

## [7]-[PRESERVATION]

A refactor relocates content; it never drops load-bearing current facts.

[PRESERVATION_RECORD]:
- Prior location: `<file-or-section>`.
- Replacement scope: `<file-or-section>`.
- Preserved: `<facts-retained>`.
- Routed: `<facts-moved-to-owner>`.
- Deleted: `<stale-or-non-owned-facts-removed>`.
- Result: `<preserved-routed-deleted-or-blocked>`.

Use the record only when replacement is broad enough that a normal diff no longer proves the carried facts at a glance.

## [8]-[AGENT_SURFACES]

Machine-facing surfaces are contracts when they affect retrieval, generated mirrors, tool selection, structured output, or automation behavior.

[DETERMINISTIC_EVALUATION]:
- Surface: `<machine-facing-surface>`.
- Baseline: `<prior-behavior-or-known-miss>`.
- Checks: `<checks-run-or-reviewed>`.
- Observed result: `<result>`.
- Refresh trigger: `<change-that-invalidates-the-result>`.

[STOCHASTIC_EVALUATION]:
- Surface: `<ranking-or-selection-surface>`.
- Case set: `<representative-task-set>`.
- Repeat rule: `<repeat-policy>`.
- Comparison: `<accepted-comparison-method>`.
- Observed result: `<result>`.
- Refresh trigger: `<change-that-invalidates-the-result>`.

Use stochastic fields only when the claim depends on ranking, selection, latency, tool choice, or provider behavior that one deterministic check cannot settle.

## [9]-[BOUNDARIES]

- `information-structure.md` carries the container that presents records, tables, diagrams, and machine-consumed Markdown.
- `style-guide.md` carries phrasing and removes hedging, report leads, and meta narration.
- `formatting.md` carries marker syntax, field rendering, fence notation, and heading style.
- `README.md` carries standards routing.
