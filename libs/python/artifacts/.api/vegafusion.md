# [PY_ARTIFACTS_API_VEGAFUSION]

`vegafusion` supplies the Rust-backed (embedded Apache DataFusion) server-side Vega transform engine for the artifacts charts rail: a module-level `runtime` singleton (`VegaFusionRuntime`, a multi-threaded worker pool) whose `pre_transform_*` family server-evaluates a Vega spec's data transforms (aggregations, joins, binning, filtering) before render, a `ChartState` that maintains the interactive server/client transform split and applies signal/dataset updates, the `vegafusion.utils` `build_pre_transform_spec_plan` planner diagnostic and `get_column_usage` referenced-column analyzer, and a `transformer` submodule that converts a dataframe to Arrow IPC `bytes`/`pa.Table` for the inline-dataset path. The package serves TWO disjoint consumer planes: the chart-render pre-pass (`visualization/chart/export#PREPASS` folds `runtime.pre_transform_spec` for the static spec and `runtime.new_chart_state(...).get_transformed_spec()` for the host-free interactive HTML row â€” the DataFusion reduction inlines INSIDE one self-contained spec because `vl-convert` exposes no external-dataset feed), and the columnar egress plane (`pre_transform_extract`'s `(name, scope, table)` 3-tuples, the `transformer` Arrow-IPC serializers, and `get_column_usage`, which belong to the `data/tabular/columnar#COLUMNAR` egress owner, never a chart-render side channel). The package owner never re-implements the Vega transform pipeline the embedded DataFusion engine already owns. The producer folds onto the universal rails: each pre-pass contributes one `msgspec` (`libs/python/.api/msgspec.md`) charts receipt under one `structlog` (`libs/python/.api/structlog.md`) event inside an `opentelemetry` (`libs/python/.api/opentelemetry-api.md`) span, a malformed spec or `PreTransformWarning` folds onto the `expression.Result` (`libs/python/.api/expression.md`) rail, and the GIL-releasing native pre-pass crosses the subprocess seam via `anyio` (`libs/python/.api/anyio.md`) `to_process`.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: runtime and chart-state roots
- rail: charts

The module `__all__` exports exactly `runtime` (the singleton `VegaFusionRuntime` *instance*), `set_local_tz`, `get_local_tz`, and `get_column_usage`. `ChartState`, `VariableUpdate`, `Variable`, `CommPlan`, and `PreTransformWarning` are REAL `class`/`TypedDict` definitions in `vegafusion.runtime`; `PreTransformSpecPlan` is a `TypedDict` in `vegafusion.utils`. The native core (`vegafusion._vegafusion`) exports ONLY `PyChartState`, `PyVegaFusionRuntime`, `build_pre_transform_spec_plan`, `get_column_usage`, `get_cpu_count`, and `get_virtual_memory` â€” `PyChartStateGrpc` is a `TYPE_CHECKING`-only annotation in `ChartState.__init__` and is NOT importable at runtime in this wheel (gRPC routing is `PyVegaFusionRuntime.new_grpc(url)`, reached through `runtime.grpc_connect`). The reachability trap is verified: `vegafusion/__init__.py` does `from .runtime import runtime`, rebinding the package attribute `runtime` to the singleton *instance*, which shadows the auto-bound submodule â€” so `vegafusion.runtime.ChartState` raises `AttributeError` (the parent attribute is the `VegaFusionRuntime` instance, not the module), while `vegafusion.runtime.pre_transform_spec` resolves correctly ON that instance. The `ChartState` class itself is reachable only through `sys.modules["vegafusion.runtime"]` / `importlib.import_module("vegafusion.runtime")`, or by typing against the structural shapes; `ChartState` is normally obtained as the `runtime.new_chart_state(...)` return value, never imported. The `vegafusion.utils` helpers (`build_pre_transform_spec_plan`, `get_inline_column_usage`) and the `vegafusion.transformer` Arrow-IPC family are reached through their submodules directly (not shadowed).

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: `runtime` transform execution
- rail: charts

`spec` accepts a `dict` or JSON `str`. The transform rows share `local_tz` (defaults to `get_local_tz()`, which falls back to `vl_convert.get_local_tz()` then `"UTC"`), `default_input_tz`, and `inline_datasets` (a `dict[str, DataFrameLike]` of named tables resolved against a `data.url` of `vegafusion+dataset://{name}` or `table://{name}`). The runtime auto-prunes inline-frame columns to those the spec references (`get_inline_column_usage(spec)` is applied internally on every `pre_transform_*`/`new_chart_state`). `pre_transform_spec`/`pre_transform_extract` carry `preserve_interactivity` + `keep_signals`/`keep_datasets` (variables retained client-side, each a `name` or `(name, scope)`); `pre_transform_datasets` instead takes an explicit `datasets` selector list and has no interactivity flags. The runtime is a lazily-initialized multi-threaded worker pool (default `cache_capacity=64`, `memory_limit=virtual_memory//2`, `worker_threads=cpu_count`); cache/memory/thread caps are read/write properties whose SETTERS restart the pool (`reset()`), never a fresh runtime per transform.

[ENTRYPOINT_SCOPE]: `ChartState`, planner, and transformer helpers
- rail: charts

`ChartState` maintains the server/client spec split and applies interactive updates; `get_comm_plan` and `get_watch_plan` (an explicit backwards-compat alias) are the two names for the one `CommPlan` signal/dataset dependency graph that drives incremental `update`. `build_pre_transform_spec_plan` (in `vegafusion.utils`) is the diagnostic that returns the SAME planned `client_spec`/`server_spec`/`comm_plan` split `pre_transform_spec` uses, without executing â€” the chart owner can read it to predict the server/client partition before paying for transform execution. The `transformer` family + `get_column_usage` belong to the columnar egress plane: `to_arrow_table` is pandas-specific (it inspects `data.dtypes`, expands `pd.CategoricalDtype`, batches at `BATCH_SIZE=8096`), while `to_arrow_ipc_bytes` accepts pandas / pyarrow / any `__dataframe__`-interchange object.

## [04]-[IMPLEMENTATION_LAW]

[CHARTS_PRE_PASS]:
- transform axis: `runtime.pre_transform_spec` is the single pre-evaluation surface keyed by spec; `preserve_interactivity`, `keep_signals`, and `keep_datasets` are rows on that surface, never a per-mode transform function. `keep_signals`/`keep_datasets` accept `name` or `(name, scope)` to retain selected interactive variables client-side. The DataFusion-backed transforms execute server-side and the REDUCED result inlines INTO the returned spec â€” the aggregation IS the size reduction, so `vl-convert` renders one self-contained spec with no client-side transform work and no oversized embedded data.
- state axis: `new_chart_state` opens a `ChartState` for the host-free interactive HTML row; `get_transformed_spec` yields the fully-transformed self-contained spec the renderer needs with no live server. The server/client split (`get_server_spec`/`get_client_spec`), `update` (a `list[VariableUpdate]` round-trip), and the `get_comm_plan`/`get_watch_plan` dependency graph (one `CommPlan`) are the interactive rows, never a duplicated spec per frame. Type-annotate against the `runtime` module reached through `sys.modules`/`importlib`, never `vegafusion.runtime.ChartState` (shadowed by the singleton instance).
- planner axis: `build_pre_transform_spec_plan` is the diagnostic projection of the same plan â€” read `client_spec`/`server_spec`/`comm_plan` to predict the partition before executing; it is one diagnostic row, never a parallel planning engine.
- resource axis: the runtime is a lazily-initialized worker pool; `worker_threads`/`cache_capacity`/`memory_limit` are properties whose setters `reset()` the pool only on a changed value, `clear_cache` reclaims without dropping the pool, `reset` drops it for re-init, and `size`/`total_memory` introspect â€” caps are properties on the singleton, never a fresh runtime per transform.
- evidence: each pre-pass captures spec identity, row limit, transformed/inlined dataset count, timezone, worker/cache state, and the collected `PreTransformWarning` list as a charts receipt folded onto the universal `msgspec`/`structlog`/`opentelemetry` rails.
- boundary: vegafusion owns server-side Vega transform execution; `altair` produces the input Vega-Lite spec; `vl-convert-python` compiles it to Vega and rasterizes (the SOLE static render path â€” it exposes NO external-dataset feed, so the reduction must cross inside the spec); the `vegafusion.jupyter` `VegaFusionWidget` and any live UI stay outside this rail.

[COLUMNAR_EGRESS]:
- extraction axis: `pre_transform_datasets` returns explicitly named server-computed tables (the `datasets` selector list, each `name` or `(name, scope)`) as native frames in `dataset_format` (`auto`/`polars`/`pandas`/`pyarrow`/`arro3`), each tz-normalized to `local_tz`; `pre_transform_extract` splits the spec from inline tables `>= extract_threshold` rows and returns them as `(name, scope, DATA)` where `DATA` follows `extracted_format` â€” `'arro3'` (default) / `'pyarrow'` give a table, `'arrow-ipc'` gives `bytes`, `'arrow-ipc-base64'` gives a base64 `str`. These belong to the `data/tabular/columnar#COLUMNAR` egress owner, NOT the chart renderer (`vl-convert` cannot consume an external frame map).
- dataset axis: `transformer.to_arrow_ipc_bytes`/`arrow_table_to_ipc_bytes`/`to_arrow_table`/`to_feather` serialize a frame to the Arrow IPC wire; `to_arrow_table` is pandas-specific, the IPC pair accepts pandas/pyarrow/`__dataframe__`-interchange. `inline_datasets` maps a spec `data.url` (`vegafusion+dataset://{name}` / `table://{name}`) to a `DataFrameLike`, so polars/pandas/pyarrow frames all enter through the one internal narwhals path (no per-format serializer).
- pruning axis: `get_column_usage(spec)` reports the columns each root dataset references (`None` when not statically determinable); the egress owner trims wide frames before `pre_transform_datasets(trim_unused_columns=True)` so only used columns cross the Arrow wire. The runtime already applies `get_inline_column_usage` internally to project inline frames on every call.

[STACK_INTEGRATION]:
- `altair` -> `vegafusion` -> `vl_convert` (chart pre-pass): `altair.Chart.to_dict()` (`libs/python/artifacts/.api/altair.md`) yields the Vega-Lite spec; the chart owner's `VegaTransform.apply` (`visualization/chart/export#PREPASS`) folds `runtime.pre_transform_spec(spec)` (the `Inline` arm) so the transforms server-evaluate and the reduced result inlines, or `runtime.new_chart_state(spec).get_transformed_spec()` (the `State` arm) for the host-free interactive HTML row; `vl_convert.vegalite_to_svg`/`vega_to_png` (`libs/python/artifacts/.api/vl-convert-python.md`) then renders that ONE self-contained spec with no client-side transform work and no oversized embedded data. There is no Arrow-IPC side channel into the renderer â€” `vl-convert` carries no `inline_datasets` parameter.
- universal-rail fold: the gated native pre-pass crosses the subprocess seam via `anyio.to_process.run_sync` (`libs/python/.api/anyio.md`) under one `CapacityLimiter` (the GIL-releasing DataFusion core); the resulting `PreTransformWarning` list and dataset facts contribute one `msgspec` (`libs/python/.api/msgspec.md`) charts receipt under one `structlog` (`libs/python/.api/structlog.md`) event inside an `opentelemetry` (`libs/python/.api/opentelemetry-api.md`) span, and a malformed spec or a `ValueError` from `_import_inline_datasets` folds onto the `expression.Result` (`libs/python/.api/expression.md`) rail rather than raising through the pipeline.
- columnar egress fold: a polars/pandas frame the `data` tier produces enters `inline_datasets` directly as a `DataFrameLike` (narwhals resolves the interchange, the owner never branches on the source library); `get_column_usage(spec)` drives pre-flight pruning; `pre_transform_extract(extracted_format='arrow-ipc')` hands `bytes` to the `data/tabular/columnar#COLUMNAR` egress owner for a genuine columnar export.
