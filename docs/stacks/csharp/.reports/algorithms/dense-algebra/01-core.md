# Dense Algebra — Core

[FACTORIZATION_SUITE]:
- Six factorizations are available on every `DenseMatrix<double>` instance as virtual factory methods: `Cholesky()`, `LU()`, `QR(QRMethod)`, `GramSchmidt()`, `Svd(bool computeVectors)`, and `Evd(Symmetricity)`. Each returns its typed abstract handle (`Cholesky<double>`, `LU<double>`, etc.) rather than a concrete sealed class; the concrete `Dense*` implementations are `internal sealed` and unreachable by name. The handle carries all downstream evidence and `Solve(Matrix<T>, Matrix<T>)` / `Solve(Vector<T>, Vector<T>)` operations, so a retained handle yields repeated right-hand-side solves with zero refactorization cost.
- `DenseCholesky.Create` clones the input, then calls `LinearAlgebraControl.Provider.CholeskyFactor` on the clone's `Values` array in place. The provider throws `ArgumentException("Matrix must be positive definite.")` if any pivot diagonal is non-positive during the column-by-column sweep. This makes `Cholesky()` the canonical definiteness probe: wrap in `Fin<Cholesky<double>>` at the admission boundary, and a successful handle is simultaneously a proved-SPD certificate. `Cholesky.Factorize(Matrix<double>)` — the in-place re-factorization method on a retained handle — copies the new matrix into the existing factor buffer via `Buffer.BlockCopy` before calling the provider, enabling efficient pipelines that supply successive near-SPD matrices without allocating a new handle each time.
- `DenseLU.Create` requires `RowCount == ColumnCount`; the factorization performs partial pivoting with row swaps recorded in an `int[] Pivots` array. `LU.Determinant` computes the product of diagonal U entries, negating each term where `Pivots[i] != i`. Non-singularity is not asserted at construction — a singular input produces a handle with `Determinant == 0.0`, not an exception. The correct singularity gate post-construction is `Math.Abs(lu.Determinant) < threshold` or, more robustly, probing `lu.Factors.Diagonal().Any(v => Math.Abs(v).AlmostEqual(0.0))`.
- `DenseQR.Create(DenseMatrix, QRMethod)` dispatches to two internal provider paths. `QRMethod.Full` stores an m×m Q and an m×n R; `QRMethod.Thin` stores an m×n Q and an n×n R. The default on `Matrix.QR()` is `QRMethod.Thin`. The `Tau` array (Householder reflector coefficients, length `min(m,n)`) is retained on the handle and forwarded to `QRSolveFactored`, so the native provider can reconstruct Q's action without materializing it. `QR.IsFullRank` checks that no diagonal entry of `FullR` satisfies `Math.Abs(FullR.At(i, i)).AlmostEqual(0.0)` — an absolute threshold of `10 · 2⁻⁵³` against the matrix's own scale. This makes `IsFullRank` the numerically correct full-rank gate only for well-scaled systems, not for ill-conditioned ones near rank deficiency; see the threshold inconsistency below.
- `DenseGramSchmidt.Create` runs modified Gram-Schmidt in pure managed code and throws `ArgumentException` only when a column norm is `== 0.0` exactly. A near-zero column norm (floating-point cancellation without bit-exact zero) passes the check, causes division by near-zero, and silently populates Q and R with `NaN`/`Inf`. The factorization-succeeded-but-result-is-corrupt failure cannot be detected without an explicit all-finite check on the factor buffer after construction. Unlike Householder QR, this path does not route through the provider and receives no native acceleration.
- `DenseEvd.Create` takes a `Symmetricity` discriminant. `Symmetricity.Unknown` calls `matrix.IsSymmetric()` at factorization time — an O(n²) exact-equality scan with no epsilon. The symmetric path routes to Householder tridiagonalization then implicit QR iteration. The asymmetric path routes to Hessenberg reduction then real Schur form. Eigenvalues are always stored as `Vector<Complex>` regardless of symmetricity; for a symmetric input, imaginary parts are numerically zero but the Complex representation is not stripped. `Evd.Solve` is implemented only for the symmetric case and throws `ArgumentException("Matrix must be symmetric.")` for asymmetric inputs.
- `DenseSvd.Create` always allocates U as m×m and VT as n×n, regardless of `computeVectors`. When `computeVectors = false`, the provider still allocates both full factor arrays but skips their computation; a subsequent `Solve` call throws `InvalidOperationException("The singular vectors were not computed.")`. Requesting vectors is therefore a requirement for any solve or pseudo-inverse use, and `false` is only a valid cost gate when rank, L2Norm, or ConditionNumber evidence is all that is needed.

[SVD_EVIDENCE]:
- `Svd<double>.Rank` computes the tolerance as `S.Maximum().EpsilonOf() * Math.Max(U.RowCount, VT.RowCount)`. `EpsilonOf(x)` returns the ULP epsilon at `x` — for a well-scaled matrix with max singular value near 1, this approximates `σ_max · ε_machine · max(m,n)`, matching the standard rank-revealing threshold. The threshold tracks the actual numerical scale of the problem via the max singular value's own ULP, not a global constant.
- `Svd<double>.ConditionNumber` returns `|S[0]| / |S[min(m,n) - 1]|`, which is `+Inf` for singular or rank-deficient matrices. Guard for this before using the result as a quality gate.
- `Matrix.PseudoInverse()` (on `Double.Matrix`) computes the truncated SVD pseudo-inverse by calling full SVD with vectors, applying a global threshold `max(m,n) · σ_max · Precision.DoublePrecision` to zero small singular values, inverting the retained values, and reconstructing as `(U · W · VT)ᵀ`. This threshold is `max(m,n) · σ_max · 2⁻⁵³`. The `Svd.Rank` threshold uses `σ_max.EpsilonOf() · max(m,n)`, where `EpsilonOf(1.0) = 2⁻⁵²` — exactly twice `Precision.DoublePrecision`. The two surfaces use different precision constants and will disagree on rank-boundary singular values even when reading from the same `S` vector on the same handle.
- `PseudoInverse()` directly mutates `S` via `s[i] = (s[i] < threshold) ? 0.0 : (1.0 / s[i])`, and `W` via `w.SetDiagonal(s)` on the lazy-cached `Matrix<T>.W`. After a `PseudoInverse()` call, `S`, `L2Norm` (which reads `S[0]`), `ConditionNumber`, and `Rank` (which iterates `S`) are all permanently corrupted on the retained handle. The handle must be treated as consumed.

[ADMISSION_GATES]:
- All six `Dense*` factorizations perform squareness and dimension checks at construction and throw `ArgumentException` or `ArgumentNullException` for violated preconditions. None perform all-finite checks; NaN and Inf propagate silently through factorization, producing corrupted factor state without any exception. An admission gate must verify `TensorPrimitives.IsFiniteAll<double>(new ReadOnlySpan<double>(matrix.Values))` before calling any factorization factory.
- `Cholesky()` is the sole correct definiteness admission gate: a successful construction certifies SPD; a thrown `ArgumentException("Matrix must be positive definite.")` is the rejection evidence. Do not substitute a separate eigenvalue probe.
- `DenseMatrix.IsSymmetric()` uses exact `!=` comparison on `double` entries — no epsilon. A matrix built via incremental accumulation (outer-product updates, normal-equation formation) is frequently not bit-exactly symmetric. When passed to `Evd(Symmetricity.Unknown)`, `IsSymmetric()` returns `false` and the EVD falls through to the asymmetric Hessenberg path, which is slower and causes `Evd.Solve` to throw. Force symmetry explicitly via `matrix.Add(matrix.Transpose()).Multiply(0.5)`, or pass `Symmetricity.Symmetric` when the structural guarantee holds.

[RESIDUAL_CAPPED_SOLVE]:
- No built-in residual cap exists on any `Dense*` solve path. The managed solve paths forward directly to in-place triangular solve or back-substitution via the provider without computing a residual. A residual-capped solve must be implemented by the caller: after `factorization.Solve(b, x)`, compute `r = A·x - b` using the original matrix, and demote success to failure when `r.L2Norm() / b.L2Norm() > eps_cap`. `DenseMatrix.Values` exposes the backing `double[]` directly, enabling span-level residual computation via `TensorPrimitives.MultiplyAdd` without boxing.

[LEAST_SQUARES_ROUTES]:
- `MultipleRegression.NormalEquations<T>` calls `x.TransposeThisAndMultiply(x).Cholesky().Solve(x.TransposeThisAndMultiply(y))`. When X is sparse or tall, `TransposeThisAndMultiply` produces a dense n×n Gram matrix regardless of the input structure — a quadratic densification cliff. For an m×n design matrix with m >> n and sparse structure, the normal-equations route discards sparsity, squares the condition number (κ(AᵀA) = κ(A)²), and allocates O(n²) dense storage. `NormalEquations` is the default `DirectRegressionMethod` and the most dangerous for near-rank-deficient systems; the choice must be explicit as a receipt field.
- `MultipleRegression.QR<T>` calls `x.QR().Solve(y)` using `QRMethod.Thin` and handles overdetermined systems without forming the Gram matrix. Query `IsFullRank` on the retained handle for full-rank evidence before accepting the solution.
- `MultipleRegression.Svd<T>` calls `x.Svd(computeVectors: true).Solve(y)` and delivers rank-deficiency-tolerant least squares, but the truncation threshold inside `SvdSolveFactored` is not applied — the SVD solve divides by each singular value without truncation (see SVD_SOLVE_TRUNCATION_GAP). Use `PseudoInverse()` when truncation is required, accepting that the handle is consumed afterward.

[PACKED_STORAGE]:
- No packed symmetric, triangular, Hermitian-packed, or band storage exists. `DenseColumnMajorMatrixStorage<T>` stores all m×n entries as a column-major `T[]` flat array. Algorithms that exploit symmetry (Cholesky, symmetric EVD, normal-equation regression) still allocate and operate on the full n×n array; the upper triangle is populated then overwritten in the Cholesky inner loop but never packed. This doubles memory over LAPACK-style DSYTRF packed storage for symmetric matrices.
- `DenseMatrix.Values` returns the backing `double[]` directly from `DenseColumnMajorMatrixStorage<T>.Data`. Constructing a `DenseMatrix` from an external `double[]` via `new DenseMatrix(rows, cols, existingArray)` binds that array without copying; `DenseColumnMajorMatrixStorage<T>` holds the same reference. Any subsequent `MapInplace` or provider in-place operation targeting that matrix mutates the original array. If the same array is wrapped into two `DenseMatrix` handles, any in-place operation on one mutates the other's view silently.

[PROVIDER_AND_VECTORIZATION]:
- The managed provider (`ManagedLinearAlgebraProvider`) contains no SIMD or `TensorPrimitives` calls. `MatrixMultiplyWithUpdate` pre-extracts all columns of B into `double[][] columnDataB = new double[columnsB][]` before the row-column dot loop, doubling memory traffic for B and allocating `columnsB` heap arrays before any computation. The inner dot loop is scalar. Under MKL, this code path is bypassed entirely and routes to BLAS DGEMM.
- Provider switching is process-global: `LinearAlgebraControl.UseNativeMKL()` replaces the `static ILinearAlgebraProvider` under a lock. It cannot be scoped per-thread or per-operation. When MKL is present, all six dense factorizations route to LAPACK equivalents (DGETRF, DPOTRF, DGEQRF, DGESVD, DSYEV, DGEEV) with full BLAS3 performance.
- The parallelization gate `rowsA + columnsB + columnsA < Control.ParallelizeOrder` uses a linear sum of three dimensions, not their product. For a 10×1000×10 GEMM the sum is 1020 — above the default threshold of 64 — so parallel dispatch fires even though the work does not warrant it. At small n, `Control.MaxDegreeOfParallelism = 1` eliminates this overhead.
- `System.Numerics.Tensors.TensorPrimitives` (in-box, net10.0) provides SIMD-backed `Dot<T>`, `Add<T>`, `Multiply<T>`, `MultiplyAdd<T>`, `FusedMultiplyAdd<T>`, and `SumOfSquares<T>` over `ReadOnlySpan<T>`. `FusedMultiplyAdd<T>` requires `IFloatingPointIeee754<T>` — satisfied by `double` — and executes with a single rounding, delivering improved accuracy for inner products. This is the platform vectorization floor for dense hot loops operating on `DenseMatrix.Values` / `DenseVector.Values` spans that the provider does not own.

[SMALL_DENSE_PROBES]:
- For small dense systems (point-set moments, per-pair precision matrices, covariance estimation), the factorization-as-probe pattern is the correct idiom: allocate a `DenseMatrix` from a `double[n*n]` span via `new DenseMatrix(n, n, data)`, call the appropriate factory, and classify the outcome. Definiteness-by-Cholesky, rank-by-SVD, and condition-by-ConditionNumber are O(n³) but with small constants for n ≤ 10 and no heap overhead beyond the storage array. At n ≤ 4 (3D covariance, 4-point alignment systems), managed factorization paths are faster than provider-switching overhead.
- Each call to `Cholesky()`, `QR()`, etc. clones the input matrix as its first operation — the clone is the internal factor buffer. For scattered-data evaluation where the coefficient system changes per query, this clone allocation is unavoidable if the original matrix must be preserved. `Cholesky.Factorize(Matrix<double>)` on a retained handle reuses the existing factor buffer via `Buffer.BlockCopy`, and is the only in-place refactorization path. No equivalent `Factorize` mutation method exists on `LU`, `QR`, `Svd`, or `Evd`; those require constructing a new handle each time.

[CONGRUENCE_AND_DENSIFICATION_CLIFFS]:
- The quadratic densification cliff via `matrix.TransposeThisAndMultiply(matrix)` is not limited to the regression API. Any algorithm that forms AᵀA routes through the managed provider's `MatrixMultiplyWithUpdate` and produces a dense n×n result regardless of whether the input was sparse or rectangular.
- Damped normal equations for iterative registration — the system `(JᵀJ + λI)x = Jᵀr` — form `JᵀJ` via `TransposeThisAndMultiply` and then solve via `Cholesky()`. This squares the condition number: for a Jacobian with condition κ(J), the unregularized Gram matrix has condition κ(J)². The correct Levenberg-Marquardt inner solve stacks the augmented system `[J; √λ I]` as an (m+n)×n dense matrix and applies thin QR directly, avoiding the squaring. The admitted package surface supports this: `DenseMatrix.OfRowArrays(J, sqrtLambdaIdentity).QR(QRMethod.Thin).Solve(b_augmented)`.

[REGISTRATION_INNER_SOLVES]:
- Closed-form orthogonal alignment (SVD of the cross-covariance matrix followed by determinant correction for reflection cases) requires forming `H = srcPts.TransposeThisAndMultiply(refPts)`, factorizing via `H.Svd(computeVectors: true)`, and constructing rotation `R = (svd.VT.Transpose() * svd.U.Transpose()).Transpose()`. The reflection check must use `Math.Sign(svd.U.Determinant() * svd.VT.Transpose().Determinant())`, not `svd.Determinant`: `Svd.Determinant` computes `|Π σᵢ|` — always non-negative — and cannot distinguish a reflection. `LU.Determinant` on a 3×3 carries the correct sign; check `det < 0.0`, not `det != 1.0`, because fp rounding means an exact rotation's determinant is only approximately ±1.
