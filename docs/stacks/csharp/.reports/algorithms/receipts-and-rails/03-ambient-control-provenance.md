# Ambient Numeric Control as Un-Receipted Policy

[THE_PROCESS_STATIC_POLICY_LATTICE]:
- The numeric library exposes one process-wide static configuration surface holding the linear-algebra provider, the Fourier provider, the sparse-solver provider, `MaxDegreeOfParallelism`, two internal parallelization thresholds (`ParallelizeOrder` default 64, `ParallelizeElements` default 300), the `TaskScheduler`, a thread-safe-RNG flag, and a distribution-parameter-validation flag. None of these is an argument to any decomposition, solve, transform, or integration call; every one is read from the static the instant the algorithm executes. A receipt that records the matrix, the tolerances, and the convergence status but omits this surface has captured the input and the verdict while discarding the machine that produced it — the policy that selected the algorithm branch and fixed its numerical reproducibility is invisible in the evidence.
- The three provider slots are lazily-initialized, double-checked-locked, mutable process statics. First touch of any provider slot with no prior configuration runs the default-resolution path; thereafter any code anywhere in the process may swap the slot to a different backend, and the swap is silent and global. A factorization computed at one instant and a re-computation of the same matrix moments later can route through two different backends with no local diff, because the discriminant lives in mutable shared state, not in either call. The receipt must snapshot the active provider identity at construction, not at re-admission — re-admission may observe a swapped slot.
- The provider singleton carries a stable low-cardinality identity through its `ToString` override (`"Managed"` for the in-house backend; native backends report their own tag). This string is the densest possible provenance anchor: one field on the receipt distinguishes the in-house LU/QR/eigensolver from a LAPACK-backed native one, which is the difference between two numerically distinct algorithms producing two distinct factor matrices for one input. Storing the provider tag is cheaper than storing the factor and strictly more discriminating than storing the matrix.
[DECOMPOSITIONS_ROUTE_THROUGH_THE_SWAPPABLE_SLOT]:
- Dense eigendecomposition delegates to `Provider.EigenDecomp(isSymmetric, n, source, eigenvectors, eigenvalues, blockDiagonal)`; dense Cholesky delegates to `Provider.CholeskyFactor(values, order)` and `Provider.CholeskySolveFactored(...)`. The factorization object is a thin owner over a provider call — the eigenvectors, the triangular factor, the singular vectors are produced by whichever backend occupies the slot at call time. The in-house backend and a native LAPACK backend compute the same decomposition by different pivoting, blocking, and orthogonalization paths; their eigenvectors agree up to sign and span but not bit-for-bit, and for clustered or degenerate spectra the chosen eigenvector basis within an eigenspace differs outright. A receipt asserting eigenvector identity across a cached-versus-recomputed pair is only sound when both carry the same provider tag; a provider mismatch makes the basis-identity check a category error, and the receipt must short-circuit to a span-equivalence predicate instead.
- Because the factor is provider-produced, the load-bearing conservation predicates split by provider class. The in-house backend's determinant, pivot parity, and rank threshold are computed by the library's own code and are auditable against the library's own tolerance ledger; a native backend computes these internally and exposes only the result, so a receipt over a native factorization cannot replay the tolerance arithmetic and must record the provider tag precisely to mark which conservation checks are replayable and which are opaque. The honesty marker is not optional metadata — it gates which re-admission predicates are even meaningful.
[PARALLELISM_BREAKS_FLOATING_POINT_REPRODUCIBILITY]:
- The in-house matrix multiply branches on `rowsX + columnsY + columnsX < ParallelizeOrder || MaxDegreeOfParallelism < 2`: below the order threshold or single-threaded it runs a deterministic serial accumulation; above it, it partitions the work across a `Partitioner` whose chunk boundaries depend on the configured degree of parallelism and the runtime scheduler. Floating-point addition is non-associative, so the parallel path accumulates partial sums in a runtime-determined tree topology and yields a result that differs in the low-order bits from the serial path and, worse, differs run-to-run when the scheduler partitions differently. The Frobenius matrix norm is computed as the diagonal sum of `A·Aᵀ` through this same multiply, so a condition number, a residual norm, or a relative convergence threshold derived from a Frobenius norm inherits the non-determinism. The crossing of the order threshold is therefore a reproducibility cliff, and the threshold value plus the active degree of parallelism are co-equal receipt evidence with the norm itself.
- The parallel reduction skeleton selects per-partition, reduces each partition, then reduces the partition results — a two-level fold whose first level is sized by `MaxDegreeOfParallelism`. For an associative-but-not-commutative-in-floating-point reducer (sum, dot product, weighted-statistic merge) the partition count changes the answer. A receipt that records a residual or a moment accumulated through this path without recording the degree of parallelism stores a value that is correct only for one core count; re-admission on a host with a different core count recomputes a different value and falsely flags tampering. The conservation check must compare against a tolerance scaled to the parallel-accumulation error bound, not against bit equality, whenever the parallel degree differs between construction and re-admission — and that decision requires the recorded degree.
- The hierarchical statistic merge that combines two moment accumulators is order-independent in exact arithmetic but order-dependent in floating point; folding partition results in scheduler order rather than a fixed order makes the combined mean and variance non-reproducible across runs at the same core count. A receipt that embeds sub-accumulator receipts and a combined parent and asserts the merge identity by bit equality is structurally wrong under parallelism; the identity holds only to the floating-point merge-error envelope, and the receipt must carry the merge fan-out alongside the moments so the envelope is computable at re-admission.
[ENVIRONMENT_AND_APPCONTEXT_GATE_THE_BACKEND]:
- Default provider resolution reads a per-family environment variable (a distinct name for linear algebra, for the Fourier transform, and for the sparse solver) and selects the named native backend or falls through to the in-house one. The backend that produced a factorization is therefore a function of ambient process environment at first-touch, set outside any call site and frequently outside the program. A receipt that omits the provider tag cannot distinguish a run configured for native acceleration from one that silently fell back to the in-house path because the environment variable was unset or the native binary failed to load — and that fallback is exactly the silent degradation a receipt exists to make visible.
- The resolution mechanics fix both the timing and the precedence of the ambient decision: the default path is a multi-way switch on the uppercased environment value, evaluated exactly once on first field access and cached on the slot — a later mutation of the variable is inert once the field is populated, so the only post-first-touch mutation channels are the explicit provider-request calls. An unrecognized value falls through a fixed probe chain that tries each native backend in a set order before the in-house fallback, and the probe resolves reflectively — an assembly-qualified type name through `Type.GetType` then `Activator.CreateInstance` cast to a creator interface — so a native sibling assembly merely PRESENT on the probe path is sufficient for the swap to fire, with no explicit opt-in anywhere in application code. The application-context switch that disables probing takes precedence ahead of the environment variable, and per-provider disable switches gate the individual probes; the choice between "host decides" and "library decides" is itself a policy value resolved at definition time of the provider field, not a runtime branch in the solver.
- Native-provider availability is further gated by `AppContext` switches that disable probing, disable all native providers, or disable a specific native backend. These switches are settable through host configuration and through code, so a process can be configured to refuse native acceleration entirely; the in-house path then runs unconditionally regardless of which `UseNative*` call the domain code issued. The `TryUseNative*` family returns a boolean that is the only signal the requested backend was actually adopted — a `Use*` call that throws leaves the previous provider active, and a `TryUse*` call that returns false leaves it active silently. A receipt factory that issues a provider request must capture the boolean outcome (or read back the provider tag after the request) as an honesty marker; assuming the requested backend is active because the request was issued is the rejected form.
- Mutating `MaxDegreeOfParallelism`, `UseSingleThread`, or `UseMultiThreading` re-runs provider `InitializeVerify` as a side effect, and the degree itself clamps to a closed interval on assignment — the native side reads the degree indirectly through re-verification rather than receiving a thread count in any call, and the single/multi-thread verbs flip the degree and the thread-safe-RNG flag together. The parallelism knob and the provider slot are therefore coupled through re-initialization, which means a receipt cannot treat the two as independent fields recorded at different instants — both must be snapshotted atomically at solve construction, because a parallelism change between recording the provider and recording the degree can re-verify (and in principle re-resolve) the provider underneath the partially-built receipt.
[HOIST_THE_AMBIENT_SNAPSHOT_INTO_THE_EVIDENCE]:
- The minimal control snapshot is a small immutable value carrying the linear-algebra provider tag, the Fourier provider tag, the degree of parallelism, the order and element thresholds, and the boolean outcome of any provider request issued for this solve. Reading the static surface once at solve construction and freezing it into this value converts ambient mutable policy into per-receipt fact; every subsequent re-admission compares the persisted snapshot against the snapshot of the re-admitting host and decides, before any numeric predicate runs, whether bit-exact comparison or envelope comparison is legitimate. The snapshot is the first field of the receipt and the gate on every downstream check.
- The reproducibility verdict is derivable, not enumerated: a single primary correspondence — provider tag plus degree of parallelism plus the relevant threshold — determines whether two receipts are bit-comparable, and the bit-versus-envelope decision is a pure function of that triple, never a per-call-site flag. Two receipts whose triples are equal admit bit equality on every derived norm and factor; two whose triples differ admit only span-and-envelope equality. The derived predicate owns this dispatch so no consumer re-derives the reproducibility rule from the raw fields.
[THE_ENVIRONMENT_THREADS_THE_POLICY_INSTEAD_OF_THE_STATIC]:
- The ambient static is the wrong owner for solve policy: it is mutable, shared, and read at an instant no call controls. Structural environment access threads the policy through the effect runtime as a trait — a `Has<RT, NumericControl>` exposing a single `static abstract K<RT, NumericControl> Ask` — so a measured solve reads its provider and parallelism policy from the threaded environment via `ask`/`asks`, never from the global. The runtime carries the snapshot the receipt will record; the solve and the receipt read one source, and the value the receipt persists is provably the value the solve consumed.

```csharp
public interface Has<in M, VALUE> { static abstract K<M, VALUE> Ask; }

static K<RT, TReceipt> MeasuredSolve<RT>(MatrixHandle a, VectorHandle b)
    where RT : Has<RT, NumericControl>, Readable<RT, RT>, Monad<RT> =>
    Readable.asksM<RT, RT, TReceipt>(_ =>
        Has<RT, NumericControl>.Ask.Bind(ctrl =>
            Decompose(a, ctrl).Bind(f =>
                Prove(f, b, ctrl).Map(ev => Receipt.Seal(ev, ctrl.Snapshot)))));
```

- Scoping an override is one combinator: `Readable.local(env => env with { Control = pinnedSnapshot }, solve)` pins a numeric-control snapshot for the duration of one sub-computation and restores the prior environment on exit, so a sibling solve in the same batch is unaffected. Pinning the snapshot through `local` rather than mutating the global means the override is lexically bounded, composable, and re-entrant — concurrent solves under different pinned snapshots never tear each other's policy, where two writes to the mutable static would race. The receipt sealed inside the scope records the pinned snapshot, and the equality of the sealed snapshot and the scoping snapshot is a structural invariant rather than a hopeful assumption.
- Lifting the snapshot read into the effect interpreter via `localIO` over a `MonadUnliftIO` runtime sequences the policy read relative to the bracketed acquisition and the retry boundary, so a retried solve that re-enters under a re-resolved provider re-reads the environment snapshot for each attempt and stamps each attempt's receipt with the policy that attempt actually ran under. A snapshot captured once outside the retry loop and reused across attempts is the rejected form: it records the policy of the first attempt for a receipt sealed by the last, and the provenance silently lies whenever a provider re-verification fires between attempts.
