## THREE STRONGEST ATTACKS ON THE MAX POLE'S PREDICTABLE POSITIONS

### Attack 1 — "Inline the doctrine, a routed read is skippable" is backwards; the inline copy is not a guarantee the law is SEEN, it is a guarantee the law is WRONG.

The MAX pole will argue that a fable agent under load trims the read, so carrying the law in-prompt is the only guarantee it is seen. The drift evidence detonates this. The engine's `L.cs.ultra` says "the 16 named laws … held as fact" — it does not reinforce `INTERFACE_SEAM`, it ERASES it: the agent reads "16, held as fact," treats the closed set as complete, and authors every interface, static-abstract bound, and variance decision with ZERO `INTERFACE_SEAM` pressure. The COLLAPSE_SCAN "12 signals" mis-attributes three moves to a table that has nine rows. A carried paraphrase is a second source of truth that drifts silently and, when it drifts, lies with the authority of "held as fact" — which is worse than a gap, because a gap is visible and a false completeness claim is not. The routed read has a disk source that cannot go stale relative to itself. The MAX position optimizes "the agent definitely reads SOMETHING" at the cost of "what it reads is true." Py's STEP-0 `ls`+read chain (already drift-immune, already on disk) makes the read a VERIFIABLE tool act — an agent cannot fake having `ls`-ed. That is strictly stronger than trusting a paraphrase that is provably broken in two places right now. The correct guarantee against trimming is not a bigger inline copy; it is (a) the enumerate-first `ls` the agent must actually run, and (b) the `beyondMap` schema field that fails the schema if the agent treated the map as a ceiling.

### Attack 2 — the MOTION/ACCEPTANCE/reconcile stages exist to serve a STALE prose obligation, not a structural requirement; re-adding them re-introduces the deferral the engine abolished.

The MAX pole will cite `DECISION:262` — "the engine's terminal reconcile carries whole-repository write authority" — as proof the engine MUST provide that stage. It proves the opposite. The engine deliberately deleted reconcile (`meta.description:4`: "every agent repairs the cross-file ripples its own work exposes IN ITS OWN PASS … the agent who sees the defect owns the fix; deferral is a defect"). The DECISION was authored against the older reconcile-bearing engine; line 262 is a dangling reference to a removed stage. The fix is to correct four words in the DECISION, not to re-grow a stage across the whole engine. And a MOTION or ACCEPTANCE stage is, by construction, a place to PARK work "for the terminal pass" — which is precisely the deferral the reconcile-free law exists to kill. The execution-fit dossier's own riders refute the need for a stage: the GShark manifest drop rides `nurbs.md`'s implement agent (the page that vendors the owner); `api-gshark.md` deletion rides the same agent (gated on the vendored owner landing); the `FAB`/`AppHost` counterpart edits ride the seam-counterpart's owning-page agent under the existing boundary-repair law. So the entire MOTION stage collapses to rider-column data + the lane's existing serialization. The MAX pole mistakes a stale prose reference for a phase the engine owes.

### Attack 3 — the removed barriers and the deleted exhortations carry no rigor; they carry idle time and noise. Rigor is structural, not ceremonial.

## THE FAILURE MODE I CANNOT DEFEND (grounding dossier) — and its bound

The MAX pole's sharpest attack on the dossier is correct in one narrow place. My three honesty rails defeat the PHANTOM-injection attack: anchored citations (`catalog:line`, verified via `assay api`) mean the dossier carries confirmed primary extracts, not a lossy digest; `dossierPhantoms` makes Implement re-verify a sample and surface any present-but-fake anchor exactly as redteam surfaces a critique miss, so self-certification is structurally impossible; and the preserved "pointer, verify, exceed" law keeps the adversarial attack on the CODE fully intact (that attack never depended on re-ingesting the whole doctrine — only on attacking the fence against the verified domain + package surface, which the dossier supplies pre-verified).

What I CANNOT defend: OMISSION. If the single discover agent's read was itself shallow — it `ls`-ed both `.api` tiers but skimmed one catalog — the dossier's anchors are all VALID (they point at real members) yet a relevant member exists that the dossier never surfaced. `dossierPhantoms` catches lies (present-but-fake), never omissions (absent-but-relevant); spot-verify only checks anchors that ARE present, never proves the negative. The bound that makes this acceptable: this is EXACTLY the omission risk the CURRENT engine already carries — the current reading map already only "POINTS," and `READ_MANDATE:553` already delegates "independently CONFIRM no other relevant catalog is missing" to each downstream agent. The dossier does not WORSEN omission risk; it RELOCATES the same completeness burden from 8 wishful re-reads to 1 verified read plus the `beyondMap` field that forces every consumer to enumerate members added BEYOND the dossier. So the bound is precise: the dossier is exactly as strong as the current engine on omission-completeness (neither can prove a negative from a single read), and STRICTLY stronger on phantom-detection and on making the read actually happen. The one claim I refuse to make is that the dossier ELIMINATES omission risk — it bounds it to the single discover read, which is the same single point of trust the engine already stakes on discovery today.
