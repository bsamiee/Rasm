# Overload Economics And Entrypoint Totality

[PRIORITY_PRUNES_BEFORE_BETTERNESS]:
- Overload resolution runs three filters in fixed order: applicability reduction, then a priority prune, then betterness. The priority prune is not a betterness rule — it groups the applicable set by declaring type, and within each group removes every member whose priority is below the highest priority found in that same group, then recombines the surviving groups and runs ordinary betterness over the union. A higher-priority overload therefore does not win by being a better conversion; it wins by deleting its lower-priority siblings from the candidate set before betterness ever runs.
- The grouping is per declaring type, so a base type can never raise its members above a derived type's: an applicable member found on the derived type stops base members from being considered at all, and priority is compared only inside each type's own group. The deprecated-overload demotion pattern — `[OverloadResolutionPriority(-1)]` on the wide overload — therefore demotes only relative to same-type siblings; it cannot suppress a competing overload on a different type in the hierarchy. Cross-type steering must place both the preferred and the demoted overload on the same declaring type.
- A high-priority overload makes its same-type, lower-priority siblings effectively uncallable from source even when an exact identity conversion exists to the sibling. `Run(long)` at priority 1 beside `Run(int)` at default means `Run(someInt)` binds to the `long` overload despite the identity conversion to `int`; the `int` overload becomes reachable only through a delegate conversion, `UnsafeAccessor`, reflection, or unrecompiled callers. This is the mechanism that collapses an arity- or width-fragmented surface into one preferred entrypoint without deleting the narrow overloads the compiler still needs for binary shape.
- Priority comes from the least-derived declaration of a member and is never inherited across an interface boundary: an implementation method carries its own priority independent of the interface member it satisfies, with no diagnostic on mismatch, and applying the attribute on an override or on an implicit interface implementation is itself an error because the value is read from the least-derived declaration. A polymorphic entrypoint that must steer through an interface seam declares its priority on the concrete least-derived method, not on the interface contract.
- The attribute is legal only where overload resolution actually selects among members: ordinary methods, constructors, and indexers. It is rejected on non-indexer properties, every accessor, conversion operators, lambdas, local functions, finalizers, and static constructors — these never participate in a method-group selection, so the priority concept is undefined there. An entrypoint family that wants priority steering cannot live behind a property getter; it must be a method group.

[COLLECTION_EXPRESSION_BETTERNESS_LATTICE]:
- A collection-expression argument has no natural type, so the entrypoint's parameter type is what target-types it, and betterness among candidate parameter types follows a dedicated lattice that does not reduce to ordinary conversion betterness. For a collection-expression argument the ordering is: a `ReadOnlySpan<E₁>` target beats a `Span<E₂>` target when an implicit conversion exists from `E₁` to `E₂`; any span target beats an array-or-array-interface target with a convertible element type; and between two non-span targets, the one with an implicit conversion to the other wins. The span-over-array preference is structural, not a tie-break — it is why a single `params ReadOnlySpan<T>` entrypoint silently absorbs every `[a, b]` call site even when an array or `IEnumerable<T>` overload is also applicable.
- The lattice has an unguarded hole: when one target is a covariant span and the other a covariant array and neither element relation subsumes the other, betterness produces no winner and the call is a hard ambiguity. `Run(Span<object>)` beside `Run(string[])` invoked with `["<value>"]` is ambiguous because `Span<string>` would beat `string[]` but the argument can also widen to `Span<object>`, and the two directions do not compose. The design pressure is to never expose two collection-shaped entrypoints whose element types differ by a reference conversion; one span entrypoint at the most-derived element type is the only ambiguity-free shape.
- A collection expression cannot be the receiver of an extension invocation precisely because it has no natural type: `["<value>"].Extension()` fails, while `Static.Extension(["<value>"])` binds, since target typing only reaches the literal through a parameter. An entrypoint family meant to be reached fluently from a bracketed literal must be an ordinary static method whose first parameter is the collection target, not an extension-receiver surface.
- For a `ReadOnlySpan<T>` parameter over a literal of constant primitive elements, the span's data is promoted to the program's data segment and the safe-context widens to the caller — no stack or heap traffic at all. For non-constant elements the literal lowers to a stack-allocated inline array bound by the span, with a declaration-block safe-context that forbids the span escaping the method. A `params ReadOnlySpan<T>` entrypoint therefore pays nothing at a constant call site and a single stack buffer at a dynamic one; the allocation-free claim is a property of the target type, not of the call.

[PARAMS_EXPANSION_TIE_BREAKS]:
- Expanded form is a last-resort applicability: a `params` member is considered in expanded form only when its normal form is not applicable, and only when no member with the already-expanded signature is independently declared on the same type. The normal-form-first rule then re-enters betterness as a tie-break — a member applicable in normal form beats one applicable only in expanded form, so passing a pre-built collection value to the `params` entrypoint binds the collection-typed overload directly and never triggers expansion.
- Among two `params` entrypoints both applicable only in expanded form over the same argument set, the one whose collection has fewer constructed elements wins, then — at equal element counts and equivalent parameter sequences — the same span-over-array, ReadOnlySpan-over-Span element lattice applies, gated on an identity (not implicit) element conversion. The gate is the consistency hinge with explicit collection expressions: when element types differ by a non-identity conversion, the params tie-break declines to fire and ordinary argument-conversion betterness decides, which can diverge from the bracketed-literal result and is the source of surprising binds.
- Two `params` entrypoints that differ only in element type are mutually ambiguous on an empty argument list and on an empty literal: `Run()` and `Run([])` against `params ReadOnlySpan<int>` beside `params ReadOnlySpan<int?>` are both ambiguous because element type is prioritized over everything and nothing disambiguates an empty construction. A polymorphic entrypoint that must accept zero arguments owns exactly one `params` element type; a second element-typed `params` overload forecloses the empty call.
- Non-array `params` construction interleaves with argument evaluation differently from array `params`: the non-array collection is created and populated at the lexical position of the params arguments, between the previous and next named or positional argument, whereas an array `params` is built immediately before the call after all arguments evaluate. An entrypoint whose `params` element construction has observable side effects exposes a different evaluation order under a non-array collection than under an array, and that order is part of the contract, not an implementation detail to be relied upon as array-like.
- A `params` parameter of a `ref struct` type is implicitly `scoped`; the span cannot escape the entrypoint without an explicit `[UnscopedRef]`, and an override must restate `params` or `scoped` to preserve the constraint. This makes `params ReadOnlySpan<T>` the safe default entrypoint shape: the absorbed arity is stack-bound by construction and the compiler forbids leaking the buffer, so no lifetime annotation burden reaches the caller.

[TOTAL_DISPATCH_AND_PARTIAL_RELAXATION]:
- The full-coverage generated `Switch`/`Map` over a closed case family is the totality proof: adding a case turns every full-coverage dispatch site into a compile error, so exhaustiveness is enforced at the type boundary rather than asserted by a default arm. A request-union entrypoint that routes through full-coverage dispatch cannot silently drop a new verb — the new case breaks the build at the dispatch body, which is the intended pressure.
- The partial-overload generation mode emits separately named partial dispatchers carrying a nullable per-case parameter and a leading nullable fallback, not extra overloads of the total method. Because the names differ, the total and partial surfaces never enter the same overload group and need no priority arbitration; the entrypoint keeps full-coverage dispatch in its own body while peripheral call sites that care about a subset bind the partial method and tolerate new cases through the fallback. Totality lives at the core, relaxation lives at the rim, and the two coexist without a knob.
- The maximally sealed entrypoint composes three generator suppressions: removing the value-to-union conversion, privatizing the case constructors, and suppressing generated factories. The only admission path is then a hand-written static factory inside the partial, and the only egress is generated dispatch — no implicit conversion, no public case constructor, and no factory backdoor exists, so input is validated exactly once at the factory and behavior is reached exactly through dispatch. Pairing this with private nested declaration scopes the construction surface to the enclosing owner alone.
- Discrimination must be recoverable from the value, never from a parallel knob: a `bool`-or-count parameter beside the input re-encodes what the input's type, case, or arity already carries, and the rejected forms are precisely a batch flag beside a sequence, a mode flag beside a value, or a count beside a span whose length already answers it. The legitimate companion parameters are capabilities orthogonal to the discriminant — a cancellation token, an environment record — which describe how the work runs, not which case it is; the test is whether removing the parameter loses information the value cannot reconstruct.

[EXTENSION_BLOCK_DISPATCH_ECONOMICS]:
- An extension-block dispatch entrypoint sees only the receiver's public surface — generated dispatch, public case members, explicit interface members — and is barred from private members, which is not a limitation for union dispatch because the generated dispatch surface is public by construction. The block itself takes no access modifier; visibility is per-member, and a `private` helper inside a block on one receiver is reachable from a block on a different receiver in the same static class, an unexpected coupling, so shared dispatch helpers belong at file or static-class scope, never inside a block.
- Extension members obey the same priority grouping as ordinary members: priority is compared only within the declaring static class, never across static classes in scope. Two static classes each offering a same-named extension on one receiver are compared by ordinary betterness regardless of either's priority, so a span-preferred extension entrypoint must declare both the preferred and the demoted member in the same static class for the priority prune to apply. A computed predicate exposed as an extension property compiles to a non-virtual receiver-parameter method, attaching derived discriminants to a union without touching its generated partial.
