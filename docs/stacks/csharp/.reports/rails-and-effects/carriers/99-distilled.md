# Carriers

[CARRIER_SELECTION_LADDER]:
- Narrowest-carrier law: `Option<A>` when absence is the only distinguishing fact — it carries zero failure semantics; promote to `Fin<A>` when the caller must know why it failed; promote to `Validation<F,A>` only when independent failures must accumulate before reporting; stay at `Either<L,R>` only when the left type is not `Error` and no effect-level convergence is needed on that path — `Either` requires an explicit `L -> Error` bridge before entering an effect chain.
- `Fin<A>` is structurally `Either<Error,A>` with the fail side pinned to the `Error` hierarchy — the narrowest carrier whose failure domain is exception-capable and composes with effect lifts without a bridge.
- The failure slot is type-parametric only above `Fin`: `Validation<F,A>` admits any monoid-capable `F`, so a closed domain rejection union instantiates as the failure type and the rail carries a discriminated rejection rather than a stringly message; the crossing to the canonical `Error` happens exactly once at the convergence seam — `ToFin` is constrained to `F = Error` — and the failure is never re-described inside the rail.

[REPRESENTATION_KINDS]:
- The carriers do not share a representation kind, and the kind dictates copy, allocation, identity, and `default` semantics: `Option<A>` is a two-field `readonly struct` — value-copied, zero-allocation, never null as a reference; `Fin<A>` is an abstract class with two sealed primary-constructor inner classes `Succ(A Value)` and `Fail(Error Error)` — not record types, `Deconstruct` hand-declared on each; `Validation<F,A>` and `Either<L,R>` are record hierarchies whose case payloads participate in compiler-synthesized record equality; the effect carriers are records wrapping a delegate-bearing inner effect.
- The struct/class split is load-bearing for `default(T)`: `default` of the absence carrier is a usable `None`; `default` of every class-shaped carrier is a null reference and any operation on it throws before the rail ever sees it — a field, array slot, or uninitialized local typed as a class-shaped carrier is a latent null; only the absence carrier is total over `default`.
- `Fin<A>` cases project spans of arity zero-or-one: `Succ` returns a one-element `ReadOnlySpan` of its value and an empty failure span, `Fail` the mirror — every result value is a bounded foldable over a stack span, so fold and comprehension behavior follows from a one-element-or-empty sequence, not from a special case.
- Serialization is non-uniform: only the absence carrier carries a hand-written binary contract (presence flag plus bound value — the legacy binary surface, not a modern codec it does not advertise); the result, accumulating, and general carriers implement none; effect carriers cannot be serialized at all. Discriminated reference carriers project to a wire-shaped union before crossing a boundary; effects run to a value first.

[CASE_SURFACE_AND_TERMINAL_EXITS]:
- `SuccValue`/`FailValue` on `Fin<A>` and `SuccessValue`/`FailValue` on `Validation<F,A>` are `internal` — inaccessible to consumers and throwing `InvalidCastException` on the wrong discriminant; the public extraction surface is the nested case types (`case Fin<A>.Succ s` exposes `s.Value`; `case Validation<F,A>.Success s` exposes `s.Value`) or the flat discriminant form `{ IsSucc: true }` over the public abstract boolean.
- `Option<A>.Case` yields `A` on `Some` and `null` on `None`; `{ IsSome: true, Case: var v }` is the canonical inline extraction that keeps the rail mid-expression. `Either<L,R>` matches on the `.Right`/`.Left` nested types with `{ IsRight: true }` guards.
- Terminal boundary exits: `Fin<A>.IfFail` (`Func<Error,A>`, `A`, `Action<Error>` overloads) and `IfSucc(Action<A>)`; `Option<A>.IfNone(A | Func<A> | Action)` and `IfSome` — the only points where `None`/`Fail` produce a default; `.Value` access is never the exit.

[FAILURE_SIDE_ALGEBRA]:
- `Fin<A>`: `MapFail` is `Error -> Error` pinned; `BindFail` delegates to `BiBind` with the succ arm as identity; `BiBind<B>` branches both cases; `BiMap<B>` maps both sides but keeps the fail side pinned to `Error`.
- `Validation<F,A>`: `MapFail<F1>` changes the failure type — the asymmetry against `Fin` — and requires `Monoid<F1>`; `BiBind<F1,A1>` carries no compile-time constraint on `F1` (structural type-change), while `BindFail` requires `Monoid<F1>` — use `BiBind` when accumulation semantics are not required, `BindFail` when the result must participate in further `&`/`*` accumulation. The instance `BiMap` constructs target cases directly without routing through `BiBind`.
- `Option<A>.BiMap` has two overloads with asymmetric null behavior: the `(Func<A,B>, Func<Unit,B>)` overload routes the executed arm's raw result through a null check that raises `ResultIsNullException` on null; the `(Func<A,B>, Func<B>)` overload converts via the implicit operator with no guard, silently folding a null result to `None` — the overloads are not interchangeable, and the `Func<Unit,B>` form is unsafe in the throwing direction on a value the sibling overload would absorb. `BiBind` carries no null guard on either arm.

[EQUALITY_ORDER_AND_IDENTITY]:
- Default structural equality on a payload-typed carrier routes through a per-type strategy struct that falls through to a reflective trait probe, not `EqualityComparer<A>.Default`: the strategy's static constructor runs once per closed payload type, searches loaded assemblies for a convention-named equality trait by string, binds its static `Equals`/`GetHashCode` as a cached reflective invocation on a hit, and falls back to the framework comparer on a miss — equality over a rich payload is a reflective call per comparison, invisible at the call site and material in hot sets and dictionary probes, and the bound delegate throws on a null boxed result, surfacing an invalid-operation failure from inside `==` far from its apparent source.
- Escape: compare on a projected scalar discriminant, or use the typeclass-keyed overloads — single-witness `Equals`/`CompareTo` on the absence and result carriers, two-witness forms on the two-sided carriers — which resolve the strategy at the type level with zero reflection; the witness form is the deterministic equality, the zero-argument form pays the resolver.
- Failure-case identity is asymmetric across the family: `Fin`'s failure case implements equality as a bare case test — any failure equals any other regardless of the carried error — hashes to a fixed constant, and orders all failures mutually equal; the accumulating and general carriers hash and compare the failure payload through its own trait. A set or dictionary keyed on `Fin` therefore coalesces all failures into a single key — distinct errors overwrite — correct for "did it fail" deduplication and wrong for "which failures occurred"; the accumulating carrier preserves each distinct failure as a distinct key, and the choice is irreversible inside the container.
- Every two-case carrier defines a total order with the failure side strictly first: failure/left/absence compares negative against any success, and within a side ordering delegates to the payload's order witness — except `Fin`'s failure side, consistent with its identity-transparent failure equality. Sorting groups all failures ahead of all successes with no predicate; a min over the sequence is "the first failure if any, else the least success"; `None` sorts before any `Some`, compares equal to `None`, and hashes to zero, so a sorted structure of optionals front-loads every absence.
- Mixed-operand comparison elides the lift: the two-case carriers carry comparison and equality operators against the bare payload and against the lift literals, not only against another carrier — no lift wrap on the constant, no unwrap on the carrier — and the mixed-operand comparison silently inherits the failure-first total order.
- Effect carriers are identity-opaque: equality and hashing compare the captured delegate reference, never the eventual result — two effects computing the same value are unequal, and the same effect equals only itself; no order, no structural projection, no serialization. The record machinery generates this equality unconditionally, so the type system permits the category error; the correct comparison is over the run result inside the rail, never over the effect value.

[COMBINATION_ALGEBRA]:
- `Bind` is sequential and fail-fast on every carrier; accumulation lives only on the applicative and semigroup operator surface, and each operator's semantics differ per carrier. LINQ query expressions are therefore always sequential — accumulation requires explicit `&`/`*`/`Apply` chains.
- `Fin<A>`: `Apply`/`*` accumulates — when both operands are `Fail` the result is `Fail(lhs + rhs)`, and in an N-ary curried chain a `Succ` function applied to a `Fail` argument passes the argument's error forward as the new accumulating failure, so the chain result carries all N independent failures combined left-to-right via `Error.Combine`. `+` (`SemigroupK.Combine`): left `Succ` is returned without examining the right operand; both `Fail` merge errors. `|` (`Alternative.Choose`, identity `Fin<A>.Empty` = `Fail(Errors.None)`): left `Succ` returns immediately, left `Fail` returns the right operand without combining errors. `op_True`/`op_False` back `||` short-circuit coherence.
- `Validation<F,A>`: `&` collects both success values into `Validation<F,Seq<A>>` and combines failures via the semigroup witness; `+`/`Combine` keeps the first success and stays at `A` — `&` widens and collects, `+` does not; `*` is the curried applicative. The internal `CombineI`/`ApplyI`/`CombineFirst` make `&` and `*` the caller-facing surface; the one public fold-friendly `CombineI` overload appends a single `Validation<F,A>` into an accumulating `Validation<F,Seq<A>>`, enabling a fold-based collection pattern.
- `Validation`'s root constraint is `Monoid<F>` across the construction and projection surface — `Success`, `Fail`, and `Empty` all carry it, and `Empty` calls `F.Empty`; `Semigroup<F>` suffices only on the static extension accumulation operators, and the `Alternative`/`MonoidK` identity resolves through a runtime instance with a `NotSupportedException` fallback when `F` is not monoid-capable — `Semigroup<F>` alone is insufficient for the carrier.

```csharp
// N-ary Apply: all three Fail errors accumulate
Fin<Func<string, Func<bool, int>>> f0 = Fin.Fail<Func<string, Func<bool, int>>>(Error.New("fn"));
Fin<string> arg1 = Fin.Fail<string>(Error.New("a1"));
Fin<bool> arg2 = Fin.Fail<bool>(Error.New("a2"));
Fin<int> result = f0 * arg1 * arg2;
// result: Fail(ManyErrors([Error "fn", Error "a1", Error "a2"]))

// SemigroupK.Combine: both Fail → merged error; left Succ → left wins immediately
Fin<int> combined = Fin.Fail<int>(Error.New("x")) + Fin.Fail<int>(Error.New("y"));
// combined: Fail(ManyErrors([Error "x", Error "y"]))
```

[INGRESS_AND_LIFT_HUBS]:
- The success and failure lift literals are `readonly record struct` single-field values whose entire purpose is deferred carrier selection: the success literal exposes `Bind` overloads across the result, either, effect, reader, and state carriers plus the kind-generic monadic form, with direct projections to the absence, result, and either carriers; the failure literal projects to the two-sided carriers and maps its payload, and the unit-failure literal converts implicitly to `None` on the absence carrier. A value or error written once as a lift literal commits to no carrier until the surrounding bind or destination type fixes one — the dispatch hub that replaces per-carrier construction at the head of a pipeline.
- `Fin<A>` declares implicit operators from `A`, `Error`, `Either<Error,A>`, `Pure<A>`, and `Fail<Error>`; `Fin.Succ<A>(A)` and `Fin.Fail<A>(Error)` are the static constructors, and `Fin.Fail<A>(string)` wraps the string in `Expected`.
- Null admission is asymmetric at the `Option` ingress: `Some` on the reference-type path has no null guard — it calls the internal constructor unconditionally; the nullable-struct overload throws `ValueIsNullException` on null; `Optional` is the null-tolerant factory that always folds null to `None` and backs `Option<A>`'s implicit conversion from `A?`.
- `guard(bool, Error)` and `guard(bool, Func<Error>)` return `Guard<Error,Unit>` — a typed conditional bound into the surrounding monad through per-rail `SelectMany` overloads; a `guard` in a query expression is the only idiomatic ingress for boolean invariants at pipeline head, `guardnot` is the negation shorthand, and the payload-free `guard<F>(bool)` form yields pure unit or the alternative empty for any alternative-capable carrier. `Guard<F,A>` is never a return type — it always exits as the destination monad.

[CROSS_RAIL_PROJECTIONS]:
- The full instance matrix: `Fin<A>` → `Option<A>` (`ToOption`), `Either<Error,A>` (`ToEither`), `Validation<Error,A>` (`ToValidation`), effect lift (`ToEff`); `Option<A>` → `Fin<A>` (zero-arg yields `Fail(Errors.None)` on `None`; `ToFin(Error)` supplies identity), `Either<L>` and `Validation<F>` each through three overloads — eager value, zero-arg, lazy `Func`; `Validation<Error,A>` → `Fin<A>` (the convergence projection, constrained to `F = Error`); `Validation<F,A>` → `Option<A>` (drops all failure information) and `Either<F,A>` (preserves both cases); `Either<Error,A>` → `Fin<A>`.
- Projectability is asymmetric: every rail projects to `Option` by discarding the failure side; widening projections require supplying the missing structure — `Option → Fin` needs an `Error`, `Either → Eff` needs an `L -> Error` map.
- Natural transformations carry round-trip loss that explicit projections avoid: the `Fin → Option` transformation discards the error, and the zero-arg reverse maps `None` to `Fail(Errors.None)` — a `Fail` survives both directions with its original error replaced by `Errors.None`, breaking every subsequent `HasCode` or `Is` predicate; the `Fin → Eff` and `Fin → IO` transformations preserve the error. When `None` must carry diagnostic identity, `option.ToFin(Error.New(...))` is always correct.

[ERROR_MONOID_FLATNESS]:
- The failure type is itself the aggregate: a single error and a thousand errors share one type, so a rail's failure slot never widens to a sequence-of-errors, and a function returning `Error` already returns the aggregate without a container change. `Errors.None` is the monoid identity — a zero-member `ManyErrors` with `IsEmpty` true — and combining any error with it returns the non-empty error unchanged; `Error.Many` is a smart constructor — empty input returns `Errors.None`, single-element input returns the scalar unchanged, multi-element input returns `ManyErrors`.
- `Combine`/`+` maintains a flat-depth invariant at every append: two collections concatenate their inner sequences, a leaf appended to a collection cons-prepends or appends into the same sequence, two leaves seed a fresh two-element collection — the result is always a leaf or a flat collection, never a tree, so a fold over an accumulated batch is a single linear pass. The accumulating carrier's gather is precisely this monoid: pinning the failure type to `Error` makes applicative accumulation produce one flat aggregate with no bespoke accumulator — the gather is the monoid, the monoid is the type.
- The flat invariant makes one-level membership total only over combine-built aggregates: `HasCode` on the base type compares only the receiver's own `Code` — causal chains composed via `Error.New(string, Error)` or `Error.New(int, string, Error)` are not searched — and `ManyErrors` overrides it as a one-level scan of direct members' `Code` properties, not a recursive call, so a manually nested aggregate is not descended into. Inner-error chains live outside the membership scan entirely; a buried cause is found only by explicit inner traversal. `IsType<E>` tests the concrete subtype; `Filter<E>` projects matching members and returns `Errors.None` when none match.
- Decomposition and typed absence: `Head`/`Tail` decompose `ManyErrors` in list order — on a scalar, `Head` is self and `Tail` is `Errors.None`; `Inner` is `Option<Error>` (`None` on `Expected` without an inner, resolved from the live `InnerException` on `Exceptional`) and `Exception` is `Option<Exception>` (`None` on `Expected`, `Some(ToException())` on `Exceptional`) — never raw nullable fields.
- Folding the aggregate splits on short-circuit: `FoldM` invokes the function over each atomic error and combines through the target monoid, continuing across sub-failures; `ForAllM` short-circuits on the first failure — indistinguishable on a single-atom error, decisive when the error is a `ManyErrors` aggregate containing multiple matching members.

[FACET_LATTICE_AND_CODES]:
- `Error` is a non-sealed public abstract record — user subtypes are a documented extension point. The built-ins: `Expected` (message, code, `Option<Error>` inner — positional record), `Exceptional` (message, code, optional live `Exception` with serialization-safe null handling), `ManyErrors` (sealed, `Seq<Error>`), and `BottomError` — sealed, inheriting `Exceptional`, a specialization representing an unevaluable state, not a sibling.
- `IsExpected`/`IsExceptional` are independent abstract booleans; every concrete leaf sets exactly one true, and the aggregate derives both structurally — exceptional if any member is, expected only if every member is. One escaped exception inside a batch of expected validation failures contaminates the whole batch to exceptional: code that routes on the aggregate facet must split the sequence by per-member facet before routing — the aggregate flag never means the batch is homogeneous.
- `Expected` equality is full structural record equality over message, code, and inner — the code is not a privileged equality key, so identity-keyed recovery breaks under message templating or localization; recovery that must survive message change keys on the code through the `HasCode` predicate surface, a separate axis from equality.
- Bottom is exceptional, not expected: a filtered-away comprehension step or a default-initialized rail value surfaces as a code-carrying exceptional bottom rather than a silent empty, and recovery that catches expected failures deliberately lets bottom escape — bottom is a logic defect, excluded from the recoverable partition by construction. `Errors` statics (`None`, `Bottom`, `Cancelled`, `TimedOut`, `SequenceEmpty`, `ValidationFailed`, `EndOfStream`) are the canonical pre-built values; `ErrorCodes` mirrors them as integer constants for `HasCode` matching.
- Reserved-band law: the runtime owns a contiguous band of large-magnitude negative codes for its structural failure singletons; a domain code inside that band is indistinguishable from the runtime singleton under code-keyed recovery — the band is an exclusion zone for domain error vocabularies. `ManyErrors` owns a reserved aggregate code distinct from any member's and renders its message at construction as the full member-message array, not a count — one predicate can target the fact of aggregation, another the presence of a specific buried failure.

[EXCEPTION_BRIDGE]:
- `Error.New` is one polymorphic factory discriminating on argument shape — `(string)` → `Expected`; `(Exception)` → `Exceptional`; `(string, Exception)` → `Exceptional` with display-sanitized message; `(string, Error)` and `(int, string, Error)` compose causal chains via `Expected` — reinforced by implicit conversions from a code-message tuple and from an exception, so tuples and exceptions are the failure literals at call sites and the discriminant is recoverable from the argument with no kind flag and no sibling named constructors.
- Exception admission canonicalizes rather than blind-wraps: `TaskCanceledException` → `Errors.Cancelled` and `TimeoutException` → `Errors.TimedOut` — pre-built expected singletons with reserved codes, so cancellation and timeout arrive on the rail expected, catchable, and code-stable, matched by one code constant rather than several exception types, never contaminating the exceptional partition; an `AggregateException` explodes through the same lift per inner exception and re-collapses zero/one/many, so a single failure never wears a collection wrapper.
- The heterogeneous inbound funnel admits the carrier itself, the wrapped error-exception forms, the rail-exception type, a raw exception, a bare string, and an option of the carrier, falling back to the bottom sentinel for anything else — one entry point collapses every foreign failure shape with no per-shape branch at the call site.
- The exceptional bridge is one-way across a wire: deserialization or default construction erases the live exception — the rebuilt value still answers exceptional and keeps message and code, but the stack trace, the exception type, and `HasException<E>` membership tests are gone (`HasException<E>` returns false on deserialized instances). Exceptions do not leak across application boundaries; only their sanitized projection does.
- Within one process the round trip is lossless: leaves project to wrapped error-exceptions preserving code, message, and inner (`ToErrorException` → `ErrorException`); the aggregate projects to an aggregate exception over member projections; and admission recognizes the wrapped forms and unwraps them back to the original carrier — a throw/catch hop through foreign code that only speaks exceptions preserves a structured aggregate, surfacing every accumulated rejection rather than the first, and re-raising routes through dispatch-info so the original stack survives the rail. `Throw()`/`Throw<T>()` escape the rail into exception flow — restricted to host-boundary adapters.
