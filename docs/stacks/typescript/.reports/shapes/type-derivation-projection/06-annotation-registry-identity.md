# Annotation Registry Identity

[THE_REGISTRY_IS_A_CLOSED_SYMBOL_VOCABULARY_PROJECTED_FROM_A_TYPED_KEY_MAP]:
- The annotation surface is two layers fused by one projection: the OUTER layer is the typed `Annotations.Schema<A, TypeParameters>` record whose keys are plain identifiers (`schemaId`, `identifier`, `title`, `description`, `examples`, `default`, `documentation`, `jsonSchema`, `arbitrary`, `pretty`, `equivalence`, `concurrency`, `batching`, `parseIssueTitle`, `parseOptions`, `decodingFallback`, `message`, `missingMessage`, `typeConstructor`), the INNER layer is the open `Annotations` record on every AST node keyed by the `*AnnotationId` unique symbols, and `builtInAnnotations` is the closed lookup table that maps each outer key to its inner symbol — so authoring an annotation is a typed record the library projects onto the symbol-keyed substrate, never a hand-built symbol record.
- `toASTAnnotations(record)` is the boundary translation that walks `builtInAnnotations`, moves each recognized key's value under its `*AnnotationId` symbol, and DELETES the plain key — so the typed-key spelling exists only at the call seam and the substrate carries pure symbol keys, the identifier-string keyspace and the symbol keyspace never colliding because the projection consumes the former; an unrecognized key passes through verbatim (the open `[_: symbol]` slot), so a custom symbol-id payload rides the same record beside the built-ins.
- The map is fixed and total over the typed slots, so adding a slot is one table row the whole annotation pipeline re-reads — the typed `Annotations.Schema` interface and the `builtInAnnotations` table are the one declaration that yields both the author-facing key set and the substrate-facing symbol set, never a parallel `set*Annotation` call per slot.

```typescript
import { Option, Schema, SchemaAST as AST } from 'effect'
const Owner = Schema.Struct({ key: Schema.String }).annotations({
    identifier: '<owner-id>',                                            // typed key -> IdentifierAnnotationId on the node
    title: '<title>',                                                    // typed key -> TitleAnnotationId
    parseOptions: { onExcessProperty: 'error' },                         // typed key -> ParseOptionsAnnotationId
})
const sameNode = Option.zipWith(                                         // the named getter and the generic symbol read resolve one slot
    AST.getIdentifierAnnotation(Owner.ast),                             // symbol pre-applied: getAnnotation(IdentifierAnnotationId)
    AST.getAnnotation<string>(AST.TitleAnnotationId)(Owner.ast),        // generic form: the symbol IS the substrate key the projection wrote
    (id, title) => `${id}:${title}`,
)
```

[THE_SCHEMA_ID_IS_AT_ONCE_THE_DISCRIMINANT_AND_THE_KEY_TO_ITS_OWN_PARAMS]:
- A built-in refinement does not carry its parameters under named field keys; it writes the SAME unique symbol into TWO positions of the one annotation record — `SchemaIdAnnotationId -> theSchemaId` marks WHICH refinement this is, and `[theSchemaId] -> { ...params }` carries that refinement's payload under the schema-id symbol used as its own key — so `minLength(n)` is `filter(predicate, { schemaId: MinLengthSchemaId, jsonSchema: { minLength: n }, title, description })` and a projector recovers the params by reading `ast.annotations[ast.annotations[SchemaIdAnnotationId]]`, the symbol serving as both the recognition tag and the indirection key into its own record.
- `fromBrand(ctor)` and `instanceOf(ctor)` follow the identical idiom — `{ schemaId: BrandSchemaId, [BrandSchemaId]: { constructor } }` and `{ schemaId: InstanceOfSchemaId, [InstanceOfSchemaId]: { constructor } }` — so the carrier's runtime dependency (the `Brand.Constructor`, the class) is stored under the schema-id symbol and recovered by the same double-index a numeric bound is; the registry is one mechanism whether the payload is a wire keyword, a constructor reference, or empty.
- `TypeConstructorAnnotation` (`{ _tag: string }`, written by `instanceOf` carriers as `{ typeConstructor: { _tag: '<ctor>' } }`) is the same family one rung up — a carrier-identity tag under its own `TypeConstructorAnnotationId`, distinct from the schema-id discriminant because it names the RECONSTRUCTING constructor rather than the validating refinement, so a native carrier (`URL`, a `Date`) declares both a schema-id for keyword/constraint recognition and a type-constructor for its `_tag`-keyed identity, two registry slots answering the recognition and the reconstruction questions on one declaration.
- The double-index degrades cleanly to a pure discriminant when a refinement needs no params: `Int`, `Finite`, `NonNaN`, and `ValidDate` carry only `SchemaIdAnnotationId -> theSchemaId` with no `[theSchemaId]` record, so a projector that dispatches `schemaId === IntSchemaId` reads the identity alone — the schema-id is a nullary tag in that case, a parameterized key in the bounded case, one slot spanning both.
- The trap is that the schema-id is the ONLY stable identity of a refinement across the projector graph: two `filter` calls with structurally-identical predicates but different schema-ids are distinct refinements to every projector, and a custom `filter` with NO `schemaId` is anonymous — recognized by no keyword fold, contributing nothing to the wire document or the generator's constraints even when its predicate enforces a real bound the codec checks.

[PROJECTOR_RECOGNITION_IS_SYMBOL_IDENTITY_NEVER_PREDICATE_INSPECTION]:
- The wire-document and generator folds cannot read a refinement's predicate body — a closure is opaque — so recognition of a built-in keyword is `===` against the schema-id symbol: the document fold tests `ast.annotations[SchemaIdAnnotationId] === IntSchemaId` to decide whether an emitted `{ type: "number" }` must become `{ type: "integer" }`, and `ParseJsonSchemaId` is tested by the same reference equality to detect a JSON-string transform — the fold branches on symbol identity, the predicate never inspected.
- The generator fold reads the schema-id to select a CONSTRAINT FAMILY rather than to read a number: a `Refinement` over `NumberKeyword` maps `schemaId === NonNaNSchemaId` to a no-NaN constraint, `schemaId === FiniteSchemaId || schemaId === JsonNumberSchemaId` to a no-infinity-no-NaN constraint, and every other id to a min/max/integer constraint synthesized from the merged keyword params — so the generator obeys the exact bound the codec validates because both read the one schema-id-keyed record, the constraint never duplicated as a generator argument.
- A leaf carrier whose structure the AST cannot see (a `Declaration`) is re-typed by its schema-id alone: the generator fold rewrites a `Declaration` tagged `DateFromSelfSchemaId` into a synthetic date node carrying date constraints, so the schema-id is the bridge that gives an opaque native carrier a structurally-derived generator — the symbol identity standing in for the structure the fold cannot traverse.
- The recognition is open at the value level and closed at the projector level: the `Annotations` record admits any symbol at any node, so a schema-id the fold does not test is carried silently and ignored — a custom schema-id earns a keyword in the document or a constraint in the generator only by the fold being extended to test it, the symbol a no-op until a projector arm names it.

[STACKED_REFINEMENT_KEYWORDS_MERGE_AS_A_NARROWEST_BOUND_MONOID]:
- Two refinements of the same axis stacked on one owner (`pipe(minLength(2), minLength(5))`, a `Between` over a `GreaterThan`) do not overwrite — the document fold runs `mergeRefinements(from, jsonSchema, ast)` which spreads the inner refinement's keywords under the outer's and, for every overlapping bound keyword, keeps the TIGHTER value inline and pushes the looser into `allOf`: `minLength` keeps the max, `maxLength` the min, `minimum` the max, `maximum` the min, `multipleOf` and `pattern` push any non-equal duplicate to `allOf` — the merge is a meet over the bound lattice, the strictest constraint surfacing as the node's own keyword and the redundant ones preserved as `allOf` members rather than discarded.
- The merge is therefore the identity-bearing fold the refinement chain composes to: the order of stacking is erased (meet is commutative), a contradictory pair (`minLength(5)` then `maxLength(2)`) is preserved verbatim as two `allOf` members so the document reflects the unsatisfiable shape rather than silently collapsing it, and the consumer reading the document sees one tightest bound plus the witness of every contributing refinement — the wire contract derived from the chain, never authored beside it.
- `isOverrideAnnotation` gates whether a refinement's own jsonSchema annotation REPLACES the inner shape rather than merging: a non-`Int` refinement whose annotation declares a `type`/`oneOf`/`anyOf`/`$ref` overrides the inner document outright, while `IntSchemaId` is the one schema-id treated specially — its `{ type: "integer" }` overrides only when the inner already typed `number`, so the integer refinement re-types in place where a generic override would replace — the schema-id deciding the merge MODE, not merely contributing a keyword.

[ANNOTATING_IS_AN_IDENTITY_BEARING_AST_REWRITE_NOT_A_FIELD_SET]:
- The merge that strips the inherited identifier also MINTS a new node identity: `AST.annotations` clones the node's own-property descriptors and `Object.create`s a fresh node on the SAME prototype, so re-annotating is referentially distinct from its source while the prototype-carried `.fields`/`.members`/`.pick` interface survives — the identifier-deletion and the identity-mint are one rewrite, the deleted `$ref` name and the new object reference together making a re-annotated sub-shape a node the document deduplicates independently of its parent rather than an alias sharing the parent's reference and name.
- The rewrite recurses through the surrogate: when the node carries a `SurrogateAnnotation` (a substitute AST a projector folds instead of the opaque carrier), `AST.annotations` re-applies the same overrides to the surrogate and rewrites that slot too, so an override placed on a native carrier reaches the projection-visible shape and the carrier in one operation — the surrogate inheriting the identity rewrite and the cleared identifier rather than freezing at the pre-annotation shape, a stale surrogate `$ref` the defect this recursion forecloses.
- A read-modify-write annotation accumulates rather than overwrites where the slot's semantics are additive: the brand modifier reads the existing `BrandAnnotation` (a `NonEmptyReadonlyArray<string | symbol>`), appends the new brand, and writes the grown array through `AST.annotations` — so `pipe(brand('A'), brand('B'))` carries `['A', 'B']` and the nominal identity stacks, the merge folding the brand list rather than replacing it, the additive slot the exception to the last-wins overwrite the scalar slots obey, and each `brand` call passing through the identity-rewrite so the accumulated array rides a fresh node with the parent identifier cleared.

```typescript
import { Option, Schema, SchemaAST as AST } from 'effect'
const Tagged = Schema.NonEmptyString.pipe(Schema.brand('<brand-a>'), Schema.brand('<brand-b>')).annotations({
    identifier: '<leaf-id>',                                             // identifier present on THIS node
})
const reIdentified = Tagged.annotations({ title: '<new-title>' })        // fresh node: IdentifierAnnotationId deleted, prototype + brand array preserved
const witness = {
    brands: AST.getBrandAnnotation(Tagged.ast),                          // Option<['<brand-a>', '<brand-b>']> — accumulated, not overwritten
    inheritedId: AST.getIdentifierAnnotation(reIdentified.ast),          // Option.none — re-annotation cleared the parent $ref name
    keptBrands: Option.isSome(AST.getBrandAnnotation(reIdentified.ast)), // true — additive slot survived the identity rewrite
}
```

[THE_STABLE_FILTER_FLAG_DECIDES_REFINEMENT_SURVIVAL_INTO_THE_ENCODED_BOUND_PHASE]:
- `encodedBoundAST` keeps refinements up to the first transformation, but NOT every refinement survives — the rewrite keeps a `Refinement` on the bound side only when `hasStableFilter(ast)` holds AND the refinement sits directly over a non-transform `from`; otherwise the refinement is shed to its `from` — so `StableFilterAnnotationId -> true` is the per-refinement identity bit that declares "this constraint is meaningful on the wire shape, not only the decoded value", and a refinement lacking it is erased from the encoded-bound projection even while it narrows the decoded type.
- The flag is set by exactly the refinements whose bound constrains the ENCODED structure rather than a decoded interior: the array-length filters (`minItems`, `maxItems`, `itemsCount`) self-mark `StableFilter: true` because a wire-shape generator must still know the array cardinality after transforms shed, while the string and number refinements (`minLength`, `greaterThan`, `pattern`) do NOT mark it because their bound is a decoded-value predicate — so the encoded-bound phase carries cardinality constraints and drops value constraints, the divergence a property of which refinements declared the identity bit, not a phase-wide rule.
- The encoded projection re-identifies the surviving node: when a suspend or referenced sub-owner crosses into the encoded phase, the rewrite writes a `JSONIdentifierAnnotationId` suffixed `Encoded` (and `EncodedBound` on the bound cut) so the encoded shape gets its own `$ref` name distinct from the decoded one — the same node yielding two document references under two derived identifiers, the wire and the decoded shapes deduplicated independently because the encoded cut mints a fresh identifier rather than reusing the decoded owner's.

[THE_RENDER_FOLD_IS_AN_ORDERED_PRECEDENCE_OVER_THE_DOCUMENTATION_SLOTS]:
- The human-facing name of a node in a parse-failure locus and a document title is one ordered fold over the documentation registry: `getExpected = identifier ?? (title ?? description ?? autoTitle) + brands` — the identifier wins outright, then a title, then a description, then the auto-title the constructors stamp (`'<id> (Type side)'`, `'Struct (Encoded side)'`), with the rendered brand chain appended unconditionally — so the failure projection and the document title draw from the same precedence, the name recoverable from the annotation registry without a parallel formatter, and a node with none of the four falls back to its `_tag`.
- `AutoTitleAnnotationId` is the LIBRARY-stamped title distinct from the user `TitleAnnotationId`: the struct/class/constructor projections write an auto-title naming the phase (`(Type side)`, `(Constructor)`, `(Encoded side)`) so a default failure message reads structurally without a user annotation, and a user `title` overrides it in the precedence — the two title slots ranked so the derived name yields where an authored name is present, never colliding.
- The document fold suppresses a built-in default title by REFERENCE identity, the sharpest registry trap: `filterBuiltIn` emits a `StringKeyword`/`NumberKeyword`/`BooleanKeyword`'s title or description into the document only when it is `!==` the prototype keyword node's own annotation object — so the default `'a string'` title carried on every bare `StringKeyword` (reference-shared with `AST.stringKeyword`) is omitted from the wire document, and a USER title set to a string equal in VALUE but distinct in reference survives — the document distinguishing an authored annotation from the inherited default by object identity, not by value, so annotating a leaf with the exact default string still surfaces it because the new record is a distinct reference.

[REACH_LIMIT_FENCE]:
- A custom schema-id earns its slot only by being TESTED in a projector arm that emits a keyword, a constraint, or a re-typing the default fold cannot reach — a `schemaId` written on a `filter` with no fold arm reading it is an anonymous tag the document and generator ignore, the constraint silently absent from every projection even when the predicate enforces it; the proof is the symbol appearing in a `===` branch of an extended fold, never a decorative id beside a predicate that decides nothing.
- A `[theSchemaId]: { params }` payload is authored only where a projector recovers it by the double-index — a numeric bound the document reads, a constructor the carrier dereferences; stashing data under a schema-id symbol that no fold reads is the parallel registration the registry deletes, the params belonging in the slot the projector indexes, never a side record the schema-id merely labels.
- The `StableFilter` bit is declared only on a refinement whose bound constrains the ENCODED structure (an array cardinality a wire generator needs); marking a decoded-value predicate stable forces a constraint into the encoded-bound projection that the wire shape does not carry, and omitting it on a genuine wire-structural bound silently drops that bound from the encoded phase — the flag set by whether the constraint is meaningful after transforms shed, never by reflex.
- A user `title`/`description` set equal in value to a leaf's built-in default is the annotation the document still emits (distinct reference) yet decides nothing the default did not — the slot restating the inherited name is deleted, the override reserved for a name the failure locus and document genuinely need to differ from the keyword default; reaching for an annotation to re-state the library's own title is the registry write that changes no projection.
- The equivalence, order, and arbitrary INSTANCE the registry's `equivalence`/`pretty`/`arbitrary` slots override are value-identity concerns the registry merely TRANSPORTS — the slot carries the instance to the projector fold, but the instance's construction, its contravariant re-basing, and its struct fold are owned where value identity is decided, not here; an `equivalence` slot is authored only to correct a comparison the structural fold gets wrong (a tolerance, a canonical collapse), the registry the carrier for that instance, never the place its algebra is built.
