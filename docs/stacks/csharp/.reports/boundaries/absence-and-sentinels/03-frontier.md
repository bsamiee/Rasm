# Absence and Sentinels — Frontier

[RETRY_AS_TRANSIENT_ABSENCE_PROBE]:
- `retryUntil(schedule, ma, e => e.HasCode(terminalCode))` is the correct surface for re-probing a transiently absent capability: the predicate stops retry when the error is terminal and allows retry on any transient code. The `retryWhile` form is the complement — retry continues while the predicate holds. Both forms accept a `Schedule` that bounds retry duration and controls inter-probe delay; without a schedule, the retry is unbounded and structurally infinite.
- Each retry iteration wraps its attempt in `BracketFail()`, which calls `WithEnvFail((EnvIO env) => env.LocalResources)`. `LocalResources` creates a new `Resources` tracker (`New(null, Token, Source, SyncContext)` — `null` resources triggers fresh allocation) while inheriting the outer `Source` and therefore the outer `CancellationToken`. The consequence: each attempt gets an isolated resource scope (any acquired-and-failed resources are released on that attempt's failure), but cancellation from the outer scope still terminates the retry loop. A terminal receipt from one attempt does not leak into the next attempt's resource scope.
- `retryUntil` without a schedule recurses via `Catch(BracketFail(), e => !predicate(e) ? RetryUntil(predicate) : IO.fail(e))` — this is a trampolined recursion on the `IO` data structure (each iteration returns a new `IO` node), not a stack recursion. The stack does not grow per attempt; the `IO` interpreter handles the chain iteratively. This is safe for large retry counts.

[BATCH_ABSENCE_PARTITION]:
- `FinExtensions.Partition<F, A>()` operates on `K<F, Fin<A>>` for any `F : Foldable<F>` and yields `(Seq<Error> Fails, Seq<A> Succs)` in a single fold — verified: the extension is defined as `extension<F, A>(K<F, Fin<A>> self) where F : Foldable<F>`. When a boundary sweeps a collection of capability probes, `Seq<Fin<CapabilityResult>>` folded through `Partition()` extracts both the typed failures and the usable results without short-circuiting and without discarding either side.
- `partitionFallible<F, M, A>(K<F, K<M, A>> fma) where F : Foldable<F>, M : Monad<M>, Fallible<M>` is the effectful form: it sequences each `K<M, A>` in the foldable, catches failures from `Fallible<M>`, and accumulates `(Seq<Error> Fails, Seq<A> Succs)` inside `K<M, ...>`. The fold body is `ms.Bind(s => ma.Bind(a => Pure((s.Fails, s.Succs.Add(a)))).Catch(e => Pure((s.Fails.Add(e), s.Succs))))` — verified. Errors are accumulated monoidally; the outer monad is never failed by individual inner failures.
- For `Option`-typed probe results, `Seq<Option<A>>.Somes()` is the absence-filtering fold: it is defined on `Seq<Option<A>>` (verified: `SeqExtensions.Somes<A>(this Seq<Option<A>> self)`) and returns `Seq<A>` of unwrapped present values. The absent results are silently dropped — this is correct only when absence carries no diagnostic content. When the None case carries evidence (a silent miss warrants a log or metric), use `Choose` with a labeled `None` branch instead.

[PURE_AS_DEFINITE_FALLBACK]:
- `option | Pure<A>(value)` is the idiomatic form for collapsing `Option<T>` absence into a definite fallback at an expression boundary. The `|` operator with a `Pure<A>` right-hand side calls `Choose(rhs.ToOption())`, which constructs `Some(rhs.Value)` and returns it when `lhs` is `None` — verified via `Pure<A>.ToOption()` returning `new Option<A>(Value)`. This is structurally equivalent to `IfNone(value)` but stays on the `Option` monad rail, composing with subsequent `Map`/`Bind` without re-wrapping.
- The `| Pure<A>` pattern eliminates the sentinel-replacement anti-pattern: instead of `IfNone(default(T))` returning a sentinel value back into the domain, the fallback is a named domain value constructed explicitly at the one seam where the absence is resolved. The fallback value at the `|` site is the single point of semantics, not a defaulted literal propagating inward.
- `option | Pure<A>(value)` returns `Option<A>`, not `A` — it does not escape the `Option` rail. To materialize the value, add `.IfNone(...)` or use `+` (the unary dereference operator, which calls `.As()`) at the final consumption site. Treating `| Pure<A>` as a terminal escape is incorrect; it only advances the fallback into the pipeline.

[SAFEHANDLE_SETHANDLEASINVALID_SENTINEL]:
- `SafeHandle.SetHandleAsInvalid()` sets the IsClosed bit (`Interlocked.Or(ref _state, 1)`) and calls `GC.SuppressFinalize(this)` — it does not call `ReleaseHandle`. This marks the handle as closed-but-not-released: the OS has already reclaimed the resource (e.g., via an inheritance-stripped or pre-closed descriptor), and no cleanup is needed. The projection law must test `handle.IsInvalid || handle.IsClosed` together before constructing `Some(handle)`, because `IsClosed` is a BCL field check (`(_state & 1) == 1`, always accurate), while `IsInvalid` is abstract and delegate to the subclass override, which may only inspect the handle value and not the closed bit.
- After `SetHandleAsInvalid`, `DangerousAddRef` throws `ObjectDisposedException` (via `ObjectDisposedException.ThrowIf((state & 1) != 0, this)` — verified on .NET 10 CoreLib) because bit 0 is set. The projection site that calls `DangerousAddRef` must occur before any `SetHandleAsInvalid` call or after an `IsInvalid || IsClosed` guard; there is no safe retry path after either flag is set.

[CANCELASYNC_CALLBACK_SCHEDULING_DIVERGENCE]:
- `CancellationTokenSource.Cancel()` calls `NotifyCancellation` which calls `ExecuteCallbackHandlers` synchronously on the calling thread. `CancellationTokenSource.CancelAsync()` — verified on .NET 10 CoreLib — checks `this.Change != null` semantics: when callbacks exist, it schedules `ExecuteCallbackHandlers` via `Task.Factory.StartNew(..., TaskScheduler.Default)` and returns the task. When no callbacks exist, it returns `Task.CompletedTask`.
- The divergence is a boundary design decision: a seam that transitions `Atom<CapabilityState>` from a cancellation callback registered via `UnsafeRegister` will have that transition run synchronously on the `Cancel()` caller's thread, but asynchronously (on a thread-pool thread) when triggered by `CancelAsync()`. The `Atom.SwapMaybe` CAS is safe from either thread, but any code that reads `atom.Value` immediately after calling `CancelAsync()` and before `await`ing the returned task may observe the pre-cancelled capability state. The seam must either `await CancelAsync()` before reading the cell, or use `Cancel()` when synchronous observation is required.
- `CancelAsync()` returns `Task.FromException(new ObjectDisposedException(...))` when the source is already disposed — verified. A seam that calls `CancelAsync()` must handle the returned task's exception; `await`ing it without a catch propagates the disposal exception. `Cancel()` on a disposed source throws synchronously via `ThrowIfDisposed()`.

[ATOM_VALIDATOR_CONSTRUCTION_ERROR_SURFACE]:
- `atomIO<A>(value, validator)` wraps `Atom.New(value, validator)` in `IO.lift`; when the initial value fails the validator, it throws `Exceptions.ValidationFailed` (verified: `throw Exceptions.ValidationFailed`), which carries `Errors.ValidationFailed` (code `-2000000011`, `IsExpected = true`). The thrown exception is caught by the `IO` interpreter and becomes a failed `IO<Atom<A>>` carrying that coded error. A recovery site that needs to distinguish "initial state invalid" from other construction failures should match against `error.HasCode(ErrorCodes.ValidationFailed)` (code `-2000000011`) or `error.Is(Errors.ValidationFailed)`.
- The two-arg `Atom.New(value, validator)` internal static (not `atomIO`) returns `Option<Atom<A>>.None` when validation fails — it does not throw. `atomIO` is the `IO`-lifted form that converts the `None` result to a thrown `ValidationFailed`. Code that calls the non-IO `atom(value, validator)` Prelude function (which returns `Option<Atom<A>>`) at a seam must handle `None` as a typed construction failure, not as a runtime error; it cannot use `atomIO` error-handling patterns because no exception is produced.

[LOCAL_CANCEL_RESOURCE_SCOPE_ASYMMETRY]:
- `EnvIO.LocalCancel` creates `New(Resources, Token, null, SyncContext)` — it inherits the outer `Resources` tracker and allocates a new `CancellationTokenSource`. Any resources acquired inside a `LocalCancel` scope are tracked in the shared outer `Resources` and will survive the inner timeout. `EnvIO.LocalCancelWithTimeout(TimeSpan)` calls `New(Resources, Token, null, SyncContext, timeout)` — same behavior, with the source auto-cancelling after the timeout.
- `IO.Timeout(TimeSpan)` creates `IOTimeout<A, A>(this, timeout, IO.pure)`, which internally uses a scoped environment with a timed source. Resources acquired inside the timeout window that are not explicitly released before expiry will remain in the outer `Resources` tracking if the outer context is not also `BracketFail`-isolated. The pattern for a bounded-lifetime probe that must not leak acquired resources on timeout is `ma.BracketFail().Timeout(timeout)`, not `ma.Timeout(timeout).BracketFail()` — `BracketFail` wraps the outer `IO`, and `Timeout` is inside that bracket, so expiry counts as failure for the bracket's resource-release logic.
- The ordering matters: `Timeout` wraps `BracketFail` inside means the timeout fires within the isolated resource scope; `BracketFail` wrapping `Timeout` outside means the bracket releases on any failure, including timeout, from the outer scope. Only the former (`BracketFail().Timeout()`) guarantees resource cleanup on expiry.

[OPTION_BIBIND_AS_DUAL_PATH_ABSENCE_ROUTER]:
- `Option<A>.BiBind<B>(Func<A, Option<B>> Some, Func<Option<B>> None)` routes both the present and absent cases to new `Option<B>` values — both branches return `Option<B>`, keeping the pipeline monadic. The `None` branch receives no argument; it is a thunk. This is the correct surface when a seam must either transform a present value (possibly producing its own absence) or derive an alternative absence from a different source, without leaving the `Option` rail.
- `BiBind` composes denser than `Match` followed by re-wrapping: `Match(Some: x => Some(f(x)), None: () => None)` is the same shape but exits and re-enters the monadic context. `BiBind(Some: x => Some(f(x)), None: () => None)` stays inside it, enabling further `Bind`/`Map` chaining without an intermediate re-entry. The distinction matters at seams that route to downstream pipeline stages rather than materializing a value.
- Unlike `BiMap`, `BiBind` does not call `Check.NullReturn` on either branch's result — the null-safety law is that neither branch must return `null`. A `Some(null)` from a `BiBind` `Some` branch produces `Option<B>` with a null `Value` field (since `Option<B>` is a struct, and `new Option<B>(null)` sets `isSome = true` and `Value = null`), resulting in a `Some`-state option with a null interior value that defeats the boundary guarantee. The `Some` branch must never return `Some(null)`.

[FOLDABLE_HASHMAP_HEALTH_DERIVATION]:
- `HashMap<K, V>` is `K<HashMap<K>, V>` — the higher-kinded form — and `HashMap<Key>` (the static class, not the struct) implements `Foldable<HashMap<Key>>`. `Fold` is therefore available as `K<HashMap<Key>, V>.Fold<S>(initialState, (v, s) => ...)` via the `FoldableExtensions` extension, not directly on the `HashMap<K, V>` struct. Code that uses `.AsIterable().Fold(...)` or LINQ `.Aggregate(...)` instead of the `Foldable` trait bypasses the trait machinery and is not polymorphic — it cannot be lifted into an `IO` fold that checks cancellation per element.
- For a health aggregate over `AtomHashMap<CapabilityKey, CapabilityState>`, the vocabulary-complete fold requires projecting every key in the vocabulary, not just the keys present in the map: `smartEnum.Values.Map(key => (key, registry.Find(key).IfNone(CapabilityState.Unknown)))`. `AtomHashMap.Find(key)` returns `Option<V>` — verified. Keys absent from the map represent capabilities not yet probed, and they must be mapped to a defined state (`Unknown` or `Unavailable`) for the health fold to produce a complete picture. Folding only over `registry.AsIterable()` silently omits unprobed keys.
