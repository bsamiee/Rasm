# [DISPATCH_INTERNALS_OVERHEAD_ADVANCED]

[SINGLEDISPATCHMETHOD_DESCRIPTOR_PROTOCOL]:
- `singledispatchmethod` stores a `singledispatch` closure as `self.dispatcher` and wraps it in `_singledispatchmethod_get` on every `__get__` call; `__get__` receives `(obj, cls)` and sets `_dispatch_arg_index` to `1` when `obj is None and isinstance(func, FunctionType)`, else `0`. The `_dispatch_arg_index=1` path means unbound access (`Class.method`) dispatches on `args[1]` (the actual payload) rather than `args[0]` (the instance), enabling correct unbound-style calls like `Class.method(instance, payload)`.
- `@classmethod` wraps the function in a `classmethod` descriptor whose `type` is not `FunctionType`; therefore `_dispatch_arg_index` stays `0` for class-level access, and dispatch fires on `args[0]` which is the payload after the descriptor strips `cls`. `singledispatchmethod` with `@classmethod`-wrapped base and `@classmethod`-wrapped registered arms dispatches correctly on both `Class.method(payload)` and `instance.method(payload)`.
- `singledispatchmethod` does not implement `__set_name__`; placing it alongside a `RegistryDescriptor` that does implement `__set_name__` causes no interaction — they are independent class-body descriptors. `@property` stacked inside `singledispatchmethod` (`@singledispatchmethod @property def …`) returns a bound-method object rather than invoking the property getter; `@property` outside (`@property @singledispatchmethod`) raises `TypeError: 'singledispatchmethod' object is not callable`. Neither combination is valid for a dispatch surface; expose computed properties as plain properties separately.
- The shared-dispatcher trap: `singledispatchmethod` stores the dispatcher in the descriptor on the owning class's `__dict__`, not on the instance or subclass. Registering via `SuperClass.method.register(T)` mutates the single shared closure regardless of which subclass calls it; subclasses that do not define their own `singledispatchmethod` inherit the base descriptor directly. All cross-class arm mutations are permanent and global: `BaseClass.process.register(int)(fn)` immediately affects `SubClass().process(42)`.
- `__isabstractmethod__` on `singledispatchmethod` delegates to `getattr(self.func, '__isabstractmethod__', False)`; stacking `@abstractmethod` inside the descriptor propagates abstract-ness to `ABCMeta` and prevents instantiation. The concrete subclass must override the base `process` implementation to resolve the abstract obligation; arm overrides alone do not satisfy it because `singledispatch`'s default arm is what's abstract.

[FIND_IMPL_MRO_RESOLUTION]:
- `_find_impl(cls, registry)` calls `_compose_mro(cls, registry.keys())` which runs a modified C3 linearization. `_compose_mro` first filters registry keys to those related to `cls` (subclasses of `cls` not already in `cls.__mro__`), strips strict bases (types whose MRO already covers another candidate), then stabilizes ABC ordering by ranking candidates via their most-subclass-specific registered entry, and feeds the result to `_c3_mro`. The final iteration: walk the composed MRO; on the first registry hit, set `match`; on the second, check ambiguity — both must be absent from `cls.__mro__` and neither a subclass of the other to raise `RuntimeError`.
- `_compose_mro` filters `GenericAlias` entries out; `list[int]` is a `GenericAlias` and is never used as a registry key. The type-erase-to-origin rule (`get_origin(tp) or tp`) follows from this: at registration, `list[int]` expands to nothing, not `list`; use unsubscripted types as keys always.
- When both `Mapping` and `MutableMapping` are registered, `_compose_mro` eliminates `Mapping` because it is a strict base of `MutableMapping` (`Mapping in MutableMapping.__mro__`). Resolution produces `MutableMapping` for `dict` with no ambiguity. Ambiguity fires only when two registered ABCs are both virtual superclasses of `cls` and neither is a subclass of the other — e.g., registering two unrelated ABCs `Sized` and `Iterable` when `cls` structurally satisfies both without explicit inheritance of either.
- After any ABC registration, `cache_token` is set to `get_cache_token()` (the global `ABCMeta._abc_invalidation_counter`). Every subsequent `dispatch(cls)` call checks `cache_token != get_cache_token()`; if they differ, `dispatch_cache.clear()` runs before the lookup. This check costs ~5% on warm paths (165 ns vs 154 ns per call). The cold miss path — `WeakKeyDictionary` miss → `registry` miss → `_find_impl` → `_compose_mro` → `_c3_mro` — runs at ~4 400 ns/call, a 28× penalty vs the 160 ns warm path.

[DISPATCH_OVERHEAD_TABLE]:
Form overhead is call overhead above a direct function call (27 ns baseline).

| form | ns/call | overhead vs direct |
| :--- | ---: | ---: |
| direct call | 27 | 1.0× |
| `match` (1 class arm, first) | 37 | 1.3× |
| `match` (3 class arms, last) | 54 | 2.0× |
| dict value table (`type(x)` key) | 52 | 1.9× |
| `singledispatch` warm, no ABC | 154 | 5.6× |
| `singledispatch` warm, ABC registered | 165 | 6.0× |
| `singledispatch` cold miss (`_find_impl`) | 4 405 | 161× |
| `match` `Protocol()` class pattern (pass) | 117 | 4.3× |
| `match` `Protocol()` class pattern (fail) | 609 | 22× |

`match` class-pattern scan is linear but per-arm cost is low (~17 ns); at ≥9–10 arms, `singledispatch` warm cost is lower per additional type. Dict value table is always faster than `singledispatch` for bounded-key dispatch because it skips the `WeakKeyDictionary` layer entirely. `runtime_checkable` Protocol class patterns have asymmetric cost: the pass path short-circuits on the first matching attribute; the fail path exhausts all protocol members.

[STRUCTURAL_PATTERN_MATCHING_DEPTH]:
- Sequence patterns use `isinstance(subject, collections.abc.Sequence)` but explicitly exclude `str`, `bytes`, and `bytearray` at the language level; a subject of type `str` never matches `[first, *rest]`. `tuple` matches because it is a `Sequence`. Custom `collections.abc.Sequence` subclasses match.
- Mapping patterns use `isinstance(subject, collections.abc.Mapping)`; any `Mapping` subclass (including custom implementations without `dict` as a base) matches. `**rest` inside a mapping pattern captures all keys not explicitly named in the pattern; the explicitly matched keys are excluded from `rest`.
- Class patterns with `__match_args__`: positional sub-patterns in `ClassName(p0, p1)` are matched against attributes named by `__match_args__[0]`, `__match_args__[1]`; keyword sub-patterns `ClassName(attr=p)` bypass `__match_args__`. Dataclasses auto-generate `__match_args__` from their field order. The class pattern calls `type.__instancecheck__` (or `ABCMeta.__instancecheck__` for ABC bases); a non-`@runtime_checkable` `Protocol` raises `TypeError` in a class pattern.
- OR patterns `p1 | p2` have a binding constraint: every alternative must bind the same set of names with compatible types. `[x, 0] | [0, x]` is valid; `[a, 0] | [0, b]` is a `SyntaxError`. Captures inside OR alternatives are union-typed from the checker's view; the runtime makes no type distinction and assigns whatever the matching branch captured.
- Guards (`if expr`) run after the pattern has successfully matched and captured all names; the guard can reference the captures from the same case arm. If the guard returns falsy, the match engine continues to the next arm — the captures are unbound in the enclosing scope. A guard that raises does not skip; it propagates.
- Mapping pattern performance asymmetry: a full mapping pattern `{'type': 'click', 'button': b, 'x': x, 'y': y}` runs at ~143 ns/call for a 3-key subject vs ~89 ns/call for `dict.get('type')` followed by a value-pattern match on the string. The full mapping pattern checks all key presence in a single `isinstance` + `__getitem__` sequence; extracting the discriminant key first and branching on a string is consistently ~1.6× faster.
- `match` has no early-exit optimization across arms: arms are evaluated top-to-bottom; the engine does not compile a jump table for class patterns. For closed families of ≤8 types, `match` is faster than warm `singledispatch` because it avoids the `WeakKeyDictionary` lookup. For ≥9 types where all arms are type-only (no value sub-patterns), `singledispatch` O(1) warm dispatch is faster.

[PLUM_FULL_SIGNATURE_RESOLUTION]:
- Plum uses `beartype.door.is_bearable` configured with `BeartypeStrategy.On` (O(n) full check, not the default O(1)) for signature matching. `Signature.match(values)` calls `is_bearable(v, t)` for every positional argument. This means plum validates `list[int]` subscript depth (e.g., rejects `[1, 'x']` for a `list[int]` annotation) at every dispatch call, not just at registration.
- Resolution algorithm in `Resolver.resolve`: collect all methods whose `Signature.match(args)` returns true (candidates). Filter to the most specific via type dominance (`Signature.__le__` using `beartype.door.TypeHint` comparisons). If exactly one candidate remains, return it. If multiple share the highest precedence, raise `AmbiguousLookupError`. Precedence can be set per-registration to break ties manually.
- `Signature.__le__` delegates to `beartype.door.TypeHint(x) <= beartype.door.TypeHint(y)` for type comparisons, the full subtype relation. This means `list[int] <= list[int | str]` is True and plum resolves `list[int]` arms as more specific than `list[int | str]` arms. No equivalent is possible with `singledispatch` alone.
- Faithful type caching: plum caches `tuple(map(type, args)) -> (method, return_type)` only when `Resolver.is_faithful` is True (all registered signatures use faithful types). A type is faithful if `isinstance(x, t)` co-incides with `issubclass(type(x), t)`. Plain classes are faithful; `ABCMeta` with custom `__subclasshook__` may not be. Subscripted generics (`list[int]`, `dict[str, int]`) are not faithful — `_is_faithful` returns False when `get_args(x) != ()` and the origin is not a Union. Registering any `list[int]` arm disables caching for the entire function; every call re-runs beartype validation.
- Union dispatch: `Union[int, str]` as an annotation type registers as a single `Signature` whose match uses `is_bearable(v, Union[int, str])`; it is not expanded to two independent keys as `singledispatch` does with `f.register(int | str)`. An arm `def f(x: int | str)` competes with `def f(x: int)` — the latter is more specific and wins for `int` arguments; the former wins for `str`.
- Keyword arguments entirely bypass plum dispatch; they are forwarded to the resolved method's implementation without type-checking. Any design that puts discriminant information in a keyword argument cannot use plum. This is not a limitation of beartype but of plum's `Signature` which only captures positional types.
- `Function.__call__` resolves the method, then calls `_convert(method(*args, **kw), return_type)`; `_convert` is a no-op when `return_type is Any`, which is the default. Plum can coerce return values when a specific `return_type` is declared, using a parametric convert function injected at import time.

[OVERLOAD_ANNOTATIONLIB_BRIDGE_DEPTH]:
- `typing.get_overloads(fn)` returns registered stubs in definition order; `annotationlib.get_annotations(stub, format=Format.VALUE)` resolves all annotations to their runtime values including forward references. Combining both builds a `type -> stub` table at module load with zero runtime overhead at dispatch. The cache key is `get_origin(ann) or ann` — the unsubscripted origin — because subscripted generics like `list[float]` survive in the hints as `list[float]` but must be keyed by `list` for `type(x)` lookup. The full parameterized annotation is preserved as metadata for downstream validation.
- `Format.FORWARDREF` in `annotationlib.Format` is used internally by `singledispatch`'s annotation-based `register` path to safely handle unresolved forward references without eagerly evaluating them. For overload bridges that must resolve at module load, `Format.VALUE` is correct; `Format.FORWARDREF` would leave names as `ForwardRef` objects.

[FREE_THREADED_REGISTRATION_LAW]:
- `dispatch_cache` is a `WeakKeyDictionary`; `register()` calls `dispatch_cache.clear()` without a lock. In a free-threaded (no-GIL) build, a concurrent `dispatch()` reading the cache while `register()` clears it will miss and re-populate — this is safe but not atomic. The resulting extra `_find_impl` traversal is a correctness-safe cache miss, not a data race. However, a second `register()` call concurrent with `dispatch()` can yield a stale result for one call before the new registration is visible. Doctrine: all `register()` calls must complete before concurrent dispatch begins; import-time registration is the only safe pattern.
