# Aspect Architecture — Edge

[ENVALGO_ISOLATION_MATRIX]:
- `EnvIO` carries four independent axes: `Resources` (child resource scope), `CancellationTokenSource` (owns the token), `CancellationToken` (the observable signal), and `SynchronizationContext` (execution continuations). Four named `EnvIO` properties slice this matrix: `Local` creates a new `Resources` scope and a new `CancellationTokenSource` linked to the parent token (passes `null` for source, `env.Token` for token — `EnvIO.New` detects null source, creates a fresh CTS, and registers `env.Token.Register(src.Cancel)` if the token can be cancelled). `LocalResources` creates a new `Resources` scope only; source and sync context propagate unchanged. `LocalCancel` creates a new linked CTS only; resources and sync context propagate unchanged. `LocalSyncContext` captures `SynchronizationContext.Current` at construction time; all other axes propagate unchanged.
- `IO<A>.Local()` calls `WithEnv(env => EnvIO.New(null, env.Token, null, env.SyncContext))` — it triggers both new-Resources and new-linked-CTS branches simultaneously. Local cancellation (`IO.cancel`) inside this scope creates a `TaskCanceledException` that propagates upward, but a cancellation of the outer token does propagate inward (via the registered callback on the linked CTS). `Local()` provides regional isolation — not cancellation blocking, but cancellation propagation through a cleanly-owned scope that disposes its resources independently.
- `IO<A>.Uninterruptible()` wraps in `IOUninterruptible`, which executes against `EnvIO.New(null, CancellationToken.None, null, env.SyncContext)` — because `CancellationToken.None` cannot be cancelled, `EnvIO.New` creates a fresh unlinked CTS with no registration to the parent token. The outer token's cancellation does not reach the wrapped computation. `Uninterruptible` is therefore a true cancellation sink — the wrapped computation runs to completion or failure regardless of any external token. The error rail (`Fin`/`Error`) propagates normally; only the `CancellationToken` path is severed.
- `IO<A>.Timeout(TimeSpan)` wraps in an `IOTimeout` DSL node that calls `env.LocalCancelWithTimeout(timeout)` — constructing `EnvIO.New(null, env.Token, null, SynchronizationContext.Current, timeout)` which passes the timeout to `new CancellationTokenSource(timeout.Value)`. If the timeout fires, the linked CTS cancels, producing `Errors.TimedOut` (code `-2000000002`). `Errors.Cancelled` (code `-2000000001`) is what `IO.cancel` produces. Both are named constants in `LanguageExt.Common.Errors`; retryWhile predicates that must distinguish timeout-expiry from explicit cancel must compare against these codes, not against exception type.
- `IO<A>.Fork(Option<TimeSpan> timeout)` creates a child `Resources` scope (`new Resources(parentResources)`) that is disposed when the fork completes, and a new `CancellationTokenSource` linked to the parent via `env.Token.Register(tsrc.Cancel)`. Resources acquired in the parent before forking are visible inside the fork as shared (parent-owned) resources and are not auto-released when the fork completes. Resources acquired inside the fork are owned by the fork's child scope and are released on fork completion regardless of success or failure. The `ForkIO<A>` result is `readonly record struct(IO<Unit> Cancel, IO<A> Await)` — cancel fires `tsrc.Cancel()`, await blocks on the task.

[DEFINITION_TIME_DISPATCH_SURFACE_DEPTH]:
- `[UnionSwitchMapOverloadAttribute]` (`AllowMultiple = true`) is placed on a Regular Union class and declares additional `Switch`/`Map` overload families that stop at a specified set of types, handling those types and their siblings but not descending into their subtypes. `StopAt = new[] { typeof(FailureCase) }` generates a dispatch overload with `Action<FailureCase> failureCase, Action<SuccessCase> successCase, ...` — all sibling types plus the stop-at types themselves, treating each stop-at type as an opaque arm rather than dispatching into its cases. This is a coarser-grained dispatch surface generated alongside the full-resolution dispatch; both coexist on the same union type. The attribute is Regular-union-only: `AdHocUnionAttribute` inherits from `UnionAttributeBase` (not `UnionAttribute`), and `NestedUnionParameterNameGeneration` does not appear on the ad-hoc form.
- `NestedUnionParameterNameGeneration.Simple` on `[Union]` uses only the final leaf type name as the Switch/Map parameter name — `notFound` instead of `failureNotFound`. The warning is compiler-enforced: if two nested union families under the same root have types sharing a leaf name, the generator produces parameter names that collide, and the C# compiler emits a duplicate-parameter error. The `Default` strategy guarantees uniqueness at cost of compound names. The choice is load-bearing for callers: once set to `Simple` with stable leaf names, renaming an intermediate union type does not break existing call sites; with `Default`, renaming an intermediate type changes every compound parameter name it prefixes.
- `[IgnoreMemberAttribute]` marks a property or field invisible to the source generator, preventing it from being included in equality, hash, serialization, comparison, or Switch/Map generation. The constraint in the docs is explicit: the ignored member must be immutable. The use case is fields that carry derived or cached state computed from the key member — they are present as runtime surface but must not affect value identity or generated dispatch. A mutable ignored member breaks the value-object contract: two instances with equal keys but different mutable-ignored-field states would compare equal while being observably different.
- `[MemberEqualityComparerAttribute<TComparer, TMember>]` applies to individual properties or fields of a `[ComplexValueObject]`. It is structurally distinct from `[KeyMemberEqualityComparerAttribute<TComparer, TKey>]` (which applies to the single key member of a simple `[ValueObject<T>]`): `MemberEqualityComparerAttribute` targets named members of a complex value object where multiple members contribute to equality. The constraint `where T : IEqualityComparerAccessor<TMember>` requires the comparer type to implement the accessor interface, which is how the generator wires the comparer without reflection at generation time.
- `SkipKeyMember = true` on `[ValueObject<TKey>]` suppresses generation of the key member field/property, requiring a hand-written member at the name given by `KeyMemberName`. The generator still uses `KeyMemberName` to route serialization, comparison, and parsing aspects through the hand-written member. The use case is value objects with a custom backing representation — a struct storing multiple native fields but exposing a computed string key for identity purposes — where the generated backing field would be wrong but all serialization and comparison aspects must still route through the declared key name.
- `CreateFactoryMethodName` and `TryCreateFactoryMethodName` on `ValueObjectAttributeBase` override the generated public factory method names from `Create`/`TryCreate` to any valid identifier. This is a naming aspect on the generated admission surface: renaming the factory to match domain vocabulary (`Of`, `From`, `Parse`, `Wrap`) while retaining the full generated factory body including `ValidateFactoryArguments` advice hook wiring, null/empty yield-null handling, and serialization routing. The default values `"Create"` and `"TryCreate"` are runtime-computed from the backing field — null field returns the default — so an explicit whitespace value is normalized to null and the default name is restored.

[FACTORY_ROUTING_ASPECTS]:
- `ObjectFactoryAttribute<T>` carries `where T : allows ref struct` (C# 13 anti-constraint), enabling `T = ReadOnlySpan<char>` — a ref struct — as the factory input type. This is the admission path for span-based zero-allocation deserialization: the generator emits `IObjectFactory<Owner, ReadOnlySpan<char>, TError>` when `T` is `ReadOnlySpan<char>`, and STJ's source-generated reader drives the span overload directly. The anti-constraint (`allows ref struct`) does not require `T` to be a ref struct; it just permits it. Any ref-struct input type — including custom `ref struct` admission types — is therefore valid.
- `UseForSerialization` on `ObjectFactoryAttribute` is `SerializationFrameworks` (a `[Flags]` enum with `SystemTextJson = 1`, `NewtonsoftJson = 2`, `MessagePack = 4`), not a `bool`. A single type can carry multiple `[ObjectFactory<T>]` attributes with different `UseForSerialization` values, routing each serialization framework through a different factory. For example, `[ObjectFactory<ReadOnlySpan<char>>(UseForSerialization = SerializationFrameworks.SystemTextJson)]` routes only STJ through the span factory while Newtonsoft routes through the primary string-based factory. This is a multi-axis factory dispatch aspect declared entirely at definition time.
- `UseForModelBinding` and `HasCorrespondingConstructor` on `ObjectFactoryAttribute` are `init`-only properties (not `set`). This means they must be set in the attribute's object initializer at the declaration site and cannot be assigned later — they are fixed at the attribute construction time, which is itself a form of immutability enforcement on the aspect payload.

[COMPOSITION_TIME_CANCELLATION_ASPECTS]:
- `IO<A>.Local()` is the composition-time regional cancellation scoping aspect. The `Errors.Cancelled` error produced by `IO.cancel` inside a `Local()` scope propagates upward past the local boundary — `Local()` does not catch cancellation errors, it merely owns the scope for resource cleanup. To contain a cancellation and substitute a fallback value, the call site must compose `local(ma) | @catch(Errors.Cancelled, _ => fallback)`, with the catch applied outside the local scope.
- `Schedule.PrependZero` is a `Schedule` property (not a method): `this.Prepend(Duration.Zero)`. Its role in `RetryUntil` and `RepeatUntil` is that the first iteration fires with zero delay — the schedule's normal delays apply only between subsequent attempts. Without `PrependZero`, a schedule of `spaced(500*ms)` would delay even the first attempt by 500ms. Both `RetryUntil` and `RepeatUntil` implementations unconditionally call `schedule.PrependZero.Run().GetIterator()` — this zero-delay prefix is not optional and not caller-controllable. The `Schedule.NoDelayOnFirst` transformer applies the same pattern to any schedule the caller composes: `NoDelayOnFirst` is `Transform(s => s.Tail.Prepend(Duration.Zero))`, which strips the schedule's natural first step and prepends zero — but for `RetryUntil`/`RepeatUntil`, the `PrependZero` is already implicit and stacking `NoDelayOnFirst` on top of those schedule calls is idempotent (it produces double-zero-prepend but does not cause extra retries because the schedule drives iteration count, not delay count alone).
- `IO<A>.Try()` is an aspect adapter: it wraps any `IO<A>` into `FinT<IO, A>` by mapping success through `Fin.Succ` and catching all errors through `Fin.Fail<A>`. The resulting `FinT<IO, A>` insulates the error rail inside the `IO` monad — `Catch` combinators applied to the outer `K<IO, Fin<A>>` computation see `Fin.Fail` values, not propagated `Error` values. This is the boundary insulation aspect: code that must inspect both success and failure paths as values (rather than as rail continuations) wraps via `Try()`, operates on `Fin<A>`, and the outer monad sees no errors unless they arise from the inspection code itself.
- `IO<A>.Combine(IO<A> rhs)` is the `Monoid<IO<A>>` implementation: it catches every error from `this` (predicate `_ => true`) and substitutes `rhs`. `IO<A>.Empty` is the monoid identity — a computation that always fails with a specialized `IOEmpty` error. The semantics are alternative resolution: `first | second | third` via `|` on the `Fallible` extension block picks the first non-failing computation; `IO<A>.Empty` acts as zero in this algebra. The distinction from retry: `Combine`/`|` provides a fallback computation, not a re-invocation of the same computation; errors from `rhs` propagate normally because no catch wraps `rhs`.
- `IO<A>.Post()` captures `env.SyncContext` at the start of chain execution, not at `Post()` call site position. The `SyncContext` stored in `EnvIO` is set by `EnvIO.New` at the time the environment is constructed — either `SynchronizationContext.Current` at construction time (when `syncContext` parameter is null) or an explicit value passed in. A `Post()` mid-chain routes back to that captured context, not the current thread's context at the moment `Post()` executes. Wrapping with `IO.lift(computation).Post()` runs `computation` on the sync context captured when `EnvIO.New` was first called; if that was a UI thread's context, this marshals back to the UI thread regardless of which thread executed the chain up to that point.

[CRON_CALENDAR_SCHEDULE_ASPECTS]:
- Four calendar-aligned schedule constructors — `secondOfMinute(int)`, `minuteOfHour(int)`, `hourOfDay(int)`, `dayOfWeek(DayOfWeek)` — each accept an optional `Func<DateTime>? currentTimeFn` clock injection parameter (defaulting to `() => DateTime.Now`). These generate window-aligned schedules: `secondOfMinute(30)` fires at second 30 of each minute, computing the next delay as the remaining time until the next occurrence of second 30 on the system clock. They are composable via `|`/`&`/`+` with other schedules and transformers. The clock injection parameter makes them deterministically testable — a fake clock advancing in controlled steps drives the schedule without wall time.
- Calendar schedules round their integer argument to the valid range: `secondOfMinute` and `minuteOfHour` round to `[0, 59]`, `hourOfDay` rounds to `[0, 23]`. The rounding is performed by `roundBetween(value, min, max)` which clamps — `secondOfMinute(-5)` yields second 0, `secondOfMinute(75)` yields second 59. The `dayOfWeek(DayOfWeek)` schedule takes the BCL `DayOfWeek` enum directly with no rounding.
- `Schedule.windowed(interval)` and `Schedule.fixedInterval(interval)` are the two clock-based non-calendar schedules. `windowed` aligns to epoch multiples of `interval` — the delay to the next step is `interval - (elapsed % interval)`. `fixedInterval` keeps a floor of zero and does not pile up: if the action took longer than `interval`, the next delay is zero rather than adding to a deficit. Both accept the optional clock function. `upto(max, currentTimeFn)` terminates after wall-clock elapsed time exceeds `max`, which diverges from `maxCumulativeDelay(max)` (terminates after sum of emitted delay values exceeds `max`) whenever action duration is non-negligible.

[OPERATOR_CASCADE_CONSTRAINTS]:
- `EqualityComparisonOperators` on `ValueObjectAttribute<T>` and `SmartEnumAttribute<TKey>` is coerced upward to match `ComparisonOperators` when `ComparisonOperators` is set higher. The getter computes `max(ComparisonOperators, EqualityComparisonOperators)` because comparison operators require equality operators in generated code. Setting `ComparisonOperators = DefaultWithKeyTypeOverloads` without explicitly setting `EqualityComparisonOperators` silently forces `EqualityComparisonOperators` to `DefaultWithKeyTypeOverloads` as well — equality overloads accepting the raw key type are generated even if the declaration did not request them. The reverse is not true: setting `EqualityComparisonOperators = DefaultWithKeyTypeOverloads` without setting `ComparisonOperators` leaves comparison operators at whatever was explicitly declared.
- `SkipEqualityComparison = true` triggers a full cascade: `IEquatable<T>`, `Equals`, `GetHashCode`, equality operators (`==`, `!=`), and comparison operators (`<`, `>`, etc.) are all suppressed simultaneously. `EqualityComparisonOperators` and `ComparisonOperators` getters return `OperatorsGeneration.None` when `SkipEqualityComparison` is set, regardless of what they were explicitly assigned — the skip is a dominating flag. On a struct value object, suppressing equality prevents the generated `default(T)` guard from being hashable, which is a constraint for use in `FrozenDictionary` and `FrozenSet` unless the hand-written `Equals`/`GetHashCode` replace the suppressed generated versions.
- `SkipFactoryMethods = true` on `ValueObjectAttributeBase` forces a cascade across five aspects simultaneously: `TypeConverter` attribute is not emitted; `IObjectFactory<T>` interface is not implemented; the conversion operator from key type to value object is not generated; `SkipIParsable` is forced to `true`; arithmetic operators (`AdditionOperators`, `SubtractionOperators`, `MultiplyOperators`, `DivisionOperators`) are all forced to `None`. Serialization converters are still generated unless `[ObjectFactory<T>]` with `UseForSerialization` is absent — if `SkipFactoryMethods` is true and no serialization-specific factory is present, no serialization converters are emitted. The cascade is one-directional: setting any of the cascaded properties individually does not set `SkipFactoryMethods`.
