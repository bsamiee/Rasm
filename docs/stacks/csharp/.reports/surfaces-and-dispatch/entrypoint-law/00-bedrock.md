# Entrypoint Law

[MODAL_ARITY_MECHANICS]:
- `SwitchMapMethodsGeneration.Default` emits full-coverage `Switch`/`Map` overloads; `DefaultWithPartialOverloads` additionally emits separately named `SwitchPartially`/`MapPartially` methods — not additional overloads of `Switch`/`Map`. Each `SwitchPartially` method places a nullable `Action<MyUnion>? @default` as its first parameter followed by one nullable per-case parameter; unhandled cases invoke `@default` if non-null. The state-threaded form is always present for both: `Switch<TState>(TState state, Action<TState, CaseA> caseA, ...)` — zero-allocation because no closure forms around captured state. Because `SwitchPartially` and `Switch` carry distinct names, no `[OverloadResolutionPriority]` is needed and there is no overload-resolution interplay between them.
- `SwitchMapStateParameterName` is a per-union string override (default `"state"`) that renames the threading parameter in generated signatures. When the same domain type hosts multiple nested `[Union]` families, distinct state-parameter names prevent generated-overload ambiguity at call sites that chain dispatchers.
- `NestedUnionParameterNameGeneration.Default` produces `failureNotFound`-style names (intermediate type name prefixed); `Simple` uses only the final leaf name and emits a compile error when two nested families share a leaf name — choice is load-bearing, not cosmetic. `NestedUnionParameterNameGeneration` governs nested case-type hierarchies only (abstract case with sub-cases); a union-typed property on a case does not affect the outer dispatcher's parameter names. `[UnionSwitchMapOverload(StopAt = [typeof(AbstractCase)])]` is `AllowMultiple = true` and generates an additional `Switch`/`Map` overload alongside the default full-coverage form, treating `AbstractCase` as atomic — exhaustive at the stop boundary, no fallback arm. Multiple attributes generate multiple additional overloads with distinct stop boundaries.
- `FactoryMethodGeneration.Default` triggers factory generation only when a member forces it: type-parameter members, `object`, interface members, or duplicate types. `None` suppresses all factories unconditionally; `Always` generates for every member. Suppressing with `None` and providing a single static factory is the canonical way to implement a controlled entrypoint surface over a union whose cases the caller must not construct directly.
- Regular `[Union]` (nested-case form): the generator emits `Switch`/`Map` overloads only. No `.IsX`/`.AsX` properties are generated; case extraction routes through `Switch`/`Map` dispatch or C# type-testing (`value is CaseName c`), which works because regular-union cases are concrete subtypes of the abstract union base. Ad-hoc `[Union<T1,T2>]`: the generator emits named `.IsT1`/`.AsT1`-style properties (e.g. `IsInt32`, `AsString`) alongside `Switch`/`Map`; the union holds a value-index field and backing slots, not a polymorphic subtype, so `value is T1 v` type-testing against the member type does not extract the case — the union instance is not an instance of the member type. The two generation shapes contradict each other on the Is/As surface by design: regular unions expose case extraction via the type hierarchy; ad-hoc unions expose it via named computed properties. `ConstructorAccessModifier.Private` on `[Union]` makes all generated implicit conversion operators private; `ConversionFromValue = ConversionOperatorsGeneration.None` removes them entirely. `ConversionToValue = ConversionOperatorsGeneration.None` removes downcast operators, eliminating `InvalidCastException` risk from premature downcasting. The maximum-sealed combination — `ConversionFromValue = None`, `ConstructorAccessModifier = Private`, `FactoryMethodGeneration = None` — produces a union where input enters only through a hand-written static factory inside the partial class and output routes only through `Switch`/`Map`, with no type-system backdoor.

[SMARTENUM_KEY_AND_SPAN_FACTORY]:
- The source generator produces `Key` as the default key property name for `[SmartEnum<TKey>]` types. `IConvertible<TKey>.ToValue()` is an explicit interface implementation delegating to `Key`; direct use of `.Key` is the correct form at call sites and in table initialization.
- The generator unconditionally adds `IConvertible<ReadOnlySpan<char>>` and `IObjectFactory<T, ReadOnlySpan<char>, TValidationError>` to every `[SmartEnum<string>]` type without any manual attribute. Every string-keyed smart enum therefore supports span-based construction via `SmartEnum.Validate(ReadOnlySpan<char>, IFormatProvider?, out T?)`. The full zero-alloc dispatch table pipeline: `Kind.Items.ToFrozenDictionary(k => k.Key, StringComparer.OrdinalIgnoreCase).GetAlternateLookup<ReadOnlySpan<char>>()` routes span-key lookups to the case's delegate behavior without allocation.
- `[ObjectFactory<ReadOnlySpan<char>>]` on a `[ValueObject<string>]` generates a span-accepting `Validate(ReadOnlySpan<char>, IFormatProvider?, out T?)` interface contract, declared `where T : allows ref struct`. Implementing this on the partial class provides a zero-alloc admission path from incoming wire buffers without a heap-allocated string intermediate. The generated path through the frozen dispatch table can then couple directly to this span-accepting factory, making the entire admit-and-dispatch spine allocation-free for string-keyed commands.
- `[SmartEnum]` (no type parameter) is the keyless variant. The generator emits `Switch`/`Map` dispatch methods and an `.Items` collection but no key property, no `Get(key)`, no `TryGet(key, out item)`, and no `IObjectFactory` interfaces. When a module uses a keyless smart enum as a case discriminant, the dispatch surface is structural: the caller holds an instance and calls `Switch(state, case1: ..., case2: ...)`. There is no admission path from an external key — the module owns the only construction paths via the static fields. This is the correct form for internally-discriminated behavior rows where the external caller receives an item by reference, not by string key.
- `SwitchMapMethodsGeneration.None` on `[Union]` or `[SmartEnum<TKey>]` suppresses `Switch` and `Map` generation entirely. The correct form when dispatch is entirely inside extension blocks that implement a custom dispatch algebra over the union's public properties — the generated `Switch` methods would create a secondary, less-restrictive dispatch surface that bypasses the extension block's policy enforcement. `MapMethods` has the same `SwitchMapMethodsGeneration` type and can be suppressed independently of `SwitchMethods`.
- `[UseDelegateFromConstructor]` on a `[SmartEnum]` partial method produces a per-item delegate field wired through the constructor. The generator produces a private delegate type whose signature matches the partial method; each static field passes a static method reference at construction. This is the mechanism by which a smart-enum item carries case-owned behavior without a switch arm — the behavior lives with the vocabulary, and the entrypoint method is the partial method shared across all items.

[AD_HOC_VS_REGULAR_UNION_DISPATCH_SELECTION]:
- `[Union<T1,T2,...>]` (generic attribute, up to five type parameters) and `[AdHocUnion(typeof(T1),typeof(T2),...)]` (non-generic attribute, same up to five) are semantically equivalent. Both produce an implicit conversion from each `Ti` to the union and an explicit conversion back; both emit the same `Switch`/`Map` pair plus named `IsTi`/`AsTi` properties. Choose ad-hoc when the payload types are already defined and the union is a positional sum without named cases. Choose regular `[Union]` when cases carry distinct named fields — because regular-union cases are declared as nested `sealed` or `abstract partial` subtypes and the generator emits the type hierarchy that makes pattern-matching extraction work.
- `T1IsStateless = true` on an ad-hoc union member stores only the discriminator index, not the payload, eliminating the member's allocation for presence-only cases. The generated `AsTi` accessor for that member returns `default(Ti)`. Use struct types for stateless members to avoid null-handling complexity in the `Value` accessors.
- `UseSingleBackingField = true` forces the generator to use a single `object` backing field regardless of member types, including structs. This trades type safety at the backing-field level for a smaller generated surface and must not be chosen for performance-sensitive dispatch paths because it introduces boxing for struct members.

[PARAMS_COLLECTIONS_AND_ARITY_COLLAPSE]:
- `params ReadOnlySpan<T>` is the span-boundary absorber; it requires no allocation at the call site when the argument list fits on the stack. The BCL encodes this via `[System.Runtime.CompilerServices.ParamCollectionAttribute]` (distinct from `[ParamArrayAttribute]`). Declaring `static Fin<Result> Execute(params ReadOnlySpan<ItemCmd> cmds)` collapses singular, multi-item, and empty call sites into one entrypoint — the caller writes `Execute(a, b)` via a collection expression or `Execute([a, b])` and the compiler lowers the literal to a stack-allocated span when the count is statically known.
- `params ReadOnlySpan<T>` and `params IEnumerable<T>` are both valid `params` collection forms. The compiler resolves to the `ReadOnlySpan<T>` overload first when both exist — span is already a better conversion target for collection-expression arguments, so `[OverloadResolutionPriority]` is only needed when candidates are equally good conversions, not to select span over `IEnumerable`.
- The input-redescribing-parameter anti-pattern: a `bool isBatch` or `int count` parameter beside the value re-encodes discriminant information the type system already provides. The discriminant must be recoverable from the value's type, case, or arity without a separate knob. When a count or mode is needed internally, derive it from `span.Length` or from pattern-matching the union case — never surface it as a parallel parameter. A structural variant: a `CancellationToken` beside a union value is not redescription (it is a capability orthogonal to the discriminant), but a `bool processInParallel` beside a `Seq<Cmd>` is — the batch intent must be encoded in the union case's type, not as a parallel flag.
- `[OverloadResolutionPriorityAttribute(-1)]` applied to a deprecated `T[]`-accepting overload downgrades it relative to the span overload without deletion. The priority is evaluated before normal tiebreaking within the same declaring type; cross-type priority comparison does not apply — derived-type candidates are evaluated separately from base-type candidates.

[FROZEN_DISPATCH_TABLE_AND_SET_MECHANICS]:
- `FrozenDictionary<TKey, TValue>` carries `[CollectionBuilder(typeof(FrozenDictionary), "Create")]`, but the collection-builder path is not taken for dictionary-shaped targets — a collection expression assigned to `FrozenDictionary<string, Handler>` fails with CS0144 + CS9215 because `FrozenDictionary<TKey,TValue>` is abstract and has no `Add` surface. The canonical construction forms are `source.ToFrozenDictionary(keySelector, comparer)` for LINQ-sourced input and `FrozenDictionary.ToFrozenDictionary(kvps, keySelector, valueSelector)` for span-shaped input. Duplicate-key behavior differs by overload: the `keySelector` overload throws `ArgumentException` on the second key (verified); the `KeyValuePair`-source overload is last-wins (verified: count=1, later value retained). `FrozenSet.ToFrozenSet` deduplicates silently (no exception). The comparer overload controls key equality; passing the comparer at construction is the only configuration point — there is no post-construction comparer replacement. A `static readonly` field initialized via `ToFrozenDictionary` executes once at type initialization with no locking overhead.
- `FrozenDictionary<TKey,TValue>.GetValueRefOrNullRef(TKey)` returns `ref readonly TValue`; absence is detected with `Unsafe.IsNullRef(in slot)`. The dispatch idiom: `ref readonly var slot = ref table.GetValueRefOrNullRef(key); if (Unsafe.IsNullRef(in slot)) return Fin<Result>.Fail(...); return slot(request);` achieves zero-allocation lookup and invocation for `Func<Cmd, Fin<Result>>` values.
- `FrozenDictionary<TKey,TValue>.GetAlternateLookup<TAlternate>()` returns an `AlternateLookup<TAlternateKey>` `readonly struct` with `ContainsKey`, `TryGetValue`, and an indexer. The constraint `where TAlternateKey : notnull, allows ref struct` enables `ReadOnlySpan<char>` as the key type when the base comparer implements `IAlternateEqualityComparer<ReadOnlySpan<char>, string>`; the BCL's `StringComparer.OrdinalIgnoreCase` satisfies this. The dispatch table key stays `string`; the call site passes a span sliced from an incoming buffer without allocating. Caching the `AlternateLookup<TAlternateKey>` struct in a `static readonly` field avoids re-invoking `GetAlternateLookup` on each call; the struct wraps the dictionary reference and contains no delegate state.
- `FrozenSet<T>.GetAlternateLookup<TAlternate>()` works identically. A `FrozenSet<string>` with `StringComparer.OrdinalIgnoreCase` accepts `ReadOnlySpan<char>` lookups via its alternate lookup instance. The zero-allocation guard pattern: `if (!_validNames.GetAlternateLookup<ReadOnlySpan<char>>().Contains(nameSpan)) return Fin<Result>.Fail(Error.New("unknown command"))` pre-validates a command name before dispatching without a separate boolean parameter.

[VERB_FAMILY_COLLAPSE_INTO_REQUEST_UNION]:
- The canonical collapse of a verb family (`Create`, `Update`, `Delete`, `Patch`) is a `[Union]` with one case per verb, `ConstructorAccessModifier = Private`, and one total dispatch method on the module. The dispatch method receives the union value and switches over it, eliminating three or more sibling entrypoints with shared preamble — each preamble becomes the constructor of its case; shared validation becomes the dispatch prologue.
- `DefaultWithPartialOverloads` on the request union emits `SwitchPartially`/`MapPartially` methods with one nullable per-case parameter and a nullable `@default` fallback. When new cases are added, the full-coverage `Switch` becomes a compile error at all dispatch sites; `SwitchPartially` lets call sites that care about a subset continue to compile while the dispatch internals use the full-coverage form. Full-coverage at dispatch internals maintains exhaustiveness. For nested `[Union]` families inside a verb-family collapse, `NestedUnionParameterNameGeneration.Simple` is the correct choice only when all leaf names across all nested families are globally distinct; any collision is a compile error, not a silent shadow.

[DEEP_SURFACE_AND_MODULE_INTERNALS]:
- The module's public surface is the single dispatch entrypoint plus the request union type. When `ConstructorAccessModifier = Private` and `FactoryMethodGeneration = None`, all case constructors and generated factories are suppressed; the only construction surface is a hand-written `public static` factory method inside the partial class, and the `Switch`/`Map` methods remain fully operative. `ConstructorAccessModifier = Internal` makes case constructors accessible to the entire assembly — not just an enclosing type. When the desired construction scope is the enclosing type only, declare the union as a `private partial class` nested inside the service class and use `ConstructorAccessModifier = Private` with explicit factory methods on the service.
- Extension blocks allow adding the dispatch entrypoint to the union type without modifying the generated partial. The pattern: `extension(RequestUnion req) { public Fin<Result> Execute(IDepA dep) => ... }`. Extension blocks must be declared inside a top-level, non-generic, `static class` (CS9283 otherwise). Multiple `extension(SameReceiver)` blocks in the same static class are additive; multiple static classes each containing an `extension(SameReceiver)` block are also additive across the compilation — this is the mechanism for splitting test-only probing methods into a test-assembly extension block without modifying the production static class.
- Extension blocks cannot access `private` members of the receiver type (CS0122). The dispatch entrypoint written in an extension block must route entirely through the receiver's public API — the generated `Switch`/`Map` methods, public case properties, and explicit interface members. This is not a limitation for union dispatch because the generated `Switch` surface is public by definition.
- The `extension(T)` block itself does not accept an access modifier (CS0106). Visibility is controlled per-member inside the block: `public`, `internal`, and `private` all compile and carry their normal access semantics. A `private` member in an extension block on `RequestUnion` inside a `static class` is callable from another extension block on a different receiver type inside the same static class — this creates unexpected coupling between dispatch helpers for different receiver types. Shared helpers must be declared as `file`-scoped at top level or as `private static` methods directly on the containing static class, not inside any extension block. Properties in extension blocks (`public bool IsA => receiver is TypeA;`) compile to non-virtual static methods with a receiver parameter, suitable for attaching computed predicate properties to a union type without touching the generated partial.
- `[OverloadResolutionPriority]` applies to members inside extension blocks with the same semantics as on regular methods. A `ReadOnlySpan<T>`-based extension block member given higher priority wins over a `string`-based member at call sites that pass a literal, enabling the zero-alloc path to be the default without renaming either member.

[COMPOSITION_TIME_ASPECTS_ON_ENTRYPOINTS]:
- `Schedule` and `ScheduleTransformer` are distinct types. Methods returning `Schedule`: `spaced`, `linear`, `exponential`, `fibonacci`, `upto`, `fixedInterval`, `windowed`, and cron variants. Methods returning `ScheduleTransformer`: `recurs`, `maxDelay`, `maxCumulativeDelay`, `jitter`, `decorrelate`, `resetAfter`, `repeat`, `intersperse`; the static fields `NoDelayOnFirst`, `RepeatForever`, `Identity` (on `Schedule`) are also `ScheduleTransformer`. `ScheduleTransformer` carries an implicit conversion to `Schedule`; a bare transformer is therefore valid wherever a `Schedule` is expected.
- The `|` operator is defined for `(Schedule, Schedule)`, `(Schedule, ScheduleTransformer)`, and `(ScheduleTransformer, Schedule)` — all returning `Schedule`. `ScheduleTransformer | ScheduleTransformer` has no `|` overload and emits CS0019; two transformers compose only via `ScheduleTransformer + ScheduleTransformer` (op_Addition), which returns `ScheduleTransformer`. `Schedule | Schedule` is union: runs to the length of the longest, yields the minimum of both delays at each step. `Schedule & Schedule` is intersect: runs to the length of the shortest, yields the maximum of both delays. `(Schedule, ScheduleTransformer)` and `(ScheduleTransformer, Schedule)` both apply the transformer to the schedule operand. These operator semantics are orthogonal and the symbols are not interchangeable.
- The correct zero-first-delay bounded exponential for a retry entrypoint: `(Schedule.recurs(n) + Schedule.NoDelayOnFirst) | Schedule.exponential(seed)` — the two transformers are composed with `+` before being applied to the schedule via `|`. Alternatively, apply the transformers sequentially: `Schedule.exponential(seed) | Schedule.recurs(n) | Schedule.NoDelayOnFirst` — each `|` applies one transformer to the accumulated `Schedule`. For parallel retry from multiple entrypoints, `Schedule.decorrelate()` is preferable to `Schedule.jitter` because it prevents thundering-herd re-collision; compose it with `+` alongside other transformers before applying to the base schedule.
- `@catch`, `catchOf<E>`, `expected`, `exceptional`, `expectedOf<E>`, `exceptionalOf<E>` are all `Prelude` methods returning `CatchM<Error, M, A>`. `catchOf<E,M,A>(Func<E,K<M,A>>)` matches errors of subtype `E` (not an exact runtime-type check — subtypes satisfy the predicate); `exceptionalOf<E,M,A>` matches `Exceptional`-subtype errors (unexpected infrastructure failures); `expectedOf<E,M,A>` matches `Expected`-subtype errors (anticipated domain rejections). Multiple `|` handlers are processed in declaration order; the first predicate match wins. `@catch(int errorCode, ...)` compares against `Error.Code`. When an `Error` value carries code 0, the `@catch(Error error, ...)` overload (not the int overload) provides message-equality matching — the int overload carries no message to fall back to; domain errors should carry non-zero codes to make code-based matching unambiguous.
- The `|` catch operator on `K<M,A>` has two right-hand operand types: `CatchM<TError,M,A>` and `Finally<M,A>` (produced by `final<F,X>(K<F,X>)`). Chaining `computation | catchHandler | final(cleanup)` is legal because the result of `computation | catchHandler` is still `K<M,A>`, which then accepts the `Finally<M,A>` right operand — cleanup always runs regardless of whether the catch matched.
- `bracketIO<M,A>(K<M,A>)` (single-arg) creates a local resource environment scoped to the computation; resources acquired via `use(resource)` (Prelude static, `resource : IDisposable`) inside the computation are registered in the local scope and released when the scope exits regardless of success or failure. `bracketIO<M,A,B,C>(Acq: K<M,A>, Use: Func<A,IO<C>>, Fin: Func<A,IO<B>>)` (three-arg): the result type is `C` from the `Use` arm; the `Fin` arm returns `IO<B>` where `B` is a free type parameter whose value is discarded — `Fin` is not forced to return `IO<Unit>` but its return value does not contribute to the final result. `bracketIO<M,A,B,C>(Acq, Use, Catch: Func<Error, IO<C>>, Finally: Func<A,IO<B>>)` (four-arg) adds `Catch` for errors raised within the `Acq` computation specifically; errors in the `Use` arm propagate through `Finally` directly without invoking `Catch`. Resources acquired inside a `retry` loop are released on each failed attempt before the next retry fires — retry does not accumulate open resources across failed iterations.
- `retryWhile(Schedule, K<IO,A>, Func<Error,bool>)` and `retryUntil(Schedule, K<IO,A>, Func<Error,bool>)` combine schedule-based delay with a predicate guard. `repeatWhile(Schedule, K<IO,A>, Func<A,bool>)` is the success-value counterpart — use it for value-dependent re-execution, not `retryWhile`.
- Three distinct scope operators govern cancellation at entrypoint boundaries. `localIO<M,A>(K<M,A>)` creates a new `CancellationToken` derived from the parent context's token, so a regional `IO.cancel` inside the scope does not propagate to computations outside the scope. `uninterruptible<M,A>(K<M,A>)` ignores all cancellation requests by substituting `CancellationToken.None` for the duration. The entrypoint design choice: `localIO` for fan-out dispatch arms that should not cancel siblings, `uninterruptible` for cleanup arms inside `bracketIO` whose interruption would leave resources open. Both accept `K<M,A>` where `M : MonadUnliftIO<M>`.
- `timeout(TimeSpan, K<M,A>)` requires `M : MonadUnliftIO<M>` (not `MonadIO<M>`). It internally forks the computation, awaits the fork, and cancels it if the duration expires — delivering `Error.New(TimeoutException)` on expiry. For entrypoints that must bound execution time, `timeout` is the correct aspect form; a `CancellationTokenSource` deadline parameter on the entrypoint signature is the policy-smuggling anti-pattern — the timeout policy belongs to the effect layer, not to the entrypoint's parameter algebra. `Prelude.cancel` is a static `IO<Unit>` field that raises cancellation into the current context; when a `retry` loop wraps a computation containing `cancel`, the cancellation error propagates through `retry` without triggering another attempt.

[OVERLOAD_RESOLUTION_ORDERING]:
- `OverloadResolutionPriorityAttribute` accepts any `int`; the compiler picks the highest priority among candidates that pass type-checking within the same declaring type. Priority 0 is the implicit default. Applying priority 1 to a `params ReadOnlySpan<T>` overload and leaving the `params T[]` overload at 0 directs the compiler to the span path at all call sites that pass a collection expression or literals. Priority comparison is scoped to the declaring-type group — the feature spec groups candidates by declaring type and applies priority within each group; a derived-type `int[]` overload at -1 beats a base-type `ReadOnlySpan<int>` overload at default because derived-type betterness is resolved before cross-type priority comparison. A `[OverloadResolutionPriority(-1)]` deprecated overload on the base type is therefore not suppressed by a higher-priority overload on the derived type — each group is compared internally first.
- Two overloads differing only in the nullability of a reference-type parameter (`T` vs `T?`) are duplicate signatures (CS0111) and cannot coexist — `[OverloadResolutionPriority]` cannot arbitrate between them because the compiler rejects the second declaration. For value types (`T` vs `Nullable<T>`) the signatures are distinct and ordinary betterness selects the non-nullable overload for a non-null argument; no attribute is needed. When nullability-distinct dispatch is structurally required, use separate method names or a union input type.
