# Capability Provision as a Typed Environment

[PROVISION_VERSUS_STATE]:
- A capability has two orthogonal absence axes that must not collapse into one carrier. *Provision* is whether the effect can reach the capability at all — a property of the monad-over-runtime pairing, proved at compile time. *State* is whether a reachable capability is currently usable — a value, observed at runtime. A capability cell models the second; the environment trait models the first. A boundary that routes "capability not wired" and "capability degraded" through one `Option`/union erases the distinction the type system would otherwise enforce for free.
- The provision axis is carried by `Has<M, VALUE>`: a trait on the *effect monad* `M` with a single `static abstract K<M, VALUE> Ask` accessor. The effect monad — not the runtime data — declares `Has<M, Cap>` for each capability reachable through its environment; a step that reads `Cap` constrains its monad to `where M : Has<M, Cap>` and calls `M.Ask` through static-virtual dispatch. A monad whose runtime lacks the capability never satisfies the constraint, so the gap is a build error at the composition root, never a runtime miss. This is the absence that needs no sentinel because the absent value cannot be constructed.
- `Has<in M, VALUE>` is contravariant in `M`: a monad over a richer runtime substitutes wherever a less-capable monad is demanded, so one concrete effect monad flows into every step whose capability subset it covers without a per-step adapter. The accessor returns `K<M, VALUE>` — the capability arrives already lifted into the effect, so the read composes with `Bind` and never escapes the rail as a bare handle. Stacking several `Has<M, ·>` constraints on one step is the declarative capability manifest: the step's signature lists exactly the capabilities it touches, and the resolver proves them all at once.

[DEFERRED_ACCESSOR_VALUE]:
- The reach into the environment is itself a first-class value, not a method call: `Ask<Env, A>(Func<Env, A> F)` captures a pure projection from environment to bound value and stays unapplied until a monad binds it. One captured accessor binds polymorphically into `Reader<Env>`, `ReaderT<Env, M>`, `RWST<Env, W, S, M>`, and any `Readable<M, Env>` via `ToReadable<M>`, `ToReader`, `ToReaderT<M>`, `ToRWST<W, S, M>` — the same projection serves every monad stack that carries the environment, selected at the bind site by the target rail, with no overload per stack.
- The accessor short-circuits the boundary's null exposure: a foreign capability read through `Ask(rt => rt.Service.MaybeHandle)` returns the raw nullable, but `Ask(rt => Optional(rt.Service.MaybeHandle))` projects absence to `Option` inside the deferred projection, so the optional is produced once at the only site that touches the foreign field and the interior binds an `Option`, never the nullable. The projection site and the environment-read site are fused into one value.
- Because the accessor is data, fallback across providers is the accessor semigroup, not a branch: a sequence of capability projections chained by first-success collapses a multi-provider lookup into one deferred value that resolves the environment once and yields the first reachable capability, with the choice made by the carrier's `Choice`/`Choose` rather than a hand-written ladder over `rt`.

[RUNTIME_AS_THE_REQUIRED_RAIL]:
- The canonical capability-bearing effect is `record Eff<RT, A>(ReaderT<RT, IO, A> effect)` — a reader of the runtime stacked on `IO`. The runtime is not an injected service object passed by value; it is the read-only environment threaded by the reader transformer, so every capability access is `runtime<RT>()` (the identity projection making the whole runtime the bound value) followed by a projection, or a direct `Has` accessor bound into the rail.
- The runtime-free effect and the runtime-bearing effect are one family with a widening arrow, not two types. A capability-agnostic step authored as the runtime-free effect lifts into any runtime-bearing pipeline by the widening that re-tags it with `RT`; the disjunction and sequencing operators apply this widening implicitly, so a runtime-free fallback or continuation composes against a runtime-bearing primary with no manual lift at the call site. Absence of a capability requirement is the most general position and flows into every more-specific runtime for free.

[ENVIRONMENT_NARROWING_AS_CAPABILITY_SCOPING]:
- Capability scoping is environment narrowing, expressed as one combinator, not a second runtime type per sub-capability. `local<OuterRT, InnerRT, A>(Func<OuterRT, InnerRT> f, innerEffect)` runs an inner-runtime effect by projecting the outer runtime down to exactly the sub-capability the inner step needs, then discards the narrowed view at scope exit. An effect written against the minimal runtime it requires is the deep, reusable surface; the outer composition supplies the projection. This is the structural collapse of "pass only the handle this function needs" into a typed environment map.
- The same-runtime `local(Func<Env, Env> f, ma)` runs `ma` under a transformed environment and is total over the transform: it cannot fail, has no restore obligation on the caller, and the outer environment is unaffected after the scope because the reader re-supplies the original on the next read. A boundary that must run a sub-computation under a swapped capability — a stubbed clock, a redirected sink, a throttled transport — expresses it as one `local` with the swap function, never as mutable set-then-reset around the call.
- Narrowing and widening are the symmetric pair that keeps capability requirements honest: widen a runtime-free step up into a richer runtime when composing; narrow a rich runtime down to a minimal sub-runtime when delegating. Neither allocates a new effect family; both are environment maps over the one reader. The minimal-runtime authoring discipline plus these two arrows replaces every bespoke "context object" the interior would otherwise thread by hand.

[ABSENCE_PLACEMENT_ACROSS_THE_PROVISION_BOUNDARY]:
- Provision-time absence is unrepresentable, so a capability that *might* be wired is a category error: either the runtime `Has` it (and the read is total) or the runtime does not (and the code does not compile). The optional must therefore sit on the *state* axis inside the provided capability, never on the provision axis. The correct shape is `Has<M, Cap>` where `Cap` is itself a closed state family or an `Atom`-gated cell — provision is proven, state is observed — so the boundary never asks "is the service present?" at runtime; it asks only "is the present service usable now?".
- A capability whose presence is genuinely dynamic — a host feature that may or may not exist at startup — is provided as `Has<M, Option<Feature>>` or `Has<M, FeatureState>` (the closed family), not as an `M` that conditionally satisfies `Has`. The constraint stays satisfiable for every runtime; the absence rides the bound value where the consumer's total dispatch handles it. Making `Has` conditional would push the absence back to the unrepresentable provision axis where it cannot be inspected.
- The environment read is the single authorized seam for foreign-sentinel projection because it is the only place the runtime's raw services are touched. Projecting the sentinel anywhere downstream means the raw runtime field leaked past the reader; the discipline is that every `Ask`/`asks` projection that reaches a nullable, an invalid handle, or a magic constant performs the `Option` projection inside its own function body, so the bound value the rail receives is already domain-shaped and no interior step re-touches the runtime to re-derive absence.

[CAPABILITY_READ_FUSED_WITH_RAIL]:
- A capability read, its sentinel projection, and its absence-to-failure conversion fuse into one bound step rather than three sequenced ones. The accessor reads the runtime, projects the foreign field to `Option`, and converts a hard-absence to a coded failure in a single deferred value bound into the effect:

```csharp
static Eff<RT, Handle> handle<RT>() =>
    from cap in Readable.asks<Eff<RT>, RT, Option<Handle>>(
                    rt => Optional(rt.RawHandle))
    from h   in cap.ToEff(Error.New(code, "capability unusable"))
    select h;
```

- The same body discriminates provision-state kinds when the capability is a closed family instead of an `Option`: the `asks` read yields the state value, the total dispatch over its cases routes `Available` to the live handle, `Degraded` to a usable-but-flagged value carrying its evidence, and `Unavailable` to a terminal coded failure — all on the effect rail, with the runtime touched exactly once. The consumer never sees the runtime, never re-reads it, and cannot reach a raw handle, because the read and its classification are a single inlined accessor.
