# Wire Discriminant Disambiguation

[TWO_TREES_ONE_OWNER]:
- The disambiguator is built twice from one owner, once per direction: `getSearchTree(ast.types, isDecoding)` is called with `isDecoding = true` on the decode path and `false` on the encode path, and the boolean is not a filter over one shared tree — it changes *which literal each member contributes*, so a union materializes two structurally distinct bucket maps from the same `ast.types` array, the decode tree keyed on each arm's encoded-side literal and the encode tree on its type-side literal.
- Direction selects the literal source inside `getLiterals` per property: a `TypeLiteral` member reads `isDecoding ? AST.encodedAST(propertySignature.type) : AST.typeAST(propertySignature.type)`, so the discriminant the decoder partitions on is the *wire* literal and the discriminant the encoder partitions on is the *domain* literal — for a plain `Schema.tag` arm both sides coincide, but the two trees are computed independently and a divergence between encoded and type literal silently produces two different partitions.
- The owner-level consequence: the decode disambiguator and the encode disambiguator are derived projections of one `Schema.Union`, never two hand-authored routing tables — adding an arm widens `ast.types` once and both trees recompute, so the wire-in and wire-out dispatch stay in lockstep off a single declaration, the multiplicative collapse a per-direction `switch (wire.kind)` plus a separate `switch (domain.kind)` forfeits.

[TRANSFORM_ARM_SPLITS_THE_DISCRIMINANT_ACROSS_DIRECTIONS]:
- A transformation-bearing arm is where the two trees diverge by construction: `getLiterals` on a `Transformation` reads `getLiterals(isDecoding ? ast.from : ast.to, isDecoding)`, so the decode tree partitions on the transform's *source* literal and the encode tree on its *target* literal — a `transformLiteral<Type, Encoded>` is `transform<Literal<[Encoded]>, Literal<[Type]>>`, its `from` the wire code and its `to` the domain name, so a union of such arms decodes by wire code and encodes by domain name, two disjoint literal sets keying the one owner's two directions.
- The recursion is direction-uniform but the input is not: the second argument threads the *same* `isDecoding` into the recursive `getLiterals(ast.from, isDecoding)`, so a transform whose `from` is itself a tagged struct contributes the wire-side tag on decode and the domain-side tag on encode all the way down — the direction is chosen once at the union and propagated, never re-decided per nesting level.
- The hazard this exposes: a member whose encoded and decoded discriminant *names* differ (a renamed wire field, a remapped literal) builds a decode bucket under one key and an encode bucket under another, so a round-trip is disambiguated by two different namespaces — correct by construction, but a hand-written decoder keying on the domain name would mis-route every wire input, the exact re-derivation the two-tree owner deletes by reading each direction's literal from the AST rather than assuming the field name is direction-stable.

```typescript
import { Schema } from 'effect'

const A = Schema.transformLiterals([0, 'alpha'], [1, 'beta'])
class B extends Schema.TaggedClass<B>()('gamma', { weight: Schema.Int }) {}
const Owner = Schema.Union(A, B)
const fromWire = Schema.decodeUnknownSync(Owner)
const toWire = Schema.encodeSync(Owner)
const decoded: Schema.Schema.Type<typeof Owner> = fromWire(0)
const reencoded: Schema.Schema.Encoded<typeof Owner> = toWire(decoded)
```

- `fromWire(0)` partitions on the decode tree whose `A` bucket is keyed `"0"` (the transform `from` literal), while `toWire` partitions on the encode tree whose `A` bucket is keyed `"alpha"` (the `to` literal) — one `Schema.Union` carries both partitions, the wire-code namespace and the domain-name namespace, derived per direction with zero second routing table; the loose spelling is a `decode` switch on the numeric code beside an `encode` switch on the string name, two parallel tables the next code/name pair must edit twice.

[STRING_COERCION_COLLAPSES_THE_LITERAL_TYPE_AXIS]:
- The bucket index is a string under both construction and lookup: `getSearchTree` hashes each literal as `String(literal.literal)` and the decoder reads `String(input[name])`, so the bucket map is a `Record<string, ...>` whose keys have lost the literal's original type — a numeric discriminant `0`, a string discriminant `"0"`, and a bigint discriminant `0n` all hash to the one key `"0"` and share a single bucket, the disambiguator collapsing the three literal-type axes the discriminant-admission gate kept distinct at the type level.
- The collision is a true bucket merge, not a type-level coincidence: a numeric discriminant is spellable only off the `AST.LiteralValue`-accepting surfaces (`Schema.tag`, `Schema.TaggedStruct`, a bare `Schema.Literal` field) since `Schema.TaggedClass` constrains `Tag extends string` — and two arms `kind: Schema.Literal(0)` and `kind: Schema.Literal("0")` joined in one union build *one* bucket under `"0"` holding both members, so inputs `{ kind: 0 }` and `{ kind: "0" }` both resolve to the same two-candidate list and are tried structurally in declaration order, the numeric arm and the string arm no longer disambiguated by the discriminant.
- The boundary: the collision is invisible at the type level (the `kind` union `0 | "0"` is a legal two-member alphabet) and surfaces only as a silent loss of the single-bucket guarantee at runtime — the law is keep a discriminant family on *one* literal-value type, never mix numeric and string tags in the same key namespace, because the `String` coercion is the disambiguator's index function and a heterogeneous-typed tag set is a hash collision the type system cannot see; that `TaggedClass` is string-only and only `TaggedStruct`/`tag` admit a numeric tag is the surface-level guard against authoring the collision through the canonical class owner.

```typescript
import { Schema } from 'effect'

const Numeric = Schema.Struct({ kind: Schema.Literal(0), reading: Schema.Int })
const Stringy = Schema.Struct({ kind: Schema.Literal('0'), label: Schema.String })
const Collided = Schema.Union(Numeric, Stringy)
const decode = Schema.decodeUnknownSync(Collided)
const ambiguous: Schema.Schema.Type<typeof Collided> = decode({ kind: 0, reading: 1 })
```

- Both arms hash to bucket `"0"`, so `decode({ kind: 0, reading: 1 })` does not short-circuit to `Numeric` — it tries `Numeric` then `Stringy` structurally, and an input ambiguous on the non-discriminant fields could resolve to the wrong arm; the gate that keeps the bucket a singleton is a homogeneous literal-value type across the discriminant namespace, the `String`-coercion index making `Schema.Literal(0)` and `Schema.Literal("0")` indistinguishable to the disambiguator that distinguished them in the `kind` type.

[CANDIDATE_CONCATENATION_IS_THE_DISAMBIGUATOR_ARITHMETIC]:
- The candidate set is accumulated, not selected: the decode loop walks `Reflect.ownKeys(searchTree.keys)` — every property namespace any member contributed a literal to — and for each namespace the input populates, `candidates = candidates.concat(buckets[literal])`, so a member appears in the candidate list once *per namespace whose bucket it sits in*. The disambiguator narrows by intersection of *presence* (only namespaces the input populates contribute) but trials by *union* of buckets (every matched bucket's members concatenate), so the trial count is the sum of matched buckets, never their intersection — the one structural asymmetry between how the search tree narrows and how it dispatches.
- Duplicate candidacy is real and order-load-bearing: concatenation does not dedupe, so a member living in two matched buckets is enqueued twice and tried twice, the first structural success winning — the disambiguator's cost is `Σ |bucket(namespace_i)|` over every namespace the input satisfies, minimized to one only when a single namespace owns a singleton bucket, and a multi-key family pays the sum even when one key would have sufficed.
- A missing or mistyped namespace value does not abort the accumulation — it forks into an error candidate while the loop continues: a namespace the input lacks pushes a `Missing` issue, a namespace whose `String`-coerced value hits no bucket pushes a located `Type` issue against that namespace's `literals` union, and either way the loop walks the *remaining* namespaces, so an input populating one valid discriminant still accumulates that bucket's candidates alongside the recorded issues from the others — the disambiguator races the valid bucket's trial against the partial-key failures, and a non-object input short-circuits before any bucket (`Predicate.isRecordOrArray(input)` false) straight to the candidate-set `Type` error.

[OVERLAPPING_FIRST_LITERAL_FORCES_A_SECOND_NAMESPACE]:
- The advance-on-collision is a `continue`-versus-terminal-push fork the build loop resolves per literal, not a give-up: registering a member at a fresh bucket `break`s, but a literal whose bucket already exists triggers `if (j < tags.length - 1) continue` — fall through to the member's *next* literal property — and only at the member's *last* literal does it `buckets[hash].push(member)` into the colliding bucket; so the member contributes its literal to *every* namespace it scans through until one is fresh, registering in the first private namespace or, with no private literal, sharing the terminal collided bucket.
- That fork is what populates multiple namespaces from one member, the source of the concatenation arithmetic: a member sharing its first literal but private on its second sits in *both* the shared first-literal bucket and its own second-literal bucket, so an input matching both prefixes accumulates two buckets — and the family discriminated on `kind: "shape"` but differing on `form: "circle" | "square"` partitions cleanly on `form` (a private bucket) while `kind` still holds both, the two namespaces the concatenation sums over.
- The terminal-push degenerate case is the silent regression: when every literal of a member collides with an already-registered bucket, the member is pushed into its last bucket beside its sibling and the family decodes by structural trial of that shared bucket's whole list — so a discriminated family stays single-bucket only when *some* literal property is unique per member, the disambiguator finding it by left-to-right scan, and the fix for a literal-identical pair is one unique literal per member, never a hand-picked discriminant field the scan already locates.

```typescript
import { Schema } from 'effect'

const Circle = Schema.Struct({ kind: Schema.Literal('shape'), form: Schema.Literal('circle'), radius: Schema.Positive })
const Square = Schema.Struct({ kind: Schema.Literal('shape'), form: Schema.Literal('square'), side: Schema.Positive })
const Shape = Schema.Union(Circle, Square)
const decode = Schema.decodeUnknownSync(Shape)
const measured = decode({ kind: 'shape', form: 'circle', radius: 2 })
```

- `kind: "shape"` collides on both members so its bucket holds both, but the builder advances each member to `form` where `"circle"` and `"square"` are private singletons — the disambiguator keys on `form`, and `decode({ kind: 'shape', form: 'circle', ... })` populates both the `kind` bucket (two candidates) and the `form` bucket (one candidate), concatenating to a three-trial list the `form` singleton would have made one; collapsing the shared `kind` to a single category arm extended across the variant union removes the redundant namespace, so only `form` partitions and the trial is a singleton.

[OTHERWISE_TAXES_THE_DISCRIMINATED_BUCKETS]:
- The undiscriminated arm is appended *once to the accumulated candidate list*, not per bucket: after the whole namespace loop completes, `if (searchTree.otherwise.length > 0) candidates = candidates.concat(searchTree.otherwise)` runs a single time against the already-accumulated candidates — so an input that hit a singleton discriminated bucket still has the full `otherwise` set concatenated onto its trial list, and a union mixing tagged arms with one bare `Schema.Struct` arm pays `bucket_hits + otherwise_count` trials per decode, the discriminated short-circuit defeated by the trailing structural arm the disambiguator cannot pre-exclude on the discriminant alone.
- The structural otherwise arm is tried by full TypeLiteral parse, so its cost is its whole field set, not a tag check: the bare-struct candidate runs the complete property loop the discriminated arms skipped, and because concatenation preserves declaration order the discriminated candidate is tried *first* — a correctly-tagged input matches its singleton bucket before the `otherwise` arm runs, so the otherwise tax is paid in trial *position*, not always in full parse, but a mistagged input falls through every discriminated bucket *and* the structural arm before failing.
- The disambiguator never promotes an otherwise member to a bucket: an arm with no required single-`Literal` property is `otherwise` for the life of the union, so the only way to remove the tax is to give that arm a discriminant — an `attachPropertySignature`-injected tag moves the bare struct out of `otherwise` and into a singleton bucket, restoring the one-trial decode without the wire carrying the tag, the disambiguator keying on the injected literal exactly as it keys on a native one.

[ONEXCESSPROPERTY_INTERACTS_WITH_ARM_SELECTION_NOT_ONLY_FIELDS]:
- `onExcessProperty: "error"` is evaluated *per candidate*, inside each TypeLiteral parse the union loop dispatches to, so it is an arm-selection force, not only a field-shape check: a candidate the disambiguator admitted fails not because its discriminant is wrong but because the *input carries a key that candidate's signature does not declare*, and under `"error"` that excess key produces an `Unexpected` issue on that candidate — so the same input that decodes cleanly under the default `"ignore"` can fail every admitted candidate under `"error"`, the union reporting failure though a structurally-correct arm exists.
- The interaction is sharpest on the otherwise path and across overlapping buckets: when an input matches a wide bucket holding several candidates, `"error"` runs each candidate's excess-property check against that candidate's *own* field set, so a key belonging to sibling arm `B` but present on an input meant for arm `A` makes `A` fail as excess — the disambiguator admitted both `A` and `B` to the candidate list, and `"error"` turns the field-set difference between them into a per-candidate rejection, the arm selection now gated by field exactness rather than discriminant alone.
- The default `"ignore"` is what lets the discriminant alone select the arm: under `"ignore"` the excess-property loop is skipped entirely (`onExcessPropertyError || onExcessPropertyPreserve` guards the whole block), so a discriminated arm matches on its tag and tolerates foreign keys, the disambiguator's bucket lookup the sole selector — the rule is that `"error"` couples arm selection to total field exactness across every admitted candidate, so a union decoded under `"error"` must have *non-overlapping field sets* per shared bucket or a legitimately-tagged input is rejected by a sibling's missing key, while `"ignore"` keeps selection on the discriminant axis the search tree owns.

```typescript
import { Schema } from 'effect'

class Open extends Schema.TaggedClass<Open>()('Open', { at: Schema.Int }) {}
class Wait extends Schema.TaggedClass<Wait>()('Wait', { until: Schema.Int }) {}
const Signal = Schema.Union(Open, Wait)
const strict = Schema.decodeUnknownSync(Signal, { onExcessProperty: 'error' })
const lenient = Schema.decodeUnknownSync(Signal)
const accepted = lenient({ _tag: 'Open', at: 1, until: 9 })
const rejected = strict({ _tag: 'Open', at: 1, until: 9 })
```

- `lenient` keys on `_tag: "Open"`, hits the `Open` singleton bucket, ignores the foreign `until`, and decodes; `strict` admits the `Open` candidate the same way but its per-candidate excess check finds `until` undeclared on `Open` and fails with `Unexpected` — the discriminant selected the arm, `onExcessProperty: "error"` then rejected it on the field set, so the two options disagree on the *same* admitted candidate; a union meant to round-trip under `"error"` keeps each arm's field set disjoint or routes the lenient policy at the boundary so arm selection stays a pure discriminant lookup.

[DECODE_TREE_IS_REUSED_AS_THE_EQUALITY_DISPATCHER]:
- The disambiguator is not decode-private: `Schema.equivalence` over a union calls `ParseResult.getSearchTree(ast.types, true)` with the decode direction hardcoded, so derived structural equality dispatches on the *encoded-side* discriminant and the identical `String(a[name])` coercion and `candidates.concat(buckets[literal])` arithmetic — the `Equivalence<A>` reads each operand's tag, hashes it to a bucket, and compares only the admitted candidates, so the search tree is one disambiguator serving decode and equality from a single build.
- The collisions therefore travel into equality: a numeric/string tag collision that merges two buckets at decode merges them at equality too — two operands hashing to the shared `"0"` bucket are compared structurally across both candidates, so the same heterogeneous-tag hazard that degrades decode degrades the derived equivalence, and the otherwise tax appends every undiscriminated arm to the equality candidate list exactly as it does the decode list.
- The owner-level fusion: decode disambiguation, encode disambiguation, and structural equality are three reads of one search-tree algorithm over one `ast.types`, so a discriminant decision pays down or regresses all three at once — a clean single-literal-type tag namespace keeps decode one-trial, equality one-candidate, and the encode tree singleton, while a mixed-type or otherwise-heavy family regresses every projection together, the disambiguator the single point where the wire-discriminant shape governs the whole derived family.

[SUSPEND_FORCES_THE_THUNK_PER_TREE_BUILD]:
- A recursive arm's discriminant is recovered by forcing the suspension: `getLiterals` on a `Suspend` node returns `getLiterals(ast.f(), isDecoding)`, invoking the lazy thunk to read the deferred member's literals, so a self-referential variant still partitions into buckets — the disambiguator reaches through the `suspend` cut point the recursive owner needed to close, and a recursive tagged family decodes by single bucket lookup at every level, never an N-arm scan per recursion frame.
- The forcing is per direction and per tree: the decode tree and the encode tree each call `ast.f()` independently because each is a separate `getSearchTree` invocation, and `typeAST`/`encodedAST` re-thread `new Suspend(() => ...)` so the direction-stripped AST is itself lazy — the thunk is evaluated when the disambiguator reads it, so a recursive union pays the thunk force once per built tree, not once per decode, the search tree memoized over the AST node while the per-input dispatch is allocation-free.
- The boundary the forcing exposes: a `suspend` thunk whose return is un-annotated infers `any`, and `getLiterals` reading an `any`-typed member finds no `Literal` AST node, so the recursive arm silently falls to `otherwise` and the recursive family loses its single-bucket decode — the discriminant survives recursion only when the `suspend` return is annotated to the real schema, the annotation the load-bearing bit that keeps the forced thunk's AST a partitionable `TypeLiteral` rather than an opaque `any` the disambiguator drops.
