# Wire-Contracts: Edge

[JSONNODE_OWNERSHIP_AND_PASS_THROUGH_SEAM_DISCIPLINE]:
- `JsonNode` enforces single-parent ownership: `JsonObject.Insert`, `SetAt`, and `ReplaceWith` all throw `InvalidOperationException` when the supplied node already has a parent. Pass-through envelope patching that takes a node from one position and inserts it at another must call `DeepClone()` on the source node first. `JsonNode.DeepClone` performs a full recursive copy; its cost is proportional to subtree size, so pre-cloning must be gated on whether the node already has a parent via `node.Parent != null`.
- `JsonNode.GetPath()` returns the JSON Path of the current node within its document tree — e.g., `$.envelope.payload[2]`. This is the diagnostic anchor for admission modifiers that log schema-contract violations: `JsonTypeInfo.Properties[i]` does not give runtime document position, but `GetPath()` on a `JsonNode` representation does. `JsonNode`-based parsing is the correct tooling for contract-violation receipts that require path evidence; `JsonDocument`/`JsonElement` has no equivalent and requires manual property-sequence tracking.
- `JsonObject.IndexOf(string propertyName)` returns the zero-based index of a named property, or `-1` if not present. Combining `IndexOf` with `SetAt` or `RemoveAt` achieves named-property positional mutation without iterating the full property list. For a pass-through seam that must upgrade a `version` field before forwarding: `var idx = obj.IndexOf("version"); if (idx >= 0) obj.SetAt(idx, "version", newNode); else obj.Insert(0, "version", newNode)`. This is the correct idempotent replace; calling `SetAt(0, "version", newNode)` when `IndexOf("version")` returns a non-zero index throws `ArgumentException`.

[JSONDOCUMENT_ELEMENT_LIFETIME_AND_ZERO_COPY_BOUNDARY]:
- `JsonElement.Clone()` produces an independent copy that survives `JsonDocument` disposal. Every `JsonElement` reference that must cross an `await` boundary, be stored in a receipt, or be held beyond the `using` scope of its owning `JsonDocument` must be cloned first. The `ObjectDisposedException` thrown by all `JsonElement` operations after document disposal manifests as a runtime fault in async pipelines where the `using` closes before the continuation executes. The clone allocates a new backing buffer; at wire seams with high message rates, pool-backed `JsonDocument.ParseAsync` with per-call `Dispose` is correct only when all element work completes synchronously within the same async step.
- `JsonElement.DeepEquals(a, b)` is a static structural comparison with no allocation, operating on two `JsonElement` values regardless of whether they share a backing document. Snapshot envelope checksum verification uses `DeepEquals` before falling back to hash comparison: `DeepEquals` handles structural JSON equivalence — key ordering in objects is not significant — while hash comparison catches byte-level identity. The two are complementary: `DeepEquals` is the canonical equality check; hash comparison is the performance path when the payload is untouched.
- `JsonMarshal.GetRawUtf8Value(JsonElement)` and `JsonElement.Clone` interact asymmetrically: `Clone` allocates a heap copy, making the clone safe beyond document disposal; `GetRawUtf8Value` returns a span into the original buffer, valid only while the document is live. A `Clone` of the element does not extend the validity of the span because the clone owns its own buffer. Any code path that calls `GetRawUtf8Value` must be statically scoped within the same `using` block as its `JsonDocument`; constructing `GetRawUtf8Value(clonedElement)` reads the clone's buffer, not the original's, and if the clone is constructed from an already-disposed document, the clone constructor itself throws.

[WRITER_DEPTH_ASYMMETRY_AND_SEAM_CONTRACT]:
- `JsonDocumentOptions.MaxDepth` defaults to 64; `JsonWriterOptions.MaxDepth` defaults to 1000. The asymmetry means a document tree written without depth limit fails to parse if it exceeds 64 levels. At a wire seam that both produces and consumes messages, the `MaxDepth` on options at the admission side must match the `MaxDepth` used on the write side, or output that passes the writer will fail its own admission round-trip. Set an explicit non-zero `MaxDepth` on both `JsonWriterOptions` and `JsonDocumentOptions`/`JsonSerializerOptions`, choosing a value strict enough to bound adversarial nesting but high enough to accommodate the maximum legitimate document depth. The protocol document must specify this value as a contract constraint.
- `JsonWriterOptions.SkipValidation = true` disables all structural JSON validation in `Utf8JsonWriter`: mismatched start/end tokens, duplicate property names, and property names written outside object context are silently emitted. At a pass-through seam that constructs a writer solely to re-emit an already-validated `ReadOnlySequence<byte>` via `WriteRawValue(sequence, skipInputValidation: true)`, setting `SkipValidation = true` eliminates the structural-state check overhead. `SkipValidation` on the writer does not affect `WriteRawValue`'s `skipInputValidation` parameter — the writer option governs the state machine; the parameter governs RFC 8259 content validation of the raw bytes.

[SERIALIZATION_CALLBACK_KIND_CONSTRAINT_AND_FAILURE_SURFACE]:
- `JsonTypeInfo.OnDeserializing`, `OnDeserialized`, `OnSerializing`, and `OnSerialized` (the modifier-assigned delegate properties) all throw `InvalidOperationException` when assigned on a `JsonTypeInfo` whose `Kind` is not `JsonTypeInfoKind.Object`. Modifiers that set these callbacks must guard `typeInfo.Kind == JsonTypeInfoKind.Object` before assignment; a modifier that unconditionally assigns will throw on `JsonTypeInfoKind.Enumerable` or `JsonTypeInfoKind.None` types. Because `MakeReadOnly` is typically called after the modifier chain runs, the exception manifests at the first serialization call rather than at construction time, making the failure site distant from the cause.
- `IJsonOnSerializing.OnSerializing()` fires on the specific instance with access to `this`; the modifier-delegate `JsonTypeInfo.OnSerializing` fires on the type-level metadata with no instance reference. Instance-level pre-conditions — required fields populated, version discriminant set — belong to `IJsonOnSerializing`; type-level policy injection — cross-cutting redaction, protocol framing — belongs to the modifier delegate. Neither can throw a typed error that escapes the serializer as a domain error; any exception becomes a `JsonException` wrapper. Pre-validation before serialization is the only mechanism for surfacing typed failures from outbound invariant checks.

[SCHEDULE_PIPE_SEMANTICS_AND_CUMULATIVE_BOUND]:
- `Schedule.maxCumulativeDelay(duration)` terminates the schedule when the sum of all emitted delays reaches the bound, regardless of the per-step delay. This differs from `maxDelay`, which caps individual delays. At a wire retry seam, `Schedule.recurs(n) | Schedule.exponential(initialDelay) | Schedule.maxDelay(ceiling) + Schedule.maxCumulativeDelay(budget)` expresses a schedule bounded both per-step and in aggregate. The `|` operator picks the minimum delay from two schedules at each step; `+` concatenates two schedules in sequence, running the second after the first is exhausted. The `|` operator selects minimum delay, not minimum count — this is critical for retry composability where `Schedule.recurs(n)` uses `|` to cap attempts while `Schedule.exponential` uses `|` to cap delay.
- `Schedule.decorrelate(factor)` transforms a schedule by de-correlating durations: given a base duration, it adds and removes jitter proportionally so that parallel clients running the same schedule produce non-synchronized retry waves. A linear schedule `(100, 200, 300...)` becomes approximately `(103.2, 97.1, 202.3, 197.3...)` — the schedule runs approximately twice as long because de-correlation doubles the number of steps. At a wire seam with high client concurrency, `Schedule.linear(initialDelay) | Schedule.decorrelate() | Schedule.recurs(n)` prevents thundering-herd re-synchronization without requiring per-client seed management; the de-correlation is seeded internally per schedule instance. Using `jitter(factor)` instead only randomizes amplitude; `decorrelate` specifically addresses phase coherence.

[IO_LOCAL_SCOPE_AND_FORK_RESOURCE_OWNERSHIP]:
- `IO<A>.Local` creates a local cancellation environment — a child-linked `CancellationToken` scope. Cancellation in this scope fires the child token; the parent scope's token is unaffected. `localComputation.Local.Timeout(duration)` produces a computation that times out independently without cancelling the enclosing session. `.Timeout(TimeSpan)` wraps the computation in an exception-throwing timeout that integrates with the surrounding `liftIO` + `.Catch(IsExceptional, ...)` boundary translation pattern.
- `IO<A>.Fork(Option<TimeSpan>)` returns `IO<FiberIO<A>>`. Resources acquired in the parent are shared and must be treated with care. A wire seam that needs to dispatch multiple outbound requests concurrently must construct each forked computation with its own `bracketIO`-scoped connection, not share a connection acquired before `Fork`. `FiberIO<A>.Cancel` is the composable cancellation point for the forked fiber, independent of the parent's `CancellationToken`. Any error inside a fork that is not awaited via `FiberIO<A>.Await` is silently dropped — an unawaited fork that fails produces no error signal in the parent chain.

[CANCELLATIONTOKENSOURCE_TIMEPROVIDER_AND_ASYNC_CANCEL_CONTRACT]:
- `CancellationTokenSource(TimeSpan delay, TimeProvider timeProvider)` takes both the delay and the `TimeProvider`, making the timeout clock injectable. `CancellationTokenSource(TimeSpan)` without a `TimeProvider` uses the system wall clock and cannot be fast-forwarded. Every per-call timeout `CancellationTokenSource` at a wire seam must use the injected `TimeProvider` overload.
- `CancellationTokenSource.CancelAsync()` returns a `Task` that completes after all registered callbacks and continuations have finished. `Cancel()` throws an `AggregateException` on the calling thread if any registered callback throws; `CancelAsync()` does not. For seam-level cancellation that cascades into `IO`-registered cleanup, `CancelAsync()` is always the correct choice.

[LOCK_SCOPE_AWAIT_PROHIBITION_AND_DOUBLE_CHECK_PATTERN]:
- `System.Threading.Lock.EnterScope()` returns `Lock.Scope`, a `ref struct` that releases on `Dispose`. Because `ref struct` values cannot be stored on the heap or cross `await` boundaries, the compiler prevents `await` inside a `using (var _ = lock.EnterScope())` block — any `await` attempt produces a compiler error. Code that calls `EnterScope()` and manually calls `scope.Dispose()` without `using` can cross `await` boundaries and will cause `InvalidOperationException` on `Dispose` if the thread changed. The safe pattern for lazy-sealed `JsonSerializerOptions` initialization: `if (!_options.IsReadOnly) { using var _ = _lock.EnterScope(); if (!_options.IsReadOnly) { /* mutate */ _options.MakeReadOnly(); } }` — no `await` inside the lock scope.
- `Lock.IsHeldByCurrentThread` returns `true` only when the current thread holds the lock, enabling nested call validation: a factory method that must be called inside a lock can assert `Debug.Assert(_lock.IsHeldByCurrentThread)`, turning a logic error into an immediate assertion failure rather than a data-race corruption. `Lock` is a non-recursive mutex — recursive entry at wire-initialization seams must be restructured to extract shared logic rather than re-entering.

[JSONOBJECT_SETONCE_INVARIANT_AND_ENVELOPE_PATCHING]:
- `JsonObject.SetAt(int index, string propertyName, JsonNode? value)` throws `ArgumentException` if `propertyName` is already present at a different index, enforcing key uniqueness. Before calling `SetAt(0, "version", newVersionNode)`, verify that `IndexOf("version") == 0` or `-1`. If `IndexOf` returns a non-zero positive index, the correct idempotent replace is: `var idx = obj.IndexOf("version"); if (idx >= 0) obj.SetAt(idx, "version", newNode); else obj.Insert(0, "version", newNode)`.
- `JsonObject.SetAt(int index, JsonNode? value)` — the two-parameter overload — replaces only the value at the given index, preserving the existing property name. This is the mutation path for in-place value upgrade — changing a version integer, replacing a discriminant string — where the key must not change. The three-parameter form can rename and re-value simultaneously; the two-parameter form changes only the value. Using the three-parameter form when only value-replacement is intended introduces accidental key changes that become wire-breaking contract mutations.

[SCHEDULE_UPTO_DATETIME_LOSS_AND_RECEIPT_DISCIPLINE]:
- `Schedule.upto(Duration duration, Func<DateTime> currentTimeFn)` uses `DateTime` internally, discarding `DateTimeOffset` offset information. Receipts emitted inside a schedule-governed retry loop must independently capture `timeProvider.GetUtcNow()` — a `DateTimeOffset` with zero offset — as the `Timestamp` field. Using the `DateTime` argument of `currentTimeFn` as the timestamp source loses UTC-offset evidence and produces a receipt whose time provenance is ambiguous on systems where local time is not UTC. The audit trail requirement makes `DateTimeOffset` mandatory on all receipt timestamp fields regardless of the schedule's internal clock type.

[DIAGNOSTICCONTEXT_PROPAGATION_CARRIER_CONTRACT]:
- `DistributedContextPropagator.Inject(Activity, object carrier, PropagatorSetterCallback setter)` and `ExtractTraceIdAndState(object carrier, PropagatorGetterCallback getter, out string traceId, out string traceState)` use `object` for the carrier — the carrier is any header container and the callbacks perform the actual key-value extraction. The `PropagatorGetterCallback` receives both a `fieldValue` (single value, non-null when exactly one value exists) and `fieldValues` (collection, non-null when multiple values exist). A wire seam's inbound carrier adapter must handle both output parameters: some transport stacks emit multi-value headers such as repeated `baggage` lines, for which `fieldValue` is null and `fieldValues` carries all values. Ignoring `fieldValues` silently drops baggage entries when the transport repeats headers.
- `DistributedContextPropagator.Current` is a static writable property — setting it is a process-wide mutation. Any concurrent serialization in progress when `Current` is replaced will use the old propagator for in-flight calls. Set `Current` exactly once during application startup before any wire calls are made. Per-request propagator selection requires constructing a local propagator and calling `Inject`/`ExtractTraceIdAndState` directly on it, bypassing the `Current` singleton.

[JSONSERIALIZEROPTIONS_ISREADONLY_SEAL_TIMING]:
- `JsonSerializerOptions.IsReadOnly` becomes `true` after `MakeReadOnly()` is called or after the first serialization call on that instance. The first-use auto-seal means a shared `static` `JsonSerializerOptions` is implicitly sealed on first use even without an explicit `MakeReadOnly()` call. A modifier added via `TypeInfoResolverChain.Insert(0, ...)` after first use throws `InvalidOperationException`. The safe canonical form: `static readonly JsonSerializerOptions _opts = new(...) { TypeInfoResolverChain = { myContext } }.Also(o => o.MakeReadOnly())` — sealing before assignment to the `readonly` field prevents any window between initialization and sealing.
- `JsonTypeInfo.IsReadOnly` mirrors the options seal at the type-info level. The correct pattern is for all modifiers to run before `MakeReadOnly()` is called on the owning options, not to check `IsReadOnly` at mutation time, which would make the modifier silently no-op rather than fail fast on contract violation. A modifier that silently skips mutation when `IsReadOnly` is true produces invisible policy omissions; the debug-phase equivalent is to assert `!typeInfo.IsReadOnly` before any property mutation.
