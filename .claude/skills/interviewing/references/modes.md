# [MODES]

## [01]-[INTERVIEW]

- Enter: Requirements ambiguous, contradictory, or living in the user's head; a brief, spec, or roadmap about to be authored from a thin prompt.
- Procedure: Establish the user's starting point, run impact-ranked rounds, challenge every conflict, and terminate on two decision-free rounds.
- Output: The decisions table — question, ruling, consequence — plus the implementation-ready prompt or spec draft that honors every entry.

## [02]-[BLINDSPOT_PASS]

- Enter: Unfamiliar territory, a plan about to seal, an explicit "what am I missing", or a corpus section feeding a large decision.
- Procedure: Read the target cold and adversarially; hunt boundaries, collisions, inversions, omissions, and load-bearing vagueness; verify each candidate against the corpus so only real gaps survive. A durable-doc target adds the prose-defect catalog as one lens.
- Output: Defect cards ranked by blast radius — each names what is missing, why it bites, and the copyable prompt that folds the fix back into the task. Zero cards is a legal result stated plainly.

## [03]-[BRAINSTORM]

- Enter: A named problem with multiple viable interventions and no committed direction.
- Procedure: Generate interventions spanning the scope-time spectrum — immediate patch, structural fix, long-horizon rebuild — each carrying thesis, cost, kills (what it forecloses), and reversibility; never one track of variations on a single idea.
- Output: The option set plotted across tiers with the leading path and the trigger conditions that flip it. A comparative board renders as a page artifact; the ranking is owned here.

## [04]-[QUIZ]

- Enter: Work about to merge or seal with comprehension of the change unproven.
- Procedure: Draw questions from the edges — failure modes, decisions that almost went the other way, the diff's least-visited corners — never from the summary; each concrete and single-answer, verifying the implementer grasps change, risk, tests, and edge cases.
- Output: The scored result; every wrong answer links to the section or diff hunk that teaches it. The gate proceeds only on a clean pass, otherwise re-teach and re-quiz.

## [05]-[BUY_IN]

- Enter: A decision needing acceptance from a reviewer or stakeholder outside the loop.
- Procedure: Lead with the demo, compress the narrative, name the tradeoffs and the ask, and pre-answer the objections that audience raises — cost, risk, alternatives not taken, precedent — each answered from evidence in the work. Opposed audiences each get their own objection lane.
- Output: The buy-in doc — decision, evidence-backed objection and answer pairs, explicit sign-off list, and a shareable message. Comparative or visual evidence renders as a page artifact.

## [06]-[TEACH_ME]

- Enter: The user cannot yet ask precisely — vague vocabulary, gesture-level intent, an unfamiliar domain.
- Procedure: Build the minimal vocabulary interactively — concept, concrete example, the technical term, then a check that the user restates their intent in it — before eliciting any decision the user cannot yet phrase; iterate until requests become decisions.
- Output: The vocabulary table — vague term, technical term, the decision it unlocks — plus the user's restated, now-precise request. The table binds the interview mode the way the decisions table binds authoring.
