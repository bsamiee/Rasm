# Spectral-Eigen Frontier

[CHOLESKY_FACTOR_STORAGE_TRUTH]:
- `ManagedLinearAlgebraProvider.CholeskyFactor(double[])` writes `a[i*order+j]` for `j > i` at each column `i`. In column-major layout (`DenseColumnMajorMatrixStorage`), index `i*order+j` maps to row `j`, column `i` — the lower triangle. The zeroing loop `a[k*order+i] = 0.0` for `k > i` clears the upper triangle. `Cholesky<T>.Factor` is therefore lower-triangular L as documented. `DoCholeskySolve` confirms: the forward-substitution loop reads `a[num3 * orderA + i]` (column `num3`, row `i`), which is the lower factor column `num3` accessed down the rows — forward substitution in L.
- `Cholesky.Solve(B)` dispatches to `CholeskySolveFactored`, which calls `DoCholeskySolve`: it performs a combined forward (L·y = B) and backward (Lᵀ·x = y) substitution in one pass. The result is `(L·Lᵀ)⁻¹·B`, not `L⁻¹·B`. Any congruence reduction step that requires `L⁻¹·A` alone cannot use `Cholesky.Solve`; it must construct a separate triangular system solve via `chol.Factor.LU()` (which for lower-triangular L costs O(n²) — the LU of a triangular matrix is itself). Both wave-1 and wave-2 congruence formulas incorrectly called `Factor.Transpose().Solve(A)`, which routes through `LU(Lᵀ)` and solves `Lᵀ·X = A`, yielding `L⁻ᵀ·A` rather than the intended first half.
- The correct two-step congruence: (1) `Y = chol.Factor.LU().Solve(A)` → `L·Y = A` → `Y = L⁻¹·A`; (2) `Z = chol.Factor.LU().Solve(Y.Transpose()).Transpose()` → `Z = (L⁻¹·(L⁻¹·A)ᵀ)ᵀ = L⁻¹·A·L⁻ᵀ`. Each `.LU()` call on the lower-triangular factor allocates an n×n copy for the PLU struct; `.Transpose()` allocates another n×n copy. Three allocations per congruence. For the back-transform `X = L⁻ᵀ·Y`: `chol.Factor.Transpose().LU().Solve(evd.EigenVectors)` is correct — `Lᵀ·X = Y` — upper-triangular backward substitution.

[NONSYMMETRIC_CONVERGENCE_ASYMMETRY_CORRECTED]:
- `NonsymmetricReduceHessenberToRealSchur(double[], ...)` (the real double path) has a bounded iteration count: `num10++` per outer step with `if (num10 >= 30 * order) throw new NonConvergenceException()`. The bound is 30n total outer iterations across all sub-problems, not 1000. For `n = 100`, this is 3000 outer steps; the counter resets to 0 only on a 2×2 block deflation. An ill-conditioned 100×100 real nonsymmetric matrix can thus exhaust the 3000-step budget and throw `NonConvergenceException` — contrary to wave-2's claim that the real nonsymmetric path never throws.
- `NonsymmetricReduceHessenberToRealSchur(Complex[], ...)` (the complex Hessenberg path) has no outer iteration bound. The `while (num >= 0)` loop terminates only on deflation (`num--` or `num -= 2`). The complex path uses shift strategies at `num2 == 10` and `num2 == 20` (exceptional shifts) but no ceiling check. A complex matrix whose Schur form fails to deflate via the shift heuristics runs indefinitely in the managed path with no exception. Residual validation after complex Evd is therefore mandatory; the API provides no evidence of whether the loop terminated by full deflation or left the eigenvalue carrier in a degenerate state.
- The asymmetry is: real nonsymmetric → catchable `NonConvergenceException` at `30n` steps; complex nonsymmetric → silent non-termination. Both contrast with symmetric/Hermitian → `NonConvergenceException` at 1000 steps per eigenvalue. A receipt that catches `NonConvergenceException` and converts to `ExhaustionFault` handles the real symmetric and real nonsymmetric paths; the complex nonsymmetric path requires a post-hoc Frobenius residual guard as the only detection mechanism.

[RANK_THRESHOLD_MAGNITUDESQUARED_HAZARD]:
- `ComplexExtensions.Norm(Complex c)` returns `c.MagnitudeSquared()` — the squared modulus — not `c.Magnitude`. `Evd.Rank` calls `EigenValues[i].AlmostEqual(Complex.Zero)`, which expands to `a.Norm().AlmostEqualNorm(0, a.NormOfDifference(Complex.Zero), threshold)` where both `Norm` and `NormOfDifference` use `MagnitudeSquared`. The effective rank threshold is `|λ|² < DoublePrecision ≈ 2.22 × 10⁻¹⁶` — equivalent to `|λ| < 1.49 × 10⁻⁸`. An eigenvalue of magnitude `1.5 × 10⁻⁸` clears the threshold; one of magnitude `1.4 × 10⁻⁸` is classified zero. For any problem with spectral radius ∼ 10⁶, this absolute `|λ|²` threshold misclassifies eigenvalues that are numerically significant at `10⁻⁸`. The `Determinant` property uses the same `AlmostEqual` call to short-circuit to `0.0` before accumulating the product — if any eigenvalue has `|λ| < 1.49 × 10⁻⁸`, the determinant returns exactly `0.0` regardless of all other eigenvalues.
- Receipt-level rank must be recomputed as `n − count(i : |λᵢ| < ε_rank × ‖A‖_spectral)` where `ε_rank` is a caller-supplied policy and `‖A‖_spectral ≈ max|λᵢ|` from the receipt's `EigenValues`. Never expose `evd.Rank` or `evd.IsFullRank` to domain logic; their thresholds are unrelated to any physical problem scale.

[NONSYMMETRIC_EVD_SOLVE_THROWS]:
- `DenseEvd.Solve(Matrix<double> input, Matrix<double> result)` throws `ArgumentException("Matrix must be symmetric.")` when `!IsSymmetric`. The Evd solve path through the spectral decomposition `V·Λ⁻¹·Vᵀ·b` only supports the symmetric case in the managed path. For a nonsymmetric Evd, `evd.Solve(b)` is not available at all — this is not a warning but a hard throw. Any use of `Evd` as a linear system solver is silently restricted to symmetric matrices; the receipt architecture must not expose `evd.Solve` for nonsymmetric inputs.

[QR_THIN_ALLOCATION_AND_Q_SHAPE]:
- `DenseQR.Create(matrix, QRMethod.Thin)` routes through `LinearAlgebraControl.Provider.ThinQRFactor`, which takes the input n×k matrix (written in-place as Q) and a k×k output array for R. The Q stored in `qr.Q` is exactly n×k. `GramSchmidt<T>` is hardcoded to `QRMethod.Full` in its constructor call `base(q, rFull, QRMethod.Full)` — calling `.GramSchmidt()` on any matrix produces a Q that is always n×n even for non-square inputs. For LOBPCG block orthonormalization where the block is n×k with k ≪ n, `QR(QRMethod.Thin).Q` avoids the n×n Q allocation that `GramSchmidt()` always incurs. The functional difference is not just stability (Householder vs. MGS) but also O(n²k) vs. O(n²) allocation.
- `QR<T>.R` is lazy: `_lazyR = new Lazy<Matrix<T>>(FullR.UpperTriangle)`. For `QRMethod.Thin`, `FullR` is already a k×k matrix; `.UpperTriangle()` allocates a k×k copy on first access. `R` is not needed for LOBPCG orthonormalization — only `Q` is used. The lazy allocation means R is never materialized unless accessed, making `QR(QRMethod.Thin)` a zero-overhead path when only Q is consumed.

[SPARSE_DENSE_MULTIPLY_NOT_DENSIFYING]:
- `SparseMatrix.DoMultiply(Matrix<double> other, Matrix<double> result)` does NOT densify when `other` is a `DenseMatrix`. When `other.Storage is DenseColumnMajorMatrixStorage<double>`, it directly reads the dense backing array `data` and loops over the sparse row pointers `_storage.RowPointers` and `_storage.ColumnIndices`. The cost is O(nnz × k) for a sparse n×n matrix with `nnz` non-zeros multiplied against a dense n×k matrix — sparse-aware and correct. Wave-1's claim that `SparseMatrix.Multiply(Matrix<double>)` densifies was false. The densification fallback (via `DenseVector` column extraction) only activates when `other.Storage` is neither `DenseColumnMajorMatrix` nor `SparseCompressedRowMatrixStorage` nor `DiagonalMatrixStorage` — i.e., for user-defined storage types.
- For LOBPCG operator application where A is sparse: `A.Multiply(W)` where A is `SparseMatrix<double>` and W is `DenseMatrix<double>` (n×3k block) is O(nnz × 3k) with no intermediate densification. The block multiply outcome `A_W = A.Multiply(W)` is a `DenseMatrix` (since `result` is built via `Matrix<double>.Build.SameAs(A, W, ...)` which for a sparse A and dense W returns a `DenseMatrix`). The subsequent Rayleigh-Ritz `W.TransposeThisAndMultiply(A_W)` is then dense-by-dense.

[MAPSINDEXEDINPLACE_FOR_RESYMMETRIZATION]:
- `DenseColumnMajorMatrixStorage.MapIndexedInplace(f, zeros)` iterates all `n × n` elements in column-major order via `CommonParallel.For(0, ColumnCount, ...)`. For the re-symmetrization `S[i,j] = (S[i,j] + S[j,i]) / 2.0`, using `s.MapIndexedInplace((i, j, v) => (v + s[j, i]) / 2.0, Zeros.Include)` is in-place and allocates nothing — no transpose copy, no intermediate matrix. The `s[j, i]` access goes through `S.At(j, i)` which for `DenseMatrix` computes `Values[i * n + j]` — a direct array lookup. This is the minimum-allocation re-symmetrization form; `(s + s.Transpose()) * 0.5` allocates three matrices (transpose copy, sum, scaled copy).
- `Zeros.Include` must be passed to `MapIndexedInplace` for re-symmetrization: the default `Zeros.AllowSkip` can skip zero-valued off-diagonal elements, leaving them unaveraged if the opposite triangle entry is nonzero. For a Rayleigh-Ritz matrix that may have zero entries from numerical cancellation, `Zeros.Include` is required for correctness.

[APPEND_ZERO_COPY_VARIANT]:
- `Matrix<T>.Append(right, result)` (the in-place overload with pre-allocated `result`) performs `CopySubMatrixToUnchecked` for both the left block and the right block directly into `result.Storage`. No intermediate allocation occurs when `result` is pre-allocated as a `DenseMatrix` of the correct shape. For the LOBPCG subspace construction `[X | A_X | P]`, pre-allocating a single n×3k `DenseMatrix workspace` and calling `X.Append(A_X, workspace); workspace.SetSubMatrix(0, n, 2*k, k, P)` (via `SetSubMatrix` which calls `CopySubMatrixTo`) avoids the chain of intermediate n×2k and n×3k allocations that `X.Append(A_X).Append(P)` produces. The zero-allocation form requires one pre-allocated workspace per iteration.

[FILTER_KERNEL_VECTORIZED_HEAT_RECIPE]:
- `TensorPrimitives.Exp<double>(ReadOnlySpan<double>, Span<double>)` compiles because `double : IExponentialFunctions<double>`. The heat filter weight vector `w[i] = exp(-t × λᵢ)` for real eigenvalues is computable as: (1) extract real parts via `MemoryMarshal.Cast<Complex, double>(evd.EigenValues.AsArray())` into a stride-2 span (only even-indexed entries are real parts); (2) or project first: `evd.EigenValues.AsArray().AsSpan().MapIndexed((i, c) => c.Real).ToArray()` — but MathNet's `Vector<T>.AsArray()` returns the internal backing for `DenseVector`. For a dedicated `double[]` weight buffer: (a) allocate `double[] weights = new double[n]`, (b) loop to project `weights[i] = eigenValues[i].Real`, (c) `TensorPrimitives.Multiply<double>(weights, -t, weights)` in-place (scalar form), (d) `TensorPrimitives.Exp<double>(weights, weights)` in-place. Steps c and d are vectorized via AVX/SSE on supported hardware. Step b is the only non-vectorized step (Complex→double projection).
- For zero-mode exclusion combined with heat filter: `weights[i] = eigenValues[i].Real < ε_zero ? 0.0 : exp(-t × eigenValues[i].Real)`. This cannot be split into the two `TensorPrimitives` calls above without a separate masking pass, because the epsilon check is a predicate per element. The correct form is a single `Vector<double>.Build.Dense(n, i => ...)` call — O(n) scalar loop — or `weights.MapIndexedInplace((i, _, w) => w < ε_zero ? 0.0 : w, Zeros.Include)` after the vectorized exp, which adds one scalar pass. The vectorized exp followed by a scalar mask pass is still faster than n scalar `Math.Exp` calls for large n.

[FIN_V5_CONSTRUCTION_SURFACE]:
- In LanguageExt 5, `Fin<A>` is an abstract class with two sealed concrete cases: `Fin<A>.Succ(A Value)` and `Fin<A>.Fail(Error Error)`. `FinSucc<A>` and `FinFail<A>` are marked `[Obsolete]`; the active factory surface is `Fin.Succ(value)` and `Fin.Fail<A>(error)` (static factory methods on the non-generic `Fin` class). `Try<A>` in v5 is a `record` wrapping `Func<Fin<A>> runTry` — it is not a monad over `IO` but a pure synchronous computation. `try_<A>(() => ...)` in the Prelude constructs a `Try<A>` whose `runTry` catches `Exception` and converts via `Error.New`. `this.Run()` on `Try<A>` calls `runTry()` wrapped in exception catching. The eigensolver boundary form is therefore `new Try<EigenReceipt>(() => Fin.Succ(BuildReceipt(matrix.Evd(sym)))).Run()`, which returns `Fin<EigenReceipt>` — a `Succ` case on success or a `Fail(Error)` case on any thrown exception including `NonConvergenceException`.
- `Error.New(Exception)` in v5 sets `IsExceptional = true` and wraps the exception as `Error.Inner`. To distinguish `NonConvergenceException` from `ArgumentException` at the receipt boundary without inspecting the exception type inside the `Fin.Fail` error, a tagged error approach is required: wrap the Evd call in a function that catches the specific exception types before returning `Fin`, converting each to a typed `Error` with a domain-specific code via `Error.New(int code, string message, Error inner)`. The code values `ExhaustionCode`, `IndefiniteBCode`, `ShapeCode` are receipt-level discriminants that survive the `Fin.Fail` carrier without requiring error.Exception inspection at use sites.

[MAPISYMMETRIC_EXACT_EQUALITY_HAZARD]:
- `DenseMatrix.IsSymmetric()` uses exact bit equality `_values[j * n + i] != _values[i * n + j]`. After any floating-point matrix operation — multiply, addition, congruence reduction — this check returns `false` for any pair where the two entries differ by even one ULP. The `Symmetricity.Unknown` → `Evd` path always triggers the full `IsSymmetric()` check (verified in `DenseEvd.Create`), and for an accumulated matrix this is guaranteed to route to the nonsymmetric Hessenberg path. The re-symmetrization `MapIndexedInplace((i, j, v) => (v + s[j, i]) / 2.0)` does produce exact bit-for-bit symmetry for the [i,j] and [j,i] entries, because the same arithmetic `(a + b) / 2` and `(b + a) / 2` produces bitwise identical results in IEEE 754 (addition is commutative in floating-point). Passing `Symmetricity.Symmetric` after `MapIndexedInplace` re-symmetrization is therefore safe and correct.

[EXHAUSTION_PARTIAL_EIGENVECTOR_STATE]:
- When `SymmetricDiagonalize` throws `NonConvergenceException` at iteration `num3 >= 1000` for eigenvalue index `j`, the `matrixEv` array (eigenvector accumulation) has been partially updated through all QR sweeps up to `j`. Eigenvectors for indices `< j` that converged earlier (their `e[k]` dropped below the relative threshold) are complete and orthonormal in their columns. Eigenvectors for indices `>= j` are intermediate products of partially-applied Givens rotations — numerically meaningless as eigenvectors. The exception discloses no information about which index `j` triggered exhaustion. A diagnostic receipt for exhaustion cannot identify which specific eigenvalue failed without instrumenting the algorithm; the `Error.Inner` of `NonConvergenceException` is null (no inner exception field in the type). The partial-convergence cutoff index is inaccessible through the public API.
- The iteration count `num3` resets per outer `j` loop iteration — it is a per-eigenvalue counter, not a global counter. A matrix with 100 eigenvalues where eigenvalue 97 is nearly degenerate could exhaust 1000 iterations for that eigenvalue while eigenvalues 0-96 converged in far fewer. The `NonConvergenceException` message `"An algorithm failed to converge."` is the only signal; the spectral radius estimate `num2` (running `max(|d[k]| + |e[k]|)`) is not exposed. A policy that records the spectral radius before attempting `Evd` and includes it in the `ExhaustionFault` payload (via the receipt's pre-computation evidence) is the only way to provide diagnostic context.

[REORTHOGONALIZATION_PROTOCOL_EXACT_COST]:
- `DenseGramSchmidt.Factorize` computes one-pass MGS: O(2nk²) flops for an n×k matrix — standard. After the pass, near-dependent columns accumulate error proportional to the condition number of the original column set. A second MGS pass on the result Q costs another O(2nk²). For LOBPCG block orthonormalization with k = 8 and n = 10000: single pass is O(1.28 × 10⁶) flops; double pass is O(2.56 × 10⁶). Compared to Householder QR at O(2nk² − 2k³/3) ≈ O(1.28 × 10⁶), the cost is identical for the stable case. The advantage of double-pass MGS over single-pass Householder is that the second pass is conditional: it is only needed when the R diagonal has entries below a floor. The recipe: `GramSchmidt().Q` → check diagonal of `.R` → if any `R[k,k] < floor`, call `GramSchmidt()` again on `.Q`. Because `GramSchmidt.Create` clones the input unconditionally, the second pass allocates a second n×k copy.
- Accessing `DenseGramSchmidt.R` (the `R` property from `QR<T>`) triggers `FullR.UpperTriangle()` lazily. For the LOBPCG near-dependence check, the R diagonal is accessible via `gs.R.Diagonal()` after the first `.GramSchmidt()` call. `Matrix<T>.Diagonal()` returns `Vector<T>` from `Storage.FoldByColumnUnchecked` with the diagonal selector. For a k×k dense R this is O(k) — negligible. The check `gs.R.Diagonal().Any(v => Math.Abs(v) < floor)` is the correct gate for the conditional second pass.

[SPECTRAL_FILTER_CLOSED_UNION_DISPATCH_SHAPE]:
- A spectral filter is fully specified by a `Func<double, double>` weight function plus zero-mode policy and bandwidth floor evidence. Collapsing heat, wave, biharmonic, diffusion, commute-time, and identity into a closed union with per-case `Invoke(double lambda)` dispatch is the correct doctrine shape. The composed filter `f₁ ∘ f₂` is a new union case whose `Invoke(λ) = f₁.Invoke(λ) * f₂.Invoke(λ)` — it is a `Product(FilterCase, FilterCase)` case, not a recursive union of unions. The `Identity` case has `Invoke(λ) = 1.0` and is the unit of composition; `Product(f, Identity)` folds to `f` by the monoid law. The union is closed because no caller can extend the weight dispatch without editing the union owner — new filter cases are added to the owner, not injected as delegates. The delegate-bundle approach (storing a raw `Func<double, double>`) is open and breaks the closed-union invariant; it is appropriate only at the receipt boundary where the weight vector is materialized, not in the filter algebra itself.
- Per-eigenvalue dispatch from the closed union via `weight[i] = filter.Invoke(eigenValues[i].Real)` is a single loop with one virtual dispatch per element (the union's abstract `Invoke` is an overridden method on each case). For n ∼ 500 eigenvalues, virtual dispatch cost is negligible. The `Product` case's `Invoke` is two virtual dispatches per eigenvalue — also negligible. Inlining through a `switch(filter) { case Heat h: ... }` or sealed-case pattern dispatch eliminates the virtual call but requires the dispatch site to enumerate all cases, violating `LIBRARY_DEPTH`. The virtual `Invoke` is correct.

**Further Considerations:**

The `Evd<T>.EigenValues` backing is always `DenseVector<Complex>` whose `.Values` property returns `Complex[]` directly. For the heat filter weight computation, `MemoryMarshal.Cast<Complex, double>` yields a `Span<double>` of length `2n` with real parts at indices 0, 2, 4, … and imaginary parts at 1, 3, 5, …. A custom `TensorPrimitives.Gather`-style strided read is not available in the public API, but `System.Runtime.InteropServices.MemoryMarshal.CreateSpan<double>(ref Unsafe.As<Complex, double>(ref eigenValues[0]), 2 * n)` followed by a stride-2 `TensorPrimitives.Multiply` with a stride-1 output buffer would be the AVX path. Since no strided TensorPrimitives overload exists, the real-part projection loop `for (int i = 0; i < n; i++) weights[i] = eigenValues[i].Real` remains the canonical route — it is O(n) scalar but LLVM-JIT-autovectorized for sequential real-part extraction on .NET 10.

`DenseEvd.Create` allocates four arrays before calling `EigenDecomp`: a new n×n `DenseMatrix` for eigenvectors, a new n×n `DenseMatrix` for the block diagonal, a new `DenseVector<Complex>` of length n for eigenvalues, and a copy of the input matrix inside `EigenDecomp` (`Buffer.BlockCopy` into `matrixEv`). Total allocation per Evd call: `4n² + n` elements — 3n² doubles plus n Complex structs (2 doubles each), totaling `3n² + 2n` doubles of working memory. For n = 1000: ≈ 24 MB per call. A cached-eigenbasis design that retains the `Evd<double>` object across filter applications avoids re-invoking `EigenDecomp` but holds the full 24 MB resident. The receipt's cache-provenance field distinguishes `Resident` (Evd object retained in memory) from `Reconstructed` (recomputed from scratch) as a memory/compute trade policy, not just a correctness annotation.

The `Symmetricity` hint affects exactly one decision inside `DenseEvd.Create` and `UserEvd.Create`: the `isSymmetric` bool which routes to either `SymmetricTridiagonalize+SymmetricDiagonalize` or `NonsymmetricReduceToHessenberg+NonsymmetricReduceHessenberToRealSchur`. Passing `Symmetricity.Hermitian` for a real symmetric `Matrix<double>` routes to `isSymmetric = true` — identical behavior. The only cost of passing the wrong hint when the matrix IS symmetric is a potential `NonConvergenceException` from the nonsymmetric path if the matrix has nearly-equal real eigenvalues that trigger exceptional shift saturation, or a silent partial-convergence for the complex matrix path. Passing the wrong hint when the matrix is NOT symmetric — claiming `Symmetricity.Symmetric` on a nonsymmetric matrix — causes `SymmetricTridiagonalize` to operate on a nonsymmetric matrix, producing incorrect eigenvectors with no exception; the only detection is the residual `‖A·V − V·D‖_F` check.
