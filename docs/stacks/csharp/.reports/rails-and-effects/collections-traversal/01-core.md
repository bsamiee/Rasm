# Collections and Traversal

[COLLECTION_CHOICE]:
- `Seq<A>` is backed by one of four concrete internal types: `SeqStrict<A>` (evaluated, array-backed with a `start` offset for O(1) `Tail`/`Init` without copying), `SeqLazy<A>` (wraps `IEnumerable<A>` and evaluates elements into a cache array at most once per element), `SeqConcat<A>` (deferred segment tree built by `Concat` or `+`), and `SeqEmptyInternal`. Calling `.Strict()` forces any lazy or concat internal to fully evaluate into a `SeqStrict<A>`.
- `Seq<A>` is preferred over `Lst<A>` in all cases except positional insertion: `Lst<A>` is an AVL-tree finger structure that supports `Insert(int, A)` and `InsertRange(int, IEnumerable<A>)` at O(log n); `Seq<A>` has no O(log n) mid-sequence insert. Outside positional mutation, `Seq<A>` is faster because structural sharing via the `start` offset avoids copying on `Cons` and `Tail`.
- `Arr<A>` is an immutable wrapper around a raw array: O(1) random read, O(n) modification because every change copies the entire backing array. Use when the access pattern is overwhelmingly read-intensive and the collection is short-lived or small; do not use when the sequence grows incrementally.
- `HashMap<K, V>` is backed by a CHAMP trie. Operations are O(1) amortized but unordered. `Map<K, V>` is backed by an AVL tree: O(log n) operations, iteration is key-ordered. Use `Map<K, V>` when ordered iteration or range queries (`FindRange(keyFrom, keyTo)`) are needed; use `HashMap<K, V>` otherwise.
- `HashSet<A>` and `Set<A>` mirror the same CHAMP vs AVL distinction. `Set<A>` supports ordered min, max, and range operations; `HashSet<A>` does not guarantee iteration order.
- `SeqEmpty` implicitly converts to every collection type — `Seq<A>`, `Arr<A>`, `Lst<A>`, `HashMap<K,V>`, `HashSet<A>`, `Map<K,V>`, `Que<A>`, `Stck<A>`, and others — via `op_Implicit(SeqEmpty)`. Use `Seq<A>.Empty` or the `SeqEmpty` sentinel at call sites that need a universal empty; do not construct any collection type explicitly just to pass an empty.
- `Iterable<A>` is a lightweight `IEnumerable<A>` wrapper that implements the full trait hierarchy (`Monad`, `Traversable`, `MonoidK`, and others) to make HKT combinators apply to arbitrary lazy enumerables. It is not a substitute for `Seq<A>` where evaluation count matters; it is the bridge when a caller-owned `IEnumerable<A>` must enter a trait-polymorphic context. Construction from a `ReadOnlySpan<A>` is available via `Iterable<A>.FromSpan(span)`, which calls `span.ToArray()` internally — not zero-copy at construction.
- `BiMap<A, B>` is a bidirectional map backed by two `Map<A,B>` and `Map<B,A>` AVL trees, giving O(log n) lookup in either direction. It is the canonical choice when a symmetric key–value bijection is the domain constraint, not a pair of `HashMap` instances.

[STRICTNESS_AND_LIFETIME]:
- `Seq<A>.Add(A)` appends to the end but forces complete evaluation of any lazy internal: the API remark states "Forces evaluation of the entire lazy sequence so the item can be appended." Operations that mutate the logical end of a lazy `Seq<A>` pay the full evaluation cost; prepend (`Cons`) does not.
- `SeqLoan<A>` wraps an `ArrayPool<T>`-rented array. It implements `IDisposable` with a finalizer safety net that returns the array to its originating pool. Any transformation (`Map`, `Filter`, `Bind`, and so on) on a `SeqLoan<A>` immediately clones the rented backing array into an owned `Seq<A>` and proceeds from there — the loan semantics are visible only during initial read-through. Calling `Dispose()` while another thread is reading the structure produces undefined behavior.
- `SeqLoan<A>.ToSeq()` creates a safe owned copy. Consume the loan in one pass, then promote to an owned `Seq<A>` and dispose the loan. Holding a `SeqLoan<A>` beyond the scope of the originating `ArrayPool` rent is a use-after-return bug that the finalizer mitigates but does not prevent.
- `SeqStrict<A>` has a `consDisallowed` flag. Once set, further `Cons` calls on that strict representation are blocked and the sequence allocates a new backing block instead of mutating the existing one. This flag protects structural sharing: a `Tail` that returns a sub-slice of the same array would be corrupted if the original sequence later prepended into that array.
- `SeqLazy<A>` sets `consDisallowed` once evaluation begins emitting elements into the internal cache array. Further `Cons` calls cannot prepend into that cache without corrupting in-flight iteration, so the flag forces a new allocation.

[TRAVERSE_AND_SEQUENCE]:
- `TraverseM<M, B>(Func<A, K<M, B>>)` on `Seq<A>`, `Arr<A>`, `Lst<A>`, `HashSet<A>`, and `Set<A>` maps into a monad `M` and collects results by sequencing via monadic bind left-to-right. This is the correct form when the monad has nontrivial short-circuit behavior (`Fin<A>`, `Eff<A>`, `Option<A>`): on the first `None` or `Fail`, the rest of the sequence is not evaluated.
- `Traverse<F, B>(Func<A, K<F, B>>)` maps into an applicative functor `F` and collects by applicative sequencing. For `Validation<FAIL, A>`, this accumulates all errors before returning; `TraverseM` with `Validation` in monad mode short-circuits on the first error. Use `Traverse` when all elements must be validated; use `TraverseM` when the first failure should abort.
- `Traversable.sequence<T, F, A>(K<T, K<F, A>>)` is the no-mapping form: when elements are already wrapped values, `sequence` swaps the container and functor layers without a prior `Map`. `sequenceM<T, M, A>` is the monadic variant with the same short-circuit semantics as `TraverseM` but without the mapping step.
- The `.As()` call is required after `TraverseM` or `Traverse` on any collection when the return type is `K<F, Seq<B>>` — the trait system returns the abstract HKT kind and `.As()` recovers the concrete type. Omitting `.As()` leaves the result in the abstract `K<F, Seq<B>>` form, which cannot be used directly.
- `Iterable<A>.TraverseIO<M, B>(Func<A, K<M, B>>, K<Iterable, A>)` is the async-aware traversal overload unique to `Iterable<A>`. When the iterable's `IsAsync` property is true, this overload correctly awaits each step. `Seq<A>` has no `TraverseIO` because it is always synchronously evaluated.

[FILTER_MAP_VIA_CHOOSE]:
- `Choose<A, B>(Func<A, Option<B>>)` on `Seq<A>`, `Arr<A>`, `Lst<A>`, `Stck<A>`, and `Map<K, V>` applies a function that returns `Option<B>`, discards all `None` results, and returns the `Some` values. This is atomic filter-map in one pass; it replaces `Filter(predicate).Map(projection)`, which makes two passes and requires the projection to repeat the same guard. `Map<K,V>.Choose` and its two-argument form `Choose(Func<K, V, Option<R>>)` carry the key through.
- `Seq.choose<A, B>(Seq<A>, Func<A, Option<B>>)` is the static-module form; `SeqExtensions.Choose<A, B>(Seq<A>, Func<A, Option<B>>)` is the extension form. Prefer the extension form at call sites that chain.
- `SeqExtensions.Choose<A, B>(Seq<A>, Func<int, A, Option<B>>)` is the indexed form: the position of each element is passed alongside the value. Use this for index-aware filter-map without a prior zip-with-index step.
- `MonoidK.choose<F, A, B>(K<F, A>, Func<A, Option<B>>)` is the HKT-polymorphic form: for any `MonoidK` instance `F`, it propagates the structure only for elements where the mapping returns `Some`. This is how `Choose` generalizes beyond concrete collection types into the trait hierarchy.

[IMMUTABLE_FOLDS]:
- `Fold<S>(S, Func<S, A, S>)` on `Seq<A>` and `SeqLoan<A>` is left-associative, reducing from head to tail. It is stack-safe for any length. `FoldBack<S>(S, Func<S, A, S>)` on `SeqLoan<A>` uses recursive processing for strict sequences shorter than 5,000 items — recursion is faster for small sizes — and falls back to an iterative CPS approach for longer sequences or lazy inputs. Treat any fold that builds a right-to-left accumulation on large lazy sequences as a candidate to rewrite as `Rev().Fold`.
- `FoldWhile<S>(Func<A, Func<S, S>>, Func<(S, A), bool>, S, K<T, A>)` from the `Foldable<T>` trait takes a curried folding function and a predicate over the `(State, Value)` pair and stops as soon as the predicate returns false. The predicate receives the current state and the next element, so it can inspect accumulated state to decide when to stop.
- `FoldUntil` is the complement: it continues while the predicate is false and stops on the first true. The monadic variants `FoldWhileM` and `FoldUntilM` take a `Func<A, Func<S, K<M, S>>>` and thread a monad through the fold; their predicate is `Func<A, bool>` (element-only), not a `(S, A)` pair.
- `FoldBackMaybe<A, S>(Func<A, Func<S, Option<S>>>, S, K<T, A>)` stops as soon as the folding function returns `None`. The function receives the element first, then the accumulator — `Func<A, Func<S, Option<S>>>` — so it decides early-exit on the element alone without inspecting accumulated state.
- `scan<S, A>(Seq<A>, S, Func<S, A, S>)` and `scanBack<S, A>(Seq<A>, S, Func<S, A, S>)` both return a `Seq<S>` of all intermediate states including the initial state. Use `scan` when downstream requires all running totals, not just the terminal value.

[STRUCTURAL_SHARING_AND_PERFORMANCE]:
- `Seq<A>.Cons(A)` prepends without copying when `consDisallowed` is false: if the backing `SeqStrict<A>` array has unused capacity before `start`, the new head is written there and `start` is decremented. Prepend is O(1) amortized and sharing is preserved for the tail. Once capacity is exhausted or `consDisallowed` is set, a new array is allocated.
- `Seq<A>.Tail` and `Seq<A>.Init` are O(1) for `SeqStrict<A>` because they adjust `start` and `count` over the same backing array rather than copying. Materializing `Head` and `Tail` in a recursive descent over a `SeqStrict<A>` is therefore structurally safe; it does not degrade like array copying would.
- `Map<K, V>` and `Set<A>` support `Skip(int)` — returning a new tree without the first N elements — in O(n) time. The operation walks the tree; there is no O(log n) structural prefix sharing.
- `Seq<A>.Case` returns an `object` shaped as `null` (empty), `A` (singleton), or `(A, Seq<A>)` (head–tail) for pattern-matching. `Map<K, V>.Case` and `Arr<A>.Case` follow the same pattern: `null` for empty, `(K, V)` for singleton, `((K, V), Seq<(K, V)>)` for more. This is how structural decomposition works across all collection types — directly into a switch expression without `.IsEmpty` guards.

[INDEXED_PROJECTION]:
- `Seq<A>.Map<B>(Func<A, int, B>)` is the indexed map overload; the remark states "Exposes an index for each map." This eliminates the zip-with-index pattern entirely when projection requires positional information.
- `Seq.generate<A>(int, Func<int, A>)` produces a strict `Seq<A>` of the given length by calling the factory with each index. It is the canonical way to build index-initialized sequences without a mutable accumulator.
- `Seq<A>` throws `IndexOutOfRangeException` from its `this[Index]` indexer when out of range; there is no safe indexer on the public struct. The safe access idiom for the public API is `.HeadOrNone()` and structural `Tail` decomposition, or `Foldable<Seq>.At` via `ta.At(index)` which returns `Option<A>`. The `Foldable<Seq>.At` override on the `Seq` static class dispatches to `seq[index]` after a bounds guard and is O(1) for `SeqStrict<A>` internally.

[HASHMAP_ALGEBRA]:
- `HashMap<K, V>.operator+` delegates to `Combine`, which calls `TrieMap.Append`, which calls `TryAddRange`. `TryAddRange` uses `TryAdd` — `UpdateType.TryAdd` — leaving the existing node unchanged if the key is already present. Both `operator+` and `Union(IEnumerable<(K,V)>)` with no merge function are left-biased: keys already in the left-hand map are preserved on collision.
- `HashMap<K, V>.Union(IEnumerable<(K, V)>, WhenMatched<K, V, V, V>)` provides controlled merge semantics. `WhenMatched` is a delegate `(key, leftValue, rightValue) => result`. Pass `(_, l, r) => r` to achieve right-bias. The no-delegate overload does not provide right-bias.
- `HashMap<K, V>.Union<W, R>(IEnumerable<(K, W)>, WhenMissing<K, V, R>, WhenMissing<K, W, R>, WhenMatched<K, V, W, R>)` is the full heterogeneous merge: left-only keys pass through `WhenMissing MapLeft`, right-only keys through `WhenMissing MapRight`, and matched keys produce the result type `R` through `WhenMatched`.
- `HashMap<K, V>.Intersect(IEnumerable<(K, V)>, WhenMatched<K, V, V, V>)` retains only keys present in both and applies the merge function. `Except(IEnumerable<K>)` removes all keys present in the right-hand set; `Except(IEnumerable<(K, V)>)` is key-only difference (values on the right are ignored).
- `AtomHashMap<K, V>.SwapKey(K, Func<Option<V>, Option<V>>)` is a single atomic CAS over one entry: `None` input means the key was absent, `Some(x)` means it was present. `Some(y)` from the delegate adds or updates; `None` removes. This replaces a read–modify–write sequence with one atomic operation and is the correct primitive for conditional update-or-insert.
- `AtomHashMap<K, V>.MapInPlace(Func<V, V>)` mutates all values atomically; the remark warns the function may be called multiple times under thread contention. Keep this function fast, pure, and free of side effects.
- `TrackingHashMap<K, V>` wraps `HashMap<K, V>` and records a change morphism since the last `Snapshot()` call. The `Changes` property returns a `HashMap<K, Change<V>>` where `Change<V>` is one of `EntryAdded`, `EntryMapped`, or `EntryRemoved`. The change map stores at most one morphism per key regardless of how many times it was mutated — it represents the net effect of all mutations since the last snapshot. `Snapshot()` resets the change ledger by constructing a new `TrackingHashMap<K, V>` with the current value map and an empty change map.

[ATOMIC_COLLECTION_STATE]:
- `AtomSeq<A>` stores the active sequence in a field typed `ISeqInternal<A>`. `Swap(Func<Seq<A>, Seq<A>>)` atomically replaces the internal with CAS retries; `SwapIO(Func<Seq<A>, Seq<A>>)` does the same but returns an `IO<Seq<A>>` capturing the resulting value.
- `AtomSeq<A>.MapInPlace(Func<A, A>)` maps all elements atomically. Like `AtomHashMap<K,V>.MapInPlace`, the mapping function may execute multiple times under contention.
- `AtomHashMap<K, V>` and `AtomSeq<A>` expose no `Subscribe` or reactive-stream surface. State is read by reading the current value via `.Value` or property accessors; change notification is not a contract of these types.

[TRAVERSAL_DECOMPOSITION]:
- `Seq.tails<A>(Seq<A>)` returns all final segments from longest to shortest: for `[a, b, c]` the result is `[[a,b,c], [b,c], [c], []]`. Each element is a `Tail` slice — O(1) per tail on `SeqStrict<A>` due to the `start` offset — so the full `Seq<Seq<A>>` costs O(n) to construct. `tailsr<A>` is a recursive implementation faster for small sequences.
- `Seq.span<A>(Seq<A>, Func<A, bool>)` splits at the first element where the predicate is false, returning `(prefix, suffix)` as a tuple. The prefix is the maximal initial segment satisfying the predicate. This is a single-pass structural split with no intermediate allocation beyond the two `Seq<A>` slices.
- `Patch<EqA, A>` is an edit-list groupoid over ordered sequences. `Patch.diff<EqA, A>(IEnumerable<A>, IEnumerable<A>)` computes the minimal patch between two sequences; `Patch.apply<EqA, A>(Patch<EqA,A>, Lst<A>)` applies it. Patches compose via `append` and invert via `inverse`. `Patch.applicable(patch, document)` and `Patch.composable(p, q)` are predicates confirming whether a patch can be validly applied or composed before doing so. This is the correct primitive for versioned or diffable sequence state, replacing ad-hoc before/after comparison.
