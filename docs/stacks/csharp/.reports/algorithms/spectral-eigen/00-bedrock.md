# Spectral-Eigen

[ROUTE_SELECTION]:
- `Matrix<T>.Evd(Symmetricity)` is the single generic entry point for all dense eigensolution routes. `Symmetricity` is a four-case enum ÔÇö `Unknown`, `Symmetric`, `Hermitian`, `Asymmetric`. Passing the known case skips the O(n┬▓) `IsSymmetric()` scan; `Unknown` pays that cost on every call.
- `DenseMatrix.Evd` calls `DenseEvd.Create`, which delegates to `LinearAlgebraControl.Provider.EigenDecomp(bool isSymmetric, int order, T[] matrix, T[] matrixEv, Complex[] vectorEv, T[] matrixD)`. A provider wired via `Control.UseNativeMKL()` or `Control.UseNativeOpenBLAS()` substitutes a LAPACK driver behind this interface; the fallback is `ManagedLinearAlgebraProvider`. `SparseMatrix` and `DiagonalMatrix` do not override `Evd`, routing to `UserEvd.Create` ÔÇö the pure-managed EISPACK path ÔÇö regardless of the active provider. `QR.Q` returns a `DenseMatrix` (a clone allocated inside `DenseQR.Create`), so it dispatches to `DenseEvd` and reaches the provider-backed path without materialization. Materializing any non-`DenseMatrix` input to `DenseMatrix` before calling `Evd` is required to reach the provider-backed path.
- The generalized symmetric-definite problem A┬Ěx = ╬╗┬ĚB┬Ěx has no native MathNet route. The congruence reduction `LÔü╗┬╣┬ĚA┬ĚLÔü╗ßÁÇ` must be built manually using `chol.Factor.LU()` ÔÇö not `Cholesky.Solve`, which returns `(L┬ĚLßÁÇ)Ôü╗┬╣┬ĚB`. The full recipe is in `[CONGRUENCE_REDUCTION]` below.
- MathNet has no sparse iterative eigensolver ÔÇö `BiCgStab`, `GpBiCg`, `MlkBiCgStab`, and `TFQMR` are linear-system solvers only. Block-preconditioned eigeniteration must be hand-assembled using MathNet's dense `Evd` for the Rayleigh-Ritz subproblem and `SparseMatrix.Multiply(Vector<double>)` for operator application.
- `DenseEvd.Solve(Matrix<double>, Matrix<double>)` throws `ArgumentException("Matrix must be symmetric.")` when `!IsSymmetric`. The spectral-decomposition solve path `V┬Ě╬ŤÔü╗┬╣┬ĚVßÁÇ┬Ěb` exists only for the symmetric case. For nonsymmetric Evd, `evd.Solve(b)` is a hard throw, not a degraded result. A receipt architecture must gate `evd.Solve` exclusively to symmetric and Hermitian receipts.

[EVD_RECEIPT_SHAPE]:
- `Evd<T>` exposes `EigenValues` (`Vector<Complex>`), `EigenVectors` (`Matrix<T>`), and `D` (`Matrix<T>`, block-diagonal). The `Vector<Complex>` carrier is uniform across all four scalar types ÔÇö the shape is not polymorphic by symmetry class. For real symmetric inputs all imaginary parts are exactly zero after `SymmetricDiagonalize` zeroes `e[]`. To recover a real weight array: project via `evd.EigenValues.AsArray().Select(c => c.Real).ToArray()`; `AsArray()` returns the raw backing `Complex[]` without allocation for `DenseVector<Complex>`. `MemoryMarshal.Cast<Complex, double>` yields a stride-2 span whose even-indexed elements are real parts; no strided `TensorPrimitives` overload exists, making the scalar projection loop canonical ÔÇö RyuJIT does not auto-vectorize this loop; explicit SIMD requires manual span splitting or `TensorPrimitives` on a pre-projected `double[]`.
- `Evd<T>.IsSymmetric` is a `bool`, not the original `Symmetricity` discriminant. Both `Symmetric` and `Hermitian` inputs set `IsSymmetric = true` because `DenseEvd.Create` maps both cases to `isSymmetric = true`. The receipt loses the original symmetry evidence; a receipt that must distinguish them must carry `Symmetricity` separately.
- Eigenvalues are sorted ascending for symmetric and Hermitian inputs by the selection sort in `SymmetricDiagonalize`. The LAPACK path follows the `dsyev`/`zheev` ascending-order contract. Nonsymmetric inputs carry eigenvalues in Schur-convergence order with no sort guarantee. The managed ascending sort is a selection sort over `d[]` that performs at most nÔłĺ1 column swaps (one per outer iteration), each touching n elements ÔÇö O(n┬▓) total swap work and O(n┬▓) comparisons; the LAPACK provider bypasses this sort entirely.
- `Evd<T>.Rank` calls `EigenValues[i].AlmostEqual(Complex.Zero)`, which expands to `a.Norm().AlmostEqualNorm(0.0, a.NormOfDifference(Complex.Zero), DefaultDoubleAccuracy)` where both `Norm` and `NormOfDifference` call `MagnitudeSquared()`. The comparison reduces to `|╬╗|┬▓ < DefaultDoubleAccuracy Ôëł 1.11 ├Ś 10Ôü╗┬╣ÔüÁ`, equivalent to `|╬╗| < sqrt(1.11 ├Ś 10Ôü╗┬╣ÔüÁ) Ôëł 3.33 ├Ś 10Ôü╗ÔüŞ`. An eigenvalue at magnitude `3.4 ├Ś 10Ôü╗ÔüŞ` clears the threshold; one at `3.2 ├Ś 10Ôü╗ÔüŞ` is classified zero. `Determinant` hits the same `AlmostEqual` call and short-circuits to exactly `0.0` the moment any eigenvalue has `|╬╗| < 3.33 ├Ś 10Ôü╗ÔüŞ`. For a matrix with ÔÇľAÔÇľÔéé Ôł╝ 10ÔüÂ, eigenvalues at magnitude 10Ôü╗ÔüŞ that are numerically significant are misclassified. Never expose `evd.Rank`, `evd.IsFullRank`, or `evd.Determinant` to domain logic; recompute rank as `n Ôłĺ count(i : |╬╗ßÁó| < ╬Á_rank ├Ś max|╬╗Ô▒╝|)` with a caller-supplied spectral-radius-relative `╬Á_rank`.
- `NonConvergenceException` (symmetric QR exhausting 1000 steps per eigenvalue) and `ArgumentException` (non-square input or non-SPD B) are both thrown during `Evd` construction. `Fin<A>` is an abstract class with two sealed nested cases; the active factory surface is `Fin.Succ(value)` and `Fin.Fail<A>(error)`. `Try<A>` is `record Try<A>(Func<Fin<A>> runTry)`; `Run()` wraps `runTry()` in a try/catch converting any `Exception` to `Fin.Fail<A>(ex)`. The boundary form is `new Try<EigenReceipt>(() => Fin.Succ(BuildReceipt(matrix.Evd(sym)))).Run()`. Both failure types collapse to `Fin.Fail(Error)`; distinguishing exhaustion from argument violation requires tagging at the boundary before propagation: wrap each specific exception with `Error.New(int code, string message, Error inner)` using domain codes `ExhaustionCode`, `IndefiniteBCode`, and `ShapeCode` ÔÇö these survive the `Fin.Fail` carrier as first-class discriminants without inspecting `error.Exception` at use sites.

[HERMITIAN_EIGENVALUE_CARRIER]:
- For `Matrix<Complex>` with `Symmetricity.Hermitian`, `DenseEvd.Create` branches on `matrix.IsHermitian()` and routes to `SymmetricTridiagonalize Ôćĺ SymmetricDiagonalize Ôćĺ SymmetricUntridiagonalize`. After convergence, `e[i] = 0.0` exactly, and `vectorEv[i] = new Complex(d[i], 0.0)` ÔÇö Hermitian eigenvalues have exactly zero imaginary part in the receipt.
- `SymmetricUntridiagonalize` applies conjugate multiplications to restore complex eigenvector structure from the real rotations used during diagonalization: `dataEv[j * order + i] = dataEv[j * order + i].Real ├Ś tau[i].Conjugate()`. Hermitian eigenvectors are genuinely complex with nonzero imaginary parts even though eigenvalues are real. Projecting eigenvectors to their real parts is incorrect for the Hermitian case.
- For nonsymmetric complex input, the complex Schur path fills `vectorEv[i]` with generally nonzero imaginary parts; no real-part projection is valid. The complex nonsymmetric path has no outer iteration bound ÔÇö the `while (num >= 0)` loop terminates only on deflation via `num--` (1├Ś1 blocks only; the complex path has no 2├Ś2 deflation branch); exceptional shift strategies fire at internal counters 10 and 20 but impose no ceiling. Silent non-termination for complex matrices whose Schur form fails to deflate through the shift heuristics makes post-hoc Frobenius residual validation mandatory for complex nonsymmetric inputs.

[REAL_COMPLEX_POLYMORPHISM_CEILING]:
- `Complex` implements `IMultiplyOperators<Complex, Complex, Complex>` but NOT `IMultiplyOperators<Complex, double, Complex>`. No BCL numeric type provides the `(Complex, double, Complex)` instantiation. A generic method constrained on `T : IMultiplyOperators<T, T, T>` cannot scale a `Vector<Complex>` by a real weight without an explicit conversion site.
- For spectral filter weight application, the delegate-bundle form is the correct wide parameterization: an explicit `Func<TM, double, TM> scale` alongside the filter function, with concrete instance `(c, w) => new Complex(c.Real * w, c.Imaginary * w)` provided at construction. `Vector<T>.Multiply(T scalar)` requires the scalar to be type `T`, so real-weight scaling of complex columns requires `new Complex(w, 0)` ÔÇö which breaks a generic delegate signature. Collapsing to `where T : IMultiplyOperators<T, TWt, T>` fails because this two-parameter form is absent from the BCL hierarchy for these types.
- `TensorPrimitives.Dot<Complex>` compiles because `Complex` satisfies the four homogeneous interfaces. `TensorPrimitives.Norm<Complex>` does not compile ÔÇö `Complex` lacks `IRootFunctions<Complex>`. `TensorPrimitives.MultiplyAdd<Complex>` compiles. `TensorPrimitives.FusedMultiplyAdd<Complex>` does not ÔÇö `IFloatingPointIeee754<Complex>` is unimplemented. The real-domain hot loop uses `TensorPrimitives.MultiplyAdd<double>` and `TensorPrimitives.Dot<double>` directly; the complex-domain equivalent requires a manual loop.
- `Complex` satisfies `INumberBase<Complex>` and `ISignedNumber<Complex>` but not `INumber<Complex>` (which requires `IComparable<T>`). A constraint `where T : INumberBase<T>` accepts both `double` and `Complex`; `where T : INumber<T>` or `where T : IFloatingPoint<T>` excludes `Complex`. The Rayleigh quotient maps `Complex Ôćĺ double` via `.Real`, breaking the generic chain regardless of constraint choice. Receipt eigenvalue scalars must be `double` for symmetric and Hermitian inputs even when the eigenvector matrix is `Matrix<Complex>`.

[LOBPCG_MECHANICS]:
- LOBPCG requires three stages per iteration: orthonormalize the block X, apply A and optionally the preconditioner, then solve the Rayleigh-Ritz dense subproblem over the stacked space [X | A┬ĚX | P].
- `Matrix<T>.GramSchmidt()` routes through `DenseGramSchmidt.Create`, which clones the input (`(DenseMatrix)matrix.Clone()`) ÔÇö the returned `.Q` is this cloned, in-place-modified m├Śn matrix (GramSchmidt produces a thin m├Śn Q by design per the class docs). `Matrix<T>.QR(QRMethod.Thin)` routes through `DenseQR.Create(matrix, Thin)`, which also clones the input (`q = matrix.Clone()`) and overwrites the clone with the Householder reflectors via `ThinQRFactor`; R lands in a separate k├Śk `DenseMatrix`. Both paths allocate one n├Śk clone; neither writes Q back into the caller's array. The stability advantage of `QR(QRMethod.Thin)` over `.GramSchmidt()` is numerical: Householder reflections are backward-stable against column-norm near-degeneracy, whereas single-pass MGS loses orthogonality in nearly dependent columns ÔÇö allocation profiles are equivalent.
- Single-pass MGS for an n├Śk block costs O(2nk┬▓) flops; a second reorthogonalization pass costs another O(2nk┬▓). For LOBPCG with k = 8 and n = 10000, single pass is O(1.28 ├Ś 10ÔüÂ) flops ÔÇö identical to Householder QR at O(2nk┬▓ Ôłĺ 2k┬│/3). The advantage of conditional double-pass MGS over single-pass Householder is the second pass is only invoked when the R diagonal has entries below the floor. `DenseGramSchmidt.Factorize` throws `ArgumentException("Matrix must not be rank deficient.")` only when column norm is exactly `0.0`; near-zero norms propagate silently. Column norms must be checked against a domain-specific floor before calling `.GramSchmidt()`. The near-dependence check `qr.R.Diagonal().Any(v => Math.Abs(v) < floor)` triggers lazy R materialization on first access: `QR<T>.R` returns `FullR.UpperTriangle()` (a k├Śk copy), then `.Diagonal()` allocates a length-k vector ÔÇö neither call is allocation-free.
- Deterministic seeded start-block construction: drawing k orthonormal columns by seeding `Matrix<double>.Build.Random(n, k, new SystemRandomSource(seed))` then calling `QR(QRMethod.Thin).Q` gives reproducible block initialization for convergence regression tests. For graph Laplacians the constant vector (all-ones normalized) is the exact ╬╗ÔéÇ = 0 eigenvector; seeding the first column as `1/ÔłÜn` and drawing the remainder randomly, then re-orthogonalizing via `QR(QRMethod.Thin).Q`, places the start block in the relevant eigenspace rather than a random orientation ÔÇö reducing iteration count for clustered near-zero spectra.
- Diagonal preconditioning: the correction for residual column `r_i` given diagonal entry `a_ii` is `(╬╗ - a_ii)Ôü╗┬╣ ┬Ě r_i`. When `|╬╗ - a_ii| < ╬Á`, fall back to identity (correction = `r_i`). The epsilon is algorithm-specific and must be supplied as a policy value, not taken from `Precision.DoublePrecision`.
- Previous-direction dropout: when a column's norm after orthogonalization falls below the floor, the P block contracts. The Rayleigh-Ritz dense subproblem shrinks from 3k to (2k + p) where p is the surviving P column count.
- Re-symmetrization of the Rayleigh-Ritz matrix is mandatory before `Evd(Symmetricity.Symmetric)`. `DenseMatrix.IsSymmetric()` uses exact bit equality `_values[j * n + i] != _values[i * n + j]`; any accumulated floating-point matrix fails this check for entries differing by even one ULP, routing through the nonsymmetric Hessenberg path. The correct form is `(s + s.Transpose()) * 0.5` ÔÇö this allocates three matrices but guarantees both `[i,j]` and `[j,i]` carry the same IEEE 754 bit pattern, so the subsequent exact-equality `IsSymmetric()` check passes unconditionally. `DenseColumnMajorMatrixStorage.MapIndexedInplace` processes entries sequentially via `CommonParallel.For`, mutating the same backing array the lambda reads; for a self-averaging step `(v + s[j,i]) / 2.0` the mirrored entry at `[j,i]` has already been modified by an earlier iteration, producing asymmetric results. Use `(s + s.Transpose()) * 0.5` for correctness.
- Hermitian Rayleigh-Ritz variant: for the Hermitian block-preconditioned case the stacked workspace is `Matrix<Complex>` and the Rayleigh-Ritz matrix S = Wß┤┤┬ĚA┬ĚW. `W.ConjugateTransposeThisAndMultiply(A.Multiply(W))` is the assembly call. Block orthonormalization uses `QR(QRMethod.Thin)` on `Matrix<Complex>` ÔÇö `DenseQR.Create` handles the complex overload, calling `ThinQRFactor` on the complex backing array. Re-symmetrization before `Evd(Symmetricity.Hermitian)` replaces the averaging step with `(s + s.ConjugateTranspose()) * 0.5` to enforce Hermitian structure (not merely symmetric).
- Rayleigh-Ritz via generalized eigendecomposition: the subspace problem S┬Ěc = ╬╗┬ĚG┬Ěc (S = WßÁÇ┬ĚA┬ĚW, G = WßÁÇ┬ĚW) reduces to a standard eigenvalue problem when W is already orthonormal (G = I), making `Evd(Symmetricity.Symmetric)` on S exact. When orthonormalization is imperfect ÔÇö accumulated floating-point drift makes G Ôëá I ÔÇö the correct route is congruence reduction via `G`'s Cholesky factor, not re-orthonormalization; however the re-orthonormalize-then-standard-Evd path is equivalent only when the orthonormality residual `ÔÇľWßÁÇ┬ĚW Ôłĺ IÔÇľ_F < ╬Á` is verified per iteration. A receipt that skips this check silently accumulates Rayleigh-quotient error.
- Subspace construction `[X | A_X | P]`: `Matrix<T>.Append(Matrix<T> right, Matrix<T> result)` (the in-place overload) calls `CopySubMatrixToUnchecked` for both blocks into the pre-allocated `result.Storage`. No intermediate allocation occurs when `result` is a pre-allocated `DenseMatrix` of the correct shape. Pre-allocating a single n├Ś3k workspace avoids the chain of intermediate n├Ś2k and n├Ś3k allocations from chained `X.Append(A_X).Append(P)`.

[MATRIX_MULTIPLY_API_PRECISION]:
- `A.TransposeAndMultiply(B)` computes A┬ĚBßÁÇ. `A.TransposeThisAndMultiply(B)` computes AßÁÇ┬ĚB. `A.ConjugateTransposeAndMultiply(B)` computes A┬ĚBß┤┤. `A.ConjugateTransposeThisAndMultiply(B)` computes Aß┤┤┬ĚB. These are verified against BLAS flag pairs passed to `MatrixMultiplyWithUpdate`.
- The Rayleigh-Ritz matrix S = WßÁÇ┬ĚA┬ĚW: `W.TransposeThisAndMultiply(A.Multiply(W))` is the two-call form. Neither form exploits S's symmetry; both allocate an intermediate n├Ś3k matrix. `SparseMatrix.DoMultiply(Matrix<double> other, Matrix<double> result)` does NOT densify when `other.Storage` is `DenseColumnMajorMatrixStorage<double>` ÔÇö it reads sparse `RowPointers` and `ColumnIndices` directly against the dense backing array, computing each result entry as a sparse inner product at O(nnz ├Ś k) cost. The densification fallback (via `DenseVector` column extraction) activates only for user-defined storage types not matching the recognized patterns. The result is constructed via `Matrix<double>.Build.SameAs(A, W, ...)` as a `DenseMatrix`. The subsequent Rayleigh-Ritz `W.TransposeThisAndMultiply(A_W)` is then a dense-by-dense multiply.

[SPECTRAL_FILTER_ALGEBRA]:
- The spectral filter algebra maps a signal through an eigenbasis: for eigenpairs `{(╬╗ßÁó, vßÁó)}` the filtered output is `ÔłĹßÁó f(╬╗ßÁó) ┬Ě (vßÁóßÁÇ ┬Ě x) ┬Ě vßÁó`. The filter cases ÔÇö heat `exp(-t┬Ě╬╗)`, wave `cos(t┬ĚÔłÜ╬╗)`, biharmonic `╬╗┬▓`, diffusion `1/(1 + t┬Ě╬╗)`, commute-time `1/ÔłÜ╬╗`, identity `1` ÔÇö form a closed union whose per-case `Invoke(double lambda)` is a virtual method on each case. The delegate-bundle form (storing a raw `Func<double, double>`) is open and breaks the closed-union invariant; it belongs only at the receipt boundary where the weight vector is materialized, not in the filter algebra itself.
- Partial-monoid composition: `fÔéü Ôłś fÔéé` produces a `Product(FilterCase, FilterCase)` union case whose `Invoke(╬╗) = fÔéü.Invoke(╬╗) ├Ś fÔéé.Invoke(╬╗)` ÔÇö two virtual dispatches per eigenvalue. `Product` is not a recursive union of unions. For n Ôł╝ 500 eigenvalues the dispatch cost is negligible; inlining through a pattern match would violate the closed-union invariant by requiring the dispatch site to enumerate all cases. `Identity`'s `Invoke(╬╗) = 1.0` is the monoid unit; `Product(f, Identity)` folds to `f` by the monoid law. Weight vectors compose via `weights1.PointwiseMultiply(weights2)` ÔÇö O(n). An identity-filter check `weights.ForAll(w => w == 1.0)` uses `Storage.Find` internally and short-circuits on the first mismatch. A weight sum below a bandwidth floor is a distinct exhaustion marker ÔÇö the receipt carries `WeightSum` and the call site gates on it, because zero weight sum (all modes excluded or underflowed) must not be interpreted as a valid zero-signal result.
- Zero-mode handling: for Laplacian-derived operators ╬╗ÔéÇ Ôëł 0; filters with `1/╬╗` or `1/ÔłÜ╬╗` diverge there. The weight at index i is `╬╗ßÁó < ╬Á_zero ? 0.0 : f(╬╗ßÁó)` ÔÇö excluded, not clamped. `╬Á_zero` is a policy parameter. The spread floor must be computed as `eigenvalues.Where(╬╗ => ╬╗.Real > ╬Á_zero).Min(╬╗ => ╬╗.Real)`, not blindly from `EigenValues[1].Real`, because that index may itself be below the exclusion threshold when multiple near-zero modes exist. `Vector<T>.Map(f, result, Zeros.Include)` passes zero-valued elements to the map function; `MapIndexed` with the index as a guard (`i == 0 ? 0.0 : f(╬╗ßÁó)`) enables index-first exclusion for normalized graph Laplacians where the 0th mode is always the zero mode.
- Heat filter weight computation with vectorized primitives: (1) scalar `Complex Ôćĺ double` projection loop to populate `weights[i] = eigenValues[i].Real` ÔÇö RyuJIT does not auto-vectorize this stride-2 read; explicit SIMD requires `MemoryMarshal.Cast<Complex, double>` and manual span arithmetic, (2) `TensorPrimitives.Multiply<double>(weights, -t, weights)` in-place, (3) `TensorPrimitives.Exp<double>(weights, weights)` in-place ÔÇö compiles because `double : IExponentialFunctions<double>`, dispatches through AVX/SSE on supported hardware. Steps 2 and 3 are SIMD; step 1 is the only non-SIMD step due to the `Complex Ôćĺ double` projection. When zero-mode exclusion combines with the heat filter, vectorized exp first, then `weights.MapIndexedInplace((i, w) => eigenValues[i].Real < ╬Á_zero ? 0.0 : w, Zeros.Include)` as one scalar mask pass ÔÇö `Vector<T>.MapIndexedInplace` takes `Func<int, T, T>` (two parameters: index, value); the matrix overload takes three.

[CONGRUENCE_REDUCTION]:
- The generalized problem A┬Ěx = ╬╗┬ĚB┬Ěx with B SPD reduces via `B = L┬ĚLßÁÇ` to `C = LÔü╗┬╣┬ĚA┬ĚLÔü╗ßÁÇ`, then `C┬Ěy = ╬╗┬Ěy`, then `x = LÔü╗ßÁÇ┬Ěy`. `ManagedLinearAlgebraProvider.CholeskyFactor(double[])` writes `a[i*order+j]` for `j > i` at each column `i`; in column-major layout this maps to row `j`, column `i` ÔÇö the lower triangle. The zeroing pass clears the upper triangle. `Cholesky<T>.Factor` is lower-triangular L, confirmed by `DoCholeskySolve`'s forward-substitution pass. `Cholesky.Solve(B)` dispatches to `DoCholeskySolve`, which performs `L┬Ěy = B` (forward) then `LßÁÇ┬Ěx = y` (backward) in one pass, returning `(L┬ĚLßÁÇ)Ôü╗┬╣┬ĚB` ÔÇö not `LÔü╗┬╣┬ĚB`. Any congruence step requiring `LÔü╗┬╣┬ĚA` alone must bypass `Cholesky.Solve` and use `chol.Factor.LU()` directly. Calling `chol.Factor.Transpose().Solve(A)` routes through `LU(LßÁÇ)` and solves `LßÁÇ┬ĚX = A`, yielding `LÔü╗ßÁÇ┬ĚA` ÔÇö the wrong half-factor.
- Correct two-step sequence: (1) `Y = chol.Factor.LU().Solve(A)` solves L┬ĚY = A, giving `Y = LÔü╗┬╣┬ĚA`; (2) `Z = chol.Factor.LU().Solve(Y.Transpose()).Transpose()` gives `Z = LÔü╗┬╣┬ĚA┬ĚLÔü╗ßÁÇ`. Each `.LU()` call allocates an n├Śn PLU struct; `.Transpose()` materializes an n├Śn copy ÔÇö three n├Śn allocations per congruence. `Matrix<T>.Solve` has no triangular-aware overload ÔÇö it always routes through `LUFactor`, which executes dense Crout inner products unconditionally with no zero-skipping: O(n┬│) regardless of whether the input is triangular. Back-transform `X = LÔü╗ßÁÇ┬ĚY`: `chol.Factor.Transpose().LU().Solve(evd.EigenVectors)` allocates a `Transpose()` copy (n├Śn), the LU factored copy (n├Śn), and the solve result (n├Śk). For k eigenvectors the total cost is two O(n┬│) LU factorizations plus O(n┬▓┬Ěk) back-substitution.
- Re-symmetrization before `Evd`: congruence asymmetry grows as O(n┬▓ ├Ś ╬Á_mach ├Ś ╬║(L)). For ill-conditioned B (╬║(B) Ôł╝ 10┬╣┬▓) the asymmetry reaches O(10Ôü╗Ôü┤) and the un-re-symmetrized C routes through the nonsymmetric Schur path, producing complex eigenvalues for a real symmetric-definite problem. Apply `(c + c.Transpose()) * 0.5` before passing with `Symmetricity.Symmetric`; this allocates three matrices but guarantees both `[i,j]` and `[j,i]` carry the same IEEE 754 bit pattern so the exact-equality `IsSymmetric()` check passes unconditionally.
- `DenseCholesky.Solve` throws `NotSupportedException("Can only do Cholesky factorization for dense matrices at the moment.")` when either input or result is not `DenseMatrix`. The type signature is `Matrix<double>` at the abstract level, so this fails at runtime, not compile time.
- `Cholesky` positive-definiteness detection checks strictly `num > 0.0`. Rank-deficient B (exactly zero diagonal) throws `ArgumentException("Matrix must be positive definite.")`. Near-singular B (very small positive diagonal) passes and produces an ill-conditioned factor; the correct guard is an ╬Á┬ĚI shift before factorization for near-singular B ÔÇö catching `ArgumentException` addresses only the indefinite case.
- Receipt route discrimination: `IndefiniteBFault` (Cholesky throws), `ExhaustionFault` (Cholesky succeeds but `Evd` on C exhausts), and success are three distinct receipt states. If Cholesky fails, no eigenpairs exist. If Evd fails, L is computed and valid; the receipt should carry L's condition evidence alongside the failure, because L's condition number diagnoses whether the problem is ill-conditioned B or a genuinely near-degenerate spectrum.

[RESIDUAL_EVIDENCE]:
- MathNet provides no built-in residual for eigendecomposition. Per-pair residual: `(A.Multiply(eigVec) - eigVec.Multiply(eigVal)).L2Norm()`. Block residual: `(A.Multiply(V) - V.Multiply(D)).FrobeniusNorm()` where D is built via `Matrix<T>.Build.DenseDiagonal(n, n, i => eigenvalues[i])`.
- A generic residual core with thin representation adapters: `double BlockResidual<TM>(TM a, Matrix<double> v, double[] eigenvalues, Func<TM, Matrix<double>, Matrix<double>> applyA)` where `applyA` is a thin adapter ÔÇö `(a, v) => ((SparseMatrix)a).Multiply(v)` for the sparse case, `(a, v) => a.Multiply(v)` for the dense case. The core computes `applyA(a, v) - v.Multiply(D)` and returns the Frobenius norm. A single generic computation path ÔÇö parameterized on the operator application ÔÇö replaces separate dense and sparse residual implementations. The adapter carries no domain logic: its entire content is the `Multiply` dispatch appropriate to the representation.
- The receipt shape for correctness must add: per-pair residual norms, the route variant (`dense-symmetric`, `dense-general`, `dense-hermitian`, `congruence-reduced`), and a sort-guarantee flag (symmetric: ascending; nonsymmetric: Schur-convergence order, no guarantee).
- The three convergence behaviors form a distinct trio: symmetric and Hermitian paths throw `NonConvergenceException` at 1000 steps per eigenvalue; the real nonsymmetric path (`NonsymmetricReduceHessenberToRealSchur`) throws `NonConvergenceException` at `iter >= 30 * order` where `iter` increments per outer step and resets to 0 on both the 1├Ś1 deflation branch (`n--; iter = 0`) and the 2├Ś2 deflation branch (`n -= 2; iter = 0`) ÔÇö the ceiling bounds consecutive non-deflating steps since the last deflation, not total steps across all sub-problems; the complex nonsymmetric path has no outer iteration bound and may not terminate under failing deflation. A receipt catching `NonConvergenceException` handles the first two paths. The complex nonsymmetric path requires `ÔÇľA┬ĚV Ôłĺ V┬ĚDÔÇľ_F` unconditionally as the only detection mechanism; the receipt must include this residual for complex nonsymmetric inputs regardless of whether an exception was thrown.
- `SymmetricDiagonalize` uses the Wilkinson relative convergence criterion: `|e[k]| ÔëĄ Precision.DoublePrecision ├Ś num2` where `num2` is a running spectral-radius estimate. Near-coincident eigenvalues (ratio Ôëł 1) are the pathological trigger for exhaustion. When `SymmetricDiagonalize` throws `NonConvergenceException` at iteration `num3 >= 1000` for eigenvalue index `j`, the `matrixEv` array has been partially updated: eigenvectors for converged indices `< j` are complete and orthonormal; those at and above `j` are intermediate Givens-rotation products ÔÇö numerically meaningless. The failing index is not exposed by the exception. The `ExhaustionFault` receipt should record the spectral radius estimate (`max(|d[k]| + |e[k]|)` at the point of failure) as the only available near-degeneracy diagnostic; this estimate must be captured before calling `Evd` since it is not accessible post-throw.

[CACHE_AND_TRUNCATION]:
- `Matrix<T>.Evd(Symmetricity.Symmetric)` always computes all n eigenpairs. For a filter using only the k smallest, over-compute-and-truncate: `evd.EigenVectors.SubMatrix(0, n, 0, k)` always allocates a new n├Śk backing array ÔÇö there is no zero-copy view because `DenseColumnMajorMatrixStorage` takes ownership. For the symmetric route, `DenseEvd.Create` allocates three arrays before calling `EigenDecomp`: the n├Śn eigenvector matrix (`eigenVectors`), the n├Śn block-diagonal matrix (`blockDiagonal`), and a length-n `DenseVector<Complex>` (`eigenValues`); inside `EigenDecomp` the symmetric path copies the input into `matrixEv` via `Buffer.BlockCopy`, so working memory is Ôëł 2n┬▓ + O(n) doubles (Ôëł 16 MB for n = 1000). The nonsymmetric path allocates a fourth n├Śn array `matrixH` for the Hessenberg reduction, bringing the working set to Ôëł 3n┬▓ + O(n) (Ôëł 24 MB for n = 1000). For cached basis reuse, retain the full `DenseMatrix` from `evd.EigenVectors`; each filter call pays one `SubMatrix` allocation, not the O(n┬│) eigensolver.
- The receipt rewritten to the truncated request carries `ComputedCount = n`, `RequestedCount = k`, and `CacheSource` distinguishing `Full` (truncated from a cached n-wide basis, O(n┬│) cost amortized) from `Fresh` (new full computation). A `Resident` vs `Reconstructed` annotation records whether the `Evd` object is retained in memory or recomputed, trading Ôëł 16 MB resident (symmetric) for n = 1000 against recomputation cost. A cache hit requires compatible n and `Symmetricity`; a mismatch on `Symmetricity` must recompute.

[DESCRIPTOR_NORMALIZATION_AND_DEGENERATE_SPREAD]:
- A descriptor normalized by `(╬╗_max Ôłĺ ╬╗_min)` requires a `DegenerateSpread` receipt case when `╬╗_max == ╬╗_min`. This is distinct from `Unusable` (non-converged pairs) and from success: degenerate spread means the filter has collapsed all frequencies into one bin and the filtered output carries zero information content, while `Unusable` pairs are finite diagnostic data explicitly flagged as structurally unreliable.
- Failing on degenerate spread is correct; clamping or silently continuing produces a descriptor with artificially uniform frequency content that is indistinguishable from a well-conditioned result. The `DegenerateSpread` case must be sealed in the receipt union alongside `Success`, `Unusable`, `ExhaustionFault`, and `IndefiniteBFault` ÔÇö not treated as a downstream validation concern ÔÇö because the spread condition is knowable only after the eigensolution is in hand and cannot be inferred from input properties before the solve.
