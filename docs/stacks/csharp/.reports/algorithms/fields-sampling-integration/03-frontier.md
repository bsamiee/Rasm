# Fields, Sampling, and Integration — Frontier

[SLICE_SAMPLER_MECHANICS]:
- `UnivariateSliceSampler` implements Neal (2003) stepping-out with no doubling or shrinkage budgets: the bracket `[num3, num4]` expands by `Scale` in each direction until the log-density falls below the slice level, then shrinks by retraction on rejection. There is no cap on bracket expansion steps — a nearly-flat log-density causes unbounded bracket growth proportional to `Scale` per iteration. For field-derived log-densities with long flat tails, `Scale` must be calibrated to the tail slope, not left at default.
- `BurnInterval` applies a constructor-side `Burn(BurnInterval)` call synchronously before any `Sample()` returns. The constructor is therefore side-effectful and stateful; `UnivariateSliceSampler` cannot be constructed lazily or pooled without re-burning. Unlike `HybridMCGeneric`, there is no `Samples` counter updated during the `Burn` call inside the constructor — only `Samples` inside `Sample()` increments the counter. `AcceptRate` computed externally before calling `Sample()` returns zero, not the burn-phase rate.
- `UnivariateSliceSampler.Scale` is mutable post-construction; changing it between `Sample()` calls alters the stepping-out interval without resetting chain state. This enables adaptive scale tuning during a warm-up phase without restarting the chain, but any scale change that widens the bracket can cause an immediate rejection cycle if the density has already tightened around the current location.

[MCMC_DIAGNOSTICS_SURFACE]:
- `MCMCDiagnostics.ACF<T>` computes autocorrelation at a single lag by materializing the series to an array and calling `Correlation.Pearson` on shifted windows. This is `O(N)` per lag but `O(N * maxLag)` for a lag ladder — there is no FFT path inside `MCMCDiagnostics`. For a full autocorrelation spectrum, `Correlation.Auto(double[])` uses `Fourier.Forward`/`Fourier.Inverse` with `AsymmetricScaling` and zero-padding to `CeilingToPowerOfTwo * 2`, delivering `O(N log N)` cost. The two are not composable: `MCMCDiagnostics.ACF` accepts `IEnumerable<T>` with a projection delegate, while `Correlation.Auto` requires `double[]` directly.
- `MCMCDiagnostics.EffectiveSize<T>` uses only lag-1 autocorrelation via the Geyer formula `(1 - ρ₁)/(1 + ρ₁) * N`. This is the first-order approximation; the full Geyer monotone sequence estimator sums positive initial sequence terms across all positive lags. The lag-1 approximation overestimates effective sample size when the chain has long-range positive autocorrelation — a common case for HMC chains with large step sizes.
- `MetropolisHastingsSampler<T>.Burn` computes `num4 = Math.Min(0.0, logP_new + logQ_reverse - logP_current - logQ_forward)` and accepts deterministically when `num4 == 0.0` (i.e. the Hastings ratio is exactly ≥ 1 in log space). Floating-point equality to exactly `0.0` fires only when all four log-density values cancel perfectly — in practice for floating-point densities this branch is unreachable, and `Bernoulli.Sample(RandomSource, Math.Exp(num4))` does all acceptance. The `== 0.0` branch is dead code for field log-densities.
- `RejectionSampler<T>.Sample` throws `ArgumentException` at runtime when `pdfQ(val) < pdfP(val)` for any sampled point. This is a design-time contract violation — the proposal envelope must dominate — but it surfaces as a runtime exception with no diagnostic information about which point violated the bound. A field-derived target density that has been modified (e.g., a truncated SDF-based density) must re-verify the envelope condition after every modification.

[QUANTILE_ESTIMATION_PIPELINE]:
- `SortedArrayStatistics.Quantile(data, tau)` hardcodes R-8 (SciPy `(1/3, 1/3)` parametrization): `h = (N + 1/3) * tau + 1/3`, interpolating between `data[h-1]` and `data[h]`. This is approximately median-unbiased but not mode-unbiased or population-quantile-unbiased. For sampling acceptance gates based on empirical quantile thresholds (e.g., discarding the bottom 5% of samples by a score field), R-8 is appropriate; for strict compatibility with a specific statistical software's quantile definition, `QuantileCustom(data, tau, QuantileDefinition)` is required.
- Nine quantile definitions are available as `QuantileDefinition` enum values with named aliases: `R7 = Excel = Mode = S`, `R8 = Median = Default`, `R6 = Weibull = Nist = SPSS`, `R5 = Hazen = Hydrology`. The `QuantileCustom(data, tau, a, b, c, d)` four-parameter overload follows Mathematica's parametrization directly: `h = a + (N + b)*tau`, interpolating via `c + d*(h - floor(h))`. This is the correct route for algorithmic compatibility with Mathematica's `Quantile[data, tau, {a,b,c,d}]`, not the enum variant.
- `QuantileFunc` extension methods (`IEnumerable<double>.QuantileCustomFunc(QuantileDefinition)`) materialize the data to a sorted array once and return a `Func<double, double>` that queries without re-sorting. For field-sampled distributions requiring many quantile probes (e.g., adaptive binning, percentile-based normalization), pre-materializing via `QuantileCustomFunc` amortizes the `O(N log N)` sort cost across all calls.

[FOURIER_FIELD_TRANSFORM_LAW]:
- `Fourier.ForwardMultiDim(Complex[] samples, int[] dimensions)` performs a multi-dimensional DFT over a flat row-major array using the default `SymmetricScaling` (`1/√N` in both forward and inverse). The `FourierOptions` enum controls the convention: `Default = SymmetricScaling`, `AsymmetricScaling` (Matlab convention: no scaling forward, `1/N` inverse), `NoScaling | InverseExponent` (Numerical Recipes: inverse exponent, no scaling either direction). For field power spectrum computation, the forward transform under `AsymmetricScaling` followed by pointwise `|X[k]|²` and an inverse transform is the standard convolution theorem path; swapping to `SymmetricScaling` incorrectly scales the result by `1/N`.
- `Fourier.Forward2D(Matrix<Complex> samples, FourierOptions)` extracts the matrix to a flat row-major `Complex[]` via `Storage.AsArray()` when the matrix is dense and column-major (`FortranArrayStorage`), requiring an internal copy. For field 2D FFT over row-major dense matrices, `Forward2D(Complex[] samplesRowWise, int rows, int columns)` avoids the copy. For column-major storage, the copy is unconditional.
- `Correlation.Auto(double[] x)` zero-mean-centers the data before the FFT: `array[i] = new Complex(x[i] - mean, 0)`. The returned autocorrelation values are normalized by `array2[0].Real` (the zero-lag variance). The field autocorrelation at lag zero is therefore always 1.0 by construction — not the empirical variance. Callers expecting an unnormalized autocorrelation (to extract variance separately) must multiply by the sample variance.
- `Correlation.Auto` zero-pads to `CeilingToPowerOfTwo * 2` which is always at least `2N` when N is a power of two, and more than `2N` otherwise. This guarantees linear (acyclic) convolution with no wrap-around aliasing. For field signals whose length is already a power of two, the padding doubles the array size unnecessarily; pre-padding to `2N` manually and calling `Forward` directly avoids the extra allocation but requires managing the zero-padding and mean-centering manually.

[SIMPSONRULE_AND_QUADRATIC_SPLINE_GAP]:
- `SimpsonRule.IntegrateComposite` requires `numberOfPartitions` to be even — passing an odd value throws `ArgumentException("Value must be even.")`. There is no automatic rounding; the caller bears full responsibility. In an adaptive field integration loop that bisects sub-intervals and accumulates partition counts, any bisection from an even count preserves evenness, but an initial odd-count call is a hard error with no recovery.
- `QuadraticSpline` is a full `IInterpolation` implementor with `SupportsDifferentiation: true` and `SupportsIntegration: true`. It is not constructable through the `Interpolate` facade — there is no `Interpolate.QuadraticSpline.*` factory. Construction requires direct instantiation: `new QuadraticSpline(x, c0, c1, c2)` with pre-computed coefficient arrays. The `Differentiate2(t)` method returns `2.0 * c2[segment]` — a piecewise constant second derivative across each segment, unlike `CubicSpline` where `Differentiate2` is piecewise linear. For smooth field curvature estimation, this piecewise-constant second derivative is a design limitation.
- `LogLinear` stores `log(y)` in a `LinearSpline` and exponentiates at evaluation time. `Differentiate(t)` is `Interpolate(t) * _spline.Differentiate(t)` — the chain rule through the log-space spline. `Differentiate2(t)` is `Differentiate(t) * logSlope + Interpolate(t) * logSpline.Differentiate2(t)` — composed from the linear spline's piecewise-constant second derivative and the first derivative. This means `Differentiate2` is piecewise-constant in log-space (zero second derivative of the log-spline), making the log-linear second derivative equal to `(logSlope)² * Interpolate(t)` within each segment. The `InterpolateSorted` factory parallelizes the `log` computation via `CommonParallel.For` with a 4096 chunk size.

[COMBINATORICS_AND_SAMPLING_WITHOUT_REPLACEMENT]:
- `Combinatorics.GenerateCombination(n, k, randomSource)` uses two paths based on density: when `k * 3 < n` (sparse), it fills a `HashSet<int>` with `k` random integers in `[0, n)` by repeated sampling; when `k * 3 >= n` (dense), it generates a full permutation of `n` indices and takes the first `k`. The sparse path has expected cost `O(k * n/(n-k))` due to collision retries as the set fills; for near-half selections (`k ≈ n/2`), the dense path triggers and costs `O(n)` regardless. The threshold `k * 3 < n` is a fixed constant — there is no tuning surface.
- `Combinatorics.SelectPermutationInplace<T>` implements Fisher-Yates (Durstenfeld) in-place shuffle. When called with `SystemRandomSource.Default`, the shuffle is non-deterministic across runs. Passing a seeded `MersenneTwister` produces a deterministic permutation sequence that is serializable via `[DataContract]` state. For reproducible field sampling permutations (farthest-point traversal order, stratified sample selection), this is the correct route; the inline `foreach` variant `SelectPermutation` (LINQ) uses a copy and yields lazily, avoiding the full materialization when only a prefix of the permutation is needed.
- `Combinatorics.GenerateVariation(BigInteger n, int k, randomSource)` handles large-cardinality sets: it samples `k` distinct indices from `[0, n)` using `RandomExtensions.NextFullRangeInt64` chained via `BigInteger` arithmetic. `NextFullRangeInt64(rnd)` constructs a 64-bit integer from two `NextFullRangeInt32` calls (via two `_crypto.GetBytes(4)` in `CryptoRandomSource`), then multiplies and shifts to map into `[0, n)`. For field sampling from large discrete spaces (voxel grids beyond `int.MaxValue`), this is the only correct path; the non-BigInteger `GenerateVariation(int n, k)` silently overflows when `n > 2^31 - 1`.

[RUNNING_STATISTICS_PARALLEL_MERGE]:
- `RunningStatistics.Combine(a, b)` implements Chan's parallel online algorithm for combined mean, variance, skewness, and kurtosis using the delta between group means. The fourth-moment formula accumulates `m4_combined = m4_a + m4_b + δ⁴ * nA * nB * (nA² - nA*nB + nB²) / n³ + 6δ² * (nA² * m2_b + nB² * m2_a) / n² + 4δ * (nA * m3_b - nB * m3_a) / n`. This is exact (not approximate) for finite-precision floating-point when N is small; numerical instability accumulates with large N and large δ. `operator+` is sugar over `Combine`.
- `RunningStatistics` is `[DataContract]` serializable with seven `[DataMember]` fields. For distributed field sampling pipelines, each worker constructs a local `RunningStatistics`, serializes via `DataContractSerializer` or Protocol Buffers, and the aggregator calls `Combine` on all partial results. The data contract carries no version compatibility guarantee for durable persistence across package upgrades.
- `RunningStatistics.Push(double value)` propagates `NaN`: `if (value < _min || double.IsNaN(value))` unconditionally sets `_min = value`. A single `NaN` in the field sample stream poisons `Minimum`, `Maximum`, and all moment estimates permanently. The only recovery is constructing a new instance; there is no `Reset()` method.

[TENSOR_OPERATION_PUBLIC_INJECTION_SURFACE]:
- `TensorOperation.Invoke<TOperation, TArg, TResult>(in ReadOnlyTensorSpan<TArg> x, in TensorSpan<TResult> destination)` is the public N-dimensional dispatch for user-defined `IUnaryOperation_Tensor<TArg, TResult>` implementations. The loop iterates `FlattenedLength` times using `AdjustToNextIndex` stride arithmetic — not vectorized at this level. SIMD vectorization is only present inside `TensorPrimitives`' private `InvokeSpanIntoSpan` dispatcher, which is inaccessible to user code. Custom `IUnaryOperation_Tensor` implementations running through `TensorOperation.Invoke` are scalar-only; the SIMD path is locked behind `internal` `IUnaryOperator<T,T>`.
- `TensorPrimitives.SoftMax(ReadOnlySpan<float>, Span<float>)` does not subtract the maximum before exponentiation: it directly calls `Exp` over the input span, then divides by `Sum`. This is numerically unstable for inputs with large magnitudes (overflow to `Infinity` when `x[i] > ~88` for `float`). The numerically stable log-sum-exp-stabilized softmax requires manual pre-subtraction of `TensorPrimitives.Max(x)` before calling `Exp`, then dividing. For field-derived energy distributions with unscaled magnitudes, the naive `SoftMax` call produces `NaN`/`Infinity` results without error.
- `TensorPrimitives.IUnaryOperator<TInput, TOutput>` is `internal`. The public custom-operator path is `TensorOperation.IUnaryOperation_Tensor<T, TResult>` with its `static abstract void Invoke(ref readonly T x, ref TResult destination)` and `static abstract void Invoke(ReadOnlySpan<T> x, Span<TResult> destination)` members. The span overload is the hot path; it does not dispatch through SIMD lanes. For SIMD-accelerated custom field kernels, the implementation must call `TensorPrimitives.*` primitives inside the span overload body rather than hoping the `TensorOperation.Invoke` dispatch will vectorize the loop.

[DISTANCE_METRIC_SURFACE_AND_GAPS]:
- `Distance.SSD(double[] a, double[] b)` dispatches to `LinearAlgebraControl.Provider.SubtractArrays` followed by `LinearAlgebraControl.Provider.DotProduct` — these are BLAS-backed when a native provider is loaded. The `Distance.Euclidean(double[] a, double[] b)` overload wraps `Math.Sqrt(SSD(a, b))` without scaled-accumulation overflow protection. For field comparison between two sample vectors with components above `√(Double.MaxValue / N)`, the intermediate squared sum overflows silently.
- `Distance.Canberra(double[] a, double[] b)` computes `Σ|a-b|/(|a|+|b|)` with a scalar loop, not BLAS-backed. The denominator is zero when both `a[i] == 0` and `b[i] == 0` simultaneously; the implementation returns `NaN` for that term (IEEE `0/0`), and the `NaN` propagates into the sum. For sparse field vectors with matching zero components, callers must guard the denominator.
- `Distance.Cosine(double[] a, double[] b)` calls `LinearAlgebraControl.Provider.DotProduct` three times: once for `a·b`, once for `a·a`, once for `b·b`. The `1 - dot/sqrt(a·a * b·b)` formula does not handle near-zero or zero vectors — `sqrt(0 * ...)` produces `0/0 = NaN`. The `Distance.Pearson` path delegates to `Correlation.Pearson` which uses online two-pass Welford-style computation and returns `NaN` for constant series, making it the safer route for field sample correlation where any dimension may be constant.

[AKIMA_AND_MAKIMA_WEIGHT_DIFFERENCE]:
- `CubicSpline.InterpolateAkimaSorted` assigns zero weight to both neighbors when `|Δm_{i+1} - Δm_{i-1}| = 0` AND `|Δm_i - Δm_{i-2}| = 0` (both differences vanish simultaneously). The fallback in this case is a harmonic mean of neighboring slopes weighted by interval widths: `((x[k+1]-x[k])*m[k-1] + (x[k]-x[k-1])*m[k]) / (x[k+1]-x[k-1])`. This is the classic Akima degenerate case: equal slopes on both sides of a node cause the spline to pass through with a linearly interpolated tangent, not a zero tangent. Callers expecting zero gradient at inflection points of smooth fields should use `InterpolatePchipSorted` instead.
- `CubicSpline.InterpolateMakimaSorted` computes weights `array5[l] = |Δm_{l+2} - Δm_{l+1}| + 0.5 * |Δm_{l+2} + Δm_{l+1}|` and `array6[l] = |Δm_{l} - Δm_{l-1}| + 0.5 * |Δm_{l} + Δm_{l-1}|`. The `0.5 * |sum|` term prevents zero weights when opposing slopes cancel — this is the MAKIMA fix. When the total weight `array7[m] = array5[m] + array6[m]` is zero (all four surrounding differences vanish, i.e., four consecutive equal slopes), the implementation returns `0.5 * (m[n+3] + m[n])` — the arithmetic mean of the extended-boundary slopes. MAKIMA therefore never produces zero-weight artifacts at constant-slope regions; Akima does.
- `CubicSpline.InterpolatePchipSorted` zeros the slope at monotone-sign-change intervals. The interior slope formula is the harmonic mean `(h0 + h1) * h0 / ((2*h1+h0)/m[j-1] + (h1+2*h0)/m[j])` (where `h` are interval widths), evaluated only when adjacent slopes have the same sign. This is the standard PCHIP shape-preserving rule — it guarantees C¹ continuity and monotonicity preservation within each monotone segment. `PchipEndPoints` applies `|endpoint_slope| ≤ 3 * |first_interval_slope|` clamping, not a harmonic mean, to prevent extrapolation overshoot at the domain boundary.
