# Carrier Core

[CARRIER_SELECTION]:
- `Fin<A>` is the concrete result carrier for computations that either succeed or fail with a typed `Error`. It is structurally `Either<Error,A>` but with the fail-side fixed to the `Error` hierarchy, making it the narrowest carrier when the failure domain is exception-capable and must compose with `Eff`/`IO` lifts. `Option<A>` is the absence carrier and carries zero failure semantics — it answers "present or not" with no failure-side information. `Validation<F,A>` is wider than `Fin` because it accumulates failures across independent checks using the applicative surface rather than short-circuiting; it is the correct choice when all failure paths must be collected before reporting. `Either<L,R>` is the most general form: the fail-side is unrestricted, enabling it to span protocol errors, domain codes, or any non-`Error` type as the left; it requires an explicit `L -> Error` bridge before entering an `Eff` chain.
- Narrowest-carrier law: admit `Option` when absence is the only distinguishing fact; promote to `Fin` when the caller must know why it failed; promote to `Validation` only when independent failures must accumulate; stay at `Either` only when the left type is not `Error` and no `Eff`-level convergence is needed on that path.
- `Fin<A>` has no `ToEither`, `ToOption`, `ToValidation`, `IfFail`, or `IfSucc` on the concrete type itself. Cross-rail projections are provided as extension methods: `FinExtensions.ToFin(Either<Error,A>)` converts inbound `Either`; `ValidationExtensions.ToFin(Validation<Error,A>)` converts accumulated validation results; `Option<A>.ToFin()` produces `Fin<A>` with `Errors.None` on `None`, while `Option<A>.ToFin(Error)` takes an explicit error for the `None` branch. There is no `Fin<A>.ToOption` — the projection is loss-inducing and is not provided to prevent silent failure-side erasure.

[FIN_ALGEBRA]:
- `Fin<A>` exposes two discriminated cases as public nested record types: `Fin<A>.Succ` carrying `.Value: A` and `Fin<A>.Fail` carrying `.Error: Error`. Pattern matching uses these directly: `{ IsSucc: true }` is the flat-match form using the public abstract `IsSucc: bool`; `case Fin<A>.Succ s` gives `s.Value` as the public positional record field. `SuccValue` and `FailValue` are also public on both the base type and each nested case but are marked unsafe — they do not guard and will throw on the wrong case.
- Construction: `Fin.Succ<A>(A)` and `Fin.Fail<A>(Error)` are static methods on the non-generic `Fin` class; `Prelude.FinSucc<A>(A)` and `Prelude.FinFail<A>(Error)` are the Prelude-imported equivalents. `Fin.Fail<A>(string)` is a convenience overload that wraps the string in an `Expected` error. There are no implicit conversions from `A` or `Error` to `Fin<A>` on the type itself — construction is always explicit.
- The failure-side algebra on `Fin<A>`: `MapFail(Error -> Error)` maps the failure without changing the success type; `BindFail(Error -> Fin<A>)` chains a recovery that may succeed or remain failed; `BiBind<B>(A -> Fin<B>, Error -> Fin<B>)` branches on both cases and converges to one return rail; `BiMap<B>(A -> B, Error -> Error)` maps both sides but keeps the fail-side pinned to `Error` — the fail transform cannot change the error type.
- `Fin<A>.Empty` is the `MonoidK` empty — `Fin<A>.Fail(Errors.None)` — which enables `Fin<A>` as an `Alternative` through `op_BitwiseOr`. The `|` operator (`op_BitwiseOr`) enables short-circuit alternative selection: the left is returned if it is `Succ`; the right is evaluated otherwise. `op_True`/`op_False` back `||` coherence but are not independently useful — imperative boolean-style access is rejected.
- `Fin<A>` has `Traverse<F,B>(A -> K<F,B>) -> K<F,Fin<B>>` and `TraverseM<M,B>(A -> K<M,B>) -> K<M,Fin<B>>` for effect sequencing. `FinExtensions.Partition<F,A>(K<F,Fin<A>>)` splits any foldable of `Fin<A>` into `(Seq<Error> Fails, Seq<A> Succs)`; `Succs` and `Fails` project either side alone. The partition surface requires a wrapping foldable; there is no `Fin<A>.Partition()` instance method.

[OPTION_ALGEBRA]:
- `Option<A>` has `op_Implicit(A)` → `Option<A>` and `op_Implicit(Fail<Unit>)` → `None`, which means an `A` value is lifted automatically and `Option<A>.None` satisfies an assignment. `Case` is the pattern-matching property: `Some(a)` yields `a`; `None` yields `null`. `{ IsSome: true, Case: var v }` is the CSP-compliant pattern for inline extraction.
- `Prelude.Some<A>(A)` throws `ValueIsNullException` if passed null — it is the strict factory asserting non-null. `Prelude.Optional<A>(A)` accepts null and folds to `None` — it is the boundary factory for nullable interop.
- `Option<A>` projections: `ToFin()` → `Fin<A>` with `Errors.None` on `None`; `ToFin(Error)` → `Fin<A>` with a caller-supplied error on `None`; `ToEither<L>(L)`, `ToEither<L>()`, `ToEither<L>(Func<L>)` — three overloads for the `None` left value; `ToValidation<F>(F)`, `ToValidation<F>()`, `ToValidation<F>(Func<F>)` — matching three overloads.
- `Option<A>.IfNone(A)`, `IfNone(Func<A>)`, `IfNone(Action)` and `IfSome(Action<A>)`, `IfSome(Func<A,Unit>)` are the terminal operators at the boundary where `Option` exits the rail. These are the only points where `None` produces a default — never use `.Value` directly.
- `OptionExtensions.Somes<A>(IEnumerable<Option<A>>)` and `Somes<A>(Seq<Option<A>>)` extract all `Some` values in order — the batch partition operator. There is no `Nones` extractor because `None` carries no payload.

[VALIDATION_ALGEBRA]:
- `Validation<F,A>` carries the constraint that `F` satisfies `Semigroup<F>` for error accumulation to function correctly. `Error` satisfies this constraint through its `Combine(Error)` / `op_Addition(Error, Error)` semigroup surface. `Seq<Error>` is an alternative lawful `F`.
- Error accumulation happens through the **applicative** surface, not through `Bind`. `Bind(A -> Validation<F,B>)` is sequential and fail-fast — it drops the current success and propagates the first failure. `ValidationExtensions.Apply<F,A,B>(K<Validation<F>,Func<A,B>>, K<Validation<F>,A>)` accumulates failures from both sides using the semigroup's combine. The `ApplyI` variant accepts an explicit `Option<SemigroupInstance<F>>` for types that do not implement the trait directly; when `None` is passed, it degrades silently to fail-first rather than accumulated behavior. A LINQ query expression on `Validation` is always sequential — accumulation requires explicit `Apply` calls.
- `CombineI<F,A>(Validation<F,A>, Validation<F,A>, Option<SemigroupInstance<F>>)` collects results: when any are `Fail`, their failures are combined and returned; when all pass, success values are collected into a `Seq`. `CombineFirst` shares this signature but returns the first success value rather than a sequence — suitable for pair-wise combining where only the first success is the goal. A third `CombineI` overload takes `Validation<F, Seq<A>>` as `lhs` and appends `Validation<F,A>` to it, which is the fold-friendly form for accumulating over an `IEnumerable<Validation<F,A>>` via `Seq.fold`.
- `Validation<F,A>.MapFail<F1>(F -> F1) -> Validation<F1,A>` changes the failure type, unlike `Fin` where the fail side is pinned to `Error`. This makes `Validation` the correct choice when failure types must be structurally transformed before convergence at a boundary.
- `ValidationExtensions.ToFin<A>(Validation<Error,A>)` is constrained to `F = Error` — it is the convergence point after accumulation where the result enters a sequential `Eff`/`IO` chain. `Validation<F,A>.ToOption` drops all failure information — only valid when the failure set is irrelevant to the caller.
- `Validation<F,A>` has `SelectMany` (enabling LINQ query syntax) and `Guard` binding via `ValidationGuardExtensions`. `ValidationExtensions.Fails<F,S>(IEnumerable<Validation<F,S>>)` and `Fails<F,S>(Seq<Validation<F,S>>)` extract failure values as a flat projection; there is no symmetric `Succs` on `Validation` through this surface.

[ERROR_HIERARCHY]:
- The `Error` abstract class has three concrete subtypes: `Expected` (code, message, optional inner `Error`), `Exceptional` (wraps a live `Exception`), `ManyErrors` (a `Seq<Error>` aggregate). A fourth, `BottomError`, represents unevaluable expressions — it is `IsExceptional = true` and `IsExpected = false`. These four cases are the closed taxonomy.
- `Error.New(string)` → `Expected`; `Error.New(Exception)` → `Exceptional`; `Error.New(string, Exception)` → `Exceptional` with message override for display sanitization; `Error.New(string, Error)` and `Error.New(int, string, Error)` compose causal chains by embedding an inner `Error`. `Error.Many(Error[])` and `Error.Many(Seq<Error>)` produce `ManyErrors` directly. `WrappedErrorExpectedException` and `WrappedErrorExceptionalException` round-trip back to their originating `Error` when passed to `Error.New(Exception)`.
- `Error.Combine(Error)` / `op_Addition` appends two errors: combining two scalar errors produces `ManyErrors(Seq(this, other))`; combining two `ManyErrors` flattens their sequences; combining a scalar with a `ManyErrors` prepends or appends the scalar into the sequence. `Errors.None` is the monoid identity — combining with `Errors.None` returns the other value unchanged.
- `Error.IsExceptional` and `Error.IsExpected` are the structural discriminants. `Error.HasException<E>()` tests whether any nested `Exceptional` wraps a live exception of type `E` — it returns `false` for deserialized `Exceptional` instances whose live value has been lost. `Error.HasCode(int)` walks the full nested structure including `ManyErrors` members. `Error.IsType<E>()` tests whether any error in the structure is of concrete type `E`. `Error.Filter<E>()` projects out only errors of type `E`, returning `Errors.None` when none match.
- `Error.Inner` is the causal chain (type not documented as `Option<Error>` — access via property). `Error.Exception` is non-null only on `Exceptional` cases. `Error.Head` / `Error.Tail` decompose `ManyErrors` in list order; for a scalar `Error`, `Head` returns self and `Tail` returns `Errors.None`. `Error.IsEmpty` is `true` for `Errors.None` (zero-member `ManyErrors`).
- `Error.FoldM<M,A>(Error -> K<M,A>) -> K<M,A>` invokes `f` over each atomic error and combines via `M`'s monoid — processing continues even on sub-failure depending on `M`. `Error.ForAllM<M,A>(Error -> K<M,A>)` short-circuits on the first failure; it is the sequential variant.
- `Errors` static class: `Errors.None` is the empty error (an error state without any error values); `Errors.Bottom` is the `BottomError` singleton; `Errors.Cancelled`, `Errors.TimedOut`, `Errors.SequenceEmpty`, `Errors.ValidationFailed`, and `Errors.EndOfStream` are canonical domain codes. `ErrorCodes` mirrors these as integer constants for `HasCode` predicate matching.
- `Error.ToErrorException()` → `ErrorException` (an `Exception` derivative) for crossing the exception boundary; `Error.Throw()` / `Error.Throw<T>()` escape the rail into exception flow — permitted only at host boundary adapters, never inside domain logic.

[GUARD_INGRESS]:
- `Prelude.guard(bool, Error)` and `Prelude.guard(bool, Func<Error>)` return `Guard<Error,Unit>` — a typed conditional that participates in query expressions via `FinGuardExtensions.SelectMany`, `EitherGuardExtensions.SelectMany`, and `ValidationGuardExtensions.SelectMany`. A `guard` in a `from`/`where` clause is the only idiomatic ingress form for boolean invariants at the top of a pipeline.
- `Prelude.guard<F>(bool)` is the `MonoidK`/`Alternative` form: it returns `Applicative.pure(())` on `true` and `Applicative.empty` on `false`. This overload applies to any monad with an `Alternative` instance — `Option`, `Seq`, `Lst` — and does not carry an error payload.
- `Prelude.guardnot(bool, Error)` is `guard(!bool, Error)` — the negation shorthand. Both forms generate `Guard<Error,Unit>` and convert to any rail via the appropriate guard-extension `ToFin`, `ToEither`, `ToValidation`, or `ToEff`.
- `Guard<F,A>` is documented as "used by various error-producing monads to have a contextual `where`" — it is never a return type; it is always immediately bound into the surrounding monad through a `SelectMany` overload and exits as the destination monad's type.
- `Prelude.Optional<A>(A)` and `Prelude.Optional<A>(Nullable<A>)` are the `null`-tolerant ingress for `Option<A>`. `Prelude.Some<A>(A)` is the strict ingress that enforces non-null. Neither produces any other carrier type — conversion to `Fin`/`Either`/`Validation` happens through the projection methods on `Option<A>` after construction.

[CROSS_RAIL_PROJECTIONS]:
- The full projection matrix (source → available targets): `Either<Error,A>` → `Fin<A>` via `FinExtensions.ToFin`. `Option<A>` → `Fin<A>` (two overloads, `Errors.None` default or explicit). `Option<A>` → `Either<L>` (three overloads). `Option<A>` → `Validation<F>` (three overloads). `Validation<Error,A>` → `Fin<A>` via `ValidationExtensions.ToFin`. `Validation<F,A>` → `Option<A>` via `.ToOption` (discards failures). `Either<L,R>` → `Option<R>` via `.ToOption`. `Either<L,R>` → `Eff<R>` via `.ToEff(L -> Error)`.
- Projectability asymmetry: every rail can project to `Option` by discarding the failure side. No rail can project to a wider rail without supplying the missing structure (`Option → Fin` needs an `Error`; `Either → Eff` needs an `L → Error` map). `Fin → Validation` does not exist as a direct projection because they differ structurally: `Fin` is sequential, `Validation` is applicative — callers build `Validation` from `Fin` values via repeated `ToValidation` at the collection boundary, not via a single promotion.
- Pattern matching on `Option<A>` uses `Case` (returns `object` — `A` on `Some`, `null` on `None`); `{ IsSome: true, Case: var v }` is the canonical form. Pattern matching on `Fin<A>` uses the public nested types `Fin<A>.Succ` and `Fin<A>.Fail`; `{ IsSucc: true }` is the safe flat-match form using public `IsSucc: bool`. `Validation<F,A>` uses `{ IsSuccess: true, SuccessValue: var v }`. `Either<L,R>` uses `.Right` / `.Left` nested types with `{ IsRight: true }` guards.
