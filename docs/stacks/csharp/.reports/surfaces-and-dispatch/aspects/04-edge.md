# Aspect Architecture — Edge

[ENVALGO_ISOLATION_MATRIX]:
- `EnvIO` carries four independent axes: `Resources` (child resource scope), `CancellationTokenSource` (owns the token), `CancellationToken` (the observable signal), and `SynchronizationContext` (execution continuations). Four named `EnvIO` properties slice this matrix. `LocalResources` creates a new `Resources` scope; `Source` and `SyncContext` propagate unchanged. `LocalCancel` creates a new linked `CancellationTokenSource`; `Resources` and `SyncContext` propagate unchanged. `LocalSyncContext` captures `SynchronizationContext.Current`; all other axes propagate unchanged. `Local` (the property on `EnvIO`) creates both a new `Resources` scope and a new linked `CancellationTokenSource` while capturing `SynchronizationContext.Current`.
- `IO<A>.Local()` calls `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — it passes the existing `Resources` scope, not a new one. `EnvIO.New` with a null `source` creates a fresh `CancellationTokenSource` and, when the incoming token can be cancelled, registers `env.Token.Register(src.Cancel)`. The result is a new linked CTS sharing the parent resources. Local cancellation via `IO.cancel` produces `Errors.Cancelled` that propagates upward past the local boundary; `Local()` does not catch cancellation errors.
- `IO<A>.Uninterruptible()` wraps in `IOUninterruptible`, which executes against `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — existing resources are preserved, and `CancellationToken.None` causes `EnvIO.New` to create a fresh `CancellationTokenSource` with no registration to the parent token. The outer token's cancellation does not reach the wrapped computation. The error rail (`Fin`/`Error`) propagates normally; only the `CancellationToken` path is severed.
- `IO<A>.Timeout(TimeSpan)` wraps in `IOTimeout`, whose `Invoke` calls `env.LocalCancelWithTimeout(TimeLimit)`. `EnvIO.LocalCancelWithTimeout(timeout)` is `New(Resources, Token, null, SyncContext, timeout)` — it creates a new `CancellationTokenSource(timeout)` linked to the parent while preserving existing `Resources` and `SyncContext`. When the timeout fires, the linked CTS cancels, producing `Errors.TimedOut` (code `−2000000002`). `Errors.Cancelled` (code `−2000000001`) is what `IO.cancel` produces. `retryWhile` predicates that must distinguish timeout from explicit cancel must compare against these code constants.
- `IO<A>.Fork(Option<TimeSpan> timeout)` creates a child `Resources` scope (`new Resources(parentResources)`) disposed when the fork completes, and a new `CancellationTokenSource` linked to the parent via `env.Token.Register(tsrc.Cancel)`. Resources acquired in the parent before forking are visible inside the fork as shared parent-owned resources and are not released on fork completion. Resources acquired inside the fork are owned by the fork's child scope and are released on fork completion regardless of success or failure. The `ForkIO<A>` result is `readonly record struct(IO<Unit> Cancel, IO<A> Await)`.

[DEFINITION_TIME_DISPATCH_SURFACE_DEPTH]:
- `[UnionSwitchMapOverloadAttribute]` (`AllowMultiple = true`) on a Regular Union class declares additional `Switch`/`Map` overload families that treat a specified set of types as opaque arms rather than descending into their subtypes. `StopAt = new[] { typeof(FailureCase) }` generates a dispatch overload treating `FailureCase` and its siblings as atomic arms — coarser-grained dispatch alongside the full-resolution overload. Both overloads coexist on the same union type. This attribute applies only to Regular Unions; the ad-hoc `[Union<T1,...>]` form does not expose `NestedUnionParameterNameGeneration`.
- `NestedUnionParameterNameGeneration.Simple` uses only the final leaf type name as the Switch/Map parameter name. If two nested union families under the same root share a leaf name, the generator produces colliding parameter names and the C# compiler emits a duplicate-parameter error. `Default` guarantees uniqueness at the cost of compound names; renaming an intermediate union type changes every compound parameter name it prefixes. Once `Simple` is set with stable leaf names, renaming an intermediate type does not break call sites.
- `[IgnoreMemberAttribute]` marks a property or field invisible to the source generator, excluding it from equality, hash, serialization, comparison, and Switch/Map generation. The ignored member must be immutable — a mutable ignored member breaks the value-object contract: two instances with equal keys but different mutable-ignored-field values would compare equal while being observably different.
- `[MemberEqualityComparerAttribute<TComparer, TMember>]` applies to individual properties or fields of a `[ComplexValueObject]`, distinct from `[KeyMemberEqualityComparerAttribute<TComparer, TKey>]` (which applies to the single key member of a simple `[ValueObject<T>]`). `TComparer` must implement `IEqualityComparerAccessor<TMember>`, which is how the generator wires the comparer without runtime reflection.
- `SkipKeyMember = true` on `[ValueObject<TKey>]` suppresses generation of the key member, requiring a hand-written member at the name given by `KeyMemberName`. The generator still routes serialization, comparison, and parsing aspects through the declared key name. The use case is value objects with a custom backing representation — for example, a struct storing multiple native fields but exposing a computed string key for identity — where the generated backing field would be incorrect but all generated aspects must still route through the key name.
- `CreateFactoryMethodName` and `TryCreateFactoryMethodName` on `ValueObjectAttributeBase` override the generated factory method names from `Create`/`TryCreate` to any valid identifier. The full generated factory body — including `ValidateFactoryArguments` hook wiring, null/empty yield-null handling, and serialization routing — is preserved under the new name.

[FACTORY_ROUTING_ASPECTS]:
- `ObjectFactoryAttribute<T>` carries `where T : allows ref struct` (C# 13 anti-constraint), enabling `T = ReadOnlySpan<char>` as the factory input type for zero-allocation deserialization on .NET 9+. The anti-constraint does not require `T` to be a ref struct; it permits any ref struct, including custom `ref struct` admission types.
- `UseForSerialization` on `ObjectFactoryAttribute` is typed as `SerializationFrameworks` (a `[Flags]` enum: `SystemTextJson = 1`, `NewtonsoftJson = 2`, `MessagePack = 4`). A single type can carry multiple `[ObjectFactory<T>]` attributes with different `UseForSerialization` values, routing each serialization framework through a different factory. For example, `[ObjectFactory<ReadOnlySpan<char>>(UseForSerialization = SerializationFrameworks.SystemTextJson)]` routes only STJ through the span factory while Newtonsoft routes through the primary string-based factory.
- `UseForModelBinding` and `HasCorrespondingConstructor` on `ObjectFactoryAttribute` are `init`-only properties — they must be set in the attribute's object initializer at the declaration site and cannot be assigned later. This is an immutability enforcement on the aspect payload itself.

[COMPOSITION_TIME_CANCELLATION_ASPECTS]:
- To contain a cancellation error and substitute a fallback value, compose `local(ma) | @catch(Errors.Cancelled, _ => fallback)`, with the catch applied outside the `local` scope. `Local()` does not catch cancellation errors — it merely owns the CTS for cleanup.
- `Schedule.PrependZero` is a property: `this.Prepend(Duration.Zero)`. `RetryUntil` and `RepeatUntil` unconditionally call `schedule.PrependZero.Run().GetIterator()`, making the first attempt immediate regardless of the schedule's natural first step. Composing `NoDelayOnFirst` on top of a schedule already passed to `RetryUntil` is idempotent — it produces a double-zero prefix that does not create extra retries.
- `IO<A>.Try()` wraps the computation into `FinT<IO, A>` by mapping success through `Fin.Succ` and catching all errors through `Fin.Fail<A>`. The resulting `FinT<IO, A>` insulates the error rail inside the `IO` monad — `Catch` combinators applied to the outer `K<IO, Fin<A>>` computation see `Fin.Fail` values as bound values, not as propagated `Error` rails. Use `Try()` when both success and failure paths must be inspected as values rather than as rail continuations.
- `IO<A>.Combine(IO<A> rhs)` catches every error from `this` (predicate `_ => true`) and substitutes `rhs`. `IO<A>.Empty` is the monoid identity — a computation that always fails with an `IOEmpty` error. The semantics are alternative resolution: `first | second | third` via `|` on the `Fallible` extension block picks the first non-failing computation. Errors from `rhs` propagate normally because no catch wraps `rhs`.
- `IO<A>.Post()` uses `env.SyncContext` captured in `EnvIO` at chain construction time, not the thread's current `SynchronizationContext` at the moment `Post()` executes. If `env.SyncContext` is `null`, `Post()` is a no-op. Wrapping a computation with `Post()` when `EnvIO` was constructed on a UI thread marshals the computation back to the UI thread regardless of which thread ran the preceding chain steps.

[CRON_CALENDAR_SCHEDULE_ASPECTS]:
- `secondOfMinute(int)`, `minuteOfHour(int)`, `hourOfDay(int)`, and `dayOfWeek(DayOfWeek)` each accept an optional `Func<DateTime>? currentTimeFn` clock injection parameter (defaulting to `() => DateTime.Now`). These generate window-aligned schedules: `secondOfMinute(30)` fires at second 30 of each minute, computing the next delay as the remaining time until the next occurrence of second 30. All four are composable via `|`/`&`/`+` with other schedules. Integer arguments for `secondOfMinute` and `minuteOfHour` are clamped to `[0, 59]`; `hourOfDay` clamps to `[0, 23]`.
- `Schedule.windowed(interval)` and `Schedule.fixedInterval(interval)` are the two clock-based non-calendar schedules. `windowed` aligns to epoch multiples of `interval`: next delay = `interval − (elapsed % interval)`. `fixedInterval` floors the next delay at zero when the action duration exceeds the interval. Both accept an optional clock function. `upto(max, currentTimeFn)` terminates after wall-clock elapsed time exceeds `max`, which diverges from `maxCumulativeDelay(max)` when action duration is non-negligible.

[OPERATOR_CASCADE_CONSTRAINTS]:
- `EqualityComparisonOperators` on `ValueObjectAttribute<T>` and `SmartEnumAttribute<TKey>` is coerced upward to match `ComparisonOperators` when `ComparisonOperators` is set higher. The getter computes `max(ComparisonOperators, EqualityComparisonOperators)` because comparison operators require equality operators in generated code. Setting `ComparisonOperators = DefaultWithKeyTypeOverloads` without explicitly setting `EqualityComparisonOperators` silently forces `EqualityComparisonOperators` to `DefaultWithKeyTypeOverloads` as well. The reverse coercion does not apply.
- `SkipEqualityComparison = true` is a dominating flag: `EqualityComparisonOperators` and `ComparisonOperators` getters return `OperatorsGeneration.None` regardless of their explicit assignments. On a struct value object, suppressing equality prevents the generated `default(T)` guard from being hashable, blocking use in `FrozenDictionary` and `FrozenSet` unless hand-written `Equals`/`GetHashCode` replace the suppressed generated versions.
- `SkipFactoryMethods = true` cascades across five aspects simultaneously: `TypeConverter` attribute is not emitted; `IObjectFactory<T>` is not implemented; the conversion operator from key type to value object is not generated; `SkipIParsable` and `SkipISpanParsable` are forced to `true`; arithmetic operators (`AdditionOperators`, `SubtractionOperators`, `MultiplyOperators`, `DivisionOperators`) are all forced to `None`. Serialization converters are suppressed unless an `[ObjectFactory<T>]` with `UseForSerialization` is present. Setting any of the cascaded properties individually does not set `SkipFactoryMethods`.
