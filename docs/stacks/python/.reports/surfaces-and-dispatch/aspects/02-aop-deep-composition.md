[STAMINA_FULL_SURFACE]:

Custom `RetryHookFactory` instances are the correct hook form whenever the hook itself needs library state (a logger, a metric counter, a tracer) that must not be resolved at module import time. [PARAMETERIZED_ASPECT_FACTORY]:
- A `Concern` ADT that owns N cross-cutting policies enables one factory to build any combination without combinatorial wrapper proliferation. The deterministic ordering guarantee requires that each concern be a distinct layer in the closure chain rather than a list iterated at call time: `beartype` innermost (modifies the callable's contract) -> `cache` outside it (reads validated values) -> observability span (encloses cache miss + hit) -> retry (encloses all failures including cache-miss failures) -> `validate_call` outermost (coerces raw ingress before the rail). Building this chain requires wrapping in reverse concern order: `apply_concerns(fn, concerns)` folds right over the reversed concern list, passing the accumulated wrapper as input to the next layer. The fold is a `functools.reduce` over `reversed(ordered_concerns)` with a two-argument applicator `(inner, concern) -> wrap(concern, inner)`.
- The `Concern` ADT must discriminate each member to its library: `Retry(on, attempts, timeout, ...)` -> `@stamina.retry`; `Observe(tracer_name, op_name)` -> OTel span + structlog bind; `Contract(conf: BeartypeConf)` -> `beartype(conf=conf)(fn)`; `Validate(model: type[BaseModel])` -> `pydantic.validate_call`; `Cache(maxsize)` -> `functools.lru_cache(maxsize=maxsize)`. Each branch is pure definition-time construction — no per-call `isinstance` dispatch. The factory builds all wrappers in one pass at decoration time; the inner closure captures only the pre-built wrappers, not the `Concern` list.
- Proven ordering law with source evidence: `beartype` MUST precede `cache` because `lru_cache` hashes arguments before `@beartype` validates them — if an invalid argument is cached before validation fires, the `Error` is cached permanently. `cache` MUST precede the observability span because a cache hit must NOT emit a span (silent fast path); wrapping the cached callable with a span means every call — hit or miss — produces a span. `retry` MUST enclose `Observe` so that each retry attempt produces its own fresh child span, preserving per-attempt trace evidence. `validate_call` MUST be outermost because it coerces raw Python types before the rail sees them; placing it inside `retry` would coerce on every retry attempt, duplicating validation cost.

[ASYNC_ASPECT_COMPOSITION_OVER_ANYIO]:
- `anyio.to_thread.run_sync(func, *args, abandon_on_cancel=False, limiter=None)` is the canonical async-aspect offload for sync handlers; `abandon_on_cancel=True` means the thread runs to completion but its result is discarded if the calling task is cancelled — use this only for fire-and-forget side effects. For sync handlers where the result feeds a `Result` rail, use the default `abandon_on_cancel=False` so cancellation blocks until the thread finishes, preventing partial state. The deprecated `cancellable=` parameter emits `DeprecationWarning` since AnyIO 4.1.0. [`anyio/to_thread.py` lines 25-65, anyio 4.13.0, 2026-06-09]
- `anyio.fail_after(delay)` is a `@contextmanager` (sync context manager factory, NOT async); it wraps a `CancelScope` and raises `TimeoutError` only AFTER the `with` block exits if `cancel_scope.cancelled_caught and current_time() >= cancel_scope.deadline`. The `TimeoutError` is stdlib's, not anyio-specific — the aspect boundary catches `TimeoutError` and lifts to `Error(Timeout(...))`. `move_on_after(delay)` returns the `CancelScope` directly (no `@contextmanager` wrapping); use `scope.cancelled_caught` after the block to detect expiry. [`anyio/_core/_tasks.py` lines 102-145, 2026-06-09]
- `anyio.create_task_group()` -> `async with tg: tg.start_soon(coro_fn, *args)` fan-out collects ALL task exceptions into a `BaseExceptionGroup` on exit. In Python 3.11+ this is the stdlib `BaseExceptionGroup`; in older Python anyio depends on the `exceptiongroup` backport. The aspect that wraps fan-out for a `Result`-returning aggregate must use `except* SomeError as eg:` (Python 3.11+ syntax) or `exceptiongroup.catch` to peel the group. The observability aspect must open a parent span BEFORE the task group and use `bind_contextvars` so child tasks inherit the trace context via `contextvars` propagation — child spans opened via `start_as_current_span` inside `start_soon`-ed coroutines automatically child the inherited context. [`anyio/_backends/_asyncio.py` lines 738-857, 2026-06-09]
- Async aspect composition over `anyio` requires that all concern layers be `async def` when any inner layer is async. Wrapping an async inner callable with a sync outer concern (e.g., a sync `move_on_after` scope placed outside an `async def`) is invalid — `move_on_after` REQUIRES an active async event loop even though it returns a sync context manager. The correct pattern: the aspect factory detects `iscoroutinefunction(fn)` and produces an `async def` wrapper for every layer in the chain when the innermost function is async; it never mixes sync and async wrapper layers.
