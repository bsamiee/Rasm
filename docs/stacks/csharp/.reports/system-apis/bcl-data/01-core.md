# BCL Data Replacement Surfaces

[FROZEN_COLLECTIONS]:
- `FrozenDictionary<TKey,TValue>` and `FrozenSet<T>` are read-only after construction; `ToFrozenDictionary` / `ToFrozenSet` from any `IEnumerable<KeyValuePair<>>` or `IEnumerable<T>`. The build step performs analysis and may allocate several internal arrays; this cost is paid once so all subsequent lookups are branch-minimal.
- The concrete runtime type is determined by key type, count, and comparer: string keys with `StringComparer.Ordinal` on large sets produce `OrdinalStringFrozenDictionary_LeftJustifiedSubstring<T>` (substring-hashing to minimize comparisons); small or int-keyed sets produce `WithFullValues<,,>` or `WithOptionalValues<,,>`; two-entry dicts may collapse to direct value tables. Callers see only the abstract `FrozenDictionary<TKey,TValue>` surface — the concrete type is an implementation detail, but it exposes the lookup performance guarantee.
- `GetAlternateLookup<ReadOnlySpan<char>>()` is the critical companion: returns a `FrozenDictionary<TKey,TValue>.AlternateLookup<ReadOnlySpan<char>>` that accepts spans for `TryGetValue`, `ContainsKey`, and indexer access without allocating a temporary `string`. Requires the underlying comparer to support the alternate key (e.g., `StringComparer.Ordinal` or `OrdinalIgnoreCase`). Same API available on mutable `Dictionary<string,TValue>` for the lookup-without-alloc case; mutable version also exposes `TryAdd`, `Remove`, and indexer set via the alternate lookup, enabling span-keyed mutation.
- `FrozenSet<T>.GetAlternateLookup<ReadOnlySpan<char>>()` provides span-based `Contains` with identical comparer constraint.
- Frozen construction is not thread-safe; `ToFrozenDictionary` must complete before concurrent readers observe the result. Post-freeze, all operations are thread-safe with no lock.
- Hand-rolled `static readonly Dictionary<,>` lookup caches are the rejected form. They require defensive `ContainsKey` / `TryGetValue` pairs, provide no span-keyed lookup, and expose the internal mutable reference. Replace with `FrozenDictionary` + `GetAlternateLookup`.

[IMMUTABLE_COLLECTIONS]:
- `ImmutableArray<T>` is a value type wrapping a single array reference; `default(ImmutableArray<T>)` is a distinct state (`.IsDefault == true`) that is not the same as `ImmutableArray<T>.Empty`. Accessing `.IsEmpty`, `.Length`, or any element on a `default` instance throws `NullReferenceException` — not `InvalidOperationException`. The safe guard is `.IsDefaultOrEmpty`, which returns `true` for both states without throwing.
- `ImmutableArray.CreateBuilder<T>(initialCapacity: int)` creates a builder that pre-allocates the backing array. When `builder.Count == builder.Capacity` exactly, `MoveToImmutable()` transfers ownership of the array without copying (zero allocation). When counts differ, `MoveToImmutable()` throws `InvalidOperationException`; `ToImmutable()` always copies and is safe at any count.
- `ImmutableArray.Create(ReadOnlySpan<T>)` allocates one array from the span and wraps it — the creation path for span-to-immutable pipelines without intermediate `List<T>`.
- `ImmutableArray<T>.AsSpan()` returns a `ReadOnlySpan<T>` over the backing array in O(1) with no copy. This is the correct path for passing immutable array data into span-consuming APIs. Do not route through `ImmutableCollectionsMarshal.AsImmutableArray(array).AsSpan()` — that is an array-to-immutable-then-span round trip that adds a wrapping cost.
- `ImmutableList<T>` is a 2-3 finger tree: indexed access is O(log n), not O(1). `ImmutableArray<T>` is O(1) random access. Choose based on whether structural sharing via `Add`/`Remove` (list) or read-heavy projection (array) dominates. Defensive copies of `IEnumerable<T>` for read-only exposure should be `ImmutableArray<T>`, not `ImmutableList<T>`.
- `ImmutableDictionary.SetItem` returns a new dictionary with one key replaced; `SetItems` batches replacements and pays one structural copy. The `Builder` pattern (`.ToBuilder()` → mutate → `.ToImmutable()`) is the correct form for multi-step construction: it avoids N intermediate dictionary allocations.
- `ImmutableSortedDictionary<TKey,TValue>` maintains key order via a sorted AVL tree; use it when sorted key enumeration is load-bearing, not as a general frozen-lookup replacement (lookup is O(log n) vs O(1) for frozen).

[SPAN_AND_MEMORY]:
- `CollectionsMarshal.AsSpan(List<T>)` returns a `Span<T>` over the list's internal array covering `[0..Count)`. Any `Add` that triggers a resize invalidates the span; concurrent structural mutation is undefined behavior. In-place bulk transforms (normalize, scale, encode) on lists use this to eliminate a copy; pin the list size before calling.
- `CollectionsMarshal.SetCount(List<T>, int)` resizes the list's `Count` property without zero-initializing new elements. Use before `AsSpan` when pre-allocating result capacity: `SetCount(list, n)` then write via the span, then the list exposes count-n filled elements. Zero-init cost is the rejected form; `new T[n]` or `Enumerable.Repeat(default, n)` for known-length outputs are replaced by `SetCount` + span write.
- `CollectionsMarshal.GetValueRefOrNullRef(dict, key)` returns a `ref TValue` into the dictionary's internal storage — no copy. Check with `Unsafe.IsNullRef(ref result)` before use. `GetValueRefOrAddDefault(dict, key, out bool existed)` atomically returns a ref to the value, inserting `default(TValue)` if absent. This is the correct form for accumulation loops that simultaneously test and update: one lookup instead of `TryGetValue` + `Add`.
- `MemoryMarshal.AsBytes<T>(Span<T>)` reinterprets a `Span<T>` of an unmanaged blittable type as `Span<byte>`. `MemoryMarshal.Cast<TFrom,TTo>` reinterprets between unmanaged types. Both are O(1) with no copy. The rejected pattern is `Buffer.BlockCopy` or `Array.Copy` for struct-to-byte or float-to-int reinterpretation.
- `MemoryMarshal.Write<T>(Span<byte>, T)` and `MemoryMarshal.Read<T>(ReadOnlySpan<byte>)` serialize/deserialize unmanaged structs without `BinaryWriter`/`BinaryReader` allocation. Use for fixed-format binary protocols over `stackalloc` buffers.
- `MemoryMarshal.GetReference<T>(Span<T>)` returns a `ref T` to the first element without bounds checking — valid for unsafe arithmetic pointer patterns inside `unsafe` blocks. Do not use as a general accessor; use indexers for safe element access.
- `MemoryMarshal.TryGetArray(ReadOnlyMemory<T>, out ArraySegment<T>)` extracts the backing array and offset when the `Memory<T>` was created from an array. Returns `false` for `MemoryManager<T>`-backed memory (e.g., pinned native buffers). Required before calling any API that accepts `ArraySegment<T>` only.
- `ArrayBufferWriter<T>` implements `IBufferWriter<T>`: `GetSpan(sizeHint)` returns a writable span, `Advance(count)` records written bytes. `WrittenSpan` and `WrittenMemory` expose the written region without additional allocation. Use as the `IBufferWriter<byte>` target for `Utf8JsonWriter`, `PipeWriter`, and protocol encoders. `ArrayBufferWriter<T>` does not implement `IDisposable` — no `using` statement.

[STRING_AND_SPAN_FORMATTING]:
- `string.Create<TState>(int length, TState state, SpanAction<char,TState> action)` allocates exactly one string of the specified length, calls the action with the backing span still mutable, then seals it. The `static` lambda constraint on the action eliminates closure allocations. Use for any fixed-length string construction that would otherwise require `StringBuilder` or string concatenation. Reject `string.Concat`, `$""` interpolation, and `StringBuilder` for known-length outputs.
- `Span<char>.TryWrite($"...", out int charsWritten)` is the interpolated string handler extension on `MemoryExtensions`. It accepts any `ISpanFormattable` argument, composing multiple formatted values into a `stackalloc` buffer in one pass with no intermediate string allocation. Return `false` when the buffer is too small; callers must handle this.
- `ISpanFormattable` (`TryFormat(Span<char> dest, out int charsWritten, ReadOnlySpan<char> format, IFormatProvider? provider)`) is the correct interface for domain types that appear in formatted output. All BCL numeric types (`int`, `double`, `Guid`, `DateTime`, `DateTimeOffset`, `TimeSpan`) implement it. Domain value types (coordinates, identifiers, measurements) implement it to participate in `TryWrite` composition without allocation.
- `ISpanParsable<T>` (`T.Parse(ReadOnlySpan<char>)`, `T.TryParse(ReadOnlySpan<char>, out T)`) is the symmetric read side. All BCL numeric types implement it. Domain types parsing from sliced spans avoid the `ToString()` intermediate that `int.Parse(string)` requires.
- `DefaultInterpolatedStringHandler` (namespace `System.Runtime.CompilerServices`) is the raw handler the compiler lowers interpolated strings to. Direct instantiation via `AppendLiteral` / `AppendFormatted` / `ToStringAndClear` is the mechanism for custom interpolation pipelines (logging, tracing) that want the format-value composition without boxing or `params object[]`.

[SEARCH_VALUES]:
- `SearchValues<char>` created via `SearchValues.Create(ReadOnlySpan<char>)` compiles the character set into a SIMD-accelerated bitmap or lookup table depending on the character range. ASCII-range sets use a 128-bit bitfield; mixed sets use a 256-entry lookup. All `MemoryExtensions` methods (`IndexOfAny`, `IndexOfAnyExcept`, `ContainsAny`, `LastIndexOfAny`) accept `SearchValues<char>` or `SearchValues<byte>` as a replacement for the `char[]` / `byte[]` overloads.
- `SearchValues<string>` created via `SearchValues.Create(string[], StringComparison)` compiles the string set using the Teddy multi-string search algorithm (SIMD vectorized Aho-Corasick variant). `StringComparison` is mandatory — no default overload exists. Use `Ordinal` or `OrdinalIgnoreCase`; culture-sensitive comparisons are not supported. The concrete runtime type names encode the algorithm variant selected based on set size and prefix structure (e.g., `AsciiStringSearchValuesTeddyNonBucketizedN3`). `MemoryExtensions.ContainsAny(ReadOnlySpan<char>, SearchValues<string>)` and `IndexOfAny` return the position of the first matched string's start — not the matched string itself. When the matched value is needed, use `IndexOfAny` with a subsequent slice comparison.
- `SearchValues<byte>` follows the same API as the char variant, targeting `ReadOnlySpan<byte>` methods. Use for binary protocol scanning (delimiter bytes, sentinel sequences).
- Pre-construct `SearchValues` as `static readonly` fields; construction cost (compilation of the search structure) is non-trivial and must not be paid per call.

[SYSTEM_IO_HASHING]:
- `System.IO.Hashing` provides `XxHash32`, `XxHash64`, `XxHash3`, `XxHash128`, `Crc32`, and `Crc64`. All extend `NonCryptographicHashAlgorithm`. Static one-shot forms (`XxHash64.HashToUInt64(ReadOnlySpan<byte>)`, `XxHash64.Hash(byte[])`) are the standard path; they allocate no state objects. `XxHash3` and `XxHash64` produce 8-byte (64-bit) hashes; `XxHash128` produces a 16-byte `UInt128`.
- Streaming form: instantiate directly (no `using` — `NonCryptographicHashAlgorithm` does not implement `IDisposable`), call `Append(ReadOnlySpan<byte>)` one or more times, then call `GetCurrentHash(Span<byte>)` (non-destructive) or `GetHashAndReset(Span<byte>)` / `GetHashAndReset()` (destructive, returns `byte[]`). `Reset()` returns to initial state without reallocation. Streaming with `GetHashAndReset` over multiple chunks avoids allocating a concatenated input buffer.
- `GetHashAndReset()` is overloaded: the zero-argument form returns a new `byte[]`; the `Span<byte>` form fills the destination without allocation. Prefer the span overload when the hash is consumed immediately (comparison, embedding in a record).
- Hand-rolled `GetHashCode`-based hashing via `HashCode.Combine` is appropriate for in-process structural equality. `System.IO.Hashing` targets cross-process or cross-session stable hashes (cache keys, file fingerprints, content-addressable storage). Do not use `GetHashCode` output for persistence; do not use `XxHash` for security-sensitive contexts — it is non-cryptographic.
- `XxHash3` outperforms `XxHash64` on modern SIMD hardware at equivalent output size; prefer it for new non-cryptographic fingerprinting.

[SYSTEM_TEXT_JSON_SOURCE_GENERATION]:
- `[JsonSerializable(typeof(T))]` on a `partial class Ctx : JsonSerializerContext` generates compile-time metadata for `T`; `Ctx.Default.T` is the typed `JsonTypeInfo<T>` that `JsonSerializer.Serialize/Deserialize` accepts directly — no reflection, no runtime type discovery.
- `[JsonSourceGenerationOptions(...)]` on the context class sets options at generation time: `PropertyNamingPolicy`, `WriteIndented`, `UseStringEnumConverter`, `DefaultIgnoreCondition`, `UnmappedMemberHandling`. These compile into the generated metadata and override `JsonSerializerOptions` defaults when using `Ctx.Default` directly.
- `JsonTypeInfoResolver.Combine(ctx1, ctx2, ...)` chains resolvers; the first resolver that returns non-null for a type wins. Use to merge a source-generated context (for owned types) with `DefaultJsonTypeInfoResolver` (for foreign types that require reflection fallback).
- `JsonSerializerContext.WithAddedModifier(Action<JsonTypeInfo>)` returns a new resolver wrapping the context with a runtime modifier. The modifier sees the generated `JsonTypeInfo` and can remove, reorder, or conditionally suppress properties via `info.Properties.Remove(prop)` or `prop.ShouldSerialize = (obj, val) => false`. This is the correct path for runtime property filtering on source-generated contracts — not a hand-rolled custom `JsonConverter`.
- `JsonSerializerOptions.MakeReadOnly(populateMissingResolver: bool)`: passing `true` installs `DefaultJsonTypeInfoResolver` automatically when no resolver was set. Calling `MakeReadOnly()` (no argument) throws `InvalidOperationException` when no `TypeInfoResolver` was assigned. In production, always assign `TypeInfoResolver` explicitly before sealing.
- `JsonNamingPolicy.SnakeCaseLower`, `SnakeCaseUpper`, `KebabCaseLower`, `KebabCaseUpper` are first-class statics — no hand-rolled converter or `string.Replace` is needed for any of the four standard identifier casing schemes.
- `JsonUnmappedMemberHandling.Disallow` (attribute `[JsonUnmappedMemberHandling(JsonUnmappedMemberHandling.Disallow)]` on the type or `UnmappedMemberHandling` in options) makes deserialization throw on unrecognized JSON properties. Use on strict protocol types to catch schema drift early rather than silently ignoring unknown fields.
- `Utf8JsonWriter` over `IBufferWriter<byte>` (e.g., `ArrayBufferWriter<byte>`) writes UTF-8 JSON without intermediate string allocation. `Utf8JsonReader` over `ReadOnlySpan<byte>` reads without string allocation. Both are the correct forms for high-throughput serialization paths; `JsonSerializer.Serialize(string)` allocates a UTF-16 string intermediate and is the rejected form for hot paths.
- `JsonNode` / `JsonObject` / `JsonArray` (the DOM API) support mutation after parse: `node["key"] = value`, `array.Add(element)`. Use for transformations where the input schema is partially unknown; use typed `JsonSerializer` for owned, schema-stable types.

[RUNE_AND_TEXT_SEGMENTATION]:
- `System.Text.Rune` represents a single Unicode scalar value (codepoint in `[U+0000, U+D7FF] ∪ [U+E000, U+10FFFF]`). `Rune.TryCreate(int, out Rune)` rejects surrogates and out-of-range values; the constructor throws on invalid input. All character-level predicates (`IsLetter`, `IsDigit`, `IsWhiteSpace`, `IsControl`, `IsPunctuation`, `GetUnicodeCategory`) operate on the full Unicode codepoint, unlike `char.*` equivalents that can misbehave on supplementary-plane characters.
- `Rune.DecodeFromUtf16(ReadOnlySpan<char>, out Rune, out int charsConsumed)` and `Rune.DecodeFromUtf8(ReadOnlySpan<byte>, out Rune, out int bytesConsumed)` advance a cursor without intermediate string construction — the correct primitives for custom parsers that handle arbitrary Unicode. Return `OperationStatus.Done`, `NeedMoreData`, or `InvalidData`.
- `Rune.EncodeToUtf8(Span<byte>)` writes 1–4 bytes and returns the count. Paired with `DecodeFromUtf8`, this is the zero-alloc UTF-8 transcoding loop: read from source span, write to destination span, advance both cursors by the consumed/written counts.
- `string.EnumerateRunes()` returns a `StringRuneEnumerator` (a `ref struct`) for forward-only Rune traversal over a string. Use in `foreach`; do not store the enumerator. This is the correct form for codepoint-level string analysis (letter count, category histogram, normalization check) that must handle supplementary-plane characters.
- `Rune.ToUpperInvariant` and `ToLowerInvariant` perform codepoint-level case mapping without culture-sensitive rules, safe for identifiers and protocol keywords. They do not handle context-dependent mappings (e.g., Turkish dotted-I); use `CultureInfo`-qualified APIs for locale-aware text.
- `System.Globalization.StringInfo.GetTextElementEnumerator(string)` and `StringInfo.LengthInTextElements` operate at the grapheme cluster level, grouping base characters with their combining marks. A string with `N` characters and `M` combining marks reports `LengthInTextElements == N - M`. Use when the unit of user-perceived character is load-bearing (cursor movement, display width estimation, text truncation). `StringInfo.LengthInTextElements != string.Length` whenever combining characters, surrogate pairs, or variation selectors are present.

[TENSOR_PRIMITIVES_SMELL_ROW]:
- `System.Numerics.Tensors.TensorPrimitives` exposes static generic methods over `ReadOnlySpan<T>` and `Span<T>` for element-wise and reduction operations. The generic constraint is `where T : INumber<T>` (or more specific interfaces for trig, log, float-only ops), enabling one method to serve `float`, `double`, `int`, `Half`, `decimal`, and custom `INumber<T>` implementations.
- Scalar loops iterating element-by-element with `+`, `*`, `Math.Sqrt`, `Math.Sin` are the rejected form. The replacement: `TensorPrimitives.Add`, `Multiply`, `Sqrt`, `Sin`, `Exp`, `Log2`, `Pow`, `Atan2` — all vectorized via `Vector128<T>` / `Vector256<T>` / `Vector512<T>` depending on hardware.
- Reductions: `Sum`, `Product`, `SumOfSquares`, `SumOfMagnitudes`, `Max`, `Min`, `Norm` (L2 norm), `Dot`. These replace hand-rolled accumulation loops.
- `CosineSimilarity`, `Dot`, `Norm` — the three distance/similarity primitives for dense vector comparison. `FusedMultiplyAdd(a, b, c, dest)` computes `a*b+c` with one rounding, replacing `a*b` then `+c`.
- `Lerp(a, b, t, dest)` — element-wise linear interpolation, replacing `a + (b-a)*t` loops.
- `ConvertChecked`, `ConvertTruncating`, `ConvertSaturating` — type-converting element-wise copies between span types. `ConvertTruncating` truncates toward zero; `ConvertSaturating` clamps to the destination type's range. Replace manual `for` loops that cast element by element.
- `SoftMax`, `Sigmoid` — applied directly to float spans. Replace manual exp-then-normalize loops.
- `IndexOfMax`, `IndexOfMin`, `IndexOfMaxMagnitude` — argmax/argmin without a manual tracking variable.
- The smell: any `for` loop over a numeric array whose body consists solely of arithmetic, math function calls, or type conversions is a `TensorPrimitives` candidate. The law resides in the algorithms page; this row identifies the data-shape signal that triggers the lookup.
