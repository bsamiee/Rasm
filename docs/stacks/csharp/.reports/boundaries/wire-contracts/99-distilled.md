# Wire Contracts — Distilled

[PROTOCOL_SHAPED_EDGE]:
- The wire shape is an attributed record family that stays protocol-shaped at the edge; the domain owner never carries codec policy, the wire record never carries domain behavior, and the converter is the only site where both schemas are simultaneously visible — the domain hierarchy and the wire hierarchy stay structurally parallel but decoupled, neither referencing the other's schema.
- `JsonTypeInfo<T>` is the cross-layer contract surface: a property's `Name`, `IsRequired`, `CustomConverter`, `Order`, presence in `Properties`, the polymorphic discriminator property name, and member order are breaking-change coordinates of equal gravity; adding a property with an ignore-when-default condition is forward-compatible, removing or renaming is not, and `[JsonPolymorphicAttribute]` offers no discriminator migration path.
- Versioned envelopes carry the schema version as a generated keyed value object riding the same converter path as any domain key — unknown versions reject at the converter, never propagate; the `kind` discriminant's valid set is versioned, validated by an admission modifier before any factory runs; `[JsonPolymorphic]` sits on the payload field, not the envelope, keeping the discriminator scoped to the payload rather than the framing.
- Codec options instances, transport objects, and provider envelope types never appear on a public boundary surface; the only crossing is a value-typed projection captured at the seam — a surface that returns or accepts a live codec object couples callers to provider lifecycle and kills provider polymorphism.
- Timestamps cross as `DateTimeOffset` with explicit offset, never `DateTime`; elapsed and duration fields cross as ISO 8601 duration via a registered `JsonConverter<TimeSpan>` when peers are non-.NET — the built-in round-trip `TimeSpan` form is .NET-only vocabulary.

[GENERATION_COVERAGE_LAW]:
- `Metadata` generation mode emits full contract metadata — property shapes, constructor resolver, polymorphism — but no fast path; `Serialization` mode emits fast-path write code only, with no deserialization support at all including `Populate`; only `Default` covers read and write simultaneously. The mode choice is a coverage contract, not an optimization flag.
- `JsonTypeInfo<T>.SerializeHandler != null` is the audit predicate for fast-path write coverage: a null handler inside a source-generated context silently falls back to reflection-style property enumeration, so sweeping `SerializeHandler` across all envelope types at initialization is the compile-adjacent proof the write path stays AoT-clean and allocation-free.
- `RespectNullableAnnotations` and `RespectRequiredConstructorParameters` set on `[JsonSourceGenerationOptions]` push nullability and required-field law into generated code at compile time, replacing post-deserialization guards.
- Types outside generator sight take exactly three routes: a registered `JsonConverter<T>`, a custom resolver composed via `JsonTypeInfoResolver.Combine` into `TypeInfoResolverChain`, or programmatic contracts via `JsonTypeInfo.CreateJsonTypeInfo<T>` plus `CreateJsonPropertyInfo` — mutable until `MakeReadOnly()`, with added properties participating exactly as generated ones.

[ADMISSION_STRICTNESS_POSTURE]:
- The strictness split is positional: `JsonUnmappedMemberHandling.Disallow` on the inner envelope type — schema drift is a failure — and `Skip` on outer wrappers and extension bags — forward-compatible noise is tolerated; the type-level attribute overrides the global setting, so the split lives on the types, not the options.
- `AllowDuplicateProperties = false` at every admission options: lenient last-write-wins silently corrupts what a strict parser would reject. `AllowOutOfOrderMetadataProperties` stays false at owned boundaries; relax only when the remote protocol demands it.
- `RespectNullableAnnotations` enforcement fires at the JSON token level, before any constructor or factory sees the value — it is the outermost null-rejection layer, ahead of `[JsonRequired]` and the domain factory. `RespectRequiredConstructorParameters` makes non-optional parameters required, but parameters with default values silently bind the default on a missing property — optionality on the constructor is wire optionality.
- `JsonIgnoreCondition.WhenReading` is the asymmetric contract shape: write-only computed fields appear outbound and are never fed back inbound.
- Reader and writer depth limits are asymmetric — document parsing defaults to 64, writing to 1000 — so a tree that writes cleanly can fail its own admission round-trip; pin one explicit `MaxDepth` on both sides and state it as a protocol constraint.

[KERNEL_FRAMEWORK_CONTRACT]:
- Null-token dispatch derives from `default(T)`, not intent: a value-type converter receives `Read` on a null token automatically; a reference-type converter is short-circuited to `default` before `Read` runs, so it cannot observe wire-`null` at all without overriding `HandleNull => true`. The override flips read and write handling together — there is no one-sided override; write-side null asymmetry lives on the property's missing-key condition, read-side inside `Read`. Calling `base.HandleNull` from an override silently re-arms the default derivation it was meant to replace — the base getter has a side effect.
- A converter implicitly declares one of two strategies. Value strategy is read-ahead-buffered: the full value is materialized before `Read` is entered, so the converter never sees a pipeline-segment straddle, and the framework verifies zero reader displacement after `Read` — any net token consumed throws a converter-read fault. Object strategy drives the reader itself, must tolerate mid-structure suspension on streaming input, and must exit on the closing `EndObject`/`EndArray` at entry depth. The write verifier symmetrically asserts balanced writer depth — an unclosed object or a property name with no value faults at the boundary, not in a downstream consumer. Reading one extra token "to be safe" corrupts the parent loop and trips the verifier.
- The dictionary-key seam is separate: `WriteAsPropertyName`/`ReadAsPropertyName`, with defaults that throw for non-string keys, so a keyed value object that round-trips as a value still faults as a map key until both seams are overridden; the key-write verifier demands exactly one property name and no value — the inverse of the value balance check — and the key-read seam must consume exactly the property-name token with zero net displacement.
- Contextual `JsonNumberHandling` is invisible to ordinary custom converters — only internal number-converter hooks observe it — so a converter that must reject string-encoded numerics demands `JsonTokenType.Number` in its own `Read`; a strict flag it never sees enforces nothing.
- Every exception escaping `Read`/`Write` is caught by the framework's core wrapper and re-thrown path-annotated — `JsonException` enriched in place, `FormatException`/`InvalidOperationException`/`NotSupportedException` converted or re-thrown with path — but a domain-specific exception type escapes unwrapped and loses path evidence: throw `JsonException` for every wire-shape rejection. Swallowing the throw to return a sentinel defeats boundary failure visibility entirely — rejection must propagate so path annotation and the seam's single exceptional-error translation both fire; the rail bridge translates once at the seam, never per converter. Lifecycle callbacks fire outside this wrapper, so their exceptions propagate unwrapped — serialize only already-validated instances.

[CLOSED_FAMILY_CONVERTER_OWNERSHIP]:
- One converter over the closed-family root owns the entire wire projection: `Read` peeks the discriminant and dispatches to the case contract; `Write` matches the case and emits shape plus discriminant — never a converter per case, and the converter is the only place both hierarchies meet.
- Discriminant-first reading is incompatible with positional-constructor binding unless buffered: parse the value subtree into a detached document, extract the discriminant, deserialize the remainder against the resolved case contract. `JsonDocument.ParseValue(ref reader)` consumes exactly the subtree and lands on its final token — satisfying the object-strategy displacement verifier by construction — and `Deserialize(JsonTypeInfo<T>)` on the element stays metadata-clean through the generated context. A hand-rolled forward-scan-and-replay pays off only where double-parse cost dominates at extreme throughput.

```csharp
public sealed class FamilyConverter : JsonConverter<Family>
{
    public override Family Read(ref Utf8JsonReader reader, Type t, JsonSerializerOptions o)
    {
        using var doc = JsonDocument.ParseValue(ref reader);
        var root = doc.RootElement;
        return root.GetProperty("$kind").GetString() switch
        {
            "a" => new Family.A(root.Deserialize(FamilyContext.Default.CaseA)!),
            "b" => new Family.B(root.Deserialize(FamilyContext.Default.CaseB)!),
            _   => throw new JsonException("unrecognized discriminant"),
        };
    }
    public override void Write(Utf8JsonWriter w, Family value, JsonSerializerOptions o) =>
        value.Switch(
            a => Emit(w, "a", a.Payload, FamilyContext.Default.CaseA),
            b => Emit(w, "b", b.Payload, FamilyContext.Default.CaseB));
}
```

- The polymorphic write path is gated by an init-only capability flag on the converter: when advertised, the framework resolves the runtime subtype's converter before the declared one, bypassing the family converter for derived instances — a hand-written family owner must not advertise it. Read re-entry branches into continuation machinery only for reference types advertising polymorphism; a sealed value-projected family rides the buffered, displacement-verified path and its streaming-suspension contract collapses to nothing.
- The attributed alternative — `[JsonPolymorphic(TypeDiscriminatorPropertyName = ...)]` with explicit `[JsonDerivedType]` rows — is the closed-hierarchy form for pure wire records: `UnknownDerivedTypeHandling = FailSerialization` and `IgnoreUnrecognizedTypeDiscriminators = false` (the defaults) make drift visible in both directions; `FallBackToBaseType` and discriminator tolerance are opt-in only where the protocol specifies forward-compatible fallback. `JsonTypeInfo.PolymorphismOptions` stays mutable until seal, so a pre-seal modifier can populate `DerivedTypes` from a capability probe — a closed polymorphic contract matched to the peer's actual capability.

[CONVERTER_SYNTHESIS_AND_SELECTION]:
- A factory converter answers `CanConvert` over an open shape and synthesizes the closed converter in `CreateConverter`; its own `Type` is null, its strategy none, and every read/write operation runs on the product. Returning null or another factory throws immediately at expansion. Expansion rides the metadata cache keyed by resolved type — reflection over closed generic arguments runs once per type per options, making this the one sanctioned reflection site at a generated-context boundary: contract-resolution time, never the hot path.
- A factory that closes over policy state must hold it by stable reference — a freshly-closured factory per options forces a distinct converter reference, a distinct cache fingerprint, and a cold metadata cache; policy that varies per call belongs in a value the synthesized converter reads at write time. The built-in string-enum converter is the model: a factory holding naming policy emits a stateless per-type converter — the template for any vocabulary-keyed family whose key spelling is policy, not type.
- The converter-selection attribute is subclassable with a virtual `CreateConverter(Type)`: the subclass synthesizes a converter parameterized by the attributed member's declared type, making the attribute a per-property selector — the same domain type serializes differently per declaration site with zero per-field converter types. A member-level attribute resolves into the property's effective converter at contract build and takes precedence over the options-level list; policy that must hold uniformly across a type belongs on a resolver modifier that sets the property converter centrally, never on per-member attributes a future field can silently omit.

[CONTRACT_MUTATION_HOOKS]:
- Hook placement follows ownership: `IJsonOnDeserializing`/`IJsonOnDeserialized`/`IJsonOnSerializing` implemented on the attributed wire record carry instance-level invariants — scratch bitmask initialization, required-population assertion, correlation preconditions; the modifier-assigned `JsonTypeInfo.OnDeserializing`/`OnDeserialized`/`OnSerializing` delegates (`Action<object>`, boxed receiver) carry type-external cross-cutting policy from one site. All fire synchronously on the (de)serialization thread — never async, never re-entrant, never writing shared state.
- Lifecycle delegates, `PolymorphismOptions`, and `CreateObject` assignment all throw on a `JsonTypeInfo` whose `Kind` is not `Object` (collection kinds accept `CreateObject`); because seal typically follows the modifier chain, the throw surfaces at first serialization — distant from the registering modifier — so every mutating modifier guards `Kind` explicitly and asserts `!typeInfo.IsReadOnly` rather than silently skipping, which converts a fast failure into an invisible policy omission.
- `ShouldSerialize` versus `OnSerializing` is install-time versus evaluation-time: `ShouldSerialize` (`Func<object, object?, bool>` over live parent and value) is installed pre-seal and evaluated per write — the only mechanism that suppresses an individual property dynamically, suppressing key and value entirely; `OnSerializing` fires after seal per instance and can assert or precompute but never suppress.
- `JsonPropertyInfo.Get`/`Set` replacement projects a property through an arbitrary transform without a full-type converter — a getter returning null for an absent payload, paired with a write-time null-ignore condition, emits missing-key rather than explicit null, keeping absence-versus-null a deliberate protocol choice at the contract layer.
- `JsonPropertyInfo.Order` drives ascending write order; default 0, ties by declaration order. For positional consumers — length-prefixed framing reading the length field first — order is enforced at the contract and treated as a breaking-change coordinate.
- A programmatically added property has no `AssociatedParameter` and is silently ignored under constructor binding — it requires a writable init-property or `JsonObjectCreationHandling.Populate`. `Populate` applies only to instance-reusable properties (collections, parameterless classes), has no effect on positional-constructor properties, and is the correct mode for extension-data overflow bags that must accumulate rather than replace.
- `JsonTypeInfo<T>.CreateObject` set to a pooling factory replaces per-message allocation; combined with preferred `Populate` handling, properties write into the recycled instance — but the deserializer never zeroes prior state, so the factory must fully re-initialize before returning.

[RESOLVER_FOLD_AND_MODIFIER_FUSION]:
- Combining resolvers folds to one flat list queried left-to-right, first non-null metadata wins: nested chains splice inline, the empty resolver drops, a single effective resolver unwraps — the runtime never walks a resolver-of-resolvers. Precedence is purely positional with no per-type priority knob; a generated context at position 0 with a modifier-bearing reflective fallback at 1 yields source-gen for covered types and modifier-shaped fallback for the rest, on one options instance.
- A modifier wrapper is atomic under flattening: it post-processes its single inner source's winning result exactly once — never once per chain element, never on null. Stacked `WithAddedModifier` calls fuse into one wrapper holding one ordered array; the fold is associative and only the final left-to-right sequence survives. Modifiers run unconditionally in order over one mutable metadata instance — an ordered transform pipeline, not independent advice — so order is load-bearing policy: a rename ahead of a redaction predicate keyed on the new name composes, the reverse silently misses.
- A modifier injecting policy must gate on `OriginatingResolver` to skip types resolved by a foreign chain member that manages its own policy.

[RESOLUTION_CACHE_LAW]:
- Per-type contract metadata caches against a structural fingerprint of the options — every scalar setting by value, plus the resolver and each registered converter by reference identity, the converter list positionally. Equal scalars with a freshly constructed resolver is a distinct fingerprint and a cold cache; the same converters reordered is a cold cache; value-equality overrides on a converter cannot opt two instances into one bucket. The discipline: resolver and converter instances are statics, reused by reference across every options that should share; per-request variance routes through fields on a value the converter reads, never through a new options shape.
- Cross-instance sharing has a hard ceiling: a fixed-capacity weak-reference table tracks distinct live shapes, and a shape past the ceiling silently receives a private untracked cache — full contract-build cost per instance, no error, no value-equality fallback; long-lived distinct options pin their slots until collected. Keep long-lived distinct shapes well under the ceiling.
- Above the dictionary sits a one-entry volatile slot for the most recent root type: one envelope type in a tight loop rides it; alternating two roots thrashes it to zero benefit. The slot keys on the metadata's reported type; polymorphic ancestor metadata hangs off a separate field consulted in sequence.
- Seal is the pipeline barrier: options auto-seal on first use, every scalar setter and both collections gate on mutability, and the sealed chain's modify hook throws — composition completes before first serialize, and the only post-seal variance channel is per-write predicate state converters already hold. The chain rejects itself and the owning options' resolver as elements (the self-recursion guard), and assigning a chain replaces the resolver field — capture the current resolver before assigning a chain that should contain it. The canonical form is a `static readonly` options never mutated after construction; the bounded exception is a request-scoped instance whose modifier payload is unavailable at startup and varies per request — never reused, lifetime explicitly bounded, resolvers combined via the span overload to skip the array allocation.
- The per-type cache stores metadata or a captured exception: a throw during resolution rethrows with original stack on every subsequent request for that type against the same cache — poison visible from every fingerprint-equal options, surfacing at first serialize of that root, distant from the registering modifier, reset only by a structurally distinct options or explicit cache clear. The seam therefore classifies a resolution fault as non-transient and non-retriable by construction — a retry against the same options replays the captured throw — distinct from network exceptional errors.
- A runtime type with no direct contract resolves by walking base types upward to the first metadata, then scanning interfaces; a contracted interface mutually unassignable with the chosen base throws an ambiguity fault — sticky like any resolution throw, memoized behind a resolved flag. The law: give each closed wire family one unambiguous contracted root — a sealed base or a single marker the converter keys on — so the walk terminates before the ambiguity branch.

[VOCABULARY_AND_MEASUREMENT_PROJECTION]:
- Bounded-vocabulary wire fields ride a generated enum-backed converter keyed on the protocol string: span-based inbound comparison without string allocation, exact protocol key outbound. The generated admission factory is the converter `Read` — one admission point for runtime construction and wire deserialization alike; the factory rejects by throw, the converter translates to `JsonException` for path anchoring, and the seam's rail bridge captures it once as a typed exceptional error.
- A null wire token projecting to domain absence takes a thin outer converter wrapping the generated inner — null maps to the absence value, non-null delegates inward — isolating null-projection policy at the boundary without touching the generated type.
- Generated value-object converters pass through only the key field, and they register in the context's converter list or chain, never inline on the wire record — codec policy stays at the boundary, no domain behavior leaks into the wire shape.
- A measurement field is two orthogonal wire concerns: a unitless numeric whose converter demands the number token — string-encoded numerics rejected at the token, not by a handling flag — and an adjacent unit discriminant as a generated keyed value object. Only the boundary converter reads both together to emit the typed domain measurement; unrecognized unit strings reject as expected errors at admission, and the protocol unit string never travels inward as a `string` field.

[READER_KERNEL_HOT_PATH_AND_STREAMING]:
- Allocation-free property dispatch: one `static readonly FrozenDictionary` with `GetAlternateLookup` — a byte-keyed `IAlternateEqualityComparer<ReadOnlySpan<byte>, string>` comparer accepts the reader's `ValueSpan` directly with no transcode; a char-span alternate key requires a `CopyString` transcode first. `ValueTextEquals(ReadOnlySpan<byte>)` compares against UTF-8 literals with escape handling and no allocation; `static readonly byte[]` name literals are the dispatch keys.
- `HasValueSequence` gates `ValueSpan`: on multi-segment input the value lives in `ValueSequence`, and reading `ValueSpan` unguarded silently truncates. `CopyString` normalizes both cases at stack-copy cost; `ValueIsEscaped` decides raw versus unescape.
- `JsonEncodedText.Encode` pre-escapes fixed-schema property names once; `WritePropertyName(JsonEncodedText)` skips the per-write escape scan, and `EncodedUtf8Bytes` feeds hash and comparison paths.
- `JsonElement` references crossing an `await`, stored in a receipt, or outliving their `JsonDocument` must be `Clone()`d — post-disposal access throws in async continuations where the `using` closes before the continuation runs. `JsonMarshal.GetRawUtf8Value` returns a zero-copy span into the document's backing buffer, valid only while the document lives, unenforced by the type system: acquire, consume within the same synchronous scope, never store across a suspension; a clone constructed after the source document's disposal throws at construction, and the clone's raw value reads the clone's own buffer. `GetRawUtf8PropertyName` gives the same zero-copy path for name dispatch; `GetRawText` allocates and `WriteTo` is the non-allocating alternative.
- Partial reads from a pipe: construct the reader over the buffered sequence with `isFinalBlock: false` and the saved state, advance, capture `CurrentState`, then advance the pipe to `reader.Position` — in the multi-segment path `BytesConsumed` counts from the slice head and is the wrong consumed argument; only `Position` resolves within the sequence. `AllowMultipleValues = true` set on the initial reader state persists across state transitions and backs newline-delimited streaming. `TrySkip()` returning false means the segment cannot complete the skip: abandon the reader, mark the buffer examined to its end with no consumed advance, retry after the next read — advancing consumed on false drops bytes silently.
- `DeserializeAsyncEnumerable` with `topLevelValues: true` streams whitespace-separated top-level values, `false` streams root-array elements; the `JsonTypeInfo<T>` overload is the AoT-safe one — the options overload is not.

[BYTE_IDENTITY_AND_SIGNED_PASS_THROUGH]:
- A wire field carries two independent contracts: semantic (the decoded value round-trips) and byte-identity (the octet sequence reproduces bit-for-bit). Structural equality normalizes away everything byte identity depends on — its number branch realigns sign, digit spans, and exponent so `1`, `1.0`, `1.00`, and `10e-1` are equal across four byte forms, and its string branch compares after unescaping so escape-form drift is invisible. Passing a structural check is zero evidence two payloads share a signature, checksum, or idempotency key; structural equality and byte hashing are complementary verifications, never interchangeable.
- A field entering a signature, HMAC, idempotency key, or content-addressed hash is a byte-identity field: its contract is the exact octet sequence, normalized once at admission — encoder, escape case, numeric spelling, member order, timestamp precision all fixed at definition time — and excluded from every parse-and-reserialize path, which is byte-lossy by construction.
- Encoder uniformity is a composition-time invariant: one encoder per byte-identity domain, asserted, never chosen per site. The conservative escaper hex-escapes everything outside a minimal allow-list while the relaxed one leaves `<`, `>`, `&`, `+`, `'`, and non-ASCII literal — semantically equivalent output, byte-divergent payloads. Hex escapes emit uppercase digits, silently diverging from lowercase-expecting canonicalization profiles with no surfacing option. The pre-escaped name type computes equality and hash on the source string alone, ignoring its encoded bytes — a cache keyed on it cannot distinguish encoder variants and emits whichever variant won the cache race, a non-deterministic byte form the type system cannot catch.
- Runtime normalization is lossy by type: binary floats write shortest-round-trippable form (`1.500` admitted re-emits as `1.5`), timestamps trim trailing zero sub-second digits so serialized width is value-dependent — a peer's fixed seven-digit fraction never byte-matches; pin a fractional precision in the contract and normalize at admission. Non-finite floats re-kind from number to quoted literal or reject — a structural break, so admission rejects non-finite for number-typed fields. Integral and fixed-decimal values ride the invariant fixed-point formatter and are byte-stable through round-trips; the contract law follows the type.
- The raw-value write copies supplied octets verbatim — no re-escape, no re-spell, no minify, validated or not — and is the only signature-preserving forwarding path; validation proves well-formedness without altering bytes and is skipped only for same-request trusted provenance. The writer's structural-validation option and the raw write's content-validation parameter are orthogonal — one governs the token state machine, the other RFC well-formedness of the supplied bytes — so a pure pass-through writer re-emitting trusted octets may disable both. A signed numeric or timestamp field whose original spelling is part of the signing input survives only as its raw token bytes, never as a parsed value. Materializing a signed sub-document into nodes and rewriting recomputes every byte and destroys the inner signature; envelope patching around an opaque signed region mutates at the positioned-node or raw-byte level only: `DeepClone` before re-insert gated on an existing parent (single-parent rule throws otherwise), the two-argument `SetAt(index, value)` for value-only replacement — the three-argument form silently renames the key, a wire-breaking mutation — `IndexOf`-then-`SetAt`-else-`Insert` as the idempotent named replace, `ReplaceWith` for one-field re-typing, and `GetPath()` as the path evidence for violation receipts that element-based parsing cannot supply.
- Member order is never normalized by the serializer — it follows the contract's property sequence — so order is a signed coordinate: canonical profiles enforce lexicographic order through the per-property order coordinate at definition time, never through incidental declaration order the next field addition breaks. A duplicate key collapsed by a lenient parser means the signature covered bytes the domain never observed — signed boundaries reject duplicates outright. Whitespace pins at the writer (compact default); key order pins at the contract; both are breaking-change axes the protocol document states.
- The signed-boundary structure: admit raw bytes once, validate and reject duplicates and re-kinding, normalize into a canonical owner with fixed serialization coordinates, derive every signing input from that owner's emitted bytes zero-copy within the live document scope, computed once and reused — the value never re-enters the serializer between signing and transmission. Inbound verification is the inverse and harder: the peer signed its exact bytes, so verification runs against the admitted raw octets captured before any parse — the local serializer cannot reproduce the peer's encoder, number forms, or member order — holding the raw span beside the parsed value until the signature clears, only then crossing into the domain. The audit receipt records the canonical-bytes hash plus the coordinate discriminants — encoder, escape case, member order, timestamp precision — never the payload, so a later byte-identity dispute resolves against recorded coordinates rather than a re-derivation that may have drifted.
