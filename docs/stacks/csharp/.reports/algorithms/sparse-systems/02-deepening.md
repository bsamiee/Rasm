# Sparse Systems: Deepening

[ITERATOR_DISPATCH_MECHANICS]:
- `Iterator<T>.DetermineStatus` short-circuits: it iterates `_stopCriteria` in insertion order and returns the moment any criterion returns a non-`Continue` status. A `Converged` result from `ResidualStopCriterion` placed before `IterationCountStopCriterion` will never be shadowed by the count cap — but the reverse placement means the count cap fires first and `Converged` is never seen if the system converges on the limit iteration. The default stack from `MatrixBuilder<double>.IterativeSolverStopCriteria(maxIterations)` is `[FailureStopCriterion, DivergenceStopCriterion, IterationCountStopCriterion, ResidualStopCriterion(1e-12)]` — `ResidualStopCriterion` is last, so the count cap suppresses a valid convergence signal on the final iteration.
- `Iterator<T>()` (no-arg constructor) pulls the default stack from `Matrix<T>.Build.IterativeSolverStopCriteria()`, which hardcodes `maxIterations = 1000`. Passing `null` as the iterator to `TrySolveIterative` causes a fresh default-constructed iterator with this 1000-cap to be substituted internally. Always construct `Iterator<T>` explicitly with a custom stack so the cap is size-derived.
- `DivergenceStopCriterion<T>` enforces `MinimumNumberOfIterations >= 3` at the setter. Its `IsDiverging` test requires every consecutive pair in the `_residualHistory` window to satisfy `history[i] > history[i-1] * (1 + MaximumRelativeIncrease)` — a strict AND over all pairs, not a mean. Default `MaximumRelativeIncrease = 0.08` and default `minimumIterations = 10` mean every one of 10 consecutive iteration pairs must show growth above 8%. A residual sequence that dips even once cannot trigger this criterion regardless of overall trend.
- `ResidualStopCriterion<T>.DetermineStatus` measures `‖residualVector‖_∞ ≤ Maximum * ‖sourceVector‖_∞`. The threshold is relative to the right-hand-side infinity norm. The convergence gate is `_iterationCount >= _minimumIterationsBelowMaximum` where `_iterationCount = iterationNumber - _lastIteration` and `_lastIteration` initializes to `-1`. With `MinimumIterationsBelowMaximum = 0` (the default), `_iterationCount = 1` on the first call (iteration 0), satisfying `1 >= 0` immediately. The criterion fires the first time the norm drops below the threshold — single-call convergence. Setting `MinimumIterationsBelowMaximum` to any positive count `k` requires the residual to be sustained below the threshold for `k` consecutive iteration increments; it fires when `_iterationCount >= k`.
- `DelegateStopCriterion<T>` accepts `Func<int, Vector<T>, Vector<T>, Vector<T>, IterationStatus>` (signature: `iterationNumber, solutionVector, sourceVector, residualVector → IterationStatus`). It stores the delegate reference without invoking it at construction; `Clone()` shares the same delegate reference, so cloned iterators share mutable closure state if the delegate captures mutable fields.

[PRECONDITIONER_FAILURE_SURFACES]:
- `MILU0Preconditioner<double>.Initialize` explicitly requires `matrix.Storage` to be `SparseCompressedRowMatrixStorage<double>` — it throws `ArgumentException("Matrix must be in sparse storage format")` for any other storage. `ILU0Preconditioner<double>.Initialize` calls `SparseMatrix.OfMatrix(matrix)` unconditionally, performing an O(n²) dense-to-sparse conversion for non-sparse input — valid but silent and expensive.
- After `MILU0Preconditioner.Initialize` throws, `IsInitialized` remains `false`; calling `Approximate` on an uninitialized instance will NPE or produce garbage depending on whether `_alu` was partially assigned before the throw. Check `IsInitialized` before `Approximate` after any potentially-throwing initialization sequence.
- `MILU0Preconditioner` and all other ILU preconditioners throw `NumericalBreakdownException` when a zero pivot is encountered during their ILU factorization. `TrySolveIterative` does not catch this exception — it propagates to the caller. The `TrySolveIterative`-returns-status contract holds only when preconditioner initialization does not break numerically. The pattern is: initialize the preconditioner outside `TrySolveIterative`, catch `NumericalBreakdownException` at that boundary, and fall back to `DiagonalPreconditioner` or a direct solve before entering the iterative path. All `IIterativeSolver<T>` implementations call `Initialize` internally on their first `Solve`, so the exception surfaces from within `solver.Solve`, not from `TrySolveIterative`.

[COMPOSITE_SOLVER_SEMANTICS]:
- `CompositeSolver<T>` sequences through its solver list until one converges. Its internal status routing is: `Converged` → copy result and return; `StoppedWithoutConvergence` → copy partial result and continue to the next solver; any other status (including `Diverged`, `Failure`, `Cancelled`) → reset input and try next solver. The partial result from a `StoppedWithoutConvergence` solver is written to the output before continuing — if the subsequent solver converges from that warm start, the result is valid; if all solvers exhaust without `Converged`, the last `StoppedWithoutConvergence` iterate is left in the output with no indication of inadequacy. The calling site must recompute the true residual after `CompositeSolver.Solve` regardless of the iterator status.
- `CompositeSolver` resets the iterator before each solver via `iterator.Reset()`, which also resets all stop criteria. The progress of each solver is therefore fully independent; there is no carry-over of convergence history between consecutive solvers.

[CSC_CLEANUP_VS_OFINDEXED]:
- `Converter.ToCompressedColumnStorage` with `cleanup: true` calls `Cleanup()` internally, which sums duplicates via a hash-map scan and produces column-sorted, duplicate-free output. `Cleanup()` does NOT remove exact zeros — it only sums duplicates. `DropZeros(tolerance)` must follow `Cleanup()` explicitly to remove cancellation residuals from assembly. `CompressedColumnStorage<T>.OfIndexed` calls `ToCompressedColumnStorage(storage, cleanup: true, inplace: false)` internally; the same zero-retention applies.
- `Helper.SortIndices(csc)` is called by `Converter.ToCompressedColumnStorage` as a separate step after column-scatter, before `Cleanup`. The sort uses a threshold of 16 elements per column: columns with ≤16 entries use insertion sort, columns with >16 use `Array.Sort`. Structural validation with `Helper.ValidateStorage(csc, strict: true)` may fail immediately after `ToCompressedColumnStorage` with `cleanup: false` — that flag skips both `SortIndices` and `Cleanup`, leaving unsorted, potentially duplicate columns that violate `strict: true`.

[SYMBOLIC_FACTORIZATION_REUSE]:
- `SymbolicFactorization` is a plain class with public fields `pinv`, `q`, `parent`, `cp`, `leftmost`, `m2`, `lnz`, `unz`. All CSparse direct factorizations store a private `SymbolicFactorization` — there is no public API to extract it from a completed factorization, so symbolic analysis cannot be reused across re-factorizations through the public surface.
- The two-argument `Create(A, int[] p)` overloads on `SparseCholesky`, `SparseLDL`, and `SparseLU` accept a pre-computed permutation vector, skipping the AMD call but still running full symbolic analysis internally. The AMD permutation is not the bottleneck; symbolic analysis (`EliminationTree` + `ColumnCounts` + `CumulativeSum`) runs every time `Create` is called regardless of overload. Savings from caching `AMD.Generate` results are real but bounded; the symbolic phase cost is unavoidable without a lower-level API accepting a pre-computed `SymbolicFactorization`.
- `SparseCholesky.SymbolicAnalysis` allocates `columnCount + 1` elements for `cp`, calls `PermuteSym` (a full upper-triangle copy of the permuted A with no values), then calls `EliminationTree`, `ColumnCounts`, and `CumulativeSum` — O(n) integer array allocations. For large n with invariant structure, this symbolic cost dominates numeric refactorization. A pattern-keyed symbolic cache holding the `SymbolicFactorization` would amortize this, but cannot be populated through the public API; it requires wrapping the symbolic phase calls directly via `GraphHelper.EliminationTree` and `GraphHelper.ColumnCounts`.

[LDL_ZERO_DIAGONAL_CONTRACT]:
- `SparseLDL.Factorize` checks `if (D[i] == 0.0) throw new Exception("Diagonal element is zero.")`. This is a plain `System.Exception`, not `NumericalBreakdownException`. A caller catching only `NumericalBreakdownException` will not intercept this failure — it propagates as an untyped exception. The receipt boundary for LDL factorization must catch `Exception` broadly or carry a typed pre-factorization check for zero diagonals in the permuted system.
- `SparseLDL.D` (the diagonal factor array) is a private field with no public accessor. The sign distribution of D — which reveals the inertia (number of positive, zero, and negative eigenvalues) — cannot be read from a completed factorization. Auditing inertia from an LDL factorization requires either a custom subclass or re-implementing the factorization loop.

[COLUMN_ORDERING_GUARD]:
- The `ColumnOrdering` enum is ordered `Natural = 0, MinimumDegreeAtPlusA = 1, MinimumDegreeStS = 2, MinimumDegreeAtA = 3`. `SparseCholesky.Create` and `SparseLDL.Create` guard with `if (order > ColumnOrdering.MinimumDegreeAtPlusA) throw ArgumentException`, rejecting both `MinimumDegreeStS` and `MinimumDegreeAtA`. `SparseLU.Create` accepts all four orderings; the guard is absent. Passing `MinimumDegreeStS` to `SparseCholesky` throws immediately with `Resources.InvalidColumnOrdering`; dynamic ordering selection must handle this at the boundary.
- `AMD.Generate` with `ColumnOrdering.Natural` short-circuits to the identity permutation without any AMD computation. This is the correct path when the matrix is already optimally ordered. The identity permutation still triggers full symbolic analysis in the subsequent `SymbolicAnalysis` call.

[SPARSE_QR_OVERDETERMINED]:
- `SparseQR.Create(A, order)` branches on `A.RowCount >= A.ColumnCount`. For overdetermined systems (m ≥ n), AMD runs on A directly; for underdetermined systems, it transposes A first and runs AMD on `A^T`. The solve direction reverses accordingly. `S.m2` tracks the augmented row dimension used for structural singletons — the actual allocation for Householder vectors is `S.m2`, not `A.RowCount`. For rank-deficient systems, `S.m2 > A.RowCount` due to structural singletons introduced during symbolic analysis.
- For QR, fill evidence must be derived from `S.lnz` (fill in Q, as Householder vectors) and `S.unz` (fill in R) from the `SymbolicFactorization` field — which is private, making it inaccessible through the public surface after factorization.

[ISYMMETRIC_COST_AND_COMPLEX_DISTINCTION]:
- `CompressedColumnStorage<double>.IsSymmetric()` iterates over all nonzeros and for each `(i, j, v)` calls `At(row, col)`, which is a binary search over `ColumnPointers[column]..ColumnPointers[column+1]` — O(log(nnz_per_column)) per lookup. Total cost is O(nnz × log(nnz_max_column)), not O(nnz). For matrices with dense columns — after Schur complement formation — this is significantly more expensive than a linear pass. The result is not cached.
- The Complex override of `IsSymmetric()` tests Hermitian symmetry (`v.Equals(Complex.Conjugate(At(i, row)))`), while the double override tests real symmetry (`v.Equals(At(row, col))`). A caller holding a base `CompressedColumnStorage<T>` reference loses the conjugation test for complex matrices.

[DULMAGE_MENDELSOHN_STRUCTURAL_ANALYSIS]:
- `DulmageMendelsohn.Generate<T>(matrix, seed)` computes the coarse Dulmage-Mendelsohn decomposition via maximum bipartite matching. The `StructuralRank` property returns `rr[3]` — the size of the maximum matching, equal to the structural rank. For a matrix with `StructuralRank < n`, no column ordering produces a non-singular factorization; the matrix is structurally singular and the gauge-handling policy applies regardless of numeric values.
- `DulmageMendelsohn.Singletons` counts 1×1 diagonal blocks in the block triangular form — variables that can be solved sequentially without entering a coupled subsystem. For near-diagonal systems, `Singletons` approaching `Blocks` indicates near-cascade-solvability with O(n) direct solves rather than a full n×n factorization. `CoarseRowDecomposition` and `CoarseColumnDecomposition` (the `rr` and `cc` arrays) partition the matrix into fine-horizontal, structurally-underdetermined, square-full-rank, structurally-overdetermined, and fine-vertical parts via five index intervals.

[GROWTH_AND_AUTOTRIM_IN_CSR_EDIT]:
- `SparseCompressedRowMatrixStorage<T>.At(row, column, value)` for a new structural entry inserts at the binary-search insertion point, shifting the suffix of `Values` and `ColumnIndices` via `Array.Copy`, and increments all row pointers above `row`. This is O(nnz) in the worst case. When `AutoTrimStorage = true`, every `Keep` call also calls `Resize(0)`, trimming arrays to exactly `NonZerosCount`. The sequence `Keep → At → Keep` triggers two O(nnz) array copies. Batching all structural modifications into a single `CoordinateStorage` and re-converting is cheaper when two or more structural edits are needed.
- `SparseCompressedRowMatrixStorage<T>.RemoveAtIndexUnchecked` triggers a defensive trim: `if (num > 1024 && num < Values.Length / 2) { Array.Resize(ref Values, num); Array.Resize(ref ColumnIndices, num); }`. The trim fires only when the array is more than half-empty after removal and has at least 1024 entries, creating a step-function memory profile for large matrices under progressive structural editing.

[BICGSTAB_BREAKDOWN_EXCEPTION]:
- `BiCgStab.Solve` throws `NumericalBreakdownException` when `ρ_k ≈ 0` — the test is `num.AlmostEqualNumbersBetween(0.0, 1L)`, a 1-ULP tolerance check. `TrySolveIterative` does not catch this throw. For indefinite systems where BiCgStab tends to stall, this exception propagates silently through the `TrySolveIterative` boundary.
- `TFQMR.Solve` handles near-zero inner products by calling `iterator.Cancel()` and breaking out of the loop, leaving `IterationStatus.Cancelled` in the iterator; it does not throw `NumericalBreakdownException`. The caller therefore encounters two incompatible failure modes for the same mathematical breakdown — a status return versus an unhandled exception — with no syntactic difference at the call site. A safe outer boundary catches `NumericalBreakdownException` from `TrySolveIterative` regardless of solver choice; TFQMR's `Cancelled` result also requires a true-residual recomputation to confirm the partial result is unusable.

[PROGRESS_REPORTING_IN_FACTORIZATION]:
- All four `Create(..., IProgress<double> progress)` overloads in CSparse call `progress?.Report((double)i / (double)n)` only at 1% granularity — they accumulate a counter incremented by `n/100.0` and report only when `i >= num`. For small matrices (n < 100), no `Report` call may be made at all. `IProgress<double>` carries no cancellation signal — it is reporting only. Cancellation during factorization requires running the factorization on a `Task` with external `CancellationToken` control; the factorization itself has no cooperative cancel point.

[COORDINATE_STORAGE_INTERNAL_CONTRACT]:
- `CoordinateStorage<T>.At(i, j, value)` unconditionally appends: `rowind[nz] = i; colind[nz] = j; values[nz] = value; nz++`, resizing by factor 2 when `nz == nzmax`. Reading `Values`, `RowIndices`, and `ColumnIndices` before conversion is valid only if the caller accounts for possible duplicates. The `inplace: true` conversion flag calls `storage.Invalidate()` after building the CSC representation — `Invalidate` sets `nz = 0` and nulls the internal arrays, leaving any outstanding references as dangling (but GC-alive) pointers into data now owned by the CSC matrix.

[KEEP_RETURN_VALUE_AND_AUTOTRIM]:
- `CompressedColumnStorage<T>.Keep(Func<int,int,T,bool>)` returns the count of surviving entries after pruning. When `AutoTrimStorage = true`, `Keep` also trims `RowIndices` and `Values` to exactly the surviving count. For the pattern of pinning a node by clearing off-diagonal entries, `Keep((i, j, v) => !(isPinned[i] && j != i) && !(isPinned[j] && i != j))` combined with an `At` pass to set boundary conditions is a complete in-place structural edit without reassembly — provided `AutoTrimStorage` is temporarily disabled during the sequence to avoid the intermediate trim between `Keep` and `At`.
