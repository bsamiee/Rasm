# [CLI_DISPATCH_ADVANCED]

[CONVERTER_AND_TOKEN_MODEL]:
- `Token` is a frozen attrs class (`cyclopts/token.py`, 4.16.1): `keyword: str | None`, `value: str`, `source: str`, `index: int`, `keys: tuple[str, ...]`, `implicit_value: Any = UNSET`. The `source` field carries the origin label (e.g. `"env"` for `Env`, file path for `Toml`/`Json`) and is preserved through to validation errors, enabling precise provenance in structured error reporting. `token.address` is a `(keys, index)` tuple hashable for subkey-destination deduplication. [cyclopts/token.py 4.16.1, installed, 2026-06-09]
- The public `convert(type_, tokens, converter, name_transform)` function in `_convert.py` is a total dispatch tree: `Literal` coerces left-to-right through allowed values; `Union` tries variants in declaration order swallowing all exceptions; `tuple[T, ...]` sums `token_count` per element; `Flag` reduces via bitwise-or across all tokens; bare abstract types (`collections.abc.Set`, `Sequence`, etc.) normalize to concrete defaults; structured types (dataclasses, attrs, pydantic, `TypedDict`, `NamedTuple`) dispatch to `_convert_structured_type`, which respects `POSITIONAL_ONLY` parameter kinds; a single `{`-prefixed token for a structured target falls through a JSON path (`json.loads` → field-validated `_convert_json` → `CoercionError` with hint diagnostics). This is the bottom of every `Parameter(converter=...)` call chain. [cyclopts/_convert.py 4.16.1, 2026-06-09]
- `token_count(type_) -> (int, bool)` is the arity oracle consumed by the binder: the `bool` signals "consume-all" for positional remainder semantics. `n_tokens=-1` in `Parameter` forces consume-all regardless of type structure; `n_tokens=N` overrides and the consume-all flag is still derived from the underlying type. A discriminated `Annotated[Cat | Dog, pydantic.Field(discriminator="type")]` collapses to `(1, False)` unconditionally — one JSON token regardless of member field counts. [cyclopts/_convert.py lines 877–983, 2026-06-09]
- Custom converter signature has two calling conventions: a regular function receives `(type_, token_value: str) -> Any`; a bound classmethod/instancemethod (detected via `inspect.ismethod`) receives `(tokens_tuple,)` with `cls`/`self` already bound; a string value in `Parameter(converter="parse")` is resolved to `getattr(type_, "parse")` at bind time, enabling type-local factory routing without coupling the annotation to the instance. [cyclopts/_convert.py lines 559–584, 2026-06-09]
- `Parameter(validator=...)` normalizes to a tuple of callables or string references. Each validator receives `(type_, coerced_value)` for regular functions or `(coerced_value,)` for bound methods. Validators run after converter on the fully coerced value; `AssertionError`, `ValueError`, and `TypeError` convert to `ValidationError` carrying the original message. Combining converter + validator is the single composition point for domain admission at the annotation layer — no separate schema step. [cyclopts/_convert.py lines 733–748; parameter.py lines 200–207, 2026-06-09]

[NEGATIVE_FLAGS_AND_CONSUME_MULTIPLE]:
- `Parameter(negative=...)` controls the prefix tokens that reset a boolean or iterable to its absent/empty state. Default prefixes are `("no-",)` for `bool`, `("empty-",)` for iterables, and `()` for `Optional` parameters (explicit reset to `None`). The three fields `negative_bool`, `negative_iterable`, `negative_none` are independent overrides on `Parameter`; the converter in `parameter.py` normalizes each to `tuple[str, ...]`. Supplying `negative=()` disables negative flags entirely for a parameter. [cyclopts/parameter.py lines 274–295, 2026-06-09]
- `consume_multiple` normalizes to `tuple[int, int | None] | None` via `_consume_multiple_converter`: `True` → `(0, None)` (zero minimum, unbounded); an `int N` → `(N, None)` (minimum N, unbounded); a `(min, max)` sequence is validated (`min <= max`, non-negative) and passed through; `False` or `None` disables. This allows a single `--tag` option to accumulate an unbounded sequence or a bounded `(2, 5)` window without changing the parameter type annotation. [cyclopts/parameter.py lines 105–140, 2026-06-09]
- `Parameter(count=True)` turns a boolean flag into an integer occurrence counter (`-v -v -v` → `3`). `Parameter(allow_repeating=True)` permits the same keyword argument to appear multiple times (last value wins unless the type is a collection). `Parameter(requires_equals=True)` enforces `--opt=value` syntax, rejecting `--opt value`. These are orthogonal modifiers composable on the same annotation. [cyclopts/parameter.py lines 297–345, 2026-06-09]
- `Parameter(parse=False)` excludes a parameter from token consumption entirely — it will never be populated from CLI tokens or config providers. `Parameter(parse=re.Pattern)` restricts the parameter to matching tokens only, enabling regex-gated positional capture. [cyclopts/parameter.py lines 230–234, 2026-06-09]

[ENV_VAR_SPLITTING]:
- `env_var_split(type_, val, *, delimiter=None)` in `_env_var.py` is the default split strategy on `Env` and `Parameter(env_var_split=...)`. It calls `token_count(type_)` to decide: scalar types return `[val]` unsplit; iterable types split on `os.pathsep` when any inner type is `Path` (platform-aware `;`/`:`) and on whitespace otherwise. A custom `Parameter(env_var_split=my_fn)` replaces this at per-parameter granularity without changing the `Env` provider. [cyclopts/_env_var.py, 2026-06-09]
- `Env` matches environment variable keys to argument names via `arguments.match(candidate[len(prefix):], transform=_transform, delimiter="_")` where `_transform` uppercases and replaces `-`/`.` with `_`. When `argument.children` is empty, remaining key segments after the match are re-joined with `_` and treated as a scalar value rather than subkeys — resolving the structural ambiguity of `_` as both key delimiter and name separator. Tokens produced carry `source="env"` and `keyword=env_key` for traceable provenance. [cyclopts/config/_env.py lines 37–70, 2026-06-09]

[CONFIG_PROVIDER_PRECEDENCE_AND_LAYERING]:
- `App._config` (accessible as `app.config`) stores `tuple[_ConfigCallable, ...] | None`, where `_ConfigCallable = Callable[[App, tuple[str, ...], ArgumentCollection], Any]`. Each callable mutates the `ArgumentCollection` in-place by appending tokens. The `app.config` property resolves via `app_stack.resolve("_config")`, which walks the live `AppStack` from the innermost context frame outward — so a sub-app's config takes precedence over its parent's config. [cyclopts/core.py lines 287–302, 626–632, 2026-06-09]
- `ConfigBase.__call__` in `config/_common.py` walks `root_keys` then `commands` to reach the relevant dict node, then filters out sibling-command keys before calling `update_argument_collection`. When `use_commands_as_keys=False` the filter uses the root app to exclude top-level command names, enabling flat-namespace configs where subcommand names are not nesting keys. `allow_unknown=True` suppresses `UnknownOptionError` for config keys with no matching argument, enabling forward-compatible config files. [cyclopts/config/_common.py lines 55–77, 2026-06-09]
- `ConfigFromFile` caches via `FileCacheKey` (mtime + size + absolute path). If mtime and size match a prior read the cached dict is returned; if they diverge the file is re-read. `search_parents=True` walks `path.parents` looking for the filename in each ancestor — the standard "nearest config file wins" pattern for monorepo-aware CLIs. `must_exist=True` raises `FileNotFoundError` rather than silently returning `{}`. [cyclopts/config/_common.py lines 80–174, 2026-06-09]
- `config.Toml` uses `tomllib` (stdlib ≥ 3.11) with `tomli` as fallback; `config.Json` uses `json.loads`; `config.Dict(data)` is the in-memory test seam. All four share `ConfigBase` — a single `__call__` protocol — so they compose identically. Layering multiple providers is a tuple: `config=[Env("APP_"), Toml(".config.toml"), Dict(overrides)]`; last writer wins for a given argument because `update_argument_collection` appends tokens and later config entries append later. [cyclopts/config/_toml.py; config/_common.py, 2026-06-09]

[META_COMPOSITION_AND_META_CHAINS]:
- `app.meta` is a lazily-instantiated `App` (`core.py` lines 1017–1029) with `_meta_parent = app`. It inherits `help_flags`, `version_flags`, `group_*`, and `result_action` from the parent. `_walk_metas(app)` yields from deepest meta to shallowest: `[meta_meta, meta, app]` for a two-deep chain, enabling stacked global-option layers (e.g., `--config-file` at outer meta, `--log-level` at inner meta). [cyclopts/core.py lines 238–247, 2026-06-09]
- A meta-chain executes because `_combined_meta_command_mapping` merges commands from the meta and the meta's meta into one dict, walking `recurse_meta` and `recurse_parent_meta` flags. When a token does not match any command, `_consume_leading_meta_options` tries to bind it as a keyword argument of any meta app in `apps` that has `_meta_parent is not None`. This means meta-app options (`--config-file`) interleave with subcommand tokens without requiring a strict `--`-delimiter separation. [cyclopts/core.py lines 193–235, 1193–1251, 2026-06-09]
- The `meta.default` function receives the raw token stream for `*tokens: str` before any binding runs. The parent app's parameters are consumed from the leading stream as keyword arguments. The meta handler therefore has access to both the global options (bound via its own signature) and the full remaining token stream for programmatic interception. Returning an `int` from meta short-circuits the downstream execution and becomes the exit code; returning `None` allows execution to continue normally. [cyclopts/core.py `__call__` dispatch path; verified against wave-1 source, 2026-06-09]
- `_get_resolution_context(execution_path)` governs which apps contribute parameters to a help page or config resolution. It includes the final app in the path, all of its metas via `_walk_metas`, and any meta apps from the live `AppStack`. A subcommand's help page therefore inherits global meta parameters without re-registering them on the subcommand's own `App`. [cyclopts/core.py lines 1156–1191, 2026-06-09]

[DISPATCHER_PROTOCOL]:
- `Dispatcher` is a `typing.Protocol` with exact signature `__call__(self, command: Callable[..., Any], bound: inspect.BoundArguments, ignored: dict[str, Any], /) -> Any`. All three parameters are positional-only. `command` is the resolved handler function. `bound` is a fully populated `BoundArguments` — calling `bound.apply_defaults()` then `command(*bound.args, **bound.kwargs)` replicates the default execution path. `ignored` carries keyword arguments that had tokens but were not bound (e.g. `Parameter(parse=False)` fields). [cyclopts/protocols.py, 2026-06-09]
- Because `Dispatcher` only wraps the post-binding invocation, tracing, timing, and structured error lifting around a single command call are its natural concerns. Pre-binding work (auth, config loading, log-level injection) belongs in `app.meta.default`. The two interception layers are thus non-overlapping: meta owns raw token visibility and global state, Dispatcher owns typed-value visibility and per-invocation span wrapping.
- `App` accepts `dispatcher` as an `__call__` keyword parameter, not as a stored field — it is a per-invocation override, not a registered attribute. A global dispatcher is threaded by passing it from the outer entry point down through programmatic `app(tokens, dispatcher=span_dispatcher)` calls, not by mutating app state.

[RESULT_ACTION_AS_EXIT_RAIL]:
- `ResultAction = ResultActionSingle | Iterable[ResultActionSingle]` where `ResultActionSingle` is a 14-member `Literal` union or a `Callable[[Any], Any]`. When a sequence is supplied, `handle_result_action` pipelines left-to-right: each action receives the return value of the previous. A callable action is applied directly. The `__cyclopts_returncode__()` zero-argument protocol on the return value overrides the default `0` exit code in all built-in actions that produce a return code — making domain rail carriers (e.g. a typed result class) control exit semantics without coupling to cyclopts. [cyclopts/_result_action.py, 2026-06-09]
- The canonical ROP embedding pattern is `result_action=["return_value"]` with `exit_on_error=False, print_error=False`. This makes `app(tokens)` return the command's return value with zero side effects, enabling `Result.is_ok()` / `.is_error()` branch logic in the caller. Adding `__cyclopts_returncode__` to a domain result type then enables process-exit semantics when the same app is invoked directly from the CLI entry point without changing command logic.
- `result_action` accepts a sequence pipeline: `["print_non_int_return_int_as_exit_code", custom_log_fn]` first maps the result to an exit code, then passes that code through `custom_log_fn`. This makes result transformation a composable data pipeline rather than imperative wrappers around the command call.

[COMMANDSPEC_LIFECYCLE_AND_AOT_ALIGNMENT]:
- `CommandSpec` in `command_spec.py` is an attrs class with `import_path: str`, `name`, `app_kwargs: dict`, `help`, `sort_key`, `group`, `show`, and private `_resolved: App | None`. `_commands: dict[str, App | CommandSpec]` stores entries; `__getitem__` resolves on first access via `cmd.resolve(self)`. `resolved_commands()` eagerly resolves all lazy entries. Internal iteration (`__iter__`, `__contains__`, `format_usage`) avoids resolution, preserving import-time performance. [cyclopts/core.py lines 910–940, 976–1016; command_spec.py, 2026-06-09]
- `CommandSpec.resolve(parent_app)` caches the resolved `App` in `self._resolved` after the first `importlib.import_module` call. On an `App` target, `app_kwargs` must be empty (raises `ValueError`) and the App's `name[0]` must match `self.name` if set — name mismatches raise `ValueError` with an explicit remediation message. On a function target, a wrapper `App` is constructed with inherited `help_flags`, `version_flags`, and group defaults from the parent. Registration-time overrides (`help`, `sort_key`, `group`, `show`) are applied after resolution. [cyclopts/command_spec.py lines 59–169, 2026-06-09]
- AOT (declaration-first) registration pattern: `CommandSpec` entries are created at module import time with string keys and zero module loads. The dispatch table (`_commands`) is fully populated with string-keyed `CommandSpec` stubs. Help pages iterate without resolution. The only trigger for a real module import is token dispatch reaching that command in `parse_commands`. This aligns with plugin systems and monorepo CLIs where a root `app` module must not import all submodule dependencies at startup. A `CommandSpec` wrapping an `App` target validates name consistency at the single resolution site rather than scattering validation across registration call sites.

[SYNONYM_AND_FLATTEN_PATTERNS]:
- `App(synonym=...)` registers alternate names that are NOT runnable but trigger a "Did you mean..." error when typed. Unlike `alias`, synonyms do not appear in `_commands` and do not execute. This is the correct surface for deprecated command names that should redirect users without silently running. [cyclopts/core.py lines 333–343, 2026-06-09]
- `app.command(sub_app, name="*")` adds `sub_app` to `_flattened_subapps`. Flattened subapp commands appear in the parent namespace but parent-level commands take precedence on collision. `__iter__` yields flattened commands after local commands; `__contains__` checks flattened subapps last. This is the namespace-composition primitive — a domain sub-app can expose its commands into a root app without re-registration, while keeping the sub-app usable standalone. [cyclopts/core.py lines 471–473, 976–1016, 2026-06-09]
