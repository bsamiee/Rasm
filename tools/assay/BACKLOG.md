### Tri-tool charter (assay + rhino-bridge + cs-analyzer) тАФ binding for the rebuild session
The three tools form one quality surface and are held to one standard, enshrined here so the next session starts from it (each tool's corpus carries its own specialization):
- **Agent-first/only.** Every consumer is an AI agent, never a human. Concretely: one structured envelope per invocation; diagnostics that carry the FIX, not just the finding (cs-analyzer emits the doctrine-compliant rewrite an agent applies; bridge failures carry the evidence an agent triages from тАФ no log spelunking); fail fast with typed faults over hangs; `--help`/README text written as agent operating contracts.
- **Monorepo-first, never coupled.** No hardcoded project paths, target names, or single-package assumptions (cs-analyzer rules describe semantic shapes per CLAUDE.md, never `Rasm.*` namespaces; assay's testkit is SUT-registered per package; the bridge serves any host-bound verification need, not one plugin). New packages integrate by declaration, not by editing tool internals.
- **Doctrine-bound.** Rebuild code per `docs/stacks/csharp/` (code doctrine + every root file) for the two C# tools, `docs/stacks/python/` for assay; all tool docs to page-craft standards. Minimal surface/dispatch shapes; external packages first (add to the plan where they own a concern); no boilerplate rule/verb authoring тАФ adding a rule or verb is one descriptor row + one body + one test.
- **Hardened, non-fragile, performance-honest.** Leases serialize contended hosts (busy = clean exit-5 envelope, never a hang under N concurrent agents); resource governors cap fan at every tier; no orphaned processes; benchmarks are real and regression-gated; mutation runs are staged, root-clean, and feed harden-or-delete decisions; per-file coverage floors and law MANIFEST gates stay binding.

Two orthogonal axes, named once and never conflated: **execution target** (`exec_target` тАФ where the tool process runs) and **storage target** (`ArtifactBackend` тАФ where artifacts/history live). A cloud-drive path is storage-remote with LOCAL execution (artifacts "remote" only in the fsspec sense); a VPS offload is execution-remote, and there artifacts default to the TARGET (co-located with execution тАФ that is the optimization; a flag overrides, but no flag forest). Invariants regardless of axis values: the calling agent ALWAYS receives the envelope/diagnostics/status locally and automatically (the agent-feedback law above), and path semantics are uniform root-down so commands AND automations behave identically across local/ssh/cloud. Sequence: (a) prove `exec_target=ssh://` end-to-end live (loopback + real host); (b) land the execution-side artifact placement + path translation; (c) decide defaults deliberately тАФ LOCAL execution stays the default for agent ergonomics; offload is explicit opt-in (one flag or a verb form, e.g. heavy lanes only: mutation FULL, static full, dotnet build closures). Strategic end-state: multi-agent resource management тАФ an agent sees local pressure via the automatic resource feedback that already exists (`_sys_pressure` load/mem/swap sampling, stall monitor) and routes its heavy lane to a remote executor; a future automation can do the same routing on pressure thresholds. ## Upstream issues worth filing

### [UPSTREAM] mutmut stats-phase crash тАФ fixed locally (`max_stack_depth = -1`); issue still worth filing
The previously documented rootdir/cwd theory is **falsified** (the captured stats argv is well-formed and pytest loads the parent pyproject config тАФ the prepended addopts prove it). Root cause, established empirically with `debug=true`: with `max_stack_depth != -1`, `record_trampoline_hit` (`mutmut/__main__.py:130`) walks caller frames calling `Path(co_filename).resolve(strict=True)`; module-level calls into mutated code during conftest import put `<frozen importlib._bootstrap>` pseudo-filenames on that stack, the strict resolve raises `FileNotFoundError`, the conftest import dies, pytest exits 4, and mutmut raises `BadTestExecutionCommandsException`. The local fix is the upstream-default `max_stack_depth = -1` (repo had 10), which disables the strict-resolve stack walk entirely. Anyone with `max_stack_depth != -1` plus import-time calls into mutated code still hits this тАФ file upstream as-is:

> **TITLE**: record_trampoline_hit crashes with FileNotFoundError on '<frozen importlib._bootstrap>' frames when max_stack_depth != -1 and mutated code runs at import time.
> **BODY**: With max_stack_depth set (e.g. the README's max_stack_depth=8), record_trampoline_hit (mutmut/__main__.py, the `while c and f:` loop) calls Path(f.f_code.co_filename).resolve(strict=True) on every caller frame. When a mutated function is invoked at module import time (module-level constant computed by a mutated function, imported transitively from a conftest), the caller stack contains importlib machinery frames whose co_filename is the pseudo-filename '<frozen importlib._bootstrap>' (also '<frozen importlib._bootstrap_external>', '<string>'). strict=True raises FileNotFoundError('<cwd>/<frozen importlib._bootstrap>'), the conftest import aborts, pytest exits 4, and mutmut raises BadTestExecutionCommandsException тАФ misleadingly suggesting the pytest command itself is malformed.
> **REPRO (minimal)**: src/m.py with `def cap() -> int: return 1`; src/pkg using `X = cap()` at module level; a conftest.py importing pkg; [tool.mutmut] source_paths=['src'], max_stack_depth=8; `mutmut run` -> stats phase exit 4 with the traceback above; with max_stack_depth=-1 the run proceeds.
> **OBSERVED ON**: mutmut 3.6.0, pytest 9.0.3, CPython 3.15.0b1, macOS.
> **FIX CANDIDATES**: (a) skip frames whose co_filename starts with '<' before resolving; (b) resolve with strict=False; (c) wrap the resolve in try/except FileNotFoundError and continue; (d) break on importlib frames the same way 'pytest'/'unittest' filenames break the walk. Additionally, the exit-4 path would be far easier to debug if BadTestExecutionCommandsException included the captured pytest output even without debug=true.

When filing, also mention the forked-children consequence: mutant workers fork from the post-clean parent, so any set-once global poisoned across in-process `pytest.main()` calls silently converts test-infrastructure failures into false "killed" verdicts тАФ an argument for subprocess-isolated stats/clean phases in mutmut core.
