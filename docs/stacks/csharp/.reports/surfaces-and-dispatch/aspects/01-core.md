# Aspect Architecture

[DEFINITION_TIME_WEAVING]:
- Attribute-directed source generation is the canonical definition-time aspect mechanism: the attribute on a `partial` type declaration is the weave point, and the generator emits a second partial file containing all cross-cutting members. No runtime proxy, no IL rewriting, no container interception.
- `[SmartEnum<TKey>]`, `[Union]`, `[ValueObject<T>]`, and `[ComplexValueObject]` each declare the type's generated surface at definition time; the emitted partial file carries equality, conversion operators, `Switch`/`Map` dispatch, `ISmartEnum<TKey,T,TValidationError>`, `IParsable<T>`, `ISpanParsable<T>`, `IFormattable`, arithmetic operator interfaces, and serialization converters — all woven from a single declaration attribute without post-compilation steps.
- `SwitchMapMethodsGeneration.DefaultWithPartialOverloads` controls whether the generator additionally emits partial overloads of `Switch`/`Map` alongside the full-coverage overloads, giving the hand-written half a seam to override specific arms while the generated dispatch covers the remainder.
- `SwitchMapStateParameterName` (default `"state"`) is an attribute property on `[SmartEnum]`, `[SmartEnum<TKey>]`, and `[Union]` that renames the thread-through parameter in every generated `Switch`/`Map` overload — aspect payload naming is therefore owned entirely by the declaration, not by downstream call sites.
- `NestedUnionParameterNameGeneration.Simple` vs. `Default` on `[Union]` controls whether intermediate type names appear in Switch/Map parameter names for hierarchical unions; `Default` produces `failureNotFound`, `Simple` produces `notFound`. The parameter name shape is load-bearing for callers and set once at the union's declaration site.
- The `[ObjectFactory<T>]` generic attribute (itself a C# 11+ generic attribute) declares an additional factory route at definition time: `UseForSerialization`, `UseWithEntityFramework`, and `UseForModelBinding` each steer the corresponding cross-cutting integration aspect to route through that specific factory path rather than the primary key-based factory.
- `FactoryMethodGeneration.Default` (on ad-hoc `[Union<T1,T2>]`) auto-detects when factory methods must be generated — triggers include type-parameter members, duplicate types, `object`, and interface arms — so the decision of whether factory methods exist as a surface aspect is derived from the type structure, not from a manual flag.

[CONSTRUCTION_ADVICE_SEAMS]:
- Generated partial hooks on `[ValueObject<T>]` and `[ComplexValueObject]`: `static partial void ValidateFactoryArguments(ref ValidationError? validationError, ref TKey value)` is the primary construction advice point. The `ref` on both parameters is load-bearing: `ref validationError` lets the implementation set the error without allocation overhead, and `ref value` lets the hook normalize the incoming raw value (casing, trimming, clamping) before the generated factory stores it. This is the admission-time cross-cutting weave — it executes inside the generator-produced `Create`/`TryCreate` factory, before the type is instantiated.
- The validation error type is itself a typed aspect payload: `IValidationError<T>` requires `static abstract T Create(string message)`, constraining the error type as a static-abstract factory rather than a base class. Custom error types that implement `IValidationError<TCustom>` with `TCustom : class, IValidationError<TCustom>` replace the default `ValidationError` across the entire generated factory surface simultaneously by naming the custom type in the `[ValueObject<T>]` declaration.
- `[UseDelegateFromConstructor]` on a `partial` method inside a `[SmartEnum]` class is a per-method aspect payload: the generator synthesizes a private delegate field with the method's matching signature, wires it through the constructor, and implements the `partial` method body as an invocation of that delegate. The `DelegateName` property forces a custom delegate type name regardless of parameter ref-kinds, which matters when `ref` or `out` parameters on the abstract method would otherwise cause the generator to produce a Func/Action shape that cannot represent the exact signature.
- Partial constructors (C# 14, feature flag `partial events and constructors`) split a constructor into declaring and implementing parts. A generator can emit the declaring part (signature and attributes only), and the hand-written file provides the body — or vice versa. The constraint is that only the implementing declaration may carry a constructor initializer (`: this(...)` or `: base(...)`), confirmed by the Roslyn diagnostic `ERR_PartialConstructorInitializer`. This makes partial constructors the right seam for source-generated initialization chains where the generator controls parameter admission and the hand-written half controls delegation to base.
- Partial properties (C# 13+, feature flag `partial properties`) and partial events (feature flag `partial events and constructors`) extend the same split-declaration model to properties and events. A partial property's definition part may not have an auto-property body (`ERR_PartialPropertyDuplicateDefinition` fires if it does), and any accessor declared in the definition part must be implemented in the implementing part (`ERR_PartialPropertyMissingAccessor`). Partial properties are therefore strictly header/body splits, not get/set splits — the definition owns the accessor contract, the implementation owns accessor bodies.
- The distinction between extended partial methods and legacy partial methods is compiler-enforced: a partial method with a non-void return type, `out` parameters, or `virtual`/`override` modifiers must carry accessibility modifiers (`ERR_PartialMethodWithNonVoidReturnMustHaveAccessMods`, etc.), and any accessible partial method must have an implementation part. Legacy partial methods may omit the implementing declaration entirely and are silently dropped; extended partial methods cannot. This means definition-time advice points for generators must use extended partial methods (with access modifier) when they need a non-void return or must guarantee execution.

[GENERIC_ATTRIBUTES_AS_TYPED_ASPECT_PAYLOAD]:
- `[SmartEnum<TKey>]` and `[ObjectFactory<T>]` are both generic attributes (C# 11, feature flag `generic attributes`). Generic attributes carry the type argument as a verified compile-time type rather than a `Type` parameter, eliminating the `typeof(T)` pattern and enabling analyzer-side type checks during source generation without reflection at generation time.
- `[ObjectFactory<T>]` when `T` is `ReadOnlySpan<char>` routes the factory through span-based parsing, which on .NET 9+ is the zero-allocation admission path. This is a typed aspect because the generator emits the span-based `IObjectFactory<Owner, ReadOnlySpan<char>, TError>` interface, which Roslyn's `StaticAbstractInvoker` uses without heap allocation; naming a non-span `T` produces a different generated interface, so the choice of `T` is the discriminant.
- `[KeyMemberComparerAttribute<TComparer, TKey>]` and `[KeyMemberEqualityComparerAttribute<TComparer, TKey>]` are generic attributes that attach a comparer type to the key member at definition time. The generator reads `TComparer` and wires the comparer into generated equality and comparison operator implementations. The hand-written `ComparerAccessors.StringOrdinal`, `OrdinalIgnoreCase`, etc. implement the `IComparerAccessor<T>`/`IEqualityComparerAccessor<T>` interfaces, providing a fully type-safe comparer injection path without any constructor or property injection at runtime.

[SERIALIZATION_GRAMMAR_IDENTITY_ASPECTS]:
- Serialization is a definition-time aspect: `System.Text.Json`, Newtonsoft.Json, MessagePack, EF Core value converters, ASP.NET Core model binding, and Swashbuckle schema filters are all generated as partial class members from the same declaration attribute, conditioned by the presence of their respective companion packages (`Thinktecture.Runtime.Extensions.Json`, etc.). No runtime registration, no DI setup — the converter types are emitted into the type itself via the source generator and discovered by the framework's reflection-based or source-generated registration.
- `SkipFactoryMethods = true` on `[ValueObject<T>]` suppresses the entire generated serialization surface unless an `[ObjectFactory<T>]` with `UseForSerialization` is present. This is the admission gate: without a factory route, no serializer can know how to reconstruct the value object, so the aspect is withheld rather than generating broken converters.
- `ConversionOperatorsGeneration.Implicit` vs. `Explicit` vs. `None` on `[ValueObject<T>]` and `[SmartEnum<TKey>]` controls the generated `op_Implicit`/`op_Explicit` operators, which are the identity-at-boundary aspect: implicit conversion means the raw key type and the value-object type are structurally interchangeable at call sites, while explicit forces an explicit cast and keeps the domain boundary visible in calling code.
- `OperatorsGeneration.DefaultWithKeyTypeOverloads` on equality/comparison operators generates overloads that accept the raw key type alongside the value-object type. This is a mixed-precision aspect: it allows comparisons between the primitive and the domain type without requiring a wrap/unwrap at the call site, at the cost of potentially admitting unvalidated primitives into comparisons. The law is to use `Default` (domain-type-only overloads) unless interop with unvalidated key values is explicitly required.
- `NullInFactoryMethodsYieldsNull` and `EmptyStringInFactoryMethodsYieldsNull` on `[ValueObject<string>]` are presence-absence aspects: `null` or empty string input produces `null` output from `Create`/`TryCreate` rather than a validation error, converting null-propagation into the generated factory's contract without a hand-written null check.

[COMPOSITION_TIME_POLICY]:
- `Schedule` is a lazy infinite stream of `Duration` values. Composition operators are `|` (union: minimum delay, length of the longer) and `&` (intersection: maximum delay, length of the shorter). These are the two algebra primitives: a retry-with-cap is `Schedule.recurs(n) | Schedule.exponential(base) & Schedule.spaced(cap)` — the intersection enforces the cap per step, the union terminates at `n`. Operators compose without allocation until the stream is consumed by `retry`/`repeat`.
- `retry(Schedule, IO<A>)` and `retryWhile(Schedule, IO<A>, Func<Error,bool>)` are the composition-time retry aspects on the `IO` monad. The `Func<Error, bool>` predicate in `retryWhile` fires on every failure; returning `false` terminates the retry loop regardless of whether the schedule has remaining steps. `retryUntil` has the inverted predicate polarity — it stops when the predicate returns `true`.
- `CatchM<E, F, A>` is the typed catch combinator: a predicate `Func<E, bool>` paired with a handler `Func<E, K<F, A>>`. `@catch(Error, Func<Error, K<F, A>>)` constructs a `CatchM` keyed to a specific `Error` value (equality-matched); `@catch(int, ...)` constructs a `CatchM` keyed to the error code; `@catch(Func<Error, bool>, ...)` is the predicate-based form. `catchOf<E, F, A>(Func<E, K<F, A>>)` matches all errors that are subtypes of `E`. The `|` operator on `K<F, A>` with a right-hand `CatchM` applies the catch combinator inline, producing a pipeline step: `computation | @catch(specificError, recovery)`.
- `Finally<F, A>` is the composition-time finally combinator. `K<F, A> | finally(K<F, B>)` executes the `K<F, B>` regardless of whether the primary computation succeeds or fails, and returns the primary result. The `|` operator is defined on every monad transformer stack (`FinT`, `OptionT`, `EitherT`, `ReaderT`, `StateT`, `WriterT`, `ChronicleT`, `RWST`), not only on bare `IO` and `Eff`. This means the `finally` aspect composes uniformly across the entire transformer tower without unwrapping.
- `bracketIO(IO<A>)` creates a local resource scope: any resource acquired via `use(...)` inside the bracket is automatically disposed when the scope exits, whether by success or failure. The resource tracking is embedded in `EnvIO`; there is no explicit `finally` call required. `bracketIO(acq, use, fin)` is the explicit three-argument form with separate acquire, use, and release functions; `bracketIO(acq, use, @catch, fin)` adds an error handler inside the acquire computation. The documentation explicitly recommends the zero-argument `bracketIO(computation)` form over the three-argument form when `use` handles resources registered with `use(...)`.
- The law separating definition-time from composition-time: admission, identity, dispatch, serialization, and grammar attach at definition time because they are properties of the type itself — they must be present at every call site, cannot be added or removed per-invocation, and are represented in the generated interface surface that callers depend on. Retry, recovery, and resource lifetime attach at composition time because they are properties of the call graph at a specific site — the same `IO<A>` can be retried with different schedules at different call sites, and the type carries no trace of the policy.

[INTERCEPTORS_AND_MODULE_INITIALIZERS]:
- Interceptors replace specific call sites at compile time: `[InterceptsLocation(version, data)]` on a static method routes invocations at the location encoded in `data` to the interceptor instead. The Roslyn API `SemanticModel.GetInterceptableLocation(InvocationExpressionSyntax)` returns the opaque `InterceptableLocation` value (version + data string), and `CSharpExtensions.GetInterceptsLocationAttributeSyntax(InterceptableLocation)` produces the attribute syntax for the generated file. This is a source-generator-only weave surface — hand-written interceptors require manually correct location data, which is fragile and unsupported in that form.
- Interceptors are constrained to invocation expressions (not constructor calls, per the Roslyn comment distinguishing interceptors from instrumenters). The intercepting method's signature must match the intercepted method's signature; a mismatch produces `ERR_InterceptorSignatureMismatch` (error) or `WRN_InterceptorSignatureMismatch` (warning, for compatible-but-not-identical cases). Location encoding is versioned (`InterceptableLocation.Version`); the version integer is an argument to `[InterceptsLocation]` so that future encoding changes do not silently corrupt existing interceptors.
- `[ModuleInitializer]` marks a static, non-virtual, parameterless, void-returning method as a CLR module initializer — it runs once before any code in the module accesses types from that module, guaranteed by the CLR. Constraints enforced by Roslyn: method must be accessible at module level, must not be generic, must not be in a generic type, and must be an ordinary method (not a property accessor, constructor, etc.). As a weaving surface it is the last-resort aspect for registration, codec installation, and process-static initialization that must be unconditional and before first use.
- The `[ModuleInitializer]` ordering across assemblies is undefined; within a single assembly, module initializers run in source-file and then declaration order within files. Aspect systems that depend on ordering (e.g., codec registrations that must happen before the first JSON operation) must not split registration across assemblies using module initializers unless they control the load order explicitly.

[STACKING_ORDER]:
- Definition-time aspects generated by Thinktecture emit into a single partial file per source-generator run. Within that file, the ordering of generated members follows the generator's internal declaration order, not the attribute order. When multiple attributes of different kinds coexist on the same type (e.g., a `[SmartEnum<string>]` with `[KeyMemberEqualityComparerAttribute<OrdinalIgnoreCase, string>]`), each attribute's generator pass is independent; the combined output is the union of both passes' partial file contributions. No stacking conflict arises because each pass targets disjoint member names.
- Composition-time aspects stack left-to-right through the monadic pipeline. `computation.Catch(...) | @catch(...) | finally(cleanup)` stacks in evaluation order: the first `.Catch` handles errors from `computation`, the `| @catch(...)` handles errors from the first catch handler, and `| finally(cleanup)` runs unconditionally after the entire left-hand side. The law is: `finally` is always the outermost `|` operand; a `@catch` between `finally` and the computation is unreachable for errors that propagate through the catch handler, because `finally` does not swallow errors, it re-propagates them after executing the cleanup.
- When a `[Union]` `Switch` dispatch carries per-case `[UseDelegateFromConstructor]`-style behavior, and a composition-time retry wraps the dispatch call site, the retry policy sees the entire dispatch (including whichever case arm executed) as one atomic unit. There is no per-case retry granularity without extracting the case into a separate `IO` computation before the switch. This is the fundamental stacking constraint: definition-time case dispatch and composition-time error handling are orthogonal and do not compose at case granularity.
