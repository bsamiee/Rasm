# State-Receipts: Core Findings

[ATOM_INTERNALS]:
- `Atom<A>` stores its value as a `volatile object` field; `Box<A>` dispatches at static-ctor time to either a CIL-emit path (unboxed field read for value types) or an identity cast (reference types), so struct-typed atoms avoid allocation on read but still box on write; the `Box<A>.New` delegate is emitted once per `A` via `DynamicMethod` under `ILCapability.Available`, with a fallback cast for platforms without emit.
- `Swap` is a CAS spin loop: it reads `value`, applies `f`, calls `validator(newVal)`, and then `Interlocked.CompareExchange`s; if the exchange fails it calls `spinWait.SpinOnce()` and retries, so `f` **will** be called multiple times under contention — any side effect inside `f` is repeated. `SwapMaybe` breaks the loop early when `f` returns `None`, leaving the atom unchanged and returning the latest value.
- `Change` event fires synchronously on the calling thread, after the CAS succeeds and before `Swap` returns. There is no buffering or dispatch queue; a slow handler directly throttles the swapper.
- `Atom<A>(A value)` (no validator) returns `Atom<A>` directly. `Atom<A>(A value, Func<A, bool> validator)` returns `Option<Atom<A>>` — the `None` case means the initial value failed the validator; construction succeeds only when the validator accepts the seed value.
- `atomIO<A>(A, Func<A, bool>)` lifts construction into `IO<Option<Atom<A>>>` — when the validator rejects the seed value, the `IO` resolves to `None`, not an exception. The no-validator overload `atomIO<A>(A)` returns `IO<Atom<A>>` directly. Callers that require a hard failure on invalid seed must map the `None` case manually after evaluating the `IO`.
- `Atom<M, A>` carries metadata `M` as a `private readonly M metadata` field — it is **not** exposed as a public property. Metadata is threaded only through `Swap(Func<M, A, A>)` and `SwapIO(Func<M, A, A>)`, so the validator and the swap function can inspect context but consumers cannot read it back out of the atom.
- `SwapIO(Func<A, A>)` and `SwapMaybeIO(Func<A, Option<A>>)` are trivial `IO.lift` wrappers over the synchronous `Swap`; they do not defer or schedule the CAS — the actual spin loop runs at `IO` evaluation time, on the evaluating thread.

[ATOM_COLLECTION_VARIANTS]:
- `AtomHashMap<K, V>` exposes two `Swap` paths. The scalar-mutation path (`Add`, `Remove`, `SetItem`, `SwapKey`, etc.) internally runs its own CAS spin loop, emits a `HashMapPatch<K, V>` carrying `From`, `To`, and a `HashMap<K, Change<V>>` for the single key. The bulk path `Swap(Func<TrackingHashMap<K,V>, TrackingHashMap<K,V>>)` also CAS-spins, but wraps the current state in a `TrackingHashMap` (a `readonly struct` that records every mutation as a `Change<V>`) before passing it to the function; the resulting tracking map's `ChangesInternal` is diffed into a `HashMap<K, Change<V>>` and forwarded to `AnnounceChanges` as a batch patch.
- `Change<A>` is an `abstract class` with four sealed subtypes: `NoChange<A>`, `EntryAdded<A>`, `EntryRemoved<A>`, and `EntryMapped<A, B>` (implements `EntryMappedFrom<A>` and `EntryMappedTo<B>`). It implements `Monoid<Change<A>>` and `Semigroup<Change<A>>`, making patches compositional. `Change<A>.None` is the static `NoChange<A>.Default` identity.
- `AtomSeq<A>.Swap(Func<Seq<A>, Seq<A>>)` runs the same CAS spin loop against the internal `ISeqInternal<A>` field; the swap function receives a `Seq<A>` wrapping the current internal and must return a new `Seq<A>` whose `.value` field becomes the next internal. There is no tracking wrapper — no diff is emitted.
- `AtomQue<A>` exposes `Enqueue(A)`, `Option<A> Dequeue()`, `Option<A> Peek()`, `A DequeueUnsafe()`, and `Unit Clear()`, each running an independent CAS spin; there is no transactional multi-step access and no `Swap` overload that accepts an `Func<Que<A>, Que<A>>`.
- `AtomHashMap` (static class) exposes `ToAtom<K,V>(this HashMap<K,V>)` as the canonical upgrade path from an immutable snapshot to a concurrent cell.

[REF_AND_STM]:
- `Ref<A>` is a `sealed class` backed by an internal `long Id` that keys into a process-global `AtomHashMap` (`STM.state`) of `RefState` values. The `Value` property setter calls `STM.Write(Id, value)`, which either writes into the current thread's `Transaction` or throws `InvalidOperationException` if called outside `atomic`/`atomicIO`.
- `atomic<R>(Func<R>, Isolation)` checks `STM.transaction.Value` (a `[ThreadStatic]`-equivalent `AsyncLocal`) first: if a transaction is already active it simply invokes `op()` and returns — nested `atomic` calls join the outermost transaction without creating a new retry scope. `RunTransaction` is called only for the root.
- `Isolation.Snapshot` (default): at commit, only the **written** `Ref`s are validated; reads of other `Ref`s inside the transaction are not checked for external mutation. `Isolation.Serialisable`: both read-set and write-set are validated; any external write to a read-from `Ref` triggers a `ConflictException` and retry.
- `commute<A>(Ref<A>, Func<A, A>)` stages the function in the transaction's `commutes` list rather than the `writes` map. Commutes are applied at commit time against the latest committed value of the `Ref`, not the transaction-local snapshot. This means commute operations never conflict with concurrent writes to the same `Ref` — they compose — but the order of concurrent commutes on the same `Ref` is unspecified.
- `ValidateAndCommit` acquires `STM.state.SwapInternal` (a lock-free CAS on the global state map) to atomically validate and commit all writes and commutes. A `ConflictException` thrown from `ValidateAndCommit` is caught by `RunTransaction`'s retry loop; a `RefValidationFailedException` (thrown when a `Ref`'s validator rejects the committed value) is **not** caught — it propagates out of `atomic` and must be handled by the caller.
- The `Eff`-integrated paths `STM.DoTransaction<RT, R>(Eff<RT,R>, Isolation)` run the full CAS retry loop inside an `Eff.Lift` delegate and return a `Fin<R>` — failed `Eff` exits do not commit. The public surface for `Eff`-based callers is `atomicIO` (wrapping `Func<R>`), not a dedicated `atomicEff`; wrapping an `Eff` chain requires materializing through `IO.Lift`.
- `Ref<A>` fires its `Change` event from `STM.OnChange`, which iterates the transaction's change patch after commit. Notification is synchronous and on the committing thread, but deferred until after `ValidateAndCommit` — unlike `Atom`, where `Change` fires immediately after the successful CAS.

[MEMO_SHAPES]:
- `Memo<A>` uses a three-state `int` field (`0 = unloaded`, `1 = loading`, `2 = loaded`) with a `SpinWait` loop and `Interlocked.CompareExchange` to implement once-only lazy initialization; the acquire function is executed exactly once. `Memo<F, A>` (the higher-kinded variant) uses the same state machine but stores `K<F, A>` — it is used as a lazy `Applicative` node.
- `memo<A, B>(Func<A, B>)` (function memoization) builds a closure over a `WeakDict<A, B>` — a `ConcurrentDictionary<A, WeakReference<OnFinalise<R>>>` where `OnFinalise` removes the entry on finalization. This means cache entries are collected when not strongly referenced by the caller, preventing unbounded growth for short-lived keys.
- `memoUnsafe<T, R>(Func<T, R>)` uses a plain `ConcurrentDictionary<T, R>` with no weak reference — values are held permanently. The "unsafe" name signals the memory-leak risk, not thread safety (both variants are thread-safe via a per-key `object` lock in a synchronized map).
- `memo<T>(IEnumerable<T>)` calls `toSeq(value)`, which calls `new Seq<A>(IEnumerable)` → `this(new SeqLazy<A>(ma))`. The result is a lazily-evaluated `Seq<T>` that materializes elements on traversal, not a precomputed snapshot. Callers expecting a frozen copy must force evaluation first.
- `memoK<F, A>(Func<K<F,A>>)` (and its `value` / `fa` overloads) are the higher-kinded lazy wrappers; `Memo<F, A>` participates in `Applicative.Apply` for all higher-kinded types in LanguageExt, enabling lazy composition of effect values without executing them at construction time.

[STATE_CELL_DISCIPLINE]:
- `Atom<T>` is the correct cell when state is independent, single-ownership, and never needs multi-variable consistency. `Ref<T>` earns its STM overhead only when two or more `Ref`s must transition atomically — a single `Ref` with `atomic(() => ...)` is identical to `Atom.Swap` in guarantees but adds transaction management overhead and requires explicit `atomic` wrapping at every write site.
- The `Atom<M, A>` form (metadata channel) exists for cases where the validator depends on stable external context that must not live inside the state value itself — for example, a capacity limit or schema version threaded alongside the state without being part of the domain value. Because `metadata` is unreachable after construction except through `Swap`, it cannot be updated; a requirement to update the metadata context signals a design that belongs inside the state value.
- `AtomHashMap` over `Atom<HashMap<K,V>>`: `AtomHashMap` provides fine-grained key-level mutations (`SwapKey`, `AddOrUpdate`, `FindOrMaybeAdd`) that `Atom<HashMap>` cannot express without replacing the whole map on every mutation. `AtomHashMap` also exposes the `Change` event with a typed `HashMapPatch<K,V>` diff; `Atom<HashMap>` fires its `Change` event with only the new whole-map value. When the change diff is needed by observers — or when key-level concurrent mutations are common — `AtomHashMap` is the owner.
- A fact stream folded with `slot`/`kind` metadata lives in `Atom<Seq<FactRecord>>` or `AtomSeq<FactRecord>` where `FactRecord` carries the kind discriminant and all evidence fields. The `AtomSeq.Swap(f)` path replaces the whole sequence atomically; there is no per-element CAS. For append-only streams under high contention the `Add(A)` method on `AtomSeq` runs an independent CAS per element.
- Typed algorithm receipts — carrying solver path, sampling parameters, spectral or mesh evidence, status flags — belong to `Atom<ReceiptRecord>` rather than an operational fact stream. The distinction is semantic: fact streams are accumulated observations projected by fold; receipts are evidence of a specific completed computation that a consumer can pattern-match for routing, replay, or audit.

[VALIDATOR_FIRST_WRITER_WINS]:
- The validator on `Atom<A>` runs on **every** CAS attempt, not just the first. If the validator is non-pure (reads external state), it may produce different answers on successive retries. A first-writer-wins pattern uses a validator that rejects if the value has already changed from a sentinel: `Atom<Option<T>>(None, v => v.IsNone || ...)`. `SwapMaybe` returning the `Option`-valued CAS result (the new value on success, the unchanged value on no-op) is the idempotency mechanism without a separate flag.
- `RefValidationFailedException` escapes the STM retry loop — it is not caught by `RunTransaction`. A `Ref` validator that rejects a transition is a hard abort of the `atomic` block, not a retry trigger. Use validators only for invariants that must never be violated; transitions that are conditionally skippable belong in the `commute` or `Swap` function logic, not the validator.
