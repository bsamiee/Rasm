# [03] Modern Analyzer Landscape — Cited Research + Recommended Patterns

2026-06 web research on building a bleeding-edge Roslyn analyzer, verified against current nuget.org, Microsoft Learn, the roslyn/roslyn-analyzers/roslyn-sdk repos, and the three reference analyzers (Meziantou, Roslynator, SonarAnalyzer). Every claim carries its source URL in §EVIDENCE. The recommended-pattern shortlist (§12) is the input to doc 04. Research-protocol note: all technical claims are 2026 sources; only stable spec mechanics (RS-rule semantics, DocumentationCommentId grammar) lean on settled official docs.

## [1]-[SDK_VERSION_POSTURE]

`Microsoft.CodeAnalysis` / `.CSharp` latest stable **5.3.0** (published 2026-03-10). `Microsoft.CodeAnalysis.Analyzers` (the meta-analyzer enforcing RS#### authoring rules) also **5.3.0** (2026-03-10). `Microsoft.CodeAnalysis.NetAnalyzers` (the CA pack in-SDK) at 10.0.300, tracking .NET 10. Roslyn→VS→SDK map (Learn, updated 2026-04-24): 5.0.0→VS 2026 v18.0; 4.14.0→17.14; 4.13.0→17.13; 4.12.0→17.12; 4.4.0→17.4 (the `ForAttributeWithMetadataName` floor). REBUILD POSTURE (agent-only, monorepo-internal, NOT shipped to external IDE consumers): the multi-target-against-old-VS concern does NOT apply — target the single repo-pinned Roslyn (5.3.0, already in `Directory.Packages.props:127-132`); skip the `roslynX.Y` multi-fan-out entirely. Reserve multi-targeting only if the analyzer must load in arbitrary external IDEs.

## [2]-[ANALYZER_VS_GENERATOR_SPLIT]

`DiagnosticAnalyzer` (override `SupportedDiagnostics` + `Initialize`) is the rule-enforcement surface; `IIncrementalGenerator` is a separate emission concern. RS1035 BANS legacy `ISourceGenerator`/`GeneratorExecutionContext` — "implement `IIncrementalGenerator` instead." For a standards-enforcement analyzer the rebuild needs `DiagnosticAnalyzer` only; the generator is irrelevant to enforcement (and is extracted, doc 04/07). Critical perf asymmetry (andrewlock part 9): the incremental-pipeline caching machinery is GENERATOR-ONLY — `DiagnosticAnalyzer` has NO equivalent incremental cache, so the equatable-data-model discipline transfers only partially. What DOES transfer: never hoist `ISymbol`/`SyntaxNode`/`Compilation` into long-lived state expecting equality; compute lazily.

## [3]-[REGISTRATION_AND_BATCHING_SEMANTICS]

Execution order is GUARANTEED (roslyn `Analyzer Actions Semantics.md`): CompilationStart first → SymbolStart brackets each symbol+members → CodeBlockStart brackets node actions in a block → CompilationEnd last. SyntaxNode/Operation actions have NO ordering guarantee relative to each other. THE canonical stateful pattern: `RegisterCompilationStartAction(ctx => { var state = new AnalyzerContext(ctx.Compilation); ctx.RegisterSymbolAction(state.AnalyzeX, SymbolKind.NamedType); ctx.RegisterCompilationEndAction(state.Finish); })` — per-compilation state instantiated inside the start callback so it never outlives the compilation. RS1008/RS1009 enforce statelessness on the analyzer TYPE (all mutable state in the per-compilation object); RS1012/RS1013 flag a CompilationStart that registers only an end action. `SymbolStartAction` is the batching primitive: fires once per symbol, registers nested node/operation + a symbol-end action, enabling cross-MEMBER accumulate-then-report with better parallelism than a global CompilationEnd — BUT only for symbol-scoped accumulation (08 B3: cross-TYPE accumulators like compilation-wide bitwise callsites cannot use SymbolStart). CRITICAL WARNING from the doc: "End actions cannot be relied upon to free resources" — hosts cancel compilations, so start actions may never get a paired end; never use end actions for disposal.

## [4]-[IOPERATION_VS_SYNTAX]

Roslyn docs RECOMMEND `IOperation`/`OperationAction` for semantic analysis of executable code: "improved performance and simpler, language-agnostic implementation" — the IOperation tree already carries bound semantics, avoiding a semantic-model round-trip per node. Tradeoff: IOperation is an abstraction, so language-specific syntax is invisible (recover via `IOperation.Syntax`). DECISION RULE: pure-syntactic rules (naming, declaration shape, forbidden keyword like the `if`/`for`/`while`/`switch` ban) → `SyntaxNodeAction` (cheapest, no semantics). Rules needing type/symbol/dataflow semantics (banned API call, allocation pattern, return-type contract) → `OperationAction`, NOT SyntaxNode-then-`GetSymbolInfo`.

## [5]-[PERFORMANCE_DOCTRINE]

(a) NEVER call `Compilation.GetSemanticModel` inside a node callback — it rebinds the entire file (documented 4-5x slowdown, roslyn#25259). The bindable AUTHORITY is RS1030 `DoNotUseCompilationGetSemanticModel` in `Microsoft.CodeAnalysis.Analyzers` — cite that (08 B2). Use the model handed in by `SemanticModelAnalysisContext`/`OperationAnalysisContext`. (b) `Initialize` MUST call `EnableConcurrentExecution()` (RS1026) and `ConfigureGeneratedCodeAnalysis(GeneratedCodeAnalysisFlags.None)` (RS1025) — None to skip generated code (typical for style rules; Thinktecture/source-gen partials carry `[GeneratedCode]`). (c) DiagnosticDescriptors MUST be static readonly immutable singletons reused across instances — never per-callback. (d) Roslyn's internal allocation strategy is `ObjectPool<T>`/`ArrayBuilder<T>.GetInstance()` (PooledObjects); custom analyzers prefer pooled/stack-local builders, avoid LINQ in hot node callbacks (foreach over LINQ). (e) `WellKnownTypeProvider` (in the Analyzers utility layer) caches `GetTypeByMetadataName` per-compilation — resolve well-known symbols ONCE in the CompilationStart state, never per node. (f) The IDE cancels running analyzers on keystroke; honor `context.CancellationToken`.

## [6]-[CONFIGURATION_SURFACE]

Per-rule + per-tree options via `.editorconfig` through `context.Options.AnalyzerConfigOptionsProvider`. Two scopes: `GetOptions(syntaxTree).TryGetValue("dotnet_diagnostic.XXXX.some_option", out v)` for TREE-scoped, and `.GlobalOptions.TryGetValue("build_property.RootNamespace", out v)` for project-wide/`.globalconfig` (MSBuild property bridge via `CompilerVisibleProperty`). The first-party CA convention namespaces options `dotnet_code_quality.*` / `dotnet_code_quality.CAxxxx.*`. AdditionalFiles metadata via `build_metadata.AdditionalFiles.<Key>`. For an agent-facing analyzer: expose severity/toggle through standard `dotnet_diagnostic.<ID>.severity`; reserve `AnalyzerConfigOptions` for genuine per-rule PARAMETERS (e.g. `csp.CSP0741.case_threshold`); use AdditionalFiles for DATA SETS (banned symbols). The per-tree `GetOptions(tree)` path is the mechanism behind the rebuild's `csp.scope` per-tree scope override (08 D2 → 04 §SCOPE).

## [7]-[TESTING_STATE_OF_THE_ART]

Canonical harness: `Microsoft.CodeAnalysis.CSharp.Analyzer.Testing` (+ `.CodeFix.Testing`) with per-framework adapters (`.XUnit`/`.MSTest`/`.NUnit`), latest 1.1.x. `DefaultVerifier` replaced the old per-framework verifier bases (`using Verify = CSharpAnalyzerVerifier<TAnalyzer, DefaultVerifier>`). `VerifyAnalyzerAsync` checks an EXACT diagnostic set; markup `{|ID:span|}` or `[|span|]` inlines expected diagnostics (no manual line/col). `VerifyCodeFixAsync` auto-runs three checks (diagnostics present, fix output matches, post-fix clean); use the codefix verifier ONLY for inputs that report diagnostics, `VerifyAnalyzerAsync` for clean inputs. The test object exposes `TestState`/`FixedState`, `ExpectedDiagnostics`, `ReferenceAssemblies` (pin TFM), and per-test `.editorconfig` via `TestState.AnalyzerConfigFiles` (the config-surface coverage mechanism). SonarAnalyzer's transferable shape: `VerifierBuilder<TAnalyzer>()` fluent builder + `TestCases/*.cs` annotated with `// Noncompliant {{message}}` / `// Secondary` comments — scales to hundreds of rules without per-case boilerplate. Snapshot/golden (Verify lib) is used by GENERATOR tests; for analyzers markup+annotation is state-of-the-art over snapshots.

## [8]-[PACKAGING_AND_MULTI_TARGETING]

NuGet layout: legacy version-agnostic `analyzers/dotnet/cs/<dll>`; since .NET 6 the SDK honors version-qualified `analyzers/dotnet/roslyn{major.minor}/cs/<dll>` to ship multiple DLLs per nupkg (System.Text.Json ships roslyn3.11/4.0/4.4 folders); a `.targets` gates old SDKs via `$(SupportsRoslynComponentVersioning)`. The `Microsoft.CodeAnalysis.CSharp` reference MUST be `PrivateAssets=all` (+ typically `ExcludeAssets=runtime`) so the Roslyn dep is build-only. RS1036/RS1037/RS1038 enforce extension assembly-reference hygiene (reference only `Microsoft.CodeAnalysis.Common`/`.CSharp`, `netstandard2.0` target). Release tracking is MANDATORY when `Microsoft.CodeAnalysis.Analyzers` is referenced: RS2008 demands `AnalyzerReleases.Shipped.md` + `.Unshipped.md`; RS2000-RS2007 enforce consistency (new IDs in Unshipped, removed IDs in a Removed Rules table). FOR THIS MONOREPO: skip the multi-targeting fan-out; single Roslyn target, `PrivateAssets=all`, project-ref with `OutputItemType=Analyzer` + `ReferenceOutputAssembly=false`.

## [9]-[BANNED_API_DATA_DRIVEN_PATTERN]

`Microsoft.CodeAnalysis.BannedApiAnalyzers` (RS0030 banned, RS0031 banned-with-message, RS0035 duplicate) is a DATA-DRIVEN analyzer: a `BannedSymbols.txt` AdditionalFile lists `DocumentationCommentId` strings (`T:System.DateTime;use DateTimeOffset`, `M:`, `N:` namespace, `P:`/`F:` member) with optional `;description`. THE transferable architecture for a banned-construct rule: read AdditionalFiles in CompilationStart, parse into an immutable lookup keyed by DocumentationCommentId, resolve to ISymbols once via `DocumentationCommentId.GetFirstSymbolForDeclarationId`, then match in OperationAction — zero hardcoded symbols, fully config-driven. This is the model for the rebuild's data-driven rules (0011/0007/0723/0742/0708 — doc 04 §CONFIG). The repo's `if`/`for`/`while`/`try` ban is the SYNTAX analog: match the forbidden `SyntaxKind` directly, no symbol resolution.

## [10]-[CODE_FIX_AND_FIXALL]

`CodeFixProvider` (`[ExportCodeFixProvider]` + `FixableDiagnosticIds` + `RegisterCodeFixesAsync` registering `CodeAction.Create` with a document-transform delegate + EQUIVALENCE KEY). FixAll: override `GetFixAllProvider`, return `WellKnownFixAllProviders.BatchFixer` (groups CodeActions by equivalenceKey across all in-scope diagnostics); custom only when batch semantics are wrong. `DocumentEditor`/`SyntaxEditor` is the modern transform API. FOR AN AGENT-FACING ANALYZER: code fixes are LOWER priority than precise, machine-parseable diagnostics — agents act on the diagnostic message+location, not the IDE lightbulb. The rebuild ships NO CodeFixProviders; the diagnostic Properties bag reserves the seam (doc 04 §AGENT_CONTRACT — but see 08 E1: the bag only earns its keep if assay adopts SARIF).

## [11]-[ARCHITECTURE_OF_MAJOR_MODERN_ANALYZERS]

- **Meziantou.Analyzer** (202 rules MA0001-MA0202): `src/.../Rules/<Name>Analyzer.cs` one file per rule; a central `RuleIdentifiers` static holds every ID + `GetHelpUri(id)`; `RuleCategories` holds category strings; descriptors are `static readonly DiagnosticDescriptor` referencing those; `Initialize` uses the CompilationStart→`AnalyzerContext(state)`→`RegisterSymbolAction` pattern verbatim. Separate `Meziantou.Analyzer.CodeFixers` project; `docs/Rules/MA####.md` one-per-rule (auto-generated).
- **Roslynator** (`dotnet/roslynator`): `Source/RoslynAnalyzers/Analyzers` with analyzer + codefix co-located per folder, `Tests/Analyzers.Tests` mirror; MIGRATING analyzers out of the IDE VSIX into `Roslynator.Analyzers` NuGet — packaging trend is NuGet-first.
- **SonarAnalyzer.CSharp** (380+ rules): rules inherit `SonarDiagnosticAnalyzer` (a wrapper over `DiagnosticAnalyzer` centralizing scope/generated-code gating — all registrations funnel through an internal `Execute` checking `HasMatchingScope`, moved into CompilationStart for perf, issue #8874); context wrappers `SonarAnalysisContext`/`SonarSyntaxNodeReportingContext`; a `DiagnosticDescriptorFactory` builds descriptors from RSPEC JSON metadata (rule docs/severity live as DATA, not code); tests use `VerifierBuilder<T>()` + `TestCases/*.cs` with `// Noncompliant` comments + an `its/` integration-regression harness over real projects.

THE CONVERGENT PATTERN across all three: (1) central ID/category registry, (2) one analyzer per rule file, (3) descriptor-as-DATA (factory or table), (4) a thin base wrapper centralizing `EnableConcurrentExecution` + generated-code + scope gating so rules don't repeat boilerplate (the `Driver` in doc 04), (5) data-driven test cases with inline annotations.

## [12]-[RECOMMENDED_PATTERN_SHORTLIST]

1. ONE `Driver : DiagnosticAnalyzer` that seals `Initialize`, forces `EnableConcurrentExecution` + `ConfigureGeneratedCodeAnalysis(None)`, and dispatches from the catalog — kills per-rule boilerplate (SonarDiagnosticAnalyzer model).
2. DESCRIPTOR CATALOG AS DATA: a single immutable table mapping each ID → its full `RuleRow`; descriptors are static readonly singletons built from the table, never per-callback. One polymorphic descriptor surface, not 87 scattered fields (satisfies repo density doctrine).
3. SYNTAX for shape/keyword bans, OPERATION for semantic rules; NEVER `GetSemanticModel` in a node callback; resolve well-known symbols once in CompilationStart state via a `WellKnownTypeProvider`-style cache.
4. PER-COMPILATION STATE in `RegisterCompilationStartAction`; `SymbolStartAction` for symbol-scoped accumulate-then-report; `CompilationEnd` for true cross-type accumulation (documented batch-only); never end actions for disposal.
5. CONFIG-DRIVEN banned-construct rules via AdditionalFiles + DocumentationCommentId (BannedApi model) so the catalog is DATA — directly serves "decision-useful to an agent" since the ban list is inspectable.
6. MANDATORY AnalyzerReleases.Shipped/Unshipped.md (RS2008) + reference `Microsoft.CodeAnalysis.Analyzers` 5.3.0 to self-enforce RS1xxx on the analyzer's own source.
7. TEST HARNESS: `CSharpAnalyzerVerifier<TAnalyzer, DefaultVerifier>`, fluent wrapper, `TestCases` with markup — scales to 80+ rules with positive AND negative cases per the repo mandate.
8. SINGLE Roslyn target (5.3.0 to match repo SDK), `PrivateAssets=all`, project-ref as Analyzer with `ReferenceOutputAssembly=false` — skip the multi-targeting fan-out.
9. AGENT-ORIENTED DIAGNOSTICS: invest in precise, structured messages + exact spans over code-fix lightbulbs; `messageFormat` names the violated standard and the canonical fix, not just "bad."
