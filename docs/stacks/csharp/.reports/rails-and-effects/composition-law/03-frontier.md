# Composition Law — Frontier

[EFF_RUNTIME_ENVIRONMENT_SEAM]:
- `Eff<A>` is a `record(Eff<MinRT, A> effect)` that implements `Readable<Eff<A>, A>` via `Deriving.Readable<Eff<A>, A, ReaderT<A, IO>>`. `Readable.Ask` returns the environment `A` as a value; `Readable.Local<A>(Func<A,A> f, K<Eff<A>,B> ma)` runs `ma` with the environment mapped through `f` without mutating the outer environment. This means `Eff<A>` is a reader-effect where the type parameter `A` is simultaneously the runtime and the environment — `asks(f)` and `local(f, ma)` are available on any `K<Eff<A>,B>` computation.
- `Eff<RT, A>` is `record(ReaderT<RT, IO, A> effect)` — the runtime is the `ReaderT` environment, not a separately tracked phantom. `runtime<RT>()` is `liftEff((Func<RT,RT>)identity)` — it projects the environment into the bound value, making the runtime available as data inside the pipeline without any special accessor protocol.
- `local<OuterRT, InnerRT, A>(Func<OuterRT, InnerRT> f, Eff<InnerRT, A> ma)` is the runtime-narrowing combinator: it reads the outer runtime via `getState<OuterRT>()`, maps it to the inner runtime via `f`, then runs `ma.effect.Run(innerRT)` inside `IO.local` (fresh `EnvIO` scope). The consequence is that an `Eff<InnerRT, A>` can be embedded inside an `Eff<OuterRT, A>` pipeline with a projection function — there is no `HasXxx` interface required; the runtime narrowing is structural via `Func<Outer, Inner>`.
- `Eff<A>.WithRuntime<RT>()` promotes an environment-free `Eff<A>` to an `Eff<RT, A>` by running `effect.RunIO(default(MinRT))` and lifting the resulting `IO<A>` via `MonadIO.liftIO<Eff<RT>, A>`. The `MinRT` default is a zero-allocation singleton struct. Calling `WithRuntime<RT>()` inside a `Bind` step in an `Eff<RT>` pipeline is the canonical way to embed pure-effect computations that were written against `Eff<A>` into a runtime-carrying context.

[IOLOCAL_SCOPE_AND_ENVIO_MECHANICS]:
- `IOLocal<X, A>` is an internal record that captures `(Func<EnvIO, EnvIO> MapEnvIO, K<IO, X> Operation, Func<X, K<IO, A>> Next)`. Its `Invoke(EnvIO envIO)` creates `using EnvIO local = MapEnvIO(envIO)` — the mapped environment is disposed when the step completes, not when the whole pipeline completes. `Map` and `Bind` on `IOLocal` fuse into the `Next` function, so the scope extension is lazy: the entire fused pipeline shares one scope creation.
- `IOUninterruptible<A, B>.Invoke` constructs `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — passing `CancellationToken.None` permanently prevents cancellation for the inner computation, while sharing the parent's resource scope and sync context. This means resources allocated during an uninterruptible step are still tracked in the parent's `Resources` and released at bracket exit; only the cancellation signal is suppressed. `ma.Uninterruptible()` should therefore only wrap the minimal atomic unit, not an entire pipeline branch, since it silently extends the resource-sharing surface.
- `IO.local<A>(K<IO, A> ma)` calls `ma.As().Local()` which constructs `IOLocal<A, A>(env => env.LocalResources, ma, IO.pure)`. `LocalResources` creates a fresh `Resources` scope with the parent's `CancellationToken` — a child resource scope that shares cancellation. Contrast with `localCancel<RT, A>(Eff<RT, A> ma)` which calls `ma.LocalIO()` — `LocalIO` creates `EnvIO.New(resources, token, new CancellationTokenSource, syncCtx)` — a fresh token source linked to the parent's via `parent.Register(() => child.Cancel())`. The asymmetry: `IO.local` isolates resource lifetime, `localCancel` isolates cancellation propagation direction; they are not interchangeable.
- `Prelude.envIO` is `IO.lift((EnvIO e) => e)` — a synchronous `IO<EnvIO>` that snapshots the current environment. `Prelude.cancelToken` is `IO.lift((EnvIO e) => e.Token)`. Inside an `Eff<RT, A>` pipeline, `lift<RT, A>(Func<EnvIO, RT, Either<Error, A>> f)` is the two-channel accessor: `+Prelude.envIO.Bind(e => Eff<RT, A>.Lift(rt => f(e, rt)))` fuses both into one step. Accessing `envIO` directly to branch on token state or extract resources inside domain logic is the `BOUNDARY_ADMISSION` violation — those concerns belong in bracket/finally/fork primitives.

[MONAD_RECUR_TRAMPOLINING]:
- `IO`'s `Monad<IO>.Recur<A, B>` implementation is `liftVAsync<B>(async (EnvIO env) => { while(true) { next = await f(value).RunAsync(env); if(next.IsDone) break; value = next.Loop; } return next.Done; })` — a heap-allocated `async` state machine that iterates via `await` without stack growth. The async state machine allocation is one-time per `Recur` call, not per iteration step. Each step produces a fresh `IO<Next<A,B>>` via `f(value)` and awaits it through `RunAsync`, which means every step respects the `EnvIO` cancellation token — the loop is automatically cancellation-cooperative.
- `Eff`'s `Monad<Eff>.Recur<A, B>` wraps `IO.liftAsync(async (EnvIO env) => { ... })` and additionally checks `fin.IsFail` after each `RunAsync`, immediately returning `Fin.Fail(fin.FailValue)` — the loop short-circuits on first failure. This is the correct behavior for domain recursion over `Eff`: an error in any step terminates the loop with that error, no recovery mid-loop.
- `Monad.iterableRecur<M, A, B>` requires `M : Natural<M, Iterable>, CoNatural<M, Iterable>`. `IO` does not implement either constraint — `IO`'s class declaration lists no `Natural<IO, Iterable>`. `iterableRecur` applies only to collection carriers (`Arr`, `Lst`, `Seq`, `HashSet`, `Set`). Using `Monad.recur<IO,A,B>` is the correct path for `IO`/`Eff`; `iterableRecur` silently miscompiles for effect carriers because the constraint check alone prevents it from being called.
- `Next<A, B>` is the trampoline decision type: `IsDone : bool`, `Done : B`, `IsLoop : bool`, `Loop : A`. Construction via `Next.Done(b)` and `Next.Loop(a)` — static factory methods. The `A` and `B` type parameters are independent, enabling accumulators of different types from loop state without boxing or type-erasing intermediate values. A `recur`-based fold over `IO<Next<Seq<T>, R>>` naturally accumulates `Seq<T>` as loop state and terminates with `R` as the final value.

[MEMO_LAZY_APPLICATIVE_COMPOSITION]:
- `Memo<F, A>` is a lazy wrapper over `K<F, A>` with a spin-lock `state` field (0=unloaded, 1=loading, 2=loaded). `Value` reads the field twice — a state check followed by the real value if loaded — with `GetValue()` performing the spin-lock acquisition. The `K<F, A>` acquired on first access is cached permanently; subsequent reads are allocation-free field reads.
- `Applicative<F>.Apply<A, B>(K<F, Func<A,B>> mf, Memo<F, A> ma)` is a `static abstract` distinct overload from the eager `Apply(K<F, Func<A,B>>, K<F, A>)`. For `Fin`, the `Memo` overload reads `ma.Value` only when `mf` is `Succ` — if `mf` is `Fail`, the memo is never forced. For `Validation<FAIL>`, the `Memo` overload forces both sides unconditionally to enable error accumulation. The behavioral difference is by carrier design, not by applicative law: laziness is only semantically safe when the carrier does not need both sides for error accumulation.
- `memoK<F, A>(K<F, A> fa)` wraps an already-resolved `K<F, A>` in a pre-loaded `Memo<F, A>` (state=2). `memoK<F, A>(A value)` wraps a pure value via `F.Pure(value)`. Neither involves suspension. `Memo<F, A>(Func<K<F, A>> f)` — the suspension constructor — is `internal`; it is used exclusively by `Memo.transform` and the `Apply` overloads inside the library. The public API provides no direct constructor for suspended memos: they are produced by the carrier's `Apply` implementation when it receives a lazy argument, not by user code.
- `Memo.transform<F, G, A>(Memo<F, A> ma)` produces `Memo<G, A>(() => F.Transform(ma.Value))` — a new suspension that lazily promotes the inner value via `Natural.transform`. This enables `Deriving.*` interfaces to propagate memoization through natural transformations without forcing the value at transform time.

[FINAL_TRAIT_PIPE_SYNTAX]:
- `Final<F>` declares `static abstract K<F, A> Finally<X, A>(K<F, A> fa, K<F, X> @finally)`. The `|` C# 14 extension operator `K<F, A> | Finally<F, X>` calls `lhs.Finally(rhs.Operation)`, enabling pipeline-style finally attachment: `computation | final(cleanup)`. `final<F, X>(K<F, X> @finally)` creates the `Finally<F, X>` record. The result is structurally equivalent to `IO.Bracket` with no acquisition — it is unconditional cleanup, not resource-scoped cleanup.
- `IO.Finally<X, A>(K<IO, A> fa, K<IO, X> @finally)` delegates to `fa.As().Finally(@finally.As())` which constructs `IOFinal<X, A, A>(this, @finally, IO.pure)`. The `Invoke` for `IOFinal` runs `fa` first, captures the `EnvIO`, then runs `@finally` on the same `EnvIO` regardless of whether `fa` succeeded or failed, and discards the finally value. Because the same `EnvIO` is reused for both, any cancellation that occurs during `fa`'s execution is still in effect when `@finally` runs — `@finally` is therefore not an uninterruptible cleanup unless explicitly wrapped in `.Uninterruptible()`.
- `Eff<A>` delegates `Finally` to `ReaderT<A, IO, T>((A env) => fa.As().RunIO(env).Finally(@finally.As().effect.Run(env)))` — both `fa` and `finally` receive the same `env`, binding the environment into both halves at composition time. The environment is captured once: if `Local` modified the environment for `fa`, the same modification is in scope for `finally`.

[STM_ATOM_IO_COMPOSITION]:
- `atomIO<A>(A value)` is `IO.lift(() => Atom(value))` — the atom is constructed inside the lifted computation, meaning it is deferred until the `IO<Atom<A>>` is run. `atomIO<A>(A value, Func<A, bool> validator)` validates the initial value inside the lift and throws `Exceptions.ValidationFailed` if the validator rejects it. Because this throw is inside `IO.lift` (synchronous capture), it is converted to `Fin.Fail(Error.New(ValidationFailed))` at `IO.RunSafe()` or `IO.RunAsync()` — the validator rejection propagates as a typed rail failure, not an unhandled exception.
- `swapIO<A>(Atom<A> ma, Func<A, A> f)` delegates to `ma.SwapIO(f)`, which is `IO.lift((EnvIO _) => Swap(f))`. The CAS spin-loop is inside the synchronous `Swap` call, which means the thread blocks in the spin loop during `IO<A>.RunAsync(env)` — there is no `await` between CAS attempts. For high-contention atoms under `IO.Fork`, each fork blocks its thread pool thread in the spin loop, which can exhaust the thread pool. The `Option<A>`-returning overload `swapIO(ma, Func<A, Option<A>> f)` calls `SwapMaybeIO` — if `f` returns `None`, the atom is not updated and the current value is returned, providing conditional-update semantics without an additional read-then-CAS race.
- `commuteIO<A>(Ref<A> r, Func<A, A> f)` is valid only inside an `STM.RunTransaction` context — calling it outside a transaction throws `InvalidOperationException`. Inside a transaction, `commute` defers the re-application: the function is recorded as a commutative operation and re-run at commit time against the latest committed value. Unlike `alter` (which records the in-transaction value), `commute` accepts that `f` may see a different value at commit than at record time, trading serializability for concurrency. The correct composition rule: `swapIO` for uncoordinated state, `alter`/`commute` inside `STM.RunTransaction` for coordinated multi-ref state.
- `refIO<A>(A value, Func<A,bool>? validator)` is `IO.lift(() => Ref(value, validator))` — the `Ref` is constructed deferred, same pattern as `atomIO`. `swapIO<A>(Ref<A> r, Func<A, A> f)` delegates to `r.SwapIO(f)` — internally calls `STM.alter(r, f)`, which requires a live transaction. Calling `swapIO(ref, f)` outside a transaction silently creates an auto-committing single-op transaction via `alter`'s implicit transaction detection. This silent wrapping means `swapIO` on a `Ref` is always transactional, while `swapIO` on an `Atom` is never transactional — they are not interchangeable even though their call shapes are identical.

[VALIDATIONT_APPLICATIVE_BIND_SEAM]:
- `ValidationT<F, M>` carries `Func<MonoidInstance<F>, K<M, Validation<F, A>>>` — it passes a `MonoidInstance<F>` witness at run time, not at construction time. `MonoidInstance<F>` is the resolved instance for `F : Monoid<F>`; it carries `Empty` and `Combine` as boxed delegates. This means `ValidationT` can compose over any `F : Monoid<F>` including custom domain error types without compiler-emitted trait dispatch overhead at each step — the monoid instance is threaded through once per `Run` call.
- `ValidationT<F, M>.Apply` uses `from ff in mf.Run(monoid) from fa in ma.Run(monoid) select ff.ApplyI(fa, monoid)` — the `from`/`from` desugars to `M.Bind(mf.Run(m), ff => M.Map(fa => ff.ApplyI(fa, m), ma.Run(m)))`. This sequences both sides inside `M` (not in parallel), then accumulates inside `Validation`. If `M = IO`, both sides execute sequentially despite both being `IO` computations — `ValidationT<F, IO>` does not parallelize field validation. To run fields in parallel and accumulate errors, use `IO.Fork` manually and combine the `Validation<F, A>` results with `+` before entering the `ValidationT` layer.
- `SemigroupK<ValidationT<F, M>>.Combine` runs both sides and calls `ea.CombineFirst(eb, monoid)` — if `ea` is success, it short-circuits; if `ea` is failure, it combines the failure with `eb`'s failure via `monoid.Combine`. `Choice<ValidationT<F, M>>.Choose` instead short-circuits on the first failure without combining: `if !ea.IsSuccess => mb.Run(m)`, meaning `|` (pipe via `Choice`) is winner-takes-first, while `SemigroupK.Combine` (via `+` or `Combine`) accumulates. The `|` operator on `ValidationT` is therefore not accumulating — it is an early-exit alternative.

[CHRONIC_TRANSFORMER_COMPOSITION]:
- `ChronicleT<Ch, M, A>` is `record(Func<SemigroupInstance<Ch>, K<M, These<Ch, A>>> runChronicleT)` where `These<Ch, A>` is the `This`/`That`/`These` bifunctor. `dictate<Ch, M, A>(A value)` produces `These.These(default(Ch), value)` — a value with an empty (via `Ch`'s default) chronicle. `confess<Ch, M, A>(Ch value)` produces `These.This(value)` — a fatal error with no bound value, aborting the computation. `chronicle<Ch, M, A>(Ch @this, A that)` produces `These.These(@this, that)` — both a chronicle entry and a bound value in one step.
- `ChronicleT<Ch, M>` is `Fallible<Ch, ChronicleT<Ch, M>>`: failure is typed as `Ch`, not `Error`. This makes `ChronicleT` the correct transformer for domain-specific accumulation where the log and the abort signal share the same type — a natural fit for typed diagnostic pipelines. Contrast with `ValidationT<F, M>` where `F` is the failure type and there is no log/value duality; `ChronicleT` carries both a partial log and a final value simultaneously via `These`.
- `memento<Ch, M, A>(K<ChronicleT<Ch, M>, A> ma)` recovers a confessed (aborted) chronicle into `Either<Ch, A>` — `This(ch)` becomes `Left(ch)`, `That(a)` becomes `Right(a)`. It requires `M : MonadIO<M>`. `absolve<Ch, M, A>(A defaultValue, ma)` substitutes a default for `This` cases, continuing computation. These are the terminal-recovery combinators for `ChronicleT`; they are analogous to `Match` on `Fin` — correct only at the consuming edge, not mid-pipeline.

[ANTI_PATTERNS_FRONTIER]:
- `iterableRecur<M, A, B>` is callable only for collection carriers implementing `Natural<M, Iterable>, CoNatural<M, Iterable>`. Passing `IO` or `Eff` as `M` does not compile because neither class lists these interfaces. A common mistake is using `iterableRecur` for trampolining effect pipelines after seeing it work on `Seq`; `Monad.recur<IO/Eff, A, B>` is the correct form.
- Wrapping `IO.Uninterruptible()` around a broad pipeline section suppresses cancellation for the entire contained computation, not just the critical atomic unit. `IOUninterruptible` passes `CancellationToken.None` into the `EnvIO` for the wrapped operation — any `yieldFor`, `Fork`, or `liftAsync` inside that section will not respond to cancellation until the uninterruptible boundary exits. The minimum-scope rule applies: wrap only the memory-barrier or CAS operation, not the pipeline step.
- Using `swapIO(Atom<A>, Func<A, A> f)` inside a `from`/`select` pipeline over `Eff<RT, A>` is correct — but using `Atom<A>.Swap(f)` (the non-IO synchronous form) inside a `Map` or `Bind` lambda is a side effect inside a pure transform. The CAS spin-loop runs synchronously inside the lambda, blocking the executing thread. `swapIO` lifts the operation into the `IO` layer correctly, preserving referential transparency of the pipeline description.
- Using `ValidationT<F, IO>.Apply` and expecting parallel execution is incorrect: `Apply` sequences both sides inside `IO.Bind` (from/from desugars to Bind). Only `IO.Fork` followed by `Await` produces parallelism; `Apply` over `ValidationT<F, IO>` is sequential with applicative error accumulation. The distinction: parallelism is a runtime scheduling concern that `IO.Fork` owns; `Apply` owns error-accumulation semantics.
- Writing `io | final(cleanup)` (pipe syntax) when the cleanup must be uninterruptible is a failure surface: `Final<IO>.Finally` runs `cleanup` on the same `EnvIO` as the primary operation, which inherits the parent's `CancellationToken`. If the primary was cancelled, the `EnvIO` token is already cancelled when `cleanup` runs. `cleanup.Uninterruptible()` inside `final(cleanup.Uninterruptible())` is required for cancel-safe teardown.
