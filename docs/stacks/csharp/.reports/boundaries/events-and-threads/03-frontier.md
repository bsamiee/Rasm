# Events and Threads — Frontier

[IO_FORK_THREAD_POOL_AND_AWAIT_SEMANTICS]:
- `IO<A>.Fork(Option<TimeSpan>)` queues work on the thread pool. `ForkIO<A>` carries `Cancel: IO<Unit>` and `Await: IO<A>`; neither executes until bound. The `IO<A>.Timeout(TimeSpan)` overload wraps a fork-with-cancel race, not a `CancellationTokenSource.CancelAfter`; the timeout is compositional.
- `awaitAny<A>(ForkIO<A>[])` cancels all remaining forks when the first succeeds — on first success it returns straight away and cancels the others; if errors accumulate equal to the fork count, all errors are returned. `awaitAll<A>(ForkIO<A>[])` collects every result or accumulates every error without short-circuiting on first failure. A seam that forks N boundary probes with `awaitAll` must treat the error sequence as a partial-availability report, not a single fault.
- `awaitAny` and `Task.WhenAny` are not interchangeable when fan-in cleanup matters: `Task.WhenAny` leaves all tasks running; `awaitAny` delivers cancellation to losers via their `CancellationTokenSource`, and their `Resources` scopes are cleaned up in their own `finally` blocks.

[TASK_COMPLETION_ORDER_AND_SPAN_OVERLOADS]:
- `Task.WhenEach<TResult>(ReadOnlySpan<Task<TResult>>)` yields tasks in completion order as `IAsyncEnumerable<Task<TResult>>`, replacing manual `Task.WhenAny`-in-a-loop without tracking a remaining set. `WithCancellation(ct)` on the returned `IAsyncEnumerable` stops iteration; it does not cancel the underlying tasks. Faulted tasks are yielded normally — `completed.IsFaulted` must be checked per task.
- `Task.WhenAll(ReadOnlySpan<Task>)`, `Task.WhenAll<TResult>(ReadOnlySpan<Task<TResult>>)`, and `Task.WhenAny(ReadOnlySpan<Task>)` avoid the `Task[]` allocation for fixed-arity fan-out. `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` eliminates the array allocation for token-merging at seams that link two or more parent tokens.
- `TaskCompletionSource<T>.SetFromTask(Task<T>)` and `TrySetFromTask(Task<T>)` propagate the terminal state (success, fault, or cancellation) of the source task atomically to the TCS. `TrySetFromTask` returns `false` if the TCS is already transitioned, making it safe for concurrent completion races. Both require the source task to be completed before calling; passing an incomplete task throws `ArgumentException`.

[CONFIGURE_AWAIT_OPTIONS_CONSTRAINT]:
- `ConfigureAwaitOptions.SuppressThrowing` is valid only on the non-generic `Task`, not `Task<T>`. Applying it to `Task<T>.ConfigureAwait(options)` throws `ArgumentOutOfRangeException` at runtime, not at compile time; CA2261 is the diagnostic. The correct pattern: cast `Task<T>` to `Task`, await with `SuppressThrowing`, then inspect the original `Task<T>` handle for `IsFaulted`, `Exception.InnerException`, or `IsCanceled`. This asymmetry makes `SuppressThrowing` a non-result-carrying boundary guard, not a value-path operator.
- `ConfigureAwaitOptions` is a `[Flags]` enum with four values: `None = 0`, `ContinueOnCapturedContext = 1`, `SuppressThrowing = 2`, `ForceYielding = 4`. `ForceYielding` forces a continuation post even on an already-completed task — awaiting a pre-completed `Task.CompletedTask` with `ForceYielding` resumes on a different managed thread. This is the correct form for host-event dispatch callbacks that must not run on the event source's stack frame.

[CHANNEL_COMPLETION_AND_ERROR_PROPAGATION_PRECISION]:
- `ChannelWriter<T>.Complete(error)` does not surface the error immediately: `ChannelReader<T>.Completion` only transitions to faulted after all buffered items are drained. The error surfaces from `ReadAsync`, `WaitToReadAsync`, and `ReadAllAsync` after the buffer is empty.
- The error from `Complete(error)` propagates as the original exception type — not wrapped in `ChannelClosedException`. `ChannelClosedException` is thrown only on a normal `Complete()` (no error) when `ReadAsync` is called after the channel is empty and closed. Consumers must catch the writer's original exception type, not `ChannelClosedException`, when the writer signals structured failures.
- A fold worker that calls `primaryWriter.Complete(error)` to terminate projection delivers the error to all `ReadAllAsync` consumers as a thrown exception of the original type. A separate `ChannelWriter<Error>` is architecturally cleaner when fault consumers and projection consumers must react differently, but the single-channel pattern is sufficient when one exception type uniformly terminates all consumers.
- `RendezvousChannel<T>` carries `CanCount = true` and `CanPeek = true`. The rendezvous invariant (each write blocks until a matching read is pending) means `Count` is always 0 or 1 in the handoff window — not a useful capacity indicator. `TryPeek` reveals what the first blocked writer wants to hand over without consuming it.

[ASYNC_LOCAL_PROPAGATION_SEMANTICS]:
- `AsyncLocal<T>` propagates into child scopes (tasks, threads started with `Thread.Start`) but modifications inside a child scope do not flow back to the parent. `Thread.UnsafeStart` completely suppresses flow — the new thread starts with `AsyncLocal<T>.Value` returning `default(T)`.
- `AsyncLocal<T>(AsyncLocalValueChangedHandler<T>)` fires on every value change, including implicit fork/join transitions: entering a child scope fires with `threadContextChanged = true`; the child setting a value fires with `threadContextChanged = false`; child scope exit fires with the reverse. This gives a seam owner a hook for ambient-context-sensitive subscription management. The callback runs synchronously on the thread that caused the change.
- `ExecutionContext.SuppressFlow()` returns an `AsyncFlowControl` that must be disposed to restore flow on the current thread. The suppression is thread-local. Calling `SuppressFlow()` without capturing the return value leaves the thread's flow suppressed for all subsequent work items queued from that thread. `ThreadPool.UnsafeQueueUserWorkItem<TState>` inherently skips EC capture regardless of ambient flow state; it is allocation-free with a struct `TState` and eliminates both the delegate allocation and EC cost for context-free callbacks.

[THREAD_UNSAFE_START_AND_BOUNDARY_SEAMS]:
- `Thread.UnsafeStart()` and `Thread.UnsafeStart(object? parameter)` skip `ExecutionContext` capture entirely: the spawned thread starts with no `AsyncLocal<T>` values from the calling context, no ambient security or activity state, and incurs no allocation for EC copy. The correct domain is callback-only workers that write to channels, swap atoms, or set completion sources. Any worker that needs to propagate activity correlation, user identity context, or caller-injected `AsyncLocal` values must use `Thread.Start`.
- `ThreadPool.UnsafeRegisterWaitForSingleObject` skips EC capture for `WaitHandle`-backed callbacks — the allocation-free registration path for boundary seams that map native wait handles to managed notifications without inheriting ambient context. The symmetric `WaitHandle.Unregister(WaitHandle)` must be called on teardown; failing to unregister leaks the OS wait registration even after the calling object is collected.

[PERIODIC_TIMER_AND_RATE_CONTROL_PROTOCOL]:
- The rate-limited consumer pattern: a `PeriodicTimer` drives the outer tick loop; inside each tick, `TryRead` drains all items that arrived in the interval. The fold worker owns the timer and disposes it on deactivation. The producer writes unconditionally; the channel's `BoundedChannelFullMode` governs back-pressure. Producer policy is expressed in channel capacity and `FullMode`, consumer pacing in the timer period, and the two are independent.

[SUBSCRIPTION_SCOPING_WITH_IO_WITHENVFAIL]:
- `IO<A>.WithEnvFail(env => env.LocalResources)` releases acquired resources only on failure. The seam pattern: an activation computation acquires subscriptions inside `WithEnvFail`; on success, subscriptions escape bound to an explicit `Atom<Option<Resources>>` ownership token; on failure, `WithEnvFail` cleans up the partial subscription set automatically. Explicit deactivation swaps `None` in and calls `DisposeU` on the evicted `Resources` value.
- `IO<A>.Retry(Schedule)` releases resources on each failed attempt and retains them on success. A retry loop that opens subscriptions on each attempt never accumulates them across failed tries; each failed attempt's scope is cleaned up before the next retry. This is the correct shape for a connect-and-subscribe pattern that must retry until the upstream is available.
- `IO<A>.Uninterruptible` composed with a `Bracket` finalizer is the canonical teardown form: cancellation of the parent token does not interrupt the finalizer chain. `IO<A>.Finally<B>(IO<B>)` runs the `finally` effect after `this` regardless of success or failure; the `Finally` effect's result is discarded and the original result or error propagates. For teardown effects with their own return value, `Bracket` is the correct form; for side-effecting cleanup, `Finally` suffices.

[CONCURRENT_FANOUT_AND_AWAIT_ANY_CANCELLATION]:
- `Parallel.ForEachAsync<TSource>(IAsyncEnumerable<TSource>, ParallelOptions, Func<TSource, CancellationToken, ValueTask>)` consumes an async source with bounded parallelism. `ParallelOptions.MaxDegreeOfParallelism` caps concurrent invocations. Cancelling the options token terminates all pending bodies. This is the correct form for bounded-parallel event processing over a channel-backed `IAsyncEnumerable` source.

[CANCELLATION_REGISTRATION_DISPOSAL_HAZARD]:
- `CancellationTokenRegistration.Dispose()` blocks until any in-flight callback on another thread finishes. `DisposeAsync()` returns immediately after unregistering without waiting for in-flight callbacks. Using `await using` combined with a callback that accesses a resource disposed in the `finally` creates a window where the resource is accessed after disposal. Synchronous `Dispose` is safer when the next step after unregistration disposes resources the callback touches.
- `IsCancellationRequested` is a cheap volatile read and the correct form inside a hot-path callback to exit early rather than inspecting state via closure.
