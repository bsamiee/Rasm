# Carrier-Polymorphic Trait Abstraction

[KIND_INDEXED_ARROW]:
- Every carrier is a kind `F` indexing the existential `K<F, A>`; an arrow generic in `F` constrained `where F : Monad<F>` (or a weaker trait) is written once and instantiates over the result carrier, the absence carrier, the accumulating carrier, the effect carrier, and every collection carrier without a single per-carrier overload. The constraint chain `Monad<F> : Applicative<F> : Functor<F>` selects the available power: `Functor` gives `Map`, `Applicative` adds `Pure`/`Apply`/`Action`, `Monad` adds `Bind`/`Recur`. A method bound only `where F : Applicative<F>` cannot sequence-with-dependency and therefore is structurally guaranteed to be parallelizable and accumulating; the constraint is the capability contract, not documentation.
- The static-virtual interface members carry the implementation: `static abstract K<M,B> Bind<A,B>(K<M,A>, Func<A,K<M,B>>)` and `static abstract K<F,A> Pure<A>(A)` dispatch on the type argument `F`/`M` at the call site with zero allocation and zero vtable indirection â€” the trait is resolved at JIT specialization, so a `where M : Monad<M>` arrow is monomorphized per carrier. The carrier type itself (`Fin`, `Option`, `Eff`, `Seq`) is the trait-implementing struct passed as the type parameter, never an instance.
- The bridge from concrete to abstract is the implicit upcast `Fin<A> : K<Fin, A>`; the bridge back is `.As()`, which downcasts the kind witness to the concrete carrier. A polymorphic arrow returns `K<F, B>`; the caller pins it with `.As()` to recover `Fin<B>`. Omitting `.As()` is the single most common carrier-polymorphic error â€” the value stays existentially kinded and refuses concrete-only operators.

```csharp
static K<M, C> combineRail<M, A, B, C>(K<M, A> first, K<M, B> second, Func<A, B, C> merge)
    where M : Monad<M> =>
    first.Bind(a => second.Map(b => merge(a, b)));
// instantiates over Fin, Option, Eff, Seq, IO with identical source; caller pins via .As()
Fin<int> pinned = combineRail<Fin, int, int, int>(Fin.Succ(2), Fin.Succ(3), (a, b) => a + b).As();
```

[STACK_SAFE_TRAMPOLINE]:
- `Monad<M>` exposes `static abstract K<M,B> Recur<A,B>(A, Func<A,K<M,Next<A,B>>>)` â€” a carrier-level trampoline that drives monadic iteration without growing the native stack. `Next<A,B>` is a two-case signal: `Next.Loop<A,B>(A)` re-enters with a new seed, `Next.Done<A,B>(B)` terminates with a result. The step function returns `K<M, Next<A,B>>`, so each iteration is itself an effect in the carrier â€” the loop body can fail, branch, or short-circuit through the rail while the recursion stays bounded. This is the FP replacement for `while`/`for` over a monadic accumulator: the loop variable threads as `A`, the exit is data (`Done`), not a `break` statement.
- `Done`/`Loop` are `static virtual` defaults (`M.Pure(Next.Done(...))`), so a carrier inherits the trampoline for free and overrides only `Recur` if it needs a non-default driving strategy. Three sibling drivers exist as one polymorphic family: `recur` is the safe default, `unsafeRecur` skips the stack guard for carriers that are already tail-safe, and `iterableRecur` (constrained `where M : Natural<M,Iterable>, CoNatural<M,Iterable>`) collects every intermediate `Loop` value into the carrier â€” turning the trampoline into a streaming unfold rather than a single-result fold.

```csharp
static K<M, long> sumTo<M>(int n) where M : Monad<M> =>
    Monad.recur<M, (int i, long acc), long>(
        (1, 0L),
        s => Monad.pure<M, Next<(int, long), long>>(
            s.i > n ? Next.Done<(int, long), long>(s.acc)
                    : Next.Loop<(int, long), long>((s.i + 1, s.acc + s.i))));
```

[LIFTED_CONTROL_FLOW]:
- The doctrine's ban on `if`/`switch` in domain logic is satisfied at carrier level by lifted combinators that consume a `K<M,bool>` predicate already in the rail. `Monad.when(K<M,bool> pred, K<M,Unit> then)` and `Monad.unless(...)` run the effect conditionally without ever materializing the boolean as a statement; `Monad.iff(K<M,bool> pred, then, else)` is the ternary on rails, returning `K<M,A>`. `iff` is overloaded across ten arms over the cross product of `{K<M,A>, Pure<A>, K<IO,A>}` for each branch, so a branch that is a pure constant or a raw `IO` effect lifts without an explicit `Pure`/`liftIO` wrap â€” the overload set absorbs the lift.
- `Monad.accumWhile`/`accumUntil` and their `M`-suffixed forms (`accumWhileM` takes `Func<A,K<M,bool>>`) replay a single effect `K<M,A>` repeatedly, collecting results into `K<M,Iterable<A>>` until the predicate flips â€” the rail-native `do { } while` whose continuation condition is itself effectful. `forever<M,A>` is the non-terminating repeat for daemon-shaped effects.

[LAZY_APPLICATIVE_SHORT_CIRCUIT]:
- `Apply` is overloaded to accept `Memo<F,A>` on the argument side. `Memo<F,A>` wraps a `Func<K<F,A>>` behind a three-state CAS gate (`0=unloaded, 1=loading, 2=loaded`) and forces the underlying effect only on first read of `.Value`, caching thereafter. A carrier whose `Apply` does not consult the right operand when the left already determines the result therefore never reads `.Value`, and the wrapped right-hand effect is never constructed; the eager `K<F,A>` overload constructs both operands unconditionally at the call site. Passing the right operand as `Memo` converts a wide applicative product from pay-for-all to pay-for-reached without changing the combiner.
- `Choice<A>(K<F,A> fa, Memo<F,A> fb)` on the `Choice` trait carries the same laziness into alternation: the second alternative is forced only when the first yields empty, giving `|`-style fallback the call-by-need semantics of a hand-written short-circuit without the boolean test.

[VARIADIC_APPLICATIVE_PRODUCT]:
- The applicative product is exposed two ways. The curried form `Applicative.apply` is overloaded for arity two through ten over `Func<A, Func<B, ... R>>`, peeling one argument per `Apply` and returning a narrower function carrier â€” the classic `f <$> a <*> b <*> c` ladder. The tuple form `Applicative.apply((fa, fb, fc), Func<A,B,C,R>)` is the dense equivalent: a single call takes a tuple of independent carriers and an uncurried combiner, eliminating the curry ceremony while preserving the parallel, accumulating semantics. `applyM` is the monadic-result variant whose combiner returns `K<M,R>` and flattens â€” the bridge from independent-accumulate to dependent-sequence in one operator.
- Because the tuple-apply derives every failure-combination from the carrier's `Apply`, an accumulating carrier collects all N failures and a short-circuit carrier reports the first; the same call site changes accumulation policy purely by changing the carrier type argument. Arithmetic lifts (`Applicative.add`/`subtract`/`multiply`/`divide`) are pre-built binary applicatives over any `IAdditionOperators`-or-`Num`-constrained payload â€” element-wise numeric combination inside any applicative carrier with no manual `Apply` of an operator function.

[CAPABILITY_TRAITS]:
- `Has<M, VALUE>` declares `static abstract K<M, VALUE> Ask` â€” a one-member capability witness stating "carrier `M` can surface a `VALUE` from its environment". An arrow constrained `where M : Monad<M>, Has<M, IClock>, Has<M, ILog>` demands exactly the capabilities it touches; the concrete runtime carrier satisfies each `Has` once, and the dependency set is the type signature, not a constructor-injected container. Adding a dependency is adding a `Has` constraint; an over-broad carrier cannot smuggle unused capability past the signature.
- The reader/state capabilities are the same shape at the trait level: `Readable<M,Env>` gives `Ask`, `Asks(Func<Env,A>)`, and `Local(Func<Env,Env>, K<M,A>)` for scoped environment rewriting; `Stateful<M,S>` gives `Get`, `Put`, `Modify(Func<S,S>)`, `Gets(Func<S,A>)`. `Ask`/`Get` are `static virtual` defaults derived from `Asks`/`Gets` with `identity`, so a carrier implements one primitive and inherits the projecting forms. `Local`/`With` scope a transformed environment to a sub-effect and restore it on exit â€” the functional replacement for mutable ambient context.

[CARRIER_POLYMORPHIC_RECOVERY]:
- `Fallible<E,F>` declares `static abstract K<F,A> Fail<A>(E)` and `static abstract K<F,A> Catch<A>(K<F,A> fa, Func<E,bool> predicate, Func<E,K<F,A>> handler)` â€” recovery as a carrier capability, not a per-carrier method. `Fallible<F> : Fallible<Error,F>` fixes the error type to the `Error` hierarchy for the common case while leaving the two-parameter form open for non-`Error` failure domains.
- `@catch` builds a `CatchM<Error,M,A>` â€” a first-class, composable recovery value, not a control statement â€” discriminated by error identity, integer code, or arbitrary predicate, each overloaded to take either a replacement effect `K<M,A>` or a handler `Func<Error,K<M,A>>`. Multiple `CatchM` values compose with `|` into one ordered recovery cascade applied to any `Fallible` carrier. `catchOf<E,M,A>` filters to a concrete error subtype and short-circuits on the first matching atom; `catchOfFold<E,M,A>` (constrained `where M : MonoidK<M>`) folds the handler across every matching atom of a `ManyErrors` aggregate, accumulating into the carrier's monoid identity â€” the distinction surfaces only when the caught error is multi-atom, where `catchOf` recovers once and `catchOfFold` recovers each.

[ALTERNATIVE_AS_GRAMMAR]:
- `Alternative<F> : Choice<F>, MonoidK<F>` is simultaneously the failure-fallback trait and a complete parser-combinator grammar over any carrier with an empty and a choice. Beyond `Empty`/`Choose`, its `static virtual` surface derives `Some` (one-or-more), `Many` (zero-or-more), `SepBy`/`SepBy1`, `EndBy`/`EndBy1`, `ManyUntil`/`SomeUntil` (and `*2` variants returning the terminator), `Between<OPEN,CLOSE>`, and `Option(value, fa)` â€” every combinator expressed once against `Choose` and `Pure` and inherited by every alternative carrier. A backtracking parser, a non-deterministic search over `Seq`, and a first-success fallback over `Option` are the same code at three carrier instantiations; the repetition combinators are stack-safe because they thread through the carrier's own `Recur`.

[PARTIAL_TRAIT_INHERITANCE]:
- A carrier opts into IO interop by implementing the partial trait `Maybe.MonadIO<M>` (one method, `LiftIOMaybe`); `MonadIO<M> : Monad<M>` then derives `EnvIO`, `Token`, `TokenSource`, and `SyncContext` as `static virtual` reader-style properties (`M.LiftIOMaybe(IO.token)`), so the whole cancellation-and-environment surface materializes from a single lift primitive. `MonadUnliftIO<M>` adds `ToIOMaybe`/`MapIOMaybe`, exposing `toIO`/`mapIO` that round-trip an arbitrary carrier effect back into a manipulable `IO<A>` â€” the basis for wrapping a polymorphic effect with timeout, retry, or bracket expressed purely in `IO` and re-lifted.
- The `Maybe.` partial-trait split is the mechanism by which a carrier can be a `Monad` without being IO-capable: the IO members live on an optional sub-trait, queried by a `cast`-or-throw at the call boundary, so a pure carrier participates in every monadic arrow yet rejects `liftIO` at the type level. Capability is additive and Ă  la carte, never all-or-nothing.

[TRAVERSAL_OVER_ANY_PAIR]:
- `Traversable<T>` turns any traversable carrier `T` inside-out against any applicative `F`: `Traverse<F,A,B>(Func<A,K<F,B>>, K<T,A>) -> K<F, K<T,B>>` is generic in both the outer structure and the effect, so one `Traverse` covers `Seq` of `Fin`, `Option` of `Eff`, and every other pairing. `TraverseM` constrains `F` to `Monad<F>` to thread effects sequentially with dependency; `Sequence`/`SequenceM` are the identity-function specializations that flip a `K<T,K<F,A>>` into `K<F,K<T,A>>`. The accumulation policy of the result is inherited from `F`: traversing with an accumulating applicative collects every failure across the structure in one pass, while traversing with a short-circuit monad stops at the first.
- `Foldable<T>` supplies the monadic and monoidal fold family from two primitives (`FoldWhile`/`FoldBackWhile`): `FoldM`/`FoldWhileM`/`FoldUntilM` thread an effect through the fold (`Func<A,Func<S,K<M,S>>>`), `FoldMap<A,B> where B : Monoid<B>` collapses via a monoid, and `Fold<A> where A : Monoid<A>` folds a structure of monoidal values with no seed. The `While`/`Until` predicate is `Func<(S State, A Value), bool>` â€” it sees both the running accumulator and the current element, so early exit is a pure data decision over the fold state, never a thrown control-flow break.
