# Parameter Algebra — Frontier

[SCHEDULE_MONAD_BIND]:
- `Schedule` implements the monad bind pattern: `Schedule.Bind(Func<Duration, Schedule> f)` fans out each emitted duration into a sub-schedule derived from it, then concatenates all sub-schedules. The generated `SchBind` runs `Schedule.Run().Bind(d => f(d).Run())` — every duration in the outer schedule produces its own inner duration stream, and the total emission is the flat concatenation. This means a duration-parameterized policy value — a schedule whose per-step spacing is itself a function of the previous duration — is expressed as `from d in baseSchedule from d2 in Schedule.spaced(d * factor) select d + d2` using C# query expression syntax.
- `SelectMany(Func<Duration, Schedule> bind, Func<Duration, Duration, Duration> project)` is generated as `SchBind2`, which runs `outer.Bind(x => bind(x).Run().Map(y => project(x, y)))`. The projection function receives both the outer duration and the inner duration, enabling combined policy values — such as capping the sub-schedule's output to the outer duration — within one LINQ expression. `ScheduleTransformer` has no `SelectMany`, `Select`, or `Bind`; only `Schedule` supports LINQ query expressions.
- The LINQ bind on `Schedule` is not the same as the `|`/`&` operators. Union and intersection compose two independent schedules at the top level. Bind composes one schedule with a function that builds the next schedule from each emission of the first. The distinction: `| Schedule.spaced(cap)` clamps maximum delay by taking the minimum per step; `Bind(d => Schedule.spaced(d.Min(cap)))` clamps by transforming each duration before producing the inner sub-schedule. Both produce the same clamped values but with different structural semantics — the bind form preserves duration-level provenance that the union form discards.

[SCHEDULE_INTERLEAVE_AND_WALL_COMBINATORS]:
- `Schedule.Interleave(Schedule b)` produces `SchInterleave`, which zips both streams into pairs and emits both durations from each pair via `SelectMany`. The emitted count is `2 * min(a.Length, b.Length)`, terminating when the shorter stream exhausts. This doubles the iteration count compared to union or intersection and interleaves step boundaries. Using `Interleave` where union was intended in a retry policy doubles the retry count silently.
- `Schedule.maxDelay(Duration max)` is a `ScheduleTransformer` wrapping `SchMaxDelay`, which maps every emitted duration through `Math.Min(x, max)`. It is a per-emission cap with no effect on iteration count. `Schedule.maxCumulativeDelay(Duration max)` wraps `SchMaxCumulativeDelay`, which yields each duration until the running sum reaches `max` and then terminates via `yield break`. These are semantically orthogonal caps: `maxDelay` restricts spacing magnitude; `maxCumulativeDelay` restricts total wall-clock budget. Both are `ScheduleTransformer`s and combine via `|` against a base schedule. A policy that needs both a per-step cap and a total budget uses `| maxDelay(2*s) | maxCumulativeDelay(30*s)` — both transformers applied to the same base.
- `Schedule.windowed(Duration interval)` produces `SchWindowed`, which re-aligns each delay to the next boundary of the fixed interval window (using `secondsToIntervalStart`). If an action runs longer than the interval, the next delay is zero. This is the cron-like recurrence form: the delay emitted is the remaining time to the next window boundary, not a fixed spacing. It is distinct from `Schedule.spaced(interval)` — spaced emits a constant `interval` regardless of when the action completes; windowed emits the residual time to the next boundary.

[ATOM_POLICY_CELL_MECHANICS]:
- `Atom<A>` carries `public static implicit operator A(Atom<A> atom)` — the atom implicitly reads its current value when used as `A`. A `static readonly Atom<Schedule>` field initialized with a composed schedule can be passed directly to `retry(schedule, computation)` without `.Value`; the implicit conversion reads the current value at call time. This is the live policy cell pattern: the schedule stored in the atom is the active policy value, and any `Swap` on the atom changes the policy for all subsequent retry calls without touching the call site.
- `Prelude.Atom<A>(value, Func<A, bool> validator)` returns `Option<Atom<A>>`, not `Atom<A>`. The validator runs on the initial value before the atom is returned; if the validator returns `false`, the result is `None` — the atom is never constructed in an invalid state. This is the construction-time invariant for policy cells: a `Schedule` atom that must always have at least one iteration encodes that constraint in the validator, and the factory returns `None` if the initial schedule violates it, forcing the caller to handle the admission failure at construction time rather than at first retry.
- `Atom<A>.SwapMaybe(Func<A, Option<A>> f)` aborts the swap and returns the current value unchanged when `f` returns `None`. The `Change` event is not fired on an aborted swap. `Swap(Func<A, A> f)` always proceeds with the new value unless the validator rejects it, in which case the validator's rejection also suppresses `Change`. `Change` fires only on the committed path — after `CompareExchange` succeeds — making the event a reliable policy-change notification: every `Change` invocation reflects a state that was actually installed in the atom.
- `Atom<M, A>` (two-type-parameter form) carries read-only metadata `M` alongside state `A`. Both `Swap` overloads receive `Func<M, A, A>` or `Func<M, A, Option<A>>`. The metadata is fixed at construction time and never changes; it is the immutable context for the swap function. The correct use for parameter algebra: a policy atom that needs access to a fixed configuration record without closing over it uses `Atom<Config, Schedule>` so the swap function can derive a new schedule from the configuration and the current state together. `atomIO<M, A>(metadata, value)` is the effect-wrapped constructor.

[CATCHOF_AGGREGATE_ERROR_WALKING]:
- `catchOf<E, M, A>(Func<E, K<M, A>> Fail)` and `@catch` (predicate form) differ in how they handle aggregate `Error` values. `catchOf` calls `e.IsType<E>()` on the error, which recursively checks whether any sub-error in an aggregate is of type `E`. If the top-level error is `E`, `Fail` receives it directly. If the top-level error is an aggregate containing an `E`, `catchOf` extracts all matching sub-errors and folds them through `Fail` using `ForAllM` (requiring all to succeed) or `FoldM` (for `MonoidK`). A plain `@catch` with a predicate `e => e is E` does NOT do this sub-error walk; it only matches the top-level error. The choice between `catchOf` and `@catch` is load-bearing when the error rail carries composite failures.
- `Error.Filter<E>()` returns an `Error` containing only sub-errors of type `E`. `Error.IsType<E>()` returns `true` if any sub-error satisfies the type predicate. These are the two primitives `catchOf` uses internally. When a custom named catch must discriminate on a structured aggregate, composing `@catch(e => e.IsType<MyError>(), handler)` with `e.Filter<MyError>().ForAllM(handler)` inside the handler body replicates `catchOf` behavior for that specific predicate without taking the `Monad` constraint that `catchOf` requires.

[FACTORY_REF_STRUCT_ADMISSION]:
- `ObjectFactoryAttribute<T>` declares `where T : allows ref struct`. A value object can declare `[ObjectFactoryAttribute<ReadOnlySpan<char>>]`, which directs the generator to produce a custom factory seam accepting `ReadOnlySpan<char>`. The generated seam is a hook for a manually-authored `Validate` implementation that receives the span directly. This enables zero-allocation admission at parsing boundaries: a span from the wire layer is passed to `Create(ReadOnlySpan<char>)` and admitted into a fully typed value object without an intermediate `string` allocation. The `UseForSerialization = SerializationFrameworks.SystemTextJson` flag additionally routes the `Utf8JsonReader`-based deserializer through this same span factory, enabling zero-copy JSON parsing end-to-end.
- `IConvertible<out T>` declares `where T : allows ref struct`, so a value object can implement `IConvertible<ReadOnlySpan<char>>` and expose its key as a span via `ToValue()`. The generated `implicit operator T(VObj vo)` from the standard factory does not use this interface — it calls `ToValue()` only when explicitly wiring the conversion. The interface is the boundary for span-returning projection at boundary adapters; interior code calls `ToValue()` on the value object to obtain the span without an allocation. This is the reverse path from the admission factory: in → span-factory → value object → domain → `ToValue()` → span out.

[PARTIAL_DISPATCH_OVERLOADS_FOR_DEEP_UNIONS]:
- `[UnionSwitchMapOverloadAttribute(StopAt = new[] { typeof(FailureCase) })]` applied to a multi-level regular union generates an additional `Switch` overload that handles `FailureCase` and its siblings at that level as a single parameter, without descending into `FailureCase`'s derived cases. The generated overload receives `Action<FailureCase>` (or `Func<FailureCase, T>` for `Map`) rather than one parameter per leaf. The full exhaustive overload (all leaves) is always generated; the `StopAt` overload is additive. This is the correct shape when an entrypoint needs to delegate handling of an entire sub-hierarchy to one handler without knowing its leaf structure — the parameter is the abstract base case, not a union of all leaves.
- `SwitchMapMethodsGeneration.DefaultWithPartialOverloads` (applicable via `UnionAttributeBase.SwitchMethods`/`MapMethods`) generates, in addition to the exhaustive overload, an overload accepting an `otherwise: Func<TState, TResult>` fallback alongside a subset of per-case handlers. The fallback receives the state for any case not handled by an explicitly provided handler. The partial overload is the correct shape when a dispatch site handles a subset of known cases and delegates the rest to a default policy value rather than forcing the caller to enumerate all cases. The state-threaded signature `Switch<TState, TResult>(TState state, Func<TState, CaseA, TResult> onA, ..., Func<TState, TResult> otherwise)` preserves the same state-threading contract as the full overload.

[IEQUALITY_COMPARER_ACCESSOR_CHAIN]:
- `IEqualityComparerAccessor<TKey>` is a static abstract interface with `static abstract IEqualityComparer<TKey> EqualityComparer { get; }`. It is the type parameter constraint used by the generator when resolving the comparer for a smart-enum's generated `GetHashCode` and `Equals` — the generated code calls `TComparerAccessor.EqualityComparer` at the type level, not at the instance level. `ComparerAccessors.StringOrdinal`, `StringOrdinalIgnoreCase`, and the culture variants all implement both `IEqualityComparerAccessor<string>` and `IComparerAccessor<string>`. Passing a custom comparer accessor as a type argument to the smart-enum key comparer attribute is the mechanism for changing hash and equality semantics without mutating the key type itself.
- The `[KeyMemberEqualityComparerAttribute<TComparerAccessor>]` and `[KeyMemberComparerAttribute<TComparerAccessor>]` attributes (applicable on the smart-enum class) are registered as `allow-multiple = false`; only one comparer accessor governs the key member. A custom accessor implementing `IEqualityComparerAccessor<TKey>` and returned through the static abstract property must be consistent with the `FrozenDictionary` comparer used for any frozen policy table keyed on the same smart-enum — if the smart-enum uses ordinal comparison for its key but the frozen table uses ordinal-ignore-case, lookup semantics diverge. The correct form uses the same `ComparerAccessors.StringOrdinal` accessor for both the smart-enum key member and the `FrozenDictionary.Create(StringComparer.Ordinal, ...)` call.

[ADHOC_UNION_STRUCTURAL_LIMITS]:
- Ad-hoc union `[Union<T1, T2, T3, T4>]` supports up to four type parameters via the generic form; a fifth member requires `AdHocUnionAttribute` (non-generic, attribute-constructor form using `Type` arguments) which supports up to five. Beyond five members, the correct shape is a regular `[Union]` with nested case records, not further extension of the ad-hoc form. The `T1Name` property on `UnionAttribute<T1, T2>` and `AdHocUnionAttribute` overrides the Switch/Map parameter names for that slot without changing the C# type name; this is the mechanism for renaming a type alias at the dispatch boundary while keeping the member type unchanged.
- `UnionAttributeBase.DefaultStringComparison` defaults to `StringComparison.OrdinalIgnoreCase` for string-keyed union discriminants. This governs the generated JSON discriminator comparison and the `ToString`/parsing path, not the structural equality of case instances. Setting it to `StringComparison.Ordinal` on a union that must participate in a case-sensitive wire protocol is the correct form; the default ignore-case behavior produces silent mismatches when union case names share an ordinal-case distinction.

[VALIDATIONERROR_TYPE_WIRE]:
- `[ValidationErrorAttribute<TError>]` applies to a value object or complex value object class/struct and requires `TError : class, IValidationError<TError>`. The attribute registers the error type at the declaring type level, not at the factory method level. The generator invokes `TError.Create(string message)` (the static abstract member) when `ValidateFactoryArguments` sets `validationError` to a non-null string, producing the fully typed error without a runtime cast. A single `[ValidationErrorAttribute<DomainError>]` on a value object type propagates `DomainError` as the error type for all generated factory methods on that type — there is no per-factory override. When two value objects in a hierarchy need different error types, they each declare their own `[ValidationErrorAttribute<T>]`.
- The default `ValidationError` type (a `sealed class` implementing `IValidationError<ValidationError>`) is used when no `[ValidationErrorAttribute<T>]` is present. It carries only a `Message: string` property and implements `Equals` by message content. The typed custom error type replaces `ValidationError` entirely in the generated factory signatures; generated factory methods return `TError?` for the validation error out-parameter rather than `ValidationError?`. This means the `IObjectFactory<T, TValue, TValidationError>` interface is parameterized on the custom error type when `SkipFactoryMethods = false` and `[ValidationErrorAttribute<TCustom>]` is present.

[FROZEN_DICTIONARY_KEYS_IMMUTABLEARRAY]:
- `FrozenDictionary<TKey, TValue>.Keys` returns `ImmutableArray<TKey>`, not `IEnumerable<TKey>` or `ICollection<TKey>`. An `ImmutableArray<TKey>` is a stack-allocated struct wrapping an array; iterating it via `foreach` never boxes and never allocates. The `IDictionary<TKey, TValue>.Keys` explicit implementation returns `ICollection<TKey>`, which allocates a wrapper. When enumerating policy-table keys for validation or bulk projection, the `Keys` property (not the `IDictionary` interface) is the zero-allocation path. The same pattern applies to `FrozenSet<T>`: its item enumeration is via `Items` returning `ImmutableArray<T>`.
- `FrozenDictionary<TKey, TValue>.Empty` is a `static` property on the generic type, returning a singleton per `(TKey, TValue)` pair. Comparing a frozen table to its `Empty` sentinel via reference equality (`ReferenceEquals(table, FrozenDictionary<TKey, TValue>.Empty)`) is the correct absence test for a policy table that may be absent entirely, rather than calling `Count == 0` and paying the property access or checking `Keys.Length`.
