# Materialization Lifecycle and the AOT Regime

[SINGLE_PUBLICATION_POINT]:
- The entire runtime state of a keyed vocabulary funnels through one publication-locked lazy whose payload is a readonly record struct holding the frozen dictionary, the optional span alternate view, and the item list; every admission verb opens with a read of that lazy's value and `Items` is its list. There is no second entry into the state, so "is this vocabulary initialized" reduces exactly to "has the lazy's value been read." A keyless vocabulary substitutes an items-only lazy for the lookup lazy — the two are mutually exclusive per declaration, so no vocabulary ever runs two independent materializers that could both attempt to publish indices.
- Materialization is one closure walking items in declaration order: reject a null item, reject a null key, reject a duplicate key, assign the dispatch index, append. Index correspondence, lookup contents, and fail-fast are products of the same pass, which is why every downstream consumer that reads through the public item accessor inherits the validated, fully-indexed view or the original failure — never an intermediate.

[POISONING_TRANSITIVITY]:
- The metadata plane is a second lazy stacked on the first: the generated metadata's item table projects over the vocabulary's own public item accessor, so it subscribes to the same single materialization and inherits the duplicate-key fail-fast and the cached-exception poisoning transitively. A serializer or persistence layer that merely walks metadata items triggers the identical process-lifetime poisoning the application's own admission would — wire infrastructure is not insulated from a declaration defect by living on the metadata plane.
- The identifier table inside that projection is an index switch mapping ordinal position to the source field-name literal — declaration order is the join key between the materialized item list and the compile-time-known identifiers, so runtime name recovery costs no reflection and no per-item name field.
- The metadata record itself materializes in the type's static constructor as a static auto-property initializer, but it is cheap by design: it carries delegates and lazies and forces no item materialization, so a vocabulary that is referenced but never admitted from pays only the record allocation — heavy work is uniformly deferred to first lazy touch, never to type load.

[CONVERSION_ROUTE_FAILURE_SHAPES]:
- The metadata's two key-conversion routes select by failure shape, not merely representation: the compiled from-key delegate routes through the throwing lookup verb, while the weakly-typed try-get routes through the null-returning `Validate` with a boxed-key value pattern that coerces a wrong-typed key to the default key. The delegate route fails loudly with a typed unknown-identifier exception; the boxed route swallows a type fault into an admission failure that reads as an empty or zero key — a generic caller holding boxed keys must pre-type them before entering the erased plane, because on that plane type mismatch and zero-key admission are indistinguishable.
- The expression-tree twin of each conversion delegate exists so a query provider can splice key conversion into a larger expression it will translate; the delegate is the in-memory path, the lambda the translatable one, and both carry the same body so the two planes cannot drift.

[AOT_CLIFF]:
- A non-generic vocabulary's serializer participation is fully static: the converter-factory attribute is stamped with both type arguments closed at generation time, so no runtime generic construction and no dynamic code is required — trim- and AOT-clean by construction. A generic vocabulary cannot close its converter at generation time, so the generator emits a file-scoped converter factory whose create path constructs the closed converter from the runtime type via runtime generic instantiation — a dynamic-code operation. The same declaration is AOT-clean closed and dynamic-code-dependent open: leaving a vocabulary generic moves its wire path across the AOT cliff while its in-process surface stays identical.
- Generic vocabularies additionally lose only the component-model type-descriptor seam (suppressed when the vocabulary or any container is generic); the factory and metadata conversion routes survive, so admission is never the casualty — design-time binding and the JSON path are.

[LOAD_CONTEXT_PARTITION]:
- The reflection metadata cache is keyed by the concrete queried type, and type identity is itself per load context, so the cache partitions cleanly along the same isolation seam as the item singletons — one context's poisoning, indices, and metadata never alias onto another's types. Negative probes are rejected before caching rather than memoized, so speculatively probing arbitrary types in serializer hot paths neither allocates nor poisons the cache, and a type that later becomes resolvable is re-probed cheaply.
