# Collections and Traversal

[COLLECTION_CHOICE]:
- `Seq<A>` is backed by one of four concrete internal types: `SeqStrict<A>` (evaluated, array-backed with a `start` offset for O(1) `Tail`/`Init` without copying), `SeqLazy<A>` (wraps `IEnumerable<A>` and evaluates elements into a cache array at most once per element), `SeqConcat<A>` (deferred segment tree built by `Concat` or `+`), and `SeqEmptyInternal`. Calling `.Strict()` forces any lazy or concat internal to fully evaluate into a `SeqStrict<A>`.
- `Seq<A>` is preferred over `Lst<A>` in all cases except positional insertion: `Lst<A>` is backed by `LstInternal<A>`, which uses a `ListItem<T>` AVL tree indexed by position — a completely independent AVL implementation from `Map<K,V>`. `ListModule.GetItem(root, index)` walks by comparing `index` against `Left.Count`, giving O(log n) positional access; `ListModule.Insert(root, index, value)` walks by index and rebalances. `Seq<A>` has no O(log n) mid-sequence insert. Outside positional mutation, `Seq<A>` is faster because structural sharing via the `start` offset avoids copying on `Cons` and `Tail`. `Lst<A>` implements no `Natural` on its trait class; converting to `Seq<A>` at the HKT layer goes through `Foldable<Lst>.ToSeq<A>`, which folds the entire tree — O(n). `Iterable` implements `NaturalMono<Iterable, Lst>`, which registers `CoNatural<Lst, Iterable>` at the dispatch position; `CoNatural<Lst, Iterable>.CoTransform` maps `K<Iterable, A> → K<Lst, A>` — i.e., it materializes an `Iterable` into a `Lst`, not the reverse. No `Natural<Lst, Iterable>` arrow exists on the `Iterable` trait class; `Lst<A>` does not enter `Iterable`-consuming combinators via this path without the O(n) fold cost.
- `Arr<A>` is an immutable wrapper around a raw array: O(1) random read, O(n) modification because every change copies the entire backing array. Use when the access pattern is overwhelmingly read-intensive and the collection is short-lived or small; do not use when the sequence grows incrementally.
- `HashMap<K, V>` is backed by a CHAMP trie. Operations are O(1) amortized but unordered. `Map<K, V>` is backed by an AVL tree: O(log n) operations, iteration is key-ordered. Use `Map<K, V>` when ordered iteration or range queries (`FindRange(keyFrom, keyTo)`) are needed; use `HashMap<K, V>` otherwise.
- `HashSet<A>` and `Set<A>` mirror the same CHAMP vs AVL distinction. `Set<A>` supports ordered min, max, and range operations; `HashSet<A>` does not guarantee iteration order.
- `SeqEmpty` implicitly converts to every collection type — `Seq<A>`, `Arr<A>`, `Lst<A>`, `HashMap<K,V>`, `HashSet<A>`, `Map<K,V>`, `Que<A>`, `Stck<A>`, and others — via `op_Implicit(SeqEmpty)`. Use `Seq<A>.Empty` or the `SeqEmpty` sentinel at call sites that need a universal empty; do not construct any collection type explicitly just to pass an empty.
- `Iterable<A>` is a lightweight `IEnumerable<A>` wrapper that implements the full trait hierarchy (`Monad`, `Traversable`, `MonoidK`, and others) to make HKT combinators apply to arbitrary lazy enumerables. It is not a substitute for `Seq<A>` where evaluation count matters; it is the bridge when a caller-owned `IEnumerable<A>` must enter a trait-polymorphic context. Construction from a `ReadOnlySpan<A>` is available via `Iterable<A>.FromSpan(span)`, which calls `span.ToArray()` internally — not zero-copy at construction.
- `BiMap<A, B>` is a bidirectional map backed by two `Map<A,B>` and `Map<B,A>` AVL trees, giving O(log n) lookup in either direction. It is the canonical choice when a symmetric key–value bijection is the domain constraint, not a pair of `HashMap` instances.
- `createRange` overloads exist for every collection type from `ReadOnlySpan<A>`: `Seq`, `Arr`, `Lst`, `Que`, `Stck`, `HashMap`, `HashSet`, `Map`, `Set`, and their `EqK`/`OrdK` variants. `Seq.createRange(ReadOnlySpan<A>)` constructs via `span.ToArray()` internally, so the resulting `SeqStrict<A>` is not zero-copy at construction but supports zero-copy `AsSpan()` thereafter. The span constructor is always the lowest-allocation creation path when data is already in a span or an array.
- `toSeq(IEnumerable<A>)` constructs a `SeqLazy<A>` for arbitrary `IEnumerable<A>` inputs, but eagerly copies into a `SeqStrict<A>` when the argument implements `IList<A>`, `ICollection<A>`, or is already an array — the strict path is taken at construction rather than on first access. Wrapping an `IAsyncEnumerable<A>` into `Iterable<A>` via `createRange` is the async parallel; `.IsAsync` is an internal flag used by the type to route operations to async overloads and is not readable from calling code.

[STRICTNESS_AND_LIFETIME]:
- `Seq<A>.Add(A)` appends to the end but forces complete evaluation of any lazy internal: the API remark states "Forces evaluation of the entire lazy sequence so the item can be appended." Operations that mutate the logical end of a lazy `Seq<A>` pay the full evaluation cost; prepend (`Cons`) does not. When building a result sequence incrementally from a lazy source, prepend and reverse outperforms repeated `Add`.
- `SeqLazy<A>.Count` calls `InternalStrict()`, which iterates the entire lazy source into the internal cache array before returning the count. Any code path that reads `.Count` on a `Seq<A>` backed by `SeqLazy<A>` forces full evaluation as a side effect. `Foldable<Seq>.Count` dispatches to `ta.As().Count` — the HKT form is not exempt. `Foldable.length` at the trait layer also calls `Count` and pays the same cost.
- `SeqLazy<A>.AsSpan()` calls `Strict()`, which also calls `InternalStrict()`. There is no zero-copy span path for a lazy `Seq<A>`; any span access forces full evaluation. `SeqStrict<A>.AsSpan()` returns `new ReadOnlySpan<A>(data, start, count)` — zero-copy. `Arr<A>.AsSpan()` returns a zero-copy slice of its backing array. Avoid `AsSpan()` on sequences of unknown internal type inside hot paths.
- `Arr<A>.AsSpan()` and its `AsSpan(int start)` and `AsSpan(int start, int count)` overloads are zero-copy slices into the backing array. These are the correct primitives when passing a slice of an `Arr<A>` to a span-consuming API without copying; elements outside the slice remain allocated.
- `SeqConcat<A>` is constructed when two sequences of mixed internal types are concatenated via `Concat` or `+`; it holds a `Seq<ISeqInternal<A>>` of segments and defers merging until an operation forces it. `SeqConcat.Count` sums the counts of all segments in O(segments) time, but `AsSpan()` and any operation requiring a contiguous array calls `Strict()`, which allocates a new backing array and copies all segments. Iterating a `SeqConcat<A>` enumerates each segment's enumerator in sequence without allocation, so `Fold` and `TraverseM` over a `SeqConcat<A>` are cheap. Repeated `+` concatenation without a terminal `Strict()` builds a segment tree that costs O(n) to enumerate each time instead of the amortized O(1) of a single `SeqStrict<A>`.
- `SeqLoan<A>` wraps an `ArrayPool<T>`-rented array. It implements `IDisposable` with a finalizer safety net that returns the array to its originating pool. `SeqLoan<A>` exposes `Fold`, `FoldBack`, `Iter`, `Exists`, `ForAll`, `Skip`, `Take`, `Append`, and `ToSeq`, but no `Map`, `Filter`, or `Bind` — those operations do not exist on this type; calling code must promote to an owned `Seq<A>` via `ToSeq()` first and then apply transformations. Calling `Dispose()` while another thread is reading the structure produces undefined behavior.
- `SeqLoan<A>.ToSeq()` creates a safe owned copy. Consume the loan in one pass, then promote to an owned `Seq<A>` and dispose the loan. Holding a `SeqLoan<A>` beyond the scope of the originating `ArrayPool` rent is a use-after-return bug that the finalizer mitigates but does not prevent.
- `SeqStrict<A>` has a `consDisallowed` flag. Once set, further `Cons` calls on that strict representation are blocked and the sequence allocates a new backing block instead of mutating the existing one. This flag protects structural sharing: a `Tail` that returns a sub-slice of the same array would be corrupted if the original sequence later prepended into that array. `SeqLazy<A>` sets `consDisallowed` when a derived view — such as `Tail` — is constructed over the shared cons buffer; the flag is not set during element-by-element emission into the cache.

[TRAVERSE_AND_SEQUENCE]:
- `TraverseM<M, B>(Func<A, K<M, B>>)` on `Seq<A>`, `Arr<A>`, `Lst<A>`, `HashSet<A>`, and `Set<A>` maps into a monad `M` and collects results by sequencing via monadic bind left-to-right. This is the correct form when the monad has nontrivial short-circuit behavior (`Fin<A>`, `Eff<A>`, `Option<A>`): on the first `None` or `Fail`, effect application for remaining elements stops. The fold still enumerates and forces the source structure up to that point — `TraverseM` over a `SeqLazy<A>` evaluates elements from the source as it proceeds, not lazily on demand; short-circuit stops the bind chain, not the source evaluation already performed.
- `Traverse<F, B>(Func<A, K<F, B>>)` maps into an applicative functor `F` and collects by applicative sequencing. For `Validation<FAIL, A>`, this accumulates all errors before returning; `TraverseM` with `Validation` in monad mode short-circuits on the first error. Use `Traverse` when all elements must be validated; use `TraverseM` when the first failure should abort.
- `Traversable.sequence<T, F, A>(K<T, K<F, A>>)` is the no-mapping form: when elements are already wrapped values, `sequence` swaps the container and functor layers without a prior `Map`. `sequenceM<T, M, A>` is the monadic variant with the same short-circuit semantics as `TraverseM` but without the mapping step.
- The `.As()` call is required after `TraverseM` or `Traverse` on any collection when the return type is `K<F, Seq<B>>` — the trait system returns the abstract HKT kind and `.As()` recovers the concrete type. Omitting `.As()` leaves the result in the abstract `K<F, Seq<B>>` form, which cannot be used directly.
- `Iterable<A>.TraverseIO<M, B>(Func<A, K<M, B>>, K<Iterable, A>)` is the async-aware traversal overload unique to `Iterable<A>`. When the iterable's `IsAsync` property is true, this overload correctly awaits each step. `Seq<A>` has no `TraverseIO` because it is always synchronously evaluated.
- `Arr<A>.Traverse<F, B>` and `Arr<A>.TraverseM<F, B>` both pre-allocate a `SeqStrict<B>` of exactly `Count` elements before folding: `new SeqStrict<B>(new B[ta.As().Count], 0, 0, 0, 0)`. The fold then calls `SeqStrict<B>.Add(b)`, writing into the pre-allocated backing array. `Seq<A>.Traverse<F, B>` and `Seq<A>.TraverseM<F, B>` do not pre-allocate; they start with `Seq<B>.Empty` and call `bs.Add(b)` per element. For short-circuit monads where most elements are expected to succeed, `Arr<A>.TraverseM` is lower-allocation than `Seq<A>.TraverseM` when the input length is known.
- `PartitionFallible<F, M, A>(K<F, K<M, A>>)` folds a collection of monad values `K<M, A>` where `M : Fallible<M>` into `K<M, (Seq<Error> Fails, Seq<A> Succs)>`. Every effect in the collection is executed sequentially without short-circuiting; failures are accumulated alongside successes. The typed-error variant `PartitionFallible<E, F, M, A>` where `M : Fallible<E, M>` accumulates `Seq<E>` instead of `Seq<Error>`. Both variants exist for `Seq<K<M,A>>`, `Iterable<K<M,A>>`, `Lst<K<M,A>>`, `HashSet<K<M,A>>`, `Set<K<M,A>>`, and `IEnumerable<K<M,A>>`. This is the dual of `TraverseM`: use `PartitionFallible` when all effects must run and all outcomes must be collected.

[FILTER_MAP_VIA_CHOOSE]:
- `Choose<A, B>(Func<A, Option<B>>)` on `Seq<A>`, `Stck<A>`, and `Map<K, V>` applies a function that returns `Option<B>`, discards all `None` results, and returns the `Some` values. This is atomic filter-map in one pass; it replaces `Filter(predicate).Map(projection)`, which makes two passes and requires the projection to repeat the same guard. `Map<K,V>.Choose` and its two-argument form `Choose(Func<K, V, Option<R>>)` carry the key through. `Arr<A>` and `Lst<A>` have no `Choose` instance or extension method; they reach choose-semantics only through the `MonoidK<M>` and `Monad<M>` constrained HKT path.
- `Seq.choose<A, B>(Seq<A>, Func<A, Option<B>>)` is the static-module form; `SeqExtensions.Choose<A, B>(Seq<A>, Func<A, Option<B>>)` is the extension form. Prefer the extension form at call sites that chain.
- `SeqExtensions.Choose<A, B>(Seq<A>, Func<int, A, Option<B>>)` is the indexed form: the position of each element is passed alongside the value. Use this for index-aware filter-map without a prior zip-with-index step.
- `MonoidK.choose<F, A, B>(K<F, A>, Func<A, Option<B>>)` is the HKT-polymorphic form: for any `F` constrained to both `MonoidK<F>` and `Monad<F>`, it propagates the structure only for elements where the mapping returns `Some`. This is how `Arr<A>` and `Lst<A>` reach choose-semantics — via this trait path, not a concrete instance method.
- `Seq<A>.Partition(Func<A, bool>)` is a single-pass, mutable-accumulator implementation that iterates via `GetEnumerator` and appends to two `Seq<A>` result values using `Add`. `Foldable.Partition<T, A>(K<T, A>, Func<A, bool>)` is the HKT form, returning `(Seq<A> True, Seq<A> False)`; for `Seq<A>` it dispatches to the single-pass implementation. `Iterable<A>.PartitionIO(Func<A, bool>)` is async-aware: when `IsAsync` is true it materializes both partitions by iterating the `IAsyncEnumerable<A>` source under a `CancellationToken` from `EnvIO` — not lazy; both result iterables are fully evaluated before returning.

[IMMUTABLE_FOLDS]:
- `Fold<S>(S, Func<S, A, S>)` on `Seq<A>` and `SeqLoan<A>` is left-associative, reducing from head to tail. It is stack-safe for any length. `FoldBack<S>(S, Func<S, A, S>)` on `SeqStrict<A>` and `SeqLoan<A>` is implemented as a plain reverse for-loop over the backing array, which is O(1) stack for any length. The XML doc for `SeqStrict.FoldBack` mentions a 5,000-item recursion branch (`FoldBackRec`) that was faster for small strict sequences, but that remark describes a prior implementation; the current implementation does not use recursion or a CPS fallback. `Rev().Fold` and `FoldBack` are equivalent over strict sequences in stack cost; the real reason to prefer `Rev().Fold` is when the accumulator must be constructed in reverse without relying on FoldBack's order semantics.
- `FoldWhile<S>(Func<A, Func<S, S>>, Func<(S, A), bool>, S, K<T, A>)` from the `Foldable<T>` trait takes a curried folding function and a predicate over the `(State, Value)` pair and stops as soon as the predicate returns false. The predicate receives the current state and the next element, so it can inspect accumulated state to decide when to stop.
- `FoldUntil` is the complement: it continues while the predicate is false and stops on the first true. The monadic variants `FoldWhileM` and `FoldUntilM` take a `Func<A, Func<S, K<M, S>>>` and thread a monad through the fold; their predicate is `Func<A, bool>` (element-only), not a `(S, A)` pair.
- `FoldBackMaybe<A, S>(Func<A, Func<S, Option<S>>>, S, K<T, A>)` stops as soon as the folding function returns `None`. The function receives the element first, then the accumulator — `Func<A, Func<S, Option<S>>>` — so it decides early-exit on the element alone without inspecting accumulated state.
- `scan<S, A>(Seq<A>, S, Func<S, A, S>)` and `scanBack<S, A>(Seq<A>, S, Func<S, A, S>)` both return a `Seq<S>` of all intermediate states including the initial state. Use `scan` when downstream requires all running totals, not just the terminal value.

[STRUCTURAL_SHARING_AND_PERFORMANCE]:
- `Seq<A>.Cons(A)` prepends without copying when `consDisallowed` is false: if the backing `SeqStrict<A>` array has unused capacity before `start`, the new head is written there and `start` is decremented. Prepend is O(1) amortized and sharing is preserved for the tail. Once capacity is exhausted or `consDisallowed` is set, a new array is allocated.
- `Seq<A>.Tail` and `Seq<A>.Init` are O(1) for `SeqStrict<A>` because they adjust `start` and `count` over the same backing array rather than copying. Materializing `Head` and `Tail` in a recursive descent over a `SeqStrict<A>` is therefore structurally safe; it does not degrade like array copying would.
- `Map<K, V>` and `Set<A>` support `Skip(int)` — returning a new tree without the first N elements — in O(n) time. The operation walks the tree; there is no O(log n) structural prefix sharing.
- `Map<OrdK, K, V>.FindRange(K, K)` delegates to a `yield return` generator that walks the AVL tree in-order and emits values only when the key is within range. The range walk is lazy: elements are yielded on demand during enumeration. Calling `FindRange` is O(log n) to position at the lower bound; subsequent iteration is O(k) for k results. There is no eager materialization. For ordered-key range queries, `Map` is the correct type and `FindRange` is the correct combinator — no `Where` filter over a full `ToSeq()` is needed.
- `Seq<A>.Case` returns an `object` shaped as `null` (empty), `A` (singleton), or `(A, Seq<A>)` (head–tail) for pattern-matching. `Map<K, V>.Case` and `Arr<A>.Case` follow the same pattern: `null` for empty, `(K, V)` for singleton, `((K, V), Seq<(K, V)>)` for more. This is how structural decomposition works across all collection types — directly into a switch expression without `.IsEmpty` guards.

[INDEXED_PROJECTION]:
- `Seq<A>.Map<B>(Func<A, int, B>)` is the indexed map overload; the remark states "Exposes an index for each map." This eliminates the zip-with-index pattern entirely when projection requires positional information.
- `Seq.generate<A>(int, Func<int, A>)` produces a `Seq<A>` whose internal representation is `SeqLazy<A>`: the implementation wraps a `Range(0, count).AsIterable().Map(generator)` pipeline and does not eagerly evaluate the factory. Elements are forced into the internal cache array on first access per element. Use `.Strict()` after construction when all elements will be accessed and repeated evaluation of the factory should be prevented; use `unfold` when length is determined by a stopping condition.
- `unfold<S, A>(S, Func<S, Option<(A, S)>>)` emits the `A` value from the tuple and advances state; the initial state is never emitted. All `unfold` overloads return `IEnumerable<A>` — they are `yield return` generators and are fully lazy. Wrapping the result in `toSeq()` produces `SeqLazy<A>`, which evaluates on demand. `unfold<S>(S, Func<S, Option<S>>)` yields the seed state first before transitioning — the initial state is always emitted; use `Skip(1)` or the tuple form to suppress it. Multi-state overloads exist up to four state components (`unfold<S1,S2,A>` through `unfold<S1,S2,S3,S4,A>`), avoiding tuple construction at call sites when the state is naturally multi-dimensional.
- `Seq<A>` throws `IndexOutOfRangeException` from its `this[Index]` indexer when out of range; there is no safe indexer on the public struct. The safe access idiom is `.HeadOrNone()` and structural `Tail` decomposition, or `Foldable<T>.At<A>(K<T, A>, Index)`, which returns `Option<A>` and handles both forward and from-end `Index` positions. The default implementation uses `FoldWhile` for forward access and `FoldBackWhile` for from-end access. The `Foldable<Seq>.At` override dispatches to `seq[index]` after a bounds guard — O(1) for `SeqStrict<A>` internally. For `Set` and `Map`, `At` uses the default fold — O(k) traversal — because AVL trees carry no O(log n) positional index.
- `Seq<T>.Zip<U, V>(Seq<U>, Func<T, U, V>)` delegates to `Enumerable.Zip`, which stops at the shorter sequence. There is no `ZipAll`, `ZipLong`, or padding overload in the `Seq` API. When zip-to-longest behavior is needed, prepend or append the shorter sequence to bring it to the required length using domain knowledge, or use `Iterable<A>.Zip<B, C>` after confirming equal length via domain invariants. `Iterator<A>.Zip(Iterator<A>)` only zips same-typed iterators and returns `Iterator<(A First, A Second)>` — no projection variant is available at the `Iterator` level.

[HASHMAP_ALGEBRA]:
- `HashMap<K, V>.operator+` delegates to `Combine`, which calls `TrieMap.Append`, which calls `TryAddRange` with `UpdateType.TryAdd` — leaving the existing node unchanged if the key is already present. Both `operator+` and `Union(IEnumerable<(K,V)>)` with no merge function are left-biased: keys already in the left-hand map are preserved on collision. The `+` operator on `Map<K,V>` has identical left-biased semantics via its AVL-tree `TryAdd` equivalent. Only `HashMap<K, V>.Union(IEnumerable<(K, V)>, WhenMatched<K, V, V, V>)` provides controlled merge semantics; pass `(_, l, r) => r` explicitly to achieve right-bias.
- `HashMap<K, V>.Union<W, R>(IEnumerable<(K, W)>, WhenMissing<K, V, R>, WhenMissing<K, W, R>, WhenMatched<K, V, W, R>)` is the full heterogeneous merge: left-only keys pass through `WhenMissing MapLeft`, right-only keys through `WhenMissing MapRight`, and matched keys produce the result type `R` through `WhenMatched`.
- `HashMap<K, V>.Intersect(IEnumerable<(K, V)>, WhenMatched<K, V, V, V>)` retains only keys present in both and applies the merge function. `Except(IEnumerable<K>)` removes all keys present in the right-hand set; `Except(IEnumerable<(K, V)>)` is key-only difference (values on the right are ignored).
- `HashMap<K, V>` (two-type-parameter form) internally uses `TrieMap<EqDefault<K>, K, V>`. `EqDefault<K>` first tries to resolve an `Eq<K>` trait implementation at static-type initialization via `EqResolve<K>`, using reflection. If no `Eq<K>` instance is found, it falls back to `EqualityComparer<K>.Default`. The reflection-based path boxes arguments and is measurably slower than the compiled delegate path. `HashMap<EqK, K, V>` (three-type-parameter form) with an explicit `EqK : Eq<K>` constraint routes directly through the static interface method `EqK.Equals` without reflection. For high-frequency insertions or lookups with custom key types, the explicit `EqK` form is the correct choice. The same `EqResolve`/`HashableResolve` pattern governs `HashSet<A>` vs `HashSet<EqA, A>`.
- `AtomHashMap<K, V>.SwapKey(K, Func<Option<V>, Option<V>>)` is a single atomic CAS over one entry: `None` input means the key was absent, `Some(x)` means it was present. `Some(y)` from the delegate adds or updates; `None` removes. This replaces a read–modify–write sequence with one atomic operation and is the correct primitive for conditional update-or-insert.
- `AtomHashMap<K, V>.MapInPlace(Func<V, V>)` mutates all values atomically; the remark warns the function may be called multiple times under thread contention. Keep this function fast, pure, and free of side effects.
- `TrackingHashMap<K, V>` wraps `HashMap<K, V>` and records a change morphism since the last `Snapshot()` call. The `Changes` property returns a `HashMap<K, Change<V>>` where `Change<V>` is one of `EntryAdded<V>`, `EntryMappedTo<V>` (records the new value), `EntryMappedFrom<V>` (records the prior value), `EntryMapped<A, B>` (records both prior and new values with distinct type parameters), or `EntryRemoved<V>`. The change map stores at most one morphism per key regardless of how many times it was mutated — it represents the net effect of all mutations since the last snapshot. `Snapshot()` resets the change ledger by constructing a new `TrackingHashMap<K, V>` with the current value map and an empty change map. This is the correct primitive for incremental UI refresh, delta sync, or change-feed emission where intermediate states are irrelevant. `AtomHashMap<K, V>.ToTrackingHashMap()` snapshots the current atomic value into a new `TrackingHashMap<K, V>`; subsequent mutations of the `AtomHashMap` are not reflected in the snapshot. The pattern for atomic-plus-change-tracking is `ToTrackingHashMap()` after each logical batch, not continuous subscription.
- `ToHashMap` on `IEnumerable<(K1, K2, V)>` produces `HashMap<(K1, K2), V>` using a tuple key. Overloads exist for 3- and 4-component composite keys. These use `ValueTuple` structural equality via `EqDefault<(K1,K2)>`. For custom key semantics over compound keys, use `HashMap<EqK, (K1, K2), V>` with an explicit `EqK` and construct via the `EqK`-scoped `ToHashMap<EqK, K1, K2, V>` overload.

[ATOMIC_COLLECTION_STATE]:
- `AtomSeq<A>` stores the active sequence in a field typed `ISeqInternal<A>`. `Swap(Func<Seq<A>, Seq<A>>)` atomically replaces the internal with CAS retries and returns `Unit`. `SwapIO(Func<Seq<A>, Seq<A>>)` wraps the same CAS-retry swap in an `IO<Unit>` — the resulting sequence value is not captured or returned; call `.Value` to read the updated state after the effect runs.
- `AtomSeq<A>.MapInPlace(Func<A, A>)` maps all elements atomically. Like `AtomHashMap<K,V>.MapInPlace`, the mapping function may execute multiple times under contention.
- `AtomHashMap<K, V>` and `AtomSeq<A>` expose no `Subscribe` or reactive-stream surface. State is read by reading the current value via `.Value` or property accessors; change notification is not a contract of these types.

[TRAVERSAL_DECOMPOSITION]:
- `Seq.tails<A>(Seq<A>)` returns all final segments from longest to shortest: for `[a, b, c]` the result is `[[a,b,c], [b,c], [c]]` — three segments, no trailing empty. The module function's loop conses only non-empty suffixes; `tailsr<A>` uses the same non-empty semantics via recursion. The `Seq<A>.Tails` instance property includes the trailing empty segment and returns four elements for a three-element input. Each segment is a `Tail` slice — O(1) per tail on `SeqStrict<A>` due to the `start` offset — so the full `Seq<Seq<A>>` costs O(n) to construct.
- `Seq<A>.NonEmptyInits` is O(n²): the implementation iterates `i` from `1` to `Count`, calling `Take(i)` each time. Each `Take(i)` on a `SeqStrict<A>` creates a new struct wrapping the same backing array with an adjusted `count` field — O(1) per `Take` — but the enclosing `Seq<Seq<A>>` is built via `Add` in a loop, paying the append cost. `tails` and `inits` are structurally asymmetric: tails are O(n), inits are O(n²).
- `Seq.span<A>(Seq<A>, Func<A, bool>)` splits at the first element where the predicate is false, returning `(prefix, suffix)` as a tuple. The prefix is the maximal initial segment satisfying the predicate. This is a single-pass structural split with no intermediate allocation beyond the two `Seq<A>` slices.
- `Patch<EqA, A>` is an edit-list groupoid over ordered sequences. `Patch.diff<EqA, A>(IEnumerable<A>, IEnumerable<A>)` computes the minimal patch between two sequences; `Patch.apply<EqA, A>(Patch<EqA,A>, Lst<A>)` applies it. Patches compose via `append` and invert via `inverse`. `Patch.applicable(patch, document)` and `Patch.composable(p, q)` are predicates confirming whether a patch can be validly applied or composed before doing so. This is the correct primitive for versioned or diffable sequence state, replacing ad-hoc before/after comparison.

[NONEMPTY_ADMISSION]:
- `AsIterableNE<A>(this IEnumerable<A>)` returns `Option<IterableNE<A>>` — the `None` case represents an empty source without throwing. `AsIterableNE<A>(this IAsyncEnumerable<A>)` returns `IO<IterableNE<A>>`, which defers the emptiness check to effect execution.
- `AsIterableNE<A>(this Seq<A>)` is implemented as `xs.Head.Map(h => new IterableNE<A>(h, xs.Tail.AsIterable()))` — a single `Option.Map` with no allocation beyond the record. This makes `Seq<A> → Option<IterableNE<A>>` the allocation-minimal non-empty admission path.
- `IterableNE<A>` is a `record(A Head, Iterable<A> Tail)` and is the required input type for `Applicative.Actions`. `actions<F, A>(IterableNE<K<F, A>>)` folds left-to-right over the tail using `h.Action(t)` — left-to-right action sequencing, discarding all but the final value. `actions` over an empty collection is not expressible because `IterableNE` guarantees at least one element, eliminating the empty-applicative-sequence problem at the type level.

[FOLDABLE_TRAIT_NUMERIC_AGGREGATION]:
- `Foldable<T>.Sum<A>` requires `A : IAdditionOperators<A, A, A>, IAdditiveIdentity<A, A>`; the default implementation is a left fold via `Fold((A x) => (A y) => x + y, A.AdditiveIdentity, ta)`. `Product<A>` requires `IMultiplyOperators<A, A, A>, IMultiplicativeIdentity<A, A>`. `Average<A>` requires `A : INumber<A>` and computes count and total in a single fold pass via a `(Count, Total)` state tuple — not two passes.
- `Average<A, B>(Func<A, B>, K<T, A>)` maps each element before accumulating, requiring `B : INumber<B>`. Use this when elements are not directly summable — extracting a `double` metric from a domain value — without a prior `Map` that allocates an intermediate collection.
- `Foldable<T>.Min<OrdA, A>` and `Max<OrdA, A>` accept an explicit `OrdA : Ord<A>` constraint enabling non-default ordering; the default implementations are full left-folds. `Set<A>` and `Map<K, V>` override the `IComparable<A>`-constrained `Min<A>` and `Max<A>` forms to return the AVL tree's `Min` and `Max` properties — O(log n); the `OrdA`-parameterized forms are not overridden on these types and fall back to the default fold.

[NATURAL_TRANSFORMATIONS_AND_COLLECTION_CONVERSION]:
- `Natural<F, G>` is a static interface with one abstract method `Transform<A>(K<F, A>) → K<G, A>`; `CoNatural<F, G>` is its dual with `CoTransform<A>(K<G, A>) → K<F, A>`. `NaturalMono<F, G>` implements `Natural<F, G>` and `CoNatural<G, F>`, registering the single F→G arrow in both dispatch positions — the default `CoTransform` on `CoNatural<G, F>` still produces `K<F, A>` from `K<G, A>` by delegating to `Transform`. `NaturalMono` does not supply a G→F arrow; it is not an isomorphism. `NaturalIso<F, G>` implements `Natural<F, G>` and `CoNatural<F, G>`, supplying both F→G and G→F arrows — the true bidirectional isomorphism interface. The static entry points are `Natural.transform<F, G, A>(K<F, A>)` and `CoNatural.transform<F, G, A>(K<G, A>)`.
- `Iterable` implements `NaturalMono<Iterable, Arr>`, `NaturalMono<Iterable, Seq>`, `NaturalMono<Iterable, Lst>`, `NaturalMono<Iterable, Set>`, and `NaturalMono<Iterable, HashSet>`. Each `NaturalMono<Iterable, X>` registers the Iterable→X materialization arrow at both the `Natural<Iterable, X>` and `CoNatural<X, Iterable>` dispatch positions; no X→Iterable natural transformation is registered by these instances. There is no round-trip path supplied by the `Iterable` trait class; converting a concrete collection to `Iterable` requires explicit `AsIterable()` or `createRange` calls, not a natural transformation.
- `Arr` implements `Natural<Arr, Seq>`, `Natural<Arr, Iterable>`, `Natural<Arr, Lst>`, `Natural<Arr, Set>`, and `Natural<Arr, HashSet>` — outbound only. `Seq` implements no `Natural` on its trait class. For generic code parameterized on a collection kind `F` that must convert to `Seq`, the constraint `where F : Natural<F, Seq>` is satisfiable by `Arr`, `Iterable`, `IterableNE`, and `Iterator` but not by `Seq` itself — `Seq`-typed inputs must be handled directly.
- `Iterator<A>` implements `Natural<Iterator, Arr>`, `Natural<Iterator, Seq>`, `Natural<Iterator, Lst>`, `Natural<Iterator, Set>`, `Natural<Iterator, Iterable>`, and `Natural<Iterator, HashSet>` — outbound only, identical matrix to `Arr`. `Iterator` has no `CoNatural`; there is no structural transformation back into an `Iterator` from another kind via the trait.
- `Memo.transform<F, G, A>(Memo<F, A>)` wraps the natural transform inside a deferred `Memo<G, A>`: the underlying `K<F, A>` is not materialized until `.Value` is first accessed, then cached. `Memo.cotransform<F, G, A>(Memo<G, A>)` is the dual. A `Memo<Arr, A>` passed to `Arr.Apply` avoids constructing the right-hand `Arr` entirely when the left-hand function set is empty.

[ITERATOR_LINKED_LIST_LIFECYCLE]:
- `Iterator<A>` is an abstract class with two structural cases: `Cons` (non-empty, abstract `Head`/`Tail`/`Clone`/`Split`) and `Nil` (terminal, `Tail` returns `this`). The backing implementations are `ConsFirst` (lazily realized from `IEnumerable<A>` via spin-lock CAS), `ConsValue` (eagerly allocated head plus tail), `ConsValueEnum` (backed by a live `IEnumerator<A>`), and `ConsValueLazy` (thunked tail function). Each is `IDisposable` and releases the source enumerator or backing value explicitly.
- `Clone()` on `ConsFirst` and `ConsValueEnum` creates a new iterator starting from the same logical position without sharing the head reference. After `Clone()`, the original node's reference is no longer retained by the traversal stack, allowing the linked chain behind the current position to be GC'd. Without `Clone()`, a long traversal builds a reachable linked list from the beginning of the sequence to the current position.
- `Split()` is stronger than `Clone()`: it breaks the backward chain so any `Iterator<A>` reference that led to the current position terminates there. `Map`, `Bind`, and `AsEnumerable` call `Clone()`, not `Split()`, so the chain break for infinite generators is the caller's responsibility.
- `Iterator<A>` is `IDisposable` with abstract `Dispose()`. `ConsFirst` CAS-disposes the `IEnumerator` and resets `firstAcquired` to 0; `ConsValueEnum` disposes the enumerator. There is no finalizer safety net on `Iterator<A>` — unlike `SeqLoan<A>` — so not disposing an `Iterator<A>` backed by a non-rewindable `IEnumerator` (database cursor, file reader) leaks the handle for the GC lifetime.

[MONAD_RECUR_COLLECTION_SEMANTICS]:
- `Monad.Recur<A, B>(A, Func<A, K<M, Next<A, B>>>)` is the trampolining recursion primitive. `Next<A, B>` is a discriminated struct with `IsLoop` (recurse with value `A`) and `IsDone` (return value `B`). Implicit conversions exist from `Loop<A>` to `Next<A, B>` and from `Pure<B>` to `Next<A, B>`, so arms are written as `new Loop<A>(nextState)` and `new Pure<B>(result)`.
- For `Seq`, `Arr`, `Lst`, `Set`, and `HashSet`, `Recur` is breadth-first: each generation step collects all loop-continuing values from the entire previous generation into a swap-buffer `List<A>`, then fans out to the next generation. `Done` values are yielded immediately. `Recur` over a collection monad therefore enumerates a tree breadth-first, not depth-first — the distinction is observable when the function returns multiple `Loop` and `Done` values from one step.
- For `Iterable`, `Recur` delegates to `Monad.iterableRecur`, which runs as an `async IAsyncEnumerable<B>` generator. When `M : Natural<M, Iterable>, CoNatural<M, Iterable>`, the recursion is materialized by transforming the result through the `Iterable` natural transformation. The `EnvIO` cancellation token is threaded through the async state machine.
- `Eff.Recur` and `IO.Recur` implement trampolining via an explicit `while` loop — O(1) stack regardless of recursion depth. Non-async collection `Recur` uses the `List<A>` swap-buffer loop — also O(1) stack depth per generation.

[COMPOSITIONS_MONOID_TREE]:
- `Compositions<A>` is a skewed binary tree over elements that implement `Monoid<A>`. It supports `Skip(k)` in O(log k), `Take(k)` in O(k log k), `TakeComposed(k)` in O(log k), and `Composed()` (fold entire tree) in O(log n) — all via tree descent rather than enumeration. `SplitAt(i)` is a tuple alias for `(Take(i), Skip(i))` and pays the combined O(k log k) + O(log k) cost.
- The design pressure: when a pipeline must query prefix-sums or interval compositions many times, building a `Compositions<A>` once and calling `TakeComposed` per query costs O(n) build plus O(log k) per query, versus O(n) per query on a flat fold. The canonical domain is running-sum windows over a `Monoid<A>` without materializing intermediate aggregates.
- `Compositions.wellFormed<EqA, A>(Compositions<A>)` validates the right-associative balance invariant. Constructing via `Combine` and `Singleton` always produces a well-formed tree; direct `Tree` manipulation can violate the invariant and silently corrupt `Skip`/`Take` results.

[SEMIGROUPK_VS_CHOICE_SEMANTICS]:
- `SemigroupK<T>.Combine<A>(K<T, A>, K<T, A>)` concatenates two structures: for `Seq`, `Arr`, and `Lst` it appends elements; for `HashMap` it delegates to `operator+` (left-biased on key collision); for `HashSet` it unions. `Choice<T>.Choose<A>(K<T, A>, K<T, A>)` selects the first non-empty structure — it short-circuits on the first non-empty, never consuming both. `Combine` always uses both sides. The two operators express different fallback idioms over the same collection type and are not interchangeable at call sites that require merge vs. fallback semantics.
- `Choice<T>.Choose<A>(K<T, A>, Memo<T, A>)` is the lazy overload: the right-hand operand is wrapped in `Memo<T, A>` and evaluated only if the left is empty. The canonical pattern for "use this collection unless it is empty, otherwise compute an alternative" wraps the alternative in `memoK<T, A>(() => computeAlternative())` and uses `Choose`.

[MEMOIZED_APPLICATIVE_INNER_COLLECTION]:
- `memoK<F, A>(Func<K<F, A>> f)` constructs a `Memo<F, A>` that defers construction of `K<F, A>` until `.Value` is first accessed, then caches the result. The cache is guarded by a `SpinWait` CAS on an `int state` field (0=unloaded, 1=loading, 2=loaded), making construction thread-safe but not the `K<F, A>` usage boundary if the underlying effect is stateful.
- The idiom for avoiding repeated inner-collection traversal in multi-function applicative application is `Apply(functions, memoK<Seq, A>(innerSeq))`. `Seq.Apply(K<Seq, Func<A,B>>, Memo<Seq, A>)` reads `ma.Value.As()` once and iterates it per function. Without `Memo`, each `Apply` call on a `SeqLazy<A>` triggers re-enumeration from the source per function.
- `Memo<F, A>.Reset()` clears the cached `K<F, A>` and reverts to the `acquire` function, but only when `acquire` was provided at construction. Memos built via `memoK<F, A>(K<F, A> fa)` (pre-loaded) return `Unit.Default` immediately from `Reset()` — no re-acquisition path exists. `Clone()` always creates a fresh unloaded memo from the original `acquire` function if present, or a pre-loaded memo with the same value if not.
