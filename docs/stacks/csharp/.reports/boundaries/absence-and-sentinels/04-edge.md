# Absence and Sentinels — Edge Cases and Structural Traps

[OPTIONТ_NONE_IS_NULL]:
- `OptionT<M, A>` is a record — a reference type. Its `static readonly OptionT<M, A> None` field has no explicit initializer in IL and is zero-initialized to `null`. Any instance operation on that field — `.Match(...)`, `.Bind(...)`, `.Run()`, `.IfNone(...)` — throws `NullReferenceException` because `runOption` is null and every operation calls `M.Map(..., runOption)` or `M.Bind(runOption, ...)`. The static method `OptionT.None<M, A>()` is a separate factory documented as "Lift a `None` value into the monad-transformer" and constructs a valid instance; it does not delegate to the zero-initialized field.
- The only safe paths to construct a known-absent `OptionT<M, A>` are `OptionT.None<M, A>()`, `OptionT.lift<M, A>(Option<A>.None)` — which evaluates `new OptionT<M, A>(M.Pure(Option<A>.None))` — and the implicit `operator OptionT<M, A>(in Option<A> ma)`, which calls that same factory. Any seam that references the `None` field directly is a structural trap.

[OPTIONТ_VS_FINT_FALLIBLE_ASYMMETRY]:
- `OptionT<M>` implements `Fallible<Unit, OptionT<M>>`. Its `Catch` predicate receives `Unit`, not `Error`. The `@catch` infrastructure with typed error codes — `error.HasCode(...)`, `error.IsType<E>()` — does not apply to `OptionT`; absence in the transformer carries no diagnostic code. A seam that needs typed recovery (distinguishing timeout-absence from resource-unavailability-absence) must use `FinT<M>`, which implements `Fallible<Error, FinT<M>>` with full typed predicates.
- `Natural<FinT<M>, OptionT<M>>` exists; this transform is lossy: a `FinT<M, A>` in `Fail(error)` state becomes `OptionT<M, A>` with `None`, silently discarding the typed `Error`. The reverse `Natural<OptionT<M>, FinT<M>>` is not implemented; absence-to-failure conversion requires an explicit `.ToFin(Error.New(code, msg))` call at the materialization point.

[TRANSFORMER_BIND_OUTER_MONAD_LIFETIME]:
- `OptionT<M, A>.Bind<B>` propagates `None` through the outer monad via `M.Pure(Option<B>.None)` — the outer monad computation succeeds and the `None` state is carried as a value inside `M`. `FinT<M, A>.Bind<B>` propagates `Fail(error)` similarly through `M.Pure(Fin<B>.Fail(error))`. In both transformers, the outer monad `M` is never short-circuited by the inner absence or failure state.
- The extract point is `optionT.Run()` returning `K<M, Option<A>>`, or `finT.Run()` returning `K<M, Fin<A>>`. The caller must handle the outer monad failure and the inner absence or failure independently. A seam that calls `.Run().Match(...)` on the raw `K<M, Option<A>>` inside a `.Bind(...)` on `IO` without checking the outer `IO` result first silently swallows any outer-monad failure that occurred before `Run()`.

[FIN_OPERATOR_ERROR_SEMANTICS]:
- The `|` operator on `Fin<A>` calls `Choose(rhs)` — the lhs error is silently discarded when lhs is `Fail`. The `+` operator (semigroup combine) accumulates: when both are `Fail`, it returns `Fin.Fail(err1 + err2)` where `+` produces a `ManyErrors` containing both.
- A fallback chain expressed as `primary | secondary | terminal` discards all intermediate errors. Use `+` (semigroup combine) when a logging or health seam must inspect why each attempt failed; use `|` only when the lhs error is genuinely irrelevant to callers.

[MANYERRORS_RECURSIVE_DISCRIMINATION]:
- `ManyErrors.HasCode(int code)` recurses into contained errors, so a terminal code is detectable even inside a combined `ManyErrors` produced by `Fin` applicative or semigroup combination. A `Catch` predicate that tests `e.HasCode(terminalCode)` correctly detects the code even when combined with transient errors.
- `ManyErrors.Filter<E>()` returns a new `ManyErrors` (or `Errors.None`) containing only errors where `IsType<E>()` holds. This is the correct surface for partial recovery: extract only the terminal typed errors from a combined failure, emit them to the health seam, and separately attempt recovery on the remaining transient errors. Discarding the combined error and re-creating a custom receipt loses the accumulated evidence.

[ERROR_INNER_CHAIN_AT_SEAM]:
- `Error.Inner` is `Option<Error>`. `Expected` records declare an `Inner` primary-constructor parameter of type `Option<Error>`. `Exceptional` derives its `Inner` from the wrapped `Exception.InnerException` via `Optional(Value?.InnerException)`. A seam that captures a chained exception should preserve the inner chain by constructing `Error.New(code, message, inner: Error.New(innerException))` so that `Inner` carries the nested cause as a typed `Option<Error>`, not as a string field. Recovering callers traverse the chain via `error.Inner.Bind(i => i.Inner)` without parsing text.

[IO_LIFT_FIN_EXCEPTION_SURFACE]:
- `IO.lift<A>(Func<Fin<A>> f)` internally calls `f().ThrowIfFail()`. `Fin<A>.ThrowIfFail()` calls `FailValue.Throw()`, which dispatches to `Error.Throw<A>()`. On `Expected`, this throws `WrappedErrorExpectedException` carrying the code and message. On `Exceptional`, it re-throws the original exception if its `Value` field is non-null, or throws `ExceptionalException`. The round-trip is lossless for `Expected`. It is potentially lossy for `Exceptional` if the original `Exception` object has been collected (serialization-deserialization paths set `Value` to null). A seam that receives a `Fin<A>` from a foreign source and lifts it via `IO.lift(fin)` must verify that exceptional errors carry live `Exception` objects before the lift if stack trace preservation is a boundary requirement.

[ENVIO_CANCEL_PROPAGATION_ASYMMETRY]:
- `EnvIO.Local` creates a fresh `CancellationTokenSource` and, when `Token.CanBeCanceled`, registers `Token.Register(source.Cancel)`. The registration is stored in `Registration` and disposed when the `EnvIO` is disposed. Cancellation propagates from parent to child via this side-channel registration, which fires synchronously on `Cancel()` and asynchronously on `CancelAsync()`.
- `EnvIO.LocalResources` passes the same `Source` as the outer `EnvIO` — no new source is created. `Cancel()` on the parent source fires immediately inside the child scope. Disposing the child `EnvIO.DisposeNonResources()` does not dispose the shared source because the ownership bit is not set. `Local` owns its source and disposes it on `DisposeNonResources()`. A timed-out `BracketFail` probe scoped via `LocalCancelWithTimeout` creates a new source that IS owned and disposed by the child.

[CANCELLATION_REGISTRATION_ALREADY_CANCELLED_REENTRANCY]:
- `CancellationToken.UnsafeRegister(Action<object?> callback, object? state)` delegates to `_source?.Register(callback, state, useSynchronizationContext: false, useExecutionContext: false)`. When `IsCancellationRequested == true` at registration time, `Register` invokes the callback synchronously on the calling thread before returning. This is a reentrancy hazard at seams that perform `Atom.SwapMaybe` inside the callback: if the swap body calls back into code that itself calls `Register` on the same already-cancelled token, the inline synchronous invocation re-enters the swap body on the same thread. Guard by checking `IsCancellationRequested` before calling `Register` or `UnsafeRegister` and handling the already-cancelled state directly, bypassing registration.

[OPTION_TRAVERSE_EFFECT_ORDERING_AT_SEAM]:
- `Option<A>.TraverseM<M, B>(Func<A, K<M, B>> f)` when `None` returns `M.Pure(Option<B>.None)` without calling `f`. `OptionT<M, A>.IfSome(Action<A> f)` runs the action inside `M` only when `Some` — identical silent-skip behavior. A side-effecting audit at a boundary that must fire unconditionally cannot use either form; it must be sequenced as a separate operation outside the traversal or conditional.
- `OptionT<M, A>.IfSome` combined with `IfNone` on separate chains creates two `M.Map` calls and two effect-bearing paths that may not compose correctly under `IO`'s resource tracking. Use `Match(Some: x => effect(x), None: () => alternateEffect())` to run exactly one effect path.

[OPTION_BIBIND_NULL_INTERIOR_TRAP]:
- `Option<A>.BiBind<B>(Func<A, Option<B>> Some, Func<Option<B>> None)` does not call `Check.NullReturn`. If the `Some` branch returns `Some(null)` for a reference-type `B`, the result is `Option<B>` with `isSome = true` and a null interior value — a `Some`-state option that defeats the boundary guarantee. A `null` return from the `None` branch produces `default(Option<B>)` — silently `None` with no diagnostic. The law: neither branch may return `null` from the factory. `Some(null)` is structurally `Some` but semantically corrupted, and a null `Option<B>` return is structurally `None` with no observable failure.

[FIN_BIMAP_ASYMMETRY_WITH_OPTION]:
- `Fin<A>.BiMap<B>(Func<A, B> Succ, Func<Error, Error> Fail)` maps to `Fin<B>` — the `Fail` branch transforms the error but stays in the `Fail` state. `Option<A>.BiMap<B>(Func<A, B> Some, Func<B> None)` maps to `B`, escaping the `Option` rail. `Fin.BiMap` cannot escape to `B` directly; it is an in-rail transform for re-coding an `Exceptional` into an `Expected` at an outbound seam. A seam that needs to collapse a `Fin<A>` into a plain `B` with distinct success and failure materializations must use `Match(Succ: ..., Fail: ...)`, not `BiMap`.

[VALIDATION_TO_FIN_NARROWING]:
- `Validation<Error, A>.ToFin()` is an extension method constrained to `Validation<Error, A>`. A seam that uses a custom `Validation<TError, A>` where `TError` is not `Error` cannot use this extension; it must manually match `Success` and `Fail` cases. The canonical projection: match the `Fail` case through `MyError.ToError()` returning `Error.New(myError.Code, myError.Message)`, then call `ToFin()` on the converted `Validation<Error, A>`. Placing this projection inline at the seam rather than inside the validation type keeps the canonical `Error` rail clean.

[MONADIO_SYNC_CONTEXT_AS_OPTIONAL]:
- `MonadIO.syncContext<M>()` is a static trait method returning `K<M, Option<SynchronizationContext>>`; when called as `MonadIO.syncContext<IO>()` it returns `IO<Option<SynchronizationContext>>`. The `SyncContext` field on `EnvIO` is nullable — `Optional(e.SyncContext)` lifts it to `Option`. The absence of a synchronization context is therefore a first-class optional: `Some(ctx)` in UI-thread environments and `None` in background-thread or console environments. A seam that conditionally marshals based on context presence must use `MonadIO.syncContext<IO>().Bind(ctx => ctx.Match(Some: c => marshalOn(c), None: () => runDirect()))` rather than reading `SynchronizationContext.Current` directly — the direct read bypasses the `EnvIO` boundary and is untestable.

[SWAPMAYBE_SUCCESS_REJECTION_OBSERVABILITY]:
- `Atom<A>.SwapMaybe(Func<A, Option<A>> f)` returns type `A` — the raw current value after the operation. When the swap succeeded, it is the new value; when the guard rejected, it is the unchanged prior value. These two cases are indistinguishable from the return value alone; there is no discriminated result type. The pattern for observable take/release encodes observability in the state structure: for `Atom<Option<Token>>`, the take function returns `None` when the cell already holds `Some(token)` (rejecting), or `Some(newToken)` when empty. After `SwapMaybe`, reading `atom.Value` and comparing against the newly constructed token indicates success; matching the prior token held by the caller indicates rejection. The caller must hold a reference to the prior value before the swap — there is no post-hoc way to determine which branch executed from the return value alone.
