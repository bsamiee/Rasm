# Awaitable Signal And Rendezvous Seam

[WAITHANDLE_TO_TASK_BRIDGE]:
- A boundary cell whose waiters must suspend asynchronously on a kernel event cannot block a pool thread on `WaitHandle.WaitOne`; the bridge registers the handle with the thread pool's single dedicated wait thread and completes a promise from the wait-or-timeout callback. One wait thread multiplexes thousands of registered handles, so the suspended waiter consumes no pool thread until the handle fires — the structural answer to "park many waiters on one boundary transition cheaply."
- The bridge promise is a bare completion source constructed with no asynchronous-continuation option, so its result transition runs the awaiting continuation inline on the thread that completes it: the pool wait thread on a handle fire, or the cancellation-callback thread on token cancel. A waiter that resumes into expensive boundary work therefore stalls the shared wait thread or the canceling thread for the duration of its synchronous continuation prefix. The fix is to construct the completion source requesting asynchronous continuations so the resumed body is queued rather than inlined, or to make the first resumed segment trivial and immediately yield.
- The registration captures execution context by default; the unsafe registration variant skips the capture. For a high-frequency boundary cell whose waiters carry no ambient security context, the context-flowing default is pure per-wait allocation overhead, and the unsafe variant is the correct seam.
- The callback registers with execute-only-once semantics and is torn down in the `finally` via the registered-wait handle's unregister plus disposal of the token registration. A waiter abandoned without reaching the `finally` — its enclosing scope faulting before the await — leaks both the registered wait slot on the shared wait thread and the token registration, and the leaked wait slot keeps the kernel handle pinned against finalization.
- A timeout argument of negative one means infinite; the callback's timed-out flag is inverted into the promise result, so the awaited boolean is true on a real signal and false on timeout. A boundary cell that must distinguish "settled" from "deadline elapsed" reads this boolean directly rather than racing a separate timeout source, collapsing the timeout into the same await.

[EVENT_HANDLE_CELL_ALGEBRA]:
- The signal cell wraps a kernel event handle and is registered into the effect runtime's resource scope at construction, so it auto-releases on scope teardown without an explicit close — the handle lifetime is the scope lifetime unless transferred. Three cell shapes share one surface: an auto-reset shape releases exactly one waiter per trigger and re-arms, a manual-reset shape releases every current and future waiter until explicitly reset, and a countdown shape releases all waiters when its decrementing counter reaches zero.
- The auto-reset shape is the rendezvous baton for a single-consumer boundary drain: each trigger hands off to exactly one parked waiter and immediately re-closes, so N triggers wake N waiters in unspecified order with no lost or doubled wake. The manual-reset shape is the latch for a one-shot boundary transition — initialization-complete, drain-finished — where every late waiter must also observe the settled state without re-arming; converting one to the other silently changes whether a second waiter is admitted.
- The countdown shape is the structured fan-in: a parent forks N boundary operations, each triggers one decrement on completion, and the parent's wait releases when the last completes — the lock-free join that a plain counter plus a manual-reset latch would have to assemble by hand. Its trigger returns whether the counter reached zero, so the final completer is identifiable at the trigger site for last-one-out teardown.
- The trigger operation is offered both as an effect-lifted form and an impure escape hatch flagged unsafe precisely because it sets the handle outside the effect tree; a cell that triggers from a host callback already off the effect spine uses the unsafe form deliberately and owns the ordering itself.
- Over-triggering a settled countdown is not a benign no-op: a decrement past zero throws rather than saturating, so a boundary cell whose completion count can race above its initial budget must gate triggers or pre-size the counter to the true maximum. The increment side likewise throws if the counter has already reached zero, so dynamic fan-in that grows the join after some children have already completed must add before the count can settle, never after.
- The countdown shape's multi-decrement trigger silently ignores its count argument and decrements by one: the effect-lifted and unsafe count-taking forms both forward to the single-step decrement, so a caller expecting a batch decrement of N undercounts by N-minus-one and the join never settles. Batch fan-in completion must loop single decrements or compute the join arithmetic externally, never trust the count-taking trigger to apply its argument.
- The auto- and manual-reset cell constructors ignore the initial-signaled argument and always construct the underlying handle unsignaled, so a cell intended to start latched — a manual-reset latch pre-set so the first waiter passes immediately — instead starts closed and parks that waiter forever. A pre-signaled start must be expressed by triggering the cell as the first effect after construction, not by the constructor argument.
- The countdown shape's wait is a synchronous blocking wait on the calling thread guarded only by the cancellation token, not the async pool-wait bridge the auto- and manual-reset shapes use. A countdown join therefore pins the awaiting thread for the whole fan-in rather than parking it cheaply; for a many-waiter boundary join the auto-reset-plus-async-bridge composition scales where the countdown wait does not.

[RACE_FANOUT_CANCELLATION]:
- The race combinator over a sequence of effects builds a fresh child cancellation scope, launches every branch under that child, awaits the first to finish, then cancels the child to tear down the losers — a structured "first wins, rest die" with no orphaned branches. The first-completion is read off a whenany over the per-branch tasks, and the loser cancellation is the asynchronous cancel form so the winner's thread is not blocked on loser callback execution.
- The race over already-forked operations carries a hazard the race over raw effects does not: the forked branches were launched under the parent scope before the race began, so the child scope the race creates and cancels does not own them — cancelling the child cancels nothing, and the losing forks run to completion in the background. Racing forks gives first-result latency but not loser termination; racing raw effects gives both. A boundary cell that must guarantee losers stop uses the raw-effect race, never the fork race.
- The any-completes combinator returns on the first success and otherwise accumulates branch errors, so a boundary cell racing redundant providers surfaces a single success immediately and a joined error only when every branch fails — the correct shape for hedged boundary requests where one good answer ends the race and total failure is the only error worth reporting.
- The all-completes combinator over forks awaits each fork's await-effect under the original parent scope rather than a child, so it inherits parent cancellation directly and produces results in branch order regardless of completion order — a deterministic fan-in whose ordering is positional, not temporal. Pairing it with a per-branch timeout on the fork itself bounds each branch independently while keeping the join's positional result shape.

[FORK_LIFETIME_OWNERSHIP]:
- A forked boundary operation runs on a dedicated long-running thread, not a pooled one, so a fork is the right primitive for a blocking native boundary call that would starve the pool, and the wrong one for short CPU work where the dedicated-thread spin-up dominates. The fork's environment is constructed owning neither its cancellation source nor its resource scope; the fork body's own `finally` is the sole owner of both teardowns, so the cancel effect cancels the source but never runs the body's resource cleanup — cleanup is bound to body completion alone.
- The fork wires a parent-fires-child-cancel registration at launch and disposes it in the body `finally` alongside the source; awaiting a fork after the parent token has already fired throws the cancellation directly rather than surfacing the body's last value, while awaiting after the body faulted surfaces the body's error through the rail. A boundary cell that forks a long native call and later wants its partial result on cancel cannot get it from the await — the cancel path is terminal, and partial state must be published into a separate cell before the cancellable region.
- The fork's await is itself a re-entrant effect that registers the awaiting environment's token against the fork source, so a second consumer awaiting the same fork from a different cancellation scope can cancel the fork by cancelling its own scope — the await is not a passive read. Sharing one fork handle across consumers with independent cancellation lifetimes lets any consumer kill the shared computation; isolate by handing each consumer its own fork or by awaiting once into a published cell the consumers then read.

[CONTINUATION_INLINING_DISCIPLINE]:
- Every promise-backed boundary signal that completes from a foreign thread — a host event, a native callback, a timer fire — inlines its awaiting continuation onto that foreign thread unless the promise was explicitly constructed to queue continuations. The default is inline. A boundary seam that completes a waiter from a host UI thread or a single shared callback thread therefore runs arbitrary downstream boundary logic on that thread, and the discipline is to construct every such signal source requesting asynchronous continuations so the resumed waiter cannot hijack the completer's thread.
- The completion-source option is the whole defense, and it is a constructor argument, not a per-await choice; a waiter cannot opt out of an inlining completer after the fact.

```csharp
// inlining: the firing thread runs the awaiter's continuation prefix
var inlineGate = new TaskCompletionSource<TState>();
// queued: the firing thread only schedules the awaiter, never runs it
var safeGate   = new TaskCompletionSource<TState>(TaskCreationOptions.RunContinuationsAsynchronously);
fromHostThread.Register(static s => ((TaskCompletionSource<TState>)s!).TrySetResult(settled), safeGate);
```

- The channel-based drain seam takes the opposite default deliberately: each pending read or write operation carries its own continuation-asynchrony flag, set at channel construction, so a channel built for asynchronous continuations never inlines a reader's resumption onto the writer's thread. For a boundary drain where producer and consumer must not share a stack, the channel seam is structurally safer than a hand-rolled event-handle signal whose promise defaults to inline.
- A cell that marshals a completed value onto a captured host context posts the body to that context and parks on a kernel event the post sets, awaiting it through the same pool-wait bridge — so the marshal cost is one kernel handle and one bridge registration per post with no pooling. High-frequency marshaling of per-item results through this seam allocates a handle per item; batch the items through one channel whose consumer runs on the host context and post once per batch, paying the handle cost once.

[SETTLED_TRANSITION_PUBLICATION]:
- The await-coordinated initialization gate pairs a state cell with a completion source allocated before any concurrent caller can reach the gate: the winning initializer commits the state transition then completes the source, and a late caller reads the cell first, returning immediately on a settled state and parking on the source only when it observes the in-progress state. The source must be process-statically present before contention, never lazily minted on first race, or two callers allocate two sources and one waiter parks on a source the winner never completes.
- Faulting outstanding waiters on a boundary replace must distinguish retriable from terminal: completing each parked source with a typed fault rather than a cancellation preserves the caller's ability to discriminate, and each source is removed from its tracking map before it is faulted so a concurrent drain-plus-replace cannot double-complete it — a second completion of an already-settled source is rejected by the try-form but a non-try completion throws. Sweeping the tracking map under the boundary's drain phase, after in-flight operations referencing the old waiters complete and before the new instance accepts work, is the only window where the sweep does not race fresh registrations.
- The signal that wakes waiters on a settled transition fires past the publishing barrier of the transition itself, so a waiter resumed by the signal observes the fully-published settled value — the wake and the value are ordered by the same barrier that committed the transition. This is why the wake belongs on the single-cell settled-transition seam and not on a multi-attempt coordination event that can fire on aborted attempts: a waiter must never be released against a value that did not commit.
