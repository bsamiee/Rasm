# [PYTHON_REPORTS_AGENTS]

Scope: `docs/stacks/python/.reports/**`. Parent Python stack instructions remain in force for trusted owner routes; this folder stores source material only and is excluded from the Python stack decision atlas unless a task explicitly names it.

## [1]-[SCOPE]

Use this folder for reusable Python stack research sessions: current source scans, maintained-doc research, package-capability findings, critique passes, synthesis, promotion maps, and candidate wording for planned Python concept pages.

This folder does not own Python doctrine, task status, planned topology, package graph truth, command behavior, proof vocabulary, or validation gates. Promote durable guidance to the matching `.planning/SPEC.<slug>.md` first, then to the concept page when the roadmap task creates that owner.

## [2]-[READ_BEHAVIOR]

When a task names a Python report session, read the session manifest first, then synthesis, correction, promotion, or pruning records before individual reports. If no manifest exists, inventory report headings first and treat the missing manifest as an archive-maintenance gap.

Before writing another report in an existing lane, read prior passes in that lane and state the prior finding the new report extends, corrects, merges, drops, holds, or routes away.

When ordinary Python stack work does not name this folder, treat `.reports/` as excluded source material. Do not include it in README chooser counts, architecture codemaps, roadmap progress, concept-page doctrine, or validation claims.

## [3]-[SESSION_SHAPE]

Name session folders as `<topic-slug>-<ddmmyy>`, where the topic is the durable concept-page subject such as `data-shapes`, `runtime`, `testing-proof`, or `platform-packages`.

Every multi-report session folder has `README.md` as the session manifest. The manifest is the archive entrypoint and routing surface, not a ledger, narrative summary, or active standard.

Required manifest sections:
- `[SESSION_CARD]`: target spec sheet, status, source set, read order, and boundary.
- `[TRACK_CARDS]`: one card per research lane with direction, facets, use, extension rule, and route.
- `[NEXT_PASS_RULE]`: one paragraph that tells a later agent how to deepen, correct, fold, or promote without repeating the same pass.
- `[PROMOTION_LOG]`: durable findings promoted to the spec sheet or concept owner.
- `[CORRECTIONS]`: only when a report has a corrected, superseded, or retracted claim.

## [4]-[REPORT_SHAPE]

Each report starts with this compact record:

```text
Question: <one concrete question>
Type: <source-scan | repo-scan | gap-critique | synthesis | wording-packet | promotion-map | correction>
Lane: <session lane>
Merge key: <spec-or-concept :: claim-or-rule :: action>
Target owner: <planned spec or concept page>
Source basis: <repo proof, maintained docs, prior report paths, or tool output>
Promotion target: <SPEC.<slug>.md or concept page>
Outcome: <PROMOTE|CORRECT|MERGE|DROP|HOLD>
```

Use `[FINDINGS]`, `[EVIDENCE]`, `[RECOMMENDATIONS]`, `[CANDIDATE_WORDING]`, `[PROOF_GAPS]`, and `[CORRECTIONS]` only when they add source material. Do not add transcript, confidence, validation, worker-role, wave, or task-history sections.

## [5]-[PROMOTION]

The final output of a roadmap research task is one focused `.planning/SPEC.<slug>.md`. The spec sheet carries the implementation-ready synthesis; reports remain evidence and candidate wording.

Promote only durable rules, proof gaps, corrected routes, adoption gaps, and rejection replacements. Strip report frames, fixed lane labels, source inventories, prompt text, and sub-agent transcript order before writing the spec sheet or concept page.

## [6]-[REJECTIONS]

- No active Python doctrine, package catalogs, command ladders, validation ledgers, provider manuals, or roadmap status in reports.
- No report paths in roadmap task bodies when a spec sheet can carry the handoff.
- No copied report headings, worker roles, wave names, confidence labels, prompt wording, task narration, local machine paths, or session chatter in active owners.
- No reports that only confirm a file was read, a task was performed, or validation was unnecessary.
