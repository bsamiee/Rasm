# [RASM_API_BCL_INTEROP]

`System.Runtime.InteropServices` BCL inbox — no package pin, shared-framework surface (`System.Runtime.InteropServices.dll`, net10.0). The substrate signal-lifecycle hub: the portable POSIX-signal registration the `Rasm.AppHost` `Runtime/lifecycle#FAULT_SPINE` drain traps and the `Runtime/config#SOURCE_AXIS` SIGHUP reload seam bind. Unix-only (Linux + macOS); a `win-*` RID exposes no signal and routes reload through `ControlTrigger` exclusively.

## [01]-[POSIX_SIGNAL]

`PosixSignalRegistration` is the sealed `IDisposable` registration owner; `PosixSignal` is the portable signal vocabulary; `PosixSignalContext` is the per-delivery handler payload. The registration is the ONLY managed signal seam — a raw `sigaction` P/Invoke, a `Console.CancelKeyPress` handler, or a second registration path are the deleted forms.

- `PosixSignalRegistration.Create(PosixSignal signal, Action<PosixSignalContext> handler) -> PosixSignalRegistration` registers one handler; the returned registration is `IDisposable` and detaches the handler on `Dispose()` (the LIFO detacher the `PhaseSubscription` composite holds). A handler runs on a dedicated thread-pool thread, not the OS signal context, so it may block and allocate. Multiple registrations for one signal all fire; a `SIGTERM`/`SIGINT`/`SIGQUIT` handler that leaves `Cancel` false lets the runtime terminate after the handlers run.
- `PosixSignal` (enum) — ten portable members carrying ABSTRACT values (`SIGTSTP = -10`, `SIGTTOU`, `SIGTTIN`, `SIGWINCH`, `SIGCONT`, `SIGCHLD`, `SIGTERM`, `SIGQUIT`, `SIGINT`, `SIGHUP`), NOT raw OS signal numbers; the runtime maps each to the platform's real signal, so a handler never encodes a numeric constant. The registrable set is exactly these ten — `SIGKILL`/`SIGSTOP` are unregistrable at the OS level and absent by construction.
- `PosixSignalContext` (sealed class) — `PosixSignal Signal { get; }` names the delivered signal (one handler may serve several registrations and discriminate on it); `bool Cancel { get; set; }` suppresses the runtime's default action when set `true` — the drain-signal path sets it before projecting the drain transition, and the SIGHUP reload sets it before enqueuing the reload outcome, so neither terminates the process. Constructed by the runtime (`PosixSignalContext(PosixSignal signal)`); consumers read `Signal` and write `Cancel`, never new it.

## [02]-[STACK]

`FaultSpine.ArmTraps` registers `SIGTERM`/`SIGQUIT` to the drain transition (each handler sets `Cancel = true` then projects `Lifecycle.Transition(RuntimePhase.Draining)`) and `SIGHUP` to the reload delegate (sets `Cancel = true` then enqueues the `ReloadOutcome` under `ReloadReceipt.SignalTrigger`); a `SIGTERM`/`SIGQUIT` fault also folds into the `FaultSource.Signalled(PosixSignal)` case, whose `FaultRecord.From` flatten projects the signal name onto the wire-stable `posix-signal` kind. Every `PosixSignalRegistration` returned by `Create` is a `Dispose` detacher collected into the one `PhaseSubscription` LIFO composite, so the trap set detaches in reverse-registration order at drain. The seam is Unix-only: `Runtime/config#SIGHUP_RELOAD` registers `SIGHUP` only where the RID resolves to Linux/macOS, and a Windows host carries no `SignalTrigger` band — reload arrives solely through the `ControlTrigger` service verb. The systemd `WATCHDOG_USEC` cadence and `sd_notify` ride `Microsoft.Extensions.Hosting.Systemd`, not this surface.
