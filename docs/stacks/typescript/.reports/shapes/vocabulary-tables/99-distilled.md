# Vocabulary Tables

[BEHAVIOR_COLUMN_OWNER]:
- The behavior-column table is one `as const satisfies Record<Key, Row>` literal: rows are keys, columns are per-key data, and the binding is the decode-free, dependency-free vocabulary owner — no schema, no codec, no library on the read path.
- Operator order is the closure: `as const` freezes every leaf to its literal and marks the structure `readonly` recursively, then `satisfies` checks each row against the contract WITHOUT widening the binding to it, so a forgotten column is a declaration-site error and the literal key/leaf unions still flow to consumers.
- The contract is a `Record<string, Row>` UPPER BOUND, never the table's own type: it asserts each row satisfies `Row` while inference discovers the exact key set and exact column literals, so the `satisfies` is the row-totality proof and the literal union is what reaches the reads.
- Two operators are the whole read surface: `keyof typeof Table` names the closed key union, indexed access `Table[k]` returns the row, a data column is `Table[k].rank`, and the column's value union is `(typeof Table)[keyof typeof Table]['rank']` — every projection derives from the one literal, none is restated.
- A function column makes the same owner a dispatch surface read `Table[k](arg)`, but a function column forfeits structural `Equal`/`Hash` and `JSON` projection — a data-projection table and a dispatch table over one vocabulary are two reads of the same key set selected by intent, never one mixed table read for both.
- The codec-backed sibling carries what the structural table cannot: `Schema.Literal(...values)` fuses the union `values[number]`, the decoding codec, and `.literals` (the ordered runtime tuple) in one owner over the admitted alphabet `string | number | boolean | null | bigint`; reach for it the moment a boundary supplies the alphabet, an annotation message must speak the rejection, or a `jsonSchema` projection is needed — the metadata column the `as const` table forfeits rides the codec declaration.

```typescript
import { pipe, Record } from 'effect'

const SEVERITY = {
    trace: { rank: 0, halts: false, run: (n: number) => n },
    warn: { rank: 1, halts: false, run: (n: number) => n + 1 },
    fatal: { rank: 2, halts: true, run: (n: number) => n * 10 }, // a row lands here; keyof typeof + every projection widen, a missing column fails the satisfies at this site
} as const satisfies Record<string, { readonly rank: number; readonly halts: boolean; readonly run: (n: number) => number }>

const halts = (k: keyof typeof SEVERITY): boolean => SEVERITY[k].halts // key union inline; rank union is (typeof SEVERITY)[keyof typeof SEVERITY]['rank'], never aliased
const ranked: Record.ReadonlyRecord<keyof typeof SEVERITY, number> = pipe(SEVERITY, Record.map((r) => r.rank)) // data read
const escalate = (k: keyof typeof SEVERITY, n: number): number => SEVERITY[k].run(n) // SAME owner read for dispatch; the function column splits no second table
```

[WIDENING_IS_TWO_INDEPENDENT_AXES]:
- The table carries two literal axes a closure depends on — the KEY axis (`keyof typeof`, the closed key union) and the LEAF axis (each column's literal union) — and a site widens them INDEPENDENTLY, so a table that still passes a `keyof typeof` audit can have already lost every column's totality.
- The defeat matrix is fixed by lever: `as const satisfies Record<K, Row>` preserves BOTH; a bare annotation `const T: Record<K, Row> = {...}` widens BOTH (key `string`, leaf primitive); a bare `satisfies` with no `as const` preserves the key but WIDENS every leaf to its primitive; `as const` alone preserves both but proves no row contract.
- The language rule under it: `satisfies` is a CHECK, not a contextual type — it verifies the inferred type assigns to the contract then discards the contract, so the binding keeps the literal type `as const` froze; object-literal KEYS never widen under inference (a key is its own literal) while VALUES widen to their primitive unless `as const` freezes them, which is why `satisfies`-without-`as-const` splits the leaf from the key.
- The defeat does not require a wrong annotation: a clean `as const satisfies` table flowed into a CONTEXTUAL position typed `Record<K, Row>` — a wide function parameter, a wide return, a wide property sink — is re-widened at that boundary because the contextual type stamps over the literal, the freeze at the value not surviving a wide sink.
- The generic-constraint partial defeat is the middle case: `<R extends Record<string, { rank: number }>>(r: R)` infers `R` so the key union SURVIVES, but the leaf widens to `number` because the constraint contextually types each leaf — and the fix is the `const` type parameter, the call-site as-const equivalent: `<const R extends ...>(r: R)` infers with deep literal preservation, so a vocabulary consumed by many sites freezes both axes at the entrypoint once rather than forcing every caller to re-`as const`.
- The leaf freeze is per-column, not all-or-nothing: `rank: 0 as number` widens ONLY that column under the same `as const`, the deliberate escape for a genuinely-open data column (a tunable threshold) sharing one owner with frozen discriminant columns; the accidental form is identical syntax — a column fed a widened source (a non-`const` variable, an arithmetic result) silently widens because `as const` freezes the shape it is given and cannot re-narrow a value that entered widened.
- `Schema.Literal` has no leaf axis to defeat: its alphabet is the rest-tuple `NonEmptyReadonlyArray<AST.LiteralValue>` inferred narrowly from the call arguments — a lower-bound constraint that ADMITS only literals rather than an upper bound that overwrites inference — so the codec owner survives contextual sinks the structural table cannot, and `transformLiterals`/`pickLiteral` carry the same `const` type parameter as built-in defeat-immunity.

```typescript
const WIDE = {
    alpha: { rank: 0, halts: false },
} satisfies Record<string, { readonly rank: number; readonly halts: boolean }>
// REJECT: satisfies without as const — keyof typeof WIDE stays 'alpha' but WIDE['alpha']['rank'] WIDENED to number; the column-as-discriminant is defeated while a keyof audit passes

declare function preserves<const Row extends Record<string, { rank: number }>>(rows: Row): Row
const kept = preserves({ alpha: { rank: 0 }, beta: { rank: 1 } }) // const type param freezes the leaf to 0 at the call; both axes survive with no as-const at the value
```

[KEY_UNION_IS_A_PROOF_NO_RECONSTRUCTION_REBUILDS]:
- The closed key union `keyof typeof Table` is a PROOF OBJECT: the proposition "every value of this key type names a present row" is true by construction and inhabited by the literal union, so a parameter typed `keyof typeof Table` already holds the membership proof and the indexed access `Table[k]` is the proof's CONSUMER — it yields `Row`, never `Row | undefined`, because the index is provably a subset of the table's own key set.
- The discharge survives the strictest posture: under no-unchecked-indexed-access `Obj[s: string]` is `Row | undefined`, but `Table[k: keyof typeof Table]` stays `Row` — the flag forces `undefined` only on indices the type cannot prove present, so totality is a property of the key's PROVENANCE, never the table; a `k: string` reads `Row | undefined` or LIES as `Row` without the flag, and the repair carries absence as `Option` through `Record.get(Table, key)`, whose key parameter `NoInfer<keyof typeof Table>` rejects a statically-literal foreign key at compile time so only a genuinely runtime-decided key reaches the rail.
- One proof discharges TWO obligations at once — case-exhaustion and pre-access membership — so a `keyof typeof`-keyed function dispatch needs neither a `default`/`orElse`/`?? throw` arm nor a membership probe; a fallback arm on a vocabulary dispatch is itself the diagnostic that the key proof was lost upstream, the dispatch degraded from total-by-type to partial-by-runtime.
- The proof cannot be rebuilt from a runtime structure — the language has no dependent path from an array's element VALUES back to a key UNION — so `Record.fromEntries(entries): Record<NonLiteralKey<Entry[0]>, Entry[1]>` widens as the only SOUND type, the erasure a structural impossibility that is silent and invisible to a value-equality audit (`fromEntries(toEntries(Table))` is a runtime identity yet a type widening).
- `NonLiteralKey<K> = K extends string ? IsFiniteString<K> extends true ? string : K : symbol` is the proof-erasure operator: a finite literal key carries a closed proof and is therefore the very thing widened to `string`, a symbol collapses to `symbol`, and `IsFiniteString` walks the key character-by-character returning `false` the instant a segment is `string` or `` `${number}` `` — so a template-pattern key `` `v${number}` `` is proof-LESS at construction (`keyof typeof` is `string`), an infinite alphabet carrying no closed-membership proof to begin with.
- The carrier-flow law subsumes the per-combinator split: a combinator KEEPS the proof iff `K` reaches only the CALLBACK position and the result key set is a union the checker can name, ERASES it iff `K` flows back into an OUTPUT KEY position whose membership reconstruction cannot prove — `Record.map` (value-position, `NoInfer<K>` callback), `Record.collect`/`Record.reduce` (callback `K`, keyless result), `Record.keys`/`Struct.keys` (`Array<K & string>`), `Record.singleton`, `Record.union`/`Record.difference` (`K0 | K1`, a nameable union) PRESERVE; `Record.fromEntries`, `Record.filterMap`, `Record.filter`, `Record.getSomes`, `Record.partition`, `Record.empty`, `Record.mapKeys`/`Record.mapEntries` whose key function widens (`key.toUpperCase()`) ERASE.
- `Record.intersection` is the one GUARDED erasure: `IntersectKeys<K1, K2> = [string] extends [K1 | K2] ? NonLiteralKey<K1> & NonLiteralKey<K2> : K1 & K2` PRESERVES `K0 & K1` for a clean two-table intersect and erases only once a side has leaked `string` — union/difference keep unconditionally (computable result set), overlap keeps only when neither input was already open, surfacing a prior erasure as a now-erased output.
- The audit is two distinct projections: `keyof typeof Result` proves the KEY axis, `(typeof Table)[keyof typeof Table]['rank']` proves the LEAF axis — a primitive there means the column is no longer a discriminant even when the key set is intact, so drop columns inside the VALUE position with `Record.map`, never key-position through `filterMap`.
- The reverse index invokes the erasure ON PURPOSE: `Record.fromEntries(Record.toEntries(Table).map(([k, row]) => [row.id, k]))` widens the new key axis to `string` because the new keys ARE the id column while the VALUES retain `K` — correct exactly when the rebuilt keys are foreign and the proof has migrated to the value position; a duplicate id collapses two keys to one entry at construction, and `Record.get(reverseIndex, id): Option<K>` reports absence as `None`.

[KEY_ERASURE_PASSES_SILENTLY_AT_THE_INDEXED_READ]:
- A key-erasing transform endangers the indexed read MORE than a keyed dispatch: `Table[k]` over an erased `k: string` types `Row` (no strict flag) or `Row | undefined` (with it) but never refuses to compile, so the read lies about totality, whereas a `Match.exhaustive` keyed on the same erased union at least fails loud — over an intact key union its residual subtracts to `never`, over `string` the residual stays `string ≠ never`.
- The erasure does not corrupt the proof, it makes the totality claim unprovable downstream where it compiled upstream: the indexed read carries no residual obligation to break, so the erased key flows into the read and the totality guarantee is gone with no diagnostic — audit the read site's key type (`keyof typeof Table` versus `string`), never the read's compilation.

[TABLE_VS_MATCH_COLLAPSE_DIRECTION]:
- The collapse direction is fixed by the per-case payload, not taste: uniform-return data-keyed arms reading only the dispatched value's own fields → a behavior-column table read by indexed access (the literal key type IS the totality proof, no exhaustiveness combinator); divergent-return or refinement-accumulating arms → `Match` over the union — never a table of mixed inert and computed columns read for both, and never a fatter table once two-plus rows need shape-dependent branching.
- The `Match` handler-row carries the guarantee the bare table cannot: its parameter type is `Extract<R, Record<D, Tag>>`, the union member narrowed to that tag, so each arm receives the precisely-narrowed case and a wrong field access fails — `Match.tags`/`Match.tagsExhaustive`/`Match.discriminators(field)`/`Match.discriminatorsExhaustive(field)` consume the SAME handler-row object literal the dispatch table is, so the two are one object shape and the choice is whether the LIBRARY proves exhaustiveness (`{ [Tag in Exclude<keyof P, Tags>]: never }` rejects a missing tag) or the literal key union does (asserts only key-presence).
- The input type must arrive before the handler row infers: `Match.valueTags(handlers)` without binding the union resolves each handler parameter to `never` and errors `(_) => _ is not assignable to never` — the curried `Match.typeTags<U>()(handlers)` binds the union first and returns a REUSABLE matcher (the dispatcher-as-value mirroring the function-column table read everywhere), where the eager `Match.value(u)` re-builds per call; the `Pr` (provided) parameter is the discriminant, `[Pr] extends [never]` selecting the function-returning overload.
- The boundary is structural, not stylistic: the uniform fold combines arm CONTENTS through `SemiApplicative.getSemigroup(F)(S)` which keeps the type `F<A>`, while divergent arms returning `X` and `Y` form a coproduct whose only combine is `SemiCoproduct.getSemigroup(F)()` — that takes NO content semigroup and SELECTS a left-biased branch widening to `F<X | Y>`, never aggregating — so a coproduct "fold" is first-present selection (identity `Coproduct.zero`, the always-losing alternative), and `Match.exhaustive` must ELIMINATE the union to a single carrier per branch before the now-uniform results fold; foldability and uniform-return coincide.

[COLUMN_IS_THE_PULLBACK_ARGUMENT]:
- A behavior column projects to a typeclass INSTANCE keyed by the vocabulary, not a hand-written comparator: `mapInput` is implemented `make((b1, b2) => self(f(b1), f(b2)))` — the projection is pre-composed onto BOTH arguments of the base instance — so `Order.mapInput(Order.number, (k) => Table[k].rank)` does not write a comparator, it pulls `number`-comparison back along the rank read, and the rank column IS the total order; this is the SAME construction the library applies to its own primitives (`Order.Date = mapInput(Order.number, (d) => d.getTime())`), reached for in place of `Order.make`/`Equivalence.make`.
- One contravariant lever `(b) => a` builds both sides from the same map: `Order.mapInput<A,B>(self: Order<A>, f: (b: B) => A): Order<B>` and `Equivalence.mapInput<A,B>(self, f): Equivalence<B>`, so the rank read builds the key-level `Order` and the group read builds the key-level `Equivalence` — and `mapInput` composes by function composition with zero instance algebra (`mapInput(mapInput(O, g), f) = mapInput(O, x => g(f(x)))`), so reading a column THROUGH a derived column is one composed projection, never a chain of intermediate `Order<intermediate>` owners.
- The instance is the one consumer the leaf-axis defeat passes ENTIRELY silently: `Order.number` accepts any `number` and `Equivalence.string` any `string`, so `mapInput` type-checks against a frozen literal column and a re-widened primitive identically — it never refuses the wider column, it merely stops proving the row set; the audit is never whether the `mapInput` compiles but whether the projected column reads `(typeof Table)[keyof typeof Table]['rank']` as a literal union, and the key-axis dual makes a key-erased binding an `Order<string>` admitting keys the table never declared, so the instance is built over the literal-key binding, never a reconstructed record.
- Multi-column composition rides the monoid on the instances themselves: `Order<K>` is a Monoid under `combine` (left-most non-zero `Ordering`, `that` only on a tie) with `empty = make(() => 0)`, so `Order.combineAll([byRank, byLabel])` lexicographically orders by rank then label and prepending a tie-break is one more fold element; the identity-laden fold deletes the empty-case branch (`combineAll([])` is the all-equal no-op), so a runtime-chosen subset of sort columns degrades gracefully with zero rows folded, never an `undefined` comparator.
- `Struct.getOrder({ col: Order.X })` and the `combineAll` chain are observationally identical orderings differing in projection ARITY, and the arity decides the owner: `getOrder` reads every column from one already-projected ROW value so the key lift is ONE pullback `Order.mapInput(getOrder({...}), (k) => Table[k])` (one row read, the struct walks columns), the `combineAll` chain applies a separate `(k) => Table[k].col` per column (N reads) — the struct form is denser when three-plus columns of one row participate (shared read, reusable over raw rows and projected keys), the chain when columns come from heterogeneous sources no single row value covers.
- The order/equivalence duality is load-bearing and breaks exactly at `reverse`: `Order.combine` is left-biased lexicographic (more columns → finer ORDER), `Equivalence.combine` is conjunctive `make((x,y) => self(x,y) && that(x,y))` (more columns → finer PARTITION, equivalent only when EVERY column agrees); `Order.reverse(O) = make((a,b) => O(b,a))` distributes non-commutatively (`reverse(combine(a,b))` descends both, `combine(reverse(a), b)` descends `a` ascends `b`), so a mixed-direction sort folds per-column reversed instances — and the equivalence side has NO `reverse` because a partition has no direction.
- The folded `Order<K>` is one owner driving the whole comparison surface — `Array.sort`, `Array.min`/`Array.max`, `Order.clamp`, `Order.between`, `Order.lessThan` all take a single instance — so a new row extends sort, extremal selection, clamp, and range predicate at once; `Array.sortBy(...orders)` is the variadic FACE giving the same lexicographic result but consumed inline by one sort, so prefer the folded named owner when more than one consumer reads the precedence; the equivalence side feeds `Array.dedupeWith` directly (the instance IS the `(a,b) => boolean` shape, no adapter) for a multi-column partition.

```typescript
import { Order, Equivalence, Array as Arr } from 'effect'

const TIER = {
    bronze: { rank: 0, halts: false, group: '<group-a>' },
    silver: { rank: 1, halts: true, group: '<group-a>' },
    gold: { rank: 2, halts: true, group: '<group-b>' }, // a new tier extends every instance and consumer below with zero edits
} as const satisfies Record<string, { readonly rank: number; readonly halts: boolean; readonly group: string }>
type Key = keyof typeof TIER

const halting = Order.combineAll<Key>([ // lexicographic: halts (false<true) dominates, rank breaks the tie; reverse flips the WHOLE precedence in one wrap
    Order.mapInput(Order.boolean, (k: Key) => TIER[k].halts),
    Order.mapInput(Order.number, (k: Key) => TIER[k].rank),
])
const congruent = Equivalence.combineAll<Key>([ // conjunctive: equivalent only when BOTH columns agree; adding a column splits more blocks, the dual direction
    Equivalence.mapInput(Equivalence.string, (k: Key) => TIER[k].group),
    Equivalence.mapInput(Equivalence.boolean, (k: Key) => TIER[k].halts),
])
const ceiling: Key = Arr.max(['bronze', 'gold', 'silver'], halting) // one folded instance also drives min/clamp/between; REJECT a re-spelled comparator per consumer
const distinct: ReadonlyArray<Key> = Arr.dedupeWith(['bronze', 'silver', 'gold'], congruent) // the Equivalence IS dedupeWith's callable
```

[ONE_TRAVERSAL_THREE_APPLICATIVES]:
- The dispatch over a COLLECTION of keys scales from one key to many to a concurrent batch through ONE traversal, the algebra swapped by the applicative supplied — `Array.Traversable.traverse(F)(keys, (k) => Table[k].run(arg))` is the universal form, the identity applicative collapsing to map-then-fold and `Effect`'s applicative threading the effect — so the per-combinator split (`Array.map` pure, `Effect.forEach` effectful) is one `traverse` whose `F` is the only change; a `dispatchAll(keys)` beside `dispatch(key)` is the modal duplication the traversal deletes, the singular `Table[k].run` the only owner.
- The pure aggregate is graded by the carrier's algebra: a `Monoid` arm-return folds over ANY key collection including empty (`Foldable.combineMap(Foldable)(Monoid)` needs the identity to return when zero arms fire), a `Semigroup`-only return folds a NON-EMPTY collection only — `Semigroup.intercalate(S, sep)` is the marker carrier whose `empty` cannot exist (an identity joining nothing with a separator is undefined), so the emptiness contract rides the carrier AND the key-collection type together (`Array.NonEmptyReadonlyArray<K>`), the absent identity surfaced as a type obligation, never a runtime length check.
- A product-shaped arm return DERIVES its monoid from its components: `Monoid.struct({ count: Number.MonoidSum, log: String.Monoid })` builds the row aggregate as the product of column aggregates seeding `empty` per-column, `Monoid.tuple(...Ms)` the positional dual, `Monoid.min`/`Monoid.max` lifting a `Bounded` (seeding `maxBound`/`minBound`) for an extremal aggregate — so foldability of a rich-row dispatch is decided once at the column types: every column monoidal makes the row monoidal, one non-monoidal column blocks the whole derivation.
- The effectful lift threads TWO carriers: `Applicative.getMonoid(Effect.getApplicative())(M)` lifts `Monoid<A>` to `Monoid<Effect<A,E,R>>` combining successes and seeding `Effect.succeed(M.empty)` (graded again — `SemiApplicative.getSemigroup` lifts a `Semigroup` over a non-empty collection), while the ERROR channel is the second carrier selected by `mode` — default short-circuits, `mode: "validate"` accumulates every failure as `Array<E>` — so a foldable effectful dispatch carries a success monoid on `A` AND an error semigroup on `E`, the carrier picking the success algebra and the dependence picking short-circuit-vs-accumulate, neither a boolean flag.
- Concurrency rides the APPLICATIVE selection, not the fold: `Effect.getApplicative(options?)` returns a different instance per `{ concurrency, batching }` and `traverse(F)` consumes it, the degree itself the closed policy alphabet `Concurrency = number | "unbounded" | "inherit"` — so sequential, capped, and unbounded batch over one vocabulary are three `getApplicative` calls with the same `Table[k].run`, the degree, the success algebra, and the error algebra three orthogonal policy choices over one arm read, widening from sequential to unbounded one literal swap.

```typescript
import { Array, Effect } from 'effect'
import { Applicative, Foldable, Monoid } from '@effect/typeclass'
import { getApplicative } from '@effect/typeclass/data/Effect'
import * as N from '@effect/typeclass/data/Number'

const STEP = {
    open: { run: (x: number) => Effect.succeed(x + 1) },
    work: { run: (x: number) => Effect.succeed(x * 2) },
    close: { run: (x: number) => Effect.succeed(x) }, // a row extends both the aggregate and the validating batch with zero combinator edits
} as const satisfies Record<string, { readonly run: (x: number) => Effect.Effect<number, string> }>
type Key = keyof typeof STEP

const effectSum = Applicative.getMonoid(getApplicative({ concurrency: 'unbounded' }))(N.MonoidSum) // Monoid<Effect<number,string>>: concurrency rides the applicative; empty = Effect.succeed(0)
const aggregate = Foldable.combineMap(Array.Foldable)(effectSum) // folds the effectful dispatch; the empty key list returns Effect.succeed(0)
const totalOf = (ks: ReadonlyArray<Key>, x: number): Effect.Effect<number, string> => aggregate(ks, (k) => STEP[k].run(x))
const accumulated = (ks: ReadonlyArray<Key>, x: number): Effect.Effect<ReadonlyArray<number>, ReadonlyArray<string>> =>
    Effect.validateAll(ks, (k) => STEP[k].run(x)) // mode-validate dual: SAME arm read, errors accumulate instead of short-circuiting
```

[ADMISSION_AND_DISPATCH_ARE_ONE_GATE_BY_PROVENANCE]:
- Admission-as-decode and admission-as-dispatch are one gate whose owner is the value's PROVENANCE, never the consumer's taste: a foreign value (`unknown` from the wire, an external string) crosses the DECODE gate, which produces the typed value from raw matter; a value already of the vocabulary's TYPE crosses the DISPATCH gate, which routes a value it does not construct, only discriminates — the within-family axes mirror exactly (`decodeUnknown` from `unknown` ↔ `Match.type<I>()` annotated; `decode` from `Encoded` ↔ `Match.value(i)` frozen by `<const I>`).
- Composition order is fixed by provenance: a foreign value crosses decode FIRST (`Effect.map(decodeUnknown(Vocab), dispatcher)`), the decode discharging membership so the dispatcher's `Match.exhaustive` runs over a value the type already proves is in the alphabet — decode answers "is this a member and which", dispatch answers "given the member, which arm", neither gate doing the other's job.
- Each gate is mandatory at a disjoint condition: the decode gate when `Encoded` differs from `Type` (a `transformLiterals` remap, a `NumberFromString` segment), the transform the dispatch gate has no bridge for; the dispatch gate when discrimination is on SHAPE beyond membership (a refinement, a per-arm `Extract` narrowing, a divergent return), the branch decode does not perform.
- `Schema.is(Vocab): (u) => u is A` is the gate's guard form — the runtime predicate the `as const` table CANNOT synthesize (its key set is a compile-time fact with no value) — narrowing a foreign value for a downstream `Match.when(Schema.is(Vocab), f)`; the structural table's runtime answers are `Record.has` (boolean presence) and `Record.get` (`Option<Row>`), the gate whichever the provenance demands.
- The two gates project ONE absence algebra: the decode terminal family is the absence-disposition projection of the Match terminal family, the same five dispositions selected by the consumer's algebra, isomorphic disposition-for-disposition.

| [DISPOSITION] | [DISPATCH_TERMINAL]   | [DECODE_TERMINAL]    | [ABSENCE_CARRIES]                       |
| :------------ | :-------------------- | :------------------- | :-------------------------------------- |
| prove-impossible | `Match.exhaustive`    | `Schema.Literal` static exhaustion | `Remaining`/residual subtracts to `never` |
| assert-phantom   | `Match.orElseAbsurd`  | `decodeUnknownSync`  | residue asserted unreachable, escaping at runtime if it inhabits |
| resolve-fallback | `Match.orElse(f)`     | a default            | any residual accepted                   |
| carry-Option     | `Match.option`        | `decodeUnknownOption`| reason ERASED                           |
| carry-Either     | `Match.either`        | `decodeUnknownEither`| residual PRESERVED                      |

- The two `either` Lefts share the carrier but carry categorically different MATTER: the decode `Left` is a RUNTIME value (a `ParseError` carrying the structured issue of WHY the foreign value was rejected), the dispatch `Left` is a TYPE (the residual union of WHICH members went unhandled, inhabited by the unmatched input) — `Either.getLeft`/`Either.merge` project either to the symmetric absence rail where the distinction is irrelevant.
- The provenance dictates WHEN a totality break surfaces: adding a member breaks `Match.exhaustive` at COMPILE time (the residual no longer subtracts to `never`), while a decode gate over a widened `Schema.Literal` keeps compiling and rejects the new-but-unlisted value at RUNTIME with a `ParseError` — a vocabulary that must fail the BUILD on an unhandled member uses the dispatch gate, one that must fail the REQUEST uses the decode gate.
- The `R` axis partitions the EXIT forms: `decodeUnknown`/`decode`/`validate` keep `Schema<A, I, R>` on `Effect<A, ParseError, R>`, while every synchronous carrier (`decodeUnknownEither`/`decodeUnknownOption`/`decodeUnknownSync` and the `decode*` duals) constrains to `Schema<A, I, never>` — an already-resolved `Either`/`Option` has nowhere to defer a pending requirement; `validateOption`/`validateSync`/`validateEither` keep `R` because `validate*` runs TYPE-side filters WITHOUT the decode transform so the `R` is inert, and `Schema.is`/`Schema.asserts` keep `R` for the same reason.
- The synchronous-exit `R = never` constraint is a routing rule, not a ceiling: a context-carrying vocabulary admits through `decodeUnknown(Vocab)` on the Effect carrier, then `Effect.either`/`Effect.option` move the `ParseError` onto the `Left`/`None` INSIDE the effect — the exact dual of `Match.either`/`Match.option` producing the same carriers synchronously because a matcher carries no requirement channel — so decode-of-context and dispatch land in one `Either<value, absence>` algebra, the requirement preserved on `R` while the absence rail is identical.

[RESOLUTION_WRITES_A_SLOT_VECTOR_ADMISSION_READS_IT]:
- Resolution is the dual fold of admission: admission folds absence onto an EXTERNAL rail (a `ParseError`, `Option.None`, a residual type) whose codomain is OUTSIDE the vocabulary; resolution folds the same missing key back to a DESIGNATED member whose codomain is the closed key union itself, so the resolved value re-enters every projection, order, and dispatch with no second gate — the two folds share one input and differ only in where they land, which is why one owner carries both.
- The resolution fold is a WRITE to the property-signature state vector `PropertySignature<TypeToken, Type, Key, EncodedToken, Encoded, HasDefault, R>` over the four orthogonal slots, recoverable from the signature type alone: `withConstructorDefault` writes ONLY `HasDefault: false → true` (the fallback fires at `make`, decode still demands the key); `withDecodingDefault` writes ONLY `TypeToken: "?:" → ":"` applying `Exclude<Type, undefined>` (fires at decode, `make` still demands); `withDefaults({ constructor, decoding })` writes both with two SEPARATE thunks because the make-time omitted-field meaning and the wire-time missing-key meaning are distinct events that may resolve to different members; the chain is direction-fixed — the decoding default requires the source to already be optional (`("?:","?:",false)`), the optionality the precondition the fold consumes.
- The `Exclude<Type, undefined>` the defaulted decode applies is the SAME `Exclude` that subtracts a member toward `never` in a partial dispatch, under two readings: the dispatch subtracts handled members, the decode default subtracts the `undefined` arm — both proving the absence case discharged at the type level, and a `Schema.optional(Vocab)` whose Type is `Vocab | undefined` reaching `Match.exhaustive` forces an `undefined` arm the alphabet does not own, the decoding default the fold that deletes it before dispatch.
- The fallback is checked against the alphabet AT THE DECLARATION: every default thunk is typed `() => NoInfer<Exclude<Type, undefined>>`, the `NoInfer` forbidding the literal from widening the field's inferred type and the `Exclude` forbidding a resolve back to `undefined` — a fallback off the alphabet fails at the field declaration, the same site the `satisfies` table rejects a forgotten column; `.annotations({ default })` is METADATA only (`DefaultAnnotation<A>` read through `getDefaultAnnotation` feeding JSON-schema/arbitrary projections), flips no token, strips no `undefined`, and conflating it with `optionalWith({ default })` leaves the field required.
- A resolution needing a member NOT in the alphabet is the signal the ALPHABET is wrong, not that the default should widen: the member is added as a vocabulary row first — one literal in the tuple or one row in the table — then named as the default, so the alphabet and its designated fallback grow from the one owner.
- The disposition over those slots is a closed three-way fold discriminated by the transform's `Option` arity over one owner: `optionalToRequired` is absence-in/member-out (`(Option<FA>) => TI`, total resolution), `requiredToOptional` is member-in/absence-out (conditional drop), `optionalToOptional` is absence-in/absence-out (resolve a present key, drop a present key, or leave a missing key missing) — the tokens partition them exactly (`(":","?:")`, `("?:",":")`, `("?:","?:")`), `optionalToOptional` the form that threads absence one more hop rather than discharging it.
- `optionalWith({ default })`'s thunk is a `LazyArg` of NO arguments — it cannot read a sibling field, a policy, or a service — so its fallback is a CONSTANT member; the moment the resolved member depends on other decoded fields or a requirement the nullary shape is the ceiling and the fold lifts to `optionalToRequired`, whose `decode` receives `Option<FA>` where `Option.none` IS the missing key, the body `Option.match({ onNone: () => member, onSome })` a fold over absence whose `onNone` reads context through its closure.
- Source and target may be DIFFERENT vocabularies, so `optionalToRequired` resolves a missing narrow-WIRE key to a wide-DOMAIN member outside the wire alphabet, the round-trip invertible because `encode` returns `Option.none` for that fallback; the token result `PropertySignature<":", TA, never, "?:", FI, false, FR | TR>` UNIONS both schemas' `R`, so a fold whose `onNone` reaches a service surfaces that requirement in the field's `R` (the same `R`-accumulation a struct extension performs), where the constant thunk's `FR | TR` collapses to `never`.
- `optionalWith(Vocab, { as: "Option" })` is the SAME owner selecting the CARRY disposition opposite to resolve: `default` writes `(":", member, true)` folding absence to a member, `as: "Option"` writes `(":", Option<Type>, false)` lifting the field to `Option<Type>` so the missing key is CARRIED as `None` — mutually exclusive in the option type itself (`OptionalOptions`'s `default` arm forbids `as`, every `as` arm forbids `default`), so a field is a resolution point OR a carry point, the exclusion proved at the declaration; `as: "Option"` carries `onNoneEncoding: LazyArg<Option<undefined>>` controlling whether a domain `None` omits the key or writes explicit `null`/`undefined`, the encode-side dual of the decode fold.

```typescript
import { Option, Schema } from 'effect'

const Wire = Schema.Literal('<a>', '<b>') // narrow wire alphabet
const Domain = Schema.Literal('<a>', '<b>', '<fallback>') // wider domain alphabet carrying a member the wire never sends

const resolved = Schema.optionalToRequired(Wire, Domain, { // the deepest resolution: onNone COMPUTES a domain-only member from absence, onSome lifts a present wire member
    decode: Option.match({ onNone: (): '<fallback>' => '<fallback>', onSome: (v): '<a>' | '<b>' => v }),
    encode: (m): Option.Option<'<a>' | '<b>'> => (m === '<fallback>' ? Option.none() : Option.some(m)), // the seam default round-trips to a dropped key, the inverse fold
})
const row = Schema.Struct({
    seam: resolved,
    carried: Schema.optionalWith(Wire, { as: 'Option', onNoneEncoding: () => Option.some(undefined) }), // CARRY disposition: absence is Option<...> downstream; the option type forbids pairing with default
})
// REJECT Schema.optionalWith(Domain, { default: () => '<fallback>' }) when the fallback must read a sibling field or service: the nullary thunk has no argument to read it from
```

[GET_OR_ELSE_WIDENS_OFF_THE_UNION]:
- The structural read resolves a foreign key with `Record.get(Table, key)` returning `Option<Row>` then `Option.getOrElse`, but the trap is that `getOrElse<B>(onNone: LazyArg<B>): <A>(self) => B | A` solves `B` from the thunk body IN ISOLATION from `A` — a fallback the checker widens (any non-`const` string operation, a parameter, a non-literal) solves `B = string`, and `B | A = string | Key = string`, silently escaping the closed union the `Some` arm preserved and defeating every downstream exhaustiveness.
- Closure is reached by constraining `B` to `A`: annotating the thunk `(): Key => '<member>'` forces `Key | Key = Key`, the annotation doing the work the codec path's `NoInfer<Type>` thunk does automatically — or resolve to the fallback ROW `Option.getOrElse(() => Table[fallbackKey])` whose type is `(typeof Table)[keyof typeof Table]`, the same type the `Some` arm carries, so `Row | Row = Row` needs no annotation; the schema fold cannot exhibit this trap because its fallback is checked against the alphabet by the combinator's typed thunk while the structural read's fallback is untyped value the inference widens.
- The widening fires at the `Option<A> → A` boundary, so a multi-candidate resolution DEFERS the single widening to one terminal: `Option.firstSomeOf(collection): <T>(Iterable<Option<T>>) => Option<T>` folds candidate resolutions to the first present member, every candidate `Option<Key>` keeping the fold `Option<Key>` while one `Option<WiderThing>` widens the whole fold — the cascade closure-safe exactly when every candidate resolves within the alphabet, and a foreign key admitted through `Schema.decodeUnknownOption(Schema.keyof(Table))` is already `Option<Key>`, chaining directly into `Option.orElse`/`firstSomeOf` so the admission `Option` and the resolution `Option` are ONE carrier the terminal `getOrElse` closes — the gate and the resolution share the rail rather than each owning a separate one.

[FINITENESS_IS_AN_AST_SPAN_PREDICATE]:
- The composite-key vocabulary is `Schema.TemplateLiteral(...params)` synthesizing the cartesian PRODUCT of its factor alphabets: a `Schema.Literal` union param distributes the interpolation across every member, so `Schema.TemplateLiteral(Scope, '.', Kind)` over two two-member factors yields the four-element `'app.warn' | 'app.fail' | 'db.warn' | 'db.fail'` — generated, never written — and growth is multiplicative (adding one member to any factor multiplies the alphabet by that factor's new cardinality with zero edits to the template owner), each factor itself a single source.
- Finiteness is a STRUCTURAL property of the lowered AST (`head: string` plus `spans: NonEmptyReadonlyArray<{ literal; type }>`), decidable by walking the span `type` set, not a type intuition: a `Literal`/finite-`Union`/finite-nested-`TemplateLiteral` span is closed, while `StringKeyword`/`NumberKeyword` are the ONLY open nodes — the lattice carries NO `BooleanKeyword` or `BigIntKeyword`, so a `Schema.Boolean` factor synthesizes the CLOSED `'true' | 'false'` and a fixed bigint stays closed: openness is decided by primitive IDENTITY, not cardinality (the unbounded `number` opens, the two-valued `boolean` does not).
- The closed product compiles to a pure-alternation regexp (a `Literal` span → `regexp.escape(...)`, a finite `Union` → `members.join("|")`) whose branch count IS the product cardinality, while an open factor injects `STRING_KEYWORD_PATTERN`/`NUMBER_KEYWORD_PATTERN` — the runtime witness that the alphabet is no longer enumerable, the same finiteness the type proves by `keyof typeof` over the product key; and a separator literal is NOT a factor — the constructor folds an adjacent literal into the preceding span's suffix (a lead literal becomes the AST `head`), so `'.'` never multiplies the product and a closed composite key stays closed across any punctuation.
- The synthesized product seeds the behavior table: `as const satisfies Record<typeof Code.Type, Row>` requires one row per product member, so a missing combination is a declaration-site error naming the absent composite key (the table's totality proof IS the product of the factor vocabularies), and widening a factor both widens `typeof Code.Type` and breaks the table until the new product rows land — the factor is the single source, the product alphabet and the table both deriving from it.
- A product subset is RECTANGULAR or it is a union of products: a per-factor narrowing (`Schema.pickLiteral` on a factor, re-synthesize) is one product contracting its regexp width and any table over it from one edit, while a subset picking composite members ACROSS factor boundaries cannot be a single product — a non-rectangular admitted set is the signal the owner is a `Schema.Union` of several products, each rectangle its own single source; the template has no `.literals`/`pickLiteral` of its own (it is a `TemplateLiteral<A>`, not a `Literal`), so the seam narrows the FACTOR, never a hand-listed allowlist of composite strings.

```typescript
import { Array, Schema, pipe, Record } from 'effect'

const Scope = Schema.Literal('app', 'db') // a finite factor; a Schema.String here injects [\s\S]*? and opens the product to a pattern
const Kind = Schema.Literal('warn', 'fail')
const InternalScope = pipe(Scope, Schema.pickLiteral('app')) // narrowing is a FACTOR operation; the product, its regexp width, and the table all contract from this one edit
const Code = Schema.TemplateLiteral(InternalScope, '.', Kind) // typeof Code.Type = 'app.warn' | 'app.fail', the cartesian product; the '.' is a span suffix, never a factor

const POLICY = {
    'app.warn': { halts: false, rank: 0 },
    'app.fail': { halts: true, rank: 2 }, // omitting one product member fails the satisfies naming the absent composite key
} as const satisfies Record<typeof Code.Type, { readonly halts: boolean; readonly rank: number }>
const blocking: ReadonlyArray<typeof Code.Type> = pipe(Record.keys(POLICY), Array.filter((c) => POLICY[c].halts)) // keys preserve the product union; total by construction
```

[PARSER_IS_THE_VALIDATOR_AS_A_TRANSFORM_FROM_SIDE]:
- `Schema.TemplateLiteralParser` is not a parallel constructor but a `transformOrFail(from, to)` whose `from` IS the validator over the joined ENCODED string and whose `to` is `Tuple(...elements)` — it compiles the validator's capturing regexp once, decode runs `re.exec(input)` then re-decodes `match.slice(1, params.length + 1)` through the tuple, so the string vocabulary and the structured tuple are one schema's two sides; every finiteness property of the product transfers unchanged because the validator is literally the parser's `from`, the only addition capture retention plus tuple re-decode.
- The decoded value is a `readonly` TUPLE aligned POSITIONALLY to the params, the structured decomposition of the matched string: a literal-value param survives as its unit literal (`['id', number, ':', 'x' | 'y']`) so the tuple carries both variable segments and structural punctuation as discriminable elements, a literal-UNION param produces a tuple slot typed as that union (a closed sub-alphabet inside the structured value), and `.params` enumerates the FACTORS and separators at runtime — never the product members — the structured analogue of a `Literal`'s `.literals`; a consumer typed by a slot reads `(typeof Parser.Type)[2]`, never a parallel `type Segment`.
- The parser is MANDATORY exactly when `Encoded` differs from `Type` for any segment: per-segment coercion (`NumberKeyword` → `compose(NumberFromString, schema)`, a non-string `Literal` separator → `transform(Literal(String(literal)), schema)`) runs ONLY in the parser, the act the validator structurally cannot perform (its `Type` IS the string union with nothing to decode into, so a `NumberFromString` segment handed to `TemplateLiteral` never executes and its typed `number` is unreachable) — the same mandate the decode gate carries over a value whose wire shape is not its domain shape, the coercion positional and independent so failures localize to the offending slot's `path` in the `ParseError`.
- The two constructors diverge on the requirement channel by their param constraint, the constructor identity itself the requirement's discharge proof: `TemplateLiteral` admits `Schema.AnyNoContext | AST.LiteralValue` and carries no `R` (it runs no transform, so a context-carrying segment would have no discharge site — the same shape as the synchronous decode carriers' `R = never`), while `TemplateLiteralParser` admits `Schema.Any | AST.LiteralValue` and accumulates `Schema.Context<Params[number]>` exactly as a struct extension unions its members' `R` — so the moment a factor must decode a captured part through a context-dependent codec the validator is rejected at the constructor call and the parser is the sole admitting owner, provenance selecting the constructor the way a value's provenance selects the decode-vs-dispatch gate.
- The product table and the per-factor dispatch are two owners of the same factors chosen by SEPARABILITY: a behavior depending on the COMBINATION (a policy differing for one specific composite) needs the full product table read by the recomposed key, a behavior depending on each factor INDEPENDENTLY needs only per-slot `Match.value(slot).pipe(Match.exhaustive)` over the decomposed tuple — collapsing `n₁·n₂·…` table rows to `n₁ + n₂ + …` per-factor arms, total because the slot's type IS the closed factor union; modeling separable logic as a full product table is the row-count blowup the decomposition deletes.

```typescript
import { Schema, Match, Effect, type ParseResult } from 'effect'

const Scope = Schema.Literal('<scope-a>', '<scope-b>') // a closed factor; its slot type IS the dispatch alphabet below
const Port = Schema.NumberFromString // a decoding segment: Encoded string, Type number — its coercion runs ONLY in the parser
const Addr = Schema.TemplateLiteralParser(Scope, ':', Port) // Type readonly ['<scope-a>'|'<scope-b>', ':', number]; R = Context<params> = never here
const decodeAddr: (u: unknown) => Effect.Effect<typeof Addr.Type, ParseResult.ParseError> = Schema.decodeUnknown(Addr)

const reach = (raw: string): Effect.Effect<number, ParseResult.ParseError> =>
    Effect.map(decodeAddr(raw), ([scope, _sep, base]) => // the ':' separator is its own captured tuple slot, discarded by binding; the parse proved the punctuation matched
        Match.value(scope).pipe( // per-factor dispatch over the decomposed slot; n₁ + n₂ arms, not the n₁·n₂ product table
            Match.when('<scope-a>', () => base + 1000),
            Match.when('<scope-b>', () => base + 2000),
            Match.exhaustive, // total because scope's slot is the finite factor union; widening Scope breaks this at compile time
        ),
    )
```
