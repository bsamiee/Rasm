# Error Projection And Recovery

[THE_FORMATTER_IS_THE_FOLD_ENGINE_WITH_THE_ISSUE_TREE_AS_SUBSTRATE]:
- `ParseResultFormatter<A>` is the failure-side mirror of the schema-AST fold engine: one interface — `formatIssue: (issue) => Effect<A>`, `formatIssueSync: (issue) => A`, `formatError`/`formatErrorSync` delegating through the wrapped `ParseError.issue` — parameterized by the carrier `A`, exactly as the success-side compiler is parameterized by what each arm returns; `TreeFormatter` is the instance at `A = string`, `ArrayFormatter` the instance at `A = Array<ArrayFormatterIssue>`, two renderers that are not parallel walkers but the SAME recursion over the runtime issue union differing only in carrier, so a third failure projection (a structured log record, a field-keyed error map) is one more `switch (issue._tag)` fold over the eight-arm union, never a hand-walked tree of `if`.
- The substrate the renderer folds is the runtime issue value, not the schema declaration — the renderer descends `Composite.issues`, `Pointer.issue`, `Refinement.issue`, `Transformation.issue` as its recursion frontier the way the codec compiler descends `propertySignatures`/`types`, so the failure projection IS structural recursion run over the residue the decode left behind: the issue sub-tree the decode produced is the input, the rendered carrier the output, the two folds (decode-forward, render-backward) sharing the path discipline and the totality the eight `_tag` arms enforce.
- The carrier choice is the only axis that distinguishes the two shipped renderers AND it forces a divergence the value-side projectors never face: `string` carries the branching as a nested `Tree<string>` forest a `drawTree` pass renders with box connectors, `Array<ArrayFormatterIssue>` collapses the branching into a flat list where each row's `path` array is the only residue of the depth — so where the value-side projectors (`Pretty`, `Arbitrary`, `JSONSchema`) all preserve the owner's tree shape in their output, the two failure renderers make OPPOSITE structural commitments at the recursive nodes, the carrier dictating whether nesting survives as forest depth or collapses into a path array.

```typescript
import { Array as Arr, Effect, ParseResult } from 'effect'
const tagCount: ParseResult.ParseResultFormatter<Record<string, number>> = {           // a third failure projection: one fold, new carrier
    formatIssue: (issue) =>
        ParseResult.isComposite(issue)
            ? Effect.reduce(Arr.ensure(issue.issues), {} as Record<string, number>, (acc, child) =>      // SingleOrNonEmpty normalized to a frontier
                  Effect.map(tagCount.formatIssue(child), (m) => ({ ...acc, ...m })),                     // recurse the frontier the codec descends
              )
            : Effect.succeed({ [issue._tag]: 1 }),                                                       // leaf: one occurrence of this failure tag
    formatIssueSync: (issue) => Effect.runSync(tagCount.formatIssue(issue)),
    formatError: (error) => tagCount.formatIssue(error.issue),
    formatErrorSync: (error) => tagCount.formatIssueSync(error.issue),
}
```

[THE_TWO_RENDERERS_ARE_THE_TREE_RETAINING_AND_THE_PATH_FLATTENING_FOLD_OF_ONE_ISSUE]:
- `TreeFormatter` folds the issue into a `Tree<string>` (`{ value: string; forest: ReadonlyArray<Tree<string>> }`) then `drawTree` renders the forest with box-drawing connectors — every recursive arm builds `makeTree(title, forest)` so a `Composite` of N children becomes a node of N sub-trees, a `Pointer` becomes a node whose single child is the inner render keyed by the rendered path, a `Refinement`/`Transformation` becomes a node whose child is a kind-naming sub-tree wrapping the inner render: the issue's branching arity is PRESERVED as the tree's branching arity, the loss locus readable as the leaf-most line of a nested diagram.
- `ArrayFormatter` folds the same issue into a flat `Array<{ _tag; path; message }>` where the recursive arms CONCATENATE rather than nest: a `Pointer` does not emit an entry, it descends `getArrayFormatterIssues(issue.issue, undefined, path.concat(issue.path))` so its path segment accumulates onto the child's path; a `Composite` maps each child through the fold and `Arr.flatten`s the results, so the tree's depth is erased and every leaf surfaces as one flat row carrying the full accumulated `path` — the same branching the tree retains as forest depth the array retains as the length of each row's path array, two encodings of one loss graph.
- The structural choice is the consumer contract: the tree is the human-facing single string for a log line or a thrown message, the array is the machine-facing list for a form's per-field error map or a batch-ingest report keyed by row index — so a boundary that renders for a person reaches `TreeFormatter.formatErrorSync` and one that renders for a UI reaches `ArrayFormatter.formatErrorSync`, the two never a re-walk of the issue but the one fold at the carrier the consumer reads, the loose hand-written error-flattening loop deleted at both consumers.

[PARENTTAG_IS_THE_FLAT_RENDERER_SUBSTITUTE_FOR_THE_LOST_NESTING]:
- Flattening discards the wrapper nodes' identity, so `ArrayFormatter` threads a `parentTag` argument to RE-ATTACH the semantic failure layer onto the leaf it descends to: a `Refinement` whose `kind === "Predicate"` calls `getArrayFormatterIssues(issue.issue, _tag, path)` passing its own `"Refinement"` tag down, and the inner `Type` arm emits `parentTag ?? _tag` — so the flat row's `_tag` reads `"Refinement"` not `"Type"`, the leaf reporting that a refinement predicate rejected rather than that a raw type mismatched, the nesting the tree would have shown collapsed into the leaf's tag.
- The rebind is conditional on the wrapper's `kind`, the discriminant the issue tree carries naming WHICH leg failed: a `Refinement` rebinds only on `kind === "Predicate"` (a `"From"` failure passes `undefined` because the inner issue is itself a genuine type failure on the refinement's source, not the predicate), and a `Transformation` rebinds only on `kind === "Transformation"` (an `"Encoded"`/`"Type"` failure passes `undefined` because the loss is on one side's schema, not in the seam) — so the flat tag distinguishes a refinement's predicate from its source-type failure and a transform's seam from its side-schema failures, the three-way leg discriminant the tree shows as a kind-naming sub-tree the array shows as the leaf's `_tag` rebinding.
- `parentTag` propagates exactly one hop and resets to `undefined` at the next `Pointer` or `Composite` boundary (both descend passing `undefined`), so the rebound tag names the IMMEDIATE wrapper of the leaf, never an ancestor two levels up — a `Composite` under a `Refinement` resets the tag so each composite child reports its own immediate failure layer, the one-hop reset the reason a deeply-nested flat row's `_tag` is the leaf's proximate cause rather than the outermost wrapper.

```typescript
import { Either, ParseResult, Schema } from 'effect'
const Bounded = Schema.Struct({ size: Schema.Number.pipe(Schema.greaterThan(0)) })       // a refinement under a struct field
const rows = Schema.decodeUnknownEither(Bounded)({ size: -1 }).pipe(
    Either.mapLeft(ParseResult.ArrayFormatter.formatErrorSync),                           // Left is a ParseError; one fold flattens it
    Either.getOrElse(() => []),
)
const failingTag = rows[0]?._tag        // "Refinement" — the Predicate-kind wrapper rebound onto the leaf, not "Type"
const failingPath = rows[0]?.path       // ["size"] — the Pointer's segment concatenated, the tree's depth flattened to a path
```

[THE_MESSAGE_FOLD_CARRIES_AN_OVERRIDE_BIT_THAT_DECIDES_PRUNE_VERSUS_DESCENT]:
- The human-facing message at a node is not a slot read but an ordered fold with a control bit: `getMessage(issue)` reads the `MessageAnnotation` off the node, and the annotation's return is `string | Effect<string> | { message; override }` — the `{ override }` arm is the discriminant that decides whether the message PRE-EMPTS the subtree render or yields to it, so a message is at once the rendered text AND a render-control instruction, the failure-projection's analogue of the policy value that selects an algebra rather than a flag that branches a body.
- `override: false` (the default for a bare-string annotation) triggers `useInnerMessage` exactly when the node is a WRAPPER whose own text should defer to its child's: a `Composite`, a `Refinement` of `kind === "From"`, or a `Transformation` of `kind !== "Transformation"` recurses `getMessage(issue.issue)` instead of using its own message — so a struct-level message annotation does NOT clobber the precise field-level failure beneath it unless it declares `override: true`, the non-override message a fallback the descent overrides, the override message a cut that stops the descent.
- The render arms consume the resolved message as a SUBTREE PRUNE: every wrapper arm (`Transformation`, `Refinement`, `Composite`) tests `message !== undefined` and, when present, returns `makeTree(message)` — a leaf node with no forest — so an authored override message at a composite node collapses the entire failure subtree to one line, the dozens of child issues beneath it never rendered; absent a message the arm descends and builds the full forest, the presence of an override the switch between a one-line summary and the exhaustive tree.
- The same `getMessage` fold feeds BOTH renderers identically — `ArrayFormatter` tests the same `message !== undefined` and emits a single flat row carrying the override text and the current `path` rather than descending to the leaves — so an override message prunes the array to one row exactly as it prunes the tree to one line, the recovery-and-render algebra carrier-agnostic, the override bit governing both projections from the one annotation.

```typescript
import { Schema } from 'effect'
const Form = Schema.Struct({
    email: Schema.String.pipe(Schema.pattern(/@/)),
    age: Schema.Number.pipe(Schema.greaterThanOrEqualTo(18)),
}).annotations({
    message: () => ({ message: 'The submitted record is invalid', override: true }),    // override: true -> renderer PRUNES the subtree
})
const Field = Schema.String.pipe(
    Schema.pattern(/@/),
    Schema.annotations({ message: () => '<email-must-contain-@>' }),                     // bare string -> override:false, yields if a wrapper
)
```

[DECODINGFALLBACK_IS_THE_DECODE_ONLY_NODE_MEMOIZED_FALLIBLE_RECOVERY_COMPOSITION]:
- `decodingFallback: (issue: ParseIssue) => Effect<A, ParseIssue>` is the recovery slot composed into the parser as `orElse(parser(input, options), fallback)` then wrapped in `handleForbidden` — the fallback RECEIVES the issue that the node's own parse produced and returns either a recovered `A` or a new `ParseIssue`, so recovery is itself FALLIBLE: a fallback that cannot salvage the value re-fails with its own issue, the recovered branch and the re-failed branch both flowing back into the same `ParseIssue` channel the original parse used, never a thrown error and never a silent `undefined`.
- Recovery is DECODE-ONLY by construction: the composition is gated `isDecoding && Option.isSome(decodingFallbackAnnotation)`, so the encode direction has no fallback hook at all — a malformed wire value can resolve to a substitute on the way in, but a value being serialized out cannot be patched, the asymmetry deliberate because the wire is the untrusted side and the interior is the side the codec already proved correct; reaching for a fallback to repair an encode is the requirement that the interior value was never valid, fixed at the owner, not at the seam.
- The recovery wraps at `goMemo`, so the composed `orElse(parser, fallback)` is CACHED per AST node in the decode memo map — the fallback is resolved once per node identity and the memoized parser carries the recovery for every value that flows through that node, so a fallback on a leaf reused across a struct recovers uniformly at every occurrence of that leaf without re-attaching the annotation, the recovery a property of the node the memo conserves rather than a wrapper re-applied per decode.
- The fallback's `handleForbidden` wrapping is the bridge to the runner-synchrony law: a fallback returning an `Effect` that reaches a service makes the recovery async, so under a sync runner the same `handleForbidden` that converts an unprovided dependency into a `Forbidden` issue applies to the RECOVERY path — a fallback needing a layer under `decodeUnknownSync` surfaces a `Forbidden` exactly as a `filterEffect` does, the recovery sharing the synchrony obligation of the parse it recovers, never an escape from the requirement the codec carries.

```typescript
import { Effect, ParseResult, Schema } from 'effect'
const Resilient = Schema.Struct({ count: Schema.NumberFromString }).annotations({
    decodingFallback: (issue) =>                                                         // receives the very issue THIS node produced
        ParseResult.ArrayFormatter.formatIssue(issue).pipe(
            Effect.flatMap((rows) =>
                rows.length === 1 && rows[0].path.length === 0                            // salvage only the single top-level defect
                    ? Effect.succeed({ count: 0 })                                        // recovered A flows back into the success channel
                    : Effect.fail(issue),                                                 // unsalvageable -> re-fail with the original issue, recovery is fallible
            ),
        ),
})
const decoded = Schema.decodeUnknownSync(Resilient)('<not-a-record>')                     // recovers to { count: 0 } at the node, memoized
```

[THE_FORMATTER_CARRIER_IS_ALWAYS_EFFECTFUL_BECAUSE_A_MESSAGE_MAY_REACH_A_SERVICE]:
- `formatIssue` returns `Effect<A>` not a bare `A` because the `MessageAnnotation` and `MissingMessageAnnotation` arms both admit an `Effect<string>` return — a message can be computed by an effectful resolution (a localized string fetched, a template rendered through a service) — so the render fold must thread `Effect` to await any node's effectful message, the whole projection lifted into the effect carrier by the possibility of one effectful message anywhere in the tree, the failure projection inheriting the success projection's carrier discipline.
- `formatIssueSync` resolves the carrier by the same fork-flush-poll discipline the decode synchrony uses: it runs `Either.getOrThrow` when the render produced an `Either` (every message was a pure string) and `Effect.runSync` otherwise — so a render whose every message is a literal stays synchronous and a render with one effectful message forces the async path, the sync formatter throwing if a message's effect cannot resolve synchronously exactly as the decoder throws on an unresolvable parse; reaching for `formatIssueSync` over a tree carrying a service-backed message is the render-side synchrony mismatch, the effectful `formatIssue` the form the service-reaching message demands.
- The effectful carrier is the reason a message annotation is the ONE failure-side slot that can widen the render's requirement set, and it does so WITHOUT a type-level `R` channel: `Effect<A>` on the formatter erases the requirement, so a message reaching a service must be provided before the render runs or it surfaces as a sync-flush defect, never as a compile-time `R` the formatter signature carries — the failure projection's service reach is a runtime obligation the carrier admits, not a static requirement the formatter declares, the asymmetry against the codec's `Context` channel deliberate because a renderer is a boundary act, not an interior one.

[THE_MISSING_AND_UNEXPECTED_LEAVES_ARE_THE_TWO_RENDER_ARMS_WITH_NO_INNER_ISSUE]:
- `Missing` and `Unexpected` are the failure-tree's true leaves — neither carries an `issue` child, so both render arms terminate without recursion — and each owns a dedicated message resolver distinct from the generic `Type` path: `Missing` reads a `MissingMessageAnnotation` (a NULLARY `() => string | Effect<string>` thunk, no issue argument, because there is no actual value to describe) falling back to `"is missing"`, while `Unexpected` reads its own `message` field falling back to `"is unexpected"`, the two absence-and-excess diagnostics rendered by resolvers that take no descent.
- The `Missing` resolver reads its thunk off the issue's OWN `ast` (`getMissingMessageAnnotation(issue.ast)`) rather than off the issue, and the thunk is NULLARY (`() => string | Effect<string>`) because the absent key has no `actual` for the message to inspect — so the absence diagnostic is the one render path that resolves its text from the expected-node identity alone, the renderer reading a field-expectation message where every other arm reads a value-dependent one; the resolver's nullary shape is the structural marker that a `Missing` leaf describes what was wanted, not what was wrong with what arrived.
- `Unexpected` is the leaf reachable only under `onExcessProperty: "error"`, so its render arm exists for a diagnostic the default decode never produces — the renderer's arm map is total over the eight issue tags but two of those tags (`Unexpected`, `Forbidden`) are option- and runner-conditioned, present in the fold's domain yet emitted only under specific decode policies, the render fold carrying arms for failure modes the value graph alone cannot produce.

[REACH_LIMIT_FENCE]:
- A bespoke `ParseResultFormatter<A>` earns its place only by folding the issue tree to a carrier no shipped renderer produces — a field-keyed map for a form, a severity-tagged log record, a count by failure layer — and deleting the hand-walked issue recursion it replaces; a formatter that re-emits what `TreeFormatter` or `ArrayFormatter` already yields, or that re-flattens the array the `ArrayFormatter` already flattened, is the renderer spelled longhand, rejected even when correct — the proof is the carrier being a target the two shipped folds cannot reach, never a re-decoration of their output.
- A `message` annotation with `override: true` earns the prune only where the subtree it cuts is genuinely noise to the consumer — a top-level "record invalid" summary for a UI that shows field errors elsewhere; an override message on a node whose children carry the actionable loci silently discards the diagnostic the consumer needs, so the override is reserved for a node whose summary is MORE useful than its descent, and a bare-string (`override: false`) message is the default that contributes a fallback without cutting the tree, reaching for override to suppress a verbose tree the consumer should instead render with `ArrayFormatter`.
- A `decodingFallback` earns its slot only where a non-fatal wire defect has a principled substitute the node can compute from the issue — a malformed optional metric defaulting to a sentinel, a deprecated enum code mapping to its successor; a fallback that masks a defect the boundary must surface (a malformed identity field, a missing required relation) is the recovery that swallows a real failure, so the fallback is authored where the substitute is a legitimate domain value and re-fails with the original issue where it is not, never a blanket `Effect.succeed` that converts every defect at the node into a silent default.
- A `MissingMessageAnnotation` or an `Unexpected` message customization earns its place only where the default `"is missing"`/`"is unexpected"` reads wrong for the boundary's audience — a required relation whose absence needs a domain explanation; a thunk restating the default phrasing is the slot that decides nothing, and the absence/excess diagnostics stay on their library defaults wherever the failure locus and tag already name the field, the per-field message reserved for an expectation the tag alone does not convey.
- The issue tree's construction, the owner-baked message and fallback policy, and the per-node annotation placement are the success-side owner's concern — the renderer CONSUMES the issue tree and the annotations the owner stamped, it does not decide which node carries which message or how the issue tree branches; a renderer that reconstructs the issue tree, re-derives which field failed, or re-attaches policy the owner should have stamped is the projection reaching back across the seam into the construction it folds, the renderer's sole obligation the carrier it produces from the tree it is handed.
