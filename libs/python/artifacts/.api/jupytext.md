# [PY_ARTIFACTS_API_JUPYTEXT]

`jupytext` supplies the notebook<->text pairing surface for the artifacts reports rail: a polymorphic `read`/`reads`/`write`/`writes` quartet that exchanges an `nbformat.NotebookNode` across `.ipynb` and the text grammars (`py:percent`, `py:light`, `md`, `md:myst`, `Rmd`, `qmd`, `R`, `jl`, marimo, ...), a paired-path family (`paired_paths`/`base_path`/`full_path`/`find_base_path_and_format`) and a `sync_pairs`/`read_pair`/`write_pair` synchronization core that keeps a `.ipynb` and its text twin in lockstep, a `combine.combine_inputs_with_outputs` merge that grafts executed outputs back onto a re-read text source, a `config.JupytextConfiguration` traitlets policy object that drives paired-format selection and metadata filtering, the `jupytext.formats` registry that resolves and normalizes the `fmt` selector, and the jupyter-server contents managers for live paired editing. The package owner threads `read`/`reads`/`writes`, `guess_format`, `paired_paths`, `combine_inputs_with_outputs`, and `config.JupytextConfiguration` through the universal-tier rails — every conversion is a `RuntimeRail`/`expression` `Result` step, `beartype`-checked at the boundary, captured under a `structlog`+OpenTelemetry span, and folded into the `core/receipt#RECEIPT` `ArtifactReceipt.Report` case; it removes any hand-rolled cell-block parser because jupytext owns the percent/light/myst grammar end to end, removes any hand-rolled `.ipynb`<->text sync because `sync_pairs` owns the timestamp-ordered merge, and never re-implements the `nbformat` round-trip jupytext already drives. jupytext is the text-source pairing layer feeding the `nbclient`/`papermill` execution rail and the `nbconvert` export rail, never a parallel notebook executor or HTML renderer.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: read and write conversion
- rail: reports

[ENTRYPOINT_SCOPE]: paired-path derivation and synchronization (`jupytext.paired_paths`, `jupytext.pairs`, `jupytext.sync_pairs`)
- rail: reports

The pairing family is jupytext's defining capability and the reports rail's paired-editing core. `paired_paths` enumerates every file path a notebook pairs to under a `formats` policy; `base_path`/`full_path` are the stem<->path projection; `find_base_path_and_format` recovers which paired format a concrete path is. `sync_pairs.read_pair` reads a paired group (`list[NotebookFile]`) into the most-recent notebook merged with the latest outputs, and `write_pair` writes a notebook back to every paired path through one `write_one_file` callback; `latest_inputs_and_outputs` resolves which member holds inputs and which holds outputs by timestamp. The owner reaches these only when authoring/maintaining a paired set on disk; the `read`/`writes` quartet stays the single-file path.

[ENTRYPOINT_SCOPE]: output merge and round-trip equivalence (`jupytext.combine`, `jupytext.compare`)
- rail: reports

`combine.combine_inputs_with_outputs` is the load-bearing graft for the execute-then-pair flow: after `nbclient`/`papermill` executes the `.ipynb`, the edited text source is re-read and `combine_inputs_with_outputs(nb_source, nb_outputs, fmt)` maps the executed outputs back onto the matching source cells (matching by content, preserving filtered metadata) so a text-paired notebook regains its outputs without re-execution. The compare family is the round-trip guard: `compare_notebooks`/`compare_cells` diff two notebooks at the cell level returning a `NotebookDifference`, and `test_round_trip_conversion` asserts a text->node->text cycle is lossless — the equivalence oracle a reports receipt cites.

## [04]-[IMPLEMENTATION_LAW]

[REPORTS_NOTEBOOK]:
- import: `import jupytext` at boundary scope only; module-level import is banned by the manifest import policy.
- conversion axis: one `read`/`reads` pair owns parsing and one `write`/`writes` pair owns serialization; the file-vs-string distinction is the `fp`/`text` argument, never a parallel reader type; `read`/`write` delegate to `reads`/`writes` after extension handling and route every text grammar through one `TextNotebookConverter`.
- format axis: `fmt` is the single format selector — short form (`py:percent`) or long-form dict — keyed per call and normalized through `long_form_*`/`short_form_*`; `.ipynb` round-trips through `nbformat` while text formats route through the registry implementation, never a per-format function family at the boundary.
- detection axis: `guess_format`/`divine_format` resolve `(format_name, format_options)` from content and extension; `get_format_implementation` resolves the `NotebookFormatDescription` row; format identity is data over `JUPYTEXT_FORMATS`, never a hard-coded branch.
- config axis: one `config.JupytextConfiguration` (via `config.load_jupytext_config`) carries paired formats, metadata filters, and cell-marker conventions threaded through `config=`; metadata filtering (`notebook_metadata_filter`/`cell_metadata_filter`) and paired-save policy (`preferred_jupytext_formats_save`) are config traits, never per-call flags scattered across the owner; `metadata_filter_as_dict`/`filter_metadata` are the functional projection of those same traits.
- exchange axis: the notebook payload is the `nbformat.NotebookNode` from `nbformat.read`/`reads`; jupytext owns the text<->node mapping and never mints a parallel notebook model.
- sync axis: on-disk paired editing routes through `paired_paths`->`sync_pairs.read_pair`/`write_pair` with `pairs.latest_inputs_and_outputs` resolving the inputs/outputs holders by timestamp; the merge of executed outputs onto a re-read text source is `combine.combine_inputs_with_outputs`, never a hand-rolled cell-zip — so an executed `.ipynb` plus an edited `.py:percent` reconcile into one notebook through the package's own timestamp-ordered algorithm.
- pairing axis: `TextFileContentsManager`/`AsyncTextFileContentsManager`, built by the contents-manager class builders, own paired `.ipynb`<->text editing inside a jupyter-server host; absent jupyter-server they are deferred raisers, never a silent no-op.
- equivalence axis: `compare.compare_notebooks`/`test_round_trip_conversion` is the round-trip oracle — a text->node->text cycle is asserted lossless and the diff is a `NotebookDifference`, the equivalence fact a reports receipt cites, never a string-equality check on the rendered text.
- evidence: each conversion captures source extension, resolved `format_name`, format options, `nbformat` major/minor version, and output byte length as a reports receipt — the `core/receipt#RECEIPT` `ArtifactReceipt.Report` case, contributed once through the runtime `ReceiptContributor` port, never a jupytext-specific receipt shape.
- boundary: jupytext owns notebook<->text conversion, content-based format detection, paired-path sync, output merge, and pairing config over `nbformat` payloads; `nbformat` owns the notebook node schema and `markdown-it-py`/`mdit-py-plugins` own the myst markdown parse; headless notebook execution routes to `nbclient`/`papermill`; executed-notebook export to HTML/Markdown routes to `nbconvert`; live jupyter-server hosting stays outside this package.

[STACK_LAW] — both-tier integration the owning design composes:
- universal `libs/python/.api` rails ON TOP OF this folder package: every `read`/`reads`/`writes`/`combine_inputs_with_outputs` call is a node in a `RuntimeRail`/`expression` `Result` chain (a `JupytextFormatError`/`PairedFilesDiffer`/`NotSupportedNBFormatVersion` is mapped to a typed `Error`, never propagated as a raise into the rail), the boundary entry is `beartype`-checked, the conversion is wrapped in a `structlog`+OpenTelemetry span carrying the resolved `format_name`/extension/`nbformat` version, and the outcome contributes the `ArtifactReceipt.Report` case via `core/receipt#RECEIPT`.
- the `fmt` selector is admitted once as a discriminant and normalized through `long_form_one_format`/`short_form_one_format` so the design models it as `msgspec`/`pydantic` data over the `JUPYTEXT_FORMATS` registry, never a hard-coded grammar branch — format identity is a `NotebookFormatDescription` row.
- the on-disk sync arm runs its `read_one_file`/`write_one_file` callbacks inside the artifacts `anyio` structured-concurrency boundary (paired writes fan under one task group), so a multi-format paired set materializes under one cancellation scope, the package supplying the merge algorithm and the universal rail supplying the concurrency + receipt fold.
- jupytext stacks BELOW the `nbclient`/`papermill` execution rail and the `nbconvert` export rail and ABOVE `nbformat`: `read`/`reads` produces the `NotebookNode` those owners execute/export, `combine_inputs_with_outputs` re-absorbs their executed outputs, and `writes` serializes the text twin — one linear reports pipeline, never a parallel executor or renderer.

[RAIL_LAW]:
- Package: `jupytext`
