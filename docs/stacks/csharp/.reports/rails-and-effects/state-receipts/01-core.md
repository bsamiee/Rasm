# State-Receipts: Core Findings

[ATOM_INTERNALS]:
- `Atom<A>` stores its value in a `private volatile object` field; `Box<A>` dispatches at static construction to either a CIL-emit path (unboxed field projection for value types via `DynamicMethod`) or an identity cast (reference types), falling back to a plain cast on platforms where `ILCapability.Available` is false. Value-type reads unbox from `Box<A>.Value` — no allocation. Value-type writes always allocate a new `Box<A>` wrapper. Reference-type reads and writes go through identity without allocation.
- `Swap` is a CAS spin loop: it reads `value`, calls `f`, calls `validator(val2)`, and then `Interlocked.CompareExchange`s; on CAS failure it calls `spinWait.SpinOnce()` and retries. `f` will be called multiple times under contention — any side effect inside `f` is repeated. `SwapMaybe` breaks the loop early when `f` returns `None`, leaving the atom unchanged and returning the current value without retrying.
- The `Change` event fires synchronously on the calling thread after the CAS succeeds, before `Swap` returns. There is no dispatch queue; a slow handler throttles the swapper directly.
- `Atom<A>(A value)` (no validator) returns `Atom<A>` directly. `Atom<A>(A value, Func<A, bool> validator)` returns `Option<Atom<A>>`; `None` means the seed value failed validation — the atom is never constructed.
- `atomIO<A>(A value)` returns `IO<Atom<A>>` directly. `atomIO<A>(A value, Func<A, bool> validator)` returns `IO<Atom<A>>` and throws `Exceptions.ValidationFailed` inside the `IO` boundary when the validator rejects the seed. The `IO` resolves to a failed execution — not `IO<Option<Atom<A>>>`. The two construction paths are not symmetric: the synchronous factory returns presence or absence in the type; `atomIO` returns failure on the error rail. Callers that want to handle rejection without an error rail must use the synchronous factory.
- `Atom<M, A>` carries metadata as a `private readonly M metadata` field. It exposes `Swap(Func<M, A, A>)`, `Swap(Func<M, A, Option<A>>)`, and their `IO`-lifting counterparts. The metadata field is not publicly readable; it flows only through these `Swap` overloads. A requirement to update the context signals that the context belongs inside the state value, not in metadata.
- `SwapIO(Func<A, A>)` and `SwapMaybeIO(Func<A, Option<A>>)` are `IO.lift` wrappers over synchronous `Swap`; they do not defer or schedule the CAS. The spin loop runs at `IO` evaluation time on the evaluating thread.

[ATOM_COLLECTION_VARIANTS]:
- `AtomHashMap<K, V>` exposes two `Swap` paths. Scalar mutations (`Add`, `Remove`, `SetItem`, `SwapKey`, etc.) run their own CAS spin loop and emit a `HashMapPatch<K, V>` carrying `From`, `To`, and a single-key `Change<V>`. The bulk path `Swap(Func<TrackingHashMap<K,V>, TrackingHashMap<K,V>>)` always constructs a `TrackingHashMap<K,V>` wrapper regardless of whether any observer is subscribed, applies the function, and derives a change trie via `TrackingHashMap.ChangesInternal`. The `TrackingHashMap` wrapper accumulates `Change<V>` values using `Change<V>.Combine` across mutations to the same key, so the emitted patch carries the net delta — not intermediate states.
- `Change<A>` is an `abstract class` with four sealed subtypes: `NoChange<A>`, `EntryAdded<A>`, `EntryRemoved<A>`, and `EntryMapped<A, B>` (implementing both `EntryMappedFrom<A>` and `EntryMappedTo<B>`). It implements `Monoid<Change<A>>` and `Semigroup<Change<A>>`. Composition laws: `NoChange + y = y`; `x + NoChange = x`; `x + Removed = Removed`; `Removed + Added(v) = Mapped(oldValue, v)`; `Added(x) + Mapped(→y) = Added(y)`; `Mapped(from, x) + Mapped(→y) = Mapped(from, y)`. An observer receiving the merged patch from a bulk `Swap` that inserts then updates the same key gets exactly one `EntryMapped` — not a separate `EntryAdded` followed by `EntryMapped`. Code that distinguishes insertion from update by pattern-matching `EntryAdded` must use key-level `SwapKey` or scalar ops to see per-operation changes.
- `AtomSeq<A>.Swap(Func<Seq<A>, Seq<A>>)` runs the same CAS spin against the internal `ISeqInternal<A>` field. There is no tracking wrapper and no diff; `AtomSeq` fires no `Change` event.
- `AtomQue<A>` exposes `Enqueue(A)`, `Option<A> Dequeue()`, `Option<A> Peek()`, `A DequeueUnsafe()`, and `Unit Clear()`, each a single-key CAS. There is no `Swap(Func<Que<A>, Que<A>>)`. Multi-step conditional operations — conditional dequeue based on peeked value, capacity-checked enqueue — require a higher-level `Atom<Que<A>>` or a `Ref<Que<A>>` inside `atomic()`.
- The static `AtomHashMap` class exposes `ToAtom<K,V>(this HashMap<K,V>)` as the canonical upgrade from an immutable snapshot to a concurrent cell.

[REF_AND_STM]:
- `Ref<A>` is a `sealed class` backed by an internal `long Id` keyed into a process-global `AtomHashMap` (`STM.state`) of `RefState` values. The `Value` setter calls `STM.Write(Id, value)`, which writes into the current thread's `Transaction` or throws `InvalidOperationException` if no transaction is active. `Ref<A>.Swap(Func<A,A>)` compiles to `Value = f(Value)` — it reads and writes via the property, so it requires an enclosing `atomic()` and is not independently thread-safe.
- `atomic<R>(Func<R>, Isolation)` checks `STM.transaction.Value` (an `AsyncLocal<Transaction?>`) first: if a transaction is already active it invokes `op()` and returns immediately — nested `atomic` calls join the outermost without creating a new retry scope.
- `Isolation.Snapshot` (default): the write-set is validated at commit; reads of unwritten `Ref`s are not checked. `Isolation.Serialisable`: the read-set (`t.reads`, a `HashSet<long>` accumulated unconditionally on every `STM.Read` call regardless of isolation mode) is validated before commit; any external write to a read-from `Ref` triggers `ConflictException` and a full retry. Under `Snapshot`, the read-set accumulates with no benefit — prefer `Serialisable` only when read-set validation is required.
- `commute<A>(Ref<A>, Func<A, A>)` stages the function in the transaction's `commutes` list rather than the `writes` map. At commit time, `CommitCommutes` runs after `CommitWrites` against the latest committed global state — not the transaction-local snapshot. Commute operations never conflict with concurrent writes to the same `Ref`, but the validator on a commuted `Ref` must be an unconditional invariant: `RefValidationFailedException` is not caught by the retry loop in `CommitCommutes`, and it escapes `atomic()` immediately.
- `STM.transaction` is `AsyncLocal<Transaction?>`. A transaction flows through `await` continuations; `Task.Run` inside `atomic()` creates a new execution context that does not inherit the value, so `Task.Run(() => ref.Value = x)` throws even when the outer `atomic` is active.

[MEMO_SHAPES]:
- `Memo<A>` uses a three-state `int` field (`0 = unloaded`, `1 = loading`, `2 = loaded`) with a `SpinWait` loop and `Interlocked.CompareExchange` to execute the acquire function exactly once. On exception, `state` resets to `0` and the exception rethrows — the memo becomes re-acquirable. A second thread spinning while `state == 1` spins indefinitely with no timeout; the acquire function must not block on any memo-protected resource.
- `memo<A, B>(Func<A, B>)` (function memoization) wraps a `WeakDict<A, B>` — a `ConcurrentDictionary` whose values are `WeakReference<OnFinalise<B>>`. `OnFinalise` removes its own dictionary entry in its finalizer. Cache entries are collected when no strong reference to the computed value exists outside the cache, preventing unbounded growth for short-lived keys. If the caller does not retain the result, `memo<A,B>(f)(key)` may call `f(key)` again on the very next invocation.
- `memoUnsafe<T, R>(Func<T, R>)` uses a plain `ConcurrentDictionary<T, R>` with no weak reference — values are held permanently. Both variants use a per-key `object` lock via `syncMap.GetOrAdd(key, new object())` to prevent double computation; the lock object is removed immediately after the protected block, leaving subsequent calls to the same key unguarded.
- `Memo<F, A>` is the higher-kinded lazy wrapper. `memoK<F, A>(Func<K<F,A>>)` constructs one in unloaded state; the `~` operator (defined as a C# 14 extension operator in `MemoExtensions`) forces acquisition by calling `.Value` and returning the `K<F, A>`. Force only at execution time — never at composition time.

[STATE_CELL_DISCIPLINE]:
- `Atom<T>` owns independent, single-ownership state. `Ref<T>` earns its STM overhead only when two or more `Ref`s must transition atomically; a single `Ref` inside `atomic(() => ...)` is equivalent to `Atom.Swap` in guarantees but adds transaction overhead and requires explicit `atomic` wrapping at every write site.
- The `Atom<M, A>` metadata form serves cases where the validator depends on stable external context that must not live inside the state value — a capacity limit, a schema version, or a generation token. Because `metadata` is unreachable after construction except through `Swap`, it cannot be updated. A requirement to update the metadata signals the context belongs inside the state value.
- `AtomHashMap` over `Atom<HashMap<K,V>>`: `AtomHashMap` provides key-level mutations (`SwapKey`, `AddOrUpdate`, `FindOrMaybeAdd`) that `Atom<HashMap>` cannot express without replacing the whole map on every operation. It also emits a typed `HashMapPatch<K,V>` diff; `Atom<HashMap>` fires `Change` with only the new whole-map value. Use `AtomHashMap` when key-level concurrent mutations are common or when change diffs drive observer logic.
- A fact stream folded with `slot` and `kind` metadata lives in `Atom<Seq<FactRecord>>` or `AtomSeq<FactRecord>` where `FactRecord` carries the kind discriminant and all evidence fields. For append-only streams under contention, `AtomSeq.Add(x)` runs an independent CAS per element; `AtomSeq.Swap(s => s.Add(...))` batches multiple adds into one CAS.
- Typed algorithm receipts — carrying solver path, sampling parameters, spectral or mesh evidence, status flags — belong to `Atom<ReceiptRecord>` rather than an operational fact stream. Fact streams are accumulated observations projected by fold; receipts are evidence of a specific completed computation that a consumer pattern-matches for routing, replay, or audit.

[VALIDATOR_FIRST_WRITER_WINS]:
- The validator on `Atom<A>` runs on every CAS attempt, not just the first. A first-writer-wins pattern uses a validator that rejects once the sentinel has changed: `Atom<Option<T>>(None, v => v.IsNone || ...)`. `SwapMaybe` returning the atom's value (unchanged on `None`, new on success) is the idempotency mechanism without a separate flag.
- A validator that rejects causes `Swap` to return the pre-swap snapshot immediately without retrying — the spin loop exits on `!validator(val2)`, not on a failed CAS. A rejecting validator is called exactly once per `Swap` invocation regardless of contention.
- `RefValidationFailedException` escapes the STM retry loop. A `Ref` validator that rejects a proposed commit is a hard abort of the `atomic` block, not a retry trigger. Use validators only for invariants that must never be violated; skippable transitions belong in the `commute` or `Swap` function logic.
