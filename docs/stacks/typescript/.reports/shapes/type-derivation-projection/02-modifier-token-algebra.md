# Modifier Token Algebra

[SEVEN_SLOT_IS_THE_FIELD_ALGEBRA_NORMAL_FORM]:
- The whole field-modifier vocabulary — `optionalWith`, `fromKey`, `propertySignature`, `withConstructorDefault`, `withDecodingDefault`, `withDefaults`, the brand modifier, the four `optionalTo*` transforms — does not produce eight parallel field kinds; it produces eight WRITES into one carrier `PropertySignature<TypeToken, Type, Key, EncodedToken, Encoded, HasDefault, R>`, so the algebra is closed: every modifier is a function `PropertySignature -> PropertySignature` (or a `Schema -> PropertySignature` lift) that rewrites a named subset of the seven slots and returns the same carrier, never a sibling type beside it.
- The two `Token` slots — `TypeToken extends ":" | "?:"` and `EncodedToken extends ":" | "?:"` — are INDEPENDENT, so the decoded-side presence and the wire-side presence are two free bits, not one optionality flag; `Type`/`Encoded` carry the per-side value type, `Key` the renamed Encoded key (`never` until `fromKey` writes it), `HasDefault` the construction-omittable bit, `R` the requirement a defaulting transform pulls in. The cross-product `2 (TypeToken) × 2 (EncodedToken) × {has-default, not}` is the full presence lattice one carrier spans, the `?`-token plus the construction-default plus the rename collapsed into positions rather than three parallel field declarations.
- `Struct.Type<F>` reads ONLY the Type token (`OptionalTypePropertySignature` matches `PropertySignature<"?:", ...>`), `Struct.Encoded<F>` reads ONLY the Encoded token (`EncodedOptionalKeys<F>` matches `PropertySignature<..., "?:", ...>`), and `Struct.Constructor<F>` reads the Type token OR `HasDefault` (`PropertySignatureWithDefault` matches `<..., true>`) — three independent mapped folds keyed on three different slots of the one carrier, so a single field's presence in the decoded record, the wire record, and the constructor input are three separate reads of three separate bits, never one optionality answer applied uniformly.

```typescript
import { Schema } from 'effect'
const Field = Schema.String.pipe(
    Schema.propertySignature,                       // <":", string, never, ":", string, false, never>
    Schema.fromKey('<wire-key>'),                   // rewrites Key only -> <":", string, "<wire-key>", ":", string, false, never>
    Schema.withConstructorDefault(() => '<value-a>'),   // rewrites HasDefault only -> <..., true, never>
)
const Row = Schema.Struct({ field: Field })
type Decoded = Schema.Schema.Type<typeof Row>       // { readonly field: string } — Type token ":" required
type Wire = Schema.Schema.Encoded<typeof Row>       // { readonly "<wire-key>": string } — renamed, Encoded token ":"
const made = Row.make({})                           // field omittable — HasDefault true is the only slot the constructor read
```

[OPTIONALWITH_IS_THE_SLOT_FORMULA_NOT_A_BRANCH]:
- `optionalWith<S, Options>` does not switch between code paths on the options shape; its result interface computes all seven slots as conditional expressions over `Types.Has<Options, "as" | "default">`, so the carrier IS the option set evaluated — `TypeToken` is `Types.Has<Options, "as" | "default"> extends true ? ":" : "?:"` (a value materializes on decode so the decoded side becomes required), while `EncodedToken` is unconditionally `"?:"` (the wire is always allowed to omit the key), the divergence between the two tokens spelled as one ternary apiece rather than a separate optional/required field pair.
- The `Type` slot is `(Types.Has<Options, "as"> extends true ? Option<Type<S>> : Type<S>) | (Types.Has<Options, "as" | "default" | "exact"> extends true ? never : undefined)` — `as: "Option"` reshapes the decoded value to `Option<A>`, while `as` OR `default` OR `exact` together suppress the `| undefined` residue; the `Encoded` slot is `Encoded<S> | (Has<"nullable"> ? null : never) | (Has<"exact"> ? never : undefined)`, so `nullable` adds `| null` to the WIRE only and `exact` strips the implicit `| undefined` from the WIRE only — every knob is one disjunct in one slot's union, the trap being that `as`/`default`/`exact` reshape the Type slot while `nullable`/`exact` reshape the Encoded slot, an option touching whichever side its semantics belongs to and never both symmetrically.
- `HasDefault` is exactly `Types.Has<Options, "default">`, so a `{ default }` field is constructor-omittable and a `{ as: "Option" }` field is not — the construction bit reads the `default` key alone, independent of the token flip the same `default` also triggers; `OptionalOptions<A>` is a six-arm union fencing the legal combinations (a bare `{ exact?, nullable? }`, a `{ default }` arm, four `{ as: "Option" }` arms varying `exact`/`nullable`/`onNoneEncoding`), so an illegal pairing — `default` with `as`, `as` with a non-`Option` literal — is rejected at the argument before any slot computes.

[MODIFIER_CONTRADICTION_IS_A_TOKEN_PRECONDITION]:
- Stacking is total only where the next modifier's input token constraint matches the current carrier's tokens, so two modifiers contradict exactly when their token preconditions disagree: `withDecodingDefault` is typed `(self: PropertySignature<"?:", Type, Key, "?:", Encoded, false, R>) => PropertySignature<":", Exclude<Type, undefined>, Key, "?:", Encoded, false, R>` — it DEMANDS both tokens already be `"?:"` and `HasDefault` already be `false`, flips the Type token to `":"`, and leaves `HasDefault` false; applying it to a carrier whose Type token is already `":"` (a `propertySignature` lift, an `optionalWith({ default })` result) is a compile rejection, not a silent no-op, the seam where a decoding default contradicts an already-required field surfaced at the call.
- `HasDefault` discriminates the two default modifiers' preconditions: `withConstructorDefault` accepts `HasDefault extends boolean` (any) and writes `true`, so it is idempotent-overwriting and stacks onto anything; `withDecodingDefault` accepts only `HasDefault extends false`, so a second decoding default is rejected while a second constructor default merely overwrites — the contradiction is one-sided, the construction slot tolerating restatement and the decoding-default slot refusing it because a decode default flips a token that cannot flip twice.
- `withDefaults({ constructor, decoding })` fuses both in one carrier rewrite — input `<"?:", Type, Key, "?:", Encoded, boolean, R>`, output `<":", Exclude<Type, undefined>, Key, "?:", Encoded, true, R>` — flipping the Type token AND setting `HasDefault`, so the two-default field is one modifier writing two slots rather than a `withDecodingDefault` chained onto a `withConstructorDefault` whose `false`-precondition the first one already violated; reaching for the chain hits the contradiction the fused modifier avoids by construction.
- `Exclude<Type, undefined>` in every default modifier's output Type slot is the boundary residue: a default materializes the value, so `undefined` cannot survive on the decoded side, and the modifier strips it from the Type slot while the Encoded token stays `"?:"` to admit the absent wire key — a field that decodes from a missing key to a guaranteed value, the asymmetry the two tokens were independent for.

[FROMKEY_IS_THE_KEY_SLOT_PROJECTION_PRESERVING_THE_REST]:
- `fromKey` is typed `(self: PropertySignature<TypeToken, Type, PropertyKey, EncodedToken, Encoded, HasDefault, R>) => PropertySignature<TypeToken, Type, Key, EncodedToken, Encoded, HasDefault, R>` — it accepts a carrier whose `Key` is still the wildcard `PropertyKey` and writes one concrete `Key`, leaving every other slot verbatim, so a rename composes AFTER any token-and-default decision and never disturbs it; the precondition `Key extends PropertyKey` means a second `fromKey` on an already-renamed field (whose `Key` is now a literal) is rejected, the one-rename-per-field invariant enforced by the same precondition mechanism the default modifiers use.
- `fromKey` is the SOLE modifier that writes the `Key` slot, and its precondition is the wildcard `PropertyKey` rather than the `never` sentinel — so it admits a freshly-lifted carrier (`Key: never`, since `never` is assignable to `PropertyKey`) and any other un-renamed carrier alike, while a literal `Key` from a prior rename is NOT assignable to the precondition the second call expects to widen, the wildcard-versus-literal distinction the exact mechanism gating one rename per field; the un-renamed `never` is what every other modifier's signature preserves untouched, so `fromKey` is the only stage in a chain that can fail on the `Key` axis.
- `fromKey.Rename<S, Key>` is the projection that lifts the rename across a structured owner: `S extends PropertySignature<infer TT, infer T, infer _K, infer ET, infer E, infer HD, infer R> ? PropertySignature<TT, T, Key, ET, E, HD, R> : S extends Schema.All ? fromKey<S, Key> : never` — it `infer`s all seven slots, rewrites ONLY `Key`, and re-emits the carrier, so a variant field's per-side token, default, and value type survive the rename intact; the fallback arm lifts a bare schema into a both-tokens-`":"` `fromKey` carrier, so the one projection handles an already-signature field and a raw schema by the same conditional.
- A variant rename mapping `{ [K]?: string }` over a variant `Config` maps each present key through `fromKey.Rename` and passes the absent ones unchanged, so renaming one variant's wire key (`json` differs from `select`) is a per-variant entry in one mapping the projection folds, the seven-slot carrier re-derived per variant rather than a field redeclared per wire shape.

[PROPERTYSIGNATURE_LIFT_IS_THE_BOTH_REQUIRED_IDENTITY]:
- `propertySignature(S)` is the carrier identity at the both-required corner of the lattice — `PropertySignature<":", Type<S>, never, ":", Encoded<S>, false, Context<S>>` — it admits a plain schema into the modifier algebra with both tokens `":"`, `Key` un-renamed (`never`), and no default, so it is the prerequisite lift before `fromKey` (which needs a `PropertySignature` to rewrite `Key` on) and the contrast against `optional`, which is the both-`"?:"` corner (`PropertySignature<"?:", Type<S> | undefined, never, "?:", Encoded<S> | undefined, false, Context<S>>`); the lift is reached only to attach a downstream modifier, never as a wrapper that decides nothing on its own.
- `tag(literal)` is the lattice corner a discriminant occupies — `PropertySignature<":", Tag, never, ":", Tag, true, never>` — both tokens `":"` with `HasDefault: true`, so the tag value is present on both the decoded and wire sides yet omittable at construction because the literal IS its own constructor default; `TaggedStruct(tag, fields)` prepends `_tag: tag<Tag>` to the field map, so the discriminant is one slot-saturated carrier the constructor fills and the codec emits, never a literal restated at both the make call and the schema.
- The lift initializes `Key` to `never` deliberately, not to a placeholder key — `never` is the inert state every modifier's signature carries through unchanged and the only state `fromKey`'s wildcard precondition can later widen, so the bare lift is the carrier in its renamable-but-un-renamed condition; the `Key` slot is thus a three-state axis the algebra reads positionally — `never` (lifted, un-renamed), a literal (renamed, locked against a second write), or `PropertyKey` (the wildcard before any decision) — one slot encoding the entire rename lifecycle rather than a has-rename flag beside the field.

[OVERRIDEABLE_IS_THE_GENERATED_LATTICE_CORNER]:
- `Overrideable<To, From, R>` is the seven-slot carrier at a corner no plain modifier reaches — `PropertySignature<":", (To & Brand<"Override">) | undefined, never, ":", From, true, R>` — both tokens `":"`, `HasDefault: true`, Type `(To & Brand<"Override">) | undefined`, Encoded `From` (a wire shape distinct from the decoded `To`), `R` the requirement an effectful generator pulls in; the carrier is present on both sides yet construction-omittable, so the type fuses a construction default with a `Context`-widening effect in slots no `withConstructorDefault` reaches — that modifier flips `HasDefault` alone and touches neither `R` nor the wire/decoded value divergence the generated corner carries.
- The `Brand<"Override">` on the Type slot is the witness an explicit caller value carries: `Override(value)` returns `value & Brand<"Override">`, so passing it at the construction site forces the field to the literal and bypasses the generator, the explicit-versus-generated decision carried in the value's brand inside the same Type slot rather than a parallel boolean argument — the `| undefined` arm is the omitted-and-generate case, the branded arm the supplied-and-keep case, one slot's union spanning both.
- `Overrideable(from, to, { generate, decode })` threads `generate: Option<ITo> -> Effect<From, ParseIssue, R>` into the `R` slot, so an insert-stamp reaching `Clock` or a generated identifier reaching entropy surfaces its requirement on the carrier and through the struct's `Context`, the generated column's effect requirement part of the field rather than smuggled into a constructor body; the wire-versus-decoded divergence (`From` ≠ `To & Brand`) makes the generated slot a transform, so its presence is bounded to the variant that uses it and absent from the variants that do not.

[BRAND_IS_A_TYPE_SLOT_INTERSECTION_OUTSIDE_THE_SIGNATURE]:
- The brand modifier is a `Schema -> Schema` combinator, NOT a `PropertySignature -> PropertySignature` one — `brand(B)(self): brand<S, B>` extends `BrandSchema<Type<S> & Brand<B>, Encoded<S>, Context<S>>`, intersecting `Brand<B>` onto the decoded Type and leaving Encoded plain — so a branded field enters the seven-slot algebra by being branded FIRST then lifted (`Number.pipe(brand("Id"), propertySignature, fromKey("..."))`), the brand riding the field's Type value the signature carrier then transports through its `Type` slot unchanged.
- The asymmetry — brand on Type, plain Encoded — composes with the token algebra without interference: a branded optional field is `optionalWith(Number.pipe(brand("Id")), { default })`, where the brand sits inside the `Type<S>` the optionalWith Type-slot formula reads, so the decoded value is `number & Brand<"Id">` (required, defaulted) while the wire stays `number | undefined` (optional, plain) — the nominal identity surviving every token flip because it lives in the value type the slot transports, not in a token.
- `BrandSchema.make(Brand.Unbranded<A>)` accepts the unbranded primitive and returns the branded value, so the brand is a derivation off the one schema, never a parallel branded type declared beside it; a brand with no runtime refinement behind it and no real nominal collision is the modifier deleted, the primitive standing — the brand earns its Type-slot intersection only by fencing a value the program elsewhere must not confuse.

[FOUR_OPTIONALTO_ARE_THE_TRANSFORM_CORNERS]:
- `optionalToRequired`, `requiredToOptional`, and `optionalToOptional` are the three field-transform carriers spanning the token cross-product an `Option`-mediated bidirectional pair can reach: `optionalToRequired` is `<":", TA, never, "?:", FI, false, FR | TR>` (wire-optional → decoded-required, `decode: Option<FA> -> TI` materializing the missing case), `requiredToOptional` the inverse `<"?:", TA, never, ":", FI, false, FR | TR>`, `optionalToOptional` the both-`"?:"` form — each writes the two tokens to OPPOSITE values across the seam and threads the transform's `FR | TR` into `R`, the presence asymmetry declared as a total bidirectional function pair rather than a post-decode presence patch.
- These differ from the default modifiers by carrying a full `Schema -> Schema` transform (a `from` and `to` schema with `decode`/`encode`), so their Type and Encoded VALUE slots may diverge (the decoded `TA` differs from the wire `FI`), where `withDecodingDefault` only flips the Type TOKEN and keeps the value; the transform corners are reached when the missing case must compute a value (`Option.getOrElse`, a sentinel collapse), the default modifiers when it is a constant — the carrier corner is the same `<":", _, _, "?:", _>` presence asymmetry, the distinction whether the value is transformed or merely defaulted.
- `encode` returning `Option.none` is the wire-key-omission channel: in `optionalToRequired` the `encode: TI -> Option<FA>` returns `none` to drop the key from the output, so the `"?:"` Encoded token is honored by the encode leg returning absence — the token is not decorative, it is the contract the transform's `Option` return type satisfies, and a transform that always returns `some` while declaring `"?:"` is the field that never actually omits, the token over-promising what the encode delivers.

[STACK_ORDER_IS_FIXED_BY_THE_PRECONDITION_CHAIN]:
- The modifiers do not commute; their token preconditions force one legal order — a raw schema must be lifted (`propertySignature` or `optionalWith`) before `fromKey` (which needs a `PropertySignature` carrying a `Key` slot), and a default that flips the Type token (`withDecodingDefault`) must precede a modifier requiring the flipped state — so the pipe reads as a precondition chain where each stage's output token set is the next stage's input constraint, a misordered pipe failing at the stage whose precondition the prior output violates, the compiler walking the chain the way the codec will walk the decode.
- `withConstructorDefault` is the one order-free modifier (precondition `HasDefault extends boolean`), so it may sit anywhere in the chain and is the safe terminal write; every other modifier carries a precise input shape, so the order is recoverable from the signatures rather than documented — reading the precondition tokens off each modifier IS reading the legal stacking, the carrier's type the proof the stack is sound before any value decodes.
- A modifier whose precondition the carrier cannot meet is the contradiction surfaced at declaration: two `fromKey` calls (second sees a literal `Key`, the precondition wanted the `PropertyKey` wildcard), two `withDecodingDefault` calls (second sees the Type token already flipped to `":"`, the precondition wanted `"?:"`), a `withDecodingDefault` after a `propertySignature` lift (same flipped-Type-token rejection) — each is a compile rejection naming the slot the second modifier expected to find in a state the first already left; the contradiction is always a token mismatch rather than a `HasDefault` clash, because the decoding default's irreversibility is encoded as a token it can only flip from `"?:"` once, the field algebra refusing an incoherent presence specification before it can produce a wrong decode.
