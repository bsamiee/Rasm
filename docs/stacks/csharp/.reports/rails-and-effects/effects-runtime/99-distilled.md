# Effects-Runtime: Distilled

[NEWTYPE_ISOMORPHISM_DERIVATION]:
- A wrapper carrier earns its entire trait stack from one bidirectional structure isomorphism, never from re-implemented operations: `Deriving<Super, Sub>` aliases `NaturalIso<Super, Sub>` — the down-arrow `static abstract K<Sub,A> Transform<A>(K<Super,A>)` plus the up-arrow `CoTransform`, variance-annotated mirror images (`Natural<out F, in G>` / `CoNatural<in F, out G>`) so one Super/Sub pair satisfies both directions. Every derived operation is the same sandwich: lower through `Transform`, run the representation's already-correct instance, lift back through `CoTransform` — structure declared once, behavior computed.
- Both arrows are `static abstract`, resolved at the constraint and devirtualized per concrete carrier: the lowering is a field read; the lifting allocates one thin wrapper record per derived call unless the wrapper is a struct record — otherwise the derived surface costs the same as running the representation directly.
- The derivation is structure-only and the compiler enforces only the type, never the naturality law: a `Transform`/`CoTransform` that inspects, reorders, drops, or re-wraps bound values with altered state breaks the derived monad/applicative laws silently. Bound values untouched is what distinguishes the natural transformation from a `Map`, and the hazard has an interpreter-level twin in fusion-node slot order.

[DERIVING_LATTICE_RUNTIME_SURFACE]:
- The deriving interfaces form a subsumption lattice mirroring the trait lattice exactly: `Monad<Super,Sub>` already carries the functor/applicative derivations, `MonadIO<Super,Sub>` adds the IO-lift, `MonadUnliftIO<Super,Sub>` at the apex transitively re-exports every lower derivation — declaring the apex hands the wrapper the full chain in one declaration.
- The lattice is self-constraining — each deriving interface requires `Sub` to implement the corresponding base trait, so an under-powered representation is a compile error, not a runtime gap. Capability is a position in the chain, never a feature flag: stop at `Monad<Super,Sub>` and `LiftIO` is not in scope on that carrier; omit the `Fallible<E,Super,Sub>` derivation and no catch surface exists, so a non-failing carrier cannot accidentally swallow errors; `Maybe.MonadUnliftIO`/`Maybe.MonadIO` marker constraints let a stack whose inner monad lacks an IO floor still type-check, degrading derived IO operations to the absent-IO branch.
- The apex materializes the complete runtime vocabulary as default-interface members — `ToIO`, `MapIO`, `LocalIO`, `PostIO`, `TimeoutIO`, both `BracketIO` arities, `RepeatIO`/`RepeatWhileIO`/`RepeatUntilIO`, `RetryIO`/`RetryWhileIO`/`RetryUntilIO`, each bare and `Schedule`-parameterized — none written for the wrapper. The mechanism is `MapIO`: every such combinator is stated once against `MonadUnliftIO<M>` as expose-the-IO-layer, apply `IO<A> -> IO<B>`, re-wrap — so timeout, bracket, retry, uninterruptible, local cancel scope, and sync-context posting are carrier-polymorphic for every deriving wrapper, and a new carrier inherits resource safety and cancellation scoping with zero bespoke combinators.
- `MonadT<Super,Sub,M>` derives a transformer wrapper whose inner monad is covariant — one declaration serves every inner monad; the `MonadIO` derivation supplies both `LiftIO` overloads with the kind form delegating to the concrete, so the bridge from the bottom carrier into any wrapper is itself derived.

```csharp
public sealed class App :
    Deriving.MonadUnliftIO<App, ReaderT<Caps, IO>>,
    Deriving.Readable<App, Caps, ReaderT<Caps, IO>>
{
    public static K<ReaderT<Caps, IO>, A> Transform<A>(K<App, A> fa) => ((App<A>)fa).Run;
    public static K<App, A> CoTransform<A>(K<ReaderT<Caps, IO>, A> fa) => new App<A>(fa.As());
}
public sealed record App<A>(ReaderT<Caps, IO, A> Run) : K<App, A>;

static K<App, B> WithPolicy<B>(K<App, B> ma, Schedule s) =>
    ma.RetryIO(s).TimeoutIO(30 * seconds).BracketIO();   // all derived, no further code
```

[ENVIRONMENT_CAPABILITY_PROJECTION]:
- `Readable<Super, Env, Sub>` derivation yields `Asks`/`Ask`/`Local` over the wrapper with the environment traveling as a third type parameter through the same isomorphism; `Stateful<Super, Sub, S>` is the symmetric state-threading derivation over identical machinery. The capability-free degenerate makes the bound type itself the environment — when the representation reads `A`, `ask<A>()` returns the bound value, which is why a no-capability carrier still satisfies the reader interface without a capability record; that form cannot extend capability slots at composition time.
- Capability resolution is derived lookup: a capability is one witness implementation on the runtime record; `Has<M, VALUE>` exposes a single `static abstract Ask`, and the three-parameter `Has<M, Env, VALUE>` stores `Env.Ask` in a `public static readonly` field, amortizing static-virtual dispatch to one initialization — read the cached field, not `Env.Ask` per bind site. The value type disambiguates sibling slots, so the accessor surface scales with zero authored members.
- `Local<M, InnerEnv> : Has<M, InnerEnv>` adds exactly one operation — `With(Func<InnerEnv,InnerEnv>, K<M,A>)` — turning a readable slot into a scopable one; dependency-isolated sub-computation embedding is `With` plus the reader derivation, nothing more.
- `asks<RT,Cap>(rt => rt.Slot)` constructs a `readonly record struct` ask that the carrier binds directly without allocating a monad wrapper. `Eff.local(f, ma)` runs an inner-runtime computation as a projection of the outer runtime inside a fresh cancel scope — no capability leakage between scopes; `runtime<RT>()` lifts the runtime record as a bound value for composition-time inspection; `getState<RT>()` yields the `(RT, EnvIO)` pair when the execution environment itself must be examined.
- `WithRuntime<RT>()` is a carrier upgrade, not capability injection: it runs the inner reader against the zero-byte minimal runtime to an `IO<A>` and re-lifts under `RT` — the runtime value is still required at the final run. `localCancel` gives the inner scope a fresh source linked to the parent token over shared resources: regional cancellation cannot tear siblings, but an exception produced by inner cancellation still propagates outward — scope isolation contains signals, not exceptions.

[INTERPRETER_EXTENSION_DISCIPLINE]:
- The runtime is a trampoline, not a call stack: a synchronous loop over a catch-handler stack pattern-matches DSL nodes and yields to an async state machine only on a genuinely-pending awaitable — fully-synchronous graphs never allocate the state machine, and the synchronous appearance of a fast effect is structural (completed value-task inspection), not a bolted-on fast path.
- `IOTail` is the trampoline step: the loop replaces the current node with the tail without advancing a continuation, and `Map`/`Bind` on a tail node throw at definition time — illegal chaining is a build-time error at the construction site, not a runtime surprise.
- Subclassing the carrier directly throws at run time; the four legal extension bases split on terminal-versus-continuing and sync-versus-async (`InvokeSync`, `InvokeSyncIO`, `InvokeAsync`, `InvokeAsyncIO`) — the continuing bases return the next carrier, enabling two-phase dispatch where phase selection is synchronous and the body is an effect chain.
- A step expressed outside the sanctioned bases is invisible to the fusion ladder: the chain cannot fold into it, so every operation around it allocates unfused nodes — a custom effect step is one legal base node whose `Invoke` returns a fused continuation, never a bespoke wrapper.
- A node's `Invoke` must reconstitute its slots in exact source order — the fields encode an ordered expression, not a set; a reordering or value-dropping `Invoke` type-checks but silently violates monad associativity.
- Node equality and hashing are structural over the captured delegates, so two fused nodes are equal only when seed and every delegate are reference-equal: fused pipelines are not value-comparable in practice, and any cache keyed on a built carrier keys on identity, never on the logical effect it denotes — the fused node is an opaque execution plan.

[CONTINUATION_FUSION_LADDER]:
- A bound `Map`/`Bind` chain is not a linked list of one-operation wrappers: each composition operator self-rewrites into a wider specialized node reifying several fused operations as record fields — a closed family (`Pure`, `Map`, `Bind`, `BindMap`, `BindBind`, `BindBindMap`, `BindBindMapBind`) holding a seed plus two to four function slots, so a four-operation pipeline allocates one node. Fusion is a builder-time allocation collapse: run cost is identical, construction cost amortizes across the fused segment.
- The family is a deterministic state machine over (current node kind, incoming operation), and it saturates at four fused operations: past saturation a node stops widening and rewrites its terminal slot functionally — a saturated tail absorbs a new map as `x => f(tail(x))` and a new bind as `x => tail(x).Bind(g)` — bounding the type explosion deliberately. Map-after-map always composes into one function slot, and same-kind tails thread through the terminal slot, so the only node-allocating transitions introduce a new operation kind against an unsaturated node.
- Two parallel ladders share the transition graph — a sync family whose `Invoke` returns a carrier, an async family whose `Invoke` awaits. The bridge is condition-driven: a sync node runs its inner effect, and on a genuinely-pending task spills its already-fused slots into the async mirror node so no fusion is lost crossing the await; the async side carries one extra collapse the sync side lacks (bind-after-bind-map folds the map into the first slot, because the mapped result must re-sequence through a bind to await correctly).
- Cancellation-token access is its own two-node sub-ladder that never widens past arity two: the token node holds the env-reader plus one accumulated tail, deferring the token read to `Invoke` where the live environment token is projected and the fused tail applied — `IO.token` participates in this fusion; `cancelToken` and `IO.source` are plain lifts with no fusion path, so token reads in tight loops belong on the fusing form. `MonadIO<M>` exposes `Token`/`TokenSource` as static-virtual properties over the fusing nodes, so carrier-generic code reads the token without breaking abstraction.
- A whole effectful sequence fuses into one node carrying an iterator of pending effects plus a single `Next` continuation: `Map`/`Bind` rewrite only `Next`, and each step re-emits a node holding the unconsumed iterator suffix — sequencing is a self-feeding fused loop, never a materialized chain of binds.

```csharp
static IO<int> Pipeline(IO<int> seed) =>
    seed.Bind(a => Lookup(a))   // -> BindBind (seed-rooted)
        .Bind(b => Widen(b))    // -> in-place tail fold, no new node
        .Map(c => c + 1)        // -> BindBindMap
        .Bind(d => Persist(d)); // -> BindBindMapBind (saturated)
// further .Map/.Bind rewrite the terminal slot's closure, never a fifth node
```

[SCHEDULE_CURVE_DERIVATION]:
- The named backoff curves are projections of one primary generator, not parallel cases: constant is an index-blind map over the infinite stream, exponential is the indexed map `seed * factor^i`, and any custom curve is one indexed map — never a new case record. Only the recurrence-bearing curves (linear `seed * (1 + factor*n)`, fibonacci) are bespoke stateful iterators, because a pure index map cannot express prior-term dependence — the recurrences are the named exception to the derivation.
- The policy type carries the functor/monad surface directly — map, indexed map, filter, bind, query-expression forms — so policy composes, slices, and merges entirely in the duration domain before any effect touches it; attempt count is stream truncation, never a counter threaded through the effect.
- The algebra is a semigroup with no identity: there is no empty policy, and `Never`/`Forever` are convenience generators (`Never` runs empty; `Forever` yields zero infinitely; `Once` is `Forever.Take(1)`), not algebraic units — a fold over policies must seed with a real schedule, not an assumed zero.
- A zero duration is a "go now" signal — the absence of a gap — structurally distinct from stream exhaustion, which is the loop's exit; conflating them mistimes the final attempt: the zero element continues immediately, the stream end terminates.
- Builder surface: `spaced` (constant), `linear`, `exponential`, `fibonacci`, `upto` (cumulative wall-clock bound), `fixedInterval` (absolute boundaries; an overrunning action fires the next step immediately with no pile-up), `windowed` (sleeps to window boundaries; may fire multiple times per window), calendar alignment (`secondOfMinute`/`minuteOfHour`/`hourOfDay`/`dayOfWeek`), `TimeSeries(params Duration[])` for literal probe series — every wall-clock builder accepts an injectable `currentTimeFn`, making time-driven schedules deterministic under test.
- Duration is a double-milliseconds `readonly struct` with implicit conversions from `double`, `TimeSpan`, and the unit type, whose extension constructors and prelude unit constants make `50 * ms` and `2 * seconds` the literal spelling; `Duration.random(min, max, seed)` as a curve seed de-correlates parallel retry storms without the expansion transformer.

[SCHEDULE_OPERATOR_BOUNDING]:
- `|` on two schedules is union — min-of-two per step, running to the longer operand; `&` is intersect — max-of-two, running to the shorter; `+` appends sequentially. With a transformer operand, `|` and `&` instead apply the transformer — operator meaning shifts by operand type, and the asymmetry is load-bearing when reading composed policy. Transformers compose left-to-right (`f + g` is `g(f(s))`), any `Schedule -> Schedule` function lifts through the public transform factory, and a bare transformer converts implicitly to a schedule by applying itself to `Forever`.
- Bounding an infinite curve requires a truncating operator — an attempt-count transformer, a cumulative-delay gate, or intersection with a finite stream. Union runs to the longer operand by design, so unioning anything finite onto an infinite curve does not bound it: the merge that tightens magnitudes is exactly the merge that cannot terminate the stream.
- `recurs(n)` is `Take(n)` on the base — exactly n attempts; `repeat(n)` cycles the entire base schedule n times, each cycle with as many steps as the base contains — the naming inversion is load-bearing: count-bounded loops take `recurs`, bounded phase cycling takes `repeat`.
- The per-step cap and the cumulative gate stack into "never wait longer than X per gap, never more than Y in total" as two composed transformers; the cumulative gate bounds applied wall-delay, not attempt count, and terminates on the step before the threshold is crossed. `resetAfter` drives the cumulative-gated inner run inside an outer loop — an infinitely resetting bounded window; `intersperse` prepends a probe series before each step.
- The driver's zero-prepend means a curve always describes the gaps between attempts, never a delay before the first try; the first-delay strip transformer exists for schedules whose seed step must fire immediately.
- Jitter is absolute (`[min,max]` additive) or proportional (`delay * factor * rand`); the de-correlating transformer emits a `+jitter`/`-jitter` pair per base step — an expansion transformer that roughly doubles the stream, built for many parallel clients de-phasing retry storms — and its optional seed is the determinism seam: pinning it makes a jittered schedule reproducible under test.

```csharp
static readonly Schedule Backoff =
    (Schedule.recurs(6)
   | Schedule.exponential(50 * ms)
   | Schedule.maxDelay(2 * seconds)
   | Schedule.decorrelate(factor: 0.2, seed: 7)
   | Schedule.maxCumulativeDelay(30 * seconds))
   & Schedule.spaced(100 * ms);   // floors every gap; inherits the bounded length
```

[DRIVE_POLARITY_RESOURCE_CONTRACT]:
- Exhaustion polarity is the load-bearing difference between the two drivers: retry threads the last error (seeded with the empty error) and stream exhaustion is the final failure surfaced to the caller; repeat threads the last value and exhaustion is success. The predicate variants compose with the iterator walk, not the schedule — the schedule bounds how many attempts and how spaced, the predicate bounds whether the observed value or error warrants another step, and whichever fires first halts.
- The resource contract is fixed by which scope each driver re-enters per step: repeat wraps every iteration in an unconditional resource scope, reclaiming each iteration's acquisitions including the successful last one — a repeated computation can never acquire-and-return a live resource, the value escapes but its backing is already released; retry wraps each attempt in a fail-only scope — failed attempts release their handles, the successful attempt's resources survive to the caller's scope, and attempts never accumulate acquisitions.
- The inversion trap: interposing an unconditional resource scope between an acquisition and its retry driver discards the successful attempt's resource before the driver observes success — a resource-producing effect that must survive its retry loop is handed to the retry combinator directly; the driver supplies the correct fail-only scope itself.
- The schedule-free retry form recurses with no delay and no exhaustion — unbounded by construction; cap it with a count-bounded schedule rather than relying on the predicate alone.
- The fold driver reuses the same iterator walk threading an accumulator instead of discarding the bound value: an unbounded schedule plus a state predicate is a converge-until-stable loop bounded only by the data; a bounded schedule is a fixed-window sampler — cadence policy as a value, no timer, no mutable accumulator. The fold-while/fold-until overload families normalize every predicate projection to one named-tuple `(State, Value)` form, and schedule-absent overloads default to the infinite schedule.
- Inter-attempt delay is a non-blocking timed yield, never a thread sleep, and the recursion is trampolined inside the interpreter — an unbounded policy drives unbounded iteration without stack growth, which is exactly why an infinite backoff on the rail must be intersected with a bound rather than trusted to stop itself.

[SCOPE_PROJECTIONS_RUN_EDGE]:
- The environment tracks two orthogonal ownership bits — one for the cancellation source, one for the resource registry — set only for halves it allocated, each disposed at-most-once via compare-exchange; ownership and disposal are one predicate: a child sharing its parent's source can be disposed without tearing the parent's still-live half.
- The child-environment projections are permutations of one constructor over those two bits, each serving exactly one scoping intent: fresh-registry-fresh-source for full isolation, fresh-registry-shared-source for resource isolation under the parent's cancellation axis (the fail-only bracket's scope, so retry attempts share one cancellation), shared-registry-fresh-source for a regionally cancellable sub-tree over shared resources, plus a context re-capture projection that re-binds the synchronization context after crossing a thread boundary. The dominant scoping bug is projection mismatch — wrapping a sub-tree in the resource-isolated projection when the intent was independent cancellation leaves it bound to the parent token, and cancelling it tears the parent.
- Cancellation cascades one direction only — the inherited token is registered to cancel the child source, never the reverse — so the cancel-isolated projection is the safe unit for speculative or deadline-bounded work. A deadline is structurally the same projection plus an expiring source, so a timed region nests with an inner regional-cancel region as two child sources, the inner strictly shorter-lived; a non-cancellable parent token elides the registration allocation entirely.
- Timeout fires as a cancellation exception through the interpreter's catch stack — the result type is not failure-wrapped; modeling timeout as a typed failure means following with a catch on the cancellation shape or using the carrier-polymorphic timeout, and the clock starts when execution begins, not at construction.
- Uninterruptible replaces the token with the non-cancellable one — replaced, not linked — but does not isolate resource tracking: acquisitions inside the uninterruptible block register in the parent scope, so composing it with scoped acquisition requires an enclosing bracket to own the lifetime.
- The two env-mapping primitives differ only in release timing — both-paths versus failure-only — and every bracket, timeout, and fork scope builds on them; a DSL extension needing a specialized environment shape constructs it through these, never by mutating environment fields.
- The run edge collapses the rail: the safe forms return the failure-carrying result; the unsafe forms throw and are reserved for composition roots that have exhausted recovery; the no-execute form extracts the underlying IO for an IO-shaped boundary. Synchronous run is the async path blocked on — there is no dedicated sync execution path — and the token-accepting run overload allocates a fresh environment and linked source: prefer passing an existing environment to share the cancellation axis and resource scope across composed runs.
- Posting onto the captured synchronization context is a scope operation: no captured context degrades to a direct run; with one, the body marshals through the context while the poster awaits a signal under the ambient token — cancellation releases the waiter as cancellation rather than dangling, and a marshalled-body exception rethrows after the wait. The context is captured at environment construction, not at post time, so which thread runs a continuation or cleanup is answered by the environment, never by an implicit capture at the call site.

[RESOURCE_REGISTRY_LAW]:
- The registry keys tracked resources by bare reference identity over a closed family of release cases — sync-disposable, async-disposable, free-function. Acquire is idempotent for identity-equal handles (a second acquire of the same reference is silently dropped); two structurally-equal instances are two entries; a value-type resource boxes to a fresh key per acquire, so its release fires once per acquire call, not once per logical value.
- Bulk teardown enumerates the registry's trie in hash-derived order — no last-in-first-out, no insertion order. Teardown that must release a dependent before its dependency encodes the ordering inside a single composite resource's release arrow; relying on two separate registrations imports a stack-unwind mental model the registry does not provide — the sharpest correctness trap on the resource surface.
- Acquisition never consults the parent scope; only release-on-miss delegates upward — a child's registry is a strict local set whose bulk teardown cannot see parent entries, while an outer-acquired handle remains explicitly releasable from inside through the upward walk.
- Every release arrow runs on a freshly-built environment carrying a non-cancellable token: cleanup always runs to completion under cancellation, and a release arrow that observes a token sees the non-cancellable one — long token-aware work does not belong in a disposer.
- A scoped acquire registers the instant the acquire effect resolves — an asynchronous acquire registers only after its task completes, eliminating the window where an exception between acquisition and registration would leak — and the release sits until the nearest enclosing auto-freeing scope disposes: a bare scoped acquire with no surrounding bracket holds its resource until the outermost run completes, so resource pressure under a long-running root is almost always eager acquires with no intervening scope.
- The unconditional and fail-only scopes differ only on the success path — one frees, the other leaves the success subtree alive — and that divergence is the entire escape mechanism by which an acquiring computation hands a live handle to its caller's scope. The finalizer combinator sequences its cleanup before the forward continuation on both paths and runs synchronously when the protected operation throws; its operator spelling composes it onto the rail as a value, and the combined use/catch/finalize bracket arity assembles recovery and cleanup in one frame rather than nesting two catch nodes.

[FORK_RACE_TEARDOWN]:
- A fork projects the scope lattice onto a pool thread: a private source cancelled by the parent token, and a child resource tracker parented to the forking scope's tracker — pre-fork handles are reachable and releasable from inside through the upward walk, while anything acquired inside is reclaimed by the child and never leaks upward.
- The fork handle is a pure value pair of cancel and await effects, so awaiting can happen in a different scope after the spawning region unwinds — the fork's lifetime is bounded by its private source, not the spawner's stack frame. Await observes the caller's environment token: if the caller cancels first, the caller's cancellation propagates, not the fork's. Await is not failure-wrapped, and re-awaiting one handle in a retry loop races an already-completed task — it can surface a disposed-object failure after the fork's cleanup has run.
- The bare fork launches on a dedicated long-running thread, bypassing the pool — excess OS-thread pressure for short-lived work; lightweight parallelism over a collection belongs on the traversal-forked await-all path, which does not request dedicated threads.
- Race teardown is asymmetric across the two ingress shapes, and the asymmetry decides loser cleanup: racing raw effects runs every competitor under one shared cancel-isolated child environment and cancels that source the moment the winner lands — losers torn down deterministically; racing already-running fork handles awaits under the caller's environment and cancels only its local await scope — losers keep running to their own completion, and deterministic teardown requires explicitly cancelling each loser through its own cancel handle inside a finalizer after the race resolves.

[CATCH_POLICY_FRAME_DISCIPLINE]:
- A recovery is a reified predicate-handler pair, not control flow: ordinary data that can be named, returned from a policy factory, parameterized, and selected from a table, retargeting its bound type by mapping the handler while reusing the predicate. The pair does not compose with siblings directly — there is no pair-to-pair operator — so a cascade is assembled at the protected effect, each successive operator application wrapping the composite to its left; the first matching predicate consumes the failure and later handlers are never consulted for it.
- The recovery matrix is two orthogonal axes chosen by combinator, never by branches inside a handler: which failures — all, by code, by error identity, by leaf subtype narrowing, expected-only, exceptional-only — crossed with which multiplicity over a matching batch — fail-fast bind-fold versus visit-every-member monoid-fold merging handler outcomes. The predicate chooses membership, the combinator chooses multiplicity, and single-failure and batch paths unify under one predicate.
- Catch is a stack discipline inside the interpreter: the protected operation pushes a handler frame and a pop node removes it on normal completion before the forward continuation runs — handler scope is exact, not lexical-approximate, and a handler structurally cannot catch a failure arising after its protected region. A recovered branch re-enters the saved forward continuation: recovery is not a terminal, which is what makes mid-pipeline recovery composition-safe rather than a dead end forcing rail reconstruction.
- The handler frame lifts any escaped exception through the canonicalizing admission before testing the predicate — a cancellation thrown deep in foreign code arrives as the canonical cancelled value, an aggregate arrives pre-flattened, a wrapped rail error arrives unwrapped — so predicates are written against codes and facet partitions, never exception types, and stay stable as the throw site changes.
- Frames have a cost: each catch node pushes one runtime stack frame, and alternative-chaining builds a left-linear chain of catch nodes each inspecting the prior failure — recovery on choice is catch-on-any-error, so stacking alternatives in tight loops allocates per attempt. The empty effect propagates the canonical none-error through handlers that filter on it; the IO-rail guard emits exactly that none-error on its false branch while the typed guard emits a named domain error — the two guard forms are not interchangeable across IO-typed and carrier-typed query expressions.
- A recovery handler may lift its recovery path to raw IO when the fallback does not belong to the outer runtime context; the result re-lifts into the carrier automatically — boundary recovery without surrendering the carrier.

```csharp
static CatchM<Error, M, A> Transient<M, A>() where M : Fallible<Error, M>, Monad<M> =>
    @catch(TransientCode, e => Recompute<M, A>(e));

K<M, A> recovered =
    protectedEffect
  | Transient<M, A>()                       // policy value from a factory
  | expected(e => Fallback<M, A>(e))        // facet-partition predicate
  | @catch(_ => true, e => Surface<M, A>(e));   // first match consumes; order is policy
```
