# State-Receipts: Core Findings

[ATOM_INTERNALS]:
- `Atom<A>` stores its value in a `private volatile object` field; `Box<A>` dispatches at static construction to either a CIL-emit path (unboxed field projection for value types via `DynamicMethod`) or an identity cast (reference types), falling back to a plain cast on platforms where `ILCapability.Available` is false. Value-type reads unbox from `Box<A>.Value` — no allocation. Value-type writes always allocate a new `Box<A>` wrapper. Reference-type reads and writes go through identity without allocation.
- `Box<A>.New` for reference types compiles (via CIL emit) to an identity function (`ldarg.0`, `ret`) — it returns the object reference unchanged, allocating nothing. When `Atom<A>.Swap(f)` calls `Box<A>.New(val2)` for a reference-type `A`, `obj2` is the same object reference as `val2`. If `f` returns the exact same reference as `val` (an idempotent or no-op swap), then `obj2 == obj` (same pointer), the `Interlocked.CompareExchange` succeeds (comparing the current atom value pointer against itself — absent a concurrent writer that changed the pointer between read and CAS), and `Change` fires with the unchanged value. There is no mechanism to detect an identity swap and suppress the event. Observers that must deduplicate must compare the new value with the expected value themselves, or use a validator that returns `false` when `f(val)` produces the same logical state.
- `Atom<A>.Swap` returns the pre-swap `val` when the validator rejects, and returns the new `val2` on CAS success. Because an idempotent reference-type swap sets `obj2 = obj`, the return value `val2 == val` in both the rejection path and the identity-swap path — the caller cannot distinguish them. Use `SwapMaybe` to make the no-op path explicit: return `None` when the intended update is not applicable, and the atom and its observers are left undisturbed.
- `Swap` is a CAS spin loop: it reads `value`, calls `f`, calls `validator(val2)`, and then `Interlocked.CompareExchange`s; on CAS failure it calls `spinWait.SpinOnce()` and retries. `f` will be called multiple times under contention — any side effect inside `f` is repeated. `SwapMaybe` breaks the loop early when `f` returns `None`, leaving the atom unchanged and returning the current value without retrying.
- The `Change` event fires synchronously on the calling thread immediately after the CAS succeeds and before `Swap` returns. There is no dispatch queue; a slow handler throttles the swapper directly.
- `Atom<A>(A value)` (no validator) returns `Atom<A>` directly. `Atom<A>(A value, Func<A, bool> validator)` returns `Option<Atom<A>>`; `None` means the seed value failed validation — the atom is never constructed.
- `atomIO<A>(A value, Func<A, bool> validator)` returns `IO<Atom<A>>` and throws `Exceptions.ValidationFailed` inside the `IO` boundary when the validator rejects the seed — the `IO` resolves to a failed execution, not `IO<Option<Atom<A>>>`. The two construction paths are not symmetric: the synchronous factory returns presence or absence in the type; `atomIO` returns failure on the error rail. Callers that want to handle rejection without an error rail must use the synchronous factory.
- `Atom<M, A>` carries metadata as a `private readonly M metadata` field. It exposes `Swap(Func<M, A, A>)`, `Swap(Func<M, A, Option<A>>)`, and their `IO`-lifting counterparts. The metadata field is not publicly readable; it flows only into these `Swap` overloads as the first argument — the validator (`Func<A, bool>`) receives only the proposed new `A` value and never receives `M`. A requirement to update the context signals that the context belongs inside the state value.
- `SwapIO(Func<A, A>)` and `SwapMaybeIO(Func<A, Option<A>>)` are `IO.lift` wrappers over synchronous `Swap`; they do not defer or schedule the CAS. The spin loop runs at `IO` evaluation time on the evaluating thread.
- `Atom<A>.ValueIO` has a `set` accessor: `value.Bind((A v) => SwapIO((A _) => v))`. `Bind` on an `IO` is a lazy combinator — it returns a new `IO<Unit>` describing the swap and discards it without evaluation. Assigning `atom.ValueIO = someIO` is a no-op; no state change occurs. The correct write pattern is `Swap(f)` or `SwapIO(f)`. The same structural hazard applies to `Atom<M,A>.ValueIO`.

[ATOM_COLLECTION_VARIANTS]:
- `AtomHashMap<K, V>` scalar mutations (`Add`, `Remove`, `SetItem`, `SwapKey`, etc.) check `this.Change != null` inside the CAS spin loop before choosing between the `WithLog` (change-tracking) trie path and the plain (no-log) path. When no observer is subscribed, the plain path runs and no `Change<V>` is allocated. A single subscribed handler forces the audit-log path on every scalar operation.
- The bulk `Swap(Func<TrackingHashMap<K,V>, TrackingHashMap<K,V>>)` path always constructs a `TrackingHashMap<K,V>` wrapper regardless of whether `Change` is subscribed, since the observer check happens in `AnnounceChanges` after the CAS rather than before construction. The wrapper accumulates `Change<V>` values using `Change<V>.Combine` across mutations to the same key, so the patch delivered to observers reflects the net delta — not intermediate states. For workloads that never subscribe observers, the constant `TrackingHashMap` allocation overhead on every bulk `Swap` call is the trade-off cost of the unified API.
- `HashMapPatch<K,V>` uses two internal representations: a scalar op stores a `(key, change)` pair; a bulk op stores a `TrieMap<K, Change<V>>`. The `Changes` property builds `HashMap<K, Change<V>>` on demand — `HashMap.Empty.Add(key, change)` for scalar ops, a `HashMap` wrapping the trie for bulk ops. There is no public scalar accessor and no way to branch before calling `Changes`; calling it on every event is unavoidable. High-frequency scalar-op observers that want minimal allocation should avoid `Changes.Count` and instead enumerate `Changes` directly, or subscribe to the typed scalar-change delegate variants that receive `(K key, Change<V> change)` directly without a `HashMapPatch` allocation.
- `Change<A>` is an `abstract class` with four sealed subtypes: `NoChange<A>`, `EntryAdded<A>`, `EntryRemoved<A>`, and `EntryMapped<A, B>` (implementing both `EntryMappedFrom<A>` and `EntryMappedTo<B>`). It implements `Monoid<Change<A>>`. Composition laws: `NoChange + y = y`; `x + NoChange = x`; `x + Removed = Removed`; `Removed + Added(v) = Mapped(oldValue, v)`; `Added(x) + Mapped(→y) = Added(y)`; `Mapped(from, x) + Mapped(→y) = Mapped(from, y)`. A bulk `Swap` that inserts then updates the same key delivers exactly one `EntryAdded` to observers — the `Added + Mapped = Added` law absorbs the in-bulk update, so the net change is still an addition. `EntryMapped` arrives only when a pre-existing key is updated (net `Removed + Added` or `Mapped + Mapped` compositions). Code that distinguishes insertion from in-place update by pattern-matching `EntryAdded` must use scalar `SwapKey` calls to receive per-operation atomics. Pattern-matching as `EntryMappedTo<V>` extracts only the new value; `EntryMappedFrom<V>` extracts only the old value — use `EntryMappedTo<V>` when the prior value is irrelevant to avoid binding to the `From` type parameter when a type-changing mutation is possible.
- Every `AtomHashMap` mutation path checks `this.Change != null` inside the CAS spin loop, so a newly subscribed observer will be seen on the next retry if the CAS fails due to contention. The race window where a newly subscribed observer may miss an event is bounded to the interval between a successful `this.Change != null` check and the CAS `CompareExchange` that commits the plain-path mutation. For scalar ops this window is a single fast trie operation; for range ops (`AddRange`, `TryAddRange`) the pre-CAS work scales with range size, widening the window proportionally. Observers that require guaranteed delivery from their subscription point must snapshot `Items` after subscribing to initialize their state.
- `AtomSeq<A>.Swap(Func<Seq<A>, Seq<A>>)` runs the same CAS spin against the internal `ISeqInternal<A>` field. There is no tracking wrapper and no diff; `AtomSeq` fires no change event.
- `AtomQue<A>` fires distinct `Dequeued` and `Enqueued` events (`AtomDequeuedEvent<A>` and `AtomEnqueuedEvent<A>` delegates) after each successful CAS. `Clear()` fires `Dequeued` for every element present at the moment of CAS success by iterating the captured pre-clear queue; a concurrent `Enqueue` after the CAS may add an element that `Clear` does not notify. Observers that must account for every element must drain with `Dequeue` in a loop rather than `Clear`. There is no `Swap(Func<Que<A>, Que<A>>)` — multi-step conditional operations require a higher-level `Atom<Que<A>>` or a `Ref<Que<A>>` inside `atomic()`.
- The static `AtomHashMap` class exposes `ToAtom<K,V>(this HashMap<K,V>)` as the canonical upgrade from an immutable snapshot to a concurrent cell.

[REF_AND_STM]:
- `Ref<A>` is a `sealed class` backed by an internal `long Id` keyed into a process-global `AtomHashMap<EqLong, long, RefState>` (`STM.state`). The `Value` setter calls `STM.Write(Id, value)`, which writes into the current thread's `Transaction` or throws `InvalidOperationException` if no transaction is active. `Ref<A>.Swap(Func<A,A>)` compiles to `Value = f(Value)` — it reads and writes via the property, so it requires an enclosing `atomic()` and is not independently thread-safe.
- `atomic<R>(Func<R>, Isolation)` checks `STM.transaction.Value` (an `AsyncLocal<Transaction?>`) first: if a transaction is already active it invokes `op()` and returns immediately — nested `atomic` calls join the outermost without creating a new retry scope.
- `Isolation.Snapshot` (default) detects write-write conflicts at commit via version comparison (`s[write].Version == refState.Version`), throwing `ConflictException` and retrying on mismatch. Blind writes are prevented. What is skipped is read-set validation (`ValidateReads`) — so non-repeatable reads and phantom reads remain possible, but two concurrent transactions writing the same `Ref` do conflict. The correct characterization is optimistic write-lock, not snapshot isolation in the database sense. `Isolation.Serialisable` additionally validates the read-set (`transaction.reads`, a `HashSet<long>` accumulated unconditionally on every `STM.Read` call regardless of isolation mode) before commit; any version change in the read-set triggers `ConflictException` and a full retry. Under `Snapshot`, the read-set accumulates with no commit-time benefit — prefer `Serialisable` only when read-set validation is required.
- The entire commit — validation, write application, and commute application — executes inside a single `state.SwapInternal(delegate ...)` CAS spin loop. Two concurrent `ValidateAndCommit` calls compete on this CAS; one wins, the other re-spins with the updated global state snapshot. This is the only global serialization point for STM commits.
- `commute<A>(Ref<A>, Func<A, A>)` stages the function in the transaction's `commutes` list rather than the `writes` map. At commit time, `CommitCommutes` runs after `CommitWrites` against the latest committed global state `s` — so commuted functions apply against values already advanced by `CommitWrites`. A `Ref` may appear in both `writes` and `commutes`; the write sets the version-verified value, then the commute applies `f` to that result. `RefValidationFailedException` is not retried inside `CommitCommutes` — it escapes `atomic()` directly.
- `Ref.Value` reads outside `atomic()` bypass read-set tracking — `STM.Read` checks `transaction.Value != null` first and reads directly from `state.Items[id].UntypedValue` when no transaction is active. The `Items` field is a `volatile` trie reference, so direct reads are visibility-safe.
- `STM.transaction` is `AsyncLocal<Transaction?>`. `Task.Run` creates a new `ExecutionContext` by capturing (copying) the current one, which includes all `AsyncLocal` values. A `Task.Run` launched inside `atomic()` therefore inherits the current `Transaction` reference and can call `ref.Value = x` without throwing — `STM.Write` sees the non-null transaction. The hazard is not a throw but unsynchronized mutation of the shared mutable `Transaction` object (`state` trie and `writes`/`reads` sets), which are not thread-safe. Concurrent writes from two threads both holding the same captured `Transaction` produce undefined behavior at commit, not an exception at write time. Code that uses `Task.Run` inside `atomic` must ensure the spawned task does not access `Ref` values through the inherited transaction.
- `Ref<A>` declares `implicit operator A(Ref<A> value)`, which calls `value.Value`, which calls `STM.Read(Id)`. Inside `atomic()`, `STM.Read` unconditionally adds `Id` to `transaction.reads`. Any code path that implicitly converts a `Ref<A>` to `A` — passing it to an overloaded method accepting `A`, using it in a LINQ expression expecting `A`, composing with operators that accept `A` via implicit conversion — silently tracks that read in the current transaction under `Serialisable` isolation. Unexpected `Ref`s appear in the read-set, broadening the conflict surface. Explicit `ref.Value` reads carry the same tracking but are visible at the call site.
- `Ref<A>.ValueIO` has a `set` accessor: `value.Map(delegate(A v) { STM.Write(Id, v); return unit; })`. `IO.Map` is a lazy combinator — it returns a new `IO<Unit>` describing the write and discards it without evaluation. Assigning `ref.ValueIO = someIO` is a no-op; no state change occurs. The correct write pattern inside `atomic()` is `ref.Value = val` or `ref.Swap(f)`.
- `Ref<A>` has a finalizer that calls `STM.Finalise(Id)`, which calls `state.Remove(Id)` on the process-global `AtomHashMap`. If a `Ref<A>` becomes unreachable outside a transaction — for example, constructed inside an `atomic` lambda and never captured externally — the GC can finalize it between the transaction's snapshot read and `ValidateAndCommit`. `CommitWrites` then looks up the now-removed `Id` via `s[write]` on the global-state trie, which throws `ArgumentException("Key doesn't exist in map")` for missing keys. This exception is not `ConflictException` and is not caught by the retry loop — it escapes `atomic()` directly and cannot be retried. Ensure all `Ref<A>` instances used inside `atomic` are strongly referenced from a live root outside the lambda for the full transaction duration. The hazard compounds with `async`/`await` inside `atomic`: a `Ref<A>` captured only as a local in an async method may become eligible for finalization when the compiler splits the state machine and the local is not live in a suspension frame. Capture `Ref<A>` instances in fields or in a closure that also captures a live root to guarantee strong retention.
- `RefValidationFailedException` escapes the STM retry loop. A `Ref` validator that rejects a proposed commit is a hard abort of the `atomic` block, not a retry trigger. Use validators only for invariants that must never be violated; skippable transitions belong in the `commute` or `Swap` function logic. The same hard-abort applies inside `CommitCommutes` — a commuted function whose result fails validation escapes without retry.

[MEMO_SHAPES]:
- `Memo<A>` uses a three-state `int` field (`0 = unloaded`, `1 = loading`, `2 = loaded`) with a `SpinWait` loop and `Interlocked.CompareExchange` to execute the acquire function exactly once. On exception, `state` resets to `0` and the exception rethrows — the memo becomes re-acquirable. A second thread spinning while `state == 1` spins indefinitely with no timeout; the acquire function must not block on any memo-protected resource.
- `Reset()` transitions `state` from `2` to `1` via `Interlocked.CompareExchange`, then writes `value = default(A)` and `state = 0` as plain field writes without memory barriers. On weakly-ordered hardware (ARM64), the CPU may reorder `state = 0` before `value = default`, making a thread spinning on `state != 0` observe `state == 0` before `value` is cleared. The thread that wins the loading CAS evaluates `acquire()` and overwrites `value` with a fresh result, making the stale non-null `value` invisible in the loaded outcome. The hazard is benign for idempotent acquire functions — the stale value is overwritten — but any `acquire` function with non-idempotent side effects (prohibited by the design contract) would have observable consequences. `Reset()` is a no-op when `acquire == null` (literal-value memos constructed via `Memo(A value)` are permanently fixed).
- `Clone()` constructs a new unloaded `Memo<A>` with the same acquire delegate, leaving the original's loaded state untouched; use `Clone` when multiple independent acquisition lifecycles share the same computation — each clone acquires independently without coordinating through the original.
- `memo<A, B>(Func<A, B>)` (function memoization) wraps a `WeakDict<A, B>` — a `ConcurrentDictionary` whose values are `WeakReference<OnFinalise<B>>`. `OnFinalise` removes its own dictionary entry in its finalizer. Cache entries are collected when no strong reference to the computed value exists outside the cache, preventing unbounded growth for short-lived keys. If the caller does not retain the result, `memo<A,B>(f)(key)` may call `f(key)` again on the very next invocation — the per-key `object` lock in `syncMap` prevents double-computation within one GC cycle but provides no cross-GC-cycle guarantee. The lock object is obtained via `syncMap.GetOrAdd(key, new object())` and then removed via `syncMap.TryRemove` immediately after the protected block; a concurrent caller that obtained the same lock before the `TryRemove` completes is not excluded — only callers that arrive after `TryRemove` and call `GetOrAdd` again get a fresh lock object. This design is correct for idempotent computations and incorrect for non-idempotent ones.
- `memoUnsafe<T, R>(Func<T, R>)` uses a plain `ConcurrentDictionary<T, R>` with no weak reference — values are held permanently. Both variants use a per-key `object` lock via `syncMap.GetOrAdd(key, new object())` to prevent double computation, with the same post-protected `TryRemove` pattern.
- `memoK<F, A>(Func<K<F,A>>)` constructs a `Memo<F, A>` in unloaded state; the `~` operator forces acquisition by calling `.Value` and returning the `K<F, A>`. The `~` operator caches construction of the `K<F, A>`, not evaluation of it — if `K<F, A>` is itself lazy, it can still be invoked multiple times; force only at execution time, never at composition time.

[STATE_CELL_DISCIPLINE]:
- `Atom<T>` owns independent, single-ownership state. `Ref<T>` earns its STM overhead only when two or more `Ref`s must transition atomically; a single `Ref` inside `atomic(() => ...)` is equivalent to `Atom.Swap` in guarantees but adds transaction overhead and requires explicit `atomic` wrapping at every write site.
- The `Atom<M, A>` metadata form serves cases where stable external context must influence swap behavior without living inside the state value — a capacity limit, a schema version, or a generation token. The metadata flows through `Swap(Func<M, A, A>)` and `Swap(Func<M, A, Option<A>>)` as the first argument; the validator (`Func<A, bool>`) receives only the proposed new `A`, not `M`. A requirement to validate against the metadata is satisfied by encoding the invariant in the swap function itself (return `None` via `SwapMaybe` when the new value violates the context-dependent condition). A requirement to update the metadata signals the context belongs inside the state value.
- `AtomHashMap` over `Atom<HashMap<K,V>>`: `AtomHashMap` provides key-level mutations (`SwapKey`, `AddOrUpdate`, `FindOrMaybeAdd`) that `Atom<HashMap>` cannot express without replacing the whole map on every operation. It also emits a typed `HashMapPatch<K,V>` diff; `Atom<HashMap>` fires `Change` with only the new whole-map value. Use `AtomHashMap` when key-level concurrent mutations are common or when change diffs drive observer logic.
- A fact stream folded with `slot` and `kind` metadata lives in `Atom<Seq<FactRecord>>` or `AtomSeq<FactRecord>` where `FactRecord` carries the kind discriminant and all evidence fields. For append-only streams under contention, `AtomSeq.Add(x)` runs an independent CAS per element; `AtomSeq.Swap(s => s.Add(...))` batches multiple adds into one CAS.
- Typed algorithm receipts — carrying solver path, sampling parameters, spectral or mesh evidence, status flags — belong to `Atom<ReceiptRecord>` rather than an operational fact stream. Fact streams are accumulated observations projected by fold; receipts are evidence of a specific completed computation that a consumer pattern-matches for routing, replay, or audit.

[VALIDATOR_FIRST_WRITER_WINS]:
- The validator on `Atom<A>` runs on every CAS attempt, not just the first, receiving only the proposed new value. A first-writer-wins pattern cannot be expressed using a validator alone: the validator sees only the candidate value `val2`, never the current value `val`, so it cannot detect whether the atom's state has already changed away from a sentinel. The correct mechanism is `SwapMaybe`: the swap function receives `val`, inspects whether it still holds the expected sentinel, and returns `None` to abort (leaving the atom unchanged) or `Some(newValue)` to proceed. `SwapMaybe` returning `None` exits the spin loop immediately without retrying — the rejection is final and the atom is undisturbed.
- A rejecting validator causes `Swap` to return the pre-swap snapshot immediately without retrying — the spin loop exits on `!validator(val2)`, not on a failed CAS. A rejecting validator is called exactly once per `Swap` invocation regardless of contention.

[TRIE_STRUCTURAL_SHARING]:
- The CHAMP trie underlying `HashMap<K,V>`, `AtomHashMap<K,V>`, and `HashSet<K>` threads an `(UpdateType Type, bool Mutate)` env tuple through every node `Update` call. `Mutate: true` is passed through the `TrieMap(IEnumerable<(K,V)>)` and `TrieMap(ReadOnlySpan<(K,V)>)` constructors for batch construction, and also pervasively via `AddOrUpdateInPlace` — used by change-log accumulation (`EmptyForMutating` trie in `*WithLog` operations) and by `STM.Transaction.WriteValue`. Every public API call on a completed, externally-visible `HashMap` or `AtomHashMap` passes `Mutate: false`, which forces path-copy on every update. The immutable sharing guarantee for consumer-facing maps is enforced by this flag at each node rewrite; the `AddOrUpdateInPlace` path is only safe because it operates on freshly-allocated tries that have not yet escaped.
- A single-key update on a map with trie depth `d` allocates `d` new `Entries` nodes, each carrying a freshly cloned `Nodes[]` array. Trie depth is bounded by hash collision depth: the `Sec` struct advances in 5-bit increments (mask = 31, `Next()` adds 5 to `Offset`), and `Merge` creates a `Collision` node when `section.Offset >= 25` — meaning at most 6 routing levels (offsets 0, 5, 10, 15, 20, 25) before collision. Per-key update allocation scales with trie depth, not map cardinality.
- The `Collision` node type handles identical-hash keys with a flat `(K,V)[]`. `Collision.Update` for an existing key calls `TrieMap.SetItem(Items, num, change, false)` — the fourth argument is a hardcoded `false`, not `env.Mutate`. The `env` tuple is not threaded into any array-write path inside `Collision`; every collision-bucket write (both existing-key replacement and insertion) allocates a new array. Maps that use a custom `Eq<K>.GetHashCode` collapsing many keys to the same hash degrade to linear scan and full clone on every update, losing all path-copy sharing benefit for those key families.

[RECEIPT_MECHANICS]:
- Operational fact streams use a single `FactRecord` shape carrying a `kind` discriminant (a `[SmartEnum]` or closed union case), a `slot` identifier (which subject the fact belongs to), a timestamp, and a payload union. All mutation buckets — adds, updates, removals, errors — are cases of the same `kind` field rather than parallel record types. The `Atom<Seq<FactRecord>>` or `AtomSeq<FactRecord>` cell is the accumulator; the `kind`-discriminated `Seq<FactRecord>` is the monoidal surface.
- Fact streams are `Monoid` by concatenation: `Seq.Empty` is the identity, `s1 + s2` is ordered append. Derived projections are pure folds: `facts.Filter(f => f.Kind == K.Added).Map(f => f.Payload)` yields additions; `facts.GroupBy(f => f.Slot).Map(g => g.Last())` yields latest-per-slot state. Neither projection mutates the stream. This means a single `Atom<Seq<FactRecord>>` simultaneously supports replay (fold from beginning), current-state projection (fold with last-wins), and audit (full chronological stream) without separate state cells.
- Typed algorithm receipts are not monoids and not streams — they are single evidence records produced by one completed computation. A `ReceiptRecord` carries fields specific to the algorithm: solver path (which branch of a numerical algorithm ran), sampling parameters, mesh or spectral measurements, residual, iteration count, and a status discriminant. The consumer pattern-matches on `status` for routing and on solver-specific fields for audit. Because receipts are the product of one computation, `Atom<ReceiptRecord>` holds the most-recent result; historical receipts require `Atom<Seq<ReceiptRecord>>` if replay is needed.
- The design pressure that distinguishes the two: fact streams answer "what happened and when" and derive current state by fold; receipts answer "how did this specific computation resolve" and are evidence for a single event, not a projection surface. Collapsing receipts into a fact stream loses the typed solver/sampling evidence fields that only apply to one algorithm's output. Collapsing a fact stream into a receipt loses the ordering and temporal projection capability. The `kind` discriminant on `FactRecord` is categorical; the `status` field on a receipt is algorithmic provenance.
- Receipt combination across multiple computation runs uses a `Semigroup` over status: `Fin.Succ(receipt)` folds over a `Seq<ReceiptRecord>` with `Fin.traverse` to produce either `Seq<SuccReceipt>` (all succeeded) or `Fin.Fail(firstError)`. This is structurally identical to `Validation` accumulation but ordered — the first failure halts. When partial success matters, the fold is over `Seq<Fin<ReceiptRecord>>` with the caller choosing between `partition` (split successes and failures) and `sequence` (fail-fast).
- Derived projections from fact streams are pure functions, not additional state cells. A projection that needs to be re-queried frequently can be memoized via `memo<Seq<FactRecord>, ProjectedResult>(project)` against a snapshot — the `WeakReference` semantics in `memo` mean a snapshot with no live references outside the cache is collected, so the memoization table does not grow without bound as streams evolve.

[SEQ_INTERNAL_STRUCTURAL_COSTS]:
- `SeqConcat<A>` (produced when `AtomSeq.Concat(Seq<A>)` merges two segments) computes `Count` as `ms.Sum(s => s.Count)` — an O(segments) traversal each time `Count` is accessed. `AtomSeq<A>.Count` delegates to the internal's `Count`, so reading `AtomSeq.Count` on a concat-heavy sequence is not O(1). Observers that call `Count` per event on a frequently-concatenated `AtomSeq` should maintain a separate `Atom<int>` counter.
- `SeqConcat<A>.Add(value)` calls `ms.Last.ValueUnsafe()?.Add(value)` and rebuilds the `ms` list via `ms.Take(ms.Count - 1).Add(value2)`. `ms` is itself a `Seq<ISeqInternal<A>>`, so `Take` and `Add` on it traverse segments. Workloads that interleave bulk `Concat` with per-element `Add` should batch adds via `AtomSeq.Swap(s => s.Add(...))` inside one CAS rather than interleaving individual calls.
- `SeqStrict.Append(right)` has a single-use in-place write gate: `Interlocked.Exchange(ref addDisallowed, 1)`. The first call that wins can copy `right.data` directly into the pre-allocated suffix space (if capacity allows), sharing the backing array; the result is `new SeqStrict<A>(data, start, count + right.count, consDisallowed: 1, addDisallowed: 0)`. The second call against this result finds `addDisallowed == 0` and can again attempt in-place widening — the gate was not set on the returned wrapper. Only the original `SeqStrict` that lost its gate is blocked. `CloneAdd` and `CloneAddRange` both construct fresh instances with `(consDisallowed: 0, addDisallowed: 0)`, so every path through the failed-gate branch also restores the gate for future callers on the new array.
