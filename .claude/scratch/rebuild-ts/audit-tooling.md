## [00]-[SIXTEEN_LAW_STATEMENT]

Read of `README.md` + all eleven routed pages (`language`, `derivation`, `values`, `computation`, `shapes`, `surfaces-and-dispatch`, `rails-and-effects`, `services-and-layers`, `concurrency`, `streams`, `boundaries`). The law set, by name, with its bearing on this audit:

FLOW — (1) `EXPRESSION_SPINE`: domain logic is expression-shaped on `Effect<A,E,R>`; `let`-rebind / `push` / `throw` in domain flow are defects — **no mechanical gate exists**. (2) `BOUNDARY_ADMISSION`: decode once at the edge, interior never sees `null`/`undefined`/sentinels, absence is `Option` — absence-leakage partly rides `strict`/`exactOptionalPropertyTypes`/`noUncheckedIndexedAccess`; "domain absence routed to `Option` by lint" has **no rule**. (3) `CAPABILITY_CHANNEL`: dependency travels the `R` channel as a `Tag`, satisfied only by Layer — the unprovided-Tag-holds-`R`-above-`never` compile error is the one structurally-gated law; module-level live singleton / ambient import promotion is ungated.

SHAPE — (4) `SCHEMA_AUTHORITY`: Schema owner is the single shape authority; a parallel interface/type/DTO is the named defect — **promotion-law target, unrealized**. (5) `SHAPE_BUDGET`: one runtime authority per concept, no standalone brand export, 3+ parallel shapes is the collapse trigger — **unrealized**. (6) `DEEP_SURFACES`: one/two exports, `_`-prefixed interior, terminal `[EXPORTS]` block, no barrels — barrel ban is a **promotion-law target, unrealized**; exports-map hides the interior but does not forbid an in-package barrel. (7) `MODAL_ARITY`: one entrypoint discriminates on input shape, `Function.dual` not a pair, arity/suffix twins are the defect — **promotion-law target, unrealized**. (8) `ANTICIPATORY_COLLAPSE`: shape for the family; review-only, unenforceable mechanically.

DERIVATION — (9) `VOCABULARY_VALUES`: one `as const` table `satisfies` its contract, secondary surfaces derive; hand-union/`enum` paralleling a vocabulary is the defect — `enum` rides `erasableSyntaxOnly`, the parallel-union case is ungated. (10) `DERIVED_TYPES`: types computed from one anchor; `isolatedDeclarations` forces export annotations, the runtime-literal-restating-a-type case is review-only. (11) `SEMANTIC_NAMING`: canonical term / role / tense; review-only.

MATERIAL — (12) `LIBRARY_DEPTH`: Effect ecosystem is stdlib; `Map`/`Set`/`Promise`/`Date`/`null` survive only at the FFI seam — stdlib-in-domain has **no rule** (the gauge bans named modules, not global built-ins). (13) `INLINE_COMPOSITION`: wrapping/decoration attaches at the owner declaration; review-only.

INTEGRATION — (14) `ROOT_REBUILD`: weave into the owner, no shims/compat aliases/barrels/migration surfaces — barrel + migrator partly gated, compat-alias ungated. (15) `ONE_HOP_RESOLUTION`: one hop to semantics, no rename/forward wrapper — **promotion-law target, unrealized**. (16) `COMPOSED_IMPLEMENTATION`: compose the page owners before new scaffold; review-only.

## [02]-[MAPPING_LAW] — every doctrine law to its gate (section a)

The four gates present and working: **compiler** strictness set (see [03]), **exports map** (interior physically unresolvable), **`_architecture` gauges** (edge ledger + acyclicity + wave direction + runtime/plane tag law + pin admission + migrator ban + snapshot hygiene), and the **`Law` witness mandate** in `_testkit/src/laws.ts` (a law value cannot construct without a refuting foil; `Law.audit` runs the tautology check beside every property — this is the tests-plane's own anti-vacuity gate and it is real).

The gap is the **third gate's promotion layer**. `biome.json` carries `linter.rules.preset: recommended` with two rules disabled and **no `plugins` array at all** (confirmed: `rg plugins biome.json` → none; no `.grit` directory on disk). [04] promotion law names six anti-patterns that "no compiler flag or shipped Biome rule rejects" and mandates each as a GritQL rule at error. None exist. The unenforced law set:

`noExplicitAny`: [04] mapping law states "`any` rides `noExplicitAny` at error". Biome `recommended` ships `suspicious.noExplicitAny` at error, so the law holds by inheritance — but it is not made explicit in `biome.json`; a preset downgrade would silently drop it. Promote it to an explicit `error` row so the `any`-ban is legislated, not inherited.

Laws with **no possible mechanical gate** (`ANTICIPATORY_COLLAPSE`, `SEMANTIC_NAMING`, `INLINE_COMPOSITION`, `COMPOSED_IMPLEMENTATION`, `DERIVED_TYPES` restatement case): these are review-only by nature. [04] says "a law with no mechanical gate is captured as a plugin rule or a gauge" — where a rule is impossible, the honest close is a documented review-only classification, not a phantom gate. State this explicitly so the corpus does not read the gap as an oversight.

## [03]-[STRICTNESS_PARITY] (section b)

The full [04] compiler-law roster is present and exact in `tsconfig.base.json`. Verified line by line: `strict` (31), `exactOptionalPropertyTypes` (9), `noUncheckedIndexedAccess` (22), `noPropertyAccessFromIndexSignature` (21), `noImplicitOverride` (20), `noFallthroughCasesInSwitch` (19), `noUncheckedSideEffectImports` (23), `strictBuiltinIteratorReturn` (32), `stableTypeOrdering` (30), `isolatedModules` (12), `moduleDetection: "force"` (16), `erasableSyntaxOnly` (8), `verbatimModuleSyntax` (34), `isolatedDeclarations` (11). Flag roster: **complete, no drift**.

The parity FAILURE is not the flag set — it is the **floor's second leg**. [04] compiler law: "Every check runs the dual floor — `tsgo` and `tsc` — and parity is the floor claim." Today:
- `nx.json` wires `@nx/js/typescript` and `@nx/vite/plugin` both with `"compiler": "tsgo"`. There is **no `tsc` invocation anywhere** — not in `nx.json`, not in a `package.json` script (there is no `scripts` block), not in any config. `typescript` (tsc) and `@typescript/native-preview` (tsgo) are both catalogued, but only tsgo runs.
- **No per-package `tsconfig.json` exists under `libs/typescript/*`** (confirmed: `fd tsconfig.json` finds only the three `tests/typescript/*` projects). The `@nx/js/typescript` plugin infers its `typecheck` target from `configName: "tsconfig.json"` per project — with none present, no lib package has a typecheck target at all. The root `tsconfig.json` `references: []` and `files` lists only the four root config files, so a solution `tsc --build` typechecks nothing but the configs. **Net: no compiler, tsgo or tsc, currently reaches `libs/typescript` source** (planning-phase-acceptable since no `.ts` ships yet, but the scaffold must land per-package `tsconfig.json` + a `tsc` parity target the day source arrives).

Stronger form: each `libs/typescript/<folder>` gains a `tsconfig.json` extending `tsconfig.base.json`; the root `tsconfig.json` becomes a real project-references solution over every lib + test project; a `tsc` parity gate runs beside the tsgo `typecheck` target (a second nx target or a `package.json` script pair), and a construct the two disagree on is rewritten, never masked.

## [05]-[CONFIG_DEBT] (section d)

- **`stryker-config.json` vs `stryker.config.json` — not a duplicate to merge**: `stryker-config.json` is the **C# Stryker.NET** config (`test-runner: mtp`, `solution: Workspace.slnx`, output `.artifacts/csharp/stryker`); `stryker.config.json` is the **StrykerJS** config (`testRunner: vitest`). They are two different tools' configs colliding by near-identical basename. StrykerJS discovers `stryker.config.{json,mjs,cjs,js}` and does not read `stryker-config.json`; Stryker.NET reads `stryker-config.json` — so no runtime collision, but the near-name at the same root is a maintenance hazard. Ruling: the TS gate keeps `stryker.config.json` as-is; the C# `stryker-config.json` is out of TS scope and must not be deleted (it is the live .NET mutation config). Stronger form: rename or relocate the C# config (e.g. under a csharp-owned path or `stryker.net.json`) to kill the namesake ambiguity — but that is a C#-scope change, flagged here only as the correct disposition of the "duplicate pair."
- **Runtime vocabulary drift (three axes)**: exports map uses conditions `server`/`browser`/`wasm`/`default`; `project.json` tags use `runtime:neutral` (core/data/runtime/security), `runtime:node` (iac), `runtime:browser` (ui); `boundaries.spec.ts` `_RUNTIME_MAY` keys are `browser`/`bun`/`neutral`/`node`. `server` has no tag counterpart (maps to `node`/`bun`), `wasm` has no tag counterpart, `bun` in the gauge is unused, and `neutral` has no export-condition counterpart (rides `default`). Collapse to one canonical runtime vocabulary shared by the exports conditions, the tags, and the gauge table.
- **`customConditions` omits the runtime conditions**: `tsconfig.base.json` `customConditions: ["types","import"]`. Under `moduleResolution: bundler`, tsc/tsgo resolving `@rasm/ts/core` never match `server`/`browser`/`wasm` and always fall to `default` (index.ts). So the type-checker validates only the default barrel — `browser.ts`/`server.ts`/`wasm.ts` are never type-checked against their runtime restriction by the compiler. This is why per-runtime purity MUST be a gauge (see [04]); it is structurally invisible to the compiler floor.
- **Root `package.json` has no `scripts`**: no committed `typecheck`/`lint`/`test`/`mutation`/`e2e` entrypoint. Everything relies on bare nx target inference. There is nowhere for the missing `tsc` parity floor to be wired. Add the script surface (or nx targets) so both compilers, Biome, vitest, Stryker, and Playwright each have a named, cache-keyed entrypoint.
- **`nx.json` `sharedGlobals` omits gate inputs**: it lists `biome.json`, `vitest.config.ts`, `vite.config.ts`, `vite.factory.ts`, `tsconfig.base.json` — but **not** `playwright.config.ts`, `stryker.config.json`, or `tsconfig.json`. A change to the Playwright or Stryker config, or the root solution tsconfig, will not invalidate the affected graph → stale-cache risk for e2e and mutation. Add all three to `sharedGlobals`.
- **No vitest `projects`**: `vitest.config.ts` uses a single flat `include` glob over `tests/typescript/**` + `libs/typescript/**`. [08]-[BROWSER_E2E] mandates a browser-mode lane through the vitest browser provider (`@vitest/browser-playwright` is catalogued), but no `test.browser` project is configured — the browser-mode capability is described in the README and unwired in the config. The header comment (lines 2-5) is also stale: it asserts "the TS test estate is currently EMPTY … every former inline project matched zero files and was deleted", contradicted by the live `_testkit` (16 files), `_architecture` (3 specs), and `e2e` (`portability.pw.ts`). Restore per-lane vitest `projects` (unit/property, browser-mode, kit-falsification) and rewrite the header.
- **Root `tsconfig.json` `references: []`**: the solution references no project, so it is not a build graph. Populate references with every `libs/typescript/*` and `tests/typescript/*` project once per-package tsconfigs land (ties to the [03] parity fix).
