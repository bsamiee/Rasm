# State-Cell Boundary Law — Core Research

[CANCELLATION_SCOPE]:
- One root `CancellationTokenSource` per process; every scoped operation derives a child via `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` (overload added in .NET 9, confirmed in .NET 10). The root owns the `IDisposable` lifetime; derived sources are disposed at scope exit, not held past drain.
- `CancellationTokenSource.TryReset()` returns `false` if the source has already been cancelled; returns `true` and restores the uncancelled state otherwise. A linked source that has been triggered by its parent also returns `false`. A linked source that has not yet been cancelled resets successfully but the parent link is **not** severed — parent cancellation still propagates after reset. Verified against .NET 10 runtime.
- Cancelled `CancellationTokenSource` instances cannot be reused; `TryReset` is not a substitute for a new instance once cancellation fires. Token-gated singleton cells that require re-open semantics must replace the entire `(CancellationTokenSource, CancellationToken)` pair on each open transition, not attempt to reset a spent source.
- `CancellationTokenRegistration` is `IDisposable` and `IAsyncDisposable`; it exposes `.Unregister()` and `.Token` for symmetric attach/detach. Registrations held past cancellation waste memory on already-fired callbacks; detach in the finally/Dispose path of the owning cell, not as a fire-and-forget side effect.
- `CancellationToken.UnsafeRegister(Action<object?>, object?)` avoids the execution-context capture of the safe overload; use only where the callback owns no ambient state that could create a security boundary leak.

[ENVIO_SCOPE_CELLS]:
- `EnvIO` is the runtime environment record threading through every `IO<A>` execution: fields are `Resources Resources`, `CancellationToken Token`, `CancellationTokenSource Source`, `SynchronizationContext SyncContext`, and a nullable timeout. The `Own` (private `int`) flag governs whether `Dispose` tears down the `CancellationTokenSource`. One `EnvIO` per execution spine; concurrent operations fork child envs, not second spines.
- `EnvIO.LocalCancel` creates a child `EnvIO` whose `Source` is a new `CancellationTokenSource` linked to the parent `Source`. Cancelling the child does not cancel the parent; cancelling the parent immediately cancels all children. This is the correct scope for bounded operations that must cancel independently without propagating upward. Verified by reflection and live execution.
- `EnvIO.LocalResources` creates a child `EnvIO` with a new `Resources` scope but the same `Token` and `Source`. Resources acquired inside the child are tracked separately from the parent scope. When the child `EnvIO` is disposed, only child-scoped resources are released; parent resources are unaffected. This is the structural mechanism for sub-operation resource containment.
- `EnvIO.LocalSyncContext` replaces the `SynchronizationContext` without creating new cancellation scope — the correct form for marshaling into a specific thread's execution context while keeping the existing cancellation chain.
- `EnvIO.LocalCancelWithTimeout(TimeSpan)` and `LocalWithTimeout(TimeSpan)` create scoped envs with an additional timeout-based cancellation deadline layered onto the existing token. The timeout is additive to the existing cancellation — if the parent fires first, the child sees it immediately; if the timeout fires first, only the child is cancelled.
- `EnvIO.New` is a static factory with signature `New(Resources, CancellationToken, CancellationTokenSource, SynchronizationContext, TimeSpan?)`. Creating a second root `EnvIO` from the same `CancellationTokenSource` creates two owners that can both cancel via the same token but dispose independently — only one should own the `CancellationTokenSource`.

[ATOM_STATE_CELLS]:
- `Atom<A>` holds a private `object value` field under a `Func<A, bool> validator`. The validator runs once at `Atom.New(value, validator)` — construction returns `Option<Atom<A>>.None` if the initial value fails — and then on every `Swap`. A failing validator leaves the atom at its current value; `SwapMaybe` returning `Option<A>` makes the failure observable at the call site. Verified by both XML docs and live execution.
- `Atom<Env, A>` carries metadata `Env` that is threaded into every `Swap(Func<Env, A, A>)` call alongside the current value. The `Env` is captured at construction and never changes, making it suitable for session-scoped configuration or stable identity keys that must accompany every update without being stored in the value itself.
- `Prelude.atomIO(A)` and `Prelude.atomIO(A, Func<A,bool>)` return `IO<Atom<A>>` — atom creation lifted into the effect pipeline. The validator form returns `IO<Option<Atom<A>>>`. This enables atoms to be created as part of an `IO` composition chain where acquisition order matters for downstream initialization.
- `AtomHashMap<K, V>` is the thread-safe mutable concurrent map. `SwapKey(K, Func<V, V>)` and `SwapKey(K, Func<Option<V>, Option<V>>)` atomically update a single key. `FindOrAdd(K, Func<V>)` atomically inserts only if absent. The `Change` event fires a `HashMapPatch<K,V>` on every structural mutation, carrying `From`, `To`, and `Changes` snapshots for downstream reactive consumption.
- `Atom<A>.Swap` is CAS-looping: the function `f` may execute more than once. Any side effect inside the swap function is repeated on contention — the swap body must be a pure transform. For operations that must fire exactly once, capture the side effect in the return value and execute it after the successful swap outside the loop.
- `Prelude.swap(Atom<A>, Func<A, A>)` and `Prelude.swapIO(Atom<A>, Func<A, A>)` are top-level alternatives to the instance `Swap`; `swapIO` lifts the mutation into an `IO<A>` for pipeline composition without losing the result snapshot.

[RESOURCES_AND_USE]:
- `Resources` tracks acquired values in an internal `AtomHashMap<object, TrackedResource>`. `Acquire<A>(A value) where A : IDisposable` registers the handle; `Acquire<A>(A value, Func<A, IO<Unit>> release)` registers with a custom `IO<Unit>` release effect. The custom-release form is the correct surface for handles whose teardown is effectful (network flush, protocol goodbye, event unsubscription) rather than a bare `.Dispose()`.
- `Resources.ReleaseAll()` returns `IO<Unit>` — releasing is itself an effectful operation. Resources are released in reverse acquisition order (LIFO) because the `AtomHashMap` tracks insertion order through the `TrackedResource` record's `Release: IO<Unit>` field. Disposal of the `Resources` object or `EnvIO` triggers `DisposeU()` which synchronously runs all releases.
- `use(IO<A> acquire, Action<A> release)` and `use(IO<A> acquire, Func<A, IO<Unit>> release)` build `IOUse<A, B>` DSL nodes that register acquisition into the current `EnvIO.Resources` scope on execution. The corresponding `IO<A>.Local()` extension creates a `IOLocal<A, B>` node that transforms `EnvIO` by calling `env.LocalResources` before the inner operation, ensuring resources acquired inside the local scope are released when that scope closes rather than when the root env closes.
- `use<A>(IO<A> acquire)` where `A : IAsyncDisposable` produces `IOUseAsyncDisposable<A, B>`; the `IDisposable` form produces `IOUseDisposable<A, B>`. Both are DSL record types in `LanguageExt.DSL` resolved at `Invoke(EnvIO)` time — they are not runtime allocation wrappers; the interpreter traverses the tree.
- `IO<A>.BracketFail()` runs cleanup only on failure, leaving the resource alive on success — correct for correlated resources that must outlive the bracket's scope (e.g., a connection handed to a caller). `IO<A>.Bracket(Use, Catch, Fin)` fires `Fin` unconditionally, which is the standard safe-acquisition shape.
- `IO<A>.Uninterruptible()` wraps the computation in `IOUninterruptible<A, B>` DSL. At interpretation, cancellation is checked before entering the node and after it exits, but not mid-execution. Confirmed by examining the DSL record and by runtime test: a pre-cancelled `EnvIO` does throw `TaskCanceledException` even on `Uninterruptible` computations because the cancellation check fires at the boundary before the guarded inner computation begins.

[TOKEN_GATED_INITIALIZATION]:
- The canonical gate for once-only native initialization is `Interlocked.CompareExchange(ref int guard, 1, 0) == 0`. The CAS returns `true` exactly for the winner; all other callers see `false` and must wait or fail. The `Atom<Option<Guid>>` form in LanguageExt maps to the same CAS: `Swap(None -> Some(id))` succeeds once; subsequent swaps on a non-None value are blocked by the validator predicate.
- `Atom<Option<T>>` with validator `v => v.IsNone || someCondition(v)` encodes a one-way gate: the atom starts as `None`, the winning open transition sets `Some(handle)`, and subsequent transitions must match the condition. `SwapMaybe(current => current.IsNone ? Some(acquire()) : None)` performs the guarded open atomically; the `None` return on the already-open branch leaves the atom unchanged and signals the caller that it lost the race.
- Re-entrant initialization — where the initializer itself calls back into the guarding layer — deadlocks on any single-permit gate (`SemaphoreSlim(1,1)`, `Lock.EnterScope`, or an `Atom` with a blocking reader). Non-re-entrant native init must hold the gate cell and *not* call the guarded layer until init completes; completion should be signaled by a second cell transition, not by releasing the gate early.
- `System.Threading.Lock.EnterScope()` returns a ref-struct `Lock.Scope` with a `Dispose` implementation that releases the lock; the `using var _ = theLock.EnterScope()` pattern is the replacement for `lock(obj)` in C# 14 code targeting .NET 9+. `Lock` is re-entrant from the same thread (`IsHeldByCurrentThread`), unlike `Monitor`. This means nested lock acquisitions from the same thread succeed silently — use `SemaphoreSlim(1,1)` for non-re-entrant protection when strict single-entry is required.
- `LazyInitializer.EnsureInitialized<T>(ref T target, ref bool initialized, ref object syncLock, Func<T> factory)` — the three-field form — is the pattern for `[ThreadStatic]`-backed per-thread caches that require a sync lock for the first initialization per-thread. The single-field form `EnsureInitialized<T>(ref T target, Func<T> factory)` uses a CAS and is appropriate for process-static initialization with a reference type target.

[RUNTIME_SPINE]:
- One process-static runtime spine record is the ownership unit for `(CancellationTokenSource, TimeProvider, channel ports, configuration)`. A re-boot creates a new record; the old record must reach a fully drained and disposed state before the new one is constructed. Concurrent second spines sharing any mutable cell (a static `Atom`, a process-static `Resources`, or a shared `Channel`) produce undefined merge semantics.
- `TimeProvider` is the clock surface injected into the spine record. `TimeProvider.System` is the production singleton; test overrides implement `TimeProvider` directly. `TimeProvider.CreateTimer(TimerCallback, object, TimeSpan, TimeSpan)` returns `ITimer`; `ITimer.Change(TimeSpan, TimeSpan)` reschedules without allocation. The runtime clock must enter state cells as `TimeProvider`, never as `DateTime.UtcNow` or `Stopwatch` calls, preserving deterministic replay in tests.
- `Channel<T>` is the seam marker for the drain phase of a runtime spine. `ChannelWriter<T>.Complete()` fences new writes (subsequent `TryWrite` returns `false`); in-flight items remain readable until `ChannelReader<T>.Completion` task completes. `ChannelReader<T>.WaitToReadAsync(CancellationToken)` returns `false` after the channel is drained — this is the structural drain-wait predicate. A spine teardown that calls `Complete()` then `await Completion` guarantees all work was processed before the rest of the spine is released.
- Mid-flight semantics during drain require explicit policy: items already dequeued but not yet committed are either completed with their result or explicitly faulted. The `Channel` drain model does not provide this; the cell owning mid-flight correlation (a `ConcurrentDictionary<CorrelationId, TaskCompletionSource<T>>`) must be separately swept on drain completion, completing outstanding sources with a `OperationCanceledException` or a domain-typed fault.

[MEMO_AND_CACHE_CELLS]:
- `Memo<A>` holds `(A value, Func<A> acquire, int state)` where `state` is `0` (not computed), `1` (computed), or `-1` (error — inferred; the `state` int field with three observable behaviors). `Memo<A>.Reset()` clears state back to `0`, re-enabling acquisition on next `.Value` access. `MemoExtensions.Memo(Func<A>)` wraps any function into a lazy-computed memoization cell. This is a single-writer, single-value cell; for multi-key memoization, `ConcurrentDictionary<K, V>.GetOrAdd(K, Func<K, TArg, V>, TArg)` is the three-argument form that avoids closure allocation.
- Cache key completeness: a memo key must encode every dimension that affects the output — for any content-dependent result this means content hash plus full configuration hash. A key missing any input dimension produces stale hits where two calls with different inputs produce the same key. For native library fingerprinting, the key must include file path, file size, and modification time; library identity is not portable across file replacement even at the same path.
- `Memo<A>.Reset()` is not thread-safe — it writes the `state` field without a CAS. A `Reset()` during concurrent reads can race to produce double acquisition. The safe form for concurrent finite-lifetime memos is an `Atom<Option<A>>` whose `SwapMaybe` clears back to `None`; the next reader races to acquire, and only one wins.
- `ConcurrentDictionary<K,V>.AddOrUpdate(K, Func<K, TArg, V>, Func<K, V, TArg, V>, TArg)` — the `TArg`-carrying factory overload — avoids closure allocation for registration patterns that thread a single argument. Both factory delegates may be called multiple times on contention; they must be pure. The update delegate receives the current value; if the update result equals the current value (by the dictionary's comparer), no update is committed.
- Generated registry keys must be stable across regeneration cycles. A key derived from a `nameof` expression, a source-generated type identity, or a versioned discriminant survives recompilation; a key derived from object identity (`RuntimeHelpers.GetHashCode`), unordered iteration index, or assembly load order does not. Stale-key entries in a persistent registry accumulate as ghost entries when the type graph changes; `AtomHashMap<K,V>.Filter(pred)` is the correct sweep operation.

[LIFECYCLE_STATE_MACHINES]:
- A boundary layer's lifecycle is a state machine with at least: `Created`, `Opening`, `Open`, `Draining`, `Closed`, and `Faulted`. `Draining` fences new work (equivalent to `ChannelWriter.Complete()`) but keeps mid-flight work alive. A cell in `Draining` must reject new acquisitions with a typed rejection carrying boundary identity, correlation id of the rejected request, and timestamp. `Faulted` is escapable if the fault is transient; the escape transition creates a new `Open` instance, not a reset of the faulted one.
- Fault re-entry: when a `Faulted` boundary cell is replaced by a new one, correlation cells tracking in-progress requests must explicitly decide replace-versus-reuse. Replace produces a fresh correlation with no memory of prior state; reuse preserves request identity but inherits any partial server-side work. Both are valid; the decision must be explicit policy in the cell, not an implicit consequence of allocation order.
- `ObjectDisposedException.ThrowIf(bool condition, object instance)` (added in .NET 7, present in .NET 10) is the idiomatic guard for post-`Close` access. The `condition` operand should be `!(_state is LifecycleState.Open)` or the equivalent typed check on the state cell. Using this instead of a manual `throw` enables the analyzer to see the disposal guard.
- State transition rejection must carry typed evidence: the rejecting state (e.g. `Draining`), the correlation id of the caller's operation, and a boundary-layer timestamp from the injected `TimeProvider`. A rejection typed as `Errors.Cancelled` loses the boundary context; a domain-typed `BoundaryDrainingFault` or equivalent carries all three fields and composes cleanly into `Fin<A>` failure rails.
- Non-re-entrant process-static initialization where the initializer has side effects (platform-specific native loading, file system scanning) must hold state in a cell that distinguishes `Initializing` from `Initialized` from `InitializationFailed`. A cell that collapses these into a boolean admits the edge case where a second caller sees `Initialized = true` while the initializer is still running due to a pre-mature write. The three-state distinction enforces that `Initializing` blocks all callers until it transitions.

[ENVIO_STRUCTURAL_NOTES]:
- `EnvIO.DisposeNonResources()` releases the `CancellationTokenSource` if `Own == 1` without touching the `Resources` scope. `EnvIO.DisposeResources()` runs `Resources.ReleaseAll()` without touching the `CancellationTokenSource`. `Dispose()` runs both. Calling `DisposeNonResources` from a scope that does not own the `CancellationTokenSource` (i.e., passed a foreign `CancellationToken` but a locally created `CancellationTokenSource`) silently cancels the parent if `Own` is incorrectly set. The `Own` flag is set by the `New` factory based on whether the caller passes the same `Source` as its parent or a new local one.
- `IO<A>.Fork(Option<TimeSpan> timeout)` returns `IO<ForkIO<A>>`. `ForkIO<A>` carries `Cancel: IO<Unit>` and `Await: IO<A>` as a destructurable pair. The forked computation runs in a child `EnvIO` whose `Source` is `LocalCancel`-derived from the calling env. `timeout` wraps the child env in a `LocalCancelWithTimeout`. Awaiting after cancellation returns the error that the cancelled computation produced, not `OperationCanceledException` directly — the error must be handled via `IO<A>.IfFail`.
- `IO<A>.Post()` routes execution through `SynchronizationContext.Post` using the `SyncContext` field of the current `EnvIO`; it does not block the caller. `IO<A>.Local()` creates `IOLocal<A, B>` that calls `env.LocalResources` at interpretation time — scoping resources but not cancellation. Confusing `Post`, `Local`, and `LocalCancel` produces subtle lifetime errors: `Post` is a thread-dispatch marker, `Local` is a resource-scope marker, and `LocalCancel` is a cancellation-scope marker. Each addresses a disjoint concern in the env.
