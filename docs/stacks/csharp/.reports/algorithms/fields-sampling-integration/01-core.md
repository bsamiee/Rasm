# Fields, Sampling, and Integration — Core Research

[INTEGRATOR_CATALOG]:
- The integration surface exposes four orthogonal routes through a single static facade: `Integrate.OnClosedInterval` (double-exponential, absolute error target), `Integrate.GaussLegendre` (fixed-order Gauss-Legendre with on-the-fly node generation, supports infinite bounds via automatic variable substitution), `Integrate.GaussKronrod` (adaptive recursive with embedded error estimate and L1-norm channel), and `Integrate.OnRectangle`/`Integrate.OnCuboid` for tensor-product 2D/3D cubature. These are structurally distinct algorithms — not wrappers of one underlying kernel — and the choice of route is the primary accuracy-versus-cost decision, not the order parameter.
- The `Integrate.GaussKronrod` overload returns `(value, error, L1Norm)` through `out` parameters. The `L1Norm` is the integral of `|f|` accumulated during the same pass; a large `|result / L1Norm|` ratio diagnoses ill-conditioning from cancellation without a second evaluation. Ignoring `L1Norm` leaves the most actionable diagnostic on the table.
- `GaussKronrod` applies automatic variable substitutions for semi-infinite and doubly-infinite domains: `(-∞, +∞)` maps through `t/(1-t²)`; `[a, +∞)` and `(-∞, b]` map through matched rational maps. `GaussLegendre` applies the same substitution family. `DoubleExponentialTransformation` applies its own family. All three therefore accept `double.NegativeInfinity`/`double.PositiveInfinity` directly — no manual change-of-variable needed.
- The adaptive recursion inside `GaussKronrod` is bounded by `maximumDepth` (default 15), which caps the recursion tree at `2^maximumDepth` leaves. The error estimate at each node compares a `(N-1)/2`-point Gauss value against the `N`-point Gauss-Kronrod value; this is the only convergence signal — there is no absolute-residual fallback. Exhaustion at `maximumDepth` returns silently without failure evidence on the returned scalar; callers must inspect the `out error` parameter.
- Precomputed Gauss-Legendre nodes exist for orders {2–20, 32, 64, 96, 100, 128, 256, 512, 1024}. Any other order triggers on-the-fly computation: Newton's method initialized with Chebyshev-cosine initial guesses, iterated with both node-value and weight-value convergence checks (`|Δx| > eps || |Δw| > eps`) up to 100 iterations. The node cache is `[ThreadStatic]` — per-thread, not shared — so concurrent calls at the same non-precomputed order recompute rather than contend.
- Gauss-Kronrod nodes for orders {15, 21, 31, 41, 51, 61} are precomputed; all others use a Stieltjes-polynomial-based on-the-fly generator (`GaussKronrodPoint.Generate`). The Stieltjes method builds the extension polynomial coefficient array, then Newton-iterates on the Legendre series to find roots, using value-and-derivative of the series computed by Horner recurrence. Cache is also `[ThreadStatic]`.
- `DoubleExponentialTransformation` uses a 7-level refinement ladder with precomputed abscissas and weights. The loop starts at level 0 and refines until the absolute difference between successive estimates falls below `targetAbsoluteError`. The precomputed per-level arrays are static readonly jagged arrays — no allocation per call. `NumberOfMaximumLevels = 10` caps refinement; reaching the cap returns without signaling failure.
- The fractional differintegral `DifferIntegrate` evaluates Riemann-Liouville derivatives of non-integer order by: differentiating (via `Differentiate.Derivative`) the convolution integral (computed by `Integrate.DoubleExponential` or `Integrate.GaussLegendre`) and dividing by `SpecialFunctions.Gamma(ceil(α) - α)`. Integer orders delegate to the ordinary derivative; zero order returns `f(x)` unchanged. The implementation composes three library calls rather than implementing the fractional calculus from scratch.
- The `NumericalDerivative` step-size management has a non-obvious invariant: `StepSize` setter rounds the input to the nearest power of 2 (`Math.Pow(2, Math.Round(log₂(input)))`). This intentionally produces a binary-representable step, improving finite-difference cancellation error. Setting `StepType.Relative` derives `BaseStepSize = ε^(1/(p - α))` where `p` is the stencil width and `α` is the derivative order — the classical optimal step for centered differences. `StepType.RelativeX` computes `BaseStepSize*(1 + |x|)` at each evaluation point, adapting to magnitude.
- `FiniteDifferenceCoefficients` generates exact rational stencil weights for all derivative orders up to `points-1` using Taylor-series expansion. The API is `GetCoefficients(center, order)` — `center` selects the stencil flavor: `0` is forward, `points-1` is backward, `points/2` (integer) is centered. `GetCoefficientsForAllOrders(center)` returns the full rectangular array in one call. The `NumericalDerivative` class uses this internally for all partial and mixed-partial evaluations.
- Mixed partial derivatives (`EvaluateMixedPartialDerivative`) recurse depth-first, evaluating the outer partial last. For a mixed derivative of order `k` at `p` stencil points, the total function evaluation count is `p^k`. For `k=3, p=5`, this is 125 evaluations — a cost that grows exponentially and is not automatically reduced by symmetry of mixed partials.

[FIELD_EVALUATION_AND_INTERPOLATION]:
- The `IInterpolation` interface defines a unified contract: `Interpolate(t)`, `Differentiate(t)`, `Differentiate2(t)`, `Integrate(t)` (indefinite), and `Integrate(a, b)` (definite). Implementations set `SupportsDifferentiation` and `SupportsIntegration` flags; calling an unsupported operation throws. The flags are evaluated at runtime — there is no compile-time prevention of calling `Differentiate` on `BulirschStoerRationalInterpolation` (`SupportsDifferentiation: false`) or `Integrate` on `NevillePolynomialInterpolation` (`SupportsIntegration: false`).
- `CubicSpline` stores four coefficient arrays (`c0, c1, c2, c3`) with `N` entries for `N+1` sample points. The indefinite integral is computed lazily via `Lazy<double[]>` and cached — first call pays `O(N)` setup; subsequent calls are `O(1)` lookup plus `O(log N)` binary search. `LinearSpline` has the same lazy-integral pattern with two coefficient arrays. Both implementations allocate the indefinite-integral array at first `Integrate` call, so field integrations inside tight loops should pre-warm the lazy field.
- `CubicSpline.InterpolateHermiteSorted` constructs Hermite cubics from sample values and first derivatives directly: the coefficients `c2, c3` are derived by solving the local two-point Hermite system per interval (no global tridiagonal solve). This gives per-interval `O(1)` construction cost and exact derivative matching at sample points — the right route when tangent estimates come from the field's own gradient evaluator.
- `BulirschStoerRationalInterpolation` supports neither differentiation nor integration through the `IInterpolation` contract. It is the only interpolant that handles poles; for field evaluation near rational singularities it is the correct choice, but the contract flags must be checked before composing it with operations that require the missing capabilities.
- `TransformedInterpolation` applies a pair of forward and inverse transforms to a base interpolant. The decompiled type exists in the namespace; this enables log-linear, log-log, and other transformed-space interpolants without implementing a new type.
- The `Interpolate` static factory exposes all interpolant constructors through named factory methods: `Interpolate.LinearBetweenPoints`, `Interpolate.CubicSpline.Natural`, `Interpolate.CubicSpline.Akima`, `Interpolate.Polynomial.Neville`, `Interpolate.RationalWithPoles`, `Interpolate.LogLinear`. `Interpolate.CubicSpline.Akima` computes Akima-weighted slopes (using median-like slope estimators) — more robust to outliers than natural splines with no user-supplied boundary conditions.

[KERNEL_DENSITY_AND_SCATTER_RECONSTRUCTION]:
- `KernelDensity.Estimate` is the generic kernel surface: `Estimate(x, bandwidth, IList<double> samples, Func<double,double> kernel)`. The kernel function takes the normalized deviation `(x - xi)/h` and returns the weight. The implementation uses `CommonParallel.Aggregate` — the sum over samples is parallelized automatically. Custom kernel families (compactly supported, anisotropic) are injected directly as delegates; no subtype is needed.
- Four named kernels provided: `GaussianKernel` (delegates to `Normal.PDF(0, 1, x)`), `EpanechnikovKernel` (optimal MSE, compactly supported on [-1,1]), `UniformKernel` (delegates to `ContinuousUniform.PDF(-1, 1, x)`), `TriangularKernel` (delegates to `Triangular.PDF(-1, 1, 0, x)`). Using the distribution PDF delegates means any `IContinuousDistribution` PDF can be adapted to a kernel with a one-line lambda.
- `Fit.LinearCombination(x[], y[], Func<double,double>[] functions)` is the scattered-data reconstruction route for global basis expansion: it builds the Vandermonde-like design matrix and calls `MultipleRegression.QR`. Calling `Fit.LinearCombination(..., DirectRegressionMethod.Svd)` uses SVD-based least squares, which is rank-revealing and the correct route when the basis is nearly linearly dependent (ill-conditioned RBF matrices, overlapping compactly supported kernels).
- `MultipleRegression.DirectMethod(Matrix<T> x, Matrix<T> y, DirectRegressionMethod method)` accepts `Matrix<T>` as the response (multi-output regression) — `y` is a matrix. This enables simultaneous reconstruction of vector-valued fields (gradient, flux, RGB) from the same scattered input without separate solve calls. The overload dispatches through the same `NormalEquations / QR / SVD` switch.
- `WeightedRegression.Weighted(designMatrix, responseVector, weightMatrix)` accepts a diagonal weight matrix, implementing weighted least squares. For scattered-data reconstruction with spatially varying reliability (e.g., data from mixed measurement sources), the weight matrix encodes per-sample confidence. This is `O(N·k²)` not `O(N³)` when the weight matrix is diagonal and exploited as such by the BLAS-backed matrix multiply.

[SAMPLING_AND_RANDOM_INFRASTRUCTURE]:
- `RandomSource` is the abstract base for all MathNet generators; it inherits `System.Random` and seals `Sample()`. The only required override is `protected abstract double DoSample()`. All generators accept an integer seed explicitly. `RandomSeed.Robust()` uses `RandomNumberGenerator.GetBytes(4)` behind a static lock — it is thread-safe but contended under heavy concurrent seeding. The deterministic alternative is to construct any `RandomSource` subclass with a fixed integer seed.
- `Xoshiro256StarStar` is the highest-quality generator available: 256-bit state, xoshiro256** 1.0 from Blackman and Vigna (2018), passes all known statistical tests, sub-nanosecond generation. It is not the default — `SystemRandomSource` (wrapping `System.Random`) is used for `Generate.Uniform` and `Normal`. Replacing the default in a sampling rail requires constructing the desired generator explicitly and passing it to distribution `Samples` overloads or to `HybridMCGeneric`.
- `Generate.Uniform(length)` delegates to `SystemRandomSource.FastDoubles(length)`, which calls the static `ThreadLocal<SystemRandomSource>` default instance. This is non-deterministic across runs. `Generate.Normal(length, mean, stddev)` calls `Distributions.Normal.Samples(SystemRandomSource.Default, ...)`. Neither method accepts a user-provided seed; deterministic sampling requires constructing distributions directly: `new Normal(mean, sd, new MersenneTwister(seed))` and calling `.Samples(array)`.
- `HybridMC` (Hamiltonian/Hybrid Monte Carlo, multivariate) and `UnivariateHybridMC` sample continuous distributions given a log-density function `DensityLn<T>`. The constructor accepts `frogLeapSteps`, `stepSize`, `burnInterval`, per-dimension momentum standard deviations, an explicit `System.Random randomSource`, and a gradient delegate `DiffMethod` (callable as a finite-difference wrapper over the log-density). This is the library's primary density-sampling route for arbitrary field distributions; Metropolis-Hastings variants exist for local and global proposals.
- `TensorPrimitives.FillGaussianNormalDistribution<T>(in TensorSpan<T> destination, Random? random)` fills a multi-dimensional `TensorSpan<T>` with Gaussian normals in-place. `Random? null` uses a shared instance. This is the SIMD-accelerated path for field initialization (scalar field grids, initial-condition arrays); it avoids the per-element boxing cost of `Generate.Normal`.
- `TensorPrimitives.Invoke<TOperation, TArg, TResult>(in ReadOnlyTensorSpan<TArg> x, in TensorSpan<TResult> destination) where TOperation : IUnaryOperation_Tensor<TArg, TResult>` is the custom-operator injection surface. `IUnaryOperation_Tensor<T, TResult>` requires scalar and Vector128/256/512 overloads of `Invoke`. When these overloads are provided on a struct, the JIT auto-vectorizes the loop at the appropriate SIMD width. This is the correct path for hot field-evaluation kernels (signed-distance functions, radial basis evaluations) where each sample is computed from one input value.
- The operator interface family: `IUnaryOperation_Tensor<T, TResult>` (one span in, one span out), `IBinaryOperation_Tensor_Tensor<T1, T2, TResult>` (two spans in), `IBinaryOperation_Tensor_Scalar<T, TResult>` (span plus scalar), and `IUnaryReduction_Tensor<T, TResult>` (span to scalar). The `Tensor.Invoke<TOperation, ...>` family dispatches to these. For a kernel blend (weighted combination of two field evaluations), `IBinaryOperation_Tensor_Tensor` with a struct implementing the blend arithmetic auto-vectorizes the inner loop.

[FIELD_LAW_GAPS_AND_DESIGN_PRESSURE]:
- No Sobol or Halton quasi-random sequence is present in the installed assembly. `MathNet.Numerics.Random` contains: `Mcg31m1`, `Mcg59`, `MersenneTwister`, `Mrg32k3a`, `Palf`, `SystemRandomSource`, `WH1982`, `WH2006`, `Xorshift`, `Xoshiro256StarStar`. Low-discrepancy sequences for quasi-Monte Carlo integration over fields must be implemented externally or via a separate package.
- Scattered-data reconstruction in the library is fully 1D (`Fit`, `Interpolate`). There is no built-in radial basis function interpolation, Kriging, or moving-least-squares in the assembly — `KernelDensity` is a density estimator, not a function interpolant. Multi-dimensional scattered-data reconstruction must be composed manually: assemble the RBF or polynomial design matrix from point coordinates, pass to `MultipleRegression.DirectMethod` with `Svd` for rank safety.
- `KernelDensity.Estimate` has no spatial index — the sum over `samples` is an all-pairs O(N) scan per query point. For N samples and M query points, cost is O(N·M). There is no built-in tree-accelerated variant; callers must pre-sort or use an external spatial structure to avoid the quadratic regime.
- The 2D and 3D Gauss-Legendre cubature (`Integrate.OnRectangle`, `Integrate.OnCuboid`) uses tensor-product rules: the same 1D Gauss-Legendre nodes applied independently in each dimension. The 3D version nests three 1D integrals, each recursive via the 2D route. There is no sparse-grid or Stroud-type high-dimensional rule — the `N`-point tensor-product rule evaluates `N^d` function values in `d` dimensions, making it impractical beyond d=3 at high order.
- `GaussLegendreRule.Integrate(Func<double,double,double>, ...)` for the 2D case uses a symmetric loop exploiting the reflection symmetry of Legendre nodes: the inner loop runs `num3 = (order+1)/2` iterations and combines symmetric pairs. The weight multiplication is `w_j * (f(x_j+, y_i+) + f(x_j+, y_i-) + f(x_j-, y_i+) + f(x_j-, y_i-))`. This halves the inner-loop count only when both dimensions share the same even-order rule — the savings vanish for odd orders.
- `DoubleExponentialTransformation` has no adaptive step-halving or error estimate returned to the caller. The absolute difference between successive levels drives termination; if the integrand is smooth, convergence is exponential. But for near-singular integrands where the decay is algebraic, the `NumberOfMaximumLevels = 10` cap is reached silently — the only way to detect this is to test multiple `targetAbsoluteError` values and check for consistency.
- `CubicSpline.InterpolateSorted` / `InterpolateNatural` / `InterpolateAkima` all require sorted input. The library does not sort in-place; unsorted input throws `ArgumentException` from within `Interpolate.CubicSpline.Akima`. The `InterpolateInplace` variants (e.g., `BulirschStoerRationalInterpolation.InterpolateInplace`) sort in-place with mutation warning — these mutate the input arrays. Field evaluation pipelines that use `InterpolateInplace` must clone data if the input is shared.
- `IInterpolation.Integrate(double a, double b)` performs definite integration by subtracting two indefinite integrals: `Integrate(b) - Integrate(a)`. For `CubicSpline`, this is done using the precomputed indefinite-integral array plus a local polynomial term; it is exact within floating-point for cubic polynomials. Calling `Integrate` before `Differentiate` on the same spline instance triggers only the integration `Lazy<>`, not differentiation — the two capabilities are independently lazy-initialized.

[SIMD_AND_TENSOR_FLOOR]:
- `TensorPrimitives` in .NET 10 exposes generic-math overloads for all arithmetic operations with constraints like `IAdditionOperators<T,T,T>`, `IFloatingPointIeee754<T>`, `IRootFunctions<T>`. For field accumulation over `float` or `double` spans — gradient dot products, weighted kernel sums, diffusion residuals — calling `TensorPrimitives.Dot<T>`, `TensorPrimitives.MultiplyAdd<T>`, or `TensorPrimitives.FusedMultiplyAdd<T>` on pre-allocated `Span<double>` eliminates scalar loop overhead entirely through hardware FMA and AVX-512 on supported processors.
- `TensorPrimitives.Norm<T>(ReadOnlySpan<T>) where T : IRootFunctions<T>` computes the Euclidean 2-norm with correct handling of overflow/underflow via scaled accumulation. This is the correct primitive for sampling acceptance criteria (checking if a sample is within a radius, computing the Euclidean distance from a kernel center) rather than a manual `sqrt(sum(x²))` loop.
- The `TensorSpan<T>` and `ReadOnlyTensorSpan<T>` types (not the flat `Span<T>` family) are N-dimensional with explicit `lengths` and `strides` in `nint[]`. A field sampled on a regular grid can be represented as a `TensorSpan<double>` with strides derived from the grid dimensions; slice operations along any axis (rows, columns, depth layers) are zero-copy subspan projections, enabling plane-by-plane or line-by-line integration without copying.
- `TensorPrimitives.FillGaussianNormalDistribution<T>` writes through `TensorSpan<T>`, not `Span<T>`. For grid-shaped field initialization the caller must construct a `TensorSpan<T>` over a flat `T[]` using `TensorMarshal.CreateTensorSpan`. The `lengths` and `strides` are `nint[]`, not `int[]` — overflow-safe for grids with total element counts above 2 billion but requiring explicit `(nint)` casts when constructing from `int` dimension arrays.
