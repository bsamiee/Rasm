# Error As Monoidal Failure Carrier

[CARRIER_IS_A_MONOID]:
- The failure type is itself `Monoid<Error>` and `Semigroup<Error>`: a single error and a thousand errors share one type, so a rail's `Error` slot never widens to `Seq<Error>` to carry accumulation, and a function returning the failure carrier already returns the aggregate without a container change. The carrier is the aggregate.
- `Combine` is the monoid `Append`, and the binary `+` operator forwards to it; associativity is structural — `(a + b) + c` and `a + (b + c)` both flatten to one `ManyErrors` over the same ordered sequence, because the combine never nests a `ManyErrors` inside another. Two `ManyErrors` concatenate their inner `Seq`; a leaf combined with a `ManyErrors` cons-prepends or appends into that same `Seq`; two leaves seed a fresh two-element `ManyErrors`. The result depth is always one — leaf or flat collection, never a tree.
- The monoid identity is an empty `ManyErrors` exposed as the absence sentinel, with `Count == 0` and `IsEmpty == true`. Combining identity on either side returns the other operand untouched, so identity is a true unit and an empty failure is structurally distinct from a single information-free failure: empty is the `None`-of-`Option` analogue at the failure layer — still a failure state, but carrying zero specific errors.
- Because the carrier is a monoid, accumulation falls out of the applicative for free: the accumulating validation carrier is parameterized as `where F : Monoid<F>`, and its multi-failure constructor folds a sequence of failures through `Monoid.combine`. Pinning that carrier's failure type to the error monoid makes applicative `Apply` gather every independent rejection into one flat `ManyErrors` with no bespoke accumulator — the gather is the monoid, the monoid is the type.

[CLOSED_LEAF_FAMILY]:
- The carrier is a closed family of three leaf shapes over one abstract owner, discriminated by two orthogonal boolean facets rather than a tag enum. `IsExpected` marks a user-domain, recoverable rejection; `IsExceptional` marks a captured runtime exception; the collection case derives both facets from its members — exceptional if any member is, expected only if all members are. The two facets are not exclusive across a collection, so a mixed aggregate reports exceptional-and-not-all-expected, and a recovery combinator keyed on either facet sees the aggregate's worst element.
- The expected leaf carries message, integer code, and an optional inner error; the exceptional leaf carries message, code, and the captured exception, projecting its `InnerException` as the inner error so the native exception chain surfaces as an error chain without manual unwinding. The collection leaf carries an ordered immutable sequence and computes message, code, and facets as projections over members — no stored aggregate state to drift out of sync.
- Construction is one polymorphic factory family discriminating on argument shape, never sibling named constructors: a bare message yields an expected leaf; code-plus-message yields a coded expected leaf; an exception yields an exceptional leaf — except recognized control-signal exception types short-circuit to pre-built cancellation or timeout sentinels, and an aggregate exception explodes into a collection leaf over its inner exceptions. The discriminant is recoverable from the argument; there is no `kind` flag.
- The collection leaf flattens an aggregate exception by binding `New` over each inner exception and re-collapsing: zero inners yield the absence sentinel, one inner yields that bare leaf, many yield a fresh collection. This is the same zero/one/many collapse the public multi-error factory performs, so an aggregate boundary and a hand-built error sequence converge on identical structure — a single failure never wears a collection wrapper.
- The collection leaf carries its own reserved aggregate code distinct from any member's code, while `HasCode` over the collection quantifies existentially across members — so the aggregate answers true both to its own structural code and to any contained member's code, and a code-keyed recovery combinator can target the fact of aggregation or the presence of a specific buried failure with the same predicate. Its message is the rendered full array of member messages, computed once at construction, so an aggregate stringifies to the complete failure set, not a count.

[STRUCTURE_POLYMORPHIC_FOLDS]:
- The carrier exposes two distinct monadic folds over its internal structure, and the distinction is load-bearing. The bind-fold sequences each contained error through a Kleisli arrow into a `Monad` that is also `Fallible`, short-circuiting on the first arrow failure — first error wins, later errors unprocessed. The combine-fold sequences each contained error into a `MonoidK` and combines results, so every contained error contributes regardless of individual outcomes — the choice between them is the choice between fail-fast and gather-all expressed at the error layer, not at the call site.
- Both folds are virtual with a leaf default of applying the arrow to `this` once; only the collection leaf overrides to iterate members. A consumer writes one fold against the abstract carrier and it does the right thing whether the carrier holds one error or a hundred — the arity of the failure is absorbed by the polymorphic surface, never branched on by the caller.
- Membership, code-presence, exception-type-presence, and leaf-type-presence are all virtual predicates that the collection leaf answers by existential quantification over members, while leaves answer by direct inspection. `IsType<E>` and `HasException<E>` therefore look *inside* an aggregate, so a typed-catch combinator narrowing to one error subtype fires when that subtype is buried among unrelated siblings in a `ManyErrors`, not only when it is the sole error.
- `Filter<E>` projects the carrier to only the errors of one subtype, returning the absence sentinel when none match and otherwise re-collapsing through the zero/one/many factory — so filtering a mixed aggregate to one domain error class yields a single bare leaf when exactly one matched, the absence sentinel when none did, and a smaller flat aggregate otherwise. Recovery that handles one error class and re-raises the remainder is `Filter<Handled>` for the handled slice and the inverse for the residual, both staying inside the carrier.
- The head/tail decomposition is virtual: a leaf is its own head with the absence sentinel as tail; the collection leaf yields its first member as head and a `with`-rebuilt collection-minus-head as tail, degenerating the tail to the absence sentinel at length one. This makes the carrier list-shaped without exposing the backing sequence — a structural recursion over errors uses head/tail and bottoms out at the absence sentinel exactly as it would over an option-terminated cons-list.

[FALLIBLE_AS_PARAMETERIZED_RAIL]:
- The failure type is a type parameter of the fallible trait, not a fixed alias: the trait is two-parameter over the error type and the carrier, and the raise operation is `static abstract` over that error type. The canonical monoidal error is the common instantiation, but a carrier whose rejection vocabulary is a closed domain union can instantiate the trait with that union as its failure type, so the rail carries a precise discriminated rejection rather than a stringly message — and the generic raise function dispatches on the carrier's own failure type with no conversion at the call site.
- The convenience raise helpers split on whether the failure type is pinned to the monoidal error or left open: the open form takes the carrier's own error type; the pinned form takes the monoidal error specifically. A boundary that admits a domain rejection union internally and must surface the monoidal error externally crosses exactly once, at the trait-instantiation seam, and never re-describes the failure inside the rail.
- Recovery combinators read the carrier's failure facets, not its message text. The expected/exceptional split lets one combinator catch only domain-recoverable rejections and let captured runtime exceptions propagate to a boundary handler, and the reverse — translate captured exceptions to domain errors while passing domain rejections through. Catch keyed on integer code, on error equality, or on leaf subtype all resolve through the same virtual predicates, so the recovery vocabulary is the carrier's own algebra rather than a parallel matcher.

[RESERVED_SENTINEL_VOCABULARY]:
- The runtime owns a reserved band of large-magnitude negative codes for its structural failure states — cancellation, timeout, empty-sequence, closed-channel, end-of-stream, validation-failed, bottom, and the trait-not-implemented diagnostics — each a pre-built singleton leaf, not an allocation per raise. A combinator distinguishes a domain rejection from a runtime-structural failure by code-band rather than by message, and the band is reserved so a domain code never collides with a runtime sentinel.
- The bottom sentinel is the failure a filtered-out or default-initialized computation collapses to: a `where`-guard that removes the only value, or a struct that was never properly constructed, surfaces as bottom rather than as a null or an undefined — the absence-of-value-due-to-elimination is itself a named, code-carrying error case, distinct from the empty-aggregate absence sentinel.
- Cancellation and timeout are recognized at construction: building an error from a cancellation, operation-cancelled, or timeout exception short-circuits to the corresponding reserved sentinel instead of wrapping the raw exception, so cancellation flowing through the failure rail is always the same canonical value regardless of which BCL exception type signalled it — a recovery combinator matches one sentinel, not three exception types.

[BOUNDARY_PROJECTIONS]:
- The carrier round-trips with the exception hierarchy at boundaries without leaking either inward. A leaf projects to a wrapped error-exception preserving code, message, and inner; the collection leaf projects to an aggregate exception over its members' projections; admitting an exception inward recognizes the wrapped forms and unwraps them back to the original carrier, so a throw-catch hop across foreign code that only speaks exceptions does not degrade a structured aggregate into a flat string.
- A recovery-from-object form admits the carrier, the wrapped exception forms, a raw exception, a bare string, or an option of the carrier, and falls back to the bottom sentinel on anything else — the single inbound funnel for heterogeneous foreign failure shapes, collapsing them to the one carrier with no per-shape branch at the call site.
- The implicit conversions make the carrier the natural failure literal: a code-message tuple and an exception both convert in, the carrier converts out to an exception, and a string converts in explicitly. A rail seam writes a tuple literal where a failure is required and the monoidal error materializes, so the failure vocabulary at call sites is tuples and exceptions, never explicit factory calls.
- The carrier throws by capturing and re-raising through dispatch-info so the original stack survives, and a collection leaf throws its aggregate-exception projection — re-throwing a gathered validation failure at a boundary surfaces every accumulated rejection as one aggregate, not only the first. The throw is the inverse of the inbound aggregate-explode, closing the round-trip.
