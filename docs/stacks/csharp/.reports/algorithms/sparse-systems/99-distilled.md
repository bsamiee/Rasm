# Sparse Systems

[TRIPLET_ADMISSION_AND_CANONICAL_OWNER]:
- One `CompressedColumnStorage<T>` instance is the canonical owner of a sparse workflow's three format contexts: triplet input converts to it exactly once, an iterative solver needing the row orientation derives it by one `Transpose` at construction, and direct factorization reads the CSC form directly — format is a named type distinction, never a flag on the owner; repeated conversion among the three is the multi-representation tax.
- Triplet admission is lossy by design: `CoordinateStorage<T>.At` skips exact-zero contributions entirely (the post-assembly count is the nonzero count, not the contribution count), throws on out-of-range indices, and doubles capacity when full; the pre-filled-array constructor requires the explicit `nonZerosCount` overload — the two-parameter array form treats the storage as empty and appends from index 0, useless for pre-built buffers.
- Conversion always sorts column indices on both code paths; the `cleanup` flag controls only duplicate summation via the marker-array scan — and zero-dropping is never internal, so exact-cancellation zeros that survive dedup stay resident in the CSC result until `DropZeros(tolerance)`, whose 0.0 default removes only binary zeros; assembly residue needs a structural tolerance near `machineEps * ‖A‖_F`.
- `inplace: true` conversion calls `Invalidate()` on the triplet — arrays nulled, the count field left stale, outstanding references dangling; never take the in-place path when the triplet buffer must survive for structural-edit increments.
- Admission gates run once, behind types: strict storage validation (sorted, duplicate-free rows per column) returns bool and never throws, and factorizing storage that fails it produces silently incorrect factors; the symmetry predicate is an uncached O(nnz·log) binary-search traversal — far costlier on dense columns — so it belongs behind a constructor-verified witness type at the boundary, never at solver call sites; the complex carrier overrides it to Hermitian symmetry and virtual dispatch through the base reference reaches the override correctly.
[CROSS_FORMAT_BRIDGE_AND_REALIFIED_EMBEDDING]:
- Both bridge factories live on the row-format storage type only, and both allocate fresh independent blocks — no zero-copy bridge exists; feeding the raw `ColumnPointers`/`RowIndices`/`Values` arrays is the correct interop and no wrapper or intermediate type is justified.
- The transposition hazard: column-format arrays fed to the row-format array factory are accepted without type rejection but with transposed semantics — silently wrong preconditioner factors from a correctly structured matrix.
- The column-format factory transposes at construction and normalizes ordering and duplicates but never zeros, so explicit zeros survive; the zero policy splits by destination — the residual-arithmetic path calls zero normalization explicitly afterward, while the external direct-solver path requires explicit diagonal zeros populated and must not remove them; bare `Normalize()` skips duplicate summation, leaving assembly duplicates unsummed — the complete sequence is ordering, duplicates, zeros.
- The external sparse-provider bridge has no managed fallback (the managed provider throws unconditionally); structure-selection enum values are raw provider option literals, and a wrong structure value never throws — the provider returns a negative status code, so every non-success status must map to a typed domain failure at the boundary because provider failures are not exceptions.
- A complex n×n system constrained to real-only factorization becomes the 2n×2n block embedding `[Re,−Im;Im,Re]` — 4·nnz triplet entries before dedup, the exact-zero admission skip handling purely-real and purely-imaginary entries; the embedding is one conversion function keyed on a structure discriminant (symmetric maps to upper half only); on the complex carrier the storage-level transpose skips conjugation while the default transpose conjugates — load-bearing for hand-built Hermitian blocks.
- Operator-application orientation is asymmetric: forward row-loop SpMV is an irregular gather, vectorizable only through explicit gather intrinsics (which live on the `Avx2` static class, not the vector types — the tensor-primitive surface has no indexed gather); transpose SpMV is scatter-accumulate with write-after-write hazards, unvectorizable without row partitioning — the asymmetry selects inner-loop orientation at operator construction; the sparse product allocates its marker and exact-fit result arrays on every multiply, unamortized across Krylov applications.
[ORDERING_ADMISSION_AND_SYMBOLIC_REUSE]:
- Ordering is the amortizable phase and the symbolic build is the floor: the `AMD.Generate` permutation `int[]` is the stable cache key for value-only refactorization over an invariant pattern, but the linear-in-pattern symbolic build reruns on every reconstruction — no public entrypoint accepts a prebuilt symbolic record plus fresh values, so the durable reuse unit is the whole factorization object reconstructed with the cached permutation.
- Ordering admission is kind-gated: the symmetric factorizations admit only natural and at-plus-a orderings and reject the two least-squares orderings; the unsymmetric factorization admits all four; natural short-circuits to the identity permutation with zero ordering work.
- Fill is knowable from the pattern alone: the symbolic record's factor nonzero counts populate before any numeric sweep, so a route policy reads the fill ratio (factor nnz over input nnz) to decide direct-versus-iterative before committing to the value sweep — but the count's meaning is per-kind, one factor for symmetric kinds and a two-factor sum for asymmetric ones, so a bare fill integer compared across kinds is meaningless.
- The graph kernels (elimination tree, column counts) are public statics, so a pattern-keyed symbolic cache is buildable outside the factory entrypoints and fully amortizes symbolic cost; through the entrypoints only the ordering call is skippable via the permutation overload.
- Factorization progress reporting gates on an n/100 step — below n=100 it fires every iteration, never zero times — and carries no cancellation channel; cancellation requires an external task boundary with its own token.
[FACTORIZATION_KIND_CAPABILITY_MATRIX]:
- Capability is a kind property recoverable from the kind alone: rank-1 update/downdate lives only on the SPD kind; transpose-solve only on the two unsymmetric kinds; the indefinite symmetric kind carries neither, and its diagonal factor is private — inertia (the sign distribution revealing eigenvalue counts) is unreachable through its public surface; the shared solver interface exposes only the forward solve, so transpose capability is invisible to the abstraction and a request surface needing edit plus transpose must close over concrete kinds.
- The symmetric kinds fail untyped: SPD pivot loss and the zero-diagonal break both throw bare `Exception`, not the numerical-breakdown type — a typed-only catch misses both, so the factorization boundary catches broadly and converts.
- Asymmetry is silent where indefiniteness is loud: symmetric kinds read only the upper triangle, so an asymmetric input factors as its symmetrization and returns a correct answer to the wrong system; the post-solve true-residual recomputation is the only structural correctness signal.
- The unsymmetric pivot tolerance is a relative column threshold in [0,1] — a pivot is accepted at `tol` times its column's largest magnitude; 1 is full partial pivoting, 0 disables pivoting — never an absolute pivot floor.
[SOLVE_PIPELINE_AND_SPAN_CONTRACT]:
- A completed factorization is a closure, not a matrix: solve is a fixed permute → triangular-sweep → permute pipeline reading only the stored factors and permutation vectors, never the input matrix; the entire numeric solve surface is four allocation-free column-walking span kernels (lower, lower-transpose, upper, upper-transpose) — the forward forms divide by the column-pointer-indexed pivot then scatter the scaled column into the suffix; the transpose forms run the columns in reverse and accumulate a gathered dot product before the final pivot divide.
- Per-kind composition: the SPD kind brackets a forward-then-back-transpose sweep over one factor inside a symmetric permutation pair; the indefinite kind inlines forward, elementwise diagonal divide, back-transpose between the same bracket; the unsymmetric kind runs row permutation in, lower, upper, column permutation out — two distinct vectors; the transpose solve is the literal stage reversal with each permutation swapped for its dual, so one factorization answers both `A x = b` and `Aᵀ x = b` at solve cost.
- The span overload is the real entrypoint (the array form forwards) and checks only non-emptiness: a too-short result span faults deep inside a permutation gather, a too-long one silently leaves a tail untouched — the boundary sizes the result from the factorization's solution dimension, never from the RHS; slices stream multi-RHS through one factorization with zero intermediate allocation on the square kinds.
- The rectangular kind is two algorithms behind one entrypoint selected by the stored shape — overdetermined least-squares, underdetermined minimum-norm — and its transpose solve swaps the branch bodies, so one stored factorization yields all four least-squares/minimum-norm corners; its work buffer and reflector count size to the symbolic augmented dimension, which exceeds the row count for structurally singular systems — sizing from the matrix shape instead of the factorization's solution dimension is the off-by-augmentation fault.
[SCRATCH_REENTRANCY_AND_FACTOR_CACHE]:
- Reuse shape is a kind property, opposite across the two families and never a caller knob: the square direct kinds hold one constructor-allocated scratch array reused by every solve — allocation-free but non-reentrant, with a second concurrent solve corrupting both results silently, no guard, no thread-local; the rectangular kind allocates a fresh augmented-length work array per solve — reentrant but allocating.
- A cached square factorization is therefore a single-flight resource per instance: the reuse shapes are a capsule that owns the factorization and serializes solves, or a pattern-keyed pool of instances when concurrent multi-RHS throughput is required.
- The factor cache is weakly-keyed and success-only: keyed by dimensions, sparsity fingerprint (hash of the pointer and index arrays), and ordering or permutation identity; only factorizations whose residual witness passed populate it, so a failed or diverged solve never poisons reuse.
- The factored operator collapses to one typed value owning the factorization instance, cached permutation, symbolic fill counts, solution dimension, and kind discriminant — the kind derives transpose availability, reentrancy, and edit-cost class; and because the solve returns a bare vector with no status, the post-solve residual witness against the original operator belongs to the operator value, not the call site.
[ELIMINATION_TREE_RANK_MODIFICATION]:
- The elimination tree `parent[]` (derived per column as the smallest row index reaching it during the upper-triangular scan; roots carry −1) is the single dependency structure governing both factorization fill and every post-factorization edit: a rank-1 modification reads nothing of the original operator and no symbolic state beyond `parent[]`, mutating only the stored factor in place along the path from the edit's lowest touched row to the root — `O(tree-depth)`, never a sibling subtree, never `O(n)` — which is the entire reason a cached factorization survives a localized structural change; holding `parent[]` beside the factor is holding the map of what any edit can perturb.
- One reachability kernel drives both phases — a sign-flip marker scheme on the shared work array with the visit stack suffix reversed into topological order — parameterized only by whether values are present; one private update body serves both edit directions with sigma ±1 threading through Givens-style rotation coefficients, and the update column is admitted unsorted and duplicated (dense gather by row, entry column via min over raw indices).
- The up/down asymmetry is exact and twofold: the axpy term reads the column entry before the elementwise subtraction when adding the rank-1 term and after it when removing; the diagonal rescale folds the gathered entry in the add direction and zero in the remove direction.
- Definiteness loss is detected inline and ends the walk: the running β² going non-positive breaks the loop and returns false with the factor left mid-walk — false means discard and refactor from scratch, never retry, because the partial walk has already corrupted the factor; the Hermitian variant is the same walk with conjugated coefficients plus a per-column phase normalization forcing the diagonal real-positive — one algorithm, two carrier instantiations.
- The edit algebra is closed over symmetric rank-1 terms and nothing else: a one-sided perturbation expressed as an update column silently applies its implied symmetric closure, and any non-symmetric structural change routes to reconstruction regardless of magnitude.
[EDIT_DESCRIPTOR_GAUGE_AND_COST_CLASSES]:
- Every structural-edit dialect — pin a node, release a constraint, prune below tolerance, clean cancellation residue, rank-1 bump, revalue, gauge treatment — is one descriptor value over a three-primitive substrate: predicate compaction (`Keep`) for removal, marker-array scatter for additive merge and duplicate folding, tree-path walk for symmetric low-rank change; a function per dialect is the rejected shape, and the descriptor carries its own predicate, update column, or tolerance rather than the body re-deriving it from flags.
- Edit cost class is derivable before touching the factor from descriptor shape plus operator kind: symmetric rank-r on the SPD kind is r tree-path walks with no symbolic rerun; a pattern-preserving revalue is a numeric-only refactor reusing the cached permutation; a pattern change — or any structural edit on a non-SPD kind — is full reconstruction.
- Batch law: one in-place compressed insertion shifts the array suffix and bumps every later column pointer — `O(nnz)` each — so more than one structural insertion crosses break-even and routes through triplet accumulation plus a single re-conversion; removal triggers a defensive trim only past half-empty at ≥1024 entries, a step-function memory profile under progressive editing.
- The exact-fit trim flag is process-global mutable static state, distinct per generic instantiation and honored by compaction and by sparse add/multiply: suppress it for the whole edit sequence and restore once — under a lock or confined to a single-threaded phase — paying one final resize instead of one `O(nnz)` resize per edit.
- Merge pre-sizing is bound-asymmetric: additive combine pre-sizes to `nnz(A)+nnz(B)`, an exact upper bound that never reallocates mid-pass but leaves exact-cancellation zeros resident until a tolerance keep; the product pre-sizes the same, a lower bound — the one merge primitive that can grow inside the operation; the advertised parallel product is a serial forwarding shim doing no concurrent work, so a genuinely parallel structural merge is hand-driven over column partitions with private scatter buffers.
- Gauge handling is decidable pre-numeric: a structural rank below n (maximum bipartite matching) means no ordering rescues the factorization and gauge policy applies regardless of values; singleton count approaching block count signals near-cascade solvability with O(n) direct solves; the four gauge dialects — large diagonal penalty, row/column elimination, identity row, uniform shift — are one policy function on a discriminant built from compaction plus targeted triplet additions, where the penalty α must reach ~1e8·max-diagonal to dominate the spectrum and the induced ill-conditioning stresses the preconditioner; the treatment kind and effective α are the gauge evidence values.

```csharp
[Union]
public abstract partial record Edit
{
    public sealed partial record Pin(int Node) : Edit;                   // predicate compaction
    public sealed partial record Prune(double Tol) : Edit;               // predicate compaction
    public sealed partial record Bump(int Sign, double[] Column) : Edit; // A ± w wᵀ ⇒ tree-path walk
    public sealed partial record Revalue(double[] Values) : Edit;        // same pattern ⇒ numeric refactor
}

static Fin<FactoredOp> Apply(FactoredOp op, Edit e) => e switch
{
    Edit.Bump b when op.Kind is FactorKind.Spd =>
        op.RankOne(b.Sign, b.Column)                       // O(tree-depth), no symbolic rerun
            ? FinSucc(op)
            : Rebuild(op, e),                              // false ⇒ factor corrupt: discard, reconstruct
    Edit.Revalue r => FinSucc(op with { Inner = Refactor(op.Permutation, r.Values) }),
    var structural => Rebuild(op, structural),             // pattern change: triplet, re-convert once
};
```

[CRITERION_STACK_POLICY]:
- The stop policy is a short-circuit fold over an ordered criterion array: insertion order is precedence, the first non-continue verdict wins and lower criteria are never consulted — the list is the policy and its order is load-bearing semantics; the correct precedence is NaN-failure, divergence, residual, count cap, so the convergence verdict wins the iteration it becomes true on and the cap is the genuine last resort.
- The default stack places the residual criterion after the count cap — suppressing a valid convergence verdict on exactly the final iteration, a precedence fault invisible at the call site because every ordering returns the same verdict type — and the default iterator hardcodes a 1000-iteration cap, silently substituted when null is passed; the iterator is always constructed explicitly, and the ceiling is a size-scaled policy value (a multiple of the expected Krylov dimension) computed once and reused across every right-hand side the operator answers.
- The residual threshold is relative to the RHS infinity norm, and its sustain counter measures the gap since its previous in-threshold call — which collapses to one under per-iteration polling, making any sustain requirement above one unreachable (and the first in-threshold call after a reset clears any sustain count immediately); the only usable mode under standard polling is the degenerate fire-on-first gate, and gap-based sustain requires consulting the criterion every k-th iteration instead.
- Each criterion gates on a last-seen iteration number and memoizes its verdict: re-asking the same iteration is free and stable, and history windows advance only on strictly increasing numbers — redundant consultation is safe, fresh re-evaluation is impossible.
- The divergence verdict is a strict all-pairs conjunction over a sliding window — one residual dip suppresses it — and the window length is a symmetry-keyed policy: short windows (minimum 3 enforced) detect blow-up early on near-monotone symmetric residuals but fire spuriously through the normal 5–20-iteration oscillatory warm-up of non-symmetric solvers, where only the long window is safe; the length keys on the operator's symmetry class, never a global default.
- Criterion internals reduce every norm to a fixed double comparison, so the stack cannot be made generic over floating precision; the delegate criterion's clone shares the delegate reference, so cloned iterators share mutable closure state.
[BREAKDOWN_NAN_AND_CANCELLATION_RAILS]:
- The same mathematical breakdown — an inner-product denominator within one ULP of zero — produces three incompatible control outcomes across the non-symmetric solver family with no syntactic difference at the call site: one solver throws the breakdown exception, one cancels the iterator and exits leaving the partial iterate under a cancelled verdict, one substitutes one for the denominator and continues to a normal verdict; recovery semantics are a solver-kind property, so swapping solvers swaps failure semantics silently — and the substitution path is the most dangerous because it certifies an arbitrary iterate.
- The ULP breakdown guard is asymmetric on non-finite input: NaN fails the near-zero test, slips past the guard, and poisons the next iterate — caught one iteration later by the criterion stack, surfacing as failure or divergence depending purely on which NaN-aware criterion sits first; placing the failure criterion first keeps NaN a terminal class separable from honest divergence, and zero versus NaN are two pathologies needing two independent rails.
- Cancellation is sticky and poisoning: a cancelled iterator returns cancelled permanently until reset; the composite loop resets between solvers, discarding a cancel verdict and continuing with the partial iterate; and a pre-cancelled master token re-cancels immediately on every reset, silently poisoning every solver in the composite chain from its first iteration.
- The composite solver wraps every sub-solver in catch-everything-and-advance — breakdown, out-of-memory, and preconditioner-init failures all vanish — and copies each non-converged partial iterate into the output before continuing, so exhaustion leaves the last partial in the buffer with no inadequacy marker; its constructor resolves each setup's preconditioner internally with a unit fallback, making the externally passed preconditioner argument dead code — each setup must return its own configured preconditioner.
- The multi-start solver throws the same breakdown type at three indistinguishable recurrence phases, and its default of dozens of dense orthonormal starting vectors — built by dense QR over random samples — allocates megabytes before the first iteration at large n; 3–5 vectors suffice under a warm start and are set explicitly.
[PRECONDITIONER_BOUNDARY]:
- The verdict rail and the exception rail are disjoint by construction: preconditioner initialization runs before the iteration loop begins, and its throw escapes the verdict-returning entrypoint entirely — a returned verdict means the loop ran, an escaped exception means it never started, and the result buffer retains stale prior content that looks like an answer; the boundary initializes the preconditioner outside the solve, catches breakdown there, and falls back to diagonal scaling (scale normalization at near-zero setup cost) or a direct solve.
- Initialization traps: the modified incomplete factorization demands compressed-row sparse storage and throws otherwise, leaving the instance uninitialized — a subsequent apply produces garbage or a null fault depending on how far assignment got, so initialization state is checked after any throwing sequence; the basic incomplete factorization converts a non-sparse input densely at O(n²).
- Policy traps: the modified variant's row-sum-preserving diagonal correction helps M-matrices and diagonally dominant systems but pushes the factors away from the true ones in a non-locally-bounded way for non-M-matrices — convection-dominated systems converge worse than under the unmodified form; the dual-threshold variant's pivot tolerance defaults to zero, which disables pivoting entirely — unsound factors that manifest only as slow convergence or late breakdown.
- No preconditioner accepts or applies a permutation: the caller symmetrically permutes the operator with an ordering-derived permutation before initialization to reduce zero-pivot probability, and the breakdown's reported row index is in input iteration order, not permuted order.
[VERDICT_ROUTING_AND_WITNESS_GATE]:
- A converged verdict certifies only that the preconditioned residual fell below tolerance — left preconditioning distorts the norm's relationship to the true error — so the independently recomputed true relative residual against the original operator is the only correctness witness that survives preconditioning, breakdown substitution, cancellation, and composite swallowing alike; the substituted-through result, the count-capped partial, and the preconditioned convergence all pass through the same finite-and-in-tolerance test before an iterate is admitted.
- Verdict routing is load-bearing because the two non-convergence verdicts demand opposite responses: divergence (sustained residual growth) routes to a regularized retry or a direct solve because more iterations cannot help; count-exhaustion routes to a direct solve optionally seeded with the partial iterate because the trajectory was sound but slow.
- One boundary owns both rails and the gate — a catch combinator converting breakdown and init exceptions into the failure rail, the verdict read for routing, the witness for admission — and the direct and iterative routes converge on the identical finite-and-in-tolerance test, so the witness is one shared gate, not a per-route check.

```csharp
static Iterator<double> Stack(int dim, double tol, bool symmetric) =>
    new(new IIterationStopCriterion<double>[]
    {
        new FailureStopCriterion<double>(),                  // NaN ⇒ Failure, first so it stays distinct
        new DivergenceStopCriterion<double>(0.08, minimumIterations: symmetric ? 3 : 10),
        new ResidualStopCriterion<double>(tol),              // convergence BEFORE the cap
        new IterationCountStopCriterion<double>(8 * dim),    // size-scaled ceiling, last resort
    });

static Fin<double[]> SolveIter(
    Matrix<double> a, Vector<double> b, IIterativeSolver<double> solver,
    IPreconditioner<double> pre, int dim, double tol, bool symmetric) =>
    new Try<(IterationStatus Verdict, Vector<double> X)>(() =>
    {
        var x = Vector<double>.Build.Dense(dim);
        var verdict = a.TrySolveIterative(b, x, solver, Stack(dim, tol, symmetric), pre);
        return Fin.Succ((verdict, x));                       // init-throw + breakdown land in Fail
    }).Run()
      .Bind(t => TrueRelResidual(a, b, t.X) is var r && double.IsFinite(r) && r <= tol
            ? Fin.Succ(t.X.AsArray())                        // witness gate: the ONLY surviving test
            : Fin.Fail<double[]>(Error.New($"residual witness failed: verdict={t.Verdict}, r={r}")));
```
