# Sparse Systems: Core Law

[TRIPLET_ASSEMBLY]:
- `CoordinateStorage<T>` is the canonical triplet format. The constructor overload accepting pre-filled arrays requires the explicit `nonZerosCount` form; the two-parameter array overload treats the storage as empty and begins appending from index 0, making it useless for pre-built buffers.
- `CoordinateStorage<T>.At(i, j, value)` accumulates â€” a subsequent call with the same `(i, j)` adds to the existing entry, not replaces it. This is the correct and complete mechanism for FEM assembly where multiple elements contribute to the same global node pair; calling `At` once per element contribution is the full assembly loop.
- `Converter.ToCompressedColumnStorage<T>(storage, cleanup: true, inplace: false)` performs duplicate summation and exact-zero dropping during CSC conversion; `cleanup: false` produces unsorted, possibly duplicate rows that break all factorization routines. The `inplace: true` flag destroys the coordinate arrays and must not be used when the triplet buffer must survive for structural-edit increments.
- `CompressedColumnStorage<T>.OfIndexed(CoordinateStorage<T>, inplace)` is the static factory shorthand for the same conversion. Prefer it over the `Converter` static class when the source is already a `CoordinateStorage<T>`.
- `DropZeros(tolerance)` on `CompressedColumnStorage<T>` removes entries at or below the tolerance. The default is 0.0, so only exact binary zeros are dropped. For assembly results that accumulate near-cancellation residuals, pass a structural tolerance â€” typically `machineEps * frobenius_norm` â€” not 0.0.

[COMPRESSED_FORMAT]:
- CSparse stores CSC: `ColumnPointers` (length `n+1`), `RowIndices`, and `Values` as public fields on `CompressedColumnStorage<T>`. MathNet stores CSR: `RowPointers` (length `m+1`), `ColumnIndices`, and `Values` as public fields on `SparseCompressedRowMatrixStorage<T>`. A CSC matrix handed to MathNet's `TrySolveIterative` is treated as CSR by the preconditioner's `Initialize` without any conversion, silently producing wrong preconditioner factors.
- `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` performs an in-memory transposition at construction time, allocating a new independent block. This is the correct bridge when converting from a CSparse solve result back to a MathNet-resident operator for residual arithmetic. `PopulateExplicitZerosOnDiagonal` must be called before passing the result to MKL DSS â€” it fills zero entries on the diagonal explicitly as required by the MKL sparse solver interface.
- `SparseCompressedRowMatrixStorage<T>.NormalizeOrdering()` sorts column indices within each row. `NormalizeDuplicates()` sums duplicate entries without sorting. `NormalizeZeros()` removes explicit zeros. `Normalize()` calls `NormalizeOrdering` then `NormalizeZeros` â€” it skips `NormalizeDuplicates`. These are strictly CSR-side operations; the CSC parallel is `Helper.SortIndices` plus `DropZeros`.
- CSparse's `Helper.ValidateStorage(storage, strict: true)` checks that row indices within each column are sorted and contain no duplicates; it returns a bool and does not throw. The `strict: false` form accepts unsorted, possibly duplicate rows. Calling any factorization on storage that fails `strict: true` produces silently incorrect factors.
- `CompressedColumnStorage<T>.AutoTrimStorage` (default `true`) causes sparse matrix addition and multiplication to resize internal arrays to exactly `NonZerosCount` after each structural operation. Disable it when building a matrix incrementally from additions; the reallocations otherwise dominate runtime.

[STRUCTURAL_INVARIANTS]:
- Symmetry verification via `CompressedColumnStorage<T>.IsSymmetric()` performs an O(nnz log nnz) fold-and-compare traversal on every call â€” for each `(i, j, v)` it calls `At(row, col)`, which is a binary search. Neither form caches its result. The correct design carries a typed witness â€” a constructor-verified value object whose admitted values are limited to matrices with confirmed structural symmetry â€” so the predicate runs once at the admission boundary, not at each solver call site.
- For `CompressedColumnStorage<Complex>`, `IsSymmetric()` tests Hermitian symmetry (conjugate equality), not storage symmetry; these are not the same property and require different methods. A caller holding a base `CompressedColumnStorage<T>` reference loses the conjugation test for complex matrices.
- `Keep(Func<int, int, T, bool>)` on both `CoordinateStorage<T>` and `CompressedColumnStorage<T>` drops entries in-place where the predicate returns false. This is the correct path for structural edit operations â€” pin a node by keeping only diagonal entries in that row/column â€” without a full decompress-reassemble cycle. The predicate receives the current value, so it can combine structural and numeric drop conditions in one pass.

[SINGLE_FORMAT_LAW]:
- A sparse workflow crosses three storage contexts with distinct format requirements: assembly (triplet, `CoordinateStorage<T>`); iterative-solver matrix-vector products (CSR or CSC, depending on operator orientation); and direct-factorization (CSC with sorted, duplicate-free rows for CSparse; CSR with explicit diagonal zeros for MKL DSS). Repeated conversion between all three is the multi-representation tax. The single-owner law: one `CompressedColumnStorage<T>` instance is the canonical owner. Triplet inputs convert to it once; iterative solvers that need CSR call `Transpose` once at construction; factorization reads the CSC form directly. Format is a named type distinction, not a flag on the owner.
- `CompressedColumnStorage<T>.OfCompressedSparseRowFormat` on the MathNet side and `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` are the only zero-copy conversion bridges between formats â€” both allocate a fresh independent block. Feeding them the raw `ColumnPointers`/`RowIndices`/`Values` arrays of a CSparse matrix is the correct interop path; no wrapper or intermediate type is justified.

[FILL_REDUCING_ORDERINGS]:
- `ColumnOrdering.MinimumDegreeAtPlusA` is the only valid ordering for symmetric positive-definite systems solved by `SparseCholesky` or `SparseLDL`. `MinimumDegreeAtA` and `MinimumDegreeStS` operate on the normal equations and produce permutation vectors that are semantically valid but numerically counterproductive for these factorizations; no exception is thrown when they are applied.
- `AMD.Generate<T>(A, ColumnOrdering)` computes the approximate minimum degree permutation vector and returns `int[]`. The `Create` overload accepting `int[]` instead of `ColumnOrdering` lets callers cache or share the permutation vector across multiple factorizations with identical sparsity patterns but different numeric values. This is the mechanism for the weakly-keyed factor cache: key on sparsity pattern identity (row-count, column-count, and a hash of `ColumnPointers`/`RowIndices`), cache the `int[]` permutation, and pass it to numeric-only re-factorizations.
- `SymbolicFactorization` exposes `lnz` (nonzeros in L) and `unz` (nonzeros in U or R) as direct integer fields. These are the fill-in evidence values. A sparse receipt carries these before the numeric factorization to expose fill ratio (`lnz / A.NonZerosCount`) as a diagnostic.
- `GraphHelper.EliminationTree` and `GraphHelper.ColumnCounts` are public static methods ordinarily called inside the symbolic analysis phase. Calling them independently allows inspection of symbolic fill structure before committing to a numeric factorization under a particular ordering.

[FACTORIZATION_LAW]:
- `SparseCholesky.Create(A, ColumnOrdering.MinimumDegreeAtPlusA)` performs symbolic analysis and numeric factorization in one call; only the upper triangular part is read. Passing a non-SPD matrix produces silently incorrect factors â€” the factorization succeeds but `Solve` returns a numerically wrong answer. The receipt must include a post-solve residual recomputation `â€–Ax âˆ’ bâ€–â‚‚ / â€–bâ€–â‚‚` as the only reliable correctness witness.
- `SparseLDL` handles indefinite symmetric matrices but performs no numeric pivoting. For strongly indefinite systems or those with near-zero pivots, LDL' produces inaccurate factors without warning. The receipt law for LDL': carry the diagonal-D sign flags (count of negative pivots) as a structural audit field so callers can detect unexpected indefiniteness.
- `SparseLU.Create` accepts a `tol` parameter in [0, 1] controlling partial pivoting: `tol = 1.0` is full partial pivoting; `tol = 0` disables pivoting entirely. The parameter is a relative threshold â€” a pivot is accepted if it is at least `tol` times the magnitude of the largest element in its column, not an absolute pivot-size threshold.
- `SparseCholesky.Update(w)` and `Downdate(w)` implement rank-1 Cholesky updates in-place without re-factorization; `UpDown(sigma, w)` is the combined `+sigma * w*w'` form with `sigma âˆˆ {+1, -1}`. These return `false` when the updated matrix loses positive definiteness rather than a typed error. For structural-edit paths â€” pinning a node modifies one row/column â€” these are significantly cheaper than full re-factorization when the modification is rank-1.

[KRYLOV_ITERATION]:
- `Matrix<T>.TrySolveIterative(input, result, solver, iterator, preconditioner)` returns `IterationStatus` without throwing. Only `Converged` represents an acceptable result. The distinction between `Diverged` (monotone residual increase detected) and `StoppedWithoutConvergence` (count cap hit without convergence) is load-bearing for fallback routing: divergence routes to a regularized retry or direct solve; count-exhaustion routes to a direct solve, optionally seeded with the partial iterate.
- `Iterator<T>` is constructed by passing an `IIterationStopCriterion<T>[]` array. The stacked stop-criteria policy short-circuits on the first non-`Continue` status. The correct minimum stack is `[FailureStopCriterion, DivergenceStopCriterion, ResidualStopCriterion, IterationCountStopCriterion]` in that order: NaN detection before divergence before convergence before hard cap. `ResidualStopCriterion` placed after `IterationCountStopCriterion` means the count cap can suppress a valid convergence signal on the final iteration.
- `ResidualStopCriterion<T>` carries both a `Maximum` threshold and a `MinimumIterationsBelowMaximum` count. The convergence gate is `_iterationCount >= MinimumIterationsBelowMaximum` where `_iterationCount = iterationNumber - _lastIteration` and `_lastIteration` initializes to `-1`. On first qualifying iteration (iteration 0), `_iterationCount = 1`. With the default `MinimumIterationsBelowMaximum = 0`, the criterion fires the first time the residual drops below `Maximum * â€–bâ€–_âˆž` â€” the threshold is relative to the right-hand-side infinity norm, not absolute. For ill-scaled problems where `â€–bâ€–_âˆž = 1e8` and `Maximum = 1e-12`, the effective absolute threshold is `1e-4`.
- `DivergenceStopCriterion<T>` fires only when the residual increases monotonically over the full window (`MinimumNumberOfIterations` defaults to 10) at a relative rate above `MaximumRelativeIncrease` (defaults to 0.08) for every consecutive pair â€” a strict AND over the entire window. A single residual dip in a 10-iteration window prevents the criterion from firing regardless of overall trend.
- The `IterationCountStopCriterion<T>` default cap is a constant, not scaled to system size. The correct design carries the size-scaled cap as a policy value computed at factorization time.
- Double-checked convergence: `TrySolveIterative` returning `Converged` means the `ResidualStopCriterion` saw the preconditioned residual fall below its threshold, not the true residual `â€–Ax âˆ’ bâ€–`. The independent recomputation `â€–Ax âˆ’ bâ€–â‚‚ / â€–bâ€–â‚‚` must be performed in the calling code because preconditioning can distort the internal residual's relationship to the true error.

[PRECONDITIONERS]:
- `ILU0Preconditioner` computes an ILU(0) factorization limited to positions already nonzero in the input. Its `Initialize` method internally calls `SparseMatrix.OfMatrix(matrix)` unconditionally â€” for a non-sparse input this is an O(nÂ²) dense-to-sparse conversion; always pass a sparse matrix.
- `ILUTPPreconditioner` carries `FillLevel`, `DropTolerance`, and `PivotTolerance` as mutable properties. `PivotTolerance` defaults to 0.0, which disables partial pivoting â€” the condition `row(i,j) > row(i,i) / PivotTolerance` is never triggered when `PivotTolerance = 0`. Setting it to a small positive value (e.g., 0.1) enables partial pivoting; the permutation is stored in `_pivots` and applied during back-substitution. Forgetting to set `PivotTolerance` for non-diagonally-dominant problems leaves the preconditioner with unsound factors that manifest only as slow convergence or `NumericalBreakdownException`.
- `MILU0Preconditioner` corrects the row sums of the L and U factors so the preconditioner preserves the row sums of the original matrix, which improves convergence for M-matrices and diagonally dominant problems. Its `UseModified` property (default `true`) selects between modified and standard forms; `IsInitialized` allows deferred initialization checks before the first `Approximate` call.
- `DiagonalPreconditioner` stores only the diagonal. For Krylov methods applied to near-diagonal systems it provides scale normalization at near-zero setup cost; for general systems it degrades convergence relative to ILU(0).

[GAUGE_HANDLING]:
- Sparse systems with pure Neumann boundary conditions or floating substructures have a singular stiffness matrix with a non-trivial null space. There is one policy concern with four named dialects: large diagonal penalty (add `Î± * I` to singular rows); row/column elimination (replace gauge node's row and column with a zero row, zero column, and 1 on the diagonal, adjusting the right-hand side); gauge-node identity row (replace one row with the identity condition directly in the assembled `CoordinateStorage<T>` before conversion); and regularization shift (add `Î± * I` uniformly). All four are implementable through `CompressedColumnStorage<T>.Keep` combined with targeted `At` additions on the triplet before conversion. A single policy function parameterized by a `GaugeTreatment` discriminant produces the appropriate modification; a separate function per dialect is the anti-pattern.
- The large diagonal penalty value `Î±` must exceed the spectral radius of the stiffness matrix to be numerically effective. A practical lower bound is `Î± â‰¥ 1e8 * max_diagonal_entry`. The resulting system is ill-conditioned; for iterative solvers this stresses the preconditioner. The receipt for gauge-treated systems carries the gauge treatment type and the effective `Î±` value.

[REALIFIED_BLOCK_EMBEDDINGS]:
- When the factorization dependency is constrained to real-only solvers, a complex `nÃ—n` system becomes a `2nÃ—2n` real system with embedding `[Re(A), -Im(A); Im(A), Re(A)]`. Each complex entry `(i, j, a+bi)` contributes four real entries; the `CoordinateStorage<double>` for the block-embedded system has `4 * nnz` entries before deduplication. The `At` duplicate-summation mechanism handles overlapping block contributions correctly. The admission boundary for complex-to-real embedding is a single conversion function keyed on a structure discriminant: `Nonsymmetric` maps to no symmetry assumption; `Symmetric` maps to the real embedding with upper half only.
- `SparseMatrix.Transpose(storage: true)` on a Complex `SparseMatrix` performs a storage-level transpose without complex conjugation; the default `Transpose()` includes conjugation. The distinction is non-trivial when building Hermitian block systems manually.

[FACTOR_CACHE]:
- A weakly-keyed success-only factor cache holds solved factorizations keyed by the complete physical parameter set: matrix dimensions, sparsity fingerprint (hash of `ColumnPointers` and `RowIndices` arrays), and the ordering enum or permutation vector identity. The cache stores only successful factorizations â€” those where post-solve residual verification passed â€” so a failed or diverged solve never populates the cache.
- The permutation vector `int[]` from `AMD.Generate` is the stable key for numeric-only re-factorizations when sparsity is invariant but values change. Caching the permutation separately from the numeric factors lets the symbolic phase run once; numeric-only phases run cheaply under new value assignments.
- The `SymbolicFactorization` fields `lnz` and `unz` report nonzeros in the factors before numeric factorization. These belong in the receipt alongside the pre-ordering nnz of the input matrix so the fill ratio is auditable without re-running the factorization.
