# Numeric Evidence Law: Receipts and Rails

[ALGORITHM_RECEIPTS]:
- Every decomposition in the admitted package hierarchy carries algorithm-specific typed evidence: `Cholesky<T>` exposes `Factor` (lower triangular `Matrix<T>`) plus computed `Determinant` and `DeterminantLn`; `LU<T>` exposes the public surface `L`, `U`, `P` (a `Permutation` built from pivot inversions), and pivot-sign-corrected `Determinant` — `Factors` and `Pivots` are private fields; `Svd<T>` exposes `S` (singular-value vector), `U`, `VT`, `Rank`, `L2Norm`, and `ConditionNumber`; `Evd<T>` exposes `EigenValues` (`Vector<Complex>`), `EigenVectors`, `D` (block-diagonal form), `Rank`, `Determinant`, and the `IsSymmetric` honesty flag. A generic receipt over all factorization families must carry the algorithm-specific properties as a concrete sub-type — no common abstraction is possible without losing evidence.
- `Symmetricity` is a four-case enum (`Unknown`, `Symmetric`, `Hermitian`, `Asymmetric`) passed to `Evd.Create` as a policy value that controls both the algorithm branch chosen at construction and the structural meaning of eigenvectors. Storing it as a field on the receipt turns an implicit algorithm choice into an auditable fact. `Unknown` triggers a symmetry check; passing `Symmetric` or `Hermitian` bypasses it and contracts that the caller has already verified the precondition — the receipt must record which case was supplied, not just what was inferred.
- `QRMethod` (`Full` vs `Thin`) is a behavior-carrying enumeration: `Full` produces a square `Q` (m×m) and full `R`; `Thin` produces economy-sized `Q` (m×n) and upper-triangular `R` (n×n). A solve over a thin QR cannot reconstruct the full column space. A receipt that records `QRMethod` is the only way a downstream consumer can know whether the orthogonal complement was discarded.
- The `Svd<T>.Rank` threshold is `S.Maximum().EpsilonOf() * Math.Max(m, n)`. Because `S` is stored descending, `S.Maximum()` scans the full vector in O(n); pre-computing and storing `S[0].EpsilonOf() * Math.Max(m, n)` on the `double` concrete type is correct and avoids the scan — but the field must carry the algorithm-qualified name, not a generic `Tolerance`, because the `Complex32` variant substitutes `S.AbsoluteMaximum().Magnitude.EpsilonOf() * (float)Math.Max(m, n)`: the `(float)` cast reduces dimension precision and widens the threshold relative to the `double` form. Capturing both the stored threshold value and the type path as named fields is the minimum auditable evidence; a receipt that stores only `Rank` as an `int` hides the tolerance policy entirely.
- `Evd<T>.Rank` uses `AlmostEqual(eigenvalue, Complex.Zero)` with the library's `DefaultDoubleAccuracy = DoublePrecision * 10` — a private constant equal to `2^{-53} * 10`. This tolerance differs from the SVD threshold. A receipt distinguishing these two rank computations as algorithm-qualified facts prevents downstream code from treating them as interchangeable. When the domain requires a different rank threshold than either hardcoded tolerance, the receipt must record both `LibraryRank` (from the API) and `DomainRank` (recomputed from the raw eigenvalue or singular-value vector against a policy constant), alongside the policy constant, so a re-admission check can verify the discrepancy is within the expected tolerance envelope.

[TOLERANCE_TAXONOMY]:
- `Precision` exposes a named ledger: `DoublePrecision = 2^{-53}`, `PositiveDoublePrecision = 2 * DoublePrecision`, `MachineEpsilon` (measured at runtime), `PositiveMachineEpsilon`, and `DefaultDoubleAccuracy = DoublePrecision * 10` (private, used implicitly by parameterless `AlmostEqual`). Inline literals such as `1e-10`, `1e-14`, `1e-15`, or `1e-8` that duplicate one of these anchors are the anti-pattern; the anti-pattern is provable when the literal equals a multiple of `DoublePrecision` for a concrete platform.
- `EpsilonOf(this double value)` returns the unit-in-the-last-place (ULP) of the argument — the distance to the adjacent representable double — computed via `BitConverter.DoubleToInt64Bits` increment/decrement arithmetic, not a multiplication by `DoublePrecision`. For values whose binade changes the ULP magnitude (e.g., `EpsilonOf(2.5)` = `4.44e-16` = 2⁻⁵¹, while `2.5 * DoublePrecision` = `2.78e-16`), the two formulas diverge; using the multiply formula as a receipt validity predicate introduces order-of-magnitude errors for large magnitudes and incorrect tightness near binade boundaries. `PositiveEpsilonOf = 2 * EpsilonOf`.
- `AlmostEqualNorm(a, b, diff, maximumAbsoluteError)` is the lowest-level comparison: it compares the pre-computed `diff` (`a - b`) against an absolute threshold, and all higher-level overloads delegate to it. A receipt validity predicate that pre-computes the residual norm and stores it alongside the threshold makes re-admission possible without recomputing `a - b`.
- `AlmostEqualNormRelative(double a, double b, double diff, double maximumError)` compares `|diff| < maximumError * Math.Max(|a|, |b|)` for the general case; when both `|a|` and `|b|` fall below `DoublePrecision`, the method falls back to comparing `|diff| < maximumError` as an absolute threshold; when either argument is exactly zero and the other is non-zero the check compares the non-zero magnitude against `maximumError`. The near-zero absolute fallback is the load-bearing nuance: a receipt that records only `IsRelative: true` and the scalar `maximumError` cannot determine at re-admission time whether the comparison was resolved by the relative branch or the near-zero branch, making the stored tolerance semantically ambiguous for small-magnitude residuals.
- `ResidualStopCriterion<T>._maximum` is a relative convergence threshold. `DetermineStatus` tests `residualVector.InfinityNorm() < _maximum * sourceVector.InfinityNorm()` — the threshold is scaled by the RHS infinity norm at each iteration. A receipt that records this field as an absolute residual threshold mis-replays the convergence decision on re-admission: a threshold of `1e-12` converges when the residual is below `1e-12 × ‖b‖_∞`, not below `1e-12` in absolute terms. The hysteresis parameter `MinimumIterationsBelowMaximum` (default `0`) must be stored alongside `Maximum` for accurate re-admission.
- `NumericalDerivative.BaseStepSize` equals `sqrt(Epsilon)` — approximately `1.49e-8` — regardless of the stencil point count. The scaling at evaluation time is `StepSize = BaseStepSize * (1 + |x|)` for `StepType.Relative`, `BaseStepSize * (1 + |x|)` with no center offset for `RelativeX`, and the stored `StepSize` field directly for `Absolute`. The `StepSize` setter rounds the assigned value to the nearest power-of-two representable double to improve finite difference accuracy; a receipt must read `StepSize` back after assignment because the stored value is the actual `h` used, not the requested one. A differentiation receipt must record `StepType`, the `BaseStepSize` (= `sqrt(Epsilon)`), the derivative `order`, and the point count separately — the effective step at a given `x` is `sqrt(Epsilon) * (1 + |x|)` for `Relative`, and neither `order` nor `points` changes `BaseStepSize`.
- `DivergenceStopCriterion<T>` constructor: `(double maximumRelativeIncrease = 0.08, int minimumIterations = 10)`. The `minimumIterations` parameter has a hard lower bound of 3 (enforced by the constructor). `_residualHistory` tracks residual norms for a look-back window of length `MinimumNumberOfIterations`. A divergence-flagged receipt must record the window length alongside `MaximumRelativeIncrease` — the same relative increase over a 3-iteration window versus a 10-iteration window can produce opposite divergence decisions: a transient residual spike over 3 steps triggers divergence where a 10-step window would not. The window length is the policy that resolves which conclusion is valid.

[ITERATIVE_SOLVER_EVIDENCE]:
- `IterationStatus` has six cases: `Continue`, `Converged`, `Diverged`, `StoppedWithoutConvergence`, `Cancelled`, `Failure`. `StoppedWithoutConvergence` is the diagnostic-but-unusable case: the solver exhausted its iteration budget without converging, diverging, or breaking down. A receipt that maps it to `Failure` destroys the distinction between "budget exceeded" and "algorithm broke down" (the `NumericalBreakdownException` / `Failure` case). The minimum evidence partition is `Converged | StoppedWithoutConvergence | (Diverged | Failure | Cancelled)`; any coarser receipt risks hiding pathological conditioning behind an innocuous timeout.
- `TrySolveIterative` on `Matrix<T>` has six public overloads, each returning `IterationStatus` directly: three accept a `Vector<T>` right-hand side and three accept a `Matrix<T>`; within each group, the parameter shapes are `(input, result, solver, Iterator<T>, IPreconditioner<T>)`, `(input, result, solver, IPreconditioner<T>, IIterationStopCriterion<T>[])`, and `(input, result, solver, IIterationStopCriterion<T>[])`. The `Iterator<T>` overload exposes the aggregated terminal status via the returned `IterationStatus` and via `iterator.Status` post-call; the stop-criteria overload exposes only what is returned. A receipt factory uses the `Iterator<T>` overload to capture both the return value and the iterator state as co-equal evidence. The receipt must also record whether a non-unit preconditioner was active — the preconditioner type changes what "residual" means.
- `Iterator<T>.DetermineStatus` iterates the stop-criteria list in registration order and returns on the first non-`Continue` status found, not on the most severe. The default constructor `Iterator<T>()` populates from `Matrix<T>.Build.IterativeSolverStopCriteria(maxIterations: 1000)`, which returns four criteria in order: `FailureStopCriterion` first, `DivergenceStopCriterion` second, `IterationCountStopCriterion` third, `ResidualStopCriterion(1e-12)` last. Under this default ordering, a solver that simultaneously triggers failure and convergence is classified as `Failure` because `FailureStopCriterion` fires first — a receipt factory must capture the full stop-criterion list in insertion order, not just the terminal `Iterator.Status`, so re-admission can replay the priority logic and verify the status was not an order-dependent artifact.
- `DelegateStopCriterion<T>` accepts `Func<int, Vector<T>, Vector<T>, Vector<T>, IterationStatus>` — the delegate receives `(iterationNumber, solutionVector, sourceVector, residualVector)`. A receipt can embed a per-step snapshot policy directly: the delegate records every k-th residual norm into a preallocated `double[]` trail without heap pressure, and the trail length relative to the total step count becomes a conservation invariant in the receipt.
- `ILUTPPreconditioner` exposes three independent numeric policy constants: `FillLevel` (default `200.0`, dimensionless multiplier of input NNZ), `DropTolerance` (default `0.0001`), and `PivotTolerance` (default `0.0`, meaning no pivoting). `UpperTriangle()`, `LowerTriangle()`, and `Pivots()` are private methods unavailable to consuming code; the fill outcome must be inferred post-factorization through a separate sparse-matrix norm or count operation rather than read from the preconditioner directly. A receipt must store all three policy constants as named fields; the fill budget constraint `NNZ(L+U) <= FillLevel * NNZ(A)` and the pivot regime (unconditional diagonal when `PivotTolerance = 0.0`, conditional column pivoting otherwise) are qualitatively different evidence states that a receipt must distinguish as an honesty marker `bool PivotingEnabled`.

[OPTIMIZATION_EVIDENCE]:
- `MinimizationResult` carries `MinimizingPoint` (via `FunctionInfoAtMinimum.Point`), `Iterations`, and `ReasonForExit` (a twelve-case `ExitCondition`). `MinimizationWithLineSearchResult` extends this with `TotalLineSearchIterations` and `IterationsWithNonTrivialLineSearch` — the line-search sub-iteration count is algorithm-specific evidence owned by BFGS that has no analog on trust-region results and must not collapse into a generic `int TotalIterations`.
- `NonlinearMinimizationResult` carries `Covariance`, `Correlation`, `StandardErrors`, and `MinimizedValues` alongside `Iterations` and `ReasonForExit`. When `DegreeOfFreedom < 1` (underdetermined system), all three matrix and vector properties are `null`. A receipt validity predicate must gate on `DegreeOfFreedom >= 1` — accessible via `IObjectiveModelEvaluation.DegreeOfFreedom` — before treating the covariance as a meaningful fact; a receipt that materializes covariance without this gate is not tamper-evident.
- `ExitCondition` has twelve cases. The split between success-adjacent cases (`WeakWolfeCriteria`, `StrongWolfeCriteria`, `RelativeGradient`, `AbsoluteGradient`) and genuine non-convergence (`ExceedIterations`, `LackOfProgress`) is load-bearing for a receipt: meeting the Wolfe criteria with very few iterations is qualitatively different from hitting `LackOfProgress`, which indicates numerical stagnation. A receipt honesty marker that records the sub-family of `ExitCondition` obtained is more discriminating than the raw enum value.
- `LevenbergMarquardtMinimizer` constructor defaults: `initialMu = 0.001`, `gradientTolerance = stepTolerance = functionTolerance = 1e-15`. These three tolerances serve distinct purposes: gradient (first-order stationarity), step (parameter change), function (residual norm). A receipt that stores one scalar "tolerance" conflates three convergence criteria. The five-parameter constructor (`initialMu`, `gradientTolerance`, `stepTolerance`, `functionTolerance`, `maximumIterations`) is the minimal policy surface; a receipt stores all five as named fields with their algorithmic roles.

[CONSERVATION_PREDICATES]:
- Rank-nullity: for an `n × n` matrix, `rank + nullity == n`. `Evd<T>.Rank` counts non-zero eigenvalues under `DefaultDoubleAccuracy`; `Svd<T>.Rank` uses the ULP-relative threshold; neither exposes `Nullity`. A receipt constructed from either must derive `Nullity = n - Rank` and record both, so re-admission can verify the conservation identity without rerunning the decomposition.
- SVD singular-value ordering: `DenseSvd` stores singular values in descending order — `L2Norm = Math.Abs(S[0])` and `ConditionNumber = Math.Abs(S[0]) / Math.Abs(S[last])` are both correct only if `S[0]` is the maximum. A receipt must assert `S[0] >= S[1] >= ... >= S[k-1] >= 0` at construction; violation signals corruption or a post-construction modification of the `S` vector. A receipt that stores `S[0]` as the "minimum" or asserts `S[0] <= S[1]` is structurally wrong — the implementation is the authority, and any conservation check that treats the sorted order as ascending corrupts every downstream rank-threshold, condition-number, and pseudoinverse computation.
- Cholesky positive-definiteness: `DenseCholesky.Create` throws `ArgumentException` when the matrix is not SPD; there is no `bool IsPositiveDefinite` property. A receipt wrapping a Cholesky factorization records success as a structural fact — existence of the `Factor` field — not as a boolean flag. For a near-SPD matrix where the domain layer must apply a diagonal shift before invoking Cholesky, the shift magnitude is degeneracy evidence that must be recorded as a named field `DiagonalShift`; downstream callers that interpret log-determinant or condition number from the factored matrix must account for the shift.
- LU pivot parity: `LU<double>.Determinant` computes the sign from the private `Pivots` array (`Pivots[i] == i` = even permutation, `Pivots[i] != i` = odd, negates). The public surface exposes only `Determinant` as the sign-corrected result; `P` is a `Permutation` derived from pivot inversions. A receipt that stores `Determinant` has all the conservation evidence available through the public API; re-admission that requires verifying pivot parity must recompute via `P.Determinant` on the public `Permutation`.
- `RunningStatistics.Combine(a, b)` is a pure static factory that merges two Welford moment accumulators. The conservation law is `combined.Count == a.Count + b.Count`. A hierarchical receipt embedding two independently computed sub-accumulator receipts and a combined parent must assert this identity at construction; violation signals a dropped sample or incorrectly initialized sub-accumulator. `Combine` is directly usable as the `f` in `Atom<RunningStatistics>.Swap` because it is pure and safe for CAS-loop retry.

[DIAGNOSTIC_STATUS_DISTINCTION]:
- `IterationStatus.StoppedWithoutConvergence` must be kept distinct from both `Converged` and `Failure` in every receipt surface and every error union that wraps iterative solver outcomes. Mapping it to `Fin.Fail` is the anti-pattern: a rail that treats a budget-exceeded solve as a hard failure prevents the caller from retrying with a relaxed criterion or a different preconditioner.
- `ExitCondition.None` and `ExitCondition.ManuallyStopped` are the two non-algorithmic exit cases. A receipt constructed with `ExitCondition.None` is structurally incomplete; the factory must reject `None` and return `Fin.Fail(Error.New(code, "optimizer did not start"))`.
- `DssStatus.MKL_DSS_ZERO_PIVOT` is a definiteness failure, not a general solver failure — it indicates the matrix is singular or indefinite given the `DssMatrixType` policy. A receipt wrapping the sparse direct solver must map this to a domain-level `SingularMatrixError` distinct from a general internal breakdown status, because collapsing both to a single `SolverFailure` discards the definiteness signal.

[INTEGRATION_AND_DIFFERENTIATION_EVIDENCE]:
- `Integrate.GaussKronrod` exposes two evidence channels through `out` parameters: `error` (the difference between the embedded Gauss approximation and the Kronrod extension) and `L1Norm` (the L1 norm of the result). These are structurally distinct: `error` is the embedded error estimator intrinsic to the G-K pair, while `L1Norm` measures whether the returned value is close to its own norm — a large disparity between the two signals a poorly conditioned integrand rather than a convergence failure. The receipt factory must capture `(Value, ErrorEstimate, L1Norm, TargetRelativeError, MaximumDepth, Order)` so re-admission can verify `ErrorEstimate <= TargetRelativeError * |Value|` and that `|Value - L1Norm| / L1Norm` stays below a policy threshold.
- `GaussKronrodRule.Order` exposes the total Kronrod point count; pre-computed point sets exist for orders 15, 21, 31, 41, 51, and 61. A K15 rule embeds G7 (7-point Gauss) and is exact for polynomials of degree up to 23 (standard Gauss-Kronrod result for odd-embedded-n: `3n+2 = 3×7+2`); a K61 rule embeds G30 and is exact to degree 91 (`3n+1 = 3×30+1` for even-embedded-n). The 2N−1 exactness formula applies to pure Gauss rules, not Gauss-Kronrod extensions — a receipt that uses `2*Order - 1` as the exactness degree overstates the polynomial coverage by roughly 25 %. A receipt that omits the rule order makes the exactness claim non-auditable. When the receipt drives downstream quadrature assembly, the order determines what source functions are integrated exactly, not merely how many function evaluations were taken.
- `DoubleExponentialTransformation.NumberOfMaximumLevels` is a private static constant (value `10`) bounding adaptive subdivision depth; the `Integrate` method returns a bare `double` with no depth-reached output parameter. A receipt for double-exponential quadrature cannot capture actual subdivision depth from the public API. The honesty marker available is a boolean `MayHaveSaturated` flag set by comparing a coarser-level estimate against the final result — the only proxy for saturation that does not require private API access.
- `NewtonCotesTrapeziumRule.IntegrateAdaptive` accepts a `targetError` parameter (not `targetAbsoluteError`) and applies a relative convergence test internally via `AlmostEqualRelative(finer, coarser, targetError)`. Because no error estimator is returned, the receipt cannot self-validate against the method's own convergence semantics. The external conservation check available to a receipt is `|estimate - coarser_estimate| / |estimate| <= targetError`, matching the relative test used internally; recording `targetError` as an absolute threshold mismodels the convergence and makes re-admission produce wrong verdicts on small-magnitude integrals.
- `NumericalDerivative.Evaluations` is a cumulative counter across all derivative computations made by a single instance, not a per-call count. A receipt wrapping a single derivative evaluation must snapshot `Evaluations` before and after the call and record the delta; that delta equals `Points` (the stencil width), providing the conservation check `EvaluationsUsed == Points` for a single-order univariate evaluation.
- `StepType.Relative` with `Center = 0` (one-sided forward difference) is fundamentally less accurate than center differences for smooth functions: the asymmetry removes the cancellation that makes centered differences second-order. A receipt carrying both `Center` and `StepType` is the minimum evidence to determine whether the derivative estimate is first-order or second-order accurate in the step size.

[EXCEPTION_BOUNDARY]:
- The exception hierarchy has two distinct roots: `NonConvergenceException` (iterative algorithm failures) and `NativeInterfaceException` (BLAS/LAPACK provider failures). The iterative root branches into `NumericalBreakdownException : NonConvergenceException`; the native root branches into `InvalidParameterException : NativeInterfaceException` (carries an integer code only in its message), `MemoryAllocationException : NativeInterfaceException` (no additional fields), and `SingularUMatrixException : NativeInterfaceException` (carries the 1-based index of the zero U diagonal entry in its formatted message string only — there is no typed `Element` property on the public surface). The optimization family is separate: `OptimizationException` (base), `MaximumIterationsException : OptimizationException`, `EvaluationException : OptimizationException` (carries `IObjectiveFunctionEvaluation ObjectiveFunction` — the last evaluation snapshot before failure), and `IncompatibleObjectiveException : OptimizationException`. None is `Error`-typed. The single boundary conversion point is `Error.New(Exception ex)`; raw exceptions must never propagate into domain code.
- `NumericalBreakdownException : NonConvergenceException` — catch it before `NonConvergenceException`. A catch ladder that catches the parent first silently swallows the more specific breakdown signal. Two `catch` arms, each producing a distinct `Fin.Fail` case, at the boundary and nowhere else.
- `SingularUMatrixException` carries no typed payload properties; the 1-based U diagonal index appears only in `ex.Message` as a formatted string. At the boundary, `Error.New(domainCode, ex.Message, Error.New(ex))` preserves both the element index (in the message) and the exception identity; attempting to extract a structured field that does not exist produces a compilation error. Map this to a domain-level `SingularMatrixError` distinct from a general internal breakdown status.
- `EvaluationException` is the only exception in the library that carries a typed evidence payload: its `ObjectiveFunction` property (type `IObjectiveFunctionEvaluation`) provides the point, gradient, and Hessian at the failing evaluation. A receipt factory that catches `EvaluationException` without extracting `ObjectiveFunction` discards the only post-failure diagnostic evidence the optimizer provides.
- `Error.New(int code, string message)` creates an `Expected` error (`IsExceptional` is `false`). `Error.New(Exception ex)` creates an `Exceptional` (`IsExceptional` is `true`). This distinction is load-bearing: algorithm convergence failures (`NonConvergenceException`, `OptimizationException`) surface as `Exceptional` errors wrapping the library exception exactly once at the boundary; domain-level precondition failures (rank deficit, non-positive-definite input) detected before invoking the library surface as `Expected` errors with domain-defined integer codes. Collapsing both families to `Expected` hides whether the error originated in a domain predicate or in library internals.
- `Error.New(int code, string message, Error inner)` chains errors with a parent code. Converting `NumericalBreakdownException` to `Error.New(domainCode, ex.Message, Error.New(ex))` preserves both the domain provenance code (for dispatch in receipt factories) and the original exception (accessible via `Error.Inner`). Flattening to a single `Error.New(ex)` discards the domain code; flattening to `Error.New(code, message)` discards the exception.
- `Error.Many(Seq<Error>)` and `Error.Many(Error[])` accumulate independent precondition failures — the correct surface for a receipt factory that validates multiple conservation identities simultaneously. `Error.Combine(Error other)` — accessible as `error1 + error2` via `operator +` — wraps both into a `ManyErrors` with left-to-right diagnostic order; the fold `errors.Fold(Error.Empty, (acc, e) => acc + e)` is an alternative path but not the only one. A factory that short-circuits on the first failure cannot distinguish "rank check failed" from "both rank and nullity checks failed independently." Accumulating with `Error.Many` and returning one `Fin.Fail(aggregated)` preserves the full violation set for diagnostic review.
- `Error.Filter<E>()` on a `ManyErrors` returns only errors assignable to `E`, and `Errors.None` if none match; on a single `Error`, it returns `this` if assignable, otherwise `Errors.None`. A re-admission check that needs to inspect only algorithm-specific error cases uses `receivedError.Filter<ExpectedError>()` to isolate expected domain errors and test `.HasCode(SolverErrorCodes.SingularU)`, leaving exceptional errors in the remainder — rather than pattern-matching the full `ManyErrors` graph.

[HONESTY_MARKERS]:
- `Symmetricity` as a stored receipt field is an honesty marker: `Symmetric` means the caller asserted real symmetry; `Hermitian` means conjugate symmetry; `Asymmetric` is explicit; `Unknown` means the library ran its own check. A receipt that omits this field makes the algorithm path ambiguous when the same receipt shape serves both symmetric and asymmetric cases.
- `QRMethod` stored on a QR receipt is a honesty marker: `Full` means the complete column space is preserved in `Q`; `Thin` means it was discarded. Re-admission of a thin-QR receipt for full-rank reconstruction is structurally invalid; any downstream factory that requires the full orthogonal complement must reject `Thin` receipts.
- `IObjectiveFunctionEvaluation.IsGradientSupported` and `IsHessianSupported` are honesty markers surfaced on every optimization evidence snapshot. A receipt that captures `IObjectiveFunctionEvaluation` must also capture these two flags; downstream Newton-based methods that inspect `Hessian` without checking `IsHessianSupported` silently receive `null`.
- `LevenbergMarquardtMinimizer` defaults all three tolerances to `1e-15`. When a receipt stores these three tolerances as `double`s and they all equal `1e-15`, the receipt cannot distinguish a caller who set them intentionally from one who accepted the defaults. An honesty marker `bool UsesDefaultTolerances` — set at construction when all three equal `1e-15` — is the minimum distinction needed for re-admission to validate the tolerance regime.
- `computeVectors` on `Svd<T>` construction controls whether `U` and `VT` are populated. When `false`, both are identity-shaped stubs. A receipt that embeds an `Svd<T>` computed with `computeVectors: false` must record this as a `bool VectorsComputed` flag; a consumer that reads `U` or `VT` without checking it silently receives stubs.
- `FourierOptions` is a `[Flags]` enum encoding two orthogonal honesty facts — exponent convention and scaling convention — as six named combinations: `Default` (`0`, symmetric `1/sqrt(N)` scaling on both directions), `InverseExponent`, `AsymmetricScaling` (alias `Matlab`, no scaling on forward and `1/N` on backward), `NoScaling`, and `NumericalRecipes` (`InverseExponent | NoScaling`, positive exponent and no scaling on either direction). These three conventions produce outputs that are scale-different by exactly `sqrt(N)` or `N` relative to each other. A spectral receipt must store the full `FourierOptions` bitfield as an `int` — not a prose string — because a bitfield preserves both orthogonal facts and the re-admission test is a bitwise equality check.
- `Fourier.ForwardReal` throws `NotSupportedException` when `FourierOptions.InverseExponent` is set; the packed real FFT surface supports only non-inverted exponent conventions. A receipt factory for real-signal forward transforms must gate on `(options & FourierOptions.InverseExponent) == 0` before invoking `ForwardReal` and return `Fin.Fail` with a typed domain error on violation — failing to gate promotes a domain precondition failure to an `Exceptional` error. `ForwardReal` packs the conjugate-symmetric spectrum into an array of length `N+2` (N even) or `N+1` (N odd). A receipt must record `N` (the time-domain sample count) separately from `data.Length` — using `data.Length` as a proxy for N produces an off-by-one in every downstream frequency bin computation. `FrequencyScale` places DC at index 0, positive frequencies at indices 1 through `length/2`, and wrapped negative frequencies at indices `length/2 + 1` through `length - 1`; for even-length N, the Nyquist bin at index `N/2` is simultaneously the most positive and most negative frequency representable — a receipt must flag this Nyquist-bin ambiguity for even-length transforms or downstream spectral analysis silently misinterprets the edge bin.
- `DirectRegressionMethod` selects among `NormalEquations` (Cholesky of XᵀX), `QR` (thin QR on the design matrix), and `Svd` (SVD on the design matrix). These have distinct numerical stability properties: `NormalEquations` squares the condition number, `QR` is stable for full-rank overdetermined systems, `Svd` handles rank-deficient systems. A fitting receipt must record the method used; re-admission into a downstream uncertainty quantifier carries different numerical guarantees depending on whether the coefficient vector was produced by Cholesky or SVD.
- `ManagedFourierTransformProvider.BluesteinSequenceLengthThreshold` is a public static field with value `46341` (greater than `sqrt(Int32.MaxValue)`). For transform lengths above this threshold, the Bluestein intermediate sequence uses double-precision multiplication of `(double)i * (double)i` to avoid `int32` `k*k` overflow — there is no `long`-indexed accumulation. A receipt wrapping a non-power-of-two FFT must record `N > 46341` as a boolean `UsesBluesteinLongPath` honesty marker: when `true`, the phase accumulation path differs from the short-path implementation and any residual-norm check on a round-trip transform is sensitive to this selection.
- `DescriptiveStatistics` constructor takes `bool increasedAccuracy = false` which switches internal computation from `double` to `decimal` (with overflow risk for large absolute values). After construction, the flag is not stored as a property — there is no `bool IncreasedAccuracy` equivalent on the instance. The computed `Mean`, `Variance`, `Skewness`, and `Kurtosis` are stored as `double` regardless of which path was used. Two instances from the same data with `true` versus `false` produce `double` properties that differ by up to O(`DoublePrecision`) on well-conditioned data but can diverge visibly on large-magnitude data near `decimal.MaxValue`. A receipt wrapping `DescriptiveStatistics` must record the flag separately at construction as a named `bool UsesDecimalArithmetic` field — the only moment the information exists.
- `MovingStatistics` is a windowed accumulator with a fixed capacity; it exposes the same `Count`, `Mean`, `Variance`, `Minimum`, `Maximum` surface as `RunningStatistics` but over a sliding window, not the full history. A snapshot of `RunningStatistics.Mean` at time T is the global mean; a snapshot of `MovingStatistics.Mean` is the window mean. A receipt must record which type produced the statistics as an honesty marker, because the two are API-compatible but semantically incompatible for receipt persistence.
- `TensorPrimitives.StdDev<T>` computes population standard deviation — `sqrt(sum((x_i - mean)^2) / N)` with the N normalizer, not the sample standard deviation with N-1. A receipt that stores a `double StdDev` computed via `TensorPrimitives.StdDev<double>` and annotates it as sample standard deviation has a structural honesty violation. `DescriptiveStatistics.Variance` and `RunningStatistics.Variance` both use the N-1 Bessel correction (unbiased); `TensorPrimitives.StdDev<T>` uses N (population). A receipt that accumulates statistics from multiple sources must carry a provenance field indicating the normalizer used — at minimum a two-case enum `{Population, Sample}` — or every downstream confidence computation is silently wrong.

[MEMOIZATION_LAW]:
- `AtomHashMap<K, V>.FindOrAdd(K key, Func<V> None)` performs a CAS-loop that atomically inserts only when absent and returns the existing or freshly-inserted value. `None` may be invoked more than once under contention, so the factory must be a pure construction function with no observable side effects. A memoization surface for numeric receipts — for example, caching eigendecompositions keyed on a matrix identity token — must supply a pure factory; any factory that acquires a lock, writes a file, or mutates external state is structurally incorrect.
- `AtomHashMap<K,V>.FindOrMaybeAdd(K, Func<Option<V>>)` is the conditional variant: when the factory returns `None`, no insertion occurs. This is the correct surface for a receipt cache that must not insert a failed-construction receipt — the factory returns `Option<TReceipt>.None` on precondition failure. The non-maybe overload `FindOrAdd(K, Func<V>)` has no suppression path; a factory that can fail must never be passed to it, because a receipt built with invalid inputs will be permanently cached on first failure.
- `Atom<A>.Swap(Func<A, A> f)` CAS-loops until the swap succeeds; `f` may be called multiple times. A receipt-accumulating fold stored in an `Atom<HashMap<K,V>>` must supply an `f` that merges the new receipt into the existing map purely via `HashMap<K,V>.Add`, not a mutation that acts on an external `Dictionary<K,V>`.
- `AtomHashMap<K, V>.SwapKey(K, Func<Option<V>, Option<V>>)` maps the current `Option<V>` — present or absent — to a new `Option<V>`: `Some(x) → Some(y)` updates, `Some(x) → None` removes, `None → Some(y)` inserts, `None → None` is a no-op. Passing `_ => None` removes the cached entry atomically within the CAS loop; a concurrent `FindOrAdd` racing with the invalidation sees either the old receipt or the absent slot, never a partially-written state.
- `AtomHashMap<K, V>.Change` is an event of delegate type `AtomHashMapChangeEvent<K, V>`, which is `delegate void(HashMapPatch<K, V> Patch)`. The `HashMapPatch<K, V>` parameter carries three `HashMap<K, V>` fields: `From` (the map before the mutation), `To` (the map after), and `Changes` (a map of changed keys to `Change<V>` discriminants that expose `HasAdded`, `HasRemoved`, `HasMapped`, and `ToOption`). For numeric receipt caches, subscribing to `Change` for staleness invalidation requires traversing `Patch.Changes` rather than reading key/old/new directly — the event delivers a patch structure, not a per-key triple.
- `Atom<A>` construction with a validator is available only through `Prelude.atom(value, validator)`, which returns `Option<Atom<A>>` — `None` when the initial value fails the validator, with no error code, message, or failure reason. A domain layer that surfaces atom construction failures on the numeric rail must convert `None` to `Fin.Fail(Error.New(domainCode, "initial state failed invariant"))` at the construction site; the library provides no error information to preserve.
- When `Atom<A>.Swap(f)` produces a value that fails the validator, the swap is silently abandoned and the atom retains the prior value. The return value is the prior value — identical to a successful swap that computed the current value — so there is no way to distinguish "validator rejected the swap" from "the function computed the current value" from the return value alone. `SwapMaybe(Func<A, Option<A>> f)` is the cleaner surface: when `f` returns `None`, the swap is skipped without ambiguity. The only pattern that fully eliminates ambiguity is to use `SwapMaybe` where `f` checks the invariant itself and returns `None` on failure, making the library validator redundant. A receipt-holding atom that encodes conservation invariants as a validator must be paired with an external `Fin<TReceipt>` check on the factory result, because a `Swap` that silently retains the old value is data loss with no error channel.
- Structural memoization of numeric receipts over atomic immutable maps requires weak coupling to the domain object that owns the key. When the key is derived from a native domain object (a mesh, a geometry handle, or a solver matrix that can be replaced without changing its pointer), the cache must use an identity token that encodes the content version — a content hash, a change counter, or a pair `(nativeHandle, generation)` — rather than the raw pointer, because a stale pointer that is recycled produces a cache hit that silently returns the wrong receipt. `AtomHashMap` CAS-safety does not extend across handle recycling; the key design is the owner's responsibility.

[GENERIC_MATH_SURFACE]:
- `TensorPrimitives` exposes generic operations with constraints that vary by operation: `Dot<T>`, `Norm<T>`, `CosineSimilarity<T>`, `FusedMultiplyAdd<T>`, `Sigmoid`, and `SoftMax` cover the hot-loop surface. For `double` and `float` these use hardware SIMD dispatch internally and are the BCL floor for any inner product, norm, or elementwise transform before reaching for MathNet's `Matrix<T>` operations.
- `TensorPrimitives.Dot<T>` has a permissive constraint (`IAdditionOperators`, `IAdditiveIdentity`, `IMultiplyOperators`, `IMultiplicativeIdentity`) that covers any algebraic ring, not just floating-point types. `Norm<T>` and `CosineSimilarity<T>` require `IRootFunctions<T>`, but `Dot<T>` does not. A receipt validity predicate for "residual within tolerance" over a custom ring type uses `TensorPrimitives.Dot` to compute the squared norm without requiring the full `IRootFunctions` surface, then applies the tolerance comparison manually — preserving exact arithmetic where `sqrt` would introduce rounding. For a `double[]` residual, `TensorPrimitives.Dot(residual, residual)` then `Math.Sqrt` is the BCL floor rather than constructing a full `DenseVector` and calling `.L2Norm()`.
- `TensorPrimitives.HammingDistance<T>` is unconstrained — it operates by equality comparison (not numeric comparison) and returns an `int` count of differing elements. Calling it on two `int[]` permutation arrays to verify bitwise identity produces the correct conservation check for permutation preservation without floating-point ambiguity; `HammingDistance == 0` is the canonical re-admission assertion for permutation-array identity across cached and recomputed factorizations.
- `ISparseSolverProvider<T>` (the generic overload) exposes one `Solve` method taking `(DssMatrixStructure matrixStructure, DssMatrixType matrixType, DssSystemType systemType, int rows, int cols, int nnz, int[] rowIdx, int[] colPtr, T[] values, int nRhs, T[] rhs, T[] solution)` and returning `DssStatus`. A receipt wrapping a sparse direct solve must record `DssMatrixStructure` and `DssMatrixType` as honesty markers: `DssMatrixStructure.Symmetric` versus `Nonsymmetric` determines which fill-reducing ordering was applied, and `DssMatrixType.PositiveDefinite` versus `Indefinite` determines whether Cholesky or LDLT was used internally. The non-generic `ISparseSolverProvider` exposes only lifecycle methods (`IsAvailable`, `InitializeVerify`, `FreeResources`) and carries no `Solve`; receipt factories must operate against the generic `ISparseSolverProvider<T>` overload.
- `Fourier.FrequencyScale(int length, double sampleRate)` generates the frequency axis in the same order as the FFT output: DC at index 0, positive frequencies up to Nyquist, then negative frequencies wrapped. A receipt that stores an FFT result without the sample rate and convention cannot produce a valid frequency axis; `(SampleRate, Length, Options)` are co-equal evidence alongside the transform output.

[HIERARCHICAL_RECEIPT_NESTING]:
- A composite solve receipt that embeds a child assembly receipt (sparse matrix assembly), a child factorization receipt (sparse Cholesky), and a child iterative refinement receipt (CG correction) satisfies the three-tier nesting law: the outer receipt owns the solve outcome and the conservation check (accepted plus rejected equals input count; residual norm at exit); the middle receipt owns the factorization evidence (`Factor`, `Determinant`, `DssMatrixType`, `DssMatrixStructure`); and the inner receipt owns the iterative trail (`IterationStatus`, residual trajectory, step count). Re-admission replays the conservation checks at each tier without re-executing any tier.
- Eigenvalue clamping in a receipt occurs when raw eigenvalues from `Evd<T>.EigenValues` lie outside an acceptable domain: negative eigenvalues on a nominally positive-semidefinite matrix, or complex eigenvalues with non-negligible imaginary parts on a nominally real-symmetric matrix. A receipt records the count of clamped eigenvalues as a per-receipt counter alongside the original eigenvalue vector — modifying the stored vector destroys the original-versus-mollified distinction needed for re-admission.
- `RunningWeightedStatistics` uses reliability weights (inverse-variance weighting), not frequency weights. `TotalWeight` is `_w1` (sum of weights); `EffectiveSampleSize` is `_w1² / _w2` (sum-of-weights squared over sum-of-squared-weights) and is directly available as a public property. `Variance` is `_m2 / _den` — the Bessel-corrected reliability-weight estimator where `_den = _w1 - _w2/_w1`; `PopulationVariance` is `_m2 / _w1` — no Bessel correction. A receipt recording `Variance` as a population variance, or `PopulationVariance` as a Bessel-corrected value, inverts the semantics and makes every downstream confidence interval derivation wrong without a compilation or runtime signal. For small effective sample sizes the Bessel correction in `Variance` is non-negligible; a receipt must carry both properties under unambiguous names alongside the `TotalWeight` and `EffectiveSampleSize` facts needed to audit the correction.

[RECEIPT_WIDTH_AND_FACTORY_DUALITY]:
- A receipt carrying more than eight distinct fields crosses the constructor-width pressure point. A nonlinear minimization receipt with `MinimizingPoint`, `Iterations`, `ExitCondition`, `Covariance`, `Correlation`, `StandardErrors`, `DegreeOfFreedom`, `GradientTolerance`, `StepTolerance`, `FunctionTolerance`, and `InitialMu` resolves through a required-evidence record (`Point`, `Iterations`, `Exit`, `DegreeOfFreedom`) and an `Option<StatisticalEvidence>` sub-receipt (`Covariance`, `Correlation`, `StandardErrors`) materialized only when `DegreeOfFreedom >= 1`. The sub-receipt is constructed independently and embeds its own conservation check (`StandardErrors.Count == Point.Count`). When `DegreeOfFreedom < 1` (underdetermined system), `NonlinearMinimizationResult.Covariance`, `Correlation`, and `StandardErrors` are all `null`; a receipt factory must gate on `DegreeOfFreedom >= 1` before treating the covariance as a meaningful fact.
- `NonlinearMinimizationResult` computes `Covariance = Hessian.PseudoInverse() * Value / DegreeOfFreedom` in its constructor via `EvaluateCovariance`, using the SVD pseudoinverse. For near-rank-deficient Jacobians, the pseudoinverse is well-defined but produces a covariance matrix whose diagonal entries for degenerate directions are numerically large rather than zero. A receipt that gates on `Covariance != null` without recording whether any singular value fell below the pseudoinverse threshold emits statistically credible-looking but inflated standard errors for rank-deficient models. `NonlinearMinimizationResult` does not expose the Hessian SVD rank; a domain receipt wrapper that recomputes `Hessian.Svd(false).Rank` post-construction is the only path to the effective parameter count. Additionally, `EvaluateCovariance` calls `objective.EvaluateAt(objective.Point)` unconditionally — a second objective evaluation at construction time; if the objective function has side effects, this second evaluation fires them. A receipt wrapping `NonlinearMinimizationResult` must be constructed immediately after the minimizer returns, before any external state the objective reads can change.
- Factory/admission duality: a receipt constructor is private; a public static `Create` factory validates preconditions — `Rank > 0`, singular values non-empty and monotone-descending — and returns `Fin<TReceipt>`. Re-admission takes a serialized or cached receipt, replays the structural invariant checks against the stored payload, and returns `Fin.Fail` on violation. The re-admission path never re-runs the decomposition; it is O(n) in the receipt field count, not O(n³) in the matrix dimension.
- `GoodnessOfFit.RSquared` computes the square of the Pearson correlation coefficient between modelled and observed values — explicitly not the coefficient of determination R². `GoodnessOfFit.CoefficientOfDetermination` computes R² = 1 − SS_res/SS_tot. These coincide only for simple linear regression with an intercept; they diverge for polynomial fits, no-intercept fits, and nonlinear models. A fitting receipt must store both quantities under unambiguous names and record the `degreesOfFreedom` used in `GoodnessOfFit.StandardError`. `GoodnessOfFit.PopulationStandardError` uses an N normalizer with no Bessel correction; `GoodnessOfFit.StandardError` takes an explicit `degreesOfFreedom` parameter and uses `N - degreesOfFreedom`. The distinction is load-bearing for confidence interval construction.
- `IPrecisionSupport<T>` with `Norm()` and `NormOfDifference(T)` is the interface that makes `AlmostEqualNorm<T>` work on custom types. A domain value type that participates in numeric tolerance comparisons implements this interface to unlock the full `Precision` extension method surface. A receipt validity predicate that hand-rolls `Math.Abs(a - b) < tol` on a custom type bypasses the named-tolerance ledger.

[DISPATCH_GRAMMAR_AND_COLLAPSE_LAW]:
- Behavior-carrying enumerations are the first tier: `DssMatrixStructure`, `DssMatrixType`, `QRMethod`, `Symmetricity`, `StepType`, `DirectRegressionMethod`, `FourierOptions`, and `ExitCondition` each carry behavior that must not be re-derived at call sites. A receipt factory that switch-dispatches on these enumerations rather than storing them as policy values re-derives the behavior and breaks re-admission.
- Closed unions with generated state-threaded switches are the second tier. When a receipt family grows beyond three parallel result shapes — for example, a `FactorizationReceipt` with separate `CholeskyCaseReceipt`, `LUCaseReceipt`, `SvdCaseReceipt`, and `EvdCaseReceipt` branches — collapsing to a `[Union]`-owned dispatch surface with one state-threaded `Switch` eliminates the four-arm match repeated at every downstream consumer. The collapse law: ≥3 parallel receipt sub-types modeling one factorization concept trigger the move; the receipt cases become `[Union]` cases, and `Switch(state, static (s, c) => ...)` owns all evidence extraction paths with no shared outer `Op` across arms — each arm constructs its own provenance.
- Runtime-type-directed projection ladders are the third tier. When receipt consumers must project a heterogeneous result sequence — for example, a pipeline that maps `IterationStatus` through four possible solver receipt types onto a unified `SolveEvidence` — and that ladder repeats across a dozen result owners, the collapse law applies: extract the projection as a `Func<TReceipt, SolveEvidence>[]` dispatch table keyed on a bounded vocabulary, and every owner dispatches through the table rather than hosting its own switch. The collapse is mandatory when the same ladder structure recurs in three or more distinct owner classes; the result is one table that owns all projection arms, satisfying `DERIVED_LOGIC`.

[COUNTING_DEGENERACY]:
- Per-receipt degeneracy counters make "counting not crashing" possible: the count of clamped eigenvalues, the count of mollified mesh edges in a geometric assembly receipt, the count of rejected weights in a quadrature receipt, the count of fallback-path function evaluations, and the count of Bluestein-path FFT blocks each belong in a named `int` field on the owning receipt. A receipt that omits these counters loses the only signal that distinguishes a well-conditioned run from one that silently degraded through multiple fallbacks.
- Degenerate element counting in sparse assembly: when building a stiffness or mass matrix, elements whose local stiffness determinant falls below a policy threshold are mollified rather than rejected. The receipt records `DegenerateElementCount` alongside `TotalElementCount`; the conservation identity `DegenerateElementCount + WellConditionedElementCount == TotalElementCount` is the re-admission check. A receipt that stores only the final assembled matrix without these counters cannot be audited for silent mollification.
