# Events and Threads â€” Edge Findings

[IOUNINTERRUPTIBLE_MECHANISM_AND_SCOPE]:
- `IO<A>.Uninterruptible()` constructs `IOUninterruptible<A,A>`, whose `Invoke` creates `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` â€” passing `CancellationToken.None` and `null` for the source. Because source is null, `New` allocates a fresh `CancellationTokenSource`, sets Own bit 1, and because `CancellationToken.None.CanBeCanceled` is false, installs no parent-propagation registration. The computation runs with an uncancellable token that is completely decoupled from the parent scope. Resources are shared (Own bit 2 = 0). On completion, the child env's `DisposeNonResources` disposes only the locally owned CTS â€” resources are never touched.
- The `IOUninterruptibleAsync` form (for async operations) receives the `LocalEnv` used during the synchronous start, carries it forward, and awaits the task. The uncancellable token is live across the full async span. A computation that awaits on a host service internally polling its own token receives the None token â€” if that service uses `env.Token` to gate its loop, it will never observe cancellation. This is intentional but means finalization paths must themselves complete within finite time.
- The distinction between `Uninterruptible` and `Bracket`/`BracketFail` is ownership of the resource scope: `Uninterruptible` shares the parent resource scope and only isolates the cancellation token. `Bracket()` creates a child resource scope via `WithEnv(env => env.LocalResources)` and disposes it on success and failure. `BracketFail()` creates a child resource scope via `WithEnvFail(env => env.LocalResources)` and disposes it only on failure. For teardown sequences that must not be interrupted AND must clean up their own subscriptions, the correct composition is `Uninterruptible` wrapping a `Bracket` whose finalizer does the detach-and-dispose sequence.

[ENVIO_OWNERSHIP_BITS_AND_SCOPE_INVARIANTS]:
- `EnvIO.New` assigns ownership via two bits in the `Own` field: bit 1 means "I allocated the CTS and will dispose it"; bit 2 means "I allocated the Resources and will dispose them". `DisposeResources` disposes only if bit 2 is set; `DisposeNonResources` disposes the CTS only if bit 1 is set and always disposes the stored `CancellationTokenRegistration`.
- `env.LocalResources` calls `New(null, Token, Source, SyncContext)` â€” source is the parent's `Source`, not null â€” so bit 1 is NOT set and the parent CTS is never disposed. `resources` is null so bit 2 IS set; a fresh `Resources` with the parent as its `parent` field is allocated and will be disposed on child env disposal. This is the child resource scope pattern: child subscriptions accumulate there and are disposed when the `using EnvIO local = ...` exits in `IOLocal.Invoke`.
- `env.LocalCancel` calls `New(Resources, Token, null, SyncContext)` â€” `resources` is the parent's `Resources`, not null â†’ bit 2 NOT set. `source` is null â†’ bit 1 set; a new CTS is allocated with the parent token registration installed to propagate cancellation. Disposal of the child env disposes the new CTS and the registration but leaves parent resources intact.
- `IOLocal.Invoke` uses `using EnvIO local = MapEnvIO(envIO)` â€” the `using` calls both `DisposeResources` and `DisposeNonResources`. For a `LocalResources` scope, this disposes the child resource scope and the (not-owned) non-resources remain untouched. For a `LocalCancel` scope, this disposes the child CTS and the registration. When a subscription depends on a resource allocated in the parent scope, it is safe to detach it inside the child scope's computation because parent resources survive child env disposal.
- `IOLocalOnFailOnly.Invoke` does NOT use `using` on the child env. On success: it calls only `local.DisposeNonResources()` in the `finally` block â€” the CTS is disposed but resources are retained. On failure: `local.DisposeResources()` in the `catch` block releases the child resource scope, then `finally` disposes non-resources. This is the verified implementation of `BracketFail` / `WithEnvFail`.

[IASYNC_DISPOSABLE_IN_RESOURCES_SYNCHRONOUS_HAZARD]:
- `Resources.AcquireAsync<A>(A value) where A : IAsyncDisposable` stores a `TrackedResourceAsyncDisposable<A>`, whose `Release()` returns `IO.liftAsync(async delegate { await Value.DisposeAsync().ConfigureAwait(false); return unit; })`. `Resources.DisposeU` calls `resource.Value.Release().Run(envIO2)` synchronously. `IO.Run` drives the async computation to completion via `GetAwaiter().GetResult()`, blocking the disposing thread until `DisposeAsync()` completes.
- The hazard: `DisposeU` is called from `IOLocal.Invoke`'s `using` block and from `IOLocalOnFailOnly.Invoke`'s `catch`. Both execute on an async state machine continuation, which may be running on the captured `SynchronizationContext`. If `DisposeAsync()` needs to marshal back to the same context for completion, the `.GetResult()` block produces a deadlock. Correct form: resources whose `DisposeAsync` marshals to a specific context must not be acquired inside a `WithEnv` or `BracketFail` block that runs on that context. Use `Prelude.use(() => new T(...))` with explicit `release` outside any scope that runs on the target context.
- When `TrackedResourceAsyncDisposable` coexists with `TrackedResourceDisposable` in the same `Resources`, disposal order between them is undefined (hash-trie traversal). If an async-disposable must be released before a synchronous disposable (e.g., flush before close), encode the order explicitly by calling `release` in sequence inside the computation rather than relying on scope disposal.

[REPEAT_AND_RETRY_RESOURCE_SEMANTICS]:
- `IO<A>.RepeatUntil(Func<A,bool> predicate)` is `Bracket().Bind(v => predicate(v) ? IO.pure(v) : RepeatUntil(predicate))`. Each iteration wraps the body in `Bracket()` = `WithEnv(env => env.LocalResources)`, which creates a child resource scope that is disposed after each iteration. Resources acquired inside the body are released at the end of each cycle. A persistent subscription must be acquired OUTSIDE the repeat body â€” opened at the level of the `IO` chain that contains the `Repeat`, not inside the repeated computation.
- `IO<A>.RetryUntil(Func<Error,bool> predicate)` is `Catch(BracketFail(), ...)`. `BracketFail()` = `WithEnvFail(env => env.LocalResources)`: resources are released only on failure. On success, the iteration's resources are retained in the parent scope. A retry loop that opens a connection on each attempt accumulates one live connection (the last successful one); each failed attempt cleans up. This asymmetry is load-bearing: the first success's resource escapes the BracketFail scope into the parent.
- Both `Repeat` and `Retry` are CPS-recursive chains (`Bind` on `IO`), not iterative loops. Each iteration adds one frame depth to the `IO` DSL evaluation chain. For schedules with thousands of repetitions, this builds a deeply nested `IOBind`/`IOFold` structure evaluated at `Run` time. `IO<A>.Fold(Schedule, S, Func<S,A,S>)` is the iteration-free fold worker: `IOFold<S,A,S>` encapsulates the schedule iteration as a state machine rather than a recursive `Bind` chain and is the correct form for long-running fold workers.

[ATOM_CHANGE_DELEGATE_LIFECYCLE_AND_GC]:
- `Atom<A>.Change` is a raw C# multicast delegate field (`event AtomChangedEvent<A>? Change`). Subscribing with `atom.Change += handler` increases the delegate's invocation list; detaching with `atom.Change -= handler` requires the exact same delegate object. A lambda registered as `atom.Change += v => SomeAction(v)` creates an anonymous delegate whose reference is lost immediately â€” detach is impossible without first capturing the delegate to a variable.
- The `Atom<A>` instance holds a strong reference to every registered delegate. The delegate holds a strong reference to its target (the subscriber's `this`). This forms a reference chain from the live atom to every subscriber. If the atom is a long-lived static cell and subscribers are activation-scoped objects, the atom prevents GC of those objects even after deactivation. The invariant is: detach all `Atom.Change` subscriptions at deactivation using the exact stored delegate reference.
- `Atom.Swap` fires `this.Change?.Invoke(val2)` synchronously after the CAS succeeds, before `Swap` returns. If the invocation list is long, every handler on it runs on the swapping thread before `Swap` returns. A handler that calls `Swap` on the same atom re-enters the CAS path from within the change event â€” the handler itself is not re-entrant for the change event, but the atom is not locked during handler execution. A concurrent swap from another thread can succeed between two sequential handler invocations in the same `?.Invoke` call chain, so late handlers in the list observe a different committed value than early ones if a concurrent swap wins after the first handler ran.

[CTS_TRYRESETAND_POOLING_CONSTRAINTS]:
- `CancellationTokenSource.TryReset()` returns `true` and unregisters all callbacks via `Volatile.Read(in _registrations)?.UnregisterAll()` only when `_state == States.NotCanceledState` AND (`_timer == null` OR the timer has not yet fired: `timer.Change(MaxValue, MaxValue)` succeeds AND `!timerQueueTimer._everQueued`). The `_everQueued` flag is set when the timer is actually queued to fire; once set it is never cleared. A CTS used with `CancelAfter` where the timeout elapsed sets `_everQueued = true` â€” `TryReset` will return `false` for it forever.
- The contract for safe CTS pooling: (1) `TryReset` returns `true`; (2) the token was never cancelled; (3) `UnregisterAll` has run â€” all registered callbacks are removed before reuse. Registrations added after `TryReset` see the fresh uncancelled state. `EnvIO.New` registers a parent-propagation callback at construction time (`token.Register(delegate { source.Cancel(); })`); for a pooled CTS to be reused in `EnvIO`, `TryReset` must be called before re-passing it to `New`, and then `New` reinstalls the registration against the fresh token.
- A cancelled `CancellationTokenSource` cannot be reset â€” `TryReset` returns `false`. Once the CTS transitions to the cancelled state, it is spent; create a new one. The pooling value is for CTS instances created with `CancelAfter` that are cancelled BEFORE the timer fires (e.g., the operation completed before the deadline): `CancelAfter`â†’`TryReset` before fire is the only poolable pattern.

[MANUAL_RESET_VALUE_TASK_SOURCE_CORE_AS_ZERO_ALLOC_EVENT]:
- `ManualResetValueTaskSourceCore<TResult>` (a `struct` in `System.Threading.Tasks.Sources`) is the zero-allocation foundation for reusable async events. `Reset()` clears all fields and increments `Version` (`_version++`); an awaiter holding a stale version number will get `ArgumentOutOfRangeException` from `GetResult` or `GetStatus` â€” the version token is the stale-detection mechanism. `SetResult` and `SetException` call `SignalCompletion`, which fires the registered continuation.
- `SignalCompletion` dispatch: when `RunContinuationsAsynchronously = false`, the continuation `Action<object>` is invoked inline on the completing thread before `SetResult` returns â€” same reentrancy hazard as `ChannelOptions.AllowSynchronousContinuations = true`. When `RunContinuationsAsynchronously = true`, continuation is submitted via `ThreadPool.UnsafeQueueUserWorkItem(action, state, preferLocal: true)` â€” no `ExecutionContext` capture, completion on a pool thread local to the completing thread's core. The `preferLocal` flag is significant for locality: a producer that sets the result and a consumer awaiting on the same core will likely resume on the same pool thread.
- The intended seam use: a host event that fires on a specific thread (not a managed thread pool thread) implements `IValueTaskSource<bool>` over a `ManualResetValueTaskSourceCore<bool>`, calls `Reset()` to prepare a new wait cycle, and calls `SetResult(true)` inside the event callback. The consumer `await`s a `new ValueTask<bool>(source, source.Version)`. After each signal, `Reset()` must be called before the next consumer `await` â€” calling `Reset()` while an awaiter is active corrupts the version, so the protocol requires one-producer, one-consumer, reset-before-next-wait discipline.

[EXECUTIONCONTEXT_RESTORE_VS_RUN_AT_SEAM]:
- `ExecutionContext.Restore(ctx)` is a one-way install: it sets the current thread's execution context to `ctx` without saving the prior context or restoring it afterward. It is non-bracketed and intentionally "unsafe" in the same sense as `UnsafeRegister` â€” it is the building block for async state machines that must restore a captured context at `await` resumption points without the overhead of `Run`'s save-restore pair. Calling `Restore` in application-level code outside an async state machine produces a context leak: the thread continues with the installed context after the call site.
- `ExecutionContext.Run(ctx, callback, state)` is bracket-shaped: it saves the prior context, installs `ctx`, invokes the callback, and restores the prior context in a `finally` block. The context snapshot taken at save time is what the thread returns to, regardless of what `callback` does to the thread's context internally. This is correct for event-dispatch wrappers that must not leak the event's context onto the calling thread after dispatch.
- The seam law: event dispatch wrappers use `Run` when the callback must not leak its context outward. `UnsafeRegister` callbacks that do only context-free work (channel write, atom swap) need neither `Run` nor `Restore` because they carry no ambient context semantics. Only callbacks that consume or propagate `AsyncLocal<T>` values need `Run`. Using `Run` in a high-frequency callback introduces allocation (context copy) and full restore overhead; profiling under load is warranted before adding `Run` to a tight event path.
- `Activity.Current` is backed by `AsyncLocal<Activity> s_current`. An event callback registered from a thread carrying an active `Activity` will inherit that activity when the callback fires (subject to `ExecutionContext` flow). A host event that fires on a dedicated thread without a managed context (e.g., a native callback via P/Invoke thread) carries `null` `Activity.Current`. `UnsafeRegister` callbacks skip EC restore entirely, so `Activity.Current` inside such a callback is the `Activity` current on the cancelling thread â€” neither the registration thread's activity nor the consuming thread's activity. The seam law: set `Activity.Current` explicitly inside the callback or use `Run` with the captured context if the callback must operate under a specific activity.

[RENDEZVOUSCHANNEL_INTERNAL_MECHANICS]:
- `RendezvousChannel<T>` (the internal type from `Channel.CreateBounded<T>(0)`) has `CanCount = true`, `Count` always returns `0`, and `CanPeek = true`. `TryPeek` inspects the head of the blocked-writers queue under a lock and returns the item waiting for handoff â€” it reveals what the first blocked writer wants to hand over without consuming it. `TryRead` pops the head writer from the queue, returns its item, and signals the writer's completion by calling `blockedWriteAsyncOperation.DangerousSetResult(default(VoidResult))`. The writer's `WriteAsync` then unblocks.
- The rendezvous is lock-mediated: both the writer and reader acquire the same `SyncObj` lock at the critical section. The `RendezvousChannelReader._readerSingleton` and `_waiterSingleton` are pooled, pre-allocated `BlockedReadAsyncOperation<T>` and `WaitingReadAsyncOperation` objects â€” reused across calls when the channel has `SingleReader`-like usage (one reader at a time). This reduces allocation for the common single-reader-single-writer rendezvous pair.
- The semantic invariant is strictly: `Count == 0` always (the channel never buffers). Using `TryRead` without a prior `WaitToReadAsync` returns `false` if no writer is blocked â€” the reader must be waiting before the writer arrives, or vice versa. The rendezvous is a synchronization point, not a queue; its correct use is explicit phase handoff between two independent async workers.

[IOTHREADPOOL_WORK_ITEM_BOXING_INVARIANT]:
- `ThreadPool.UnsafeQueueUserWorkItem(IThreadPoolWorkItem callBack, bool preferLocal)` routes to `UnsafeQueueUserWorkItemInternal(object callBack, bool preferLocal)` â€” the work item is stored as `object`. A `struct` implementing `IThreadPoolWorkItem` is boxed at the call site; the single box becomes the work item. This is one allocation â€” better than a delegate wrapper plus closure â€” but still allocates. The allocation-free path for struct state is `ThreadPool.UnsafeQueueUserWorkItem<TState>(Action<TState>, TState, preferLocal)` with a struct `TState`; the generic overload stores the struct state inline in `QueueUserWorkItemCallbackDefaultContext<TState>` without boxing the state itself.
- The `IThreadPoolWorkItem` interface (`void Execute()`) is the correct form when the work item IS the context (carries its own state) and needs to be cancellable: implement `Execute` to check a volatile flag before doing work. The `UnsafeQueueUserWorkItem(IThreadPoolWorkItem, preferLocal)` form skips EC capture and delivers `Execute` directly to the work-stealing queue. Structures implementing this interface without closures achieve the minimum allocation path for per-event work submission.

[SIGNAL_AND_COUNTDOWN_LIFECYCLE_AT_SEAM]:
- `Signal.autoReset<M>()` and `Signal.manualReset<M>()` both call `Prelude.use(() => new Signal<M>(new EventWaitHandle(...)))` â€” the `Signal<M>` instance is wrapped by `use` and tracked in the current `Resources` scope. The underlying `EventWaitHandle` is disposed when the resource scope is released. A `Signal` consumed inside a `Repeat` body is disposed after each iteration; a `Signal` used as a cross-iteration synchronization point must be acquired OUTSIDE the `Repeat` computation.
- `CountdownSignal<M>.Wait()` calls `handle.Wait(e.Token)` â€” this is `CountdownEvent.Wait(CancellationToken)`, which blocks the thread pool thread (or dedicated thread) until the count reaches zero or the token fires. If called inside an `IO` chain running on a thread-pool thread, it occupies that thread for the duration. For high-count drains, prefer `PeriodicTimer`-driven polling of `handle.IsSet` or submit the wait to a dedicated thread via `Fork`.
- `Signal<M>.Wait()` calls `IO.liftAsync(e => handle.WaitOneAsync(e.Token))`. `WaitOneAsync` uses `ThreadPool.RegisterWaitForSingleObject` (with EC capture) â€” not the `Unsafe` variant. The resumption carries the `ExecutionContext` from the `await` site inside the `liftAsync` lambda, not from the thread that called `SetResult` on the TCS. This is correct for most seam uses but incurs one EC capture per `Wait` call.

[FORK_SYNC_CONTEXT_AND_POST_ALLOCATION_PRESSURE]:
- `IO<A>.Fork` passes `env.SyncContext` to the child `EnvIO.New(resources, token, tsrc, env.SyncContext)`. The forked `LongRunning` thread carries the parent's captured sync context. Any `IO.Post()` call inside the fork targets the same context as the parent â€” if the parent captured a UI thread context, forked work that calls `Post()` marshals back to the same UI thread. A fork spawned to protect the UI thread from blocking work that still uses `Post()` does not reduce context-switch count; it only moves the blocking off the UI thread.
- `IO<A>.Post()` allocates an `AutoResetEvent` per invocation (an OS kernel handle) and calls `SynchronizationContext.Post` (queue, non-blocking) followed by `await wait.WaitOneAsync(env.Token)`. `WaitOneAsync` uses `ThreadPool.RegisterWaitForSingleObject` internally â€” another OS kernel registration. This is three kernel transitions per `Post()` call (Post, WaitForSingleObject registration, Set). For per-frame or per-event dispatch at high frequency, `Post()` is not viable; use `Channel<T>` or `AtomHashMap` state cells to batch work and marshal the batch once.
- When `env.SyncContext == null`, `Post()` falls through to `return await RunAsync(env)` â€” the computation runs on the current scheduler, inline, with no context switch. Code that relies on `Post()` for thread safety while running in a null-context environment (e.g., background workers in tests or CLI processes) silently gets inline execution and no actual marshaling. Seam contracts that require specific thread affinity must verify `env.SyncContext != null` before relying on `Post()` for enforcement.

[RESOURCES_DISPOSAL_ORDER_AND_ORDERED_TEARDOWN]:
- `Resources.DisposeU` iterates the `AtomHashMap<object, TrackedResource>` and calls each resource's `Release().Run(envIO2)` synchronously. `AtomHashMap` iteration follows HAMT (Hash Array Mapped Trie) traversal order, which is determined by the hash values of the keys â€” effectively arbitrary from the caller's perspective. There is no LIFO, FIFO, or insertion-order guarantee.
- For teardown sequences where a subscriber must be detached before its connection is disposed: acquire resources with explicit `use`/`release` pairs in sequence. The subscription token is obtained first, but its `release` must be called explicitly inside the computation before the connection `use` exits. Do not rely on scope disposal order for ordered teardown â€” encode the order as a sequential `IO` chain:
```csharp
from conn  in use(IO.lift(OpenConnection))
from token in use(IO.lift(() => Subscribe(conn)))
from _     in IO.yieldFor(Duration.Zero) // ... actual work ...
from __    in release(token)             // explicit detach first
select unit                              // conn release follows scope exit
```
- `TrackedResourceDisposable<A>` calls `Value.Dispose()` synchronously. `TrackedResourceAsyncDisposable<A>` calls `await Value.DisposeAsync().ConfigureAwait(false)` wrapped in `IO.liftAsync`, then `Run(envIO2)` blocks via `.GetAwaiter().GetResult()`. This synchronous block-on-async is only safe when the disposing thread is NOT the SynchronizationContext target. Disposing an async-disposable resource from the UI thread via scope disposal is a deadlock risk when the resource's `DisposeAsync` schedules continuation back to the same UI thread.
