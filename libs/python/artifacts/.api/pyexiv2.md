# [PY_ARTIFACTS_API_PYEXIV2]

`pyexiv2` is the unified in-process EXIF / IPTC / XMP / ICC / comment / thumbnail read-write engine for the artifacts descriptive-metadata rail â€” a thin Python binding over the Exiv2 C++ library (the same metadata core Darktable/digiKam embed) that collapses the four-provider raster fold (`exif` EXIF scalars + `iptcinfo3` IIM block + `python-xmp-toolkit`/`libxmp` XMP packet + `pillow` `ImageCms` ICC header) into ONE object parse: a single `ImageData(payload)` (or `Image(path)`) handle exposes `read_exif`/`read_iptc`/`read_xmp`/`read_icc`/`read_comment`/`read_thumbnail` as dict/str/bytes projections keyed by the Exiv2 family-form tag grammar (`Exif.Image.Artist`, `Iptc.Application2.ObjectName`, `Xmp.dc.creator`), `modify_*(dict)` writers where a `None` value deletes one tag, `clear_*` namespace scrubbers, and `get_bytes()` re-encode egress â€” every standard read and written through one native pass with no Python-side IIM/EXIF/XMP codec. It is SUPERSEDED on the `exchange/metadata#METADATA` `RASTER` carrier â€” the rebuilt `Metadata` owner folds EXIF/IPTC/XMP/ICC in ONE cross-format `pyexiftool` pass â€” and carries no manifest row; this catalog documents its unified in-process Exiv2 surface (one `ImageData` parse over EXIF + IPTC + XMP + ICC replacing a four-import + `NamedTemporaryFile` round-trip) as API reference, not a live path. The owner composes `ImageData(payload, encoding="utf-8")` construction under the `with` context, the `read_exif`/`read_iptc`/`read_xmp`/`read_icc` projections folded through the one `_FIELD_KEYS` correspondence into the single `MetaFacts.from_logical`, the `modify_exif`/`modify_iptc`/`modify_xmp` + `clear_*` writers under the `MetaBind` disposition, `get_bytes()` re-encode into the metadata carrier evidence triple, and module-level `registerNs(uri, prefix)`/`set_log_level(2)` namespace+log setup; it never re-implements the Exiv2 tag codec, the IIM/XMP/EXIF serialization, the cross-standard `convert_*` mapping, or the file framing the native core owns, and the raw `Image`/`ImageData` handle never crosses the owner boundary â€” only the projected `dict[str, object]` logical fold and the `get_bytes()` payload do.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: the two metadata handles
- rail: metadata

`Image` (file-path-backed) and `ImageData` (in-memory bytes-backed) are the two public objects, sharing the identical read/modify/clear/copy method surface â€” the only difference is construction source and egress (`ImageData.get_bytes()` returns the mutated buffer; `Image` writes through to the file in place). The owner uses `ImageData` exclusively: the `exchange/metadata#METADATA` carrier contract is bytes-in/bytes-out, so `ImageData(payload)` parses the in-memory artifact and `get_bytes()` returns the re-encoded payload with ZERO filesystem round-trip â€” the `NamedTemporaryFile(suffix=".jpg")` dance the `iptcinfo3` arm needs (because that module is path-only) disappears under pyexiv2. Both are context managers (`__enter__`/`__exit__` â†’ `close()`); the native handle MUST be closed (it holds a C++ object), so the owner always parses under `with`. There is no public Python exception hierarchy: Exiv2 errors surface as `RuntimeError` carrying the C++ message (`"No namespace info available for XMP prefix ..."`, `"the image size exceeds ..."`), which the owner traps as `(RuntimeError, OSError, ValueError)` at the companion seam â€” best-effort enrichment, never a typed package rail.

[PUBLIC_TYPE_SCOPE]: the tag-key grammar (data, not types)
- rail: metadata

Exiv2 keys are the closed three-part family grammar `<family>.<group>.<tagname>` â€” the vocabulary the owner's `_FIELD_KEYS` rows map their logical names onto. Reads return the family-prefixed key as the dict key; writes take the same.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: construct, read, modify, clear, egress
- rail: metadata

`ImageData(data, encoding="utf-8")` parses on construction (the native Exiv2 reader runs once over the in-memory buffer); the `encoding` governs how bytes-typed tag values decode to `str` (the owner passes `"utf-8"`). `read_exif`/`read_iptc`/`read_xmp` each return a `dict[str, str | list | dict]` of the present tags in that family â€” scalar values as `str`, IPTC repeatable datasets and XMP Bag/Seq as `list[str]`, XMP LangAlt as `dict[lang, str]` â€” so one carrier read is three native projections folded into the owner's one logical map (versus the four imports + temp-file the provider split needs). `read_icc()` returns the raw ICC profile `bytes` (the owner hands these to `pillow` `ImageCms` for the header parse, or carries them forward on re-encode), `read_comment()` the JPEG COM `str`, `read_thumbnail()` the embedded preview `bytes`. The `read_*_detail` siblings return typed rows (`{key: {"type": ..., "value": ...}}`) for the rare cases the owner needs the Exiv2 type tag rather than the coerced value. `modify_exif`/`modify_iptc`/`modify_xmp` take a `dict` and apply it tag-by-tag â€” a `None` value DELETES that tag (the per-field `MetaBind.STRIP`/scrub primitive), any other value sets it (lists for repeatable IPTC/XMP, plain strings for scalars). `clear_exif`/`clear_iptc`/`clear_xmp` scrub a whole family (the namespace-level `REPLACE`/`STRIP` primitive). `get_bytes()` returns the re-encoded payload after the modifies â€” the owner's egress.

[ENTRYPOINT_SCOPE]: module-level namespace + conversion + log surface
- rail: metadata

`registerNs(namespace: str, prefix: str)` registers a custom XMP namespace URI under a prefix BEFORE a `modify_xmp` writes a tag in it â€” writing `Xmp.<prefix>.<tag>` for an unregistered prefix raises `RuntimeError("No namespace info available for XMP prefix ...")`, so the owner pre-registers any non-standard namespace (the `Iptc4xmpExt:DigitalSourceType` content-origin namespace the metadata page composes is one such registration) at module-load scope, once per process, never per parse. `set_log_level(level=2)` mutes the Exiv2 native warning chatter to stderr (`0`=debug â€¦ `4`=mute; the owner sets `2`=warn-and-up so a malformed tag logs without flooding); it is a global call made once at boundary scope. The four `convert_*` functions cross-walk a tag dict from one standard to another (`convert_exif_to_xmp({"Exif.Image.Artist": ...}) -> {"Xmp.dc.creator": ...}` etc.) using Exiv2's own metadata-mapping tables â€” the native primitive that lets the owner author one logical field once and project it to the EXIF + IPTC + XMP spellings without hand-maintaining a third correspondence beyond `_FIELD_KEYS`.

- read fold: `_pyexiv2_read` opens `with pyexiv2.ImageData(payload, encoding="utf-8") as img`, folds `img.read_exif()`, `img.read_iptc()`, and `img.read_xmp()` into one `dict[str, object]` keyed by Exiv2 family-form key, projects each through the `_FIELD_KEYS[logical].exif`/`.iptc`/`.xmp` Exiv2-key column into the logical name â€” the IPTC `Iptc.Application2.Keywords` and XMP `Xmp.dc.subject` repeatable datasets arriving already as `list[str]` (no `;`/`,` split the libvips/av string forms need), the XMP LangAlt `Xmp.dc.title` dict reduced to its default-language value, the EXIF GPS `Exif.GPSInfo.GPSLatitude` rational triple folded through `_dms` with its `GPSLatitudeRef` into the signed `Place.gps` pair â€” then folds `img.read_icc()` through the existing `_icc` `pillow` parse into the `Color` header and materializes the one `MetaFacts.from_logical`. - write fold: `_pyexiv2_write` opens `with pyexiv2.ImageData(payload, encoding="utf-8") as img`, on `MetaBind.REPLACE`/`STRIP` calls `img.clear_exif()`/`img.clear_iptc()`/`img.clear_xmp()` (the per-family scrub, PRESERVING the ICC profile unless STRIP also `clear_icc()`s it â€” `Color` write parity the libvips `strip=` flag lacks granularity for), on a non-STRIP bind builds the per-standard modify dicts from `_flat(facts)` through the `_FIELD_KEYS` Exiv2-key columns (a `None` value per field the page wants deleted, the EXIF rationals bound as the Exiv2 `num/den` string the native type round-trips, the `facts.descriptive.keywords` tuple bound as a `list` to `Iptc.Application2.Keywords` and `Xmp.dc.subject`), calls `img.modify_exif(exif_dict)` / `img.modify_iptc(iptc_dict)` / `img.modify_xmp(xmp_dict)`, then `img.get_bytes()` returns the re-encoded payload.
