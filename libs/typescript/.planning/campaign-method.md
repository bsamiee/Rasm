# [TS_CAMPAIGN_METHOD]

The branch-local campaign method for `libs/typescript`, a first-class sibling library branch named in the suite [campaign-method](../../csharp/.planning/campaign-method.md) `[SIBLING_BRANCHES]` law and held to the same `[THE_BAR]`, `[THE_METHOD]`, `[TWO_MODES]`, `[FENCE_LAW]`, and `[IDEATION_LAW]`. This file states why the branch is built this way, the method that produces it, and the staged work that remains. The branch is an agnostic, universal, professional Effect/TS library in its own right — designed on its own merits FIRST, then aligned hand-in-glove to serve every C# modality through the wire contracts and the infrastructure boundary, never by importing C# interiors. Folder-specific entry points live in each charter (`<domain>/.planning/README.md`); open work lives in `TASKLOG.md`.

[TWO_MODES]: PLANNING and IMPLEMENTATION never interleave. PLANNING does all discovery — catalogue extraction over the `.api` set, decompile and spike resolution (the XxHash128 byte-identity harness is the named highest-risk SPIKE), gap-hunt, app-ideation, inherited-pool binding — and produces a decision-complete plan in which every change is specified to the domain, owner, page, row, and signature it lands on. IMPLEMENTATION is a later run that transcribes the finalized fences and finds nothing new. There is NO `.ts` source under this branch during planning; the unit of work is the design-page code fence.

[FENCE_LAW]: TypeScript is a FRAGILE language. The unit of work is the transcription-complete CODE FENCE — real `Effect`/`Schema`/`Layer`/`Match`/`Stream` declarations with members, bodies where logic exists, and exact API spellings verified against the `.api` catalogues — never prose. Every decision earns an ultra-harsh critique plus a distinct red-team plus a hardening pass. A page enriched only with prose, a card, or a bare ledger row is UNDONE; the done-measure is the diff to the code fence. The named TS defect classes the critique refuses: loose schema spam, a fresh Schema per shape, branded-type/type/const nonsense, barrel re-export files, and thin wrapper layers — every such occurrence collapses into one dense polymorphic owner.

[IDEATION_LAW]: ideation is aggressive and unanchored — each turn imagines the most ambitious universal-library applications the branch must power from first principles (a flagship browser analytics/control SPA, a node durable-execution + multi-tenant Postgres backend, a self-hosted-or-cloud two-mode IaC tier) and measures the gap to that frontier, never re-confirming existing capability. New high-value concepts land in `FEATURES.md`; every actionable shortfall converts into authored fences the same turn.

## [1]-[THE_BAR]

World-class library code a company would adopt if public. No wrappers, no rename adapters, no grab-bag abstractions — every surface adds real higher-order capability built FROM the admitted packages. Dense, optimized, polymorphic: one owner per axis, one entrypoint family per rail, one fault family per boundary, total `Match` dispatch, `Schema`-derived shapes, fold algebras, table-driven logic. Files are as large as the owned concern requires. Full doctrine adherence: `docs/stacks/typescript/` (README + `language.md`, `shapes.md`, `surfaces-and-dispatch.md`, `rails-and-effects.md`, `boundaries.md`, `system-apis.md`) is the GOVERNING world-class doctrine, the idiomatic-TS equivalent of the C# bar, composed as given and never restated.

## [2]-[UNIVERSAL_PRINCIPLES]

- One unified body per domain, not loose feature arms. A new capability is a row, case, or column on an existing axis — the pressure to add a second surface is the signal to deepen the first.
- Modality completeness from the root: the branch fully serves every C# topology it touches — web-fed SPA, remote/companion/sidecar/hub/service tiers — through the wire contracts and the infrastructure boundary, never one face built and the rest stubbed.
- Parameterization as a universal need: anything that can vary is a `Schema.Literal` row value — a codec, a provider, an export format, a runner protocol, a deploy mode, a renderer backend — never a hardcode and never a parallel name.
- External-lib maximization, truthfully: every admitted package is mined to its full capability and integrated through the owning axis; the Effect-owned abstraction owns the surface (the Effect `KeyValueStore` owns offline persistence with idb-keyval as backing only; `@effect/platform` `Worker.makePool` owns the decode pool; the native `PgClient` + `Model.Class` own SQL with drizzle/kysely never parallel). An admitted capability no page exploits is a named gap.
- Plug-in-play composition: domains are complete in isolation and integrate only through the eleven wire contracts and the documented cross-domain seams (`region-map/seam-splits.md`). Coupling beyond those is a defect; so is a seam re-taught instead of consumed.

## [3]-[THE_METHOD]

1. INFRA TRUTH FIRST — `pnpm-workspace.yaml` catalog, the ONE package `package.json` with `catalog:` refs + subpath `exports` + the folder-scoped `no-restricted-imports` strata, `tsconfig.base.json`, and `vitest.workspace.ts` are made honest before authoring; admissions execute with `pnpm install` resolution proof under `catalogMode: strict`; every admitted package carries a decompile-grounded `.api` catalogue page with zero member gaps, re-verified against installed `node_modules` `.d.ts` (phantom APIs are the named defect class).
2. RESEARCH RESERVOIR — catalogue extraction, adversarial verifiers, gap-hunt lenses, and decompile probes run BEFORE authoring; conflicts resolve by the suite standard's binding precedence.
3. IDEATION BEFORE DESIGN — exhaustive concept enumeration across every modality drives the pages; survivors ride an axis row; concert concepts land in `FEATURES.md`.
4. AUTHOR-THEN-REPAIR — page authors transcribe the blueprint under the suite standard; consolidated review-and-repair waves find and fix in one pass; the orchestrator is the only ledger writer.
5. CROSS-DOMAIN COHESION — ledger consolidation (owner symbols, seam splits), then a final deep pass per domain: under-utilization hunt, heavy-code critique on every fence, doc integrity, zero-guessing validation.
6. EVERYTHING DURABLE — working material persists gitignored; law lands in charters, pages, the ledger, atlases; versions live only in the workspace catalog; the charter ADMISSIONS_RECORD owns the package→page→catalogue→status map, never the pin. `architecture-tree.md` draws the planned source tree; the `.planning/` corpus stays flat.

## [4]-[STAGED_WORK_BEFORE_IMPLEMENTATION]

1. PER-DOMAIN EXTREME-DETAIL DEEPENING — each domain in isolation: re-read every `.api` catalogue against every page, push capabilities further, close TASKLOG rows, re-run the heavy-code critique until any world-class app is buildable from the pages alone.
2. COMPREHENSIVE APP IDEATION — a dedicated ideation campaign across every modality (browser SPA, node backend, two-mode IaC, hub-peer sync) producing new concept rows integrated root-up into pages and `FEATURES.md`.
3. SPIKE RESOLUTION — the XxHash128 byte-identity harness (tier-2, throwaway, against `xxhash-wasm` and C# `System.IO.Hashing.XxHash128` seed=0 with fixed endianness and two-64-bit-half byte order) flips `ArtifactFrameRail` SPIKE→FINALIZED before the content-addressed cache is trusted across C#, the TS browser worker pool, and the future Python IFC->GLB companion; the residual SPIKE set at exit is the cross-branch precondition-DAG entries only.
4. FINAL ARCHITECTURE PASS — the per-domain file plan finalized against each charter BUILD_ORDER, logic-flow walkthrough of every entrypoint rail, cross-domain wire choreography rehearsed on paper, the proof-gate matrix confirmed green-able; the source tree in `architecture-tree.md` re-derived from BUILD_ORDER. Exit: implementation begins with transcription only.

## [5]-[THE_LOOP]

The campaign is a repeatable cycle; each session is one turn and the corpus deepens every turn. Consume the prior distillation as the binding starting work-list (the refreshed concept pool, the per-domain TASKLOG, the `FEATURES.md` rows); integrate the entire inherited pool in full before ideating anything new. Deepen and gap-hunt per domain in one full-depth read; resolve every open unknown in-session through decompile and throwaway-harness probes; cohere ground-up within each domain and cross-domain without coupling (per-domain upgrade, then alignment/refinement reconciling owners/seams/charter state/tree against the ledger, then cross-cutting authoring the suite-level surface). Distill forward: verify 100% of the turn's work is a transcription-complete fence driven to FINALIZED, then run the closeout deep-read plus app-driven ideation emitting the next turn's decision-complete pool. The forward-emitted ideation pool is the only thing a session defers; every actionable shortfall lands on an owner now. `campaign-method.md` records the loop so every turn launches from the same protocol.
