# Foldable Kernel Derivation

[TWO_PRIMITIVE_BASIS]:
- The entire ~50-method aggregation surface of the universal folding trait is generated from exactly two abstract methods: `FoldWhile<A, S>(Func<A, Func<S, S>>, Func<(S State, A Value), bool> predicate, S, K<T, A>)` and its right-associative dual `FoldBackWhile`. Every other member — `Fold`, `FoldBack`, `FoldM`, `FoldMap`, `Count`, `IsEmpty`, `Exists`, `ForAll`, `Find`, `Head`, `Last`, `Sum`, `Product`, `Min`, `Max`, `Average`, `At`, `Partition`, `Iter`, `ToSeq`/`ToLst`/`ToArr` — is a `static virtual` default body expressed as one call into those two. Implementing a new foldable structure costs two method bodies; the structure inherits the full numeric, search, decomposition, and conversion vocabulary with no further code.
- `Fold` is `FoldWhile(f, _ => true, state, ta)` — the unconditional predicate degenerates the early-exit fold into a total fold. `FoldUntil` is `FoldWhile(f, not(predicate), ...)` and `FoldBackUntil` is `FoldBackWhile(f, not(predicate), ...)`; the until/while polarity is a single `Prelude.not` wrap, never a second traversal kernel. The collapse signal is structural: any pair of combinators differing only by predicate negation is one combinator plus a negation, not two.
- The predicate receives the `(State, Value)` pair, so early-exit can inspect accumulated state, not just the next element. This is strictly more expressive than a `takeWhile`-style element-only guard: a running-sum fold can halt when the accumulator crosses a budget, deciding termination from the state the fold itself produced.

[SHORT_CIRCUIT_AS_STATE_PREDICATE]:
- Search and existence combinators do not break a loop; they encode the stop condition as a predicate over a sentinel-carrying state. `Find` folds with state `Option<A>` and predicate `s => s.State.IsNone` — once the accumulator becomes `Some`, the predicate fails and the fold returns. `Head` is the same shape with the fold function ignoring state and emitting `Some(x)` for the first element. `IsEmpty` folds `bool` with predicate `s => s.State`, initial `true`, and a fold function that returns `false`: the first element flips state to `false`, the predicate then fails, and the fold returns after touching one element. `Exists` folds `bool || predicate(c)` with predicate `s => s.State` over `FoldUntil`; `ForAll` is its `&&` mirror over `FoldWhile`.
- The early-exit is checked *before* the fold function runs: the concrete kernel tests `predicate((state, item))` and returns `state` on failure, applying `f(item)(state)` only on pass. The unprocessed element is never folded, so the discriminant of "stop" is the prior state, and the halting element is observed by the predicate but not consumed by the accumulator — the basis for O(1)-on-match search over an arbitrary structure with no concrete `Find` method.
- Positional access reuses the search machinery rather than indexing: `At` threads `(int Index, Option<A> Result)`, halts on the same `Result.IsNone` sentinel as `Find`, and emits `Some(a)` when the running counter hits the target. Forward and from-end access are the *same* state machine — the only branch is which primitive carries it — so a default foldable's indexed access is an O(n) early-exit fold, and constant-time positional access exists only where a concrete type overrides `At` against a real layout. The lesson for callers: `At` on an unknown foldable is not random access, and treating it as such on a large structure is a hidden linear scan per lookup.

[MONADIC_FOLD_VIA_CONTINUATION]:
- The monadic fold family threads an effect through the accumulation without a bespoke kernel by folding into the *continuation* `Func<S, K<M, S>>` and applying `initialState` at the end. `FoldM` is `Fold<A, Func<S, K<M, S>>>(acc, Monad.pure<M, S>, ta)(initialState)` where each step builds `bind => state => Monad.bind(f(value)(state), bind)`. The pure fold composes a chain of binds left-to-right; the seed continuation is `Monad.pure`; invoking the resulting continuation on `initialState` runs the whole monadic chain. The monad is never special-cased in the kernel — it rides the existing pure fold as a higher-order accumulator.

```csharp
static K<M, S> FoldM<A, M, S>(Func<A, Func<S, K<M, S>>> f, S seed, K<T, A> ta) where M : Monad<M> =>
    Fold<A, Func<S, K<M, S>>>(
        a => bind => s => Monad.bind(f(a)(s), bind),
        Monad.pure<M, S>, ta)(seed);
```

- `FoldWhileM`/`FoldUntilM`/`FoldBackWhileM`/`FoldBackUntilM` apply the same continuation-accumulator trick over the predicate-gated primitive, so monadic early-exit folds inherit both the effect threading and the state-predicate halt with no new traversal. Their guard is element-only (`Func<A, bool>`) because the state is now an effectful continuation, not an inspectable value — the one place the `(State, Value)` predicate degrades to a `Value` predicate, and the reason a state-dependent monadic halt must be expressed inside the bound effect rather than the predicate.
- Traversal is itself a fold: the concrete `Traverse` and `TraverseM` over a sequence are both `Foldable.fold(add, F.Pure(Empty), ta).Map(Kind)`, differing only in `add`. The applicative variant lifts `(bs, b) => bs.Add(b)` through `Applicative.lift`; the monadic variant binds `state.Bind(bs => f(value).Bind(b => F.Pure(bs.Add(b))))`. Applicative traversal accumulates because `lift` always evaluates both operands; monadic traversal short-circuits because `Bind` on a failed left never invokes the right. The accumulate-vs-abort distinction is the applicative-vs-monad sequencing law, not two separate algorithms.

[DERIVED_NUMERICS_SINGLE_PASS]:
- `Sum`, `Product`, `Min`, `Max`, and `Average` are folds against generic-math constraints, so one declaration covers every numeric carrier. `Sum` requires `IAdditionOperators<A,A,A>, IAdditiveIdentity<A,A>` and folds `x + y` from `A.AdditiveIdentity`; `Product` folds `x * y` from `A.MultiplicativeIdentity`. No element type is enumerated; the constraint is the polymorphism.
- `Average` carries its denominator inside the accumulator: the fold state is `(Count, Total)` advanced as `(s.Count + A.One, s.Total + x)` from `(AdditiveIdentity, AdditiveIdentity)`, so the count is itself a value of the numeric carrier `A`, divided once at the end — the entire mean is one `Fold` against `INumber<A>` with no auxiliary length read. The projecting overload `Average<A, B>(Func<A, B>, ...)` lifts each element into a distinct summable carrier `B : INumber<B>` inside that same fold, so a `double` mean over a domain value never materializes a projected collection and never calls `Count` (which would force a second traversal on a lazy backing).
- The `Ord`-parameterized `Min<OrdA, A>`/`Max<OrdA, A>` and the `IComparable`-constrained `Min<A>`/`Max<A>` are distinct only in comparator source — `OrdA.Compare(x, v)` versus `x.CompareTo(v)` — and the explicit-`Ord` form carries a null guard the `IComparable` form omits, because a custom total order can be defined over a reference type whose default comparison would NRE. The seeded `Min/Max(ta, initial)` forms collapse the `Option<A>` empty case away by folding into a plain `A` from a caller-supplied floor, trading the absence rail for a known identity.

[OVERRIDE_LATTICE_AND_DIVERGENCE]:
- Default `Count` is `FoldBack(c => _ => c + 1, 0, ta)` — left-associative, so it forces a complete traversal and *diverges on an infinite structure*. Structures that carry a length override `Count` to read it directly; a sequence override is `ta.As().Count`. The doctrine consequence: `Count` on a structure of unknown concrete type is a full fold with side effects on lazy backings, and the only safe constant-time count is one a concrete type promises by overriding.
- The correct override set is exactly the structurally-faster operations, never the derivations. A finite indexed sequence overrides `Count`, `IsEmpty`, `Head`, `Last`, and `At` (each O(1) against the backing layout) plus the two abstract primitives, and inherits `Find`, `Sum`, `Partition`, `FoldMap`, and the rest as default folds. Overriding a derived method that already reduces to the primitives adds surface without changing behavior and is the rejected form; the override budget is one method per genuine asymptotic win.
- `FoldBackWhile` on an enumerable backing is implemented by reversing then forward-iterating (`ta.As().Rev()` then the same predicate-gated loop). Right-associative folds over such a structure therefore pay a reversal materialization; only structures with native O(1) reverse access (a backing array walked from the end, an offset-sliced strict sequence) avoid it. `FoldBack` being "lazy in the accumulator" is a Haskell-inherited contract that the strict CLR backing cannot honor without the reversal — the documented laziness is nominal, the cost is real.

[DUAL_ENTRY_CURRYING]:
- Every static entry point ships both the curried fold function `Func<A, Func<S, S>>` and the uncurried `Func<S, A, S>` over the same body; the uncurried overload adapts to the curried kernel at the call boundary. The curried form composes cleanly under point-free pipelines and partial application of the per-element step; the uncurried form reads as a conventional reducer. Choosing the uncurried overload at a call site that then needs to partially apply the element step reintroduces a lambda the curried form would have avoided — the currying is a real call-site decision, not cosmetic.
- `Contains<A>` resolves equality through `EqDefault<A>`, which performs reflection-based trait resolution at static-type initialization with a boxing fallback to `EqualityComparer<A>.Default`; the `Contains<EqA, A>` overload routes through an explicit `EqA : Eq<A>` static interface method with no reflection and no boxing. For a custom key type folded for membership in a hot path, the explicit-`Eq` overload is the correct surface, and `Contains` is `Exists((x) => EqA.Equals(value, x))` — membership is a search fold, carrying the same first-match early-exit as `Find`.
