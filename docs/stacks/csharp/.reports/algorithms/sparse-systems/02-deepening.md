# Sparse Systems: Deepening

[ITERATOR_DISPATCH_MECHANICS]:
- `Iterator<T>.DetermineStatus` short-circuits: it iterates `_stopCriteria` in insertion order and returns the moment any criterion returns a non-`Continue` status, writing that status to `_status` and discarding all later criteria. A `Converged` result from `ResidualStopCriterion` placed before `IterationCountStopCriterion` will never be shadowed by the count cap — but the reverse placement means the count cap fires first and `Converged` is never seen if the system happens to converge exactly on the limit iteration. The default stack returned by `MatrixBuilder<double>.IterativeSolverStopCriteria(maxIterations)` is `[FailureStopCriterion, DivergenceStopCriterion, IterationCountStopCriterion, ResidualStopCriterion(1e-12)]` — `ResidualStopCriterion` is last, meaning the count cap can suppress a valid convergence signal on the final iteration.
- `Iterator<T>()` (no-arg constructor) pulls the default stack from `Matrix<T>.Build.IterativeSolverStopCriteria()`, which hardcodes `maxIterations = 1000`. Passing `Iterator<T>` as null to `TrySolveIterative` causes a fresh default-constructed iterator with this 1000-cap to be substituted internally. The safe path always constructs `Iterator<T>` explicitly with a custom stack so the cap is size-derived.
- `DivergenceStopCriterion<T>` enforces `MinimumNumberOfIterations >= 3` at the setter (throws `ArgumentOutOfRangeException` for values below 3). Its `IsDiverging` test requires every consecutive pair in the `_residualHistory` window to show a relative increase exceeding `MaximumRelativeIncrease` — the check is an AND over all consecutive pairs, not a mean or majority. Default `MaximumRelativeIncrease = 0.08` and default `minimumIterations = 10` mean 10 consecutive iterations each growing by >8% triggers `Diverged`. Non-monotone residuals with occasional dips cannot trigger this criterion even if the overall trend is diverging.
- `ResidualStopCriterion<T>.DetermineStatus` measures `‖residualVector‖_∞ ≤ Maximum * ‖sourceVector‖_∞`. The threshold is therefore relative to the right-hand-side infinity norm, not absolute and not an L2 norm. When `Maximum = 1e-12` and `‖b‖_∞ = 1e8`, the absolute convergence threshold is `1e-4` — orders of magnitude larger than a naive reading implies. For ill-scaled problems this is a loose criterion that can signal convergence prematurely. The calling code's independent recomputation must use the same norm for a meaningful comparison.
- `DelegateStopCriterion<T>` accepts `Func<int, Vector<T>, Vector<T>, Vector<T>, IterationStatus>` — signature is `(iterationNumber, solutionVector, sourceVector, residualVector) -> IterationStatus`. It stores the delegate reference without invoking it at construction; `Clone()` shares the same delegate reference, so cloned iterators share mutable closure state if the delegate captures mutable fields.

[PRECONDITIONER_FAILURE_SURFACES]:
- `MILU0Preconditioner<double>.Initialize` explicitly requires `matrix.Storage` to be `SparseCompressedRowMatrixStorage<double>`; it throws `ArgumentException("Matrix must be in sparse storage format")` for any other storage, including the `DiagonalMatrixStorage` or a dense matrix. `ILU0Preconditioner<double>.Initialize` internally calls `SparseMatrix.OfMatrix(matrix)`, which performs an O(n²) dense-to-sparse conversion for a non-sparse input — valid but silent and expensive.
- `MILU0Preconditioner` exposes `IsInitialized` (public bool property, initially false). After `Initialize` throws, `IsInitialized` remains false; `Approximate` called on an uninitialized instance will NPE or produce garbage depending on whether `_alu` was partially assigned before the throw. Checking `IsInitialized` before `Approximate` is the only safe path after a potentially-throwing initialization sequence.
- `MILU0Preconditioner` and all other preconditioners throw `NumericalBreakdownException` (subtype of `NonConvergenceException`) when a zero pivot is encountered during their ILU factorization. `TrySolveIterative` does not catch this exception — it propagates to the caller. The contract of `TrySolveIterative` returning `IterationStatus` without throwing holds only when preconditioner initialization does not break numerically. The pattern is: initialize preconditioner outside `TrySolveIterative`, catch `NumericalBreakdownException` at that boundary, and fall back to `DiagonalPreconditioner` or direct solve before entering the iterative solve. Calling `Initialize` inside the solver (which all `IIterativeSolver<T>` implementations do internally) means the exception surfaces from within the `Solve` call, not from `TrySolveIterative` itself.
- `ILUTPPreconditioner` carries `PivotTolerance` defaulting to 0.0, meaning no pivoting. The pivoting condition is `row(i,j) > row(i,i) / PivotTolerance` — division by zero when `PivotTolerance = 0.0` means the condition is never triggered. Setting `PivotTolerance` to a small positive value (e.g., 0.1) enables partial pivoting; the permutation is stored in `_pivots` which the `Approximate` back-substitution applies. Forgetting to set `PivotTolerance` for non-diagonally-dominant problems leaves the preconditioner with unsound factors that only manifest as slow convergence or `NumericalBreakdownException`.

[COMPOSITE_SOLVER_SEMANTICS]:
- `CompositeSolver<T>` sequences through its solver list until one converges. Its internal status routing is: `Converged` → copy result and return immediately; `StoppedWithoutConvergence` → copy partial result and continue to the next solver; any other status (including `Diverged`, `Failure`, `Cancelled`) → reset input and try next solver. The partial result from a `StoppedWithoutConvergence` solver is written to the output before continuing — if the subsequent solver converges from that warm start, the result is valid; if all solvers exhaust without `Converged`, the last `StoppedWithoutConvergence` iterate is left in the output with no indication of inadequacy. The caller must recompute the true residual after `CompositeSolver.Solve` regardless of the `Iterator.Status`.
- `CompositeSolver` resets the iterator before each solver via `iterator.Reset()`, which also resets all stop criteria. The progress of each solver is therefore fully independent; there is no carry-over of convergence history between consecutive solvers.

[CSC_CLEANUP_VS_OFINDEXED]:
- The CSparse `Converter.ToCompressedColumnStorage` with `cleanup: true` calls `Cleanup()` internally, which sums duplicates via a hash-map scan (`int[] array` indexed by row, length `rows`) and produces column-sorted, duplicate-free, exact-zero-retaining output. `Cleanup()` does NOT remove exact zeros — it only sums duplicates. The call to `DropZeros(tolerance)` must follow `Cleanup()` explicitly to remove cancellation residuals from assembly. `CompressedColumnStorage<T>.OfIndexed` (the static factory shorthand) calls `ToCompressedColumnStorage(storage, cleanup: true, inplace: false)` internally; the same zero-retention applies.
- `Helper.SortIndices(csc)` is called by `Converter.ToCompressedColumnStorage` as a separate step after column-scatter, before `Cleanup`. The sort uses a threshold of 16 elements per column: columns with ≤16 entries use insertion sort, columns with >16 use `Array.Sort`. This means structural validation with `Helper.ValidateStorage(csc, strict: true)` may fail immediately after `ToCompressedColumnStorage` with `cleanup: false` if some columns have >16 entries that arrived out-of-order and `SortIndices` was not called — `cleanup: false` skips both `SortIndices` and `Cleanup`.

[SYMBOLIC_FACTORIZATION_REUSE]:
- `SymbolicFactorization` is a plain class with public fields `pinv`, `q`, `parent`, `cp`, `leftmost`, `m2`, `lnz`, `unz`. All CSparse direct factorizations store a private `SymbolicFactorization S` and a private factor (`L` for Cholesky/LDL, `L+U` for LU). There is no public API to extract `S` from a completed factorization, so symbolic analysis cannot be reused across re-factorizations through the public surface.
- The two-argument `Create(A, int[] p)` overloads on `SparseCholesky`, `SparseLDL`, and `SparseLU` accept a pre-computed permutation vector directly — skipping the AMD call but still running full symbolic analysis internally. The AMD permutation is not the bottleneck; symbolic analysis (`EliminationTree` + `ColumnCounts` + `CumulativeSum`) is, and it runs every time `Create` is called regardless of overload. For repeated factorizations with an invariant sparsity pattern, the savings from caching `AMD.Generate` results are real but bounded; the symbolic phase cost is unavoidable without a lower-level API that accepts a pre-computed `SymbolicFactorization`.
- `SparseCholesky.SymbolicAnalysis` allocates `columnCount + 1` elements for `cp`, calls `PermuteSym` (a full upper-triangle copy of the permuted A with no values), then calls `EliminationTree` and `ColumnCounts` and `CumulativeSum`. This sequence allocates O(n) integer arrays. For large n with invariant structure, the symbolic cost dominates numeric refactorization; a pattern-keyed symbolic cache holding the `SymbolicFactorization` (all its fields are value arrays that serialize trivially) would amortize this. The cache cannot be extracted through the public API — it requires a custom subclass or wrapping the symbolic phase calls directly via `GraphHelper.EliminationTree` and `GraphHelper.ColumnCounts`.

[LDL_ZERO_DIAGONAL_CONTRACT]:
- `SparseLDL.Factorize` sets `D[i]` to the computed diagonal value and then checks `if (D[i] == 0.0) throw new Exception("Diagonal element is zero.")`. This is a plain `System.Exception`, not a typed domain failure and not `NumericalBreakdownException`. The throw contract is absent from `ISparseFactorization<T>`, which means a caller catching only `NumericalBreakdownException` will not intercept this failure — it propagates as an untyped exception. The receipt boundary for LDL factorization must catch `Exception` broadly or carry a typed pre-factorization check for zero diagonals in the permuted system.
- `SparseLDL.D` (the diagonal factor array) is a private field with no public accessor. The sign distribution of `D` — which reveals the inertia of the matrix (number of positive, zero, and negative eigenvalues) — cannot be read from a completed factorization. The only way to audit inertia from an LDL factorization is to intercept inside the `Factorize` loop, which requires subclassing or re-implementing the factorization.

[COLUMN_ORDERING_GUARD]:
- The `ColumnOrdering` enum is ordered `Natural = 0, MinimumDegreeAtPlusA = 1, MinimumDegreeStS = 2, MinimumDegreeAtA = 3`. `SparseCholesky.Create(A, ColumnOrdering)` and `SparseLDL.Create(A, ColumnOrdering)` both guard with `if (order > ColumnOrdering.MinimumDegreeAtPlusA) throw ArgumentException`. This integer comparison rejects both `MinimumDegreeStS` and `MinimumDegreeAtA` for symmetric factorizations. `SparseLU.Create` accepts all four orderings — the guard is absent. Passing `MinimumDegreeStS` to `SparseCholesky` throws immediately with `Resources.InvalidColumnOrdering`; a caller switching ordering dynamically must handle this at the boundary.
- `AMD.Generate` with `ColumnOrdering.Natural` short-circuits to `Permutation.Create(columnCount)` — the identity permutation — without any AMD computation. This is the correct path when the caller knows the matrix is already optimally ordered (e.g., from a prior AMD call on an earlier time step with the same sparsity pattern). The identity permutation still triggers full symbolic analysis in the subsequent `SymbolicAnalysis` call.

[SPARSE_QR_OVERDETERMINED]:
- `SparseQR.Create(A, order)` branches on `A.RowCount >= A.ColumnCount`. For overdetermined systems (m ≥ n), AMD runs on A directly; for underdetermined systems (m < n), it transposes A first and runs AMD on `A^T`. The solve direction is reversed accordingly: for overdetermined, `Solve` applies Householder reflectors forward then solves upper triangular `R`; for underdetermined, it solves `R^T` then applies reflectors backward. `S.m2` tracks the augmented row dimension used for structural singletons — the actual allocation for Householder vectors is `S.m2`, not `A.RowCount`. For rank-deficient systems, `S.m2 > A.RowCount` due to structural singletons introduced during symbolic analysis.
- `SparseQR` does not expose a `NonZerosCount` that counts only R; `L.NonZerosCount + U.NonZerosCount - n` in `SparseLU` is the fill metric. For QR, the fill evidence must be derived from `S.lnz` (fill in Q, as Householder vectors) and `S.unz` (fill in R) from the `SymbolicFactorization` field — which is again private.

[ISYMMETRIC_COST_AND_COMPLEX_DISTINCTION]:
- `CompressedColumnStorage<double>.IsSymmetric()` iterates over all nonzeros, and for each `(i, j, v)` pair calls `At(rowIndices[j], i)`. `At(row, column)` is a binary search over `ColumnPointers[column]` to `ColumnPointers[column+1]` — O(log(nnz_column)) per lookup. Total cost is O(nnz * log(nnz_max_column)), not O(nnz). For matrices with dense columns (e.g., after Schur complement formation), this is significantly more expensive than a linear pass. The result is not cached; every call pays the full cost.
- The Complex override of `IsSymmetric()` tests `v.Equals(Complex.Conjugate(At(i, row)))` — Hermitian symmetry. The base double override tests `v.Equals(At(row, col))` — real symmetry. These are structurally different predicates. The method name `IsSymmetric` is shared; a caller using the base `CompressedColumnStorage<T>` interface reference loses the conjugation test for complex matrices.

[DULMAGE_MENDELSOHN_STRUCTURAL_ANALYSIS]:
- `DulmageMendelsohn.Generate<T>(matrix, seed)` computes the coarse Dulmage-Mendelsohn decomposition via maximum bipartite matching with `MaximumMatching.Generate`. The `StructuralRank` property returns `rr[3]`, which is the size of the maximum matching — equal to the structural rank of the matrix (the maximum rank over all numerical instances with the same sparsity pattern). For a matrix with `StructuralRank < n`, no column ordering produces a non-singular factorization — the matrix is structurally singular and the gauge-handling policy applies regardless of numeric values.
- `DulmageMendelsohn.Singletons` counts 1×1 diagonal blocks in the block triangular form — the number of variables that can be solved sequentially without entering a coupled subsystem. For structural analysis of near-diagonal systems (e.g., lightly-coupled physical networks), `Singletons` approaching `Blocks` indicates the system is nearly cascade-solvable with O(n) direct solves rather than a full n×n factorization. `CoarseRowDecomposition` and `CoarseColumnDecomposition` (the `rr` and `cc` arrays) partition the matrix into fine-horizontal, structurally-underdetermined, square-full-rank, structurally-overdetermined, and fine-vertical parts via five index intervals.

[GROWTH_AND_AUTOTRIM_IN_CSR_EDIT]:
- `SparseCompressedRowMatrixStorage<T>.At(row, column, value)` for a new structural entry inserts it at the binary-search insertion point in the row, shifting the suffix of `Values` and `ColumnIndices` by one position via `Array.Copy`. All row pointers above `row` are incremented. This is O(nnz) in the worst case. When `AutoTrimStorage = true` (the CSparse default), every `Keep` call also calls `Resize(0)`, which trims the arrays to exactly `NonZerosCount`. For incremental structural edits (pin a node by zeroing a row/column), the sequence `Keep → At → Keep` triggers two O(nnz) array copies. Batching all structural modifications into a single `CoordinateStorage` and re-converting is cheaper when ≥2 structural edits are needed.
- `SparseCompressedRowMatrixStorage<T>.RemoveAtIndexUnchecked` triggers a defensive trim: `if (num > 1024 && num < Values.Length / 2) { Array.Resize(ref Values, num); Array.Resize(ref ColumnIndices, num); }`. The trim fires only when the array is more than half-empty after removal and has at least 1024 entries. This avoids repeated reallocation for small-matrix deletions but creates a step-function memory profile for large matrices under progressive structural editing.

[BICGSTAB_BREAKDOWN_EXCEPTION]:
- `BiCgStab.Solve` throws `NumericalBreakdownException` when `ρ_k ≈ 0` (the dot product `ρ = r̃ · rk` approaches zero). The test is `num.AlmostEqualNumbersBetween(0.0, 1L)` — this uses the ULP-based comparison with 1 ULP tolerance, meaning any `double` within one ULP of zero causes a throw. `TrySolveIterative` does not catch this throw. For indefinite systems or near-singular operators where BiCgStab tends to stall, this exception propagates silently through the `TrySolveIterative` boundary. `GpBiCg` is the correct solver for non-symmetric indefinite systems — it alternates `_numberOfBiCgStabSteps = 1` BiCGStab steps with `_numberOfGpbiCgSteps = 4` GPBiCG steps, reducing the incidence of near-zero `ρ` while retaining BiCGStab's residual-reduction behavior.
- `TFQMR.Solve` contains an internal call to `CalculateTrueResidual` (computing `b - Ax` explicitly) at certain iterations for residual monitoring, but it passes the internal residual vector to `iterator.DetermineStatus`. The `ResidualStopCriterion` therefore tests an internally-maintained residual, not the true `b - Ax`. `TFQMR` does not throw `NumericalBreakdownException` in its main loop — it tolerates near-zero updates differently from `BiCgStab`. For problems where `BiCgStab` frequently breaks down, `TFQMR` is the safer convergence path at comparable per-iteration cost.

[PROGRESS_REPORTING_IN_FACTORIZATION]:
- All four `Create(..., IProgress<double> progress)` overloads in CSparse (Cholesky, LDL, LU, QR) call `progress?.Report((double)i / (double)n)` only at 1% granularity — they accumulate a counter `num` incremented by `n/100.0` and report only when `i >= num`. For small matrices (n < 100), no `Report` call may be made at all. For cancellable factorizations, `IProgress<double>` is the only hook but carries no cancellation signal — it is reporting only. Cancellation during factorization requires running the factorization on a `Task` with `CancellationToken` operating externally; the factorization itself has no cooperative cancel point.

[COORDINATE_STORAGE_INTERNAL_CONTRACT]:
- `CoordinateStorage<T>` exposes `At(i, j, value)` which unconditionally appends: it writes `rowind[nz] = i; colind[nz] = j; values[nz] = value; nz++`, resizing by factor 2 when `nz == nzmax`. There is no deduplication path in `At` — duplicates are permitted and required for assembly. However, `CoordinateStorage<T>` also exposes `Values`, `RowIndices`, and `ColumnIndices` as public arrays and `NonZerosCount` as a public property. Reading these arrays before conversion is valid only if the caller accounts for possible duplicates and does not treat the raw arrays as a normalized representation. The `inplace: true` conversion flag calls `storage.Invalidate()` after building the CSC representation from the raw arrays — `Invalidate` sets `nz = 0` and nulls the internal arrays, leaving any outstanding references to the old arrays as dangling (but GC-alive) references pointing at data now owned by the CSC matrix.

[KEEP_RETURN_VALUE_AND_AUTOTRIM]:
- `CompressedColumnStorage<T>.Keep(Func<int,int,T,bool>)` returns the count of surviving entries after pruning. When `AutoTrimStorage = true`, `Keep` also calls `Resize(0)` which trims `RowIndices` and `Values` to exactly the surviving count. The returned count equals `ColumnPointers[ColumnCount]` before any subsequent structural write. For the pattern of "pin a node by clearing its off-diagonal entries", `Keep((i, j, v) => !(isPinned[i] && j != i) && !(isPinned[j] && i != j))` combined with an `At` pass to set boundary conditions is a complete in-place structural edit without reassembly — provided `AutoTrimStorage` is temporarily disabled during the sequence to avoid the intermediate trim between `Keep` and `At`.
