# Wire-Contracts: Core Findings

[SOURCE_GEN_CONTRACT_COVERAGE]:
- `[JsonSerializable(typeof(T), GenerationMode = JsonSourceGenerationMode.Metadata)]` on a `JsonSerializerContext` subclass generates compile-time `JsonTypeInfo<T>` instances; all `JsonSerializer` overloads accept `JsonTypeInfo<T>` directly, eliminating every reflection call at the serialization boundary. `Metadata` mode generates full contract metadata ŌĆö property shapes, constructor resolver, polymorphism ŌĆö but defers fast-path serialization; `Serialization` mode generates fast-path write code only and omits deserialization support entirely, including `Populate` mode. Only `Default` (both) gives full read-write coverage simultaneously.
- `[JsonSourceGenerationOptions]` on the context mirrors every `JsonSerializerOptions` property that affects code generation. Setting `RespectNullableAnnotations = true` and `RespectRequiredConstructorParameters = true` at this attribute propagates those constraints into generated code, enforcing nullability and required-field law at compile time rather than in ad-hoc post-deserialization guards.
- Types not covered by any `[JsonSerializable]` declaration require a `JsonConverter<T>` registered via `JsonSerializerOptions.Converters` or a custom `IJsonTypeInfoResolver` returned by `JsonTypeInfoResolver.Combine(...)` and added to `JsonSerializerOptions.TypeInfoResolverChain`. The chain is evaluated left-to-right; the first resolver returning non-null wins. This is the insertion point for attributed payload records whose types are assembled at runtime or come from assemblies not under the generator's sight.
- `JsonTypeInfo<T>.SerializeHandler` is an `Action<Utf8JsonWriter, T>?` delegate set by the source generator when `GenerationMode` includes `Serialization` (i.e., `Default` or `Serialization` modes). When non-null, the serializer uses this generated fast-path delegate instead of the reflection-based property-enumeration path. A modifier that inspects `typeInfo.SerializeHandler != null` confirms that a given type has fast-path write coverage; types with `null` `SerializeHandler` will use the reflection fallback path even inside a source-generated context. Auditing `SerializeHandler` coverage across all envelope types at initialization time is the compile-time contract check that the wire lane stays AoT-clean and allocation-free on the write path.

[RESOLVER_CHAIN_AND_MODIFIERS]:
- `JsonTypeInfoResolver.WithAddedModifier(IJsonTypeInfoResolver, Action<JsonTypeInfo>)` wraps a resolver and post-processes every `JsonTypeInfo` it produces. Modifiers are the correct attachment point for cross-cutting wire-contract policy ŌĆö renaming properties for a protocol version, injecting redaction converters on sensitive fields via `JsonPropertyInfo.CustomConverter`, or forcing `IsRequired = true` on envelope fields ŌĆö without coupling that policy to any domain type. Chaining `WithAddedModifier` calls stacks the `Action<JsonTypeInfo>` invocations in declaration order.
- `JsonSerializerOptions.TypeInfoResolverChain` exposes the live resolver list as a mutable `IList<IJsonTypeInfoResolver>`. Inserting a source-generated context at position 0 and a `DefaultJsonTypeInfoResolver` with modifiers at position 1 gives source-gen coverage for known types with modifier-shaped reflection fallback for unknowns, without splitting options instances per call site.
- `JsonTypeInfo.CreateJsonTypeInfo<T>(JsonSerializerOptions)` and `JsonTypeInfo.CreateJsonPropertyInfo(Type, string)` build contract metadata programmatically for types the source generator cannot see. The resulting `JsonTypeInfo` is mutable until `MakeReadOnly()` is called; properties added via `Properties.Add(...)` participate in all subsequent serialization exactly as generated ones do. This is the sanctioned path for attributed payload records covering types from foreign assemblies or dynamic shapes.
- `JsonSerializerOptions` auto-seals on the first serialization call ŌĆö `IsReadOnly` becomes `true` without an explicit `MakeReadOnly()` call. A modifier inserted via `TypeInfoResolverChain.Insert(0, ...)` after first use throws `InvalidOperationException`. The scoping law: the canonical form is a `static readonly` field initialized once and never mutated after construction; per-instance metadata caches are rebuilt on construction, so the default must never be constructed per-request. The bounded exception: a seam that requires a request-scoped modifier ŌĆö such as a per-request redaction predicate derived from a request-context authorization snapshot ŌĆö may construct a bounded-lifetime `JsonSerializerOptions` with `Combine(ReadOnlySpan<...>)` provided the instance is not reused and its lifetime is explicitly bounded by the request. This is an explicit structural exception, not an optimization; the admission criterion is that the modifier payload is unavailable at startup and varies per request. For lazy-sealed initialization of the static form from a concurrent context: `if (!_options.IsReadOnly) { using var _ = _initLock.EnterScope(); if (!_options.IsReadOnly) { /* mutate */ _options.MakeReadOnly(); } }` using `System.Threading.Lock` ŌĆö whose `EnterScope()` returns a `ref struct` `Lock.Scope` that releases on `Dispose` without allocation ŌĆö is the correct double-checked pattern. `Lock.Scope.Dispose` throws `SynchronizationLockException` when the current thread does not hold the lock; calling `scope.Dispose()` manually without `using` after crossing an `await` boundary will hit this path. Because `Lock.Scope` is a `ref struct`, the compiler prevents `await` inside a `using (var _ = lock.EnterScope())` block; `Lock` must not be used across `await` boundaries.
- `Lock.IsHeldByCurrentThread` returns `true` only when the current thread holds the lock. A factory method that must be called inside a lock can assert `Debug.Assert(_lock.IsHeldByCurrentThread)`, turning a logic error into an immediate assertion failure rather than a data-race corruption. `Lock` is a non-recursive mutex ŌĆö recursive entry at wire-initialization seams must be restructured to extract shared logic rather than re-entering.
- `JsonTypeInfo.IsReadOnly` mirrors the options seal at the type-info level. Modifiers must assert `!typeInfo.IsReadOnly` before any property mutation; a modifier that silently skips mutation when `IsReadOnly` is true produces invisible policy omissions rather than a fast failure. All modifiers must run before `MakeReadOnly()` is called on the owning options, not check `IsReadOnly` at mutation time.
- `JsonTypeInfoResolver.Combine(ReadOnlySpan<IJsonTypeInfoResolver>)` accepts a `ReadOnlySpan` argument. When constructing a bounded-lifetime options instance for a request-scoped seam (the explicit exception described above), combining via span avoids a heap allocation for the resolver array.
- `JsonTypeInfo<T>.CreateObject` is a `Func<T>?` delegate invoked by the deserializer to construct instances before property population. Setting it via a modifier to a factory delegate that pools objects from a pre-allocated slab replaces heap allocation per deserialized message. Combined with `JsonObjectCreationHandling.Populate` on the type via `JsonTypeInfo.PreferredPropertyObjectCreationHandling`, properties are written into the recycled instance rather than a freshly constructed one. Pooled instances must be fully re-initialized before returning from the factory; the deserializer writes properties but does not zero fields from a prior use. The modifier must guard `typeInfo.Kind == JsonTypeInfoKind.Object` before assigning; `Kind.None` produces `InvalidOperationException` when the converter for that kind does not support a create-object delegate ŌĆö the runtime error text is 'not supported for the current metadata Kind', not an enumeration-name check. Parameterless-constructor collection converters (`Enumerable` kind) accept `CreateObject` and do not throw.

[SERIALIZATION_LIFECYCLE_HOOKS_AND_CONTRACT_INVARIANTS]:
- `IJsonOnDeserializing` and `IJsonOnDeserialized` are interface-level contracts on any type in the object graph; implementing the interface is sufficient ŌĆö no attribute involvement ŌĆö and the runtime invokes them before and after property assignment respectively. The boundary pattern is to implement them on the attributed wire record, not the domain type: `IJsonOnDeserializing` initializes a scratch field such as a seen-field bitmask, and `IJsonOnDeserialized` asserts that all required fields were populated. The callback either produces a validated value via the factory or throws a `JsonException` captured and translated by the enclosing `liftIO` + `.Catch` rail.
- `JsonTypeInfo.OnDeserializing` and `JsonTypeInfo.OnDeserialized` (callback delegates on the metadata instance, both typed `Action<object>?`) are the equivalent modifier-based attachment points for types the caller does not own. Both fire synchronously on the deserialization thread; they cannot be async, cannot be re-entrant, and must not write to shared state. Assigning them via `WithAddedModifier` keeps the domain type clean while enforcing cross-cutting wire invariants ŌĆö such as schema-version range checks on every envelope type ŌĆö from one policy site.
- `JsonTypeInfo.OnSerializing` fires before property enumeration begins. `JsonPropertyInfo.ShouldSerialize` is a `Func<object, object?, bool>` evaluated per write with the live parent instance and current property value ŌĆö it is the dynamic predication mechanism, and it must be installed via a modifier before `MakeReadOnly()`. The two mechanisms are therefore distinguished by installation time versus evaluation time: `ShouldSerialize` is installed before `MakeReadOnly()` and evaluated at every serialize call with live state; `OnSerializing` fires after the contract is sealed and is the correct hook for pre-serialize assertions or pre-compute side-effects, but it cannot suppress individual properties ŌĆö only `ShouldSerialize` can. A modifier that needs dynamic per-write suppression installs `ShouldSerialize`; one that needs to inspect or mutate instance-level fields before serialization begins uses `OnSerializing`.
- `IJsonOnSerializing.OnSerializing()` and the modifier-delegate `JsonTypeInfo.OnSerializing` (typed `Action<object>?`) both receive the object instance ŌĆö the modifier receives it as a boxed `object`, the interface method receives it as `this`. Instance-level pre-conditions ŌĆö required correlation fields non-null, version discriminant set ŌĆö belong to `IJsonOnSerializing`; type-level cross-cutting policy ŌĆö redaction, protocol framing ŌĆö belongs to the modifier delegate. Neither can surface a typed failure: exceptions thrown from either callback propagate unwrapped through the serializer, not wrapped as `JsonException`. Pre-validate before calling `JsonSerializer.Serialize` and invoke serialization only on already-validated objects.
- `JsonTypeInfo.OnDeserializing`, `OnDeserialized`, `OnSerializing`, and `OnSerialized` (the modifier-assigned delegate properties) all throw `InvalidOperationException` when assigned on a `JsonTypeInfo` whose `Kind` is not `JsonTypeInfoKind.Object`. Modifiers that set these callbacks must guard `typeInfo.Kind == JsonTypeInfoKind.Object` before assignment; a modifier that unconditionally assigns will throw on `JsonTypeInfoKind.Enumerable` or `JsonTypeInfoKind.None` types. Because `MakeReadOnly` is typically called after the modifier chain runs, the exception manifests at the first serialization call rather than at construction time, making the failure site distant from the cause.

[WIRE_CONTRACT_PROPERTY_SHAPE_MECHANICS]:
- `JsonPropertyInfo.ShouldSerialize` is a `Func<object, object?, bool>` where the first argument is the parent object instance and the second is the property value. Returning `false` suppresses the property entirely ŌĆö no key, no null. Role-based or capability-based field redaction at a boundary injects this predicate via a modifier on the source-generated context, holding a closure over an authorization snapshot; the domain record carries no serialization concerns.
- `JsonPropertyInfo.Get` and `JsonPropertyInfo.Set` can be replaced via modifier with arbitrary delegates that project through an additional transformation. Setting `Get` to a delegate that reads an `Option<T>` field and returns `null` when `IsNone` ŌĆö combined with `[JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)]` ŌĆö projects LanguageExt absence onto the wire as missing-key rather than explicit null, without adding a custom converter for the entire type.
- `JsonPropertyInfo.AssociatedParameter` exposes the constructor parameter linked to a property during `[JsonConstructor]`-attributed deserialization. When a modifier adds a new `JsonPropertyInfo` programmatically via `JsonTypeInfo.CreateJsonPropertyInfo`, that property will not have an `AssociatedParameter` and will be ignored during constructor-binding deserialization unless the type uses `JsonObjectCreationHandling.Populate`. Programmatically added properties on constructor-bound records must be on writable init-properties, or the type must use `Populate` mode.
- `JsonObjectCreationHandling.Populate` re-uses the existing instance on a property rather than replacing it, applying only to properties capable of instance-reuse: collections and classes with a parameterless constructor. For attributed payload records using positional constructors, `Populate` has no effect on those properties and the constructor-path binding dominates. This mode is correct for `[JsonExtensionData]`-backed overflow bags where an existing `Dictionary<string, JsonElement>` must accumulate unknown keys rather than be replaced.
- `JsonPropertyInfo.Order` controls the serialization order of properties within a type. For streaming wire protocols where consumers read properties positionally ŌĆö such as length-prefixed framing where the length is always the first field ŌĆö the property order must be enforced at the contract level. A modifier sets `Order` on each `JsonPropertyInfo` to match the protocol's canonical property sequence; the serializer writes properties in ascending `Order` value. This order is protocol-stable by contract and must be treated as a breaking change coordinate equal in gravity to property name changes. The default `Order` is 0; ties within the same order are resolved by the order `Properties` appear in the `JsonTypeInfo.Properties` list, which reflects declaration order in source-generated types.

[UNMAPPED_MEMBERS_AND_CONTRACT_STRICTNESS]:
- `JsonSerializerOptions.UnmappedMemberHandling = JsonUnmappedMemberHandling.Disallow` causes deserialization to throw on any JSON property without a corresponding .NET member. At the type level, `[JsonUnmappedMemberHandling(JsonUnmappedMemberHandling.Disallow)]` overrides the global setting. The correct wire-contract posture for versioned envelopes at an admission boundary is `Disallow` on the inner envelope type ŌĆö detecting schema drift as a failure ŌĆö and `Skip` on outer wrapper or extension bags ŌĆö tolerating forward-compatible noise.
- `JsonSerializerOptions.AllowDuplicateProperties` controls whether duplicate JSON keys throw or silently take last-write-wins. Set it explicitly to `false` on any `JsonSerializerOptions` used at admission to reject malformed input that a lenient parser would silently corrupt.
- `JsonSerializerOptions.AllowOutOfOrderMetadataProperties` relaxes the constraint that `$type`, `$id`, and `$values` metadata properties must precede regular properties. Set it only when the remote protocol requires it; leave it `false` at owned boundaries.
- `JsonIgnoreCondition.WhenReading` ignores a property only during deserialization, not serialization. Use it on write-only computed fields that must appear in outbound payloads but must not be fed back into the domain on inbound deserialization ŌĆö a precise asymmetric contract shape.

[NULLABLE_AND_REQUIRED_CONTRACT_ENFORCEMENT]:
- `RespectNullableAnnotations = true` causes deserialization of a `null` JSON value into a non-nullable reference type property to throw `JsonException`. Enforcement happens before any `[JsonConstructor]`-attributed factory sees the value ŌĆö the admission gate fires at the JSON token level, not at the factory. A wire type with `required string Field` where `null` arrives on the wire will throw before the domain factory is ever invoked, making the `[JsonRequired]` + `RespectNullableAnnotations` combination the outermost null-rejection layer.
- `RespectRequiredConstructorParameters = true` treats non-optional constructor parameters as required: a missing JSON property for a parameter that has no default value causes deserialization to throw. Parameters with a default value are optional ŌĆö a missing property silently binds the default and no exception is thrown. Without this flag even non-optional parameters are silently bound to `default(T)` when the JSON property is absent, which can produce domain objects in states the admission logic was never designed to reject.
- `[JsonNumberHandling(JsonNumberHandling.Strict)]` applied at the property level via attribute or via `JsonPropertyInfo.NumberHandling` set in a modifier is the correct posture for measurement fields: it prevents string-encoded numeric values ŌĆö a common interop ambiguity ŌĆö from silently binding to typed measurement fields. Physical-unit conversion fields must use `Strict` by default; `AllowReadingFromString` is opt-in only for protocols that document string-encoded numerics.

[POLYMORPHIC_WIRE_CONTRACTS]:
- `[JsonPolymorphic(TypeDiscriminatorPropertyName = "$kind", IgnoreUnrecognizedTypeDiscriminators = false)]` with explicit `[JsonDerivedType(typeof(CaseA), "a")]` annotations is the attributed-record form of a closed-hierarchy wire contract. The discriminator property name must be protocol-stable across versions ŌĆö changing it is a wire-breaking change with no migration path in `JsonPolymorphicAttribute`. The domain union hierarchy and the wire hierarchy must stay structurally parallel but decoupled: a `JsonConverter<TDomain>` at the boundary translates between the domain union and its attributed wire projection, so neither carries the other's schema.
- `JsonPolymorphicAttribute.UnknownDerivedTypeHandling = JsonUnknownDerivedTypeHandling.FailSerialization` is the safe default: attempting to serialize an undeclared runtime subtype throws, preventing silent data loss. At the admission boundary, `IgnoreUnrecognizedTypeDiscriminators = false` (the default) causes deserialization of an unknown discriminator to throw. Both behaviors make schema-contract drift visible immediately; set `FallBackToBaseType` or `IgnoreUnrecognizedTypeDiscriminators = true` only when the protocol explicitly specifies forward-compatible fallback semantics.
- `JsonTypeInfo.PolymorphismOptions` is mutable until `MakeReadOnly()` is called on the owning `JsonSerializerOptions`. A modifier that runs before seal can populate `PolymorphismOptions.DerivedTypes` from a probe receipt: the probe result supplies the list of discriminator strings the remote accepts, the modifier adds `JsonDerivedType` entries for each, and the resulting type info has a closed polymorphic contract matched to the peer's actual capability. This is the extension point for capability-gated wire cases that must not appear in the base contract. The modifier must guard `typeInfo.Kind == JsonTypeInfoKind.Object` before assigning `PolymorphismOptions`; the exception and late-failure mechanics are the same as for the lifecycle delegate callbacks.

[WIRE_CODEC_COVERAGE_GAPS_AND_ALTERNATE_LOOKUP]:
- `FrozenDictionary<TKey, TValue>.GetAlternateLookup<TAlternateKey>()` returns an `AlternateLookup<TAlternateKey>` that performs lookups without allocating a `TKey` instance. When `TKey` is `string` and the comparer implements `IAlternateEqualityComparer<ReadOnlySpan<char>, string>`, the lookup accepts a `ReadOnlySpan<char>` key. `Utf8JsonReader` yields UTF-8 byte spans, not char spans; to use a `ReadOnlySpan<char>` alternate key, first transcode via `Utf8JsonReader.CopyString(Span<char>)` to produce a char span, or design the dictionary with `IAlternateEqualityComparer<ReadOnlySpan<byte>, string>` so the lookup key comes directly from the reader's `ValueSpan` without transcoding. A single `static readonly FrozenDictionary<string, Action<...>>` with `GetAlternateLookup<ReadOnlySpan<byte>>()` at startup eliminates both dictionary boxing and string allocation per property read when the byte-keyed comparer path is used.
- `Utf8JsonReader.HasValueSequence` must be checked before using `ValueSpan`; when `true`, the token value spans multiple `ReadOnlyMemory<byte>` segments and `ValueSequence` is the correct source. A custom converter that reads `ValueSpan` directly without checking `HasValueSequence` silently reads partial data on multi-segment pipeline inputs. `CopyString(Span<byte>)` normalizes this: it always returns the complete unescaped UTF-8 value regardless of segmentation at the cost of a stack allocation. `ValueIsEscaped` gates whether the span is already in raw form or requires unescaping.
- `Utf8JsonWriter.WriteRawValue(ReadOnlySpan<byte>, skipInputValidation: false)` validates the input as RFC 8259-compliant JSON before writing. At a pass-through boundary where a wire field carries an opaque JSON sub-document to be re-emitted verbatim, `skipInputValidation: true` is safe only when the input was received from a trusted internal source in the same request context. For externally-sourced payloads, leave validation enabled; the cost is proportional to the sub-document byte length.
- `JsonElement.Clone()` produces an independent copy that survives `JsonDocument` disposal. Every `JsonElement` reference that must cross an `await` boundary, be stored in a receipt, or be held beyond the `using` scope of its owning `JsonDocument` must be cloned first. The `ObjectDisposedException` thrown by all `JsonElement` operations after document disposal manifests as a runtime fault in async pipelines where the `using` closes before the continuation executes; pool-backed `JsonDocument.ParseAsync` with per-call `Dispose` is correct only when all element work completes synchronously within the same async step.
- `JsonElement.GetRawText()` allocates a `string`. At high-throughput seams where the raw JSON sub-document must be compared or hashed but not decoded, `JsonElement.WriteTo(Utf8JsonWriter)` followed by `Utf8JsonWriter.BytesCommitted` is the allocation-minimizing path. `JsonElement.DeepEquals(a, b)` (static, no allocation) performs structural JSON equality ŌĆö key ordering in objects is not significant ŌĆö without producing text. Snapshot envelope verification uses `DeepEquals` for the structural claim and hash comparison for the byte-identity claim ŌĆö separate verdicts answering separate contracts, never a fallback ladder; structural equality is zero evidence of byte identity.
- `System.Runtime.InteropServices.JsonMarshal.GetRawUtf8Value(JsonElement)` returns a `ReadOnlySpan<byte>` view directly into the `JsonDocument`'s backing buffer ŌĆö no allocation, no copy. The span is valid only while the owning `JsonDocument` is live; it becomes dangerously stale the moment the document is disposed. `JsonElement.Clone` and `GetRawUtf8Value` interact asymmetrically: `Clone` allocates a heap copy making the clone safe beyond document disposal, but calling `GetRawUtf8Value` on a cloned element reads the clone's own buffer ŌĆö if the original document was already disposed when the clone was constructed, the clone constructor itself throws. Acquire raw spans, perform all comparisons or hash-inputs within the same synchronous scope, and never store the span across an `await` boundary or beyond the document's `using` scope. `JsonMarshal.GetRawUtf8PropertyName(JsonProperty)` provides the same zero-copy span for property names, enabling allocation-free property-name dispatch in a custom converter without constructing a `string`. The class is in `System.Runtime.InteropServices`, not `System.Text.Json`; the safety invariant (live document) is not enforced by the type system.
- `JsonEncodedText.Encode(ReadOnlySpan<char>, JavaScriptEncoder?)` pre-escapes a property name once at initialization time and stores the result as `JsonEncodedText`, a value type backed by the encoded UTF-8 bytes. `Utf8JsonWriter.WriteString(JsonEncodedText, ...)` and `WritePropertyName(JsonEncodedText)` skip all escaping at write time because the name is already encoded. For high-throughput serialization seams with a fixed schema, a `static readonly JsonEncodedText` per property name eliminates both the per-write string allocation and the per-write escape scan. `JsonEncodedText.EncodedUtf8Bytes` exposes the pre-encoded bytes as a `ReadOnlySpan<byte>` for use in hash or comparison operations. `JsonEncodedText` instances must be created with the same `JavaScriptEncoder` as the `Utf8JsonWriter` they will be written to; mixing `UnsafeRelaxedJsonEscaping` instances with default-encoder writers produces byte-identity divergence for characters above ASCII 127 ŌĆö the decoded semantics of the two outputs are equivalent (both are valid RFC-compliant JSON), but the raw byte sequences differ, which breaks byte-exact contexts such as HMAC signatures, checksums, and canonical-form comparisons. Keep encoder choice uniform across all writers that produce payloads entering such contexts.
- `Utf8JsonReader.ValueTextEquals(ReadOnlySpan<byte>)` compares the current token value to a UTF-8 literal without allocating a string, handling JSON escape sequences in the token. `static readonly byte[]` fields holding UTF-8 property name literals encoded at module initialization are the dispatch-table keys for allocation-free property routing in custom converters.

[DOMAIN_VOCABULARY_WIRE_PROJECTION_MECHANICS]:
- Bounded vocabulary wire fields should use a generated enum-backed converter whose key type matches the wire string. The generated converter reads keys via span-based comparison without heap string allocation on the inbound path; the generated write path emits the exact protocol key string. Admission logic ŌĆö range checks, sentinel rejection ŌĆö belongs in the generated factory that is also used as the converter `Read` implementation, making the factory the single admission point for both runtime construction and wire deserialization. The factory receives the key value already parsed from the wire; it returns a valid domain value or throws, and the surrounding `JsonConverter.Read` translates the throw into a `JsonException` which the enclosing `liftIO` + `.Catch(e => e.IsExceptional, ...)` rail captures as a typed exceptional error at the seam.
- A null wire value for a bounded-vocabulary field that projects to an absence value in the domain requires a thin outer converter wrapping the generated one: `Read` returns `Option.None<T>()` on null and delegates to the inner converter for non-null, isolating the null-projection policy at the boundary without modifying the generated type.
- Generated value-object converters pass through only the key field; no domain behavior leaks into the wire shape. The `JsonSerializerContext` subclass annotated with `[JsonSerializable]` must include the generated converter in its `Converters` list or the chain, not inline it into the wire record, so codec policy stays at the boundary.

[STREAMING_AND_PIPELINE_SEAM]:
- `JsonSerializer.DeserializeAsyncEnumerable<T>(PipeReader, JsonTypeInfo<T>, bool topLevelValues, CancellationToken)` is the sanctioned streaming-deserialization path over `System.IO.Pipelines`. When `topLevelValues = true`, it deserializes a sequence of whitespace-separated top-level JSON values (NDJSON semantics); when `false`, it deserializes elements of a root-level JSON array. The `JsonTypeInfo<T>` overload is AoT-safe; the `JsonSerializerOptions` overload is not. Wire seams that stream event sequences use the `topLevelValues = true` path with a source-generated `JsonTypeInfo<T>` to stay AoT-clean throughout.
- `ChannelReader<T>.ReadAllAsync(CancellationToken)` returns `IAsyncEnumerable<T>`, composing directly with `await foreach` and async LINQ combinators. A `Channel<TWire, TDomain>` shape ŌĆö where the write side accepts deserialized wire messages and the read side emits projected domain events ŌĆö is the canonical seam marker: it decouples the network-read loop from the domain consumer, absorbs backpressure via `BoundedChannelOptions`, and makes the concurrency contract explicit in the type.
- `Channel.CreateBounded<T>(BoundedChannelOptions, Action<T> itemDropped)` with `FullMode = BoundedChannelFullMode.DropOldest` surfaces a drop-notification callback when the channel is at capacity. The callback is the single correct site for emitting a drop-receipt ŌĆö a typed evidence record with the dropped item's identity, `TraceId`, `Timestamp`, and `DropReason` ŌĆö into a diagnostic side-channel, keeping the channel itself stateless and the observability surface separate. `ChannelOptions.AllowSynchronousContinuations` defaults to `false`; do not opt in on cross-seam channels ŌĆö enabling it allows the writer's call to `TryWrite` to synchronously execute a consumer continuation on the writer's thread, re-entering the writer's synchronization context and inverting the execution model.
- `Utf8JsonReader` is a `ref struct` over a `ReadOnlySpan<byte>` or `ReadOnlySequence<byte>`. For partial reads from a `PipeReader`, call `PipeReader.ReadAsync`, take the buffer from `ReadResult.Buffer`, construct a new `Utf8JsonReader(sequenceSlice, isFinalBlock: false, state: _savedState)`, advance as far as possible, capture `reader.CurrentState` into `_savedState`, then call `PipeReader.AdvanceTo(consumed: reader.Position, examined: buffer.End)`. In the multi-segment `ReadOnlySequence<byte>` path, `reader.Position` is a `SequencePosition`, not a byte count; passing `reader.BytesConsumed` to `AdvanceTo` is incorrect because `BytesConsumed` counts from the head of the current sequence slice. `reader.Position` (which resolves within the sequence) is the only correct consumed argument.
- `JsonReaderOptions.AllowMultipleValues = true` enables `Utf8JsonReader` to tolerate zero or more whitespace-separated top-level values in the same buffer; without it, a second top-level value causes `JsonException`. Set this on the `JsonReaderState` before the first partial-read constructor call; it is preserved across state transitions because it is stored in `JsonReaderState.Options`. This is the reader-level option backing the NDJSON streaming path.
- `Utf8JsonReader.TrySkip()` returns `false` when the current segment does not contain enough data to skip the complete value. Check the return value; if `false`, abandon the current reader, advance `PipeReader.AdvanceTo` with `examined: buffer.End` and no consumed advance, and retry after the next `ReadAsync`. Advancing the consumed position on a `false` return drops bytes silently.

[BACKPRESSURE_AND_PIPELINE_CONTRACT]:
- `PipeOptions.UseSynchronizationContext = false` prevents `PipeReader.ReadAsync` continuations from marshaling back to a captured synchronization context. At a server-side or background-pipeline seam, this must always be `false`; defaulting to `true` on synchronization-context-bearing threads produces unnecessary thread switches between every read/process cycle.
- `PipeOptions.PauseWriterThreshold = 0` disables backpressure: `FlushAsync` never blocks regardless of how much data is buffered. The correct posture for boundary-facing seams is a non-zero threshold ŌĆö 32 KB or a protocol-MTU multiple ŌĆö so that the producer is back-pressured when the consumer is slow. Setting it to zero is safe only when the consumer is guaranteed to drain faster than the producer fills, which is never a safe assumption at a wire boundary with an adversarial remote.
- `PipeReader.ReadAtLeastAsync(minimumSize, CancellationToken)` blocks until at least `minimumSize` bytes are available or the pipe is completed. This is the correct primitive for framing protocols with fixed-length headers before variable-length bodies: read the header with `ReadAtLeastAsync(headerSize)`, parse the body-length field, then `ReadAtLeastAsync(bodyLength)`. Without `ReadAtLeastAsync`, a manual loop over `ReadAsync` with segment-position tracking is error-prone when the header straddles a segment boundary. The `examined` parameter in `AdvanceTo(consumed, examined)` must cover all bytes inspected but not fully consumed to avoid re-reading them.

[TEMPORAL_VOCABULARY_AT_BOUNDARY]:
- `TimeProvider.GetUtcNow()` returns `DateTimeOffset` with `.Offset == TimeSpan.Zero`; `GetLocalNow()` returns local time using `TimeProvider.LocalTimeZone`. Timestamps in persisted or transmitted wire envelopes must use `DateTimeOffset` ŌĆö never `DateTime` ŌĆö with the UTC offset made explicit. `DateTime.MinValue` and `DateTimeOffset.MinValue` must not appear in any serialized field; the sentinel projection at the host boundary converts them to `None` before they reach any serialized shape. Absent timestamps are represented by the absence of the property via `[JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)]` on a `DateTimeOffset?` field, never by `MinValue`. `[JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingDefault)]` is a second-line defense for uninitialized value-type fields that bypass the boundary projection.
- `TimeProvider` is not itself serializable; it is a service injected at composition time. The boundary projects `GetUtcNow()` into an attributed record field at the moment of capture. The receipt field name, format (ISO 8601 via `DateTimeOffset`), and JSON property name are fixed at definition time and never vary across codec versions.
- `TimeProvider.GetTimestamp()` returns a high-frequency counter value. `TimeProvider.GetElapsedTime(long startingTimestamp)` and `GetElapsedTime(long startingTimestamp, long endingTimestamp)` convert a timestamp pair to a `TimeSpan`. A typed receipt carrying an elapsed field captures `timeProvider.GetTimestamp()` before the outbound call and stores `timeProvider.GetElapsedTime(start)` at receipt construction. `Stopwatch` is not injectable and `DateTime` subtraction loses sub-millisecond precision on platforms where the system clock ticks coarsely. The receipt's elapsed field type is `TimeSpan`; `JsonMetadataServices.TimeSpanConverter` exists as the built-in round-trip representation, but an ISO 8601 duration format (`P0DT0H0M0.000S`) is preferred at wire boundaries that must be consumed by non-.NET peers ŌĆö register a custom `JsonConverter<TimeSpan>` for ISO 8601 when the protocol requires it.
- `CancellationTokenSource(TimeSpan delay, TimeProvider timeProvider)` creates a token source whose cancellation is driven by the injected clock. Per-call timeout tokens constructed this way remain deterministic under `FakeTimeProvider` in tests; tokens constructed with `CancellationTokenSource(TimeSpan)` alone use the real wall clock and cannot be fast-forwarded. Every per-request `CancellationTokenSource` for call timeout at a wire seam must take the injected `TimeProvider`.
- `Schedule.upto(duration, Func<DateTime>)` and `Schedule.fixedInterval(duration, Func<DateTime>)` both accept a `Func<DateTime>` clock delegate. At wire seams, this delegate must be `() => timeProvider.GetLocalNow().DateTime` (where `timeProvider` is the injected `TimeProvider`), never `DateTime.UtcNow`. Receipts emitted inside a schedule-governed retry loop must independently capture `timeProvider.GetUtcNow()` ŌĆö a `DateTimeOffset` with zero offset ŌĆö as the `Timestamp` field; the `DateTime` argument of `currentTimeFn` loses UTC-offset evidence and must not serve as the timestamp source. The audit trail requirement makes `DateTimeOffset` mandatory on all receipt timestamp fields regardless of the schedule's internal clock type.

[FAILURE_CAPTURE_AT_MARSHAL_BOUNDARY]:
- `IO<A>.Catch(Func<Error, bool>, Func<Error, K<IO, A>>)` is the composable predicate-based error handler. At a wire marshal boundary ŌĆö where network I/O, codec throws, and timeout exceptions arrive as untyped `Exception` ŌĆö the pattern is `liftIO(Func<Task<A>>)` (wrapping `Exception` into an `IsExceptional` error) followed by `.Catch(e => e.IsExceptional, e => IO.Fail<A>(translateToExpected(e)))` to project exceptional errors into typed expected errors before they escape the seam. The result travels the `Fin<A>` rail inward; the raw `Exception` does not.
- `Error.New(string, Exception)` creates an **exceptional** error with an overridden display message ŌĆö `IsExceptional` is `true`, `IsExpected` is `false`. The original exception is preserved for diagnostic extraction via `Error.HasException<T>()` and `Error.Inner` but the public message is sanitized before any error crosses an export boundary. To create an expected error carrying only a message, use `Error.New(string)` (no exception argument). This distinction is load-bearing: `retryWhile` predicates and `.Catch` guards that test `.IsExpected` versus `.IsExceptional` route errors to different handling paths; mixing the two constructors at translation sites produces silent misclassification. `Error.Many(Seq<Error>)` aggregates multiple field-level failures into one composite error at the validation layer, distinct from the single boundary-crossing exceptional error that wraps an I/O failure.

[IO_MONAD_RESOURCE_AND_THREAD_SEAM_MECHANICS]:
- `IO<A>.Post` re-queues the IO computation onto the `SynchronizationContext` captured in `EnvIO` at the time the chain started. The captured context is the one present when `IO<A>.Run(EnvIO)` was called, not the context at the point `.Post` appears in the chain. A wire seam that calls `IO.Post` after a network awaitable marshals back to the caller's synchronization context only if `EnvIO` was created with a context present. When `EnvIO.New()` is called on a thread-pool thread with no synchronization context, `.Post` is a no-op and the continuation executes wherever the awaitable completes.
- `IO<A>.Local` creates an isolated cancellation scope whose `CancellationToken` child-links to the parent environment's token. Cancellation in the local scope signals only the child token; the parent scope continues. `localComputation.Local.Timeout(TimeSpan)` wraps the computation in a timeout that times out independently without cancelling the enclosing session; the exception integrates with the surrounding `liftIO` + `.Catch(IsExceptional, ...)` boundary translation pattern. The local scope's resources are released when the scope exits regardless of cancellation.
- `IO<A>.WithEnv(Func<EnvIO, EnvIO>)` threads a modified `EnvIO` through a sub-computation and releases resources acquired within that sub-computation when it exits. `IO<A>.WithEnvFail(...)` releases only on failure, propagating the acquired resources upward on success. The correct pattern for a wire connection capsule is `WithEnv` (always-release at the seam boundary) combined with `bracketIO` at the inner call site for any sub-resources the connection acquires. Mixing `Bracket` and `WithEnv` on the same resource produces double-release if not structured carefully: only `WithEnv` should own connection teardown, and inner `bracketIO` should own stream and reader lifetimes.
- `IO<A>.Fork(Option<TimeSpan>)` queues the computation on the thread pool and returns an `IO<ForkIO<A>>` immediately. `ForkIO<A>` is a record with two properties: `Cancel` (`IO<Unit>`) that cancels the forked operation, and `Await` (`IO<A>`) that awaits its result. Wire boundary seams must not fork an `IO` that owns an undisposed connection handle acquired by the parent; the fork should receive a value-projected snapshot such as a deserialized message or typed receipt, with any resource acquisition inside the forked computation's own `bracketIO` scope. Any error inside a fork that is not awaited via `ForkIO<A>.Await` is silently dropped ŌĆö an unawaited fork that fails produces no error signal in the parent chain.
- `Prelude.cancelToken` (the `IO<CancellationToken>` field) extracts the `CancellationToken` from the current `EnvIO`. Passing this token to `liftIO`-wrapped network calls integrates IO monad cancellation into BCL `CancellationToken`-accepting APIs without additional plumbing. The token reflects the state at the time `Run` was called, not at the time it is consumed; if `localCancel` creates a child scope after extraction, the extracted token is the parent's, not the child's. Extract `cancelToken` inside the computation, not before it is queued.
- `Atom<T>.SwapIO(Func<T, T>)` returns an `IO<T>` that performs the compare-and-swap atomically and yields the committed value after the swap. This is the correct cell type for a wire-seam state record that multiple forked `IO` computations must update ŌĆö a connection registration table or a per-lane sequence counter ŌĆö because `SwapIO` integrates into the IO effect chain without requiring explicit locking. `Atom<T>.ValueIO` is an `IO<T>` that reads the current value when evaluated; it defers the read into the chain rather than closing over the value at construction time, which is essential for correctness when the atom is updated between chain construction and execution.
- `Atom<T>.SwapMaybeIO(Func<T, Option<T>>)` performs the swap only when the function returns `Some`; a `None` result leaves the atom unchanged and returns the current unmodified value. This is the gate mechanism for a conditional state transition ŌĆö retiring a pending receipt only when its sequence number matches the expected drain position ŌĆö without introducing explicit branching. The function may be called multiple times under contention; it must be pure and free of observable side effects.
- `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` creates a composite token that fires when any constituent fires. At a wire seam with a per-session token and a per-call timeout token, pass them directly to the two-parameter overload `CreateLinkedTokenSource(sessionToken, callTimeoutCts.Token)` ŌĆö `CancellationToken` is a managed struct (holds a reference to `CancellationTokenSource`) so `stackalloc CancellationToken[]` does not compile. The source must be disposed after the call to release the registrations; a leaked `CancellationTokenSource` holds a registration in the parent token's callback list for the session lifetime.
- `CancellationTokenSource.CancelAsync()` communicates cancellation and returns a `Task` that completes after all registered callbacks and continuations have finished. At a wire seam where cancellation triggers cleanup effects registered via `CancellationToken.Register`, `CancelAsync()` ensures the cleanup runs to completion before the seam proceeds; `Cancel()` runs callbacks synchronously on the caller's thread, throws `AggregateException` if any registered callback throws, and may deadlock if a callback acquires a lock held by the caller. For seam-level cancellation that cascades into `IO`-registered cleanup, `CancelAsync()` is always the correct choice.

[RETRY_SEAM_LAW_AND_CONFLICT_EVIDENCE]:
- `retryWhile(schedule, computation, e => e.HasCode(ErrorCodes.TimedOut) || e.IsExceptional)` is the correct root-level retry predicate for transient wire failures: it retries on timeout and exceptional errors while propagating expected application-level errors ŌĆö validation failures, authorization denials ŌĆö immediately. The predicate is the only correct location to discriminate retriable from non-retriable errors; lower-layer operations emit `Fin.Fail(Error.New(conflictMessage))` (expected, `IsExpected = true`) for known conflict modes and rely on the outer seam's `liftIO + .Catch` translation (described in `[FAILURE_CAPTURE_AT_MARSHAL_BOUNDARY]`) to wrap raw I/O exceptions into exceptional errors that the predicate will retry. Domain-level conflict evidence must never use `Error.New(message, exception)` ŌĆö that form produces `IsExceptional = true` and the predicate would incorrectly retry it.
- `Schedule.exponential(initialDelay, factor)` produces a growing-delay schedule. `Schedule.maxDelay(ceiling)`, `Schedule.jitter(factor)`, `Schedule.maxCumulativeDelay(budget)`, and `Schedule.recurs(n)` all return `ScheduleTransformer`, not `Schedule`; a transformer operand on either side of `|` or `&` applies the transformer to the schedule operand, so `recurs(n) | exponential(d)` and `recurs(n) & exponential(d)` both yield the exponential curve capped at `n`, and transformers compose with `+` before applying. Union-versus-intersect semantics arise only between two `Schedule` values: `|` runs to the longer operand ŌĆö unioning a finite schedule onto an infinite curve does not bound it ŌĆö while `&` runs while both continue, taking the maximum delay. A bounded jittered exponential schedule:

```csharp
var transformer = Schedule.maxDelay(5 * seconds) + Schedule.jitter(0.25);
var bounded = transformer.Apply(Schedule.recurs(5) & Schedule.exponential(10 * ms));
```

- `Schedule.maxCumulativeDelay(budget)` is a `ScheduleTransformer` that terminates the schedule when the sum of all emitted delays reaches the bound, regardless of per-step delay. To bound both per-step delay and cumulative budget, compose transformers before applying: `(Schedule.maxDelay(ceiling) + Schedule.maxCumulativeDelay(budget)).Apply(baseSchedule)`.
- `Schedule.decorrelate(factor)` transforms a schedule by de-correlating durations: given a base duration, it adds and removes jitter proportionally so that parallel clients running the same schedule produce non-synchronized retry waves. A linear schedule `(100, 200, 300...)` becomes approximately `(103.2, 97.1, 202.3, 197.3...)` ŌĆö de-correlation approximately doubles the number of steps because it normalizes amplitude around the base. `Schedule.decorrelate()` returns a `ScheduleTransformer` and must be applied via `.Apply(baseSchedule)`, not composed with `|`. `jitter(factor)` only randomizes amplitude; `decorrelate` specifically addresses phase coherence across parallel clients.
- Exactly one retry owner exists per outbound hop, held at the composition root. Lower-layer operations emit `Fin.Fail(Error.New(conflictMessage))` ŌĆö an expected error ŌĆö carrying the conflict reason. The root uses `retryWhile` to decide whether to retry based on `.IsExceptional` (transient infrastructure failure) versus `.IsExpected` (domain conflict that should not be retried). Lower layers translate raw infrastructure exceptions via `liftIO(...).Catch(e => e.IsExceptional, e => IO.Fail<A>(Error.New(sanitizedMessage)))`, producing expected errors for known failure modes and letting unknown exceptional errors propagate to the root predicate. The root predicate asserts no-conflict by propagating all expected errors immediately; it retries only exceptional errors and coded expected errors with known transient codes. This is the only structure that prevents retry loops from stacking at multiple layers.

[TRACING_AND_CORRELATION_AT_WIRE_SEAMS]:
- `ActivitySource.HasListeners()` is a zero-allocation check that gates `Activity` span creation. At a wire seam producing typed receipt records, correlation fields ŌĆö `TraceId`, `SpanId`, `TraceFlags` ŌĆö must be captured only when listeners are active. The pattern: a `static readonly ActivitySource _source = new("boundaryName")` owned by the seam; receipt construction calls `_source.StartActivity(name, ActivityKind.Client)` guarded by `_source.HasListeners()`, captures `Activity.Current?.TraceId.ToHexString()` into the receipt's correlation slot, then disposes the activity. No `Activity` object appears on a public boundary surface.
- `DistributedContextPropagator.Current.Inject(activity, carrier, setter)` and `ExtractTraceIdAndState(carrier, getter, out traceId, out traceState)` are the vendor-neutral context propagation points for inbound and outbound wire calls. `CreateW3CPropagator()` produces the W3C `traceparent`/`tracestate` propagator. At an outbound seam, `Inject` populates header carrier objects before the wire call. At an inbound seam, `ExtractTraceIdAndState` reads the propagated context strings; the extracted `traceId` and `traceState` must be passed as the `parentId` and optional `traceState` arguments to `ActivitySource.StartActivity(name, kind, parentId: traceId)` on a freshly constructed-but-unstarted activity ŌĆö `SetParentId` is valid only before `Start()`, and `Activity.Current` is by definition already started, making `Activity.Current?.SetParentId(...)` a silent no-op. There is no `SetParentId(string, string)` overload; W3C trace state is set separately via `activity.TraceStateString = traceState` before `Start()`. `DistributedContextPropagator.Current` is a static writable property ŌĆö setting it is a process-wide mutation; set it exactly once during application startup before any wire calls are made, because any concurrent serialization in progress when `Current` is replaced will use the old propagator for in-flight calls. Per-request propagator selection requires constructing a local propagator and calling `Inject`/`ExtractTraceIdAndState` directly on it, bypassing the `Current` singleton.
- `PropagatorGetterCallback` receives both a `fieldValue` (single value, non-null when exactly one value exists) and `fieldValues` (collection, non-null when multiple values exist). A wire seam's inbound carrier adapter must handle both output parameters: some transport stacks emit multi-value headers such as repeated `baggage` lines, for which `fieldValue` is null and `fieldValues` carries all values. Ignoring `fieldValues` silently drops baggage entries when the transport repeats headers.
- `Activity.AddException(Exception, in TagList, DateTimeOffset)` records an exception event onto the active span using W3C-conventional span event semantics. The `TagList tags` parameter is `in` and defaults to `default`, making the bare `Activity.Current?.AddException(ex)` form valid ŌĆö the `in` modifier is optional at the call site. The `in TagList` overload avoids allocating a `ReadOnlyCollection<Tag>` for the tag bag when additional diagnostic tags are supplied. At a wire seam that catches an exception inside a `liftIO` lambda, call `Activity.Current?.AddException(ex)` before translating the exception into an `Error` rail value so the span carries the exception event even though the exception is not re-thrown.

[WIRE_CONTRACT_STABILITY_AS_BREAKING_CHANGE_CONTRACT]:
- `JsonTypeInfo<T>` is the cross-layer serialization contract surface. A change to a property's `JsonPropertyInfo.Name`, `IsRequired`, `CustomConverter`, or its omission from `Properties` is a wire-breaking change regardless of whether the domain type changes. Receipt shapes ŌĆö typed records carrying correlation, timestamp, discriminant, and policy fields ŌĆö must treat their `JsonPropertyInfo` configuration as a versioned contract. Adding a new property with `[JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingDefault)]` is forward-compatible; removing a property or changing its JSON name is not.
- Snapshot envelopes encode the schema version as a generated enum value object keyed on `int` or `string` so that the version token participates in the same generated converter path as any other keyed domain value, and unknown version values are rejected at the converter boundary rather than silently propagated. The envelope carries both a `version` field and a `kind` discriminant field whose valid values are versioned per schema version; the admission modifier validates that the `kind` value is in the permitted set for the declared `version` before the domain factory sees it. `[JsonPolymorphic]` on the envelope's payload field rather than on the envelope itself keeps the discriminator scoped to the payload, not the outer framing.
- `JsonTypeInfo.OriginatingResolver` identifies which resolver in the chain produced a given `JsonTypeInfo`. A modifier injecting redaction logic must gate on `OriginatingResolver` to avoid injecting onto types resolved by a foreign resolver in the chain that manages its own redaction: `if (typeInfo.OriginatingResolver == myContext) { inject modifier logic; }`.
- `JsonNode` enforces single-parent ownership: `JsonObject.Insert`, `SetAt`, and `ReplaceWith` all throw `InvalidOperationException` when the supplied node already has a parent. Pass-through envelope patching that takes a node from one position and inserts it at another must call `DeepClone()` on the source node first. `JsonNode.DeepClone` performs a full recursive copy; its cost is proportional to subtree size, so pre-cloning must be gated on whether the node already has a parent via `node.Parent != null`.
- `JsonObject.GetAt(int index)` returns a `KeyValuePair<string, JsonNode?>` by position; `SetAt(int index, string key, JsonNode? value)` replaces the property at that position with a new key-value pair, throwing `ArgumentException` if `propertyName` is already present at a different index; `SetAt(int index, JsonNode? value)` replaces only the value at the given index preserving the existing key ŌĆö use this two-parameter form when only the value changes, as the three-parameter form accidentally renames the key when the intent is value-only replacement, producing a wire-breaking contract mutation; `Insert(int index, string key, JsonNode? value)` inserts at a position; `RemoveAt(int index)` removes by position. `IndexOf(string propertyName)` returns the zero-based index or `-1`. The idempotent named-property replace: `var idx = obj.IndexOf("version"); if (idx >= 0) obj.SetAt(idx, "version", newNode); else obj.Insert(0, "version", newNode)`. For pass-through envelope patching where the codec must forward an unknown envelope with one additional field injected, index-positioned mutation is the only correct tool; a modifier-based approach via `JsonTypeInfoResolver` is preferred for typed admission.
- `JsonNode.ReplaceWith<T>(T value)` replaces the current node in its parent with a new value, where `T` can be any type that `JsonValue.Create<T>` accepts: primitives, `JsonNode` subtypes, or boxed values. The parent's reference is updated atomically in the node graph. This is the mutation primitive for in-place schema migration of a pass-through envelope where exactly one field value must be re-typed or re-keyed before forwarding, without reconstructing the full node tree.
- `JsonNode.GetPath()` returns the JSON Path of the current node within its document tree ŌĆö e.g., `$.envelope.payload[2]`. This is the diagnostic anchor for admission modifiers that log schema-contract violations: `JsonTypeInfo.Properties[i]` does not give runtime document position, but `GetPath()` on a `JsonNode` representation does. `JsonNode`-based parsing is the correct tooling for contract-violation receipts that require path evidence; `JsonDocument`/`JsonElement` has no equivalent and requires manual property-sequence tracking.

[WIRE_FRAMING_CONSTRAINTS]:
- `JsonDocumentOptions.MaxDepth` defaults to 64; `JsonWriterOptions.MaxDepth` defaults to 1000. The asymmetry means a document tree written without depth limit fails to parse if it exceeds 64 levels. At a wire seam that both produces and consumes messages, the `MaxDepth` on options at the admission side must match the `MaxDepth` used on the write side, or output that passes the writer will fail its own admission round-trip. Set an explicit non-zero `MaxDepth` on both `JsonWriterOptions` and `JsonDocumentOptions`/`JsonSerializerOptions`, choosing a value strict enough to bound adversarial nesting but high enough to accommodate the maximum legitimate document depth. The protocol document must specify this value as a contract constraint.
- `JsonWriterOptions.SkipValidation = true` disables all structural JSON validation in `Utf8JsonWriter`: mismatched start/end tokens, duplicate property names, and property names written outside object context are silently emitted. At a pass-through seam that constructs a writer solely to re-emit an already-validated `ReadOnlySequence<byte>` via `WriteRawValue(sequence, skipInputValidation: true)`, setting `SkipValidation = true` eliminates the structural-state check overhead. `SkipValidation` on the writer does not affect `WriteRawValue`'s `skipInputValidation` parameter ŌĆö the writer option governs the state machine; the parameter governs RFC 8259 content validation of the raw bytes.

[RECEIPT_FAMILY_LAW]:
- One polymorphic receipt family per layer is the structural invariant for wire-seam observability. A receipt record carries: a UTC `DateTimeOffset` timestamp, a correlation identifier (`TraceId`/`SpanId` strings), a discriminant (the operation kind as a generated enum value object), a policy snapshot (retry count, timeout budget, schedule identity), elapsed `TimeSpan` via `TimeProvider.GetElapsedTime`, and a source-boundary tag identifying which seam produced it. These fields are invariant across all cases in the family; cases add evidence specific to the operation outcome ŌĆö success payloads, conflict codes, partial results.
- Extension adds a case to the existing receipt family, never a parallel result system. A second receipt type for the same layer-crossing concern is a collapse signal: the two types share the invariant fields and differ only in the discriminant and case-specific evidence, which is the definition of a union case, not a sibling type. Three or more sibling receipt types with overlapping fields is the mandatory collapse trigger.
- Toolkit and provider types ŌĆö transport channel objects, codec options instances, connection handles, provider-specific envelope types ŌĆö must not appear on public boundary surfaces. The only crossing is a read-only snapshot or projection: a receipt record with value-typed fields derived from the provider state at capture time. A method that returns a live provider object or accepts one as a parameter couples the caller to the provider's lifecycle and makes the surface non-polymorphic across providers.

[COMPOSITE_DIAGNOSTIC_BUNDLES]:
- Composite diagnostic bundles aggregate wire-seam evidence across multiple layers into one exportable unit: the outbound receipt, the inbound response receipt, any drop receipts from intermediate channels, and the final `Fin<A>` outcome. Split ownership means each layer populates its own slice of the bundle and the seam root assembles them; no layer reaches into a sibling's slice. The bundle type is a record with named receipt slots, not a generic bag ŌĆö the slot names encode the layer topology and make schema drift visible as a compile error.
- Redaction-before-export is mandatory: any bundle field derived from external input ŌĆö remote error messages, response headers, path values ŌĆö passes through a redaction predicate before the bundle is emitted to the diagnostic side-channel. The redaction predicate is injected at composition time and closes over an authorization snapshot; it is not a property of the individual receipt type. The redacted bundle is structurally identical to the unredacted one ŌĆö same field names, same types ŌĆö with sensitive string fields replaced by a canonical redacted marker, so downstream analysis code requires no special-casing.

[PHYSICAL_UNIT_AND_STORAGE_NAME_BOUNDARIES]:
- Physical-unit measurement fields crossing a wire seam require two orthogonal concerns separated: the numeric value ŌĆö unitless in the wire representation ŌĆö and the unit discriminant, a generated enum value object keyed on the protocol unit string. A `JsonNumberHandling.Strict` numeric field plus an adjacent unit enum field is the minimal protocol-correct shape. The domain receipt carries only the typed measurement after unit conversion; only the boundary converter reads both the raw numeric and the unit raw string together to perform the conversion and emit the typed domain value.
- Wire payloads that use string keys for physical units must never let those strings propagate into internal code as `string` fields. A generated bounded-vocabulary value object with span-based conversion is the correct admission type: the admission factory rejects unrecognized unit strings as expected errors, converting them to `Fin.Fail(Error.New(...))` before they reach the converter output. Domain code refers to the enum case; the protocol string appears only in the key definition.
- Storage naming is internal: public code addresses entity kinds and query shapes exclusively, never physical storage names such as table names, column names, collection names, or index names. A type whose public API surface carries a literal storage name ŌĆö as a property value, a method parameter, or a generic type argument that maps to a physical name ŌĆö leaks the persistence topology into the domain layer. Storage-to-domain name binding lives at the persistence boundary adapter and is never repeated above it; renaming a storage artifact requires a change only in the adapter, not across the domain.
