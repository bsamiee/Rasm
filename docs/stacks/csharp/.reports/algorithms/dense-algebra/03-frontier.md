# Dense Algebra — Frontier

[SVD_W_MUTATION_HAZARD]:
- `Svd<T>.W` is a `Lazy<Matrix<T>>` backed by `ComputeW()`, which allocates a fresh m×n `DenseMatrix` and writes `S[i]` onto the diagonal via `matrix.At(i, i, S[i])`. The returned matrix is a live allocation, not a read-only projection. `PseudoInverse()` exploits this by calling `w.SetDiagonal(s)` after zeroing small singular values — it mutates the lazy-cached W in place. A second call to `PseudoInverse()` on the same `DenseSvd` handle returns a corrupted W because the lazy value was already mutated; the zeroed singular-value slots from the first truncation are never restored. The handle is not reusable for pseudo-inverse computation. A `Rank` call after `PseudoInverse()` is unaffected because `Rank` reads from `S` (the original `DenseVector`), not from `W`, but any downstream code that inspects `svd.W` gets the truncated diagonal.
- `Svd<T>.S` is the same `DenseVector` instance stored on the base class — it is not a copy. `PseudoInverse()` mutates `S` directly: `s[i] = (s[i] < num) ? 0.0 : (1.0 / s[i])`. After a `PseudoInverse()` call, `S`, `L2Norm` (`S[0]`), `ConditionNumber` (`S[0] / S[last]`), and `Rank` (which iterates over `S`) are all permanently corrupted on the retained handle. The handle must be treated as consumed after `PseudoInverse()` is called; re-use for evidence extraction or solve calls produces silently wrong results.

[QR_ISFULRANK_THRESHOLD_INCONSISTENCY]:
- `QR.IsFullRank` (double, abstract base) checks `Math.Abs(FullR.At(i, i)).AlmostEqual(0.0)`, where `AlmostEqual(0.0)` resolves to `AlmostEqualNorm(x, 0.0, |x - 0|, DefaultDoubleAccuracy)` with `DefaultDoubleAccuracy = 10 · 2⁻⁵³ ≈ 1.11e-15` — an absolute threshold at the magnitude of machine epsilon, not a relative threshold scaled to the matrix. For a well-scaled problem with R diagonal entry `~1e-8` (legitimately rank-deficient relative to the problem scale but far above machine epsilon in absolute terms), `IsFullRank` returns `true`. The correct rank gate scales the threshold to the largest diagonal of R: a pivot ratio `|R[i,i]| / |R[0,0]| < sqrt(eps)` is the standard relative rank-deficiency criterion. `IsFullRank` is reliable only for matrices with diagonal entries either exactly zero (exactly rank-deficient) or well above machine epsilon — not for ill-conditioned systems near rank deficiency.
- `QR.Determinant` (double abstract base) also uses `Math.Abs(FullR.At(i, i)).AlmostEqual(0.0)` for its zero-short-circuit, consistent with `IsFullRank`. All diagonal accesses go through the virtual `Matrix<T>.At(i, i)` dispatch rather than direct array indexing, imposing a virtual call per diagonal element in both properties.

[SVD_REGRESSION_VECTORS_DEFAULT]:
- `MultipleRegression.Svd(x, y)` calls `x.Svd().Solve(y)`, which routes to `Matrix<T>.Svd(computeVectors: true)` — the default. This means every SVD regression call allocates the full m×m U and n×n VT factor matrices even though the least-squares solve only needs the singular values and VT (not the full U) for the minimum-norm computation. The allocation of a full m×m U for a 5000×20 design matrix adds 200 MB of working storage for 25 million unreferenced doubles. The managed `SvdSolveFactored` uses all of U (it computes Uᵀb column by column), so the full U is required for correctness under the current provider implementation; there is no thin-SVD solve path. The cost is unavoidable under the admitted package surface for SVD regression on tall matrices.

[CHOLESKY_SOLVE_INPUT_OVERWRITE]:
- `DenseCholesky.Solve(input, result)` copies `input.Values` into `result.Values` via `Buffer.BlockCopy` before calling `CholeskySolveFactored`. The LAPACK POTRS-equivalent solve overwrites the right-hand side buffer in place with the solution. The result buffer, not a separate workspace, is the solve target. When the caller passes the same instance for both `input` and `result` — or when `result` was constructed by wrapping the same `double[]` as `input` (via the aliasing hazard) — the copy via `BlockCopy` is a no-op and the result is correct. But if the caller's `input` is separately held and expected to be unmodified after the call, the copy-then-overwrite pattern preserves it. The key risk: passing a `result` matrix pre-loaded with a prior solution expecting additive update is silently overwritten. The solve is unconditionally destructive to `result.Values` before the provider call.
- `DenseLU.Solve(input, result)` follows the identical `Buffer.BlockCopy(input → result)` then in-place solve pattern. The same overwrite semantics apply. `DenseQR.Solve` does not copy into result before the provider call — instead `QRSolveFactored` receives both Q, FullR, Tau, and the input array; the result buffer is written directly. The asymmetry between Cholesky/LU (copy-then-overwrite) and QR (direct write) means that QR does not preserve the input buffer under any calling pattern, while Cholesky/LU do.

[EVD_DETERMINANT_MAGNITUDE_ONLY]:
- `Evd<double>.Determinant` computes `Π eigenvalues[i].Magnitude` — an absolute-value product of complex eigenvalues. For a symmetric matrix with a negative eigenvalue (indefinite matrix passed with `Symmetricity.Symmetric`), the true determinant is negative but `Evd.Determinant` returns a positive value. Only `LU.Determinant` carries the correct sign. For definiteness-sensitive algorithms that need `sign(det(A))` — computing the orientation of a rotation, detecting a definite-to-indefinite transition — `Evd.Determinant` is incorrect and `LU(matrix).Determinant` is the only admitted surface with the correct sign.
- `Cholesky.Determinant` computes `Π L[i,i]²` using `Factor.At(i, i)` virtual dispatch — one virtual call per diagonal for both `Determinant` and `DeterminantLn`. The diagonal of the lower Cholesky factor is accessible as a contiguous column-major stride: `values[i * n + i]` for column `i`. A `TensorPrimitives.Log<double>(diagonalSpan, logSpan)` followed by `TensorPrimitives.Sum<double>(logSpan) * 2.0` replaces n virtual-dispatch scalar calls with a single SIMD reduction for `DeterminantLn`, matching the numerically stable result without the dispatch overhead. At n = 200 this is ~200 virtual calls vs. a single AVX2-vectorized log+sum pass.

[GRAM_SCHMIDT_EXACT_ZERO_ONLY]:
- `DenseGramSchmidt.Factorize` throws `ArgumentException("Matrix must not be rank deficient.")` only when a column norm during orthogonalization is `== 0.0` exactly — the comparison is `if (num == 0.0)` with no epsilon guard. A near-zero column norm (numerically zero due to floating-point cancellation, but not bit-exactly zero) passes the check and results in division by near-zero, producing `NaN`/`Inf` in Q and R without any exception. This is a silent failure mode: the returned `DenseGramSchmidt` handle has an `R` matrix populated with `NaN`, and `IsFullRank` subsequently returns `true` because `Math.Abs(NaN).AlmostEqual(0.0)` is `false`. The factorization-succeeded-but-result-is-corrupt failure cannot be detected without an explicit all-finite check on the factor buffer after construction.

[LU_INVERSE_ALLOCATION_PATTERN]:
- `DenseLU.Inverse()` clones `Factors` then calls `LUInverseFactored(clone.Values, n, Pivots)`. The managed `LUInverseFactored` allocates a second `double[n²]` identity matrix, calls `LUSolveFactored(n, factored, n, pivots, identity)` to solve n right-hand sides simultaneously, then copies the result back over the input array. Total allocations: one `n²` clone in `Inverse()` plus one `n²` identity in `LUInverseFactored`. For n=50, this is two 20KB allocations on the LOH boundary (LOH threshold ~85KB), below LOH pressure. For n≥100, both arrays cross the LOH boundary together. The correct pattern for matrix inversion in a hot loop is to retain the `DenseLU` handle and call `Solve` on the identity in-place, reusing pre-allocated result and right-hand side buffers across calls.

[MANAGED_GEMM_ROW_COLUMN_EXTRACT_COST]:
- The managed `MatrixMultiplyWithUpdate` kernel pre-extracts all columns of B into `double[][] columnDataB = new double[columnsB][]` before the row-column multiply loop. Each column of B is extracted by `GetColumn(transposeB, i, ...)`, which for the non-transposed case copies `b[i*rowsB .. i*rowsB + rowsB - 1]` — a contiguous slice in column-major layout — into a new `double[rowsB]`. This pre-extraction doubles the memory traffic for B (once to read it during extraction, once during the multiply loop) and allocates `columnsB` separate arrays. For an `m × k` times `k × n` multiply, the column pre-extraction allocates `n` arrays of length `k` on the heap before any multiply occurs. Under MKL, this code path is bypassed entirely — the `MatrixMultiplyWithUpdate` call goes directly to BLAS DGEMM without pre-extraction. The managed GEMM is therefore significantly slower than raw cache-locality would suggest, independent of vectorization.
- The parallelization gate `rowsA + columnsB + columnsA < Control.ParallelizeOrder` uses a linear sum of three dimensions, not their product. For a 10×1000×10 GEMM (many columns, small square factors), the sum is 1020 — well above the default threshold of 64 — so parallel dispatch fires. For a 64×64×64 GEMM, the sum is 192, also above the threshold. The threshold is poorly calibrated for problem shapes where parallel dispatch overhead exceeds the parallelism benefit; at small n, `Control.MaxDegreeOfParallelism = 1` eliminates this overhead.

[SYMMETRIC_FORCE_ADMISSION_PATTERN]:
- The canonical symmetric-force expression `matrix.Add(matrix.Transpose()).Multiply(0.5)` allocates two new matrices (one for the transpose, one for the add result) and one scalar multiply, for three total dense allocations at O(n²) each before the EVD call. For an accumulated covariance matrix at n ≤ 10 (3D covariance, a 3×3 or 6×6), this is negligible. At n = 500 (medium covariance matrices), the three-allocation chain produces 750KB + 750KB + 750KB = ~2.25MB of transient working storage. The in-place alternative — iterate the `Values` array and average symmetric pairs `(a[i*n+j] + a[j*n+i]) / 2` for `j < i`, writing both — requires zero extra allocations and completes in a single pass. `TensorPrimitives` has no direct symmetric-force primitive; the span-level loop with direct index arithmetic is the efficient form, exploiting column-major layout to write the lower and upper triangles in the same pass.

[FACTORIZATION_RESULT_BUFFER_REUSE]:
- All six `Dense*.Create` paths call `matrix.Clone()` as their first operation before passing the clone to the provider. The clone is the internal factor buffer — it is never released to the caller. For repeated factorizations of identically-sized matrices (scattered-data interpolation, MCMC covariance updates), the clone allocation dominates at small n because the managed allocator cannot reuse the previous factor buffer. `Cholesky.Factorize(newMatrix)` on a retained handle reuses the existing internal factor buffer via `Buffer.BlockCopy(newMatrix.Values → factor.Values)` before calling `CholeskyFactor`. This is the only in-place refactorization path; it requires that the new matrix be the same size and type as the one used to create the handle, which is checked by dimension equality. The refactorization path exists only on `Cholesky`; there is no equivalent `Factorize` mutation method on `LU`, `QR`, `Svd`, or `Evd` handles — those require constructing a new handle and a new clone each time.

[ISFINITE_ADMISSION_GATE]:
- The correct all-finite admission gate over `DenseMatrix.Values` is `TensorPrimitives.IsFiniteAll<double>(new ReadOnlySpan<double>(matrix.Values))`, which maps to a vectorized SIMD predicate over the entire flat array in a single call. This replaces a scalar `LINQ All(double.IsFinite)` that would box or use interface dispatch per element. `IsFiniteAll<T>` is available in `System.Numerics.Tensors` 10.0.0 under the constraint `T : INumberBase<T>`, which `double` satisfies. For a 100×100 matrix (10,000 doubles, 80KB), the difference between vectorized and scalar is roughly 20-50× depending on SIMD width, critical for admission gates on matrices updated at high frequency.
- `TensorPrimitives.IsNaNAny<T>` and `IsInfinityAny<T>` are also available as early-exit alternatives when NaN and Inf must be distinguished in evidence receipts. The combined test `!IsNaNAny && !IsInfinityAny` is equivalent to `IsFiniteAll` with the advantage that the failing predicate is identified in the evidence.

[PSEUDOINVERSE_THRESHOLD_VS_SOLVE_INCONSISTENCY]:
- The `PseudoInverse()` threshold is `max(m, n) · σ_max · Precision.DoublePrecision` where `Precision.DoublePrecision = 2⁻⁵³`. The `Svd.Rank` threshold is `σ_max.EpsilonOf() · max(m, n)` where `EpsilonOf(x) = MathNet.Numerics.Precision.EpsilonOf(x)` — computed as the next representable double above `σ_max` minus `σ_max`, i.e., the machine epsilon at the scale of `σ_max`. For `σ_max = 1.0`, `EpsilonOf(1.0) = 2⁻⁵²` — exactly double `Precision.DoublePrecision`. This means `Rank` uses a threshold that is twice as large as `PseudoInverse`'s threshold for the same matrix. A singular value that `PseudoInverse` retains (above `σ_max · ε_machine · max(m, n)`) can still cause `Rank` to report one fewer non-zero singular value (if the singular value is below `σ_max · 2ε_machine · max(m, n)`). The two surfaces are calibrated to different precision constants and will disagree on rank-boundary cases, even though they both read from the same `S` vector on the same handle.
