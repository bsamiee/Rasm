# Parse Option Control Surface

[THE_TRIPLE_IS_STATIC_THE_OPTIONS_ARE_THE_RUNTIME_FOURTH_AXIS]:
- `Type`/`Encoded`/`Context` are the compile-time triple a codec owns; `ParseOptions` is the orthogonal RUNTIME axis — `{ errors?: "first" | "all"; onExcessProperty?: "ignore" | "error" | "preserve"; propertyOrder?: "none" | "original"; exact?: boolean }` — that reshapes the SAME codec's failure family and excess-key handling without touching any of the three static slots, so a struct decoded `errors: "all"` and the same struct decoded `errors: "first"` are one declaration with two failure projections, never two schemas.
- The four fields are a closed policy product, each a literal-keyed knob the compiler walks as data: `errors` selects first-failure short-circuit versus full accumulation, `onExcessProperty` selects strip/reject/retain for keys the field map does not name, `propertyOrder` selects internal-fast versus input-preserving output key order, `exact` selects whether an absent key is `Missing` or treated as present-with-`undefined` — a behavior table, never a body branch, the option value selecting the algebra exactly as a `Schedule` or a `Layer` selects its.
- `defaultParseOption` is the empty record `{}` — every field absent falls to its documented default (`"first"`, `"ignore"`, `"none"`, `false`), so a decode with no options is the strictest-failure, lossiest-output, laxest-presence corner of the product; the policy is read off the option record at parse time, not baked into the AST, the one owner answering to every corner of the lattice by the record it is handed.

[ERRORS_FLIPS_THE_FAILURE_FAMILY_BETWEEN_SHORT_CIRCUIT_AND_FOLD]:
- `errors: "first"` makes the struct/tuple compiler return `Either.left(new Composite(ast, input, e, output))` at the FIRST rejecting field — the fold abandons the remaining fields the instant one issue forms; `errors: "all"` switches the same fold to push each issue as `[stepKey++, e]` into an accumulator, sort it by `sortByIndex`, and return one `Composite` whose `issues` is the `SingleOrNonEmpty<ParseIssue>` of every field that failed — the failure ARITY is the option, the issue tree the same `Composite` node either way.
- The accumulation is index-stamped so the multi-issue report is deterministic regardless of which fields resolve effectfully out of order: every queued issue carries its `stepKey` and `sortByIndex` re-orders the collected `es` into declaration order before the `Composite` is built, so the `"all"` projection's child order mirrors the field-map order rather than the resolution order — the failure tree's branching is isomorphic to the slice of the value graph the fold visited, the index the discipline that keeps it stable under async legs.
- `errors` reaches every composite node uniformly — the same `options?.errors === "all"` read gates the `Refinement` from-leg, the `TupleType` element loop, and the `TypeLiteral` property loop — so accumulation is a recursive property of the whole decode, a nested struct under an `"all"` decode reporting all of ITS field failures inside its `Pointer`, the full-error mode a single switch the entire AST fold honors rather than a per-node flag.

[ONEXCESSPROPERTY_IS_THE_THIRD_LEG_THE_FIELD_MAP_DOES_NOT_NAME]:
- The default `"ignore"` SKIPS the excess-key scan entirely — the compiler never calls `Reflect.ownKeys(input)` and copies only the field-map keys to `output`, so unrecognized wire keys are silently dropped and the scan cost is never paid; `"error"` and `"preserve"` BOTH force the `Reflect.ownKeys` enumeration up front (`inputKeys` materializes once and is reused by `propertyOrder`), the two non-default modes sharing the one scan the default avoids — the perf axis is which mode you choose, not a separate flag.
- `"error"` turns an unexpected key into `new Pointer(key, input, new Unexpected(input[key], "is unexpected, expected: ..."))` — the `Unexpected` issue is the ONLY `ParseIssue` arm that exists solely because of an option, present in the family but reachable only under `onExcessProperty: "error"`, so the issue alphabet the boundary handles is itself a function of the decode policy, the strict-key failure a leg the default decode never produces.
- `"preserve"` copies each unrecognized key straight into `output` (`output[key] = input[key]`) so the decoded value carries fields the static `Type` does not declare — the one mode where the runtime value is WIDER than its compile-time type, the escape hatch for a pass-through envelope, paid for with a `Type` that under-describes the value; the strict default and the strict-key `"error"` keep value and type aligned, `"preserve"` is the deliberate divergence named at the call.
- Excess handling and accumulation compose: under `errors: "all"` an `"error"` excess key is pushed onto the same `es` accumulator and continues the scan, so a payload with two stray keys and one malformed field reports all three issues in one `Composite`; under `errors: "first"` the excess `Unexpected` short-circuits immediately — the two options are read in the same loop, the excess leg obeying the accumulation mode the failure leg does.

[EXACT_IS_THE_PRESENCE_SEMANTICS_OF_AN_ABSENT_KEY]:
- `exact: false` (the decode default) treats a missing non-optional key as present-with-`undefined` and lets the field's own codec reject the `undefined`; `exact: true` makes the absent key itself a `Missing` issue at the property loop (`isExact` short-circuits before the field codec runs), so the SAME field map yields a `Type` issue under one option and a `Missing` issue under the other — the absence diagnostic is the option, the field schema untouched.
- `is` and `asserts` inject `exact: true` as their base options because a type guard must treat an absent key as definitively absent rather than coercibly-`undefined`, while `decodeUnknown`/`validate` default `exact: false` to admit the boundary convention that a missing key and an explicit `undefined` are interchangeable — the guard family and the decode family disagree on presence by default, the disagreement encoded as the option each pre-sets, not as two property loops.
- `exact` is orthogonal to field-level optionality: a `PropertySignature` whose Type token is `"?:"` continues past an absent key under both `exact` values (the optional field legitimately omits), so `exact` governs only the REQUIRED-key absence — the modifier-token optionality decides whether a key MAY be absent, `exact` decides whether a required key's absence is `Missing` versus `Type`, two independent axes the property loop reads in sequence.

[PROPERTYORDER_IS_THE_OUTPUT_KEY_PERMUTATION]:
- `propertyOrder: "none"` (default) emits output keys in the internal fast order the compiler chooses and explicitly does NOT guarantee stability across releases; `propertyOrder: "original"` re-walks the input key order at `computeResult`, seeding the output from `inputKeys` (the same array `onExcessProperty` already materialized when set) then appending any expected key the input omitted — the output key sequence becomes a function of the wire key sequence, the one option that reshapes the SHAPE of the success value rather than the failure family.
- The reordering reuses `inputKeys` when an excess mode populated it and falls back to a fresh `Reflect.ownKeys(input)` otherwise, so `onExcessProperty: "error" | "preserve"` and `propertyOrder: "original"` SHARE the one key enumeration — combining them pays the scan once, where each alone pays it once too, the two output-shaping options collapsing onto a single substrate read rather than two passes.
- `"original"` matters only where a downstream consumer is order-sensitive (a deterministic serialization, a canonical-hash input, a human-readable diff) — it is the deliberate cost paid to make the decoded record's key order load-bearing; the default `"none"` is the correct choice for every consumer that reads by key, and reaching for `"original"` without an order-consuming downstream is the option set that decides nothing.

[THE_ANNOTATION_PINS_A_PER_NODE_DECODE_POLICY_THAT_OVERRIDES_THE_CALLER]:
- `getParseOptionsAnnotation(ast)` reads the `ParseOptionsAnnotationId` slot off any node, and the compiler wraps the node's parser so a present annotation re-runs the parser under `mergeInternalOptions(callerOptions, annotationOptions)` — the annotation is the OVERRIDE argument, last-wins, so a sub-owner that annotates `onExcessProperty: "error"` enforces strict keys for its own sub-tree EVEN when the top-level decode passes `"ignore"`; the node's policy wins over the caller's, the inversion of the obvious precedence.
- This makes excess-key strictness, error accumulation, and presence semantics a COMPOSITIONAL property a sub-shape carries rather than a global the boundary dictates: a permissive envelope decoded `"ignore"` can embed a strict inner payload annotated `"error"`, each node's own slot governing its own descent, the policy declared once on the owner that needs it and every consumer inheriting it without re-passing options at the call.
- The annotation rides the SAME `Annotations.Schema` record as `arbitrary`/`pretty`/`message` — `parseOptions: ParseOptionsAnnotationId` is one slot among them — so attaching a decode policy is a slot change at the owner exactly like attaching a generator override, the secondary-family overrides and the runtime-control policy sharing the one annotation surface, never a parallel configuration call beside the schema.
- The merge is shallow spread (`{ ...caller, ...annotation }`), so an annotation setting only `onExcessProperty` leaves the caller's `errors` and `exact` intact — the per-node policy patches the fields it names and inherits the rest, a struct annotated `{ onExcessProperty: "error" }` still accumulating under a caller's `errors: "all"`, the two policies merging field-wise rather than the annotation replacing the whole record.

```typescript
import { Schema } from 'effect'
const Strict = Schema.Struct({ key: Schema.NonEmptyString }).annotations({
    parseOptions: { onExcessProperty: 'error', exact: true },   // pins strict-key + strict-presence on THIS node
})
const Envelope = Schema.Struct({ meta: Schema.Unknown, body: Strict })   // lax outer, strict inner by annotation
const decode = Schema.decodeUnknownEither(Envelope)
const lax = decode({ meta: 0, body: { key: '<value-a>' }, extra: 1 })    // outer drops `extra` (caller "ignore" default)
const strict = decode({ meta: 0, body: { key: '<value-a>', extra: 1 } }) // Left: inner annotation forces Unexpected on `body.extra`
```

[OPTIONS_BIND_AT_TWO_PHASES_AND_THE_NUMBER_OVERLOAD_IS_INERT]:
- Every decode/encode/validate constructor takes options at TWO phases — `decodeUnknown(schema, options)` binds a policy at schema-bind time, and the returned `(u, overrideOptions) => ...` accepts a per-call override merged via `mergeInternalOptions(boundOptions, overrideOptions)` — so a parser is built once with a default policy and re-tuned per invocation without re-deriving the codec, the two-argument shape a curried policy partial the call site refines.
- The merge is the SAME `mergeInternalOptions` the per-node annotation uses, so the override precedence is uniform across the whole surface: call-time override beats bind-time default beats the empty `defaultParseOption`, and the per-node annotation beats the resolved caller policy on top — one merge rule chained at three levels, the effective policy at any node the field-wise spread of bind, call, and annotation in that order.
- `is(schema, options)` returns a guard whose second argument is typed `overrideOptions?: ParseOptions | number`, and `mergeInternalOptions` returns the base options unchanged when the override `isNumber` — the numeric second argument is the React/recursion-depth escape the guard ignores for policy purposes, so passing a number to a guard tunes nothing about excess/errors/exact, a trap where a position that looks like an options slot is silently inert.

[FILTEREFFECT_AND_THE_SYNC_RUNNER_PRODUCE_THE_FORBIDDEN_ISSUE]:
- The option record carries a hidden runtime bit `isEffectAllowed` set ONLY by the effectful runners (`decodeUnknown`/`validate` returning `Effect`), so a `filterEffect`/`transformOrFail` leg that reaches a service yields an async parse the SYNC runners cannot resolve — `decodeUnknownSync`/`decodeUnknownEither` over a schema with an effectful node return `Either.left(new Forbidden(ast, actual, "cannot be resolved synchronously..."))`, the `Forbidden` issue the failure the option surface produces when the codec's `Context` requirement collides with the runner's synchrony.
- `Forbidden` is the bridge between the static `Context` axis and the runtime option axis: a non-`never` `Context` on the codec means an effectful leg exists, and the runner choice (sync versus `Effect`) decides whether that leg is permitted — so the `Forbidden` issue is reachable purely by mismatching the runner to the codec's requirement, never by malformed data, the only `ParseIssue` arm whose cause is the execution model rather than the value.
- `handleForbidden` attempts a synchronous flush of the effect and converts a defect (a missing layer) into `Forbidden` carrying `Cause.pretty(cause)` — so a sync decode of a codec whose `filterEffect` needs an unprovided service surfaces the missing-dependency as a parse failure, not a runtime throw, the requirement gap rendered as a recoverable issue at the owning node the same way a malformed value is.

[CONCURRENCY_AND_BATCHING_ARE_THE_SAME_PER_NODE_POLICY_FAMILY]:
- `concurrency` and `batching` ride the identical `Annotations.Schema` record (`concurrency: ConcurrencyAnnotationId`, `batching: BatchingAnnotationId`) as `parseOptions`, and the tuple, array, and union compilers each read `getConcurrency(ast)`/`getBatching(ast)` to parameterize the `Effect.all` resolving their children — so the decode of a collection of effectful members is a structured-concurrency policy the OWNER declares per node, the member-parse strategy a slot exactly as the failure policy is, one annotation family spanning the failure axis and the concurrency axis of one decode.
- The union arm pins `getConcurrency(ast) ?? 1` — sequential candidate resolution unless the node annotates otherwise — so a union of `filterEffect`-bearing members tries its arms one service call at a time by default and the annotation widens that to a declared degree, the concurrency of the decode a policy value on the schema rather than an improvised parallel map; this composes with `errors: "all"` so a concurrent member decode still index-stamps each child's issues by `stepKey` into the order-stable `Composite`, the structured-concurrency degree never disturbing the declaration-order failure tree.

[REACH_LIMIT_FENCE]:
- A `parseOptions` annotation earns its slot only by pinning a policy a sub-tree genuinely needs against the caller's — a strict inner payload inside a lax envelope, a presence-exact key set inside a coercible outer; an annotation restating the caller's policy or the documented default is the option that decides nothing, deleted, and a global decode policy the whole boundary shares is passed at the one `decodeUnknown` call, never sprayed as identical annotations on every node.
- `onExcessProperty: "preserve"` is admitted ONLY where a pass-through envelope genuinely must carry undeclared keys, accepting that the decoded value is wider than its `Type`; reaching for `"preserve"` to dodge declaring a field is the field map left incomplete, the missing field added to the struct rather than smuggled through the excess channel — the `Type` must describe the value the codec produces, and `"preserve"` is the one deliberate exception named at the seam.
- `errors: "all"` is the decode mode for a boundary that reports every defect at once (a form, a batch ingest); a single-value decode whose first failure aborts the operation stays on the default `"first"` short-circuit, since accumulating issues no consumer reads is the full-error fold paid for with no reader — the option is selected by whether a downstream consumes the multi-issue `Composite`, not by reflex.
- `propertyOrder: "original"` is reached only by an order-consuming downstream (canonical serialization, deterministic hashing); applying it to a record read by key is the key-permutation that changes nothing observable, the scan cost paid for no effect — the default `"none"` is correct wherever output key order is not load-bearing, and naming `"original"` without that consumer is the rejected register.
- A `Forbidden` issue from a sync runner over an effectful codec is a runner-codec MISMATCH, not a data defect — the fix is the `Effect` runner the codec's non-`never` `Context` demands, never `runSync` wrapped in a retry or the effectful leg downgraded to a pure one to satisfy a sync caller; the option surface surfaces the synchrony requirement at the boundary, and dodging it by forcing the codec synchronous is the requirement erased rather than provided.
