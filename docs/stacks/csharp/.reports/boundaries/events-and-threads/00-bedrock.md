# Events and Threads — Core Findings

[SUBSCRIPTION_AS_DISPOSABLE_VALUE]:
- A subscription is the detacher token returned by the attach site. Because `event -= handler` requires the exact delegate identity used for `+=`, the wrapper delegate must be stored at subscribe time and reused for detach. The canonical form is a `sealed class` implementing `IDisposable` that holds a `Seq<Action>` of detach lambdas, each closing over the exact wrapper instance — not a secondary reference to the handler. A lambda registered directly as `source.Change += v => Action(v)` creates an anonymous delegate whose reference is immediately lost — detach is impossible without capturing the delegate identity to a variable before registration.
- `CancellationToken.Register` returns `CancellationTokenRegistration`, which is both `IDisposable` and `IAsyncDisposable`. Failing to dispose leaks the callback reference and prevents the registered action from being collected. When the token is already cancelled at registration time, the callback fires synchronously on the calling thread before `Register` returns; no subscription object is enqueued. Synchronous `Dispose` unregisters and, if the callback is currently executing on another thread, blocks synchronously until it finishes via `WaitForCallbackToComplete`. `DisposeAsync` provides the same rendezvous guarantee asynchronously — `WaitForCallbackToCompleteAsync` is returned when a callback is executing, allowing the caller to `await` completion without blocking a thread. Both forms wait for in-flight callbacks; the difference is whether the wait occupies a thread.
- `CancellationToken.UnsafeRegister(Action<object?, CancellationToken>, object?)` skips both `ExecutionContext.Capture` and `SynchronizationContext.Current` capture entirely — the callback runs with whatever context the cancelling thread carries. Every `Register` overload passes `useExecutionContext: true` regardless of `useSynchronizationContext`; the two flags are independent. `UnsafeRegister` is the correct form for callbacks that perform only context-free work — `ChannelWriter.TryWrite`, `Atom.Swap`, `ManualResetEventSlim.Set` — and do not read `AsyncLocal<T>` values from the registration context. The `Action<object?, CancellationToken>` overload receives the token directly, allowing `IsCancellationRequested` inspection without a closure. Inside a hot-path callback, `IsCancellationRequested` is a cheap volatile read; inspect it directly to exit early rather than closing over external state.
- `CancellationTokenSource.TryReset()` returns `true` and unregisters all callbacks via `UnregisterAll()` only when `_state == NotCanceledState` AND either the timer is null or the timer has not yet fired. The `_everQueued` flag is set when the timer is queued to fire; once set it is never cleared. A CTS used with `CancelAfter` where the timeout elapsed sets `_everQueued = true` — `TryReset` returns `false` permanently. Safe CTS pooling requires: `TryReset` returns `true`, the token was never cancelled, and `UnregisterAll` has run. The only poolable pattern is `CancelAfter`→`TryReset` before the timer fires. `EnvIO.New` registers a parent-propagation callback at construction; a pooled CTS reused in `EnvIO` must have `TryReset` called before re-passing it to `New`.
- Subscription sets opened outside a reactivation scope accumulate across reactivation cycles. The invariant is: the subscription set is destroyed at the same point the owner transitions away from the live state; reactivation constructs a fresh set rather than appending to a retained one.
- On deactivation or disposal, all event handlers must be detached before any acquired handle is disposed. Disposal before detach allows a racing callback to touch a handle mid-teardown. The order is strictly: detach all subscriptions, then dispose handles. Host-owned buses deregister independently of the local subscription record — re-querying availability after any host lifecycle event is required; a liveness probe at every callback entry is the correct form before acting on a consumed item.

[RESOURCE_DISPOSAL_ORDER_AND_ASYNC_HAZARD]:
- `Resources.DisposeU` iterates the `AtomHashMap<object, TrackedResource>` and calls each resource's `Release().Run(envIO2)` synchronously. `AtomHashMap` iteration follows HAMT traversal order, which is determined by hash values — effectively arbitrary from the caller's perspective. There is no LIFO, FIFO, or insertion-order guarantee. For teardown sequences where a subscriber must be detached before its connection is disposed, acquire resources with explicit `use` and `release` pairs in sequence and call `release` on the subscription token explicitly inside the computation before the connection `use` exits. Do not rely on scope disposal order for ordered teardown — encode the order as a sequential `IO` chain:

```csharp
from conn  in use(IO.lift(OpenConnection))
from token in use(IO.lift(() => (IDisposable)Subscribe(conn)))
from _     in IO.yieldFor(Duration.Zero) // ... actual work ...
from __    in release(token)             // explicit detach first
select unit                              // conn release follows scope exit
```

- `Resources.AcquireAsync<A>(A value) where A : IAsyncDisposable` stores a `TrackedResourceAsyncDisposable<A>`, whose `Release()` returns `IO.liftAsync(async delegate { await Value.DisposeAsync().ConfigureAwait(false); return unit; })`. `Resources.DisposeU` calls `resource.Value.Release().Run(envIO2)` synchronously — `IO.Run` drives the async computation to completion via `GetAwaiter().GetResult()`, blocking the disposing thread until `DisposeAsync()` completes. The hazard: `DisposeU` is called from `IOLocal.Invoke`'s `using` block and from `IOLocalOnFailOnly.Invoke`'s `catch`, both of which may execute on an async state machine continuation running on the captured `SynchronizationContext`. If `DisposeAsync()` needs to marshal back to the same context for completion, the `.GetResult()` block deadlocks. Resources whose `DisposeAsync` marshals to a specific context must not be acquired inside a `WithEnv` or `BracketFail` block that runs on that context — use `Prelude.use(() => new T(...))` with explicit `release` outside any scope that runs on the target context.
- When `TrackedResourceAsyncDisposable` and `TrackedResourceDisposable` coexist in the same `Resources`, disposal order between them is HAMT-order (undefined). When an async-disposable must be released before a synchronous disposable, encode the order explicitly by calling `release` in sequence inside the computation rather than relying on scope disposal sequence.

[SAFEHANDLE_CALLBACK_WINDOW_PROTOCOL]:
- `SafeHandle.DangerousAddRef(ref bool success)` increments the internal reference counter, preventing `ReleaseHandle` from being called while the counter is elevated. The `success` parameter must be checked before calling `DangerousGetHandle`; if `success` is `false`, the handle was already closed. The symmetric form is always `try { handle.DangerousAddRef(ref success); if (success) { /* use */ } } finally { if (success) handle.DangerousRelease(); }`. Calling `DangerousRelease` without a matching `DangerousAddRef` corrupts internal state.
- In an event callback window, the subscribe–use–unsubscribe triplet for a `SafeHandle`-wrapped resource is: (1) `DangerousAddRef` before registering the handler, (2) the handler body calls `DangerousGetHandle`, (3) `DangerousRelease` after detach completes. Detach synchronously before calling `DangerousRelease`; guard the handler body with an `IsInvalid` or liveness check on the acquired pointer.

[ATOM_CHANGE_EVENTS]:
- `Atom<A>.Change` is a raw C# multicast delegate field (`event AtomChangedEvent<A>`). It fires synchronously on the swapping thread immediately after the `Interlocked.CompareExchange` succeeds, before `Swap` returns. The swap function may be called multiple times in the spin loop; the event fires exactly once with the committed value. The swap function must be side-effect free; the change event is the single notification point. Because `Change` fires synchronously inside the CAS success path, subscribers that block inside the change handler deadlock — the swap does not return until the change event returns. Change handlers must be non-blocking; cross-thread work must be submitted via channel write, not executed inline. Every handler in one `?.Invoke` call receives the identical committed value as its event argument; a handler that re-reads `atom.Value` after the event may observe a newer value from a concurrent swap, as line 32 notes.
- `Atom<A>` holds a strong reference to every registered delegate, which holds a strong reference to its target. A long-lived static atom with activation-scoped subscribers prevents GC of those objects after deactivation. Detach all `Atom.Change` subscriptions at deactivation using the exact stored delegate reference — capture the delegate identity to a variable before registration, then use that variable for `-=`.
- `Atom<M, A>` (the metadata-carrying form) exposes the same `Change` event; the swap function receives `(M metadata, A current)`. The change event carries only the new value, not the metadata or prior value. A subscriber needing the prior value must maintain a shadow read of `atom.Value` before the swap. Reading `atom.Value` after the event fires may already reflect a further-advanced value if another swap occurred during the handler.
- `AtomHashMap<K, V>.Change` fires as `AtomHashMapChangeEvent<K, V>`, carrying a `HashMapPatch<K, V>` that includes the previous map, the current map, and either a single key with its `Change<V>` or a trie of all changed keys. The event fires synchronously inside `AnnounceChange` on the swapping thread. `AtomHashMap<K, V>.Swap(TrackingHashMap → TrackingHashMap)` emits change tracking only for transforms applied to the `TrackingHashMap` argument provided by the swap function — if the swap function ignores the argument and builds a fresh `TrackingHashMap`, the change event carries a diff showing only the final state with no tracked intermediate changes. The correct pattern always threads the provided map through transforms (`map.SetItem`, `map.Remove`) rather than constructing a new collection.
- `Ref<A>.Change` (STM-backed) fires inside the STM `finally` block for every transaction iteration, including conflict-aborted ones. `t.changes` is the in-transaction proposed diff accumulated by `WriteValue`; it is not empty on conflict. Subscribers to `Ref<A>.Change` receive notifications for rolled-back writes and must re-read the committed value if consistency is required.

[THREAD_MARSHAL_EXPLICIT_BOUNDARY]:
- `IO<A>.Post()` marshals the computation onto the `SynchronizationContext` captured in `EnvIO.SyncContext` at construction time. The context is captured once at `EnvIO.New()`; all `Post()` calls inside that environment target the same captured context. When `env.SyncContext` is `null`, `Post` falls through to `RunAsync` on the current scheduler — no actual marshaling occurs, so seams requiring thread affinity must verify the context is non-null before composition. Code that relies on `Post()` for thread safety while running in a null-context environment silently gets inline execution with no actual marshaling.
- `IO.Post()` allocates an `AutoResetEvent` per invocation and calls `SynchronizationContext.Post` followed by `await wait.WaitOneAsync(env.Token)`. `WaitOneAsync` uses `ThreadPool.RegisterWaitForSingleObject` internally — three kernel transitions per `Post()` call. For per-frame or per-event dispatch at high frequency, `Post()` is not viable; use `Channel<T>` or `AtomHashMap` state cells to batch work and marshal the batch once.
- `IO.syncContext` is an `IO<Option<SynchronizationContext>>` effect that reads the live `EnvIO.SyncContext` at evaluation time. Evaluate it at the composition root on the correct thread, inject the result into `EnvIO.New`, and pass the environment down. Never re-read `SynchronizationContext.Current` inside an effect.
- `EnvIO` exposes four distinct local-scope constructors with different ownership profiles. `Local` allocates new `Resources`, new `CancellationTokenSource`, and reads `SynchronizationContext.Current` at access time — full isolation. `LocalResources` allocates new `Resources` only; it inherits the parent `Source` and `SyncContext`. `LocalCancel` allocates a new `CancellationTokenSource` only; it inherits the parent `Resources` and `SyncContext`. `LocalSyncContext` passes the parent `Source` through (`New(Resources, Token, Source, SynchronizationContext.Current)`) — no new CTS is allocated, Own bit 1 is not set, and the parent CTS is never disposed when the child env is released. It only re-reads the ambient `SynchronizationContext` at access time. `LocalSyncContext` and `Local` are property getters, not construction-time captures — calling them on a background thread captures the wrong context. Evaluate them on the host thread and pass the resulting env down. `EnvIO.New` assigns ownership via two bits in the `Own` field: bit 1 means the caller allocated the CTS and will dispose it; bit 2 means the caller allocated the `Resources` and will dispose them. `DisposeResources` disposes only if bit 2 is set; `DisposeNonResources` disposes the CTS only if bit 1 is set and always disposes the stored `CancellationTokenRegistration`. `LocalResources` sets `Source` to the parent's source (not null), so bit 1 is not set and the parent CTS is never disposed. `LocalCancel` sets `Resources` to the parent's resources (not null), so bit 2 is not set — a subscription that depends on a resource allocated in the parent scope can be safely detached inside the child scope's computation because parent resources survive child env disposal.
- `SynchronizationContext.Post` is fire-and-queue; `SynchronizationContext.Send` is synchronous and deadlocks if the target thread is blocked waiting. `IO.Post()` uses `Post` and `AutoResetEvent.WaitOneAsync` for the caller-side rendezvous — the calling fiber blocks on the wait handle while the host thread processes synchronously. If the host context queues work behind an already-blocked host thread, this becomes a deadlock. The guard: only cross to the host thread from a fiber that is not itself holding the host thread. `Send` is never the correct form for a bidirectional host-thread call; `Post` with a completion signal via channel write or `TaskCompletionSource.SetResult` is the correct shape.
- `TaskScheduler.FromCurrentSynchronizationContext()` throws `InvalidOperationException` when `SynchronizationContext.Current` is `null` — this happens on thread-pool threads, `LongRunning` task threads, and non-UI background threads. The capture must happen on the host thread while that thread's context is installed; store it in the environment and inject it into effects rather than re-reading at call sites.
- `MonadUnliftIO<M>.PostIO` delegates to `MapIO(ma, io => io.Post())`, meaning `Eff<RT, A>.PostIO()` lifts through the runtime transformer and applies `Post()` to the inner `IO`. The sync context must be in the environment when the `Eff` runs; it cannot be injected after the fact.
- `IO<A>.Fork` passes `env.SyncContext` to the child `EnvIO.New`. The forked thread carries the parent's captured sync context. Any `IO.Post()` call inside the fork targets the same context as the parent — a fork spawned to protect the host thread from blocking work that still uses `Post()` does not reduce context-switch count; it only moves the blocking off the host thread.
- Host invoke primitives (`SynchronizationContext.Post`, `Dispatcher.InvokeAsync`, and equivalents) swallow any exception thrown inside the posted delegate — the exception does not propagate to the posting fiber. `IO.Post()` implements the explicit capture protocol: the posted delegate catches all exceptions into a local `error` variable and sets the `AutoResetEvent`; after the caller's `await wait.WaitOneAsync` completes, the caller fiber calls `error.Rethrow<A>()`, which re-throws preserving the original stack. Any marshaled work that is NOT wrapped in `IO.Post()` and instead fires-and-forgets via raw `SynchronizationContext.Post` loses exceptions silently. Every marshaled effect must either flow through `IO.Post()` or run inside an explicit catch-then-publish-to-fault-channel wrapper; there is no ambient exception propagation across a context post.

[EXECUTION_CONTEXT_FLOW_DISCIPLINE]:
- `CancellationToken.Register` captures the ambient `ExecutionContext` unconditionally at registration time — every overload passes `useExecutionContext: true` internally regardless of the `useSynchronizationContext` parameter. The two capture flags are independent; `useSynchronizationContext: false` suppresses `SynchronizationContext.Current` capture but does not affect EC capture. Only `UnsafeRegister` skips EC capture by passing `useExecutionContext: false`. The registered callback is always invoked with the captured EC restored unless `UnsafeRegister` was used.
- `ExecutionContext.Restore(ctx)` is a one-way install: it sets the current thread's execution context without saving the prior context or restoring it afterward. It is the building block for async state machines that must restore a captured context at `await` resumption points without the overhead of `Run`'s save-restore pair. Calling `Restore` in application-level code outside an async state machine produces a context leak — the thread continues with the installed context after the call site. `ExecutionContext.Run(ctx, callback, state)` saves the prior context, installs `ctx`, invokes the callback, and restores the prior context in a `finally` block — correct for event-dispatch wrappers that must not leak the event's context onto the calling thread after dispatch. Captured `ExecutionContext` instances are immutable and can be passed to `Run` multiple times without error.
- `ExecutionContext.SuppressFlow` suppresses `ExecutionContext` propagation to work queued from the current thread until the returned `AsyncFlowControl` is disposed. The suppression is thread-local; failing to capture the return value and call `Dispose` leaves the thread's flow suppressed for all subsequent work items queued from that thread. Nesting suppression calls throws.
- `ThreadPool.UnsafeQueueUserWorkItem<TState>(Action<TState>, TState, preferLocal: bool)` avoids both `ExecutionContext` capture and delegate allocation when combined with a struct `TState`. The `preferLocal: true` flag biases toward the current thread's local work-stealing queue, reducing cross-core contention for tight producer-consumer pairs on the same logical thread. Because `UnsafeQueueUserWorkItem` inherently skips EC capture regardless of ambient flow state, it is the allocation-free form for context-free callbacks even without an explicit `SuppressFlow`. The `IThreadPoolWorkItem` interface overload (`void Execute()`) is the correct form when the work item is itself the cancellable context: implement `Execute` to check a volatile flag before doing work. Note that passing a `struct` implementing `IThreadPoolWorkItem` directly to the `IThreadPoolWorkItem` overload boxes it at the call site — use the generic `Action<TState>` overload with a struct `TState` to avoid that allocation.
- `AsyncLocal<T>` propagates into child scopes (tasks, threads started with `Thread.Start`) but modifications inside a child scope do not flow back to the parent. `Thread.UnsafeStart` and `Thread.UnsafeStart(object?)` skip EC capture entirely — the spawned thread starts with no `AsyncLocal<T>` values, no ambient security or activity state, and incurs no EC copy allocation. The correct domain is callback-only workers that write to channels, swap atoms, or set completion sources; any worker that must propagate correlation or caller-injected `AsyncLocal` values must use `Thread.Start`.
- `AsyncLocal<T>(AsyncLocalValueChangedHandler<T>)` fires on every value change including implicit fork/join transitions: entering a child scope fires with `threadContextChanged = true`; a child setting a value fires with `threadContextChanged = false`; scope exit fires with the reverse. This hook enables ambient-context-sensitive subscription management without explicit plumbing. The callback runs synchronously on the thread that caused the change.
- `Activity.Current` is backed by `AsyncLocal<Activity> s_current`. A host event that fires on a dedicated thread without a managed context (such as a native callback via P/Invoke) carries `null` `Activity.Current`. `UnsafeRegister` callbacks skip EC restore entirely, so `Activity.Current` inside such a callback is the activity current on the cancelling thread — neither the registration thread's activity nor the consuming thread's activity. Set `Activity.Current` explicitly inside the callback or use `Run` with the captured context if the callback must operate under a specific activity. Using `Run` in a high-frequency callback introduces allocation (context copy) and full restore overhead; profile under load before adding `Run` to a tight event path.
- `ThreadPool.UnsafeRegisterWaitForSingleObject` is the allocation-free registration path for mapping native `WaitHandle`-backed signals to managed notifications without inheriting ambient context. The symmetric `WaitHandle.Unregister(WaitHandle)` must be called on teardown; failing to unregister leaks the OS wait registration even after the calling object is collected.

[CONFIGUREAWAIT_OPTIONS_AT_SEAM]:
- `ConfigureAwaitOptions` is a `[Flags]` enum with four values: `None = 0`, `ContinueOnCapturedContext = 1`, `SuppressThrowing = 2`, `ForceYielding = 4`. `ForceYielding` forces a continuation post even on an already-completed task — without it, awaiting a pre-completed task runs the handler synchronously on the event thread, causing reentrancy if the handler schedules more events or acquires shared state. `ForceYielding` on a host-event dispatch callback ensures the continuation does not run on the event source's stack frame; the resumption thread is determined by the captured context, not the event dispatcher.
- `ConfigureAwaitOptions.SuppressThrowing` is valid only on the non-generic `Task`, not `Task<T>`. Applying it to `Task<T>.ConfigureAwait(options)` throws `ArgumentOutOfRangeException` at runtime, not at compile time; CA2261 is the diagnostic. The correct pattern: cast `Task<T>` to `Task`, await with `SuppressThrowing`, then inspect the original `Task<T>` handle for `IsFaulted`, `Exception.InnerException`, or `IsCanceled`. This asymmetry makes `SuppressThrowing` a non-result-carrying boundary guard — correct only when the caller has a separate fault channel and the task fault is already emitted there; using it without inspecting the outcome silently discards errors.
- The combination `ForceYielding | SuppressThrowing` (value 6) yields before running and does not propagate faults through the await site; the non-generic `Task` cast is still required for the `SuppressThrowing` component.

[FORK_AND_CONCURRENCY]:
- `IO<A>.Fork(Option<TimeSpan>)` spawns a dedicated OS thread via `Task.Factory.StartNew` with `TaskCreationOptions.LongRunning`, bypassing the managed thread pool entirely. The forked computation receives a child `Resources` (constructed from the parent's `Resources`) and a child `CancellationTokenSource`; parent cancellation propagates via `env.Token.Register(tsrc.Cancel)`. Both the `CleanUp` (owns `CancellationTokenSource` + `CancellationTokenRegistration`) and the child `Resources` are disposed in the task's `finally` block on the forked thread. Because `LongRunning` threads are not pool-bounded, spawning many forks concurrently creates a proportional number of OS threads; the thread budget must be treated as a construction-time constraint on the seam owner.
- `ForkIO<A>` is a `readonly record struct` with two `IO` effects: `Cancel` and `Await`. Neither executes unless bound. The forked task runs unconditionally after `Fork` returns; the only cancellation paths are binding `Cancel` or cancelling the parent `EnvIO` token.
- Parent-to-child resource sharing is by reference: the child `Resources` holds a `parent` pointer. `Release` walks up to the parent if the key is not found locally. Disposing the parent environment while the fork is live invalidates the parent resource scope; the fork's own `finally` releases only the child scope.
- `IO<A>.Timeout(TimeSpan)` runs the body inline under a timer-armed local environment constructed by `envIO.LocalCancelWithTimeout(TimeLimit)`, which calls `EnvIO.New(Resources, Token, null, SyncContext, timeout)` — allocating a fresh `CancellationTokenSource(timeout)` and linking parent cancellation. The body executes synchronously on the calling fiber; there is no fork and no race. When the timer fires, the child CTS cancels, and the next cancellation-observing primitive in the body throws `OperationCanceledException`. `awaitAny` over raw effects or fork-constructing arguments terminates losers when the first completes; `awaitAny` over already-running `ForkIO<A>` handles awaits under the parent environment and cancels nothing it owns — losers run to completion, and guaranteed fan-in cleanup over handles cancels each loser explicitly in a `Finally`. `Task.WhenAny` leaves all tasks running. These are not interchangeable when fan-in cleanup matters. `awaitAll<A>(ForkIO<A>[])` collects every result or accumulates every error without short-circuiting on first failure — a seam that forks N boundary probes with `awaitAll` must treat the error sequence as a partial-availability report, not a single fault.
- `Parallel.ForEachAsync<TSource>(IAsyncEnumerable<TSource>, ParallelOptions, Func<TSource, CancellationToken, ValueTask>)` consumes an async source with bounded parallelism. `ParallelOptions.MaxDegreeOfParallelism` caps concurrent invocations; cancelling the options token terminates all pending bodies. This is the correct BCL form for bounded-parallel event processing over a channel-backed `IAsyncEnumerable` source when effect-rail fork management is not required.
- `ThreadPool.SetMaxThreads` and `SetMinThreads` are the construction-time caps for the managed pool. Any native thread-pool or host-specific cap that differs must be recorded as construction evidence on the seam owner.

[CHANNEL_AS_SEAM_MARKER]:
- `Channel.CreateUnbounded<T>()` runs continuations asynchronously by default. The single-reader optimization (`SingleReader = true`) routes to a lock-free `SingleConsumerUnboundedChannel<T>`, which is safe only when a single consumer issues `ReadAsync` or `TryRead` at a time; concurrent readers corrupt state.
- `Channel.CreateBounded<T>(0)` produces an internal `RendezvousChannel<T>`: every `WriteAsync` blocks until a matching `ReadAsync` is pending. Using it as a queue is an error; using it as an explicit phase-handoff between two independent async workers is the intended shape. `RendezvousChannel<T>` carries `CanCount = true` and `CanPeek = true`; `Count` always returns `0`; `TryPeek` inspects the head of the blocked-writers queue under a lock and returns the item waiting for handoff without consuming it. `TryRead` pops the head writer from the blocked-writers queue, returns its item, and signals the writer's completion, unblocking `WriteAsync`. The `_readerSingleton` and `_waiterSingleton` are pooled, pre-allocated async operation objects reused across calls for the common single-reader-single-writer pair. `TryRead` without a prior `WaitToReadAsync` returns `false` if no writer is blocked — the reader must be waiting before the writer arrives, or vice versa; the rendezvous is a synchronization point, not a queue.
- `Channel.CreateUnboundedPrioritized<T>(UnboundedPrioritizedChannelOptions<T>)` accepts an `IComparer<T>`; items are dequeued in comparator order regardless of insertion order. When `Comparer` is `null`, `IComparable<T>` default ordering applies.
- `BoundedChannelFullMode.DropWrite` with the `itemDropped` callback is the correct form for observer-owned rate limiting: the channel defines the capacity policy; the callback surfaces the dropped value for telemetry without blocking the producer.
- `ChannelReader<T>` and `ChannelWriter<T>` are independent abstract surfaces. Passing `ChannelReader<T>` to a consumer and `ChannelWriter<T>` to a producer enforces the seam. The `Channel<T>` owner holds both and is responsible for completing the writer side on shutdown; `Completion` on the reader settles only after `TryComplete` is called on the writer.
- `ChannelReader<T>.ReadAllAsync(CancellationToken)` consumes until the channel is completed; it does not complete on cancellation alone — the channel must be closed by the producer. Rate-limited consumers must use `WaitToReadAsync` + `TryRead` in a loop to interpose pacing between dequeues.
- `ChannelOptions.AllowSynchronousContinuations = true` allows the thread calling `WriteAsync` or `TryWrite` to invoke the continuation of an awaiting `ReadAsync` inline, before the write returns. This eliminates a thread switch for tightly coupled pairs but introduces two hazards: unbounded stack growth when the continuation itself writes to the same or a downstream channel, and reentrancy when the writer holds a lock or is inside a callback the reader's continuation also needs. The default `false` posts continuations asynchronously, breaking the call chain.
- `ChannelWriter<T>.Complete(Exception? error)` closes the channel; when `error` is non-null the error does not surface immediately — `ChannelReader<T>.Completion` transitions to faulted only after all buffered items are drained. Once drained, `ReadAsync` throws `ChannelClosedException` with the original error as `InnerException`, via `ChannelUtilities.CreateInvalidCompletionException(inner)`. `ChannelClosedException` wraps the fault in both the normal and error-completed cases; the `InnerException` is `null` for a normal close and the original exception for an error close. `WaitToReadAsync` and `ReadAllAsync` on an error-completed channel surface the original exception type directly rather than wrapping it in `ChannelClosedException`. A fold worker that uses `primaryWriter.Complete(error)` to terminate projection delivers the fault to `ReadAsync` callers as `ChannelClosedException(inner: error)` and to `ReadAllAsync` callers as the unwrapped original; a dedicated `ChannelWriter<Error>` is architecturally cleaner when fault consumers and projection consumers must react differently.
- `ChannelReader<T>.CanCount` and `CanPeek` are capability flags; `Count` throws `NotSupportedException` and `TryPeek` returns `false` if the respective capability is absent. Capability guards before using these must be explicit rather than try/catch, because the exception path obscures logic.

[STATE_CELLS_AND_SYNCHRONIZATION_GATES]:
- `System.Threading.Lock` replaces `object` as the lock target. The `lock` keyword on a `Lock` instance generates `EnterScope()` and `Scope.Dispose()` instead of `Monitor.Enter`/`Exit`, eliminating the hidden boxing and `SyncBlock` slot overhead. `IsHeldByCurrentThread` allows reentrancy detection without a try/finally guard.
- `Lock.EnterScope()` returns a `ref struct Scope`, making it correct inside `unsafe` kernels and stack-allocated contexts where heap-allocated `IDisposable` is illegal. The `Scope` is not valid across `await` points; a gate that must span an async region requires `SemaphoreSlim(1, 1)`. `Lock.Scope.Dispose` throws `SynchronizationLockException` if the current thread does not hold the lock at disposal time. The `ref struct` constraint prevents the most common misuses at compile time but does not cover IL-level or reflection paths. `Lock.IsHeldByCurrentThread` enables reentrancy detection; the correct form for seams that may be called from both lock-owning and non-owning code: `if (!_gate.IsHeldByCurrentThread) using (_gate.EnterScope()) { ... }`.
- An `Atom<Option<T>>` cell with `SwapMaybe` is the correct shape for a singleton ownership seam: acquire sets `Some(token)`, release clears back to `None`. `SwapMaybe` short-circuits when `f` returns `None`, leaving the cell unchanged; this is the safe conditional-take pattern.
- A `Lock` or `Monitor` gate is earned when the invariant requires multiple in-scope reads and writes to be mutually exclusive. An `Atom` is earned when a single value must transition atomically with no requirement for mutual exclusion across multiple values. An STM `Ref` is earned when multiple values must change atomically as a unit. Conflating these produces either over-serialization (wrapping independent atoms in a lock) or under-protection (using atoms for multi-value invariants).

[SUBSCRIPTION_FENCING_AND_LATE_SUBSCRIPTION]:
- Late-subscription semantics must be declared at the seam: a subscriber that arrives after the first emission must be told whether it receives replay-last, snapshot-plus-delta, or rejection. Absence of this declaration causes ordering hazards where a consumer misses the initial state and receives only incremental diffs without a baseline.
- `Channel`-borne delivery is commit-ordered and non-missable for active consumers. A subscriber that starts consuming after items have been written but before the channel is completed sees all buffered items in order — snapshot-plus-delta for free when the channel carries committed changes.
- Fault receipts must flow on a separate `ChannelWriter<Error>` from projection results. Mixing errors into a `Channel<Either<Error, T>>` forces every consumer to unpack the sum type on every read; a dedicated fault channel lets nominal consumers remain total and fault monitors remain disjoint.
- A hot-path that submits typed intent and consumes cached receipts must never block on compute or storage in the submission loop. The pattern: `TryWrite` the intent onto an unbounded channel, return a pre-allocated receipt handle to the caller, and resolve the receipt from a single named serial fold worker. The fold worker owns the channel read loop and mutates no external state outside its own result cell.

[PRODUCERS_AND_CONSUMERS]:
- Producer never throttles for consumer. Back-pressure is the consumer's observable, surfaced via the channel's `FullMode`. `WaitToWriteAsync` blocks the producer when a bounded channel is full; the consumer sets its own pace by controlling the read rate. A producer that delays to pace for an assumed consumer speed is incorrect.
- Observer-owned rate limiting: the consumer reads at a rate governed by its own schedule-driven loop, `WaitToReadAsync` return timing, or explicit delay in the read loop. Combining `DropOldest` or `DropNewest` with an `itemDropped` callback gives the consumer telemetry about shed load without blocking the producer.
- Pre-frame propagation of scale or DPI changes follows the same law: the producer writes to a channel; the consumer reads with `TryRead` at frame start, applying the latest value. Reading the change inside the frame work causes a one-frame-behind rendering artifact.
- A single named serial fold worker is the `IO`/`Eff` surface pattern for repeating accumulation: `IO<A>.Repeat(Schedule)` with state threaded across iterations, or a `PeriodicTimer`-driven loop with a `TryRead` drain per tick. The fold runs in sequence; concurrent writes to the state cell from outside the fold break the serial invariant. Drain rendezvous is via `RepeatUntil` with a terminal predicate or by cancelling the environment token.

[TASK_COMPLETION_AND_FAN_OUT_PRIMITIVES]:
- `Task.WhenEach<TResult>(ReadOnlySpan<Task<TResult>>)` yields tasks as `IAsyncEnumerable<Task<TResult>>` in completion order, replacing manual `Task.WhenAny`-in-a-loop without tracking a remaining set. `WithCancellation(ct)` on the returned `IAsyncEnumerable` stops iteration but does not cancel the underlying tasks. Faulted tasks are yielded normally — `completed.IsFaulted` must be checked per task.
- `Task.WhenAll(ReadOnlySpan<Task>)`, `Task.WhenAll<TResult>(ReadOnlySpan<Task<TResult>>)`, and `Task.WhenAny(ReadOnlySpan<Task>)` accept span overloads that avoid the `Task[]` allocation for fixed-arity fan-out. `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` eliminates the array allocation for token-merging at seams that link two or more parent tokens.
- `TaskCompletionSource<T>.SetFromTask(Task<T>)` and `TrySetFromTask(Task<T>)` propagate the terminal state (success, fault, or cancellation) of a completed source task atomically to the `TaskCompletionSource`. `TrySetFromTask` returns `false` if the TCS is already transitioned, making it safe for concurrent completion races. Both require the source task to be completed before calling; passing an incomplete task throws `ArgumentException`.

[PERIODIC_TIMER_AND_TIME_PROVIDER_SEAM]:
- `PeriodicTimer(TimeSpan, TimeProvider)` accepts a `TimeProvider` injection point for deterministic testing. `WaitForNextTickAsync` cancellation via its `CancellationToken` affects only the single outstanding wait; the underlying timer continues firing. Stopping the timer requires calling `Dispose()`, which completes the current `WaitForNextTickAsync` with `false`. The read loop pattern `while (await timer.WaitForNextTickAsync(ct)) { ... }` exits on either cancellation (throws `OperationCanceledException`) or disposal (returns `false`). Missing the `false` return causes the loop to exit without the worker having been explicitly stopped.
- `PeriodicTimer.Period` is a mutable property. Setting it after construction changes the period for all future ticks but does not affect the currently pending `WaitForNextTickAsync`. Rate changes at a seam boundary update `Period` on the next cycle rather than requiring a new timer instance.
- A `PeriodicTimer` used as a fold worker driver must be owned and disposed by the worker itself, not by a parent scope that may outlive the worker's subscription set. If the parent disposes the timer while the worker is between ticks, `WaitForNextTickAsync` returns `false` cleanly. If the parent disposes it while the worker is inside the loop body, the current iteration completes and `false` surfaces on the next wait. No in-flight work is interrupted.
- In a rate-limited consumer, the `PeriodicTimer` drives the outer tick loop; inside each tick, `TryRead` drains all items that arrived in the interval. Producer writes unconditionally; the channel's `BoundedChannelFullMode` governs back-pressure. Consumer pacing in the timer period and producer back-pressure in the channel capacity are independent — conflating them by delaying producer writes to match consumer rate is incorrect.

[IO_REPEAT_RETRY_AND_SUBSCRIPTION_LIFETIME]:
- `IO<A>.RepeatUntil(Func<A,bool>)` is implemented as `Bracket().Bind(v => predicate(v) ? IO.pure(v) : RepeatUntil(predicate))`. Each iteration wraps the body in `Bracket()` = `WithEnv(env => env.LocalResources)`, creating a child resource scope disposed after each cycle. Resources acquired inside the body are released at the end of each cycle. A persistent subscription must be acquired outside the repeat body — opened at the level of the `IO` chain that contains the `Repeat`, not inside the repeated computation. `IO<A>.Repeat(Schedule)`, `RepeatWhile`, and `RepeatUntil` are all CPS-recursive `Bind` chains, not iterative loops — each iteration adds one frame depth to the `IO` DSL evaluation chain; for long-running fold workers, `IO<A>.Repeat(schedule).WithState` or equivalent schedule-driven state-machine patterns avoid deep nesting.
- `IO<A>.RetryUntil(Func<Error,bool>)` is `Catch(BracketFail(), ...)`. `BracketFail()` = `WithEnvFail(env => env.LocalResources)`: resources are released only on failure. On success, the iteration's resources are retained in the parent scope. A retry loop that opens a connection on each attempt accumulates one live connection (the last successful one); each failed attempt cleans up. The first success's resource escapes the `BracketFail` scope into the parent.
- `IO<A>.Uninterruptible` constructs `IOUninterruptible<A,A>`, whose `Invoke` creates `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — passing `CancellationToken.None` and `null` for the source. Because source is null, `New` allocates a fresh `CancellationTokenSource`, sets Own bit 1, and because `CancellationToken.None.CanBeCanceled` is false, installs no parent-propagation registration. The computation runs with an uncancellable token completely decoupled from the parent scope. Resources are shared (Own bit 2 = 0). On completion, the child env's `DisposeNonResources` disposes only the locally owned CTS — resources are never touched. A computation that awaits on a host service internally polling its own token receives the `None` token — if that service uses `env.Token` to gate its loop, it will never observe cancellation. Finalization paths must themselves complete within finite time. For teardown sequences that must not be interrupted AND must clean up their own subscriptions, the correct composition is `Uninterruptible` wrapping a `Bracket` whose finalizer runs the detach-and-dispose sequence.
- `IO<A>.Finally<B>(IO<B>)` runs the `finally` effect after `this` regardless of success or failure and discards the `finally` result; the original result or error propagates. For teardown effects whose own return value must be preserved, `Bracket` is the correct form; for side-effecting cleanup where the result is irrelevant, `Finally` suffices.
- `WithEnv(env => env.LocalResources)` is the scoping form for subscription lifetime within an `IO` chain: resources acquired inside the mapped computation are released in a child scope on completion without touching the parent scope's resources or cancellation token. `Bracket()` wraps the whole computation in a child resource scope and releases all tracked resources on both success and failure. `BracketFail()` creates a child resource scope and releases it only on failure — correct for subscriptions where success deliberately transfers resource ownership out of the bracket. `IOLocalOnFailOnly.Invoke` does not `using` the child env — on success it calls only `local.DisposeNonResources()` in the `finally` block (CTS disposed, resources retained); on failure, `local.DisposeResources()` in the `catch` block releases the child resource scope. `WithEnvFail` paired with an outer `Atom<Option<Resources>>` ownership token allows manual release on deactivation: the activation computation acquires subscriptions inside `WithEnvFail`; on success, subscriptions escape bound to the `Atom<Option<Resources>>` token; on failure, `WithEnvFail` cleans up the partial subscription set automatically. Explicit deactivation swaps `None` in and calls `DisposeU` on the evicted `Resources` value.
- `EnvIO.LocalCancel` forks a new cancellation token sourced from the parent while inheriting the parent's resources and sync context. This is the correct form for a guarded region that must be cancellable independently of the parent — for example, a subscription whose read loop can be stopped without tearing down the parent environment's resources. Combined with `Retry`, each failed activation attempt cleans up its partial subscriptions via the child `BracketFail` scope before the next attempt begins.

[SIGNAL_AND_COUNTDOWN_LIFECYCLE_AT_SEAM]:
- `Signal.autoReset<M>()` and `Signal.manualReset<M>()` both wrap `new EventWaitHandle(...)` with a `use` operator, tracking it in the current `Resources` scope. The underlying `EventWaitHandle` is disposed when the resource scope is released. A `Signal` consumed inside a `Repeat` body is disposed after each iteration; a `Signal` used as a cross-iteration synchronization point must be acquired outside the `Repeat` computation.
- `Signal<M>.Wait()` calls `IO.liftAsync(e => handle.WaitOneAsync(e.Token))`. `WaitOneAsync` uses `ThreadPool.RegisterWaitForSingleObject` with EC capture — not the `Unsafe` variant. The resumption carries the `ExecutionContext` from the `await` site inside the `liftAsync` lambda, not from the thread that called `SetResult` on the TCS.
- `CountdownSignal<M>.Wait()` calls `CountdownEvent.Wait(CancellationToken)`, blocking the thread pool thread until the count reaches zero or the token fires. For high-count drains, prefer `PeriodicTimer`-driven polling of `handle.IsSet` or submit the wait to a dedicated thread via `Fork`.

[MANUAL_RESET_VALUE_TASK_SOURCE_CORE_AS_ZERO_ALLOC_EVENT]:
- `ManualResetValueTaskSourceCore<TResult>` (a `struct` in `System.Threading.Tasks.Sources`) is the zero-allocation foundation for reusable async events. `Reset()` clears all fields and increments `Version`; an awaiter holding a stale version number causes `ValidateToken` inside `GetResult` or `GetStatus` to call `ThrowHelper.ThrowInvalidOperationException()`, throwing `InvalidOperationException` — the version token is the stale-detection mechanism. The protocol requires one-producer, one-consumer, reset-before-next-wait discipline: `Reset()` must be called before the next consumer `await`; calling it while an awaiter is active corrupts the version.
- When `RunContinuationsAsynchronously = false`, the continuation is invoked inline on the completing thread before `SetResult` returns — the same reentrancy hazard as `ChannelOptions.AllowSynchronousContinuations = true`. When `RunContinuationsAsynchronously = true`, continuation is submitted via `ThreadPool.UnsafeQueueUserWorkItem(action, state, preferLocal: true)` — no `ExecutionContext` capture; the `preferLocal` flag promotes same-core resumption for tightly coupled producer-consumer pairs.
- The intended seam use: a host event that fires on a specific thread implements `IValueTaskSource<bool>` over a `ManualResetValueTaskSourceCore<bool>`, calls `Reset()` to prepare a new wait cycle, and calls `SetResult(true)` inside the event callback. This is the correct zero-allocation shape for high-frequency host-event-to-managed-awaiter bridging when the allocation of `TaskCompletionSource<T>` per event is unacceptable.
