# Wake and Handoff Substrate

[SYNCHRONOUS_CHANGE_DELIVERY]:
- The single-value cell's change event fires synchronously on the committing thread inside the post-CAS path before the swap returns, which makes a handler that re-swaps the same cell a genuine reentrancy hazard distinct from ordinary blocking: the inner swap completes and fires its own nested change before the outer swap's call frame returns, so a handler observing-then-reacting sees its own reaction's notification interleaved with the original. The substrate has no reentrancy guard; the handler that must trigger a further transition routes it through a channel so the second transition runs on a drained continuation, not nested on the committer's stack.
- A swap that loses the CAS race re-reads and re-applies, so under a storm of writes the last writer always wins and intermediate values are coalesced away rather than queued — the cell is structurally a latest-value register, not a log. This is exactly the semantics a per-frame or per-tick consumer wants from a high-frequency host event: the callback's only work is a context-free swap, the cell absorbs unbounded write frequency by collapsing to the latest, and a downstream drain reads one current value per tick. A consumer that must see every intermediate cannot use the cell as its transport and needs a channel, whose buffer is the log the cell deliberately is not.
- The cell holds a strong reference to every registered handler and transitively to its target, and the transactional reference shares the same retention shape; neither provides a weak-subscription mode. An activation-scoped subscriber on a process-static cell keeps its target alive past deactivation until detached by the exact delegate identity used to attach — and that detach must sequence before the handles the handler closes over are disposed, or a late commit fires the handler against a torn target.

[WAITHANDLE_TO_TASK_BRIDGE]:
- A boundary waiter that must suspend asynchronously on a kernel event cannot block a pool thread on a synchronous wait; the bridge registers the handle with the thread pool's single dedicated wait thread and completes a promise from the wait-or-timeout callback. One wait thread multiplexes thousands of registered handles, so a parked waiter consumes no pool thread until the handle fires — the structural answer to "park many waiters on one boundary transition cheaply."
- The bridge promise is a bare completion source constructed without the asynchronous-continuation option, so its result transition runs the awaiting continuation inline on the thread that completes it: the shared pool wait thread on a handle fire, or the cancellation-callback thread on token cancel. A waiter that resumes into expensive boundary work therefore stalls the shared wait thread or the cancelling thread for its whole synchronous continuation prefix; the fix is to request asynchronous continuations at construction or make the first resumed segment trivial and immediately yield.
- The registration captures execution context by default; the unsafe registration variant skips the capture. For a high-frequency boundary cell whose waiters carry no ambient context, the context-flowing default is pure per-wait allocation overhead and the unsafe variant is the correct seam.
- The callback registers with execute-only-once semantics and tears down in the `finally` via the registered-wait unregister plus disposal of the token registration. A waiter abandoned before reaching that `finally` — its enclosing scope faulting before the await — leaks both the registered wait slot on the shared wait thread and the token registration, and the leaked slot keeps the kernel handle pinned against finalization.
- A timeout argument of negative one means infinite; the callback's timed-out flag is inverted into the promise result, so the awaited boolean is true on a real signal and false on deadline expiry. A boundary that must distinguish "settled" from "deadline elapsed" reads this boolean directly rather than racing a separate timeout source — the timeout collapses into the same await.

[SIGNAL_CELL_ALGEBRA]:
- The signal cell wraps a kernel event handle registered into the effect runtime's resource scope at construction, so it auto-releases on scope teardown; three shapes share one surface. The auto-reset shape is the rendezvous baton for a single-consumer drain: each trigger hands off to exactly one parked waiter and immediately re-closes, so N triggers wake N waiters with no lost or doubled wake. The manual-reset shape is the latch for a one-shot transition — initialization-complete, drain-finished — where every late waiter must observe the settled state without re-arming; converting one to the other silently changes whether a second waiter is admitted.
- The countdown shape is the structured fan-in: a parent forks N boundary operations, each completion triggers one decrement, and the parent's wait releases when the last completes; the trigger returns whether the counter reached zero, so the final completer is identifiable at the trigger site for last-one-out teardown. The trigger is offered both effect-lifted and as an impure unsafe escape hatch — a cell triggered from a host callback already off the effect spine uses the unsafe form deliberately and owns the ordering itself.
- The auto- and manual-reset constructors ignore their initial-signaled argument and always construct the underlying handle unsignaled, so a cell intended to start latched — a manual-reset latch pre-set so the first waiter passes immediately — instead starts closed and parks that waiter forever. A pre-signaled start must be expressed by triggering the cell as the first effect after construction, never by the constructor argument.
- The countdown shape's count-taking trigger silently ignores its count argument and decrements by one — both the effect-lifted and unsafe count-taking forms forward to the single-step decrement — so a caller expecting a batch decrement of N undercounts by N−1 and the join never settles. Batch fan-in completion must loop single decrements or compute the join arithmetic externally.
- Over-triggering a settled countdown is not a benign no-op: a decrement past zero throws rather than saturating, so a completion count that can race above its initial budget must gate triggers or pre-size the counter to the true maximum; the increment side likewise throws once the counter has reached zero, so dynamic fan-in must grow the join before the count can settle, never after.
- The countdown shape's wait blocks the calling thread under token guard rather than riding the async pool-wait bridge the auto- and manual-reset shapes use, so a countdown join pins the awaiting thread for the whole fan-in. For a many-waiter join, the auto-reset-plus-bridge composition scales where the countdown wait does not.

[CONTINUATION_INLINING_DISCIPLINE]:
- Every promise-backed boundary signal that completes from a foreign thread — a host event, a native callback, a timer fire — inlines its awaiting continuation onto that foreign thread unless the promise was constructed to queue continuations. The default is inline, the option is a constructor argument and not a per-await choice, and a waiter cannot opt out of an inlining completer after the fact. A seam that completes waiters from a host UI thread or a single shared callback thread therefore runs arbitrary downstream logic on that thread unless every such signal source is constructed queuing.

```csharp
// inlining: the firing thread runs the awaiter's continuation prefix
var inlineGate = new TaskCompletionSource<TState>();
// queued: the firing thread only schedules the awaiter, never runs it
var safeGate   = new TaskCompletionSource<TState>(TaskCreationOptions.RunContinuationsAsynchronously);
fromHostThread.Register(static s => ((TaskCompletionSource<TState>)s!).TrySetResult(settled), safeGate);
```

- The channel-based drain seam takes the opposite default deliberately: each pending read or write operation carries its own continuation-asynchrony flag fixed at channel construction, so a channel built for asynchronous continuations never inlines a reader's resumption onto the writer's thread. For a producer-consumer pair that must not share a stack, the channel seam is structurally safer than a hand-rolled event-handle signal whose promise defaults to inline.

[FORK_HANDLE_SHARING]:
- The fork's await is itself a re-entrant effect that registers the awaiting environment's token against the fork's source — the await is not a passive read. A second consumer awaiting the same fork handle from a different cancellation scope can kill the shared computation by cancelling its own scope; isolate by handing each consumer its own fork, or by awaiting once into a published cell the consumers then read.
- Awaiting a fork after the parent token has already fired throws the cancellation directly rather than surfacing the body's last value, while awaiting after the body faulted surfaces the body's error through the rail. A seam that forks a long native call and later wants its partial result on cancel cannot get it from the await — the cancel path is terminal, and partial state must be published into a separate cell before the cancellable region begins.
