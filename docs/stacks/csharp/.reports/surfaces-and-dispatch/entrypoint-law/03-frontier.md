# Entrypoint Law — Frontier

[SCHEDULE_OPERATOR_TYPE_ALGEBRA]:
- `Schedule` and `ScheduleTransformer` are distinct types with a strict operator lattice. Methods returning `Schedule`: `spaced`, `linear`, `exponential`, `fibonacci`, `upto`, `fixedInterval`, `windowed`, and cron variants. Methods returning `ScheduleTransformer`: `recurs`, `maxDelay`, `maxCumulativeDelay`, `jitter`, `decorrelate`, `resetAfter`, `repeat`, `intersperse`; static fields `NoDelayOnFirst`, `RepeatForever`, `Identity` are also `ScheduleTransformer`. This is the load-bearing distinction for composition.
- The `|` operator is defined for `(Schedule, Schedule)`, `(Schedule, ScheduleTransformer)`, and `(ScheduleTransformer, Schedule)` — all returning `Schedule`. It is **not** defined for `(ScheduleTransformer, ScheduleTransformer)`. Chaining `Schedule.NoDelayOnFirst | Schedule.recurs(5) | Schedule.exponential(seed)` fails with CS0019 because the left-associative evaluation produces `ScheduleTransformer | ScheduleTransformer` in the first subexpression.
- The `+` operator on two `ScheduleTransformer` values produces function composition: `(t1 + t2).Apply(s)` is `t2.Apply(t1.Apply(s))`. This is the correct form for combining multiple transformers before applying them to a schedule: `Schedule.exponential(seed) | (Schedule.recurs(5) + Schedule.maxDelay(2000.0))` applies both transformers in one step. Alternatively: `Schedule.NoDelayOnFirst | (Schedule.recurs(5) | Schedule.exponential(seed))` — here the inner `|` reduces `ScheduleTransformer | Schedule` to a `Schedule` first, then the outer `|` applies `NoDelayOnFirst`.
- `Schedule | Schedule` is `Union`: runs to the length of the longest, yields the minimum of both delays at each step. `Schedule & Schedule` is `Intersect`: runs to the length of the shortest, yields the maximum of both delays. `Schedule + Schedule` is `Combine`: sequential append — when the first schedule exhausts its steps, the second schedule continues. These three semantics are orthogonal and the symbols are not interchangeable with intent.
- `ScheduleTransformer` carries an implicit conversion to `Schedule` via `Schedule.Forever | t`. This means a bare `ScheduleTransformer` used where a `Schedule` is expected (e.g., as an argument to `retry`) quietly wraps it in `Forever` and then applies the transformer — yielding an infinite base schedule bounded only by the transformer's own count limit. An unwrapped `Schedule.recurs(5)` passed to `retry` is therefore equivalent to `retry(Schedule.Forever | Schedule.recurs(5), computation)`, which is finite. This implicit conversion is the mechanism by which bare transformers are valid retry-schedule arguments.

[EXTENSION_BLOCK_STRUCTURAL_CONSTRAINTS]:
- C# 14 extension blocks must be declared inside a top-level, non-generic, `static class`; placing an `extension(T)` block at top-level or inside a generic class produces CS9283. Multiple `extension(SameReceiver)` blocks in the same static class are permitted and additive — the compiler merges their members. Multiple static classes each containing an `extension(SameReceiver)` block are also additive across the compilation. This is the mechanism for splitting test-only probing methods into a test-assembly extension block without modifying the production static class.
- The `extension(T)` block itself does not accept an access modifier (CS0106). Visibility is controlled per-member inside the block: `public`, `internal`, and `private` all compile and carry their normal access semantics. A `private` member in an extension block is inaccessible from outside the static class containing the block — equivalent to a private static method on that class.
- Extension blocks cannot access `private` members of the receiver type (CS0122). The dispatch entrypoint written in an extension block must route entirely through the receiver's public API — the generated `Switch`/`Map` methods, public case properties, and explicit interface members. This is not a limitation for union dispatch because the generated `Switch` surface is public by definition.
- Extension blocks support property syntax: `public bool IsA => receiver is TypeA;` compiles without restriction. Properties defined this way are non-virtual and compile to static methods with a receiver parameter, identical to extension methods in earlier C# versions. This makes extension blocks suitable for attaching computed predicate properties to a union type without touching the generated partial.
- `[OverloadResolutionPriority]` applies to members inside extension blocks with the same semantics as on regular methods. A `ReadOnlySpan<T>`-based extension block member can be given priority over a `string`-based member to make the span path win at call sites that pass a literal, without the caller needing to be aware of the distinction.

[FROZENDICTIONARY_PRECISE_SURFACE]:
- `FrozenDictionary<TKey, TValue>.GetValueRefOrNullRef(TKey)` returns `ref readonly TValue`, not `ref TValue`. The caller checks absence with `Unsafe.IsNullRef(in ref)` (noting the `in` modifier since the ref is readonly) and reads the value directly through the reference without boxing. The dispatch idiom: `ref readonly var slot = ref table.GetValueRefOrNullRef(key); if (Unsafe.IsNullRef(in slot)) return Fin<Result>.Fail(...); return slot(request);` achieves zero-allocation lookup and invocation for `Func<Cmd, Fin<Result>>` values.
- `FrozenDictionary<TKey, TValue>` carries `[CollectionBuilder(typeof(FrozenDictionary), "Create")]`. This means a collection expression `FrozenDictionary<string, Handler> t = [(k1, h1), (k2, h2)];` lowers directly to `FrozenDictionary.Create<string, Handler>(span)` — no `new Dictionary<>()` intermediate, no `.ToFrozenDictionary()` call. The static field initializer `static readonly FrozenDictionary<string, Handler> _t = [(k1, h1), (k2, h2)];` is the canonical construction form.
- `FrozenDictionary<TKey, TValue>.AlternateLookup<TAlternateKey>` is a `readonly struct` with `ContainsKey(TAlternateKey)`, `TryGetValue(TAlternateKey, out TValue)`, and an indexer. The alternate key constraint is `where TAlternateKey : notnull, allows ref struct`, enabling `ReadOnlySpan<char>` as the key type when the base comparer implements `IAlternateEqualityComparer<ReadOnlySpan<char>, string>`. The BCL `StringComparer.OrdinalIgnoreCase` satisfies this. Storing the `AlternateLookup` struct in a `static readonly` field avoids repeated allocation of the lookup delegate captured inside the struct.
- `FrozenDictionary.Create<TKey, TValue>(IEqualityComparer<TKey>?, params ReadOnlySpan<KeyValuePair<TKey, TValue>>)` is the comparer-accepting overload. Both overloads are `params ReadOnlySpan<>` — verified from source. Passing the comparer at construction is the only way to configure the alternate lookup; there is no post-construction comparer replacement.

[SMARTENUM_KEY_AND_SPAN_FACTORY]:
- The generated `[SmartEnum<TKey>]` key property is `.Key`, not `.Value` or `.ToValue()`. `ToValue()` is an explicit interface implementation of `IConvertible<TKey>` delegating to `.Key`. Direct use of `.Key` is the correct form at call sites and in table initialization; `((IConvertible<string>)item).ToValue()` works but is unnecessary ceremony.
- On `NET9_OR_GREATER`, the generator unconditionally adds `IConvertible<ReadOnlySpan<char>>` and `IObjectFactory<T, ReadOnlySpan<char>, TValidationError>` to every `[SmartEnum<string>]` type. This means every string-keyed smart enum automatically supports span-based construction via `SmartEnum.Validate(ReadOnlySpan<char>, IFormatProvider?, out T?)` without any manual attribute. The full zero-alloc dispatch table pipeline: `Kind.Items.ToFrozenDictionary(k => k.Key, StringComparer.OrdinalIgnoreCase).GetAlternateLookup<ReadOnlySpan<char>>()` is verified working — span key lookup routes to the case's delegate behavior without allocation.
- `ObjectFactoryAttribute<T>` is declared `where T : allows ref struct`, enabling `[ObjectFactory<ReadOnlySpan<char>>]` on a `[ValueObject<string>]` to generate a span-accepting `Validate(ReadOnlySpan<char>, IFormatProvider?, out T?)` interface contract. Implementing this `public static` method directly on the partial class provides a zero-alloc admission path from incoming wire buffers without converting to a heap-allocated string first. This is the correct form for the admission boundary when the wire protocol gives a span.

[UNION_COARSE_DISPATCH_OVERLOAD]:
- `[UnionSwitchMapOverload(StopAt = [typeof(AbstractCase)])]` is an `AllowMultiple = true` attribute that generates an **additional** `Switch`/`Map` overload alongside the default full-coverage overload. The generated overload treats `AbstractCase` and all its siblings as atomic parameters — it does not descend into `AbstractCase`'s subtypes. A nested `[Union]` hierarchy with `[UnionSwitchMapOverload(StopAt = [typeof(Failure)])]` produces: (1) the full overload `Switch(ok: ..., failureNotFound: ..., failureUnauthorized: ...)` and (2) the coarse overload `Switch(ok: ..., failure: ...)` that delivers the `Failure` abstract base without decomposing it. Verified compiling and running.
- This coarse overload is architecturally distinct from `DefaultWithPartialOverloads`. A partial overload omits some leaf cases and has a fallback arm for unhandled cases. The `[UnionSwitchMapOverload]` coarse overload is **exhaustive at the stop boundary** — it has no fallback arm and covers all cases at the stop level. It is the correct form when some callers legitimately need the full decomposition and others only need the grouped behavior, and both caller populations must be exhaustively covered.
- Multiple `[UnionSwitchMapOverload]` attributes on one union generate multiple additional overloads, each with distinct stop-at boundaries. A union with three levels of nesting can expose three dispatch granularities — leaf-level, mid-level, and top-level — all simultaneously, without writing any dispatch code by hand. The full-coverage overload from `SwitchMapMethodsGeneration.Default` is always the zeroth generated form; each attribute appends one more.

[PRIVATE_CONSTRUCTOR_SEALING_MECHANICS]:
- `ConstructorAccessModifier = UnionConstructorAccessModifier.Private` on `[Union]` seals the constructor AND makes all generated implicit conversion operators (`operator implicit Cmd(Cmd.SetValue)`) private simultaneously. Without explicit factory methods or with `FactoryMethodGeneration = FactoryMethodGeneration.None`, case construction becomes impossible from outside the declaring class. The only surviving public construction path is a `public static` factory method declared in the `partial class` body that calls `new CaseName(...)` directly — which is still accessible because the factory method is inside the partial class, not outside it.
- `ConversionFromValue = ConversionOperatorsGeneration.None` removes the implicit conversion operators entirely, not just their accessibility. Pattern matching (`is CaseName c`) still works because pattern matching uses type testing, not conversion operators. `Switch` still works because `Switch` dispatches through the generated discriminator field, not through conversion. Setting `None` is the correct choice when the union must prevent accidental implicit upcasting in expression contexts while preserving the full dispatch surface.
- `ConversionToValue = ConversionOperatorsGeneration.Explicit` (the default) generates explicit downcast operators from the union to each case type. Applying the explicit cast to the wrong case type throws `InvalidCastException` at runtime. When case types share no base beyond the union, the cast succeeds only when the stored case matches — pattern matching with `is` is always safer than the explicit cast at call sites where the case is not statically known.

[SCHEDULE_RETRY_RESOURCE_SCOPE_PRECISION]:
- `retry(Schedule, K<IO,A>)` releases resources registered via `use<A>` inside the computation on each failed attempt before the next retry fires. The LanguageExt docs state: "Any resources acquired within a retrying IO computation will automatically be released if the operation fails." This means a `bracketIO`-wrapped computation inside a `retry` loop acquires and releases cleanly per attempt — retry does not accumulate open resources across failed iterations. The interaction is: `retry` catches the failure before the `bracketIO` scope exits, triggers cleanup, then re-enters the computation. An `IO.use<A>` registration inside the computation is scoped to the computation's lifetime, which ends (with cleanup) on failure before retry re-runs.
- `bracketIO<M,A,B,C>(Acq, Use, Catch, Fin)` (four-arg) adds an error-path handler `Func<Error, IO<C>> Catch` that runs **before** `Fin` on failure, but does not replace `Fin`. The `Fin` cleanup runs in all paths: success, failure with catch, and failure without catch match. The `Catch` arm is for transforming or logging the error before cleanup; it cannot suppress the cleanup. The four-arg overload's `Acq` and `Use` type parameter `C` is the use-phase result type; `Fin` returns `IO<B>` (a separate type) whose result is discarded. This allows the `Fin` arm to return a different type than the main computation without a cast.
- `retryWhile(Schedule, K<IO,A>, Func<Error, bool>)` and `retryUntil(Schedule, K<IO,A>, Func<Error, bool>)` are the four-argument forms that combine schedule-based delay with a predicate guard. Without the schedule argument, `retryWhile(computation, predicate)` fires immediately on each failure. The schedule argument applies the delay policy independently of the predicate — the predicate gates whether to retry at all, and the schedule gates how long to wait before the next attempt. Both predicate and schedule must permit a retry for it to proceed. `repeatWhile(Schedule, K<IO,A>, Func<A, bool>)` is the success-value counterpart: it inspects the successful result and re-runs the computation while the predicate holds, applying the schedule between re-runs.
