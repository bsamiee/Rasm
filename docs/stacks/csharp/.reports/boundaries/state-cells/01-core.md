# State-Cell Boundary Law — Core Research

[CANCELLATION_SCOPE]:
- One root `CancellationTokenSource` per process; every scoped operation derives a child via `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` (overload added in .NET 9, confirmed in .NET 10). The root owns the `IDisposable` lifetime; derived sources are disposed at scope exit, not held past drain.
- `CancellationTokenSource.TryReset()` returns `false` when the source has already been cancelled and `true` otherwise, restoring the uncancelled state. A linked source triggered by its parent also returns `false`. A linked source not yet cancelled resets successfully, but the parent link is not severed — parent cancellation still propagates after reset.
- Cancelled `CancellationTokenSource` instances cannot be reused; `TryReset` is not a substitute for a new instance once cancellation fires. Token-gated singleton cells that require re-open semantics must replace the entire `(CancellationTokenSource, CancellationToken)` pair on each open transition, not attempt to reset a spent source.
- `CancellationTokenRegistration` is both `IDisposable` and `IAsyncDisposable`; it exposes `.Unregister()` and `.Token` for symmetric attach/detach. Registrations held past cancellation waste memory on already-fired callbacks; detach in the finally or Dispose path of the owning cell, not as a fire-and-forget side effect.
- `CancellationToken.UnsafeRegister(Action<object?>, object?)` avoids the execution-context capture of the safe overload; use only where the callback owns no ambient state that could create a security boundary leak.

[ENVIO_SCOPE_CELLS]:
- `EnvIO` is the runtime environment record threading through every `IO<A>` execution: public fields are `Resources Resources`, `CancellationToken Token`, `CancellationTokenSource Source`, and `SynchronizationContext? SyncContext`. The private `Own` int governs whether `Dispose` tears down the `CancellationTokenSource` (bit 1) and the `Resources` scope (bit 2). One `EnvIO` per execution spine; concurrent operations fork child envs, not second spines.
- `EnvIO.LocalCancel` creates a child `EnvIO` whose `Source` is a new `CancellationTokenSource` linked to the parent `Source`. Cancelling the child does not cancel the parent; cancelling the parent immediately cancels all children. This is the correct scope for bounded operations that must cancel independently without propagating upward.
- `EnvIO.LocalResources` creates a child `EnvIO` with a new `Resources` scope but the same `Token` and `Source`. Resources acquired inside the child are tracked separately from the parent scope. When the child `EnvIO` is disposed, only child-scoped resources are released; parent resources are unaffected.
- `EnvIO.LocalSyncContext` replaces the `SynchronizationContext` without creating a new cancellation scope — the correct form for marshaling into a specific thread's execution context while keeping the existing cancellation chain.
- `EnvIO.LocalCancelWithTimeout(TimeSpan)` and `LocalWithTimeout(TimeSpan)` create scoped envs with an additional timeout-based cancellation deadline layered onto the existing token. If the parent fires first, the child sees it immediately; if the timeout fires first, only the child is cancelled.
- `EnvIO.New` is a static factory with signature `New(Resources?, CancellationToken, CancellationTokenSource?, SynchronizationContext?, TimeSpan?)`. When `source` is null, a new `CancellationTokenSource` is created and Own bit 1 is set; when `resources` is null, a new `Resources` scope is created and Own bit 2 is set. Creating a second root `EnvIO` from the same `CancellationTokenSource` creates two owners that can both cancel via the same token but dispose independently — only one should own the `CancellationTokenSource`.

[ATOM_STATE_CELLS]:
- `Atom<A>` holds a `private volatile object value` field under a `Func<A, bool> validator`. The validator runs once at construction — `internal static Option<Atom<A>> New(A value, Func<A, bool> validator)` returns `Option<Atom<A>>.None` when the initial value fails — and then on every `Swap`. A failing validator leaves the atom at its current value; `SwapMaybe` returning the unchanged current value makes the rejection observable at the call site.
- `Atom<Env, A>` carries metadata `Env` threaded into every `Swap(Func<Env, A, A>)` call alongside the current value. The `Env` is captured at construction and never changes, making it suitable for session-scoped configuration or stable identity keys that must accompany every update without being stored in the value itself.
- `Prelude.atomIO(A)` and `Prelude.atomIO(A, Func<A,bool>)` return `IO<Atom<A>>` — atom creation lifted into the effect pipeline. The validator form returns `IO<Option<Atom<A>>>`. This enables atoms to be created as part of an `IO` composition chain where acquisition order matters for downstream initialization.
- `AtomHashMap<K, V>` is the thread-safe mutable concurrent map. `SwapKey(K, Func<V, V>)` and `SwapKey(K, Func<Option<V>, Option<V>>)` atomically update a single key. `FindOrAdd(K, Func<V>)` atomically inserts only if absent. The `Change` event fires a `HashMapPatch<K,V>` on every structural mutation, carrying `From`, `To`, and `Changes` snapshots for downstream reactive consumption.
- `Atom<A>.Swap` is CAS-looping: the function `f` may execute more than once. Any side effect inside the swap function is repeated on contention — the swap body must be a pure transform. For operations that must fire exactly once, capture the side effect in the return value and execute it after the successful swap outside the loop.
- `Prelude.swap(Atom<A>, Func<A, A>)` and `Prelude.swapIO(Atom<A>, Func<A, A>)` are top-level alternatives to the instance `Swap`; `swapIO` lifts the mutation into an `IO<A>` for pipeline composition without losing the result snapshot.

[RESOURCES_AND_USE]:
- `Resources` tracks acquired values in an internal `AtomHashMap<object, TrackedResource>`. `Acquire<A>(A value) where A : IDisposable` registers the handle; `Acquire<A>(A value, Func<A, IO<Unit>> release)` registers with a custom `IO<Unit>` release effect. The custom-release form is the correct surface for handles whose teardown is effectful (network flush, protocol goodbye, event unsubscription) rather than a bare `.Dispose()`.
- `Resources.DisposeU(EnvIO)` iterates the `AtomHashMap` via `foreach` and calls each `TrackedResource.Release().Run(envIO2)` where `envIO2` is a fresh `EnvIO` constructed with `CancellationToken.None`. The HAMT has no insertion-order guarantee; release order is hash-tree traversal order. Designs that require ordered teardown must encode the sequence explicitly — a `Seq<IO<Unit>>` tracked alongside the map, or nested `Resources` parent chains where child-scope disposal precedes parent-scope disposal by construction.
- `use(IO<A> acquire, Action<A> release)` and `use(IO<A> acquire, Func<A, IO<Unit>> release)` build `IOUse<A, B>` DSL nodes that register acquisition into the current `EnvIO.Resources` scope on execution. The corresponding `IO<A>.Local()` extension creates a `IOLocal<A, B>` node that transforms `EnvIO` by calling `env.LocalResources` before the inner operation, ensuring resources acquired inside the local scope are released when that scope closes rather than when the root env closes.
- `use<A>(IO<A> acquire)` where `A : IAsyncDisposable` produces `IOUseAsyncDisposable<A, B>`; the `IDisposable` form produces `IOUseDisposable<A, B>`. Both are DSL record types in `LanguageExt.DSL` resolved at `Invoke(EnvIO)` time.
- `IO<A>.BracketFail()` runs cleanup only on failure, leaving the resource alive on success — correct for correlated resources that must outlive the bracket's scope. `IO<A>.Bracket(Use, Catch, Fin)` fires `Fin` unconditionally, which is the standard safe-acquisition shape.
- `IO<A>.Uninterruptible()` wraps the computation in `IOUninterruptible<A, B>` DSL. At interpretation, `Invoke` constructs a child `EnvIO` via `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — the null source causes a new `CancellationTokenSource` to be created, and because `CancellationToken.None.CanBeCanceled == false`, no parent→child registration is created. Parent cancellation does not propagate into the uninterruptible body; the isolation is structural, not a guard check.

[TOKEN_GATED_INITIALIZATION]:
- The canonical gate for once-only native initialization is `Interlocked.CompareExchange(ref int guard, 1, 0) == 0`. The CAS returns `true` exactly for the winner; all other callers see `false` and must wait or fail. The `Atom<Option<Guid>>` form in LanguageExt maps to the same CAS: `SwapMaybe(curr => curr.IsNone ? Some(id) : None)` succeeds once; the `None` return on the already-open branch leaves the atom unchanged and signals the caller that it lost the race.
- The correct one-way gate validator is `_ => true` (no validator); the gate logic lives in the swap function: `SwapMaybe(curr => curr.IsNone ? Some(acquire()) : None)`. A validator that rejects `Some(handle)` after the first open freezes the atom permanently at `Some(handle)` because no new value can pass — the validator is the wrong surface for transition constraints.
- Re-entrant initialization — where the initializer itself calls back into the guarding layer — deadlocks on any single-permit gate (`SemaphoreSlim(1,1)`, `Lock.EnterScope`, or an `Atom` with a blocking reader). Non-re-entrant native init must hold the gate cell through the full side-effect sequence; completion must be signaled by a second cell transition, not by releasing the gate early.
- `System.Threading.Lock.EnterScope()` returns a ref-struct `Lock.Scope` whose `Dispose` releases the lock; `using var _ = theLock.EnterScope()` is the replacement for `lock(obj)` in C# 14 code targeting .NET 9+. `Lock` is re-entrant from the same thread (`IsHeldByCurrentThread`), unlike `Monitor`. Use `SemaphoreSlim(1,1)` for non-re-entrant protection when strict single-entry is required.
- `LazyInitializer.EnsureInitialized<T>(ref T target, ref bool initialized, ref object syncLock, Func<T> factory)` — the three-field form — is the pattern for `[ThreadStatic]`-backed per-thread caches that require a sync lock for the first initialization per thread. The single-field form `EnsureInitialized<T>(ref T target, Func<T> factory)` uses a CAS and is appropriate for process-static initialization with a reference-type target.

[RUNTIME_SPINE]:
- One process-static runtime spine record is the ownership unit for `(CancellationTokenSource, TimeProvider, channel ports, configuration)`. A re-boot creates a new record; the old record must reach a fully drained and disposed state before the new one is constructed. Concurrent second spines sharing any mutable cell (a static `Atom`, a process-static `Resources`, or a shared `Channel`) produce undefined merge semantics.
- `TimeProvider` is the clock surface injected into the spine record. `TimeProvider.System` is the production singleton; test overrides implement `TimeProvider` directly. `TimeProvider.CreateTimer(TimerCallback, object, TimeSpan, TimeSpan)` returns `ITimer`; `ITimer.Change(TimeSpan, TimeSpan)` reschedules without allocation. The runtime clock must enter state cells as `TimeProvider`, never as `DateTime.UtcNow` or `Stopwatch` calls, preserving deterministic replay in tests.
- `Channel<T>` is the seam marker for the drain phase of a runtime spine. `ChannelWriter<T>.Complete()` fences new writes (subsequent `TryWrite` returns `false`); in-flight items remain readable until `ChannelReader<T>.Completion` task completes. `ChannelReader<T>.WaitToReadAsync(CancellationToken)` returns `false` after the channel is drained — this is the structural drain-wait predicate. A spine teardown that calls `Complete()` then `await Completion` guarantees all work was processed before the rest of the spine is released.
- Mid-flight semantics during drain require explicit policy: items already dequeued but not yet committed are either completed with their result or explicitly faulted. The `Channel` drain model does not provide this; the cell owning mid-flight correlation (a `ConcurrentDictionary<CorrelationId, TaskCompletionSource<T>>`) must be separately swept on drain completion, completing outstanding sources with an `OperationCanceledException` or a domain-typed fault.

[MEMO_AND_CACHE_CELLS]:
- `Memo<A>` uses three states encoded in an `internal int state` field: `0` (unloaded), `1` (loading), `2` (loaded). The winner of `CAS(state: 0→1)` runs `acquire()`, writes `value = result`, then writes `state = 2`. A failed `acquire()` reverts to `state = 0` in the catch block; the exception is rethrown, not absorbed. These two post-CAS writes are non-atomic; any concurrent caller in state `1` spins via `SpinWait.SpinOnce()`.
- `Memo<A>.Reset()` CASes `state: 2→1`, then writes `value = default(A)`, then writes `state = 0`. The `state` field carries no `volatile` annotation; the `Interlocked.CompareExchange` calls provide memory-barrier coherence for state transitions, but the hot-path `if (state != 2)` guard in the `Value` getter is a plain `int` read. On weakly ordered hardware (ARM64), a thread could observe `state == 2` before the write to `value` from the winning CAS thread becomes visible, reading a stale default. The safe form is `Volatile.Read(ref state) != 2` in the getter, or replacement with `Atom<Option<A>>` which carries the full volatile guarantee on both state and value simultaneously.
- `Reset()` is not safe for concurrent invocation alongside `GetValue()`: if state is `1` (loading), `Reset()` observes `CAS(state, 1, 2)` return `1` (compare failed) and spins. For boundary cells that need periodic re-acquisition under high contention, `Memo<A>.Clone()` — which creates a fresh `Memo` from the same `acquire` delegate without cached state — combined with replacement via an outer `Atom<Memo<A>>` is safer than calling `Reset()` under load.
- Cache key completeness: a memo key must encode every dimension that affects the output — for content-dependent results this means content hash plus full configuration hash. A key missing any input dimension produces stale hits. For native library fingerprinting, the key must include file path, file size, and modification time; library identity is not portable across file replacement even at the same path.
- `ConcurrentDictionary<K,V>.AddOrUpdate(K, Func<K, TArg, V>, Func<K, V, TArg, V>, TArg)` — the `TArg`-carrying factory overload — avoids closure allocation for registration patterns that thread a single argument. Both factory delegates may be called multiple times on contention and must be pure. For factories that acquire resources (native init, file handles), wrap with a `Lazy<V>` value in the dictionary and use `LazyThreadSafetyMode.ExecutionAndPublication` to guarantee single execution.
- Generated registry keys must be stable across regeneration cycles. A key derived from a `nameof` expression, a source-generated type identity, or a versioned discriminant survives recompilation; a key derived from `RuntimeHelpers.GetHashCode`, unordered iteration index, or assembly load order does not. Stale-key entries in a persistent registry accumulate as ghost entries when the type graph changes; `AtomHashMap<K,V>.Filter(pred)` is the correct sweep operation.

[LIFECYCLE_STATE_MACHINES]:
- A boundary layer's lifecycle is a state machine with at least: `Created`, `Opening`, `Open`, `Draining`, `Closed`, and `Faulted`. `Draining` fences new work (equivalent to `ChannelWriter.Complete()`) but keeps mid-flight work alive. A cell in `Draining` must reject new acquisitions with a typed rejection carrying boundary identity, correlation id of the rejected request, and timestamp. `Faulted` is escapable if the fault is transient; the escape transition creates a new `Open` instance, not a reset of the faulted one.
- Fault re-entry: when a `Faulted` boundary cell is replaced by a new one, correlation cells tracking in-progress requests must explicitly decide replace-versus-reuse. Replace produces a fresh correlation with no memory of prior state; reuse preserves request identity but inherits any partial server-side work. Both are valid; the decision must be explicit policy in the cell, not an implicit consequence of allocation order.
- `ObjectDisposedException.ThrowIf(bool condition, object instance)` is the idiomatic guard for post-`Close` access. The `condition` operand should be `!(_state is LifecycleState.Open)` or the equivalent typed check on the state cell. `ObjectDisposedException.ThrowIf(bool, Type)` accepts a `Type` directly — use this inside static methods or factories where `this` is unavailable: `ObjectDisposedException.ThrowIf(_closed, typeof(BoundaryCell))`. The `typeof(T)` form is a compile-time constant and avoids the virtual dispatch on `GetType()` that the `object`-accepting overload implies.
- State transition rejection must carry typed evidence: the rejecting state (e.g., `Draining`), the correlation id of the caller's operation, and a boundary-layer timestamp from the injected `TimeProvider`. A rejection typed as `Errors.Cancelled` loses the boundary context; a domain-typed `BoundaryDrainingFault` carrying all three fields composes cleanly into `Fin<A>` failure rails.
- Non-re-entrant process-static initialization where the initializer has side effects must hold state in a cell that distinguishes `Initializing` from `Initialized` from `InitializationFailed`. A cell that collapses these into a boolean admits the window where a second caller observes `Initialized = true` while the initializer's side effects are still running. The three-state distinction enforces that `Initializing` blocks all callers until the state transitions to `Initialized` or `InitializationFailed`.

[ENVIO_STRUCTURAL_NOTES]:
- `EnvIO.DisposeNonResources()` releases the `CancellationTokenSource` if `Own bit 1` is set and disposes any `Registration`, without touching the `Resources` scope. `EnvIO.DisposeResources()` runs `Resources.DisposeU()` only when `Own bit 2` is set, without touching the `CancellationTokenSource`. Both methods use `Interlocked.CompareExchange` to ensure idempotency. Calling `DisposeNonResources()` on an `EnvIO` that does not own its source (bit 1 clear) is a no-op for the source but still disposes the `Registration`, which is correct.
- `IO<A>.Fork(Option<TimeSpan> timeout)` returns `IO<ForkIO<A>>`. `ForkIO<A>` carries `Cancel: IO<Unit>` and `Await: IO<A>` as a destructurable pair. The forked computation runs in a child `EnvIO` whose `Source` is a new `CancellationTokenSource` linked to the calling env's token via a registration. `timeout` wraps the child source in a timeout-scoped source. Awaiting after cancellation returns the error that the cancelled computation produced, not `OperationCanceledException` directly — handle via `IO<A>.IfFail`.
- `IO<A>.Post()` routes execution through `SynchronizationContext.Post` using the `SyncContext` field of the current `EnvIO`; it allocates an `AutoResetEvent` per call — a kernel-backed synchronization primitive — which is non-trivial on hot paths. `IO<A>.Local()` creates `IOLocal<A, B>` that calls `env.LocalResources` at interpretation time — scoping resources but not cancellation. `Post`, `Local`, and `LocalCancel` address disjoint concerns: thread-dispatch, resource-scope, and cancellation-scope respectively.
