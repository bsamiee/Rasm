# [RASM_API_BCL_CRYPTOGRAPHY]

`System.Security.Cryptography` (+`.X509Certificates`) BCL inbox — no package pin, shared-framework surface (`System.Security.Cryptography.dll`, net10.0). The RFC-7468 PEM armor, X.509 round-trip, and zeroization primitives bind the AppHost secret lease and credential wire plus the Persistence KMS-unwrapped data-encryption key custody. Snapshot and rotation identity use the non-cryptographic `ContentHash.Of` digest, so only `ZeroMemory` and PEM/X.509 armor cross into live consumers here.

## [01]-[PEM_ENCODING]

`PemEncoding` is a static class owning RFC-7468 textual-encoding armor with a full write family, a full locate family, and a size estimator. Both directions carry a `char`/`string` form and a UTF-8 `byte` mirror, and every allocating form has a caller-buffer `Try*` span form — the span forms are the zero-alloc credential path.

- `PemEncoding.WriteString(ReadOnlySpan<char> label, ReadOnlySpan<byte> data) -> string` writes one armored element (`-----BEGIN {label}-----`/`-----END {label}-----`, base-64 body wrapped at 64 chars) — the `CredentialPem.Encode` per-block armor writer.
- `PemEncoding.Write(ReadOnlySpan<char> label, ReadOnlySpan<byte> data) -> char[]` is the `char[]` allocating form for a caller that pools or mutates the armor buffer.
- `PemEncoding.WriteUtf8(ReadOnlySpan<byte> utf8Label, ReadOnlySpan<byte> data) -> byte[]` is the UTF-8 armor form — writes armor bytes directly when the wire is `byte`-framed, skipping a `string` transcode before egress.
- `PemEncoding.TryWrite(ReadOnlySpan<char> label, ReadOnlySpan<byte> data, Span<char> destination, out int charsWritten) -> bool` / `PemEncoding.TryWriteUtf8(ReadOnlySpan<byte> utf8Label, ReadOnlySpan<byte> data, Span<byte> destination, out int bytesWritten) -> bool` — the caller-buffer forms; size `destination` through `GetEncodedSize`, `false` signals an undersized buffer without a partial write.
- `PemEncoding.TryFind(ReadOnlySpan<char> pemData, out PemFields fields) -> bool` walks the next armored element and returns `false` at end-of-input — the `CredentialPem.Decode` multi-element loop condition; `PemEncoding.TryFindUtf8(ReadOnlySpan<byte> pemData, out PemFields fields) -> bool` is the UTF-8 mirror over a `byte`-framed bundle.
- `PemEncoding.Find(ReadOnlySpan<char> pemData) -> PemFields` / `PemEncoding.FindUtf8(ReadOnlySpan<byte> pemData) -> PemFields` are the throwing forms (`ArgumentException` on no armor) — reserved for a boundary that already proved armor presence; domain decode stays on the `Try*` rail so an absent element folds to `PemFault.EmptyBundle`, never an exception.
- `PemEncoding.GetEncodedSize(int labelLength, int dataLength) -> int` sizes a `TryWrite`/`TryWriteUtf8` destination before a span write.
- `PemFields` (readonly struct) carries `Range Location`, `Range Label`, `Range Base64Data`, and `int DecodedDataLength` — the decode fold sizes the DER through `DecodedDataLength`, fills through `Convert.TryFromBase64Chars(pemData[fields.Base64Data], destination, out _)`, reads the label through `pemData[fields.Label]`, and advances the walk past `fields.Location.End`. The ranges index the ORIGINAL span, so the walk re-slices from `Location.End..` without recomputing offsets.

## [02]-[X509_ROUNDTRIP]

`X509Certificate2.CreateFromPem`/`ExportCertificatePem` own the certificate PEM round-trip and are the NON-obsolete admission — the raw `byte[]`/path constructors carry `SYSLIB0057` and are the deleted forms; a certificate enters through `CreateFromPem` and leaves through `ExportCertificatePem`/`RawData`.

- `X509Certificate2.RawData -> byte[]` is the certificate DER the `CredentialBundle.Cert` factory armors; `X509Certificate2.RawDataMemory -> ReadOnlyMemory<byte>` is the alloc-free mirror the `PemBlock.Der` `ReadOnlyMemory<byte>` slot binds without a defensive copy.
- `X509Certificate2.CreateFromPem(ReadOnlySpan<char> certPem) -> X509Certificate2` parses a `CERTIFICATE` block, proving armored bytes decode to a real certificate before admission; `CreateFromPem(ReadOnlySpan<char> certPem, ReadOnlySpan<char> keyPem) -> X509Certificate2` pairs a private key from a second armored block; `CreateFromPemFile(string certPemFilePath, string? keyPemFilePath = null) -> X509Certificate2` reads both from disk when the credential source is a mounted PEM file.
- `X509Certificate2.ExportCertificatePem() -> string` is the reverse projection (a `CERTIFICATE` armored string); `X509Certificate2.TryExportCertificatePem(Span<char> destination, out int charsWritten) -> bool` is the caller-buffer form for a pooled export path, `false` on an undersized destination.

## [03]-[ZEROIZE]

`CryptographicOperations` is the static integrity-primitive owner. Only `ZeroMemory` binds into live consumers; the constant-time compare and the one-shot hash/HMAC family are catalogued as the full surface but RECORDED NON-USE for this suite's identity path.

- `CryptographicOperations.ZeroMemory(Span<byte> buffer) -> void` overwrites a rented credential buffer so no managed copy survives collection — the one zeroization path the `SecretLease` renew, the drain-forced `Zeroize` terminal, and the KMS-unwrapped DEK share.
- `CryptographicOperations.FixedTimeEquals(ReadOnlySpan<byte> left, ReadOnlySpan<byte> right) -> bool` is the constant-time compare — RECORDED NON-USE: the rotation-diff identity is the kernel non-cryptographic `ContentHash.Of` digest, so no constant-time compare is bound over a non-crypto hash (a `FixedTimeEquals` over the digest is the deleted form). It is the correct primitive for any future MAC-tag or secret-token equality that carries a real security claim.
- `CryptographicOperations.HashData(HashAlgorithmName,...)` / `HashDataAsync(...)` / `TryHashData(...)` and the `HmacData(...)` / `HmacDataAsync(...)` / `TryHmacData(...)` family are the allocation-controlled one-shot cryptographic digest surface (byte-array, `Stream`, and `ReadOnlySpan<byte>` sources; `Span<byte>` destinations; `ValueTask` async mirrors) — RECORDED NON-USE: identity and cache keys route to the kernel `ContentHash.Of` (`api-hashing` `XxHash128`), never a cryptographic digest, so no member of this family is bound for identity. This is the owner a future cryptographic-integrity claim (signed manifest, attested payload) composes, distinct from the non-crypto identity rail.

## [04]-[STACK]

The AppHost credential lifecycle composes `[01]`/`[02]` for the RFC-7468 bundle wire (`CredentialPem.Encode`/`Decode`/`Carrier`, `CredentialBundle.Cert(X509Certificate2)`) and `[03]` for zeroize — the `SecretLease` holds the live rented `byte[]` and owns the in-memory lifecycle, while `CredentialPem` owns the at-rest/on-wire armor, so material never carries two encodings. `PemBlock.Der` binds `X509Certificate2.RawDataMemory` alloc-free; the `Decode` fold reads `PemFields.DecodedDataLength`/`Base64Data` on the `TryFind` rail so an absent element folds to `PemFault.EmptyBundle` on `Fin`, never an exception, and an unknown armor label folds to `PemFault.LabelUnknown` against the closed `PemLabel` `[SmartEnum<string>]` vocabulary. Secret material carries `DataClassification.Secret` so `api-redaction` erases it at every egress; the redacted `CredentialPemWire` crosses the label set, per-block `ContentHash.Of` digests, and the `Redactor`-redacted key-id — never a private-key byte. The KMS-unwrapped data-encryption key zeroizes through the same `ZeroMemory` path the lease owns. The armored bundle text crosses to TS through `jose` `importPKCS8`/`importSPKI`/`importX509` and to Python through `cryptography` `load_pem_*`, both parsing the identical BCL-written armor; the RFC-7468 `-----END-----`/`-----BEGIN-----` boundary joined by `\n` IS the self-delimiting separator, so no `--SEP--` token is minted. The non-crypto identity boundary is load-bearing: `api-hashing` owns snapshot/rotation identity, this catalog owns armor + zeroization, and the crypto hash/compare family stays RECORDED NON-USE until a real security claim admits it.
