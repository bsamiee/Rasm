# Map-Fold Spine and the Zeros Policy Algebra

[ONE_FLAG_DERIVES_THREE_DECISIONS]:
- A single two-state `Zeros` argument on every elementwise transform derives three independent structural decisions, so the operation never carries three parallel knobs. `Zeros.AllowSkip` permits structural zeros to go unvisited; `Zeros.Include` forces every cell. At the public arrow this one value selects: the result-storage mutability via `Build.SameAs(this, rows, cols, zeros == Zeros.Include)`, the target-clearing convention via `existingData = zeros == Zeros.Include ? ExistingData.AssumeZeros : ExistingData.Clear`, and the traversal completeness inside the storage kernel. One declared correspondence — does this kernel preserve zeros — fans out to mutability, clearing, and visitation; the three are never set independently.
- The mutability decision is load-bearing because a structure-destroying map applied to triangular or diagonal storage must spill into off-structure cells. `SameAs` with `fullyMutable: true` promotes a diagonal example to sparse (or dense) storage that can hold the new fill; with `fullyMutable: false` it returns same-structured storage that would silently drop every off-structure result. A map that adds a scalar to a diagonal matrix without `Include` therefore loses the fill it just computed — the flag is the only thing standing between a correct full result and a structurally truncated one.
- The clearing decision matters only against a reused target buffer. `ExistingData.Clear` zeroes the target before a sparse-skipping map so stale entries at skipped positions do not survive; `ExistingData.AssumeZeros` skips the clear because the target is freshly allocated or about to be fully overwritten. A hot loop that reuses a result buffer across a skipping map must route `Clear`, or prior-iteration values leak through at the skipped structural-zero positions.

[KERNEL_OVERRIDES_THE_HINT_BY_PROBING_F_AT_ZERO]:
- `Zeros` is a hint the storage layer is free to override by interrogating the function at the additive identity. The sparse map kernel computes `forceAll = zeros == Zeros.Include || !Zero.Equals(f(Zero))` — even when the caller asked to skip, a map whose value at zero is non-zero forces full traversal, because skipping would leave the structural zeros holding the wrong value. The caller declares intent; the kernel derives necessity. A structure-preserving map (`x => x * k`) keeps `f(0) == 0` and the skip stands; a structure-destroying map (`x => x + 1`) is detected and upgraded to full visitation regardless of the flag. The policy value is advisory at the boundary and authoritative at the kernel, and the kernel's authority is itself derived from one probe.
- This probe makes the correct elementwise contract emergent rather than caller-asserted. The operation author need not reason about whether a lambda preserves sparsity; the kernel reasons for them by evaluating once at zero. The only failure mode is a lambda that is non-zero-preserving but accidentally returns zero at the literal zero input while being non-zero on some other entry — the probe cannot see that, so an operation known to destroy structure on nonzero inputs must still pass `Include` explicitly. The probe handles the common case; the flag remains the escape for the pathological one.

[OPERATION_IS_A_LAMBDA_PLUS_A_DECLARED_ZERO_POLICY]:
- Every scalar arithmetic primitive on the dense double surface is one lambda over the elementwise transform with its zero-preservation declared as the policy argument, not enumerated as a branch. Scalar-add passes `Include` because adding to zero is nonzero; scalar-multiply passes `AllowSkip` because scaling zero stays zero; divide-by-scalar passes the policy conditionally — `AllowSkip` for a nonzero divisor, `Include` when the divisor is zero so the resulting infinities populate the structural zeros; scalar-divided-by-element passes `Include` because the reciprocal of zero is not zero. The arithmetic family is a table of `(lambda, zero-policy)` rows, and the policy column is computed from the lambda's behavior at zero, not chosen by hand per row.

```csharp
void DoAddScalar(double s, Matrix<double> r)   => Map(x => x + s, r, Zeros.Include);
void DoMulScalar(double s, Matrix<double> r)   => Map(x => x * s, r, Zeros.AllowSkip);
void DoDivByScalar(double d, Matrix<double> r)  => Map(x => x / d, r, d == 0.0 ? Zeros.Include : Zeros.AllowSkip);
void DoScalarDivBy(double n, Matrix<double> r)  => Map(x => n / x, r, Zeros.Include);
```

- The conditional divide row proves the policy is data, not control flow: the same lambda carries different zero-traversal depending on a runtime property of its own constant, and that property is read once to seat the policy rather than re-tested inside the loop body. Pushing the `d == 0.0` test into the elementwise kernel would re-evaluate it per cell; lifting it to the policy argument evaluates it once and lets the storage layer skip the entire zero block when safe.

[STORAGE_PAIR_DISPATCH_IS_PATTERN_MATCH_NOT_VIRTUAL]:
- Binary arithmetic discriminates on the runtime storage pair by pattern match, taking a provider fast path only when both operands and the target are the contiguous column-major storage, and falling to the elementwise base otherwise. `DoAdd(other, result)` tests `other.Storage is DenseColumnMajorMatrixStorage && result.Storage is DenseColumnMajorMatrixStorage` and routes to a flat array kernel; any sparse or diagonal participant falls to `base.DoAdd`, which threads the operation through the shared elementwise map. The acceleration is keyed on the storage shape of the operands, recovered by `is`-pattern, never by a mode parameter the caller supplies.
- `Map2` over two storages is the canonical zip and its kernel is itself storage-pair-specialized. The dense-dense `Fold2` walks both flat arrays in lockstep; the dense-diagonal `Fold2` synthesizes the off-diagonal zeros from the builder's zero rather than materializing them; the dense-sparse `Fold2` advances the compressed-row cursor and supplies zero for absent columns. One `Fold2` arrow over a function `(state, a, b) => state` thus expresses residual norms, elementwise comparisons, and weighted inner products without ever densifying the sparse or diagonal operand — the structural skip is preserved through the fold, not just the map.

```csharp
// r = A·x − b without forming A·x, over column-major A, one MultiplyAdd per column
static double RelativeResidual(Matrix<double> a, Vector<double> x, Vector<double> b)
{
    var acc = Vector<double>.Build.Dense(a.RowCount);
    var src = ((DenseColumnMajorMatrixStorage<double>)a.Storage).Data;
    Enumerable.Range(0, a.ColumnCount).Iter(c =>
        TensorPrimitives.MultiplyAdd(src.AsSpan(c * a.RowCount, a.RowCount),
            x[c], acc.AsArray().AsSpan(), acc.AsArray().AsSpan()));
    return acc.Subtract(b).L2Norm() / b.L2Norm();
}
```

[FOLD_DIRECTION_IS_A_TRAVERSAL_CONTRACT]:
- The row fold and the column fold are not symmetric in cache behavior under column-major storage, so the fold direction is a performance contract, not a free choice. `FoldByColumn` walks each column as a contiguous slice (`data[col*rows + j]` for ascending `j`) and is stride-one; `FoldByRow` walks each row across columns (`data[col*rows + row]` for ascending `col`) and is strided at the row count. A per-column reduction — column norms, per-feature statistics, Gram diagonal — is the cache-aligned direction; the same reduction phrased per row pays a stride penalty proportional to the row dimension. The fold that matches the storage major axis is the one to reach for, and transposing the problem to align the fold beats folding against the grain.
- Both folds thread a typed accumulator and a finalize step, so a fold is a stateful left-scan with a terminal projection rather than a bare reduction. The signature carries `f : (acc, value) => acc` and `finalize : (acc, count) => acc`, letting a mean fold accumulate a sum and divide by the count once at the end of each row or column, and letting a variance fold carry a running pair. The finalize hook removes the second pass a naive reduce-then-normalize would require — one traversal, accumulate, finalize per line.

[INPLACE_VERSUS_ALLOCATING_IS_THE_SAME_KERNEL_TWICE]:
- The allocating transform and the in-place transform are one kernel selected by buffer identity, never two code paths. `Map(f, result)` tests `this == result` and routes to `MapInplace` when source and target alias, otherwise to `MapTo` with the derived clearing policy; the allocating `Map(f)` is `MapTo` into a freshly built `SameAs` target. A pipeline that reuses one buffer across a chain of elementwise steps invokes the in-place arm at each step and never allocates past the first; the allocating arm is the convenience leak that materializes a value only where one must leave the pipeline. The choice is recovered from whether the result handle is the source handle, so aliasing the buffer is the way to request in-place, not a separate method name.
- The in-place dense map and the allocating dense map share the identical flat-array parallel loop, differing only in whether the write target is the source array or a sibling array; the indexed variants additionally reconstruct the row and column from the flat offset under column-major arithmetic (`col*rows + row`) so the index-aware lambda sees logical coordinates while the loop stays linear over the contiguous buffer. The cost of the indexed form over the unindexed form is the per-cell coordinate reconstruction, not a change in traversal order — both sweep the buffer once, in storage order, under the same partitioning.

[ELEMENTWISE_TRANSCENDENTALS_FALL_THROUGH_TO_SCALAR_MAP]:
- The provider accelerates only a narrow arithmetic core; every elementwise transcendental routes through the scalar map and receives no vectorization. Dense-dense add, subtract, scale, and pointwise multiply/divide/power reach the provider's array kernels; the pointwise exponential, logarithm, root, trigonometric, and hyperbolic families have no provider entry and resolve to a `Map` over a scalar lambda, one managed call per element. An elementwise `exp` or `sqrt` over a large dense buffer is therefore a scalar loop, and the contiguous `Values` span fed to a vectorized elementwise transcendental primitive replaces that loop with one SIMD pass — the elementwise transcendental layer is the primary vectorization gap in the algebra surface, mirroring the factorization-kernel gap but on the map side.
- The pointwise unary wrapper allocates a `SameAs` result then applies the kernel, and the pointwise binary wrapper validates shape agreement before applying, so the allocation-versus-reuse and the validation cost are uniform across the entire transcendental family. Reaching past the per-element `Map` to a span-level vectorized elementwise call must reconstruct the same `SameAs` result selection and shape guard the wrapper provides, or it skips the structural-mutability decision the policy layer makes for free.

[POLICY_VALUE_REPLACES_A_FLAG_TRIPLE_AT_THE_BOUNDARY]:
- The rejected shape is a transform taking a clear-target boolean, a force-full boolean, and a result-is-mutable boolean — three parallel knobs whose legal combinations the implementation re-derives, and whose illegal combinations (skip-traversal with a non-clearing reused target, immutable result with a structure-destroying lambda) are silently wrong. The admitted shape is the single enum whose two states the kernel expands into all three, with the probe at zero as the derived authority that makes the structure-preserving common case need no flag at all. The combination space collapses from eight boolean triples to two enum states plus one runtime probe, and the unreachable-but-expressible bad triples cease to exist.
- An operation that re-describes its zero handling with a parallel boolean beside the lambda has smuggled the collapsed knob back in. The discriminant must remain recoverable from the one policy value and the lambda's behavior at zero — never from a second flag that restates what `f(Zero)` already reveals. A custom elementwise operation built on this spine therefore exposes at most the `Zeros` enum and the function; any further structural parameter is the rejected form, because the kernel can derive every remaining decision from those two.
