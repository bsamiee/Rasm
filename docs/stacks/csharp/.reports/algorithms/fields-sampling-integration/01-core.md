# Fields, Sampling, and Integration — Core Research

[INTEGRATOR_CATALOG]:
- The `Integrate` static class exposes four orthogonal routes: `OnClosedInterval` (double-exponential, absolute-error target), `GaussLegendre` (fixed-order, with infinite-bound substitution), `GaussKronrod` (adaptive recursive with embedded error estimate and L1-norm channel), and `OnRectangle`/`OnCuboid` for tensor-product 2D/3D cubature. These are structurally distinct algorithms — not wrappers of one underlying kernel — and the route choice is the primary accuracy-versus-cost decision, not the order parameter.
- `Integrate.GaussKronrod` returns `(value, error, L1Norm)` through `out` parameters on its four-parameter overload. The `L1Norm` is the integral of `|f|` accumulated in the same pass; a large `|result / L1Norm|` ratio diagnoses ill-conditioning from cancellation without a second evaluation. The two-parameter overload discards both `error` and `L1Norm`, making the cancellation diagnostic invisible. The facade default `targetRelativeError = 1e-8`; `GaussKronrodRule.Integrate` defaults to `1e-10` — these are independent constructor defaults.
- `GaussKronrod`, `GaussLegendre`, and `DoubleExponential` accept `double.NegativeInfinity`/`double.PositiveInfinity` at the facade level — the infinite-domain variable substitutions live in `Integrate.*`, not in the underlying rule classes. Calling `DoubleExponentialTransformation.Integrate` directly with infinite bounds passes infinity into abscissa evaluation, producing `NaN` weights; the substitution fires only through `Integrate.DoubleExponential`.
- The adaptive recursion inside `GaussKronrodRule.recursive_adaptive_integrate` is bounded by `maximumDepth` (default 15), capping the recursion tree at `2^maximumDepth` leaves. At `maximumDepth = 15` and `order = 15`, full recursion reaches ~260,000 function evaluations. Depth exhaustion returns silently without a convergence flag; callers must inspect the `out error` parameter. Reducing to 8–10 trades accuracy for bounded evaluation count on oscillatory or near-singular integrands.
- Precomputed Gauss-Legendre nodes exist for orders {2–20, 32, 64, 96, 100, 128, 256, 512, 1024}. Any other order triggers on-the-fly computation through `GaussLegendrePoint.Generate`: Newton's method initialized with Chebyshev-cosine guesses, with both node-value and weight-value convergence checks. The node cache is `[ThreadStatic]` on a single slot per thread — a field integration loop that alternates between two non-precomputed orders recomputes on every call. The remedy is to construct named `GaussLegendreRule` instances at setup; each instance holds its own `GaussPoint` and is not subject to thread-local eviction.
- Precomputed Gauss-Kronrod nodes exist for orders {15, 21, 31, 41, 51, 61}; all others use a Stieltjes-polynomial generator. `GaussKronrodRule` takes only `order` in its constructor — bounds are supplied to the instance `Integrate` method, not the constructor.
- `DoubleExponentialTransformation.Integrate` uses `NewtonCotesTrapeziumRule.IntegrateAdaptiveTransformedOdd` with level-indexed abscissas and weights. The termination test is the absolute difference between successive level estimates; `NumberOfMaximumLevels = 10` caps refinement and is reached silently.
- `NumericalDerivative.StepSize` setter rounds the input to the nearest power of 2, producing a binary-representable step and improving finite-difference cancellation error. `StepType.Relative` derives `BaseStepSize = ε^(1/(p - α))` — the classical optimal step for centered differences at stencil width `p` and derivative order `α`. `StepType.RelativeX` computes `BaseStepSize*(1 + |x|)` per evaluation point, adapting to magnitude.
- `FiniteDifferenceCoefficients.GetCoefficients(center, order)` generates exact rational stencil weights: `0` is forward, `points-1` is backward, `points/2` is centered. `GetCoefficientsForAllOrders(center)` returns the full 2D rectangular array in one call. The construction cost is `O(p³)` for `p` points, via Vandermonde-matrix inversion, and is dominated by the constructor; subsequent queries are `O(1)` table lookups.
- `EvaluateMixedPartialDerivative` recurses depth-first, evaluating the outer partial last. For a mixed derivative of order `k` at `p` stencil points, total function evaluations are `p^k` — 125 at `k=3, p=5`. The cost grows exponentially and is not reduced by symmetry of mixed partials.

[FIELD_EVALUATION_AND_INTERPOLATION]:
- The `IInterpolation` interface defines a unified contract: `Interpolate(t)`, `Differentiate(t)`, `Differentiate2(t)`, `Integrate(t)` (indefinite), and `Integrate(a, b)` (definite). The `SupportsDifferentiation` and `SupportsIntegration` flags are explicit-interface properties evaluated at runtime — there is no compile-time prevention of calling unsupported operations.
- `CubicSpline` stores four coefficient arrays (`c0, c1, c2, c3`). The indefinite integral is computed lazily via `Lazy<double[]>` and cached — first call pays `O(N)` setup; subsequent calls are `O(1)` lookup plus `O(log N)` binary search. `LinearSpline` follows the same lazy-integral pattern. Field integrations inside tight loops should pre-warm the `Lazy<>` via a single `Integrate` call at construction rather than at first query.
- `CubicSpline.InterpolateHermiteSorted` constructs Hermite cubics from sample values and first derivatives directly: coefficients `c2, c3` are derived per interval by the local two-point Hermite system, with no global tridiagonal solve. This gives per-interval `O(1)` construction and exact derivative matching at sample points — the correct route when tangent estimates come from the field's own gradient evaluator.
- `BulirschStoerRationalInterpolation` supports neither differentiation nor integration through the `IInterpolation` contract. It is the only interpolant that handles poles; for field evaluation near rational singularities it is the correct choice, but the contract flags must be checked before composing it with operations that require those capabilities. The pole-avoidance perturbation shifts `D[i]` by `1e-25`; when the rational update denominator rounds to zero under ULP tolerance, the method returns `NaN` rather than throwing.
- `TransformedInterpolation.InterpolateSorted(transform, transformInverse, x, y)` hardcodes `LinearSpline` as the base interpolant — the factory does not accept a `CubicSpline` in the base position. Both `SupportsDifferentiation` and `SupportsIntegration` are `false`; chain-rule differentiation and change-of-variables integration are not implemented.
- The `Interpolate` static facade exposes named constructors: `CubicSpline`, `CubicSplineRobust`, `CubicSplineMonotone`, `CubicSplineWithDerivatives`, `LogLinear`, `Linear`, `Step`, `Polynomial`, `RationalWithPoles`, and `RationalWithoutPoles`. The facade has no nested types; `CubicSpline.InterpolateAkimaSorted` and `InterpolateMakimaSorted` are accessed directly on the `CubicSpline` class, not through fluent facade nesting.

[KERNEL_DENSITY_AND_SCATTER_RECONSTRUCTION]:
- `KernelDensity.Estimate(x, bandwidth, samples, kernel)` parallelizes via `CommonParallel.Aggregate` — the per-element computation `kernel((x - samples[i]) / bandwidth)` runs in parallel, and partial sums are combined with floating-point addition. Parallel reduction does not preserve associativity order, so estimates are non-reproducible across thread schedules even with a fixed seed. Serial evaluation is required for reproducibility, which forfeits the automatic parallelism. There is no spatial index — the sum over `samples` is an all-pairs `O(N)` scan per query point; `O(N·M)` for `N` samples and `M` query points.
- Four named kernels are provided: `GaussianKernel` (delegates to `Normal.PDF(0, 1, x)`), `EpanechnikovKernel` (optimal MSE, compactly supported on [-1,1]), `UniformKernel`, `TriangularKernel`. Any `IContinuousDistribution` PDF can be adapted to a kernel with a one-line lambda; no subtype is needed.
- `Fit.LinearCombination(x[], y[], Func<double,double>[] functions)` and `Fit.LinearCombination(..., DirectRegressionMethod)` build the design matrix and dispatch to `MultipleRegression`. Passing `DirectRegressionMethod.Svd` is the rank-revealing route for ill-conditioned basis matrices (overlapping compactly supported kernels, near-dependent RBFs).
- `MultipleRegression.DirectMethod(Matrix<T> x, Matrix<T> y, DirectRegressionMethod)` accepts a matrix response — `y` is a matrix — enabling simultaneous reconstruction of vector-valued fields (gradient, flux) from the same scattered input without separate solve calls. Named `NormalEquations`, `QR`, and `Svd` static methods are also available as direct dispatch alternatives.
- `WeightedRegression.Weighted(designMatrix, responseVector, weightMatrix)` implements weighted least squares with a diagonal weight matrix encoding per-sample confidence. `WeightedRegression.Local(x, y, t, radius, kernel)` computes a locally weighted fit around query point `t` within `radius`, using an injected kernel — the library-level locally-weighted regression surface that eliminates the need to manually window and solve.

[SAMPLING_AND_RANDOM_INFRASTRUCTURE]:
- `RandomSource` is the abstract base; the only required override is `protected abstract double DoSample()`. All generators accept an explicit integer seed. `RandomSeed.Robust()` uses `RandomNumberGenerator.GetBytes(4)` behind a static lock — thread-safe but contended under heavy concurrent seeding. The thread-safe instance flag wraps all `DoSample` calls in `lock(_lock)`, documented as 2.5× slower than non-thread-safe instances.
- `Xoshiro256StarStar` is the highest-quality generator available: 256-bit state across four `UInt64` fields, marked `[DataContract]` with `[DataMember(Order=1..4)]`. A seeded instance can be serialized mid-sequence and deserialized to reproduce the exact continuation. `MersenneTwister` carries the same `[DataContract]` attribute; `SystemRandomSource` wraps `System.Random`, which does not support serialization.
- `SystemRandomSource.Default` is a `ThreadLocal<SystemRandomSource>` initialized with `RandomSeed.Robust()` per thread. `Generate.Uniform(length)` and `Generate.Normal(length, ...)` delegate to this thread-local — non-deterministic across runs. Deterministic sampling requires constructing distributions directly with an explicit seeded generator: `new Normal(mean, sd, new MersenneTwister(seed)).Samples(array)`.
- `HybridMC` accepts `frogLeapSteps`, `stepSize`, `burnInterval`, per-dimension momentum standard deviations, an explicit `Random randomSource`, and a `DiffMethod diff` gradient delegate. The default gradient is a centered three-point finite difference with step `max(0.001, 1e-6 * |x[i]|)` — a hardcoded absolute floor of `0.001` that dominates near zero and is unsuitable for fields with sharp gradients at small scales. Supplying a user-provided `DiffMethod` is mandatory for such fields.
- `TensorPrimitives.FillGaussianNormalDistribution<T>(in TensorSpan<T> destination, Random? random)` fills a multi-dimensional `TensorSpan<T>` with Gaussian normals in-place — the SIMD-accelerated path for field-grid initialization that avoids per-element boxing from `Generate.Normal`. The `TensorSpan<T>` is constructed over a flat `T[]` using `TensorMarshal.CreateTensorSpan` with `nint[]` lengths and strides.

[FIELD_LAW_GAPS_AND_DESIGN_PRESSURE]:
- No Sobol or Halton quasi-random sequence is present. The `MathNet.Numerics.Random` namespace contains: `Mcg31m1`, `Mcg59`, `MersenneTwister`, `Mrg32k3a`, `Palf`, `SystemRandomSource`, `WH1982`, `WH2006`, `Xorshift`, `Xoshiro256StarStar`. Low-discrepancy sequences for quasi-Monte Carlo field integration must be sourced externally.
- Multi-dimensional scattered-data reconstruction is absent from the library: `KernelDensity` is a density estimator, not a function interpolant, and `Fit`/`Interpolate` are 1D surfaces. The correct composition is to assemble an RBF or polynomial design matrix from point coordinates and pass it to `MultipleRegression.DirectMethod` with `Svd` for rank safety.
- `OnRectangle`/`OnCuboid` use tensor-product rules: the same 1D Gauss-Legendre nodes applied independently in each dimension. At order `N` and dimension `d`, the evaluation count is `N^d`; no sparse-grid or Stroud-type rule is available. This makes the 3D cubature impractical beyond moderate order.
- `CubicSpline.InterpolateAkimaSorted` and all `*Sorted` variants require pre-sorted input and throw `ArgumentException` on unsorted data. The `*Inplace` variants sort in-place and mutate the input arrays. Field evaluation pipelines using `*Inplace` must clone data if the input is shared.
- Neither `DoubleExponentialTransformation` nor any adaptive Gauss-Kronrod overload catches `NaN` or `Infinity` propagated by the integrand. A signed-distance function evaluated at its zero set returns `NaN`; this propagates silently through the integrand without diagnostic. The only defense is a finite-fallback guard on the integrand delegate, with the failure recorded typed outside the integral call.
- `StepInterpolation` reports `SupportsDifferentiation: true` as an explicit interface implementation, but `Differentiate(t)` returns `0.0` everywhere within a segment and silently elides the distributional derivative at the jumps. Callers using `StepInterpolation` in a gradient-accumulating quadrature loop will receive zero gradient throughout.

[SIMD_AND_TENSOR_FLOOR]:
- `TensorPrimitives` on .NET 10 exposes generic-math overloads constrained by `IAdditionOperators<T,T,T>`, `IFloatingPointIeee754<T>`, `IRootFunctions<T>`. For field accumulation over `float` or `double` spans — gradient dot products, weighted kernel sums, diffusion residuals — `TensorPrimitives.Dot<T>`, `MultiplyAdd<T>`, and `FusedMultiplyAdd<T>` eliminate scalar loop overhead through hardware FMA and AVX-512 on supported processors.
- `TensorPrimitives.Norm<T>(ReadOnlySpan<T>)` computes the Euclidean 2-norm with overflow/underflow protection via scaled accumulation — the correct primitive for sampling acceptance criteria and kernel-center distance checks rather than a manual `sqrt(sum(x²))` loop.
- `TensorSpan<T>` and `ReadOnlyTensorSpan<T>` are N-dimensional with `nint[]` lengths and strides. A field sampled on a regular grid can be represented as a `TensorSpan<double>` with strides derived from grid dimensions; slice operations along any axis are zero-copy subspan projections, enabling plane-by-plane integration without copying.
- The public custom-operator injection surface is `TensorOperation.IUnaryOperation_Tensor<T, TResult>` with `static abstract void Invoke(ref readonly T x, ref TResult destination)` and a span overload. The `TensorOperation.Invoke<TOperation, ...>` dispatch loop is scalar-only — SIMD is locked inside `internal IUnaryOperator<T,T>`. For SIMD-accelerated custom field kernels, the span overload body must explicitly call `TensorPrimitives.*` primitives to reach the vectorized path.
