# Projection Boundary As Ref-Safety Construct

[PROJECTION_LAW_IS_TYPE_ENFORCEMENT_NOT_CONVENTION]:
- The capsule rule "a live handle never escapes; only a value copy leaves" is not a discipline the author upholds — it is a context the compiler computes. Every variable carries a *ref-safe-context* (where a reference to it is valid) and every expression a *safe-context* (where the value is valid), drawn from a four-level lattice ordered narrowest-to-widest: declaration-block, function-member, return-only, caller-context. A projection that compiles is one whose escaped value has safe-context at least return-only; a handle whose safe-context is function-member is rejected at the `return` by construction. The capsule's projection seam is therefore a place where the language's escape analysis is the enforcement mechanism, and the author's job is to shape the seam so the analysis decides correctly rather than to manually audit escapes.
- The live-pointer surface inside a capsule — a `Span<T>` over native memory, a `void*` from a pin, a `ref T` into a fixed buffer — is exactly the family the lattice governs: `ref struct` values and `ref` locals. A value copy of the projected result is a non-`ref-struct` `T` whose safe-context is caller-context unconditionally. The two project differently because they live at different lattice levels, and the capsule is correct precisely when the only thing crossing the seam is the caller-context-level value, never the function-member-level span.

[REF_STRUCT_IS_THE_ANTI_ESCAPE_TYPE]:
- A `ref struct` cannot reach the managed heap, and the heap is every long-lived escape route a capsule must close. The compiler forbids it as an array element, as a field of any `class` or non-`ref struct`, as a boxed `object`/`ValueType`, as a captured variable in a lambda or local function, and across an `await` in the same block or across a `yield return`. A capsule that holds its live view as a `ref struct` field of a `ref struct` representation gets all of these closures for free: the view physically cannot be stashed in a class field, cached in a static, enqueued into a channel, or captured by a continuation, because each of those operations requires a heap residency the type denies.
- This is why the live view must be surfaced *only* as a `ref struct`, never as a `Memory<T>` or an array slice. `Memory<T>` is a normal struct: it can sit in a class field and survive arbitrarily, so a capsule that returns `Memory<T>` over a pinned region has handed the caller a value that outlives the pin with no compiler objection. `Span<T>` is a `ref struct`: returning it from inside the pinned scope is a function-member-level escape the compiler refuses. The lattice level of the surfaced type *is* the lifetime contract; choosing `Span<T>` over `Memory<T>` for the in-scope view is choosing type-enforcement over convention.

[SCOPED_CLOSES_THE_SEAM_THE_BOUNDED_CALLBACK_LEAVES_OPEN]:
- The bounded-callback projection — a capsule method that hands the handle into `Func<H, Fin<T>>` and returns `Fin<T>` — does not, by itself, prevent escape. When the surfaced thing is a `ref struct` view, the projection delegate must be a `ref struct`-aware delegate, because `Func<,>` cannot carry a `ref struct` argument and cannot be applied to a `Span<T>` that must not escape the lambda. The correct seam takes the view by a custom `delegate` whose span parameter is marked `scoped`, which lowers that parameter's ref-safe-context to function-member and makes any attempt to assign it outward a compile error inside the delegate body.
- The deeper gap: a generic projection callback `Func<Span<byte>, T>` constrains nothing about `T`. With `T` unconstrained, the caller can instantiate `T = Span<byte>` (where the language allows) or smuggle the span into a closed-over field, and the projection law is silently broken at the type level even though the capsule "uses a bounded callback." The closure is to make the view parameter `scoped` and to forbid the return type from being a `ref struct` — express the callback as `delegate TResult View<TResult>(scoped ReadOnlySpan<byte> view)` where the return position is structurally a value. `scoped` on the parameter says "you may read this view here and nowhere else"; the non-`ref struct` return says "what you compute leaves, the view does not."

```csharp
// The view parameter is `scoped`: function-member ref-safe-context, cannot escape the body.
// TResult is a value (no `allows ref struct`): only a copy crosses the seam.
delegate TResult ViewOf<TResult>(scoped ReadOnlySpan<byte> view);

readonly ref struct NativeView      // ref struct: cannot be stored, boxed, captured, or awaited across
{
    readonly ref byte _origin;      // ref field: compiler guarantees it never outlives its referent
    readonly int      _length;

    public TResult Project<TResult>(ViewOf<TResult> read) =>
        read(MemoryMarshal.CreateReadOnlySpan(ref _origin, _length));   // span never named outside read
}
```

[REF_FIELD_BINDS_THE_VIEW_TO_THE_CAPSULE_LIFETIME]:
- A `ref struct` may hold a `ref` field — a stored reference, possibly null, probed with the null-ref intrinsic. The compiler enforces one invariant on it that is the whole point for capsules: a reference stored in a `ref` field cannot outlive its referent. The ref-reassignment rule `e1 = ref e2` admits the bind only when `e2`'s ref-safe-context is at least as wide as `e1`'s and their safe-contexts match, so a capsule view whose `ref` field points into native memory is structurally barred from being constructed over a referent narrower than itself. The view's validity window is computed, not asserted.
- The two-phase split between a logical extent and a physical address lock collapses into one type. The physical lock is the `ref` field's referent; the logical extent is the length beside it; the `ref struct` containing both has a safe-context no wider than the narrowest input it was built from. Pass it a span over a pinned region and the view inherits the pin's scope; pass it a span over a stackalloc and it inherits the stack frame. The author never tracks which — the lattice propagates it through every method that takes or returns the view.

[OUT_AND_THIS_DEFAULTS_ARE_THE_FACTORY_ESCAPE_RULES]:
- A `ref`/`in` parameter defaults to return-only ref-safe-context: it may be returned by `return` but not stored where it would outlive the call. An `out` parameter defaults to function-member — it is implicitly `scoped out`, so a factory that produces a view through an `out` parameter cannot, by default, let that view escape to the caller's context. A struct's `this` is scoped (function-member) by default, so an instance method on a `ref struct` capsule cannot return a `ref` into `this` without opting out. These three defaults are the implicit walls of a capsule factory: the language assumes the conservative lifetime and forces the author to *declare* any widening.
- `[UnscopedRef]` is that declaration — it widens a ref by exactly one lattice level: an instance member's `this`-ref from function-member to return-only, an `in`/`ref` parameter from return-only to caller-context, an `out` from function-member to return-only. It is the precise tool when a capsule legitimately must surface a `ref` into its own backing storage (an accessor returning `ref T` into a fixed buffer the capsule owns for the capsule's whole life). Its absence is the safe default; its presence is an audited assertion that the referent outlives every caller — exactly the assertion a long-lived pinned-heap capsule can make and a stack-frame capsule cannot.

[DISPOSABLE_REF_STRUCT_IS_THE_GUARANTEED_NON_DEFERRABLE_SCOPE]:
- A `ref struct` participates in `using` through the disposal *pattern* — a public parameterless void `Dispose` — without implementing any interface, and overload resolution prefers that pattern method even when the interface is also implemented. The value of the pattern over the interface here is structural: because the type is a `ref struct`, its `using` scope provably cannot be deferred across an `await`, cannot be captured into a closure that defers `Dispose`, and cannot be boxed into an `IDisposable` collection that disposes later. The release runs as the stack frame unwinds, at a point the type system pins to the lexical scope.
- This is the correct shape for any critical-section or address-lock disposal where ordering must hold against asynchrony — the lock scope, the pin scope, the disposal-ordering guard. A class-based `IDisposable` lock holder can be smuggled across an `await` and released on the wrong thread or after the protected teardown; the `ref struct` form makes that a compile error. Implementing `IDisposable` on a `ref struct` is available but defeats the guarantee precisely when used for its interface-ness: converting the `ref struct` to `IDisposable` is a boxing conversion the compiler forbids, so the interface is reachable only through generic code over a type parameter that admits ref structs, never through a heap-resident `IDisposable` slot.

[ALLOWS_REF_STRUCT_PROPAGATES_THE_VIEW_THROUGH_GENERIC_PIPELINES]:
- A `ref struct` view can be a type argument only where the type parameter declares the anti-constraint admitting ref structs, and that anti-constraint forbids the generic body from doing anything that would heap-escape the argument — no boxing, no field storage, no `default`-to-`object`. A generic projection combinator that must accept either a value carrier or a live `ref struct` view declares the anti-constraint and inherits, inside its own body, every escape restriction the concrete `ref struct` carries. The constraint flows the lifetime contract into the generic surface instead of forcing a non-generic duplicate for the view case.
- This is what lets one polymorphic projection entrypoint span both modalities — a value result and a live-view result — without a parallel method per shape. The anti-constraint is the single declaration that makes the generic surface honor ref-safety for the view instantiation; omitting it forces the view through a boxed path the language rejects, which is the signal that the combinator was written for heap values and must be widened, not copied. The constraint is a definition-time decision that determines, once, whether a generic resource-projection surface can carry live views at all.

[NARROWEST_ARGUMENT_RULE_IS_THE_COMPOSITION_SEMANTICS]:
- When a method returns a `ref struct`, the returned value's safe-context is the narrowest of caller-context and the safe-contexts and ref-safe-contexts contributed by all of its arguments — and a `scoped ref` argument contributes nothing. This is the compositional law a capsule pipeline rides: chain a view through three transforms and the result's lifetime is automatically the minimum of every input view's lifetime, computed without annotation. A transform that takes one view by `scoped` deliberately removes that view from the lifetime calculation, asserting "my output does not borrow from this input," which lets the output outlive the borrowed-but-not-retained source.
- The practical consequence for a capsule chain: a stage that reads a live native view and emits a value copy must take the view `scoped`, because that severs the output's safe-context from the view's and lets the value flow downstream past the view's disposal. A stage that re-slices a view and returns the sub-view must take it un-`scoped`, because the sub-view genuinely borrows and must inherit the parent's narrower lifetime. The choice of `scoped` on each stage's view parameter is the per-stage declaration of whether that stage's output borrows or copies — the entire capsule pipeline's escape behavior is encoded in those modifiers and verified by the narrowest-argument rule at every call.
