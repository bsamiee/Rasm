# Numeric Evidence Law: Receipts and Rails â€” Frontier

[SVD_ORDERING_CONTRACT_INVERSION]:
- `Svd<T>.S` is documented as "ascending value" in the XML but the `double` managed provider implementation contradicts this: `L2Norm = Math.Abs(S[0])` (the two-norm is by definition the largest singular value) and `ConditionNumber = Math.Abs(S[0]) / Math.Abs(S[last])` (ratio max-to-min). Both formulas are correct only if `S` is in **descending** order â€” `S[0]` is the maximum. A receipt that stores `S[0]` as the "minimum" or that asserts `S[0] <= S[1]` fails structurally. The ascending XML claim is wrong; the implementation is the authority. Any conservation check on a receipt that treats the sorted order as ascending rather than descending corrupts every downstream rank-threshold, condition-number, and pseudoinverse computation derived from the stored singular values.
- The rank threshold for the `double` concrete type is `S.Maximum().EpsilonOf() * Math.Max(U.RowCount, VT.RowCount)`. Because `S` is descending, `S.Maximum()` returns `S[0]` in O(n) by scanning the full vector rather than using `S[0]` directly. A receipt that pre-computes and stores the rank threshold as `S[0].EpsilonOf() * Math.Max(m, n)` is both correct and avoids a scan â€” but it must name the constant as the algorithm-specific threshold, not as a generic `Tolerance` field, because the formula differs for the `Complex32` type (where a type-narrowing cast to `float` multiplies the dimension): `tolerance = S.AbsoluteMaximum().Magnitude.EpsilonOf() * (float)Math.Max(m, n)` â€” the `(float)` cast reduces the effective dimension precision and produces a slightly wider threshold than the `double` variant.

[ILUTP_PRECONDITIONER_EVIDENCE_SURFACE]:
- `ILUTPPreconditioner` exposes three independent numeric policy constants: `FillLevel` (default `200.0`, dimensionless multiplier of the input NNZ), `DropTolerance` (default `0.0001`), and `PivotTolerance` (default `0.0`, meaning no pivoting). These three scalars are independent: `FillLevel` controls the storage budget of the factors (`NNZ(L+U) <= FillLevel * NNZ(A)`), `DropTolerance` controls which fill-in entries below that threshold are zeroed, and `PivotTolerance` triggers column pivoting only when the diagonal entry is below `PivotTolerance * â€–rowâ€–`. A receipt wrapping an ILUTP preconditioner must store all three as named fields; a receipt that stores only the final `NNZ(L)` and `NNZ(U)` from `UpperTriangle().NonZerosCount` and `LowerTriangle().NonZerosCount` preserves the fill outcome but loses the policy â€” making re-admission impossible because the same fill outcome can arise from different (`FillLevel`, `DropTolerance`) combinations.
- `ILUTPPreconditioner.UpperTriangle()` and `.LowerTriangle()` expose the factored matrices post-`Initialize`. The actual fill-in ratio `(NNZ(L) + NNZ(U)) / NNZ(A)` is a post-factorization degeneracy counter: when this ratio approaches `FillLevel`, the factorization saturated its budget and the preconditioner quality is controlled by the drop tolerance, not the fill target. A receipt that records `ActualFillRatio` alongside `FillLevel` encodes a conservation check: `ActualFillRatio <= FillLevel`.
- The `Pivots()` method returns the pivot permutation array after `Initialize`. A missing or null pivot array (when `PivotTolerance == 0`) means the preconditioner made no pivoting decisions. A receipt factory must distinguish between a zero-pivot-tolerance run (unconditional diagonal) and a positive-pivot-tolerance run (conditional pivoting applied) â€” these are qualitatively different evidence states, not both "no pivoting needed."

[ITERATOR_FIRST_WIN_SEMANTICS]:
- `Iterator<T>.DetermineStatus` iterates the stop-criteria list in registration order and returns on the **first** non-`Continue` status found, not on the most severe. This is a silent priority-by-insertion-order semantics: if `ResidualStopCriterion` is registered before `DivergenceStopCriterion`, a diverged solver that simultaneously satisfies the residual threshold will be declared `Converged`, not `Diverged`. The receipt factory must capture the full stop-criterion list in the same insertion order it was registered â€” not just the terminal `Iterator.Status` â€” so a re-admission can replay the priority logic and verify the status was not an order-dependent artifact.
- The `Iterator<T>()` zero-argument constructor populates itself from `Matrix<T>.Build.IterativeSolverStopCriteria(maxIterations: 1000)` â€” a virtual method on the builder. The default set contains `ResidualStopCriterion`, `DivergenceStopCriterion`, and `IterationCountStopCriterion`. The insertion order is: `ResidualStopCriterion` first, `DivergenceStopCriterion` second, `IterationCountStopCriterion` third. This means a solver that both converges and exceeds the iteration count on the same step returns `Converged` â€” the iteration count criterion never fires if the residual criterion fires first. A receipt must record whether an explicit `Iterator<T>` was constructed with explicit stop criteria, or whether the default constructor was used â€” the two paths can produce materially different status outcomes under borderline conditions.
- `ResidualStopCriterion<T>` constructor: `(double maximum, int minimumIterationsBelowMaximum = 0)`. The default for `minimumIterationsBelowMaximum` is `0` â€” meaning convergence is declared on the **first** iteration where the residual drops below the threshold, with no hysteresis. A receipt that records both the threshold and the hysteresis count is the only way to re-audit whether a premature convergence declaration was caused by a zero hysteresis policy.
- `DivergenceStopCriterion<T>` constructor: `(double maximumRelativeIncrease = 0.08, int minimumIterations = 10)`. The `minimumIterations` parameter has a hard lower bound of 3 (enforced by the constructor with `ArgumentOutOfRangeException`). The 10-iteration window is not the minimum possible â€” it can be reduced to 3. A receipt must record `MinimumNumberOfIterations`, because divergence at a 3-iteration window is a qualitatively weaker claim than at a 10-iteration window: a transient residual spike over 3 steps triggers divergence where a 10-step window would not.

[EXCEPTION_BOUNDARY_HIERARCHY]:
- The exception hierarchy has two distinct roots: `NonConvergenceException` (iterative algorithm failures) and `NativeInterfaceException` (BLAS/LAPACK provider failures). `NumericalBreakdownException : NonConvergenceException` â€” catch it before `NonConvergenceException`. The native root branches into `InvalidParameterException : NativeInterfaceException` (parameter validation, integer code carried), `MemoryAllocationException : NativeInterfaceException` (no fields), and `SingularUMatrixException : NativeInterfaceException` (integer element index carried: the specific diagonal position of the zero in U).
- `SingularUMatrixException(int element)` carries the 1-based index of the zero diagonal entry in U. This index is the boundary-level evidence that identifies which equation is degenerate â€” it is not a generic "singular" tag. At the boundary conversion point, `Error.New(SolverErrorCodes.SingularU, ex.Message, Error.New(ex))` preserves both the element index (in the message) and the exception identity. A domain receipt that maps `SingularUMatrixException` to an `Expected` error with domain code but reconstructs the element index separately from `ex.Message` string parsing is brittle; the correct pattern is to catch `SingularUMatrixException` specifically and extract `ex.Element` before constructing the error.
- The optimization exceptions form a separate hierarchy: `OptimizationException` (base, carries only message), `MaximumIterationsException : OptimizationException` (carries only the iteration count in the message string), `EvaluationException : OptimizationException` (carries `IObjectiveFunctionEvaluation ObjectiveFunction` â€” the last evaluation snapshot before failure), and `IncompatibleObjectiveException : OptimizationException` (structural incompatibility). `EvaluationException` is the only exception in the library that carries a typed evidence payload; its `ObjectiveFunction` gives the point, gradient, and Hessian at the failing evaluation. A receipt factory that catches `EvaluationException` and does not extract `ObjectiveFunction` discards the only post-failure diagnostic evidence the optimizer provides.
- `MinimizerBase.VerySmall = 1e-15` is a `protected const double` used as the floor for gradient/step/function progress tolerances. `LevenbergMarquardtMinimizer` uses `1e-15` for all three tolerance defaults. When a domain receipt stores these three tolerances as `double`s and they all equal `1e-15`, the receipt cannot distinguish a caller who set them intentionally from one who accepted the defaults. An honesty marker `bool UsesDefaultTolerances` â€” set at construction when all three are at `1e-15` â€” is the minimum distinction needed for re-admission to validate the tolerance regime.

[ATOM_VALIDATOR_RETURN_SEMANTICS]:
- `Atom<A>.New(A value, Func<A, bool> validator)` is `internal static Option<Atom<A>>`. The return type is `Option<Atom<A>>`, not `Fin<Atom<A>>`. When the initial value fails the validator, the result is `None` â€” no error code, no error message, no failure reason. A domain layer that surfaces atom construction failures on the numeric rail must convert `None` to `Fin.Fail(Error.New(domainCode, "initial state failed invariant"))` at the construction site; there is no error information to preserve because the library provides none. The `Prelude.atom(value, validator)` helper returns the same `Option<Atom<A>>`.
- When `Atom<A>.Swap(f)` produces a value that fails the validator, the swap is silently abandoned and the atom retains the prior value. The return value of `Swap` in this case is the prior value â€” identical to a successful swap that already held the prior value. There is no way to distinguish "validator rejected the swap" from "the function computed the value that happened to be the current value" from the return value alone. A receipt-accumulating `Swap` must compare the return value to the input value to detect a potential rejection â€” but this test is also ambiguous when the pure function is idempotent on the current state. A sentinel-typed `SwapMaybe(Func<A, Option<A>> f)` overload exists: when `f` returns `None`, the swap is skipped cleanly with no ambiguity, but the validator can still silently reject the `Some` case. The only pattern that eliminates ambiguity is to use `SwapMaybe` where `f` checks the invariant itself and returns `None` on failure, making the library validator redundant.

[RUNNING_WEIGHTED_STATISTICS_WEIGHT_SUM_GAP]:
- `RunningWeightedStatistics` tracks `_w1` (sum of weights) and `_w2` (sum of squared weights) internally, but exposes only `Count` (number of non-zero-weight pushes) and the computed moment statistics. The sum of weights (`_w1`) is fully private. The effective sample size under reliability weighting is `_w1^2 / _w2`, which cannot be computed from the public API. A receipt that needs to report effective sample size for confidence interval derivation must compute `_w1^2 / _w2` by accumulating it in a domain-layer parallel accumulator alongside `RunningWeightedStatistics`, or by replacing `RunningWeightedStatistics` with a domain type that exposes these fields â€” there is no re-entry point.
- The `Variance` property on `RunningWeightedStatistics` computes `_m2 / _w1` â€” the weighted population variance with no Bessel correction. This is reliability-weighted population variance, not sample variance. A receipt that records `Variance` as "sample variance" and uses it in downstream interval construction without the Bessel correction applied to `_w1` over-constrains the confidence bounds for small effective sample sizes.

[TENSOR_PRIMITIVES_STDDEV_HONESTY]:
- `TensorPrimitives.StdDev<T>` computes `sqrt(sum((x_i - mean)^2) / N)` â€” population standard deviation with the N normalizer, not the sample standard deviation `sqrt(.../(N-1))`. This is the same computation as `RunningWeightedStatistics.PopulationStandardDeviation` rather than `StandardDeviation`. A receipt that stores a `double StdDev` computed via `TensorPrimitives.StdDev<double>` and annotates it as sample standard deviation has a structural honesty violation. The distinction matters for any downstream F-test, chi-square test, or sample-to-population inference that uses the stored value.
- `TensorPrimitives.Dot<T>` constraint: `IAdditionOperators<T,T,T>, IAdditiveIdentity<T,T>, IMultiplyOperators<T,T,T>, IMultiplicativeIdentity<T,T>`. The `IRootFunctions<T>` constraint is not required for `Dot`, so `Dot<T>` works on any algebraic ring including integer types â€” unlike `Norm<T>` and `CosineSimilarity<T>`, which require `IRootFunctions<T>`. A receipt validity predicate for "residual within tolerance" over a custom ring type can use `TensorPrimitives.Dot` to compute the squared norm without requiring the full `IRootFunctions` surface, then manually apply the tolerance comparison â€” preserving exact arithmetic where `sqrt` would introduce rounding.
- `TensorPrimitives.HammingDistance<T>` has constraint only `T` (unconstrained generic) â€” it operates by bitwise equality comparison, not numeric. It returns an `int`. A receipt that calls `HammingDistance` on two `double[]` index vectors (e.g., pivot arrays, permutation arrays) to verify bitwise identity gets the number of differing elements as an integer, which is the correct conservation check for permutation preservation without floating-point ambiguity.

[FOURIER_CONVENTION_AS_HONESTY_MARKER]:
- `FourierOptions.InverseExponent` combined with `FourierOptions.NoScaling` corresponds to `NumericalRecipes` convention: `X_k = sum x_n * exp(+2Ď€i*k*n/N)` unscaled for forward, `x_n = sum X_k * exp(-2Ď€i*k*n/N) / N` for backward. `FourierOptions.Matlab` is an alias for `FourierOptions.AsymmetricScaling` â€” forward unscaled, backward by `1/N`. `FourierOptions.Default` is `0`, meaning symmetric `1/sqrt(N)` scaling on both forward and backward. These three conventions produce outputs that are scale-different by exactly `sqrt(N)` or `N` relative to each other. A receipt that encodes `FourierOptions` as an `int` bitfield preserves all two orthogonal facts (exponent sign and scale) in one field; a receipt that stores a prose string "Matlab convention" destroys the bitwise re-admission test.
- `Fourier.ForwardReal` throws `NotSupportedException` when `FourierOptions.InverseExponent` is set. This is a hard boundary: the packed real FFT surface supports only non-inverted exponent conventions. A receipt factory for real-signal forward transforms must gate on `(options & FourierOptions.InverseExponent) == 0` before invoking `ForwardReal`, returning `Fin.Fail` with a typed domain error rather than relying on the runtime exception. Failing to gate means the exception fires inside the library and must be caught at the external boundary, promoting it from a domain precondition failure to an `Exceptional` error.
- `FrequencyScale` places DC at index 0, positive frequencies at indices 1 through `length/2`, and wrapped negative frequencies at indices `length/2 + 1` through `length - 1`. For even-length N, the Nyquist frequency at index `N/2` is simultaneously the most positive and most negative frequency representable â€” it is a shared edge bin. A receipt that stores a frequency-axis receipt without flagging the Nyquist-bin ambiguity for even-length transforms leaves downstream spectral analysis to silently misinterpret the edge bin's contribution.

[NONLINEAR_COVARIANCE_CONSTRUCTION_GAP]:
- `NonlinearMinimizationResult` computes `Covariance = Hessian.PseudoInverse() * Value / DegreeOfFreedom` inside its constructor via `EvaluateCovariance`. `Hessian.PseudoInverse()` uses the SVD pseudoinverse, not the Cholesky inverse. For near-rank-deficient Jacobians, the pseudoinverse is well-defined but produces a covariance matrix whose diagonal entries for the degenerate directions are numerically large â€” not zero. A receipt that gates on `Covariance != null` but does not record whether any singular value was below the pseudoinverse threshold emits statistically credible-looking but inflated standard errors for rank-deficient models. The receipt must capture the `Svd.Rank` from the Hessian decomposition (computed internally during `PseudoInverse`) alongside `StandardErrors` â€” but `NonlinearMinimizationResult` does not expose this rank. A domain receipt wrapper that recomputes `Hessian.Svd(false).Rank` post-construction is the only path to the effective parameter count.
- The `EvaluateCovariance` path calls `objective.EvaluateAt(objective.Point)` unconditionally â€” a second objective evaluation after the minimizer already computed the final point. If the objective function has side effects, this second evaluation fires them at construction time. A receipt wrapping `NonlinearMinimizationResult` must be constructed immediately after the minimizer returns and before any external state the objective reads can change; delayed receipt construction from a cached `NonlinearMinimizationResult` does not re-trigger this evaluation but also cannot detect whether the second evaluation at construction time succeeded or mutated state.

[DESCRIPTIVE_STATISTICS_ACCURACY_FLAG_INVISIBILITY]:
- `DescriptiveStatistics` constructor takes `bool increasedAccuracy = false` which switches internal computation from `double` to `decimal`. After construction, the flag is not stored as a property â€” there is no `bool IncreasedAccuracy` or equivalent on the instance. The computed `Mean`, `Variance`, `Skewness`, and `Kurtosis` values are stored as `double` regardless of which path was used. Two `DescriptiveStatistics` instances from the same data with `increasedAccuracy = true` vs `false` will produce double properties that differ by up to `O(DoublePrecision)` in absolute terms on well-conditioned data, but may diverge visibly on large-magnitude data close to `decimal.MaxValue`. A receipt wrapping `DescriptiveStatistics` must record the `increasedAccuracy` flag separately at construction as a named `bool UsesDecimalArithmetic` field â€” the only moment the information exists.
- `DescriptiveStatistics.Variance` is the **unbiased** estimator using N-1 Bessel correction; `RunningStatistics.Variance` is also N-1 unbiased. But `TensorPrimitives.StdDev<T>` uses N (population). These three are not interchangeable in a receipt even though all produce a `double`. A receipt that accumulates statistics from multiple sources must carry a provenance field indicating the normalizer used â€” minimum: a two-case enum `{Population, Sample}` â€” or every downstream confidence computation is silently wrong.

[ERROR_COMBINE_OPERATOR_SEMANTICS]:
- `Error.Combine(Error other)` â€” accessible as `error1 + error2` via `operator +` â€” wraps both into a `ManyErrors` whose internal `Iterable<Error>` carries them in left-to-right order. `ManyErrors.Errors` is a `Seq<Error>` that preserves insertion order. A receipt factory that validates multiple conservation identities and uses `Error.Combine` to accumulate failures preserves the diagnostic order. `Error.Many(Seq<Error> errors)` does not exist as a static factory on `Error` â€” the correct surface for constructing a multi-error from a sequence is to fold over the sequence with `error1 + error2`, or to use `new ManyErrors(errors)` if the internal type is accessible. The public-API pattern is: `errors.Fold(Error.Empty, (acc, e) => acc + e)`.
- `Error.Filter<E>()` on a `ManyErrors` filters to only errors of type `E`, returning `Errors.None` if none match. On a single `Error`, it returns `this` if it is assignable to `E`, otherwise `Errors.None`. A receipt re-admission that needs to inspect only algorithm-specific error cases (e.g., only `SolverErrorCode.SingularU`) should use `receivedError.Filter<ExpectedError>()` to isolate expected domain errors and test `.HasCode(SolverErrorCodes.SingularU)`, leaving exceptional errors in the remainder â€” rather than pattern-matching the full `ManyErrors` graph.

[MANAGED_SVD_BLUESTEIN_OVERFLOW_BOUNDARY]:
- `ManagedFourierTransformProvider` uses a Bluestein FFT for non-power-of-two lengths and documents `BluesteinSequenceLengthThreshold = 46341` (the floor of `sqrt(Int32.MaxValue)`) as the threshold above which `k*k` overflows an `int32`. For lengths `N > 46341`, the implementation switches from `int`-indexed to `long`-indexed accumulation in the Bluestein sequence. A receipt wrapping a non-power-of-two FFT must record `N > 46341` as a boolean `UsesLongBluesteinSequence` honesty marker: when `true`, the frequency output may have accumulated slightly different rounding than the short-path implementation due to the widened intermediate accumulator, and any residual-norm check on a round-trip transform is sensitive to this path selection.
