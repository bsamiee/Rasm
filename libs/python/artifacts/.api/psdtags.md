# [PY_ARTIFACTS_API_PSDTAGS]

`psdtags` is the layered-TIFF container owner the `export/layered#LAYERED` `TIFF` arm composes: a pure-Python codec for the two Photoshop-specific TIFF tags — `ImageSourceData` (tag `37724`, the layer/mask/channel graph) and `ImageResources` (tag `34377`, the resource-block list incl. the embedded thumbnail) — that Photoshop, Affinity Photo, and Krita read as a fully separable, re-orderable, re-colorable layer stack inside an ordinary RGB TIFF the rest of the pipeline already writes through `tifffile`/`pyvips`. It owns the WHOLE typed object graph the TIFF-PSD format defines and the codec/container owners do NOT: `TiffImageSourceData -> PsdLayers -> PsdLayer -> PsdChannel` (the layer/channel spine, each channel a `numpy` 2-D array slot), the `PsdLayerMask`/`PsdUserMask`/`PsdFilterMask` mask records, the 90-member `PsdKeyABC` additional-layer-information tagged-block family (`PsdSectionDividerSetting` group folders, `PsdUnicodeString` names, `PsdMetadataSettings`, `PsdReferencePoint`, adjustment/effect/smart-object keys), and the `PsdResourceBlockABC` resource-block family (`PsdThumbnailBlock`, `PsdVersionBlock`, `PsdColorBlock`, the Pascal/Unicode string blocks). It is the structure authority only — it never compresses a channel byte (`imagecodecs` owns the `none`/`packbits`/`zip`/`zip-prediction` channel codecs through `psdtags.compress`/`decompress`) and it never reads or writes the TIFF directory itself (`tifffile` owns the container: `psdtags` emits one `(code, dtype, count, value, writeonce)` extratag tuple per tag via `TiffImageSourceData.tifftag(...)`/`TiffImageResources.tifftag()` that `tifffile.imwrite(..., extratags=...)` lays into the IFD, and reads them back through `TiffImageSourceData.fromtiff(...)`). Per the `[03]-[PACKAGE_ROSTER]` PSD decision this package is the layered-TIFF container only — native `.psd`/`.psb` authority is `PhotoshopAPI`/`psd-tools`; `psdtags` is retained for the Photoshop-compatible TIFF egress where the document/composite plane already produces a flat RGB TIFF and the only additive concern is the re-editable layer structure laid alongside it.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: the layered-TIFF author/parse round-trip the `export/layered` `TIFF` arm composes, plus the channel-codec bridge and the low-level stream primitives
- rail: layered

The `TIFF` arm authors a `TiffImageSourceData`, derives one extratag through `tifftag(...)`, and hands it to `tifffile.imwrite(sink, merged, photometric='rgb', extrasamples=('unassalpha',), metadata=None, byteorder=source_data.byteorder, extratags=(source_data.tifftag(maxworkers=4), resources.tifftag()))`; the byte-order MUST be read off the authored graph (`byteorder=source_data.byteorder`) and `photometric='rgb'`+`metadata=None` are the Photoshop-read contract. The `maxworkers` argument on `tifftag` is the channel-compression thread fan inside `imagecodecs`, NOT an `anyio` worker — it stays bounded inside the already-`to_process`-offloaded arm. `compress`/`decompress` are the explicit channel-codec bridge for a raw-channel path that bypasses the per-`PsdChannel.tobytes` default.

## [04]-[IMPLEMENTATION_LAW]

[LAYERED_PSDTAGS]:
- import: `import psdtags` at boundary scope only; module-level import is banned by the manifest import policy. The `export/layered` page reifies the structure authority through its `lazy import psdtags` proxy on first `TIFF`-arm use, exactly as it `lazy import`s `tifffile`/`pyvips`/`numpy` — `psdtags` is never imported on the runtime loader path because its `tifffile`/`imagecodecs`/`matplotlib` deps are optional and pulled only when the layered-TIFF arm runs.
- structure here, codec there, container elsewhere — three disjoint owners meeting at typed seams: `psdtags` authors the `TiffImageSourceData`/`PsdLayers`/`PsdLayer`/`PsdChannel` graph and emits the `tifftag(...)` extratag tuple; `imagecodecs` compresses each `PsdChannel.data` array (`PsdCompressionType -> imagecodecs.<codec>` — `RAW->none`, `RLE->packbits`, `ZIP->zlib/deflate`, `ZIP_PREDICTED->delta+deflate`, reached through `psdtags.compress`/`decompress` so the page never calls `imagecodecs` directly for a channel); `tifffile.imwrite(..., extratags=[isd.tifftag(maxworkers=4), res.tifftag()])` lays the tags into the IFD. Never re-implement a channel codec, a TIFF directory write, or a PSD layer record across these — the `_psd_layer` arm builds the typed graph, `compress` produces the channel bytes, `tifftag` produces the IFD value.
- byte-order + photometric are a single contract, not three knobs: the `tifffile.imwrite` MUST carry `byteorder=isd.byteorder` and `photometric='rgb'` (and `metadata=None` so no tifffile-private metadata corrupts the Photoshop read) — the `PsdFormat.BE32BIT` the `TiffImageSourceData` is authored with fixes `byteorder` to `'>'`, and a mismatch produces a TIFF Photoshop reads as flat or rejects. Author the source data, READ its `byteorder` prop, pass it through — never hardcode `'>'` at the `imwrite` site independently of the format the graph carries.
- the flat image is the merged composite, each layer's channels are that layer's own RGBA — two distinct `numpy` arrays, not one: the `TIFF` arm composites the visible placed layers to ONE `merged` canvas RGBA array (the `pyvips` `composite(placed[1:], modes[1:])` the `export/layered` reuses from the ORA path) and writes THAT as the flat TIFF image, while it builds each `PsdLayer` from the per-layer `image`'s own `_rgba_array` as four `PsdChannel`s — `CHANNEL0/1/2` the RGB and `TRANSPARENCY_MASK` the alpha — over the layer's `PsdRectangle(top, left, bottom, right)` bbox derived from `Layer.bbox`. Each channel is a contiguous big-endian `uint8` 2-D slice (`rgba[:, :, c]`); a discontiguous view must be `.copy()`-d before it reaches `PsdChannel.data`/`compress`, never handed in as a stride view a codec rejects. The alpha is UNASSOCIATED — the flat image declares `extrasamples=("unassalpha",)` on the `tifffile.imwrite` so a reader treats `TRANSPARENCY_MASK` as straight (non-premultiplied) alpha matching the `PsdChannelId.TRANSPARENCY_MASK` channel semantics.
- group folders are `PsdSectionDividerSetting` blocks on `PsdLayer.info`, not a nesting structure: the layered-TIFF format is a FLAT ordered layer list; a group is expressed by a `PsdSectionDividerSetting(PsdSectionDividerType.OPEN_FOLDER/CLOSED_FOLDER)` block opening the group and a `BOUNDING_SECTION_DIVIDER` divider closing it, both as `PsdKeyABC` entries on `PsdLayer.info` around the member layers. Author the `group` label as these divider blocks in list order — never assume a tree the format does not carry.
- blend mode is a closed map with one explicit fallback, never a passthrough string: the `export/layered` `_PSD_BLEND` `frozendict[BlendMode, psdtags.PsdBlendMode]` maps each pipeline `BlendMode` to its `PsdBlendMode` signature; a `BlendMode` with no Photoshop equivalent routes to one explicit `PsdBlendMode.NORMAL` fallback recorded on the layer receipt, never a `getattr(PsdBlendMode, name, default)` lookup or a raw 4-char bytes literal. The 29-member vocabulary is the totality — dispatch over it, do not stringly-spell a signature.
- round-trip equality is the correctness oracle: `TiffImageSourceData.fromtiff(written) == source_data` (and the same for `TiffImageResources`) is the verification the format defines — `unknown=True` on every parse preserves unrecognized blocks as `PsdUnknown` so the equality holds even for blocks `psdtags` does not model. The layered-TIFF receipt records the authored-layer count, the canvas shape, the per-channel compression method, and this round-trip assertion result, never a precision-loss field (the channel codecs are all lossless).
- evidence: the `TIFF`-arm `LayerFact` captures the authored-layer count, the canvas `(H, W)` shape and `byteorder`, the `PsdCompressionType` per channel, the `tifftag` `maxworkers` fan, whether a `TiffImageResources` thumbnail was embedded, and the `fromtiff`-round-trip equality result, folded onto the layer-export span as the existing `core/receipt#RECEIPT` `ArtifactReceipt.Preview` case (the editable layered document is a preview-class artifact), the authored-layer count on the shared `overlays` slot.
- boundary: `psdtags` owns the layered-TIFF tag STRUCTURE only — it never compresses a channel byte (`imagecodecs` via `compress`/`decompress`), never reads/writes the TIFF directory (`tifffile`), never composites the canvas (`pyvips`), never authors a native `.psd`/`.psb` document (`PhotoshopAPI`/`psd-tools` per the roster PSD decision), and never plots (the `matplotlib` viewer is the CLI only, not a library path). It re-renders nothing and re-lays-out nothing — it receives a placed canvas RGBA array and the per-layer bboxes and authors the re-editable layer tags alongside the flat image the container already writes.

## [05]-[INTEGRATION_LAW]

[STACKING] — the catalog stacks BOTH `.api` tiers: the shared/universal `libs/python/.api/*` rails ON TOP OF this folder package, never the folder set alone.

- container seam (folder `tifffile` `.api`): `psdtags` and `tifffile` are the two halves of the layered-TIFF egress — `psdtags` owns the `ImageSourceData`/`ImageResources` tag graph and emits each as a `tifftag(...)` extratag tuple; `tifffile.imwrite(sink, merged, photometric='rgb', extrasamples=('unassalpha',), metadata=None, byteorder=source_data.byteorder, extratags=(source_data.tifftag(maxworkers=4), resources.tifftag()))` lays the flat RGBA composite AND those two tags into one TIFF, and `TiffImageSourceData.fromtiff(file)` reads them back through `tifffile`. They meet at the extratag tuple and the file path, never re-implementing each other; the `extrasamples=('unassalpha',)` marks the 4th sample as straight alpha and `metadata=None` keeps tifffile-private metadata out of the Photoshop read. `read_tifftag` is the explicit `tifffile`-backed primitive for pulling the raw `37724`/`34377` bytes when a streamed source bypasses `fromtiff`.
- codec seam (folder `imagecodecs` `.api`): the per-channel byte compression is `imagecodecs`' — `psdtags.compress(channel_array, PsdCompressionType.ZIP_PREDICTED, rlecountfmt)` runs `imagecodecs.delta_encode` + `imagecodecs.deflate_encode` under the hood, and `PsdChannel.tobytes(psdformat, compression=…)` selects the codec by the channel's `PsdCompressionType`. The `imagecodecs.md` sibling already declares `psdtags` as the `TiffImageSourceData`/`PsdLayer`/`PsdChannel` structure owner it compresses channels FOR — this catalog is the mirror: `psdtags` hands a contiguous channel array to `imagecodecs` and gets compressed bytes; the `<CODEC>.available` capability probe lives on the `imagecodecs` side, so an absent channel codec routes to `RAW`/`none` with the achieved method recorded on the channel receipt.
- numeric seam (universal `numpy` `.api`): every `PsdChannel.data` is a contiguous big-endian 2-D `numpy` array, `PsdLayer.asarray`/`overlay` return `NDArray`, and `decompress` reconstructs `(shape, dtype)` arrays — the universal `numpy` rail owns the canvas RGBA array the `TIFF` arm composites and slices, `psdtags` owns its TIFF-PSD tag serialization. Slice each layer's channels from the placed canvas (`canvas[top:bottom, left:right, c]`), `.copy()` a non-contiguous slice before it reaches `PsdChannel.data`, and size nothing manually — the array carries its own shape/dtype into `compress`.
