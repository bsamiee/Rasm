# Sparse Systems: Core Law

[TRIPLET_ASSEMBLY]:
- `CoordinateStorage<T>` is the canonical triplet format. The constructor overload accepting pre-filled arrays requires the explicit `nonZerosCount` form; the two-parameter array overload treats the storage as empty and begins appending from index 0, making it useless for pre-built buffers.
- `CoordinateStorage<T>.At(i, j, value)` skips the append entirely when `value.Equals(Zero)` is true and throws `ArgumentOutOfRangeException` on out-of-range indices; only non-zero contributions increment `nz`. This matters for assembly: element stiffness contributions that produce exact-zero global entries are silently dropped at admission, so the triplet count after assembly is the non-zero count, not the element-contribution count. Resizing by factor 2 occurs when `nz == nzmax` on a non-zero contribution.
- `Converter.ToCompressedColumnStorage<T>(storage, cleanup, inplace)` always calls `Helper.SortIndices(result)` on both code paths — the `cleanup` flag controls only whether duplicates are summed via the marker-array scan (`Cleanup()`), not whether column indices are sorted. `cleanup: false` yields sorted but possibly-duplicated columns; `cleanup: true` (the default) sums duplicates via the marker-array scan and then sorts. The distinction: `SortIndices` is the sort pass; `Cleanup` is the duplicate-summation pass only. `DropZeros` is never called internally — exact zeros that survive deduplication (e.g., equal-and-opposite contributions that cancel to exactly 0.0) remain in the CSC representation after conversion.
- `CoordinateStorage<T>.Invalidate()` nulls `rowind`, `colind`, and `values` but leaves `nz` at its current value; the field `nz` is not reset to zero. Outstanding references to the nulled arrays become dangling. The `inplace: true` path in both `Converter.ToCompressedColumnStorage` and `CompressedColumnStorage<T>.OfIndexed` calls `Invalidate` after building the CSC result. `OfIndexed(coordinateStorage, inplace)` forwards the caller's `inplace` argument directly — the default is `false`. Do not use `inplace: true` when the triplet buffer must survive for structural-edit increments.
- `DropZeros(tolerance)` on `CompressedColumnStorage<T>` removes entries at or below the tolerance. The default is 0.0, so only exact binary zeros are dropped. For assembly results that accumulate near-cancellation residuals, pass a structural tolerance — typically `machineEps * frobenius_norm` — not 0.0.

[COMPRESSED_FORMAT]:
- CSparse stores CSC: `ColumnPointers` (length `n+1`), `RowIndices`, and `Values` as public fields on `CompressedColumnStorage<T>`. MathNet stores CSR: `RowPointers` (length `m+1`), `ColumnIndices`, and `Values` as public fields on `SparseCompressedRowMatrixStorage<T>`. The format hazard at the interop boundary is feeding CSC arrays to the CSR array factory `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseRowFormat` — the arrays are accepted without type-system rejection but the semantics are transposed, producing silently wrong preconditioner factors from a correctly-structured CSC matrix.
- `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` performs an in-memory transposition at construction time, allocating a new independent block. It calls `NormalizeOrdering()` and `NormalizeDuplicates()` but does NOT call `NormalizeZeros()` — explicit zeros in the input column-format arrays survive as explicit zeros in the resulting CSR storage. For the path from CSparse CSC to MathNet CSR for residual arithmetic, explicit zeros inflate `ValueCount` and slow preconditioner initialization; call `NormalizeZeros()` explicitly after `OfCompressedSparseColumnFormat`. For the path into `PopulateExplicitZerosOnDiagonal` before MKL DSS, diagonal explicit zeros are required and must not be removed. `PopulateExplicitZerosOnDiagonal` must be called before passing the result to MKL DSS — it fills zero entries on the diagonal explicitly as required by the MKL sparse solver interface. This is the correct bridge when converting from a CSparse solve result back to a MathNet-resident operator for residual arithmetic.
- `SparseCompressedRowMatrixStorage<T>.NormalizeOrdering()` sorts column indices within each row. `NormalizeDuplicates()` sums duplicate entries without sorting. `NormalizeZeros()` removes explicit zeros. `Normalize()` calls `NormalizeOrdering` then `NormalizeZeros` — it skips `NormalizeDuplicates` and therefore leaves structural duplicates from assembly intact, with values unsummed. The complete normalization sequence after building from coordinate format is `NormalizeOrdering()`, `NormalizeDuplicates()`, then `NormalizeZeros()` — exactly what `OfCompressedSparseColumnFormat` does minus the zero step. These are strictly CSR-side operations; the CSC parallel is `Helper.SortIndices` plus `DropZeros`. `Helper.SortIndices` uses a threshold of 16 elements per column: columns with ≤16 entries use insertion sort, columns with >16 use `Array.Sort`.
- `Helper.ValidateStorage(storage, strict: true)` checks that row indices within each column are sorted and contain no duplicates; it returns a bool and does not throw. The `strict: false` form accepts unsorted, possibly duplicate rows. Calling any factorization on storage that fails `strict: true` produces silently incorrect factors.
- `CompressedColumnStorage<T>.AutoTrimStorage` is `public static bool AutoTrimStorage { get; set; } = true` — a process-global mutable flag that is distinct per generic instantiation (`CompressedColumnStorage<double>` and `CompressedColumnStorage<Complex>` have independent static state). Setting `AutoTrimStorage = false` on one instantiation does not affect the other. Because it is process-global, disabling it during a structural-edit sequence mutates the shared state for all matrices of that type on all threads; the correct pattern is to save and restore the value under a lock or to confine structural edits to a single-threaded phase. `Keep` and sparse addition/multiplication both honor `AutoTrimStorage` — with it enabled, every structural operation resizes internal arrays to exactly `NonZerosCount` after the operation, O(nnz) copies that dominate incremental-build cost.
- `SparseMatrix.DoMultiplySparse` uses a two-pass algorithm: the first pass counts nonzeros in each result row using a marker array `int[other.ColumnCount]` initialized to `-1`, then allocates exact-fit `int[]` and `double[]` for the result; the second pass fills the values. This per-multiply O(n) work array allocation is not amortized across repeated multiplications — for Krylov subspace methods where the preconditioner involves a sparse product, this allocation recurs on every application.
- `SparseCompressedRowMatrixStorage<T>.At(row, column, value)` for a new structural entry inserts at the binary-search insertion point, shifting the suffix of `Values` and `ColumnIndices` via `Array.Copy`, and increments all row pointers above `row` — O(nnz) worst case. When two or more structural edits are needed, batching all modifications into a single `CoordinateStorage` and re-converting is cheaper than sequential in-place insertions. `RemoveAtIndexUnchecked` triggers a defensive trim — firing only when the array is more than half-empty after removal and has at least 1024 entries — creating a step-function memory profile for large matrices under progressive structural editing.

[SPMV_VECTORIZATION]:
- `SparseMatrix.DoMultiply(Vector<double> rightSide, Vector<double> result)` is a scalar CSR row-loop: for each row `i`, it accumulates `values[j] * rightSide[columnIndices[j]]` into a local `double` via a scalar loop. The `rightSide[columnIndices[j]]` access is a gather with irregular indices, which blocks vectorized gather-then-multiply. `TensorPrimitives` provides no indexed-gather API. The only SIMD path for sparse mat-vec in hot loops is to hand-vectorize using `Avx2.GatherMaskVector256` or `Avx2.GatherMaskVector128` from `System.Runtime.Intrinsics.X86` — gather intrinsics live on the `Avx2` static class, not on `Vector256<T>` or `Vector128<T>`.
- `SparseMatrix.DoTransposeThisAndMultiply(Vector<double> rightSide, Vector<double> result)` is a scatter-accumulate kernel: for each nonzero `(i, j, v)`, `result[j] += v * rightSide[i]`. The scatter-write pattern has write-after-write hazards across rows and cannot be vectorized without explicit atomics or row-partitioning. The forward SpMV (row-loop) is vectorizable via gather; the transpose SpMV (column-scatter) is not — this asymmetry governs solver inner-loop orientation selection for a given sparsity pattern.

[STRUCTURAL_INVARIANTS]:
- Symmetry verification via `CompressedColumnStorage<double>.IsSymmetric()` performs an O(nnz × log(nnz_max_column)) traversal — for each `(i, j, v)` it calls `At(row, col)`, which is a binary search over `ColumnPointers[column]..ColumnPointers[column+1]`. For matrices with dense columns (e.g., after Schur complement formation) this is significantly more expensive than a linear pass. The result is not cached. The correct design carries a typed witness — a constructor-verified value object whose admitted values are limited to matrices with confirmed structural symmetry — so the predicate runs once at the admission boundary, not at each solver call site.
- `CompressedColumnStorage<double>.IsSymmetric()` is `virtual` and declared on the base `CompressedColumnStorage<T>` class, where it tests real symmetry. `CSparse.Complex.SparseMatrix` overrides `IsSymmetric()` to test Hermitian symmetry (`v.Equals(Complex.Conjugate(At(i, row)))`). Virtual dispatch through a base `CompressedColumnStorage<Complex>` reference correctly reaches the Hermitian override; no down-cast is required for the test to be correct.
- `Keep(Func<int, int, T, bool>)` on both `CoordinateStorage<T>` and `CompressedColumnStorage<T>` drops entries in-place where the predicate returns false and returns the count of surviving entries. This is the correct path for structural edit operations — pin a node by keeping only diagonal entries in that row/column — without a full decompress-reassemble cycle. The predicate receives the current value, so it can combine structural and numeric drop conditions in one pass. With `AutoTrimStorage = true` (process-global), `Keep` also trims `RowIndices` and `Values` to exactly the surviving count; save and restore the flag around the `Keep`-then-`At` sequence to avoid intermediate reallocation.

[SINGLE_FORMAT_LAW]:
- A sparse workflow crosses three storage contexts with distinct format requirements: assembly (triplet, `CoordinateStorage<T>`); iterative-solver matrix-vector products (CSR or CSC, depending on operator orientation); and direct-factorization (CSC with sorted, duplicate-free rows for CSparse; CSR with explicit diagonal zeros for MKL DSS). Repeated conversion between all three is the multi-representation tax. The single-owner law: one `CompressedColumnStorage<T>` instance is the canonical owner. Triplet inputs convert to it once; iterative solvers that need CSR call `Transpose` once at construction; factorization reads the CSC form directly. Format is a named type distinction, not a flag on the owner.
- `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` and `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseRowFormat` are both members of the MathNet CSR storage type — there is no `OfCompressedSparseRowFormat` on CSparse's `CompressedColumnStorage<T>`. Both MathNet factory methods allocate a fresh independent block; neither is a zero-copy bridge. Feeding them the raw `ColumnPointers`/`RowIndices`/`Values` arrays of a CSparse matrix is the correct interop path; no wrapper or intermediate type is justified.

[FILL_REDUCING_ORDERINGS]:
- `SparseCholesky.Create` and `SparseLDL.Create` guard with `(int)order > 1`, which admits `Natural` (0) and `MinimumDegreeAtPlusA` (1) and rejects `MinimumDegreeStS` (2) and `MinimumDegreeAtA` (3) via `ArgumentException`. `MinimumDegreeAtPlusA` produces the best fill reduction for SPD systems and is the recommended choice; `Natural` is admitted but bypasses AMD entirely, producing no fill reduction — it short-circuits to the identity permutation without any AMD computation. `SparseLU.Create` accepts all four orderings; the guard is absent.
- `AMD.Generate<T>(A, ColumnOrdering)` computes the approximate minimum degree permutation vector and returns `int[]`. With `ColumnOrdering.Natural`, it short-circuits to the identity permutation without any AMD computation — the correct path when the matrix is already optimally ordered. The `Create` overload accepting `int[]` instead of `ColumnOrdering` lets callers cache or share the permutation vector across multiple factorizations with identical sparsity patterns but different numeric values.
- `SymbolicFactorization` exposes `lnz` (nonzeros in L) and `unz` (nonzeros in U or R) as direct integer public fields alongside `pinv`, `q`, `parent`, `cp`, `leftmost`, and `m2`. These are the fill-in evidence values. A sparse receipt carries these before the numeric factorization to expose fill ratio (`lnz / A.NonZerosCount`) as a diagnostic. `GraphHelper.EliminationTree` and `GraphHelper.ColumnCounts` are public static methods ordinarily called inside the symbolic analysis phase; calling them independently allows inspection of symbolic fill structure before committing to a numeric factorization.
- `SparseCholesky.SymbolicAnalysis` allocates `columnCount + 1` elements for `cp`, calls `PermuteSym` (a full upper-triangle copy of the permuted A with no values), then calls `EliminationTree`, `ColumnCounts`, and `CumulativeSum` — O(n) integer array allocations. The two-argument `Create(A, int[] p)` overloads on `SparseCholesky`, `SparseLDL`, and `SparseLU` accept a pre-computed permutation vector, skipping the AMD call but still running full symbolic analysis. Savings from caching `AMD.Generate` results are real but bounded by the symbolic phase cost, which is unavoidable through the public API. A pattern-keyed symbolic cache holding the `SymbolicFactorization` itself would fully amortize this, but requires calling `GraphHelper.EliminationTree` and `GraphHelper.ColumnCounts` directly — there is no public API to extract a `SymbolicFactorization` from a completed factorization.

[FACTORIZATION_LAW]:
- `SparseCholesky.Create(A, ColumnOrdering.MinimumDegreeAtPlusA)` performs symbolic analysis and numeric factorization in one call; only the upper triangular part is read. When a negative or zero diagonal pivot is encountered, it throws `System.Exception` with message "Matrix must be symmetric positive definite." — not `NumericalBreakdownException`, not `ArgumentException`. Callers catching only typed numeric exceptions will not intercept this failure. The silent hazard is asymmetry: only the upper triangle is read, so a nonsymmetric matrix whose upper triangle is SPD factors successfully and returns a correct solution for the symmetrized system — indefiniteness throws explicitly; asymmetry is silent. The receipt for Cholesky solves must include a post-solve residual recomputation `‖Ax − b‖₂ / ‖b‖₂` as the structural correctness witness.
- `SparseLDL` handles indefinite symmetric matrices but performs no numeric pivoting. `SparseLDL.Factorize` checks `if (D[i] == 0.0) throw new Exception("Diagonal element is zero.")` — this is `System.Exception`, not `NumericalBreakdownException`. A caller catching only `NumericalBreakdownException` will not intercept this failure; the receipt boundary for LDL factorization must catch `Exception` broadly. `SparseLDL.D` (the diagonal factor array) is a private field with no public accessor, so the inertia (sign distribution of D, revealing count of positive, zero, and negative eigenvalues) cannot be read from a completed factorization through the public surface. Auditing inertia requires a custom subclass or re-implementing the factorization loop. For strongly indefinite systems or those with near-zero pivots, LDL' produces inaccurate factors without warning; carry the negative-pivot count as a structural audit field in the receipt.
- `SparseLU.Create` accepts a `tol` parameter in [0, 1] controlling partial pivoting: `tol = 1.0` is full partial pivoting; `tol = 0` disables pivoting entirely. The parameter is a relative threshold — a pivot is accepted if it is at least `tol` times the magnitude of the largest element in its column, not an absolute pivot-size threshold.
- `SparseCholesky.Update(w)` and `Downdate(w)` implement rank-1 Cholesky updates in-place without re-factorization; both return `false` when the updated matrix loses positive definiteness rather than a typed error. There is no public `UpDown` method; `Update` and `Downdate` are the two public entry points. For structural-edit paths — pinning a node modifies one row/column — these are significantly cheaper than full re-factorization when the modification is rank-1.
- `SparseQR.Create(A, order)` branches on `A.RowCount >= A.ColumnCount`. For overdetermined systems (m ≥ n), AMD runs on A directly; for underdetermined systems, it transposes A first and runs AMD on `A^T`. `S.m2` tracks the augmented row dimension used for structural singletons — the actual Householder vector allocation is `S.m2`, not `A.RowCount`, and `S.m2 > A.RowCount` for rank-deficient systems. Fill evidence for QR derives from `S.lnz` and `S.unz` of the `SymbolicFactorization` — which is private after factorization, same as Cholesky/LDL/LU.
- All four `Create(..., IProgress<double> progress)` overloads report at a granularity controlled by `step = n / 100.0`. The gate `k >= current` fires as soon as the integer iteration counter crosses the accumulated threshold. For `n < 100`, `step < 1`, so the gate passes on every iteration — `Report` fires at a higher-than-1% rate, not zero times. For `n = 100`, `step = 1.0` and `Report` fires exactly 100 times. `IProgress<double>` carries no cancellation signal; cancellation requires running the factorization on a `Task` with external `CancellationToken` control.
- `ManagedSparseSolverProvider.Solve` (all four overloads) unconditionally throws `NotImplementedException` — there is no managed fallback implementation. The `ISparseSolverProvider<T>.Solve` interface accepts raw CSR arrays (`rowPointers`, `columnIndices`, `values`), RHS, and solution arrays as the bridge to MKL. Without an MKL-backed provider registered via `SparseSolverControl`, calling `Control.SparseSolverProvider.Solve(...)` throws immediately. The `DssMatrixStructure` enum values are MKL DSS option literals mapping to specific bit patterns — `Symmetric = 536870976`, `Nonsymmetric = 536871104`, etc. Passing the wrong `DssMatrixStructure` does not throw; MKL DSS returns `DssStatus.MKL_DSS_STRUCTURE_ERR = -14`. Code calling `ISparseSolverProvider<T>.Solve` must map every non-`MKL_DSS_SUCCESS` status to a typed domain failure before returning, because MKL DSS failure codes are not typed exceptions.

[KRYLOV_ITERATION]:
- `Matrix<T>.TrySolveIterative(input, result, solver, iterator, preconditioner)` returns `IterationStatus` without throwing under normal conditions — but preconditioner initialization failures and certain solver breakdowns propagate as uncaught exceptions (see preconditioner section). Only `Converged` represents an acceptable result. The distinction between `Diverged` (monotone residual increase detected) and `StoppedWithoutConvergence` (count cap hit without convergence) is load-bearing for fallback routing: divergence routes to a regularized retry or direct solve; count-exhaustion routes to a direct solve, optionally seeded with the partial iterate.
- `Iterator<T>` is constructed by passing an `IIterationStopCriterion<T>[]` array. `Iterator<T>.DetermineStatus` short-circuits: it iterates `_stopCriteria` in insertion order and returns the moment any criterion returns a non-`Continue` status. The correct minimum stack is `[FailureStopCriterion, DivergenceStopCriterion, ResidualStopCriterion, IterationCountStopCriterion]` in that order: NaN detection before divergence before convergence before hard cap. The default stack from `MatrixBuilder<double>.IterativeSolverStopCriteria(maxIterations)` is `[FailureStopCriterion, DivergenceStopCriterion, IterationCountStopCriterion, ResidualStopCriterion(1e-12)]` — `ResidualStopCriterion` is last, so the count cap suppresses a valid convergence signal on the final iteration. `Iterator<T>()` (no-arg constructor) hardcodes `maxIterations = 1000`; passing `null` as the iterator to `TrySolveIterative` substitutes a fresh default-constructed iterator with this cap. Always construct `Iterator<T>` explicitly with a custom stack so the cap is size-derived. `IterationCountStopCriterion<T>` default cap is a constant, not scaled to system size; the correct design carries the size-scaled cap as a policy value computed at factorization time.
- `ResidualStopCriterion<T>.DetermineStatus` measures `‖residualVector‖_∞ ≤ Maximum * ‖sourceVector‖_∞` — the threshold is relative to the right-hand-side infinity norm. The internal gate is `_iterationCount >= _minimumIterationsBelowMaximum` where `_iterationCount = iterationNumber - _lastIteration` and `_lastIteration` is updated to `iterationNumber` on every call where the residual is in-threshold. Under per-iteration invocation (the normal pattern), `_iterationCount` is always 1 because `_lastIteration` advances every call: `k >= 2` can never satisfy the gate regardless of how many consecutive in-threshold iterations occur. The criterion measures the gap between the current iteration number and the previous invocation, not a sustained count. A gap-based design requires calling `DetermineStatus` every `k`-th iteration rather than every iteration; the default `MinimumIterationsBelowMaximum = 0` degenerates to fire on the first in-threshold call, which is the only usable mode under standard per-iteration invocation.
- `DivergenceStopCriterion<T>` allocates `_residualHistory` of length `_minimumNumberOfIterations + 1` on first use (not at construction). The `IsDiverging` check tests all consecutive pairs in the window: `history[i] > history[i-1] * (1 + MaximumRelativeIncrease)` must hold for every pair — a strict AND over the entire window. Default `MaximumRelativeIncrease = 0.08` and default window of 10 mean every one of 10 consecutive pairs must show growth above 8%. A single residual dip in a 10-iteration window prevents the criterion from firing regardless of overall growth trend. The constructor enforces `MinimumNumberOfIterations >= 3` at the setter. For Krylov methods on non-symmetric systems (BiCGStab, TFQMR, GpBiCG), where residual oscillation is normal for the first 5–20 iterations, reducing the window to 3 (the minimum) produces spurious `Diverged` signals during the oscillatory warm-up phase; the default 10-iteration AND window is conservative enough to avoid this on symmetric problems but too aggressive for non-symmetric ones.
- `DelegateStopCriterion<T>` accepts `Func<int, Vector<T>, Vector<T>, Vector<T>, IterationStatus>` (signature: `iterationNumber, solutionVector, sourceVector, residualVector → IterationStatus`). Its `Clone()` shares the same delegate reference, so cloned iterators share mutable closure state if the delegate captures mutable fields.
- `Iterator<T>.DetermineStatus` guards at the top: `if (_status == IterationStatus.Cancelled) return _status`. Once cancelled, the iterator permanently returns `Cancelled` on all subsequent `DetermineStatus` calls. `Reset()` clears `_status` back to `Continue` and resets all stop criteria. `CompositeSolver` calls `iterator.Reset()` between solvers, so a `Cancelled` status from TFQMR's near-zero handling is discarded and the composite loop continues with the partial iterate in `result`. `CancellationStopCriterion<T>` links a `CancellationToken` into the stop-criteria stack; `Iterator<T>.Reset()` creates a new `CancellationTokenSource` linked to the original `_masterToken`. If `_masterToken` is already cancelled at the time of `Reset()`, the new TCS is immediately cancelled and every subsequent solver in a `CompositeSolver` loop cancels on the first iteration — a pre-cancelled token silently poisons the entire composite chain.
- `MlkBiCgStab.Solve` checks `_startingVectors` at the start: if non-null and of correct count and size, it uses them; otherwise it calls `CreateStartingVectors(DefaultNumberOfStartingVectors, n)`, where `DefaultNumberOfStartingVectors = 50`. The method returns `min(50, n-1)` orthonormal vectors — for `n = 10,000`, this produces 50 dense vectors of length 10,000, allocating 4 MB at solver-initialization time before the first iteration; for `n = 50`, it produces only 49. The orthonormal basis is constructed via Gram-Schmidt QR on a matrix filled with normally-distributed samples — the dense QR dominates the allocation cost and is the primary startup expense. Setting `NumberOfStartingVectors` to 3–5 eliminates this startup cost and is appropriate when a warm start from a prior time step reduces the benefit of multiple starting vectors. `MlkBiCgStab` throws `NumericalBreakdownException` at three distinct phases of the multi-Lanczos recurrence, all with the same message — which phase broke down is not distinguishable from outside `Solve`.
- Double-checked convergence: `TrySolveIterative` returning `Converged` means the `ResidualStopCriterion` saw the preconditioned residual fall below its threshold, not the true residual `‖Ax − b‖`. The independent recomputation `‖Ax − b‖₂ / ‖b‖₂` must be performed in the calling code because preconditioning can distort the internal residual's relationship to the true error.
- `TrySolveIterative` is defined on `Matrix<T>` where `T : struct, IEquatable<T>, IFormattable`. It does not constrain T to `IFloatingPoint<T>` or `INumber<T>`. `ResidualStopCriterion<T>.DetermineStatus` calls `residualVector.InfinityNorm()` and compares the result as `double` via a hard-coded `double.IsNaN` guard — the norm reduces to a real `double` scalar regardless of T. Criterion implementations cannot be made generic over floating-point precision using generic math constraints; the `double`-typed norm comparison is a fixed implementation detail, not a parameterizable policy surface.

[PRECONDITIONERS]:
- `ILU0Preconditioner` computes an ILU(0) factorization limited to positions already nonzero in the input. Its `Initialize` method internally calls `SparseMatrix.OfMatrix(matrix)` unconditionally — for a non-sparse input this is an O(n²) dense-to-sparse conversion; always pass a sparse matrix.
- `MILU0Preconditioner<double>.Initialize` explicitly requires `matrix.Storage` to be `SparseCompressedRowMatrixStorage<double>` — it throws `ArgumentException("Matrix must be in sparse storage format")` for any other storage. After `Initialize` throws, `IsInitialized` remains `false`; calling `Approximate` on an uninitialized instance produces garbage or NPE depending on whether `_alu` was partially assigned before the throw. Check `IsInitialized` before `Approximate` after any potentially-throwing initialization sequence. `MILU0Preconditioner` with `UseModified = true` (default) accumulates dropped fill into the diagonal so the preconditioner matrix preserves row sums of the original matrix — beneficial for M-matrices and diagonally dominant systems. For problems where the matrix is not an M-matrix — non-symmetric convection-dominated PDEs, for example — the correction pushes the approximation away from the true L and U factors in a non-locally-bounded way, degrading convergence relative to standard ILU(0). The `NumericalBreakdownException` thrown carries the message `"Zero pivot encountered on row " + num + " during ILU process"` where the row index corresponds to the CSR iteration order of the input matrix, not a permuted index. Pre-ordering with AMD before calling `MILU0Preconditioner` reduces zero-pivot probability; the preconditioner does not accept or apply a permutation — the caller must symmetrically permute the matrix using `Matrix<T>.PermuteRows` and `PermuteColumns` with a permutation derived from `AMD.Generate` before passing it to the preconditioner.
- `ILUTPPreconditioner` carries `FillLevel`, `DropTolerance`, and `PivotTolerance` as mutable properties. `PivotTolerance` defaults to 0.0, which disables partial pivoting — the condition `row(i,j) > row(i,i) / PivotTolerance` is never triggered when `PivotTolerance = 0`. Setting it to a small positive value (e.g., 0.1) enables partial pivoting. Forgetting to set `PivotTolerance` for non-diagonally-dominant problems leaves the preconditioner with unsound factors that manifest only as slow convergence or `NumericalBreakdownException`.
- `MILU0Preconditioner` and all other ILU preconditioners throw `NumericalBreakdownException` when a zero pivot is encountered during their ILU factorization. `TrySolveIterative` does not catch this exception — it propagates to the caller. The pattern is: initialize the preconditioner outside `TrySolveIterative`, catch `NumericalBreakdownException` at that boundary, and fall back to `DiagonalPreconditioner` or a direct solve before entering the iterative path.
- `BiCgStab.Solve` throws `NumericalBreakdownException` when `ρ_k ≈ 0` — the test is `num.AlmostEqualNumbersBetween(0.0, 1L)`, a 1-ULP tolerance check. `TFQMR.Solve` handles the same near-zero inner-product condition by calling `iterator.Cancel()` and breaking out of the loop, leaving `IterationStatus.Cancelled` — it does not throw. `GpBiCg.Solve` handles near-zero inner products by substituting `1.0` in place of the near-zero value and continuing — a damage-control heuristic that may yield a valid result for mildly ill-conditioned problems or an arbitrary vector for severely ill-conditioned ones; GpBiCg never throws `NumericalBreakdownException` and never calls `iterator.Cancel()`, continuing through all substituted-zero events until a stop criterion fires. Three incompatible failure modes for the same mathematical breakdown — unhandled exception, cancelled status, and silent substitution — with no syntactic difference at the call site. A safe outer boundary catches `NumericalBreakdownException` from `TrySolveIterative` regardless of solver choice; TFQMR's `Cancelled` and GpBiCg's continued-with-substitution results both require true-residual recomputation to confirm whether the partial result is usable.
- `CompositeSolver<T>` sequences through its solver list until one converges. Its routing: `Converged` → copy result and return; `StoppedWithoutConvergence` → copy partial result and continue to next solver; any other status (including `Diverged`, `Failure`, `Cancelled`) → reset input and try next solver. `CompositeSolver.Solve` wraps each sub-solver call in `try { ... } catch (Exception) { input.CopyTo(vector); continue; }` — any exception including `NumericalBreakdownException`, `OutOfMemoryException`, or preconditioner initialization failures is silently caught and the solver advances. The result after exhausting all solvers carries the last `StoppedWithoutConvergence` partial iterate with no indication of which solvers threw versus which ran to completion. The partial result from a `StoppedWithoutConvergence` solver is written to the output before continuing — if all solvers exhaust without `Converged`, the last such iterate is left in the output with no indication of inadequacy. `CompositeSolver.Solve` result always requires true-residual recomputation regardless of iterator status. `CompositeSolver` resets the iterator before each solver via `iterator.Reset()`, so convergence history is fully independent with no carry-over. `CompositeSolver`'s constructor resolves each preconditioner via `setup.CreatePreconditioner() ?? new UnitPreconditioner<double>()`, making `solver.Item2` always non-null. The external `preconditioner` argument passed to `Solve` is therefore dead — the expression `solver.Item2 ?? preconditioner` never reaches the right-hand side; each `IIterativeSolverSetup<T>` must return a configured preconditioner from `CreatePreconditioner()`.
- `DiagonalPreconditioner` stores only the diagonal. For near-diagonal systems it provides scale normalization at near-zero setup cost; for general systems it degrades convergence relative to ILU(0).

[GAUGE_HANDLING]:
- Sparse systems with pure Neumann boundary conditions or floating substructures have a singular stiffness matrix with a non-trivial null space. One policy concern admits four named dialects: large diagonal penalty (add `α * I` to singular rows); row/column elimination (replace gauge node's row and column with a zero row, zero column, and 1 on the diagonal, adjusting the right-hand side); gauge-node identity row (replace one row with the identity condition directly in the assembled `CoordinateStorage<T>` before conversion); and regularization shift (add `α * I` uniformly). All four are implementable through `CompressedColumnStorage<T>.Keep` combined with targeted `At` additions on the triplet before conversion. A single policy function parameterized by a `GaugeTreatment` discriminant produces the appropriate modification; a separate function per dialect is the anti-pattern.
- The large diagonal penalty value `α` must exceed the spectral radius of the stiffness matrix to be numerically effective. A practical lower bound is `α ≥ 1e8 * max_diagonal_entry`. The resulting system is ill-conditioned; for iterative solvers this stresses the preconditioner. The receipt for gauge-treated systems carries the gauge treatment type and the effective `α` value.
- `DulmageMendelsohn.Generate<T>(matrix, seed)` computes the coarse Dulmage-Mendelsohn decomposition via maximum bipartite matching. `StructuralRank` returns `rr[3]` — the size of the maximum matching. For a matrix with `StructuralRank < n`, no column ordering produces a non-singular factorization; the matrix is structurally singular and the gauge-handling policy applies regardless of numeric values. `DulmageMendelsohn.Singletons` counts 1×1 diagonal blocks in the block triangular form; for near-diagonal systems, `Singletons` approaching `Blocks` indicates near-cascade-solvability with O(n) direct solves rather than a full n×n factorization. `CoarseRowDecomposition` and `CoarseColumnDecomposition` partition the matrix into fine-horizontal, structurally-underdetermined, square-full-rank, structurally-overdetermined, and fine-vertical parts via five index intervals.

[REALIFIED_BLOCK_EMBEDDINGS]:
- When the factorization dependency is constrained to real-only solvers, a complex `n×n` system becomes a `2n×2n` real system with embedding `[Re(A), -Im(A); Im(A), Re(A)]`. Each complex entry `(i, j, a+bi)` contributes four real entries; the `CoordinateStorage<double>` for the block-embedded system has `4 * nnz` entries before deduplication. The `At` zero-skip mechanism (exact-zero contributions dropped at admission) handles purely-real or purely-imaginary entries correctly. The admission boundary for complex-to-real embedding is a single conversion function keyed on a structure discriminant: `Nonsymmetric` maps to no symmetry assumption; `Symmetric` maps to the real embedding with upper half only.
- `SparseMatrix.Transpose(storage: true)` on a Complex `SparseMatrix` performs a storage-level transpose without complex conjugation; the default `Transpose()` includes conjugation. The distinction is non-trivial when building Hermitian block systems manually.

[FACTOR_CACHE]:
- A weakly-keyed success-only factor cache holds solved factorizations keyed by the complete physical parameter set: matrix dimensions, sparsity fingerprint (hash of `ColumnPointers` and `RowIndices` arrays), and the ordering enum or permutation vector identity. The cache stores only successful factorizations — those where post-solve residual verification passed — so a failed or diverged solve never populates the cache.
- The permutation vector `int[]` from `AMD.Generate` is the stable key for numeric-only re-factorizations when sparsity is invariant but values change. Caching the permutation separately from the numeric factors lets the AMD phase run once. However, the symbolic analysis phase (`EliminationTree` + `ColumnCounts` + `CumulativeSum`) still runs on every `Create` call even with a pre-computed permutation — it cannot be bypassed through the public API. A pattern-keyed `SymbolicFactorization` cache (populated via `GraphHelper.EliminationTree` + `GraphHelper.ColumnCounts` directly) would fully amortize symbolic cost; this requires stepping outside the `SparseCholesky.Create` entry point. The `SymbolicFactorization` fields `lnz` and `unz` report nonzeros in the factors before numeric factorization; both belong in the receipt alongside the pre-ordering nnz of the input matrix so the fill ratio is auditable without re-running the factorization.
