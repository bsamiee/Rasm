# Spectral-Eigen

[ROUTE_SELECTION]:
- `Matrix<T>.Evd(Symmetricity)` is the single generic entry point for all dense eigensolution routes. `Symmetricity` is a four-case enum — `Unknown`, `Symmetric`, `Hermitian`, `Asymmetric`. Passing the known case skips the O(n²) `IsSymmetric()` scan; `Unknown` pays that cost on every call.
- `DenseMatrix.Evd` calls `DenseEvd.Create`, which delegates to `LinearAlgebraControl.Provider.EigenDecomp(bool isSymmetric, int order, T[] matrix, T[] matrixEv, Complex[] vectorEv, T[] matrixD)`. A provider wired via `Control.UseNativeMKL()` or `Control.UseNativeOpenBLAS()` substitutes a LAPACK driver behind this interface; the fallback is `ManagedLinearAlgebraProvider`. Non-dense types (`SparseMatrix`, `QR.Q`) route to `UserEvd.Create` — the pure-managed EISPACK path — regardless of the active provider. Materializing to `DenseMatrix` before calling `Evd` is required to reach the provider-backed path.
- The generalized symmetric-definite problem A·x = λ·B·x has no native MathNet route. The congruence reduction `L⁻¹·A·L⁻ᵀ` must be built manually using `chol.Factor.LU()` — not `Cholesky.Solve`, which returns `(L·Lᵀ)⁻¹·B`. The full recipe is in `[CONGRUENCE_REDUCTION]` below.
- MathNet has no sparse iterative eigensolver — `BiCgStab`, `GpBiCg`, `MlkBiCgStab`, and `TFQMR` are linear-system solvers only. Block-preconditioned eigeniteration must be hand-assembled using MathNet's dense `Evd` for the Rayleigh-Ritz subproblem and `SparseMatrix.Multiply(Vector<double>)` for operator application.

[EVD_RECEIPT_SHAPE]:
- `Evd<T>` exposes `EigenValues` (`Vector<Complex>`), `EigenVectors` (`Matrix<T>`), and `D` (`Matrix<T>`, block-diagonal). The `Vector<Complex>` carrier is uniform across all four scalar types — the shape is not polymorphic by symmetry class. For real symmetric inputs all imaginary parts are exactly zero after `SymmetricDiagonalize` zeroes `e[]`. To recover a real weight array: project via `evd.EigenValues.AsArray().Select(c => c.Real).ToArray()`; `AsArray()` returns the raw backing `Complex[]` without allocation for `DenseVector<Complex>`.
- `Evd<T>.IsSymmetric` is a `bool`, not the original `Symmetricity` discriminant. Both `Symmetric` and `Hermitian` inputs set `IsSymmetric = true` because `DenseEvd.Create` maps both cases to `isSymmetric = true`. The receipt loses the original symmetry evidence; a receipt that must distinguish them must carry `Symmetricity` separately.
- Eigenvalues are sorted ascending for symmetric and Hermitian inputs by the selection sort in `SymmetricDiagonalize`. The LAPACK path follows the `dsyev`/`zheev` ascending-order contract. Nonsymmetric inputs carry eigenvalues in Schur-convergence order with no sort guarantee.
- `Evd<T>.Rank` calls `EigenValues[i].AlmostEqual(Complex.Zero)`, which expands to `|λᵢ|² < DefaultDoubleAccuracy` where `DefaultDoubleAccuracy = DoublePrecision × 10 ≈ 1.11 × 10⁻¹⁵`. Because `ComplexExtensions.Norm(c)` returns `MagnitudeSquared`, not `Magnitude`, the effective magnitude floor is `|λ| < sqrt(1.11 × 10⁻¹⁵) ≈ 3.33 × 10⁻⁸`. This is an absolute threshold unrelated to spectral radius; for a matrix with ‖A‖₂ ∼ 10⁶, eigenvalues at magnitude 10⁻⁸ that are numerically significant are misclassified as rank-zero. Receipt-level rank must be recomputed as `n − count(i : |λᵢ| < ε_rank × max|λⱼ|)` with a caller-supplied `ε_rank`. `Determinant` uses the same `AlmostEqual` call to short-circuit to `0.0` the moment any eigenvalue has `|λ| < 3.33 × 10⁻⁸`.
- `NonConvergenceException` (symmetric QR exhausting 1000 steps per eigenvalue) and `ArgumentException` (non-square input or non-SPD B) are both thrown during `Evd` construction. In LanguageExt v5, `Try<A>` is `record Try<A>(Func<Fin<A>> runTry)`; `Run()` is an extension that wraps `runTry()` in a try/catch, converting any `Exception` to `Fin.Fail<A>(ex)`. The factory surface is `Fin.Succ` and `Fin.Fail<A>` — `FinSucc`/`FinFail` are `[Obsolete]`. Both failure types collapse to a single `Fin.Fail(Error)`; distinguishing exhaustion from argument violation requires tagging before the boundary: `ExhaustionCode`, `IndefiniteBCode`, and `ShapeCode` as domain-specific error codes survive the `Fin.Fail` carrier without inspecting `error.Exception` at use sites.

[REAL_COMPLEX_POLYMORPHISM_CEILING]:
- `Complex` implements `IMultiplyOperators<Complex, Complex, Complex>` but NOT `IMultiplyOperators<Complex, double, Complex>`. No BCL numeric type provides the `(Complex, double, Complex)` instantiation. A generic method constrained on `T : IMultiplyOperators<T, T, T>` cannot scale a `Vector<Complex>` by a real weight without an explicit conversion site.
- For spectral filter weight application, the delegate-bundle form is the correct wide parameterization: an explicit `Func<TM, double, TM> scale` alongside the filter function, with concrete instance `(c, w) => new Complex(c.Real * w, c.Imaginary * w)` provided at construction. `Vector<T>.Multiply(T scalar)` requires the scalar to be type `T`, so real-weight scaling of complex columns requires `new Complex(w, 0)` — which breaks a generic delegate signature. Collapsing to `where T : IMultiplyOperators<T, TWt, T>` fails because this two-parameter form is absent from the BCL hierarchy for these types.
- `TensorPrimitives.Dot<Complex>` compiles because `Complex` satisfies the four homogeneous interfaces. `TensorPrimitives.Norm<Complex>` does not compile — `Complex` lacks `IRootFunctions<Complex>`. `TensorPrimitives.MultiplyAdd<Complex>` compiles. `TensorPrimitives.FusedMultiplyAdd<Complex>` does not — `IFloatingPointIeee754<Complex>` is unimplemented. The real-domain hot loop uses `TensorPrimitives.MultiplyAdd<double>` and `TensorPrimitives.Dot<double>` directly; the complex-domain equivalent requires a manual loop.
- `Complex` satisfies `INumberBase<Complex>` and `ISignedNumber<Complex>` but not `INumber<Complex>` (which requires `IComparable<T>`). A constraint `where T : INumberBase<T>` accepts both `double` and `Complex`; `where T : INumber<T>` or `where T : IFloatingPoint<T>` excludes `Complex`. The Rayleigh quotient maps `Complex → double` via `.Real`, breaking the generic chain regardless of constraint choice. Receipt eigenvalue scalars must be `double` for symmetric and Hermitian inputs even when the eigenvector matrix is `Matrix<Complex>`.

[LOBPCG_MECHANICS]:
- LOBPCG requires three stages per iteration: orthonormalize the block X, apply A and optionally the preconditioner, then solve the Rayleigh-Ritz dense subproblem over the stacked space [X | A·X | P].
- `Matrix<T>.GramSchmidt()` routes through `DenseGramSchmidt.Create`, which unconditionally clones the input (`(DenseMatrix)matrix.Clone()`) and performs single-pass modified Gram-Schmidt. `DenseGramSchmidt` inherits `GramSchmidt<T>` whose constructor hardcodes `QRMethod.Full` — `.GramSchmidt()` always produces an n×n Q regardless of block width. `Matrix<T>.QR(QRMethod.Thin)` routes through `DenseQR.Create` with `ThinQRFactor`, producing an n×k Q that avoids the n×n allocation. For LOBPCG, `QR(QRMethod.Thin).Q` is both more numerically stable (Householder vs. single-pass MGS) and cheaper in allocation.
- `DenseGramSchmidt.Factorize` throws `ArgumentException("Matrix must not be rank deficient.")` only when column norm is exactly `0.0`. A near-zero column norm propagates silently — the column is normalized by its near-zero value, producing amplified rounding noise. Column norms must be checked against a domain-specific floor before calling `.GramSchmidt()`; the API provides no tolerance parameter. For the second reorthogonalization pass on near-dependent blocks: `GramSchmidt().Q` → check R diagonal via `gs.R.Diagonal().Any(v => Math.Abs(v) < floor)` → if triggered, call `GramSchmidt()` again on `.Q`. `QR<T>.R` is lazy (`_lazyR = new Lazy<Matrix<T>>(FullR.UpperTriangle)`), so diagonal access does not materialize R until needed.
- Diagonal preconditioning: the correction for residual column `r_i` given diagonal entry `a_ii` is `(λ - a_ii)⁻¹ · r_i`. When `|λ - a_ii| < ε`, fall back to identity (correction = `r_i`). The epsilon is algorithm-specific and must be supplied as a policy value, not taken from `Precision.DoublePrecision`.
- Previous-direction dropout: when a column's norm after orthogonalization falls below the floor, the P block contracts. The Rayleigh-Ritz dense subproblem shrinks from 3k to (2k + p) where p is the surviving P column count.
- Re-symmetrization of the Rayleigh-Ritz matrix is mandatory before `Evd(Symmetricity.Symmetric)`. `DenseMatrix.IsSymmetric()` uses exact bit equality; any accumulated matrix will route through the nonsymmetric Hessenberg path if passed with `Symmetricity.Unknown`. The minimum-allocation re-symmetrization uses `s.MapIndexedInplace((i, j, v) => (v + s[j, i]) / 2.0, Zeros.Include)` — `DenseColumnMajorMatrixStorage` overrides this with a `CommonParallel.For` loop over columns and direct array access, allocating nothing. `Zeros.Include` is required; the default `Zeros.AllowSkip` may leave zero off-diagonal entries unaveraged when the opposite entry is nonzero.

[SPECTRAL_FILTER_ALGEBRA]:
- The spectral filter algebra maps a signal through an eigenbasis: for eigenpairs `{(λᵢ, vᵢ)}` the filtered output is `∑ᵢ f(λᵢ) · (vᵢᵀ · x) · vᵢ`. The filter cases — heat `exp(-t·λ)`, wave `cos(t·√λ)`, biharmonic `λ²`, diffusion `1/(1 + t·λ)`, commute-time `1/√λ`, identity `1` — form a closed union whose per-case `Invoke(double lambda)` is a virtual method on each case. The delegate-bundle form (storing a raw `Func<double, double>`) is open and breaks the closed-union invariant; it is appropriate only at the receipt boundary where the weight vector is materialized.
- Zero-mode handling: for Laplacian-derived operators λ₀ ≈ 0; filters with `1/λ` or `1/√λ` diverge there. The weight at index i is `λᵢ < ε_zero ? 0.0 : f(λᵢ)` — excluded, not clamped. `ε_zero` is a policy parameter. When the Laplacian is normalized by degree, the effective spread floor must be computed as `eigenvalues.Where(λ => λ.Real > ε_zero).Min(λ => λ.Real)`, not blindly taken from `EigenValues[1].Real`, because that index may itself be below the exclusion threshold.
- `Vector<T>.Map(f, result, Zeros.Include)` applies the map to all elements including zero-valued eigenvalues. `MapIndexed` enables index-first exclusion (the 0th mode is always the zero mode for normalized graph Laplacians) without the epsilon comparison.
- Partial-monoid composition: `f₁ ∘ f₂` produces a new `Product(f₁, f₂)` case whose `Invoke(λ) = f₁.Invoke(λ) × f₂.Invoke(λ)`. `Identity` is the monoid unit. The materialized weight vectors compose via `weights1.PointwiseMultiply(weights2)` — O(n). A weight sum below a bandwidth floor is a distinct exhaustion marker: the receipt carries `WeightSum` and the call site gates on it, because a zero weight sum (all modes excluded or underflowed) must not be interpreted as a valid zero-signal result.

[CONGRUENCE_REDUCTION]:
- The generalized problem A·x = λ·B·x with B SPD reduces via `B = L·Lᵀ` to `C = L⁻¹·A·L⁻ᵀ`, then `C·y = λ·y`, then `x = L⁻ᵀ·y`. `Cholesky<T>.Solve(B)` dispatches to `CholeskySolveFactored`, which performs both forward and backward substitution — it returns `(L·Lᵀ)⁻¹·B`, not `L⁻¹·B`. Any congruence step that requires `L⁻¹·A` alone must bypass `Cholesky.Solve` and use `chol.Factor.LU()` directly.
- Correct two-step sequence: (1) `Y = chol.Factor.LU().Solve(A)` → solves L·Y = A → `Y = L⁻¹·A`; (2) `Z = chol.Factor.LU().Solve(Y.Transpose()).Transpose()` → `Z = L⁻¹·A·L⁻ᵀ`. `Matrix<T>.Solve` has no bare triangular overload — it always routes through LU; for lower-triangular L this is O(n²). Each step allocates an n×n matrix. `Transpose()` materializes a copy, not a lazy view.
- Re-symmetrization before `Evd`: congruence asymmetry grows as O(n² × ε_mach × κ(L)). For ill-conditioned B (κ(B) ∼ 10¹²) the asymmetry reaches O(10⁻⁴) and the un-re-symmetrized C routes through the nonsymmetric Schur path, producing complex eigenvalues for a real symmetric-definite problem. Apply `MapIndexedInplace((i, j, v) => (v + c[j, i]) / 2.0, Zeros.Include)` before passing with `Symmetricity.Symmetric`; the IEEE 754 commutativity of addition makes the result bitwise symmetric, so the `IsSymmetric()` check passes.
- Back-transform: `X = L⁻ᵀ·Y` via `chol.Factor.Transpose().LU().Solve(evd.EigenVectors)`. `Transpose()` allocates an n×n copy; `.LU()` allocates the PLU struct. Three n×n allocations for the complete back-transform.
- `DenseCholesky.Solve` throws `NotSupportedException("Can only do Cholesky factorization for dense matrices at the moment.")` when either input or result is not `DenseMatrix`. The type signature is `Matrix<double>` at the abstract level, so this fails at runtime, not compile time.
- `Cholesky` positive-definiteness detection checks strictly `num > 0.0` (diagonal after Schur complement). Rank-deficient B (exactly zero diagonal) throws `ArgumentException("Matrix must be positive definite.")`. Near-singular B (very small positive diagonal) passes and produces a numerically poor factor; the correct remedy is to perturb with ε·I before factorization.
- Receipt route discrimination: `IndefiniteBFault` (Cholesky throws), `ExhaustionFault` (Cholesky succeeds but `Evd` on C exhausts), and success are three distinct receipt states. If Cholesky fails, no eigenpairs exist. If Evd fails, L is computed and valid; the receipt should carry L's condition evidence alongside the failure.

[RESIDUAL_EVIDENCE]:
- MathNet provides no built-in residual for eigendecomposition. Per-pair residual: `(A.Multiply(eigVec) - eigVec.Multiply(eigVal)).L2Norm()`. Block residual: `(A.Multiply(V) - V.Multiply(D)).FrobeniusNorm()` where D is built via `Matrix<T>.Build.DenseDiagonal(n, n, i => eigenvalues[i])`.
- The receipt shape for correctness must add: per-pair residual norms, the route variant (`dense-symmetric`, `dense-general`, `dense-hermitian`, `congruence-reduced`), and a sort-guarantee flag (symmetric: ascending; nonsymmetric: Schur-convergence order, no guarantee). `Evd<T>` carries none of these.
- Symmetric exhaustion throws `NonConvergenceException` before `SymmetricDiagonalize` completes the sort and back-transform. Eigenvectors for indices below the failing `j` are complete and orthonormal; those at and above `j` are intermediate Givens-rotation products — numerically meaningless. The failing index is not exposed by the exception. The `ExhaustionFault` receipt should record the spectral radius estimate (`max(|d[k]| + |e[k]|)` at the point of failure) as the only available diagnostic for near-degeneracy.
- For nonsymmetric real inputs, the managed path throws `NonConvergenceException` after `30 × n` total outer Schur iterations — not per-eigenvalue. For nonsymmetric complex inputs, the managed path has no iteration bound and no exception; the `while (num >= 0)` loop terminates only on deflation. Silent non-termination for complex inputs makes post-hoc Frobenius residual validation the only detection mechanism.

[CACHE_AND_TRUNCATION]:
- `Matrix<T>.Evd(Symmetricity.Symmetric)` always computes all n eigenpairs. For a filter using only the k smallest, over-compute-and-truncate: `evd.EigenVectors.SubMatrix(0, n, 0, k)` allocates an n×k backing array — there is no zero-copy truncation because `DenseColumnMajorMatrixStorage` takes ownership. `DenseEvd.Create` allocates four arrays before calling `EigenDecomp`: the n×n eigenvector matrix, n×n block-diagonal matrix, length-n `DenseVector<Complex>`, and a copy of the input — totaling `3n² + 2n` doubles of working memory (≈ 24 MB for n = 1000).
- The receipt rewritten to the truncated request carries `ComputedCount = n`, `RequestedCount = k`, and `CacheSource` distinguishing `Full` (truncated from a cached n-wide basis, O(n³) cost amortized) from `Fresh` (new full computation). A `Resident` vs `Reconstructed` annotation records whether the `Evd` object is retained in memory or recomputed.
- A cache hit requires compatible n and `Symmetricity`. A mismatch on `Symmetricity` must recompute. The full O(n³) eigensolver is not re-invoked for a cache hit; only `SubMatrix` is called.
