# Generated Value Objects

[OWNER_DECLARATION]:
- `[ValueObject<TKey>]` and `[ComplexValueObject]` attach to a `partial` class or struct; the generator weaves factories, identity, equality, conversion, parsing, and metadata into the owner, leaving the visible body for invariants only ‚Äî the validation partials, optional comparer attributes, and domain operations. Reference-type owners are emitted `sealed partial class`, struct owners `readonly partial struct`: derivability and member mutability are closed by generation, not convention, and an author-declared `abstract` or mutable field collides at compile time. The read-only discipline reaches inherited state ‚Äî base-class fields and properties must also be read-only (TTRESG034/035) ‚Äî so mutability cannot be laundered through a base type.
- The generated interface set on a keyed owner is the dispatch contract: `IEquatable<T>`, `IKeyedObject<TKey>`, `IConvertible<TKey>` (explicit-interface `ToValue()`, aggressively inlined), `IObjectFactory<T,TKey,TError>` (static abstract `Validate`), and an internal metadata-owner interface; complex owners drop the keyed interfaces and gain `IEqualityOperators<T,T,bool>`.
- One attribute per owner: stacking both attributes is an analyzer error. All constructors must be private (generated-constructor accessibility is the `ConstructorAccessModifier` property, default private); primary constructors are rejected because they cannot be private. Record declarations are rejected outright ‚Äî record-synthesized equality, formatting, and copy semantics would collide with every generated surface, so the positional-record spelling of a value object is unrepresentable rather than subtly wrong.
- Generic owners are first-class: `[ValueObject<TypeParamRef1>]` on `Box<T>` generates a fully generic owner ‚Äî generic factory triad, generic conversions ‚Äî with `TypeParamRef1`‚Äì`TypeParamRef5` as positional placeholders into the owner's type parameters, and each closed constructed type materializes its own static metadata instance. A type parameter in the key position must carry a `notnull` constraint (TTRESG074) ‚Äî generic owners cannot quietly admit nullable type arguments into identity.
- Struct owners are stamped `[StructLayout(LayoutKind.Auto)]`; field-order layout assumptions and unmanaged-interop use are void by construction, and the auto layout yields only to an author-declared explicit layout attribute, which transfers all layout-correctness obligations to the author.
- `SkipEqualityComparison` deletes `IEquatable<T>`, both equality operators, and the `Equals`/`GetHashCode` overrides in one cut: a class owner silently degrades to reference identity and a struct owner to reflection-based field equality ‚Äî flipping it without replacements turns a value type into an entity type with no diagnostic.
- Access-modifier configuration is one composable lattice (private, protected, internal, public, plus the two composite protections as bitwise unions) applied uniformly across the constructor and key-member knobs ‚Äî one vocabulary governs every generated accessibility decision on the owner.

[DERIVATION_VISIBILITY]:
- Derived surfaces are emitted as one satellite declaration per capability ‚Äî factory/identity core, then separate partials for `IComparable`, comparison operators, equality operators, each arithmetic family, `IFormattable`, `IParsable`, `ISpanParsable` ‚Äî and each satellite exists only when the generator can see the key's corresponding capability on the key type's declared (not generated) surface.
- That visibility test has a structural blind spot: source generators cannot observe generated output, including their own. A keyed owner whose key is itself a generated owner in the same compilation derives only the core ‚Äî factories, equality, conversions, metadata ‚Äî and silently loses comparison, parsing, formatting, and arithmetic even though the inner owner demonstrably implements all of them after generation. The identical declaration with the inner owner in a referenced assembly derives the full surface, because compiled metadata is visible: owner-keyed owners reach maximal derivation only across an assembly boundary, and composing them intra-assembly is a capability downgrade with no diagnostic.
- The blindness extends to nullability inference: generated `ToString()` delegates to the key, and for a generated-owner key the generator resolves only `object.ToString()`, emitting `public override string? ToString()` ‚Äî the nullable-annotated override is a visible fingerprint of an under-derived owner.
- Conversion operators do not compose transitively: an owner keyed by another owner gets exactly one hop in each direction, no outer‚Üíraw operator is manufactured through the chain, and double-cast spelling at call sites is the designed shape. The component-model converter on such an owner is closed over the inner owner as key ‚Äî converter-driven binding chains through the inner owner's own converter, so string-shaped input crosses two admissions to reach the outer type.

[FACTORY_TRIAD]:
- `Validate` is the primal factory; `Create` and `TryCreate` are generated derivations that call it. Scalar shape: `static TError? Validate(TKey? value, IFormatProvider? provider, out T? obj)` (struct owners emit a non-nullable `out T obj`); complex shape drops the provider and takes one parameter per member. Only `Validate` returns the typed error and the instance in one call ‚Äî it is the sole signature that preserves both for a single downstream projection.
- `Create` throws `ValidationException` carrying `validationError.ToString()` ‚Äî the typed error is string-flattened and lost on the throwing path. `TryCreate` ships two overloads (`out obj` and `out obj, out error`) with full `NotNullWhen(true)`/`NotNullWhen(false)` flow annotations.
- `Create` and `TryCreate` hard-code a null `IFormatProvider` into their `Validate` call; the provider-aware admission entries are `Validate` itself, `Parse`/`TryParse` (which thread the provider through both the key parse and the validation), and the component-model converter (which forwards the ambient culture). A culture-sensitive validation partial is unreachable from the bare factory pair by design.
- `CreateFactoryMethodName` and `TryCreateFactoryMethodName` rename the generated pair so the owner's public grammar matches domain language while `Validate` keeps its contract name for the static-abstract interface; assigning whitespace or empty reverts silently to canonical names rather than erroring ‚Äî renames driven from computed constants must not assume an invalid value will be caught.
- `FactoryPostInit()` is a generated instance partial invoked after successful construction on the factory path only ‚Äî a definition-time advice point for derived-cache priming that constructor-route materialization never touches; an undeclared body compiles away to nothing.

[VALIDATION_HOOK_PIPELINE]:
- The factory hook is a static partial with the error slot first and every argument by `ref`: scalar `static partial void ValidateFactoryArguments(ref TError? validationError, ref TKey value)`; complex `static partial void ValidateFactoryArguments(ref TError? validationError, ref M1 a, ref M2 b)`. Writing back through a `ref` parameter is the sanctioned normalization channel ‚Äî trim, clamp, canonicalize before capture; the stored state is the post-hook value. An undeclared partial is elided by the compiler ‚Äî the hook is pay-for-play per owner.
- Framework null gates run before the user hook: class owners reject a null key with a generated message; complex owners null-check every reference-type member per member, minting fault text that names the offending member and the owning type in member declaration order ‚Äî field attribution on the null tier is generator-authored, so a flat fault stream from a multi-member admission never loses which member produced the rejection before any hand-written labeling downstream. The hook body can assume non-null inputs.
- The two hooks form an ordered pipeline, not two independent guards: factory-hook normalization completes before construction begins, so the constructor-scoped hook observes only canonical input and never re-normalizes. The ordering inverts on bypass routes ‚Äî rehydration construction runs only the constructor hook against stored values, so any normalization that must also hold for stored material is either re-derivable from the stored form or it silently does not exist on that path.
- `ValidateConstructorArguments(ref ...)` runs inside the private constructor on every construction path, including persistence rehydration through the raw constructor; it can only throw, never report a typed error. Normalization placed only in the factory hook is silently bypassed on the constructor path.
- Complex admission is fail-first single-error by construction: generated `Validate` returns on the first null reference member, then runs the single hook, which can surface exactly one error object. Per-member error accumulation belongs inside the hook ‚Äî the only point that sees all members simultaneously, post-null-gate and pre-capture ‚Äî and the natural shape is an applicative sweep folded into one composite error:

```csharp
static partial void ValidateFactoryArguments(ref BundleError? validationError, ref string name, ref string code)
{
   name = name.Trim();
   code = code.Trim();
   var faults = Seq(
      name.Length is 0 ? Some("<message-a>") : None,
      code.Length > 12 ? Some("<message-b>") : None).Somes();
   validationError = faults.IsEmpty ? null : BundleError.Create(string.Join("; ", faults));
}
```

[STATE_THREADED_ADMISSION_EVIDENCE]:
- `ValidateFactoryArguments` is signature-polymorphic on its return type: declaring the partial with a non-`void` return changes the generated declaration to `private static partial TState ValidateFactoryArguments(...)` and rewires `FactoryPostInit` to accept that value ‚Äî `Validate` captures the hook's return into a local and passes it to `FactoryPostInit(state)` after construction.
- The non-`void` return is the only channel for evidence the members themselves cannot hold: by the time the instance exists its members are `init`-frozen, so a value computed during validation ‚Äî the pre-normalization original, a parsed sidecar, a derived classification ‚Äî cannot be stored on the instance after the fact and cannot be a member without widening the public surface. The hook return threads it across the construction boundary into the post-construction partial, where derived caches prime against it exactly once, with no second pass.
- The pair is transactional admission: construction and the post-construction partial both sit inside the success branch of generated `Validate`, so a rejected candidate never constructs and never posts ‚Äî a side effect placed in the post hook (one-time registration, derived-field publish, diagnostic emit) is reached if and only if the value is genuinely admitted, with no half-constructed instance ever observable to it.
- The threaded value exists only on the factory path; constructor-route materialization invokes neither hook nor `FactoryPostInit`, so any state reconstructed from it must be derivable again or the rehydrated instance silently lacks it.

```csharp
private static partial (int Major, int Minor) ValidateFactoryArguments(
   ref TagError? validationError, ref string value)
{
   var parsed = TryParseParts(value);
   validationError = parsed is null ? TagError.Create("<message-a>") : null;
   return parsed ?? default;
}

partial void FactoryPostInit((int Major, int Minor) state) { }
```

[TIGHTENING_PLACEMENT_LAW]:
- Invariant tightening is the one admission change with zero compile signal ‚Äî the factory narrows silently while values persisted under the looser rule still exist ‚Äî and the hook chosen for the tightened check decides the failure mode of that drift. Placed in the factory hook, the tightening rejects fresh input with a recoverable typed error while old stored data continues to rehydrate through the bypass constructor; placed in the constructor hook, it throws on the rehydration path too and converts invariant drift into a load-time fault. The first placement makes tightening a data-migration decision with a code rider; the second makes it an availability decision.
- The split generalizes: the factory hook owns every rejection that may legitimately occur on fresh input, because it is the only hook with an error channel; the constructor hook owns the invariant-of-record that must hold on every birth path including deserialization and trusted rehydration, and can only throw. An invariant placed in the factory hook alone is a statement that stored data is exempt from it ‚Äî that statement should be deliberate, never an accident of which partial was convenient. The type system stays silent either way; the proof obligation lives entirely outside it, against both construction routes.

[ERROR_CONTRACT]:
- `IValidationError<TSelf>` demands exactly one member: `static abstract TSelf Create(string message)` ‚Äî no message property, no base class. Any sealed type with a static `Create(string)` satisfies it; `[ValidationError<TError>]` on the owner rewires every generated signature ‚Äî `Validate` return type, `TryCreate` out-parameter, the `IObjectFactory` arity, the type-converter arity ‚Äî to the custom error type.
- Framework-originated failures route through the same contract: null-key rejection, parse-validation failure, and converter errors are all minted via `TError.Create(message)` ‚Äî a custom error type observes every admission failure, and its `Create` must accept arbitrary framework prose, not a closed vocabulary. For complex owners the custom type also receives the framework's per-member null-rejection messages, so a structured error type that parses its message string must tolerate the framework's prose grammar on those paths, or model itself as message-carrying with structure added only by its own factory call sites.
- The throwing path calls `validationError.ToString()`; a custom error type that does not override `ToString()` produces exceptions whose message is the error type's fully-qualified name ‚Äî `ToString()` returning the message is a load-bearing obligation the interface does not state.
- The shipped default error type is value-semantic ‚Äî equatable by message with message-returning `ToString` ‚Äî so accumulated error collections deduplicate and compare by message content out of the box.

```csharp
public sealed class TagError : IValidationError<TagError>
{
   public string Message { get; }
   private TagError(string message) => Message = message;
   public static TagError Create(string message) => new(message);
   public override string ToString() => Message;
}
```

[KEY_MEMBER_IDENTITY]:
- The default key member is `private readonly TKey _value` ‚Äî a private field, not a public property; the key escapes only through the generated conversion operator and the explicit-interface `ToValue()`. Exposing `.Value` requires `KeyMemberAccessModifier = AccessModifier.Public, KeyMemberKind = MemberKind.Property`, with `KeyMemberName` following the declared kind. Default-hidden identity is deliberate: consumers compare and dispatch on the owner, not its raw key.
- `SkipKeyMember = true` hands the key member to the author; the generator targets the declared member by `KeyMemberName`, and a property implementation must have a private `init` accessor ‚Äî a public `init` is an analyzer error, closing the object-initializer bypass around the factory. When the declared member is missing, the generator emits a member-not-found diagnostic followed by a member-type-mismatch cascade through every derived satellite ‚Äî cascade size proportional to the key's arithmetic capability.
- A nullable key member is rejected by the analyzer; absence belongs to the factory's null-handling modes or to an option carrier outside the owner.

[COMPARER_ACCESSOR_TYPECLASS]:
- Identity policy is carried by a type, not a value: a comparer accessor is a class or struct whose only obligation is one or two static-abstract properties ‚Äî `IEqualityComparerAccessor<T>.EqualityComparer` and `IComparerAccessor<T>.Comparer` ‚Äî threaded purely as a type argument on `[KeyMemberEqualityComparer<TAccessor, TKey>]` and its ordering sibling. Generated `Equals`, `GetHashCode`, `CompareTo`, and every relational operator reach the comparer through the constraint, so policy resolves at generic-instantiation time with no field, no constructor parameter, and no per-owner allocation: the comparison surface is a typeclass dictionary passed at compile time, and one attribute swap changes every comparison surface at once.
- String-keyed owners default to ordinal-case-insensitive equality, hashing, and ordering, silently merging identifiers differing only in case; the analyzer warns (TTRESG048) until the policy is explicit via an accessor, and complex owners with string members get the same default and parallel warning (TTRESG049) until `DefaultStringComparison` is set. Shipped accessors cover ordinal, ordinal-ignore-case, culture variants, and `Default<T>`.
- Both accessor interfaces are contravariant (`in T`): an accessor written over a base or interface type satisfies the constraint for every key assignable to it ‚Äî one accessor class supplies policy for a whole inheritance family of keys. The interfaces place no `class` constraint, so a `readonly struct` accessor is legal, and because generated code reaches the comparer only through the static member the accessor type is never instantiated ‚Äî its sole runtime footprint is the comparer object the property returns.
- The two concerns are split into two interfaces precisely so they can be satisfied independently ‚Äî ordering without custom equality and vice versa ‚Äî while pairing diagnostics re-couple them at the surface: a comparer without its counterpart draws a warning because hash containers and sorted containers would disagree on identity, and the generic default accessor satisfies the pairing rule for non-string keys without inventing a comparer.
- Every generated comparison is a two-hop dispatch: a static-abstract property read resolved at compile time, then a virtual `Equals`/`Compare` on the returned object. The property body is evaluated on every comparison ‚Äî an accessor that constructs a fresh comparer per get manufactures a new object on each `Equals` inside a hot dictionary probe; the correct shape stores the comparer in a `static readonly` field and returns it, because the type is the dictionary and the comparer is its only state.

```csharp
public sealed class CodepointAccessor : IEqualityComparerAccessor<string>, IComparerAccessor<string>
{
   private static readonly IEqualityComparer<string> EqInstance =
      new ProjectionEqualityComparer<string, string>(static s => s.Normalize().Trim(), StringComparer.OrdinalIgnoreCase);
   public static IEqualityComparer<string> EqualityComparer => EqInstance;
   public static IComparer<string> Comparer => StringComparer.Ordinal;
}
```

- The shipped projection comparer composes the same way the owner composes the accessor: the selector picks a sub-value, then defers to an inner comparer that can itself be a string-policy comparer ‚Äî a two-level policy (normalized derived substring, compared case-insensitively) is one accessor whose field stacks selector over inner comparer, inherited through a single type argument; its hash arm returns zero for a null projection, so null-projecting elements bucket together rather than throwing. Derived-field and collection-member semantics become a one-line accessor body rather than a hand-written comparer class.
- The keyed owner's `GetHashCode` is the accessor's hash of the key member, never the member's own hash ‚Äî the `Equals`-implies-same-hash contract is structurally guaranteed under any custom equivalence, not reviewed. A reference-typed key from a type parameter null-coalesces its hash to zero, so a null-keyed generic owner is hashable in the degenerate case the comparer would otherwise reject.
- Accessor/member type agreement is a compile error; on violation the generator still threads the mismatched accessor into `Equals`/`GetHashCode`, so the genuine diagnostic is followed by a type-mismatch cascade inside generated code ‚Äî triage always starts from the first analyzer error, never from the generated-file errors below it.
- Container-level identity override is a shipped surface: a generic comparer over any owner convertible to string exposes six prebuilt static instances (ordinal, ordinal-ignore-case, and both culture pairs) that compare owners by egressed key under the container's chosen policy ‚Äî a case-sensitive set of case-insensitive owners, or a culture-ordered index over ordinal owners, without re-keying or touching the owner's intrinsic policy; it rides the universal explicit-interface egress, so it works for every string-keyed generated owner uniformly.

[ORDERING_CONVERGENCE_AND_RAW_COMPARISON]:
- `CompareTo` and the comparison operators are independent derivations that agree only through an accessor: `CompareTo` routes accessor ‚Üí string-policy comparer ‚Üí a cast of the key to `IComparable<TKey>`, while the operators forward the key's own `<` `<=` `>` `>=` directly when no accessor is configured ‚Äî two routes that can disagree when a key implements one but not the other. A configured ordering accessor forces both through the accessor's `Compare`, restoring a single ordering definition; this convergence is the structural reason the pairing diagnostics treat equality-and-ordering accessors as one policy.
- Relational operators couple downward: `ComparisonOperators` broader than `EqualityComparisonOperators` draws a divergence warning (TTRESG105) ‚Äî set the pair together or not at all. The pairing is also self-healing at the attribute surface: the equality-operators getter returns the maximum of itself and the comparison setting, so requesting key-type overloads for comparison transparently grants them for equality in the generated output, while the divergence diagnostic still warns on the explicit spelling ‚Äî the clamp keeps the heterogeneous operator pair co-present even where the source attribute spells only one.
- `OperatorsGeneration.DefaultWithKeyTypeOverloads` equality generates `owner == rawKey` and `rawKey == owner` backed by one private static `Equals(owner, key)` that applies the owner's comparer policy ‚Äî the same case or accessor policy that governs owner-to-owner equality. It is a policy-aware membership test against the owner's value space, never a proof the raw operand is admissible: a raw value the factory would reject simply returns unequal, indistinguishable from an admissible-but-different value; the raw side is never validated.
- The relational key-type overloads compare the owner directly against an unadmitted raw bound through the same accessor ‚Äî a range gate at the boundary runs the owner's collation against a raw threshold without admitting it, without casting the owner out to the key, and without materializing a second owner; this is the sanctioned spelling for raw-threshold gating.
- `CompareTo(object)` returns 1 for null and throws `ArgumentException` for foreign types ‚Äî sorted non-generic containers fail loudly on heterogeneous content rather than ordering arbitrarily.
- The operators-generation vocabulary makes generation the zero state: `None` is `-1` and `Default` is `0`, so an unset attribute property means "generate" and opting out is an explicit negative assignment ‚Äî tooling that reads attribute data and treats zero as "off" inverts the semantics.

[COMPLEX_EQUALITY_MEMBERSHIP]:
- `[MemberEqualityComparer<...>]` on any complex-owner member flips equality membership from all-members to opt-in: unmarked members vanish from `Equals`, `GetHashCode`, and the generated `ToString` while remaining factory parameters and stored state ‚Äî and `DefaultStringComparison` becomes inert for them. Two instances differing only in an unmarked member compare equal; the change is an equality-semantics rewrite, not a local tweak. The reified member list on complex-owner metadata remains the full assignable set regardless of equality membership ‚Äî identity is opt-in, materialization is total.
- A member-level accessor is the same one-property class attached per member, so the member list carries a heterogeneous vector of typeclass policies ‚Äî one field case-insensitive, another ordinal, another sequence-equated ‚Äî all resolved statically inside the single generated `Equals`. The accessor tier is the only tier that overrides a member's intrinsic semantics: no owner-wide switch reaches a single member without an accessor, so correcting one wrong member equality is always a per-member accessor promotion, never a global setting.
- Per-member equality dispatch is a fixed four-tier chain: explicit accessor, then the owner-level string policy for direct string members, then null-tolerant `a.Equals(b)` for reference members, then direct `.Equals` for value members. No tier consults `EqualityComparer<T>.Default` for references ‚Äî a collection-typed member (`IReadOnlyList<T>`, arrays) compares by the collection's own `Equals`, which is reference identity; the only sanctioned fix is an explicit member comparer accessor implementing sequence semantics.
- `GetHashCode` in complex owners has two shapes: a `HashCode.Combine` fast path used only when equality membership has fewer than eight members, zero accessors, and zero string members; otherwise incremental `HashCode.Add` with the comparer threaded per member ‚Äî adding one string member or one accessor flips the whole method's form, keeping hash and equality policy-aligned by construction.
- `[IgnoreMember]` removes a stored member from the generator's world entirely (no factory parameter, no equality participation); immutability of the ignored member is the author's obligation. Expression-bodied computed properties are invisible to the generator without any attribute.
- An equality membership of zero members generates `Equals` returning constant true and `GetHashCode` returning 0 ‚Äî every instance is one equivalence class: structurally legal for marker values, a trap when members were removed from equality incrementally.

[GENERIC_MATH_OPERATOR_SYNTHESIS]:
- Operator capability on the key is computed once into a per-key flag record that drives every arithmetic and relational satellite; the record is filled by two independent passes that never re-probe. The interface pass walks the key's implemented interfaces and, on finding a homogeneous generic-math operator interface, grants the entire operator family in one assignment ‚Äî both the unchecked and checked flag of each arithmetic family, all four relational flags for comparison ‚Äî with no per-operator method inspection. The method pass then sets flags individually from the presence of each corresponding operator method, only for families the interface pass left dark: a key either lights up a whole family through an interface or accretes operators one method at a time.
- The interface pass demands exact self-self operand symmetry: the framework numerics interface by name and namespace, three type arguments, the first two both equal to the key. The result argument is unconstrained ‚Äî a key whose binary operator yields a third type still triggers full-family generation ‚Äî while a key implementing only the mixed-operand form fails the second-argument equality and is invisible to the interface pass entirely: operand symmetry, not result-type identity, is the admission gate.
- The method pass classifies an operator as arithmetic only when it is static, takes exactly two by-value parameters, and has return type and both parameter types all equal to the key type. A hand-written key operator returning a delta type, or mixing operand types, satisfies neither pass and produces no owner operator ‚Äî the owner's generated arithmetic is strictly the homogeneous closed-operation subset of the key's algebra; richer key algebra surfaces only as hand-written domain operations on the owner.
- Checked detection is a separate flag bit fed by a separate operator-method name: a key carrying only the unchecked operator yields an owner with no checked variant, and a key carrying only the checked operator yields the checked operator with no generic-math interface ‚Äî the interface emits only when the unchecked bit is set, because the checked member lives inside the same interface and is meaningless without the default one. The owner mirrors a partially checked-capable key exactly rather than fabricating the missing mode.
- Operator availability is detected per operator from the key's `op_*` method set, not only from generic-math interface implementation ‚Äî custom key types with hand-written operators qualify.

[INTERFACE_EMISSION_GATING]:
- The equality-operator interface (`IEqualityOperators<T,T,bool>`) is the unconditional floor ‚Äî always emitted with self-self type arguments because it derives from the owner's own equality, not from any key operator. Arithmetic and ordering participation are earned from the key; equality participation in generic math is reached by every owner.
- The comparison interface declaration is gated on full four-operator presence: a key with a partial relational set yields the matching partial operator subset on the owner with no `IComparisonOperators<T,T,bool>` ‚Äî a silent capability split where source-level `<` compiles while generic-constraint dispatch rejects the owner.
- The string-key and comparer-accessor arms of the comparison gate manufacture the interface over keys with no relational operators at all ‚Äî a string-keyed owner with an ordering accessor routes `<` `<=` `>` `>=` through the accessor's `Compare` and implements the full comparison interface, becoming more generic-math-capable than its key. Policy attributes do not merely configure comparison; they synthesize operator algebra the key never had, lifting the owner to full interface citizenship where the key's own algebra cannot reach it.
- Every generated arithmetic family declares its generic-math interface on the owner ‚Äî `IAdditionOperators<T,T,T>` through `IDivisionOperators<T,T,T>`, with `checked` variants satisfied inside the same interface ‚Äî so admitted owners flow directly into operator-constrained generic algorithms.

[ARITHMETIC_REENTERS_ADMISSION]:
- Arithmetic operator generation defaults on whenever the key implements the corresponding generic-math interface: a bare `[ValueObject<int>]` ships `+ - * /` and their `checked` variants with no configuration. Every operator body routes the raw result back through the throwing factory ‚Äî arithmetic re-enters admission, so `a - b` on validated operands can throw `ValidationException` when the result violates the invariant: an innocent-looking subtraction is a validation site, and owners with range invariants must treat generated arithmetic as a policy decision. Renaming the throwing factory renames the arithmetic interior ‚Äî a `From`-renamed owner's `+` re-enters admission through `From`.
- The `checked` variants wrap only the key arithmetic in `checked(...)` and still route through the throwing factory ‚Äî one `checked +` has two failure species: `OverflowException` from the key math and `ValidationException` from re-admission.
- Every arithmetic body wraps the raw key arithmetic in a cast back to the key type exactly when the key is one of the integral types the language promotes under binary arithmetic ‚Äî the byte, signed-byte, short, unsigned-short, and char family ‚Äî computed from the key's special type, never from operator inspection. The cast target is the key type, not whatever result type the key's operator interface declares, so a key designed to widen under arithmetic has that widening erased at the owner boundary: the owner's arithmetic is closed over the key by force.
- The narrowing cast sits inside the checked wrapper for the checked variant and outside it for the unchecked variant, so one narrow-key operator pair carries two failure surfaces: unchecked promotes, computes, truncates silently, then re-admits the wrapped value ‚Äî an out-of-range sum validates its modular residue with no signal under a permissive invariant ‚Äî while checked computes under overflow detection on the wide type and throws before any narrowing. Identical operands, divergent failure species, selected by the call site's checked context; range-invariant owners get accidental protection, full-range owners get silent modular arithmetic.
- `OperatorsGeneration.DefaultWithKeyTypeOverloads` adds mixed-operand arithmetic in both directions (`owner op key`, `key op owner`), each returning the owner and each re-validating ‚Äî ergonomic at call sites, but it reopens raw-key arithmetic adjacent to the owner. Mixed-operand bodies null-guard their two operands asymmetrically and structurally: each operand's guard is computed independently from that operand's own nullability, so a class owner with a non-nullable struct key produces a mixed operator that throws on a null owner and can never throw on the key ‚Äî the asymmetry is per-operand origin, not a single owner-side rule.

[ALGEBRAIC_CEILING]:
- No identity-element or aggregate numeric interface is ever synthesized ‚Äî no additive or multiplicative identity, no min-max carrier, no full number interface ‚Äî because an identity element is a constant minted outside admission, an unvalidated owner value the generator refuses to manufacture. The owner's advertised structure under each operator is a magma, never a monoid: the unit is unrepresentable without bypassing the factory.
- Every seed, zero, or bound a generic kernel needs therefore enters as an admitted parameter; an algorithm written against a full-number constraint excludes owners by construction, and relaxing it to the operator families is the owner-compatible spelling.

[CONVERSION_OPERATOR_MATRIX]:
- The generated operator matrix encodes three risk tiers, each with its own attribute property: safe owner‚Üíkey conversion (`ConversionToKeyMemberType`, default implicit); unsafe owner‚Üíkey conversion that can throw (`UnsafeConversionToKeyMemberType`, default explicit ‚Äî a class owner with a struct key dereferences a null owner); and key‚Üíowner conversion (`ConversionFromKeyMemberType`, default explicit), which routes through `Create` and throws `ValidationException` on bad input.
- The key‚Üíowner cast is admission-preserving but exception-shaped: `(Tag)"<value-a>"` is a hidden `Create` call. Setting it implicit manufactures an implicit conversion that throws ‚Äî the most hazardous configuration on the surface; it also cannot exist when factories are skipped, because the operator body is the factory.
- The key‚Üíowner conversion on a class owner with a reference-type key null-propagates before admission: a null key casts to a null owner with no factory call and no error, even when the null-yield factory modes are off. The cast is not a strict `Create` alias ‚Äî it is `Create` for non-null inputs and identity-null otherwise; a pipeline that admits via cast accepts an absence the factory would reject.
- The operator family encodes four distinct null contracts on one class owner: equality operators are null-tolerant (`null == null` is true); comparison operators throw `ArgumentNullException` on either operand; arithmetic operators likewise throw; the unsafe owner‚Üíkey conversion throws `NullReferenceException` deliberately. Code that treats `<` as null-safe because `==` is will fault at runtime.
- `IConvertible<TKey>.ToValue()` is an explicit interface implementation ‚Äî the raw key stays off the owner's IntelliSense surface while remaining reachable for generic boundary code constrained on the interface. Interface-typed and `object`-typed keys suppress the entire conversion-operator matrix in both directions; identity escape for such owners is only the explicit-interface `ToValue()`.

[PARSING_FORMATTING_SURFACES]:
- `IParsable<T>`/`ISpanParsable<T>` generation composes two static-abstract hops: parse the key via the key type's own parsable implementation (threading the `IFormatProvider`), then validate via `IObjectFactory`. The exception taxonomy splits by entry point: `Parse` throws `FormatException` (string-flattened error), `Create` throws `ValidationException`, `TryParse` discards the typed error entirely, and only `Validate`/`TryCreate` preserve it.
- `IFormattable` and `ToString()` delegate to the key; complex owners interpolate equality members as `{ A = ..., B = ... }` ‚Äî opt-in equality membership also edits the diagnostic string.
- `SkipIFormattable`, `SkipToString`, `SkipIComparable`, `SkipIParsable` (which forces `SkipISpanParsable`) prune surfaces individually; `SkipFactoryMethods` is the master cut: it kills factories, type converter, object-factory interface, key‚Üíowner conversion, parsing, and forces all arithmetic to none. The surviving surface is principled ‚Äî equality, comparison, formatting, and owner‚Üíkey egress conversions remain because none depends on a factory; everything that constructs from the key is gone.
- The single generic `TypeConverter` is transitive through the key's own converter in both directions: any source type the key's converter accepts becomes admissible, and any destination it can produce becomes an egress, with validation applied at the key‚Üíowner hop. Converter-route admission failures throw `FormatException` carrying the flattened error text ‚Äî not `ValidationException` ‚Äî and converting null to a struct owner throws `NotSupportedException`. The converter attribute is not emitted for generic owners or owners nested inside generic types: component-model binding and designer support silently see no converter on generic owners while the rest of the generated surface is intact.

[STRUCT_ZERO_INIT_GATES]:
- Struct owners implement `IDisallowDefaultValue` by default, and the analyzer escalates zero-init to a compile error (TTRESG047) at `default` and parameterless-`new` sites ‚Äî the admission bypass via zero-init is closed at declaration sites. The gate has holes: array allocation then indexing, generic `default(T)` in unconstrained code, and field zero-init all compile cleanly and materialize an unvalidated instance ‚Äî a struct owner whose invariant excludes the zero value must still tolerate observing it from those routes or remain a class.
- The gate extends to optional parameters: `= default` on a parameter of a disallow-default owner is the same compile error, and the canonical static instance is not a constant expression, so it cannot be a parameter default ‚Äî optionality enters as an `Option<T>` parameter resolved against the canonical instance, or as overloads.
- The gate escapes the owner: any embedding type declaring an instance field or settable property whose type is a default-disallowing struct owner draws the must-be-`required` diagnostic (TTRESG104) unless the member is initialized inline, read-only, static, less accessible than its container, or already `required` ‚Äî the generated admission discipline disciplines every composition site in the graph.
- `AllowDefaultStructs = true` legalizes zero-init and emits a canonical instance as a `public static readonly` field named by `DefaultInstancePropertyName` (default `Empty`). The analyzer forbids the setting in two sound cases: a struct owner with a reference-type key (default would hold a null key behind a non-nullable member) and a complex struct owner containing members that themselves disallow defaults ‚Äî zero-init permission cannot be granted outward when an inner owner denies it.
- The generator computes disallowance independently of the requested permission: a complex struct owner with `AllowDefaultStructs = true` still emits the `IDisallowDefaultValue` marker when any member's type denies defaults ‚Äî the computed state overrides the attribute setting, so downstream embedding diagnostics key off the computed truth, never the requested one.

[NULL_YIELD_MODES]:
- `NullInFactoryMethodsYieldsNull = true` converts null input from rejection into success-with-null: `Validate` returns a null error and a null instance, `Create` becomes nullable-returning, and `TryCreate` returns true with a null out-value ‚Äî yet the `NotNullWhen(true)` annotation on the out-parameter is retained, producing an unsound annotation: flow analysis trusts that a `true` return implies a non-null out-value, masking the reachable null at every call site.
- `EmptyStringInFactoryMethodsYieldsNull = true` extends the same null-yield behavior to empty and whitespace strings and force-enables the null mode; the generator removes `NotNullWhen(true)` only under this empty-string mode. Both settings are inert for struct owners, struct keys, or skipped factories.
- Any single-projection consumer of `Validate` must handle the (null error, null instance) quadrant explicitly ‚Äî `obj!` on the success path is wrong under these modes; the honest projection target is an option-of-owner with absence decided at the boundary, never propagated as null inward.

[ALTERNATE_ADMISSION_FACTORIES]:
- `[ObjectFactory<TValue>]` adds a second admission key type and demands a hand-written `static Validate(TValue value, IFormatProvider? provider, out T item)` returning the owner's error type ‚Äî the custom factory joins the triad grammar rather than inventing a parallel one. `IObjectFactory<TOwner,TValue,TError>` is woven unconditionally, but `IConvertible<TValue>` ‚Äî and the `ToValue()` egress obligation ‚Äî only when the factory participates in serialization or persistence: a parse-only factory is one-way by type system, not just by convention. Both obligations are enforced as errors with exact shapes.
- Skipping factory methods while declaring a string object factory composes into a fully custom admission grammar on a generated chassis: the key-typed triad, component-model converter, key‚Üíowner cast, and key-typed parsing all vanish; the hand-written `Validate` over the foreign shape becomes the sole admission; and the object-factory weave regenerates `IParsable` over the custom grammar, with `Parse`/`TryParse` dispatching into the hand-written `Validate` through the static-abstract invoker ‚Äî host parameter binding via the parsable interface is restored after the master cut removed its key-typed form.

```csharp
[ValueObject<long>(SkipFactoryMethods = true)]
[ObjectFactory<string>(UseForSerialization = SerializationFrameworks.All)]
public partial class Ticket
{
   public static ValidationError? Validate(string? value, IFormatProvider? provider, out Ticket? item)
   {
      item = long.TryParse(value, provider, out var parsed) && parsed > 0 ? new Ticket(parsed) : null;
      return item is null ? ValidationError.Create("<message-a>") : null;
   }

   public string ToValue() => _value.ToString(System.Globalization.CultureInfo.InvariantCulture);
}
```

- A string object factory on a complex owner unlocks the parse surface: `Parse`/`TryParse` route through the static-abstract `Validate` over `string`, so a multi-member owner gains string-shaped admission ‚Äî and with it host parameter binding ‚Äî while keeping its member-wise factory triad intact.
- The factory value type admits `ref struct` arguments end-to-end (`allows ref struct` on both the attribute and the factory interface): a `ReadOnlySpan<char>` factory gives allocation-free wire admission where the deserializer transcodes UTF-8 directly to a char span and calls the span `Validate`; span pattern matching against known literals then admits hot values with zero allocation. Only the span-capable serialization gate supports it ‚Äî declaring it for other frameworks buys nothing.
- `HasCorrespondingConstructor = true` is a persistence-trust declaration: a single-parameter constructor of the factory value type that persistence materialization uses directly, bypassing `Validate` on the premise that the store holds already-admitted truth. Wire deserialization and model binding ignore the flag and always route through `Validate` ‚Äî the trust boundary is persistence-only by design.
- Multiple object factories partition responsibilities under analyzer enforcement: at most one may own persistence materialization, at most one model binding, and serialization frameworks may not overlap across factories ‚Äî each wire concern resolves to exactly one admission route. Conversion-metadata resolution selects the last declared object factory satisfying a custom filter ‚Äî attribute declaration order is a semantic input wherever more than one factory can match; only the serialization, persistence, and binding claims are analyzer-deduplicated.
- The serialization-framework gate is a closed three-bit flag set ‚Äî two JSON families and one binary format, with a combined-JSON alias and an all-mask ‚Äî materialized verbatim into each factory metadata row, so runtime integrations resolve the wire partition by flag intersection on metadata instead of re-reading attributes; any framework outside the vocabulary enters only through an object factory or the component-model converter, never through the flags. Converter code materializes only when the corresponding integration assembly is present ‚Äî on a runtime without an integration package the flags are dormant metadata; the always-present seam is the generated `TypeConverter` registration, which exists unless factories are skipped.

[WIRE_MATERIALIZATION_HAZARDS]:
- Factory-routed deserialization re-validates and re-normalizes; constructor-routed materialization runs only the constructor hook. Values normalized into storage by the factory and read back via the constructor path bypass current normalization ‚Äî normalization changes are data migrations, not code edits.
- Without the converter-generating integration assembly, reflection-based wire serialization sees a keyed class owner with the default private key field as an empty object ‚Äî `{}` out, data silently gone ‚Äî and cannot rehydrate it through the private constructor, failing at deserialization time. The struct variant is worse than failure: reflection deserialization of a struct owner zero-initializes and sets nothing, producing exactly the unvalidated default instance the compile-time gate exists to forbid ‚Äî admission bypass by the serializer, with no exception and no diagnostic. Wire exposure of any owner is a hard dependency on its converter gate, not an optimization.

[METADATA_DISPATCH]:
- Every generated owner publishes a static metadata record through an internal interface: key type, validation-error type, key-extraction delegates and expression-tree forms, a throwing from-key delegate (via `Create`), a non-throwing weakly-typed `TryGetFromKey` (via `Validate`), and a from-key-via-constructor expression that bypasses factory validation by design, for persistence materialization of already-admitted values.
- Complex owners reify their member set as `MemberInfo` lists extracted from an anonymous-type projection expression at static-init time ‚Äî frameworks bind members without reflection-over-attributes scans, and the member list is the full assignable set, not the equality-filtered set: equality-ignored members appear in metadata but not in `Equals`/`GetHashCode`/`ToString`.
- The metadata family is a closed variant set with exhaustive `Switch`/`Map` over the shape kinds (keyed value object, complex value object, and sibling generated shapes), including state-threaded overloads taking a context argument ‚Äî boundary code dispatches on the kind of generated owner a type is without type-testing the owner itself.
- Metadata registration is pull, not push: no module initializer. Lookup reflects the explicit-interface static metadata property on first touch, walks base types, unwraps nullable-of-owner, and memoizes per type in a process-wide concurrent map ‚Äî cold-start dispatch pays one reflection probe per type, steady-state a dictionary hit; constructed generic owners each materialize their own metadata instance, so per-closed-type lookup is automatic.
- The weakly-typed key bridge does not fail fast on foreign key types: it pattern-matches the boxed key and falls through to `default` on mismatch ‚Äî for a struct-keyed owner a wrong-typed key degenerates into validating the zero value, which a permissive invariant will admit; for a class-keyed owner it degenerates into the null path. On the weak route, type mismatch is indistinguishable from zero-value admission; strongly-typed dispatch through the factory interface is the only route that makes mismatch unrepresentable.
- Factory-skipped owners keep their keyed metadata but null out the factory-routed delegates (from-key delegate, expression, weak try-get) while retaining the constructor-route expression ‚Äî persistence materialization survives the master cut by design. Metadata consumers selecting a conversion route must filter on delegate presence, not on metadata kind.
- Internal-surface policing keys on the internal namespace or the internal marker attribute at warning severity, with a per-type suppression on every generated owner ‚Äî reuse of the internal static-abstract `Validate` invokers outside generated code costs one auditable warning per use site.
