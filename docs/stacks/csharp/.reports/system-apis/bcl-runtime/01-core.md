# BCL Runtime Replacement Surfaces

[LOCK]:
- `System.Threading.Lock` is a purpose-built monitor type; `lock (obj)` where `obj` is a `Lock` instance compiles to `Lock.EnterScope()` rather than `Monitor.Enter/Exit`. This is a C# 13 language lowering rule — no attribute marks the type; the compiler pattern-matches `EnterScope()` returning a `ref struct` with `Dispose()`. `object` locks do not benefit from this path.
- `Lock.EnterScope()` returns `Lock.Scope`, a `ref struct` implementing `Dispose()` — the `using var` pattern is the preferred non-keyword form when scope boundaries must be explicit. `Lock.IsHeldByCurrentThread` enables fast re-entrancy checks without acquiring.
- `Lock` has no implicit `IDisposable` and carries no interfaces; it cannot be used as a `SyncRoot` or boxed. Code that passes `this` or a field as the lock object to `Monitor.Enter` must be rewritten; `Lock` fields are not compatible drop-in replacements — the `lock` keyword must target the `Lock` instance directly.
- `TryEnter(TimeSpan)` and `TryEnter(int millisecondsTimeout)` overloads exist; both return `bool` and do not return a `Scope`. When the timeout path must also release via `using`, use `Enter()` + `Exit()` manually with the timeout result.

[TIMEPROVIDER]:
- `TimeProvider` is abstract with two virtual override points: `GetUtcNow()` and `GetTimestamp()`. `GetLocalNow()`, `GetElapsedTime(long)`, and `GetElapsedTime(long, long)` are concrete and derive from those two. A test double only needs to override `GetUtcNow` and `GetTimestamp` to control all derived time surfaces.
- `TimeProvider.CreateTimer(TimerCallback, object, TimeSpan, TimeSpan)` returns `ITimer` (not `Timer`). `ITimer` exposes only `Change(TimeSpan, TimeSpan)`. A test `TimeProvider` can return a fake `ITimer` and advance time by invoking the callback directly, giving deterministic timer tests without wall-clock coupling.
- `Task.Delay(TimeSpan, TimeProvider)` and `Task.Delay(TimeSpan, TimeProvider, CancellationToken)` accept a `TimeProvider`, as do `Task.WaitAsync(TimeSpan, TimeProvider)` and `Task.WaitAsync(TimeSpan, TimeProvider, CancellationToken)`. `PeriodicTimer(TimeSpan, TimeProvider)` and `CancellationTokenSource(TimeSpan, TimeProvider)` complete the injectable surface: every time-dependent async primitive is testable without real wall-clock delay.
- `TimeProvider.GetElapsedTime(long startingTimestamp)` calls `GetTimestamp()` internally for the end timestamp — one-arg form is not simply `GetElapsedTime(start, GetTimestamp())`; it reads the clock at call time, which matters for accuracy under fake providers that advance monotonically.
- `TimeProvider.TimestampFrequency` is the ticks-per-second for `GetTimestamp()` output. A fake provider must return a consistent `TimestampFrequency` to make `GetElapsedTime` arithmetic correct; returning 0 will produce division-by-zero in `GetElapsedTime`.

[THROW_HELPERS]:
- The complete static throw-helper surface: `ArgumentNullException.ThrowIfNull(object, [CallerArgumentExpression] string?)`, `ArgumentException.ThrowIfNullOrEmpty(string, [CAE] string?)` and `ThrowIfNullOrWhiteSpace`, `ArgumentOutOfRangeException.ThrowIfZero/ThrowIfNegative/ThrowIfNegativeOrZero/ThrowIfEqual/ThrowIfNotEqual/ThrowIfGreaterThan/ThrowIfGreaterThanOrEqual/ThrowIfLessThan/ThrowIfLessThanOrEqual` (all generic over `IComparable<T>`, all `[CAE]` on `paramName`), `ObjectDisposedException.ThrowIf(bool, object)` and `ThrowIf(bool, Type)`.
- `CallerArgumentExpressionAttribute` on `paramName` means the argument expression string is captured automatically at compile time when `paramName` is omitted. The pattern `ArgumentNullException.ThrowIfNull(value)` emits the parameter name `"value"` into the exception without a string literal. Guard-block reimplementations — `if (x is null) throw new ArgumentNullException(nameof(x))` — are the deleted form.
- `ObjectDisposedException.ThrowIf(bool, object instance)` accepts any object and uses `instance.GetType().FullName` as the type name in the message; the `Type` overload accepts a closed generic type and renders its name. Neither overload has a `[CallerArgumentExpression]` — the condition expression is not captured.
- `ArgumentOutOfRangeException` throw helpers are constrained to `IComparable<T>` (not `INumber<T>`), so they work for any comparable value type and string. The `ThrowIfEqual`/`ThrowIfNotEqual` family is the only symmetric pair; all others compare against a named `other` parameter that is not `[CAE]`-decorated, meaning the bound value does not capture as an expression.

[RANDOM]:
- `Random.Shared` is a thread-safe `Random` instance backed by a per-thread `Xoshiro256**` PRNG with thread-local state — no lock on read. It is not cryptographically random; use `RandomNumberGenerator` static methods for security-sensitive material.
- `Random.GetString(ReadOnlySpan<char> choices, int length)` allocates and returns a new `string`. `Random.GetHexString(int stringLength, bool lowercase)` and its `Span<char>` sink overload are available on both `Random.Shared` and instance `Random`. These replace hand-rolled character selection loops.
- `Random.GetItems<T>(ReadOnlySpan<T> choices, int length)` allocates `T[]`. The `Span<T>` destination overload `GetItems<T>(ReadOnlySpan<T> choices, Span<T> destination)` writes into a caller-owned buffer for zero allocation. `Random.Shuffle<T>(Span<T>)` and `Shuffle<T>(T[])` perform Fisher-Yates in place.
- `RandomNumberGenerator` mirrors the same static surface — `GetItems`, `GetString`, `GetHexString`, `Shuffle`, `Fill`, `GetBytes`, `GetInt32` — making the selection between pseudo-random and cryptographic independent of call-site shape; the only change is the owning type.

[CANCELLATION]:
- `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` avoids the `CancellationToken[]` allocation path for the variable-arity case. The two-token overload `(token1, token2)` and single-token overload `(token)` are optimized fast paths; the span overload is the correct form for three or more tokens.
- `CancellationTokenSource.TryReset()` resets a non-cancelled source to its initial state, allowing reuse across periodic operations without allocation. Returns `false` if cancellation was already requested; returns `true` only when the source is in a clean state after the reset.
- `CancellationTokenSource.CancelAsync()` schedules cancellation asynchronously and returns a `Task`. Unlike `Cancel()`, callbacks registered via `Register` are invoked on the thread pool rather than inline on the calling thread — avoids re-entrancy when callbacks dispatch further async work.
- `CancellationToken.UnsafeRegister(Action<object?>, object?)` registers a callback without flowing `ExecutionContext`. Where callbacks are known-safe closures with no ambient context dependency (no async locals, no security context), this form eliminates the context capture allocation that `Register` incurs. The `Action<object?, CancellationToken>` two-argument overload on `UnsafeRegister` passes the token itself to the callback, removing the need to close over it.
- `CancellationToken.ThrowIfCancellationRequested()` is the canonical inline check; it throws `OperationCanceledException` carrying the token. It does not allocate an exception object if cancellation is not requested; `IsCancellationRequested` avoids even the throw-path overhead for tight polling loops where cancellation is rare.

[PERIODIC_TIMER]:
- `PeriodicTimer.WaitForNextTickAsync(CancellationToken)` returns `ValueTask<bool>` — `false` signals disposal; `true` means tick fired. The loop idiom is `while (await timer.WaitForNextTickAsync(ct))`. There is no callback, no reentrancy risk, and no thread affinity.
- `PeriodicTimer.Period` is a readable and writable `TimeSpan` property. Changing it mid-loop takes effect on the next tick's scheduling. Timer callbacks that need backpressure-sensitive intervals can write `Period` from inside the loop body.
- `PeriodicTimer(TimeSpan period, TimeProvider timeProvider)` accepts a `TimeProvider` — the timer advances with the provider's clock, enabling deterministic interval tests without sleeping.
- `PeriodicTimer` does not skip ticks; if processing takes longer than the period, the next tick fires immediately upon return from `WaitForNextTickAsync`. Callers that need skip-on-overrun behavior must compare elapsed time against the period inside the loop body.

[TASK_COMBINATION]:
- `Task.WhenEach(Task[])`, `Task.WhenEach(ReadOnlySpan<Task>)`, and `Task.WhenEach(IEnumerable<Task>)` return `IAsyncEnumerable<Task>` — tasks complete in arrival order, not input order. Generic overloads return `IAsyncEnumerable<Task<T>>`. This replaces `Task.WhenAny` retry loops that discard and rebuild task collections on each iteration.
- `Task.WhenAll` and `Task.WhenAny` both now have `ReadOnlySpan<Task>` overloads avoiding array allocation when the task set is stack-or-pool-allocated. The generic `Task.WhenAll<T>(ReadOnlySpan<Task<T>>)` returns `Task<T[]>` — same semantics, no heap allocation for the input sequence.
- `Task.WaitAsync(CancellationToken)`, `WaitAsync(TimeSpan)`, `WaitAsync(TimeSpan, TimeProvider)`, and `WaitAsync(TimeSpan, TimeProvider, CancellationToken)` wrap any task with a timeout or cancellation boundary without blocking. The returned task faults with `TimeoutException` on expiry, cancels with the token, or completes with the original task's result. The original task continues running — this is not a cancellation of the underlying work.
- `Task.Delay(TimeSpan, TimeProvider)` and its `CancellationToken` variant fully decouple delay scheduling from the system clock, needed for test determinism.

[INTERLOCKED_VOLATILE]:
- `Interlocked.CompareExchange` and `Exchange` now cover `byte`, `sbyte`, `short`, `ushort`, `uint`, `ulong`, `UInt16`, `UInt32`, `UInt64` in addition to the traditional `int`/`long`/`float`/`double`/`T`. CAS loops on narrow integer state fields no longer require widening to `int`.
- `Interlocked.And` and `Or` exist for `int`, `long`, `uint`, `ulong` — atomic bitwise operations without a load-mask-store sequence. `Interlocked.Increment` and `Decrement` cover `uint` and `ulong` as well as `int` and `long`.
- `Volatile.ReadBarrier()` and `Volatile.WriteBarrier()` issue one-directional memory fences directly, cheaper than `Thread.MemoryBarrier()` which issues a full two-way fence. Producers that write data then publish a flag can use `WriteBarrier` after the data stores and before the flag write; consumers can use `ReadBarrier` after reading the flag and before reading the data.
- `Interlocked.MemoryBarrierProcessWide()` issues a process-wide fence that makes all writes visible to all threads and processors — necessary after pointer-publishing patterns on architectures with weak ordering. Stronger and more expensive than `MemoryBarrier`.

[ARRAY_AND_MEMORY_POOLING]:
- `ArrayPool<T>.Shared.Rent(int minimumLength)` returns an array of at least `minimumLength` elements but potentially larger (bucket-aligned). `Return(T[], bool clearArray)` must be called with the same array reference; `clearArray: true` required when the array held sensitive or reference-typed data to prevent leaks across rentals.
- `MemoryPool<T>.Shared.Rent(int minBufferSize)` returns `IMemoryOwner<T>` whose `Memory<T>` property gives the usable segment. The segment length may exceed `minBufferSize`. `IMemoryOwner<T>` is `IDisposable` — ownership is explicit and lifetime-scoped, unlike bare `ArrayPool<T>` which requires manual pairing of `Rent`/`Return`.
- `GC.AllocateUninitializedArray<T>(int length, bool pinned)` bypasses zero-initialization for blittable element types — semantically equivalent to `stackalloc` for heap allocation. The `pinned` flag allocates in the pinned object heap (POH), eliminating GC pressure from repeated pinning. Use only when the entire array is written before any read; uninitialized reads are undefined behavior.
- `GC.AllocateArray<T>(int length, bool pinned)` zero-initializes and optionally pins. The POH path avoids the GC relocation cost for long-lived buffers used with native interop.

[LAZY_INITIALIZATION]:
- `Lazy<T>(LazyThreadSafetyMode mode)` with `LazyThreadSafetyMode.PublicationOnly` allows concurrent initialization races — multiple threads may construct `T` but only one wins; losers are discarded. Use when construction is idempotent, cheap, and racing threads should not block.
- `LazyThreadSafetyMode.ExecutionAndPublication` (the default `isThreadSafe: true` mode) blocks competing threads behind a `Monitor` until the winner completes initialization. If the factory throws, the exception is cached and re-thrown on every subsequent access — the `Lazy<T>` is permanently faulted. `PublicationOnly` mode does not cache exceptions; a faulted attempt allows the next thread to try again.
- `LazyThreadSafetyMode.None` provides no synchronization — suitable only when the `Lazy<T>` is guaranteed never to be accessed concurrently (e.g., per-instance field accessed only from the owning thread). It is the highest-performance mode and should be the default when the owning type already guarantees single-threaded access.

[GC_CONTROL]:
- `GC.RefreshMemoryLimit()` re-reads the memory limit from the container or OS after a live cgroup change, without restarting the process. Relevant for containerized workloads where memory limits are adjusted at runtime.
- `GC.GetConfigurationVariables()` returns `IReadOnlyDictionary<string, object>` of active GC configuration keys — heap size, generation limits, LOH threshold, concurrent GC mode. Diagnostic-only; the returned values reflect the live configuration, not what was set at startup if JIT tuning modified them.
- `GC.GetTotalPauseDuration()` returns the cumulative `TimeSpan` spent in GC pauses since process start — useful for SLO monitoring alongside `GC.GetTotalAllocatedBytes`.
- `GC.TryStartNoGCRegion(long totalSize)` suppresses GC for a region of known-bounded allocation. `RegisterNoGCRegionCallback(long size, Action)` triggers a callback when allocation within the no-GC region approaches the limit, allowing graceful fallback rather than forced collection.

[THREAD_POOL]:
- `ThreadPool.QueueUserWorkItem<TState>(Action<TState>, TState, bool preferLocal)` is allocation-free — the generic state parameter avoids boxing compared to the `WaitCallback`/`object` form. `preferLocal: true` queues to the calling thread's local work-stealing queue, reducing contention for work that has locality with the producer.
- `ThreadPool.UnsafeQueueUserWorkItem<TState>(Action<TState>, TState, bool preferLocal)` and `UnsafeQueueUserWorkItem(IThreadPoolWorkItem, bool preferLocal)` suppress `ExecutionContext` flow. The `IThreadPoolWorkItem` form allows custom work items with zero delegate overhead — implement `Execute()` on a struct to avoid heap allocation entirely.
- `Thread.UnsafeStart()` and `Thread.UnsafeStart(object parameter)` start a thread without flowing `ExecutionContext` — appropriate when the thread establishes its own context and ambient values from the creating thread must not leak (e.g., async locals carrying request context).
