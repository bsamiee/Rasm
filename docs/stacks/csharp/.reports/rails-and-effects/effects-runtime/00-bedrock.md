# Effects-Runtime: Core

[CARRIER_HIERARCHY]:
- `IO<A>` is the bottom carrier — `abstract record` implementing `K<IO, A>`, `Monoid<IO<A>>`, `Semigroup<IO<A>>`. Every execution path bottoms out here; `Eff<A>` and `Eff<RT, A>` are readers over `IO`.
- `Eff<A>` is `record Eff<A>(Eff<MinRT, A> effect)` where `MinRT` is a zero-byte `readonly struct`. It implements `Readable<Eff<A>, A>` via `Deriving.Readable<Eff<A>, A, ReaderT<A, IO>>` — the type parameter `A` itself is the reader environment, so `ask<A>()` projects the bound type, not a capability record. Use `Eff<A>` only when the computation needs no capability beyond cancellation and resource tracking.
- `Eff<RT, A>` is `record Eff<RT, A>(ReaderT<RT, IO, A> effect)` — a `ReaderT` over `IO` named distinctly for ergonomics. The `RT` type carries capability slots via `Has<Eff<RT>, Cap>` implementations; the computation reads them via `asks<RT, Cap>()` or `Has<Eff<RT>, RT, Cap>.ask`.
- `Eff` (non-generic) is the higher-kinded type-class implementing `Readable<Eff, MinRT>`, `Natural<Eff, Eff<MinRT>>`, and all monad/applicative/fallible traits. It is the constraint surface for carrier-polymorphic arrows: `K<Eff, A>` abstracts over `Eff<A>` and `Eff<MinRT, A>` simultaneously via `Natural` conversions.

[IO_DSL_INTERPRETER]:
- The runtime is a trampoline, not a call stack. `IO<A>.RunAsync()` opens a synchronous loop over a `Seq<StackItem>` catch-handler stack; each iteration pattern-matches the current `IO<A>` node against `InvokeSync<A>`, `InvokeSyncIO<A>`, `InvokeAsyncIO<A>`, `IOCatch<A>`, `IOCatchPop<A>`, and `IOTail<A>`. Only when the node is `InvokeAsync<A>` — carrying an unfinished `ValueTask` — does the loop yield to `RunAsyncInternal`, which is an `async`/`await` state machine. This two-phase design avoids the state-machine allocation for fully-synchronous computation graphs.
- `IOTail<A>` is the trampoline step: it holds the next `IO<A>` and overrides `Map` and `Bind` with `NotSupportedException`. When the loop encounters `IOTail<A>` it replaces the current node with `iOTail.Tail` and continues without advancing a continuation; `.Map` and `.Bind` on a tail node throw at definition time, not at run time, making illegal chaining a build error, not a runtime surprise.
- `tail<A>(IO<A>)` from `Prelude` is the public constructor for `IOTail<A>`. `IOTail<A>.resolve<B,C>` is the select-partner: it unwraps a chained `IOTail<B>` into its `Tail` for the `B == C` case, and falls back to `Map` otherwise. The constraint `typeof(B) == typeof(C)` is a runtime equality check at the resolve site.
- `IOCatch<X,A>` carries `Operation`, `Predicate`, `Failure`, `Final?`, and `Next` as record fields. The loop pushes a `StackItem(MakeHandler())` onto the catch stack before running `MakeOperation()`; `MakeOperation()` inserts an `IOCatchPop` node after the success path; `MakeHandler()` constructs the failure branch that checks `Predicate` and routes to `Failure`. When the `Final` field is set, it runs unconditionally via `Finally` on both paths — the four-argument `Bracket(Use, Catch, Fin)` form assembles this in a single `Bind` expression rather than nesting two separate `IOCatch` nodes. Avoid allocating these nodes per-call in tight loops; each `IOCatch` node pushes one `StackItem` onto the catch stack at run time.
- Extending `IO<A>` by subclassing directly causes `Errors.IODSLExtension` to be thrown at run time; the four legal extension base types are `InvokeSync<A>`, `InvokeAsyncIO<A>`, `InvokeSyncIO<A>`, and `InvokeAsync<A>`. `InvokeSyncIO<A>` is the only synchronous base whose `Invoke` returns `IO<A>` rather than `A` — it enables two-phase dispatch where phase selection is synchronous and the body is an `IO` chain. `InvokeAsyncIO<A>` is the async counterpart: `Invoke` returns `ValueTask<IO<A>>`. `InvokeSync<A>` returns the raw `A`, terminating that step.

[LIFT_FORMS]:
- `Eff.lift<A>(Func<A>)`, `Eff.lift<A>(Func<Fin<A>>)`, `Eff.lift<A>(Func<Either<Error,A>>)` — pure synchronous lifts; the `Fin`/`Either` overloads admit pre-computed failure.
- `Eff.lift<A>(Func<EnvIO, A>)` and `Eff.lift<A>(Func<EnvIO, Fin<A>>)` — synchronous lifts with `EnvIO` access; implemented as `+Prelude.envIO.Bind(e => Eff<A>.Lift(() => f(e)))`. The unary `+` on `K<Eff, A>` is the downcast operator, not a custom combinator.
- `Eff.lift<A>(Func<Task<A>>)`, `Eff.lift<A>(Func<Task<Fin<A>>>)`, `Eff.lift<A>(Func<EnvIO, Task<A>>)` — async lifts; `Task<Fin<A>>` unwraps via `.ThrowIfFail()` before storage, so failures surface at run-time, not at lift time.
- `Eff.lift<A>(IO<A>)` — admits an already-constructed `IO` monad directly into the `Eff` carrier; the canonical bridge between the two layers.
- `Prelude.liftEff` mirrors all static `Eff.lift` overloads; prefer the static form on `Eff<A>` for explicitness at call sites.
- `IO.lift<A>(Func<EnvIO, A>)` carries the `EnvIO` token for cancellation polling; `IO.liftAsync<A>(Func<EnvIO, Task<A>>)` and `IO.liftVAsync<A>(Func<EnvIO, ValueTask<A>>)` are the async equivalents. `IO.env` is the built-in effect `IO<EnvIO>` for accessing the context without manual lift construction.
- `Eff<A>` carries nine implicit conversions: from `Pure<A>`, `Fail<Error>`, `Lift<A>`, `Lift<Fin<A>>`, `Lift<MinRT,A>`, `Lift<MinRT,Fin<A>>`, `Either<Error,A>`, `Fin<A>`, and `IO<A>`. Every already-computed carrier value becomes an `Eff<A>` without an explicit lift call; the `Fin<A>` implicit is load-bearing at boundary seams where a validation rail rejoins the effect rail. `Eff<RT,A>` adds one further implicit from `Eff<A>`, widening the no-RT carrier to the RT form — the single direction of implicit widening matches the type-parameter direction.
- `IO<A>` is not implicitly convertible to `Eff<A>` at the type level; `Eff<A>.op_Implicit(IO<A>)` is the one-way bridge. When an `IO<A>` value must participate in a `K<Eff, A>` context without explicit wrapping, use `Eff.lift(io)` — the static form is canonical. The `Lift<A>` implicit on `Eff<A>` accepts a `Lift<A>` ref-struct: an inline `Lift(() => expr)` value cannot be stored in a variable before passing — it must be constructed at the conversion site.

[MONAD_TRANSFORMER_IO_LIFT]:
- `MonadIO<M>.LiftIO<A>(IO<A> ma)` is `static abstract` — every concrete `MonadIO` type must implement it. The default `LiftIO(K<IO,A>)` overload delegates to `LiftIO(IO<A>)` via `.As()`. For transformer stacks such as `OptionT<IO, A>`, `FinT<IO, A>`, and `EitherT<L, IO, A>`, `LiftIO` wraps the `IO` computation in the transformer's inner layer.
- `IO<A>.Try()` returns `FinT<IO, A>` — it maps `Fin.Succ` over the result and catches via `Fin.Fail<A>`. The `FinT<IO, A>` wrapper is a lifted `IO<Fin<A>>`; running the inner `IO` returns `Fin<A>` directly without an outer `Fin` wrapping. Use `io.Try()` when the computation must run inside a transformer stack that accumulates outcomes without re-wrapping failure.
- `IO.SelectMany` extension block defines overloads from `K<IO,A>` into `OptionT<M,B>`, `EitherT<L,M,B>`, `FinT<M,B>`, and `ValidationT<F,M,B>` via `OptionT.liftIO<M,A>(ma).SelectMany(...)`. This enables mixed-carrier query expressions where an `IO` step binds directly into a transformer query without an explicit `.LiftIO` call.

[MAPIO_AND_TOID_USES]:
- `mapIO<M,A,B>(K<M,A> ma, Func<IO<A>, IO<B>> f)` where `M : MonadUnliftIO<M>` is the canonical slot for injecting raw `IO` transformations into any carrier. Every `Eff` or transformer that implements `MonadUnliftIO` exposes its `IO<A>` layer via `ToIO`, applies `f`, and re-wraps. This is how `uninterruptible`, `timeout`, `fork`, `localIO`, and `postIO` are all carrier-polymorphic: each calls `MapIO` with the appropriate `IO<A> → IO<B>` wrapper.
- `localIO<M,A>(K<M,A> ma)` calls `M.LocalIO(ma)` which uses `MapIO(ma, io => io.Local())` — it wraps the operation in a fresh cancel scope. `postIO<M,A>(K<M,A> ma)` calls `M.PostIO(ma)` which uses `MapIO(ma, io => io.Post())` — it marshals execution onto the captured `SynchronizationContext`. Both are expressed purely through `MapIO` so they compose with any `MonadUnliftIO` stack.
- `ToIO<M,A>(K<M,A> ma)` returns `K<M, IO<A>>` — it embeds the `IO<A>` layer as a bound value inside the carrier. This is used in `awaitAny(Seq<K<M,ForkIO<A>>>)` to extract the `IO<ForkIO<A>>` for each fork and run them independently.
- `IO<A>.WithEnv(Func<EnvIO, EnvIO>)` and `IO<A>.WithEnvFail(Func<EnvIO, EnvIO>)` are the low-level env-mapping primitives that `Bracket`, `BracketFail`, `Timeout`, and `Fork` all build on. `WithEnv` releases the mapped resources on both success and failure; `WithEnvFail` releases only on failure and leaks resources back to the parent scope on success. A custom DSL extension that requires a specialized `EnvIO` shape constructs it via `WithEnv`/`WithEnvFail` — never by accessing `EnvIO` fields directly.

[RUN_COLLAPSE]:
- `Eff<A>.Run()` (via `EffExtensions`) — synchronous; returns `Fin<A>`; creates a fresh `EnvIO.New()`. Exceptions from unmanaged boundaries are caught and converted to `Error`.
- `Eff<A>.Run(EnvIO envIO)` — synchronous with caller-supplied environment; allows sharing a cancellation token across composed operations.
- `Eff<A>.RunAsync()` — returns `Task<Fin<A>>`; creates `EnvIO.New()` internally.
- `Eff<A>.RunAsync(EnvIO envIO)` — `Task<Fin<A>>` with explicit env.
- `Eff<A>.RunUnsafe()` / `RunUnsafeAsync()` — omit the `Fin` wrapper; throw on failure. Reserve for composition roots that have already exhausted recovery.
- `Eff<A>.RunIO()` — returns `IO<A>` without executing it; the escape hatch for passing an `Eff` computation into an `IO`-shaped boundary or test harness that runs the `IO` layer directly.
- `Eff<RT, A>.Run(RT env)` / `RunAsync(RT env)` — the RT form passes the runtime record; the `Fin<A>` wrapping and exception capture are identical to the no-RT form.
- `IO<A>.Run()` / `Run(EnvIO)` — synchronous; throws on failure because `IO` does not return `Fin`. When `ValueTask.IsCompleted` is false on the synchronous path, the runtime blocks. Use only when the call site is guaranteed synchronous (e.g., plug-in `OnLoad`). Internally calls `RunAsync()` and blocks — there is no dedicated synchronous execution path in the DSL; synchronous appearance is an optimization for already-completed `ValueTask`.
- `IO<A>.RunAsync()` / `RunAsync(EnvIO)` / `RunAsync(CancellationToken)` — `ValueTask<A>` return; the `CancellationToken` overload allocates `EnvIO.New(null, token)` — fresh resources and a fresh `CancellationTokenSource` linked to the supplied token. Prefer passing an existing `EnvIO` to preserve resource scope and avoid the allocation; the `CancellationToken` overload is for integration points that hand tokens from external callers.

[ENVIO_AND_CONTEXT]:
- `EnvIO.New(resources, token, source, syncContext, timeout)` — all parameters optional. The `source` and `token` are kept separate: `source` is disposable and owns cancellation; `token` is the read-only observable. Passing `source: null` creates a fresh `CancellationTokenSource` internally.
- `EnvIO` tracks two orthogonal ownership bits via the `Own` int: bit `1` for the `CancellationTokenSource` (owned when created internally), bit `2` for the `Resources` object (owned when created internally). `Dispose()` calls `DisposeResources()` then `DisposeNonResources()` in order. Each half uses `Interlocked.CompareExchange` to guarantee at-most-once disposal regardless of concurrent callers.
- `EnvIO.Local` — `New(null, Token, null, SynchronizationContext.Current)`; both source and resources are null so both are created fresh and both ownership bits are set. The new CTS links to the parent token via a registration. The sync context is captured at construction time, not at `Post()` call time. `EnvIO.LocalResources` — `New(null, Token, Source, SyncContext)`; creates fresh resources (bit 2 owned) but reuses the parent source (bit 1 not owned); used by `BracketFail()` so retry scopes share the same cancellation axis.
- `EnvIO.LocalCancel` — inherits `Resources` and `Token`, clears `Source`. Used for cancel-scope isolation without resource isolation.
- `EnvIO.LocalCancelWithTimeout(TimeSpan)` is the mechanism behind `IO<A>.Timeout(TimeSpan)` via `IOTimeout<A,B>`: it creates a new CTS with the timeout, registers the parent token to cancel it, and the `using` on the local `EnvIO` in `IOTimeout.Invoke` disposes both the CTS and the registration once the operation completes.
- `EnvIO.LocalSyncContext` preserves all existing scope fields but re-captures `SynchronizationContext.Current` from the calling thread. Used to re-bind context after crossing a thread boundary before calling `IO.Post()`.
- When `EnvIO.New` receives a non-cancellable parent token, the registration is null and no linkage is established, eliminating the allocation. The `Own` bit for the source is still set, so `DisposeNonResources` cleans up the source regardless.
- `IO.Post()` — marshals execution onto `env.SyncContext` when non-null; no-op when `SyncContext` is null.

[CANCELLATION_TOKEN_ACCESS]:
- `IO.token` is `new IOToken<CancellationToken>(pure)` — a DSL node subclassing `InvokeSyncIO<CancellationToken>` whose `Invoke` calls `Next(env.Token)`. It participates in fusion: `Map` and `Bind` produce `IOTokenMap` and `IOTokenBind` nodes that carry the `Next` continuation, deferring token access until run time. Prefer `IO.token` over `cancelToken` in tight loops where the fusion-node chain can eliminate continuations.
- `IO.source` is `IO.lift((EnvIO e) => e.Source)` — a plain `InvokeSync` lift, not an `IOToken` variant; it does not participate in the token-bind fusion chain. `cancelToken` in `Prelude` is identically `IO.lift(e => e.Token)`, also a plain lift with no fusion path.
- `MonadIO<M>` exposes `Token` and `TokenSource` as `static virtual` properties defaulting to `M.LiftIOMaybe(IO.token)` and `M.LiftIOMaybe(IO.source)`. Any `MonadIO`-constrained generic function can read the token without breaking carrier abstraction.

[RESOURCE_SAFETY]:
- `IO<A>.Bracket()` — `WithEnv(env => env.LocalResources)`; creates a fresh `Resources` scope that disposes on both success and failure. Every `use<A>()` call and `use<A>(acquire, release)` call registers an `IDisposable` or custom release function in `env.Resources`; disposal runs at scope boundary.
- `IO<A>.BracketFail()` — `WithEnvFail(env => env.LocalResources)`; disposes the fresh resource scope only on failure. Used as the atom inside `Retry`: each failed attempt releases its acquired handles while the successful attempt's resources propagate to the enclosing scope.
- `IO<C>.Bracket<B,C>(Func<A, IO<C>> Use, Func<A, IO<B>> Fin)` — the three-argument form is sugar for `Bind(x => Use(x).Finally(Fin(x)))`.
- `IO<C>.Bracket<B,C>(Use, Catch, Fin)` — adds an error recovery path: `Bind(x => Use(x).Catch(Catch).Finally(Fin(x)))`. `Fin` always runs; `Catch` fires only on error.
- `Prelude.use<A>(IO<A> acquire) where A : IDisposable` — lifts `acquire` into the current `Resources` scope; the value is released when the enclosing `Bracket()` scope exits. `use<A>(Func<A> acquire, Func<A, IO<Unit>> release)` accepts a custom release function for non-`IDisposable` handles.
- `Prelude.useAsync<A>` — for `IAsyncDisposable`.
- `IO<A>.Finally<X>(K<IO, X> finally)` — runs `finally` unconditionally after `fa`, regardless of success or failure; result of `finally` is discarded. The `Final<IO>` trait static abstract exposes `Finally` generically, enabling use on `K<IO, A>`. The `|` operator between `K<F,A>` and `Finally<F,X>` is defined in a C# extension block: `(K<F,A> lhs, Finally<F,X> rhs) => F.Finally(lhs, rhs.Operation)`, so `finalEffect | final(cleanup)` is the expression-spine spelling.
- `Resources` holds an `AtomHashMap<object, TrackedResource>` (MVCC-safe). `Acquire<A>` calls `TryAdd` with the handle as the key — if the same object reference is acquired twice, the second call is silently dropped, making acquire idempotent for identity-equal handles. `Release<A>` removes the key and runs the release IO; if not found locally it delegates to `parent`, enabling scope nesting where a child scope can release a handle registered in a parent.
- `IOLocal<X,A>` (the implementation of `IO<A>.Bracket()`) creates a local env via `MapEnvIO`, awaits the operation inside a `using` statement, then disposes the local env on exit — both resource and non-resource cleanup run. `IOLocalOnFailOnly<X,A>` (the implementation of `IO<A>.BracketFail()`) only calls `local.DisposeResources()` in the `catch` block; on the success path, `local.DisposeNonResources()` runs in `finally` but the resource scope is left live, intentionally propagating acquired handles to the enclosing scope.
- `IOUseDisposable<X,A>.Invoke` checks whether the acquire `ValueTask` is already completed: synchronous acquires register into `env.Resources` immediately and return the next node; asynchronous acquires return `IOAcquireDisposableAsync<X,A>`, which registers the handle only after the `ValueTask` resolves. This eliminates the window between acquisition completion and registration where an exception would cause a leak.
- `Resources.DisposeU` creates a fresh CTS before iterating and uses `CancellationToken.None` in the disposal env, so resource release is uninterruptible by design — cancellation of the outer scope does not abort cleanup.
- `IO<A>.Finally<X>` constructs `IOFinal<X,A,A>`. If the operation's `RunAsync` returns a completed task, `Finally` sequences as `Final.Bind(_ => Next(result))` — the finally IO runs before the continuation. If the task is incomplete, `IOFinalAsync` handles the await and similarly sequences finally before continuing. The `Finally` action runs synchronously in the `catch` block if `RunAsync` throws before completing, guaranteeing cleanup on both paths.

[UNINTERRUPTIBLE_SEMANTICS]:
- `IO<A>.Uninterruptible()` constructs `IOUninterruptible<A,A>`. `Invoke` creates a new `EnvIO` with `CancellationToken.None` — the token is completely replaced, not linked. The source is `null` so a fresh CTS is created internally, making the scope unowned from a lifecycle perspective.
- The uninterruptible scope does not isolate resource tracking: it passes `envIO.Resources` (from the parent) into the local env. Resources acquired inside an uninterruptible block register in the parent scope and are released when the parent bracket exits, not when the uninterruptible scope does. Composing `uninterruptible` with `use` requires an enclosing `Bracket()` to own the resource lifetime.
- `uninterruptible<M,A>(K<M,A> ma)` on `MonadUnliftIO<M>` is implemented as `MapIO(ma, io => io.Uninterruptible())`. The uninterruptible wrapping operates at the `IO` layer regardless of the outer carrier.

[TIMEOUT_BEHAVIOR]:
- `IO<A>.Timeout(TimeSpan)` uses `IOTimeout<A,A>`, which calls `Fa.Run(envIO.LocalCancelWithTimeout(limit))`. When the CTS fires, `Run` propagates a `TaskCanceledException` through the trampoline's catch stack — the return type is `IO<A>`, not `IO<Fin<A>>`, so the exception is not wrapped automatically. To model timeout as a typed failure, either follow with `@catch` on the cancellation type or use the carrier-polymorphic `TimeoutIO<M, A>(K<M, A>, TimeSpan)` extension, which delegates to `MapIO(ma, io => io.Timeout(ts))`. The timeout clock starts when `IO` execution begins, not at construction.
- `TimeoutIO<M, A>` being implemented as `MapIO(ma, io => io.Timeout(ts))` makes it carrier-polymorphic via `MonadUnliftIO<M>`: any `Eff<A>`, `Eff<RT, A>`, or transformer stack picks it up without specialization.

[RECOVERY_COMBINATORS]:
- `@catch<M,A>(Func<Error, K<M,A>> Fail)` — unconditional catch; the `@` prefix is mandatory because `catch` is a C# keyword. Returns `CatchM<Error, M, A>`.
- `@catch<M,A>(Func<Error, bool> predicate, Func<Error, K<M,A>> Fail)` — guarded catch; only fires when `predicate` returns true.
- `@catch<M,A>(Error error, Func<Error,K<M,A>> Fail)` / `@catch<M,A>(int errorCode, ...)` — match by equality or numeric code.
- `@catch<M,A>(K<M,A> Fail)` — recovery value, no inspection of the error.
- `catchOf<E, M, A>(Func<E, K<M,A>> Fail) where E : Error` — typed catch; automatically narrows to `E` subtype errors and dispatches, ignoring the rest.
- `expected<M,A>` / `exceptional<M,A>` / `expectedOf<E,M,A>` / `exceptionalOf<E,M,A>` — `expected` targets `Expected` (user-defined, recoverable) errors; `exceptional` targets `Exceptional` (system exception wrappers). Both forward to `catchOf`.
- `CatchM<E, M, A>` is a value — `(Match predicate, Action handler)`. Multiple `CatchM` values compose via the `|` operator on `K<F, A>`: `lhs | @catch(...) | @catch(...)` builds a left-to-right guarded chain; first matching predicate wins.
- `Catch(fa, CatchM)` extension on `K<F, A>` is the method form; `FallibleExtensions` provides all arity variants.
- `IfFail(Func<Error, A>)` / `IfFailEff(Func<Error, Eff<A>>)` — instance members on `Eff<A>` and `Eff<RT,A>`; equivalent to `.Catch(Fail)` and `.Catch(Fail).As()`.
- `FallibleExtensions.CatchIO<M, A>(K<M, A>, Func<Error, bool>, Func<Error, K<IO, A>>)` matches the `@catch` family but lifts the recovery path to `IO` rather than the outer carrier `M`. This is used at the boundary where recovery produces a raw IO effect that does not belong to the outer runtime context; the recovery result is then lifted back into `M` via `MonadIO.liftIO`. The three-overload family covers predicate, Error-equality, and numeric-code matching.
- `guardIO(bool)` returns `IO<Unit>` — on `false` it returns `IO.empty<Unit>()`, which propagates as `Errors.None` through the catch stack. `guard<E>(bool, Func<E>)` returns `Guard<E, Unit>` for generic query expressions over `Fallible<E>` carriers. The two forms are not interchangeable: `guardIO` belongs in `IO`-typed query expressions; `guard<Error>(...)` belongs in `Eff`-typed query expressions that must emit a named typed error on the false branch.
- `succs<M, A>(Seq<K<M, A>>)` runs each effect in the sequence and returns only the successful values; failed effects are silently dropped. `fails<M, A>(Seq<K<M, A>>)` returns only the failure values. Both execute eagerly and neither accumulates. When both halves are needed, pair two passes or follow `TraverseM` with a partition — the API provides no single `partitionM` form.

[IOEMPTY_AND_MONOIDAL_CHOICE]:
- `IOEmpty<A>` is `InvokeSync<A>` whose `Invoke` throws `Errors.None` — not `null`, but the `Error.None` sentinel wrapped as an exception. This makes empty propagate through `Catch` handlers that filter on `Errors.None`, so `io | IO.empty<A>()` is `io` on success and `IO.empty<A>()` on error — the exact `SemigroupK` zero semantics. `IOEmpty` fuses into `Map`, `Bind`, `FoldWhile`, and `FoldUntil` via `IOEmptyAsync` without executing continuations.
- `IO<A>.Combine(IO<A> rhs)` is `Catch((Error _) => true, (Error _) => rhs)` — `MonoidK` addition is recovery-on-any-error. This means `io1 | io2` (the operator spelling of `Combine`) uses `Choose` which is itself built on `Catch`. Stacking three alternatives `io1 | io2 | io3` builds a left-linear chain of `IOCatch` nodes, each inspecting the previous failure.

[SCHEDULE_BUILDERS]:
- `Schedule.spaced(Duration)` — constant delay between every step; produces an infinite series.
- `Schedule.linear(Duration seed, double factor = 1.0)` — delay grows linearly: `seed * (1 + factor * n)`.
- `Schedule.exponential(Duration seed, double factor = 2.0)` — delay doubles (or multiplies by `factor`) each step.
- `Schedule.fibonacci(Duration seed)` — Fibonacci-shaped delay series.
- `Schedule.upto(Duration max)` — emits until cumulative wall-clock time exceeds `max`; accepts an optional `currentTimeFn` for deterministic testing.
- `Schedule.fixedInterval(Duration, currentTimeFn?)` — fires at absolute wall-clock boundaries regardless of how long the action took. If the action runs longer than the interval, the next fire is immediate — no pile-up of queued executions. `Schedule.windowed(Duration, currentTimeFn?)` subdivides the timeline into windows and sleeps until the next window boundary; it can fire multiple times per window if execution is fast. Both accept an optional `currentTimeFn` for deterministic test injection.
- `Schedule.recurs(int times)` — returns `ScheduleTransformer`, not `Schedule`; it applies `.Take(times)` to the base schedule it modifies. Composed as `Schedule.recurs(5) | Schedule.exponential(10.Milliseconds())`.
- `Schedule.repeat(int times)` produces `SchRepeat` — it repeats the *entire base schedule* `n` times, not individual retry iterations. `Schedule.recurs(int times)` is `.Take(times)` on the base — it limits the total number of steps. The naming inversion is load-bearing: `recurs(5)` gives exactly 5 attempts; `repeat(5)` gives 5 full cycles of the base schedule, each with as many steps as the base contains. Use `recurs` for count-bounded loops, `repeat` for bounded phase cycling such as resetting a window pattern `n` times.
- `Schedule.maxDelay(Duration max)` — `ScheduleTransformer`; caps each individual delay at `max`.
- `Schedule.maxCumulativeDelay(Duration max)` — `ScheduleTransformer`; terminates the schedule once total accumulated delay meets or exceeds `max`. Terminates on the step *before* the threshold is crossed.
- `Schedule.jitter(Duration min, Duration max)` — `ScheduleTransformer`; adds absolute random jitter in `[min, max]`.
- `Schedule.jitter(double factor = 0.5)` — `ScheduleTransformer`; adds proportional jitter: `delay * factor * rand`.
- `Schedule.decorrelate(double factor = 0.1, Option<int> seed = default)` — `ScheduleTransformer`; emits two durations per base step — `current + jitter` then `current - jitter` — doubling the schedule's length; `decorrelate` is an expansion transformer, not a per-step scalar. Each jitter is an independent `SingletonRandom.Uniform(0, current * factor)` draw. Accepts an optional `seed` for reproducible jitter.
- `Schedule.NoDelayOnFirst` — `ScheduleTransformer`; strips the first step delay to zero then restores it. Equivalent to `s => s.Tail.Prepend(Duration.Zero)`.
- `Schedule.RepeatForever` — `ScheduleTransformer`; wraps the schedule in a repeat loop; use `Schedule.Forever` (the bare Schedule) for an infinite constant-zero-delay sequence.
- `Schedule.resetAfter(Duration max)` produces `SchResetAfter`, which drives `(s | maxCumulativeDelay(max)).Run()` in an outer `while (true)` loop — once cumulative delay reaches `max`, the inner iterable exhausts and the outer loop restarts from the beginning of the original schedule. This produces an infinitely resetting bounded-window schedule.
- `Schedule.intersperse(Schedule inner)` is `Transform(s => s.Bind(inner.Prepend))` — it inserts every duration from `inner` before each step of the outer schedule. The `params Duration[]` overload accepts inline duration literals as a variadic probe series; `Duration` and `Schedule` are not implicitly convertible so the two overloads do not conflict at call sites.
- Calendar-aligned schedules — `secondOfMinute(int)`, `minuteOfHour(int)`, `hourOfDay(int)`, `dayOfWeek(DayOfWeek)` — accept an optional `Func<DateTime>? currentTimeFn` parameter. Passing a deterministic `currentTimeFn` makes these schedules testable without wall-clock coupling; the default is `() => DateTime.Now`.
- `Schedule.TimeSeries(params Duration[] durations)` constructs `SchItems` from an `Iterable<Duration>`. `Duration` stores milliseconds as `double` and all arithmetic operators work over `double`, converting back via `new Duration(double)`.

[DURATION_UNIT_SURFACE]:
- `Duration` is `readonly struct` over a `double Milliseconds` field. Implicit conversions exist from `double`, `TimeSpan`, and `Time`; explicit conversion from `Duration` to `TimeSpan`. Arithmetic operators (`+`, `-`, `*`, `/`, comparison) all operate on the underlying `double`.
- `Time` is `readonly struct` storing seconds as `double`. Extension methods `.Milliseconds()`, `.Seconds()`, `.Minutes()`, `.Hours()`, `.Days()` on `int`, `float`, and `double` produce `Time` values. Prelude constants `ms`, `seconds`, `minutes`, `hours`, `days` are pre-constructed `Time` values equal to `1` of each unit — `10 * ms` produces `Time(0.010)` which converts implicitly to `Duration(10.0)`.
- `Duration.random(Duration min, Duration max, Option<int> seed)` is the public surface for `SingletonRandom.Uniform`. Unseeded draws use `Random.Shared` internally. Using `Duration.random(...)` as the seed to a `Schedule.linear` de-correlates parallel retry storms without requiring `decorrelate()`.

[SCHEDULE_ALGEBRA]:
- `|` on two `Schedule` values is `Union`: min-of-two semantics at each step, runs to the *longer* schedule. The `SchUnion` source: when one is exhausted, yields from the other's remaining steps.
- `&` on two `Schedule` values is `Intersect`: max-of-two semantics, runs to the *shorter* schedule. `SchIntersect`: `Math.Max` at each zipped step.
- `+` on two `Schedule` values is `Combine` (sequential append): appends one series after the other exhausts.
- `|` on `Schedule | ScheduleTransformer` and `ScheduleTransformer | Schedule` applies the transformer to the schedule — it is *not* a union. Same for `&` with a transformer operand. This asymmetry means `|` means different things depending on operand types.
- `ScheduleTransformer + ScheduleTransformer` composes transformers left-to-right: `f + g` produces `g(f(s))` for any schedule `s`. `ScheduleTransformer` is a `readonly struct` wrapping a `Func<Schedule, Schedule>` field with `Apply(Schedule)` and `op_Addition`. The constructor is `internal`; `Schedule.Transform(s => ...)` is the public factory — any `Schedule -> Schedule` function qualifies.
- `ScheduleTransformer` is implicitly convertible to `Schedule` via `static implicit operator Schedule(ScheduleTransformer t) => Schedule.Forever | t`, making a bare transformer in an operator chain produce a `Schedule`.
- `Schedule.Interleave(b)` — alternates durations from two schedules element-by-element; both must run for the result to continue.
- `Schedule.Run()` returns `Iterable<Duration>`, not `IEnumerable<Duration>`. Retry/repeat drives it via `schedule.PrependZero.Run().GetIterator()`, calling `MoveNext()` once per attempt. `PrependZero` injects a `Duration.Zero` as the first step so the initial attempt fires without delay. `Schedule.Never` is `SchNever.Default` — its `Run()` returns an empty `Iterable<Duration>`. `Schedule.Once` is `Forever.Take(1)`. `Schedule.Forever` is `SchForever.Default` whose `Run()` yields `Duration.Zero` infinitely.

[RETRY_AND_REPEAT]:
- Retry uses `BracketFail()` internally: each attempt runs in a scoped resource environment that disposes on failure but not on success.
- Repeat uses `Bracket()` internally: each iteration disposes resources on completion regardless of outcome. The `RepeatUntil` source confirms: `IO.yieldFor(head).Bind(_ => Bracket().Bind(v => ...))`.
- Both `RetryUntil(Schedule, predicate)` and `RepeatUntil(Schedule, predicate)` call `schedule.PrependZero.Run()` — the zero prepended means the first attempt fires immediately (zero delay), with scheduled delays between subsequent attempts.
- `RetryUntil(Func<Error, bool> predicate)` — no-schedule form; recursively calls itself without `yieldFor`, so there is no delay between attempts and no schedule exhaustion. Use with a maximum-iteration guard via a `recurs`-capped schedule.
- `Fold<S>(Schedule, S, Func<S,A,S>)` — runs `ma` repeatedly, accumulating state; terminates when the schedule is exhausted.
- `FoldWhile<S>(Schedule, S, folder, Func<(S State, A Value), bool>)` — the predicate receives a named tuple `(State, Value)`, not two separate parameters. The overloads accepting `Func<S, bool>` and `Func<A, bool>` project to this form internally.
- `IO<A>.FoldWhile<S>` has six overloads discriminating on schedule presence and predicate projection: `(Schedule, S, folder, Func<S,bool>)`, `(S, folder, Func<S,bool>)`, `(Schedule, S, folder, Func<A,bool>)`, `(S, folder, Func<A,bool>)`, `(Schedule, S, folder, Func<(S,A),bool>)`, and `(S, folder, Func<(S,A),bool>)`. The schedule-absent forms use `Schedule.Forever` as the default, producing an infinite fold bounded only by the predicate. All six delegate to `IOFoldWhile<S,A,S>` with the predicate normalized to `Func<(S State, A Value), bool>`.
- `FoldUntil` mirrors `FoldWhile` with negated predicate semantics — the fold continues while predicate is false. Both families are also exposed via `FoldWhileIO<M,S,A>` and `FoldUntilIO<M,S,A>` on `MonadUnliftIO<M>` extension methods, enabling carrier-polymorphic stateful repetition. The `IOFoldWhile` node carries the schedule's `IEnumerator<Duration>` obtained via `schedule.PrependZero.Run().GetEnumerator()` and advances it per iteration.
- `IO.yieldFor(Duration)` emits a `Task.Delay`-based pause; used inside schedule-driven retry/repeat to honour the inter-attempt delay without blocking the thread.

[TRAVERSAL_AND_SEQUENCING]:
- `seq.TraverseM<M, B>(Func<A, K<M, B>>)` on `Seq<A>`, `Arr<A>`, `Lst<A>`, `Set<A>`, `HashSet<A>`, `Iterable<A>` — monadic traversal; returns `K<M, Seq<B>>` (or the appropriate container type). Call `.As()` immediately to recover the concrete `Eff<Seq<B>>`.
- `Traversable.traverseM<T, M, A, B>(f, ta)` — the free-standing static form; `T` must satisfy `Traversable<T>`. Prefer the instance `.TraverseM` on concrete collection types to avoid type inference failures on `T`.
- `Seq<K<Eff, A>>.Actions()` (via `Applicative<Eff>`) — sequences effects, discards values; used when only side-effects matter. The `Eff` trait class implements `Actions` by running each effect's `IO` as a left-fold via `io.Actions()`.
- `fork<M, A>(K<M, A> ma)` — `where M : MonadUnliftIO<M>`; lifts `ma` into a background task returning `ForkIO<A>(Cancel: IO<Unit>, Await: IO<A>)`. The forked computation runs in a child `EnvIO` with its own `CancellationTokenSource` linked to the parent token via a registration. `IO<A>.Fork(Option<TimeSpan> timeout)` launches via `Task.Factory.StartNew` with `TaskCreationOptions.LongRunning` — it bypasses the thread pool and requests a dedicated thread from the OS, creating excess OS-thread pressure for short-lived tasks. For lightweight parallelism over a collection, prefer `awaitAll(Seq<K<M,A>>)`, which forks via `TraverseM` and does not use `LongRunning`. The forked computation runs inside a child `Resources` scope parented to `env.Resources`, so acquired handles in the fork register locally and are released when the fork completes.
- `ForkIO<A>` is a `readonly record struct` with two fields: `Cancel: IO<Unit>` and `Await: IO<A>`. `Cancel` fires `tsrc.Cancel()` swallowing `OperationCanceledException` and `ObjectDisposedException`. `Await` uses `AwaitAsync` which awaits the task while also observing the calling environment's cancellation token — if the caller cancels before the fork finishes, `Await` propagates the caller's cancellation, not the fork's. `Await` is `IO<A>`, not `IO<Fin<A>>`; re-awaiting the same `ForkIO<A>` handle in a retry loop is unsafe: the underlying `Task<A>` completes once, and subsequent awaits race on an already-completed task, potentially producing `ObjectDisposedException` if the fork's cleanup has run.
- `awaitAll(Seq<ForkIO<A>>)` uses `Task.WhenAll` over `f.Await.RunAsync(eio).AsTask()` projections — each fork's `Await` IO runs with the outer `EnvIO`, meaning outer cancellation cancels all awaits simultaneously. `awaitAny(Seq<K<M, A>>)` forks each effect, uses `Task.WhenAny`, picks the winner's value, and returns it. Losers are neither cancelled nor awaited — their tasks continue running until their own timeout or completion. When guaranteed cleanup is required, take explicit `ForkIO<A>` handles, call `awaitAny(Seq<ForkIO<A>>)`, then cancel losers via `forkIO.Cancel.Run(env)` inside a `Finally` block.
- `IOExtensions.WaitOneIO(AutoResetEvent)` and `WaitOneIO<A>(AutoResetEvent)` lift BCL synchronization primitives into the `IO` monad. The typed overload returns `IO<A>` where `A` is a phantom — useful when a downstream bind requires a typed IO. `Semaphore`, `ManualResetEvent`, and `Monitor` have no built-in `IO` wrappers and must be lifted via `IO.lift(() => evt.WaitOne())`.

[OPERATOR_ALGEBRA]:
- `>>` on `K<Eff, A>` with `Func<A, K<Eff, B>>` — Kleisli bind; desugars to `+ma.Bind(f)`. The unary `+` downcast is emitted automatically by the operator implementation, not by user code.
- `>>>` (`op_UnsignedRightShift`) on `K<Eff, A>` with `K<Eff, B>` — runs the left effect for its side-effects, discards the value, then runs the right effect and returns its value. It is distinct from `>>` (`Kleisli`): `>>>` ignores the left value entirely, while `>>` feeds it as input. The operator is available on both `Eff` and `IO` via their respective extension classes.
- `*` on `K<Eff, Func<A,B>>` with `K<Eff, A>` — applicative `Apply`; both sides run, function is applied to value. `IOExtensions` defines symmetric overloads up to 11-arity curried functions — `Func<A0,...,A10, B>` applied to `K<IO, A0>` through the final partial. All effects run left-to-right and each application strips one function argument, returning a new `K<IO, Func<...>>` until fully applied; this is the dense form for combining up to 11 sequentially-executed effects without nested `SelectMany`.
- `*` on `Func<A,B>` with `K<Eff, A>` — lifts the pure function and applies; `map` without explicit `.Map()`.
- `|` on two `K<Eff, A>` — `Choose`; runs left, falls back to right on failure.
- `|` on `K<Eff, A>` with `CatchM<Error, Eff, A>` — applies the catch combinator.
- `|` on `K<Eff, A>` with `Pure<A>` — fallback to a pure value on failure.
- Downcast via `+ma` (`operator+(K<Eff,A>)` → `(Eff<A>)ma`) and `.As()` (`this K<Eff,A>` → `(Eff<A>)ma`, plus `this Eff<MinRT,A>` → `Eff<A>`) are semantically identical casts, not copies. The `>> Lower` form is also a downcast alias.
- `IO<A>` carries the same operator set: `>>`, `>>>`, `*`, `|`.
- `EffExtensions` provides `SelectMany` overloads whose source is a `ValueTuple` of effects — `(K<Eff,A> First, K<Eff,B> Second)` and the 3-tuple form. In a query expression the bound tuple exposes named fields (`First`, `Second`, `Third`), not positional destructuring. The semantics are monadic left-to-right sequential; both effects run before either result is accessible. Applicative concurrency requires explicit `fork`/`awaitAll`.
- `MonadIOExtensions.SelectMany<M, A, B, C>(K<M, A>, Func<A, IO<B>>, Func<A, B, C>)` and the symmetric `SelectMany<M, A, B, C>(IO<A>, Func<A, K<M, B>>, Func<A, B, C>)` enable mixed-carrier LINQ queries where one step is `IO` and another is `Eff<RT>` without explicit `.LiftIO`. The first form binds an `IO` step inside an `M`-shaped query; the second binds an `M` step inside an `IO`-shaped query. Both delegate to the carrier's `LiftIO` automatically.

[RUNTIME_CAPABILITY_PROJECTION]:
- `Eff<RT, A>` reads the runtime via `asks<RT, Cap>(Func<RT, Cap>)` — returns `Ask<RT, Cap>`, which `Eff<RT,A>.Bind` accepts directly. The `Ask<RT,A>` record resolves via the `Readable<M, Env>` trait without allocating a new monad wrapper.
- `Has<M, VALUE>` is an `interface` with a single `static abstract K<M, VALUE> Ask { get; }`. The three-parameter static class `Has<M, Env, VALUE>` (where `Env : Has<M, VALUE>`) stores `Env.Ask` in a `public static readonly` field, amortizing static-virtual dispatch to a single initialization. Prefer `Has<Eff<RT>, RT, MyService>.ask` over calling `Env.Ask` at each bind site to avoid repeated static-virtual dispatch.
- `asks<Env,A>(Func<Env,A> f)` from `Prelude` constructs `Ask<Env,A>` — a `readonly record struct` that exposes `ToReadable<M>()`, `ToReader()`, `ToReaderT<M>()`, and `SelectMany` for query-expression compatibility. Inside `Eff<RT,A>` pipelines, `asks<RT,Cap>(rt => rt.MyService)` is the idiomatic slot projection; the resulting `Ask<RT,Cap>` binds directly into `Eff<RT>` via `Readable<Eff<RT>, RT>.Asks`.
- `Eff<A>` (no-RT) is `Readable<Eff<A>, A>` where the env is the type parameter itself — useful for dependency-injection patterns where the outer type binds all capabilities before running, but this form cannot extend capability slots at composition time.
- `WithRuntime<RT>()` on `Eff<A>` converts to `Eff<RT, A>` via `MonadIO.liftIO<Eff<RT>, A>(effect.RunIO(default(MinRT))).As()` — it runs the inner `ReaderT<MinRT, IO, A>` against the zero-byte `MinRT` value to produce `IO<A>`, then re-lifts that `IO<A>` under the `RT` carrier. This is a carrier upgrade, not a capability injection; the `RT` value is still required at the final `Run(RT env)` call.
- `Eff.local<OuterRT, InnerRT, A>(Func<OuterRT, InnerRT> f, Eff<InnerRT, A> ma)` returns `Eff<OuterRT, A>`. Its implementation reads `getState<OuterRT>()` to get the outer runtime, maps it through `f` to produce the inner runtime, runs `ma.effect.Run(innerRT)` wrapped in `IO.local` (fresh cancel scope), then re-surfaces the result in the outer carrier. This is the canonical mechanism for embedding a dependency-isolated sub-computation whose runtime is a projection of the outer one — no capability leakage between scopes.
- `Eff.runtime<RT>()` lifts the runtime record itself into the bound value as `Eff<RT, RT>`. Combined with `.Bind(rt => ...)` it enables a computation to inspect its own runtime at composition time without breaking the reader monad abstraction. Used to derive derived capabilities or perform runtime-conditional branching without breaking the monadic spine.
- `Eff.getState<RT>()` returns `Eff<RT, (RT Runtime, EnvIO EnvIO)>` — it exposes the full internal state of the running effect, including both the capability record and the `EnvIO`. Use it when downstream logic must inspect the execution environment directly; `Eff.runtime<RT>()` gives only the `RT` value, `getState` gives the pair.
- `Eff.localCancel<RT, A>(Eff<RT, A> ma)` delegates to `ma.LocalIO().As()`, which calls `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — fresh CTS linked to the parent token, shared resources. Cancellation of the inner scope does not propagate to the outer scope's other computations; an exception produced by `IO.cancel` inside the local does propagate outward because it is an exception, not a signal. Scope isolation prevents regional cancellation from tearing sibling effects; it does not suppress the exception.

[STM_AND_REF_INTEGRATION]:
- `STM.DoTransaction` is `internal`; the public surface is `atomic<R>(Func<R> op, Isolation)` in `Prelude`, with `atomicIO<R>` wrapping the same call in `IO.lift`. Effect-pipeline integration requires wrapping: `IO.lift(() => atomic(op, isolation))` is what `atomicIO` does. The `Eff`-overloaded `DoTransaction` forms exist internally and nest when a parent transaction is active; `RunTransaction` is the raw form that never nests. Prefer `atomicIO` at call sites to stay inside the `IO` rail.
- `Isolation.Snapshot` rolls back only when a `Ref` *written within* the transaction is externally written between start and commit. `Isolation.Serialisable` rolls back when any `Ref` *read or written within* the transaction is externally written. Choosing `Snapshot` admits more concurrency but leaves read-write skew possible; `Serialisable` prevents all anomalies at the cost of higher conflict rate.
- `Ref<A>.Commute(Func<A, A>)` re-runs the function at commit time against the most-recently-committed value, not the in-transaction snapshot. This means `Commute` is only correct for commutative or last-writer-wins operations — non-commutative uses will silently produce stale-basis results.
- `Ref<A>.SwapIO(Func<A, A>)` returns `IO<A>` — it wraps `Swap` in `IO.lift`, enabling participation in effect pipelines. The swap is transactional when called inside an `atomic` boundary; outside a transaction it operates as a standalone assignment (not a CAS spin, since `Ref` mutation uses STM commit, not lock-free CAS). `Atom<A>.SwapIO` returns `IO<A>` from `IO.lift(_ => Swap(f))` — an actual CAS spin loop that runs until no contention; the two types have the same IO return shape but different internal mutation semantics.
- `Atom<A>.SwapMaybeIO(Func<A, Option<A>>)` wraps `SwapMaybe` in `IO.lift`: when the function returns `None` the atom is unchanged and the IO yields the current value; when it returns `Some(v)` the atom is updated to `v` and the IO yields `v`. The function runs inside a spin loop on contention, so it must be side-effect-free and cheap. This is the idiomatic form for conditional test-and-set where the update decision depends on the current value.
