# Algebraic Owner Operator Lattice

[OPERATOR_AXES_AS_DIMENSIONAL_ALGEBRA]:
- Six independent operator axes are declared per keyed owner — addition, subtraction, multiply, division, comparison, and equality-comparison — and each axis is an explicit grant of one binary generic-math interface (`IAdditionOperators<T,T,T>`, `ISubtractionOperators`, `IMultiplyOperators`, `IDivisionOperators`, `IComparisonOperators<T,T,bool>`, `IEqualityOperators<T,T,bool>`) onto the owner. The axes are not a single "is it a number" toggle; they are a per-operation grant set, and the selection of which axes are on is the dimensional algebra of the quantity the owner models.
- A like-quantity owner enables addition and subtraction and holds multiply and division at `None`, because adding two lengths yields a length but multiplying two lengths changes dimension to area — the product type is a different owner. The axis set is therefore a statement that the closed binary operations of the quantity's group are `+` and `-` only, and the cross-dimension operators (`Length * Length -> Area`) are hand-written against the foreign result type, which no per-axis grant can express because every generated axis is homogeneous `T op T -> T` or key-typed `T op TKey -> T`.
- Generation per axis is conditional on the key type itself implementing the corresponding generic-math interface or exposing its `op_` method pair, so an arithmetic axis declared over a non-arithmetic key is silently inert — the setting compiles, the attribute carries it, and no operator appears. A no-effect axis declaration is a latent lie about the owner's algebra worth deleting, and the inverse — a numeric key with no axis declared — is a quantity whose group operations are unexpressed at the type and re-derived by casting out to the raw key at every arithmetic site.

```csharp
[ValueObject<decimal>(
    AdditionOperators = OperatorsGeneration.DefaultWithKeyTypeOverloads,
    SubtractionOperators = OperatorsGeneration.DefaultWithKeyTypeOverloads,
    MultiplyOperators = OperatorsGeneration.None,
    DivisionOperators = OperatorsGeneration.None)]
public readonly partial struct Meters;

[ValueObject<decimal>(MultiplyOperators = OperatorsGeneration.None)]
public readonly partial struct SquareMeters;

static SquareMeters Area(Meters width, Meters height) =>
    SquareMeters.Create((decimal)width * (decimal)height);

static TQuantity Total<TQuantity>(IEnumerable<TQuantity> parts, TQuantity zero)
    where TQuantity : IAdditionOperators<TQuantity, TQuantity, TQuantity> =>
    parts.Aggregate(zero, static (acc, next) => acc + next);
```

[CHECKED_VARIANT_DUALITY]:
- Every generated arithmetic axis emits both the unchecked operator and its `operator checked` counterpart — `op_Addition` and `op_CheckedAddition`, and the same pairing for subtraction, multiply, and division — conditioned on the key exposing both. The owner therefore inherits the BCL's two-mode arithmetic contract whole: an evaluation inside a `checked` context binds the checked operator and an overflow on the underlying key throws at the owner's surface rather than wrapping silently, while the same expression outside the context binds the unchecked form. The overflow-trapping mode is a property of the call site's context, not a second owner type and not a flag on the value.
- The checked duality is what lets a quantity owner be a sound substitute for its key under `checked`-disciplined accumulation: a generic accumulator constrained on `IAdditionOperators` run inside a checked region traps overflow at the quantity boundary, so a domain quantity does not weaken the overflow guarantees the raw key had — the evidence wrapper is overflow-transparent in both directions.

[EQUALITY_PRESUPPOSITION_AS_A_TYPE_LAW]:
- Comparison operators presuppose equality operators because the comparison generic-math interface inherits the equality one, so a type that claims to be comparable but not equatable is unsatisfiable. The owner encodes this as an attribute-level invariant: the comparison axis is forced to `None` whenever equality comparison is off, and the equality-comparison axis getter returns the join of its own backing value and the comparison axis — declaring comparison at the key-typed-overload level silently lifts equality to the same level. The lattice point "ordered but not equatable" is therefore inexpressible by construction, mirroring the interface hierarchy rather than re-checking it.
- `EqualityComparisonOperators` is a monotone function of `ComparisonOperators`, not an independent slot: setting comparison high and equality low yields the high value at read, so the two axes form a chain `None < Default < DefaultWithKeyTypeOverloads` along which comparison can only pull equality up. The only way to suppress equality operators is to suppress the entire equality comparison stance at the owner, which forces both axes to `None` together — identity and order collapse to off as one move, never independently.
- The whole operator surface is downstream of admission: disabling factory generation forces all four arithmetic axes to `None` regardless of their declared values, because the generated operators delegate to the factory to re-admit the arithmetic result. An owner with no admission has no arithmetic, no parsing, no conversion-from-key — the operator lattice is one more capability that is a projection of the single `Validate` funnel, never a free-standing surface. An axis declared on a factory-suppressed owner is inert in the same silent way as an axis declared over a non-arithmetic key.

[KEY_TYPED_OVERLOAD_AS_HETEROGENEOUS_ALGEBRA]:
- `DefaultWithKeyTypeOverloads` adds the heterogeneous operator overloads against the raw key type alongside the homogeneous owner-to-owner ones, so `quantity + rawScalar` and `quantity < rawScalar` bind without converting either operand and without casting the owner out. The owner stays in play at exactly the sites that would otherwise drop to the key to do arithmetic against a literal or an external scalar, which is the readability defect the overload set deletes: a quantity that must cast to its key to add a constant has surrendered its evidence for the duration of the expression.
- Key choice decides whether order is operator-expressible at all, because the axis grant is inert over a key that lacks the operators: a numeric-keyed rank ladder reaches `<`/`>=` plus heterogeneous key-typed comparison, while a string-coded vocabulary can never reach operator-form ordering — the comparison axis declared over a string key is silently inert. A vocabulary whose consumers dispatch by range therefore needs the numeric key even when a human-readable code exists, and the code demotes to a non-key column. The operator surface is a forcing function on key selection, not a free decoration over whatever key reads best.

[GRANT_AND_REVOKE_ORDER_INDEPENDENT_OF_KEY]:
- `IComparable<T>` is grantable over a non-comparable key purely by declaring a comparer accessor — the owner gains ordering the key does not have, because the generated comparison routes through the declared `IComparer<T>` rather than the key's natural order. Ordering is thus a property assignable at the owner independent of the key's own capability, which decouples "this quantity has a domain rank" from "the underlying representation is naturally ordered": a string-keyed vocabulary gains a domain sort by supplying a comparer accessor, without the string's lexical order leaking in.
- The inverse is a positive semantic statement, not an omission: a surrogate identifier declares `SkipIComparable` with comparison operators at `None` because its ordering is genuinely meaningless, and the stance is then enforced at every sort and range expression at compile time. Granting order where the key has it but the domain forbids it, or withholding order where the key lacks it but the domain needs it, are both first-class owner decisions made against the key's capability rather than inherited from it — capability subtraction is as meaningful as capability addition because both are declared and both are enforced.
- Comparer policy is typeclass-shaped through static-abstract accessor interfaces (`IComparerAccessor<T>.Comparer`, `IEqualityComparerAccessor<T>.EqualityComparer`), with prebuilt accessor classes for the string-comparison family and a default-comparer accessor for any `T`. Equality and ordering accessors are a mandatory pair — declaring one without the other lets hashing and sorting disagree on identity — and the comparison axis coerces the equality axis upward for the operator surface exactly as the comparer pair coerces at the comparer surface: the operator lattice and the comparer lattice are the same equality-presupposition law expressed at two surfaces of one owner.

[PARTIAL_NUMBER_CITIZENSHIP]:
- The grantable axes are precisely the binary additive and multiplicative group interfaces plus the two relational interfaces; there is no axis for unary negation, modulus, increment/decrement, or the aggregate number interface that bundles parsing, formatting, radix, and the numeric identity constants. An owner is therefore a deliberately partial generic-math citizen: it satisfies the binary-operation constraints a generic accumulator or a generic interpolation routine actually names, while never claiming to be a full number with a multiplicative identity, a radix, or saturating conversions. The partiality is the correctness property — a domain quantity that advertised full number-hood would invite `One`, `Abs`, and radix operations that are nonsense on a dimensioned value.
- Constraining a generic algorithm on the exact binary interface it needs — `IAdditionOperators` for a sum, `IComparisonOperators` for a clamp, both for a running min-with-accumulation — lets every owner that grants those axes flow through one algorithm with zero per-owner overload, and an owner that withholds an axis is rejected at the constraint rather than at a runtime cast. The algorithm names the algebra it requires and the owner advertises the algebra it has; the intersection is checked by the constraint solver, so a quantity used in an operation outside its group is a compile error at the call, not a representation bug at runtime.
- Unary operations, identity constants, and dimension-changing products that the axis set cannot grant are hand-written on the owner's user-declared partial against the foreign result type, and this hand-written remainder is the exact shape of the quantity's algebra that is not a closed homogeneous binary operation — the generated axes carry the group structure that is closed and homogeneous, and the manual operators carry the structure that crosses dimensions or arity. Reading an owner's generated axis set against its hand-written operators recovers the full algebraic signature of the quantity: which operations are internal to the type, which leave it, and which the type forbids.
