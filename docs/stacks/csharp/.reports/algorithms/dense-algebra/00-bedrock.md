# Dense Algebra — Bedrock

[FACTORIZATION_SUITE]:
- Six factorizations are available on every `Matrix<double>` instance as virtual factory methods: `Cholesky()`, `LU()`, `QR(QRMethod)`, `GramSchmidt()`, `Svd(bool computeVectors)`, and `Evd(Symmetricity)`. Each returns its typed abstract handle (`Cholesky<double>`, `LU<double>`, etc.) rather than a concrete sealed class; the concrete `Dense*` implementations are `internal sealed` and unreachable by name. The handle carries all downstream evidence and `Solve(Matrix<T>, Matrix<T>)` / `Solve(Vector<T>, Vector<T>)` operations, so a retained handle yields repeated right-hand-side solves with zero refactorization cost.
- `DenseCholesky.Create` clones the input, then calls `LinearAlgebraControl.Provider.CholeskyFactor` on the clone's `Values` array in place. The managed `CholeskyFactor` reads only upper-triangle entries (`a[i * order + j]` for `j >= i`) and then zeroes the lower triangle — an asymmetric input whose upper triangle is positive-definite factors successfully, certifying only the definiteness of the upper-symmetrized matrix. A call preceded by `IsSymmetric()` is required to certify full SPD; without it, `Cholesky()` certifies only that the upper triangle is positive-definite. Wrap in `Fin<Cholesky<double>>` at the admission boundary for the definiteness probe. `Cholesky.Factorize(Matrix<double>)` — the in-place re-factorization method on a retained handle — copies the new matrix into the existing factor buffer via `Buffer.BlockCopy` before calling the provider, enabling efficient pipelines that supply successive near-SPD matrices without allocating a new handle each time.
- `DenseLU.Create` requires `RowCount == ColumnCount`; the factorization performs partial pivoting with row swaps recorded in an `int[] Pivots` array. `LU.Determinant` computes the product of diagonal U entries, negating each term where `Pivots[i] != i`. Non-singularity is not asserted at construction — a singular input produces a handle with `Determinant == 0.0`, not an exception. The correct singularity gate post-construction is `Math.Abs(lu.Determinant) < threshold` or, more robustly, probing `lu.Factors.Diagonal().Any(v => Math.Abs(v).AlmostEqual(0.0))`.
- `DenseQR.Create(DenseMatrix, QRMethod)` dispatches to two internal provider paths. `QRMethod.Full` stores an m×m Q and an m×n R; `QRMethod.Thin` stores an m×n Q and an n×n R. The default on `Matrix.QR()` is `QRMethod.Thin`. The `Tau` array (Householder reflector coefficients, length `min(m,n)`) is retained on the handle and forwarded to `QRSolveFactored`, so the native provider can reconstruct Q's action without materializing it. `QR.IsFullRank` checks that no diagonal entry of `FullR` satisfies `Math.Abs(FullR.At(i, i)).AlmostEqual(0.0)` — an absolute threshold of `10 · 2⁻⁵³` against the matrix's own scale. This makes `IsFullRank` the numerically correct full-rank gate only for well-scaled systems; see `[QR_ISFULLRANK_THRESHOLD_INCONSISTENCY]` for the failure mode.
- `DenseGramSchmidt.Create` runs modified Gram-Schmidt in pure managed code and throws `ArgumentException("Matrix must not be rank deficient.")` only when a column norm is `== 0.0` exactly — the comparison is `if (num == 0.0)`. A near-zero column norm (floating-point cancellation without bit-exact zero) passes the check, causes division by near-zero, and silently populates Q and R with `NaN`/`Inf`. The returned handle has an R matrix with `NaN` entries, and `IsFullRank` subsequently returns `true` because `Math.Abs(NaN).AlmostEqual(0.0)` is `false` — the factorization-succeeded-but-result-is-corrupt failure is undetectable without an explicit all-finite check on `((DenseMatrix)gs.Q).Values` and `((DenseMatrix)gs.FullR).Values` after construction. Unlike Householder QR, this path does not route through the provider and receives no native acceleration.
- `DenseEvd.Create` takes a `Symmetricity` discriminant. `Symmetricity.Unknown` calls `matrix.IsSymmetric()` at factorization time — an O(n²) exact-equality scan with no epsilon. The symmetric path routes to Householder tridiagonalization then implicit QR iteration. The asymmetric path routes to Hessenberg reduction then real Schur form. Eigenvalues are always stored as `Vector<Complex>` regardless of symmetricity; for a symmetric input, imaginary parts are numerically zero but the Complex representation is not stripped. `Evd.Solve` is implemented only for the symmetric case and throws `ArgumentException("Matrix must be symmetric.")` for asymmetric inputs. `DenseEvd.Create` pre-allocates eigenvector matrix `denseMatrix` (n×n) and block-diagonal matrix `denseMatrix2` (n×n) before calling `LinearAlgebraControl.Provider.EigenDecomp` with those buffers as output parameters; on the asymmetric path the provider additionally allocates an internal n² working array for Hessenberg reduction. `DenseEvd.Solve` for the symmetric case executes triple nested loops; eigenvector access uses direct `Values` array indexing (`((DenseMatrix)EigenVectors).Values[j * count + k]`) while input and result access goes through virtual `At` dispatch — it receives no BLAS3 acceleration under MKL and does not scale with `Control.MaxDegreeOfParallelism`.
- `DenseSvd.Create` always allocates U as m×m and VT as n×n, regardless of `computeVectors`. When `computeVectors = false`, the provider still allocates both full factor arrays but skips their computation; a subsequent `Solve` call throws `InvalidOperationException("The singular vectors were not computed.")`. Requesting vectors is therefore a requirement for any solve or pseudo-inverse use, and `false` is only a valid cost gate when rank, L2Norm, or ConditionNumber evidence is all that is needed. For tall matrices under the managed provider, `SvdSolveFactored` iterates U column-by-column for the Uᵀb step, requiring the full m×m U — there is no thin-SVD solve path. For a 5000×20 design matrix, the full m×m U allocates ~200MB of working storage; MKL provides a thin SVD via DGESDD but the managed path has no equivalent.
- `DenseCholesky.Solve(input, result)` copies `input.Values` into `result.Values` via `Buffer.BlockCopy` before calling `CholeskySolveFactored`, then overwrites `result.Values` in place with the solution — the solve is unconditionally destructive to any prior content of `result.Values`. Any `result` matrix pre-loaded with a prior solution expecting additive update is silently overwritten before the provider call. `DenseLU.Solve` follows the same copy-then-overwrite pattern. `DenseQR.Solve` does not copy — it passes Q, FullR, Tau, and the input array directly to `QRSolveFactored` and writes the result without preserving input.
- `DenseLU.Inverse()` clones `Factors` into a new `double[n²]` array, then calls `LinearAlgebraControl.Provider.LUInverseFactored` on that clone. The managed `LUInverseFactored` allocates a second `double[n²]` identity array and solves n right-hand sides simultaneously into it. For n ≥ 104, both arrays cross the LOH threshold (~85,000 bytes); at n = 100 the flat array is 80,000 bytes and remains below the LOH boundary. The correct pattern for matrix inversion in a hot loop is to retain the `DenseLU` handle and call `Solve` on an identity matrix with pre-allocated result and right-hand-side buffers reused across calls.

[SVD_EVIDENCE]:
- `Svd<double>.Rank` computes the tolerance as `S.Maximum().EpsilonOf() * Math.Max(U.RowCount, VT.RowCount)`. `EpsilonOf(x)` returns the ULP epsilon at `x` — for a well-scaled matrix with max singular value near 1, this approximates `σ_max · ε_machine · max(m,n)`, matching the standard rank-revealing threshold. The threshold tracks the actual numerical scale of the problem via the max singular value's own ULP, not a global constant.
- `Svd<double>.ConditionNumber` returns `|S[0]| / |S[min(m,n) - 1]|`, which is `+Inf` for singular or rank-deficient matrices. Guard for this before using the result as a quality gate.
- The managed provider delivers singular values in descending order via an insertion-sort step in the convergence loop. `S[0]` is guaranteed to be the largest, making `L2Norm = |S[0]|` and `ConditionNumber = |S[0]| / |S[min(m,n)-1]|` correct without the caller sorting.
- `Matrix.PseudoInverse()` on `Double.Matrix` opens with `Svd<double> svd = Svd()` — a fresh method-local handle per call — then mutates the local `s` vector and `W` matrix (`Lazy<Matrix<T>>` forced via `svd.W`) on that private handle. A caller-retained `Svd` handle is neither read nor touched; all mutations are confined to the locally constructed handle, which is discarded after reconstruction. A second `PseudoInverse()` call is correct. The threshold `PseudoInverse` uses (`max(m,n) · σ_max · Precision.DoublePrecision = max(m,n) · σ_max · 2⁻⁵³`) differs from `Svd.Rank`'s threshold (`σ_max.EpsilonOf() · max(m,n)`, where `EpsilonOf(1.0) ≈ 2⁻⁵²`) by a factor of two. A singular value above the `PseudoInverse` threshold but below the `Rank` threshold is retained by `PseudoInverse` yet counted as zero by `Rank` — the two surfaces give inconsistent rank evidence for boundary-singular values even when called on independently constructed handles from the same matrix.
- `SvdSolveFactored` (the managed provider path) executes `num2 /= s[j]` for every `j < min(m,n)` with no zero guard and no truncation. If any singular value is numerically near-zero but not exactly zero, the solve produces a numerically exploded solution. The pseudo-inverse path applies truncation; `Solve` on the handle does not. As a result, `Solve` and `Rank` / `ConditionNumber` evidence on the same handle are computed under different conventions and can disagree — `Rank` can report deficiency while `Solve` still divides by all singular values.

[EVD_RANK_AND_DETERMINANT_FAILURES]:
- `Evd.Rank` and `Evd.IsFullRank` test `eigenValue.AlmostEqual(Complex.Zero)`, which routes through `a.Norm().AlmostEqualNorm(b.Norm(), a.NormOfDifference(b), DefaultDoubleAccuracy)` where `Norm()` = `MagnitudeSquared()`. For eigenvalue `λ`, the test is `|λ|² < DefaultDoubleAccuracy ≈ 1.11e-15`. An eigenvalue of ~1e-12 gives `|λ|² ≈ 1e-24 < 1.11e-15`, so it is classified as zero and rank reports deficiency. Meanwhile `Svd.Rank` uses `σ_max.EpsilonOf() · max(m,n)` — with `σ_max ~ 1` the threshold is ~n · 2⁻⁵², far above 1e-12, so SVD counts that singular value as nonzero and reports full rank. The two surfaces disagree in opposite directions: EVD is over-sensitive to small eigenvalues (classifying 1e-12 as zero), while SVD is over-tolerant for the same magnitude. Neither alone is a reliable conditioning gate for near-singular problems; SVD with an explicit scale-relative threshold is the correct evidence surface.
- `Evd.Determinant` computes `one.Magnitude` from `Π eigenvalues[i]` — an absolute-value product of complex eigenvalues. For a symmetric matrix with a negative eigenvalue (indefinite matrix passed with `Symmetricity.Symmetric`), the true determinant is negative but `Evd.Determinant` returns a positive value. Only `LU.Determinant` carries the correct sign. For algorithms that need `sign(det(A))` — detecting a definite-to-indefinite transition — `Evd.Determinant` is incorrect.

[QR_ISFULLRANK_THRESHOLD_INCONSISTENCY]:
- `QR.IsFullRank` checks `Math.Abs(FullR.At(i, i)).AlmostEqual(0.0)`, which resolves to `|R[i,i]| < DefaultDoubleAccuracy = 10 · 2⁻⁵³`. This is an absolute threshold at the magnitude of machine epsilon, not a relative threshold scaled to the matrix. For a problem where R has a legitimate smallest diagonal entry ~1e-8 — legitimately rank-deficient relative to problem scale but far above machine epsilon in absolute terms — `IsFullRank` returns `true` while `Svd.Rank` correctly reports rank deficiency (because it uses a relative threshold scaled to `σ_max`). `QR.IsFullRank` is reliable only for problems that are either exactly rank-deficient or well-conditioned; for ill-conditioned problems SVD rank evidence must replace it. The correct relative pivot criterion scales to the largest diagonal: `|R[i,i]| / |R[0,0]| < sqrt(eps)`. All diagonal accesses go through the virtual `Matrix<T>.At(i, i)` dispatch — one virtual call per diagonal element in both `IsFullRank` and `Determinant`.
- For `QRMethod.Full`, the lazy `QR<T>.R` property allocates a new dense matrix and extracts the upper triangle from `FullR` on first access. `FullR` (not `R`) is used internally in every `Solve` call. Accessing `qr.R` from performance-sensitive code during repeated solves allocates unnecessarily. `DenseQR` stores an eagerly materialized Q matrix in its constructor (either m×m for Full or m×n for Thin, computed by `QRFactor`/`ThinQRFactor` at Create time); `Solve` accesses `((DenseMatrix)base.Q).Values` directly with no lazy materialization cost.

[CHOLESKY_DETERMINANT_LOG]:
- `Cholesky<double>.DeterminantLn` computes `2 · Σ log(L[i,i])` using `Factor.At(i, i)` — one virtual call per diagonal element — and is numerically stable for large n where `Determinant = Π L[i,i]²` underflows to zero. Gaussian process log-likelihood, Kalman filter innovation covariance, and multivariate normal normalization all require `log|Σ|`; `DeterminantLn` is the correct surface — `Math.Log(Determinant)` silently returns `-Inf` for large SPD matrices because `Determinant` returns `0.0` without any underflow indication.
- The column-major diagonal of the lower Cholesky factor lies at `values[i * n + i]`. Projecting the diagonal into a contiguous span then calling `TensorPrimitives.Log<double>(diagonalSlice, logSlice)` followed by `TensorPrimitives.Sum<double>(logSlice) * 2.0` replaces n virtual-dispatch scalar calls with a single SIMD pass. At n = 200 this converts ~200 virtual `At` calls to one AVX2-vectorized log+sum; the strided layout requires the diagonal projection step before invoking `TensorPrimitives`.

[ADMISSION_GATES]:
- All six `Dense*` factorizations perform squareness and dimension checks at construction and throw `ArgumentException` or `ArgumentNullException` for violated preconditions. None perform all-finite checks; NaN and Inf propagate silently through factorization, producing corrupted factor state without any exception. An admission gate must verify `TensorPrimitives.IsFiniteAll<double>(new ReadOnlySpan<double>(matrix.Values))` before calling any factorization factory. `IsNaNAny<T>` and `IsInfinityAny<T>` are early-exit alternatives that distinguish the failure condition in the evidence receipt when a typed `FiniteCheckFailure` case must name the cause.
- `Cholesky()` is the correct definiteness admission gate: a thrown `ArgumentException("Matrix must be positive definite.")` is the rejection evidence. Because the managed `CholeskyFactor` reads only the upper triangle, pass `Symmetricity.Symmetric` explicitly or apply `IsSymmetric()` before calling `Cholesky()` when the input may be asymmetric from accumulation; otherwise the probe certifies only that the upper-triangrized matrix is positive-definite. Do not substitute a separate eigenvalue probe.
- `DenseMatrix.IsSymmetric()` uses exact `!=` comparison on `double` entries — no epsilon. A matrix built via incremental accumulation (outer-product updates, normal-equation formation) is frequently not bit-exactly symmetric. When passed to `Evd(Symmetricity.Unknown)`, `IsSymmetric()` returns `false` and the EVD falls through to the asymmetric Hessenberg path, which is slower and causes `Evd.Solve` to throw. Force symmetry explicitly via `matrix.Add(matrix.Transpose()).Multiply(0.5)`, or pass `Symmetricity.Symmetric` when the structural guarantee holds. The allocation cost of the three-expression chain — transpose, add result, scalar multiply result — is three dense matrices; at n = 500 this produces ~6MB of transient working storage. The in-place alternative iterates `Values` and averages symmetric pairs `(values[i*n+j] + values[j*n+i]) * 0.5` for `j < i`, writing both positions in a single pass exploiting column-major layout — zero extra allocations. `TensorPrimitives` has no symmetric-force primitive.

[RESIDUAL_CAPPED_SOLVE]:
- No built-in residual cap exists on any `Dense*` solve path. The managed solve paths forward directly to in-place triangular solve or back-substitution via the provider without computing a residual. A residual-capped solve must be implemented by the caller: after `factorization.Solve(b, x)`, compute `r = A·x - b` using the original matrix, and demote success to a typed failure when `r.L2Norm() / b.L2Norm() > eps_cap`. The demotion follows the carrier rail: the factorization is wrapped in `Fin<TFactorization>` at admission; the solve step returns `Fin<Vector<double>>`; the residual check is a `Bind` that maps a successful solve to `FinFail(new ResidualCapExceeded(relative_residual))` when the cap is exceeded. `DenseMatrix.Values` exposes the backing `double[]` directly, enabling span-level residual computation via `TensorPrimitives.MultiplyAdd` without boxing — column-major layout means each column of A is a contiguous slice, so the Ax accumulation is one `MultiplyAdd` call per column.

[LEAST_SQUARES_ROUTES]:
- `MultipleRegression.NormalEquations<T>` calls `x.TransposeThisAndMultiply(x).Cholesky().Solve(x.TransposeThisAndMultiply(y))`. When X is sparse or tall, `TransposeThisAndMultiply` produces a dense n×n Gram matrix regardless of the input structure — a quadratic densification cliff. For an m×n design matrix with m >> n and sparse structure, the normal-equations route discards sparsity, squares the condition number (κ(AᵀA) = κ(A)²), and allocates O(n²) dense storage. `NormalEquations` is the default `DirectRegressionMethod` and the most dangerous for near-rank-deficient systems; the route must be explicit as a receipt field.
- `MultipleRegression.QR<T>` calls `x.QR().Solve(y)` using `QRMethod.Thin` and handles overdetermined systems without forming the Gram matrix. Query `IsFullRank` on the retained handle for full-rank evidence before accepting the solution; see `[QR_ISFULLRANK_THRESHOLD_INCONSISTENCY]` for the absolute-threshold failure mode.
- `MultipleRegression.Svd<T>` calls `x.Svd(computeVectors: true).Solve(y)` with `computeVectors: true` as the default, allocating a full m×m U and n×n VT. `SvdSolveFactored` delivers rank-deficiency-tolerant notation but applies no truncation — the solve divides by each singular value without consulting `Svd.Rank`. Use `PseudoInverse()` when truncation is required, accepting that the handle is consumed afterward.
- `WeightedRegression.Weighted<T>(x, y, w)` forms `xᵀ(Wx)` by evaluating `x.TransposeThisAndMultiply(w * x)`. When `w` is a `DiagonalMatrix`, `DiagonalMatrix.DoMultiply(Matrix<double>, Matrix<double>)` has an explicit `DenseColumnMajorMatrixStorage` fast path that scales each input row by the corresponding diagonal entry in a nested loop — no m×n intermediate matrix is allocated. The result matrix allocation (m×n) remains unavoidable. For a diagonal weight vector supplied as `double[]` rather than a `DiagonalMatrix`, there is no analogous fast path; the efficient form is to scale each row `i` of `x` by `Math.Sqrt(wDiag[i])` into a working matrix `xw`, then compute `xw.TransposeThisAndMultiply(xw)`. `NormalEquations<T>` has no singularity check before calling `Cholesky()`; treat a thrown `ArgumentException` as a `ConditioningFailure` evidence tag at the admission boundary.

[PACKED_STORAGE_AND_ALIASING]:
- No packed symmetric, triangular, Hermitian-packed, or band storage exists. `DenseColumnMajorMatrixStorage<T>` stores all m×n entries as a column-major `T[]` flat array. Algorithms that exploit symmetry (Cholesky, symmetric EVD, normal-equation regression) still allocate and operate on the full n×n array; the upper triangle is populated then overwritten in the Cholesky inner loop but never packed. This doubles memory over LAPACK-style DSYTRF packed storage for symmetric matrices.
- `DenseMatrix.Values` returns the backing `double[]` directly from `DenseColumnMajorMatrixStorage<T>.Data`. Constructing a `DenseMatrix` from an external `double[]` via `new DenseMatrix(rows, cols, existingArray)` binds that array without copying. `DenseCholesky.Create`, `DenseLU.Create`, `DenseQR.Create`, and `DenseSvd.Create` each call `matrix.Clone()` before passing the buffer to the provider, so factorization never mutates the source array for those four paths. `DenseEvd.Create` passes `matrix.Values` directly to `EigenDecomp` without cloning; on the symmetric path the provider copies the matrix via `Buffer.BlockCopy` into the pre-allocated `matrixEv` buffer before operating, leaving the original intact. `MapInplace` and provider-level operations targeting a `DenseMatrix` constructed from an external array do mutate that array in place. If the same `double[]` is wrapped into two `DenseMatrix` handles, any in-place operation on one mutates the other's view silently. The canonical safe pattern is `matrix.Clone()` before any in-place operation when the input originates from an external buffer; reserve direct array construction exclusively for hot paths with single-owner scope.

[PROVIDER_AND_VECTORIZATION]:
- The managed provider (`ManagedLinearAlgebraProvider`) contains no SIMD or `TensorPrimitives` calls. `MatrixMultiplyWithUpdate` pre-extracts all columns of B into `double[][] columnDataB = new double[columnsB][]` before the row-column dot loop: each column of B is extracted by `GetColumn` (a contiguous `Data[c*n .. (c+1)*n-1]` slice copy) into a new `double[rowsB]` heap array. This pre-extraction doubles memory traffic for B and allocates `columnsB` separate heap arrays of length `k` before any computation begins. The inner dot loop is scalar. Under MKL, this code path is bypassed entirely and routes to BLAS DGEMM.
- Provider switching is process-global: `LinearAlgebraControl.UseNativeMKL()` replaces the `static ILinearAlgebraProvider` under a lock. It cannot be scoped per-thread or per-operation. When MKL is present, five factorization families route to LAPACK equivalents (DGETRF for LU, DPOTRF for Cholesky, DGEQRF for Householder QR, DGESVD for SVD, DSYEV/DGEEV for EVD) with full BLAS3 performance. `DenseGramSchmidt` runs managed modified Gram-Schmidt exclusively and receives no MKL acceleration regardless of provider state.
- The parallelization gate `rowsA + columnsB + columnsA < Control.ParallelizeOrder` uses a linear sum of three dimensions, not their product. For a 10×1000×10 GEMM the sum is 1020 — above the default threshold of 64 — so parallel dispatch fires even though the work does not warrant it. For a 64×64×64 GEMM the sum is 192, also above threshold. `Control.MaxDegreeOfParallelism = 1` eliminates the `CommonParallel.For` overhead for any small-n batch.
- `System.Numerics.Tensors.TensorPrimitives` provides SIMD-backed `Dot<T>`, `Add<T>`, `Multiply<T>`, `MultiplyAdd<T>`, `FusedMultiplyAdd<T>`, and `SumOfSquares<T>` over `ReadOnlySpan<T>`. `FusedMultiplyAdd<T>` requires `IFloatingPointIeee754<T>` — satisfied by `double` — and executes with a single rounding, delivering improved accuracy for inner products. `TensorPrimitives` has no GEMM surface; dense GEMM remains the exclusive domain of the MathNet provider. `TensorPrimitives` is the platform floor for element-wise and reduction hot loops the provider does not own: all-finite checks via `IsFiniteAll<T>` over the flat `Values` span as a vectorized SIMD predicate; `IsNaNAny<T>` and `IsInfinityAny<T>` as early-exit alternatives that distinguish failure modes; diagonal log-sum for `DeterminantLn`; residual norm computation; and symmetric force on the `Values` span.
- All `DenseColumnMajorMatrixStorage<double>` data is laid out as `Data[col * RowCount + row]`. Column `c` occupies `Data[c*n .. (c+1)*n - 1]` as a contiguous slice — `new ReadOnlySpan<double>(values, c * n, n)`. Row access is strided at stride `n`. For dense residual computation — `r = A·x - b` — the correct vectorized form accumulates each column contribution via `TensorPrimitives.MultiplyAdd(columnSpan, x[col], accumulatorSpan, accumulatorSpan)` over the column-major layout, processing one full n-element column per call. The Cholesky inner loop (`DoCholeskyStep`) operates column-by-column: the rank-1 update `targetCol -= multiplier * sourceCol` maps directly to `TensorPrimitives.MultiplyAdd(sourceColSpan, -multiplier, targetColSpan, targetColSpan)` — the managed Cholesky does not use this; it is the primary vectorization gap in the managed factorization kernel.

[SMALL_DENSE_PROBES]:
- For small dense systems (per-pair precision matrices, covariance estimation, polynomial moment fits), the factorization-as-probe pattern is the correct idiom: allocate a `DenseMatrix` from a `double[n*n]` span via `new DenseMatrix(n, n, data)`, call the appropriate factory, and classify the outcome. Definiteness-by-Cholesky, rank-by-SVD, and condition-by-ConditionNumber are O(n³) but with small constants for n ≤ 10 and no heap overhead beyond the storage array. At n ≤ 4 (3D covariance, 4-point alignment systems), managed factorization paths are faster than provider-switching overhead.
- Moment-system solves via normal equations — fitting polynomials or radial basis functions to scattered data by forming the Vandermonde system `VᵀV x = Vᵀf` — fall into the small dense probe category when the basis dimension n ≤ 10 but the point count m is large. The correct route is `V.TransposeThisAndMultiply(V).Cholesky().Solve(V.TransposeThisAndMultiply(f))` for SPD Gram matrices, with the `Fin<Cholesky<double>>` gate demoting degenerate configurations to a `ConditioningFailure` receipt. For moment systems approaching rank-deficiency (near-collinear basis functions, nearly uniform sample distribution), QR on V directly is more stable at O(mn²) cost; the normal-equations route squares the condition number and must be rejected when `V.ConditionNumber() > 1/eps_cap`.
- For repeated factorizations of identically-sized matrices (MCMC covariance updates, scattered-data interpolation), the clone allocation inside `Dense*.Create` dominates at small n because the managed allocator cannot reuse the previous factor buffer across construction calls. `Cholesky.Factorize(newMatrix)` on a retained handle reuses the existing factor buffer via `Buffer.BlockCopy` — the only in-place refactorization path in the admitted surface. No equivalent `Factorize` mutation method exists on `LU`, `QR`, `Svd`, or `Evd`; those require constructing a new handle each time. For per-query factorization cost in scattered-data evaluation where the coefficient system changes per query, this is the controlling allocation budget.

[CONGRUENCE_AND_DENSIFICATION_CLIFFS]:
- The quadratic densification cliff via `matrix.TransposeThisAndMultiply(matrix)` is not limited to the regression API. Any algorithm that forms AᵀA routes through the managed provider's `MatrixMultiplyWithUpdate` and produces a dense n×n result regardless of whether the input was sparse or rectangular.
- Damped normal equations for iterative registration — the system `(JᵀJ + λI)x = Jᵀr` — form `JᵀJ` via `TransposeThisAndMultiply` and then solve via `Cholesky()`. This squares the condition number: for a Jacobian with condition κ(J), the unregularized Gram matrix has condition κ(J)². The correct Levenberg-Marquardt inner solve stacks the augmented system `[J; √λ I]` as an (m+n)×n dense matrix and applies thin QR directly, avoiding the squaring. The admitted package surface supports this: `J.Stack(sqrtLambdaIdentity).QR(QRMethod.Thin).Solve(b_augmented)`, where `sqrtLambdaIdentity` is an n×n dense matrix with `√λ` on the diagonal.

[REGISTRATION_INNER_SOLVES]:
- Closed-form orthogonal alignment (SVD of the cross-covariance matrix followed by determinant correction for reflection cases) requires forming `H = srcPts.TransposeThisAndMultiply(refPts)`, factorizing via `H.Svd(computeVectors: true)`, and constructing rotation `R = (svd.VT.Transpose() * svd.U.Transpose()).Transpose()`. The reflection check must use `Math.Sign(svd.U.Determinant() * svd.VT.Transpose().Determinant())`, not `svd.Determinant`: `Svd.Determinant` computes `|Π σᵢ|` — always non-negative — and cannot distinguish a reflection. `LU.Determinant` on a 3×3 carries the correct sign; check `det < 0.0`, not `det != 1.0`, because fp rounding means an exact rotation's determinant is only approximately ±1.
- Iterative non-linear registration requires a backtracking line search over the dense inner solve: after computing the Newton or Gauss-Newton step `Δp` from the factorized normal system, evaluate the objective at `p + αΔp` for α ∈ {1, ½, ¼, ...} and accept the first α satisfying the Armijo or sufficient-decrease condition. This costs one forward-model evaluation per backtrack — not a refactorization — so the retained factorization handle from the Cholesky or QR solve is reused across all backtrack steps at zero factorization cost.
- Log-space robust scaling for outlier-contaminated registration: when residuals span several decades, a single global eps_cap for residual demotion is too tight for large residuals and too loose for small ones. The correct form pre-scales each residual by `Math.Log1p(Math.Abs(r[i]))` — log-space compression that assigns finite weight to large residuals without hard rejection. This changes the Gram matrix: `JᵀWJ` with `W[i,i] = 1 / (1 + |r[i]|)` (the derivative of log1p) must be recomputed each iteration because the weights depend on current residuals; the factorization must not be cached across iterations. The resulting n×n symmetric `JᵀWJ` is SPD when J is full rank and all weights positive, so Cholesky remains the correct inner solve, and the `Fin<Cholesky<double>>` admission gate fires on degenerate configurations before iterating.
