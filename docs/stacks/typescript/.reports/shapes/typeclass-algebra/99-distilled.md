# Typeclass Algebra Candidates

[KIND_IS_A_POSITIONAL_RECORD_WHOSE_SLOT_VARIANCE_IS_A_TYPE_LEVEL_FREE_MONOID]:
- `Kind<F, R, O, E, A>` is a four-field positional record `{ In; Out2; Out1; Target }`; a combinator manipulates slots by POSITION while the carrier's `TypeLambda` alone binds each position to a domain channel — `EffectTypeLambda.type = Effect<Target, Out1, Out2>` routes the requirement to the slot the algebra calls `O` (`Out2`), so the slot named `R` is UNUSED by Effect. The `&`/`|` arithmetic is FORCED by slot variance, not carrier content: `In` `Contravariant` (intersected), `Out2`/`Out1` `Covariant` (unioned), `Target` `Invariant` (replaced, never accumulated) — `EitherTypeLambda` makes `Left` a covariant `Out1` so `product` unions `E1 | E2` at the same slot Effect's failure rides, one slot-driven channel-agnostic arithmetic, never per-carrier merge rules.
- `product`/`flatMap`/`coproduct` merge `R1 & R2` over `(&, unit unknown)` and `O1 | O2`, `E1 | E2` over `(|, unit never)` — `&` and `|` are the two free monoids on the lattice of types, so composing two carrier values runs the SAME monoid combine the value owner runs on its content, lifted one level onto the type parameters. `Of.of: <A>(a) => Kind<F, unknown, never, never, A>` is that monoid's TWO-SIDED IDENTITY (`R & unknown = R`, `O | never = O`), not a permissive default, and `Coproduct.zero` carries the IDENTICAL triple — the empty product and the empty alternation seed from one type-level neutral, the value-layer Semigroup/Monoid identity boundary recurring at the channel layer.
- A single-content carrier (`OrderTypeLambda`, `EquivalenceTypeLambda`, `OptionTypeLambda`, `ReadonlyArrayTypeLambda`) populates ONLY `Target`, leaving the channel slots at defaults, so a comparison instance composes through the SAME `SemiProduct` whose signature unions `O1 | O2` — one applicative law spans channel-rich and channel-free carriers, and a hand-spelled lexicographic tuple-order pins the one cell `SemiProduct<OrderTypeLambda>.product` spans generically. `composeK(F)(afb, bfc)` performs the union at the ARROW level before any value flows, fusing two continuations into `(a) => Kind<F, R1 & R2, O1 | O2, E1 | E2, C>` so a pipeline's total requirement and failure compute once at composition time; `productComposition(F, G)`/`mapComposition(F, G)` merge both layers' slots independently in one signature.
- `IdentityTypeLambdaFix<A>` sets `type` to a constant `Identity<A>` (ignoring `this["Target"]`) so a content-FIXED carrier can still present the slot-shaped interface a typeclass demands — `data/Identity` carries `Covariant`/`Monad`/`Traversable` over the polymorphic `IdentityTypeLambda` as bare consts, but `getSemiCoproduct: <A>(S) => SemiCoproduct<IdentityTypeLambdaFix<A>>` returns an instance over the FIXED lambda because the coproduct of two bare values has no structural choice point and must be decided by `S.combine`, the encoding that converts "this instance needs the content first" into one extra type parameter rather than a parallel carrier.
- Boundary — heterogeneous accumulation lives ONLY in the static struct shape: `product` widens two distinct channel sets, but `productMany`/`coproductMany`/`combineMany`/`combineAll` pin ONE `R, O, E, A` across the iterable, so a runtime-length fold over carriers whose failure sets differ silently NARROWS the element type to the common channel — distinct channels survive only through `Product.struct(F)({ a, b })` where each field's channels union into the result.
- Reject — a wrapper re-spelling `flatten`/`zipRight` for one carrier re-derives the free function the four-slot record already owns; an owner hand-merging channels per combinator re-derives what the slot variances state once.

[INSTANCE_IS_A_WITNESS_VALUE_THREADED_ONCE_NOT_A_METHOD]:
- An instance is a record of operations passed to a call site, never a `class extends` heritage re-resolved per value: `Semigroup<A>` is `{ combine, combineMany }`; one `S: Semigroup<A>` threads through a whole computation and every accumulation reads `S.combine`, the rule never restated per site.
- `Semigroup.make(combine, combineMany?)` derives `combineMany` from `combine` as a STRICT LEFT fold `reduce(self, combine)` (`(((self ⊕ a₀) ⊕ a₁) ⊕ a₂)`); the second argument is supplied only when an `Iterable` fold beats left-folding, so the witness ships one operation and the bulk operation is free.
- The interface lattice means a combinator demands its WEAKEST sufficient algebra — `Of` → `Pointed extends Covariant + Of` → `Monad extends FlatMap + Pointed`; `SemiProduct + Covariant` → `SemiApplicative` → `Applicative extends SemiApplicative + Product`; `SemiCoproduct` → `Coproduct`; the recovery duals `SemiAlternative = SemiCoproduct & Covariant` → `Alternative`. `Monoid<A> extends Semigroup<A>` adding `empty` + `combineAll`, so a `Monoid` is accepted wherever a `Semigroup` is, with zero adaptation; passing too weak a witness is a compile error, never a runtime gap.
- Variance witnesses RETARGET an existing instance, never `make` a second: `Covariant.imap(map)` builds `imap` by discarding `from`, `Contravariant.imap(contramap)` by discarding `to` — one interface, two derivations, the discarded arrow naming the variance. `contramapComposition(F, G)` over two contravariant layers yields a COVARIANT `map` (two input-flips restore forward variance). `bindTo`/`tupled` need only `Invariant`, so they run over either.

[STRENGTH_PROMOTION_IS_A_ONE_WAY_TRUST_GATE_FUNNELING_FOUR_IDENTITY_SOURCES]:
- Every promotion from a non-empty accumulation to a unit-bearing one is the SAME act — supply the one missing two-sided identity — funneled through `Monoid.fromSemigroup(S, empty)` whose `combineAll` is `(c) => S.combineMany(empty, c)`: it reuses the operand's exact fold and the only new fact is the seed. The four sources are four ways of computing the second argument, forming a strict ladder by where the datum originates:
  - LITERAL asserted at the declaration — `fromSemigroup(SemigroupSum, 0)`, `fromSemigroup(SemigroupEvery, true)`; the shipped `MonoidSum`/`MonoidEvery` ARE these calls frozen as consts, the only source where the identity can be wrong while every other layer is lawful.
  - RETARGET of an existing unit — `fromSemigroup(Semigroup.imap(M, to, from), to(M.empty))`; `Monoid` exposes NO `imap`, forcing the identity to be re-stated through `to`, lawful only when `to`/`from` are mutual inverses (a clamping `to` silently breaks both associativity and neutrality).
  - BOUND read off the order witness — `Monoid.min(B) = fromSemigroup(Semigroup.min(B.compare), B.maxBound)`; the identity is the LOSING bound (`min(maxBound, x) = x`), read off the `Bounded` witness so a wrong extremum identity is unrepresentable, where `fromSemigroup(Semigroup.min(O), 0)` over a non-max `0` silently clamps.
  - CARRIER native unit — `Option.getOptionalMonoid(S)` returns `Monoid<Option<A>>` with `empty = None` from a unit-LESS `Semigroup<A>`; `Coproduct.getMonoid(F)()` takes NOTHING because the unit is the carrier's `zero`.
- A promotion is provably weaker→stronger because the operand cannot exhibit the new datum and the output declares it; the carrier source uniquely CHANGES the carrier (the identity lives in the `Option` layer, not in `A`) where the literal/bound sources keep the carrier and add a value.
- `Monoid.struct(fields)` does not introduce a fifth source: it COMPOSES the per-field promotions, assembling `empty` field-wise as `{ [k]: fields[k].empty }`, so the record is buildable only when EVERY column carries a unit — a column frozen by `Semigroup.first`/`last` is unit-less and forces the whole struct down to a `Semigroup`.
- The gate composes laws, never synthesizes them: `SemiApplicative.getSemigroup(F)(S)` builds the elementwise combine as `F.map(F.product(self, that), ([a, b]) => S.combine(a, b))` and is associative IFF `F.product` AND `S.combine` are — a non-associative inner `S` yields a defined-but-incoherent lifted semigroup even when `F` is lawful, requiring only `SemiApplicative` because a semigroup has no identity to lift. `Applicative.getMonoid(F)(M) = fromSemigroup(getSemigroup(F)(M), F.of(M.empty))` requires the FULL `Applicative` precisely because the lifted identity is `F.of(M.empty)`, the `Of` half of `Product` — the rung where the gate demands the stronger carrier is exactly the rung where it acquires an identity.
- Identity acquisition is ORTHOGONAL to the associativity premise: `getOptionalMonoid` over a non-associative `S` yields a `Monoid<Option<A>>` whose `None`-identity is genuinely two-sided yet whose `Some ⊕ Some` step is exactly as non-associative as `S` — the strongest identity source cannot rescue the weakest law axis. `SemiApplicative.lift2(F)(f)` lifts any binary `(a, b) => c` into the carrier with full channel accumulation; `Foldable.combineMap(F)(M)(f)` is the witness-driven foldMap where changing the accumulation swaps the `Monoid`, never the traversal.
- Law — the property test asserting associativity (and an `imap` retarget's round-trip inverse) belongs at the LEAF `Semigroup` admission, never at any `fromSemigroup`/`getOptionalMonoid`/`Monoid.min` site: the gate is one-way trust, and a re-check downstream re-derives a fact the source rung already owed.

```typescript
import { SemiApplicative } from '@effect/typeclass'
import * as EffectInstances from '@effect/typeclass/data/Effect'
import * as N from '@effect/typeclass/data/Number'
import type { Semigroup } from '@effect/typeclass/Semigroup'
import { Effect } from 'effect'

const merge = <A>(S: Semigroup<A>, opts: EffectInstances.ConcurrencyOptions): Semigroup<Effect.Effect<A>> =>
    SemiApplicative.getSemigroup(EffectInstances.getSemiApplicative(opts))(S)

const Peak: Semigroup<Effect.Effect<number>> = merge(N.SemigroupMax, { concurrency: 'unbounded' })
const widest: Effect.Effect<number> = Peak.combineMany(Effect.succeed(1), [Effect.succeed(7), Effect.succeed(3)])
```

[IDENTITY_IS_THE_ONE_REIFIED_LAW_AND_ITS_PRESENCE_DOES_NOT_VERIFY_IT]:
- The strength lattice is one bit per rung — does a two-sided identity exist — materialized as which interface a witness inhabits, asked at four layers by ONE discriminant: `combineMany`-vs-`combineAll`, `getSemigroup`-vs-`getMonoid`, `SemiCoproduct`-vs-`Coproduct`, `SemiApplicative`-vs-`Applicative`. Associativity/commutativity/idempotence live in the unchecked premise (the shape `{ combine, combineMany }` records none); identity alone is reified as a MEMBER (`Monoid extends Semigroup` adds `{ empty; combineAll }`), so the boundary is checkable.
- `combineMany`/`combineAll` are the IDENTICAL left-fold differing only in seed — `self` vs `empty` — so the empty fold is not a missing FEATURE but a missing VALUE: `self` is the only value a non-empty fold has, and `empty` is the value that lets the seed exist with no element. The same split recurs at three more layers: `SemiProduct.nonEmptyStruct(F)` constrains `EnforceNonEmptyRecord<R> = keyof R extends never ? never : R` so a zero-field struct is a compile error while `Product.struct(F)` carries no guard because `Of` supplies the empty-struct lift (the empty product is `of` applied to the empty tuple); `SemiCoproduct.coproduct` is the first-success/choice dual to `product` while `Coproduct extends SemiCoproduct` adds `zero<A>()` and `coproductAll`, so `SemiCoproduct.getSemigroup(F)()` lifts the choice into a semigroup whose `combine` is `coproduct` and `Coproduct.getMonoid(F)()` adds `empty = zero`, a fallback chain folded over the coproduct (mon/semi)oid, never a hand-written sentinel.
- `data/Option` carries the FULL `Coproduct`/`Alternative` because `None` IS the `zero` (`coproduct = isSome(self) ? self : that` is associative, idempotent, two-sided); `data/Either` carries ONLY `SemiCoproduct`/`SemiAlternative` because a `Left` requires a witness `E` and there is no neutral failure to serve as `zero` — `coproductAll` is uninhabitable, foreclosed at the type level, so a recovery fold over `Option` admits the empty case and one over `Either` requires at least one alternative, the wrong strength surfacing as an absent export, never a runtime empty-list crash.
- Paradox — the reified field's PRESENCE does not verify its own law: `combineAll([])` returns `empty` UNCOMBINED and `combineAll([a])` touches one side of one combine, so a left-neutral-only unit passes every empty and singleton test and corrupts ONLY a multi-element `combineAll` where `empty` lands at the head and a later element re-reads the right side — the checker confirms an `empty` of the right TYPE and asserts nothing about two-sidedness, the bit being SET necessary but not sufficient.
- Boundary — the empty-fold property test (`combine(a, empty) === a === combine(empty, a)`) is irreplaceable AND distinct from the associativity test (a left-neutral-only unit is associative and type-correct); a fact encoded as ABSENCE is discharged by the checker with no test (the foreclosed combinator cannot be named) while a fact encoded as PRESENCE is a premise it TRUSTS, so a missing single-member CONST removes a WHOLE derived family (foreclosing `Foldable`'s one `reduce` subtracts `combineMap`/`reduceKind`/`coproductMapKind`/`toArray`/`toArrayMap` at once) where a missing FACTORY removes exactly one operation — the export surface a one-sided gate concentrating all verification at the leaf witness admission.

[THE_FOUR_LAWS_ARE_FOUR_INDEPENDENT_EDIT_LICENSES]:
- Each property licenses exactly ONE fold-body edit and a witness carrying one authorizes that edit while carrying none of the others: associativity licenses RE-PARENTHESIZING (tree/parallel fold), commutativity REORDERING the sequence, idempotence DEDUPLICATING repeats, a two-sided identity DROPPING-AND-SEEDING the empty case. Name the edit a refactor performs and the license it cashes is fixed; a reassociating edit over a non-associative combine and a reordering edit over a non-commutative one are the same fault — cashing a license the instance never issued — both producing a defined, type-checking, silently-wrong value.
- Each license maps one-to-one onto a fold EXECUTION strategy: associativity → tree/parallel evaluation, commutativity → unordered ingestion (`Set`/hash-grouped stream), idempotence → at-least-once delivery tolerance (retry-safe), identity → empty-and-chunk-boundary seeding. A parallel/chunked `combineAll` needs BOTH associativity (to split) AND identity (to seed each chunk) — the `Monoid` contract — and out-of-order chunk completion additionally needs commutativity, so an unbounded-concurrency `getApplicative`-lifted fold is reorder-safe ONLY when the inner value monoid is commutative.
- The shipped instances partition by which licenses they issue — the table a refactor consults before reordering, deduping, or tree-folding:

| instance source | assoc | comm | idem | identity | reorder-safe | dedup-safe |
| :-------------- | :---: | :--: | :--: | :------: | :----------- | :--------- |
| `Number.MonoidSum` / `BigInt.MonoidSum` / `MonoidMultiply` | yes | yes | no | `0`/`1` | yes | no |
| `Number.MonoidMin` / `MonoidMax` (extremum) | yes | yes | yes | bound | yes | yes |
| `Boolean.MonoidEvery` (∧) / `MonoidSome` (∨) | yes | yes | yes | `true`/`false` | yes | yes |
| `Boolean.MonoidXor` / `MonoidEqv` (↔) | yes | yes | NO | `false`/`true` | yes | NO |
| `String.Monoid` (concat) | yes | NO | no | `""` | NO | no |
| `Semigroup.first` / `last` | yes | NO | yes-on-equal | none | NO | yes-on-equal |
| `Order.combine` (comparator monoid) | yes | NO | yes | `empty()` | NO | yes |
| `Equivalence.combine` (∧ of relations) | yes | yes | yes | always-true | yes | yes |

- The reorder-safe and dedup-safe columns DIVERGE on exactly the `Xor`/`Eqv` rows: both are associative AND commutative (reorder-safe) but NOT idempotent (`x ⊻ x = false`), so `MonoidXor.combineAll` over a multiset counts PARITY — even multiplicity cancels. Removing a duplicate predicate "for clarity" is sound under `MonoidEvery` and corrupts under `MonoidXor`, and reaching for `MonoidXor` where `MonoidSome` (∨) was meant passes every distinct-element test because XOR and OR agree until a value repeats — the parity collapse invisible until the fold runs over duplicated production data.
- Hazard — lifting `String.Monoid` (associative, NON-commutative) through an unbounded-concurrency applicative folds a concatenation whose chunk-completion order scrambles the result: the type checks, the fold runs, the string assembles nondeterministically — non-commutativity surfacing only under the concurrency the factory admitted. The predictor composes by the WEAKEST contributing column: a struct fold is reorder-safe iff EVERY column's value algebra is commutative, so one `Semigroup.last` discriminant poisons the whole record's reorder license even amid commutative `MonoidSum` columns.

[ORDER_AND_EQUIVALENCE_ARE_COMPARATOR_MONOIDS_THAT_INVERT_THE_REORDER_LICENSE]:
- `effect/Order` is a complete monoid OVER COMPARATORS: `combine`/`combineMany`/`combineAll`/`empty()` all consume and produce `Order<A>`. `combine(self, that)` is LEFTMOST-DECISIVE (`out !== 0 ? out : that(a1, a2)`) and IS associative with two-sided identity `empty() = make(() => 0)` — the always-equal comparator, provably two-sided because it can neither pre-empt nor be pre-empted, so `combineAll([])` is the "preserve input order" total order, never an undefined seed. `Order.product`/`all`/`tuple`/`array` apply the SAME law positionally (lexicographic = order monoid folded over slots), one algebra at three arities.
- The extremum comparator-monoid occupies the strength rung gated by whether a zero exists: `Semigroup.min(O)`/`max(O)` lift any `Order<A>` to a last-extremum semigroup WITHOUT bounds, while `Monoid.min(B)`/`max(B)` require `Bounded<A>` (`{ compare: Order<A>; maxBound; minBound }`) because the identity IS a bound — the SAME extremum logic at two strengths. `Bounded` derives `between(B)`/`clamp(B)` (membership, saturating projection) and `reverse(B)` (flips `compare`, swaps both bounds atomically, so `Monoid.max(Bounded.reverse(B)) === Monoid.min(B)`) off the one witness.
- `Equivalence<A>` is the callable `(self, that) => boolean` carrying its own monoid where `combine` is logical AND (`make((x, y) => self(x,y) && that(x,y))`), `empty` the always-true relation, so a multi-field equality is `combineAll([byId, byVersion])` — every field must agree, never a hand-spelled `&&` chain. `Equivalence.make` wraps as `self === that || isEquivalent(...)`, reflexive-by-reference even when the predicate is not.
- The fork is exact and the two reads identically: `Order.combine` is NON-commutative (short-circuit on the FIRST decisive operand re-prioritizes the sort), so key ORDER in `combineAll` is a CORRECTNESS decision — `combine(byA, byB) !== combine(byB, byA)`; `Equivalence.combine` IS commutative (conjunction), so its short-circuit on the first DISAGREEING relation makes field order a pure PERFORMANCE lever (cheapest-discriminating first prunes earliest). The same `combineAll(fieldRows)` shape is a reorder bug on one owner and the recommended tuning on the other.
- `Order.mapInput(O, f)` / `Equivalence.mapInput(self, f)` are the contravariant retarget; `effect/Order` and `effect/Equivalence` expose contravariance as the standalone `mapInput`, NOT a `Contravariant` instance — neither exports a typeclass instance over its `TypeLambda`, only the lambda, so the comparison family retargets through the free function and slots into generic machinery only as the `F` carrier its `product`/`combine` already drive.

```typescript
import { Order, Equivalence, Equal } from 'effect'

type Owner = { readonly rank: number; readonly tier: number; readonly tag: string }
const byField = <K extends keyof Owner>(k: K, O: Order.Order<Owner[K]>): Order.Order<Owner> => Order.mapInput(O, (o) => o[k])

const ranked: Order.Order<Owner> = Order.combineAll([byField('tier', Order.number), byField('rank', Order.number)])
const reordered: Order.Order<Owner> = Order.combineAll([byField('rank', Order.number), byField('tier', Order.number)])
const same: Equivalence.Equivalence<Owner> = Equivalence.combineAll([
    Equivalence.mapInput(Equal.equivalence<string>(), (o: Owner) => o.tag),
    Equivalence.mapInput(Equivalence.number, (o: Owner) => o.rank),
])
const diverge = ranked({ rank: 1, tier: 2, tag: 'a' }, { rank: 2, tier: 1, tag: 'a' }) === reordered({ rank: 1, tier: 2, tag: 'a' }, { rank: 2, tier: 1, tag: 'a' })
```

[INTERCALATE_AND_FIRST_LAST_OCCUPY_THE_RUNG_WHERE_THE_BIT_IS_UNSETTABLE]:
- `Semigroup.intercalate(S, sep) = make((self, that) => S.combineMany(self, [sep, that]))` is the ONE combinator with NO `Monoid` counterpart: any candidate unit `e` forces `combine(e, a)` to emit `sep` before `a`, contradicting `combine(e, a) = a` — so the bit is UNSETTABLE in principle (not merely absent for a carrier), and the missing `Monoid.intercalate` encodes "associative, no unit" permanently. `Monoid.reverse(M)` CROSSES the boundary because argument-flip preserves two-sidedness, so which combinators a `Monoid` exports beyond `Semigroup` is the executable map of which operations preserve the unit.
- `Semigroup.first()` (`a => a`) and `last()` (`(_, b) => b`) are the lawful no-merge instances — associative, non-commutative, idempotent-on-equal — and unit-LESS by construction (a projection always keeping one operand has no vanishing `e`); `last`'s bespoke `combineMany` loops keeping the FINAL element (cannot left-fold `combine`, which keeps the first), proving the `combineMany` rung is reachable by an instance the `combineAll` rung provably is not. `Semigroup.constant(a)` (`() => a`) is the INVERSE degeneracy — an ABSORBING element (`combine(a, x) = a` discards `x`), the algebraic opposite of a neutral.
- `Semigroup.imap(S, to, from)` makes `Semigroup` itself an invariant owner — a `Semigroup<number>` becomes `Semigroup<Branded>` by `imap(wrap, unwrap)`, the lawful path to give a value object its parent's algebra without re-implementing `combine`, provided the round-trip is identity; `reverse` on a COMMUTATIVE semigroup is the identity transform, a no-op the checker permits but the algebra empties of meaning, so `reverse` carries information ONLY over a non-commutative owner.

[THE_TWO_WITNESS_GRID_IS_A_SUM_OF_DECLARATIONS_OVER_A_PRODUCT_OF_CELLS]:
- `Traversable<T> extends TypeClass<T>` directly — NOT `Covariant`, NOT `Foldable`; `Foldable<T>` carries only `reduce`, `Filterable<F>` only `partitionMap`/`filterMap` — so the type lattice REFUSES to bundle walk, fold, and filter into one heritage, and the product space is a grid of independently-present cells, each an export const read off before any call. `traverse(F: Applicative<F>)` takes TWO witnesses: the outer owns the walk shape, the inner owns what accumulates, and swapping `F` reshapes the result with zero change to `T` or `f`.
- The three fold combinators over one `Foldable<F>` are ONE `F.reduce` with a different `(seed, step)` per algebra witness: `combineMap(F)(M)` seeds `M.empty` and steps `M.combine`; `reduceKind(F)(G: Monad<G>)` seeds `G.of(b)` and sequences via `flatMap`; `coproductMapKind(F)(G: Coproduct<G>)` seeds `G.zero()` and alternates via `coproduct` — the structure witness supplies iteration, the algebra witness supplies BOTH seed and step, and the identity boundary surfaces on the SECOND witness (`coproductMapKind` demands FULL `Coproduct` for `G.zero()`, not `SemiCoproduct`).
- The grid's economy is arithmetic: `m` structure owners × `n` algebra witnesses span `m × n` reachable cells from `m + n` declarations where the naive form pays `m × n` hand-written loops, each welding one (structure, algebra) pair into one body. `*Composition` derivers extend this to `m × k × n` cells from `m + k + n` declarations — a stacked structure's traversal grows by one composition call, never a rewritten recursion per (outer, inner) pair.
- The inner axis uniquely admits a POLICY sub-witness the algebra axis forbids: `data/Effect` ships NO `Traversable`/`Foldable`/`Filterable` (uninhabitable as a walkable container) yet ships `getApplicative(options?)` as a FACTORY, so `traverse(getApplicative({ concurrency: 'unbounded' }))` evaluates per-element effects in parallel — same walk, same `f`, the parallelism an axis-of-an-axis no fold combinator can express.
- The structure axis is not free over the algebra axis — the carrier's shape forecloses cells, read off as a present-or-absent const: `data/Either` carries `Foldable`/`Traversable` but NO `Filterable` (a single-success slot cannot DROP without a neutral `Left`); `data/Option` carries the FULL filterable AND coproduct stack because `None` is BOTH the drop target and the `zero` identity; `data/Array` carries everything EXCEPT `Coproduct`/`Alternative`; `data/Record` carries `Traversable`/`Filterable`/`Covariant` but NO `Foldable` const. An algorithm constrained on `Traversable<T> & Filterable<T>` accepts `Array`/`Option` and REJECTS `Either` at the call site — the checker discharging a grid coordinate, never a runtime assertion.
- `Filterable<F>` reduces filtering to TWO discriminating-carrier members — `partitionMap(f: (a) => Either<C, B>)` splits into `[F<B>, F<C>]` by tag, `filterMap(f: (a) => Option<B>)` keeps `Some` — so `filter`/`partition`/`compact`/`separate` all derive (`filter` is `filterMap` of a predicate-gated `Option`, `separate` is `partitionMap(identity)`); `filter`/`partition` carry refinement overloads `(a) => a is B` that NARROW the output carrier. Trap — `partitionMap` returns `[Right/accepted, Left/rejected]`, the INVERSE of the `Either` value's left-then-right reading, and `partition`'s refinement overload returns `[original C, narrowed B]`, so the narrowed/accepted half is the SECOND slot and `const [rejected, accepted] = partitionMap(f)` silently swaps whenever `B` and `C` unify.
- Hazard — passing `Either.Applicative` as the inner traverse witness where every fault must survive keeps only the LEFTMOST `Left` (`product` returns the first non-`Right`), so a validator that should accumulate all faults silently reports one; both `Either.Applicative` and a lifted accumulating applicative satisfy the same `Applicative<F>` constraint, so the bug surfaces as missing faults, never a type error.

```typescript
import { Foldable } from '@effect/typeclass'
import * as ArrayInstances from '@effect/typeclass/data/Array'
import * as OptionInstances from '@effect/typeclass/data/Option'
import * as N from '@effect/typeclass/data/Number'
import { Option } from 'effect'

const walk = ArrayInstances.Foldable
const score = (s: string): number => s.length
const probe = (s: string): Option.Option<number> => (s.length > 2 ? Option.some(s.length) : Option.none())

const total: number = Foldable.combineMap(walk)(N.MonoidSum)(score)(['<value-a>', '<value-bb>'])
const peak: number = Foldable.combineMap(walk)(N.MonoidMax)(score)(['<value-a>', '<value-bb>'])
const firstWide: Option.Option<number> = Foldable.coproductMapKind(walk)(OptionInstances.Coproduct)(probe)(['<value-a>', '<value-bbb>'])
```

[THE_FUSED_FILTER_CELL_DERIVES_ITS_WITNESS_BUDGET_AS_A_GRID_COORDINATE]:
- `TraversableFilterable<T>` fuses walk, inner-applicative effect, and filter into ONE pass — `traverseFilterMap(F)(f: (a) => Kind<F,…,Option<B>>)` walks, runs each element's `F`, keeps inner `Some`s, where the unfused form is `traverse(F)` then `filterMap` (two passes, two intermediate carriers); `traversePartitionMap(F)(f: (a) => Kind<F,…,Either<C, B>>)` returns `Kind<F,…,[T<B>, T<C>]>`, the two-channel dual.
- The deriver's constraint difference IS the partition-versus-filter cost made checkable: `traverseFilterMap(T)` derives from `Traversable<T> & Filterable<T>`, but `traversePartitionMap(T)` ADDS `Covariant<T>` — a partition's single walk rebuilds TWO output carriers and `Covariant` re-maps the one traversal result into the second, the extra witness the algebraic price of the second output channel. `filterMapComposition(F: Covariant, G: Filterable)` demands `Covariant` of the OUTER (only re-maps) and `Filterable` of the INNER (the drop happens at the leaf), so an outer that is only `Covariant` still composes — the foreclosure is per-position inside the deriver, not carrier-wide.
- `traverseFilter(F)(predicate: (a) => Kind<F,…,boolean>)` is where the boolean knob the grid otherwise forbids is LAWFULLY re-admitted: the keep-test runs INSIDE the inner applicative, accumulating `F`'s channels while the refinement narrows `B extends A` — a `(a) => boolean` permitted only because wrapped in the inner witness, the synchronous-second-pass form the fusion deletes.

```typescript
import * as ArrayInstances from '@effect/typeclass/data/Array'
import * as O from '@effect/typeclass/data/Option'
import { Either, Option } from 'effect'

const split = ArrayInstances.TraversableFilterable.traversePartitionMap(O.Applicative)
const classify = (n: number): Option.Option<Either.Either<number, number>> =>
    n === 0 ? Option.none() : Option.some(n > 0 ? Either.right(n) : Either.left(-n))

const halves: Option.Option<[ReadonlyArray<number>, ReadonlyArray<number>]> = split([5, -2, 0, 8], classify)
```

[THE_DO_BUILDER_DEGRADES_TO_THE_WEAKEST_SUFFICIENT_WITNESS_PER_STEP]:
- The struct-accreting notation is five free functions each closed over the MINIMUM interface its step consumes, so a carrier exposes the EXACT subset of the five its instance set satisfies: `Do(Of)` seeds the empty struct, `bindTo(Invariant)`/`tupled(Invariant)` wrap a scalar, `let(Covariant)` accretes a field computed PURELY from prior fields, `andThenBind(SemiProduct)` accretes an INDEPENDENT carrier field, `bind(Chainable)` accretes a DEPENDENT carrier field. The heritage encodes a strict climb (`Pointed extends Covariant, Of`; `Chainable extends FlatMap, Covariant`; `SemiProduct extends Invariant` NOT Covariant; `Product extends SemiProduct, Of`), and a missing const is the absent builder step surfaced as a combinator failing to type-resolve.
- `let`/`bind`/`andThenBind` constrain the carrier `A extends object` and the name `Exclude<N, keyof A>`, so the builder grows strictly monotonically by one provably-fresh key per step with result `{ [K in keyof A | N]: K extends keyof A ? A[K] : B }`; `bind`'s continuation alone is typed `(a: NoInfer<A>) => ...`, freezing the prior fields before the dependent carrier is built FROM `A`, where `let`'s synchronous body and `andThenBind`'s fixed `that` carry no such suppression — the three steps differ precisely in how each relates to the prior struct.
- A contravariant owner (`data/Predicate` carries `Of`/`Invariant`/`SemiProduct`/`Product`/`Contravariant` but NO `Covariant`/`FlatMap`) degrades to a SHARP ceiling: `andThenBind(SemiProduct)` accretes an independent predicate field but `let(Covariant)` is foreclosed (no forward `map` to compute a field FROM prior fields) and `bind` is foreclosed (a predicate cannot SEQUENCE). Its real growth axis is a different composition — `Contravariant.contramap(leaf, accessor)` retargets a leaf predicate onto a field, folded through the conjunction monoid `getMonoidEvery<A>()`, so a record validator is the every-monoid fold of field-retargeted leaf predicates, not the do-builder's struct of predicates.
- `zipRight` is the ONLY combining function `FlatMap<F>` exports beyond derivation (keeping the SECOND value needs no re-read of the first); `zipLeft`/`tap`/`bind` live on `Chainable<F> = FlatMap<F> & Covariant<F>` because re-surfacing a held value through `map` needs the `Covariant` half — the placement is the executable proof of what each operation costs. `zipRight`/`zipLeft` ALSO exist on `SemiApplicative` with the same signatures but run INDEPENDENTLY via `product` (the chainable forms run DEPENDENTLY via `flatMap`), so the selection is the import, not a flag — the dependence axis surfaces as which module the combinator is read from.

```typescript
import { Invariant, Covariant, Chainable } from '@effect/typeclass'
import * as OptionInstances from '@effect/typeclass/data/Option'
import { Option, pipe } from 'effect'

const seed: (s: Option.Option<number>) => Option.Option<{ readonly raw: number }> = Invariant.bindTo(OptionInstances.Invariant)('raw')
const label = (a: { readonly scaled: number }): Option.Option<string> => (a.scaled > 0 ? Option.some('<value-a>') : Option.none())
const built: Option.Option<{ readonly raw: number; readonly scaled: number; readonly tag: string }> = pipe(
    Option.some(7),
    seed,
    Covariant.let(OptionInstances.Covariant)('scaled', (a) => a.raw * 2),
    Chainable.bind(OptionInstances.Chainable)('tag', label),
)
```

[THE_EXPORT_FORM_IS_A_FIVE_WAY_WITNESS_SIGNATURE_OF_THE_FREE_DATUM]:
- An instance's export FORM declares what the carrier cannot decide alone, read off with zero implementation inspection: a BARE CONST means the behavior is fully fixed by the carrier (`Option.Monad`); a `get*(subAlgebra)` factory means an inner value algebra is undetermined (`getMonoidUnion(M)`, `getOptionalMonoid(S)`); a `get*(options)` factory means a runtime POLICY is unfixed (`getApplicative(options)`); a `get*<K extends string>()` factory means the carrier's OWN type-lambda shape is parameterized (`getCovariant<K>()`); a `<A>()` phantom factory means only an element type floats (`Array.getMonoid<A>()`, `getMonoidEvery<A>()`); an ABSENT export means the operation is uninhabitable. The phantom `<A>()` versus the value parameter is the sharpest cut: `getMonoidEvery<A>()` takes NO value because `constTrue` is determined by the operation, while `getMonoidUnion(M)` takes a real `Monoid<A>` because the per-key merge is not.
- A constant instance composes UP the heritage with zero new datum (`Monad extends FlatMap, Pointed`, so an owner exporting both consts exports their join `Monad` as a const — `map`/`of`/`flatMap` each have one lawful behavior), while a factory STOPS at the free datum: `data/Effect`/`data/Micro` ship `Monad`/`FlatMap`/`Chainable` as CONSTS (sequential evaluation FORCED by data dependence, no free datum) but `getSemiProduct`/`getApplicative` as FACTORIES over `ConcurrencyOptions = { concurrency?: number | "unbounded" | "inherit"; batching?: boolean | "inherit" }` (`Micro` drops `batching`) because independent composition admits a parallelism choice the monad cannot — the channel that selects the algebra also selects whether the instance can be a const, splitting the stack at the dependence seam. The distinction is INVISIBLE in the consuming pipeline body — both reach the fold as a fully-formed witness, so the policy is resolved at the witness boundary and the fold never re-decides it.
- The factory parameter is ALWAYS a domain value carrying its own behavior — a `Semigroup`, a `Monoid`, a `Bounded`, a `ConcurrencyOptions`, a phantom key witness — never a boolean knob whose combinations the instance re-derives; `data/Record`'s `getMonoidUnion` (union, has identity `{}`) versus `getSemigroupIntersection` (intersection, NO neutral — the would-be unit must contain every key, an absorbing element) selects set behavior by WHICH FACTORY is read and the per-key merge by which value witness is fed — two orthogonal axes, both instance selection plus one parameter.

[ONE_OWNER_DERIVES_THE_ALGEBRA_FAMILY_AND_THE_DRIFT_IS_THE_REJECT]:
- A value owner's first declaration is the only place a comparator, accumulator, identity, or hash key-set is spelled: from one `Schema.Class<Self>(fields)` / `Data.taggedEnum` / `Data.struct`, the derivations `Schema.equivalence(schema)`, `Struct.getOrder(fieldOrders)`, `Struct.getEquivalence(fieldEquivs)`, `Semigroup.struct(fieldSemis)`, `Hash.structureKeys(value, keys)` all yield from the same `keyof` projection — none re-states the field shape. The collapse deletes the four-link loose chain (`const fields` → `interface Shape` → `const eq` → `const ord` → `const merge`) where every link after the first re-reads the field list and drifts.
- The fan is asymmetric by what the owner CAN carry: the schema AST has annotation slots `EquivalenceAnnotationId`/`ArbitraryAnnotationId`/`PrettyAnnotationId`/`JSONSchemaAnnotationId` but NO `OrderAnnotationId` — ordering is the one surface the schema cannot auto-derive, so `Struct.getOrder` over per-field `Order` is the sanctioned assembly, never a `Schema.order` that does not exist; a field's wrong deep-equality is corrected by an `equivalence: () => Equivalence<A>` annotation INSIDE the owner (a closed thunk), folded through the conjunction monoid, never a parallel `Equivalence.make` beside it.
- `Struct.getOrder`/`getEquivalence`/`Semigroup.struct`/`Monoid.struct` are ONE shape — `Record<keyof R, Instance>` → `Instance<record>` — at three algebras, but completeness has a boundary: the struct lifts INFER their key-set from the argument map, so a partial map silently yields a partial-record instance; the completeness check fires only when the result is annotated `Order.Order<T>` with `T = Schema.Schema.Type<owner>`. `Schema.equivalence` is field-complete for FREE (its result type is fixed by the schema), the asymmetry exactly the absent `OrderAnnotationId` — the per-field `Order` map is the one structure a property test over the full field set must guard.
- Law — the collapse concentrates the law obligation onto the LEAF instances the owner feeds: `Semigroup`/`Equivalence` carry no law field, so one unlawful leaf (a tolerance `|x - y| < ε` annotated on one field) folds cleanly through `Schema.equivalence`'s conjunction yet violates transitivity, the multiplicative fan working in reverse as a fault-amplifier — the property test admitting the leaf guards the entire chain at one point. The rejected shape is any parallel hand-written second comparator living BESIDE the owner: the next field updates the owner-derived chain and forgets the loose one.

```typescript
import { Schema, Struct, Order, Equivalence } from 'effect'

class Owner extends Schema.Class<Owner>('Owner')({
    rank: Schema.Number,
    label: Schema.String.annotations({ equivalence: () => Equivalence.strict<string>() }),
    span: Schema.Number,
}) {}
type T = Schema.Schema.Type<typeof Owner>

const fromOwner = Schema.equivalence(Owner)
const byField: { readonly [K in keyof T]: Order.Order<T[K]> } = { rank: Order.number, label: Order.string, span: Order.reverse(Order.number) }
const ranked: Order.Order<T> = Struct.getOrder(byField)
const same: Equivalence.Equivalence<T> = Struct.getEquivalence({ rank: Equivalence.number, label: Equivalence.strict<string>(), span: Equivalence.number })
```
