# Effect-Rail Resource Scoping And State-Cell Lifting

[ENV_OWNERSHIP_ALGEBRA]:
- The effect runtime carries one ambient environment record per evaluation that bundles four orthogonal capabilities: a resource registry, a cancellation token, the token source, and a captured synchronization context. Lifetime is governed by a single `int Own` bit-field — bit `1` marks ownership of the source, bit `2` marks ownership of the registry — and disposal is gated on those bits: a child that shares its parent's source disposes neither, a child that shares the registry never frees the registry. This bit-field is the entire mechanism by which nested scopes decide what survives scope exit; there is no reference counting and no parent back-pointer on the environment itself.
- Three child-environment projections compose from the same factory with different sharing choices, and each is the correct tool for a distinct scoping intent: a fully-isolated scope allocates a new registry and a new linked source (independent resources, independent cancellation); a resource-isolated scope allocates a new registry but shares the source (its own resources, cancelled by the same token as the parent); a cancel-isolated scope shares the registry but allocates a new linked source (shared resources, independently cancellable sub-tree). Choosing the wrong projection is the dominant scoping bug: wrapping a sub-tree in the resource-isolated projection when the intent was to cancel it independently leaves the sub-tree bound to the parent token, and cancelling it tears the parent.
- The linked source is established by the factory only when the new environment owns its source and the inherited token can be cancelled: it registers a callback that cancels the child source when the parent token fires, then rebinds the working token to the child source's token. Parent-to-child cancellation therefore propagates exactly one direction; cancelling a child never cancels its parent, which is why a cancel-isolated sub-scope is the safe unit for speculative or timeout-bounded work whose failure must not abort the enclosing computation.

[RESOURCE_REGISTRY_MECHANICS]:
- The resource registry is itself a concurrent trie cell keyed by the resource instance as a bare object reference, holding a closed family of tracked-resource cases — synchronous-disposable, async-disposable, and free-function — each carrying the release arrow for its case. Three consequences follow directly from reference-identity keying: re-acquiring the same instance is a silent insert-if-absent no-op, two structurally-equal but distinct instances are two independent entries, and a value-type resource boxes to a fresh key on every acquire so its release fires once per acquire call rather than once per logical value.
- Release on a registry miss walks the parent chain: a scope releases what it owns, then delegates upward, so a resource acquired in an outer scope can be released explicitly from an inner scope by upward delegation, but is otherwise freed only when its owning scope is disposed. This upward delegation is the only cross-scope coupling the registry has; acquisition never consults the parent, so a child scope's registry is a strict local set and the parent's entries are invisible to the child's bulk teardown.
- Bulk teardown iterates the trie in hash-bucket order, not acquisition order. There is no last-in-first-out guarantee and no insertion-order guarantee; any teardown that depends on releasing a dependent before its dependency must encode that ordering inside a single composite resource's release arrow rather than relying on two separate registrations. This is the sharpest correctness trap in the whole resource surface, because the common mental model imports stack-unwind ordering from language-level scope blocks that the registry does not provide.
- Disposal runs every release arrow on a freshly-constructed environment carrying a non-cancellable token, regardless of whether the ambient token was already cancelled. Cleanup is therefore guaranteed to run to completion under cancellation — but a release arrow that itself honors a token observes the non-cancellable one and cannot be interrupted, so a disposer must never embed long token-aware work expecting the original cancellation to cut it short.

[ACQUIRE_EAGER_RELEASE_DEFERRED]:
- A scoped-acquire combinator registers its resource on the ambient registry the instant the acquire effect resolves, then continues the computation; the release arrow is not run at that point — it sits in the registry until the owning scope is disposed. The acquire is eager and the release is deferred to the nearest enclosing auto-freeing boundary, which means a bare scoped-acquire with no surrounding bracket holds its resource alive until the outermost run completes. Resource pressure under a long-running root effect is almost always this: many eager acquires with no intervening bracket to bound their release.
- The auto-freeing boundary projects the resource-isolated child environment and disposes that child's owned registry on both success and failure at the end of the bracketed effect's run — the "ultimate using" over the effect tree. Its fail-only sibling projects the same child environment but disposes the registry only on the failure path; a successful result keeps its acquired resources alive so they can escape the scope into the caller's registry. The success-path divergence between the two is the load-bearing distinction: one frees unconditionally, the other hands successful resources back.

```csharp
// the two scoping primitives differ only in success-path release
IO<Handle> openAndKeep = AcquireScoped(open, h => releaseIO(h)).BracketFail(); // survives on success
IO<Result>  openAndUse  = AcquireScoped(open, h => releaseIO(h)).Map(use).Bracket(); // freed at scope end
```

[SCHEDULE_RETRY_RESOURCE_LAW]:
- Iterated effects are driven by a lazy duration transformer whose run produces a lazy sequence of delays; a zero delay is prepended so the first iteration runs with no wait and subsequent waits come from the schedule. Each iteration yields through a cancellation-observing delay step, so a cancelled ambient token short-circuits the iteration loop between attempts without an explicit predicate.
- The repeat family wraps each iteration in the unconditional auto-freeing boundary and the retry family wraps each attempt in the fail-only boundary, and this single wiring choice dictates the entire resource contract of iteration. Repeat frees an iteration's resources at the end of every iteration including the successful final one — a repeated effect therefore cannot acquire-and-return a resource, because the value escapes but its backing resource is already released. Retry frees only failed attempts' resources before the next attempt and retains the successful attempt's resources, so retries never accumulate leaked handles and a final success hands its resource to the caller's scope.
- The practical inversion: stacking the unconditional boundary under a retry — bracket-then-retry over a resource-returning effect — discards the resource on the successful attempt because the inner unconditional boundary frees it before the retry combinator sees success. A resource-producing effect that must survive its retry loop is wrapped by the retry combinator directly, which supplies the correct fail-only boundary itself; an explicit unconditional bracket must never be interposed between the acquire and the retry.

[STATE_CELL_LIFTING_DISCIPLINE]:
- Lifting an independent-ownership cell into the effect rail defers only the spin-loop execution, not the loop's contention semantics: the lifted swap is a thin lift over the synchronous compare-and-exchange spin, evaluated on the running thread at effect-evaluation time. The swap function may run several times under contention exactly as in the synchronous form, so it must stay side-effect-free even inside the effect, and the lift gives no scheduling, no yielding, and no back-pressure.
- The lifted-value setter on both the independent cell and the transactional reference is a latent no-op: it binds the supplied effect to a swap and discards the resulting effect without evaluating it, so assigning an effect through the value-as-effect setter performs no write. The only correct write paths are the lifted swap combinator or the synchronous swap; the value-as-effect property is read-only in practice despite exposing a setter. This is the single most common silent failure when threading cell writes through an effect pipeline.
- There is no transaction-as-effect primitive. Software-transactional commit is exposed only synchronously, and its commit is a blocking spin-retry loop; composing a transaction into an effect requires an explicit lift around the synchronous transaction call, and that lift blocks the evaluating worker for the full retrying-commit duration without yielding to the async scheduler. A transactional multi-cell update inside an effect pipeline is a synchronous island, and sizing it against the effect runtime's worker budget is a real constraint, not a theoretical one.
- The transactional reference's transaction handle rides an async-local slot, so a task spawned inside a transaction inherits the same handle by execution-context capture and can mutate the shared transaction object across threads — producing undefined commit behavior rather than a clean throw. Composing structured concurrency under a transaction is therefore unsafe by construction; the transaction is a single-threaded synchronous unit and any fan-out inside it must avoid touching transactional references through the inherited handle.
