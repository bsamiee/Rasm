# Aspect Architecture — Frontier

[EXTENSION_BLOCKS_AS_HKT_OPERATOR_SURFACES]:
- C# 14 extension blocks are the mechanism that makes constrained operators on higher-kinded types compilable. The syntax `extension<E, F, A>(K<F, A> _) where F : Fallible<E, F>` declares an extension over `K<F, A>` with generic constraints, enabling `operator |` to be defined directly on the carrier type with trait constraints — impossible with C# 13 extension methods, which require operator parameters to include the declaring type. The decompiled `FallibleExtensions` and `FinalExtensions` in LanguageExt 5 use exactly this form; the `$T0`, `$T1` placeholders in decompiled output are ilspycmd's representation of extension block type parameters.
- `MonadUnliftIOExtensions` defines the entire retry/repeat/bracket/timeout/post/uninterruptible surface in a single extension block `extension<M, A>(K<M, A> ma) where M : MonadUnliftIO<M>`, exposing `LocalIO`, `UninterruptibleIO`, `PostIO`, `TimeoutIO`, `BracketIO` (three overloads), `RepeatIO`, `RepeatWhileIO`, `RepeatUntilIO`, `RetryIO`, `RetryWhileIO`, and `RetryUntilIO` as receiver-syntax methods. The `FoldIO`/`FoldWhileIO`/`FoldUntilIO` family remains as classical `this K<M, A>` static extension methods because their additional `S` state and predicate overloads prevent uniform representation inside one extension block constraint.
- The extension block form is load-bearing for constrained operator definition: you cannot define a constrained `operator |` on a generic type via classical extension methods because C# requires operator parameters to include the containing type. Extension blocks lift this restriction by making the block's receiver type the containing type at the constraint level. Any domain operator aspect constrained to a trait requires an extension block in C# 14.

[COMPOSITION_TIME_POLICY_FRONTIER]:
- `catchOf<E, M, A>` requires `where E : Error, M : Fallible<Error, M>, Monad<M>`. It chains through all `E`-subtype matching errors in the error structure via `ForAllM` — sequential monadic `Bind` over each matching error. On `IO`/`Eff`, a single error is the root, so `catchOf` handles it directly. `catchOfFold<E, M, A>` additionally requires `MonoidK<M>` and uses `FoldM` — monoidal `Combine` over each matching error — which accumulates results rather than sequencing them. The operative distinction: `catchOf` sequences via `Bind` (short-circuits at first failure, correct for `IO`/`Eff`); `catchOfFold` combines via `MonoidK.Combine` (accumulates, correct for `Validation`). The type constraint enforces the right tool — `catchOf` does not compile for a type lacking `Monad<M>`, and `catchOfFold` does not compile for one lacking `MonoidK<M>`.
- `IO<A>.Fold(schedule, initialState, folder)` re-invokes the computation on success according to the schedule, accumulating a state `S` via `folder(S, A) -> S`. `FoldWhile(schedule, state, folder, stateIs: Func<S, bool>)` continues while the state predicate returns `true`; `FoldUntil` stops when it returns `true`. Both accept three termination predicate shapes — `Func<S, bool>` (state only), `Func<A, bool>` (value only), and `Func<(S State, A Value), bool>` (combined). `IO.Fold` without a schedule defaults to `Schedule.Forever`, producing an infinite accumulation loop that runs until an exception.
- `IO<A>.WithEnv(Func<EnvIO, EnvIO> f)` and `IO<A>.WithEnvFail(Func<EnvIO, EnvIO> f)` are the protected virtual methods underlying all isolation aspects. `Bracket()` calls `WithEnv(_ => env.LocalResources)` — `env.LocalResources` creates a new `Resources` child scope while propagating the existing `CancellationTokenSource` and `SyncContext`. `BracketFail()` calls `WithEnvFail(_ => env.LocalResources)` — the same local resources scope, but disposal fires only on failure. The resource scope is created identically; the disposal trigger is the sole difference.
- `IO<A>.Post()` marshals the computation onto the `SynchronizationContext` captured in `env.SyncContext` when the `EnvIO` was constructed. If `env.SyncContext` is `null`, `Post()` is a no-op that runs inline. The key constraint: `Post()` routes back to the context captured at the start of the `IO` chain — the one embedded in `EnvIO` at construction — not the current thread's context at the moment `Post()` executes. A `Post()` call mid-chain with a UI-thread-constructed `EnvIO` marshals back to the UI thread regardless of which thread runs the preceding chain steps.
- `IO<A>.Uninterruptible()` wraps in `IOUninterruptible`, which executes against `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — the existing `Resources` scope is preserved, `CancellationToken.None` ensures the new `CancellationTokenSource` is not linked to the outer token. The outer token's cancellation does not reach the wrapped computation; the error rail (`Fin`/`Error`) propagates normally. `Uninterruptible` severs only the `CancellationToken` path.
- `IO<A>.Fork(Option<TimeSpan> timeout)` returns `IO<ForkIO<A>>` where `ForkIO<A>` is `readonly record struct(IO<Unit> Cancel, IO<A> Await)`. Resources acquired inside the fork run against a child `Resources` scope — `new Resources(parentResources)` — and are released when the fork completes. Resources acquired in the parent before the fork are visible inside the fork as shared parent-owned resources and are not auto-released on fork completion. The fork's `CancellationTokenSource` is linked to `env.Token` via `env.Token.Register(tsrc.Cancel)`.

[SCHEDULE_ALGEBRA_FRONTIER]:
- `Schedule` is `Semigroup<Schedule>` with `+` as `Combine` (sequential append). The `|` operator is `Union` (min delay, length of the longer); `&` is `Intersect` (max delay, length of the shorter). These are three algebraically distinct binary operations on the same type. Conflating `|` with `+` changes semantics entirely: `exponential(50*ms) | recurs(5)` terminates after 5 steps, while `exponential(50*ms) + recurs(5)` runs the exponential forever then appends 5 zero-delay steps.
- `Schedule.Bind(Func<Duration, Schedule> f)` and `Schedule.SelectMany` make `Schedule` a monad in LINQ query expressions. `intersperse(schedule)` is implemented entirely as `Transform(s => s.Bind(schedule.Prepend))`: for each duration `d` in `s`, the bind returns `schedule` with `d` prepended, then flattens, producing `d1, (schedule steps), d2, (schedule steps), ...`. This is categorically different from `Interleave(b)`, which produces `a1, b1, a2, b2, ...` — `intersperse` inserts the entire interspersal schedule between each step, while `Interleave` alternates individual steps.
- `ScheduleTransformer` has an implicit conversion to `Schedule` via `Schedule.Forever | t`. A `ScheduleTransformer` from `recurs(5)` converts to a 5-step schedule; one from `maxDelay(1*second)` converts to an infinite schedule with per-step cap. The `+` operator on two `ScheduleTransformer` values is function composition (`g.Apply(f.Apply(x))`), not schedule concatenation — transformer `+` is left-to-right function composition, not the same as `Schedule.Combine`.
- `Schedule.upto(max, currentTimeFn)` terminates after wall-clock elapsed time exceeds `max` from the moment `Run()` is first called. The optional clock function (defaulting to `() => DateTime.Now`) is the injection point for deterministic testing. `maxCumulativeDelay(max)` terminates after the sum of emitted `Duration` values exceeds `max` — in computations where actions take variable time, the two diverge significantly.
- `Schedule.NoDelayOnFirst` is a static `ScheduleTransformer` implemented as `Transform(s => s.Tail.Prepend(Duration.Zero))`. It replaces the first step with `Duration.Zero` regardless of what the base schedule emits. `RetryUntil` and `RepeatUntil` unconditionally call `schedule.PrependZero.Run().GetIterator()` — the zero-delay first retry is implicit and not caller-controllable. Wrapping such a schedule in `NoDelayOnFirst` is idempotent: it produces a double-zero prefix, which does not cause extra retries because iteration count is driven by the schedule's step count, not delay count alone.

[DEFINITION_TIME_ASPECT_FLAGS]:
- `UnsafeConversionToKeyMemberType` on `[ValueObject<T>]` is an independent axis from `ConversionToKeyMemberType`. The documentation defines "unsafe" as a conversion that can throw — converting a value object back to its key type when the key type allows values the value object's validation would reject. Setting one axis to `None` does not suppress the other.
- `AllowDefaultStructs = true` suppresses the generated guard that throws when `default(T)` or the parameterless constructor is used. The flag has no effect on class value objects — classes use null as the default, which is already inadmissible.
- `DefaultInstancePropertyName` overrides the generated static property name for the `default` instance of a struct value object from `"Empty"` (analogous to `Guid.Empty`) to any valid identifier. The property exists only when `AllowDefaultStructs = true`.
- `SerializationFrameworks` on `[ValueObject<T>]` and `[SmartEnum<TKey>]` is a `[Flags]` enum (`SystemTextJson = 1, NewtonsoftJson = 2, MessagePack = 4`, `Json = 3`, `All = 7`). Setting it to `None = 0` suppresses all serialization aspects without suppressing the factory methods — unlike `SkipFactoryMethods = true`, which suppresses serialization as a side effect. The `[Flags]` nature enables selective suppression: `SerializationFrameworks.Json` generates STJ and Newtonsoft converters but not MessagePack formatters. Absent companion packages cause the generator to skip that framework aspect silently.
- `FactoryMethodGeneration.Always` on `[Union<T1,T2>]` forces factory method generation for every member regardless of whether any auto-detect trigger is present. The generated methods follow the `FromT1(T1)`, `FromT2(T2)` naming pattern. `FactoryMethodGeneration.Default` generates factory methods only when auto-detect triggers are present (duplicate types, interface arms, `object` arms, type-parameter arms); `FactoryMethodGeneration.None` suppresses factory methods unconditionally. `FactoryMethodGeneration` is on `UnionAttributeBase`, so it applies to both the typed ad-hoc form and the Regular Union.

[INTERCEPTOR_REVERSE_QUERY]:
- C# 14 adds `CSharpExtensions.GetInterceptorMethod(SemanticModel?, InvocationExpressionSyntax, CancellationToken)` returning `IMethodSymbol?`. Given an invocation that has been intercepted, it returns the interceptor method symbol. Before this API, verifying that a generated interceptor resolved correctly required manual source inspection. With `GetInterceptorMethod`, a generator can confirm its own interceptor wired correctly for a target call site within a single compilation pass, closing the generation-verification loop.
- `InterceptsLocationAttribute` is not in `System.Runtime`; the source generator must emit it as `namespace System.Runtime.CompilerServices { [AttributeUsage(...)] sealed class InterceptsLocationAttribute(int version, string data) : Attribute {} }`. The version integer must be taken from `GetInterceptableLocation`'s result at generation time — never hardcoded. A version mismatch between the attribute declaration and the compiler's expected encoding produces a compile error, not a silent failure.
