# Table Widening Defeat Law

[WIDENING_IS_AXIS_INDEPENDENT]:
- The table carries two literal axes the closure depends on — the KEY axis (`keyof typeof Table`, the closed key union) and the LEAF axis (each column's literal, the closed column union) — and a construction site widens them INDEPENDENTLY: a site can preserve the key union while widening every leaf, so a table that still types-checks against `keyof typeof` can have already lost every column's totality.
- The widening matrix by construction lever is fixed: `as const satisfies Record<K, Row>` preserves BOTH axes (key narrow, leaf narrow); a bare type annotation `const T: Record<K, Row> = {...}` widens BOTH (key `string`, leaf primitive); a bare `satisfies Record<K, Row>` with no `as const` preserves the key union but WIDENS every leaf to its primitive; `as const` alone preserves both but proves no row contract.
- The two axes degrade through different mechanisms, so the defeat is two distinct failures wearing one symptom: the key axis collapses when a `Record<string, Row>` upper bound or a key-erasing transform stamps `string` over the inferred union, while the leaf axis collapses whenever the literal-freeze (`as const`) is absent even if the key stayed narrow — `satisfies` re-widening the leaf is the trap a `keyof`-only audit never catches.
- The defeat is total over the closure the table seeds: a widened key axis defeats every downstream `keyof typeof` dispatch and `Match.exhaustive`, a widened leaf axis defeats every column-as-discriminant and column-as-instance read — the rank-order, group-equivalence, and function-column dispatch all silently degrade to `number`/`string` keyed instances that no longer prove the row set.

[SATISFIES_DOES_NOT_INFLUENCE_INFERENCE]:
- The load-bearing language rule is that `satisfies` is a check, NOT a contextual type: it verifies the already-inferred type assigns to the contract and then DISCARDS the contract, so the binding keeps exactly the type the literal inferred — `as const` makes that inferred type the deep-literal one, and without `as const` the inferred type is the freshly-widened one the literal would have had standalone.
- This is why `as const satisfies R` and `satisfies R` diverge on the leaf axis but agree on the key axis: object-literal KEYS are never widened by inference (a key is always its own literal), so the inferred type's `keyof` is the narrow union regardless of `as const`; object-literal VALUES widen to their primitive under fresh inference unless `as const` freezes them — the contract influences neither, it only gates.
- The operator-order corollary is exact: `as const satisfies R` freezes first then checks the frozen type against `R`, so a row missing a column fails because the frozen literal lacks it; `satisfies R as const` is rejected because `satisfies` produces an expression whose type is the binding's, and `as const` then re-applies to that already-checked value — the freeze must precede the check or the check runs over the widened leaf and a too-permissive contract passes a row the frozen form would reject.

```typescript
const TABLE = {
    alpha: { rank: 0, halts: false, glyph: '<glyph-a>' },
    beta: { rank: 1, halts: true, glyph: '<glyph-b>' },
} as const satisfies Record<string, { readonly rank: number; readonly halts: boolean; readonly glyph: string }>
// keyof typeof TABLE = 'alpha' | 'beta'; TABLE['alpha']['rank'] = 0; both axes narrow, contract checks row totality and discards

const LEAF_WIDE = {
    alpha: { rank: 0, halts: false, glyph: '<glyph-a>' },
} satisfies Record<string, { readonly rank: number; readonly halts: boolean; readonly glyph: string }>
// REJECT: satisfies without as const — keyof typeof LEAF_WIDE stays 'alpha' but LEAF_WIDE['alpha']['rank'] WIDENED to number; column-as-discriminant defeated, keyof audit passes
```

[CONTEXTUAL_TYPE_WIDENS_AT_THE_USE_SITE]:
- The defeat does not require a wrong annotation on the binding — the table widens whenever its literal flows into a CONTEXTUAL position typed `Record<K, Row>` without a literal-freeze: passing the object literal directly to a function parameter, returning it from a function whose return type is the wide record, or assigning it into a wide-record property all stamp the contextual type over the literal and erase both axes, even when the binding never named the type.
- The contextual-position defeat is invisible at the table's declaration — the construction looks clean — and surfaces only where the widened value is read; a table authored `as const satisfies` and then handed to `f(table)` where `f` takes `Record<string, Row>` is re-widened at the argument boundary unless `f` declares a literal-preserving parameter, so the freeze at the value does not survive a wide contextual sink.
- The generic-constraint partial defeat is the subtle middle case: a parameter `<R extends Record<string, { rank: number }>>(r: R)` infers `R` from the argument and so PRESERVES the key union, but the leaf still widens to `number` because the constraint contextually types each leaf — the key survives the generic, the column does not, reproducing the `satisfies`-without-`as-const` split at the call boundary.
- The fix is the `const` type parameter on the consuming surface, which is the as-const-equivalent at the call site: `<const R extends Record<string, { rank: number }>>(r: R)` infers `R` with deep literal preservation, freezing every leaf to its literal exactly as `as const` would at the value — so a vocabulary consumed by many sites declares its entrypoint with `const` type parameters once rather than forcing every caller to re-freeze with `as const`.

```typescript
declare function widens<Row extends Record<string, { rank: number }>>(rows: Row): Row
declare function preserves<const Row extends Record<string, { rank: number }>>(rows: Row): Row
const wide = widens({ alpha: { rank: 0 }, beta: { rank: 1 } })        // keyof wide = 'alpha'|'beta', but wide['alpha']['rank'] widened to number
const kept = preserves({ alpha: { rank: 0 }, beta: { rank: 1 } })     // const type param freezes the leaf to 0; both axes survive without an as-const at the call
```

[PER_LEAF_WIDENING_IS_LOCAL]:
- A single column annotated to its primitive at the literal — `rank: 0 as number` — widens ONLY that leaf and leaves every sibling column at its literal under the same `as const`, so the leaf axis is not all-or-nothing: a table can carry a closed `glyph` union and a widened `rank` column simultaneously, and only the column-as-discriminant reads against `rank` lose totality while `glyph`-keyed dispatch stays closed.
- This local widening is the deliberate escape, not the accident: when one column is genuinely open (a tunable threshold, a count that is not a discriminant) it is annotated wide at the leaf while the discriminant columns stay frozen, so the same owner carries a closed-vocabulary column and an open-data column without splitting into two tables — the freeze is per-leaf, applied exactly where a closed union is load-bearing.
- The accidental form is the same syntax used unintentionally: a column whose value arrives from a widening expression (a non-`const` variable, an arithmetic result, a function return typed wide) silently widens that leaf under `as const` because `as const` freezes the literal SHAPE it is given and cannot re-narrow a value that already entered widened — the freeze preserves, it does not recover, so a leaf fed a widened source is a widened column the `as const` cannot rescue.

[KEY_ERASURE_IS_THE_FINITE_STRING_RULE]:
- The key axis collapses to `string` precisely when the key-producing transform yields a non-finite-string key type, governed by `NonLiteralKey<K> = K extends string ? IsFiniteString<K> extends true ? string : K : symbol` — a finite literal key like `'alpha'` is finite and so widens to `string`, while the rule's role is that any reconstruction routes the key through this collapse: `Record.fromEntries`, `Record.filterMap`, `Record.filter`, `Record.getSomes`, and `Record.partition` all return `Record<NonLiteralKey<K>, B>`, erasing the union the table existed to carry.
- `IsFiniteString<T>` walks the string character-by-character and returns `false` the moment a `Head` segment is `string` or `` `${number}` ``, so a template-literal key `` `v${number}` `` is ALREADY non-finite before any transform — a vocabulary keyed on an open template pattern has `keyof typeof` of `string` at construction, never a closed union, which is why a closed composite vocabulary requires every template factor to be a finite literal union and one primitive factor opens the whole key set to `string`.
- The transforms split on whether they touch the KEY position: value-position transforms (`Record.map`, the `NoInfer<K>` key argument forbidding the callback from widening `K`) return `Record<K, B>` preserving the exact union, while key-position transforms route through `NonLiteralKey` — so the rule is to drop columns inside the value position with `Record.map` and never through `Record.filterMap`, because filtering by entry makes the surviving key set statically unknown and forces the `NonLiteralKey` widening even when the runtime keys are a strict subset of the closed union.
- `Record.mapKeys`/`Record.mapEntries` widen exactly when their key-producing function widens: their return is `Record<K2, B>` keyed by the function's RETURNED type, so a function returning a literal preserves `K2` while `key.toUpperCase()` returns `string` and collapses the axis — the combinator is innocent, the key function decides, and the only key-rekeying transforms that preserve a closed union are those whose function returns a literal-typed key.

[ROUND_TRIP_SILENTLY_ERASES_THE_PROOF]:
- The most dangerous defeat is structural, not nominal: `Record.fromEntries(Record.toEntries(Table))` is an identity at runtime — the same key-value pairs — but its type is `Record<string, Row>` because `fromEntries` cannot prove the reconstructed key set is the original union, so a round-trip that looks like a no-op silently erases the totality proof and every downstream `Match.exhaustive` keyed on the result compiles with no exhaustiveness obligation.
- The asymmetry is `keys` versus `fromEntries`: `Record.keys(Table)` returns `Array<K & string>` preserving the closed union for runtime enumeration, and `Record.union(A, B)` returns `Record<K0 | K1, ...>` preserving the merged union, but `fromEntries` and `toEntries`-then-rebuild lose it — enumeration and merge preserve, reconstruction-from-entries widens, so the table is iterated and combined freely but never round-tripped through entries when its key union must survive.
- The reverse index is the one site that invokes this erasure ON PURPOSE: `Record.fromEntries(toEntries(Table).map(([k, row]) => [row.id, k]))` deliberately widens the new key axis to `string` because the new keys ARE the id column, not the vocabulary keys, while the VALUES retain the closed key union `K` — so the round-trip widening is correct exactly when the keys being rebuilt are foreign and the closed union has migrated to the value position.

[KEYOF_TYPEOF_IS_NOT_THE_TOTALITY_PROOF]:
- A passing `keyof typeof` audit proves the KEY axis survived but says nothing about the leaf axis, so a table re-widened on the leaf still satisfies every key-position check while having lost every column-as-discriminant — the totality proof is the conjunction of both axes, and verifying only that `keyof typeof Table` is a closed union is the incomplete audit a `satisfies`-without-`as-const` defeat passes.
- The column union is the leaf-axis proof and it is read `(typeof Table)[keyof typeof Table]['rank']`: this projection is the closed union of a column ACROSS rows, and it degrades to `number` exactly when the leaf widened, so the audit that catches the leaf defeat is whether this column projection is a literal union or a primitive — a primitive there means the column is no longer a discriminant even when the key set is intact.
- `Match.value(input)` is `<const I>(i: I)` and so freezes the input literal at the dispatch entry, whereas `Match.type<I>()` takes `I` as an explicit annotation with no inference site — so a value routed through `Match.value(table[k])` carries the frozen leaf into the matcher while a `Match.type<Row>()` annotated wide re-widens it, making the dispatch entry itself a widening site governed by the same const-vs-annotation split as the table construction.
- The defeat composes with the closure the table seeds, so a single widening at construction or at a contextual sink propagates: the rank-derived `Order<K>`, the group-derived `Equivalence<K>`, the function-column dispatch arm union `(typeof Table)[keyof typeof Table]['run']`, and the `Match.exhaustive` over the key union all read the table's axes, and any one widened axis silently relaxes all of them from closed-union-keyed to primitive-keyed instances that no longer break loudly when a row is added.

[CODEC_VOCABULARY_HAS_NO_LEAF_AXIS_TO_DEFEAT]:
- The `Schema.Literal(...values)` vocabulary cannot suffer the leaf-widening defeat because its alphabet is the rest-tuple `Literals extends NonEmptyReadonlyArray<AST.LiteralValue>` inferred narrowly from the call arguments — there is no contextual record annotation to stamp over it and no `as const` to omit, so the codec form trades the structural table's two-axis fragility for a single inferred alphabet that widens only if a value argument is itself a widened expression.
- The contract-vs-inference asymmetry inverts at the codec: a structural `Record<K, Row>` annotation defeats the table by being an upper bound that overwrites inference, whereas `Schema.Literal`'s constraint `NonEmptyReadonlyArray<AST.LiteralValue>` is a lower-bound shape that admits only literals and infers the exact tuple — the constraint narrows admission rather than widening the result, which is why the codec owner is the form to reach for once a vocabulary must survive contextual sinks the structural table cannot.
- The codec uses the same `const` lever the structural table needs at its call sites: `Schema.transformLiterals` is `<const A extends ...>(...pairs: A)`, freezing every pair literal at the call exactly as `as const` freezes a table's leaves, so a remap vocabulary preserves both its encoded-column and decoded-column unions without an `as const` the structural spread would require — the const-on-the-type-parameter is the codec's built-in defeat-immunity, the lever the loose structural table must apply by hand at every construction and consumption site.
