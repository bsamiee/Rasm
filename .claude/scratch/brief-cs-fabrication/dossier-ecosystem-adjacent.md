## [2]-[THE SILENCES] — concerns the target SHOULD own, none named yet

**S1 — 3-axis+ SURFACE FINISHING (waterline/Z-level, drop-cutter, pencil-trace, constant-scallop, rest machining).** The single largest CAM silence. `Toolpath/motion.md` owns 8 `CutStrategy` rows (`boundary-pass`/`pocket-clear`/`peck`/`adaptive`/`radial-sweep`/`plunge-dwell`/`helical`/`layer-walk`) — ALL 2.5D contour/pocket/turning/additive over the Geometry2D offset. NONE samples a tool against a 3D triangle model. A production CAM system finishing a 3D surface needs drop-cutter (point-sampled Z against a tool at (x,y)) and waterline/Z-level (constant-Z contour of the surface). No managed C# library owns this; **OpenCAMLib (LGPL, native libocl)** is the categorical-best OSS owner, dynamic-link-clean, native-binding posture identical to admitted PicoGK/OcctNet. This is a new `CutStrategy` family (`waterline`/`drop-cutter`/`pencil`/`scallop`) whose generator hands off to OCL over the `MeshSpace`, plus **rest machining** (residual-stock tracking between tools) as a companion. 5-axis tool-axis control (lead/lean, collision-tilt) is the adjacent multi-axis silence the current "one orientation column" growth hand-waves.

**S2 — NC VERIFICATION / material-removal simulation.** `Toolpath/guard.md` is DESIGN-TIME swept-collision/gouge (a single move's Minkowski envelope vs. keep-outs). There is NO post-toolpath material-removal verification: swing the whole program against a STOCK MODEL, diff the removed volume, detect gouge/uncut/overcut/air-cutting. This is the CAMotics concern (GPL, reference only). The PERMISSIVE answer is already on disk and DEAD: **PicoGK** owns `Voxels.BoolSubtract` — a stock voxel field minus the accumulated swept-tool volume IS the verified result; MillSimSharp (MIT, .NET-native voxel+SDF, 3/5-axis) corroborates this is the right approach. Verification has no folder and no owner; it wires the dead PicoGK admission.

**S3 — ADDITIVE at PRODUCTION depth.** `Toolpath/slicing.md` is planar FFF only (4 infill patterns over Clipper2). The additive plane is thin against the mandate's "FFF/DED plus the resin/powder lanes, slicing, supports, orientation, lattice/infill sophistication, machine profiles": (a) NO support-generation owner (slicing.md L19 names it as growth, never owns); (b) NO orientation-optimization owner (Tweaker-3 algorithm, unclaimed); (c) NO resin/powder owner — PicoGK's grayscale/`.cli` + TPMS/gyroid lattice + support scaffolds are ADMITTED but UNWIRED; (d) NO modern format — STL-implied, no 3MF (lib3mf, BSD, official C# binding, beam-lattice extension that maps PicoGK `Lattice` output directly); (e) NO machine-profile model (build volume, nozzle, filament, printer kinematics). FFF slicer engines are all AGPL (REJECT) — the author-kernel posture is CORRECT — but the surrounding additive-production plane (format/support/orient/lattice/resin) is absent.

**S4 — PRODUCTION SPECS / GD&T / TOLERANCES / SURFACE-FINISH / SPC / measurement.** Entirely SILENT. No owner for geometric dimensioning & tolerancing, datum/FCF, tolerance stack, surface-finish (Ra/Rz) specification-or-prediction, process capability (Cp/Cpk/SPC), setup sheets, shop travelers, job routing. The mandate names all of these. No permissive .NET package exists (proprietary MTK/ODA/STEP-Tools REJECT), so it is a data-model author-kernel — grounded on the FREE **QIF (ISO 23952:2020)** XML schemas (models ASME GD&T + ISO GPS, datums, FCF, surface finish, SPC, measurement plans/results). QIF is the metrology counterpart to the folder's MTConnect-aligned tooling and ISO-13399 geometry; the IDEAS-queued `Toolpath/probing` (measured-feature receipt) is the natural QIF consumer. STEP AP242 carries semantic PMI, readable via OcctNet IF PMI surfaced (it is not today).

**S5 — Fast robust MESH BOOLEAN.** `Posting/projection.md` defers watertight-silhouette CSG to the kernel arrangement seam (correct per geometry-flow law). **Manifold (Apache-2, C FFI, v3.5.2, very active)** is the categorical-best robust mesh-boolean engine (guaranteed-manifold output, first-of-its-kind). Belongs KERNEL-side (one mesh-boolean owner per runtime); this is a WATERFALL to `Rasm/Geometry` with Fabrication the named demanding consumer (watertight silhouette + additive part/support boolean). Note as upstream ripple, not a Fabrication roster row.

**S6 — G-code INGESTION / re-post / DNC round-trip.** `Posting/program.md` WRITES a `CutProgram` AST but there is NO reader — an existing NC program cannot be re-ingested for verification, re-posting to a different dialect, or DNC round-trip. **gsGCode (Boost/BSL, depends on already-admitted geometry3Sharp)** is the same-ecosystem G-code parse/manipulate owner.

**S7 — SHEET-METAL bending / unfold / flat-pattern (press-brake).** The folder cuts 2D profiles + nests + emits cut-programs, but has no bend/unfold owner. `DSTV.Net` even carries `KA` bend blocks (its .api) yet no page reads them. Press-brake fabrication needs K-factor bend allowance + unfold-to-flat-pattern; **FreeCAD_SheetMetal (LGPL Python, reference)** is the algorithm (K-factor ANSI/DIN piecewise + tangent-face-graph unfold), authorable over **OcctNet** B-rep faces (already admitted). A real fabrication-process silence.

**S8 — MACHINE CONNECTIVITY / shop-floor streaming.** `Process/magazine.md` correctly firewalls transport (admits `MTConnect.NET-Common` model-only). The mandate names "machine-connectivity landscape." The full **MTConnect.NET (MIT, agent/adapter/SHDR/HTTP/MQTT, v6.9.0)** and OPC-UA (GPL/RCL gated) are APP-PLATFORM/transport concerns → WATERFALL to `Rasm.AppHost`, NOT the host-neutral Fabrication interior. The seam: machine tool-life telemetry streamed back feeds the magazine mid-job reload logic.

**S9 — POST-PROCESSOR dialect breadth / canned cycles / macros / subprograms.** `Posting/program.md` `GCommand` is 21 rows; `PostDialect` (Process/family) is 8 rows skewed hobby/open (linuxcnc/grbl/marlin/reprap) + 4 industrial (fanuc/haas/hypertherm/mazak). ABSENT vs. production: **canned cycles** first-class (G81-G89 drilling/boring/tapping — only `peck` exists), **macro programming** (Fanuc Macro B: `#`-variables, WHILE, GOTO), **subprograms** (M98/M99), **cutter compensation** (G41/G42, distinct from the geometry-side kerf), **work offsets** (G54-G59), and the **conversational-dialect family** — Heidenhain Klartext and Siemens SINUMERIK 840D are NON-RS-274 languages (the `PostDialect.Rs274` bool already anticipates a second family for Marlin; Klartext is a bigger family gap). Author-kernel deepening on the AST (a `CannedCycle`/`Macro`/`Subprogram` vocabulary) grounded on the free NIST RS274NGC v3 spec.

**S10 — FEEDS & SPEEDS data-model depth (cutting-force).** CONFIRMED GAP (README correctly records it): no free feeds/speeds DATASET exists — FreeCAD REMOVED its FeedsAndSpeeds cards for copyright (PR#15910). But the MODEL is richer than owned: `Process/physics.md` `CuttingData` is SFM+chipload only; the FreeCAD machinability model adds **Kienzle unit-cutting-force kc**, chip-thickness exponent, per-operation surface-speed (slotting/roughing/finishing), feed-per-tooth tables, cooling. The cutting-force term is load-bearing downstream: `Posting/program.md:183` hardcodes `double radialForce = 0.0012 * policy.RemovalRate` — a magic constant a real unit-cutting-force model would replace.
