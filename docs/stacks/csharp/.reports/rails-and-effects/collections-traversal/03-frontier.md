# Collections and Traversal — Frontier

[NATURAL_TRANSFORMATIONS_AND_COLLECTION_CONVERSION]:
- `Natural<F, G>` is a static interface with one abstract method `Transform<A>(K<F, A>) → K<G, A>`; `CoNatural<F, G>` is its dual with `CoTransform<A>(K<G, A>) → K<F, A>`. `NaturalMono<F, G>` composes both directions: it supplies a canonical isomorphism between two collection functor kinds via `Natural<F,G>` + `CoNatural<G,F>`. The static entry point is `Natural.transform<F, G, A>(K<F, A>)` / `CoNatural.transform<F, G, A>(K<G, A>)`.
- `Iterable` implements `NaturalMono<Iterable, Arr>`, `NaturalMono<Iterable, Seq>`, `NaturalMono<Iterable, Lst>`, `NaturalMono<Iterable, Set>`, and `NaturalMono<Iterable, HashSet>` — a full bidirectional isomorphism to every concrete collection kind. `Iterable` is therefore the universal collection interchange type at the trait layer; any collection that implements `CoNatural<F, Iterable>` can round-trip through it without writing explicit conversion code.
- `Arr` implements `Natural<Arr, Seq>`, `Natural<Arr, Iterable>`, `Natural<Arr, Lst>`, `Natural<Arr, Set>`, `Natural<Arr, HashSet>` — outbound only. `Seq` implements no `Natural` at all on its trait class. For generic code parameterized on a collection kind `F` that must convert to `Seq`, the constraint `where F : Natural<F, Seq>` is satisfiable by `Arr`, `Iterable`, `IterableNE`, and `Iterator` but not by `Seq` itself — `Seq`-typed inputs must be handled directly.
- `Iterator<A>` (the abstract linked-list class, distinct from `Iterable`) implements `Natural<Iterator, Arr>`, `Natural<Iterator, Seq>`, `Natural<Iterator, Lst>`, `Natural<Iterator, Set>`, `Natural<Iterator, Iterable>`, `Natural<Iterator, HashSet>` — outbound only, identical matrix as `Arr`. `Iterator` has no `CoNatural` — there is no structural transformation back into an `Iterator` from another kind via the trait.
- `Memo.transform<F, G, A>(Memo<F, A>)` wraps the natural transform inside a deferred `Memo<G, A>`: the underlying `K<F, A>` is not materialized until `.Value` is first accessed. `Memo.cotransform<F, G, A>(Memo<G, A>)` is the dual. This means `Applicative.Apply(K<F, Func<A,B>>, Memo<F, A>)` passes the right-hand collection lazily — the inner collection is accessed only if the left-hand function set is non-empty. `Arr.Apply` with `Memo<Arr, A>` uses this directly: it reads `ma.Value.As()` only after confirming `arr.Count > 0`, avoiding construction of the right-hand `Arr` entirely for the empty-function case.

[ITERATOR_LINKED_LIST_LIFECYCLE]:
- `Iterator<A>` is an abstract class with two structural cases: `Cons` (non-empty, abstract `Head` / `Tail` / `Clone` / `Split`) and `Nil` (terminal, `Tail` returns `this`). The backing implementations are `ConsFirst` (lazily realized from `IEnumerable<A>` via spin-lock CAS), `ConsValue` (eagerly allocated head + tail), `ConsEnum` (backed by a live `IEnumerator<A>`), and `ConsLazy` (thunked tail function). Each is `IDisposable` and releases the source enumerator or backing value explicitly.
- `Clone()` on `ConsFirst` and `ConsEnum` creates a new iterator starting from the same logical position without sharing the head reference. The XML remarks state this "will stop any GC pressure when processing large or infinite sequences." The mechanism: after `Clone()` the original `ConsFirst` node's reference is no longer retained by the recursive traversal stack, allowing the linked chain behind the current position to be GC'd. Without `Clone()`, a long traversal builds a reachable linked list from the beginning of the sequence to the current position.
- `Split()` is stronger than `Clone()`: it breaks the backward chain so that any `Iterator<A>` reference that led to the current position terminates there. The remarks state "Any other iterator references that came before this one will terminate at this point." Use `Split()` when consuming a node and no longer needing any backward references — inside `Map`, `Bind`, `AsEnumerable`, these call `Clone()` not `Split()`, so the chain break is the caller's responsibility for infinite generators.
- `Iterator<A>` is `IDisposable` with abstract `Dispose()`. `ConsFirst` CAS-disposes the `IEnumerator` and resets `firstAcquired` to 0. `ConsEnum` disposes the enumerator. Not disposing an `Iterator<A>` backed by a non-rewindable `IEnumerator` (e.g., database cursor, file reader) leaks the handle for the lifetime of the GC cycle — there is no finalizer safety net on `Iterator<A>`, unlike `SeqLoan<A>`.

[MONAD_RECUR_COLLECTION_SEMANTICS]:
- `Monad.Recur<A, B>(A, Func<A, K<M, Next<A, B>>>)` is the trampolining recursion primitive. `Next<A, B>` is a discriminated struct with `IsLoop` (recurse with value `A`) and `IsDone` (return value `B`). Implicit conversions exist from `Loop<A>` → `Next<A, B>` and `Pure<B>` → `Next<A, B>`, so arms are written as `new Loop<A>(nextState)` and `new Pure<B>(result)`.
- For `Seq`, `Arr`, `Lst`, `Set`, and `HashSet`, `Recur` is breadth-first: each generation step collects all loop-continuing values from the entire previous generation into a `List<A>`, then fans out to the next generation. `Done` values are yielded immediately. This means `Recur` over a collection monad enumerates a tree breadth-first, not depth-first — the distinction is observable when the function returns multiple `Loop` and `Done` values from one step.
- For `Iterable`, `Recur` delegates to `Monad.iterableRecur` which runs as an `async IAsyncEnumerable<B>` generator. When `M : Natural<M, Iterable>, CoNatural<M, Iterable>`, the recursion is materialized by transforming the result through the `Iterable` natural transformation and iterating it. The `EnvIO` cancellation token from the current `Eff` runtime is threaded through the async state machine.
- `Eff.Recur` and `IO.Recur` implement trampolining via an explicit `while` loop: each step awaits `f(value)`, reads the `Next<A,B>` out of `Fin<Next<A,B>>`, and continues or terminates. This is O(1) stack regardless of recursion depth. `Arr` and non-async collection Recur use `Monad.enumerableRecur` — an imperative `List<A>` swap-buffer loop that is also O(1) stack depth per generation.

[COMPOSITIONS_MONOID_TREE]:
- `Compositions<A>` is a skewed binary tree over elements that implement `Monoid<A>`. It supports `Skip(k)` in O(log k), `Take(k)` in O(k log k), `TakeComposed(k)` in O(log k), and `composed()` (fold entire tree) in O(log n) — all via tree-descent rather than enumeration. `SplitAt(i)` is a tuple alias for `(Take(i), Skip(i))` and pays the combined O(k log k) + O(log k) cost.
- The design pressure for `Compositions<A>`: when a pipeline must query prefix-sums or interval compositions of a sequence many times, building a `Compositions<A>` once and calling `TakeComposed` per query costs O(n) build + O(log k) per query, versus O(n) per query on a flat fold. The canonical domain is running-sum windows over a `Monoid<A>` without materializing intermediate aggregates.
- `Compositions.wellFormed<EqA, A>(Compositions<A>)` validates the right-associative balance invariant. The tree must have strictly increasing node sizes from right to left at each level. Constructing via `Combine` and `Singleton` always produces a well-formed tree; constructing by direct `Tree` manipulation can violate the invariant and silently corrupt `Skip`/`Take` results.

[FOLDABLE_TRAIT_NUMERIC_AGGREGATION]:
- `Foldable<T>.Sum<A>` requires `A : IAdditionOperators<A, A, A>, IAdditiveIdentity<A, A>` — the generic math interfaces from .NET 7+. The default implementation is a left fold via `Fold((A x) => (A y) => x + y, A.AdditiveIdentity, ta)`. `Product<A>` requires `IMultiplyOperators<A, A, A>, IMultiplicativeIdentity<A, A>`. `Average<A>` requires `A : INumber<A>` and computes count and total in a single fold pass via a `(Count, Total)` state tuple.
- `Average<A, B>(Func<A, B>, K<T, A>)` maps each element before accumulating, requiring `B : INumber<B>`. This is the correct form when elements are not directly summable — e.g., extracting a `double` metric from a domain value — without a prior `Map` that allocates an intermediate collection.
- `Foldable<T>.At<A>(K<T, A>, Index)` returns `Option<A>` and handles both forward (`!index.IsFromEnd`) and from-end (`Index` with `IsFromEnd = true`) positions. The default implementation uses `FoldWhile` for forward access and `FoldBackWhile` for from-end access. For `Seq`, this is overridden to use direct `seq[index]` when in-range, which is O(1) for `SeqStrict<A>`. For `Set` and `Map`, `At` uses `FoldWhile` — O(k) traversal — because AVL trees have no O(log n) positional index.
- `Foldable<T>.Min<OrdA, A>` and `Max<OrdA, A>` accept an explicit `OrdA : Ord<A>` constraint, enabling non-default ordering. `Min<A>` and `Max<A>` use `IComparable<A>`. `Set<A>` and `Map<K, V>` override both to use `FindMin` / `FindMax` on the AVL tree — O(log n) — rather than the default fold.

[SEMIGROUPK_VS_CHOICE_SEMANTICS]:
- `SemigroupK<T>.Combine<A>(K<T, A>, K<T, A>)` concatenates two structures: for `Seq`, `Arr`, `Lst` it appends elements; for `HashMap` it delegates to `operator+` (left-biased on key collision); for `HashSet` it unions. `Choice<T>.Choose<A>(K<T, A>, K<T, A>)` selects the first non-empty structure. These are semantically disjoint: `Combine` always uses both sides; `Choose` short-circuits on the first non-empty. The two operators express different fallback idioms over the same collection type.
- `Choice<T>.Choose<A>(K<T, A>, Memo<T, A>)` is the lazy overload: the right-hand operand is wrapped in `Memo<T, A>` and evaluated only if the left is empty. `Iterable<A>` exposes this as `operator|(Iterable<A> x, Memo<Iterable, A> y)`. The canonical pattern for "use this collection unless it is empty, otherwise compute an alternative" wraps the alternative in `memoK<T, A>(() => computeAlternative())` and uses `|` or `Choose`.

[LST_AVL_POSITIONAL_INSERTION]:
- `Lst<A>` is backed by `LstInternal<A>` which uses a `ListItem<T>` AVL tree with `Height`, `Count`, `Left`, `Right`, and `Key` fields. This is a completely independent AVL implementation from `Map<K,V>` — it is index-addressed, not key-addressed. `ListModule.GetItem(root, index)` walks the tree by comparing `index` against `Left.Count`, giving O(log n) positional access. `ListModule.Insert(root, index, value)` similarly walks by index to find the insertion point and rebalances. This confirms that `Lst<A>` is genuinely the only collection with O(log n) positional insertion; `Seq<A>` cannot insert mid-sequence without O(n) work.
- `Lst<A>` does not implement `Natural<Lst, Seq>` or any outbound natural transformation on its trait class. Converting a `Lst<A>` to `Seq<A>` at the HKT layer goes through `Foldable<Lst>.ToSeq<A>` which folds the entire tree into a new `Seq<A>` — O(n). `Iterable` implements `CoNatural<Lst, Iterable>`, so `Lst<A>` can be admitted to `Iterable`-consuming trait combinators via the `CoTransform` path on the `Iterable` trait class.

[MEMOIZED_APPLICATIVE_INNER_COLLECTION]:
- `memoK<F, A>(Func<K<F, A>> f)` constructs a `Memo<F, A>` that defers construction of `K<F, A>` until `.Value` is first accessed, then caches the result. `memoK<F, A>(K<F, A> fa)` preloads. The cache is guarded by a `SpinWait` CAS on an `int state` field (0=unloaded, 1=loading, 2=loaded), making the memo thread-safe at the construction boundary but not at the `K<F, A>` usage boundary if the underlying effect is stateful.
- The idiom for avoiding repeated inner-collection traversal in multi-function applicative application is to wrap the inner collection in `memoK`: `Apply(functions, memoK<Seq, A>(innerSeq))`. `Seq.Apply(K<Seq, Func<A,B>>, Memo<Seq, A>)` reads `ma.Value.As()` once and iterates it per function. Without `Memo`, each `Apply` call on a `SeqLazy<A>` would trigger re-enumeration from the source on each function.
- `Memo<F, A>.Reset()` clears the cached `K<F, A>` and reverts to the `acquire` function, but only when `acquire` was provided at construction. Memos built via `memoK<F, A>(K<F, A> fa)` (pre-loaded) return `Unit.Default` immediately from `Reset()` — there is no re-acquisition path. `Clone()` always creates a fresh unloaded memo from the original `acquire` function if present, or a pre-loaded memo with the same value if not.
