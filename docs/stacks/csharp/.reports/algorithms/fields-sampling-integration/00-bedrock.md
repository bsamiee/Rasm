# Fields, Sampling, and Integration — Core Research

[INTEGRATOR_CATALOG]:
- The `Integrate` static class exposes four orthogonal routes: `OnClosedInterval` (double-exponential, absolute-error target), `GaussLegendre` (fixed-order, with infinite-bound substitution), `GaussKronrod` (adaptive recursive with embedded error estimate and L1-norm channel), and `OnRectangle`/`OnCuboid` for tensor-product 2D/3D cubature. These are structurally distinct algorithms — not wrappers of one underlying kernel — and the route choice is the primary accuracy-versus-cost decision, not the order parameter.
- `Integrate.GaussKronrod` returns `(value, error, L1Norm)` through `out` parameters on its four-parameter overload. The `L1Norm` is the integral of `|f|` accumulated in the same pass; a large `|result / L1Norm|` ratio diagnoses ill-conditioning from cancellation without a second evaluation. The two-parameter overload discards both `error` and `L1Norm`, making the cancellation diagnostic invisible. The `Integrate` facade `targetRelativeError` defaults to `1e-8`; `GaussKronrodRule.Integrate` directly also defaults to `1e-10` — these are independent method-parameter defaults.
- `GaussKronrod`, `GaussLegendre`, and `DoubleExponential` accept `double.NegativeInfinity`/`double.PositiveInfinity` at the facade level, and `GaussKronrodRule.Integrate` directly also owns its own infinity substitution — the facade forwards to the same static and adds no additional substitution logic. Calling `DoubleExponentialTransformation.Integrate` directly with infinite bounds passes infinity into abscissa evaluation, producing `NaN` weights; the substitution fires only through `Integrate.DoubleExponential`. Symmetrically, `Integrate.GaussLegendre` at the facade applies a cubic change-of-variable `(b-a)/4 * t*(3-t²) + (a+b)/2` on top of the already-mapped Gauss-Legendre nodes — composing two substitutions. Callers expecting raw Gauss-Legendre nodes on `[a,b]` must use the static `GaussLegendreRule.Integrate(f, begin, end, order)` directly, which applies only the linear affine map; the instance `GaussLegendreRule` constructor exposes nodes and weights via `GetAbscissa`/`GetWeight` but has no instance `Integrate` method — weighted sums must be hand-rolled from those accessors.
- The adaptive recursion inside `GaussKronrodRule.recursive_adaptive_integrate` is bounded by `maximumDepth` (default 15), capping the recursion tree at `2^maximumDepth` leaves. At `maximumDepth = 15` and `order = 15`, full recursion reaches ~260,000 function evaluations. Depth exhaustion returns silently without a convergence flag; callers must inspect the `out error` parameter. Reducing to 8–10 trades accuracy for bounded evaluation count on oscillatory or near-singular integrands.
- `GaussKronrodRule.recursive_adaptive_integrate` initializes `absTol = 0.0` at the root, then sets it to `|value * relTol|` after the root interval evaluation. Recursive calls halve `absTol` at each level, so for integrands with a sharp spike covered by many recursion levels the accumulated error can exceed `relTol * |result|` because `absTol` splits geometrically rather than proportionally to local magnitude. The error estimate per node clamps to `max(|GK_N - G_{(N-1)/2}|, |GK_N * 2ε|)` to prevent recursion beyond numerical resolution, but this clamping does not recover the proportional bound.
- Precomputed Gauss-Legendre nodes exist for orders {2–20, 32, 64, 96, 100, 128, 256, 512, 1024}. Any other order triggers on-the-fly computation through `GaussLegendrePoint.Generate`: Newton's method initialized with Chebyshev-cosine guesses, with both node-value and weight-value convergence checks. The node cache (`GaussLegendrePointFactory._gaussLegendrePoint`) is `[ThreadStatic]` on a single slot per thread — a field integration loop that alternates between two non-precomputed orders recomputes on every call, and parallel integration with `Task.WhenAll` or `Parallel.For` where each task uses a different order produces cache misses on every call. The remedy is to construct named `GaussLegendreRule` instances at setup; each instance holds its own `GaussPoint` as a private field, bypassing the thread-local slot entirely.
- Precomputed Gauss-Kronrod nodes exist for orders {15, 21, 31, 41, 51, 61}; all others use a Stieltjes-polynomial generator. `GaussKronrodRule` takes only `order` in its constructor; the instance surface exposes `Order`, `KronrodAbscissas`, `KronrodWeights`, and `GaussWeights` only — bounds are passed to the static `GaussKronrodRule.Integrate`, not the instance.
- `GaussLegendreRule.Integrate` for 2D and 3D cubature performs a straightforward tensor-product quadrature: order-`N` integration costs exactly `N²` function evaluations in 2D and `N³` in 3D for all parities. There is no reflection-symmetry inner-loop halving. At order `N` and dimension `d`, the general count is `N^d` — no sparse-grid or Stroud-type rule is available, making 3D cubature impractical beyond moderate order.
- `DoubleExponentialTransformation.Integrate` uses `NewtonCotesTrapeziumRule.IntegrateAdaptiveTransformedOdd` with level-indexed abscissas and weights. The termination test is the absolute difference between successive level estimates; `NumberOfMaximumLevels = 10` caps refinement and is reached silently. `DoubleExponentialTransformation.ContourIntegrate` applies the same tanh-sinh kernel to `Func<double, Complex>` for complex-valued line integrals on finite intervals; `GaussLegendreRule.ContourIntegrate` and `GaussKronrodRule.ContourIntegrate` expose the same real-parameterized complex-valued integrand path — the `L1Norm` output in the Kronrod contour variant is `double`, accumulating `Complex.Abs` at each evaluation, keeping the cancellation diagnostic `|result|/L1Norm` valid for complex-valued fields.
- `NumericalDerivative.StepSize` setter rounds the input to the nearest power of 2, producing a binary-representable step and improving finite-difference cancellation error. `StepType.Relative` derives `BaseStepSize = ε^(1/2)` — the classical generic step for centered differences, constant across all `points` and `center` settings. Both `StepType.Relative` and `StepType.RelativeX` share this `BaseStepSize`; `RelativeX` multiplies the per-evaluation step by `(1 + |x|)`, adapting to the point magnitude. The distinction matters only when evaluating at large `|x|`: `RelativeX` produces a coarser absolute step that tracks field scale.
- `FiniteDifferenceCoefficients.GetCoefficients(center, order)` generates exact rational stencil weights: `0` is forward, `points-1` is backward, `points/2` is centered. `GetCoefficientsForAllOrders(center)` returns the full 2D rectangular array in one call. Construction solves a `p×p` Vandermonde-like system via `DenseMatrix.Inverse()` in `O(p³)`, rounding weights to `MidpointRounding.AwayFromZero`; subsequent queries are `O(1)` table lookups. At `p=15` the 3375-operation solve runs once at construction.
- `EvaluateMixedPartialDerivative` recurses depth-first, evaluating the outer partial last. For a mixed derivative of order `k` at `p` stencil points, total function evaluations are `p^k` — 125 at `k=3, p=5`. The cost grows exponentially and is not reduced by symmetry of mixed partials.
- `SimpsonRule.IntegrateComposite` requires `numberOfPartitions` to be even — passing an odd value throws `ArgumentException` with no automatic rounding. In an adaptive field integration loop that bisects sub-intervals, any bisection from an even count preserves evenness; an initial odd-count call is a hard error with no recovery.

[FIELD_EVALUATION_AND_INTERPOLATION]:
- The `IInterpolation` interface defines a unified contract: `Interpolate(t)`, `Differentiate(t)`, `Differentiate2(t)`, `Integrate(t)` (indefinite), and `Integrate(a, b)` (definite). The `SupportsDifferentiation` and `SupportsIntegration` flags are explicit-interface properties evaluated at runtime — there is no compile-time prevention of calling unsupported operations.
- `CubicSpline` stores four coefficient arrays (`c0, c1, c2, c3`). `ComputeIndefiniteIntegral` prefixes segment integrals via Horner evaluation: `integral[i+1] = integral[i] + h*(c0 + h*(c1/2 + h*(c2/3 + h*c3/4)))` where `h = x[i+1] - x[i]` — exact in floating-point for polynomials, `O(N)`. The result is cached in a `Lazy<double[]>`; first call pays `O(N)` setup; subsequent `Integrate(a, b)` calls cost one binary search each. `LinearSpline` and `StepInterpolation` share the same `Lazy<double[]>` + `Array.BinarySearch` pattern; the two capabilities (`Lazy<>` for integration, `Lazy<>` for differentiation) are independently lazy-initialized. Field integrations inside tight loops should pre-warm via a single `Integrate` call at construction.
- `CubicSpline.InterpolateHermiteSorted` constructs Hermite cubics from sample values and first derivatives directly: coefficients `c2, c3` are derived per interval by the local two-point Hermite system, with no global tridiagonal solve. This gives per-interval `O(1)` construction and exact derivative matching at sample points — the correct route when tangent estimates come from the field's own gradient evaluator.
- `CubicSpline.InterpolateMakimaSorted` computes weights `|Δm_{i+1} - Δm_i| + 0.5 * |Δm_{i+1} + Δm_i|`. The `0.5 * |sum|` term prevents zero weights when opposing slopes cancel — the failure mode of classic Akima at those nodes. When all four surrounding differences vanish (constant-slope region), the fallback is `0.5 * (m[n+3] + m[n])` — the arithmetic mean of extended-boundary slopes — so MAKIMA never produces a zero-weight artifact, at the cost of a boundary-slope dependency in interior nodes. `InterpolateAkimaSorted` does produce zero-weight artifacts at constant-slope regions; its degenerate fallback is a linearly interpolated tangent weighted by interval widths, not a zero tangent — callers expecting zero gradient at inflection points of smooth fields must use `InterpolatePchipSorted` instead.
- `CubicSpline.InterpolatePchipSorted` zeros the slope at monotone-sign-change intervals and applies the formula `(w1 + w2) / (w1/m[j-1] + w2/m[j])` for interior slopes with matching signs, where `w1 = 2*h1 + h0` and `w2 = 2*h0 + h1`. This is a weighted harmonic mean with numerator `3*(h0 + h1)`, guaranteeing C¹ continuity and monotonicity preservation within each monotone segment. `PchipEndPoints` clamps boundary slopes to `|endpoint_slope| ≤ 3 * |first_interval_slope|` to prevent extrapolation overshoot.
- `SplineBoundaryCondition` has exactly four members: `Natural` (second derivative = 0 at endpoints, minimizes overshoot), `ParabolicallyTerminated` (parabolic boundary extension), `FirstDerivative` (caller supplies endpoint slope, value 0.0 for clamped-flat), and `SecondDerivative` (caller supplies endpoint curvature, value 0.0 reproduces the natural condition). No periodic mode is exposed through this enum; periodic fields must be reconstructed by mirroring or extending the sample array manually.
- `BulirschStoerRationalInterpolation` supports neither differentiation nor integration through the `IInterpolation` contract and is the only interpolant that handles poles. The pole-avoidance perturbation initializes `D[i]` with `y[i] + 1e-25`; when the rational update denominator rounds to zero under ULP tolerance, the method returns `NaN` rather than throwing — callers must test for `NaN` on each evaluation.
- `Barycentric.InterpolateRationalFloaterHormannSorted(x, y, order)` computes alternating-sign barycentric weights over blending windows of width `order`; omitting `order` defaults to `x.Length - 1`, which is full-degree polynomial. This interpolant supports neither differentiation nor integration through the `IInterpolation` contract; it is the correct route for near-singular data but the contract flags must be checked before composing it with operations that require those capabilities.
- `TransformedInterpolation.InterpolateSorted(transform, transformInverse, x, y)` applies `transformInverse` to the `y` values via `CommonParallel.For` with 4096-element chunks, then constructs a `LinearSpline` over the transformed values. The `_interpolation` field is always `LinearSpline` — the factory hardcodes this and does not accept a `CubicSpline` in the base position. Both `SupportsDifferentiation` and `SupportsIntegration` are `false`. `LogLinear` wraps a `LinearSpline` in log-space, exposes `SupportsDifferentiation: true` through the explicit interface, and throws `NotSupportedException` on `Integrate` — it is the correct surface for log-linear interpolation with differentiation only. The `InterpolateSorted` factory also parallelizes the `log` computation via `CommonParallel.For` with 4096-element chunks; `TransformedInterpolation` exposes neither capability.
- `QuadraticSpline` is a full `IInterpolation` implementor with `SupportsDifferentiation: true` and `SupportsIntegration: true`. It requires direct construction — `new QuadraticSpline(x, c0, c1, c2)` with pre-computed coefficient arrays — because there is no `Interpolate.*` facade entry for it. Its `Differentiate2(t)` returns `2.0 * c2[segment]` — a piecewise-constant second derivative. For smooth field curvature estimation this constant curvature per segment is a design limitation relative to `CubicSpline`.
- The `Interpolate` static facade exposes these constructors: `Common`, `RationalWithoutPoles`, `RationalWithPoles`, `PolynomialEquidistant`, `Polynomial`, `Linear`, `LogLinear`, `CubicSpline`, `CubicSplineRobust`, `CubicSplineMonotone`, `CubicSplineWithDerivatives`, and `Step`. `CubicSpline.InterpolateAkimaSorted`, `InterpolateMakimaSorted`, and `InterpolatePchipSorted` are accessed directly on the `CubicSpline` class, not via the facade.
- `CubicSpline.InterpolateAkimaSorted` and all `*Sorted` variants require pre-sorted input and throw `ArgumentException` on unsorted data. The `*Inplace` variants sort in-place and mutate the input arrays. Field evaluation pipelines using `*Inplace` must clone data if the input is shared.
- `StepInterpolation` reports `SupportsDifferentiation: true` as an explicit interface implementation, but `Differentiate(t)` returns `double.NaN` exactly at sample points (where `Array.BinarySearch` returns a non-negative index) and `0.0` everywhere within a segment. The `NaN` is the distributional derivative signal at the jump discontinuity — callers using `StepInterpolation` in a gradient-accumulating quadrature loop must guard against `NaN` propagation at every sample point.

[KERNEL_DENSITY_AND_SCATTER_RECONSTRUCTION]:
- `KernelDensity.Estimate(x, bandwidth, samples, kernel)` parallelizes via `CommonParallel.Aggregate` with no degree-of-parallelism cap exposed to the caller — the inner thread usage is independent of any outer `ParallelOptions.MaxDegreeOfParallelism`. Parallel reduction does not preserve associativity order, so estimates are non-reproducible across thread schedules even with a fixed seed. When a field integration pipeline spawns multiple concurrent KDE evaluations, the two parallelism layers do not compose and can saturate the thread pool. Serial evaluation is required for reproducibility. There is no spatial index — the sum over `samples` is an all-pairs `O(N)` scan per query point; `O(N·M)` for `N` samples and `M` query points.
- Four named kernels are provided: `GaussianKernel` (delegates to `Normal.PDF(0, 1, x)`), `EpanechnikovKernel` (optimal MSE, compactly supported on [-1,1]), `UniformKernel`, `TriangularKernel`. Any `IContinuousDistribution` PDF can be adapted to a kernel with a one-line lambda; no subtype is needed.
- `Fit.LinearCombination(x[], y[], Func<double,double>[] functions)` and `Fit.LinearCombination(..., DirectRegressionMethod)` build the design matrix and dispatch to `MultipleRegression`. Passing `DirectRegressionMethod.Svd` is the rank-revealing route for ill-conditioned basis matrices (overlapping compactly supported kernels, near-dependent RBFs).
- `MultipleRegression.DirectMethod(Matrix<T> x, Matrix<T> y, DirectRegressionMethod)` accepts a matrix response — `y` is a matrix — enabling simultaneous reconstruction of vector-valued fields (gradient, flux) from the same scattered input without separate solve calls. Named `NormalEquations`, `QR`, and `Svd` static methods are also available as direct dispatch alternatives.
- `WeightedRegression.Weighted(designMatrix, responseVector, weightMatrix)` implements weighted least squares with a diagonal weight matrix encoding per-sample confidence. `WeightedRegression.Local(x, y, t, radius, kernel)` computes a locally weighted fit around query point `t` within `radius`, using an injected kernel — the library-level locally-weighted regression surface that eliminates the need to manually window and solve.

[SAMPLING_AND_RANDOM_INFRASTRUCTURE]:
- `RandomSource` is the abstract base; the only required override is `protected abstract double DoSample()`. All generators accept an explicit integer seed. `RandomSeed.Robust()` uses `RandomNumberGenerator.GetBytes(4)` behind a static lock — thread-safe but contended under heavy concurrent seeding. The thread-safe instance flag wraps all `DoSample` calls in `lock(_lock)`, documented as 2.5× slower than non-thread-safe instances.
- `Xoshiro256StarStar` is the highest-quality generator available: 256-bit state across four `UInt64` fields, marked `[DataContract]` with `[DataMember(Order=1..4)]`. A seeded instance can be serialized mid-sequence and deserialized to reproduce the exact continuation — the correct approach for blue-noise sampling or deterministic field sampling that must checkpoint and resume. `MersenneTwister` carries the same `[DataContract]` attribute; `SystemRandomSource` wraps `System.Random`, which does not support this.
- `SystemRandomSource.Default` is a `ThreadLocal<SystemRandomSource>` initialized with `RandomSeed.Robust()` per thread. `Generate.Uniform(length)` and `Generate.Normal(length, ...)` delegate to this thread-local — non-deterministic across runs. `SystemRandomSource.FastDoubles(values)` bypasses this for arrays of length ≥2048: it parallelizes via `CommonParallel.For` with block sizes of 1024, 2048, or 8192, constructing a fresh `System.Random` from `RandomSeed.Robust()` per block — non-deterministic regardless of any per-thread seeding. Deterministic sampling requires constructing distributions directly with an explicit seeded generator: `new Normal(mean, sd, new MersenneTwister(seed)).Samples(array)`.
- `HybridMC` accepts `frogLeapSteps`, `stepSize`, `burnInterval`, per-dimension momentum standard deviations (`pSdv`), an explicit `Random randomSource`, and a `DiffMethod diff` gradient delegate — all in an eight-parameter constructor. The default gradient is a centered three-point finite difference with step `max(0.001, 1e-6 * |x[i]|)` — a hardcoded absolute floor of `0.001` that dominates when `|x[i]| < 1000` and is unsuitable for fields with sharp gradients at small scales or large-scale parameter spaces. Supplying a user-provided `DiffMethod` via the eight-parameter constructor overload is mandatory for such fields; `UnivariateHybridMC` follows the same floor rule.
- `Tensor.FillGaussianNormalDistribution<T>(ref TensorSpan<T> destination, Random random)` is a static method on `Tensor` — not on `TensorPrimitives` — and fills a multi-dimensional `TensorSpan<T>` with Gaussian normals in-place, enabling field-grid initialization that avoids per-element boxing from `Generate.Normal`. The `TensorSpan<T>` is constructed over a flat `T[]` using the `TensorMarshal`/`Tensor` construction APIs with `nint[]` lengths and strides.
- `UnivariateSliceSampler` implements Neal stepping-out with no expansion budget: the bracket `[low, high]` expands by `Scale` in each direction until the log-density falls below the slice level, then shrinks by retraction on rejection. There is no cap on bracket expansion — a nearly-flat log-density causes unbounded bracket growth proportional to `Scale` per iteration. `Scale` is mutable post-construction, allowing adaptive scale tuning between `Sample()` calls without restarting the chain; a scale increase that widens the bracket can cause an immediate rejection cycle if the density has already tightened around the current location. `Burn(burnInterval)` runs synchronously in the constructor; `AcceptanceRate` queried before any `Sample()` call returns zero, not the burn-phase acceptance rate.
- `MCMCDiagnostics.ACF<T>` computes autocorrelation at a single lag by materializing the series to an array and calling `Correlation.Pearson` on shifted windows — `O(N)` per lag, `O(N * maxLag)` for a lag ladder. For a full autocorrelation spectrum, `Correlation.Auto(double[])` uses `Fourier.Forward`/`Fourier.Inverse` with `AsymmetricScaling` and zero-padding to `CeilingToPowerOfTwo * 2`, delivering `O(N log N)`. The two are not composable: `MCMCDiagnostics.ACF` accepts `IEnumerable<T>` with a projection delegate; `Correlation.Auto` requires `double[]` directly.
- `MCMCDiagnostics.EffectiveSize<T>` uses only lag-1 autocorrelation via `(1 - ρ₁)/(1 + ρ₁) * N` — the first-order approximation. The full Geyer monotone sequence estimator sums positive initial sequence terms across all positive lags; the lag-1 approximation overestimates effective sample size when the chain has long-range positive autocorrelation, a common case for HMC chains with large step sizes.
- `RejectionSampler<T>.Sample` throws `ArgumentException` at runtime when `pdfQ(val) < pdfP(val)` for any sampled point. The proposal envelope must dominate — this is a design-time contract violation that surfaces as a runtime exception with no diagnostic about which point violated the bound. A field-derived target density modified after construction must re-verify the envelope condition after every modification.
- `MetropolisHastingsSampler<T>.Burn` computes the log Hastings ratio and tests `ratio == 0.0` after computing `ratio = Math.Min(0.0, logHastings)`. Since `Math.Min(0.0, x) == 0.0` whenever `x >= 0.0`, the deterministic-accept branch fires for every uphill-or-equal proposal — it is the primary acceptance path for MCMC chains on log-concave targets with symmetric proposals, not dead code. `AcceptanceRate` is the ratio of accepted samples to total samples (`Accepts / Samples`); the base-class property is `McmcSampler<T>.AcceptanceRate`.
- `Combinatorics.GenerateCombination(n, k, randomSource)` uses two paths based on density: when `k * 3 < n` (sparse), it fills a `bool[]` mask with repeated sampling and retries on collision; when `k * 3 >= n` (dense), it generates a full permutation and takes the first `k`. The sparse path has expected cost `O(k * n/(n-k))` due to collision retries; near-half selections (`k ≈ n/2`) trigger the dense path at `O(n)`.
- `Combinatorics.SelectPermutationInplace<T>` implements Fisher-Yates (Durstenfeld) in-place shuffle. Passing a seeded `MersenneTwister` produces a deterministic, serializable permutation sequence for reproducible field sampling traversal order. The LINQ `SelectPermutation` variant copies and yields lazily, avoiding full materialization when only a prefix of the permutation is needed.
- `Combinatorics.GenerateVariation(BigInteger n, int k, randomSource)` samples `k` distinct indices from `[0, n)` using `RandomExtensions.NextFullRangeInt64` chained via `BigInteger` arithmetic. For field sampling from large discrete spaces (voxel grids beyond `int.MaxValue`), this is the only correct path; the `int`-overloaded `GenerateVariation(int n, k)` silently overflows when `n > int.MaxValue`.
- `SortedArrayStatistics.Quantile(data, tau)` hardcodes R-8 (approximately median-unbiased): `h = (N + 1/3) * tau + 1/3`, interpolating between adjacent order statistics. For sampling acceptance gates based on empirical quantile thresholds, R-8 is appropriate. `QuantileCustom(data, tau, QuantileDefinition)` covers R1–R9 and named aliases: `R7 = Excel`, `R8 = Median`, `R6 = Weibull = Nist = SPSS`, `R5 = Hazen`. The four-parameter form `QuantileCustom(data, tau, a, b, c, d)` follows the Mathematica convention directly: `h = a + (N + b)*tau`. Extension methods such as `IEnumerable<double>.QuantileCustomFunc(QuantileDefinition)` materialize the data to a sorted array once and return a `Func<double, double>`, amortizing the `O(N log N)` sort cost across all quantile probes.
- `RunningStatistics.Combine(a, b)` implements Chan's parallel online algorithm for combined mean, variance, skewness, and kurtosis, with the fourth-moment formula accumulating the delta and cross-terms exactly. `operator+` is sugar over `Combine`. The type is `[DataContract]` serializable with seven `[DataMember]` fields, enabling distributed field sampling pipelines where each worker serializes a local `RunningStatistics` and the aggregator calls `Combine` on all partial results. `RunningStatistics.Push(double value)` propagates `NaN`: a single `NaN` in the field sample stream permanently poisons `Minimum`, `Maximum`, and all moment estimates; there is no `Reset()` method — recovery requires constructing a new instance.

[FOURIER_FIELD_TRANSFORM]:
- `Fourier.ForwardMultiDim(Complex[] samples, int[] dimensions, FourierOptions)` performs a multi-dimensional DFT over a flat row-major array. The `FourierOptions` enum controls convention: `Default = SymmetricScaling` (`1/√N` in both directions), `AsymmetricScaling` (no forward scaling, `1/N` inverse), `NoScaling | InverseExponent`. For field power spectrum computation, the forward transform under `AsymmetricScaling` followed by pointwise `|X[k]|²` is the standard convolution-theorem path; using `SymmetricScaling` introduces an incorrect `1/N` factor in the spectral density.
- `Fourier.Forward2D(Matrix<Complex> samples, FourierOptions)` extracts the matrix via `DenseColumnMajorMatrixStorage<T>.AsColumnMajorArray()` for dense column-major storage — this returns the internal `Data` array directly with no copy and passes swapped `(columns, rows)` dimensions to the row-major FFT kernel. `AsRowMajorArray()` on the same storage always produces a copy. For 2D FFT where zero-copy is critical, the `Forward2D(Complex[] samplesRowWise, int rows, int columns, FourierOptions)` overload on a pre-assembled row-major array avoids the storage branch entirely.
- `Correlation.Auto(double[] x)` zero-mean-centers the data before the FFT and normalizes by the zero-lag variance, so the autocorrelation at lag zero is always `1.0` by construction — not the empirical variance. Callers expecting unnormalized autocorrelation must multiply by the sample variance. The method zero-pads to `CeilingToPowerOfTwo * 2`, guaranteeing linear (acyclic) convolution with no wrap-around aliasing.

[ODE_AND_FIXED_STEP_INTEGRATION]:
- `RungeKutta` exposes two static fixed-step solvers: `SecondOrder` (Heun's method) and `FourthOrder` (classical RK4). Both accept scalar `double y0` or vector `Vector<double> y0`, with a `Func<double, T, T>` right-hand side, start/end times, and partition count `N`. Return is `T[]` — the full trajectory at uniform steps. There is no adaptive step control, no dense output, no error estimate, and no event hook. The tableau is hardcoded per method; there is no tableau-injection surface.
- `AdamsBashforth` exposes `FirstOrder` (Euler), `SecondOrder`, `ThirdOrder`, and `FourthOrder` explicit Adams-Bashforth linear multistep methods, all fixed-step, all returning `double[]` trajectories. No predictor-corrector (Adams-Bashforth-Moulton) surface exists in the library. Startup for orders ≥2 uses the prior steps from the array directly — there is no Runge-Kutta startup phase, so initial accuracy depends on step size alone.
- The entire `OdeSolvers` surface is fixed-step with no adaptive stepping, no dense output, no event localization, and no tableau validation. The lane's declared design pressures — tableau receipt validation, order-moment conditions, generic dense output from stage derivatives, adaptive step safety factors, clamped step ratios, reject budgets, and event localization by sign-change bracketing — have no implementation surface in the admitted package universe. These are build gaps: the field, sampling, and integration design law must declare them as required custom surfaces and specify that any integrator implementation must record tableau structural laws (consistency, order conditions) and adaptive step evidence (proposed step, safety factor, accept/reject state) as typed receipts rather than bare scalar outputs.

[SIMD_AND_TENSOR_FLOOR]:
- `TensorPrimitives` on current runtimes exposes generic-math overloads constrained by `IAdditionOperators<T,T,T>`, `IFloatingPointIeee754<T>`, `IRootFunctions<T>`. For field accumulation over `float` or `double` spans — gradient dot products, weighted kernel sums, diffusion residuals — `TensorPrimitives.Dot<T>`, `MultiplyAdd<T>`, and `FusedMultiplyAdd<T>` eliminate scalar loop overhead through hardware FMA and AVX-512 on supported processors.
- `TensorPrimitives.Norm<T>(ReadOnlySpan<T>)` computes `T.Sqrt(SumOfSquares(x))` with no scaled accumulation and no overflow or underflow protection — it is the naive squared-sum-then-sqrt path. For field vectors with components near `sqrt(double.MaxValue / N)`, the squared-sum overflows to `Infinity` before the square root. Sampling acceptance criteria and kernel-center distance checks that use `Norm` directly must guard the input or use a two-pass scaled implementation.
- `TensorSpan<T>` and `ReadOnlyTensorSpan<T>` are N-dimensional with `nint[]` lengths and strides. A field sampled on a regular grid can be represented as a `TensorSpan<double>` with strides derived from grid dimensions; slice operations along any axis are zero-copy subspan projections, enabling plane-by-plane integration without copying.
- `TensorPrimitives.SoftMax(ReadOnlySpan<float>, Span<float>)` calls `Exp` over the input span, then divides by the sum without subtracting the maximum first — numerically unstable for inputs above ~88 for `float`, where `Exp` overflows to `Infinity`. The numerically stable path requires manual pre-subtraction of `TensorPrimitives.Max(x)` before calling `Exp`. For field-derived energy distributions with unscaled magnitudes, the naive `SoftMax` produces `NaN` or `Infinity` without error.
- Custom per-element field kernels over `TensorSpan<T>` must be built by calling `TensorPrimitives.*` span overloads directly inside a scalar stride-walking loop; there is no public operator-injection interface in `TensorPrimitives` — the `IUnaryOperator<TInput,TOutput>` interface is internal. `TensorOperation` is an internal static class whose dispatch infrastructure is not accessible for custom kernel injection.

[FIELD_LAW_GAPS_AND_DESIGN_PRESSURE]:
- No Sobol or Halton quasi-random sequence is present. The `MathNet.Numerics.Random` namespace contains: `Mcg31m1`, `Mcg59`, `MersenneTwister`, `Mrg32k3a`, `Palf`, `SystemRandomSource`, `WH1982`, `WH2006`, `Xorshift`, `Xoshiro256StarStar`. Low-discrepancy sequences for quasi-Monte Carlo field integration must be sourced externally.
- Multi-dimensional scattered-data reconstruction is absent from the library: `KernelDensity` is a density estimator, not a function interpolant, and `Fit`/`Interpolate` are 1D surfaces. The correct composition is to assemble an RBF or polynomial design matrix from point coordinates and pass it to `MultipleRegression.DirectMethod` with `Svd` for rank safety; the rank check, condition number, and orientation agreement for acceptance gating are caller responsibilities via `SvdEvd` from the linear algebra layer.
- Neither `DoubleExponentialTransformation` nor any adaptive Gauss-Kronrod overload catches `NaN` or `Infinity` propagated by the integrand. A signed-distance function evaluated at its zero set returns `NaN`; this propagates silently through the integrand without diagnostic. The only defense is a finite-fallback guard on the integrand delegate, with the failure recorded typed outside the integral call.
- `Distance.SSD(double[] a, double[] b)` dispatches to `LinearAlgebraControl.Provider.SubtractArrays` and `DotProduct` — BLAS-backed when a native provider is loaded. `Distance.Euclidean` wraps `Math.Sqrt(SSD(a, b))` without scaled-accumulation overflow protection; for field comparison vectors with components above `√(Double.MaxValue / N)`, the intermediate squared sum overflows silently. `Distance.Canberra` returns `NaN` per term when both `a[i] == 0` and `b[i] == 0` (IEEE `0/0`), which propagates into the sum — the denominator must be guarded for sparse field vectors. `Distance.Cosine` calls `LinearAlgebraControl.Provider.DotProduct` three times and does not handle zero vectors; `Distance.Pearson` via `Correlation.Pearson` uses a two-pass Welford-style computation and returns `NaN` for constant series, making it the safer route for field sample correlation where any dimension may be constant.
- The library provides no spatial index for point location or pairwise statistics. `KernelDensity.Estimate` and all distance-matrix computations are `O(N²)` all-pairs. Scattered-data reconstruction and blue-noise sample validation at scale require an external k-d tree or BVH; the library has no surface for this, making any pairwise statistic a design-owned gap.
- Entropic optimal transport is entirely absent from the admitted package universe. Log-space Sinkhorn (max-shifted log-sum-exp, underflow clamping, debiasing via paired self-transport, mode-forked residual semantics) and the hard nonconvergence contract are build gaps with no partial implementation to compose from. Any transport-distance field integration must be implemented from scratch with explicit typed failure evidence for nonconvergence — a best-effort fallback is architecturally inadmissible under the lane's law.
