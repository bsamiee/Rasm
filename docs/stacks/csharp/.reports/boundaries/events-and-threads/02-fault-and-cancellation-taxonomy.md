# Fault and Cancellation Taxonomy at the Seam

[CANCELLATION_IS_A_TYPED_VALUE_NOT_AN_ESCAPE]:
- Every exception that crosses the seam interpreter is funneled through one normalizer that maps the host's three control-flow exception families onto fixed-code rail values: `TaskCanceledException` and `OperationCanceledException` both collapse to `Errors.Cancelled` (code `-2000000001`), `TimeoutException` collapses to `Errors.TimedOut` (code `-2000000002`), and `AggregateException` fans out into a coalesced multi-error. The consequence is that cancellation is never an out-of-band stack unwind inside the effect rail ‚Äî it is an ordinary catchable, inspectable, retryable error value sitting in the failure channel beside domain faults.
- Because the normalizer runs inside the catch arm of the effect interpreter's per-node handler, the typed predicate of a catch combinator sees `Errors.Cancelled` exactly as it sees any expected error. There is no separate cancellation channel; the discriminant is the code, recovered with `e.HasCode(cancelCode)`, never a `catch (OperationCanceledException)` clause in domain code.
- The collapse of both `OperationCanceledException` and `TaskCanceledException` to the identical code means a seam cannot distinguish "the linked parent token cancelled" from "this fiber's local token cancelled" by error code alone ‚Äî the code carries the kind of stop, not the source of the stop. A seam that must react differently to caller-initiated shutdown versus an internal deadline must carry the source as separate evidence (a distinct error case constructed at the originating boundary), because the rail flattens both into one value.

[THE_TIMEOUT_CANCELLED_COLLISION]:
- A deadline combinator runs its body inline on the calling fiber under a child environment carrying a fresh timer-armed cancellation source; when the timer fires, it cancels that source and the next cancellation-observing primitive in the body throws `OperationCanceledException`. That exception normalizes to `Errors.Cancelled`, not `Errors.TimedOut` ‚Äî so a body that timed out is, in the rail, indistinguishable by code from a body whose parent token was cancelled. `Errors.TimedOut` is produced only when a raw `TimeoutException` is lifted directly, which the inline deadline path never does.
- The practical rule: a seam that needs "did this hop exceed its deadline" as a recoverable signal cannot read it off the deadline combinator's own output. It must arm an observable side-channel ‚Äî a flag set in the timer registration, or an explicit `fail(Errors.TimedOut)` raised by a body-internal deadline check ‚Äî and treat the combinator's `Cancelled` purely as the unwind mechanism. Reading `Cancelled` as "timed out" is wrong whenever the parent scope can also cancel.

[RETRY_LOOPS_ON_CANCELLATION_BY_DEFAULT]:
- The retry algebra is `Catch(BracketFail, e => predicate(e) ? fail(e) : recurse)` ‚Äî it retries on every error the predicate does not stop on. The bare retry form supplies the stop predicate `_ => false`, meaning "stop on nothing, retry on everything." Because cancellation is a normalized `Errors.Cancelled` value flowing through that same `Catch`, an unqualified retry loop treats a cancelled token as a retryable failure and spins ‚Äî each attempt re-enters a body whose cancellation-observing primitives immediately re-throw, producing a tight retry storm against a token that will never un-cancel.
- The correct stop predicate for any retry that may run under a cancellable scope must short-circuit the cancellation code first: the predicate stops when `e.HasCode(cancelCode)` or when the parent token reports cancellation, and only then applies the domain-fault retry policy. Cancellation is the one error that is never transiently recoverable and therefore must be the first arm of every retry predicate at a seam.

```csharp
static IO<A> reactivate<A>(IO<A> attach, Schedule policy) =>
    attach.RetryUntil(policy, e => e.HasCode(Errors.Cancelled.Code) || e.HasCode(Errors.TimedOut.Code));
```

- The schedule-driven retry threads its accumulator across iterations and, on schedule exhaustion, fails with the *last* observed error rather than the first ‚Äî so a probe that fails N times surfaces only the final fault unless the seam coalesces the sequence itself. The non-schedule retry recurses without an attempt accumulator and so carries no envelope, making the schedule form mandatory whenever the seam needs a bounded reactivation count.

[ERROR_IS_A_MONOID_AND_FAN_IN_COALESCES]:
- The rail's error type is a monoid: `operator +` (Combine) appends, single errors promote to a many-error trie on append, and many+many concatenates the underlying error sequences. `Errors.None` is the identity ‚Äî an empty many-error trie, semantically "a failure with no information," distinct from "no failure." This is the structural substrate for fault coalescing at any fan-in seam: collecting N boundary-probe failures is a fold with `+`, never a list the caller post-processes.
- A coalesced many-error answers discriminant queries by quantifier, not by its own identity: `IsExceptional` is `Exists(member.IsExceptional)` ‚Äî the aggregate is exceptional if *any* member is ‚Äî while `IsExpected` is `ForAll(member.IsExpected)` ‚Äî expected only if *every* member is. `HasCode(c)` is `Exists(member.Code == c)`, and `IsType<E>` is `Exists(member is E)`. A fan-in result therefore reports "contains a cancellation" or "contains a non-recoverable exception" without the consumer ever iterating the trie; the quantifier semantics are the recovery-decision surface.
- An aggregate exception crossing the seam is decomposed at the boundary: its inner-exception set is mapped through the same normalizer and folded into a many-error. A single-inner aggregate flattens to that one normalized error rather than a one-element trie, so a fan-out that produced exactly one fault does not pay the aggregate wrapper. This means cancellation hidden inside an aggregate (the common all-tasks fan-in shape) surfaces as a `Cancelled` member of the coalesced rail, catchable by code without unwrapping.

[TYPE_DIRECTED_CATCH_OVER_A_COALESCED_RAIL]:
- A type-directed catch arm matches not only a directly-typed error but a many-error that *contains* a member of the target type, because the match predicate is `e is E || e.IsType<E>()` and the aggregate's `IsType<E>` is membership. When the matched value is the aggregate rather than the bare type, the handler does not run once on the whole trie ‚Äî it filters the trie to only the `E`-typed members and chains the per-member handler across them. Two filter-and-chain strategies are exposed: a monad-bind variant that threads each matched member sequentially through the recovery (short-circuiting on the first recovery failure), and a monoid-fold variant that accumulates every member's recovery via the rail's `+`.
- The filter is total and order-preserving: it chooses the `E`-typed members out of the trie and yields `Errors.None` (identity, not a thrown "not found") when none match, so composing several type-directed arms over one coalesced fan-in fault is associative and never throws on a miss. A seam with heterogeneous probe failures ‚Äî some cancellations, some expected unavailabilities, some genuine exceptions ‚Äî installs one arm per error type, each draining its slice of the trie, and the arms compose into a single recovery surface rather than a per-member switch.

```csharp
var recover =
    catchOf<Exceptional, M, A>(ex => reportFatal(ex)) >>
    @catch(Errors.Cancelled.Code, () => stop) >>
    catchOf<Expected, M, A>(ex => retryLater(ex));
```

[RACE_LOSER_TERMINATION_DEPENDS_ON_WHO_LAUNCHED]:
- The race combinator is a doubly-awaited first-completion over the branch tasks: the *first task to complete* ‚Äî success or fault ‚Äî is the result, and a faulted first-completer rethrows directly. The combinator does **not** collect failures and continue waiting for a later success, despite its own documented intent reading that way ‚Äî the first-completion await rethrows immediately. Conflating the race with an error-collecting fan-in is a real availability bug, because the race returns the first failure as eagerly as the first success.
- Loser termination is decided by which scope launched the branches, and the three race overloads diverge silently. Racing raw effects runs every body under a fresh child cancellation scope and cancels that scope after the first completion ‚Äî losers observe the cancel and die. Racing effects that *construct* forks runs the fork construction under the child scope, so each fork's source links to it and the post-completion cancel propagates into every loser. Racing already-launched fork handles awaits each fork under the caller's own environment while the child scope the race creates owns nothing ‚Äî cancelling it terminates no loser, and the losing forks run to completion in the background. Racing pre-launched forks buys first-result latency, never loser termination; a seam that must guarantee losers stop hands the race the un-launched effects and lets the race itself launch them.
- The race cancels its child scope, not the parent source ‚Äî winning short-circuits the rest without tearing the parent scope, and loser cancellation surfaces nowhere because the losing tasks are never awaited. A pre-entry guard short-circuits the whole combinator: if the ambient token is already cancelled when the lifted body begins, it throws before any branch spawns, so a race cannot be relied on to "run at least one probe" under an already-cancelled scope ‚Äî the fan-out is skipped entirely and surfaces as `Cancelled` with no probe ever started.
- The all-completes fan-in is the structural opposite on every axis: it awaits every branch under the original parent environment (inheriting parent cancellation directly), never cancels a sibling on failure, and returns results in branch order ‚Äî a deterministic join whose ordering is positional, not temporal. Per-branch bounding therefore belongs on each fork itself (a timeout on the forked effect), not on the join; the join has no per-branch cancellation lever at all.
