## [FINDINGS]

Finding 1
    Active owner/section: `docs/standards/proof.md` `[2][EVIDENCE_HIERARCHY]`, `[3][CONFLICT_HANDLING]`, `[10][AGENT_SURFACE_EVALUATION]`.
    Evidence source URLs: https://www.w3.org/TR/prov-overview/; https://slsa.dev/spec/v1.2/verifying-artifacts; https://c2pa.wiki/specifications/C2PA_Specification.pdf.
    Finding: W3C PROV treats provenance as object identity, attribution, processing steps, derivation, reproducibility, versioning, and provenance-of-provenance. SLSA separates provenance from verification: provenance has value only when inspected against expected values and a root of trust. C2PA makes validation a failure-classed process where unresolved, mismatched, undeclared, malformed, or missing references reject the claim.
    Weakness/inconsistency: Rasm already has strong evidence fields, but the current wording can still let `Evidence:` read like "a source exists" rather than "this claim was verified against an expectation." Generated and provider claims need a visible split between declaration, expected value, verification method, and failure class.
    Proposed correction: Add a proof-owned rule: evidence records for generated, provider, renderer, support, and machine-facing claims must name the claimed fact, the controlling source, the verification or comparison performed, and the failure state when the check cannot prove the claim.
    Active owner: `docs/standards/proof.md`.
    Ripple files: `docs/standards/agentic-documentation.md`; `docs/standards/reference/api.md`; `docs/standards/reference/reference.md`; `docs/standards/reference/support-matrix.md`.
    Decision: PROMOTE.
    Proof gap/question: This report did not test any local generated ledger. Promotion should stay generic unless a local parser, generator, or validator is sourced.

Finding 2
    Active owner/section: `docs/standards/proof.md` `[6][ASSERTION_UNCERTAINTY]`, `[8][SOURCE_RESEARCH]`.
    Evidence source URLs: https://developers.google.com/style/word-list; https://developers.cloudflare.com/style-guide/formatting/dates-and-times/; https://developer.mozilla.org/en-US/docs/MDN/Writing_guidelines/Experimental_deprecated_obsolete.
    Finding: Google warns against time-decaying words such as `future`, `soon`, and unsupported `latest`, and says `latest` needs a reference point such as a version or release date. Cloudflare states that documentation should prefer universal truth over time-bound wording because dates and explicit time references become stale. MDN defines `experimental` through explicit support/spec-change criteria and defines when that status may expire.
    Weakness/inconsistency: `proof.md` says to mark gaps and drift triggers, but the active standards do not yet provide a sharp stale-source decision rule for vague freshness terms. Without a source or review trigger, `latest`, `current`, `legacy`, `experimental`, `deprecated`, and `obsolete` can become unowned assertions.
    Proposed correction: Add a stale-source handling rule: any claim using a freshness, lifecycle, or support term must either carry a source-specific definition plus `Last verified:` or `Review trigger:`, or be rewritten into a source-independent invariant.
    Active owner: `docs/standards/proof.md`.
    Ripple files: `docs/standards/reference/support-matrix.md`; `docs/standards/reference/reference.md`; `docs/standards/agents-md.md`.
    Decision: PROMOTE.
    Proof gap/question: Exact product lifecycle labels remain type-local; `proof.md` should define the evidence obligation, not the support vocabulary.

Finding 3
    Active owner/section: `docs/standards/proof.md` `[4][PROOF_FIELDS]`; `docs/standards/information-structure.md` `[6][STRUCTURED_RECORDS]`, `[9][DEFINITION_BLOCKS]`.
    Evidence source URLs: https://docs.gitlab.com/development/documentation/styleguide/; https://kubernetes.io/docs/contribute/generate-ref-docs/kubectl/.
    Finding: GitLab uses source comments to tell maintainers when a table is autogenerated, what source file to edit, and what script to run. Kubernetes generated `kubectl` reference docs route edits through source comments first, then generation commands, version inputs, generated output existence checks, and copied-file checks.
    Weakness/inconsistency: Rasm templates already include `Generated from:` and `Source of truth:`, but generated ledgers need a compact carrier that also records refresh command, output artifact, source edit route, and mismatch behavior. This belongs in form; proof owns field meaning.
    Proposed correction: Add a generated-ledger packet in `information-structure.md` that uses proof-owned fields: `Generated from:`, `Source of truth:`, `Refresh command:`, `Generated artifact:`, `Expected output:`, `Proof gap:`, `Last verified:`, and `Review trigger:`. Omit absent fields; do not hand-maintain generated rows.
    Active owner: `docs/standards/information-structure.md`.
    Ripple files: `docs/standards/proof.md`; `docs/standards/reference/api.md`; `docs/standards/reference/reference.md`; generated reference pages if accepted later.
    Decision: PROMOTE.
    Proof gap/question: If any local generated ledger has a parser-enforced schema, the packet must cite that parser before field order becomes machine-enforced.

Finding 4

Finding 5
    Active owner/section: `docs/standards/agentic-documentation.md` `[7][TASK_OUTPUT_CONTRACTS]`, `[10][RETRIEVAL_PROVENANCE]`; `docs/standards/proof.md` `[10][AGENT_SURFACE_EVALUATION]`.
    Evidence source URLs: https://www.w3.org/TR/prov-overview/; https://slsa.dev/spec/v1.2/verifying-artifacts.
    Finding: PROV and SLSA both support the current Rasm split between source provenance, semantic validation, and runtime safety. Provenance records identify where something came from and how it was produced; they do not by themselves prove correctness, safety, authorization, or current applicability.

    Proposed correction: Add candidate wording to `agentic-documentation.md`: "A source trace is not a semantic pass. Pair claim provenance with the validation that checks the claim, or mark `Proof gap:`."
    Active owner: `docs/standards/agentic-documentation.md`.
    Ripple files: `docs/standards/proof.md`; retrieval indexes, generated mirrors, and MCP catalog standards if promoted later.
    Decision: MERGE.
    Proof gap/question: Do not imply probabilistic retrieval quality without evaluation receipts; that remains governed by `proof.md`.

Finding 6
    Active owner/section: `docs/standards/reference/api.md` `[5][CONTRACT_RECORDS]`; `docs/standards/reference/reference.md` `[3][REQUIRED_STRUCTURE]`, `[5][FACT_ENTRIES]`; `docs/standards/reference/support-matrix.md` `[5][LIFECYCLE_BASELINES]`.
    Evidence source URLs: https://developer.mozilla.org/en-US/docs/MDN/Writing_guidelines/Experimental_deprecated_obsolete; https://developers.google.com/style/word-list; https://docs.gitlab.com/ee/development/documentation/styleguide/deprecations_and_removals.html.
    Finding: Mature docs systems make status words operational. MDN defines status criteria and expiry conditions for `experimental`; Google requires a reference point for `latest`; GitLab uses process-backed deprecation/removal documentation before content deletion when a feature is generally available.
    Weakness/inconsistency: Reference templates often include evidence fields but omit `Proof gap:` in examples where missing proof is a valid state. Support and API templates should normalize proof gaps where generated, current, upstream, lifecycle, or provider behavior can be unavailable.
    Proposed correction: Insert `Proof gap:` after `Source of truth:` in proof-bearing templates that can represent generated output, support state, current command output, renderer behavior, provider behavior, or upstream facts. Produced docs omit it when the claim is proved.
    Active owner: `docs/standards/proof.md`.
    Ripple files: `docs/standards/reference/api.md`; `docs/standards/reference/reference.md`; `docs/standards/reference/support-matrix.md`.
    Decision: PROMOTE.
    Proof gap/question: The field belongs in templates only where missing proof is allowed. Do not add filler `Proof gap: none` to produced docs.

Finding 7
    Active owner/section: `docs/standards/formatting.md` `[2][STATUS_RESULT_MARKERS]`; `docs/standards/proof.md` `[6][ASSERTION_UNCERTAINTY]`.
    Evidence source URLs: https://c2pa.wiki/specifications/C2PA_Specification.pdf; https://developer.mozilla.org/en-US/docs/MDN/Writing_guidelines/Experimental_deprecated_obsolete.
    Finding: Validation and support states are not interchangeable. C2PA validation failures are machine-check result classes. MDN status labels are reader-decision states about technology maturity and stability. Both use closed criteria, but each belongs to its own domain.
    Weakness/inconsistency: Rasm formatting owns marker spelling but not proof or support semantics. The standards should keep `Proof gap:`, `[UNKNOWN]`, `[PARTIAL]`, support labels, lifecycle labels, and validation result tokens separated by owner and use case.
    Proposed correction: Add a boundary note: uncertainty is carried by `Proof gap:` when evidence is missing; `[UNKNOWN]` is an inline value marker only after a local vocabulary declares it; support and lifecycle states stay in their type standard.
    Active owner: `docs/standards/formatting.md`.
    Ripple files: `docs/standards/proof.md`; `docs/standards/reference/support-matrix.md`; type standards with status vocabularies.
    Decision: MERGE.
    Proof gap/question: User choice on compact glyphs remains separate and should not block proof-gap wording.

## [RECOMMENDATIONS]

[PROMOTE]:
- Add proof wording that separates declaration, provenance, verification, expected value, and failure class.
- Add stale-source wording for `latest`, `current`, `soon`, `future`, `legacy`, `experimental`, `deprecated`, `obsolete`, support, and lifecycle claims.
- Add `Proof gap:` to proof-bearing templates where missing proof is a valid state, then instruct produced docs to omit it when proved.
- Add a configured-gate record that names what each docs gate proves and what remains unproved.
- Add a generated-ledger packet in form guidance, with proof labels routed to `proof.md`.

[MERGE]:
- Merge agent-surface wording that says source provenance does not prove semantic correctness.
- Merge formatting boundary wording so status/result markers do not replace `Proof gap:`.
- Merge reference/support-matrix wording so status labels require source criteria and review triggers.

[DROP]:
- Do not create a parallel uncertainty vocabulary outside `proof.md`.
- Do not add generic `Confidence:` fields; use `Evidence:`, `Proof gap:`, validation receipts, and source strength instead.
- Do not add calendar review dates where a source-event `Review trigger:` is enough.

## [CANDIDATE_WORDING]

For `docs/standards/proof.md`:

```markdown conceptual
[CLAIM_VERIFICATION]:
- Declaration: the claim made in prose, a row, a generated artifact, or a status record.
- Provenance: source path, generated artifact, upstream source, or provider page that produced the fact.
- Expectation: value, status, schema, source version, renderer behavior, support phase, or generated output the claim must match.
- Verification: command, source comparison, generated-output check, maintained-source read, renderer proof, or human review.
- Failure state: `Proof gap:` when verification is unrun or unavailable; specific failure class when a validator provides one.

Provenance identifies origin. Verification proves agreement with expectation. A source link without comparison is not a semantic pass.
```

For `docs/standards/proof.md` stale-source handling:

```markdown conceptual
[STALE_SOURCE_RULE]:
- Use a source-independent invariant when possible.
- When a claim depends on freshness, support, lifecycle, deprecation, generated output, provider behavior, or the word `latest`, include a reference point and either `Last verified:` or `Review trigger:`.
- When the source cannot be checked, write `Proof gap:` beside the claim and remove any wording that presents the claim as verified.
```

For `docs/standards/information-structure.md`:

```text template
Generated ledger: `<artifact or table>`
Generated from: `<source, generator, command, schema, or query>`
Source of truth: `<source path, manifest, contract, upstream source, or generated source model>`
Refresh command: `<exact command; omit when no command exists>`
Expected output: `<artifact path, row set, schema, status, or comparison target>`
Proof gap: `<missing generator, unrun command, unsupported source, or unavailable renderer; omit when proved>`
Last verified: YYYY-MM-DD
Review trigger: `<source, generator, schema, command, package, provider, or renderer changes>`
```

For `docs/standards/agentic-documentation.md`:

```markdown conceptual
A source trace is not a semantic pass. Pair claim provenance with the validation that checks the claim, or mark `Proof gap:`.
```
