# [PY_ARTIFACTS_API_FAST_SUGIYAMA]

`fast-sugiyama` is the categorical-best layered-graph placement engine for the diagram-layout rail: a Rust/PyO3 (`_fast_sugiyama.abi3.so`, abi3) Sugiyama pipeline — cycle removal, layer ranking, dummy-vertex insertion, ordered crossing minimization, and horizontal coordinate assignment — exposed through one polymorphic `from_edges(edges, ...)` boundary that accepts a mixed-type `(source, target)` edge list, int-maps it for the Rust core, runs the full barycenter/median crossing-reduction placement, and re-projects the result back onto the original node ids as a `Layouts` value. `Layouts` is a `list[LayoutType]` subclass carrying the post-placement transform algebra (per-component alignment, dot-style top-align+compaction, rect-pack disjoint-component arrangement, horizontal/vertical sort, origin/bbox query, dict projection) and the composite `dot_layout`/`compact_layout` recipes — so a hierarchy diagram gets Graphviz-`dot`-grade layered coordinates plus multi-component packing in one in-process native call, no `dot` subprocess and no Python placement loop. The package owner SUPERSEDES the `grandalf` `SugiyamaLayout` placement computation in `visualization/diagram/layout#LAYOUT`'s `HierarchyEngine.GRANDALF` arm; `grandalf` is retained only as the parity-fallback placement provider and the spline/line edge-route source (`route_with_lines`/`route_with_splines`) until `fast-sugiyama` placement parity is proven, after which `grandalf` is flagged for removal in the final `pyproject` reconciliation. It owns layered coordinate assignment only: it computes node `(x, y)` and component `(width, height)`, never SVG (that is `visualization/diagram/draw#DRAW`'s), never graph analysis (the `data/graph#GRAPH` `rustworkx` analysis kernel stays at the data plane), and its synchronous native placement is offloaded onto the `anyio.to_thread.run_sync` `CapacityLimiter` lane the layout owner already runs `_render` under, so the Rust call never blocks the event loop.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: the `Layouts` placement container and its layout tuple
- rail: diagram-layout

`from_edges` returns one `Layouts` — a `list[LayoutType]` subclass, ONE element per connected component of the input graph. Each `LayoutType` is the 4-tuple `(positions, width, height, edges)`: `positions` is the `Sequence[(node_id, (x, y))]` placed-node coordinate list keyed on the ORIGINAL node id (re-projected from the Rust int indices), `width`/`height` are the component's laid-out extent, and `edges` is the optional re-projected edge list (carrying inserted dummy-vertex routing points when `dummy_vertices=True`, else the original edges or `None`). A diagram consumer reads coordinates off the typed positions, never re-parses a `dot` string; the `__all__` surface is exactly `{from_edges, layout, Layouts}` — `layout` is the transform-function module, every other name is a `_types` alias or the private `_fast_sugiyama` extension.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: `from_edges` — the one placement boundary
- rail: diagram-layout

`from_edges(edges, ...)` is the single polymorphic placement entrypoint — there is no `layout_tree`/`layout_dag`/`layout_components` family; one call discriminates on the edge list and returns every component laid out. It collects the unique node set, builds an `int`-index map, lowers the edges to int pairs, calls the Rust `_from_edges` (with `encode_edges=False` forced — the public boundary never returns Rust-encoded edges), then re-projects each component's positions and edges back onto the original `NodeIDType` ids into a `Layouts`. Every knob defaults to `None` (the Rust core supplies its built-in default) EXCEPT `vertex_spacing`, which the boundary defaults to `PYDOT_SPACING` (72, the Graphviz-`dot` grid pitch) so default output matches `dot` spacing. The Rust core (`_from_edges` in the `.pyi`) additionally exposes `encode_edges`; `from_edges` deliberately does not — a consumer needing the encoded-edge wire calls the extension directly, which the diagram owner never does.

## [04]-[IMPLEMENTATION_LAW]

- import: `from fast_sugiyama import from_edges, Layouts` (and `from fast_sugiyama import layout` for the free functions) at boundary scope only; the distribution is `fast-sugiyama`, the import package is `fast_sugiyama`, and the version is both `importlib.metadata.version("fast-sugiyama")` and the package `__version__`.
- placement axis: `from_edges(edges, ...)` is the one Sugiyama entrypoint; it is fed the `(source, target)` pairs from the `data/tabular#GRAPH` adjacency frame (or the `rustworkx` `PyDiGraph.edge_list()`), keyed on the SAME stable integer node index the layout owner already assigns and the glyph marks read back on, so no third re-keying is introduced. The whole `from_edges` call is synchronous native Rust — it runs INSIDE the layout owner's existing `_render` kernel that `_compute` offloads onto `to_thread.run_sync(self._render, limiter=_LAYOUT_LANES)`, never on the event loop and never under its own second limiter.
- knob axis: the `LayeredPolicy` fields lower to the `from_edges` knob arguments — `direction` (TB/BT/LR/RL) maps to `ranking_type` (`down`/`up`) plus a coordinate transpose for LR/RL (or the `transpose` pass), `crossing_minimization` selects `median` vs `barycenter`, `minimum_length`/`vertex_spacing`/`dummy_size` carry the spacing geometry; `dummy_vertices=True` is set exactly when the layered policy needs edge-bend routing points (long-edge waypoints) returned in the per-component `edges`, which is the in-engine substitute for `grandalf`'s `route_with_lines` polyline. A new layout knob is one `LayeredPolicy` field mapped to one `from_edges` argument, never a new entrypoint.
- result axis: the return is a `Layouts` — ONE `(positions, width, height, edges)` tuple per connected component, positions re-projected onto the original node ids. A single-graph diagram reads component `[0]`; a forest/multi-component diagram arranges every component through `dot_layout` (top-align + compaction, the single-`dot`-graph look) or `compact_layout` (rect-pack + sort + compaction, dense packing), then `to_dict()` flattens to the `{node_index: Point}` `LayoutMap` the glyph-construction fold consumes — so the layout owner's `LayoutResult` `(LayoutMap, RouteMap)` is built straight off the `Layouts` (positions → `LayoutMap`, the dummy-vertex `edges` bend points → `RouteMap`), no intermediate `dot` parse.
- routing axis: when `dummy_vertices=True`, the returned per-component `edges` carry the inserted dummy-vertex bend points — the native long-edge routing waypoints. The layout owner threads these into `RouteMap[(source, target)]` as the laid-out `points`, REPLACING the `grandalf` `EdgeViewer._pts` read for the placement-driven route; `grandalf` `route_with_splines` is retained only where a curved spline route (not a polyline bend) is wanted, until `fast-sugiyama` placement parity is signed off and `grandalf` is removed.
- gate axis: `Layouts.rect_pack_layouts` (and therefore `compact_layout`) does `import rpack` at call time; `rpack`/`rectangle-packer` is NOT in the admitted roster, so that path raises `ModuleNotFoundError` on the core interpreter. The disjoint-component arrangement defaults to `dot_layout` (pure-`Layouts`, no extra dependency); dense rect-packing is enabled by either admitting `rpack` centrally or composing the `Rasm.Fabrication` `RectanglePacker`/NFP nesting kernel over the component `to_bboxes()` rectangles — never a hand-rolled bin-packer.
- evidence: each layered placement records node count, edge count, connected-component count (`len(layouts)`), the resolved `ranking_type`/`crossing_minimization`, and the per-component extent `(width, height)` into the `visualization/diagram/layout#LAYOUT` `core/receipt#RECEIPT` `ArtifactReceipt.Diagram(key, kind, nodes, edges, algorithm)` case — `algorithm` becomes the `fast-sugiyama` placement tag, distinguishing it from the `grandalf`/`rustworkx`/projected arms in the same receipt family; the content key is the `xxh3_128` digest over the serialized graph wire joined to the resolved positions, so a re-placement under different knobs keys distinctly.
- boundary: `fast-sugiyama` owns layered-graph COORDINATE assignment (the Sugiyama pipeline) and component-level arrangement; it emits no SVG (that is `visualization/diagram/draw#DRAW`'s), runs no graph analysis (centrality/shortest-path/community stay in the `data/graph#GRAPH` `rustworkx` kernel), builds no presentation graph (the layout owner builds the `rustworkx` `PyDiGraph` from the tabular feed and passes only its `edge_list()` here), and rasterizes nothing. Force/radial/topological layout stays with `rustworkx`; spline edge-routing and the parity-fallback placement stay with `grandalf` until removal; the deleted forms are a hand-rolled Sugiyama loop where this engine owns it, a `dot` subprocess where this in-process native call replaces it, a re-keying of coordinates off the stable node index, a synchronous native placement left on the event loop, and a second graph-analysis kernel.

[STACKING]:
- A `STACKING`/`PROGRAM` (or any layered) diagram kind in `visualization/diagram/layout#LAYOUT` flips its `LayeredPolicy.engine` arm to `fast-sugiyama`: the `_render` kernel builds the `rustworkx` `PyDiGraph` from the `data/tabular#GRAPH` adjacency frame, calls `from_edges(graph.edge_list(), vertex_spacing=..., ranking_type=..., crossing_minimization=..., dummy_vertices=True)`, takes `dot_layout()` for the single-graph arrangement, and folds the resulting `Layouts` into the `LayoutResult` — positions → `LayoutMap`, dummy-vertex `edges` bends → `RouteMap` — all inside the one `to_thread.run_sync(self._render, limiter=_LAYOUT_LANES)` `anyio` lane, so the native Rust placement composes the universal structured-concurrency rail with zero new async surface.
- The `LayeredPolicy` `expression.tagged_union` case in `visualization/diagram/layout#LAYOUT` gains a `from_edges` knob mapping rather than a new case: `direction`/`route`/`crossing` fields lower to `ranking_type`/`dummy_vertices`/`crossing_minimization` arguments, the `HierarchyEngine` `StrEnum` discriminates `fast-sugiyama` from the retained `grandalf` and `rustworkx` arms, and the `match`/`case` over `policy.tag` stays the one total dispatch — the universal `expression` rail owns the discriminant, this engine owns one arm's body.
- The placement is content-addressed on the universal `xxh3_128` rail: the layout owner digests the `rustworkx` `node_link_json` graph wire joined to `repr(sorted(Layouts.to_dict().items()))` and the routed bend points, so two placements of one graph under distinct `ranking_type`/`crossing_minimization` knobs key distinctly and a laid-out diagram is content-addressable on its full form (topology + knobs + coordinates + routes), keyed by the same `ContentIdentity` owner every artifact receipt uses.
- Disjoint architectural components (a multi-zone program graph) arrange through `Layouts.dot_layout()` by default; when dense packing is wanted the component `to_bboxes()` rectangles feed the `Rasm.Fabrication` nesting kernel at the wire (the catalog's gated `rect_pack_layouts` `rpack` path is the in-package alternative iff `rpack` is admitted) — the layered placement and the component packing compose without either owner re-implementing the other.
- `grandalf` is co-resident only as the parity oracle and the curved-spline route source: `visualization/diagram/layout#LAYOUT` keeps the `route_with_splines` arm for spline edge routes and the `SugiyamaLayout` placement behind a `HierarchyEngine.GRANDALF` policy selection so a regression test can diff `fast-sugiyama` coordinates against `grandalf` coordinates on the same graph; once parity is signed off, `grandalf` drops out of `pyproject` and the catalog `grandalf.md` is flagged removed, leaving `fast-sugiyama` the sole layered-placement owner.
