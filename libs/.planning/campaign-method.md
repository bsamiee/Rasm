# [CAMPAIGN_METHOD]

The doctrinal approach behind every library campaign — the three first-class library branches `libs/csharp`, `libs/typescript`, and `libs/python`, each held to the same bar, and the cross-language core that binds them. Sessions load this file first: it states why these libraries are built this way, the method that produces them, the bar they are held to, and the staged work that remains before implementation. The authoring standard is `libs/.planning/README.md`; folder-specific entry points live in each package's root `README.md`; open work lives in `TASKLOG.md` at the tier that owns it.

[TWO_MODES]: PLANNING and IMPLEMENTATION are distinct modes and never interleave. PLANNING — the mode this file governs — does ALL the discovery (research, decompile and spike resolution, gap-hunt, app-ideation, inherited-pool binding) and produces a decision-complete, implementation-ready plan in which every change is already specified to the file, owner, row or case, and signature it lands on. Every working artifact a session builds — the gitignored research briefs and registers under `.artifacts/planning-briefs/`, the integration work-list manifest, the spike proofs under `.artifacts/spikes/` — is a PLANNING output that exists to make the plan decision-complete; none is an implementation-time activity. IMPLEMENTATION is a later, separate run that does no discovery: it dispatches workflows that apply the finalized plan — authoring or transcribing exactly what planning mapped — and finds nothing new. The test is simple: finding, researching, or building a manifest is PLANNING; dispatching workflows to apply a finished plan is IMPLEMENTATION.

[FENCE_LAW]: The unit of work is the transcription-complete CODE FENCE, never prose. For every owner, planning authors the real declaration — named members, union and enum cases, record fields, method signatures AND bodies wherever logic exists, and exact decompile-verified API spellings — so implementation copy-pastes it. Implementing, integrating, or ENRICHING an item means authoring or deepening that code fence: the measure of a done item is new or expanded code inside the fence, not new prose around it. An intent card, a rationale paragraph, a lone OWNER_REGISTRY / FEATURES / IDEAS row, a TASKLOG closure, or a bulleted description frames a fence — it never substitutes for one. An ENRICH or NEW item that adds only prose, a card, a bare row, or a ledger registration without deepening the owner's code fence is UNDONE: a defect to repair, not a completed item. The done-measure is the diff to the code fence, not the diff to the page.

[IDEATION_LAW]: Ideation is the campaign's engine and it is AGGRESSIVE and UNANCHORED. Each turn imagines the most ambitious applications the suite must power FROM FIRST PRINCIPLES — the apps a flagship company would build on it — and never lets prior completion cap ambition: a finished, world-class owner is the FLOOR for the next idea, never the ceiling, and "it is already done" is a forbidden stopping condition. Ideation deliberately resists context-anchoring — it does not derive ideas only from what the owners already do; it derives the most ambitious capability frontier the domain admits and then measures the gap to it (imagine first, compare second). The forward pool is `IDEAS.md` at the owning tier (per-folder `FEATURES.md` is the realized list; `IDEAS.md` is the reservoir of imagined concepts): every turn pushes new high-value concepts into IDEAS, maps exactly where each folder AND the cross-cutting surface fall short of building the imagined apps, and converts every actionable shortfall into authored fences this turn (APP-IDEATION → SHORTFALL → IMPLEMENT, with [FENCE_LAW] applying to the implementation). A turn whose ideation only re-confirms existing capability has failed its most important job; the measure of ideation is net-new ambitious capability mapped and implemented, not coverage re-stated.

[SIBLING_BRANCHES]: The campaign is polyglot across THREE first-class, peer library branches. `libs/csharp` (the four app packages and the geometry kernel), `libs/typescript` (the web/edge platform), and `libs/python` (the science/compute/data/geometry/IFC stack) are equal first-class library branches, never afterthoughts and never thin couplings to one another. Focus is ADDITIVE, never subtractive: each branch receives its OWN dedicated concurrent allocation, never effort diverted from a sibling. Each branch is a complete, agnostic, universal, professional library adoptable in ANY monorepo — every concern owned ground-up — re-derived from first principles every campaign (its package/folder/domain topology re-derived from the finalized owner set, never anchored to a stale layout), with its OWN central manifest (`Directory.Packages.props`/`Directory.Build.props`; `pnpm-workspace.yaml`; the single root `pyproject.toml`), catalogue extraction, ideation, pages, root docs, and ledger, running THIS method on its own cadence with its own workflow passes. The branches own their concerns natively, never as tacked-on mirrors: TypeScript owns its full web/edge platform — infrastructure and provisioning, the web app surface, the node tier, its OWN store/Postgres/SQL access, and a browser UI/UX domain distinct in kind from the C# Avalonia AppUi; Python owns a full science/compute/data stack — compute, data, geometry, and IFC pipelines driven root-up into a real pipeline that is an alternative to OR complement of the rhino/gh2 + C# stack. Aligned hand-in-glove only at the shared seam, NEVER coupled: each branch integrates with the others ONLY through the wire contracts and the companion/offline seams declared in the Tier-0 ledger — never another branch's interiors, host lifecycles, durable stores, or UI state. Each branch runs its OWN aggressive-ideation and keeps its OWN additive `IDEAS.md` + `FEATURES.md` + `TASKLOG.md` (informed by the sibling corpora for cross-pollination, never coupled to them); the closeout distillation is tri-language and unified (THE_METHOD phase 8, [9]). Driving the weaker-language doctrines (`docs/stacks/python`, `docs/stacks/typescript`) to match-or-exceed `docs/stacks/csharp` is an OPTIONAL operator-requested side-track, documented in the session-entry prompt, run per session only when asked.

## [1]-[THE_BAR]

These are lib-grade foundations, not app scaffolding. The bar for every package, current and future:

- World-class library code a company would adopt if it were public: the library internalizes the heavy logic, boilerplate, host quirks, native lifetimes, and failure handling so downstream apps compose capability instead of re-deriving it.
- No wrappers, no wrapper-faces, no rename adapters, no grab-bag abstractions. Every surface adds real functionality: high-value, higher-order capability built FROM the admitted packages, never a thinner face OVER them.
- Effortless complex apps without sacrifice: single entry points and unified rails reduce downstream agent overhead, and they do it by absorbing complexity internally — never by limiting tooling, hiding capability, or simplifying away power.
- Dense, optimized, polymorphic code: one owner per axis, one entrypoint family per rail, one fault family per package, total dispatch, fold algebras, table-driven logic — density is anti-redundancy and polymorphic collapse, not a line budget. Flat code, parallel rails, and near-duplicate shapes are defects.
- Files are as large as the owned concern requires; size follows the concern, never a line ceiling.
- Full doctrine adherence: the route-owned code doctrine (`docs/stacks/<lang>/`) is law, composed as given, never restated.

## [2]-[UNIVERSAL_PRINCIPLES]

- One unified body of functionality, not loose feature arms. A new capability is a row, case, or column on an existing axis — never a sibling surface. The pressure to add a second surface is the signal to deepen the first.
- Modality completeness from the root: every axis that varies by where code runs carries the full row set — host modalities, process topologies, deployment placements. Apps load and offload compute, data, and UI across those rows without new surfaces.
- Parameterization as a universal need: anything that can vary is a row value, never a hardcode. A new data source, asset source, theme probe, transport, store engine, or schedule is one row on the owning axis.
- External-lib maximization, truthfully: every admitted package is mined to its full capability and integrated through the owning axis; first-party companions and extensions are first-class rows from the root. An admitted capability no page exploits is a named gap.
- Plug-in-play composition, glove-in-hand integration: packages are complete in isolation and integrate only through the declared ports and the ledger seam laws. Coupling beyond the ports is a defect; so is a seam re-taught instead of consumed.
- Unified telemetry/logging/observability from the source: every package contributes through the contributor and receipt-sink ports; signals are plug-in-play at composition, never bolted on per app.

## [3]-[THE_METHOD]

A planning turn runs these phases in order, each finalized before the next. Every phase is PLANNING work — find, map, and FINALIZE to the decision-complete bar, never deferred to implementation — and every phase runs TRI-LANGUAGE across all three branches. The order front-loads all discovery so that by the integration gate the turn's entire work-set is realized as transcription-complete fences and implementation begins with zero re-research, re-mapping, or admission/layout surprise. The ALIGN and IDEATE phases run at three nested altitudes — package, language, cross-language.

1. INHERIT — consume the prior turn's distillation as the binding, decision-complete starting work-list: each language's pool (its `IDEAS.md` + `TASKLOG.md`) and the cross-language pool in `libs/.planning/`. Integrate it in FULL before any new ideation; the loop never restarts from a blank slate, and a pre-existing finding left unintegrated is a defect.
2. INFRA TRUTH — per language, make root config, manifests, lockfiles, tool pins, classification, and every `.api` catalogue honest before authoring; admissions execute with restore/resolve proof; every admitted package carries a decompile-grounded catalogue with zero member gaps and every existing catalogue is re-verified (phantom APIs are the named defect). The build stays centralized — versions, targets, RIDs, and settings live in the manifest, project files stay versionless, and a project-file edit is admitted only to drop a dead RID or off-platform native asset or to centralize a setting. Admission is bleeding-edge: every concern is served by the strongest modern library at newest stable, verify-before-pin — retaining a legacy package where a stronger equivalent exists is a named defect. A dedicated agent subset works only the manifest and the `.api` catalogues (the package-and-API lane). RE-ENTRANT: a finding in any later phase that needs a new pin, tool, catalogue, build edit, or structural change reopens this phase within the same turn ([9]).
3. RESEARCH + SPIKES — open one research-and-artifact track per open question, uncatalogued surface, version/compatibility unknown, and spike; resolve each in-session through decompile ([8].1) and throwaway-harness probes against the real packages, native libraries, servers, and runtimes ([8].2); the verified result folds back to FINALIZE the dependent fence and flip its `[STATE]` in the owner registry, and the brief persists gitignored under `.artifacts/planning-briefs/`. Conflicts resolve by the standard's binding precedence. The residual `SPIKE` set at exit is exactly the tier-3 live-host probes.
4. EXECUTE — author and deepen every owner to the world-class bar, per package, in isolation, parallel across the packages of a language. This integrates the full inherited pool, runs the categorical world-class gap-hunt ([7]) — naming and owning every missing category and every surface-level owner root-up — and mines every admitted package to its deepest useful surface. Every capability lands as a fence under the [FENCE_LAW]; every addition resolves through all of the language's code doctrine (`docs/stacks/<lang>/`). Imagine the concept set first, prune bloat ruthlessly, and ride every survivor on an axis row — never a sibling surface.
5. HARDEN — improve, critique, red-team, and harden the authored work; parallel authors transcribe under the standard and consolidated review-and-repair waves find AND fix in one pass. Ambitious synthesis is producer → red-team → merge, never a single agent. A page is finalized only on a cold review-law grade converging to a zero-edit pass ([5]); the producer's grade admits, the cold grade decides. The fragile branches earn an explicit ultra-harsh critique, a distinct red-team, and a hardening pass before any page is final.
6. ALIGN — three nested altitudes, in order:
   (a) PACKAGE — each package coherent in itself: owners, seams, the ARCHITECTURE owner registry state, the ARCHITECTURE source tree (the planned non-flat namespaced implementation tree, one leaf per build-order file, re-derived from the finalized owner set), lean FEATURES, and the package open-work surface reconciled against its ledger. A greenfield package nests its `.planning/` under source-mirroring sub-domain folders; a mature folder carries no `.planning/` and routes open work to `existing-work.md`; the ARCHITECTURE source tree is the build order (`architecture.md` [5]).
   (b) LANGUAGE — cross-package alignment within a language: ledger consolidation (owner symbols, seam splits, adjudications), one owner per axis, consumers reference and never re-mint, every two-package fact records its altitude split in the branch `region-map/seam-splits.md`, and a cross-package signature change propagates to every sibling caller this phase. Then the branch cross-cutting surface is authored — its cross-package laws in `ARCHITECTURE.md`, its concert IDEAS, its branch TASKLOG — and anchor integrity + single-state-surface verified across the language. The orchestrator is the sole ledger writer.
   (c) CROSS-LANGUAGE — tri-language seam coherence: every shared canonical concept (content identity, receipts, wire contracts, graduation evidence, the companion seam) carries ONE name and ONE owner across the three branches, consumed at the boundary and never re-minted. The cross-language alignment and its seam ledger live in `libs/.planning/region-map/` (the core); all three branches are first-class participants, aligned hand-in-glove and never coupled beyond the declared seam.
7. INTEGRATION GATE — verify 100% of the turn's contained work — every inherited-pool item, every gap-hunt finding, every research result, every admission integration — is realized as a transcription-complete fence (not a mention) and driven to FINALIZED, across all three languages, with a complete forbidden-token sweep. A single unintegrated finding or surviving forbidden token fails the turn and is repaired before ideation.
8. IDEATE + DISTILL — two nested altitudes, in order, and only after the gate ([9] carries the two-sided closeout-audit detail and the structural homes):
   (a) PER-LANGUAGE — each language runs its own aggressive, unanchored app-ideation ([IDEATION_LAW]): imagine the most ambitious apps that branch must power FROM FIRST PRINCIPLES, map exactly where each package and the language's cross-cutting surface fall short, implement every actionable shortfall now, and emit that language's additive next-loop pool to its branch `IDEAS.md`.
   (b) CROSS-LANGUAGE — the unified distillation: imagine the apps and concerts that SPAN languages (a flagship product built on C# + python + typescript at once), map the cross-lib shortfalls, implement what is actionable, and emit the cross-language next-loop pool to `libs/.planning/IDEAS.md`. The cross-language pool, the tri-language concert, and the cross-lib big ideas live in `libs/.planning/`; each branch's per-language pool lives in its own `libs/<language>/.planning/IDEAS.md`. The forward-emitted ideation pool is the ONLY thing the turn defers; a genuinely new categorical concern warranting a new package is surfaced as a candidate, never silently built.

## [4]-[STAGED_WORK_BEFORE_IMPLEMENTATION]

The campaign runs THE_METHOD ([3]) every turn, deepening the corpus across all three languages until it is implementation-complete; the arc toward implementation, in order:

1. DEEPENING TURNS — each turn runs THE_METHOD end to end: per-package execute + categorical gap-hunt (phase 4) re-reading every catalogue against every page and pushing capabilities further, the nested package→language→cross-language alignment (phase 6), and the per-language + cross-language ideation (phase 8) — so the corpus deepens every cycle until any world-class app is buildable from the pages alone.
2. SIBLING LIBRARY BRANCHES — the `[SIBLING_BRANCHES]` law governs in full; each branch (entry: `libs/<lang>/.planning/README.md`) runs THE_METHOD on its own cadence with its own workflow passes, as a concurrent additive stream. Each branch re-derives its topology from first principles, mines its catalogues to depth, drives its own pages to the bar, runs its own aggressive-ideation, and feeds the tri-language unified closeout (THE_METHOD phase 8). The branches align hand-in-glove at the shared seam and never couple beyond it.
3. FINAL ARCHITECTURE PASS — the closing pre-implementation session: per-package file plan finalized against the ARCHITECTURE source tree, logic-flow walkthrough of every entrypoint rail end to end, cross-package boot/drain/wire choreography rehearsed on paper, implementation-start gates executed (bridge spikes, native probes), and the proof-gate matrix confirmed green-able; the per-package source tree in `ARCHITECTURE.md` is refactored each session to match the finalized build order, so the tree is the live file plan, not a one-time sketch. Exit: implementation begins with transcription only.

## [5]-[QUALITY_MACHINERY]

- Every page graded cold per the standard's review law; research sections carry unresolved fact statements only — the executable proof rails live once in each folder's README PROOF_GATES by rail owner.
- The ledger (`region-map/` at the owning tier) is the single registry of owner symbols and seam laws; consult before naming any cross-package type; the orchestrating session is its only writer.
- Owner state lives in exactly one surface: the ARCHITECTURE `[OWNER_REGISTRY]` `[STATE]` column. FEATURES, IDEAS, pages, and the gitignored register carry no state marker; FEATURES and IDEAS route to the registry for state. A state value duplicated across surfaces is the named drift defect — one owner, one state cell, one edit site.
- Bleeding-edge posture is verified, not assumed: newest stable of every package at admission time with restore proof; performance laws are page rows that specs encode.
- Compute/threading/tensors are first-class architecture: one CPU budget across lane readers, pools, and parallel partitions; allocation classes with receipts; solve-path isolation; benchmark claims gated by host fingerprint.
- Proof gates per folder README: locked restore, api doctor/resolve, static plan/build, test target, bridge scenarios for host seams, local diagram render — each routed by rail owner, never a command catalog.
- Spike resolution ([8]): every `SPIKE` is driven to `FINALIZED` during planning by tier-1 decompile and tier-2 throwaway-harness proofs against the real packages, native libraries, and servers; only live-integrated-host probes remain open for the implementation session.

## [6]-[SESSION_ENTRY_PROTOCOL]

1. Read this file, then the cross-language `TASKLOG.md` and the target folder's `TASKLOG.md`, then the target folder's root `README.md` and `ARCHITECTURE.md` end-to-end.
2. Read the standard (`libs/.planning/README.md`) and the relevant ledger; code doctrine (`docs/stacks/<lang>/`) arrives settled.
3. Orchestrate with the method in [3]: ambitious synthesis is producer→red-team→merge (never a single agent), with parallel agents for reading, authoring, and repair — per-folder lanes parallel across folders and sequential within each. The orchestrator is the ledger's sole writer and the sole committer: after each phase gate it commits a scoped path set with explicit `git add -- <path>` of that gate's enumerated planning files only, never `git add -A` and never a foreign path. Workflow agents never run git.
4. Never re-derive: catalogues are decompile truth; pages are decision-complete; the binding precedence resolves conflicts; anything genuinely new lands as rows through the same machinery.

## [7]-[DEPTH_AND_GAP_HUNT]

The standing enrichment lens binding every per-folder deepening pass, beyond the session's starting register. Each session reads every folder in full depth and grades each owner against a world-class library that must power many diverse, ambitious applications at once; a surface-level owner, an under-exploited admitted package, or a missing capability category is a defect, not an acceptable baseline.

- World-class-or-defect: grade every owner at the bar of "nothing a flagship app dev would hand-roll," not "present." For each page, name what an ambitious app would still build by hand from this folder's domain — a query shape, a store engine, a codec, a data type, an import/export path, a chart kind, a transport, a streaming or chunking path, a viewport binding, a data-source adapter — and own it now. An owner that stops at table-stakes is a deepening target.
- Categorical gap-hunt, not row top-up: the hunt finds whole concerns the folder overlooks, not only rows on existing axes. A missing category is a new owned concern, surfaced and integrated this session.
- External-lib maximization: every admitted package is mined to its deepest useful capability from multiple angles — operator, combinator, extension point, generated surface, zero-alloc and performance path — and integrated at the deepest form the package reaches; an admitted capability no owner exploits is a named defect.
- Topology completeness: every modality an app can take — backend, server, remote, on-host, companion, sidecar, standalone, web-fed, hub, paired — is equally and fully realized through the host-modality row set, never one face built and the rest stubbed.
- Root-up integration, never flat spam: a found gap lands by deepening the canonical owner with a row, case, column, fold, or policy value, or by a NEW design page when a genuinely new owned concern warrants one — authored ground-up, woven into the densest owner, never a loose arm. Every addition resolves through `docs/stacks/<lang>/` in full and the standard.
- Additive and recurring: the gap-hunt runs every session as design pressure on top of the starting register, and each session closes by distilling the next frontier into `IDEAS.md`, so the corpus deepens every cycle and never plateaus.

## [8]-[SPIKE_RESOLUTION]

A `SPIKE` owner is a transcription-complete fence whose runtime correctness is gated on a probe. Spikes are driven to `FINALIZED` during planning, never deferred to implementation, by proving each probe against the real external packages, native libraries, servers, and runtimes in throwaway scaffolding, then folding the verified spelling and behavior back into the fence and flipping its `[STATE]` in the owner registry. The design corpus stays the deliverable; the proof harness is disposable.

The closure tiers, cheapest first; a spike rides the lowest tier that proves it.
1. Decompile and reflection — the external-artifact decompile/reflection rail and the `.api` catalogues confirm member existence and exact spelling with no execution, closing every uncatalogued-member RESEARCH row: the placeholder spelling in the fence is replaced with the verified one and the dependent owner row leaves `SPIKE`.
2. Throwaway harness against the real package — a minimal standalone proof under the gitignored temp root `.artifacts/spikes/<spike-id>/`, referencing the admitted package versions from the manifests, compiled and run against the real runtime to confirm behavior: a native library load on the target RID, a live server response against a throwaway database, an inference session run, a headless render, a serializer round-trip, a byte-identity or codec check. Tier-2 service provisioning runs through the repo-owned spike rail; the rail returns the proof as a structured envelope. The harness proves exactly the named probe; its verified output folds into the fence and card, and the temp dir is deleted.
3. Live integrated host — only probes that strictly require the running integrated host (Generic Host boot and unload inside the live plugin load context, native view embedding, in-process host-seam behavior) stay `SPIKE` until the host exists, and even these are de-risked standalone first. Any probe requiring interactive OS authentication is tier-3 by the same rule: a keychain item read, a biometric or credential-store read, or a signal-to-self delivery all prompt the operator and never run unattended — their owners are proven by tier-1 decompile member-shape ONLY and stay `SPIKE` until the integrated host.

- Every closed spike updates the page (RESEARCH row removed or member confirmed) and the owner registry `[STATE]` column (`SPIKE` to `FINALIZED`); a probe that reveals a runtime divergence reshapes the fence root-up — the spike has done its job by surfacing the unknown before it cost the system.
- The throwaway harness never enters package source, the planning pages, or the region ledger; it lives only under `.artifacts/spikes/` (gitignored), is referenced by no durable artifact, and is deleted once its result is folded in. Versions stay in the owning manifest; the harness reads them there and never re-declares them, and the ADMISSIONS_RECORD carries no pin.
- [WARNING] A throwaway harness NEVER triggers an OS credential, keychain, biometric, or password prompt: an unsigned harness reading a keychain item raises the login-keychain dialog and spams the operator with a password request that has no answer. Keychain, secret-store, biometric, and signal-delivery routes are proven by tier-1 decompile member-shape only and stay tier-3. Tier-2 provisioning is clean and additive: public container images only, operator containers and volumes are sacrosanct, and the provisioning architecture is full, powerful, and current — then torn down to its pre-run state except where a later phase reuses the stack.
- A session closes with every tier-1 and tier-2 spike finalized; the residual `SPIKE` set is exactly the tier-3 live-host probes, each named in the page RESEARCH cluster and carried as a TASKLOG row whose `STATUS` is `SPIKE`. The `SPIKE` count at session exit measures only host-bridge surface, never unproven library, native, or server capability.

## [9]-[THE_LOOP]

The campaign is a repeatable cycle, not a one-shot; each session is one turn that runs every phase of THE_METHOD ([3]) and the corpus deepens every turn. A turn begins by consuming the prior turn's distillation (phase 1) and ends by emitting the next turn's (phase 8); the next session begins at phase 1 against that fresh pool. Every phase is PLANNING: an unknown that surfaces during a turn is resolved in that turn, never parked for implementation. The loop is a cycle, not a one-way pipeline — a finding that surfaces late re-enters the phase that owns it and is finalized before the turn closes: a new package, stronger modern library, manifest pin, `.api` catalogue, build edit, or folder/page-set reshape reopens INFRA TRUTH (phase 2), the package-and-API lane, or the architecture re-derivation within the same turn. The only forward-deferred artifact is the closeout ideation pool (phase 8); a genuinely new categorical concern warranting a NEW package is surfaced as a candidate with its couples-to seam and the contradiction it must ratify, never silently built.

STRUCTURAL HOMES — where planning state lives:
- `libs/.planning/` is the CROSS-LANGUAGE CORE: the monorepo hierarchy law (`architecture.md`), this method, the authoring standard (`README.md`), the cross-language `IDEAS.md`, the cross-language `TASKLOG.md`, AND the cross-language alignment (`region-map/seam-splits.md`, `region-map/shared-owners.md`). It is the home of every concern that spans a language boundary; all three branches are first-class peers participating in the main workflow.
- A WIP folder's `.planning/` is a TRANSIENT greenfield scaffold: a greenfield package nests its design pages under source-mirroring `<pkg>/<SubDomain>/.planning/` folders; a mature folder with real code carries no `.planning/` and routes its open split/cleanup/re-architect work to `libs/csharp/.planning/existing-work.md`. A genuinely-new unbuilt sub-domain inside a mature package gets a scoped transient `.planning/`; its mature siblings do not. The full lifecycle law is `architecture.md` [5].
- `libs/<language>/.planning/` (csharp, typescript, python) is each branch's own peer home: its branch `README.md` (charter + TEST_POLICY), `ARCHITECTURE.md` (branch source tree + dependency graph + cross-package laws), its own additive `IDEAS.md` (the branch's big ideas and its concert), its own `TASKLOG.md`, its `api-catalogues.md`, and its branch `region-map/`. Each branch keeps the logical groupings its own domain needs — never a 1-1 name mirror of a sibling.
- `IDEAS.md` is the idea reservoir and the engine of phase 8: it holds the BIG ideas (the most ambitious apps, imagined first) and the lean concept rows distillation draws from (`[CONCEPT] | [PAGE#CLUSTER]` — no state, no owner-symbol chains). `FEATURES.md` (per WIP folder) is the realized list — one concrete capability + its page#cluster. A turn whose ideation only re-confirms existing capability has failed phase 8.

EXPECTED ADDITIONS PER ALTITUDE — what each phase nets:
- Package (phases 4-6a): deepened owners, new rows/cases/columns/folds closing categorical gaps, mined package capability, telemetry/diagnostics/metrics deepened with every capability.
- Language (phase 6b): cross-package alignment, one-owner-per-axis collapse, the branch concert surface (cross-package laws + concert IDEAS + branch TASKLOG), recorded seam-splits.
- Cross-language (phase 6c): shared-concept single-owner enforcement, cross-lib seam coherence, the tri-language concert — all homed in `libs/.planning/`.
- Per-language ideation (phase 8a): each branch's new ambitious apps, packages, and wider-branch concepts, emitted to its own `IDEAS.md`.
- Cross-language ideation (phase 8b): the apps and concerts that span C# + python + typescript, emitted to `libs/.planning/IDEAS.md`.

THE CLOSEOUT IS A TWO-SIDED AUDIT, NOT NEW-CONCEPTS-ONLY (phase 8): the closeout deep-read both ideates and audits existing depth. (i) Grade EVERY existing owner against the world-class bar ([7]) by deep-reading every `.api/` catalogue and every admitted external library end to end, and name each owner that is surface-level or under-exploits an admitted capability — pushing an existing body deeper is as much a closeout target as a new concept. (ii) Audit EVERY fence against ALL of the language's code doctrine (`docs/stacks/<lang>/`) for conformance and surface-collapse — a fence that violates a code law, or that could collapse parallel types/enums/methods into a denser union/smart-enum/fold/data-table, is a named target. Every such target joins the refreshed `IDEAS.md` pool as a detailed, owner-anchored next-loop task. Enrichment is universal and root-up: every existing file is pushed, deep-package-leveraged, and aligned every turn — never left at table-stakes — and its telemetry, diagnostics, and metrics deepen with every capability it gains.

The optional operator-requested doctrine-hardening side-track — driving `docs/stacks/python`/`docs/stacks/typescript` to match-or-exceed `docs/stacks/csharp` — runs per session only when asked, documented in the session-entry prompt; it is not a standing phase. The next session begins at phase 1 against the fresh distillation, and `campaign-method.md` records the loop so every turn launches from the same protocol.
