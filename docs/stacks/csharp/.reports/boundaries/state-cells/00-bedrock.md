# State-Cell Boundary Law

[CANCELLATION_SCOPE]:
- One root `CancellationTokenSource` per process; every scoped operation derives a child via `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` — the span overload that accepts a `ReadOnlySpan<CancellationToken>` directly, avoiding the params-array allocation path of the older overloads. The root owns the `IDisposable` lifetime; derived sources are disposed at scope exit, never held past drain.
- `CancellationTokenSource.TryReset()` returns `false` when the source is already cancelled — including a linked source whose parent fired — and `true` when the source is still live. A linked source that has not yet been cancelled resets successfully, but the parent link is not severed; parent cancellation still propagates after reset. `TryReset` is not a substitute for a new instance once cancellation fires; token-gated singleton cells that require re-open semantics must replace the entire `(CancellationTokenSource, CancellationToken)` pair on each open transition.
- `CancellationTokenRegistration` is both `IDisposable` and `IAsyncDisposable` and exposes `.Unregister()` and `.Token`. Registrations held past cancellation waste memory on already-fired callbacks; detach in the finally or Dispose path of the owning cell, not as a fire-and-forget side effect.
- `CancellationToken.UnsafeRegister(Action<object?>, object?)` avoids the execution-context capture of the safe overload; use only where the callback owns no ambient state that could create a security boundary leak.
- `Cancel()` runs `NotifyCancellation` inline, which calls `ExecuteCallbackHandlers` synchronously — all registered callbacks complete before `Cancel()` returns. `CancelAsync()` checks whether cancellation was already requested first; if not, it transitions the state and then, when there are registered callbacks, dispatches `ExecuteCallbackHandlers` to a thread-pool task via `Task.Factory.StartNew` and returns that task. The asymmetry: `Cancel()` blocks the caller until all callbacks complete; `await source.CancelAsync()` allows the caller to yield and resume after the callback-execution task completes on a pool thread. For a boundary cell whose teardown logic requires all callbacks to run before any subsequent work, `Cancel()` provides the synchronous seam and the caller proceeds only after callbacks complete; `CancelAsync()` is the correct form when blocking the calling thread on callbacks is unacceptable, because the await suspends rather than stalls.
- `new CancellationTokenSource(TimeSpan delay, TimeProvider provider)` creates a source whose timer uses the injected `TimeProvider`. A `FakeTimeProvider` can advance the clock to trigger the timeout in deterministic tests without real-time elapsed. `PeriodicTimer(TimeSpan, TimeProvider)` accepts the same overload. Both constructors start the timer immediately on construction. `PeriodicTimer.Period` is a read-write property; setting it mid-operation changes the interval immediately — the next `WaitForNextTickAsync` uses the new period, allowing a boundary cell to rate-adapt its polling loop without disposing and recreating the timer.
- `EnvIO.New` registers `token.Register(() => source.Cancel())` only when it creates a new `CancellationTokenSource` AND `token.CanBeCanceled`. The resulting `CancellationTokenRegistration` is stored in `Registration` and disposed in `DisposeNonResources()`. Failing to dispose a non-root `EnvIO` leaks the registration, which holds a delegate reference keeping the source alive until the parent token fires or the parent source is disposed. This contrasts with `CancellationTokenSource.CreateLinkedTokenSource`, which registers callbacks on all parent tokens at construction and allocates a linked source — each parent registration holds a delegate reference back to the source. For bounded-lifetime boundary cells that create linked sources per-operation, `Dispose()` must be called in every path (success, failure, and timeout) or parent cancellation callbacks accumulate as garbage.
- A `CancellationTokenSource` created with a timeout (`new CancellationTokenSource(TimeSpan)`) starts the timer immediately at construction. When `EnvIO.New` creates a timeout-scoped source inside a lazy `IO` tree, the timeout window begins at `IO` tree evaluation time. Stacking multiple `LocalCancelWithTimeout` calls does not compound the timeout — each call creates a new source from the current token at evaluation time.

[ENVIO_SCOPE_CELLS]:
- `EnvIO` is the runtime environment record threading through every `IO<A>` execution: public fields are `Resources Resources`, `CancellationToken Token`, `CancellationTokenSource Source`, and `SynchronizationContext? SyncContext`. The private `Own` int governs whether `Dispose` tears down the `CancellationTokenSource` (bit 1) and the `Resources` scope (bit 2). One `EnvIO` per execution spine; concurrent operations fork child envs, not second spines.
- `EnvIO.New` signature: `New(Resources?, CancellationToken, CancellationTokenSource?, SynchronizationContext?, TimeSpan?)`. When `source` is null, a new `CancellationTokenSource` is created and Own bit 1 is set; when `resources` is null, a new `Resources` scope is created and Own bit 2 is set. Creating a second root `EnvIO` from the same `CancellationTokenSource` creates two owners that can both cancel via the same token but dispose independently — only one should own the source.
- `EnvIO.Local` (property) computes `New(null, Token, null, SynchronizationContext.Current)` — allocates both a fresh `Resources` and a fresh `CancellationTokenSource`, sets `Own = 3`, and registers `Token.Register(() => source.Cancel())`. Full isolation: new resource scope, new cancellation scope, parent cancellation propagates inward, parent resources are inaccessible.
- `EnvIO.LocalResources` computes `New(null, Token, Source, SyncContext)` — null resources but the parent `Source` is passed. Allocates fresh `Resources` (Own bit 2 set) but reuses the parent `CancellationTokenSource` (Own bit 1 NOT set). No `Registration` is created. Disposing this env does not dispose the parent source; operations inside share the same token so parent cancellation still fires them.
- `EnvIO.LocalCancel` computes `New(Resources, Token, null, SyncContext)` — passes parent resources but null source. A new linked `CancellationTokenSource` is created (Own bit 1 set) with `Token.Register(() => source.Cancel())`. Disposing this env disposes the child source but not parent resources. Child cancellation does not propagate to parent; parent cancellation propagates to child.
- `EnvIO.LocalSyncContext` replaces the `SynchronizationContext` without creating a new cancellation scope — the correct form for marshaling into a specific thread's context while keeping the existing cancellation chain.
- `EnvIO.LocalCancelWithTimeout(TimeSpan)` and `LocalWithTimeout(TimeSpan)` create scoped envs with an additional timeout-based cancellation deadline layered onto the existing token. Parent fires first if earlier; child fires on timeout without cancelling the parent.
- `IO<A>.Local()` calls `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — a fresh cancellation scope with shared resources. The `.Local()` name is misleading: it is a cancellation isolation wrapper, not a resource isolation wrapper; `.Bracket()` is the resource isolation wrapper.
- `EnvIO.DisposeNonResources()` releases the `CancellationTokenSource` if Own bit 1 is set and disposes any `Registration`, without touching the `Resources` scope. `EnvIO.DisposeResources()` runs `Resources.DisposeU()` only when Own bit 2 is set, without touching the source. Both use `Interlocked.CompareExchange` for idempotency. Calling `DisposeNonResources()` then `DisposeResources()` separately (as `IOLocalOnFailOnly` does) is equivalent to `Dispose()`; the CAS guards prevent double-release.
- `EnvIO.DisposeResources()` checks `(Own & 2) == 2` before touching resources — an `EnvIO` that does not own its resource scope silently skips resource release even when `Dispose()` is called. This is correct for scopes that borrow a parent's resources; `Own bit 2` tracks allocation, not logical ownership. Additionally, `DisposeResources()` constructs a fully owned fresh `EnvIO` (Own=3) to pass to `Resources.DisposeU()`, but never calls `Dispose()` on that intermediate env — its `CancellationTokenSource` and `Resources` scope are abandoned to GC finalization. This is a minor resource leak per disposal and a pressure point for high-frequency disposal paths.
- `IO<A>.Fork(Option<TimeSpan> timeout)` returns `IO<ForkIO<A>>`. `ForkIO<A>` carries `Cancel: IO<Unit>` and `Await: IO<A>`. The forked computation runs in a child `EnvIO` constructed via `EnvIO.New(resources, token, tsrc, env.SyncContext)` with all four parameters non-null, so `Own = 0`; neither the `CancellationTokenSource` nor the `Resources` are owned by this env. The fork task's `finally` block is the sole owner of cleanup — it calls `resources.Dispose()` (child scope) and `cleanup.Dispose()` (which disposes `tsrc` and the parent→child `CancellationTokenRegistration`). The `Cancel` IO only cancels the source; it does not trigger `cleanup.Dispose()`. Awaiting after cancellation returns the error the cancelled computation produced, not `OperationCanceledException` directly — handle via `IO<A>.IfFail`.
- `IO<A>.Post()` allocates an `AutoResetEvent` per call — a kernel-backed synchronization primitive that is `IDisposable`. The implementation disposes it correctly in a `finally` block, but there is no pooling. For marshaling high-frequency operations to a specific context thread, batch work via a channel whose consumer runs on that context and returns aggregated results, calling `Post()` at the boundary only once per batch. `Post`, `Local`, and `LocalCancel` address disjoint concerns: thread-dispatch, resource-scope, and cancellation-scope respectively.
- `EnvIO.New()` with `syncContext = null` captures `SynchronizationContext.Current` at the moment `New()` is called — not at construction of the `IO` tree and not at `Run()` time. A state cell that must always marshal updates to a specific context must inject the context explicitly via `EnvIO.New(syncContext: theContext)`, not rely on ambient capture. `EnvIO.LocalSyncContext` similarly creates a new `EnvIO` with `SynchronizationContext.Current` at the point of the property access; deferring it into a lazy `IO` node means the context is captured at node creation, not at evaluation, which may be on a different thread.
- `IO<A>.RepeatUntil(Func<A,bool> predicate)` — the no-schedule overload — is defined as `Bracket().Bind(v => !predicate(v) ? RepeatUntil(predicate) : IO.pure(v))`. Each iteration adds one `Bind` closure to the managed continuation stack; `Bracket()` produces an async-path `IOLocal` node, so each iteration suspends and resumes. Use only for predicates known to fire within a bounded number of iterations. `IO<A>.RepeatUntil(Schedule schedule, Func<A,bool> predicate)` avoids recursive bind via a local `go(Iterator<Duration> iter, A? value)` that injects `IO.yieldFor(head)` per step — this is the correct form for long-running polling cells.

[ATOM_STATE_CELLS]:
- `Atom<A>` is initialized via `internal static Option<Atom<A>> New(A value, Func<A, bool> validator)`, which returns `None` when the initial value fails the validator. The validator runs on every `Swap` attempt; a failing validator leaves the atom at its current value. `SwapMaybe` returning `None` makes the rejection observable at the call site without throwing.
- `Atom<A>.Swap` is CAS-looping: the transform `f` may execute more than once on contention. Any side effect inside the swap function is repeated — the swap body must be a pure transform. For operations that must fire exactly once, capture the side effect in the return value and execute it after the successful swap outside the loop.
- `Atom<Env, A>` carries metadata `Env` threaded into every `Swap(Func<Env, A, A>)` call alongside the current value. The `Env` is captured at construction and never changes — suitable for session-scoped configuration or stable identity keys that must accompany every update without being stored in the value itself.
- `Prelude.atomIO(A)` and `Prelude.atomIO(A, Func<A,bool>)` return `IO<Atom<A>>` and `IO<Option<Atom<A>>>` respectively, lifting atom creation into the effect pipeline for ordered acquisition in `IO` composition chains.
- `Prelude.swap(Atom<A>, Func<A, A>)` and `Prelude.swapIO(Atom<A>, Func<A, A>)` are top-level alternatives to the instance `Swap`; `swapIO` lifts the mutation into `IO<A>` for pipeline composition without losing the result snapshot.
- `AtomHashMap<K, V>` is the thread-safe mutable concurrent map. `SwapKey(K, Func<V, V>)` returns `unit` immediately when the key is absent — the swap function is never called and the map is unchanged; use `SwapKey(K, Func<Option<V>, Option<V>>)` for conditional-insertion semantics (receives `None` for absent keys). `FindOrAdd(K, Func<V>)` calls `FindOrAddWithLog` inside a CAS spin loop; if two threads both observe the key absent, both invoke `None()`, only one wins the CAS, the loser retries and finds the key present — the factory executes at least once per concurrent first-insert per key. Any factory that acquires resources leaks the handle produced by the losing thread. The safe surface: `TryAdd(key, new Lazy<V>(factory))` then read `.Value` — `Lazy<T>` with `LazyThreadSafetyMode.ExecutionAndPublication` guarantees single execution even when multiple threads race before it completes. `FindOrAdd` always calls the `WithLog` trie variant regardless of whether `Change` is subscribed — the allocation cost is unconditional for `FindOrAdd`-dominant workloads. `SetItem` switches between `SetItemWithLog` and `SetItem` based on the subscription check; for `SetItem`-dominant workloads the no-subscriber path avoids the allocation.
- `AtomHashMap.Swap(Func<TrackingHashMap<K,V>, TrackingHashMap<K,V>>)` checks reference identity of the swap function's result before attempting the CAS — if the swap function returns the SAME underlying trie reference, the CAS is skipped entirely and no `Change` event fires. A swap function that conditionally modifies should return the original `TrackingHashMap` when no change is desired, not a structurally equivalent new one.
- `AtomHashMap<K,V>.Change` fires via `AnnounceChange` after the CAS succeeds, carrying `HashMapPatch<K,V>` with `prev` and `current` trie snapshots. `AnnounceChange` gates on `change != null && change.HasChanged` before invoking; the event does not fire when the new and old tries are equal. For `SetItem`-dominant subscribed workloads, `SetItemWithLog` pre-allocates a `Change<V>` diff record before the `HasChanged` gate — the allocation occurs even for idempotent swaps on subscribed maps. For `FindOrAdd`-dominant workloads, `Change` subscription has no additional allocation impact on that path because `FindOrAddWithLog` is always taken.
- `Atom<M, A>` (the metadata form) has a `Change` event signature that receives only the new `A` value, not the `M` metadata. Handlers that need the metadata must capture it from the construction context.
- `SwapMaybe(curr => curr.IsNone ? Some(acquire()) : None)` for once-only resource acquisition has a double-acquisition hazard: the swap function runs inside a CAS loop, so two threads both seeing `None` will both invoke `acquire()`, with the losing thread's handle leaked after the CAS. The safe form is a two-step protocol — a first `SwapMaybe` that gates without acquiring (transitions from `None` to a sentinel `Some(pending)` value), then acquiring outside the swap, then a second swap to commit — or, if the acquiring call itself is idempotent and the resource is self-cleaning on GC finalization, structured disposal of the loser's handle via the swap function's return value when the function observes a concurrent `IsSome` on retry.
- `AtomHashMap<K,V>.FindOrMaybeAdd(K key, Func<Option<V>> None)` calls the `None` delegate only when the key is absent and inserts only when it returns `Some(v)`. If `None` returns `None`, the map is left unchanged and the returned `Option<V>` is `None`. The factory still executes inside the CAS loop and may run multiple times on contention; it must produce the same insertion decision on repeated calls (though possibly different `Some` values, in which case only the CAS winner's value is stored). `FindOrMaybeAdd(K, Option<V>)` — the value-direct overload — accepts a pre-computed `Option<V>`; inserting a pre-computed `None` is a read-only probe: if the key is absent, nothing is inserted; if present, the stored value is returned. This is the correct surface for a lookup-or-no-op pattern where key absence is a valid terminal state rather than a trigger for insertion.

[STM_REF_MULTI_CELL_COORDINATION]:
- `Ref<T>` is the correct primitive when two or more state cells must transition atomically. `Atom<A>` CAS-loops on a single cell; two independent atoms cannot be CAS-swapped together without a window where one has committed and the other has not. `Ref<T>` inside `STM.DoTransaction` / `sync(...)` provides snapshot isolation across any number of refs with automatic retry on conflict — the unit of atomicity is the transaction body, not a single cell.
- `atomic<R>(Func<R> op, Isolation isolation = Isolation.Snapshot)` and `snapshot<R>(Func<R> op)` are the public transaction entry points; `DoTransaction` overloads are `internal` and not callable from application code. Both `atomic` and `snapshot` nest: when a transaction is already running on the current thread, the inner call participates in the outer transaction rather than starting a new one. `STM.TransactionId` is a `long` property getter that throws `InvalidOperationException("Transaction not running")` when no transaction is active on the current thread — it cannot serve as a guard for conditional transaction entry. The correct guard is tracking transaction context externally, or structuring code so conditional entry is not needed (favor `atomic` nesting which participates safely in an outer transaction).
- `Isolation.Snapshot` triggers a retry only when a ref written inside the transaction was also written externally between the transaction's read-point and its commit. A ref that is read but not written is not tracked for conflict. `Isolation.Serialisable` tracks both reads and writes; any external write to any ref touched causes retry. For boundary lifecycle cells where the correctness invariant is "the set of open connections did not change while I was deciding to add one", `Serialisable` is required; `Snapshot` permits a torn read.
- `commute(Ref<T>, Func<T,T>)` applies `f` to the current in-transaction snapshot and writes the result back into the transaction's local state immediately — the in-transaction read of the same ref after `commute` observes the tentative commuted result, not the original snapshot value. At commit time, `f` is re-applied to the most recently committed external value rather than the in-transaction value. Use `commute` only for operations where last-one-in-wins is acceptable — append-only counters and idempotent flags. For boundary cells whose value determines whether an operation is permitted, `swap(ref, f)` (not `commute`) enforces snapshot-at-read within the body.
- `Ref<T>.Value` setter calls `STM.Write`, which throws `InvalidOperationException("Refs can only be written to from within a \`sync\` transaction")` when no transaction is active. There is no implicit transaction wrapping — out-of-transaction writes fail hard.
- A lifecycle state machine encoded as multiple `Ref<T>` cells — one for state, one for in-flight count, one for drain deadline — inside a single `Serialisable` transaction is the correct shape for drain-with-count correctness. The `Draining` transition sets the state ref and reads the in-flight count atomically; any concurrent operation that increments the in-flight count will conflict with the state-read, triggering a retry. Either the drain commits before any in-flight increment, or the in-flight increment commits and the drain retries with the updated count.
- `STM.DoTransaction<T>(Eff<RT, T>, Isolation)` accepts an `Eff<RT, T>` effect as the transaction body. If the transaction retries, the effect's `IO` subtree is re-evaluated. Any effect with external side effects (logging, metrics, I/O) re-executes on each retry. Transaction bodies must be confined to pure `Ref` reads and writes; side-effectful `IO` must run in a continuation after the transaction commits using the committed values.
- `commute` returns the in-transaction commuted value directly (not a `CommuteRef<T>`) — the return type is `A` after casting the updated in-transaction state. A transaction body that commutes a ref and then reads it back observes the tentative commuted value because `commute` wrote it into the transaction's local trie. Boundary cells that must make permit decisions based on the value they are about to write should use `swap` (not `commute`) so the decision is based on the committed snapshot, not on a tentative value that may be re-applied with a different external base at commit time.

[RESOURCES_AND_USE]:
- `Resources` tracks acquired values in an internal `AtomHashMap<object, TrackedResource>`. `Acquire<A>(A value) where A : IDisposable` registers the handle; `Acquire<A>(A value, Func<A, IO<Unit>> release)` registers with a custom `IO<Unit>` release effect. The custom-release form is correct for handles whose teardown is effectful (network flush, protocol goodbye, event unsubscription) rather than a bare `.Dispose()`.
- `Resources.DisposeU(EnvIO envIO)` creates a fresh internal `EnvIO` with `CancellationToken.None` to run releases. Release effects are therefore never cancellable through the caller's token — they run to completion regardless of outer cancellation state. A teardown effect that itself acquires a temporary resource must use a manually scoped inner `IO.Bracket` to isolate that acquisition, or the temporary resource outlives the teardown call.
- `Resources.DisposeU` iterates `foreach (var resource in resources)` over the `AtomHashMap` backed by a HAMT. HAMT iteration follows hash-tree traversal order, not insertion order — no LIFO or FIFO guarantee on release. Designs that require ordered teardown must encode the sequence explicitly: a `Seq<IO<Unit>>` tracked alongside the map, or nested `Resources` parent chains where child-scope disposal precedes parent-scope disposal by construction.
- `Resources.Release<A>(A value)` walks the parent chain when the key is absent in the current scope: `resources.Find(obj).Match(f, () => parent != null ? parent.Release(value) : unitIO)`. A forked computation runs inside a child `Resources` constructed as `new Resources(parentResources)`. Any `Release` call that misses the child map traverses into the parent scope and releases a parent-scoped resource — correct design for shared-resource release from a fork, but also the mechanism by which a fork can accidentally release a parent resource via key collision on `object` identity.
- `Fork` creates `new Resources(parentResources)` as the forked child scope. The fork's `finally` block calls `resources.Dispose()` on the child only; parent resources remain alive. Any resource the fork explicitly releases via `IO<A>.Bracket()` or `use(...)` that was registered in the parent scope will be released through the parent chain traversal — the parent-scope resource is then gone. Never register a shared parent resource in a child bracket unless the intent is to transfer ownership to the fork.
- `Resources.Merge(Resources rhs)` is `internal` and adds all entries from `rhs.resources` into `this.resources` via `Swap`. It is used by the framework to migrate resources from a completed sub-scope into a parent. Designs that need to manually promote resources from a child scope to a parent must go through the `use`/`bracket` DSL.
- `Resources.Acquire<A>(A value, Func<A, IO<Unit>> release)` stores `(Value, Dispose: Func<A, IO<Unit>>)` in `TrackedResourceWithFree<A>` and calls `Dispose(Value)` when released. The `IO<Unit>` release runs inside the `EnvIO` provided to `DisposeU`, which carries the sync context — correct for resources whose cleanup must run on a specific thread or must be sequenced relative to other effects.
- `use(IO<A> acquire, Action<A> release)` and `use(IO<A> acquire, Func<A, IO<Unit>> release)` build `IOUse<A, B>` DSL nodes that register acquisition into the current `EnvIO.Resources` scope on execution.
- `IO<A>.Bracket(Use, Catch, Fin)` fires `Fin` unconditionally — the standard safe-acquisition shape. `IO<A>.BracketFail()` runs cleanup only on failure, leaving the resource alive on success — correct for correlated resources that must outlive the bracket's scope. `IO<A>.Bracket()` calls `WithEnv(env => env.LocalResources)` producing `IOLocal` (disposes on both success and failure). `IO<A>.BracketFail()` calls `WithEnvFail(env => env.LocalResources)` producing `IOLocalOnFailOnly`: on success the local env's resources are NOT disposed (only `DisposeNonResources()` runs in `finally`), leaving them live for the caller.
- `IO<A>.Uninterruptible()` wraps in `IOUninterruptible<A, B>`. At interpretation, `Invoke` constructs a child `EnvIO` via `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — `CancellationToken.None.CanBeCanceled == false` so no parent→child registration is created. Parent cancellation does not propagate into the uninterruptible body; isolation is structural, not a guard check. The async path (`IOUninterruptibleAsync.Invoke`) is an `async ValueTask<IO<B>>` with no `try/finally` around `await Fa`; if `Fa` faults, the awaiter propagates the exception before `LocalEnv.Dispose()` is called, abandoning the isolated `CancellationTokenSource` (Own bit 1 set) to GC finalization. For high-frequency fault paths, this accumulates live `CancellationTokenSource` objects. Composing `Uninterruptible` inside a `Fork` is safe: the fork's source is cancelled by parent cancellation via its own registration, but the `Uninterruptible` inside the fork constructs from `CancellationToken.None` so the fork's cancellation also does not propagate; the fork's cleanup disposes its own source and parent registration cleanly.
- `IOLocal<X,A>.Invoke` uses `using EnvIO local = MapEnvIO(envIO)` with no explicit `try/catch`. When `Operation.RunAsync(local)` throws, the compiler-generated `finally` calls `local.Dispose()` — both resources and non-resources are released before the exception propagates. `IOLocalOnFailOnly<X,A>.Invoke` manually separates: `catch { local.DisposeResources(); throw; }` followed by `finally { local.DisposeNonResources(); }`. On success, neither runs in the local scope's body — the result is returned without disposal because `LocalResources` shares the parent `Source`, transferring resource ownership to the caller.
- `Resources.ReleaseAll()` runs via an `IO.lift` lambda that calls `resources.Swap(r => { foreach (item in r) item.Value.Release().Run(envIO); return empty; })`. The swap function executes inside `AtomHashMap.Swap`'s CAS spin loop; if the CAS fails on concurrent mutation, the entire sweep — including already-executed `Release().Run(envIO)` calls from the previous snapshot — re-executes on the retried snapshot. Release effects must be idempotent, or `ReleaseAll` can double-release on CAS contention. The safe form for non-idempotent teardown is `Release<A>(value)` per key, which removes the key before calling the release effect, preventing re-entry on retry.

[TOKEN_GATED_INITIALIZATION]:
- The canonical gate for once-only native initialization is `Interlocked.CompareExchange(ref int guard, 1, 0) == 0`. The CAS returns the comparand value (`0`) exactly for the winner; all other callers see the updated value (`1`) and must wait or fail. The `Atom<Option<Guid>>` form maps to the same CAS: `SwapMaybe(curr => curr.IsNone ? Some(id) : None)` succeeds once; `None` return on the already-open branch leaves the atom unchanged and signals the caller that it lost the race.
- The gate validator must be `_ => true`; the transition guard belongs in the swap function body, not in the validator. A validator that rejects `Some(handle)` after the first open freezes the atom permanently at `Some(handle)` because no subsequent swap value can pass, preventing any future state transition including the release path. The two-step gate protocol (gate-CAS first with a `Pending` sentinel, then acquire outside the CAS loop, then commit-CAS) eliminates the double-acquisition hazard inherent in acquire-inside-swap while still using `SwapMaybe` as the concurrency boundary.
- Re-entrant initialization — where the initializer itself calls back into the guarding layer — deadlocks on any single-permit gate (`SemaphoreSlim(1,1)`, `Lock.EnterScope`, or an `Atom` with a blocking reader). Non-re-entrant native init must hold the gate cell through the full side-effect sequence; completion must be signaled by a second cell transition, not by releasing the gate early.
- `System.Threading.Lock.EnterScope()` returns a ref-struct `Lock.Scope` whose `Dispose` releases the lock; `using var _ = theLock.EnterScope()` replaces `lock(obj)`. `Lock` is re-entrant from the same thread (`IsHeldByCurrentThread`) — the same thread can enter multiple times without deadlock and must exit the same number of times. For boundary cells that guard process-static initialization whose callers may themselves be initializers of dependent cells, `Lock` silently permits the recursion while `SemaphoreSlim(1,1)` hard-deadlocks on the second entry from the same thread. `SemaphoreSlim(1,1)` is the right primitive when initialization must not recurse and the desired behavior on recursion is to surface the failure — the caller will deadlock, which is observable and diagnosable, rather than silently reinitializing.
- `Lock.Scope` is a `ref struct` and cannot be boxed, stored in a field, or captured in a lambda — it is mechanically scoped to the method frame. The compiler rejects `Lock.Scope` across an `await` boundary at compile time; the `ref struct` constraint makes the cross-await failure a build error, not a runtime hazard. `Lock.Scope.Dispose()` calls `lockObj.Exit(_currentThreadId)` with the thread id captured at `EnterScope` time; a call from a different thread id would throw `SynchronizationLockException`, but this path is unreachable in correctly compiled code because the `ref struct` constraint prevents the scope from surviving an await. For boundary cells that need mutual exclusion across async continuations, use `SemaphoreSlim.WaitAsync` and `Release`; for passing "locked" context to nested synchronous methods, `Lock.IsHeldByCurrentThread` is the assertion surface.
- `LazyInitializer.EnsureInitialized<T>(ref T target, ref bool initialized, ref object syncLock, Func<T> factory)` — the three-field form — is the pattern for `[ThreadStatic]`-backed per-thread caches that require a sync lock for the first initialization per thread. The single-field form `EnsureInitialized<T>(ref T target, Func<T> factory)` uses a CAS and is appropriate for process-static initialization with a reference-type target.
- Native library identity for conflict detection requires three fields at minimum: absolute path, file size (`FileInfo.Length`), and last-write timestamp (`FileInfo.LastWriteTimeUtc`). Path alone is insufficient because file replacement at the same path changes the library without changing the path. `NativeLibrary.Load(string libraryPath)` returns an `IntPtr` handle; `NativeLibrary.TryLoad(string, Assembly, DllImportSearchPath?, out IntPtr)` populates the handle and returns `false` on failure without throwing. The handle must be stored in the gate cell alongside the file fingerprint. `NativeLibrary.Free(handle)` is the symmetric release; releasing a handle still referenced by exported function pointers produces an access violation on next call. The gate cell must hold the handle until the process exits or explicit teardown.
- Two process-static native library cells wrapping the same underlying library must share a fingerprint check to detect version conflicts at the seam. An `AtomHashMap<string, (IntPtr handle, long size, DateTime modified)>` keyed by canonical name with a `SwapKey` that validates fingerprint consistency on each `TryAdd` surfaces conflicts as a typed failure at the second load attempt rather than silent undefined behavior from loading two incompatible versions.
- `Interlocked.CompareExchange(ref byte location, byte value, byte comparand)` and the `sbyte`, `short`, `ushort` overloads allow lifecycle state machines with ≤255 states to encode state in a `byte` field and CAS-transition without widening to `int`. Four `byte` state fields fit in one cache word versus four `int` fields spanning multiple words. The state values must fit in the unsigned range of the chosen type; a state machine that uses negative sentinel values must use `int` or `sbyte` with care.
- `Interlocked.Or(ref int location, int value)` and the `uint`, `long`, `ulong` overloads atomically set bits in a flag word and return the original value. For a lifecycle state cell that tracks multiple concurrent properties as a bitfield, a single `Interlocked.Or` or `And` can atomically set or clear a flag without a CAS loop. Reading the current flags requires `Volatile.Read` or `Interlocked.CompareExchange(ref state, state, state)` to get a fresh value with acquire semantics; a plain field read is not coherent on weakly ordered hardware.

[LAZY_INITIALIZATION_CELLS]:
- `Lazy<T>` with `LazyThreadSafetyMode.ExecutionAndPublication` caches the first exception thrown by the factory permanently — every subsequent `.Value` access rethrows the cached exception without retrying the factory. This makes it unsuitable as an initialization cell for anything with transient failure modes (network, file system, native library load). A boundary cell whose initialization can fail transiently must use `LazyThreadSafetyMode.PublicationOnly`, which does not cache exceptions and retries on each subsequent `.Value` access until one succeeds, or must use `Atom<Option<T>>` with explicit retry policy via `IO` schedule.
- `LazyThreadSafetyMode.PublicationOnly` allows multiple threads to run the factory concurrently; the first to complete sets the value, and all others discard their results. For factories that acquire a native resource, concurrent initialization means competing factories all acquire the resource but only one is retained — the others are silently discarded without disposal. `PublicationOnly` is only safe for factories that do not acquire resources or for resources that are self-cleaning on GC finalization. The safe concurrent-initialization cell for resource-acquiring factories is `Lazy<T>(factory, LazyThreadSafetyMode.ExecutionAndPublication)` with explicit acknowledgment that transient failures are permanent.
- Recursive access to `Lazy<T>.Value` inside the factory throws `InvalidOperationException` under `ExecutionAndPublication` (the lock is not re-entrant). Under `PublicationOnly`, recursive access does not throw — the recursion sees the uninitialized state and re-enters the factory. A boundary cell that chains lazy initialization cells must prevent cross-cell circular dependencies or choose `PublicationOnly` explicitly and accept the double-initialization risk.

[RUNTIME_SPINE]:
- One process-static runtime spine record is the ownership unit for `(CancellationTokenSource, TimeProvider, channel ports, configuration)`. A re-boot creates a new record; the old record must reach fully drained and disposed state before the new one is constructed. Concurrent second spines sharing any mutable cell (a static `Atom`, a process-static `Resources`, or a shared `Channel`) produce undefined merge semantics.
- `TimeProvider` is the clock surface injected into the spine record. `TimeProvider.System` is the production singleton; test overrides implement `TimeProvider` directly. `TimeProvider.CreateTimer(TimerCallback, object, TimeSpan, TimeSpan)` returns `ITimer`; `ITimer.Change(TimeSpan, TimeSpan)` reschedules without allocation. The runtime clock must enter state cells as `TimeProvider`, never as `DateTime.UtcNow` or `Stopwatch` calls, preserving deterministic replay in tests. When a boundary cell's polling interval must be adaptive, the correct cell is `(ITimer timer, Atom<TimeSpan> currentPeriod)` with the atom driving `timer.Change` calls from the `Atom.Change` handler. `ITimer.Change(TimeSpan.Zero, Timeout.InfiniteTimeSpan)` fires the callback immediately and disables periodic rescheduling — correct for a boundary cell that must immediately trigger a drain-start callback on an explicit `Shutdown()` call when the cell uses a timer as its drain-trigger rather than a `CancellationToken`.
- When the same `FakeTimeProvider` instance is injected into the runtime spine record and into every timeout-scoped `CancellationTokenSource` created from the spine, advancing the fake clock in a test synchronously fires all outstanding timeouts. This gives deterministic coverage of timeout-triggered state transitions — `Draining` on grace period expiry, `InitializationFailed` on setup timeout, `Closed` on idle timeout — without requiring real-time waits in tests. The `TimeProvider` must be part of the spine record, not hidden inside closure captures, for this propagation to be complete.
- `Channel<T>` is the seam marker for the drain phase of a runtime spine. `ChannelWriter<T>.Complete()` fences new writes (subsequent `TryWrite` returns `false`); in-flight items remain readable until `ChannelReader<T>.Completion` task completes. `ChannelWriter<T>.TryComplete(Exception? error)` returns `false` when the channel is already completed (idempotent on the `false` path); `Complete(Exception?)` throws `ChannelClosedException` when already completed. For drain-phase boundary cells that may receive concurrent `Shutdown()` calls, `TryComplete` is the safe form. When `error` is non-null, `ChannelReader<T>.Completion` task faults with that exception after drain; callers awaiting `Completion` receive the fault, not a clean `RanToCompletion`. `ChannelReader<T>.WaitToReadAsync(CancellationToken)` returns `false` after the channel is drained — this is the structural drain-wait predicate. A spine teardown that calls `TryComplete()` then `await Completion` guarantees all work was processed before the rest of the spine is released.
- `ChannelReader<T>.ReadAllAsync(CancellationToken)` produces an `IAsyncEnumerable<T>` that completes when the channel is drained and completed. Cancellation of the token cancels the enumeration but does not complete the channel — the channel writer must be separately completed. A drain loop using `ReadAllAsync` that is cancelled mid-drain without the channel being completed will restart enumeration on the next consumer call, rediscovering unread items. The correct drain-then-stop pattern: cancel the write-side via `TryComplete()`, then `await foreach` with `CancellationToken.None` until the enumerable completes.
- `Channel.CreateUnboundedPrioritized<T>(UnboundedPrioritizedChannelOptions<T>)` creates a channel backed by a heap whose dequeue order is determined by `options.Comparer`. Items are dequeued in ascending order — smallest-first per the comparer. For a drain-phase boundary cell, an `IComparer<WorkItem>` that sorts by priority class then enqueue timestamp provides structured drain: high-priority in-flight items drain before low-priority ones, avoiding starvation that a FIFO unbounded channel cannot prevent. `ChannelReader<T>.TryPeek(out T item)` is implemented on both bounded and unbounded non-prioritized channels — `UnboundedChannel<T>` and `SingleConsumerUnboundedChannel<T>` both override `CanPeek => true` and provide working `TryPeek`. The base `ChannelReader<T>` defaults `CanPeek => false`; always gate `TryPeek` calls on `CanPeek` to guard against future channel implementations that do not override it. For back-pressure surfacing as a typed rejection up the `Fin<A>` rail, `BoundedChannelFullMode.Wait` is the correct mode — `TryWrite` returns `false` when the channel is full and the caller must treat the rejection as a fault. `DropWrite`, `DropNewest`, and `DropOldest` all cause `TryWrite` to return `true` even when the item is discarded (for `DropWrite`, `_itemDropped` is invoked but `true` is returned); all three modes silently lose work from the caller's perspective and are inappropriate when every write must be acknowledged or rejected through the Fin rail.
- Mid-flight semantics during drain require explicit policy: items already dequeued but not yet committed are either completed with their result or explicitly faulted. The `Channel` drain model does not provide this; the cell owning mid-flight correlation (a `ConcurrentDictionary<CorrelationId, TaskCompletionSource<T>>`) must be separately swept on drain completion, completing outstanding sources with an `OperationCanceledException` or a domain-typed fault.

[MEMO_AND_CACHE_CELLS]:
- `Memo<A>` uses three states encoded in an `internal int state` field: `0` (unloaded), `1` (loading), `2` (loaded). The winner of `CAS(state: 0→1)` runs `acquire()`, writes `value = result`, then writes `state = 2`. A failed `acquire()` reverts to `state = 0` in the catch block; the exception is rethrown, not absorbed. These two post-CAS writes are non-atomic; concurrent callers in state `1` spin via `SpinWait.SpinOnce()`.
- `Memo<A>.state` carries no `volatile` annotation. The `Interlocked.CompareExchange` calls in `GetValue` and `Reset` carry implicit full memory barriers, so state machine transitions (0→1→2 and 2→1→0) are coherent between CAS participants. The gap is the hot-path `if (state != 2)` guard in the `Value` property getter — a plain `int` read. On ARM64, a thread can observe `state == 2` before the write to `value` on the winning CAS thread becomes visible, reading a stale default. The safe form is `Volatile.Read(ref state) != 2` in the getter, or replacement with `Atom<Option<A>>` which carries the full volatile guarantee on both state and payload simultaneously.
- `Memo<A>.Reset()` CASes `state: 2→1`, writes `value = default(A)`, then writes `state = 0`. Both reset writes are sequential after the CAS with no further atomic protection. Two concurrent `Reset()` calls are safe — the second sees `CAS(state, 1, 2)` fail while state is `1` and spins until the first writes `state = 0`. However, `Reset()` is not safe for concurrent invocation alongside `GetValue()` in the general case; `Clone()` combined with replacement via an outer `Atom<Memo<A>>` is safer than calling `Reset()` under load.
- `Memo<F, A>` (the higher-kinded form) uses the same three-state protocol on `K<F, A>?`. `Clone()` creates a new `Memo` from the same `acquire` delegate without cached state — enabling controlled re-acquisition without the concurrency hazard of `Reset()`.
- `MemoExtensions.Memo<T>(this Func<T> func)` wraps any function into a lazy-computed memoization cell. The multi-key form `memo<A,B>(this Func<A,B> func)` uses a `WeakDict<A,B>` backed by `ConcurrentDictionary<A, WeakReference<...>>` with a per-key synchronization object — `syncMap.GetOrAdd(key, new object())` obtains a per-key lock object, and `lock(value2) { cache.GetOrAdd(key, func) }` serializes factory execution per key. The factory executes exactly once per key per concurrent group. `memoUnsafe<T,R>` uses the same per-key lock pattern with a `ConcurrentDictionary<T,R>` value cache but without weak-reference eviction. Neither surface has the raw `GetOrAdd` factory-race hazard; the per-key lock is the guarantee mechanism.
- Cache key completeness: a memo key must encode every dimension that affects the output — for content-dependent results this means content hash plus full configuration hash. A key missing any input dimension produces stale hits. Native-library cache keys follow the same law: path alone does not identify library identity because file replacement at the same path is transparent to path-keyed caches.
- `ConcurrentDictionary<K,V>.AddOrUpdate(K, Func<K, TArg, V>, Func<K, V, TArg, V>, TArg)` — the `TArg`-carrying factory overload — avoids closure allocation for registration patterns that thread a single argument. Both factory delegates may be called multiple times on contention and must be pure. For factories that acquire resources, wrap with a `Lazy<V>` value using `LazyThreadSafetyMode.ExecutionAndPublication` to guarantee single execution.
- Generated registry keys must be stable across regeneration cycles. A key derived from a `nameof` expression, a source-generated type identity, or a versioned discriminant survives recompilation; a key derived from `RuntimeHelpers.GetHashCode`, unordered iteration index, or assembly load order does not. Stale-key entries in a persistent registry accumulate as ghost entries when the type graph changes; `AtomHashMap<K,V>.Filter(pred)` is the correct sweep operation, and the sweep must execute during the boundary layer's drain phase — before `Closed`, after all in-flight operations referencing the old keys complete. A filter sweep during `Open` state races with concurrent `FindOrAdd` calls; the filter predicate must check entry validity rather than key existence to be safe under concurrent use.
- `AtomHashMap.FindOrAdd(K key, Func<V> None)` is the correct once-only insertion primitive for singleton registries; the factory races under CAS and must be idempotent. For resource-acquiring factories, the `TryAdd + Lazy` pattern described in the state-cells group applies identically here.

[LIFECYCLE_STATE_MACHINES]:
- A boundary layer's lifecycle is a state machine with at least: `Created`, `Opening`, `Open`, `Draining`, `Closed`, and `Faulted`. `Draining` fences new work (equivalent to `ChannelWriter.Complete()`) but keeps mid-flight work alive. A cell in `Draining` must reject new acquisitions with a typed rejection carrying boundary identity, correlation id of the rejected request, and timestamp. `Faulted` is escapable if the fault is transient; the escape transition creates a new `Open` instance, not a reset of the faulted one.
- A lifecycle state cell encoded as `Atom<LifecycleState>` with `validator = _ => true` and transition constraints in the swap function — `curr => curr.IsValidPredecessor(next) ? next : curr` — correctly separates domain invariants from relational transition constraints. The validator is the wrong surface for transition constraints because it receives only the candidate next state; the current state must be read inside the CAS loop. `SwapMaybe` returning `None` to signal rejection is the correct shape.
- Non-re-entrant process-static initialization where the initializer has side effects must hold state in a cell that distinguishes `Initializing` from `Initialized` from `InitializationFailed`. A cell that collapses these into a boolean admits the window where a second caller observes `Initialized = true` while the initializer's side effects are still running. The three-state distinction enforces that `Initializing` blocks all callers until the state transitions to `Initialized` or `InitializationFailed`. Callers that see `Initializing` must wait — the correct awaitable seam is a `TaskCompletionSource<InitState>` created before the first `Swap` and placed process-statically alongside the atom. The winning initializer calls `tcs.TrySetResult(newState)` immediately after committing the `Initialized` swap. Waiters read `atom.Value` first; on `Initializing`, they `await tcs.Task`. If the winner commits before the waiter checks the atom, the waiter reads `Initialized` directly; if after, the waiter blocks on `tcs.Task`. Both the atom and the `TaskCompletionSource` must be initialized before any concurrent caller can reach the gate — the `TaskCompletionSource` must not be lazily allocated on first contention.
- Fault re-entry: when a `Faulted` boundary cell is replaced by a new one, correlation cells tracking in-progress requests must explicitly decide replace-versus-reuse. Replace produces a fresh correlation with no memory of prior state; reuse preserves request identity but inherits any partial server-side work. The decision must be explicit policy in the cell, not an implicit consequence of allocation order. The correlation `ConcurrentDictionary` must be fully swept before the new open instance begins accepting work; sweep must fault all outstanding sources via `TrySetException(new BoundaryFaultException(...))` rather than `TrySetCanceled()`, because callers may distinguish cancelled (retriable) from faulted (non-retriable). Remove each entry before faulting it to prevent double-fault from concurrent drain-plus-replace.
- State transition rejection must carry typed evidence: the rejecting state (e.g., `Draining`), the correlation id of the caller's operation, and a boundary-layer timestamp from the injected `TimeProvider`. A rejection typed as `Errors.Cancelled` loses the boundary context; a domain-typed `BoundaryDrainingFault` carrying all three fields composes cleanly into `Fin<A>` failure rails.
- `ObjectDisposedException.ThrowIf(bool condition, object instance)` is the idiomatic guard for post-`Close` access. `ObjectDisposedException.ThrowIf(bool, Type)` accepts a `Type` directly — use this inside static methods or factories where `this` is unavailable: `ObjectDisposedException.ThrowIf(_closed, typeof(BoundaryCell))`. The `typeof(T)` form is a compile-time constant and avoids the virtual dispatch on `GetType()` that the `object`-accepting overload implies.
