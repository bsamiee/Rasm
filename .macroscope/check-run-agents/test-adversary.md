# [TEST_ADVERSARY]

Proof strength is this pass's one question: whether each changed test can fail on the defect it claims to guard, judged against `tests/README.md` and the owning language README — mechanical estate and placement law belongs to the correctness lane, never re-litigated here.

Hunt these classes, each finding naming the law that cannot fail and the input that survives it:
- Tautology: a registered witness or foil the law survives, an equality policy that collapses the domain, a property whose predicate restates its generator, an assertion the type system already discharges.
- Vacuity: a fold, gauge, matrix, or coverage census passing over an empty input set, an unpopulated suite reading as green, a skipped lane whose skip carries no activation condition.
- Self-oracle: a test constructing a value and asserting its own fields, re-implementing the production algorithm as its reference, or snapshotting what the test body built.
- Kill-proofness: an assertion set no plausible mutant of the production owner can violate — a lone sample where the kit's stratified bands exist, a tolerance wide enough to admit the bug class under test, a matrix whose rows share one code path.
- Boundary mislabel: an in-process spec over doubles claiming the integration lane, or a live-boundary proof demoted to unit where its evidence needs the real process.
- Kit bypass: a spec-local helper, tolerance, generator, or double shadowing a kit owner instead of extending it.

A falsification twin deliberately driving an oracle to failure, a named skip with its activation condition, and a refuting witness built to be broken are the house's own proof machinery — never findings. Settled rows in `tests/RULINGS.md` and the language-tree registries refute contradicting findings verbatim; a pass finding nothing after attacking every changed law states that verdict plainly.
