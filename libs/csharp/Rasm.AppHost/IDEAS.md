## [1]-[OPEN]

[AGENT_TOOL_FEDERATION]: the runtime spine consumes external MCP servers, not only serves its own — the `agent/` sub-domain's missing client leg.
- The `agent/` sub-domain is single-page (`mcp-projection.md`) and one-directional: it projects the capability registry onto the MCP **server** surface but never consumes a peer MCP server, leaving the full catalogued client surface (`McpClient.CreateAsync(IClientTransport, …)`, `ListToolsAsync`/`CallToolAsync`, `McpClientTool : AIFunction`, the `StdioClientTransport`/`HttpClientTransport`/`StreamableHttpClientSessionTransport` transports) entirely unexploited. A new `agent/tool-federation.md` page folds each attached external MCP server's `McpClientTool` set into the one `CapabilityRegistry` as brokered `CapabilityDescriptor` rows under a `federated.{server}.{tool}` surface key, so an external tool is metered, cost-previewed, grant-scoped, and determinism-logged exactly as a native op — the broker gates a federated call, the command algebra wraps it in commit-or-rollback, and the event log chains its content hash.
- Unlocks the host as an agent **client** — a remote filesystem MCP server, a database MCP server, or another Rasm node's MCP surface becomes a first-class federated capability the same descriptor source drives, so a new external tool source is one `IClientTransport` admission plus a projection fold, never a parallel client or an unbrokered side channel. The federation inverts the existing projection: descriptors flow inward (peer tool → descriptor) as cleanly as they flow outward (descriptor → tool).
- Draws on the official SDK's symmetric `McpClient`/`McpClientTool : AIFunction` client surface (already catalogued in `.api/api-mcp.md` but never composed) and the one-broker-one-registry law: a federated tool that bypassed the grant broker or minted a second tool catalog is the named drift defect. The cross-language leg is the `libs/.planning` `ONE_AGENT_TOOL_PROJECTION` seam consumed from the consumer side.

[AGENTIC_REASONING_RUNTIME]: the in-process `IChatClient`/`IEmbeddingGenerator` agent loop the registry already implies but no page owns — function-calling, semantic discovery, and modal input over the one descriptor source.
- The `mcp-projection.md` page names in-process `IChatClient` function-calling as a consumer of the descriptor projection but no page owns the agent **reasoning** loop, and the catalogued `Microsoft.Extensions.AI` surface (`IChatClient.GetStreamingResponseAsync`, `ChatOptions.Tools`/function-invocation, `IEmbeddingGenerator<TInput,TEmbedding>.GenerateAsync`, the `IImageGenerator`/`ISpeechToTextClient` modal clients) is reduced to a single `IChatClient` mention. A new `agent/reasoning-runtime.md` owns the host-side agent loop: each `CapabilityDescriptor` projects to an `AIFunction` the chat client invokes through the same brokered `CommandAIFunction` the MCP projection adopts (one tool-adoption seam, two front doors), an embedding-indexed `SemanticDiscovery` fold ranks descriptors by an `IEmbeddingGenerator` similarity over the op surface so an agent resolves intent to a capability by meaning rather than exact id, and the function-invocation transcript chains into the determinism event log so an agent reasoning trace replays bit-identically.
- Unlocks a self-hosted agentic surface — the host runs its own tool-calling loop over its own capabilities with no external orchestrator, the semantic-discovery fold turns the flat `DiscoveryQuery.ById` lookup into an intent-resolved ranking, and a reasoning session is a replayable, grant-metered, cost-bounded unit. A new modality (speech-to-command, image-to-intent) is one modal-client row reading the same descriptor catalog.
- Draws on the `Microsoft.Extensions.AI` function-invocation and embedding surface (catalogued, single-member-exploited) and the determinism-replay law: an agent reasoning trace is a command sequence the macro engine records and the event log chains, so an agent run is auditable and reproducible, never an opaque LLM call.

[INDUSTRIAL_DEEPENING]: the live-wire transport axis carries eight rows but only two protocol clients are bound — the six poll transports are `IO.fail` stubs, the Modbus/serial clients are a phantom package admission, and the OPC-UA PubSub-over-MQTT fan-in leg is unbuilt.
- The `live-wire.md` `TransportRows.Read`/`Write` dispatch routes Modbus/serial/REST/GraphQL/spreadsheet/ERP-PLM to `PollRead`/`PollWrite` bodies that return `WireFault.ReadFailed`/`WriteRejected` literals — six of eight transports are unimplemented stubs; the Modbus TCP/RTU register read/write and the serial fieldbus line protocol have no owner because the central manifest carries only `OPCFoundation.NetStandard.Opc.Ua(.PubSub)` and `MQTTnet`, never a Modbus or serial-port client; and the OPC-UA PubSub-over-MQTT leg (`OPCFoundation.NetStandard.Opc.Ua.PubSub` `UaPubSubApplication`) is named-but-unbound. The deepening admits `FluentModbus` (the maintained managed Modbus TCP/RTU/ASCII client over the abandoned `NModbus` — `ModbusTcpClient`/`ModbusRtuClient` `ReadHoldingRegisters<T>`/`WriteMultipleRegisters` over a `Span<T>` register window, `ModbusEndianness` byte order) and `System.IO.Ports` (`SerialPort` `Open`/`ReadLine`/`WriteLine`/`DataReceived` for the RS-232/485 line protocol) as the two named owners the `modbus`/`serial` rows bind, builds each remaining poll transport's real binding on its `OutboundHop` row (REST/GraphQL over `HttpApi` with the GraphQL query body, spreadsheet over a read-only range fetch, ERP-PLM over the vendor REST contract), and lands the PubSub `UaPubSubApplication.Create`/`Start`/`Stop` dataset-subscriber leg as the high-throughput OPC-UA path beside the per-node monitored-item subscription. The register-window decode and the line-frame parse each project one `ExternalValue` exactly as the OPC-UA `DataValue` and MQTT payload do, riding the row's `OutboundHop`.
- Unlocks the studio's actual industrial breadth — a real Modbus PLC's holding-register block, a serial sensor's NMEA/ASCII line, a REST/GraphQL ERP feed, and a PubSub dataset stream all bind through the one `TransportRow` contract, so the eight-row axis is eight working transports rather than two bound rows, six stubs, and a phantom admission; a new register map or line grammar is policy data on the binding spec, never a parallel poller. The PubSub leg adds a fan-in dataset path the per-item subscription cannot scale to.
- Draws on the maintained managed Modbus stack (`FluentModbus`), the BCL `System.IO.Ports` serial owner, the certified OPC-UA PubSub application (`OPCFoundation.NetStandard.Opc.Ua.PubSub`, admitted, the `UaPubSubApplication` lifecycle catalogued but unbound), the `OutboundHop` resilience the poll transports compose, and the one-transport-row-per-edge law: a new fieldbus is one bound row, never a parallel poller.

[RESILIENT_REPLAY_FRONTIER]: the determinism kernel pins reproduction but never models adversarial divergence — chaos-fault injection, divergence bisection, and counterfactual replay are the missing depth.
- The `determinism-and-replay.md` kernel proves a recorded run reproduces and names the first divergent step, but the runtime spine already admits `Simmy` chaos through the outbound Polly pipeline yet the determinism kernel never composes it — there is no recorded-fault-injection replay, no divergence bisection over a long log (the `ReplayVerify` walk is linear), and no counterfactual replay (re-run a recorded log with one command's arguments perturbed to observe downstream divergence). A new `[CHAOS_REPLAY]`/`[DIVERGENCE_BISECT]`/`[COUNTERFACTUAL]` cluster set on the page records the Simmy chaos decisions into the event log as deterministic fault-injection entries so a fault campaign replays identically, bisects a divergence over the content-hash chain in log time rather than linear scan, and replays a recorded log with a bound argument override to surface the downstream cone a change would alter — composing the existing `RecomputeGraph.Invalidate` walk over the perturbed input.
- Unlocks adversarial reproducibility — a recorded chaos campaign re-runs deterministically (the injected faults are log entries, not live randomness), a non-determinism source in a thousand-step log is found by bisection in log-time, and a "what if this command's input changed" question replays against the recorded log without re-recording. The counterfactual replay is the recompute graph applied to history rather than to a live edit.
- Draws on the recorded-fault-injection technique (chaos decisions as deterministic log entries), binary-search divergence bisection over a tamper-evident hash chain, and the existing `RecomputeGraph` content-address invalidation reused over a perturbed historical input — every new cluster composes the kernel's own owners, never a second determinism surface.
