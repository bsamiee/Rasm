# [DOCS_AGENTS]

Scope: `docs/**`. Root instructions own repository policy; `docs/standards` owns documentation form. This file adds the docs-corpus routing overlay.

## [1]-[READ_BEHAVIOR]

- When changing documentation form, document type, headings, tables, examples, proof wording, or prose density, read `standards/README.md` and the specific standard it routes to.
- When changing cross-stack owner precedence, composition, proof order, platform-spine behavior, or package-backed implementation flow, read `usage/` and extend the owner ladder or composition route.
- When changing language, package, platform, testing, or first-class capability guidance, read the owning `stacks/<language>/` folder and extend the concept page or platform page that owns the coding decision.
- When changing host SDK semantics, read `hosts/` where present and route current member claims to host proof.
- When changing an `AGENTS.md`, read `standards/agents-md.md` and keep this overlay delta-only.

## [2]-[OWNER_CONTRACT]

Docs carry implicit agent guidance, current owner routes, and future-facing target architecture. Source, manifests, generated contracts, local tool output, and maintained provider documentation prove present-tense claims; standards choose the document form and do not lower the target.

External packages are first-class implementation material when a source route admits them. Package graph state belongs to stack/platform owners; package-backed coding decisions belong to concept owners; project-local architecture and roadmaps own exact package adoption sequence.

Promote source material into active docs only as durable owner routes, target architecture, source-backed claims, proof gaps, and rejection replacements. Strip report process, prompt framing, source-history narration, broad provenance, and sub-agent orchestration before promotion.

## [3]-[ROUTE_AWAY]

Route README orientation to the nearest README, document form to `standards/`, cross-stack precedence to `usage/`, language/platform policy to `stacks/`, host SDK truth to `hosts/`, package graph state to stack platform pages, and implementation sequence to the local roadmap.

Reusable standards, templates, and examples stay project-agnostic. Project-specific docs may name real paths, packages, manifests, and source facts only when the page exists to describe that exact repository surface.

Stack pages own durable language and package-backed guidance for their language folders. Future reusable artifacts extract from `docs/` after project guidance is made portable.

## [4]-[REJECTIONS]

- No package/API catalogs, command inventories, support matrices, roadmap sequences, implementation maps, or generated contract bodies in this overlay.
- No duplicate standards prose, root policy, C# stack policy, command ladders, validation ceremonies, or provider manuals.
- No wording that treats missing consumers as a reason to weaken an accepted target capability.
- No source/provenance sections, prompt-source narration, source-link collections, or report-process carryover in active docs; route claim evidence to `standards/proof.md` and package decisions to stack owners.
- No route-only docs when a reader decision, owner boundary, proof route, or rejection would make the page useful.
