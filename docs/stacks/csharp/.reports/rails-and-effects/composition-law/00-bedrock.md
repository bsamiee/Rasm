# Composition Law

[BOUNDARY_CONVERSION]:
- `Optional<A>(nullable)` and `Optional<A>(Nullable<A>)` are the canonical null-to-`Option<T>` projections; both are on `Prelude`. `Option<A>` carries an implicit operator from `A?` so direct assignment projects null to `None` with no call for non-struct types, but the `Optional` overload is load-bearing for value types where `A?` means `Nullable<A>`, not `A`.
- `Option<A>.ToFin()` calls `ToFin(Errors.None)` — absence maps to `Errors.None`, not a meaningful error. Supply a domain `Error` via `ToFin(Error fail)` at the boundary to preserve diagnostic intent rather than post-hoc at the consuming edge.
- `Validation<Error, A>.ToFin()` is a direct case-projection: `Succ(value)` becomes `Fin.Succ(value)`, `Fail(e)` becomes `Fin.Fail(e)`. The `Error` produced by `Validation`'s applicative accumulation is a `ManyErrors` whose `Combine` walks the error list; `ToFin` transfers the entire accumulated `Error` as a single `Fin.Fail` — no partial extraction, no further enumeration at the call site. Once inside `Fin`, there is no accumulation: subsequent `Bind` steps short-circuit, and re-entering `ToValidation()` mid-pipeline cannot recover failures already short-circuited by `Fin.Bind`.
- `Either<Error, A>.ToFin()` is on `FinExtensions`: `Right` maps to `Fin.Succ`, `Left` maps to `Fin.Fail`. The conversion is one-hop with no allocation beyond the `Fin` record case.
- `Try<A>` is `record(Func<Fin<A>> runTry)`. Its `Run()` extension executes the delegate inside a catch block, routing exceptions through `Error.New(Exception)`: `TaskCanceledException` and `OperationCanceledException` produce `Errors.Cancelled`; `TimeoutException` produces `Errors.TimedOut`; `AggregateException` produces `ManyErrors.FromAggregate`; `ErrorException` round-trips through `.ToError()`; all other exceptions produce `Exceptional`. `Try<A>` is the canonical synchronous exception-capture boundary. `liftEff<A>(Func<Fin<A>> f)` lifts the same `Func<Fin<A>>` into `Eff<A>` via `IO.lift`; exception capture for the `Eff` form occurs at `RunSafe()` through the `IO` layer's own `Try` wrapping.
- `Error.New(Exception)` performs the same exception routing: `WrappedErrorExceptionalException` and `WrappedErrorExpectedException` round-trip their inner `Error`, `ErrorException` restores via `.ToError()`, cancellation and timeout map to the static `Errors` sentinels. Applying this at the boundary guarantees re-captured exceptions never lose their original `Error` payload.
- `Eff<A>` carries implicit operators from `Fin<A>`, `IO<A>`, `Either<Error,A>`, `Pure<A>`, `Fail<Error>`, and `Lift<Fin<A>>` — all declared on `Eff<A>` itself, not on the source types. `Fin` implements `Natural<Fin, Eff>`: `Transform` wraps `Succ` in `Eff<A>.Pure` and `Fail` in `Eff<A>.Fail`, so `Natural.transform<Fin, Eff, A>(fin)` is the explicit carrier-promotion path when the implicit operator is ambiguous.

[TERMINAL_COLLAPSE]:
- `Match<B>(Func<A,B> Succ, Func<Error,B> Fail)` on `Fin<A>` is the only bi-directional terminal; it exits the rail entirely. Rail-preserving alternatives are `Map`, `Bind`, `MapFail`, and `BiMap` — these keep the failure rail intact and never produce a raw value. Using `Match` mid-pipeline discards the rail and forces reconstruction; the mid-pipeline `Match` anti-pattern is a lane constraint encoded in the repository's analyzer.
- `Eff<A>.IfFail(Func<Error,A>)` and `Eff<A>.IfFail(A)` return raw `A` — they are terminal, rail-collapsing operators that recover a value out of the effect. `Eff<A>.IfFailEff(Func<Error,Eff<A>>)` is the rail-preserving recovery form, returning `Eff<A>`. `MapFail(Func<Error,Error>)` is the non-collapsing error-transform; it keeps the failure inside the envelope.
- `Eff<A>.Run()` returns `Fin<A>` and never throws; `Eff<A>.RunUnsafe()` throws on failure. For `Expected` failures the thrown exception is `ExpectedException`; for `Exceptional` failures the original captured exception is rethrown via `ExceptionDispatchInfo`, preserving the original stack. The `Run` / `RunUnsafe` axis is the consuming-edge terminal dichotomy: internal code uses `Run`, exception-required interop uses `RunUnsafe`. `RunAsync` returns `Task<Fin<A>>`; `RunUnsafeAsync` returns `ValueTask<A>`.
- `IO<A>.RunSafe()` wraps the computation in `Try().Run()`, capturing any exception as `Fin.Fail(Error.New(ex))`. It is the non-throwing exit for `IO<A>` when no `Eff` envelope is present. The `Eff` form inherits this behaviour — `Eff.Run()` reaches `RunSafe()` through the `ReaderT` unwrap, so all exception routing follows the same `Error.New` path.
- `FinGuardExtensions.ToFin(Guard<Error,Unit>)` converts a `guard(flag, error)` into `Fin<Unit>.Succ` or `Fin<Unit>.Fail(error)`. `Guard<E,Unit>.Bind<B>(Func<Unit,Fin<B>> f)` provides direct monadic continuation, making `guard` composable inside `from`/`select` query expressions without introducing a terminal collapse mid-chain.

[CATCH_COMBINATOR_COMPOSITION]:
- `CatchM<E,M,A>` is `readonly record struct(Func<E,bool> Match, Func<E,K<M,A>> Action)`. Its `Run(E error, K<M,A> otherwise)` applies the predicate: if false, returns `otherwise`; if true, calls `Action(error)`. The `|` operator on `K<F,A>` where `F : Fallible<E,F>` is an extension method in `FallibleExtensions`: `lhs | rhs` calls `lhs.Catch(rhs)`, which delegates to `F.Catch` with the predicate and recovery from `rhs`. Multiple `|` handlers compose left-to-right; the first matching handler consumes the error and subsequent handlers are not tested.
- `CatchMExtensions.Map<B>` on `CatchM<E,M,A>` produces `CatchM<E,M,B>(self.Match, e => self.Action(e).Map(f))`. This enables reusing a handler predicate while transforming its recovery output without re-writing the predicate — a shared handler factory returns `CatchM<Error,M,Unit>` and callers `Map` it to their specific bound type.
- `Prelude.@catch<M,A>(Error, K<M,A>)` matches via `error.Is` — deep structural equality on the `Error` record, not reference equality. `@catch<M,A>(int errorCode, ...)` matches via `e.Code == errorCode || e.HasCode(errorCode)`, traversing `ManyErrors` children. A `ManyErrors` error containing one child matching the code triggers the handler even though the outer type doesn't match.
- `Prelude.catchOf<E,M,A>(Func<E,K<M,A>>)` matches errors that are `E` or contain a child `E` via `e.IsType<E>()`. For `ManyErrors` inputs, the handler body calls `es.Filter<E>().ForAllM(e => Fail((E)e))` — `Filter` is the first operation the handler performs, selecting matching children, then `ForAllM` sequences them monadically with short-circuit on the first failure. Use `catchOfFold` when `M : MonoidK<M>` and partial failures across children should be combined rather than short-circuited.
- `exceptional<M,A>(Func<Exceptional,K<M,A>>)` and `expected<M,A>(Func<Expected,K<M,A>>)` are the typed `Error` subtype matchers, delegating to `catchOf<Exceptional,...>` / `catchOf<Expected,...>`. The distinction is load-bearing at retry logic: `Exceptional` errors (unplanned, host exceptions) should not be retried; `Expected` errors (domain, planned) are retry candidates.
- `Fallible<E,F>.Catch` is the abstract static method every carrier implementing `Fallible` must provide. The instance-level `.Catch(Func<Error,bool>, Func<Error,K<F,A>>)` extension on `FallibleExtensions` delegates to `F.Catch`, meaning the carrier provides the actual recovery semantics. For `Fin`, recovery produces a new `Fin<A>`; for `Eff`, recovery produces a new `Eff<A>` without touching the outer monad stack.
- `CatchIO<M,A>(Func<Error,bool>, Func<Error,K<IO,A>>)` is the `MonadIO`-specific catch: the recovery returns `K<IO,A>` rather than `K<M,A>`. Internally it calls `M.Catch(ma, Predicate, e => M.LiftIO(Fail(e)))` — the recovery is injected into `M` via `LiftIO`. This is the entry point for recovery paths requiring `IO` effects (logging, metrics) without constraining the algorithm's outer monad to exactly `IO`.
- `K<F,A> | Error rhs` (where `F : Fallible<F>`) catches all errors and replaces them with `rhs`, implemented as `F.Catch(lhs, _ => true, _ => F.Fail<A>(rhs))`. It stays in the failure rail — it forces a different error but does not recover. Its correct use is contextual error replacement at a layer boundary, not retry.

[NO_DOUBLE_VALIDATION_SEAM]:
- The seam law: the generated owner's factory or validation partial produces `Validation<Error,A>` or `Fin<A>` exactly once. Internal code calls `.ToFin()` at the transition point and operates on `Fin<A>` inward. No code downstream of the boundary calls `.IsValid`, re-checks fields, or re-runs admission predicates.
- `Validation<FAIL>` `Bind` short-circuits (monad instance returns `Fail(value)` immediately on failure); `Apply` accumulates (applicative instance combines both failure sides via `Semigroup<FAIL>.Combine` when both are `Fail`). The admission phase uses `Apply` — building the success value via `*` applicative chains — precisely to collect all field errors. Switching to `Bind` inside the admission chain silently disables accumulation. When accumulation across fields is needed inside a `ValidationT<Error,IO>` context, build the field product via `*` applicative operators — not via `from`/`select` query expressions, which desugar to `Bind`.
- `guard<E>(bool flag, Func<E> False)` produces `Guard<E,Unit>` which binds into query expressions via `SelectMany`. Inside `Fin` pipelines, `FinGuardExtensions.Bind` replaces the pattern `if (!condition) return Fail(...); return f(unit)`. This is the mechanism for inline domain invariant checks after admission without creating a new validation layer.
- `ValidationT<F,M>` carries `Func<MonoidInstance<F>, K<M, Validation<F,A>>>` — the `MonoidInstance<F>` witness is passed at run time, threading `Empty` and `Combine` as boxed delegates so custom domain error types compose without per-step compiler-emitted trait dispatch. `Apply` sequences both sides inside `M` via `Bind` then accumulates inside `Validation`: when `M = IO`, both sides execute sequentially — `ValidationT<F,IO>` does not parallelize field validation. To run fields in parallel and accumulate errors, use `IO.Fork` manually and combine the `Validation<F,A>` results with `+` before entering the `ValidationT` layer.
- `SemigroupK<ValidationT<F,M>>.Combine` runs both sides: success short-circuits; failure combines with the other side's failure via `monoid.Combine`. `Choice<ValidationT<F,M>>.Choose` short-circuits on the first failure without combining — `|` on `ValidationT` is an early-exit alternative, not an accumulating combinator. `+` or `Combine` is the accumulating form; `|` is not.

[CARRIER_POLYMORPHIC_ALGORITHMS]:
- `K<F,A>` is a phantom-type witness interface with no members, enabling HKT simulation. Every carrier implementing `Monad<M>` also implements `K<M,A>` for its bound type, which means algorithms parameterized over `K<M,A> where M : Monad<M>` work uniformly over `Fin`, `Eff`, `Option`, `IO`, and monad-transformer stacks. `Fallible.fail<E,F,A>(E error)` and `Fallible.error<F,A>(Error error)` are the static entry points for raising failures in carrier-polymorphic algorithms — using `Fin.Fail(e)` directly inside `K<M,A> where M : Fallible<Error,M>` binds the implementation to a concrete carrier, defeating the abstraction.
- `Traversable.traverse<T,F,A,B>(Func<A,K<F,B>> f, K<T,A> ta)` requires `F : Applicative<F>` and uses applicative `Apply` semantics — for `Validation<FAIL>` this produces error accumulation across the traversed structure; for `Fin` the applicative `Apply` also accumulates failures when both sides fail, combining them via `Semigroup<Error>.Combine`. `traverseM<T,M,A,B>(...)` requires `M : Monad<M>` and is implemented as a `Bind`-based fold — for `Fin` this short-circuits on first failure. The two overloads differ in what they use: `traverse` dispatches through the applicative instance, `traverseM` through the monad instance. Choosing the wrong overload silently changes error-collection behavior with no compile-time indication.
- `MonadIO.liftIO<M,A>(IO<A> ma)` where `M : Maybe.MonadIO<M>, Monad<M>` is the generic IO injection for any monad in the `MonadIO` hierarchy, including monad-transformer stacks — the mechanism for sharing `IO`-bound operations across different `Eff<RT>` flavors without carrier-specific plumbing.
- `Natural<Fin, Eff>`, `Natural<Fin, IO>`, `Natural<Fin, Option>`, `Natural<Fin, Try>`, `Natural<Fin, Either<Error>>` are all implemented on `Fin`. `Natural.transform<F,G,A>(K<F,A>)` performs the structural promotion with zero domain logic. An algorithm returning `K<Fin,A>` can be promoted to any of these targets at the instantiation site via `transform` or the implicit operator, keeping the algorithm carrier-agnostic.

[NATURAL_TRANSFORM_HIERARCHY]:
- `Natural<F,G>` declares `static abstract K<G,A> Transform<A>(K<F,A>)`. `CoNatural<F,G>` is the dual: `static abstract K<F,A> CoTransform<A>(K<G,A>)`. `NaturalMono<out F, in G>` implements both via a default `CoTransform` delegating to `F.Transform` — unidirectional promotion with zero manual impl. `NaturalIso<F,G>` implements both interfaces directly, asserting a full roundtrip. `NaturalEpi<in F, out G>` implements `Natural<G,F>` by calling `F.CoTransform`, so the "from" and "to" type parameters are inverted relative to the interface names. The distinctions are load-bearing: `NaturalMono` is correct for unidirectional promotions; `NaturalIso` asserts a roundtrip that fails to hold when failure information is lost on one path.
- `FinT<M>` implements `Natural<FinT<M>, EitherT<Error,M>>`, `Natural<FinT<M>, OptionT<M>>`, and `Natural<FinT<M>, TryT<M>>` — all one-way promotions via `Natural.transform`. The inverse is not provided: `EitherT` cannot be narrowed to `FinT` via `CoNatural` because the transformation is not an isomorphism.
- `CoNatural.transform<F,G,A>(K<G,A>)` is callable when `F : CoNatural<F,G>`. The distinction from `Natural.transform` is which type parameter bears the constraint: the target type `F` provides `CoTransform`. For transformer stacks where the outer monad implements `CoNatural` against its inner monad, the call site uses `CoNatural.transform<Outer,Inner,A>` to go downward and `Natural.transform<Inner,Outer,A>` to go upward.

[FINT_AS_MONAD_TRANSFORMER]:
- `FinT<M,A>` is `record(K<M, Fin<A>> runFin)` where `M : Monad<M>`. `FinT<M>` implements `MonadT<FinT<M>, M>`, supplying `Lift<A>(K<M,A>) → K<FinT<M>,A>` via `FinT.lift(ma)` — the canonical path for lifting an existing `M` computation into the transformer stack. An algorithm that accumulates failure via `Fin` while tracking effects in `M` composes `FinT<IO,A>` or `FinT<Eff,A>` rather than nesting the two independently.
- `FinT<M>.MapM<B>(Func<K<M,A>,K<M,B>>)` transforms the inner `K<M,A>` while preserving the `Fin` layer: extracts the success value, runs `f(M.Pure(value))`, then re-wraps in `Fin.Succ`; the `Fin.Fail` case short-circuits into `M.Pure(Fail)`. `MapT<M1,B>(Func<K<M,Fin<A>>,K<M1,Fin<B>>>)` transforms the monad type itself — the outer `M` to `M1` — enabling hoisting across monad boundaries without flattening the `Fin` envelope.
- `FinT<M>` implements `Applicative` via `Apply` that uses the `*` applicative operator on `K<M, Fin<Func<A,B>>>` and `K<M, Fin<A>>` concurrently — `Apply` enables the outer monad's concurrency semantics (parallelism if `M = IO`) while keeping `Fin` accumulation semantics intact for the inner carrier. This differs from `Bind` which sequences.
- `FinT<M>` implements `MonoidK` via `Combine`: if the left `runFin` is `Fin.Succ`, it short-circuits; if `Fin.Fail`, the right `runFin` is evaluated via `M.Bind`. This is the lazy `<|>` alternative behaviour inside the transformer — the first successful branch wins without evaluating the second.
- `FinT<M>` implements `Monad<FinT<M>>.Recur` via `M.Recur` applied to the step function after adapting `Next<A,Fin<B>>` from the inner result: a `Fin.Fail` inside the step produces `Next.Done(fail)` — short-circuiting the loop with the error as the loop result. A domain loop over `FinT<IO,A>` is stack-safe, cancellable, and error-aborting without any composition glue.

[EFF_RUNTIME_ENVIRONMENT_SEAM]:
- `Eff<A>` is `record(Eff<MinRT, A> effect)` and derives `Readable<Eff<A>, A>` via `Deriving.Readable<Eff<A>, A, ReaderT<A, IO>>`. `Readable.Ask` returns the environment `A` as a value; `Readable.Local<A>(Func<A,A> f, K<Eff<A>,B> ma)` runs `ma` with the environment mapped through `f` without mutating the outer environment. The type parameter `A` is simultaneously the runtime and the reader environment — `asks(f)` and `local(f, ma)` are available on any `K<Eff<A>,B>` computation.
- `Eff<RT,A>` is `record(ReaderT<RT, IO, A> effect)` — the runtime is the `ReaderT` environment, not a separately tracked phantom. `runtime<RT>()` is `liftEff((Func<RT,RT>)identity)` — it projects the environment into the bound value, making the runtime available as data inside the pipeline without any special accessor protocol.
- `localEff<OuterRT, InnerRT, A>(Func<OuterRT, InnerRT> f, Eff<InnerRT, A> ma)` is the runtime-narrowing combinator: it reads the outer runtime via `getState<OuterRT>()`, maps it to the inner runtime via `f`, then runs `ma.effect.Run(innerRT)` inside `IO.local`. An `Eff<InnerRT,A>` embeds inside an `Eff<OuterRT,A>` pipeline with a projection function — no `HasXxx` interface required; the runtime narrowing is structural.
- `Eff<A>.WithRuntime<RT>()` promotes an environment-free `Eff<A>` to an `Eff<RT,A>` by running `effect.RunIO(default(MinRT))` and lifting the resulting `IO<A>` via `MonadIO.liftIO<Eff<RT>, A>`. The `MinRT` default is a zero-allocation singleton struct. Calling `WithRuntime<RT>()` inside a `Bind` step in an `Eff<RT>` pipeline is the canonical way to embed pure-effect computations written against `Eff<A>` into a runtime-carrying context.

[MEMO_LAZY_APPLICATIVE_COMPOSITION]:
- `Memo<F,A>` has three construction paths: `internal Memo(Func<K<F,A>> f)` (state=0, suspended), `internal Memo(K<F,A> fa)` (state=2, pre-loaded), and an internal two-arg form for pre-loaded with a retained acquire function. The public `Prelude.memoK<F,A>(Func<K<F,A>> f)` is the suspension entry point; `memoK<F,A>(K<F,A> fa)` and `memoK<F,A>(A value)` produce pre-loaded memos. Pre-loaded memos involve no spin-lock on first access; suspended memos enter a `SpinWait`/`Interlocked.CompareExchange` loop serializing initial evaluation, with state reset to 0 on exception so a subsequent read retries acquisition.
- `Applicative<F>.Apply<A,B>(K<F,Func<A,B>> mf, Memo<F,A> ma)` is a distinct static abstract overload from the eager `Apply(K<F,Func<A,B>>, K<F,A>)`. For `Fin`, the memo overload forces `ma.Value` unconditionally as its first operation, then both the function result and the memo value participate in the carrier's failure-combination logic when either side is `Fail` — there is no short-circuit before the memo is evaluated. For `Validation<FAIL>`, both sides are similarly forced to enable error accumulation. The behavioral consequence is that a suspended `Memo` is always evaluated when it reaches `Apply`, regardless of the function side's state.
- `Memo.transform<F,G,A>(Memo<F,A> ma)` produces `Memo<G,A>(() => F.Transform(ma.Value))` — a new suspension that lazily promotes the inner value via `Natural.transform`. `Deriving.*` interfaces propagate memoization through natural transformations without forcing the value at transform time. `Memo<F,A>.Clone()` returns a fresh unsuspended `Memo` for reacquisition when `acquire != null`; `Reset()` clears the cached value via the same spin-lock so the next `Value` read re-invokes acquisition.

[FINAL_TRAIT_PIPE_SYNTAX]:
- `Final<F>` declares `static abstract K<F,A> Finally<X,A>(K<F,A> fa, K<F,X> @finally)`. The `|` extension operator `K<F,A> | Finally<F,X>` calls `lhs.Finally(rhs.Operation)`, enabling pipeline-style finally attachment: `computation | final(cleanup)`. `final<F,X>(K<F,X> @finally)` creates the `Finally<F,X>` record. The result is unconditional cleanup — not resource-scoped cleanup. `IO<A>`, `Eff<A>`, and `Eff<RT,A>` all implement `Final<F>`.
- `IOFinal<X,A,B>.Invoke` runs `Fa.RunAsync(envIO)`, then returns `Final.As().Bind(_ => Next(result))` as an un-run `IO<B>` — the same `envIO` is reused for both `Fa` and the final. Because `envIO.Token` is shared, any cancellation that occurs during `Fa` execution is already signalled when the finally executes. Cancel-safe teardown requires `final(cleanup.Uninterruptible())`.
- `Eff<A>.Finally` delegates to `ReaderT<A, IO, T>((A env) => fa.RunIO(env).Finally(@finally.As().effect.Run(env)))` — both primary and finally receive the same `env`, binding the environment into both halves at composition time. If `Local` modified the environment for the primary, the same modification is in scope for the finally.

[CHRONIC_TRANSFORMER_COMPOSITION]:
- `ChronicleT<Ch,M,A>` is `record(Func<SemigroupInstance<Ch>, K<M, These<Ch,A>>> runChronicleT)` where `These<Ch,A>` is the `This`/`That`/`These` bifunctor. `dictate(value)` lifts `That(value)` — a bound value with no chronicle entry. `confess(ch)` lifts `This(ch)` — a fatal abort. `chronicle(ch, value)` lifts `Both(ch, value)` — a chronicle entry alongside a bound value. The three constructors map to the three `These` cases; `dictate` does not produce a `Both` with a default chronicle entry.
- `ChronicleT<Ch,M>` implements `Fallible<Ch, ChronicleT<Ch,M>>` — failure is typed as `Ch`, not `Error`. `ChronicleT` is correct when the log type and the abort type share one vocabulary: `Both(ch, a)` carries a partial log entry alongside a bound value simultaneously. `ValidationT` models validation failures with no log/value duality; `ChronicleT` carries both.
- `Bind` on a `Both(ch, a)` case runs `f(a)` and combines chronicle entries: a `This(ch2)` result produces `This(ch.Combine(ch2))`; a `That(b)` result produces `Both(ch, b)`; a `Both(ch2, b)` result produces `Both(ch.Combine(ch2), b)` — chronicle entries accumulate across `Bind` steps without short-circuiting the bound value.
- `memento(ma)` recovers a confessed chronicle into `ChronicleT<Ch,M,Either<Ch,A>>`: `This(ch)` → `That(Left(ch))`; `That(a)` → `That(Right(a))`; `Both(ch, a)` → `Both(ch, Right(a))` — the chronicle entry is preserved alongside the bound value. `absolve(defaultValue, ma)` substitutes a default for `This` cases and continues; `condemn(ma)` converts `Both` into `This`, discarding the bound value when any chronicle entry exists. All three require only `M : Monad<M>` and are consuming-edge recovery combinators.

[EXCEPTION_CAPTURE_AND_CONTEXT]:
- `Error.New(string message, Error inner)` and `Error.New(int code, string message, Error inner)` produce `Expected` records that form a chain via `Error.Inner : Option<Error>`. Wrapping boundary failures with `MapFail(e => Error.New("layer context", e))` preserves the original error as `Inner` while adding layer context to `Message`. Walking `.Inner` reconstructs the full causal chain without any custom error types.
- `Eff<A>.MapFail(Func<Error,Error> f)` is implemented as `this.Catch(f).As()` — it creates a catch handler that transforms every failure, preserving the `Eff` envelope. This is the correct composition-time error-context addition; it attaches as a transformer to the pipeline, not as a terminal that unwraps and re-wraps.
- `Error.Combine(Error rhs)` and the `+` operator produce `ManyErrors` by merging. `Error.Many(params Error[])` and `Error.Many(Seq<Error>)` are the batching constructors. An error context accumulating multiple sub-errors — e.g., parallel validation passes — uses `+` or `Error.Many` before entering the rail.
- `Exceptional.Inner` traverses to `Error.New(Value.InnerException)` when the wrapped `Exception` has an inner exception, preserving the full exception chain as nested `Error` values. `Expected.Inner` carries an explicit `Option<Error>` field set at construction. Neither requires unwrapping the error to access the chain.

[ERRORHIERARCHY_STRUCTURAL_QUERYING]:
- `ManyErrors.IsType<E>()` returns true when `Errors.Exists(e => e.IsType<E>())` — it traverses direct children, and each child recurses via `Error.IsType<E>` which delegates to `this is E`. For `Expected` and `Exceptional` leaf nodes the check is a direct `is`-check; for nested `ManyErrors` it is a `Seq.Exists` loop. `catchOf<Expected,M,A>` on a `ManyErrors` containing mixed expected and exceptional children triggers the handler even though the outer wrapper is neither.
- `ManyErrors.Filter<E>()` where `E : Error` returns a new `ManyErrors` containing only children where `IsType<E>()` is true, via `Errors.Choose`. The base `Error.Filter<E>()` returns `this` if `this is E`, else `Errors.None`. `Filter` is called inside `catchOf` and `catchOfFold` handlers — it is the first operation they perform to select matching children before `ForAllM` sequences them. As a consumer-side call it is also available post-rail-termination for error analysis.
- `Error.ForAllM<M,A>(Func<Error,K<M,A>> f)` on a single error calls `f(this)`; on `ManyErrors` it sequences via `Bind` across direct children, short-circuiting on the first failure `f` produces. The complementary `FoldM<M,A>` uses `MonoidK.Combine` instead of `Bind`, combining partial failures rather than short-circuiting. `AsIterable()` on `ManyErrors` returns a flat `Iterable<Error>` of direct children without deep-flattening nested `ManyErrors` — the depth is caller-managed.
- `Errors.None` is `ManyErrors(Seq.empty<Error>)`: `IsEmpty = true`, `Code = -2000000006`. `Error.Combine(empty)` returns `this`; `empty.Combine(rhs)` returns `rhs` — `Errors.None` is the monoidal zero for accumulation. It is still an error value that produces `Fin.Fail` if it reaches the failure rail; `IsEmpty` is never equivalent to `null` or a recovered state.

[FALLIBLE_PARTITION_COMBINATORS]:
- `FallibleExtensions.PartitionFallible<F,M,A>(K<F,K<M,A>>)` where `F : Foldable<F>` and `M : Monad<M>, Fallible<M>` returns `K<M,(Seq<Error> Fails, Seq<A> Succs)>`. It left-to-right folds over the foldable, binding each effect in sequence and routing successes and failures into separate accumulators via `Catch`. The fold neither short-circuits on the first failure nor requires a `Validation` carrier — the correct choice for batches where all operations must be attempted and results partitioned.
- `FallibleExtensions.Succs<F,M,A>` and `Fails<F,M,A>` are the single-side variants. `Fails` accumulates via `ma.Bind(_ => M.Pure(s)).Catch(e => M.Pure(s.Add(e)))` — the success arm returns the unchanged accumulator and the failure arm appends the error — making both combinators stack-safe for any `M` supporting `Bind` and `Catch`.
- The generic `fails<E,F,M,A>(K<F,K<M,A>>)` in the traits module carries a general error type parameter `E` distinct from `Error`, enabling partition over carriers with typed error rails — `ValidationT<F,M,A>` where `F` is a custom monoid. The concrete overloads pin `E = Error`. The generic versions require `M : Monad<M>, Fallible<E,M>` and `F : Foldable<F>`.

[ANTI_PATTERNS]:
- Throwing inside a `from`/`select` query expression over `Fin`, `Option`, or `Eff` does not capture the exception into the rail; it propagates to the caller. `Error.New(ex)` wrapped in `Fin.Fail` or `Eff.Fail` is the correct form. `liftEff(Func<A> f)` captures synchronous throws; `liftEff(Func<Task<A>> f)` captures async throws. Any raw `throw` inside domain logic is a violation.
- `Fin<A>.Match(Succ, Fail)` at a mid-pipeline `Bind` step is identity-equivalent to `curr.Map(f)` but destroys type inference, creates unnecessary closure allocation, and violates the mid-pipeline collapse constraint. The correct rail-preserving forms are `Map`, `Bind`, `MapFail`, and `BiMap`.
- Re-calling the generated owner's factory or running a validation partial inside a `Bind` step is the double-validation anti-pattern. The `Fin<A>` produced by the first admission already carries the invariant; all downstream operations bind against the admitted value.
- Ambient state accessed inside `Bind` or `Map` closures (static fields, thread-static, context-accessor patterns) breaks referential transparency and makes the `Eff` pipeline non-testable. Boundary state belongs in `Atom<T>` (lock-free CAS), `Ref<T>` (STM), or the `Eff<RT>` reader environment, not in ambient singletons inside lambda captures.
- Per-carrier pipeline duplication — writing one pipeline for `Fin`, another for `Eff`, another for `Option` sharing identical structure — is eliminated by parameterizing over `K<M,A> where M : Monad<M>, Fallible<Error,M>`. The algorithm body uses `Monad.bind`, `Fallible.fail`, `M.Pure`, and `Natural.transform` at carrier-specific edges.
- Calling `RetryUntil` or `Retry` directly on an `IO<A>` after `BracketFail()` has already been applied in an outer scope means every failed attempt releases the outer scope's resources, not just the retrying computation's resources. The retry methods internally call `BracketFail()` — wrap only the minimal retrying unit, not the outer scope.
- Writing `io.Fork().Bind(f => f.Await)` instead of `Prelude.awaitIO<M,A>(K<M,ForkIO<A>>)` re-implements the await pattern manually and breaks the `IOEmpty<A>` fast-path, adding unnecessary allocation. The carrier-polymorphic form delegates to `MonadUnliftIO<M>.Await` and preserves the fast-path.
- `Atom<A>.Swap(f)` (the non-IO synchronous form) inside a `Map` or `Bind` lambda is a side effect inside a pure transform — the CAS spin-loop runs synchronously, blocking the executing thread. `swapIO(Atom<A>, Func<A,A> f)` lifts the operation into the `IO` layer and preserves pipeline referential transparency.
- `IO.Uninterruptible()` passes `CancellationToken.None` into the `EnvIO` for the entire wrapped computation — any `yieldFor`, `Fork`, or `liftAsync` inside that section ignores cancellation until the uninterruptible boundary exits. Resources allocated inside are still tracked in the parent's scope. Wrap only the minimum atomic unit, not a full pipeline branch.
- Accessing `Prelude.envIO` or `Prelude.cancelToken` inside domain logic to branch on token state or extract resources violates boundary admission — `envIO` is `IO.lift((EnvIO e) => e)`, a synchronous snapshot of the current environment. Those concerns belong in bracket, finally, and fork primitives.
- `io | final(cleanup)` runs `cleanup` on the same `EnvIO` as the primary, inheriting the parent's `CancellationToken`. When the primary is cancelled, the token is already signalled when `cleanup` executes. Cancel-safe teardown requires `final(cleanup.Uninterruptible())`.
