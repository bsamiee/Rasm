# Effects-Runtime: Frontier

[IMPLICIT_CONVERSION_AS_LIFT_SURFACE]:
- `Eff<A>` carries nine implicit conversions: from `Pure<A>`, `Fail<Error>`, `Lift<A>`, `Lift<Fin<A>>`, `Lift<MinRT,A>`, `Lift<MinRT,Fin<A>>`, `Either<Error,A>`, `Fin<A>`, and `IO<A>`. Every already-computed carrier value becomes an `Eff<A>` without an explicit lift call; the `Fin<A>` implicit is load-bearing at boundary seams where a validation rail rejoins the effect rail. `Eff<RT,A>` adds one further implicit: from `Eff<A>`, widening the no-RT carrier to the RT form — the single direction of implicit widening matches the type-parameter direction.
- `IO<A>` is not implicitly convertible to `Eff<A>` at the type level; the `Eff<A>.op_Implicit(IO<A>)` is the one-way bridge. When an `IO<A>` value must participate in a `K<Eff, A>` context without explicit wrapping, pass it through `Eff.lift(io)` — the static form is the canonical spelling and matches the core lift shape.
- The `Lift<A>` implicit on `Eff<A>` accepts a `Lift<A>` ref-struct by-ref (`@`). This means an inline `Lift(() => expr)` value cannot be stored in a variable before passing — it must be constructed at the conversion site. The lifetime constraint is not enforced by the compiler for method-parameter contexts but signals that these ref-struct lifts are not general values.

[RUNTIME_SCOPING_WITH_LOCAL]:
- `Eff.local<OuterRT, InnerRT, A>(Func<OuterRT, InnerRT> f, Eff<InnerRT, A> ma)` returns `Eff<OuterRT, A>`. Its implementation reads `getState<OuterRT>()` to get the outer runtime, maps it through `f` to produce the inner runtime, runs `ma.effect.Run(innerRT)` wrapped in `IO.local` (fresh cancel scope), then re-surfaces the result in the outer carrier. This is the canonical mechanism for embedding a dependency-isolated sub-computation whose runtime is a projection of the outer one — no capability leakage between scopes.
- `Eff.runtime<RT>()` lifts the runtime record itself into the bound value as `Eff<RT, RT>`. Combined with `.Bind(rt => ...)` it enables a computation to inspect its own runtime at composition time without breaking the reader monad abstraction. Used to derive derived capabilities or perform runtime-conditional branching without breaking the monadic spine.
- `Eff.getState<RT>()` returns `Eff<RT, (RT Runtime, EnvIO EnvIO)>` — it exposes the full internal state of the running effect, including both the capability record and the `EnvIO`. Use it when downstream logic must inspect the execution environment directly; `Eff.runtime<RT>()` gives only the `RT` value, `getState` gives the pair.
- `Eff.localCancel<RT, A>(Eff<RT, A> ma)` delegates to `ma.LocalIO().As()`, which calls `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — fresh CTS linked to the parent token, shared resources. Cancellation of the inner scope does not propagate to the outer scope's other computations; an exception produced by `IO.cancel` inside the local does propagate outward because it is an exception, not a signal. Scope isolation prevents regional cancellation from tearing sibling effects; it does not suppress the exception.

[STM_AND_REF_INTEGRATION]:
- `STM.DoTransaction` is `internal`; the public surface is `atomic<R>(Func<R> op, Isolation)` in `Prelude`, with `atomicIO<R>` wrapping the same call in `IO.lift`. Effect-pipeline integration requires wrapping: `IO.lift(() => atomic(op, isolation))` is what `atomicIO` does. The `Eff`-overloaded `DoTransaction` forms exist internally and nest when a parent transaction is active; `RunTransaction` is the raw form that never nests. Prefer `atomicIO` at call sites to stay inside the `IO` rail.
- `Isolation.Snapshot` rolls back only when a `Ref` *written within* the transaction is externally written between start and commit. `Isolation.Serialisable` rolls back when any `Ref` *read or written within* the transaction is externally written. Choosing `Snapshot` admits more concurrency but leaves read-write skew possible; `Serialisable` prevents all anomalies at the cost of higher conflict rate.
- `Ref<A>.Commute(Func<A, A>)` re-runs the function at commit time against the most-recently-committed value, not the in-transaction snapshot. This means `Commute` is only correct for commutative or last-writer-wins operations — non-commutative uses will silently produce stale-basis results. The `CommuteRef<A>` wrapper returned by `DoTransaction(Func<CommuteRef<A>>)` provides the same semantics with explicit commute-function registration via `STM`.
- `Ref<A>.SwapIO(Func<A, A>)` returns `IO<A>` — it wraps `Swap` in `IO.lift`, enabling participation in effect pipelines. The swap is transactional when called inside an `atomic` boundary; outside a transaction it operates as a standalone assignment (not a CAS spin, since `Ref` mutation uses STM commit, not lock-free CAS). `Atom<A>.SwapIO` returns `IO<A>` from `IO.lift(_ => Swap(f))` — an actual CAS spin loop that runs until no contention; the two types have the same IO return shape but different internal mutation semantics.
- `Atom<A>.SwapMaybeIO(Func<A, Option<A>>)` wraps `SwapMaybe` in `IO.lift`: when the function returns `None` the atom is unchanged and the IO yields the current value; when it returns `Some(v)` the atom is updated to `v` and the IO yields `v`. The function runs inside a spin loop on contention, so it must be side-effect-free and cheap. This is the idiomatic form for conditional test-and-set where the update decision depends on the current value.

[TIMEOUT_BEHAVIOR_AND_CARRIER_POLYMORPHISM]:
- `IO<A>.Timeout(TimeSpan)` uses `IOTimeout<A,A>`, which calls `Fa.Run(envIO.LocalCancelWithTimeout(limit))`. When the CTS fires, `Run` propagates a `TaskCanceledException` through the trampoline's catch stack — the return type is `IO<A>`, not `IO<Fin<A>>`, so the exception is not wrapped automatically. To model timeout as a typed failure, either follow with `@catch` on the cancellation type or use the carrier-polymorphic `TimeoutIO<M, A>(K<M, A>, TimeSpan)` extension, which delegates to `MapIO(ma, io => io.Timeout(ts))`. The timeout clock starts when `IO` execution begins, not at construction.
- `MapIO(ma, io => io.Timeout(ts))` being the implementation means `TimeoutIO` is carrier-polymorphic via `MonadUnliftIO<M>`: any `Eff<A>`, `Eff<RT, A>`, or transformer stack picks it up without specialization.
- `IO<A>.WithEnv(Func<EnvIO, EnvIO>)` and `IO<A>.WithEnvFail(Func<EnvIO, EnvIO>)` are the low-level env-mapping primitives that `Bracket`, `BracketFail`, `Timeout`, and `Fork` all build on. `WithEnv` releases the mapped resources on both success and failure; `WithEnvFail` releases only on failure and leaks resources back to the parent scope on success. A custom DSL extension that requires a specialized `EnvIO` shape constructs it via `WithEnv`/`WithEnvFail` — never by accessing `EnvIO` fields directly.

[APPLICATIVE_TUPLE_AND_OPERATOR_FORMS]:
- `EffExtensions` provides `SelectMany` overloads whose source is a `ValueTuple` of effects — `(K<Eff,A> First, K<Eff,B> Second)` and the 3-tuple form. In a query expression the bound tuple exposes named fields (`First`, `Second`, `Third`), not positional destructuring. The semantics are monadic left-to-right sequential; both effects run before either result is accessible. Applicative concurrency requires explicit `fork`/`awaitAll`. The symmetric overloads where `bind` returns a tuple also exist: `from a in effA from (b, c) in f(a)` destructures the pair returned by `f`.
- The `*` operator on `K<Eff, Func<A,B>>` and `K<Eff, A>` is curried applicative application; `IOExtensions` defines symmetric overloads up to 11-arity curried functions — `Func<A0,...,A10, B>` applied to `K<IO, A0>` through the final partial. All effects run left-to-right and the function is applied to all results. This is the dense form for combining up to 11 sequentially-executed effects without nested `SelectMany`; each application strips one function argument and returns a new `K<IO, Func<...>>` until fully applied.
- `>>>` (`op_UnsignedRightShift`) on `K<Eff, A>` with `K<Eff, B>` runs the left effect for its side-effects, discards the value, then runs the right effect and returns its value. It is distinct from `>>` (`Kleisli`): `>>>` ignores the left value entirely, while `>>` feeds it as input. The operator is available on both `Eff` and `IO` via their respective extension classes.
- `MonadIOExtensions.SelectMany<M, A, B, C>(K<M, A>, Func<A, IO<B>>, Func<A, B, C>)` and the symmetric `SelectMany<M, A, B, C>(IO<A>, Func<A, K<M, B>>, Func<A, B, C>)` enable mixed-carrier LINQ queries where one step is `IO` and another is `Eff<RT>` without explicit `.LiftIO`. The first form binds an `IO` step inside an `M`-shaped query; the second binds an `M` step inside an `IO`-shaped query. Both delegate to the carrier's `LiftIO` automatically.

[FALLIBLE_COLLECTION_PARTITIONING]:
- `succs<M, A>(Seq<K<M, A>>)` runs each effect in the sequence and returns only the successful values; failed effects are silently dropped. `fails<M, A>(Seq<K<M, A>>)` returns only the failure values. Both execute eagerly and neither accumulates. When both halves are needed, pair two passes or follow `TraverseM` with a partition — the API provides no single `partitionM` form.
- `FallibleExtensions.CatchIO<M, A>(K<M, A>, Func<Error, bool>, Func<Error, K<IO, A>>)` matches the `@catch` family but lifts the recovery path to `IO` rather than the outer carrier `M`. This is used at the boundary where recovery produces a raw IO effect (typically a BCL call wrapped in a single `IO.lift`) that does not belong to the outer runtime context. The recovery result is then lifted back into `M` via `MonadIO.liftIO`. The three-overload family covers predicate, Error-equality, and numeric-code matching.
- `guardIO(bool)` returns `IO<Unit>` — on `false` it returns `IO.empty<Unit>()`, which propagates as `Errors.None` through the catch stack. `guard<E>(bool, Func<E>)` returns `Guard<E, Unit>` for generic query expressions over `Fallible<E>` carriers. The two forms are not interchangeable: `guardIO` belongs in `IO`-typed query expressions; `guard<Error>(...)` belongs in `Eff`-typed query expressions that must emit a named typed error on the false branch.

[SCHEDULE_ADVANCED_PATTERNS]:
- `Schedule.repeat(int times)` produces `SchRepeat` — it repeats the *entire base schedule* `n` times, not individual retry iterations. `Schedule.recurs(int times)` is `.Take(times)` on the base — it limits the total number of steps. The naming inversion is load-bearing: `recurs(5)` gives exactly 5 attempts; `repeat(5)` gives 5 full cycles of the base schedule, each with as many steps as the base contains. Use `recurs` for count-bounded loops, `repeat` for bounded phase cycling such as resetting a window pattern `n` times.
- `Schedule.fixedInterval(Duration, currentTimeFn?)` fires at absolute wall-clock boundaries regardless of how long the action took. If the action runs longer than the interval, the *next* fire is immediate — no pile-up of queued executions. `Schedule.windowed(Duration, currentTimeFn?)` subdivides the timeline into windows and sleeps until the next window boundary; it can fire multiple times per window if the execution is fast. Both accept an optional `currentTimeFn` for deterministic test injection.
- `ScheduleTransformer` is a `readonly struct` wrapping a `Func<Schedule, Schedule>` field with `Apply(Schedule)` and `op_Addition`. Composing transformers with `+` produces a new transformer that applies left-then-right: `(a + b).Apply(s)` equals `b(a(s))`. The constructor is `internal`, but `Schedule.Transform(s => ...)` is the public factory — any `Schedule -> Schedule` function qualifies.
- `Schedule.intersperse(Duration[] durations)` accepts a `params Duration[]` and builds an `Iterable<Duration>` from the array before constructing `SchItems`. The `Schedule.intersperse(Schedule inner)` overload binds each step of `inner` between the base steps. These two overloads do not conflict at call sites because `Duration` and `Schedule` are not implicitly convertible; the array overload admits inline duration literals as a variadic probe series.

[FORK_RESOURCE_AND_AWAIT_SEMANTICS]:
- `IO<A>.Fork(Option<TimeSpan>)` uses `Task.Factory.StartNew` with `TaskCreationOptions.LongRunning`, requesting a dedicated OS thread. For short-lived tasks this creates excess OS-thread pressure; `fork` is designed for computations that would otherwise block the thread pool. For lightweight parallelism over a collection, prefer `awaitAll(Seq<K<M,A>>)`, which forks via `TraverseM` and does not use `LongRunning`.
- `awaitAny(Seq<K<M, A>>)` forks each effect, uses `Task.WhenAny`, picks the winner's value, and returns it. Losers are neither cancelled nor awaited — their tasks continue running until their own timeout or completion. When guaranteed cleanup is required, take explicit `ForkIO<A>` handles, call `awaitAny(Seq<ForkIO<A>>)`, then cancel losers via `forkIO.Cancel.Run(env)` inside a `Finally` block.
- `ForkIO<A>.Await` is `IO<A>`, not `IO<Fin<A>>`. Cancellation of the *awaiting* environment (not the fork's own token) unblocks `Await` with a cancellation exception while the fork continues running. Re-awaiting the same `ForkIO<A>` handle in a retry loop is unsafe: the underlying `Task<A>` completes once, and subsequent awaits race on an already-completed task, potentially returning the result multiple times or producing `ObjectDisposedException` if the fork's cleanup has run.
- `IOExtensions.WaitOneIO(AutoResetEvent)` and `WaitOneIO<A>(AutoResetEvent)` lift BCL synchronization primitives into the `IO` monad. The typed overload returns `IO<A>` where `A` is a phantom — useful when a downstream bind requires a typed IO. These are the only BCL sync-primitive adapters in the extension class; `Semaphore`, `ManualResetEvent`, and `Monitor` have no built-in `IO` wrappers and must be lifted via `IO.lift(() => evt.WaitOne())`.

[DSL_EXTENSION_POINTS_AND_CORRECTNESS]:
- The four legal `IO<A>` DSL extension base types are `InvokeSync<A>`, `InvokeAsync<A>`, `InvokeSyncIO<A>`, and `InvokeAsyncIO<A>`. `InvokeSyncIO<A>` is the only synchronous base whose `Invoke` returns `IO<A>` rather than `A` — it enables two-phase dispatch where phase selection is synchronous and the body is an `IO` chain. `InvokeAsyncIO<A>` is the async counterpart: `Invoke` returns `ValueTask<IO<A>>`. Subclassing any other `IO<A>` type throws `Errors.IODSLExtension` at run time, not compile time.
- `IOCatch<X, A>` carries `Operation`, `Predicate`, `Failure`, `Final?`, and `Next` as record fields. `MakeOperation()` inserts an `IOCatchPop` node after the success path; `MakeHandler()` constructs the failure branch that checks `Predicate` and routes to `Failure`. The `Final` field, when set, runs unconditionally as a `Finally` on both paths. Each `IOCatch` node pushes one `StackItem` onto the catch stack at run time — avoid allocating these nodes per-call in tight loops.
