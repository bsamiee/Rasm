# Const-type-parameter inference pull

[PULL_THESIS]:
- The `<const T>` modifier migrates the literal-preservation assertion from the call expression into the signature, so the assertion is paid once at declaration and every consumer omits `as const` — the inward pull is the collapse: a clean call site is the deleted `as const`, and the entrypoint owns the narrowness its whole result type depends on. The naive chain writes the assertion outward, once per call (`f(['<a>', '<b>'] as const)`); the pulled form writes it once inward (`f: <const T>(x: T)`) and every call drops the operator.
- The pull amortizes one assertion across N call sites: the outward `as const` is paid once per call and scales with consumer count, the inward `const` is paid once in the signature and scales with nothing — so the collapse is quantified, an entrypoint with `K` consumers deleting `K` assertions, and the lever's value grows with the surface's fan-in rather than being a per-call constant. An entrypoint whose argument is asserted `as const` at every call is the un-pulled signature, the repeated assertion the missing modifier would have absorbed.
- The pull reaches every depth the outward assertion would have in one modifier: `const` is recursive, so a nested object inside an array inside the argument is pinned to its leaf literals without a per-level `as const`, and the depth a downstream type reads — `T[number][K]`, a positional tuple of records — is exactly the depth the single signature-side modifier preserved.
- The pull is what makes a downstream conditional, mapped, or template type resolve against literals it could otherwise never see: a result type that branches on `[T] extends [ReadonlyArray<...>]`, indexes `keyof T`, or reads `T[K]['field']` is computed against the widened constraint the moment `const` is dropped, so the entrypoint's `const` is the precondition for its own return-type derivation — the lever and the derived family are one obligation, the lever at the parameter and the obligation discharged in the return type.
- The reach is the call expression alone and stops at type erasure: `<const T>` shapes the inference of one parameter for one call, leaves no runtime artifact, enforces no `readonly` on the value (the `readonly` is the inferred type's, the object stays mutable), and survives no re-binding through a widened alias — the entrypoint narrows against the literal as written at the source, never the dynamic shape a value carries when it arrives from outside.

[SLOT_SEEDING]:
- One `const` modifier seeds multiple type-parameter slots of a single returned carrier, and the count of seeded slots is the measure of how far the pull reaches: `Match.value: <const I>(i: I) => Matcher<I, Without<never>, I, never, I>` threads the inferred `const I` into THREE positions of the matcher — the `Input` slot, the `RemainingApplied` remainder slot initialized to the full union, and the `Provided` slot carrying the runtime value — so a single inferred literal union simultaneously fixes what the matcher accepts, what it has left to consume, and what value it folds over.
- The `Provided` slot is the value-carrier the type-source entrypoint lacks: `Match.type: <I>() => Matcher<I, Without<never>, I, never, never>` takes no value, supplies `I` as an explicit type argument, and seeds `Provided = never` because there is no value to carry — so the matcher type itself records whether a value was pulled in (`Provided = I`) or a type was named (`Provided = never`), and the absence of `const` on `Match.type` is structural, not omitted: with no value to widen there is no pressure for `const` to relieve.
- The remainder slot is the inward pull's leverage on the exhaustiveness proof: because `const I` seeds `RemainingApplied` with the full narrow union rather than a widened primitive, each arm subtracts a real member and the terminal `never`-proof can reach zero — a missing `const` would seed the remainder with a base type that subtraction never empties, so the inward pull at the value position is the precondition for the proof at the terminal position, the two ends of the matcher type bound by one modifier.

```typescript
import { Match } from 'effect'
const classify = Match.value({ status: 200, body: '<ok>' }).pipe(
    Match.when({ status: 200 }, (r) => r.body.length),
    Match.when({ status: Match.number }, (r) => -r.status),
    Match.orElse(() => 0),
)
```

[SHAPE_DISCRIMINANT_PULL]:
- The deepest reach of `<const T>` is driving a modality-polymorphic entrypoint: when a single function accepts both a positional tuple and a keyed record and returns a shape-dependent result, `const` is the lever that lets the return-type conditional tell the two apart. `all: <const Arg extends Iterable<F<any>> | Record<string, F<any>>>(arg: Arg) => [Arg] extends [ReadonlyArray<F<any>>] ? { -readonly [K in keyof Arg]: ... } : ...` resolves the tuple branch — a positional mapped type preserving arity, order, and each element's heterogeneous payload — only because `const` pinned the array argument to a tuple; drop `const` and the array widens to `Array<F<any>>`, `keyof Arg` collapses to `number`, the positional projection dissolves to the homogeneous `Array<A>` fallback, and the heterogeneous-tuple result is lost.
- This is the chain-collapse thesis at its sharpest: one entrypoint discriminates on input shape — `MODAL_ARITY` realized — and the discriminant is recoverable from the value's own type only because the `const`-pulled inference preserved the distinction between a tuple and an unbounded array that widening would have erased. The naive form splits the entrypoint into `allTuple`/`allRecord` siblings precisely because, without the inward pull, one signature cannot keep the shapes apart; `const` is what permits the single polymorphic surface.
- The discriminant pull is structural across the carrier family: the same `<const Arg extends Iterable | Record>` shape drives the result conditional for every applicative collector, so a heterogeneous fixed-arity gather and a homogeneous iterable gather are one entrypoint whose return type the `const`-pinned `Arg` selects — the tuple-positional projection and the array fallback are two arms of one derived type, never two functions.

```typescript
import { Either } from 'effect'
const gather = Either.all([Either.right(1), Either.right('<v>'), Either.right(true)])
type Gathered = typeof gather extends Either.Either<infer A, infer _E> ? A : never
const keyed = Either.all({ count: Either.right(0), label: Either.right('<x>') })
```

[IDENTITY_LITERAL_PULL]:
- `<const Id extends string>` pulls a key literal into a non-instantiable owner's identity slot so the owner's type records the exact key, never the widened `string`: `Tag: <const Id extends string>(id: Id) => <Self, Shape>() => TagClass<Self, Id, Shape>` carries the literal `Id` into the `TagClass`'s second type parameter, so two tags built from different string literals are distinct types and the literal is the de-duplication key the runtime context resolves on — without `const`, both ids widen to `string`, the `Id` slot cannot tell the tags apart, and a `typeof id` restatement is the only way back to the literal the pull would have kept.
- The pull conditions a downstream proxy projection: `Service` threads `<const Key extends string, const Make extends ...>` so the produced `Service.Class<Self, Key, Make>` indexes both the pinned key and the pinned shape — the constructor make-record's accessor surface and the service's resolution key both read the literals the two `const` modifiers captured, so one declaration's identity AND its constructor faces derive from the call-site literals, no second statement naming the key as a constant.
- The identity pull is the dual of the value pull: the value entrypoint pulls a runtime value's literal type inward, the identity entrypoint pulls a key literal inward, and both delete the same restatement — a follow-on `const KEY = '<id>'` next to a tag, or a `typeof KEY` alias re-deriving the literal the signature already pinned, is the value-after-value link the inward pull removes at the declaration.

[CONSTRAINT_CLAMP_ORTHOGONALITY]:
- `<const T extends C>` stacks two independent forces on one parameter: the constraint `C` is the floor the argument must clear, the `const` modifier is the ceiling against widening — they act on opposite directions and never substitute. `fixed: <const T extends NonEmptyArray<any>>(values: T)` requires the argument be a non-empty array (constraint) AND pins it to its exact tuple so the length and element literals survive (`const`); the constraint alone would accept a non-empty array widened to `Array<element>`, losing the very arity the `NonEmpty` proof and any positional projection depend on.
- The clamp is what lets a rest-spread entrypoint derive per-position results: `make: <const Steps extends NonEmptyReadonlyArray<Step>>(...steps)` constrains each spread argument to the step shape AND pins the whole tuple, so the return type reads `Steps[number]['field']` and per-index projections off a tuple the constraint admits but `const` preserves — the constraint proves each element is well-shaped, the `const` proves the tuple kept its members, and the derived result needs both.
- The orthogonality is the selection rule: reach for the constraint when the argument must clear a shape floor, add `const` when a downstream type reads the argument's literal value or positional structure, and stack both when the entrypoint both admits a bounded family and derives its result from the family's exact members — a `const` with no literal-reading return type is the precautionary-lever defect (it pins what nothing downstream consumes), and a bare constraint where the return type indexes positions is the missing pull (the result derives against the widened constraint and collapses).
- The clamp interacts with an excess-property fence at the same parameter: `<const Make extends NoExcessProperties<{ ... }, Make>>` pins `Make`'s key set with the modifier so the self-referential `NoExcessProperties<Base, Make>` can compute `Make`'s keys against the exact literal set written, rejecting a stray key — so on one parameter the constraint is the shape floor, the `const` is the literal ceiling, and the `NoExcessProperties` self-reference is the no-extra-key wall, three forces the entrypoint stacks where the naive form would split into a validate-then-construct sequence.

[DEFAULTED_PARAMETER_PULL]:
- The `const` modifier composes with a literal default on the SAME type parameter, a pairing no call-site `as const` can express: `<const Preload extends boolean = false>(..., options?: { readonly preload?: Preload })` declares the default narrow value `false` AND pins whatever literal the call passes — so an omitted option resolves the parameter to the pinned default `false` and a passed `true` resolves it to the pinned `true`, both narrow, where an outward assertion has no syntactic home (there is no argument to assert when the option is omitted). The default-plus-`const` is the only spelling that pins both the absent-argument and the present-argument case to a literal.
- The pinned literal then steers a return-type conditional reading the option's VALUE: `Preload extends true ? never : E` elides the error channel exactly when `true` was passed, so the entrypoint's result type derives from the flag the call supplied — drop `const` and `preload: true` widens to `boolean`, the conditional stays unresolved, and the channel elision the literal was meant to trigger never fires. The lever is mandated here because the downstream type matches on the literal value, the same condition that earlier governed which option records carry the modifier.
- The defaulted-`const` collapses a boolean-knob-plus-two-bodies into one signature whose return type the pinned flag selects: the default declared once on the parameter, the call site passing the literal or omitting it, the result type deriving — a sibling overload per flag value, or a runtime branch re-deriving the channel the type already knows, is the surface this parameter deletes.

[ABSENCE_IS_A_DESIGN_DECISION]:
- The author of an entrypoint includes or omits `<const T>` by one test — does the parameter face a value-side widening pressure — so the modifier's presence in a verified signature is positive evidence the parameter is value-sourced and inferred fresh, and its absence is evidence the type is named as an explicit argument or already inherited narrow from an upstream pin. `Match.value<const I>` carries it because the literal union flows from a runtime value; `Match.type<I>` omits it because the union arrives as an explicit type argument with nothing to widen — the modifier set is a readable map of which parameters are inference entry points and which are type sinks.
- The omission failure surface is silent and surfaces downstream: an entrypoint whose result type indexes `keyof T` or branches on `T`'s literal value but whose author forgot `const` compiles at the declaration and fails only where a consumer's call passes an array or inline literal — the argument widens, the result type collapses to the fallback arm, and the diagnostic (or worse, the silently-widened result) lands at the call site far from the missing modifier. The defect is the signature's omitted `const`; the symptom is a consumer's lost literals, so the repair is read at the entrypoint, never the call.
- The inverse defect is the precautionary `const` on a parameter no return type reads as a literal: it pins narrowness nothing consumes, adds no correctness, and signals a misread of where the pressure lies — so an entrypoint carries `const` exactly on the parameters whose literal value or positional structure a downstream type derives from, and a parameter whose result type is invariant under widening carries none, the modifier set a precise record of which inferences the return type depends on.

[ERASURE_AND_SOURCE_SEAM_BOUND]:
- The inward pull's narrowness is the inferred type's, never the value's, so the entrypoint's `const` proves nothing the runtime enforces: the argument's object stays structurally mutable behind the deep-`readonly` type, and a value re-bound through a widened alias before the call loses the literals the parameter would have pulled — the pull narrows against the literal as authored at the source, so a prior `: string` annotation or a widened intermediate binding destroys the inference the `const` parameter can no longer recover. The lever recovers literals the call expression would lose, never literals a prior statement already erased.
- The pull stops at the source seam exactly as every literal-preservation lever does: a value entering from `unknown` carries no authored literal for the parameter to pin, so a `<const T>` entrypoint fed a decoded or external value narrows against the static type the signature declares, not the dynamic shape that arrived — the inward pull is the authoring-time gate on the source seam, and a vocabulary that must reject unknown input is admitted through decode past that seam, the two never swapped.
- The erasure is the reach edge that bounds the collapse: because `<const T>` leaves no runtime artifact, the pulled narrowness cannot persist past compilation, drive a runtime check, or survive serialization — so the entrypoint that pulls inference inward to clean its call sites pairs with a runtime owner wherever the narrowness must outlive the type checker, the `const` parameter owning the authoring-time shape and the decode owning the runtime invariant.
