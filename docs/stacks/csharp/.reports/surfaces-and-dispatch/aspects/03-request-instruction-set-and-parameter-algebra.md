# The Request Instruction Set And The Entrypoint's Parameter Algebra

[VERB_FAMILY_COLLAPSES_TO_A_CLOSED_INSTRUCTION_UNION]:
- A verb family — the sibling methods that name what to do rather than what they consume — collapses into one closed two-case recursive union: a pure terminal carrying a result and a suspension carrying the next instruction wrapped in an instruction functor. The instruction functor is the closed set of verbs reified as data; the entrypoint is one total arrow that interprets that data. The arrow is selected by the carrier the verbs target, not by a verb name, so the verb family stops being a method-name proliferation and becomes a value the interpreter folds.
- The suspension holds the instruction functor wrapped around the continuation, so each verb is a constructor of the functor and a program is a tree of stacked verbs. Lifting a verb is mapping the functor's payload through the pure constructor; sequencing verbs is the monad bind over the union. The verb grammar is declared once as the functor's case set, and every program is the two union constructors plus bind — no per-verb entrypoint exists at any layer, and the verb set is closed by the functor's own closure rather than by a sealed hierarchy the interpreter must exhaustively match.
- Swapping the interpreter is a natural transformation between two instruction functors lifted to a transformation between the two programs: the recursive hoist rewrites every suspension's functor through the transformation and leaves the pure terminals intact. The interpreter that actually runs a program is itself a natural transformation from the instruction functor to a target monad, so "run", "log", "mock", and "optimize" are all the same shape — one arrow from the verb functor to a carrier — never four methods. There is no first-class collapse-to-monad combinator; the interpreter is authored as the transformation and the program is consumed by recursing the closed union directly:

```csharp
public abstract record Program<F, A> : K<Program<F>, A> where F : Functor<F>;
public sealed record Return<F, A>(A Value)                  : Program<F, A> where F : Functor<F>;
public sealed record Step<F, A>(K<F, Program<F, A>> Next)   : Program<F, A> where F : Functor<F>;

static Program<F, A> lift<F, A>(K<F, A> verb) where F : Functor<F> => new Step<F, A>(verb.Map(pure));
static Program<G, A> reinterpret<F, G, A>(Program<F, A> p)
    where F : Functor<F>, Natural<F, G> where G : Functor<G> => hoist<F, G, A>(p);
```

[ARITY_COLLAPSES_AT_THE_CALL_BOUNDARY_NOT_IN_THE_SIGNATURE]:
- The collection-builder attribute names a static builder method and binds collection-expression literal syntax to it, so `[a, b, c]` materializes through the builder rather than through a constructor overload. Every immutable carrier carries this attribute, so one entrypoint typed against the carrier absorbs zero, one, and many at the call site through identical literal syntax — the arity is a property of the literal, never a `single` versus `batch` split in the signature. The builder receives a `ReadOnlySpan<T>`, so the literal path is allocation-lean and the entrypoint never sees the difference between a one-element and a thousand-element call.
- The builder method name is part of the dispatch identity and varies by carrier: the sequence, set, map, queue, and stack carriers route their literal through a range-builder, while the array carrier routes through a plain create-builder. The distinction is load-bearing only at the definition of a new carrier — consumers see one literal syntax — but it means the builder method is selected at the type level, the same place the comparer and the conversion policy are selected, so arity, equality, and conversion are one coherent definition-time surface on the carrier, not three runtime knobs.
- The first-success entrypoint absorbs arity through a `params` array of carriers and folds them left to right through the choice operator, returning the first inhabited result. One call with a `params` spread is the entire try-this-then-that cascade; the alternatives are not a switch over a count and not a chain of null checks, they are the operands of one fold whose operator is the carrier's choice. The constraint that the carrier implements the alternative trait is what makes the fold total over an empty spread — empty yields the trait's empty, never an exception — so the arity-zero call is a legal value, not a guarded edge.

[THE_UNION'S_PARAMETER_SURFACE_IS_A_DEFINITION_TIME_POLICY_SET]:
- Inbound and outbound conversion are independent axes with opposite defaults: admission into a case is implicit, extraction back to the underlying value is explicit. The asymmetry is the parameter algebra of the boundary — a raw value flows into the union without ceremony while pulling it back out forces a visible cast, so the union is a one-way valve by construction and the direction is a declaration-site policy, not a per-call decision. Setting the inbound axis to require an explicit cast converts the union into an admission-controlled surface where even construction is deliberate.
- Collapsing inbound conversion to none and forcing the constructor access modifier to private simultaneously removes the implicit conversion operator, so the only construction surface is a hand-written static factory that runs validation before materializing the case. The access modifier coupling is load-bearing: the same setting hides both the constructor and the generated conversion operator, which is precisely the lever that forces every case through one admitting factory. The constructor access enum is non-contiguous, encoding three distinct visibility levels rather than an ordered scale, so the choice is a discrete policy selection, not a permissiveness dial.
- Each case carries a member-name override and a nullable-reference flag, and the stateless flag derives the nullable flag for reference-typed cases because a stored discriminator with no payload reads back as the type's default, which for a class is null. A marker or signal case set stateless allocates no field and the accessor returns the type's default; recommending a struct for such a case eliminates the null path entirely. The single-backing-field flag collapses every case's storage into one object slot — an unambiguous win when all cases are reference types and a per-access box when any case is a struct — so the storage layout is a topology-driven definition-time choice, not a runtime concern the caller ever observes.
- A coarse-versus-fine dispatch overload is added at the declaration by naming a set of cases to treat as opaque atomic arms; both the descending and the stop-at overloads target the same generated type. Resolution granularity is therefore an overload the caller selects per call site, never a depth flag threaded through the parameters and never a second flattened family — the union exposes both granularities and the call picks one by which overload it binds.

[KNOB_TAILS_COLLAPSE_INTO_POLICY_VALUES_AND_TYPE_PARAMETERS]:
- A trailing set of boolean and enum knobs that the body re-derives into behavior collapses into one domain value that carries the behavior with it — a smart-enum row whose constructor wires delegate fields, a union case, or a frozen policy table keyed by the vocabulary that selects it. The knob set is the rejected form precisely because its combinations are a derived computation the implementation must reconstruct on every call; the policy value is the reconstruction done once at definition and selected by name. The behavior rows live with the vocabulary, so adding a behavior is adding a row, never threading a new flag through every entrypoint that consumes the policy.
- Optional context enters as the absence carrier or a runtime record, never as a nullable flag tail, and the default derives from the policy owner rather than from a literal at the call site. A parameter that re-describes the input — a count, a mode, or a batch flag standing beside the value it counts or modes — is arity or a discriminant smuggled back into a knob, because the program already holds that fact in the value's shape; the recoverable discriminant must come from the value, so the re-describing parameter is deleted and the entrypoint reads the shape.
- The validation-error type is a type parameter constrained to a static-abstract factory, so naming a custom error type in the owner declaration replaces the default across the entire generated admission surface at once — the factory's failure slot, the parsing rail, the serialization round-trip. The error is a typed parameter of the owner, not a string the body formats, so every rejection across the owner's whole surface speaks the same typed failure, and pairing that owner with an accumulating carrier merges those typed errors through their semigroup rather than concatenating strings.
- The factory input type is itself a type parameter carrying an anti-constraint that permits but does not require a ref struct, so a span or any custom ref-struct shape is a legal factory discriminant. Naming a span input routes the factory through the zero-allocation parse path and emits a different generated factory interface than a heap input would; the input type is therefore a definition-time selection between two generated admission surfaces, and the entrypoint that admits a transient key never allocates it. The serialization-framework selector is a flags enum, so one owner routes each framework through a different factory by stacking factory attributes — a span factory for one serializer and a string factory for another — selected by flag, never by a per-serializer method.

[CAPABILITY_DISPATCH_KEYS_OFF_A_TYPE_PARAMETER_NOT_A_METHOD_FAMILY]:
- An effect entrypoint discriminates capability by its environment type parameter through a static-abstract accessor trait: the environment that has a capability exposes a static accessor yielding the capability inside the effect carrier, and a three-parameter disambiguating accessor caches that capability as a readonly static keyed by the carrier, the environment, and the value. The capability is read by naming the type, not by calling a named getter on a service, so a service surface that would have been a family of `get`-prefixed methods becomes one accessor selected by which capability type the call site requests. Adding a capability is adding a trait the environment implements, never a method on a service.
- The reader, state, and mutation capabilities are sibling traits over the same environment parameter: the readable trait projects and locally rewrites the environment, the stateful trait puts and modifies a state value and projects from it, and the mutation trait yields a lock-free atomic cell of the inner environment. The mutation trait exposing an atomic cell unifies capability dispatch with lock-free mutation — the same trait that selects the capability hands back the cell that mutates it — so a stateful native cache is one capability accessor plus an atomic swap, not a service method guarding a dictionary. Every capability keys off the one environment parameter, so a call site widens its requirements by intersecting trait constraints on that parameter, never by depending on a wider service interface.
- The keyless variant of the vocabulary attribute carries no key member, so the operator-generation mode that would emit key-typed comparison overloads collapses to the plain mode by construction — there is no key primitive to compare against. This is the precise discriminator between the two vocabulary forms: the keyed form is an admission surface because the key is the factory's input, while the keyless form is behavior-only — it dispatches and sweeps its items but admits nothing, because admission is parameterized over the key the keyless form deliberately omits. A vocabulary whose cases need no external identity therefore selects the keyless form and loses no dispatch power; one whose cases must be reconstructed from a wire or host primitive selects the keyed form and gains the whole factory, parsing, and serialization surface keyed on that primitive in the same declaration.
- The key member's equality and ordering law is injected at the type level by a comparer-accessor type named on the key, and a complex value object sets a single string-comparison mode for all its string members without per-member annotation. Two vocabularies over the same key primitive with different injected comparers are different dispatch surfaces — the resolution law is part of the surface's identity, fixed at definition, never a comparer argument the caller threads. The comparer is a type-parameterized accessor rather than a constructed instance, so the injection is reflection-free and resolved statically alongside the conversion, arity-builder, and error-type selections that together form the one definition-time policy surface of the keyed owner.
