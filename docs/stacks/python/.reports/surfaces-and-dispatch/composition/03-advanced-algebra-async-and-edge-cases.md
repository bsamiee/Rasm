# [ADVANCED_ALGEBRA_ASYNC_AND_EDGE_CASES]

[STARPIPE_AND_STARCOMPOSE_TUPLE_THREADING]:
- `starpipe(args, fn1, fn2, ...)` and `starcompose(fn1, fn2, ...)` thread tuple arguments across every step: each intermediate function must return a tuple that is unpacked as positional args for the next. `starpipe((a, b), fn)` calls `fn(a, b)`. `starpipe((a, b), fn, gn)` calls `gn(*fn(a, b))`. The intermediate tuple arity must match the next function's positional arity exactly — there is no partial-application bridge. These are the correct surfaces when a dispatch step produces a labelled pair `(value, metadata)` that the next step consumes positionally. Using `pipe` on the same steps flattens the tuple into a single arg and breaks the shape. [expression/core/pipe.py lines 250-266; expression/core/compose.py lines 177-194, 5.6.0, 2026-06-09]
- `starcompose` accumulates via `reduce(lambda fields, f: f(*fields), fns, source)` — the `source` must therefore be a tuple (or any unpackable) on the first call, and each subsequent `f` receives the prior return unpacked. A zero-arg `starcompose()` returns `lambda *args: args` (the identity over `starid`). This is load-bearing when the initial value is already a positional pair and you want to build the composed transform before applying it. [expression/core/compose.py lines 177-194, 5.6.0, 2026-06-09]
- `pipe2((a, b), fn)` applies `fn(a)(b)` — it expects the first function to be a **curried** two-argument function, not a two-positional-arg function. `pipe2` is for `curry_flip`-decorated functions applied over a two-value context; `starpipe` is for multi-positional functions. Mixing them produces a `TypeError` at call time with no static signal from the checker. [expression/core/pipe.py lines 186-188, 5.6.0, 2026-06-09]

[PIPE_OVERLOAD_CEILING_AND_COMPOSE_ESCAPE_HATCH]:
- `pipe` has typed overloads up to 9 transform arguments (value + fn1..fn9). The implementation `return compose(*fns)(value)` delegates to `compose`, which uses an untyped `reduce`. At 10+ steps the checker falls back to `Any` for the return; the escape is `compose(*step_list)(value)` with an explicit annotation on the variable, or splitting the pipeline into two named stages. The `PipeMixin.pipe` method has overloads only up to 6 transforms; beyond 6 the method form also widens to `Any`. [expression/core/pipe.py lines 42-156, 314-316; expression/core/compose.py lines 116-141, 5.6.0, 2026-06-09]
- `compose()` with zero args returns `identity` equivalent: `lambda source: reduce(lambda acc, f: f(acc), [], source)` which is `source` unchanged. This is the correct seed for a `fold`-built transform accumulator: `pipe(compose(), *accumulated_transforms)` applied one step at a time degrades to `compose(*accumulated)` — keep `compose(*transforms)` as the final form, not iterative piping. [expression/core/compose.py lines 136-141, 5.6.0, 2026-06-09]

[OPTION_MAP2_MODULE_LEVEL_ARGUMENT_ORDER_HAZARD]:
- The module-level `option.map2` is `@curry_flip(2)` over `(opt1, opt2, mapper)`. After decoration the call shape is `option.map2(mapper)(opt1)(opt2)`. The method form `opt1.map2(mapper, opt2)` takes `mapper` first then `other` — NOT `other` then `mapper`. These two surfaces have the same semantic intent but opposite first-argument positions. In particular, the method `map2` in `option.py` is declared as `def map2(self, mapper, other)` at line 100, while the module-level function curries `(opt1, opt2, mapper)`. Passing `opt2` as the mapper argument to the module-level form type-checks when both `Option` types unify; the silent swap is detectable only at runtime. [expression/core/option.py lines 100-111, 431-433, 5.6.0, 2026-06-09]
- `Result.map2` (method) is `def map2(self, other, mapper)` — `other` before `mapper`. This is the inverse of the module-level `result.map2` which is `@curry_flip(2)` over `(x, y, mapper)` producing `result.map2(mapper)(x)(y)`. The asymmetry between method and module-level argument order is a persistent trap: method forms receive `(other, mapper)` while module-level curried forms always receive `(mapper)` first after flip. [expression/core/result.py lines 108-122, 385-391, 5.6.0, 2026-06-09]

[EFFECT_OPTION_ZERO_AND_NOTHING_SHORT_CIRCUIT]:
- `effect.option` is `OptionBuilder`. Its `zero()` returns `Nothing`. When an `@effect.option()`-decorated generator body reaches `return None` (or falls off the end without returning), the builder calls `zero()` and the function returns `Nothing`. This is distinct from `@effect.result` where `zero()` raises `NotImplementedError` — a generator body that falls off the end inside `@effect.result` raises at runtime. [expression/effect/option.py lines 14-30; expression/effect/result.py lines 32-33, 5.6.0, 2026-06-09]
- `@effect.option()` short-circuits on `yield from Nothing` via `EffectError(Nothing)` in `Option.__iter__`. The `Nothing` singleton is compared by identity inside `EffectError`; it is the same object as the module-level `Nothing = Option[Any].Nothing()`. A freshly constructed `Option(none=None)` that is NOT the `Nothing` singleton also raises `EffectError` correctly because `__iter__` is defined on the `Option` class, not on the singleton instance — both paths short-circuit identically. [expression/core/option.py lines 282-287, 361-362, 5.6.0, 2026-06-09]

[TRY_BUILDER_AS_EXCEPTION_BOUNDARY]:
- `effect.try_` is `TryBuilder(ResultBuilder[_TSource, Exception])` — it is exactly `@effect.result` with the error type fixed to `Exception`. The `@effect.try_()` decorator produces `Callable[_P, Result[T, Exception]]` (i.e., `Try[T]`). Use this as the narrow boundary form when the entire generator body wraps third-party calls that may raise arbitrary exceptions; the error type is `Exception`, not a domain error, so `map_error` is needed at the seam to convert to the canonical domain error type before downstream composition. [expression/effect/result.py lines 45-47; expression/core/try_.py lines 15-42, 5.6.0, 2026-06-09]
- `extra.result.catch(exception=ExcType)` is a decorator that wraps a function returning `T` (or already returning `Result[T, E]`) into `Result[T, ExcType]`. The `_TError` bound is `bound=Exception`, so `ExcType` must be an `Exception` subclass. When the wrapped function already returns a `Result`, the decorator passes it through via `isinstance(out, Result)` check — this means `catch` can layer over a function that sometimes raises and sometimes returns a `Result` without double-wrapping. [expression/extra/result/catch.py lines 28-52, 5.6.0, 2026-06-09]
- `catch` has two call forms: as a decorator factory `@catch(exception=ValueError)` and as a direct wrapper `catch(fn, exception=ValueError)`. In the factory form the `exception` argument is keyword-only; positional usage `catch(ValueError)` matches the first `@overload` and wraps the argument as a function, not as an exception type — the result is a decorator expecting a callable. Calling `catch(ValueError)(fn)` correctly produces `fn_wrapped`. [expression/extra/result/catch.py lines 15-52, 5.6.0, 2026-06-09]

[BLOCK_COLLECT_AS_MONAD_BIND]:
- `Block.collect(mapping)` where `mapping: A -> Block[B]` is the monadic `bind`/`flatMap` for `Block`. It applies `mapping` to each element and concatenates the results using a generator comprehension — `(y for x in mapped for y in x)`. This is the correct surface for expanding one `Block[A]` into `Block[B]` where each element maps to zero-or-more outputs; `map` followed by concatenation is always replaceable by one `collect` call. The module-level `block.collect` is `@curry_flip(1)`. [expression/collections/block.py lines 115-131, 570-586, 5.6.0, 2026-06-09]
- `Block.of_option(option)` returns a singleton `Block` for `Some` and `empty` for `Nothing`. This is the bridge from an `Option` rail to a `Block` context — use it inside `collect` when each element lookup may return `Option[B]`: `pipe(blk, block.collect(lambda a: block.of_option(lookup(a))))`. This is equivalent to `block.choose(lookup)` but explicit about the `Option -> Block` expansion; `block.choose` is the idiomatic shorthand. [expression/collections/block.py lines 822-827, 5.6.0, 2026-06-09]

[MAP_TRY_PICK_AS_FIRST_MATCH_COMBINATOR]:
- `Map.try_pick(chooser)` where `chooser: (K, V) -> Option[R]` iterates the AVL tree in key-sorted order and returns the first `Some(R)` via `maptree.try_pick`. This is a fold-short-circuit — it stops at the first non-`Nothing` result. It is the correct surface for a priority-based dispatch registry stored as a `Map`: given a `Map[Priority, Handler]`, `try_pick(lambda k, h: h(input))` returns the first handler result that is `Some`, in ascending priority order. No equivalent short-circuit combinator exists on `Block`; only `Block.choose` collects all `Some` values. [expression/collections/map.py lines 163-164; expression/collections/maptree.py (try_pick implementation), 5.6.0, 2026-06-09]
- `Map.partition(predicate)` splits an `Map[K, V]` into `(Map[K, V], Map[K, V])` via `maptree.partition` — truthy keys first, falsy second. Both halves are `Map` instances preserving key-sorted order. This is the correct surface for splitting a `Map`-backed registry into active and inactive handlers, or into two dispatch tables for different execution contexts, before composing downstream pipelines over each half separately. [expression/collections/map.py lines 117-119, 5.6.0, 2026-06-09]

[TAILREC_AND_TAILREC_ASYNC_AS_FIXPOINT_SURFACE]:
- `tailrec(fn)` and `tailrec_async(fn)` in `expression.core.fn` implement a trampoline. `fn` must return `TailCallResult[T, P]` which is `T | TailCall[P]`. Returning `TailCall(*args, **kw)` continues the trampoline; returning a plain `T` terminates. This is the correct surface for recursive algorithms that cannot be expressed as `block.unfold` (because each step is not a `Option[tuple[A, State]]` but a conditional dispatch over multiple arities). Use `tailrec` instead of recursive `@effect.result` nesting, which builds an O(N) chain of `Builder.bind` closures on the heap. [expression/core/fn.py lines 14-68, 5.6.0, 2026-06-09]
- `tailrec_async` trampolines an `async def` function returning `Awaitable[TailCallResult[T, P]]`. Each `TailCall` iteration `await`s the next call; no stack frames accumulate. The trampoline does NOT compose with `@effect.result` — the outer async function must return a plain value or `TailCall`, not a `Result`; wrap the final result in `Ok` after termination. [expression/core/fn.py lines 53-68, 5.6.0, 2026-06-09]

[ASYNC_BOUNDARY_COMPOSITION_CONSTRAINTS]:
- `@effect.result[T, E]()` is synchronous only. The decorated generator must be a regular `def`, not `async def`. Applying `@effect.result` to an `async def` produces a wrapper whose body is an async generator; `gen.send(value)` on an async generator raises `AttributeError` at the first `_send` call because async generators do not have a synchronous `send`. There is no `@effect.aresult` or async equivalent in `expression` 5.6.0. [expression/core/builder.py lines 48-69; expression/effect/result.py lines 15-43, 5.6.0, 2026-06-09]
- `extra.result.pipeline` and `extra.option.pipeline` are both synchronous Kleisli composers (plain `def` steps only). An async dispatch surface returning `Result[T, E]` cannot be composed directly with `pipeline`; each step must be `def` returning `Result`. The pattern for async composition is: run each async step with `await`, collect results into `Result`, then compose sync post-processing steps via `pipe` or `pipeline`. [expression/extra/result/pipeline.py lines 71-96; expression/extra/option/pipeline.py lines 68-92, 5.6.0, 2026-06-09]
- `expression.core.aiotools` (NOT `expression.system.aiotools`) provides `from_continuations`, `start`, `start_immediate`, and `run_synchronously` — all asyncio-native, no anyio integration. `CancellationToken` / `CancellationTokenSource` in `expression.system` are thread-based (using `RLock` and a dict of callbacks), not tied to asyncio task cancellation. Bridging them to an `anyio` cancel scope requires calling `CancellationTokenSource.cancel()` from inside an `anyio.CancelScope`'s cleanup, not the reverse. [expression/core/aiotools.py lines 32-127; expression/system/cancellation.py lines 10-113, 5.6.0, 2026-06-09]

[MAILBOXPROCESSOR_AS_ASYNC_MESSAGE_DISPATCH_SURFACE]:
- `MailboxProcessor[Msg]` in `expression.core.mailbox` is an asyncio-backed actor: `post(msg)` queues synchronously; `receive()` is `async def` that resolves the next message via `from_continuations`; `post_and_async_reply(build_message)` returns an `Awaitable[Reply]`. `MailboxProcessor.start(body, token)` bootstraps an async loop body `async def body(mb: MailboxProcessor[Msg])` that calls `await mb.receive()` repeatedly. This is the correct surface for serializing access to mutable state across an async dispatch surface — the mailbox owns the state, the body handler applies commands and sends replies, and all external callers use `post`/`post_and_async_reply`. [expression/core/mailbox.py lines 38-165, 5.6.0, 2026-06-09]
- `MailboxProcessor.receive()` may only be called once at a time; a second concurrent `receive()` call raises `Exception("Receive can only be called once!")`. The loop body must `await mb.receive()` at the top of every iteration and must not call it from parallel tasks. The `continuation` field stores only the single pending waiter; concurrent receipt requires wrapping in a queue or fan-out at the call site. [expression/core/mailbox.py lines 109-132, 5.6.0, 2026-06-09]

[DISPOSABLE_AND_ASYNC_DISPOSABLE_AS_RESOURCE_COMPOSITION]:
- `Disposable.create(action)` returns an `AnonymousDisposable` that is idempotent — calling `dispose()` twice is safe because `_is_disposed` is checked under `RLock`. `AsyncDisposable.composite(*disposables)` creates an `AsyncCompositeDisposable` that disposes each member sequentially via `await`. Neither has a `using`-style context combinator that produces `Result`; wrapping in `async with` and catching `ObjectDisposedException` at the boundary converts disposal errors to the `Error` track. [expression/system/disposable.py lines 46-144, 5.6.0, 2026-06-09]

[TAGGED_UNION_CONSTRUCTION_AND_FROZEN_HASH_INVARIANT]:
- `@tagged_union` transforms the decorated class into a dataclass with `init=False`; it then installs a custom `__init__` that accepts exactly one keyword argument matching a declared field name. Passing `tag=` explicitly is accepted but validated: `match tag or name, name: case str(tag), name if tag == name` — passing `tag="other_field"` with `other_field=value` raises `TypeError("Tag other_field does not match case name other_field")`. The `tag` kwarg is redundant for construction and serves only round-trip pickling via `__getstate__`/`__setstate__`. [expression/core/tagged_union.py lines 56-78, 5.6.0, 2026-06-09]
- `@tagged_union(frozen=True)` generates `__hash__` as `hash((cls.__name__, self.tag, getattr(self, self.tag)))`. The inner value must itself be hashable; a `@tagged_union` case holding a `Block` or `list` raises `TypeError` at hash time. `@tagged_union(frozen=True)` is usable as a `Map` key only when every case payload is itself hashable. For `Option` and `Result`, which are also `@tagged_union(frozen=True)`, this means `Option[Block[T]]` is not hashable even though `Option[str]` is. [expression/core/tagged_union.py lines 93-100; expression/core/option.py line 42, 5.6.0, 2026-06-09]
- `@tagged_union` sets `cls.__match_args__` to `field_names` — the tuple of all declared case field names. Structural pattern matching with positional patterns therefore matches on the first field name, not on `tag`. `case MyUnion(value):` (positional) binds the first case field regardless of which case is active. Always use keyword patterns `case MyUnion(tag="x", x=v):` for unambiguous case discrimination. [expression/core/tagged_union.py lines 135-136, 5.6.0, 2026-06-09]

[PYDANTIC_INTEGRATION_SCHEMA_FOR_OPTION_AND_RESULT]:
- `Option` and `Result` both implement `__get_pydantic_core_schema__` as classmethods. For `Option[T]`, JSON deserialization treats `null` as `Nothing` and any other value as `Some` via `lambda data: cls(some=data) if data is not None else cls(none=data)`. For `Result[T, E]`, JSON deserialization expects `{"tag": "ok", "ok": value}` or `{"tag": "error", "error": value}` — the `tag` field is present in the typed dict schema. Direct `result.dict()` produces `{"tag": "ok", "ok": ...}`, not the bare value. When a Pydantic model field is `Option[T]`, the JSON wire format is the unwrapped value (or null); when a field is `Result[T, E]`, the JSON wire format includes the tag envelope — these differ and must not be conflated at egress serialization. [expression/core/option.py lines 299-355; expression/core/result.py lines 268-333, 5.6.0, 2026-06-09]
- `Option.to_optional()` / `option.to_optional()` returns `T | None` for use at JSON/wire egress boundaries — it is the correct projection from the `Option` rail to a nullable Python value. `Result.default_with(getter)` accepts `getter: Callable[[E], T]`, making it the correct projection from the `Result` rail to a plain value at egress. Neither of these projections should appear inside domain logic; they are boundary-exit combinators only. [expression/core/option.py lines 227-233; expression/core/result.py lines 84-91, 5.6.0, 2026-06-09]

[CURRY_VS_CURRY_FLIP_ARITY_SEMANTICS]:
- `@curry(N)` curries N+1 leading arguments left-to-right: `@curry(1)` on `def f(a, b, *P)` produces `f(a)(b, *P)`. `@curry_flip(N)` defers N leading arguments to the end: `@curry_flip(1)` on `def f(source, mapper)` produces a function that takes `(mapper)` first, returning a closure that accepts `(source)`. The `N` in `curry` counts leading args to curry; the `N` in `curry_flip` also counts leading args to defer — both accept `Literal[0, 1, 2, 3, 4]` only. Arity 0 is identity in both decorators: `@curry(0)` and `@curry_flip(0)` both return the function unchanged. [expression/core/curry.py lines 88-109, 174-205, 5.6.0, 2026-06-09]
- `curry_flip(N)` with N=2 over `def f(a, b, *P, **kw)` produces a function that first accepts `(*P, **kw)` (trailing args), then returns a two-level curried closure `(a) -> (b) -> result`. This means `result.map2` as `@curry_flip(2)` over `(x, y, mapper)` produces `result.map2(mapper)(x)(y)` — both `x` and `y` are curried individually in the order they appear in the source signature after the trailing args have been captured. Introducing a `**kwargs` default in the decorated function shifts the trailing capture incorrectly and is not supported. [expression/core/curry.py lines 174-205, 5.6.0, 2026-06-09]

[EFFECT_SEQ_BUILDER_BROKEN_BIND]:
- `effect.seq` is `SeqBuilder`. Its `bind` implementation is `for x in xs: return fn(x)` — it returns the result of `fn` for the FIRST element only, discarding all subsequent elements. This is not a faithful sequence monad bind. `effect.seq` is suitable only for generators that yield exactly one value or zero values (the `zero()` returns `seq.empty`). Using `@effect.seq()` on a multi-yield generator silently processes only the first yielded element. For multi-element sequence effects, use `block.collect` or an explicit list comprehension outside the effect. [expression/effect/seq.py lines 12-29, 5.6.0, 2026-06-09]

[RESULT_FILTER_AND_FILTER_WITH_ASYMMETRY]:
- `Result.filter(predicate, default)` returns the original `Result` unchanged if it is already `Error`; it only applies the predicate to `Ok` values. If the predicate fails on an `Ok`, it returns `Error(default)` where `default` is an already-evaluated value. The return type widens to `Result[T, E | DefaultE]` because `default` may have a different type than the original error. `Result.filter_with(predicate, default_fn)` where `default_fn: T -> E` keeps the error type uniform because the function must return the same `E`. Use `filter_with` when the domain requires the ok value as context for the error; `filter` is for simple guard conditions with a static error sentinel. [expression/core/result.py lines 156-188, 5.6.0, 2026-06-09]

[RESULT_SWAP_AND_MERGE_AS_CHANNEL_INVERTERS]:
- `Result.swap()` / `result.swap()` inverts ok and error channels: `Ok(v)` becomes `Error(v)` and `Error(e)` becomes `Ok(e)`. This is the correct combinator when a downstream surface treats the error channel as the primary output (e.g., extracting error details from a validation that is expected to fail) and you need to map over it using standard `map`/`bind`. After `swap`, apply `map`/`bind` on the now-ok error value, then `swap` back. [expression/core/result.py lines 204-210, 439-441, 5.6.0, 2026-06-09]
- `Result.merge()` / `result.merge()` is constrained to `Result[T, T]` (same type on both channels) and returns `T` unconditionally via `default_with(lambda x: x)`. It is the correct terminal projection when a pipeline intentionally produces the same type on both rails (e.g., a validated string that is either the input or a normalized default) and you need a plain value at egress. The method-level type constraint enforces `Result[_TSource, _TSource]` statically. [expression/core/result.py lines 222-227, 457-458, 5.6.0, 2026-06-09]
