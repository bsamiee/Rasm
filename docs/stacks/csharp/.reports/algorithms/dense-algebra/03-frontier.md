# Dense Algebra — Frontier

[SVD_W_MUTATION_HAZARD]:
- `Svd<T>.W` is a `Lazy<Matrix<T>>` backed by `ComputeW()`, which allocates a fresh m×n `DenseMatrix` and writes `S[i]` onto the diagonal. `PseudoInverse()` calls `Matrix.PseudoInverse()`, which first mutates `S` directly via `s[i] = (s[i] < threshold) ? 0.0 : (1.0 / s[i])`, then calls `w.SetDiagonal(s)` on the lazy-cached W. Both mutations are permanent. A second `PseudoInverse()` call on the same handle produces a corrupted result because `S` already holds inverted or zeroed values. `Rank` reads from `S` and also returns corrupted results. The handle must be treated as consumed after a single `PseudoInverse()` call; never retain an SVD handle across pseudo-inverse calls.

[QR_ISFULRANK_THRESHOLD_VS_SVD]:
- `QR.IsFullRank` uses an absolute threshold `|R[i,i]| < 10 · 2⁻⁵³`, while `Svd.Rank` uses a relative threshold `|S[i]| > σ_max.EpsilonOf() · max(m,n)`. For a design matrix with R diagonal entry ~1e-8 — legitimately rank-deficient relative to the problem scale but far above machine epsilon in absolute terms — `QR.IsFullRank` returns `true` while `Svd.Rank` correctly reports rank deficiency. When the full-rank gate controls whether a solution is accepted, prefer the SVD rank evidence for ill-conditioned problems; the QR `IsFullRank` gate is reliable only for problems that are either exactly rank-deficient or well-conditioned.

[SVD_REGRESSION_VECTORS_DEFAULT]:
- `MultipleRegression.Svd(x, y)` calls `x.Svd()` with the default `computeVectors: true`, allocating a full m×m U and n×n VT. `SvdSolveFactored` in the managed provider iterates over U column-by-column for the Uᵀb step, so the full U is required for correctness under the current implementation. There is no thin-SVD solve path. For a 5000×20 design matrix, the full m×m U allocates ~200MB of working storage (25M doubles). When SVD regression on tall matrices is required, MKL provides a thin SVD via its DGESDD routine; without MKL the full allocation is unavoidable.

[CHOLESKY_SOLVE_INPUT_OVERWRITE]:
- `DenseCholesky.Solve(input, result)` copies `input.Values` into `result.Values` via `Buffer.BlockCopy` before calling `CholeskySolveFactored`. The provider then overwrites `result.Values` in place with the solution. When the caller passes the same instance for both `input` and `result`, or when `result` wraps the same `double[]` as `input` via the aliasing hazard, the copy is a no-op and the result is correct. However, any `result` matrix pre-loaded with a prior solution expecting additive update is silently overwritten — the solve is unconditionally destructive to `result.Values` before the provider call. `DenseLU.Solve` follows the same `Buffer.BlockCopy(input → result)` then in-place pattern. `DenseQR.Solve` does not copy — it passes both Q, FullR, Tau, and the input array directly to `QRSolveFactored`; the result buffer is written directly without preserving the input.

[GRAM_SCHMIDT_EXACT_ZERO_ONLY]:
- `DenseGramSchmidt.Factorize` throws `ArgumentException("Matrix must not be rank deficient.")` only when a column norm is `== 0.0` exactly — the comparison is `if (num == 0.0)`. A near-zero column norm (floating-point cancellation without bit-exact zero) passes the check, causes division by near-zero, and populates Q and R with `NaN`/`Inf` without any exception. The returned handle has an R matrix with `NaN` entries, and `IsFullRank` subsequently returns `true` because `Math.Abs(NaN).AlmostEqual(0.0)` is `false`. The factorization-succeeded-but-result-is-corrupt failure is undetectable without an explicit all-finite check on `((DenseMatrix)gs.Q).Values` and `((DenseMatrix)gs.FullR).Values` after construction.

[LU_INVERSE_ALLOCATION_PATTERN]:
- `DenseLU.Inverse()` clones `Factors` into a new `double[n²]` array, then calls `LinearAlgebraControl.Provider.LUInverseFactored` on that clone. The managed `LUInverseFactored` allocates a second `double[n²]` identity array, calls `LUSolveFactored` to solve n right-hand sides simultaneously into it, then copies the result back over the clone. Total allocations: one n² clone in `Inverse()` plus one n² identity in `LUInverseFactored`. For n ≥ 100, both arrays cross the LOH threshold (~85KB). The correct pattern for matrix inversion in a hot loop is to retain the `DenseLU` handle and call `Solve` on an identity matrix with pre-allocated result and right-hand side buffers reused across calls.

[MANAGED_GEMM_ROW_COLUMN_EXTRACT_COST]:
- The managed `MatrixMultiplyWithUpdate` kernel pre-extracts all columns of B into `double[][] columnDataB` before the multiply loop: each column of B is extracted by `GetColumn` (a contiguous `Data[c*n .. (c+1)*n-1]` slice copy) into a new `double[rowsB]` heap array. This pre-extraction doubles memory traffic for B and allocates `columnsB` separate arrays before any computation begins. For an m × k × n multiply, `n` arrays of length `k` are allocated on the heap. Under MKL this code path is bypassed entirely — `MatrixMultiplyWithUpdate` calls BLAS DGEMM directly.
- The parallelization gate uses `rowsA + columnsB + columnsA < Control.ParallelizeOrder` — a linear sum of three dimensions, not their product. For a 10×1000×10 GEMM the sum is 1020, above the default of 64, so parallel dispatch fires for a problem too small to benefit from it. For a 64×64×64 GEMM the sum is 192, also above threshold. Setting `Control.MaxDegreeOfParallelism = 1` disables parallel dispatch and eliminates the `CommonParallel.For` overhead for small n.

[SYMMETRIC_FORCE_ADMISSION_PATTERN]:
- The canonical symmetric-force expression `matrix.Add(matrix.Transpose()).Multiply(0.5)` allocates three dense matrices (transpose, add result, scalar multiply result) before the EVD call. At n = 500, the three-allocation chain produces ~2.25MB of transient working storage. The in-place alternative iterates the `Values` array and averages symmetric pairs `(values[i*n+j] + values[j*n+i]) * 0.5` for `j < i`, writing both positions — zero extra allocations and a single pass exploiting column-major layout to write both triangles simultaneously. `TensorPrimitives` has no direct symmetric-force primitive.

[PSEUDOINVERSE_THRESHOLD_VS_SOLVE_INCONSISTENCY]:
- The `PseudoInverse()` threshold is `max(m, n) · σ_max · Precision.DoublePrecision` where `Precision.DoublePrecision = 2⁻⁵³`. The `Svd.Rank` threshold is `σ_max.EpsilonOf() · max(m, n)` where `EpsilonOf(1.0) = 2⁻⁵²` — exactly twice `Precision.DoublePrecision`. A singular value that `PseudoInverse` retains (above `σ_max · 2⁻⁵³ · max(m,n)`) may still cause `Rank` to report one fewer non-zero singular value (if the value is below `σ_max · 2⁻⁵² · max(m,n)`). The two surfaces disagree on rank-boundary singular values even when reading from the same `S` vector on the same handle. Because `PseudoInverse` also permanently mutates `S`, `Rank` and `ConditionNumber` become unreliable after `PseudoInverse` is called regardless of which threshold is in use.

[ISFINITE_ADMISSION_GATE]:
- `TensorPrimitives.IsFiniteAll<double>(new ReadOnlySpan<double>(matrix.Values))` maps to a vectorized SIMD predicate over the entire flat `Values` array in a single call, replacing scalar LINQ `All(double.IsFinite)` that would use interface dispatch per element. For a 100×100 matrix (10,000 doubles, 80KB), the vectorized form is roughly 20–50× faster depending on SIMD width — critical for admission gates on matrices updated at high frequency. `IsNaNAny<T>` and `IsInfinityAny<T>` are available as early-exit alternatives that distinguish the failure condition in the evidence receipt.
