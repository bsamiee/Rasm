# Structural Edit And Rank Modification

[ELIMINATION_TREE_AS_EDIT_DEPENDENCY_GRAPH]:
- The elimination tree `parent[]` computed in symbolic analysis is the single dependency structure that governs both numeric factorization fill and every post-factorization structural edit. `EliminationTree` derives `parent[j]` as the smallest row index whose column reaches `j` during the upper-triangular pattern scan; the result is a forest whose root columns have `parent == -1`. Factorization fills each column `j` by walking the reach set returned from `EtreeReach(A, j, parent, ...)`, and rank-1 modification walks the identical tree from the edit's lowest touched row to the root. Holding `parent[]` beside the factor is holding the only map that says which factor columns any given edit can perturb.
- `EtreeReach` returns the topologically ordered set of columns that the `k`-th column's nonzeros reach through the tree, using a sign-flip marker scheme on the work array `w` so the pattern carrier needs no separate visited set: `w[k] = -w[k] - 2` marks, the ancestor walk pushes onto the stack `s` until it hits an already-marked node, then the suffix of `s` is reversed into topological order and the marks are cleared on exit. The same call drives both the symbolic column-count pass and the numeric fill pass — the symbolic phase and the numeric phase share one reachability kernel parameterized only by whether values are present.
- The rank-1 update reads no part of the original operator and no symbolic record beyond `parent[]`; it mutates only the stored factor `L` in place along one root-bound tree path. The structural locality is exact: an edit whose update vector first touches row `r` perturbs precisely the columns on the path `r → parent[r] → … → root`, never a sibling subtree. For an edit confined to a deep leaf of a wide tree this touches `O(tree-depth)` columns rather than `O(n)`, which is the whole reason a cached factorization survives a localized structural change.

[RANK_ONE_MODIFICATION_AS_DELTA_OPERATOR]:
- One private `UpDown(sigma, w)` body serves both directions: `Update` passes `sigma = +1`, `Downdate` passes `sigma = −1`, and the sign threads through the rotation coefficients so a single dispatch covers `A ± w wᵀ`. The update vector `w` is a single-column compressed storage; row order inside that column is irrelevant because the body gathers `w` into a dense length-`n` scratch indexed by row and finds the entry column via `Math.Min` over the raw row indices, so an unsorted or duplicated update column is admitted without a pre-sort.
- The walk is a Givens-style rescaling of each touched `L` column: at column `j` on the path it forms the ratio `array[j] / L_diag[j]`, advances a running scalar `β² ← β² + sigma·ratio²`, takes `√` of it, derives a multiply factor and an axpy factor from `(old β, new β)`, rescales the diagonal, then sweeps the column suffix applying `array[row] −= ratio · L[row]` and `L[row] ← mul·L[row] + add·(value before or after that subtraction, branch on sign)`. The axpy term reads the column entry's value before the subtraction when adding the rank-1 term and after the subtraction when removing it — the one structural difference between an up and a down step is which side of that elementwise update feeds the factor rescale. The diagonal carries a second asymmetry: the add direction folds the gathered entry into the diagonal rescale, the remove direction folds in zero.
- Loss of definiteness is detected inline and ends the walk: when the running `β²` goes non-positive the loop breaks and the method returns `false`, signalling the updated matrix is no longer factorable in this form. There is no typed failure, no partial-rollback — on a `false` return the factor is left in whatever state the partial walk produced, so a downdate that breaks definiteness corrupts `L` and the operator value must treat `false` as "discard this factor, refactor from scratch," never "retry the edit."
- The Hermitian variant carries an extra per-column phase normalization: after rescaling the diagonal it multiplies the whole column by `Abs(diag)/diag`, which forces the diagonal real-positive and keeps the factor a valid Cholesky factor of a Hermitian system. The conjugate enters the coefficient as `sigma · conj(ratio)` and the definiteness scalar uses `(ratio · conj(ratio)).Real`. The real and complex update kernels are the same path-walk with conjugation and phase folded in — one algorithm, two carrier instantiations, not two implementations.

[EDIT_CAPABILITY_IS_KIND_ASYMMETRIC]:
- Rank-1 modification is a capability of one factorization kind only. The symmetric-positive-definite factor exposes `Update`/`Downdate`; the indefinite symmetric factor exposes none — it carries only solve, create, and factorize, and its diagonal factor is a private field with no accessor, so neither a rank update nor an inertia read is reachable through its surface. The general unsymmetric factor likewise has no update path; it carries a transpose-solve the symmetric factors lack. Edit capability and transpose capability are disjoint across the family, and a request surface that must both modify rank and solve the transpose cannot rest on any shared abstraction — each capability lives on a different concrete kind.
- The consequence for a factored-operator value is a capability discriminant that is a property of the kind, not a caller flag: only the SPD kind answers "can this edit be applied as a rank-1 delta," and for every other kind a structural edit forces full reconstruction. A pin, a release, or a single-coefficient change on an SPD operator is `O(tree-path)`; the identical edit on an indefinite or unsymmetric operator is a complete refactor. The edit-cost class is therefore recoverable from the kind alone and belongs in the operator value beside the kind discriminant.
- Because the symmetric factors read only the upper triangle, an SPD rank update that preserves symmetry is well-defined, but an asymmetric perturbation expressed as a single update column silently applies its implied symmetric closure — the factor cannot represent a one-sided edit. The edit algebra is closed over symmetric rank-1 terms and nothing else; a non-symmetric structural change is outside the update surface entirely and routes to reconstruction regardless of magnitude.

[IN_PLACE_COMPACTION_ALGEBRA]:
- Structural removal collapses to one predicate-driven compaction primitive: a keep pass walks every column, copies surviving `(row, value)` pairs down to a write cursor, rewrites each column pointer to the cursor as it passes, and returns the survivor count. Numeric-tolerance dropping is the same primitive with a generated predicate — an exact-zero test below a zero tolerance, an absolute-magnitude test above a positive tolerance — so there is no separate drop path, only a parameterized keep. A pin (retain only the diagonal of a row and column), a magnitude prune, and a zero clean are three predicate values over one compaction, not three functions.
- The keep walk shares a structural-merge cousin built on a marker-array scatter: column-wise accumulation marks each touched row in a length-`n` work array against a per-column mark, writing fresh contributions and folding repeats by index, so combining two patterns or summing duplicates is one `O(nnz)` scatter per column rather than a sort-merge. Both additive combination and duplicate summation reduce to this scatter; the only difference is whether the second operand is the same matrix's later entries or a separate matrix.
- A process-wide trim flag turns every compaction into an exact-fit resize: with it enabled, keep, additive combine, and product all shrink the index and value arrays to exactly the survivor count immediately, an `O(nnz)` copy per structural operation. The flag is static and distinct per carrier instantiation, so disabling it for one element type leaves the other element type still trimming, and disabling it at all mutates shared state for every matrix of that type on every thread. A progressive structural-edit sequence wants the trim suppressed across the whole batch and restored once at the end — confined to a single-threaded phase or guarded — so the cost is one final resize, not one resize per edit.

[BATCH_VERSUS_SEQUENTIAL_EDIT_COST]:
- A single structural insertion into compressed storage shifts the suffix of the index and value arrays and bumps every column pointer above the insertion site — `O(nnz)` per insertion. Sequential point edits therefore cost `O(edits · nnz)`; the dominating alternative is to accumulate all modifications into a triplet buffer and re-convert once, paying one sort-and-deduplicate pass for the whole batch. The decision is structural: more than one structural insertion crosses the break-even and should route through a single re-conversion, not a sequence of in-place inserts.
- Additive combine pre-sizes its result buffer to `nnz(A) + nnz(B)`, the exact upper bound for a sum, so two-matrix overlay never reallocates mid-pass but leaves exact-cancellation zeros resident until a tolerance keep removes them. Sparse product pre-sizes to the same `nnz(A) + nnz(B)`, which is only a lower bound for a product whose fill can exceed it, so the product's result buffer may have to grow during its fill pass — a structural product is the one merge primitive that can reallocate inside the operation. A structural-edit pipeline that expresses a constraint as a product carries that latent growth cost; one expressed as a keep-then-combine does not.
- The advertised parallel product is a thin forwarding shim to the serial product and performs no concurrent work; relying on it for throughput on a large structural product is a silent no-op, and a genuinely parallel structural merge must be hand-driven over column partitions with private scatter buffers. Treating the parallel entry as load-bearing is a false economy the surface invites.

[EDIT_OPERATOR_COLLAPSE]:
- Every structural-edit dialect — pin a node, release a constraint, prune below tolerance, apply a rank-1 stiffness bump, clean cancellation residue — is one predicate-or-vector parameter over a fixed three-primitive substrate: predicate compaction for removal, marker scatter for additive merge, tree-path rank-1 walk for symmetric low-rank change. A separate function per dialect is the rejected shape; one edit entrypoint discriminating on an edit-descriptor value drives the right primitive, and the descriptor carries its own predicate, update column, or tolerance rather than the body re-deriving it from flags.
- The edit-cost class is derivable, not measured: a symmetric rank-`r` change on the SPD kind is `r` tree-path walks at `O(r · tree-depth)` with no symbolic rerun; a pattern-preserving value-only change is a numeric refactor that reuses the cached permutation and symbolic record but reruns the value sweep; a pattern change is a full reconstruction. The descriptor's shape plus the operator's kind fixes which of the three the edit costs, so a reuse policy reads the descriptor and the kind to choose between in-place delta, numeric-only refactor, and reconstruction before touching the factor.

```csharp
// The edit descriptor is the policy value; the operator's kind decides which primitive runs.
// Symmetric low-rank ⇒ tree-path delta on the cached factor (no symbolic rerun).
// Pattern-preserving value change ⇒ numeric refactor reusing the cached permutation.
// Pattern change ⇒ reconstruction. One entrypoint, no per-dialect functions.
[Union]
public partial interface IEdit
{
    Edit Pin(int node);                          // keep diagonal of row/col ⇒ predicate compaction
    Edit Prune(double tol);                      // |aij| ≤ tol ⇒ predicate compaction
    Edit Bump(int sign, double[] column);        // A ± w wᵀ ⇒ symmetric rank-1 tree-path walk
    Edit Revalue(ReadOnlyMemory<double> values); // same pattern, new values ⇒ numeric refactor
}

static Fin<FactoredOp<double>> Apply(FactoredOp<double> op, Edit e) => e switch
{
    Edit.Bump b when op.Kind is FactorKind.Spd =>
        op.RankOne(b.Sign, b.Column)                       // false ⇒ definiteness lost
            ? FinSucc(op)                                   //   ⇒ discard factor, reconstruct
            : Rebuild(op, b),
    Edit.Revalue r =>
        FinSucc(op with { Inner = Refactor(op.Permutation, op.Symbolic, r.Values) }),
    var structural =>                                       // Pin / Prune / pattern change
        Rebuild(op, structural),                            // keep-compact triplet, re-convert once
};
```
