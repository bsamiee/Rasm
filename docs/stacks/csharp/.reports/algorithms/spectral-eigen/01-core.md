# Spectral-Eigen Core Research

[ROUTE_SELECTION]:
- `Matrix<T>.Evd(Symmetricity)` is the single generic entry point for all dense eigensolution routes. `Symmetricity` is a four-case enum: `Unknown`, `Symmetric`, `Hermitian`, `Asymmetric`. Passing the known case skips the O(n²) `IsSymmetric()` check; passing `Unknown` pays that cost at every call.
- `DenseMatrix` overrides `Evd` to call `DenseEvd.Create`, which delegates to `LinearAlgebraControl.Provider.EigenDecomp` — a six-argument void method `(bool isSymmetric, int order, T[] matrix, T[] matrixEv, Complex[] vectorEv, T[] matrixD)`. Any provider wired through `Control.UseNativeMKL()` or `Control.UseNativeOpenBLAS()` substitutes a native LAPACK call behind this interface; the managed fallback is `ManagedLinearAlgebraProvider`. Non-dense matrix types (`SparseMatrix`, QR output) inherit from `Double.Matrix`, which overrides `Evd` with `UserEvd.Create` — the pure-managed EISPACK path. Materializing to `DenseMatrix` before calling `Evd` is required to reach the provider-backed path; calling `.Evd()` on a QR.Q result routes through `UserEvd` regardless of the active provider.
- The generalized symmetric-definite problem A·x = λ·B·x has no native MathNet route. The congruence reduction `L⁻¹·A·L⁻ᵀ` must be built manually: `B.Cholesky()` yields `Factor` (lower-triangular L); forward and back substitution via `Factor.Solve(A)` and its transpose solve produce the standard-form matrix; `Evd(Symmetricity.Symmetric)` solves the reduced problem; the back-transform is `L⁻ᵀ·Z` via `Factor.Transpose().Solve(Z)`. Cholesky throws `ArgumentException` with the message `"Matrix must be positive definite."` synchronously during `Create` if B is indefinite — this failure must be caught at the Fin boundary before Evd is attempted.
- MathNet has no sparse iterative eigensolver (no Lanczos, no block Krylov, no LOBPCG). The sparse iterative solvers (`BiCgStab`, `GpBiCg`, `MlkBiCgStab`, `TFQMR`) are linear system solvers only. Block-preconditioned eigeniteration must be hand-assembled on top of MathNet's dense `Evd` (for the Rayleigh-Ritz dense subproblem) and the sparse matrix-vector product (for the action of A on the current block).

[EVD_RECEIPT_SHAPE]:
- `Evd<T>` carries three payload properties: `EigenValues` (`Vector<Complex>`), `EigenVectors` (`Matrix<T>`), and `D` (`Matrix<T>`, block diagonal). For a real symmetric matrix, eigenvalues are real; `EigenValues[i].Imaginary == 0.0` for all i. The `Vector<Complex>` carrier is uniform across all four scalar types (double, float, Complex, Complex32) — the representation is not polymorphic by input symmetry class. To recover real eigenvalues: `evd.EigenValues.Map(c => c.Real)` returns `Vector<double>`; `evd.EigenValues.AsArray()` returns the raw backing `Complex[]` without a copy for dense vectors.
- `IsSymmetric` on `Evd<T>` is a bool, not an enum discriminant. It does not carry the original `Symmetricity` case — a Hermitian complex input sets `IsSymmetric = true` because `Symmetricity.Hermitian` branches into the same `flag = true` path. The receipt loses the original symmetry evidence.
- Eigenvalues are sorted ascending for symmetric inputs by the terminal selection sort in `SymmetricDiagonalize` (verified in `ManagedLinearAlgebraProvider`). The LAPACK/MKL path through `DenseEvd` follows the LAPACK `dsyev`/`zheev` contract, which also returns ascending order. Non-symmetric inputs carry eigenvalues in Schur-convergence order — no sort guarantee.
- `Evd<T>.Rank` uses `AlmostEqual(EigenValues[i], Complex.Zero)` with `DefaultDoubleAccuracy = DoublePrecision × 10 ≈ 1.11 × 10⁻¹⁵` as the absolute threshold. This is a fixed absolute test against zero, not a threshold relative to the spectral radius, making it unreliable as a numerical rank oracle for anything other than exactly-zero eigenvalues. Receipt-level rank must be recomputed with a caller-supplied threshold.
- `NonConvergenceException` (from `UserEvd` symmetric QR iteration exhausting 1000 steps) and `ArgumentException` (from `DenseEvd` MKL path or non-square validation) are both thrown by `Evd` construction. `Try<A>.Run()` catches all exceptions and converts via `Error.New(Exception)` to `Fin.Fail`. The boundary form is `new Try<EigenReceipt>(() => Fin.Succ(BuildReceipt(matrix.Evd(sym)))).Run()`. Both failure types collapse to a single `Fin.Fail(Error)` — the caller cannot distinguish convergence failure from argument violation without inspecting `Error.Inner` or `Error.IsExceptional`.

[REAL_COMPLEX_POLYMORPHISM_CEILING]:
- `System.Numerics.Complex` implements `IMultiplyOperators<Complex, Complex, Complex>` and `IMultiplicativeIdentity<Complex, Complex>` but does NOT implement `IMultiplyOperators<Complex, double, Complex>`. `double` does not implement `IMultiplyOperators<double, Complex, Complex>`. C# 14 generic math has no cross-type multiply interface for real-by-complex. A generic method constrained on `T : IMultiplyOperators<T, T, T>` cannot scale a `Vector<Complex>` by a real weight without an explicit cast or conversion site.
- The consequence for spectral filter kernels: a weight function `f : double → double` cannot be applied to a `Vector<Complex>` column block via a single generic Multiply overload. The minimal fix is a delegate bundle: `(Func<Complex, double, Complex> scaleComplex, Func<double, double, double> scaleReal)` where callers provide `(c, w) => c * w` and `(r, w) => r * w`. Collapsing to one generic kernel with `where T : IMultiplyOperators<T, TWt, T>` fails because the two-parameter form is not on the BCL hierarchy for these types.
- `TensorPrimitives.Dot<T>` requires `IAdditionOperators<T, T, T>`, `IAdditiveIdentity<T, T>`, `IMultiplyOperators<T, T, T>`, `IMultiplicativeIdentity<T, T>` — Complex satisfies all four (homogeneous), so `TensorPrimitives.Dot<Complex>` compiles. `TensorPrimitives.Norm<T>` requires `IRootFunctions<T>` — Complex does NOT implement `IRootFunctions<Complex>`, so `TensorPrimitives.Norm<Complex>` does not compile. For real-valued column norms, `TensorPrimitives.Norm<double>` is available; for complex column norms, use `Complex.Abs(c)` folded manually or `Vector<Complex>.L2Norm()` from MathNet.
- `Complex` is not `IFloatingPointIeee754<Complex>`, `INumber<Complex>`, or `IBinaryFloatingPoint<Complex>`, so no `TensorPrimitives.FusedMultiplyAdd<Complex>` or `TensorPrimitives.MultiplyAdd<Complex>` compile. The real-domain hot loop (`double` Gram-Schmidt columns) can use `TensorPrimitives.MultiplyAdd<double>` and `TensorPrimitives.Dot<double>` directly; the complex-domain equivalent requires a manual loop or explicit SIMD with `Vector<double>` pairs.

[LOBPCG_MECHANICS]:
- LOBPCG requires three stages per iteration: (1) orthonormalize the block X, (2) apply the operator A and optionally the preconditioner, (3) form the Rayleigh-Ritz dense subproblem over the stacked space [X | A·X | P].
- MathNet provides `Matrix<T>.QR(QRMethod.Thin)` for Householder orthogonalization, `Matrix<T>.GramSchmidt()` for modified Gram-Schmidt via `GramSchmidt<T> : QR<T>`. Both expose `.Q` (the orthonormal factor). For column-norm-floored rank collapse, `Matrix<T>.ColumnNorms(2.0)` returns a `Vector<double>` of L2 column norms; columns below the floor are dropped by extracting the surviving submatrix via `SubMatrix(0, n, surviving_start, surviving_count)`.
- Block stacking: `Matrix<T>.Append(right)` concatenates column-wise (n × k ++ n × k → n × 2k); `Matrix<T>.Stack(bottom)` row-wise. The LOBPCG subspace [X | R | P] is formed by chained `Append` calls. After Rayleigh-Ritz (`subspace.Evd(Symmetricity.Symmetric)`), the leading k eigenvectors of the small dense problem select the new X as `subspace.Multiply(V.SubMatrix(0, 3k, 0, k))`.
- Diagonal preconditioning: for a diagonal entry `a_ii`, the preconditioned correction is `(λ - a_ii)⁻¹ · r_i`. A degenerate diagonal (|λ - a_ii| < ε) must fall back to identity (correction = r_i) to prevent division by near-zero. This epsilon is algorithm-specific, not `Precision.DoublePrecision`, and must be supplied as a policy value.
- Previous-direction dropout: when a column's norm after orthogonalization falls below the floor, the P block is truncated. The dropout condition `colNorm < floor` is evaluated after MGS; the P block's column count contracts. The Rayleigh-Ritz dense problem then shrinks from 3k to (2k + p) where p is the surviving P column count.
- Re-symmetrization of the Rayleigh-Ritz matrix: `S = W^T · A · W` may be numerically asymmetric due to floating-point accumulation across block products. Explicit re-symmetrization `S <- (S + S^T) / 2` before passing to `Evd(Symmetricity.Symmetric)` prevents the managed path from re-checking `IsSymmetric()` or branching into the nonsymmetric Schur path. `Matrix<T>.TransposeThisAndMultiply(Matrix<T>)` (alias: W^T·A) exists; `Matrix<T>.ConjugateTransposeAndMultiply` handles the Hermitian case.

[SPECTRAL_FILTER_ALGEBRA]:
- The spectral filter algebra maps a graph signal through an eigenbasis: for eigenpairs `{(λ_i, v_i)}` the filtered output is `∑ᵢ f(λᵢ) · (vᵢᵀ · x) · vᵢ`. The filter function `f : double → double` selects the kernel family: heat `f(λ) = exp(-t·λ)`, wave `f(λ) = cos(t·√λ)`, biharmonic `f(λ) = λ²`, diffusion `f(λ) = (I + t·Δ)⁻¹` approx `f(λ) = 1/(1 + t·λ)`, commute-time `f(λ) = 1/√λ` (zero mode excluded), identity `f(λ) = 1`. These belong to a closed union — the filter case is a policy value, not a boolean flag.
- Zero-mode handling: for Laplacian-derived operators, λ₀ = 0 (or numerically near zero). Filters with 1/λ or 1/√λ diverge at λ₀. The zero-mode epsilon floor is domain-driven: for graph Laplacians normalized by degree, λ₁ > 0 after zero-mode exclusion. The weight at index i is `λᵢ < ε_zero ? 0.0 : f(λᵢ)` — set to zero, not clamped, because the zero mode carries the DC component and must be excluded, not preserved at an arbitrary value. The ε_zero value is a policy parameter, not a global constant.
- Bandwidth floor: filters composed as `f₁ ∘ f₂` must prove the combined weight sum is non-trivial. A filter application that produces a weight-sum below a bandwidth floor signals degenerate filtering — all input energy is in excluded modes. This is a diagnostic exit condition: the receipt carries `WeightSum` and the call site gates on it before interpreting the filtered output.
- Partial-monoid composition: two filters `f₁` and `f₂` compose pointwise as `f(λ) = f₁(λ) · f₂(λ)`. The identity element is `f(λ) = 1`. Composition preserves the closed union if both cases carry the same per-eigenvalue dispatch delegate — the composed filter is a new case with product weights, not a new recursive union.

[CONGRUENCE_REDUCTION]:
- The generalized problem A·x = λ·B·x with B SPD reduces to standard form via Cholesky factorization: `B = L·Lᵀ`, `C = L⁻¹·A·L⁻ᵀ`, `C·y = λ·y`, `x = L⁻ᵀ·y`. In MathNet: `var chol = B.Cholesky()` (throws if not SPD); `var Linv_A = chol.Factor.Transpose().Solve(A)` — this produces `L⁻ᵀ·A`; then `var C = chol.Factor.Solve(Linv_A.Transpose()).Transpose()` completes `C = L⁻¹·A·L⁻ᵀ`. Alternatively: `var C = chol.Solve(A).Multiply(chol.Factor.Inverse())` — but `.Inverse()` materializes a full n×n matrix and is O(n³); prefer the two-solve approach.
- Re-symmetrization after congruence: floating-point error in `L⁻¹·A·L⁻ᵀ` accumulates asymmetry. The receipt-level approach: compute `C_sym = (C + C^T) / 2` before `C_sym.Evd(Symmetricity.Symmetric)`. Without this step, the managed path runs `IsSymmetric()` on C and may fall through to the nonsymmetric Hessenberg path when accumulated asymmetry exceeds the IsSymmetric tolerance.
- Back-transform: given standard-form eigenvectors Y (columns of `evd.EigenVectors`), the generalized eigenvectors are `X = L⁻ᵀ·Y = chol.Factor.Transpose().Solve(Y)`. This is a column-wise triangular solve — `Cholesky.Factor.Transpose().Solve(Matrix)` applies the upper-triangular back-substitution in one call.
- Route name honesty: "generalized symmetric-definite" is a Cholesky-congruence standard-form reduction followed by a standard `Evd`, not a single compound primitive. The receipt should carry the route as a discriminated case, not collapse the Cholesky step into the eigenreceipt invisibly.

[RESIDUAL_EVIDENCE]:
- MathNet has no built-in residual computation for eigendecomposition. The residual `‖A·v - λ·v‖₂` must be computed independently: `var residual = A.Multiply(eigVec) - eigVec.Multiply(eigVal)` where eigVal is a scalar; then `residual.L2Norm()`. For block residuals, `A.Multiply(V) - V.Multiply(D)` where D is diagonal: use `Matrix<T>.Build.DenseDiagonal(n, n, i => eigenvalues[i])` to construct D from the receipt's eigenvalue array.
- An independently computed residual closes the evidence loop: the `Evd<T>` receipt carries `EigenValues` and `EigenVectors`, but gives no residual norm, no convergence count, and no numerical rank threshold. The receipt shape for numeric correctness must add: per-pair residual norms, the method variant (dense-symmetric / dense-general / dense-hermitian / congruence-reduced), and whether the eigenvalue sort is guaranteed (symmetric: yes; nonsymmetric: no).
- Exhaustion semantics: `NonConvergenceException` means the QR iteration reached 1000 steps without convergence. The partially-computed eigenpairs stored in `matrixEv` before the throw are **unusable** — they are intermediate Hessenberg/tridiagonal values, not converged eigenvectors. An exhaustion failure is distinct from a successful Evd with near-zero eigenvalues and from a domain error (non-square). The Fin.Fail error for exhaustion should carry a distinct code, not the same error as an ArgumentException from non-SPD input.

[CACHE_AND_TRUNCATION]:
- Over-compute-and-truncate: `matrix.Evd(Symmetricity.Symmetric)` always computes all n eigenvalues. For a spectral filter that uses only the k smallest, the full decomposition is computed then truncated: `evd.EigenValues.AsArray().Take(k)` and `evd.EigenVectors.SubMatrix(0, n, 0, k)`. The receipt rewritten to the truncated request carries `requested_k`, `computed_k = n`, and `cache_source = Full`. The full basis may be cached for re-use across multiple filter applications with different k.
- The cost for full Evd (MKL dsyev) is O(n³). For a problem where k ≪ n, shift-invert with sparse-direct factorization is nominally O(n·k²) per iteration, but MathNet provides no shift-invert path. The choice to over-compute-and-truncate should be explicit in the receipt: it is a deliberate trade of full dense cost against zero iterative infrastructure cost.
- Cache provenance: a cached eigenbasis computed at full resolution that is re-used for truncated requests must record the original `n` and the `Symmetricity` hint. A cache miss due to `Symmetricity` mismatch must recompute. The truncated receipt's `cache_source` distinguishes `Full` (truncated from full) vs `Requested` (recomputed at exact k, which is only possible with an external iterative solver).

[GENERIC_MATH_CONSTRAINTS]:
- C# 14 generic math allows `where T : INumberBase<T>` constraints that subsume `IAdditionOperators`, `IMultiplyOperators` (homogeneous), and `IMultiplicativeIdentity`. Complex satisfies `INumberBase<Complex>` and `ISignedNumber<Complex>`, but NOT `INumber<Complex>`, `IFloatingPoint<Complex>`, or `IBinaryFloatingPoint<Complex>`. This means generic algorithms constrained on `INumber<T>` (which requires `IComparable<T>`) exclude Complex — Complex has no total order.
- For a generic spectral kernel parameterized by scalar type: `where T : INumberBase<T>` accepts both double and Complex; `where T : IFloatingPoint<T>` accepts only double/float. The Rayleigh quotient (a real scalar for Hermitian inputs, even when V is complex) requires the map `Complex → double` by `.Real`, which breaks the generic chain. The receipt scalar type for eigenvalues should be `double` regardless of input matrix scalar type when the matrix is real-symmetric or Hermitian.
- `IMultiplyOperators<TLeft, TRight, TResult>` is the correct interface for cross-type expressions, but no BCL numeric type defines the `(Complex, double, Complex)` instantiation. The work-around is a `IEigenScaleOps<TMatrix, TWeight>` interface (or equivalent delegate bundle) where `TMatrix` is the eigenvector type and `TWeight` is the real eigenvalue weight type. This is the delegate-bundle wide parameterization: explicit `Func<TMatrix, double, TMatrix> scale` passed alongside the filter function.
