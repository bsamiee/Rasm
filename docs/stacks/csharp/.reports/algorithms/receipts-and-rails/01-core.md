# Numeric Evidence Law: Receipts and Rails

[ALGORITHM_RECEIPTS]:
- Every decomposition in the admitted package hierarchy carries algorithm-specific typed evidence: `Cholesky<T>` exposes `Factor` (lower triangular `Matrix<T>`) plus computed `Determinant` and `DeterminantLn`; `LU<T>` exposes `Factors`, `Pivots` (`int[]`), and pivot-sign-corrected `Determinant`; `Svd<T>` exposes `S` (singular-value vector), `U`, `VT`, `Rank`, `L2Norm`, and `ConditionNumber`; `Evd<T>` exposes `EigenValues` (`Vector<Complex>`), `EigenVectors`, `BlockDiagonal`, `Rank`, `Determinant`, and the `isSymmetric` honesty flag. A generic receipt over all factorization families must carry the algorithm-specific properties as a concrete sub-type — no common abstraction is possible without losing evidence.
- `Symmetricity` is a four-case enum (`Unknown`, `Symmetric`, `Hermitian`, `Asymmetric`) passed to `Evd.Create` as a policy value that controls both the algorithm branch chosen at construction and the structural meaning of eigenvectors. Storing it as a field on the receipt turns an implicit algorithm choice into an auditable fact. `Unknown` triggers a symmetry check; passing `Symmetric` or `Hermitian` bypasses it and contracts that the caller has already verified the precondition — the receipt must record which case was supplied, not just what was inferred.
- `QRMethod` (`Full` vs `Thin`) is a behavior-carrying enumeration: `Full` produces a square `Q` (m×m) and full `R`; `Thin` produces economy-sized `Q` (m×n) and upper-triangular `R` (n×n). A solve over a thin QR cannot reconstruct the full column space. A receipt that records `QRMethod` is the only way a downstream consumer can know whether the orthogonal complement was discarded.
- The `Svd<T>.Rank` computation uses `S.Maximum().EpsilonOf() * max(m, n)` as the rank threshold — a machine-epsilon-relative tolerance tied to the largest singular value. Capturing both `S[0]` (largest) and the computed rank integer in a receipt makes the threshold auditable; a receipt that stores only `Rank` as an `int` hides the tolerance policy used.
- `Evd<T>.Rank` uses `AlmostEqual(eigenvalue, Complex.Zero)` with the library's `DefaultDoubleAccuracy = DoublePrecision * 10` — a private constant equal to `2^{-53} * 10`. This tolerance differs from the SVD threshold. A receipt distinguishing these two rank computations as algorithm-qualified facts prevents downstream code from treating them as interchangeable.

[TOLERANCE_TAXONOMY]:
- `Precision` exposes a named ledger: `DoublePrecision = 2^{-53}` (Demmel/LAPACK convention), `PositiveDoublePrecision = 2 * DoublePrecision` (Higham/ISO C convention), `MachineEpsilon` (measured at runtime), `PositiveMachineEpsilon`, and `DefaultDoubleAccuracy = DoublePrecision * 10` (private, used implicitly by parameterless `AlmostEqual`). Inline literals such as `1e-10`, `1e-14`, `1e-15`, or `1e-8` that duplicate one of these anchors are the anti-pattern; the anti-pattern is provable when the literal equals a multiple of `DoublePrecision` for a concrete platform.
- `EpsilonOf(this double value)` returns `Math.Abs(value) * DoublePrecision` — the unit-in-the-last-place (ULP) relative epsilon for that specific value, not the global machine epsilon. `PositiveEpsilonOf = 2 * EpsilonOf`. Using a global threshold where `EpsilonOf` is semantically correct causes false acceptances near zero and false rejections for large magnitudes. A receipt's validity predicate for "result within one ULP of input" must use `EpsilonOf` on the actual residual, not on a policy constant.
- `AlmostEqualNorm(a, b, diff, maximumAbsoluteError)` is the lowest-level comparison: it compares the pre-computed `diff` (`a - b`) against an absolute threshold, and all higher-level overloads delegate to it. A receipt validity predicate that pre-computes the residual norm and stores it alongside the threshold makes re-admission possible without recomputing `a - b`.
- `AlmostEqualNormRelative` divides the norm of the difference by the norm of `a` before comparing against `maximumError`. The distinction between absolute and relative error modes is not preserved by any existing result type — it must be recorded explicitly as an enum or boolean honesty marker on any receipt that performs a comparison.
- Three policy objects form the named-constant ledger for iterative convergence: `ResidualStopCriterion<T>` owns `_maximum` (absolute residual threshold) and `_minimumIterationsBelowMaximum` (hysteresis count); `DivergenceStopCriterion<T>` owns `_maximumRelativeIncrease` and `_minimumNumberOfIterations`; `IterationCountStopCriterion<T>` has `DefaultMaximumNumberOfIterations = 1000`. Duplicating any of these defaults as bare literals in a caller-side receipt is the tolerance anti-pattern.

[ITERATIVE_SOLVER_EVIDENCE]:
- `IterationStatus` has six cases: `Continue`, `Converged`, `Diverged`, `StoppedWithoutConvergence`, `Cancelled`, `Failure`. `StoppedWithoutConvergence` is the diagnostic-but-unusable case: the solver exhausted its iteration budget without converging, diverging, or breaking down. A receipt that maps it to `Failure` destroys the distinction between "budget exceeded" and "algorithm broke down" (the `NumericalBreakdownException` / `Failure` case). The minimum evidence partition is `Converged | StoppedWithoutConvergence | (Diverged | Failure | Cancelled)`; any coarser receipt risks hiding pathological conditioning behind an innocuous timeout.
- `TrySolveIterative` on `Matrix<T>` is void; it mutates the `result` parameter and leaves terminal status on `Iterator<T>.Status` or on each `IIterationStopCriterion<T>` instance. A wrapping domain receipt must capture the status read from `Iterator<T>.Status` post-call, the final residual vector from the mutated `result`, the iteration count drawn from `IterationCountStopCriterion.MaximumNumberOfIterations`, and whether a non-unit preconditioner was active — the preconditioner type changes what "residual" means.
- `DelegateStopCriterion<T>` accepts `Func<int, Vector<T>, Vector<T>, Vector<T>, IterationStatus>` — the delegate receives `(iterationNumber, solutionVector, sourceVector, residualVector)`. A receipt can embed a per-step snapshot policy directly: the delegate records every k-th residual norm into a preallocated `double[]` trail without heap pressure, and the trail length relative to the total step count becomes a conservation invariant in the receipt.

[OPTIMIZATION_EVIDENCE]:
- `MinimizationResult` carries `MinimizingPoint` (via `FunctionInfoAtMinimum.Point`), `Iterations`, and `ReasonForExit` (a twelve-case `ExitCondition`). `MinimizationWithLineSearchResult` extends this with `TotalLineSearchIterations` and `IterationsWithNonTrivialLineSearch` — the line-search sub-iteration count is algorithm-specific evidence owned by BFGS that has no analog on trust-region results and must not collapse into a generic `int TotalIterations`.
- `NonlinearMinimizationResult` carries `Covariance`, `Correlation`, `StandardErrors`, and `MinimizedValues` alongside `Iterations` and `ReasonForExit`. When `DegreeOfFreedom < 1` (underdetermined system), all three matrix and vector properties are `null`. A receipt validity predicate must gate on `DegreeOfFreedom >= 1` — accessible via `IObjectiveModelEvaluation.DegreeOfFreedom` — before treating the covariance as a meaningful fact; a receipt that materializes covariance without this gate is not tamper-evident.
- `ExitCondition` has twelve cases. The split between success-adjacent cases (`WeakWolfeCriteria`, `StrongWolfeCriteria`, `RelativeGradient`, `AbsoluteGradient`) and genuine non-convergence (`ExceedIterations`, `LackOfProgress`) is load-bearing for a receipt: meeting the Wolfe criteria with very few iterations is qualitatively different from hitting `LackOfProgress`, which indicates numerical stagnation. A receipt honesty marker that records the sub-family of `ExitCondition` obtained is more discriminating than the raw enum value.
- `LevenbergMarquardtMinimizer` constructor defaults: `initialMu = 0.001`, `gradientTolerance = stepTolerance = functionTolerance = 1e-15`. These three tolerances serve distinct purposes: gradient (first-order stationarity), step (parameter change), function (residual norm). A receipt that stores one scalar "tolerance" conflates three convergence criteria. The five-parameter constructor (`initialMu`, `gradientTolerance`, `stepTolerance`, `functionTolerance`, `maximumIterations`) is the minimal policy surface; a receipt stores all five as named fields with their algorithmic roles.

[CONSERVATION_PREDICATES]:
- Rank-nullity: for an `n × n` matrix, `rank + nullity == n`. `Evd<T>.Rank` counts non-zero eigenvalues under `DefaultDoubleAccuracy`; `Svd<T>.Rank` uses the ULP-relative threshold. Neither class exposes `Nullity`; a receipt constructed from either must derive `Nullity = n - Rank` and record both, so re-admission can verify the conservation identity without rerunning the decomposition.
- SVD singular-value ordering: `DenseSvd` stores singular values in descending order — `L2Norm = Math.Abs(S[0])` and `ConditionNumber = Math.Abs(S[0]) / Math.Abs(S[last])` are both correct only under descending order. A receipt must assert `S[0] >= S[1] >= ... >= S[k-1] >= 0` at construction; violation signals corruption or a post-construction modification of the `S` vector.
- Cholesky positive-definiteness: `DenseCholesky.Create` throws `ArgumentException` when the matrix is not SPD; there is no `bool IsPositiveDefinite` property. A receipt wrapping a Cholesky factorization records success as a structural fact — existence of the `Factor` field — not as a boolean flag computed after the fact. Existence equals success; no separate property is needed.
- LU pivot parity: `LU<double>.Determinant` computes the sign from `Pivots[i] == i` (even permutation) versus `Pivots[i] != i` (odd, negates). A receipt that stores `Determinant` must verify at construction that `Pivots.Length == RowCount` — the pivot array length equals the input dimension, and the conservation check `Pivots.Length == RowCount` is the structural invariant.

[DIAGNOSTIC_STATUS_DISTINCTION]:
- `IterationStatus.StoppedWithoutConvergence` must be kept distinct from both `Converged` and `Failure` in every receipt surface and every error union that wraps iterative solver outcomes. Mapping it to `Fin.Fail` is the anti-pattern: a rail that treats a budget-exceeded solve as a hard failure prevents the caller from retrying with a relaxed criterion or a different preconditioner.
- `ExitCondition.None` and `ExitCondition.ManuallyStopped` are the two non-algorithmic exit cases. A receipt constructed with `ExitCondition.None` is structurally incomplete; the factory must reject `None` and return `Fin.Fail(Error.New(code, "optimizer did not start"))`.
- `DssStatus.MKL_DSS_ZERO_PIVOT` is a definiteness failure, not a general solver failure — it indicates the matrix is singular or indefinite given the `DssMatrixType` policy. A receipt wrapping the sparse direct solver must map this to a domain-level `SingularMatrixError` distinct from a general internal breakdown status, because collapsing both to a single `SolverFailure` discards the definiteness signal.

[EXCEPTION_BOUNDARY]:
- The admitted numeric library emits three exception families: `NonConvergenceException` / `NumericalBreakdownException` (algorithm convergence); `OptimizationException` / `MaximumIterationsException` / `EvaluationException` (optimization); `ArgumentException` / `SingularUMatrixException` (structural preconditions). None is `Error`-typed. The single boundary conversion point is `Error.New(Exception ex)`, which wraps any exception into an `Exceptional` record. This call belongs in exactly one `catch` per algorithm invocation, immediately producing `Fin.Fail(Error.New(ex))`; raw exceptions must never propagate into domain code.
- `NumericalBreakdownException` is a sub-type of `NonConvergenceException`. A catch ladder that catches the parent first silently swallows the more specific breakdown signal. Catch `NumericalBreakdownException` first, map it to a domain `NumericalBreakdown` error case, then catch the parent for the general non-convergence case — two `catch` arms, each producing a distinct `Fin.Fail` case, at the boundary and nowhere else.
- `SingularUMatrixException : NativeInterfaceException` is thrown by the native BLAS/LAPACK provider when U is singular. At the boundary it maps to a typed domain error that preserves the "singular U" semantics, not a generic "solver failed" case.

[HONESTY_MARKERS]:
- `Symmetricity` as a stored receipt field is an honesty marker: `Symmetric` means the caller asserted real symmetry; `Hermitian` means conjugate symmetry; `Asymmetric` is explicit; `Unknown` means the library ran its own check. A receipt that omits this field makes the algorithm path ambiguous when the same receipt shape serves both symmetric and asymmetric cases.
- `QRMethod` stored on a QR receipt is a honesty marker: `Full` means the complete column space is preserved in `Q`; `Thin` means it was discarded. Re-admission of a thin-QR receipt for full-rank reconstruction is structurally invalid; any downstream factory that requires the full orthogonal complement must reject `Thin` receipts.
- `IObjectiveFunctionEvaluation.IsGradientSupported` and `IsHessianSupported` are honesty markers surfaced on every optimization evidence snapshot. A receipt that captures `IObjectiveFunctionEvaluation` must also capture these two flags; downstream Newton-based methods that inspect `Hessian` without checking `IsHessianSupported` silently receive `null`.
- `computeVectors` on `Svd<T>` construction controls whether `U` and `VT` are populated. When `false`, both are identity-shaped stubs. A receipt that embeds an `Svd<T>` computed with `computeVectors: false` must record this as a `bool VectorsComputed` flag; a consumer that reads `U` or `VT` without checking it silently receives stubs.

[MEMOIZATION_LAW]:
- `AtomHashMap<K, V>.FindOrAdd(K key, Func<V> None)` performs a CAS-loop that atomically inserts only when absent and returns the existing or freshly-inserted value. `None` may be invoked more than once under contention, so the factory must be a pure construction function with no observable side effects. A memoization surface for numeric receipts — for example, caching eigendecompositions keyed on a matrix identity token — must supply a pure factory; any factory that acquires a lock, writes a file, or mutates external state is structurally incorrect.
- `Atom<A>.Swap(Func<A, A> f)` CAS-loops until the swap succeeds; `f` may be called multiple times. A receipt-accumulating fold stored in an `Atom<HashMap<K,V>>` must supply an `f` that merges the new receipt into the existing map purely via `HashMap<K,V>.Add`, not a mutation that acts on an external `Dictionary<K,V>`.
- `AtomHashMap<K, V>.Change` is an event raised after each successful mutation. For numeric receipt caches, subscribing to `Change` for staleness invalidation is preferable to polling the map count; the event carries key, old-value, and new-value through `AtomHashMapChangeEvent<K, V>`.

[GENERIC_MATH_SURFACE]:
- `TensorPrimitives` on .NET 10 exposes generic operations with constraints that vary by operation: `Dot<T>`, `Norm<T>`, `CosineSimilarity<T>`, `FusedMultiplyAdd<T>`, `Sigmoid`, and `SoftMax` cover the hot-loop surface. For `double` and `float` these use hardware SIMD dispatch internally and are the BCL floor for any inner product, norm, or elementwise transform before reaching for MathNet's `Matrix<T>` operations.
- `TensorPrimitives.Dot<T>` has a permissive constraint (`IAdditionOperators`, `IAdditiveIdentity`, `IMultiplyOperators`, `IMultiplicativeIdentity`) that covers any algebraic ring, not just floating-point types. A receipt that stores a `double[]` residual vector and needs a norm for its validity predicate calls `TensorPrimitives.Dot(residual, residual)` then `Math.Sqrt` rather than constructing a full `DenseVector` and calling `.L2Norm()`.
- The `ISparseSolverProvider<T>` interface takes raw CSR arrays (`rowPointers`, `columnIndices`, `values`) plus `DssMatrixStructure` and `DssMatrixType` as behavior-carrying enumerations. A receipt wrapping a sparse direct solve must record both as honesty markers: `DssMatrixStructure.Symmetric` versus `Nonsymmetric` determines which fill-reducing ordering was applied, and `DssMatrixType.PositiveDefinite` versus `Indefinite` determines whether Cholesky or LDLT was used internally.

[RECEIPT_WIDTH_AND_FACTORY_DUALITY]:
- A receipt carrying more than eight distinct fields crosses the constructor-width pressure point. A nonlinear minimization receipt with `MinimizingPoint`, `Iterations`, `ExitCondition`, `Covariance`, `Correlation`, `StandardErrors`, `DegreeOfFreedom`, `GradientTolerance`, `StepTolerance`, `FunctionTolerance`, and `InitialMu` resolves through a required-evidence record (`Point`, `Iterations`, `Exit`, `DegreeOfFreedom`) and an `Option<StatisticalEvidence>` sub-receipt (`Covariance`, `Correlation`, `StandardErrors`) materialized only when `DegreeOfFreedom >= 1`. The sub-receipt is constructed independently and embeds its own conservation check (`StandardErrors.Count == Point.Count`).
- Factory/admission duality: a receipt constructor is private; a public static `Create` factory validates preconditions — `Rank > 0`, `Pivots.Length == n`, `S` non-empty and monotone-descending — and returns `Fin<TReceipt>`. Re-admission takes a serialized or cached receipt, replays the structural invariant checks against the stored payload, and returns `Fin.Fail` on violation. The re-admission path never re-runs the decomposition; it is O(n) in the receipt field count, not O(n³) in the matrix dimension.
- `IPrecisionSupport<T>` with `Norm()` and `NormOfDifference(T)` is the interface that makes `AlmostEqualNorm<T>` work on custom types. A domain value type that participates in numeric tolerance comparisons implements this interface to unlock the full `Precision` extension method surface. A receipt validity predicate that hand-rolls `Math.Abs(a - b) < tol` on a custom type bypasses the named-tolerance ledger.
