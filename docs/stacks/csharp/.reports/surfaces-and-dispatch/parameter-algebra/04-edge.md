# Parameter Algebra â€” Edge

[RETRY_PREDICATE_POLICY_FORMS]:
- `retryWhile(Schedule, K<M, A>, Func<Error, bool>)` and `retryUntil(Schedule, K<M, A>, Func<Error, bool>)` add a predicate gate to the schedule-driven retry loop. The predicate is a second axis of the policy value: the schedule governs spacing and count; the predicate governs the error domain that admits retries. Both are named fields on a policy record, not flags passed at call sites. Storing `static readonly Func<Error, bool> IsTransient = e => e.IsExpected && !e.IsType<TerminalError>()` alongside the schedule makes the full retry policy one cohesive value.
- `repeatWhile` and `repeatUntil` accept the same `(Schedule, K<M, A>, Func<A, bool>)` shape; the predicate receives the bound value, not an error. The distinction between retry-gate (error predicate) and repeat-gate (value predicate) is load-bearing: using `repeatWhile` when the intent is to abort on certain errors silently keeps repeating until the schedule expires because a success value always satisfies `Func<A, bool>`. The correct split: schedule-driven retry for error recovery, schedule-driven repeat-with-value-predicate for polling, never conflated on one call site.

[EXPECTED_EXCEPTIONAL_CATCH_VOCABULARY]:
- `exceptional<M, A>(Func<Exceptional, K<M, A>>)` and `expectedOf<E, M, A>(Func<E, K<M, A>>)` are the two orthogonal catch axes on the `Error` type hierarchy. `Expected` errors (created via `Error.New(string)` or `Error.New(int, string)`) carry `IsExpected = true`; `Exceptional` errors (created via `Error.New(Exception)`) carry `IsExceptional = true`. A policy that handles domain errors (`Expected`) and a policy that handles infrastructure faults (`Exceptional`) are two separate `static readonly CatchM` values chained with `|`; merging them into one predicate-catch erases the type distinction.
- `catchOfFold<E, M, A>(Func<E, K<M, A>>)` and `exceptionalOfFold<E, M, A>(...)` are the monoid-fold variants of `catchOf`/`exceptionalOf`. Where `catchOf` uses `ForAllM` (fail-fast on first handler failure), the `Fold` variants use `FoldM`, which continues processing sub-errors even after individual handler failures â€” the behavior depends on `M`'s `MonoidK` implementation. Choosing `catchOf` vs `catchOfFold` is a policy decision about aggregate error semantics: fail-fast vs collect-all.
- `Error.Filter<E>()` returns an `Error` containing only sub-errors of type `E`; if no sub-errors match, it returns `Errors.None`. `Errors.None` is a `ManyErrors` with an empty `Seq<Error>` â€” it is equal to `Error.Many(Seq<Error>())`. Using `error.Filter<MyError>().IsEmpty` to gate a handler is incorrect because `ManyErrors` has no `IsEmpty`; the correct guard is `error.IsType<MyError>()` before calling `Filter<MyError>()` to collect the matching sub-errors.

[SCHEDULE_TIME_INJECTION_POLICY]:
- `Schedule.upto(Duration max, Func<DateTime> currentTimeFn)`, `fixedInterval`, `windowed`, `secondOfMinute`, `minuteOfHour`, `hourOfDay`, and `dayOfWeek` all accept a `Func<DateTime> currentTimeFn` parameter. The default form (no `currentTimeFn`) uses `DateTime.UtcNow`; the injected form accepts any clock function. This is the clock-injection seam for these wall-clock-aware schedules â€” the policy value carries its own time source rather than reading a global. A `static readonly Schedule CronLike = Schedule.minuteOfHour(0, () => _clock.UtcNow)` binds the clock at construction, not at each retry step.
- `Schedule.fixedInterval` differs from `Schedule.windowed` in resync semantics: `fixedInterval` suppresses delay (emits zero) when the action overruns the interval, avoiding pile-up, but the next window starts from the completion time of the previous action; `windowed` always emits the residual time to the next absolute boundary regardless of action duration. `fixedInterval` is the load-leveling form; `windowed` is the cron-alignment form. Using `windowed` when the intent is load-leveling causes delayed actions to catch up immediately but then run at fixed boundary times, potentially causing burst retries at boundary crossings.
- `Schedule.repeat(int n)` repeats the entire schedule `n` times (documented with a NOTE: "use `Schedule.recurs(n)` for 'number of attempts'"). A `Schedule.exponential(100*ms) + Schedule.recurs(5)` is a 5-attempt exponential backoff; `Schedule.exponential(100*ms) + Schedule.repeat(2)` runs the entire exponential stream twice end-to-end, doubling the total iteration count. This is the collapse-point: `repeat` is a schedule multiplier, not an iteration count â€” storing it with a comment is insufficient; the stored policy value should use `recurs` exclusively when the intent is a retry count.

[VALUE_OBJECT_UNSAFE_CONVERSION_AND_NULL_ADMISSION]:
- `ValueObjectAttribute<TKey>.UnsafeConversionToKeyMemberType` (default `Explicit`) generates an `explicit operator TKey(TObj obj)` that can throw when `obj` represents the default struct case (`AllowDefaultStructs = true`). The "unsafe" designation means the operator can throw an `InvalidOperationException` when the value object wraps an unset/empty state. Setting `UnsafeConversionToKeyMemberType = None` removes this operator; callers must use `ToValue()` or `.Key` explicitly. The safe implicit operator (`ConversionToKeyMemberType = Implicit`) never throws and is the only conversion generated when `AllowDefaultStructs = false`.
- `NullInFactoryMethodsYieldsNull = true` changes the factory contract: `Create(null)` returns `null` instead of a validation error. This is the boundary-null passthrough for nullable reference types â€” it permits a `T?` input to map to a `T?` output without a validation failure. The cascade: setting this also sets `EmptyStringInFactoryMethodsYieldsNull` to `false` unless that is also explicitly `true`. The two flags are independent null-admission policies for two different input shapes (null vs empty/whitespace). Their combination defines the full nullable-string admission policy for one value object.
- `EmptyStringInFactoryMethodsYieldsNull = true` sets `NullInFactoryMethodsYieldsNull = true` automatically (documented: "by setting this property to true, `NullInFactoryMethodsYieldsNull` will also be true"). The reverse is not guaranteed: `NullInFactoryMethodsYieldsNull = true` does NOT set `EmptyStringInFactoryMethodsYieldsNull`. Treating them as independent knobs produces a null-passes-but-empty-fails policy; the only way to get both null and empty to yield null is to set `EmptyStringInFactoryMethodsYieldsNull = true` alone.

[SKIP_KEY_MEMBER_MANUAL_BACKING]:
- `ValueObjectAttribute<TKey>.SkipKeyMember = true` suppresses generation of the key field/property and all default-name references to it. The generator instead reads `KeyMemberName` to locate the manually-declared member, which must be a property with an `init` setter if it is a property (the generated constructor uses `init` assignment). This is the correct form when the key value requires domain-specific storage semantics â€” for example, a `readonly field` with a custom backing type, or a property that delegates to a native handle. The generator still produces the factory, equality, and conversion surfaces using the named member.
- When `SkipKeyMember = true` and `KeyMemberKind = MemberKind.Property`, the manual property must declare an `init` accessor. A `get`-only property without `init` produces a compile error in the generated code: the generated constructor body tries to assign `this.KeyMemberName = value` which requires a writable accessor. The constraint is specific to properties; for fields (`MemberKind.Field`), a `readonly` field is accepted because the generated constructor is allowed to initialize `readonly` members.

[OBJECT_FACTORY_HAS_CORRESPONDING_CONSTRUCTOR]:
- `ObjectFactoryAttribute.HasCorrespondingConstructor` (bool, default `false`) controls whether the generator emits a `ConvertFromKeyExpressionViaConstructor` in the factory metadata. When `true`, the generator assumes a constructor accepting a single parameter of type `T` exists on the target type and emits the direct-constructor expression as a fast path for conversion from a trusted source (EF Core materialization, post-validate internal construction). When `false`, all conversion paths go through the full validation factory. The `HasCorrespondingConstructor = true` form is the domain-internal trusted-source seam: values arriving from a validated store bypass re-validation while values from wire sources still pass through `Validate`.
- Multiple `[ObjectFactoryAttribute<T>]` declarations on one type with different `UseForSerialization` values register independent converter registrations per serialization framework. A type can declare `[ObjectFactoryAttribute<Guid>](UseForSerialization = SerializationFrameworks.SystemTextJson)` for JSON (Guid â†’ string-like path) and `[ObjectFactoryAttribute<string>](UseForSerialization = SerializationFrameworks.MessagePack)` for binary (string key path) with different admission validators for each. The `UseWithEntityFramework` and `UseForModelBinding` flags are always per-attribute, so EF Core can use a different factory than the JSON deserializer on the same type.

[ATOM_SWAP_IO_VS_SWAP]:
- `Atom<A>.SwapIO(Func<A, A>)` returns `IO<A>` rather than `A`. The IO wrapping defers the swap to effect evaluation time and allows the swap to participate in IO composition and error handling. `Atom<A>.Swap(Func<A, A>)` executes immediately and returns the committed value. The distinction is load-bearing in a pipeline: `swap.SwapIO(update).Bind(newVal => downstream(newVal))` threads the committed state through the subsequent IO computation without a separate read; `swap.Swap(update)` in a raw expression inside an IO pipeline produces a side effect that is not sequenced with the IO monad's execution order.
- `Atom<A>.SwapMaybeIO(Func<A, Option<A>>)` returns `IO<A>`: it evaluates inside IO, returns `None` from the function to abort without committing, and returns `Some(next)` to commit. The returned `IO<A>` produces the committed value on success and the unchanged value on abort â€” both paths are `IO<A>`, never `IO<Option<A>>`. Callers that need to distinguish "aborted" from "committed" must capture state outside the `SwapMaybeIO` call; the IO return value itself does not encode the abort-vs-commit distinction.
- `Atom<M, A>.SwapIO(Func<M, A, A>)` and `Atom<M, A>.Swap(Func<M, A, Option<A>>)` are the metadata-carrying forms. The two-argument form on `Atom<M, A>` exposes the metadata `M` as the first parameter to the swap function and the current state `A` as the second. This is the correct shape for a policy cell that must derive the next policy value from a fixed configuration record without closing over it at construction time: `Atom<PolicyConfig, Schedule>(config, initialSchedule)` with swap `(cfg, s) => s | Schedule.maxDelay(cfg.MaxBackoff)`.

[ATOMHASHMAP_AS_POLICY_REGISTRY]:
- `AtomHashMap<K, V>` exposes `SwapKey(K key, Func<Option<V>, Option<V>> f)` â€” the swap function receives `None` when the key is absent and `Some(v)` when present; returning `None` removes the key, returning `Some(next)` sets it. This is the atomic policy-registry mutation primitive: adding, updating, and removing entries are all expressed as one `Func<Option<V>, Option<V>>` with no separate `Add`/`Update`/`Remove` surface. The function may be called multiple times on CAS retry; it must be pure and allocation-minimal.
- `AtomHashMap<K, V>.Change` is an `event Action<AtomHashMapChangeEvent<K, V>>` that fires after each committed mutation. `AtomHashMapChangeEvent<K, V>` carries the key, the old value (`Option<V>`), and the new value (`Option<V>`), enabling downstream policy consumers to react to registry updates without polling. The event fires only on committed paths â€” failed CAS retries and validator rejections do not fire `Change`. This is the live-policy-registry notification seam.

[COMPLEX_VALUE_OBJECT_MEMBER_EQUALITY_COMPARER]:
- `[MemberEqualityComparerAttribute<TMember, TComparerAccessor>]` applies to a property or field of a `[ComplexValueObject]` and overrides the equality comparer used for that specific member in `GetHashCode` and `Equals`. The attribute takes the member type as `TMember` and a comparer accessor as `TComparerAccessor`. Multiple `[MemberEqualityComparerAttribute]` declarations on different members of the same complex value object are independent â€” each member can carry its own comparer. This is the per-field equality policy: a complex value object with a `string` field that requires ordinal comparison and a numeric field that requires exact equality declares both without a shared global policy.
- `StringKeyedObjectComparer<T>` is a concrete `IEqualityComparer<T>` for types where `T` is a smart enum or value object keyed on `string`. The static fields `Ordinal`, `OrdinalIgnoreCase`, `CurrentCulture`, etc. provide the most common string-semantic comparers for the domain type directly, without going through the key member. Using `StringKeyedObjectComparer<T>.Ordinal` as the comparer for a `FrozenDictionary<T, Policy>` that holds smart-enum-keyed policy values is the correct form: the comparer delegates to the same `StringComparer.Ordinal` that the smart enum's key member uses, preserving consistency between lookup semantics and enum identity.

[UNION_SKIP_EQUALITY_AND_BACKING_FIELD_INTERACTION]:
- `UnionAttributeBase.SkipEqualityComparison = true` suppresses `IEquatable<T>`, `Equals`, `GetHashCode`, and all comparison operators on the union. This is the correct form for unions that are used as discriminated command/request types where identity is meaningless (no two requests are "equal" by structural comparison) â€” equality generation is wasted code at that use site. The suppression interacts with `UseSingleBackingField`: the backing-field layout affects `GetHashCode` bucket distribution when equality is enabled; suppressing equality makes the backing-field choice irrelevant to correctness.
- `UnionAttributeBase.UseSingleBackingField = true` with a mix of struct and reference-type cases produces boxing for every struct case on both storage and retrieval. The discriminator index is stored separately from the payload; the struct case is boxed into the single `object` backing field on `IsT1` assignment and unboxed on `AsT1` access. When the union carries only reference-type cases, `UseSingleBackingField` saves memory by eliminating per-case backing fields at the cost of one interface dispatch on unboxing. The struct-case boxing cost is only paid when `T1IsStateless = false` (a stateless struct case has no payload to box).

[PROJECTION_EQUALITY_COMPARER_AS_POLICY_TABLE_KEY]:
- `Thinktecture.Collections.ProjectionEqualityComparer<T, TItem>` constructs an `IEqualityComparer<T>` from a `Func<T, TItem> selector` and an optional inner `IEqualityComparer<TItem>`. This is the correct form for using a complex domain type as a `FrozenDictionary` key when equality should be defined by one projected field, not by the type's full structural equality. A `FrozenDictionary<RequestType, PolicyRow>` keyed on a union-case type uses `ProjectionEqualityComparer<RequestType, Kind>` where `Kind` is the discriminant, so two requests with the same kind map to the same policy row regardless of payload differences. This is the policy-table key algebra: the comparer is the policy, not just infrastructure.
