[STAMINA_GENERATOR_AND_CONTEXT_MECHANICS]:
- `@stamina.retry` handles generator functions (`isgeneratorfunction`) and async generator functions (`isasyncgenfunction`) separately from coroutines and plain callables, each producing a distinct inner wrapper (`sync_gen_inner` with `yield from`, `async_gen_inner` with full `__anext__`/`asend`/`athrow`/`aclose` protocol). An aspect factory that calls `iscoroutinefunction(fn)` to distinguish sync from async is insufficient — it silently misclassifies generator and async-generator surfaces: `iscoroutinefunction` returns `False` for generator functions and `True` only for true coroutine functions. The correct gate sequence is `isgeneratorfunction` -> `isasyncgenfunction` -> `iscoroutinefunction` -> sync fallback, matching stamina's own dispatch in `retry_decorator`. [`stamina/_core.py` lines 848–893, 26.1.0, 2026-06-09]
- `stamina.set_testing(testing=True, attempts=N, cap=True)` caps rather than replaces the decorated attempt count: `_Testing.get_attempts` returns `min(N, non_testing_attempts or N)`, so a decorator configured with `attempts=2` is capped at `min(3, 2) = 2` when called with `cap=True, attempts=3` — the user's tighter bound wins. `cap=False` (default) unconditionally overrides to `N`. The `_RestoreTestingCM` returned by `set_testing` is a sync context manager whose `__exit__` restores `CONFIG.testing` to the prior `_Testing | None` state, making it safe to nest test-mode scopes. [`stamina/_config.py` lines 16–45, 158–195, 2026-06-09]
- `stamina.retry_context`'s `_RetryContextIterator.with_name(name, args, kw)` creates a shallow copy via `dataclasses.replace` carrying a new `_name`, `_args`, `_kw` — the `_t_kw` dict reference and the tenacity `AsyncRetrying` are shared. The `@retry` decorator calls this per invocation to bind the actual arguments into `RetryDetails` populated in the `before_sleep` callback. An aspect that builds a `retry_context` at decoration time and calls `.with_name(guess_name(fn), args, kw)` per call replicates `@retry`'s behavior for code blocks that span multiple operations — the correct pattern for imperative retry inside a `Result`-returning aggregate pipeline. [`stamina/_core.py` lines 544–550, 667–708, 2026-06-09]
- `stamina.get_default_hooks()` auto-discovers `prometheus_client` and `structlog` via `importlib`-level `try/import` at call time, not at module import. `PrometheusOnRetryHook = RetryHookFactory(init_prometheus)` is a module-level constant. `init_prometheus` is guarded by a `global RETRIES_TOTAL` sentinel so repeated calls from test teardown are idempotent. An aspect that calls `set_on_retry_hooks([])` to silence retry noise in tests and restores the default by passing `None` (`set_on_retry_hooks(None)` resets `CONFIG.on_retry` to `None`, triggering lazy re-initialization on the next retry) is the idiomatic teardown pattern — not re-registering the default hooks explicitly. [PYDANTIC_VALIDATE_CALL_ADVANCED_MECHANICS]:
- `@validate_call(validate_return=True)` generates a second `pydantic_core` validator for the return annotation. For async callables, this validator is wrapped as an `async def return_val_wrapper(aw)` that `await`s the coroutine and then calls `validator.validate_python(result)` — so return validation is async-transparent. The `ValidateCallWrapper.__call__` checks `__pydantic_complete__` on every invocation and lazily triggers `_create_validators()` if not yet built (deferred-build path via `ConfigDict(defer_build=True)`). In a cold-start scenario where the first call constructs the validator, race conditions are possible in multithreaded code — `ValidateCallWrapper` has no internal lock. For thread-safety, build at decoration time (do not pass `defer_build=True` to the `config` argument of `validate_call`). [ANYIO_ASYNC_CACHE_AND_RUNVAR]:
- `anyio.functools.lru_cache` and `anyio.functools.cache` discriminate on `iscoroutinefunction`: coroutine functions get `AsyncLRUCacheWrapper` (task-safe, TTL, `RunVar`-scoped); sync functions fall through to `functools.lru_cache`. `AsyncLRUCacheWrapper` stores cache entries in a `RunVar`-backed `WeakKeyDictionary[wrapper, OrderedDict[key, (value, Lock, expires_at)]]` — a `RunVar` is event-loop-scoped (keyed by the asyncio `Task` token), not process-global. This means the async cache is ISOLATED per event-loop run: in tests that create a fresh event loop per test (the anyio default), the cache is automatically cleared between tests with no manual `cache_clear()`. [`anyio/functools.py` lines 49–221, anyio 4.13.0, 2026-06-09]
- `anyio.functools.lru_cache(maxsize=N, ttl=T)` accepts an integer `ttl` in seconds; expired entries are detected lazily on the next call to the same key (TTL is not proactively evicted). The check is `current_time() >= expires_at` using anyio's `_core._eventloop.current_time`, which is event-loop-clock based — deterministic in tests using a mock clock. An aspect that caches expensive async operations (e.g., token refresh, slow config reads) should use `anyio.functools.lru_cache` with a TTL rather than `functools.cache`, which has no TTL and caches `Error` results permanently. [CROSS_CONCERN_FAILURE_MODES]:
- `stamina.retry_context` increments `rcs.attempt_number` before running `before_sleep`; `_make_before_sleep` builds `RetryDetails(retry_num=rcs.attempt_number)` where `retry_num=1` means the first retry (second attempt), not the first attempt. A `RetryHook` that uses `details.retry_num` as an OTel span attribute must document this as 1-based retry count, not attempt count. `Attempt.num` from `retry_context` iteration IS attempt number (1-based attempt, not retry count). The two `num` fields are semantically different and must not be aliased in structured log output. [`stamina/_core.py` lines 181, 685–707, 2026-06-09]
- `structlog.contextvars.clear_contextvars()` sets each known `ContextVar` to `Ellipsis` (not `None`) — `Ellipsis` is the sentinel for "no value" in structlog's scheme. A `merge_contextvars` processor skips keys where `ctx[k] is Ellipsis`. An aspect that calls `clear_contextvars()` at the start of a request handler clears only keys previously bound in this contextvars copy; it does NOT clear keys set in ancestor contexts (tasks inherit their parent's context snapshot, so `clear_contextvars()` only affects the copy for the current task). For a fresh task with clean logging context, bind a new context via `bind_contextvars` after creation — the inherited parent bindings are read-only copy-on-write. [`structlog/contextvars.py` lines 103–117, 2026-06-09]
- `anyio.functools.AsyncLRUCacheWrapper` uses an anyio `Lock(fast_acquire=not always_checkpoint)` per cache key to prevent concurrent cache population (dog-pile / thundering-herd). `fast_acquire=True` (default) skips a checkpoint when acquiring an uncontested lock — the lock acquisition is synchronous and does not yield to the event loop. Setting `always_checkpoint=True` forces a checkpoint on every cache call including hits, which is required when a cached `async def` is called in a tight loop that must yield periodically. For aspect-level caching of expensive I/O operations, `always_checkpoint=False` (default) is correct. [`anyio/functools.py` lines 170–221, `_synchronization.py` lines 83–101, anyio 4.13.0, 2026-06-09]
- `pydantic.validate_call` raises `ValidationError` (not `BeartypeCallHintViolation`) — pydantic's own error type with `errors()` returning a list of `InitErrorDetails`. The boundary lift in the aspect must catch `pydantic.ValidationError` specifically; catching `BeartypeCallHintViolation` in the outermost layer will NOT intercept `validate_call` rejections because the two decorators operate on orthogonal execution paths. A catch-all `except Exception` at the aspect boundary erases this diagnostic distinction: use `except pydantic.ValidationError as exc: Error(ValidationFailure(exc.errors()))` and `except BeartypeCallHintViolation as exc: Error(ContractViolation(str(exc)))` as separate arms in the boundary converter.
