# Dense Algebra — Core Research

[FACTORIZATION_SUITE]:
- Six factorizations are available on every `DenseMatrix<double>` instance as virtual factory methods: `Cholesky()`, `LU()`, `QR(QRMethod)`, `GramSchmidt()`, `Svd(bool computeVectors)`, and `Evd(Symmetricity)`. Each returns its typed abstract handle (`Cholesky<double>`, `LU<double>`, etc.) rather than a concrete sealed class; the concrete `Dense*` implementations are `internal sealed` and unreachable by name. The handle carries all downstream evidence and the `Solve(Matrix<T>, Matrix<T>)` / `Solve(Vector<T>, Vector<T>)` operations, so a retained handle yields repeated right-hand-side solves with zero refactorization cost.
- `DenseCholesky.Create` clones the input, overwrites the clone with the lower-triangular factor in place, and throws `ArgumentException("Matrix must be positive definite.")` if any pivot diagonal is `<= 0.0` during the column-by-column Banachiewicz sweep. This makes `Cholesky()` the canonical definiteness probe: wrap in `Fin<Cholesky<double>>` at the admission boundary and a successful handle is simultaneously a proved-SPD certificate. `Cholesky.Factorize(Matrix<double>)` is a live re-factorization method on the retained handle that mutates the internal factor buffer in place, enabling efficient rank-one update pipelines that supply a fresh near-symmetric-positive-definite matrix without allocating a new handle.
- `DenseLU.Create` requires `RowCount == ColumnCount`; the factorization performs partial pivoting with row swaps recorded in a `int[] Pivots` array (one pivot index per row). `LU.Determinant` computes the product of diagonal U entries, negating each term where `Pivots[i] != i`. Non-singularity is not asserted at construction time — a singular input produces a handle with `Determinant == 0.0`, not an exception; singularity detection must be done by the caller post-construction.
- `DenseQR.Create(DenseMatrix, QRMethod)` dispatches to two internal provider paths: `QRMethod.Full` stores an m×m orthogonal matrix Q and an m×n upper-triangular R; `QRMethod.Thin` stores an m×n Q and an n×n R. The default on `Matrix.QR()` is `QRMethod.Thin`. The `Tau` array (Householder reflector coefficients, length `min(m,n)`) is retained on the handle and forwarded to the provider's `QRSolveFactored` call, so the provider can reconstruct the action of Q without materializing it. `QR.IsFullRank` checks that no diagonal entry of `FullR` is `AlmostEqual(0.0)` — this is the numerically correct full-rank gate for the regression and least-squares routes.
- `DenseGramSchmidt.Create` runs modified Gram-Schmidt (not classical) in pure managed code and throws `ArgumentException` when a column norm during orthogonalization is exactly zero — it detects rank deficiency only at this threshold, not at a machine-epsilon-relative threshold. Unlike Householder QR, this path does not route through the provider and receives no native acceleration; it is numerically inferior for ill-conditioned systems and should not be used when the design matrix may be near-rank-deficient.
- `DenseEvd.Create` takes a `Symmetricity` discriminant. When `Symmetricity.Unknown`, it calls `matrix.IsSymmetric()` at factorization time (an O(n²) scan). When the caller knows the structure, passing `Symmetricity.Symmetric` or `Symmetricity.Asymmetric` skips that scan. The symmetric path routes to `SymmetricTridiagonalize` (Householder tridiagonalization, EISPACK tred2 derivation) followed by `SymmetricDiagonalize` (implicit QR iteration). The asymmetric path routes to `NonsymmetricReduceToHessenberg` then `NonsymmetricReduceHessenberToRealSchur`. Eigenvalues are always stored as `Vector<Complex>` regardless of symmetricity; for a symmetric input, imaginary parts are numerically zero but the Complex representation is not stripped. The `Evd.Solve` method is only implemented for the symmetric case and throws for asymmetric inputs.
- `DenseSvd.Create` allocates U as m×m and VT as n×n unconditionally, regardless of `computeVectors`. When `computeVectors = false` the provider still allocates the full U and VT arrays but skips their computation; the subsequent `Solve` call throws `InvalidOperationException("The singular vectors were not computed.")` if vectors were not requested. Requesting vectors is therefore a requirement for any solve or pseudo-inverse use, and the `bool` parameter is meaningful only as a cost gate when evidence extraction alone (rank, L2Norm, ConditionNumber) is needed.

[SVD_EVIDENCE]:
- `Svd<double>.Rank` computes the tolerance as `S.Maximum().EpsilonOf() * max(U.RowCount, VT.RowCount)`. `EpsilonOf(x)` is the ULP epsilon at the value `x` — for a well-scaled matrix with max singular value near 1, this approximates `σ_max · ε_machine · max(m,n)`, which matches the standard rank-revealing threshold. Critically, this is evaluated against the max singular value's ULP, not against `Precision.DoublePrecision` directly, so the threshold tracks the actual numerical scale of the problem.
- `Svd<double>.ConditionNumber` returns `|S[0]| / |S[min(m,n) - 1]|` — the ratio of largest to smallest singular value regardless of rank. This can be `+Inf` for singular or rank-deficient matrices, and the caller must guard for this case before using the result as a quality gate.
- `Matrix.PseudoInverse()` (on `Double.Matrix`) builds the pseudo-inverse by: (1) calling full SVD with vectors, (2) zeroing singular values below `max(m,n) * σ_max * Precision.DoublePrecision` in the diagonal weight matrix W, (3) inverting the retained values, (4) reconstructing as `(U · W · VT)ᵀ`. This is a correct truncated SVD pseudo-inverse. The rank threshold used here differs from `Svd.Rank`'s per-value ULP threshold — it uses a global `σ_max · ε_machine · max(m,n)` tolerance, which is more conservative and standard.

[ADMISSION_GATES]:
- All six `Dense*` factorizations perform squareness and dimension checks at construction and throw `ArgumentException` or `ArgumentNullException` for violated preconditions. None perform all-finite checks (NaN/Inf propagation is silent through the factorization). An admission gate must check `matrix.Values.All(double.IsFinite)` or equivalent before calling any factorization factory; the provider's inner loops do not guard against NaN/Inf and will silently produce corrupted factor state.
- `Cholesky()` is the definiteness admission gate: a successful construction certifies SPD; a thrown `ArgumentException("Matrix must be positive definite.")` is the rejection evidence. This should be the sole path for detecting definiteness — attempted factorization, not a separate eigenvalue probe.
- For the `LU` path, singularity is silent at construction. The correct singularity admission gate post-construction is `Math.Abs(lu.Determinant) < threshold` or, more robustly, probing `lu.Factors.Diagonal().Any(v => Math.Abs(v).AlmostEqual(0.0))`.

[RESIDUAL_CAPPED_SOLVE]:
- No built-in residual cap exists on any `Dense*` solve path. The managed solve paths (`LUSolveFactored`, `QRSolveFactored`, `SvdSolveFactored`, `CholeskySolveFactored`) forward directly to in-place triangular solve or back-substitution via the provider without computing a residual. A residual-capped solve must be implemented by the caller: after `factorization.Solve(b, x)`, compute `r = A·x - b` (using the original matrix, not the factored form), and demote success to failure when `r.L2Norm() / b.L2Norm() > eps_cap`. The `DenseMatrix.Values` array is accessible via the public `Values` property, enabling direct span-level residual computation without boxing.

[LEAST_SQUARES_ROUTES]:
- `MultipleRegression.NormalEquations<T>` calls `x.TransposeThisAndMultiply(x).Cholesky().Solve(x.TransposeThisAndMultiply(y))`. When X is sparse or tall, `TransposeThisAndMultiply` produces a dense n×n Gram matrix regardless of the original structure — a quadratic densification cliff. For an m×n design matrix with m >> n and sparse structure, the normal-equations route discards sparsity, squares the condition number (κ(AᵀA) = κ(A)²), and allocates O(n²) dense storage.
- `MultipleRegression.QR<T>` calls `x.QR().Solve(y)` using the default `QRMethod.Thin`, which correctly handles overdetermined systems without forming the Gram matrix. The retained thin QR handle can be queried for `IsFullRank` to obtain full-rank evidence before accepting the solution.
- `MultipleRegression.Svd<T>` calls `x.Svd(computeVectors: true).Solve(y)` and delivers rank-deficiency-tolerant least squares. The SVD solve replaces small singular values with zero rather than inverting them, but the threshold is not exposed; the caller cannot supply a custom tolerance.
- `DirectRegressionMethod` is the policy discriminant. `NormalEquations` is the default and the most dangerous for near-rank-deficient or badly conditioned systems; the choice must be made explicit as a receipt field, not left at the default.

[PACKED_STORAGE]:
- No packed symmetric (triangular) storage exists in the admitted package surface. `DenseColumnMajorMatrixStorage<T>` stores all m×n entries as a column-major `T[] Data` flat array. There is no triangular, Hermitian-packed, or band storage. Algorithms that exploit symmetry (Cholesky, symmetric EVD, symmetric regression via normal equations) still allocate and operate on the full n×n array; the upper triangle is written then zeroed in the Cholesky inner loop but is not packed. This doubles memory for symmetric matrices over LAPACK SSYTRF/DSYTRF-style packed storage and prevents cache-friendly packed symmetric BLAS operations.
- The storage backing array is accessible for direct array work: `DenseMatrix.Values` returns the private `double[] _values` via a public property; `DenseColumnMajorMatrixStorage<T>.Data` exposes the same array. This is the injection point for `TensorPrimitives`-based hot loops that the provider does not cover.

[PROVIDER_AND_VECTORIZATION]:
- The managed provider (`ManagedLinearAlgebraProvider`) contains no SIMD or `TensorPrimitives` calls. The `DotProduct`, `AddArrays`, and `MatrixMultiplyWithUpdate` implementations are plain scalar loops. `MatrixMultiplyWithUpdate` allocates a `double[][]` column-cache of B's columns before the row-column dot loop; under the managed provider, this is the dominant dense GEMM kernel and has no vectorization. For dense loops the library does not own, `System.Numerics.Tensors.TensorPrimitives` (in-box on .NET 9+) provides SIMD-backed `Dot<T>`, `Add<T>`, `Multiply<T>`, `MultiplyAdd<T>`, and `SumOfSquares<T>` over `ReadOnlySpan<T>`, and is the platform vectorization floor for any dense hot loop operating on `DenseMatrix.Values` / `DenseVector.Values` spans directly.
- Provider switching at startup: `LinearAlgebraControl.UseNativeMKL()` / `TryUseNativeMKL()` replaces the managed provider with the MKL-backed provider if `MathNet.Numerics.Providers.MKL` is loaded. The global state is a `static ILinearAlgebraProvider` under a lock; setting `Control.MaxDegreeOfParallelism` calls `Provider.InitializeVerify()` which re-initializes the provider under the new degree. Provider selection is process-global — it cannot be scoped per-thread or per-operation. When MKL is present, all six dense factorizations route to LAPACK DGETRF/DPOTRF/DGEQRF/DGESVD/DSYEV/DGEEV equivalents with full BLAS3 performance; without it, the managed provider's O(n³) scalar loops dominate.
- The managed Cholesky inner loop parallelizes via `CommonParallel.Invoke` when the remaining column range exceeds `Control.ParallelizeElements` (default 300) and `MaxDegreeOfParallelism > 1`. The managed GEMM parallelizes via `CommonParallel.For` when `rowsA + columnsB + columnsA >= Control.ParallelizeOrder` (default 64). These thresholds are not exposed as public API and cannot be tuned per-call; they respond only to `Control.ParallelizeOrder` and `Control.ParallelizeElements` which are `internal`.

[SMALL_DENSE_PROBES]:
- For small dense systems (point-set moments, per-pair precision matrices, covariance estimation), the factorization-as-probe pattern is the correct idiom: allocate a `DenseMatrix` from a `double[n*n]` span via `new DenseMatrix(n, n, data)`, call the appropriate factory, and classify the outcome. Definiteness-by-Cholesky, rank-by-SVD, and condition-by-ConditionNumber are all O(n³) but with small constants for n ≤ 10 and no heap overhead beyond the storage array. At n ≤ 4 (3D covariance, 4-point alignment systems), the managed factorization paths are faster than provider switching overhead.
- Per-query factorization cost in scattered-data evaluation: each call to `Cholesky()`, `QR()`, etc. clones the input matrix via `matrix.Clone()` before factorizing (verified in `DenseLU.Create` and `DenseCholesky.Create`). For a scattered-data interpolation kernel where the coefficient system changes per query, factorizing a cloned copy is unavoidable — the original matrix is preserved. The clone allocation dominates at small n; reusing pre-allocated factor buffers requires implementing a custom provider path or using `Cholesky.Factorize(Matrix<double>)` against a retained handle with a pre-allocated clone that is overwritten each query.

[CONGRUENCE_AND_DENSIFICATION_CLIFFS]:
- The quadratic densification cliff via normal equations (`AᵀA` formation) is not limited to the regression API. Any algorithm that uses `matrix.TransposeThisAndMultiply(matrix)` (a `DenseMatrix` → `DenseMatrix` path through the managed provider's `MatrixMultiplyWithUpdate`) produces a dense n×n result regardless of whether the input was sparse or rectangular. The provider implementation allocates a `double[][] columnDataB` scratch of size `columnsB × rowsB` before the loop, so for a 1000×20 design matrix the ATA formation allocates 20 columns of 1000 doubles each, plus the 20×20 output. The result is dense whether or not the input was.
- `DenseEvd.Solve` for a symmetric factored matrix performs two O(n²) manual loops (Vᵀb then VΛ⁻¹(Vᵀb)) without routing through the provider or using `MatrixMultiplyWithUpdate`. This path receives no BLAS acceleration even under MKL. The EVD solve is more expensive per operation than a Cholesky solve of the same problem and should be used only when eigenpairs are separately needed for evidence extraction.

[REGISTRATION_INNER_SOLVES]:
- Closed-form orthogonal alignment (SVD of the cross-covariance matrix followed by determinant correction for reflection cases) requires: `H = DenseMatrix.OfRowArrays(pts).TransposeThisAndMultiply(DenseMatrix.OfRowArrays(refs))`, then `var svd = H.Svd(computeVectors: true)`, then rotation `R = (svd.VT.Transpose() * svd.U.Transpose()).Transpose()` with `det(R)` correction by negating the last column of VT when `det(R) < 0`. The determinant of R is `svd.Determinant` on the 3×3 cross-covariance SVD handle. The SVD factorization on a 3×3 matrix (m=3, n=3) takes under 1µs under the managed provider and should not be cached across calls unless the point set is identical.
- Damped normal equations for iterative registration (ICP refinement, nonlinear least squares with Levenberg-Marquardt damping): the system `(JᵀJ + λI)x = Jᵀr` is formed by computing the dense Gram matrix `JᵀJ` via `TransposeThisAndMultiply`, adding `λ` to the diagonal via `AddDiagonal` or a loop over `SetDiagonal`, and solving via `Cholesky()`. The Cholesky factorization of `JᵀJ + λI` succeeds for any `λ > 0` regardless of the rank of J, so the SPD certificate from `Cholesky()` is guaranteed at sufficient damping; this is the correct admission gate for the damped step, not a pre-check on J itself.
