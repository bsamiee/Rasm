# State-Cell Boundary Law — Edge

[STM_REF_MULTI_CELL_COORDINATION]:
- `Ref<T>` is the correct primitive when two or more state cells must transition atomically. `Atom<A>` CAS-loops on a single cell; two independent atoms cannot be CAS-swapped together without a window where one has committed and the other has not. `Ref<T>` inside `STM.DoTransaction` / `sync(...)` provides snapshot isolation across any number of refs with automatic retry on conflict — the unit of atomicity is the transaction body, not a single cell.
- `STM.DoTransaction<T>(Func<T>, Isolation)` nests: when a transaction is already running on the current thread, the inner `DoTransaction` call participates in the outer transaction rather than starting a new one. `STM.TransactionId` returns the currently running transaction id; its value is non-null only when called from within a live transaction body. A state cell that conditionally starts a transaction only when not already inside one can read `STM.TransactionId` as a guard — null means no enclosing transaction; non-null means nested participation.
- `Isolation.Snapshot` triggers a retry only when a ref written inside the transaction was also written externally between the transaction's read-point and its commit. A ref that is read but not written is not tracked for conflict — external writes to read-only refs do not cause retry. `Isolation.Serialisable` tracks both reads and writes; any external write to any ref touched (read or written) causes retry. For boundary lifecycle cells where the correctness invariant is "the set of open connections did not change while I was deciding to add one", `Serialisable` is required; `Snapshot` permits a torn read.
- `commute(Ref<T>, Func<T,T>)` re-runs the function with the most recently committed value at commit time rather than with the in-transaction snapshot. The in-transaction read returns the snapshot value for consistency within the body; the commit write uses the live value. Use `commute` only for operations where `f(f(x)) == f(x)` or where last-one-in-wins is acceptable policy — append-only counters and idempotent flags. For boundary cells whose value determines whether an operation is permitted, `swap(ref, f)` (not `commute`) enforces the snapshot-at-read invariant.
- `Ref<T>.Value` setter called outside a transaction wraps itself in an implicit `Snapshot` transaction. The implicit transaction has the same retry semantics as an explicit `sync(() => ref.Value = v, Isolation.Snapshot)`. This means a lone assignment is safe but has undefined ordering relative to concurrent explicit transactions — it does not participate in their atomicity guarantee. Two consecutive outside-transaction assignments are two separate implicit transactions and can interleave with any concurrent explicit transaction between them.

[STM_ISOLATION_PRESSURE_FOR_LIFECYCLE_CELLS]:
- A lifecycle state machine encoded as multiple `Ref<T>` cells — one for state, one for in-flight count, one for drain deadline — inside a single `Serialisable` transaction is the correct shape for drain-with-count correctness. The `Draining` transition sets the state ref and reads the in-flight count atomically; any concurrent operation that increments the in-flight count will conflict with the state-read, triggering a retry. The result is that either the drain commits before any in-flight increment, or the in-flight increment commits and the drain retries with the updated count.
- The `STM.DoTransaction<T>(Eff<RT, T>, Isolation)` overload accepts an `Eff<RT, T>` effect as the transaction body. The effect runs inside the transaction; if the transaction retries, the effect's `IO` subtree is re-evaluated. Any effect with external side effects (logging, metrics, I/O) that runs inside the transaction body re-executes on each retry. Transaction bodies must be confined to pure `Ref` reads and writes; side-effectful `IO` must run in a continuation after the transaction commits using the committed values.
- `CommuteRef<T>` is returned by `commute(Ref<T>, f)`. It is a proxy that signals to the commit algorithm that the final write should use the live value of the ref, not the snapshot. `CommuteRef<T>` cannot be read back within the same transaction — its current value is undefined until commit. A transaction body that reads a `CommuteRef<T>` after commuting it will observe the pre-commute snapshot value, not the commuted result. Boundary cells that must read-then-commute-write within one transaction should restructure as swap — the commute pattern is for accumulators whose intermediate reads do not matter.

[CANCELLATIONTOKEN_ASYNC_CANCEL_PATTERN]:
- `CancellationTokenSource.CancelAsync()` returns a `Task` that completes after all synchronous callbacks registered via `CancellationToken.Register` have executed and any asynchronous continuations scheduled through them have been queued (but not necessarily completed). It does not await the async continuations themselves. The behavioral contract is: `Cancel()` completes synchronously after running all callbacks; `CancelAsync()` returns a task, runs all callbacks asynchronously, and the task completes when all callbacks return. For a boundary cell that must ensure all token callbacks complete before proceeding to teardown, `await source.CancelAsync()` provides the right seam; `Cancel()` followed by `await Task.Yield()` does not have the same guarantee.
- `new CancellationTokenSource(TimeSpan delay, TimeProvider provider)` creates a source whose timer uses the injected `TimeProvider`. A `FakeTimeProvider` from the `Microsoft.Extensions.TimeProvider.Testing` package can advance the clock to trigger the timeout in deterministic tests without real-time elapsed time. The same `TimeProvider` overload is available for `PeriodicTimer(TimeSpan, TimeProvider)`. Both constructors accept the `TimeProvider` as the second argument and start the timer/ticker immediately on construction — the timer window begins at object creation, not at first `await`.
- `PeriodicTimer.Period` is a read-write property. Setting it mid-operation changes the interval immediately: the next `WaitForNextTickAsync` will use the new period. There is no `Reset` — a running tick that was triggered before the period change completes normally, and the new period applies to the gap between that tick's completion and the next scheduled tick. A boundary cell that must rate-adapt its polling loop can mutate `Period` directly rather than disposing and recreating the timer.

[LAZY_EXCEPTION_CACHING_FOR_INIT_CELLS]:
- `Lazy<T>` with `LazyThreadSafetyMode.ExecutionAndPublication` caches the first exception thrown by the factory permanently — every subsequent `.Value` access rethrows the cached exception without retrying the factory. This makes it unsuitable as an initialization cell for anything with transient failure modes (network, file system, native library load). A boundary cell whose initialization can fail transiently must use `LazyThreadSafetyMode.PublicationOnly`, which does not cache exceptions and retries on each subsequent `.Value` access until one succeeds, or must use `Atom<Option<T>>` with explicit retry policy via `IO` schedule.
- `LazyThreadSafetyMode.PublicationOnly` allows multiple threads to run the factory concurrently; the first to complete sets the value, and all others discard their results. For factories that acquire a native resource (file handle, connection, native library load), concurrent initialization means competing factories all acquire the resource but only one is retained — the others are silently discarded without disposal. `PublicationOnly` is only safe for factories that do not acquire resources or for resources that are self-cleaning on GC finalization. The safe concurrent-initialization cell for resource-acquiring factories is `Lazy<T>(factory, LazyThreadSafetyMode.ExecutionAndPublication)` with the explicit acknowledgment that transient failures are permanent.
- Recursive access to `Lazy<T>.Value` inside the factory throws `InvalidOperationException` under `ExecutionAndPublication` (the lock is not re-entrant). Under `PublicationOnly`, recursive access does not throw — the recursion sees the uninitialized state and re-enters the factory, producing a second initialization attempt. The behavioral divergence means a boundary cell that chains lazy initialization cells must prevent cross-cell circular dependencies or choose `PublicationOnly` explicitly and accept the double-initialization risk.

[PRIORITIZED_DRAIN_CHANNEL_AS_STATE_BOUNDARY_SEAM]:
- `Channel.CreateUnboundedPrioritized<T>(UnboundedPrioritizedChannelOptions<T>)` creates a channel backed by a heap whose dequeue order is determined by `options.Comparer` (defaults to `Comparer<T>.Default` when null). Items are dequeued in ascending order — smallest-first per the comparer. For a drain-phase boundary cell, an `IComparer<WorkItem>` that sorts by priority class and then by enqueue timestamp provides a structured drain: high-priority in-flight items drain before low-priority ones. This avoids starvation of high-priority items during a backlogged drain, which a FIFO unbounded channel cannot guarantee.
- `ChannelReader<T>.TryPeek(out T item)` is available on both bounded and prioritized channels but returns `false` on unbounded non-prioritized channels. `CanPeek` is the capability flag. A drain predicate that decides whether to accept new items based on the current queue depth can use `Count` (available when `CanCount` is true) and `TryPeek` together to inspect the head of the prioritized queue without dequeuing. For a backpressure cell that must reject new work when high-priority drain items fill the head, peek-and-count is the structural surface.
- `BoundedChannelFullMode.DropWrite` causes `TryWrite` to return `false` immediately when the channel is full, without blocking or dropping an existing item. This is the correct mode for a boundary cell that surfaces back-pressure as a typed rejection: the caller receives `false` from `TryWrite`, wraps it in a `BoundaryCapacityFault`, and returns it up the `Fin<A>` rail. `DropNewest` and `DropOldest` silently discard items, which obscures loss from the fault rail. `Wait` blocks, which prevents the boundary cell from returning a typed fault synchronously.

[INTERLOCK_SMALL_INT_GATE_CELLS]:
- `Interlocked.CompareExchange(ref byte location, byte value, byte comparand)` and the `sbyte`, `short`, `ushort` overloads are available in .NET 9+. Lifecycle state machines with ≤255 states can encode state in a `byte` field and CAS-transition without widening to `int`. The reduced field size improves cache-line density when multiple independent state cells share an object — four `byte` state fields fit in one cache word versus four `int` fields spanning multiple words. The tradeoff is that the state values must fit in the unsigned range of the chosen type; a state machine that uses negative sentinel values must use `int` or `sbyte` with care.
- `Interlocked.Or(ref int location, int value)` and the `uint`, `long`, `ulong` overloads atomically set bits in a flag word and return the original value. For a lifecycle state cell that tracks multiple concurrent properties as a bitfield — `IsOpen`, `IsDraining`, `HasCorruptedItems` — a single `Interlocked.Or` or `And` call can atomically set or clear a flag without a CAS loop. Reading the current flags requires `Volatile.Read` or `Interlocked.CompareExchange(ref state, state, state)` to get a fresh value with acquire semantics; a plain field read is not coherent on weakly ordered hardware.

[TESTCLOCK_INTEGRATION_FOR_BOUNDARY_TIMEOUT_CELLS]:
- `CancellationTokenSource(TimeSpan, TimeProvider)` and `PeriodicTimer(TimeSpan, TimeProvider)` both accept a `TimeProvider` that drives their respective timers. When the same `FakeTimeProvider` instance is injected into the runtime spine record and into every timeout-scoped `CancellationTokenSource` created from the spine, advancing the fake clock in a test synchronously fires all outstanding timeouts. This gives deterministic coverage of timeout-triggered state transitions — `Draining` on grace period expiry, `InitializationFailed` on setup timeout, `Closed` on idle timeout — without requiring real-time waits in tests. The `TimeProvider` must be part of the spine record, not hidden inside closure captures, for this propagation to be complete.
- `TimeProvider.CreateTimer(callback, state, dueTime, period)` returns `ITimer` whose `Change(TimeSpan, TimeSpan)` reschedules without allocation. When a boundary cell's polling interval must be adaptive — backing off under contention, accelerating under load — the correct cell is `(ITimer timer, Atom<TimeSpan> currentPeriod)` with the atom driving `timer.Change` calls from the `Atom.Change` handler. The handler fires synchronously on the thread that committed the swap, so it must return quickly; `timer.Change` is O(1) and safe to call from the handler.
- `ITimer.Change(TimeSpan.Zero, Timeout.InfiniteTimeSpan)` fires the callback immediately and disables periodic rescheduling — the equivalent of `CancellationTokenSource.CancelAfter(TimeSpan.Zero)` but for a timer-driven boundary signal. This is the correct form for a boundary cell that must immediately trigger a drain-start callback (e.g., on an explicit `Shutdown()` call) when the cell uses a timer as its drain-trigger rather than a `CancellationToken`.

[LOCK_SCOPE_SEMANTICS_VS_MONITOR]:
- `System.Threading.Lock.Enter()` is re-entrant: the same thread can enter multiple times without deadlock, and must exit the same number of times (`Exit()` or `Dispose()` on each `Scope`). This differs from `SemaphoreSlim(1,1)`, which deadlocks on recursive entry from the same thread. For boundary cells that guard process-static initialization but whose callers may themselves be initializers of dependent cells (recursive initialization chains), `Lock` silently permits the recursion while `SemaphoreSlim(1,1)` hard-deadlocks. Choose `SemaphoreSlim(1,1)` explicitly when recursive entry from the same thread must be detected and rejected rather than silently permitted.
- `Lock.EnterScope()` returns `Lock.Scope`, a `ref struct` with a `Dispose()` method. Calling `using var _ = theLock.EnterScope()` is the canonical form; because `Scope` is a `ref struct`, it cannot be boxed, stored in a field, or captured in a lambda — it is mechanically scoped to the method frame. This constraint prevents the pattern of accidentally storing a scope in a closure (the `lock(obj)` equivalent allows no such storage either, but the error mode is different). For a boundary cell that must pass a "locked scope" token to a nested method, the token cannot be a `Lock.Scope`; use `Lock.IsHeldByCurrentThread` as an assertion in the nested method instead.
- `Lock.Scope.Dispose()` throws `SynchronizationLockException` if called from a thread other than the one that entered the lock. This is a correctness guard against async-continuation-based misuse: if a `using Lock.Scope` block spans an `await`, the continuation may resume on a different thread, and `Dispose()` at the end of the `using` block will throw. `Lock` must not span `await` expressions; use `SemaphoreSlim.WaitAsync` / `Release` for asynchronous mutual exclusion.

[ATOMHASHMAP_FINDORMAYBE_ADD_SURFACE]:
- `AtomHashMap<K,V>.FindOrMaybeAdd(K key, Func<Option<V>> None)` is a conditional insertion: if the key is absent, the `None` delegate is called and the result is inserted only when it returns `Some(v)`. If `None` returns `None`, the map is left unchanged and the returned `Option<V>` is `None`. This is the correct surface for conditional resource registration — a factory that may decide not to register based on runtime conditions — without requiring a separate `TryAdd` guard. The factory still executes inside the CAS loop and may run multiple times on contention; it must be pure in the sense that repeated calls produce the same insertion decision (though possibly different `Some` values, in which case only the CAS winner's value is stored).
- `AtomHashMap<K,V>.FindOrMaybeAdd(K, Option<V>)` — the value-direct overload — accepts a pre-computed `Option<V>` without a factory. Inserting a pre-computed `None` is a read-only probe: if the key is absent, nothing is inserted; if present, the stored value is returned. This is the correct surface for a lookup-or-no-op pattern where the absence of the key is a valid terminal state rather than a trigger for insertion. Using `TryAdd` for this pattern would require a dummy value to signal the "no-op" intent.

[CHANNEL_DRAIN_MID_FLIGHT_EXACT_SEMANTICS]:
- `ChannelWriter<T>.TryComplete(Exception? error)` returns `false` when the channel is already completed (idempotent on the `false` path). `Complete(Exception?)` throws `ChannelClosedException` when already completed. For drain-phase boundary cells that may receive concurrent `Shutdown()` calls, `TryComplete` is the safe form; `Complete` requires external synchronization to prevent double-fault. When `error` is non-null, `ChannelReader<T>.Completion` task faults with that exception after drain; callers awaiting `Completion` receive the fault, not a clean `RanToCompletion`.
- `ChannelReader<T>.ReadAllAsync(CancellationToken)` produces an `IAsyncEnumerable<T>` that completes when the channel is drained and completed. Cancellation of the token passed to `ReadAllAsync` cancels the enumeration but does not complete the channel — the channel writer must be separately completed. A drain loop using `ReadAllAsync` that is cancelled mid-drain without the channel being completed will restart enumeration on the next consumer call, rediscovering unread items. The correct drain-then-stop pattern is: cancel the write-side via `TryComplete()`, then `await foreach` with `CancellationToken.None` until the enumerable completes.

[OBJECTDISPOSEDEXCEPTION_THROWIF_TYPE_OVERLOAD]:
- `ObjectDisposedException.ThrowIf(bool, Type)` accepts a `Type` directly rather than an `object` instance. Use this overload inside static methods, class factories, or extension methods where `this` is not available but the type is known at compile time: `ObjectDisposedException.ThrowIf(_closed, typeof(BoundaryCell))`. The `typeof(T)` form is a compile-time constant and avoids the virtual dispatch on `GetType()` that the `object`-accepting overload implies. Both overloads produce equivalent exception messages containing the full type name; the distinction is purely syntactic.
