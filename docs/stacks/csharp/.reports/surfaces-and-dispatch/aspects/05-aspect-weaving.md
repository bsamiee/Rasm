# Aspect Weaving and Advice Points

[CONSTRUCTION_ADVICE_POINTS]:
- The generated factory threads four advice points in a fixed order, and the order is load-bearing: reference-null guards run first, then `validationError` is initialized to `null`, then the mutating advice point `ValidateFactoryArguments` runs, then construction proceeds only when `validationError is null`, then the post-construction advice point `FactoryPostInit` runs on the built instance. A hook that needs to see normalized arguments must be the factory hook, not the post-init hook; a hook that needs the live instance must be the post-init hook. The two are not interchangeable advice slots.
- `ValidateFactoryArguments` is emitted as a `static partial` hook whose every argument is passed by `ref`, so the hook is a normalizer, not merely a validator: writing through the `ref` parameter canonicalizes the value before it reaches the constructor — trimming, casing, range-clamping, or unit-converting at definition time so the interior never re-normalizes. The first parameter is `ref TValidationError? validationError`; setting it to a non-null error short-circuits construction without throwing. A hook that throws instead of setting the error rail breaks the admission contract — the rail bridge expects a returned error, not an exception.
- The author may declare `ValidateFactoryArguments` with a non-`void` return type; the generator detects this by the symbol's return `SpecialType` and threads the returned value into `FactoryPostInit(channelValue)` as a typed side channel. When the hook returns `void`, the generated `FactoryPostInit()` is parameterless. This is a definition-time channel between the pre-construction normalizer and the post-construction initializer with no field, no closure, and no second pass over the arguments — computation done once during validation is handed forward to initialization.
- When a typed channel is declared, the generated `ValidateFactoryArguments` hook is emitted `private static partial`; the void form is emitted `static partial` without an explicit accessibility token. The channel return type detection scans only non-generic `static` members named `ValidateFactoryArguments`; declaring the hook as a generic method makes it invisible to the generator and the channel collapses to `void` silently.
- `ValidateConstructorArguments` is a separate `static partial void` advice point emitted inside the constructor body, before field assignment, with every field passed by `ref`. It always returns `void` — it carries no error rail and no channel. It is the only advice point that fires on every construction path, including the trusted direct-constructor fast path emitted when a factory declares `HasCorrespondingConstructor = true`. A normalization that must hold for store-materialized values as well as wire-admitted values belongs in the constructor hook; one that must run only on untrusted admission belongs in the factory hook.
- The constructor-hook-versus-factory-hook split is the definition-time expression of trusted-versus-untrusted provenance: EF Core materialization via the corresponding-constructor fast path bypasses `ValidateFactoryArguments` and `FactoryPostInit` entirely but still runs `ValidateConstructorArguments`. Placing an expensive admission check in the constructor hook taxes every read from the store; placing an invariant-repair in the factory hook leaves store values unrepaired.

```csharp
// Definition-time advice: normalize-and-channel in one pass, consume in post-init.
public sealed partial class Handle  // [ValueObject<string>] owner elsewhere
{
    static partial void ValidateConstructorArguments(ref string value) =>
        value = value.AsSpan().Trim().ToString();          // fires on every path, incl. store reads

    private static partial int ValidateFactoryArguments(   // non-void => channel exists
        ref ValidationError? validationError, ref string value)
    {
        value = value.ToLowerInvariant();                  // ref-normalize before construct
        var checksum = Crc32(value);                       // computed once during admission
        validationError = checksum is 0 ? Err("empty") : null;
        return checksum;                                   // handed forward, not recomputed
    }

    partial void FactoryPostInit(int checksum) =>          // sees the channel + the live instance
        _registry.Index(this, checksum);
}
```

[ASPECT_ROUTING_REGISTRY]:
- Cross-cutting capability attaches at definition time as data, not code: each `[ObjectFactory<T>]` declaration on an owner emits one `ObjectFactoryMetadata` row carrying `ValueType`, `ValidationErrorType`, the `SerializationFrameworks` flag set, `UseWithEntityFramework`, `UseForModelBinding`, and a `ConvertFromKeyExpressionViaConstructor` lambda expression. An owner exposes its full set through the generated `static IReadOnlyList<ObjectFactoryMetadata> ObjectFactories` member of the factory-owner interface. The serialization, EF, and model-binding adapters are reflection-free consumers that read these rows — the attribute is the policy, the adapter is the mechanism, and the two never share state beyond this row.
- The runtime registry caches metadata per type in a concurrent map and walks base types once on first lookup; the conversion-selection entry point takes two predicate filters — one over the factory rows, one over the keyed metadata — so an adapter selects the row whose `SerializationFrameworks` flag matches its framework and ignores rows registered for other frameworks. One owner can therefore route a binary wire format through one source-type factory and JSON through another, each with its own validation error type, by declaring two attributes with disjoint framework flags; the registry resolves each adapter to its own row with no shared policy and no precedence question.
- `HasCorrespondingConstructor = true` on a factory attribute populates the row's via-constructor conversion expression as a fast path; the EF adapter reads this expression to materialize trusted store values by direct construction, skipping the validating factory while still passing through the constructor advice point. The flag is the only switch that changes which advice points fire for that conversion path — it is policy data on the row, never a runtime branch in the adapter.
- A custom error type implements the validation-error interface's `static abstract Create(string)`; the owner declares it through the validation-error attribute, and every generated factory on that owner produces that error type in its `Validate` signature and threads it through every adapter's `ConvertFromKey`. The type-converter adapter projects a non-null returned error into a format exception at the boundary, so the typed error rail and the exception-throwing host boundary are reconciled at one generated seam — the domain never sees the exception, the host never sees the rail.

[COMPOSITION_TIME_RESOURCE_ASPECT]:
- Resource lifetime attaches as a composition-time aspect through an ambient tracker, not a lexical block: acquiring a resource registers it into the effect environment's resource set and returns the naked value, decoupling acquisition from release. Release happens at the next bracket boundary, at an explicit release of the tracked value, or at scope teardown — the acquire site and the release site are two separate aspect points, so a pipeline can acquire deep inside a bind chain and have the resource freed by an outer bracket without the acquire site naming the release. This is the dual of the definition-time construction advice: there the advice is woven into the type, here it is woven into the effect.
- The zero-argument bracket scopes the ambient resource set to the bracketed computation and frees everything acquired inside it on completion, success or failure; the fail-only bracket frees only on failure, leaving successfully-acquired resources alive for the continuation to return. Selecting the fail-only form where the full bracket was meant leaks every resource the successful path acquired; the distinction is the policy, not a flag on one call.
- The iteration combinators wrap each pass in a bracket, so per-iteration resources are freed before the next iteration begins — the resource aspect composes underneath the schedule aspect automatically. The consequence is a hard rule that the language cannot state but the runtime enforces: a resource acquired inside a repeated or retried computation cannot be returned out of it, because the bracket that bounds the iteration has already released it by the time the loop yields its result.
- The local-scope aspect forks a fresh child resource set off the parent and runs the computation against it, releasing the child set on completion regardless of outcome — the ultimate scoped using, expressed as one combinator with no acquire or release argument. The fork aspect goes further: it allocates a child resource set, a child cancellation source linked to the parent token, and disposes both in a `finally` when the forked task completes — structured-concurrency resource and cancellation scoping is one woven aspect, never an author-written try/finally pair, and parent-acquired resources remain visible-but-shared inside the fork.

[ASPECT_STACKING_ORDER]:
- Definition-time and composition-time aspects stack in opposite directions and must not be conflated. Definition-time aspects are ordered by the generator's emission sequence and are fixed: null-guard, then factory normalizer, then construct, then constructor normalizer, then post-init — the author cannot reorder them, only opt into the slots. Composition-time aspects are ordered by the operator-application sequence in the pipeline and are author-controlled: a retry wrapped around a bracket retries the whole acquire-use-release cycle, while a bracket wrapped around a retry acquires once and retries only the use. The same two aspects in the two orders are two different policies.
- A named catch is a composition-time recovery aspect that composes as a linear chain via the choice operator; the first matching predicate consumes the error, so a broad catch placed before a narrow one shadows the narrow one with no diagnostic. Stacking order is precedence order. A finalizer aspect composes orthogonally to catch and runs on both the success and failure exit of the bracketed region, so a catch that recovers an error still runs the finalizer — recovery and cleanup are separate aspects on separate axes, and ordering them against each other is meaningless.
- The two weaving layers meet at exactly one seam: the admission boundary where a generated factory's typed error rail is lifted into the effect carrier. Above that seam every aspect is composition-time and author-ordered; below it every aspect is definition-time and generator-ordered. A design that pushes retry or recovery policy into a construction hook, or admission normalization into a pipeline combinator, has crossed the seam in the wrong direction — the hook gains hidden control flow, the pipeline gains hidden validation, and both lose the property that the aspect is recoverable from the declaration alone.
