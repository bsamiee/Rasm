# Validation Hook Advice Channel

[CANONICAL_INPUT_ORDERING]:
- Normalization through the factory hook's `ref` parameters completes before construction begins, so the constructor-scoped hook observes only canonical input and never re-normalizes — the two hooks form an ordered pipeline, not two independent guards, and an invariant written in the constructor hook may assume the factory path has already trimmed, clamped, and canonalized. The ordering inverts on bypass routes: rehydration construction runs only the constructor hook against stored values, so any normalization that must also hold for stored material is either re-derivable from the stored form or it silently does not exist on that path.
- The generated per-member null guards that precede the hook mint their faults with text naming the offending member and the owning type, in member declaration order — field attribution on the null tier is generator-authored, so a flat fault stream from a multi-member admission never loses which member produced the rejection even before any hand-written labeling at the bridge.

[ONE_SHOT_EVIDENCE_CHANNEL]:
- A non-`void` hook return is the only channel for evidence the members themselves cannot hold: by the time the instance exists its members are `init`-frozen, so a value computed during validation — the pre-normalization original, a parsed sidecar, a derived classification — cannot be stored on the instance after the fact and cannot be a member without widening the public surface. The hook return threads it across the construction boundary into the post-construction partial, where derived caches prime against it exactly once.
- The pair is transactional admission: construction and the post-construction partial both sit inside the success branch of the generated `Validate`, so a rejected candidate never constructs and never posts — a side effect placed in the post hook (one-time registration, derived-field publish, diagnostic emit) is reached if and only if the value is genuinely admitted, with no half-constructed instance ever observable to it.

[TIGHTENING_PLACEMENT_LAW]:
- Invariant tightening is the one admission change with zero compile signal — the factory narrows silently while values persisted under the looser rule still exist — and the hook chosen for the tightened check decides the failure mode of that drift. Placed in the factory hook, the tightening rejects fresh input with a recoverable typed error while old stored data continues to rehydrate through the bypass constructor; placed in the constructor hook, it throws on the rehydration path too and converts invariant drift into a load-time fault. The first placement makes tightening a data-migration decision with a code rider; the second makes it an availability decision — and because the type system stays silent either way, the proof obligation lives entirely in tests against both construction routes.
- The split generalizes: the factory hook owns every rejection that may legitimately occur on fresh input, because it is the only hook with an error channel; the constructor hook owns the invariant-of-record that must hold on every birth path including deserialization and trusted rehydration, and can only throw. An invariant placed in the factory hook alone is a statement that stored data is exempt from it — that statement should be deliberate, never an accident of which partial was convenient.
