The three tools form one quality surface and are held to one standard, enshrined here so the next session starts from it (each tool's corpus carries its own specialization):

### Cross-invocation resource governance — machine-altitude admission (verified live 2026-06-12, loadavg 48)
Every existing protection operates at the wrong altitude for multi-agent contention: the flock lease serializes IDENTICAL closure keys only (two different closures fan freely in parallel); `governed_concurrency` caps each invocation's OWN fan — memory backpressure included — but is blind to system load and to sibling invocations; and `--disable-build-servers` (the deliberate isolation posture) makes every concurrent build maximally expensive — full cold process trees, no shared compiler. Observed: one 8-project static-build fan overlapping two agents' dotnet/Rhino brackets drove 1-min loadavg to 48 with zero protection triggering. Fix directions, in order of invariant strength: (1) **machine-wide dotnet admission slots** — one flock-counted slot pool (e.g. `locks/machine.dotnet`) acquired per spawned dotnet process across ALL invocations; bounds the total exactly; busy = queue with a visible envelope note, never a silent pile-on. Slot unit DECIDED: dotnet-family process census (= MSBuild node count in practice), NOT per-invocation; land together with (3) since re-enabled servers make node accounting smaller and more stable. (2) **reactive census backpressure — LANDED 2026-06-12**: `governed_concurrency` halves DOTNET batches when the foreign dotnet/MSBuild/VBCSCompiler process census (own tree excluded) reaches `cpu_count` (`_foreign_dotnet_count`, law-pinned incl. the non-DOTNET-batch exclusion and telemetry fields). This is the census substitute for loadavg, which stays rejected (macOS load1 counts uninterruptible waits — the in-code rejection comment stands). (3) **shared-compilation re-enable** (research-verified: VBCSCompiler pipe identity is per-user+per-SDK, stateless, isolation-safe with per-closure `--artifacts-path`) — reduces per-build amplitude so contention costs less when it happens; ride the existing A/B criteria; (4) pressure facts surface AUTOMATICALLY in the envelope per the agent-feedback law (an orchestrating agent must see "queued 40s behind 2 invocations / census 14" without running a probe — the backpressure event exists in telemetry; the envelope note does not yet). The previously documented rootdir/cwd theory is **falsified** (the captured stats argv is well-formed and pytest loads the parent pyproject config — the prepended addopts prove it). Root cause, established empirically with `debug=true`: with `max_stack_depth != -1`, `record_trampoline_hit` (`mutmut/__main__.py:130`) walks caller frames calling `Path(co_filename).resolve(strict=True)`; module-level calls into mutated code during conftest import put `<frozen importlib._bootstrap>` pseudo-filenames on that stack, the strict resolve raises `FileNotFoundError`, the conftest import dies, pytest exits 4, and mutmut raises `BadTestExecutionCommandsException`. The local fix is the upstream-default `max_stack_depth = -1` (repo had 10), which disables the strict-resolve stack walk entirely. Anyone with `max_stack_depth != -1` plus import-time calls into mutated code still hits this — file upstream as-is:

> **TITLE**: record_trampoline_hit crashes with FileNotFoundError on '<frozen importlib._bootstrap>' frames when max_stack_depth != -1 and mutated code runs at import time.
> **BODY**: With max_stack_depth set (e.g. the README's max_stack_depth=8), record_trampoline_hit (mutmut/__main__.py, the `while c and f:` loop) calls Path(f.f_code.co_filename).resolve(strict=True) on every caller frame. When a mutated function is invoked at module import time (module-level constant computed by a mutated function, imported transitively from a conftest), the caller stack contains importlib machinery frames whose co_filename is the pseudo-filename '<frozen importlib._bootstrap>' (also '<frozen importlib._bootstrap_external>', '<string>'). strict=True raises FileNotFoundError('<cwd>/<frozen importlib._bootstrap>'), the conftest import aborts, pytest exits 4, and mutmut raises BadTestExecutionCommandsException — misleadingly suggesting the pytest command itself is malformed.
> **REPRO (minimal)**: src/m.py with `def cap() -> int: return 1`; src/pkg using `X = cap()` at module level; a conftest.py importing pkg; [tool.mutmut] source_paths=['src'], max_stack_depth=8; `mutmut run` -> stats phase exit 4 with the traceback above; with max_stack_depth=-1 the run proceeds.
> **OBSERVED ON**: mutmut 3.6.0, pytest 9.0.3, CPython 3.15.0b1, macOS.
> **FIX CANDIDATES**: (a) skip frames whose co_filename starts with '<' before resolving; (b) resolve with strict=False; (c) wrap the resolve in try/except FileNotFoundError and continue; (d) break on importlib frames the same way 'pytest'/'unittest' filenames break the walk. Additionally, the exit-4 path would be far easier to debug if BadTestExecutionCommandsException included the captured pytest output even without debug=true.

When filing, also mention the forked-children consequence: mutant workers fork from the post-clean parent, so any set-once global poisoned across in-process `pytest.main()` calls silently converts test-infrastructure failures into false "killed" verdicts — an argument for subprocess-isolated stats/clean phases in mutmut core.

### [UPSTREAM] cyclopts help renderer crashes on Enum-hinted params with None defaults
cyclopts 4.16.1 `help/help.py` (the `if argument.show_default:` block) renders Enum-hinted defaults via `default_val.name` BEFORE consulting any `show_default` callable, so `language: Language | None = None` crashes `--help` with `AttributeError: 'NoneType' object has no attribute 'name'` regardless of the callable form (`show_default=lambda d: "" if d is None else d.name` was tried and is structurally unreachable for Enum hints). Local posture: `Parameter(show_default=False)` on the field with a source-cited comment; the law `test_main_enum_param_help_renders` pins the regression. Revisit on cyclopts bump.
