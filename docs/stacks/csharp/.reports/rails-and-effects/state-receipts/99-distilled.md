# State-Receipts

[CELL_AND_THREAD_SELECTION]:
- The selection law: one producer inside one run threads the receipt through a transformer channel; many producers or many runs fold into a cell. A design that wants both layers the threaded receipt per run and folds each run's final output into the cell once at the run edge — one swap per run instead of one per fact.
- A threaded receipt dies at the run boundary: cross-run accumulation, cross-thread visibility, or observation by code outside the pipeline requires a cell. Within one run the thread is strictly stronger — pure, replayable, free of the multiple-execution-under-contention constraint on every cell swap function, and free of the subscribe-then-snapshot initialization discipline cell observers need.
- Contended append-only streams stay on a cell because the monoid thread has no concurrent writers by construction: the cell's CAS is the serialization point, and batching is the contention lever — `AtomSeq.Add` runs an independent CAS per element, while `AtomSeq.Swap(s => s.Add(...))` lands multiple appends in one CAS.
- Within the cell branch: `Atom<T>` owns independent single-ownership state; `Ref<T>` earns its STM overhead only when two or more `Ref`s must transition atomically — a single `Ref` inside `atomic()` matches `Atom.Swap` guarantees while adding transaction overhead and `atomic()` wrapping at every write site. `AtomHashMap<K,V>` beats `Atom<HashMap<K,V>>` when key-level concurrent mutations are common or typed change diffs drive observers: key-level swaps and a typed patch versus whole-map replacement per operation and a whole-value `Change` event.

[FACT_STREAM_VERSUS_TYPED_RECEIPT]:
- One `FactRecord` shape carries a kind discriminant (smart-enum or closed-union case), a slot identifier, a timestamp, and a payload union; all mutation buckets — adds, updates, removals, errors — are kind cases, never parallel record types. The cell is `Atom<Seq<FactRecord>>` or `AtomSeq<FactRecord>`, and the kind-discriminated `Seq` is the monoidal surface: empty identity, ordered append.
- Derived projections are pure folds, never additional cells: filter-by-kind yields a bucket, group-by-slot last-wins yields current state — one stream simultaneously supports replay (fold from origin), current-state projection, and audit (full chronology) without separate state cells.
- Typed algorithm receipts are neither monoids nor streams: single evidence records carrying solver path, sampling parameters, residual, iteration count, and a status discriminant; the consumer pattern-matches status for routing and solver-specific fields for audit. `Atom<ReceiptRecord>` holds the most recent result; history escalates to `Atom<Seq<ReceiptRecord>>`.
- The split is capability, not taste: streams answer "what happened and when" and derive state by fold; receipts answer "how did this specific computation resolve". Collapsing receipts into a stream erases the typed solver and sampling evidence that applies to one algorithm's output; collapsing a stream into a receipt erases ordering and temporal projection. The kind discriminant is categorical; the status field is algorithmic provenance.
- Multi-run receipt combination folds `Seq<Fin<ReceiptRecord>>`: sequence for fail-fast (ordered — the first failure halts), partition when partial success matters; the caller, never the receipt, owns the choice.

[THREADED_STATE_CHANNEL]:
- The state transformer wraps `Func<S, K<M, (A Value, S State)>>` — state is the run argument threaded in and the second tuple element threaded out, never a field. Each bind runs with the incoming state, extracts the advanced state from the inner monad, and feeds it into the continuation's run; the consumer reads the final receipt as the second tuple element of the top-level run — accumulation with no atom, no spin loop, and no contention semantics.
- Threading is strictly left-to-right and builds a left-nested tower of inner-bind closures forced only at the final run: a long fold costs one closure per step, not intermediate state-cell allocation. `Map` passes the state untouched beside the projected value; `pure` injects through the inner monad and pairs the incoming state unchanged.
- The state-effect vocabulary is a trait, not methods on the type: the whole-state read derives from the read-and-map form applied to identity — one declared correspondence yields the projection family, never a second primitive.

[WRITER_DIFFERENCE_LIST]:
- The log transformer wraps `Func<W, K<M, (A Value, W Output)>>` under a monoid constraint and threads the accumulator in as an argument: the run seeds with the monoid identity, and each bind feeds the first computation's accumulated output as the second's input accumulator. This difference-list shape avoids the quadratic left-nested combine of a log-returning writer — the combine is one threaded parameter, appended on the right.
- Emission appends the new fact as the right operand of the existing accumulator, so a fold of receipt emissions lands in emission order. The listen, pass, and censor projections each call the seeded run and re-prepend the prior accumulator — nested inside a tight loop they reconstruct the monoid combine per iteration; the cheap form is one outer listen or censor over the whole fold, never one per step.
- A fact stream expressed as the writer accumulator is a monoid append threaded through the effect with no cell, no observer subscription, and no delivery-window discipline: the stream is the transformer output, and replay, current-state projection, and audit are folds over the single final accumulator.

[RWST_FUSED_CHANNELS]:
- The fused reader-writer-state transformer collapses all three threadings into one wrapped function over a triple, and one bind threads every channel in its correct discipline simultaneously — environment held constant, log and state advanced:

```csharp
new RWST<R, W, S, M, B>(input =>
    ma.runRWST(input).Bind(x =>
        f(x.Value).runRWST((input.Env, x.Output, x.State))));
```

- The constant-environment / advanced-output-and-state split is visible in the single continuation call: the environment is re-supplied unchanged while output and state carry forward. This is the canonical shape for an effect that reads a runtime, emits a receipt log, and threads accumulating algorithm state in one pass — replacing three parallel cells and three observer wirings with one function and one fold over the final triple. The modify, gets, censor, and local projections are case projections of this single carrier, not sibling surfaces.

[ATOM_SWAP_MECHANICS]:
- `Swap` is a CAS spin loop — read value, apply `f`, validate, `Interlocked.CompareExchange`, `SpinOnce` and retry on CAS failure — so `f` runs multiple times under contention and any side effect inside it repeats. `SwapMaybe` breaks the loop early when `f` returns `None`, leaving the atom unchanged and returning the current value without retrying — the rejection is final.
- The validator runs on every CAS attempt and receives only the proposed new value, never the current one; rejection exits the loop immediately, returning the pre-swap snapshot — the loop exits on validator rejection, not on a failed CAS, so a rejecting validator is invoked exactly once per `Swap` call regardless of contention.
- First-writer-wins cannot be expressed by a validator: it cannot see whether the state has already moved off a sentinel. The mechanism is `SwapMaybe` — the swap function inspects the incoming value and returns `None` to abort (atom undisturbed) or `Some` to proceed.
- `Box<A>` dispatches at static construction: value-type reads unbox without allocation while every value-type write allocates a fresh `Box<A>`; reference types pass through a CIL-emitted identity (plain cast where emit is unavailable), so `Box<A>.New` returns the same reference. An idempotent reference-type swap therefore yields the same pointer, the CAS succeeds against itself (absent a concurrent writer), and `Change` fires with the unchanged value — no identity-swap suppression exists; deduplication is the observer's comparison or `SwapMaybe`'s `None`.
- `Swap` returns the pre-swap value on validator rejection and the new value on success; under an identity swap the two are indistinguishable — `SwapMaybe` is what makes the no-op path explicit.
- `Change` fires synchronously on the calling thread after CAS success and before `Swap` returns; there is no dispatch queue — a slow handler throttles the swapper directly.

[ATOM_SURFACE_ASYMMETRIES]:
- Construction is asymmetric across factories: `Atom(value)` returns `Atom<A>` directly; `Atom(value, validator)` returns `Option<Atom<A>>` where `None` means the seed failed validation and the atom was never constructed; `atomIO(value, validator)` throws `Exceptions.ValidationFailed` inside the `IO` — seed rejection lands on the error rail, not as `IO<Option<Atom<A>>>`. Handling rejection without an error rail requires the synchronous factory.
- The `ValueIO` set accessors on `Atom<A>`, `Atom<M,A>`, and `Ref<A>` build a lazy `IO` via `Bind`/`Map` and discard it unevaluated — assignment is a structural no-op and no state changes. Writes go through `Swap`/`SwapIO` on atoms and `Value`/`Swap` inside `atomic()` on refs.
- `SwapIO` and `SwapMaybeIO` are `IO.lift` wrappers over synchronous `Swap`: no deferral, no scheduling — the spin loop runs at `IO` evaluation time on the evaluating thread.
- `Atom<M,A>` metadata is a private readonly field that flows only as the first argument of its `Swap` overloads; the validator receives only the proposed `A`, never `M`. Context-dependent invariants encode in the swap function itself — `SwapMaybe` returning `None` on violation. The metadata form fits stable external context (a capacity limit, a schema version, a generation token); a requirement to update the metadata signals the context belongs inside the state value.

[ATOM_COLLECTION_FAMILIES]:
- `AtomHashMap` scalar mutations check `Change != null` inside the CAS spin loop: unsubscribed maps take the plain trie path with no `Change<V>` allocation; a single subscriber forces the change-tracking path on every scalar operation. Bulk `Swap` always constructs a `TrackingHashMap` regardless of subscription because the observer check happens post-CAS in announcement — the constant wrapper cost on never-subscribed workloads is the price of the unified API.
- Bulk patches combine per-key through the `Change<A>` monoid and deliver net deltas, not intermediates: `NoChange` is identity, `x + Removed = Removed`, `Removed + Added(v) = Mapped(old, v)`, `Added + Mapped = Added` — an insert-then-update inside one bulk `Swap` delivers a single `EntryAdded`, and `EntryMapped` arrives only when a pre-existing key is updated, so code distinguishing insertion from in-place update must use scalar `SwapKey` atomics. Pattern-match `EntryMappedTo<V>` when the prior value is irrelevant, avoiding a bind to the `From` type parameter under type-changing mutation.
- `HashMapPatch<K,V>` stores scalar ops as a `(key, change)` pair and bulk ops as a trie; `Changes` builds a `HashMap` on demand and no scalar accessor exists — high-frequency scalar observers enumerate `Changes` directly or subscribe the typed scalar delegates that receive `(K, Change<V>)` with no patch allocation.
- The subscription race window is bounded between the `Change != null` check and the committing CAS: a single trie operation for scalar ops, widening proportionally with range size for `AddRange`/`TryAddRange`. Observers requiring guaranteed delivery from their subscription point snapshot `Items` immediately after subscribing.
- `AtomSeq` runs the same CAS against its internal sequence with no tracking wrapper, no diff, and no change event. `AtomQue` fires distinct `Dequeued`/`Enqueued` events; `Clear` notifies `Dequeued` only for elements captured at CAS success — a concurrent post-CAS `Enqueue` goes unnotified, so full accounting drains via a `Dequeue` loop. No functional `Swap` exists on `AtomQue`; multi-step conditional operations escalate to `Atom<Que<A>>` or a `Ref` inside `atomic()`.
- `ToAtom<K,V>(this HashMap<K,V>)` is the canonical upgrade from immutable snapshot to concurrent cell.

[STM_COMMIT_MODEL]:
- `Ref<A>` is a `long Id` keyed into a process-global concurrent map of ref states; the `Value` setter writes into the current thread's transaction or throws when none is active, and `Ref.Swap(f)` compiles to `Value = f(Value)` — property reads and writes that require an enclosing `atomic()` and are never independently thread-safe.
- `atomic()` checks the `AsyncLocal` transaction first: a nested call joins the outermost transaction without creating a new retry scope.
- `Isolation.Snapshot` is optimistic write-lock, not database snapshot isolation: write-write conflicts are detected at commit by version comparison (conflict, then retry), but read-set validation is skipped — non-repeatable and phantom reads remain possible while concurrent same-`Ref` writers still conflict. `Serialisable` additionally validates the read-set before commit. The read-set accumulates unconditionally on every transactional read in both modes, so under `Snapshot` it accrues with no commit-time benefit — prefer `Serialisable` only when read-set validation is required.
- The entire commit — validation, write application, commute application — executes inside one CAS spin on the global state cell; two concurrent commits compete there and the loser re-spins against the updated snapshot. This is the only global serialization point for STM.
- `commute` stages the function in the transaction's commutes list; commutes apply after writes against the freshly committed state, so commuted functions see values already advanced by the write set. A `Ref` in both sets gets the version-verified write first, then the commute on that result; a validation failure inside commute application escapes `atomic()` without retry.
- `Ref.Value` outside `atomic()` bypasses read tracking and reads the volatile trie directly — a visibility-safe point read.

[STM_HAZARDS]:
- `Task.Run` inside `atomic()` copies the `ExecutionContext` including the `AsyncLocal` transaction, so spawned work can write `Ref`s without throwing; the hazard is unsynchronized mutation of the shared transaction's non-thread-safe state — undefined behavior at commit, not an exception at write time. Spawned tasks must not touch `Ref`s through the inherited transaction.
- `implicit operator A(Ref<A>)` routes through the transactional read, which inside `atomic()` unconditionally adds the `Id` to the read-set: overload resolution, LINQ expressions, and operator positions silently broaden the `Serialisable` conflict surface. An explicit `.Value` read carries the same tracking but is visible at the call site.
- `Ref<A>` has a finalizer that removes its global-map entry: a `Ref` reachable only from inside the `atomic` lambda can be finalized between the snapshot read and commit, making the commit throw a missing-key `ArgumentException` — not a conflict, not retried, escaping `atomic()` directly. The hazard compounds with async state machines where a local `Ref` is not live across a suspension frame; every `Ref` used in a transaction must be strongly rooted outside the lambda for its full duration.
- A `Ref` validator rejection hard-aborts the `atomic` block rather than retrying — validators are only for invariants that must never be violated; skippable transitions belong in the `Swap` or `commute` function logic.

[MEMO_EXACTLY_ONCE]:
- `Memo<A>` drives exactly-once acquisition through a three-state CAS (unloaded, loading, loaded); an exception resets to unloaded and rethrows, leaving the memo re-acquirable. A second thread spins without timeout while loading — the acquire function must never block on a memo-protected resource.
- `Reset()` CASes loaded to loading, then clears the value and state as plain stores; weakly-ordered hardware may expose the unloaded state before the value clear — benign because the loading winner overwrites, and the design contract already prohibits non-idempotent acquire. `Reset` is a no-op on literal-value memos. `Clone()` yields an independent unloaded memo sharing the acquire delegate — independent acquisition lifecycles without coordinating through the original.
- `memo(f)` caches through weak references whose finalizers self-evict: an unretained result may recompute on the very next call. The per-key lock — `GetOrAdd` a lock object, `TryRemove` it after the protected block — prevents double computation only within one GC cycle and only for callers arriving after the removal; the pattern is correct exclusively for idempotent computations. `memoUnsafe(f)` holds values permanently in a plain concurrent dictionary with the same lock pattern.
- `memoK` caches construction of the `K<F,A>`, not its evaluation — the force operator returns a carrier that, if itself lazy, still re-evaluates per run; force at execution time, never at composition time.
- A frequently re-queried fact-stream projection memoizes via `memo` over a stream snapshot: the weak-reference semantics collect snapshots with no live references, so the memo table stays bounded as the stream evolves.

[STRUCTURAL_COST_FLOOR]:
- The CHAMP trie under `HashMap`, `AtomHashMap`, and `HashSet` threads a mutate flag through every node update: every public call on an escaped map forces path-copy per update, and in-place mutation runs only on unescaped tries — batch constructors, change-log accumulation, and the STM transaction write path — which is exactly what keeps the sharing guarantee safe.
- A single-key update allocates one cloned node array per trie level, and depth is bounded at six routing levels (5-bit hash sections, collision past the last offset) — per-update allocation scales with trie depth, never map cardinality. The collision node hardcodes copy-on-write with linear scan regardless of the mutate flag: a custom `Eq<K>` hash that collapses key families degrades those families to full-clone linear updates, losing all path-copy sharing.
- A concat-backed sequence computes `Count` by summing segments, so `AtomSeq.Count` on a concat-heavy cell is O(segments), not O(1) — observers that read the count per event maintain a separate `Atom<int>`. Per-element `Add` on a concat backbone rebuilds the segment list each call; interleaved bulk `Concat` and per-element `Add` batches through one `Swap`.
- Strict-sequence append wins a single-use in-place widening gate (`Interlocked.Exchange` on the gate field) to share the backing array when capacity allows; the returned wrapper restores the gate, so only the instance that lost its gate stays blocked, and every clone path reconstructs a fresh gate.
