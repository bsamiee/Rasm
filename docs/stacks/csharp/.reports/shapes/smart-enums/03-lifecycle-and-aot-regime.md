# [SMART_ENUM_LIFECYCLE_AND_AOT_REGIME]

[MATERIALIZATION_CHOKEPOINT_AND_POISONING]:
- The entire runtime state of a keyed vocabulary funnels through one `static readonly Lazy<Lookups>` constructed with `ExecutionAndPublication`, where `Lookups` is a `readonly record struct` holding the frozen dictionary, the optional span alternate view, and the read-only item list. Every admission verb — `Get`, `TryGet`, `Validate`, every span overload — opens with `_lookups.Value.Lookup.TryGetValue(...)`, and `Items` is `_lookups.Value.List`. There is no second entry into the state: the lazy is the sole publication point, so the question of "is this vocabulary initialized" reduces to "has `_lookups.Value` been read."
- A keyless vocabulary substitutes a `static readonly Lazy<IReadOnlyList<T>> _items` for the lookup lazy because it has no dictionary; its `Items` is `_items.Value`. The two lazy fields are mutually exclusive per declaration — a vocabulary owns exactly one materialization lazy, and dispatch always forces the one its `Items` accessor reads, so no vocabulary runs two independent materializers that could both attempt to publish indices.
- Materialization is a single closure that builds the mutable dictionary with the configured equality comparer accessor (ordinal-ignore-case for string keys absent an accessor), then runs a local `AddItem` per item in declaration order. `AddItem` rejects a null item, rejects a null key for nullable-key types, rejects a duplicate key, assigns the dispatch index via `item._itemIndex.Set(list.Count)`, then appends. The frozen dictionary is built from the mutable one with the same comparer instance, so the build-time and freeze-time equality laws cannot diverge by construction.
- `ExecutionAndPublication` means a closure that throws caches the thrown exception instance: the lazy never retries, and the identical exception object resurfaces on every subsequent `Items`, `Get`, `TryGet`, `Validate`, `Parse`, conversion, serializer, and dispatch touch for the life of the load context. A duplicate-key or null-key defect is therefore not a per-call failure but a one-time process poisoning deferred to first touch — recovery is restart-only, which is why forcing `Items` in a startup probe is the only place this defect class is cheap to surface deterministically.

[TWO_LAYER_LAZY_PROJECTION]:
- The metadata plane is a second lazy stacked atop the first. The generated `IMetadataOwner.Metadata` static auto-property carries its own `Items` as `new Lazy<IReadOnlyList<SmartEnumItemMetadata>>(() => Enumerable.Select(T.Items, (item, index) => ...))` — it projects over the vocabulary's own `Items` accessor, which is itself `_lookups.Value.List`. The metadata item table does not re-materialize the vocabulary; it subscribes to the same single materialization through the public accessor, so it inherits the duplicate-key fail-fast and the cached-exception poisoning transitively. A serializer or ORM that walks the metadata `Items` triggers the identical poisoning the application's own admission would.
- The `index switch` inside that projection maps each ordinal position to its field-name identifier literal, so the runtime identifier table is recovered without reflection and without a per-item name field — declaration order is the join key between the materialized item list and the compile-time-known identifier strings.

[STATIC_INITIALIZATION_TIMELINE]:
- The `Metadata` static property is a static auto-property with an inline initializer, so it materializes in the type's static constructor — first access to any static member of the vocabulary triggers it. The initializer is cheap: it builds the metadata record carrying delegates and lazies but forces no item materialization, because both the vocabulary `Items` and the metadata `Items` are lazies. Heavy work is uniformly deferred to first lazy touch, never to type load. A vocabulary that is referenced but never admitted from pays nothing beyond the metadata record allocation.
- Item field initializers run in declaration order inside the static constructor before any lazy can fire, so a row whose constructor argument names a later-declared item captures null silently — the materialization guard checks item and key non-nullness, not what an item's constructor captured. Cross-row references and cyclic successor graphs must defer behind a `static () => Other` delegate or derive as a lazy projection from the item list; the by-ref constructor-validation hook is the declaration-time place to reject a null cross-reference when eager capture was intended.
- Dispatch over an unmaterialized vocabulary is self-healing in timing terms: the index read falls negative, the slow path forces `Items`, and the same single pass that publishes the lookup publishes every index, so a dispatch issued before any admission still resolves — it pays the materialization on its own thread rather than failing. Reading through `Items` is the happens-before edge that subscribes a consumer to that pass, which is why a derived index must project from the item accessor and never from an eager static field that can run before the pass and observe sentinel indices and sentinel cross-references.

[DUAL_CONVERSION_REPRESENTATION]:
- The metadata carries key conversion in two parallel forms with identical bodies: a compiled `Delegate ConvertFromKey = static T (TKey key) => T.Get(key)` and a `LambdaExpression ConvertFromKeyExpression` with the same shape. The delegate is the direct runtime path; the expression tree exists so a query provider can splice key conversion into a larger expression it will translate or compile. The `ConvertToKey`/`ConvertToKeyExpression` pair mirrors this on the projection side, and a boxing `GetKey = static object (object item) => ((T)item).Key` serves reflection-plane callers that hold `object`.
- The two conversion routes select by failure shape, not just representation: `ConvertFromKey` routes through throwing `Get`, while the metadata's `TryGetFromKey` routes through null-returning `Validate` with a boxed-key value pattern `key is TKey typedKey ? typedKey : default`. A boundary that feeds a wrong-typed boxed key validates the default key and fails with the empty-key message rather than a type error — the boxed plane swallows a type fault into an admission failure, so a generic caller holding `object` keys must pre-type before entering it.

[AOT_AND_TRIMMING_CLIFF]:
- A non-generic vocabulary's serializer participation is fully static: the converter-factory attribute is stamped with both type arguments resolved at generation time (`ThinktectureJsonConverterFactory<T, TError>`, or the span-parsable factory above the span plane), so no runtime `MakeGenericType` and no dynamic code is required — the path is trim- and AOT-clean.
- A generic vocabulary cannot resolve its closed converter at generation time, so the generator emits a `file`-scoped `JsonConverterFactory` whose `CreateConverter` calls `typeof(ThinktectureJsonConverter<,>).MakeGenericType(typeToConvert, typeof(TError))` against the runtime `Type`. `MakeGenericType` over a generic definition is a dynamic-code operation, so a generic vocabulary's JSON converter is the AOT cliff: the same declaration that is AOT-clean when closed at definition becomes dynamic-code-dependent the moment it is left open and constructed per closed instantiation at runtime.
- The component-model type converter is suppressed entirely when the vocabulary or any containing type is generic, while the factory and metadata conversion routes survive — generic vocabularies lose only the design-time type-descriptor seam, not admission. The serializer-integration converters generate per framework only when the matching integration assembly is referenced by the compilation, so a per-enum framework flag naming an absent integration is inert rather than an error, and the integration set is a compile-time fact, not a runtime probe.

[LOAD_CONTEXT_ISOLATION_BOUNDARY]:
- Items are static singletons, and static state is per load context, so an assembly loaded into multiple isolation contexts yields disjoint item universes: instances from one context fail both reference identity and runtime type identity against the other. The lazy materialization state, the published dispatch indices, and the cached poisoning are all per context — a vocabulary poisoned in one context is healthy in another loaded from the same source.
- Vocabulary values therefore cross an isolation seam as keys re-admitted on the far side, never as instances; an instance marshaled across the boundary is an alien singleton that no lookup on the far side will reference-equal. The reflection metadata cache is a process-wide `ConcurrentDictionary<Type, Metadata>` keyed by the concrete queried type, but `Type` identity is itself per context, so the cache partitions cleanly along the same seam rather than aliasing one context's metadata onto another's type.

[METADATA_CACHE_RESOLUTION_REGIME]:
- The runtime metadata lookup fast-rejects non-candidates — primitives, arrays, language enums, pointers — and unwraps nullable before any cache interaction, so probing arbitrary types in a hot serializer path neither allocates nor poisons the cache with non-candidates. The single reflection touch is one static non-public property read per type, performed once and cached; a type that is not an `IMetadataOwner` is rejected without caching, so a negative is re-probed cheaply rather than memoized.
- Base-type walking resolves a derived-case instance's concrete type to the owning vocabulary's metadata: the lookup climbs `BaseType` reading the static metadata property, so an integration handed a case instance finds the family without special-casing, and the result caches at the queried concrete type. Conversion resolution over the object-factory list is factory-first and last-wins — `LastOrDefault` over the factory metadata under the integration's filter — so object-factory attribute declaration order is a precedence list, and a serialization-claiming factory rewires every metadata-driven integration that resolves conversion at runtime, not merely the converters stamped on the type.

[STARTUP_PROBE_AS_THE_ONLY_CHEAP_SEAM]:
- Because materialization is deferred and its failure is a process-lifetime poisoning, the deterministic place to convert a latent duplicate-key, null-key, or null-cross-reference defect into a build-adjacent failure is a startup probe that forces `Items` on every vocabulary once. A generic admission constraint over the factory interface lets one probe walk every keyed generated shape through `T.Validate` or a forced enumeration, so a single pass surfaces the whole program's materialization defects at startup rather than at the first production admission. Past that probe, every code path holding an item never re-checks validity — validity is a property of keys resolved at admission, and the item is valid by construction.
