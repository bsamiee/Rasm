# State-Receipts: Frontier

[TRIE_STRUCTURAL_SHARING]:
- The CHAMP trie underlying `HashMap<K,V>`, `AtomHashMap<K,V>`, and `HashSet<K>` threads an `(UpdateType Type, bool Mutate)` env tuple through every node `Update` call. `Mutate: true` is passed only inside the `TrieMap(IEnumerable<(K,V)>)` and `TrieMap(ReadOnlySpan<(K,V)>)` constructors — a single owner mutates its own freshly-allocated nodes during batch construction. Every public API call after construction passes `Mutate: false`, which forces path-copy on every update. The immutable sharing guarantee is enforced by the flag at each node rewrite, not by field finality.
- A single-key update on a map with trie depth `d` allocates `d` new `Entries` nodes, each with a freshly cloned `Nodes[]` array. Map depth is bounded by hash collision depth: distinct keys with disjoint hash bits share at most 7 levels (32-bit hash / 5-bit sections). Per-key update allocation scales with trie depth, not map cardinality.
- The `Collision` node type handles identical-hash keys with a flat `(K,V)[]`. Every `Collision.Update` clones this array on every write regardless of the `Mutate` flag — `Collision` does not thread `Mutate` into its internal `SetItem`. Maps using an `Eq<K>.GetHashCode` that collapses many keys into collision nodes degrade to linear scan and full clone on every update, losing all path-copy sharing benefit.

[SEQ_INTERNAL_STRUCTURAL_COSTS]:
- `SeqConcat<A>` (produced when `AtomSeq.Concat(Seq<A>)` merges two segments) computes `Count` as `ms.Sum(s => s.Count)` — an O(segments) traversal each time `Count` is accessed. `AtomSeq<A>.Count` delegates to the internal's `Count`, so reading `AtomSeq.Count` on a concat-heavy sequence is not O(1). Observers that call `Count` per event on a frequently-concatenated `AtomSeq` should maintain a separate `Atom<int>` counter.
- `SeqConcat<A>.Add(value)` calls `ms.Last.ValueUnsafe()?.Add(value)` and rebuilds the `ms` list via `ms.Take(ms.Count - 1).Add(value2)`. `ms` is itself a `Seq<ISeqInternal<A>>`, so `Take` and `Add` on it traverse segments. Workloads that interleave bulk `Concat` with per-element `Add` should batch adds via `AtomSeq.Swap(s => s.Add(...))` inside one CAS rather than interleaving individual calls.
- `SeqStrict.Append(right)` has a single-use in-place write gate: `Interlocked.Exchange(ref addDisallowed, 1)`. The first call that wins can copy `right.data` directly into the pre-allocated suffix space (if capacity allows), sharing the backing array. The second call finds `addDisallowed == 1` and always allocates a new array. The gate is not reset by `Clone` — two `AtomSeq.Concat` calls on a freshly-strict sequence: the first may widen in place, the second forces a full copy.

[ATOM_SWAP_REFERENCE_TYPE_CHANGE_HAZARD]:
- `Box<A>.New` for reference types compiles (via CIL emit) to an identity function (`ldarg.0`, `ret`) — it returns the object reference unchanged, allocating nothing. When `Atom<A>.Swap(f)` calls `Box<A>.New(val2)` for a reference-type `A`, `obj2` is the same object reference as `val2`. If `f` returns the exact same reference as `val` (an idempotent or no-op swap), then `obj2 == obj` (same pointer), the `Interlocked.CompareExchange` succeeds unconditionally (comparing the current atom value pointer against itself), and `Change` fires with the unchanged value. There is no mechanism to detect an identity swap and suppress the event. Observers that must deduplicate must compare the new value with the expected value themselves; or use a validator that returns `false` when `f(val)` produces the same logical state.
- `Atom<A>.Swap` returns the pre-swap `val` when the validator rejects, and returns the new `val2` on CAS success. Because an idempotent reference-type swap sets `obj2 = obj`, the return value `val2 == val` in both the rejection path and the identity-swap path — the caller cannot distinguish them. Use `SwapMaybe` to make the no-op path explicit: return `None` when the intended update is not applicable, and the atom and its observers are left undisturbed.

[REF_FINALIZER_GC_HAZARD]:
- `Ref<A>` has a finalizer that calls `STM.Finalise(Id)`, which calls `state.Remove(Id)` on the process-global `AtomHashMap`. If a `Ref<A>` becomes unreachable outside a transaction — for example, constructed inside an `atomic` lambda and never captured externally — the GC can finalize it between the transaction's snapshot read and `ValidateAndCommit`. `CommitWrites` then looks up a now-removed `Id` from the global state, obtains a zero-initialized `RefState`, and compares its version against the transaction-snapshot version. The mismatch triggers `ConflictException` and an infinite retry loop, because the `Ref` id is monotonic and the removed entry never reappears. Ensure all `Ref<A>` instances used inside `atomic` are strongly referenced from a live root outside the lambda for the full transaction duration.
- The hazard compounds with `async`/`await` inside `atomic`: since `STM.transaction` is `AsyncLocal`, the transaction flows through continuations. A `Ref<A>` captured only as a local in an async method may become eligible for finalization when the compiler splits the state machine and the local is not live in a suspension frame. Capture `Ref<A>` instances in fields or in a lambda closure that also captures `this` to guarantee strong retention.

[REF_VALUEIO_SETTER_DEAD_WRITE]:
- `Ref<A>.ValueIO` has a `set` accessor: `value.Map(delegate(A v) { STM.Write(Id, v); return unit; })`. `IO.Map` is a lazy combinator — it returns a new `IO<Unit>` describing the write and discards it without evaluation. The accessor returns immediately. Assigning `ref.ValueIO = someIO` is a no-op; no state change occurs. The correct write pattern inside `atomic()` is `ref.Value = val` or `ref.Swap(f)`. The same `ValueIO` setter exists on `Atom<A>` with identical semantics: the set accessor calls `value.Bind((A v) => SwapIO((A _) => v))` — `Bind` on an `IO` is also lazy. Neither setter can write state; they exist as type-safe composition points that must be forced.

[REF_IMPLICIT_OPERATOR_TRANSACTION_READ_TRACKING]:
- `Ref<A>` declares `implicit operator A(Ref<A> value)`, which calls `value.Value`, which calls `STM.Read(Id)`. Inside `atomic()`, `STM.Read` unconditionally adds `Id` to `transaction.reads`. Any code path that implicitly converts a `Ref<A>` to `A` — passing it to an overloaded method accepting `A`, using it in a LINQ expression expecting `A`, composing with LanguageExt operators that accept `A` via implicit conversion — silently tracks that read in the current transaction under `Serialisable` isolation. Unexpected `Ref`s appear in the read-set, broadening the conflict surface. Explicit `ref.Value` reads carry the same tracking but are visible at the call site.

[ATOMHASHMAP_OBSERVER_SUBSCRIPTION_RACE]:
- Every `AtomHashMap` mutation path checks `this.Change != null` inside the CAS spin loop before choosing between the `WithLog` and plain trie path. If the CAS fails due to contention, the next iteration re-evaluates `this.Change`, so a newly subscribed observer will be seen on the next retry. The race window where a newly subscribed observer may miss an event is therefore bounded to the interval between a successful `this.Change != null` check and the CAS `CompareExchange` that commits the plain-path mutation. For scalar ops this window is a single fast trie operation; for range ops (`AddRange`, `TryAddRange`) the pre-CAS work scales with range size, widening the window proportionally. Observers that require guaranteed delivery from their subscription point must snapshot `Items` after subscribing to initialize their state.

[MEMO_RESET_MEMORY_ORDERING]:
- `Memo<A>.Reset()` transitions `state` from `2` to `1` via `Interlocked.CompareExchange`, then writes `value = default(A)` and `state = 0` as plain field writes without memory barriers. On weakly-ordered hardware (ARM64), the CPU may reorder `state = 0` before `value = default`, making a thread spinning on `state != 0` observe `state == 0` before `value` is cleared. The thread that wins the loading CAS (`Interlocked.CompareExchange(ref state, 1, 0)`) evaluates `acquire()` and overwrites `value` with a fresh result, making the stale non-null `value` invisible in the loaded outcome. The hazard is benign for idempotent acquire functions — the stale value is overwritten — but any `acquire` function with non-idempotent side effects (prohibited by the design contract) would have observable consequences.
