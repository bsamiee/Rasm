# Marker As Field Config Fold: The Six-Slot Config Row Behind Every Named Marker

[THE_ROW_IS_THE_TYPE]:
- Every named marker is one interface `extends VariantSchema.Field<{ ...config }>`, and `Field<A extends Field.Config>` is exactly `{ readonly [FieldTypeId]: FieldTypeId; readonly schemas: A }` — the marker carries no behavior, no method, no second field beyond the brand symbol and the `schemas` object; the entire difference between `Generated`, `Sensitive`, and `JsonFromString` is the object literal in the type argument. A marker is a SATURATED config row, not a subtype with overrides.
- `Field.Config` is `{ readonly [key: string]: Schema.Schema.All | Schema.PropertySignature.All | undefined }` — a partial slot map keyed by variant name where each present key holds the schema (or `PropertySignature`) that variant projects; a slot is a `PropertySignature.All` precisely so an `Overrideable` or a `Schema.tag` can occupy it, which is how the generative and discriminant slots ride the same row shape as a plain leaf.
- The marker zoo collapses to ONE data-table because `ExtractFields` reads only two facts off a `Field<Config>` — `V extends keyof Config` (does the variant get a column) and `Config[V]` (which schema) — so two markers with identical `schemas` objects are indistinguishable to every downstream projection, equivalence, and constructor. There is no marker identity past the config; renaming `Sensitive(s)` to `FieldExcept("json", "jsonCreate", "jsonUpdate")(s)` yields a byte-identical `Field` row.

[NAMED_MARKERS_ARE_PRESET_ROWS]:
- The named-marker interfaces are the preset rows of one table, each a fixed `keyof Config` membership and per-slot schema, replaceable by the open constructors with zero semantic drift:

```typescript
import { Model } from '@effect/sql';
import { Schema } from 'effect';

// `Model.Generated(s)` IS this row — select|update|json, no insert/jsonCreate/jsonUpdate:
const generated = <S extends Schema.Schema.All>(s: S) => Model.FieldExcept('insert', 'jsonCreate', 'jsonUpdate')(s);
// `Model.GeneratedByApp(s)` IS this row — every db + json-read variant, only the create bodies dropped:
const byApp = <S extends Schema.Schema.All>(s: S) => Model.FieldExcept('jsonCreate', 'jsonUpdate')(s);
// `Model.Sensitive(s)` IS this row — db round-trip, wholly absent from every json body:
const sensitive = <S extends Schema.Schema.All>(s: S) => Model.FieldOnly('select', 'insert', 'update')(s);

class Owner extends Model.Class<Owner>('Owner')({
    id: Model.Field({ select: Schema.Number, update: Schema.Number, json: Schema.Number }),
    secret: Model.Field({ select: Schema.String, insert: Schema.String, update: Schema.String }),
}) {}
```

- `Model.Field({ select, update, json })` and `Model.Generated(s)` produce the same `Field<{ select: S; update: S; json: S }>` — the named marker is the row pre-filled with one repeated schema; the open form lets the THREE slots differ, which the preset cannot express. The preset is the special case where every present slot holds the SAME schema.
- `Model.FieldOnly(...keys)(s)` and `Model.FieldExcept(...keys)(s)` are the two complement-authoring spellings of one row: `FieldOnly` types `Field<{ [K in Keys[number]]: S }>` (exactly the listed variants), `FieldExcept` types `Field<{ [K in Exclude<AllVariants, Keys[number]>]: S }>` (the complement) — both fill every present slot with one schema `S`, so they author a uniform-schema row whose membership is a key set, the slot schema constant. A row whose slots carry DIFFERENT schemas per variant is unreachable from `FieldOnly`/`FieldExcept` and routes through `Field({...})` or `fieldEvolve`.

[TWO_DISJOINT_GATES_GUARD_THE_KEY_DOMAIN]:
- The row's key domain is closed by TWO independent gates that catch DIFFERENT authoring shapes, neither subsuming the other. `Field`'s own parameter type is `A & { readonly [K in Exclude<keyof A, Variants[number]>]: never }` — an out-of-alphabet key typed `never` AT THE `Field({...})` LITERAL, so `Model.Field({ selct: s })` rejects where the row is written, before the row reaches any owner. This gate fires only when the literal passes through `Field`'s constructor.
- The second gate, `Struct.Validate<Fields, Variant>`, fires at the `Model.Class({...})` field-map argument and catches the row that BYPASSED the constructor — a row spread from `Model.fields(donor)`, a row assembled by `fieldEvolve` whose mapping widened `keyof`, a row constructed under a different alphabet: `A[K] extends Field<infer Config> ? [keyof Config] extends [Variant] ? {} : "field must have valid variants"` resolves the field-map position to that literal string when `keyof Config` escapes the six. A directly-written `Field` literal is checked TWICE (constructor `never`, then `Validate`); a programmatically-built row is checked ONCE (only `Validate`), which is exactly why the assembly gate cannot be folded into the constructor.
- `Validate` recurses a nested `Struct` — `A[K] extends { readonly [TypeId]: infer _ } ? Validate<A[K], Variant>` — so a malformed row buried in a `Model.Struct({...})` fragment surfaces `"field must have valid variants"` at the NESTED field position, the row-validity law threading one level down; the bad column is rejected at the fragment that declares it, not at the parent that spreads the fragment in.

[ROW_DECIDES_MEMBERSHIP_AND_REPRESENTATION_IN_ONE_LITERAL]:
- The single `ExtractFields` clause `[Config[V]] extends [Schema.Schema.All | Schema.PropertySignature.All] ? Config[V] : never` reads BOTH axes off one row in one indexed access: the `V extends keyof Config` key-presence test decides MEMBERSHIP and the `Config[V]` lookup decides REPRESENTATION, so a row is not "a membership table plus a schema choice" but one literal where membership and per-slot schema are the same object. A row whose every present slot holds one schema (the uniform-schema rows `FieldOnly`/`FieldExcept` author) is the degenerate case; the general row's slots differ, and that difference is what no preset marker beyond `JsonFromString` expresses.
- A bare `Schema.X` field is the IMPLICIT all-slots uniform row: it falls to the non-`Field` arm and appears in all six variants identically, equivalent to `Field({ select: s, insert: s, update: s, json: s, jsonCreate: s, jsonUpdate: s })`. A marker is therefore never an ADDITION to a base column — it is the narrowing of this universal row to a key subset, or the per-slot rewrite of its schemas, which is why a column has exactly one row and stacking two markers is structurally a single rewrite, never a wrapper chain.
- One row expresses BOTH the variant-family split and the per-leaf representation split as a single literal: `JsonFromString`'s row is all-six membership but `Schema<Type<S>, string, _>` in the three db slots versus `S` in the three json slots — the family partition is just "the set of slots sharing a schema," derivable from the row by grouping `keyof Config` on `Config[K]` equality, never a separate declaration. Membership and representation are not two axes the row reconciles; they are one object the projection reads, the split a fact recoverable by inspecting the literal.

[REMAP_NOT_NEST_IS_THE_COMPOSITION_LAW]:
- Markers compose by per-slot REWRITE of an existing row, never by wrapping a row in a row: `fieldEvolve({ <variant>: s => ... })` over a `Field<S>` types `Field<{ [K in keyof S]: K extends keyof Mapping ? ReturnType<Mapping[K]> : S[K] }>` — each named slot's schema is REPLACED by the mapper's return, the unmapped slots passed through, and the result is a flat `Field<Config'>` with the same `keyof` membership, not a nested `Field<Field<...>>`. The mapper receives the SLOT SCHEMA (`(variant: S[K]) => ...`), so a transform composes onto the exact per-variant schema in place.

```typescript
import { Model } from '@effect/sql';
import { Schema } from 'effect';

// A bare schema is the universal row; `fieldEvolve` over it promotes it to an explicit six-slot row,
// rewriting ONLY the json-family slots' representation while the db slots inherit the bare schema:
const widened = Model.fieldEvolve({
    json: (base: typeof Schema.String) => Schema.compose(Schema.Trim, base),
    jsonCreate: (base: typeof Schema.String) => Schema.compose(Schema.Trim, base),
    jsonUpdate: (base: typeof Schema.String) => Schema.compose(Schema.Trim, base),
})(Schema.String);

class Owner extends Model.Class<Owner>('Owner')({
    code: widened,
}) {}
```

- `fieldEvolve` has two branches selected by its operand: over a bare schema (the universal row) its mapping is keyed by ALL variants and each mapper receives the whole schema, PROMOTING it to a full six-slot row with named slots rewritten (the snippet's `Schema.String` becoming an explicit row); over an existing `Field<S>` its mapping is `{ readonly [K in keyof S]?: ... }`, keyed by the row's EXISTING membership, so it cannot add a slot the row lacks — evolving a `jsonCreate` slot on a `Generated` row (which has no `jsonCreate`) is a type error. Membership change routes through `Field`/`FieldOnly`/`FieldExcept`; representation change of present slots routes through `fieldEvolve` — orthogonal operations, neither nesting.
- `Model.FieldOption` over a `Field<infer S>` (the overload distinct from the raw-schema overload) maps `{ [K in keyof S]: S[K] extends Schema.Schema.Any ? K extends VariantsDatabase ? OptionFromNullOr<S[K]> : optionalWith<S[K], { as: "Option"; nullable: true }> : never }` — it REWRITES every PRESENT slot of an already-variant-shaped row in place, preserving the row's membership while wrapping each slot's schema; optionality layers onto a `Generated` or `JsonFromString` row by per-slot remap, NEVER by nesting a `Field` inside a `Field`. The two `FieldOption` overloads are the proof: the raw-`Schema` overload BUILDS the canonical six-slot optional row, the `Field` overload REMAPS an existing row's slots — same result type shape (`Field<Config>`), different starting point.
- The remap-not-nest law has a sharp boundary: `FieldOption`'s `Field` overload dispatches on `K extends VariantsDatabase` to pick `OptionFromNullOr` (the db `null`-tolerant decode) versus the nullable `optionalWith` (the json missing-key-tolerant decode), reusing `VariantsDatabase = "select" | "insert" | "update"` — so wrapping a row preserves its db/json family split rather than flattening it; a custom slot-rewrapping marker reuses this partition predicate, never re-enumerating the six keys. A row that already dropped a json slot (e.g. a `Sensitive` row) yields a `FieldOption` whose missing slot stays missing — `never` in, `never` out — because the rewrite is `[K in keyof S]`, not `[K in AllVariants]`.

[ONE_SLOT_SCHEMA_KIND_UNIFIES_LEAF_GENERATIVE_DISCRIMINANT]:
- A slot's value is `Schema.Schema.All | Schema.PropertySignature.All`, so a single row places three structurally different things in different slots without three different row types: a plain leaf schema (`Schema.String`), an auto-stamp `VariantSchema.Overrideable` (`PropertySignature<":", (To & Brand) | undefined, never, ":", From, true, R>`), and a `Schema.tag` discriminant (`PropertySignature<":", Tag, never, ":", Tag, true, never>`) all satisfy the slot bound. `DateTimeUpdate`'s row holds a plain `Schema.DateTimeUtc` in `select`/`json` and an `Overrideable` in `insert`/`update` — one row, two schema KINDS across slots, the generative behavior a property of WHICH SLOT, not which marker.
- The `PropertySignature.All` admission is what lets a single row carry a slot whose schema KIND differs per variant, not merely a schema whose representation differs: a `tag` and an `Overrideable` are `PropertySignature`s with distinct property-signature TOKENS, a plain leaf is a `Schema`, and the row holds whichever each slot needs — so the per-variant difference in a column can be a difference in signature kind (leaf vs property-signature-with-default), not only in encoded leaf. A custom auto-stamp marker is one config placing an `Overrideable` in its write slots and a plain schema in its read slots; the downstream construct asymmetry the kind produces is a consequence the projection reads, the row's job being only to HOLD the kind per slot.
- `UuidV4Insert`'s row is the canonical mixed-kind row: `select`/`update`/`json` hold `Schema.brand<Uint8ArrayFromSelf, B>` (a branded leaf, required, identical across reads) while `insert` holds `Overrideable<Uint8Array & Brand<B>, Uint8Array>` (the generative slot, elided, minting the key) — one row threads a branded read shape and a generative write shape through the six slots, and `Model.UuidV4WithGenerate(branded)` is the raw `Overrideable` value the row drops into the `insert` slot, so a bespoke generative key is a hand-built `Model.Field` placing a `*WithGenerate` overrideable into the write slot and the branded leaf into the rest.

[THE_TABLE_IS_THE_GROWTH_AXIS]:
- The row is the smallest portable unit of the persistence shape: a column's full six-variant behavior is one `Field<Config>` value, so reuse, evolution, and validation all operate on the row, never on a re-derived per-variant projection. A column moves between owners as its row; a column's representation changes as a `fieldEvolve` over its row; a column's membership changes as a fresh `Field`/`FieldOnly`/`FieldExcept` — three operations, one operand, no projection ever the unit of reuse because a projection has already collapsed the row to one slot and lost the table.
- A new persisted column is one new row in the field map — a preset marker when its membership matches a named row, a `Model.Field({...})` when the six-slot membership or the per-slot representation is bespoke — and every variant projection, constructor, equivalence, arbitrary, JSON Schema, and repository operation absorbs it by re-reading `ExtractFields<V, Fields>` over the widened map; the next column changes no projection code, only the table. The proof of the table being the right shape is the next column's diff: one row added, every consumer recomputed at compile time, none respelled.
- A new VARIANT modality (a row that exists in a slot no preset names — say wire-only-and-store-redacted, or insert-once-but-json-mutable) lands as a `Model.Field` whose `keyof Config` is the exact slot set, not a new marker constructor and not a parallel model; the open `Field` form is the escape that makes the named markers a non-exhaustive convenience layer over a complete config algebra, so the table never runs out of expressible rows. A marker the library does not ship is a row the author writes directly — the same `Field<Config>` the shipped markers ARE.
- The whole six-variant toolkit is one application of `VariantSchema.make({ variants, defaultVariant })`, which closes `Field`, `FieldOnly`, `FieldExcept`, `fieldEvolve`, and `fieldFromKey` over the variant alphabet; a different alphabet yields a parallel config-row table with identical authoring mechanics — the row algebra is alphabet-agnostic, the six-variant persistence table its persistence-specific instantiation, so the marker-as-config-fold pattern transfers to any owner that needs N representations of one shape with no new machinery.
