# State-Cell Boundary Law — Deepening

[RESOURCES_RELEASE_ORDERING_CORRECTION]:
- `Resources.DisposeU` and `ReleaseAll` iterate `foreach (var resource in resources)` over an `AtomHashMap<object, TrackedResource>` backed by a `TrieMap` (HAMT). HAMT iteration follows hash-tree traversal order, not insertion order. There is no LIFO guarantee on resource release. Any design that depends on teardown ordering must encode the order explicitly — a sequenced `Seq<(object, IO<Unit>)>` tracked alongside the map, or nested `Resources` parent chains where child-scope disposal precedes parent-scope disposal by construction.
- `Resources.Release<A>(A value)` walks the parent chain when the key is absent in the current scope: `resources.Find(obj).Match(f, () => parent != null ? parent.Release(value) : unitIO)`. A forked computation runs inside a child `Resources` constructed as `new Resources(parentResources)`. Any `Release` call that misses the child map traverses into the parent scope and releases a parent-scoped resource. This is the correct design for shared-resource release from a fork, but it is also the mechanism by which a forked computation can accidentally release a parent resource by key collision on `object` identity.
- `Fork` creates `new Resources(parentResources)` as the forked child scope. The fork's `finally` block calls `resources.Dispose()` on the child only; parent resources remain alive. However, any resource the fork explicitly releases via `IO<A>.Bracket()` or `use(...)` that was registered in the parent scope will be released through the parent chain traversal above — the fork acquires, uses, and releases without issue, but the parent-scope resource is then gone. The design pressure is to never register a shared parent resource in a child bracket unless the intent is to transfer ownership to the fork.

[ENVIO_ENV_TOPOLOGY_PRECISION]:
- `EnvIO.Local` (property, not method) computes `New(null, Token, null, SynchronizationContext.Current)` — passing `null` for both `resources` and `source`. The factory then allocates both a fresh `Resources` and a fresh `CancellationTokenSource`, sets `Own = 3` (owns both), and registers `Token.Register(() => source.Cancel())` to forward parent cancellation. This is a full isolation: new resource scope, new cancellation scope, parent cancellation propagates inward, parent resources are inaccessible.
- `EnvIO.LocalResources` computes `New(null, Token, Source, SyncContext)` — null resources but the parent `Source` is passed. The factory allocates a fresh `Resources` (Own bit 2 set) but reuses the parent `CancellationTokenSource` (Own bit 1 NOT set). No `Registration` is created because the source is not new. Disposing this env does NOT dispose the parent source; cancelling the parent token still cancels operations run inside this env because they share the same `Source`.
- `EnvIO.LocalCancel` computes `New(Resources, Token, null, SyncContext)` — passes parent resources but null source. A new linked `CancellationTokenSource` is created (Own bit 1 set), with `Token.Register(() => source.Cancel())` wiring parent→child. Disposing this env disposes the child source but not the parent resources. Child cancellation does not propagate to parent; parent cancellation propagates to child via the registration.
- `IO<A>.Local()` calls `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — this is a fresh cancellation scope with shared resources. The `IOLocal` DSL node wraps the mapped env in `using EnvIO local = MapEnvIO(envIO)`, so both the new source and the registration are disposed when the inner operation completes (success or failure). The confusingly named `.Local()` is a cancellation isolation wrapper, not a resource isolation wrapper; `.Bracket()` is the resource isolation wrapper.
- `IO<A>.Bracket()` calls `WithEnv(env => env.LocalResources)` which produces `IOLocal` (disposes on both success and failure). `IO<A>.BracketFail()` calls `WithEnvFail(env => env.LocalResources)` which produces `IOLocalOnFailOnly`: on success the local env's resources are NOT disposed (the `using` is absent; only `DisposeNonResources()` runs in `finally`), leaving them live for the caller. On failure, `DisposeResources()` runs in the `catch` before `DisposeNonResources()` in `finally`. The structural difference is `IOLocal` uses `using EnvIO local` while `IOLocalOnFailOnly` manually calls `DisposeResources()` only in catch.

[UNINTERRUPTIBLE_MECHANISM_PRECISE]:
- `IOUninterruptible.Invoke` constructs `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)`. Passing `CancellationToken.None` with a null source means: a new `CancellationTokenSource` is created (Own bit 1 set), and because `CancellationToken.None.CanBeCanceled == false`, NO registration is created connecting the parent token to the new source. Parent cancellation does not propagate into the uninterruptible body during execution. This is the actual mechanism — isolation via a non-linked fresh source, not a "checked before/after" guard.
- The asynchronous uninterruptible path (`IOUninterruptibleAsync`) holds the local `EnvIO` across the `await Fa` continuation and calls `LocalEnv.Dispose()` after the await completes. If the outer env is cancelled while `Fa` is awaiting, the inner operation continues to completion because its token is unlinked. The `LocalEnv.Dispose()` runs after the awaited value is produced, releasing the isolated scope. Callers who need to cooperate with cancellation after the uninterruptible region must check `envIO.Token` in the continuation (`Next`) not inside the guarded computation.
- Composing `Uninterruptible` with an outer `Fork` creates a subtlety: the fork creates a child `EnvIO` linked to the parent via `env.Token.Register(() => tsrc.Cancel())`, so the fork's source is cancelled when the parent is cancelled. The `Uninterruptible` inside the fork creates its own local env from `CancellationToken.None`, so the fork's cancellation also does not reach the uninterruptible body. The fork's cleanup (`cleanup.Dispose()` in finally) disposes the fork's own `tsrc` and the parent registration, so no leak occurs even if the parent fires during the uninterruptible region.

[MEMO_STATE_MACHINE_RACE_SURFACE]:
- `Memo<A>` uses three states: `0` (unloaded), `1` (loading), `2` (loaded). `GetValue()` wins on `CAS(state: 0→1)`, runs `acquire()`, writes `value = result`, then writes `state = 2`. The two writes after the CAS are non-atomic. Any concurrent caller in state `1` spins via `SpinWait.SpinOnce()`. A failed `acquire()` writes `state = 0` in the catch block — the memo reverts to unloaded, and the next `GetValue()` call retries acquisition. The exception is rethrown, not absorbed.
- `Memo<A>.Reset()` does `CAS(state: 2→1)` (sets loading), then `value = default(A)`, then `state = 0`. All three steps are sequential after the CAS; no further atomic protection. Any concurrent `GetValue()` that observes state `1` spins correctly. However, a concurrent `Reset()` that also wins `CAS(2→1)` is impossible — only one thread wins the CAS. If a second thread tries `Reset()` while state is `1` (first Reset is in progress), it gets `1` back from CAS and spins until the first Reset writes `state = 0`. At that point, the second Reset's CAS loop fires again with state `0`, returns immediately via the `case 0: return default(Unit)` branch — it sees unloaded and exits without re-resetting. This means two concurrent `Reset()` calls are safe, but only one successfully clears the value.
- `Memo<F, A>` (the higher-kinded form) uses the same three-state protocol but on `K<F, A>?` instead of `A?`. The `Clone()` method creates a new `Memo` from the same `acquire` delegate without the cached state — allowing controlled re-acquisition without the concurrency hazard of `Reset()`. For boundary cells that need periodic re-acquisition under high contention, `Clone()` and replacement via an outer `Atom<Memo<A>>` is safer than `Reset()` under concurrent load.
- `MemoExtensions.Memo<T>(this Func<T> func)` delegates to `Prelude.memo(func)` which constructs `Memo<T>(acq)` with state `0`. The multi-key form `memo<T,R>(this Func<T,R> func)` uses `ConcurrentDictionary<T, R>.GetOrAdd(key, f)`. Note: `GetOrAdd` does NOT guarantee the factory runs exactly once when multiple threads race for the same key — the factory may execute on multiple threads, but only one result is stored. For factories with side effects (native init, file handles), this is a correctness violation; wrap with a `Lazy<R>` value in the dictionary.

[ATOM_CHANGE_EVENT_TOPOLOGY]:
- `Atom<A>.Change` fires synchronously on the thread that won the CAS, immediately after `Interlocked.CompareExchange` succeeds. It does NOT fire when the validator rejects the candidate value (the CAS is not attempted when the validator returns false). It does NOT fire when `SwapMaybe` returns `None` (the CAS is not attempted). It fires exactly once per successful committed transition.
- The `Change` handler receives the newly committed value as its argument (`AtomChangedEvent<A>` is `delegate void AtomChangedEvent<A>(A value)`). The handler runs before any subsequent `Swap` calls can observe the committed value through the change event. However, between the CAS winning and the `Change` invocation, any concurrent reader of `atom.Value` already sees the new value — the change event is a notification that lags slightly behind the actual visibility.
- `Atom<M, A>` (the metadata form) also has a `Change` event, but the event signature is `AtomChangedEvent<A>` — it receives only the new `A` value, not the `M` metadata. Handlers that need the metadata must capture it from the construction context. The two `Swap` overloads for `Atom<M, A>` — `Swap(Func<M, A, A>)` and `Swap(Func<M, A, Option<A>>)` — both thread metadata through the update function, but only the committed value reaches the handler.
- `AtomHashMap<K,V>.Change` fires via `AnnounceChange`/`AnnounceChanges` after the CAS succeeds, carrying a `HashMapPatch<K,V>` with `prev` and `current` trie snapshots. When `this.Change == null`, the mutation path skips the `WithLog` trie variant and uses the cheaper no-log variant — no patch is constructed and no allocation occurs. Subscribing to the `Change` event shifts all mutations to the more expensive `WithLog` path. For high-frequency registry cells that do not need change notifications, leave `Change` unsubscribed.

[CANCELLATION_REGISTRATION_LIFETIME_PRECISION]:
- `EnvIO.New` registers `token.Register(() => source.Cancel())` only when it creates a new `CancellationTokenSource` AND the incoming `token.CanBeCanceled`. The resulting `CancellationTokenRegistration` is stored in `Registration` and disposed in `DisposeNonResources()` (not `DisposeResources()`). Failing to call `Dispose()` or `DisposeNonResources()` on a non-root `EnvIO` leaks the registration, which holds a delegate reference keeping the source alive until the parent token fires or the parent `CancellationTokenSource` is disposed.
- `EnvIO.DisposeNonResources()` uses `Interlocked.CompareExchange(ref nonResourcesDisposed, 1, 0)` to ensure it runs at most once. `DisposeResources()` uses the same pattern on `resourcesDisposed`. `Dispose()` calls both sequentially. This means double-dispose is safe — idempotent by CAS. Calling `DisposeNonResources()` then `DisposeResources()` separately (as `IOLocalOnFailOnly` does in the failure path) is equivalent to a single `Dispose()` call; the CAS guards prevent double-release.
- A `CancellationTokenSource` created with a timeout (`new CancellationTokenSource(TimeSpan)`) starts the timer immediately at construction. If the `EnvIO.New` call that creates a timeout-scoped source is delayed (e.g. in a lazy `IO` tree that is not yet executed), the timeout window begins at the `IO` tree evaluation time, not at any earlier point in the pipeline construction. An `IO` tree that constructs `LocalCancelWithTimeout` is therefore always fresh at execution time; stacking multiple `LocalCancelWithTimeout` calls does not compound the timeout — each call creates a new source from the current token at execution time.

[ATOM_VALIDATOR_GATE_INVARIANT]:
- `Atom<A>.New(A value, Func<A, bool> validator)` returns `Option<Atom<A>>.None` if the initial value fails the validator. This is the only way for construction to fail; the validator is also applied on every `Swap`. The validator receives the candidate value before the CAS. If the validator is expensive, it runs once per CAS attempt plus one extra per contended retry — pure validator functions with O(1) cost are required.
- The validator is stored as `Func<A, bool>` (never nullable — substituted with `_ => true` when omitted). Two-phase open semantics using `Atom<Option<T>>` with validator `v => v.IsNone || condition(v.Case)` encodes a one-way gate, but a validator that rejects `Some(handle)` after the first open will prevent all subsequent swaps — the atom freezes at `Some(handle)` permanently once the gate opens, because no new value can pass. The correct one-way gate validator is `v => true` (no validator) and the gate logic lives in the swap function: `SwapMaybe(curr => curr.IsNone ? Some(acquire()) : None)` — the gate is in the swap predicate, not the validator.
- A concurrent `SwapMaybe(curr => curr.IsNone ? Some(acquire()) : None)` pattern: two threads both see `None`, both call `acquire()` (the factory runs on both), one wins the CAS and commits `Some(handle1)`, the other's CAS fails, it retries with the new current value `Some(handle1)`, the swap function returns `None` (already open), the atom is unchanged. The losing thread's `handle2` is now leaked — it was acquired but never committed. The winning CAS is not guaranteed to be deterministic. The only safe form for once-only acquisition is to call `acquire()` only after confirming the CAS will succeed, which requires a two-step protocol: gate CAS first (no acquisition), then acquire, then a second CAS to commit. Or: accept the double-acquisition race and ensure the losing handle is explicitly released in the swap function's returned value (e.g., by returning `Option<(A handle, A toDispose)>`).

[RESOURCES_MERGE_AND_SCOPE_MIGRATION]:
- `Resources.Merge(Resources rhs)` is `internal` and adds all entries from `rhs.resources` into `this.resources` via `Swap`. This is the mechanism for absorbing a child scope into a parent. It is not exposed publicly; it is used by the framework when migrating resources from a completed sub-scope into a parent. A design that needs to manually promote resources from a child scope to a parent must go through the framework's `use`/`bracket` DSL rather than calling `Merge` directly.
- `Resources.Acquire<A>(A value, Func<A, IO<Unit>> release)` accepts a custom `IO<Unit>` release effect for handles whose teardown is itself effectful. `TrackedResourceWithFree<A>` stores `(Value, Dispose: Func<A, IO<Unit>>)` and calls `Dispose(Value)` when released. This is the correct surface for resources whose cleanup must run on a specific thread (via `Post()`), requires network flush, or must be sequenced relative to other effects — the `IO<Unit>` release runs inside the `EnvIO` provided to `DisposeU`, which carries the sync context.
- `Resources.DisposeU(EnvIO envIO)` creates a NEW internal `EnvIO` with `CancellationToken.None` (not the caller's token) to run releases: `new EnvIO(envIO.Resources, CancellationToken.None, source, envIO.SyncContext, null, 0)`. Release effects are therefore never cancellable through the caller's token — they run to completion regardless of the outer cancellation state. This is a deliberate design: teardown must complete even when the scope's operation was cancelled.

[LIFECYCLE_FAULT_REENTRY_CELL_DESIGN]:
- A lifecycle state cell encoded as `Atom<LifecycleState>` with a validator enforces valid transitions: `v.IsValidTransitionFrom(currentState)`. The validator runs inside the CAS loop on the candidate next state. Invalid transitions are rejected by returning the current state unchanged; the transition winner is observable only through the return value distinguishing "I committed" from "someone else committed". For a bounded state machine with ≤8 states, a `[SmartEnum<LifecycleState>]` with a `bool IsValidSuccessor(LifecycleState)` method on each case is denser than a free validator function.
- Correlation cell replace-versus-reuse under fault re-entry: when a boundary layer transitions `Faulted → Opening → Open`, a `ConcurrentDictionary<CorrelationId, TaskCompletionSource<T>>` representing in-flight requests from the faulted instance must be fully swept before the new open instance begins accepting work. The sweep must fault all outstanding sources via `TrySetException(new BoundaryFaultException(...))` rather than `TrySetCanceled()`, because callers may distinguish cancelled (retriable) from faulted (non-retriable). The dictionary sweep is idempotent if each entry is removed before being faulted, preventing double-fault from a concurrent drain+replace.
- `ObjectDisposedException.ThrowIf(bool, object instance)` (added in .NET 7, verified present in .NET 10) accepts an `object` for the instance parameter; passing the boundary cell itself as the instance provides a meaningful type name in the exception message without reflection. The guard pattern `ObjectDisposedException.ThrowIf(_state is not LifecycleState.Open, this)` is the canonical post-`Open` access guard; the JIT can eliminate the branch when the state check is trivially false in hot paths.
- A three-state initialization cell (`Initializing`, `Initialized`, `InitializationFailed`) with `Atom<InitState>` prevents the window where a second caller observes `Initialized = true` while the initializer's side effects are still running. The winning initializer holds `Initializing` (set via the first CAS) through the full side-effect sequence; it commits `Initialized` (or `InitializationFailed`) as the final state transition. Subsequent callers that see `Initialized` after that commit see a fully complete initialization. Callers that see `Initializing` must wait — use a `SemaphoreSlim(0, 1)` or a `TaskCompletionSource<Unit>` that the winner signals on completion; `SpinWait` busy-looping on `Initializing` is only acceptable for microsecond-range initializations.

[GENERATED_REGISTRY_KEY_STABILITY]:
- A generated registry backed by `AtomHashMap<K, V>` where keys derive from `nameof` on a source-generated type survives recompilation because `nameof` is resolved at compile time to a stable string literal. Keys derived from `GetType().Name` or `GetType().GUID` are stable across recompilation but not across assembly signing changes or obfuscation. Keys derived from `RuntimeHelpers.GetHashCode` or object identity are per-process and produce ghost entries on every restart.
- When a generated registry accumulates ghost entries (keys whose types were removed in a later code generation cycle), `AtomHashMap<K,V>.Filter(pred)` atomically removes entries matching the predicate. The filter is applied inside a `Swap` call using `items.FilterWithLog(pred)` when the `Change` event is subscribed, or `items.Filter(pred)` otherwise. The sweep must execute during the boundary layer's drain phase — before `Closed`, after all in-flight operations referencing the old keys complete. Executing a filter sweep during `Open` state races with concurrent `FindOrAdd` calls and is safe only if the filter predicate checks entry validity rather than key existence.
- `AtomHashMap.FindOrAdd(K key, Func<V> None)` is the correct once-only insertion primitive for singleton registries. The factory `None` runs inside the CAS loop and may execute more than once if the CAS fails; the factory must be pure and idempotent. For factories that acquire resources (handle construction), the factory must either be idempotent (re-acquiring a logically equivalent resource) or the cell must use `SwapKey(key, f)` with a function that constructs only when the current value is `None` and disposes any leaked handle when the swap fails.

[NATIVE_LIBRARY_IDENTITY_AND_FINGERPRINTING]:
- Native library identity for conflict detection requires three fields at minimum: absolute path, file size (from `FileInfo.Length`), and last-write timestamp (from `FileInfo.LastWriteTimeUtc`). Path alone is insufficient because file replacement at the same path changes the file without changing the path. Using `Assembly.LoadFrom(path)` does not prevent loading two physically different DLLs at the same path if one was loaded before the file was replaced; the CLR does not re-check file contents on a second load request.
- For process-static native library cells, `NativeLibrary.Load(string libraryPath)` returns an `IntPtr` handle; `NativeLibrary.TryLoad(string, Assembly, DllImportSearchPath?, out IntPtr)` populates the handle and returns `false` on failure without throwing. The handle from `Load` must be stored in the gate cell alongside the file fingerprint. `NativeLibrary.Free(handle)` is the symmetric release; releasing a handle that is still referenced by exported function pointers produces an access violation on next call. The gate cell must hold the handle until the process exits or the library is explicitly unloaded as a deliberate teardown step.
- Two process-static native library cells that both wrap the same underlying native library must share a fingerprint check to detect version conflicts at the seam. An `AtomHashMap<string /* canonical lib name */, (IntPtr handle, long size, DateTime modified)>` keyed by canonical name with a `SwapKey` that validates fingerprint consistency on each `TryAdd` surfaces conflicts as a typed failure at the second load attempt rather than silent undefined behavior from loading two incompatible versions.
