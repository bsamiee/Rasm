# Publication And Fence Algebra

[BOXED_CARRIER_PUBLICATION]:
- A single-cell primitive publishes a value of arbitrary type `A` through one slot typed `volatile object`, never through a slot typed `A`. The carrier is a generated `Box<A>` whose static `New` and `GetValue` delegates are IL-emitted once per closed `A` at type-initialization: for a reference `A` both delegates are bare identity casts so the reference is stored and read with zero allocation and zero indirection; for a struct `A` `New` is `newobj Box<A>` and `GetValue` is `castclass` + `ldfld Value`, so the struct is copied into one immutable heap box at write and read straight out of the box field. The cell never boxes a reference type and boxes a struct exactly once per published value.
- The immutability of the box is the tear-freedom guarantee: a struct wider than a machine word — a `(state, count, deadline)` triple, a sixteen-byte identifier pair — cannot be observed half-written, because the writer constructs the whole box before the compare-exchange and the slot only ever holds a fully-initialized box reference. Word-tearing of multi-field value cells, the classic hazard of a plain `volatile struct` field, is structurally impossible here: the atomic unit is the reference swap, not the field write.
- The fence pairing is the reference store into the `volatile object` slot publishes with release order, the reference load reads with acquire order, and the immutable box contents are transitively visible because they were fully written before the publishing store. No explicit `Volatile.Write`/`Volatile.Read` call appears; the field qualifier carries the ordering, and the compare-exchange that performs every transition is itself a full barrier, so the box payload is guaranteed visible to any thread that observes the new reference.
- When runtime IL emission is unavailable the `Box<A>` delegates degrade to closures — `(object x) => (A)x` and `(A x) => (object)x ?? throw` — preserving identical publication semantics at higher call cost. A boundary cell on a restricted runtime keeps its correctness and loses only the unbox-elision throughput; the fence algebra is unchanged.

[CAS_VISIBILITY_LADDER]:
- A successful compare-exchange is a full two-way barrier: every prior store retires before the swap and no subsequent load floats above it, so the transform's reads of the old value and the publication of the new value are sequenced. A spin-loop transform therefore re-reads the slot with full ordering on each iteration; the read that feeds the transform is never stale relative to a competing committed swap, which is what makes the loser's retry observe the winner's value rather than a cached duplicate.
- The change event fires once, after the loop, off the committed value — never on a losing iteration. The single-cell primitive is therefore the only correct seam for "signal the host exactly once on a settled transition"; the firing is past the barrier, so a handler reads a fully-published value. The accompanying validator runs inside the loop before the swap; a rejected transform returns the unchanged prior snapshot with no barrier-crossing publication and no event, so rejection is invisible to any observer reading the slot.
- The concurrent map cell raises the same ladder one level: its backing immutable trie sits in a `volatile` field and every mutation is a compare-exchange of the whole trie reference. A point read indexes the volatile trie snapshot and is wait-free, lock-free, and linearizable against any concurrent writer — the reader acquires a coherent whole-map snapshot with one acquire load and never spins. Bulk readers that capture the trie reference once and query it repeatedly are reading a frozen consistent snapshot for free; re-reading the field per query instead re-acquires and can observe a newer map mid-iteration.
- The transactional ref registry is itself one of these map cells, so a non-transactional ref read is a single acquire load of the global trie followed by an index — the cheapest coherent read in the family, with no transaction, no snapshot copy, and no lock. The expensive path is reserved for multi-cell commit; isolated reads ride the same volatile-trie acquire as the map cell.

[NON_VOLATILE_DOUBLE_WRITE_TEAR]:
- A three-state lazy cell that encodes load progress in a plain `int state` and the payload in a plain typed field carries a publication hole the boxed-carrier cells close. The winner writes payload then writes `state = 2` as two ordinary stores with no release fence between them; the fast-path reader does a plain `if (state != 2)` load and, on seeing `2`, returns the plain payload field. On a weakly-ordered core the two writer stores may retire out of order and the two reader loads may be satisfied out of order, so a reader can observe `state == 2` while the payload load still returns the pre-initialization default.
- The compare-exchange that claims the loading transition is a full barrier, but it guards only the slow path; the fast `state != 2` short-circuit bypasses the barrier entirely, and that is exactly the read that races. The hazard is invisible on strongly-ordered hardware and surfaces only under weak ordering, making it a latent correctness bug that passes every strongly-ordered test bench.
- Two closes of the fix space: read the progress flag with an explicit acquire load so the dependent payload load cannot be hoisted above it, or migrate the cell to the boxed-carrier primitive whose `volatile`-slot-plus-CAS publishes state and payload as one indivisible reference swap. The second is strictly safer because it removes the two-field split that the fence pair has to bridge; a lazy cell with transient-failure retry semantics additionally needs a payload carrier that does not permanently cache a thrown initialization fault, which the boxed-carrier primitive backed by an absence case provides and the plain three-state field does not.
- The reset transition of such a cell writes the default payload then writes `state = 0` as two more unfenced stores after its claiming compare-exchange; concurrent reset-versus-read in the general case is unsafe for the same hoisting reason. Replacing the whole cell through an outer boxed-carrier slot — swap a fresh cell in atomically — is safer than mutating the inner unfenced fields under load.

[FALSE_SHARING_AND_WIDTH]:
- Lock-free lifecycle state encoded as a narrow integer admits compare-exchange overloads on the byte, signed-byte, short, and unsigned-short widths, so a state machine of at most a few hundred states transitions atomically in a single byte; four such byte cells co-reside in one machine word rather than four words. Width selection is a memory-density decision, not only an encoding one, and negative sentinel states force the signed widths.
- Co-residence is a double-edged blade: four byte state cells in one word share a cache line, so independent compare-exchanges on sibling cells contend on the line and ping-pong it between cores even though no logical conflict exists. False sharing converts logically independent transitions into a serialized hot line. The defense is to pad genuinely-independent high-frequency cells onto separate lines and to pack into one word only cells that transition together under one logical operation — the same cells a multi-ref transaction would coordinate anyway.
- A multi-property lifecycle flag word transitions without a compare-exchange loop at all: an atomic bit-set or bit-clear sets or clears one property and returns the prior word in a single instruction, so a property toggle costs one atomic and never spins. Reading the current flags, however, requires an acquire-ordered read — an explicit volatile read or a no-op compare-exchange of the field against itself — because a plain field load is not coherent on weakly-ordered hardware and can return a stale word. The asymmetry is load-bearing: the write side is fence-free per the atomic primitive, the read side must opt into the acquire fence explicitly.
- The version counter that a multi-cell coordinator compares for conflict is a monotonic field whose only safe cross-thread reader is the commit barrier; a seqlock-style optimistic reader that wants to read a coordinated payload without entering a transaction must read the version with acquire ordering, read the payload, then re-read the version and restart if it moved — the payload read must not be hoisted across either version read, which only an acquire-ordered version load enforces.

[COMMIT_PHASE_FENCE_NESTING]:
- A multi-cell commit nests two retry loops with distinct fence roles. The outer loop owns the snapshot-and-replay: it captures the global trie at entry, runs the transaction body against the local snapshot, and re-runs the whole body on a conflict. The inner commit is itself a compare-exchange on the global trie that validates reads, commits writes with version increment, and rebases commutes — all inside one swap of the whole-map reference, so the entire validate-plus-commit is a single atomic publication against concurrent committers.
- The inner commit swap therefore re-executes its validation and write-application on every inner compare-exchange retry, independently of any outer body retry. Validators and commute functions invoked at commit must be pure and idempotent for the same reason the outer body must be: both replay, on different triggers, and a side effect in either replays per attempt. The version-equality check that decides a write conflict, and the read-version equality check that serializable isolation adds, both execute inside this inner swap and re-run on inner contention without re-entering the body.
- Change notifications for a committed multi-cell transaction fire only in the outer loop's teardown, off the transaction's accumulated change set, regardless of how many inner commit retries occurred — so the notification count is decoupled from the inner CAS retry count but coupled to the outer attempt count. A handler driving a host effect from a coordinated cell's change event still fires once per outer attempt including aborted ones; the inner retries are invisible to it, the outer retries are not. Host-visible effects belong in a post-commit continuation reading the committed values, never on the change subscription of a coordinated cell.
- The metadata-carrying single-cell variant fires its change event with the new value only and never the metadata, which is captured at construction and immutable; a handler needing both publishes them as one immutable payload through a single boxed slot so the acquire load delivers value and metadata together, rather than reading the metadata from a second unfenced field after the event.
