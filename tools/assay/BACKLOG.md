## Upstream issues worth filing

### [UPSTREAM] mutmut stats-phase crash â€” fixed locally (`max_stack_depth = -1`); issue still worth filing
The previously documented rootdir/cwd theory is **falsified** (the captured stats argv is well-formed and pytest loads the parent pyproject config â€” the prepended addopts prove it). Root cause, established empirically with `debug=true`: with `max_stack_depth != -1`, `record_trampoline_hit` (`mutmut/__main__.py:130`) walks caller frames calling `Path(co_filename).resolve(strict=True)`; module-level calls into mutated code during conftest import put `<frozen importlib._bootstrap>` pseudo-filenames on that stack, the strict resolve raises `FileNotFoundError`, the conftest import dies, pytest exits 4, and mutmut raises `BadTestExecutionCommandsException`. The local fix is the upstream-default `max_stack_depth = -1` (repo had 10), which disables the strict-resolve stack walk entirely. Anyone with `max_stack_depth != -1` plus import-time calls into mutated code still hits this â€” file upstream as-is:

> **TITLE**: record_trampoline_hit crashes with FileNotFoundError on '<frozen importlib._bootstrap>' frames when max_stack_depth != -1 and mutated code runs at import time.
> **BODY**: With max_stack_depth set (e.g. the README's max_stack_depth=8), record_trampoline_hit (mutmut/__main__.py, the `while c and f:` loop) calls Path(f.f_code.co_filename).resolve(strict=True) on every caller frame. When a mutated function is invoked at module import time (module-level constant computed by a mutated function, imported transitively from a conftest), the caller stack contains importlib machinery frames whose co_filename is the pseudo-filename '<frozen importlib._bootstrap>' (also '<frozen importlib._bootstrap_external>', '<string>'). strict=True raises FileNotFoundError('<cwd>/<frozen importlib._bootstrap>'), the conftest import aborts, pytest exits 4, and mutmut raises BadTestExecutionCommandsException â€” misleadingly suggesting the pytest command itself is malformed.
> **REPRO (minimal)**: src/m.py with `def cap() -> int: return 1`; src/pkg using `X = cap()` at module level; a conftest.py importing pkg; [tool.mutmut] source_paths=['src'], max_stack_depth=8; `mutmut run` -> stats phase exit 4 with the traceback above; with max_stack_depth=-1 the run proceeds.
> **OBSERVED ON**: mutmut 3.6.0, pytest 9.0.3, CPython 3.15.0b1, macOS.
> **FIX CANDIDATES**: (a) skip frames whose co_filename starts with '<' before resolving; (b) resolve with strict=False; (c) wrap the resolve in try/except FileNotFoundError and continue; (d) break on importlib frames the same way 'pytest'/'unittest' filenames break the walk. Additionally, the exit-4 path would be far easier to debug if BadTestExecutionCommandsException included the captured pytest output even without debug=true.

When filing, also mention the forked-children consequence: mutant workers fork from the post-clean parent, so any set-once global poisoned across in-process `pytest.main()` calls silently converts test-infrastructure failures into false "killed" verdicts â€” an argument for subprocess-isolated stats/clean phases in mutmut core.
