# [PY_ARTIFACTS_API_PYVIPS]

`pyvips` is the native-`libvips` fused-streaming-raster engine the `graphic/raster/io#IO` `RasterEngine.LIBVIPS` arm and the `graphic/color/managed#MANAGED` `Managed` arm compose: an `Image` whose `new_from_file`/`new_from_buffer`/`new_from_source`/`new_from_memory`/`new_from_array` factory family loads a source, whose generated libvips-operation methods (`thumbnail_image`/`thumbnail_source`/`resize`/`smartcrop`/`extract_area`/`gravity`/`embed`/`icc_transform`/`colourspace`/`flatten`/`autorot`/`composite2`/`arrayjoin`) transform it through a lazily-evaluated demand-driven pipeline that computes only at egress, and whose `write_to_file`/`write_to_buffer`/`write_to_target`/`write_to_memory`/`numpy` family flushes it — plus `Source`/`Target` (and the `SourceCustom`/`TargetCustom` Python-callback variants) for stream IO and `Region` for random-access pixel fetch. The package owner composes `new_from_buffer(payload, "", access=Access.SEQUENTIAL)` then `autorot()`, `thumbnail_image`, `icc_transform`, `extract_area`/`smartcrop`, `composite2`, and `write_to_buffer` over ONE `Access.SEQUENTIAL` pipeline that fuses decode + downscale + ICC + crop/composite in one streamed pass; it is the fused-pipeline / large-image / managed-ICC-egress owner where `pillow` (`.api/pillow.md`) is the in-process pure-Python working surface, and it never re-implements the demand-driven region scheduler, the SIMD resamplers, the liblcms2 ICC engine, or the per-format codec loaders libvips already owns. Every arm crosses the ONE `graphic/raster/io#IO`-/`faults`-owned `anyio.to_process.run_sync(_worker_raster, op, limiter=WORKER_BAND)` subprocess seam — pyvips binds a Forge-provisioned native `libvips` that is NOT on the runtime loader path, so it rides the worker band like the other host-native worker packages, never a thread slot and never the loop — captures `pyvips.Error` into `RasterFault.engine` and the `cffi` dlopen `OSError` of an unprovisioned `libvips` into `RasterFault.provision`, folds into one `msgspec`-shaped `RasterFact`/`ManagedFact`, and projects `core/receipt#RECEIPT` `ArtifactReceipt.Preview(key, width, height, scores)`.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: image, connection, region, interpolator roots
- rail: image

`Image` is the lazily-evaluated pixel-pipeline node; transforms return a new `Image` and nothing computes until a `write_to_*`/`numpy`/`tolist` egress pulls pixels. `Source`/`Target` are the streaming IO endpoints (descriptor/file/memory); `SourceCustom`/`TargetCustom` route reads/writes/seeks through Python callbacks (any file-like, S3 object, in-flight stream). `Region` exposes random-access pixel fetch over a computed image. `Interpolate` is the named sub-pixel interpolator passed to warp/affine operations. Every failure raises the single typed `Error` carrying the libvips error buffer.

[PUBLIC_TYPE_SCOPE]: operation-selector enums (string-constant vocabularies, `pyvips.enums`)
- rail: image

The enums are string-constant holders (`pyvips.enums`) feeding the generated operations as keyword rows — pass the constant string (`Interesting.ATTENTION`, `Access.SEQUENTIAL`), never a raw libvips int, and never a parallel per-strategy method. `Access` is the load-time streaming hint that decides whether the whole pipeline can run single-pass; `Interesting` is the `smartcrop`/`thumbnail crop=` strategy; `Intent` is the ICC rendering intent; `Kernel` is the resampler; `BlendMode` drives `composite2`. The `Foreign*` family carries the per-encoder option vocabularies (`dzsave`/`tiffsave`/`webpsave`/`heifsave`/`pngsave` keyword rows) so a codec reaches its native option set through a typed constant. The operation-algebra enums select the arithmetic/relational/boolean/complex/round/morphology operation for the generic `math`/`relational`/`boolean`/`complex`/`round`/`morph` ops.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: image construction (one factory family, source-discriminated)
- rail: image

`new_from_file`/`new_from_buffer`/`new_from_source` are the codec-decoding factories; the loader is chosen from the source bytes/extension, and per-loader options (`access`, `fail_on`, `page`, `n`, `dpi`, `shrink`, `unlimited`) ride `**kwargs` or the trailing option string (`"img.jpg[shrink=2]"`). `new_from_memory` wraps an already-decoded raw buffer (no codec), `new_from_array`/`new_from_list` lift host arrays, `new_from_image` builds a constant image matching an existing one's geometry, and `black`/`new_temp_file` mint blanks. Pass `access=Access.SEQUENTIAL` at load when the whole pipeline is one-pass so libvips streams instead of buffering the full raster.

[ENTRYPOINT_SCOPE]: pipeline egress and stream IO
- rail: image

[ENTRYPOINT_SCOPE]: generated libvips operations (one `__getattr__` dispatch, not a per-op type)
- rail: image

Most libvips operations are methods generated on `Image` via `Image.__getattr__` -> `Operation.call(name, *args, **kwargs)`; there is no per-operation type and no explicit `def` for them in the binding source — the method set is introspected from the live libvips build, so the names below are the canonical libvips operation nicknames (`Image.get_suffixes()` lists the loaders/savers the build exposes, `pyvips.base.type_find("VipsOperation", nick)` confirms an op exists — the capability-detection probe before composing a build-dependent op). A small set are explicit Python convenience `def`s in `vimage.py` (`bandjoin`/`bandsplit`/`bandrank`/`composite`/`ifthenelse`/`scaleimage`/`maxpos`/`minpos`/`get_n_pages`/`pagesplit`/the trig/round band wrappers/the band-arithmetic operator overloads). Operations chain into ONE lazy pipeline (each returns a new `Image`, nothing computes until an egress pulls pixels); positional args are the required inputs and keyword rows are the optional libvips arguments. `Image.get`/`set`/`get_typeof`/`get_fields`/`remove` read/write image metadata (EXIF/ICC/orientation/page) and `Image.get("icc-profile-data")` recovers an embedded ICC blob — `get_typeof("icc-profile-data") != 0` is the canonical embedded-ICC-presence probe the managed/convert arms gate on (NOT a `get_value` that raises on absence).

## [04]-[IMPLEMENTATION_LAW]

[IMAGE_STREAMING_RASTER]:
- import: `lazy import pyvips` inside the `graphic/raster/io#IO` `_worker_raster` / `graphic/color/managed#MANAGED` `_icc_apply` worker-arm body ONLY — the binding's `cffi.dlopen('libvips.42')` runs at import time, so an unprovisioned native `libvips` raises a dlopen `OSError`/`ImportError` the worker's outer `try` converts to `RasterFault.provision` (the host-readiness fault distinct from a content fault, the same gate `python-magic`/libmagic + `skimage` ride); module-level import on the core page is banned because it does crash the page load on a host without `libvips`.
- load axis: `new_from_file`/`new_from_buffer`/`new_from_source` is the single decode factory across every libvips loader; loader selection is introspected from the source bytes/extension, never a per-format reader type. The composed ingest is `pyvips.Image.new_from_buffer(payload, "", access=Access.SEQUENTIAL)` because the raster arms hold the payload as `bytes` (the `""` is the empty trailing option string); the `Source`/`SourceCustom` endpoint is the ingest when admission holds a live stream/descriptor rather than a full buffer. `access=Access.SEQUENTIAL` at load is load-bearing: it tells libvips the pipeline is one-pass so it streams instead of buffering the full raster, which is the property that makes the fused decode+downscale+ICC pass O(scanline) not O(image).
- transform axis: every transform is a generated libvips operation method on `Image` dispatched through `__getattr__` -> `Operation.call`, returning a new lazy `Image`; `autorot`/`thumbnail_image`/`resize`/`smartcrop`/`extract_area`/`gravity`/`icc_transform`/`colourspace`/`flatten`/`composite2`/`arrayjoin` chain into one pipeline that computes only at egress. `Kernel` (resampler), `Interesting` (smartcrop / thumbnail `crop=`), `Intent`+`PCS` (ICC), `BlendMode` (`composite2`, passed by `mode.value` libvips nickname), `Extend` (`embed`/`gravity`), `Align` (`arrayjoin`), and `Size` (thumbnail clamp) are keyword-row enums, never raw libvips ints and never a parallel per-strategy method. `autorot()` precedes a crop/resize so the box is in display orientation; the `FitMode` (`CONTAIN`/`COVER`/`STRETCH`/`PAD`) maps to `crop=Interesting.NONE`/`crop=Interesting.ATTENTION`/`size=Size.FORCE`/`embed(extend=Extend.BACKGROUND)` — the same geometry the `RasterEngine.PILLOW` arm resolves, so one `Thumbnail` op never diverges by engine.
- shrink-on-load axis: `thumbnail`/`thumbnail_buffer`/`thumbnail_source` is the fused decode+downscale row — it asks the codec to shrink during decode (JPEG DCT scaling, libvips `shrink-on-load`) so the full-resolution raster is never materialized; `thumbnail_image(width, height=, size=Size, crop=Interesting)` shrinks an already-opened `Access.SEQUENTIAL` pipeline (the composed `new_from_buffer(...).thumbnail_image(...)` shape). A `new_from_file` + separate `resize` two-step is rejected where `thumbnail*` fuses them, and a Pillow round-trip for downscale is rejected where the source is already a libvips pipeline.
- colour axis: `icc_transform(dst, *, input_profile=src, intent=Intent.RELATIVE, black_point_compensation=, pcs=PCS.LAB, depth=)` runs liblcms2-backed device→device ICC conversion — the `graphic/color/managed#MANAGED` `_icc_apply` managed-egress engine AND the `graphic/raster/io#IO` `Convert`-arm sRGB normalization. The embedded-profile gate is `Image.get_typeof("icc-profile-data") != 0` (a `get_typeof` probe that returns 0 for absent, NOT a `get_value` that raises); only when it proves a profile does the arm run `icc_transform`, never a naive `colourspace('srgb')` that discards the source profile. `embedded=True` lets `icc_transform` read the source's own profile (the `Convert` arm on an already-encoded raster); a `new_from_array` numpy source carries no embedded profile so the managed arm always passes an explicit `input_profile`. pyvips owns the device→device managed egress; `pillow` `ImageCms.buildProofTransform` owns ONLY the soft-proof/gamut-warning transform pyvips has no member for, and `ImageCms.getProfile*` the profile-header READ — a second pillow ICC device-egress engine is the deleted divergence.
- egress axis: `write_to_buffer(format_string, **kwargs)` keys the encoder by `format_string` (`".png"`/`".jpg"`/`".webp"`/`".tif"`, the `_VIPS_SUFFIX`/`_CODEC_OPTS` row); encoder options (`Q`/`compression`/`lossless`/`effort`/`subsample_mode`/`bitdepth`, and `keep=ForeignKeep` for EXIF/ICC/XMP retention) ride `**kwargs` per the `_VIPS_KWARGS[codec](quality, effort)` builder, never a parallel encoder type. `write_to_target` streams the encode to a `Target`/`TargetCustom` for a network/segmented sink; `dzsave`/`dzsave_buffer` emits a DeepZoom pyramid; `numpy(dtype=)`/`write_to_memory` extract raw pixels for the host array edge. The pipeline computes EXACTLY ONCE at the chosen egress (the lazy-pull point), so the whole fused decode+orient+ICC+flatten+encode runs in one streamed pass.
- random-access axis: a consumer that needs scattered pixel windows (tiling, deep-zoom) takes a `Region.new(image)` over the computed image and `fetch(x, y, w, h)`s; a `tilecache`/`linecache` is inserted to break a re-computation bottleneck when an `Access.RANDOM` pipeline is read non-sequentially, and `copy_memory()` forces a reused intermediate to RAM — never a manual full-`write_to_memory` then slice.
- runtime axis: `concurrency_set(n)` bounds libvips's OWN internal thread pool DOWN at worker init so it does not oversubscribe the host against the outer `WORKER_BAND` `to_process` fan (the two-pool guard the `graphic/raster/io#IO` page names); `cache_set_max_mem`/`cache_set_max` bound the operation cache; `block_untrusted_set(True)`/`operation_block_set(...)` harden the loader set for hostile uploads; `set_progress`/`set_kill` give a long evaluation a cancellation hook. These are boundary-init calls, never per-op knobs.
- fault axis: a failed libvips call raises the single typed `pyvips.Error` carrying the libvips error buffer; the worker's `_vips_guarded` capture maps `pyvips.Error -> RasterFault.engine` and the outer dlopen `OSError`/`ImportError -> RasterFault.provision`, lifting once at the worker boundary, never letting a bare exception escape into the domain rail (`@beartype` violation -> `RasterFault.contract`, exhausted worker death -> `RasterFault.worker`).
- evidence: each image op captures source kind, loader/encoder name, the `Access` mode, output width/height/bands/`Interpretation`, ICC presence (`get_typeof("icc-profile-data") != 0`) + applied intent/BPC/PCS/depth, page count (`get_n_pages`), the chained operation list, the native `libvips` version (`pyvips.base.version`), and output byte length as the `msgspec.Struct` `RasterFact`/`ManagedFact` field the consuming owner folds — pyvips mints no receipt of its own.
- boundary: pyvips owns the libvips fused streaming decode/transform/encode + the device→device managed-ICC egress + smartcrop + DeepZoom + large-image pipeline; `pillow` (`.api/pillow.md`) owns the in-process pure-Python raster working surface (drawing, FreeType text, per-pixel Python LUTs, ICC soft-proof, profile-header read); scientific filtering/morphology/registration routes to `scikit-image` (the `graphic/raster/process#PROCESS`+`measure#MEASURE` acceptors); CMYK/spectral math is `colour-science`; layered PSD/PSB authoring is `PhotoshopAPI`/`psd-tools`; the rendered raster bytes feed the `document/emit#DOCUMENT`/figure/export-bundle owners directly; live UI stays outside this package.

[STACKING]:
- universal `numpy` tier (`libs/python/.api/numpy.md`): the host pixel seam is `Image.numpy(dtype=...)` out / `Image.new_from_array(arr)` (the `graphic/color/managed#MANAGED` `_icc_apply` lifts the colour-science-toned `uint8`/`uint16` array via `pyvips.Image.new_from_array(toned)`) / `new_from_memory(buf, w, h, bands, format)` in — a canonical `numpy` band-interleaved buffer is the one host raster surface, so a `scikit-image`/`colour-science`/`pillow`-produced array enters the libvips pipeline and a libvips result leaves to the array tier without a bespoke pixel struct; `numpy` is preferred over `tolist` for any real raster (the buffer-protocol zero-copy path).
- universal `anyio` tier (`libs/python/.api/anyio.md`): the `write_to_*`/egress pull is native CPU work AND pyvips binds a Forge-provisioned native `libvips` off the runtime loader path, so every arm crosses the ONE `graphic/raster/io#IO`-/`faults`-owned `anyio.to_process.run_sync(_worker_raster, op, limiter=WORKER_BAND)` SUBPROCESS seam (the `graphic/color/managed#MANAGED` arm `to_process.run_sync(_icc_apply, ..., limiter=_ICC_LANE)`) bounded by the shared `execution/lanes#LANE` `WORKER_BAND` `CapacityLimiter` the `exchange/detect`/`exchange/metadata`/`graphic/raster/measure`/`process`/`graphic/color/managed` worker lane shares — never a per-owner `CapacityLimiter(slots)` knob that oversubscribes the host against libvips's own internal pool, never the unbounded per-loop default, never a thread slot, and never a `to_interpreter` subinterpreter (which shares the host interpreter version and cannot host the native stack); `concurrency_set` is bound DOWN against the outer fan. - universal `expression` tier (`libs/python/.api/expression.md`): the typed `pyvips.Error` maps at the worker boundary into the consuming owner's closed `RasterFault`/`ManagedFault` `@tagged_union` (`pyvips.Error -> .engine`, dlopen `OSError -> .provision`) and returns through `Result[RasterFact, RasterFault]` (`Ok`/`Error`, `bind`/`map`, the batch `Block`-collected as `RuntimeRail[Block[Result[ArtifactReceipt, RasterFault]]]`) so one corrupt input faults its own slot without aborting the farm; the `try/except pyvips.Error` lives only in the `_vips_guarded` worker adapter, never in the domain pipeline.
- universal `structlog`/`opentelemetry` tier: the per-op evidence (loader, `Access` mode, dimensions, ICC presence + intent/BPC/PCS/depth, page count, chained operation list, `pyvips.base.version`, byte length) is the structured event/span payload folded into the `RasterFact`/`ManagedFact` `msgspec.Struct`; the native `libvips` version and `get_suffixes()` feature set are read once at boundary init so they ride the receipt as deployment facts.
