# Exhaustiveness Under Tag Widening

[TWO_DETECTORS_ONE_OBLIGATION]:
- Totality is enforced by two structurally distinct detectors, never one: *residual totality* gates `Match.exhaustive` on the recomputed third matcher slot being literally `never` (`Matcher<I, F, never, A, Pr, Ret>`), and *record-shape totality* gates `$match`/`tagsExhaustive`/`discriminatorsExhaustive` on a handler record `P` whose key set must equal the tag alphabet — `[Tag in Types.Tags<"_tag", R> & string]: Fn` demands every tag and `[Tag in Exclude<keyof P, Types.Tags<"_tag", R>>]: never` forbids any foreign tag. The same widening event re-opens the obligation through two different type-level paths, so which detector a dispatch site uses decides *how* a new arm breaks it and *what the diagnostic says*.
- Residual totality is a one-sided subtraction proof: each arm folds `AddWithout<F, consumed>` and `exhaustive` accepts only when `ApplyFilters<I, F>` collapsed to `never` — it proves *nothing is left*, blind to whether an arm matched a case the input never contained. Record-shape totality is a two-sided set-equality proof: the keys must *cover* `Tags<"_tag", R>` (missing-arm error) and must *not exceed* it (stale-arm error) — it proves *the handler set equals the tag set* in both directions. The subtraction proof cannot catch a stale arm; the set-equality proof catches both gaps with one keyed error per offending key.
- The selection is not stylistic: a dispatch that must forward its leftover takes the residual detector (the leftover is a typed channel); a dispatch that closes in place over a `_tag` family takes the record detector (no leftover, the most collapsed spelling). Routing a closed `_tag` dispatch through `when`/`tag` arms plus a trailing `exhaustive` forfeits the stale-arm diagnostic the record form gives for free — a removed tag whose `when`-arm survives is dead code residual totality never reports.

[WIDENING_VECTOR_TRICHOTOMY]:
- A variant family widens along three vectors, each re-opening totality at a different declaration site with a different blast radius: *arm-append re-declaration* (`Schema.Union(...existing, NewArm)`) widens `Schema.Type<Members[number]>` at the union value, so every `Schema.Type<typeof Union>` extraction and every matcher seeded on it re-opens at once; *type-level alphabet widen* (a new key on the `Data.taggedEnum<A>` literal definition, or a new self-type slot inhabitant) widens `A["_tag"]` at the type owner, re-opening every `$match` and `$is` keyed on it; *input-position widen* (a matcher seeded `Match.type<Schema.Type<typeof Wider>>()` over a union that absorbed an arm) re-opens `Remaining` purely by `I` growing under a frozen filter sequence.
- The vectors are not interchangeable in what they break. Arm-append over a `Schema.Union` re-opens both detector families — `Schema.is`/decode see the new arm structurally, and any `Match.type<Schema.Type<...>>` over the decoded value re-opens residually — so the wire owner's widen is the loudest, breaking decode totality and dispatch totality together. The `Data.taggedEnum` type-level widen re-opens only the record detector (`$match`'s `[Tag in A["_tag"]]`), because the decode-free owner carries no `Schema.is` to re-open and no codec search tree to re-partition — its sole compile-time alarm is the `$match` key gap.
- Input-position widen is the silent one: nothing about the new arm appears at the matcher's *arm* declarations — the filter sequence `AddWithout<...>` is byte-identical — yet `ApplyFilters<I_wide, F>` recomputes a non-`never` residual purely because `Exclude<I_wide, consumed>` now retains the new member. The break surfaces only at the `exhaustive` finalizer with `Type '<NewArm>' is not assignable to type 'never'`, pointing at the *finalizer*, never at the union edit that caused it — the diagnostic locality is one declaration removed from the cause.

[RECORD_TOTALITY_IS_VECTOR_BLIND]:
- Record-shape totality keys on `Types.Tags<"_tag", R> = R extends Record<"_tag", infer X> ? X : never` — the literal-union projection of the `_tag` field across the input union — so it re-opens identically under *every* widening vector that grows that projection, whether the arm arrived by `Schema.Union` re-declaration, by a new `Data.taggedEnum` key, or by an `Extract`-narrowed input being re-widened. The detector reads only the resulting `_tag` alphabet, never the path that produced it, so one `$match`/`tagsExhaustive` site gives the same keyed `Property '<NewTag>' is missing` error regardless of vector.
- The vector-blindness is what makes the record detector's diagnostic *byte-stable* across vectors where the residual detector's is not: the next arm — by any vector — lands as exactly one required key named at the dispatch site (`Property '<NewTag>' is missing`), and a stale arm as exactly one forbidden key (`Exclude<...>: never`), whereas residual totality's diagnostic for the same widen names the *finalizer* and varies with which `I` member survived (`Type '<NewArm>' is not assignable to type 'never'`). Diagnostic locality and stability — not just detection — is the axis the record detector wins on the closed-family case.

```typescript
import { Match, Schema } from 'effect'

class Alpha extends Schema.TaggedClass<Alpha>()('Alpha', { weight: Schema.Int }) {}
class Beta extends Schema.TaggedClass<Beta>()('Beta', { label: Schema.NonEmptyString }) {}
const Owner = Schema.Union(Alpha, Beta)
const classify = Match.typeTags<Schema.Schema.Type<typeof Owner>>()({
    Alpha: ({ weight }) => weight,
    Beta: ({ label }) => label.length,
})
```

- `typeTags()` self-finalizes on the handler record alone, so appending `Gamma` to `Owner` makes `classify`'s record fail with `Property 'Gamma' is missing` at the record literal — one keyed diagnostic at the dispatch site, not a deferred `never`-mismatch. The rejected spelling is `Match.type<...>().pipe(Match.tag('Alpha', ...), Match.tag('Beta', ...), Match.exhaustive)`: it covers identically today, but a later *removal* of `Beta` leaves `Match.tag('Beta', ...)` as a stale arm matching a tag the union no longer contains, dead code `exhaustive`'s subtraction proof cannot flag — `typeTags` rejects the dead `Beta` key as `never`.

[DEFAULT_HANDLER_SWALLOW]:
- The optional-handler twins are the silent-swallow surface that survives widening without a diagnostic: `Match.tags(handlers)` and `Match.discriminators(field)(handlers)` type their record as `[Tag in Types.Tags<"_tag", R> & string]?: ((_: ...) => Ret) | undefined` — every key *optional* — and thread `AddWithout<F, Extract<R, Record<"_tag", keyof P>>>` so the matcher stays open with `Remaining` non-`never`, returning the matcher rather than self-finalizing. The trailing `Match.orElse` then absorbs whatever the partial record declined.
- Under widening this is a true silent failure, not a loud one: append an arm, and the `tags` record — keys optional — still type-checks unchanged, the new arm falling through to the `orElse` fallback whose handler is typed at the *recomputed* residual `RA` and so silently widens to include the new case. No key is demanded, no `never` mismatch fires; the new arm is swallowed by the default at runtime with full type approval. The `?`-keyed record is the one `_tag` dispatch shape where a new variant lands in the catch-all without the checker ever naming it.
- The boundary that converts the swallow back into a loud break is the finalizer choice, not the combinator: `tags` followed by `exhaustive` is impossible to satisfy with a true catch-all (the `orElse` would have consumed the residual already), but `tags` whose *intended* total set is closed must instead be `tagsExhaustive` — the required-key record — so a partial `tags(...).pipe(orElse(default))` is correct only when the default is a *deliberate* policy over genuinely-open future arms, and is a latent swallow everywhere the family is meant to be closed. The rule: a closed family takes the required-record detector; the optional-record plus default is reserved for an explicitly-open family whose new arms *should* route to the default.

```typescript
import { Match } from 'effect'

type Signal =
    | { readonly _tag: 'ready'; readonly at: number }
    | { readonly _tag: 'busy'; readonly since: number }
    | { readonly _tag: 'down'; readonly code: number }

const handled = Match.type<Signal>().pipe(
    Match.tags({ ready: ({ at }) => at, busy: ({ since }) => since }),
    Match.orElse(({ code }) => code),
)
```

- `tags({ ready, busy })` leaves `down` in the residual, `orElse` narrows its handler param to exactly `{ _tag: 'down'; code: number }` — correct *today*. Append `{ _tag: 'fault'; severity: number }` and the record still compiles (keys optional), `fault` silently joining `down` in the `orElse` residual whose param widens to `down | fault` with `code` no longer present on every member — the first downstream property read breaks, two edits removed from the swallow. The loud-break spelling is `Match.tagsExhaustive({ ready, busy, down })`: appending `fault` fails the record immediately with `Property 'fault' is missing`, the swallow impossible because the detector demands the key.

[PHANTOM_RESIDUAL_POISONS_THE_GATE]:
- A bare-predicate arm (one whose consumed slice computes to `never`, matching at runtime yet subtracting nothing) leaves the case it handles sitting in `Remaining`, so the matcher reads as incomplete at the finalizer even though every case is handled — and the programmer reaches for a non-totality close to silence the phantom. That close is the widening hazard: a finalizer chosen to paper over a phantom residual cannot distinguish the phantom from a genuinely-new arm a later widen adds.
- The trap compounds under widening precisely because the silencing close hides the new arm too: closing the phantom-residual matcher with `orElseAbsurd` (which does *not* demand `Remaining = never`) asserts the leftover unreachable at runtime — correct while the leftover is genuinely the handled-but-unsubtracted case, but the *same* `orElseAbsurd` now also covers any arm a later widen adds, because both the phantom case and the genuinely-new case sit in the same non-`never` residual the absurd close swallows. The phantom residual converts `orElseAbsurd` from an empty-residual proof into a blanket catch that a new arm hides inside.
- The fix is upstream of the finalizer and survives widening: route the narrowing arm as a `Predicate.Refinement<A, B>` (`(x): x is B`) — mapping to `SafeRefinement<B, B>` whose `B` second slot *is* subtracted — so the arm consumes its slice, `Remaining` shrinks honestly, and the matcher closes on `exhaustive` (the `never`-demanding finalizer). A new arm then re-opens `Remaining` loudly because no phantom residual is masking the gap, and `exhaustive`'s `Type '<NewArm>' is not assignable to type 'never'` fires at the finalizer. The boundary: a refinement keeps the residual honest so widening breaks loudly; a bare predicate forces a non-`never` close that a widen silently joins.

[NOT_FLIP_COLLAPSES_TO_NEVER_UNDER_WIDEN]:
- The `not` combinator is the one arm that can turn widening into a *false* exhaustiveness pass rather than a missed one. Because a `not`-chain can collapse its `Filters` to `never` (an `Only` window a matched slice escapes), and `ApplyFilters<I, never>` is `never` for *any* `I`, a poisoned `Filters` makes `exhaustive` accept unconditionally — the residual is `never` by collapse, not by coverage, and the collapse condition is a function of `I`.
- Widening weaponizes this: a `not`-built matcher tuned so its `Only` window exactly fit the pre-widen `I` can, after an arm-append grows `I`, have its window-escape condition flip — the new arm pushes a `not`'s matched slice outside the surviving `Only` set, collapsing `Filters` to `never` and satisfying `exhaustive` *for the wrong reason*: the new arm is unhandled at runtime yet the type-level residual is empty by lattice poisoning. This is the inverse failure of input-position widen — there the residual re-opens loudly, here it stays `never` falsely.
- The discipline that immunizes a `not`-bearing matcher against widening: pair each `not(pattern, f)` with exactly one positive arm consuming `pattern`'s own slice (`AddWithout<Only<X>, X> = Only<never>`, `ApplyFilters` honestly `never`), and never let two `not` arms intersect their `Only` windows in a way a future `I` can escape. The `not` lattice flip is safe only when the `Only` set is closed by a positive consumer in the same chain — an open `Only` window is a widening landmine, because the next arm decides whether it shrinks honestly or collapses falsely.

```typescript
import { Match } from 'effect'

type State =
    | { readonly _tag: 'idle' }
    | { readonly _tag: 'active'; readonly n: number }
    | { readonly _tag: 'failed'; readonly why: string }

const describe = Match.type<State>().pipe(
    Match.withReturnType<string>(),
    Match.not({ _tag: 'failed' }, (live) => `live:${live._tag}`),
    Match.tag('failed', ({ why }) => `failed:${why}`),
    Match.exhaustive,
)
```

- `not({ _tag: 'failed' }, ...)` flips `Filters` to `Only<failed>` (the complement `idle | active` handled, `failed` still owing), the trailing `tag('failed', ...)` folds `Only<never>` so `Remaining` is honestly `never`. Append `{ _tag: 'killed'; signal: number }`: `I` grows, the `not`'s matched slice is now `idle | active | killed` and its `Only` window recomputes — the positive `tag('failed')` no longer closes the widened window, `Remaining` re-opens at `killed`, and `exhaustive` breaks loudly. The rejected spelling is two stacked `not` arms with no positive consumer: their intersecting `Only` windows can collapse to `never` under the widen, passing `exhaustive` while `killed` runs unhandled.

[NULLOR_WIDENS_THE_ALPHABET_SILENTLY]:
- Wrapping a union owner in `NullOr`/`UndefinedOr` widens the input alphabet with a member carrying *no* `_tag`, and this interacts with the two detectors asymmetrically. `Schema.NullOr(Owner)` yields `Schema.Type` of `Owner | null`, so a `Match.type<Schema.Type<typeof NullOr>>()` matcher's `Tags<"_tag", I>` projection is unchanged (`null` contributes no `_tag`), and a `tagsExhaustive` record over the original tags type-checks — yet `Remaining` under residual totality retains `null`, because `Exclude<I, consumed>` keeps the tagless member. The record detector is blind to the nullable widen; the residual detector catches it.
- This splits the two detectors on the one widen that touches the input alphabet without touching the discriminant axis: a wire field that becomes nullable re-opens `Match.exhaustive` (loud, at the finalizer, with `Type 'null' is not assignable to type 'never'`) but does *not* re-open `tagsExhaustive` (the `_tag` alphabet is identical), so a family dispatched through the record detector silently admits the new `null` inhabitant to whatever its fallback or its first structural arm claims. The detector that keys on the discriminant alphabet cannot see a widen that adds a member *outside* the discriminant axis.
- The reconciliation routes the tagless widen back onto the discriminant axis before dispatch: admit `null` as a genuine tagged arm (a `Schema.tag`-defaulted `Absent` variant) so the alphabet widen *is* a `_tag` widen both detectors see, rather than leaving a tagless member only residual totality tracks. The rule: keep every inhabitant on the discriminant axis the dispatch keys on, so one widen re-opens both detectors uniformly — a member off the axis is the asymmetry that lets the record detector pass a case the residual detector would have caught.

[RETURN_TOTALITY_IS_A_THIRD_AXIS]:
- A new arm re-opens a *third* obligation orthogonal to case totality: return-type consistency, gated only when `Match.withReturnType<Ret>()` sits first in the chain. The finalizer's constraint `[Ret] extends [[A] extends [never] ? any : A]` passes `any` while `Result` (`A`) is still `never`, so a `withReturnType` placed before any arm pins `Ret` and then constrains every subsequent arm's `ReturnType<Fn>` against it — placed *after* an arm, it compares against an already-populated `A` and stops gating later arms, including any the next widen adds.
- The widening consequence: an arm-append on the union re-opens case totality (the new arm re-enters `Remaining`) and, *if and only if* `withReturnType` was placed first, also gates the new arm's return type — so a matcher built with `withReturnType` first breaks twice on a widen that returns the wrong type (missing-case at the finalizer, return-mismatch at the new arm), while one with `withReturnType` placed late breaks only on the case gap and silently admits a new arm returning a divergent type that `Unify` then widens the whole `Result` to absorb. Return totality is a per-arm gate the placement decision arms or disarms for *future* arms, not present ones.
- The rule that keeps both axes loud under widening: `withReturnType` is the first pipe step on any matcher whose return contract must survive arm growth, because its gate is forward-looking — it constrains arms declared *after* it, so it must precede the entire arm sequence to cover the arms a later widen appends. A matcher that pins its return type with an arm-return annotation instead of `withReturnType` first leaves the family's return contract un-gated against the next case, the divergent return absorbed into a widened `Unify<A>` rather than rejected.

[STALE_ARM_IS_NARROWING_WIDENING]:
- Removal — the dual of append — is a widening of the *consumer's obligation surface* the residual detector cannot see and the record detector can. Drop an arm from a `Schema.Union` and `Schema.Type<Members[number]>` narrows; a `Match.type<...>().pipe(tag(removed), ..., exhaustive)` chain still type-checks because `AddWithout<F, Extract<R, Record<"_tag", removed>>>` over a tag no longer in `R` subtracts `never` — a harmless no-op — and `exhaustive` sees the same `never` residual, so the dead `tag(removed)` arm survives as unreachable code with full type approval.
- The record detectors reject it: `tagsExhaustive`/`$match`/`typeTags` all carry `[Tag in Exclude<keyof P, Types.Tags<"_tag", R>>]: never`, so a handler key for the removed tag is forbidden — `Type '(_: never) => ...' is not assignable to type 'never'` at the stale key. This is the asymmetry the two-detector law turns into a maintenance rule: append-widening breaks every detector loudly, but remove-widening (stale arm) breaks *only* the record detectors — a codebase relying on `exhaustive` over `when`/`tag` ladders accumulates dead arms invisibly, while one built on `tagsExhaustive`/`$match` has its dead arms rejected at the next compile.
- The consolidated law across append and remove: residual totality is a coverage proof (loud on append, silent on removal), record-shape totality is an equality proof (loud on both), so a dispatch surface that must stay tight under *both* directions of tag change takes the record detector — the equality proof is the only one that keeps the handler set and the tag set in lockstep, and the `when`/`tag`-plus-`exhaustive` form is admitted only for genuinely heterogeneous patterns (refinements, structural shapes) the record detector cannot key on.
