# State-Receipts: Deepening

[STM_ISOLATION_MECHANISM]:
- `Isolation.Snapshot` (default) does NOT mean reads go unchecked: `CommitWrites` compares `s[write].Version == refState.Version` (the global committed version versus the transaction-snapshot version) and throws `ConflictException` on mismatch. Write-write conflicts are always detected and cause a retry. Only the read-set is skipped during `ValidateReads` — so non-repeatable reads and phantom reads are possible, but blind writes are not. The correct characterization is optimistic write-lock, not snapshot isolation in the database sense.
- `Isolation.Serialisable` adds `ValidateReads` before commit: it iterates `transaction.reads` (a `HashSet<long>` populated by every `transaction.Value.ReadValue(id)` call) and checks `s[read].Version != trieMap[read].Version`. Any version change in the read-set triggers `ConflictException` and a full retry. This is a true serializable optimistic check, not pessimistic locking.
- Reads are tracked unconditionally inside a transaction regardless of isolation mode — `STM.Read` always calls `transaction.Value.ReadValue(id)` which always calls `reads.Add(id)`. The isolation mode only controls whether `ValidateReads` runs at commit time. A transaction that reads many `Ref`s under `Snapshot` accumulates a full read-set with no benefit; prefer `Serialisable` only when read-set validation is required.
- `Ref.Value` reads outside `atomic()` do not track reads — `STM.Read` checks `transaction.Value != null` first and reads directly from `state.Items[id].UntypedValue` when no transaction is active. The `state` field is `AtomHashMap<EqLong, long, RefState>`, and `Items` is a `volatile` reference to the current `TrieMap`, so direct reads are visibility-safe.

[STM_COMMIT_ARCHITECTURE]:
- The entire commit — validation, write application, and commute application — executes inside a single `state.SwapInternal(delegate ...)` CAS spin loop. `SwapInternal` is a pure CAS on the `volatile TrieMap` reference with `SpinWait.SpinOnce()` on contention. There is no mutex, no monitor, and no `lock`. Two concurrent `ValidateAndCommit` calls compete on this CAS; one wins and the other re-spins with the updated global state snapshot. This is the only global serialization point for STM commits.
- `commute` applies `f` to the transaction-local state snapshot immediately (for in-transaction reads), AND appends `(id, f)` to the `commutes` list. At commit time, `CommitCommutes` runs AFTER `CommitWrites` with the updated global state `s` — so commuted functions apply against the latest committed values, not the transaction snapshot. A `Ref` can appear in both `writes` and `commutes`; the write sets the version-verified value, then the commute applies `f` to that written result.
- `commute`'s conflict-freedom comes at cost: `RefValidationFailedException` is not retried inside `CommitCommutes` — if the commuted `f` produces a value that fails the `Ref`'s validator, `RefValidationFailedException` escapes directly from `atomic()`. Validators on commuted `Ref`s must be unconditional invariants, never conditional-on-current-value checks, because the commute function sees the latest committed value, not the transactional snapshot.
- `STM.transaction` is `AsyncLocal<Transaction?>` not `[ThreadStatic]`. `AsyncLocal` flows through `await` continuations — a transaction started on thread A, suspended at `await`, and resumed on thread B continues the same transaction. `Task.Run` inside `atomic()` creates a new execution context that does NOT inherit the `AsyncLocal` value; `Task.Run(() => ref.Value = x)` throws `InvalidOperationException` even though the outer `atomic` is active.

[ATOMHASHMAP_OBSERVER_OPTIMIZATION]:
- Every scalar mutation on `AtomHashMap` (`SetItem`, `Add`, `Remove`, `SwapKey`, `FindOrAdd`, etc.) checks `this.Change != null` before entering the `WithLog` variant of the underlying trie operation. When no observer is subscribed, the plain mutation path runs and no `Change<V>` object is allocated. This optimization is pervasive — every one of the ~40 mutation paths carries the conditional. Subscribing even one handler to `Change` forces the full audit-log path on every operation.
- The bulk `Swap(Func<TrackingHashMap, TrackingHashMap>)` path always constructs a `TrackingHashMap` wrapper (allocates the change-tracking struct), regardless of whether `Change` is subscribed. The `TrackingHashMap` accumulates per-operation `Change<V>` values using `ChangesInternal.AddOrUpdate(key, ex => ex.Combine(pair.Change), pair.Change)` — i.e., the `Change<V>` monoid `Combine` collapses multiple mutations to the same key across one `Swap` call into a single compressed change record. The patch delivered to `AnnounceChanges` reflects the net delta of the entire bulk function body, not intermediate states.
- `HashMapPatch<K,V>` internally stores either a `(key, change)` pair (scalar ops) or a `TrieMap<K, Change<V>>` (bulk ops), and constructs the `HashMap<K, Change<V>> Changes` property lazily. For a scalar op this construction is a single-entry `HashMap.Empty.Add(key, change)` — O(1) allocation on demand. Observers that only call `patch.Changes` for every event pay this allocation on every mutation; observers that pattern-match on whether `patch.Changes.Count == 1` can take a fast path for the common scalar case.

[CHANGE_MONOID_COMPOSITION]:
- `Change<A>.Combine(y)` is the semigroup/monoid `Combine` used to collapse sequential changes to the same key. The composition laws sourced from the implementation: `NoChange + y = y`, `x + NoChange = x`, `any + Removed = Removed`, `Removed + Added(v) = Mapped(removed_value, v)`, `Added(x) + Mapped(->y) = Added(y)` (the add wins, updated to the final value), `Mapped(from, x) + Mapped(->y) = Mapped(from, y)` (origin preserved, destination updated). An observer receiving the merged patch from a bulk `Swap` that inserts then updates the same key gets exactly one `EntryMapped` with the original-absent origin — not an `EntryAdded` followed by an `EntryMapped`. Code that distinguishes insertion from update based on `EntryAdded` vs `EntryMapped` must use key-level `SwapKey` or listen to scalar ops to see truly atomic per-operation changes.
- `EntryMapped<A,B>` implements both `EntryMappedFrom<A>` and `EntryMappedTo<B>` as separate covariant/contravariant interfaces. Pattern matching the change as `EntryMappedTo<V>` extracts only the new value; `EntryMappedFrom<V>` extracts only the old value; pattern matching as `EntryMapped<V,V>` gives both. Change observers that only need the new state should pattern-match `EntryMappedTo<V>` to avoid binding to the `From` type when in-place value mutation changes the wrapper type.

[ATOMQUE_EVENT_ASYMMETRY]:
- `AtomQue<A>` fires `Dequeued` and `Enqueued` events (distinct `AtomDequeuedEvent<A>` and `AtomEnqueuedEvent<A>` delegate types) after each successful CAS. This is fundamentally different from the other collection variants: `AtomHashMap` fires one `Change` event with a typed patch, `AtomSeq` has no events at all, and `Atom<A>` fires `Change` with the new value. The asymmetric per-operation signals on `AtomQue` map naturally to producer-consumer notification patterns without requiring a full patch diff.
- `AtomQue.Clear()` fires `Dequeued` for every element that was in the queue at the moment of the CAS success — it iterates the captured pre-clear `QueInternal` and fires per-element. `Clear` is not atomic with respect to `Dequeued` delivery: a concurrent `Enqueue` after the CAS may put an element in the queue that `Clear` does not notify about. Observers that must account for every element must use `Dequeue` in a drain loop rather than `Clear`.
- `AtomQue` exposes no `Swap(Func<Que<A>, Que<A>>)` overload. Multi-step conditional queue operations — conditional dequeue based on peeked value, conditional enqueue with capacity check — require a higher-level `Atom<Que<A>>` or a `Ref<Que<A>>` inside `atomic()`. The scalar-CAS-only API trades bulk transactional capability for the `Dequeued`/`Enqueued` event granularity.

[MEMO_FAILURE_AND_RESET]:
- `Memo<F,A>.GetValue()` uses `Interlocked.CompareExchange(ref state, 1, 0)` to transition from `unloaded(0)` to `loading(1)`. A thread that wins this CAS evaluates `acquire()`. On exception, it resets `state` to `0` and rethrows — the memo becomes re-acquirable. A second thread spinning while `state == 1` simply `SpinWait.SpinOnce()` until the first either sets `state = 2` or resets to `0`. There is no timeout: if the acquire function deadlocks, all waiting threads spin indefinitely. The acquire function must be safe to call exactly once and must not block on anything that requires a memo-protected resource.
- `Memo<F,A>.Reset()` transitions `state` from `2(loaded)` to `1(loading)` via CAS, sets `value = null`, then sets `state = 0`. It is a no-op if `acquire == null` (literal-value memos created via `memoK<F,A>(K<F,A>)` store no acquire function and are permanently fixed). `Clone()` creates a new unloaded `Memo` using the same `acquire` delegate — this is the correct pattern for re-usable lazy computation cells that must be independently re-acquired per use site.
- The `~` operator on `Memo<F,A>` is `extension<F,A>(Memo<F,A> self) { public static K<F,A> operator ~(Memo<F,A> ma) => ma.Value; }` — it materializes the lazy node by calling `Value` and returning the `K<F,A>`. This is the forcing operator; it triggers acquisition if not yet loaded. `memoK` used as a lazy `Applicative` node should only be forced (`~`) at the point where the effect is to be executed, never at composition time.

[MEMO_WEAK_EVICTION_HAZARD]:
- `memo<A,B>(Func<A,B>)` uses `WeakDict<A,B>`: values are wrapped in `OnFinalise<B>` which removes its own `ConcurrentDictionary` entry in its finalizer. The cache entry is keyed by `A` in the dict and the value is a `WeakReference<OnFinalise<B>>`. Eviction is GC-driven: if no strong reference to the computed value exists outside the cache, the `WeakReference` becomes collectible and the entry is evicted. For value types and newly-created reference types with no external reference, this means: `memo<int, SomeRef>(compute)(42)` may call `compute(42)` again on the very next call if `SomeRef` was not captured elsewhere. The per-key `object` lock in `syncMap` prevents double-computation within the same GC cycle but offers no cross-GC-cycle guarantee.
- The per-key lock in both `memo` and `memoUnsafe` uses `syncMap.GetOrAdd(key, new object())` — this may allocate a new lock object even on a cache hit path, then discard it if the key already exists. `ConcurrentDictionary.GetOrAdd` is documented to potentially call the value factory multiple times under contention; multiple lock objects for the same key may be allocated and one survives. The surviving lock object governs exclusive computation for that key for the duration of the `lock(value2)` block only; `syncMap.TryRemove` is called immediately after the lock exits, so the key is unguarded for subsequent calls. This design is correct for idempotent computations and incorrect for non-idempotent ones.

[SNAPSHOT_ISOLATION_WRITE_CONFLICT_DETAIL]:
- `CommitWrites` checks `s[write].Version == refState.Version` where `refState = t.state[write]` (the transaction-snapshot version). The comparison is a long equality on `Version` fields, not a reference check. A concurrent transaction that writes and commits the same `Ref` between the current transaction's snapshot and commit will have incremented the version via `refState.Inc()`, making the versions unequal and triggering `ConflictException`. The retry loop in `RunTransaction` catches `ConflictException` and re-runs `op()` with a fresh snapshot of the global state — reading `Ref.Value` in the new `op()` invocation sees the committed concurrent value.
- Both `Snapshot` and `Serialisable` isolation use the same `ConflictException` retry loop. There is no `abort()` escape hatch — all `ConflictException`s inside `RunTransaction` trigger retries. The only non-retryable exits are: `RefValidationFailedException` (propagates immediately), any exception thrown by `op()` itself (propagates immediately), and `atomic` called when already in a transaction (the inner call is a no-op join returning `op()` directly — no retry scope).
