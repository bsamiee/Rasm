# Rich Class Owner Mechanics

# Owner-Surface Triage Under Admitted Packages

- `msgspec.Struct` owns wire/decode/report carriers with C-accelerated encode/decode, `gc=False`, `omit_defaults`, `forbid_unknown_fields`, `tag_field`; behavior beyond field bags stays on the same owner via methods and `classmethod` factories, never parallel DTO siblings.
- `pydantic.BaseModel` / `BaseSettings` owns ingress/configuration with env parsing, alias routing, `field_validator` / `model_validator`, `computed_field`, `extra="forbid"`; domain transitions that re-run validators use `model_validate` over a field snapshot, not attribute mutation.
- Stdlib `@dataclass(frozen=True, slots=True, kw_only=True)` owns decorator-driven CLI/param capsules needing `copy.replace` without pydantic overhead; promote to msgspec or pydantic when wire or boundary validation becomes load-bearing.
- cyclopts reads dataclass field metadata, not pydantic `computed_field` — CLI params and settings export remain split owners until a decorator admits both projection lattices without tier collapse.
- Repo semantic policy treats all three bases as mandatory-frozen model owners; omitting `frozen=True` is a shape defect regardless of method density.

# Memory Layout, Sealing, And Disjointness

- `slots=True` and msgspec slot layout eliminate per-instance `__dict__`; pair with `frozen=True` so assignment errors surface at the owner.
- `gc=False` on msgspec structs disables GC tracking only when the graph cannot form cycles; cyclic rich owners keep `gc=True` or relocate cycle-prone state.
- `cache_hash=True` memoizes `__hash__` for dict/set key duty in hot folds — registry/catalog rows with typed enum axes; bare string fields signal passive-record drift unsuitable for keying.
- msgspec `cache_hash=True` owners used as registry or catalog keys must not be re-keyed after `structs.replace` when any replaced field participates in equality — hash memoization assumes immutability after materialization.
- `@typing.final` and `@typing.override` document non-extensible algorithm arms and checker-dead branches after `isinstance`; they do not runtime-block subclassing — do not rely on `@final` alone for wire security.
- PEP 800 (`typing.disjoint_base`, Python ≥3.15) auto-infers disjointness on non-empty `__slots__` including `@dataclass_transform`-synthesized pydantic layouts; explicit `@disjoint_base` on nested `_Seal` remains required when variant siblings share a non-slot base or exhaustiveness must be nominal beyond slot inference — supplements but does not replace explicit `TypeIs` predicates at ingress.
- Rich owners cannot multiply-inherit unrelated disjoint bases; composition uses nested frozen sub-owners, not dual msgspec/pydantic inheritance.

# Replacement Algebra And Validated Transitions

- `copy.replace(obj, **changes)` (3.13+) is the canonical cross-family replacement entry for dataclasses, msgspec structs, and pydantic models; doctrine treats `__replace__` as implementation detail routing through `copy.replace` or family aliases (`msgspec.structs.replace`, `dataclasses.replace`).
- Pydantic `__replace__` forwards to `model_copy(update=...)` without re-validation — trusted same-type swaps may use `model_copy`/`copy.replace`; untrusted or computed deltas require `model_validate(snapshot | delta)`.
- No admitted package provides validated `copy.replace`; `with_configuration`-style full-field snapshot + `model_validate` remains the validated path; partial nested env merge is construction-only (`nested_model_default_partial_update=True`), not a domain transition primitive.
- `copy.replace` on pydantic accepts Python field names only, not `Field(alias=...)` keys — boundary adapters normalize before replacement.
- msgspec/dataclass `__post_init__` re-executes on `copy.replace`, decode, and `convert(..., from_attributes=True)`; it does not run on `copy.copy`.
- msgspec `copy.deepcopy` still invokes `__post_init__` (msgspec 0.21.1, PR #933 target) while dataclass deepcopy does not — ban deepcopy on struct owners with load-bearing post-init until upstream alignment.
- Decode/convert exceptions inside msgspec `__post_init__` wrap as `msgspec.ValidationError`; the same from `copy.replace` propagates raw — split hooks when surfaces must differ.
- Pydantic has no `__post_init__`; cross-field replay after replacement is absent (`model_copy`) or full (`model_validate`); post-init performing I/O or reading ambient environment violates replacement purity; dataclass `init=False` fields are not carried by `copy.replace`.
- `model_copy(update=...)` shallow-copies `__dict__` including `@cached_property` caches — updated fields can leave stale caches; Pydantic ≥2.13 permits `del` on frozen-model cache entries but doctrine prefers `@computed_field` without caching or recompute-via-new-identity (`model_validate` snapshot).
- Partial nested `model_copy(update={"nested": {"field": v}})` does not construct typed nested models (pydantic #12312) — owned transition methods assemble explicit child snapshots then `model_validate`; this primitive must not appear in owned transition methods.
- Frozen dataclass synthesized `__replace__` can defeat PEP 695 read-only generic boxes — set `__replace__ = None` for covariant read-only carriers; public transition methods pin `Self` or closed unions because `copy.replace` generic typing remains checker-unsettled.
- Full-field snapshot transitions iterate `type(self).model_fields`, materialize `{name: getattr(self, name) for name in fields}`, apply delta, call `__class__.model_validate(snapshot)` — every validator and coercion replays, not only the changed axis.
- Selective overrides on nested sub-owners merge explicit child snapshots (`{**self.artifact_backend.model_dump(), **{k: v for k, v in overrides.items() if v is not None}}` then `ArtifactBackend.model_validate(...)`) before parent fabrication — unfiltered spreads conflate omission with clearing; partial nested dict updates break typed nested models.
- Transition kwargs use domain vocabulary (`configuration`, `protocol`, `root`), not wire aliases — alias normalization stays on ingress validators; identity transitions after ambient coercion drift require full snapshot `model_validate`, not bare `model_copy()` without update — env reload is out of scope for materialized instances.
- dataclass/param transitions return `Self` via `copy.replace`; settings transitions return validated pydantic identity via `model_validate`; msgspec owners use `copy.replace` with optional `model_validate` at pydantic seams — never mix families in one method without explicit boundary conversion.
- `typing.Self` on instance methods (`bound`, `with_configuration`) gives checker evidence transitions stay inside the family; projection methods return boundary types; `Self` on `__new__` binds payload-carrying `StrEnum` member attributes — the enum case is the owner, not a bare string token.
- Enrichment kwargs on settings close as PEP 728 `TypedDict` with `closed=True` and `NotRequired[...]`; `Unpack[ConfigurationDelta]` on transition methods replaces loose `**overrides: object` when multiple named axes co-vary — one polymorphic entry owns all configuration deltas without stringly key routing; `closed=True` forbids ad hoc extension keys at enrichment call sites — surplus kwargs must fail static checking, not silently drop at runtime; keys omitted from the spread mean leave stored value; `None` assigned means clear optional field when the field type is optional.
- param-owner enrichment stays `copy.replace`-typed with field-name keys — cyclopts surplus never enters delta TypedDicts; delta assembly filters `None` at the method boundary to preserve omission versus explicit null.
- Transitions that can fail return `Result[Self, E]` or `Self | Fault`; shallow replace shares nested mutable sub-objects unless `deep=True` — `frozen=True` on parent does not freeze nested dict/list payloads; parent `copy.replace` swapping the entire child reference is preferred over mutating nested payloads.

# Construction Hooks And Validator Ownership

- msgspec forbids overriding `__init__`/`__new__` on `Struct`; alternate constructors are `classmethod` factories; ingress is `msgspec.convert`/`Decoder` plus `Annotated[..., msgspec.Meta(...)]` — document which lane runs first when the same invariant spans decode post-init and pydantic `model_validator`.
- Pydantic `field_validator`/`model_validator` are the owned hook surface; cross-field invariants belong in `mode="after"` on the model, not external guards; validators are classmethods by protocol — never instance methods or standalone functions receiving the model; `mode="after"` returns `Self`, not `cls`.
- `mode="before"` owns wire-safe single-field gates and ingress normalization for every construction path including env (`run_id` surrogate scrubbing, `exec_target` URL shape); `mode="after"` owns cross-field storage invariants provable from stored fields alone (`protocol` × `root` traversal, empty root on non-file backends) — must not call context-bound projections depending on parent settings or assume parent existence when sub-owners validate in isolation (`ArtifactBackend()`).
- `classmethod` settings sources (`settings_customise_sources` with `@override`) are construction hooks with source-ordering invariant — not module-level bootstrap; `nested_model_default_partial_update=True` merges env deltas at construction only when defaults were non-factory values (`Field(default_factory=ArtifactBackend)`), not on materialized `model_copy` or parent transition methods.
- Underscore validators (`_bounded_non_file_root`, `_wire_safe`, `_exec_target`) encode construction law; public methods encode projection and fabrication — law matrices and `register_law` target validator surfaces versus dispatch surfaces independently.
- `@dataclass(kw_only=True)` on rich owners with more than two fields keeps `copy.replace` call sites auditable and cyclopts binding explicit — positional-only param owners fight decorator introspection.

# Owned Projections: Field-Closed, Context-Bound, Serialization Tiers

- Three tiers on one pydantic rich owner: stored fields, `@computed_field` (schema + serialization + settings export), stdlib `@property` (pre-serialization interior only); mixing tiers without policy yields silent egress gaps.
- `@computed_field` over explicit `@property` preserves IDE/checker evidence — `@computed_field` alone auto-wraps with `property` but degrades static analysis; promote to `@computed_field` when the fold uses only stored fields and must appear in schema or settings export (`solution`, `agent_context`); keep `@property` when the fold is subprocess-interior only (`python_tool_env`, `store_root`) — interior `@property` folds omit computed-field export deliberately; exporting them widens settings schema and OTEL/env surfaces without policy review; reserve methods for folds needing parameters beyond `self` or fabricating non-owner products.
- `exclude_computed_fields` (pydantic ≥2.13) is an egress knob — interior transitions must not rely on dumps excluding computed evidence validators assume on re-validate.
- Field-closed projections fold stored fields only — `RailStatus.from_returncode`, enum payload reads (`exit_code`, `prefix`, `strategy`), and `match`/`case` arms on the owner need no ambient process state.
- Context-bound projections require explicit typed runtime parameters (`ArtifactBackend.target(settings: AssaySettings)`, `AssaySettings.artifact(kind, *parts)`, `AssaySettings.store(...)`) because results depend on anchors, workspace markers, or caller overrides not fully encoded in callee stored fields — context parameters must be typed owner surfaces, not bare `Path`/`dict`; ad hoc context bags reintroduce scattered-key routing the rich owner eliminated.
- Filesystem normalization (`UPath.expanduser`, absolute-vs-relative resolution against `settings.root`, segment safety) belongs in validators or context-bound methods, not `@computed_field` — running at construction, schema export, or on every property access bakes stale paths or executes IO during schema compilation and static introspection walks.
- Context-bound methods own laws requiring sibling fields on a parent settings owner or anchored paths only after materialization (`target(settings)`, segment safety on `artifact(kind, *parts)` against `store_root`).
- One projection concept, one owner method family — `store_root` on `AssaySettings`; `ArtifactBackend.target` resolves backend-relative roots against that anchor; duplicating root-resolution on both owners is a seam defect unless each owns a distinct projection role documented in the adapter.
- Context-bound methods that consume only a subset of a large settings graph accept structural port slices (`RootAnchor` as `@runtime_checkable` `Protocol` exposing `root: UPath` and `store_root` law) with `TypeIs[RootAnchor]` predicates proving minimal context law before `target(slice)` — narrowing replaces dict-shaped context bags or casts of the full settings owner.
- Declare slices as `@runtime_checkable` `Protocol` definitions colocated with the projecting sub-owner; re-decorate every protocol subclass used in `isinstance` checks — inheriting checkability without re-decoration is deprecated in Python 3.15 and becomes `TypeError` in 3.20.
- Generate law-matrix fixtures from `typing.get_protocol_members(Port)` — assert each required name exists on doubles before exercising projection behavior; reject erased `Mock` objects as conformance proof; parent settings owners remain the production context type — slices narrow test and cross-context imports, not production signatures, unless the method family genuinely depends on fewer axes.
- `Annotated[T, msgspec.Meta(gt=0, max_length=N)]` encodes per-field decode policy in the type position; `msgspec.inspect.type_info` enables owner-local operations (`field_cap`) without parallel constraint tables.
- Writable descriptors on ostensibly frozen owners break the faux-immutability contract — mutation routes through replacement.

# Nested Vocabularies, Variant Seals, And Dispatch

- Payload-carrying `StrEnum` cases colocate wire token, CLI flags, routing strategy, and severity via `__new__` binding (`Runner.prefix`, `Input.flag`, `Language.suffixes`, `RailStatus.exit_code`, `strategy`) — class-body names with annotation only are non-member descriptors; vocabulary tables outside the enum duplicate ownership and defeat `match`/`case` exhaustiveness.
- Python 3.13+ `enum._add_value_alias_` registers wire aliases at member construction; `@enum.verify(EnumCheck.UNIQUE | EnumCheck.CONTINUOUS | ...)` (≥3.15) replaces post-hoc repair; `enum.nonmember(...)` escapes class-body constants colocated with vocabulary.
- `classmethod` factories (`from_returncode`) belong on the enum when construction from an external scalar is vocabulary law — enum modules do not get parallel free functions for the same mapping.
- `@typing.disjoint_base` nested seals mark nominal variant families with runtime-class membership; pair with `TypeIs`/`isinstance` against sealed member classes, not string tags.
- Tagged generic owners (`Shape[T: Tag]`) collapse families when discriminant is a field; `@disjoint_base` split is for distinct runtime classes owning distinct method sets.
- Enum class-level operations (`join`, `fold`, `from_returncode`) may stay module-adjacent as the public semigroup or homomorphism when `cls` polymorphism adds no value — promote to `@classmethod` only when `cls` polymorphism adds value; do not promote enum folds to instance methods on unrelated owners (`rail_fold` consuming `RailStatus` streams stays adjacent to the status owner, not on `Report` or `Completed`); never scatter into `*_utils.py`.
- Dataclasses emit `__match_args__` by default; msgspec tagged unions require consistent `tag_field` — decode to tagged struct union first, then `match`.
- `assert_never` witnesses exhaustiveness under mypy `exhaustive-match`; closed variant sets use `match` on the rich class, not scattered isinstance ladders, string routing tables, or `@singledispatch` for sealed families — `singledispatch` remains the open-extension axis.
- `@tagged_union(frozen=True)` owns closed in-process error/envelope variants — promote to pydantic/msgspec when validators, settings sources, computed egress, or cross-call transitions attach; wire tagged msgspec unions project from canonical domain owners, not parallel wire/domain class hierarchies mirroring tags one-for-one unless wire shape is policy-canonical.
- Registry/catalog msgspec rows (`Tool`, `Check`, `Bind`) reference payload enums on fields and param types on `Bind.params` — behavior dispatches through registry handlers and param subclasses, not methods on wire structs; richness on rows is typed field vocabularies and bind params, not method density.
- Registry binds pair row identity with param subclass types — handler dispatch uses `match` on routed param products and fault morphisms, not string tool names alone; method families stay on settings, param, and service owners referenced by bind rows, not on wire structs unless the operation is pure projection from stored axes.

# Method Families, Fabrication, And Enrichment Composition

- Rich owners carry algorithm families: path projection (`target`, `artifact`), env projection (`remote_env`), fabrication (`store`), scope validation (`bound`), status folds (`join`, `fold` on `RailStatus`); fields without such families are record candidates for inline collapse.
- Static methods construct product types (`Fault`, `Completed`) without instance state; `classmethod` when dispatch varies by `cls` or construction from non-CLI carriers — CLI surplus routing stays instance-bound after cyclopts materialization.
- Protected hooks (`_arity(verb) -> int | None`) are the single override point; default hook bodies that ignore the discriminant (`_ = verb; return None`) document unbounded arity on the base param owner; subclasses narrow by overriding the hook alone, preserving one polymorphic `bound(verb) -> Self | Fault` surface across static, bridge, docs, and package param types — callers never override `bound` per subclass.
- Hook methods intentionally accept unused parameters for polymorphic signature stability; analyzer noqa on the hook signals virtual-dispatch intent, not passive-record drift.
- Static fault constructors (`surplus(verb, tokens) -> Fault`) stay on the base because fault product shape and wire-cap clipping are shared law; subclasses customize arity, not fault framing.
- Param-owner transitions never raise at the decorator boundary — surplus tokens become `Fault` values inside `bound`, matching registry `_bound` routing that expects `Self | Fault` evidence rather than parse exceptions.
- Fabrication constructs service types outside the owner family (`AssaySettings.store(...) -> ArtifactStore`): merge overrides into sub-owner fields, `model_validate` sub-owner, context-bound projection (`backend.target(self)`), construct external service with explicit constructor kwargs — order is load-bearing: validate sub-owner backend snapshot → project path through context-bound method → construct service with validated opts kwargs — opts validation runs before filesystem handle construction, not after; returned product is not `Self`.
- Fabrication does not mutate settings; repeated `store()` from the same frozen settings yields new `ArtifactStore` identities with equivalent logical roots — caches belong on the service or scope owner, not on settings.
- Fabrication methods that forward `**opts` to third-party constructors (`fsspec.filesystem`) admit a frozen sub-owner (`FabricationOpts` dataclass or pydantic model) when opts carry schema-worthy constraints — validate opts before passthrough, keep truly opaque backend knobs as documented escape hatches only; sub-owner defaults live in validated fields on the fabrication method signature — `store(..., opts: FabricationOpts = FabricationOpts())` or `opts: FabricationOpts | None = None` merged through `model_validate`, not bare `**opts: object` as the primary surface; fabrication sub-owners never mutate the parent settings owner — they are ephemeral validation capsules consumed at the service constructor boundary; when opts are empty for a backend, omit the sub-owner parameter entirely.
- Exclusive-resource storage law (memory filesystem singleton, process-global backend) belongs in `model_validator(mode="after")` on the sub-owner; fabrication methods assume legality already proved at construction.
- `@Parameter(name="*")` + frozen dataclass binds CLI surplus to `paths: tuple[str, ...]`; promote param owners to pydantic only when env/settings validation becomes load-bearing with explicit `model_validate` at the seam.
- Python ≥3.15 treats enrichment as the materialization stage for validated transitions, context projections, and service fabrication — ingress/wire never call owner method families directly; phantom stage parameters (`Record[Stage: Literal["canonical", "enriched"]]`) encode lifecycle without `{Concept}Draft` classes; transition methods return `Self` narrowed to the next stage literal when the pipeline is closed.
- Settings rich owners materialize at the settings-slice stage; domain enrichment consumes the frozen instance through composition-root handoff — env sources and `settings_customise_sources` do not re-run inside transition methods; `with_configuration`, `store`, and nested sub-owner override paths assume construction-law validators already proved storage invariants at process start.
- Wire egress and CLI cyclopts slices reject enrichment method inputs — only canonical field snapshots or adapter-projected structs cross those edges; method products like `ArtifactStore` are seam-local, not wire fields; stage mismatch at a projection edge is a static defect when `ty`/`mypy` binds stage parameters on the owner generic — runtime string stage fields are reserved for persistence envelopes, not in-process enrichment law.
- Enrichment transitions compose with `copy.replace` on dataclass/param owners and `model_validate` snapshots on pydantic settings owners within one handler — family mixing requires an explicit adapter at the seam, not an implicit `.model_dump()` handoff.
- Composition roots materialize settings once, bind port implementations, then thread the immutable settings instance through handlers — re-reading env inside enrichment or rail interiors violates snapshot discipline; enrichment return values rebind local names — callers must not assume prior settings identity remains authoritative after `with_configuration` or nested backend override fabrication.
- `ContextVar` tokens may hold request-scoped settings overrides when parallel tasks need divergent enrichment inputs — prefer immutable snapshot threading; ContextVar is for narrow override lanes, not a substitute for validated transitions on the owner.
- Import cycles resolve at the composition root — rich owner modules stay acyclic; registry rows reference owners by type, roots wire handlers to settings and param classes; cross-owner handoff exports immutable snapshots or fold-derived receipts — live mutable buffers and shared service handles are not re-published as canonical state at seams.
- Parent holds nested pydantic sub-owners; children expose context projections parameterized by parent (`target(settings: AssaySettings)`), not back-references — parent fabrication re-validates child snapshots on override instead of mutating child fields in place; nested sub-owner defaults via `Field(default_factory=ArtifactBackend)` participate in env partial-update merge at settings construction only; parent transition methods do not rely on `nested_model_default_partial_update` semantics after materialization; intermediate string roots are not stored on settings when the backend sub-owner is authority for protocol-specific root law.
- Path projection chains compose without duplicating anchors: `settings.store(...)` → `backend.target(settings)` → `ArtifactStore(root=...)`.
- `ArtifactScope` and `ArtifactStore` are separate rich owners with their own method families (`open`, `write_bytes`, `retain_history`); they consume projected paths from settings/backend owners rather than re-deriving roots from raw env.
- Remote execution projections (`remote_env(env) -> dict[str, str]`) filter caller-supplied env dicts against allowlists derived from stored fields and inject `ASSAY_RUN_ID` from settings — env assembly stays on the settings owner, not engine helpers; SSH launches inherit one projection surface.
- Agent and run correlation (`agent_context` computed export, `run_id` field-validator scrubbing) flows from settings into logging and wire envelopes — rich settings owners own tag shape; rail modules consume dumps, not re-derive tags.
- Service owners hold port-typed capabilities (`ArtifactFileSystem`), not vendor classes — richness on the service owner is path algebra and store operations, not fsspec import leakage to domain modules; port definitions mirror minimal third-party surfaces at the boundary module — optional capabilities fold into protocol default members instead of nominal mixin trees parallel to the port; conformance proof combines static assignability, `@runtime_checkable` smoke checks, and `get_protocol_members` drift tables — production backends register at composition roots, not inside rich settings modules; negative conformance cases assert `not isinstance(partial, Port)` when any required member is missing — happy-path isinstance alone does not prove port compatibility for service fabrication.
- Workers pass frozen snapshots and rebuild stores locally through fabrication methods inside the worker entrypoint — parent port handles are not inherited; port-typed fields on frozen dataclass service owners are not serialized across process seams.

# Fallible Envelopes And Construction Versus Runtime Law

- Parse and arity rails on param owners return `Self | Fault` — success preserves param identity, failure returns a canonical fault product from static methods on the same owner, not bare strings or ad hoc exception types.
- Param-owner `Self | Fault` and rail `@tagged_union` faults are parallel envelope families — morphism functions live at the composition-root registry boundary; param owners stay canonical for surplus and arity law; registry `_bound` handlers own total morphism from parse-time `Fault` structs to rail fault variants, not ad hoc string re-wraps inside each handler arm; rail modules consume tagged faults after morphism, not raw parse faults mixed with envelope cases; root-module import graph forbids rail arms from importing param-owner fault types directly — allowed edge is root morphism export consumed by both sides; cyclic envelope imports fail collapse tests.
- `TypeIs` predicates on morphism outputs prove the narrowed fault variant before `match` dispatch — bool helpers plus casts are policy-banned on enrichment boundaries; the envelope is not a second rich owner for the same concept.
- Parse faults clip diagnostic text at static constructors using struct metadata caps (`field_cap(Diagnostic, "hint", ...)`); morphism preserves clipped evidence; registry handlers do not re-truncate unless a downstream wire cap is stricter; surplus token messages respect egress limits without a separate truncation helper module; `expression` `Result`/`Fault` envelopes pair with param owners at parse boundaries and tagged unions at rail interiors.
- Settings use `raise ValueError` in validators for construction law (`exec_target` scheme, non-file backend root) and `@property` gates for runtime preconditions (`local_root` for non-file protocols) — construction rails and runtime projection rails are intentionally split; do not unify into `Result` when pydantic owns construction as `ValidationError`; the settings instance may exist in configurations where only remote execution is legal — `local_root` raising expresses subprocess-launch precondition, not construction defect.
- Context projections on materialized owners return plain types; fallible projections return `Result`/`Self | Fault` only for expected domain outcomes, not programmer errors; registry handlers `match` on `Self | Fault` without re-wrap from the owner's own `bound`; rich owners do not catch and re-wrap faults from their own `bound` — the fault type is already canonical; success arms thread param `Self` and immutable settings snapshots separately — do not embed settings inside param owners.
- `model_validate` on full snapshot always allocates new pydantic identity — `with_configuration` roundtrips compare field-wise equality, not `id()` stability; law matrices must use value equality or explicit field reads, not identity caches keyed on settings instances.
- `copy.replace` and `model_copy(update=...)` preserve shallow sharing of unchanged nested sub-objects — enrichment transitions that swap one nested axis must replace the entire child reference or re-validate the child snapshot; partial nested dict updates break typed nested models.
- Built-in `frozendict` allowlists compare/hash order-insensitively; enrichment transitions that rebuild allowlists must publish new `frozendict` instances, not mutate backing maps.
- `@computed_field` values are not stored fields — two settings instances with identical stored fields but different materialization order share equality on stored axes only; computed exports re-derive on access and serialization; identity transitions that re-run validators (`model_validate` on unchanged snapshot) still produce new instance identity — use only when ambient coercion policy drift requires replay, not as a no-op refresh primitive.

# Checker, Runtime Introspection, And Decorator Admission

- Pydantic models are `@dataclass_transform()` carriers; mypy pydantic plugin and ty treat synthesized `__init__`/`__replace__` — subclass overrides need `@override`.
- Ruff `runtime-evaluated-base-classes` includes `msgspec.Struct` and `pydantic.BaseModel` — `TYPE_CHECKING`-only field imports break runtime validation, schema compilation, and beartype on rich owner fields.
- Python ≥3.15 deferred annotations (PEP 649/749); pydantic, beartype, and cyclopts resolve field types at class-body or first-schema-compile time; introspect via `annotationlib.get_annotations(owner, format=annotationlib.Format.VALUE)` when building metadata-driven operations — direct `__annotations__` misses deferred materialization.
- Dataclass `fields()[n].type` may retain `ForwardRef` — resolve with `ForwardRef.evaluate()` at operation sites; PEP 695 generic owners defer type-parameter binding; use `types.get_original_bases()`, not private `__orig_bases__`, in catalog or registry walkers.
- Rich owner modules export types consumed by rails unconditionally; interior helpers that only serve validators stay private to the module — export stability is for types consumed by runtime-evaluated decorators and registry rows.
- Decorator stacks on owner methods must preserve PEP 695 `**P` shape under ruff `runtime-evaluated-decorators` (`beartype`, `pydantic.validate_call`, `pydantic.field_validator`).

# Integration Boundaries And Canonical Collapse

- One canonical class per concept — projections are `computed_field`, `model_dump` slices, or methods returning `Self`/views, never parallel `{Concept}Read/Update/Patch` when `copy.replace` or `model_validate` express the transition.
- Ingress pydantic; egress/evidence msgspec; CLI frozen dataclass — cross families with explicit `model_validate`/`msgspec.convert`, not implicit `.dict()`/`.model_dump()` handoff without re-validation policy.
- `frozendict` nested policy tables require hashable equality; shallow owner immutability plus mutable `dict` fields is doctrine violation.
- `expression` tagged unions own closed error/result variants in rail code — do not mirror the same variant set as a pydantic model hierarchy; pick union owner per bounded context.
- Context path projections prefer `os.path.realpath(..., strict=os.path.ALLOW_MISSING)` at the boundary — manual symlink-prefix resolution loops duplicate stdlib law already on the Python 3.15 surface; local enumeration inside service owners prefers `Path.walk()` over ad hoc `os.walk()` string stitching when the store root is local — backend-backed stores stay on structural port methods (`find`, `glob`) where stdlib walk does not apply; hot existence or type probes on local artifact paths use `Path.info` cache semantics instead of repeated bare `stat()` — context projections normalizing anchors must not stat child paths at settings materialization time.
- `UPath` remains the pathlib-compatible anchor type on settings owners — projection methods return backend path strings or `UPath` values consistently per method family; mixing untyped `str` roots on one owner breaks seam auditing.

# Proof Contracts And Side-Effect-Free Introspection

# Concurrency And Cross-Process Re-Materialization

- PEP 779 free-threaded builds: frozen instance owners share without locks after materialization — concurrent reads require no locks on instance fields; mutable class attributes (`classmethod` memo dicts, lazy registry caches) race under parallel import or task spawn — use `ContextVar` snapshots, import-published `frozendict`/`frozenset`, or explicit `threading.Lock`; class-scoped memo tables on rich owners must use import-published `frozendict`, explicit locks, or `ContextVar` lanes — unguarded class dict mutation under parallel import fails concurrency modules; `concurrent.interpreters` isolation applies to CPU-bound folds on owner-derived evidence — shared class-level memo tables on rich owners race under free-threaded builds unless published as immutable `frozendict` at import or guarded by explicit locks.
