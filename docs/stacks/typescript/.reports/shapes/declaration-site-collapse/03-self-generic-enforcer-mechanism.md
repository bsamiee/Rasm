# Self-generic enforcer mechanism

[ENFORCER_THESIS]:
- The duality collapse — one `class Self extends Factory<Self>(...) {}` carrying value, type, codec, equality, and constructor — is not a convention the author follows but a contract the factory's return type forces: the factory defaults `Self = never` and gates its real return behind `[Self] extends [never] ? Diagnostic : Owner`, so omitting the self-reference does not produce a looser owner, it produces a value whose static type is an unusable string subtype. The collapse is enforced from the return position, the one place a missing call-site generic is observable, and the enforcement is total because no `extends` clause accepts the diagnostic.
- The diagnostic is a template-literal type, a `string` subtype computed from the factory's own name and argument prefix, so the error text IS the corrected re-declaration: a value typed `` `Missing \`Self\` generic - use \`class Self extends Factory<Self>()({ ... })\`` `` cannot extend anything (`class X extends "..."` is not a class expression), and the only way to satisfy the heritage position is to write exactly the spelling the string spells. The enforcer turns the most common authoring mistake into a compile error whose message is the fix, never a runtime surprise.
- The `[Self] extends [never]` guard is the inference-direction probe specialized to one position: `Self` is supplied by the author at the call site (`Factory<Self>`), never inferred from an argument, so its sole un-supplied resolution is the `= never` default — the guard reads "did the author name the class as its own type argument" and answers in the return type, the only signal a non-inferred generic can carry.
- This is the chain-collapse thesis made non-degradable: the prior links of the naive chain (`type X = typeof x`, a second `const`, a trailing `new X()`) are deletable but their deletion is voluntary — nothing forbids re-growing them. The self-generic is the one link whose ABSENCE is forbidden: the fused owner cannot exist without it, so where the other levers make the collapse possible the enforcer makes the un-collapsed form impossible to compile.

[PARAMS_IS_A_CURRIED_SIGNATURE]:
- The `${Params}` slot is not a label but the family's curried-second-call signature spelled as a literal prefix, so the diagnostic reproduces the exact leading argument list the corrected form must pass: `` `...class Self extends ${Usage}<Self>()(${Params}{ ... })\`` `` resolves `Params = '"Tag", '` for the tagged owner and `Params = '"Tag", SuccessSchema, FailureSchema, '` for the request owner, the trailing `{ ... }` always the field block. The enforcer does not say "name yourself," it emits the precise call shape — the longer the family's fixed second-call prefix, the longer its `Params`, the message growing in lockstep with the factory's fixed-argument arity rather than authored per family.
- The empty-`Params` default `= ""` is the signal a family's second call takes only the field block: the schema owner reads `MissingSelfGeneric<"Class">` so its corrected form is `Class<Self>()({ ... })` with nothing before the braces — a non-empty `Params` is positive evidence the family fixes a discriminant or schema pair ahead of its fields, so the diagnostic's prefix length reads off the family's curried arity without consulting the factory signature.
- The inheritance verbs flow a DOTTED owner name into `${Usage}` so the wall locates the omission to one method call: `.extend` returns the template with `"Base.extend"`, `.transformOrFail`/`.transformOrFailFrom` with `"Base.transformOrFail"`/`"Base.transformOrFailFrom"`, so a tree several levels deep that fails to compile names the exact edge by the verb that produced it — the enforcer distinguishes a forgetting base from a forgetting child by the dot in the prefix, not just "somewhere a self-generic is missing."

```typescript
import { Schema } from 'effect'
class Owner extends Schema.Class<Owner>('Owner')({
    code: Schema.String,
    span: Schema.Number.pipe(Schema.positive()),
}) {
    get widened(): Owner {
        return Owner.make({ code: this.code, span: this.span * 2 })
    }
}
class Extended extends Owner.extend<Extended>('Extended')({
    weight: Schema.Number,

const admit = Schema.decodeUnknown(Extended)
type ExtendedWire = Schema.Schema.Encoded<typeof Extended>
```

[TEMPLATE_ARITY_DIVERGENCE]:
- The "one polymorphic enforcer" claim holds only within a package: three structurally-distinct template definitions exist across the owner families, differing in arity and in which slots are parameterized. The schema family's `MissingSelfGeneric<Usage, Params>` parameterizes both name and argument prefix; the variant-model family's `MissingSelfGeneric<Params extends string = "">` hardcodes the family name to `Class` and parameterizes only `Params`; the service family's `MissingSelfGeneric` takes NO parameters and inlines its whole text `` `Missing \`Self\` generic - use \`class Self extends Effect.Service<Self>()...\`` ``.
- The hardcoded-name template is the tell that an owner family has exactly one factory whose diagnostic never needs a `${Usage}` slot: the variant model spells `` `...class Self extends Class<Self>()(${Params}{ ... })\`` `` with `Class` fixed because the family has one entry point and the name is invariant — parameterizing what cannot vary is the surface the fixed template deletes, so a family's enforcer arity is a readable signature of how many factories it exposes.
- The parameterless service template and the parameterized schema template differ because their corrected spellings differ in shape, not just text: the schema correction is `Class<Self>()({ ... })` (curried, fields in the second call), the service correction is `Effect.Service<Self>()(key, make)` (curried, key-and-make in the second call), so the service template inlines a complete sentence rather than composing one from slots — the enforcer's structure mirrors the factory's call shape, a per-family fact no shared parameterization could carry uniformly.

[GUARD_POSITION_SPLIT]:
- The `[Self] extends [never]` guard sits at different curry depths across families, and the depth decides WHERE the diagnostic surfaces. The schema family curries `(<Self>(id) => (fields) => [Self] extends [never] ? Diagnostic : Class<...>)`: the guard wraps the INNERMOST return, so forgetting `<Self>` types the value of the fully-applied `Class<Self>('id')({ ... })` as the diagnostic and the error lands on the heritage expression. The service family curries `(<Self>() => [Self] extends [never] ? Diagnostic : { <overloads>(key, make): Class })`: the guard wraps the SECOND function — the object carrying the call overloads — so forgetting `<Self>` types `Effect.Service<Self>()` itself as a `string`, and the subsequent `(key, make)` application fails as "this expression is not callable" on a `string`, a node earlier than the heritage clause.
- The split is the reason the two diagnostics read differently at the call site: the schema author sees the template-literal text as the heritage type, the service author sees a not-callable error on the inner application BEFORE the heritage resolves, because the missing generic collapsed the curried factory to a string one call up. The enforcer's guarantee — non-degradable collapse — holds for both, but the surfaced diagnostic and its source node are a function of the guard's curry depth, so reading the error requires knowing which family's guard wraps which call.
- The variant-model family inherits the schema-style inner-return guard `(<Self>(id) => (fields) => [Self] extends [never] ? Diagnostic : VariantClass<...>)`, so its diagnostic surfaces at the heritage clause like the schema family despite its template arity matching neither — guard position and template arity are independent axes, and a family is fully specified only by naming both.

[TAG_PARAMS_LINKAGE]:
- The enforcer's `'"Tag", '` prefix points at an argument that is structurally a declaration-time default, never a per-construction literal: the `_tag` the tagged owner injects is a `tag<Tag>` property signature pinned `PropertySignature<':', Tag, never, ':', Tag, true, never>` whose `HasDefault = true` bit drops it from the constructor's required set, so the corrected spelling the diagnostic emits passes the tag in the SECOND call (where the family fixes it once) and never in the construction argument (where the owner auto-supplies it). The self-generic enforcement and the tag-injection collapse meet on one owner — the prefix names the discriminant's single entry point.
- The two collapses share a producer consequence: a `TaggedClass` sets its constructor argument to `Struct.Constructor<Omit<Fields, '_tag'>>` so a hand-passed `_tag` at construction is forbidden, while the enforcer's prefix shows the tag's only legitimate site is the curried second call — so the owner that forgets `<Self>` and the producer that re-passes `_tag` are walled by the same declaration's two mechanisms, the discriminant flowing through exactly one position the type fixes at both layers.

[TUPLE_WRAP_IS_LOAD_BEARING]:
- The guard wraps both sides in single-element tuples — `[Self] extends [never]`, not `Self extends never` — and the wrap is the mechanism, not cosmetic: a naked `Self extends never` is a DISTRIBUTIVE conditional that maps over a union member-wise and, on the `never` default, distributes over the empty union to yield `never` outright, never reaching the diagnostic branch. The tuple-wrap suppresses distribution so `never` is compared as a whole and hits the true branch, producing the diagnostic exactly when `Self` resolved to the `= never` default — the one resolution an un-supplied, never-inferred type parameter can take.
- The same tuple-wrap fixes the boundary of the guarantee: supplying `any` for `Self` makes `[any] extends [never]` resolve to the BOTH-branches union, so a deliberate `Factory<any>(...)` types the return as `Diagnostic | Owner` and the wall partially dissolves — the enforcer is total against the `never` default (forgetting) but not against an author who explicitly widens the self-reference to `any`. The collapse is enforced against omission, the one failure mode that occurs by accident, and leaves the deliberate `any`-escape open because no type-level guard can distinguish a mistake from a sanctioned override.
- The wall's teeth are `class X extends "string-literal"` being syntactically ill-formed, not merely type-mismatched: a heritage clause requires a construct-signature-bearing expression, and a `string` subtype has none, so the diagnostic type cannot satisfy the position under any coercion — the enforcer does not emit a warning the author may suppress, it makes the surrounding `class` declaration itself unparseable as a class, the failure mode no discipline-based convention can match for immediacy or totality.

[ANTI_RESTATEMENT_LAW]:
- The self-generic forbids the one chain link the other levers merely discourage: a forward-declared `type Self = { ... }` written above the factory to feed its own `Self` argument is the broken-cycle restatement — the class IS its own type the instant `<Self>` threads the name, so the standalone alias is a second statement naming what the heritage already names, and worse it cannot reference the class's own getters and methods because they are not yet declared when the alias is read. The enforcer's correct spelling `Factory<Self>` consumes the class name directly, so the only conformant form is the one with no separate type source.
- The threaded `Self` decides what `this`, the constructor return, and `.make`'s `InstanceType` are typed against but carries no runtime artifact of its own: codec, `Equal`/`Hash`, and the validating constructor come from the factory machinery, while `Self` only pins which type those surfaces wear — so a forgotten `<Self>` does not break the runtime owner (it would build identical values) but breaks its type, which is why the enforcer must live in the type system and cannot be a runtime check; there is nothing to check at runtime, only a type to refuse.
- The self-reference closes without a value-level cycle because `<Self>` is consumed at the type level while the heritage expression evaluates eagerly at the declaration line: the `Self` binding is hoisted but TDZ-protected, and the factory reads the NAME (a type) not the BINDING (a value), so the class names itself in its own heritage without a circular read — a field that named the class at the VALUE level would throw TDZ, the precise reason the self-reference is a type generic and never a value-side back-reference.

[INHERITANCE_RE_ENFORCEMENT]:
- Every inheritance verb re-runs the enforcer on the child with a fresh `[Child] extends [never]` guard, so the collapse is non-degradable across the whole inheritance tree, not just at the root: `.extend<Extended>` threads `Extended` and walls a forgetting child with `MissingSelfGeneric<"Base.extend">`, `.transformOrFail<Transformed>` and `.transformOrFailFrom<Transformed>` do the same — a subclass cannot quietly inherit the parent's `Self` and skip naming itself, because the child's instance type, its own getters, and its `.make` return all depend on the child's `Self`, and the enforcer refuses the child declaration that omits it.
- The parent's `Self` does NOT satisfy the child's guard: it migrates into the child's `Inherited` slot — `Class<Extended, Fields & NewFields, ..., Self, Proto>` carries the parent `Self` as the sixth parameter so the child instance type is `Struct.Type<Fields & NewFields> & ParentSelf & Proto` — but the child's own first parameter must be the child's name or the guard fires. The two slots are orthogonal: `Inherited` threads the parent's typed surface (its methods and getters) through the edge, the child's `Self` pins the child's own this-type, and the enforcer guards only the latter.
- The dotted `${Usage}` is the diagnostic's diagnostic: `"Base.extend"` versus `"Base.transformOrFail"` in the error text tells the author which inheritance verb's child forgot its generic, so a tree several levels deep that fails to compile names the exact edge by the verb that produced it — the enforcer is parameterized finely enough that the error locates the omission to one method call, not just "somewhere a self-generic is missing."
