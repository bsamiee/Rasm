# Composition Law

[BOUNDARY_CONVERSION]:
- `Optional<A>(nullable)` and `Optional<A>(Nullable<A>)` are the canonical null-to-`Option<T>` projections; both are on `Prelude`. `Option<A>` carries an implicit operator from `A?` so direct assignment projects null to `None` with no call needed for non-struct types, but the `Optional` overload is load-bearing for value types where `A?` means `Nullable<A>`, not `A`.
- `Option<A>.ToFin()` calls `ToFin(Errors.None)` — the absence case maps to `Errors.None`, not a meaningful error; to preserve diagnostic intent, pass a domain `Error` via `ToFin(Error fail)` at the boundary rather than post-hoc at the consuming edge.
- `Validation<Error, A>.ToFin()` is a direct case-projection: `Succ(value)` becomes `Fin.Succ(value)`, `Fail(e)` becomes `Fin.Fail(e)`. The `Error` produced by Validation's applicative error accumulation is a `ManyErrors` whose `Combine` walks the error list via `Semigroup<Error>.Combine`; `ToFin` transfers the whole accumulated `Error` as a single `Fin.Fail` — no partial extraction, no further enumeration needed at the call site.
- `Either<Error, A>.ToFin()` is on `FinExtensions`: `Right` maps to `Fin.Succ`, `Left` maps to `Fin.Fail`. The conversion is one-hop with no allocation beyond the Fin record case.
- `Try<A>` is a `Func<Fin<A>>` wrapper whose `Run()` executes the delegate inside a catch block and returns `Fin<A>`; exception types route through `Error.New(Exception)`. `TaskCanceledException` and `OperationCanceledException` produce `Errors.Cancelled`; `TimeoutException` produces `Errors.TimedOut`; `AggregateException` produces `ManyErrors.FromAggregate`; `ErrorException` round-trips through `.ToError()`; all other exceptions produce `Exceptional`. This means `Try<A>` is the canonical synchronous exception-capture boundary; `liftEff<A>(Func<Fin<A>> f)` wraps the same pattern into `Eff<A>` directly.
- `Error.New(Exception)` follows the same routing as `Try.Run()`: `WrappedErrorExceptionalException` / `WrappedErrorExpectedException` round-trip their inner `Error`, `ErrorException` restores via `.ToError()`, cancellation and timeout map to the static `Errors` sentinels. Calling this at the boundary guarantees that re-captured exceptions never lose their original `Error` payload.
- `Fin<A>` has implicit operators from `Fin<A>`, `IO<A>`, `Either<Error,A>`, `Pure<A>`, `Fail<Error>`, `Lift<Fin<A>>`, and bare `Error` into `Eff<A>`. `Fin` implements `Natural<Fin, Eff>`: `Transform` wraps `Succ` in `Eff<A>.Pure` and `Fail` in `Eff<A>.Fail`, so `Natural.transform<Fin, Eff, A>(fin)` is the explicit carrier-promotion path when the implicit operator is ambiguous.

[TERMINAL_COLLAPSE]:
- `Match<B>(Func<A,B> Succ, Func<Error,B> Fail)` on `Fin<A>` is the only bi-directional terminal; it exits the rail entirely. Every other operator that branches on state (`IfFail`, `IfSucc`, `MapFail`) stays inside the rail and propagates a `Fin<A>` or `Eff<A>` — not a raw value. Using `Match` mid-pipeline discards the rail and forces reconstruction; the CSP rule CSP0705 (Match mid-pipeline forbidden) encodes this.
- `Eff<A>.Run()` returns `Fin<A>` and never throws; `Eff<A>.RunUnsafe()` delegates to `IO<A>.Run()` which throws `ErrorException` on failure. The `Run` / `RunUnsafe` axis is the consuming-edge terminal dichotomy: internal code uses `Run`, exception-required interop uses `RunUnsafe`. `RunAsync` returns `Task<Fin<A>>`; `RunUnsafeAsync` returns `ValueTask<A>`.
- `IO<A>.RunSafe()` returns `Fin<A>` without throwing; any exception thrown inside the IO computation is captured and returned as `Fin.Fail(Error.New(ex))`. This is the non-throwing exit point for `IO<A>` when it must be unwrapped without an `Eff` envelope.
- `FinGuardExtensions.ToFin(Guard<Error,Unit>)` converts a `guard(flag, error)` into `Fin<Unit>.Succ` or `Fin<Unit>.Fail(error)`. `Guard<E,Unit>.Bind<B>(Func<Unit,Fin<B>> f)` provides direct monadic continuation, making `guard` composable inside `from`/`select` query expressions without introducing a terminal collapse mid-chain.

[CATCH_COMBINATOR_COMPOSITION]:
- `CatchM<E, M, A>` is a record pairing a predicate `Func<E,bool>` with a recovery `Func<E,K<M,A>>`. The `|` operator on `K<F,A>` where `F : Fallible<E,F>` is defined via `FallibleExtensions` C# 14 extension block: `lhs | rhs` calls `lhs.Catch(rhs)`, which delegates to `F.Catch` with the predicate and recovery from `rhs`. Chaining multiple `|` handlers produces left-to-right predicate testing; the first matching `CatchM` wins and subsequent handlers are not evaluated.
- `Prelude.@catch<M,A>(Error, K<M,A>)` matches via `error.Is` — deep structural equality on the `Error` record, not reference equality. `@catch<M,A>(int errorCode, ...)` matches via `e.Code == errorCode || e.HasCode(errorCode)`, traversing `ManyErrors` children. This means a `ManyErrors` error containing one child matching the code triggers the handler even though the outer type doesn't match.
- `Prelude.catchOf<E,M,A>(Func<E,K<M,A>>)` matches errors that are `E` or that contain a child `E` via `e.IsType<E>()`. For `ManyErrors` inputs, matching children are iterated and sequenced via `MonoidK`; use `catchOfFold` when `M : MonoidK<M>` and partial failure should produce an empty rather than a composed `K<M,A>`.
- `exceptional<M,A>(Func<Exceptional,K<M,A>>)` and `expected<M,A>(Func<Expected,K<M,A>>)` are the typed `Error` subtype matchers. Both delegate to `catchOf<Exceptional,...>` / `catchOf<Expected,...>`. The distinction is load-bearing at retry logic: `Exceptional` errors (unplanned, host exceptions) should typically not be retried; `Expected` errors (domain, planned) are retry candidates.
- `Fallible<E,F>.Catch` is the abstract static method that every carrier implementing `Fallible` must provide. The instance-level `.Catch(Func<Error,bool>, Func<Error,K<F,A>>)` extension on `FallibleExtensions` delegates directly to `F.Catch`, meaning the carrier provides the actual recovery semantics. For `Fin`, recovery produces a new `Fin<A>`; for `Eff`, recovery produces a new `Eff<A>` without touching the outer monad stack.

[NO_DOUBLE_VALIDATION_SEAM]:
- The seam law: the generated owner's factory or validation partial produces `Validation<Error,A>` or `Fin<A>` exactly once. Internal code calls `.ToFin()` at the transition point and operates on `Fin<A>` inward. No code downstream of the boundary calls `.IsValid`, re-checks fields, or re-runs admission predicates. The `Validation<Error,A>` applicative accumulates all field errors via `Semigroup<Error>.Combine` before the seam; `.ToFin()` transfers the entire `ManyErrors` as a single `Fin.Fail`.
- `Validation<FAIL>` Bind short-circuits (Monad instance returns `Fail(value)` immediately on failure); Apply accumulates (Applicative instance calls `trait.Combine(Value4, Value5)` when both sides are `Fail`). The admission phase uses Apply — building the success value via `*` applicative chains — precisely to collect all field errors. Switching to `Bind` inside the admission chain silently disables accumulation.
- `guard<E>(bool flag, Func<E> False)` produces `Guard<E,Unit>` which binds into query expressions via `SelectMany`. Inside `Fin` pipelines, `FinGuardExtensions.Bind` replaces the pattern `if (!condition) return Fail(...); return f(unit)`. This is the mechanism for inline domain invariant checks after admission without creating a new validation layer.

[CARRIER_POLYMORPHIC_ALGORITHMS]:
- `K<F,A>` is a phantom-type witness interface with no members, enabling HKT simulation. Every carrier implementing `Monad<M>` also implements `K<M,A>` for its bound type, which means algorithms parameterized over `K<M,A> where M : Monad<M>` work uniformly over `Fin`, `Eff`, `Option`, `IO`, and monad-transformer stacks.
- `Traversable.traverse<T,F,A,B>(Func<A,K<F,B>> f, K<T,A> ta)` and `traverseM<T,M,A,B>(...)` are the generic traversal entry points. The distinction: `Traverse` requires `F : Applicative<F>` and uses applicative `Apply` semantics (for `Validation<FAIL>` this means error accumulation); `TraverseM` requires `M : Monad<M>` and uses monadic `Bind` semantics (for `Fin` this means short-circuit on first failure). Choosing the wrong overload changes error-collection behavior for the same carrier.
- `Seq<A>.Traverse<F,B>(Func<A,K<F,B>>)` and `Seq<A>.TraverseM<M,B>(...)` are the concrete entry points on sequences. The `TraverseM` default implementation delegates to `Traversable.traverse(f, ta)` — the same applicative path — meaning override by concrete types matters; `Fin` traversal on `Seq` uses the carrier's own `Apply`, which for `Fin` is short-circuit, not accumulating. Only `Validation<FAIL>` accumulates across a sequence.
- `MonadIO.liftIO<M,A>(IO<A> ma)` where `M : Maybe.MonadIO<M>, Monad<M>` is the generic IO injection for any monad in the `MonadIO` hierarchy, including monad-transformer stacks. This is the mechanism for sharing `IO`-bound operations across different `Eff<RT>` flavors without carrier-specific plumbing.
- `Natural<Fin, Eff>`, `Natural<Fin, IO>`, `Natural<Fin, Option>`, `Natural<Fin, Try>`, `Natural<Fin, Either<Error>>` are all implemented on `Fin`. `Natural.transform<F,G,A>(K<F,A>)` performs the structural promotion with zero domain logic. An algorithm returning `K<Fin,A>` can be promoted to any of these targets at the instantiation site via `transform` or via the implicit operator, keeping the algorithm itself carrier-agnostic.

[EXCEPTION_CAPTURE_AND_CONTEXT]:
- `Error.New(string message, Error inner)` and `Error.New(int code, string message, Error inner)` produce `Expected` records that form a chain via `Error.Inner : Option<Error>`. Wrapping boundary failures with `MapFail(e => Error.New("layer context", e))` preserves the original error as `Inner` while adding layer context to `Message`. Walking `.Inner` reconstructs the full causal chain without any custom error types.
- `Eff<A>.MapFail(Func<Error,Error> f)` is implemented as `this.Catch(f).As()` — it creates a catch handler that transforms every failure, preserving the `Eff` envelope. This is the correct composition-time error-context addition; it attaches as a transformer to the pipeline, not as a terminal that unwraps and re-wraps.
- `Error.Combine(Error rhs)` and the `+` operator produce `ManyErrors` by merging. `Error.Many(params Error[])` and `Error.Many(Seq<Error>)` are the batching constructors. An error context that accumulates multiple sub-errors — e.g., parallel validation passes — uses `+` or `Error.Many` before entering the rail.
- `Exceptional.Inner` traverses to `Error.New(Value.InnerException)` when the wrapped `Exception` has an inner exception, preserving the full exception chain as nested `Error` values. `Expected.Inner` carries an explicit `Option<Error>` field set at construction. Neither requires unwrapping the error to access the chain.

[SCHEDULE_COMPOSITION]:
- `Schedule.recurs(n) | Schedule.exponential(seed)` produces a schedule that fires `n` times with exponential delay. `|` implements union (min of the two durations, length of the longest); `&` implements intersection (max of durations, length of the shortest). Both operators are on the `Schedule` record itself. The result is a `Schedule` value composable with `RetryIO`, `RepeatIO`, etc.
- `MonadUnliftIO<M>.RetryIO<A>(K<M,A> ma, Schedule s)` is implemented as `MapIO(ma, io => io.Retry(s))` — it delegates to `IO<A>.Retry(Schedule)` internally. The same pattern holds for `RepeatIO`, `RepeatWhileIO`, `RepeatUntilIO`, `RetryWhileIO`, `RetryUntilIO`. Any `M : MonadUnliftIO<M>` participates in schedule-driven retry without carrier-specific overrides.
- `Schedule.PrependZero` (prepend a zero-duration delay at the front) ensures the first execution runs immediately; it is used internally by `IO.Retry` and `IO.Repeat`. The implication: `retry(Schedule.recurs(3))` on a failing `Eff<A>` attempts 4 total runs (1 initial + 3 retries), not 3. `PrependZero` is documented on the `IO` schedule-driven loop.

[BRACKET_AND_RESOURCE_LIFETIME]:
- `IO<A>.Bracket()` creates a local resource environment via `WithEnv(env => env.LocalResources)`. Resources allocated inside the bracketed computation are released when it completes. `BracketFail()` releases only on failure. `IO<C> Bracket<B,C>(Func<A,IO<C>> Use, Func<A,IO<B>> Fin)` acquires `A`, uses it to produce `C`, and runs `Fin(a)` regardless of outcome — the bracket pattern. The three-argument overload adds `Func<Error,IO<C>> Catch` between Use and Fin.
- `IO.Bracket` and `IO.Finally` are the correct resource-management primitives; `IDisposable` disposal inside an `Eff` pipeline must route through these, not through a `using` statement wrapping the pipeline. A `using` inside `liftEff(Func<A> f)` works for synchronous cases but silently fails to dispose if the `Eff` is cancelled after the lambda returns and before the caller reads the value.
- `STM.RunTransaction<R>(Eff<R> op, Isolation)` executes an `Eff<R>` inside an MVCC transaction. `Ref<T>` mutations inside the transaction are snapshot-isolated; conflicting writes trigger automatic retry. `STM.RunTransaction<R>(Func<R> op, Isolation)` is the synchronous variant. Both are the public surface; `DoTransaction` is internal. `Isolation.Snapshot` vs `Isolation.Serialisable` selects the MVCC isolation level.

[ANTI_PATTERNS]:
- Throwing inside a `from`/`select` query expression over `Fin`, `Option`, or `Eff` does not capture the exception into the rail; it propagates to the caller. `Error.New(ex)` wrapped in `Fin.Fail` or `Eff.Fail` is the correct form. `liftEff(Func<A> f)` captures synchronous throws; `liftEff(Func<Task<A>> f)` captures async throws. Any raw `throw` inside domain logic is a violation.
- `Fin<A>.Match(Succ, Fail)` at a mid-pipeline `Bind` step — i.e., `prev.Bind(_ => curr.Match(Succ: x => Fin.Succ(f(x)), Fail: e => Fin.Fail(e)))` — is identity-equivalent to `curr.Map(f)` but destroys type inference, creates unnecessary closure allocation, and violates CSP0705. The correct rail-preserving forms are `Map`, `Bind`, `MapFail`, and `BiMap`.
- Re-calling the generated owner's factory or running a validation partial inside a `Bind` step to perform a secondary check is the double-validation anti-pattern. The `Fin<A>` produced by the first admission already carries the invariant; all downstream operations bind against the admitted value, not against raw field values.
- Calling `Validation<Error,A>.ToFin()` and then immediately calling `.ToValidation()` or re-wrapping into `Validation` downstream re-enters the admission phase. `Fin` and `Validation<Error>` share the same `Error` type and have round-trip conversions, but the round-trip is only meaningful at the admission seam; mid-pipeline conversions indicate a missing `Applicative` composition step.
- Ambient state accessed inside `Bind` or `Map` closures (static fields, thread-static, `HttpContext.Current`-style accessors) breaks referential transparency and makes the `Eff` pipeline non-testable. Boundary state belongs in `Atom<T>` (lock-free CAS), `Ref<T>` (STM), or the `Eff<RT>` reader environment, not in ambient singletons accessed inside lambda captures.
- Per-carrier pipeline duplication — writing one pipeline for `Fin`, another for `Eff`, another for `Option` sharing identical structure — is eliminated by parameterizing over `K<M,A> where M : Monad<M>, Fallible<Error,M>`. The algorithm body uses `Monad.bind`, `Fallible.fail`, `M.Pure`, and `Natural.transform` at the carrier-specific edges; only the carrier-polymorphic algorithm is written once.
