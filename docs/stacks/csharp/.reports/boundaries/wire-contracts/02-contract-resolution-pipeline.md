# Contract Resolution Pipeline

[RESOLVER_FOLD_IS_FIRST_NON_NULL_OVER_A_FLATTENED_LIST]:
- Combining several resolvers does not build a tree of resolvers; it folds them into one flat list and queries left-to-right, returning the first non-null metadata. Each combine step splices any nested chain operand inline rather than nesting it, and drops the empty resolver entirely, so the runtime never walks a resolver-of-resolvers — the list is always one level deep regardless of how the combinators were stacked at the call site.
- The flatten also collapses the degenerate cases: combining a single effective resolver yields that resolver unwrapped, not a one-element chain, and combining nothing yields the shared empty resolver. The consequence for ordering law is that precedence is purely positional in the final flat list — a generated context placed ahead of a reflective fallback wins for every type it covers, and the fallback is consulted only for the types the generated context returns null for. There is no per-type priority knob; position is the entire policy.
- A modifier-wrapping resolver is not a chain, so flattening leaves it atomic: it applies its modifier sequence to whatever its single inner source returns, once, and only when that source returns non-null. Wrapping a chain in a modifier therefore post-processes the chain's winning result exactly once, never once per chain element — the modifier sees the resolved metadata, not each candidate.

[MODIFIER_FUSION_IS_ASSOCIATIVE_AND_FLAT]:
- Adding a modifier to a resolver that is already a modifier-wrapper does not nest a second wrapper; it copies the existing modifier array, appends, and returns one wrapper holding the combined sequence. Stacking N modifier additions yields one wrapper with an N-element array invoked in addition order, not N wrappers each invoking one. The fold is associative: the grouping of the additions is invisible, only the final left-to-right sequence survives.
- The wrapper runs every modifier unconditionally after a non-null resolve, with no guard between them, so a modifier that mutates a property the next modifier reads observes the prior mutation — the sequence is an ordered transform pipeline over one mutable metadata instance, not a set of independent advice. Order is therefore load-bearing policy: a rename modifier ahead of a redaction predicate keyed on the new name composes; the reverse silently misses.

[STRUCTURAL_CACHE_KEY_GOVERNS_INSTANCE_DISCIPLINE]:
- Per-type contract metadata is cached against a structural fingerprint of the owning options, not the options reference. The fingerprint hashes every scalar serialization setting plus, by reference identity, the resolver and each registered converter; two distinct options instances that agree on all scalars and hold the same resolver and converter references share one metadata cache. The reference-identity inclusion of the resolver and converters is the trap: equal scalar settings with a freshly constructed resolver instance produce a distinct fingerprint and a cold cache, so building the resolver inline per request silently defeats sharing even when nothing semantically differs.
- For unsealed participants — the resolver and converters — the fingerprint uses runtime reference-hash, not value equality, so a converter that overrides equality cannot opt two instances into the same bucket; only the same instance shares. The discipline that follows is singular: construct the resolver and converter instances once as statics and reuse the references across every options instance that should share metadata. A per-request options that points at the shared static resolver and converter list still lands on the shared cache; a per-request options that news up its own resolver does not.
- The converter list participates positionally — order and reference identity of each entry are part of the key — so reordering the same converters yields a distinct fingerprint and a cold cache even though resolution behavior is unchanged. Treat the converter list as an ordered, identity-stable contract coordinate, not a bag.

[CROSS_INSTANCE_SHARING_HAS_A_HARD_CEILING]:
- The shared caches are tracked in a fixed-capacity weak-reference table; once that many structurally-distinct live options shapes are in flight, a new distinct shape is handed an untracked cache that is private to its own instance and never shared, with no error and no fallback to value-equality matching. A system that mints many genuinely-distinct options shapes — per-tenant encoders, per-route naming policies, per-call resolver variants — silently crosses from shared to per-instance metadata once the live distinct-shape count exceeds the table, and pays full contract-build cost per instance thereafter.
- The table holds weak references, so a shape's slot frees only after its options instance is collected; long-lived distinct options pin their slots for their lifetime. The design pressure is to keep the count of long-lived distinct options shapes well under the ceiling and to route per-request variance through fields on a value object the converter reads, never through a new options shape per request.

[EXCEPTION_AS_CACHE_ENTRY_MAKES_RESOLUTION_FAULTS_STICKY]:
- The per-type cache stores either the metadata or a captured exception; a throw during contract resolution for a type is captured and re-thrown, preserving its original stack, on every subsequent metadata request for that type against the same cache. A resolver or modifier that throws for one type does not fail once and recover — it poisons that type's slot for the lifetime of the shared cache, and because the cache is shared structurally, the poison is visible from every options instance with the same fingerprint.
- The capture happens inside the cache-fill delegate, so the fault surfaces at first resolution of that type, which for a root value is the first serialize or deserialize call, not at options construction or seal. A modifier that assigns a lifecycle delegate or polymorphism options to a metadata instance of the wrong kind throws here, and the failure site is distant from the modifier registration. The only reset is a fresh cache: a new structurally-distinct options, or an explicit cache clear, never a retry against the same instance.
- Because the captured throw rethrows with original stack via dispatch-info, the rail bridge at the seam must translate it once at the call site and must not assume a transient: a resolution fault that retries against the same options is non-transient by construction and the retry predicate must classify it as expected-non-retriable, distinct from a network exceptional error.

[ROOT_TYPE_FAST_PATH_IS_A_SINGLE_VOLATILE_SLOT]:
- Above the per-type dictionary sits a one-entry volatile cache holding the metadata for the most recent root type. A serialize or deserialize of the same root type as the previous call skips the dictionary lookup entirely; a call with a different root type displaces the slot. Alternating between two root types at a hot seam thrashes this slot to zero benefit and falls through to the dictionary every call, whereas a seam that serializes one envelope type in a tight loop rides the slot.
- The slot keys on the metadata's reported type, not the requested type, which matters for polymorphic roots: a root resolved through nearest-ancestor fallback caches under the requested runtime type, and the polymorphic ancestor metadata is reached through a separate field on that entry, so the fast-path slot and the polymorphic dispatch are distinct caches consulted in sequence.

[NEAREST_ANCESTOR_RESOLUTION_AND_AMBIGUITY_FAULT]:
- A runtime type with no direct contract — a derived instance serialized through a base-typed root — is resolved by walking base types upward to the first that has metadata, then scanning interfaces for additional contracted ancestors. When a contracted interface and the chosen base type are mutually unassignable, resolution throws an ambiguous-metadata fault rather than picking arbitrarily; this is the failure surface for a wire root whose runtime type implements two unrelated contracted interfaces with no single most-derived contract.
- The ancestor result is memoized on the type's own cache entry behind a resolved flag, so the walk runs once per type per cache and the ambiguity fault, once captured, is sticky exactly like any other resolution throw. The design law is to give the closed wire family one unambiguous contracted root — a sealed base or a single marker the converter keys on — so the ancestor walk terminates at one metadata and never reaches the interface-ambiguity branch.

[FACTORY_RESOLUTION_FUNNELS_THROUGH_THE_TYPEINFO_CONVERTER_SLOT]:
- A converter is never reached directly; it is the converter slot of the resolved metadata, so every converter selection — including factory expansion — passes through the same structurally-keyed metadata cache. A factory's synthesized per-type converter is therefore cached at the metadata layer keyed by resolved type, not re-synthesized per message, and a factory's contract is enforced at expansion time: returning null or returning another factory throws immediately, and that throw is captured as the type's sticky cache entry.
- Because factory expansion rides the metadata cache, a factory that closes over policy state must hold that state by stable reference for the structural fingerprint to keep two options on the same cache; a factory whose closure varies per options instance forces a distinct converter reference, a distinct fingerprint, and a cold cache. Policy that varies per call belongs in a value the synthesized converter reads at write time, not in a freshly-closured factory per options.

[SEAL_IS_THE_PIPELINE_BARRIER]:
- Every options scalar setter and the converter and resolver-chain collections gate on a mutability check that throws once the instance is sealed, and the instance auto-seals on first use. The resolver chain exposed from a sealed options is read-only and its modify hook throws; the modifier and resolver composition must therefore be complete before first serialize. There is no post-seal patch path for the contract pipeline — the only post-seal variance channel is per-write predicate state the converters already hold, evaluated live, never a new modifier.
- The chain bound to an options rejects adding the options' own resolver or itself as a chain element, guarding the self-referential cycle that would otherwise recurse during resolution. Composition that wants the options' current resolver inside a new chain must capture it before assigning the new chain, because assigning the chain replaces the resolver field it would have read.
