# [DISPATCH_COMPOSITION_SEAMS_AND_BOUNDARY_MECHANICS]

[TAGGED_UNION_STRUCTURAL_MECHANICS]:
- `@tagged_union` rewrites `__init__` to accept exactly one keyword argument naming the active case; two kwargs raise `TypeError: One and only one case can be specified`, an unknown kwarg raises `TypeError: Unknown case name`. Both fire at construction, never at match — a malformed variant cannot reach the dispatch site.
- `__match_args__` becomes the full field tuple in definition order — `('tag', case1, case2, ...)` — so positional class patterns bind `tag` at index 0: `Event('click', click=x)` and `Event(tag='click', click=x)` both work, the latter bypassing `__match_args__`. Pin the tag in every arm; positional-only arms silently match the wrong case when tags share a value shape.
- Inactive cases are never assigned as instance attributes, and per-instance `__dataclass_fields__` is pruned to exactly `{'tag', active_case}`. Accessing `variant.inactive_case` raises `AttributeError`, not a sentinel; the only safe read is a `match` arm or `getattr(variant, variant.tag)`. The pruning makes `dataclasses.asdict` emit only the active case, which is already the correct wire shape — no custom encoder needed for tagged-union egress.
- `frozen=True` synthesizes `__hash__ = hash((cls.__name__, tag, active_value))` and blocks field mutation; `order=True` synthesizes `__lt__` over `(_index, active_value)` where `_index` is the field position (tag at 0, first case at 1). This makes variants directly sortable and usable as dict keys, but `order=True` raises `TypeError` at comparison when two cases hold incomparable value types — it is safe only when all case payloads share an ordering or comparisons stay within one case.
- Pickle and Pydantic `model_copy` both round-trip: `__getstate__`/`__setstate__` reconstruct via `__init__`, and injected `__copy__`/`__deepcopy__` rebuild via `cls(**{tag: deepcopy(active_value)})`. A frozen union survives unpickling because reconstruction routes through `__init__`'s `object.__setattr__`, not the frozen `__setattr__` guard.

[EFFECT_RESULT_COROUTINE_MECHANICS]:
- `effect.result` is the `ResultBuilder` class (carrying `ABCMeta`), not an instance; `@effect.result[T, E]()` subscripts for the checker then instantiates, and the builder wraps the decorated generator. The short-circuit channel is `Result.__iter__`: `Ok(v)` runs `return (yield v)` to hand the value to the builder and receive the echoed send-back; `Error(e)` raises `EffectError(self)` carrying the whole `Result`. The builder catches `EffectError`, marks done, and returns `error.args[0]` unchanged — the exact `Error` instance, error value intact, propagates out.
- Multiple `yield from` statements compose through `combine = xs.bind(lambda _: ys)`: each statement binds to the next, discarding the left value, with every result after the first delayed. Execution is eager up to the first `Error`, after which all later statements are unreachable. Nesting composes the same way — `yield from inner()` returning `Error(e)` re-raises `EffectError(Error(e))` through the outer builder, short-circuiting with the identical error; depth is O(calls), not O(generator stack), since each block is a fresh builder invocation.
- A generator that returns `None` (implicit or explicit) raises `StopIteration` with no value, invoking `zero()`, which raises `NotImplementedError`. There is no silent `Ok(None)`: every `@effect.result` body must `return` an explicit value, making a missing return a hard contract failure rather than a coerced success.
- `@effect.result` is sync-only: `yield from` is a `SyntaxError` inside `async def`, and no async builder exists. Fallible async composition is therefore explicit — `await step` then `match`/`bind` on the result — because `result.bind` cannot await its function. The async monadic rail is structurally absent, not merely unconfigured.

[ROP_COMPOSITION_COMBINATORS]:
- `result.pipeline(*fns)` is Kleisli composition (`>=>`), not value piping: `pipeline(f, g)(x)` equals `f(x).bind(g)`, short-circuiting on the first `Error`; `pipeline()` is the identity returning `Ok`, so `pipeline()(x)` is `Ok(x)`. `option.pipeline` is the identical construct over `Option`, short-circuiting on `Nothing` and discarding the absence reason. The two are structurally identical and never interchangeable — `Result` carries an error payload, `Option` carries none. Use these for any chain of rail-returning steps sharing one error type `E`.
- A pipeline fixes `E` across all steps. Crossing subsystem boundaries with distinct error vocabularies requires explicit `map_error` to a common error union before each step — there is no error-widening pipeline combinator. This is the binding constraint when composing dispatch surfaces that own different failure rails.
- `compose(*fns)` chains plain functions left-to-right with no rail threading; feeding it `Result`-returning steps yields `Result[Result[T, E], E]` nesting. Reserve `compose` for pure transforms and `pipeline` for monadic steps. `starcompose(*fns)` pipelines multi-argument stages: each stage returns a tuple unpacked into the next via `f(*fields)`, and the initial call passes a single tuple — `starcompose(f, g)((a, b))` runs `f(a, b)` then `g(*result)`, a 1-tuple for single-valued intermediates. This is the form for multi-arg dispatch stages that thread several values forward.
- `result_catch(exception=E)` is the boundary lift: it wraps any callable — including a `singledispatch` surface — converting matching exceptions to `Error(e)`, passing an already-`Result` return through without double-wrapping, and letting non-matching exceptions propagate. Wrapping a dispatch surface captures the default arm and every registered arm uniformly, giving one lift point for a whole open family.

[CURRY_ALIGNMENT_FOR_PIPE]:
- `curry_flip(n)` moves the first `n` positional parameters to the end of the partial-application order, placing config before source: `@curry_flip(1) def scale(source, factor)` yields `scale(factor)(source)`, so `pipe(data, scale(10))` reads as data flowing through configured operations. `curry(n)` instead fixes the first `n` left-to-right: `@curry(2) def f(a, b, c)` yields `f(a)(b)(c)`. They are not symmetric — choose by which argument is the data flowing through: that argument must land last, which is what `curry_flip` arranges and `curry` does not. The `n` counts config args, not total arity.

[APPLICATIVE_AND_RECURSIVE_FOLDS]:
- `traverse(fn, lst)` threads `fn: A -> Result[B, E]` across a `Block`, accumulating `Result[Block[B], E]` and short-circuiting on the first error; `sequence` is `traverse(identity, lst)`, lifting a block of results to a result of block. Both require `expression.collections.Block` — pass `block.of_seq(iterable)` — and both are strictly fail-fast: there is no parallel or error-accumulating variant, so collecting all errors needs a manual fold over `Result.map_error` plus a list monoid.
- `@tailrec` is a trampoline: the body returns `TailCall(*next_args)` instead of recursing, and a loop re-invokes until a non-`TailCall` value emerges, giving unbounded depth for recursive dispatch over nested tagged unions. `@tailrec_async` is the same for `async def`. The return path is exclusive — a tail-recursive body returns either the final value or `TailCall`, never a `Result`; make it fallible by returning `Ok`/`Error` as the terminal value and lifting with `result_catch` separately, because mixing `TailCall` and `Result` on one return path is a type error.

[SINGLEDISPATCH_STRUCTURAL_AMBIGUITY]:
- A `@runtime_checkable` Protocol registered with `singledispatch` enters MRO resolution as a virtual superclass. Ambiguity raises `RuntimeError` at dispatch — not registration — only when two registered types are both virtual superclasses of the target and neither is a subclass of the other. Two independent structural Protocols that each match a class but share no method overlap collide this way.
- A Protocol overlapping a concrete ABC always loses to the ABC: `issubclass(Protocol, ABC)` is structurally `True` when the ABC satisfies the Protocol's members, so the ABC is the more specific candidate and resolution succeeds without error. The safe rule for open structural dispatch is to register concrete types and ABCs and never two independent structural Protocols on one surface; disjoint Protocols (no shared methods) are safe because no single type satisfies both, and the collision only fires when one concrete type matches multiple registered virtual types.

[STRUCTURAL_PREDICATE_TABLES]:
- `frozendict[str, Callable]` is a hashable, immutable, definition-time-closed dispatch table; mutation raises `TypeError`, and the table object is itself usable as a dict key for table-of-tables meta-dispatch. `TypeIs` predicates serve as table values — `frozendict[str, tuple[predicate, handler]]` — and the narrowing propagates into the handler call site. A linear predicate scan beats `singledispatch`'s warm path for small families (a dict lookup plus a predicate costs well under the `WeakKeyDictionary` traversal), and for structural checks `singledispatch` cannot express — element-depth distinctions like `list[int]` versus `list[str]` — a `TypeIs` predicate table is the only correct form.

[CYCLOPTS_DISPATCH_SEAMS]:
- `App.__call__` takes no `dispatcher`; the `Dispatcher` hook exists only on `interactive_shell`. Programmatic rail dispatch goes through `app.parse_args(tokens)`, which returns `(command, BoundArguments, ignored_dict)` for a manual dispatcher call; `parse_args` raises `CycloptsError` subclasses on parse failure, which must be caught and lifted to `Error` at that call site rather than escaping the rail.
- `result_action` accepts a callable, a literal string, or a sequence of either, where a sequence is a left-to-right pipeline — `[fn1, fn2]` applies `fn2(fn1(value))`. `result_action=Ok` wraps every command return in `Ok`, turning the whole app surface into `Result`-returning. `resolve_returncode` reads a `__cyclopts_returncode__` callable off the result to derive a CLI exit code; attaching that method (via a thin boundary subclass — an egress concern, not a domain one) lets a `Result`-returning command set its own exit status without a separate transformer.
- `App.meta` is a lazily-created child `App` sharing `result_action` with a `_meta_parent` back-reference. `@app.meta.default` registers a token-level interceptor that receives raw string tokens before command resolution and must call `app(tokens)` itself — the seam for cross-cutting concerns (auth, tracing, dry-run) without touching command bodies. A re-entrant `app(tokens)` from inside that interceptor is detected through the app stack and silently forced to `result_action="return_value"` to avoid double-output; an interceptor needing its own result handling should call `parse_args` plus manual dispatch instead of re-entering `__call__`.
- `App.run_async` is the async dispatch entry, the counterpart to `__call__`, taking `backend: Literal["asyncio", "trio"]`. Async commands registered with `@app.command` are detected by coroutine-function inspection and routed through the chosen backend's runner, so the CLI boundary owns the event-loop entry while command bodies stay `async def -> Result[T, E]`.
