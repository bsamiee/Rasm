# Fault and Cancellation Taxonomy at the Seam

[CANCELLATION_IS_A_TYPED_VALUE_NOT_AN_ESCAPE]:
- Every exception that crosses the seam interpreter is funneled through one normalizer that maps the host's three control-flow exception families onto fixed-code rail values: `TaskCanceledException` and `OperationCanceledException` both collapse to `Errors.Cancelled` (code `-2000000001`), `TimeoutException` collapses to `Errors.TimedOut` (code `-2000000002`), and `AggregateException` fans out into a coalesced multi-error. The consequence is that cancellation is never an out-of-band stack unwind inside the effect rail — it is an ordinary catchable, inspectable, retryable error value sitting in the failure channel beside domain faults.
- Because the normalizer runs inside the catch arm of the effect interpreter's per-node `MakeHandler`, the typed predicate of a catch combinator sees `Errors.Cancelled` exactly as it sees any expected error: `Error error = Error.New(e); return !Predicate(error) ? fail(error) : Failure(e).Bind(Next)`. There is no separate cancellation channel; the discriminant is the code, recovered with `e.HasCode(cancelCode)`, never a `catch (OperationCanceledException)` clause in domain code.
- The collapse of both `OperationCanceledException` and `TaskCanceledException` to the identical code means a seam cannot distinguish "the linked parent token cancelled" from "this fiber's local token cancelled" by error code alone — the code carries the kind of stop, not the source of the stop. A seam that must react differently to caller-initiated shutdown versus an internal deadline must carry the source as separate evidence (a distinct error case constructed at the originating boundary), because the rail flattens both into one value.

[THE_TIMEOUT_CANCELLED_COLLISION]:
- A deadline combinator runs its body inline on the calling fiber under a child environment carrying a fresh timer-armed cancellation source; when the timer fires, it cancels that source and the next cancellation-observing primitive in the body throws `OperationCanceledException`. That exception normalizes to `Errors.Cancelled`, not `Errors.TimedOut` — so a body that timed out is, in the rail, indistinguishable by code from a body whose parent token was cancelled. `Errors.TimedOut` is produced only when a raw `TimeoutException` is lifted directly, which the inline deadline path never does.
- The practical rule: a seam that needs "did this hop exceed its deadline" as a recoverable signal cannot read it off the deadline combinator's own output. It must arm an observable side-channel — a flag set in the timer registration, or an explicit `fail(Errors.TimedOut)` raised by a body-internal deadline check — and treat the combinator's `Cancelled` purely as the unwind mechanism. Reading `Cancelled` as "timed out" is wrong whenever the parent scope can also cancel.

[RETRY_LOOPS_ON_CANCELLATION_BY_DEFAULT]:
- The retry algebra is `Catch(BracketFail, e => predicate(e) ? fail(e) : recurse)` — it retries on every error the predicate does not stop on. The bare retry form supplies the stop predicate `_ => false`, meaning "stop on nothing, retry on everything." Because cancellation is a normalized `Errors.Cancelled` value flowing through that same `Catch`, an unqualified retry loop treats a cancelled token as a retryable failure and spins — each attempt re-enters a body whose cancellation-observing primitives immediately re-throw, producing a tight retry storm against a token that will never un-cancel.
- The correct stop predicate for any retry that may run under a cancellable scope must short-circuit the cancellation code first: the predicate stops when `e.HasCode(cancelCode)` or when the parent token reports cancellation, and only then applies the domain-fault retry policy. Cancellation is the one error that is never transiently recoverable and therefore must be the first arm of every retry predicate at a seam.

```csharp
static IO<A> reactivate<A>(IO<A> attach, Schedule policy) =>
    attach.RetryUntil(policy, e => e.HasCode(Errors.Cancelled.Code) || e.HasCode(Errors.TimedOut.Code));
```

- The schedule-driven retry threads its accumulator across iterations and, on schedule exhaustion, fails with the *last* observed error rather than the first — so a probe that fails N times surfaces only the final fault unless the seam coalesces the sequence itself. The non-schedule retry recurses without an attempt accumulator and so carries no envelope, making the schedule form mandatory whenever the seam needs a bounded reactivation count.

[ERROR_IS_A_MONOID_AND_FAN_IN_COALESCES]:
- The rail's error type is a monoid: `operator +` (Combine) appends, single errors promote to a many-error trie on append, and many+many concatenates the underlying error sequences. `Errors.None` is the identity — an empty many-error trie, semantically "a failure with no information," distinct from "no failure." This is the structural substrate for fault coalescing at any fan-in seam: collecting N boundary-probe failures is a fold with `+`, never a list the caller post-processes.
- A coalesced many-error answers discriminant queries by quantifier, not by its own identity: `IsExceptional` is `Exists(member.IsExceptional)` — the aggregate is exceptional if *any* member is — while `IsExpected` is `ForAll(member.IsExpected)` — expected only if *every* member is. `HasCode(c)` is `Exists(member.Code == c)`, and `IsType<E>` is `Exists(member is E)`. A fan-in result therefore reports "contains a cancellation" or "contains a non-recoverable exception" without the consumer ever iterating the trie; the quantifier semantics are the recovery-decision surface.
- An aggregate exception crossing the seam is decomposed at the boundary: its inner-exception set is mapped through the same normalizer and folded into a many-error. A single-inner aggregate flattens to that one normalized error rather than a one-element trie, so a fan-out that produced exactly one fault does not pay the aggregate wrapper. This means cancellation hidden inside an `AggregateException` (the common `Task.WhenAll` shape) surfaces as a `Cancelled` member of the coalesced rail, catchable by code without unwrapping.

[TYPE_DIRECTED_CATCH_OVER_A_COALESCED_RAIL]:
- A type-directed catch arm matches not only a directly-typed error but a many-error that *contains* a member of the target type, because the match predicate is `e is E || e.IsType<E>()` and the aggregate's `IsType<E>` is membership. When the matched value is the aggregate rather than the bare type, the handler does not run once on the whole trie — it filters the trie to only the `E`-typed members and chains the per-member handler across them. Two filter-and-chain strategies are exposed: a monad-bind variant that threads each matched member sequentially through the recovery (short-circuiting on the first recovery failure), and a monoid-fold variant that accumulates every member's recovery via the rail's `+`.
- The filter is total and order-preserving: it chooses the `E`-typed members out of the trie and yields `Errors.None` (identity, not a thrown "not found") when none match, so composing several type-directed arms over one coalesced fan-in fault is associative and never throws on a miss. A seam with heterogeneous probe failures — some cancellations, some expected unavailabilities, some genuine exceptions — installs one arm per error type, each draining its slice of the trie, and the arms compose into a single recovery surface rather than a per-member switch.

```csharp
var recover =
    catchOf<Exceptional, M, A>(ex => reportFatal(ex)) >>
    @catch(Errors.Cancelled.Code, () => stop) >>
    catchOf<Expected, M, A>(ex => retryLater(ex));
```

[FIRST_COMPLETION_RACE_HAS_NO_COLLECT_SEMANTICS]:
- The race combinator over a set of forks is implemented as `await (await WhenAny(...))` followed by cancelling the losing forks via the local source. The doubly-awaited shape means the *first task to complete* — success or fault — is the result; a faulted first-completer rethrows directly. The combinator does **not** collect failures and continue waiting for a later success, despite the intuition that a race should keep trying. The "collect every error and only fail if all fail" behavior is a separate accumulating combinator; conflating the two is a real availability bug, because the race returns the first failure as eagerly as the first success.
- The race seals its losers by cancelling a fresh local cancellation source derived from the parent (a fork-local scope), not the parent source — so winning forks short-circuit the rest without tearing the parent scope, and the loser cancellation propagates as `Cancelled` into each loser's own rail where it is swallowed by the loser never being awaited. The accumulating fan-in, by contrast, awaits all and returns either every result or the coalesced error trie, making it the correct shape for partial-availability reporting where the race is the correct shape only for genuine first-wins latency hedging.
- A pre-entry guard short-circuits both combinators: if the ambient token is already cancelled at the moment the lifted async body begins, it throws immediately rather than spawning any fork. The seam therefore cannot rely on a race to "run at least one probe" under an already-cancelled scope — the guard fires first and the entire fan-out is skipped, surfacing as `Cancelled` with no probe ever started.

[STALE_NOTIFICATION_IS_THE_DUAL_OF_CANCELLATION]:
- The transactional reference's change notification fires inside the transaction's `finally` for *every* iteration, including conflict-aborted retries, carrying the in-transaction proposed diff which is non-empty on abort. This is the inverse failure surface to cancellation: where cancellation is a real stop delivered as a value, an abort notification is a non-event delivered as if it were real. A subscriber that treats the change payload as committed truth acts on a rolled-back write; the only safe read is to re-read the committed value after the notification rather than trusting the payload.
- The change event on the lock-free map cell carries a patch only for transforms threaded through the tracking argument the swap function receives; a swap that ignores the argument and constructs a fresh collection emits a patch showing the final state with no tracked intermediate diff. The notification fidelity is therefore a property of how the swap body is written, not a guarantee of the cell — a seam that needs precise change deltas must thread the provided collection through `SetItem`/`Remove` and never rebuild, exactly as a seam that needs precise fault provenance must construct its error at the originating boundary and never let the rail flatten it.
