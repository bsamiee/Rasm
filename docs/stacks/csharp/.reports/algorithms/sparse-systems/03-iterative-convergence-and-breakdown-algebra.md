# Iterative Convergence And Breakdown Algebra

[CRITERION_STACK_AS_ORDERED_FOLD]:
- The stop policy is a short-circuit fold over an ordered criterion list, not a set: the iterator walks the list in insertion order and returns the first non-`Continue` verdict, so the array index is a priority, not a label. The verdict an iteration receives is the highest-priority criterion that fires, and every lower-priority criterion that would also fire on that iteration is never consulted. Composing the stop policy is therefore choosing both the criterion vocabulary and its precedence in one ordered value — the list is the policy, and its order is load-bearing semantics.
- The correct precedence is NaN-failure before divergence before convergence before the hard ceiling: under that order the convergence verdict wins the iteration it becomes true on and the count cap is the genuine last resort, whereas any order placing the cap ahead of the residual converts an in-tolerance final iterate into a non-convergence verdict on exactly the iteration where both fire. The precedence fault is invisible at the call site because every ordering returns the same verdict type; only the ordered value itself records which partition the solve was judged under.
- The iteration ceiling belongs to the operator's dimension, not to a library constant: a size-scaled cap — a multiple of the expected Krylov subspace dimension — is a policy value the boundary owner computes once and threads into the count criterion at construction. Constructing the iterator from an explicit ordered criterion array is the only way to fix both the precedence and the cap, and the reuse shape is one criterion-stack policy value built at factorization time from the operator's dimension and the caller's tolerance, reused across every right-hand side that operator answers.
[PER_ITERATION_IDEMPOTENCY_CONTRACT]:
- Each criterion guards its body with a last-seen-iteration gate: if the iteration number it is handed does not exceed the number it last processed, it returns its cached verdict without recomputing. Re-invoking the fold with the same iteration number is therefore free and stable — the stop decision for iteration `k` is computed once and memoized inside each criterion. This makes the criterion stack safe to consult redundantly but means a criterion that tracks history advances its internal window only on a strictly increasing iteration number, and a solver that re-asks the same number twice gets the first answer, not a fresh evaluation.
- The convergence criterion's internal counter is the gap between the current iteration number and the iteration number at its previous in-threshold call, reset to a sentinel below zero on construction and on reset. On the first in-threshold call after a reset the gap is the iteration number plus one, which clears any required-sustain count immediately at any non-trivial iteration; under the normal per-iteration invocation pattern the gap collapses toward one because the previous-call marker advances every call. A sustain requirement above one is therefore unreachable under per-iteration polling and only becomes meaningful when the criterion is consulted every `k`-th iteration. The usable configuration is the degenerate sustain of zero — fire on the first in-threshold iteration — which is the only mode that behaves as a relative-residual gate under standard polling.
[NAN_VERDICT_IS_ASSIGNED_BY_PRECEDENCE]:
- Three separate criteria each detect non-finite state and each assign a different terminal verdict to it. The failure criterion flags a non-finite solution-or-residual norm as `Failure`; the divergence criterion flags a non-finite residual norm as `Diverged`; the convergence criterion flags a non-finite source-or-residual scaled norm as `Diverged`. Which verdict a NaN actually receives is decided purely by criterion order — the first NaN-aware criterion in the stack wins — so the same numerical event surfaces as `Failure` or `Diverged` depending only on precedence. Placing the failure criterion first makes NaN a distinct terminal class separable from honest divergence; demoting it folds NaN into the divergence bucket and erases the distinction the recovery dispatch needs.
- The breakdown guard's unit-in-last-place test is a binary-representation distance and is asymmetric on non-finite input: a NaN denominator fails the near-zero test and is therefore not caught as a breakdown — it slips past the guard and propagates as a poisoned value into the next iterate, where only the NaN-aware stop criteria can catch it, one iteration later and as `Failure` or `Diverged` rather than as a breakdown. The breakdown guard catches the zero case, never the NaN case; the two pathologies need two independent rails, and conflating them under one "breakdown" handler leaves the NaN path entirely to the criterion stack's precedence.
[BREAKDOWN_SEMANTICS_ARE_A_KIND_PROPERTY]:
- The same mathematical breakdown — an inner product collapsing to a near-zero denominator in the bi-Lanczos recurrence, tested at one-unit-in-last-place from zero — produces three incompatible control outcomes across the non-symmetric solver family with no syntactic difference at the call site: one solver throws a numerical-breakdown exception, one cancels the iterator and exits leaving a cancelled verdict and the partial iterate, one substitutes one in place of the near-zero denominator and continues to a normal verdict. The recovery contract is a property of the solver kind, recoverable from the kind alone, and a request surface that swaps solvers swaps failure semantics silently.
- The three shapes converge on one obligation: recompute the true residual of the original operator against the returned iterate before trusting it. The substitution path is the most dangerous because it returns a normal verdict; the cancelled path is recoverable because a composite loop resets the verdict and continues; the exception path is loud. One catch combinator plus one residual witness collapses three control shapes into one failure-handling surface.
[DIVERGENCE_WINDOW_IS_A_SYMMETRY_KEYED_POLICY]:
- The divergence verdict is a strict all-pairs conjunction over a sliding window — it fires only on uninterrupted super-threshold growth across the entire window, so a single dip suppresses it. The window length is the one knob that trades false positives against detection latency, and the trade runs in opposite directions for the two operator classes: on symmetric systems the residual descends near-monotonically, so a short window detects genuine blow-up early without spurious firing; on non-symmetric systems the residual oscillates through the warm-up phase, so the same short window fires spuriously during the normal transient and only the long window is safe. The window length is therefore a policy value keyed on the operator's symmetry class, carried beside the criterion stack, never a global default.
[VERDICT_RAIL_AND_EXCEPTION_RAIL_ARE_DISJOINT]:
- The solve entrypoint returns the iterator's terminal verdict and wraps nothing in a catch, but the very first action inside the solver body — before the iteration loop begins — is to initialize the preconditioner against the operator. A preconditioner that rejects the storage shape, or hits a zero pivot during its incomplete factorization, throws out of that initialization, and the throw escapes the verdict-returning entrypoint entirely. The two rails are disjoint by construction: a returned verdict means the loop ran; an escaped exception means it never started. Code that reads only the verdict treats a preconditioner-init failure as if no solve occurred, with stale prior content left in the result buffer — the most deceptive state the surface can produce, because the buffer looks like an answer.
[THE_WITNESS_IS_THE_ONLY_SURVIVING_GATE]:
- The convergence verdict is computed against the preconditioned residual the loop carries, not against the true residual of the original operator: a left-preconditioned system measures the transformed residual, whose norm relationship to the true error is distorted by the preconditioner's conditioning, so `Converged` certifies the preconditioned residual fell below tolerance and nothing more. The independent recomputation of the true relative residual against the original operator is the only correctness witness that survives preconditioning, breakdown substitution, cancellation, and composite-solver exception swallowing alike.
- One boundary owns both rails and the gate: it runs the solve under a catch combinator that converts the breakdown exception and the preconditioner-init exception into the failure rail, reads the verdict for routing — divergence routes to a regularized retry or a direct solve because more iterations will not help, count-exhaustion routes to a direct solve optionally seeded with the partial iterate because the trajectory was sound but slow — and accepts the iterate only if the recomputed true relative residual is finite and within tolerance. A substituted-through breakdown, a count-capped partial, and a preconditioned `Converged` all pass through the same finite-and-in-tolerance test before the iterate is admitted as a solution.

```csharp
// One criterion-stack policy value fixes precedence, a size-scaled cap, and a symmetry-keyed
// divergence window; one solve boundary folds verdict-routing, breakdown-catching, and the
// true-residual witness into a single Fin rail.
static Iterator<double> Stack(int dim, double tol, bool symmetric) =>
    new(new IIterationStopCriterion<double>[]
    {
        new FailureStopCriterion<double>(),                  // NaN ⇒ Failure, first so it stays distinct
        new DivergenceStopCriterion<double>(0.08, minimumIterations: symmetric ? 3 : 10),
        new ResidualStopCriterion<double>(tol),              // convergence BEFORE the cap
        new IterationCountStopCriterion<double>(8 * dim),    // size-scaled ceiling, last resort
    });

static Fin<double[]> SolveIter(
    Matrix<double> a, Vector<double> b, IIterativeSolver<double> solver,
    IPreconditioner<double> pre, int dim, double tol, bool symmetric) =>
    new Try<(IterationStatus Verdict, Vector<double> X)>(() =>
    {
        var x = Vector<double>.Build.Dense(dim);
        var verdict = a.TrySolveIterative(b, x, solver, Stack(dim, tol, symmetric), pre);
        return Fin.Succ((verdict, x));                       // init-throw + breakdown land in Fail
    }).Run()
      .Bind(t => TrueRelResidual(a, b, t.X) is var r && double.IsFinite(r) && r <= tol
            ? Fin.Succ(t.X.AsArray())                        // witness gate: Converged, Cancelled,
            : Fin.Fail<double[]>(Error.New(                  //   and substituted-through all pass
                $"residual witness failed: verdict={t.Verdict}, r={r}")));  // ONLY this one test
```
