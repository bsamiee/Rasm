# Sparse Systems: Frontier

[SPARSEMATRIX_SPMV_VECTORIZATION_GAP]:
- `SparseMatrix.DoMultiply(Vector<double> rightSide, Vector<double> result)` is a scalar CSR row-loop with no `TensorPrimitives` involvement: for each row `i`, it accumulates `values[j] * rightSide[columnIndices[j]]` into a local `double num3` via a scalar for-loop. The `rightSide[columnIndices[j]]` access is a gather with irregular indices, which blocks vectorized gather-then-multiply. `TensorPrimitives` provides no indexed-gather API on .NET 10. `TensorPrimitives.Dot` is limited to contiguous span pairs and is inapplicable here. The only SIMD path for the sparse mat-vec product in hot loops is to hand-vectorize using `Vector256.GatherMaskVector256` (x86 AVX2) or `Vector128.GatherMaskVector128` (ARM SVE is not yet stable on .NET 10) via `System.Runtime.Intrinsics` — there is no higher-level API that reaches this pattern through admitted packages.
- `SparseMatrix.DoTransposeThisAndMultiply(Vector<double> rightSide, Vector<double> result)` is a scatter-accumulate kernel: for each nonzero `(i, j, v)`, `result[j] += v * rightSide[i]`. The scatter-write pattern has write-after-write hazards across rows and cannot be vectorized without explicit atomics or row-partitioning. The forward SpMV (row-loop) is vectorizable via gather; the transpose SpMV (column-scatter) is not — this asymmetry matters when selecting solver inner-loop orientation for a given sparsity pattern.

[COMPOSITE_SOLVER_EXCEPTION_SWALLOWING]:
- `CompositeSolver.Solve` wraps each sub-solver call in `try { ... } catch (Exception) { input.CopyTo(vector); continue; }`. Any exception — including `NumericalBreakdownException`, `OutOfMemoryException`, or preconditioner initialization failures — is silently caught and the solver advances to the next entry in the list. The result after exhausting all solvers carries the last `StoppedWithoutConvergence` partial iterate with no status indication of which solvers threw versus which ran to completion. The composite pattern is therefore not a typed-error surface; it is a best-effort loop with silent exception burial. The calling site's true-residual recomputation is the only discriminant between a buried-exception partial result and a legitimate non-convergence partial result.
- `CompositeSolver` ignores the `preconditioner` parameter passed to `Solve`: each sub-solver uses `solver.Item2 ?? preconditioner` where `solver.Item2` is the per-entry preconditioner from `IIterativeSolverSetup<T>.CreatePreconditioner()`. When `CreatePreconditioner()` returns non-null, the external preconditioner is never used. When it returns null, the fallback is `UnitPreconditioner<T>`, not the external one, because `solver.Item2 ?? preconditioner` is the expression but `CreatePreconditioner()` null → `UnitPreconditioner` substitution happens in the constructor. The external `preconditioner` argument is dead code in the current implementation — each `IIterativeSolverSetup<T>` must return a configured preconditioner from `CreatePreconditioner()` or accept `UnitPreconditioner` as the default.

[MILU0_RESIDUAL_CORRECTION_LIMIT]:
- `MILU0Preconditioner` with `UseModified = true` accumulates the dropped fill into the diagonal during the ILU factorization so that the preconditioner matrix preserves row sums of the original matrix. This is numerically beneficial for M-matrices (negative off-diagonal, non-negative diagonal, positive real eigenvalues) and diagonally dominant systems, but it is not equivalent to a general shifted incomplete factorization. For problems where the matrix is not an M-matrix but has significant off-diagonal structure (e.g., non-symmetric convection-dominated PDEs), the MILU(0) modification can make the preconditioner worse than standard ILU(0) because the diagonal correction pushes the approximation away from the true L and U factors in a non-locally-bounded way.
- The `NumericalBreakdownException` thrown by `MILU0Preconditioner.Initialize` carries the message `"Zero pivot encountered on row " + num + " during ILU process"`. The row index in the message corresponds to the zero-pivot row in the ILU iteration order, which is the row index in the CSR storage of the input matrix, not a permuted index. For symmetric matrices, pre-ordering with AMD before calling MILU0 reduces the probability of zero pivots, but `MILU0Preconditioner` does not accept or apply a permutation; the caller must symmetrically permute the matrix before calling `Initialize` using `SparseCompressedRowMatrixStorage<T>.MapIndexedTo` and an explicit permutation vector.

[DIVERGENCE_CRITERION_HISTORY_WINDOW_MECHANICS]:
- `DivergenceStopCriterion<T>` allocates `_residualHistory` of length `_minimumNumberOfIterations + 1` on first use (not at construction). The `IsDiverging` check tests all consecutive pairs in the window: if ANY pair has `history[i] <= history[i-1]` or `history[i] < history[i-1] * (1 + maxRelIncrease)`, the test short-circuits and returns false. For a window of 10 (default), all 10 consecutive pairs must show monotone increase above the 8% relative threshold — this is a strict AND over the entire window. A residual sequence that oscillates even once in a 10-iteration window cannot trigger divergence, regardless of how large the overall growth is. For non-symmetric indefinite systems where BiCgStab produces chaotic residual histories (not monotonically increasing), this criterion never fires, leaving the iteration to exhaust the `IterationCountStopCriterion` and return `StoppedWithoutConvergence` instead of `Diverged`.
- `DivergenceStopCriterion<T>.MinimumNumberOfIterations` setter enforces `value >= 3` (throws `ArgumentOutOfRangeException` for 1 or 2). The constructor default of 10 with `MaximumRelativeIncrease = 0.08` was designed for well-behaved conjugate-gradient-style residual sequences. For Krylov methods on non-symmetric systems (BiCGStab, TFQMR, GpBiCG), where residual oscillation is normal for the first 5-20 iterations, reducing the window to 3 produces spurious `Diverged` signals during the oscillatory warm-up phase.

[TFQMR_NEAR_ZERO_CANCEL_VS_BICGSTAB_THROW]:
- `TFQMR.Solve` handles near-zero inner products by calling `iterator.Cancel()` and breaking out of the iteration loop, leaving `IterationStatus.Cancelled` in the iterator. It does not throw `NumericalBreakdownException`. `BiCgStab.Solve` throws `NumericalBreakdownException` on the same near-zero condition. The caller of `TrySolveIterative` therefore encounters two incompatible failure modes for the same mathematical breakdown event depending on which solver is in use — a status return versus an unhandled exception — with no syntactic difference at the call site. A safe outer boundary catches `NumericalBreakdownException` from `TrySolveIterative` regardless of the solver; but TFQMR's `Cancelled` result also requires a true-residual recomputation to confirm that the cancelled partial result is unusable, since `Cancel` leaves whatever partial iterate is in `result` at the time of cancellation.
- `GpBiCg.Solve` handles near-zero inner products by substituting `1.0` in place of the near-zero value (`num2 = 1.0` and `num6 = 1.0`) and continuing. This is a damage-control heuristic, not breakdown recovery. The continued iteration after substitution may produce a valid result for mildly ill-conditioned problems or an arbitrary vector for severely ill-conditioned ones. GpBiCg never throws `NumericalBreakdownException` and never calls `iterator.Cancel()` — the iteration continues through all substituted-zero events until a stop criterion fires. A post-solve residual check is mandatory; the solver's own convergence status cannot be trusted when substitutions have occurred.

[MLKBICGSTAB_STARTING_VECTOR_DEPENDENCY]:
- `MlkBiCgStab.Solve` checks `_startingVectors` at the start: if `_startingVectors` is non-null and has the correct count and size, it uses them; otherwise it calls `CreateStartingVectors(DefaultNumberOfStartingVectors = 50, n)`, which generates up to 49 random unit vectors via `DistributionVector.CreateRandom`. The 50 starting vectors are dense `DenseVector` instances of length n. For a system of size n = 10,000, this allocates 50 × 10,000 × 8 bytes = 4 MB at solver-initialization time before the first iteration. The `NumberOfStartingVectors` property controls this; setting it to a small value (3-5) reduces startup allocation and is appropriate when the approximate solution is already close to the true solution (warm-start from a prior time step), eliminating the benefit of multiple starting vectors.
- `MlkBiCgStab` throws `NumericalBreakdownException` at three distinct points in the iteration. Unlike `BiCgStab` (which has one throw point), the three break-down conditions in ML(k)-BiCGStab correspond to different phases of the multi-Lanczos recurrence. The exception message for all three is the default `NumericalBreakdownException` (no additional context). Distinguishing which phase broke down requires a wrapping try/catch at three distinct code regions — not achievable from outside the `Solve` method.

[RESID_STOP_CRITERION_MINIMUMITERATIONS_DEFAULT]:
- `ResidualStopCriterion<T>(double maximum, int minimumIterationsBelowMaximum = 0)` — the default `minimumIterationsBelowMaximum` is 0, not a positive count. The `DetermineStatus` convergence gate is `_iterationCount >= _minimumIterationsBelowMaximum`, so with the default of 0, the criterion signals `Converged` on the first iteration the residual falls below `maximum`, without requiring the residual to be sustained below the threshold for multiple iterations. The default stack from `MatrixBuilder.IterativeSolverStopCriteria` passes `1e-12` as the threshold but uses the default 0 for the minimum-iterations guard — single-point convergence. The wave 1 finding that `MinimumIterationsBelowMaximum = 1` degrades to single-point convergence is not accurate: the actual default is 0, which is more aggressive than `1` — it fires on iteration k=0 if `_iterationCount (= k - _lastIteration = 0 - (-1) = 1) >= 0`. In practice the guard is `_iterationCount >= 0`, which is always true, meaning the criterion fires the moment the norm drops below the threshold regardless of how long it has been below.

[NORMALIZEZEROS_ABSENT_IN_OF_COMPRESSED_COLUMN_FORMAT]:
- `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` calls `NormalizeOrdering()` and `NormalizeDuplicates()` but does NOT call `NormalizeZeros()`. Explicit zeros in the input column-format arrays survive as explicit zeros in the resulting CSR storage. For the path from CSparse CSC (which may contain explicit zeros after `DropZeros` with threshold 0.0) to MathNet CSR for residual arithmetic, explicit zeros in the output are benign for arithmetic but inflate `ValueCount` and slow preconditioner initialization. To remove them, call `storage.NormalizeZeros()` after `OfCompressedSparseColumnFormat`. For the path into `PopulateExplicitZerosOnDiagonal` before MKL DSS, explicit zeros on the diagonal are required and should NOT be removed.
- `SparseCompressedRowMatrixStorage<T>.Normalize()` calls `NormalizeOrdering()` then `NormalizeZeros()` — it does not call `NormalizeDuplicates()`. Using `Normalize()` on a matrix that has structural duplicates from assembly or accumulation leaves those duplicates intact; the values are not summed. The correct complete normalization sequence after building from coordinate format is `NormalizeOrdering()` then `NormalizeDuplicates()` then `NormalizeZeros()` — which is exactly what `OfCompressedSparseColumnFormat` does minus the zero step. Using the public `Normalize()` method skips the duplicate-summation step.

[ITERATOR_CANCEL_SEMANTICS_STICKY]:
- `Iterator<T>.DetermineStatus` guards at the top: `if (_status == IterationStatus.Cancelled) return _status`. Once cancelled (via `iterator.Cancel()` or a `CancellationStopCriterion`), the iterator permanently returns `Cancelled` on all subsequent `DetermineStatus` calls, regardless of the residual vector contents. `Reset()` clears `_status` back to `Continue` and also resets all stop criteria. `CompositeSolver` calls `iterator.Reset()` between solvers; a cancelled iterator from TFQMR inside a `CompositeSolver` is therefore reset before the next solver, and the `Cancelled` status from TFQMR's near-zero handling is discarded — the composite loop continues as if the solver had returned normally with its partial iterate in `result`.
- `CancellationStopCriterion<T>` links a `CancellationToken` into the stop-criteria stack. When the token is cancelled, `DetermineStatus` returns `IterationStatus.Cancelled` on the next criteria evaluation. The `Iterator<T>.Reset()` call creates a new `CancellationTokenSource` linked to the original `_masterToken`. If `_masterToken` is already cancelled at the time of `Reset()`, the new TCS is immediately cancelled and the criterion returns `Cancelled` on the first `DetermineStatus` call after reset. A composite solver using an already-cancelled token will never make progress through its solver list — each solver cancels on the first iteration.

[DSS_PROVIDER_UNIMPLEMENTED_FALLBACK]:
- `ManagedSparseSolverProvider.Solve` (all four overloads) unconditionally throws `NotImplementedException`. There is no fallback managed implementation in `MathNet.Numerics`. The `ISparseSolverProvider<T>.Solve` interface — accepting raw CSR arrays (`rowPointers`, `columnIndices`, `values`), RHS, and solution arrays — is the bridge to external providers (MKL). Without an MKL-backed provider registered via `SparseSolverControl`, calling `Control.SparseSolverProvider.Solve(...)` throws immediately. This interface is not used internally by any `IIterativeSolver<T>` implementation; it is a separate, high-performance single-call path intended for direct sparse factorization bypassing the `Matrix<T>` type hierarchy entirely.
- The `DssMatrixStructure` enum values are not sequential: `Symmetric = 536870976`, `Nonsymmetric = 536871104`, `SymmetricComplex = 536871168`. These are MKL DSS option literals that map to specific bit patterns in the MKL DSS interface. Passing the wrong `DssMatrixStructure` to a solve call does not throw — MKL DSS returns `DssStatus.MKL_DSS_STRUCTURE_ERR = -14`. The `DssStatus` enum is the only type-safe diagnostic for MKL DSS errors; code calling `ISparseSolverProvider<T>.Solve` must map every non-`MKL_DSS_SUCCESS` status to a typed domain failure before returning to the caller, because MKL DSS failure codes are not typed exceptions.

[SPARSEMATRIX_TIMES_SPARSEMATRIX_TWO_PASS_ALLOCATION]:
- `SparseMatrix.DoMultiplySparse` uses a two-pass algorithm: the first pass counts the number of nonzeros in each result row using a marker array `int[] array = new int[columnCount]` initialized to `-1`, then allocates `int[num]` and `double[num]` for the result; the second pass fills the values. The marker array size is `other.ColumnCount`, not `this.RowCount`, and uses `j` (the output row index) as the marker sentinel. This means the first pass allocates O(n) extra work array at the start of every sparse-sparse multiply; for repeated multiplications (e.g., Krylov subspace methods where the preconditioner involves a product), this allocation is not amortized. The `DoMultiplySparse` result is assigned to the input `SparseMatrix result`'s storage via `result._storage.ColumnIndices = array2; result._storage.Values = array3`, bypassing the storage's `AutoTrimStorage` mechanism — the allocated arrays are exact-fit from allocation.

[GENERIC_MATH_CONSTRAINT_ON_NUMERIC_RAIL]:
- `TrySolveIterative` is defined on `Matrix<T>` where `T : struct, IEquatable<T>, IFormattable`. It does not constrain T to `IFloatingPoint<T>` or `INumber<T>`. `ResidualStopCriterion<T>.DetermineStatus` calls `residualVector.InfinityNorm()` which dispatches to `Vector<T>.InfinityNorm()`, which internally calls provider BLAS. For `T = double`, this is a MathNet-provided managed scalar loop. The `double.IsNaN(num)` guard in `DetermineStatus` is a hard-coded `double` comparison — the `num` is `residualVector.InfinityNorm()` cast to `double` — which means the NaN guard works correctly for double but would be a type-erasure issue for a custom T. The practical constraint is that `ResidualStopCriterion<T>` and `DivergenceStopCriterion<T>` always compute norms as `double` regardless of T; this is fine for `float`, `double`, and `Complex32`/`Complex` because the InfinityNorm reduces to a real scalar, but it means criterion implementations cannot be made generic over precision using C# 14 generic math constraints.
