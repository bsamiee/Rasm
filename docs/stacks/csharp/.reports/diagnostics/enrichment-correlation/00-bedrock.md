# enrichment-correlation — bedrock

## one-correlation-identity

- A suite mints exactly one correlation root at process boot and stamps it everywhere a signal leaves: into the resource fold as the pinned service-instance identity, onto every log record via a static enricher row, into cross-process hops via one baggage key, and into every receipt and manifest the process writes.
- The correlation envelope deletes per-call-site correlation parameters — no method signature carries a correlation id, because ambient context owns it.
- One boot mint, three projections: the same minted identity materializes as the stamp envelope's origin id, the resource fold's pinned instance identity, and the correlation-root tag — three consumers, one constructor, and the equality of the three is itself a boot assertion.
- In-process ambient transport is execution-context flow: the current activity flows on async-local storage; `RuntimeContext.RegisterSlot<T>(name)` / `GetSlot<T>` provide typed ambient slots for suite-owned context beyond the activity.
- Slot names are minted once and collide loudly — registering an existing slot name throws — so slot registration belongs to the envelope owner's boot path, the same single-mint discipline as every other identity in this layer.
- Marshaling seams carry an ambient-integrity obligation: a hop onto a host or UI thread asserts the ambient activity survived the hop (or restores it from the captured context) — the assertion is one line at the seam and the alternative is a class of unparented spans diagnosable only by their absence.
- Ambient slots hold context, never dependencies: a slot carrying a service handle is ambient service location wearing correlation's clothes — slots are for identity-grade values the envelope schema names.
- Register slots once at boot and hold the returned `RuntimeContextSlot<T>`; slot lookup by string at call sites is the rejected spelling.
- The trace tuple is the fine-grained correlation layer under the root id: trace id groups one causal request tree, span id names the hop.
- Structured log events capture the ambient activity's identifiers natively as first-class fields (`LogEvent.TraceId` / `SpanId` typed as `ActivityTraceId` / `ActivitySpanId`, with `OutputProperties.TraceIdPropertyName` / `SpanIdPropertyName` for text projection) — no trace-id enricher is ever hand-written; one existing in a pipeline marks a stale design.
- Cross-thread capture for deferred work is explicit: `LogContext.Clone()` yields an enricher representing the current ambient property stack, for work that hops executors outside execution-context flow — native callbacks, pooled workers with suppressed context.
- The symmetric trap: starting an activity on a thread with no ambient parent silently roots a new trace — deferred work must carry `ActivityContext` and start child spans from it, never from the ambient current at execution time.
- The three classic ambient breaks share one carry rule: pooled callbacks scheduled without context flow, native event callbacks entering from foreign stacks, and manually created threads all receive captured context values (activity context, baggage value, cloned ambient properties) as data and restore them at entry — capture-as-value, restore-at-entry is the universal repair.
- New roots mint random trace ids — deterministic or derived trace ids are forbidden because the ratio sampler's verdict is a function of the id bytes; a biased id generator silently reshapes sampling.
- Identity layering law: correlation root (process/boot scope) ⊃ trace id (request scope) ⊃ span id (hop scope) ⊃ causal stamp (event-order scope) — four layers, each minted by exactly one owner, none derivable from another, all four present on exported evidence.
- Any operational view that cannot answer "same process? same request? same hop? what order?" is missing one layer's stamp — the four-question test is the review rubric for new signal surfaces.
- The ambient current is observable: `Activity.CurrentChanged` fires on every ambient swap — the hook for context-integrity probes and for marshaling seams that must mirror the ambient activity into a foreign execution model; it is a diagnostics seam, never a propagation mechanism.
- Activity-attached baggage reads walk the parent chain (`GetBaggageItem` resolves through ancestors), so a child span sees baggage set on its root — attachment scope is the trace tree, not the single span.
- `TraceStateString` on the current activity is the writable vendor-stamp channel — suite-owned entries compose with foreign vendors' entries under the key caps; overwriting the whole string clobbers foreign tracing vendors silently, so mutation is read-modify-write by law.
- Ambient slot storage is pluggable (`RuntimeContext.ContextSlotType`) — the default is execution-context flow; replacing the slot type is a process-wide decision made once, because mixed slot backends partition ambient visibility by thread model.

## w3c-propagation-mechanics

- `traceparent` carries `version-traceid-spanid-flags` (16-byte trace id, 8-byte span id, lowercase hex; the sampled bit rides flags) — the only mandatory hop header.
- The version prefix makes the header forward-parseable: a receiver reading a higher version still extracts the fields it knows — receivers reject malformed headers, never merely newer ones.
- `tracestate` carries vendor-keyed entries under hard caps — key ≤ 256 chars (tenant part ≤ 241, vendor part ≤ 14), value ≤ 256 chars — and the propagator drops violating entries.
- `tracestate` is the standards-sanctioned slot for vendor metadata that must survive foreign hops; its caps make it a stamp channel, not a payload channel.
- `baggage` is the cross-process key-value envelope: the spec floor is 64 list-members and 8192 total bytes; values percent-encode outside the printable-safe set (the percent character itself always encoded); `;k=v` properties attach per member.
- Baggage member properties are the metadata slot most designs forget exists: a `;` -suffixed property on a member can mark provenance or scope (which tier minted the key) without spending a second member — the envelope schema assigns property semantics, the spec deliberately does not.
- The runtime propagator implementation caps baggage at 180 items / 8192 bytes — looser than the spec floor on count, identical on bytes; budget against the bytes.
- Baggage is broadcast — every downstream hop and every instrumented client on the path sees it — so it carries opaque identifiers only, never payload and never anything classification-bearing; a secret in baggage is a secret in every downstream access log.
- Two baggage stores exist in one process and they do not synchronize: the activity-attached store (`Activity.Baggage`, consumed by the platform's distributed-context propagator) and the SDK's ambient store (`Baggage.Current`, riding its own runtime-context slot, consumed by the SDK's `BaggagePropagator`).
- A suite picks one write surface — the SDK store when the SDK owns propagation — and treats the other as read-only foreign material; writing both is double-emission, reading the wrong one is silent absence.
- The SDK store also composes non-ambiently: every baggage operation takes and returns an explicit `Baggage` value (with the ambient current as default), so pipeline stages that must not touch ambient state thread baggage as an argument — the same dual ambient/explicit grammar the rest of the correlation layer uses.
- `Baggage` is an immutable value: every `SetBaggage` / `RemoveBaggage` returns a new instance, and the static surface operates on the ambient current — mutation-style code that discards the return value changes nothing; the no-op compiles cleanly.
- Propagator selection is process policy with two seams that must agree: the platform seam (`DistributedContextPropagator.Current`) governs platform-native injection on outbound HTTP; the SDK seam (`Sdk.SetDefaultTextMapPropagator` / `Propagators.DefaultTextMapPropagator`) governs SDK-side inject/extract.
- Platform propagator factory rows: `CreateW3CPropagator` (spec-conformant baggage encoding), `CreatePreW3CPropagator` (legacy encoding), `CreatePassThroughPropagator` (forwards inbound headers untouched — the proxy/relay row), `CreateNoOutputPropagator` (suppression row), `CreateDefaultPropagator`.
- SDK initialization installs composite trace-context + baggage as the SDK-seam default; before SDK init — or in an API-only process that never loads the SDK — the default is a no-op propagator, so library code calling inject in such a process emits nothing, silently.
- Divergence between the two seams is the classic "baggage arrives, trace breaks" (or inverse) split — set both at boot to the same convention and assert it in the startup proof.
- Custom transports propagate by delegation, never by hand-rolled headers: `TextMapPropagator.Inject(PropagationContext, carrier, setter)` / `Extract(context, carrier, getter)` adapt any keyed carrier — message headers, IPC frames, job rows; the carrier adapter is the only code a new transport writes.
- `PropagationContext` pairs `ActivityContext` with `Baggage` — one value carries both propagation planes; extraction failure yields an empty context, not an exception, so admission of a context-less message is a policy decision the receiver makes explicitly.
- The context-less admission verdicts are a closed three-row policy: adopt-as-root (mint a new trace and tag it foreign-rooted), reject (transports where context is contractual), or quarantine-tag (process but mark unjoinable) — the receiver declares one row per transport, and the foreign-rooted tag is what keeps "traces that start nowhere" countable.
- Baggage values round-trip through percent-encoding transparently — consumers always see decoded values, and a value that arrives still-encoded marks a non-conforming producer upstream, not a decode step to add.
- The platform propagator contract is delegate-shaped on both sides — getter callbacks return single or multi-valued fields, setters write one field — so one adapter pair serves header collections, dictionaries, and frame maps without intermediate allocation; `Fields` advertises which keys a carrier must reserve.
- `Fields` is also the relay-hygiene contract: a relay clears the propagator-owned keys from a carrier before re-injecting, or forwarded and freshly injected headers coexist and downstream extraction becomes carrier-order-dependent.
- An all-zero trace id is the invalid sentinel: extraction yielding a default context means "no parent", and code must branch on context validity rather than constructing children of the zero trace — a child of the invalid context corrupts downstream joins with a plausible-looking tree.
- The sampled flag is the only cross-process volume signal: the parent's recorded bit arriving in the flags field is what the parent-respecting sampler matrix consumes — stripping or forcing flags at a proxy rewrites the suite's sampling topology from outside, which is why pass-through propagation exists as a distinct row.

## hlc-stamp-envelope

- The stamp envelope is one value: physical component `l` (wall milliseconds), logical counter `c`, origin id — packed as 48 bits of millis plus 16 bits of counter into one 64-bit word, ordered as `(l, c, origin)`.
- Packing is the law, not an optimization: numeric order on the packed word equals causal-consistent order, the origin tiebreak makes order total, and 16 counter bits absorb 65k same-millisecond events per origin before saturation.
- Advance algebra, local/send event: `l' = max(l, now)`; `c' = c + 1` when `l'` unchanged, else `0`.
- Advance algebra, receive of stamp `m`: `l' = max(l, m.l, now)`; `c' = max(c, m.c) + 1` when all three tie, `c + 1` when `l'` follows the local component, `m.c + 1` when it follows the message, else `0`.
- Two invariants fall out and are assertable in tests: `l ≥ now` always, and `l − now` is bounded by the largest peer clock offset — the stamp never runs slower than the wall and runs ahead only by observed skew.
- One primitive per suite: each process holds exactly one stamp cell, advanced atomically (a compare-and-swap loop over the packed word — cell mechanics compose the established state-cell law; the stamp law is what the cell holds).
- The atomic advance also serializes intra-process concurrency: two threads issuing simultaneously cannot receive equal stamps from one cell, so within an origin the stamp is already a strict total order — the origin tiebreak exists only for the cross-origin case.
- Every signal, receipt, op-log entry, and manifest in the process stamps from that one cell; two cells in one process is the named defect — each is internally monotone, their merged output is not, and the defect is invisible until a consumer sorts.
- The stamp rides baggage, not tracestate, by a survivability argument: tracestate survives only trace-aware hops and is vendor-namespaced under tight caps; baggage survives any propagator-honoring hop and is suite-keyed — evidence seams are not always trace-aware, so the stamp channel must not depend on trace participation.
- Stamps travel in the correlation envelope: one baggage/header key carries the packed stamp on every hop, and the receiver's extract performs the receive-advance before any stamped work.
- The receive-advance executes at the transport admission seam — once per transport adapter, where extraction already happens — never inside handlers; per-handler advancing both duplicates work and lets an un-advanced code path slip through.
- A hop that forwards context without advancing turns the envelope into a vector for stale time; send stamps after local advance — the stamp on the wire is the sender's post-advance value.
- Skew-as-evidence, split from ordering: the receive rule consumes `m.l` unconditionally — causal order never depends on whether skew is acceptable.
- Separately, `m.l − now` at receive is a measured peer-offset observation; when it exceeds the declared bound, the event becomes a typed skew fact (peer origin, observed offset, bound) folded into the evidence stream — and the stamp still advances.
- Rejecting messages on skew is a policy a consumer may layer; the envelope itself only measures. This split is the settled altitude: ordering algebra lives here; last-writer-wins adjudication over already-stamped versions belongs to the durability layer that composes stamps.
- Skew facts feed health as a governed signal — clock-skew dashboards become a fold over the skew-fact stream rather than inference from timestamp anomalies.
- Serialization precision is part of the contract: the packed word serializes fixed-width big-endian (16 hex chars) so lexicographic order equals numeric order in any store or log line.
- Rejected serializations, each with its failure: floating-point epochs (precision loss reorders), locale-formatted times (unsortable), split `(l, c)` text fields (field-wise sort breaks at the boundary), raw 64-bit integers in JSON (exceeds the interoperable safe-integer range — serialize as string).
- Sub-millisecond wall precision is deliberately forfeited; intra-millisecond order is the counter's job, and widening `l` to microseconds spends tiebreak bits on false precision.
- Counter saturation policy is declared, not improvised: at counter exhaustion within one millisecond the cell either spins to the next millisecond or the suite widened the field at design time — silent wraparound is the one unrecoverable corruption, producing order inversion with valid-looking stamps.
- Duplicate delivery is safe without being idempotent: re-receiving a stamp advances the counter again — order never regresses, the cost is counter burn; deduplication is the transport's concern, the envelope only guarantees monotonicity under it.
- The envelope's law matrix is property-testable: monotonicity under arbitrary interleavings of local and receive events, receive-dominance (output ≥ every input), and pack/unpack round-trip identity — three properties cover the algebra, and the third catches serialization edits the first two cannot see.
- The physical component reads from the suite's one clock seam (composed, not re-taught here); a stamp cell reading a second clock source re-introduces the dual-clock drift the envelope exists to absorb.
- Restart continuity is part of the envelope law: a rebooted process must not issue stamps below its own previously issued maximum — either the high-water mark persists and rehydrates the cell, or the origin id incorporates a boot epoch so the new sequence is a new origin; silently restarting at wall time risks reissuing the past when the process previously ran ahead under skew.
- High-water persistence has a cadence/risk trade stated once: flushing the mark per issued stamp is overkill, flushing per coarse checkpoint bounds the crash window to (checkpoint interval + observed run-ahead) — the boot-epoch-in-origin alternative deletes the persistence requirement entirely at the cost of widening the origin space, and small suites should take that trade.
- The design rationale is a two-axis dominance over the alternatives and worth recording as the answer to "why not simpler": pure logical counters order events but drift unboundedly from wall time (no temporal queries); vector clocks capture full causality but cost a slot per participant (unbounded width in open suites); the hybrid keeps O(1) width, bounded wall divergence, and happened-before consistency — the only property surrendered is detecting concurrency, which the total order deliberately erases.
- Stamp comparison is the ONLY ordering verb consumers get: exposing the raw physical component invites wall-clock comparisons that the skew bound makes wrong by up to ε — the envelope type hides `l` and `c` behind `CompareTo` precisely to make the wrong comparison unwritable.
- Skew-fact retention is a policy row of its own: per-peer offset observations are a bounded-cardinality stream (one row per origin pair), foldable to current-offset and worst-offset projections — the fold, not the raw stream, is what health consumes.
- The skew bound ε is not a constant to guess: it is derivable from the skew-fact stream itself as the observed worst offset plus margin, which turns the bound from a deployment superstition into a measured, revisable policy value.
- Stamps order, they do not identify: equal stamps cannot occur within one origin but the stamp alone is not a unique key across origins — anything using stamps as identifiers keys on (stamp, origin), and the packed-word-plus-origin pair is the suite's canonical event identity where one is needed.
- Stamp coverage is a completeness law, not an option: an unstamped fact in a stamped suite sorts nowhere — it cannot participate in merge, LWW, or causal views — so the envelope stamps at the evidence-emission seam where every fact already passes, not per producer discretion.
- The consumer-side merge is mechanical once coverage holds: k-way merge by stamp with origin tiebreak reconstructs one suite-wide causal timeline from per-process streams — no collector clock, no reconciliation pass, no heuristic windowing.
- The bound ε is dominated by clock-sync quality among peers — well-synced fleets keep stamps within wall-clock intuition while unsynced peers stretch `l` ahead of the wall; both regimes order correctly, only temporal-query interpretation changes, and the skew-fact stream is what tells the operator which regime holds.

## cross-signal-join-table

- Each signal kind carries correlation differently, and the join table is worth stating once because every operational query composes it: log records carry trace and span ids as typed fields plus the root id as an enriched tag; spans carry the full context natively; metrics carry no per-point correlation tags at all — their join to traces is exemplars, and their join to a process is resource identity.
- The metrics row is the one engineers get wrong: stamping correlation ids as measurement tags is a cardinality explosion (one series per request), which is why exemplars exist — the join rides the exemplar's trace reference, not a tag.
- The correlation root rides two transports deliberately: resource identity travels with exported telemetry (collector-side join, zero per-record cost) while the baggage key travels with requests (in-band join available to every hop) — one id, two declared transports, and dropping either halves the joinable surface.
- Span start and end times remain wall-clock and are NOT ordered evidence across machines: a cross-process waterfall can show negative gaps up to the skew bound, and that UI artifact is expected — the stamp channel is the ordering truth, the span clock is duration truth; views that mix the two channels inherit both error modes.
- Receipts and operational facts join through the stamped envelope: correlation root plus causal stamp, which makes the evidence stream sortable against log and span timelines without sharing their identifiers.
- The frozen latency record joins through its tag row — it carries no native trace identity, so the composition law is that one registered tag carries the root or trace id, declared with the vocabulary.
- The table's completeness check: for every signal surface a process exports, name the field or mechanism answering each of the four identity questions; a surface with an unanswerable question is the gap an incident will find first.
- The join table is testable end-to-end: one synthetic request through a process should produce a log record, a span, and an evidence fact that join on every declared layer — the single-request join test is the cheapest spanning proof the correlation layer has, and it catches enricher, propagator, and stamp regressions in one assertion.
- Tag namespaces stay partitioned across signal kinds: latency tag names, log enrichment tags, and span tag vocabularies are three registries — one shared registry owner may hold them, but a name's meaning never silently spans kinds, because per-kind semantics (sampled, frozen, per-record) differ.

## enrichment-governance-rows

- Enrichment is declared as typed rows at the root, split by cost class: `ILogEnricher` rows run per record; `IStaticLogEnricher` rows run once per provider and their tags ride every record for free.
- Anything process-constant — correlation root, build, environment, region — is static by law; a constant in a per-record enricher is pure waste, a per-request value in a static row is a correctness bug.
- Static rows evaluate once per provider, so with multiple providers the value must be provider-independent — a static enricher reading provider-relative state stamps different providers with different "constants".
- Registration is `AddLogEnricher<T>` / `AddStaticLogEnricher<T>` (generic or instance); collection is `IEnrichmentTagCollector`; activation is `EnableEnrichment` on the logging builder — enrichment without the enable call is registered but inert.
- An enricher is a pure projection from ambient state to bounded tags: it never throws into the pipeline, never blocks on I/O, and never smuggles domain payloads — the row declares dimensions, and a dimension that needs a network call is a design error at the row, not a caching problem at the enricher.
- Enricher tag names share the record's flat tag namespace with generated-emission tags — the namespace is one vocabulary, so enricher rows reserve their prefixes in the same registry that owns event tag names; collisions overwrite silently in flat-map sinks.
- The stock application enricher row carries deliberate defaults: environment name on, application name on, deployment ring off, build version off — the off half is off because ring and version multiply tag cardinality fleet-wide; flipping them is a recorded options decision.
- The stock process enricher row: process id on, thread id off — thread id at enrichment time is not the thread that did the work under async flow; its default-off is a correctness stance, not economy.
- The application enricher's registration verb has an obsolete predecessor (service-enricher naming) — compositions still calling the old verb are emitting the same dimensions under a deprecated registration; migrate the row, not the tags.
- Exception detail is governed enrichment, not formatter behavior: `LoggerEnrichmentOptions.CaptureStackTraces` emits a `stackTrace` tag; `IncludeExceptionMessage` folds messages into traces; `MaxStackTraceLength` defaults 4096 and validates to [2 KB, 32 KB] — out-of-range throws at options validation, not at log time.
- Captured traces chain inner exceptions into the one tag — the truncation budget covers the whole chain, so deeply nested exception wrappers consume it fastest and are the first casualty of a tight length row.
- `UseFileInfoForStackTraces` consults debugging symbols for richer frames at substantial per-exception cost — a development row only.
- Stack capture is a volume decision: at error-burst rates the truncation length decides whether an incident's logs fit the export budget.
- Outgoing-dependency metadata is its own governance row family: request route metadata and downstream-dependency declarations (`RequestMetadata`, ambient request context, dependency metadata contracts) classify outbound calls for telemetry without leaking raw URLs — boundary material, never domain state.
- Per-signal enrichment law: logs enrich via the row system; spans enrich via processor hooks and options-bound enrich delegates at the instrumented seam; metrics never enrich per-point — static dimensions live on meter tags, dynamic dimensions are measurement tags governed by view projection.
- One fact therefore has one enrichment seat per signal kind, and "add it everywhere" means three declared rows, not one helper that scatters.

## latency-context

- The latency context is a request-scoped measurement envelope with pre-registered vocabulary: checkpoint, measure, and tag names register at composition (`RegisterCheckpointNames` / `RegisterMeasureNames` / `RegisterTagNames`).
- Runtime code resolves tokens (`CheckpointToken` / `MeasureToken` / `TagToken`) through the issuer and records exclusively via token — string names at record time are unregistered by definition.
- `ThrowOnUnregisteredNames` defaults to no-op: an unregistered token records nothing, silently — flip it on under test, leave it off in production; the silent default is a data-loss trap during refactors that rename vocabulary without re-registering.
- The recording algebra is small and closed: `AddCheckpoint(token)` marks a point-in-flow; `AddMeasure(token, value)` accumulates; `RecordMeasure(token, value)` sets; `SetTag(token, value)` dimensions the context; `Freeze()` seals for export — post-freeze recording is inert.
- Exporters (`ILatencyDataExporter`; the console row carries per-kind output toggles for checkpoints, measures, tags) receive the sealed `LatencyData` spans.
- Exporters fan in over the one frozen record — multiple exporter rows consume the same sealed data, so adding a destination is a registration, not a second measurement pass.
- The registered name set is the closed measurement vocabulary: renaming a checkpoint is a registry edit plus token re-resolution, and the registry is the diff surface for "what do we measure" exactly as the instrument registry is for metrics.
- The context is an explicit handle, not ambient — measured work receives it where it flows, which is exactly why it survives executor hops and suppressed-context callbacks that break ambient correlation; explicitness here is the feature, not a missing convenience.
- Seat the latency context as the in-flight hot-path ledger distinct from spans: checkpoints are cheaper than child spans, carry no sampling coupling, and export as one record per scope rather than a span tree.
- Use it where span-per-step would distort the measured path; fold its exported data into the evidence stream keyed by the correlation envelope — the context carries no trace identity of its own, so the tag row carrying the root or trace id is the join.
- The null-context row (`AddNullLatencyContext` / the null context object) is the absent-measurement composition for processes that wire latency-aware components without paying for measurement.
- The provider/issuer split is the lifecycle law: the context provider creates one context per measured scope, the token issuer resolves names to tokens once — token resolution is composition-time work, context creation is per-scope work; inverting them (resolving tokens per scope) reintroduces the string-lookup cost the tokens exist to delete.
- Checkpoint sequences double as in-flight phase evidence: because checkpoints are cheap and ordered, the frozen record is a phase histogram source — per-phase duration views derive from checkpoint deltas at export, never from per-phase measures recorded redundantly.

## divergent

- Correlation-propagation maximal form: the suite's entire cross-process context is one declared envelope schema — a versioned set of baggage keys (correlation root, stamp, tenant-grade routing facts), each with an owner, a classification verdict (identifiers only), and a size budget against the byte/item ceiling — plus one propagator convention asserted at both seams.
- New context is a schema row; ad-hoc baggage writes outside the envelope owner are the rejected form — they bypass classification review and erode the byte budget invisibly.
- The envelope owner exposes typed accessors over the slots; consumers never touch string keys — the same one-hop-resolution law the rest of the corpus applies to names applies to context keys.
- HLC-stamp maximal form: one stamp type owns pack/unpack, advance (local and receive), comparison, serialization, and skew measurement as a closed algebra — consumers see `Now()`, `Receive(stamp)`, `CompareTo`, and a skew-fact stream, nothing else.
- Every other ordering construct in the suite — op-log sequence, receipt order, event versioning — composes this one type; a second timestamp-based ordering primitive anywhere is the collapse trigger.
- The growth axes are origin count and event rate, both absorbed by packing parameters declared once; nothing about adding a process to the suite touches the stamp type.
- Envelope/stamp interlock: the stamp riding the correlation envelope means causal order is recoverable across every signal of every process from exported data alone — log lines, spans, and op-log entries from different machines merge-sort by stamp without collector-side clock reconciliation.
- The page-level payoff line: ordering is producer-stamped evidence, never consumer-inferred — that single sentence is the design boundary between this architecture and timestamp-trusting telemetry.
- Enrichment-budget arithmetic: every per-record tag costs at every volume tier (sampling keeps records post-enrichment), so an enricher's governance row includes its byte cost × kept-record rate.
- The static/per-record split, stack-trace truncation, and envelope size budget compose into one declarable telemetry-weight budget per process — overweight is detectable at review time from the rows alone, which is the point of rows.
- Dual-seam audit as a boot proof: assert at startup that the platform propagator and SDK propagator encode the same convention, that the envelope's keys round-trip one inject/extract cycle, and that a stamp survives the loop with a correct receive-advance — three cheap assertions that convert the three classic silent propagation failures into startup faults.
- Envelope versioning is the forgotten growth axis: baggage keys carry an envelope version key so a mixed-version fleet can detect schema drift at extract — a receiver seeing a newer envelope version records a typed compatibility fact instead of misreading a renamed key as absence.
- The latency context and the stamp envelope answer different "when" questions and must not merge: the context measures durations within one scope (relative time, wall-clock honest), the stamp orders events across scopes (causal time, wall-clock skew-tolerant) — a design that derives durations from stamp differences inherits ε of error per endpoint and is wrong by construction.
