# Aspect Architecture

[DEFINITION_TIME_WEAVING]:
- The attribute on a `partial` type declaration is the weave point; the generator emits a second partial file containing all cross-cutting members. No runtime proxy, no IL rewriting, no container interception.
- `[SmartEnum<TKey>]`, `[Union]`, `[ValueObject<T>]`, and `[ComplexValueObject]` declare the generated surface at definition time; the emitted partial file carries equality, conversion operators, `Switch`/`Map` dispatch, `ISmartEnum<TKey,T,TValidationError>`, `IParsable<T>`, `ISpanParsable<T>`, `IFormattable`, arithmetic operator interfaces, and serialization converters — all from a single declaration attribute.
- `SwitchMapMethodsGeneration.DefaultWithPartialOverloads` causes the generator to emit partial overloads of `Switch`/`Map` alongside the full-coverage overloads, giving the hand-written half a seam to override specific arms while generated dispatch covers the rest.
- `SwitchMapStateParameterName` (default `"state"`) on `[SmartEnum]`, `[SmartEnum<TKey>]`, and `[Union]` renames the thread-through parameter in every generated `Switch`/`Map` overload — aspect payload naming is owned entirely by the declaration site, not by downstream call sites.
- `NestedUnionParameterNameGeneration.Simple` vs. `Default` on `[Union]` controls whether intermediate type names appear in Switch/Map parameter names for hierarchical unions; `Default` produces `failureNotFound`, `Simple` produces `notFound`. The choice is load-bearing for callers: `Simple` insulates call sites from intermediate type renames, while `Default` guarantees uniqueness when leaf names collide across union families. Once `Simple` is set with stable leaf names, renaming an intermediate type does not break call sites; renaming any intermediate type under `Default` changes every compound parameter name it prefixes.
- `[UnionSwitchMapOverloadAttribute]` (`AllowMultiple = true`) on a Regular Union class declares additional `Switch`/`Map` overload families treating a specified set of types as opaque arms rather than descending into their subtypes. `StopAt = new[] { typeof(FailureCase) }` generates a dispatch overload treating `FailureCase` and its siblings as atomic arms — coarser-grained dispatch alongside the full-resolution overload. Both overloads coexist on the same union type; this attribute applies only to Regular Unions.
- `[ObjectFactory<T>]` (a generic attribute) declares an additional factory route at definition time; `UseForSerialization`, `UseWithEntityFramework`, and `UseForModelBinding` each steer the corresponding cross-cutting integration aspect to route through that factory path rather than the primary key-based factory. `UseForSerialization` is typed as `SerializationFrameworks` (a `[Flags]` enum), so a single type can carry multiple `[ObjectFactory<T>]` attributes routing each serialization framework through a different factory — for example, STJ through a span factory while Newtonsoft routes through the primary string-based factory. `UseForModelBinding` and `HasCorrespondingConstructor` are `init`-only properties, enforcing immutability of the aspect payload at the declaration site.
- `FactoryMethodGeneration` on the generic `[Union<T1,T2>]` and `[AdHocUnion]` forms is inherited from `UnionAttributeBase`. Regular `[Union]` derives directly from `Attribute` and carries only `SwitchMethods`, `MapMethods`, `ConversionFromValue`, `SwitchMapStateParameterName`, and `NestedUnionParameterNames`; `FactoryMethodGeneration` is absent from it. On `UnionAttributeBase`-derived forms, `Default` derives factory-method presence from type topology (triggers: type-parameter members, duplicate types, `object`, interface arms); `Always` forces generation regardless of triggers; `None` suppresses unconditionally.
- `UseSingleBackingField = true` on `UnionAttributeBase`-derived union forms collapses all per-case backing fields into one `object` field. Field-per-case stores value types in their native representation (no boxing); a single backing field always boxes structs. For unions whose cases are all reference types this is an unambiguous memory-layout improvement; for unions with struct cases it introduces per-access boxing. The decision is driven by case type topology at definition time.
- `ConstructorAccessModifier = UnionConstructorAccessModifier.Private` on `UnionAttributeBase`-derived union forms makes both the generated constructors and the generated implicit conversion operators private simultaneously — the coupling is load-bearing. The only externally visible factory surface is then hand-written static factory methods on the union type; no caller can construct a case by direct implicit assignment. This is the definition-time enforcement mechanism for admission-controlled unions where construction must pass validation or enrichment before the case is materialized.
- `T1IsStateless = true` on `[Union<T1,T2>]` instructs the generator to store only the discriminator index for that arm — no field is allocated, and accessors like `AsT1` return `default(T1)`. This eliminates the case field entirely for marker or signal cases. The generator recommends struct types for stateless members to eliminate null-handling at the accessor.
- `ConversionFromValue` and `ConversionToValue` are independent axes on `UnionAttributeBase`-derived forms. A union can admit values implicitly while requiring an explicit cast to extract them, or vice versa. Regular `[Union]` carries only `ConversionFromValue`; `ConversionToValue` is absent from it. Decoupling admission from extraction is the definition-time aspect that enforces directional policy — inbound permissiveness does not imply outbound permissiveness.
- `UnsafeConversionToKeyMemberType` on `[ValueObject<T>]` is an independent axis from `ConversionToKeyMemberType`. "Unsafe" means the conversion can throw — converting back to the key type when the key type admits values the value object's validation would reject. Setting one axis to `None` does not suppress the other.
- `AllowDefaultStructs = true` suppresses the generated guard that throws on `default(T)` or parameterless construction. The flag has no effect on class value objects. `DefaultInstancePropertyName` overrides the generated static property name for the `default` instance from `"Empty"` to any valid identifier and exists only when `AllowDefaultStructs = true`.
- `SerializationFrameworks` on `[ValueObject<T>]` and `[SmartEnum<TKey>]` is a `[Flags]` enum (`SystemTextJson = 1, NewtonsoftJson = 2, MessagePack = 4`, `Json = 3`, `All = 7`). Setting it to `None = 0` suppresses all serialization aspects without suppressing the factory methods. The `[Flags]` nature enables selective suppression: `SerializationFrameworks.Json` generates STJ and Newtonsoft converters but not MessagePack formatters. Absent companion packages cause the generator to skip that framework aspect silently.
- `[IgnoreMemberAttribute]` marks a property or field invisible to the source generator, excluding it from equality, hash, serialization, comparison, and Switch/Map generation. The ignored member must be immutable — a mutable ignored member breaks the value-object contract: two instances with equal keys but different mutable-ignored-field values would compare equal while being observably different.
- `SkipKeyMember = true` on `[ValueObject<TKey>]` suppresses generation of the key member, requiring a hand-written member at the name given by `KeyMemberName`. The generator still routes serialization, comparison, and parsing aspects through the declared key name — the use case is value objects with a custom backing representation where the generated backing field would be incorrect but all generated aspects must still route through the key name.
- `CreateFactoryMethodName` and `TryCreateFactoryMethodName` on `ValueObjectAttributeBase` override the generated factory method names from `Create`/`TryCreate` to any valid identifier. The full generated factory body — including `ValidateFactoryArguments` hook wiring, null/empty yield-null handling, and serialization routing — is preserved under the new name.

[SMART_ENUM_KEY_SURFACE_ASPECTS]:
- `KeyMemberKind = MemberKind.Field` changes the generated key member from a public property to a field. Combined with `KeyMemberAccessModifier = AccessModifier.Private`, this generates a private backing field named `_key` that carries the discriminant without exposing it as public surface. The enum's behavioral identity is then entirely through `Switch`/`Map` and delegate-backed partial methods, not through key inspection.
- `DisableSpanBasedJsonConversion = true` on `[SmartEnum<string>]` falls back from the zero-allocation `ReadOnlySpan<char>`-based JSON reader to a string-based path. The span path uses the generated `IObjectFactory<Owner, ReadOnlySpan<char>, TError>` interface, avoiding heap allocation for the transient key during deserialization. Enabling this flag is a signal that the serialization context is not driving the span reader, and the zero-allocation admission guarantee is silently lost.
- `SkipIParsable = true` forces `SkipISpanParsable` to `true` simultaneously because `ISpanParsable<T>` inherits from `IParsable<T>`. Selectively suppressing `IParsable` while retaining `ISpanParsable` is impossible — the BCL inheritance chain makes the skip cascade. Any grammar or text-protocol aspect relying on `ISpanParsable` is disabled by the same flag that disables string-based parsing.

[CONSTRUCTION_ADVICE_SEAMS]:
- `static partial void ValidateFactoryArguments(ref ValidationError? validationError, ref TKey value)` on `[ValueObject<T>]` and `[ComplexValueObject]` is the primary construction advice point. Both `ref` annotations are load-bearing: `ref validationError` sets the error without extra allocation, and `ref value` normalizes the raw input (casing, trimming, clamping) before the generated factory stores it. The hook executes inside the generator-produced `Create`/`TryCreate` body, before the type is instantiated.
- The validation error type is a typed aspect payload: `IValidationError<T>` requires `static abstract T Create(string message)`, constraining the error type as a static-abstract factory rather than a base class. A custom type implementing `IValidationError<TCustom>` where `TCustom : class, IValidationError<TCustom>` replaces the default `ValidationError` across the entire generated factory surface simultaneously by naming it in the `[ValueObject<T>]` declaration.
- `[UseDelegateFromConstructor]` on a `partial` method inside a `[SmartEnum]` class is a per-method aspect payload: the generator synthesizes a private delegate field with the method's matching signature, wires it through the constructor, and implements the `partial` method body as a delegate invocation. `ref` or `out` parameters already force a custom delegate type rather than `Func`/`Action`; the `DelegateName` property provides a custom name for that delegate type and is generated "regardless of parameter ref-kinds" — it is a naming control, not a representability workaround.
- `[ObjectFactory<T>]` with `UseForSerialization` steers the generated serialization converters to route through the custom `IObjectFactory<Owner, T, TError>` implementation's `Validate(TValue value, IFormatProvider provider, out T result)` method. The `out` parameter and `IFormatProvider` argument are load-bearing for the STJ span-reader path. A type implementing only `IObjectFactory<T>` (the one-type-parameter marker interface) satisfies the runtime but is not wired by the generator for serialization.
- `HasCorrespondingConstructor = true` on `[ObjectFactory(typeof(T))]` informs the generator that a constructor taking a single `T` parameter exists, enabling the generator to emit a conversion expression that routes through the constructor rather than only through the static factory. This is the construction-advice aspect for EF Core value converters: EF materializes instances via constructor expression trees, and the generator wires the constructor path only when this flag is explicitly set.
- Partial constructors split a constructor into declaring and implementing parts. Only the implementing declaration may carry a constructor initializer (`: this(...)` or `: base(...)`), enforced by `ERR_PartialConstructorInitializer` (9280). Partial constructors are the correct seam for source-generated initialization chains where the generator controls parameter admission and the hand-written half controls delegation to base.
- A partial property's definition part is exactly auto-property-shaped (semicolon accessors only); the implementing part carries the accessor bodies. Providing an auto-property body in the implementing part is what the compiler parses as a second defining declaration and flags with `ERR_PartialPropertyDuplicateDefinition` (9250). Any accessor declared in the definition part must be matched in the implementing part (`ERR_PartialPropertyMissingAccessor`, 9252).
- Partial events may not carry an initializer on either part, enforced by `ERR_PartialEventInitializer` (9279). Partial events are the generated/hand-written seam for event accessor implementation: the generator emits the definition part declaring `add`/`remove`, the hand-written half implements the subscription logic. Unlike partial methods, the implementing part is mandatory — a partial event without an implementing declaration is a compile error (`ERR_PartialMemberMissingImplementation`, 9275).
- A partial method with a non-void return type requires an access modifier (`ERR_PartialMethodWithNonVoidReturnMustHaveAccessMods`, 8796); `out` parameters carry a separate enforcement via `ERR_PartialMethodWithOutParamMustHaveAccessMods` (8797); `virtual`/`override` and other extended modifiers carry `ERR_PartialMethodWithExtendedModMustHaveAccessMods` (8798). All three are independently diagnosed — correcting only one does not suppress the others. Any accessible partial method must have an implementing part; legacy implicit-void partial methods without an access modifier may omit the implementing declaration and are silently dropped at the call site. Definition-time advice points for generators must use extended partial methods (with access modifier) when a non-void return or guaranteed execution is required.

[GENERIC_ATTRIBUTES_AS_TYPED_ASPECT_PAYLOAD]:
- `[SmartEnum<TKey>]` and `[ObjectFactory<T>]` are generic attributes. The type argument is a verified compile-time type rather than a `Type` parameter, eliminating `typeof(T)` and enabling analyzer-side type checks in the source generator without runtime reflection.
- `[ObjectFactory<T>]` where `T` is `ReadOnlySpan<char>` routes the factory through span-based parsing — the zero-allocation admission path. The generator emits `IObjectFactory<Owner, ReadOnlySpan<char>, TError>`; naming a non-span `T` produces a different generated interface. The `where T : allows ref struct` anti-constraint on `ObjectFactoryAttribute<T>` enables this: it permits `T` to be a ref struct without requiring it. Any custom `ref struct` input type is therefore also a valid factory discriminant.
- `[KeyMemberComparerAttribute<TComparer, TKey>]` and `[KeyMemberEqualityComparerAttribute<TComparer, TKey>]` attach a comparer type to the key member at definition time; `[MemberEqualityComparerAttribute<TComparer, TMember>]` attaches to individual properties or fields of a `[ComplexValueObject]`. All three require `TComparer` to implement `IComparerAccessor<T>` or `IEqualityComparerAccessor<T>`, providing a type-safe comparer injection path without runtime reflection or constructor injection. `DefaultStringComparison` on `[ComplexValueObject]` sets the comparison mode for all `string`-typed members without per-member annotation, serving as a type-level default-comparison aspect.

[SERIALIZATION_GRAMMAR_IDENTITY_ASPECTS]:
- Serialization is a definition-time aspect: `System.Text.Json`, Newtonsoft.Json, MessagePack, EF Core value converters, ASP.NET Core model binding, and Swashbuckle schema filters are generated as partial class members from the declaration attribute, conditioned by the presence of their respective companion packages. No runtime registration, no DI setup — converters are emitted into the type itself and discovered by framework reflection or source-generated registration.
- `ConversionOperatorsGeneration.Implicit` vs. `Explicit` vs. `None` on `[ValueObject<T>]` and `[SmartEnum<TKey>]` governs the generated `op_Implicit`/`op_Explicit` operators — the identity-at-boundary aspect. Implicit conversion makes the raw key type and the value-object type structurally interchangeable at call sites; explicit forces a cast and keeps the domain boundary visible.
- `OperatorsGeneration.DefaultWithKeyTypeOverloads` generates equality/comparison overloads that accept the raw key type alongside the value-object type. It allows comparisons between the primitive and the domain type without wrap/unwrap, at the cost of admitting unvalidated primitives into comparisons. Use `Default` (domain-type-only overloads) unless interop with unvalidated key values is explicitly required.
- `NullInFactoryMethodsYieldsNull` and `EmptyStringInFactoryMethodsYieldsNull` on `[ValueObject<string>]` convert null or empty-string input into `null` output from `Create`/`TryCreate` rather than a validation error. Setting `EmptyStringInFactoryMethodsYieldsNull = true` also forces `NullInFactoryMethodsYieldsNull = true` — the cascade is one-directional.

[EXTENSION_BLOCKS_AS_HKT_OPERATOR_SURFACES]:
- Extension blocks are the mechanism that makes constrained operators on higher-kinded types compilable. The syntax `extension<E, F, A>(K<F, A> _) where F : Fallible<E, F>` declares an extension over `K<F, A>` with generic constraints, enabling `operator |` to be defined directly on the carrier type with trait constraints — operator parameters are not required to include the declaring type. The `$T0`, `$T1` placeholders in decompiled IL output are the representation of extension block type parameters.
- `MonadUnliftIOExtensions` defines the entire retry/repeat/bracket/timeout/post/uninterruptible surface in a single extension block `extension<M, A>(K<M, A> ma) where M : MonadUnliftIO<M>`, exposing `LocalIO`, `UninterruptibleIO`, `PostIO`, `TimeoutIO`, `BracketIO` (three overloads), `RepeatIO`, `RepeatWhileIO`, `RepeatUntilIO`, `RetryIO`, `RetryWhileIO`, and `RetryUntilIO` as receiver-syntax methods. The `FoldIO`/`FoldWhileIO`/`FoldUntilIO` family remains as classical `this K<M, A>` static extension methods because their additional `S` state and predicate overloads prevent uniform representation inside one extension block constraint.
- The extension block form is load-bearing for constrained operator definition: a domain operator aspect constrained to a trait — a `|` catch combinator, a `|` finally operator, or any binary operator keyed to `Fallible<E, F>` or `Final<F>` — requires an extension block. Any local aspect that introduces a constrained operator on `K<F, A>` must use this form.

[COMPOSITION_TIME_POLICY]:
- Admission, identity, dispatch, serialization, and grammar attach at definition time because they are properties of the type itself — present at every call site, represented in the generated interface surface. Retry, recovery, and resource lifetime attach at composition time because they are properties of a specific call site — the same `IO<A>` can be retried with different schedules, and the type carries no trace of the policy.
- `Schedule` is a lazy stream of `Duration` values. `|` is Union (minimum delay per step, length of the longer schedule); `&` is Intersect (maximum delay per step, length of the shorter); `+`/`Combine` is sequential append (all steps of the left, then all steps of the right). These are three algebraically distinct binary operations: `exponential(50*ms) | recurs(5)` terminates after 5 steps at exponential delays, while `exponential(50*ms) + recurs(5)` runs the exponential to exhaustion and then appends 5 zero-delay steps. `Schedule` is a `Semigroup<Schedule>` where `+` is `Combine`. On `ScheduleTransformer`, `+` is left-to-right function composition (`g.Apply(f.Apply(x))`), not schedule concatenation; a `ScheduleTransformer` implicitly converts to `Schedule` via `Schedule.Forever | t`.
- `Schedule.Interleave(b)` alternates steps from two schedules — a, b, a, b — for the length of the longer. `Schedule.Bind(Func<Duration, Schedule> f)` and `Schedule.SelectMany` make `Schedule` a monad; `intersperse(schedule)` implements as `Transform(s => s.Bind(schedule.Prepend))`, inserting the entire interspersal schedule between each step. `Interleave` alternates individual steps; `intersperse` inserts a full sub-schedule between each step — categorically different combinators.
- `Schedule.decorrelate(factor, seed)` introduces both upward and downward jitter. For a linear schedule `(100, 200, 300, ...)`, decorrelation produces roughly `(103.2, 96.8, 202.3, 197.7, ...)` — the effective step count approximately doubles because each step expands into two. Its purpose is reducing thundering-herd re-collision among parallel instances of the same computation, not delay expansion.
- `Schedule.fixedInterval(d)` guarantees a gap of `d` between action completions — if the action takes longer than `d`, the next gap is zero rather than accumulating a deficit. `Schedule.windowed(d)` aligns to window boundaries anchored to the moment `Run()` is first called: each emitted duration is `d - (currentTime - startTime) % d`, so windows are relative to the schedule's own start time, not clock-epoch multiples. `Schedule.upto(max, currentTimeFn)` terminates after wall-clock elapsed time exceeds `max` from the moment `Run()` is first called; the optional clock function is the injection point for deterministic testing. All three accept or default to a `Func<DateTime>` clock. `secondOfMinute(int)`, `minuteOfHour(int)`, `hourOfDay(int)`, and `dayOfWeek(DayOfWeek)` generate window-aligned calendar schedules, each computing the delay to the next occurrence of the specified calendar slot; all four accept an optional clock injection parameter and compose via `|`/`&`/`+`.
- `Schedule.resetAfter(max)` resets once cumulative delay reaches `max`, restarting from zero indefinitely. `Schedule.maxCumulativeDelay(max)` terminates when the cumulative total exceeds `max`. `Schedule.maxDelay(max)` clamps each individual step. `upto` terminates on wall-clock elapsed; `maxCumulativeDelay` terminates on sum of emitted durations — in computations with variable action time these diverge significantly. `Schedule.NoDelayOnFirst` is a static `ScheduleTransformer` implemented as `Transform(s => s.Tail.Prepend(Duration.Zero))`; `RetryUntil`/`RepeatUntil` call `schedule.PrependZero.Run().GetIterator()` unconditionally, so the zero-delay first retry is implicit — composing `NoDelayOnFirst` on a schedule already passed to `RetryUntil` is idempotent and produces a double-zero prefix, not an extra retry.
- `retry` and `repeat` are semantically antipodal: `retry` fires on failure (re-invokes when the computation produces an error), `repeat` fires on success (re-invokes when it produces a value). `retryWhile`/`retryUntil` predicate on `Error`; `repeatWhile`/`repeatUntil` predicate on the bound value `A`. Both accept `K<F, A>`, so mixing them is not a type error — the distinction is purely semantic. `retryWhile(Schedule, IO<A>, Func<Error,bool>)` terminates the loop when the predicate returns `false` regardless of remaining schedule steps; `retryUntil` inverts polarity.
- `CatchM<E, F, A>` is a predicate–handler pair. `@catch(Error, Func<Error, K<F, A>>)` constructs a `CatchM` keyed to a specific `Error` value (equality-matched); `@catch(Func<Error, bool>, ...)` is the predicate form. The `|` operator on `K<F, A>` with a right-hand `CatchM` is defined in `FallibleExtensions` via an extension block constrained to `F : Fallible<E, F>` — the operator is not available on types that do not implement that trait. `catchOf<E, M, A>` requires `where E : Error, M : Fallible<Error, M>, Monad<M>`; for a multi-error aggregate it routes through `ForAllM` (the `Monad<M>`-constrained traversal that Bind-chains each matched `E` sub-error), short-circuiting at the first failure — correct for `IO`/`Eff`. `catchOfFold<E, M, A>` additionally requires `MonoidK<M>` and uses `FoldM` — monoidal `Combine`, accumulating results. The type constraints enforce the distinction: `catchOf` does not compile without `Monad<M>`, `catchOfFold` does not compile without `MonoidK<M>`.
- `K<F, A> | final(K<F, B>)` executes `K<F, B>` regardless of success or failure and returns the primary result. `final` is the function name (`Prelude.final<F,X>(K<F,X>)` returning `Finally<F,X>`); `finally` is a C# reserved word and does not exist as a callable surface. The `|` operator is defined in `FinalExtensions` for `F : Final<F>`. `final` does not swallow errors — `IOFinal.Invoke` uses a C# `finally` block: the finalizer runs, and the originating exception re-propagates up the interpreter stack. A `@catch` composed after `final` in the chain (i.e., `computation | final(cleanup) | @catch(pred, handler)`) receives errors normally — `final` is a transparent pass-through for the error channel, not a terminal barrier. Position relative to `final` determines only whether cleanup runs before or after the catch handler, not whether the handler is reachable.
- `IO<A>.Bracket()` calls `WithEnv(env => env.LocalResources)` — a new `Resources` child scope propagating the existing `CancellationTokenSource` and `SyncContext`. `IO<A>.BracketFail()` calls `WithEnvFail(env => env.LocalResources)` — identical child scope creation, but disposal fires only on failure, allowing a successfully acquired resource to escape as the result. `WithEnv` and `WithEnvFail` are the protected virtual methods underlying all isolation aspects; the resource scope creation is identical, the disposal trigger is the sole difference. The three-argument `bracketIO(acq, use, fin)` is implemented as `acq.Bind(x => use(x).Finally(fin(x)))` — a manual bracket without automatic resource tracking.
- `IO<A>.Post()` marshals onto the `SynchronizationContext` captured in `env.SyncContext` when the `EnvIO` was constructed, not the thread's current context at the moment `Post()` executes. If `env.SyncContext` is `null`, `Post()` runs inline. Wrapping a computation with `Post()` when `EnvIO` was constructed on a UI thread marshals it back to that thread regardless of which thread ran preceding chain steps.
- `IO<A>.Uninterruptible()` wraps in `IOUninterruptible`, executing against `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)` — existing `Resources` are preserved, and `CancellationToken.None` causes `EnvIO.New` to create a fresh `CancellationTokenSource` with no registration to the parent token. The outer token's cancellation does not reach the wrapped computation; the error rail propagates normally. Only the `CancellationToken` path is severed.
- `IO<A>.Timeout(TimeSpan)` wraps in `IOTimeout` and creates a new `CancellationTokenSource(timeout)` linked to the parent via `LocalCancelWithTimeout`, preserving existing `Resources` and `SyncContext`. When the timeout fires, `Errors.TimedOut` (code `−2000000002`) propagates. `Errors.Cancelled` (code `−2000000001`) is what `IO.cancel` produces. `retryWhile` predicates that must distinguish timeout from explicit cancel must compare against these code constants.
- `IO<A>.Fork(Option<TimeSpan> timeout)` returns `IO<ForkIO<A>>` where `ForkIO<A>` is `readonly record struct(IO<Unit> Cancel, IO<A> Await)`. A child `Resources` scope (`new Resources(parentResources)`) is disposed when the fork completes. Resources acquired in the parent before forking are visible inside as shared parent-owned resources and are not released on fork completion; resources acquired inside the fork are owned by the fork's child scope and released on fork completion regardless of outcome. The fork's `CancellationTokenSource` is linked to `env.Token` via `env.Token.Register(tsrc.Cancel)`.
- `IO<A>.Local()` calls `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — it passes the existing `Resources` scope, not a new one. `EnvIO.New` with a null `source` creates a fresh linked `CancellationTokenSource`; when the incoming token can be cancelled, `env.Token.Register(src.Cancel)` links the scopes. Local cancellation via `IO.cancel` produces `Errors.Cancelled` that propagates upward past the local boundary — `Local()` does not catch cancellation errors. To contain a cancellation error and substitute a fallback value, compose `local(ma) | @catch(Errors.Cancelled, _ => fallback)` with the catch applied outside the `local` scope.
- `IO<A>.Try()` wraps the computation into `FinT<IO, A>` by mapping success through `Fin.Succ` and catching all errors through `Fin.Fail<A>`. The resulting `FinT<IO, A>` insulates the error rail inside the `IO` monad — `Catch` combinators applied to the outer `K<IO, Fin<A>>` computation see `Fin.Fail` values as bound values, not as propagated `Error` rails. Use `Try()` when both success and failure paths must be inspected as values rather than as rail continuations.
- `IO<A>.Combine(IO<A> rhs)` catches every error from `this` (predicate `_ => true`) and substitutes `rhs`. `IO<A>.Empty` is the monoid identity — `IOEmpty<A>.Invoke` throws `(Exception)Errors.None`, making `Errors.None` the identity error propagated by the empty computation. Catch predicates that distinguish empty from failure must compare against `Errors.None`. The semantics are alternative resolution: `first | second | third` via `|` on the `Fallible` extension block picks the first non-failing computation. Errors from `rhs` propagate normally.
- `IO<A>.Fold(schedule, initialState, folder)` re-invokes on success, accumulating state `S` via `folder(S, A) -> S`. `FoldWhile`/`FoldUntil` accept three termination predicate shapes: `Func<S, bool>` (state only), `Func<A, bool>` (value only), `Func<(S State, A Value), bool>` (combined). `Fold` without a schedule defaults to `Schedule.Forever`, producing an infinite accumulation loop.
- `Eff<A>.MapIO(Func<IO<A>, IO<B>>)` calls `effect.MapIO(f)` directly — it unwraps the inner `IO<A>`, applies `f`, and wraps the result back into `Eff<B>`. The static `Prelude.mapIO<M,A,B>(ma, f)` routes through `M.MapIO(ma, f)`, which for `Eff` delegates to `effect.MapIO(f)`. The lower-level `MapIOMaybe<M,A,B>(this K<M,A>, Func<IO<A>,IO<B>>)` is a public extension on any `K<M,A>` where `M : MonadIO<M>` (defined in `MaybeMonadUnliftIOExtensions`), with a trait-default implementation on the `Maybe.MonadIO` interface that transformers such as `ReaderT` implement by delegation — it is not restricted to `IdentityT<M>` composition. `MapIO` (the `MonadUnliftIO`-constrained form) is the correct mechanism when full unlifting is available; `MapIOMaybe` is the weaker form available to any monad that can lift IO but cannot fully unlift. The generic `retry<M, A>(Schedule, K<M, A>)` overloads operate on any `M : MonadUnliftIO<M>`; both `Eff<A>` and `Eff<RT, A>` satisfy this constraint, so retry/repeat aspects compose uniformly without explicit lowering to `IO` first.

[INTERCEPTORS_AND_MODULE_INITIALIZERS]:
- `[InterceptsLocation(version, data)]` on a static method routes invocations at the location encoded in `data` to the interceptor at compile time. `SemanticModel.GetInterceptableLocation(InvocationExpressionSyntax)` returns the opaque `InterceptableLocation` value; `CSharpExtensions.GetInterceptsLocationAttributeSyntax(InterceptableLocation)` produces the attribute syntax for the generated file. Interceptors are a source-generator-only weave surface — hand-written interceptors require manually correct location data, which is fragile. `InterceptsLocationAttribute` is not in `System.Runtime`; the generator must emit it as `namespace System.Runtime.CompilerServices { [AttributeUsage(...)] sealed class InterceptsLocationAttribute(int version, string data) : Attribute {} }` in the generated file. The version integer must be taken from `GetInterceptableLocation`'s result at generation time — never hardcoded.
- `GetInterceptableLocation` returns an `InterceptableLocation` (version integer + opaque data string) only for invocation expressions — constructor calls, property accessors, and delegate invocations are not interceptable. A mismatch between the version used during generation and the version expected by the compiler at build time produces a compile error, preventing silent data corruption from format-version drift. Any source generator producing interceptors must regenerate location data on every compilation, not cache it across builds.
- `CSharpExtensions.GetInterceptorMethod(SemanticModel?, InvocationExpressionSyntax, CancellationToken)` returns `IMethodSymbol?`. Given an intercepted invocation, it returns the interceptor method symbol. A generator can confirm its own interceptor resolved correctly for a target call site within a single compilation pass, closing the generation-verification loop without manual source inspection.
- Interceptors require opt-in via the `<InterceptorsNamespaces>` MSBuild property (or `<Features>InterceptorsNamespaces=...</Features>` in a `.editorconfig` features block) listing the namespaces that contain interceptor types. Without this, the compiler emits `ERR_InterceptorsFeatureNotEnabled` (9137) on every `[InterceptsLocation]` attribute — the weave is entirely suppressed. A source generator that emits interceptors must either document the required MSBuild property or emit it into a generated `.props` file via the generator's companion `build` folder.
- `ERR_InterceptorSignatureMismatch` fires when an interceptor has a parameter or return type incompatible with the intercepted method's resolved signature; a warning fires for compatible-but-not-identical cases (differing parameter names or `params` vs. explicit array). An interceptor produces a strict call-site replacement — the original method is never called unless the interceptor explicitly calls it. Only one interceptor per location is valid; a second interceptor for the same location is a compile error.
- `[ModuleInitializer]` marks a static, non-virtual, parameterless, void-returning method as a CLR module initializer — it runs once before any code in the module accesses types from that module. The method must not be generic, must not be in a generic type, and must not be a property accessor, constructor, or operator. These constraints mean it cannot be a generic factory installer, cannot be placed on generated partial constructors or partial methods, and cannot be embedded inside generic-type aspects. It is the last-resort aspect for unconditional registration, codec installation, and process-static initialization that must occur before first use.
- Within a single assembly, `[ModuleInitializer]` methods run in source-file order and then in declaration order within each file. This ordering is a consequence of IL emission order, not a language specification guarantee — it may change across compiler versions. Aspect registrations that depend on ordering must be consolidated into one method or use explicit dependency calls inside that method rather than relying on cross-method ordering across multiple `[ModuleInitializer]` declarations.

[STACKING_ORDER]:
- Definition-time aspects from the generator emit multiple per-concern partial files per type. The hint name is constructed as `namespace + "." + prefix + typeName + arity + fileNameSuffix + ".g.cs"` where `fileNameSuffix` varies per concern (equality, serialization, factory, operators, parsing). Multiple `[ObjectFactory<T>]` attributes on one type, or multiple `SerializationFrameworks` flags, produce additional files from the same generator invocation. All emitted files target disjoint generated member names, so no stacking conflict arises between the concern files for one type.
- When two definition-time aspects produce generated members that share a name — which the generator avoids by construction — the conflict manifests as a duplicate member error at compile time, not a silent override. Partial type merging does not resolve naming conflicts; it only merges disjoint sets of members into one type.
- Composition-time aspects stack left-to-right through the monadic pipeline in the order they appear as `|` operands or chained method calls. `computation | @catch(pred, handler) | @catch(pred2, handler2) | final(cleanup)` evaluates strictly in that order: the first `@catch` sees only errors from `computation`; the second `@catch` sees errors that escape `handler`; `final` runs unconditionally on success or failure of the entire chain to its left. A `@catch` placed after `final` in the chain receives errors normally — `IOFinal.Invoke` re-propagates the exception through its C# `finally` block into the outer interpreter stack, where the subsequent `@catch` is on the handler chain. Position relative to `final` is load-bearing for cleanup ordering, not handler reachability: `computation | final(cleanup) | @catch(pred, h)` runs cleanup before the handler sees the error; `computation | @catch(pred, h) | final(cleanup)` runs cleanup after the handler (or after success). Both forms reach the handler.
- `retry(schedule, computation)` wraps the entire computation as one retriable unit. If `computation` is a composed chain of `| @catch` handlers, the retry sees only errors that escape all inner handlers. A `retry` around an inner `@catch` and a `@catch` around an outer `retry` are not equivalent: `retry(s, computation | @catch(pred, h))` retries the combined expression, while `retry(s, computation) | @catch(pred, h)` catches errors from both the computation and any retry exhaustion. Schedule exhaustion propagates the last error unchanged; `retryUntil`/`retryWhile` termination returns the last observed error when the predicate halts iteration.
- The boundary between definition-time and composition-time stacking is the call site: a `[Union]` `Switch` dispatch backed by `[UseDelegateFromConstructor]` delegate rows executes case behavior at the call site as a pure synchronous call — the composition-time retry policy wrapped around the call site sees the dispatch and all its case behavior as one atomic unit. Per-case retry granularity is not expressible by wrapping the dispatch; it requires each case arm to produce a `K<M, A>` that is individually wrapped with retry before the dispatch assembles them.

[OPERATOR_CASCADE_CONSTRAINTS]:
- `EqualityComparisonOperators` on `ValueObjectAttribute<T>` and `SmartEnumAttribute<TKey>` is coerced upward to match `ComparisonOperators` when `ComparisonOperators` is set higher — the getter computes `max(ComparisonOperators, EqualityComparisonOperators)` because comparison operators require equality operators in generated code. Setting `ComparisonOperators = DefaultWithKeyTypeOverloads` without explicitly setting `EqualityComparisonOperators` silently forces `EqualityComparisonOperators` to `DefaultWithKeyTypeOverloads` as well. The reverse coercion does not apply.
- `SkipEqualityComparison = true` is a dominating flag: `EqualityComparisonOperators` and `ComparisonOperators` getters return `OperatorsGeneration.None` regardless of their explicit assignments. On a struct value object, suppressing equality prevents the generated `default(T)` guard from being hashable, blocking use in `FrozenDictionary` and `FrozenSet` unless hand-written `Equals`/`GetHashCode` replace the suppressed generated versions.
- `SkipFactoryMethods = true` cascades across five aspects simultaneously: `TypeConverter` attribute is not emitted; `IObjectFactory<T>` is not implemented; the conversion operator from key type to value object is not generated; `SkipIParsable` and `SkipISpanParsable` are forced to `true`; arithmetic operators (`AdditionOperators`, `SubtractionOperators`, `MultiplyOperators`, `DivisionOperators`) are all forced to `None`. Setting any of the cascaded properties individually does not set `SkipFactoryMethods`.
