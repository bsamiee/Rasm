# Effect Lifecycle Algebra for Measured Solves

[ATTEMPT_SEMANTICS_OF_RETRY_VS_REPEAT]:
- `RetryUntil(schedule, pred)` and `RepeatUntil(schedule, pred)` are the same fold skeleton over `schedule.PrependZero.Run().GetIterator()` but differ in one load-bearing detail: retry wraps each attempt in `BracketFail()` (resources released only on the failing branch) and repeat wraps each iteration in `Bracket()` (resources released unconditionally at iteration end). The consequence is a hard contract: a retried solve that succeeds carries its acquired native handles out to the continuation, but a repeated solve cannot return any resource it acquired inside the loop — the disposal fires before the next iteration sees the value. A measured solve that must hand a live factorization to its caller composes through retry, never repeat; a polling probe that must not leak across ticks composes through repeat.
- `PrependZero` makes the first attempt immediate (`Duration.Zero`), so `recurs(n)` produces `n` total attempts, not `n` delays after a first free run. The schedule iterator drives delay-then-attempt via `yieldFor(head).Bind(_ => attempt)`; the head duration is consumed before the body, so the zero-prepend is what aligns "attempts" with "schedule length".
- On schedule exhaustion the retry fold returns `IO.fail(error)` carrying the LAST captured error, not the first and not an aggregate. A pipeline that needs the full failure trajectory across attempts cannot recover it from the retry combinator; it must thread an external accumulator (an `Atom<Seq<Error>>` swapped inside a `Catch` arm) because the combinator discards every intermediate error except the terminal one.
- The retry predicate is an UNTIL predicate over the error: retry continues while the predicate is false and stops (re-throwing) when it returns true. `RetryWhile(p)` is defined as `RetryUntil(not(p))`. For a numeric rail this is the seam where a budget-exceeded status is made retryable and a structural-breakdown status is made terminal: the predicate inspects the converted domain error code and returns true (stop) for a singular-matrix code, false (retry) for a transient conditioning code.
[BRACKET_AS_THE_SOLVE_CAPSULE]:
- The three-argument bracket desugars to one expression — `Acq.Bind(x => Use(x).Catch(Catch).Finally(Fin(x)))` — so acquire, use, exception-recovery, and release are one composed value, not four call sites. The `Catch` arm runs INSIDE the resource scope, before `Finally`, so a recovery path still sees the live resource and the release still fires after recovery. A measured factorization that must attempt a diagonal-shift fallback on an indefinite-matrix throw places that fallback in the `Catch` arm and keeps the working-buffer release in `Fin`, and both are guaranteed in the failure path. The discipline inside the `Catch` arm is that any resource the recovery acquires must itself be bracketed — a recovery that rents a fresh workspace inline has stepped outside the capsule and leaks it on a second failure.
- `Bracket()` with no arguments is not resource acquisition — it re-roots the computation in a fresh `LocalResources` environment so every resource acquired transitively is released at the bracket boundary regardless of outcome. `BracketFail()` re-roots identically but only releases on failure, letting a successful computation export its acquired resources upward. These two are the primitives the retry/repeat fold selects between; understanding the lifecycle algebra means reading retry as "BracketFail per attempt" and repeat as "Bracket per iteration".

```csharp
static Eff<RT, TReceipt> MeasuredSolve(MatrixHandle a, VectorHandle b) =>
    liftIO(IO.lift(static () => Workspace.Rent(a.Rows))
        .Bracket(
            Use:   ws  => Decompose(a, ws).Bind(f => SolveAndProve(f, b, ws)),
            Catch: err => err.HasCode(Codes.Indefinite)
                              ? ShiftedSolve(a.WithShift(Policy.Shift), b)   // itself a bracket
                              : IO.fail<TReceipt>(err),
            Fin:   ws  => IO.lift(() => ws.Return())));
```

[SCHEDULE_AS_A_POLICY_LATTICE]:
- A `Schedule` is a value-typed time-series of `Duration` with a full lattice algebra over composition, so backoff policy is data carried by one value, never imperative delay arithmetic at the call site. Between two SCHEDULES, `a | b` (Union) emits the MIN duration and runs to the LONGER of the two — the breadth combinator — and `a & b` (Intersect) emits the MAX duration and runs to the SHORTER — the bound combinator; `a + b` (Combine) appends the series. The same two symbols beside a TRANSFORMER are plain application: `recurs(n)` is `Take(n)` over whatever schedule it touches, so `recurs(n) | exponential(seedMs)` and `recurs(n) & exponential(seedMs)` are one identical capped exponential and the operator choice is inert there. The unbounded-policy defect lives entirely on the schedule-schedule side: `exponential(seed)` derives from the infinite `Forever`, so unioning it with any other infinite series never terminates, and a bound must arrive as a transformer — `recurs(n)` caps on COUNT, `maxCumulativeDelay(budget)` caps on TIME — or as an intersect against a genuinely finite series. The symbol does not determine the semantics; the pair of operand kinds does, and a policy review that reads the operator without reading both operand kinds misjudges whether the policy is bounded at all.
- Transformers are first-class functions over schedules: `maxDelay(d)` clamps per-step delay, `maxCumulativeDelay(d)` halts when accumulated delay crosses a budget, `resetAfter(d)` rewinds the series after a cumulative threshold, `jitter(factor)` and `jitter(min, max, seed)` add bounded randomization, and `decorrelate(factor, seed)` produces decorrelated jitter where each delay derives from the previous realized delay rather than the nominal one. These compose with schedules by the same `|`/`&` operators (`ScheduleTransformer` has operator overloads against `Schedule`), so a contention-aware retry policy for a shared solver pool is `exponential(baseMs) | maxCumulativeDelay(budget) | decorrelate()` — one value, fully auditable, droppable into any retry surface.
- `recurs(n)` is a transformer meaning "number of ATTEMPTS"; the `repeat(n)` transformer means "replay the whole schedule n times" — replaying `recurs(3)` under `repeat(2)` yields six attempts in two bursts. Conflating them silently doubles or halves the attempt budget. The base series themselves are derived, not enumerated: `exponential` is `Forever.Map((_, i) => seed * pow(factor, i))`, `spaced` is `Forever.Map(_ => space)`, so a custom backoff law is a single `Map` over `Forever` rather than a hand-built duration array.
[CANCELLATION_AND_FORK_SCOPE]:
- `Local()` installs a fresh `CancellationTokenSource` whose token is independent of the parent — a child solve can be cancelled or time-limited without tearing the enclosing scope, and cancelling the parent still cascades because the parent token is registered to cancel the child source. This is the seam for a per-solve deadline that must not abort a sibling solve in the same batch.
- `Timeout(span)` is a structural node (`IOTimeout`), not a wall-clock sleep race — it threads the deadline through the effect interpreter, so a timed-out measured solve releases its bracketed workspace through the normal `Finally` path rather than abandoning it. `Fork(timeout)` returns `ForkIO<A>` = `(IO<Unit> Cancel, IO<A> Await)`: a handle pair where `Cancel` is itself an effect (composed, not invoked imperatively) and `Await` blocks on completion. Resources acquired inside a forked solve release on the fork's own completion, and resources acquired in the parent are SHARED into the fork — a forked numeric solve that reads a parent-owned matrix buffer must treat that buffer as shared lifetime, because either side may release it.
- Fork honors the ambient cancellation token at launch: if the parent token is already cancelled when `Fork` runs, it throws `TaskCanceledException` before queuing, so a fan-out of forked solves under an already-cancelled batch deadline never schedules dead work.
[ATOM_COMMIT_SEMANTICS_AS_ACCUMULATION_RAIL]:
- The `Change` event fires only after a real `Interlocked.CompareExchange` commit, and the validator-rejected branch of `Swap(f)` returns the PRIOR value with no event and no signal — byte-identical to a successful identity swap. An accumulation fold that encodes a conservation invariant as the atom validator therefore cannot learn from the return value whether a step was rejected; the only ambiguity-free rail is `SwapMaybe(f)` with the invariant checked inside `f`, plus an external `Fin` on the factory result — a `Swap` that silently retains the old receipt map is data loss with no error channel.
- `SwapIO(f)` and `SwapMaybeIO(f)` lift the CAS commit into `IO` (`IO.lift(_ => Swap(f))`), so receipt accumulation joins the effect pipeline as a node sequenced by `Bind`, ordered relative to the bracketed solve and the retry boundary rather than executing out of band. Because `f` is re-invoked on every CAS retry, it must be the pure receipt-merge (`map => map.Add(key, receipt)`); a merge that closes over a mutable buffer corrupts under contention, and the pure statistics-merge combinator is directly usable as the `SwapIO` `f` for moment accumulation.
[ERROR_CODE_DIRECTED_RECOVERY]:
- `@catch(int errorCode, Fail)` and `@catch(Func<Error,bool> predicate, Fail)` build `CatchM` recovery arms that compose against an effect via `operator |`, so the receipt factory's domain error CODE becomes the catch-arm DISCRIMINANT — recovery dispatches on the same integer the factory stamped at the boundary, never on exception type matching. A measured-solve rail reads `solve | @catch(Codes.BudgetExceeded, relaxAndRetry) | @catch(Codes.Singular, fail)` as a total recovery table keyed on provenance code, where budget-exceeded relaxes a stop criterion and re-enters while singular terminates.
- The recovery arms chain left-to-right and the first matching predicate wins, so arm ORDER is policy: a broad transient-conditioning predicate placed before a specific singular-matrix code swallows the singular case. The chained-`|` table must order from most specific code to most general predicate, mirroring the convergence-status partition where breakdown must be caught before generic non-convergence. Each arm produces its own typed `Fin.Fail` or recovery effect; no shared outer handler spans arms, so provenance stays per-arm.
[APPLICATIVE_ACCUMULATION_VS_MONADIC_SHORTCIRCUIT]:
- `Validation<F, A>` is the accumulating applicative: `Map`/`BiMap`/`Apply` collect every `Fail` across independent checks, while its `Bind` short-circuits monadically on the first failure. A receipt factory that validates several INDEPENDENT conservation identities at once (rank-nullity, descending-singular-value order, standard-error arity) composes them applicatively so a single returned `Validation.Fail` carries the full violation set, then converts to the `Fin` rail once at the boundary; sequencing those same checks with `Bind` hides every violation after the first. The dividing rule is dependency: applicative for independent predicates, monadic `Bind` only where a later check needs an earlier check's success value.
- The applicative path keeps `SuccessSpan()` as a zero-allocation read of the success payload when present, so a validated receipt's fields are projected without re-boxing. The boundary conversion is one hop — `Validation` to `Fin` — and the accumulation lives entirely on the `Validation` side; the `Fin` rail never re-accumulates, preserving the law that each rail owns exactly one concern.
[LIFECYCLE_COMPOSITION_LAW]:
- The full measured-solve lifecycle is one composed `Eff<RT, TReceipt>` whose order is fixed by the algebra: resource bracket is the innermost capsule (release guaranteed), the schedule-driven retry wraps the bracket (each attempt is its own `BracketFail`), the error-code catch table wraps the retry (recovery after the budget is spent), the cancellation/timeout scope wraps the catch (deadline dominates recovery), and the atomic receipt accumulation is a sibling `SwapIO` node sequenced after a successful return. Inverting any two layers changes semantics: putting the catch inside the retry recovers per-attempt and defeats the schedule; putting accumulation inside the bracket disposes the receipt with the workspace. The composition is read inside-out as release < attempt-policy < recovery < deadline < commit, and that ordering is the lifecycle law for any algorithm that must produce a tamper-evident receipt under retry, timeout, and contention.
