# [PY_ARTIFACTS_API_ZXING_CPP]

`zxing-cpp` (dist `zxing-cpp`, import `zxingcpp`) supplies the 2D-matrix encoded-mark surface AND the only decode inverse across the `graphic/marks` rail: it is the `matrix` `EncodeArm` of `graphic/marks/encode#MARK` (DataMatrix/PDF417/Aztec/MaxiCode, the symbologies the segno QR arm and the python-barcode linear arm drop) and the sole owner of `graphic/marks/decode#DECODE` (`read_barcodes` over a `numpy` frame, a PIL image, a `Buffer`, or a custom-stride `ImageView`), closing the encode/decode round-trip neither generation-only arm can express. The encode surface is the modern `create_barcode(content, format, **kwargs) -> Barcode` writer (string or `bytes`), `Barcode.to_svg` (dependency-free SVG string) and `Barcode.to_image` (an 8-bit grayscale `Image` exporting the buffer protocol straight into `numpy`); the decode surface is `read_barcodes(image, ...) -> list[Barcode]` returning the full per-symbol evidence shape (`text`/`bytes`/`format`/`symbology`/`valid`/`position`/`orientation`/`error`/`content_type`/`symbology_identifier`/`ec_level`/`extra`). The `matrix` arm composes `create_barcode` + `to_svg`/`to_image` into the `MarkOp` encode path beside segno/python-barcode under one `Mark` owner; the decode owner composes `read_barcodes` + `ImageView` + the `Error`/`ContentType`/`ErrorType`/`Position` shapes into the `MarkOp.Decode` body, projecting every symbol through the shared `RasterFact` onto `core/receipt#RECEIPT` `ArtifactReceipt.Preview` — it reads the widest wrapped OSS symbology set, and the 2D-matrix concern is dropped from python-barcode and routed here rather than re-implemented.

## [02]-[PUBLIC_TYPES]

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: encode (create + serialize)
- rail: imaging

`create_barcode` builds a `Barcode` from `str` (encoded UTF-8, `content_type=Text`) or `bytes` (raw, `content_type=Binary`) content for any creatable format; the `CreatorOptions` axis rides `**kwargs` keyword-only, never positional — `ec_level` is the verified content key (a string `'L'`/`'M'`/`'Q'`/`'H'` for QR or an integer-as-string for Aztec/PDF417; an out-of-range value raises `ValueError("Invalid ecLevel: ...")`). The binding's `**kwargs` silently ignores unrecognized keys (a bogus key does not raise), so GEOMETRY (`scale`/`width`/`height`/`margin`/`add_quiet_zones`) is a WRITER concern on `to_svg`/`to_image`/`write_barcode_to_*`, NOT a `create_barcode` key — passing geometry to `create_barcode` is silently dropped; the consuming arm key-filters its admitted band against the genuine creator keys (`ec_level`) rather than spraying writer keys at the creator. `Barcode.to_svg` serializes a dependency-free SVG string; `Barcode.to_image` yields an 8-bit grayscale `Image` whose buffer protocol feeds `numpy.array` directly; the `write_barcode_to_*` module functions are the equivalent free-function forms over the same `(scale, add_hrt, add_quiet_zones)` axis. The 2.x `write_barcode(format, text, width, height, quiet_zone, ec_level) -> Image` one-shot is DEPRECATED (it emits a `DeprecationWarning` directing to `create_barcode` + `write_barcode_to_image`) and uniquely carries `width`/`height`/`quiet_zone` as positional creator-plus-writer args — the catalog admits it only as the superseded form, never on the rail.

[ENTRYPOINT_SCOPE]: decode (read)
- rail: imaging

`read_barcodes` decodes every symbol in an image and returns a `list[Barcode]`; `read_barcode` returns the first `Barcode | None`. Both default `formats=BarcodeFormats()` (empty = all readable). The `formats` scope narrows the search to one `BarcodeFormat`, a set member, a `BarcodeFormats`, or `barcode_formats_from_str(...)`; the remaining keyword axis (`try_rotate`/`try_downscale`/`try_invert`/`text_mode`/`binarizer`/`is_pure`/`ean_add_on_symbol`/`return_errors`) tunes the detector. `return_errors=True` is load-bearing: without it an invalid symbol is silently dropped; with it the symbol returns carrying its `Error` so a checksum/format miss is a typed per-symbol fact. `barcode_formats_list(filter=)` enumerates the supported format roster (optionally filtered) for capability probing. Each decoded `Barcode` carries `text`, `bytes`, `format`, `symbology`, `valid`, `position`, `orientation`, `error`, `content_type`, `symbology_identifier`, `ec_level`, and the per-symbology `extra` dict.

## [04]-[IMPLEMENTATION_LAW]

[IMAGING_MATRIX]:
- format axis: one `BarcodeFormat` enum keys symbology selection on both encode and decode — `create_barcode(content, BarcodeFormat.DataMatrix)` and `read_barcodes(image, formats=BarcodeFormat.DataMatrix)`. In 3.0 the enum `str()` is the human display name with separators (`'Data Matrix'`/`'QR Code'`/`'Code 128'`/`'EAN-13'`) while `.name` is separatorless (`'DataMatrix'`/`'QRCode'`/`'Code128'`/`'EAN13'`); resolve a format through the enum attribute or `barcode_format_from_str` (which accepts both `'EAN13'` and `'EAN-13'`), never by formatting the enum to a string and re-parsing the separatorless name — the 3.0 ToString rename breaks that pattern. `Barcode.format` (precise) and `Barcode.symbology` (rolled-up family, e.g. `EAN13 -> EANUPC`, `MicroQRCode -> QRCode`) are distinct facts; the decode owner records `str(format)` for the per-symbol `symbology` field and may carry the family separately, never collapsing the two.
- creatable axis: 3.0 encodes DataMatrix, PDF417 (and Compact PDF417), Aztec, QR, Micro-QR, AND MaxiCode — MaxiCode is creatable in 3.0 (the 2.x decode-only limitation was lifted by the ZXing-C++ writer rewrite), verified by a successful `create_barcode("...", BarcodeFormat.MaxiCode).to_svg()`; `BarcodeFormat.AllCreatable` names the full encode set and `barcode_formats_list(BarcodeFormat.AllCreatable)` enumerates it. Linear EAN/UPC/Code128/ITF also encode but overlap python-barcode's owned domain, so the admitted scope of this package on the preview rail is the 2D-matrix concern python-barcode drops.
- content axis: `create_barcode` accepts `str` (encoded as UTF-8 text, `content_type=Text`, `bytes` = the UTF-8 octets) or `bytes` (raw binary, `content_type=Binary`, `bytes` = the input verbatim); a binary DataMatrix/Aztec/PDF417 payload rides the `bytes` path, never a lossy str round-trip. `Barcode.bytes` is the uninterpreted symbol bytes on both directions; the decode owner carries it as `raw` so a binary payload survives the seam without a text reinterpretation.
- option axis: `create_barcode(**kwargs)` carries the `CreatorOptions` axis keyword-only; the verified creator key is `ec_level` (a string `'L'`/`'M'`/`'Q'`/`'H'` for QR, an integer-as-string for Aztec/PDF417 — an out-of-range string raises `ValueError("Invalid ecLevel: 'Z'")`), never a positional `ec_level` argument and never a per-format encoder function. The binding's `**kwargs` silently ignores an unrecognized key, so geometry (`scale`/`width`/`height`/`margin`/`add_quiet_zones`) is the SERIALIZE axis on `to_svg`/`to_image`/`write_barcode_to_*`, not a `create_barcode` key — the arm key-filters its admitted band against the genuine creator keys rather than passing writer geometry to the creator where it is silently dropped. `Barcode.ec_level` is the empty string when not applicable to the symbology (e.g. EAN/UPC/DataMatrix return `''`), so the receipt records `ec_level` as present-or-empty, never asserting an EC level on a format that lacks one.
- decode axis: `read_barcodes` returns a `list[Barcode]` and `read_barcode` the first `Barcode | None` — the decode owner COLLAPSES `read_barcode` into `read_barcodes` (a decode is always plural over the raster; "first symbol" is the consumer's `[0]` projection, never a sibling entrypoint). Scope the search with `formats=` (a single `BarcodeFormat`, a set member, a `BarcodeFormats`, or `barcode_formats_from_str("Data Matrix,QR Code")` — the `\|` union operator is deprecated in 3.0, pass a set member or parse a string instead). The eight set-valued classes (`AllMatrix`/`AllLinear`/`AllRetail`/`AllGS1`/`AllIndustrial`/`AllCreatable`/`AllReadable`/`All`) are direct `formats=` scopes reaching readable formats with no single-format member (`Code93`/`UPCE`/`DataBar*`/`MicroPDF417`). Each decoded `Barcode` yields `text`, `bytes` (raw), `format`, `symbology` (family), `valid`, `position` (a four-corner `Position`), `orientation`, `error` (an `Error` with `.type`/`.message`, or `None`), `content_type`, `symbology_identifier` (the AIM `]d1`-style id), `ec_level`, and `extra`; map a non-`valid` symbol or a non-`None` `error` at the boundary rather than trusting `text`.
- metadata axis: `Barcode.extra` is a real `dict[str, str]`-shaped Python dict carrying symbology-specific fields (a QR symbol yields `{'DataMask', 'Version', 'ECLevel'}`, a DataMatrix `{'Version': '8x32'}`) — the decode owner captures it stringified onto the per-symbol evidence (`extra={k: str(v) for k, v in barcode.extra.items()}`), never discarding the structural metadata. `Barcode.error` is `None` on a clean symbol or a `zxingcpp.Error` whose `.type` is an `ErrorType` (`Checksum`/`Format`/`Unsupported`) and whose `.message` is the human string; the owner maps `error.type` onto its closed fault vocabulary.
- frame axis: decode input is a numpy BGR/grayscale array (a bare three-channel array is read BGR, swapping RGB luminance weights, and a four-channel array misreads), a PIL image, a `Buffer`, or a `zxingcpp.ImageView(buffer, width, height, format, row_stride=0, pix_stride=0)` that declares the channel order through `ImageFormat` and supports a non-contiguous stride — the decode owner wraps a typed frame in an `ImageView` carrying its declared `PixelFormat -> ImageFormat` rather than a shape-inferred guess. The `to_image` output is an 8-bit grayscale `Image` whose buffer protocol makes `numpy.array(image)` zero-copy, so the encode-then-decode raster never round-trips through PNG bytes.
- round-trip axis: `create_barcode(content, fmt).to_image(scale=N)` then `numpy.array(image)` then `read_barcodes(frame)` recovers the content with `valid=True` and the matching `format`, so the encoded mark carries a generation-correctness proof from one decode pass — the round-trip a QR-only or linear-only owner cannot express. The decode-confidence fact (`valid`, `format`, `text`, `error`, `position`) lands on the imaging receipt.
- dimensionality axis: this package owns the 2D-matrix concern (DataMatrix/PDF417/Aztec/MaxiCode) plus the QR/Micro-QR and linear sets; on the artifacts preview rail it is admitted for the 2D-matrix symbologies python-barcode explicitly drops and the decode round-trip segno and python-barcode lack, never as a second QR or linear encoder displacing the segno/python-barcode arms.
- evidence: each encoded/decoded symbol captures format display name, family symbology, content, error-correction level, validity, decode error, position quad, orientation, AIM identifier, and the symbology `extra` dict as an imaging receipt fact.

[STACKING]:
- The `matrix` arm folds onto the SHARED `libs/python/.api` substrate exactly as the segno `qr` and python-barcode `linear` arms do: the SVG string from `barcode.to_svg()` (`.encode()` to bytes) or the grayscale `to_image()` buffer crosses the seam as a `Result[RasterFact, MarkFault]` (`expression` `Ok`/`Error`), the encode `MarkPayload` band is admitted once through a module-level `TypeAdapter` and a `ValidationError` maps onto the options-fault case as the full `tuple[str, ...]` of `.errors()` `loc` paths (`pydantic`), the closed `Symbology`/`MarkFault` encode vocabularies and the `RasterFact` value object are `msgspec.Struct`/`expression.tagged_union` shapes, the `create_barcode` call is offloaded off the event loop under the one `_MARK_LANE` `CapacityLimiter` via `anyio.to_thread.run_sync`, and the resolved `format`/`ec_level` evidence threads onto `ArtifactReceipt.Preview.scores` for the `structlog`/OTel receipt — never a flat folder-only call. A `create_barcode` `ValueError` (`Invalid ecLevel`, oversize content) is named at the matrix arm and railed onto the encode `MarkFault`, never a bare unwrapped raise escaping the encode capsule.
- The decode owner (`graphic/marks/decode#DECODE`) stacks the SAME substrate at greater depth: `DecodeScope` collapses the eight-argument `read_barcodes` detector tail into one frozen value keyed by a `ScopeKind` preset, with the format scope an explicit `Symbology` tuple or a `FormatFamily` class member resolved through `barcode_formats_from_str` / the `_FAMILY` set-value table; `DecodedSymbol` is a frozen `msgspec.Struct` over the eleven `Barcode` read properties that doubles as its own wire form (`msgspec.json`) crossing a process seam onto the receipt; the canonical `Binarize`/`TextRead`/`EanAddOn`/`PixelFormat`/`FormatFamily` `StrEnum` vocabularies remap to the provider `Binarizer`/`TextMode`/`EanAddOnSymbol`/`ImageFormat`/`BarcodeFormat` enums through `frozendict` tables only at the `read_barcodes` call so zxing names stay at the provider edge; the `_admitted` core is `@beartype`-contracted at the foreign-pybind boundary; and a raster source rides a crash-isolated `anyio.to_process.run_sync` worker (a PIL `Image.open`/`load` `UnidentifiedImageError`/`DecompressionBombError`/`OSError` converting to a typed `UNREADABLE` fault) while a trusted already-decoded `numpy` frame rides an `anyio.to_thread.run_sync` slot through `ImageView` (a wrong-rank/dtype/channel frame converting to `MALFORMED`) — both off the event loop under the one `_MARK_LANE` `CapacityLimiter`.
