# EnvIO Resource Registry Substrate

[OWNERSHIP_BITMASK]:
- The effect environment threaded through every computation carries four fields that decide resource lifetime: a `Resources` registry, a `CancellationToken`, its owning `CancellationTokenSource`, and an optional captured `SynchronizationContext`. A private `Own` bitmask records which of the source (bit 1) and the registry (bit 2) the environment instance allocated itself versus inherited from a parent. Disposal is gated on ownership: the registry is released only when `(Own & 2) == 2`, the source disposed only when `(Own & 1) == 1`. An environment that inherited a shared registry never disposes it — the inheriting scope is a borrower, and the bitmask is the borrowed/owned discriminant expressed as bit flags rather than a case type.
- Disposal splits into two idempotent halves guarded by separate interlocked flags: one releases tracked resources, the other disposes the source and its cancellation registration. Each half uses `Interlocked.CompareExchange(ref flag, 1, 0)` so a double-dispose is a no-op, and the two halves are independently invokable — the failure-only bracket disposes resources in a catch but defers the source-and-registration half to a finally, so the two cleanup concerns never couple. A scope that wants to dispose its cancellation plumbing while letting acquired resources survive calls only the second half.
- Four derived environments clone different field subsets, and each names exactly what it isolates: the fully-local form takes a fresh registry, a fresh source, and re-captures the ambient sync context (isolates everything); the resources-local form takes a fresh registry but shares the parent source (isolates resources, shares cancellation); the cancel-local form shares the parent registry but takes a fresh source (isolates cancellation, shares resources); the sync-context-local form shares both registry and source and only re-captures the ambient context. Selecting the wrong derivation silently shares the lifetime axis the caller meant to isolate, and the bug surfaces only as a cross-scope release.

[LOCAL_ISOLATES_CANCELLATION_NOT_RESOURCES]:
- The cancellation-local operator passes the existing registry into the new environment and passes a null source; the environment factory, seeing a non-null registry, leaves the registry-ownership bit clear and, seeing a null source, allocates a fresh source and sets the source-ownership bit. The consequence is that this operator isolates the cancellation token but continues to share the parent's resource registry — resources registered inside it are tracked in and released by the outer scope, not at this operator's boundary. Reaching for it expecting an isolated resource scope leaves acquisitions live past the intended boundary.
- The fully-isolated child scope — independent registry and independent token — is the resources-local environment, not the cancellation-local one. The acquire-bracket operator is defined as mapping the environment through the resources-local derivation; it is the canonical way to open a child registry that disposes on exit. A per-iteration cancel-and-retry pattern that also needs per-iteration resource disposal must wrap each attempt in the bracket operator, not the cancellation-local operator, or per-iteration acquisitions accumulate in the outer registry and only release at the outermost exit.

[REGISTRY_INTERNALS]:
- The registry stores tracked resources in an atomic hash map keyed by the resource value itself under object identity. Registration is a try-add: a second registration of a reference-equal value is silently dropped, and — critically — a value type or record whose equality is structural collides with a prior structurally-equal registration, so the second acquisition is never tracked and never released. Resources that must be independently tracked must be reference types with reference identity, or must be wrapped in a distinct owner per acquisition.
- Registries form a parent chain. Release of a value searches the local map first; on a miss it walks to the parent and releases there. A child scope can therefore release a resource its parent acquired — the lookup reaches up the chain — which is the precise mechanism by which a forked or nested computation can free a resource a sibling still holds. The release-all operation atomically swaps the local map to empty while running each tracked release, leaving parent registrations untouched.
- A merge operation folds another registry's entries into this one via an atomic add-range. This is how applicative and parallel composition unify two independently-populated registries into a single disposal scope; the merged-in entries inherit the target registry's lifetime and dispose with it.

[RELEASE_ORDER_AND_FAILURE_SURFACE]:
- Bulk disposal iterates the registry's hash-map enumerator and runs each tracked release in turn. Release order is therefore hash-bucket iteration order, not last-in-first-out acquisition order. Any disposal sequence that depends on reverse-acquisition ordering — a child handle that must close before the container handle it was carved from — cannot rely on the registry to enforce it and must encode the ordering as an explicit nested bracket or an explicit dependency in a single tracked finalizer.
- Bulk disposal constructs a fresh environment with a non-cancellable token before running the releases, so cleanup is immune to the outer token: a cancelled or timed-out computation still runs its registered releases to completion under a clean token. The release loop has no per-item failure isolation — a throwing release aborts the loop and leaves every not-yet-iterated resource unreleased. A tracked release that can throw must catch internally and convert to a tracked failure, or one faulty handle leaks every handle the enumerator had not yet reached.
- That throwaway release environment is itself fully owned and never disposed — its fresh source is abandoned to finalization on every bulk teardown. The cost is one leaked source per scope disposal, which compounds on a boundary cell that opens and tears down a scope per operation; long-lived cells batch their resource lifetime into one outer scope rather than paying the per-teardown abandonment on a hot open/close path.
- The release-all sweep runs inside the registry map's own CAS swap: on contention the swap function re-executes against the retried snapshot, re-running release effects already executed from the previous snapshot. Release effects routed through the sweep must therefore be idempotent; non-idempotent teardown uses the per-key release, which removes the key before running the effect and so cannot re-enter on retry.

[TRACKED_RESOURCE_FAMILY]:
- Acquisition produces one of three tracked-resource records, each a closed case over how the value is freed: a synchronous-disposable case, an asynchronous-disposable case, and a custom-finalizer case carrying an explicit release arrow. All three expose one release operation returning a deferred unit effect, so the disposal loop is uniform over the family and the per-case freeing logic lives with the case, not in the loop.
- The synchronous-disposable case performs a release-time interface probe: if the value also implements the asynchronous-disposable interface, release routes through the asynchronous path with non-captured continuation rather than the synchronous one. Registering a dual-interface resource through the synchronous acquire path does not pin it to synchronous disposal — the registry prefers the asynchronous drain at release. The dedicated asynchronous acquire path is required only when the value implements solely the asynchronous-disposable interface, not as an opt-in for a value that implements both.
- The custom-finalizer case stores the release as a function from the value to a deferred unit effect, so a resource with no disposal interface at all — a raw handle freed by a native call, a slot returned to a pool — is tracked identically to an interface-bearing one. The optional-acquire form registers nothing when the producing effect yields absence, so a typed-absent optional resource leaves the registry untouched rather than tracking a null.

[BRACKET_DISPOSAL_SEAMS]:
- The two bracket flavors differ only in their disposal seam. The release-on-every-exit flavor runs the inner computation inside a using-bound local environment, so success, failure, and cancellation all trigger the using-disposal that releases the local registry. The release-on-failure-only flavor runs the inner computation in a try and calls only the resource-disposal half from its catch; on success it calls only the source-and-registration half, leaving the local registry's resources unreleased. The single-argument acquire-bracket resolves to the release-on-every-exit flavor — it always frees on exit and does not carry survive-on-success semantics.

```csharp
// Distilled seams. Release-on-every-exit: using disposes the local registry on any path.
// Release-on-failure-only: catch disposes resources; success path disposes only the
// source, so acquired resources survive into the outer scope's registry.
async ValueTask<IO<A>> InvokeEager(EnvIO env)
{
    using EnvIO local = MapEnv(env);                       // disposes resources + source on exit
    return Next(await Operation.RunAsync(local));          // value projected before using-dispose
}

async ValueTask<IO<A>> InvokeOnFailOnly(EnvIO env)
{
    EnvIO local = MapEnv(env);
    try     { return Next(await Operation.RunAsync(local)); }
    catch   { local.DisposeResources(); throw; }           // resources freed only on fault
    finally { local.DisposeNonResources(); }               // source/registration freed always
}
```

- The continuation that consumes the bracketed value runs after the using-disposal returns, because the inner run projects a value and the disposal fires as the invoking frame unwinds. The value is a projection, fully materialized before release; a handle smuggled out as the value is a handle to an already-released resource. This is the registry-level restatement of the projection law: brackets return values, never live handles.
- The survive-on-success flavor does not merge the local registry back into the parent. Survival means the local registry is simply never disposed on the success path and the resource object remains reachable through the returned value; from the registry's bookkeeping the registration is orphaned, but the resource lives because nothing released it. Composing repeated survive-on-success acquisitions accumulates orphaned local registries, each holding a live resource that no enclosing scope will release — the surviving resource must be re-registered in an outer bracket to regain a disposal owner.

[MULTI_ARG_BRACKET_IS_FINALLY_NOT_REGISTRY]:
- The multi-argument instance bracket — acquire, use, finalize, optionally catch — does not touch the resource registry at all. It binds the acquired value, runs the use arrow, and attaches the finalize arrow as a finally combinator; the catch arrow, when present, wraps the use arrow before the finally. The finalizer runs exactly once on completion regardless of outcome, but it is the caller's explicit arrow, not a registry release. Mixing registry-tracked acquisitions inside a multi-argument bracket's use body means two independent cleanup mechanisms run — the explicit finalizer and the enclosing registry's disposal — and their relative order is the bind order, not a unified release.

[FORK_CHILD_REGISTRY_CHAIN]:
- Forking allocates a child registry parented to the forking scope's registry and runs the forked computation under a fresh environment built from that child, a fresh linked source, and the inherited sync context. The fork's finally disposes the child registry on every exit. Because the child is parented, a release call inside the fork for a parent-acquired resource walks up and frees it in the parent — the documented shared-resource hazard is the parent-chain lookup, not a copy. The fork's cancel and await members never dispose resources; disposal is owned solely by the child registry's finally, so awaiting a fork to then dispose its resources observes already-released state.
- Competitive fork combinators cancel the losing forks, and each loser's child registry disposes at that fork's exit — including the winner's. A value projected out of a fork race is a copy; a session opened inside a racing fork is gone by the time the race returns. The winning identity must be a reconstructable token, and the live resource acquired in the outer scope after the race settles.
