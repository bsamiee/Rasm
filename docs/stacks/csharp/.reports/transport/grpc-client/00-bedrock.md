# grpc-client bedrock

## channel law

- One `GrpcChannel` per remote authority per process, created once at the composition root via `GrpcChannel.ForAddress(address, GrpcChannelOptions)` and held for process lifetime; a channel owns connection pooling, retry buffers, balancer state, and compression registry — per-call channel creation forfeits HTTP/2 multiplexing and re-pays TLS plus connection ramp on every call.
- `GrpcChannel` is `IDisposable`; `Dispose` closes underlying connections only when the channel owns its handler — supplying `HttpClient`/`HttpHandler` without `DisposeHttpClient = true` leaves handler lifetime with the caller, which is the correct shape when one `SocketsHttpHandler` is shared across channel rows.
- Channel policy is settled once: every knob lives on `GrpcChannelOptions` (`Credentials`, `ServiceConfig`, `HttpHandler`, `MaxSendMessageSize`, `MaxReceiveMessageSize`, `MaxRetryAttempts`, `MaxRetryBufferSize`, `MaxRetryBufferPerCallSize`, `CompressionProviders`, `LoggerFactory`, `ServiceProvider`, `HttpVersion`, `HttpVersionPolicy`, `InitialReconnectBackoff`, `MaxReconnectBackoff`, `ThrowOperationCanceledOnCancellation`, `UnsafeUseInsecureChannelCallCredentials`, `DisableResolverServiceConfig`); a second configuration site (per-call handler tweaks, ambient statics) is the rejected form — per-call variance is `CallOptions` only.
- Verified defaults: receive cap 4_194_304 bytes, send cap absent (unbounded), retry attempts capped at 5, retry buffer 16_777_216 bytes per channel and 1_048_576 per call, reconnect backoff 1 s initial to 120 s max. Defaults are policy rows to restate explicitly in the channel policy value, not ambient truths to rediscover in production.
- Connectivity is a first-class state machine: `GrpcChannel.State` reports `ConnectivityState`, `ConnectAsync(ct)` forces eager connection (warm-up before first deadline-bearing call), `WaitForStateChangedAsync(lastObservedState, ct)` parks until departure from the observed state — the watch loop is `state -> wait -> re-read`, never polling. `ConnectAsync` is unavailable when the channel wraps a caller-supplied `HttpClient`.
- Oversize messages fail with `StatusCode.ResourceExhausted` after transfer/serialization work is already spent; caps are negotiated policy on both ends of a seam, and the producer-side pre-check before serialization is owned by the IPC framing law.

## seam vocabulary — typed port records

- The only cross-package transport seam is a fixed small set of typed port records: each record carries delegates (`Func`-shaped call arrows for the stream shapes the seam serves) plus policy values (deadline budget, cap pair, codec choice) — zero interfaces, zero inheritance, zero provider types in the signature. A consumer package binds the record; it never references channel, invoker, or handler types.
- The byte mover is a vocabulary column on each seam row, not a call-site branch: a seam row declares `(shape, transport, codec, cap, budget)` and the composition root materializes the delegate from the transport column — in-process direct call, UDS channel, remote TLS channel, and browser-translated channel are four column values over one row schema; `if (remote)` at a call site is the named defect.
- The aggregate port (one record bundling every operation a package exposes) and the provider-branded port (a record named after or shaped by the transport library) are the two named defect forms: the first re-creates an interface by another spelling, the second leaks the byte mover into consumer vocabulary.
- Generated clients are edge adapters materialized at the root from `GrpcChannel.CreateCallInvoker()`; consumers receive the port record's delegates closed over the generated client — the generated type itself never crosses a package boundary.

## transport rows with stream-shape capability columns

- Transport is a closed row axis with stream-shape capability columns; the row decides which shapes exist, the shape never decides the row:

| row | unary | server-stream | client-stream | duplex | mechanics |
| --- | --- | --- | --- | --- | --- |
| HTTP/2 TCP+TLS | yes | yes | yes | yes | default `SocketsHttpHandler` |
| HTTP/2 UDS | yes | yes | yes | yes | `ConnectCallback` row (IPC lane owns mechanics) |
| HTTP/3 | yes | yes | yes | yes | `HttpVersion` + `HttpVersionPolicy` on channel options |
| browser-translated | yes | yes | no | no | `GrpcWebHandler` wrapped inner handler |

- The browser row is a handler wrap, not a different client: `GrpcWebHandler(GrpcWebMode, HttpMessageHandler)` enters through `GrpcChannelOptions.HttpHandler`; `SendAsync` rewrites only `application/grpc` content and passes everything else through, response trailers are restored and the version normalized to HTTP/2 so generated clients are row-blind. `GrpcWebMode.GrpcWebText` (base64) is mandatory for server-streaming to actually stream in browser environments; binary `GrpcWeb` buffers. Client-streaming and duplex columns are structurally absent on this row — route those calls to an HTTP/2 row or reshape the contract.
- Long-lived duplex and server-stream columns require keep-alive rows on the handler: `SocketsHttpHandler.KeepAlivePingDelay`, `KeepAlivePingTimeout`, `KeepAlivePingPolicy` (`WithActiveRequests` pings only while calls are in flight; `Always` keeps idle channels alive for push streams); without pings an idle middlebox silently kills the connection and the next write surfaces as `Unavailable` minutes later.
- HTTP/2 multiplexing is capped by the server's max-concurrent-streams (commonly 100); beyond it, calls queue invisibly on one connection — `SocketsHttpHandler.EnableMultipleHttp2Connections = true` is the row column for high-fan-out clients, and `PooledConnectionIdleTimeout`/`PooledConnectionLifetime` govern pooled connection turnover.
- Per-shape lifetime law: `AsyncUnaryCall<T>` awaits once; `AsyncServerStreamingCall<T>` drains `ResponseStream`; `AsyncClientStreamingCall<TReq,TRes>` requires `RequestStream.CompleteAsync()` before the response materializes; `AsyncDuplexStreamingCall` requires `CompleteAsync()` to end the request side while the response side drains independently. All four are `IDisposable` — disposing an undrained call is the cancellation idiom, not a leak.
- Stream drain rides `ResponseStream.ReadAllAsync(ct)` (`IAsyncEnumerable<T>` adapter over `MoveNext`/`Current`); the adapter lives in the common transitive assembly, not the call-contract assembly — `await foreach` is the canonical drain spelling, hand-rolled `MoveNext` loops are flatter retellings.
- `IAsyncStreamWriter<T>.WriteAsync(message, cancellationToken)` exists alongside the token-less overload; `WriteOptions` with `WriteFlags.BufferHint`/`NoCompress` is per-writer policy. One write at a time per writer — concurrent `WriteAsync` is undefined; serialize producers through a channel lane upstream.

## credential rows

- Channel trust is a closed row set: `ChannelCredentials.Insecure` (plaintext rows: UDS, loopback dev), `ChannelCredentials.SecureSsl` (TLS), `ChannelCredentials.Create(channelCredentials, callCredentials)` (TLS plus per-call identity). Client TLS identity (mutual TLS) is not a credential row — it rides `SocketsHttpHandler.SslOptions` (`SslClientAuthenticationOptions`) on the transport row.
- Per-call identity is `CallCredentials.FromInterceptor(AsyncAuthInterceptor)` — the delegate `(AuthInterceptorContext, Metadata) -> Task` receives `ServiceUrl`, `MethodName`, `CancellationToken` and appends entries; `CallCredentials.Compose(params CallCredentials[])` stacks identities. Token refresh lives inside the interceptor delegate, never in call sites.
- Composed call credentials transmit only over TLS by default; `UnsafeUseInsecureChannelCallCredentials = true` is the explicit channel row that permits bearer material on a plaintext transport — the knob exists precisely for trusted-boundary rows where the transport itself is the perimeter; on any network row it is the named defect.

## one stamping interceptor

- The stamping interceptor is `CallInvoker.Intercept(Func<Metadata, Metadata>)` — the built-in metadata interceptor covers all five call shapes (blocking unary, async unary, server-stream, client-stream, duplex) from one delegate that receives existing headers (or a fresh `Metadata`) and returns the stamped set; subclassing `Interceptor` for header stamping re-implements what the package already derives.
- A full `Interceptor` subclass is earned only when the response side is inspected (receipt capture, trailer harvesting): the four async client overrides each receive a continuation delegate and return the call object — wrapping `ResponseAsync`/`ResponseStream` inside the returned `AsyncUnaryCall<TResponse>`/`AsyncServerStreamingCall<TResponse>` is the interception point; the request payload is mutated never, the context is rebuilt via `new ClientInterceptorContext<TReq,TRes>(context.Method, context.Host, context.Options.WithHeaders(...))`.
- Composition order is verified law: `Intercept(params Interceptor[])` applies `Enumerable.Reverse`, so the first array element is outermost and executes first on the way out; chained `a.Intercept(x).Intercept(y)` makes `y` outermost. Exactly one stamping interceptor per channel, installed at invoker creation — a second stamper is a merge conflict, not a layer.
- Binary metadata entries require the `-bin` key suffix (`Metadata.BinaryHeaderSuffix`); the entry constructor enforces the suffix for `byte[]` values and rejects it for string values, keys are lowercased, and `GetValueBytes(key)` is the read side. `Metadata.Empty` is a frozen instance — never mutate it; stamping always allocates.

## deadline and cancellation propagation

- `CallOptions(headers, deadline, cancellationToken)` is the per-call policy triple; the deadline is an absolute UTC instant transmitted as the wire timeout header so the server observes remaining budget, and every hop inherits it — a deadline is set once at the outermost call site from a budget policy value, inner hops only shrink it (server-side downstream propagation is the server lane's law).
- `ThrowOperationCanceledOnCancellation = true` converts deadline expiry and cancellation from `RpcException(DeadlineExceeded | Cancelled)` into `OperationCanceledException`, unifying transport cancellation with the process-wide cancellation rail so one exception-capture conversion at the boundary serves both; leaving it false forces dual-path capture at every call site.
- Failure taxonomy at the client seam: `RpcException.StatusCode` discriminates — `DeadlineExceeded` (budget spent; the client enforces locally even when the server never responds), `Cancelled` (caller token), `Unavailable` (connect refusal, connection loss, drain), `ResourceExhausted` (cap breach), `Unimplemented` (contract drift — method unknown to server). `RpcException.Trailers` carries the structured-fault payload (decode law lives with the server lane). The typed-rail projection maps this taxonomy once at the boundary bridge; matching on status strings anywhere interior is the rejected form.
- A deadline already in the past fails immediately with `DeadlineExceeded` without network I/O — clock skew between budget computation and call issue manifests as instant local failure, which is the diagnostic signature distinguishing local budget exhaustion from server slowness (no trailers, zero latency).

## retry-config interplay with the hop owner

- Transport retry is data, not code: `ServiceConfig.MethodConfigs` rows pair `MethodName` selectors (service/method, or empty for default-all) with exactly one of `RetryPolicy` or `HedgingPolicy` — the two are mutually exclusive per method row. `RetryPolicy` carries `MaxAttempts`, `InitialBackoff`, `MaxBackoff`, `BackoffMultiplier`, `RetryableStatusCodes`; `HedgingPolicy` carries `MaxAttempts`, `HedgingDelay`, `NonFatalStatusCodes`.
- When the channel retry row is enabled, the channel IS the hop's one retry owner — an outer resilience pipeline wrapping the same calls is the second-owner conflict the resilience law degrades and receipts; the composition rule is binary per seam row: channel-config retry (status-code-driven, wire-aware, hedging-capable) or pipeline retry at the seam, never both.
- Retry commitment is structural: once any response data is observed, or the buffered request bytes exceed `MaxRetryBufferPerCallSize`/`MaxRetryBufferSize`, the call commits to the in-flight attempt and stops being retryable — large request payloads silently exit retry protection at the 1 MiB per-call default, which is the non-obvious cap to align with payload-cap policy.
- `RetryThrottlingPolicy` (`MaxTokens`, `TokenRatio`) is the channel-wide brake: failures drain tokens, successes refill at the ratio, and retries pause below half tokens — it converts a downstream brownout into reduced retry pressure instead of amplification. `MaxRetryAttempts` on channel options caps whatever the service config requests.
- Hedging duplicates the call to multiple in-flight attempts after each `HedgingDelay`; it is admissible only for idempotent methods, and a non-fatal status on one attempt lets others continue — the receipt for a hedged call must record attempt cardinality or the diagnostics fold under-counts wire traffic.

## load balancing and resolution

- Balancing activates by URI scheme plus `ServiceProvider` on channel options: `dns:///host` enables periodic re-resolution against all A records; `static:///` with a registered `StaticResolverFactory(Func<Uri, IEnumerable<BalancerAddress>>)` serves fixed address sets (the multi-process suite row: addresses from a discovery manifest). `BalancerAddress(host, port)` wraps `DnsEndPoint`, with `BalancerAttributes` as the per-address metadata bag.
- Policy selection is a `ServiceConfig.LoadBalancingConfigs` row: `PickFirstConfig` (single connection, failover ordering) or `RoundRobinConfig` (connection per address, rotating picker); policy names are the wire constants `pick_first`/`round_robin`. Custom strategies implement `LoadBalancer`/`SubchannelPicker` with `PickResult.ForSubchannel`/`ForFailure`, registered as `LoadBalancerFactory` rows in the service provider — the picker runs per call, so it must be allocation-light.
- `Resolver.Refresh()` is the manual re-resolution hook — the seam between discovery-manifest change detection and balancer address updates; `DisableResolverServiceConfig` blocks resolver-supplied service config from overriding root-declared policy, which is the correct posture when policy is a compiled value.

## compression rows

- `CompressionProviders` on channel options registers `ICompressionProvider` rows keyed by `EncodingName`; gzip and deflate rows ship in the common assembly (`GzipCompressionProvider(CompressionLevel)`). Response decompression is automatic from the registry; request compression is per-call opt-in via the metadata key `grpc-internal-encoding-request` naming the encoding — a header-shaped policy switch, not a channel-wide default, so per-call payload entropy decides.
- `WriteFlags.NoCompress` on `WriteOptions` exempts individual messages inside a compressed stream — the row for mixed-entropy streams (already-compressed artifacts interleaved with compressible frames).

## divergent

- Channel-configuration depth as one policy record: every channel in a suite derives from one frozen channel-policy row set — `(authority, transport row, credential row, cap pair, retry-or-hedge row, balancer row, keep-alive row)` — materialized into `GrpcChannelOptions` by one fold at the composition root; the diff of adding a peer process is one row. The rejected forms this forecloses: per-call-site `ForAddress` scatter, handler construction inside services, and environment-conditional channel branches.
- Interceptor-deadline fusion: the stamping interceptor and the budget law compose into one seam — the stamper writes correlation/causal headers while the budget policy computes the deadline from a per-row time allowance; both are values consumed at `CallOptions` construction inside the port-record delegate, so a call site can neither omit the stamp nor invent a deadline. The hop-owner interplay then collapses to a per-row enum: `RetryOwner.Channel` (service-config row present, pipeline forbidden) vs `RetryOwner.Seam` (no method config, pipeline owns) — representable as one policy column, auditable without reading code.
- Stream-shape columns as carrier dispatch: the four shapes are one request-union dispatch where the case shape selects the column — a value case maps to unary, a sequence case to client-streaming, a subscription case to server-streaming, a duplex case to the bidirectional column; the port record exposes one polymorphic arrow per seam, and shape-suffixed method families (`CallX`/`CallXStream`/`CallXDuplex`) are the foreclosed spelling. Flow control falls out: the only backpressure primitives are await-on-`WriteAsync` (HTTP/2 window) and pull-on-`ReadAllAsync`; buffering between them re-introduces unreceipted loss, which the concurrency lane's lane law already rejects.
