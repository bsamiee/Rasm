# BCL Runtime Replacement Surfaces

[LOCK]:
- `System.Threading.Lock` is a purpose-built monitor type; `lock (obj)` where `obj` is a `Lock` instance compiles to `Lock.EnterScope()` rather than `Monitor.Enter/Exit`. This is a C# 13 language lowering rule — no attribute marks the type; the compiler pattern-matches `EnterScope()` returning a `ref struct` with `Dispose()`. `object` locks do not benefit from this path.
- `Lock.EnterScope()` returns `Lock.Scope`, a `ref struct` that exits the lock on `Dispose()`. The `using var` pattern is the preferred non-keyword form when scope boundaries must be explicit. `Lock.IsHeldByCurrentThread` enables fast re-entrancy checks without acquiring.
- `Lock` is sealed and carries no interfaces; it cannot be used as a `SyncRoot` or boxed. Code that passes `this` or a field as a lock object to `Monitor.Enter` must be rewritten; `Lock` fields are not compatible drop-in replacements — the `lock` keyword must target the `Lock` instance directly.
- `Lock` allows recursive entry up to an implementation-defined limit and throws `LockRecursionException` when that limit is exceeded; it is not a reentrant-once type like `ReaderWriterLockSlim`.
- `TryEnter(TimeSpan)` and `TryEnter(int millisecondsTimeout)` return `bool` and do not return a `Scope`. When the timeout path must also release via `using`, use `Enter()` + `Exit()` manually with the timeout result.

[TIMEPROVIDER]:
- `TimeProvider` exposes two virtual override points: `GetUtcNow()` and `GetTimestamp()`. `GetLocalNow()`, `GetElapsedTime(long)`, and `GetElapsedTime(long, long)` are concrete and derive from those two. A test double needs to override only `GetUtcNow` and `GetTimestamp` to control every derived time surface.
- `TimeProvider.CreateTimer(TimerCallback, object?, TimeSpan, TimeSpan)` returns `ITimer`, not `Timer`. `ITimer` exposes only `Change(TimeSpan, TimeSpan)`. A test `TimeProvider` can return a fake `ITimer` and advance time by invoking the callback directly, giving deterministic timer tests without wall-clock coupling.
- `Task.Delay(TimeSpan, TimeProvider)`, `Task.Delay(TimeSpan, TimeProvider, CancellationToken)`, `Task.WaitAsync(TimeSpan, TimeProvider)`, `Task.WaitAsync(TimeSpan, TimeProvider, CancellationToken)`, `PeriodicTimer(TimeSpan, TimeProvider)`, and `CancellationTokenSource(TimeSpan, TimeProvider)` all accept a `TimeProvider` — every time-dependent async primitive is testable without real wall-clock delay.
- `TimeProvider.GetElapsedTime(long startingTimestamp)` calls `GetTimestamp()` at call time to produce the end timestamp; it is not equivalent to `GetElapsedTime(start, GetTimestamp())` called ahead of time. This matters under a fake provider that advances the clock between calls.
- `TimeProvider.TimestampFrequency` must return a nonzero value on a custom provider. `GetElapsedTime` divides by `TimestampFrequency`; returning `0` produces division-by-zero in the concrete `GetElapsedTime` implementations.

[THROW_HELPERS]:
- The complete static throw-helper surface: `ArgumentNullException.ThrowIfNull(object?, [CAE] string?)` and its `void*` overload; `ArgumentException.ThrowIfNullOrEmpty(string?, [CAE] string?)` and `ThrowIfNullOrWhiteSpace`; `ArgumentOutOfRangeException.ThrowIfZero/ThrowIfNegative/ThrowIfNegativeOrZero` (constrained to `INumberBase<T>`), `ThrowIfGreaterThan/GreaterThanOrEqual/LessThan/LessThanOrEqual` (constrained to `IComparable<T>`), `ThrowIfEqual/ThrowIfNotEqual` (unconstrained `T`); `ObjectDisposedException.ThrowIf(bool, object)` and `ThrowIf(bool, Type)`.
- `[CallerArgumentExpression]` on the `paramName` or `value` parameter captures the argument expression string at compile time when the caller omits that argument. The pattern `ArgumentNullException.ThrowIfNull(value)` emits `"value"` into the exception automatically. Guard-block reimplementations — `if (x is null) throw new ArgumentNullException(nameof(x))` — are the deleted form.
- The `INumberBase<T>` constraint on `ThrowIfNegative`, `ThrowIfNegativeOrZero`, and `ThrowIfZero` means they accept signed and unsigned numeric types — not just `IComparable<T>` comparables. `ThrowIfEqual` and `ThrowIfNotEqual` carry no constraint, making them applicable to any equality-comparable type.
- `ObjectDisposedException.ThrowIf(bool, object instance)` uses `instance.GetType().FullName` as the type name in the message; the `Type` overload accepts a closed generic type and renders its name. Neither overload has a `[CallerArgumentExpression]` — the condition expression is not captured.
- The `other` bound parameter in `ThrowIfGreaterThan`, `ThrowIfLessThan`, and similar comparisons is not decorated with `[CAE]`; only the `value` or `paramName` argument captures as an expression. The bound value itself appears in the message as its formatted value, not as a source expression.

[RANDOM]:
- `Random.Shared` is a thread-safe `Random` instance backed by per-thread `Xoshiro256**` state — no lock on read. Use `RandomNumberGenerator` static methods for security-sensitive material.
- `Random.GetString(ReadOnlySpan<char> choices, int length)` allocates and returns a `string`. `Random.GetHexString(int stringLength, bool lowercase = false)` and its `Span<char>` sink overload are available on both `Random.Shared` and instance `Random`. These replace hand-rolled character selection loops.
- `Random.GetItems<T>(ReadOnlySpan<T> choices, int length)` allocates a `T[]`. `GetItems<T>(ReadOnlySpan<T> choices, Span<T> destination)` writes into a caller-owned buffer for zero allocation. `Random.Shuffle<T>(Span<T>)` and `Shuffle<T>(T[])` perform Fisher-Yates in place.
- `RandomNumberGenerator` mirrors the same static surface — `GetItems`, `GetString`, `GetHexString`, `Shuffle`, `Fill`, `GetBytes`, `GetInt32` — making the selection between pseudo-random and cryptographic independent of call-site shape; the only change is the owning type.

[CANCELLATION]:
- `CancellationTokenSource.CreateLinkedTokenSource(params ReadOnlySpan<CancellationToken> tokens)` avoids the `CancellationToken[]` allocation path for the variable-arity case. The two-token overload `(token1, token2)` and single-token overload `(token)` are optimized fast paths; the span overload is the correct form for three or more tokens.
- `CancellationTokenSource.TryReset()` resets a non-cancelled source to its initial state, allowing reuse across periodic operations without allocation. It returns `false` if cancellation was already requested and returns `true` only when the source is in a clean state after the reset.
- `CancellationTokenSource.CancelAsync()` schedules cancellation asynchronously and returns a `Task`. Unlike `Cancel()`, callbacks registered via `Register` are invoked on the thread pool rather than inline on the calling thread — avoids re-entrancy when callbacks dispatch further async work.
- `CancellationToken.UnsafeRegister(Action<object?>, object?)` registers a callback without flowing `ExecutionContext`. Where callbacks are known-safe closures with no ambient context dependency, this form eliminates the context capture allocation that `Register` incurs. The `Action<object?, CancellationToken>` overload of `UnsafeRegister` passes the token itself to the callback, removing the need to close over it.
- `CancellationToken.ThrowIfCancellationRequested()` does not allocate an exception object when cancellation is not requested; `IsCancellationRequested` avoids even the throw-path overhead for tight polling loops where cancellation is rare.

[PERIODIC_TIMER]:
- `PeriodicTimer.WaitForNextTickAsync(CancellationToken)` returns `ValueTask<bool>` — `false` signals disposal; `true` means the tick fired. The idiomatic loop is `while (await timer.WaitForNextTickAsync(ct))`. There is no callback, no re-entrancy risk, and no thread affinity.
- `PeriodicTimer.Period` is a readable and writable `TimeSpan` property. Changing it mid-loop takes effect on the next tick's scheduling; callers that need backpressure-sensitive intervals can write `Period` from inside the loop body.
- `PeriodicTimer(TimeSpan period, TimeProvider timeProvider)` advances with the provider's clock, enabling deterministic interval tests without sleeping.
- `PeriodicTimer` does not skip ticks; if processing takes longer than the period, the next tick fires immediately upon return from `WaitForNextTickAsync`. Callers that need skip-on-overrun behavior must compare elapsed time against the period inside the loop body.

[TASK_COMBINATION]:
- `Task.WhenEach` overloads — `(params Task[])`, `(params ReadOnlySpan<Task>)`, and `(IEnumerable<Task>)` — return `IAsyncEnumerable<Task>`, yielding tasks in completion order rather than input order. Generic overloads return `IAsyncEnumerable<Task<T>>`. This replaces `Task.WhenAny` retry loops that discard and rebuild task collections on each iteration.
- `Task.WhenAll` and `Task.WhenAny` have `params ReadOnlySpan<Task>` overloads that avoid array allocation when the task set is stack- or pool-allocated. `Task.WhenAll<T>(params ReadOnlySpan<Task<T>>)` returns `Task<T[]>` with the same semantics as the array form.
- `Task.WaitAsync(CancellationToken)`, `WaitAsync(TimeSpan)`, `WaitAsync(TimeSpan, TimeProvider)`, and `WaitAsync(TimeSpan, TimeProvider, CancellationToken)` wrap any task with a timeout or cancellation boundary without blocking. The returned task faults with `TimeoutException` on expiry or cancels with the token; the original task continues running — this is not cancellation of the underlying work.

[INTERLOCKED_VOLATILE]:
- `Interlocked.CompareExchange` and `Exchange` cover `byte`, `sbyte`, `short`, `ushort`, `uint`, `ulong`, and `nuint` in addition to `int`, `long`, `float`, `double`, and `T`. CAS loops on narrow integer state fields no longer require widening to `int`.
- `Interlocked.And` and `Or` exist for `int`, `long`, `uint`, and `ulong` and return the original value before the operation — the pre-operation value, not the result. `Interlocked.Increment` and `Decrement` cover `uint` and `ulong` as well as `int` and `long`.
- `Volatile.ReadBarrier()` prevents memory reads that follow it from being reordered before it; `Volatile.WriteBarrier()` prevents memory writes that follow it from being reordered before accesses that precede it. The producer pattern writes data, calls `WriteBarrier`, then publishes the flag; the consumer reads the flag, calls `ReadBarrier`, then reads the data. Both are cheaper than `Thread.MemoryBarrier()`, which issues a full bidirectional fence.
- `Interlocked.MemoryBarrierProcessWide()` issues a process-wide fence that makes all prior writes visible across all threads and processors — necessary after pointer-publishing on architectures with weak ordering. It is stronger and more expensive than `Thread.MemoryBarrier`.

[ARRAY_AND_MEMORY_POOLING]:
- `ArrayPool<T>.Shared.Rent(int minimumLength)` returns an array of at least `minimumLength` elements but potentially larger due to bucket alignment. `Return(T[], bool clearArray)` must be called with the same array reference; `clearArray: true` is required when the array held sensitive or reference-typed data to prevent leaks across rentals.
- `MemoryPool<T>.Shared.Rent(int minBufferSize = -1)` returns `IMemoryOwner<T>` whose `Memory<T>` property gives the usable segment. The segment length may exceed `minBufferSize`. `IMemoryOwner<T>` is `IDisposable` — ownership is explicit and lifetime-scoped, unlike `ArrayPool<T>`, which requires manual pairing of `Rent`/`Return`.
- `GC.AllocateUninitializedArray<T>(int length, bool pinned = false)` bypasses zero-initialization for blittable element types. The `pinned` flag allocates in the pinned object heap, eliminating GC pressure from repeated pinning. Use only when every element is written before any read; uninitialized reads are undefined behavior.
- `GC.AllocateArray<T>(int length, bool pinned = false)` zero-initializes and optionally pins to the pinned object heap, avoiding the GC relocation cost for long-lived buffers used in native interop.

[LAZY_INITIALIZATION]:
- `Lazy<T>(LazyThreadSafetyMode.PublicationOnly)` allows concurrent initialization races — multiple threads may construct `T` but only one published value wins; losers are discarded. Use when construction is idempotent and racing threads must not block.
- `LazyThreadSafetyMode.ExecutionAndPublication` (the default `isThreadSafe: true` mode) blocks competing threads behind a `Monitor` until the winner completes. If the factory throws, the exception is cached and re-thrown on every subsequent access — the `Lazy<T>` is permanently faulted. `PublicationOnly` does not cache exceptions; a faulted attempt allows the next thread to retry.
- `LazyThreadSafetyMode.None` provides no synchronization and is the highest-performance mode. Use it only when the `Lazy<T>` is guaranteed single-threaded — for example, a per-instance field accessed only from the owning thread.
- `LazyInitializer.EnsureInitialized<T>(ref T? target)` is the non-allocating alternative for reference types when the owning type manages its own synchronization. The single-reference overload uses an internal `Interlocked.CompareExchange` to publish the winner without a separate lock object; the three-argument form `(ref T target, ref bool initialized, ref object? syncLock)` is the general-purpose variant for value types and factory overrides.

[GC_CONTROL]:
- `GC.RefreshMemoryLimit()` re-reads the memory limit from the container or OS after a live cgroup change, without restarting the process — relevant for containerized workloads where memory limits are adjusted at runtime.
- `GC.GetConfigurationVariables()` returns `IReadOnlyDictionary<string, object>` of active GC configuration keys — heap size, generation limits, LOH threshold, concurrent GC mode. The returned values reflect the live configuration, which may differ from startup configuration if the JIT adjusted them.
- `GC.GetTotalPauseDuration()` returns the cumulative `TimeSpan` spent in GC pauses since process start — useful for SLO monitoring alongside `GC.GetTotalAllocatedBytes`.
- `GC.TryStartNoGCRegion(long totalSize)` suppresses GC for a region of known-bounded allocation. `RegisterNoGCRegionCallback(long size, Action)` triggers a callback when allocation within the no-GC region approaches the limit, allowing graceful fallback rather than forced collection.

[THREAD_POOL]:
- `ThreadPool.QueueUserWorkItem<TState>(Action<TState>, TState, bool preferLocal)` is allocation-free — the generic state parameter avoids boxing compared to the `WaitCallback`/`object` form. `preferLocal: true` queues to the calling thread's local work-stealing queue, reducing contention for work with producer locality.
- `ThreadPool.UnsafeQueueUserWorkItem<TState>(Action<TState>, TState, bool preferLocal)` and `UnsafeQueueUserWorkItem(IThreadPoolWorkItem, bool preferLocal)` suppress `ExecutionContext` flow. The `IThreadPoolWorkItem` form allows struct-typed custom work items with zero delegate overhead — implement `Execute()` on a struct to avoid heap allocation entirely.
- `Thread.UnsafeStart()` and `Thread.UnsafeStart(object? parameter)` start a thread without flowing `ExecutionContext` — appropriate when the thread establishes its own context and ambient values from the creating thread must not leak.
