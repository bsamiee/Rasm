# Cartesian Product Finiteness And Parser Context

[FINITENESS_IS_AN_AST_SPAN_PREDICATE_NOT_A_TYPE_INTUITION]:
- The synthesized template-literal vocabulary lowers to one AST shape — `head: string` plus `spans: NonEmptyReadonlyArray<TemplateLiteralSpan>` where each span is `{ literal: string; type: StringKeyword | NumberKeyword | Literal | TemplateLiteral | Union<...> }` — so the alphabet's finiteness is a structural property of the span `type` set, decidable by walking the spans, not a feeling about whether the type "looks closed".
- The span `type` alphabet is the exhaustive finiteness lattice: a `Literal` span contributes one fixed member, a `Union` of `Literal` spans a finite alternation, a nested `TemplateLiteral` span the finiteness of its own factors, while a `StringKeyword` or `NumberKeyword` span is the one open node — a single open span anywhere in the list opens the whole product, and the predicate is "every span `type` reduces to a `Literal`/finite-`Union`/finite-nested-`TemplateLiteral`".
- The closed product compiles to a regular expression of pure alternation — `getTemplateLiteralCapturingRegExp` maps a `Literal` span to `regexp.escape(String(literal))` and a finite `Union` span to `members.map(pattern).join("|")`, so the compiled source contains only escaped literals and `|` — while an open factor injects `STRING_KEYWORD_PATTERN = "[\\s\\S]*?"` or `NUMBER_KEYWORD_PATTERN = "[+-]?\\d*\\.?\\d+(?:[Ee][+-]?\\d+)?"`, the runtime witness that the alphabet is no longer enumerable: the finiteness the type system proves by `keyof typeof` over the product key set is the same finiteness the AST proves by alternation-only span patterns.
- The product cardinality is the runtime cost of the regexp: a closed product of factor arities `n₁·n₂·…` compiles to an alternation whose branch count is that product, so the cartesian growth the type-level union exhibits is the literal branch-count growth of the compiled pattern — adding a member to one factor multiplies both the synthesized union arity and the regexp alternation width, the cost visible in the AST before any decode runs.

[BOOLEAN_AND_BIGINT_FACTORS_ARE_FINITE_BY_CONSTRUCTION]:
- The span `type` lattice admits `StringKeyword` and `NumberKeyword` as open nodes but carries NO `BooleanKeyword` or `BigIntKeyword` node, so a boolean or bigint factor cannot enter the product as an open primitive — it enters only as a `Literal` span (`true`, `false`, a fixed `1n`) or a `Union` of such literals, which makes every boolean and bigint segment provably finite regardless of how it is spelled.
- The asymmetry is load-bearing for the finiteness audit: a `Schema.String` or `Schema.Number` factor is the ONLY way to open the product, and a `Schema.Boolean` factor — though `boolean` is a two-member union — synthesizes the closed `` `${boolean}` `` alphabet `'true' | 'false'` because the keyword has no open span representation, so a factor's openness is decided by its primitive identity, not its cardinality: the unbounded `number` opens, the two-valued `boolean` stays a closed two-branch alternation.
- This bounds where the structural-vs-open distinction actually fires: the only factor schemas that degrade `typeof Code.Type` from a closed union to a template-string pattern are the string and number keyword primitives and any schema whose encoded form lowers to them, so a product over literal unions, enums-as-literals, booleans, and fixed bigints is closed by construction and only a string/number-shaped segment forfeits the `as const satisfies` table contract over the product key.

[SEPARATOR_LITERALS_ARE_SPAN_SUFFIXES_NOT_FACTORS]:
- A literal passed between two variable factors does not become its own product dimension — the `TemplateLiteral` constructor folds an adjacent literal into the PRECEDING span's `literal` suffix field (consuming both params in one step), so `Schema.TemplateLiteral(Scope, '.', Kind)` builds spans `[{ type: Scope, literal: '.' }, { type: Kind, literal: '' }]` and the `'.'` never multiplies the cartesian set — the product is `|Scope|·|Kind|`, the separator a fixed glue between factors.
- A leading literal becomes the AST `head` rather than a span at all, and a trailing literal with no following variable becomes a span whose `type` is a `Literal` and whose `literal` suffix is empty — so the three literal positions (lead, inter-factor, trail) lower to three distinct AST slots (head, prior-span suffix, own `Literal` span), none of which is a multiplicative factor; only a variable `type` node contributes to the cartesian arity.
- The synthesized type mirrors the fold by recursing `GetTemplateLiteralType<Params> = Params extends [...infer Init, infer Last] ? AppendType<...Init, Last> : ''` right-to-left, where `AppendType` interpolates a literal `Next` as `` `${Template}${Next}` `` and a schema `Next` as `` `${Template}${A}` `` distributing over the schema's union — so the separator literals appear in the produced string but never widen the union arity, exactly as the AST fold keeps them out of the span `type` count.
- This is why a closed composite key stays closed across any punctuation: arbitrary fixed separators (`.`, `:`, `/`, multi-character glue) interleave the factors with zero effect on the product cardinality or the `as const satisfies Record<typeof Code.Type, Row>` table totality, the punctuation a presentational detail of the synthesized key, never a dimension the behavior table must cover a row for.

[THE_PARSER_IS_THE_VALIDATOR_AS_THE_FROM_SIDE_OF_A_TRANSFORM]:
- `TemplateLiteralParser` is not a parallel constructor but a `transformOrFail(from, to)` whose `from` IS the validator over the joined ENCODED string and whose `to` is `Tuple(...elements)` — the parser builds `from = TemplateLiteral(...encodedSchemas)`, compiles its capturing regexp once, and the decode runs `re.exec(input)` then re-decodes the captures through the tuple, so the string vocabulary and the structured vocabulary are one schema's two sides, never two independent owners.
- The decode body is `match ? succeed(match.slice(1, params.length + 1)) : fail(Type(...))` — the validator's capturing regexp extracts the segment captures, and the tuple `to` schema re-decodes each capture through its own element codec — so the validator answers "does the string match the pattern" and the same compiled artifact, run with capture groups, answers "what are the segments", the parse a strict refinement of the validation that additionally retains the captures.
- The transform is declared `strict: false` because the decode hands the tuple schema the raw STRING captures and lets each element codec coerce them (a numeric capture is the string `"42"` the element decodes to `42`), so the from-encoded and to-decoded shapes legitimately differ per element and the relaxed strictness is the type-level admission that the captures are pre-coercion strings the tuple elements finish decoding.
- Because the validator is literally the parser's `from`, every finiteness property of the product transfers unchanged: the parser's matching cost, alternation width, and open-vs-closed character are the validator's, and the only thing the parser adds over the validator is the capture retention plus the tuple re-decode — the structured tuple is the validator's match decomposed, never a second pattern compiled.

[CAPTURE_SLICE_ALIGNMENT_IS_THE_PARSER_INVARIANT]:
- The decode's `match.slice(1, params.length + 1)` takes exactly the first `params.length` capture groups as the tuple slots, so the parser's correctness rests on the capturing regexp emitting one top-level group per param in param order — `getTemplateLiteralPattern(ast, capture=true, top=true)` wraps the head and each span suffix literal in `(…)` at the top level and each span `type` in `(…)`, the group sequence aligning one-to-one with the param sequence the validator folded.
- A nested `Union` span is wrapped non-capturing `(?:…)` when not at the top level (`handleTemplateLiteralSpanTypeParens` returns `(?:${s})` for a union below top), so the alternation's internal branches do NOT each open a capture group that would shift the slice — the union contributes exactly one top-level capture for its whole alphabet, preserving the positional alignment the slice arithmetic depends on, the non-capturing wrap the mechanism that keeps a multi-member factor a single tuple slot.
- The separator literals, absorbed into span suffixes, ARE captured as their own top-level groups and therefore occupy their own tuple slots — this is why a parser's decoded tuple carries the punctuation as discriminable unit-literal elements (`readonly ['<scope-a>' | '<scope-b>', ':', number]`) while the validator's synthesized string carries the same punctuation as inert glue: the slice keeps every captured group, separators included, so the tuple is wider than the variable-factor count by exactly the captured separators.
- The encode is the inverse with no regexp — `tuple.join("")` concatenates the decoded slots back to the matching string — so the parser's two directions are asymmetric in machinery (capturing regexp forward, plain join backward) but symmetric in the alphabet: a value encodes to a string the validator admits because the tuple slots were the validator's captures, the round-trip closed by construction rather than by a second pattern.

[PER_SEGMENT_COERCION_IS_WHY_A_DECODING_SEGMENT_DEMANDS_THE_PARSER]:
- The parser coerces each element through `getTemplateLiteralParserCoercedElement(encoded, schema)`: a `NumberKeyword` encoded element becomes `compose(NumberFromString, schema)`, a non-string `Literal` element (a numeric/boolean/bigint separator) becomes `transform(Literal(String(literal)), schema, …)` bridging the captured string to the literal value, and a `Union` element recurses coercing each member — so the parser's decode actually transforms the captured string segments into typed parts, the act the validator structurally cannot perform.
- The validator `TemplateLiteral` produces a `Type` that IS the string union and has nothing to decode into — its result is `SchemaClass<A>` over the pattern string, no transform runs, no capture is retained — so handing it a `NumberFromString` segment is wasteful: the segment's decode never executes, the matched substring stays an undifferentiated part of the string, and the typed `number` the segment would produce is unreachable; the constructor selecting validate-only versus decode-into-parts is the choice of whether a segment's transform is ever run, never a flag on one constructor.
- A wire-decoding segment — a `NumberFromString` port, a `transformLiterals` channel remap inside a position, a branded refinement of a captured part — therefore DEMANDS `TemplateLiteralParser` because the coercion that produces the typed part is built only there, and the same params handed to `TemplateLiteral` type-check only when every segment is context-free and still discard the decoded parts: the parser is mandatory exactly when `Encoded` differs from `Type` for any segment, the identical mandate the decode gate carries over a value whose wire shape is not its domain shape.
- The coercion is positional and independent: each tuple slot decodes through its own element codec with no cross-slot dependency, so a parser mixing a fixed-literal separator slot, a `NumberFromString` slot, and a literal-union slot runs three independent segment decodes whose failures localize to the offending slot's `path` in the `ParseError` — the structured tuple's per-slot codec is the per-segment admission the flat string vocabulary collapses into one opaque match-or-fail.

[CONTEXT_IS_FORBIDDEN_WHERE_NO_TRANSFORM_CONSUMES_IT]:
- The validator's `AnyNoContext` param constraint is not a stylistic restriction but a structural consequence: requirement `R` is consumable only by a decode transform, the validator runs none (its result is the matched string), so a context-carrying segment would carry a requirement with no discharge site — the `Schema<any, any, never>` bound is the type system refusing to admit a service-dependent segment into a constructor that never runs its decode, the identical shape as the synchronous decode carriers constraining their schema to `R = never` because an already-resolved `Either`/`Option` has no channel to thread context.
- The parser's `R = Schema.Context<Params[number]>` accumulates the union of every segment's requirement exactly as a struct extension unions its members' `R`, so a parser whose port segment decodes through a service-backed codec surfaces that service in the whole parser's requirement — the composite string-to-tuple vocabulary declares one requirement recovered from the param list, the segment requirements discharged once at the parser's own provision, never per segment.
- The constructor identity therefore IS the requirement's discharge proof: a segment carrying context type-checks only in the parser form because only there does its transform run to consume the context, so the moment a factor must decode a captured part through a context-dependent codec the validator is rejected at the constructor call and the parser is the sole admitting owner — provenance selecting the constructor the way a value's provenance selects the decode-vs-dispatch gate, the context channel the static signal that the segment's decode is load-bearing rather than discarded.

```typescript
import { Schema, Match, Effect, type ParseResult } from 'effect'

const Scope = Schema.Literal('<scope-a>', '<scope-b>') // a closed finite factor; a Schema.String here injects [\s\S]*? and opens the product
const Port = Schema.NumberFromString // a decoding segment: Encoded string, Type number — its coercion runs ONLY in the parser

const Addr = Schema.TemplateLiteralParser(Scope, ':', Port) // Type readonly ['<scope-a>'|'<scope-b>', ':', number]; R = Schema.Context<params> = never here
const decodeAddr: (u: unknown) => Effect.Effect<typeof Addr.Type, ParseResult.ParseError> = Schema.decodeUnknown(Addr)
const encodeAddr = Schema.encodeSync(Addr) // (a: typeof Addr.Type) => `${'<scope-a>'|'<scope-b>'}:${string}` — the inverse is tuple.join(''), no regexp

const reach = (raw: string): Effect.Effect<number, ParseResult.ParseError> =>
    Effect.map(decodeAddr(raw), ([scope, _sep, base]) => // the ':' separator is its own captured tuple slot, discarded by binding
        Match.value(scope).pipe( // scope's slot type IS the closed factor alphabet; the parse already proved the punctuation matched
            Match.when('<scope-a>', () => base + 1000),
            Match.when('<scope-b>', () => base + 2000),
            Match.exhaustive, // total because the slot is the finite factor union; widening Scope breaks this at compile time
        ),
    )
```

[A_PRODUCT_SUBSET_IS_RECTANGULAR_OR_IT_IS_A_UNION_OF_PRODUCTS]:
- A product alphabet is closed under per-FACTOR narrowing but not under arbitrary member selection: a subset expressible as a product of per-factor subsets (a rectangle in the cartesian grid) is one re-synthesis from narrowed factors, while a subset that picks composite members across factor boundaries (a non-rectangular region of the grid) cannot be a single product at all — the product form admits exactly the rectangular subsets, and a non-rectangular admitted set is the signal the vocabulary is a `Schema.Union` of several products, each rectangle its own single-source product.
- The decision between one product and a union of products is geometric: variation at fixed positions over a shared format is one product narrowed by its factors, variation in the format itself (different separator structure, different segment count, members that pair only in specific combinations) is a union of products — so a vocabulary whose admitted composites do not factor cleanly is over-modeled as one template and under-modeled as a flat literal list, the correct owner the disjunction of the products its grid decomposes into.
- The parser's `.params` enumerates the FACTORS and separators at runtime, never the product members — the structured analogue of a `Literal`'s `.literals` reads the constituent dimensions, so walking `.params` recovers the grid axes while the grid cells stay implicit, and a runtime need for the cells reconstructs them as the cartesian product of each factor's own `.literals` joined through the separators, computed from the same single sources the type-level product derives from rather than a hand-maintained array of composite strings drifting from its factors.

[DECOMPOSITION_DISSOLVES_THE_PRODUCT_TABLE_INTO_PER_FACTOR_LOGIC]:
- The parser's tuple recurses LEFT-to-right (`GetTemplateLiteralParserType<Params> = Params extends [infer Head, ...infer Tail] ? readonly [SlotType, ...rest] : []`), the opposite fold direction from the validator's right-associative `[...Init, Last]` string synthesis — so the same param list produces a flattened string by right-fold and a positional tuple by left-fold, the two folds the structural reason the validator's `Type` is a single union while the parser's `Type` is an indexable product preserving each factor's position.
- A behavior that FACTORS — its per-composite logic decomposes into independent per-segment contributions — does not need the full product table at all: the parse splits the product key into per-factor slots each carrying its own finite sub-alphabet, so `Match.value(slot).pipe(Match.exhaustive)` over one decomposed factor replaces the `as const satisfies Record<typeof Code.Type, Row>` table whose row count is the FULL product, collapsing `n₁·n₂·…` rows to `n₁ + n₂ + …` per-factor arms.
- The product table and the per-factor dispatch are the two owners of the same factors, chosen by whether the behavior is separable: a behavior depending on the COMBINATION (a policy that differs for `'<scope-a>'+'<kind-b>'` specifically) needs the product table read by the recomposed composite key, a behavior depending on each factor INDEPENDENTLY (a scope-derived base plus a kind-derived offset) needs only per-slot dispatch over the decomposed tuple — the table is the dense owner exactly when the cells are irreducible, the per-slot dispatch exactly when they factor, and modeling separable logic as a full product table is the row-count blowup the decomposition deletes.

```typescript
import { Schema, pipe } from 'effect'

const Scope = Schema.Literal('<scope-a>', '<scope-b>', '<scope-c>') // one factor; its .literals enumerates this slot's sub-alphabet at runtime
const Kind = Schema.Literal('<kind-a>', '<kind-b>')

// product narrowing is a FACTOR operation: pickLiteral the factor, re-synthesize — the product, its regexp width, and any table over it all contract from this one edit
const InternalScope = pipe(Scope, Schema.pickLiteral('<scope-a>', '<scope-b>')) // a non-rectangular composite subset would be a Schema.Union of products instead
const Code = Schema.TemplateLiteral(InternalScope, '.', Kind) // closed product: |InternalScope|·|Kind| members; a Schema.String factor would open it to a pattern

const Parsed = Schema.TemplateLiteralParser(InternalScope, '.', Kind) // Type readonly ['<scope-a>'|'<scope-b>', '.', '<kind-a>'|'<kind-b>'] — separators are captured slots
const members: ReadonlyArray<string> = InternalScope.literals.flatMap((s) => Kind.literals.map((k) => `${s}.${k}`)) // product reconstructed from factor .literals, never a hand-listed composite array
```
