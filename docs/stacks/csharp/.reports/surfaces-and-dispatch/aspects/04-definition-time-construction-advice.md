# Definition-Time Construction Advice

[FACTORY_HOOK_REF_NORMALIZES_THE_ADMITTED_VALUE]:
- The factory-path advice point is one author-supplied `static partial` whose key parameter is `ref`, not `ref readonly` or `in` — assigning through it rewrites the value the backing field receives, so normalization and admission are one weave, never a pre-pass that hands a clean value to a second validator. The generator emits the stub and detects the body purely by the method's exact name and `static` non-generic shape; no attribute marks it, which is why renaming it silently un-weaves the advice without a diagnostic.
- The error channel is a leading `ref` slot of the type's validation-error type, so the hook both mutates the value and raises a typed rejection through the same call; the rejection is a value written into the slot, never a thrown exception, and the generated factory constructs only when that slot stays null. A `Create` raises the slot as a thrown rejection; a `TryCreate` returns it as the out-error — one hook body serves both factory spellings because the rail bridge happens in the generated wrapper, not in the advice.
- The hook does not see the format provider even though the enclosing factory does: the provider parameter is consumed by the generated parse/convert layer that runs before the hook, so locale-dependent admission must live in the validation-error type's own parsing, not in the normalization advice. A hook that needs culture is the signal that the key member's own parsable surface is the correct weave point instead.

```csharp
// admit-and-normalize in one weave; the ref rewrites the stored value
static partial void ValidateFactoryArguments(ref ValidationError? error, ref string value)
{
    var trimmed = value.AsSpan().Trim();
    error = trimmed.IsWhiteSpace() ? ValidationError.Create("<blank>") : null;
    value = trimmed.ToString();              // normalized value reaches the backing field
}
```

[NULL_GUARD_PRECEDES_THE_HOOK_SO_ABSENCE_IS_NOT_ADVISABLE]:
- The generated factory runs its null and blank-string short-circuit before the advice fires; when the configured absence policy yields null, the factory returns a null instance with a null error and the hook never executes. Absence is therefore not a thing the normalization advice can validate or repair — it is decided by the absence-policy knobs on the owner, and the hook only ever sees a present, non-blank value. Code that tries to reject null inside the hook is dead.
- The blank-string-yields-null policy and the null-yields-null policy are independent definition-time settings on the owner; one treats whitespace as absence, the other treats only true null as absence. Selecting the blank policy moves the whitespace decision out of every call site and into the generated guard, which is why a boundary value object for wire input chooses it at definition rather than re-checking emptiness downstream.

[CONSTRUCTOR_HOOK_GUARDS_THE_REHYDRATION_PATH_THE_FACTORY_NEVER_TOUCHES]:
- A second `static partial` advice point fires inside the instance constructor, with every assignable member as a `ref` parameter and no error slot — it is the only weave point on the deserialization, ORM-materialization, and direct-construction paths, none of which route through the validating factory. The factory-path hook does not run there, so an invariant that must hold on a rehydrated instance lives in the constructor hook and signals by throwing, while the same logic on the admission path populates the error slot instead.
- The two hooks share their normalization but split their failure rail: the factory hook writes a typed rejection value, the constructor hook can only throw, because rehydration has no result rail to fail into. A fully defended owner centralizes the rewrite in one private static and calls it from both hooks — soft-fail on admission, hard-assert on rehydration — so the normalization is declared once and the failure shape is chosen by which path reached it.
- A complex multi-member owner emits both hooks with one `ref` parameter per assignable member, the error slot still leading on the factory variant; per-member null guards precede the factory hook in member-declaration order, so a member already short-circuited to absence is not visible to the cross-member advice, and any invariant spanning two members must tolerate either being the absence default.

[HOOK_RETURN_TYPE_REWIRES_A_SECOND_GENERATED_PARTIAL]:
- Declaring the factory hook with a non-void return type rather than the void form silently changes the signature of a separate generated post-construction partial: the returned value is captured and forwarded into a post-init advice point that otherwise emits parameterless. The hook's return type is thus a definition-time switch over a second weave point's shape — a coupling invisible at the call site, where changing void to a returning form alters which post-init overload the owner must implement.
- The post-construction partial runs after a successful construct, receiving the forwarded factory diagnostic, and is the place for derived-field caching or registration that must observe the finished instance; it never runs on the rejection branch, so it carries no failure rail and is purely an after-the-fact observation weave.

[BEHAVIOR_ROWS_WEAVE_A_CONSTRUCTOR_DELEGATE_FIELD]:
- Marking a partial method on a vocabulary owner as delegate-from-constructor makes the generator emit a per-item delegate field wired through the constructor: each static row passes a method reference at its own construction, and the shared partial method forwards to that field. The behavior travels with the row, not in a dispatch arm, so the entrypoint is the one partial method and the variance is the field — no switch over the vocabulary exists anywhere.
- The emitted field's type is a synthesized named delegate when the method shape needs it — named `<Method>Delegate` by default, or a supplied name — and a plain `Func<…>`/`Action<…>` otherwise; the named form is what carries by-ref or span parameters a `Func` cannot express. Choosing the delegate name at definition time is the lever for two vocabularies whose rows must carry structurally distinct behavior signatures without colliding on a shared synthesized type.

[COMPARERS_AND_PROTOCOL_INTERFACES_ATTACH_AS_DEFINITION_TIME_ACCESSORS]:
- Equality and ordering attach through a comparer-accessor type named at the member, not through a hand-written `Equals`/`CompareTo`: a static accessor exposing a comparer is bound to the key member or to one member of a complex owner, and the generated equality and comparison surface routes through it. The same accessor type satisfies both the equality and the ordering accessor contract, so one definition-time choice fixes both `==` and ordering for that member, and a per-member accessor on a complex owner overrides the type-wide default for that field alone.
- The interface-implementation generators each gate on the member's own capability plus the per-owner skip flags: the formattable, comparable, parsable, and span-parsable surfaces emit only when the key member already satisfies the underlying capability and the corresponding skip is off. Suppressing the parsable surface cascades to suppress the span-parsable surface, and suppressing the factory methods cascades to suppress parsable entirely — the skips are not independent knobs but a derivation where the broader suppression forces the narrower, so a single skip can remove a chain of generated protocol surfaces.
- Equality-comparison operators are monotonically coerced up to the ordering-operators setting: requesting ordering operators silently drags the equality operators along to at least the same generation level, and the type-wide equality suppression collapses both to none. Ordering without equality is therefore not expressible through the operator knobs alone; the coercion is a definition-time invariant, not a default a call site can locally undo.

[SERIALIZATION_WEAVES_ONLY_FOR_REFERENCED_FRAMEWORKS]:
- The serializer-generation stage keys off the compilation's metadata references, not off an attribute alone: the converter for a given serialization framework emits only when that framework's assembly is actually referenced, so the framework-selection flags on the owner are a ceiling the present references intersect. An owner declaring all-frameworks emits exactly the converters whose assemblies are on the reference graph, and adding a framework reference later weaves its converter with no edit to the owner.
- A foreign or wire-shaped construction surface attaches through a factory attribute carrying its own serialization-usage flags and a constructor-correspondence marker, so one owner can expose a validated domain factory and a separate deserialization-only factory whose usage flags route each serializer to the correct entry. The correspondence marker tells the generator the foreign factory mirrors a real constructor, which is what lets the model-binding and serialization weaves target it without a hand-written converter — the protocol shape stays at the edge attribute, the domain factory stays clean.
