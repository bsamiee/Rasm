# Dense Algebra — Deepening

[SVD_SOLVE_TRUNCATION_GAP]:
- `SvdSolveFactored` (the double managed provider path) executes `num2 /= s[j]` for every `j < min(m,n)` with no zero guard and no truncation. If any singular value is numerically near-zero but not exactly zero, the solve produces a numerically exploded solution. The pseudo-inverse path (`Matrix.PseudoInverse()`) applies a `σ_max · ε_machine · max(m,n)` threshold before inversion and is the only surface with truncation. A caller that holds a `DenseSvd` handle and calls `Solve(b, x)` receives no truncation; the threshold from `Svd.Rank` is never consulted by `Solve`. As a result, `Solve` and the `Rank` / `ConditionNumber` evidence on the same handle are computed under different conventions and can disagree — `Rank` can report deficiency while `Solve` still divides by all singular values.
- The SVD singular values in the managed provider are delivered in descending order by an insertion-sort step in the convergence loop. `S[0]` is guaranteed to be the largest, making `L2Norm = |S[0]|` and `ConditionNumber = |S[0]| / |S[min(m,n)-1]|` correct without the caller sorting.

[EVD_RANK_THRESHOLD_FAILURE]:
- `Evd.Rank` and `Evd.IsFullRank` test `eigenValue.AlmostEqual(Complex.Zero)`, which resolves to `|eigenvalue_norm - 0| < DefaultDoubleAccuracy` where `DefaultDoubleAccuracy = 10 · 2⁻⁵³ ≈ 1.11e-15` — a fixed absolute threshold independent of matrix scale. For a well-scaled covariance matrix with true smallest eigenvalue ~1e-12, the rank probe classifies that eigenvalue as non-zero, returning full rank even though the numeric problem is near-singular. `Svd.Rank` uses `S.Maximum().EpsilonOf() · max(m,n)`, scaling to the spectral radius; for the same matrix it detects the deficiency. EVD-based rank is unreliable as a conditioning gate and must not substitute SVD rank evidence.
- `Evd.Determinant` computes `one.Magnitude` from `Π eigenvalues[i]` — an absolute-value product of complex eigenvalues. For a symmetric matrix with a negative eigenvalue (indefinite matrix passed with `Symmetricity.Symmetric`), the true determinant is negative but `Evd.Determinant` returns a positive value. Only `LU.Determinant` carries the correct sign. For algorithms that need `sign(det(A))` — detecting a definite-to-indefinite transition — `Evd.Determinant` is incorrect.

[QR_ISFULRANK_THRESHOLD_INCONSISTENCY]:
- `QR.IsFullRank` checks `Math.Abs(FullR.At(i, i)).AlmostEqual(0.0)`, which resolves to `Math.Abs(|R[i,i]|) < DefaultDoubleAccuracy = 10 · 2⁻⁵³`. This is an absolute threshold at the magnitude of machine epsilon, not a relative threshold scaled to the matrix. For a problem where R has a legitimate smallest diagonal entry ~1e-8 (rank-deficient relative to problem scale but far above machine epsilon in absolute terms), `IsFullRank` returns `true`. The correct rank gate scales to the largest diagonal of R: `|R[i,i]| / |R[0,0]| < sqrt(eps)` is the standard relative pivot criterion. All diagonal accesses go through the virtual `Matrix<T>.At(i, i)` dispatch — one virtual call per diagonal element in both `IsFullRank` and `Determinant`.
- For `QRMethod.Full`, the lazy `QR<T>.R` property allocates a new dense matrix and extracts the upper triangle from `FullR` on first access. `FullR` (not `R`) is used internally in every `Solve` call. Accessing `qr.R` from performance-sensitive code during repeated solves allocates unnecessarily. Under MKL, accessing `qr.Q` in a hot solve path also incurs cost because MKL preserves the compact Householder form in `Tau` and materializes Q on demand; do not access `qr.Q` in hot paths under the native provider.

[CHOLESKY_DETERMINANT_LOG]:
- `Cholesky<double>.DeterminantLn` computes `2 · Σ log(L[i,i])` using `Factor.At(i, i)` — one virtual call per diagonal element — and is numerically stable for large n where `Determinant = Π L[i,i]²` underflows to zero. Gaussian process log-likelihood, Kalman filter innovation covariance, and multivariate normal normalization all require `log|Σ|`; `DeterminantLn` is the correct surface — `Math.Log(Determinant)` silently returns `-Inf` for large SPD matrices because `Determinant` returns `0.0` without any underflow indication.
- The column-major diagonal of the lower Cholesky factor is accessible as a strided pattern: element at column `i` is at `values[i * n + i]`. A `TensorPrimitives.Log<double>(diagonalSlice, logSlice)` followed by `TensorPrimitives.Sum<double>(logSlice) * 2.0` replaces n virtual-dispatch scalar calls with a single SIMD pass for `DeterminantLn`. At n = 200 this is ~200 virtual calls vs. one AVX2-vectorized log+sum; the strided nature requires projecting the diagonal into a contiguous span before calling `TensorPrimitives`.

[ARRAY_ALIASING_HAZARD]:
- `new DenseMatrix(rows, columns, storage)` binds the provided `double[]` directly to `DenseColumnMajorMatrixStorage<T>.Data` without copying. Any subsequent factorization that calls `matrix.Clone()` internally (all six `Dense*.Create` paths) produces an independent copy; the original array is preserved. However, `MapInplace` and provider-level operations targeting a `DenseMatrix` constructed from an external array mutate that array in place. If the same `double[]` is wrapped into two `DenseMatrix` handles, any in-place operation on one mutates the other's view without exception.
- The canonical safe pattern is `matrix.Clone()` before any factorization call when the input originates from an external buffer, and reserving direct array construction exclusively for cases where the hot-path allocation cost must be avoided and the aliasing scope is single-owner. `Cholesky.Factorize(Matrix<double>)` on a retained handle copies via `Buffer.BlockCopy` into the handle's internal factor buffer and never mutates the input — this is safe for external arrays.

[WEIGHTED_REGRESSION_DENSIFICATION]:
- `WeightedRegression.Weighted<T>(x, y, w)` forms `xᵀ(Wx)` by evaluating `x.TransposeThisAndMultiply(w * x)`. The sub-expression `w * x` always allocates a new m×n dense matrix regardless of whether `w` is diagonal — the `Matrix<T>` multiplication operator has no diagonal-aware fast path when the left operand is `DiagonalMatrix`. The efficient form for a diagonal weight vector `wDiag: double[]` is to scale each row `i` of `x` by `Math.Sqrt(wDiag[i])` into a working matrix `xw`, then compute `xw.TransposeThisAndMultiply(xw)` — this avoids the m×n dense intermediate and executes one half-multiplication.
- `NormalEquations<T>` has no singularity check before calling `Cholesky()`. When the design matrix is rank-deficient, `Cholesky()` throws `ArgumentException("Matrix must be positive definite.")` without identifying which column caused the pivot failure. Wrap the `Cholesky()` call in `Fin<Cholesky<T>>` and treat the thrown `ArgumentException` as a `ConditioningFailure` evidence tag at the admission boundary.

[EVDSOLVE_MISSING_BLAS_PATH]:
- `DenseEvd.Solve` for the symmetric case executes two scalar loops: one accumulates `Vᵀb` (accessing `EigenVectors.Values[]` directly as a column-major flat array, but reading `input` via virtual `At(k, i)`), and one accumulates `V(Λ⁻¹Vᵀb)` (accessing `EigenVectors.Values[]` directly, writing `result` via virtual `At(l, i, num2)`). Neither loop routes through `MatrixMultiplyWithUpdate` or any provider path, so they receive no BLAS3 acceleration under MKL. EVD solve throughput does not scale with `Control.MaxDegreeOfParallelism`, unlike Cholesky, LU, and QR solves.
- `DenseEvd.Create` passes `matrix.Values` directly to `LinearAlgebraControl.Provider.EigenDecomp`, which allocates `matrixEv` internally as an n×n buffer. Total allocations for EVD of an n×n matrix include the `matrixEv` n² buffer in the provider plus intermediate working storage. For a symmetric system where EVD is needed only for eigenvalue extraction, prefer Cholesky for the actual solve.

[TENSOR_PRIMITIVES_DENSE_FLOOR]:
- `System.Numerics.Tensors.TensorPrimitives` on net10.0 provides `FusedMultiplyAdd<T>` requiring `IFloatingPointIeee754<T>` — satisfied by `double` — which executes multiply-add in a single rounding for improved inner-product accuracy vs. separate `Multiply + Add`. For dense residual computation after a factorization solve — `r = A·x - b` — the correct vectorized form accumulates each column contribution via `TensorPrimitives.MultiplyAdd(columnSpan, x[col], accumulatorSpan, accumulatorSpan)` over the column-major layout, processing one full n-element column per call.
- `TensorPrimitives` has no GEMM surface on net10.0. Dense GEMM remains the exclusive domain of the MathNet provider (managed or MKL). `TensorPrimitives` is the floor for element-wise and reduction hot loops that the provider does not own: all-finite checks via `IsFiniteAll<T>`, diagonal log-sum for `DeterminantLn`, residual norm computation, and symmetric force on the `Values` span.
- `IsFiniteAll<T>` is available in `System.Numerics.Tensors` under the constraint `T : INumberBase<T>`, which `double` satisfies. It maps to a vectorized SIMD predicate over the entire flat array in a single call, replacing a scalar LINQ `All(double.IsFinite)` that would use interface dispatch per element. `IsNaNAny<T>` and `IsInfinityAny<T>` are available as early-exit alternatives that identify which condition failed in the evidence receipt.

[COLUMN_MAJOR_LAYOUT_EXPLOITATION]:
- All `DenseColumnMajorMatrixStorage<double>` data is laid out as `Data[col * RowCount + row]`. Column `c` occupies `Data[c*n .. (c+1)*n - 1]` as a contiguous slice — `new ReadOnlySpan<double>(values, c * n, n)`. Row access is strided at stride `n`. Dense algorithms that operate column-by-column can use direct span slicing on `matrix.Values` without marshaling. The managed `MatrixMultiplyWithUpdate` pre-extracts columns of B exactly this way; replacing its scalar inner dot loop with `TensorPrimitives.Dot` on each extracted column is the primary vectorization opportunity in the managed GEMM kernel.
- The Cholesky inner loop (`DoCholeskyStep`) operates column-by-column on the factor buffer. The rank-1 update for each column corresponds to a SAXPY-style pattern `targetCol -= multiplier * sourceCol` on contiguous column spans — precisely the shape `TensorPrimitives.MultiplyAdd(sourceColSpan, -multiplier, targetColSpan, targetColSpan)` would vectorize. The managed Cholesky does not use `TensorPrimitives`; replacing the inner update with a `MultiplyAdd` span call is a direct vectorization path requiring no provider involvement.

[FACTORIZATION_RESULT_BUFFER_REUSE]:
- All six `Dense*.Create` paths call `matrix.Clone()` as their first operation. The clone is the internal factor buffer and is never released to the caller. For repeated factorizations of identically-sized matrices (MCMC covariance updates, scattered-data interpolation), the clone allocation dominates at small n because the managed allocator cannot reuse the previous factor buffer across construction calls. `Cholesky.Factorize(newMatrix)` on a retained handle reuses the existing factor buffer via `Buffer.BlockCopy(newMatrix.Values → factor.Values)` — the only in-place refactorization path in the admitted surface. No equivalent `Factorize` mutation method exists on `LU`, `QR`, `Svd`, or `Evd`.
