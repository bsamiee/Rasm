# Factorization Route as One Closed Family

[SHARED_ARROW_DISJOINT_EVIDENCE]:
- The five solver handles implement one arrow and five disjoint evidence vocabularies. `ISolver<T>` declares the entire shared contract — four `Solve` overloads spanning matrix/vector right-hand sides and allocating/in-place forms — and nothing else. Every handle carries `Determinant`, but `IsFullRank` lives only on the QR and eigen handles, `Rank`/`ConditionNumber`/`L2Norm`/`DeterminantLn` each on a single handle, `Inverse()` only on the LU handle, in-place re-factorization only on the definite handle. There is no `IFactorizationEvidence` interface to query; the evidence surface is structurally non-uniform and cannot be reached by a shared base reference.
- The orthonormal handle is a subtype of the triangular-rectangular handle, not a sibling. The modified-Gram-Schmidt path derives from the same abstract base as Householder QR and inherits its `Q`/`R`/`Determinant`/`IsFullRank`/`Solve` surface verbatim, overriding only construction. The "six factories" are therefore five solver families over one shared arrow, and any closed family that enumerates them as peer cases double-counts a subtype as a sibling — the orthonormal case is a discriminant of the QR case, selected by orthogonalization policy, not a parallel arm.
- The element carrier is closed-set runtime dispatch, never open generic math. The matrix type is constrained to `struct, IEquatable, IFormattable` — a constraint that admits exactly four runtime element types and excludes the IEEE-754 and number interfaces, so the family cannot be made generic over a numeric typeclass. A factorization route is monomorphic in its real-double specialization at the point of use; the carrier-polymorphism the constraint hints at is dispatched inside the provider by runtime type test, and lifting it to a typeclass arrow at the call site is impossible against this surface — the route's type parameter is decorative, and a generic route signature buys nothing the concrete one lacks.
[ONE_REQUEST_RECOVERABLE_DISCRIMINANT]:
- One request family replaces six factory call sites, and the discriminant must be recoverable from the request value. Collapse the six entrypoints into a closed union of route cases — definite-SPD, square-pivoting, rectangular-orthonormal, full-spectral, and rank-revealing — each case carrying its own admission policy as data: the orthonormal case carries thin-vs-full mode and orthogonalization method as enum fields, the spectral case carries the symmetricity discriminant, the rank-revealing case carries its rank-tolerance scaling. A `bool computeVectors` or `QRMethod mode` riding beside the matrix as a parallel parameter is arity smuggled back into a knob; the mode belongs inside the case as a row, so the route is recoverable by matching the case, never by reading a flag tail.
- The route case is a policy value whose constructor seats the threshold convention. Each case binds its conditioning gate at construction rather than re-deriving it at solve time: the definite case binds the symmetrize-then-Cholesky probe, the rectangular case binds the relative pivot criterion `|R[i,i]| / |R[0,0]| < sqrt(eps)`, the spectral case binds the scale-relative eigenvalue floor. The case row owns the threshold; the dispatcher reads it; no solve site re-states a magnitude. This converts five inconsistent built-in thresholds — absolute, magnitude-squared, and scale-relative, one per handle — into one declared convention per route, selected by the case.

```csharp
[Union]
public abstract partial record FactorRoute
{
    public sealed partial record DefinitePsd(Matrix<double> A) : FactorRoute;                               // symmetrize → Cholesky probe
    public sealed partial record SquarePivoting(Matrix<double> A) : FactorRoute;                            // LU, signed determinant
    public sealed partial record Orthonormal(Matrix<double> A, QRMethod Mode, bool Modified) : FactorRoute; // QR subsumes Gram-Schmidt
    public sealed partial record Spectral(Matrix<double> A, Symmetricity Sym) : FactorRoute;                // EVD; symmetricity is data
    public sealed partial record RankRevealing(Matrix<double> A, bool Vectors) : FactorRoute;               // SVD; Vectors gates evidence-only vs solvable
}
```

[EVIDENCE_AS_DERIVED_PROJECTION]:
- The disjoint evidence surfaces fold into one declared receipt, never an enumerated arm per handle. Because no shared evidence interface exists, the route's `Switch` over its cases is the single place that knows which handle exposes which probe: the rank-revealing case reads `Rank`/`ConditionNumber`/`L2Norm` from its handle, the definite case reads `DeterminantLn` (the underflow-stable log-determinant) from its handle, the orthonormal and spectral cases read `IsFullRank` from theirs, the pivoting case reads the signed `Determinant` from its handle. The receipt is the closed projection of these reads; a case that lacks a probe contributes absence for that field, and the projection is total because the route is closed. One fold over the request union derives the full evidence record from five non-uniform surfaces — the derived-logic law applied to a library that refused to derive it itself.
- The signed-determinant evidence is recoverable from exactly one case and must be routed there deliberately. The pivoting handle's determinant is the product of upper-triangular diagonal entries negated per recorded pivot inversion — the only signed value in the family. The spectral handle's determinant is a magnitude of a complex product and the rank-revealing handle's is a non-negative singular-value product; both discard sign. A route that needs the sign of the determinant — a definite-to-indefinite transition, a reflection detection in closed-form alignment — must carry the pivoting case for that field even when another case is the primary solver, so the receipt's signed slot is filled from the pivot evidence and its conditioning slot from the rank-revealing evidence; one route, two handles, one fold.
- The rank-revealing case is dual-modality and the solve modality overrides its own field. With vectors uncomputed the handle still allocates both full singular-vector buffers but skips computing them, yielding `Rank`/`ConditionNumber`/`L2Norm` evidence while any `Solve` throws on the absent vectors; with vectors computed the same handle solves. The case's vector field therefore selects evidence-only versus solvable, but the solve route ignores the field and forces vectors true, because a solve that consulted the field could fail at the arrow rather than at admission. The receipt route honors the field; the solve route discards it — one case, two routes, the modality recoverable from which route consumes the case, not from a second flag.
- Rank from the spectral case and rank from the rank-revealing case disagree by construction and must not share a receipt slot. The eigen rank test squares the eigenvalue magnitude against a fixed accuracy, classifying a small eigenvalue as zero far too eagerly, while the singular-value rank test scales tolerance to the largest singular value's own unit-in-last-place. Folding both into one `Rank` field erases a divergence that runs in opposite directions — the spectral test over-classifies small eigenvalues as zero, the singular test tolerates them; the receipt carries the rank-revealing rank as the conditioning rank and treats any spectral rank purely as definiteness signature, never as the same quantity.
[SOLVE_AS_RAIL_BIND_OVER_SHARED_ARROW]:
- The single arrow makes the solve a rail bind that is agnostic to which case factorized. Admission lifts the matrix to a finite-checked owner; the route fold lifts factorization to `Fin<ISolver<double>>` with the case's conditioning gate firing as the rejection; the solve is one `Bind` against `ISolver.Solve(Vector, Vector)` into a pre-allocated result; the residual cap is the next `Bind` computing `A·x − b` against the original matrix and demoting to a typed failure when the relative norm exceeds the cap. The solve composition is written once over the shared arrow and reused across every case, because `Solve` is the only member all five surfaces have in common — the collapse is load-bearing, not cosmetic: it is the one place a single `Bind` chain can serve all factorizations.
- The in-place overload is the rail's allocation contract and the allocating overload is its convenience leak. The shared arrow's `Solve(input, result)` writes destructively into a caller-owned result buffer; the `Solve(input)` overload allocates a fresh result and forwards. A hot route binds the in-place overload with result buffers reused across right-hand sides, so the rail's success value is a written span rather than a new vector; the allocating overload appears only at the route's outermost boundary where a value, not a buffer, leaves. Both overloads being abstract on every handle means this choice is uniform across the family and selected by the route, not the case.

```csharp
static Fin<Vector<double>> SolveRoute(FactorRoute route, Vector<double> b, double cap) =>
    route.Switch(                                                                           // gate fires inside Admit
        definitePsd:    c => Admit(c.A).Map(a => (A: a, S: (ISolver<double>)a.Cholesky())),
        squarePivoting: c => Admit(c.A).Map(a => (A: a, S: (ISolver<double>)a.LU())),
        orthonormal:    c => Admit(c.A).Map(a => (A: a, S: (ISolver<double>)a.QR(c.Mode))),
        spectral:       c => Admit(c.A).Map(a => (A: a, S: (ISolver<double>)a.Evd(c.Sym))),
        rankRevealing:  c => Admit(c.A).Map(a => (A: a, S: (ISolver<double>)a.Svd(true)))) // solve forces vectors
    .Bind(t => Capped(t.A, t.S.Solve(b), b, cap));                                          // one bind, every case
```

[REFACTOR_STREAM_IS_A_DEFINITENESS_PRIVILEGE]:
- Exactly one handle in the family can re-factor a successor operator into its existing factor buffer: the definite handle alone declares an in-place re-factorization method that block-copies a new operand over the held factor storage before re-invoking the provider. The pivoting handle, both orthogonal handles, the spectral handle, and the rank-revealing handle expose no such method — each demands a freshly constructed handle per operand. The amortization profile of a route is therefore a property of the case, not a caller knob: an iterative-registration or sequential-update route whose coefficient matrix mutates per step but stays definite must bind the definite case to capture the refactor privilege; binding any other case for the same loop pays a full handle allocation every iteration. Choosing the case is choosing the allocation budget.
- A stream of near-identical definite operators amortizes to one factor-buffer lifetime: construct the definite handle once, drive successive operators through the in-place re-factorization, and solve each against the refreshed buffer — the per-step cost is the provider's factorization work with zero managed allocation beyond the reused right-hand-side and result buffers. The privilege exists only while the operator stays definite: a step that loses definiteness throws at re-factorization, and that rejection is the regime boundary, not an error to retry. A streaming route that tracks a log-likelihood reads the log-determinant accessor per refreshed factor — one diagonal pass — because the determinant accessor flushes to zero under underflow with no indication and its logarithm is negative infinity.
- The damped-normal route is the augmented-stack case, not the Gram case, and the choice is a conditioning decision encoded in the route. Forming the normal system squares the condition number through the densifying transpose-multiply; stacking the design matrix over a scaled identity and applying the rectangular-orthonormal case to the augmented operator avoids the squaring at the cost of an `(m+n)×n` dense allocation. The two are the same mathematical solve with opposite conditioning, so they are two cases of one route discriminated by a conditioning-budget policy field — when the operator's condition exceeds the inverse cap the route must select the augmented-orthonormal case, and that selection is a derived consequence of the case's bound threshold, not a separate entrypoint the caller picks by name.
