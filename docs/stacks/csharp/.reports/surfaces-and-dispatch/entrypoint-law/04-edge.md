# Entrypoint Law — Edge Cases

[CANCELLATION_SCOPE_OPERATORS_ON_ENTRYPOINTS]:
- Three distinct scope operators govern cancellation at entrypoint boundaries and must not be conflated. `localIO<M,A>(K<M,A>)` creates a new `CancellationToken` derived from the parent context's token, so a regional `IO.cancel` inside the scope does not propagate to computations outside the scope — but the local cancel still raises an exception that propagates upward from the local scope, requiring a catch. `localCancel<A>(Eff<A>)` is the `Eff` counterpart: same semantics, different carrier type. `uninterruptible<M,A>(K<M,A>)` ignores all cancellation requests — including parent `CancellationToken` cancellation — by substituting `CancellationToken.None` for the duration; the computation runs to completion. The entrypoint design choice: `localIO` for fan-out dispatch arms that should not cancel siblings, `uninterruptible` for cleanup arms inside `bracketIO` whose interruption would leave resources open.
- `Prelude.cancel` is a static `IO<Unit>` field that raises cancellation into the current context. Placing it as a dispatch arm in a `Switch` branch aborts the entire computation at that case — it is not a no-op return, it is a structured abort that propagates as an exception through `|` catch chains. The resulting error code is the standard `TaskCanceledException`-mapped error. When a `retry` loop wraps a computation containing `cancel`, the cancellation error propagates through `retry` without triggering another attempt; `retry` does not swallow `TaskCanceledException`.
- `timeout(TimeSpan, K<M,A>)` is a Prelude function available for any `MonadIO` carrier. It internally forks the computation, awaits the fork, and cancels it if the duration expires — delivering `Error.New(TimeoutException)` on expiry. The carrier constraint is `MonadIO`, not `IO` specifically, so `timeout` composes with `Eff<RT,A>` and any user-defined transformer stack that implements `MonadIO`. For entrypoints that must bound execution time, `timeout` is the correct aspect form rather than a `CancellationTokenSource` knob on the entrypoint signature, which would be the input-redescribing anti-pattern.

[FORK_AND_PARALLEL_COMPOSITION_AT_ENTRYPOINTS]:
- `fork<M,A>(K<M,A>, Option<TimeSpan>)` (Prelude, carrier-polymorphic) queues the computation on the thread-pool and returns `IO<ForkIO<A>>`. The `ForkIO<A>` struct has two IO effects: `.Await` (blocks until completion and returns the result or failure) and `.Cancel` (sends a cancellation signal). The carrier constraint is `MonadIO`; the fork always executes on the thread-pool regardless of the outer monad's synchronous or asynchronous nature. When a timeout is supplied, the forked operation is automatically cancelled after that duration and `.Await` delivers `TimeoutException`-mapped `Error`. A forked computation that throws is not swallowed — `.Await` delivers the error through the normal `IO` error rail.
- `awaitAll<M,A>(K<M,A>[])` and `awaitAny<M,A>(K<M,A>[])` are Prelude functions over arrays of `K<M,A>` (not `ForkIO`). `awaitAll` runs all computations and returns `Seq<A>` when all succeed; if any fail, it collects all failures. `awaitAny` returns the first successful result and cancels all remaining computations; if no computation succeeds, it returns all errors. The semantic difference from `ForkIO` arrays: `awaitAll`/`awaitAny` handle the full fork lifecycle without requiring the caller to manage `ForkIO` handles. For a dispatch entrypoint that fans out to N independent suboperations, `awaitAll(arms)` is denser than N explicit `fork` calls.
- `Applicative.zip` over `K<F,A>` tuples (the `(K<F,A>, K<F,B>)` tuple syntax) composes two independent applicative effects. For IO applicative `F = IO`, the two effects run independently; the compiler lowers `(opA, opB).Apply(f)` via applicative lifting. The tuple overloads go to arity 5: `zip<F,A,B,C,D,E>((K<F,A>, K<F,B>, K<F,C>, K<F,D>, K<F,E>))`. This is the idiomatic form for parallel-independent entrypoint fan-out without introducing `fork` handles: `zip((readX, readY, readZ)).Map((x,y,z) => process(x,y,z))`. Unlike `>>` sequential bind, `zip` does not impose an ordering requirement on the IO effects — the runtime may or may not execute them concurrently; for guaranteed concurrency, explicit `fork` + `awaitAll` is required.

[KEYLESS_SMART_ENUM_AS_ENTRYPOINT_DISCRIMINANT]:
- `[SmartEnum]` (no type parameter) is the keyless variant of `[SmartEnum<TKey>]`. The generator emits `Switch`/`Map` dispatch methods and an `.Items` collection but no key property, no `Get(key)`, no `TryGet(key, out item)`, and no `IObjectFactory` interfaces. The `SwitchMethods` and `MapMethods` properties on `[SmartEnum]` accept the same `SwitchMapMethodsGeneration` enum, including `None` to suppress generation entirely. When a module uses a keyless smart enum as a case discriminant, the dispatch surface is structural: the caller holds an instance of the enum and calls `Switch(state, case1: ..., case2: ...)`. There is no admission path from an external key — the module owns the only construction paths via the static fields. This is the correct form for internally-discriminated behavior rows where the external caller receives an item by reference, not by string key.
- `SwitchMapMethodsGeneration.None` on `[Union]` or `[SmartEnum<TKey>]` suppresses `Switch` and `Map` generation entirely. When `None` is set, the only dispatch surface is the generated implicit/explicit conversion operators and the union's own partial class. Setting `None` is the correct form when the dispatch is entirely inside extension blocks that implement a custom dispatch algebra over the union's public properties — the generated `Switch` methods would create a secondary, less-restrictive dispatch surface that bypasses the extension block's policy enforcement. The `MapMethods` property has the same `SwitchMapMethodsGeneration` type and can be suppressed independently of `SwitchMethods`.

[BRACKETIO_CATCH_PARAM_PHASE_PRECISION]:
- The four-argument `bracketIO<M,A,B,C>(Acq, Use, Catch, Finally)` receives the `Catch: Func<Error, IO<C>>` argument for errors raised within the `Acq` computation, not the `Use` computation. The XML doc states: "Function called when an Error is raised within the Acq computation." An error in the `Use` arm propagates through the `Finally` arm directly; the `Catch` arm is not invoked for `Use` failures. This is architecturally significant: entrypoints that acquire a resource through a fallible acquisition step (e.g., opening a connection that may fail on TLS validation) use the `Catch` arm to reclassify or log the acquisition error while still guaranteeing `Finally` cleanup. The `Catch` arm must return `IO<C>` where `C` matches the use-phase result type — the catch cannot change the overall result type.
- `bracketIO(Acq, Use, Finally)` three-argument form: the `Finally` arm type parameter `B` is independent of both `Acq`'s `K<M,B>` result type and `Use`'s result type `C` — verified from the XML doc which shows `bracketIO<M,A,B,C>` where `A` is the bound result, `B` is the acquisition value, and `C` is the use result. The `Finally` returns `IO<A>` where `A` is the final result type, not `IO<Unit>`. This means the `Finally` arm can contribute a derived value rather than being forced to return `Unit` — a distinction that matters when cleanup has a meaningful output (e.g., confirming a commit was acknowledged before the connection closed).

[CONVERSION_OPERATOR_ASYMMETRY_AND_SWITCH_INVARIANCE]:
- `ConversionFromValue = ConversionOperatorsGeneration.None` removes the implicit conversion from member types to the union type. Pattern-matching (`req is CmdA a`) still works because pattern matching uses type testing, not conversion operators. Generated `Switch` still works because `Switch` dispatches through the generator's internal discriminator field, not through conversion operators. Setting `None` on `FromValue` prevents the accidental implicit widening `CmdA cmd = ...; Union req = cmd;` while preserving the full dispatch and pattern-match surface. The union is still constructable through explicit factory methods.
- `ConversionToValue = ConversionOperatorsGeneration.Explicit` (the default) generates explicit downcast operators from the union to each case type. Setting `ConversionToValue = None` removes them entirely. Removing `ToValue` operators eliminates the runtime `InvalidCastException` risk at call sites that attempt to downcast the union to a case before switching. With `ToValue = None`, the only ways to extract a case value are through the generated `Switch`/`Map` methods or through the generated `.IsX`/`.AsX` properties on the union. This is the correct form for a dispatch union whose callers should never hold a typed case reference — all access routes through the dispatch entrypoint.
- The combination `ConversionFromValue = None, ConstructorAccessModifier = Private, FactoryMethodGeneration = None` produces a union whose only construction surface is a hand-written static factory inside the partial class. No caller can implicitly widen a case, no caller can call a generated factory, and no caller can construct any case at all from outside the class. The `Switch` and `Map` methods remain fully operative. This is the maximum-sealed entrypoint shape: input enters only through the one public factory, output routes only through the dispatch method, and no type-system backdoor exists.

[PARAMS_COLLECTION_GENERIC_SPAN_HEAP_FALLBACK_DEPTH]:
- When a `params ReadOnlySpan<T>` parameter appears in a generic method and `T` is unconstrained, the compiler emits a heap-allocated `T[]` and passes it as a span when the call site provides a collection expression with a non-statically-known count. Adding the constraint `where T : allows ref struct` narrows the type to ref-struct-safe types only; for non-ref-struct domain types, the correct mitigation is to use a concrete typed `params ReadOnlySpan<KnownCmd>` rather than a generic parameter. The constraint `allows ref struct` is a .NET 10 feature (`AntiConstraint`); a parameter typed as `T where T : allows ref struct` can receive `ReadOnlySpan<char>` directly, but it cannot be stored on the heap — this is compatible with a dispatch entrypoint that consumes the span and returns a result without retaining the items.
- `[OverloadResolutionPriority(-1)]` applied to a `T[]`-accepting overload that exists for binary compatibility downgrades it relative to the span overload without deletion. The priority is evaluated before normal tiebreaking, so the span overload wins at all call sites that could call either, even when the array overload would otherwise be preferred by specificity. The negative priority works regardless of the overload being in the same type or a base type — it is a binding-time compiler hint, not a runtime decision.

[MODULE_ENTRYPOINT_INTERNALS_PRIVACY_FAILURE_MODES]:
- An extension block member declared as `private` inside a `static class` is accessible to any code inside the same static class, including other extension blocks declared in the same static class for different receiver types. This means a `private` dispatch helper defined in an extension block on `RequestUnion` is callable from another extension block on `Result` inside the same static class — an unexpected coupling surface. The mitigation is to declare helper methods as `file`-scoped (`file static`) at the top level or as `private static` methods on the containing static class that are explicitly not placed inside any extension block.
- `internal sealed partial class` on a generated `[Union]` type restricts the type to the assembly and prevents subclassing from outside. When `ConstructorAccessModifier = Internal`, the case constructors are accessible to the entire assembly. The assembly-level constraint is coarser than an enclosing type's visibility, so a `private` nested request union inside a `public` service class still has `internal` case constructors accessible to all assembly code — not just the enclosing service. When the desired scope is the enclosing type only, declare the union as a `private partial class` nested inside the service and use `ConstructorAccessModifier = Private` with explicit factory methods on the service.
- `[UnionSwitchMapOverload(StopAt = [typeof(AbstractCase)])]` attaches to the outer union, not to `AbstractCase`. When `AbstractCase` is itself a `[Union]`-derived abstract partial class, the stop-boundary is structural: the coarse overload receives `AbstractCase` instances that can be further dispatched via `AbstractCase`'s own generated `Switch`. The caller population that needs leaf-level dispatch uses the outer full-coverage overload; the population that needs aggregate handling uses the coarse overload. Both overloads are exhaustive at their respective levels. This two-tier dispatch surface is the correct design for a command hierarchy where some handlers care about the specific case and others treat the category as atomic.
