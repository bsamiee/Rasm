# Persistent Trie and AVL Node Mechanics

[CHAMP_DUAL_BITMAP_INVARIANT]:
- The keyed trie node carries two disjoint 32-bit bitmaps, `EntryMap` and `NodeMap`, over the same 32 logical slots of one trie level. A slot is either an inline key-value pair or a descent into a sub-node, never both, and the two bitmaps are mutually exclusive by construction. The compression follows from this split: only occupied slots consume an array cell, and inline entries are stored densely in one `(K,V)[]` while child nodes are stored densely in a parallel `Node[]`. There is no per-slot wasted reference and no sentinel-tagged union cell — the bitmaps replace the tag.
- Slot membership and array position are both derived from population count over the bitmap prefix. A hash fragment selects a single bit `mask = 1 << fragment`; presence is `(map & mask) == mask`; the dense array index is the count of set bits below that position, `popcount(map & (mask - 1))`. The popcount is a SWAR Hamming-weight: `bits - ((bits>>1)&0x55555555)`, then `(n&0x33333333)+((n>>2)&0x33333333)`, then `((n+(n>>4))&0x0F0F0F0F)*0x01010101 >> 24` — no table, no loop, one branchless reduction per level. Read, insert, and remove all pay exactly this per level of descent, so a lookup is O(level-count) popcounts plus one final key equality, not O(log n) comparisons.
- Iteration order is structural, not insertion or key order: a node yields its inline entries first, then recurses into each child node in bitmap order. Two maps with identical contents inserted in different orders enumerate identically because position is a pure function of hash fragment, but the order itself is hash-derived and carries no domain meaning — never treat keyed-trie enumeration as stable across key-type or hash changes.

[FIVE_BIT_SECTION_DESCENT]:
- A level consumes exactly five hash bits. The descent state is a single `int Offset` advanced by five per level; the fragment is `(hash & (31 << offset)) >> offset`. Five bits index 32 slots, matching the 32-bit bitmap width exactly — the arity of the trie and the width of the bitmap are the same constant, which is why one `uint` suffices per map. Maximum useful depth is six levels: the descent stops branching once `offset >= 25` because the seventh level would need bits beyond 30, leaving only the top two hash bits unused by branching.
- When two distinct keys collide on all six fragments — identical low 30 hash bits — the trie cannot branch further and falls to a flat collision node holding a `(K,V)[]` plus the shared hash. Collision nodes do linear key scan, so a hash function that clusters in its low 30 bits silently degrades that cluster from popcount-indexed descent to linear array scan. The cost model therefore hinges on low-bit dispersion, not on the full 32-bit hash range; a high-quality avalanche over the low 30 bits is what preserves the O(level) guarantee.
- Branch creation on a fragment clash is computed, not enumerated: when a new entry collides with an existing inline entry at a slot, a recursive `Merge` walks both keys' hashes downward one section at a time, building single-child wrapper nodes until the two fragments diverge, then places both as inline entries in one node with the correct relative order. The merge terminates at the collision-node floor when the keys agree through every section. Insertion that splits a slot is thus a tail of empty wrapper nodes whose depth equals the shared-fragment-prefix length — a pathological pair of near-colliding hashes builds a chain up to six nodes deep for two elements.

[TRANSIENT_BATCH_MUTATION]:
- Bulk construction routes through a private transient: an internal empty map flagged for mutation, against which each insert is applied with an in-place flag set. Under that flag the node-level array writer overwrites the existing backing array cell in place and returns the same array reference instead of cloning it; the pure path clones the array on every write. A batch of N inserts into a fresh transient therefore allocates backing arrays proportional to the final shape, not proportional to N intermediate shapes — the array-copy cost of repeated single inserts collapses to amortized in-place writes along each touched path.
- The transient does not eliminate per-step wrapper allocation. Even under the mutate flag, the top-level map handle is rebuilt as a fresh wrapper carrying the new root and updated count on every step that changes the structure, and a step that is a pure no-op returns the receiver unchanged. The amortization is on the node arrays, not the handle; reaching for the transient batch path wins on collections built element-by-element from a source, and wins nothing when the structure is already materialized.
- Range-add with a change log threads a second transient in parallel: the value map and a `K → Change<V>` map are both built under the mutate flag in one pass, and the change map records only entries whose insert reported an actual mutation, so an idempotent re-add contributes no change entry. The net-effect change map is therefore a byproduct of the same single traversal that builds the result, not a second diff pass — one walk yields both the new structure and the minimal delta against the prior structure.

[BULK_SET_ALGEBRA_IS_REPLAY_NOT_NODE_MERGE]:
- Union, intersection, difference, and append over the keyed trie are implemented as enumerate-one-side-and-replay-into-a-transient, never as structural node alignment. The full heterogeneous union iterates the right operand, looks up each key in the left, applies the matched or right-only resolver, and inserts into a fresh transient, then iterates the left operand and inserts every left-only key the transient does not already hold. Cost is O((|L|+|R|) · level-count) with no reuse of the operands' shared sub-trees — two structurally similar maps do not short-circuit identical sub-nodes the way a node-aligned CHAMP merge would.
- The consequence for collection choice: the keyed trie wins on point operations and on transient batch build, but a frequent whole-map merge of large maps pays full replay every time and shares nothing structural between input and output. A workload dominated by repeated set-algebra over large persistent maps is the regime where the popcount trie's per-element constant is multiplied by the entire union cardinality, and a domain that genuinely needs structural merge sharing must model the merge differently rather than assume the operator reuses sub-trees.
- Default `operator+` and unmerged union are left-biased because the replay inserts the right operand first under try-add semantics and then fills left-only keys, so a key present on both sides keeps the right operand's value only if the resolver chooses it; with no resolver the existing entry wins on collision. Right-bias is not a flag — it is the explicit matched-resolver `(_, _, r) => r`, and a call site that wants right-bias must pass that resolver or it silently keeps the wrong side.

[AVL_PATH_REBUILD_HAS_NO_TRANSIENT]:
- The ordered map is an AVL tree of immutable nodes, each carrying a one-byte cached `Height` and a subtree `Count`. Insert descends by key comparison, rebuilds every node on the path with a freshly allocated node, and rebalances via single or double rotation when the balance factor — right height minus left height — exceeds one. Every insert allocates O(log n) fresh nodes along the path plus up to two rotation nodes; there is no mutate-flag transient and no in-place array write, so bulk construction of an ordered map cannot amortize node allocation the way the keyed trie's transient does.
- The cached `Height` byte and `Count` per node are what make ordered range and positional operations sublinear: a range query positions at the lower bound in O(log n) and yields in-order on demand, and a positional fetch compares the target index against the left subtree's count to choose a descent direction — both ride the stored counts rather than walking. The keyed trie carries no per-node count and no ordering, so it offers neither operation; the count-and-height payload is precisely the cost the ordered tree pays at every node to buy ordered access.
- Rotation preserves the in-order key sequence while re-rooting a subtree, so the rebalanced tree is a different node graph with the same logical order; structural sharing survives only for the untouched sibling subtrees hanging off the rebuilt path. An ordered map built by N sequential inserts is therefore N path-rebuilds with rotations, and converting from an unordered keyed trie to an ordered map is a full re-insert, not a re-view — the two structures share no backing and the conversion is O(n log n), never a cast.

[EQUALITY_DISPATCH_AT_NODE_BOUNDARY]:
- The hash and equality used at every node boundary are resolved through the trie's type-parameter comparator, invoked as a static interface call with no instance and no virtual dispatch. The two-parameter map form binds a default comparator that resolves a trait equality instance once at static initialization via reflection and falls back to boxing the framework default when none is found; the three-parameter form binds an explicit comparator type whose static `Equals` and `GetHashCode` are called directly with no reflection and no boxing. Every level of every operation calls `GetHashCode` once and `Equals` on the final key match, so for a custom key type in a hot path the explicit-comparator form removes a per-call boxing the default form pays.
- Compound keys carry the same dispatch decision: a value-tuple key routes through the framework's structural tuple equality under the default comparator, which boxes each component, whereas an explicit comparator over the tuple shape calls the typed equality directly. A composite-key map under heavy churn is the canonical case where binding the explicit comparator type, rather than relying on tuple default equality, changes the allocation profile of the whole workload.

[CHANGE_TRACKED_NET_DELTA]:
- The change-tracking wrapper records one morphism per key representing the net effect since the last snapshot, collapsing any number of intermediate mutations on a key into a single tagged change — added, removed, value-mapped-to, value-mapped-from, or both-ends-typed. Reading the change ledger yields a keyed map of net deltas, and snapshotting constructs a fresh wrapper over the current value map with an empty ledger; it does not retain intermediate states. This is the correct primitive for delta-sync and incremental refresh where only the final state per key matters and the path taken to it is irrelevant.
- Net-delta tracking composes with the transient change-log build: each value mutation that the underlying insert reports as an actual change updates the per-key morphism, and an idempotent transition contributes none, so the ledger size is bounded by the count of keys actually touched, not by the count of operations applied. A burst of writes that cancels out — add then remove of the same key — can leave that key absent from the ledger entirely, which is the intended semantics for a change feed but a trap for any consumer that expects one ledger entry per operation.
