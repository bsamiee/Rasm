# Numeric Evidence Law: Receipts and Rails

[ALGORITHM_RECEIPTS]:
- Every decomposition in the admitted package hierarchy carries algorithm-specific typed evidence: `Cholesky<T>` exposes `Factor` (lower triangular `Matrix<T>`) plus computed `Determinant` and `DeterminantLn`; `LU<T>` exposes `Factors`, `Pivots` (`int[]`), and pivot-sign-corrected `Determinant`; `Svd<T>` exposes `S` (singular-value vector), `U`, `VT`, `Rank`, `L2Norm`, and `ConditionNumber`; `Evd<T>` exposes `EigenValues` (`Vector<Complex>`), `EigenVectors`, `BlockDiagonal`, `Rank`, `Determinant`, and the `isSymmetric` honesty flag. A generic receipt over all factorization families must extend `ISolver<T>` and carry the algorithm-specific properties as a concrete sub-type — no abstraction over them is possible without losing evidence.
- `Symmetricity` is a four-case enum (`Unknown`, `Symmetric`, `Hermitian`, `Asymmetric`) passed to `Evd.Create` as a policy value that controls both the algorithm branch chosen at construction and the structural meaning of eigenvectors. Passing it as a stored field on a receipt turns an implicit algorithm choice into an auditable fact. `Unknown` triggers a symmetry check; passing `Symmetric` or `Hermitian` bypasses it and contracts that the caller has already verified the precondition — so the receipt must record which case was supplied, not just what was inferred.
- `QRMethod` (`Full` vs `Thin`) is similarly a behavior-carrying enumeration: `Full` produces a square `Q` (m×m) and full `R`; `Thin` produces economy-sized `Q` (m×n) and upper-triangular `R` (n×n). A solve over a thin QR cannot reconstruct the full column space. A receipt that records `QRMethod` is the only way a downstream consumer can know whether the orthogonal complement was discarded.
- The `Svd<T>.Rank` computation uses `S.Maximum().EpsilonOf() * max(m, n)` as the rank threshold — a machine-epsilon-relative tolerance tied to the largest singular value. Embedding this threshold in a custom receipt by capturing both `S[0]` (largest) and the computed rank integer makes the threshold auditable; a receipt that stores only `Rank` as an `int` hides the tolerance policy used.
- `Evd<T>.Rank` uses `AlmostEqual(eigenvalue, Complex.Zero)` with the library's `DefaultDoubleAccuracy = DoublePrecision * 10` — a private constant equal to `2^{-53} * 10`. This tolerance differs from the SVD threshold. A receipt distinguishing these two rank computations as algorithm-qualified facts prevents downstream code from treating them as interchangeable.

[TOLERANCE_TAXONOMY]:
- `Precision` exposes a named ledger: `DoublePrecision = 2^{-53}` (Demmel/LAPACK convention), `PositiveDoublePrecision = 2 * DoublePrecision` (Higham/ISO C convention), `MachineEpsilon` (measured at runtime), `PositiveMachineEpsilon`, and `DefaultDoubleAccuracy = DoublePrecision * 10` (private, used implicitly by parameterless `AlmostEqual`). Inline literals such as `1e-10`, `1e-14`, `1e-15`, or `1e-8` that duplicate one of these anchors are the anti-pattern; the anti-pattern is provable when the literal equals a multiple of `DoublePrecision` for a concrete platform.
- `EpsilonOf(this double value)` returns `Math.Abs(value) * DoublePrecision` — the unit-in-the-last-place (ULP) relative epsilon for that specific value, not the global machine epsilon. `PositiveEpsilonOf = 2 * EpsilonOf`. Using a global threshold where `EpsilonOf` is semantically correct causes false acceptances near zero and false rejections for large magnitudes. A receipt's validity predicate for "result within one ULP of input" must use `EpsilonOf` on the actual residual, not on a policy constant.
- `AlmostEqualNorm(a, b, diff, maximumAbsoluteError)` is the lowest-level comparison: it compares `diff` (pre-computed `a - b`) against an absolute threshold. All higher-level overloads delegate to this. A receipt validity predicate that pre-computes the residual norm and stores it alongside the threshold makes re-admission possible without recomputing `a - b`.
- `AlmostEqualNormRelative` divides the norm of the difference by the norm of `a` before comparing against `maximumError`. The distinction between absolute and relative error modes is not preserved by any existing result type — it must be recorded explicitly as an enum or boolean honesty marker on any receipt that performs a comparison.
- Three tolerance forms exist in the iterative solver surface: `ResidualStopCriterion<T>` owns a `_maximum` absolute residual threshold and a `_minimumIterationsBelowMaximum` stability count; `DivergenceStopCriterion<T>` owns `_maximumRelativeIncrease` and `_minimumNumberOfIterations`; `IterationCountStopCriterion<T>` has `DefaultMaximumNumberOfIterations = 1000`. These three policy objects are the named-constant ledger for iterative convergence; duplicating their defaults as bare literals in a caller-side receipt is the tolerance anti-pattern.

[ITERATIVE_SOLVER_EVIDENCE]:
- `IterationStatus` is a six-case enum: `Continue`, `Converged`, `Diverged`, `StoppedWithoutConvergence`, `Cancelled`, `Failure`. `StoppedWithoutConvergence` is the diagnostic-but-unusable case: the solver ran to max iterations without convergence and without divergence or failure. A receipt that collapses this into `Failure` destroys the distinction between "exceeded budget" and "algorithm broke down" (`NumericalBreakdownException` / `Failure`). The three-way distinction `Converged | StoppedWithoutConvergence | (Diverged | Failure | Cancelled)` is the minimum evidence partition; any coarser receipt risks hiding pathological conditioning behind an innocuous timeout.
- `TrySolveIterative` returns `IterationStatus` directly on `Matrix<T>` — the status is the receipt of the iterative solve's outcome. A wrapping domain receipt must capture this status plus the final residual vector (accessible post-call via the mutated `result` parameter), the iteration count drawn from `IterationCountStopCriterion.MaximumNumberOfIterations`, and whether a non-unit preconditioner was active — the preconditioner type changes what "residual" means.
- `DelegateStopCriterion<T>` accepts `Func<int, Vector<T>, Vector<T>, Vector<T>, IterationStatus>` — the delegate receives `(iterationNumber, solutionVector, sourceVector, residualVector)`. A domain receipt can embed a per-step snapshot policy here: the delegate can record every k-th residual norm into a preallocated `double[]` trail without heap pressure, and the trail becomes part of the receipt's step-count-vs-trail-length conservation invariant.

[OPTIMIZATION_EVIDENCE]:
- `MinimizationResult` (unconstrained) carries `MinimizingPoint` (via `FunctionInfoAtMinimum.Point`), `Iterations`, and `ReasonForExit` (twelve-case `ExitCondition`). `MinimizationWithLineSearchResult` extends this with `TotalLineSearchIterations` and `IterationsWithNonTrivialLineSearch` — the line-search sub-iteration count is algorithm-specific evidence owned by BFGS that has no analog on trust-region results and must not be collapsed into a generic `int TotalIterations`.
- `NonlinearMinimizationResult` carries `Covariance`, `Correlation`, `StandardErrors`, and `MinimizedValues` alongside `Iterations` and `ReasonForExit`. When `DegreeOfFreedom < 1` (underdetermined) all three matrix/vector properties are `null`. A receipt validity predicate must check `DegreeOfFreedom >= 1` — exposed via `IObjectiveModelEvaluation.DegreeOfFreedom` — before treating the covariance as a meaningful fact. A receipt that materializes covariance without this gate is not tamper-evident.
- `ExitCondition` carries twelve cases. The split between success-adjacent (`WeakWolfeCriteria`, `StrongWolfeCriteria`, `RelativeGradient`, `AbsoluteGradient`) and genuine non-convergence (`ExceedIterations`, `LackOfProgress`) is load-bearing for a domain receipt: meeting the Wolfe criteria with very few iterations is a qualitatively different outcome from hitting `LackOfProgress`, which indicates numerical stagnation. A receipt honesty marker that records which sub-family of `ExitCondition` obtained is more discriminating than the raw enum value.
- `LevenbergMarquardtMinimizer` constructor defaults: `initialMu = 0.001`, `gradientTolerance = stepTolerance = functionTolerance = 1e-15`. These three tolerances serve distinct purposes: gradient (first-order stationarity), step (parameter change), function (residual norm). A receipt that records only one scalar "tolerance" conflates three different convergence criteria. The five-parameter constructor signature is the minimal policy surface; a domain receipt should store all five as named fields with their algorithmic roles.

[CONSERVATION_PREDICATES]:
- Rank-nullity: for `n × n` matrix, `rank + nullity == n`. `Evd<T>.Rank` counts non-zero eigenvalues under `DefaultDoubleAccuracy`; `Svd<T>.Rank` uses the ULP-relative threshold. Neither class exposes `Nullity`; a receipt constructed from either must derive `Nullity = n - Rank` and record both, so that a re-admission check can verify the conservation identity without rerunning the decomposition.
- SVD singular-value ordering: `DenseSvd` stores singular values in descending order (the library contract). A receipt should assert `S[0] >= S[1] >= ... >= S[k-1] >= 0` at construction; violation signals corruption or a hand-rolled modification of the `S` vector between construction and use.
- Cholesky positive-definiteness: `DenseCholesky.Create` throws `ArgumentException` if the matrix is not SPD. There is no `bool IsPositiveDefinite` property. A receipt wrapping a Cholesky factorization therefore must record the factorization success as a structural fact (existence of the `Factor` field), not as a boolean flag computed after the fact — the factory-result-as-admission-evidence pattern: existence == success, no property needed.
- LU pivot parity: `LU<double>.Determinant` computes the sign from `Pivots[i] == i` (even permutation) versus `Pivots[i] != i` (odd, negates). A receipt that stores `Determinant` must also store the pivot array length equals the matrix rank at construction time, since the pivot count is tied to the input dimension as a conservation check: `Pivots.Length == RowCount`.

[DIAGNOSTIC_STATUS_DISTINCTION]:
- `IterationStatus.StoppedWithoutConvergence` must be kept distinct from both `Converged` and `Failure` in every receipt surface and every error union that wraps iterative solver outcomes. The temptation to map it to `Fin.Fail` is the exact anti-pattern: a rail that treats a converge-budget-exceeded solve as a hard failure prevents the caller from deciding whether to retry with a relaxed criterion or a different preconditioner.
- `ExitCondition.None` and `ExitCondition.ManuallyStopped` are the two non-algorithmic exit cases. A receipt that can be constructed with `ExitCondition.None` is structurally incomplete; the receipt factory must reject `None` and emit a typed error, using `Fin.Fail(Error.New(code, "optimizer did not start"))`.
- `DssStatus.MKL_DSS_ZERO_PIVOT` (`-1`) is a definiteness failure, not a general solver failure — it indicates the matrix is singular or indefinite given the `DssMatrixType` policy. A receipt wrapping the sparse direct solver must map this to a domain-level `SingularMatrixError` distinct from `DssStatus.MKL_DSS_FAILURE` (`-3`), which is an internal MKL breakdown. Collapsing both to a single `SolverFailure` discards the definiteness signal.

[EXCEPTION_BOUNDARY]:
- The admitted numeric library emits three exception families: `NonConvergenceException` / `NumericalBreakdownException` (algorithm convergence), `OptimizationException` / `MaximumIterationsException` / `EvaluationException` (optimization), and `ArgumentException` / `SingularUMatrixException` (structural precondition violations). None of these is `Error`-typed. The single boundary conversion point is `Error.New(Exception? thisException)`, which wraps any exception into an `ErrorException` record. This call belongs in exactly one `catch` per algorithm invocation, immediately converting to `Fin.Fail(Error.New(ex))` — never propagated as raw exceptions deeper into domain code.
- `NumericalBreakdownException` is a sub-type of `NonConvergenceException`. A catch ladder that catches the parent first silently swallows the more specific breakdown signal. The correct pattern is to catch `NumericalBreakdownException` first, map it to a domain `NumericalBreakdown` error case, then catch the parent for the general non-convergence case — two `catch` arms, each producing a distinct `Fin.Fail` case, at the boundary and nowhere else.
- `SingularUMatrixException : NativeInterfaceException` is thrown by the native BLAS/LAPACK provider when U is singular. It carries no additional fields beyond the base exception message. At the boundary, this maps to a typed domain error that preserves the "singular U" semantics — not a generic "solver failed" case.

[HONESTY_MARKERS]:
- `Symmetricity` as a stored receipt field is an honesty marker: `Symmetric` means the caller verified or asserted real symmetry; `Hermitian` means conjugate symmetry; `Asymmetric` is explicit; `Unknown` means the library performed its own check. A receipt that omits this field makes the algorithm path ambiguous when the same receipt shape serves both symmetric and asymmetric cases.
- `QRMethod` stored on a QR receipt is a honesty marker: `Full` means the complete column space is preserved in `Q`; `Thin` means it was discarded. Re-admission of a thin-QR receipt for a full-rank reconstruction is structurally invalid; the receipt factory for a downstream step that requires the full orthogonal complement must reject `Thin` receipts.
- `IObjectiveFunctionEvaluation.IsGradientSupported` and `IsHessianSupported` are honesty markers surfaced on every optimization evidence snapshot. A receipt that captures the current `IObjectiveFunctionEvaluation` must also capture these two flags; downstream Newton-based second-order methods that inspect `Hessian` without checking `IsHessianSupported` will silently receive `null`.
- `vectorsComputed` on `Svd<T>` construction controls whether `U` and `VT` are populated. When `false`, `U` and `VT` are identity-shaped stubs. A receipt that embeds an `Svd<T>` computed with `computeVectors: false` must record this as a `bool VectorsComputed` flag; otherwise a consumer that reads `EigenVectors` or `VT` receives silent stubs.

[MEMOIZATION_LAW]:
- `AtomHashMap<K, V>.FindOrAdd(K key, Func<V> None)` performs a CAS-loop that atomically inserts only when absent and returns the existing or freshly-inserted value. The invariant is that `None` may be invoked more than once under contention, so `None` must be a pure construction function with no observable side effects. A domain memoization surface for numeric receipts — e.g., caching eigendecompositions keyed on a matrix identity token — must supply a pure factory to `FindOrAdd`; any factory that acquires a lock, writes a file, or mutates state is structurally incorrect here.
- `Atom<A>.Swap(Func<A, A> f)` CAS-loops until the swap succeeds; `f` may be called multiple times. A receipt-accumulating fold stored in an `Atom<HashMap<K,V>>` must supply an `f` that merges the new receipt into the existing map purely (immutable `HashMap<K,V>.Add`) — not a mutation that acts on an external `Dictionary<K,V>`.
- `AtomHashMap<K, V>.Change` is an event raised after each successful mutation. For numeric receipt caches, subscribing to `Change` for staleness invalidation is preferable to polling the map count; the event carries `(Key, OldValue, NewValue)` semantics via `AtomHashMapChangeEvent<K, V>`.

[GENERIC_MATH_SURFACE]:
- `TensorPrimitives` on .NET 10 exposes generic operations constrained on `IFloatingPointIeee754<T>`, `INumber<T>`, `IRootFunctions<T>`, and `IBinaryInteger<T>` depending on the operation. `Dot<T>`, `Norm<T>`, `CosineSimilarity<T>`, `FusedMultiplyAdd<T>`, `Sigmoid`, and `SoftMax` cover the hot-loop surface. For `double` and `float`, these operations use hardware SIMD dispatch internally; they are the BCL floor for any inner product, norm, or elementwise transform before reaching for MathNet's `Matrix<T>` operations.
- `TensorPrimitives.Dot<T>` has constraint `IAdditionOperators<T,T,T>, IAdditiveIdentity<T,T>, IMultiplyOperators<T,T,T>, IMultiplicativeIdentity<T,T>` — this is strictly more permissive than the MathNet provider surface. A numeric receipt that stores a `double[]` residual vector and needs a norm for its validity predicate can call `TensorPrimitives.Dot(residual, residual)` then `Math.Sqrt` rather than constructing a full `DenseVector` and calling `.L2Norm()`.
- The `ISparseSolverProvider<T>` interface takes raw CSR arrays (`rowPointers`, `columnIndices`, `values`) plus `DssMatrixStructure` and `DssMatrixType` as behavior-carrying enumerations. A receipt wrapping a sparse direct solve must record both enumerations as honesty markers — `DssMatrixStructure.Symmetric` versus `Nonsymmetric` determines which fill-reducing ordering was used, and `DssMatrixType.PositiveDefinite` versus `Indefinite` determines whether Cholesky or LDLT was applied internally.

[RECEIPT_WIDTH_AND_FACTORY_DUALITY]:
- A receipt carrying more than eight distinct fields — e.g., a nonlinear minimization receipt with `MinimizingPoint`, `Iterations`, `ExitCondition`, `Covariance`, `Correlation`, `StandardErrors`, `DegreeOfFreedom`, `GradientTolerance`, `StepTolerance`, `FunctionTolerance`, `InitialMu` — crosses the constructor-width pressure point. The shape law resolves this through a required-evidence record (`Point`, `Iterations`, `Exit`, `DegreeOfFreedom`) plus an optional-evidence `Option<StatisticalEvidence>` sub-receipt (`Covariance`, `Correlation`, `StandardErrors`) that is only materialized when `DegreeOfFreedom >= 1`. This is a hierarchical receipt: the outer receipt nests the statistical sub-receipt, which is constructed independently and embeds its own conservation check (`StandardErrors.Count == Point.Count`).
- Factory/admission duality: a receipt constructor is private; a public static `Create` factory validates preconditions — `Rank > 0`, `Pivots.Length == n`, `S` non-empty and monotone-descending — and returns `Fin<TReceipt>`. Re-admission takes a serialized or cached receipt and replays the preconditions against the current payload to verify it has not been tampered with, returning `Fin.Fail` on violation. The re-admission path never re-runs the decomposition; it re-runs only the structural invariant checks. This separation means re-admission is O(n) in the receipt field count, not O(n³) in the matrix dimension.
- `IPrecisionSupport<T>` with `Norm()` and `NormOfDifference(T)` is the interface that makes `AlmostEqualNorm<T>` work on custom types. A domain value type that participates in numeric tolerance comparisons implements this interface to unlock the full `Precision` extension method surface. A receipt validity predicate for a custom type that hand-rolls `Math.Abs(a - b) < tol` instead of using `AlmostEqualNorm<T>` is bypassing the named-tolerance ledger.
