# Events and Threads — Deepening

[ENVO_SCOPE_VARIANTS_AND_SUBSCRIPTION_ISOLATION]:
- `EnvIO` exposes four distinct local-scope constructors with different ownership profiles. `Local` allocates new `Resources`, new `CancellationTokenSource`, and reads `SynchronizationContext.Current` at access time — full isolation. `LocalResources` allocates new `Resources` only; it inherits the parent `Source` and `SyncContext`. `LocalCancel` allocates a new `CancellationTokenSource` only; it inherits the parent `Resources` and `SyncContext`. `LocalSyncContext` reads `SynchronizationContext.Current` at access time and allocates a new `CancellationTokenSource`, but inherits the parent `Resources`. `LocalSyncContext` and `Local` are property getters, not construction-time captures — calling them on a background thread captures the wrong context. Evaluate them on the host thread and pass the resulting env down.
- `WithEnv(env => env.LocalResources)` is the scoping form for subscription lifetime within an `IO` chain: resources acquired inside the mapped computation are released in a child scope on completion without touching the parent scope's resources or cancellation token. `Bracket()` wraps the whole computation in a child resource scope and releases all tracked resources on both success and failure. `BracketFail()` creates a child resource scope and releases it only on failure — correct for subscriptions where success deliberately transfers resource ownership out of the bracket.
- `Resources.DisposeU` iterates the `AtomHashMap` in hash-traversal order, not insertion or reverse order. Disposal sequence is undefined. When disposal of one resource depends on another still being live — for example, a subscriber must be detached before closing the connection it reads — encode the order explicitly inside the computation rather than relying on scope disposal order.

[EXECUTION_CONTEXT_FLOW_DISCIPLINE]:
- `CancellationToken.Register` captures the ambient `ExecutionContext` at registration time and restores it when invoking the callback, unless `useSynchronizationContext: false` is also specified. `CancellationToken.UnsafeRegister` skips context capture entirely — the callback runs with whatever `ExecutionContext` the cancelling thread carries. `UnsafeRegister` is the correct form for callbacks that are context-free (writing to a channel writer, swapping an `Atom`) and must not pay the context-capture allocation or risk inherited context leaking into the callback.
- `ExecutionContext.Run` is single-use per captured instance: passing the same captured context to `Run` a second time throws `InvalidOperationException`. Event subscription tokens that store a captured `ExecutionContext` for replay must re-capture after each use.
- `ExecutionContext.SuppressFlow` suppresses `ExecutionContext` propagation to work queued from the current thread until the returned `AsyncFlowControl` is disposed. The intended use is inside queue-submission loops where ambient context must not bleed into workers. The suppression is thread-local; disposing it restores flow. Nesting suppression calls throws.
- `ThreadPool.UnsafeQueueUserWorkItem<TState>(Action<TState>, TState, preferLocal: bool)` avoids both `ExecutionContext` capture and delegate allocation when combined with a struct `TState`. The `preferLocal: true` flag biases toward the current thread's local work-stealing queue, reducing cross-core contention for tight producer-consumer pairs on the same logical thread.

[LOCK_SCOPE_REF_STRUCT_HAZARDS]:
- `Lock.EnterScope()` returns `Lock.Scope`, a `ref struct`. It cannot be stored in a field, passed to a local function, used inside an `async` method, or captured in a lambda. A gate that must span an `await` point requires `SemaphoreSlim(1, 1)` with `WaitAsync`.
- `Lock.Scope.Dispose` throws `SynchronizationLockException` if the current thread does not hold the lock at disposal time. The `ref struct` constraint prevents the most common misuses at compile time but does not cover IL-level or reflection paths.
- `Lock.IsHeldByCurrentThread` allows reentrancy detection; the lock supports recursive entry up to an implementation-defined limit. The intended pattern for seams that may be called from both lock-owning and non-owning code: `if (!_gate.IsHeldByCurrentThread) using (_gate.EnterScope()) { ... }`.

[CHANNEL_SYNCHRONOUS_CONTINUATION_HAZARD]:
- `ChannelOptions.AllowSynchronousContinuations = true` allows the thread calling `WriteAsync` or `TryWrite` to invoke the continuation of an awaiting `ReadAsync` inline, before the write returns. This eliminates a thread switch for tightly coupled pairs but introduces two hazards: unbounded stack growth when the continuation itself writes to the same or a downstream channel, and reentrancy when the writer holds a lock or is inside a callback the reader's continuation also needs. The default `false` posts continuations asynchronously, breaking the call chain.
- `ChannelWriter<T>.Complete(Exception? error)` both closes the channel and, when `error` is non-null, propagates it to `ChannelReader<T>.Completion` and to `WaitToReadAsync` as an exception on the next call after the buffer is drained. The error propagates as the original exception type — not wrapped in `ChannelClosedException`. `ChannelClosedException` is thrown only on a normal `Complete()` when `ReadAsync` is called after the channel is empty and closed. Consumers must catch the writer's original exception type when the writer signals structured failures via `Complete(error)`.
- `ChannelReader<T>.CanCount` and `CanPeek` are capability flags; `Count` throws `NotSupportedException` and `TryPeek` returns `false` if the respective capability is absent. Capability guards before using these must be explicit rather than try/catch, because the exception path obscures logic.

[PERIODIC_TIMER_AND_TIME_PROVIDER_SEAM]:
- `PeriodicTimer(TimeSpan, TimeProvider)` accepts a `TimeProvider` injection point for deterministic testing. `WaitForNextTickAsync` cancellation via its `CancellationToken` affects only the single outstanding wait; the underlying timer continues firing. Stopping the timer requires calling `Dispose()`, which completes the current `WaitForNextTickAsync` with `false`. The read loop pattern `while (await timer.WaitForNextTickAsync(ct)) { ... }` exits on either cancellation (throws `OperationCanceledException`) or disposal (returns `false`). Missing the `false` return causes the loop to exit without the worker having been explicitly stopped.
- `PeriodicTimer.Period` is a mutable property. Setting it after construction changes the period for all future ticks but does not affect the currently pending `WaitForNextTickAsync`. Rate changes at a seam boundary update `Period` on the next cycle rather than requiring a new timer instance.
- A `PeriodicTimer` used as a fold worker driver must be owned and disposed by the worker itself, not by a parent scope that may outlive the worker's subscription set. If the parent disposes the timer while the worker is between ticks, `WaitForNextTickAsync` returns `false` cleanly. If the parent disposes it while the worker is inside the loop body, the current iteration completes and `false` surfaces on the next wait. No in-flight work is interrupted.

[SAFEHANDLE_CALLBACK_WINDOW_PROTOCOL]:
- `SafeHandle.DangerousAddRef(ref bool success)` increments the internal reference counter, preventing `ReleaseHandle` from being called while the counter is elevated. The `success` parameter must be checked before calling `DangerousGetHandle`; if `success` is `false`, the handle was already closed. The symmetric form is always `try { handle.DangerousAddRef(ref success); if (success) { /* use */ } } finally { if (success) handle.DangerousRelease(); }`. Calling `DangerousRelease` without a matching `DangerousAddRef` corrupts internal state.
- In an event callback window, the subscribe–use–unsubscribe triplet for a `SafeHandle`-wrapped resource is: (1) `DangerousAddRef` before registering the handler, (2) the handler body calls `DangerousGetHandle`, (3) `DangerousRelease` after detach completes. Detach synchronously before calling `DangerousRelease`; guard the handler body with an `IsInvalid` or liveness check on the acquired pointer.

[ATOM_SWAP_CHANGE_EVENT_HAZARDS]:
- `AtomHashMap<K, V>.Swap(TrackingHashMap → TrackingHashMap)` emits change tracking only for transforms applied to the `TrackingHashMap` argument provided by the swap function. If the swap function ignores the argument and builds a fresh `TrackingHashMap`, the change event carries a diff showing only the final state with no tracked intermediate changes. The correct pattern always threads the provided map through transforms (`map.SetItem`, `map.Remove`) rather than constructing a new collection.
- `Atom<A>.Change` fires synchronously on the swapping thread inside the CAS success path, before `Swap` returns. Subscribers that block inside the change handler deadlock because the swap does not return until the change event returns. Change handlers must be non-blocking; cross-thread work must be submitted via channel write, not executed inline.
- The two-argument `Atom<M, A>` form's change event carries only the new value `A`. A subscriber that needs the prior value must maintain a shadow read of `atom.Value` before the swap. Reading `atom.Value` after the event fires may already reflect a further-advanced value if another swap occurred during the handler.

[IO_REPEAT_AND_SUBSCRIPTION_LIFETIME_CONSTRAINT]:
- `IO<A>.Repeat(Schedule)`, `RepeatWhile`, and `RepeatUntil` release all resources acquired within each iteration after that iteration completes. A subscription opened inside `Repeat` is detached and disposed after each cycle. The correct pattern for a persistent subscription with a repeating read loop: open the subscription outside the `Repeat` via `use` at the outer `IO` level, then bind the repeating reader inside. The outer `use` persists until the parent scope is released.
- `IO<A>.Retry(Schedule)` releases resources on each failed attempt but retains them on success. A retry loop that opens a subscription on each attempt and fails repeatedly does not accumulate subscriptions; each failed attempt's resources are cleaned up before the next retry begins. The first successful attempt's resource escapes the `BracketFail` scope into the parent.
- `IO<A>.Uninterruptible` wraps the computation in a local environment that ignores cancellation-token signals. This is the correct form for finalization paths that must complete regardless of cancellation — detaching subscribers, draining a queue, flushing state. `Uninterruptible` composed with `Bracket`'s finalizer lambda is the canonical resource-safe teardown form.

[CONFIGUREAWAIT_OPTIONS_AT_SEAM]:
- `ConfigureAwaitOptions.ForceYielding` forces an already-completed `Task` or `ValueTask` to post its continuation asynchronously rather than completing inline. Without `ForceYielding`, awaiting a pre-completed task runs the handler synchronously on the event thread, causing reentrancy if the handler schedules more events or acquires shared state.
- `ConfigureAwaitOptions.SuppressThrowing` is valid only on the non-generic `Task`, not `Task<T>`. Applying it to `Task<T>.ConfigureAwait(options)` throws `ArgumentOutOfRangeException` at runtime; CA2261 is the diagnostic. The correct pattern: cast `Task<T>` to `Task`, await with `SuppressThrowing`, then inspect the original `Task<T>` handle. Using `SuppressThrowing` without inspecting the outcome silently discards errors; this is only correct when the caller has a separate fault channel and the task fault is already emitted there.
- `ConfigureAwaitOptions` is a `[Flags]` enum. The combination `ForceYielding | SuppressThrowing` (value 6) yields before running and does not propagate faults through the await site; the non-generic `Task` cast is still required for the `SuppressThrowing` component.

[TASKSCHEDULER_SYNCHRONIZATIONCONTEXT_CAPTURE]:
- `TaskScheduler.FromCurrentSynchronizationContext()` throws `InvalidOperationException` when `SynchronizationContext.Current` is `null`. This happens on thread pool threads, `LongRunning` task threads, and non-UI background threads. The capture must happen on the host thread while that thread's context is installed; store it in the environment and inject it into effects rather than re-reading at call sites.
- `SynchronizationContext.Send` is synchronous and blocks the calling thread until the callback returns. Calling `Send` from a thread that participates in the target context's message loop deadlocks. `Post` with a completion signal (channel write or `TaskCompletionSource.SetResult`) is the correct bidirectional form.

[SUBSCRIPTION_RESOURCE_SCOPE_ISOLATION_WITH_IO_WITHENV]:
- `IO<A>.WithEnv(env => env.LocalResources)` creates a child resource scope for the duration of the mapped computation. Any `use` calls inside that computation register in the child scope. On completion, the child scope's `DisposeU` runs. Combined with `Retry`, each failed activation attempt cleans up its partial subscriptions before the next attempt.
- `WithEnvFail` releases only on failure. When the activation succeeds and the caller wants to retain subscriptions until an explicit deactivation, `WithEnvFail` paired with an outer `Atom<Option<Resources>>` ownership token allows manual release: swap `None` in and call `DisposeU` explicitly on the evicted `Resources` value.
- `EnvIO.LocalCancel` forks a new cancellation token sourced from the parent while inheriting the parent's resources and sync context. This is the correct form for a guarded region that must be cancellable independently of the parent — for example, a subscription whose read loop can be stopped without tearing down the parent environment's resources.

[UNSAFE_REGISTER_AND_CONTEXT_FREE_CALLBACKS]:
- `CancellationToken.UnsafeRegister(Action<object?, CancellationToken>, object?)` is the preferred form for callbacks that (a) do not use `AsyncLocal<T>` values from the registration context and (b) perform only context-free work such as `ChannelWriter.TryWrite`, `Atom.Swap`, or setting a `ManualResetEventSlim`. It avoids `ExecutionContext.Capture` allocation and the restore overhead during cancellation. The `Action<object?, CancellationToken>` overload receives the cancellation token itself, allowing the callback to inspect `IsCancellationRequested` without a closure.
- `CancellationTokenRegistration` is both `IDisposable` and `IAsyncDisposable`. Synchronous `Dispose` unregisters and, if the callback is currently executing on another thread, blocks until it finishes. `DisposeAsync` returns immediately after unregistering without waiting for in-flight callbacks. Using `await using` on a `CancellationTokenRegistration` when an in-flight callback writes to a resource that the caller then disposes creates a window where the resource is accessed after disposal. Synchronous `Dispose` is safer when the next step after unregistration disposes resources the callback touches.
