# Carrier-Polymorphic Resource Lifecycle

[UNLIFT_PRIMITIVE_COLLAPSE]:
- The entire resource-lifecycle algebra — bracket, local-scope, repeat, retry, timeout, fork, uninterruptible, post, fold — is not enumerated per carrier. It collapses onto exactly one abstract primitive on the unlift trait: a single `ToIO` arrow that lowers an arbitrary effect `K<M, A>` into `K<M, IO<A>>` — the inner `IO` action exposed as a value inside the same carrier. Every other lifecycle operator is a `static virtual` default expressed through one derived combinator, `MapIO(ma, io => f(io))`, which lowers via `ToIO`, transforms the surfaced `IO` with the concrete `IO`-level operator, then re-lifts through the carrier's own `LiftIO`. A carrier that supplies `ToIO` inherits the complete bracket-and-scope surface for free; it overrides an individual operator only to change that one operator's semantics, never to re-implement the family.
- `MapIO` is the load-bearing reduction: `ToIO(ma).Bind(io => M.LiftIO(f(io)))`. Read literally, it round-trips the effect through the innermost `IO` layer — surface the `IO`, rewrite it with the pure `IO` operator, sink it back. This is why the carrier-polymorphic operators carry the exact resource-release guarantees of their concrete `IO` counterparts: the release semantics live entirely in the surfaced `IO` action, and the carrier wrapper is transparent to them. A bracket lowered through `MapIO` releases its registry on the same success/failure/cancellation seams as a bare-`IO` bracket because it *is* that bracket, momentarily wearing the carrier's clothes.
- The reduction means the resource registry is never carrier-aware. Registration, parent-chaining, and bulk disposal happen at the `IO`/environment layer that `ToIO` surfaces; the transformer stack above it cannot see or perturb the registry. A capsule whose lifecycle operator is invoked polymorphically over `M` acquires identical tracking to one invoked over concrete `IO`, because both resolve to the same surfaced `IO` action under one environment.

```csharp
// Every lifecycle operator is one MapIO over the single ToIO primitive.
// Carrier-polymorphic bracket — works for any M that provides ToIO.
static K<M, C> Bracketed<M, A, B, C>(
    K<M, A> acquire, Func<A, IO<C>> use, Func<A, IO<B>> finalise)
    where M : MonadUnliftIO<M> =>
    M.MapIO(acquire, io => io.Bracket(use, finalise));   // lower → rewrite IO → re-lift

// Carrier-polymorphic isolated scope, retry, timeout — same lowering, different IO op.
static K<M, A> Isolated<M, A>(K<M, A> ma) where M : MonadUnliftIO<M> =>
    M.MapIO(ma, io => io.Local());                       // fresh registry + token at IO layer
```

[PARTIAL_TRAIT_FALLBACK_AND_INNERMOST_IO_LAW]:
- The lift and unlift surfaces split into a fully-derived public form and a partial-implementation form the carrier author overrides. The public `liftIO` and the lifecycle operators delegate to a partial trait whose members carry a distinct suffix — a `LiftIOMaybe` arrow and a parallel `*Maybe` member per lifecycle operator. The "maybe" here is structural, not optional-valued: it encodes that the operation succeeds only if the stack admits an `IO` at its base. A transformer stack whose innermost monad is not `IO` returns a failed lift at runtime, not a compile error — the partial trait is the seam where "this stack can reach `IO`" is asserted dynamically.
- This is the hard constraint on carrier-polymorphic capsules: resource lifecycle is available over `M` only when `M`'s transformer stack bottoms out at `IO`. A reader-over-state-over-`IO` stack supports the full bracket surface; a reader-over-state with no `IO` base silently cannot, and the failure surfaces as a lift fault deep inside the first lifecycle operator rather than at the call that built the stack. Capsule code that is generic over `M : MonadUnliftIO<M>` must treat "innermost is `IO`" as a precondition the type system does not enforce.
- The partial form is also the override point for carriers that must intercept lifecycle. A carrier overrides `BracketIOMaybe` or `LocalIOMaybe` when the carrier's own layer carries lifetime-relevant state that the naive `MapIO` round-trip would drop — the override threads that layer's state through the surfaced `IO` explicitly instead of letting `ToIO` flatten it.

[STATE_THREADING_HAZARD_UNDER_LOWERING]:
- The `MapIO` round-trip is transparent for reader-shaped carriers because a reader contributes no outbound state: lowering a `ReaderT<Env, IO, A>` surfaces an `IO<A>` that closes over the environment, and re-lifting restores the reader with the environment intact. It is *not* transparent for state-shaped carriers. A `StateT<S, IO, A>` lowered to `IO` must surface an `IO` that also carries the post-computation state `S`; a lifecycle operator that rewrites only the `A`-typed surface and discards the threaded `S` silently loses state mutations performed inside the bracketed body.
- The consequence for resource capsules built over a state-threaded carrier: a bracket whose use-body mutates carrier state — incrementing a lease counter, recording an acquisition ledger in the threaded `S` — must use the carrier's overridden lifecycle member, not the default `MapIO` lowering, or the post-bracket state reverts to the pre-bracket value while the resource itself was correctly acquired and released. The resource lifecycle is correct; the carrier-state ledger of it is lost. This is the precise failure mode of treating a state-threaded effect as if it were a bare `IO` for bracketing.
- The safe shape over a state-threaded carrier is to keep the resource-lifetime ledger in the resource environment (a reader-asked runtime, below) rather than in the threaded `S`, so the lowering surfaces an `IO` whose closure already holds the ledger and no outbound state needs threading. Lease accounting belongs in a reader-provisioned cell, not in `StateT`'s carried value, precisely so the unlift round-trip stays transparent.

[TRAIT_PROVISIONED_RESOURCE_RUNTIME]:
- A resource runtime — the bundle of pools, native contexts, clocks, and capsule registries an operation may acquire from — is provisioned as a reader environment carried by the carrier itself, asked through a typed accessor, and scoped through a typed `With`. The accessor trait exposes a single cached `ask` arrow yielding the carried value inside the carrier; the scoping trait extends it with `With(Func<InnerEnv, InnerEnv>, K<M, A>)`, which runs a subcomputation under a transformed environment and restores the outer environment on exit. A capsule asks for its sub-runtime by accessor rather than receiving it as a constructor parameter, and a caller swaps the runtime for a bounded region by `With` rather than re-wiring construction.
- The runtime-carrying effect is structurally a reader over `IO`: the runtime-provisioned effect type is exactly a `ReaderT<RT, IO, A>` under a thin record. This identity is what makes resource provisioning and resource lifecycle compose without friction — the reader layer carries the runtime, the `IO` base carries the registry, and `ToIO` surfaces the `IO` with the runtime already closed into it. `runtime<RT>()` asks the current runtime; `local`-scoped runtime swap (`Func<RT, RT'>` over a subcomputation) runs a region against a derived runtime — a child pool, a quarantined connection lane, a clock advanced by an injected provider — and the outer runtime is restored when the region exits, identically to how a reader restores its environment.
- Because the runtime is reader-carried, a scoped runtime swap does not touch the resource registry: swapping the runtime changes which pools and contexts the region acquires *from*, while the registry that tracks what the region acquired remains the environment's registry. Isolating the acquisition source and isolating the disposal scope are two orthogonal axes — the first is a reader `With`, the second is the lifecycle operator's `Local`/bracket — and conflating them by swapping the runtime when an isolated disposal scope was meant leaves acquisitions tracked in the outer registry.

```csharp
// Capsule asks its sub-runtime by accessor; a region swaps the runtime by With.
// The runtime-carrying effect is ReaderT<RT, IO, A> under a thin record.
static K<M, Conn> Acquire<M, RT>()
    where RT : Has<M, Pool>                     // runtime carries a connection pool
    where M  : Monad<M> =>
    from pool in Has<M, RT, Pool>.ask           // ask the sub-runtime, not a ctor param
    from conn in pool.Open()
    select conn;

// Bounded region against a derived runtime; outer runtime restored on exit.
static K<M, A> InQuarantineLane<M, RT, A>(Func<RT, RT> toLane, K<M, A> region)
    where RT : Local<M, RT> =>
    RT.With(toLane, region);                     // swaps acquisition source, not the registry
```

[ORTHOGONAL_PROVISION_VS_LIFETIME_AXES]:
- Three independent axes govern a carrier-polymorphic capsule, and each has its own operator: *what it acquires from* is the reader-carried runtime, set by `With`; *how long the acquisition lives* is the registry scope, set by the lifecycle operator's bracket/`Local`; *whether the body can be interrupted* is the cancellation mask, set by `Uninterruptible`. All three are expressible polymorphically over `M` through the single `MapIO`-lowering for the lifetime and mask axes and through the reader trait for the provision axis. A capsule that needs a child pool, a private disposal scope, and an uninterruptible release composes the three operators in sequence; none subsumes another, and each lowers independently.
- The correct stacking for an acquire-once-release-uninterruptibly capsule against a scoped pool: swap to the scoped pool (`With`), open the private registry (`Local`-shaped bracket), and mask only the release step (`Uninterruptible` on the finaliser, not the body). Masking the whole region defeats the timeout-and-drain window; swapping the pool but not opening a private registry leaks acquisitions to the outer scope; opening a private registry but not swapping the pool acquires from the wrong source. The three-operator stack is the minimal correct form, and each operator is one `MapIO` or one reader `With`.

[FORK_OVER_ARBITRARY_CARRIER]:
- Fork is also a `MapIO` default: `ForkIO(ma, timeout)` lowers `ma` to `IO`, forks the surfaced `IO`, and re-lifts a `K<M, ForkIO<A>>`. The forked child receives a fresh isolated registry parented to the surfaced environment's registry — but the carrier layers above the `IO` base are *captured by closure into the surfaced `IO`, not re-threaded*. A reader-carried runtime is visible inside the fork because it was closed into the lowered `IO`; a state-threaded ledger is captured at its pre-fork value and any mutation the fork performs is invisible to the parent because the fork's `IO` carries its own state thread that never rejoins the parent's. The fork-isolation hazard from the registry layer is thus doubled for state-threaded carriers: not only is the fork's registry isolated, the fork's carrier-state is isolated too, and only reader-carried context survives the fork boundary as a shared value.
- The competitive-fork combinators (await-any, await-all) over an arbitrary carrier return projected carrier values, never live handles, for the same reason at the carrier layer that the bare-`IO` race returns copies at the registry layer: the winning fork's registry disposes at the fork's exit and the winning fork's carrier-state thread terminates at the fork's exit. A session opened inside a racing fork over `M` is gone when the race returns; the winning identity must be a reconstructable reader-runtime key, and the live resource re-acquired in the outer carrier scope after the race settles.

[CAPSULE_AS_DEFINITION_TIME_TRAIT_INSTANCE]:
- The lifecycle surface a capsule's carrier exposes is fixed at definition time by which traits the carrier implements, not at call time by which methods exist. A carrier declaring the unlift trait gains the full bracket-and-scope family as inherited statics; a carrier declaring only the lift trait gains `liftIO` but no bracketing; a carrier declaring the reader-accessor trait gains runtime provisioning. The capsule author chooses the carrier's resource capabilities by trait membership, and the available operators are derived from that membership rather than written. Adding bracket capability to a carrier is implementing one `ToIO` member, after which the entire family materialises — the canonical case of a cross-cutting capability attaching through a single definition-time hook and the rest deriving.
- This makes the resource-lifecycle surface uniform across every effect in the system that shares the carrier: a domain effect, a wire-boundary effect, and a native-capsule effect built over the same `M` all bracket, retry, and scope identically, because all three inherit the same derived operators from the same trait instance. There is one bracket spelling, one timeout spelling, one isolation spelling across the whole effect surface — the carrier owns them once, and no per-effect or per-capsule lifecycle re-statement exists. A capsule that re-implements bracketing locally instead of inheriting the carrier's trait-derived operator is the rejected form: it forks the lifecycle algebra the carrier already owns.
