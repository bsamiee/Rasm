# Aspect Architecture — Deepening

[UNION_MEMORY_AND_ACCESS_ASPECTS]:
- `UseSingleBackingField = true` on `[Union]` collapses all per-case backing fields into one `object` field. Field-per-case stores value types in their native representation (no boxing); a single backing field always boxes structs. For unions whose cases are all reference types this is an unambiguous memory-layout improvement; for unions with struct cases it introduces per-access boxing. The decision is driven by case type topology at definition time.
- `ConstructorAccessModifier = UnionConstructorAccessModifier.Private` on `[Union]` makes both the generated constructors and the generated implicit conversion operators private simultaneously — the coupling is load-bearing. The only externally visible factory surface is then hand-written static factory methods on the union type; no caller can construct a case by direct implicit assignment. This is the definition-time enforcement mechanism for admission-controlled unions where construction must pass validation or enrichment before the case is materialized.
- `T1IsStateless = true` on `[Union<T1,T2>]` instructs the generator to store only the discriminator index for that arm — no field is allocated, and accessors like `AsT1` return `default(T1)`. This eliminates the case field entirely for marker or signal cases. The generator recommends struct types for stateless members to eliminate null-handling at the accessor.
- `ConversionFromValue` and `ConversionToValue` on `[Union]` are independent axes. A union can admit values implicitly (implicit `from T` operator) while requiring an explicit cast to extract them, or vice versa. Decoupling these two directions is the definition-time aspect that enforces directional admission policy — inbound permissiveness does not imply outbound permissiveness.

[SMART_ENUM_KEY_SURFACE_ASPECTS]:
- `KeyMemberKind = MemberKind.Field` changes the generated key member from a public property to a field. Combined with `KeyMemberAccessModifier = AccessModifier.Private`, this generates a private backing field named `_key` that carries the discriminant without exposing it as public surface. The enum's behavioral identity is then entirely through `Switch`/`Map` and delegate-backed partial methods, not through key inspection.
- `DisableSpanBasedJsonConversion = true` on `[SmartEnum<string>]` falls back from the zero-allocation `ReadOnlySpan<char>`-based JSON reader to a string-based path. The span path uses the generated `IObjectFactory<Owner, ReadOnlySpan<char>, TError>` interface, avoiding heap allocation for the transient key during deserialization on .NET 9+. Enabling this flag is a signal that the serialization context is not driving the span reader, and the zero-allocation admission guarantee is silently lost.
- `SkipIParsable = true` forces `SkipISpanParsable` to `true` simultaneously because `ISpanParsable<T>` inherits from `IParsable<T>`. Selectively suppressing `IParsable` while retaining `ISpanParsable` is impossible — the BCL inheritance chain makes the skip cascade. Any grammar or text-protocol aspect relying on `ISpanParsable` is disabled by the same flag that disables string-based parsing.

[CONSTRUCTION_SEAM_DEEPENING]:
- `[ObjectFactory<T>]` with `UseForSerialization` steers the generated serialization converters to route through the custom `IObjectFactory<Owner, T, TError>` implementation's `Validate(TValue value, IFormatProvider provider, out T result)` method. The `out` parameter and `IFormatProvider` argument are load-bearing for the STJ span-reader path. A type implementing only `IObjectFactory<T>` (the one-type-parameter marker interface) satisfies the runtime but is not wired by the generator for serialization.
- `HasCorrespondingConstructor = true` on `[ObjectFactory(typeof(T))]` informs the generator that a constructor taking a single `T` parameter exists, enabling the generator to emit a conversion expression that routes through the constructor rather than only through the static factory. This is the construction-advice aspect for EF Core value converters: EF materializes instances via constructor expression trees, and the generator wires the constructor path only when this flag is explicitly set.
- `DefaultStringComparison` on `[ComplexValueObject]` sets the comparison mode for all `string`-typed members that do not carry their own `[KeyMemberEqualityComparerAttribute<...>]`. This is a type-level default-comparison aspect that prevents per-member comparer annotation proliferation when all string members share one comparison policy.

[COMPOSITION_TIME_POLICY_DEEPENING]:
- `repeat` and `retry` are semantically antipodal: `retry` fires on failure (re-invokes when the computation produces an error), `repeat` fires on success (re-invokes when it produces a value). `retryWhile`/`retryUntil` predicate on `Error`; `repeatWhile`/`repeatUntil` predicate on the bound value `A`. A schedule-driven heartbeat loop is `repeat(schedule, computation)`; a resilience loop is `retry(schedule, computation)`. Both accept `K<F, A>`, so mixing them is not a type error — the distinction is purely semantic.
- `IO<A>.BracketFail()` calls `WithEnvFail(env => env.LocalResources)` — it creates a new `Resources` child scope but disposes resources only on failure, allowing a successfully acquired resource to escape as the computation's result. `IO<A>.Bracket()` calls `WithEnv(env => env.LocalResources)` and disposes on both success and failure. Using `Bracket` when the acquired resource escapes as the result would dispose it before the caller can use it; `BracketFail` is the correct form when the resource is the result.
- `Eff<A>.MapIO(Func<IO<A>, IO<B>>)` is the static extension method on `K<M, A>` where `M : MonadUnliftIO<M>`. It calls `M.MapIOMaybe(ma, f)`, extracting the `IO<A>` from the transformer stack, applying `f`, and re-lifting into `M`. This is the correct mechanism for applying IO-layer aspects (retry, bracket, catch) from within an `Eff` pipeline when the generic `retry(schedule, eff)` Prelude form is insufficient — for example, when composing `io.Timeout(ts).Retry(schedule)` as a chained IO expression before re-lifting.
- The generic `retry<M, A>(Schedule, K<M, A>)` overloads operate on any monad `M` that satisfies `MonadUnliftIO<M>`. Both `Eff<A>` and `Eff<RT, A>` satisfy this constraint, so retry/repeat aspects compose uniformly across bare `IO<A>`, `Eff<A>`, and `Eff<RT, A>` without explicit lowering to `IO` first.

[SCHEDULE_ALGEBRA_DEEPENING]:
- `Schedule.Interleave(b)` alternates steps from two schedules — a, b, a, b — for the length of the longer. This is distinct from `Combine` (sequential append: all of `a`, then all of `b`) and from Intersect/Union (parallel max/min per step). A burst-then-paced pattern is `Schedule.recurs(3).Interleave(Schedule.spaced(1*second))`: attempt, 1 s, attempt, 1 s, attempt, 1 s — gaps are interleaved between retries, not appended after them.
- `Schedule.decorrelate(factor, seed)` introduces both upward and downward jitter. For a linear schedule `(100, 200, 300, ...)`, decorrelation produces roughly `(103.2, 96.8, 202.3, 197.7, ...)`. The effective run length approximately doubles because each step expands into two. It is not a delay-expansion combinator — it is a statistical de-correlation aspect whose purpose is reducing thundering-herd re-collision among parallel instances of the same computation.
- `Schedule.resetAfter(max)` resets the schedule once the cumulative delay reaches `max`, restarting the delay sequence from zero and continuing indefinitely. `Schedule.maxCumulativeDelay(max)` terminates the schedule when the cumulative total exceeds `max`. Both transform the same base schedule differently: `resetAfter` is a periodic restart; `maxCumulativeDelay` is a terminal cap.
- `Schedule.fixedInterval(d)` guarantees a gap of `d` between action completions — if the action takes longer than `d`, the next gap is zero rather than accumulating a deficit. `Schedule.windowed(d)` aligns to clock-epoch multiples of `d` — the next step waits until the nearest multiple of `d` regardless of when the action completed. Both accept an optional `Func<DateTime>` clock argument for deterministic testing.
- `Schedule.maxDelay(max)` clamps each individual step duration to `max`; `Schedule.maxCumulativeDelay(max)` terminates when the sum of all emitted durations exceeds `max`. These are orthogonal caps: `maxDelay` bounds each sleep, `maxCumulativeDelay` bounds total wait time. A production retry policy uses both: `Schedule.exponential(50*ms) & Schedule.maxDelay(30*seconds) & Schedule.maxCumulativeDelay(5*minutes)`.

[INTERCEPTOR_WEAVE_SURFACE]:
- `GetInterceptableLocation` on `SemanticModel` returns an `InterceptableLocation` (version integer + opaque data string) only for invocation expressions — constructor calls, property accessors, and delegate invocations are not interceptable. A mismatch between the version used during generation and the version expected by the compiler at build time produces a compile error, preventing silent data corruption from format-version drift. Any source generator producing interceptors must regenerate location data on every compilation, not cache it across builds.
- `ERR_InterceptorSignatureMismatch` fires when an interceptor has a parameter or return type incompatible with the intercepted method's resolved signature; a warning fires for compatible-but-not-identical cases (for example, differing parameter names or `params` vs. explicit array). Generators must track both: parameter-name divergence is tolerable, type divergence is a build break.
- An interceptor produces a strict call-site replacement — the original method is never called unless the interceptor explicitly calls it. A generated interceptor replacing a factory call can inject logging, validation, or caching purely at the call site without touching the factory implementation. Only one interceptor per location is valid; a second interceptor for the same location is a compile error.

[MODULE_INITIALIZER_ORDERING_CONSTRAINT]:
- Within a single assembly, `[ModuleInitializer]` methods run in source-file order and then in declaration order within each file. This ordering is a consequence of IL emission order, not a language specification guarantee — it may change across compiler versions. Aspect registrations that depend on ordering must be placed in a single method or must use explicit dependency calls inside that method rather than relying on cross-method ordering.
- `[ModuleInitializer]` cannot appear on a generic method, a method in a generic type, or a method in a `ref struct`. These constraints narrow its use: it cannot be a generic factory installer, cannot be placed on generated partial constructors or partial methods, and cannot be embedded inside generic-type aspects. All registration logic it carries must be fully specialized at the type level.
