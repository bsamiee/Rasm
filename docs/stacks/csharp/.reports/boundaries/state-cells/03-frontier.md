# State-Cell Boundary Law — Frontier

[ATOM_VOLATILE_BOX_MEMORY_MODEL]:
- `Atom<A>.value` is declared `private volatile object`. All reads use `volatile` semantics (acquire fence); all CAS operations via `Interlocked.CompareExchange` carry full memory barriers. The `volatile` annotation on a reference field means any thread reading `value` observes the most recently written reference — no stale reference to a prior boxed value is possible. For value types, `Box<A>` wraps the struct; `Box<A>.GetValue` is a static IL-emitted delegate (`DynamicMethod`) that field-loads directly from the box object. When `ILCapability.Available` is false (AOT/NativeAOT), the fallback is `(object x) => ((Box<A>)x).Value` — correct but adds a cast; no correctness hazard, only a minor allocation shape difference on NativeAOT.
- `Memo<A>.state` is `internal int` with no `volatile` annotation. The `Interlocked.CompareExchange(ref state, 1, 0)` calls in `GetValue` and `Reset` carry implicit full memory barriers, so state machine transitions are coherent. However, the hot-path `if (state != 2)` guard in the `Value` property getter is a plain `int` read — no memory barrier. On architectures with weak memory ordering (ARM64), a thread could observe `state == 2` before observing the write to `value` that occurred on the winning CAS thread, reading a stale `default(A)`. This window is a real data race on weakly ordered hardware; the safe form is `Volatile.Read(ref state) != 2` in the getter, or a `Memo<A>` replaced by an `Atom<Option<A>>` which carries the full volatile guarantee on both state and payload simultaneously.

[ENVIO_DISPOSAL_LAYERING_GAPS]:
- `EnvIO.DisposeResources()` checks `(Own & 2) == 2` before touching resources — an `EnvIO` that does not own its resource scope (passed a non-null `resources` argument to `New`) will SILENTLY skip resource release even when `Dispose()` is called. This is correct for scopes that borrow a parent's resources, but any design where a caller passes `null` to `New` to get a new scope AND does not want to own it will be surprised: `Own bit 2` is always set when resources are freshly allocated. The invariant is: `Own bit 2` tracks allocation, not logical ownership; it cannot be selectively suppressed after `New` returns.
- `EnvIO.DisposeResources()` calls `EnvIO.New()` — a fully owned fresh `EnvIO` with `Own=3`, a new `CancellationTokenSource`, and a new `Resources` — then passes that to `Resources.DisposeU(envIO)`. `DisposeU` constructs a SECOND inline `EnvIO` (`new EnvIO(envIO.Resources, CancellationToken.None, source, ...)` with `Own=0`) to actually run releases. The fresh `EnvIO` from `New()` is handed to `DisposeU` but `Dispose()` is never called on it within `DisposeResources()`; its own `CancellationTokenSource` and `Resources` scope are abandoned. On `GC`, the `CancellationTokenSource` will be finalized. This is a minor resource leak per disposal — not a correctness hazard but a pressure point for high-frequency disposal paths.
- `Resources.DisposeU(EnvIO envIO)` passes `envIO.Resources` (the outer caller's resources) into the release env. Custom `IO<Unit>` release effects that call `use(...)` or `Acquire(...)` inside them will register new resources into the CALLER's resource scope, not a local one. A teardown effect that itself acquires a temporary resource (e.g., a connection for a protocol goodbye) must use a manually scoped inner `IO.Bracket` to isolate that acquisition from the caller's scope, or the temporary resource outlives the teardown call.

[IOUNINTERRUPTIBLE_ASYNC_LEAK]:
- `IOUninterruptibleAsync<A,B>.Invoke` awaits `Fa`, then calls `LocalEnv.Dispose()` after the await, then returns. There is no `try/finally` wrapping the `await Fa` expression; if `Fa` faults, the awaiter propagates the exception before `LocalEnv.Dispose()` is called. The `LocalEnv` holds an isolated `EnvIO` constructed with `CancellationToken.None` and a fresh `CancellationTokenSource` (via `EnvIO.New(envIO.Resources, CancellationToken.None, null, envIO.SyncContext)`). On exception, the `CancellationTokenSource` inside `LocalEnv` is abandoned until GC finalizes it. For high-frequency fault paths (network faults, timeout races inside uninterruptible regions), this accumulates live `CancellationTokenSource` objects. The correct form wraps `await Fa` in `try { A arg = await Fa; LocalEnv.Dispose(); return +Next(arg); } catch { LocalEnv.Dispose(); throw; }`.

[IOLOCAL_EXCEPTION_PROPAGATION_CONTRACT]:
- `IOLocal<X,A>.Invoke` uses `using EnvIO local = MapEnvIO(envIO)` with no explicit `try/catch`. When `Operation.RunAsync(local)` throws, the `using` block's compiler-generated `finally` calls `local.Dispose()` — both resources and non-resources are released — before the exception propagates. The returned `IO<B>` is never constructed; the exception surfaces to the caller's `catch` layer. This is the NORMAL path: `Bracket()` therefore guarantees cleanup on any failure path.
- `IOLocalOnFailOnly<X,A>.Invoke` manually separates the cleanup: `catch { local.DisposeResources(); throw; }` followed by `finally { local.DisposeNonResources(); }`. On success, neither `DisposeResources` nor `DisposeNonResources` run in the local scope's guarded body — the `await Operation.RunAsync(local)` returns normally and `Next(result)` is returned without any disposal. On success the local `CancellationTokenSource` (created if `MapEnvIO` calls `LocalResources`, which passes the parent source, so `Own bit 1 = 0`) is NOT disposed. This is intentional — `BracketFail` keeps resources alive on success for caller use. The caller takes ownership of `local`'s resources by the resources being in the parent scope (`LocalResources` passes `Source`, not a new one; the new scope holds resources only via `Own bit 2`).

[ATOMHASHMAP_FACTORY_MULTI_INVOKE_SURFACE]:
- `AtomHashMap<K,V>.FindOrAdd(K, Func<V> None)` calls `Items.FindOrAddWithLog(key, None)` inside a CAS spin loop. If two threads both observe `key` absent, both call `None()` inside the loop body, only one wins the CAS, the loser retries and now finds the key present, returns the stored value. The `None` factory therefore executes at least once per concurrent first-insert per key, potentially more under high contention. Any `None` factory that acquires resources (file handles, connections, native memory) will leak the handle produced by the losing thread. The safe surface is `SwapKey(key, curr => curr.IsNone ? Some(factory()) : curr)` — but this carries the same factory-multi-invoke problem because the swap function also runs inside a CAS loop. True once-only insertion requires an outer gate: check first with `Find(key)`, then gate on a `Lazy<V>` inserted via `TryAdd(key, new Lazy<V>(factory))`, then read `.Value` from the stored lazy — `Lazy<T>` with `LazyThreadSafetyMode.ExecutionAndPublication` ensures the factory runs exactly once even if multiple threads observe the `Lazy` before it completes.
- `AtomHashMap<K,V>.SwapKey(K, Func<V,V>)` returns `unit` immediately when the key is absent — the swap function is never called and the map is unchanged. This is the correct sentinel-free behavior: absent key + value-transform swap = no-op. For conditional insertion semantics, use `SwapKey(key, Func<Option<V>, Option<V>>)` which receives `None` for absent keys, allowing the swap function to return `Some(value)` to insert.
- `AtomHashMap.Swap(Func<TrackingHashMap<K,V>, TrackingHashMap<K,V>>)` checks `(object)items == trackingHashMap.Value` before attempting the CAS — if the swap function returns the SAME underlying trie reference (no structural change), the CAS is skipped entirely and no `Change` event fires. A swap function that conditionally modifies should return the original `TrackingHashMap` when no change is desired, not a structurally equivalent new one — equality is by reference identity of the underlying `TrieMap`, not value equality.

[FORK_CLEANUP_AND_RESOURCE_CHAIN_HAZARDS]:
- `IO<A>.Fork` creates a `CleanUp` record holding `(CancellationTokenSource tsrc, CancellationTokenRegistration reg)`. `CleanUp.Dispose()` uses `Interlocked.Exchange(ref disposed, 1) == 0` as an idempotency guard, then disposes `tsrc` and `reg` with independent `try/catch` swallowing all exceptions. The fork task's `finally` block calls `resources.Dispose()` (the child resource scope) and `cleanup.Dispose()`. If `cleanup.Dispose()` is called both from the task's `finally` AND from the `Cancel` IO (which calls `tsrc.Cancel()`), the idempotency guard prevents double-dispose. The `Cancel` IO only cancels the source; it does not call `cleanup.Dispose()` — `Dispose` is called exclusively from the task's `finally`. This means the `CancellationTokenRegistration` (the parent→child link) is disposed in the task's `finally` regardless of whether cancellation was requested externally or via timeout.
- The fork passes `EnvIO.New(resources, token, tsrc, env.SyncContext)` to `RunAsync` — not `env.LocalCancel` or `env.LocalResources`. This creates a new `EnvIO` with the child resource scope and the fork's own `tsrc`, but the `Own` flag is `0` because all four parameters are non-null (no auto-allocation). Neither the `CancellationTokenSource` (`tsrc`) nor the `Resources` are owned by this env. `Dispose()` on this env is a no-op for both fields. The actual cleanup responsibility falls entirely on the `finally` block's explicit `resources.Dispose()` and `cleanup.Dispose()` calls. A fork task that throws before reaching `finally` (which cannot happen in .NET because `finally` always runs for normal and exception paths on a managed thread) would leak — but this is not possible in practice.

[REPEAT_STACK_GROWTH_PRESSURE]:
- `IO<A>.RepeatUntil(Func<A,bool> predicate)` is defined as `Bracket().Bind(v => !predicate(v) ? RepeatUntil(predicate) : IO.pure(v))`. Each iteration adds one `Bind` closure to the pending continuation chain. The interpreter in `RunAsync(EnvIO)` is a trampoline-style stack machine using a `Seq<StackItem>` stack, so the growth is in that managed stack, not the call stack. However, `Bracket()` calls `WithEnv(env => env.LocalResources)` which produces `IOLocal` — an async-path node — meaning each iteration suspends and resumes via the managed continuation stack. Under a `Schedule`-based `RepeatUntil`, the DSL uses `IOFoldUntil` which avoids the recursive bind and iterates via a `Schedule.Run().GetIterator()` enumerator. Boundless `RepeatUntil` (no schedule) should therefore be used only for terminating predicates known to fire within a bounded number of iterations; for long-running polling loops, use `RepeatUntil(schedule, predicate)` to stay in the non-recursive `IOFoldUntil` path.

[IO_POST_ALLOCATION_COST]:
- `IO<A>.Post()` allocates an `AutoResetEvent` per call — a kernel-backed synchronization primitive — to bridge the `SynchronizationContext.Post` callback back to the awaiting thread. This is non-trivial allocation on hot paths. An `AutoResetEvent` is `IDisposable`; the `Post()` implementation disposes it correctly in a `finally` block. There is no pooling. For marshaling high-frequency operations to a UI or platform thread, composing `Post()` calls per-operation is expensive; the pattern should batch work via a channel or actor whose consumer runs on the target context and returns aggregated results, calling `Post()` at the boundary only once.

[ENVIO_SYNC_CONTEXT_CAPTURE]:
- `EnvIO.New()` with `syncContext = null` captures `SynchronizationContext.Current` at the moment `New()` is called — not at construction of the `IO` tree, and not at `Run()` time. If `IO.RunAsync()` (no-arg form) is called from a thread with a non-null `SynchronizationContext`, that context is stored in the spine env and propagates to all `Post()` calls inside the operation. If `RunAsync()` is called from a thread-pool thread (null `SynchronizationContext`), `Post()` inside the operation falls through to `await RunAsync(env)` directly with no dispatching. A state cell that must always marshal updates to a specific context must inject the context explicitly via `EnvIO.New(syncContext: theContext)`, not rely on ambient capture.
- `EnvIO.LocalSyncContext` creates a new `EnvIO` with `SynchronizationContext.Current` at the point of the property access, sharing the existing `Resources` and `Source`. This is the correct form for an operation that must switch execution context within a pipeline while preserving the existing cancellation and resource scope. The returned `EnvIO` must be used immediately — deferring it into a lazy `IO` node means the context is captured at node creation, not at evaluation, which may be on a different thread.

[RESOURCES_RELEASEALL_TRIE_SWEEP]:
- `Resources.ReleaseAll()` runs via an `IO.lift` lambda that calls `resources.Swap(r => { foreach (item in r) item.Value.Release().Run(envIO); return empty; })`. The swap function runs inside `AtomHashMap.Swap`'s CAS loop. If the CAS fails (concurrent mutation), the entire sweep — including already-executed `Release().Run(envIO)` calls for items in the previous snapshot — re-executes on the retried snapshot. Release effects must be idempotent, or `ReleaseAll` can double-release resources on CAS contention. The `TrackedResourceWithFree<A>` release effect is whatever the caller provides; if it is not idempotent (e.g., sends a network goodbye message), double-release on CAS contention produces duplicate protocol messages. The safe form for non-idempotent teardown is to drain the map key-by-key via `Release<A>(value)` which removes the key before calling the release effect — single-entry removal via `resources.Remove(obj)` before `f.Release()` prevents re-entry on CAS retry.

[LIFECYCLE_FAULT_CELL_WAITING_MECHANISM]:
- A three-state initialization cell (`Initializing`, `Initialized`, `InitializationFailed`) encoded as `Atom<InitState>` with a `validator = _ => true` requires waiters that see `Initializing` to block until the winner transitions to `Initialized` or `InitializationFailed`. A `TaskCompletionSource<InitState>` created before the first `Swap` and signaled in the winning initializer's completion path provides an awaitable seam. The cell's swap function returns the new state; the initializer calls `tcs.TrySetResult(newState)` immediately after the committed `Initialized` swap. Waiters read `atom.Value` first; on `Initializing`, they `await tcs.Task`. This is race-free: if the winner commits `Initialized` BEFORE the waiter checks the atom, the waiter reads `Initialized` directly; if AFTER, the waiter blocks on `tcs.Task` which will be set. The `TaskCompletionSource` must be process-static alongside the atom — both initialized before any concurrent caller can reach the gate.
- A lifecycle state machine encoded as `Atom<LifecycleState>` with a validator `v => v.IsValidSuccessor(atom.Value)` has a critical flaw: the validator receives the CANDIDATE next state, not the current state. The current state must be read inside the CAS loop, not captured once before the loop. The correct validator for a state machine is `_ => true` (no structural constraint in the validator), with the transition constraint enforced in the swap function: `curr => curr.IsValidPredecessor(next) ? next : curr` — the swap function returns the current state unchanged if the transition is invalid, and `SwapMaybe` returning `None` signals the rejection. The validator is the wrong surface for transition constraints; it is for value-domain invariants on the new state, not relational constraints between old and new.

[GENERATED_REGISTRY_CHANGE_SUBSCRIPTION_COST]:
- `AtomHashMap<K,V>.Change` subscription switches ALL subsequent mutations to the `WithLog` trie variant. `WithLog` paths produce a `TrieMap<EqDefault<K>, K, Change<V>> changes` diff map alongside the updated trie on every `SetItem`, `Remove`, `Add`, and `TryAdd`. For singleton registry cells whose occupancy is bounded (e.g., a component registry with ≤100 entries), the diff map allocation per mutation is acceptable. For high-frequency cells (per-request correlation maps, per-frame annotation caches), subscribing `Change` shifts the mutation from zero-diff-allocation to one-allocation-per-mutation even when subscribers do nothing with the patch. The decision to subscribe `Change` on a boundary registry cell must account for mutation frequency, not just the utility of the change notification.
- `AnnounceChange(prev, current, key, change)` checks `change.HasChanged` before calling `this.Change?.Invoke(...)`. For `SwapKey(K, Func<V,V>)`, `change` is constructed as `Change<V>.None` when the event is unsubscribed — the diff is skipped entirely. When subscribed, `SetItemWithLog` builds a `Change<V>` with `(Old: prev, New: new)` — allocating per call. For no-op swaps (swap function returning the same value), `SetItemWithLog` still allocates the `Change<V>` record even if `HasChanged` will be false, because the change is constructed from the trie operation before the predicate is checked. This is a pre-materialized-allocation hazard for idempotent swap functions on subscribed maps.

[CANCELLATION_LINKED_SOURCE_DISPOSAL_CONTRACT]:
- `CancellationTokenSource.CreateLinkedTokenSource(CancellationToken[])` (the params overload) and `CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` (added in .NET 9) both return a new `CancellationTokenSource` that registers callbacks on all parent tokens at construction time. Each parent registration holds a delegate reference back to the linked source. If the linked source is not disposed, those registrations remain live until each parent source fires or is disposed, extending the lifetime of the linked source's callback delegate. The linked source's `Dispose()` removes all parent registrations. For bounded-lifetime boundary cells that create linked sources per-operation, `Dispose()` must be called in every path — success, failure, and timeout — or parent cancellation callbacks accumulate as garbage. In LanguageExt, `EnvIO.New` handles this via `Registration?.Dispose()` in `DisposeNonResources()`. For hand-rolled linked sources outside `EnvIO`, the pattern is a `using CancellationTokenSource linked = CancellationTokenSource.CreateLinkedTokenSource(parent, localToken)` block scoped to the operation.
- `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` — the .NET 9+ `ReadOnlySpan` overload — uses an internal optimized path that registers all tokens in a single allocation when the span length is known at compile time. For the two-token case (parent + per-operation deadline), this overload avoids the intermediate array allocation of the `params CancellationToken[]` form. The `EnvIO.New` factory uses `token.Register(() => source.Cancel())` rather than `CreateLinkedTokenSource`, which registers a single callback without allocating a linked source; this is cheaper but does not expose a `Token` derived from both inputs — the `EnvIO.Source.Token` is a new token that is cancelled transitively by the registration, which is correct for LanguageExt's model.
