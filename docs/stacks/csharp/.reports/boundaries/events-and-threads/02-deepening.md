# Events and Threads — Deepening

[ENVO_SCOPE_VARIANTS_AND_SUBSCRIPTION_ISOLATION]:
- `EnvIO` exposes four distinct local-scope constructors with different ownership profiles. `Local` allocates new `Resources`, new `CancellationTokenSource`, and reads `SynchronizationContext.Current` at access time — full isolation. `LocalResources` allocates new `Resources` only; it inherits the parent's `Source` and `SyncContext`. `LocalCancel` allocates a new `CancellationTokenSource` only; it inherits the parent `Resources` and `SyncContext`. `LocalSyncContext` reads `SynchronizationContext.Current` at access time and allocates a new `CancellationTokenSource`, but inherits the parent `Resources`. Every access to `LocalSyncContext` or `Local` captures `Current` at that moment — these are property getters, not construction-time captures. Calling them from a background thread captures the wrong context; the correct pattern is to evaluate these on the host thread and pass the resulting env down.
- `WithEnv(env => env.LocalResources)` is the scoping form for subscription lifetime within an `IO` chain: resources acquired inside the mapped computation are released in a child scope on completion, without touching the parent scope's resources or cancellation token. `Bracket()` wraps the whole computation in its own resource scope and releases all tracked resources on both success and failure. `BracketFail()` releases only on failure — correct for subscriptions where success deliberately transfers resource ownership out of the bracket.
- The parent resource chain is reference-shared: a forked computation's child `Resources` holds a `parent` reference to the parent scope. `Release` walks up to the parent if the key is not found locally. Disposing the parent `EnvIO` while a fork is live does not automatically cancel the fork — the fork runs until its own cancellation or natural completion. The child `Resources.DisposeU` on the fork's thread releases only the child scope; parent resources remain until the parent env is disposed or explicitly released.
- `Resources.DisposeU` iterates the `AtomHashMap` in hash-traversal order, not insertion or reverse order. Disposal sequence is undefined. When disposal of resource B depends on A still being live (e.g., a subscriber must be detached before closing the connection it reads), the sequence must be encoded explicitly: detach explicitly inside the computation before `use` releases the connection, rather than relying on disposal order.

[EXECUTION_CONTEXT_FLOW_DISCIPLINE]:
- `CancellationToken.Register` captures the ambient `ExecutionContext` at registration time and restores it when invoking the callback, unless `useSynchronizationContext: false` is also specified. `CancellationToken.UnsafeRegister` skips context capture entirely — the callback runs with whatever `ExecutionContext` the cancelling thread carries. `UnsafeRegister` is the correct form for callbacks that are themselves context-free (e.g., writing to a channel writer, swapping an `Atom`) and must not pay the context-capture allocation or risk inherited context leaking into the callback.
- `ExecutionContext.Run` is single-use: a captured context cannot be passed to `Run` more than once. The runtime throws `InvalidOperationException` on a second call. Event subscription tokens that store a captured `ExecutionContext` for replay must re-capture after each use; the common mistake is capturing once and reusing across multiple callback invocations, which works until the second invocation.
- `ExecutionContext.SuppressFlow` suppresses `ExecutionContext` propagation to work queued from the current thread until the returned `AsyncFlowControl` is disposed. The intended use is inside queue-submission loops where ambient context (e.g., `AsyncLocal<T>` values set by a caller) must not bleed into workers. The suppression is thread-local and scoped to the `AsyncFlowControl` lifetime; disposing it restores flow. Nesting suppression calls throws.
- `ThreadPool.UnsafeQueueUserWorkItem<TState>(Action<TState>, TState, preferLocal: bool)` avoids both `ExecutionContext` capture and delegate allocation when combined with a struct `TState`. The `preferLocal: true` flag biases toward the current thread's local work-stealing queue, reducing cross-core contention for tight producer-consumer pairs on the same logical thread. The typed generic overload is allocation-free compared to `WaitCallback` boxing.

[LOCK_SCOPE_REF_STRUCT_HAZARDS]:
- `Lock.EnterScope()` returns `Lock.Scope`, a `ref struct`. It cannot be stored in a field, passed to a local function, used inside an `async` method, or captured in a lambda. A gate that must span an `await` point requires `SemaphoreSlim(1, 1)` with `WaitAsync`. Attempting to use `Lock` across an await compiles to a diagnostic and the `Scope` is silently not disposed, leaving the lock permanently held.
- `Lock.Scope.Dispose` throws `SynchronizationLockException` if the current thread does not hold the lock at the time of disposal. This fires when `Scope` is passed across thread boundaries (illegal for a `ref struct` but can occur through unsafe or codegen paths) or when the finalizer runs on a GC thread after the scope was incorrectly stored. The `ref struct` constraint prevents the most common misuses at compile time but does not cover IL-level or reflection paths.
- `Lock.IsHeldByCurrentThread` allows reentrancy detection: the lock supports recursive entry up to an implementation-defined limit. Detecting reentrancy before taking a lock eliminates the need for a parallel recursion counter. The intended pattern for seams that may be called from both lock-owning and non-owning code: `if (!_gate.IsHeldByCurrentThread) using (_gate.EnterScope()) { ... }`.
- `Lock` generates `Monitor.Enter`/`Monitor.Exit`-equivalent IL when used with the `lock` keyword but routes through `Lock.EnterScope`/`Scope.Dispose` instead of `object`-based monitor. This eliminates the hidden SyncBlock slot allocation on `object` lock targets and removes the boxing that occurs when a value type is used as a lock object. Profiler and debugger tooling that inspects monitor contention correctly identifies `Lock` instances.

[CHANNEL_SYNCHRONOUS_CONTINUATION_HAZARD]:
- `ChannelOptions.AllowSynchronousContinuations = true` allows the thread calling `WriteAsync` or `TryWrite` to directly invoke the continuation of an awaiting `ReadAsync` or `WaitToReadAsync` on that same thread, inline, before the write returns. This eliminates a thread switch for tightly coupled producer-consumer pairs but introduces two hazards: (1) unbounded stack growth when the continuation itself writes to the same or a downstream channel, creating a synchronous call chain; (2) reentrancy when the writer holds a lock or is inside a callback that the reader's continuation also needs. The default `false` posts continuations asynchronously, breaking the call chain.
- `ChannelWriter<T>.Complete(Exception? error)` both closes the channel and, if `error` is non-null, propagates it to `ChannelReader<T>.Completion` as a faulted task and to `WaitToReadAsync` as an exception on the next call. Fault delivery and nominal projection therefore share the same completion signal; this is the mechanism for writer-originated errors surfacing to consumers. Separate fault channels are needed only when the consumer must continue processing buffered items after a partial failure — when the error should terminate consumption, `Complete(error)` is sufficient.
- `Channel.CreateBounded<T>(0)` produces an internal `RendezvousChannel<T>` where every `WriteAsync` blocks until a matching `ReadAsync` is pending. Neither side proceeds until both are ready. Using this as a single-item buffer is incorrect and deadlocks when both sides are on the same thread. The correct use is explicit synchronization between two independent async workers that must hand off exactly once per cycle.
- `ChannelReader<T>.CanCount` and `CanPeek` are capability flags; `Count` throws `NotSupportedException` and `TryPeek` returns `false` if the respective capability is absent. Calling `Count` on a channel created with `SingleReader = false` or unbounded variants may throw. Capability guards before using these must be explicit rather than try/catch, because the exception path obscures logic.

[PERIODIC_TIMER_AND_TIME_PROVIDER_SEAM]:
- `PeriodicTimer(TimeSpan, TimeProvider)` accepts a `TimeProvider` injection point. `FakeTimeProvider` from `Microsoft.Extensions.TimeProvider.Testing` drives deterministic ticks in tests. The timer respects the `TimeProvider`'s notion of elapsed time, not wall clock; advancing the fake provider past the period fires `WaitForNextTickAsync`. This is the correct testing seam for schedule-driven workers that own a `PeriodicTimer`.
- `PeriodicTimer.WaitForNextTickAsync` cancellation via its `CancellationToken` affects only the single outstanding wait; the underlying timer continues firing. Cancelling the token does not stop subsequent ticks. Stopping the timer requires calling `Dispose()`, which completes the current `WaitForNextTickAsync` with `false`. The read loop pattern is `while (await timer.WaitForNextTickAsync(ct)) { ... }` — the loop exits on either cancellation (throws `OperationCanceledException`) or disposal (returns `false`). Missing the `false` return causes the loop to exit without the worker having been explicitly stopped.
- `PeriodicTimer.Period` is a mutable property. Setting it after construction changes the period for all future ticks but does not affect the currently pending `WaitForNextTickAsync`. Rate changes at a seam boundary — e.g., responding to a DPI or refresh-rate change — update `Period` on the next cycle rather than requiring a new timer instance.
- A `PeriodicTimer` used as a fold worker driver must be owned and disposed by the worker itself, not by a parent scope that may outlive or outlast the worker's subscription set. If the parent disposes the timer while the worker is between ticks, the worker's `WaitForNextTickAsync` returns `false` cleanly. If the parent disposes it while the worker is inside the loop body, the worker completes its current iteration and sees `false` on the next wait. No in-flight work is interrupted.

[SAFHANDLE_CALLBACK_WINDOW_PROTOCOL]:
- `SafeHandle.DangerousAddRef(ref bool success)` increments the internal reference counter, preventing `ReleaseHandle` from being called by the finalizer or `Dispose` while the counter is elevated. The `success` parameter must be checked before calling `DangerousGetHandle`; if `success` is `false`, the handle was already closed and `DangerousGetHandle` returns a stale value. The symmetric pair is always `try { handle.DangerousAddRef(ref success); if (success) { /* use */ } } finally { if (success) handle.DangerousRelease(); }`. Calling `DangerousRelease` without a matching `DangerousAddRef` decrements below zero, corrupting internal state.
- In an event callback window, the subscribe–use–unsubscribe triplet for a `SafeHandle`-wrapped resource is: (1) `DangerousAddRef` before registering the handler, (2) the handler body calls `DangerousGetHandle`, (3) `DangerousRelease` after detach completes. If the handler is called on a thread pool thread after step (3) — possible when the bus fires independently — the handle value obtained is stale. The fence is: detach synchronously before calling `DangerousRelease`, and guard the handler body with an `IsInvalid` or liveness check on the acquired pointer.

[ATOM_SWAP_CHANGE_EVENT_HAZARDS]:
- `AtomHashMap<K, V>.Swap(TrackingHashMap → TrackingHashMap)` carries a critical invariant: change tracking is only emitted for transforms applied to the `TrackingHashMap` argument provided by the swap function. If the swap function ignores the argument and builds a fresh `TrackingHashMap`, the change event carries a diff that shows only the final state with no tracked intermediate changes. Consumers depending on fine-grained per-key change events receive only a bulk replace notification. The correct pattern always threads the provided map through transforms (`map.SetItem`, `map.Remove`, etc.) rather than constructing a new collection.
- `Atom<A>.Change` fires synchronously on the swapping thread inside the CAS success path, before `Swap` returns. Because `Swap`'s function may be called multiple times across competing threads, the event fires exactly once for each committed value. Subscribers that block inside the change handler (e.g., waiting on a semaphore held by the swapping thread) deadlock because the swap does not return until the change event returns. Change handlers must be non-blocking; cross-thread work must be submitted via channel write, not executed inline.
- The two-argument `Atom<M, A>` metadata form's change event carries only the new value `A`, not the metadata `M` or the previous value. A subscriber that needs the prior value to compute a delta must maintain a shadow read of `atom.Value` before the swap. Reading `atom.Value` after the change event fires gives the committed value; it may already have advanced past the value that triggered the notification if another swap occurred during the handler.

[IO_REPEAT_AND_SUBSCRIPTION_LIFETIME_CONSTRAINT]:
- `IO<A>.Repeat(Schedule)`, `RepeatWhile`, and `RepeatUntil` release all resources acquired within each iteration after that iteration completes. This is explicit in the contract: resources cannot be acquired inside a `Repeat` body and held across iterations. A subscription opened inside `Repeat` is detached and disposed after each cycle. The correct pattern for a persistent subscription with a repeating read loop is: open the subscription outside the `Repeat` (e.g., via `use` at the outer `IO` level), then bind the repeating reader inside. The outer `use` persists until the parent scope is released.
- `IO<A>.Retry(Schedule)` releases resources on each failed attempt but retains them on success. This asymmetry means a retry loop that opens a connection on each attempt and succeeds on attempt N holds only one connection — the others were cleaned up on failure. A retry loop that opens a subscription on each attempt and fails repeatedly does not accumulate subscriptions.
- `IO<A>.Uninterruptible` wraps the computation in a local environment that ignores cancellation-token signals. The computation runs to completion even if the parent token is cancelled. This is the correct form for finalization paths that must complete regardless of cancellation — detaching subscribers, draining a queue, flushing state — because a parent cancel must not interrupt these cleanup steps. `Uninterruptible` composed with `Bracket`'s finalizer lambda is the canonical resource-safe teardown form.

[CONFIGUREAWAIT_OPTIONS_AT_SEAM]:
- `ConfigureAwaitOptions.ForceYielding` forces an already-completed `Task` or `ValueTask` to post its continuation asynchronously rather than completing inline. The intended use is inside event dispatch loops where the event source fires synchronously and the handler must not run on the event source's stack frame. Without `ForceYielding`, awaiting a pre-completed task runs the handler synchronously on the event thread, causing reentrancy if the handler schedules more events or acquires shared state.
- `ConfigureAwaitOptions.SuppressThrowing` prevents the awaited expression from throwing when the task faults or is cancelled; it converts the fault into a silent `default(T)` result. The correct use at a seam boundary is to `await task.ConfigureAwait(ConfigureAwaitOptions.SuppressThrowing)` and then check `task.IsCompletedSuccessfully` or `task.Exception` on the original task handle. Using `SuppressThrowing` without inspecting the outcome silently discards errors; this is only correct when the caller has a separate fault channel and the task fault is already emitted there.
- `ConfigureAwaitOptions` are flags and compose: `ConfigureAwaitOptions.None | ConfigureAwaitOptions.ForceYielding | ConfigureAwaitOptions.SuppressThrowing` yields asynchronous, non-throwing completion regardless of prior state. This is the form for a host-thread dispatch wrapper that must yield before running and must not propagate exceptions through the await site.

[TASKSCHEDULER_SYNCHRONIZATIONCONTEXT_CAPTURE]:
- `TaskScheduler.FromCurrentSynchronizationContext()` throws `InvalidOperationException` when `SynchronizationContext.Current` is `null`. This happens on any thread that has not had a context installed — thread pool threads, `LongRunning` task threads, and non-UI background threads all have `null` `Current`. The capture must happen on the host thread while that thread's context is installed. The correct pattern is to capture the scheduler or context at composition-root initialization on the host thread, store it in the environment, and inject it into effects rather than re-reading at call sites.
- `SynchronizationContext.Send` is synchronous and dispatches the callback on the target context's thread, blocking the calling thread until the callback returns. Calling `Send` from a thread that participates in the target context's message loop deadlocks. `Post` queues asynchronously and returns immediately. At a seam boundary, `Send` is only safe when the calling thread is provably not the target context's thread and the target loop is not blocked. `Post` with a completion signal (channel write or `TaskCompletionSource.SetResult`) is the correct bidirectional form.

[SUBSCRIPTION_RESOURCE_SCOPE_ISOLATION_WITH_IO_WITHENV]:
- `IO<A>.WithEnv(env => env.LocalResources)` creates a child resource scope for the duration of the mapped computation. Any `use`/`useAsync` calls inside that computation register in the child scope. On completion (success or failure) the child scope's `DisposeU` runs. The parent scope is not disturbed. This is the correct form for one-shot activation/deactivation cycles: the activation acquires subscriptions inside a `WithEnv(LocalResources)` block, and the block's completion auto-releases them regardless of how the computation terminates. Combined with `Retry`, each failed activation attempt cleans up its partial subscriptions before the next attempt.
- `WithEnvFail` releases only on failure, not on success — the complementary form of `BracketFail`. When the activation succeeds and the caller wants to retain subscriptions until an explicit deactivation rather than until the computation exits, `WithEnvFail` paired with an outer `Atom<Option<Resources>>` ownership token allows explicit manual release on the deactivation path: the `Resources` value is swapped out of the atom and `DisposeU` called explicitly.
- `EnvIO.LocalCancel` forks a new cancellation token sourced from the parent token while inheriting the parent's resources and sync context. This is the correct form for a guarded region that must be cancellable independently of the parent — e.g., a subscription whose read loop can be stopped without tearing down the parent environment's resources. The child source is disposed when the `EnvIO.LocalCancel` env is disposed; parent cancellation propagates to the child via the registration installed in `EnvIO.New`.

[CHANNEL_FAULT_AND_COMPLETION_PROTOCOL]:
- `ChannelWriter<T>.Complete(error)` with a non-null `error` propagates the fault to `ChannelReader<T>.Completion` and surfaces it from `WaitToReadAsync` on the next call after the buffer is drained. This means buffered items written before the fault are still readable; the fault is only delivered after the last buffered item has been consumed. Consumers that read via `ReadAllAsync` will see the fault as an exception thrown from the `IAsyncEnumerable` iteration, not as a separate signal.
- A fold worker that detects an unrecoverable input error must write to a dedicated fault `ChannelWriter<Error>` before calling `TryComplete` on the primary projection writer. The two operations are non-atomic; a consumer that reads from both must handle the case where the fault channel has an item but the projection channel has not yet been completed. Using a `Channel<Either<Error, T>>` avoids this race but forces every nominal consumer to handle the error arm; the two-channel split is architecturally cleaner when fault consumers and projection consumers are distinct.

[UNSAFE_REGISTER_AND_CONTEXT_FREE_CALLBACKS]:
- `CancellationToken.UnsafeRegister(Action<object?, CancellationToken>, object?)` is the preferred form for all callbacks that (a) do not use `AsyncLocal<T>` values from the registration context and (b) perform only context-free work such as `ChannelWriter.TryWrite`, `Atom.Swap`, or setting a `ManualResetEventSlim`. It avoids `ExecutionContext.Capture` allocation and the restore overhead during cancellation. The `Action<object?, CancellationToken>` overload receives the cancellation token itself, which allows the callback to inspect `IsCancellationRequested` without a closure.
- `CancellationTokenRegistration` is both `IDisposable` and `IAsyncDisposable`. Synchronous `Dispose` unregisters the callback; if the callback is currently executing on another thread, `Dispose` blocks until the callback finishes. `DisposeAsync` returns immediately after unregistering; it does not wait for in-flight callbacks. Using `await using` on a `CancellationTokenRegistration` in a context where an in-flight callback writes to a resource that the caller then disposes creates a window where the resource is accessed after disposal. Synchronous `Dispose` is safer when the callback accesses resources whose disposal is the next step.
