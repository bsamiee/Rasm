# AST Is The Single Source

[ONE_FOLD_ENGINE_UNDER_EVERY_PROJECTOR]:
- `Compiler<A> = (ast: AST, path: ReadonlyArray<PropertyKey>) => A` and `Match<A> = { [K in AST["_tag"]]: (ast: Extract<AST, { _tag: K }>, compile: Compiler<A>, path) => A }` are the entire structural-recursion machinery of the lane: `getCompiler<A>(match)` ties the open recursion into a closed `Compiler<A>`, threading the resulting compiler back into every arm as the `compile` argument so each node folds its children by the same compiler rather than re-dispatching — the fixpoint is supplied, never re-opened per node.
- The arm map is keyed on `AST["_tag"]`, so a `Match<A>` is exhaustive over all twenty-three node tags by the mapped-type domain; omitting an arm is a missing-key error at construction, the same totality the codec, the wire-contract emitter, and the renderer all inherit from one table shape — a projector cannot silently skip a node kind.
- `Pretty.match` is exported with type `Match<Pretty<any>>` and `Pretty.make(schema)` is `getCompiler(match)` applied to `schema.ast` at `path = []`; the renderer is not a sibling traversal beside the codec, it is the same fold engine over a different `A` (`Pretty<To> = (a: To) => string`), so a value's printer and its decoder differ only in the carrier the arms produce.
- A bespoke projection is one `Match<A>` filled for the node kinds the target cares about, defaulting the rest to a base case — never a hand-walked tree of `if (ast._tag === ...)`; the table is the projector, `getCompiler` is the only recursion, and the `path` accumulates the failure/document locus for free as the fold descends.

```typescript
import { Record, Schema, SchemaAST as AST } from 'effect'
const tags = ['Declaration', 'Literal', 'UniqueSymbol', 'UndefinedKeyword', 'VoidKeyword', 'NeverKeyword', 'UnknownKeyword', 'AnyKeyword', 'StringKeyword', 'NumberKeyword', 'BooleanKeyword', 'BigIntKeyword', 'SymbolKeyword', 'ObjectKeyword', 'Enums', 'TemplateLiteral', 'Refinement', 'TupleType', 'Suspend', 'Transformation'] as const
const depth: AST.Compiler<number> = AST.getCompiler({
    ...Record.fromEntries(tags.map((t) => [t, () => 0] as const)),               // every leaf arm closed at zero — the table is total over AST["_tag"]
    TypeLiteral: (node, compile, path) => 1 + node.propertySignatures.reduce((m, p) => Math.max(m, compile(p.type, [...path, p.name])), 0),   // node narrowed to AST.TypeLiteral by the arm key
    Union: (node, compile, path) => 1 + node.types.reduce((m, t) => Math.max(m, compile(t, path)), 0),                                       // recursion threads through `compile`, not a re-dispatch
} as AST.Match<number>)
const owner = Schema.Struct({ tag: Schema.Literal('<variant-a>'), inner: Schema.Struct({ leaf: Schema.Number }) })
const nesting = depth(owner.ast, [])   // 2 — one Match table, path accumulated, no parallel walker
```

[OWNER_IS_AN_AST_CARRYING_VALUE]:
- Every schema owner satisfies `readonly ast: AST` and `Schema.make<A, I, R>(ast): SchemaClass<A, I, R>` re-promotes any `AST` back to a typed schema — the runtime declaration IS its AST plus a phantom triple, so the value-level toolkit (`typeSchema`, `encodedSchema`, subschema projection) is the AST-level toolkit (`typeAST`, `encodedAST`, `pick`/`omit`) wrapped in `make`, the two phases never a hand-written mirror of one another.
- `typeAST(ast)`, `encodedAST(ast)`, `encodedBoundAST(ast)` are the three phase cuts as pure `AST -> AST` rewrites the schema-level `typeSchema`/`encodedSchema`/`encodedBoundSchema` delegate to: `typeAST` discards every `Transformation` and keeps the `to` side, `encodedAST` keeps the `from` side and strips refinements, `encodedBoundAST` keeps refinements up to the first `Transformation` — the chain `make(typeAST(owner.ast))` is identical to `typeSchema(owner)`, so a projector that needs the decoded interior reads the rewritten AST node, never re-derives the cut.
- `keyof(ast)`, `pick(ast, keys)`, `omit(ast, keys)`, `partial(ast, options?)`, `required(ast)`, `mutable(ast)` are AST→AST transforms whose return is `TypeLiteral | Transformation` (pick/omit) or `AST` — the schema-level subschema family lifts these through `make`, so a sub-shape's codec, type, and validator descend from one AST rewrite, the field-set surgery happening on the substrate and the triple re-extracted on the way out.
- `annotations(ast, overrides)` is the AST-level merge that DELETES any prior `IdentifierAnnotationId` while overwriting duplicate slots — the same delete-on-merge the owner-method form inherits, so re-identifying a node is a fresh AST with the stale identifier gone, never a node carrying two competing identifiers a downstream `$ref` emitter would race on.

[PROJECTORS_DIVERGE_ONLY_IN_THE_CARRIER]:
- The codec compiler produces `(u: unknown, opts, ast) => Effect<A, ParseIssue, R>`, `Pretty.match` produces `Pretty<A>`, the JSON-schema fold produces `JsonSchema7`, the arbitrary fold produces `LazyArbitrary<A>`, and `Schema.equivalence` produces `Equivalence<A>` — five `A`s, one `Match` shape, one `getCompiler` recursion; the four secondary families are not parallel walkers but the same fold parameterized by what each arm returns, so adding a node kind to the schema AST extends every projector's obligation through the one mapped-type key set.
- A projector's phase is selected by which AST rewrite it is handed, not by a flag or a mode argument: the document fold runs over the `encodedAST` and the generation fold over the `typeAST`, so the two consume disjoint sub-graphs of one owner and the well-known document-versus-generation phase divergence is a structural consequence of the two rewrites, not a behavior the projector chooses — a custom secondary family inherits the same phase selection by which cut it folds, never by re-deciding it.
- A leaf node carries its projector overrides as annotations on the node itself, so the fold reads them in-arm: a `NumberKeyword` with a narrowed `ArbitraryAnnotationId` slot, a `Refinement` whose `JSONSchemaAnnotationId` slot emits `minimum`/`pattern`, a `Declaration` whose `SurrogateAnnotationId` slot redirects the fold onto a substitute AST — the override is data the arm consults, never a registration the projector tracks beside the tree.
- A custom secondary family (a SQL-column-type emitter, a form-field descriptor, a binary-layout planner) is authored as one `Match<ColumnSpec>` over the owner's AST, reusing every leaf annotation already present and inheriting the totality, the `path`-based locus, and the suspend handling from `getCompiler` — the new projection is a table read off the same substrate, the hand-walked recursion the doctrine deletes.

[NODE_TAG_IS_THE_DISCRIMINANT_THE_VALUE_ALREADY_CARRIES]:
- The twenty-three-arm `AST` union is itself a `_tag`-discriminated owner — `Declaration | Literal | UniqueSymbol | ...Keyword | Enums | TemplateLiteral | Refinement | TupleType | TypeLiteral | Union | Suspend | Transformation` — so `Match<A>` dispatching on `AST["_tag"]` is the schema doctrine applied to the schema's own metalanguage: the discriminant is recoverable from the node, never a parallel kind enum, and the per-tag `isLiteral`/`isUnion`/`isTransformation` guards are the narrowing reads when only one branch is wanted outside a full fold.
- The composite nodes expose their children as the fold's recursion frontier with no accessor indirection: `TypeLiteral.propertySignatures` (each a `PropertySignature` carrying `name`, `isOptional`, `isReadonly`, `type`) and `.indexSignatures`, `Union.types` (a `Members<M> = readonly [M, M, ...M[]]` proving arity ≥ 2 at the type level), `TupleType.elements`/`.rest`, `Refinement.from`, `Transformation.from`/`.to`, `Suspend.f` — a structural query maps over the typed child arrays and the fold descends each, the value-level `.members`/`.fields` re-derivation of the bedrock being the schema wrapper over these AST children.
- `Suspend.f: () => AST` is the one node the fold must not eagerly force — a recursive owner's compiler arm calls `compile(node.f(), path)` only under a depth or memo guard the `ArbitraryGenerationContext.maxDepth`/`depthIdentifier` carries, so the `Suspend` arm is where a projector either terminates the recursion or diverges; the `identifier` annotation on the suspend is the `$ref` target a `JSONSchema` fold emits instead of inlining, the same node serving recursion termination and document deduplication.
- `OptionalType extends Type` and `PropertySignature extends OptionalType` add `isOptional`/`isReadonly`/`name` as the field-level modifier slots a fold reads to emit the `?` token and the readonly marker — the field algebra the bedrock's `PropertySignature` carrier extracts at the type level is these boolean slots at the value level, one source for the optionality the codec validates and the optionality a form-descriptor projector renders.

[TRANSFORMATION_NODE_IS_THE_THREE_LEG_PIVOT]:
- A `Transformation` node carries `from: AST`, `to: AST`, and `transformation: TransformationKind` where `TransformationKind = FinalTransformation | ComposeTransformation | TypeLiteralTransformation` — the per-node discriminant telling a fold whether the seam is a user `decode`/`encode` pair (`FinalTransformation`), a `compose` join with no own functions (`ComposeTransformation`), or a key-level rename/default rewrite (`TypeLiteralTransformation` holding `PropertySignatureTransformation` records) — so a projector that must reason about the seam reads the `_tag` of the kind, never inspects opaque closures.
- The decode and wire phases are exactly the two halves of this one node: `typeAST` walks toward `.to` and discards the transformation, `encodedAST` walks toward `.from` — so the Encoded/Type asymmetry the bedrock catalogs is a single graph with two traversal directions, not two declarations, and a fold authored once over the substrate already has both phases available by which child it descends.
- `FinalTransformation.decode`/`encode` and `ComposeTransformation`/`TypeLiteralTransformation` are the AST residue of `transform`/`transformOrFail`, `compose`, and the property-signature default/rename machinery respectively — `PropertySignatureTransformation` carries `from: PropertyKey`, `to: PropertyKey`, and `Option -> Option` `decode`/`encode` where `none` means key-absent — so the field rename and the defaulted-presence asymmetry are one node kind a fold pattern-matches, the wire-key↔type-key correspondence read off the node rather than tracked in a parallel rename map.
- The transform `decode`/`encode` receive `self: AST.Transformation` as an argument, so a leg can read its own node's annotations mid-parse — the `parseIssueTitle`, `identifier`, and `message` slots that shape the failure projection are reachable from inside the success projection, one annotated node feeding both the value families and the error family.

[ISSUE_TREE_MIRRORS_THE_AST_AND_NAMES_THE_NODE]:
- The decode-failure projection is the structural recursion run in reverse: `ParseIssue = Type | Missing | Unexpected | Forbidden | Pointer | Refinement | Transformation | Composite`, and the recursive issues carry the AST node that rejected — `Type.ast: AST`, `Refinement.ast: AST.Refinement`, `Transformation.ast: AST.Transformation`, `Composite.ast: AST` — so a failure is a sub-tree of the same graph the codec folded, the locus derived from structure rather than flattened to a string.
- `Pointer` carries `path: Path` and a nested `issue` built from the same `ReadonlyArray<PropertyKey>` the `Compiler` threads as its second argument — the success fold accumulates `path` to name a `$defs` reference site and the failure fold accumulates it to locate an error, one path discipline spanning both directions, so a multi-field decode error names `["outer", "inner", 0]` by the identical descent a JSON-schema fold uses to place a `$ref`.
- The `Refinement` issue's `kind: "From" | "Predicate"` and the `Transformation` issue's `kind: "Encoded" | "Transformation" | "Type"` are the leg discriminants naming WHICH part of the node the fold was inside when it rejected: a `Transformation` issue tagged `"Encoded"` failed before the seam, `"Type"` after — the three-leg pivot of the transform node surfaced as a three-way failure discriminant, the leg recoverable from the issue without re-deriving the pipeline position.
- `Composite` holds `SingleOrNonEmpty<ParseIssue>`, so the failure sub-tree's branching arity mirrors the AST's composite arity — a `TypeLiteral` rejecting two fields yields a `Composite` of two child issues at two `Pointer` paths, the failure graph isomorphic to the slice of the value graph the fold visited, never a flattened single message that loses which node objected.

[ANNOTATION_IS_THE_SLOT_THE_FOLD_READS_NOT_A_REGISTRY]:
- Every node implements `Annotated` (`readonly annotations: Annotations` where `Annotations` is an open `[_: symbol]: unknown` record) and `getAnnotation<A>(key: symbol)(annotated): Option<A>` is the single typed read for all of them — `ArbitraryAnnotationId`, `PrettyAnnotationId`, `EquivalenceAnnotationId`, `JSONSchemaAnnotationId`, `MessageAnnotationId`, `SurrogateAnnotationId`, `DecodingFallbackAnnotationId`, `IdentifierAnnotationId` are symbol keys into the one record, so a projector arm consults `Option`-typed slot reads, never a per-projector lookup table indexed beside the tree.
- The substrate exposes named getters only for the slots a fold consults structurally (`getIdentifierAnnotation`, `getJSONSchemaAnnotation`, `getJSONIdentifier`, `getDecodingFallbackAnnotation`, `getParseIssueTitleAnnotation`, `getSurrogateAnnotation`) — each is `getAnnotation` pre-applied to its `*AnnotationId` symbol returning `Option<Slot>`; the generation-side slots (`ArbitraryAnnotationId`, `PrettyAnnotationId`, `EquivalenceAnnotationId`) carry no named getter, so a projector reads them through the generic `getAnnotation<Slot>(id)`, the asymmetry encoding that codec-shaping slots are first-class to the substrate while generation slots are consumed only by their owning module's fold.
- A missing override is `Option.none` the fold falls through to its derived arm — the override-or-derive decision is one `Option` read at the node, the value families and the failure family drawing every override from the one annotated record on the substrate, never a per-projector registration tracked beside the tree.
- `SurrogateAnnotation = AST` is the slot that redirects the fold itself: a `Declaration` carrier with no inner structure attaches a surrogate AST so `JSONSchema.make` and `Pretty.make` traverse the surrogate's shape, the projection-visible structure being a node slot — a native carrier presents a struct-shaped document while decoding through an opaque constructor, the fold reading the surrogate where the real node is structurally silent.
- `getJSONIdentifier` prefers the `JSONIdentifierAnnotationId` slot then falls back to `IdentifierAnnotationId`, so the identifier that names a node in an error message is the same identifier that drives `$ref` deduplication unless a JSON-specific name overrides it — one slot family serving the diagnostic locus and the wire-document reference, the override scoped per concern by which symbol the fold reads first.

[TEMPLATE_NODE_COMPILES_TO_A_REGEX_THE_SUBSTRATE_OWNS]:
- A `TemplateLiteral` node holds `head: string` and `spans: NonEmptyReadonlyArray<TemplateLiteralSpan>` where each span carries a `literal` separator and a `type: StringKeyword | NumberKeyword | Literal | TemplateLiteral | Union<...>` — the structured-key grammar is data on the substrate, and `getTemplateLiteralRegExp(ast)` / `getTemplateLiteralCapturingRegExp(ast)` compile that node to a validating or capturing `RegExp`, so the structured-string validator is derived from the AST by the library, never a hand-authored pattern beside the schema.
- The capturing variant is the parse direction the bedrock's `TemplateLiteralParser` rides: the same node yields a validator (non-capturing) and a destructurer (capturing) by which compiler is run over it, so a flat key both validates and decomposes from one declaration, the regex the substrate emits replacing the loose `string.match` plus index destructure at every consumer.
- The span `type` admitting a nested `TemplateLiteral` or a `Union` of spans means the regex fold is itself recursive over the substrate — a multi-segment key with an alternation segment compiles to a nested group by the same descent a codec fold performs, so the structured-key grammar's depth is bounded only by the AST, never by a hand-maintained pattern that must be re-edited per added segment.

[REACH_LIMIT_FENCE]:
- A bespoke `Match<A>` earns its place only by deriving a real secondary family (a column-type plan, a form descriptor, a wire-layout) off the owner's AST and deleting the hand-walked recursion it replaces; a `Match` that re-emits what the codec, `Pretty`, `JSONSchema`, `Arbitrary`, or `equivalence` fold already yields is the projector spelled longhand, rejected even when correct — the proof is the carrier `A` being a target no shipped projector produces, the fold reaching nodes the default cannot.
- Reading or rewriting an owner's `.ast` directly is admitted only where the schema-level surface cannot express the rewrite — a phase cut beyond `typeAST`/`encodedAST`/`encodedBoundAST`, a node-kind-conditional surgery the `pick`/`omit`/`partial`/`mutable` family does not cover; a hand-built `new TypeLiteral(...)` reconstructing what `Schema.Struct` already declares is the substrate touched gratuitously, the owner re-stated at the AST level the doctrine deletes — descend to the AST to FOLD it, never to rebuild a node a combinator constructs.
- `getCompiler` over a `Suspend`-bearing AST without a depth or memo guard in the suspend arm diverges, not errors — an unbounded recursive generator or an infinite document; the `ArbitraryGenerationContext.maxDepth`/`depthIdentifier` and the suspend's `identifier` annotation are the termination the fold MUST consult, so a custom `Match` over a recursive owner that omits the suspend guard is the silent non-termination the substrate cannot catch for the author.
- A projector override fires only if its `*AnnotationId` symbol sits on the exact node the fold visits for that carrier, and `typeAST`/`encodedAST` hand different projectors different nodes — so a generation override placed on a `Transformation` node is never reached by an `Arbitrary.make` that folds the `to`-side rewrite, the slot compiling and silently never firing, a node-routing mismatch the type system does not flag because the open `Annotations` record accepts any symbol at any node.
- A `Refinement` whose predicate carries no recognized `jsonSchema` slot and no schema-id contributes nothing to the `JSONSchema` fold while still narrowing the codec and constraining the arbitrary — so a wire-contract document silently omits a bound the decoder enforces; the refinement must annotate the keyword (`minLength`, `pattern`) for the three projections to agree, the unannotated refinement the trap where the document under-specifies what the codec checks.
