# Error Monoid And Expected/Exceptional Catch Algebra

[FAILURE_VALUE_LATTICE]:
- The failure value is one abstract record that is simultaneously `Monoid<Error>` and `Semigroup<Error>`, so the failure carried on every rail is already a combinable value, not a leaf. Two orthogonal abstract booleans partition the lattice: `IsExpected` marks a domain failure that belongs on the rail and may be caught and recovered; `IsExceptional` marks an escaped exception that the runtime lifted onto the rail. The two are not complementary by construction — they are independently abstract — but every concrete leaf sets exactly one true, and the aggregate case derives both from its children.
- The expected leaf carries message, integer code, and an optional inner error, and declares equality by code: two expected errors with equal nonzero code are equal regardless of message, and fall back to message equality only when the code is zero. This single decision makes a localized or templated message irrelevant to matching — the code is the catchable identity, the message is display-only. Any recovery key that must survive translation is a code, never a string.
- The exceptional leaf wraps a live exception in a field that is deliberately non-serialized and private. Crossing a serialization boundary erases the exception object while the record stays exceptional: the reconstructed value still answers `IsExceptional` true, still carries message and the exception's `HResult` as its code, but its stack trace and exception type are gone. An exceptional error is therefore a one-way bridge — it carries the throw inward as a value but does not carry it back out across a wire intact, which is exactly the containment intent: exceptions do not leak across application boundaries, only their sanitized projection does.
- The aggregate case holds an ordered sequence of errors and folds the two flags structurally: it is exceptional if *any* child is exceptional, and expected only if *every* child is expected. One exceptional failure inside a batch of expected validation errors contaminates the whole batch to exceptional. Code that routes on the batch flag — leak the exceptional, recover the expected — must therefore split the sequence by per-child flag before routing, never trust the aggregate flag to mean the batch is homogeneous.

[MONOID_FLATTENING_LAW]:
- The combine operation is associative append that auto-promotes: a single error appended to a single error yields a two-element aggregate; an aggregate appended to an aggregate concatenates their sequences into one flat aggregate; a single appended to an aggregate cons-prepends. The structure never nests an aggregate inside an aggregate — the monoid maintains a flat sequence invariant at every append, so depth stays one and the fold over a batch is always a single linear pass.
- The empty aggregate is the monoid identity and behaves as a present-but-information-free failure, the failure analogue of absence: still a failure, but carrying nothing, with count zero. Identity absorption is wired into combine on both sides — appending the empty aggregate to anything returns the other operand unchanged, so accumulating validation failures across a traversal where some steps contribute the empty error produces exactly the informative errors with no empty padding. The framework's canonical no-information error is this empty aggregate, reused as a shared singleton.
- Because combine is the monoid and the same operation backs the `+` operator, a sequence of independent fallible checks accumulates into one flat aggregate by reduction, and the validation applicative's failure-combination is precisely this monoid. The accumulation surface and the catch-matching surface share one structure: a predicate that reaches into a batch sees a flat sequence with no recursion to unwind.

[REACH_INTO_AGGREGATE]:
- Containment predicates descend into the aggregate without the caller unwrapping it. Code-match tests the top code then asks each child; type-match tests `this is E` then asks each child; identity-match against another error tests the pair then asks each child. The consequence is that a catch keyed on a code or an error-type matches when that code or type appears *anywhere* in an accumulated batch, even mixed among non-matching siblings — a catch is a membership test over the failure set, not an equality test against a single failure.
- Filtering by error subtype returns only the matching children, collapsing to the no-information empty error when none match rather than throwing or returning a null. This makes "extract the validation errors, ignore the rest" a total projection: filter to the expected subtype, fold what remains, and the absence case is already the identity that downstream combine absorbs.
- The first/rest projection treats the aggregate as a list: head is the first child or the empty error when there are none, tail is the aggregate with head removed or the empty error when one or none remain. A single non-aggregate error is its own head with an empty tail. This lets a fail-fast consumer pattern-match head without a separate "is this a batch" branch — the projection is uniform across single and aggregate.

[EXCEPTION_CANONICALIZATION]:
- Lifting an exception to an error is not a blind wrap — it is a canonicalizing dispatch. Cancellation and its task-cancellation subtype both collapse to one shared expected `Cancelled` singleton; timeout collapses to a shared expected `TimedOut` singleton; an aggregate exception flattens each inner exception through the same lift and rebuilds the error aggregate; the framework's own wrapper exceptions unwrap back to the exact error they were carrying, so a round trip through `throw` and catch is lossless for framework errors. Only a genuinely foreign exception becomes a fresh exceptional leaf. The load-bearing result: cancellation and timeout arrive on the rail as *expected, catchable, code-stable* values rather than as exceptional leaks — they can be matched by code and recovered, and they are not contaminated into the exceptional partition.
- The reserved-code band is the framework's private vocabulary: a contiguous block of large negative integers names cancelled, timed-out, empty-sequence, closed, bottom, validation-failed, end-of-stream, source-closed, sink-full, and the lift/unlift/fork capability gaps. Each is a pre-built singleton with a fixed code, so a consumer matches them by code constant without constructing a probe error. Domain codes must avoid this band; a domain error that reuses a reserved code becomes indistinguishable from the framework singleton under code-keyed catch.
- The bottom error — the value of an expression that could not produce one, from a filtered-away monadic step or an uninitialized struct — is exceptional, not expected. It does not travel the recoverable partition; a catch that recovers expected failures deliberately leaves bottom to escape, because a bottom is a logic defect, not a domain outcome. The constructed-default of a failure-carrying rail is this bottom, which is why an unmatched `where`/filter in a fallible comprehension surfaces as an exceptional bottom rather than a silent empty.

[CATCH_AS_FAILURE_DISPATCH]:
- The recover capability is one trait method: given the structure, a predicate over the failure value, and a handler over the failure value, it runs the handler only when the structure is in a failed state and the predicate accepts the failure. Catch is dispatch on the failure value, symmetric to the way forward composition dispatches on the success value — the predicate is the failure-side match arm and the handler is the failure-side continuation. There is no second mechanism; every catch spelling reduces to a predicate-plus-handler pair.
- That pair is reified as a small immutable value — a `(match predicate, handler)` record — built by the catch combinators and applied by an operator so the recovery reads as an infix alternative beside the protected effect. Building the recovery as a *value* means a library of named recovery policies are ordinary data: a catch policy can be stored, passed, parameterized, and selected from a policy table, and a chain of them composes by left-to-right operator application where the first matching predicate wins and the rest are skipped. The recovery surface is therefore a policy value, not control flow embedded in the call site.

```csharp
// recovery policies are values; the alternative operator selects the first match
static readonly CatchM<Error, M, A> retryable =
    @catch(TransientCode, e => recompute(e))
  | expected(ex => fallback(ex))            // every Expected leaf, fold-free
  | @catch(_ => true, e => surfaceFatal(e)); // catch-all tail
var recovered = protectedEffect | retryable; // infix alternative
```

[FOLD_VERSUS_FAILFAST_RECOVERY]:
- The subtype-keyed catch comes in two algebras that differ only in how they treat a *batch* of matching failures. The monadic-bind variant walks the matching children and binds the handler across them so the first handler that itself fails short-circuits the rest — fail-fast recovery. The monoid-combine variant requires the structure to be an alternative monoid and folds the handler across the matching children combining the results — accumulating recovery that visits every matching failure and merges all handler outcomes. Selecting between them is selecting the recovery's own failure semantics, and it is encoded purely by which combinator is reached for, never by a branch inside the handler.
- The single-failure path and the batch path unify under one predicate: when the failure is already the exact subtype, the handler runs once on it directly; when the failure is an aggregate, the handler runs over the filtered matching subset via the chosen fold-or-bind algebra. A handler author never branches on "is this one error or many" — the combinator chooses the multiplicity strategy and the predicate chooses membership.
- The expected-only and exceptional-only catches are the subtype catch specialized to the two partition leaves, and each has a fold-variant and a bind-variant. Catching expected with the fold variant accumulates recovery across every expected failure in a batch; catching exceptional with the bind variant recovers the first lifted exception and stops. Pairing the partition with the algebra is the full recovery matrix: which failures (expected, exceptional, subtype, code, identity, all) crossed with which multiplicity (recover-first, recover-all).

[RUNTIME_CATCH_BRIDGE]:
- Inside the effect runtime, catch is a stack discipline, not a wrapper closure. The protected operation is pushed with its predicate, handler, an optional finalizer, and the forward continuation; on normal completion a pop frame removes the handler before the continuation runs so the handler cannot accidentally catch a failure that arises *after* the protected region. The handler frame's exception path runs the canonicalizing lift on any escaped exception, tests the predicate against the resulting error, and either re-fails with that error or runs the handler and threads its result into the saved continuation. The recovered branch re-enters the same forward pipeline, so recovery is not a dead end — the handler's success continues the original computation.
- Because the handler frame lifts through the same canonicalization, an exception thrown by a raw side-effecting operation inside the protected region reaches the predicate already normalized: a cancellation thrown deep in foreign code is matched by the cancelled code at the catch site, an aggregate of failures arrives pre-flattened, and a framework wrapper arrives unwrapped to its original error. The catch predicate never sees a raw exception — it always sees the canonical error — so predicates are written against codes and partitions, never against exception types, and remain stable as the underlying throw site changes.
