# Collections and Traversal — Deepening

[SEQ_INTERNAL_FOURTH_TYPE]:
- `Seq<A>` has four internal representations, not three: `SeqStrict<A>`, `SeqLazy<A>`, `SeqEmptyInternal<A>`, and `SeqConcat<A>`. `SeqConcat<A>` is constructed when two sequences of mixed types are concatenated via `Concat` or `+`; it holds a `Seq<ISeqInternal<A>>` of segments and defers merging until an operation forces it. `SeqConcat.Count` sums the counts of all segments in O(segments) time — O(1) per strict segment — but `AsSpan()` and any operation requiring a contiguous array calls `Strict()` which allocates a new backing array and copies all segments.
- Iterating a `SeqConcat<A>` enumerates each segment's enumerator in sequence without allocation. Calling `Fold` or `TraverseM` over a `SeqConcat<A>` is therefore cheap; calling `AsSpan()`, `Strict()`, or the `Count` property if segments include `SeqLazy<A>` sub-segments triggers full materialization. Repeated `+` concatenation without a terminal `Strict()` builds a segment tree that costs O(n) to enumerate each time instead of the amortized O(1) of a single `SeqStrict<A>`.

[LAZY_EVALUATION_TRAPS]:
- `SeqLazy<A>.Count` calls `InternalStrict()` which iterates the entire lazy source into the internal cache array before returning the count. Any code path that reads `.Count` on a `Seq<A>` backed by `SeqLazy<A>` forces full evaluation as a side effect. This includes `Foldable<Seq>.Count` which dispatches to `ta.As().Count` — the HKT form is not exempt. `Foldable.length` at the trait layer also calls `Count` and pays the same cost.
- `SeqLazy<A>.AsSpan()` calls `Strict()` which also calls `InternalStrict()`. There is no zero-copy span path for a lazy `Seq<A>`; any span access forces full evaluation. `SeqStrict<A>.AsSpan()` returns `new ReadOnlySpan<A>(data, start, count)` — zero-copy. `Arr<A>.AsSpan()` returns a zero-copy slice of its backing array. Avoid `AsSpan()` on sequences of unknown internal type inside hot paths.
- `Seq<A>.Add(A)` and `Seq<A>.Concat(Seq<A>)` both carry XML remarks stating they force evaluation of any lazy internal so the item can be appended. `Seq<A>.Cons(A)` (prepend) does not force evaluation; it is O(1) amortized under the capacity check. When building a result sequence incrementally from a lazy source, prepend and reverse outperforms repeated `Add`.

[COLLECTION_CONSTRUCTION_SPANS]:
- `Seq.createRange(ReadOnlySpan<A>)` constructs via `new Seq<A>(span)` which calls `span.ToArray()` internally, allocating a copy. The result is a `SeqStrict<A>`. It is not zero-copy at construction, but the resulting `Seq<A>` supports zero-copy `AsSpan()` thereafter.
- `Arr<A>.AsSpan()` and its `AsSpan(int start)` and `AsSpan(int start, int count)` overloads are zero-copy slices into the backing array. These are the correct primitives when passing a slice of an `Arr<A>` to a span-consuming API without copying; the remarks note that elements outside the splice remain allocated.
- `createRange` overloads exist for every collection type from `ReadOnlySpan<A>`: `Seq`, `Arr`, `Lst`, `Que`, `Stck`, `HashMap`, `HashSet`, `Map`, `Set`, and their `EqK`/`OrdK` variants. The span constructor is always the lowest-allocation creation path when the data is already in a span or an array.
- `toSeq(IEnumerable<A>)` constructs a `SeqLazy<A>` that evaluates elements into a cache array on first access per element. Wrapping an `IAsyncEnumerable<A>` into `Iterable<A>` via `createRange` is the async parallel; `.IsAsync` is true and operations on it route to async overloads.

[HASHMAP_OPERATOR_BIAS_CORRECTION]:
- `HashMap<K, V>` `operator+` delegates to `Combine` which calls `TrieMap.Append` which calls `TryAddRange`. `TryAddRange` uses `TryAdd` which is `UpdateType.TryAdd` — it returns the existing node unchanged if the key exists. **`operator+` is left-biased**: existing left-hand keys are preserved on collision.
- `HashMap<K, V>.Union(IEnumerable<(K, V)>)` with no `WhenMatched` argument also delegates to `TryAddRange` — also left-biased. Only `HashMap<K, V>.Union(IEnumerable<(K, V)>, WhenMatched<K, V, V, V>)` provides controlled merge semantics. Passing `(_, l, r) => r` explicitly achieves right-bias. The wave 1 claim of a "right-bias default" is incorrect for both `+` and the no-delegate `Union` overload.
- `Map<K, V>` `operator+` has identical left-biased semantics: the AVL-tree append uses `TryAdd` equivalents. The `+` operator on both collection types is additive identity (monoid) for non-overlapping keys and silently retains left values for overlapping keys.

[EQ_RESOLUTION_AND_HASHMAP_KEY_EQUALITY]:
- `HashMap<K, V>` (two-type-parameter form) internally uses `TrieMap<EqDefault<K>, K, V>`. `EqDefault<K>` first tries to resolve an `Eq<K>` trait implementation at static-type initialization via `EqResolve<K>`, using reflection (`Resolver.Find`, `MethodInfo.Invoke`). If no `Eq<K>` instance is found, it falls back to `EqualityComparer<K>.Default`. The reflection-based path boxes arguments and is measurably slower than the compiled delegate path.
- `HashMap<EqK, K, V>` (three-type-parameter form) with an explicit `EqK : Eq<K>` constraint routes directly through the static interface method `EqK.Equals` without reflection. For high-frequency insertions or lookups with custom key types, the explicit `EqK` form is the correct choice. The two-parameter form is a convenience that pays a reflection cost per comparison until the JIT can devirtualize.
- The same `EqResolve` / `HashableResolve` pattern governs `HashSet<A>` vs `HashSet<EqA, A>`. The three-type form is the performance-safe choice for all set and map types with non-trivial key types.

[PARTITION_SEMANTICS]:
- `Seq<A>.Partition(Func<A, bool>)` is a single-pass, mutable-accumulator implementation that iterates via `GetEnumerator` and appends to two `Seq<A>` result values using `Add`. It does not use `Choose` or two-pass `Filter`; it is internally imperative but has value-semantics externally.
- `Iterable<A>.PartitionIO(Func<A, bool>)` is async-aware: when `IsAsync` is true it materializes both partitions by iterating the `IAsyncEnumerable<A>` source under a `CancellationToken` from `EnvIO`. The sync path mirrors `Seq.Partition`. The async path is not lazy — it fully evaluates both result iterables before returning.
- `Foldable.Partition<T, A>(K<T, A>, Func<A, bool>)` is the HKT form, returning `(Seq<A> True, Seq<A> False)`. For `Seq<A>` it dispatches to the above single-pass implementation; for other foldables it uses `FoldBack` on the trait default, which may be less efficient.

[PARTITION_FALLIBLE]:
- `PartitionFallible<F, M, A>(K<F, K<M, A>>)` folds a collection of monad values `K<M, A>` where `M : Fallible<M>` into `K<M, (Seq<Error> Fails, Seq<A> Succs)>`. The fold runs each effect, catching failures via `ma.Catch(e => M.Pure((fails.Add(e), succs)))`. This means every effect in the collection is executed sequentially; the monad is not short-circuited. Failures are accumulated alongside successes, unlike `TraverseM` which aborts on first failure.
- The typed-error variant `PartitionFallible<E, F, M, A>` where `M : Fallible<E, M>` accumulates `Seq<E>` instead of `Seq<Error>`. Both variants exist for `Seq<K<M,A>>`, `Iterable<K<M,A>>`, `Lst<K<M,A>>`, `HashSet<K<M,A>>`, `Set<K<M,A>>`, and `IEnumerable<K<M,A>>`.

[UNFOLD_AND_GENERATION]:
- `unfold<S>(S, Func<S, Option<S>>)` yields `state` first (before calling the unfolder), then transitions. It always emits the seed — there is no "unfold that can skip the first element" overload. For sequences that must suppress the initial state, wrap in `Skip(1)` or use `unfold<S, A>(S, Func<S, Option<(A, S)>>)` where the unfolder controls emission.
- `unfold<S, A>(S, Func<S, Option<(A, S)>>)` emits the `A` value from the tuple and advances state; the initial state is never emitted. All `unfold` overloads return `IEnumerable<A>` — they are `yield return` generators and are fully lazy. Wrapping the result in `toSeq()` produces `SeqLazy<A>` which evaluates on demand.
- Multi-state overloads exist up to four state components: `unfold<S1,S2,A>`, `unfold<S1,S2,S3,A>`, `unfold<S1,S2,S3,S4,A>`. These avoid tuple construction at call sites when the state is naturally multi-dimensional (e.g., numerator + denominator in a continued-fraction expansion).
- `Seq.generate<A>(int, Func<int, A>)` is the index-driven alternative to `unfold` for finite sequences of known length. It produces a strict `Seq<A>` immediately — no laziness, no `IEnumerable` wrapping. Use it when the length is known and the generation function is cheap and pure; `unfold` when length is determined by the stopping condition.

[INITS_TAILS_COST]:
- `Seq<A>.NonEmptyInits` is O(n²) in the number of elements: the implementation iterates `i` from `1` to `Count`, calling `Take(i)` each time and appending to a result `Seq`. Each `Take(i)` on a `SeqStrict<A>` creates a new struct wrapping the same backing array with an adjusted `count` field — O(1) per `Take` — but `result.Add(...)` forces evaluation of any lazy prefix. The full quadratic cost arises from building the enclosing `Seq<Seq<A>>` via `Add` in a loop.
- `Seq.tails<A>(Seq<A>)` from the static `Seq` module produces all final segments from longest to shortest. Each element is a `Tail` slice — O(1) per tail on `SeqStrict<A>` due to the `start` offset. The resulting `Seq<Seq<A>>` thus costs O(n) total to construct. `NonEmptyTails` on the instance has the same implementation. `Inits` and `tails` are structurally asymmetric in cost.

[NONEMPTY_ADMISSION_PATTERN]:
- `AsIterableNE<A>(this IEnumerable<A>)` returns `Option<IterableNE<A>>` — the `None` case represents an empty source without throwing. `AsIterableNE<A>(this IAsyncEnumerable<A>)` returns `IO<IterableNE<A>>` which defers the emptiness check to effect execution.
- `AsIterableNE<A>(this Seq<A>)` is implemented as `xs.Head.Map(h => new IterableNE<A>(h, xs.Tail.AsIterable()))` — a single `Option.Map` with no allocation beyond the record. This makes `Seq<A> → Option<IterableNE<A>>` the allocation-minimal non-empty admission path for sequences.
- `IterableNE<A>` is a `record(A Head, Iterable<A> Tail)` and is the required input type for `Applicative.Actions`. `actions<F, A>(IterableNE<K<F, A>>)` folds left-to-right over the tail using `h.Action(t)` — semantics are left-to-right action sequencing, discarding all but the final value. `actions` over an empty collection is not expressible because `IterableNE` guarantees at least one element, eliminating the empty-applicative-sequence problem at the type level.

[ARR_TRAVERSE_PREALLOCATION]:
- `Arr<A>.Traverse<F, B>` and `Arr<A>.TraverseM<F, B>` both pre-allocate a `SeqStrict<B>` of exactly `Count` elements before folding: `new SeqStrict<B>(new B[ta.As().Count], 0, 0, 0, 0)`. The fold then calls `SeqStrict<B>.Add(b)` which writes into the pre-allocated backing array. The final `Arr<B>` slices the populated region via `bs.data.AsSpan().Slice(bs.start, bs.Count)`.
- `Seq<A>.Traverse<F, B>` and `Seq<A>.TraverseM<F, B>` do not pre-allocate; they start with `Seq<B>.Empty` and call `bs.Add(b)` per element. For short-circuit monads (`Fin`, `Option`, `Eff`) where most elements are expected to succeed, `Arr<A>.TraverseM` is lower-allocation than `Seq<A>.TraverseM` when the input length is known. The `Traverse` (applicative) path on both types pays the same fold structure; the pre-allocation advantage of `Arr` is most visible on `TraverseM`.

[TRACKING_HASHMAP_SNAPSHOT_SEMANTICS]:
- `TrackingHashMap<K, V>.Snapshot()` returns a new `TrackingHashMap<K, V>` initialized with the current value map and an **empty** change map (`TrieMap.Empty`). It does not consume or process the accumulated changes — it simply resets the change ledger. The `Changes` property returns a `HashMap<K, Change<V>>` reflecting the net morphism since the last `Snapshot()` call; each key appears at most once in the change map regardless of how many times it was mutated.
- `Change<V>` is one of `EntryAdded<V>`, `EntryMapped<V>` (with `From` and `To`), or `EntryRemoved<V>`. The change map never records intermediate states — only the net transition. This is the correct primitive for incremental UI refresh, delta sync, or change-feed emission where intermediate states are irrelevant.
- `AtomHashMap<K, V>.ToTrackingHashMap()` snapshots the current atomic value into a new `TrackingHashMap<K, V>`. Subsequent mutations of the `AtomHashMap` are not reflected in the snapshot. The pattern for atomic-plus-change-tracking is `ToTrackingHashMap()` after each logical batch, not continuous subscription.

[MAP_FINDRANGE_LAZINESS]:
- `Map<OrdK, K, V>.FindRange(K, K)` delegates to `MapModule.FindRange<OrdK, K, V>` which is a `yield return` generator that walks the AVL tree in-order and emits values only when the key is within range. The result is wrapped in `AsIterable()` before being returned as `Iterable<V>`. The range walk is lazy: elements are yielded on demand during enumeration. Calling `FindRange` itself is O(log n) to position at the lower bound; subsequent iteration is O(k) for k results. There is no eager materialization.
- `Map<K, V>.FindRange(K, K)` has the same implementation via `OrdDefault<K>`. For ordered-key range queries, `Map` is the correct type and `FindRange` is the correct combinator — no `Where` filter over a full `ToSeq()` is needed.

[ZIP_BEHAVIOR_AND_LIMITS]:
- `Seq<T>.Zip<U, V>(Seq<U>, Func<T, U, V>)` delegates to `Enumerable.Zip` which stops at the shorter sequence. There is no `ZipAll`, `ZipLong`, or padding overload in the `Seq` API. When zip-to-longest behavior is needed, align the sequences manually with `Pad` before zipping, or use `Iterable<A>.Zip<B, C>` after confirming equal length via domain invariants.
- `Iterator<A>.Zip(Iterator<A>)` only zips same-typed iterators. The return type is `Iterator<(A First, A Second)>` — no projection variant is available at the `Iterator` level. This is more restrictive than `Seq` or `Iterable` where `Zip<B, C>(Iterable<B>, Func<A, B, C>)` is available.

[TO_HASHMAP_COMPOSITE_KEYS]:
- `ToHashMap` on `IEnumerable<(K1, K2, V)>` produces `HashMap<(K1, K2), V>` using a tuple key. Overloads exist for 3- and 4-component composite keys. These use `ValueTuple` structural equality and `GetHashCode()`, which relies on `EqDefault<(K1,K2)>` — falling back to `EqualityComparer<ValueTuple<K1,K2>>.Default` if no explicit `Eq<(K1,K2)>` is registered. For custom key semantics over compound keys, use `HashMap<EqK, (K1, K2), V>` with an explicit `EqK` and construct via the `EqK`-scoped `ToHashMap<EqK, K1, K2, V>` overload.
