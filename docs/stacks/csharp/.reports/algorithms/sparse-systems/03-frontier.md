# Sparse Systems: Frontier

[SPARSEMATRIX_SPMV_VECTORIZATION_GAP]:
- `SparseMatrix.DoMultiply(Vector<double> rightSide, Vector<double> result)` is a scalar CSR row-loop: for each row `i`, it accumulates `values[j] * rightSide[columnIndices[j]]` into a local `double` via a scalar loop. The `rightSide[columnIndices[j]]` access is a gather with irregular indices, which blocks vectorized gather-then-multiply. `TensorPrimitives` provides no indexed-gather API on .NET 10. The only SIMD path for the sparse mat-vec product in hot loops is to hand-vectorize using `Vector256.GatherMaskVector256` (AVX2) or `Vector128.GatherMaskVector128` via `System.Runtime.Intrinsics` — no higher-level admitted package reaches this pattern.
- `SparseMatrix.DoTransposeThisAndMultiply(Vector<double> rightSide, Vector<double> result)` is a scatter-accumulate kernel: for each nonzero `(i, j, v)`, `result[j] += v * rightSide[i]`. The scatter-write pattern has write-after-write hazards across rows and cannot be vectorized without explicit atomics or row-partitioning. The forward SpMV (row-loop) is vectorizable via gather; the transpose SpMV (column-scatter) is not — this asymmetry governs solver inner-loop orientation selection for a given sparsity pattern.

[COMPOSITE_SOLVER_EXCEPTION_SWALLOWING]:
- `CompositeSolver.Solve` wraps each sub-solver call in `try { ... } catch (Exception) { input.CopyTo(vector); continue; }`. Any exception — including `NumericalBreakdownException`, `OutOfMemoryException`, or preconditioner initialization failures — is silently caught and the solver advances. The result after exhausting all solvers carries the last `StoppedWithoutConvergence` partial iterate with no indication of which solvers threw versus which ran to completion. The composite pattern is not a typed-error surface; it is a best-effort loop with silent exception burial. The calling site's true-residual recomputation is the only discriminant between a buried-exception partial result and a legitimate non-convergence partial result.
- `CompositeSolver`'s constructor resolves each preconditioner at construction via `setup.CreatePreconditioner() ?? new UnitPreconditioner<double>()`, making `solver.Item2` always non-null. The external `preconditioner` argument passed to `Solve` is therefore dead — the expression `solver.Item2 ?? preconditioner` never reaches the right-hand side. Each `IIterativeSolverSetup<T>` must return a configured preconditioner from `CreatePreconditioner()` or accept `UnitPreconditioner` as the default; passing a preconditioner to the `Solve` call has no effect.

[MILU0_RESIDUAL_CORRECTION_LIMIT]:
- `MILU0Preconditioner` with `UseModified = true` accumulates the dropped fill into the diagonal so the preconditioner matrix preserves row sums of the original matrix. This is beneficial for M-matrices (negative off-diagonal, non-negative diagonal, positive real eigenvalues) and diagonally dominant systems. For problems where the matrix is not an M-matrix — non-symmetric convection-dominated PDEs, for example — the MILU(0) correction can make the preconditioner worse than standard ILU(0) because the diagonal correction pushes the approximation away from the true L and U factors in a non-locally-bounded way.
- The `NumericalBreakdownException` thrown by `MILU0Preconditioner.Initialize` carries the message `"Zero pivot encountered on row " + num + " during ILU process"`. The row index corresponds to the zero-pivot row in the CSR iteration order of the input matrix, not a permuted index. Pre-ordering with AMD before calling `MILU0Preconditioner` reduces the probability of zero pivots, but the preconditioner does not accept or apply a permutation. The caller must symmetrically permute the matrix before calling `Initialize` — using `Matrix<T>.PermuteRows` and `PermuteColumns` with a permutation derived from `AMD.Generate` — before passing the permuted matrix to the preconditioner.

[DIVERGENCE_CRITERION_HISTORY_WINDOW_MECHANICS]:
- `DivergenceStopCriterion<T>` allocates `_residualHistory` of length `_minimumNumberOfIterations + 1` on first use (not at construction). The `IsDiverging` check tests all consecutive pairs in the window: if any pair has `history[i] <= history[i-1]` or `history[i] < history[i-1] * (1 + maxRelIncrease)`, the test short-circuits and returns false. For the default window of 10, all 10 consecutive pairs must show monotone increase above the 8% threshold — an AND over the entire window. A residual sequence that oscillates even once in a 10-iteration window cannot trigger divergence regardless of overall growth.
- The constructor default of 10 iterations with `MaximumRelativeIncrease = 0.08` was designed for well-behaved conjugate-gradient-style residuals. For Krylov methods on non-symmetric systems (BiCGStab, TFQMR, GpBiCG), where residual oscillation is normal for the first 5–20 iterations, reducing the window to 3 (the minimum enforced by the setter) produces spurious `Diverged` signals during the oscillatory warm-up phase.

[TFQMR_NEAR_ZERO_CANCEL_VS_BICGSTAB_THROW]:
- `TFQMR.Solve` handles near-zero inner products by calling `iterator.Cancel()` and breaking out of the loop, leaving `IterationStatus.Cancelled` in the iterator. `BiCgStab.Solve` throws `NumericalBreakdownException` on the same condition. The caller of `TrySolveIterative` encounters two incompatible failure modes for the same mathematical event — a status return versus an unhandled exception — with no syntactic difference at the call site. TFQMR's `Cancelled` result leaves whatever partial iterate is in `result` at cancellation time; a true-residual recomputation is required to determine whether that partial result is usable.
- `GpBiCg.Solve` handles near-zero inner products by substituting `1.0` in place of the near-zero value and continuing. This is a damage-control heuristic, not breakdown recovery. The continued iteration after substitution may yield a valid result for mildly ill-conditioned problems or an arbitrary vector for severely ill-conditioned ones. GpBiCg never throws `NumericalBreakdownException` and never calls `iterator.Cancel()` — the iteration continues through all substituted-zero events until a stop criterion fires. A post-solve residual check is mandatory; the solver's own convergence status cannot be trusted when substitutions have occurred.

[MLKBICGSTAB_STARTING_VECTOR_DEPENDENCY]:
- `MlkBiCgStab.Solve` checks `_startingVectors` at the start: if non-null and of correct count and size, it uses them; otherwise it calls `CreateStartingVectors(DefaultNumberOfStartingVectors = 50, n)`, which generates up to 49 random unit vectors via `DistributionVector.CreateRandom`. For a system of size n = 10,000, this allocates 50 × 10,000 × 8 bytes = 4 MB at solver-initialization time before the first iteration. Setting `NumberOfStartingVectors` to 3–5 eliminates this startup cost and is appropriate when a warm start from a prior time step reduces the benefit of multiple starting vectors.
- `MlkBiCgStab` throws `NumericalBreakdownException` at three distinct points corresponding to different phases of the multi-Lanczos recurrence. All three throw the same message. Distinguishing which phase broke down is not achievable from outside the `Solve` method.

[NORMALIZEZEROS_ABSENT_IN_OF_COMPRESSED_COLUMN_FORMAT]:
- `SparseCompressedRowMatrixStorage<T>.OfCompressedSparseColumnFormat` calls `NormalizeOrdering()` and `NormalizeDuplicates()` but does NOT call `NormalizeZeros()`. Explicit zeros in the input column-format arrays survive as explicit zeros in the resulting CSR storage. For the path from CSparse CSC to MathNet CSR for residual arithmetic, explicit zeros inflate `ValueCount` and slow preconditioner initialization; call `NormalizeZeros()` explicitly after `OfCompressedSparseColumnFormat`. For the path into `PopulateExplicitZerosOnDiagonal` before MKL DSS, diagonal explicit zeros are required and must not be removed.
- `SparseCompressedRowMatrixStorage<T>.Normalize()` calls `NormalizeOrdering()` then `NormalizeZeros()` — it does not call `NormalizeDuplicates()`. Using `Normalize()` on a matrix with structural duplicates from assembly or accumulation leaves those duplicates intact; values are not summed. The complete normalization sequence after building from coordinate format is `NormalizeOrdering()`, `NormalizeDuplicates()`, then `NormalizeZeros()` — exactly what `OfCompressedSparseColumnFormat` does minus the zero step.

[ITERATOR_CANCEL_SEMANTICS_STICKY]:
- `Iterator<T>.DetermineStatus` guards at the top: `if (_status == IterationStatus.Cancelled) return _status`. Once cancelled, the iterator permanently returns `Cancelled` on all subsequent `DetermineStatus` calls, regardless of residual content. `Reset()` clears `_status` back to `Continue` and resets all stop criteria. `CompositeSolver` calls `iterator.Reset()` between solvers; a cancelled iterator from TFQMR inside `CompositeSolver` is therefore reset before the next solver, and the `Cancelled` status from TFQMR's near-zero handling is discarded — the composite loop continues as if the solver returned normally with its partial iterate in `result`.
- `CancellationStopCriterion<T>` links a `CancellationToken` into the stop-criteria stack. `Iterator<T>.Reset()` creates a new `CancellationTokenSource` linked to the original `_masterToken`. If `_masterToken` is already cancelled at the time of `Reset()`, the new TCS is immediately cancelled and the criterion returns `Cancelled` on the first `DetermineStatus` call after reset. A composite solver using an already-cancelled token will never make progress through its solver list — each solver cancels on the first iteration.

[DSS_PROVIDER_UNIMPLEMENTED_FALLBACK]:
- `ManagedSparseSolverProvider.Solve` (all four overloads) unconditionally throws `NotImplementedException`. There is no managed fallback implementation. The `ISparseSolverProvider<T>.Solve` interface — accepting raw CSR arrays (`rowPointers`, `columnIndices`, `values`), RHS, and solution arrays — is the bridge to external providers (MKL). Without an MKL-backed provider registered via `SparseSolverControl`, calling `Control.SparseSolverProvider.Solve(...)` throws immediately.
- The `DssMatrixStructure` enum values are not sequential: `Symmetric = 536870976`, `SymmetricStructure = 536871040`, `Nonsymmetric = 536871104`, `SymmetricComplex = 536871168`, `SymmetricStructureComplex = 536871232`, `NonsymmetricComplex = 536871296` — these are MKL DSS option literals mapping to specific bit patterns. Passing the wrong `DssMatrixStructure` does not throw — MKL DSS returns `DssStatus.MKL_DSS_STRUCTURE_ERR = -14`. Code calling `ISparseSolverProvider<T>.Solve` must map every non-`MKL_DSS_SUCCESS` status to a typed domain failure before returning to the caller, because MKL DSS failure codes are not typed exceptions.

[SPARSEMATRIX_TIMES_SPARSEMATRIX_TWO_PASS_ALLOCATION]:
- `SparseMatrix.DoMultiplySparse` uses a two-pass algorithm: the first pass counts nonzeros in each result row using a marker array `int[other.ColumnCount]` initialized to `-1`, then allocates exact-fit `int[]` and `double[]` for the result; the second pass fills the values. This per-multiply O(n) work array allocation is not amortized across repeated multiplications — for Krylov subspace methods where the preconditioner involves a sparse product, this allocation recurs on every application. The result arrays are assigned directly to the output `SparseMatrix`'s storage, bypassing `AutoTrimStorage` — they are exact-fit from allocation.

[GENERIC_MATH_CONSTRAINT_ON_NUMERIC_RAIL]:
- `TrySolveIterative` is defined on `Matrix<T>` where `T : struct, IEquatable<T>, IFormattable`. It does not constrain T to `IFloatingPoint<T>` or `INumber<T>`. `ResidualStopCriterion<T>.DetermineStatus` calls `residualVector.InfinityNorm()` and compares the result as `double` via a hard-coded `double.IsNaN(num2) || double.IsNaN(num)` guard — the norm reduces to a real `double` scalar regardless of T. This means criterion implementations cannot be made generic over floating-point precision using C# 14 generic math constraints; the `double`-typed norm comparison is a fixed implementation detail, not a parameterizable policy surface.
