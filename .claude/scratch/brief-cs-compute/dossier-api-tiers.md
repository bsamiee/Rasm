# DOSSIER — Rasm.Compute API TIERS (both tiers, manifests)

Lane: api-tiers. Scope read: `libs/csharp/.api/*.md` (30 substrate catalogs, 4574 LOC), `libs/csharp/Rasm.Compute/.api/*.md` (37 domain catalogs, 6539 LOC), `Directory.Packages.props`, `Rasm.Compute.csproj`, `packages.lock.json`, plus targeted consumption census across the 41 `.planning` pages and full deep-reads of the load-bearing catalogs.

## [00]-[HEADLINE]

The Compute domain catalogs are, on the CRAFT axis, **the strongest tier in the corpus**: decompile-verified against consumer-bound TFMs, integration-shaped (PACKAGE_SURFACE → PUBLIC_TYPES → ENTRYPOINTS → IMPLEMENTATION_LAW/RAIL_LAW), rail-law-bearing with explicit REJECTED surfaces, honest cross-catalog deferral (torchsharp defers all native RID/ABI facts to libtorch-cpu; genai/gpu/directml defer to onnxruntime), and truthful about transitive/Bim-owned/test-only status. `onnxruntime` (518 LOC, 127 planning occ), `torchsharp`+`libtorch-cpu` (the `[CLASSICAL_ML_BLAS]` managed+native pair), `mathnet-symbolics` (F# member-mangling split + FSharp.Core interop seam), and `extensions-ai` (contracts-only-by-design) are exemplary and should be the pattern every other tier in `libs/csharp` is dragged up to.

The defects are therefore NOT catalog-quality — they are **tiering, roster-vs-catalog contradictions, and a small set of admitted-but-unrealized packages**. The corpus is naive on the APPROACH axis in exactly one place (ONNX GPU providers admitted as packages without ExecutionProvider rows) and on COVERAGE in exactly one place (LanguageExt.Core, the ROP rail, entirely uncatalogued). Everything else is a placement/duplication cleanup.

## [05]-[CROSS_CUTTING]

- **Duplication:** 9 md5-identical cross-tier copies (~1,971 LOC); 2 drifted overlays (tensors, unitsnet). See [01]. This is the single largest mechanical debt in the tier.
- **Concern mixing:** none material inside catalogs (the rail-law discipline prevents it). The one boundary blur is the TestHost test-package in the domain tier ([02]).
- **Hardcoding-vs-generator:** not a catalog concern in this tier (catalogs are evidence, not generated code). The one APPROACH-naive spot is upstream in the roster: DirectML/Gpu packages admitted as enumerated instances without the ExecutionProvider generator-row that would consume them ([02]-C) — providers.md:15 states "growth is one ExecutionProvider row" yet ships the packages without the rows.
- **Dead carriers / unused capability the target never demands:** (1) DirectML — 0 planning occ, 0 osx-arm64 payload; (2) Gpu (CUDA/TensorRT) — 5 occ but all in growth-seam notes, 0 osx-arm64 payload; (3) SharpGLTF.Runtime full decode+animation+skinning surface — catalog-declared Compute-deleted-form; (4) TorchSharp/ONNX training surfaces — correctly REJECTED in rail-law (not dead, properly fenced). Platform reality (NREL.OpenStudio.macOS-arm64, libtorch-cpu-osx-arm64) confirms osx-arm64 primary target, making DirectML/Gpu premature.
- **Unwired seams (catalog-declared, roster/page-unrealized):** DirectML/Cuda ExecutionProvider rows (declared in onnxruntime.md + providers.md growth note, no row); SharpGLTF.Ext.3DTiles (admitted, uncatalogued, TILE_PARTITION consumer exists in codecs.md but no catalog seam).
- **Unmined capability with catalog anchors:** minimal — the tier is honestly scoped. The extensions-ai IEmbeddingGenerator embedding-production stacking (catalog:251) vs embedding.md's hand-rolled vocabulary is the one page-side wiring to verify (soft, likely a legitimate retrieval-vs-generation split).
- **Folder-architecture verdict:** the two-tier split (shared substrate `libs/csharp/.api/` + per-package domain `Rasm.Compute/.api/`) is the RIGHT structure and matches CLAUDE.md [04]. It is undermined ONLY by (a) the 9 verbatim duplicates that violate the overlay principle, (b) the MathNet core/satellite tier split, and (c) LanguageExt uncatalogued at the substrate root. Fix those three and the folder architecture is clean. No flat-sprawl or one-file-one-folder rot; the domain tier is a healthy flat set of ~37 focused catalogs.

Either (a) it IS Compute's — document `SharpGLTF.Schema2.Tiles3D` in the Compute tier and correct the catalog's Bim-owned claim, or (b) it is Bim's — drop the direct admission and route tileset emission through a Bim seam. 4. **Drop (or realize) the ONNX GPU-provider admissions (DirectML + Gpu) for the osx-arm64 target.** onnxruntime.md:122-131 proves neither ships an osx-arm64 payload (DirectML=win-only 1.24.4 HOLD, Gpu=CUDA win/linux); providers.md realizes only Cpu+CoreMl rows (:12), naming Cuda/DirectMl only in a growth note (:173); DirectML planning occ=0. Accept as the single sanctioned production-seam-shaped-for-test overlay OR relocate to a test-substrate location — but rule explicitly so test packages do not proliferate into domain `.api/` tiers.
