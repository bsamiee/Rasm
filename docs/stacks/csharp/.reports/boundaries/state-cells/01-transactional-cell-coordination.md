# Transactional Cell Coordination

[MVCC_VERSION_ALGEBRA]:
- Every shared cell that participates in multi-cell coordination carries a monotonic `Version` alongside its value; conflict detection compares versions, never value equality. A transaction snapshots `(value, version)` per touched cell at entry, and commit succeeds only when the live version still equals the snapshot version. Two transactions that write structurally identical values to the same cell still conflict — sameness of payload is irrelevant; sameness of version is the sole admission predicate.
- The engine is snapshot-plus-one-CAS, not a lock: a transaction opens by capturing the global cell map into a thread-flow-local transaction object, all in-transaction reads and writes hit the snapshot's copy while recording touched ids into read and write sets, and commit re-acquires the live map under a single swap that validates and applies. The persistent-trie backing makes the snapshot capture free and the structural-sharing writes cheap — which is the precondition the engine silently assumes its payloads honor; a cell payload that defeats structural sharing converts every snapshot into a deep-copy tax.
- Commit's validate-plus-apply runs inside the global map's own swap closure, so validation and application are themselves atomic against other committers — a second transaction cannot interleave between the read-set check and the write application. Version numbers are per-cell and monotonic; the increment is a commit-time event, so a transaction reading its own just-written cell inside the body sees the in-transaction value at the snapshot version and must not infer "I committed" from any version it can observe in the body.
- The write-commit check is unconditional across isolation levels: a write whose snapshot version no longer matches the committed version aborts the whole transaction regardless of whether reads are tracked. Snapshot isolation does not relax this — it relaxes only the read-set check. The practical consequence: any cell a boundary lifecycle transaction *writes* is fully serialized against concurrent writers to that cell, and only cells it merely *reads* are exposed to the snapshot/serializable distinction.
- Serializable isolation additionally validates the read set: every cell read but not written must still hold its snapshot version at commit, or the transaction retries. This is the only correct mode for a permit decision whose invariant is "the set of open handles I observed did not change while I decided to add one" — under snapshot isolation a concurrent open commits between the read and the decision, the read is torn, and the count the decision was based on is stale at commit with no abort.

[ISOLATION_ENTRYPOINT_TRAP]:
- The convenience entrypoint whose name reads as the serializable level dispatches the snapshot level to the engine — only the explicitly-parameterized atomic entrypoint with the serializable isolation argument performs read-set validation. A seam that needs a read-only cell protected from concurrent mutation mid-transaction must spell the isolation argument explicitly; relying on the serially-named helper silently gets write-set-only validation, and the read-protection it appears to promise never runs. The dedicated snapshot-named helper and the serially-named helper are behaviorally identical, which makes the divergence invisible to review unless the isolation argument is explicit at every permit-decision site.

[NOTIFICATION_FIRING_ASYMMETRY]:
- A transactional cell's change event fires once per *attempt*, in the attempt's teardown, from the attempt's local change-set — including attempts that then abort on a version conflict and silently retry. A subscriber to a transactional cell therefore observes change callbacks for values that were never committed: the event is a notification of *intent*, not of *commitment*. Any handler driving an external effect (rescheduling a timer, emitting a metric, marshaling onto a host thread) from a transactional cell's change event executes once per retry and once more for the winning attempt; the effect is neither once-only nor commit-gated.
- A single-cell CAS cell fires its change event exactly once, after the successful compare-exchange, outside the spin loop — no firing on losing iterations. This is the inverted guarantee: the single-cell primitive is the correct seam for "notify the host exactly once on a settled transition," and the transactional primitive is the wrong one for that, despite both exposing a same-named change event.
- The rule the asymmetry forces: drive host-visible side effects from the *return value* of the coordination, in a continuation after the coordination settles — never from the cell's change subscription. Read the committed value out of the transaction result, then run the effect once. Reserve change subscriptions on transactional cells for idempotent, retry-tolerant observers only.

[VALIDATOR_FAILURE_SURFACES]:
- A single-cell CAS cell's validator runs after the transform inside the spin; rejection returns the prior value with no exception and no event — rejection is observable only by inspecting the returned snapshot, and the maybe-form makes rejection explicit by returning absence. A boundary cell built on the single-cell primitive must treat "value unchanged" as the rejection signal, not an exception.
- A transactional cell's commit-time validator failure throws a distinct validation exception that is *not* the conflict exception — it escapes the retry loop and propagates to the caller. Conflict aborts retry; validation failure terminates: the world is not re-snapshotted, there is no retry, and the caller receives the throw. A lifecycle invariant encoded as a transactional-cell validator is therefore a hard wall, and the boundary must convert that fault into a typed rail outcome at the seam rather than letting it cross into domain code.
- The two surfaces compose into a deliberate split: relational transition constraints (is this a legal predecessor→successor edge) belong in the swap-body decision and reject by returning the unchanged state or absence; absolute domain invariants (a count never goes negative, a deadline never precedes now) belong in the cell validator and fault on violation. Collapsing both into one mechanism either swallows an invariant breach as a silent no-op or escalates an ordinary illegal-edge into a thrown fault.

[COMMUTE_REBASE_SEMANTICS]:
- A commuting update applies its function to the in-transaction snapshot immediately for the body's benefit, then *re-applies the same function to the latest committed value at commit* — the committed result is the function rebased onto whatever external value won the race, not the tentative in-body value. The transaction's return for a commute is read from the rebased committed state, so the value the caller receives can differ from the value the body computed.
- The rebase is also the concurrency unlock: because the commit-time application reads the live committed value rather than the snapshot, a commute never conflicts on its own cell — two transactions both commuting the same counter both commit without either retrying, where two plain writes would force one to spin. The contract this buys concurrency with is commutativity: the function must be correct regardless of which committed base it sees at the rebase, because the order of concurrent commutes is unspecified. A non-commutative function does not fail loudly — it degrades silently to last-application-wins — so an increment, a set-union, or a max is a correct commute while "set to a value derived from a stale read" is silently wrong.
- A permit decision must never read a commuted cell to decide admission: the in-body value it reads is the un-rebased tentative result, and the value that actually commits is rebased onto a base the decision never saw. Permit logic uses the swap form, whose committed value equals the body value under the version check; commute is strictly for blind accumulation where no decision is conditioned on the cell.
- Commute increments the cell version and runs the cell validator on the rebased result, so an accumulator with a domain bound (a capacity ceiling) still faults at commit when the rebased value violates the bound — the higher concurrency of commute does not buy past an invariant, it only defers the check to the rebased value.

[TRANSACTION_CONTEXT_FLOW]:
- The ambient transaction is carried in an async-local slot, so it flows across continuation boundaries within the same logical async flow rather than being pinned to a physical thread. A transaction body may suspend and resume on a different worker and still be inside its transaction. The guard that reports "no transaction running" is therefore a flow-context probe, not a thread probe, and cannot be used as a cheap conditional-entry test — entering or not must be decided structurally.
- Nested entry participates in the enclosing transaction instead of opening a second one; an inner coordination called from within an outer transaction silently joins it, sharing the snapshot and the abort fate. This makes composition safe — an inner permit check inherits the outer atomicity — but it also means an inner body's writes are not independently durable until the outer commit, and an inner body that performs an external effect re-runs that effect on every outer-level retry.
- Because the whole body re-executes on retry, an effectful body — one that reads or writes outside the coordinated cells, logs, or touches I/O — replays its effects per attempt. Transaction bodies must be confined to coordinated-cell reads and writes; the committed values flow into a single post-commit continuation that runs the effect exactly once.

[CELL_PRIMITIVE_SELECTION]:
- Single-cell CAS is the floor: it coordinates exactly one location, validates after transform, fires once on success, and rejects by returning the unchanged snapshot. It is correct when the entire lifecycle invariant fits in one value and no second cell must move atomically with it.
- Transactional refs are the multi-cell ceiling: state, in-flight count, and drain deadline as three coordinated cells inside one serializable transaction give drain-with-count correctness — the drain transition reads the count and writes the state atomically, and any concurrent increment of the count conflicts with the drain's read and forces one side to retry against the other's committed value. The boolean collapse of these cells into one value reintroduces the torn-read window the multi-cell transaction exists to close.
- Both CAS cells and transactional refs are value-replacement primitives, not critical sections: neither can express "these reads and writes are mutually exclusive as a block." The scoped lock — whose enter returns a stack-only scope — is earned only for that block-exclusion invariant, is illegal across an await point, and an async-spanning gate needs a counting semaphore instead. Reaching for a lock where a CAS cell or a transaction suffices is the over-serialization failure; the lock is the last resort, not the default.
- The cost asymmetry is load-bearing: single-cell CAS allocates nothing on the success path and spins only under contention; the transactional path snapshots the global cell map at entry, re-runs the whole body on every conflict, and fires change notifications per attempt. Reach for the transactional primitive only when two or more cells must transition as a unit; using it for a single cell pays the snapshot-and-replay tax for atomicity a CAS already provides.
- A transactional cell's identity entry lives in a process-global registry until the cell is finalized; the cell object is garbage-collectible but its registry slot is reclaimed only by the finalizer, coupling cell lifetime to finalization latency. A boundary layer that mints transactional cells per operation accumulates registry slots faster than finalization reclaims them under load — coordinated cells are long-lived spine fixtures, not per-request ephemera.
