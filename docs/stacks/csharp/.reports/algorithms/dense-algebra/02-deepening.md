# Dense Algebra — Deepening

[SVD_SOLVE_TRUNCATION_GAP]:
- `SvdSolveFactored` (the managed provider path that `DenseSvd.Solve` routes through) divides each component by `s[j]` with no zero guard and no truncation: `num2 /= s[j]` executed for every `j < min(m,n)`. If any singular value is numerically near-zero but not exactly zero, the solve produces a numerically exploded solution rather than the least-norm minimum-norm answer. The pseudo-inverse path (`Matrix.PseudoInverse()`) applies a `σ_max · ε_machine · max(m,n)` threshold before inversion and is the only surface with truncation. A caller that holds an `Svd<double>` handle and calls `Solve(b, x)` directly receives no truncation; the threshold from `Svd.Rank` is never consulted by `Solve`. The consequence: `Solve` and the `Rank` / `ConditionNumber` evidence on the same handle are computed under different threshold conventions and can disagree — `Rank` can report deficiency while `Solve` still inverts all singular values.
- The SVD singular values in the managed provider are delivered in descending order. The final case-4 convergence branch executes an insertion sort step (`for (; i != num16 - 1 && !(array4[i] >= array4[i + 1]); i++)`) that bubbles each converged singular value into its descending position. `S[0]` is therefore guaranteed to be the largest, making `Svd.L2Norm = |S[0]|` and `Svd.ConditionNumber = |S[0]| / |S[min(m,n)-1]|` correct without the caller sorting.

[EVD_RANK_THRESHOLD_FAILURE]:
- `Evd.Rank` and `Evd.IsFullRank` test `eigenValue.AlmostEqual(Complex.Zero)`, which resolves to `|λ|.AlmostEqualNorm(0.0, |λ - 0|, DefaultDoubleAccuracy)` where `DefaultDoubleAccuracy = 10 · 2⁻⁵³ ≈ 1.11e-15` — a fixed absolute threshold independent of matrix scale. For a well-scaled covariance matrix with true smallest eigenvalue `~1e-12`, the rank probe classifies that eigenvalue as non-zero, returning full rank even when the numeric problem is near-singular. The SVD-based `Svd.Rank` uses `S.Maximum().EpsilonOf() · max(m,n)`, scaling to the spectral radius; for the same matrix it would detect the deficiency. EVD-based rank is therefore unreliable as a conditioning gate and must not be substituted for the SVD rank evidence receipt.
- `DenseMatrix.IsSymmetric()` uses exact `!=` comparison on `double` entries — no epsilon. A matrix built via incremental accumulation (e.g., successive outer-product updates, normal-equation formation through loops) is frequently not exactly symmetric because `a[i,j]` and `a[j,i]` traverse different floating-point paths. When passed to `Evd(Symmetricity.Unknown)`, `IsSymmetric()` returns `false`, and the EVD falls through to the asymmetric Hessenberg path (slower, produces complex eigenvalue pairs, and `Evd.Solve` throws). The admission gate must force symmetry explicitly — `matrix.Add(matrix.Transpose()).Multiply(0.5)` — or pass `Symmetricity.Symmetric` with the caller accepting the structural guarantee.

[QR_LAZY_R_AND_THIN_PATH]:
- The public `QR<T>.R` property is `Lazy<Matrix<T>>`, initialized from `FullR.UpperTriangle()` — it allocates a new dense matrix and extracts the upper triangle on first access. For the `QRMethod.Full` path, `FullR` is already the m×m full upper triangular storage so `R` adds an allocation. For the `QRMethod.Thin` path, `FullR` is an n×n matrix containing R; `Q` is the m×n thin orthogonal factor. In both paths `FullR` (not `R`) is used internally in every `Solve` call — the lazy `R` property exists only for inspection and evidence extraction, never for the solve route. Accessing `qr.R` from performance-sensitive code during repeated solves allocates unnecessarily; the solve path uses `FullR` directly.
- `DenseQR.Create(matrix, QRMethod.Full)` stores the full Q as an m×m matrix and leaves the Householder reflector coefficients in `Tau`. The provider's `QRSolveFactored` receives both `Q.Values`, `FullR.Values`, and `Tau`, which allows the native path (MKL) to reconstruct Q implicitly from the reflectors rather than materializing it. Under the managed provider, `Q` is already materialized during factorization so `Tau` is vestigial for the managed solve path. Under MKL, `Tau` is the compact representation and materializating `qr.Q` for inspection causes an implicit Q-application; do not access `qr.Q` in hot solve paths under MKL.

[CHOLESKY_DETERMINANT_LOG]:
- `Cholesky<double>.DeterminantLn` computes `2 · Σ log(L[i,i])` where `L[i,i]` are the diagonal entries of the lower Cholesky factor. This is numerically stable for large `n` where `Determinant = Π L[i,i]²` underflows to zero (n > ~750 for a unit-diagonal L). Gaussian process log-likelihood, Kalman filter innovation covariance, and multivariate normal normalization all require `log|Σ|`; `DeterminantLn` is the correct surface — `Math.Log(Determinant)` is not, because `Determinant` returns `0.0` for large SPD matrices without any indication of underflow, making the log return `-Inf` silently.
- The Cholesky `Determinant` property iterates `Factor.At(i, i)` via the virtual `MatrixStorage<T>.At` dispatch — one virtual call per diagonal element. For n > 50, this is non-trivial overhead compared to accessing `((DenseMatrix)cholesky.Factor).Values[i * n + i]` directly (column-major indexing: diagonal element of column `i` is at offset `i * n + i`). `DeterminantLn` has the same overhead. For repeated determinant computation over a hot loop (MCMC, expectation-maximization), cache the diagonal span from `Values` and use `TensorPrimitives.Log` + `TensorPrimitives.Sum` over a diagonal-index projected `ReadOnlySpan<double>`.

[ARRAY_ALIASING_HAZARD]:
- `new DenseMatrix(rows, columns, storage)` binds the provided `double[]` directly to `DenseColumnMajorMatrixStorage<T>.Data` without copying ("used directly without copying" is stated in the XML doc). Any subsequent factorization that calls `DenseMatrix.Clone()` internally (all six `Dense*.Create` paths) produces an independent copy; the original array is preserved. However, `Cholesky.Factorize(matrix)` — the in-place re-factorization method on a retained handle — uses `Buffer.BlockCopy(matrix.Values, 0, factor.Values, 0, ...)` to copy the input into the handle's internal factor buffer first, then overwrites factor in place. The input `matrix` is not mutated. But `MapInplace` and provider-level operations like `MatrixMultiplyWithUpdate` targeting a `DenseMatrix` constructed from an external array mutate that array in place. This is the aliasing hazard in assembly loops: if the same `double[]` is wrapped into two `DenseMatrix` handles, any in-place operation on one mutates the other's view without exception.
- `DenseMatrix.Values` (on the `Double.DenseMatrix` class) returns the `double[] _values` field from the storage — the same reference as `DenseColumnMajorMatrixStorage<double>.Data`. External code that holds both the raw array and the matrix wrapper observes mutations through either surface. The canonical safe pattern is `matrix.Clone()` before any factorization call when the input originates from an external buffer, and reserving direct array construction (`new DenseMatrix(rows, cols, existingArray)`) exclusively for cases where the hot-path allocation cost must be avoided and the aliasing scope is single-owner.

[WEIGHTED_REGRESSION_DENSIFICATION]:
- `WeightedRegression.Weighted<T>(Matrix<T> x, Vector<T> y, Matrix<T> w)` forms `xᵀ(Wx)` by evaluating `x.TransposeThisAndMultiply(w * x)`. The sub-expression `w * x` always allocates a new m×n dense matrix regardless of whether `w` is diagonal — the `Matrix<T>` multiplication operator has no diagonal-aware fast path when the left operand is `DiagonalMatrix` and the right is `DenseMatrix`. When `w = Matrix.Build.Diagonal(weights)`, use `x.PointwiseMultiply(...)` on column-scaled rows instead, or scale `x`'s rows directly to form the weighted design matrix without materializing the full `Wx` intermediate. The correct efficient form for a diagonal weight vector `wDiag: double[]` is to scale each row `i` of `x` by `Math.Sqrt(wDiag[i])` into a new matrix `xw`, then compute `xw.TransposeThisAndMultiply(xw)` — this avoids the m×n dense intermediate and executes one half-multiplication.
- `NormalEquations<T>` has no singularity check before calling `Cholesky()`. When the design matrix is rank-deficient or nearly so, `Cholesky()` throws `ArgumentException("Matrix must be positive definite.")` — but the throw message does not identify which column caused the pivot failure. The evidence receipt at the admission boundary must wrap the `Cholesky()` call in `Fin<Cholesky<T>>` and treat the thrown `ArgumentException` as a `ConditioningFailure` evidence tag, not a generic error. The singular values of the original design matrix — computable from the retained `QR.IsFullRank` check — are the correct pre-admission gate.

[EVDSOLVE_MISSING_BLAS_PATH]:
- `DenseEvd.Solve` for the symmetric case executes two O(n²) scalar triple-nested loops using `matrix.At(k, i)` virtual dispatch for the eigenvector products — one for Vᵀb and one for Vd⁻¹(Vᵀb). No `MatrixMultiplyWithUpdate` call, no BLAS3 path, no MKL acceleration. The asymmetric case throws `ArgumentException("Matrix must be symmetric.")`. This means EVD solve throughput does not scale with MKL or with `Control.MaxDegreeOfParallelism`, unlike Cholesky, LU, and QR solves which forward directly to the provider. For a symmetric system where EVD is needed for evidence (eigenvalues for spectrum analysis, positive-definiteness certificate), prefer Cholesky for the actual solve and use EVD only for eigenvalue extraction.
- The managed EVD `EigenDecomp(double)` initializes `matrixEv` by copying the full input matrix via `Buffer.BlockCopy(matrix, 0, matrixEv, 0, matrix.Length * 8)` before tridiagonalization. The input matrix double-allocates: the `DenseEvd.Create` caller provides a `DenseMatrix` whose `Values` are cloned (not passed); then `EigenDecomp` copies again into `matrixEv`. Total allocations for EVD of an n×n matrix: `matrix.Clone()` (n²), `matrixEv` (n²), `denseMatrix2` (block-diagonal, n²), `DenseVector` for `denseVector` (n). Three n² allocations before any computation.

[TENSOR_PRIMITIVES_DENSE_FLOOR]:
- `System.Numerics.Tensors.TensorPrimitives` (in-box, net10.0) exposes `Dot<T>`, `MultiplyAdd<T>`, `FusedMultiplyAdd<T>`, `SumOfSquares<T>`, `Norm<T>`, and `CosineSimilarity<T>` over `ReadOnlySpan<T>` with generic math constraints. `FusedMultiplyAdd<T>` requires `IFloatingPointIeee754<T>` — satisfied by `double` — and executes a fused multiply-add in a single rounding, delivering improved accuracy for inner products compared to separate multiply + add. For dense residual computation after a factorization solve — `r = A·x - b` — this is the correct vectorized primitive: accumulate each column contribution via `TensorPrimitives.MultiplyAdd(columnSpan, x[col], accumulatorSpan, accumulatorSpan)` down the column-major layout.
- `TensorPrimitives` has no GEMM surface — no matrix-multiply primitive exists at the span level in net10.0. Dense GEMM remains the exclusive domain of the MathNet provider (managed or MKL). `TensorPrimitives` is the floor for element-wise and reduction hot loops that the provider does not own: all-finite checks (`TensorPrimitives.IndexOfAnyExcept<double>(span, ...)`), diagonal extraction and log-sum for `DeterminantLn`, residual norm computation, and symmetric force (`TensorPrimitives.Add` applied to transposed span pairs).
- `Dot<T>` constraint is `IAdditionOperators<T,T,T>, IAdditiveIdentity<T,T>, IMultiplyOperators<T,T,T>, IMultiplicativeIdentity<T,T>` — satisfied by `double`. `FusedMultiplyAdd<T>` adds `IFloatingPointIeee754<T>`. Both constraints are known at compile time for `double`, so no boxing occurs. The SIMD dispatch inside `TensorPrimitives` checks hardware capabilities at runtime (AVX512 → AVX2 → SSE2) and vectorizes accordingly; the managed MathNet provider's `DotProduct` loop is scalar and receives none of this.

[COLUMN_MAJOR_LAYOUT_EXPLOITATION]:
- All `DenseColumnMajorMatrixStorage<double>` data is laid out as `Data[col * RowCount + row]`. For a column of column index `c`, the entire column occupies `Data[c*n .. (c+1)*n - 1]` as a contiguous slice — `new ReadOnlySpan<double>(values, c * n, n)`. Row access is strided with stride `n`. Dense algorithms that need to operate column-by-column (GEMM column panel, AXPY update, triangular solve) can use `MemoryMarshal.Cast` or direct span slicing on `matrix.Values` without marshaling overhead. The managed `MatrixMultiplyWithUpdate` already exploits this by extracting columns of B into `columnDataB` before the outer loop; a `TensorPrimitives.Dot` call on each extracted column would replace the inner scalar loop with SIMD, yielding the primary optimization opportunity in the managed GEMM kernel.
- The Cholesky inner loop (`DoCholeskyStep`) operates column-by-column on the factor buffer in column-major layout. The update `data[col * n + row] -= multipliers[row] * multipliers[col]` for `row >= col` corresponds to a SAXPY-style rank-1 update on the column panel. This is the precise shape `TensorPrimitives.MultiplyAdd(sourceSpan, multipliers[col], targetColSpan, targetColSpan)` would vectorize. The managed Cholesky does not use `TensorPrimitives`; replacing the inner loop with a `MultiplyAdd` span call is a direct vectorization path that does not require MKL.

[REGISTRATION_DETERMINANT_CORRECTION]:
- In the closed-form orthogonal alignment (cross-covariance SVD), the determinant correction for reflection cases uses `svd.Determinant` on the rotation handle, which for a 3×3 is the product `|σ₁ · σ₂ · σ₃|` — guaranteed non-negative by the absolute-value in `Svd.Determinant`. This cannot detect reflections. The correct test is `det(V · Uᵀ)` = `sign(det(VT.Transpose()) · det(U.Transpose()))`, which equals the sign of the product of the two independent determinants. `DenseMatrix.Determinant()` (via LU) or `Math.Sign(svd.U.Determinant() * svd.VT.Determinant())` provides the correct signed quantity. `LU.Determinant` on a `DenseLU` handle for a 3×3 is three multiplications plus pivoting and is not deferred — it executes each time the property is accessed with no caching. For a 3×3, the determinant of a rotation matrix is ±1.0 exactly in theory; due to fp rounding, the check must be `det < 0.0`, not `det != 1.0`.
- Damped normal equations via `(JᵀJ + λI).Cholesky().Solve(JᵀR)`: the Jacobian gram matrix `JᵀJ` formed by `TransposeThisAndMultiply` squares the condition number. For a Jacobian with condition `κ(J)`, the damped system has condition approximately `(κ(J)² + λ) / λ` near the optimal λ — it does not recover from the squaring for small λ. The correct Levenberg-Marquardt inner solve uses the augmented Jacobian `[J; √λ I]` QR-factored directly: overdetermined QR on the (m+n)×n stacked system avoids squaring the condition number. This requires forming an (m+n)×n dense matrix but preserves κ(J) rather than κ(J)² in the solve. The admitted package surface (thin QR) supports this directly — `DenseMatrix.OfRowArrays(J, sqrtLambdaIdentity).QR(QRMethod.Thin).Solve(b_augmented)`.
