# Composition Law — Deepening

[NATURAL_TRANSFORM_HIERARCHY]:
- `Natural<F,G>` declares `static abstract K<G,A> Transform<A>(K<F,A>)`. `CoNatural<F,G>` is the dual: `static abstract K<F,A> CoTransform<A>(K<G,A>)`. `NaturalMono<F,G>` implements both via default `CoTransform` that delegates to `F.Transform`, providing one-way promotion with zero manual impl. `NaturalIso<F,G>` implements both independently for true bidirectional conversion. `NaturalEpi<F,G>` is the reverse-dual: it implements `Natural<G,F>` by calling `F.CoTransform`, so the "from" and "to" type parameters are inverted relative to the interface names. These are load-bearing distinctions: `NaturalMono` is correct for one-directional promotions like `Fin→Eff`; using `NaturalIso` wrongly asserts a roundtrip that doesn't hold if any failure information is lost.
- `FinT<M>` implements `Natural<FinT<M>, EitherT<Error,M>>`, `Natural<FinT<M>, OptionT<M>>`, and `Natural<FinT<M>, TryT<M>>` — all one-way promotions via `Natural.transform`. These allow an algorithm producing `K<FinT<M>,A>` to be promoted to an `EitherT` or `OptionT` context at the call site through `Natural.transform<FinT<M>, EitherT<Error,M>, A>(value)` — no structural extraction required. The inverse direction is not provided: `EitherT` cannot be narrowed to `FinT` via `CoNatural` because the natural transformation is not an isomorphism (the `Right`/`Left` distinction of `Either` is structurally richer than `Fin`'s success/failure).
- `CoNatural.transform<F,G,A>(K<G,A>)` is the dual call-site combinator, callable when `F : CoNatural<F,G>`. The distinction from `Natural.transform` is in which type parameter bears the constraint: the target type `F` provides `CoTransform`. This matters for transformer stacks where the outer monad implements `CoNatural` against its inner monad — the call site uses `CoNatural.transform<Outer,Inner,A>` to go downward, and `Natural.transform<Inner,Outer,A>` to go upward.

[FINT_AS_MONAD_TRANSFORMER]:
- `FinT<M,A>` is `record(K<M, Fin<A>> runFin)` — the outer monad `M` is load-bearing and must satisfy `Monad<M>`. `FinT<M>` (the trait class) implements `MonadT<FinT<M>, M>`, which implies `Lift<A>(K<M,A>) → K<FinT<M>,A>` via `FinT.lift(ma)`. This is the canonical path for lifting an existing `M` computation — such as `IO<A>` or `Eff<A>` — into the transformer stack. An algorithm that accumulates failure via `Fin` while also tracking effects in `M` composes `FinT<IO,A>` or `FinT<Eff,A>` rather than nesting the two independently.
- `FinT<M>.MapM<B>(Func<K<M,A>,K<M,B>>)` on the instance transforms the inner `K<M,A>` while preserving the `Fin` layer. It extracts the success value, runs `f(M.Pure(value))`, then re-wraps in `Fin.Succ`. The `Fin.Fail` case short-circuits into `M.Pure(Fail)`. `MapT<M1,B>(Func<K<M,Fin<A>>,K<M1,Fin<B>>>)` transforms the monad type itself — the outer `M` to `M1` — enabling hoisting across monad boundaries without flattening the `Fin` envelope.
- `FinT<M>` implements `Applicative` via `Apply` that uses the `*` applicative operator on `K<M, Fin<Func<A,B>>>` and `K<M, Fin<A>>` concurrently — the lambda `(ff, fa) => ff.Apply(fa)` runs the applicative product inside `M`, then applies the function inside `Fin`. This differs from `Bind` which sequences: `Apply` enables the outer monad's concurrency semantics (parallelism if `M = IO`) while keeping `Fin` accumulation semantics intact for the inner carrier.
- `FinT<M>` implements `MonoidK` via `Combine`: if the left `runFin` is a `Fin.Succ`, it short-circuits; if it is `Fin.Fail`, the right `runFin` is evaluated via `M.Bind`. This is the `<|>` alternative behaviour inside the transformer: the first successful branch wins without evaluating the second, which is the correct lazy semantics for alternatives in a transformer stack.

[FALLIBLE_PARTITION_COMBINATORS]:
- `FallibleExtensions.PartitionFallible<F,M,A>(K<F,K<M,A>>)` where `F : Foldable<F>`, `M : Monad<M>, Fallible<M>` returns `K<M,(Seq<Error> Fails, Seq<A> Succs)>`. It folds over the foldable, running each effect and accumulating successes and failures independently using `Catch`. The fold is left-to-right and sequential — each `K<M,A>` is `Bind`-ed in order. This is the correct combinator when a batch of operations must all be attempted and results partitioned: it neither short-circuits on the first failure nor requires a `Validation` carrier. Overloads exist for `Seq<K<M,A>>`, `Iterable<K<M,A>>`, `Lst<K<M,A>>`, `IEnumerable<K<M,A>>`, `HashSet<K<M,A>>`, `Set<K<M,A>>`.
- `FallibleExtensions.Succs<F,M,A>` and `Fails<F,M,A>` are the single-side variants: `Succs` collects values from successful effects, silently discarding failures; `Fails` collects errors, silently discarding successes. The implementation of `Fails` uses `ma.Bind(_ => M.Pure(s)).Catch(e => M.Pure(s.Add(e)))` — the success case returns the unchanged accumulator, the failure case appends the error — making both combinators stack-safe for any `M` that supports `Bind` and `Catch`.
- The generic `fails<E,F,M,A>(K<F,K<M,A>>)` in the traits module carries a general error type parameter `E` distinct from `Error`, enabling partition over carriers with typed error rails — `ValidationT<F,M,A>` where `F` is a custom monoid. The concrete overloads in the non-trait module pin `E = Error`. The generic versions require `M : Monad<M>, Fallible<E,M>` and `F : Foldable<F>`.

[ERRORHIERARCHY_STRUCTURAL_QUERYING]:
- `ManyErrors.IsType<E>()` where `E : Error` returns true if `Errors.Exists(e => e.IsType<E>())` — it traverses children recursively because the base `Error.IsType<E>` delegates to `this is E`. For `Expected` and `Exceptional` at leaf nodes, `IsType<E>` is a direct `is`-check with no recursion. For nested `ManyErrors`, the traversal is a `Seq.Exists` loop. The implication: `catchOf<Expected,M,A>` on a `ManyErrors` containing mixed expected and exceptional children triggers the handler even though the outer wrapper is neither.
- `ManyErrors.Filter<E>()` where `E : Error` returns a new `ManyErrors` containing only the children where `IsType<E>()` is true, via `Errors.Choose`. Children failing the predicate are removed. The base `Error.Filter<E>()` returns `this` if `this is E`, else `Errors.None`. `Filter` is not called by any `CatchM` handler internally — it is a consumer-side extraction tool for post-hoc error analysis after the rail has terminated.
- `Error.ForAllM<M,A>(Func<Error,K<M,A>> f)` on a single error calls `f(this)`. On `ManyErrors`, it sequences: `k = Errors[0].ForAllM(f)` then `foreach(e in Errors.Tail) k = k.Bind(_ => e.ForAllM(f))`. This is monadic sequential execution across all child errors, short-circuiting on the first failure produced by `f`. The complementary `FoldM<M,A>` uses `MonoidK` combine instead of `Bind`, enabling parallel collection semantics. These are the composable error-as-structure traversal combinators; `AsIterable()` on `ManyErrors` returns a flat `Iterable<Error>` of direct children without deep-flattening nested `ManyErrors`.
- `Error.IsEmpty` is distinct from null: `ManyErrors(Seq.empty<Error>)` is the canonical `Errors.None` value. `IsEmpty = true` and `Code = -2000000006`. An `Error.Combine(empty)` short-circuits to `this`; `empty.Combine(rhs)` returns `rhs`. This means `ManyErrors.Empty` functions as the monoidal zero for error accumulation — `+` with it is an identity — but it is still an error value that causes `Fin.Fail` if it appears on the failure rail. Code that checks `e.IsEmpty` before routing to a recovery path is the correct form; `.IsEmpty` is never equivalent to `null` or a recovered state.

[SCHEDULE_TRANSFORMER_COMPOSITION]:
- `ScheduleTransformer` is a `readonly struct` wrapping `Func<Schedule,Schedule>`. The `+` operator composes two transformers left-to-right: `(f + g).Apply(s) = g.Apply(f.Apply(s))`. The implicit conversion `ScheduleTransformer → Schedule` applies the transformer to `Schedule.Forever`, producing an infinite schedule with the transformed delays. This enables writing `Schedule.jitter(0.5) + Schedule.maxDelay(5.Seconds())` as a reusable modifier policy separate from the retry count.
- `Schedule.NoDelayOnFirst` is a `ScheduleTransformer` that prepends a zero-duration delay: `s.Tail.Prepend(Duration.Zero)`. Applied to `Schedule.recurs(3) | Schedule.exponential(100*ms)`, the first iteration runs immediately (zero delay), and subsequent iterations use the exponential delays. `Schedule.PrependZero` on a `Schedule` instance does the same directly. The distinction: `NoDelayOnFirst` is a transformer for reuse across multiple schedules; `PrependZero` is a one-shot instance method.
- `IO.RetryUntil(Schedule, Func<Error,bool>)` iterates the schedule via `schedule.PrependZero.Run().GetIterator()`, waits `IO.yieldFor(head)` before each attempt, and `BracketFail()`s each attempt so resources acquired in a failed attempt are released before the next retry begins. The schedule drives iteration count and inter-attempt delay independently of the predicate. When the iterator is `Nil` (schedule exhausted), the last error is raised. When the predicate returns `true`, the error is re-raised immediately regardless of remaining schedule capacity.
- `Schedule.RepeatForever` is a `ScheduleTransformer` that wraps the schedule in `SchRepeatForever`, causing the schedule to restart from its beginning when exhausted rather than terminating. Composing `Schedule.recurs(3) | Schedule.exponential(100*ms) | Schedule.RepeatForever` produces a schedule that cycles through 3 exponential delays indefinitely. This is the correct primitive for watchdog-style loops; combining with `RepeatWhile` on the `IO` layer terminates the loop on a value predicate rather than a schedule limit.

[ENV_IO_SCOPE_MANAGEMENT]:
- `EnvIO` has four local-scope constructors: `.Local` (new resources, new token source — full isolation), `.LocalResources` (new resources, shared token source), `.LocalCancel` (shared resources, new token source), `.LocalSyncContext` (capture current `SynchronizationContext.Current`). `IO.Bracket()` uses `WithEnv(env => env.LocalResources)` — a fresh resource scope that shares the cancellation token of the parent. `IO.Local()` uses `WithEnv(env => EnvIO.New(env.Resources, env.Token, null, env.SyncContext))` — fresh token source but shared resources. `localCancel<A>(Eff<A>)` calls `ma.LocalIO()` which delegates to `LocalIO<A>` on `MonadUnliftIO`, which calls `.Local()` — full scope isolation including cancellation. The design pressure: cancellation propagation direction is a scope decision, not a carrier decision.
- `IO<A>.Fork(Option<TimeSpan>)` creates a new `CancellationTokenSource` linked to the parent's token via `parent.Register(() => tsrc.Cancel())`, runs the computation on the thread pool with `RunAsync(EnvIO.New(resources, token, tsrc, env.SyncContext)).GetAwaiter().GetResult()` on a thread pool thread, and returns `ForkIO<A>(Cancel: IO<Unit>, Await: IO<A>)`. The forked computation inherits the parent's `Resources` — resources acquired inside the fork are visible to the parent and vice versa. The `Cancel` IO signals the forked `CancellationTokenSource`. `Await` is `f.Await` — the stored task result. Forked computations that fail capture the error inside the `Await` IO rather than propagating to the parent thread.
- `IO<A>.Post()` marshals the computation onto the `SynchronizationContext` captured in `EnvIO.SyncContext` using `SyncContext.Post(delegate, null)` with an `AutoResetEvent` for synchronization. If `SyncContext` is null, it falls through to `RunAsync(env)` directly. `PostIO<A>(K<IO,A>)` on `MonadUnliftIO<IO>` is the carrier-polymorphic form that calls `.Post()`. This is the correct primitive for UI-thread dispatch inside a pipeline; `localCancel` then `PostIO` is the composition for scoped UI work with independent cancellation.

[CARRIER_POLYMORPHIC_TAIL_RECURSION]:
- `Monad<M>` declares `static abstract K<M,B> Recur<A,B>(A value, Func<A,K<M,Next<A,B>>> f)`. The contract requires implementors to provide a stack-neutral (trampoline) implementation. `unsafeRecur` is the non-trampolining fallback provided only for carriers where deep recursion is unlikely; using it for `IO` or `Eff` is explicitly unsupported for deep pipelines. `iterableRecur<M,A,B>` is the prebuilt trampolining form via `Iterable` async enumeration — it converts `M` to `Iterable` via `Natural.transform`, iterates asynchronously on a `CancellationToken`, and converts back via `CoNatural.transform`. It requires `M : Natural<M,Iterable>, CoNatural<M,Iterable>`, which `IO` and `Eff` satisfy.
- `Monad.recur<M,A,B>(A value, Func<A,K<M,Next<A,B>>> f)` is the public entry point; `Next<A,B>` is a discriminated union with `IsDone/IsLoop` discriminants and `Done : B` / `Loop : A` payloads. Writing a tail-recursive loop in domain logic uses `recur` with `Next.Done(result)` and `Next.Loop(nextState)` inside the step function. The result type `B` is independent of the accumulator type `A`, enabling folds that produce different types from their loop state. This replaces `while`/`for` loops without stack growth for deep iteration.
- `FinT<M>` implements `Monad<FinT<M>>.Recur` via `M.Recur` applied to the step function after adapting `Next<A,Fin<B>>` from the inner result: a `Fin.Fail` inside the step produces `Next.Done(fail)` — short-circuiting the loop with the error as the loop result. This means `recur` on `FinT<M>` participates in both the outer monad's trampoline and the inner `Fin` error short-circuit; a domain loop over `FinT<IO,A>` is stack-safe, cancellable, and error-aborting without any composition glue.

[CATCHM_AND_PIPE_OPERATOR_MECHANICS]:
- `CatchM<E,M,A>` is `readonly record struct(Func<E,bool> Match, Func<E,K<M,A>> Action)`. Its `Run(E error, K<M,A> otherwise)` method applies the predicate: if false, returns `otherwise` (the original computation); if true, calls `Action(error)`. The `|` operator on `K<F,A>` where `F : Fallible<E,F>` is a C# 14 extension method inside `FallibleExtensions`: `lhs | rhs` calls `lhs.Catch(rhs)` which calls `F.Catch(lhs, rhs.Match, rhs.Action)`. Multiple `|` handlers compose left-to-right by the left-associativity of `|`, producing nested `Catch` calls each delegating to `F.Catch`. The first matching handler consumes the error and subsequent handlers are not tested.
- `CatchMExtensions` adds a `Map<B>` to `CatchM<E,M,A>`: it produces `CatchM<E,M,B>(self.Match, e => self.Action(e).Map(f))`. This enables reusing a handler predicate while transforming its recovery output, without re-writing the predicate. A shared handler factory returns `CatchM<Error,M,Unit>` and callers `Map` it to their specific bound type.
- `CatchIO<M,A>(Func<Error,bool>, Func<Error,K<IO,A>>)` is the `MonadIO`-specific catch: the recovery function returns `K<IO,A>` rather than `K<M,A>`. Internally it calls `M.Catch(ma, Predicate, e => M.LiftIO(Fail(e)))`, meaning the recovery is injected into `M` via `LiftIO`. This is the entry point for recovery paths that require `IO` effects (logging, metrics) without constraining the algorithm's outer monad to exactly `IO`.
- `K<F,A> | Error rhs` (where `F : Fallible<F>`) is a convenience overload that catches all errors and replaces with the given `Error` failure, implemented as `F.Catch(lhs, _ => true, _ => F.Fail<A>(rhs))`. This is the terminal "replace error" combinator, not a recovery — it forces a different error but stays in the failure rail. The correct use is contextual wrapping at a layer boundary, not retry.

[VALIDATION_APPLICATIVE_SEAM_PRESSURE]:
- `Traversable.TraverseM<T,M,A,B>` has a default implementation that delegates to `Traversable.traverse<T,M,A,B>` — the applicative form. This means `Seq.TraverseM<Fin,A,B>` uses `Fin`'s applicative `Apply`, which short-circuits on first failure rather than accumulating. The only carrier that accumulates across a traversed sequence is `Validation<FAIL>`. Passing `traverseM` when applicative traversal was intended, or using `traverse` when monadic short-circuit was intended, silently produces different error semantics with no compile-time indication.
- `Validation<F,M>` (the transformer) implements `Alternative<ValidationT<F,M>>` and `MonoidK<ValidationT<F,M>>`. Its `Bind` short-circuits (monad). Its applicative `Apply` is generated via the `Bimonad` bifunctor: `BiMap` maps both sides. When accumulation across fields is needed inside a `ValidationT<Error,IO>` context, the correct path is to build a field product via `*` applicative operators — not via `from`/`select` query expressions which desugar to `Bind`.
- `Validation<Error,A>.ToFin()` transfers the entire accumulated `ManyErrors` as a single `Fin.Fail`. Once inside `Fin`, there is no accumulation — subsequent `Bind` steps short-circuit. The seam between the admission phase and the domain pipeline is therefore the only location where `Validation`'s `ManyErrors` accumulation is meaningful. Code that calls `.ToFin().ToValidation()` mid-pipeline re-enters a monadic (Bind-based) `Validation` whose applicative accumulation cannot recover earlier failures that were already short-circuited by `Fin.Bind`.

[ANTI_PATTERNS_DEEPENED]:
- `Fallible.fail<E,F,A>(E error)` and `Fallible.error<F,A>(Error error)` are the static module entry points for raising failures in a carrier-polymorphic context. Using `Fin.Fail(e)`, `Eff.Fail(e)`, or `Option.None` directly inside a carrier-polymorphic algorithm (`K<M,A> where M : Fallible<Error,M>`) binds the implementation to a concrete carrier, defeating the entire K-trait abstraction. `Fallible.fail<Error,M,A>(e)` is the correct form inside such algorithms.
- `ScheduleTransformer`'s implicit conversion to `Schedule` applies the transformer to `Schedule.Forever`. Storing a `ScheduleTransformer` in a variable and passing it where `Schedule` is expected triggers this conversion silently, producing an infinite schedule where a bounded one was intended. Transformers and schedules must be kept at their declared types until a deliberate `|` compose or explicit `Apply` call produces the final `Schedule`.
- Calling `RetryUntil` or `Retry` directly on an `IO<A>` after `BracketFail()` has already been applied in an outer scope means every failed attempt releases the outer scope's resources, not just the retrying computation's resources. The correct structure wraps only the minimal retrying unit in `BracketFail()` via the retry methods themselves — which internally call `BracketFail()` — rather than pre-wrapping and then retrying.
- Writing `io.Fork().Bind(f => f.Await)` instead of `MonadUnliftIO.await(io.Fork())` re-implements the `Await` pattern manually. The carrier-polymorphic `Monad.await<M,A>(K<M,ForkIO<A>>)` delegates to `MonadUnliftIO.Await` and ensures the IOEmpty fast-path is preserved. The manual form breaks the fast-path for `IOEmpty<A>`, adding unnecessary allocation.
