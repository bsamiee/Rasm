# Carrier Deepening

[IMPLICIT_CONVERSION_SURFACE]:
- Every carrier type exposes implicit lifts from its value type: `Fin<A>` accepts `A`, `Error`, `Either<Error,A>`, `Pure<A>`, and `Fail<Error>` as implicit sources. `Either<L,R>` accepts both `L` and `R` implicitly. `Validation<F,A>` accepts `A` and `F` implicitly. `Option<A>` accepts `A?` via `Optional()` null-guard. None of these appear in carrier-selection reasoning, but each one changes how the compiler resolves expression positions — a bare `A` value in an `Eff<A>`-returning expression auto-promotes through `Fin<A>` → `Eff<A>` via two implicit hops without any explicit operator call.
- The `Fin<A>` implicit from `A` is not null-guarded: a `null` reference type value becomes `Succ(null)`, not `Fail`. The `Option<A>` implicit from `A?` calls `Optional()` and null-folds to `None`. This asymmetry means the carrier choice at nullable ingress is not interchangeable: `Option` is the null-safe boundary carrier; `Fin` is not.
- `Eff<A>` carries a direct implicit from `Fin<A>`, so a `Fin<A>` value returned in any `Eff<A>` position is silently promoted. The inverse does not exist. Explicit `.Run()` or effect materialization is always required to recover `Fin<A>` from `Eff<A>`.

[EXPLICIT_CAST_TRAP]:
- `Validation<F,A>` exposes `explicit operator A` and `explicit operator F`. Both delegate to the internal `SuccessValue`/`FailValue` properties, which are `internal` — accessible only through the cast operator. A cast on the wrong case throws `InvalidCastException`. Both `Fin<A>` and `Validation<F,A>` have `internal` discriminant value accessors; there are no `public` unsafe value properties in v5. The `.SuccValue`/`.FailValue` description from v4 documentation is gone — match exclusively via `case Fin<A>.Succ`, `case Fin<A>.Fail`, or the `{ IsSucc: true }` flat-match form.

[APPLY_SEMIGROUP_DEGRADATION]:
- `Validation<F,A>.Apply` carries `where F : Semigroup<F>` at the call site, enforced by the compiler. When `F` satisfies the constraint, `F.Instance` (a static virtual property) resolves to `SemigroupInstance<F>` at JIT time and accumulation works. When `F` does not implement `Semigroup<F>`, `Apply` cannot be called — the compiler rejects it. This means the runtime failure described as `TypeLoadException` cannot occur through `Apply`; it applies only to internal/indirect dispatch paths (e.g., `ValidationT` transformer stack operations that call `F.Instance` via reflection through `SemigroupInstance<A>.GetInstance()`).
- The reflection fallback in `SemigroupInstance<A>.GetInstance()` uses `typeof(Semigroup<>).MakeGenericType(typeof(A)).GetProperty("Instance")?.GetValue(null)`. When `A` is not `Semigroup<A>`, this returns `null`, and `SemigroupInstance<A>.Instance` is `Option<SemigroupInstance<A>>.None`. The `ApplyI` path that receives `Option<SemigroupInstance<F>>` gracefully handles `None` by falling through to first-fail (not accumulated) behavior. The accumulation guarantee requires `F : Semigroup<F>` at compile time; the degradation mode is not a crash but a silent collapse to fail-first semantics.
- `SelectMany` on `Validation<F,A>` (instance method) carries no `F` constraint and always operates sequentially (fail-first). Error accumulation requires `Apply` (or `Applicative.lift` / `CombineI`), never `SelectMany`. A LINQ query expression on `Validation` is always sequential.

[VALIDATION_MAPFAIL_TARGET_CONSTRAINT]:
- `Validation<F,A>.MapFail<F1>(Func<F,F1> f)` requires `where F1 : Monoid<F1>` — the target failure type must be a monoid, not merely the source. This prevents narrowing to an arbitrary type at the failure-transform site; it also prevents MapFail as a free converter between validation error shapes when the target does not satisfy the monoid law. `BiBind<F1,A1>` has no such constraint on `F1` — it provides the escape hatch when the target failure type is not a monoid, because `BiBind` bypasses accumulation entirely.

[ERROR_HIERARCHY_MECHANICS]:
- `Error.Combine` (op_Addition) flattens two `ManyErrors` into a single `ManyErrors` by appending their `Seq<Error>` sequences. Combining a scalar `Error` with a `ManyErrors` prepends or appends the scalar into the sequence. Combining two scalars produces `ManyErrors(Seq(this, other))`. `Errors.None` is `ManyErrors([])` with `IsEmpty = true`; it is the monoid identity — `Combine(Errors.None, x) = x`. This means `ToFin()` on `None` Option produces `Fail(Errors.None)`, which passes `IsEmpty = true` and is structurally distinct from a real error.
- `Error.Is(Error other)` determines identity by code (when `Code != 0`) or by message string equality (when `Code == 0`). It does not use reference equality or type identity. For `ManyErrors`, `Is` walks each member asking `member.Is(other)`. `HasCode(int)` also walks the full nested structure. `IsType<E>()` tests `this is E` — structural type, not semantic code — and `ManyErrors` overrides it to test each member. The `Filter<E>()` operator on `ManyErrors` runs `Choose` over members, keeping only those that `IsType<E>()`, and returns `Errors.None` when the set is empty.
- `Error.New(Exception? e)` performs smart routing: `null` → `Errors.None`; `OperationCanceledException` / `TaskCanceledException` → `Errors.Cancelled`; `TimeoutException` → `Errors.TimedOut`; `AggregateException` → `ManyErrors` (flattens inner exceptions); `WrappedError*` exceptions round-trip back to the original `Error`. The live `Exception` is stored in `Exceptional.Value` as `private readonly` and marked `[IgnoreDataMember]` — it is lost on serialization. `Error.Exception` (the property) always returns `Some(ToException())` when `IsExceptional`, but `ToException()` for a deserialized `Exceptional` rebuilds from `Message`/`Code` only, discarding the stack trace. `HasException<E>()` returns `true` only when the live `Value` field holds an instance of `E` — it returns `false` for deserialized `Exceptional` regardless of type.

[CATCHM_ANATOMY]:
- `CatchM<E,M,A>` is a `readonly record struct` with two fields: `Func<E,bool> Match` and `Func<E,K<M,A>> Action`. Its `Run(E error, K<M,A> otherwise)` method: if `Match` returns false, returns `otherwise` unchanged (propagating the error); if true, calls `Action`. This means chaining multiple named catches via `|` is ordered: each catch's predicate is evaluated left-to-right until one matches, and the original error propagates through all mismatches. There is no `CatchM` composition operator (no `&&`, `||`) — only `Map<B>` to transform the recovery value. Multiple catches must be chained as separate `|` applications.
- The `|` operator on `Fin<A>` has two distinct overloads: `Fin<A> | Fin<A>` (alternative — returns left if `Succ`, right otherwise) and `K<Fin, A> | CatchM<Error, Fin, A>` (named recovery). Both `Validation<F,A>` and `Option<A>` carry the same `| CatchM` overload with their respective failure types (`F` and `Unit`). The named catch combinators (`@catch`, `expected`, `exceptional`, `catchOf<E>`, `expectedOf<E>`) each produce a `CatchM<Error,M,A>` composable with `|` against any `Fallible<Error,M>` carrier including `Fin`, `Eff`, `IO`, `Try`, and `Either<Error,_>`.

[FIN_NATURAL_TRANSFORMATION]:
- `Fin` implements `Natural<Fin, Eff>`, `Natural<Fin, IO>`, `Natural<Fin, Option>`, `Natural<Fin, Either<Error>>`, and `Natural<Fin, Try>`. The `Transform` implementations for `Eff` and `IO` pattern-match on `Fin<A>.Succ`/`Fin<A>.Fail` and construct the target monad's pure or fail. This is the mechanism behind the `Eff<A>` implicit conversion from `Fin<A>` — the implicit operator calls `Match(Pure, Fail)` rather than `Natural.Transform`, but the structural effect is identical. The `Natural<Fin, Option>` implementation discards the error: `Succ → Some`, `Fail → None`. There is no `Natural<Fin, Validation>` — the direction from sequential to applicative is never a natural transformation; callers must apply `ToValidation()` projection explicitly at the collection boundary.

[GUARD_FORMS_CONTRASTED]:
- `guard(bool, Error)` produces `Guard<Error, Unit>` which carries the error payload and integrates into any `Fallible<Error,M>` via `SelectMany` extension overloads. `guard<F>(bool)` with `where F : MonoidK<F>, Applicative<F>` produces `K<F, Unit>` by calling `F.Empty<Unit>()` on false and `F.Pure(unit)` on true — no error payload, no `Guard` wrapper, just the applicative's empty or pure. The two forms are not interchangeable: the payload form is the correct choice inside `Eff`/`IO`/`Fin` pipelines; the applicative form is correct for `Option`/`Seq`/`Lst` where absence rather than failure is the semantic.
- `guardnot(bool, Error)` is a strict alias that negates the condition before calling `guard`. `iff<F,A>(bool, K<F,A>, K<F,A>)`, `when<F>(bool, K<F,Unit>)`, and `unless<F>(bool, K<F,Unit>)` are the complementary carrier-polymorphic branching forms: `iff` is a non-short-circuiting value selection; `when` runs the alternative on true; `unless` runs it on false. All three require `F : Applicative<F>` only — no `Fallible` constraint.

[OPTION_AS_ALTERNATIVE]:
- `Option<A>` implements `Fallible<Unit, Option>` — its failure side is `Unit`. `@catch` against `Option<A>` requires `CatchM<Unit, Option, A>`; the named catch forms that dispatch on `Error` subtypes cannot be applied to `Option` without first projecting to `Fin` or `Eff`. The `|` on `K<Option,A>` also accepts `K<Option,A> | Pure<A>` (promotes the pure to Some) and `K<Option,A> | Unit` (no-op on the None path, returning None). These forms allow option-chain alternatives in compact expressions. The `guard<F>(bool)` applicative form with `F = Option` is the correct inline absence gate: `None` on false, `Some(())` on true, without constructing a `Guard<Error,Unit>` that would require a later `ToOption`.

[VALIDATION_COMBINEONLY_ERROR_ACCUMULATION]:
- Error accumulation in `Validation<Error,A>` is strictly applicative — `CombineI(lhs, rhs, F.Instance)` and `Apply` are the two public surfaces. `CombineI(Validation<F,A>, Validation<F,A>, Option<SemigroupInstance<F>>)` returns `Validation<F, Seq<A>>`: when both succeed, their values are collected into a `Seq`; when any fail, their errors are combined via `F.Instance.Combine`. `CombineFirst` shares the same signature but returns the first success value rather than a sequence — suitable for pairwise combining where only the first success is the goal. The public `CombineI(Validation<F, Seq<A>>, Validation<F, A>)` overload (without the `Option<SemigroupInstance<F>>` parameter) takes the already-accumulated sequence as `lhs` and appends to it, which is the fold-friendly form for accumulating over an `IEnumerable<Validation<F,A>>` via `Seq.fold`.

```csharp
// Applicative accumulation: all three checks run; all failures collected
var result =
    Validation.Succ<Error, int>(42)
        .Apply(Validation.Succ<Error, Func<int, int>>(x => x + 1))
        .CombineI(Validation.Fail<Error, int>(Error.New("b")), F.Instance);
// result: Fail(ManyErrors([Error "b"]))

// BiBind with type-changing failure side — target F1 need not be Monoid at compile time
Validation<string, int> coerced =
    someValidation.BiBind(
        f => Validation<string, int>.Fail(f.Message),
        v => Validation<string, int>.Success(v));
```
