# redaction-compliance — bedrock

## classification-taxonomy

- A classification is a two-part value — `DataClassification(taxonomyName, value)` — and a suite declares exactly one taxonomy: one closed family of attribute rows, each a sealed subclass of `DataClassificationAttribute` binding one classification value.
- The attribute targets cover class, struct, property, field, interface, parameter, return value, and generic parameter, with multiple allowed — classification can annotate every place data is declared, including type parameters of generic carriers.
- Multiple-application is how stacked sensitivity is declared: two classification attributes on one member produce the union set at the seam — stacking is annotation arithmetic, not a special combined attribute.
- The taxonomy is the suite's sensitivity vocabulary; a second taxonomy in one suite forks every downstream verdict.
- The base attribute's constructor is protected — anonymous classification is unrepresentable; every classification application is a named taxonomy row by construction, which is what makes the reachable-set audit a finite walk.
- The sentinels live outside every user taxonomy by construction (they carry an empty taxonomy name through a private constructor) — a user-declared row can never collide with `None` or `Unknown`, and equality between user rows is (taxonomy, value) pairwise.
- Two sentinels are built in and they are not interchangeable: `DataClassification.None` is an affirmative "reviewed, public" verdict; `DataClassification.Unknown` is "never reviewed".
- Unannotated data is Unknown-shaped, not None-shaped — governance treats it as sensitive-until-classified; a default that maps absence to None is the single most dangerous misconfiguration in the seam.
- The fail-closed posture rests entirely on keeping the None/Unknown distinction: None resolves to pass-through by an explicit auto-mapped row, Unknown resolves to nothing and falls to the erasing fallback.
- Values combine as sets: `DataClassificationSet` unions classifications (implicit conversion from a single classification; `Union` for composition), and the set — not the individual classification — is the redaction key.
- Two annotation sites carry one verdict: the canonical site is the contract member (property or field on the shared type), and the parameter site exists for one-off boundary values that have no contract home — a parameter annotation duplicating a member annotation is drift waiting to disagree.
- A value tagged with two sensitivities is a different compliance object than either alone — set-keyed resolution makes stacked sensitivity a first-class case instead of a precedence puzzle.
- `DataClassification` carries a type converter, so classification values bind from configuration — redaction maps can be configuration-declared without weakening the typed taxonomy.
- Classification is declaration metadata consumed by generators and providers; it never branches domain logic — code reading classification attributes at runtime to decide behavior has moved compliance policy out of the seam and into call sites.
- The attribute carries a `Notes` channel — the review rationale lives on the annotation itself, beside the data it judges, instead of in an external register that drifts.
- `DataClassificationSet` equality is set equality with a cached hash — two sets with the same members are one key regardless of construction order, which is what makes the frozen redactor map deterministic across composition refactors.
- Taxonomy design heuristics that survive contact with real suites: classes partition by disclosure consequence, not by data type (an identifier and a free-text field with the same consequence share a class); the class count stays single-digit (every class is a redactor row and an audit row); and a class whose members would all map to pass-through is `None` by another name — delete it.
- Generic-parameter classification is the deep cut: a generic carrier annotated at its type parameter classifies whatever flows through it, so wrapper and envelope types propagate sensitivity without per-instantiation annotation — classify the carrier once, every payload inherits the verdict.

## redactor-algebra

- `Redactor` is a span-first two-phase contract: `GetRedactedLength(source)` sizes, `Redact(ReadOnlySpan<char>, Span<char>)` writes; string overloads derive from the span core via exact-size allocation.
- The generic overload routes any span-formattable value through a fixed stack buffer into redaction without materializing the cleartext as a string — sensitive values flow from typed field to redacted output with no intermediate heap copy for a dump or profiler to find.
- Empty input yields empty output by contract — redactors never invent placeholder text for absent values.
- The package ships exactly three redactor rows and they partition the disclosure spectrum: erasure (`ErasingRedactor.Instance` — zero-length output; only the value's existence survives), pseudonymization (`HmacRedactor` — keyed one-way token), pass-through (`NullRedactor` — the None row).
- Anything beyond the three — masking, truncation, format-preserving substitution — is a custom `Redactor` subclass admitted as a row; inventing it inline at a sink is the rejected form.
- The span-formattable fast path runs under a fixed stack budget — values formatting within it redact with zero heap traffic; longer values take the allocating path, so high-volume classified values that format compactly are effectively free to redact.
- The choice per classification is a disclosure-utility trade declared once: erase what nobody joins on, HMAC what operations must correlate, pass what review cleared.
- `HmacRedactor` mechanics define its guarantees: HMAC-SHA256 over the value, first 16 hash bytes emitted as 24 base64 chars, optional `keyId:` prefix; output length is constant per configuration — redacted output also length-hides.
- The key is supplied as base64 with a 44-char (32-byte) minimum enforced by options validation; `SetHmacRedactor` accepts a configurator or a configuration section, so key material binds from secret stores, never source.
- Equal inputs under one key yield equal tokens — joins and distinct-counts survive redaction — while the keyed hash blocks dictionary reversal without the key.
- `KeyId` makes rotation epochs visible in the output itself: tokens from different key generations are distinguishable, so a join silently degrading across a rotation boundary is diagnosable from the data alone.
- The epoch prefix is a partition marker, not a strippable decoration: joining tokens across epochs by discarding the prefix is wrong by construction — different epochs are disjoint pseudonym spaces, and the correct cross-epoch question is "same epoch?" before "same token?".
- The HMAC correlation property is double-edged and pairs with the discriminator law: stable tokens enable exactly the cross-record correlation erasure prevents — admit HMAC only where that correlation is the requirement.
- The generic redaction overload accepts format and format-provider arguments — a formattable value redacts under an explicit format, so the redacted token is stable across call sites only when the format is also stable; format drift forks pseudonyms for one value.
- HMAC options are validated at startup by a dedicated options validator — a malformed or short key is a boot failure, never a runtime surprise mid-stream; this is the pattern every custom redactor's options should copy.
- The fallback stays erasure by law: `SetFallbackRedactor<T>` can replace it, but a non-erasing fallback (pass-through or pseudonymizing) converts every future taxonomy gap from silent destruction into silent disclosure — uncovered sets get explicit rows; the fallback is the safety net, not a policy slot.
- No fast-hash redactor ships — pseudonymization is keyed HMAC or nothing; a performance-motivated unkeyed hash is reversible by dictionary and must not be admitted as a custom row without a named disclosure argument that survives that attack.

## provider-and-enforcement-seam

- `AddRedaction(builder)` is the single registration root: `SetRedactor<T>(params DataClassificationSet[])` maps redactor rows to classification sets; `SetFallbackRedactor<T>` declares the no-match row; resolution is `IRedactorProvider.GetRedactor(DataClassificationSet)` over a frozen map.
- `SetHmacRedactor` is the one builder verb that refuses inline secrets gracefully: its configuration-section overload binds key material from the secret store path, and the delegate overload exists for composition that resolves keys through its own typed options — both shapes keep the key out of the registration line.
- One redactor type may serve many sets (`SetRedactor<T>` takes the sets as params), so the row count tracks the taxonomy, not the redactor inventory — three shipped redactors plus rare custom rows cover taxonomies of any width.
- The fallback defaults to erasure — an unmapped classification destroys data rather than leaking it; fail-closed is the constructed default, not a configuration achievement.
- The map binds classification sets to redactor TYPES and instances resolve from the container — redactors are options-bound, constructor-injected services, so one redactor type can serve multiple sets under one configuration and a redactor needing key material declares it as options like any service.
- The parameterless `AddRedaction()` is a valid bootstrap state: infrastructure present, zero rows — everything classified erases via the fallback; a suite can therefore turn enforcement on before the taxonomy mapping is finished and tighten by adding rows, never by loosening a default.
- Map lookup is exact-set match, not subset match: a value classified `{A, B}` does not resolve the redactor registered for `{A}`; it falls to the fallback eraser.
- Every classification set the suite can produce — including unions from multi-classified members — therefore needs its own explicit row; a missing combination row manifests as silent erasure, not as an error.
- Definition-time enforcement has exactly one seam: classified members annotated at the type, the telemetry generator separating classified tags from plain tags at compile time, and `EnableRedaction` on the logging builder activating redaction before any provider or exporter observes the record.
- Everything downstream of the seam — providers, sinks, exporters, stores — receives post-redaction data and never re-redacts; a second masking pass anywhere downstream is the named defect: it implies the seam is untrusted, and it operates on rendered text where the seam operated on typed values.
- Stores' role is admission verification — rejecting unclassified payloads as a typed rejection — never transformation; verification and enforcement are different verbs at different altitudes.
- `LoggerRedactionOptions.ApplyDiscriminator` defaults true: the tag name is folded into the redacted value, so identical cleartext under two different tags yields two different tokens — cross-tag correlation is off by default.
- The discriminator is a pseudonym-only concern: erasure output is empty regardless, so the discriminator verdict is decided with and recorded beside the HMAC rows it actually affects.
- Redactor resolution is a frozen-map lookup — hot-path cheap — and emission sites that dominate one classification set may hold the resolved redactor; the provider is the policy authority, not a per-call ceremony.
- Re-enabling cross-tag correlation (for deliberate pseudonym joins across tags) is a recorded governance decision, not a tuning knob — the default exists to make linkability opt-in.
- Flipping the discriminator mid-retention mixes token spaces: records before and after the flip pseudonymize the same value differently, so the verdict is set once per suite and a change is treated like a key rotation — an epoch boundary, not a toggle.
- Exporter governance closes the loop at the wire: route parameters carry their own redaction mode policy (`HttpRouteParameterRedactionMode`) so path segments classified as identifying are masked in telemetry route tags.
- Outbound-HTTP instrumentation redacts URL query values by default with an explicit environment opt-out — the default-redacted posture extends to instrumentation-generated telemetry, not just first-party tags.
- The governance table for a process is five auditable rows: classified-tag redaction active, discriminator verdict, route-parameter mode, query-redaction verdict, classification→redactor map — the full compliance posture without reading code.
- The seam's compile-time half makes violations cheap to find: classified data enters telemetry only through generated methods that route it into the classified array, so the audit surface is a grep — no string interpolation of classified-typed values, no `ToString` on classification-annotated members at emission sites — plus a boot assertion that redaction is enabled.
- The `None` row is auto-registered to pass-through when not explicitly mapped — reviewed-public data flows unredacted without a declaration, which is correct and also why `None` annotations deserve the same review rigor as sensitive ones: `None` is the only classification that disables the seam for a value.
- Redaction applies where classification metadata exists — the typed tag path through the generator; payloads that bypass the generated path (hand-built state objects, foreign provider records) carry no classification and therefore no redaction, which is the structural argument for the generated-emission monopoly rather than a redaction-specific rule.
- The redaction seam exists only on the log path — spans, measurement tags, latency tags, and baggage have no redactor in their pipelines; their governance is prevention by review: classified values are barred from span enrich delegates, instrument tag vocabularies, latency tag registrations, and the baggage envelope schema.
- The seamless-surface inventory is the enforcement law's completeness proof: one redacting seam (logs) plus four prevention rows (spans, metrics, latency, baggage) covers every emission surface a process has — an enforcement design that stops at the log seam governs a fraction of the exfiltration surface and calls it whole.
- Span-side prevention has a precise location: enrich delegates and processors are the only code that writes span tags from payload data, so the review boundary is those declared rows — small, enumerable, and root-located by the interception law.
- `EnableRedaction` and `AddRedaction` both bind from configuration sections as well as code — the posture is environment-diffable like every other pipeline policy.
- Repeated `AddRedaction` calls compose builder actions over one idempotently registered provider — modular composition roots can each contribute rows without double-registering the seam.
- Projection surfaces beyond live telemetry inherit the same seam: retained snapshots, support bundles, and receipt projections consume the classification verdicts and redactor rows — redaction class is part of the projection contract, and pseudonymizing projections declare key identity and output-length policy in their own receipts so a bundle's redaction posture is self-describing.
- Unclassified persisted data is rejected before it reaches a support bundle — the projection-side twin of the store-admission verification, and the second place the Unknown/None distinction does load-bearing work.
- Erasure leaves structural presence: the field exists, the content is gone — bundles and snapshots remain schema-debuggable (shape questions answerable) even where content is destroyed, which is the operational argument for erasure over field deletion.
- Query semantics inherit the distinction: an empty classified field reads "erased", an absent field reads "never emitted" — views that conflate the two cannot distinguish a redaction working from an emission missing, which are opposite findings.
- Cross-suite bundle exchange carries provenance: a bundle records the taxonomy name and key epoch its redactions were produced under, so a receiving suite interprets classifications and pseudonyms against the producing posture instead of assuming its own.
- The wire exporter serializes post-seam arrays untouched — redaction is wire-format independent by construction, and no exporter, codec, or transport choice can weaken it, because the cleartext never reaches the serialization layer.

## classified-data-lifecycle

- The classified value's lifecycle is five owned stages — annotated at declaration, separated at generation, redacted at the enforcement seam, projected to exporters and bundles, retained under epoch policy — and each stage has exactly one owner; a compliance question is answered by naming the stage, not by reading the flow.
- The annotated contract types double as the data inventory: the catalogue of what sensitive data the suite processes, where it is declared, and how it is disclosed is generated from the attributes — an inventory maintained in parallel documents is the drifting duplicate the annotation design exists to delete.
- Secrets are not a classification: credential material never enters telemetry at all — it is barred at the emission grammar (no log method parameter may be secret-typed), because a redacted secret is still a secret that transited the pipeline buffers; classification governs data that legitimately flows, secrecy governs data that must not.
- Redactors are stateless singletons by contract — resolved once per classification set, shared across threads, all state in options; a redactor holding mutable per-call state breaks under the pipeline's concurrency silently.
- The two-phase span contract (`GetRedactedLength` then `Redact`) exists so callers write into exact-size or pooled buffers — custom redactors preserve the property that length depends only on configuration and input length, never on input content, or they leak information through output size.
- The destination contract is enforced: an undersized destination buffer throws at the redactor, never truncates — partial redaction output is unrepresentable by construction.
- Null sources are empty sources: the string overloads project null through the empty-span path, so a nullable classified member never needs a null guard at the emission site.
- Key-epoch allocation is monotone bookkeeping: `KeyId` values increment per rotation and every pseudonymizing projection records the epoch it used — the rotation registry is one integer sequence plus dates, and its absence is unanswerable "which key produced this token".
- The correlation algebra under HMAC plus discriminator is exactly three rows: same tag and same key epoch — joinable; different tag — severed by the discriminator; different key epoch — severed by rotation. Every privacy review of a pseudonymized dataset reduces to those three rows plus the rotation cadence.
- Audit cadence binds to taxonomy edits: the reachable-set diff, the canary boot proof, and the uncovered-set scan run on every change to an annotation or a redactor row — the cheap time to find a gap is the commit that creates it.
- Redaction precedes every volume policy by construction: classified values redact at generated-method invocation, before sampling, buffering, or batching see the record — a buffered record holds redacted values, so an incident flush can never resurrect cleartext that emission already destroyed.
- HMAC output is base64 text — it survives every text sink, JSON encoder, and query language without escaping interactions, which is a quiet reason the pseudonym row composes with arbitrary projection backends while binary-token schemes do not.
- Enrichment dimensions and classified data are disjoint populations by review: enricher rows emit bounded operational dimensions and are unclassified by definition — a fact that would need classification has no business as an enrichment dimension, and the enricher review enforces that boundary, not the redactor.
- Foreign-process data is unclassified at arrival regardless of its origin suite's posture: relays and aggregators re-classify by mapping foreign fields onto the local taxonomy before re-emission — forwarding foreign payload tags blind launders unclassified data through a classified pipeline.

## resource-signals-governed

- Resource monitoring is admitted as a governed signal source, never a decision-maker: `AddResourceMonitoring()` registers the monitor, snapshot provider, and publisher fan-out; `AddPublisher<T>` admits publisher rows; publishers receive utilization via `PublishAsync` with cancellation.
- Produced values are read-only evidence — throttling, shedding, and degradation decisions belong to the owning policy machines that consume the signal; the machines themselves are composed, not legislated here.
- Two consumption bridges exist and they serve different consumer kinds: the meter feeds the metric pipeline (dashboards, alerting, export), publisher rows feed in-process consumers (health cells, degradation policy) — choosing the bridge by consumer kind keeps the metric pipeline out of control loops and control loops off the export cadence.
- The container/process instrument split tracks deployment class: container rows are meaningful only under control-group or job limits, process rows always — dashboards bind per deployment class, and a bare-process fleet alerting on container rows alerts on absence.
- Utilization percentages are limit-relative, not host-relative: a full reading means the declared resource envelope is exhausted, not the machine — host headroom beyond the limit is invisible by design, because the limit is what the orchestrator enforces.
- The platform snapshot layer is sealed behind its contract — the extension surface is publishers and options, never snapshot-source replacement; a deployment needing a foreign resource signal admits it as its own meter, not as a forged snapshot provider.
- Utilization is delivered as a value type — publishers receive immutable evidence by value, and holding the latest reading is a copy, never a live reference that mutates under the consumer.
- Consuming the obsolescence-marked pull surface emits a build diagnostic — migration pressure is compile-time visible, so a composition still on the windowed API is a warning in the build log, not an archaeology finding.
- The canonical consumption route is the instrument surface on the meter `Microsoft.Extensions.Diagnostics.ResourceMonitoring`; the windowed pull API (`IResourceMonitor.GetUtilization(window)`) and its window/interval option rows carry obsolescence markers steering to the instruments.
- New composition admits the meter into the metric pipeline and treats the pull API as a legacy seam — catalogue surfaces describing the pull API as primary are stale against the installed package.
- Process instrument rows: `process.cpu.utilization`, `dotnet.process.memory.virtual.utilization`.
- Container instrument rows: `container.cpu.time`, `container.cpu.limit.utilization`, `container.cpu.request.utilization`, `container.memory.limit.utilization`, `container.memory.request.utilization`, `container.memory.usage`.
- Flag-gated system rows: `system.disk.io`, `system.disk.io_time`, `system.disk.operations` (one enable flag), and `system.network.connections` (platform-scoped; connection tracking filters by a declared source-address set).
- The request/limit split is first-class and is the correct alarm axis: guaranteed units model the orchestrator request (1000 CPU units per core), maximum units model the limit, and the container instruments emit utilization against each separately.
- The same split exists for memory — guaranteed bytes versus maximum bytes ride the limits record — so memory alarms get the same two-axis treatment as CPU, and a single "memory percent" alarm is the same collapsed mistake.
- Admitting the monitoring meter into the signal pipeline is one admission row at the signal root — the governed-source pattern in one line: the source registers its instruments, the root admits the meter, governance rows shape the rest.
- Publishers are cadence-coupled consumers: delivery awaits each publisher with cancellation, so a slow publisher stretches the publishing loop for all — publishers record and return; anything slow queues internally behind its own lane.
- Alerting on limit-utilization detects imminent throttling and eviction; alerting on request-utilization detects under-provisioning — collapsing the two into one "CPU percent" loses exactly the distinction the orchestrator acts on.
- Metric range is a declared convention with a live migration trap: utilization emits [0,1] under the platform-specific flag (default on for its platform) but [0,100] under the general default — and the general default is documented as destined to flip.
- A fleet pins one range explicitly in options and binds dashboards to it; mixed ranges across platforms in one fleet is the silent hundred-fold alerting error.
- A cgroup-v2 calculation flag exists for container CPU accounting — container platforms with v2 hierarchies flip it; the default remains the legacy calculation.
- Cadence rows (collection window 5s, sampling interval 1s, refresh 5s) bound the staleness of every derived signal: a health policy reading utilization can never react faster than the sampling interval.
- A publisher window wider than the consuming policy's evaluation period aliases — declare the monitoring cadence and the consuming machine's period together.
- Utilization values are classification-clean — no payload, no identity — so the resource lane needs no redaction row; its governance is range, cadence, and consumption-route discipline.
- That asymmetry is why one page owns both concerns without the lanes touching: redaction governs what leaves, resource governance governs what is measured — both are exporter-side governance tables.
- The legacy pull model's value shape survives in publishers: utilization carries CPU and memory percentages, used bytes, the limits record, and the latest raw snapshot (kernel/user/total time since start plus memory bytes) — windowed utilization folds buffered snapshots, so a one-second spike inside a five-second window reports diluted; spike-sensitive policy needs the instruments at their native cadence, not the windowed fold.
- The kernel/user time split in the snapshot is a free diagnosis axis: kernel-heavy saturation points at syscall and I/O pressure, user-heavy at compute — one ratio separates the two standard CPU-saturation incident classes before any profiler attaches.
- The legacy knobs deprecate as a family — collection window, sampling interval, publishing window, both refresh intervals — all steering to the instrument surface together; a composition migrating one knob at a time is half on each model and inherits both staleness semantics at once.
- The disk-metrics enable flag was renamed once already (its own contract notes the predecessor) — composition binding these flags from configuration strings must track the current name; a stale key is silently false, and silently-false enable flags are absent dashboards, not errors.
- Several monitoring options are experimental-gated behind compile-time diagnostics — opting in is a recorded suppression at the composition file, which keeps "we depend on a pre-stable knob" greppable when the knob stabilizes or moves.
- Network-connection tracking filters by a declared source-address set — the one monitoring row that carries address-shaped configuration; addresses are deployment material bound from configuration, never literals in composition code.

## divergent

- Taxonomy-design maximal form: the taxonomy is a policy table with one row per sensitivity class — attribute type, classification value, redactor row, disclosure rationale, store-admission verdict — and the entire compliance posture of a suite is recoverable from that table plus the five-row exporter governance table.
- Growth is row-shaped: a new sensitivity class is one attribute subclass plus one `SetRedactor` row; a new combination of sensitivities is one exact-set row; a new wire surface inherits the posture without new code.
- Forms the table forecloses, each with its damage: per-sink scrub lists (drift per sink), regex masking middleware (operates on rendered text, misses structured paths), "sanitize" helpers (policy hidden in call graphs), serializer-buried redaction converters (couples compliance to wire format) — each is policy invisible to the table and therefore unauditable.
- Exact-set semantics as a design forcing function: because `{A, B}` does not inherit `{A}`'s redactor, multi-classified members force an explicit decision per combination — the API turns "which policy wins when sensitivities stack" into a declared row instead of an implicit precedence rule.
- The audit procedure that falls out is static: enumerate the classification sets reachable from annotated members (a walk over attributes), diff against registered rows, and treat uncovered sets as defects even though the runtime would only erase.
- Silent erasure of a value operations needed is an availability bug with a compliance-shaped cause — the uncovered-set audit catches it at review, not during an incident.
- Pseudonym lifecycle law: HMAC tokens are stable per key epoch, so the suite's joinability horizon equals its key-rotation period — rotation severs longitudinal joins by design.
- The `KeyId` prefix is what lets retention policy reason about token epochs (drop epochs older than the key's retirement); declaring rotation cadence beside the redactor row turns "how long can redacted data be correlated" into a read-off fact.
- Omitting the rotation declaration leaves an unbounded pseudonym horizon — which is itself a compliance posture, just an undeclared one; the table format makes the omission visible.
- Two-sided verification of the seam: forward, a canary value classified at each taxonomy row is logged at boot and the exported record asserted to carry the redacted form — one check catches a missing enable call, a missing redactor row, and a provider bypass simultaneously.
- Backward, store admission rejecting an unclassified payload proves the verification half without any re-redaction machinery — the two checks compose into the suite's startup proof and make "is redaction actually on, everywhere" a boot-time fact rather than an incident finding.
- Disclosure-spectrum completeness check: every taxonomy row maps to exactly one of {erase, pseudonymize, pass, custom-row}; a row whose rationale cannot name which of the four it needs is an unfinished classification, and a custom redactor without a named disclosure property (what survives, what is destroyed) is a masking superstition.
- The seam's deepest property is that it is the ONLY place compliance and observability meet: everything upstream is type design (annotations on owners), everything downstream is plumbing (providers consuming arrays) — which means compliance posture migrates with the types when code moves between processes, and a new process inherits a correct posture by referencing the contract assemblies; per-process compliance configuration reduces to the map and the five governance rows.
- Posture versions with contracts: because annotations live on shared contract types, a compliance change is a contract-assembly release — every consuming process updates posture by reference bump, and a fleet's posture skew is readable from its dependency graph rather than from per-process configuration archaeology.
- The redactor map is the one per-process half: contracts carry classifications, processes carry redactor verdicts — splitting it this way is deliberate, because what data IS does not vary by process while what a process may DISCLOSE does (an operator console and a headless worker legitimately map one taxonomy to different redactor rows).
- Resource governance generalizes the same shape one level up: measured-source admission (which instruments exist), range convention, cadence rows, and consumption bridges are to machine evidence what taxonomy, redactor map, discriminator, and exporter rows are to human-data evidence — one page, two governed source families, one table grammar.
- The governed-source pattern is the template for every future machine-evidence admission: a source registers typed instruments under one meter, the root admits by name, governance declares range, cadence, and bridge rows — ad-hoc polling loops against platform counters are the form this retires, and the retirement is checkable as "no timer reads a system counter outside a registered source".
- The page's whole compliance surface folds to three declared tables — taxonomy rows, per-process redactor map plus governance rows, rotation registry — and one inventory of prevention rows for the seamless surfaces; anything claiming to be compliance policy that lives outside those tables is either a duplicate or a bypass.
