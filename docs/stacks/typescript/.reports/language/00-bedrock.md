# language — bedrock (TS 6.0.3 / TS7-native type discipline)

`[TS6_TO_TS7_SURFACE]:`
- TS7 = native (Go) compiler, beta enabled-by-default in VS 2026 18.6 (April 2026); a port, not a language fork — no authored-code type-checking semantic deltas claimed; gains are speed (~10x build, ~8x LS load) and LSP-based editor service. Treat authored-code law as TS6-stable; the only authoring-relevant TS7 fact is "type-checking is a port, write to TS6 semantics."
- TS6 is the final JS-based release; defaults flipped to `strict:true`, `noUncheckedSideEffectImports:true`, `module:esnext`, lowest target ES2015 (`target:es5` deprecated, `--downlevelIteration` deprecation-errors), `moduleResolution: node10/classic` deprecated/removed (use `nodenext`/`bundler`), `types:[]` default (no auto `@types` pull), `esModuleInterop`/`allowSyntheticDefaultImports` cannot be `false`. `--stableTypeOrdering` exists to pre-align with TS7 type ordering.
- `erasableSyntaxOnly` errors on: `enum` (regular AND `const enum` — both require non-erasable emit), `namespace`/`module` containing runtime code, class constructor parameter properties (`constructor(public x)`), non-ECMAScript `import x = require(...)` / `import X = NS.Y` aliases, `export =`. PERMITTED: ambient `declare enum`, `declare namespace`, `declare module` (type-only, zero emit) — usable for typing untyped JS at the boundary.
- Doctrine replacements: enum → `as const` object + `(typeof X)[keyof typeof X]` value-union, or Effect `Schema.Literal(...)` / `Data.taggedEnum` when a runtime shape is owned; namespace → ES module file or merged `interface`/`type` for type-only grouping; parameter properties → explicit field decls + assignment (or skip classes entirely for `Data`/`Schema` records); `import =` → ESM `import` + `package.json#imports` (`#/`) subpaths (TS6 adds `#/` prefix support); `export =` → named/default ESM export. These compose with `verbatimModuleSyntax` (`import type`/`export type` mandatory for type-only) and `isolatedModules` (no cross-file const-enum/`export =` re-export).

`[ADVANCED_TYPE_TOOLKIT_WHEN]:`
- `const T` type parameters: capture literal/tuple shape without caller `as const`. Earn use on generic builders that must preserve literal keys/tuple order (dispatch-table keys, fluent spec builders). Redundant where the value already flows through `as const satisfies`.
- `NoInfer<T>` (intrinsic, present in lib.es5): block a type param from being inferred at one position so another position drives it. Earn use on `(table: Record<K, H>, key: NoInfer<K>)`-style APIs and default-value params that must not widen the param. Native — never reach for a library polyfill.
- `satisfies`: validate a value against a constraint while keeping its narrow literal type. `as const satisfies T` is the canonical totality+literal idiom: `{ ... } as const satisfies Record<Tag, Handler>` proves every `Tag` is covered AND keeps per-key narrowing. Prefer over `: T` annotation (which widens) and over `as T` (which lies).
- Variance annotations `in`/`out`/`in out`: explicit, checked variance on type params; surfaces structural mistakes at declaration and speeds checking. Effect's own surfaces use them (`Effect<out A, out E, out R>`, `Schema<in out A, in out I, out R>`, `Brand<in out K>`, `Tag<in out Id, in out Value>`). Earn use on any phantom/branded carrier and any HKT-shaped interface; annotate to match the semantic role (covariant outputs `out`, invariant identity `in out`).
- Template literal types + intrinsic `Uppercase`/`Lowercase`/`Capitalize`/`Uncapitalize` (all `intrinsic` in lib.es5): typed string construction/parsing (route patterns, event names, key prefixes). Intrinsics do single-token casing only — delimiter-aware multi-word casing is not native.
- Key remapping in mapped types `as`: rename/filter keys (`[K in keyof T as `on${Capitalize<K>}`]`), drop keys via `as never`. Earn use when deriving a surface (event handlers, getters) from a shape — the alternative is a parallel type (forbidden).
- Conditional types + `infer` with `extends` constraints (`infer R extends string`): constrained extraction; the constraint narrows `R` without a second conditional. Distributive over naked union type params (`T extends U ? ... : ...` distributes when `T` is a bare param); suppress distribution with tuple wrapping `[T] extends [U]` — Effect uses `[T] extends [Effect<...>]` exactly for this. Law: wrap in `[...]` whenever the union should be tested as a whole (never-collapse, exhaustiveness).
- Tuple types: labeled (`[first: A, ...rest: B[]]`) for self-documenting positional APIs; variadic (`[...A, ...B]`) for typed concat/curry; optional elements (`[A, B?]`) for arity modality. Prefer a single variadic signature over overload stacks for concat/pipe shapes.
- Recursive type aliases: legal for self-referential trees/paths; TS caps instantiation depth (~50 nesting / ~100 instantiations) and requires tail-recursion-eliminable form for deep accumulation. Bound recursion with a depth-counter tuple param when walking deep paths; prefer Effect `Schema.suspend` for recursive runtime shapes rather than a parallel recursive type.
- `unique symbol` branding: the nominal primitive. Effect `Brand` is `A & Brand<K>` where `Brand<K>` carries `readonly [BrandTypeId]: { [k in K]: K }` over a `unique symbol`. Use Effect `Brand`/`Brand.nominal`/`Brand.refined` (or `Schema.brand`) — do NOT hand-roll a second `unique symbol` brand.
- Phantom type parameters: a param used only in types (carried via variance helpers `(_: never) => A` etc., as in `effect/Types` `Covariant`/`Contravariant`/`Invariant`). Earn use for compile-time state machines / units-of-measure that have no runtime field.
- F-bounded polymorphism (`<T extends Self<T>>`): self-referential constraints for fluent/recursive builders and comparable hierarchies. Use sparingly — Effect `Pipeable` already gives method chaining without F-bounds.
- Overloads vs union-discriminated single signature: prefer ONE signature discriminating on input shape (polymorphic entry) over `get`/`getMany` overload proliferation. Reach for true overloads only when return type is a non-uniform function of arg arity that a single conditional return cannot express cleanly (Effect `dual` is the canonical data-first/data-last two-overload pattern). The modal-arity answer: single variadic/optional-tuple signature first; overloads only for irreducible return-shape divergence.
- Assertion functions `asserts x is T` / type predicates `x is T`: narrowing at boundaries. Predicates for filters/guards that return `boolean`; `asserts` for invariant checks that throw. In domain code prefer Effect `Schema.is`/`Schema.asserts`/`Schema.decodeUnknown` (owns both predicate and refinement) over hand-written guards.
- `using`/`await using` + `Symbol.dispose`/`Symbol.asyncDispose`: typed via `Disposable`/`AsyncDisposable`/`DisposableStack` (lib.esnext.disposable, present in 6.0.3). For Effect code, `Effect.acquireRelease`/`Scope` own resource lifetime — reserve `using` for non-Effect host-API scopes at the boundary.

`[EFFECT_TYPE_MACHINERY_COMPOSE]:`
- HKT (`effect/HKT`): `interface TypeLambda { In; Out2; Out1; Target }` + `Kind<F, In, Out2, Out1, Target>`; defunctionalized higher-kinded encoding via `URI` `unique symbol` on `TypeClass<F>`. Author typeclass-generic code by declaring a `TypeLambda` and projecting through `Kind` — never re-encode HKTs.
- `Effect<out A, out E, out R>`: tri-variant (success/error/requirements), all covariant; variance witnessed by `Effect.Variance` `[EffectTypeId]: { _A; _E; _R }` (each `Covariant`). Extract via `Effect.Success<T>`/`Effect.Error<T>`/`Effect.Context<T>` (all use `[T] extends [Effect<...>]` non-distributive form). Mirror this extractor idiom for any custom three-channel carrier.
- `Context.Tag<in out Id, in out Value>`: invariant identity + service; `Tag.Identifier<T>`/`Tag.Service<T>` extractors; `TagClass<Self, Id extends string, Type>` is the `class X extends Context.Tag("Id")<X, Shape>()` pattern giving a nominal `Self` identifier. Use `Tag`/`TagClass` for all service identity — the `Self`-as-identifier trick is the canonical phantom-identity use.
- `Schema<in out A, in out I = A, out R = never>` (Type=A, Encoded=I, Context=R): extract with `Schema.Schema.Type<typeof X>`, `Schema.Schema.Encoded<typeof X>`, `Schema.Schema.Context<typeof X>` (each matches `Schema.Variance<infer A, infer I, infer R>`). LAW: derive the static type FROM the schema (`type T = Schema.Schema.Type<typeof T>`); never declare a parallel `interface`/`type` beside a schema. `Struct.Type`/`Struct.Encoded` show field-level derivation via `UnionToIntersection`.
- `Brand`: `Branded<A,K> = A & Brand<K>`; constructors `Brand.nominal<T>()` (no validation) / `Brand.refined<T>(pred, onFail)` (validated, returns `Brand.Constructor` with `.option`/`.either`/`.is`). `Brand.Unbranded<P>` strips brands. This is THE nominal layer — type-fest `Tagged`/`Opaque`, ts-essentials `Opaque`/`Newtype`/`Branded`/`Nominal`/`Token`, ts-toolbelt `A.Type` are all structurally-disjoint competing brands; never mix.
- `Pipeable`: 1–N overloaded `.pipe(...)` method interface; compose by extending `Pipeable`, never reimplement the overload ladder. `dual<DataLast, DataFirst>(arity|predicate, impl)` is the standard for exposing both styles from one symbol.
- `effect/Types` ships the reusable type-level primitives: `Simplify`, `Equals`/`EqualsWith` (the `(<T>()=>T extends X?1:2)` identity test), `UnionToIntersection`, `Mutable`, `Tags`, `Covariant`/`Contravariant`/`Invariant` variance witnesses. Prefer these over external libs for the overlapping cases.

`[EXHAUSTIVENESS_LAW]:`
- `never`-check (`const _: never = x` in the `default`/else arm): zero-dependency, works on any discriminated union; failure is a compile error at the unhandled member. Cost: imperative `switch` shape, one dead binding per site.
- `Match.exhaustive` (`effect/Match`): build `Match.type<U>()`/`Match.value(v)` → `Match.tag`/`Match.when`/`Match.not` → `Match.exhaustive`; the finalizer type-errors if any case is unhandled and erases the fallback. Composes via `pipe`, returns a value, integrates with `Data.taggedEnum` (`$match`). Capability edge: predicate/`not`/nested-shape matching, not just tag equality.
- `as const satisfies Record<Tag, Handler>` dispatch table: a data-table totality proof — missing key = compile error, extra key = compile error, per-key handler narrowing preserved; dispatch is `table[tag]` (under `noUncheckedIndexedAccess`, indexing a `Record<Tag,H>` by a `Tag`-typed key is NOT widened to `H | undefined` because the key is a known finite union, but a `string` key IS). This is the densest exhaustiveness form for pure value→value dispatch.
- `Data.TaggedEnum`'s generated `$match`/`$is` (`TaggedEnum.GenericMatchers`) give exhaustiveness with `[K in Exclude<keyof Cases, _tag>]: never` rejecting unknown arms — totality without a separate Match pipeline. Capability ranking: table-`satisfies` (densest, pure dispatch) ⊂ `$match`/`Match.exhaustive` (effectful arms, predicate matching) ⊃ `never`-check (lowest-level fallback, no value return). Pick table-satisfies for record dispatch, `Match`/`$match` for effectful or predicate-shaped dispatch; reserve `never`-checks for boundary `switch` the others cannot reach.

`[HELPER_LIB_VERDICT_MATERIAL]:`
- All three are 100% type-only `.d.ts`, fully erased under `isolatedModules`/`verbatimModuleSyntax` (import via `import type`); zero runtime/bundle cost. Hazards are checker-correctness, version co-evolution, and conceptual duplication — not emit.
- type-fest 5.7 (active, ~monthly; TS≥5.9, ESM-only): unique value over native = `Simplify`/`SimplifyDeep` (no native display-flatten), delimiter-aware case transforms `CamelCase`/`KebabCase`/`*CasedPropertiesDeep`, lens types `Paths`/`Get`, key-modifier algebra `SetOptional`/`SetRequired`/`SetFieldType`, cardinality `RequireAtLeastOne`/`RequireExactlyOne`, type-level arithmetic `Sum`/`IntRange`, `OptionalKeysOf`/`ReadonlyKeysOf` introspection, `Jsonify`. Defers to native intrinsics (does not re-ship `Uppercase`/`NoInfer`/`Awaited`). REDUNDANT here: `Tagged`/`Opaque` (Effect `Brand`), `PartialDeep`/`ReadonlyDeep`/`Jsonify`/`Simplify`-of-schema (Effect `Schema`/`Data` own the shape; `Schema.parseJson`/`Schema.Json` own JSON). Brand clash: type-fest `tag` symbol ≠ Effect `BrandTypeId`.
- ts-essentials 10.2 (active, low cadence; peer `typescript>=4.5` — NOT co-evolved with TS7): unique value = `StrictOmit`/`StrictExclude`/`StrictExtract` (native equivalents still lax in TS6/7), `DeepReadonly` with per-builtin passthrough config, path-targeted `DeepMarkRequired`/`StrictDeepPick`. REDUNDANT: ships FIVE parallel brand encodings (`Opaque`/`Newtype`/`Branded`/`Nominal`/`Token`) all superseded by Effect `Brand`; deep transforms redundant where Schema owns shape. Hazards: highest brand-collision surface; within-minor name churn (`Exact`→`IsExact`, `NonNever`→`OmitNeverProperties` in 10.2.0); lax peer floor signals possible deep-recursion breakage under TS6/7 instantiation-depth rules.
- ts-toolbelt 9.6 (UNMAINTAINED — last release 2021-03; no TS6/7 remediation path): namespace-organized (`O.*`/`U.*`/`L.*`/`S.*`/`F.*`/`N.*`/`A.*`). Almost entirely superseded — native template literals replace `S.*`, Effect `pipe`/`flow` replace `F.Curry`/`F.Compose`/`F.Pipe`, `A.Compute`==`Simplify`, `A.Type` is a third incompatible brand. Heavy recursive types (`N.*` arithmetic, `O.P.*` deep paths) authored pre-TS-4.5 tail-recursion and are known to break/slow under modern instantiation-depth rules. No validation against `exactOptionalPropertyTypes`/`noUncheckedIndexedAccess`. Verdict material: discontinued; replaceable by native + Effect + type-fest.
- Six+ mutually-incompatible nominal systems exist across these libs (type-fest `tag`, ts-essentials ×5, ts-toolbelt `A.Type`, Effect `BrandTypeId`); a single-brand law (Effect `Brand`/`Schema.brand` only) is required. `Simplify`/`Prettify`/`Compute` is the one capability with NO native TS6/7 equivalent in any lib — `effect/Types.Simplify` already supplies it in-ecosystem.

`[OPTIONAL_AND_INDEX_FLAG_IDIOMS]:`
- `exactOptionalPropertyTypes`: `{ x?: T }` means "absent OR T", NOT "T | undefined" — you cannot assign `undefined` to `x` unless the type is `x?: T | undefined`. Consequence: optional-property writes use spread-presence (`...(cond ? { x: v } : {})`) not `{ x: cond ? v : undefined }`. Reads: `"x" in obj` distinguishes absent from present-undefined.
- `noUncheckedIndexedAccess`: every index/record access yields `T | undefined` (`arr[i]`, `rec[k]` for `string`-typed `k`); finite-union-keyed `Record<Tag, H>` accessed by a `Tag`-typed key is NOT widened (so `satisfies`-driven dispatch tables stay `undefined`-free). `for...of`/`.entries()`/destructuring-with-default are the `undefined`-free traversal idioms; raw numeric indexing always needs an `undefined` discharge.
- DOMAIN ABSENCE = Effect `Option<T>`, NOT optional properties: model fields as `readonly x: Option.Option<T>` (always-present, `Option.none()`/`Option.some(v)`), reserve `?:` for genuine wire/partial-DTO boundary shapes only. Convert at the boundary: `Option.fromNullable(rec[k])` discharges the `noUncheckedIndexedAccess` `| undefined` directly into the rail; `Option.getOrElse`/`Option.match`/`Schema.OptionFromUndefinedOr`/`Schema.optionalWith({ as: "Option" })` bridge optional-property ⇄ `Option`. Net law: index/optional `undefined` lives only at the boundary line; it becomes `Option<T>` before entering domain code, and absence is never re-encoded as a parallel `?:` field inside the domain.

Further Considerations:
- `as const satisfies` + `noUncheckedIndexedAccess` interaction is the single highest-leverage law to state precisely: a finite-union-keyed dispatch table indexed by its own key type is provably `undefined`-free, which makes the satisfies-table the densest exhaustiveness primitive in the whole toolkit — but only if the doctrine forbids `string`-typed keys leaking into the index position. Worth a dedicated rule.
- The `effect/Types` module (`Simplify`, `Equals`, `UnionToIntersection`, `Mutable`, variance witnesses) makes a hard external-helper-lib dependency nearly unnecessary; the only residual type-fest pull is delimiter-aware case transforms and lens `Paths`/`Get`, both of which are boundary/codegen concerns rather than domain-type concerns — the doctrine can plausibly mandate zero type-helper deps in domain code and confine type-fest to a build/boundary tier.
- TS7 being a faithful port means the durable risk is not semantic drift but instantiation-depth/perf regressions in deep recursive types; pinning recursion-bounded patterns (depth-counter tuples, `Schema.suspend` for recursive shapes) future-proofs the doctrine against the exact class of breakage that already killed ts-toolbelt.
