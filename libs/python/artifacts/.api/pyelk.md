# [PY_ARTIFACTS_API_PYELK]

`pyelk` is the categorical-best PORTS + NESTING + ORTHOGONAL-ROUTING graph-layout engine for the diagram-layout rail: a pure-Python port of the Eclipse Layout Kernel exposing nine ELK layout providers (`layered`, `force`, `stress`, `mrtree`, `radial`, `rectpacking`, `sporeOverlap`, `sporeCompaction`, `fixed`) through ONE polymorphic `ELK().layout(graph)` boundary that consumes and returns the standard ELK-JSON graph document — a recursive `{id, children, edges, ports, labels, layoutOptions}` tree — and writes `x`/`y`/`width`/`height` onto every node, `x`/`y` onto every port, container extents onto every nested `children` group, and an `edges[].sections[]` route geometry (`startPoint`/`endPoint`/`bendPoints`) onto every edge. It is the one layout owner in the admitted roster that handles the three structural concerns `fast-sugiyama`/`grandalf`/`rustworkx` cannot: typed PORTS with side/index constraints, recursive HIERARCHICAL NESTING (a compound node laid out with its own algorithm, sized to enclose its children), and native ORTHOGONAL edge routing emitting per-edge bend-point waypoints. The package owner is the layered/orthogonal/ports/nesting placement provider behind `visualization/diagram/layout#LAYOUT`'s `HierarchyEngine.ELK` arm — the engine an ER diagram, a flowchart with record ports, a circuit/schematic, or a nested zone-and-program AEC diagram selects when edges must route orthogonally around boxes and connect to named ports rather than node centers. It owns coordinate assignment + edge-route geometry only: it emits no SVG (that is `visualization/diagram/draw#DRAW`'s), runs no graph analysis (the `data/graph#GRAPH` `rustworkx` analysis kernel stays at the data plane), and its synchronous Python layout pass is offloaded onto the `anyio.to_thread.run_sync` `CapacityLimiter` lane the layout owner already runs `_render` under, so the layout call never blocks the event loop.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: the ELK-JSON graph document (the one model — a plain `dict`, not a class)
- rail: diagram-layout

pyelk's model IS the ELK-JSON graph format: a plain recursive `dict`, never a class-based node/edge object graph (this is the deliberate design — the same JSON `elkjs`/upstream ELK consume, so a `rustworkx`/`polars` adjacency frame lowers to it by construction and the laid-out result reads back by key). `ELK.layout(graph: dict, ...) -> dict` takes a graph document and returns the SAME document shape with coordinates and route geometry written in place. The `__all__` surface is exactly `{ELK, ElkError, UnsupportedConfigurationException, UnsupportedGraphException, InvalidGraphException}`; the `dict` document keys below are the load-bearing contract a design page builds and reads.

| [INDEX] | [DOCUMENT_KEY]     | [KIND]      | [ROLE]                                                                                              |
| :-----: | :----------------- | :---------- | :-------------------------------------------------------------------------------------------------- |
|  [01]   | `id`               | str (req)   | unique element id (root/node/port/edge/label); `validate_id` enforces non-empty unique ids          |
|  [02]   | `children`         | list[dict]  | child NODES of a compound node; recursion is hierarchical nesting, parent sized to enclose children |
|  [03]   | `edges`            | list[dict]  | edges at this scope: `{id, sources, targets}`; hyperedge ID lists; endpoint may be a port id        |
|  [04]   | `ports`            | list[dict]  | typed PORTS `{id, width, height, layoutOptions}`; side/index under `elk.portConstraints`            |
|  [05]   | `labels`           | list[dict]  | text labels on a node/edge/port: `{text, width, height}`; placement via `elk.nodeLabels.placement`  |
|  [06]   | `width` / `height` | float       | INPUT leaf-node fixed size; OUTPUT computed container extent enclosing children + padding           |
|  [07]   | `x` / `y`          | float (out) | OUTPUT: assigned top-left coordinate relative to parent; absent on input leaf nodes                 |
|  [08]   | `layoutOptions`    | dict        | per-element overrides (`{"elk.algorithm": ..., "elk.direction": ...}`); override global + parent    |
|  [09]   | `edges[].sections` | list[dict]  | OUTPUT routed edge geometry: `{id, startPoint, endPoint, bendPoints}`; orthogonal waypoints         |
|  [10]   | `logging`          | dict (out)  | OUTPUT: present when `layout(..., logging=True)`; the per-run diagnostic trace                      |

[PUBLIC_TYPE_SCOPE]: the exception family (`pyelk.exceptions`)
- rail: diagram-layout

The four typed exceptions are the layout boundary's failure rail — every one descends from `ElkError`, so the layout owner's `async_boundary` traps `ElkError` as the one diagram-layout failure class (a malformed adjacency frame, an unsupported option combination, or an algorithm that rejects the graph shape), never a bare `Exception`. The boundary catches `ElkError`; the three leaf subclasses discriminate the cause for the receipt/diagnostic. This version's `validate_graph` checks id presence/shape, not duplicate ids, so the caller dedups upstream (keying off `str(node_index)` already guarantees uniqueness).

| [INDEX] | [TYPE]                              | [RAISED_WHEN]                                                                        |
| :-----: | :---------------------------------- | :----------------------------------------------------------------------------------- |
|  [01]   | `ElkError`                          | `Exception` subclass; base ELK layout error — the one class `async_boundary` traps   |
|  [02]   | `InvalidGraphException`             | malformed graph: bad ids or missing required fields (`validate_graph`/`validate_id`) |
|  [03]   | `UnsupportedGraphException`         | valid JSON but the selected algorithm cannot lay the shape out                       |
|  [04]   | `UnsupportedConfigurationException` | the `layoutOptions` combination is unsupported by the algorithm                      |

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: `ELK` — the one layout engine boundary
- rail: diagram-layout

`ELK` is the single layout engine; `ELK().layout(graph)` is the one polymorphic placement boundary — there is no `layout_layered`/`layout_force`/`layout_tree` method family. One `layout` call discriminates the algorithm off the graph's `elk.algorithm` option (or the constructor's `default_layout_options`), dispatches to the registered provider, and returns the positioned ELK-JSON document. The engine is constructed once (cheap — it only seeds default options) and reused across layouts; `default_layout_options` sets a global option floor every `layout` call inherits, and the per-call `layout_options` argument overlays it. `measure_execution_time=True` and `logging=True` are the two diagnostic switches. Option dicts are `dict[str, str] | None = None`; the full layout signature is `ELK.layout(graph, layout_options=None, logging=False, measure_execution_time=False) -> dict`; the three `known_*` methods return `list[dict]`.

| :-----: | :-------------------------------------------------- | :--------- | :---------------------------------------------------------------- |
|  [01]   | `ELK(default_layout_options=None, algorithms=None)` | construct  | the engine; `default_layout_options` sets the global option floor |
|  [02]   | `ELK.layout(graph, …) -> dict`                      | layout     | placement boundary; writes positions + edge `sections`            |
|  [03]   | `ELK.known_layout_algorithms()`                     | introspect | the nine providers as `{id, name}` rows                           |
|  [04]   | `ELK.known_layout_options()`                        | introspect | layout options as `{id, type}` rows                               |
|  [05]   | `ELK.known_layout_categories()`                     | introspect | category roster as `{id, name, knownLayouters}` rows              |

[ENTRYPOINT_SCOPE]: the nine layout providers (`elk.ALGORITHM_REGISTRY` / `ALGORITHM_ALIASES`)
- rail: diagram-layout

The `elk.ALGORITHM_REGISTRY` maps each fully-qualified `org.eclipse.elk.*` id to its provider CLASS; `elk.ALGORITHM_ALIASES` maps the short names + `elk.`-prefixed names onto the FQ ids, so `"layered"`, `"elk.layered"`, and `"org.eclipse.elk.layered"` all resolve to one provider (`elk.resolve_algorithm` does the resolution). A `LayoutPolicy`/`HierarchyEngine` selection maps onto the SHORT alias as the `elk.algorithm` option value; the layout owner never imports a provider class directly — it sets the option and lets `layout` dispatch. The `disco` alias maps but has no registered provider (selecting it raises an `ElkError`), so the layout owner's policy admits only the nine REGISTERED ids below.

Each FQ id is `org.eclipse.elk.<short>`; the layout owner selects by the short alias.

## [04]-[IMPLEMENTATION_LAW]

- import: `from pyelk import ELK, ElkError, InvalidGraphException` at boundary scope only; the distribution and import package are both `pyelk`, and the version is `importlib.metadata.version("pyelk")` (`0.3.0`). The option/graph helpers import as `from pyelk import options, graph` (or `from pyelk.options import get_effective_options, parse_padding`).
- model axis: the graph IS a plain ELK-JSON `dict`, NEVER a wrapped class model — the `visualization/diagram/layout#LAYOUT` owner builds it by lowering the `rustworkx` `PyDiGraph` (or the `data/tabular#GRAPH` adjacency frame directly) into `{"id":"root","layoutOptions":{...},"children":[{"id":str(i),"width":w,"height":h,"ports":[...],"labels":[...]} for each node],"edges":[{"id":..., "sources":[str(s)],"targets":[str(t)]} for each edge]}`, keyed on the SAME stable integer node index (stringified) the glyph marks read back on, so no third re-keying is introduced. The node id is `str(node_index)`; `ELK.layout` writes `x`/`y` onto each child and `sections` onto each edge, and the glyph-construction fold reads positions off `collect_nodes(result)[str(i)]["x"]`.
- placement axis: `ELK().layout(graph, layout_options=...)` is the one synchronous placement boundary — it runs INSIDE the layout owner's existing `_render` kernel that `_compute` offloads onto `to_thread.run_sync(self._render, limiter=_LAYOUT_LANES)`, never on the event loop and never under its own second limiter. The engine is constructed once at module scope (or per-render — construction only seeds `DEFAULTS`) and `layout` is the call; the algorithm is selected by the `elk.algorithm` option the policy sets, so one `layout` call covers every ELK algorithm without a per-algorithm method.
- ports + nesting axis: this is the capability `fast-sugiyama`/`grandalf`/`rustworkx` LACK and the reason pyelk is admitted alongside them — a node carries typed `ports` (`{"id":..., "layoutOptions":{"port.side":"EAST","port.index":N}}`) under `elk.portConstraints`, and an edge endpoint id may be a PORT id so the routed edge connects to the port coordinate rather than the node center; a compound node carries its own `children` + `edges` and is laid out as an enclosed sub-graph then sized to contain them (verified: a `grp` node with two children + an internal edge is positioned and sized to `44.0×84.0` to enclose them). An ER diagram with field ports, a flowchart-with-record-fields, a circuit/schematic, or a nested zone→program AEC diagram routes through this arm; the `STACKING`/`PROGRAM` kinds with simple center-to-center edges stay on the lighter `fast-sugiyama` arm.
- routing axis: a long layered edge is ORTHOGONALLY routed and its `sections[].bendPoints` carry the polyline waypoints (verified: an `n0→n3` edge over four layers returns `bendPoints=[{x:72,y:52},{x:72,y:92}]`); the layout owner threads `[startPoint, *bendPoints, endPoint]` into `RouteMap[(source, target)]` as the laid-out `points`, so an orthogonal route is one point sequence — the native in-engine equivalent of `grandalf`'s `route_with_lines` polyline and the `fast-sugiyama` `dummy_vertices` bend list, but emitted directly by the ELK algorithm with true port-aware orthogonal geometry. This is the only admitted engine whose routing respects ports and obstacle boxes.
- option axis: the `LayeredPolicy` fields lower to the `layoutOptions` dict — `direction` (TB/BT/LR/RL) maps to `elk.direction` (`DOWN`/`UP`/`RIGHT`/`LEFT`), the crossing-minimization knob maps to `elk.layered.crossingMinimization.strategy` (`LAYER_SWEEP`), the layering knob to `elk.layered.layering.strategy` (`LONGEST_PATH`/`NETWORK_SIMPLEX`), nesting to `elk.hierarchyHandling` (`INCLUDE_CHILDREN`/`SEPARATE_CHILDREN`), ports to `elk.portConstraints` (`FIXED_SIDE`/`FIXED_ORDER`/`FIXED_POS`) — never a free-form string the policy invents. A design page validates a chosen option against `ELK().known_layout_options()` (capability detection) and re-reads a resolved value (padding/spacing) through `options.get_padding`/`get_spacing` to keep swimlane/band geometry aligned to the laid-out container. A new layout knob is one `LayeredPolicy` field mapped to one `elk.*` option key, never a new entrypoint.
- result axis: `layout` returns the SAME ELK-JSON document with positions filled in — `children[].{x,y,width,height}`, `ports[].{x,y}`, `edges[].sections[].{startPoint,endPoint,bendPoints}`, and compound `children[].{width,height}` extents. The layout owner folds it into the `LayoutResult` `(LayoutMap, RouteMap)`: `nodes = collect_nodes(result)` (a flat id → element dict that ALSO carries the port entries), then `{i: (nodes[str(i)]["x"], nodes[str(i)]["y"]) for i in node_indices}` indexed on the stable node ids so the port entries never enter the `LayoutMap`, and each edge's `[startPoint, *bendPoints, endPoint]` → `RouteMap[(source, target)]`, no intermediate parse — the document is already the typed dict. The compound-node extents feed the `Swimlane`/zone glyph sizes directly.
- failure axis: every pyelk failure is an `ElkError` (or a leaf subclass), so the layout owner's `async_boundary` traps `ElkError` as the one diagram-layout failure class — a malformed graph surfaces as `InvalidGraphException`, an unsupported option combination as `UnsupportedConfigurationException`, an unlayoutable shape as `UnsupportedGraphException` — each mapped to the runtime fault rail and recorded in the diagnostic, never a bare `Exception` leak. `graph.validate_graph` at the boundary fails a bad adjacency frame BEFORE the layout pass.
- evidence: each ELK placement records node count, edge count, the resolved algorithm id (the `elk.algorithm` value), the layout direction, and the port-constraint/hierarchy-handling mode into the `visualization/diagram/layout#LAYOUT` `core/receipt#RECEIPT` `ArtifactReceipt.Diagram(key, kind, nodes, edges, algorithm, bytes)` case — `algorithm` becomes the ELK algorithm tag (`elk-layered`, `elk-radial`, ...), distinguishing it from the `fast-sugiyama`/`grandalf`/`rustworkx`/projected arms in the same receipt family; the content key is the `xxh3_128` digest over the serialized input graph wire joined to the resolved positions + edge sections, so a re-placement under different `layoutOptions` keys distinctly. `measure_execution_time=True` and `logging=True` are diagnostic-only switches (the timing/trace ride the result dict, not the receipt scalars).
- boundary: `pyelk` owns ELK graph COORDINATE assignment + edge-route geometry (the nine providers) for the ports/nesting/orthogonal-routing kinds; it emits no SVG (that is `visualization/diagram/draw#DRAW`'s), runs no graph analysis (centrality/shortest-path/community stay in the `data/graph#GRAPH` `rustworkx` kernel), builds no presentation graph as a CLASS model (the layout owner lowers the `rustworkx`/tabular feed to the ELK-JSON dict and reads positions back), and rasterizes nothing. The deleted forms are a hand-rolled port-placement or orthogonal-routing loop where ELK owns it, an `elkjs`/Java-ELK subprocess where this pure-Python in-process call replaces it, a wrapped node/edge class model layered over the ELK-JSON dict, a re-keying of coordinates off the stable node index, a synchronous layout left on the event loop, and a second graph-analysis kernel.

[STACKING]:
- A ports/nesting/orthogonal diagram kind in `visualization/diagram/layout#LAYOUT` selects its `LayeredPolicy.engine = HierarchyEngine.ELK` arm: the `_render` kernel builds the `rustworkx` `PyDiGraph` from the `data/tabular#GRAPH` adjacency frame, lowers it to the ELK-JSON `dict` (node index stringified to `id`, ports + labels attached from the node attributes, edges from `edge_list()`), calls `ELK().layout(graph, layout_options={"elk.algorithm":"layered","elk.direction":..., "elk.portConstraints":..., "elk.hierarchyHandling":...})`, and folds the positioned document into the `LayoutResult` — `collect_nodes(result)` filtered to the stable node ids (the flatten also carries port entries, which never enter the `LayoutMap`) → `LayoutMap`, each edge's `[startPoint, *bendPoints, endPoint]` → `RouteMap` — ALL inside the one `to_thread.run_sync(self._render, limiter=_LAYOUT_LANES)` `anyio` lane, so the pure-Python ELK placement composes the universal structured-concurrency rail with zero new async surface and the result is wrapped by the runtime `async_boundary` `RuntimeRail`.
- The `HierarchyEngine` `StrEnum` in `visualization/diagram/layout#LAYOUT` gains an `ELK` member beside `GRANDALF`/`RUSTWORKX` (and `fast-sugiyama`): the `match`/`case` over `policy.tag` stays the one total dispatch, the `LayeredPolicy` `expression.tagged_union` case gains an `elk.*` option mapping rather than a new case, and the universal `expression` rail owns the discriminant while pyelk owns one arm's body — the engine an edge-routing-with-ports or nested-container diagram selects, the lighter center-to-center layered diagrams staying on `fast-sugiyama`.
- The placement is content-addressed on the universal `xxh3_128` rail: the layout owner digests the `rustworkx` `node_link_json` graph wire joined to `repr(sorted(collect_nodes(result)...))` and the routed edge sections, so two ELK placements of one graph under distinct `layoutOptions` (direction, port constraints, crossing strategy) key distinctly and a laid-out diagram is content-addressable on its full form (topology + options + coordinates + routes), keyed by the same `ContentIdentity` owner every artifact receipt uses.
- The ELK-JSON `dict` is validated and traced through the universal rails: `graph.validate_graph` guards the boundary (a malformed adjacency frame → `InvalidGraphException` → the runtime fault rail), `ELK.layout(..., logging=True)` rides a `structlog`-bound diagnostic when the layout owner traces, and the `beartype`-checked `LayoutResult` shape (`dict[int, Point]`, `dict[tuple[int,int], tuple[Point,...]]`) is built straight off the positioned document — no class-model intermediary, the ELK-JSON dict folding directly into the `msgspec`-backed glyph owners.
- pyelk and `fast-sugiyama`/`grandalf` are co-resident by DISTINCT capability, not redundancy: `fast-sugiyama` owns fast native layered placement for center-to-center hierarchical diagrams (the dense common case), `grandalf` owns curved-spline edge routes until removal, `rustworkx` owns force/radial/topological data-plane layout + the content-key wire, and pyelk owns the ports/nesting/orthogonal-routing kinds none of the others handle — the `HierarchyEngine` selection routes each diagram to its categorical-best provider behind the one `LayoutPolicy` owner, no two engines owning the same diagram shape.
