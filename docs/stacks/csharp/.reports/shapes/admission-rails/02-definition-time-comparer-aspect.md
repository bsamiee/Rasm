# Definition-Time Comparer And Equality Aspect

[COMPARER_ACCESSOR_IS_THE_POLICY_VALUE]:
- Equality and ordering semantics enter the owner as a typed accessor — a static class exposing `static IEqualityComparer<TKey> EqualityComparer` and `static IComparer<TKey> Comparer` behind two marker interfaces — never as a comparison-mode flag the generated body must re-interpret. A library family of accessors covers the string-comparison axis (`Ordinal`, `OrdinalIgnoreCase`, the four culture pairings) and an open-generic structural-default accessor binds `EqualityComparer<T>.Default`/`Comparer<T>.Default` for any key. The accessor type is the carrier of comparison policy; the generated `Equals`, `GetHashCode`, `CompareTo`, and all relational operators route through the same one, so equality, hashing, dictionary keying, and sort order can never diverge by construction.
- An accessor whose `EqualityComparer` property folds in normalization (case folding, cultural collation, accent stripping) makes vocabulary identity a definition-time decision: two raw keys differing only along the accessor's equivalence collapse to one admitted instance, and the admitted instance carries its declaration-time canonical key, not the wire spelling. Choosing the accessor is choosing the admission equivalence relation in one place, and every downstream `Equals`/hash/sort inherits it without a hook.
- Three attributes wire the accessor at definition time, each parameterized by the accessor type and the comparison-target type as generic arguments: one binds the equality accessor for a keyed owner's key member, one binds the ordering accessor for that key, and a per-member variant binds an accessor onto a single field of a multi-field owner. Absent any of them the generator falls to a defaulted policy — ordinal-ignore-case for string keys, structural default otherwise — so the unconfigured owner still has a total, named comparison policy, never an ad-hoc inline comparison.

[GENERATED_BODY_ROUTES_THROUGH_ONE_ACCESSOR]:
- The keyed owner's `GetHashCode` is the accessor's hash of the key member, never the member's own `GetHashCode` — so a custom accessor that hashes case-insensitively keeps the hash/equality contract intact automatically. A string key with no accessor hashes through the ordinal-ignore-case static comparer to match the default equality; a value key hashes its own value; a reference type parameter null-coalesces to zero. The hash strategy is selected by the same policy that selects equality, in the same generated member, so the invariant `Equals ⇒ same hash` is structurally guaranteed rather than reviewed.
- Relational operators (`<`, `<=`, `>`, `>=`) and `CompareTo` route through the ordering accessor's `Comparer.Compare` on the key member; with no accessor, string keys compare ordinal-ignore-case and other keys defer to the member's own `CompareTo`. The same accessor that decides equality-by-fold decides sort order, so a case-insensitive owner sorts case-insensitively without a second declaration.
- Two relational-surface settings form one ordered axis, not two independent knobs: a comparison-operator generation level and an equality-operator generation level, each taking values none / default / default-with-key-type-overloads. The settings are coupled at definition time — comparison-operator strength may never exceed equality-operator strength (the generator silently raises equality to match a stronger comparison request), and skipping equality comparison or setting equality operators to none forces both to none. One cannot ask for `<` over the owner without also getting `==`; the lattice of relational surfaces is derived from one ordering, never enumerated as a free flag set.

[KEY_TYPE_OVERLOADS_COLLAPSE_THE_BOUNDARY_COMPARE]:
- The default-with-key-type-overloads level emits, alongside the owner-versus-owner operators, a full mirrored set of operators comparing the owner directly against the raw key type in both operand orders — `owner == key`, `key == owner`, and the negations, each delegating to a static `Equals(owner, value)` that runs the same accessor. A boundary that holds an admitted owner and a raw wire key compares them in one operator with the owner's own comparison policy, with zero materialization of a second owner and zero manual unwrap to the key. The comparison-against-raw is a generated surface, so the boundary never re-implements the equivalence the owner already owns.
- The relational key-type overloads similarly compare owner against raw key through the ordering accessor, so range checks at the edge (`owner < rawLimit`) run the owner's collation against an un-admitted bound. This is the sanctioned spelling for gating on a raw threshold without admitting it: the threshold stays raw, the comparison stays the owner's policy.

[PER_MEMBER_FAULT_AND_COMPARER_THREADING]:
- A multi-field owner's generated member-wise `Equals` folds the equality members through `&&`, each member compared by its own accessor when one is wired, by the owner-level default string comparison when the member is a string, by a null-safe self-equals when it is a nullable reference, and by direct value-equals otherwise. Per-member comparer policy is therefore independent across fields of one owner — one field case-insensitive, another ordinal, another structural — all selected at definition time and all flowing into the single generated `Equals`.
- `GetHashCode` for a multi-field owner crosses a generation threshold: with fewer than eight equality members and no custom or string comparers it emits a single `HashCode.Combine(...)` over the raw members; the moment any member carries a string or a wired accessor, or the count reaches eight, it switches to a `HashCode` accumulator that `Add`s each member with its accessor (bang-suppressing nullable-oblivious reference members). The fast combine path and the accumulator path produce the same contract; the threshold is an emission detail, but it means a member's comparer choice silently changes the whole owner's hash codegen shape.
- Per-member admission faults are self-naming at definition time: the generated `Validate` of a multi-field owner emits, ahead of the hook and in member-declaration order, one short-circuit null guard per non-nullable reference member, each manufacturing a fault through the fault family's `Create` whose text names the offending member and the owning type. Admission inside one owner is short-circuit at the null tier before the hook ever runs, and the member attribution is generator-authored, not call-site-authored — the flat fault never loses which field produced it.

[VALUE_CARRYING_VALIDATE_HOOK]:
- The factory-arguments hook is normally `void` with one `ref` fault slot, but an owner may declare the hook with a non-void return type; the generator then captures the hook's return value into a `factoryArgumentsValidationError` local and threads it into the post-construction init hook. The construction hook can normalize the by-`ref` arguments but has no fault channel; the factory hook owns rejection through the `ref` slot and may additionally hand a typed by-value artifact forward to derived-state initialization — so derived state computed during validation survives into the constructed instance without a second pass or a mutable field.
- The construction hook runs on every birth path including trusted rehydration and deserialization, while the factory hook runs only on admission; invariants that must hold even for storage-materialized instances belong in the construction hook, faults belong only in the factory hook. This split is the structural reason a deserialized owner is still an admitted owner — the wire path and the domain path terminate in the same construction hook, and the no-double-validation guarantee crosses the process boundary because there is no surface that re-runs the factory hook on an already-born instance.

[DETERMINISTIC_ASPECT_STACKING]:
- Comparison, equality, hashing, parsing, formatting, and serialization are independent generators each keyed off the same immutable settings record and each emitting into the same partial; the settings record is the single source every aspect reads, so no aspect can observe a half-built owner or contradict another's policy. A change to the comparer accessor reaches equality, hash, sort, and the relational and equality operators in one resolution pass because they all read it from that one record — the aspect is woven once and fanned out, never re-decided per surface.
- The settings record's own equality and hash are structural over every knob (key kind, factory names, the absent-as-null knobs, each operator level, the conversion levels, the default string comparison), which is what makes the generator incremental: a settings record that compares equal short-circuits regeneration of the entire owner. Two owners with identical policy and differing only in name share a cached emission shape; flipping one comparison level invalidates the whole owner's generated output, not a fragment, because the equality is whole-record.
- A self-referential definition-time constraint gates the aspect before it runs: the validation-error type bound on the owner is checked for self-satisfaction of its own contract at definition time, so a malformed fault family — a derived case that fails to re-implement the static create for its precise type — fails the constraint check before generation begins. The aspect that produces admission is itself guarded by an aspect over the fault type, and a broken fault lattice is a definition-time error, not a runtime surprise inside a generated body.
