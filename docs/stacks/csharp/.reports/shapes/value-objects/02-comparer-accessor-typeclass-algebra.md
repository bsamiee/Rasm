# Comparer-Accessor Typeclass Algebra

[ACCESSOR_AS_STATIC_TYPECLASS]:
- The identity policy of a generated owner is carried by a type, not a value: a comparer accessor is a class or struct whose only obligation is one or two `static abstract` properties — `static abstract IEqualityComparer<T> EqualityComparer` and `static abstract IComparer<T> Comparer` — and the policy attribute threads the accessor purely as a type argument (`[KeyMemberEqualityComparer<TAccessor, TKey>]`, `[KeyMemberComparer<TAccessor, TKey>]`). The generated `Equals`, `GetHashCode`, `CompareTo`, and every relational operator call the accessor's property through the constraint, so the comparison policy is resolved at the generic-instantiation site with no field, no constructor parameter, and no per-owner allocation. The owner's comparison surface is a typeclass dictionary passed at compile time.
- The two accessor concerns are split into two interfaces precisely so they can be satisfied independently: a key that needs only ordering (a custom struct key with hand-written `<`) takes a `IComparerAccessor` accessor while equality stays on the default; a key that needs only a custom equality bucket takes a `IEqualityComparerAccessor` accessor while ordering stays absent. The diagnostics re-couple them — a lone equality or ordering accessor draws a pairing warning because hash containers and sorted containers would then disagree on identity — but the type system keeps the concerns orthogonal so the author opts into exactly the surface the domain needs.
- An accessor is variance-flexible at the type boundary: both interfaces are `in T` contravariant, so an accessor written over a base or interface type satisfies the constraint for any key assignable to it, and one accessor class supplies the policy for a whole family of keys related by inheritance rather than one accessor per concrete key.
- The accessor interface places no `class` constraint on its implementers, so a `readonly struct` accessor is legal and carries the same `static abstract` properties. Because the generated code reaches the comparer only through the static member — never through an accessor instance — the accessor type is never instantiated; its sole runtime footprint is the comparer object the property returns, and that object is shared across every comparison the owner performs.

[STATIC_TO_INSTANCE_HOP]:
- The accessor returns a comparer *instance* (`IEqualityComparer<T>` / `IComparer<T>`), not a static method, so the generated comparison is a two-hop dispatch: a static-abstract property read resolved at compile time, then a virtual `comparer.Equals`/`comparer.Compare` call on the returned object. The first hop is free; the second is the cost of expressing policy as a reusable comparer object rather than inlined arithmetic. The shipped accessors return process-shared singletons (`StringComparer.Ordinal`, `EqualityComparer<T>.Default`), so the property body is a field read, not an allocation.
- A custom accessor's property is evaluated on every comparison unless the author caches: a body that constructs a fresh comparer per get manufactures a new comparer object on each `Equals` call inside a hot dictionary probe. The correct shape stores the comparer in a `static readonly` field and returns it, matching the shipped accessors' discipline; the typeclass contract rewards a singleton comparer because the type is the dictionary and the comparer is the only state.

```csharp
public sealed class CodepointAccessor : IEqualityComparerAccessor<string>, IComparerAccessor<string>
{
   private static readonly IEqualityComparer<string> EqInstance = new ProjectionEqualityComparer<string, int>(s => s.Length);
   public static IEqualityComparer<string> EqualityComparer => EqInstance;
   public static IComparer<string> Comparer => StringComparer.Ordinal;
}
```

[PROJECTION_ACCESSOR_DERIVATION]:
- The projection equality comparer collapses "compare by a derived field" into one selector: it takes `Func<T, TItem>` plus an optional inner comparer and folds null-handling, selection, and inner comparison into `Equals(x, y) => x is null ? y is null : y is not null && inner.Equals(select(x), select(y))` with a matching `GetHashCode` that returns zero for a null projection. Wrapping it in a one-property accessor turns any pure projection into a complete identity policy — case-insensitive-by-normalized-form, ordering-by-rank, equality-by-canonical-key — without a hand-written comparer class. The projection is the policy; the accessor is its type-level carrier.
- A projection accessor composes the inner comparer the same way the owner composes the accessor: the projection selects a sub-value, then defers to an inner `IEqualityComparer` for that sub-value, which can itself be a string-policy comparer. A two-level policy — "equal when the trimmed, ordinal-ignore-case form of a derived substring matches" — is one accessor whose field is `new ProjectionEqualityComparer<string, string>(Normalize, StringComparer.OrdinalIgnoreCase)`, and the owner inherits the full stack through a single type argument.
- A member-level equality accessor on a complex owner is the same one-property class, attached per member rather than per key (`[MemberEqualityComparer<TAccessor, TMember>]`); the member list of a multi-field owner thus carries a heterogeneous vector of typeclass policies, one accessor type per member, all resolved statically inside the single generated `Equals`. A collection-typed member — whose default reference identity is almost always wrong — is corrected by attaching a sequence-equality projection accessor, and that is the only sanctioned fix because no generated tier consults `EqualityComparer<T>.Default` for reference members.

[OPERATOR_ALGEBRA_SYNTHESIS]:
- A configured comparer accessor synthesizes relational operators the key never had. With no accessor the generated `<`, `<=`, `>`, `>=` forward to the key's own operators, so a `string`-keyed owner has none (string has no relational operators); with a comparer accessor the operators route through `Comparer.Compare(a, b) {<,<=,>,>=} 0`, and the owner exposes the full `IComparisonOperators<T,T,bool>` interface over a key type that is itself not generic-math-comparable. The accessor is not a configuration of existing behavior — it manufactures an operator algebra on top of a key that lacks one, and the owner becomes more constraint-satisfying than its underlying type.
- The two ordering surfaces converge only through the accessor. `CompareTo` defaults to casting the key to `IComparable<TKey>`; the relational operators default to the key's own `op_*` methods — two independent routes that can disagree when a key implements one but not the other. A comparer accessor forces both `CompareTo` and the operators through `Comparer.Compare`, restoring a single ordering definition. This is the structural reason the explicit-comparer diagnostics treat equality-and-ordering accessor pairing as one policy: only with the accessor present do the two ordering surfaces provably agree.
- Operator availability is detected per operator from the key's `op_*` set, so a key with a partial hand-written operator set yields the matching partial operator subset on the owner and no `IComparisonOperators` interface declaration; an accessor is the way to lift a partial-operator or no-operator key to the complete four-operator interface, closing the gap where the owner supports `<` at a call site yet silently fails a `where T : IComparisonOperators<T,T,bool>` constraint.

[SELF_HEALING_PAIRING]:
- The equality/comparison operator-generation settings self-reconcile in the generated output while still diagnosing the explicit spelling: the equality-operators property getter returns the maximum of its own value and the comparison-operators setting, so requesting key-type operator overloads for comparison transparently grants the same overloads for equality, and a divergence (comparison broader than equality) is clamped up rather than emitted asymmetrically. The divergence warning still fires on the written attribute, demanding the two be spelled identically even where the clamp already reconciled them — the generated code is self-healing, the source contract is not.
- The generic default accessor (`Default<T>` over `EqualityComparer<T>.Default` and `Comparer<T>.Default`) is the pairing-satisfier for non-string keys: when ordering is wanted but only default equality, or vice versa, naming the default accessor on the missing side resolves the pairing diagnostic without inventing a bespoke comparer, because one accessor type satisfies both interfaces at once. The default accessor is the identity element of the accessor algebra — it supplies the absent half of a policy pair with zero semantic change.

[GENERATED_HASH_FORM_COUPLING]:
- The complex-owner `GetHashCode` switches whole form on the accessor population: a `HashCode.Combine` fast path is emitted only when equality membership has fewer than eight members, zero accessors, and zero string members; the presence of a single member accessor or one string member flips the entire method to the incremental `HashCode.Add` form with the comparer threaded per member. Attaching one projection accessor therefore rewrites the hash method's shape, and the rewrite is exactly what keeps hash and equality policy-aligned — the hash cannot fall back to default hashing on a member whose equality is custom.
- Per-member equality in a complex owner is a fixed four-tier resolution evaluated independently for each member: an explicit member accessor first, then the owner-level string policy for direct string members, then null-tolerant `a.Equals(b)` for reference members, then direct `.Equals` for value members. The accessor tier is the only one that overrides intrinsic semantics, so a member whose intrinsic `Equals` is wrong is correctable solely by promoting it to the accessor tier — there is no owner-wide switch that reaches a single member without an accessor.

[CONTAINER_OVERRIDE_VS_INTRINSIC]:
- The accessor sets the owner's intrinsic identity — the one used by `==`, `Equals`, and any default-comparer container — while the shipped string-keyed object comparer overrides identity at the container without touching the owner: it compares two owners by egressing both keys through the universal `ToValue()` and applying a chosen string policy, exposing ordinal and culture pairs as prebuilt static instances over any `IConvertible<string>` owner. The two mechanisms are orthogonal layers — accessor decides what the owner *is*, container comparer decides how one collection *views* it — so a case-insensitive owner can be indexed case-sensitively in one dictionary without changing the owner's intrinsic comparer or constructing a second owner type.
- Because the container comparer is generic over the egress contract rather than over any concrete owner, the override layer is written once and applies to every string-keyed owner uniformly, whereas the accessor layer is per-owner policy fixed at declaration. A domain mixes them deliberately: intrinsic accessor policy for the owner's natural equality, container comparer for a single index that needs a divergent view, never duplicating the owner to obtain the second policy.
