# Aspect Architecture — Deepening

[UNION_MEMORY_AND_ACCESS_ASPECTS]:
- `UseSingleBackingField = true` on any `[Union]` declaration collapses all per-case backing fields into one `object` field. The trade-off is explicit: field-per-case stores the value in its native type (no boxing for value types), while a single backing field always boxes structs. For unions whose cases are all reference types this is an unambiguous memory-layout improvement; for unions with struct cases it introduces per-access boxing. The correct decision is therefore driven by case type topology at declaration time, not by a blanket policy.
- `ConstructorAccessModifier = UnionConstructorAccessModifier.Private` on `[Union]` or `[Union<T1,T2>]` makes both the generated constructors and the generated implicit conversion operators (`op_Implicit`) private simultaneously — the coupling is documented and load-bearing. The effect is that the only factory surface visible externally is hand-written static factory methods on the union type itself; no caller can construct a case by direct implicit assignment. This is the definition-time enforcement mechanism for admission-controlled unions where construction must go through validation or enrichment logic before the case is materialized, without requiring a separate `[ValueObject]` layer.
- `T1IsStateless = true` (and the corresponding `T2IsStateless`, etc.) on `[Union<T1,T2>]` instructs the generator to store only the discriminator index for that arm — no field is allocated for that type's instance data, and accessors like `AsT1` and `Value` return `default(T1)`. This eliminates the case field entirely when a case carries no payload (marker/signal cases). The generator recommends struct types for stateless members to eliminate null-handling at the accessor, confirming that the stateless aspect is size-driven and the type choice follows from it.
- `ConversionFromValue` on `[Union]` (inheriting from `UnionAttributeBase`) controls the implicit operator direction from member type to union type. `ConversionToValue` controls the reverse. These are independent axes: a union can admit values implicitly (implicit `from T` operator) but require an explicit cast to extract them, or vice versa. Decoupling these two directions is the definition-time aspect that enforces directional admission policy — inbound permissiveness does not imply outbound permissiveness.

[SMART_ENUM_KEY_SURFACE_ASPECTS]:
- `KeyMemberKind = MemberKind.Field` on `[SmartEnum<TKey>]` changes the generated key member from a public property to a field. The name default changes accordingly: `Key` for a property, `_key` for a private field. This matters architecturally when the key must be inaccessible from outside the type — `KeyMemberAccessModifier = AccessModifier.Private` combined with `KeyMemberKind = MemberKind.Field` generates a private backing field named `_key` that carries the discriminant without exposing it as a public surface. The enum's behavioral identity is then entirely through `Switch`/`Map` and the delegate-backed partial methods, not through key inspection.
- `DisableSpanBasedJsonConversion = true` on `[SmartEnum<string>]` falls back from the generated `ReadOnlySpan<char>`-based JSON reader path to a string-based path. The span path is the zero-allocation admission path on .NET 9+ (the `IObjectFactory<Owner, ReadOnlySpan<char>, TError>` generated interface avoids heap allocation for the transient key during deserialization). The disable flag exists for compatibility with STJ configurations that do not drive the span reader overload, but enabling it for correctness reasons is a signal that the serialization context is not running the span path and the admission zero-allocation guarantee is silently lost.
- `SkipIParsable = true` on `[SmartEnum<TKey>]` forces `SkipISpanParsable` to `true` simultaneously because `ISpanParsable<T>` inherits from `IParsable<T>`. Selectively suppressing `IParsable` while retaining `ISpanParsable` is not possible — the inheritance chain in the BCL makes the skip cascade. Any grammar or text-protocol aspect that relies on `ISpanParsable` for span-based parsing is therefore disabled by the same flag that disables string-based parsing.

[CONSTRUCTION_SEAM_DEEPENING]:
- `[ObjectFactory<T>]` with `UseForSerialization` steers the generated serialization converters to route through the custom `IObjectFactory<Owner, T, TError>` implementation's `Validate` method instead of the primary key-based `Create`/`TryCreate` factory. The constraint is that the custom factory type must implement `IObjectFactory<T, TValue, TError>.Validate(TValue value, IFormatProvider provider, out T result)` — the `out` parameter and the `IFormatProvider` argument are load-bearing for the STJ span-reader path. A factory that implements only `IObjectFactory<T>` (the marker interface with one type parameter) satisfies the runtime but will not be wired by the generator for serialization purposes.
- `HasCorrespondingConstructor = true` on `[ObjectFactory(typeof(T))]` informs the generator that a constructor taking a single `T` parameter exists on the value object type, enabling the generator to emit a conversion expression that routes through the constructor rather than only through the static factory method. This is the construction-advice aspect for EF Core value converters: EF often needs to materialize instances via constructor expression trees, and the generator wires the constructor path only when this flag is explicitly asserted.
- `DefaultStringComparison` on `[ComplexValueObject]` sets the comparison mode for all `string`-typed members that do not carry their own `[KeyMemberEqualityComparerAttribute<...>]`. The default is `OrdinalIgnoreCase`. This is a type-level default-comparison aspect: declaring it at the type level prevents per-member comparer annotation proliferation when all string members share one comparison policy.

[COMPOSITION_TIME_POLICY_DEEPENING]:
- `repeat` and `retry` are semantically antipodal: `retry` fires on failure (re-invokes the computation when it produces an error), `repeat` fires on success (re-invokes when it produces a value). `retryWhile`/`retryUntil` predicate on `Error`; `repeatWhile`/`repeatUntil` predicate on the bound value `A`. A schedule-driven heartbeat loop is `repeat(schedule, computation)`; a schedule-driven resilience loop is `retry(schedule, computation)`. Mixing the two produces a behavior that is not obviously wrong at the type level because both accept `K<F, A>` — the distinction is purely semantic and must be stated as a design constraint.
- `IO<A>.BracketFail` is distinct from `IO<A>.Bracket`: `Bracket` releases acquired resources on both success and failure, while `BracketFail` releases them only on failure. The asymmetry exists for cases where the resource itself is the result — acquiring a network connection and returning it as the `A` value must not dispose it on success, because the caller now owns it. Using `Bracket` in that scenario would dispose the resource before the caller can use it. `BracketFail` is the correct form whenever the acquired resource escapes the bracket scope as the computation's result.
- `Eff<A>.MapIO(Func<IO<A>, IO<B>>)` is the unlift bridge: it extracts the `IO` monad from the `Eff` transformer stack, applies a function that is only expressible on `IO` directly (such as `.Retry(schedule)`, `.Bracket`, or `.BracketFail`), and re-lifts the result back into `Eff`. This is the correct mechanism for applying IO-layer aspects (retry, bracket, catch) from within an `Eff` pipeline — calling `retry(schedule, eff)` via the Prelude works at the `K<M, A>` level but uses the `MonadUnliftIO` trait dispatch. `Eff<A>.MapIO` is the instance-method form that makes the unlift explicit and avoids the trait indirection.
- The generic `retry<M, A>(Schedule, K<M, A>)` overloads (two type parameters, no IO-specific constraint) operate on any monad `M` that satisfies `MonadUnliftIO<M>`. `Eff<RT, A>` satisfies this constraint; `Eff<A>` (single type parameter, using `MinRT`) also satisfies it. The consequence is that retry/repeat aspects compose uniformly across bare `IO<A>`, `Eff<A>`, and `Eff<RT, A>` without requiring explicit lowering to `IO` first.

[SCHEDULE_ALGEBRA_DEEPENING]:
- `Schedule.Interleave(b)` alternates steps from two schedules — a, b, a, b... — for the length of the longer. This is distinct from `Combine` (sequential append: all of `a`, then all of `b`) and from `Intersect`/`Union` (parallel min/max per step). A burst-then-steady retry pattern is `Schedule.recurs(3).Interleave(Schedule.spaced(1*second))`: attempt, immediate, attempt, 1s, attempt, 1s — interspersing the spaced gaps between each fast retry.
- `Schedule.decorrelate(factor, seed)` transforms a schedule by introducing both upward and downward jitter, effectively doubling the schedule's run length. Per the documentation, given a linear schedule `(100, 200, 300, ...)` decorrelation produces roughly `(103, 97, 202, 197, ...)`. The semantic contract is: the decorrelated schedule is less correlated with parallel instances of the same computation, reducing thundering-herd re-collision at cost of doubled total elapsed time. It is not a pure delay-expansion combinator; it is a statistical de-correlation aspect.
- `Schedule.resetAfter(max)` resets the schedule once the cumulative delay reaches `max`, restarting the delay sequence from zero. Paired with `Schedule.exponential(base) | Schedule.recurs(n)`, this implements bounded exponential backoff with a reset window: the exponential grows until the cumulative sum exceeds `max`, then resets and begins growing again, indefinitely. This is strictly different from `maxCumulativeDelay(max)` which terminates the schedule at `max` total delay rather than resetting it.
- `Schedule.fixedInterval(d)` guarantees that the gap between action completions is `d`, absorbing action duration into the next gap — if the action takes longer than `d`, the next gap is zero rather than piling up. `Schedule.windowed(d)` instead aligns to window boundaries on a clock timeline — the next step waits until the nearest multiple of `d` from an epoch, regardless of when the action completed. `windowed` is calendar-clock-aligned; `fixedInterval` is relative-to-completion-aligned. Both take a `Func<DateTime>` clock argument, making them injectable for deterministic testing.
- `Schedule.maxDelay(max)` clamps each individual duration to `max`; `Schedule.maxCumulativeDelay(max)` terminates the schedule once the sum of all emitted durations exceeds `max`. These are orthogonal caps: `maxDelay` ensures no single sleep exceeds `max`, while `maxCumulativeDelay` ensures the total wall time spent waiting does not exceed `max`. A production retry policy uses both: `Schedule.exponential(50*ms) & Schedule.maxDelay(30*seconds) & Schedule.maxCumulativeDelay(5*minutes)` caps the per-step delay and the total wait window independently.

[INTERCEPTOR_WEAVE_SURFACE]:
- `GetInterceptableLocation` on `SemanticModel` returns an `InterceptableLocation` value (version integer + opaque data string) only for invocation expressions — constructor calls, property accessors, and delegate invocations are not interceptable surfaces. The version integer is emitted as the first argument to `[InterceptsLocation(version, data)]`; a mismatch between the version used during generation and the version expected by the Roslyn compiler at build time produces a compile error, preventing silent data-corruption from format-version drift. Any source generator that produces interceptors must regenerate location data on every compilation rather than caching it across builds.
- The interceptor signature match requirement distinguishes two error levels: `ERR_InterceptorSignatureMismatch` fires when the interceptor has a parameter or return type that is incompatible with the intercepted method's resolved signature; a warning-level mismatch fires for compatible-but-not-identical cases (e.g., differing parameter names, `params` vs. explicit array). The distinction matters for generated interceptors because parameter-name divergence (warning) is tolerable but type divergence (error) is a build break. Generators that produce interceptors must track both.
- Interceptors produce a strict call-site replacement rather than a before/after wrapping: the original method is never called unless the interceptor explicitly calls it. This is categorically different from Roslyn analyzer-side fixups or post-compilation IL rewriting — the interceptor owns the entire call body for that location. The implication for aspect stacking is that a generated interceptor replacing a factory call can inject logging, validation, or caching purely at the call site without touching the factory implementation, but it cannot compose with a second interceptor for the same location: only one interceptor per location is valid.

[MODULE_INITIALIZER_ORDERING_CONSTRAINT]:
- Within a single assembly, `[ModuleInitializer]` methods run in source-file order and then declaration order within each file. This ordering is stable but not documented as a guarantee across compiler versions — it is a consequence of the IL emission order, not a language specification promise. Aspect registrations that depend on ordering (codec installation before first deserialization, registry population before first lookup) must either be placed in a single module initializer method or must use a chain of explicit dependency calls inside that method rather than relying on cross-method ordering across multiple `[ModuleInitializer]` declarations.
- `[ModuleInitializer]` cannot appear on a generic method, a method in a generic type, or a method that is not an ordinary method (no property accessors, constructors, operators). It also cannot appear on a method in a `ref struct`. These constraints narrow its use as a weaving surface: it cannot be a generic factory installer, it cannot be placed on generated partial constructors or partial methods, and it cannot be embedded inside generic-type aspects. All registration logic it carries must be fully specialized at the type level.
