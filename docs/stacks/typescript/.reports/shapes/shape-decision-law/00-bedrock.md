# Shape Decision Law

[SIGNATURE_KEYS]:
- The concept signature is a five-axis tuple read before any shape is written: identity regime (structural value vs reference cell vs nominal primitive), variant cardinality (one shape vs N tagged shapes), payload timing (fixed at definition vs per-occurrence per-context), equality demand (deep structural `Equal`/`Hash` vs reference vs none), persistence/wire demand (codec + per-channel projection vs in-memory only). The chosen owner is a pure function of this tuple, not of field count or perceived complexity.
- Identity regime is the dominant axis: it splits the entire decision tree at the root, because it selects whether the owner must carry an `Equal`/`Hash` instance (`Schema.Class`/`Data.*`), a brand witness (`Schema.brand`), or neither (`as const` table read by `keyof typeof`).
- Variant cardinality of one routes to a single `Schema.Class`/`Model.Class`; cardinality of N-with-shared-discriminant routes to a `Schema.TaggedClass` union or `Data.taggedEnum`; cardinality of N-as-closed-keyset-of-behaviors routes to an `as const` behavior table. A second conceivable case promotes a one-shape owner to a tagged union before any call site exists, never after.
- Payload-timing-per-occurrence is the sole signal that forces `Model.Class`: a field whose admitted shape differs across read, write, and serialization channels cannot be expressed by a flat `Schema.Class` without spawning sibling schemas, and the sibling-schema spawn is exactly the multiplicative artifact `Model.Class` collapses.

[AXIS_SHIFT_LAW]:
- A concept signature is mutable: each of the five axes can flip after a route exists, and a flip re-routes the owner — the chooser is re-run, never patched. A wrong route locked in by call sites is the multiplicative failure the anticipatory shape prevents; the cost asymmetry is the law — a late flip is measured in fanned-out consumer edits, an early-over-shaped owner is one unused union arm.
- Only two axis flips re-route within the same owner family with zero consumer churn: cardinality one→N when the owner was already a `Schema.Union` of one `TaggedClass` member (the new member is one declaration), and persistence none→wire when the owner was already a `Schema.Class` (the codec was always present, only the decode boundary is new). Every other flip crosses owner families and breaks consumers loudly — which is the design goal, not the defect.
- Identity-regime flip is the only irrecoverable one: a structural-value owner that becomes a reference cell cannot be re-derived by indexed access or `extend`, because `Equal`/`Hash` semantics are baked into the constructor at declaration. The flip forces a hand-migration of every comparison site, so identity regime is the axis `ANTICIPATORY_COLLAPSE` reads most conservatively — when reference identity is conceivable, the value owner is the wrong first route.
- Equality-demand flip none→deep on an `as const` table does not extend the table; it re-routes to `Data.struct`/`Schema.Class`, because a table read by `keyof typeof` has no instance to compare. The promotion ladder is monotone — each added axis can only move up the owner tier, never down — so reading the table for behavior and the class for identity in the same concept is the split a single owner must absorb before either is written.

[ROUTING_TABLE]:
- Structural value, one shape, fixed payload, deep equality, wire need -> `Schema.Class<Self>(id)({...fields})`: one declaration yields constructor, static `Type`/`Encoded`, codec, `Equal`/`Hash`, and `make`. The reject is an `interface` plus a parallel `Schema.Struct` plus a hand-written `Equivalence`.
- Structural value, N tagged shapes, codec/wire need -> a union of `Schema.TaggedClass<Self>(id)(tag, {...})` members closed by `Schema.Union`; `Match.tagsExhaustive` folds them. The reject is a discriminated `interface` union plus a separate decode `switch`.
- Structural value, N tagged shapes, in-memory only, no codec -> `Data.taggedEnum<T>()`; each tag constructor plus the generated `$is`/`$match` reflective matchers ship in one statement. The reject is hand-written constructors plus a hand-written type guard zoo.
- Nominal primitive carrying a refinement invariant -> `Schema.<base>.pipe(Schema.<filter>(), Schema.brand("<Name>"))`: the brand and its validating `make` derive from the filtered base. The reject is `type X = number & Brand<"X">` declared beside a separate runtime check.
- Closed keyset of behaviors, decode-free, dependency-free -> `as const` object read by `keyof typeof` and indexed access. The reject is an `enum` plus a parallel lookup `Record` plus a `switch` mapping keys to behavior.
- Closed set of bare literal values, no per-value payload and no behavior column -> `Schema.Literal(...values)`, whose `Type` is the `values[number]` union, carrying codec and exhaustive matching with no struct, table, or class. The reject is an `as const` array plus a hand-written `is`-guard; the reject above it is a behavior-`as const`-table when the values carry no behavior — a behavior table with empty rows is a `Literal` wearing keys.
- Per-occurrence payload across persistence channels -> `Model.Class<Self>(id)({...})` with `Model.Generated`/`Model.Sensitive`/`Model.FieldOption`/`Model.DateTimeInsert` field modifiers. The reject is one base schema plus five `pick`/`omit`-derived sibling schemas for select/insert/update/json variants.

[IDENTITY_TIER_LADDER]:
- Four owners satisfy the structural-value-one-shape signature, ranked by what each adds: `as const` table (no instance, behavior by key) → `Data.struct(obj)` (`Equal`/`Hash`, no codec, no constructor reuse) → `Schema.Data(struct)` (`Equal`/`Hash` plus codec, a `transform` over `DataFromSelf`) → `Schema.Class<Self>(id)(fields)` (all of the prior plus a named constructor, instance methods, and `extend` lineage). The tie-break is the lowest tier that carries every demanded axis — over-tiering is dead surface, under-tiering is the missing axis.
- `Data.struct` is the terminal owner exactly when deep equality is demanded but no value ever crosses a decode boundary: it returns a frozen structural-equality record with no `ast`, so `Schema.decodeUnknown` cannot consume it and `pick`/`omit` cannot project it. The moment a wire channel appears, `Data.struct` is the wrong owner and `Schema.Data` or `Schema.Class` takes the route.
- `Schema.Data(innerStruct)` is the surgical middle tier: it lifts an identity-free `Schema.Struct` into an `Equal`/`Hash`-bearing schema without a class declaration, so a transient decoded value gains structural comparison without earning instance methods or a `Self` generic. Choose it over `Schema.Class` when the value is compared and decoded but never carries derived getters — the class is the wrong owner when its body would be empty.
- `Schema.Class` earns its tier only when the owner carries behavior (instance getters/methods reading fields directly) or anchors an `extend`/`transformOrFail` lineage; an empty-bodied `Schema.Class` over a struct is `Schema.Data` wearing a constructor, and the constructor is ceremony. The presence of a method in the body is the discriminant that promotes `Schema.Data` to `Schema.Class`.

```typescript
const Coordinate = Schema.Data(Schema.Struct({ lat: Schema.Number, lon: Schema.Number }))
class Waypoint extends Schema.Class<Waypoint>("Waypoint")({
    at: Coordinate,
    label: Schema.NonEmptyTrimmedString,
}) {
    get key(): string {
        return `${this.at.lat}:${this.at.lon}`
    }
}
const a = Waypoint.make({ at: Data.struct({ lat: 0, lon: 0 }), label: "<value-a>" })
const b = Waypoint.make({ at: Data.struct({ lat: 0, lon: 0 }), label: "<value-a>" })
const same = Equal.equals(a.at, b.at) && Equal.equals(a, b) // structural Equal recurses through the embedded Data
```

[SHAPE_BUDGET_ARITHMETIC]:
- Artifact count is multiplicative, not additive: a concept needs a value, a static type, a codec, an equality, a key projection, and one wire projection per channel. A naive chain spells each link as its own declaration, so a six-facet concept across three wire channels is `1 value + 1 type + 1 codec + 1 equality + 1 keys + 3 projections = 8` artifacts that drift independently; the owner collapses all eight into one `class` declaration where every facet is a derived member.
- The collapse trigger fires on concept density, not line count: three or more parallel shapes for one concept, three or more sibling factories sharing a prefix, three or more repeated dispatch arms, or three or more single-call private helpers. Any one signal triggers; three make the move mandatory.
- The break-even for promoting a hand-spelled tagged union to a generated owner is the per-member facet count, not the member count: a two-member union each needing constructor + type + codec + equality + match-arm already pays for `Schema.TaggedClass`, because the generated form deletes four facets per member while adding one declaration per member.
- The owner is sized for the family it anchors, never the instance in hand: the diff of the next case is the proof. A correct owner absorbs the next case as one union member the checker chases or one behavior row; an incorrect owner forces a new sibling surface and a fan-out edit across existing call sites.

[CLASS_AS_FUSED_OWNER]:
- `Schema.Class<Self>(id)({...})` returns a value that is simultaneously a class constructor (`new`), a schema (carries `ast`), a codec source, an `Equal`/`Hash` instance, and a static-type source via `Schema.Schema.Type<typeof X>`. No `type X = typeof x` and no second `const` ever restate it; the static type is extracted, the value is the runtime, and the two are one declaration.
- The class body holds derived getters and methods as ordinary members reading fields directly, so projection logic lives inside the owner, never in external functions keyed on the owner's type — the single mechanism fusing `DEEP_SURFACES` and `SINGLE_SOURCE` in one statement.
- `Self` is mandatory and self-referential — `class X extends Schema.Class<X>("X")({...})` — and omitting it collapses the return to a `MissingSelfGeneric` string-literal error type, not a usable class; the self-reference is what lets the schema's `Type` equal the class instance type rather than a structural twin.
- A `Schema.Class`'s `ast` is an `AST.Transformation`, not a `TypeLiteral` — the class is a transform between its decoded instance type and its struct encoded shape, so the identifier names the transform: two classes with identical fields but distinct identifiers are distinct owners with distinct codecs and distinct `$defs` keys, and identifier identity, not structural identity, individuates a class in the schema graph. A copy-pasted class body with a fresh identifier is a genuinely new owner.
- A `Class` presents itself to generated members through a `SurrogateAnnotation` (read by `getSurrogateAnnotation`) carrying the underlying struct `ast`, so `Schema.pick`, `Schema.omit`, `Schema.encodedSchema`, and field-level introspection see the class as its struct without unwrapping the transformation. The reject is re-spelling the struct beside the class to make it projectable; the surrogate already exposes it.

[LINEAGE_GROWTH_FORK]:
- Growing a `Schema.Class` owner in place forks into three lineage members, each a different relationship between the new fields and the parent, and new capability lands inside the owner lineage rather than a parallel schema: `extend<Next>(id)(newFields)` adds fields orthogonal to existing ones (a parent value is a valid prefix of the child), `transformOrFail<Next>(id)(newFields, { decode, encode })` brackets the new fields by a decode computed from the parent's *Type* (the child holds a computed projection the parent cannot carry without the source), and `transformOrFailFrom<Next>(id)(...)` brackets them by a decode applied to the parent's *Encoded* before the parent decodes. The fork is read by where the dependency sits — in the decoded value, or in the raw input. A derived field spelled as an `extend` field plus an external mutation step is the re-route the lineage member deletes.
- Both lineage forms return a `Class` whose `Inherited`/`Proto` parameters carry the parent's instance methods, so the lineage shares behavior without re-declaration. Lineage growth has a hard ceiling: `extend` requires a fresh `Self` generic or collapses to `MissingSelfGeneric<"Base.extend">`, and the child's constructor accumulates `C & Struct.Constructor<NewFields>` — the parent is never re-routed, only succeeded.
- When a "child" would override a parent field's *type* rather than add to it, the lineage is the wrong tool: type override is a distinct concept, routed to a sibling class, not an `extend` that silently widens the union of constructors. Growth re-routes out of lineage entirely when the new requirement is a *variant* rather than a *field*: a second tagged shape sharing the parent's discriminant is a `Schema.Union` member, never an `extend`, because `extend` adds fields to one shape and a union adds shapes to one discriminant. Modeling a variant as an optional-field-laden superclass is the parallel-shape smell the cardinality axis exists to catch.

[EXTEND_AS_OWNER_FUSION]:
- `Class.extend<Next>(id)(fields)` and `Schema.extend(self, that)` fuse two owners into one when the concept is genuinely the *intersection* of two shape families, not a parent-child reference: the result carries `Fields & NewFields`, the combined constructor `C & Struct.Constructor<NewFields>`, and the union of contexts `R | Struct.Context<NewFields>`. Fusion is the route when two vocabularies co-occur on every instance; embedding is the route when one is a sub-value of the other.
- `extend` accepts a `HasFields` argument — a bare `Schema.Struct`, a refinement of a struct, a `suspend` of a struct, or a struct unioned with a `Schema.Record` index signature — so a fixed-key owner fuses with an open index signature in one `extend`, yielding a value that carries both named fields and a typed string-keyed tail. The reject is a parallel `Schema.Record` declared beside the struct plus a hand-merged read.
- Field-overlap support is asymmetric and the trap is silent at the call site: two structs with overlapping field *names* extend only when the overlapping fields' schemas are themselves extension-compatible (string-refinement over string-refinement, struct over struct), and an overlap whose schemas conflict produces a parse-time duplicate-property error, not a type error — the fusion compiles and fails at decode. Overlapping fields that are not co-refinable are the signal the two owners are distinct concepts, routed to embed or to a tagged union, never to `extend`.
- `extend` composes left-to-right under `pipe`, so a base owner fuses with a struct of new fields and then with a record index signature in one chain (`base.pipe(Schema.extend(struct), Schema.extend(record))`), accumulating one schema; the growth axis is a further `extend` in the chain, never a new sibling owner. A union-valued `extend` argument distributes the extension over each union member, so fusing a base struct with a tagged union yields a union of extended members — one declaration spelling N fused shapes.

[TAGGED_DISCRIMINANT]:
- `Schema.TaggedClass` injects `_tag` as a literal-typed field automatically; the constructor argument omits `_tag` (`Struct.Constructor<Omit<Fields, "_tag">>`), so the tag is never passed at construction and never drifts from the declared literal.
- `Data.taggedEnum` ships `$is(tag)` returning a `u is Extract<A, {_tag: tag}>` guard and `$match` accepting an exhaustive case record `{[Tag]: (args) => R}` with excess-key rejection (`[K in Exclude<keyof Cases, A["_tag"]>]: never`); the discriminant routing is generated, total, and checked, never a hand-written `switch`.
- The injected literal `_tag` is the discriminant the union owner already declares, so widening the union with a new tagged member breaks every exhaustive consumer loudly at compile time rather than silently — the absorbed-growth guarantee that makes the union the correct owner the moment a second case is conceivable.
- `Data.taggedEnum` accepts a `WithGenerics<N>` interface for type-parameterized unions (`numberOfGenerics` 1–4, slots `A`/`B`/`C`/`D`), so a generic result-like union is one owner, not a per-instantiation copy. The reject is a hand-spelled generic discriminated union plus per-arity constructors.

[CODEC_PRESENCE_FORK]:
- The variant-cardinality-N route forks on persistence demand into three near-peers sharing `_tag` injection: `Data.taggedEnum<T>()` (in-memory, generated `$is`/`$match`, no codec), `Schema.TaggedStruct(tag, fields)` (codec plus `_tag`, no `Equal`/`Hash`, no constructor), and `Schema.TaggedClass<Self>(id)(tag, fields)` (codec plus `Equal`/`Hash` plus constructor and methods). The fork is read as two independent bits — codec-need and equality-need — not one ladder.
- The identity-free wire member is chosen by what consumes it downstream: a tagged member used only to decode and `Match.tag`-route is `Schema.TaggedStruct`, and a member ever deduplicated, cached by identity, or used as a `HashMap`/`HashSet` key is `Schema.TaggedClass` — structural-key membership is the discriminant, because only the class member's `Equal`/`Hash` makes structural-key lookup correct. Mis-routing here is silent: a `TaggedStruct` used as a `HashMap` key compiles and dedupes by reference, corrupting the map without a diagnostic.
- `Data.taggedEnum` is the correct N-owner only when no member ever decodes: its `$match` is a generated total record with excess-arm rejection (`[K in Exclude<keyof Cases, A["_tag"]>]: never`), and adding a member breaks every `$match` site at compile time. A wire need on any single member re-routes the whole union to a `Schema.Union` of `TaggedClass` members, because mixing a `taggedEnum` arm with a decoding arm splits one concept across two owner families.
- A union mixing equality-bearing and equality-free members is itself the signal that two concepts were fused: split the union by codec/equality demand, not by tag count. Heterogeneous identity regimes inside one `Schema.Union` is the rejected shape — every member of a union shares one identity tier.

[CONSTRUCTOR_VALIDATION_BOUNDARY]:
- The generated owner's constructor validates by default; `make(props, { disableValidation: true })` (and the `new` second argument, typed `MakeOptions = boolean | { disableValidation?: boolean }`) is the deliberate trusted-source escape for a value already proven valid upstream — the in-owner answer to re-validation at a known-good seam, never a second un-validating sibling constructor. The decision is per-call-site, riding the existing owner, not a fork into a parallel shape.
- The owner's constructor is the only admission path that runs filter/brand refinements eagerly; a value built by spreading object fields and asserting the class type bypasses every refinement silently. Construction through `new`/`make` is therefore load-bearing for correctness, not stylistic — the bypass is the boundary leak that admits an unrefined value into interior code that assumes the invariant holds.

[EQUALITY_OVERRIDE_BOUNDARY]:
- Structural `Equal`/`Hash` from `Schema.Class`/`Data.*` compares every field by deep structural equality; when a concept's identity is a subset of its fields (two records equal iff their `id` matches, ignoring a mutable `revision`), the default is wrong and the route is `Schema.Class` plus an `equivalence` annotation supplying the custom `Equivalence`, never a hand-written external comparator. `Schema.equivalence(schema)` extracts the resolved `Equivalence` for reuse at comparison sites.
- Subset-identity does not re-route off the generated owner: the custom comparator rides the existing `Schema.Class` as its `equivalence` annotation, so the concept stays one owner rather than splitting into a schema plus a sibling `Equivalence`. A standalone comparator declared beside the schema is the parallel artifact the route deletes — choosing the annotation over the sibling is the whole decision.
- Custom equivalence and a brand are orthogonal routes that compose: a branded-id field inside a `Schema.Class` whose `equivalence` compares only that field gives a value whose identity is its id and whose id is nominally distinct from other same-base ids — one owner carrying both the nominal axis and the subset-identity axis.

[BRAND_TERMINAL]:
- `Schema.brand("X")` over a filtered base is the correct terminal owner for a nominal primitive that carries a real refinement: the resulting `BrandSchema` exposes `make(unbranded)` that validates, and the static type is `Schema.Schema.Type` = `base & Brand<"X">`, fused in one pipe. No standalone `type` alias restates the brand.
- A brand with no runtime predicate and no real nominal collision is deleted, not kept: `Brand.nominal<X>()` applies zero runtime checks and only distinguishes same-base values by meaning, so when no two same-base values are ever confused, the bare primitive stands and the brand is ceremony. The deletion is a law, not a preference.
- `Brand.refined<X>(predicate, onFailure)` (or the `Option<BrandErrors>`-returning overload) is the bare-`Brand` terminal when no codec/wire need exists; its `Constructor` carries `.option`, `.either`, and `.is` alongside the throwing call. Reach for `Schema.brand` instead the moment the branded value crosses a decode boundary, because only the schema form participates in `Schema.decodeUnknown`.
- The nominal-tier promotion is monotone: above the bare primitive sits `Brand.nominal` (meaning-only distinction), above it `Brand.refined(predicate, onFailure)` (a runtime invariant exists to check), above it `Schema.brand` (the value crosses a decode boundary). Each rung is reached only when its discriminant appears; an unused rung is the over-tier the demotion law strips.

[MULTI_INVARIANT_NOMINAL_FUSION]:
- A nominal primitive carrying two or more independent refinements is one fused constructor, not a chain: `Brand.all(a, b, c)` composes the constructors under `Brand.EnsureCommonBase`, a type-level constraint comparing each brand's `Brand.Unbranded<Brand.FromConstructor<B>>` against the first's and resolving to `"ERROR: All brands should have the same base type"` on mismatch, so a brand whose unbranded base differs from the first is rejected at the call to `all`, not at runtime. The reject is `c(b(a(value)))` nesting, which loses the simultaneous error accumulation.
- `Brand.all` returns a `Constructor<UnionToIntersection<Brands>>` whose `.either` collects every failed refinement into one `Brand.BrandErrors` array rather than short-circuiting at the first — the multi-invariant primitive reports all violations in one pass, the property a nested-constructor chain cannot give. The fused constructor carries `.option`, `.either`, and `.is` alongside the throwing call, so the recovery modality is selected at the call site without a second constructor.
- `Schema.fromBrand(constructor, annotations)` lifts an existing `Brand.Constructor` into a `BrandSchema` reusing the constructor's predicate as the schema's filter, so the brand declared once for in-memory use participates in `Schema.decodeUnknown` at the boundary without restating the refinement as a `Schema.filter`. The reject is a `Schema.brand` re-spelling the same predicate beside the `Brand.refined` constructor — one invariant, two declarations that drift.

[CONSTRAINT_AXIS_IS_NOT_THE_SHAPE_AXIS]:
- A constraint (an invariant narrowing a value, a remap rewriting its representation) is a seventh routing question orthogonal to the five-axis shape tuple: the shape axes decide the owner family, the constraint axis decides which combinator attaches the invariant *inside* that owner, and conflating them is the error that spawns a sibling type beside the schema. The constraint never earns its own owner — it rides the owner the shape axes already chose.
- The constraint fork reads two independent bits before any combinator is named: does the invariant introduce a *nominal* distinction the type system must track (brand), or only *prove a property* of the existing type (refine); and is the invariant *pure* (decidable from the value) or *service-consulting* (decidable only against an `R` requirement). The four combinations route to four distinct combinators, and choosing on a third bit — line count, perceived complexity — is the misroute.
- A refinement that narrows `A` to a subtype `B extends A` without minting a new nominal identity routes to `Schema.filter`, whose return type is `refine<C & B, From>` — the `C & B` intersection narrows the static type at the type level while the runtime stays the base, so the consumer reads the narrowed type with zero brand ceremony. The reject is `Schema.brand` for a constraint that distinguishes nothing — a `filter` that returns a same-base subtype is the route the deletion law leaves standing.
- The brand-vs-filter discriminant is whether two same-base values that both satisfy the invariant must still be kept apart by meaning: a positive integer that is also a valid count where counts must never be assigned to indices is nominal (brand); a string that is merely non-empty is a refined type the rest of the program treats as a string (filter). Nominal collision is the only signal that promotes `filter` to `brand`.

[SERVICE_CONSULTING_REFINEMENT]:
- A refinement decidable only against a dependency — a uniqueness check against a store, a policy lookup, a cryptographic verification — routes to `Schema.filterEffect`, whose predicate returns `Effect<FilterReturnType, never, FD>` and whose result is a `filterEffect<S, FD>` extending `transformOrFail<S, SchemaClass<Type<S>>, FD>`: the `FD` requirement threads into the schema's `R` context, so the constraint's dependency rides the schema graph and is provided once at the same composition root as every other requirement. The reject is a post-decode `Effect` guard the interior must remember to call — that guard lives outside the schema, fires after admission, and re-admits an unverified value.
- `filterEffect` is the only refinement combinator that lifts the owner's context from `never`, and that lift is the load-bearing signal: a schema whose `R` is non-`never` carries a service-consulting invariant in its decode, and `Schema.decodeUnknown` over it returns an `Effect` requiring that service rather than a bare value. A pure `filter` keeps `R = never` and decodes synchronously; the context parameter is the type-level proof of which kind of refinement was attached.
- The refinement predicate's `FilterReturnType` is richer than a boolean: it returns `undefined | boolean | string | ParseIssue | FilterIssue | ReadonlyArray<FilterOutput>`, so a single `filter`/`filterEffect` reports *which* sub-invariant failed by returning a `FilterIssue { path, message }` array — multiple independent invariants on one field accumulate their failures in one combinator rather than chaining three filters that short-circuit at the first. The array return is the in-owner answer to multi-invariant refinement on a non-nominal field; `Brand.all` is its nominal counterpart.

[REPRESENTATION_REMAP_FORK]:
- A constraint that *changes the representation* between wire and domain — not narrows it — routes to `transform`/`transformOrFail`, never `filter`: `filter` keeps `Encoded === Type` and only proves a property, `transform` declares `From` and `To` schemas with distinct `Type`/`Encoded` and a decode/encode bracket. The discriminant is whether the encoded and decoded values differ in *shape* (transform) or only in *which subset is admitted* (filter). A `filter` used to reshape silently keeps the wrong encoded type; a `transform` used to merely narrow spawns a needless second schema side.
- `transformOrFail` is the fallible remap and `transform` the total one: the fork is whether the decode can reject the input (`transformOrFail` returns `Effect<To, ParseIssue, R>`) or is a total bijection (`transform` returns `To` directly). A total remap spelled as `transformOrFail` with a never-failing decode is the ceremony a `transform` deletes; a fallible remap spelled as `transform` plus an external validation step is the boundary leak `transformOrFail` closes.
- A closed wire↔domain literal correspondence — a set of stored codes mapped to domain literals — routes to `Schema.transformLiterals(...pairs)`, which expands `[from, to]` pairs into a `Union` of `transformLiteral` members, deriving the decode, the encode, and the exhaustive `Type`/`Encoded` literal unions from the one pair table. The reject is a `Schema.Literal` of codes plus a hand-written `Match` mapping each code to its domain literal plus an inverse map for encode — three artifacts the pair table collapses into one, where a new code lands as one pair and both directions update.
- `transformLiterals` is the correct owner for a remap whose entire content *is* the correspondence (no per-value computation); the moment the decode must compute a value rather than look it up, the route is `transformOrFail`. The pair table is a data-driven remap; the computed decode is a logic-driven one, and the data form is preferred wherever the correspondence is enumerable.
- `Schema.compose(from, to)` chains two transforms into one schema when the decoded `Type` of `from` equals the `Encoded` of `to` (`Schema.Type<From> = Schema.Encoded<To>`) — the route for a two-stage admission (raw → intermediate → domain) that stays one owner; an `Encoded`-mismatch is admitted only under `{ strict: false }`, which relaxes the boundary check. The reject is decoding to the intermediate, reading it, then decoding again at the call site, which splits one admission across two seams; composition is the in-owner answer to staged decode, never a second schema the interior threads by hand.

[SYNTHESIZED_DISCRIMINANT]:
- A structural wire shape carrying no discriminant field, that the interior must route by tag, routes through `Schema.attachPropertySignature("_tag", literal)`: it injects a `_tag` literal *at decode time* that is absent from the input shape, turning a tag-free wire union into a tagged domain union without polluting the encoded type with a field the source never sends. The reject is a `Schema.transform` that hand-adds the tag, or a domain shape that demands the producer stamp a discriminant it does not own.
- The injected-vs-declared discriminant decision is part of the owner choice: a producer that *sends* a discriminant routes to `Schema.TaggedClass`/`Schema.TaggedStruct`/`Schema.tag` (the `_tag` is part of the wire shape and round-trips); a producer that sends only a structurally-distinguishable shape (a payload present in one arm, absent in another) routes to `attachPropertySignature` over a `Schema.Union`, where the tag is *derived from structure* on decode and *stripped* on encode. Demanding the wire carry a tag it has no field for is the boundary the synthesized discriminant removes.
- `Schema.tag(literal)` declares a single-value literal field whose `make` defaults it — `Schema.tag` inside a `Schema.Struct` is the manual spelling of what `TaggedStruct`/`TaggedClass` inject, and is the route only when the discriminant key is not `_tag` or the struct already carries other tags. The reject is a `Schema.Literal("X")` field that the constructor demands at every call site; `tag` makes it omittable-and-defaulted in one declaration.

[FIELD_MODALITY_VS_OWNER_MODALITY]:
- A field whose presence varies across contexts forks on whether the variance is per-channel or per-call. Per-call optionality is a field modifier — `Schema.optional`, `Schema.optionalWith({ as: "Option", default, exact, nullable })` — staying inside one `Schema.Class`. Per-channel optionality (present on read, absent on insert) is the `Model.Class` route, because the variance is a property of the persistence variant, not the value.
- The decisive trap: reaching for `Schema.partial(self)` or a `pick`/`omit`/`pluck` chain to spawn a "partial variant" of a class is the sibling-schema spawn `Model.Class` exists to delete — each returns an anonymous `SchemaClass` that drifts independently on field change and carries no relationship to its source. Three such derived schemas off one owner is the collapse trigger that mandates the `Model` re-route; one is tolerable only as a genuinely local projection consumed once.
- `Schema.optionalWith({ as: "Option" })` is the in-owner answer to "absence is a domain state": the field decodes to `Option<A>` and the absence rail is the value, no second schema. Reach past it to `Model.FieldOption` only when the same field must be `OptionFromNullOr` on the database channel and `optionalWith` on the json channel simultaneously — the dual-channel split is the `Model` signal, the single-channel `Option` is not.
- `Schema.pluck(schema, key)` extracting a single field's schema is a read-time projection, never an owner: using it to define a concept ("the id type is `pluck(User, "id")`") inverts ownership — the field's brand or schema is declared once on the owner and the owner is the source, not a projection of it. `pluck` is admitted only as a transient decoder for an external single-value payload.

[BEHAVIOR_TABLE_VS_GENERATED_OWNER]:
- The `as const` table and the generated owner are not interchangeable along the cardinality axis; they are split by whether the variants carry per-instance payload. A closed keyset of fixed behavior rows (each key maps to the same-shaped static record) is the table read by indexed access; a closed set of variants each carrying distinct per-instance fields is the generated owner. Counting cases never decides this — payload uniformity does.
- A table whose rows reference a function or a later-declared owner is no longer dependency-free and re-routes out of the `as const` exemption: the exemption holds only for the decode-free, dependency-free, behavior-by-key surface. A row holding a `Schedule`, a `Layer`, or a closure is a policy-value table that belongs with the vocabulary it dispatches, not a launchpad for derived predicates built atop it.
- The table promotes to `Data.taggedEnum` the moment a row needs to carry call-site data alongside its key: a `severity` key with a fixed `rank` is a table row, but a `severity` event carrying a per-occurrence `message` and `at` is a tagged variant. The discriminant is whether the variant's payload is known at declaration (table) or supplied at construction (generated owner).

[RAW_TABLE_EXEMPTION]:
- The `as const` table is admitted only as the decode-free, dependency-free vocabulary owner, and even there it is a behavior-column dispatch surface read by `keyof typeof` and indexed access — never a launchpad for hand-rolled type derivation built atop it. A standalone `is-*` predicate or union-arithmetic utility computed from a table is the rejected register regardless of correctness.
- The table earns its place over a generated owner exactly when there is no value identity to compare, no codec to derive, and no per-occurrence payload — pure literal-keyed behavior. Adding any one of equality, codec, or wire need promotes it to `Schema.Class`/`Data.taggedEnum` and the table is the wrong owner.

```typescript
const Severity = {
    info: { rank: 0, retain: false, escalateAfter: 0 },
    warn: { rank: 1, retain: true, escalateAfter: 3 },
    error: { rank: 2, retain: true, escalateAfter: 1 },
} as const satisfies Record<string, { readonly rank: number; readonly retain: boolean; readonly escalateAfter: number }>
// rank read inline by indexed access; the behavior columns dispatch together off `keyof typeof Severity`
const escalates = (k: keyof typeof Severity, seen: number): boolean => Severity[k].retain && seen >= Severity[k].escalateAfter
```

[VOCABULARY_OWNER_TIER]:
- The nominal/keyset axis has a sub-tier the structural chooser misses: a concept that *is* a closed set of bare literal values, with no per-value payload and no behavior column, routes to `Schema.Literal(...values)`. The discriminant between `Schema.Literal` and an `as const` behavior table is whether the keys *carry behavior columns*: bare alternatives with no associated data are `Literal`; alternatives each mapping to a static behavior record are the indexed-access table. Promoting `Literal` to a table the moment the first behavior column appears keeps the vocabulary one owner.
- `Schema.Enums(nativeEnum)` is the terminal owner *only* at the seam admitting a foreign or generated TypeScript `enum`; its `Type` is `A[keyof A]` and it exists to interop with an enum the program does not own. A native `enum` declared *for* this concept is the rejected source — `Literal` or a behavior table owns a vocabulary the program declares; `Enums` is the boundary adapter for one the program inherits, and reaching for `Enums` to model a new vocabulary is the foreign-shape smuggled inward.
- A structured-string vocabulary — keys with an enforced lexical grammar, a prefixed or templated identifier family — routes to `Schema.TemplateLiteral(...params)` for the type-only constraint, or `Schema.TemplateLiteralParser(...params)` when the parts must *decode into a typed tuple*. `TemplateLiteralParser` is the owner whose decode splits the string into its typed components and whose encode rejoins them, so a `"prefix:id:version"` string becomes a parsed `[Prefix, Id, Version]` in one schema; the reject is a `Schema.String` plus a regex `filter` plus a manual split — the parser owns the grammar, the membership, and the round-trip.

[REFINEMENT_PLACEMENT_LAW]:
- The constraint attaches at exactly one altitude and the altitude is load-bearing: a field-level invariant lives on the field schema (`name: Schema.String.pipe(Schema.filter(...))`), a cross-field invariant is a whole-record `Schema.filter` *piped over the class schema* (`Account.pipe(Schema.filter(...))`), and a service-consulting cross-field invariant is `Schema.filterEffect` piped over it. A cross-field invariant pushed down to a single field cannot see the sibling it constrains; a field invariant lifted to the record runs on every decode of the whole value when the field alone changed.
- The whole-record `filter` is the in-owner answer to "these two fields must agree" — the predicate receives the decoded record and returns a `FilterIssue` whose `path` points at the offending field, so the failure is attributed precisely without splitting the invariant into a sibling validator. The reject is a post-construction assertion or a smart-constructor wrapper around the class — both re-admit a value the class already produced unchecked.
- Constraint stacking under `pipe` is left-to-right and each combinator narrows the prior: `Schema.String.pipe(Schema.minLength(1), Schema.pattern(re), Schema.brand("X"))` runs three refinements then mints the brand on the proven base, so the brand's `make` is reachable only by values that passed every prior filter. The order is semantic — the brand must be last so the nominal type wraps the fully-refined base, and a brand placed before a filter brands an unrefined value whose later filter the brand's type no longer reflects.
- The cross-cutting derivation an annotation supplies (`equivalence`, `arbitrary`) attaches *after* the constraints because it derives from the refined `ast`; `Schema.equivalence(schema)` over a filtered-and-branded schema yields a comparator that already respects the refinement, so the constraint chain and the derived comparator are one owner. Supplying a hand-written comparator beside the chain is the parallel artifact the annotation deletes.

```typescript
const Slug = Schema.String.pipe(
    Schema.minLength(1),
    Schema.pattern(/^[a-z]+(?:-[a-z]+)*$/),
    Schema.brand('Slug'), // static type is `Schema.Schema.Type<typeof Slug>` = `string & Brand<'Slug'>`, never restated
)
const Tier = Schema.transformLiterals([0, 'free'], [1, 'pro'], [2, 'team'])
class Account extends Schema.Class<Account>('Account')({
    slug: Slug,
    tier: Tier,
    seats: Schema.Number,
    active: Schema.Number,
}) {
    get plan(): Schema.Schema.Type<typeof Tier> {
        return this.tier
    }
}
// cross-field invariant rides the owner at class altitude; FilterIssue.path attributes the failure
const Seated = Account.pipe(
    Schema.filter((a) => (a.active <= a.seats ? undefined : { path: ['active'], message: '<over-seat-limit>' })),
)
const route = Match.type<Schema.Schema.Type<typeof Tier>>().pipe(
    Match.whenOr('pro', 'team', () => '<billed>'),
    Match.when('free', () => '<gratis>'),
    Match.exhaustive,
)
```

[CHOOSER_IS_ITSELF_AN_OWNER]:
- The owner chooser is not a prose checklist run by hand at each concept — it is a closed dispatch over a signature value, and the moment a third concept is routed the chooser collapses to one `Match.type<Signature>()` whose arms are the owner tiers. The signature is the input value, the owner tier is the result, and the routing law is the executable specification, not commentary beside an `if`-ladder.
- The signature is shaped as a discriminated union keyed on the dominant axis, not a flat record of independent fields, because the chooser dispatches by narrowing: each `identity` member carries only the secondary axes its tier reads (a `nominal` member carries `wire`, a `structural` member carries `cardinality` and `wire`, a `reference` member carries neither), so the arm receives a value already narrowed to the axes it routes on. A flat record with parallel literal fields cannot narrow and forces every arm to re-read every axis.
- `Match.type<I>()` and `Match.value(i)` diverge in what terminating returns and the divergence is load-bearing: `Match.type<Signature>()` builds a matcher with `Pr = never`, so the terminal yields a reusable `(s: Signature) => Tier` function — the chooser as a value applied at every routing site; `Match.value(sig)` builds a matcher with `Pr = Signature`, so the terminal yields the resolved `Tier` directly — the chooser inlined for one concept. The reusable chooser is the route the moment a second concept is shaped, and the inlined form is the nested sub-router inside an arm.
- Exhaustiveness over the signature union is the proof the routing law is total: a new dominant-axis member (a fourth `identity` regime, a `stream` payload member) widens the `Signature` union and `Match.discriminatorsExhaustive` fails to compile until its record gains the member's arm — the chooser breaks loudly when an unrouted signature appears, the absorbed-growth guarantee applied to the decision law itself.

```typescript
type Signature =
    | { readonly identity: 'nominal'; readonly wire: boolean }
    | { readonly identity: 'reference' }
    | { readonly identity: 'structural'; readonly cardinality: 'one' | 'many'; readonly wire: boolean }
const chooseOwner: (s: Signature) => string = Match.type<Signature>().pipe(
    Match.discriminatorsExhaustive('identity')({
        nominal: (s) => (s.wire ? 'Schema.brand' : 'Brand.refined'),
        reference: () => 'as-const-table',
        structural: (s) =>
            Match.value(s).pipe(
                Match.when({ cardinality: 'many', wire: true }, () => 'Schema.Union<TaggedClass>'),
                Match.when({ cardinality: 'many', wire: false }, () => 'Data.taggedEnum'),
                Match.orElse(() => 'Schema.Class'),
            ),
    }),
)
```

[DISCRIMINATOR_DISPATCH_VS_PATTERN_DISPATCH]:
- `Match.discriminators(D)` and `Match.discriminatorsExhaustive(D)` narrow each arm by `Extract<R, Record<D, P>>`, so the field `D` must be a genuine discriminant — a union whose members differ by that field — for the arm to receive a narrowed value; a flat record carrying `D` as a free literal-union field narrows every arm to `never` and the arm bodies fail to read any sibling axis — the trap that forces the signature into a tagged union, because the dominant-axis dispatch is unavailable until the signature is shaped as one.
- The exhaustive record form rejects an excess arm at the type level (the record's keys are constrained to the discriminant's literal union) and a missing arm fails the terminal, so `discriminatorsExhaustive` is total by construction — no `Match.exhaustive` tail is needed. The reject is `Match.discriminators` (non-exhaustive) closed by a hand-written `Match.orElse` default that silently swallows an unrouted member.
- `Match.discriminatorStartsWith("kind")(prefix, f)` routes a hierarchical discriminant whose literal is a dotted or prefixed family (a `"db.generated"` vs `"db.sensitive"` column-policy family), folding a prefix family into one arm without enumerating each member; the reject is one `Match.when` per literal sharing a prefix.
- `Match.withReturnType<Tier>()` pins the result type of every arm before the arms are written, so an arm returning the wrong tier shape is caught at the arm, not at the call site — the chooser's codomain is declared once and every arm is checked against it. The reject is an untyped chooser whose arms drift in return shape and surface the mismatch only downstream.
- Inside a `structural` arm where no single field discriminates, `Match.whenAnd({ cardinality: 'many' }, { wire: true }, f)` is the conjunction router that subtracts the joint pattern, and `Match.orElse` catches the remaining `cardinality: 'one'` signatures — the inner sub-dispatch is `Match.value(s)` over the narrowed member, never a re-run of the outer chooser. The reject is two near-identical arms differing only in the literal they match.

[ANNOTATION_COMPLETENESS_AXIS]:
- A sixth axis decides between a raw owner and a generated owner that the five structural axes miss: which derived capabilities (`Equivalence`, `Arbitrary`, `Pretty`, JSON Schema, a decode fallback) the concept demands. A generated owner derives all of them from its `ast` for free; a raw `as const` table or a bare `Schema.declare` derives none, so demanding any one of them on a non-generated owner forces a hand-supplied annotation that the generated owner would have produced — the demand re-routes toward the generated tier.
- The `Annotations.Schema<A>` interface is the closed annotation surface every generated owner accepts: `identifier`, `title`, `description`, `documentation`, `examples`, `default`, `message`, `jsonSchema`, `arbitrary`, `pretty`, `equivalence`, `concurrency`, `batching`, `parseIssueTitle`, `parseOptions`, `decodingFallback`. Each is a derivation hook, not metadata — supplying `arbitrary` swaps the property-test generator the schema yields, supplying `equivalence` swaps the structural comparator, supplying `decodingFallback` makes the schema total on the failing input.
- `decodingFallback: (issue: ParseIssue) => Effect<A, ParseIssue>` is the annotation that converts a fallible field into a total one at the owner, never at the call site: a field with a known default-on-malformed-input policy carries the fallback as a definition-time value, and the reject is a post-decode `Effect.orElse` repeated at every read. The fallback returns `Effect<A>` so it composes a recovered value or re-raises the issue — the recovery policy rides the field.
- `equivalence` and `arbitrary` annotations take the type parameters' own instances as arguments (`EquivalenceAnnotation<A, TypeParameters>` is `(...equivalences) => Equivalence<A>`), so a container or generic owner derives its comparator and its generator from its elements' — the annotation is compositional, not a flat override. Supplying a flat `Equivalence<A>` that ignores the type-parameter slots on a generic owner severs the composition and is the reject.
- The `message: (issue) => string | Effect<string> | { message; override: boolean }` annotation carries the `override` flag deciding whether the message replaces or augments nested issues; a bare-string return carries no `override`, so it cannot replace and is shadowed by inner field messages — the flag is the load-bearing field the string form silently omits. The `Effect<string>` return resolves a message asynchronously against a service.

[ANNOTATION_PLACEMENT_TUPLE]:
- A generated class owner takes annotations as either one `Annotations.Schema<Self>` record or a positional tuple `[typeSideAnnotations?, transformationAnnotations?, encodedSideAnnotations?]` — three independent annotation slots because a `Schema.Class` is a `Transformation` with a decoded type side, the transformation node, and an encoded side. An annotation that must attach to the wire shape (`jsonSchema` describing the encoded payload) goes in slot three, not slot one; placing it in slot one annotates the decoded instance and the generated JSON Schema describes the wrong shape.
- The placement decision is part of the owner choice, not a downstream tweak: a concept whose decoded and encoded representations both need distinct identifiers, descriptions, or arbitraries is the signal the owner carries a non-trivial transform, and the three-slot tuple is where that duality is declared once. The reject is two sibling schemas — one for the type side, one for the encoded side — each annotated separately and drifting.
- `Class.annotations(record)` re-annotates an existing class owner in place, returning a `SchemaClass<Self, I, R>` carrying the merged annotations but *dropping* the `new`/`make` constructor and instance methods — it is a schema view, not the class. The decision is whether the annotation is known at declaration (the slot tuple, which keeps the constructor and methods) or added later for a derived schema view (`.annotations`, which sheds the class); a late identifier or description requirement that must keep the constructor uses the construction-time annotation slot, never `.annotations`.

[DEMOTION_DIRECTION]:
- The chooser runs in both directions: stacking combinators on an over-tiered owner reveals the lower tier the signature actually demands, and the demotion is the inverse of the brand-deletion law applied to every tier. An empty-bodied `Schema.Class` over a struct demotes to `Schema.Data(struct)`; a `Schema.Data` whose value never decodes demotes to `Data.struct`; a `Data.struct` whose value is never compared demotes to a plain `Schema.Struct`; a `Schema.brand` over a base with no codec need demotes to `Brand.refined`; a brand with no predicate and no nominal collision demotes to the bare primitive.
- The demotion trigger is a dead derived capability: a tier carrying a capability no consumer reads is ceremony, and the signature is re-read to find the tier that carries only the demanded axes. Over-tiering is not free — every unused capability is surface a reader must trace to discover it is unused, and `ONE_HOP_RESOLUTION` is violated by a constructor, an `Equal` instance, or a codec that resolves to nothing.
- The demotion is bounded by the same axis-shift law that governs promotion: an irrecoverable axis (identity regime) cannot be demoted across after call sites exist without a hand-migration, so demotion is a definition-time correction caught before consumers bind, not a refactor of a live owner. Demoting a structural-value owner to a reference cell after comparison sites exist breaks every `Equal.equals` call silently.
- The constraint fork demotes by the same law applied to every combinator: a `transform` whose decode and encode are identity is a `filter` wearing two function slots; a `filter` whose predicate is constantly true is dead surface; a `transformLiterals` of one pair is a `transformLiteral`; an `attachPropertySignature` on a union of one member is a `tag` field. Each degenerate combinator demotes to the next-lower one the signature actually demands.

[TERMINAL_DECLARE_ESCAPE]:
- `Schema.declare` is the terminal owner when a concept's decode logic the library cannot generate — a foreign opaque value, a parse that consults a service, a representation no combinator composes. Its unary form `declare(is, annotations)` admits a value by a type guard `(input: unknown) => input is A` with no transformation; its parametric form `declare(typeParameters, { decode, encode }, annotations)` carries explicit `decode`/`encode` Effects whose context is constrained to the type parameters' contexts, so `Schema.to`/`Schema.from` recover a `never`-context schema.
- The `declare` decode and encode return `Effect<A, ParseResult.ParseIssue, never>` — the custom owner participates in the same `ParseError` rail as every generated owner, so a `declare`-routed concept composes inside a `Schema.Struct` field, a `Schema.Union` arm, or a `Schema.Class` field with zero special-casing at the consumer. The reject is a hand-written parser outside the schema graph that the interior must call separately and whose failures live in a different channel.
- `Schema.instanceOf(Ctor, annotations)` is the narrow `declare` for an opaque host class: it admits a value by `instanceof` as a leaf `instanceOf<InstanceType<Ctor>>` schema with no field projection, the correct terminal owner for a platform object embedded as a field that the interior treats as a black box. Reach past `instanceOf` to the parametric `declare` only when the foreign value must decode to a canonical interior shape rather than pass through opaque.
- `declare` is the escape of last resort, not a convenience: every axis the structural owners carry (`Equal`/`Hash`, codec projections, `extend` lineage, the variant schemas of `Model.Class`) is forfeited unless hand-supplied through annotations, so a concept routed to `declare` that later demands structural equality or a wire projection must hand-write what a generated owner would derive. The signature is re-read before reaching for `declare` — a concept expressible by composing generated combinators is never a `declare`, and `declare` survives only where no composition reaches the representation.
- The terminal owner is admitted only at the seam, never inside: a foreign shape is wrapped once at the boundary and the interior sees the decoded canonical owner. `Schema.standardSchemaV1(schema)` exposes a Standard Schema V1 surface for interop with a foreign validator the program does not own, the wire-facing complement of the `instanceOf`/`declare` admission of a foreign value inward.

[COMPOSITION_AXIS]:
- A field whose value is itself an owner-bearing concept adds a sixth routing question the five-axis tuple resolves recursively: the field's own signature is run through the chooser, and the parent embeds the resulting owner schema directly as a field value. The parent never re-declares the child's shape, equality, or codec — it references the child owner, so a child route flip re-flows the parent at compile time with zero parent edits.
- Embed-vs-reference is decided by identity ownership, not by nesting depth: a child whose identity lives *inside* the parent value (a value object the parent fully owns and compares structurally) is embedded as the child schema directly, and the parent's deep `Equal`/`Hash` recurses into it; a child that has its own independent identity and lifecycle (separately persisted, separately mutated) is referenced by its branded id field, and the full child is decoded only at the seam that joins them. Embedding an independently-identified entity duplicates its state across every parent and corrupts the single-source guarantee.
- A `Schema.Class` field typed as another `Schema.Class` is the canonical embed: the inner class's instance methods, getters, and refinements travel into the outer value, and the outer constructor validates the inner recursively. The reject is re-spelling the inner shape as inline `Schema.Struct` fields on the parent — that fork drifts the moment the child gains a field and severs the child's behavior from the embedded copy.
- The composite owner stays one declaration even across owner families: a parent `Schema.Class` may embed a `Data.taggedEnum` variant field, a `Schema.brand`-ed primitive field, and an `as const`-keyed enum literal simultaneously, each child routed independently and fused as a parent field. Heterogeneous child regimes are licensed; what is rejected is a parent re-deriving any child's type, codec, or equality beside the child owner.

[RECURSIVE_OWNER]:
- A self-referential concept (a tree node whose children are the same concept) routes to a `Schema.Class` whose recursive field is `Schema.suspend((): Schema.Schema<Self> => Self)` — the thunk defers the self-reference past the class binding, and the explicit return annotation is mandatory because the inferred type would be circular. The reject is an `interface` plus a hand-written recursive `Equivalence` plus a hand-written recursive decoder; the one class carries recursive codec, recursive `Equal`/`Hash`, and recursive validation.
- The recursive field's type parameters diverge the moment any node field has a distinct decoded vs encoded shape: the thunk is annotated `Schema.Schema<Self, SelfEncoded, SelfContext>` so the recursion threads the `Type`, the `Encoded` projection, and the context. Omitting the second parameter silently collapses the encoded recursion to the decoded type — a codec that round-trips wrong without a diagnostic; omitting `R` collapses the recursion's context to `never`, so a recursive owner whose leaf carries a `filterEffect` loses its service requirement on the recursive arm — the context gate breached silently at the recursion point, admitting an unverified value past the depth the thunk hides. Recursive owners with any inner transform or service-consulting leaf must spell all three parameters.
- Mutual recursion (two concepts each referencing the other) is two `Schema.Class` owners whose cross-reference fields are each a `suspend` thunk pointing at the sibling; the thunks break the declaration-order cycle so neither owner must precede the other, and the sibling's identifier is what the generated JSON Schema `$ref` targets — so the namespace is what closes the cross-owner cycle, not the suspend. The reject is collapsing both into one over-general owner with optional fields standing in for the two shapes — that erases the cardinality the discriminant should carry; two mutually-recursive owners sharing one identifier produce a `$ref` graph pointing each owner at the other's definition, a self-referential corruption the per-owner identifier prevents.
- A recursive *variant* family routes to a `Data.taggedEnum` (in-memory) or a `Schema.Union` of `TaggedClass` members (wire) where the recursive arm's payload field is a `suspend` over the union schema itself; the generated `$match`/`Match.tagsExhaustive` still closes the recursion totally, and a new recursive arm lands as one tagged member the checker chases through every fold. Recursion never re-routes a union out of its owner family — it only adds a `suspend`-typed field to one arm.
- `Schema.suspend<A, I, R>(() => Schema<A, I, R>)` is identifier-less by construction — it is a deferral thunk, not a named owner — so a recursive owner's identity lives on the enclosing `Schema.Class`, and the recursion is named only at the class that owns it. A `suspend` annotated with its own identifier injects a second `$defs` key for the recursion point that competes with the owning class's key; the recursion is referenced through the owner's identifier, and the suspend stays anonymous.

[CONTAINER_OWNER]:
- A concept that *is* a collection-keyed-by-identity (a registry of entities by id, a set of value objects deduplicated by structure) routes to a structural-identity container — `Schema.HashMapFromSelf`, `Schema.HashSetFromSelf`, `Schema.ReadonlyMapFromSelf`, `Schema.ReadonlySetFromSelf` — never a bare `ReadonlyArray` plus a hand-written dedup or lookup pass. The container's membership and key-equality derive from the element owner's `Equal`/`Hash`, so a value object with structural equality is correctly deduplicated and looked up by structure, not reference.
- The `*FromSelf` schemas and their plain counterparts split on what the in-memory value is: `HashMapFromSelf({key, value})` decodes a `HashMap` directly and the runtime value is the persistent map (its `Equal`/`Hash` preserved end-to-end), whereas `Schema.HashMap({key, value})` is a `transform` whose *wire* form is an array of `[key, value]` tuples and whose decoded value is the map. Choose `*FromSelf` when the value is already a persistent container in memory; choose the plain form as the decode boundary that admits a JSON array into a map. Mis-routing a `*FromSelf` over JSON input fails at decode because the input is an array, not a map instance.
- A map keyed by a string-shaped id whose wire form is a JSON *object* (not a tuple array) routes to `Schema.ReadonlyMapFromRecord({key, value})`: the encoded side is a `{ [k]: V }` record and the decoded side is a `ReadonlyMap`, so a string-keyed JSON dictionary becomes a typed map in one schema. The reject is `Schema.Record` plus an `Object.entries`-to-`Map` mapping step — the record schema yields a plain object that loses ordered/typed-key membership.
- `Schema.Record({key, value})` is the container owner only when the concept is an open string-keyed dictionary with no element identity to deduplicate by and no persistent-container semantics — a bag of named slots. The moment elements must be compared or deduplicated by structure, or the key carries a brand, the route is a `*FromSelf` container; the moment keys are a fixed closed set, the route is a `Schema.Struct`, not a `Record`.

[POSITIONAL_OWNER]:
- A fixed-arity heterogeneous sequence (a coordinate pair, a labeled span, a versioned triple) routes to `Schema.Tuple(...elements)` — each positional element its own owner schema — never a `Schema.Struct` with `0`/`1`/`2` keys nor a `ReadonlyArray` of a union element. The tuple's `Type` is a fixed-length labeled tuple the checker indexes positionally, and a wrong-arity literal breaks at compile time.
- `Schema.optionalElement(s)` marks a trailing tuple slot optional and `Schema.element(s)` annotates a required slot; a tuple of required head elements followed by a `Schema.Array(rest)` rest argument (`Schema.Tuple([head], rest)`) is the fixed-prefix-plus-variadic-tail owner — one declaration spelling a head-and-spread shape that a struct cannot express. The reject is a union of fixed-arity tuples enumerating each possible length.
- Tuple vs struct is decided by whether position or name is the load-bearing accessor: a sequence whose elements are addressed by ordinal and whose order is semantic (a pair where swapping breaks meaning) is a `Tuple`; a record whose elements are addressed by name is a `Struct`. A `Tuple` of homogeneous elements is the smell that the concept is actually a `Schema.Array` or a `*FromSelf` container — positional fixed arity over a single element type is rarely the true shape.

[CONTAINER_TIER_RULE]:
- The container choice mirrors the element's identity tier: an element owner with structural `Equal`/`Hash` (`Schema.Class`/`Data.*`) earns a `HashSetFromSelf`/`HashMapFromSelf` because structural keying is now correct and free; an element that is a bare branded primitive earns a `ReadonlySetFromSelf` keyed by the primitive's value; an element with no identity at all stays a `Schema.Array` because a set over reference-equal elements deduplicates nothing. The container tier is read off the element route, never chosen independently.
- A container of independently-identified entities is the place the embed-vs-reference law inverts the element: the registry holds full child owners only when it is the child's canonical home; a secondary index over the same entities holds branded-id elements and resolves the full owner through the canonical registry. Two registries holding full copies of one entity is the multiplicative duplication the id-reference container deletes.
- Nesting a container inside a `Schema.Class` field keeps the parent one owner: a class field typed `Schema.HashMapFromSelf({ key: IdBrand, value: ChildClass })` gives the parent a recursive structural `Equal` that compares the whole map by structure, and a new entry lands as map data with zero schema change. The reject is a parent field of `ReadonlyArray<ChildClass>` plus an external id-keyed lookup function — the lookup is the map the container owner already is.

```typescript
type NodeEncoded = { readonly id: string; readonly children: ReadonlyArray<NodeEncoded> }
class Node extends Schema.Class<Node>("Node")({
    id: Schema.String.pipe(Schema.brand("NodeId")),
    children: Schema.Array(Schema.suspend((): Schema.Schema<Node, NodeEncoded> => Node)),
}) {
    get depth(): number {
        return this.children.length === 0 ? 0 : 1 + Math.max(...this.children.map((c) => c.depth))
    }
}
class Forest extends Schema.Class<Forest>("Forest")({
    roots: Schema.HashMapFromSelf({ key: Schema.String.pipe(Schema.brand("NodeId")), value: Node }),
}) {
    static readonly Indexed = Schema.extend(
        Forest,
        Schema.Struct({ revision: Schema.Number }),
    )
}
```

[REFERENCE_RESOLUTION_BOUNDARY]:
- The id-reference container defers child decoding to one join seam: the parent owner carries branded-id fields, and the join is a single `Effect.all` over the registry container's lookups, never a per-field decode scattered across the parent's construction. The reference owner's correctness is that the parent is decodable without the children present — a dangling reference is a domain state the join surfaces, not a parse failure that blocks parent admission.
- A `*FromSelf` container used as a parent field participates in the parent's structural `Equal` automatically, so two parents are equal iff their containers hold structurally-equal entries — the precise reason the registry holds full owners only at its canonical home: an entity embedded in two parents makes both parents structurally unequal to a third holding a referenced id, splitting one identity across incompatible equality regimes. The reference-by-id discipline keeps every parent comparing on the id, the registry comparing on the structure, one identity per concept.

[CALCULUS_IS_THE_ROUTE]:
- The owner is chosen by counting derived members, not by reading prose: each concept demands a facet set — value, decoded type, encoded type, codec, structural `Equal`/`Hash`, key projection, an `Arbitrary`, an `Equivalence`, a `Pretty`, and one persistence projection per channel — and the route is the owner whose single declaration yields the largest covered subset of that set. The chooser's output is the owner that minimizes hand-spelled artifacts, and the count is computed before the owner is named.
- Naive artifact count is multiplicative across three independent dimensions — channels `C`, variant cardinality `V`, derived facets `F` — so a hand-spelled concept costs `C × V × F` declarations that drift independently, while the generated owner collapses the product to one declaration whose `C × V × F` members are derived by indexed access. The route is decided by where on the lattice the product crosses the cost of the owner's one declaration, never by the absolute size of any single dimension.
- The facet count `F` is the per-owner constant the lattice is read against: `Schema.Class` covers eight facets in one declaration (constructor, `.fields`, `.ast`, decoded `Type`, encoded `Encoded`, codec, `Equal`/`Hash`, `make`) plus every annotation-derived facet on demand; `Data.taggedEnum` covers five per member (constructor, `Type`, `Equal`/`Hash`, the `$is` guard, the `$match` arm) and zero codec; an `as const` table covers one (keyed lookup). The owner with the higher `F` wins exactly when the concept demands the facets it uniquely carries.
- The break-even is per-member, never per-concept: a tagged union of `M` members each needing `F` facets costs `M × F` hand-spelled artifacts against `M + 1` generated declarations (one union, `M` members), so the generated owner overtakes the hand chain at the first member whose `F ≥ 2` — a two-facet member already pays, and counting members before facets is the misread that keeps a two-member union hand-spelled.

[CROSS_PRODUCT_LATTICE]:
- The three dimensions are read in fixed precedence because each higher dimension's owner subsumes the lower: identity-and-facets (`F`) chooses the owner family, variant cardinality (`V`) chooses tagged-union-vs-single inside it, channel count (`C`) chooses variant-owner-vs-flat-owner inside that. Reading `C` before `F` is the misroute that reaches for `Model.Class` on a concept with no structural identity — the variant cross-product is a property layered onto an identity-bearing owner, never a substitute for choosing one.
- Each dimension at unit is a degenerate axis that demotes: `C = 1` demotes the variant owner to a flat `Schema.Class` (no channel divergence to project), `V = 1` demotes the tagged union to a single member (no discriminant to fold), `F = 1` demotes the owner to an `as const` table (only keyed lookup demanded). The lattice corner `C = 1, V = 1, F = 1` is the bare branded primitive or raw literal — the owner that carries exactly one facet, and any richer owner there is dead surface the demotion law strips.
- The lattice grows monotonically: a concept at `(C, V, F)` whose channel count rises to `C + 1` lands the new channel as one variant in the alphabet (a `VariantSchema.make` member, every `Model.Field` config gaining one key), whose cardinality rises to `V + 1` lands the new variant as one union member, whose facet demand rises lands as one annotation on the existing owner. Every dimension's growth is one declaration inside the owner; a growth that forces a new sibling surface is the proof the owner was sized for the instance, not the cross-product.
- The anticipatory reading sizes the owner for the product, not the corner: the moment a second channel, a second variant, or a second derived facet is conceivable, the owner is declared at the variant tier so the next requirement lands as a row, a member, or an annotation — sizing at the corner and widening after consumers bind multiplies the `C × V × F` edits the cross-product owner makes one. The cost asymmetry is the law — early over-shaping is one unused variant key, late under-shaping is a fan-out across every channel and consumer. The variant cross-product never absorbs a *cardinality* growth that is a new identity: a new variant member carrying a distinct discriminant is a `Model.Union` member (the cardinality dimension), never a new variant in the alphabet (the channel dimension), and the union-vs-alphabet decision is read off whether the new requirement is *the same value seen differently* (channel) or *a different value sharing a discriminant* (cardinality).

[VARIANT_AXIS_IS_GENERAL]:
- The persistence dimension is not a fixed `read/write/json` triple — it is an open variant set declared once by `VariantSchema.make({ variants, defaultVariant })`, which returns the whole owner toolkit (`Struct`, `Class`, `Union`, `Field`, `FieldOnly`, `FieldExcept`, `fieldEvolve`, `fieldFromKey`, `extract`) specialized to that variant alphabet. The six-projection `Model.Class` is one instantiation over `variants: ["select", "insert", "update", "json", "jsonCreate", "jsonUpdate"]` with `defaultVariant: "select"`; a domain whose channels are not database CRUD declares its own variant alphabet and inherits the identical generated cross-product.
- `Model.Class` is the owner when payload timing is per-occurrence: one declaration yields six derived schemas — base/select (`X`), `X.insert`, `X.update`, `X.json`, `X.jsonCreate`, `X.jsonUpdate` — each a real `Schema.Struct` projecting only the fields a given channel admits. The reject is six sibling schemas hand-derived by `pick`/`omit`, each drifting on field change.
- The `defaultVariant` is the projection the bare owner *is*: `VariantSchema.Class`'s instance type is `ExtractFields<Default, Fields, true>`, so the class read without a variant suffix is its default projection and every other variant is a named member `[V in Variants[number]]`. Choosing the default wrong forces every consumer to spell the suffix; choosing it as the most-read channel makes the common path suffix-free and the rare channels explicit.
- A custom variant set is the route the moment a concept has three or more representation channels that are neither pure wire (`Schema.transform`) nor pure persistence CRUD — a `public`/`internal`/`audit` projection family, a `summary`/`detail`/`export` family — because the hand-spelled alternative is `C` sibling schemas off one base, each a `pick`/`omit` that drifts. The variant owner is the cross-product collapse the moment `C ≥ 3`; below that the in-owner `Schema.optionalWith` modifier carries per-call presence without a variant alphabet.
- The variant alphabet is closed and total: `Struct.Validate<Fields, Variants[number]>` rejects a field config naming a variant outside the alphabet at the type level, so a field declared for a non-existent channel fails at declaration, not at projection — the alphabet is the checked vocabulary every field's per-variant config is validated against.

[FIELD_IS_A_PER_VARIANT_FUNCTION]:
- A field in a variant owner is not one schema — it is a function from variant to schema, declared by `Model.Field({ select: A, insert: B, json: C })` (a `Field.ConfigWithKeys` whose excess keys are rejected `[K in Exclude<keyof A, Variants[number]>]: never`), so a single field carries a distinct admitted shape per channel without spawning a sibling schema per channel. The column-policy modifiers are pre-composed `Field` configs; `Model.Field` is the open form for a per-variant shape no canned modifier names.
- Field modifiers encode the per-variant projection as data, not branches: `Model.Generated(S)` admits the field in select/update/json but omits it from insert; `Model.GeneratedByApp(S)` admits it in all database variants but omits json-create; `Model.Sensitive(S)` admits it in select/insert/update but strips it from all json variants; `Model.FieldOption(S)` makes it `OptionFromNullOr` in database variants and `optionalWith({as: "Option"})` in json. The modifier is the column-policy value; selecting it is the whole decision.
- `Model.DateTimeInsert`/`Model.DateTimeUpdate` (and the `FromDate`/`FromNumber` variants) carry an `Overrideable` that defaults to the current `DateTime.Utc` on the write channel and a plain `DateTimeUtc` on select/json — the timestamp policy rides the field, never an external mutation step. `Model.Override(value)` brands a value to bypass the auto-default at a call site.
- `Model.FieldOnly(...variants)(schema)` admits a field on exactly the named channels and omits it elsewhere; `Model.FieldExcept(...variants)(schema)` is its complement. The two are the closed-set and complement spellings of the same per-variant presence function, and choosing between them is choosing whether the field's home set or its absent set is shorter to name — both collapse to one `Field` the cross-product reads, never an optional field plus a per-channel branch.
- `Model.fieldEvolve({ insert: (s) => f(s) })` rewrites a field's schema on selected variants from the existing per-variant schema, so a field whose insert channel must wrap the select channel's schema (a write-side transform over a read-side type) is one evolution step, not a second field. The evolution is keyed on the same variant alphabet, so a field's per-channel divergence is derived from its base rather than re-declared per channel.
- `Model.fieldFromKey({ select: "<wire-name>" })` renames a field's wire key per variant via `fromKey.Rename`, threading the rename through the `PropertySignature`'s `Key` parameter, so a field whose stored column name differs from its domain name on one channel carries the rename as a per-variant config — the reject is a `Schema.propertySignature(s).pipe(Schema.fromKey(...))` repeated per sibling schema.

[REPOSITORY_IS_A_DERIVED_FACET]:
- The CRUD surface is itself a member of the facet set the variant owner yields: `Model.makeRepository(Model, { tableName, spanPrefix, idColumn })` returns an `Effect` over `insert`, `insertVoid`, `update`, `updateVoid`, `findById`, `delete` derived directly from the model's variant schemas — `insert` consumes `S["insert"]["Type"]`, `findById` returns `Option<S["Type"]>` keyed on `Schema.Schema.Type<S["fields"][Id]>`, and `delete` keys on the same id field type. No hand-written column-to-field mapping exists; the repository is the variant cross-product read as operations.
- The id column is type-checked against the model's field set at the route: `Id extends (keyof S["Type"]) & (keyof S["update"]["Type"]) & (keyof S["fields"])` rejects an `idColumn` that is not simultaneously a select field, an update field, and a declared field — an id chosen on a column absent from the update projection fails at the repository declaration, so the repository's correctness is the variant owner's field discipline surfaced as a constraint.
- The repository threads the model's context through every operation: each derived method's `R` is `S["Context"] | Schema.Schema.Context<S["fields"][Id]>` (or the variant's own context for `insert`), so a model whose field carries a `filterEffect` service requirement propagates that requirement into every CRUD operation's signature — the service-consulting field's dependency rides the entire derived surface, provided once at the composition root.
- A new column lands once and flows everywhere: adding a `Model.Field` to the model widens all six projections, every repository operation's parameter and return type, and the JSON Schema simultaneously, with zero edit to any operation body. The diff of the next column is one field declaration; the absence of any operation-body edit is the proof the repository is a derived facet, not a hand-written surface.

[UNION_OF_VARIANTS]:
- A tagged union whose members each carry the full persistence cross-product routes to `Model.Union(...members)` / `VariantSchema.Union(...members)`, whose result is `Union<Members> & Union.Variants<Members, Variants[number]>` — one declaration yielding a tagged union *and* its six per-channel projections, so the union's `insert` projection is the union of each member's `insert` shape. The reject is a hand-spelled `Schema.Union` plus six sibling unions, one per channel, each enumerating every member.
- The variant-union is the route the moment both the cardinality dimension `V ≥ 2` and the channel dimension `C ≥ 3` are non-trivial: the naive cost is `V × C` sibling schemas (each member projected to each channel) while the owner is `V + 1` declarations (each member, one union), so the collapse ratio is the full cross-product. A union of variant members where any single member is not a `VariantSchema.Struct` fails the `Members extends ReadonlyArray<Struct<any>>` constraint — every member of a variant union shares the variant alphabet.
- Mixing a variant member with a flat `Schema.Class` member in one `Model.Union` is the signal two concepts were fused: the variant union's projections cannot derive a channel the flat member does not declare, so a heterogeneous member set splits by whether each member participates in the cross-product. A concept where only some members are persisted is two owners — a persisted variant union and a transient tagged union — joined at the seam, never one union straddling both regimes.

[DERIVED_PROJECTION_TRIO]:
- The wire dimension is internalized by three derived projections every schema yields, so a concept never spells its encoded twin: `Schema.typeSchema(s)` projects the decoded-side schema (`SchemaClass<A>`, the value with all transforms applied), `Schema.encodedSchema(s)` projects the wire-side schema (`SchemaClass<I>`, the raw input shape), and `Schema.encodedBoundSchema(s)` projects the wire side clamped at the outermost transformation boundary. The reject is a hand-spelled `XEncoded` interface declared beside `X` — the encoded shape is `Schema.Schema.Encoded<typeof X>`, never restated.
- `encodedSchema` and `encodedBoundSchema` diverge on nested transforms and the divergence is load-bearing: `encodedSchema` recurses through every inner transform to the deepest raw representation, while `encodedBoundSchema` stops at the first transformation boundary, preserving inner refinements' decoded types. A wire contract describing the outermost payload uses `encodedBoundSchema`; a contract describing the fully-raw storage uses `encodedSchema`, and confusing them yields a JSON Schema for the wrong representation depth.
- `Schema.pick(...keys)` and `Schema.omit(...keys)` are the per-projection multiplier the variant owner deletes: each returns an anonymous `SchemaClass<Pick<A, K>>` / `SchemaClass<Omit<A, K>>` that carries no relationship to its source, so three pick/omit projections off one base is the `C × V` sibling spawn the variant owner internalizes as named members. A pick/omit chain survives only as a genuinely single-consumer transient read; a recurring projection is a channel the alphabet should name.

[TWO_COORDINATES_BESIDE_SHAPE]:
- The five shape axes choose the owner family, but two further coordinates decided at the same declaration govern whether the chosen owner is *routable* across the corpus, and both fail silently when wrong: the owner's `identifier` string, which is its key in a global namespace consumed by JSON Schema `$ref`/`$defs`, parse-error attribution, and AST de-duplication; and the owner's `R` context parameter, which gates which extraction, embedding, and synchronous surfaces accept it. A correct shape with a colliding identifier or an unexpected non-`never` context is a misroute the shape axes cannot detect.
- The identifier is mandatory on `Schema.Class<Self>(identifier)` and on `Class.extend<Next>(identifier)`, but optional on `Schema.TaggedClass<Self>(identifier?)`, `Schema.TaggedError<Self>(identifier?)`, and `Schema.TaggedStruct` — the tagged owners default their identifier to the tag value, so a tagged variant needs no second name while a structural class must mint one. Supplying an identifier to a tagged owner overrides the tag-default and is the route only when two distinct concepts share a `_tag` literal and must stay namespace-distinct.
- The identifier is not metadata: it materializes as the `identifier` annotation on the owner's `ast`, is read by `getIdentifierAnnotation`, and doubles as the default `title` — so it surfaces verbatim in every `ParseError` path string and every generated JSON Schema definition key. A concept's identifier is its corpus-global true name, chosen with the same discipline as the value's canonical term, never a throwaway label.
- The two coordinates are orthogonal to each other and to shape: an `as const` table carries neither (no `ast`, no context), a `Schema.brand` carries an identifier-via-annotation and `R = never`, a `filterEffect`-bearing `Schema.Class` carries both an identifier and a non-`never` `R`. The owner choice is incomplete until both coordinates are read off the same five-axis signature that chose the family.

[CONTEXT_PARAMETER_IS_A_ROUTABILITY_GATE]:
- The `R` in `Schema.Schema<A, I, R>` is `never` for every owner whose decode is pure, and non-`never` exactly when a `filterEffect` or a `transformOrFail` whose decode consults a service is attached — and that lift is a routability fact, not a rail fact: a non-`never` owner is rejected at the type level by every surface constrained to `Schema.AnyNoContext` (`= Schema<any, any, never>`), so the construction decision is read off the owner's context before the owner is embedded anywhere.
- `ParseResult.decodeUnknownSync`, `decodeSync`, and `encodeSync` are typed `<A, I>(schema: Schema<A, I, never>)` — they accept *only* `R = never`, so a service-consulting owner cannot be decoded or encoded synchronously at all; the synchronous extraction surface is unavailable the moment a `filterEffect` field lands. `validateSync`, `is`, and `asserts` are typed `<A, I, R>` and tolerate any context (they run the synchronous filter chain and ignore the effectful one), so the gate is asymmetric: a service-consulting owner is still type-guardable but not sync-decodable.
- `Schema.TemplateLiteral` and `Schema.TemplateLiteralParser` accept parameters typed `Schema.AnyNoContext | AST.LiteralValue`, so a structured-string vocabulary whose part schema carries a service-consulting refinement cannot be a template parameter — the part must be a pure refinement. A vocabulary part needing a service lookup re-routes the whole grammar out of the template form into a `filterEffect` over the assembled string, because the template parameter slot structurally forbids the context.
- The context is recovered, never restated, by `Schema.Schema.Context<typeof X>` (the third `Variance` slot), the companion to `Type` and `Encoded`; the owner's requirement set is read off the owner exactly as its static type is, so the layer that provides the service is selected against `Schema.Schema.Context<typeof X>` at the composition root with no hand-tracked requirement list.
- An owner whose `R` rises to non-`never` after consumers bound to its synchronous extraction breaks every `decodeSync`/`encodeSync` call site at compile time — the loud-break guarantee applied to the context coordinate, where the broken sites are the exact inventory of the lift to `Schema.decodeUnknown` and a sync site that survives is one the new requirement does not reach. The service-consulting refinement cannot be added to a sync-extracted owner without surfacing every sync site.

[IDENTIFIER_IS_THE_CORPUS_NAMESPACE]:
- `JSONSchema.make(schema)` and `JSONSchema.fromAST(ast, opts)` lift the identifier into the JSON Schema definition graph: an owner carrying an `identifier` becomes a `$ref` into `$defs/<identifier>`, and the `definitionPath` (default `"#/$defs/"`) plus `topLevelReferenceStrategy` (`"keep"` default, `"skip"`) decide whether the top-level owner is a named reference or inlined. An owner with no identifier inlines its full shape at every use site, so a recursive or widely-embedded owner without an identifier explodes the schema instead of referencing one definition.
- Two distinct owners sharing one identifier collide in the `$defs` namespace — the second silently overwrites the first's definition while both `$ref` the same key, so a single JSON Schema describes one shape for two concepts with no diagnostic. The identifier is therefore a globally-unique key, and the collision is the namespace failure the structural-shape axes cannot see; uniqueness is read against the whole `$defs` namespace, not the local module.
- `getJSONIdentifier` reads the `JSONIdentifierAnnotation` (keyed by `JSONIdentifierAnnotationId`) first and falls back to the plain `identifier`, so a concept whose wire-definition name must differ from its domain identifier carries that `$defs` name as a separate `jsonSchema`-side annotation — the dual-name need is the signal an owner's domain term and its published contract name diverge, declared once as two annotations on one owner rather than a sibling schema renamed for the wire.
- The identifier feeds parse-error attribution: a `ParseError` over a deeply-nested owner names each level by its identifier in the issue path, so an unnamed nested owner produces an anonymous structural path that cannot be traced to a concept. Naming every embedded owner is the precondition for legible decode failures, and the missing-identifier path is the diagnostic cost the `as const`-table exemption pays — a table has no path to attribute.
- An owner reused at three or more sites with its full shape inlined (no identifier) is the JSON-Schema-explosion re-route trigger: minting an identifier collapses the inlined copies to one `$defs` definition and N `$ref`s, the namespace-level analogue of the variant cross-product collapse. An owner referenced more than once earns an identifier the moment the second reference is conceivable; leaving a widely-embedded owner anonymous and accepting the duplicated definitions is the reject.

[FAILURE_CONCEPT_FUSES_SHAPE_AND_CARRIER]:
- A concept that *is* a failure routes to `Schema.TaggedError<Self>(identifier?)(tag, fields)`, whose generated type extends `Class<...>` with `cause_.YieldableError` in its `Proto` slot — the owner is simultaneously a shape (codec, `Equal`/`Hash`, constructor), a discriminated `_tag` member, and a yieldable value the typed failure channel carries directly. The reject is a `Schema.TaggedClass` shape declared beside a separate error type; the fault concept is one owner that is its own carrier.
- The `TaggedError` identifier defaults to the tag exactly as `TaggedClass`, so a fault family is a `Schema.Union` of `TaggedError` members each named by its tag, and a new fault lands as one tagged member whose tag is its namespace key — the closed `E` alphabet and the `$defs` namespace are the same set of literals. A fault whose identifier must differ from its tag (two fault concepts sharing a tag across bounded contexts) supplies the explicit identifier, the same namespace-collision escape the structural class uses.
- `TaggedError`'s constructor omits `_tag` (`Struct.Constructor<Omit<Fields, "_tag">>`) and the yieldable instance is constructed with `new`/`make`, so the fault value participates in structural-key membership and in the parse rail with one declaration — a fault deduplicated in a `HashSet` of recent failures or matched by `Match.tagsExhaustive` is the same owner the channel yields. The fault concept never splits across a shape owner and an error owner; the tier that carries both is `TaggedError`.
