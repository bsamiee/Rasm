# Events and Threads

[SUBSCRIPTION_IS_A_DISPOSABLE_VALUE]:
- A subscription is the detacher token returned by the attach site, and detach requires the exact delegate identity used to attach: the wrapper delegate is stored at subscribe time and reused for `-=`; a lambda registered inline loses its reference immediately and is undetachable. The canonical owner is one sealed `IDisposable` holding a `Seq<Action>` of detach lambdas, each closing over the exact wrapper instance, never a secondary reference to the handler.
- `CancellationTokenRegistration` is the model subscription: `Dispose` unregisters and, when the callback is mid-flight on another thread, blocks until it completes; `DisposeAsync` gives the same rendezvous without occupying a thread. A token already cancelled at registration fires the callback synchronously on the calling thread before `Register` returns and enqueues nothing — attach itself can run the handler inline.
- Every `Register` overload captures `ExecutionContext` unconditionally — `useSynchronizationContext` and EC capture are independent flags; only `UnsafeRegister` skips EC, making it the form for context-free callbacks (channel `TryWrite`, cell swap, event set) that read no `AsyncLocal` state. Inside the callback, `IsCancellationRequested` on the delivered token is a cheap volatile read — inspect it directly rather than closing over external state.
- Subscriptions retain strongly in both directions: the source holds every registered delegate and transitively its target, with no weak mode — an activation-scoped subscriber on a process-static source outlives deactivation until detached by stored identity. Registered kernel waits share the shape: the registration leaks the OS wait slot until explicitly unregistered, even after the owning object is collected.
- A handle used inside a callback window is ref-counted around the subscription, not the call: ref-acquire before attach, release only after detach completes, liveness check guarding the handler body — the window, not the invocation, is the borrow.
- Detach all subscriptions before disposing any handle; dispose-before-detach lets a racing callback touch a handle mid-teardown. The order is encoded as a sequential effect chain in the bracket finalizer — detach sequenced before dispose — never inferred from scope-collection disposal order, which is hash-trie traversal and therefore undefined.
- The subscription set dies at the same point the owner leaves the live state; reactivation constructs a fresh set and a fresh child resource scope — the scope is the set, so releasing the scope destroys the set atomically and no append-to-retained path exists; this is the structural cure for cross-cycle subscription accumulation. Host-owned buses deregister independently of the local record: every host lifecycle event invalidates assumed registration, so a liveness probe at callback entry precedes acting on any consumed item.

[SEAM_TRANSFORMER_COMPOSITION]:
- Attach, detach, marshal, retry, and fault-fence are not five wiring sites; they are one effect transformer `K<M, A> -> K<M, A>` composed by arrow juxtaposition: the bracket arrow owns acquire-project-release as a unit, the catch arrow owns the typed fault rail, the finally arrow owns idempotent teardown, the schedule-driven retry arrow owns reactivation cadence. No marshaled effect leaves the capsule except through the composed transformer.
- The trait constraint set is the capability manifest: the unlift trait supplies `MapIO`/`ToIO` and the bracket arrow, `Fallible<Error, M>` the predicate-keyed catch, `Final<M>` the finally arrow — declare the intersection once and the composition is available generically; the capsule never names the concrete carrier in its body.
- `toIO` projects the inner `IO` out as a value so the seam can `Fork`/`Await` host-event delivery on a dedicated thread and re-bind the result into the same `M` — the surrounding pipeline never learns a thread boundary was crossed.
- Reactivation cadence is one `Schedule` value, never enumerated attempt logic: an open-ended host-recovery probe unioned with a bounded envelope reads "probe on the host's cadence, give up at the ceiling"; a frame-rate ceiling intersected with a back-off floor honors the stricter of two cadences. The schedule is a pure value — the cadence is provable by enumerating its duration stream, no live host required — and one policy value is reused across every seam instance.

```csharp
var reactivation = Schedule.exponential(20 * ms) | Schedule.recurs(6) | Schedule.maxDelay(2 * sec) | Schedule.jitter(0.25);
K<M, A> Activate<M>(K<M, A> attach) where M : MonadUnliftIO<M> =>
    MonadUnliftIO.mapIO(io => io.Bracket().Retry(reactivation), attach);
```

[SUBSCRIPTION_LIFETIME_UNDER_BRACKETS]:
- Repeat combinators wrap each iteration in a child resource scope disposed per cycle — a subscription or signal acquired inside the repeated body is torn down every iteration; persistent subscriptions are acquired at the chain level containing the repeat, never inside it. Repeat and retry are CPS-recursive bind chains, not loops: long-running fold workers use schedule-threaded state forms to avoid unbounded evaluation depth.
- Failure-only bracketing (`BracketFail`/`WithEnvFail`) releases the child scope only on failure: each failed activation attempt cleans its partial subscription set before the next attempt, while the first success's subscriptions escape into the parent scope by design. Paired with an external ownership token holding the escaped scope, explicit deactivation swaps the token out and releases manually — success transfers ownership, failure self-cleans.
- `Uninterruptible` rebuilds the environment with an uncancellable token decoupled from the parent: inner work polling the environment token never observes cancellation, so the region must terminate in finite time on its own. Teardown that must not be interrupted and must detach its own subscriptions composes `Uninterruptible` around a bracket whose finalizer runs detach-then-dispose.
- `Finally` runs cleanup on both outcomes and discards its result; bracket is the form when the teardown's own value matters. A child-cancel environment inherits parent resources while forking a fresh token — a read loop stoppable without tearing down parent state, and detach inside the child stays safe because parent resources survive child disposal; ownership bits guarantee whoever allocated the cancellation source or resource scope disposes it, never the inheritor.

[HOST_THREAD_MARSHAL_EFFECT]:
- `IO.Post()` marshals onto the `SynchronizationContext` captured once at environment construction; when that context is null, `Post` silently degrades to inline execution with no marshaling — seams requiring thread affinity verify the context is non-null before composition.
- Context-bearing reads are effects evaluated at the composition root on the host thread and injected once: the ambient local-environment accessors are property getters that re-capture at access time, so touching them on a background thread seals the wrong context; `TaskScheduler.FromCurrentSynchronizationContext()` throws outright on null-context threads. Capture on the host thread, then travel in the environment — never re-read ambient context inside an effect.
- Host post primitives swallow exceptions thrown inside the posted delegate; there is no ambient propagation across a context post. `IO.Post()` implements the capture protocol — the posted delegate catches into a local, signals the rendezvous handle, and the awaiting fiber rethrows preserving the original stack. Marshaled work outside this protocol flows through the composed transformer or an explicit catch-then-publish-to-fault-channel wrapper, never bare.
- Each post allocates a wait event and costs multiple kernel transitions — non-viable per-frame or per-event. The high-frequency form writes into a channel and marshals one batch drain per tick: the repeat arrow over a drain body, driven by the tick schedule, owns both the cadence and the marshal point in one composed value.
- Synchronous send is never the bidirectional host-call shape — it deadlocks when the host thread is blocked waiting; fire-and-queue posting with a completion signal is. The structural deadlock guard: cross to the host only from a fiber that does not itself hold the host thread, achieved by sealing the host context at the root and forking blocking work off it before any marshal — the marshaling fiber is distinct from the host fiber by construction.
- The lifted post operates through the runtime transformer, so the context must already be in the environment when the effect runs; it cannot be injected after the fact. A fork carries the parent's captured context into the child environment: a fork spawned to protect the host thread still posts to the same context — it moves the blocking, not the context-switch count.

[EXECUTION_CONTEXT_FLOW]:
- `ExecutionContext.Run` saves, installs, invokes, and restores in a finally — the event-dispatch wrapper that must not leak the event's context onto the calling thread; captured contexts are immutable and reusable. `Restore` is a one-way install with no save — outside an async state machine resumption point it leaks the installed context onto the thread.
- `SuppressFlow` is thread-local and undone only by disposing the returned flow control: dropping the return value suppresses flow for all subsequent work queued from that thread, and nesting throws.
- `ThreadPool.UnsafeQueueUserWorkItem` with a generic struct state is the allocation-free, EC-free queue form — the work-item interface overload boxes a struct at the call site; `preferLocal: true` biases the current thread's local queue for same-core producer-consumer pairs. `Thread.UnsafeStart` spawns with no EC copy at all — correct only for callback workers that write channels, swap cells, or set completions; correlation-carrying workers must use the flowing start.
- The async-local change handler fires synchronously on every value change including fork/join transitions, with a flag discriminating context switch from explicit set — ambient-context-sensitive subscription management without explicit plumbing.
- Diagnostic ambient state rides async-locals: a native callback thread carries none, and an EC-skipping cancellation callback runs under the cancelling thread's ambient state — neither the registration thread's nor the consumer's. Set the ambient explicitly inside the callback, or pay the context-copy and restore cost of `Run` only after profiling the hot path.

[CANCELLATION_AS_RAIL_VALUE]:
- One normalizer in the interpreter's catch arm maps the host's control-flow exception families onto fixed-code rail values: both cancellation exception types collapse to `Errors.Cancelled`, a raw timeout to `Errors.TimedOut`, and aggregates fan out into a coalesced many-error. Cancellation is an ordinary catchable, inspectable, retryable error value in the failure channel — discriminated by `HasCode`, never by a typed catch clause in domain code.
- The code carries the kind of stop, not the source: parent-token cancel and local-token cancel are indistinguishable by code. A seam that reacts differently to caller shutdown versus internal deadline carries the source as separate evidence — a distinct error case constructed at the originating boundary — because the rail flattens both into one value.
- The timeout-cancelled collision: a deadline combinator runs its body inline under a timer-armed child cancellation source, the body's next cancellation-observing primitive throws, and the rail reads `Cancelled` — not `TimedOut`, which only a directly-lifted raw timeout produces. "Did this hop exceed its deadline" cannot be read off the combinator's output; arm a side-channel flag in the timer registration or raise an explicit timed-out failure from a body-internal check, and treat the combinator's `Cancelled` purely as unwind.
- Bare retry retries on everything — its default stop predicate stops on nothing — so a cancelled token becomes a tight retry storm: every attempt re-enters a body whose first cancellation-observing primitive re-throws against a token that never un-cancels. Cancellation is never transiently recoverable; it is the first arm of every seam retry predicate.

```csharp
static IO<A> reactivate<A>(IO<A> attach, Schedule policy) =>
    attach.RetryUntil(policy, e => e.HasCode(Errors.Cancelled.Code) || e.HasCode(Errors.TimedOut.Code));
```

- Schedule-driven retry fails with the last observed error on exhaustion — a probe failing N times surfaces only the final fault unless the seam coalesces the sequence itself — and the schedule-less retry carries no attempt envelope at all, making the schedule form mandatory for bounded reactivation.
- Source pooling edge: a cancellation source whose timer ever queued is permanently un-resettable — reset succeeds only before the timer fires, with callbacks unregistered and the token never cancelled; environment construction registers a parent-propagation callback, so a pooled source must reset before being re-passed.

[FAN_IN_FAULT_COALESCING]:
- The rail error is a monoid: append promotes singles to a many-error trie, and the identity is the empty trie — "a failure with no information," distinct from "no failure." Collecting N boundary-probe failures is a fold with `+`, never a list the caller post-processes; the all-completes fan-in accumulates every branch error without short-circuiting, so its error sequence is a partial-availability report, not a single fault.
- The aggregate answers discriminants by quantifier: exceptional if any member is, expected only if every member is, code and type membership by existence — the recovery decision reads "contains a cancellation" or "contains a non-recoverable exception" without iterating the trie.
- Aggregate exceptions decompose at the boundary through the same normalizer and fold into a many-error; a single-inner aggregate flattens to the bare normalized error rather than a one-element trie. Cancellation hidden inside a fan-in aggregate surfaces as a `Cancelled` member, catchable by code without unwrapping.
- A type-directed catch arm matches an aggregate containing the target type and chains its handler across only the matching members — sequentially short-circuiting or monoid-accumulating; the filter is total and order-preserving and yields the identity on a miss, so arms compose associatively into one recovery surface over heterogeneous probe failures — one arm per error type draining its slice, never a per-member switch.

[RACE_FORK_AND_FAN_OUT_TERMINATION]:
- The race returns the first completion, success or fault — a faulted first-completer rethrows immediately; it never collects failures while awaiting a later success. Conflating the race with an error-collecting fan-in is an availability bug: the race returns the first failure as eagerly as the first success.
- Loser termination depends on who launched: racing raw effects runs branches under a fresh child cancellation scope and cancels it after first completion — losers die; racing fork-constructing effects links each fork's source under that scope — the cancel propagates; racing pre-launched fork handles awaits under the caller's environment while the child scope owns nothing — losers run to completion in the background. A seam guaranteeing loser stop hands the race un-launched effects; first-result fan-in over already-running handles cancels each loser explicitly in a finally.
- A pre-entry guard skips the whole combinator under an already-cancelled ambient token — no branch ever spawns, and the fan-out surfaces as `Cancelled` with no probe started; a race cannot be relied on to run at least one probe under a cancelled scope.
- The all-completes join awaits every branch under the parent environment in positional order with no per-branch cancellation lever — per-branch bounding belongs on each forked effect as its own timeout, not on the join.
- A fork spawns a dedicated long-running OS thread outside the managed pool: the thread budget is a construction-time constraint on the seam owner, recorded beside any divergent native pool caps. The forked body runs unconditionally once launched — the only stops are binding its cancel effect or parent-token propagation — and disposing the parent environment while the fork is live invalidates the parent resource scope the child references.
- The fork handle's await is a re-entrant effect registering the awaiting environment's token against the fork's source: a second consumer awaiting the same handle from a different scope can kill the shared computation by cancelling its own scope — give each consumer its own fork, or await once into a published cell. Awaiting after the parent token fired throws the cancellation rather than surfacing the body's last value — partial results must be published to a separate cell before the cancellable region begins; awaiting after a fault surfaces the body's error through the rail.
- Completion-order iteration over a task set yields faulted tasks normally — per-task fault inspection is mandatory — and cancelling the iteration stops yielding without cancelling the underlying tasks; span overloads remove the array allocation for fixed-arity fan-out and linked-token construction. Atomic terminal-state propagation onto a completion source requires the source already completed and returns false on a lost completion race.

[WAKE_SUBSTRATE_SYNCHRONOUS_DELIVERY]:
- The single-value cell's change event fires synchronously on the committing thread inside the post-CAS success path before the swap returns: a blocking handler deadlocks the swap, and a handler that re-swaps the same cell nests its own change notification inside the outer commit's frame — the substrate has no reentrancy guard, so a handler triggering a further transition routes it through a channel where the second transition runs on a drained continuation, not on the committer's stack.
- The swap function may run multiple times in the CAS spin loop and must be side-effect free; the event fires exactly once with the committed value, every handler in one invocation receives that identical value, and a handler re-reading the cell may already observe a newer value from a concurrent swap. The metadata-carrying form delivers only the new value — prior-value consumers maintain a shadow read before the swap.
- Under a write storm the last writer wins and intermediates coalesce — the cell is structurally a latest-value register, not a log: exactly right for a per-tick consumer of a high-frequency host event (context-free swap in the callback, one current value per drain), exactly wrong for a consumer that must see every intermediate, which needs a channel whose buffer is the log the cell deliberately is not.
- The keyed-map cell emits change tracking only for transforms threaded through the map argument the swap function receives — building a fresh collection yields a diff with no tracked intermediates; always thread the provided map. The transactional reference's change event fires on every transaction iteration including conflict-aborted ones, delivering rolled-back proposed diffs — subscribers re-read the committed value when consistency matters.

[HANDOFF_CHANNEL_AND_DRAIN_CADENCE]:
- The host-callback handoff is non-blocking write plus paced drain: the callback's only work is `TryWrite`; a tick-driven worker drains all interval arrivals per tick. Consumer pacing lives in the tick period and producer back-pressure in channel capacity — independent axes; delaying producer writes to match an assumed consumer rate conflates them. State changes are read at tick start with `TryRead` — reading inside the tick body lands one frame behind.
- The hot path submits typed intent and consumes cached receipts without ever blocking on compute: `TryWrite` the intent, return a pre-allocated receipt handle, and resolve it from one named serial fold worker that owns the read loop and mutates nothing outside its own result cell — concurrent writes to that cell from outside the fold break the serial invariant. Drop-mode with the dropped-item callback is observer-owned rate limiting: capacity policy in the channel, shed-load telemetry in the callback, producer never blocked; bounded-parallel consumption over a channel-backed async source with a parallelism cap is the form when fork management is not required, and a comparator-ordered channel dequeues by priority regardless of insertion order.
- Zero-capacity construction yields a rendezvous, not a queue: every write parks until a read is pending, a bare read returns false unless a writer is already parked, the count is always zero, and peek inspects the head parked writer without consuming — a synchronization point for phase handoff between two independent workers.
- Completion fault delivery is asymmetric: an error completion surfaces to single reads as a closed-channel exception wrapping the original — and only after buffered items drain — while streaming reads surface the unwrapped original directly. A dedicated fault-channel writer is architecturally cleaner whenever fault consumers and projection consumers react differently; mixing errors into a sum-typed item forces every consumer to unpack on every read, while a separate fault channel keeps nominal consumers total.
- Late-subscription semantics are declared at the seam — replay-last, snapshot-plus-delta, or rejection; an undeclared seam hands consumers incremental diffs without a baseline. A channel carrying committed changes gives snapshot-plus-delta free: delivery is commit-ordered and non-missable for active consumers, and buffered items remain visible to a consumer arriving before completion.
- Streaming reads do not end on cancellation alone — the producer must complete the channel; rate-limited consumers interpose pacing via wait-then-drain loops, not the streaming form. The single-reader optimization routes to a lock-free implementation that corrupts under concurrent readers — the flag is a contract, not a hint.
- The tick timer is owned and disposed by the worker it drives, never a parent that may outlive the worker's subscription set: disposal completes the pending wait with false and never interrupts an in-flight iteration, while cancelling a single wait throws without stopping the timer — a drain loop must handle both exits. The period is mutable for next-cycle rate changes without reconstruction, and a time-provider injection point makes the cadence deterministic under test.

[KERNEL_WAIT_BRIDGE_AND_SIGNAL_CELLS]:
- The kernel-event-to-await bridge registers the handle on the pool's single dedicated wait thread and completes a promise from the wait-or-timeout callback: one wait thread multiplexes thousands of parked waiters at zero pool-thread cost. The bridge promise defaults to inline continuations — resumption runs on the shared wait thread or the cancelling thread, stalling every other waiter for the continuation prefix — so construct it queuing or make the first resumed segment trivial and yield. Deadline expiry inverts into the awaited boolean — settled-versus-elapsed reads off one await, no separate timeout race — and the negative-one timeout means infinite.
- A waiter abandoned before its teardown finally — the enclosing scope faulting pre-await — leaks both the registered wait slot and the token registration, and the leaked slot pins the kernel handle against finalization. The EC-flowing registration default is pure per-wait allocation for context-free waiters — the unsafe variant is the high-frequency seam; the effect-lifted wait resumes carrying the EC from the await site inside the lift, not from the completing thread.
- Signal cells register their kernel handle into the effect runtime's resource scope at construction and auto-release on scope teardown — a cell consumed inside a repeat body dies every iteration; cross-iteration cells are acquired outside the repeat. Auto-reset is the rendezvous baton — N triggers wake exactly N waiters, no lost or doubled wake; manual-reset is the one-shot latch every late waiter observes settled; converting one to the other silently changes whether a second waiter is admitted.
- The reset constructors ignore their initial-signaled argument and always start unsignaled — a pre-latched start is expressed by triggering as the first effect after construction, never by the constructor, or the first waiter parks forever. The countdown's count-taking trigger ignores its count and decrements by one — batch completion loops single decrements or computes the join externally, or the join never settles; decrementing past zero throws rather than saturating and incrementing after settle throws, so dynamic fan-in grows the join before the count can reach zero. The trigger reports whether zero was reached, identifying the final completer for last-one-out teardown; a host callback already off the effect spine uses the impure trigger deliberately and owns the ordering itself.
- The countdown wait blocks its thread under token guard rather than riding the async bridge — many-waiter joins compose auto-reset-plus-bridge instead. The zero-allocation reusable awaiter is the high-frequency host-event bridge when per-event promise allocation is unacceptable: a struct core whose version token makes a stale awaiter's get throw, under one-producer, one-consumer, reset-before-next-wait discipline — reset while an awaiter is active corrupts the version; the host event sets the result inside its callback and the consumer awaits a fresh cycle after each reset.

[CONTINUATION_INLINING_DISCIPLINE]:
- Every promise-backed boundary signal completed from a foreign thread — host event, native callback, timer fire — inlines its awaiting continuation onto that thread unless constructed to queue: the choice is a constructor argument, never per-await, and a waiter cannot opt out of an inlining completer after the fact. A seam completing waiters from a host thread or shared callback thread runs arbitrary downstream logic there unless every signal source is constructed queuing; the queued form schedules through the EC-free pool queue with same-core preference.

```csharp
var inlineGate = new TaskCompletionSource<TState>();   // firing thread runs the awaiter's continuation prefix
var safeGate   = new TaskCompletionSource<TState>(TaskCreationOptions.RunContinuationsAsynchronously);
hostToken.Register(static s => ((TaskCompletionSource<TState>)s!).TrySetResult(settled), safeGate);
```

- The channel seam fixes continuation asynchrony per pending operation at construction — a channel built asynchronous never inlines a reader's resumption onto the writer's thread, structurally safer than a hand-rolled signal whose promise defaults inline. Enabling synchronous continuations buys one thread switch and two hazards: unbounded stack growth when the continuation writes the same or a downstream channel, and reentrancy when the writer holds a lock the reader's continuation needs.
- Force-yielding the await of a pre-completed task prevents the handler from running synchronously on the event source's stack frame — resumption lands on the captured context, not the dispatcher's stack — closing the reentrancy hole where an already-completed await schedules more events inline. Throw suppression is valid only on the non-generic task shape — the generic form throws at runtime, not compile time; the pattern is cast down, await suppressed, then inspect the original handle for fault or cancel — a non-result-carrying boundary guard correct only when a separate fault channel already carries the outcome, and the combined yield-plus-suppress flags still require the cast.
