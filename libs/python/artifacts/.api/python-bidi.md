# [PY_ARTIFACTS_API_PYTHON_BIDI]

`python-bidi` (dist `python-bidi`, import `bidi`) supplies the UAX#9 Unicode Bidirectional Algorithm reorder surface for the artifacts text-shaping rail ahead of HarfBuzz shaping: a top-level `get_display(str_or_bytes, encoding='utf-8', base_dir=None, debug=False)` that resolves a logical-order string into its visual display order over the Rust `unicode-bidi` crate, and a `get_base_level(text) -> int` paragraph-direction probe (`0` LTR / `1` RTL). Both public names re-export from `bidi.wrapper`, which calls the native `bidi.bidi` PyO3 extension (`get_display_inner(text, base_dir, debug)` / `get_base_level_inner(text)`). The package also ships a pure-Python reference engine `bidi.algorithm` that exposes the FULL UAX#9 stage pipeline as separate callables (`explicit_embed_and_overrides` -> `resolve_weak_types` -> `resolve_neutral_types` -> `resolve_implicit_levels` -> `reorder_resolved_levels` -> `apply_mirroring`) over a mutable storage dict, plus the `bidi.mirror.MIRRORED` 362-entry BidiMirroring pairing table. The package owner composes `get_display` and `get_base_level` into the `typography/shape#SHAPE` `Shaping` owner's `ShapeOp.BIDI` arm — the mixed-direction reorder pass that runs over a logical-order run BEFORE `uharfbuzz` itemises and shapes it, so an Arabic/Hebrew + Latin run reaches `Buffer.shape` already in visual order with the per-paragraph base level resolved. It never re-implements the bidi resolution rules (explicit embeddings/overrides/isolates, weak/neutral/implicit type resolution, level-run reordering, mirroring) the `unicode-bidi` Rust core owns, and on the runtime floor the arm dispatches onto the gated `runtime` `anyio.to_process.run_sync` subprocess seam because the native binding has no in-process gated package.

## [02]-[PUBLIC_TYPES]

[PUBLIC_TYPE_SCOPE]: reorder value and direction probe
- rail: text-shaping

`python-bidi` is a function surface, not a type surface: `get_display` returns the same kind it was given (`str` in -> `str` out, `bytes` in -> `bytes` out re-encoded with `encoding`), so there is no wrapper value object to allocate. ## [04]-[IMPLEMENTATION_LAW]

[TEXT_SHAPING_BIDI]:
- ordering axis: the `BIDI` reorder runs BEFORE `uharfbuzz` shaping, never after — `get_display` produces the visual-order run that `uharfbuzz` `Buffer.add_str` + `shape` then itemises and positions; reordering after shaping does scramble the glyph cluster mapping. The arm is a pre-pass on a mixed-direction logical run, so on a pure-LTR run with no strong-RTL character the arm is a no-op the owner elides rather than a mandatory hop (`get_display('hello') == 'hello'`; `get_display('אבג abc') == 'abc גבא'`). The `PositionedGlyphRun` the `Shaping` owner emits therefore carries glyphs already in visual order with the per-paragraph base level threaded from `get_base_level`.
- direction axis: `get_base_level(text)` resolves the first paragraph's base direction (`0` LTR / `1` RTL) from the first strong character (`get_base_level('abc') == 0`, `get_base_level('אבג abc') == 1`); `get_display(..., base_dir='L'|'R')` overrides that computation when the document column direction is already fixed by the layout. Compute the base level once at the `BIDI` arm and thread it into the shaping `direction`/`script` decision rather than re-deriving it per fragment — `base_dir` is a 2-member `'L'`/`'R'` vocabulary the boundary lifts into the closed paragraph-direction value, never a free string passed verbatim (the native `get_display_inner` takes `base_dir` as the second positional, so the owner passes the lifted value, not a character).
- bytes axis: `get_display` accepts `str` or `bytes`; on `bytes` `bidi.wrapper` decodes with `encoding` (default `'utf-8'`), reorders the decoded `str` over the native core, and RE-ENCODES the result with the same `encoding`, so a `bytes` in yields `bytes` out and a `str` in yields `str` out (`get_display('אבג abc'.encode()) == b'abc \xd7\x92\xd7\x91\xd7\x90'`). The arm passes a decoded `str` and keeps the byte path off the shaping seam, since `uharfbuzz` `Buffer.add_str` shapes a `str`. Never mix the kinds at the boundary (a `str` reorder must not be re-encoded), and never pass the `bytes` form when the upstream is already a decoded run.
- debug axis: `debug=True` makes the native `get_display` return the Rust `BidiInfo` resolution dump (a formatted `'BidiInfo { text: ..., original_classes: [R, R, ...], ... }'` string carrying the original bidi classes and resolved levels) instead of the reordered string — a diagnostic dump, never the production reorder; the `BIDI` arm runs `debug=False`. For a structured per-character level array (not the native dump string) the owner routes through the pure-Python `bidi.algorithm.get_embedding_levels`/`calc_level_runs` over a `get_empty_storage()` dict and `debug_storage` for the trace. The pure-Python `bidi.algorithm.get_display` additionally exposes `upper_is_rtl=True` (treat uppercase Latin as strong RTL) — a reference-test affordance the native binding IGNORES (the `pybidi -u` flag is no-op under `-r`), never a production knob.
- mirroring axis: the L4 character-mirror swap (brackets/quotes flipped in RTL runs) is INSIDE the native reorder — `get_display` already returns mirrored output. `bidi.mirror.MIRRORED` (362 entries) is the same pairing table exposed as data: the owner reads it only to resolve a single mirrored pair without re-running the algorithm (e.g. an annotation glyph), never to re-implement L4 over the native result.
- evidence axis: each reorder pass captures the resolved paragraph base level (`0`/`1` from `get_base_level`), whether a `base_dir` override was applied, the presence of a strong-RTL character (whether the arm was a no-op or an active reorder, derivable from input-vs-output equality or a base-level of `1`), and the run length as a text-shaping receipt fact folded into the `Shaping` owner's `ArtifactReceipt.Document` content-key derivation (the `BIDI` arm contributes the reordered-text byte count; the base level / override / strong-RTL flag stay interior content-key evidence, never new receipt fields the shared `Document` case cannot carry); map a Rust panic at the subprocess seam into the `runtime` `RuntimeRail` fault via the `async_boundary` trap rather than surfacing it bare across the interpreter boundary.
- boundary: python-bidi owns the UAX#9 logical->visual WHOLE-PARAGRAPH reorder and the paragraph base-level probe — the mixed-direction pre-pass `uharfbuzz` does NOT perform (HarfBuzz shapes a run in a single given direction; it does not run the bidi resolution that splits a mixed run into directional level runs). It is the locale-free DEFAULT bidi path and the active path on this cp315 interpreter (where `PyICU` is cp-gated absent). When a design needs the LINE-RELATIVE reorder over already-resolved paragraph levels (`Bidi.setLine` after line-break), the per-character level array AND the explicit directional-VISUAL-RUN decomposition (`getVisualRun(i) -> (logicalStart, length, level)` to itemise a mixed run into single-direction spans before shaping), or combined reorder+Arabic-join+digit-shape (`BidiTransform.transform`), the owner routes to the richer `pyicu` `Bidi` engine instead — `python-bidi`'s whole-string `get_display` cannot give a line-scoped or visual-run-itemised result. `uharfbuzz` owns the OpenType itemisation/shaping AFTER reorder, `fonttools` the binary font model and the `SVGPathPen` outline pen, `blackrenderer` the COLRv1 color-glyph raster of the shaped run, `uniseg`/`pyicu` the UAX#14/#29 line/grapheme segmentation (a separate concern, not bidi); the `BIDI` reorder feeds the shaped run that `document/emit#DOCUMENT` places and `composition/compose#COMPOSE` annotates. The reorder is the gated Rust hop; live UI stays outside this package.

[STACK_INTEGRATION]:
- `python-bidi` -> `typography/shape#SHAPE`: the `Shaping` owner reaches `get_display` through `lazy from bidi import get_display`, dispatching the `ShapeOp.BIDI` arm onto the worker via `to_process.run_sync(_gated_bidi, self.params.text, self.params.base_direction, limiter=_SHAPE_LIMITER)`, where `_gated_bidi(text, base_direction)` is `get_display(text, base_dir=base_direction).encode("utf-8")` — the reorder runs in a worker subprocess (the binding has no in-process gated package) and crosses the visual-order text back as bytes. The reordered text feeds the `ShapeOp.SHAPE` arm's `Buffer.add_str` -> `shape` so the shaped `PositionedGlyphRun` is already visual-order.
- universal-tier rails (`libs/python/.api/` ON TOP OF the folder shaping partners): the gated reorder rides the `anyio` (`libs/python/.api/anyio.md`) `to_process.run_sync(func, *args, cancellable=False, limiter=...)` process-dispatch seam under one shared `CapacityLimiter` bounding the whole shaping subsystem (distinct from `uharfbuzz`/`blackrenderer`'s GIL-releasing `to_thread` lane — bidi needs the SUBPROCESS lane because its PyO3 binding is not the in-process gated package), with the worker raise/death surfacing as `anyio.BrokenWorkerProcess`; every `BIDI` arm returns the same `expression` (`libs/python/.api/expression.md`) `Result`/`RuntimeRail[ContentKey]` rail the rest of the `ShapeOp` family returns, so a Rust panic folds into the `runtime` `RuntimeRail` fault via `async_boundary` rather than crossing the interpreter boundary bare; the `base_dir`/`text` boundary inputs are validated by `beartype` (`libs/python/.api/beartype.md`) and the reorder params carried as a frozen `msgspec.Struct` (`libs/python/.api/msgspec.md`) field set whose reordered-text fact encodes through one module-level `msgspec.msgpack.Encoder` into the content key; a strong-RTL-present / base-level / override span is recorded through `structlog` + OpenTelemetry (`libs/python/.api/structlog.md`, `opentelemetry-api.md`) on the worker hop. The pure-Python `bidi.algorithm` level array, when surfaced as evidence, lands as a `numpy` (`libs/python/.api/numpy.md`) `int8` array of embedding levels keyed by character index.
- `python-bidi` vs `pyicu` (no-overlap owner selection, `libs/python/artifacts/.api/pyicu.md`): both run UAX#9, but they are NOT a redundant pair — `python-bidi` owns the whole-paragraph `get_display` reorder + `get_base_level` probe (the locale-free default, cp315-active path), `pyicu` `Bidi` owns the locale-aware engine surface python-bidi does not expose (`setPara`/`getLevels`/`getVisualRun` visual-run itemisation, `setLine` line-relative reorder, `writeReordered` with `UBiDiReorderingOption`, `BidiTransform.transform` reorder+reshape). The owner picks `python-bidi` for the default whole-run reorder and escalates to `pyicu` `Bidi` only when visual-run itemisation or line-relative reordering is needed; `pyicu` is cp-gated absent on this interpreter, so `python-bidi` is the operative bidi owner today.
- `python-bidi` vs `uniseg` (`libs/python/artifacts/.api/uniseg.md`): orthogonal concerns at the same shaping altitude — `uniseg` owns UAX#14/#29 line/grapheme/word/sentence SEGMENTATION (the break-opportunity stream), `python-bidi` owns UAX#9 bidi REORDER; they sit side by side in the text pipeline, never overlapping. `uniseg`'s `unicodedatawrapper.bidirectional` Bidi_Class re-export is a per-codepoint property read (interpreter UCD), not a reorder, and does not duplicate the `get_display` algorithm. Unlike a `uniseg` segmentation pass (pure, offline, inline), the bidi reorder is the one shaping-pipeline step that REQUIRES the `runtime` worker seam because of the native binding.
