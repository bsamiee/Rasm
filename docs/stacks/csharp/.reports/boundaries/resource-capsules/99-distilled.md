# Resource Capsule Law

[CAPSULE_OWNER_AND_HANDLE_LAW]:
- One capsule owner per foreign resource — it acquires, projects, and disposes; borrowed and owned lifetimes are cases of one closed surface (the borrowed case's `Dispose` is a no-op because the caller owns the lifetime, the owned case disposes), never sibling types; projections leave as fully materialized values and the raw handle never crosses into caller-controlled scope.
- A raw `IntPtr` native handle lives inside a `SafeHandle`-derived inner representation, never a bare field: `SafeHandle` reference-counts via `DangerousAddRef`/`DangerousRelease`, runs `ReleaseHandle` from the finalizer, and the marshaller's add-ref/release pair keeps a handle alive during in-flight marshaled calls so the OS cannot recycle the value under a stale raw copy on another thread. `CriticalHandle` is admissible only when the handle never passes through a `[LibraryImport]` marshaling layer and never races teardown with a concurrent call — no ref-count means no in-flight protection; any concurrent topology forces `SafeHandle`.
- `SafeHandleZeroOrMinusOneIsInvalid` and `SafeHandleMinusOneIsInvalid` (in `Microsoft.Win32.SafeHandles`, not `System.Runtime.InteropServices`) encode the two OS sentinel conventions — POSIX descriptors are minus-one-only; cross-platform capsules take the zero-or-minus-one base as the conservative form. A capsule extending a helper base must not override `IsInvalid`: `InternalRelease` gates `ReleaseHandle` on `_ownsHandle && !IsInvalid`, so an override that reports a live handle invalid silently skips `ReleaseHandle` — a diagnostic-free native leak. The sanctioned suppression path is `SetHandleAsInvalid()`: it atomically sets the closed bit and suppresses finalization so an externally-freed resource is never double-freed.
- `DangerousGetHandle` returns the raw pointer even after invalidation; any non-`[LibraryImport]` egress wraps it in `DangerousAddRef(ref success)`/`DangerousRelease` where the `success` guard is load-bearing — releasing when `success` is `false` decrements a count never incremented and corrupts the ref-count into premature `ReleaseHandle`. The capsule entry-point `IsInvalid`/`IsClosed` pre-check is a fast path only; `DangerousAddRef`'s own closed check is the true gate.
- `[LibraryImport]` with typed `SafeHandle` parameters replaces the manual pattern through `SafeHandleMarshaller<T>`'s three modes: in/by-value (`FromManaged` add-refs, `Free` releases), out-factory (`ToManaged` constructs the subclass and performs the `Marshal.InitHandle` equivalent — no manual call), and ref-bidirectional (`OnInvoked` must run between invocation and `ToManagedFinally`; interruption leaves a partial wrapper whose `Free` disposes partial state). `Marshal.InitHandle` is the public bridge to protected `SetHandle` for start-invalid-publish-after-success construction outside a marshaled boundary.

[PINNED_AND_NATIVE_MEMORY_LIFETIME]:
- The pinned-buffer capsule is a sealed class around a single `GCHandle` field: `Dispose` frees iff `IsAllocated`, and the address projects as `nint` via `AddrOfPinnedObject` — never the `GCHandle` itself. `GCHandle` is a struct, so it cannot be a `SafeHandle` inner representation; a capsule needing both a pinned buffer and a native handle holds two independently lifecycle-managed fields. Normal-type handles never freed are managed leaks invisible to native-resource diagnostics.
- Handle-scoped pinning and permanent pinned-heap residency are two distinct lifetime shapes, not equivalents: `GCHandle.Alloc(Pinned)` holds the address only until `Free`; `GC.AllocateUninitializedArray(pinned: true)` pins permanently in the pinned object heap and pairs with `MemoryMarshal.CreateFromPinnedArray` — which validates nothing, so a non-POH array yields a `Memory<T>` the GC may move between span-take and native use. POH slots are never compacted; the owning capsule's `Dispose` nulls the backing field to release the slot, and projection is gated on disposal state.
- Unmanaged-heap capsules pair the free with the allocation variant — plain free on aligned-alloc memory is CRT undefined behavior — and security-sensitive buffers zero in `ReleaseHandle` via `NativeMemory.Clear` before the free: zero-on-release, never zero-on-allocate, because an allocation-site invariant has no enforcement point.
- `Memory<T>.Pin()` returns `MemoryHandle` — the two-phase shape: `Memory<T>` is the logical extent, the handle the physical address lock; `Pointer` is valid only inside the handle's scope, retention past `Dispose` is a dangling dereference, and an undisposed handle pins indefinitely.
- `MemoryManager<T>` is the custom-backing extension point: `Pin(elementIndex)`/`Unpin` form a thread-safe pin-count pair; disposing with outstanding pins is undefined behavior at the native layer; `Dispose` is explicit-interface only (`using` binds, a direct call needs an `IDisposable` cast); `MemoryMarshal.TryGetMemoryManager` recovers the manager from a `Memory<T>` slice to rejoin the pin-count lifecycle.

[REGISTRY_OWNERSHIP_AND_DERIVED_SCOPES]:
- The effect environment is the boundary's lifetime substrate: a resource registry, a cancellation token, its owning source, an optional captured synchronization context — and a private ownership bitmask (source = bit 1, registry = bit 2) recording allocated-here versus inherited. Disposal is ownership-gated: an environment that inherited a shared registry never disposes it — borrowed/owned expressed as bit flags rather than a case type.
- Environment disposal splits into two independently invokable idempotent halves under separate interlocked CAS flags — release-tracked-resources and dispose-source-plus-registration; the failure-only bracket runs the first from its catch and the second from its finally, so the two cleanup concerns never couple, and a scope may dispose its cancellation plumbing while letting acquisitions survive.
- Four derived environments each isolate a named axis: fully-local (fresh registry, fresh source, recaptured context), resources-local (fresh registry, shared source), cancel-local (shared registry, fresh source), context-local (recaptured context only). Selecting the wrong derivation silently shares the axis the caller meant to isolate, and the bug surfaces only as a cross-scope release.
- The cancel-local operator (`localIO`) isolates cancellation, not resources: acquisitions inside it track in and release with the outer scope. `Local()` is not the isolated child — it shares the registry and token and refreshes only the cancellation-source slot; the acquire-bracket is the resources-local derivation and the canonical child registry that disposes on exit. A per-iteration cancel-and-retry that needs per-iteration disposal wraps each attempt in the bracket, not the cancel-local operator, or acquisitions accumulate until the outermost exit.

[REGISTRY_TRACKING_AND_RELEASE_SEMANTICS]:
- Registration is a try-add keyed by the resource value under object identity: re-registering a reference-equal value silently drops, and a struct or record with structural equality collides with a prior equal registration — the second acquisition is never tracked and never released. Independently tracked resources must carry reference identity or a distinct wrapper per acquisition.
- Registries chain to parents and release walks up on a local miss — the precise mechanism by which a fork or nested scope frees a parent-acquired resource a sibling still holds. A merge folds one registry into another atomically (the applicative/parallel composition path); merged entries adopt the target's lifetime.
- Acquisition produces one of three tracked cases — synchronous-disposable, asynchronous-disposable, custom-finalizer arrow — behind one uniform release. The synchronous case probes at release time and prefers the asynchronous drain when the value implements both interfaces (the dedicated async acquire path is required only for async-only values); the custom case tracks interface-free handles — native frees, pool returns — identically to interface-bearing ones; the optional-acquire form registers nothing on typed absence. Per-key release runs the finalizer immediately and removes the registration; releasing an unregistered value is a no-op.
- Bulk release runs in hash-bucket iteration order, never LIFO acquisition order: child-before-container ordering must be encoded as nested brackets or one composite finalizer, never assumed from the registry. Cleanup runs under a fresh non-cancellable environment — immune to the outer token — which is itself never disposed, abandoning one source to finalization per scope teardown; hot open/close paths batch lifetime into one outer scope to avoid the compounding cost.
- The release loop has no per-item failure isolation: one throwing release aborts the loop and leaks every not-yet-iterated resource — a tracked release that can throw catches internally and converts to a tracked failure. The release-all sweep runs inside the map's CAS swap and re-executes on contention, re-running already-executed release effects: sweep-routed teardown must be idempotent; non-idempotent teardown uses per-key release, which removes the key before the effect and cannot re-enter.

[BRACKET_SEAMS_AND_SCOPE_OPERATORS]:
- The two bracket flavors differ only in the disposal seam: release-on-every-exit runs the body under a using-bound local environment — success, failure, and cancellation all dispose; release-on-failure-only disposes resources from catch and only source-plus-registration on success, so acquisitions survive as orphans — the local registry is never merged back, the registration is unowned, and repeated survive-on-success acquisitions accumulate live resources no scope will release until re-registered in an outer bracket. The seam pair is protected on the carrier; acquire-on-success is expressed through the failure-only bracket operator, not by calling the seam directly.

```csharp
async ValueTask<IO<A>> InvokeEager(EnvIO env)
{
    using EnvIO local = MapEnv(env);                  // every-exit: using disposes registry + source
    return Next(await Operation.RunAsync(local));     // value projected before using-dispose
}
async ValueTask<IO<A>> InvokeOnFailOnly(EnvIO env)
{
    EnvIO local = MapEnv(env);
    try     { return Next(await Operation.RunAsync(local)); }
    catch   { local.DisposeResources(); throw; }      // resources freed only on fault
    finally { local.DisposeNonResources(); }          // source/registration freed always
}
```

- The continuation consuming a bracketed value runs after the using-disposal fires: the value is fully materialized before release, so a handle smuggled out as the value is a handle to an already-released resource — brackets return values, never live handles, by construction of the seam.
- The multi-argument bracket (acquire, use, finalize, optional catch) never touches the registry: the finalizer is a finally combinator — the caller's explicit arrow, run exactly once on any outcome. Registry-tracked acquisitions inside its use body engage two independent cleanup mechanisms whose relative order is bind order, not a unified release. The catch-bearing form is for partially-allocating acquisition: catch recovers from the partial fault while the finalizer runs unconditionally over whatever registered before it.
- A synchronization-context switch threads the same environment: the registry neither resets nor re-scopes across it, so post-switch acquisitions needing a shorter lifetime take an inner bracket.
- Timeout fires the environment token cooperatively: a non-cancellable blocking call keeps executing past the deadline while the timeout returns — a native call inside a timed bracket either accepts the token or runs inside a fork whose own scope cleans up when abandoned. Bracket composed with timeout is safe: the timeout path releases at scope exit like any other.
- The uninterruptible mask suppresses cancellation without altering release — scope-exit disposal still runs. Mask only the atomic acquire or release step; masking a whole bracket defeats the drain window. The per-iteration pattern is cancel-isolation around each attempt with the mask on acquire alone.
- Retry releases a failing iteration's acquisitions before the next attempt; repeat releases each iteration's acquisitions even on success — nothing acquired inside a repeated body survives the iteration boundary. Resources persisting across iterations are acquired in a surrounding bracket and enter the body as values.

[FORK_AND_RACE_PROJECTION]:
- Fork allocates a child registry parented to the forker's and disposes it in the fork's own finally on every exit; the fork value's await re-raises and its cancel signals — neither disposes, so awaiting a fork to then dispose its resources observes already-released state. Because the chain is parented, a release inside the fork for a parent-acquired resource walks up and frees it under a sibling — share projections into forks, never releasable handles; fork-bounded resources are acquired inside the fork.
- The competitive race cancels losers, and every fork's registry — including the winner's — disposes at that fork's exit: a session opened inside a racing fork is gone when the race returns. The race yields a reconstructable identity token and the live resource is acquired in the outer scope after settlement; the await-all form never cancels siblings and releases each fork independently at its own completion.
- Over an arbitrary carrier the isolation doubles: fork lowers through the effect base and captures carrier layers by closure — reader-carried context survives into the fork as a shared value, but a state-threaded ledger is captured at its pre-fork value and the fork's mutations never rejoin the parent.

[CARRIER_POLYMORPHIC_LIFECYCLE]:
- The entire lifecycle algebra — bracket, local scope, repeat, retry, timeout, fork, mask, post, fold — collapses onto one abstract unlift primitive: `ToIO` lowers `K<M,A>` to `K<M, IO<A>>`, and every operator is the derived reduction `MapIO(ma, f) = ToIO(ma).Bind(io => M.LiftIO(f(io)))` — surface the inner action, rewrite it with the concrete operator, re-lift. A carrier supplying `ToIO` inherits the whole family; it overrides a member only to change that member's semantics, never to re-implement the family — and a capsule re-implementing bracketing locally forks the algebra the carrier already owns, the rejected form.
- Because release semantics live wholly in the surfaced action, carrier-polymorphic operators carry exactly the concrete release guarantees, and the registry is never carrier-aware: the transformer stack above the base cannot see or perturb registration, parent-chaining, or bulk disposal.

```csharp
static K<M, C> Bracketed<M, A, B, C>(K<M, A> acquire, Func<A, IO<C>> use, Func<A, IO<B>> finalise)
    where M : MonadUnliftIO<M> =>
    M.MapIO(acquire, io => io.Bracket(use, finalise));   // lower → rewrite IO → re-lift
```

- The public operators delegate to partial-trait members that assert "this stack admits an `IO` base" dynamically: a transformer stack not bottoming out at `IO` fails at runtime inside the first lifecycle operator, not at the call that built the stack — innermost-`IO` is a precondition the `MonadUnliftIO<M>` constraint does not enforce.
- The lowering round-trip is transparent for reader-shaped carriers (no outbound state) and lossy for state-threaded ones: a bracket whose body mutates carrier state acquires and releases correctly while the post-bracket state reverts to its pre-bracket value — the resource lifecycle succeeds, its ledger is lost. Either the carrier overrides the partial member to thread its state through the surfaced action, or the lease ledger moves into a reader-provisioned cell so the lowering stays transparent.
- A capsule's lifecycle surface is fixed at definition time by trait membership: the unlift trait yields the full bracket family, lift-only yields lifting without bracketing, the reader-accessor trait yields provisioning. One bracket, timeout, and isolation spelling holds across every effect sharing the carrier — no per-effect or per-capsule lifecycle re-statement exists.

[PROVISION_LIFETIME_MASK_AXES]:
- Three orthogonal axes govern a carrier-polymorphic capsule, each with its own operator: what it acquires FROM is the reader-carried runtime (typed accessor ask; scoped swap restoring the outer runtime on exit); how long acquisitions LIVE is the registry scope (bracket / local isolation); whether the body can be INTERRUPTED is the cancellation mask. None subsumes another and each lowers independently — the runtime-carrying effect is structurally a reader over the effect base under a thin record, so the reader layer carries the runtime, the base carries the registry, and unlifting surfaces the action with the runtime already closed in.
- Swapping the runtime does not isolate the registry: the scoped swap changes the acquisition source while acquisitions still track in the outer registry — conflating the axes leaves them live past the intended boundary. The minimal correct stack for acquire-once-release-uninterruptibly against a scoped pool: swap to the lane, bracket-open a private registry, mask only the finalizer — drop any one and the capsule acquires from the wrong source, leaks to the outer scope, or defeats the drain window.

```csharp
static K<M, Conn> Acquire<M, RT>()
    where RT : Has<M, Pool> where M : Monad<M> =>
    from pool in Has<M, RT, Pool>.ask                 // sub-runtime by accessor, not ctor param
    from conn in pool.Open()
    select conn;

static K<M, A> InLane<M, RT, A>(Func<RT, RT> toLane, K<M, A> region)
    where RT : Local<M, RT> =>
    RT.With(toLane, region);                          // swaps acquisition source, not the registry
```

[PROJECTION_SEAM_REF_SAFETY]:
- "Only a value copy leaves" is compiler-computed, not convention: every expression carries a safe-context from a four-level lattice (declaration-block, function-member, return-only, caller-context); a projection compiles only when the escaping value sits at return-only or wider, and a live view at function-member is rejected at the return. The author's job is shaping the seam so escape analysis decides correctly, never auditing escapes by hand.
- The live view surfaces only as a `ref struct`, never `Memory<T>` or an array slice: a `ref struct` cannot be an array element, a class field, boxed, lambda-captured, or carried across `await`/`yield` — every heap escape route is type-denied — while `Memory<T>` is an ordinary struct that sits in a class field and outlives the pin with no objection. The lattice level of the surfaced type is the lifetime contract.
- A bounded callback alone does not close the seam: `Func<Span<byte>, T>` cannot carry a `scoped` span and constrains nothing about `T`, so the view can smuggle through a closure or a ref-struct instantiation. The closed seam is a custom delegate with a `scoped` view parameter — function-member ref-safe-context, outward assignment a compile error inside the body — and a non-ref-struct return: the computation leaves, the view does not.

```csharp
delegate TResult ViewOf<TResult>(scoped ReadOnlySpan<byte> view);

readonly ref struct NativeView
{
    readonly ref byte _origin;          // ref field: cannot outlive its referent, compiler-enforced
    readonly int      _length;

    public TResult Project<TResult>(ViewOf<TResult> read) =>
        read(MemoryMarshal.CreateReadOnlySpan(ref _origin, _length));
}
```

- A `ref` field collapses the two-phase extent/lock split into one type: the referent is the physical lock, the length beside it the logical extent, and the containing `ref struct`'s safe-context is no wider than its narrowest input — built over a pin it inherits the pin's scope, over a stackalloc the frame's, propagated through every method without manual tracking; ref-reassignment admits a bind only when the source's contexts are at least as wide as the target's.
- Factory defaults are the implicit walls: `ref`/`in` parameters default to return-only, `out` is implicitly `scoped` (function-member), and a struct's `this` is scoped — so a factory cannot leak a view through `out` and an instance member cannot return a `ref` into `this` without declaring it. `[UnscopedRef]` widens exactly one lattice level and is an audited assertion that the referent outlives every caller — valid for a process-lifetime pinned capsule, never for a stack-frame one.
- Generic pipelines carry views only through type parameters declaring the ref-struct anti-constraint, which imports every escape restriction into the generic body — one polymorphic projection entrypoint spans value results and live-view results without a per-shape duplicate; a boxed-path rejection signals the combinator must be widened, not copied.
- Composition is the narrowest-argument rule: a returned `ref struct`'s safe-context is the minimum over caller-context and every argument's contexts, and a `scoped` argument contributes nothing. Per stage: take the view `scoped` when emitting a value copy — severing the output from the view lets it outlive the view's disposal; take it un-`scoped` when returning a sub-view that genuinely borrows and must inherit the parent's narrower lifetime. The pipeline's entire escape behavior is those modifiers, verified at every call.

[TEARDOWN_PROTOCOL_AND_DRAIN_ORDER]:
- Exactly-once teardown: `Dispose` gates on `Interlocked.CompareExchange(ref _disposed, 1, 0)`; projections guard with `ObjectDisposedException.ThrowIf(_disposed == 1, this)` — the non-atomic read is correct only because disposal is terminal, so a resettable type cannot use the pattern.
- Dual `IDisposable`/`IAsyncDisposable` with non-equivalent bodies is an uncomposable contract: `await using` never reaches `Dispose`. When an async drain is required for correctness the type is async-disposable only — a sync `Dispose` that skips the drain is a partial-cleanup bug; the only valid dual form is a documented sync-over-async bridge whose deadlock surface includes any single-thread-affinity scheduler, not just UI contexts. A handle-backed capsule with an async flush disposes the handle at the end of `DisposeAsync` after the flush — a racing sync path freeing the handle mid-flush is a native use-after-free. Internal async flushes avoid context capture per await; callers compose the configured form at the use site.
- Child-before-container, strictly sequential: stop intake, await child drains, dispose children, dispose container. Memory-mapped views are the silent instance — the BCL enforces no ordering, the OS reference count keeps the mapping live, and a writeable view flushed after the file object is marked disposed loses data without error: flush and dispose all views, then the file.
- Critical-section disposal rides the ref-struct pattern-dispose guarantee: the lock's scope value cannot be boxed, captured, or carried across `await`, so release is pinned to the lexical frame — a class-based disposable lock holder can be smuggled across `await` and released on the wrong thread or after the protected teardown. Pattern `Dispose` wins overload resolution over the interface, and converting a `ref struct` to a disposable interface is a forbidden boxing conversion, reachable only through anti-constraint generic code. No try-enter scope form exists — bounded-deadline disposal is try-enter with timeout plus explicit exit; a scope-dispose throwing the lock-ownership exception means a continuation rerouted the thread, and the held-by-current-thread probe is the diagnostic.
- The clock-driven polling capsule separates shutdown from interrupt: disposal makes current and future tick waits return `false` — the drain-time termination check — while a per-wait token throws and affects only that wait as the timer keeps firing; disposing from a cancellation callback conflates the two and suppresses false-return detection.
- Global drain is strict: fence new work, signal the root cancellation source, drain compute, flush persistence, complete UI, dispose handles child-before-container. Each step's deadline is a clock-derived cancellation window — a source constructed over an injected time provider, never a hardcoded delay; the fake clock fires the timer synchronously inside the advance call on the advancing thread, so advancing while holding a lock the callback acquires deadlocks.
- Source lifecycle: async cancel completes after all callbacks ran, where sync cancel can block on long callbacks; disposing races concurrent registration against freed state — dispose only after cancellation completed and the drain confirms no new registrations; reset succeeds only pre-cancellation with no pending timer and clears all registrations — a failed reset is terminal, allocate fresh. Linking accepts a span of parents without cascading intermediate sources.
- Channel drain: reader completion fires only after writer completion AND an empty buffer — the canonical drained await. For resource-bearing items the dropped-item callback is the only disposal seam for items rejected under drop modes (completion and drain never dispose buffered items; the reader disposes drained ones); it fires synchronously on the writer's thread and must not block, throw, or write the same channel; under drop-write it receives the just-rejected item that never entered the buffer. Synchronous-continuation opt-in lets a writer re-enter reader continuations inline and requires proving no reader holds a lock the writer takes; a prioritized drain follows the comparer's total order — a partial order drains non-deterministically.
- Pipe teardown order: writer completes, reader drains to completed, reader completes, then reset — which throws unless both sides completed and releases segments to the pool; reader-first completion silently discards unconsumed segments. Advance commits exactly the bytes written — overcommit is unvalidated garbage to the reader; examining past consumed is the partial-parse back-pressure signal; a zero pause threshold disables back-pressure entirely (the sentinel defaults resolve to real thresholds); routing continuations through a captured synchronization context must stay off for capsules holding a lock across read continuations.

[POOLED_AND_LEASED_RESOURCES]:
- Rented extents are logical, not physical: pool rentals return at-least-length arrays, the logical extent travels as a slice and never the raw array (trailing bytes carry stale data), and clear-on-return is mandatory for credential or keying material. Memory-pool rentals return an owner whose disposal returns the memory; the owner's memory is the usable surface and the owner itself never crosses the lifetime seam.
- Pool vocabulary never appears on public signatures: surfaces express extent as spans, read-only memory, or a domain value slicing internally — a visible owner type couples callers to return timing. Buffers return at phase boundaries — before emitting into the next stage, not after processing ends — or the buffer is held past the next renter's fill.
- Object-pool return does not reset: the resettable hook runs before pooling and a `false` result discards without disposing — identical to a policy rejection — so a degraded resettable or a rejecting policy must dispose in-band before returning `false`, or every rejection leaks. State bleeding across rentals is a silent correctness bug: a pooled object fully resets or holds nothing that matters across rental boundaries. The leak-tracking wrapper pays a weak-table allocation per rental in every build while diagnosing only debug — admit it behind the pool interface via the provider, debug only.
- Every rate lease disposes exactly once, success or failure — a custom limiter's failed lease may record denial evidence on disposal, so guarding disposal on the acquired flag silently loses it. A cancelled asynchronous acquire throws with no lease returned — a finally-dispose must guard the still-null local. Chained partitioned limiters release earlier successful inner leases internally when a later one fails; the caller still disposes the outer lease, a no-op over already-released inners.

[GATED_ACQUISITION_AND_REGISTRY_OWNED_SESSIONS]:
- The registry that constructs a session owns its disposal: call sites receive projected values, never the session reference, and never dispose — a call-site dispose tears down a session concurrently held elsewhere. Ephemeral per-call construct-use-dispose is forbidden: it defeats caching, pays initialization on every call, and races concurrent initializations toward one underlying resource.
- Cache keys are structurally complete — content identity plus full configuration; a partial key produces stale-hit false positives invisible until an excluded field diverges.
- Probe and construction are atomic from the caller's view — a gap where another caller observes probed-but-not-constructed yields stale-probe authorization after the resource has changed — and atomic-cell mutation lambdas are side-effect-free and may re-execute under contention, so construction and disposal both happen outside the swap. Idempotent construction may fuse probe and construct into one CAS; non-idempotent construction separates the claim from the build: a side-effect-free sentinel CAS takes exclusive ownership, the winner constructs exactly once outside the lambda, a second swap publishes; eviction snapshots the outgoing value, atomically removes, then disposes the snapshot. Non-idempotent construction inside a contended lambda leaks every losing invocation's allocation, and registry registrations made inside a spinning lambda leave earlier registrations pointing at disposed resources.
- Operation-scoped contexts — connections, transactions, unit-of-work — acquire at operation start inside the bracket and never persist as fields or across `await` continuations: resumption may change threads and violate provider affinity. Ambient context (a static async-local or injected singleton) makes lifetime invisible and disposable before forked consumers observe it; the explicit form injects a context factory called at the top of each operation.
- Maintenance-class connections acquire only inside the lifecycle state that permits them, and the transition out is the abort signal: an interrupted maintenance connection is quarantined — disposed immediately without further writes, never returned to any pool — and the operation emits a typed failure receipt carrying the interruption step index; the gate reopens only after the connection is closed, and the cancellation registration drives the quarantine path synchronously. Retry lives at the lifecycle machine, never inside the capsule.
