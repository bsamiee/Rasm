# Byte-Identity Wire Contract

[TWO_EQUALITIES_ARE_DISTINCT_CONTRACTS]:
- A wire field carries two independent contracts that must never be conflated: the *semantic* contract (does the decoded value round-trip) and the *byte-identity* contract (does the encoded octet sequence reproduce bit-for-bit). The structural-equality primitive over parsed values normalizes away every byte difference the byte-identity contract depends on, so passing a structural-equality check is no evidence that two payloads share a signature, checksum, or idempotency key.
- The number branch of structural equality parses each numeric token into sign, integral span, fractional span, and exponent, then realigns the integral/fractional boundary by the exponent delta before comparing digit runs. The consequence is that the four tokens `1`, `1.0`, `1.00`, and `10e-1` are all structurally equal yet have four distinct byte forms; any field whose value enters a signed envelope must be normalized to one canonical numeric spelling at admission, because the structural check will not catch the drift and the signature will.
- The string branch compares after unescaping, so two strings differing only in escape form — one carrying `<`, the other a literal `<` — are structurally equal and byte-divergent. A redaction or pass-through path that re-encodes a string with a different escaper passes every structural assertion while silently breaking byte identity.
- The design law: a field that participates in a signature, an HMAC, an idempotency key, a content-addressed hash, or a canonical-form comparison is a byte-identity field, and its contract is the exact octet sequence, not the parsed value. Such fields require a normalization pass at admission that fixes encoder, escape form, numeric spelling, and member order before any signing input is derived — and they must be excluded from any path that parses and reserializes, which is byte-lossy by construction.

[ENCODER_UNIFORMITY_IS_A_GLOBAL_INVARIANT]:
- The pre-escape value type stores both the source string and the encoded UTF-8 bytes, but its equality and hash are computed on the *source string alone*, ignoring the encoded bytes entirely. Two pre-escaped names built from the same source with different escapers are equal and hash equal while their `EncodedUtf8Bytes` differ. A cache or set keyed on the pre-escaped value therefore cannot distinguish encoder variants, and a write-time dispatch that selects the encoded bytes from such a cache will silently emit whichever variant won the cache race — a non-deterministic byte form.
- The conservative escaper hex-escapes every scalar outside a minimal allow-list — the structural characters, the C0 control set, and a short pre-escaped map (`\b \t \n \f \r \\` plus the quote) — encoding `<`, `>`, `&`, `+`, `'`, and every non-ASCII scalar as `\uXXXX`. The relaxed escaper leaves those literal. The two encoders therefore diverge on byte identity for any payload containing those characters even though both emit valid documents whose decoded semantics are identical. Mixing a relaxed-encoded writer with a default-encoded pre-escaped name inside one document yields a payload whose byte form depends on which sites used which encoder.
- The hex escape emits uppercase digits. Any external canonicalization profile or peer implementation that expects lowercase `\uxxxx` will compute a divergent hash over a byte-identical-by-intent payload; the escape-case is a silent byte-identity coordinate not surfaced by any option.
- The governing rule is one encoder per byte-identity domain, fixed at composition time and asserted, not chosen per site. Encoder choice is a contract coordinate equal in gravity to a property name; a per-call-site encoder selection is the same class of error as a per-call-site property rename, and the pre-escaped value's source-only equality means the type system cannot catch the divergence.

[NUMBER_AND_TIMESTAMP_FORMS_ARE_RUNTIME_NORMALIZED_LOSSILY]:
- Floating-point values are written through the shortest-round-trippable formatter with no format specifier — the byte form is the minimal decimal string that round-trips to the same bit pattern. This is deterministic for a given value on a given runtime, but a value that arrived on the wire as `1.500` is admitted, parsed to its bit pattern, and re-emitted as `1.5`: parse-then-reserialize is not a byte-preserving identity for numbers even when the value is unchanged. A signed numeric field cannot survive a deserialize/reserialize cycle and must be carried as its raw token bytes if its original spelling is part of the signed input.
- Timestamp serialization trims trailing zero sub-second digits: the round-trip writer formats the value, then walks the fractional region backward and drops trailing-zero digit positions, so a whole-second instant collapses from a seven-digit fractional form to no fractional form, and an instant with three significant fractional digits keeps exactly three. The serialized width of a timestamp field is therefore value-dependent, not fixed. A protocol that signs over fixed-width timestamps, or a peer that emits a fixed seven-digit fraction, will not byte-match this output; the wire contract must specify a fractional precision and the admission path must normalize to it before signing rather than relying on the formatter's variable-width output.
- Non-finite floating-point values are not written as numbers at all — they are emitted as quoted token literals under a dedicated handling mode, or rejected. A measurement field that may carry a non-finite sentinel changes JSON *kind* between number and string depending on the value, which is a structural contract break, not merely a byte difference; the admission gate must reject non-finite values for a number-typed field rather than let the writer silently re-kind it.
- Integral and decimal values are written through the invariant fixed-point formatter and carry no exponent, so their byte form is stable across cultures and runtimes; only the binary floating-point and timestamp paths carry runtime-normalization risk. The contract law follows the type: integral and fixed-decimal fields are byte-stable through a round-trip, binary-float and timestamp fields are not.

[PASS_THROUGH_PRESERVES_BYTES_PARSE_DOES_NOT]:
- The raw-value write copies the supplied octets verbatim into the output buffer and only sets the current token kind; with input validation skipped it performs no structural read and no re-minification. This is the one path that preserves the byte identity of an already-signed sub-document: an opaque inner payload received under signature is forwarded by capturing its raw span and re-emitting it through the raw write, never by parsing it into nodes and reserializing.
- The corollary is the prohibition: any path that materializes the sub-document into a parsed node tree and writes it back recomputes every byte — re-applies the active encoder, re-spells every number, re-orders nothing but re-escapes everything — and destroys the inner signature. Envelope patching that must add or replace an outer field while preserving an inner signed payload must mutate at the byte or positioned-node level around the opaque region, never round-trip the whole envelope through the serializer.
- Skipping input validation on the raw write is safe only for octets whose provenance is trusted within the same request context; for externally sourced bytes the validation pass is mandatory and costs a full structural read, but it does not alter the bytes — it only proves they are well-formed before the verbatim copy. Validation and minification are orthogonal: the raw write never minifies, validated or not, so insignificant whitespace inside a trusted-but-pretty-printed sub-document is preserved and becomes part of the forwarded byte identity.
- A zero-copy raw span obtained directly from a parsed document's backing buffer is the signing input of choice precisely because it is the exact admitted bytes, but it is valid only while that document is live; deriving a signature from it after the document is disposed reads freed memory. Acquire the raw span, derive the hash or feed the signer, and complete that work inside the document's synchronous scope — never carry the span across a suspension point, and never carry the hash-input derivation past the document lifetime.

[CANONICAL_MEMBER_ORDER_IS_A_SIGNED_COORDINATE]:
- Member order in the output is not normalized by the serializer; it follows the contract's property sequence, which for a generated contract reflects declaration order and is overridable per property. Two payloads carrying the same members in different orders are structurally equal but byte-divergent, so member order is a byte-identity coordinate. A canonical-form profile that mandates lexicographic member order must enforce it through the contract's per-property order coordinate at definition time; relying on incidental declaration order makes the canonical form an undeclared invariant that the next field addition silently breaks.
- A duplicate key admitted under a lenient parser collapses to last-write-wins and is invisible to the parsed value, but the duplicate's bytes were part of the signing input. The admission posture for a signed boundary rejects duplicate keys outright so the byte form the signer saw and the value the domain sees cannot diverge; a silent collapse means the signature covered bytes the domain never observed.
- Insignificant whitespace and key ordering are the two normalization axes a canonical profile must pin, and they sit at opposite layers: whitespace is a writer concern fixed by emitting in the compact mode the serializer uses by default, while key order is a contract concern fixed at the property level. A byte-identity contract specifies both explicitly and treats either drifting as a breaking change.

[NORMALIZE_AT_ADMISSION_DERIVE_SIGNING_INPUT_ONCE]:
- The single correct structure for a signed boundary is: admit the raw bytes once, validate structure and reject duplicates and non-finite re-kinding, normalize into a canonical owner whose serialization coordinates — encoder, escape case, numeric spelling, member order, timestamp precision — are all fixed at definition time, then derive every signing input and idempotency key from that canonical owner's emitted bytes, captured zero-copy within the live scope. The canonical bytes are computed exactly once and reused; the value never re-enters the serializer between signing and transmission.
- Verifying an inbound signature requires the inverse and is the harder direction: the signature was computed over the peer's exact bytes, so verification must run against the admitted raw octets — captured before any parse — not against a re-serialization of the parsed value, because the local serializer's encoder, number form, and member order will not reproduce the peer's bytes. The admission path therefore holds the raw span alongside the parsed value through verification, and only after the signature checks against the raw bytes does the parsed value cross into the domain.
- A typed receipt that records what was signed carries the canonical-bytes hash and the encoder, escape-case, member-order, and timestamp-precision coordinates as discriminant fields — not the raw payload — so that a later byte-identity dispute is resolved against recorded coordinates rather than a re-derivation that may have drifted. The receipt makes the byte-identity contract auditable as a set of fixed coordinates rather than an implicit property of the code that happened to run.
