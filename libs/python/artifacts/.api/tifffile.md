# [PY_ARTIFACTS_API_TIFFFILE]

`tifffile` is the TIFF/BigTIFF/OME-TIFF container owner the `export/layered#LAYERED` `TIFF` arm composes BENEATH its layer-structure author: a pure-Python reader/writer that owns the WHOLE IFD directory, strip/tile layout, byte-order/word-size selection, the standard + private tag set, and the `numpy`-array<->container round-trip the codec and structure owners do NOT — `psdtags` authors the Photoshop `ImageSourceData`/`ImageResources` tag GRAPH and `imagecodecs` compresses each channel byte, but `tifffile` is the one place a TIFF directory is laid out, a tag is written into an IFD, and a strip is read back to an `NDArray`. For the layered-egress plane it owns exactly the container concern: `imwrite(sink, merged, photometric='rgb', extrasamples=('unassalpha',), metadata=None, byteorder=source_data.byteorder, extratags=(source_data.tifftag(maxworkers=4), resources.tifftag()))` lays the flat RGBA composite AND the two `psdtags`-emitted extratags into one TIFF that Photoshop/Affinity/Krita read as a separable layer stack, and `TiffFile`/`imread` read the directory back. It never compresses a channel (it DELEGATES every codec to `imagecodecs` through its internal `_imagecodecs` registry — `tifffile` selects the codec by `COMPRESSION` enum and hands the strip to `imagecodecs`, never re-implementing deflate/PackBits/LZW), never authors a PSD layer record (`psdtags`), and never authors a native `.psd`/`.psb` document (`PhotoshopAPI`/`psd-tools` per the `[03]-[PACKAGE_ROSTER]` PSD decision). Beyond the layered-TIFF egress it is the artifacts TIFF SPINE: the `imread`/`imwrite` polymorphic array<->TIFF face for any single-image or multi-page TIFF the document/figure/raster owners emit or ingest, the `memmap`/`aszarr` lazy-tiled access for a large drawing-sheet raster, the GeoTIFF tag surface for a georeferenced site plan, the C2PA-manifest detection for a provenance-sealed TIFF, and `validate_jhove` as the TIFF/IT (TIFF for image-technology) preflight oracle. It stacks the artifacts raster band as the TIFF-CONTAINER discriminant: `tifffile` owns the TIFF directory (incl. layered-TIFF extratags), while `psdtags` owns the PSD tag structure laid INTO it, `imagecodecs` owns the strip codec, `pyvips`/`Pillow` own the streaming/in-process raster decode of every OTHER format, and the universal `numcodecs`/`zarr` own array-chunk storage — one container owner per concern, never a duplicate TIFF directory writer across the boundary.

## [03]-[ENTRYPOINTS]

[ENTRYPOINT_SCOPE]: the array<->TIFF round-trip the `export/layered` `TIFF` arm composes, the multi-page/pyramid writer, and the lazy-tiled read
- rail: layered

The `TIFF` arm's whole container concern is ONE `imwrite` call: it composites the visible layers to a merged RGBA `NDArray`, authors the `psdtags` `TiffImageSourceData`/`TiffImageResources`, derives each as a `tifftag(...)` extratag tuple, and lays the flat image + both tags into one TIFF with `photometric='rgb'`, `extrasamples=('unassalpha',)`, `metadata=None` (no tifffile-private shaped/JSON metadata, which would corrupt the Photoshop read), and `byteorder=source_data.byteorder` (read off the authored graph, never hardcoded). The `extratags` element is a `TagTuple` = `(code, dtype, count, value, writeonce)` — exactly what `psdtags.TiffImageSourceData.tifftag(...)` returns — so the structure owner emits the tuple and the container owner lays it into the IFD; the two meet at the tuple and the file path. `maxworkers` on `imwrite`/`write` is the STRIP-compression thread fan inside `imagecodecs` (a thread pool the GIL-releasing native codec parallelizes), NOT an `anyio` worker — it stays bounded inside the already-`to_process`-offloaded `TIFF` arm.

## [04]-[IMPLEMENTATION_LAW]

[LAYERED_TIFFFILE]:
- import: `import tifffile` at boundary scope only; module-level import is banned by the manifest import policy. The `export/layered` page reifies the container owner through its `lazy import tifffile` proxy on first `TIFF`-arm use, exactly as it `lazy import`s `psdtags`/`pyvips`/`numpy` — the codec, OME-XML, and zarr deps are optional extras pulled only when the dependent path runs, and the `TIFF` arm runs on the `Band.WORKER` `to_process` lane, so `tifffile` never loads on the runtime loader path.
- container here, structure there, codec elsewhere — three disjoint owners meeting at typed seams: `tifffile` lays out the IFD directory and writes the flat image + the `psdtags`-emitted extratags; `psdtags` authors the `TiffImageSourceData`/`PsdLayers`/`PsdLayer`/`PsdChannel` GRAPH and emits each as a `tifftag(...)` TagTuple; `imagecodecs` compresses each strip (selected by the `COMPRESSION` enum, dispatched through `tifffile._imagecodecs`, never re-implemented). Never re-implement a TIFF directory write in `psdtags`, a channel codec in `tifffile`, or a PSD layer record in `imagecodecs` — the `_psd_layer` arm builds the typed graph, `tifftag` produces the IFD value tuple, `imwrite` lays it in, and the strip codec is `imagecodecs`'.
- byte-order + photometric + metadata are a single contract, not three knobs: the layered `imwrite` MUST carry `byteorder=source_data.byteorder` (read off the authored `PsdFormat.BE32BIT` graph -> `'>'`, never hardcoded at the `imwrite` site), `photometric='rgb'` (the Photoshop colour-read contract), and `metadata=None` (so no tifffile shaped/JSON metadata corrupts the read). The default `metadata={}` writes a shaped-NumPy JSON ImageDescription — explicitly pass `metadata=None` for a Photoshop-compatible layered TIFF; a mismatched byte-order or a non-`None` metadata produces a TIFF Photoshop reads as flat or rejects.
- the extratag is a 5-tuple, not a method call across the boundary: the `extratags=` element is the `TagTuple` `(code, dtype, count, value, writeonce)` `psdtags.TiffImageSourceData.tifftag(maxworkers=4)` and `TiffImageResources.tifftag()` return — `tifffile` receives the tuple and lays it into the IFD verbatim. The `maxworkers` inside `tifftag` is the `imagecodecs` channel-compression thread fan, and the `maxworkers` on `imwrite` is the STRIP-compression thread fan; both are `imagecodecs` thread pools bounded INSIDE the one `to_process` crossing, never a second `anyio` worker layered on the arm.
- the flat image is the merged composite, the extratags carry the editable layers — one `NDArray` plus two tags, not a per-layer page write: the `TIFF` arm writes ONE merged RGBA canvas as the flat TIFF image (the `pyvips` `composite(placed[1:], modes[1:])` result the ORA path produces) and lays the editable layer structure alongside it as the `37724`/`34377` extratags `psdtags` authored — NOT a multi-page TIFF with one page per layer (Photoshop reads the `ImageSourceData` tag, not extra pages). The merged array is a contiguous `numpy` `uint8` `(H, W, 4)` RGBA; `extrasamples=('unassalpha',)` declares the 4th sample straight (non-premultiplied) alpha.
- `imwrite`/`imread` is the polymorphic face, `TiffWriter`/`TiffFile` the incremental owner — one entry per concern, not a `write_tiff`/`write_bigtiff`/`write_ome` family: a single-image write is `imwrite(path, array, **opts)`, a single-image read `imread(path)`; a multi-page/pyramid/SubIFD build opens `with TiffWriter(...) as w: w.write(...)` per page; a large/lazy read opens `with TiffFile(...) as t:` and walks `t.pages`/`t.series`/`t.asarray(level=)`. `bigtiff=True` and `kind=` ('ome'/'imagej'/'shaped'/'generic') are keyword discriminants on the ONE face, never parallel writer types.
- the native handle is `with`-bracketed, never GC-reaped: both `TiffWriter` and `TiffFile` are context managers; the `TIFF` arm opens the writer in a `with` whose body runs `write`/`overwrite_description` while the handle is live and whose `__exit__` finalizes the directory, exactly as the `PDF` arm brackets its `pymupdf`/`pikepdf` handles. A writer left for GC may flush an incomplete directory; a reader left open holds the file/mmap. The `TIFF` arm's `BytesIO()` sink is the in-memory target so the finalized bytes are `sink.getvalue()` once the `with` closes.
- round-trip + JHOVE are the correctness oracles: `psdtags.TiffImageSourceData.fromtiff(written) == source_data` (the layered-tag round-trip `psdtags` owns) plus an OPTIONAL `validate_jhove(path)` (the container-level TIFF validation) are the integrity signals; a read-back that re-parses to an identical graph and a JHOVE-clean container is the layered-TIFF correctness contract. The layered-TIFF receipt records the canvas shape, `byteorder`, the per-channel compression, and the round-trip equality result; the codecs are all lossless so no precision-loss field is recorded.
- evidence: the `TIFF`-arm `LayerFact` captures the canvas `(H, W)` shape and `byteorder`, the `PHOTOMETRIC`/`EXTRASAMPLE`/`COMPRESSION` the container wrote, the `maxworkers` strip-fan, whether a `TiffImageResources` thumbnail rode the `34377` tag, the `imwrite` `returnoffset` `(offset, bytecount)` when taken, and the `fromtiff`-round-trip equality result — folded onto the layer-export span as the `core/receipt#RECEIPT` `ArtifactReceipt.Preview` case (the editable layered TIFF is a preview-class artifact), the authored-layer count on the shared `overlays` slot.
- boundary: `tifffile` owns the TIFF CONTAINER only — it never authors a PSD layer record (`psdtags`), never compresses a strip byte (`imagecodecs` via its `_imagecodecs` registry), never composites the canvas (`pyvips`), never authors a native `.psd`/`.psb` document (`PhotoshopAPI`/`psd-tools` per the roster PSD decision), and never decodes a non-TIFF raster (`pyvips`/`Pillow`). It re-renders nothing and re-lays-out nothing — it receives a merged canvas `NDArray` and the `psdtags`-emitted extratags and writes the TIFF directory the structure is laid into. The CLI `imshow` viewer and `matplotlib` plot are the console-script path only, never a library render.

## [05]-[INTEGRATION_LAW]

[STACKING] — the catalog stacks BOTH `.api` tiers: the shared/universal `libs/python/.api/*` rails ON TOP OF this folder package, never the folder set alone.

- structure seam (folder `psdtags` `.api`): `tifffile` and `psdtags` are the two halves of the layered-TIFF egress — `psdtags` owns the `ImageSourceData`/`ImageResources` tag GRAPH and emits each as a `tifftag(...)` `TagTuple`; `tifffile.imwrite(sink, merged, photometric='rgb', extrasamples=('unassalpha',), metadata=None, byteorder=source_data.byteorder, extratags=(source_data.tifftag(maxworkers=4), resources.tifftag()))` lays the flat RGBA composite AND those two tags into one TIFF, and `psdtags.TiffImageSourceData.fromtiff(file)` reads them back THROUGH `tifffile` (`psdtags` opens the file with `tifffile.TiffFile` and pulls the raw `37724`/`34377` tag values via its OWN `psdtags.read_tifftag`, which delegates the directory parse to `tifffile`). They meet at the `TagTuple` and the file path, never re-implementing each other; the `psdtags.md` sibling already declares `tifffile` as the container owner it emits extratags FOR — this catalog is the mirror: `tifffile` receives the extratag and owns the directory write, `psdtags` owns the tag structure. `byteorder=source_data.byteorder` is the single shared contract, not two independent knobs.
- codec seam (folder `imagecodecs` `.api`): every TIFF strip/tile codec is `imagecodecs`' — `tifffile` selects the codec by the `COMPRESSION` enum and dispatches the strip byte transform through its internal `tifffile._imagecodecs` registry (`ADOBE_DEFLATE`/`DEFLATE` -> `imagecodecs.deflate_*`, `PACKBITS` -> `imagecodecs.packbits_*`, `LZW` -> `imagecodecs.lzw_*`, the `PREDICTOR` pre-pass -> `imagecodecs.delta_*`/`floatpred_*`), so `tifffile` NEVER re-implements a codec and a `COMPRESSION` whose `imagecodecs` core is absent raises at the write boundary (the `<CODEC>.available` capability the `imagecodecs.md` sibling owns), never mid-strip. The `imagecodecs.md` `tiff_decode` row is the whole-container decode `imagecodecs` itself exposes — the two packages share the codec layer; the layered arm uses `tifffile` for the directory and lets it reach `imagecodecs` for the strips, never calling `imagecodecs` directly for a TIFF strip.
- numeric seam (universal `numpy` `.api`): the container boundary is `NDArray -> TIFF file` (`imwrite(sink, merged)`) and `TIFF file -> NDArray` (`imread`/`TiffPage.asarray`); the merged composite is a contiguous `numpy` `uint8` `(H, W, 4)` RGBA the `TIFF` arm writes, and a read returns an `NDArray` whose dtype/shape `tifffile` derives from the IFD (`SAMPLEFORMAT`+`bitspersample`). `numpy>=2.1` is the one HARD runtime dep — the universal `numpy` rail owns the array, `tifffile` owns its TIFF serialization; `memmap`/`aszarr` return a `numpy.memmap`/`zarr`-backed `NDArray` view for over-RAM access, never a manual whole-image load-then-slice.
- model seam (universal `msgspec`/`pydantic` `.api`): the layered-export policy is a typed `msgspec.Struct` — the `LayerPolicy` save knobs (`garbage`/`deflate`) and the `ExportTarget.TIFF` discriminant are validated tags, and the container options the arm passes (`photometric='rgb'`, the `COMPRESSION`/`PREDICTOR` member, `byteorder`) are closed-enum members or fixed contract values, never free-form strings reaching `imwrite`. `tifffile`'s own `PHOTOMETRIC`/`COMPRESSION`/`EXTRASAMPLE`/`PREDICTOR` `IntEnum`s are the authored target, the pipeline `msgspec` model the validated source — they meet at the `imwrite` keyword. A compression choice surfacing from policy is a closed `COMPRESSION` member, never a `compression='deflate'` string the writer re-parses.
- contract seam (universal `beartype` `.api`): annotate the `TIFF`-arm boundary `Canvas RGBA NDArray + extratags Sequence[TagTuple] -> LayerFact` so a non-array canvas, a wrong-dtype merged image, or a malformed `TagTuple` is rejected at the contract, not deep in a `tifffile` IFD-write frame or an `imagecodecs` Cython strip; the `merged.ndim == 3 and merged.shape[2] == 4` and `merged.dtype == uint8` RGBA invariants are the contract predicates the arm reads before `imwrite`, surfacing a shape/dtype defect as the `export/layered` `ExportFault` rail rather than a raw `TiffFileError` on read-back.
- resilience seam (universal `stamina` `.api`): the `tifffile` directory write and `imagecodecs` strip codecs are CPU-pure and deterministic — never wrap them in `@retry`; a malformed array, an absent codec core, or a corrupt-TIFF parse surfaces immediately as a terminal `TiffFileError`/`ExportFault`, never a transient retry trigger. Retry belongs only on the artifact-store IO around the produced TIFF bytes (owned by `core/plan`) and on the `validate_jhove` JRE subprocess (a flaky external process, the one retriable boundary), never on the container author or the strip transform.
- observability seam (universal `structlog`+`opentelemetry-api` `.api`): record the layered-TIFF receipt — canvas shape, `byteorder`, `PHOTOMETRIC`/`EXTRASAMPLE`/`COMPRESSION`, the strip `maxworkers`, the thumbnail-tag flag, the `returnoffset` `(offset, bytecount)`, and the `fromtiff`-round-trip equality result — as a structured event on the layer-export span; the `validate_jhove` verdict (when run) is the container-integrity signal a downstream reader trusts that the written TIFF is a valid directory, paired with the `psdtags` round-trip as the layer-structure-integrity signal.
- concurrency seam (universal `anyio` `.api`): the `TIFF` arm runs on the `export/layered` `Band.WORKER` lane (`_offloaded` over `anyio.to_process.run_sync` bounded by the module-level `_GATE` `CapacityLimiter`=4) because it reuses the ORA `pyvips` placement/composite path and libvips is off the runtime loader path — the `pyvips` composite, the `psdtags` graph author, the `tifffile.imwrite` directory write, and the `imagecodecs` strip compression all cross ONE process worker seam, never the in-process `Band.CORE`/`_THREAD_GATE` `to_thread` lane the `SVG`/`PDF` arms use. The `imwrite(maxworkers=…)` strip-compression thread fan (and the `tifftag(maxworkers=…)` channel fan) stay bounded INSIDE that single process crossing — they are `imagecodecs`' internal pools, never a second `anyio` worker layered on the arm. A many-file `TiffSequence.asarray(ioworkers=…)` read similarly bounds its own per-file fan inside one worker.
- artifacts-rail boundary: `tifffile` owns the TIFF CONTAINER — the `TIFF` arm of the `export/layered` editable-export family AND the artifacts TIFF array<->container spine. The sibling layered-export owners stay disjoint: `drawsvg` owns the named-`<g>` SVG layer document, `pymupdf`+`pikepdf` own the PDF optional-content-group layers, `pyvips`/`lxml`/`stream-zip` own the OpenRaster `ORA` composite/manifest/container, and `PhotoshopAPI`/`psd-tools` own the native `.psd`/`.psb` document (the roster PSD decision keeps PSD authority off this stack — `psdtags`+`tifffile` are the layered-TIFF container ONLY). The sibling RASTER owners stay disjoint too: `pyvips` owns the streaming decode/transform of EVERY non-TIFF raster (and the TIFF composite the `TIFF` arm feeds in), `Pillow` the in-process raster edge, `imagecodecs` the strip codec, and the universal `numcodecs`/`zarr` the array-chunk store — `tifffile` reaches `zarr` (`aszarr`) for the lazy-tiled TIFF view but never owns array-chunk storage of a non-TIFF array. Route a Photoshop-compatible layered TIFF through `psdtags`+`tifffile`, a generic single/multi-page TIFF through `tifffile.imwrite`/`imread`, a native `.psd` through `PhotoshopAPI`/`psd-tools`, an editable PDF through `pymupdf`/`pikepdf`, a GIMP/Krita stack through the `ORA` arm — one container owner per format, never a `tifffile`-authored `.psd` or a duplicate TIFF directory writer.
