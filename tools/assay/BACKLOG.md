- **DOC502 vs structured `Raises:`** — ruff DOC502 rejects a `Raises:` section without an in-body `raise`, so propagate-through-helper exceptions (e.g. `ArtifactStore.write_bytes`/`write_bytes_path` -> `_write_at`'s `FileExistsError`) cannot use the machine-readable `Raises:` spelling an agent-first envelope wants; they document propagation as a prose docstring body line. Candidate for a repo-wide convention decision or a pydoclint config exception for propagated exceptions, tracked not fixed per-site.
- **`registry._distill` reaches private `engine._measure` (composition->core)** — the resource-snapshot fallback imports the private measurement owner (beside the pre-existing private `_RESOURCE`/`fan_out` reach, `# noqa: PLC2701`). One-hop co-owned seam, stable today; if the `Measurements`/`to_resources` shape ever forks, promote it to a public engine seam rather than widening the private cross-module coupling.
- **`fold` STATIC EMPTY→OK promotion over-applies via claim-keying** — `core/model.fold` promotes `claim in _PROCESS_BACKED_OK_CLAIMS` (incl. `Claim.STATIC`) + folded-EMPTY + outcomes + no-defects → `OK`. This is correct and LOAD-BEARING for the static RAIL (clean run = OK; `test_rail_static` depends on it — do NOT carve EMPTY out of the central fold, it would break the rail). But it over-applies to non-rail folds reusing `Claim.STATIC`: `registry._delta_report` works around it with a local `msgspec.structs.replace(status=EMPTY)` re-pin (run-not-found must stay EMPTY), and `automation.drive`'s `fold(Claim.STATIC,"program",…)` inherits it unintentionally. Root-up fix: make the EMPTY→OK promotion an explicit rail-opt-in (e.g. a `fold(..., promote_empty=True)` the rails pass) rather than a blanket claim rule, then delete the delta local guard and let the automation program fold report EMPTY. Touches every `fold` call site — a careful refinement, not a plan tier.
- **`Diagnostic.failing_step` carries `Step` members in-process but plain `str` after a wire round-trip** — `==`/`!=` are safe (StrEnum), but identity (`is Step.X`) on decoded envelopes silently fails. The one comparison site uses `!=`; flag if identity comparison is ever introduced.
- **Static streaming gap** — `fan.schedule`/`dotnet.slot_wait`/`concurrency.pressure` are structured events at the execution boundary (structlog + OTel `add_event`, one-envelope stdout preserved), but per-phase start/end, per-process receipts, SARIF folds, and discovered-diagnostics emission are still buffered until the final envelope.
- **`SeamProbe.projected[K]` unused** in the assay specialization (`commands`/`checks` read `captured` directly) — retained as a real engine affordance (post-hoc `calls`-log view distinct from the per-call `project` stream); flag so a reviewer does not strip it as dead.

## Upstream issues worth filing

### [UPSTREAM] mutmut stats-phase crash — fixed locally (`max_stack_depth = -1`); issue still worth filing
The previously documented rootdir/cwd theory is **falsified** (the captured stats argv is well-formed and pytest loads the parent pyproject config — the prepended addopts prove it). Root cause, established empirically with `debug=true`: with `max_stack_depth != -1`, `record_trampoline_hit` (`mutmut/__main__.py:130`) walks caller frames calling `Path(co_filename).resolve(strict=True)`; module-level calls into mutated code during conftest import put `<frozen importlib._bootstrap>` pseudo-filenames on that stack, the strict resolve raises `FileNotFoundError`, the conftest import dies, pytest exits 4, and mutmut raises `BadTestExecutionCommandsException`. The local fix is the upstream-default `max_stack_depth = -1` (repo had 10), which disables the strict-resolve stack walk entirely. Anyone with `max_stack_depth != -1` plus import-time calls into mutated code still hits this — file upstream as-is:

> **TITLE**: record_trampoline_hit crashes with FileNotFoundError on '<frozen importlib._bootstrap>' frames when max_stack_depth != -1 and mutated code runs at import time.
> **BODY**: With max_stack_depth set (e.g. the README's max_stack_depth=8), record_trampoline_hit (mutmut/__main__.py, the `while c and f:` loop) calls Path(f.f_code.co_filename).resolve(strict=True) on every caller frame. When a mutated function is invoked at module import time (module-level constant computed by a mutated function, imported transitively from a conftest), the caller stack contains importlib machinery frames whose co_filename is the pseudo-filename '<frozen importlib._bootstrap>' (also '<frozen importlib._bootstrap_external>', '<string>'). strict=True raises FileNotFoundError('<cwd>/<frozen importlib._bootstrap>'), the conftest import aborts, pytest exits 4, and mutmut raises BadTestExecutionCommandsException — misleadingly suggesting the pytest command itself is malformed.
> **REPRO (minimal)**: src/m.py with `def cap() -> int: return 1`; src/pkg using `X = cap()` at module level; a conftest.py importing pkg; [tool.mutmut] source_paths=['src'], max_stack_depth=8; `mutmut run` -> stats phase exit 4 with the traceback above; with max_stack_depth=-1 the run proceeds.
> **OBSERVED ON**: mutmut 3.6.0, pytest 9.0.3, CPython 3.15.0b1, macOS.
> **FIX CANDIDATES**: (a) skip frames whose co_filename starts with '<' before resolving; (b) resolve with strict=False; (c) wrap the resolve in try/except FileNotFoundError and continue; (d) break on importlib frames the same way 'pytest'/'unittest' filenames break the walk. Additionally, the exit-4 path would be far easier to debug if BadTestExecutionCommandsException included the captured pytest output even without debug=true.

When filing, also mention the forked-children consequence: mutant workers fork from the post-clean parent, so any set-once global poisoned across in-process `pytest.main()` calls silently converts test-infrastructure failures into false "killed" verdicts — an argument for subprocess-isolated stats/clean phases in mutmut core.

### [UPSTREAM] cyclopts help renderer crashes on Enum-hinted params with None defaults
cyclopts 4.16.1 `help/help.py` (the `if argument.show_default:` block) renders Enum-hinted defaults via `default_val.name` BEFORE consulting any `show_default` callable, so `language: Language | None = None` crashes `--help` with `AttributeError: 'NoneType' object has no attribute 'name'` regardless of the callable form (`show_default=lambda d: "" if d is None else d.name` was tried and is structurally unreachable for Enum hints).
