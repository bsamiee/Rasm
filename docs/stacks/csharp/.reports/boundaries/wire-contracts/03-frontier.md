# Wire-Contracts: Frontier

[ZERO_COPY_DESERIALIZED_DOCUMENT_ACCESS]:
- `System.Runtime.InteropServices.JsonMarshal.GetRawUtf8Value(JsonElement)` returns a `ReadOnlySpan<byte>` view directly into the `JsonDocument`'s backing buffer — no allocation, no copy. The span is valid only while the owning `JsonDocument` is live; it becomes dangerously stale the moment the document is disposed. Acquire the span, perform all comparisons or hash-inputs within the same synchronous scope, and never store the span across an `await` boundary or beyond the document's `using` scope. `JsonMarshal.GetRawUtf8PropertyName(JsonProperty)` provides the same zero-copy span for property names, enabling allocation-free property-name dispatch in a custom converter without constructing a `string`.
- `JsonMarshal` is in `System.Runtime.InteropServices`, not `System.Text.Json` — callers must add the using. The class is marked explicitly unsafe because the safety invariant (live document) is not enforced by the type system, though `unsafe` is not required to call the methods. Wire seam code that caches the span across the document boundary is a memory-corruption vector, not merely a logic bug.

[INCREMENTAL_READER_STATE_FOR_PARTIAL_PIPELINE_INPUT]:
- `Utf8JsonReader` is a `ref struct` over a `ReadOnlySpan<byte>` or `ReadOnlySequence<byte>`. For partial reads from a `PipeReader`, call `PipeReader.ReadAsync`, take the buffer from `ReadResult.Buffer`, construct a new `Utf8JsonReader(sequenceSlice, isFinalBlock: false, state: _savedState)`, advance as far as possible, capture `reader.CurrentState` into `_savedState`, then call `PipeReader.AdvanceTo(consumed: reader.Position, examined: buffer.End)`. In the multi-segment `ReadOnlySequence<byte>` path, `reader.Position` is a `SequencePosition`, not a byte count; passing `reader.BytesConsumed` to `AdvanceTo` is incorrect because `BytesConsumed` counts from the head of the current sequence slice. `reader.Position` (which resolves within the sequence) is the only correct consumed argument.
- `JsonReaderOptions.AllowMultipleValues = true` enables `Utf8JsonReader` to tolerate zero or more whitespace-separated top-level values in the same buffer; without it, a second top-level value causes `JsonException`. Set this on the `JsonReaderState` before the first partial-read constructor call; it is preserved across state transitions because it is stored in `JsonReaderState.Options`. This is the reader-level option backing the NDJSON streaming path.
- `Utf8JsonReader.TrySkip()` returns `false` when the current segment does not contain enough data to skip the complete value. Check the return value; if `false`, abandon the current reader, advance `PipeReader.AdvanceTo` with `examined: buffer.End` and no consumed advance, and retry after the next `ReadAsync`. Advancing the consumed position on a `false` return drops bytes silently.
- `Utf8JsonReader.ValueTextEquals(ReadOnlySpan<byte>)` compares the current token value to a UTF-8 literal without allocating a string. The comparison handles JSON escape sequences in the token. `static readonly byte[]` fields holding UTF-8 property name literals encoded at module initialization are the dispatch-table keys for allocation-free property routing in custom converters.

[PREALLOCATED_PROPERTY_NAME_ENCODING]:
- `JsonEncodedText.Encode(ReadOnlySpan<char>, JavaScriptEncoder?)` pre-escapes a property name once at initialization time and stores the result as `JsonEncodedText`, a value type backed by the encoded UTF-8 bytes. `Utf8JsonWriter.WriteString(JsonEncodedText, ...)` and `WritePropertyName(JsonEncodedText)` skip all escaping at write time because the name is already encoded. For high-throughput serialization seams with a fixed schema, a `static readonly JsonEncodedText` per property name eliminates both the per-write string allocation and the per-write escape scan. `JsonEncodedText.EncodedUtf8Bytes` exposes the pre-encoded bytes as a `ReadOnlySpan<byte>` for use in hash or comparison operations.
- `JsonEncodedText` instances must be created with the same `JavaScriptEncoder` as the `Utf8JsonWriter` they will be written to. Mixing `UnsafeRelaxedJsonEscaping` instances with default-encoder writers produces protocol-incorrect output for characters above ASCII 127. Fix the `JavaScriptEncoder` at the outbound seam initialization site, and use the same encoder in all `JsonEncodedText.Encode` calls at that site.

[LOCK_SCOPE_FOR_CONTRACT_FREEZE_GUARDS]:
- `System.Threading.Lock` is a dedicated mutex type; `Lock.EnterScope()` returns a `Lock.Scope` `ref struct` that releases when `Dispose` is called, composing with `using` without allocation. `Lock.IsHeldByCurrentThread` allows diagnostic assertion that a critical section is already held before entering a nested call. For lazy-sealed `JsonSerializerOptions` initialization: `if (!_options.IsReadOnly) { using var _ = _initLock.EnterScope(); if (!_options.IsReadOnly) { /* mutate */ _options.MakeReadOnly(); } }` is the correct double-checked sealed singleton pattern. `Lock` must not be used across `await` boundaries — the `using` block must not contain any `await`.

[TIMEPROVIDER_TIMESTAMP_FOR_RECEIPT_ELAPSED_FIELDS]:
- `TimeProvider.GetTimestamp()` returns a high-frequency counter value. `TimeProvider.GetElapsedTime(long startingTimestamp)` and `GetElapsedTime(long startingTimestamp, long endingTimestamp)` convert a timestamp pair to a `TimeSpan`. A typed receipt carrying an elapsed field captures `timeProvider.GetTimestamp()` before the outbound call and stores `timeProvider.GetElapsedTime(start)` at receipt construction. This is the only clock-agnostic elapsed measurement: `Stopwatch` is not injectable, `DateTime` subtraction loses sub-millisecond precision on platforms where the system clock ticks coarsely, and `Environment.TickCount64` is not accessible through `TimeProvider`. The receipt's elapsed field type is `TimeSpan`, serialized as an ISO 8601 duration via a registered `JsonConverter<TimeSpan>` since `TimeSpan` has no built-in `System.Text.Json` representation.
- `CancellationTokenSource(TimeSpan delay, TimeProvider timeProvider)` creates a token source whose cancellation is driven by the injected clock. Per-call timeout tokens constructed this way remain deterministic under `FakeTimeProvider` in tests; tokens constructed with `CancellationTokenSource(TimeSpan)` alone use the real wall clock and cannot be fast-forwarded. Every per-request `CancellationTokenSource` for call timeout at a wire seam must take the injected `TimeProvider`.

[ATOMIC_CELL_AS_WIRE_STATE_GATE]:
- `Atom<T>.SwapIO(Func<T, T>)` returns an `IO<T>` that performs the compare-and-swap atomically and yields the committed value after the swap. This is the correct cell type for a wire-seam state record that multiple forked `IO` computations must update — a connection registration table or a per-lane sequence counter — because `SwapIO` integrates into the IO effect chain without requiring explicit locking. `Atom<T>.ValueIO` is an `IO<T>` that reads the current value when evaluated; it defers the read into the chain rather than closing over the value at construction time, which is essential for correctness when the atom is updated between chain construction and execution.
- `Atom<T>.SwapMaybeIO(Func<T, Option<T>>)` performs the swap only when the function returns `Some`; a `None` result leaves the atom unchanged and returns the current unmodified value. This is the gate mechanism for a conditional state transition — retiring a pending receipt only when its sequence number matches the expected drain position — without introducing explicit branching. The function may be called multiple times under contention; it must be pure and free of observable side effects.

[CANCELLATIONTOKEN_SCOPE_HIERARCHY_AT_WIRE_SEAMS]:
- `CancellationTokenSource.CreateLinkedTokenSource(ReadOnlySpan<CancellationToken>)` creates a composite token that fires when any constituent fires, accepting a stack-allocated span to avoid the heap allocation of the array overload. At a wire seam with a per-session token and a per-call timeout token, `CreateLinkedTokenSource(stackalloc CancellationToken[] { sessionToken, callTimeoutCts.Token })` produces the composite token for the outbound network call. The source must be disposed after the call to release the registrations; a leaked `CancellationTokenSource` holds a registration in the parent token's callback list for the session lifetime.
- `CancellationTokenSource.CancelAsync()` communicates cancellation and returns a `Task` that completes after all registered callbacks and continuations have finished. At a wire seam where cancellation triggers cleanup effects registered via `CancellationToken.Register`, `CancelAsync()` ensures the cleanup runs to completion before the seam proceeds; `Cancel()` runs callbacks synchronously on the caller's thread and may deadlock if a callback acquires a lock held by the caller. For seam-level cancellation that cascades into `IO`-registered cleanup, `CancelAsync()` is always the correct choice.

[JSONOBJECT_INDEXED_MUTATION_AS_ENVELOPE_PATCHING]:
- `JsonObject.GetAt(int index)` returns a `KeyValuePair<string, JsonNode?>` by position; `SetAt(int index, string key, JsonNode? value)` replaces the property at that position with a new key-value pair; `Insert(int index, string key, JsonNode? value)` inserts at a position; `RemoveAt(int index)` removes by position. These index-based operations make `JsonObject` usable as an ordered property bag where schema-version-migration code must insert a new property at a specific canonical position — position 0 for a version discriminant, for example — without iterating and rebuilding the whole object. A modifier-based approach via `JsonTypeInfoResolver` is preferred for typed admission, but for pass-through envelope patching where the codec must forward an unknown envelope with one additional field injected, index-positioned mutation is the only correct tool.
- `JsonNode.ReplaceWith<T>(T value)` replaces the current node in its parent with a new value, where `T` can be any type that `JsonValue.Create<T>` accepts: primitives, `JsonNode` subtypes, or boxed values. The parent's reference is updated atomically in the node graph. This is the mutation primitive for in-place schema migration of a pass-through envelope where exactly one field value must be re-typed or re-keyed before forwarding, without reconstructing the full node tree.

[TYPEINFO_CREATEOBJECT_AS_FACTORY_INJECTION]:
- `JsonTypeInfo<T>.CreateObject` is a `Func<T>?` delegate invoked by the deserializer to construct instances before property population. Setting it via a modifier to a factory delegate that pools objects from a pre-allocated slab replaces heap allocation per deserialized message. Combined with `JsonObjectCreationHandling.Populate` on the type via `JsonTypeInfo.PreferredPropertyObjectCreationHandling`, properties are written into the recycled instance rather than a freshly constructed one. Pooled instances must be fully re-initialized before returning from the factory; the deserializer writes properties but does not zero fields from a prior use.
- `JsonTypeInfo.CreateObject` must not be `null` for types with `JsonTypeInfoKind.Object`; the deserializer throws if the kind is `Object`, the property-population path executes, and `CreateObject` is null. A modifier that replaces it with a pooling factory must check `typeInfo.Kind == JsonTypeInfoKind.Object` before assigning to avoid setting it on `None` or `Enumerable` kinds where it is irrelevant.

[SERIALIZEHANDLER_AS_FAST_PATH_EVIDENCE]:
- `JsonTypeInfo<T>.SerializeHandler` is a `SerializeHandler<T>?` delegate set by the source generator when `GenerationMode` includes `Serialization` (i.e., `Default` or `Serialization` modes). When non-null, the serializer uses this generated fast-path delegate instead of the reflection-based property-enumeration path. A modifier that inspects `typeInfo.SerializeHandler != null` confirms that a given type has fast-path write coverage; types with `null` `SerializeHandler` will use the reflection fallback path even inside a source-generated context. Auditing `SerializeHandler` coverage across all envelope types at initialization time is the compile-time contract check that the wire lane stays AoT-clean and allocation-free on the write path.

[ISONSERIALIZING_INTERFACE_BOUNDARY_INVARIANT]:
- `IJsonOnSerializing.OnSerializing()` is called on the instance immediately before property enumeration begins. Unlike the `JsonTypeInfo.OnSerializing` modifier delegate, it fires on the specific instance, giving access to `this`. Use it to assert outbound invariants: required correlation fields are non-null, the version discriminant has been set, and redaction-sensitive fields have been cleared or replaced with placeholder values. The assertion can set a flag that `IJsonOnSerialized.OnSerialized()` inspects to confirm the serialization completed from a valid state. Neither interface method can surface a typed failure — any exception thrown propagates through the serializer as a `JsonException` wrapper. Pre-validate before calling `JsonSerializer.Serialize` and invoke serialization only on already-validated objects.

[ENVELOPE_POLYMORPHISM_OPTIONS_LATE_BINDING]:
- `JsonTypeInfo.PolymorphismOptions` is mutable until `MakeReadOnly()` is called on the owning `JsonSerializerOptions`. A modifier that runs before seal can populate `PolymorphismOptions.DerivedTypes` from a probe receipt: the probe result supplies the list of discriminator strings the remote accepts, the modifier adds `JsonDerivedType` entries for each, and the resulting type info has a closed polymorphic contract matched to the peer's actual capability. Seal the options immediately after modifier-chain assembly to prevent concurrent serialization calls from racing on modifier application.
- `JsonTypeInfo.PolymorphismOptions` throws `InvalidOperationException` if the `Kind` is not `JsonTypeInfoKind.Object` or if the instance has been associated with a different `JsonTypeInfo`. Modifiers that set `PolymorphismOptions` must guard on `typeInfo.Kind == JsonTypeInfoKind.Object` before assigning.

[RECEIPT_PROPERTY_ORDER_AS_STREAMING_CONTRACT]:
- `JsonPropertyInfo.Order` controls the serialization order of properties within a type. For streaming wire protocols where consumers read properties positionally — such as length-prefixed framing where the length is always the first field — the property order must be enforced at the contract level. A modifier sets `Order` on each `JsonPropertyInfo` to match the protocol's canonical property sequence; the serializer writes properties in ascending `Order` value. This order is protocol-stable by contract and must be treated as a breaking change coordinate equal in gravity to property name changes. The default `Order` is 0; ties within the same order are resolved by the order `Properties` appear in the `JsonTypeInfo.Properties` list, which reflects declaration order in source-generated types.
