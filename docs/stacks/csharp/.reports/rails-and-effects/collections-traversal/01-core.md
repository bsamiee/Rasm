# Collections and Traversal

[COLLECTION_CHOICE]:
- `Seq<A>` is a cons-sequence backed by one of three concrete internal types — `SeqStrict<A>` (evaluated, array-backed with a `start` offset for O(1) Tail/Init without copying), `SeqLazy<A>` (wraps `IEnumerable<A>` and evaluates elements at most once, caching into the strict representation as traversal occurs), or `SeqEmptyInternal`. The correct internal type is selected at construction; calling `.Strict()` forces any lazy internal to fully evaluate and promote to `SeqStrict<A>`.
- `Seq<A>` is preferred over `Lst<A>` in all cases except positional insertion: `Lst<A>` is an AVL-tree finger structure that supports `Insert(int, A)` and `InsertRange(int, IEnumerable<A>)` at O(log n); `Seq<A>` has no O(log n) mid-sequence insert. Outside positional mutation, `Seq<A>` is faster because structural sharing via `start` offset avoids copying on `Cons` and `Tail`.
- `Arr<A>` is an immutable wrapper around a raw array: O(1) random read, O(n) modification because every change copies the entire backing array. Use when the access pattern is overwhelmingly read-intensive and the collection is short-lived or small; do not use when the sequence grows incrementally.
- `HashMap<K, V>` is backed by a CHAMP trie (Compressed Hash Array Map Trie, per Steindorfer's thesis). Operations are O(1) amortized but unordered. `Map<K, V>` is backed by an AVL tree: O(log n) operations, iteration is key-ordered. The choice between them is purely access-pattern driven: use `Map<K, V>` when ordered iteration or range queries (`FindRange(keyFrom, keyTo)`) are needed, `HashMap<K, V>` otherwise.
- `HashSet<A>` and `Set<A>` mirror the same CHAMP vs AVL distinction. `Set<A>` supports ordered min/max and range operations; `HashSet<A>` does not guarantee iteration order.
- `SeqEmpty` is a value that implicitly converts to every collection type (`Seq<A>`, `Arr<A>`, `Lst<A>`, `HashMap<K,V>`, `HashSet<A>`, `Map<K,V>`, `Que<A>`, `Stck<A>`, etc.) via `op_Implicit(SeqEmpty)`. Use `Seq<A>.Empty` or the `SeqEmpty` sentinel at call sites that need a universal empty; do not construct any collection type explicitly just to pass an empty.
- `Iterable<A>` is a lightweight `IEnumerable<A>` wrapper that implements the full trait hierarchy (`Monad`, `Traversable`, `MonoidK`, etc.) to make HKT combinators apply to arbitrary lazy enumerables. It is not a substitute for `Seq<A>` where evaluation count matters; it is the bridge when a caller-owned `IEnumerable<A>` must enter a trait-polymorphic context. Construction from a `ReadOnlySpan<A>` is available via `Iterable<A>.FromSpan(span)`.
- `BiMap<A, B>` is a bidirectional map indexed on both `A` and `B` for O(log n) lookup in either direction. It is the canonical choice when a symmetric key–value bijection is the domain constraint, not a pair of `HashMap` instances.

[STRICTNESS_AND_LIFETIME]:
- `Seq<A>.Add(A)` appends to the end but forces complete evaluation of any lazy internal: the remark on the API states "Forces evaluation of the entire lazy sequence so the item can be appended." Operations that mutate the logical end of a lazy `Seq<A>` pay the full evaluation cost; prepend (`Cons`) does not.
- `SeqLoan<A>` wraps an `ArrayPool<T>`-rented array. It implements `IDisposable` with a finalizer safety net that returns the array to its originating pool. Any transformation (Map, Filter, Bind, etc.) on a `SeqLoan<A>` immediately clones the rented backing array into an owned `Seq<A>` and proceeds from there — the loan semantics are visible only during initial read-through. Calling `Dispose()` while another thread is reading the structure produces undefined behavior.
- `SeqLoan<A>.ToSeq()` creates a safe owned copy. This is the deliberate escape hatch: consume the loan in one pass, then promote to an owned `Seq<A>` and dispose the loan. Holding a `SeqLoan<A>` beyond the scope of the originating `ArrayPool` rent is a use-after-return bug that the finalizer mitigates but does not prevent.
- `SeqStrict<A>` has a `consDisallowed` flag. Once set, further `Cons` calls on that strict representation are blocked (the sequence allocates a new backing block instead of mutating the existing one). This flag protects structural sharing: a `Tail` that returns a sub-slice of the same array would be corrupted if the original sequence later prepended into that array.
- `SeqLazy<A>` similarly has `consDisallowed`. Once evaluation begins emitting elements into the internal cache array, further `Cons` calls cannot prepend into that cache without corrupting in-flight iteration; the flag forces a new allocation.

[TRAVERSE_AND_SEQUENCE]:
- `TraverseM<M, B>(Func<A, K<M, B>>)` on `Seq<A>`, `Arr<A>`, `Lst<A>`, `HashSet<A>`, `Set<A>` maps into a monad `M` and collects results by sequencing via monadic bind (left-to-right). This is the correct form when the monad has nontrivial short-circuit behavior (`Fin<A>`, `Eff<A>`, `Option<A>`): on the first `None` or `Fail`, the rest of the sequence is not evaluated.
- `Traverse<F, B>(Func<A, K<F, B>>)` maps into an applicative functor `F` and collects by applicative sequencing. For `Validation<FAIL, A>`, this accumulates all errors before returning; `TraverseM` with `Validation` in monad mode short-circuits on the first error. The choice between them is not stylistic: use `Traverse` when all elements must be validated, `TraverseM` when the first failure should abort.
- `Traversable.sequence<T, F, A>(K<T, K<F, A>>)` is the no-mapping form: when elements are already wrapped values, `sequence` swaps the container and functor layers without a prior `Map`. `sequenceM<T, M, A>` is the monadic variant with the same semantics as `TraverseM` but without the mapping step.
- The `.As()` call pattern is required after `TraverseM` or `Traverse` on any collection when the return type is `K<F, Seq<B>>` — the trait system returns the abstract HKT kind and `.As()` recovers the concrete type. Omitting `.As()` leaves the result in the abstract `K<F, Seq<B>>` which cannot be used directly. The extension is on the concrete HKT type (e.g., `EffExtensions.As<A>` for `Eff`).
- `Iterable<A>.TraverseIO<M, B>(Func<A, K<M, B>>, K<Iterable, A>)` is the async-aware traversal overload unique to `Iterable<A>`. When the iterable's `IsAsync` property is true (the iterable wraps an async source), this overload correctly awaits each step. `Seq<A>` has no `TraverseIO` because it is always synchronous-evaluated.

[FILTER_MAP_VIA_CHOOSE]:
- `Choose<A, B>(Func<A, Option<B>>)` on `Seq<A>`, `Arr<A>`, `Lst<A>`, `Stck<A>`, and `Map<K, V>` applies a function that returns `Option<B>`, discards all `None` results, and returns the `Some` values. This is atomic filter-map in one pass; it replaces `Filter(predicate).Map(projection)` which makes two passes and requires the projection to repeat the same guard. `Map<K,V>.Choose` and its two-argument form `Choose(Func<K, V, Option<R>>)` carry the key through.
- `Seq.choose<A, B>(Seq<A>, Func<A, Option<B>>)` is the static-module form; `SeqExtensions.Choose<A, B>(Seq<A>, Func<A, Option<B>>)` is the extension form. Both exist; prefer the extension form at call sites that chain.
- `SeqExtensions.Choose<A, B>(Seq<A>, Func<int, A, Option<B>>)` is the indexed form: the position of each element is passed alongside the value. Use this for index-aware filter-map without a prior `zip`-with-index step.
- `MonoidK.choose<F, A, B>(K<F, A>, Func<A, Option<B>>)` is the HKT-polymorphic form: for any `MonoidK` instance `F`, it propagates the structure only for elements where the mapping returns `Some`. This is how `Choose` generalizes beyond concrete collection types into the trait hierarchy.

[IMMUTABLE_FOLDS]:
- `Fold<S>(S, Func<S, A, S>)` on `Seq<A>` and `SeqLoan<A>` is left-associative: it reduces from head to tail. This matches the natural order of `Seq<A>` enumeration and is stack-safe for any length. `FoldBack<S>(S, Func<S, A, S>)` on `SeqLoan<A>` uses recursive processing (`FoldBackRec`) for strict sequences shorter than 5000 items because recursion is faster for small sizes; it falls back to an iterative CPS approach for longer sequences or lazy inputs. This threshold is documented in the XML remarks and is an implementation detail that can change; treat any fold that builds a right-to-left accumulation on large lazy sequences as a candidate to rewrite as `Rev().Fold`.
- `FoldWhile<S>(Func<A, Func<S, S>>, Func<(S, A), bool>, S, K<T, A>)` from the `Foldable<T>` trait takes a curried folding function and a predicate over the `(state, element)` pair and stops as soon as the predicate returns false. This is the principled early-exit fold; the predicate receives the current state and the element that would next be processed, so it can inspect accumulated state to decide when to stop.
- `FoldUntil` is the complement: it continues while the predicate is false and stops on the first true. Both have monadic variants `FoldWhileM` and `FoldUntilM` that take a `Func<A, Func<S, K<M, S>>>` and thread a monad through the fold.
- `FoldBackMaybe<A, S>(Func<A, Func<S, Option<S>>>, S, K<T, A>)` stops as soon as the folding function returns `None`. This is the canonical formulation for a fold that can decide "no more results are possible" based on the element alone, without needing to inspect accumulated state.
- `scan<S, A>(Seq<A>, S, Func<S, A, S>)` and `scanBack<S, A>(Seq<A>, S, Func<S, A, S>)` both return a `Seq<S>` of all intermediate states including the initial state. `scan` runs left-to-right; `scanBack` runs right-to-left. Use scan when downstream requires all running totals, not just the terminal value.

[STRUCTURAL_SHARING_AND_PERFORMANCE]:
- `Seq<A>.Cons(A)` prepends without copying when `consDisallowed` is false: if the backing `SeqStrict<A>` array has unused capacity before `start`, the new head is written there and `start` is decremented. This means prepend is O(1) amortized and sharing is preserved for the tail. Once capacity is exhausted or `consDisallowed` is set, a new array is allocated.
- `Seq<A>.Tail` and `Seq<A>.Init` are O(1) for `SeqStrict<A>` because they adjust `start` and `count` over the same backing array rather than copying. Materializing `Head` and `Tail` in a recursive descent over a `SeqStrict<A>` is therefore safe to use structurally; it does not degrade like array copying would.
- `Map<K, V>` and `Set<A>` (AVL trees) support `Skip(int)` — returning a new tree without the first N elements — in O(n) time. The operation walks the tree, it does not slice; there is no O(log n) structural prefix sharing.
- `Seq<A>.Case` returns an `object` shaped as `null` (empty), `A` (singleton), or `(A, Seq<A>)` (head–tail) for pattern-matching. `Map<K, V>.Case` and `Arr<A>.Case` follow the same pattern: `null` for empty, `(K, V)` for singleton, `((K, V), Seq<(K, V)>)` for more. This is how structural decomposition works across all collection types — directly into a switch expression without `.IsEmpty` guards.

[INDEXED_PROJECTION]:
- `Seq<A>.Map<B>(Func<A, int, B>)` is the indexed map overload; the remark states "Exposes an index for each map." This eliminates the `zip`-with-index pattern entirely when projection requires positional information.
- `Seq.generate<A>(int, Func<int, A>)` produces a strict `Seq<A>` of the given length by calling the factory with each index. It is the canonical way to build index-initialized sequences without a mutable accumulator.
- `Seq<A>.Item[System.Index]` throws `IndexOutOfRangeException`; there is no safe-indexer on `Seq<A>` itself. `SeqStrict<A>.At(int)` is the `Option<A>`-returning safe accessor exposed on the internal type. Access via `.HeadOrNone()` and structural `Tail` decomposition is the safe access idiom for the public API.

[HASHMAP_ALGEBRA]:
- `HashMap<K, V>.Union(IEnumerable<(K, V)>, WhenMatched<K, V, V, V>)` merges two maps, calling `WhenMatched` for every key present in both. `WhenMatched` is a delegate `(key, leftValue, rightValue) => result`. The right-bias default (no `WhenMatched` argument) keeps the right-hand value on collision.
- `HashMap<K, V>.Union<V2>(IEnumerable<(K, V2)>, WhenMissing<K, V2, V>, WhenMatched<K, V, V2, V>)` is the heterogeneous merge: left-only keys pass through `WhenMissing MapRight`, right-only keys pass through `WhenMissing MapLeft`, and matched keys produce the result type through `WhenMatched`. The three-argument overload `Union<R>(…, WhenMissing<K,V,R>, WhenMissing<K,V2,R>, WhenMatched<K,V,V2,R>)` produces a completely different output type `R`.
- `HashMap<K, V>.Intersect(IEnumerable<(K, V)>, WhenMatched<K, V, V, V>)` retains only keys present in both and applies the merge function. `Except(IEnumerable<K>)` removes all keys present in the right-hand set; `Except(IEnumerable<(K, V)>)` is key-only difference (values on the right are ignored).
- `AtomHashMap<K, V>.SwapKey(K, Func<Option<V>, Option<V>>)` is a single atomic CAS over one entry: `None` input means the key was absent, `Some(x)` means it was present. The return value from the delegate determines the final state: `Some(y)` → add or update, `None` → remove. This replaces a read–modify–write sequence with one atomic operation and is the correct primitive for conditional update-or-insert.
- `AtomHashMap<K, V>.MapInPlace(Func<V, V>)` mutates all values atomically; the remark warns the function may be called multiple times under thread contention. Keep this function fast, pure, and free of side effects.
- `TrackingHashMap<K, V>` wraps `HashMap<K, V>` and records a change morphism since the last `Snapshot()` call. The `Changes` property returns a `HashMap<K, Change<V>>` where `Change<V>` is one of `EntryAdded`, `EntryMapped`, or `EntryRemoved`. Crucially the change map stores at most one morphism per key, not a log — it represents the net effect of all mutations since the last snapshot. `AtomHashMap<K,V>.ToTrackingHashMap()` bridges between the atomic mutable form and the change-tracking snapshot model.

[ATOMIC_COLLECTION_STATE]:
- `AtomSeq<A>` stores the active sequence in a field typed `ISeqInternal<A>` (the interface shared by all three internal variants). `Swap(Func<Seq<A>, Seq<A>>)` atomically replaces the internal with CAS retries; `SwapIO(Func<Seq<A>, Seq<A>>)` does the same but returns an `IO<Seq<A>>` capturing the resulting value. `SwapInternal(Func<ISeqInternal<A>, ISeqInternal<A>>)` exposes the low-level form.
- `AtomSeq<A>.MapInPlace(Func<A, A>)` maps all elements atomically. Like `AtomHashMap<K,V>.MapInPlace`, the mapping function may execute multiple times under contention.
- `AtomHashMap<K, V>` and `AtomSeq<A>` do not expose `Subscribe` or reactive streams. State is read by reading the current value (`.Value` or the property accessors); change notification is not a contract of these types.

[TRAVERSAL_DECOMPOSITION]:
- `Seq.tails<A>(Seq<A>)` returns all final segments from longest to shortest: for `[a, b, c]` the result is `[[a,b,c], [b,c], [c], []]`. `tailsr<A>` is the recursive implementation of the same operation, faster for small sequences. Use `tails` for large or lazy inputs.
- `Seq.span<A>(Seq<A>, Func<A, bool>)` splits at the first element where the predicate is false, returning `(prefix, suffix)` as a tuple. The prefix is the maximal initial segment satisfying the predicate. This is a single-pass structural split with no intermediate allocation beyond the two `Seq<A>` slices.
- `Patch<EqA, A>` is an edit-list groupoid over ordered sequences. `Patch.diff(lst1, lst2)` computes the minimal patch between two sequences; `Patch.apply(patch, seq)` applies it. Patches compose (`append`) and invert (`inverse`). `applicable(patch, document)` and `composable(p, q)` are predicates that confirm whether a patch can be validly applied or composed before doing so. This is the correct primitive for versioned or diffable sequence state — it replaces ad-hoc before/after comparison.
