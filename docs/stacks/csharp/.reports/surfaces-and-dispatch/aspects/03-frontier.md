# Aspect Architecture — Frontier

[EXTENSION_BLOCKS_AS_HKT_OPERATOR_SURFACES]:
- C# 14 extension blocks (`IDS_FeatureExtensions`, language version 1400) are the mechanism that makes constrained operators on higher-kinded types compilable. The syntax `extension<E, F, A>(K<F, A> _) where F : Fallible<E, F>` declares an extension over `K<F, A>` with generic constraints, enabling `operator |` to be defined directly on the carrier type with trait constraints — impossible with C# 13 extension methods. Decompiled from the installed LanguageExt 5 assembly, `FallibleExtensions` uses exactly this form to define the `|` operator that composes `K<F, A>` with `CatchM<E, F, A>`, and `FinalExtensions` uses it for `K<F, A>` with `Finally<F, X>`. The `$T0`, `$T1` placeholders in the decompiled output are the representation of extension block type parameters in the current ilspycmd decompilation of C# 14.
- `MonadUnliftIOExtensions` defines the entire retry/repeat/bracket/timeout surface inside a single extension block `extension<M, A>(K<M, A> ma) where M : MonadUnliftIO<M>`, exposing `LocalIO()`, `UninterruptibleIO()`, `PostIO()`, `TimeoutIO(TimeSpan)`, `BracketIO()`, `BracketIO<B,C>(Use, Fin)`, `BracketIO<B,C>(Use, Catch, Fin)`, `RepeatIO`, `RepeatWhileIO`, `RepeatUntilIO`, `RetryIO`, `RetryWhileIO`, and `RetryUntilIO` as receiver-syntax methods on any `K<M, A>` where `M` satisfies `MonadUnliftIO`. The FoldIO family (`FoldIO`, `FoldWhileIO`, `FoldUntilIO`) remains as classical static extension methods with explicit `this K<M, A>` parameters because their additional generic `S` and predicate overloads prevent uniform representation inside one extension block. This split is structural, not arbitrary.
- The extension block form is load-bearing for operator definition: you cannot define a constrained `operator |` on a generic type via classical `this` extension methods because C# requires operator parameters to include the containing type. Extension blocks lift this restriction by making the block's receiver type the containing type at the constraint level. Any domain-level operator aspect that must be constrained to a trait (e.g., a `|` that only applies when the carrier is `Fallible`) requires an extension block in C# 14.

[COMPOSITION_TIME_POLICY_FRONTIER]:
- `catchOf<E, M, A>` requires `where E : Error, M : Fallible<Error, M>, Monad<M>` — it handles the **first** occurrence of an `E`-subtype error by calling `Fail((E)e)`. `catchOfFold<E, M, A>` additionally requires `MonoidK<M>` (a monad with an empty element and a combine operation) and accumulates **all** `E`-subtype errors in the monad via `FoldM`: `es.Filter<E>().FoldM((Error e) => Fail((E)e))`. The distinction: `catchOf` handles the single root error (appropriate for `IO`/`Eff`), `catchOfFold` aggregates multiple errors into a monoidal context (appropriate for `Validation`). Using `catchOf` on a `Validation` monad discards accumulated errors past the first; using `catchOfFold` on `IO` does not compile.
- `IO<A>.Fold(schedule, initialState, folder)` re-invokes the computation on success according to the schedule's steps, accumulating a state value `S` via `folder(S, A) -> S`. `FoldWhile(schedule, state, folder, stateIs: Func<S, bool>)` continues while the state predicate returns `true`; `FoldUntil` stops when the predicate returns `true`. Both accept three predicate shapes — `Func<S, bool>` (state), `Func<A, bool>` (value), and `Func<(S State, A Value), bool>` (combined) — giving the termination criterion access to both the accumulated state and the last produced value. `IO.Fold` without a schedule defaults to `Schedule.Forever`, producing an infinite accumulation loop that runs until an exception.
- `IO<A>.WithEnv(Func<EnvIO, EnvIO> f)` and `IO<A>.WithEnvFail(Func<EnvIO, EnvIO> f)` are the protected virtual methods underlying all isolation aspects. `Bracket()` calls `WithEnv(_ => env.LocalResources)` — on success and failure, the `Resources` child scope is disposed. `BracketFail()` calls `WithEnvFail(_ => env.LocalResources)` — disposal occurs only on failure, allowing a successfully acquired resource to escape as the computation's result. This is the mechanism behind the `BracketFail` vs `Bracket` distinction: the local resource scope is created identically, but the disposal trigger is different.
- `IO<A>.Post()` is the explicit `SynchronizationContext` marshaling aspect. Its implementation captures `env.SyncContext` at execution time — the `SyncContext` instance stored in `EnvIO` when the computation was created — and marshals the continuation via `SyncContext.Post(...)`. If `env.SyncContext` is `null` (no captured context), `Post()` is a no-op that runs inline. The key architectural implication: `Post()` is not "post to current sync context" — it posts to the context captured at the **start** of the `IO` chain (the one embedded in `EnvIO`). A `Post()` call mid-chain routes back to whatever thread context started the chain, not the current dispatcher.
- `IO<A>.Uninterruptible()` wraps the computation in `IOUninterruptible`, which executes against a fresh `EnvIO` with a new `CancellationTokenSource` that is never linked to the outer token. The outer cancellation token's cancellation does not propagate into the wrapped computation. This is an aspect that removes one composition-time concern (cancellation) from a specific sub-computation, and it is orthogonal to the error rail — `Uninterruptible` does not affect `Fin`/`Error` propagation, only the `CancellationToken` path.
- `IO<A>.Fork(Option<TimeSpan> timeout)` returns `IO<ForkIO<A>>`, where `ForkIO<A>` is `readonly record struct(IO<Unit> Cancel, IO<A> Await)`. Resources acquired inside a forked computation are tracked in an independent child `Resources` scope — they are released when the forked computation completes (successfully or otherwise). Resources acquired in the parent before the fork are visible inside the fork as shared resources without automatic release on fork completion. This is the documented resource-sharing asymmetry: parent-to-fork sharing is explicit and unguarded, fork-to-parent is not possible.

[SCHEDULE_ALGEBRA_FRONTIER]:
- `Schedule` implements `Semigroup<Schedule>` with `+` as `Combine` (sequential append: all steps of `a`, then all steps of `b`). The `|` operator between two `Schedule` instances is `Union` (min delay per step, length of the longer). The `&` operator is `Intersect` (max delay per step, length of the shorter). These are three algebraically distinct binary operations on the same type — `+`/Semigroup is sequential composition, `|`/Union is parallel min, `&`/Intersect is parallel max. A composition that conflates `|` with `+` changes the semantics entirely: `exponential(50*ms) | recurs(5)` terminates after 5 steps at exponential delays, while `exponential(50*ms) + recurs(5)` runs the exponential forever and then appends 5 zero-delay steps.
- `Schedule.Bind(Func<Duration, Schedule> f)` and `Schedule.SelectMany` make `Schedule` a monad in LINQ query expressions. The `intersperse(schedule)` static method is implemented entirely as `Transform(s => s.Bind(schedule.Prepend))`: for each duration `d` in `s`, `f(d)` returns a new schedule that is `schedule` with `d` prepended. The bind flattens this, producing `d1, (schedule steps), d2, (schedule steps), ...`. This is categorically different from `Interleave(b)` which produces `a1, b1, a2, b2, ...` — `intersperse` inserts the entire interspersal schedule between each step of the base, while `Interleave` alternates individual steps.
- `ScheduleTransformer` has an implicit conversion to `Schedule` via `Schedule.Forever | t`. This means any `ScheduleTransformer` used where a `Schedule` is expected produces an infinite-base schedule with the transformer applied to it. A `ScheduleTransformer` from `recurs(5)` converts to a 5-step schedule; a `ScheduleTransformer` from `maxDelay(1*second)` converts to an infinite schedule with delay capped per step. The `+` operator on two `ScheduleTransformer` values composes them left-to-right (`g.Apply(f.Apply(x))`) — this is transformer function composition, not schedule concatenation.
- `Schedule.upto(max, currentTimeFn)` generates steps until a wall-clock duration has elapsed from the moment `Run()` is first called. The optional `currentTimeFn` (defaulting to `() => DateTime.Now`) is the clock injection point, making `upto`-based schedules fully testable without real time. The distinction from `maxCumulativeDelay(max)`: `maxCumulativeDelay` terminates when the sum of emitted `Duration` values exceeds `max`; `upto` terminates when elapsed wall time exceeds `max` — in computations where actions take variable time, these diverge significantly.
- `Schedule.NoDelayOnFirst` is a static `ScheduleTransformer` that replaces the first step with `Duration.Zero` regardless of what the base schedule emits. Its implementation is `Transform(s => s.Tail.Prepend(Duration.Zero))`. The practical use: wrapping any retry schedule in `NoDelayOnFirst | schedule` ensures the first retry is immediate even if the underlying schedule starts with a non-zero delay (e.g., `exponential(100*ms)` would otherwise delay the first retry by 100ms). The transformer is applied to the schedule, not the computation — the delay behavior changes, not the retry logic.

[DEFINITION_TIME_ASPECT_FLAGS]:
- `UnsafeConversionToKeyMemberType` is an axis on `[ValueObject<T>]` independent of `ConversionToKeyMemberType`. The documentation defines "unsafe" as a conversion that can throw an exception — converting a value object back to its key type when the key type allows values that the value object's validation rejects. Default is `ConversionOperatorsGeneration.Explicit`. `ConversionToKeyMemberType` controls the safe (non-throwing) direction; `UnsafeConversionToKeyMemberType` controls the explicit-cast direction when a throw is possible. Both axes are required to fully specify the conversion surface, and setting one to `None` does not suppress the other.
- `SkipEqualityComparison = true` on `[ValueObject<T>]`, `[ComplexValueObject]`, or `[Union]` suppresses the generation of `IEquatable<T>`, `Equals`, `GetHashCode`, and any equality or comparison operators simultaneously. When set, `EqualityComparisonOperators` and arithmetic comparison operators are also forced to `OperatorsGeneration.None` — equality is a prerequisite for comparison operators in generated code, and the flag enforces the full suppression cascade. The use case is types that must delegate equality to a custom implementation or an identity registry where value-based equality is incorrect.
- `AllowDefaultStructs = true` on struct value objects suppresses the generated guard that throws when `default(T)` or the parameterless constructor is used. The guard is a generated `[ModuleInitializer]`-adjacent mechanism: the struct's generated code intercepts default-construction via the parameterless constructor and throws unless `AllowDefaultStructs` is set. The flag does not affect class value objects — classes use null as the default, which is already an inadmissible value by the boundary doctrine.
- `DefaultInstancePropertyName` overrides the generated static property name for the `default` instance of a struct value object from "Empty" to any valid identifier. This property exposes `default(T)` as a named sentinel — `Guid.Empty` is the BCL model. The generated property exists only when `AllowDefaultStructs = true` (structurally, it only makes sense when the default is admitted); the name override lets the domain vocabulary align with the sentinel's semantics (e.g., `None`, `Zero`, `Root`, `Missing`).
- `SerializationFrameworks` on `[ValueObject<T>]` and `[SmartEnum<TKey>]` is a `[Flags]` enum (`SystemTextJson = 1, NewtonsoftJson = 2, MessagePack = 4`). Setting it to `None = 0` suppresses all serialization aspects without suppressing the factory methods — unlike `SkipFactoryMethods = true`, which suppresses serialization as a side effect. The `[Flags]` nature enables selective suppression: `SerializationFrameworks.Json` (value 3) generates STJ and Newtonsoft converters but not MessagePack formatters. Companion packages for each framework must be present for the generator to actually emit the corresponding converter type; absent companion packages cause the generator to skip that aspect silently.
- `FactoryMethodGeneration.Always` on `[Union<T1,T2>]` forces factory method generation for every member regardless of whether any auto-detect trigger is present (duplicate types, interface arms, `object` arms, type-parameter arms). The use case is an ad-hoc union where all member types are distinct concrete types — the auto-detect (`Default`) would produce no factory methods, but uniform factory-method style is desired for API consistency. The generated methods follow the `FromT1(T1)`, `FromT2(T2)` naming pattern and produce implicit-conversion-equivalent construction with an explicit method name.

[INTERCEPTOR_REVERSE_QUERY]:
- C# 14 SDK adds `CSharpExtensions.GetInterceptorMethod(SemanticModel?, InvocationExpressionSyntax, CancellationToken)` returning `IMethodSymbol?`. This is the reverse-query API: given an invocation that has been intercepted, it returns the interceptor method symbol rather than the intercepted method. Prior to C# 14, verifying that an interceptor was correctly wired required manual inspection of the generated source. With `GetInterceptorMethod`, an analyzer or code generator can confirm at semantic analysis time that its own generated interceptor has resolved correctly for a target call site, closing the generation-verification loop within a single compilation pass.
- The `InterceptsLocationAttribute` is not in `System.Runtime`; it must be defined by the source generator in its own generated output, typically as `namespace System.Runtime.CompilerServices { [AttributeUsage(...)] sealed class InterceptsLocationAttribute(int version, string data) : Attribute {} }`. The version integer is the `InterceptableLocation.Version` property value. Any version mismatch between the attribute declaration and the Roslyn compiler's expected encoding version produces a compile error, not a silent failure — the location data format is versioned at the Roslyn level and the attribute signature must match. The consequence for generators: the version value must be taken from `GetInterceptableLocation`'s result at generation time, not hardcoded.
