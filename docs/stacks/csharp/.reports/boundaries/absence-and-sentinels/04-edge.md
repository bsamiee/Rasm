# Absence and Sentinels — Edge Cases and Structural Traps

[OPTIONТ_NONE_IS_NULL]:
- `OptionT<M, A>` is a record (reference type). Its `static readonly OptionT<M, A> None` field has no static initializer visible in IL — it is zero-initialized to `null`. Any instance operation on `OptionT<M, A>.None` — `.Match(...)`, `.Bind(...)`, `.Run()`, `.IfNone(...)` — NullReferenceExceptions at the call because `runOption` is null and every operation calls `M.Map(..., runOption)` or `M.Bind(runOption, ...)`. The factory `OptionT.None<M, A>()` delegates to the same null field; it is equally unsafe.
- The only safe path to construct a known-absent `OptionT<M, A>` is `OptionT.lift<M, A>(Option<A>.None)`, which evaluates `new OptionT<M, A>(M.Pure(Option<A>.None))` — verified: `OptionT.lift<M,A>(Option<A> monad) => new OptionT<M, A>(M.Pure(monad))`. The implicit `operator OptionT<M, A>(in Option<A> ma)` calls this same factory. Any seam that constructs a None transformer value must use one of these two paths; the static `None` field is a structural trap.

[OPTIONТ_VS_FINT_FALLIBLE_ASYMMETRY]:
- `OptionT<M>` implements `Fallible<Unit, OptionT<M>>`. Its `Catch` predicate receives `Unit`, not `Error`. The `@catch` infrastructure with typed error codes (`error.HasCode(...)`, `error.IsType<E>()`) does not apply to `OptionT` — absence in the transformer carries no diagnostic code. A seam that needs typed recovery (e.g., distinguish timeout-absence from resource-unavailability-absence) cannot use `OptionT<M>`; it must use `FinT<M>` which implements `Fallible<Error, FinT<M>>` with full typed predicates.
- `Natural<FinT<M>, OptionT<M>>` exists — verified on `FinT<M>`'s class declaration. This transform is lossy: a `FinT<M, A>` in `Fail(error)` state becomes `OptionT<M, A>` with `None`, silently discarding the typed `Error`. The inverse `Natural<OptionT<M>, FinT<M>>` does not exist; the absence-to-failure conversion requires an explicit `.ToFin(Error.New(code, msg))` call at the materialization point.

[TRANSFORMER_BIND_OUTER_MONAD_LIFETIME]:
- `OptionT<M, A>.Bind<B>` propagates `None` through the outer monad via `M.Pure(Option<B>.None)` — the outer monad computation succeeds and the `None` state is carried as a value inside `M`. `FinT<M, A>.Bind<B>` propagates `Fail(error)` similarly through `M.Pure(Fin<B>.Fail(error))`. In both transformers, the outer monad `M` is never short-circuited by the inner absence or failure state. The outer `IO` monad can still independently fail (from `IO.fail(e)` or an uncaught exception inside the body) — the inner `Option.None` or `Fin.Fail` state and an outer `IO` failure are orthogonal conditions that both require handling at the `Run()` extract point.
- The extract point is `optionT.Run()` which returns `K<M, Option<A>>`, or `finT.Run()` which returns `K<M, Fin<A>>`. The caller must handle the outer monad failure AND the inner absence/failure independently. A seam that calls `.Run().Match(...)` on the raw `K<M, Option<A>>` inside a `.Bind(...)` on `IO` without checking the outer `IO` result first silently swallows any outer-monad failure that occurred before `Run()`.

[FIN_OPERATOR_ERROR_SEMANTICS]:
- The `|` operator on `Fin<A>` calls `Choose(rhs)` — verified: `static K<Fin, A> Choice<Fin>.Choose<A>(K<Fin, A> ma, K<Fin, A> mb)` returns `mb` when `ma` is `Fail`. The lhs error is silently discarded. `SemigroupK<Fin>.Combine(ma, mb)` accumulates: when both are `Fail`, it returns `Fin.Fail(err1 + err2)` where `+` produces a `ManyErrors` containing both. `Applicative<Fin>.Apply` also accumulates: `Fail(mf) + Fail(ma)` = `Fail(err_f + err_a)`.
- A fallback chain expressed as `primary | secondary | terminal` discards all intermediate errors if the first failing attempt had a diagnostic code needed downstream. The correct surface for fallback-with-error-retention is `SemigroupK.Combine` (accessible via the `+` operator inherited from `SemigroupK`) which builds a `ManyErrors` that survives `HasCode` queries on the combined result. Use `|` only when lhs error is genuinely irrelevant to callers; use `Combine`/`+` when a logging or health seam must inspect why each attempt failed.

[MANYERRORS_RECURSIVE_DISCRIMINATION]:
- `ManyErrors.HasCode(int code)` evaluates `Errors.Exists(e => e.Code == code)` — verified. `ManyErrors.IsType<E>()` evaluates `Errors.Exists(e => e.IsType<E>())`. `ManyErrors.Is(Error error)` evaluates `Errors.Exists(e => e.Is(error))`. All three predicates recurse into contained errors, so a terminal code is detectable even inside a combined `ManyErrors` produced by `Fin.Applicative.Apply` or `SemigroupK.Combine`. A `Catch` predicate at a recovery site that tests `e.HasCode(myTerminalCode)` correctly detects the terminal code even when it has been combined with transient errors.
- `ManyErrors.Filter<E>()` returns a new `ManyErrors` (or `Errors.None`) containing only errors where `IsType<E>()` holds — verified: it projects `Errors.Choose(e => e.IsType<E>() ? Some(e.Filter<E>()) : None)`. This is the correct surface for partial recovery: extract only the terminal typed errors from a combined failure, emit them to the health seam, and separately attempt recovery on the remaining transient errors. A seam that discards the combined error and re-creates its own typed receipt loses the accumulated evidence.

[ERROR_INNER_CHAIN_AT_SEAM]:
- `Error.Inner` is `Option<Error>` — verified on the abstract base class: `public virtual Option<Error> Inner => Prelude.None`. `Expected` records declare an `Inner` primary-ctor parameter of type `Option<Error>` — verified on `Expected(string Message, int Code, Option<Error> Inner = default)`. `Exceptional` derives its `Inner` from the wrapped `Exception.InnerException` via `Optional(Value?.InnerException) |> Error.New(...)`. A seam that captures a chained exception should preserve the inner chain by constructing `Error.New(code, message, inner: Error.New(innerException))` so that `Inner` carries the nested cause as a typed `Option<Error>`, not as a string field. Recovering callers can then traverse the chain via `error.Inner.Bind(i => i.Inner)` without parsing text.

[IO_LIFT_FIN_EXCEPTION_SURFACE]:
- `IO.lift<A>(Func<Fin<A>> f)` internally calls `f().ThrowIfFail()` — verified. `Fin<A>.ThrowIfFail()` calls `FailValue.Throw()`, which dispatches to `Error.Throw<A>()`. On `Expected`, `Throw<R>()` throws `WrappedErrorExpectedException` (an `ErrorException` subclass carrying the code and message). On `Exceptional`, it re-throws the original exception if its `Value` field is non-null, or throws `ExceptionalException`. The `IO` interpreter catches the thrown exception and re-wraps it into an `IO` failure. The round-trip is lossless for `Expected` because `ErrorException` carries the original `Expected` record. It is potentially lossy for `Exceptional` if the original `Exception` object has been garbage-collected (serialization deserialization path sets `Value` to null). A seam that receives a `Fin<A>` from a foreign source and lifts it via `IO.lift(fin)` must verify that exceptional errors carry live `Exception` objects before the lift if stack trace preservation is a boundary requirement.

[ENVIO_CANCEL_PROPAGATION_ASYMMETRY]:
- `EnvIO.Local` (used by `IO.Local()`) calls `New(null, Token, null, SyncContext.Current)` — creates a fresh `CancellationTokenSource` and, when `Token.CanBeCanceled`, registers `Token.Register(source.Cancel)`. The registration is stored in `Registration` and disposed when the `EnvIO` is disposed. Cancellation propagates from parent to child via this side-channel registration, which fires synchronously on `Cancel()` and asynchronously on `CancelAsync()` per the `CancellationTokenSource.Cancel()` vs `CancelAsync()` divergence already established.
- `EnvIO.LocalResources` (used by `BracketFail`) calls `New(null, Token, Source, SyncContext)` — passes the same `Source` as the outer `EnvIO`. No new source is created and no registration is needed: the child directly shares the parent source. `Cancel()` on the parent `Source` fires immediately inside the child scope. The asymmetry: `LocalResources` shares the source lifetime, so disposing the child `EnvIO.DisposeNonResources()` does NOT dispose the shared source (the `Own & 1` bit is not set). `Local` owns its source (`Own & 1 == 1`) and disposes it on `DisposeNonResources()`. A timed-out `BracketFail` probe that is scoped via `LocalCancelWithTimeout` creates a new source (timeout variant passes `null` source to trigger creation) — the timeout source IS owned and disposed by the child.

[CANCELLATION_REGISTRATION_ALREADY_CANCELLED_REENTRANCY]:
- `CancellationToken.UnsafeRegister(Action<object?> callback, object? state)` verified source: delegates to `_source?.Register(callback, state, useSynchronizationContext: false, useExecutionContext: false)`. When `_source` is null (token is `CancellationToken.None`), returns `default(CancellationTokenRegistration)` — the callback is never registered. When `IsCancellationRequested == true` at registration time, `Register` invokes the callback synchronously on the calling thread before returning (because the token is already cancelled and the source handles this in `Register`). This is a reentrancy hazard at seams that perform `Atom.SwapMaybe` inside the callback: if `SwapMaybe`'s spin-loop body calls back into code that itself calls `Register` on the same already-cancelled token, the inline synchronous invocation will re-enter the swap body on the same thread. The guard is to check `IsCancellationRequested` before calling `Register`/`UnsafeRegister` and handle the already-cancelled state directly, bypassing registration.

[OPTION_TRAVERSE_EFFECT_ORDERING_AT_SEAM]:
- `Option<A>.TraverseM<M, B>(Func<A, K<M, B>> f)` when `Some`: runs `f` inside `M`, maps the result to `Option.Some`, and wraps in `M.Pure`. When `None`: returns `M.Pure(Option<B>.None)` without calling `f`. This is the standard traversable law but the boundary consequence is: a seam that threads a logging or state-mutation effect through a traversal will silently skip the effect for every absent value. Contrast with `Option<A>.Traverse<F, B>` (applicative traversal) which has identical None behavior but is constrained to `Applicative<F>`, not `Monad<M>`. A side-effecting audit at a boundary that must fire unconditionally — even for absent values — cannot use either Traverse form; it must be sequenced as a separate operation outside the traversal.
- `OptionT<M, A>.IfSome(Action<A> f)` runs the action inside `M` only when `Some` — identical silent-skip behavior. `OptionT<M, A>.IfNone(Action)` runs the action only when `None`. A seam that must always execute an effect and conditionally branch on the value must use `Match(Some: x => effect(x), None: () => alternateEffect())` rather than combining `IfSome` + `IfNone` on separate chains — the latter creates two `M.Map` calls and two effect-bearing paths that may not compose correctly under `IO`'s resource tracking.

[OPTION_BIBIND_NULL_INTERIOR_TRAP]:
- `Option<A>.BiBind<B>(Func<A, Option<B>> Some, Func<Option<B>> None)` does not call `Check.NullReturn` — verified. If the `Some` branch returns `Some(null)` (constructing `new Option<B>(null)` for a reference-type `B`), the result is an `Option<B>` with `isSome = true` and `Value = null`. Subsequent `Map`, `Bind`, or domain code that extracts the value will receive `null` at a type-safe location. The `None` branch returning `null` produces a `default(Option<B>)` which is `None` — silently dropping the absent case without error. The law: both branches of `BiBind` must never return `null` from the factory — `Some(null)` is structurally `Some` but semantically corrupted, and a `null` `Option<B>` return is structurally `default(None)` with no diagnostic.

[FIN_BIMAP_ASYMMETRY_WITH_OPTION]:
- `Fin<A>.BiMap<B>(Func<A, B> Succ, Func<Error, Error> Fail)` maps to `Fin<B>` — the `Fail` branch transforms the error but stays in the `Fail` state; both branches return `Fin<B>`. Compare `Option<A>.BiMap<B>(Func<A, B> Some, Func<B> None)` — both branches return `B`, escaping the `Option` rail. `Fin.BiMap` cannot escape to `B` directly; it is an in-rail transform. A seam that needs to collapse a `Fin<A>` into a plain `B` with distinct Succ and Fail materializations must use `Match(Succ: ..., Fail: ...)`, not `BiMap`. `BiMap` is for error-retransform (e.g., re-coding an `Exceptional` into an `Expected` at an outbound seam) while staying within the `Fin` rail.

[VALIDATION_TO_FIN_NARROWING]:
- `Validation<Error, A>.ToFin()` is an extension method constrained to `Validation<Error, A>` — verified signature `Fin<A> ToFin<A>(this Validation<Error, A> ma)`. A seam that uses a custom `Validation<TError, A>` where `TError` is not `Error` cannot use this extension; it must manually match `Success` and `Fail` cases. The canonical shape: a `Validation<MyError, A>` at a seam projects the `Fail` case through `MyError.ToError()` (a conversion returning `Error.New(myError.Code, myError.Message)`) before calling `ToFin()` on the converted `Validation<Error, A>`. Placing this projection inline at the seam rather than inside the validation type keeps the canonical `Error` rail clean.

[ENUMIO_SYNCHRONIZATION_CONTEXT_AS_OPTIONAL]:
- `IO.syncContext` is declared as `IO<Option<SynchronizationContext>>` — verified: `public static readonly IO<Option<SynchronizationContext>> syncContext = lift((EnvIO e) => Prelude.Optional(e.SyncContext))`. The `SyncContext` field on `EnvIO` is nullable (`SynchronizationContext? SyncContext`). The absence of a synchronization context is a first-class optional: `IO.syncContext` is `Some(ctx)` in UI-thread environments and `None` in background-thread or console environments. A seam that conditionally marshals based on the presence of a context must use `IO.syncContext.Bind(ctx => ctx.Match(Some: c => marshalOn(c), None: () => runDirect()))` rather than reading `SynchronizationContext.Current` directly — the direct read would be a sentinel projection that bypasses the `EnvIO` boundary.

[SWAPMAYBE_SUCCESS_REJECTION_OBSERVABILITY]:
- `Atom<A>.SwapMaybe(Func<A, Option<A>> f)` returns type `A` — the raw current value after the operation, whether the swap succeeded or was rejected. The return value is the post-swap state if successful, or the unchanged pre-swap state if the function returned `None`. These two cases are indistinguishable from the return value alone. There is no discriminated result type. The pattern for observable take/release (e.g., `Atom<Option<Token>>` singleton ownership) encodes observability in the state structure: the take function returns `None` when the cell is already `Some(token)` (rejecting), or `Some(newToken)` when empty. After `SwapMaybe`, read `atom.Value`: if it equals the newly constructed token, the take succeeded; if it equals the prior token (which the caller holds a reference to), it was rejected. The caller must hold a reference to the prior value before the swap to distinguish the two outcomes — there is no post-hoc way to determine which branch executed from the return value alone.
