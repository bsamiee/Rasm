# Spectral-Eigen Deepening

[CONGRUENCE_REDUCTION_CORRECTNESS]:
- `Cholesky<T>.Factor` is documented and stored as the **lower** triangular L (A = L·Lᵀ). In the managed double provider `CholeskyFactor`: column i, rows j > i hold L[j,i]; rows k < i in column k > i are zeroed. Solving L·X = B must therefore use the factor directly as a matrix, not via `Cholesky.Solve` — `Cholesky.Solve(B)` internally calls `CholeskySolveFactored`, which performs both forward and backward substitution (POTRS equivalent: solves L·Lᵀ·X = B). `Cholesky.Solve` returns (L·Lᵀ)⁻¹·B, not L⁻¹·B.
- The wave-1 congruence formula `chol.Factor.Transpose().Solve(A)` is incorrect as a first step. `Factor.Transpose()` = Lᵀ; `.Solve(A)` routes to `DenseLU.Create(Lᵀ).Solve(A)`, which solves Lᵀ·X = A → X = L⁻ᵀ·A. This gives the wrong half-factor. The correct two-step sequence is: (1) `Y = chol.Factor.LU().Solve(A)` → solves L·Y = A → Y = L⁻¹·A; (2) `Z = chol.Factor.LU().Solve(Y.Transpose())` → Z = L⁻¹·Yᵀ; (3) `C = Z.Transpose()` → C = (L⁻¹·Yᵀ)ᵀ = Y·L⁻ᵀ = L⁻¹·A·L⁻ᵀ. LU of L is O(n²) since L is already triangular.
- `Matrix<T>` has no bare triangular solve API. `Matrix<T>.Solve` always routes through `LU()` (verified in `DenseMatrix`); there is no `ForwardSolve`, `BackwardSolve`, or `TriangularSolve` overload. For the congruence back-transform x = L⁻ᵀ·y: `chol.Factor.Transpose().LU().Solve(Y)` is the only abstract route. `Transpose()` materializes a copy (column-major transposition: new allocation, not a lazy view), making each congruence step allocate an n×n dense matrix.
- `DenseCholesky.Solve` throws `NotSupportedException("Can only do Cholesky factorization for dense matrices at the moment.")` when either the input or result is not a `DenseMatrix`. Passing a `SparseMatrix` or a submatrix view will throw at runtime, not compile time — the type is `Matrix<double>` at the abstract level.
- `Cholesky` positive-definiteness detection is exact: the managed path checks `num > 0.0` (strictly positive diagonal after previous Schur complement updates). A diagonal that reaches exactly zero (rank-deficient B on the boundary) throws `ArgumentException("Matrix must be positive definite.")`. Near-indefinite B (very small positive diagonal entry) passes the check and produces a numerically poor factor; the Cholesky factor becomes ill-conditioned rather than failing. Catching `ArgumentException` distinguishes indefinite B from genuine SPD; near-singular B requires conditioning with a small shift ε·I before factorization.

[GRAM_SCHMIDT_FAILURE_SURFACE]:
- `DenseGramSchmidt.Create` always clones the input matrix before factorizing (`DenseMatrix denseMatrix = (DenseMatrix)matrix.Clone()`). The `.Q` accessor returns the cloned, modified matrix — it is NOT the input. Extra O(n·k) allocation is unconditional.
- `DenseGramSchmidt.Factorize` throws `ArgumentException("Matrix must not be rank deficient.")` only when the column norm is **exactly** `0.0` (computed as `Math.Sqrt(sum_of_squares)` after the normalization phase). A near-zero column norm (numerically small but nonzero) propagates silently: the column is normalized by the near-zero value, producing a unit-norm vector with catastrophically amplified rounding noise. For LOBPCG block orthonormalization: every column norm must be checked against a domain-specific floor before `GramSchmidt()` is called; the API provides no tolerance parameter and no graceful degradation.
- `DenseGramSchmidt` performs **single-pass** modified Gram-Schmidt (one `i`-loop, one `l`-loop per column). There is no second reorthogonalization pass. For nearly-dependent columns — which arise in LOBPCG when the P block loses independence after many iterations — a second manual pass is required: compute `GramSchmidt()` on the surviving block, extract `.Q`, then call `GramSchmidt()` again on `.Q` if the R diagonal has entries below the floor. The two-pass protocol is not built into any MathNet surface.
- `DenseQR.Create(matrix, QRMethod.Thin)` uses Householder reflections via `ThinQRFactor` (provider-dispatched LAPACK `dgeqrf`/`dorgqr` equivalent in the managed path). For the same n×k block, `QR(QRMethod.Thin).Q` is numerically more stable than `GramSchmidt().Q` for near-dependent inputs because Householder reflections maintain orthogonality to machine precision without column-order sensitivity. The trade-off: Householder QR discards R but Q is n×k; thin QR storage is identical to GramSchmidt output. For LOBPCG orthonormalization, `QR(QRMethod.Thin).Q` is the preferred surface when near-dependence is expected.

[CONVERGENCE_ASYMMETRY]:
- `SymmetricDiagonalize` (real symmetric, complex Hermitian paths) uses **relative** convergence: `|e[k]| ≤ Precision.DoublePrecision × num2`, where `num2 = max over j of (|d[j]| + |e[j]|)` is a running spectral-radius estimate. This is the Wilkinson relative criterion. The iteration count is bounded at 1000 per sub-eigenvalue; `NonConvergenceException` (no inner exception, message `"An algorithm failed to converge."`) fires when the count is reached.
- `NonsymmetricReduceHessenberToRealSchur` (real nonsymmetric path) has **no iteration bound** and throws **no exception**. The `num9` counter resets to 0 on each successful deflation (`num--`); it triggers exceptional shift strategies at `num9 == 10` and `num9 == 30` but is never tested against a ceiling. For a pathological nonsymmetric matrix that never deflates, the managed path loops indefinitely. The MKL-backed path uses `dlahqr`/`dlaqr0` which do bound iterations internally; the managed path has no such guard.
- The asymmetry is receipt-critical: `NonConvergenceException` is catchable and produces a clean `Fin.Fail` from `Try<A>.Run()`; the nonsymmetric divergence case produces no exception and instead returns partially-converged eigenvalues silently. A receipt built over a nonsymmetric Evd must add a residual validation step: compute `‖A·V - V·D‖_F` and emit a distinct diagnostic `Unusable` flag when the Frobenius residual exceeds the policy threshold, because the API provides no convergence evidence for nonsymmetric inputs.
- `Evd<double>.Rank` uses `EigenValues[i].AlmostEqual(Complex.Zero)` with the default threshold `DefaultDoubleAccuracy = Precision.DoublePrecision × 10 ≈ 1.11 × 10⁻¹⁵`. `Complex.AlmostEqual(Complex.Zero)` uses the norm-of-difference path: `|a.MagnitudeSquared() − 0| < 1.11 × 10⁻¹⁵`. Because MathNet's `Norm(Complex)` returns `MagnitudeSquared` (not `Magnitude`), an eigenvalue with `|λ| ≈ 3.3 × 10⁻⁸` has `|λ|² ≈ 1.1 × 10⁻¹⁵` and is classified as zero by `Rank`. This threshold is not spectral-radius-relative; for matrices with large spectral radius (e.g., ‖A‖ ∼ 10⁶), eigenvalues of magnitude ∼ 10⁻⁸ that are numerically significant may be incorrectly counted as zero-rank contributions. Receipt-level rank must be recomputed with `n − count(λᵢ : |λᵢ| < ε_rank × ‖A‖₂)` where ε_rank is a caller-supplied policy.

[HERMITIAN_EIGENVALUE_CARRIER]:
- For a `Matrix<Complex>` with `Symmetricity.Hermitian`, `DenseEvd.Create` branches on `matrix.IsHermitian()` (abstract override) rather than `IsSymmetric()`. The `isSymmetric` flag routes to `SymmetricTridiagonalize(Complex[], double[], double[], Complex[], int)` → `SymmetricDiagonalize(Complex[], double[], double[], int)` → `SymmetricUntridiagonalize`. After `SymmetricDiagonalize`, the `e[]` (off-diagonal) array is zeroed by `e[j] = 0.0` inside each convergence pass and the terminal `e[order-1] = 0.0` assignment. The eigenvalue vector is then filled as `vectorEv[i] = new Complex(d[i], e[i])` where `e[i] == 0.0` exactly — Hermitian eigenvalues have zero imaginary part in the receipt's `Vector<Complex>`.
- `SymmetricUntridiagonalize` (complex Hermitian untransformation) applies `dataEv[j * order + i] = dataEv[j * order + i].Real × tau[i].Conjugate()` for each (i, j). The `.Real` extraction plus conjugate multiplication restores the full complex eigenvector structure from the real rotations used during `SymmetricDiagonalize`. This means complex Hermitian eigenvectors are not real-valued — they are genuinely complex with nonzero imaginary parts, but eigenvalues are real. A receipt that projects eigenvectors to real is incorrect for the Hermitian case.
- Eigenvalue ascending sort applies to both real symmetric (`UserEvd.SymmetricDiagonalize`) and complex Hermitian (`ManagedLinearAlgebraProvider.SymmetricDiagonalize`) paths: a selection sort over `d[]` swapping corresponding eigenvector columns. This is O(n²) in eigenvalue permutations with O(n) cost per swap of n-element columns — O(n³) total in the worst case, not negligible relative to the QR iteration. The LAPACK provider follows `dsyev`/`zheev` ascending-order contract.
- For a nonsymmetric complex `Matrix<Complex>`, the eigenvalue carrier is also `Vector<Complex>` but imaginary parts are generally nonzero. `NonsymmetricReduceHessenberToRealSchur` for complex inputs runs the full complex Schur path and fills `vectorEv[i]` directly. No real-parts-only projection is valid for the nonsymmetric complex case.

[MATRIX_MULTIPLY_API_PRECISION]:
- `A.TransposeAndMultiply(B)` computes A·Bᵀ (this times other-transposed). `A.TransposeThisAndMultiply(B)` computes Aᵀ·B (this-transposed times other). `A.ConjugateTransposeAndMultiply(B)` computes A·Bᴴ. `A.ConjugateTransposeThisAndMultiply(B)` computes Aᴴ·B. All four are verified against the BLAS flags passed to `MatrixMultiplyWithUpdate`: `TransposeAndMultiply` → `(DontTranspose, Transpose)`; `TransposeThisAndMultiply` → `(Transpose, DontTranspose)`.
- For the Rayleigh-Ritz matrix S = Wᵀ·A·W over subspace W: `W.TransposeThisAndMultiply(A.Multiply(W))` is the two-call form. An equivalent but more cache-friendly single-pass is `W.TransposeThisAndMultiply(A).Multiply(W)` if A·W is reused across the iteration. Neither form exploits the symmetry of S; both allocate intermediate n×3k matrices. The re-symmetrization `S = (S + Sᵀ) / 2` uses `(s + s.Transpose()).Multiply(0.5)` — note `Transpose()` allocates a new matrix, so a pointwise average `(s + s.Transpose()) * 0.5` allocates three matrices. In-place form using `s.PointwiseMultiply(one_half) + s.Transpose().PointwiseMultiply(one_half)` is equivalent but still two allocations; the minimum is a `MapIndexedInplace` that writes `(s[i,j] + s[j,i]) / 2` directly.
- `Matrix<T>.IsSymmetric()` for the dense real case uses exact equality `_values[j * n + i] != _values[i * n + j]` (confirmed in `DenseMatrix.IsSymmetric`). After floating-point accumulation from `Wᵀ·A·W`, the result will NEVER be exactly symmetric bit-for-bit. Passing `Symmetricity.Unknown` to `Evd` on an accumulated Rayleigh-Ritz matrix always triggers `IsSymmetric()` → `false` → nonsymmetric Schur path, even when the matrix is mathematically symmetric. The `Symmetricity.Symmetric` hint must be passed after explicit re-symmetrization to route to the symmetric QR path.

[GENERIC_MATH_SCALAR_CEILING]:
- `System.Numerics.Complex` satisfies `INumberBase<Complex>` and `ISignedNumber<Complex>` but not `INumber<Complex>` (which requires `IComparable<T>`). `Complex` has no total order. A generic spectral kernel constrained on `where T : INumber<T>` excludes `Complex` at compile time. The correct constraint for a kernel that handles both `double` and `Complex` is `where T : INumberBase<T>`, but this constraint excludes `IComparable<T>`, blocking any sorting or ordering inside the generic body.
- `TensorPrimitives.Dot<T>` requires `T : IAdditionOperators<T,T,T>, IAdditiveIdentity<T,T>, IMultiplyOperators<T,T,T>, IMultiplicativeIdentity<T,T>`. `Complex` satisfies all four (homogeneous). `TensorPrimitives.Norm<T>` requires `T : IRootFunctions<T>` — `Complex` does NOT implement `IRootFunctions<Complex>`. `TensorPrimitives.MultiplyAdd<T>` requires `T : IAdditionOperators<T,T,T>, IMultiplyOperators<T,T,T>` — `Complex` satisfies this. `TensorPrimitives.FusedMultiplyAdd<T>` additionally requires `T : IFloatingPointIeee754<T>` — `Complex` does not satisfy this.
- The cross-type gap (`IMultiplyOperators<Complex, double, Complex>` not defined on `Complex`) means a generic weight-application function `void ApplyWeights<TM>(Matrix<TM> V, double[] weights)` cannot scale complex columns by real weights via a single generic multiply. The delegate-bundle form — `Func<TM, double, TM> scale` passed alongside the matrix — is the only correct wide-parameterization. The concrete instance is `(c, w) => new Complex(c.Real * w, c.Imaginary * w)` for `Matrix<Complex>`, provided at construction time. Attempting to use `Vector<Complex>.Multiply(double scalar)` directly: `Vector<T>.Multiply(T scalar)` requires `scalar : T`, so the scalar must be `Complex`, not `double`. Wrapping `w` as `new Complex(w, 0)` and calling `.Multiply(new Complex(w, 0))` is the correct workaround but breaks the generic delegate signature.

[OVER_COMPUTE_TRUNCATION_ALLOCATION]:
- `evd.EigenValues.AsArray()` returns the raw backing `Complex[]` without copy for `DenseVector<Complex>` (which is the concrete type). `evd.EigenVectors.SubMatrix(0, n, 0, k)` returns a new `DenseMatrix` allocating an n×k backing array. For cached basis reuse with different k: the full eigenbasis should be retained as the `DenseMatrix` from `evd.EigenVectors`; truncation is then a `SubMatrix` call at filter time (read-only view is NOT available — `SubMatrix` always allocates). A zero-copy truncation requires accessing `evd.EigenVectors.Values` (the `double[]` backing) and constructing a `DenseMatrix` with offset columns directly via `DenseColumnMajorMatrixStorage<double>.OfColumnMajorArray(n, k, values_subset)` — but this requires constructing a new backing array since `DenseColumnMajorMatrixStorage` takes ownership.
- The receipt shape for over-compute-and-truncate must carry three fields: `ComputedCount = n` (full eigensystem), `RequestedCount = k`, `CacheSource` distinguishing `Full` (truncated from a cached n-wide basis) vs `Fresh` (new full computation). When the cache hit has a compatible n and Symmetricity, reuse requires only SubMatrix on the already-computed EigenVectors — the O(n³) eigensolver is not re-invoked.
- `evd.EigenValues` is always `Vector<Complex>` regardless of input scalar type. For real symmetric inputs: all imaginary parts are 0 (confirmed by e[] zeroing in `SymmetricDiagonalize`). Constructing a `double[]` weight array from the receipt: `evd.EigenValues.AsArray().Select(c => c.Real).ToArray()` copies and projects in O(n). A faster route accesses the backing directly and strides: `MemoryMarshal.Cast<Complex, double>(evd.EigenValues.AsArray())` gives a `Span<double>` with interleaved real/imaginary pairs; the real parts are at even indices (stride 2), usable directly via `TensorPrimitives` with stride — but `TensorPrimitives` has no strided overload. The select-project copy remains the clean route.

[SPECTRAL_FILTER_ZERO_MODE_MECHANICS]:
- `Vector<T>.Map(f, result, Zeros.Include)` passes zero-valued elements to the map function. For a spectral weight vector `w[i] = λᵢ < ε_zero ? 0.0 : f(λᵢ)`, calling `eigenvalues.Map(λ => λ.Real < ε_zero ? 0.0 : f(λ.Real), weightVector, Zeros.Include)` is the correct form. `Zeros.AllowSkip` (the default) may skip zero-valued eigenvalues in sparse storage; for `DenseVector` the distinction is immaterial since all elements are stored. Using `MapIndexed` instead of `Map` when the index carries semantic rank (e.g., the 0th mode is always the zero mode for normalized graph Laplacians) enables index-first exclusion without the epsilon comparison.
- The partial-monoid composition `f_composed(λ) = f₁(λ) × f₂(λ)` applied via `weights1.PointwiseMultiply(weights2)` is an O(n) dense vector product. The identity filter (all weights = 1.0) is `Vector<double>.Build.Dense(n, 1.0)`. An identity filter check before multiplication: `weights.ForAll(w => w == 1.0)` avoids the O(n) multiply; this `ForAll` uses `Storage.Find` internally and short-circuits on the first mismatch.
- The bandwidth floor condition — weight sum below threshold signals degenerate filtering — is computed as `weights.Sum()` (O(n), returns `double`). For a heat filter with large t, all weights can underflow to 0.0 in `double` precision when `exp(-t × λᵢ) < double.Epsilon` for all i. The receipt must carry the pre-filter weight sum as evidence; a zero weight sum is an exhaustion marker distinct from a successful zero-signal result.

[EXHAUSTION_SEMANTICS_RECEIPT]:
- `NonConvergenceException` carries no partial-result payload: the exception is thrown inside the QR sweep loop before the final sort and back-transform complete. The partially-modified `eigenVectors` matrix (passed by reference into `SymmetricDiagonalize`) may contain intermediate Householder/rotation accumulation that is neither the original eigenvectors nor the final converged ones. The `matrixEv` argument to `EigenDecomp` (which becomes `DenseEvd.Create`'s eigenvector storage) is in this corrupted intermediate state when the exception fires. `Try<A>.Run()` wraps the exception into `Error.New(NonConvergenceException)`, but the `Evd` object is never returned — there is no partial-result inspection path.
- Exhaustion, indefinite-B, and non-square failures are all `ArgumentException` or `NonConvergenceException` from the same Evd construction site. A receipt-level error discriminant must be established at the `Fin.Fail` boundary using `error.Exception?.GetType()` or a custom error tag: `ExhaustionFault`, `IndefiniteBFault`, `ShapeFault`. Collapsing all three to `Error.New` destroys the diagnostic signal; each requires a different recovery action (increase matrix size, perturb B by ε·I, fix input shape).
- The nonsymmetric path has no exhaustion semantics at all. For a nonsymmetric matrix whose Schur form fails to converge in the managed path, the function returns normally with eigenvalues that are partially correct (some deflated correctly, others not). The only detection path is residual validation: `(A.Multiply(V) - V.Multiply(D)).FrobeniusNorm()` compared against a policy threshold. A receipt for nonsymmetric Evd must include this post-hoc residual; without it, partially-converged nonsymmetric eigenvalues are indistinguishable from fully-converged ones.

[CONGRUENCE_BACK_TRANSFORM_SHAPE]:
- After congruence reduction C = L⁻¹·A·L⁻ᵀ and standard Evd C·y = λ·y with Y = EigenVectors, the generalized eigenvectors X satisfy A·X = λ·B·X where X = L⁻ᵀ·Y. The back-transform step: `chol.Factor.Transpose().LU().Solve(evd.EigenVectors)`. This is two O(n²) triangular LU factorizations of Lᵀ (one for setup, one per right-hand column) plus an O(n·k) forward-elimination for k eigenvectors. Forming `chol.Factor.Transpose()` allocates an n×n copy; then `.LU()` allocates the LU factored copy. Three n×n allocations for the back-transform step.
- Re-symmetrization of C after congruence is mechanically equivalent to re-symmetrization after Rayleigh-Ritz: `(c + c.Transpose()).Multiply(0.5)`. But the congruence asymmetry grows as `O(n² × machine_epsilon × κ(L))` where κ(L) is the condition number of L. For well-conditioned B (κ(B) ∼ 1), re-symmetrization adds only negligible change; for ill-conditioned B (κ(B) ∼ 10¹²), the asymmetry can reach O(10⁻⁴) and the un-re-symmetrized C would route through the nonsymmetric Schur path, producing complex eigenvalues for what should be a real symmetric-definite problem.
- The route name `generalized-symmetric-definite` in a receipt must discriminate the two-stage nature: Cholesky reduction succeeded (L exists), standard Evd was called on C, and the back-transform applied. If Cholesky fails (indefinite B), the receipt carries `IndefiniteBFault` and no eigenpairs. If Cholesky succeeds but Evd on C fails (non-convergence), the receipt carries `ExhaustionFault` with the Cholesky evidence still valid (L is computed and correct). These are distinct receipt states, not collapsed into one `Fail`.

[DESCRIPTOR_NORMALIZATION_AND_DEGENERATE_SPREAD]:
- Spectral descriptor normalization using eigenvalue spread: a descriptor normalized by `(λ_max − λ_min)` fails when `λ_max == λ_min` (degenerate spectrum). The correct guard is a failure receipt with a `DegenerateSpread` case, not a `0/0 → NaN` clamp or a silent `1.0` fallback. A degenerate spread means the filter has collapsed all frequencies into one bin — the filtered output carries zero information content. Receipt-level `DegenerateSpread` is a distinct exhaustion marker: not `Unusable` (which applies to non-converged pairs), not `Success` (which implies well-conditioned output).
- For Laplacian-derived operators with zero-mode excluded, the effective λ_min is `λ₁` (the algebraic connectivity). If λ₁ is below the zero-mode epsilon ε_zero, the exclusion removes it and the next eigenvalue becomes the spread floor. A policy that blindly uses `EigenValues[1].Real` as λ_min will use a near-zero mode that was supposed to be excluded; the spread floor must be computed as `eigenvalues.Where(λ => λ.Real > ε_zero).Min(λ => λ.Real)`.

**Further Considerations:**

The Rayleigh-Ritz matrix S = Wᵀ·A·W, built via `W.TransposeThisAndMultiply(A.Multiply(W))`, has a subtle memory-layout interaction: `DenseMatrix.Values` is column-major, and `TransposeThisAndMultiply` dispatches to `MatrixMultiplyWithUpdate` with `(Transpose, DontTranspose)`. When W is n×3k and A·W is n×3k, the BLAS call computes a 3k×3k result; the 3k BLAS columns of the intermediate are O(n) each. Providing A as a `SparseMatrix` would avoid the O(n²) dense intermediate for sparse A, but `SparseMatrix.Multiply(DenseMatrix)` internally densifies before calling the provider — there is no sparse-dense BLAS dispatch in the managed path. For large sparse A, the matrix-vector product used in the LOBPCG operator application must be hand-built via `SparseMatrix.Multiply(Vector<double>)` column-by-column, not `SparseMatrix.Multiply(Matrix<double>)`.

The `SymmetricDiagonalize` selection sort (both real and Hermitian paths) runs after QR convergence and has time complexity O(n²) in comparisons and O(n²) in eigenvector column swaps — each swap touches n elements. For n ∼ 5000 (a moderate graph Laplacian full Evd), the sort costs O(25 × 10⁶) operations on top of the O(n³) QR work, which is negligible. For n ∼ 500 where full Evd is routine in spectral mesh processing, the sort is O(250,000) — invisible. The real performance note is that the LAPACK provider path (`MKL` / `OpenBLAS`) bypasses `SymmetricDiagonalize` entirely and uses the native driver's sort contract; only the managed path runs this selection sort.

The `NonConvergenceException` thrown by `SymmetricDiagonalize` at iteration count 1000 per outer `j` loop corresponds to a specific eigenvalue pair failing to reduce its off-diagonal residual below `Precision.DoublePrecision × spectral_radius`. A matrix that triggers this is one where the ratio between two near-coincident eigenvalues is extremely close to 1 — nearly degenerate eigenvalues. Near-degeneracy is itself spectral information: it indicates a graph with near-symmetric structure or a repeated geometric eigenvalue. A receipt that converts `NonConvergenceException` to `ExhaustionFault` should also record the spectral radius estimate at the time of exhaustion, enabling the caller to diagnose whether the failure arose from near-degeneracy (ratio ∼ 1) or from numerical ill-conditioning (small off-diagonals that oscillate rather than decay).
