# Lock-Free Coordination Substrate Beneath the Change Event

[CAS_SPIN_IS_THE_REAL_PRIMITIVE_NOT_THE_EVENT]:
- The single-value cell's update is a read-apply-`CompareExchange` spin: read the current boxed reference, apply the transform to its unboxed value, allocate a fresh box for the result, and `Interlocked.CompareExchange` the new box reference against the read reference. A losing exchange means another thread committed in the window; the loop calls `SpinWait.SpinOnce` and re-reads. The transform is therefore re-invoked once per lost race, which is the structural reason it must be side-effect free — not a style guideline but a correctness precondition of the retry mechanism.
- The compare token is the boxed object reference, never the value. For a reference-typed payload the box is the payload; for a value-typed payload each update allocates a new heap box and the CAS compares box identities. Two updates that produce equal values still allocate distinct boxes and still contend, so a hot cell carrying a struct pays one allocation per successful commit plus one per lost race — the allocation count is a load metric, and a high-frequency value cell is a candidate for a reference-wrapped payload that mutates a pre-boxed slot.
- The change notification fires exactly once, after the loop breaks, with the committed value as its only argument. It does not fire on a lost race, does not fire per transform invocation, and does not carry the prior value or any metadata the cell holds beside the value. A subscriber that needs the delta maintains its own shadow read taken before the swap; a subscriber that re-reads the cell inside the handler may observe a value already advanced past the one the event delivered, because the notification runs outside the atomic region.
- The metadata-carrying cell threads its immutable metadata into the transform as a second argument but excludes it from both the CAS comparand and the change payload — metadata is read-only context for the transform, never part of the compared or announced state. The conditional-update form short-circuits before the CAS when the transform returns absence, leaving the cell and its change event untouched: a no-op is structurally distinct from a committed identity write, and only the latter announces.

[VALIDATOR_FAILURE_IS_A_NON_EVENT_NOT_A_RETRY]:
- The cell's validator runs on the candidate value after the transform and before the CAS attempt. A rejected candidate returns the current (unchanged) value to the caller, does not retry, and does not fire the change event — the rejection is silent and indistinguishable at the call site from a successful no-op unless the caller compares the returned snapshot against its own prior read. A guarded cell therefore cannot signal rejection through its rail; the seam must inspect the returned snapshot or carry the accept/reject decision on a separate channel.
- A cell constructed with a validator runs that validator once at construction and refuses to materialize if the seed fails, returning absence from the factory. The cell can consequently never hold an invalid value across its whole lifetime — the construction-time check plus the per-swap check are the two gates, and interior code reading the cell never re-validates.

[MVCC_TRANSACTION_IS_SNAPSHOT_PLUS_VERSION_NOT_A_LOCK]:
- The transactional reference family runs under multi-version concurrency control with no locks on the read or write path. A transaction opens by snapshotting the entire global reference map into a thread-local transaction object keyed by an ambient async-local slot; all in-transaction reads and writes hit the snapshot's copy, recording touched ids into per-transaction read and write sets. Commit re-acquires the live map under one CAS-style swap, validates, applies, and bumps per-reference versions. The whole engine is one CAS at commit plus a snapshot copy at open — the persistent trie makes the snapshot free and the structural-sharing writes cheap, which is the precondition the engine assumes its payloads honor.
- Conflict detection is version-equality at commit, scoped by isolation. Snapshot isolation validates only the write set: a write commits only if the live reference version still equals the snapshot version it was read at, otherwise the whole transaction throws an internal conflict marker, is caught, spins once, and re-runs against a fresh snapshot. Serialisable isolation additionally validates the read set version-by-version before applying writes, so a reference that was only read — never written — still forces a retry if a concurrent committer advanced it. The two isolations are a read-set-validated-or-not switch, not two engines; the stricter one trades higher conflict probability for cross read-write consistency.
- Commit runs inside the global map's own swap closure, so validation and application are themselves atomic against other committers — a second transaction cannot interleave between the read-set check and the write application. Version numbers are per-reference and monotonic; the engine never compares values, only versions, so two transactions writing structurally-equal values to one reference still conflict on version.

[COMMUTE_DOUBLE_EVALUATION_IS_THE_CONCURRENCY_UNLOCK]:
- A commuting update is the lock-free engine's tool for contention-free accumulation. The commute function runs once in-transaction against the snapshot value to produce the caller-visible result, and is recorded; then at the commit point it runs *again* against the most-recently-committed value, and that second result is what lands. Because the second application reads the live committed value rather than the snapshot, a commute never conflicts on its own reference — two transactions both commuting the same counter both succeed without retry, where two plain writes would force one to spin.
- The contract this buys concurrency with is commutativity: the function must produce a correct result regardless of which committed value it sees at the commit re-run, because the order of concurrent commutes is unspecified. A non-commutative function degrades to last-application-wins semantics rather than failing loudly, so an increment, a set-union, or a max is correct as a commute while a "set to a value derived from a stale read" is silently wrong. The commute is the precise instrument for a fan-in counter or accumulator written from many event-source threads at once; a plain transactional write is correct only when the new value does not depend on the prior committed value.

[ABORT_AND_VALIDATION_NOTIFICATION_ASYMMETRY]:
- The transactional change event fires inside the transaction's `finally` for every iteration, conflict-aborted retries included, carrying the in-transaction proposed diff rather than the committed diff. A conflict-aborted attempt accumulated proposed writes, so the diff is non-empty even though nothing committed; a subscriber that trusts the payload acts on a rolled-back write. The only safe read is to re-read the committed reference value after the notification and treat the payload purely as a "something was attempted" signal — the change event on a transactional reference is a notification of *intent*, not of *commitment*, which is the exact inverse of the single-value cell whose event fires only post-commit.
- Reference validation and cell validation fail in opposite shapes. A cell validator that rejects returns the old value and retries nothing — a soft, silent no-op. A reference validator that rejects throws a hard validation-failed exception from inside the commit closure, and that exception is *not* the conflict marker the spin loop catches, so it propagates out of the entire transaction and tears it down permanently — there is no retry, the world is not re-snapshotted, and the caller receives the throw. A seam mixing both must know that the same word "validation" denotes a recoverable no-op on one surface and a fatal abort on the other.

[ISOLATION_HELPER_DOES_NOT_MATCH_ITS_NAME]:
- The convenience entrypoint whose name reads as the serialisable level actually passes the snapshot level to the engine — only the explicitly-parameterized atomic entrypoint with the serialisable argument performs read-set validation. A seam that needs a read-only reference protected from concurrent mutation mid-transaction must spell the isolation argument explicitly; relying on the named helper silently gets write-set-only validation and the read-protection it appears to promise never runs. This is a real divergence between the documented intent and the dispatched behavior, and the explicit-argument form is the only correct route to read-set protection.

[CHOOSING_THE_COORDINATION_PRIMITIVE_IS_AN_INVARIANT_QUESTION]:
- The three lock-free surfaces partition by the shape of the invariant, not by convenience. One value transitioning atomically with no cross-value constraint is a single-value CAS cell — the cheapest, one allocation and one exchange per commit. Multiple values that must change as an indivisible unit are transactional references under one transaction — the engine guarantees all-or-nothing across the set. A keyed collection whose per-key and whole-map mutations both need atomic announcement is the lock-free map cell, which CAS-swaps the whole trie and announces a key-scoped or trie-scoped patch after the exchange. Wrapping independent single-value cells in a transaction over-serializes; using one cell for a multi-value invariant under-protects. The map cell's change fidelity is conditional on threading the provided tracking collection through structural edits — a swap body that ignores the argument and rebuilds the collection emits a patch showing only the final state with no tracked intermediate diff, so precise deltas are a property of how the swap is written, not a guarantee of the cell.
- A coordinated lock — the scoped value-typed lock whose enter returns a stack-only scope — is earned only when the invariant requires multiple reads and writes to be mutually exclusive as a block, which the CAS cells and the transaction cannot express because they are value-replacement primitives, not critical sections. The scope is illegal across an await point; an async-spanning gate needs a counting semaphore instead. Reaching for a lock where a CAS cell or a transaction suffices is the over-serialization failure; the lock is the last resort, not the default.

[CAS_SUBSTRATE_AS_AN_EVENT_SINK]:
- The change event fires synchronously on the committing thread inside the post-CAS path before the swap returns, which makes a handler that re-swaps the same cell a genuine reentrancy hazard distinct from ordinary blocking: the inner swap completes and fires its own nested change before the outer swap's call frame returns, so a handler observing-then-reacting sees its own reaction's notification interleaved with the original. The substrate has no reentrancy guard; the handler that must trigger a further transition routes it through a channel so the second transition runs on a drained continuation, not nested on the committer's stack.
- A swap that loses the CAS race re-reads and re-applies, so under a storm of writes the last writer always wins and intermediate values are coalesced away rather than queued — the cell is structurally a latest-value register, not a log. This is exactly the semantics a per-frame or per-tick consumer wants from a high-frequency host event: the callback's only work is a context-free swap, the cell absorbs unbounded write frequency by collapsing to the latest, and a downstream drain reads one current value per tick. A consumer that must see every intermediate cannot use the cell as its transport and needs a channel, whose buffer is the log the cell deliberately is not.
- The cell holds a strong reference to every registered handler and transitively to its target, so an activation-scoped subscriber on a process-static cell keeps its target alive past deactivation until detached by the exact delegate identity used to attach — a leak the transactional reference shares because its change delegate is held the same way. The substrate provides no weak-subscription mode; symmetric attach-detach keyed on captured identity is the only teardown, and it must run before the handles the handler closes over are disposed, or a late race fires the handler against a torn target.
