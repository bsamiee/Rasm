# [TYPESCRIPT_ARCHITECTURE_POSTURE]

One page fixes the Effect application doctrine across the whole five-domain flat TS lib branch — the altitude grammar that admits exactly one form per concern, the closed app-service owner budget, the platform-neutrality strata over three reachability levels (neutral interior, the two-folder browser publication, the node publication), the runtime-and-version posture every owner peers against, the admission discipline, the two cross-package laws (anti-spam cluster, altitude-co-location), and the RULE_ENFORCEMENT table naming every mechanical guard. The mechanics of each owner live on its finalized domain page; this page legislates the altitudes those owners obey across domains. It crosses no wire contract and authors no decode. This is a `CROSS_PACKAGE_LAWS`-tier page, not a domain page.

## [2]-[APP_SHAPE]

- Owner: the altitude grammar, the closed app-service budget, and the key-discriminated state-layer posture every domain page obeys. Each suite altitude admits exactly one TS form; a second form on the same altitude is the named defect the grammar deletes.
- Cases: vocabulary rides `Schema.Literal` unions with behavior in a companion `Record`, never parallel const enums; domain shape is one `Schema.Class` per boundary concept with projections derived from the owning class via `Model.fields`/`Schema.pick`, never a parallel struct; the error rail is `Data.TaggedError` families, one fault family per rail, and `FaultDetailRail` reconstructs the .NET faults as one tagged family bound by `Match.tagsExhaustive`; effects ride `Effect<A, E, R>`, pure branching rides `Either`, connect-es server-streams ride `Stream`, and a thrown exception never crosses a module boundary; services are `Effect.Service` classes, one owner per axis; composition is one `Layer` graph per publication entry and one runtime per host surface; the boundary decodes with `Schema.decodeUnknown` at the wire edge while the interior never re-validates, `Option` carries absence, and `null` exists only at the JSON boundary.
- Entry: the app-service owner budget is closed at five — `WireClients` (the four browser-dialable generated services over one shared grpc-web transport, `interchange`), `CommandGateway` (the outbound verb face with intent keys and availability gating, `interchange`), `SnapshotFeed` (msgpack-decoded persistence fold, `projection`), `RuntimeFeed` (JSON record decode and receipt-envelope payload binding, `projection`), and `EvidenceFeed` (correlation-keyed envelope ingestion and the HLC skew-band confidence-interval fold, `projection`); a new capability lands as a method or row on one of the five, and a sixth sibling app-service is the named defect; the five strengthened `platform` infrastructure owners (`AppRouter`/`ServiceWorkerHost`/`CrashTelemetry`/`RemoteConfig`/`PerformanceBudget`) are platform-bound HOST owners, NOT app-services — they sit at the host-substrate altitude exactly as `BrowserPlatform`/`AuthSession`/`RuntimeConfig` do and never count against the closed five-app-service budget; the state layer is one fold per boundary concept over a server-stream or a receipt sequence into an immutable keyed map, the discriminant driving the fold consumed verbatim from the wire vocabulary; a parallel view-state library over domain data is the rejected form.
- Packages: `effect` for `Schema`, `Effect`, `Either`, `Stream`, `Option`, `Data.TaggedError`, `Match`, `Layer`, and `Effect.Service`.
- Growth: a new altitude form is never admitted — the grammar is closed; a new boundary concept lands as one fold owner and one app-service method, never a new altitude or a sixth service.
- Boundary: this cluster legislates the altitudes; `WireTransport`, the decode rails, and the gateway are owned by `interchange`, the fold owners by `projection`, the UI/UX render and binding surfaces by `ui` (the AppUi-analog library), the SPA infrastructure and browser entry by `platform` (the AppHost-analog host), and the node tiers plus the provisioning subpath by `services`, each composed as settled vocabulary and never re-taught; the closed budget is the architecture invariant every domain transcribes.

## [3]-[PLATFORM_NEUTRALITY]

- Owner: the domain-folder strata that keep the platform-neutral interior, the two-folder browser publication, and the node publication separate inside ONE package, and the inward dependency direction every domain folder obeys. The branch is one npm package `@rasm/ts` of five flat domain folders under `libs/typescript/` (NO `src/` nesting — module leaves sit directly at each domain root); the bundle split is a subpath-export attribute plus a folder-scoped lint stratum, never five published packages, never the folder axis, and never a runtime guard.
- Cases: three reachability strata over five folders — the platform-neutral interior (`interchange` + `projection`, stratum `neutral`) both publications compose; the browser stratum of TWO folders (`ui`, stratum `browser`, the `./ui` UI-library entry; `platform`, stratum `browser`, the `./web` SPA-application entry); the node-only stratum (`services`, stratum `node`, the `./node` + `./provisioning` publication entries). A browser folder CANNOT import a node folder and vice versa because the folder-scoped ESLint `no-restricted-imports` rule forbids the cross-stratum source path across `browser`/`node`/`neutral` folder globs; the rule is browser↔node forbidden with the neutral interior shared by both publications. Inside the browser stratum the direction is one-way: `platform` MAY import `ui` (the SPA entry composes the UI library) but `ui` MAY NEVER import `platform`, so the browser stratum is authored as TWO rules — a shared browser↔neutral/node rule applied to both folders, PLUS a `ui/**`-specific rule banning every `../platform` and `platform/**` path; a single combined `{ ui/**, platform/** }` group lets a `ui` -> `platform` import silently pass, the named lint-authoring defect the two-rule split prevents (RULE_ENFORCEMENT row [7]). Because `interchange` and `projection` are BOTH neutral, the stratum graph does not separate the `CommandGateway`-reads-`AvailabilityStore` intra-neutral seam, so the banned-import lint rule on the `projection` folder forbidding the entire `@connectrpc/*` set is the sole remaining mechanical guard keeping the fold interior transport-free (RULE_ENFORCEMENT row [1]); the read is an intra-package module edge, not a published-package boundary.
- Entry: dependencies flow inward from the publication entries toward the wire boundary — the committed `FileDescriptorSet` drives `buf generate` into `interchange/gen`, `interchange` derives transport and decode rails, `projection` folds decoded values into keyed stores, `ui` subscribes to the stores and renders the leaf surfaces, `platform` composes `ui` into the SPA root and binds the host infrastructure (composition root, substrate, routing, service-worker, crash telemetry, remote config, vitals), and `services` composes interchange+projection plus its own durable/runner/SQL/RPC/provisioning owners; no domain folder depends on a folder above it in the flow, the node folder never depends on a browser folder, and inside the browser stratum `ui` never depends on `platform`.
- Packages: `@effect/platform` for the neutral platform primitives, `@effect/platform-browser` for the browser bindings the two browser folders share, `@effect/platform-node` as the node driver host; the ESLint flat-config `no-restricted-imports` rule as the folder-stratum bundle-fence owner.
- Growth: a new platform binding lands as one platform-layer row on its tier; a new domain owner lands as a leaf in its owning domain folder, never a new domain folder unless a genuinely new bounded concern warrants one.
- Boundary: integration crosses only the eleven wire contracts the `interchange` inventory fixes, exactly as the C# packages integrate only through ports; no import, path mapping, or build edge resolves into `libs/csharp`, and a cross-stratum browser-to-node source import or a `ui` -> `platform` import is the named coupling defect the folder-scoped lint rule is the single enforcement point against — `package.json` subpath `exports` plus the stratum lint carry the bundle fence, build-config not domain types; the `services` `./provisioning` subpath keeps the `@pulumi/*` deploy-time closure off the durable runtime hot path (RULE_ENFORCEMENT row [2]).

## [4]-[RUNTIME_AND_TOOLCHAIN]

- Owner: the held Effect runtime line every admitted satellite peers against, and the dual-compiler typecheck posture the whole branch builds on. The runtime is the single design constant under the entire owner set; the toolchain is the proof gate every page transcribes to.
- Cases: the Effect core holds at the v3 line past the async-local-storage context-loss patched point, and every admitted satellite — platform, rpc, sql, sql-pg, cluster, workflow, experimental, cli, ai, atom, opentelemetry, vitest — peers on the v3 line, so the whole satellite set resolves together under `catalogMode: strict`; the v4 line carries no stable release and is a designed-only growth row, never a catalog pin; the typecheck runs the native tsgo binary as the canonical `--noEmit` proof while stable `typescript` runs as the peer-bridge smoke, the catalog pin being the newest published dev-nightly refreshed each loop; the strictness floor the native checker enforces is `isolatedDeclarations`, `erasableSyntaxOnly`, `exactOptionalPropertyTypes`, `verbatimModuleSyntax`, `noUncheckedIndexedAccess`, and `customConditions`, encoded in `tsconfig.base.json` so every domain inherits it.
- Auto: `Schema.Class` and `Model.Class` field declarations carry their own decoded and encoded types, so `isolatedDeclarations` emits the public surface without a hand-written `.d.ts`; the strictness floor makes a `null`-in-interior or an unchecked index access a typecheck failure, not a review finding. `isolatedDeclarations` is load-bearing, not stylistic: the five cross-domain app-services need explicit `Schema.Class`/`Model.Class` field annotations to emit `.d.ts` without cross-file resolution, so the Model.Class-single-owner law (RULE_ENFORCEMENT row [5]) is a typecheck requirement.
- Packages: `effect` (core, v3-held); the native-preview tsgo compiler as the canonical typecheck; the Effect-language-service-aware tsgo runner beside the native binary; stable `typescript` as the peer-bridge smoke.
- Growth: the v4 line lands as a runtime swap only when stable and peer-aligned across the satellite set; a new strictness knob lands as one `tsconfig.base.json` row inherited by every domain.
- Boundary: the v3-to-v4 jump is the one runtime axis held open; no owner imports a v4-only path under the held line, and no satellite is admitted whose peer range excludes the held core; package pins live only in the workspace catalog and never in this prose; `tsconfig.base.json` lives at the branch root and each domain extends it, registering a `references` row in the root solution `tsconfig.json` — four divergent configs is loose-config-spam one level up from schemas, the named defect.

## [5]-[ADMISSION_DOCTRINE]

- Owner: the admission discipline that gives every catalog package a real owner across the four domains — the architecture-altitude statement of the every-admitted-package-owns-a-page law, with the provider-vocabulary collapse that keeps a multi-provider concern one polymorphic axis rather than parallel services.
- Cases: the full Effect package set is admitted under the held v3 line, and each admitted package carries a page-to-catalogue row on its owning charter ADMISSIONS_RECORD; the AI surface — anthropic, openai, google, amazon-bedrock, openrouter — lands as one closed `Schema.Literal` provider axis on `ActivityOwner` (`services` durable-execution.md, the sole declaration site of the `AiProvider` literal), so the multi-provider concern is one durable activity with a provider row, never five parallel `Effect.Service` owners; the export-codec concern — exceljs, papaparse, jspdf, jszip — lands as one `AssetTransfer` codec axis on `services` persistence.md; the superseded merged cluster-workflow package is deleted from the catalog, its concern covered by the admitted cluster-plus-workflow split; `oauth4webapi` is void, superseded by `arctic` as the single OAuth owner; `@effect/printer-ansi` is void, the CLI renderer being `@effect/cli`'s own.
- Entry: the provider axis is the architecture-altitude instance of the one-owner-per-axis law — provider arity lives in the literal vocabulary, never in a name, so adding a sixth provider is one literal and one Layer-selection arm; the export-codec axis is the second architecture-side instance, format arity living in one `Schema.Literal` rather than four sibling exporters; each is one row of the admission-honesty closure grounding an admitted-but-unconsumed package into a real owner. The `AiProvider` literal is declared once in `services` durable-execution.md and referenced here as settled vocabulary per the prose-budget law.
- Packages: the AI set (`@effect/ai` and the five provider satellites, each exporting a `LanguageModel` Layer); the export-codec set (`exceljs`/`papaparse`/`jspdf`/`jszip`); `@effect/cli` as the typed-command binding; the cluster-plus-workflow successor pair.
- Growth: a new AI provider lands as one literal on the provider axis and one Layer arm; a new export codec lands as one literal on the asset-transfer axis; a new admitted package lands as one ADMISSIONS_RECORD row bound to its owning page, or the row is removed.
- Boundary: the AI activity body and the export-codec binding are owned for mechanics by their `services` pages and consumed here only as the architecture-altitude admission-honesty statement; a provider modeled as a parallel service rather than a literal-axis row is the named defect; package versions live in the workspace catalog and never in this prose.

## [6]-[ANTI_SPAM_AND_COLOCATION]

- Owner: the two cross-package laws that defeat the named TS defect classes — the anti-spam cluster law (every literal-vocabulary owner declared once) and the altitude-co-location law (co-location follows altitude and outbound-effect surface, never read-dependency).
- Cases: the anti-spam cluster law — every literal-vocabulary owner is declared once as a `Schema.Literal` union plus a behavior `Record`, never a parallel const enum, never a fresh Schema per shape, and never a proliferated one-off branded type; a derived projection brands/filters off the one owning Schema, never a sibling Schema; the `region-map/owner-symbols.md` registry is the single declaration site, so `AiProvider` lives once in `services` and the four prior geo aliases collapse into one `GeoSeriesLayer` union owner in `ui`. The altitude-co-location law — two owners share a page only when they sit at the same `[ONE FORM]` altitude and the same outbound-effect surface; a read-dependency between owners never forces co-location, so `AvailabilityStore` (a transport-free fold in `projection`) and `CommandGateway` (a dialing dispatch surface in `interchange`) live in different domains even though the gateway READS the fold — co-locating them leaks `@connectrpc/connect-web` into the platform-neutral fold interior. The same law homes `CommandGateway` in `interchange` rather than the fold domain: dialing dispatch is its outbound-effect surface even though it reads the availability fold.
- Packages: `effect` for the `Schema.Literal` + `Record` vocabulary form; the folder-scoped ESLint `no-restricted-imports` strata for the cross-domain altitude separation inside the one package.
- Growth: a new literal vocabulary lands as one `Schema.Literal` owner and one `owner-symbols.md` row; a new owner co-locates only with a same-altitude same-effect-surface sibling, otherwise it homes in the domain its outbound effect surface dictates.
- Boundary: a literal vocabulary declared in two places is the named const/type-spam defect; a fresh Schema per shape where a derivation off one owner suffices is the named loose-schema-spam defect; a barrel re-export file or a thin wrapper layer is the named wrapper defect; co-location by read-dependency rather than altitude is the named co-location defect this law deletes.

## [7]-[RULE_ENFORCEMENT]

- Owner: the mechanical-guard table — each named TS defect class is caught by exactly one mechanical guard at build or typecheck time, never a prose review check. The single-package collapse makes every import guard a folder-scoped `no-restricted-imports` rule rather than a cross-project tag edge, and the intra-neutral co-residence of interchange+projection means the gateway-reads-fold seam carries no stratum separation, so the guard set carries the residual mechanical guards explicitly.
- Cases: each row pairs a defect class with the single mechanical surface that fails the build when it occurs.
- Packages: the ESLint flat-config `no-restricted-imports` rule keyed by folder glob for the stratum and connect-ban import guards; tsgo for the typecheck guards; `package.json` subpath `exports` for the subpath guard; `effect` `Match.tagsExhaustive` for the fault guard.
- Growth: a new defect class lands as one guard row bound to one mechanical surface; a guard expressible only as prose is the named un-enforceable defect.
- Boundary: a guard that lives only in prose is not a guard; every row names the file or tool that fails the build, never a reviewer obligation.
