# Cross-Shape Metadata Algebra and Universal Admission

[CLOSED_SHAPE_FAMILY]:
- The generated-owner population is one closed discriminated family, not a set of independent generators: a keyed value object is a sibling case of a keyed smart enum under a shared `Keyed` parent that owns key type, validation-error type, the to-key conversion delegate and expression, and a weakly-typed key extractor; that parent sits beside the complex value object, the keyless smart enum, and both union shapes under one sealed metadata union. Boundary code that must branch on what kind of generated owner a type is dispatches through this union's exhaustive case set, never through reflection type-tests against the owner — a value object's identity-as-a-shape is one arm of a total match, recoverable without naming its concrete type.
- The metadata family describes itself with the same machinery it generates for domain owners: the union root and its keyed sub-union each publish a self-metadata entry enumerating their own case types, so the closed-family discipline is total — the introspection layer is its own first instance, and a fold over shape kinds is total against a compiler-checked case list rather than an open type switch that silently drops a future kind.
- Shape-kind dispatch ships in four forms on the metadata owner — a `void` form, a result-returning form, each with a state-threaded overload that takes a context value `where TState : allows ref struct`, plus a constant-arm `Map`. A ref-struct context threads a span or a stack-only accumulator through the case arms with zero capture, so a single span-backed buffer drives shape-polymorphic projection across the whole owner population without a closure allocation.
- The keyed sub-union's two arms are not symmetric in optionality: the smart-enum arm's from-key conversion delegate, conversion expression, and validating key extractor are non-nullable (a smart enum always converts a key to a member), while the value-object arm makes all three nullable (a factory-skipped owner nulls them out). A fold that reads the from-key route must treat the value-object arm as partial and the smart-enum arm as total — the nullability is the shape-level encoding of "this owner can be constructed from its key" versus "this owner's construction was cut."

[UNIVERSAL_EGRESS_CONTRACT]:
- One covariant single-method egress interface — `ToValue()` returning the carried value, declared `out` over a `notnull, allows ref struct` payload — is the universal escape hatch shared by every keyed shape: the keyed-value-object marker and the smart-enum marker both derive from it, so any container, comparer, or serializer constrained on the egress interface operates identically over value objects and smart enums with no shape awareness. The `out` variance erases the key's exact type at the constraint: an owner egressing a derived reference key satisfies a consumer constrained on the egress of any base it widens to, so one consumer arrow written against the egress of a base vocabulary type absorbs every owner whose key is a subtype without per-owner specialization.
- The container-level identity comparer is generic over the egress interface, not over any one owner: `where T : IConvertible<string>` admits every string-keyed value object and every string-keyed smart enum to the same six prebuilt comparer instances, which compare two owners by egressing both keys and applying the chosen string policy. A case-sensitive index over case-insensitive owners, or a culture-ordered view over ordinal owners, is one static field reference — and the same field works across both shapes because the constraint is the egress, never the owner.

```csharp
public static FrozenDictionary<T, V> ByKey<T, V>(IEnumerable<(T, V)> rows)
   where T : IConvertible<string> =>
   rows.ToFrozenDictionary(r => r.Item1, r => r.Item2, StringKeyedObjectComparer<T>.OrdinalIgnoreCase);
```

[UNIFIED_ADMISSION_ARROW]:
- Admission is one static-abstract arrow over the egress family's factory face, identical for both shapes. The trinary factory interface carries the sole `static abstract Validate(value, provider, out item)` returning the typed error; the smart-enum interface derives from that exact trinary factory, so a smart enum is an object factory by type, validated through the same `Validate` signature a value object exposes. One generic admitter constrained on the trinary factory therefore admits a value object or a smart enum with no overload, no shape branch, and the typed error preserved to a single rail projection.
- The package ships the arrow already inlined: a static invoker holds aggressively-inlined generic `Validate` thunks in three key shapes — a `where TKey : notnull, allows ref struct` form, a `ReadOnlySpan<char>` form, and parse/try-parse forms over the standard parsable interfaces — each a one-line forward to the owner's static-abstract member. Boundary code reusing these thunks buys the package's exact dispatch instead of re-deriving a per-owner bridge; the ref-struct key constraint is what lets a UTF-8 deserializer transcode into a char span and admit allocation-free across both shapes.
- The factory marker splits into a bare single-parameter face and the trinary validating face; the bare face exists only to let `where TValue : notnull, allows ref struct` flow without naming the error type, and the trinary face adds the validating method. A generic kernel that only needs "this can be built from a value of type `TValue`" constrains on the bare face and stays error-type-agnostic; one that performs admission constrains on the trinary face and recovers the typed error.

```csharp
public static Fin<T> Admit<T, TKey, TError>(TKey key, IFormatProvider? provider = null)
   where T : IObjectFactory<T, TKey, TError>
   where TKey : notnull, allows ref struct
   where TError : class, IValidationError<TError> =>
   StaticAbstractInvoker.Validate<T, TKey, TError>(key, provider, out var item) is { } error
      ? Error.New(error.ToString() ?? "<message-a>")
      : FinSucc(item!);
```

[CONVERSION_METADATA_FUSION]:
- One concrete conversion-route record — owner type, key type, validation-error type, a from-key-via-constructor expression, and a from-key-via-factory expression — is the single reified shape a binder or serializer reads, and it is computed by fusing two independent sources: the keyed metadata's key-typed conversion and the owner's object-factory list. The fusion is conditional on type identity: the keyed metadata's conversion expressions are folded in only when the keyed key type equals the selected factory's value type, otherwise the route carries only the factory's own constructor expression. A value object whose alternate factory admits a different value type than its key therefore yields a conversion route that constructs through the alternate value, not through the key — the binding shape follows the matched value type, not the declared key.
- Object-factory selection is last-write-wins under a caller-supplied predicate: the fusion picks the last metadata row satisfying the filter, so when several factories can match one wire concern, attribute declaration order is the disambiguator. The analyzer deduplicates only the exclusive claims — at most one persistence owner, at most one model-binding owner, no overlapping serialization frameworks — leaving custom-filter ambiguity to resolve by source order.
- The from-key route is selected by the shape arm, not by a flag: a fold over the keyed sub-union returns the constructor-bypass expression plus the validating-factory expression for the value-object arm, and a null constructor-bypass plus the validating expression for the smart-enum arm. Persistence materialization reads the constructor-bypass slot to rehydrate already-admitted truth without re-validation; a smart enum has no such slot because its members are interned, so its rehydration is always a validating key lookup — the absence is the shape's encoding of "members are not reconstructed, they are resolved."

[SHARED_CONVERTER_CHASSIS]:
- A single generic component-model converter type serves every keyed shape; it is closed over owner, key, and error and routes both directions through the universal contracts. Inbound from the key calls the owner's static-abstract `Validate` and throws a format exception carrying the flattened error on rejection; inbound from any other source delegates to the key's own converter and then re-enters `Validate`; outbound to the key egresses through the universal `ToValue()` and, for any further destination, hands that egressed key to the key's own converter. The converter is therefore transitive through the key's converter in both directions and shape-blind — a value object and a smart enum reuse the identical converter body.
- The converter encodes the struct-null contract structurally: converting null to a class owner yields null, converting null to a struct owner throws not-supported. Wire exposure of a struct owner without the converter-generating integration assembly is worse than a class owner's silent empty object — the reflection serializer zero-initializes the struct and sets nothing, manufacturing exactly the unvalidated default the compile-time gate forbids, with no exception. The converter gate is the only thing standing between a struct owner and serializer-driven admission bypass.

[CROSS_SHAPE_COMPOSITION_PRESSURE]:
- The universal egress is the seam to compose shapes without nesting their generators: an owner keyed by another generated owner reaches its full derived surface only across an assembly boundary, because a generator cannot observe another generator's output in the same compilation. The shape-agnostic alternative composes at the boundary instead of in the declaration — a value object carrying a domain string key, with a separate smart enum or value object validating that key's vocabulary, fused through one admission arrow that runs the inner admission then the outer, keeps every derived surface intact intra-assembly and pays one extra admission hop instead of an under-derived owner.
- Because admission, egress, conversion, and identity are all expressed against the shared contracts rather than against either concrete shape, a domain that holds a mixed population of value objects and smart enums needs exactly one boundary layer: one admitter generic over the trinary factory, one identity comparer generic over the egress, one conversion-route reader over the fused metadata. Writing a value-object boundary and a parallel smart-enum boundary is the rejected duplication — the shapes were already one dispatch surface at the contract level, and the second boundary re-fragments what the metadata family unified.
- The weakly-typed key bridge on the shared keyed metadata does not fail fast on a foreign key: it pattern-matches the boxed key and falls through to the default, so a struct-keyed owner degenerates a wrong-typed key into validating the zero value and a class-keyed owner into the null path. Shape-polymorphic dispatch that must reject mismatch unrepresentably routes through the strongly-typed factory arrow, never the weak metadata bridge — the weak route trades type safety for shape-erased reflection and admits mismatch as zero-value or null on a permissive invariant.
