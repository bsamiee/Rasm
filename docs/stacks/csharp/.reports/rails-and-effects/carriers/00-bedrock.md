# Carrier Core

[CARRIER_SELECTION]:
- `Fin<A>` is the concrete result carrier for computations that either succeed or fail with a typed `Error`. It is structurally `Either<Error,A>` but with the fail-side fixed to the `Error` hierarchy, making it the narrowest carrier when the failure domain is exception-capable and must compose with `Eff`/`IO` lifts. `Option<A>` is the absence carrier and carries zero failure semantics — it answers "present or not" with no failure-side information. `Validation<F,A>` is wider than `Fin` because it accumulates failures across independent checks using the applicative surface rather than short-circuiting; it is the correct choice when all failure paths must be collected before reporting. `Either<L,R>` is the most general form: the fail-side is unrestricted, enabling it to span protocol errors, domain codes, or any non-`Error` type as the left; it requires an explicit `L -> Error` bridge before entering an `Eff` chain.
- Narrowest-carrier law: admit `Option` when absence is the only distinguishing fact; promote to `Fin` when the caller must know why it failed; promote to `Validation` only when independent failures must accumulate; stay at `Either` only when the left type is not `Error` and no `Eff`-level convergence is needed on that path.
- `Fin<A>` exposes three `IfFail` overloads (`Func<Error,A>`, `A`, `Action<Error>`) and `IfSucc(Action<A>)` as terminal boundary operators; cross-rail projections are enumerated in full under `[CROSS_RAIL_PROJECTIONS]`.

[FIN_ALGEBRA]:
- `Fin<A>` is an abstract class with two sealed inner classes: `Fin<A>.Succ(A Value)` and `Fin<A>.Fail(Error Error)`, both declared with primary constructors. They are not record types — they decompile as primary-constructor classes, not positional records; `Deconstruct` is hand-declared on each. `IsSucc`/`IsFail` are public abstract booleans on the base. `SuccValue`/`FailValue` are `internal abstract` — inaccessible to consumers — and throw `InvalidCastException` on the wrong discriminant. Pattern matching uses the nested types directly (`case Fin<A>.Succ s` exposes `s.Value`; `case Fin<A>.Fail f` exposes `f.Error`) or the flat form `{ IsSucc: true }` using the public abstract property.
- Construction: `Fin.Succ<A>(A)` and `Fin.Fail<A>(Error)` are static methods on the non-generic `Fin` class; `Fin.Fail<A>(string)` wraps the string in `Expected`. `Fin<A>` declares implicit operators from `A`, `Error`, `Either<Error,A>`, `Pure<A>`, and `Fail<Error>` — assigning any of these directly to a `Fin<A>` variable requires no explicit constructor call.
- The failure-side algebra on `Fin<A>`: `MapFail(Error -> Error)` maps the failure without changing the success type; `BindFail(Error -> Fin<A>)` delegates to `BiBind` with the succ arm as identity; `BiBind<B>(A -> Fin<B>, Error -> Fin<B>)` branches on both cases; `BiMap<B>(A -> B, Error -> Error)` maps both sides but keeps the fail-side pinned to `Error`.
- `Fin<A>` has two structurally distinct combination behaviors. `Bind` is sequential and fail-fast: on `Fail`, the continuation is skipped. `Apply` (the applicative surface) accumulates: when both `mf` and `ma` are `Fail`, the implementation produces `Fail(mf_error + ma_error)`. In an N-ary curried application chain using `op_Multiply` (`*`), step 1 where `mf` is `Succ` and `ma` is `Fail` passes `ma`'s error forward as the new `mf` failure; step 2 combines that accumulated error with the next `ma` failure. The result of an N-ary curried chain carries all N independent failures combined left-to-right via `Error.Combine`. `SemigroupK<Fin>.Combine` (the `+` operator on `K<Fin,A>`) differs from `Apply`: when `lhs` is `Succ`, it returns `lhs` without examining `rhs`; when both are `Fail`, it combines their errors. `Alternative.Choose` (`|` operator, backed by `Fin<A>.Empty` = `Fail(Errors.None)` as the `MonoidK` identity) short-circuits: left `Succ` is returned immediately; left `Fail` returns the right operand without combining errors. `op_True`/`op_False` back `||` short-circuit coherence.

```csharp
// N-ary Apply: all three Fail errors accumulate
Fin<Func<string, Func<bool, int>>> f0 = Fin.Fail<Func<string, Func<bool, int>>>(Error.New("fn"));
Fin<string> arg1 = Fin.Fail<string>(Error.New("a1"));
Fin<bool> arg2 = Fin.Fail<bool>(Error.New("a2"));
Fin<int> result = f0 * arg1 * arg2;
// result: Fail(ManyErrors([Error "fn", Error "a1", Error "a2"]))

// SemigroupK.Combine: both Fail → merged error; left Succ → left wins immediately
Fin<int> combined = Fin.Fail<int>(Error.New("x")) + Fin.Fail<int>(Error.New("y"));
// combined: Fail(ManyErrors([Error "x", Error "y"]))
```

- `Fin<A>` has `Traverse<F,B>(A -> K<F,B>) -> K<F,Fin<B>>` and `TraverseM<M,B>(A -> K<M,B>) -> K<M,Fin<B>>` for effect sequencing. `FinExtensions.Partition<F,A>(K<F,Fin<A>>)` splits any foldable of `Fin<A>` into `(Seq<Error> Fails, Seq<A> Succs)`; it operates on a wrapping foldable (e.g., `Seq<Fin<A>>`), not on a `Fin<A>` instance method.

[OPTION_ALGEBRA]:
- `Option<A>` has `op_Implicit(A?) -> Option<A>` (delegates to `Prelude.Optional` — null folds to `None`) and `op_Implicit(Fail<Unit>) -> None`. `Case` is the pattern-matching property: yields `A` on `Some`, `null` on `None`. `{ IsSome: true, Case: var v }` is the CSP-compliant inline extraction form.
- `Prelude.Some<A>(A)` for the unconstrained overload delegates to `Option.Some(value)` which calls the internal `Option<A>(value)` constructor unconditionally — no null guard is present on the reference-type path. The nullable-struct overload (`A? value where A : struct`) does throw `ValueIsNullException` on `null`. `Prelude.Optional<A>(A?)` is the null-tolerant factory for any nullable interop and always folds null to `None`.
- `Option<A>` projections: `ToFin()` → `Fin<A>` with `Errors.None` on `None`; `ToFin(Error)` → `Fin<A>` with a caller-supplied error on `None`; `ToEither<L>(L)`, `ToEither<L>()`, `ToEither<L>(Func<L>)` — three overloads for the `None` left value; `ToValidation<F>(F)`, `ToValidation<F>()`, `ToValidation<F>(Func<F>)` — matching three overloads.
- `Option<A>.IfNone(A)`, `IfNone(Func<A>)`, `IfNone(Action)` and `IfSome(Action<A>)`, `IfSome(Func<A,Unit>)` are the terminal operators at the boundary where `Option` exits the rail. These are the only points where `None` produces a default — never use `.Value` directly.
- `Option<A>.BiMap<B>` has two overloads with asymmetric null behavior. `BiMap<B>(Func<A,B> Some, Func<Unit,B> None)`: the IL calls `Check.NullReturn<B>` on the raw `B` returned by whichever arm executes; `Check.NullReturn` raises `ResultIsNullException` if that `B` is null, then converts the non-null `B` to `Option<B>` via the implicit operator. A null-returning arm on this overload throws. `BiMap<B>(Func<A,B> Some, Func<B> None)`: the IL calls `op_Implicit<B>` directly without any null guard; a null `B` from either arm passes into `Prelude.Optional`, which folds it to `None`. The distinction is not syntactic — the `Func<Unit,B>` overload is null-unsafe in the dangerous direction (throws on a null value that the `Func<B>` overload would silently fold), and the two overloads are therefore not interchangeable. `BiBind<B>(Func<A,Option<B>>, Func<Option<B>>)` carries no null guard — both arms return `Option<B>` directly.
- `OptionExtensions.Somes<A>(IEnumerable<Option<A>>)` and `Somes<A>(Seq<Option<A>>)` extract all `Some` values in order.

[VALIDATION_ALGEBRA]:
- `Validation<F,A>` imposes `Monoid<F>` at its construction surface: `Validation.Success<F,A>`, `Validation.Fail<F,A>`, and `Validation.Empty<F,A>()` all carry `where F : Monoid<F>`. `MapFail<F1>` and `BindFail<F1>` also carry `where F1 : Monoid<F1>` on the result type. The `Semigroup<F>` constraint appears only on the static extension operators (`Apply`, `Action`, `CombineFirst` via `SemigroupInstance<F>`) — the minimum for accumulation on the extension path. The monoid identity is load-bearing: `Validation.Empty` calls `F.Empty` to construct the initial fail value, and `MonoidK`/`Alternative.Empty` resolve via `MonoidInstance<F>.Instance` at runtime with a `NotSupportedException` fallback when `F` is not monoid-capable. The root constraint is therefore `Monoid<F>` for the full construction and projection surface; `Semigroup<F>` alone is insufficient.
- Accumulation happens through the applicative surface, not through `Bind`. `Bind(A -> Validation<F,B>)` is sequential and fail-fast. The collect-and-accumulate surface is the `&` operator family: `K<Validation<F>,A> & K<Validation<F>,A>` delegates to `CombineI`, returns `Validation<F,Seq<A>>`, collects both success values into a `Seq` when both succeed, and combines failure values via `SemigroupInstance<F>` when either fails. `Combine<F,A>(K<Validation<F>,A>, K<Validation<F>,A>) where F : Semigroup<F>` delegates to `CombineFirst` — it is the named form of `SemigroupK.Combine` (the `+` operator), not of `&`: both-success keeps the first success value and returns `Validation<F,A>` (not `Seq<A>`); both-fail combines failures. The result type and success semantics differ — `&` widens to `Seq<A>` and collects both; `Combine`/`+` keeps first and stays at `A`. LINQ query expressions on `Validation` are always sequential — accumulation requires explicit `&`/`*`/`Apply` chains.
- `CombineI`, `ApplyI`, and `CombineFirst` are `internal` — the caller-facing accumulation surface is `&` and `*` (via `K<Validation<F>,Func<A,B>> * K<Validation<F>,A>`). The fold-friendly `public` overload is `CombineI<F,A>(this Validation<F,Seq<A>>, Validation<F,A>, Option<SemigroupInstance<F>>)` which appends a single result into an accumulating sequence; it is `public` and enables a `Seq.fold`-based collection pattern.
- `Validation<F,A>.MapFail<F1>(F -> F1) -> Validation<F1,A> where F1 : Monoid<F1>` changes the failure type, unlike `Fin` where the fail side is pinned to `Error`. The instance `BiMap<F1,A1>(Func<F,F1> Fail, Func<A,A1> Succ)` constructs the target cases directly without going through `BiBind`; the `Bifunctor<Validation>` trait's `BiMap` delegates to the instance `BiMap`. `BiBind` backs `Bimonad.BindFirst` and `BindSecond`.
- `Validation<F,A>.BiBind<F1,A1>(Func<F,Validation<F1,A1>>, Func<A,Validation<F1,A1>>)` carries no compile-time constraint on `F1`. `BindFail` is the conservative form — it delegates to `BiBind` with the success arm as identity and requires `F1 : Monoid<F1>`. Use `BiBind` for structural type-change where accumulation semantics are not required; use `BindFail` when the result must participate in further `&`/`*` accumulation.
- `ValidationExtensions.ToFin<A>(Validation<Error,A>) where F = Error` is constrained to `F = Error` — it is the convergence point after accumulation where the result enters a sequential `Eff`/`IO` chain. `Validation<F,A>.ToOption()` drops all failure information; `Validation<F,A>.ToEither()` preserves both cases as `Either<F,A>`.
- `Validation<F,A>` has `SelectMany` (enabling LINQ query syntax) and `Guard` binding via `ValidationGuardExtensions`. `ValidationExtensions.Fails<F,S>(IEnumerable<Validation<F,S>>)` and `Fails<F,S>(Seq<Validation<F,S>>)` extract failure values as a flat projection; there is no symmetric `Succs` on `Validation` through this surface — contrast `FinExtensions.Partition` which returns both `Fails` and `Succs` as a tuple.

[ERROR_HIERARCHY]:
- `Error` is a non-sealed `public abstract record` — user-defined subtypes are permitted and are a documented extension point. The built-in concrete subtypes are `Expected` (message, code, `Option<Error>` inner — a positional record), `Exceptional` (message, code, plus an optional live `Exception` value — a positional record with serialization-safe null handling), and `ManyErrors` (a sealed record carrying `Seq<Error>`). `BottomError` is a sealed record that inherits from `Exceptional` — not a sibling of `Expected`/`Exceptional`/`ManyErrors` but a specialization of `Exceptional` representing an unevaluable expression state.
- `Error.New(string)` → `Expected`; `Error.New(Exception)` → `Exceptional` (with special handling for `WrappedError*Exception` round-trips, `TaskCanceledException` → `Errors.Cancelled`, `TimeoutException` → `Errors.TimedOut`, `AggregateException` → `ManyErrors.FromAggregate`); `Error.New(string, Exception)` → `Exceptional` with display-sanitized message; `Error.New(string, Error)` and `Error.New(int, string, Error)` compose causal chains via `Expected`. `Error.Many(params Error[])` and `Error.Many(Seq<Error>)` are smart constructors: empty input returns `Errors.None`; single-element input returns the scalar error unchanged; multi-element input returns `ManyErrors`.
- `Error.Combine(Error)` / `op_Addition` appends two errors: combining two scalar errors produces `ManyErrors(Seq(this, other))`; combining two `ManyErrors` flattens their sequences; combining a scalar with `ManyErrors` prepends or appends the scalar. `Errors.None` is the monoid identity — combining any error with `Errors.None` returns the non-empty error unchanged.
- `Error.IsExceptional` and `Error.IsExpected` are the structural discriminants. `Error.HasException<E>()` tests whether any nested `Exceptional` wraps a live exception of type `E` — returns `false` for deserialized instances whose live `Exception` has been lost. `Error.HasCode(int)` on the base `Error` type compares only the receiver's own `Code` field (`Code == code`); neither `Expected` nor `Exceptional` override it, so causal chains composed via `Error.New(string, Error)` or `Error.New(int, string, Error)` are not searched. `ManyErrors` overrides `HasCode` as `Errors.Exists(e => e.Code == code)` — a one-level scan that checks each direct member's `Code` property, not a recursive `HasCode` call, so `ManyErrors` members that are themselves `ManyErrors` are not descended into. `Error.IsType<E>()` tests for concrete subtype `E`. `Error.Filter<E>()` projects out only errors of type `E`, returning `Errors.None` when none match.
- `Error.Inner` is declared `Option<Error>` — it is `None` on `Expected` when no inner error was supplied, and resolves the inner exception to an `Error` on `Exceptional` via `Error.New` when the live `Exception` has an `InnerException`. `Error.Exception` is declared `Option<Exception>`: it returns `None` on `Expected` cases and `Some(ToException())` on `Exceptional` cases — it is never a raw nullable field; both are typed `Option<T>` carriers. `Error.Head` / `Error.Tail` decompose `ManyErrors` in list order; for a scalar `Error`, `Head` returns self and `Tail` returns `Errors.None`. `Error.IsEmpty` is `true` for `Errors.None` (zero-member `ManyErrors`).
- `Error.FoldM<M,A>(Error -> K<M,A>) -> K<M,A>` invokes `f` over each atomic error and combines via `M`'s monoid — processing continues even on sub-failure depending on `M`. `Error.ForAllM<M,A>(Error -> K<M,A>)` short-circuits on the first failure. `catchOf<E,M,A>` and `catchOfFold<E,M,A>` expose this distinction at the catch-handler level: `catchOf` calls `es.Filter<E>().ForAllM(Fail)` (sequential-short-circuit); `catchOfFold` calls `es.Filter<E>().FoldM(Fail)` (accumulating, starting from `M.Empty<A>()`). The distinction is invisible for a single-atom error; it matters when the caught error is a `ManyErrors` aggregate containing multiple instances of `E`.
- `Errors` static class: `Errors.None` is the empty error; `Errors.Bottom` is the `BottomError` singleton; `Errors.Cancelled`, `Errors.TimedOut`, `Errors.SequenceEmpty`, `Errors.ValidationFailed`, and `Errors.EndOfStream` are canonical domain codes. `ErrorCodes` mirrors these as integer constants for `HasCode` predicate matching. `Error.ToErrorException()` → `ErrorException`; `Error.Throw()` / `Error.Throw<T>()` escape the rail into exception flow — restricted to host-boundary adapters.

[GUARD_INGRESS]:
- `Prelude.guard(bool, Error)` and `Prelude.guard(bool, Func<Error>)` return `Guard<Error,Unit>` — a typed conditional that participates in query expressions via `FinGuardExtensions.SelectMany`, `EitherGuardExtensions.SelectMany`, and `ValidationGuardExtensions.SelectMany`. A `guard` in a `from`/`where` clause is the only idiomatic ingress form for boolean invariants at the top of a pipeline.
- `Prelude.guard<F>(bool)` is the `MonoidK`/`Alternative` form: it returns `Applicative.pure(())` on `true` and `Applicative.empty` on `false`. This overload applies to any monad with an `Alternative` instance — `Option`, `Seq`, `Lst` — and does not carry an error payload.
- `Prelude.guardnot(bool, Error)` is `guard(!bool, Error)` — the negation shorthand. Both forms generate `Guard<Error,Unit>` and convert to any rail via the appropriate guard-extension `ToFin`, `ToEither`, `ToValidation`, or `ToEff`.
- `Guard<F,A>` is never a return type; it is always immediately bound into the surrounding monad through a `SelectMany` overload and exits as the destination monad's type.
- `Prelude.Optional<A>(A?)` and `Prelude.Optional<A>(Nullable<A>)` are the null-tolerant ingress for `Option<A>`. `Prelude.Some<A>(A)` for the reference-type overload has no null guard in its code path — the null guard resides in the nullable-struct overload only. Conversion to `Fin`/`Either`/`Validation` from `Option<A>` uses the projection methods on the constructed `Option<A>` value.

[CROSS_RAIL_PROJECTIONS]:
- The full instance projection matrix: `Fin<A>` → `Option<A>` (`ToOption()`), `Either<Error,A>` (`ToEither()`), `Validation<Error,A>` (`ToValidation()`), `Eff<A>` (`ToEff()`). `Option<A>` → `Fin<A>` (two overloads), `Either<L>` (three overloads), `Validation<F>` (three overloads). `Validation<Error,A>` → `Fin<A>` (`ValidationExtensions.ToFin`); `Validation<F,A>` → `Option<A>` (`.ToOption()`) and `Either<F,A>` (`.ToEither()`). `Either<Error,A>` → `Fin<A>` (`FinExtensions.ToFin`).
- Projectability asymmetry: every rail projects to `Option` by discarding the failure side. Widening projections require supplying missing structure (`Option → Fin` needs an `Error`; `Either → Eff` needs an `L → Error` map). `Fin<A>.ToValidation()` exists as a direct instance projection and is constrained to `F = Error`; the resulting `Validation<Error,A>` is suitable for feeding into a multi-step accumulation chain.
- Natural transformations carry round-trip loss that explicit projections avoid. `Natural<Fin,Option>` discards the error: `Succ → Some(value)`, `Fail → None`. The reverse via `ToFin()` (zero-arg) maps `None` to `Fail(Errors.None)`. A `Fail` case survives both directions but with its original error replaced by `Errors.None`, making subsequent `HasCode` or `Is` predicates incorrect. `Natural<Fin,Eff>` preserves the error: `Fail → Eff.fail(error)`. `Natural<Fin,IO>` also preserves the error: `Fail → IO.fail(error)`. Any carrier-polymorphic pipeline using `Natural.transform<Option,M,A>` or `Natural.transform<Fin,Option,A>` generates `Errors.None`-tagged failures on the `None`/`Fail` path; when `None` must carry diagnostic identity, `option.ToFin(Error.New(...))` is always correct.
- Pattern matching: `Option<A>` uses `Case` (returns `object` — `A` on `Some`, `null` on `None`); `{ IsSome: true, Case: var v }` is canonical. `Fin<A>` uses the public nested types `Fin<A>.Succ` and `Fin<A>.Fail`; `{ IsSucc: true }` is the flat-match form using the public abstract `IsSucc: bool`. `Validation<F,A>` uses the public sealed records `Validation<F,A>.Success` and `Validation<F,A>.Fail` with their `Value` properties — `SuccessValue`/`FailValue` are `internal` and inaccessible; the correct form is `case Validation<F,A>.Success s` exposing `s.Value`. `Either<L,R>` uses `.Right` / `.Left` nested types with `{ IsRight: true }` guards.
