# Carrier-Polymorphic Trait Algebra

[CONSTRAINT_AS_CAPABILITY_CONTRACT]:
- The constraint chain selects the available power: a functor constraint gives mapping, applicative adds pure/apply/action, monad adds bind and recursion. An arrow bound only to the applicative constraint cannot sequence-with-dependency and is therefore structurally guaranteed to be accumulation-compatible and parallelizable — the constraint is the capability contract, not documentation, and weakening a constraint is a provable statement about what the body cannot do.
- The static-virtual trait members dispatch on the type argument at the call site with zero allocation and zero vtable indirection; a trait-constrained arrow is monomorphized per carrier at JIT specialization. The carrier type itself is the trait-implementing witness passed as the type parameter, never an instance.
- The bridge back from the erased kind is per-carrier: a polymorphic arrow returns the kind surface, the caller pins it with the carrier's own re-anchoring cast, and there is no universal erasure-removing cast — each carrier owns its own. Omitting the pin is the single most common carrier-polymorphic error: the value stays existentially kinded and refuses concrete-only operators. Nested stacks use a distinct lift/lower pair through the outer map for two-level values, and the resilient lowering rebuilds via an identity map instead of throwing when a hand-constructed kind value is genuinely foreign.

[RECUR_AS_DATA_DRIVEN_LOOP]:
- The monad trait exposes a carrier-level trampoline whose step function returns an effect producing a two-case signal — loop with a new seed, or done with a result — so each iteration is itself an effect in the carrier: the loop body can fail, branch, or short-circuit through the rail while the recursion stays bounded. The loop variable threads as data and the exit is data, never a break statement.
- Three sibling drivers form one family: the safe default, an unguarded variant for carriers already tail-safe, and a streaming variant (constrained to carriers with a lazy-enumerable natural transformation pair) that collects every intermediate loop value into the carrier — turning the trampoline into a streaming unfold rather than a single-result fold.

[LIFTED_CONTROL_FLOW]:
- Conditional execution consumes a predicate already in the rail: when/unless run an effect conditionally without materializing the boolean as a statement, and the lifted ternary returns an effect chosen by an effectful condition. The ternary's overload set spans the cross product of carrier-effect, pure-literal, and raw-IO branches, so a branch that is a constant or a bare effect lifts without explicit wrapping — the overload set absorbs the lift.
- The accumulate-while/until family replays a single effect, collecting results until the predicate flips — with the monadic-suffix forms taking an effectful predicate — giving the rail-native do-while whose continuation condition is itself an effect; the unbounded repeat exists for daemon-shaped effects.

[VARIADIC_APPLICATIVE_PRODUCT]:
- The applicative product ships curried and tuple-driven: the curried form peels one argument per apply across the arity ladder, and the tuple form takes a tuple of independent carriers (up to ten) plus an uncurried combiner in one call — internally currying once and threading apply across the elements, eliminating the curry ceremony while preserving the independent-effects semantics. The monadic-result twin flattens a combiner that itself returns an effect, bridging independent-accumulate into a final dependent step without an explicit bind.
- Because the tuple form derives every failure-combination from the carrier's apply, an accumulating carrier collects all failures and a short-circuit carrier reports the first — the same call site changes accumulation policy purely by changing the carrier type argument. Pre-built arithmetic lifts provide element-wise numeric combination inside any applicative carrier with no manual application of an operator function.

[LAZY_OPERAND_SELECTION]:
- The apply and choice traits carry deferred-operand overloads taking a memoized effect whose construction is forced on first read behind a compare-and-swap gate. The laziness pays off only for carriers whose combination can decide from the left operand alone — a collection carrier with an empty function side never constructs the right operand, and a choice falls to the alternative only when the left is empty. Failure-combining carriers force the memo unconditionally because accumulation needs both sides — pay-for-reached versus pay-for-all is a property of the carrier's combination logic, not a guarantee of the memo wrapper.

[ALTERNATIVE_AS_GRAMMAR]:
- The alternative trait — choice plus applicative with its own identity — is simultaneously the failure-fallback surface and a repetition grammar over any carrier with emptiness and choice: one-or-more, zero-or-more, separated and terminated repetition, and repeat-until-terminator (with twins that keep the terminator) are derived members expressed once against choice and pure, inherited by every alternative carrier. A backtracking parser, a non-deterministic search over a sequence carrier, and a first-success fallback over the absence carrier are the same code at three instantiations, and the repetition combinators stay stack-safe by threading the carrier's own trampoline.

[CAPABILITY_SIGNATURE]:
- A one-member capability witness per dependency makes the dependency set the type signature: an arrow constrained over exactly the capabilities it touches cannot smuggle unused capability past the signature, and adding a dependency is adding a constraint, not editing a container registration. The reader and state capabilities share the witness shape — environment ask/asks/local and state get/put/modify/gets — with the whole-value form derived from the projecting form applied to identity, so a carrier implements one primitive per axis and inherits the family.
