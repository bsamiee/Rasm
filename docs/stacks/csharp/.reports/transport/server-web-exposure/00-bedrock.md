# server-web-exposure bedrock

## server composition at app roots

- gRPC server exposure is two declarations at the app root: `AddGrpc(Action<GrpcServiceOptions>)` settles global policy once, `MapGrpcService<TService>()` binds each service onto endpoint routing; the returned `GrpcServiceEndpointConventionBuilder` is the per-endpoint convention seam (host filtering, metadata, web enablement) so per-service variance is conventions, never a second options site.
- Per-service override rides `AddServiceOptions<TService>(Action<GrpcServiceOptions<TService>>)` with inherit-unless-specified semantics carried by paired flags: setting `MaxReceiveMessageSize`/`MaxSendMessageSize` flips `MaxReceiveMessageSizeSpecified`/`MaxSendMessageSizeSpecified` even for a null value, so a service can explicitly LIFT a global cap by assigning null — clearing the flag restores inheritance. The flag pair is the only way to distinguish "unset" from "set to unlimited"; copying global values into per-service options by hand re-derives what the flags already encode.
- Defaults are policy rows to restate in the root policy value: receive cap 4_194_304 bytes, send cap absent, and a compression-provider registry pre-seeded with gzip and deflate at `CompressionLevel.Fastest` whenever the list is left empty — supplying any provider row suppresses both defaults, so a custom-codec root must re-add gzip explicitly or older peers lose a negotiable encoding.
- `ResponseCompressionAlgorithm` names the registry row used when the client's accept-encoding admits it; `ResponseCompressionLevel` tunes that row; compression is response-side negotiation, never a per-call branch. `EnableDetailedErrors` widens unhandled-exception projection into status detail text — a trusted-lane row only, since it leaks interior message content to every caller.
- `IgnoreUnknownServices` (default false) routes unmatched gRPC-content-type requests to a terminal unimplemented-status endpoint; true lets them continue down the middleware pipeline — the correct row only when a non-gRPC handler legitimately shares the gRPC content-type path space, otherwise contract drift surfaces as confusing 404s instead of the unimplemented status the client taxonomy expects.
- Three `MapGrpcService` ingress shapes serve one law: the generic overload activates a service type whose generated base carries the bind method; `MapGrpcService(ServerServiceDefinition)` and `MapGrpcService(Func<IServiceProvider, ServerServiceDefinition>)` mount dynamically composed definitions (`SuppressCreatingService = true` skips DI activation for these) — the definition route is the seam for contract surfaces assembled at runtime rather than generated at build.
- Stub generation is one item-metadata switch per contract file: `GrpcServices` = `Both` (default) | `Client` | `Server` | `None`; server-only packages compile without client stubs by declaring the row, never by post-build pruning.
- Server interceptors are option rows: `GrpcServiceOptions.Interceptors.Add<TInterceptor>(params object[] args)` with constructor arguments as data; global rows run before per-service rows; an interceptor type registered in the container resolves from it, otherwise it is activated per call with the supplied args — stateful interceptors therefore demand explicit container registration or their state silently resets every call.
- Transport prerequisites live on the endpoint row: the listener's `Protocols` selects `HttpProtocols.Http2` for plaintext trusted lanes (no ALPN), TLS endpoints negotiate via ALPN; gRPC exposure without an HTTP/2-capable row fails at the protocol layer before any service code runs.

## server call seam

- `ServerCallContext` is the one per-call capability surface: `Deadline` (already client-budgeted), `CancellationToken` (fires on client disconnect, cancellation, and deadline expiry — the only token to thread into effects), `RequestHeaders`, `ResponseTrailers`, `WriteResponseHeadersAsync(Metadata)` (one-shot early-header flush before first response message), `WriteOptions`, `UserState` (per-call bag shared with interceptors), `AuthContext`, and `GetHttpContext()` — the bridge to connection features when transport evidence (negotiated protocol, connection-level identity) is needed.
- Inbound `-bin`-suffixed headers are decoded to byte entries during `RequestHeaders` materialization — the server reads binary metadata as bytes without hand-decoding, mirroring the entry constructor contract that enforces the suffix for byte values and rejects it for strings.
- `Peer` degrades to the literal `"unknown"` on any non-IP transport because it is derived from the connection's remote IP; caller identity on local transports is connection-level evidence, never a call-context read — the kernel-credential mechanics live with the IPC topology law.
- Server streaming lifetime: a handler returning while a client is mid-read terminates the call cleanly; the handler observing `CancellationToken` is the drain contract — catching the cancellation to "finish the write" sends into a dead call and is the rejected form.

## structured faults in status details

- The fault contract is structured-detail-in-trailers, never code-plus-string: the suite declares one fault-detail wire message (its own contract file, evolving under descriptor law), serializes it into a trailer entry under a `-bin` key — canonically `grpc-status-details-bin` — and the server raises `RpcException(new Status(code, briefText), trailers)`; the four-argument exception constructors accept the trailer set directly, so fault projection is one expression at the boundary bridge.
- `Status.Detail` is a human-readable summary only; encoding machine-readable discriminants, retry hints, or field paths into the detail string is the named defect — string parsing at the consumer re-derives what the structured entry already carries, and the detail text is not part of any contract gate.
- The trailer-entry constructor enforces the `-bin` suffix for byte payloads, so a structured fault cannot accidentally travel as text; the client-side read and status taxonomy arrive settled — the server lane owns only the emission law: exactly one fault-detail family per suite, packed at exactly one projection point, with unhandled exceptions converging into the same family via the boundary bridge rather than leaking through `EnableDetailedErrors`.
- Fault details compose the extension slot law: a closed suite owns its detail cases as one message family; a foreign-extensible detail slot is the packed-any row of the contract vocabulary — the choice is contract design, not transport code.

## health service

- Health exposure is two rows: `AddGrpcHealthChecks(Action<GrpcHealthChecksOptions>)` and `MapGrpcHealthChecksService()`; registration installs the health service implementation as a singleton plus a health-publisher row, and pre-maps the empty-string service name to ALL registered health checks — the empty name is the whole-process answer every probe understands without configuration.
- Per-service health rows are predicate mappings: `options.Services.Map(name, ctx => ...)` filters the suite's health-check registrations by name and tags into a named gRPC health service; mapping is name-keyed and replace-on-remap, so each exposed service name is one declared row.
- The wire status fold is fixed: any matched unhealthy entry folds to NOT_SERVING; healthy AND degraded entries fold to SERVING (degraded still serves — clients needing degradation visibility read diagnostics signals, not the health protocol); zero matched entries fold to UNKNOWN; a Check against an unmapped service name fails with the not-found status, while the Watch path reports SERVICE_UNKNOWN instead — the two surfaces disagree by design, and probes must tolerate both spellings of "no such service".
- `UseHealthChecksCache = false` (default) executes the mapped health checks inline on every Check call — accurate but cost-bearing per probe; true serves the last publisher-cadence snapshot. The Watch stream splits the difference structurally: its FIRST write is replaced with a freshly computed result so a new watcher never acts on a stale snapshot, then subsequent updates ride the publisher cadence — cadence policy itself is runtime-owned and composed, never re-tuned here.
- Shutdown is a first-class health transition: on application stopping, active Watch streams are completed and a final NOT_SERVING is written to each watcher (suppressible via `SuppressCompletionOnShutdown`) — peers observe drain through the same channel they observe liveness, which is the readiness signal the attach choreography consumes; polling Check during shutdown races the listener teardown, watching does not.

## grpc-web translation at the server

- Server-side browser translation is one middleware plus per-endpoint consent: `UseGrpcWeb(new GrpcWebOptions { DefaultEnabled })` installs the translator; consent is endpoint metadata via `.EnableGrpcWeb()`/`.DisableGrpcWeb()` conventions on the mapped service, with `DefaultEnabled` as the blanket row — a detected grpc-web request without consent falls through untranslated and fails as a non-gRPC request, which is the diagnostic signature of a missing enable row.
- Detection is structural: POST plus a grpc-web content type selects translation; the request's Accept header selects the RESPONSE mode independently, defaulting to binary grpc-web when absent — request and response modes are decoupled, so a binary request may legitimately stream back base64 text.
- The translation trick is protocol spoofing: the middleware rewrites the request protocol to HTTP/2 and the content type to native gRPC (preserving any `+`-suffix subtype), lets the full gRPC server pipeline run unmodified, then on response start restores the original protocol and rewrites the content type to the negotiated web mode; trailers are written into the response body after the handler completes — the server pipeline is row-blind exactly as the client side is, and no service code can detect translation.
- Browser callers additionally require a CORS policy exposing the gRPC trailer-carried headers (`Grpc-Status`, `Grpc-Message`, encoding headers) since browsers hide non-safelisted response headers from scripts; the CORS row lives beside the enable convention at the app root. Stream-shape limits of the translated row arrive settled from the transport-row law.

## minimal-API seam and JsonPatch

- `JsonPatchDocument<TModel>` is a self-binding minimal-API parameter: the type carries its converter factory via its converter attribute (deserializes from the standard operations array) and implements the endpoint-metadata provider contract, auto-emitting accepts metadata for `application/json-patch+json` with the model type — patch endpoints get correct binding and schema metadata from the parameter declaration alone, with zero filter or convention code.
- The document's `SerializerOptions` property is the codec coupling point: operation values deserialize and convert through it, so it must be assigned the suite contract options or patch value conversion diverges from the suite codec on every converter-bearing type (temporal values foremost) — the silent-divergence trap is patches that apply cleanly in tests using default options and corrupt converter-typed properties in production.
- Apply is a three-overload law: `ApplyTo(target)` throws a patch exception carrying the failed operation and affected object on first failure; `ApplyTo(target, Action<JsonPatchError>)` converts per-operation failures into reported errors and continues — the rail-friendly shape, accumulating `JsonPatchError` rows (affected object, operation, message) into a typed validation outcome instead of an exception ladder; the adapter overloads swap path-resolution behavior and are the extension seam, not the default.
- The operation vocabulary is closed and typed at authoring: expression-tree builders (`Add`/`Remove`/`Replace`/`Move`/`Copy`/`Test`, each with positional list overloads) construct operations against the model's static shape, so a renamed property breaks patch construction at compile time; the untyped string-path document exists for relay scenarios where the server forwards patches it does not model.
- `Test` is the in-document concurrency guard: a patch carrying a `Test` of the expected version value fails atomically when the precondition no longer holds — optimistic concurrency travels INSIDE the patch rather than as a separate header, which keeps replayed patches self-guarding.
- Patch application is a boundary act: operations apply to a mutable wire-shape projection, and the mutated projection then re-enters domain admission as a whole value — applying patches directly to admitted domain owners would bypass admission and is the rejected form; the patch document is wire vocabulary, never interior vocabulary.

## suite contract merge and schema export

- The federated suite JSON law: each package ships ONE source-generated contract context owning its wire family; app roots merge per-package contexts once into the suite options via the resolver chain — `JsonTypeInfoResolver.Combine(params ...)` flattens nested combinations into a single ordered chain, and `JsonSerializerOptions.TypeInfoResolverChain` is the equivalent mutable view bound to the options instance.
- Chain order is first-non-null resolution, and the merge law forbids exploiting it: a type resolvable by two contexts is a CONFLICT, not a fallback — the root proves disjointness at startup by probing each advertised contract type against each context individually and rejecting any type with two owners; after the probe, `MakeReadOnly()` freezes the suite options and `IsReadOnly` is the audit bit. Order-dependent resolution that "works" is a latent contract fork: whichever package reorders the chain re-decides another package's wire format.
- `WithAddedModifier(Action<JsonTypeInfo>)` is the cross-cutting seam over a merged chain — one modifier enforcing suite invariants (naming policy conformance, required-property posture) across every package's contract without touching any package's generator; modifiers stack without re-wrapping.
- Per-package payload opacity: a routing envelope carries foreign package payloads as raw JSON (element or bytes), decoded only by the owning package's context at the consuming edge — the root never merges payload SHAPES, only resolver OWNERSHIP, so package contract evolution never recompiles the envelope or any non-owning package.
- Schema export is contract publication: `options.GetJsonSchemaAsNode(type, JsonSchemaExporterOptions)` (or per-`JsonTypeInfo`) emits the schema the FROZEN suite options actually serialize — exporting from anything but the merged, frozen options publishes a schema no wire byte obeys. `TreatNullObliviousAsNonNullable` tightens unannotated reference types for consumers that enforce nullability; `TransformSchemaNode` receives each schema node with its exporter context and is the injection point for vocabulary stamps and constraint tightening during export, not after.
- Exported schemas join the same evolution discipline as binary contracts: the schema node set is hashed into the suite's contract checksum material and judged by the same injected additive-vs-breaking classifier verdict pattern — JSON surfaces and binary surfaces evolve under one gate with two projections.
- One codec per surface: every wire surface in the suite declares exactly one codec (binary contract, suite JSON, or a framing codec row); a second codec observed for the same surface is a conflict receipt at composition, never a runtime fallback — fallback codecs mask drift by silently re-encoding what the primary rejected.

## causal-stamp serialization precision

- The cross-process causal stamp serializes as TWO JSON number fields — physical (epoch milliseconds) and logical (counter) — never as one packed 64-bit value: JSON numbers interoperate at IEEE-754 double precision, exact only to 2^53, and a packed physical-shifted-plus-logical encoding exceeds the mantissa immediately while epoch milliseconds alone sit three orders of magnitude under the bound for centuries.
- The split form stays precise because the logical component resets to zero whenever the physical component advances (advance algebra arrives settled from the diagnostics spine) — the logical counter only grows within one millisecond of skew, bounding it far below any precision edge; an envelope whose logical component grows monotonically across physical advances indicates a broken advance composition, making the serialized shape itself a drift detector.
- Binary contract projection of the same stamp uses two integer fields under identical reset semantics — one stamp vocabulary, two codec projections, zero re-derivation at any consumer.

## divergent

- Server-health-web fusion — one exposure record per app root: `(service binding rows, global option row, per-service override rows, interceptor rows, health mapping rows, web-enablement row, fault-detail family)` materialized by one fold into `AddGrpc`/`AddServiceOptions`/`MapGrpcService`/`MapGrpcHealthChecksService`/`EnableGrpcWeb` calls — the diff of exposing a new service is one binding row plus at most one health mapping row. The fold makes the second-configuration-site defect structurally impossible: nothing outside the record can reach the options, and the record is auditable as data. Rejected forms this forecloses: per-service `AddGrpc` scatter, health predicates inline at map sites, interceptors registered both globally and per-service for the same concern (the order trap: the global row always runs first, so a per-service row can never wrap a global one), and grpc-web enabled by blanket default when only one endpoint faces a browser.
- Suite-contract-merge as the one JSON legislation: resolver disjointness probe, freeze, schema export, checksum gate, and stamp-precision law compose into a single root ceremony with one receipt — `(contexts merged, conflicts: none, frozen, schema hash)` — emitted once at boot; every package's JSON contract is thereafter immutable-by-construction for the process lifetime. The conflict probe doubles as the suite's only defense against the quiet failure mode of reflection fallback: a chain accidentally containing a reflection resolver resolves EVERY type, so the probe also asserts that no chain member resolves a type outside its declared family — catching the unbounded resolver by its over-breadth rather than by name.
- JsonPatch as recorded intent, not computed diff: the patch document is constructed where the intent occurs (typed builders at the mutation site), travels with its own concurrency guard (`Test` row), applies through the error-accumulating overload onto a wire projection, and re-enters admission whole. This ordering makes patch generation a free byproduct of the write path and makes a JSON-diff dependency structurally unnecessary — diffing two states reconstructs WHAT changed but destroys WHY and HOW (a move diffs as remove-plus-add; a list reorder explodes into per-index replaces), producing larger patches with weaker semantics than the recorded operations. The named rejected form is the state-diff endpoint: accept two documents, compute operations, apply — it doubles payload, loses move/copy/test semantics, and re-derives at the consumer what the producer already knew.
