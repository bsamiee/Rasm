# Spectral-Eigen Deepening

[CONGRUENCE_CORRECTNESS]:
- `Cholesky<T>.Factor` stores the lower triangle L in column-major layout: column `i`, rows `j > i` hold `L[j,i]`; the upper triangle is zeroed. `Cholesky.Solve(B)` calls `CholeskySolveFactored`, which performs the combined forward pass (`L·y = B`) and backward pass (`Lᵀ·x = y`) in a single `POTRS`-equivalent routine, returning `(L·Lᵀ)⁻¹·B`. Any congruence step requiring `L⁻¹·A` alone cannot use `Cholesky.Solve`; it must use `chol.Factor.LU()` directly.
- The correct two-step congruence: (1) `Y = chol.Factor.LU().Solve(A)` solves `L·Y = A`, giving `Y = L⁻¹·A`; (2) `Z = chol.Factor.LU().Solve(Y.Transpose()).Transpose()` gives `Z = L⁻¹·A·L⁻ᵀ`. Calling `chol.Factor.Transpose().Solve(A)` routes through `LU(Lᵀ)` and solves `Lᵀ·X = A`, yielding `L⁻ᵀ·A` — the wrong half-factor. `Matrix<T>` has no bare triangular solve; `.Solve` always routes through LU. For lower-triangular L, the LU factorization is O(n²) since L is already triangular.
- `DenseCholesky.Solve` throws `NotSupportedException("Can only do Cholesky factorization for dense matrices at the moment.")` when either the input or result is not `DenseMatrix`. The abstract type is `Matrix<double>`, so the failure is runtime, not compile-time.
- Cholesky's positive-definiteness check is `num > 0.0` strictly. Exactly-zero diagonal (rank-deficient B) throws `ArgumentException("Matrix must be positive definite.")`. Near-singular B (very small positive diagonal) passes and produces an ill-conditioned factor rather than throwing. The correct guard is a small `ε·I` shift before factorization for near-singular B; catching `ArgumentException` addresses only the indefinite case.

[GRAM_SCHMIDT_FAILURE_SURFACE]:
- `DenseGramSchmidt.Create` unconditionally clones the input (`(DenseMatrix)matrix.Clone()`) before factorizing — the `.Q` accessor returns this cloned, modified matrix. An extra O(nk) allocation is unconditional.
- `DenseGramSchmidt.Factorize` throws `ArgumentException("Matrix must not be rank deficient.")` only when column norm is exactly `0.0` after `Math.Sqrt(sum_of_squares)`. A near-zero column norm (small but nonzero) propagates silently: the column is divided by the near-zero value, producing a unit-norm vector with catastrophically amplified rounding noise. For LOBPCG block orthonormalization, every column norm must be checked against a domain-specific floor before calling `.GramSchmidt()`.
- `DenseGramSchmidt` performs single-pass MGS. For nearly-dependent columns arising in LOBPCG when the P block loses independence, a second manual pass is required: compute `.GramSchmidt().Q`, check the R diagonal via `gs.R.Diagonal().Any(v => Math.Abs(v) < floor)`, then call `.GramSchmidt()` again if triggered. `QR<T>.R` is lazy (`_lazyR = new Lazy<Matrix<T>>(FullR.UpperTriangle)`), so the diagonal access does not force allocation unless the check fires.
- `DenseQR.Create(matrix, QRMethod.Thin)` uses Householder reflections via `ThinQRFactor`. For an n×k block, the resulting `qr.Q` is exactly n×k. `GramSchmidt<T>` hardcodes `QRMethod.Full` in its base constructor, so `.GramSchmidt()` always produces an n×n Q even for non-square input — a difference in both stability (Householder vs. single-pass MGS) and allocation (O(n²k) vs. O(n²)). `QR(QRMethod.Thin)` is the correct orthonormalization surface for LOBPCG block columns.

[CONVERGENCE_ASYMMETRY]:
- `SymmetricDiagonalize` uses the Wilkinson relative convergence criterion: `|e[k]| ≤ Precision.DoublePrecision × num2` where `num2` is a running spectral-radius estimate. The iteration counter resets per outer `j` loop (per eigenvalue); exhaustion at 1000 steps for a single eigenvalue throws `NonConvergenceException`. Near-coincident eigenvalues (ratio ≈ 1) are the pathological trigger.
- The real nonsymmetric path (`NonsymmetricReduceHessenberToRealSchur(double[])`) has a bounded iteration count: `if (num10 >= 30 * order) throw new NonConvergenceException()`. The counter `num10` increments per outer step and resets only on a 2×2 block deflation; the ceiling is `30n` total steps across all sub-problems, not per-eigenvalue. A pathological 100×100 matrix exhausts 3000 steps and throws.
- The complex nonsymmetric path (`NonsymmetricReduceHessenberToRealSchur(Complex[])`) has no outer iteration bound. The `while (num >= 0)` loop terminates only on deflation via `num--` or `num -= 2`. Exceptional shift strategies fire at internal counters 10 and 20 but impose no ceiling. Silent non-termination is possible for complex matrices whose Schur form fails to deflate through the shift heuristics.
- Receipt consequences: `NonConvergenceException` is catchable for symmetric and real nonsymmetric inputs; the complex nonsymmetric path requires post-hoc Frobenius residual validation as the only termination signal. A nonsymmetric Evd receipt must include `‖A·V − V·D‖_F` compared against a policy threshold; without it, partially-converged eigenvalues are indistinguishable from fully-converged ones.
- `Evd<double>.Rank` uses `EigenValues[i].AlmostEqual(Complex.Zero)`, which expands to `|λ|² < DefaultDoubleAccuracy ≈ 1.11 × 10⁻¹⁵` because `ComplexExtensions.Norm(c)` returns `MagnitudeSquared`. The effective magnitude floor is `|λ| < 3.33 × 10⁻⁸`. For matrices with spectral radius ∼ 10⁶, eigenvalues at magnitude ∼ 10⁻⁸ that are numerically significant are classified zero-rank. The same `AlmostEqual` call short-circuits `Determinant` to `0.0` when any eigenvalue has `|λ| < 3.33 × 10⁻⁸`. Receipt-level rank must be recomputed with a spectral-radius-relative threshold.

[HERMITIAN_EIGENVALUE_CARRIER]:
- For `Matrix<Complex>` with `Symmetricity.Hermitian`, `DenseEvd.Create` branches on `matrix.IsHermitian()` and routes to `SymmetricTridiagonalize → SymmetricDiagonalize → SymmetricUntridiagonalize`. After convergence, `e[i] = 0.0` exactly, and `vectorEv[i] = new Complex(d[i], 0.0)` — Hermitian eigenvalues have exactly zero imaginary part in the receipt.
- `SymmetricUntridiagonalize` applies conjugate multiplications to restore complex eigenvector structure from the real rotations used during diagonalization: `dataEv[j * order + i] = dataEv[j * order + i].Real × tau[i].Conjugate()`. Hermitian eigenvectors are genuinely complex with nonzero imaginary parts even though eigenvalues are real. Projecting eigenvectors to their real parts is incorrect for the Hermitian case.
- Eigenvalue ascending sort applies to both real symmetric and complex Hermitian paths: a selection sort over `d[]` that swaps corresponding eigenvector columns. Each swap touches n elements, making the sort O(n²) in permutations and O(n³) in work in the worst case. The LAPACK provider bypasses this managed sort and uses the native driver's ascending-order contract.
- For nonsymmetric complex input, `vectorEv[i]` is filled by the complex Schur path with generally nonzero imaginary parts; no real-part projection is valid.

[MATRIX_MULTIPLY_API_PRECISION]:
- `A.TransposeAndMultiply(B)` computes A·Bᵀ. `A.TransposeThisAndMultiply(B)` computes Aᵀ·B. `A.ConjugateTransposeAndMultiply(B)` computes A·Bᴴ. `A.ConjugateTransposeThisAndMultiply(B)` computes Aᴴ·B. These are verified against BLAS flag pairs passed to `MatrixMultiplyWithUpdate`.
- The Rayleigh-Ritz matrix S = Wᵀ·A·W: `W.TransposeThisAndMultiply(A.Multiply(W))` is the two-call form. Neither form exploits S's symmetry; both allocate an intermediate n×3k matrix. For sparse A, `SparseMatrix.DoMultiply(Matrix<double> other, Matrix<double> result)` does NOT densify when `other.Storage` is `DenseColumnMajorMatrixStorage<double>` — it reads sparse row pointers directly against the dense backing array. The O(nnz × k) cost is sparse-aware. The densification fallback triggers only for user-defined storage types.
- `DenseMatrix.IsSymmetric()` uses exact bit equality `_values[j * n + i] != _values[i * n + j]`. After floating-point accumulation from Wᵀ·A·W, this never returns true. Passing `Symmetricity.Unknown` to `Evd` on any accumulated matrix always routes through the nonsymmetric Hessenberg path. `Symmetricity.Symmetric` must be passed explicitly after re-symmetrization via `MapIndexedInplace((i, j, v) => (v + s[j, i]) / 2.0, Zeros.Include)`. The IEEE 754 commutativity of addition makes `(a + b) / 2` and `(b + a) / 2` bitwise identical, so the subsequent `IsSymmetric()` check (if triggered by a provider that re-checks) passes.

[OVER_COMPUTE_TRUNCATION_ALLOCATION]:
- `evd.EigenValues.AsArray()` returns the raw backing `Complex[]` without copy for `DenseVector<Complex>`. `evd.EigenVectors.SubMatrix(0, n, 0, k)` always allocates a new n×k backing array — there is no zero-copy view. For cached basis reuse, retain the full `DenseMatrix` from `evd.EigenVectors`; each filter call pays one `SubMatrix` allocation, not the O(n³) eigensolver.
- The receipt shape for over-compute-and-truncate carries `ComputedCount = n`, `RequestedCount = k`, and `CacheSource` — `Full` (truncated from a cached n-wide basis) vs `Fresh` (new full computation). A `Resident` annotation records whether the `Evd` object is retained in memory, trading ≈ 24 MB resident (for n = 1000) against recomputation cost.
- `evd.EigenValues` is always `Vector<Complex>`. For real symmetric inputs the imaginary parts are exactly zero. Projecting to a `double[]` weight array: `evd.EigenValues.AsArray().Select(c => c.Real).ToArray()` is O(n). `MemoryMarshal.Cast<Complex, double>` yields a stride-2 span whose even-indexed elements are real parts; no strided `TensorPrimitives` overload exists, making the scalar projection loop the canonical route — it is LLVM-JIT-autovectorized for sequential real-part extraction on .NET 10.

[SPECTRAL_FILTER_ZERO_MODE_MECHANICS]:
- `Vector<T>.Map(f, result, Zeros.Include)` passes zero-valued elements to the map function. `Zeros.AllowSkip` (the default) may skip zero-valued eigenvalues; for `DenseVector` the distinction is immaterial since all elements are stored. `MapIndexed` with the index as a guard (`i == 0 ? 0.0 : f(λᵢ)`) enables index-first exclusion for normalized graph Laplacians where the 0th mode is always the zero mode.
- Partial-monoid weight composition via `weights1.PointwiseMultiply(weights2)` is O(n). An identity-filter check `weights.ForAll(w => w == 1.0)` uses `Storage.Find` internally and short-circuits on the first mismatch, avoiding the O(n) multiply for the common case.
- Heat filter weight computation with vectorized primitives: project real parts first (`weights[i] = eigenValues[i].Real`, O(n) scalar loop), then `TensorPrimitives.Multiply<double>(weights, -t, weights)` in-place, then `TensorPrimitives.Exp<double>(weights, weights)` in-place. `TensorPrimitives.Exp<double>` compiles because `double : IExponentialFunctions<double>`. The projection loop is the only non-vectorized step and is LLVM-JIT-autovectorized on .NET 10.
- When zero-mode exclusion combines with the heat filter: `weights[i] = eigenValues[i].Real < ε_zero ? 0.0 : exp(-t × eigenValues[i].Real)` cannot split into the two vectorized calls without a separate masking pass. Apply vectorized exp first, then `weights.MapIndexedInplace((i, _, w) => eigenValues[i].Real < ε_zero ? 0.0 : w, Zeros.Include)` as one additional scalar pass. Vectorized exp plus one scalar mask is still faster than n scalar `Math.Exp` calls for large n.

[EXHAUSTION_SEMANTICS_RECEIPT]:
- `NonConvergenceException` carries no partial-result payload. When thrown at iteration 1000 for eigenvalue index `j`, eigenvectors for indices `< j` that converged earlier are complete and orthonormal in their columns. Eigenvectors at index `>= j` are intermediate Givens-rotation products — not usable as eigenvectors. The failing index `j` is not exposed by the exception (`NonConvergenceException.Message = "An algorithm failed to converge."`, no inner exception).
- The iteration counter `num3` is per-eigenvalue — it resets per outer `j` loop. A matrix where eigenvalue 97 (of 100) is nearly degenerate exhausts 1000 iterations for that eigenvalue alone while eigenvalues 0–96 converged cheaply. The spectral radius estimate `num2` (running `max(|d[k]| + |e[k]|)`) is not exposed at the exception site. Recording the spectral radius before calling `Evd` and including it in the `ExhaustionFault` payload is the only way to supply the near-degeneracy diagnostic.
- Exhaustion, indefinite-B, and non-square failures all collapse to `Fin.Fail(Error)` at the boundary. Domain-specific error codes (`ExhaustionCode`, `IndefiniteBCode`, `ShapeCode`) established before the catch site are the correct discriminants; inspecting `error.Exception?.GetType()` at use sites leaks exception-handling logic into domain code.

[CONGRUENCE_BACK_TRANSFORM_SHAPE]:
- The back-transform `X = L⁻ᵀ·Y` via `chol.Factor.Transpose().LU().Solve(evd.EigenVectors)` allocates three n×n matrices: the `Transpose()` copy, the LU factored copy, and the solve result. For k eigenvectors, the total cost is two O(n²) LU factorizations of Lᵀ plus O(n·k) back-substitution.
- The two-stage failure structure of congruence reduction must be reflected in the receipt: if Cholesky throws, no eigenpairs exist; if Cholesky succeeds but Evd on C exhausts, the `ExhaustionFault` receipt should carry the Cholesky evidence (L is valid) alongside the failure, because L's condition number diagnoses whether the problem is ill-conditioned B or a genuinely near-degenerate spectrum.

[DESCRIPTOR_NORMALIZATION_AND_DEGENERATE_SPREAD]:
- A descriptor normalized by `(λ_max − λ_min)` requires a `DegenerateSpread` receipt case when `λ_max == λ_min`. This is distinct from `Unusable` (non-converged pairs) and from success. A degenerate spread means the filter has collapsed all frequencies into one bin; the filtered output carries zero information content.
- For Laplacian-derived operators with zero-mode excluded, the spread floor is `eigenvalues.Where(λ => λ.Real > ε_zero).Min(λ => λ.Real)`. Using `EigenValues[1].Real` directly is incorrect when the second eigenvalue is itself below the exclusion threshold.
