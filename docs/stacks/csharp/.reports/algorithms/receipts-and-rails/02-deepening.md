# Numeric Evidence Law: Receipts and Rails — Deepening

[INTEGRATION_EVIDENCE_RECEIPTS]:
- `Integrate.GaussKronrod` exposes two evidence channels through `out` parameters: `error` (the difference between the (N-1)/2-point Gauss approximation and the N-point Kronrod approximation) and `L1Norm` (the L1 norm of the result). These are structurally distinct: `error` is the embedded error estimator intrinsic to the G-K pair, while `L1Norm` measures whether the returned value is close to its own norm — a large disparity between the two signals a poorly conditioned integrand rather than a convergence failure. A receipt that stores only the return value discards both. The receipt factory must capture `(Value, ErrorEstimate, L1Norm, TargetRelativeError, MaximumDepth, Order)` so re-admission can verify `ErrorEstimate <= TargetRelativeError * |Value|` and that `|Value - L1Norm| / L1Norm` stays below a policy threshold.
- `GaussKronrodRule.Order` exposes the integer rule order as a property on the rule object. Pre-computed point sets exist for orders 15, 21, 31, 41, 51, and 61. The order is an algorithm-specific fact: a 15-point rule integrates polynomials exactly up to degree 29; a 61-point rule up to degree 121. A receipt that omits the rule order makes the exactness claim non-auditable. When the receipt drives downstream quadrature assembly that sums element integrals, the order determines what source functions are integrated exactly, not merely how many function evaluations were taken.
- `DoubleExponentialTransformation.NumberOfMaximumLevels` is a fixed constant bounding the adaptive subdivision depth. A receipt for double-exponential quadrature must record the actual depth reached alongside `NumberOfMaximumLevels`; the ratio of actual-to-maximum depth is the degeneracy counter for near-singularity or endpoint behavior that forced early saturation.
- `NewtonCotesTrapeziumRule.IntegrateAdaptive` is the only quadrature surface that operates without embedded error output parameters; it returns a scalar and accepts only a `targetAbsoluteError`. A receipt wrapping this call must record the target absolute error and the method choice. Because no error estimator is returned, the receipt cannot self-validate — the conservation check is limited to `|estimate - coarser estimate| <= targetAbsoluteError`, which must be performed externally and stored as the receipt's validity fact.

[DIFFERENTIATION_EVIDENCE]:
- `NumericalDerivative.StepType` is a three-case enum (`Absolute`, `RelativeX`, `Relative`) that governs how the base step `h` is scaled. `Relative` computes `dx = (1 + |x|) * h^(2/(order+1))` from `Epsilon` (defaults to machine epsilon); `RelativeX` uses `BaseStepSize * (1 + |x|)`; `Absolute` uses `h` directly. The scaling choice changes both the accuracy and the function-evaluation count, and it changes the meaning of `BaseStepSize` — under `Relative`, `BaseStepSize` is derived from `Epsilon` and the order, while under `RelativeX` it is a caller policy. A differentiation receipt that records only a scalar `StepSize` conflates three distinct policies.
- `NumericalDerivative.Evaluations` is a cumulative counter across all derivative computations made by a single instance, not a per-call count. A receipt wrapping a single derivative evaluation must snapshot `Evaluations` before and after the call and record the delta. That delta equals `Points` (the stencil width), providing the conservation check `EvaluationsUsed == Points` for a single-order evaluation on a univariate function.
- `NumericalDerivative.StepSize` get-after-set may return a different value than what was assigned, because the implementation rounds to the nearest base-2 representable number to improve finite difference accuracy. A receipt that records `StepSize` must read it back after assignment — the stored value is the actual `h` used, not the requested `h`. Recording the requested value produces a gap between the policy and the execution fact.
- `StepType.Relative` with `Center = 0` (one-sided forward difference) is fundamentally less accurate than center differences for smooth functions: the step scales as `h^(1/(order+1))` for one-sided versus `h^(2/(order+1))` for centered. A receipt carrying both `Center` and `StepType` is the minimum evidence to determine whether the derivative estimate is first-order or second-order accurate in the step size.

[SPECTRAL_FFT_EVIDENCE]:
- `FourierOptions` is a `[Flags]` enum with six named combinations: `Default` (symmetric scaling, standard exponent), `InverseExponent` (positive exponent on forward), `AsymmetricScaling` (1/N on inverse only), `NoScaling`, `Matlab` (alias for `AsymmetricScaling`), and `NumericalRecipes` (alias for `InverseExponent | NoScaling`). These encode two orthogonal honesty facts — exponent convention and scaling convention — that are independent and composable. A spectral receipt must store the full `FourierOptions` bitfield, not a prose description, because two passes of `Forward` under different options produce outputs that are scale-different by `1/N` and phase-conjugate relative to each other.
- `Fourier.FrequencyScale(int length, double sampleRate)` generates the frequency axis in the same order as the FFT output: DC at index 0, positive frequencies up to Nyquist, then negative frequencies wrapped. A receipt that stores an FFT result without the sample rate and convention cannot produce a valid frequency axis; the receipt must capture `(SampleRate, Length, Options)` as co-equal evidence alongside the transform output.
- `Fourier.ForwardReal` packs the conjugate-symmetric spectrum into an array of length `N+2` (N even) or `N+1` (N odd), not `N`. A receipt wrapping a real-signal forward transform must record `N` (the time-domain sample count) separately from `data.Length` — they differ by 1 or 2, and using `data.Length` as a proxy for N produces an off-by-one in every downstream frequency bin computation.

[FITTING_AND_GOODNESS_RECEIPTS]:
- `GoodnessOfFit.RSquared` computes r², the square of the Pearson product-moment correlation coefficient between modelled and observed values — explicitly not the coefficient of determination R². `GoodnessOfFit.CoefficientOfDetermination` computes R² = 1 − SS_res/SS_tot. These two coincide only for simple linear regression with an intercept; they diverge for polynomial fits, no-intercept fits, and nonlinear models. A fitting receipt that stores a single scalar "R²" without recording which computation produced it is an honesty failure. The receipt must store both quantities under unambiguous names and record the degrees-of-freedom count used in `GoodnessOfFit.StandardError`.
- `GoodnessOfFit.PopulationStandardError` uses an N normalizer with no Bessel correction; `GoodnessOfFit.StandardError` takes an explicit `degreesOfFreedom` parameter and uses `N - degreesOfFreedom`. The distinction matters when the model parameter count is non-trivial relative to the sample count. A receipt that reports population standard error when the caller expects the sample standard error from a fitted model violates the honesty-marker law because the degrees-of-freedom adjustment is load-bearing for confidence interval construction.
- `DirectRegressionMethod` selects among `NormalEquations` (Cholesky of XᵀX), `QR` (thin QR on the design matrix), and `Svd` (SVD on the design matrix). These three paths have distinct numerical stability properties: `NormalEquations` squares the condition number, `QR` is stable for full-rank overdetermined systems, and `Svd` handles rank-deficient systems. A fitting receipt must record the method used, because re-admission into a downstream uncertainty quantifier carries different numerical guarantees depending on whether the coefficient vector was produced by Cholesky or SVD.

[STATISTICS_ACCUMULATOR_RECEIPTS]:
- `RunningStatistics.Combine(a, b)` is a pure static factory that merges Welford moment accumulators from two independent runs. The conservation law is `combined.Count == a.Count + b.Count`. A hierarchical receipt that embeds two independently computed `RunningStatistics` instances and a combined parent must assert this identity at construction; violation signals either a dropped sample or an incorrectly initialized sub-accumulator.
- `RunningWeightedStatistics` uses reliability weights (inverse-variance weighting), not frequency weights. The `Count` property returns the number of samples with non-zero weight, not the sum of weights. A receipt that conflates reliability-weighted statistics with frequency-weighted statistics produces wrong estimates: the effective sample size under reliability weighting is `w1² / w2` (where `w1` is the sum of weights and `w2` is the sum of squared weights), not `Count`. The receipt must record the weight sum alongside `Count` so the effective sample size can be derived.
- `MovingStatistics` is a windowed accumulator with a fixed capacity. It exposes the same `Count`, `Mean`, `Variance`, `Minimum`, `Maximum` surface as `RunningStatistics` but over a sliding window, not the full history. A snapshot of `RunningStatistics.Mean` at time T is the global mean; a snapshot of `MovingStatistics.Mean` is the window mean. The two are API-compatible but semantically incompatible for receipt persistence; a receipt must record which type produced the statistics as an honesty marker.
- `DescriptiveStatistics` carries an `increasedAccuracy` constructor flag that internally switches from `double` to `decimal` arithmetic; the documentation notes this overflows for large absolute values. The flag is not exposed as a property after construction. A receipt wrapping `DescriptiveStatistics` must record it separately at construction time, because it determines whether the computed moments meet the NIST accuracy threshold or only the standard `double` threshold.

[ATOM_VALIDATOR_AND_RECEIPT_CONSTRUCTION]:
- `Atom<T>.New(T, Func<T, bool> validator)` runs the validator on the initial value at construction time and refuses to return an `Atom<T>` whose initial state fails the validator. When a numeric receipt is stored in an `Atom<TReceipt>` with a validator encoding conservation invariants (for example, `r => r.Rank + r.Nullity == r.Dimension`), the atom cannot enter an invalid state at construction or through subsequent `Swap` calls. When `Swap(f)` produces a value that fails the validator, the atom silently retains its previous value and returns a snapshot of it — there is no error channel. A receipt-holding atom that uses a validator must be paired with an external `Fin<TReceipt>` check on the factory result, because a `Swap` that returns the old value rather than the computed value is silent data loss.
- `AtomHashMap<K,V>.FindOrMaybeAdd(K, Func<Option<V>>)` is the conditional variant: when the factory returns `None`, no insertion occurs and `None` is returned. This is the correct surface for a receipt cache that must not insert a failed-construction receipt — the factory returns `Option<TReceipt>.None` on precondition failure. The non-maybe overload `FindOrAdd(K, Func<V>)` has no suppression path; a factory that can fail must never be passed to it, because a receipt built with invalid inputs will be permanently cached on first failure.
- `AtomHashMap<K,V>.SwapKey(K, Func<Option<V>, Option<V>>)` maps the current `Option<V>` — present or absent — to a new `Option<V>`: `Some(x) → Some(y)` updates, `Some(x) → None` removes, `None → Some(y)` inserts, and `None → None` is a no-op. Passing `_ => None` removes the cached entry without a lock and is atomic within the CAS loop; a concurrent `FindOrAdd` racing with the invalidation sees either the old receipt or the absent slot, never a partially-written state.

[ERROR_RAIL_PROVENANCE]:
- `Error.New(int code, string message)` creates an `Expected` error (`IsExceptional` is `false`, `IsExpected` is `true`). `Error.New(Exception ex)` creates an `Exceptional` (`IsExceptional` is `true`). This distinction is load-bearing: algorithm convergence failures (`NonConvergenceException`, `OptimizationException`) surface as `Exceptional` errors wrapping the library exception exactly once at the boundary; domain-level precondition failures (rank deficit, non-positive-definite input) detected before invoking the library surface as `Expected` errors with domain-defined integer codes. Collapsing both families to `Expected` hides whether the error originated in a domain predicate or in library internals.
- `Error.New(int code, string message, Error inner)` chains errors with a parent code. Converting `NumericalBreakdownException` to `Error.New(domainCode, ex.Message, Error.New(ex))` preserves both the domain provenance code (for dispatch in receipt factories) and the original exception (for diagnostic extraction); the chain is accessible via `Error.Inner`. Flattening to a single `Error.New(ex)` discards the domain code; flattening to `Error.New(code, message)` discards the exception.
- `Error.Many(Seq<Error>)` and `Error.Many(Error[])` both exist and accumulate independent precondition failures — the correct surface for a receipt factory that validates multiple conservation identities simultaneously. A factory that short-circuits on the first failure cannot distinguish "rank check failed" from "both rank and nullity checks failed independently." Accumulating with `Error.Many` and returning one `Fin.Fail(aggregated)` preserves the full violation set for diagnostic review.

[TOLERANCE_POLICY_SPECIFICITY]:
- `NumericalDerivative.StepType.Relative` produces a per-order base step via `h = Epsilon^(2/(order+1))`. For order 1 this gives `h ≈ epsilon^(2/3) ≈ 6.1e-6` when `Epsilon = machine_epsilon ≈ 2.2e-16`; for order 2, `h ≈ epsilon^(1/2) ≈ 1.5e-8`. A tolerance receipt for a finite-difference gradient must record `StepType`, the `Epsilon` base, and the derivative `order` together — the effective `h` is fully determined by these three values and cannot be inferred from any one alone.
- `ResidualStopCriterion<T>._minimumIterationsBelowMaximum` introduces a hysteresis count: convergence is not declared until the residual has been below the threshold for this many consecutive iterations. The constructor default is `0`, meaning convergence is declared on the first iteration where the residual drops below the threshold. A receipt that records only the final `IterationStatus` cannot be re-audited to determine whether convergence was declared prematurely. The receipt must capture both `Maximum` (threshold) and `MinimumIterationsBelowMaximum` from the criterion instance.
- `DivergenceStopCriterion<T>._residualHistory` tracks residual norms for the look-back window; the window length equals `MinimumNumberOfIterations`. A divergence-flagged receipt must record the window length alongside the `MaximumRelativeIncrease` threshold — the same relative increase over a 5-iteration window versus a 20-iteration window can produce opposite divergence decisions, and the window length is the policy that resolves which conclusion is valid.

[HIERARCHICAL_RECEIPT_NESTING_MECHANISMS]:
- A composite solve receipt that embeds a child assembly receipt (sparse matrix assembly), a child factorization receipt (sparse Cholesky), and a child iterative refinement receipt (CG correction) satisfies the three-tier nesting law: the outer receipt owns the solve outcome and the conservation check (accepted plus rejected equals input count; residual norm at exit); the middle receipt owns the factorization evidence (`Factor`, `Determinant`, `DssMatrixType`, `DssMatrixStructure`); and the inner receipt owns the iterative trail (`IterationStatus`, residual trajectory, step count). Re-admission replays the conservation checks at each tier without re-executing any tier.
- Both `TrySolveIterative` overloads are void and mutate the `result` parameter. The `TrySolveIterative(input, result, solver, iterator, preconditioner)` overload that takes `Iterator<T>` exposes the aggregated terminal status via `iterator.Status` after the call; the `IIterationStopCriterion<T>[]` overload requires querying each criterion individually to read the terminal status. A receipt factory uses the `Iterator<T>` overload to capture the aggregated status as a first-class post-call fact rather than assembling it from per-criterion queries.
- `RunningStatistics.Combine(a, b)` is a pure static factory that produces a new accumulator without mutating its arguments. This makes it directly usable as the `f` in `Atom<RunningStatistics>.Swap` — the state transition from `current` to `RunningStatistics.Combine(current, newSample)` is pure and safe for CAS-loop retry. The conservation invariant is carried through the atom's validator: `s => s.Count >= 0 && !double.IsNaN(s.Mean)` is both cheap and load-bearing.

[DEGENERACY_COUNTING_AND_MOLLIFICATION]:
- Eigenvalue clamping in a receipt occurs when raw eigenvalues from `Evd<T>.EigenValues` lie outside an acceptable domain: negative eigenvalues on a nominally positive-semidefinite matrix, or complex eigenvalues with non-negligible imaginary parts on a nominally real-symmetric matrix. A receipt records the count of clamped eigenvalues as a per-receipt counter — it does not replace the clamped values in the stored vector. The original eigenvalue vector and the count of mollifications are co-equal evidence; modifying the vector destroys the original-versus-mollified distinction needed for re-admission.
- `Cholesky.Create` throws `ArgumentException` on non-SPD input; there is no partial-success path. For a near-SPD matrix, the domain layer must shift the diagonal before invoking Cholesky — and the shift magnitude is degeneracy evidence. A receipt wrapping a shifted Cholesky records the applied shift as a named field `DiagonalShift` so downstream callers that interpret factored quantities (log-determinant, condition number) can account for the shift rather than silently treating the shifted matrix as the original.
- The `Precision.AlmostEqual` overloads applied to eigenvalues for rank counting use `DefaultDoubleAccuracy = DoublePrecision * 10`. This tolerance cannot be parameterized through the `Evd<T>` API — it is hardcoded. A receipt that needs a different rank threshold must recount from the raw `EigenValues` vector using a domain policy tolerance, and must record both the hardcoded-threshold rank (as `LibraryRank`) and the domain-policy rank (as `DomainRank`) alongside the policy constant, so a re-admission check can verify that the discrepancy between the two is within the expected tolerance envelope.
