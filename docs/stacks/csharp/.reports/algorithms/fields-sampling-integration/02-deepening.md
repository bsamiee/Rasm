# Fields, Sampling, and Integration — Deepening

[INTEGRATOR_CONVERGENCE_MECHANICS]:
- `GaussKronrodRule.recursive_adaptive_integrate` applies a cubic change-of-variable on the finite interval before calling the non-adaptive kernel: the map `(b-a)/4 * t*(3-t²) + (a+b)/2` on `[-1,1]` is a degree-3 polynomial — a different substitution than the affine map a raw Gauss-Legendre rule on `[a,b]` applies. The error estimate per node is `max(|GK_N - G_{(N-1)/2}|, |GK_N * 2ε|)`, clamping to machine-epsilon relative noise to prevent recursion beyond numerical resolution.
- The `absTol` halving in `recursive_adaptive_integrate` means that refinement in deep sub-intervals is not bounded to the original relative tolerance: for integrands with a sharp spike covered by many levels, the accumulated error can exceed `relTol * |result|` because `absTol` splits geometrically rather than proportionally to local magnitude.
- `Integrate.GaussLegendre` at the facade applies the same cubic substitution `(b-a)/4 * t*(3-t²) + (a+b)/2` on top of the already-mapped Gauss-Legendre nodes — composing two changes of variable. Callers expecting raw Gauss-Legendre nodes on `[a,b]` must use `new GaussLegendreRule(a, b, order)` with its instance `Integrate` method, which applies only the linear affine map.
- The 3D cubature `GaussLegendreRule.Integrate(Func<double,double,double,double>, ...)` reduces to a lambda-captured outer quadrature over the 2D inner rule. Each outer evaluation re-fetches the thread-local `GaussPoint` for the same order; because order is constant across all invocations, no recomputation occurs. The cost is `n² + n` evaluations for even order `n` rather than `((n+1)/2)²` for odd order, because the outer dimension does not exploit reflection symmetry.
- `GaussLegendrePointFactory._gaussLegendrePoint` is `[ThreadStatic]` with a single slot per thread — it caches only the most recently requested order. Parallel integration with `Task.WhenAll` or `Parallel.For`, where each task uses a different order, produces cache misses on every call. The correct pattern is to construct named `GaussLegendreRule` instances once at setup and capture them in closures; the instance holds `_gaussKronrodPoint` as a private field rather than in the thread-local slot.

[INTERPOLATION_ALGEBRA_AND_CAPABILITY_MAP]:
- `Barycentric.InterpolateRationalFloaterHormannSorted(x, y, order)` computes alternating-sign barycentric weights over blending windows of width `order`. The overload without `order` defaults to `x.Length - 1`, which is the full-degree polynomial. This interpolant supports neither differentiation nor integration through the `IInterpolation` contract; it is the correct route for near-singular data, but the contract flags must be checked before composing it with operations that require those capabilities.
- `CubicSpline.InterpolateMakimaSorted` uses the modified Akima weight formula: `|Δm_{i+1} - Δm_i| + 0.5 * |Δm_{i+1} + Δm_i|`. The `0.5 * |sum|` term prevents zero weights when opposing slopes cancel — the failure mode of classic Akima at those nodes. When all four surrounding differences vanish (four consecutive equal slopes), the fallback is `0.5 * (m[n+3] + m[n])` — the arithmetic mean of the extended-boundary slopes. MAKIMA therefore never produces zero-weight artifacts at constant-slope regions; Akima does.
- `CubicSpline.InterpolatePchipSorted` zeros the slope at monotone-sign-change intervals and applies the harmonic-mean formula `(h0 + h1) * h0 / ((2*h1+h0)/m[j-1] + (h1+2*h0)/m[j])` for interior slopes with matching signs. This guarantees C¹ continuity and monotonicity preservation within each monotone segment. `PchipEndPoints` applies `|endpoint_slope| ≤ 3 * |first_interval_slope|` clamping at domain boundaries to prevent extrapolation overshoot.
- `TransformedInterpolation.InterpolateSorted` applies `transformInverse` to the `y` values via `CommonParallel.For` with 4096-element chunks, then constructs a `LinearSpline` over the transformed values and wraps it with a forward `transform` at evaluation time. The `_interpolation` field is always `LinearSpline` — the factory hardcodes this; `SupportsDifferentiation` and `SupportsIntegration` are both `false`. For log-linear interpolation via this surface, see `LogLinear`, which wraps `LinearSpline` directly and implements both capabilities.
- `FiniteDifferenceCoefficients` builds the full coefficient table at construction via `DenseMatrix.Inverse()`: a `p×p` Vandermonde-like system, inverted in `O(p³)`, producing exact rational stencil weights rounded to `MidpointRounding.AwayFromZero`. The constructor allocates one dense matrix per center position sequentially; at `p=15`, the 3375-operation solve runs once at construction and the resulting table is reused for all subsequent queries.
- `BulirschStoerRationalInterpolation.Interpolate(t)` initializes two working arrays `C` and `D` with `y[i]` and `y[i] + 1e-25` respectively. The `1e-25` shift on `D` is the pole-avoidance perturbation; when the rational update denominator rounds to zero under ULP tolerance, the method returns `NaN` rather than throwing. Callers composing this over a field with poles must test for `NaN` on each evaluation.
- `QuadraticSpline` is a full `IInterpolation` implementor with `SupportsDifferentiation: true` and `SupportsIntegration: true` as explicit interface implementations. It requires direct construction — `new QuadraticSpline(x, c0, c1, c2)` with pre-computed coefficient arrays — because there is no `Interpolate.*` facade entry for it. Its `Differentiate2(t)` returns `2.0 * c2[segment]` — a piecewise-constant second derivative across each segment. For smooth field curvature estimation, this constant curvature per segment is a design limitation relative to `CubicSpline`.

[SAMPLING_DISTRIBUTION_AND_DENSITY_MECHANICS]:
- `SystemRandomSource.FastDoubles(values)` bypasses `Default` for arrays of length ≥2048: it parallelizes via `CommonParallel.For` with block sizes of 1024, 2048, or 8192, constructing a fresh `System.Random` from `RandomSeed.Robust()` per block. The result is non-deterministic regardless of any per-thread seeding, because each block receives an independent OS-entropy seed.
- `HybridMC`'s default gradient uses step `max(0.001, 1e-6 * |x[i]|)` — a hardcoded absolute floor of `0.001` that dominates when `|x[i]| < 1000`. This is unsuitable for fields with sharp gradients near zero or for large-scale parameter spaces. Supplying a `DiffMethod` via the eight-parameter constructor overload is mandatory for such fields; the `UnivariateHybridMC` scalar gradient follows the same floor rule.
- `KernelDensity.Estimate` uses `CommonParallel.Aggregate` with no degree-of-parallelism cap exposed to the caller. When a field integration pipeline spawns multiple concurrent KDE evaluations, the inner `CommonParallel.Aggregate` thread usage is independent of any outer `ParallelOptions.MaxDegreeOfParallelism` — the two parallelism layers do not compose. This can saturate the thread pool when many KDE calls run concurrently.
- `Xoshiro256StarStar` serializes its 256-bit state as four `[DataMember(Order=1..4)]` `UInt64` fields via `[DataContract]`. A seeded instance can be serialized mid-sequence and deserialized to reproduce the exact continuation — the correct approach for blue-noise sampling or deterministic field sampling that must checkpoint and resume. `MersenneTwister` carries the same `[DataContract]`; `SystemRandomSource` wraps `System.Random`, which does not support this.

[SPLINE_INTEGRATION_EXACTNESS_AND_LAZY_SEMANTICS]:
- `CubicSpline.ComputeIndefiniteIntegral` computes a prefix sum of segment integrals via Horner evaluation: `integral[i+1] = integral[i] + h*(c0 + h*(c1/2 + h*(c2/3 + h*c3/4)))` where `h = x[i+1] - x[i]`. The result is exact in floating-point for polynomials and runs in `O(N)`. Calling `Integrate(a, b)` evaluates `Integrate(b) - Integrate(a)`, each requiring one binary search; warming the `Lazy<>` via a single `Integrate` call at construction eliminates lazy-initialization overhead from the hot path.
- `SplineBoundaryCondition.SecondDerivative` with value `0.0` (natural spline) minimizes overshoot at domain boundaries with no knowledge of the field's behavior outside the sample range. `SplineBoundaryCondition.FirstDerivative` with value `0.0` (clamped flat) is appropriate when the field is known to be extremal at the boundary. `SplineBoundaryCondition.Periodic` enforces equal first and second derivatives at both ends, suitable for periodic fields.
- `StepInterpolation.ComputeIndefiniteIntegral` accumulates `integral[i+1] = integral[i] + y[i] * (x[i+1] - x[i])` — a rectangle-rule sum over piecewise-constant segments. `LinearSpline` and `CubicSpline` share the same `Lazy<double[]>` + `Array.BinarySearch` pattern; the only difference is the polynomial degree evaluated per segment. The two capabilities are independently lazy-initialized — calling `Integrate` before `Differentiate` triggers only the integration `Lazy<>`.

[DE_TRANSFORM_AND_CONTOUR_INTEGRATION]:
- `GaussLegendreRule.ContourIntegrate` and `GaussKronrodRule.ContourIntegrate` accept `Func<double, Complex>` — a real-parameterized complex-valued integrand — and return `Complex`. These are line-integral contour variants: the integration variable is still a real scalar parameterizing a path. The `L1Norm` output in the Kronrod contour variant is `double`, accumulating `Complex.Abs` of each evaluation; the cancellation diagnostic `|result|/L1Norm` remains valid for complex-valued cancellation.
- `DoubleExponentialTransformation.Integrate` is a `static` method on the class itself (not accessible through a facade overload). It accepts only finite `[a, b]` and passes the integrand unchanged to `NewtonCotesTrapeziumRule.IntegrateAdaptiveTransformedOdd`. The `ContourIntegrate` static method on the same class accepts `Func<double, Complex>` and applies the same `tanh-sinh` kernel for complex-valued line integrals on finite intervals.

[ADAPTIVE_QUADRATURE_FAILURE_AND_DEPTH_SEMANTICS]:
- `GaussKronrodRule.recursive_adaptive_integrate` initializes `absTol = 0.0` and sets it to `|value * relTol|` at the root interval. Recursive calls halve `absTol` at each level; the dual condition `absTol < error` drives recursion. An integrand with a spike covered by many recursion levels can produce accumulated error exceeding `relTol * |result|` because `absTol` splits geometrically rather than proportionally to local magnitude.
- `GaussLegendreRule.Integrate(Func<double,double,double>, ...)` for the 2D case exploits reflection symmetry: the inner loop combines symmetric node pairs, halving the inner-loop count only when both dimensions share the same even-order rule. The savings vanish for odd orders.
- Neither `DoubleExponentialTransformation` nor any adaptive Gauss-Kronrod overload catches `NaN` or `Infinity` from the integrand. A field evaluator that returns `NaN` at a quadrature node propagates it silently through the returned scalar — no diagnostic is raised. The only defense is a finite-fallback guard on the integrand delegate, with any singular-point event recorded as typed failure outside the integral call.
